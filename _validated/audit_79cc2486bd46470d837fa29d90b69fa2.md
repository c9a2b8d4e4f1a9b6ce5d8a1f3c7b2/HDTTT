Audit Report

## Title
IDOR in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId` but never validates that the approver record identified by `:id` belongs to that transaction. An authenticated user who has created any transaction can pass their own `transactionId` for the ownership check while targeting an approver `id` from a completely different transaction, silently deleting it.

## Finding Description

**Root cause — missing cross-reference check in `removeTransactionApprover`**

The controller performs an ownership check on `transactionId`, then unconditionally delegates deletion using the unrelated `id` parameter: [1](#0-0) 

```typescript
@Delete('/:id')
async removeTransactionApprover(...) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId only
  await this.approversService.removeTransactionApprover(id);               // deletes by id — no cross-check
  return true;
}
```

The service method `removeTransactionApprover` accepts only the approver primary key and performs no transaction-membership validation: [2](#0-1) 

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id); // recursive soft-delete, no transactionId check
  emitTransactionStatusUpdate(...);
  return result;
}
```

`getCreatorsTransaction` only verifies `creatorKey.userId === user.id` for the supplied `transactionId` — the returned `Transaction` object is never used to validate the approver: [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-reference check that `removeTransactionApprover` omits: [4](#0-3) 

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
```

The two IDs (`transactionId` from the URL and the approver's own `transactionId`) are never compared in the delete path.

## Impact Explanation

- **Bypasses approval requirements**: Approvers represent the multi-signature gate before a Hedera transaction can execute. Stripping all approvers from a victim transaction can allow it to advance to `WAITING_FOR_EXECUTION` without any human approval, enabling unauthorized on-chain execution.
- **Disrupts organizational workflows**: Silently removing approvers corrupts the governance model for any transaction in the system.
- **Permanent state corruption**: `removeNode` performs a recursive soft-delete via raw SQL with no automatic restoration path. [5](#0-4) 

## Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (any registered organization user). Creating one transaction is sufficient to pass the ownership check.
- **Approver ID discovery**: IDs are sequential integers exposed in API responses to any user who is a participant in at least one transaction (via `GET /transactions/:transactionId/approvers`).
- **No rate limiting or anomaly detection** is visible in the codebase.
- The attack requires a single HTTP request per targeted approver.

## Recommendation

In `removeTransactionApprover` (controller or service), add the same cross-reference check already present in `updateTransactionApprover`:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to get the root node.
2. Assert `rootNode.transactionId === transactionId` (the URL parameter), throwing `UnauthorizedException` on mismatch.
3. Only then proceed with `removeNode`.

This mirrors the existing pattern in `updateTransactionApprover` at lines 386–394 of `approvers.service.ts`. [4](#0-3) 

## Proof of Concept

1. Attacker registers as a normal user and creates any transaction → receives `transactionId = 1`. This satisfies `getCreatorsTransaction`.
2. Attacker observes approver IDs from any transaction they participate in (sequential integers), or enumerates them. Target: approver `id = 42`, belonging to `transactionId = 99` (a victim's transaction).
3. Attacker sends:
   ```
   DELETE /transactions/1/approvers/42
   ```
4. Controller calls `getCreatorsTransaction(1, attacker)` → passes (attacker owns transaction 1).
5. Controller calls `removeTransactionApprover(42)` → service fetches approver 42, calls `removeNode(42)`, recursively soft-deletes approver 42 and all its children — which belong to transaction 99 — without any complaint.
6. Victim's transaction 99 now has its approver(s) silently deleted. [6](#0-5)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
