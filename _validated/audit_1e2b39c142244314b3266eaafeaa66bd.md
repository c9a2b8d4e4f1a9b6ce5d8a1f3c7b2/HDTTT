# Audit Report

## Title
TOCTOU Race in `archiveTransaction` Allows Unconditional Overwrite of `EXECUTED` Status with `ARCHIVED`

## Summary
`archiveTransaction` in `TransactionsService` reads the transaction status, validates it, then performs an unconditional `UPDATE` with no `WHERE status IN (...)` guard. A concurrent scheduler execution that transitions the transaction to `EXECUTED` between the read and the write is silently overwritten with `ARCHIVED`, permanently corrupting the audit trail.

## Finding Description

**Root cause — non-atomic state transition in `archiveTransaction`:**

`back-end/apps/api/src/transactions/transactions.service.ts`, lines 708–733:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user); // READ (line 709)

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP); // guard (lines 711–718)
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED }); // UNCONDITIONAL WRITE (line 720)
  ...
}
```

The `UPDATE` at line 720 carries no `WHERE status = <expected>` predicate. Whatever status the row holds at write time is overwritten.

**Contrast with the correctly guarded `cancelTransactionWithOutcome`:**

```typescript
const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.CANCELED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses }) // guarded
  .execute();
``` [2](#0-1) 

`cancelTransactionWithOutcome` is a no-op if the status changed between read and write. `archiveTransaction` has no such protection.

**The execute service's own write is also guarded:**

```typescript
.where('id = :id AND status = :currentStatus', {
  id: transaction.id,
  currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
})
``` [6](#0-5) 

This means the execute service correctly refuses to overwrite a status that has already changed — but `archiveTransaction` does not reciprocate that discipline.

**Race window:**

The scheduler fires every 10 seconds for transactions whose `validStart` is within 3 minutes: [4](#0-3) 

The window between `archiveTransaction`'s read (line 709) and its unconditional write (line 720) is the time for one DB round-trip — small but non-zero, and the creator controls when the archive request is issued.

**Exploit path:**

1. Creator submits a transaction; it reaches `WAITING_FOR_EXECUTION` (all required signatures collected).
2. Creator sends `PATCH /transactions/archive/:id` — `archiveTransaction` reads `status = WAITING_FOR_EXECUTION`, passes the guard.
3. Concurrently, the scheduler's `prepareTransactions` → `collateAndExecute` → `_executeTransaction` path submits the transaction to Hedera and writes `status = EXECUTED` (guarded, succeeds).
4. `archiveTransaction` then executes `repo.update({ id }, { status: ARCHIVED })` unconditionally, overwriting `EXECUTED`.

## Impact Explanation

- A transaction that was successfully executed on the Hedera network (real-world effect already occurred) is recorded as `ARCHIVED` in the database.
- All participants (signers, observers, approvers) receive notifications for `ARCHIVED` instead of `EXECUTED`.
- The audit trail is permanently corrupted: `ARCHIVED` carries no `executedAt` timestamp or `statusCode` (Hedera response code), so the execution result is lost.
- The creator can use this to conceal that a transaction was executed, misleading co-signers and observers about the outcome of a multi-party workflow.

## Likelihood Explanation

- **Attacker profile:** any authenticated user who is the creator of a transaction — no privileged access required.
- **Precondition:** the transaction must be in `WAITING_FOR_EXECUTION`, a normal, reachable state in every multi-signature workflow.
- **Timing:** `validStart` is stored in the database and returned to the creator, making the scheduler's execution window predictable. The creator can issue the archive request immediately before `validStart` to maximize the race probability.
- **Repeatability:** the request can be retried at zero cost until the race is won.

## Recommendation

Replace the unconditional `repo.update` in `archiveTransaction` with a guarded query builder UPDATE, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  // Status changed between read and write — re-fetch and surface the correct error
  const latest = await this.getTransactionForCreator(id, user);
  throw new ConflictException(
    `Transaction status changed to ${latest.status}; archive rejected.`,
  );
}
```

This makes the write atomic with respect to the status check: if the scheduler wins the race and sets `EXECUTED`, the `archiveTransaction` UPDATE affects 0 rows and the conflict is surfaced to the caller rather than silently corrupting the record.

## Proof of Concept

```
1. Create a multi-sig transaction with validStart = T+30s.
2. Collect all required signatures so status transitions to WAITING_FOR_EXECUTION.
3. At T+29s, send PATCH /transactions/archive/:id.
   - archiveTransaction reads status = WAITING_FOR_EXECUTION → guard passes.
4. At T+30s, the scheduler fires handleTransactionsBetweenNowAndAfterThreeMinutes,
   calls prepareTransactions → collateAndExecute → _executeTransaction.
   - Hedera confirms the transaction; _executeTransaction writes status = EXECUTED
     (guarded WHERE status = WAITING_FOR_EXECUTION → succeeds).
5. archiveTransaction's repo.update({ id }, { status: ARCHIVED }) executes
   unconditionally, overwriting EXECUTED → ARCHIVED.
6. Query the database: status = ARCHIVED, executedAt = NULL, statusCode = NULL.
   The on-chain execution is invisible in the audit trail.
```
