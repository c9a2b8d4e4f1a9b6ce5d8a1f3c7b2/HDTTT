Audit Report

## Title
Unbounded String Fields in Transaction and Comment DTOs Enable Authenticated DoS via Storage and Memory Exhaustion

## Summary
`CreateTransactionDto`, `CreateCommentDto`, and `CreateTransactionGroupDto` accept arbitrarily large strings with no `@MaxLength()` constraint. Any authenticated user can repeatedly submit near-2 MB payloads, accumulating unbounded data in PostgreSQL. When `getTransactionById` is called, it eagerly loads all stored comment bodies into server memory simultaneously, enabling OOM crashes and service degradation for all users.

## Finding Description

**Missing length constraints — confirmed in source:**

`CreateTransactionDto` declares `name` and `description` with only `@IsString()`: [1](#0-0) 

`CreateCommentDto` declares `message` with only `@IsString()`: [2](#0-1) 

`CreateTransactionGroupDto` declares `description` with only `@IsString()`: [3](#0-2) 

The `TransactionComment` entity maps `message` as a plain `@Column()` with no database-level length constraint either: [4](#0-3) 

The only server-side size guard is the global Express body parser limit of 2 MB per request: [5](#0-4) 

**No access control on comment creation — confirmed and explicitly flagged by developers:**

The `CommentsController` only requires a valid JWT and verified user status. There is no check that the commenter is associated with the target transaction. The developers themselves left a TODO acknowledging this gap: [6](#0-5) 

This means any authenticated user — regardless of whether they are a signer, observer, creator, or approver — can post comments on **any** transaction ID.

**Eager loading of all comments on every transaction fetch — confirmed:**

`getTransactionById` unconditionally loads the full `comments` relation for every call: [7](#0-6) 

This method is called from multiple code paths including `getTransactionWithVerifiedAccess`, `shouldApproveTransaction`, and `getTransactionForCreator`: [8](#0-7) [9](#0-8) [10](#0-9) 

**Rate limiting is insufficient:**

The global `IpThrottlerGuard` throttles by IP and request count, not by data volume. The `EmailThrottlerGuard` is scoped only to auth endpoints. Neither guard bounds cumulative per-field data written to the database: [11](#0-10) [12](#0-11) 

## Impact Explanation
- **Memory exhaustion / OOM crash**: When any code path calls `getTransactionById` on a transaction with thousands of large comments, all comment bodies are materialized into heap memory simultaneously, exhausting available RAM and crashing or severely degrading the API process for all users.
- **Database storage exhaustion**: Unbounded accumulation of large `name`, `description`, and `message` values with no per-user or per-field quota bloats PostgreSQL storage.
- **Query performance degradation**: Transaction listing endpoints that return `name`/`description` for all visible transactions are slowed by large field values.

## Likelihood Explanation
- **Precondition**: Valid authenticated session only — no admin or privileged role required.
- **Broadened attack surface**: Due to the missing access control on `createComment`, the attacker does not even need to own or be associated with the target transaction. Any authenticated user can flood comments on any transaction.
- **Attack complexity**: Trivial — a simple loop of HTTP POST requests to `POST /transactions/:id/comments` with a ~2 MB `message` body suffices.
- **Partial mitigations present but insufficient**: The 2 MB body limit caps each individual request but does not bound cumulative storage. The IP throttler limits request rate but not data volume per field.

## Recommendation
1. Add `@MaxLength()` constraints to all user-controlled string fields in `CreateTransactionDto` (`name`, `description`), `CreateCommentDto` (`message`), and `CreateTransactionGroupDto` (`description`). Reasonable limits (e.g., 500–2000 characters) should be chosen based on application requirements.
2. Add a database-level column length constraint in the `TransactionComment` entity (e.g., `@Column({ length: 2000 })`).
3. Implement access control in `CommentsController.createComment` to verify the requesting user is actually associated with the target transaction (signer, observer, creator, or approver) before allowing comment creation. The existing TODO comment in the code acknowledges this gap.
4. Consider lazy-loading or paginating the `comments` relation in `getTransactionById` rather than eagerly loading all comments unconditionally.

## Proof of Concept

```bash
# Step 1: Authenticate and obtain a JWT token
TOKEN=$(curl -s -X POST http://localhost:3000/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"attacker@example.com","password":"password"}' \
  | jq -r '.accessToken')

# Step 2: Generate a ~2MB message payload
LARGE_MSG=$(python3 -c "print('A' * 1900000)")

# Step 3: Flood a target transaction with large comments
# Note: no association with transaction ID 1 is required
for i in $(seq 1 500); do
  curl -s -X POST http://localhost:3000/transactions/1/comments \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d "{\"message\": \"$LARGE_MSG\"}" &
done
wait

# Step 4: Trigger memory exhaustion by fetching the transaction
# Any code path calling getTransactionById will now load ~950 GB of comment data
curl -s http://localhost:3000/transactions/1 \
  -H "Authorization: Bearer $TOKEN"
# Expected result: OOM crash or extreme latency / timeout
```
