Audit Report

## Title
Missing Transaction-Level Authorization on Comment Endpoints Allows Any Verified User to Read and Write Comments on Arbitrary Transactions

## Summary
`CommentsController` enforces only JWT authentication and account-status verification, but performs no check that the requesting user has any relationship to the target transaction. Any registered, verified user can post comments on and read all comments from any transaction in the system, including those they have no legitimate access to.

## Finding Description

**Root cause — missing transaction-level access check in `CommentsController`**

The controller applies three guards at the class level:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
``` [1](#0-0) 

`VerifiedUserGuard.canActivate` returns `user.status === UserStatus.NONE`, meaning it passes for any user who has completed account setup. It does not inspect which transaction is being targeted. [2](#0-1) 

The `createComment` handler passes the caller's user object and the attacker-supplied `transactionId` directly to the service with no further check. The developer left an explicit acknowledgement of the missing control:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(...)
``` [3](#0-2) 

`CommentsService.createComment` performs no access check; it creates and persists the comment unconditionally: [4](#0-3) 

The `GET` handler for listing comments is equally unguarded — it accepts a `transactionId` and returns all comments without verifying the caller's relationship to that transaction: [5](#0-4) [6](#0-5) 

**Contrast with the correct pattern used elsewhere**

Every other sensitive transaction endpoint calls `getTransactionWithVerifiedAccess`, which internally calls `verifyAccess`. That method checks whether the user is the creator, a signer, an observer, or an approver before granting access: [7](#0-6) [8](#0-7) 

The comments subsystem is the only area that skips this check entirely.

## Impact Explanation

1. **Unauthorized write — comment injection on private transactions.** A verified user with no relationship to a transaction can `POST /transactions/{id}/comments` with any `transactionId`. The comment is persisted and associated with that transaction, polluting the audit trail and enabling injection of misleading information into transactions belonging to other users.

2. **Unauthorized read — information disclosure.** A verified user can `GET /transactions/{id}/comments` for any `transactionId` and retrieve all comments ever posted on that transaction. Comments may contain sensitive operational details (signing instructions, key identifiers, business context) that the transaction's access control model was designed to keep private.

## Likelihood Explanation

**Attacker precondition:** Must hold a valid JWT for a verified account (`UserStatus.NONE`). This is a realistic insider-threat or compromised-account scenario within the organization.

**Exploit steps:** Once authenticated, the attacker enumerates sequential integer `transactionId` values (sequential primary keys, trivially guessable) and issues standard HTTP requests. No special tooling, no privilege escalation, no cryptographic attack required.

**Likelihood: High.** The vulnerability is reachable by any registered user with a single HTTP request, and the codebase's own `//TODO` comment confirms the gap was known but unresolved.

## Recommendation

In `CommentsController`, before delegating to `CommentsService`, call `TransactionsService.getTransactionWithVerifiedAccess(transactionId, user)` (or an equivalent access-check method). This will throw `UnauthorizedException` if the requesting user is not the creator, a signer, an observer, or an approver of the target transaction — matching the access control pattern already enforced on all other transaction endpoints.

Apply this check to both the `POST` (create) and `GET` (list) handlers.

## Proof of Concept

```
# Step 1: Authenticate as any verified user (UserStatus.NONE)
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Read all comments on a transaction the attacker has no access to
GET /transactions/42/comments
Authorization: Bearer <JWT>
→ 200 OK — returns all comments for transaction 42

# Step 3: Inject a comment on a transaction the attacker has no access to
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "message": "Attacker-injected comment" }
→ 201 Created — comment persisted against transaction 42
```

No special privileges are required beyond a valid verified-user JWT. The `transactionId` path parameter is the only input needed, and sequential integer IDs make enumeration trivial.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
```
