Audit Report

## Title
Missing Unique Database Constraint on `transaction_signer` Enables Duplicate Signer Insertion via Concurrent Requests (TOCTOU)

## Summary
The `transaction_signer` table has no unique constraint on `(transactionId, userKeyId)`. The application-level duplicate-signer check in `loadTransactionData` / `processTransactionSignatures` is a read-then-write pattern with no atomicity guarantee. Two concurrent `POST /transactions/:id/signers` requests from the same authenticated user can both pass the in-memory check and both insert duplicate `TransactionSigner` rows, corrupting the signer ledger for a transaction.

## Finding Description

**Root cause — no database-level uniqueness enforcement**

The `TransactionSigner` entity declares only a non-unique index:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← performance index, NOT unique
export class TransactionSigner { … }
``` [1](#0-0) 

The migration confirms no `UNIQUE` constraint exists on the table — the index created for `transaction_signer` is a plain `CREATE INDEX`, not `CREATE UNIQUE INDEX`:

```sql
CREATE INDEX "IDX_f8cbac3e0cae3730a762e4905e"
  ON "transaction_signer" ("transactionId", "userKeyId")
``` [2](#0-1) 

Compare this to other tables in the same migration that correctly use `CREATE UNIQUE INDEX` (e.g., `transaction_observer`, `cached_account_key`). [3](#0-2) 

**TOCTOU race in the application-level guard**

`uploadSignatureMaps` follows a read → validate → write pattern split across separate async steps:

1. **Read** — `loadTransactionData` fetches existing signers **outside** any database transaction and builds `signersByTransaction`: [4](#0-3) 

2. **Validate** — `processTransactionSignatures` uses the stale snapshot to decide which keys are "new":

```typescript
if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);
}
``` [5](#0-4) 

3. **Write** — `bulkInsertSigners` inserts without any conflict-resolution clause:

```typescript
await manager.createQueryBuilder()
  .insert().into(TransactionSigner).values(signersToInsert)
  .returning('*').execute();
``` [6](#0-5) 

The `persistSignatureChanges` method wraps only the writes in `this.dataSource.transaction(...)`, but the critical read in step 1 occurs entirely before that transaction begins, providing no isolation guarantee against concurrent reads. [7](#0-6) 

## Impact Explanation

- **Duplicate `TransactionSigner` rows** for the same `(transactionId, userKeyId)` pair are permanently persisted.
- Any downstream logic that counts `TransactionSigner` records to evaluate whether a signing threshold is met (e.g., `processTransactionStatus`) will see an inflated count, potentially advancing a transaction to `WAITING_FOR_EXECUTION` before all *distinct* required signers have actually signed.
- `bulkUpdateNotificationReceivers` is triggered for each duplicate insert, causing spurious notification state changes.
- The corrupted signer ledger is visible to all organization members and cannot be self-corrected by normal users.

## Likelihood Explanation

Any authenticated user who owns at least one key registered on a transaction can trigger this by issuing two HTTP POST requests to `POST /transactions/:id/signers` simultaneously (e.g., with `Promise.all` or two browser tabs). No privileged access, no leaked credentials, and no special network position are required. The race window spans the entire async processing pipeline from `loadTransactionData` through `bulkInsertSigners` — tens of milliseconds — which is wide enough to be reliably hit.

## Recommendation

Apply both a database-level and application-level fix:

1. **Add a unique constraint at the database level** — change the entity decorator and add a migration:
   ```typescript
   @Index(['transactionId', 'userKeyId'], { unique: true })
   ```
   And in the migration:
   ```sql
   CREATE UNIQUE INDEX "UQ_transaction_signer_tx_key"
     ON "transaction_signer" ("transactionId", "userKeyId");
   ```

2. **Use `ON CONFLICT DO NOTHING`** in `bulkInsertSigners` so that concurrent inserts are idempotent rather than error-prone:
   ```typescript
   await manager.createQueryBuilder()
     .insert().into(TransactionSigner).values(signersToInsert)
     .orIgnore()   // generates ON CONFLICT DO NOTHING
     .returning('*').execute();
   ```

3. **Move the signer read inside the database transaction** in `persistSignatureChanges` so the read and write share the same serializable snapshot, eliminating the TOCTOU window entirely.

## Proof of Concept

```typescript
// Attacker owns key K registered on transaction X
const payload = [{ id: X, signatureMap: signatureMapForKeyK }];

// Fire two concurrent requests
await Promise.all([
  fetch('/transactions/X/signers', { method: 'POST', body: JSON.stringify(payload) }),
  fetch('/transactions/X/signers', { method: 'POST', body: JSON.stringify(payload) }),
]);

// Both requests call loadTransactionData concurrently:
//   Request A reads: signersByTransaction.get(X) = Set{}  (key K absent)
//   Request B reads: signersByTransaction.get(X) = Set{}  (key K absent)
// Both pass the existingSignerIds.has(K) check → K is "new" in both
// Both call bulkInsertSigners → two rows (transactionId=X, userKeyId=K) inserted
// Result: duplicate TransactionSigner rows in the database
```

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
