Audit Report

## Title
Internal Infrastructure Services Exposed via Host Port Bindings in Production Docker Compose

## Summary
`back-end/docker-compose.prod.yaml` binds four internal-only infrastructure services — PostgreSQL, pgAdmin, Redis, and NATS — directly to host-level ports with no `127.0.0.1` loopback restriction. A secondary issue exists in `back-end/apps/api/src/setup-app.ts` where CORS is configured with `origin: true` and `credentials: true`, reflecting any `Origin` header as allowed.

## Finding Description

**Host port bindings confirmed in `back-end/docker-compose.prod.yaml`:**

- NATS: `4222:4222` [1](#0-0) 

- PostgreSQL: `5432:5432` [2](#0-1) 

- pgAdmin: `5050:80` [3](#0-2) 

- Redis: `6380:6379` [4](#0-3) 

None of these services have a `127.0.0.1:` prefix on the host-side binding, so Docker publishes them on all host interfaces via its `iptables` rules, bypassing host-level software firewalls (`ufw`, `firewalld`).

**CORS misconfiguration confirmed in `back-end/apps/api/src/setup-app.ts`:** [5](#0-4) 

`origin: true` causes Express to echo back whatever `Origin` header the client sends as the `Access-Control-Allow-Origin` response header. Combined with `credentials: true`, this allows any website to make credentialed cross-origin requests to the API.

**NATS authentication is absent.** The `nats-values.yaml` Kubernetes dev config has the entire `accounts` block (user/password auth) commented out: [6](#0-5) 

The production Docker Compose NATS service likewise has no `-auth` or `-config` flag in its command, confirming unauthenticated operation. [7](#0-6) 

**JWT blacklist is Redis-backed.** `BlacklistService` stores and checks revoked tokens exclusively in Redis: [8](#0-7) 

## Impact Explanation

| Service | Port | Impact |
|---|---|---|
| PostgreSQL | 5432 | Direct brute-force or credential-stuffing against the database; full read/write of all user data, transaction records, and key metadata |
| pgAdmin | 5050 | Web UI for the database; successful login yields full database control including schema modification |
| Redis | 6380 | JWT blacklist stored here; flushing Redis (`FLUSHALL`) invalidates the blacklist, allowing previously revoked tokens to be reused |
| NATS | 4222 | No authentication; attacker can publish to internal subjects, injecting fake transaction status updates or disrupting the scheduler |

The Redis exposure is the most critical: because `BlacklistService.isTokenBlacklisted` reads exclusively from Redis, an attacker who can reach port `6380` and issue `FLUSHALL` or `DEL <jwt>` effectively un-revokes any logged-out or administratively revoked token.

## Likelihood Explanation

The file is explicitly named `docker-compose.prod.yaml` and is the designated production deployment artifact. Any operator running this file on an internet-facing host without an independent cloud security group or network ACL exposes all four services. Docker's `iptables` rules are inserted at a lower priority than `ufw`/`firewalld`, so host-level software firewalls do **not** protect ports published by Docker. Likelihood is **High** for any deployment not fronted by a dedicated cloud security group or network ACL.

## Recommendation

1. **Restrict host port bindings to loopback** for all internal services. Change each internal port mapping to `127.0.0.1:<host_port>:<container_port>` in `docker-compose.prod.yaml`, or remove the `ports:` stanza entirely and rely on Docker's internal network for service-to-service communication.

2. **Enable NATS authentication.** Uncomment and configure the `accounts` block in the NATS config and pass a config file to the NATS container via `-c /etc/nats/nats.conf`.

3. **Fix CORS.** Replace `origin: true` with an explicit allowlist of trusted origins (e.g., `origin: ['https://app.example.com']`) in `back-end/apps/api/src/setup-app.ts`.

4. **Protect Redis.** At minimum, enable Redis `requirepass` authentication and bind Redis to the Docker internal network only.

## Proof of Concept

**Redis blacklist bypass (requires network access to port 6380):**
```bash
# 1. Obtain a valid JWT and log out (token is blacklisted in Redis)
# 2. From a host that can reach the production server on port 6380:
redis-cli -h <prod-host> -p 6380 FLUSHALL
# 3. Replay the previously revoked JWT — the API now accepts it because
#    BlacklistService.isTokenBlacklisted() returns false (key no longer exists)
```

**NATS unauthenticated publish (requires network access to port 4222):**
```bash
nats pub --server nats://<prod-host>:4222 "transaction.status" '{"id":"X","status":"EXECUTED"}'
# No credentials required; message is delivered to all internal subscribers
```

**CORS credential reflection:**
```javascript
// From any attacker-controlled origin:
fetch('https://<prod-api>:3001/auth/me', {
  credentials: 'include',
  headers: { 'Origin': 'https://evil.example.com' }
})
// Response includes: Access-Control-Allow-Origin: https://evil.example.com
//                    Access-Control-Allow-Credentials: true
```
