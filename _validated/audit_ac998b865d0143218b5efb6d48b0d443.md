Audit Report

## Title
Unbounded Observer and Approver Arrays Enable Resource-Exhaustion DoS

## Summary
A transaction creator can submit a single authenticated HTTP request containing an arbitrarily large `userIds` array (observers) or a deeply nested `approversArray` (approvers) with no enforced upper bound at the DTO, service, or database layer. This causes unbounded DB writes per request and, when combined with the `full=true` group fetch path, unbounded reads proportional to attacker-controlled data.

## Finding Description

**Observer path — missing `@ArrayMaxSize` in DTO:**

`CreateTransactionObserversDto` enforces only a minimum size (`@ArrayMinSize(1)`) with no maximum: [1](#0-0) 

The service iterates the entire unbounded array and bulk-saves all new entries in one call: [2](#0-1) 

There is no service-level length check before the loop or before `repo.save(observers)`.

**Approver path — unbounded recursive tree with no depth/node limit:**

`CreateTransactionApproverDto` has no `@ArrayMaxSize` on the nested `approvers` field, and `CreateTransactionApproversArrayDto` has no `@ArrayMaxSize` on `approversArray`: [3](#0-2) 

The inner `createApprover` function recurses into every element of `dtoApprover.approvers` with no depth limit or total-node cap. Each node triggers at least one `isNode` DB query and one `transactionalEntityManager.insert` call: [4](#0-3) 

**Expensive read path — `getTransactionGroup` with `full=true`:**

When `full=true` (the default in the controller), the service fetches all signers, approvers, and observers for every transaction in the group in parallel with no pagination or cap: [5](#0-4) 

The controller defaults `full` to `true`: [6](#0-5) 

## Impact Explanation
1. A single `POST /transactions/:id/observers` with `userIds: [1, 2, ..., 10000]` causes up to 10,000 `TransactionObserver` entity creations and a bulk `INSERT` in one request, plus one `emitTransactionUpdate` NATS emission.
2. A single `POST /transactions/:id/approvers` with a wide or deeply nested `approversArray` causes one DB query + one `INSERT` per node, all within a single DB transaction, with no cap on total nodes.
3. Repeated across many transactions, the `transaction_observer` and `transaction_approver` tables grow unboundedly, degrading query performance for all users.
4. Any subsequent `GET /transaction-groups/:id` (defaulting to `full=true`) must load all observers and approvers for every group item via `getTransactionObserversForTransactions` and `getTransactionApproversForTransactions`, making read cost proportional to attacker-controlled table size. [7](#0-6) 

## Likelihood Explanation
Any authenticated organization member who creates a transaction is its creator and can immediately call the observer/approver endpoints. The attack requires only a crafted JSON body — no special privilege, no race condition, no token accumulation. The endpoint is a standard REST POST reachable by any verified user.

## Recommendation
1. Add `@ArrayMaxSize(N)` (e.g., `N = 50`) to `userIds` in `CreateTransactionObserversDto`.
2. Add `@ArrayMaxSize(N)` to `approversArray` in `CreateTransactionApproversArrayDto` and to `approvers` in `CreateTransactionApproverDto`.
3. Add a service-level total-node counter in `createTransactionApprovers` that throws before recursing if the cumulative node count exceeds a configured limit.
4. Consider adding a database-level constraint (e.g., a partial index or trigger) limiting the number of observers per transaction.
5. For the group read path, consider paginating or capping the number of group items and/or the number of observers/approvers returned per transaction.

## Proof of Concept

```http
POST /transactions/1/observers HTTP/1.1
Authorization: Bearer <valid_creator_jwt>
Content-Type: application/json

{
  "userIds": [1, 2, 3, 4, ..., 10000]
}
```

This single request causes the server to:
1. Load the transaction with its existing observers.
2. Loop 10,000 times, calling `this.repo.create(...)` for each new userId.
3. Call `repo.save(observers)` with up to 10,000 entities.
4. Emit a NATS `emitTransactionUpdate` event.

For the approver tree variant:

```json
POST /transactions/1/approvers
{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        { "userId": 1 },
        { "userId": 2 },
        ...
        { "userId": 500 }
      ]
    }
  ]
}
```

This causes 501 sequential DB queries (`isNode` check) + 501 sequential `INSERT` statements inside a single DB transaction, all from one HTTP request.
