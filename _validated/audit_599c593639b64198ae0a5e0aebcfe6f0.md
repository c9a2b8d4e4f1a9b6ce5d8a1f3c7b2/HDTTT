Audit Report

## Title
Any Authenticated User Can Post Comments on Arbitrary Transactions Due to Missing Authorization Check

## Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` applies only authentication guards and performs no authorization check to verify the requesting user has any relationship to the target transaction. Any registered, verified user can inject comments into any transaction in the system.

## Finding Description
In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the `createComment` handler at line 21–29 is decorated with controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but carries no per-route authorization guard or service-layer ownership check: [1](#0-0) 

The developer-left TODO at line 22 explicitly acknowledges the missing control. The `CommentsService.createComment` method performs no relationship check either — it directly creates and persists the comment entity against the supplied `transactionId` with no validation: [2](#0-1) 

Contrast this with other state-mutating endpoints in the same codebase that enforce ownership:

- `GET /transactions/:id` delegates to `getTransactionWithVerifiedAccess`, which verifies the user is creator, signer, observer, or approver before returning data. [3](#0-2) 
- `DELETE /transactions/:transactionId/approvers/:id` calls `this.approversService.getCreatorsTransaction(transactionId, user)` before mutating state. [4](#0-3) 
- `getCreatorsTransaction` throws `UnauthorizedException` if the requesting user is not the transaction creator. [5](#0-4) 

No equivalent check exists anywhere in the comments flow.

## Impact Explanation
- **Unauthorized state mutation**: Any authenticated user can inject comments into transactions they have no legitimate access to, polluting the audit trail and comment history for all participants.
- **Integrity degradation**: Transaction comment threads — used by signers, approvers, and observers to coordinate — can be flooded or manipulated by any user in the organization, disrupting multi-party workflows.
- **Bypass of access control model**: The rest of the codebase enforces a strict relationship-based access model (creator / signer / observer / approver). The comments endpoint silently bypasses this model entirely.

## Likelihood Explanation
Exploitation requires only a valid user account obtainable via the standard signup flow. No privileged role, leaked credential, or special network position is needed. The attack is trivially scriptable against any known or enumerated transaction ID. The developer-left TODO confirms the gap was known but unaddressed.

## Recommendation
Add a service-layer check in `CommentsService.createComment` (or a dedicated guard) that verifies the requesting user has an established relationship to the target transaction before persisting the comment. The existing `getTransactionWithVerifiedAccess` or `getVerifiedApproversByTransactionId` patterns demonstrate the correct approach: load the transaction with its `creatorKey`, `observers`, `signers`, and `approvers` relations, then assert the user appears in at least one of those collections before proceeding. [6](#0-5) 

## Proof of Concept
1. Register as a normal user via `POST /auth/signup` (or use any existing organization account).
2. Obtain a valid JWT via `POST /auth/login`.
3. Issue the following request with any `transactionId` value belonging to a transaction the attacker has no relationship to:

```http
POST /transactions/42/comments
Authorization: Bearer <jwt>
Content-Type: application/json

{ "message": "Attacker-injected comment" }
```

4. The server responds with `201 Created` and the comment is persisted against transaction 42 with no ownership validation, as `CommentsService.createComment` directly calls `this.repo.save(comment)` without any prior relationship check.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
