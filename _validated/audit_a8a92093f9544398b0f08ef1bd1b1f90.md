Audit Report

## Title
Wrong Group Item Edited Due to String Character Indexing in `BaseGroupHandler.vue`

## Summary
In `BaseGroupHandler.vue`, the expression `route.params.seq[0]` is used to recover the sequence index of the group item being edited. Because Vue Router serialises route params as strings, `[0]` performs JavaScript **string character indexing** rather than array element access. For any transaction group with 10 or more items, this silently truncates the index to its first digit, causing `editGroupItem` in `storeTransactionGroup.ts` to overwrite the wrong transaction in the group.

## Finding Description

**Root cause — `BaseGroupHandler.vue` line 53–54:**

When a user clicks "Edit", `CreateTransactionGroup.vue` navigates to the edit route with the array `index` as the `seq` param:

```js
// CreateTransactionGroup.vue lines 130–134
router.push({
  name: 'createTransaction',
  params: { type, seq: index },   // index is a number, e.g. 10
  query: { groupIndex: index, group: 'true' },
});
``` [1](#0-0) 

Vue Router serialises `seq: 10` to the string `"10"` in `route.params.seq`.

Back in `BaseGroupHandler.vue`, the seq is recovered as:

```js
// BaseGroupHandler.vue lines 53–54
const seq =
  action === 'add' ? transactionGroup.groupItems.length.toString() : route.params.seq[0];
``` [2](#0-1) 

`route.params.seq` is the plain string `"10"`. Applying `[0]` to a JavaScript string returns the character at position 0, i.e. `"1"`, not the intended `"10"`. The developer likely intended to handle the `string[]` case from Vue Router's union type `string | string[]`, but single-value params are always serialised as a plain string, never as an array.

**Propagation — `storeTransactionGroup.ts` lines 116–117:**

```js
const editIndex = Number.parseInt(newGroupItem.seq);  // parseInt("1") = 1
if (!(editIndex >= 0 && editIndex < groupItems.value.length)) return;
``` [3](#0-2) 

The bounds guard does **not** catch this because `1` is a valid index in any group with 2+ items. The item at position `1` is then silently overwritten:

```js
groupItems.value = [
  ...groupItems.value.slice(0, editIndex),
  newGroupItem,
  ...groupItems.value.slice(editIndex + 1),
];
``` [4](#0-3) 

Truncation table for groups of 10–19 items:

| Intended index | `seq[0]` | Item actually overwritten |
|---|---|---|
| 10 | `"1"` | item 1 |
| 11 | `"1"` | item 1 |
| 15 | `"1"` | item 1 |
| 20 | `"2"` | item 2 |

## Impact Explanation
A user managing a transaction group with 10 or more items who edits any item at position ≥ 10 will silently overwrite a different, earlier item in the group. The intended item remains unchanged. If the group is then saved or submitted, the wrong Hedera transaction bytes are persisted or executed on-chain. Depending on the transaction type (e.g. token transfers, account updates), this can result in unintended fund movements or account state changes on the Hedera network with no warning to the user.

## Likelihood Explanation
The trigger condition — a group with 10+ items — is a normal workflow; no upper bound on group size is enforced anywhere in the codebase. Any user who builds a moderately large transaction group and attempts to edit a late-position item will hit this silently. No privileged access or adversarial intent is required; it is a pure correctness failure reachable through normal UI interaction.

## Recommendation
Replace the string character index access with a proper string coercion. In `BaseGroupHandler.vue` line 54, change:

```js
// Before (broken)
route.params.seq[0]

// After (correct)
Array.isArray(route.params.seq) ? route.params.seq[0] : route.params.seq
```

Or more simply, since Vue Router always serialises single params as strings:

```js
route.params.seq as string
```

This ensures the full numeric string (e.g. `"10"`) is passed to `editGroupItem` rather than just its first character.

## Proof of Concept

1. Open the application and navigate to **Create Transaction Group**.
2. Add 11 or more transactions to the group.
3. Click **Edit** on the transaction at position index `10` (the 11th item).
4. Modify any field (e.g. description or payer account) and confirm.
5. Observe: the transaction at index `1` (the 2nd item) has been overwritten with the new data. The transaction at index `10` is unchanged.

This is reproducible deterministically because `"10"[0] === "1"` in JavaScript, and `parseInt("1") === 1` is always a valid index in a group of 11+ items.
