Audit Report

## Title
Concurrent Signature Uploads Overwrite Each Other's Transaction Bytes, Causing Signature Loss

## Summary
`uploadSignatureMaps` in `SignersService` and `importSignatures` in `TransactionsService` both follow a read-modify-write pattern with no concurrency control. The read of `transactionBytes` occurs outside the database transaction that wraps the write. Two concurrent signers can each read the same stale bytes, independently add their own signature in memory, and then both write back — the second write silently discards the first signer's cryptographic signature while the `TransactionSigner` row for the first signer is still committed, creating a persistent inconsistency.

## Finding Description

**`uploadSignatureMaps` — `SignersService`**

`loadTransactionData` fetches `transactionBytes` for all requested transaction IDs using a plain `find` call that is entirely outside any database transaction: [1](#0-0) 

`processTransactionSignatures` then calls `sdkTransaction.addSignature(publicKey, map)` on the in-memory copy of those bytes: [2](#0-1) 

`persistSignatureChanges` opens a DB transaction only for the write phase. The read that produced the bytes being written back happened before this transaction started, so no isolation is provided against concurrent readers: [3](#0-2) 

`bulkUpdateTransactions` issues a plain `UPDATE … SET "transactionBytes" = CASE id … END WHERE id = ANY(…)` — no `WHERE "transactionBytes" = :expected` guard, no `SELECT FOR UPDATE`, no version column: [4](#0-3) 

**`importSignatures` — `TransactionsService`**

The identical pattern exists here: `entityManager.find` reads bytes outside any transaction, signatures are merged in memory, and a CASE-based `UPDATE` writes back with no concurrency guard: [5](#0-4) 

**Race window**

```
Signer A                              Signer B
──────────────────────────────────────────────────────────
loadTransactionData → bytes = B0
                                      loadTransactionData → bytes = B0
addSignature(SA) → B1 = B0 + SA
                                      addSignature(SB) → B2 = B0 + SB
UPDATE bytes = B1  ✓
                                      UPDATE bytes = B2  ← overwrites B1
                                                            SA is gone
```

The `signersByTransaction` deduplication only prevents inserting a duplicate `TransactionSigner` row for the same `userKeyId`; it does not prevent the bytes overwrite because the two signers have different keys and different `userKeyId` values: [6](#0-5) 

The endpoint that triggers this path requires only a valid JWT and signer membership — no elevated privileges: [7](#0-6) 

## Impact Explanation
- **Signature loss**: Signer A's cryptographic signature is silently dropped from `transactionBytes`. The `TransactionSigner` row for A is still committed (it is written in a separate `INSERT`), so the UI reports A as having signed, but the actual serialized transaction bytes no longer carry A's signature.
- **Persistent DB inconsistency**: The `TransactionSigner` table and `transactionBytes` diverge — the database records N signers as having signed, but the bytes contain only a subset of their signatures.
- **Permanent transaction failure**: Hedera transactions have a fixed valid-start window (default 180 s). If the lost signatures were needed to reach the required threshold and the window closes before the victim re-signs, the transaction expires and cannot be re-executed with the same transaction ID.
- **Malicious insider amplification**: A signer can deliberately loop `POST /transactions/:id/signers` with their own valid signature map to continuously overwrite the bytes and prevent the transaction from ever accumulating enough signatures before expiry.

## Likelihood Explanation
- Any two verified organization users who are both signers of the same transaction can trigger this by submitting signatures within the same database round-trip window (~50–200 ms).
- No special privileges are required beyond a valid JWT and membership as a signer.
- In organizations with many required signers (e.g., a 5-of-10 threshold key), concurrent signing is the normal workflow, making accidental collisions likely without any adversarial intent.
- The same vulnerability exists in both `uploadSignatureMaps` (`SignersService`) and `importSignatures` (`TransactionsService`), doubling the attack surface.

## Recommendation
Move the read inside the same database transaction as the write, and use `SELECT … FOR UPDATE` (or equivalent TypeORM locking) to serialize concurrent access to the same transaction row. For example, inside `persistSignatureChanges`, acquire a row-level lock before reading `transactionBytes`:

```sql
SELECT id, "transactionBytes" FROM transaction WHERE id = ANY($1) FOR UPDATE;
```

Then re-derive the updated bytes from the freshly locked row before issuing the `UPDATE`. Alternatively, add an optimistic-locking version column and include a `WHERE version = :expected` guard in the `UPDATE`, retrying on conflict.

## Proof of Concept

1. Create a Hedera transaction requiring signatures from two keys (Key A and Key B).
2. Signer A and Signer B both call `POST /transactions/:id/signers` simultaneously with their respective signature maps.
3. Both requests execute `loadTransactionData` and receive the same `transactionBytes` (B0) before either has written back.
4. Signer A's request computes B1 = B0 + SA and writes it.
5. Signer B's request computes B2 = B0 + SB and writes it, overwriting B1.
6. Query the `transaction_signer` table: both A and B appear as signers.
7. Deserialize `transactionBytes` from the `transaction` table: only SB is present; SA is absent.
8. If the threshold requires both signatures, the transaction will fail on submission to Hedera despite the UI showing it as fully signed.

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
