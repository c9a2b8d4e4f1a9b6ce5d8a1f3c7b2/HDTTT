Audit Report

## Title
Transaction Creator Can Modify Approver Structure After Approvals Are Given, Invalidating Multi-Signature Integrity

## Summary
The transaction creator can add, remove, or replace approvers and change approval thresholds at any time — including while the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status — because the sole authorization gate, `getCreatorsTransaction`, performs no status check. This allows a malicious creator to invalidate existing approvals, evict rejecting approvers, or lower the threshold to bypass the intended multi-signature quorum.

## Finding Description

**Root cause — `getCreatorsTransaction` has no status check:**

`getCreatorsTransaction` is the sole authorization gate for all three approver-mutation operations. It only verifies `transaction.creatorKey?.userId !== user.id` and never inspects `transaction.status`. Any transaction status — including `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION` — passes through. [1](#0-0) 

**All three mutation paths rely solely on this gate:**

1. **Create approvers** — `createTransactionApprovers` calls `getCreatorsTransaction` as its only guard before entering the DB transaction: [2](#0-1) 

2. **Update approver** — `updateTransactionApprover` calls `getCreatorsTransaction` inside the DB transaction with no status check: [3](#0-2) 

3. **Remove approver** — the controller calls `getCreatorsTransaction` then immediately removes, with no status check: [4](#0-3) 

**Approval state is silently wiped on `userId` update:**

When the creator replaces an approver's `userId`, the existing `signature`, `userKeyId`, and `approved` fields are explicitly cleared: [5](#0-4) 

**Threshold can be lowered at any time:** [6](#0-5) 

## Impact Explanation

The multi-signature approval workflow's integrity guarantee is broken. Concrete outcomes:

- **Approval erasure**: Creator replaces Approver A (who already approved) with a new user, clearing A's recorded `signature` and `approved` flag. The transaction reverts to needing fresh approvals.
- **Rejection bypass**: Approver B rejects the transaction. Creator removes B and adds a colluding user C, who then approves. The rejection is silently discarded.
- **Threshold manipulation**: Creator lowers the threshold from 3-of-5 to 1-of-5 after only one approval is collected, causing the transaction to immediately satisfy the approval requirement without the originally-agreed quorum.
- **DoS against execution**: Creator removes all approvers who have approved, then re-adds them, resetting the approval state and forcing re-approval indefinitely.

## Likelihood Explanation

- **Attacker profile**: Any authenticated user who has created a transaction. No admin role, no leaked credentials, no privileged keys required.
- **Entry point**: Standard REST API endpoints `POST /transactions/:id/approvers`, `PATCH /transactions/:id/approvers/:id`, `DELETE /transactions/:id/approvers/:id` — all reachable by any authenticated creator.
- **Trigger**: The creator simply calls these endpoints while the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. No race condition or timing dependency is required; the window is the entire active lifetime of the transaction.

## Recommendation

Add a status guard inside `getCreatorsTransaction` (or as a dedicated check at the top of each mutation method) that rejects modifications when the transaction has already progressed past `DRAFT`/`WAITING_FOR_SIGNATURES` (depending on the intended design). For example:

```typescript
const IMMUTABLE_STATUSES = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
  TransactionStatus.EXECUTED,
  TransactionStatus.CANCELED,
  TransactionStatus.EXPIRED,
  TransactionStatus.ARCHIVED,
];

if (IMMUTABLE_STATUSES.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify approvers after the transaction has been submitted');
}
```

This check should be placed in `getCreatorsTransaction` so it is enforced uniformly across all three mutation paths (`createTransactionApprovers`, `updateTransactionApprover`, and the `removeTransactionApprover` controller handler).

## Proof of Concept

1. Creator (user A) creates a transaction; it transitions to `WAITING_FOR_SIGNATURES` with approvers [B, C, D] and threshold 2.
2. Approver B calls `POST /transactions/:id/approvers/:id` (approve) — B's `signature` and `approved=true` are recorded.
3. Approver C calls the same endpoint and rejects (`approved=false`).
4. Creator A calls `PATCH /transactions/:id/approvers/:cId` with `{ "userId": <colluder_id> }` — C's record is replaced; `signature`, `userKeyId`, and `approved` are cleared.
5. Creator A calls `PATCH /transactions/:id/approvers/<threshold_node_id>` with `{ "threshold": 1 }` — threshold drops from 2 to 1.
6. The colluder approves. With threshold now 1 and one approval (B's), the transaction satisfies the approval requirement — the original 2-of-3 quorum is bypassed entirely.

All steps use standard authenticated API calls with no special privileges beyond being the transaction creator. [1](#0-0) [7](#0-6)

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
