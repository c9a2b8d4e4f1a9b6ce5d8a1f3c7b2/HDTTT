# Audit Report

## Title
Transaction Creator Can Execute Manually Without Satisfying Required Approvals

## Summary
The `executeTransaction` endpoint (`PATCH /transactions/execute/:id`) allows the transaction creator to trigger execution of a manual transaction without verifying that the configured approval threshold has been met. The approval system is completely decoupled from the execution path, allowing a creator to unilaterally execute any transaction they own regardless of the approval policy.

## Finding Description
`executeTransaction` in `back-end/apps/api/src/transactions/transactions.service.ts` (lines 736–751) enforces only two guards before proceeding to execution:

```typescript
async executeTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user); // guard 1: creator only
  if (!transaction.isManual) {                                        // guard 2: isManual flag
    throw new BadRequestException(ErrorCodes.IO);
  }
  if (transaction.validStart.getTime() > Date.now()) {
    await this.repo.update({ id }, { isManual: false });
    emitTransactionUpdate(...);
  } else {
    await this.executeService.executeTransaction(transaction);        // no approval check
  }
  return true;
}
```

`getTransactionForCreator` (lines 879–891) only verifies `creatorKey.userId === user.id` — it performs no status check and no approval check. [2](#0-1) 

The downstream `executeService.executeTransaction` calls `getValidatedSDKTransaction` → `validateTransactionStatus`, which only throws for terminal statuses (NEW, FAILED, EXECUTED, REJECTED, EXPIRED, CANCELED, ARCHIVED). It never queries the `TransactionApprover` table. [6](#0-5) 

The approval mechanism in `approvers.service.ts` (`approveTransaction`, lines 547–620) only writes `TransactionApprover.approved` and emits a notification. It never changes the transaction's execution eligibility and is never consulted by the execution path. [7](#0-6) 

## Impact Explanation
The approval mechanism is the primary organizational authorization control for multi-party transaction governance. Bypassing it allows a single authenticated user (the creator) to unilaterally execute any transaction they created, regardless of the approval policy configured (e.g., a 2-of-3 threshold). This directly violates the integrity of the multi-approval workflow and enables unauthorized state changes on the Hedera network — including unauthorized token transfers, account updates, or file operations — that were supposed to require organizational consensus.

## Likelihood Explanation
The attacker is a normal authenticated user with no special privileges beyond being the transaction creator. The attack requires only two API calls:
1. Create a transaction with `isManual: true` and configure required approvers.
2. Immediately call `PATCH /transactions/:id/execute`.

No race condition, cryptographic break, or privileged credential is needed. Any organization member who can create transactions can exploit this.

## Recommendation
Add an approval threshold check inside `executeTransaction` before proceeding to execution. Specifically:

1. Query all `TransactionApprover` records for the transaction via `approversService.getApproversByTransactionId`.
2. Evaluate the approval tree using the same threshold logic already present in the front-end utility `isApproved` (in `front-end/src/renderer/utils/sdk/index.ts`, lines 342–360) — or implement an equivalent server-side function.
3. If the threshold is not met, throw a `BadRequestException` before calling `executeService.executeTransaction`.

```typescript
async executeTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);
  if (!transaction.isManual) throw new BadRequestException(ErrorCodes.IO);

  // NEW: enforce approval threshold
  const approvers = await this.approversService.getApproversByTransactionId(id);
  if (approvers.length > 0 && !this.approversService.isApprovalThresholdMet(approvers)) {
    throw new BadRequestException('Approval threshold not met');
  }

  if (transaction.validStart.getTime() > Date.now()) {
    await this.repo.update({ id }, { isManual: false });
    emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
  } else {
    await this.executeService.executeTransaction(transaction);
  }
  return true;
}
```

## Proof of Concept
1. Authenticate as a normal organization member (User A).
2. `POST /transactions` with `isManual: true` — record the returned `transactionId`.
3. `POST /transactions/:transactionId/approvers` — add a 2-of-3 approval threshold with three other users as required approvers.
4. Without waiting for any approver to respond, call `PATCH /transactions/:transactionId/execute`.
5. Observe that `executeService.executeTransaction` is invoked and the transaction is submitted to the Hedera network with zero approvals collected, bypassing the configured 2-of-3 threshold entirely. [1](#0-0) [8](#0-7)
