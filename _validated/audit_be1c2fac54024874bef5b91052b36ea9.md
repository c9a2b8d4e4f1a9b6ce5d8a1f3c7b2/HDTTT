Audit Report

## Title
Unbounded In-Memory Iteration Over All Active Transactions in `getTransactionsToSign` Causes Authenticated DoS

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction system-wide from the database with no row limit, then performs an expensive per-row SDK deserialization and key-tree computation in application memory for each one. Pagination is applied only after the full scan. A single authenticated API call to `GET /transactions/sign`, after sufficient transaction accumulation, can exhaust Node.js heap memory or cause DB query timeouts, degrading or crashing the shared API service for all users.

## Finding Description

**Unbounded DB query — no `take` / LIMIT clause:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–299, `getTransactionsToSign` issues a `repo.find()` with no `take` option:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only excludes terminal statuses — no user scope, no row cap
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`) and does **not** scope the query to the requesting user's transactions at the database level: [2](#0-1) 

**In-memory iteration with expensive per-row work:**

Every returned row is then processed in application memory, calling `userKeysToSign` for each transaction: [3](#0-2) 

Pagination is applied only after the full scan completes: [4](#0-3) 

**`userKeysToSign` / `computeSignatureKey` is expensive per row:**

`computeSignatureKey` calls `SDKTransaction.fromBytes(transaction.transactionBytes)` (full SDK deserialization) and then performs multiple async account/node cache lookups per transaction: [5](#0-4) 

**Contrast with correctly paginated endpoints:**

`getTransactions` and `getHistoryTransactions` both pass `skip: offset, take: limit` to the ORM so the database enforces pagination before any data is loaded into memory: [6](#0-5) [7](#0-6) 

**The endpoint is live and authenticated-only:**

The controller exposes `GET /transactions/sign` protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — reachable by any verified user: [8](#0-7) [9](#0-8) 

## Impact Explanation
A single authenticated HTTP request to `GET /transactions/sign` causes the NestJS process to load every non-terminal transaction in the database into memory and deserialize each one via the Hedera SDK. As the number of active transactions grows, each call consumes proportionally more heap memory and CPU. At sufficient scale this causes request timeouts, OOM crashes, or sustained hangs that affect all concurrent users of the shared API service — a full service outage until the process is restarted and the transaction backlog is cleared.

## Likelihood Explanation
Any registered, verified user can reach this endpoint. Transaction creation via `POST /transactions` is unrestricted in volume per user. An attacker needs only a valid account and the ability to submit HTTP requests — no privileged access or leaked credentials are required. The degradation also occurs organically as legitimate usage grows, meaning the vulnerability can be triggered without malicious intent.

## Recommendation
1. **Push pagination to the database**: Add `take: limit` and `skip: offset` to the `repo.find()` call in `getTransactionsToSign`, mirroring the pattern used in `getTransactions` and `getHistoryTransactions`.
2. **Pre-filter at the DB level**: Join against `transaction_signer` or `user_key` tables to restrict the query to transactions where the requesting user's keys are relevant, eliminating the need to load and deserialize all active transactions.
3. **Add a hard cap**: Enforce a maximum page size (e.g., 100 rows) to bound worst-case memory usage regardless of client-supplied parameters.

## Proof of Concept

```
# 1. Authenticate as a normal user and obtain a JWT token.

# 2. Create N transactions with future validStart values so they remain
#    in WAITING_FOR_SIGNATURES status indefinitely:
for i in $(seq 1 10000); do
  curl -s -X POST https://<host>/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"transactionBytes":"...","creatorKeyId":1,...}'
done

# 3. Issue a single request to the vulnerable endpoint:
curl -v "https://<host>/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer $TOKEN"

# The server must now:
#   - SELECT * FROM transaction WHERE status NOT IN (...) -- no LIMIT
#   - Load all N rows into Node.js heap
#   - Call SDKTransaction.fromBytes() + computeSignatureKey() for each row
#   - Only then slice the first 10 results
#
# With N large enough, the process exhausts heap or the DB query times out,
# returning 500 / hanging for all concurrent users.
```

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
