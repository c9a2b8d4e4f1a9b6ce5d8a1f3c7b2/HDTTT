Audit Report

## Title
OTP (TOTP) Code Exposed in Plaintext via Custom Protocol Deep Link URL

## Summary
The Electron desktop application delivers the 8-digit TOTP code to the user via a custom protocol deep link of the form `hedera-transaction-tool://token=<TOTP_CODE>`. This URL is processed by the Electron main process and the TOTP code is forwarded to the renderer to auto-fill the OTP input. Because the sensitive code lives in a URL, it is subject to capture by OS-level URL scheme logs, browser history, and email client logs.

**Correction to the submitted report:** The value embedded in the deep link is the 8-digit TOTP code (generated by `totp.generate(secret)` and emailed as `otp`), **not** a JWT. The JWT (`token`) is returned from `POST /auth/reset-password` to the frontend via the API response and is never placed in the URL. This distinction changes the exploit flow but does not eliminate the vulnerability.

## Finding Description

**1. Deep link handler — `front-end/src/main/modules/deepLink.ts`**

The handler splits the URL on the protocol prefix and extracts whatever follows `token=`, then sends it to the renderer via IPC: [1](#0-0) 

**2. Protocol registration and `open-url` binding — `front-end/src/main/index.ts`**

The app registers itself as the default handler for `hedera-transaction-tool://` and fires `handleDeepLink` on every `open-url` event: [2](#0-1) [3](#0-2) 

**3. Preload bridge — `front-end/src/preload/localUser/deepLink.ts`**

The IPC message `deepLink:otp` is surfaced to the renderer via the preload bridge: [4](#0-3) 

**4. Renderer auto-fill — `front-end/src/renderer/components/ForgotPasswordModal.vue`**

On mount, the component subscribes to `onOTPReceived` and calls `setOTP` to populate the 8-digit input fields: [5](#0-4) 

**5. Backend OTP generation — `back-end/apps/api/src/auth/auth.service.ts`**

`createOtp()` generates the 8-digit TOTP code and emails it as `otp`; the JWT (`token`) is returned to the API caller separately and is never placed in the email or the deep link: [6](#0-5) 

**6. TOTP configuration**

`step=60, window=20` gives a ~20-minute validity window for any generated code: [7](#0-6) 

**7. Secondary observation — `LoggerMiddleware` logs `originalUrl` verbatim**

`maskedQuery` masks the parsed `query['token']` field, but `originalUrl` (which includes the raw query string) is logged unmasked. No current endpoint passes a token as a query parameter, so this is latent rather than immediately exploitable: [8](#0-7) 

## Impact Explanation

An attacker who captures the deep link URL (from OS URL-scheme logs, browser history, or email client storage) obtains the 8-digit TOTP code. The corrected exploit flow:

1. Victim requests a password reset; the backend emails a deep link containing the TOTP code.
2. Attacker captures the URL (e.g., from macOS Console, Windows Event Viewer, or browser history).
3. Attacker calls `POST /auth/reset-password` for the victim's email address, obtaining their own OTP JWT (`verified: false`).
4. Attacker submits their JWT + the captured TOTP code to `POST /auth/verify-reset`. Because `window=20` keeps the code valid for ~20 minutes, the check passes and a `verified: true` JWT is issued.
5. Attacker calls `PATCH /auth/set-password` with the verified JWT to overwrite the victim's password — full account takeover. [9](#0-8) 

Step 3 sends a second TOTP email to the victim, which may alert them, but the original TOTP code remains valid within the window.

## Likelihood Explanation

The attacker must be able to read OS-level URL scheme logs, browser history, or email client storage on the victim's machine, or have a process running locally that intercepts URL scheme activations. This requires either local access or malware already present. The TOTP validity window is approximately 20 minutes. Likelihood is low but realistic in shared-machine or malware scenarios.

## Recommendation

1. **Remove the TOTP code from the URL.** Instead of embedding the code in the deep link, the email should instruct the user to copy-paste the code manually. The deep link can carry only a non-sensitive session identifier that opens the app

    const user = await this.usersService.getUser({ email });

    if (!user) return;

    const secret = this.getOtpSecret(user.email);
    const otp = totp.generate(secret);

    emitUserPasswordResetEmail(this.notificationsPublisher, [{ email: user.email, additionalData: { otp } }]);

    const token = this.getOtpToken({ email: user.email, verified: false });
    return { token };
  }
