Audit Report

## Title
Authorization Bypass: `verifyAccess` Unconditionally Grants Access to All Terminal-Status Transactions, Enabling Cross-Tenant Data Exposure

## Summary
The `verifyAccess` function in `transactions.service.ts` short-circuits all ownership and relationship checks and returns `true` for any transaction in a terminal status (`EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, `ARCHIVED`). Because `GET /transactions/:id` delegates authorization entirely to this function, any authenticated user can retrieve full details of any completed transaction belonging to any other user or organization. The `GET /transactions/history` endpoint compounds this by returning all terminal-status transactions system-wide with no per-user filtering.

## Finding Description

**Root cause — `verifyAccess` bypasses all ownership checks for terminal states**

In `transactions.service.ts`, the `verifyAccess` function performs an early unconditional `return true` for any transaction whose status is in the terminal set, before any ownership predicate is evaluated: [1](#0-0) 

The ownership checks (`creatorKey.userId`, `observers`, `signers`, `approvers`) at lines 800–808 are never reached for terminal-status transactions.

**Call site — `GET /transactions/:id`**

`getTransactionWithVerifiedAccess` calls `verifyAccess` and throws only when it returns `false`. For every terminal-status transaction it always returns `true`, so the guard never fires: [2](#0-1) 

The controller wires this directly to `GET /:id`: [3](#0-2) 

**Compounding issue — `GET /transactions/history` has no per-user filter**

`getHistoryTransactions` builds a `FindManyOptions` with only a status filter and no `userId` predicate, returning all terminal-status transactions across all users: [4](#0-3) 

The controller exposes this endpoint with no user context passed to the service: [5](#0-4) 

**Controller — both endpoints are reachable by any verified JWT holder**

The controller class applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard session guards — with no ownership guard: [6](#0-5) 

## Impact Explanation

A malicious authenticated user can:

1. **Enumerate all completed transactions** via `GET /transactions/history` (paginated, no ownership check) and obtain transaction IDs, types, mirror networks, and group metadata for every transaction ever processed by the platform.
2. **Retrieve full transaction detail** for any enumerated ID via `GET /transactions/:id` (served as `TransactionFullDto`), obtaining:
   - Raw `transactionBytes` (the full signed Hedera transaction body)
   - `transactionHash`
   - `publicKeys` embedded in the transaction
   - Signer user-key associations (`signers[].userKey`)
   - Observer user identities
   - Creator key and linked user identity

This constitutes cross-tenant data exfiltration: organizational transaction history, key-to-user mappings, and signing patterns of unrelated users are fully exposed. Severity is **High** (confidentiality breach across all tenants).

## Likelihood Explanation

- **Attacker precondition**: a valid, non-admin account — obtainable by any user the admin has ever invited.
- **Effort**: a single authenticated HTTP `GET /transactions/history` returns the first page of all terminal transactions system-wide; subsequent pages enumerate the entire history. No brute-force of IDs is required.
- **Detection difficulty**: the requests are indistinguishable from legitimate history browsing.
- **Likelihood**: **High**.

## Recommendation

1. **Fix `verifyAccess`**: Remove the unconditional early `return true` for terminal statuses. Apply the same ownership checks (creator, observer, signer, approver) regardless of transaction status.
2. **Fix `getHistoryTransactions`**: Add a per-user filter (e.g., `creatorKey.userId`, `observers.userId`, `signers.userKey.userId`) so the history endpoint only returns transactions the requesting user is associated with, mirroring the pattern already used in `getTransactions`.

## Proof of Concept

```
# Step 1: Enumerate all terminal-status transactions (no ownership check)
GET /transactions/history?page=1&limit=50
Authorization: Bearer <any_valid_jwt>
# Returns all terminal transactions system-wide

# Step 2: Retrieve full details of a transaction belonging to another user
GET /transactions/42
Authorization: Bearer <any_valid_jwt>
# verifyAccess returns true unconditionally because status is EXECUTED
# Full TransactionFullDto returned including transactionBytes, publicKeys, signers, observers
```

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
