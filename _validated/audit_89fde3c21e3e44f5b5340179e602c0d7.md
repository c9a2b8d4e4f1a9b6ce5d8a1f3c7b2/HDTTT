Audit Report

## Title
Sequential/Atomic Transaction Group Execution Permanently Blocked When Any Member Transaction Changes State (TOCTOU)

## Summary
`executeTransactionGroup` in `execute.service.ts` validates every transaction in a group against the live database before executing any of them. If any single transaction's DB status has changed since the group was loaded into memory (e.g., canceled by its creator or expired by the background cron), the entire group throws and is silently abandoned — no other transaction in the group is ever executed. The group execution timeout is deleted and never rescheduled, permanently blocking all remaining valid transactions.

## Finding Description

**Vulnerability class**: State-transition / TOCTOU batch processing.

**Code path**:

**Step 1 — Scheduling.** `handleTransactionsBetweenNowAndAfterThreeMinutes` (cron, every 10 s) calls `prepareTransactions`, which loads the full `TransactionGroup` from the DB — including all `groupItem.transaction` objects with their current statuses — and calls `collateGroupAndExecute`. [1](#0-0) [2](#0-1) 

**Step 2 — Two-stage timeout registration.** `collateGroupAndExecute` registers a `setTimeout` that fires at `validStart − 10 s` to run `smartCollate`, then calls `addGroupExecutionTimeout`, which registers a second `setTimeout` that fires at `validStart + 5 s` and calls `executeService.executeTransactionGroup(transactionGroup)`. [3](#0-2) [4](#0-3) 

**Step 3 — Stale in-memory filter.** Inside `executeTransactionGroup`, group items are first filtered using the **stale in-memory** `transaction.status` field captured at scheduling time (T1): [5](#0-4) 

**Step 4 — DB re-read.** For every item that passed the in-memory filter, `getValidatedSDKTransaction` is called, which immediately re-reads the status **from the database** via `validateTransactionStatus`: [6](#0-5) 

**Step 5 — Group-level throw.** If the DB status is anything other than `WAITING_FOR_EXECUTION` (e.g., `CANCELED`, `EXPIRED`, `REJECTED`), `validateTransactionStatus` throws. That exception is caught and **re-thrown** as a group-level error, aborting the entire validation loop: [7](#0-6) [8](#0-7) 

**Step 6 — Silent abandonment.** The group-level error propagates to `addGroupExecutionTimeout`'s callback, where it is silently swallowed with `console.log`. The timeout is deleted in the `finally` block and is **never re-registered**. All remaining valid transactions in the group are permanently skipped. [9](#0-8) 

**TOCTOU window**: The group is loaded at cron time (T1). The execution callback fires at `validStart + 5 s` (T2). Any status change to any group member between T1 and T2 causes the entire group to fail.

**Natural trigger — expiry cron**: `handleExpiredTransactions` runs every 10 seconds and sets `WAITING_FOR_EXECUTION` transactions to `EXPIRED` when `validStart < now − 3 min`. If the scheduler is under load and the execution callback fires late (past the 3-minute window), the expiry cron can race ahead and expire one or more group members, triggering the group-level throw. [10](#0-9) 

**Adversarial trigger**: A user who is the creator of any one transaction in a sequential or atomic group can cancel that transaction in the window between group scheduling (T1) and execution (T2). The cancellation changes the DB status to `CANCELED`. When the execution callback fires, `validateTransactionStatus` detects the mismatch and throws, blocking all other transactions in the group from executing.

## Impact Explanation

All transactions in a sequential or atomic group are permanently abandoned when any single member changes state between scheduling and execution. The group execution timeout is deleted and never rescheduled. Affected transactions remain in `WAITING_FOR_EXECUTION` until the expiry cron eventually marks them `EXPIRED` (up to 3 minutes later). Users lose the ability to execute their transactions with no error surfaced to them — only a `console.log` on the server side.

**Severity: Medium** — permanent loss of transaction execution for all group members, triggered by a single member's state change. No funds are directly stolen, but the integrity of the transaction lifecycle is broken.

## Likelihood Explanation

- **Natural race**: The expiry cron (`handleExpiredTransactions`) runs every 10 seconds against `WAITING_FOR_EXECUTION` transactions. Under scheduler load or clock skew, the execution callback can fire after the 3-minute expiry window, causing a natural collision.
- **Adversarial**: Any legitimate user who is the creator of one transaction in a group can deliberately cancel it in the scheduling window to block the entire group. No privileged access is required — only a valid user account with a transaction in the group.

## Recommendation

1. **Per-transaction error handling in `executeTransactionGroup`**: Replace the group-level re-throw with per-item skip logic. If a single transaction fails validation (status changed), log the error and skip that item rather than aborting the entire group. For sequential groups, decide whether to halt-on-first-failure or skip-and-continue based on business requirements.

2. **Re-read status from DB before filtering**: Replace the stale in-memory filter at line 65–67 of `execute.service.ts` with a fresh DB query so the in-memory and DB checks are consistent.

3. **Avoid silent swallowing**: Replace the `console.log` in `addGroupExecutionTimeout`'s catch block with structured error logging and, where appropriate, emit a status-update notification to affected users.

4. **Idempotent timeout re-registration**: If a group-level error is unrecoverable, explicitly mark all remaining `WAITING_FOR_EXECUTION` transactions in the group as `FAILED` (with a descriptive status code) rather than leaving them to be expired silently by the cron.

## Proof of Concept

**Adversarial scenario (sequential group, 3 transactions):**

1. User A creates a sequential transaction group with transactions T1, T2, T3, all with `validStart = now + 2 min`. All three are signed and reach `WAITING_FOR_EXECUTION`.
2. The `handleTransactionsBetweenNowAndAfterThreeMinutes` cron fires and loads the group into memory. `collateGroupAndExecute` schedules the collate timeout at `validStart − 10 s` and the execution timeout at `validStart + 5 s`.
3. User A (creator of T2) calls the `cancelTransaction` API for T2. The DB status of T2 is now `CANCELED`.
4. At `validStart + 5 s`, `addGroupExecutionTimeout`'s callback fires and calls `executeTransactionGroup(transactionGroup)`.
5. The stale in-memory filter (line 65–67) passes T1, T2, T3 (all were `WAITING_FOR_EXECUTION` at scheduling time).
6. `getValidatedSDKTransaction(T1)` succeeds (T1 is still `WAITING_FOR_EXECUTION` in DB).
7. `getValidatedSDKTransaction(T2)` calls `validateTransactionStatus`, which reads `CANCELED` from DB and throws `"Transaction has been canceled."`.
8. The catch block at line 76–80 re-throws: `"Transaction Group cannot be submitted. Error validating transaction T2: Transaction has been canceled."`.
9. The error propagates to `addGroupExecutionTimeout`'s catch block (line 341), is logged with `console.log`, and the timeout is deleted (line 344).
10. T1 and T3 are never executed. They remain `WAITING_FOR_EXECUTION` until `handleExpiredTransactions` marks them `EXPIRED` up to 3 minutes later. No notification is sent to the users of T1 and T3. [11](#0-10) [9](#0-8)

  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },
            });
            // All the transactions for the group are now pulled. If there is an issue validating for even one
            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);
        }
      }
    }
  }

  addGroupExecutionTimeout(transactionGroup: TransactionGroup) {
    const name = `group_execution_timeout_${transactionGroup.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart =
      transactionGroup.groupItems[0].transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransactionGroup(transactionGroup);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }
