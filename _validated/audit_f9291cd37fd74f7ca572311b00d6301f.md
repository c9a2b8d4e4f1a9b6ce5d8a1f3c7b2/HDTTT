Audit Report

## Title
Silent Error Swallowing in Transaction Scheduler Callbacks Leaves Transactions Permanently Stuck

## Summary
In `TransactionSchedulerService`, async callbacks registered via `setTimeout` in `collateAndExecute` and `collateGroupAndExecute` catch all errors but only call `console.log(error)` without updating the transaction status or emitting a notification. When `computeSignatureKey` or `smartCollate` throws an unexpected error, the transaction is silently abandoned in `WAITING_FOR_EXECUTION` status — the execution timeout is never registered, the transaction is never submitted to Hedera, and users receive no meaningful notification of the failure. The same pattern exists in `addExecutionTimeout` and `addGroupExecutionTimeout`.

## Finding Description

In `collateAndExecute`, the callback fires 10 seconds before the transaction's `validStart`. It calls `computeSignatureKey` (which calls `SDKTransaction.fromBytes` and performs mirror node HTTP calls) and `smartCollate` (which also calls `SDKTransaction.fromBytes` and `_makeRequestAsync`). If either throws, the `catch` block at lines 319–321 only logs to stdout: [1](#0-0) 

Because `addExecutionTimeout` (line 318) is never reached, no execution timeout is ever registered. The transaction remains in `WAITING_FOR_EXECUTION` indefinitely.

The identical pattern exists in `collateGroupAndExecute`: [2](#0-1) 

And in `addExecutionTimeout` and `addGroupExecutionTimeout`, where unexpected errors from `executeService.executeTransaction` / `executeService.executeTransactionGroup` are also silently swallowed: [3](#0-2) [4](#0-3) 

The `collateAndExecute` and `collateGroupAndExecute` cases are the most severe: the `smartCollateFailed` path (lines 287–309 / 234–257) correctly updates status to `FAILED` and emits a notification, but this only handles the `null` return from `smartCollate`. Any thrown exception bypasses this path entirely.

The own test suite acknowledges this behavior without asserting any status update: [5](#0-4) [6](#0-5) 

## Impact Explanation

A transaction in `WAITING_FOR_EXECUTION` that silently fails the collation step is never submitted to the Hedera network. `handleExpiredTransactions`, which runs every 10 seconds, transitions `WAITING_FOR_EXECUTION` transactions past their valid start to `EXPIRED`: [7](#0-6) 

The result is:
- The transaction is silently dropped — never executed on-chain.
- The final status is `EXPIRED` rather than `FAILED`, masking the true cause.
- No failure notification is emitted for the actual error.
- For grouped/atomic transactions, all transactions in the group are silently abandoned.
- Operators have no visibility into the failure beyond a `console.log` that may be lost in log noise.

## Likelihood Explanation

`computeSignatureKey` calls `SDKTransaction.fromBytes(transaction.transactionBytes)` at line 42, which is not wrapped in a try-catch and will throw on corrupted or malformed bytes: [8](#0-7) 

`smartCollate` also calls `SDKTransaction.fromBytes` at line 291 and `_makeRequestAsync()` via `isTransactionOverMaxSize` at line 296, both of which can throw on unexpected input: [9](#0-8) 

Both failure modes — corrupted bytes in the database, network timeouts from mirror node calls, or SDK deserialization errors — are realistic in production environments, making this a moderate-to-high likelihood issue.

## Recommendation

In the `catch` blocks of `collateAndExecute` and `collateGroupAndExecute`, update the transaction status to `FAILED` and emit a notification, mirroring the existing `smartCollateFailed` handling path. For example, in `collateAndExecute`:

```typescript
} catch (error) {
  console.log(error);
  // Update status to FAILED and notify
  const result = await this.transactionRepo
    .createQueryBuilder()
    .update(Transaction)
    .set({ status: TransactionStatus.FAILED, executedAt: new Date() })
    .where('id = :id AND status = :currentStatus', {
      id: transaction.id,
      currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
    })
    .returning('id')
    .execute();
  if (result.raw.length > 0) {
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      result.raw.map(row => ({ entityId: row.id })),
    );
  }
}
```

Apply the equivalent fix to `collateGroupAndExecute` (updating all group transaction IDs). For `addExecutionTimeout` and `addGroupExecutionTimeout`, consider re-throwing or logging with a structured logger (`Logger`) so the error is observable in monitoring systems, since by that point the transaction has already been submitted.

## Proof of Concept

1. Create a transaction with `WAITING_FOR_EXECUTION` status and a `validStart` more than 10 seconds in the future.
2. Ensure `computeSignatureKey` will throw — e.g., by storing corrupted `transactionBytes` in the database (causing `SDKTransaction.fromBytes` to throw at `transaction-signature.service.ts:42`).
3. Wait for `collateAndExecute` to fire its `setTimeout` callback (10 seconds before `validStart`).
4. Observe: the `catch` block logs the error to stdout; `addExecutionTimeout` is never called; the transaction remains in `WAITING_FOR_EXECUTION`.
5. Wait for `handleExpiredTransactions` to run (every 10 seconds, at `transaction-scheduler.service.ts:100`).
6. Observe: the transaction is transitioned to `EXPIRED` — not `FAILED` — with no indication of the collation error that caused it.

export async function smartCollate(
  transaction: Transaction,
  requiredKeys: KeyList,
): Promise<SDKTransaction | null> {
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // HIP-1300: isTransactionOverMaxSize is fee-payer aware. Privileged payers
  // (0.0.2, 0.0.42-0.0.799) can carry up to 128 KB, so transactions under that
  // limit pass through here untouched and signatures are NOT stripped.
  if (await isTransactionOverMaxSize(sdkTransaction)) {
    const publicKeys = computeShortenedPublicKeyList(
      sdkTransaction._signerPublicKeys,
      requiredKeys,
    );

    const signatureMap = sdkTransaction.getSignatures();
    sdkTransaction.removeAllSignatures();

    for (const key of publicKeys) {
      sdkTransaction.addSignature(key, signatureMap);
    }

    // If the transaction is still too large,
    // set it to failed with the TRANSACTION_OVERSIZE status code
    // update the transaction, emit the event, and delete the timeout
    if (await isTransactionOverMaxSize(sdkTransaction)) {
      return null;
    }
  }

  return sdkTransaction;
}
