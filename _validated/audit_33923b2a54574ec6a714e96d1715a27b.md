Audit Report

## Title
Wrong NATS Event Emitted in `approveTransaction`: Approval Always Fires `emitTransactionUpdate` Instead of `emitTransactionStatusUpdate`

## Summary
In `approveTransaction` (`back-end/apps/api/src/transactions/approvers/approvers.service.ts`), the post-write branching condition that selects which NATS event to publish evaluates a stale in-memory array that is structurally guaranteed to contain at least one `approved: null` entry at the point of evaluation. As a result, every successful approval emits `emitTransactionUpdate` (lightweight UI-refresh ping) instead of `emitTransactionStatusUpdate` (full notification pipeline). The rejection path is unaffected.

## Finding Description

**Step 1 — Fetch and filter**

`userApprovers` is populated from the DB and filtered to the current user's records. [1](#0-0) 

**Step 2 — Double-submission guard**

The function throws if every record already has a `signature`. Passing this guard guarantees at least one record in `userApprovers` has `signature: null` (and therefore `approved: null`, since both fields are always written together). [2](#0-1) 

**Step 3 — DB write without in-memory refresh**

All `userApprovers` records are updated in the DB with `approved: dto.approved`. The in-memory `userApprovers` array is **never re-fetched** after this write. [3](#0-2) 

**Step 4 — Stale branch condition**

```typescript
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
``` [4](#0-3) 

Because the in-memory array still reflects the pre-write state, `userApprovers.every(a => a.approved)` evaluates against stale `null` values and is always `false` on the approval path:

| `dto.approved` | `!dto.approved` | `every(a=>a.approved)` (stale) | Condition | Event emitted |
|---|---|---|---|---|
| `true` | `false` | always `false` | `false` | `emitTransactionUpdate` ← **wrong** |
| `false` | `true` | always `false` | `true` | `emitTransactionStatusUpdate` ← correct |

**The two NATS subjects route to entirely different handlers:**

`emitTransactionUpdate` publishes to `TRANSACTION_UPDATE` → `processTransactionUpdateNotifications`: [5](#0-4) [6](#0-5) 

`processTransactionUpdateNotifications` only computes affected user IDs and sends a WebSocket ping (`TRANSACTION_EVENT_TYPE.UPDATE`). It does **not** delete old in-app indicators, create new ones, or send emails. [7](#0-6) 

`emitTransactionStatusUpdate` publishes to `TRANSACTION_STATUS_UPDATE` → `processTransactionStatusUpdateNotifications`: [8](#0-7) 

`processTransactionStatusUpdateNotifications` runs the full pipeline: deletes stale in-app indicators, creates new ones, sends email notifications, and sends the WebSocket ping. [9](#0-8) 

## Impact Explanation

Every time any approver submits `dto.approved = true`:
- In-app notification indicators (`TRANSACTION_INDICATOR_SIGN`, `TRANSACTION_INDICATOR_EXECUTABLE`, etc.) are never updated — stale indicators persist and new ones are never created.
- Email notifications (`TRANSACTION_WAITING_FOR_SIGNATURES`, `TRANSACTION_READY_FOR_EXECUTION`, etc.) are never dispatched.
- Signers, observers, and the creator receive only a silent UI-refresh WebSocket ping, not the actionable notification they depend on to know the transaction is ready for the next step.

The rejection path correctly fires `emitTransactionStatusUpdate`, creating a direct asymmetry: rejections are fully notified while approvals are silently under-notified.

## Likelihood Explanation

This fires on **every** approval call without exception. No special conditions, race conditions, or attacker-controlled inputs are required. Any authenticated approver submitting `dto.approved = true` through the normal API endpoint deterministically triggers the wrong event. The bug is 100% reproducible.

## Recommendation

After the DB write, re-fetch the updated `userApprovers` from the database before evaluating the branch condition, or replace the stale in-memory check with a direct comparison against `dto.approved` combined with a fresh DB query to determine whether all approvers have now approved. The simplest correct fix is:

```typescript
// After the DB write, always emit the full status update on approval
if (!dto.approved || /* fresh DB check: all userApprovers now approved */) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

Alternatively, since any approval or rejection is a status-relevant event, always emit `emitTransactionStatusUpdate` unconditionally after a successful write, removing the stale-data branch entirely.

## Proof of Concept

1. Set up a transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status with at least one approver.
2. As the approver, call the approval endpoint with `dto.approved = true`.
3. Observe that `emitTransactionUpdate` is called (publishes to `TRANSACTION_UPDATE`).
4. Observe that `processTransactionUpdateNotifications` runs — only a WebSocket ping is sent.
5. Confirm that `processTransactionStatusUpdateNotifications` is **not** called — no in-app indicators are updated, no emails are sent.
6. Repeat with `dto.approved = false` (rejection) and observe that `emitTransactionStatusUpdate` is correctly called and the full notification pipeline runs.

The existing test at line 1278 of `approvers.service.spec.ts` appears to assert `emitTransactionStatusUpdate` was called, but it passes only because the test calls `approveTransaction` twice in sequence — once with `approved: true` and once with `approved: false` — and the assertion is satisfied by the second (rejection) call, masking the bug on the first (approval) call.     const ctx = await this.prepareEventContext(events, true);
    if (!ctx) return;

    const {
      cache,
      keyCache,
      transactionMap,
      approversMap,
      deletionNotifications,
      inAppNotifications,
      emailNotifications,
      inAppReceiverIds,
      emailReceiverIds,
      affectedUsers,
    } = ctx;

    // Process each event
    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping status-update notifications`);
        continue;
      }
      const approvers = approversMap.get(transactionId) || [];

      if (transaction.deletedAt && transaction.status !== TransactionStatus.CANCELED) {
        console.error(
          `Soft-deleted transaction ${transactionId} has unexpected status: ${transaction.status} (expected CANCELED)`
        );
        transaction.status = TransactionStatus.CANCELED;
      }

      const syncType = this.getInAppNotificationType(transaction.status);
      const emailType = this.getEmailNotificationType(transaction.status);

      // Single transaction for both notification types
      await this.entityManager.transaction(async entityManager => {
        await this.handleTransactionStatusUpdateNotifications(
          entityManager,
          transaction,
          approvers,
          syncType,
          emailType,
          cache,
          keyCache,
          deletionNotifications,
          inAppNotifications,
          inAppReceiverIds,
          emailNotifications,
          emailReceiverIds,
          affectedUsers,
          transactionId,
        );
      });
    }

    // Send all notifications in batch
    await this.sendDeletionNotifications(deletionNotifications);
    await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
    await this.sendEmailNotifications(emailNotifications, emailReceiverIds);
    await this.sendNotifyClients(affectedUsers, TRANSACTION_EVENT_TYPE.STATUS_UPDATE);
  }
