Audit Report

## Title
`mnemonicHash` and `index` of All Users' Keys Exposed to Any Authenticated User via `GET /user-keys`

## Summary
The `GET /user-keys` endpoint, served by `UserKeysAllController`, returns every user's `mnemonicHash` and `index` fields to any verified, authenticated user. A parallel endpoint (`GET /user/:userId/keys`) correctly suppresses these fields for non-owners, but the all-keys path applies no equivalent restriction, creating a complete bypass of the intended field-level access control.

## Finding Description

**Root cause — two service methods with asymmetric field filtering:**

`getUserKeysRestricted` (used by `GET /user/:userId/keys`) conditionally selects `mnemonicHash` and `index` only when the requesting user is the key owner: [1](#0-0) 

`getUserKeys` (used by `GET /user-keys`) performs an unrestricted `findAndCount` with no `select` clause, returning all columns for all users: [2](#0-1) 

**`UserKeyDto` unconditionally exposes both sensitive fields:** [3](#0-2) 

**`UserKeysAllController` carries no admin guard** — only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`, meaning any verified user can reach the endpoint: [4](#0-3) 

**Developer intent is explicit in the e2e test suite**, which asserts that `mnemonicHash` and `index` must not appear when a user fetches another user's keys via the restricted path: [5](#0-4) 

No equivalent assertion exists for `GET /user-keys`.

## Impact Explanation

`mnemonicHash` is the argon2 hash of a user's 24-word BIP39 recovery phrase stored in the `user_key` table. Exposure enables:

- **Recovery phrase confirmation**: An attacker who obtains a candidate mnemonic (via phishing or social engineering) can verify it belongs to a specific user by computing its argon2 hash offline and comparing it to the leaked value — without touching the backend's authentication flow.
- **Key correlation**: Keys sharing the same `mnemonicHash` reveal that they were derived from the same recovery phrase, exposing the full key derivation structure across all users.
- **Derivation index disclosure**: The `index` field reveals the exact BIP32 derivation index, reducing the search space in any offline brute-force attack against the recovery phrase.

## Likelihood Explanation

- **Precondition**: A valid, verified account — achievable through the standard product registration flow.
- **No elevated privilege required**: The endpoint has no admin guard.
- **Trivially exploitable**: A single authenticated HTTP `GET /user-keys` request with a valid JWT is sufficient to retrieve paginated `mnemonicHash` and `index` values for every user in the system.

## Recommendation

Apply the same field-level restriction in `getUserKeys` that exists in `getUserKeysRestricted`. Specifically, the `findAndCount` call in `getUserKeys` should use a `select` clause that excludes `mnemonicHash` and `index` entirely (since the caller identity is not scoped to a single user in this path). Alternatively, remove `@Expose()` from `mnemonicHash` and `index` in `UserKeyDto` and introduce a separate owner-only DTO used exclusively by the self-key path.

Add an e2e test for `GET /user-keys` asserting that `mnemonicHash` and `index` are absent from all returned items, mirroring the existing assertion at: [6](#0-5) 

## Proof of Concept

```
# 1. Register and verify an attacker account (standard product flow)
# 2. Obtain a JWT via login

GET /user-keys
Authorization: Bearer <attacker_jwt>

# Response (paginated):
{
  "items": [
    {
      "id": 1,
      "userId": 1,
      "publicKey": "...",
      "mnemonicHash": "$argon2id$...",   # victim's mnemonic hash
      "index": 0                          # victim's BIP32 derivation index
    },
    ...
  ],
  "totalItems": N,
  "page": 1,
  "size": 10
}
```

The attacker receives `mnemonicHash` and `index` for every user in the system from a single unelevated request, because `getUserKeys` applies no field-level filtering: [7](#0-6)
