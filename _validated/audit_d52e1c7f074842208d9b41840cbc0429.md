Audit Report

## Title
Transaction Creator Can Delete Approvers Belonging to Transactions They Do Not Own (Cross-Transaction IDOR in `removeTransactionApprover`)

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller owns the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without confirming that approver belongs to that transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause — missing cross-transaction ownership check in the delete path**

In `approvers.controller.ts`, the delete handler calls two service methods sequentially:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId only
await this.approversService.removeTransactionApprover(id);               // deletes approver `id` — no ownership check
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the user is the creator of the transaction identified by `transactionId`:

```typescript
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
``` [2](#0-1) 

`removeTransactionApprover` then fetches the approver by its own `id` and deletes it with no check that it belongs to the verified transaction:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [3](#0-2) 

**The fix already exists in the update path but is absent from the delete path.** `updateTransactionApprover` performs the missing ownership check:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [4](#0-3) 

The delete path omits this guard entirely.

**`removeNode` performs a recursive soft-delete** via a SQL CTE, meaning the entire subtree rooted at the targeted approver is deleted in one operation: [5](#0-4) 

## Impact Explanation
An attacker can silently delete any approver — including entire approval subtrees — from any transaction they do not own. Because `removeNode` is a recursive soft-delete, a single request can wipe the full subtree rooted at the targeted approver. This permanently disrupts the multi-signature approval workflow: required approvers disappear, threshold conditions can no longer be met, and the victim transaction may become permanently stuck or require the creator to rebuild the entire approver tree from scratch.

## Likelihood Explanation
- **Precondition**: attacker must only be a registered, verified user — no admin or special role required.
- **Approver ID discovery**: approver IDs are sequential integers. Any user who is an observer, signer, or approver of the victim transaction can retrieve them directly via `GET /transactions/:id/approvers` (`getVerifiedApproversByTransactionId`). [6](#0-5) 
- **No rate-limit barrier**: the delete endpoint is a standard authenticated REST call with no additional throttling.
- **Realistic scenario**: a malicious participant added as an observer to a high-value multi-sig transaction can enumerate approver IDs and destroy the entire approval tree before execution.

## Recommendation
In `removeTransactionApprover` (service) or in the controller's delete handler, add the same cross-transaction ownership check that already exists in `updateTransactionApprover`:

1. After fetching the approver by `id`, resolve its root node via `getRootNodeFromNode`.
2. Assert `rootNode.transactionId === transactionId`. If not, throw `UnauthorizedException`.

This mirrors the existing guard at line 390–391 of `approvers.service.ts` and closes the decoupling between the authorization target (`transactionId`) and the mutation target (`id`). [7](#0-6) 

## Proof of Concept

1. Attacker registers as a normal user (no admin required).
2. Attacker creates their own transaction → receives `transactionId_A`.
3. Attacker is added as an observer or signer to victim transaction `transactionId_B`, or enumerates sequential approver IDs. They call `GET /transactions/{transactionId_B}/approvers` and obtain `approver_id_B`.
4. Attacker sends:
   ```
   DELETE /transactions/{transactionId_A}/approvers/{approver_id_B}
   ```
5. `getCreatorsTransaction(transactionId_A, user)` passes — attacker owns transaction A. [8](#0-7) 
6. `removeTransactionApprover(approver_id_B)` executes — approver from transaction B (and its entire subtree) is recursively soft-deleted with no further authorization check. [3](#0-2)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
