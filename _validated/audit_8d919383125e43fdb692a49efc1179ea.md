# Audit Report

## Title
`removeTransactionApprover` Does Not Update Parent Threshold, Causing Permanent Approval DoS

## Summary
`removeTransactionApprover` in `ApproversService` delegates to `removeNode`, which soft-deletes a child approver and its descendants via a recursive SQL query but never adjusts the parent tree node's `threshold` column. When a child is removed from a threshold group, the parent's `threshold` can exceed the remaining live child count, making the approval requirement mathematically unsatisfiable. The transaction becomes permanently stuck unless the creator manually lowers the threshold — a remediation step that is not surfaced by the API.

## Finding Description

**Root cause — `removeTransactionApprover` calls `removeNode` with no threshold fixup:** [1](#0-0) 

**`removeNode` performs a recursive soft-delete via raw SQL and never touches the parent's `threshold` column:** [2](#0-1) 

**By contrast, `updateTransactionApprover` (the `dto.listId = null` branch) correctly adjusts the parent threshold when a child is detached — it either soft-deletes the parent if no children remain, or clamps the threshold to the new child count:** [3](#0-2) 

`removeTransactionApprover` has no equivalent adjustment, leaving the parent's `threshold` stale after deletion.

**The controller only verifies the caller is the transaction creator before invoking `removeTransactionApprover`:** [4](#0-3) 

**Additional secondary bug**: `removeTransactionApprover` emits `emitTransactionStatusUpdate` using `approver.transactionId`. For a child node (non-root), `transactionId` is `null` (only root nodes carry `transactionId`), so the status-update event fires with `entityId: null`. [5](#0-4) 

## Impact Explanation

After the deletion, the parent node's `threshold` exceeds the count of live child approvers. The only remaining approver(s) can submit their approval, but the threshold can never be reached. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. A manual recovery path exists — the creator can call `PATCH /transactions/{id}/approvers/{parentId}` with `{ threshold: N }` — but this is not surfaced by the API, not documented as a required follow-up step, and the `updateTransactionApprover` threshold branch validates against the live DB count (which is correct post-deletion), so the fix is technically possible but entirely non-obvious. [6](#0-5) 

## Likelihood Explanation

The trigger is a normal, documented product flow: a transaction creator removes one approver from a threshold group. No special privilege beyond being the authenticated transaction creator is required. The creator can trigger this accidentally (e.g., removing the wrong approver from a 2-of-2 group) or intentionally to sabotage their own transaction. The `DELETE /:id` endpoint is publicly documented and requires no elevated role. [7](#0-6) 

## Recommendation

In `removeTransactionApprover`, after calling `removeNode`, load the parent of the deleted node (if any) with its live children and apply the same threshold-clamping logic already present in `updateTransactionApprover`:

- If the parent has zero remaining live children, soft-delete the parent.
- If the parent's `threshold` exceeds the remaining live child count, update `threshold` to `remainingCount`.

This mirrors the existing logic at lines 417–428 of `approvers.service.ts` and closes the inconsistency. [3](#0-2) 

## Proof of Concept

1. Creator creates a transaction and adds a threshold approver tree:
   ```
   POST /transactions/1/approvers
   { approversArray: [{ threshold: 2, approvers: [{ userId: A }, { userId: B }] }] }
   ```
   DB state: parent node `threshold=2`, live children = [A, B].

2. Creator deletes child A:
   ```
   DELETE /transactions/1/approvers/{A.id}
   ```
   `removeNode(A.id)` soft-deletes A. Parent's `threshold` remains `2`.

3. DB state: parent node `threshold=2`, live children = [B] (count = 1).

4. User B approves. The system evaluates the tree: 1 approval received, threshold requires 2. Condition is never satisfied. The transaction is permanently stuck.

5. The `isApproved` utility (front-end) confirms this: `approvals.length >= (approver.threshold || approvals.length)` — with `approvals.length = 1` and `threshold = 2`, the condition `1 >= 2` is false and can never become true. [8](#0-7)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }
