Audit Report

## Title
`getTransactionsToSign` Unbounded DB Query and Sequential Per-Transaction Async Loop Enables Single-User Resource Exhaustion DoS

## Summary
`GET /transactions/sign` is a live, JWT-authenticated endpoint that internally calls `getTransactionsToSign`. The service method issues an unbounded `repo.find()` loading every non-terminal transaction in the database into the Node.js heap, then serially awaits one `userKeysToSign` call per row before applying pagination. A single authenticated request against a large transaction table can exhaust heap memory, saturate the DB connection pool, and block the event loop, causing service degradation or crash for all concurrent users.

## Finding Description

**Unbounded DB query — no `take` limit:**

`getTransactionsToSign` calls `repo.find()` with no `take` field:

```ts
// transactions.service.ts:295-299
const transactions = await this.repo.find({
  where: whereForUser,   // only filters terminal statuses — no user scope
  relations: ['groupItem'],
  order,
  // ← no take
});
``` [1](#0-0) 

The `whereForUser` condition excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`) and applies no user-scoped filter, so the query returns **all** non-terminal transactions system-wide. [2](#0-1) 

Compare with `getTransactions` and `getHistoryTransactions`, which both pass `take: limit` in their find options: [3](#0-2) [4](#0-3) 

**Sequential per-transaction async loop:**

Every loaded transaction triggers a sequential `await userKeysToSign(...)` call, which internally calls `transactionSignatureService.computeSignatureKey` (a DB/network operation) and `entityManager.find` for `UserKey` records:

```ts
// transactions.service.ts:301-309
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [5](#0-4) 

`userKeysRequiredToSign` (the underlying utility) calls `attachKeys` and `keysRequiredToSign`, each issuing additional DB queries per transaction: [6](#0-5) 

**Pagination applied only after full in-memory scan:**

`limit`/`offset` from `PaginationParams` are used only to slice the already-built result array: [7](#0-6) 

`PaginationParams` enforces `size <= 100`, but this only limits the final slice — it does not bound the DB query or the loop: [8](#0-7) 

**Live, minimally-guarded endpoint:**

The route is active and requires only a valid JWT session (no admin/role guard): [9](#0-8) 

## Impact Explanation
- **Heap exhaustion**: All non-terminal transactions with their `groupItem` relations are loaded into the Node.js process heap at once. With thousands of active transactions, this can trigger OOM and crash the API process.
- **DB connection starvation**: The sequential `await` loop holds or re-acquires DB connections for the entire duration of the request (one per transaction), starving the connection pool for all concurrent requests.
- **Event-loop blocking / HTTP timeouts**: A single long-running request can delay responses for all other users until the Node.js event loop is freed.
- **Severity**: High — a single authenticated request can render the API unavailable for all users.

## Likelihood Explanation
- Any registered, authenticated user can call `GET /transactions/sign?page=1&size=1`.
- No elevated privilege or admin role is required.
- In organizations running bulk transaction workflows, the condition is naturally met without attacker-specific setup.
- An attacker can also deliberately inflate the `transactions` table via `POST /transactions` before triggering the endpoint, amplifying the effect.

## Recommendation
1. **Apply `take: limit` to the `repo.find()` call** inside `getTransactionsToSign`, mirroring the pattern used in `getTransactions` and `getHistoryTransactions`.
2. **Restructure the query** to push the user-key filtering into the database (e.g., a JOIN or subquery on `transaction_signer`/`user_key`) rather than performing it in application code, eliminating the N+1 loop entirely.
3. **Add a user-scoped `where` clause** so the query only fetches transactions relevant to the requesting user, not all non-terminal transactions system-wide.
4. **Consider removing or rate-limiting the endpoint** given the comment "NO LONGER USED BY FRONT-END" — if it is truly unused, it should be disabled.

## Proof of Concept
```
# Authenticated request — triggers full table scan + N sequential DB calls
GET /transactions/sign?page=1&size=1
Authorization: Bearer <valid_jwt>
```

With N non-terminal transactions in the database:
1. `repo.find()` loads all N rows (with `groupItem` relations) into heap.
2. The `for...of` loop issues N sequential `userKeysToSign` calls, each performing `computeSignatureKey` + `entityManager.find(UserKey, ...)`.
3. After all N iterations complete, `result.slice(0, 1)` returns 1 item to the caller.
4. Total DB queries: `1 (find) + N×2 (per-transaction)`. Total heap: proportional to N × (transaction + groupItem size).
5. Repeating the request concurrently multiplies the load linearly.

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
