Audit Report

## Title
Cross-Transaction Approver Deletion via Mismatched Authorization Key

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller owns `transactionId`, but then deletes the approver record identified by `id` without verifying that approver belongs to `transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other transaction by supplying their own `transactionId` for the auth check and a victim's approver `id` for the deletion.

## Finding Description

**Root cause — controller:**

The `removeTransactionApprover` handler in `approvers.controller.ts` performs authorization and deletion as two independent, uncorrelated operations: [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` confirms the caller owns `transactionId`. The subsequent `removeTransactionApprover(id)` then operates on a completely independent approver record identified only by `id` — no cross-check is performed.

**Root cause — service:**

`removeTransactionApprover` in `approvers.service.ts` fetches the approver by `id` and immediately deletes it without ever comparing `approver.transactionId` against the route's `transactionId`: [2](#0-1) 

**Contrast with `updateTransactionApprover`:**

The sibling `updateTransactionApprover` method correctly performs this cross-check — it fetches the root node and explicitly validates `rootNode.transactionId !== transactionId` before proceeding: [3](#0-2) 

The fix was applied to `update` but not to `remove`, making the omission in `removeTransactionApprover` a clear inconsistency.

**Recursive deletion scope:**

`removeNode` uses a recursive CTE to soft-delete the entire approver subtree rooted at the given `id`: [4](#0-3) 

A single DELETE request targeting a root approver node wipes the entire approval tree of the victim's transaction.

## Impact Explanation

- An attacker can silently remove required approvers from any transaction they do not own, bypassing the multi-signature approval workflow.
- Because `removeNode` is recursive, a single call targeting a root approver node deletes the entire approval tree, not just a leaf.
- A transaction that previously required N-of-M approvals can be reduced to zero required approvals, allowing it to proceed to execution without any authorization.
- This is a critical integrity failure in the transaction approval trust model.

## Likelihood Explanation

- **Precondition:** attacker must be a registered, verified user who has created at least one transaction — a normal product workflow requiring no elevated privilege.
- **ID discovery:** approver IDs are sequential auto-increment integers. They are also returned by `GET /transactions/:transactionId/approvers` for any transaction the attacker has visibility into (as observer, signer, or approver), or can be brute-forced with small integers.
- **Attack complexity:** a single crafted HTTP DELETE request. No race condition, no complex setup.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the authorized `transactionId`. The pattern already exists in `updateTransactionApprover`:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to get the root of its tree.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.
3. Only then proceed with `removeNode`.

Alternatively, pass `transactionId` into `removeTransactionApprover` and perform the check inside the service, consistent with how `updateTransactionApprover` handles it. [3](#0-2) 

## Proof of Concept

```
# Setup
POST /auth/register  → attacker account
POST /transactions   → creates T_attacker (attacker is creator)

# Victim setup (separate session)
POST /transactions   → creates T_victim
POST /transactions/{T_victim}/approvers → creates approver tree, returns approver IDs

# Attack (attacker session)
DELETE /transactions/{T_attacker}/approvers/{approver_id_from_T_victim}

# Result:
# 1. getCreatorsTransaction(T_attacker, attacker) → PASSES (attacker owns T_attacker)
# 2. removeTransactionApprover(approver_id_from_T_victim) → EXECUTES
# 3. removeNode recursively soft-deletes victim's entire approver subtree
# 4. T_victim's approval requirements are silently destroyed
```

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
