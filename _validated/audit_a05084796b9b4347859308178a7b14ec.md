Audit Report

## Title
Creator Can Execute Individual Transactions from an Atomic Group, Breaking Atomicity Guarantee

## Summary
The `PATCH /transactions/execute/:id` endpoint allows the creator of any `isManual` transaction to submit it individually to Hedera, even when that transaction belongs to an `atomic: true` `TransactionGroup`. No group-membership or atomicity check exists at the execution gate. When the scheduler later fires `executeTransactionGroup` for the remaining members, it silently filters out the already-executed member and proceeds with a partial set — permanently violating the atomicity invariant.

## Finding Description

**Entry point — `PATCH /transactions/execute/:id`**

`TransactionsService.executeTransaction` performs exactly two checks before calling `executeService.executeTransaction(transaction)`:

1. Is the caller the creator? (`getTransactionForCreator`)
2. Is the transaction flagged `isManual`? [1](#0-0) 

There is no check for whether the transaction belongs to a `TransactionGroup` with `atomic: true` or `sequential: true`. The `TransactionGroup` entity explicitly carries both flags: [2](#0-1) 

**Silent partial-execution in `executeTransactionGroup`**

When the scheduler later calls `executeTransactionGroup`, the very first operation is a filter that silently drops any group member not in `WAITING_FOR_EXECUTION`: [3](#0-2) 

There is no guard that checks `transactionGroup.atomic` before applying this filter. A member already moved to `EXECUTED` by the individual-execution path is silently dropped, and the remaining members are submitted without it.

**Scheduler path that triggers group execution**

`addGroupExecutionTimeout` calls `executeTransactionGroup` without re-verifying that all group members are still in `WAITING_FOR_EXECUTION`: [4](#0-3) 

**`isManual` is a valid per-transaction flag in a group**

`CreateTransactionGroupDto` accepts individual `CreateTransactionDto` objects per group item, each of which can carry `isManual: true` through `validateAndPrepareTransaction`: [5](#0-4) [6](#0-5) 

## Impact Explanation

The `atomic` flag on a `TransactionGroup` is the protocol's guarantee that all member transactions execute together or not at all. A malicious creator can violate this guarantee by individually executing one member before the group fires. Other signers approved the full group under the assumption of atomicity. Selective execution can exploit intermediate on-chain state — for example, an account key update in transaction A that changes authorization for B and C, or a token transfer in A that the creator exploits before B and C settle. The DB records will show mixed statuses (`EXECUTED` for A at an earlier time, `EXECUTED` for B and C later) with no indication that atomicity was violated.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The only precondition is that at least one transaction in the atomic group is created with `isManual: true`, which is a standard product feature accepted by the group creation API. The endpoint `PATCH /transactions/execute/:id` is reachable by any authenticated creator. No special tooling is required beyond a standard HTTP client.

## Recommendation

1. **Add a group-membership check in `executeTransaction`**: Before calling `executeService.executeTransaction`, check whether the transaction belongs to a `TransactionGroup` with `atomic: true` or `sequential: true`. If it does, reject the individual execution with an appropriate error code.

   ```typescript
   // In TransactionsService.executeTransaction
   const transaction = await this.getTransactionForCreator(id, user);
   if (!transaction.isManual) throw new BadRequestException(ErrorCodes.IO);

   // NEW: block individual execution for atomic/sequential group members
   if (transaction.groupItem?.group?.atomic || transaction.groupItem?.group?.sequential) {
     throw new BadRequestException(ErrorCodes.CANNOT_EXECUTE_GROUP_MEMBER_INDIVIDUALLY);
   }
   ```

2. **Add an atomicity guard in `executeTransactionGroup`**: Before filtering, check whether the group is `atomic: true`. If any member is not in `WAITING_FOR_EXECUTION`, abort the entire group execution rather than proceeding with a partial set.

   ```typescript
   if (transactionGroup.atomic) {
     const allReady = transactionGroup.groupItems.every(
       tx => tx.transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
     );
     if (!allReady) throw new Error('Atomic group cannot be partially executed.');
   }
   ```

## Proof of Concept

1. Creator submits an atomic `TransactionGroup` (A, B, C) with transaction A having `isManual: true`:
   ```
   POST /transaction-groups
   { "atomic": true, "groupItems": [
     { "seq": 0, "transaction": { ..., "isManual": true } },
     { "seq": 1, "transaction": { ... } },
     { "seq": 2, "transaction": { ... } }
   ]}
   ```
2. Organization members sign all three transactions expecting atomic execution.
3. After A's `validStart` passes, creator calls:
   ```
   PATCH /transactions/execute/{A_id}
   ```
4. `TransactionsService.executeTransaction` passes both checks (creator ✓, `isManual` ✓); `ExecuteService.executeTransaction` submits A to Hedera; A's DB status becomes `EXECUTED`.
5. The chain-service scheduler picks up B and C (still `WAITING_FOR_EXECUTION`), calls `executeTransactionGroup`.
6. The filter at `execute.service.ts` lines 65–67 drops A (status is `EXECUTED`, not `WAITING_FOR_EXECUTION`); B and C are submitted without A.
7. Atomicity is broken: A executed at a different time and independently of B and C, with no error or audit trail indicating the violation.
