Audit Report

## Title
Transaction Creator Can Manipulate Approver Structure After Approvals Are Collected, Bypassing Governance Controls

## Summary
Both `removeTransactionApprover` and `updateTransactionApprover` allow the transaction creator to delete approvers or lower the approval threshold at any point in the transaction lifecycle — including after approvals have already been collected — because neither endpoint enforces any transaction-status guard. The only check performed is creator identity. This allows the creator to retroactively weaken or nullify the organizational approval requirement that other participants already acted upon.

## Finding Description

**`removeTransactionApprover` controller** — only calls `getCreatorsTransaction` (creator identity check), then immediately invokes the service to soft-delete the approver node: [1](#0-0) 

**`removeTransactionApprover` service** — performs no status check before soft-deleting and emitting a status-update event: [2](#0-1) 

**`updateTransactionApprover` service** — calls `getCreatorsTransaction` (creator check only) and then allows threshold reduction with no status guard: [3](#0-2) [4](#0-3) 

**`getCreatorsTransaction`** — confirms it checks only creator identity, never transaction status: [5](#0-4) 

After either mutation, `emitTransactionStatusUpdate` (or `emitTransactionUpdate`) is published via NATS. The chain service's scheduler periodically calls `processTransactionStatus`, which re-evaluates whether the transaction has sufficient cryptographic signatures to advance to `WAITING_FOR_EXECUTION`: [6](#0-5) 

**Note on mechanism precision:** `processTransactionStatus` evaluates Hedera cryptographic signature requirements (via `hasValidSignatureKey` and `smartCollate`), not the approver-table threshold directly. The approver table is the organizational governance layer. When an approver approves, their cryptographic signature is stored in the `TransactionApprover` table and is later collated into the SDK transaction bytes. Removing an approver record or lowering the threshold retroactively changes which signatures are considered valid for the organizational gate, and the next scheduler cycle re-evaluates the resulting state. [7](#0-6) 

## Impact Explanation
The approval system is the primary organizational governance control. Approvers who already signed believed their approval was a meaningful gate. A creator who retroactively removes an approver or lowers the threshold renders those approvals irrelevant. This allows execution of arbitrary Hedera network transactions (account updates, file changes, token operations, node operations) without the full organizational sign-off that was in place when participants acted. The integrity of the governance record is silently broken.

## Likelihood Explanation
The attack requires only that the caller be the transaction creator — a standard, unprivileged role reachable by any registered organization user. No leaked credentials, admin keys, or race conditions are needed. export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }

  if (updatesByStatus.size > 0) {
    await Promise.all(
      Array.from(updatesByStatus.values()).map(async ({ newStatus, oldStatus, ids }) => {
        const result = await transactionRepo
          .createQueryBuilder()
          .update(Transaction)
          .set({ status: newStatus })
          .where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
          .returning('id')
          .execute();

        for (const row of result.raw) {
          statusChanges.set(row.id, newStatus);
        }
      })
    );
  }

  return statusChanges;
}

  /* Checks if the signers are enough to sign the transactions and update their statuses */
  async updateTransactions(from: Date, to?: Date) {
    //Get the transaction, creatorKey, groupItem, and group. We need the group info upfront
    //in order to determine if the group needs to be processed together
    const transactions = await this.transactionRepo.find({
      where: {
        status: In([
          TransactionStatus.WAITING_FOR_SIGNATURES,
          TransactionStatus.WAITING_FOR_EXECUTION,
        ]),
        validStart: to ? Between(from, to) : MoreThan(from),
      },
      relations: {
        creatorKey: true,
        groupItem: {
          group: true,
        },
      },
      order: {
        validStart: 'ASC',
      },
    });

    const results = await processTransactionStatus(this.transactionRepo, this.transactionSignatureService, transactions);

    if (results.size > 0) {
      const events = Array.from(results.keys(), id => ({ entityId: id }));
      emitTransactionStatusUpdate(this.notificationsPublisher, events);
    }

    return transactions;
