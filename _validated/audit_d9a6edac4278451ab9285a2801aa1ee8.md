Audit Report

## Title
Any Authenticated User Can Post Comments on Any Transaction Without Access Verification

## Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` performs no check to verify that the requesting user is a participant (creator, signer, observer, or approver) of the target transaction. Any authenticated organization user can post comments on any transaction by supplying an arbitrary `transactionId`. The `GET` comment endpoints are similarly unguarded. The developer acknowledged this gap with an inline `//TODO`.

## Finding Description

**Root cause — `CommentsController`:**

The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level — these only confirm the caller holds a valid, non-blacklisted JWT and has a verified account. There is no guard or service-level check that the user is the creator, a signer, an observer, or an approver of the transaction identified by `:transactionId`. The developer acknowledged this gap inline: [1](#0-0) 

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
```

**Root cause — `CommentsService.createComment`:**

The service blindly creates and persists the comment for the supplied `transactionId` with no membership lookup: [2](#0-1) 

```typescript
async createComment(user, transactionId, dto): Promise<TransactionComment> {
  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);
}
```

Compare this to `TransactionsService.verifyAccess`, which correctly checks creator, observer, signer, and approver membership before granting access: [3](#0-2) 

The `createComment` path never calls `verifyAccess` or any equivalent.

**Read endpoints are also unguarded:**

`getComments` and `getCommentById` return comment data for any `transactionId` or comment `id` to any authenticated user, leaking the existence and content of comments on transactions the user has no business accessing: [4](#0-3) 

## Impact Explanation
- **Unauthorized write on any transaction:** Any authenticated user can inject comments into transactions they have no role in, polluting the audit trail and potentially confusing legitimate participants.
- **Information disclosure:** Reading comments on a transaction exposes its comment history to unauthorized users, leaking workflow and decision metadata.
- **Integrity degradation:** The comment history is part of the transaction's collaborative record. Unauthorized entries corrupt the integrity of that record.

## Likelihood Explanation
The attacker precondition is only a valid organization account — the lowest privilege level in the system. No admin access, no leaked secrets, and no special network position are required. The endpoint is reachable over the standard REST API. The `//TODO` comment confirms the developers are aware the check is missing, meaning the gap is not accidental but deferred, making it reliably reproducible.

## Recommendation
In `CommentsService.createComment`, before persisting the comment, call `TransactionsService.getTransactionWithVerifiedAccess(transactionId, user)` (which internally calls `verifyAccess`) and throw `UnauthorizedException` if the user is not a participant. Apply the same check in `getTransactionComments` and `getTransactionCommentById` — either by injecting `TransactionsService` into `CommentsService`, or by adding a dedicated guard at the controller level analogous to the one used in `ObserversService.getTransactionObserversByTransactionId`: [5](#0-4) 

## Proof of Concept

1. Register as a normal organization user (no admin, no special role).
2. Obtain a valid JWT via `POST /auth/login`.
3. Send the following request with any `transactionId` the attacker does not own:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   Content-Type: application/json

   { "message": "Injected comment" }
   ```
4. The request passes all guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) and `CommentsService.createComment` persists the comment without any membership check.
5. Enumerate comments on any transaction via:
   ```
   GET /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   ```
   All comments are returned with no access verification.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
