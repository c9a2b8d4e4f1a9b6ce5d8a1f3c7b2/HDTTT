# Audit Report

## Title
`archiveTransaction` Skips Status Validation for Manual Transactions, Enabling Terminal-State Overwrite

## Summary
The `archiveTransaction` function in `transactions.service.ts` contains a boolean logic error in its guard condition. When `transaction.isManual` is `true`, the status check is entirely bypassed, allowing a transaction creator to archive a manual transaction in any status — including terminal states `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED` — overwriting them with `ARCHIVED` and corrupting the audit trail.

## Finding Description

**Root cause — `back-end/apps/api/src/transactions/transactions.service.ts` lines 711–718:**

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual        // ← short-circuits the entire guard when isManual=true
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard throws only when **both** of the following are true:
- `status` is not `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`
- `isManual` is `false`

When `isManual = true`, the second operand of `&&` is always `false`, so the condition is always `false` and the `throw` is never reached. The status of the transaction is **never validated** for manual transactions.

The registered error message for `OMTIP` is `"Only manual transaction in progress can be archived"`, confirming the **intended** semantics require both conditions: the transaction must be manual **and** in-progress. [2](#0-1) 

The correct guard should use `||` (OR), not `&&` (AND):

```typescript
// Correct intent: throw if NOT manual OR NOT in-progress
if (
  !transaction.isManual ||
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

**Contrast with `cancelTransaction`**, which correctly enforces status unconditionally:

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [3](#0-2) 

**Test gap.** The existing "not archiveable" unit test uses a transaction with `isManual` unset (falsy), so it never exercises the `isManual = true` + terminal-status combination: [4](#0-3) 

The success-path test uses `isManual: true` only with `WAITING_FOR_EXECUTION`, never with a terminal status: [5](#0-4) 

## Impact Explanation

A transaction creator can:

1. Overwrite `EXECUTED` → `ARCHIVED`, erasing on-chain execution evidence from the organization's shared view.
2. Overwrite `FAILED` → `ARCHIVED`, hiding a failed Hedera submission and preventing observers/approvers from investigating.
3. Overwrite `CANCELED` → `ARCHIVED`, obscuring the cancellation history.
4. Overwrite `EXPIRED` → `ARCHIVED`, hiding the fact that a transaction was never executed.

The `repo.update` call at line 720 is unconditional once the guard passes: [6](#0-5) 

There is no secondary check at the controller layer — the endpoint is protected only by JWT and email-verification guards: [7](#0-6) 

## Likelihood Explanation

**Medium.** The attacker must be an authenticated, email-verified user who is the creator of a manual transaction. This is a standard user role with no elevated privileges required. The exploit requires only a single `PATCH /transactions/archive/:id` API call with a valid transaction ID. No race condition, chaining, or special timing is needed.

## Recommendation

Replace the `&&` with `||` and invert the conditions to match the stated intent:

```typescript
if (
  !transaction.isManual ||
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Additionally, add a unit test covering `isManual: true` with each terminal status (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) to assert that `archiveTransaction` throws `ErrorCodes.OMTIP`.

## Proof of Concept

```
1. Authenticate as a user with a valid JWT (email verified).
2. Create a manual transaction (isManual: true) — e.g., via POST /transactions with isManual=true.
3. Allow the transaction to reach EXECUTED status (or directly seed the DB with status=EXECUTED).
4. Call: PATCH /transactions/archive/<id>
   Authorization: Bearer <jwt>
5. Observe: HTTP 200, transaction status is now ARCHIVED.
6. Expected: HTTP 400 with ErrorCodes.OMTIP ("Only manual transaction in progress can be archived").
```

The guard at line 715 evaluates `!transaction.isManual` as `false` (since `isManual=true`), making the entire `&&` condition `false`, so the `throw` at line 717 is never reached and `repo.update` stamps `ARCHIVED` unconditionally. [8](#0-7)
