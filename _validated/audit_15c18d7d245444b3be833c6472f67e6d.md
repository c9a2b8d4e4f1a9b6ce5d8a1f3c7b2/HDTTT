Audit Report

## Title
Any Transaction Creator Can Delete Approvers Belonging to Other Users' Transactions

## Summary
The `DELETE /:id` handler in `ApproversController` verifies the authenticated user is the creator of the transaction identified by `transactionId`, but then deletes the approver identified by the separate `id` parameter without verifying that approver belongs to that same transaction. This allows any transaction creator to delete approvers from any other transaction in the system.

## Finding Description

**Root cause:** The controller-level authorization check and the service-level deletion operate on two independent identifiers with no cross-validation.

In `approvers.controller.ts`, the `removeTransactionApprover` handler:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id alone
``` [1](#0-0) 

Step 1 verifies the caller is the creator of `transactionId`. Step 2 deletes the approver with the separate `id` parameter. There is no check that approver `id` belongs to `transactionId`.

The service-level `removeTransactionApprover` accepts only an approver `id` and performs no ownership or transaction-scope validation: [2](#0-1) 

`getTransactionApproverById` fetches by `id` alone with no transaction scope: [3](#0-2) 

`removeNode` soft-deletes the entire approver subtree rooted at the given `id` with no ownership check: [4](#0-3) 

**Contrast with `updateTransactionApprover`:** The service's update path correctly performs this cross-validation — it calls `getRootNodeFromNode` and explicitly checks `rootNode.transactionId !== transactionId`, throwing `UnauthorizedException` on mismatch. The delete path is missing this equivalent guard. [5](#0-4) 

## Impact Explanation
An attacker with a valid account can silently delete approvers from any transaction in the organization, including transactions they have no legitimate access to. This breaks the multi-signature approval workflow: removing a required approver can allow a transaction to proceed without the intended oversight, or irreversibly corrupt the approval tree structure. This constitutes unauthorized state mutation of another user's transaction data.

## Likelihood Explanation
The attacker only needs to be a registered, verified user — no admin or special role is required. Approver IDs are sequential integers (auto-incremented primary keys), making enumeration trivial. The attack requires a single authenticated HTTP request. Any malicious organization member can exploit this immediately.

## Recommendation
Add a cross-validation check in `removeTransactionApprover` — either in the controller or service — mirroring what `updateTransactionApprover` already does:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to get the root of the approver's tree.
2. Verify `rootNode.transactionId === transactionId` (the URL parameter). Throw `UnauthorizedException` on mismatch.
3. Only then proceed with `removeNode`.

This ensures the approver being deleted actually belongs to the transaction the caller is authorized to modify.

## Proof of Concept

1. Attacker registers as a normal user and creates **Transaction A**. They are now the creator of A.
2. Attacker enumerates or guesses an approver `id` (e.g., `42`) belonging to victim's **Transaction B** (IDs are sequential integers).
3. Attacker sends:
   ```
   DELETE /transactions/{A_id}/approvers/42
   Authorization: Bearer <attacker_token>
   ```
4. `getCreatorsTransaction(A_id, attacker)` passes — attacker is creator of A.
5. `removeTransactionApprover(42)` executes — fetches approver 42 (belonging to Transaction B) and calls `removeNode(42)`, soft-deleting the entire approver subtree from Transaction B without any further check.
6. Transaction B's approval structure is now corrupted, with no error returned to the attacker.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
