**Audit Report**

## Title
`TransferTransactionModel.getSigningAccounts()` and `getReceiverAccounts()` Only Inspect HBAR Transfers, Omitting Token and NFT Transfer Participants

## Summary
Both the back-end and front-end `TransferTransactionModel` implementations compute required signers by iterating exclusively over `hbarTransfersList`. The Hedera SDK `TransferTransaction` also exposes `tokenTransfersList` and `nftTransfersList`, whose senders and receivers are never added to the signing-accounts set. When a `TransferTransaction` carrying HTS fungible-token or NFT transfers is submitted to the tool via the API (using externally crafted transaction bytes), the resulting `KeyList` built by `TransactionSignatureService.computeSignatureKey()` is incomplete, causing either on-chain rejection or silent under-counting of required approvers.

## Finding Description

**Affected files and confirmed lines:**

- `back-end/libs/common/src/transaction-signature/model/transfer-transaction.model.ts` lines 10–19 and 22–31
- `front-end/src/renderer/utils/transactionSignatureModels/transfer-transaction.model.ts` lines 6–16 and 19–28

Both `getSigningAccounts()` implementations iterate only over `this.transaction.hbarTransfersList`:

```ts
for (const transfer of this.transaction.hbarTransfersList) {
  if (transfer.amount.isNegative() && !transfer.isApproved) {
    accounts.add(transfer.accountId.toString());
  }
}
``` [1](#0-0) [2](#0-1) 

The same omission applies to `getReceiverAccounts()`, which also only reads `hbarTransfersList`, so token/NFT receivers with `receiverSignatureRequired = true` are silently dropped. [3](#0-2) [4](#0-3) 

The result of `getSigningAccounts()` is consumed directly by `TransactionSignatureService.computeSignatureKey()` to build the `KeyList` of required signatures:

```ts
const requirements = this.extractSignatureRequirements(transactionModel);
await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);
``` [5](#0-4) 

This `KeyList` is then flattened and used to determine which user keys are required to sign, driving the entire organization multi-sig approval workflow. [6](#0-5) 

**Root cause:** `TransferTransactionModel` assumes a `TransferTransaction` only carries HBAR transfers. The Hedera SDK exposes three distinct lists (`hbarTransfersList`, `tokenTransfersList`, `nftTransfersList`), but only the first is consulted. No grep match for `addTokenTransfer`, `addNftTransfer`, `tokenTransfersList`, or `nftTransfersList` exists anywhere in the codebase, confirming the omission is total. [7](#0-6) 

## Impact Explanation

**Primary impact — transaction failure / workflow DoS:** Any `TransferTransaction` submitted to the API that carries HTS fungible-token or NFT transfers where the token sender is not the fee payer will be processed with an incomplete `KeyList`. The tool will either fail to collect the required signatures or submit the transaction to Hedera without the sender's signature, resulting in an `INVALID_SIGNATURE` rejection. The transaction fee is consumed and the workflow is permanently blocked for that transaction. [8](#0-7) 

**Secondary impact — approval-threshold bypass:** When the token sender coincides with the fee payer, the tool collects that account's key once (as fee payer) and never adds it as a distinct "signing account." If the organization's threshold key structure requires the token sender to be an explicitly counted approver, the threshold is silently under-counted, allowing the transaction to be marked ready with fewer approvals than policy requires. [9](#0-8) 

## Likelihood Explanation

**Corrected likelihood (lower than claimed):** The tool's built-in UI (`TransferHbar.vue` / `createTransferHbarTransaction`) only constructs HBAR-only `TransferTransaction`s using `addHbarTransfer` / `addApprovedHbarTransfer`. There is no UI path that calls `addTokenTransfer` or `addNftTransfer`. [10](#0-9) 

However, the API accepts externally crafted raw `transactionBytes` (hex-encoded), as documented in the API reference. An authenticated organization user can craft a `TransferTransaction` with HTS token or NFT transfers externally, submit the bytes to the API, and trigger the defective `computeSignatureKey` path. No elevated privilege beyond normal authenticated access is required. The vulnerability is therefore real and reachable, though it requires use of the API rather than the GUI.

## Recommendation

In both `TransferTransactionModel` implementations, extend `getSigningAccounts()` to also iterate over `tokenTransfersList` (checking `amount < 0 && !isApproved`) and `nftTransfersList` (checking `senderAccountId` where `!isApproval`). Similarly extend `getReceiverAccounts()` to include token and NFT receivers. Example addition for the back-end model:

```ts
for (const transfer of this.transaction.tokenTransfersList) {
  if (transfer.amount < 0 && !transfer.isApproved) {
    accounts.add(transfer.accountId.toString());
  }
}
for (const transfer of this.transaction.nftTransfersList) {
  if (!transfer.isApproval) {
    accounts.add(transfer.senderAccountId.toString());
  }
}
```

Add corresponding unit tests using mocked `tokenTransfersList` and `nftTransfersList` entries, mirroring the existing test structure in `transfer-transaction.model.spec.ts`. [11](#0-10) 

## Proof of Concept

1. Craft a `TransferTransaction` externally using the Hedera SDK:
   ```ts
   const tx = new TransferTransaction()
     .setTransactionId(TransactionId.generate('0.0.PAYER'))
     .setNodeAccountIds(['0.0.3'])
     .addTokenTransfer('0.0.TOKEN', '0.0.SENDER', -100)
     .addTokenTransfer('0.0.TOKEN', '0.0.RECEIVER', 100)
     .freeze();
   const hexBytes = Buffer.from(tx.toBytes()).toString('hex');
   ```
2. Submit `hexBytes` to the organization API as a new transaction (authenticated as a normal user).
3. Observe that `computeSignatureKey` returns a `KeyList` containing only the fee-payer key — `0.0.SENDER`'s key is absent.
4. The tool proceeds to collect signatures without ever requesting `0.0.SENDER`'s signature.
5. On execution, Hedera rejects the transaction with `INVALID_SIGNATURE` because the token sender's key is missing. [12](#0-11)

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```

  private async addSigningAccountKeys(
    signatureKey: KeyList,
    transaction: Transaction,
    signingAccounts: Set<string>
  ): Promise<void> {
    for (const account of signingAccounts) {
      try {
        const accountInfo = await this.accountCacheService.getAccountInfoForTransaction(
          transaction,
          account,
        );
        if (accountInfo?.key) {
          signatureKey.push(accountInfo.key);
        }
      } catch (error) {
        this.logger.error(`Failed to get key for account ${account}: ${error.message}`);
      }
    }
  }
