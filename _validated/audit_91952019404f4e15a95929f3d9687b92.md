Audit Report

## Title
Any Verified User Can Claim Another User's Public Key, Permanently Blocking Their Key Registration

## Summary
The `POST /user/:userId?/keys` endpoint in `UserKeysService.uploadKey` enforces global uniqueness of public keys across all users but requires no cryptographic proof of ownership. Any verified user can register any syntactically valid Hedera public key under their own account. Because the uniqueness check rejects a second registration of the same key by a different user, an attacker who pre-registers a victim's public key permanently prevents the victim from registering it.

## Finding Description

**Root Cause**

`uploadKey` in `UserKeysService` performs a global `findOne` on `publicKey` with `withDeleted: true`, then rejects the request if the found record belongs to a different user: [1](#0-0) 

No challenge-response or signature verification is performed to prove the caller controls the private key corresponding to the submitted public key.

`UploadUserKeyDto` validates only that the submitted string is parseable as a Hedera public key — no ownership proof field exists: [2](#0-1) 

The controller is mounted at `POST /user/:userId?/keys` and guarded by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. The `userId` route parameter is **not read** in the `uploadKey` handler — the key is always assigned to the JWT-authenticated caller: [3](#0-2) 

This is explicitly confirmed by the e2e test: [4](#0-3) 

**Exploit Flow**

1. Attacker (verified user) calls `GET /user/:victimId/keys` — accessible to all verified users — to enumerate the victim's registered public keys, or learns the victim's upcoming public key out-of-band (e.g., from a transaction setup conversation).
2. Before the victim registers the key (or after the victim soft-deletes it), the attacker calls `POST /user/<any_id>/keys` with `{ publicKey: <victim_key> }`.
3. The key is now owned by the attacker's `userId` in the `user_key` table.
4. When the victim calls `POST /user/keys` with their own public key, `uploadKey` finds `userKey.userId !== user.id` and throws `ErrorCodes.PU` (Bad Request).
5. The victim can never register that key in the organization back-end.

## Impact Explanation

The victim is permanently blocked from registering their public key. Transaction signing and approval workflows depend on `user.keys`:

- `userKeysRequiredToSign` matches the user's registered keys against the transaction's required signers: [5](#0-4) 

- `approveTransaction` resolves the signing key exclusively from the authenticated user's own key list via `user.keys.find(key => key.id === dto.userKeyId)`: [6](#0-5) 

A victim whose key is squatted by an attacker cannot be identified as a required signer, cannot submit signatures, and cannot approve transactions — constituting a targeted denial-of-service against their participation in any multi-signature workflow within the organization.

## Likelihood Explanation

- **Attacker precondition:** must be a verified user in the same organization; no admin or privileged access is required.
- **Key discoverability:** `GET /user/:id/keys` is accessible to all verified users, exposing already-registered keys. New keys can be learned out-of-band (e.g., shared during transaction creation).
- **Attack complexity:** a single authenticated HTTP POST; trivially automatable.
- **Victim recourse:** none — the attacker owns the key record and can soft-delete and immediately re-register it to maintain the lock indefinitely.

## Recommendation

Require cryptographic proof of ownership before accepting a public key registration. The standard approach is a **challenge-response**:

1. Client requests a short-lived nonce from the server (tied to the authenticated user's session).
2. Client signs the nonce with the private key corresponding to the submitted public key.
3. Server verifies the signature against the submitted public key before persisting the record.

This ensures only the holder of the private key can register the corresponding public key, regardless of whether another user submits the same public key string.

## Proof of Concept

```
# Step 1 – Attacker learns victim's upcoming public key (e.g., from GET /user/2/keys or out-of-band)
VICTIM_KEY="e0c8ec2758a5879ffac226a13c0c516b799e72e35141a0dd828f94d37988a4b7"

# Step 2 – Attacker (authenticated as attacker_token) registers the victim's key under their own account
curl -X POST https://<org-server>/user/99999/keys \
  -H "Authorization: Bearer <attacker_token>" \
  -H "Content-Type: application/json" \
  -d "{\"publicKey\": \"$VICTIM_KEY\"}"
# → 201 Created; key is now owned by attacker's userId

# Step 3 – Victim attempts to register their own key
curl -X POST https://<org-server>/user/2/keys \
  -H "Authorization: Bearer <victim_token>" \
  -H "Content-Type: application/json" \
  -d "{\"publicKey\": \"$VICTIM_KEY\"}"
# → 400 Bad Request: ErrorCodes.PU ("Public key in use")
# Victim is permanently blocked from registering this key.
```
