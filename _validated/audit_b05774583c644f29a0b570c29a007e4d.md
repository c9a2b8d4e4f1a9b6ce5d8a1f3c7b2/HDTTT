Audit Report

## Title
`removeTransactionApprover` Authorization Scope Mismatch Allows Any Transaction Creator to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of the transaction identified by `transactionId`, but then deletes the approver record identified by the separate `id` parameter with no verification that the approver actually belongs to `transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Controller — decoupled authorization and deletion:**

In `approvers.controller.ts` lines 102–113, the handler performs two independent operations:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes by id — no binding
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction identified by `transactionId`. It says nothing about the approver identified by `id`. [2](#0-1) 

**Service — no cross-check in `removeTransactionApprover`:**

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [3](#0-2) 

There is no assertion that `approver.transactionId === transactionId`. The two URL parameters are completely decoupled after the initial creator check.

**Contrast with the protected `updateTransactionApprover` path:**

The update path correctly performs this cross-check before any mutation:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [4](#0-3) 

The delete path has no equivalent guard.

## Impact Explanation

- An attacker can delete approvers from any transaction they do not own, silently destroying the multi-signature approval workflow of victim transactions.
- A transaction requiring threshold approvals (e.g., 2-of-3) can have its entire approver tree removed, causing it to stall permanently or proceed without the intended governance controls.
- The `emitTransactionStatusUpdate` call at line 541 fires with `approver.transactionId` (the victim's transaction ID), potentially triggering a premature status recalculation on the victim's transaction — advancing it to `WAITING_FOR_EXECUTION` if the approval threshold is no longer enforced.
- `removeNode` performs a recursive soft-delete of the entire subtree rooted at the targeted approver, so a single request can wipe an entire approver tree. [5](#0-4) 

## Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated, verified user with at least one created transaction. This is a normal user role — no admin or privileged access required.
- Approver IDs are sequential auto-increment integers (`id` primary key on `TransactionApprover`), making enumeration of valid approver IDs trivial via a simple loop.
- The attack requires only a single HTTP `DELETE` request with a crafted `transactionId` (attacker's own) and `id` (victim's approver ID).
- No rate limiting or anomaly detection is visible on this endpoint.

## Recommendation

In `removeTransactionApprover` (service), add a cross-check that the approver's root node belongs to the supplied `transactionId` before deletion — mirroring the guard already present in `updateTransactionApprover`:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root node.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.
3. Then proceed with `removeNode`.

Alternatively, the controller can pass `transactionId` into `removeTransactionApprover` and perform the binding check there, keeping the service method consistent with the update path.

## Proof of Concept

**Setup:**
- Attacker (User A) owns Transaction #1 with approver ID #10.
- Victim (User B) owns Transaction #2 with approver ID #99 (a root approver node in a 2-of-3 threshold tree).

**Attack request:**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer <User A's JWT>
```

**Execution trace:**
1. `getCreatorsTransaction(1, userA)` — passes, because User A is the creator of Transaction #1.
2. `removeTransactionApprover(99)` — fetches approver #99 (belongs to Transaction #2), calls `removeNode(99)`, recursively soft-deletes the entire approver subtree of Transaction #2.
3. `emitTransactionStatusUpdate` fires with `entityId: 2` (Transaction #2's ID), potentially advancing Transaction #2's status prematurely.

**Result:** User B's Transaction #2 has its entire approver tree destroyed by User A, with no error returned to the attacker. The attack requires no special privileges beyond being a registered, verified user with one transaction.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
