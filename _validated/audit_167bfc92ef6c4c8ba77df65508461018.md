Audit Report

## Title
Any Authenticated User Can Enumerate All Transaction Groups via Unguarded "Testing Only" Endpoint

## Summary
The `GET /transaction-groups` endpoint in `TransactionGroupsController` is explicitly marked "TESTING ONLY" in source code but is deployed in production with no user-scoping authorization. Any authenticated, verified user can call it and receive the full list of every `TransactionGroup` record in the database, bypassing the per-user access controls enforced on every other transaction-related endpoint.

## Finding Description

**Root cause — missing authorization on a deliberately unrestricted endpoint:**

The handler at line 52–57 of the controller takes no `user` parameter and passes nothing to the service: [1](#0-0) 

The service's `getTransactionGroups()` issues an unconditional `find(TransactionGroup)` — no `WHERE` clause, no user filter: [2](#0-1) 

The controller class carries only three guards: [3](#0-2) 

- `JwtBlackListAuthGuard` — checks the token is not blacklisted, no ownership check. [4](#0-3) 

- `JwtAuthGuard` — validates the JWT signature only. [5](#0-4) 

- `VerifiedUserGuard` — checks `user.status === UserStatus.NONE` (i.e., account is verified), nothing more. [6](#0-5) 

None of these guards perform any ownership or membership check.

**Contrast with every other endpoint in the same controller**, which correctly scopes results to the requesting user: [7](#0-6) 

`getTransactionGroup` passes `user` to the service, which then enforces membership via a scoped SQL query and throws `UnauthorizedException` if the user has no relation to the group: [8](#0-7) 

## Impact Explanation

Transaction groups aggregate multiple `Transaction` records (Hedera account operations, token transfers, file updates, system operations). Full enumeration of all groups leaks:

- The existence, IDs, descriptions, and metadata of every group in the system
- Which transactions belong to which groups and their statuses
- Organizational workflow structure and timing across all tenants

This is an **unauthorized cross-tenant data disclosure**. Any compromised or malicious account can silently harvest the entire organizational transaction group history without triggering any per-resource access check.

## Likelihood Explanation

The endpoint is reachable by any user who completes the standard registration and login flow — no elevated privilege is required. The route is `GET /transaction-groups` with no additional path segment. Swagger is enabled (`@ApiTags('Transaction Groups')` is present on the controller), making the route trivially discoverable. The attacker needs only a valid JWT obtained through the normal login process. [9](#0-8) 

## Recommendation

**Immediate:** Remove or disable the `getTransactionGroups()` handler entirely. It is explicitly marked "TESTING ONLY" and has no legitimate production use case.

**If retention is required:** Restrict the endpoint to admin users only by adding `@UseGuards(AdminGuard)` (the `AdminGuard` already exists in the codebase at `back-end/apps/api/src/guards/admin.guard.ts`) and pass the authenticated `user` to the service to scope results appropriately. [10](#0-9) 

## Proof of Concept

```
# 1. Register and verify a new account (standard flow)
POST /auth/register  { "email": "attacker@example.com", "password": "..." }

# 2. Obtain a JWT
POST /auth/login  { "email": "attacker@example.com", "password": "..." }
# → { "accessToken": "<JWT>" }

# 3. Enumerate all transaction groups
GET /transaction-groups
Authorization: Bearer <JWT>

# Response: full array of every TransactionGroup in the database,
# including groups belonging to other users/organizations,
# with their descriptions, IDs, and associated transaction metadata.
```

The service call `this.dataSource.manager.find(TransactionGroup)` at line 37 of `transaction-groups.service.ts` issues `SELECT * FROM transaction_group` with no predicate, confirming the full-table read. [2](#0-1)
