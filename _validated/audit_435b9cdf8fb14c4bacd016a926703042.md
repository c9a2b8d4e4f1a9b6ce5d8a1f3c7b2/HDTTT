Audit Report

## Title
Any Transaction Creator Can Delete Approvers Belonging to Other Users' Transactions

## Summary
The `DELETE /:id` handler in `ApproversController` verifies the caller is the creator of the transaction identified by `transactionId`, but then deletes the approver identified by the separate `id` parameter with no check that the approver actually belongs to that transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause:** The controller-level authorization check and the service-level deletion operate on two independent identifiers with no cross-validation.

In `approvers.controller.ts`, the handler at `DELETE /:id`:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks transactionId
await this.approversService.removeTransactionApprover(id);               // deletes by id — no link to transactionId
``` [1](#0-0) 

`getCreatorsTransaction` only validates that the caller created the transaction at `transactionId`: [2](#0-1) 

`removeTransactionApprover` fetches the approver by `id` alone and immediately deletes it — no transaction scope check: [3](#0-2) 

`getTransactionApproverById` queries only by `id` with no `transactionId` filter: [4](#0-3) 

`removeNode` performs a recursive soft-delete of the entire approver subtree with no ownership check: [5](#0-4) 

**The fix already exists in the sibling method.** `updateTransactionApprover` correctly validates that the approver's root transaction matches the URL parameter before proceeding: [6](#0-5) 

This cross-validation is simply absent from the delete path.

## Impact Explanation
An attacker with a valid account can silently delete approvers — including entire approver subtrees — from any transaction in the organization. This breaks the multi-signature approval workflow: removing a required approver can allow a transaction to proceed without the intended oversight, or permanently corrupt the approval tree structure. The deletion is a soft-delete (`deletedAt = now()`), so it is not immediately visible but is irreversible through normal API flows.

## Likelihood Explanation
The attacker only needs to be a registered, verified user — no admin or special role is required. Approver IDs are sequential integers (`id` is a standard auto-increment primary key), making enumeration trivial. The attack requires a single authenticated HTTP `DELETE` request. Any malicious organization member can exploit this immediately with no special tooling.

## Recommendation
In `removeTransactionApprover` (service), after fetching the approver, resolve its root node via `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId` before calling `removeNode`. This is exactly the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Add this cross-validation:
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call accordingly.

## Proof of Concept
1. Attacker (user A) registers and creates transaction **T_A**. They are now the creator of T_A.
2. Victim (user B) creates transaction **T_B** with approver record `id = 42`.
3. Attacker sends:
   ```
   DELETE /transactions/{T_A_id}/approvers/42
   Authorization: Bearer <attacker_token>
   ```
4. `getCreatorsTransaction(T_A_id, attacker)` passes — attacker is creator of T_A.
5. `removeTransactionApprover(42)` executes — fetches approver 42 (belonging to T_B), calls `removeNode(42)`, soft-deletes the entire subtree rooted at approver 42 from T_B.
6. Approver 42 and all its children are now deleted from T_B with no error returned to the attacker.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
