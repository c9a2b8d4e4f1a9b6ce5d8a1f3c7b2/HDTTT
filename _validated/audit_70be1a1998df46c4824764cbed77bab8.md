Audit Report

## Title
Cross-Transaction Approver Deletion: Creator of Any Transaction Can Delete Approvers Belonging to Any Other Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller owns the transaction at `:transactionId`, then unconditionally deletes the approver at `:id` with no check that the approver belongs to that transaction. Any authenticated user who is the creator of at least one transaction can permanently delete approvers from any other transaction in the system.

## Finding Description

**Root cause — decoupled authorization and destructive action**

In `approvers.controller.ts`, the `DELETE /:id` handler performs two fully independent operations:

```typescript
// lines 102-113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of :transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver :id — no binding check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction referenced by `:transactionId`:

```typescript
// lines 640-641
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
``` [2](#0-1) 

`removeTransactionApprover` then deletes the approver row identified by the separate `:id` parameter with no check that it belongs to the same transaction:

```typescript
// lines 534-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [3](#0-2) 

The two URL parameters (`:transactionId` and `:id`) are never cross-validated. The authorization gate is satisfied by owning *any* transaction, while the destructive action operates on an *arbitrary* approver row.

**The fix already exists in the sibling `PATCH` handler** — `updateTransactionApprover` correctly traverses to the root node and validates the binding before proceeding:

```typescript
// lines 386-394
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [4](#0-3) 

This cross-check is entirely absent from the `DELETE` path.

## Impact Explanation
- Approval requirements on any transaction can be silently gutted. A transaction requiring M-of-N approvals can be reduced to zero approvers, allowing it to proceed to execution without the mandated governance sign-off.
- The `removeNode` function performs a recursive soft-delete of the entire approver subtree, so a single request can eliminate an entire approval tree, not just one approver.
- Notifications fire against the *victim* transaction (via `emitTransactionStatusUpdate` with `approver.transactionId`), not the attacker's transaction, providing no audit trail attributing the deletion to the attacker.
- In a multi-signature organizational workflow this directly enables unauthorized fund movement or unauthorized Hedera network operations (account updates, node operations) that the approval gate was designed to prevent.

## Likelihood Explanation
- **Attacker precondition**: authenticated, verified user who has created at least one transaction — the default state for any active organizational participant.
- **No privileged access required**: no admin key, no leaked credential, no internal network access.
- **Approver ID enumeration**: approver IDs are sequential integers. The `GET /transactions/:transactionId/approvers` endpoint exposes them to any participant of a transaction they legitimately belong to, and IDs are globally sequential so adjacent IDs from other transactions are trivially guessable.
- **No anomaly barrier**: the malicious request is structurally identical to a legitimate creator deleting their own approver.

## Recommendation
Mirror the validation already present in `updateTransactionApprover`. In `removeTransactionApprover` (or in the controller before calling it), after fetching the approver by `id`, traverse to its root node via `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId` before proceeding with deletion. The corrected flow in `approvers.service.ts`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-validate: ensure the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover` and update the controller call accordingly.

## Proof of Concept

**Setup:**
- Attacker (user A) is the creator of transaction `TX_A` (id = 1).
- Victim transaction `TX_B` (id = 2) is owned by user B and has approver with id = 99.

**Attack:**
```http
DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>
```

**Execution trace:**
1. `getCreatorsTransaction(1, userA)` — passes, attacker owns TX_A.
2. `removeTransactionApprover(99)` — fetches approver 99 (which belongs to TX_B), calls `removeNode(99)`, soft-deletes the entire approver subtree rooted at 99.
3. Returns `true`. Approver 99 and all its children are permanently removed from TX_B.

**Result:** TX_B's approval requirement is silently reduced. The notification fires against TX_B (not TX_A), and there is no log entry linking the deletion to the attacker's session on TX_A. [5](#0-4)
