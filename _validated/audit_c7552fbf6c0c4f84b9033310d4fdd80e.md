# Audit Report

## Title
TOCTOU Race Condition in `createTransactions` Allows Duplicate Transaction Records for the Same Hedera Transaction ID

## Summary
`createTransactions` in `back-end/apps/api/src/transactions/transactions.service.ts` performs a uniqueness check for `transactionId` via a plain `SELECT` outside any database transaction or row-level lock, then writes the record in a separate `entityManager.transaction` block. Because the `transaction` table has no `UNIQUE` constraint on `transactionId`, two concurrent requests carrying the same Hedera transaction ID both pass the check and both succeed in writing to the database, producing duplicate rows.

## Finding Description

**Phase 1 — uniqueness check (lines 415–427):**

```typescript
const existing = await this.repo.find({
  where: {
    transactionId: In(transactionIds),
    status: Not(In([TransactionStatus.CANCELED, TransactionStatus.REJECTED, TransactionStatus.ARCHIVED])),
  },
  select: ['transactionId'],
});
```

This is a plain `SELECT` with no `SELECT ... FOR UPDATE` row-level lock and is executed **outside** the DB transaction that follows. [1](#0-0) 

**Phase 2 — write (lines 436–462):**

```typescript
const savedTransactions = await this.entityManager.transaction(async (entityManager) => {
  // ...
  return await entityManager.save(Transaction, transactions);
});
```

The `entityManager.transaction` block wraps only the `INSERT`; it does not re-validate uniqueness inside the same serializable scope. [2](#0-1) 

**No database-level safety net:**

The `Transaction` entity declares `transactionId` as a plain `@Column()` with no `unique: true`:

```typescript
@Column()
transactionId: string;
``` [3](#0-2) 

The initial migration confirms the column is created without a `UNIQUE` constraint (`"transactionId" character varying NOT NULL`). [4](#0-3) 

No subsequent migration adds one. All `UNIQUE` index additions in later migrations target cache-related join tables, not `transaction.transactionId`. [5](#0-4) 

## Impact Explanation

- **Duplicate execution risk**: The chain service polls for `WAITING_FOR_EXECUTION` transactions. With two rows for the same Hedera transaction ID, it may attempt to submit the same transaction to the Hedera network twice. The second submission will be rejected by Hedera (duplicate transaction ID), but the second row's status will be set to `FAILED`, creating a permanently inconsistent record.
- **Signature-collection split**: Signers who sign one duplicate row do not satisfy the threshold on the other. Neither row may ever accumulate enough signatures to execute, permanently locking the transaction in `WAITING_FOR_SIGNATURES`.
- **State corruption**: Observers, approvers, and notification records attached to one duplicate are invisible to queries against the other, breaking the integrity of the entire transaction lifecycle.
- **Severity**: High — permanent state corruption and potential double-submission to the Hedera network, reachable by any authenticated user.

## Likelihood Explanation

- **Attacker precondition**: A valid JWT token — obtainable by any registered user.
- **Trigger mechanism**: Two simultaneous HTTP `POST /transactions` requests, trivially achievable with `Promise.all`, `curl --parallel`, or any load-testing tool.
- **No special timing required**: Node.js's event loop and async I/O make the window between the `SELECT` and the `INSERT` reliably exploitable under normal server load; no millisecond-precision timing is needed.
- **Likelihood**: High.

## Recommendation

Apply one or more of the following fixes in combination:

1. **Add a `UNIQUE` constraint at the database level** via a new migration:
   ```sql
   CREATE UNIQUE INDEX "UQ_transaction_transactionId_active"
   ON "transaction" ("transactionId")
   WHERE status NOT IN ('CANCELED', 'REJECTED', 'ARCHIVED');
   ```
   This is a partial unique index that mirrors the application-level check and provides an atomic, database-enforced guard.

2. **Move the uniqueness check inside the same DB transaction** and use a pessimistic lock (`SELECT ... FOR UPDATE`) or use `INSERT ... ON CONFLICT DO NOTHING` / `ON CONFLICT DO UPDATE` (upsert) semantics so the check and write are atomic.

3. **Add `unique: true` to the entity column** (or use a TypeORM `@Index({ unique: true })`) and generate the corresponding migration, so TypeORM schema sync also enforces the constraint.

Both requests reach `repo.find` (lines 415–427) before either has committed; both see zero existing rows and pass the guard. Both proceed to `entityManager.save` (line 458); both succeed because there is no DB-level uniqueness enforcement. [6](#0-5)
