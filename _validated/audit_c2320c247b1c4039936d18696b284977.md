Audit Report

## Title
Missing Rate Limiting on `/verify-reset` Combined with Oversized TOTP Window Enables Brute-Force Account Takeover

## Summary
The `POST /verify-reset` endpoint has no rate-limiting guard, while `/signup`, `/login`, and `/reset-password` are all protected by `EmailThrottlerGuard`. Simultaneously, the TOTP configuration uses `window: 20` with `step: 60`, making 41 distinct 8-digit codes simultaneously valid. An attacker who triggers a password reset for a victim (receiving the OTP JWT in the response body) can brute-force the OTP code against the unthrottled endpoint to take over the account.

## Finding Description

**Root Cause 1 — Oversized TOTP window (confirmed):** [1](#0-0) 

`window: 20` causes `otplib` to accept any code from `currentStep - 20` to `currentStep + 20`, meaning **41 distinct 8-digit codes are simultaneously valid** at any moment. This reduces the expected brute-force attempts from 100,000,000 to ~2,439,024.

**Root Cause 2 — No rate limiting on `/verify-reset` (confirmed):** [2](#0-1) 

Compare with the protected endpoints: [3](#0-2) [4](#0-3) 

**Root Cause 3 — OTP JWT returned in response body of `/reset-password` (confirmed):** [5](#0-4) 

The `createOtp` method returns `{ token }` directly, and the controller passes it through to the HTTP response. The attacker does not need access to the victim's email inbox to obtain the OTP JWT.

**Root Cause 4 — Failed attempts do not blacklist the JWT:**

On a failed OTP check, `verifyOtp` throws `UnauthorizedException` at line 115 before the blacklist call at line 157 is reached. The attacker can reuse the same OTP JWT for unlimited attempts. [6](#0-5) [7](#0-6) 

**`EmailThrottlerGuard` tracks by `req.body.email`:** [8](#0-7) 

Even if `EmailThrottlerGuard` were applied to `/verify-reset`, it would not function correctly there because the request body contains an OTP code (`OtpDto`), not an email field.

## Impact Explanation
Full account takeover for any user whose email is known. The attacker calls `POST /reset-password` once to obtain the OTP JWT, then iterates 8-digit codes against the unthrottled `POST /verify-reset`. On success, the server returns a `verified: true` OTP JWT. The attacker then calls `PATCH /set-password` with that JWT to set an arbitrary password. [9](#0-8) 

Once the account is controlled, the attacker can access all organization transactions, approve or cancel multi-signature workflows, and exfiltrate any data visible to that user. Admin accounts are equally vulnerable.

## Likelihood Explanation
Preconditions are minimal: only the victim's email address is required. The OTP JWT is returned directly in the HTTP response of `/reset-password`, so no inbox access is needed. The absence of any rate limiting on `/verify-reset` means the attack is a simple HTTP loop. The `window: 20` setting meaningfully reduces the search space by a factor of 41, increasing success probability per unit time.

## Recommendation

1. **Add `EmailThrottlerGuard` (or a per-IP throttler) to `/verify-reset`.** Since the body contains an OTP code rather than an email, a per-IP throttler (NestJS default `ThrottlerGuard`) is more appropriate here.
2. **Reduce `window` to `1` (the `otplib` default).** This limits valid codes to 3 at any moment (current ± 1 step), restoring the full 10^8 search space.
3. **Blacklist the OTP JWT after a configurable number of failed attempts**, or on any failed attempt, forcing the attacker to re-trigger `/reset-password` (which is throttled).
4. **Do not return the OTP JWT in the HTTP response body.** Deliver it only via the email channel, so the attacker must control the victim's inbox to proceed.

## Proof of Concept

```
# Step 1: Obtain OTP JWT from response body (no inbox access needed)
TOKEN=$(curl -s -X POST https://target/auth/reset-password \
  -H 'Content-Type: application/json' \
  -d '{"email":"victim@example.com"}' | jq -r '.token')

# Step 2: Brute-force /verify-reset — no throttle guard present
for CODE in $(seq -w 0 99999999); do
  RESULT=$(curl -s -o /dev/null -w "%{http_code}" \
    -X POST https://target/auth/verify-reset \
    -H 'Content-Type: application/json' \
    -H "otp: Bearer $TOKEN" \
    -d "{\"token\":\"$CODE\"}")
  if [ "$RESULT" = "200" ]; then
    echo "Valid OTP: $CODE"; break
  fi
done

# Step 3: Use verified JWT to set arbitrary password
curl -X PATCH https://target/auth/set-password \
  -H 'Content-Type: application/json' \
  -H "otp: Bearer $VERIFIED_TOKEN" \
  -d '{"password":"attacker_controlled_password"}'
```

With `window: 20`, ~41 codes are valid at any moment, so the expected number of attempts before a hit is ~2,439,024. At 1,000 req/s this completes in ~40 minutes — well within a typical OTP expiration window.
