Audit Report

## Title
Unbounded In-Memory Loop in `getTransactionsToSign` Causes Server-Side Resource Exhaustion via Single Authenticated Request

## Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction from the database with no row limit, then performs expensive per-transaction async operations — including database-backed cache lookups and potential Hedera mirror-node HTTP calls — before applying pagination in memory. A single authenticated request from a user with many active transactions can cause disproportionate CPU, memory, and I/O load, degrading service for all concurrent users.

## Finding Description

**Root cause — `back-end/apps/api/src/transactions/transactions.service.ts`, lines 295–316**

The `getTransactionsToSign` method accepts `Pagination` parameters (`limit`, `offset`) but never passes them to the database query:

```typescript
// Lines 295-299: NO `take` or `skip` — fetches every matching row
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then iterates over the entire result set in a sequential `for` loop, calling `userKeysToSign` for every transaction:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { console.log(error); }
}
``` [2](#0-1) 

Pagination is applied only after the full loop completes:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

Compare this with `getHistoryTransactions` and `getTransactionsToApprove`, which correctly pass `skip: offset, take: limit` to the database query before returning results. [4](#0-3) [5](#0-4) 

**Cost of each iteration**

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` → `computeSignatureKey` performs, per transaction:

1. Deserialization of `transactionBytes` via `SDKTransaction.fromBytes`
2. One `accountCacheService.getAccountInfoForTransaction` call for the fee-payer account
3. One call per signing account (`addSigningAccountKeys` loop)
4. One call per receiver account (`addReceiverAccountKeys` loop)
5. One call per node account for node transactions (`addNodeKeys`) [6](#0-5) 

Each `getAccountInfoForTransaction` call performs a database lookup against the cache DB. When the cache is stale or absent, it additionally performs a mirror-node HTTP request and writes back to the DB. [7](#0-6) 

The `importSignatures` endpoint in the same file contains an explicit developer acknowledgment of a related missing limit: *"Added a batch mechanism, probably should limit this on the api side of things"* (line 575), confirming awareness that input-size controls are absent in this service layer. [8](#0-7) 

## Impact Explanation

A single authenticated request to `GET /transactions/sign` causes the server to:
- Load all N non-terminal transactions into heap memory
- Execute N × M sequential async DB/cache operations (and potentially mirror-node HTTP calls for uncached accounts) before returning even a single page of results
- Hold the Node.js event loop for the duration of the sequential `await` chain, blocking other requests
- Exhaust server memory (OOM) or saturate the cache database connection pool at large N

The `GET /sign` endpoint has no per-endpoint `@Throttle` decorator. [9](#0-8) 

The global user-level throttler allows 100 requests/minute and 10 requests/second per user — this limits request frequency but does not bound the computational cost of a single request. [10](#0-9) 

## Likelihood Explanation

- **No elevated privilege required**: any registered, verified user can create transactions — the core product feature.
- **Attacker-controlled growth**: a user creates N transactions with future `validStart` timestamps, keeping them in `WAITING_FOR_SIGNATURES` status (the filter applied at lines 266–277). Hedera transactions can have a `validStart` up to ~180 days in the future.
- **Single request trigger**: one HTTP request to `GET /transactions/sign` is sufficient to trigger the full unbounded loop.
- **Endpoint still accessible**: although the controller comment notes "NO LONGER USED BY FRONT-END", the endpoint remains live and unauthenticated clients can reach it after login. [11](#0-10) [12](#0-11) 

## Recommendation

1. **Push pagination into the database query**: add `take: limit` and `skip: offset` to the `repo.find()` call in `getTransactionsToSign`, mirroring the pattern used in `getHistoryTransactions`.
2. **Enforce a hard maximum page size**: cap `limit` at a reasonable value (e.g., 100) in the `PaginationParams` decorator or service layer.
3. **Add a per-endpoint throttle**: apply a `@Throttle` decorator to `GET /sign` to limit how frequently a single user can invoke the expensive loop.
4. **Consider deprecating the endpoint**: since the front-end no longer uses it, evaluate removing or disabling `GET /sign` entirely, as the per-transaction `GET /sign/:transactionId` endpoint already serves the same purpose without the unbounded scan. [13](#0-12) 

## Proof of Concept

```
# 1. Register and verify a user account
# 2. Create N transactions with validStart set ~180 days in the future
#    (keeps them in WAITING_FOR_SIGNATURES indefinitely)
for i in $(seq 1 5000); do
  curl -s -X POST https://<host>/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"name":"tx'$i'","transactionBytes":"<bytes>","creatorKeyId":<id>,...}'
done

# 3. Issue a single request — server must process all 5000 transactions
#    before returning page 1 of 10 results
curl -s "https://<host>/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer $TOKEN"
# Server performs 5000 × M sequential DB/mirror-node calls before responding.
# Repeat within the 100 req/min throttle window to sustain load.
```

The server must complete all N iterations of the `for` loop — including N deserialization operations and up to N × M `getAccountInfoForTransaction` DB calls — before the response is sent, regardless of the requested page size. [14](#0-13)

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
