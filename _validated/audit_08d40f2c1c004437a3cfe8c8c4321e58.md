## Audit Report

## Title
Any Authenticated User Can Corrupt Transaction Group Structure via Missing Ownership Check in `removeTransactionGroup`

## Summary
`removeTransactionGroup` in `TransactionGroupsService` fetches a `TransactionGroup` and all its `TransactionGroupItem` records with no user-ownership filter. It then deletes each `TransactionGroupItem` row **before** the per-transaction ownership check in `removeTransaction` fires. When `removeTransaction` throws `UnauthorizedException`, the `TransactionGroupItem` row is already permanently deleted with no wrapping database transaction to roll it back, leaving the group in a corrupted, partially-dismembered state. Any verified, authenticated user can trigger this against any group ID.

## Finding Description

**Entry point** — `DELETE /transaction-groups/:id` in `TransactionGroupsController`: [1](#0-0) 

The controller passes the call directly to `removeTransactionGroup(user, groupId)` with no additional ownership guard beyond `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — all of which only verify that the caller is a valid, verified user, not that they own the group.

**Root cause** — `removeTransactionGroup` in `TransactionGroupsService`: [2](#0-1) 

Step-by-step breakdown:

1. **Line 173** — `findOneBy(TransactionGroup, { id })` fetches the group by ID with **no user ownership check**. Any group ID resolves successfully for any authenticated user.

2. **Lines 177–186** — `find(TransactionGroupItem, { where: { group: { id: group.id } } })` fetches **all** group items with **no user filter**. Compare this to `cancelTransactionGroup`, which calls `getTransactionGroup(user, groupId, false)` → `getTransactionGroupItemsQuery(sql, id, user)`, which applies `roles: { signer: true, creator: true, observer: true, approver: true }` filtering. [3](#0-2) 

3. **Lines 187–191** — The loop removes the `TransactionGroupItem` row **first** (line 189), then calls `removeTransaction` (line 190). There is **no wrapping database transaction** around this loop.

4. `removeTransaction` calls `getTransactionForCreator`, which throws `UnauthorizedException` if the caller is not the creator: [4](#0-3) 

5. When the exception is thrown, the `TransactionGroupItem` deletion at line 189 is already committed. The group and the underlying `Transaction` record remain, but the linking `TransactionGroupItem` row is gone permanently.

## Impact Explanation
The group is left in an inconsistent state: one or more `TransactionGroupItem` rows are deleted while the parent `TransactionGroup` and the underlying `Transaction` records remain. This is a data-integrity violation. The attacker can repeat the request to strip all group items from any group they do not own, rendering the group permanently unusable. Because the operation is not wrapped in a database transaction, there is no rollback path.

## Likelihood Explanation
The attacker only needs to be a verified, authenticated user — a low bar in a multi-user deployment. Group IDs are sequential integers (`ParseIntPipe` on the route parameter), making enumeration trivial. No special knowledge of the victim's data is required beyond a valid group ID.

## Recommendation
Apply two independent fixes:

1. **Add an ownership check before any mutation.** Reuse the existing `getTransactionGroup(user, groupId, false)` helper (which already applies the role-based filter and throws `UnauthorizedException` on mismatch) at the top of `removeTransactionGroup`, replacing the bare `findOneBy` call.

2. **Wrap the entire loop in a database transaction.** Use `this.dataSource.transaction(async manager => { ... })` so that any `UnauthorizedException` thrown by `removeTransaction` rolls back all preceding `TransactionGroupItem` deletions atomically. [5](#0-4) 

## Proof of Concept

1. Attacker (authenticated user A) discovers group ID `42`, which belongs to victim user B (sequential IDs make enumeration trivial).
2. Attacker sends: `DELETE /transaction-groups/42`
3. `removeTransactionGroup` fetches group `42` — succeeds with no ownership check (line 173).
4. All `TransactionGroupItem` rows for group `42` are fetched — no user filter (lines 177–186).
5. First iteration of the loop:
   - `this.dataSource.manager.remove(TransactionGroupItem, groupItem)` executes and commits (line 189).
   - `this.transactionsService.removeTransaction(transactionId, user, false)` calls `getTransactionForCreator`, which finds `transaction.creatorKey.userId !== user.id` and throws `UnauthorizedException` (line 887).
6. The exception propagates out of `removeTransactionGroup`. The `TransactionGroup` and `Transaction` records are untouched, but the first `TransactionGroupItem` row is permanently deleted.
7. Repeating the request strips additional group items on each call until the group is fully dismembered while the group record and all transactions remain in the database.

```typescript
  async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
    if (!group) {
      throw new Error('group not found');
    }
    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
      relations: {
        group: true,
      },
      where: {
        group: {
          id: group.id,
        },
      },
    });
    for (const groupItem of groupItems) {
      const transactionId = groupItem.transactionId;
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(transactionId, user, false);
    }

    await this.dataSource.manager.remove(TransactionGroup, group);

    emitTransactionUpdate(this.notificationsPublisher, groupItems.map(gi => ({ entityId: gi.transactionId })));

    return true;
  }
```
