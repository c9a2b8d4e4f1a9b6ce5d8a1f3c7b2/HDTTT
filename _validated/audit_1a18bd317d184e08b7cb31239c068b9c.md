Audit Report

## Title
IDOR in `removeTransactionApprover`: Transaction Creator Can Delete Approvers Belonging to Any Other Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller is the creator of `:transactionId`, but then passes the approver `:id` directly to `removeTransactionApprover()` with no check that the approver belongs to `:transactionId`. An authenticated user who is the creator of any one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause — decoupled authorization and operation:**

In `approvers.controller.ts`, the `DELETE /:id` handler performs two independent, unlinked calls:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id only
``` [1](#0-0) 

The service method `removeTransactionApprover` accepts only the approver `id` and performs no cross-check against the transaction:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // deletes whatever approver has this id
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly validates ownership before acting:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)          // ← cross-check present here
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The delete path is missing this exact cross-check, creating an IDOR.

## Impact Explanation
An attacker can unilaterally soft-delete any approver node — including entire approval subtrees via `removeNode` — from any transaction they do not own. The `removeNode` function recursively soft-deletes the target and all its children: [4](#0-3) 

This directly undermines the multi-signature approval workflow: required approvals can be silently removed, reducing or eliminating the threshold of signatures needed before a transaction is executed on the Hedera network. This constitutes unauthorized state mutation of another user's transaction governance structure.

## Likelihood Explanation
Preconditions are minimal and represent normal product usage:
1. The attacker must be an authenticated, verified user — standard account.
2. The attacker must be the creator of at least one transaction — a normal product flow.
3. Being listed as an approver on the victim transaction (also a normal flow) grants access to approver IDs via `getVerifiedApproversByTransactionId`: [5](#0-4) 

Sequential integer primary keys also make brute-force enumeration feasible even without that relationship. No privileged access, leaked credentials, or admin keys are required.

## Recommendation
Mirror the ownership cross-check already present in `updateTransactionApprover`. Before calling `removeTransactionApprover(id)`, resolve the root node of the target approver and verify its `transactionId` matches the `:transactionId` route parameter:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);

  // ADD: verify the approver belongs to this transaction
  const approver = await this.approversService.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const rootNode = await this.approversService.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

Alternatively, modify `removeTransactionApprover` in the service to accept and enforce `transactionId`, consistent with how `updateTransactionApprover` is implemented.

## Proof of Concept

**Setup:**
- Alice (attacker) creates Transaction A — she is its creator.
- Bob (victim) creates Transaction B with a multi-approver workflow. Alice is added as one of the approvers of Transaction B (a normal, unprivileged role).

**Steps:**

1. Alice calls `GET /transactions/{B_id}/approvers`. Because she is listed as an approver of B, `getVerifiedApproversByTransactionId` returns all approver records including their integer IDs.

2. Alice identifies a victim approver ID (e.g., `victim_approver_id = 42`) belonging to Transaction B.

3. Alice sends:
   ```
   DELETE /transactions/{A_id}/approvers/42
   Authorization: Bearer <alice_token>
   ```

4. Server evaluates `getCreatorsTransaction(A_id, alice)` — Alice is the creator of Transaction A → **passes**.

5. Server calls `removeTransactionApprover(42)` — fetches approver 42 (which belongs to Transaction B), soft-deletes it and its entire subtree via `removeNode`. **No transaction ownership check occurs.**

6. The approval requirement for Transaction B is silently removed or corrupted, without Bob's knowledge or consent.
