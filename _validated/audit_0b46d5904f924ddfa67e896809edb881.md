Audit Report

## Title
`removeTransactionApprover` Does Not Adjust Parent Threshold After Child Removal, Permanently Locking Transaction Approval State

## Summary
`removeTransactionApprover` in `approvers.service.ts` soft-deletes an approver node without reading or updating the parent list node's `threshold`. If the parent's threshold exceeds the remaining child count after deletion, the approval condition becomes permanently unsatisfiable and the transaction is irreversibly stuck. The `updateTransactionApprover` code path handles the identical structural change correctly, making the omission in `removeTransactionApprover` a clear inconsistency.

## Finding Description

**Root cause — `removeTransactionApprover` never touches the parent threshold:**

`removeTransactionApprover` fetches the target node, calls `removeNode`, and emits a status event. It never loads the parent, never reads `parent.threshold`, and never issues an update to the parent record. [1](#0-0) 

`removeNode` itself is a recursive SQL soft-delete that marks the target node and all its descendants as deleted. It operates only on the subtree rooted at the given id and has no awareness of the parent's `threshold` column. [2](#0-1) 

**Contrast with `updateTransactionApprover` (`dto.listId === null` branch):**

The detach path in `updateTransactionApprover` explicitly handles the same structural change — one fewer child under a parent — and performs the threshold correction: [3](#0-2) 

It decrements the effective child count, soft-removes the parent if it becomes empty, and reduces `threshold` to `newParentApproversLength` when the current threshold would exceed the remaining children. `removeTransactionApprover` has no equivalent logic.

**Secondary finding — cross-transaction authorization bypass in the controller:**

The controller enforces creator-only access by calling `getCreatorsTransaction(transactionId, user)` using the `transactionId` URL parameter, then calls `removeTransactionApprover(id)` with the approver `id` URL parameter. [4](#0-3) 

`removeTransactionApprover` never verifies that the approver identified by `id` actually belongs to `transactionId`. A user who is the creator of transaction A can issue `DELETE /transactions/A/approvers/{id}` where `id` is an approver node belonging to transaction B. The creator check passes (they own A), but the deleted node is from B. This is a cross-transaction authorization bypass. [5](#0-4) 

Note: the original report's claim that "any authenticated user" can delete any approver is **incorrect**. The controller does enforce creator-level authorization for the `transactionId` in the URL. The actual bypass is narrower: a creator of any transaction can delete approvers belonging to a different transaction.

## Impact Explanation

**Threshold corruption (primary):** After a child node is removed, if `parent.threshold > remaining_children`, no combination of approvals can ever satisfy the threshold condition. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. The creator cannot advance or recover the transaction through the approval path; the only escape is cancellation if that path remains reachable.

**Cross-transaction approver deletion (secondary):** A creator of any transaction can delete approver nodes from transactions they do not own, silently corrupting another transaction's approver tree and potentially triggering the same stuck-threshold condition on a transaction they have no legitimate authority over.

## Likelihood Explanation

The threshold corruption is triggered by a normal, in-scope API call (`DELETE /transactions/:transactionId/approvers/:id`) made by the transaction creator. No cryptographic break or privileged access beyond transaction ownership is required. Any organization workflow that uses threshold approver trees (threshold > 1 with multiple children) is exposed whenever the creator removes a child node. The cross-transaction bypass requires only that the attacker be the creator of at least one transaction.

## Recommendation

1. **In `removeTransactionApprover`**, after calling `removeNode`, load the deleted node's parent (via `approver.listId`), recount its remaining non-deleted children, and apply the same threshold correction already present in `updateTransactionApprover`:
   - If `remaining_children === 0`, soft-remove the parent.
   - Else if `parent.threshold > remaining_children`, update `parent.threshold = remaining_children`.

2. **In the controller's `removeTransactionApprover` handler** (or in the service), verify that the approver identified by `id` belongs to `transactionId` before proceeding with deletion. The `getRootNodeFromNode` helper already exists and returns the root's `transactionId`, making this check straightforward. [6](#0-5) 

## Proof of Concept

**Threshold corruption:**

1. Creator builds: `parent (threshold=2, listId=null)` → children `[UserA (id=10), UserB (id=11)]`.
2. Creator calls `DELETE /transactions/1/approvers/10`.
3. Controller calls `getCreatorsTransaction(1, creator)` — passes.
4. Controller calls `removeTransactionApprover(10)`.
5. `removeNode(10)` soft-deletes UserA. Parent record is untouched: `threshold=2`, children now `[UserB]`.
6. Only 1 approver remains; threshold of 2 can never be reached.
7. Transaction is permanently stuck.

**Cross-transaction approver deletion:**

1. Attacker is creator of transaction 1 (owns it legitimately).
2. Attacker discovers approver node `id=99` belonging to transaction 2 (owned by another user).
3. Attacker calls `DELETE /transactions/1/approvers/99`.
4. Controller calls `getCreatorsTransaction(1, attacker)` — passes (attacker owns tx 1).
5. `removeTransactionApprover(99)` deletes the approver from transaction 2 with no further ownership check.
6. Transaction 2's approver tree is silently corrupted.

    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"

      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
