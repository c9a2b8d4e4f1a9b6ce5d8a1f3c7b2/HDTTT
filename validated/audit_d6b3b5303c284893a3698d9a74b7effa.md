### Title
Unimplemented `getTransactionApproversForTransactions` Causes Approver Data to Always Be Empty in Transaction Group Details

### Summary

The `getTransactionApproversForTransactions` function in `transactions.service.ts` is explicitly stubbed out with a comment "To be implemented when approver functionality is added" and unconditionally returns an empty array. This function is the sole data source for populating approver information when a transaction group is fetched with `full=true`. As a result, every transaction inside a group always reports zero approvers to the client, regardless of how many approvers have been assigned, making the approval workflow invisible and non-functional for the transaction-group view.

### Finding Description

**Root cause — the stub:** [1](#0-0) 

```ts
async getTransactionApproversForTransactions(
  transactionIds: number[],
): Promise<TransactionApprover[]> {
  if (!transactionIds.length) {
    return [];
  }

  //To be implemented when approver functionality is added.
  return [];
}
```

**Call site — group detail fetch:** [2](#0-1) 

```ts
const [
  transactionSigners,
  transactionApprovers,   // always []
  transactionObservers,
] = await Promise.all([
  this.transactionsService.getTransactionSignersForTransactions(transactionIds),
  this.transactionsService.getTransactionApproversForTransactions(transactionIds),
  this.transactionsService.getTransactionObserversForTransactions(transactionIds),
]);

// approverMap is always empty
const approverMap = this.groupBy(transactionApprovers, a => a.transactionId);

for (const groupItem of group.groupItems) {
  groupItem.transaction.approvers = approverMap.get(txId) ?? [];  // always []
}
```

The `TransactionApprover` entity, the `approversService.createTransactionApprovers`, `approveTransaction`, and `getApproversByTransactionId` are all fully implemented and functional for individual transactions. The database schema documents the `approved`, `signature`, `threshold`, and `userKeyId` fields. The approver tree structure, threshold logic, and approval signing flow are all wired up — but the batch-fetch path used exclusively by the group view is a dead stub. [3](#0-2) [4](#0-3) 

### Impact Explanation

Any user who calls `GET /transactions/groups/:id?full=true` receives `approvers: []` for every transaction in the group, regardless of the actual approver configuration. Concretely:

1. **Approval workflow is invisible for group transactions.** The front-end renders approval UI based on the `approvers` array returned in the group response. Since it is always empty, approvers assigned to group transactions cannot see their pending approval tasks through the group view.

2. **Approval-gate bypass risk.** If any client-side or downstream logic gates execution readiness on `transaction.approvers.length > 0`, the stub causes every group transaction to appear approval-free, potentially allowing the transaction to be treated as ready for execution before required approvals are collected.

3. **Data integrity gap.** The `approved` and `signature` fields on `TransactionApprover` records are never surfaced in the group context, so the group detail response is structurally incorrect and cannot be trusted by any consumer. [5](#0-4) 

### Likelihood Explanation

This is triggered by any authenticated user calling the group detail endpoint with `full=true`. No special privileges are required. The `getTransactionGroup` endpoint is a standard REST call available to any organization member who has visibility into the group. The stub is reachable on every invocation of that code path — it is not conditional on any feature flag or configuration. [6](#0-5) 

### Recommendation

Implement `getTransactionApproversForTransactions` in `transactions.service.ts` analogously to `getTransactionSignersForTransactions` and `getTransactionObserversForTransactions`:

```ts
async getTransactionApproversForTransactions(
  transactionIds: number[],
): Promise<TransactionApprover[]> {
  if (!transactionIds.length) return [];

  return this.entityManager.find(TransactionApprover, {
    where: {
      transactionId: In(transactionIds),
    },
    withDeleted: false,
  });
}
``` [7](#0-6) 

### Proof of Concept

1. As an organization admin, create a transaction group via `POST /transactions/groups` and assign at least one approver to one of the group's transactions via `POST /transactions/:id/approvers`.
2. Confirm the approver exists by calling `GET /transactions/:id/approvers` — it returns the approver correctly (this path uses `getApproversByTransactionId`, which is implemented).
3. Now call `GET /transactions/groups/:groupId?full=true`.
4. Inspect the response: every `groupItem.transaction.approvers` array is `[]`, even though step 2 confirmed approvers exist.
5. The approval state, threshold, and signature data are entirely absent from the group response, breaking any client logic that depends on this data. [8](#0-7) [1](#0-0)

```typescript
  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
