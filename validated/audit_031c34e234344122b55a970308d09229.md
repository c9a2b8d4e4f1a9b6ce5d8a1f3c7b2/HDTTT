Audit Report

## Title
Any Authenticated User Can Permanently Corrupt Another User's Transaction Group via Missing Ownership Check in `removeTransactionGroup`

## Summary
`removeTransactionGroup` in `back-end/apps/api/src/transactions/groups/transaction-groups.service.ts` performs a bare, unfiltered `findOneBy` on `TransactionGroup` with no ownership verification, then immediately deletes `TransactionGroupItem` rows outside any database transaction before delegating to `removeTransaction` (which does check ownership). Because each `remove` call is committed before the ownership check fires, an attacker can permanently delete one group item per request, eventually rendering the victim's group inaccessible.

## Finding Description

**Root cause — missing ownership gate before destructive mutation:**

`removeTransactionGroup` opens with an unfiltered lookup that accepts any group ID from any authenticated user:

```typescript
const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
``` [1](#0-0) 

It then iterates over every `TransactionGroupItem` and removes each one **before** the ownership check runs:

```typescript
await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // committed immediately
await this.transactionsService.removeTransaction(transactionId, user, false); // ownership check here
``` [2](#0-1) 

`removeTransaction` calls `getTransactionForCreator`, which throws `UnauthorizedException` when the caller is not the creator:

```typescript
if (transaction.creatorKey?.userId !== user?.id) {
  throw new UnauthorizedException('Only the creator has access to this transaction');
}
``` [3](#0-2) 

Because there is no wrapping database transaction around the loop, the `remove(TransactionGroupItem, …)` call on line 189 is committed to the database before `removeTransaction` throws. The loop aborts on the first item, but that `TransactionGroupItem` row is already permanently gone.

**Contrast with `cancelTransactionGroup`**, which correctly calls `getTransactionGroup(user, groupId, false)` first — a method that filters results through a user-scoped SQL query: [4](#0-3) 

`getTransactionGroup` enforces access by throwing when `groupItems.length === 0`: [5](#0-4) 

`removeTransactionGroup` never calls `getTransactionGroup`; it uses the unfiltered `findOneBy` instead.

**The controller exposes this endpoint to any verified user with no additional guard:** [6](#0-5) 

## Impact Explanation
After each attacker request, one `TransactionGroupItem` row is permanently deleted while the underlying `Transaction` record and the `TransactionGroup` shell remain intact. For a single-item group, one request is sufficient: `getTransactionGroup` subsequently returns `groupItems.length === 0` and throws `UnauthorizedException`, making the group permanently inaccessible to its owner. For multi-item groups, the attacker repeats the request once per item. The underlying transactions still exist but are no longer reachable through the group, permanently breaking the multi-signature workflow the group coordinates. The damage is irreversible without direct database intervention.

## Likelihood Explanation
The attacker requires only a valid, verified account — the same precondition as any normal user. `TransactionGroup` IDs are sequential integers (auto-increment primary keys), making them trivially enumerable. The attacker sends `DELETE /transaction-groups/1`, `DELETE /transaction-groups/2`, etc. until a group belonging to another user is hit. No privileged access, no leaked credentials, and no special knowledge are required.

## Recommendation
1. **Add an ownership check at the top of `removeTransactionGroup`**, mirroring `cancelTransactionGroup`, by calling `getTransactionGroup(user, id, false)` before any mutation. This reuses the existing user-scoped query and throws `UnauthorizedException` if the caller has no items in the group.
2. **Wrap the entire loop in a database transaction** (`this.dataSource.transaction(async manager => { … })`) so that any `UnauthorizedException` thrown by `removeTransaction` rolls back all preceding `remove` calls atomically.

## Proof of Concept
1. Attacker (user A, verified) enumerates group IDs: `DELETE /transaction-groups/5` where group 5 belongs to victim (user B).
2. `removeTransactionGroup` fetches group 5 with no ownership check (line 173).
3. Fetches all `TransactionGroupItem` rows for group 5 (lines 177–186).
4. First loop iteration: `manager.remove(TransactionGroupItem, groupItem)` executes and commits — item 1 is deleted (line 189).
5. `removeTransaction(transactionId, userA, false)` → `getTransactionForCreator` → `creatorKey.userId (B) !== user.id (A)` → throws `UnauthorizedException` (line 886–887).
6. Loop aborts. Group 5's first `TransactionGroupItem` is permanently gone; the `Transaction` and `TransactionGroup` rows remain.
7. Victim (user B) calls `GET /transaction-groups/5` → `getTransactionGroupItemsQuery` returns one fewer item. If only one item existed, `groupItems.length === 0` → `UnauthorizedException("You don't have permission to view this group.")` — group is permanently inaccessible.
8. Attacker repeats for remaining items to fully corrupt multi-item groups.
