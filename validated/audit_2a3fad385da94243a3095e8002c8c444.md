### Title
Duplicate `accountId` Key Collision in Transfer Rendering Causes Silent Omission of Transfer Entries

### Summary
The `v-for` loops in `TransferDetails.vue` and `TransferHbarFormData.vue` use `:key="debit.accountId"` (and `:key="credit.accountId"`) as the Vue rendering key. Because the Hedera `TransferTransaction` permits multiple HBAR transfers to/from the same account with different `isApproved` flags, a transaction containing two entries for the same `accountId` will cause Vue's virtual DOM to silently drop one entry from the rendered list. A signer reviewing the transaction sees fewer transfers than actually exist, violating rendering invariability and preventing informed consent.

### Finding Description

`getTransferHbarData` in `getData.ts` returns `transaction.hbarTransfersList` verbatim:

```typescript
// front-end/src/renderer/utils/sdk/getData.ts:213-218
export const getTransferHbarData = (transaction: Transaction): TransferHbarData => {
  assertTransactionType(transaction, TransferTransaction);
  return {
    transfers: transaction.hbarTransfersList,
  };
};
``` [1](#0-0) 

This list can legitimately contain two entries with the same `accountId` — one with `isApproved=false` and one with `isApproved=true` — because `createTransferHbarTransaction` calls both `addHbarTransfer` and `addApprovedHbarTransfer` for the same account: [2](#0-1) 

The `handleAddTransfer` logic in `TransferHbarFormData.vue` only merges transfers that share the same `accountId`, direction **and** `isApproved` value, so it explicitly allows two entries for the same account with differing approval status to coexist in `data.transfers`: [3](#0-2) 

Both the creation-time preview (`TransferHbarFormData.vue`) and the signing-time review (`TransferDetails.vue`) then render the list with `accountId` alone as the Vue `:key`:

```html
<!-- TransferHbarFormData.vue line 204 -->
<template v-for="(debit, i) in data.transfers" :key="debit.accountId">

<!-- TransferDetails.vue line 86 -->
<template v-for="debit in transaction.hbarTransfersList" :key="debit.accountId">
``` [4](#0-3) [5](#0-4) 

When two transfers share the same `accountId`, Vue treats them as the same DOM node and renders only one, silently discarding the other.

### Impact Explanation
A signer in organization mode loads a transaction via `TransferDetails.vue`. If the transaction contains both an approved and a non-approved debit from the same account (e.g., `-5 HBAR approved` + `-5 HBAR not approved` from `0.0.123`), the UI displays only one `-5 HBAR` row. The signer believes they are authorizing a `-5 HBAR` debit but actually signs a `-10 HBAR` debit. This directly undermines informed consent for transaction signing.

### Likelihood Explanation
The creation UI (`TransferHbarFormData.vue`) does not block adding an approved and a non-approved transfer to the same account — the merge guard at lines 80–88 only merges entries that match on all three fields (`accountId`, direction, `isApproved`). A transaction creator can therefore produce this state through normal UI interaction. In organization mode, the resulting transaction bytes are stored server-side and presented to other signers who have no way to detect the omission.

### Recommendation
Replace `:key="debit.accountId"` with a composite key that uniquely identifies each transfer entry, for example:

```html
:key="`${debit.accountId}-${debit.amount.toTinybars()}-${debit.isApproved}`"
```

Apply this fix to all four `v-for` loops in `TransferHbarFormData.vue` (lines 204 and 273) and `TransferDetails.vue` (lines 86 and 159). [6](#0-5) [7](#0-6) 

### Proof of Concept

1. In organization mode, create a `TransferTransaction` with two debits from the same account `0.0.123`:
   - `-5 HBAR`, `isApproved = false` (via "Add Transfer From")
   - `-5 HBAR`, `isApproved = true` (via "Add Transfer From" with "Approve Transfer" toggled)
2. Submit the transaction for signing.
3. A second user opens the signing view (`TransferDetails.vue`).
4. `transaction.hbarTransfersList` contains two entries both with `accountId = 0.0.123`.
5. Vue renders the `v-for` with `:key="debit.accountId"` — both entries share the key `"0.0.123"`.
6. Vue deduplicates and renders only one row: `-5 HBAR from 0.0.123`.
7. The signer sees a `-5 HBAR` debit and approves, but the actual on-chain debit is `-10 HBAR`.
