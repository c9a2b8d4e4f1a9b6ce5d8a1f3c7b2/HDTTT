Audit Report

## Title
`verifyAccess` Unconditionally Returns `true` for Terminal-Status Transactions, Bypassing Per-User Authorization

## Summary
The `verifyAccess` function in `transactions.service.ts` returns `true` for any transaction in a terminal status (EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED) without verifying that the requesting user has any relationship to that transaction. Any authenticated user can retrieve the full details of any historical transaction by supplying its integer ID.

## Finding Description

**Root cause — `verifyAccess` skips the user-relationship check for terminal statuses:**

Lines 789–798 of `transactions.service.ts` unconditionally return `true` the moment the transaction's status is in the terminal set. The user-relationship evaluation (creator, signer, observer, approver) that follows on lines 800–808 is never reached for those transactions. [1](#0-0) 

**Call site — `getTransactionWithVerifiedAccess` relies on this gate:**

`getTransactionWithVerifiedAccess` (lines 754–763) fetches the full transaction record — including `transactionBytes`, `transactionHash`, signers with `userKey`, observers, comments, and creator key — via `getTransactionById`, then calls `verifyAccess`. Because `verifyAccess` returns `true` for any terminal-status transaction regardless of the caller, the `UnauthorizedException` branch is never triggered for those records. [2](#0-1) 

`getTransactionById` eagerly loads `creatorKey`, `creatorKey.user`, `observers`, `comments`, and `signers` (with `userKey`) before the access check is applied. [3](#0-2) 

**Contrast with the list endpoint:**

`getTransactions` (lines 150–217) correctly scopes results to transactions where the user is creator, signer, observer, or approver. The detail endpoint provides no equivalent scoping for historical transactions, creating a direct inconsistency. [4](#0-3) 

**The behavior is explicitly tested and confirmed:**

The test suite at line 2100 asserts: *"should return history transaction, even if the user does not have verified access"* — confirming this is the actual runtime behavior, not a theoretical path. [5](#0-4) 

## Impact Explanation
Any authenticated user (no elevated privilege required) can enumerate integer transaction IDs and call the transaction-detail endpoint. For every transaction in a terminal status they receive:
- Full `transactionBytes` (serialized Hedera transaction, including node account IDs and memo fields)
- `transactionHash`
- The list of signers and their public keys
- The list of observers and their user IDs
- Creator key and associated user identity
- All comments

In an organizational deployment this constitutes unauthorized cross-tenant data disclosure: one organization's members can read another organization's completed transaction history in full.

## Likelihood Explanation
The precondition is a valid JWT — any registered user. Transaction IDs are sequential integers, making enumeration trivial. The terminal-status population grows monotonically; in any production system the vast majority of stored transactions are historical. No special timing, race condition, or privileged role is needed.

## Recommendation
Remove the unconditional early-return for terminal statuses in `verifyAccess`. Apply the same user-relationship check (creator, signer, observer, approver) regardless of transaction status. If the product requirement is that historical transactions should be broadly visible, that policy should be expressed explicitly and scoped (e.g., visible only within the same organization/tenant), not implemented as a blanket bypass of all access control.

// Remove lines 789–798:

return (

  transaction.creatorKey?.userId === user.id ||
  !!transaction.observers?.some(o => o.userId === user.id) ||

## Proof of Concept
1. Register two accounts, User A and User B, in the same deployment.
2. As User A, create and execute a transaction (status becomes EXECUTED). Note its integer ID, e.g. `42`.
3. As User B (no relationship to transaction 42), call the transaction-detail endpoint:
   ```
   GET /api/transactions/42
   Authorization: Bearer <User B JWT>
   ```
4. The response returns the full transaction object including `transactionBytes`, signers, observers, comments, and creator identity — despite User B having no relationship to the transaction.
5. Repeat for IDs 1, 2, 3, … to enumerate all historical transactions across all tenants. [6](#0-5)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,

      !!transaction.approvers?.some(a => a.userId === user.id)
