Audit Report

## Title
Unauthorized Signature Injection via Missing Transaction Access Control in `uploadSignatureMaps`

## Summary
The `uploadSignatureMaps` endpoint (`POST /transactions/:transactionId/signers`) allows any authenticated, verified user with at least one registered key to inject their cryptographic signature into any active transaction in the system. The `validateAndProcessSignatures` private method in `signers.service.ts` performs no transaction-level authorization check, in direct contrast to the parallel `importSignatures` path in `transactions.service.ts`, which correctly calls `verifyAccess` before processing.

## Finding Description

**Root Cause**

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, the `validateAndProcessSignatures` method performs only three checks:

1. The transaction exists in the DB
2. The transaction status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`
3. The public key in the submitted `SignatureMap` belongs to the requesting user's registered keys [1](#0-0) 

The key ownership check at lines 244–248 only verifies the submitted key is in the user's own key set — it does not verify the user has any legitimate relationship to the target transaction: [2](#0-1) 

There is no check that the user is the creator, a designated signer, an observer, or an approver of the transaction.

**Contrast with `importSignatures`**

The parallel endpoint `POST /transactions/signatures/import` in `transactions.service.ts` correctly calls `this.verifyAccess(transaction, user)` before any processing: [3](#0-2) 

The `verifyAccess` function enforces that the user must be the creator, observer, signer, approver, or hold a key required by the transaction: [4](#0-3) 

**Controller Layer**

The controller applies only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) — no transaction-ownership guard exists at this layer: [5](#0-4) 

## Impact Explanation

An attacker who is a normal authenticated user with at least one registered key can:

1. **Inject their signature into any active transaction**, permanently mutating the stored `transactionBytes` in the database via `bulkUpdateTransactions`. This is a direct integrity violation of the transaction record. [6](#0-5) 

2. **Create a fraudulent `TransactionSigner` record** linking their user ID to a transaction they have no legitimate role in, corrupting the audit trail and notification system. [7](#0-6) 

3. **Advance transaction status**: `processTransactionStatus` is called after every upload. If the attacker's key satisfies a required signing key in the transaction's Hedera key structure (e.g., the attacker controls a key that is part of a threshold key on the paying account), the transaction can be moved to `WAITING_FOR_EXECUTION` and subsequently submitted to the Hedera network — causing unauthorized fund movement or account state changes. [8](#0-7) 

4. **Inflate transaction size**: Repeated signature injection can push `transactionBytes` past the Hedera network's size limit, causing the transaction to fail permanently with `TRANSACTION_OVERSIZE`.

## Likelihood Explanation

The preconditions are minimal:
- A valid account on the organization server (normal user, no admin role required)
- At least one registered key pair (required to use the system at all)
- Knowledge of a target transaction ID (sequential integers; existence is confirmable via the difference in error codes between `TNF` and `TNRS`/`TE` in the response)

No privileged access, no leaked credentials, and no cryptographic break are required. The endpoint is reachable over HTTPS by any verified user.

## Recommendation

Add a `verifyAccess` call inside `validateAndProcessSignatures` (or in `uploadSignatureMaps` before delegating) in `signers.service.ts`, mirroring the pattern already used in `importSignatures`:

```typescript
// In validateAndProcessSignatures, after confirming the transaction exists:
if (!(await this.transactionsService.verifyAccess(transaction, user))) {
  return { id, error: ErrorCodes.TNF };
}
```

This requires injecting `TransactionsService` (or extracting `verifyAccess` into a shared utility) into `SignersService`. The `loadTransactionData` method should also be updated to eagerly load the `creatorKey`, `observers`, `signers`, and `approvers` relations (as `importSignatures` does at line 509) so that `verifyAccess` has the data it needs without additional queries. [9](#0-8) 

## Proof of Concept

1. Register as a normal user (User A) and upload one key pair via `POST /user/:userId/keys`.
2. Enumerate transaction IDs by submitting `POST /transactions/:id/signers` with an empty body and observing whether the response error is `TNF` (transaction not found) vs. `TNRS`/`TE` (transaction exists but wrong status/expired). Both non-`TNF` responses confirm the transaction exists.
3. Obtain the target transaction's `transactionBytes` (e.g., via `GET /transactions/:id` if the user can view it, or by any other means). Construct a valid `SignatureMap` by signing the transaction body bytes locally with User A's private key using the open-source Hedera SDK.
4. Submit `POST /transactions/:transactionId/signers` with the crafted `UploadSignatureMapDto` payload containing User A's public key and the computed signature.
5. `validateAndProcessSignatures` accepts the payload because the public key is in User A's registered key set (`userKeyMap`).
6. `processTransactionSignatures` calls `sdkTransaction.addSignature(publicKey, map)`, mutating the transaction bytes.
7. The mutated `transactionBytes` are written back to the database via `bulkUpdateTransactions`, and a `TransactionSigner` record is inserted linking User A's ID to the target transaction.
8. `processTransactionStatus` is invoked, potentially advancing the transaction's status if User A's key satisfies a required signing threshold. [10](#0-9)

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
