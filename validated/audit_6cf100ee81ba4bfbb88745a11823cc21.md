### Title
IP-Based Rate Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header

### Summary
The global `IpThrottlerGuard` — the sole rate-limiting mechanism applied to every API endpoint — derives its throttle key directly from the raw `req.headers['x-forwarded-for']` value. Because this header is fully attacker-controlled, any HTTP client can rotate arbitrary values in it to generate a fresh throttle key per request, completely bypassing the configured rate limits and enabling unbounded resource exhaustion against the API.

### Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`:** [1](#0-0) 

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;   // raw, attacker-supplied header used as throttle key
}
```

`req.headers['x-forwarded-for']` is a plain HTTP request header. Any HTTP client — including an unauthenticated attacker — can set it to any arbitrary string. The NestJS throttler uses the return value of `getTracker()` as the Redis key for the rate-limit counter. A different string means a different counter, so rotating the header value resets the counter on every request.

**This guard is registered as the global `APP_GUARD`**, meaning it is the only rate-limiting layer protecting every controller in the application: [2](#0-1) 

Only `IpThrottlerModule` and `EmailThrottlerModule` are imported; `UserThrottlerModule` is defined but **never imported** and `UserThrottlerGuard` is **never registered**, so no per-user fallback exists: [3](#0-2) 

In production the application calls `app.enable('trust proxy')`, which makes Express populate `req.ip` correctly from the proxy chain — but the guard ignores `req.ip` whenever `x-forwarded-for` is present, preferring the raw, unsanitised header: [4](#0-3) 

The unit tests confirm the guard accepts any string in the header without validation: [5](#0-4) 

**Affected endpoints (all controllers, examples):**

- `POST /auth/login` — credential brute-force, OTP exhaustion
- `POST /auth/reset-password` — OTP spam / email flooding
- `POST /transactions` — transaction creation flood, DB exhaustion
- `POST /notifications/remind-signers` — email spam (its per-IP `@Throttle` override is equally bypassable) [6](#0-5) [7](#0-6) 

### Impact Explanation
An unauthenticated attacker can send an unlimited number of requests to any endpoint by cycling the `X-Forwarded-For` header (e.g., `10.0.0.1`, `10.0.0.2`, …). This exhausts server CPU, PostgreSQL connection pool, and Redis write capacity, rendering the API unavailable or severely degraded for legitimate users. Sensitive endpoints such as `/auth/login` and `/auth/reset-password` become open to unlimited brute-force and OTP-spray attacks as a secondary consequence.

**Impact: 3/5** — Service degradation/unavailability; no direct asset theft, but authentication controls are undermined.

### Likelihood Explanation
No privileges, credentials, or special network position are required. Any HTTP client (e.g., `curl`, Python `requests`) can set arbitrary headers. The bypass is trivial to script and requires no knowledge of the application internals. The `X-Forwarded-For` header is a standard HTTP header accepted by every web framework by default.

**Likelihood: 4/5** — Trivially reachable by any external attacker.

### Recommendation

1. **Use `req.ip` exclusively** (which Express populates correctly when `trust proxy` is enabled) and strip any direct reliance on the raw `x-forwarded-for` header in `getTracker()`:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip; // Express resolves this correctly with trust proxy
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

2. **Import and register `UserThrottlerModule`/`UserThrottlerGuard`** as a secondary guard for authenticated endpoints so that per-user limits apply independently of IP. [8](#0-7) 

3. **Validate the `X-Forwarded-For` header** at the proxy/ingress layer (e.g., Nginx, Kubernetes ingress) to strip client-supplied values before they reach the application.

### Proof of Concept

```bash
# Baseline: normal requests are rate-limited after GLOBAL_MINUTE_LIMIT hits
for i in $(seq 1 200); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<api-host>/auth/login \
    -H "Content-Type: application/json" \
    -d '{"email":"victim@example.com","password":"wrong"}'
done
# Expected: 429 after GLOBAL_MINUTE_LIMIT requests

# Bypass: rotate X-Forwarded-For to get a fresh counter each request
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<api-host>/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.${i%256}.$((i%256))" \
    -d '{"email":"victim@example.com","password":"wrong"}'
done
# Observed: all responses are 200 or 401 — never 429
# Result: unlimited login attempts, unlimited OTP requests, unlimited DB writes
```

Each iteration uses a unique `X-Forwarded-For` value, producing a unique Redis throttle key, so the counter never reaches the configured limit. The attack requires no authentication and no privileged access.
