## Audit Report

## Title
Missing Cross-Object Authorization in `removeTransactionApprover`: Any Transaction Creator Can Delete Approvers from Transactions They Don't Own

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId`, but then deletes the approver record identified by `:id` without verifying that `:id` belongs to `:transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description
In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two sequential calls:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction referenced by `:transactionId`. It does not confirm that the approver record `:id` is associated with `:transactionId`. [2](#0-1) 

`removeTransactionApprover` in the service fetches the approver by `id` alone (no `transactionId` filter) and deletes it unconditionally: [3](#0-2) 

`getTransactionApproverById` queries only by `id` with no `transactionId` constraint: [4](#0-3) 

The sibling `updateTransactionApprover` method correctly performs this cross-check — it fetches the root node from the approver and verifies `rootNode.transactionId === transactionId` before proceeding — but `removeTransactionApprover` omits this entirely: [5](#0-4) 

## Impact Explanation
An attacker can silently remove required approvers from any transaction in the organization. This breaks the multi-signature approval workflow: a transaction that required N approvers can have those approvers stripped, potentially allowing it to proceed with fewer approvals than intended, or corrupting the approval state permanently. This is a direct integrity failure in the trust model of the multi-sig coordination system.

## Likelihood Explanation
The attacker only needs a valid authenticated account and ownership of at least one transaction. No admin privileges, no leaked secrets, and no special network access are required. Approver IDs are sequential integers, making enumeration trivial. This is fully exploitable by any malicious normal user.

## Recommendation
In `removeTransactionApprover` (service), after fetching the approver by `id`, verify that the approver's root node belongs to the expected `transactionId` before deleting — exactly as `updateTransactionApprover` does:

```typescript
// After fetching the approver:
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to the specified transaction');
}
```

The `transactionId` parameter must be threaded through from the controller into `removeTransactionApprover` in the service.

## Proof of Concept
1. Attacker registers as a normal user and creates Transaction A (attacker-owned, `transactionId = 10`).
2. Attacker enumerates or guesses approver ID `42` belonging to Victim's Transaction B (`transactionId = 20`).
3. Attacker sends: `DELETE /transactions/10/approvers/42`
4. `getCreatorsTransaction(10, attacker)` passes — attacker owns Transaction A.
5. `removeTransactionApprover(42)` executes — approver `42` (belonging to Transaction B) is deleted with no cross-check.
6. Transaction B now has a missing approver, breaking its approval threshold requirement.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
