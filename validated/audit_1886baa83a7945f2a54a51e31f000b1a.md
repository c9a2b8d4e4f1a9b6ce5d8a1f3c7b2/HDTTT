### Title
Internal Infrastructure Services Exposed via Host Port Bindings in Production Docker Compose

### Summary
The production Docker Compose file (`back-end/docker-compose.prod.yaml`) binds internal infrastructure services — PostgreSQL, pgAdmin, Redis, and NATS — directly to host-level ports. This is the direct analog of the external report's finding: internal services that should be network-isolated are reachable on the host's network interface. On an internet-facing production host, these ports are accessible from `0.0.0.0` with no firewall restriction enforced at the application layer.

### Finding Description
`back-end/docker-compose.prod.yaml` contains the following host port bindings for services that are purely internal and require no external access:

- **NATS message broker** — `4222:4222`
- **PostgreSQL database** — `5432:5432`
- **pgAdmin web UI** — `5050:80`
- **Redis cache/session store** — `6380:6379` [1](#0-0) [2](#0-1) 

None of these services require direct external access. The API (`3001`) and Notifications (`3020`) services are the only intended public-facing endpoints, and they are correctly exposed. The infrastructure services, however, are bound to the host with no `127.0.0.1` restriction, meaning Docker publishes them on all host interfaces including any public NIC.

A secondary compounding issue exists in `back-end/apps/api/src/setup-app.ts`: CORS is configured with `origin: true`, which reflects any incoming `Origin` header back as an allowed origin, combined with `credentials: true`. This allows any website to make authenticated cross-origin requests to the API. [3](#0-2) 

### Impact Explanation
| Service | Port | Impact if Internet-Exposed |
|---|---|---|
| PostgreSQL | 5432 | Direct brute-force or credential-stuffing against the database; full read/write of all user data, transaction records, and private key metadata |
| pgAdmin | 5050 | Web UI for the database; successful login yields full database control including schema modification |
| Redis | 6380 | JWT blacklist and session data are stored here; an attacker can flush the blacklist to un-revoke tokens, or read cached session state |
| NATS | 4222 | Unauthenticated by default (confirmed by `nats-values.yaml` which has auth commented out); attacker can publish to internal subjects, injecting fake transaction status updates or disrupting the scheduler | [4](#0-3) 

The Redis exposure is particularly severe because the JWT blacklist (`BlacklistService`) is stored there. Flushing Redis invalidates the blacklist, allowing previously revoked tokens to be reused. [5](#0-4) 

### Likelihood Explanation
The file is explicitly named `docker-compose.prod.yaml` and is the designated production deployment artifact. Any operator following the repository's deployment path and running this file on an internet-facing host — without an independent host-level firewall — exposes all four services. Docker's default `iptables` rules bypass `ufw`/`firewalld`, so host-level software firewalls do not protect ports published by Docker. The likelihood is **High** for any deployment not fronted by a dedicated cloud security group or network ACL.

### Recommendation
1. **Remove host port bindings** for all internal services in `back-end/docker-compose.prod.yaml`. Services that only need intra-container communication should have no `ports:` entry at all — Docker's internal network handles service discovery by name.
2. If local admin access to pgAdmin or the database is needed, bind only to loopback: `127.0.0.1:5050:80` instead of `5050:80`.
3. **Enable NATS authentication** by uncommenting and configuring the `accounts` block in the NATS values file.
4. **Restrict CORS** in `setup-app.ts` to an explicit allowlist of known frontend origins rather than `origin: true`.

### Proof of Concept

**Step 1 — Deploy production stack on an internet-facing host:**
```bash
docker compose -f docker-compose.prod.yaml up -d
```

**Step 2 — Verify PostgreSQL is reachable from the internet:**
```bash
psql -h <host-public-ip> -p 5432 -U postgres
# Brute-force or credential-stuff the password
```

**Step 3 — Verify pgAdmin is reachable:**
```
http://<host-public-ip>:5050
# Login with PGADMIN_DEFAULT_EMAIL / PGADMIN_DEFAULT_PASSWORD from environment
```

**Step 4 — Verify NATS is reachable and unauthenticated:**
```bash
nats pub -s nats://<host-public-ip>:4222 "transaction.status" '{"id":1,"status":"EXECUTED"}'
# Injects a fake transaction execution event into the internal message bus
```

**Step 5 — Flush Redis to un-revoke all blacklisted JWTs:**
```bash
redis-cli -h <host-public-ip> -p 6380 FLUSHALL
# All previously invalidated tokens are now valid again
``` [6](#0-5)
