Audit Report

## Title
TOCTOU Race Condition in `approveTransaction` Allows Approval Decision Override

## Summary
`approveTransaction` in `approvers.service.ts` performs a check-then-act sequence without any database-level serialization. Two concurrent HTTP requests from the same approver both read stale in-memory state showing `signature = null`, both pass the double-approval guard, and both proceed to write — with the second write unconditionally overwriting the first. This allows a registered approver to flip their stored approval decision (`approved: true` → `false` or vice versa) in a way the application explicitly tries to prevent.

## Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` executes the following sequence:

**Step 1 — Read (line 553):** Fetches approver records from the database into an in-memory snapshot. [1](#0-0) 

**Step 2 — Guard (line 563):** Checks the in-memory snapshot to prevent double-approval. [2](#0-1) 

**Step 3 — Write (lines 599–609):** Unconditionally overwrites `signature`, `userKeyId`, and `approved` for all matching approver IDs, with no `WHERE signature IS NULL` condition. [3](#0-2) 

Between Step 1 and Step 3 there are multiple `await` points (`attachKeys`, `findOne`, signature verification). In Node.js's event loop, two concurrent HTTP requests are interleaved at these `await` points. Both requests capture the same in-memory snapshot where `signature = null`, both pass the guard at Step 2, and both proceed to Step 3. The second `UPDATE` unconditionally overwrites the first because `whereInIds` carries no conditional predicate on the current `signature` value.

There is no distributed lock, no database row-level lock (`SELECT ... FOR UPDATE`), no optimistic-locking version column, and no unique constraint that would serialize these two writes.

## Impact Explanation

A registered approver sends two concurrent `POST /transactions/:transactionId/approvers/approve` requests — one with `approved: true` and one with `approved: false` — both carrying a valid cryptographic signature (the signature is verified independently per request at line 593–596, and the same valid signature can be reused in both requests). The second request to complete its DB write wins, silently overwriting the first. This allows the approver to:

- Retroactively flip their approval from `true` to `false` after the transaction has reached its approval threshold, causing it to fall below the threshold and be rejected.
- Retroactively flip their approval from `false` to `true` after they have rejected the transaction.

This is an unauthorized state mutation of a live approval record, directly affecting whether a transaction proceeds to execution. [4](#0-3) 

## Likelihood Explanation

The attacker only needs to be a legitimately registered approver for any transaction — a normal, unprivileged role. Sending two concurrent HTTP requests requires nothing beyond a standard HTTP client (`Promise.all`, `curl --parallel`, etc.). No special timing precision is required; the race window spans the entire duration of the async operations between the DB read and the DB write (lines 553–598), which in practice is tens to hundreds of milliseconds — a very wide window. [5](#0-4) 

## Recommendation

Move the duplicate-approval check inside a serialized database transaction with a row-level lock, and add a conditional predicate to the `UPDATE` so it only writes when `signature IS NULL`:

```typescript
await this.dataSource.transaction(async em => {
  // Re-read with a row lock to serialize concurrent requests
  const locked = await em
    .createQueryBuilder(TransactionApprover, 'a')
    .whereInIds(userApprovers.map(a => a.id))
    .setLock('pessimistic_write')
    .getMany();

  if (locked.every(a => a.signature)) {
    throw new BadRequestException(ErrorCodes.TAP);
  }

  await em
    .createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .whereInIds(locked.map(a => a.id))
    .andWhere('signature IS NULL')   // conditional predicate
    .execute();
});
```

This ensures the guard and the write are atomic: the second concurrent request will either block on the row lock until the first commits (and then see `signature IS NOT NULL` and throw `TAP`), or its conditional `UPDATE` will affect zero rows.

## Proof of Concept

```javascript
// Attacker is a legitimately registered approver for transactionId 42.
// They hold a valid private key and produce a valid signature.
const axios = require('axios');
const headers = { Authorization: 'Bearer <APPROVER_JWT>' };
const url = 'https://api.example.com/transactions/42/approvers/approve';
const signature = '<valid_signature_bytes>';

// Fire both requests concurrently before either write completes.
const [r1, r2] = await Promise.all([
  axios.post(url, { userKeyId: 1, signature, approved: true  }, { headers }),
  axios.post(url, { userKeyId: 1, signature, approved: false }, { headers }),
]);

// Both return HTTP 200.
// The second DB write wins; the stored `approved` value is `false`
// even though the approver's "real" intent (first request) was `true`.
// If the transaction had just reached its approval threshold via the
// first write, the second write retroactively drops it below threshold.
```

Both requests pass the `userApprovers.every(a => a.signature)` guard because both read `signature = null` from the same pre-write snapshot. The second `UPDATE ... whereInIds(...)` has no conditional predicate and unconditionally overwrites the first. [6](#0-5)
