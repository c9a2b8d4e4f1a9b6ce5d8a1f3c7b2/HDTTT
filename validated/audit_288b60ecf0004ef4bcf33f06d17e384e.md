Audit Report

## Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header in `IpThrottlerGuard`

## Summary
The global `IpThrottlerGuard` derives its throttle key directly from the raw, attacker-controlled `x-forwarded-for` HTTP header without sanitization or trusted-proxy validation. Any unauthenticated client can rotate arbitrary spoofed IPs in that header to obtain a fresh rate-limit counter on every request, completely nullifying the IP-based throttling layer across the entire API surface.

## Finding Description

**Root cause — `getTracker` reads a raw, attacker-controlled header:**

`IpThrottlerGuard.getTracker` resolves the throttle key as:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [1](#0-0) 

`req.headers['x-forwarded-for']` is the raw HTTP header string. It is never validated, never parsed for the trusted segment, and never compared against a known-good proxy list. Any HTTP client can set it to any arbitrary value.

**Production deployment enables full proxy trust:**

```typescript
app.enable('trust proxy');   // trusts ALL proxies
``` [2](#0-1) 

`trust proxy = true` (boolean form) tells Express to trust every hop in the forwarding chain. Critically, the guard reads `req.headers['x-forwarded-for']` directly — bypassing even the partial protection Express's `req.ip` would provide under a stricter numeric trust-proxy setting.

**The guard is applied globally to every endpoint:**

```typescript
{ provide: APP_GUARD, useClass: IpThrottlerGuard }
``` [3](#0-2) 

The bypass is not limited to one route — it covers the entire API surface.

**`EmailThrottlerGuard` does not compensate:**

The `EmailThrottlerGuard` on `POST /auth/login` keys on `req.body.email`, not IP. [4](#0-3) 

It is a separate, orthogonal control and does not mitigate the IP throttler bypass.

## Impact Explanation

- **Authentication brute force:** `POST /auth/login` is protected by `LocalAuthGuard` + `EmailThrottlerGuard` (email-keyed) and the global IP throttler. With the IP throttler bypassed, an attacker can attempt unlimited passwords against any known email address.
- **OTP brute force / resource exhaustion:** `POST /auth/reset-password` triggers OTP generation. Unlimited calls exhaust email-sending resources and allow OTP enumeration.
- **General DoS:** Every endpoint becomes unthrottled, enabling resource exhaustion of the API service, PostgreSQL connection pool, and Redis.

## Likelihood Explanation

- **No privilege required.** Any unauthenticated HTTP client can set arbitrary headers.
- **Trivially scriptable.** A single `curl` loop incrementing the spoofed IP is sufficient.
- **No detection barrier.** The real source IP never appears in the throttle key, so the attacker's actual IP is never rate-limited.
- **Deployment-confirmed.** The `trust proxy` flag is only set in `createAppForDeployment`, confirming this path is active in production. [5](#0-4) 

## Recommendation

Replace the raw header read with Express's sanitized `req.ip`, which respects the `trust proxy` configuration and returns only the trusted client IP:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

Additionally, change `app.enable('trust proxy')` to a numeric value (e.g., `app.set('trust proxy', 1)`) to trust only the single known reverse-proxy hop, preventing header injection from further upstream. [1](#0-0) 

## Proof of Concept

```bash
# Exhaust the rate limit under IP 1.1.1.1
for i in $(seq 1 100); do
  curl -s -o /dev/null -X POST https://target/auth/login \
    -H "X-Forwarded-For: 1.1.1.1" \
    -H "Content-Type: application/json" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done

# Rotate to a fresh IP — counter resets, unlimited attempts continue
for i in $(seq 1 100); do
  curl -s -o /dev/null -X POST https://target/auth/login \
    -H "X-Forwarded-For: 1.1.1.2" \
    -H "Content-Type: application/json" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
```

Each new `X-Forwarded-For` value produces a fresh throttle key in Redis, giving the attacker an unbounded number of login attempts with no real rate limit applied.
