Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated DoS via `GET /transactions/sign`

## Summary

`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then iterates over the entire result set in a sequential `for` loop, invoking the expensive `userKeysToSign` (which deserializes transaction bytes and makes multiple async account-cache/DB calls) for each row. Pagination is applied only after the full scan completes. Any authenticated, verified user can call `GET /transactions/sign` to trigger this unbounded work, causing request timeouts and Node.js event-loop saturation that degrades the API for all users.

## Finding Description

**Entry point:** `GET /transactions/sign` → `TransactionsController.getTransactionsToSign` → `TransactionsService.getTransactionsToSign`

The controller route is live and protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — the minimum privilege level in the system. No throttler guard is applied to this endpoint. [1](#0-0) 

**Step 1 — Unbounded DB fetch (no `take:`):**

The `whereForUser` filter excludes only terminal statuses. It does **not** filter by the requesting user's identity, meaning it fetches **all** non-terminal transactions in the system. No `take:` is passed to `repo.find()`. [2](#0-1) 

**Step 2 — Sequential per-transaction async work:**

Each transaction in the result set is processed one-by-one via `await this.userKeysToSign(transaction, user)`. [3](#0-2) 

**Step 3 — Pagination applied after full scan:**

`result.slice(offset, offset + limit)` is applied only after all N transactions have been fully processed. [4](#0-3) 

**Per-transaction cost inside `userKeysToSign` → `keysRequiredToSign`:**

Each iteration calls `computeSignatureKey`, which:
- Deserializes `transaction.transactionBytes` via `SDKTransaction.fromBytes()` (CPU + memory)
- Calls `accountCacheService.getAccountInfoForTransaction()` for the fee-payer account, each signing account, and each receiver account (multiple async I/O calls per transaction)
- Issues an `entityManager.find(UserKey, ...)` DB query [5](#0-4) [6](#0-5) 

## Impact Explanation

- **Service degradation / DoS:** With N active (non-terminal) transactions in the system, every call to `GET /transactions/sign` forces the API worker to deserialize N transaction byte blobs, make O(N × accounts_per_tx) async cache/DB calls, and hold all N transaction objects in memory before returning a single page. Repeated calls from one or more authenticated users saturate the Node.js event loop and exhaust the DB connection pool, causing timeouts for all concurrent users.
- **Memory pressure:** All non-terminal transactions are loaded into the process heap simultaneously; large deployments can trigger OOM.
- **No rate-limit or cap:** No `ThrottlerGuard` is applied to the transactions controller. The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`); every other transaction across all users is included regardless of who created it.

## Likelihood Explanation

- **Attacker precondition:** A valid, verified user account — the minimum privilege level in the system. No admin role, no special key, no internal access required.
- **Trigger:** A single HTTP GET to `/transactions/sign` (with any pagination params) is sufficient. Repeated calls amplify the effect.
- **Realistic scenario:** The load-test seed script itself targets 500 sign-transactions per user (`DATA_VOLUMES.SIGN_ALL_TRANSACTIONS = 500`), and the `sign-all.ts` k6 test script confirms this scale is operationally expected. In a multi-user organization, the total non-terminal transaction count grows proportionally with the number of users, making the attack surface larger over time. [7](#0-6) [8](#0-7) 

## Recommendation

1. **Push pagination into the DB query:** Add `skip: offset` and `take: limit` to the `repo.find()` call. This requires a different approach to filtering (see point 2).
2. **Pre-filter in SQL:** Instead of fetching all transactions and then checking `userKeysToSign` in application code, add a DB-level join or subquery to narrow the candidate set to transactions where the user's public keys appear as required signers (e.g., via a `transaction_cached_account` or similar relation). This eliminates the O(N) application-layer loop entirely.
3. **Apply a hard cap:** As an immediate mitigation, add a maximum `take` value (e.g., 100) to the `repo.find()` call to prevent unbounded fetches, even before a full architectural fix is in place.
4. **Remove or gate the endpoint:** Since the controller comment already states "NO LONGER USED BY FRONT-END", consider removing the route entirely or protecting it with an admin-only guard until it is properly optimized.
5. **Add rate limiting:** Apply a `ThrottlerGuard` to the transactions controller, consistent with the throttler infrastructure already present for auth endpoints.

## Proof of Concept

```
# Prerequisites: valid JWT for any verified user account
TOKEN="<valid_jwt>"
BASE_URL="https://<target>/api"

# Single request — triggers full table scan + O(N) async work
curl -s -o /dev/null -w "%{time_total}\n" \
  -H "Authorization: Bearer $TOKEN" \
  "$BASE_URL/transactions/sign?page=1&limit=10"

# Amplified attack — concurrent requests from one user
for i in $(seq 1 20); do
  curl -s -o /dev/null \
    -H "Authorization: Bearer $TOKEN" \
    "$BASE_URL/transactions/sign?page=1&limit=10" &
done
wait
```

With 500+ non-terminal transactions in the system (the documented operational scale), each request forces the server to deserialize 500 transaction byte blobs and make thousands of async DB/cache calls before returning 10 rows. Concurrent requests from a single authenticated user will exhaust the Node.js event loop and DB connection pool, causing timeouts for all other API consumers.

export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
