### Title
Incorrect Boolean Guard in `archiveTransaction` Allows Creator to Overwrite Terminal Transaction States on Manual Transactions

### Summary

`archiveTransaction` in `transactions.service.ts` uses a logically flawed `&&` guard that completely bypasses the status check whenever `isManual` is `true`. As a result, the creator of any manual transaction can archive it regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED` — overwriting the authoritative on-chain outcome with `ARCHIVED` and corrupting the audit trail.

### Finding Description

**Root cause — incorrect boolean guard:** [1](#0-0) 

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← short-circuits the entire guard to false
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** sub-expressions are true:

| `status NOT IN [WFS, WFE]` | `!isManual` | throws? |
|---|---|---|
| true | true | yes (non-manual, wrong status) |
| true | **false** | **no — isManual bypasses status check entirely** |
| false | true | no (correct status) |
| false | false | no (correct status) |

When `isManual === true`, the second operand is `false`, the `&&` short-circuits to `false`, and the guard never fires — regardless of whether the transaction is `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`.

The unconditional write that follows immediately overwrites whatever terminal state the transaction holds: [2](#0-1) 

**Entry point — authenticated HTTP endpoint, no privilege required:** [3](#0-2) 

The only access control is `getTransactionForCreator`, which verifies the caller is the transaction's creator — a normal user role: [4](#0-3) 

**Test gap — the existing e2e test uses a non-manual transaction:** [5](#0-4) 

`addedTransactions.userTransactions[0]` does not have `isManual: true`, so the guard's `!isManual` branch is never exercised. The unit test similarly omits `isManual`: [6](#0-5) 

### Impact Explanation

A transaction creator can overwrite any terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) with `ARCHIVED` on any manual transaction they own. Concretely:

- An `EXECUTED` transaction — one that has already been submitted to and confirmed by the Hedera network — can be relabeled `ARCHIVED`, hiding the on-chain execution from the organization's audit trail and notification history.
- A `FAILED` transaction can be silently buried as `ARCHIVED`, preventing operators from investigating the failure.
- The notification system will emit a spurious `STATUS_UPDATE` event for a transaction already in a terminal state, potentially confusing downstream consumers.

The `TransactionStatus` enum treats `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED` as final states that the rest of the system (scheduler, chain service, notification service) relies on for correctness: [7](#0-6) 

### Likelihood Explanation

- **Attacker preconditions**: authenticated user, creator of at least one manual transaction. No admin or privileged key required.
- **Trigger**: a single `PATCH /transactions/archive/:id` HTTP request after the transaction reaches a terminal state.
- **Discoverability**: the `isManual` flag is set by the creator at creation time and is visible in the transaction response body, so the creator knows exactly which transactions are eligible.
- Manual transactions are an explicitly supported workflow: [8](#0-7) 

Any user who creates a manual transaction and then executes it is in the exact position to trigger this bug.

### Recommendation

Replace the `&&` with `||` so that the guard enforces **both** conditions independently — the status must be in the allowed set **and** (optionally) the transaction may be manual only for the allowed statuses:

```typescript
// Correct: throw if status is not archiveable, regardless of isManual
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If the intent is that manual transactions have a wider set of archiveable statuses, enumerate them explicitly rather than removing the status check entirely. Add a unit test covering `isManual: true` with each terminal status to prevent regression.

### Proof of Concept

1. Authenticate as a normal user (no admin role needed).
2. Create a manual transaction: `POST /transactions` with `isManual: true`.
3. Wait for or trigger execution so the transaction reaches `status = EXECUTED` (or manually set it to `FAILED`/`EXPIRED`/`CANCELED` in a test environment).
4. Call `PATCH /transactions/archive/<id>` as the creator.
5. **Expected (correct)**: `400 Bad Request` — terminal state cannot be archived.
6. **Actual (vulnerable)**: `200 OK` — transaction status is now `ARCHIVED`, overwriting the terminal state.

The unit-test equivalent: mock `getTransactionForCreator` to return `{ isManual: true, status: TransactionStatus.EXECUTED, ... }` and assert that `archiveTransaction` throws `ErrorCodes.OMTIP` — it does not, confirming the bypass.

  async executeTransaction(id: number, user: User): Promise<boolean> {

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }
