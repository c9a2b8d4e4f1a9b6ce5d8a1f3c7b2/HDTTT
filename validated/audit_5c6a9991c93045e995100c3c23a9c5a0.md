### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side DoS via Repeated API Calls

### Summary
`GET /transactions/sign` is backed by `getTransactionsToSign` in `transactions.service.ts`, which fetches **every non-terminal transaction** from the database with no row limit, then iterates over all of them calling `userKeysToSign()` per row before applying pagination in memory. Any authenticated, verified user can repeatedly call this endpoint to exhaust server CPU, memory, and database connection resources, degrading or blocking service for all users.

### Finding Description

The root cause is in `back-end/apps/api/src/transactions/transactions.service.ts` at `getTransactionsToSign`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});                                          // ← no `take`, no `skip`

for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}

return {
  items: result.slice(offset, offset + limit),  // pagination applied AFTER full scan
  ...
};
``` [1](#0-0) 

The `PaginationParams` decorator caps `size` at 100, but this cap is irrelevant — the unbounded `repo.find()` runs before any slicing occurs, loading the entire active transaction table into the Node.js process heap. [2](#0-1) 

The endpoint is exposed at `GET /transactions/sign` and is guarded only by JWT authentication and email verification — no admin role required. [3](#0-2) 

The controller comment even acknowledges the endpoint is `/* NO LONGER USED BY FRONT-END */`, meaning it exists solely as a legacy attack surface with no active monitoring. [4](#0-3) 

### Impact Explanation

Each request to `GET /transactions/sign`:
1. Issues a full-table `SELECT` on the `transaction` table (with `groupItem` JOIN) with no row limit.
2. Deserializes every returned row into TypeORM entity objects in the Node.js heap.
3. Calls `userKeysToSign()` — a cryptographic key-comparison routine — once per row, synchronously blocking the event loop iteration.

As the transaction table grows (normal platform usage), each request becomes progressively more expensive. An attacker sending concurrent requests can:
- Exhaust the PostgreSQL connection pool, blocking all other queries.
- Spike Node.js heap to OOM, crashing the API process.
- Saturate CPU with key-comparison work, causing request timeouts across all endpoints.

This constitutes a **service availability failure** for all organization users.

### Likelihood Explanation

- **Precondition**: A valid JWT token for any verified user account. Registration is a normal user flow.
- **Trigger**: Repeated `GET /transactions/sign?page=1&size=1` requests. No special knowledge needed; the endpoint is documented in `transactions-requests.http`.
- **Amplification**: Impact scales with the number of active (non-terminal) transactions in the system, which grows naturally with platform use.
- **No rate limit** is applied to this endpoint (unlike `remind-signers` which has `@Throttle`). [5](#0-4) 

### Recommendation

Apply the database-level limit **before** the iteration, mirroring the pattern used by every other paginated endpoint in the service:

```typescript
// Replace the unbounded find:
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // ← add DB-level limit
  skip: offset,  // ← add DB-level offset
});
```

Because `userKeysToSign` must be evaluated per row to determine membership, a two-pass approach (fetch a larger batch, filter, then re-page) may be needed for correctness, but the batch size must still be capped (e.g., `MAX_SCAN_BATCH = 1000`). Alternatively, push the key-membership check into the SQL query to eliminate the in-process loop entirely, as done in `getTransactionsToApprove`.

Additionally, apply `@Throttle` to this endpoint and consider removing it entirely since the front-end no longer uses it.

### Proof of Concept

```bash
# Authenticated as any verified user:
TOKEN="<valid_jwt>"

# Fire concurrent unbounded scans:
for i in $(seq 1 50); do
  curl -s -H "Authorization: Bearer $TOKEN" \
    "http://localhost:3001/transactions/sign?page=1&size=1" &
done
wait
```

With a large transaction table, each request loads all rows into memory and iterates them. Fifty concurrent requests will spike heap usage proportionally to `50 × (row_count × avg_row_size)` and saturate the DB connection pool, causing 503/timeout responses for all other API consumers.
