### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Authenticated DoS via Resource Exhaustion

### Summary
`GET /transactions/sign` is backed by `getTransactionsToSign`, which fetches **every** non-terminal transaction from the database with no row limit, then performs an expensive per-row async loop (transaction deserialization + DB query + cryptographic key computation) before applying pagination in memory. As the number of active transactions grows through normal platform use, any authenticated user can trigger unbounded memory, CPU, and database connection consumption with a single request, degrading or crashing the API service for all users.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues an unlimited `repo.find` call: [1](#0-0) 

No `take` (SQL `LIMIT`) is passed, so the ORM returns every row whose status is not in the terminal set. The result set is then iterated one-by-one: [2](#0-1) 

Each iteration calls `userKeysToSign`, which in turn calls `userKeysRequiredToSign`: [3](#0-2) 

`userKeysRequiredToSign` performs:
1. `attachKeys` — a database query to load the user's keys
2. `SDKTransaction.fromBytes` — CPU-bound deserialization of raw transaction bytes
3. `transactionSignatureService.computeSignatureKey` — potentially a mirror-node network call
4. `entityManager.find(UserKey, ...)` — another database query [4](#0-3) 

Pagination (`slice(offset, offset + limit)`) is applied only **after** all N transactions have been processed: [5](#0-4) 

The endpoint is still live and requires only a valid JWT: [6](#0-5) 

The comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is abandoned but not removed, leaving it fully reachable by any API client.

**Exploit path:**
1. Attacker registers a verified account (normal sign-up flow).
2. Attacker (or normal platform growth) accumulates a large number of non-terminal transactions. Any authenticated user can create transactions via `POST /transactions`.
3. Attacker sends `GET /transactions/sign?page=1&size=1` — the `size` parameter is irrelevant because the full scan happens before slicing.
4. The server loads all N transactions into memory, then performs 2–3 async DB queries + deserialization + crypto per transaction, blocking the Node.js event loop and exhausting the DB connection pool.
5. Repeated calls (or a single call on a large dataset) cause request timeouts, OOM, or connection pool starvation for all concurrent users.

### Impact Explanation

- **Memory exhaustion:** All non-terminal transaction rows (including their `transactionBytes` blobs) are loaded into the Node.js heap simultaneously.
- **CPU exhaustion:** `SDKTransaction.fromBytes` is synchronous CPU work executed N times per request.
- **DB connection pool starvation:** Two DB queries are issued per transaction in a sequential `await` loop, holding connections for the full duration.
- **Service-wide degradation:** Because Node.js is single-threaded, a blocked event loop or exhausted DB pool degrades all concurrent API users, not just the attacker.

Severity: **High** — a single authenticated request can render the API unresponsive when the transaction table is large, which is the expected steady state of a production deployment.

### Likelihood Explanation

- Any registered, verified user can reach this endpoint — no admin or privileged role is required.
- Transaction accumulation is a natural consequence of normal platform use; no adversarial seeding is required.
- The endpoint is still registered in the router despite being marked as unused by the front-end, so it is reachable by any HTTP client.
- A single crafted request is sufficient; no sustained flood is needed.

### Recommendation

1. **Apply the database-level limit before processing.** Pass `take: limit` and `skip: offset` directly to `repo.find` and restructure the query so that key-matching is done in SQL (join against `user_key` on public key), eliminating the per-row application-level loop entirely.
2. **Remove or gate the endpoint.** Since the front-end no longer uses `GET /transactions/sign`, either delete the route or protect it with an additional guard that prevents abuse (e.g., a strict rate limiter per user).
3. **Cap the maximum page size.** Enforce a hard upper bound (e.g., 100) on the `limit` parameter in the `PaginationParams` pipe so that even a correctly-implemented scan cannot be forced to return an arbitrarily large result set.

### Proof of Concept

**Preconditions:** A verified user account exists. The database contains a large number of non-terminal transactions (e.g., 10 000 rows in `WAITING_FOR_SIGNATURES` status — achievable by any user with `POST /transactions` access, or simply through normal platform growth).

**Steps:**
```
# 1. Authenticate
POST /auth/login  →  { "accessToken": "<JWT>" }

# 2. Trigger the unbounded scan with a minimal page size
#    (page size does not limit the DB fetch or the per-row loop)
GET /transactions/sign?page=1&size=1
Authorization: Bearer <JWT>
```

**Expected outcome:** The server loads all N non-terminal transactions, executes N×(2 DB queries + deserialization + crypto), exhausts available memory/DB connections, and either returns after a multi-second delay or times out. Concurrent requests from other users fail or queue indefinitely during this window.

**Verification:** Monitor PostgreSQL `pg_stat_activity` during the request — the connection count will spike to the pool maximum and remain there for the duration of the scan. Node.js heap usage (via `process.memoryUsage()`) will grow proportionally to the number of transactions loaded.

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
```
