Audit Report

## Title
Negative `threshold` Persisted via Create Path Due to Missing `@Min(1)` in `CreateTransactionApproverDto` and Incomplete Service Guard

## Summary
`CreateTransactionApproverDto` lacks the `@Min(1)` constraint present on `UpdateTransactionApproverDto.threshold`. The service-level guard in `createTransactionApprovers` only rejects `threshold === 0`, not negative values. A crafted POST request with `threshold: -1` bypasses all validation and writes a negative threshold to the database, corrupting the approval-tree invariant.

## Finding Description

**DTO-level inconsistency:**

`CreateTransactionApproverDto` has no lower-bound constraint on `threshold`: [1](#0-0) 

`UpdateTransactionApproverDto` enforces `@Min(1)`: [2](#0-1) 

**Incomplete service guard:**

The only range check in `createTransactionApprovers` is: [3](#0-2) 

For `threshold = -1` with `approvers = [{userId: 1}, {userId: 2}]`:
- `-1 > 2` → `false`
- `-1 === 0` → `false`

Both branches are false; the guard does not throw.

**`validateApprover` also does not catch negatives:** [4](#0-3) 

This only throws when the approver is entirely empty. With `threshold = -1` and a non-empty `approvers` array, the condition is false.

**Negative threshold is then stored:** [5](#0-4) 

In JavaScript, `-1` is truthy, so `dtoApprover.threshold && dtoApprover.approvers` evaluates to truthy when `approvers` is non-empty, and `threshold: -1` is written to the database.

## Impact Explanation

A negative threshold stored in `transaction_approver.threshold` corrupts the approval-tree invariant in two ways:

1. **Auto-correction branch never fires.** When a child approver is detached, the parent's threshold is supposed to be reduced if it exceeds the remaining child count: [6](#0-5) 
With `threshold = -1`, `newParentApproversLength < -1` is always `false` for any non-negative count, so the correction is silently skipped.

2. **Downstream approval evaluation is trivially satisfied.** Any logic that checks `approvals >= threshold` will pass with zero approvals when `threshold` is `-1`, effectively bypassing the multi-party approval requirement for that tree node.

## Likelihood Explanation

Any authenticated user who is the creator of a transaction can reach the `POST /transactions/:id/approvers` endpoint. The NestJS global `ValidationPipe` will not reject the request because `CreateTransactionApproverDto.threshold` carries no `@Min` constraint. No privileged access beyond being the transaction creator is required. The exploit body is trivial:

```json
{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [{ "userId": 1 }, { "userId": 2 }]
    }
  ]
}
```

## Recommendation

Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// create-transaction-approver.dto.ts
@IsNumber()
@Min(1)        // add this
@IsOptional()
threshold?: number;
```

Additionally, change the service guard from `=== 0` to `< 1` (or `<= 0`) in both `createTransactionApprovers` and `updateTransactionApprover` as defence-in-depth: [7](#0-6) [8](#0-7) 

## Proof of Concept

1. Authenticate as any user who is the creator of a transaction (e.g., `transactionId = 1`).
2. Send:
```http
POST /transactions/1/approvers
Authorization: Bearer <valid_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [{ "userId": 1 }, { "userId": 2 }]
    }
  ]
}
```
3. The request passes DTO validation (no `@Min` on `CreateTransactionApproverDto.threshold`).
4. In `createTransactionApprovers`, the guard at line 305 evaluates `-1 > 2 || -1 === 0` → `false`, so no exception is thrown.
5. The record is inserted with `threshold = -1` in `transaction_approver`.
6. Verify by querying the `transaction_approver` table: the row will have `threshold = -1`.
