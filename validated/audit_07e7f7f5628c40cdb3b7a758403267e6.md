Audit Report

## Title
Public Key Squatting Allows Malicious Organization Member to Permanently Block Legitimate User Key Registration

## Summary
The `uploadKey` method in `user-keys.service.ts` enforces global public-key uniqueness through a non-atomic application-level check-then-insert. Any verified organization member can register a victim's known public key under their own account, permanently preventing the legitimate owner from registering it. No admin recovery path exists to evict a squatted key.

## Finding Description

**Root Cause**

The `UserKey` entity declares `publicKey` with only `@Index()`, not `@Unique()`: [1](#0-0) 

There is no database-level uniqueness constraint. The uniqueness invariant is enforced entirely in application code inside `uploadKey`: [2](#0-1) 

The pattern is:
1. **Read**: `findOne({ where: { publicKey: dto.publicKey }, withDeleted: true })`
2. **Check**: `if (userKey.userId !== user.id) → throw ErrorCodes.PU`
3. **Write**: `repo.create(dto)` + `repo.save(userKey)`

No database transaction, advisory lock, or `INSERT … ON CONFLICT` guard wraps steps 1–3.

**Exploit Flow**

1. Attacker (verified user A) learns victim B's Hedera public key — trivially available from the Hedera network, shared transaction bytes, or via the `GET /users/public-owner/:publicKey` endpoint already in the codebase. [3](#0-2) 

2. Attacker sends `POST /user/:userId/keys` with `{ publicKey: "<victim's key>" }` while authenticated as user A. Because no record exists, `findOne` returns `null`, the `else` branch executes, and the key is saved with `userId = A`. [4](#0-3) 

3. Victim B later calls `POST /user/:userId/keys` with the same `publicKey`. `findOne` returns the record owned by A (including soft-deleted records due to `withDeleted: true`). The ownership check fires: `userKey.userId (A) !== user.id (B)` → `throw new BadRequestException(ErrorCodes.PU)`. Victim's registration is permanently rejected. [5](#0-4) 

**Controller Entry Point**

The endpoint is guarded by JWT + `VerifiedUserGuard` only — no `AdminGuard` required: [6](#0-5) 

The `@GetUser()` decorator binds the key to the authenticated caller's identity, so the attacker cannot impersonate the victim — but they can register the victim's public key under their own account.

**No Admin Recovery Path**

The controller's `DELETE /:id` route calls `removeUserKey`, which enforces ownership: [7](#0-6) 

Only the key owner (the attacker) can remove the squatted key. There is no admin-privileged endpoint to forcibly reassign or evict a key from another user's account.

## Impact Explanation

- The victim cannot register their Hedera public key with the organization backend.
- Without a registered key, the victim cannot be set as a `creatorKey` for any transaction, cannot appear as a required signer, and cannot participate in any multi-signature coordination workflow — the core purpose of the tool.
- The `MAX_USER_KEYS` cap means the attacker can squat multiple keys per victim, potentially blocking all keys the victim might use.
- The state is permanent unless the attacker voluntarily removes the squatted key; no admin can force removal.

## Likelihood Explanation

- **Attacker precondition**: Must be a registered, verified organization member. No admin role required. This is the "malicious normal user" profile explicitly listed in `RESEARCHER.md`.
- **Key discovery**: Hedera public keys are public by design and embedded in every signed transaction visible on-chain or shared within the organization workflow. The `GET /users/public-owner/:publicKey` endpoint confirms keys are treated as known identifiers.
- **Attack complexity**: A single authenticated HTTP POST. No timing precision required — the attacker only needs to act before the victim registers.

## Recommendation

1. **Add a database-level unique constraint** on `publicKey` in the `UserKey` entity (change `@Index()` to `@Index({ unique: true })` or add `@Unique()`). This makes the invariant atomic and removes the TOCTOU window.
2. **Scope the uniqueness check to non-deleted records only**, or use `INSERT … ON CONFLICT` semantics so a soft-deleted key owned by a different user can be recovered by the legitimate owner.
3. **Add an admin endpoint** to forcibly reassign or remove a key from any user's account, providing a recovery path if squatting occurs.
4. **Wrap the check-then-insert in a database transaction** with a serializable isolation level or a `SELECT … FOR UPDATE` lock as a defense-in-depth measure.

## Proof of Concept

**Prerequisites**: Two verified organization accounts — attacker (`userA@org.com`) and victim (`userB@org.com`). Victim's Hedera public key is `<VICTIM_PUBLIC_KEY>` (observable from the network or via `GET /users/public-owner/`).

**Step 1 — Attacker squats the key:**
```http
POST /user/1/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "publicKey": "<VICTIM_PUBLIC_KEY>" }
```
Response: `201 Created` — key saved with `userId = attacker_id`.

**Step 2 — Victim attempts registration:**
```http
POST /user/2/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json

{ "publicKey": "<VICTIM_PUBLIC_KEY>" }
```
Response: `400 Bad Request` — `ErrorCodes.PU` ("Public key is already in use").

**Step 3 — Confirm no recovery:**
- Victim cannot remove the key (ownership check blocks it).
- No admin endpoint exists to forcibly remove it.
- Victim is permanently unable to register their own key.
