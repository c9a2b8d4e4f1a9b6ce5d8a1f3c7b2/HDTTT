Audit Report

## Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header Renders Global Throttler Ineffective

## Summary
The global `IpThrottlerGuard` registered as `APP_GUARD` derives its throttle tracking key from the client-supplied `X-Forwarded-For` header, which takes priority over `req.ip`. Because any HTTP client can forge this header, an attacker can rotate the tracking key on every request and never accumulate hits against the configured limits, completely bypassing the only global rate-limiting protection.

## Finding Description

`IpThrottlerGuard` is registered as a global `APP_GUARD` in `api.module.ts`, making it the sole global rate-limiting protection for every HTTP route: [1](#0-0) 

The guard's `getTracker()` method resolves the throttle key as follows: [2](#0-1) 

`req.headers['x-forwarded-for']` is evaluated **first** and takes priority over `req.ip`. Because `X-Forwarded-For` is a plain HTTP header, any client can set it to an arbitrary value. The throttler stores hit counts keyed by this value, so an attacker who sends a different `X-Forwarded-For` value on each request (e.g., `1.2.3.1`, `1.2.3.2`, …) is treated as a distinct IP each time and never accumulates hits against the configured limits.

The `IpThrottlerModule` configures two throttlers (`global-minute` and `global-second`) whose limits are read from environment variables: [3](#0-2) 

Both limits are rendered meaningless once the tracking key is attacker-controlled.

`EmailThrottlerGuard` is applied only to three auth endpoints (`/auth/signup`, `/auth/login`, `/auth/reset-password`): [4](#0-3) [5](#0-4) [6](#0-5) 

All other controllers carry no per-route throttle guard beyond the bypassable global IP guard.

`UserThrottlerModule` and `UserThrottlerGuard` are defined but `UserThrottlerModule` is never imported into `api.module.ts` and `UserThrottlerGuard` is never exported from the guards index or applied to any route, so they provide no protection: [7](#0-6) [8](#0-7) [9](#0-8) 

## Impact Explanation

An unauthenticated or authenticated attacker can send unlimited requests to any API endpoint:

- **Authentication brute-force**: `/auth/login` has `EmailThrottlerGuard` keyed on `req.body.email`, but the global IP guard is bypassed. An attacker targeting many different email addresses faces no IP-level restriction.
- **OTP/reset-password flooding**: `/auth/reset-password` triggers email dispatch on each call. With IP throttling bypassed and email throttling bypassable by rotating target emails, an attacker can cause mass email spam and exhaust email-sending quotas.
- **Transaction and signature endpoint DoS**: `POST /transactions`, `POST /transactions/:id/signers`, `POST /transaction-groups`, and all other business-logic endpoints have no effective rate limiting. An attacker can flood these endpoints to exhaust database connections, CPU, and memory.
- **Unbounded resource creation**: Authenticated users can create unlimited transactions, signature maps, and approver records, causing database bloat and degraded performance.

## Likelihood Explanation

The precondition is zero: setting an HTTP header requires no privilege, no leaked credential, and no special tooling. Any HTTP client (`curl`, Python `requests`, `k6`) can rotate `X-Forwarded-For` values in a loop. The attack is fully scriptable and deterministic.

## Recommendation

1. **Do not trust `X-Forwarded-For` directly.** If the application sits behind a trusted reverse proxy (nginx, AWS ALB, Cloudflare), configure NestJS/Express to use `app.set('trust proxy', 1)` (or the appropriate proxy count), which causes `req.ip` to be populated with the real client IP extracted from `X-Forwarded-For` by the framework after validating the proxy chain. Then change `getTracker()` to use only `req.ip`:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

2. **Import and apply `UserThrottlerModule`/`UserThrottlerGuard`** for authenticated endpoints to add a per-user rate limit layer that cannot be bypassed by header manipulation.

3. **Enforce rate limiting at the infrastructure layer** (reverse proxy or API gateway) as a defense-in-depth measure, so limits are applied before requests reach the application.

## Proof of Concept

```bash
# Bypass global IP throttler by rotating X-Forwarded-For on every request
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://target/auth/reset-password \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 1.2.3.$((i % 255))" \
    -d '{"email": "victim@example.com"}'
done
```

Each request presents a unique `X-Forwarded-For` value, so the throttler treats each as a new IP and never triggers the rate limit. The server will process all 10,000 requests, dispatching an OTP email for each, exhausting the email-sending quota and flooding the victim's inbox.
