Audit Report

## Title
Unauthenticated Public Key Ownership Claim Allows Key Squatting in `uploadKey()`

## Summary
Any verified organization member can register an arbitrary Hedera public key as their own without proving possession of the corresponding private key. Because `uploadKey()` performs no cryptographic ownership verification, a malicious user who learns a victim's public key can register it first, permanently preventing the legitimate owner from associating that key with their account.

## Finding Description
The root cause is in `uploadKey()` in `back-end/apps/api/src/user-keys/user-keys.service.ts`.

When a new public key is submitted, the service queries the database for an existing record matching `dto.publicKey`: [1](#0-0) 

If no record exists, the key is unconditionally assigned to the caller with no challenge, no signed message, and no proof that the caller controls the corresponding private key: [2](#0-1) 

The only cross-user conflict guard fires **after** a key is already registered — it cannot prevent a race to register first: [3](#0-2) 

The controller endpoint `POST /user/:userId?/keys` is protected only by JWT authentication guards, meaning any verified organization member is a valid attacker: [4](#0-3) 

Public keys are discoverable by any authenticated user via `getUserKeysRestricted()`, which returns `publicKey` for any `userId`: [5](#0-4) 

## Impact Explanation
Once a key is squatted, the victim's every attempt to register their own key is rejected by the `ErrorCodes.PU` check at line 52, because the key is already owned by a different `userId`. The victim is permanently blocked from associating their key with their account. [6](#0-5) 

The `removeUserKey()` function enforces ownership — a user can only remove their own keys — so the victim cannot evict the squatted record: [7](#0-6) 

There is no admin-specific key-removal endpoint in the controller; the only admin-level remediation path is removing the attacker's entire user account, which soft-deletes all their keys: [8](#0-7) 

In Organization Mode, `UserKey` records are the mechanism by which users participate in multi-signature threshold workflows. A user whose key is squatted cannot participate in signing, effectively locking them out of the core product function without admin intervention.

## Likelihood Explanation
- **Attacker precondition**: Must be a verified organization member. No admin privileges required — any user who has completed the sign-up flow qualifies.
- **Key discovery**: Hedera public keys are not secret. They are returned by `getUserKeysRestricted()` for any `userId`, appear in Hedera mirror node account info queries, and are present in transaction records.
- **Timing**: Unlike mempool front-running, this is a simple first-write-wins registration with no time pressure. The attacker can register a key at any time before the victim does. [9](#0-8) 

## Recommendation
Implement a cryptographic proof-of-possession challenge before accepting a public key registration:

1. **Challenge-response**: Before accepting `POST /user/:userId?/keys`, issue a server-generated nonce to the client. Require the client to sign the nonce with the private key corresponding to `dto.publicKey`. Verify the signature server-side using the submitted public key before persisting the record.
2. **Alternatively**, require the client to sign a well-known message (e.g., the authenticated user's ID or email concatenated with a timestamp) and include the signature in `UploadUserKeyDto`. Reject any upload where `PublicKey.verify(signature, message)` fails.

This ensures only the holder of the private key can register the corresponding public key, eliminating the squatting attack entirely. [10](#0-9) 

## Proof of Concept

**Setup**: Two verified organization members — `attacker` (user A) and `victim` (user B).

**Step 1 — Key discovery**: Attacker calls `GET /user/:victimId/keys` to retrieve the victim's registered or soon-to-be-registered public key. Alternatively, the attacker queries the Hedera mirror node for the victim's account public key. [5](#0-4) 

**Step 2 — Key squatting**: Before the victim registers the key, the attacker sends:
```
POST /user/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "publicKey": "<victim_public_key_hex>" }
```
The service finds no existing record for this key and unconditionally assigns it to the attacker: [2](#0-1) 

**Step 3 — Victim is permanently blocked**: When the victim later attempts to register their own key with the same `POST /user/keys` request, the service finds the existing record owned by the attacker (`userKey.userId !== user.id`) and throws `BadRequestException(ErrorCodes.PU)`: [3](#0-2) 

The victim cannot remove the squatted record (ownership check blocks them), and no admin key-removal endpoint exists. The victim is locked out of multi-signature workflows until an admin removes the attacker's account entirely.
