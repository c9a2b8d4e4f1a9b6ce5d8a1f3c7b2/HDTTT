I have verified all code references in `back-end/apps/api/src/transactions/approvers/approvers.service.ts`. The vulnerability is real, but the exploit flow in the submission contains one material inaccuracy that must be corrected before issuing the report.

**Inaccuracy in the submitted exploit flow:** Step 3 (soft-deleting Alice's row) actually *prevents* the signature copy. `getApproversByTransactionId` filters with `WHERE "deletedAt" IS NULL`, so a soft-deleted row is invisible to the copy block. The deletion step is unnecessary and counterproductive. The correct exploit skips deletion entirely — Alice's original signed row remains active and is found by the query, causing the copy.

# Audit Report

## Title
Stale Approval Signature Carried Over When Re-Adding an Approver at a Different Tree Position, Allowing a Single Signature to Satisfy Multiple Approval Slots

## Summary
In `createTransactionApprovers`, when a user is added as an approver at a tree position that does not already exist, the code fetches any existing **active** (non-soft-deleted) approver records for that user on the same transaction and copies their `signature`, `userKeyId`, and `approved` fields into the new record. Because the duplicate-check (`isNode`) is position-specific — it matches on `listId + userId + transactionId` — a malicious transaction creator can add the same already-signed user at one or more additional positions in the tree without removing the original. Each new position receives the copied signature and `approved=true`, allowing a single genuine signature to satisfy multiple approval slots simultaneously.

## Finding Description

**Root cause — `approvers.service.ts` lines 318–329:**

```typescript
if (typeof dtoApprover.userId === 'number') {
  const userApproverRecords = await this.getApproversByTransactionId(
    transactionId,
    dtoApprover.userId,
    transactionalEntityManager,
  );

  if (userApproverRecords.length > 0) {
    data.signature = userApproverRecords[0].signature;
    data.userKeyId = userApproverRecords[0].userKeyId;
    data.approved = userApproverRecords[0].approved;
  }
}
``` [1](#0-0) 

`getApproversByTransactionId` returns all **active** (non-soft-deleted) approver rows for the given `(transactionId, userId)` pair, regardless of their position in the tree: [2](#0-1) 

The duplicate guard `isNode` only blocks re-adding a user at the **exact same position** (same `listId`, same `transactionId`). Adding the same user at a different `listId` produces a count of 0 and returns `false`: [3](#0-2) 

Because `isNode` is position-specific, adding the same user at a different `listId` passes the guard, reaches the signature-copy block, and inserts a new row with the old signature already populated. This behavior is even confirmed by the existing test suite: [4](#0-3) 

## Impact Explanation

A malicious transaction creator can bypass the multi-party approval threshold by duplicating a single approver's signature across multiple tree positions. The transaction is submitted to the Hedera network as if all required parties approved it, when in reality only a subset did. This undermines the entire purpose of the multi-signature orchestration model and can result in unauthorized Hedera transactions being executed (e.g., unauthorized token transfers, account updates, or system file changes).

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The required API calls (`POST /approvers`) are standard product flows available to any creator. The only precondition is that at least one approver has already signed. No cryptographic break, no admin access, and no external dependency is required.

## Recommendation

1. **Remove the signature-copy block entirely (lines 318–329).** There is no legitimate reason to pre-populate a brand-new approver row with a signature from a different row. Approval state should only be written by `approveTransaction`.
2. **Add a cross-position uniqueness check.** Before inserting a new user-type approver node, verify that no active (non-soft-deleted) row for the same `(transactionId, userId)` pair already exists anywhere in the tree, regardless of `listId`. This prevents the same user from occupying multiple positions simultaneously.
3. **Audit `approveTransaction`'s bulk-update behavior.** The `whereInIds(userApprovers.map(a => a.id))` pattern (line 608) updates all rows for a user at once. If the cross-position uniqueness check above is enforced, this becomes safe; without it, it amplifies the impact of the duplication. [5](#0-4) 

## Proof of Concept

**Setup:** Transaction T, threshold-2 root node → [Alice (`listId=root`), Bob (`listId=root`)].

1. Alice calls `POST /transactions/T/approvers/approve` → her row (`listId=root`) gets `signature=S`, `approved=true`.
2. Creator calls `POST /transactions/T/approvers` with body `{ approversArray: [{ threshold: 1, approvers: [{ userId: Alice }], listId: root }] }`.
   - `isNode` checks for `listId=root, userId=null, threshold=1` → 0 rows → **passes**.
   - A new threshold node (threshold=1) is inserted as a third child of root.
   - Recursively, `createApprover` is called for Alice with `listId=newThresholdNode`.
   - `isNode` checks for `listId=newThresholdNode, userId=Alice` → 0 rows → **passes**.
   - `getApproversByTransactionId(T, Alice)` returns Alice's active row at `listId=root` with `signature=S, approved=true`.
   - New row inserted: `{ userId=Alice, listId=newThresholdNode, signature=S, approved=true }`.
3. State: root threshold node (threshold=2) now has 3 children — Alice (approved), Bob (not approved), new threshold node (threshold=1, approved because its only child Alice is approved). Two of three children are approved → threshold of 2 is satisfied.
4. Transaction advances to `WAITING_FOR_EXECUTION` without Bob ever signing. [6](#0-5)
