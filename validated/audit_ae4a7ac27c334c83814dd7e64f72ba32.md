Audit Report

## Title
Mutable Approver Structure Allows Transaction Creator to Nullify Rejections After Approvals Are Submitted

## Summary
`updateTransactionApprover()` and `removeTransactionApprover()` in `approvers.service.ts` allow the transaction creator to modify the approver tree — including swapping approver identities and removing approvers — at any time after approvals or rejections have been submitted. The sole guard, `getCreatorsTransaction()`, verifies only creator identity and performs no transaction status check. A creator can exploit this to silently erase rejections and force a transaction through multi-party approval.

## Finding Description

**Root cause**: `getCreatorsTransaction()` is the only authorization check applied before structural mutations to the approver tree. It fetches the transaction and verifies `creatorKey.userId === user.id`, but performs no check on `transaction.status`: [1](#0-0) 

In `updateTransactionApprover()`, when a `userId` replacement is requested, the existing approval state is **explicitly cleared** — `signature`, `userKeyId`, and `approved` are all set to `undefined` — with no check on whether the approver had already voted: [2](#0-1) 

The threshold can also be lowered at any time, with no status guard: [3](#0-2) 

`removeTransactionApprover()` contains no status check whatsoever: [4](#0-3) 

The controller confirms `getCreatorsTransaction` is the only guard before removal: [5](#0-4) 

And before update: [6](#0-5) 

By contrast, `approveTransaction()` correctly enforces a status check before accepting votes: [7](#0-6) 

This asymmetry means votes can be cast only on active transactions, but the approver tree can be mutated at any time — including after votes are in.

**Exploit flow**:
1. Creator creates a transaction with approvers A, B, C and threshold = 2.
2. User A approves (`approved = true`). User B rejects (`approved = false`).
3. Creator calls `PATCH /transactions/:txId/approvers/:approverId` with `{ userId: userD }` to replace User B. The record is updated: `signature = null`, `userKeyId = null`, `approved = null`. User B's rejection is silently erased.
4. User D approves.
5. Threshold of 2 is now met with User A and User D. The transaction proceeds as if it was never rejected.

Alternatively, the creator calls `DELETE /transactions/:txId/approvers/:approverId` to remove User B entirely, reducing the approver pool and potentially satisfying the threshold with only the remaining approvals.

## Impact Explanation

The approval system is the primary governance mechanism preventing a transaction creator from unilaterally executing transactions that require organizational consensus. By swapping out or removing rejecting approvers, the creator can force any transaction through regardless of how many approvers reject it.

Concrete impact:
- A creator can execute a transaction that was explicitly rejected by designated approvers.
- The rejection history is silently erased — no audit trail of the removed approver's decision is preserved.
- Organizational multi-party approval policies are rendered unenforceable.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges. The attack requires only standard API calls (`PATCH` or `DELETE` on approver endpoints) that the creator is already authorized to make. No leaked credentials, no admin access, and no race conditions are required. The creator can observe rejections in real time via the notification system (the `emitTransactionStatusUpdate` call in `approveTransaction`) and respond immediately.

## Recommendation

Add a transaction status check at the beginning of both `updateTransactionApprover()` and `removeTransactionApprover()` (or inside `getCreatorsTransaction()` when called from these paths). Structural mutations to the approver tree should be rejected if the transaction status is anything other than `NEW` (i.e., before any approver has had the opportunity to vote). Specifically:

- In `getCreatorsTransaction()`, or at the call sites in the controller, verify `transaction.status === TransactionStatus.NEW` before allowing approver tree mutations.
- Alternatively, check whether any approver in the tree has a non-null `approved` field and reject the mutation if so.

## Proof of Concept

```
# Step 1: Creator sets up approvers A (id=10), B (id=11), C (id=12), threshold=2
POST /transactions/42/approvers
{ "approversArray": [{ "threshold": 2, "approvers": [{"userId":10},{"userId":11},{"userId":12}] }] }

# Step 2: User B rejects
POST /transactions/42/approvers/approve  (authenticated as User B)
{ "userKeyId": 5, "signature": "...", "approved": false }

# Step 3: Creator replaces User B with User D — rejection is erased
PATCH /transactions/42/approvers/11  (authenticated as Creator)
{ "userId": 20 }
# Result: approver record 11 now has userId=20, approved=null, signature=null

# Step 4: User D approves
POST /transactions/42/approvers/approve  (authenticated as User D)
{ "userKeyId": 9, "signature": "...", "approved": true }

# Threshold of 2 is now met (User A + User D). Transaction proceeds.
# User B's rejection is gone with no trace.
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
