Audit Report

## Title
Missing Lower-Bound Validation on `reminderMillisecondsBefore` Allows Redis Memory Exhaustion via Unbounded Key TTL

## Summary
`CreateTransactionDto.reminderMillisecondsBefore` accepts any number including negative values. A negative value bypasses the only guard in `createTransactions`, produces a far-future `remindAt` date, and causes `SchedulerService.addReminder` to write a Redis key with an effectively infinite TTL. An authenticated user can repeatedly exploit this to accumulate non-expiring Redis keys and exhaust server memory.

## Finding Description

**Root cause — no `@Min` constraint on `reminderMillisecondsBefore`:**

`CreateTransactionDto` decorates the field with only `@IsOptional()` and `@IsNumber()`, imposing no lower bound: [1](#0-0) 

**Guard bypass — negative values are truthy in JavaScript:**

The only guard before the reminder logic is:

```typescript
if (!dto.reminderMillisecondsBefore) return null;
``` [2](#0-1) 

In JavaScript, `!(-9007199254740991)` evaluates to `false` because negative numbers are truthy. The guard is not triggered, and execution continues.

**Arithmetic inversion — negative value produces a far-future date:**

```typescript
const remindAt = new Date(tx.validStart.getTime() - dto.reminderMillisecondsBefore);
``` [3](#0-2) 

Subtracting a large negative number is equivalent to addition. With `reminderMillisecondsBefore = -Number.MAX_SAFE_INTEGER`, `remindAt` resolves to approximately year ~285,000.

**Sink — Redis key written with far-future `PXAT`:**

`SchedulerService.addReminder` writes the key unconditionally with no validation of the date:

```typescript
await this.pubClient.set(key, key, 'PXAT', date.getTime());
``` [4](#0-3) 

A `PXAT` timestamp centuries in the future means the key never expires under normal operation. The key name is derived from the saved transaction's database ID (`getTransactionSignReminderKey(tx.id)`), so each new transaction submission creates a distinct, immortal key.

**No Redis memory cap configured:**

The deployed `redis.conf` contains no `maxmemory` or `maxmemory-policy` directive, so Redis will consume all available host memory without eviction. [5](#0-4) 

## Impact Explanation

An authenticated user who submits transactions with `reminderMillisecondsBefore: -<large_number>` causes Redis to accumulate non-expiring `schedule:transaction:sign:<id>` keys indefinitely. Because Redis is an in-memory store with no configured `maxmemory` limit, unbounded key growth leads to:

- **Memory exhaustion** on the Redis instance, degrading or crashing the scheduler and notification subsystems for all organization users.
- **Service unavailability** for the Chain and Notifications microservices that depend on the same Redis instance for scheduling.
- **Persistence**: keys remain even after the attacker stops, requiring manual `SCAN` + `DEL` intervention to purge them.

## Likelihood Explanation

The attacker precondition is a valid authenticated account — a normal user role, reachable without any privileged access. The endpoint `POST /transactions` is the standard transaction-creation path used by every organization user. The field name `reminderMillisecondsBefore` is self-documenting. Rate limiting (100 req/min per user) slows but does not prevent accumulation; over hours or days, thousands of immortal keys can be written. The attack is repeatable and persistent. [6](#0-5) 

## Recommendation

1. Add `@Min(1)` (or `@Min(0)`) to `reminderMillisecondsBefore` in `CreateTransactionDto` to reject non-positive values at the validation layer:

```typescript
@IsOptional()
@IsNumber()
@Min(1)
reminderMillisecondsBefore?: number;
``` [1](#0-0) 

2. Add a server-side guard in `createTransactions` to reject or clamp `remindAt` dates that are in the past or unreasonably far in the future before calling `addReminder`.

3. Set a `maxmemory` and `maxmemory-policy` (e.g., `allkeys-lru`) in `redis.conf` as a defense-in-depth measure to prevent unbounded memory growth regardless of application bugs. [7](#0-6)

The key never expires. Repeating this loop (within the 100 req/min rate limit) accumulates immortal keys until Redis memory is exhausted. [8](#0-7)
