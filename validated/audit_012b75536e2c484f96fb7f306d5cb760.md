### Title
Race Condition in `DebouncedNotificationBatcher.flush()` Allows Duplicate Notification Delivery and Message Loss

### Summary
`DebouncedNotificationBatcher` uses a Redis-backed lock in `processExpiration()` to prevent concurrent flushes triggered by key-expiry events, but the `add()` method calls `flush()` directly — without acquiring that lock — when the batch size threshold is reached. Because both code paths are async and can interleave via `await`, two concurrent `flush()` invocations for the same `groupKey` can execute simultaneously, causing the same batch of messages to be read, deleted, and dispatched twice (duplicate notifications) or causing messages added between the `lrange` and `del` calls to be silently dropped.

### Finding Description
`DebouncedNotificationBatcher` is a singleton service that batches notification messages in Redis and flushes them either when a debounce timer expires or when the batch reaches `maxBatchSize`.

**Path 1 — size-triggered flush (no lock):** [1](#0-0) 

When `length >= maxBatchSize`, `flush()` is called directly with no distributed lock.

**Path 2 — expiry-triggered flush (with lock):** [2](#0-1) 

`processExpiration()` acquires a Redis `NX` lock before calling `flush()`. However, this lock only serialises two concurrent `processExpiration()` calls; it does **not** block a concurrent `add()` → `flush()` call.

**The non-atomic read-then-delete in `flush()`:** [3](#0-2) 

Between `lrange` (line 101) and `del(batchKey)` (line 107), the event loop yields on every `await`. A concurrent `add()` call can `rpush` new messages into `batchKey` during that window; those messages are then deleted by `del(batchKey)` but never passed to `flushCallback`, resulting in silent message loss.

**Concurrent double-flush scenario:**

1. `add()` receives the Nth message (`N >= maxBatchSize`), `rpush` returns `length >= maxBatchSize`.
2. `add()` calls `flush()` → `await lrange(batchKey)` → yields.
3. The flush key expires; `processExpiration()` fires → acquires lock (lock is free because `add()` never takes it) → calls `flush()` → `await lrange(batchKey)` → yields.
4. Both `flush()` calls read the same messages.
5. Both call `flushCallback(groupKey, parsedMessages)` → **duplicate notifications sent**.

### Impact Explanation
Duplicate notification delivery causes every affected user to receive the same signing/approval notification more than once for the same transaction. Message loss means some users never receive a notification at all, stalling the multi-signature workflow: transactions remain in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely because required signers are never alerted.

### Likelihood Explanation
The race is triggered by ordinary high-throughput usage — specifically when a large transaction group (>100 accounts, as noted in the file's own comment) causes `maxBatchSize` to be reached at the same time a debounce timer fires. The codebase's own comment acknowledges this scenario explicitly: [4](#0-3) 

No attacker input is required; normal organization-mode usage with large transaction groups is sufficient to trigger it.

### Recommendation
Replace the non-atomic `lrange` + `del` sequence with a Redis `GETDEL`/`LMPOP` or a Lua script that atomically reads and removes the list in a single round-trip. Additionally, extend the distributed lock to cover **all** callers of `flush()`, including the size-triggered path in `add()`, so that at most one flush can execute per `groupKey` at any time:

```typescript
// In add(), before calling flush():
const lockKey = `${this.batchKeyPrefix}${groupKeyStr}:lock`;
const acquired = await this.pubClient.set(lockKey, '1', 'PX', 1000, 'NX');
if (acquired) {
  await this.flush(groupKey);
  await this.pubClient.del(lockKey);
}
```

And in `flush()`, use an atomic pop instead of `lrange` + `del`:

```typescript
// Atomically pop all messages (Redis 7.0+)
const messages = await this.pubClient.lmpop(1, batchKey, 'LEFT', 'COUNT', this.maxBatchSize);
```

### Proof of Concept

1. Create a `DebouncedNotificationBatcher` with `maxBatchSize = 3`, `delayMs = 50 ms`.
2. Add 2 messages (batch not yet full; flush key is set with 50 ms TTL).
3. Wait ~45 ms so the flush key is about to expire.
4. Add a 3rd message: `rpush` returns `length = 3 >= maxBatchSize` → `add()` calls `flush()` → `await lrange` yields.
5. The flush key expires at ~50 ms → `processExpiration()` fires → lock acquired (free) → `flush()` called → `await lrange` yields.
6. Both `flush()` calls complete: `flushCallback` is invoked **twice** with the same 3 messages. [5](#0-4) [6](#0-5)

```typescript
  async add(message: T, groupKey: string | number | null = null): Promise<void> {
    const groupKeyStr = groupKey === null ? this.GLOBAL_KEY : String(groupKey);
    const batchKey = `${this.batchKeyPrefix}${groupKeyStr}`;
    const flushKey = `${this.flushKeyPrefix}${groupKeyStr}`;

    // Add message to Redis list and set TTL for max flush delay
    const length = await this.pubClient.rpush(batchKey, JSON.stringify(message));
    if (length === 1) {
      await this.pubClient.pexpire(batchKey, this.maxFlushMS);
    }

    // If batch size reached, flush immediately, removing the flush key in the process
    if (length >= this.maxBatchSize) {
      await this.flush(groupKey);
      return;
    }

    // If flush key exists, renew expiration; otherwise, set it to trigger flush via keyevent
    const isFlushScheduled = await this.pubClient.get(flushKey);
    if (isFlushScheduled) {
      await this.pubClient.pexpire(flushKey, this.delayMs);
    } else {
      await this.pubClient.set(flushKey, '1', 'PX', this.delayMs);
    }
  }
```

```typescript
  async flush(groupKey: string | number | null): Promise<void> {
    const groupKeyStr = groupKey === null ? this.GLOBAL_KEY : String(groupKey);
    const batchKey = `${this.batchKeyPrefix}${groupKeyStr}`;
    const flushKey = `${this.flushKeyPrefix}${groupKeyStr}`;

    // Retrieve all messages for the group
    const messages = await this.pubClient.lrange(batchKey, 0, -1);
    if (!messages || messages.length === 0) {
      return;
    }

    // Clear both the batch and flush keys
    await this.pubClient.del(batchKey);
    await this.pubClient.del(flushKey);

    // Parse messages and call the flush callback
    const parsedMessages = messages.map((msg) => JSON.parse(msg));
    await this.flushCallback(groupKey, parsedMessages);
  }
```
