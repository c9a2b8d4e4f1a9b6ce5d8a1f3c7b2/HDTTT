Audit Report

## Title
`mnemonicHash` and `index` of All Users' Keys Exposed to Any Authenticated User via `GET /user-keys` Endpoint

## Summary
The `GET /user-keys` endpoint, handled by `UserKeysAllController`, returns the `mnemonicHash` and `index` fields for every user's keys to any verified, authenticated user. A parallel endpoint (`GET /user/:userId/keys`) explicitly suppresses these fields for non-owners, confirming the developers' intent that they are private. The `GET /user-keys` path bypasses this restriction entirely.

## Finding Description

**Root Cause — Two service methods with inconsistent field-level access control:**

`getUserKeysRestricted` (used by `UserKeysController` at `GET /user/:userId/keys`) conditionally hides `mnemonicHash` and `index` based on ownership: [1](#0-0) 

`getUserKeys` (used by `UserKeysAllController` at `GET /user-keys`) performs a plain `findAndCount` with no `select` clause, returning every column for every row: [2](#0-1) 

**Serializer unconditionally exposes the sensitive fields:**

`UserKeyDto` applies `@Expose()` to both `mnemonicHash` and `index` with no conditional logic: [3](#0-2) 

**Controller has no admin guard:**

`UserKeysAllController` is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — any verified user can reach it: [4](#0-3) 

**Developer intent confirmed by e2e test on the restricted path:**

The e2e test for `GET /user/:userId/keys` explicitly asserts that `mnemonicHash` and `index` must not appear when the requester is not the key owner: [5](#0-4) 

No equivalent assertion exists for `GET /user-keys`, and the service code confirms the protection is absent there.

**Entity definition confirms the fields are stored and retrievable:** [6](#0-5) 

## Impact Explanation

Any verified user can retrieve the `mnemonicHash` and `index` for every key registered by every user in the system. `mnemonicHash` is a fingerprint of a user's BIP39 recovery phrase. Exposure enables:

- **Recovery phrase confirmation**: An attacker who obtains a candidate recovery phrase (e.g., via phishing) can compute its hash and compare it to the leaked value to confirm a match — entirely offline, without touching authentication.
- **Key derivation structure disclosure**: Knowing which keys share the same `mnemonicHash` reveals the full key family of every user.
- **Reduced offline attack surface**: The `index` field reveals the exact BIP32 derivation index, narrowing the search space in any offline attack against the recovery phrase.

## Likelihood Explanation

- **Precondition**: A valid, verified account — the standard product registration flow.
- **No elevated privilege required**: The endpoint carries no admin guard.
- **Trivially reachable**: A single authenticated HTTP GET to `/user-keys` (paginated) is sufficient to enumerate all keys system-wide. The front-end client already implements `getAllUserKeys` which iterates all pages automatically: [7](#0-6) 

## Recommendation

Apply the same field-level restriction in `getUserKeys` that exists in `getUserKeysRestricted`. Specifically, add a `select` clause to the `findAndCount` call in `getUserKeys` that excludes `mnemonicHash` and `index` unconditionally (since the caller identity is not scoped to a single user in this endpoint), or introduce a separate DTO — analogous to `UserKeyCoreDto` — that omits those fields and use it as the serializer for `UserKeysAllController` instead of `UserKeyDto`. [8](#0-7) 

## Proof of Concept

```
# 1. Register and verify an account (standard flow).
# 2. Obtain a JWT for the verified account.

curl -H "Authorization: Bearer <JWT>" \
     https://<host>/user-keys?page=1&size=100

# Response includes, for every user in the system:
# {
#   "id": 3,
#   "userId": 7,
#   "publicKey": "...",
#   "mnemonicHash": "<hash of victim's 24-word recovery phrase>",
#   "index": 0
# }
```

Iterating pages exhausts the full `user_key` table. The front-end helper `getAllUserKeys` already does exactly this in a loop.

```typescript
export const getAllUserKeys = async (organizationServerUrl: string): Promise<IUserKey[]> => {
  let page = 1;
  const size = 100;
  let totalItems = 0;
  const allUserKeys: IUserKey[] = [];

  do {
    const { data, error } = await safeAwait(
      getUserKeysPaginated(organizationServerUrl, page, size),
    );
    if (data) {
      totalItems = data.totalItems;
      allUserKeys.push(...data.items);
      page++;
    }

    if (error) {
      break;
    }
  } while (allUserKeys.length < totalItems);

  return allUserKeys;
```
