### Title
`PaginationParams` Decorator Allows `size=0`, Bypassing the 100-Item Limit via TypeORM `take:0` Behavior

### Summary
The `PaginationParams` decorator validates `size` with `size < 0` instead of `size <= 0`, allowing `size=0` to pass all guards. When `size=0` is accepted, `limit=0` is forwarded to TypeORM's `take` option. TypeORM treats `take:0` as "no LIMIT clause," returning every row in the table. Any authenticated user can exploit this to dump all transactions, notifications, or other paginated resources in a single request, bypassing the 100-item cap the code explicitly tries to enforce.

### Finding Description

**Root cause — incorrect boundary in `PaginationParams`:** [1](#0-0) 

```
if (isNaN(page) || page <= 0 || isNaN(size) || size < 0)   // ← size=0 passes
```

`page` is correctly rejected when `<= 0`. `size` is only rejected when `< 0`, so `size=0` is accepted. The subsequent upper-bound guard (`size > 100`) also passes for `0`. The decorator then computes:

```
limit  = 0
offset = (page - 1) * 0 = 0
```

and returns `{ page:1, limit:0, size:0, offset:0 }`.

**Propagation to TypeORM — `take:0` removes the SQL LIMIT clause:**

Every paginated service method passes `limit` directly to TypeORM's `take` option: [2](#0-1) [3](#0-2) [4](#0-3) 

TypeORM omits the `LIMIT` clause from the generated SQL when `take` is `0`, so the query returns every matching row. The same path exists in the notifications service: [5](#0-4) 

**Affected endpoints (all use `@PaginationParams()`):** [6](#0-5) 

Any controller decorated with `@PaginationParams()` is reachable via `?page=1&size=0`.

**Analog to the external report:**
The external report shows `dataStoreId` checked with `<= 0` when `< 0` was intended, making `dataStoreId=1` unreachable. Here the mirror image occurs: `size` is checked with `< 0` when `<= 0` was intended, making `size=0` reachable and exploitable.

### Impact Explanation
An authenticated user (no elevated privileges required) sends `GET /transactions?page=1&size=0` and receives every transaction row visible to them in a single response. For an organization with thousands of transactions this is:
- **Data over-exposure**: full dataset returned instead of a bounded page.
- **Server-side DoS**: unbounded query and response serialization can exhaust memory and CPU, degrading service for all users.
- The 100-item cap (`size > 100`) is completely bypassed.

### Likelihood Explanation
- Requires only a valid session cookie/JWT — any registered user qualifies.
- The attack is a single HTTP GET with a trivially crafted query string (`size=0`).
- No brute-force, timing, or cryptographic work is needed.
- The bug is in a shared decorator applied to every paginated endpoint, so the attack surface is wide.

### Recommendation
Change the boundary check from `size < 0` to `size <= 0`:

```typescript
// back-end/libs/common/src/decorators/pagination-params.decorator.ts
if (isNaN(page) || page <= 0 || isNaN(size) || size <= 0) {
  throw new BadRequestException(ErrorCodes.IPP);
}
``` [7](#0-6) 

This mirrors the correct treatment already applied to `page` and eliminates the `limit=0` / `take:0` path entirely.

### Proof of Concept

**Precondition:** valid authenticated session (any normal user account).

**Request:**
```
GET /transactions?page=1&size=0
Authorization: Bearer <valid_jwt>
```

**Step-by-step:**
1. `PaginationParams` parses `size=0`; `0 < 0` is `false` → guard does not throw.
2. `0 > 100` is `false` → upper-bound guard does not throw.
3. Decorator returns `{ page:1, limit:0, size:0, offset:0 }`.
4. `getTransactions` calls TypeORM with `{ skip:0, take:0 }`.
5. TypeORM generates `SELECT … FROM "transaction" … ORDER BY … ` with **no LIMIT clause**.
6. All rows matching the user's visibility predicate are returned in the response body.

**Expected (correct) behavior:** `size=0` should throw `400 Bad Request` with `ErrorCodes.IPP`, identical to `size=-1`.
