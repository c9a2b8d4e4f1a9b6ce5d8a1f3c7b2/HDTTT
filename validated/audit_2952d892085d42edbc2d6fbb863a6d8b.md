Audit Report

## Title
Unbounded Full-Table Scan with Per-Row Sequential Async Work in `getTransactionsToSign` Causes Server Resource Exhaustion

## Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction in the database without any row limit, then performs a sequential async DB call chain per row before applying pagination. Any authenticated, verified user can trigger `GET /transactions/sign`. As active transactions accumulate, each request consumes unbounded CPU and database connections, degrading or hanging the API service.

## Finding Description

**Root cause — no `take` on the initial query:**

The `whereForUser` filter excludes only a subset of terminal statuses and does not scope to the requesting user's own transactions. It returns every matching transaction in the entire organization: [1](#0-0) 

Additionally, `REJECTED` is present in `terminalStatuses` but is absent from the `whereForUser` exclusion list, meaning rejected transactions are also included in the full scan: [2](#0-1) 

**Unbounded sequential async loop:**

For each of the N returned rows, `userKeysToSign` is awaited sequentially: [3](#0-2) 

Internally, `userKeysRequiredToSign` calls `attachKeys` (a DB call) and then `keysRequiredToSign`, which:
- Deserializes the Hedera SDK transaction from raw bytes (CPU-intensive)
- Calls `transactionSignatureService.computeSignatureKey(transaction)` — an async call that itself invokes `accountCacheService.getAccountInfoForTransaction` per signing account [4](#0-3) [5](#0-4) 

**Pagination applied only after full scan:** [6](#0-5) 

The caller receives one page of results, but the server has already processed every active transaction.

**Exposed endpoint — no rate limiting:**

The route `GET /transactions/sign` is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level. No rate-limit decorator or middleware is applied at the handler level. The comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is still live and reachable: [7](#0-6) [8](#0-7) 

## Impact Explanation

Each call to `GET /transactions/sign` causes the server to:
1. Load all N non-terminal transactions from PostgreSQL (full table scan)
2. Execute N sequential async operations per transaction: `attachKeys` (DB), `SDKTransaction.fromBytes` (CPU deserialization), and `computeSignatureKey` (async account/node cache lookups)

With a sufficiently large active-transaction set, a single request can exhaust the Node.js event loop and the PostgreSQL connection pool, causing timeouts for all concurrent users. Because the endpoint is still publicly reachable and requires only a valid JWT, a single malicious authenticated user can trigger this to degrade or hang the API service.

## Likelihood Explanation

- **Attacker precondition:** A valid, verified account in the organization — the lowest privilege level the application grants.
- **Attack vector:** A single HTTP GET request to `GET /transactions/sign`. No special tooling required.
- **Amplification:** The attacker can first create many transactions via `POST /transactions` (no creation limit enforced), then call the sign endpoint. Alternatively, in a busy organization with many legitimate active transactions, the endpoint is already slow without any attacker-created data.
- **No rate limiting** is present on this endpoint.

## Recommendation

1. **Apply a database-level limit before the loop.** Add `take` and `skip` to the initial `repo.find()` call, scoped to transactions where the user's keys are actually required. This requires restructuring the query to filter at the DB level rather than in application code.
2. **Scope the query to the requesting user.** The `whereForUser` filter should include a join or subquery that restricts results to transactions where the user's public keys appear in the required signers, rather than returning all non-terminal transactions.
3. **Add `REJECTED` to the exclusion list** in `whereForUser` to match `terminalStatuses`.
4. **Apply rate limiting** to `GET /transactions/sign` using a NestJS throttler guard or equivalent middleware.
5. **Consider parallelizing** the `userKeysToSign` calls with `Promise.all` as a short-term mitigation to reduce wall-clock time, though this does not address the unbounded scan itself.

## Proof of Concept

```
# Precondition: attacker has a valid JWT for a verified account
# Step 1: Create many transactions (no creation limit enforced)
for i in $(seq 1 1000); do
  curl -X POST https://<api>/transactions \
    -H "Authorization: Bearer <jwt>" \
    -d '<valid_transaction_payload>'
done

# Step 2: Trigger the unbounded scan — a single request
# The server will now deserialize 1000+ transactions and perform
# 1000+ sequential async DB/network calls before returning page 1
curl -X GET "https://<api>/transactions/sign?page=1&limit=10" \
  -H "Authorization: Bearer <jwt>"

# Result: server CPU and DB connection pool are saturated;
# concurrent legitimate requests time out
```

Even without Step 1, in an organization with many naturally accumulated active transactions, Step 2 alone is sufficient to trigger the resource exhaustion.

```typescript
  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
