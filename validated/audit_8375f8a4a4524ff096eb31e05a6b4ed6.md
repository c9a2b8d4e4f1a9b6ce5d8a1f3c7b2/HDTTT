# Audit Report

## Title
Silent Signature Upload Failure in Batch `uploadSignatureMaps` Masked by Flawed Frontend Detection, Causing Transactions to Expire Without User Notification

## Summary
When `uploadSignatureMaps` processes a batch of signature uploads, per-item validation failures (expired transaction, wrong status, not found) are silently logged to console and skipped. The HTTP response always returns HTTP 201 with only the successfully created `TransactionSigner` records and no per-item error information. The frontend `signItems` function then attempts to detect unsigned items using a unit-mismatched comparison (`items.length - signers.length`) that evaluates to zero even when some transactions were silently rejected, completely hiding the failure from the user.

## Finding Description

**Root cause — backend silent skip:**

In `persistSignatureChanges`, any item whose validation produced an error is silently skipped with only a `console.error` and a `continue`:

```typescript
// signers.service.ts lines 282–286
for (const result of validationResults) {
  if (result.error) {
    console.error(`[TX ${result.id}] Validation failed: ${result.error}`);
    continue;
  }
``` [1](#0-0) 

The `validateAndProcessSignatures` method returns `{ id, error: ErrorCodes.TE }` for expired transactions, `{ id, error: ErrorCodes.TNF }` for not-found transactions, and `{ id, error: ErrorCodes.TNRS }` for wrong-status transactions — all of which are silently dropped here. [6](#0-5) 

The outer `uploadSignatureMaps` then returns only the successfully created signers with no error field:

```typescript
// signers.service.ts lines 120–123
return {
  signers: Array.from(signers),
  notificationReceiverIds: notificationsToDismiss,
};
``` [2](#0-1) 

The controller always responds HTTP 201 with no per-item error field: [3](#0-2) 

**Root cause — frontend flawed detection:**

`signItems` tries to detect unsigned items using:

```typescript
// front-end/src/renderer/utils/index.ts lines 260–264
const signers: { transactionId: number }[] = uploadResults?.data?.signers ?? [];
const signedTransactionIds = new Set<number>(signers.map((s) => s.transactionId));
const unsignedCount = items.length - signers.length;
const result: SignatureItem[] = [];
if (unsignedCount > 0) {
``` [7](#0-6) 

`items` is `SignatureItem[]` — one element **per transaction**. `signers` is `TransactionSigner[]` — one element **per key per transaction**. These are different units. The inner loop at lines 266–270 correctly uses `signedTransactionIds` (a set of transaction IDs) to identify unsigned transactions, but it is gated behind the broken `unsignedCount > 0` guard that uses the mismatched lengths. [8](#0-7) 

**Concrete failing scenario:**

| Submitted | Keys | Outcome |
|---|---|---|
| Transaction A | key1, key2 | Succeeds → 2 `TransactionSigner` records |
| Transaction B | key3 | Expired → silently dropped, 0 records |

- `items.length = 2`, `signers.length = 2` → `unsignedCount = 0`
- Inner loop never executes
- Transaction B's failure is invisible to the user
- `signItems` returns `[]`, signaling complete success

The expiry validation that produces the silent error is confirmed at: [5](#0-4) 

## Impact Explanation
A user submits a batch of signatures and receives HTTP 201. They believe all signatures were recorded. Transaction B never receives the required signature. If no other signer covers it, Transaction B expires without being executed. The transaction cannot be re-submitted with the same transaction ID. This is an unrecoverable state loss for the transaction — a permanent freeze of that transaction's lifecycle — caused by a silent failure the user had no way to detect. The `SignAllController` and `SignTransactionController` both rely on `signItems` returning non-empty to surface errors, so both signing workflows are affected. [9](#0-8) [10](#0-9) 

## Likelihood Explanation
This is a realistic race condition requiring no privileged access. Any authenticated user can trigger it through the ordinary signing workflow: fetch a list of transactions to sign, one expires in the window between fetch and submission (Hedera transactions have a short validity window, typically 120–180 seconds), submit a batch that includes the now-expired transaction alongside a multi-key transaction. The flawed `unsignedCount` check masks the failure. The combination of a multi-key success and a single-key failure is the most common real-world batch composition.

## Recommendation

**Backend:** Return per-item error information alongside the successful signers. The response shape should include a `failures` array:

```typescript
return {
  signers: Array.from(signers),
  notificationReceiverIds: notificationsToDismiss,
  failures: validationResults
    .filter(r => r.error)
    .map(r => ({ id: r.id, error: r.error })),
};
```

**Frontend:** Remove the unit-mismatched `unsignedCount` guard entirely. The inner loop already uses `signedTransactionIds` (a set of transaction IDs) which is the correct unit for detecting unsigned transactions. The guard should simply be removed or replaced with a check on the `failures` array from the backend:

```typescript
// Remove the broken guard; always run the loop
for (const item of items) {
  if (!signedTransactionIds.has(item.transactionId)) {
    result.push(item);
  }
}
```

Additionally, if the backend returns `failures`, surface them to the user with a specific error message identifying which transaction IDs failed and why (e.g., "Transaction #42 expired before your signature could be recorded").

## Proof of Concept

1. Authenticate as a normal user with two keys: `key1`, `key2`, `key3`.
2. Create Transaction A requiring `key1` and `key2` (multi-key).
3. Create Transaction B requiring `key3` (single-key) with a short validity window.
4. Fetch both transactions for signing.
5. Wait for Transaction B to expire (past its `validStart + validDuration`).
6. Submit a batch `POST /transactions/signers` with both signature maps.
7. Observe: HTTP 201 is returned. The response `signers` array contains 2 entries (both for Transaction A). The backend logs `[TX <B_id>] Validation failed: TE` to console only.
8. On the frontend: `items.length = 2`, `signers.length = 2`, `unsignedCount = 0`. The inner detection loop never runs. `signItems` returns `[]`.
9. The UI displays "Transaction signed successfully" (from `SignTransactionController` or `SignAllController`).
10. Transaction B never receives its required signature and expires permanently.
