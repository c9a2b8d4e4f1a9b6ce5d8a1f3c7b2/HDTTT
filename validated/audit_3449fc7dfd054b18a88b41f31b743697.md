### Title
Unbounded Loop in `getTransactionsToSign` Causes Server-Side Denial of Service for Any Authenticated User

### Summary
The `getTransactionsToSign` method in `TransactionsService` fetches **all** transactions from the database without any row limit, then iterates over every result performing expensive per-transaction async work (SDK deserialization, signature-key computation, and a database query) before applying pagination. Any authenticated user can repeatedly call `GET /transactions/sign` to exhaust server CPU, memory, and database connections, degrading or halting service for all users.

### Finding Description

**Root cause — no `take` limit on the database fetch:**

In `back-end/apps/api/src/transactions/transactions.service.ts` the query at line 295 has no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `limit` here
});
``` [1](#0-0) 

**Unbounded sequential async loop:**

Every fetched transaction is then processed one-by-one in a `for` loop:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

**Pagination applied only after full processing:**

The `offset`/`limit` from the request are applied only after the entire result set has been computed in memory:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

**Each iteration is expensive:**

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs, per transaction:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive protobuf deserialization.
2. `transactionSignatureService.computeSignatureKey(transaction)` — cryptographic key-tree traversal.
3. `entityManager.find(UserKey, { where: { publicKey: In(flatPublicKeys) } })` — a database round-trip. [4](#0-3) 

**Endpoint is publicly reachable by any authenticated user:**

The controller exposes `GET /transactions/sign` behind only the standard JWT guards — no admin or special role is required: [5](#0-4) 

The endpoint is marked `/* NO LONGER USED BY FRONT-END */` but remains fully active and callable. [6](#0-5) 

Even the developers noted the need for limits elsewhere in the codebase (see the comment at line 575 of `transactions.service.ts`: *"Added a batch mechanism, probably should limit this on the api side of things"*), but no such limit was applied to `getTransactionsToSign`. [7](#0-6) 

### Impact Explanation

- **CPU exhaustion**: Each request triggers N SDK deserializations and N cryptographic key-tree computations, where N is the total number of non-terminal transactions in the system. With thousands of transactions, a single request can peg a CPU core for seconds.
- **Memory exhaustion**: All N transaction entities (including `transactionBytes` blobs) are loaded into the Node.js heap simultaneously.
- **Database connection starvation**: N sequential `entityManager.find(UserKey, ...)` calls hold or re-acquire DB connections for the full duration of the request.
- **Cascading effect**: Concurrent repeated calls from even a single attacker can exhaust the NestJS thread pool and PostgreSQL connection pool, making the API unresponsive for all users.

Severity: **High** — complete service unavailability is achievable by a single authenticated user with no special privileges.

### Likelihood Explanation

- **Attacker precondition**: A valid JWT token (i.e., a registered, verified account). Registration is a normal user flow.
- **Attack complexity**: Trivially low — a simple `GET /transactions/sign` request in a loop, or even a single request on a large deployment.
- **No rate limiting** is visible on this endpoint in the controller or any middleware shown in the codebase.
- The endpoint remains active despite being marked as unused by the front-end, meaning it will not be naturally removed by front-end deprecation.

### Recommendation

1. **Apply the database-level limit before the loop.** Pass `take: limit` and `skip: offset` directly to the `repo.find()` call, and restructure the query to filter at the SQL level (e.g., via a subquery or join) rather than in application memory.
2. **Move the per-transaction key check into the database query** using the same recursive CTE pattern already used in `getTransactions` and `getTransactionsToApprove`, so only transactions the user actually needs to sign are returned.
3. **Enforce a hard maximum page size** (e.g., 100) in the `PaginationParams` decorator to prevent a caller from requesting an arbitrarily large page.
4. **Remove or gate the endpoint** with a deprecation notice or feature flag if it is truly no longer used by the front-end.

### Proof of Concept

**Setup**: An organization with 10,000 active transactions in `WAITING_FOR_SIGNATURES` status (achievable by any user who can create transactions, or naturally over time in a production deployment).

**Attack**:
```bash
# Attacker holds a valid JWT token
TOKEN="eyJ..."

# Single request — forces the server to deserialize and process all 10,000 transactions
curl -H "Authorization: Bearer $TOKEN" \
     "https://api.example.com/transactions/sign?page=1&limit=10"
```

**Expected outcome**:
- The server fetches all 10,000 `Transaction` rows (including `transactionBytes` blobs) into memory.
- It then executes 10,000 sequential iterations of: SDK deserialization → signature key computation → `UserKey` DB query.
- Response time grows linearly with N; at scale the request times out or the Node.js event loop stalls.
- Repeated concurrent calls from the same attacker exhaust the PostgreSQL connection pool, returning 500 errors to all other users.

The attacker receives only `limit=10` results, but the server has done O(N) work — a classic work-amplification DoS.
