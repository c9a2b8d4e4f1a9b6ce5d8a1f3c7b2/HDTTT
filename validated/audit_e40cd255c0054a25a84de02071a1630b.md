### Title
Transaction Creator Can Immediately Manipulate Approval Parameters to Bypass Approval Workflow

### Summary
The transaction creator can modify or remove approvers and lower the approval threshold at any time — including after approvers have already voted — with no transaction-status guard. This is the direct analog of the `changeSpotPrice()`/`changeDelta()` frontrunning issue: a privileged-but-reachable role can immediately change critical parameters that govern a pending operation, invalidating prior participant actions.

### Finding Description

**Vulnerability class:** State transition / privileged parameter manipulation (no-delay, no-status guard)

**Root cause:**

`updateTransactionApprover` and `removeTransactionApprover` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` both delegate their authorization check to `getCreatorsTransaction`, which only verifies that the caller is the transaction creator. Neither function checks the current transaction status before allowing modifications. [1](#0-0) 

`getCreatorsTransaction` performs no status check — it only validates creator identity: [2](#0-1) 

`updateTransactionApprover` calls `getCreatorsTransaction` and then immediately proceeds to mutate threshold, userId, or listId with no status guard: [3](#0-2) 

`removeTransactionApprover` has no status check at all — it fetches the approver and deletes it: [4](#0-3) 

The controller exposes both mutations over HTTP with no additional guards: [5](#0-4) 

The test suite itself confirms that `updateTransactionApprover` is exercised against a transaction in `WAITING_FOR_EXECUTION` status — the state after all required approvals have been collected — with no rejection: [6](#0-5) 

**Exploit paths:**

**Path A — Threshold reduction after partial approval:**
1. Creator creates a transaction with a threshold-tree requiring 3-of-5 approvals.
2. Only 1 approver has approved so far; the transaction is still in `WAITING_FOR_SIGNATURES`.
3. Creator calls `PATCH /transactions/:id/approvers/:approverId` with `{ threshold: 1 }`.
4. The threshold is immediately lowered to 1.
5. The chain service re-evaluates and moves the transaction to `WAITING_FOR_EXECUTION` with only 1 approval, bypassing the 3-of-5 requirement. [7](#0-6) 

**Path B — Remove a rejecting approver:**
1. Creator creates a transaction requiring approval from a compliance officer (User A).
2. User A reviews and rejects the transaction (`approved: false`).
3. Creator calls `DELETE /transactions/:id/approvers/:approverId` to remove User A.
4. `removeTransactionApprover` deletes the approver and emits `emitTransactionStatusUpdate`, triggering a re-evaluation.
5. Creator adds a colluding User B as the new approver, who approves.
6. The transaction executes, bypassing the compliance officer's rejection. [8](#0-7) 

**Path C — Replace an approver, clearing their signature:**
When `updateTransactionApprover` changes `userId`, it explicitly clears the existing approver's signature and approval: [9](#0-8) 

This allows the creator to silently erase a rejection by swapping the approver to a colluding user.

### Impact Explanation

The approval workflow is the core security mechanism of Organization Mode. It is designed to enforce multi-party authorization before a Hedera transaction is submitted to the network. By manipulating threshold or approver membership after votes have been cast, the creator can:

- Execute transactions that were explicitly rejected by designated approvers.
- Reduce a multi-signature requirement to a single approval unilaterally.
- Erase audit evidence of rejections by replacing approvers.

This constitutes unauthorized state change and integrity failure in the trust model — the approval record no longer reflects the actual decisions of the designated approvers.

### Likelihood Explanation

The attacker is any authenticated user who has created a transaction in Organization Mode — a role reachable by any registered user with no special privileges. The API endpoints are standard REST calls requiring only a valid JWT. No race condition or timing precision is required; the creator can make these calls at any point during the transaction lifecycle. The missing status guard is a straightforward omission, not a complex bypass.

### Recommendation

Add a transaction-status guard in both `updateTransactionApprover` and `removeTransactionApprover` (and `createTransactionApprovers`) that rejects modifications once the transaction has progressed past a configurable "lock" status (e.g., `WAITING_FOR_EXECUTION` or after any approver has voted). For example:

```typescript
// In updateTransactionApprover, after getCreatorsTransaction:
const lockedStatuses = [
  TransactionStatus.WAITING_FOR_EXECUTION,
  TransactionStatus.EXECUTED,
  TransactionStatus.CANCELED,
  TransactionStatus.EXPIRED,
];
if (lockedStatuses.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify approvers: transaction is locked');
}
```

Apply the same guard in `removeTransactionApprover` before calling `removeNode`.

### Proof of Concept

**Threshold reduction (Path A):**

```
# Step 1: Creator creates transaction with threshold=3, 5 approvers
POST /transactions
POST /transactions/1/approvers  { approversArray: [{ threshold: 3, approvers: [...5 users] }] }

# Step 2: One approver approves
POST /transactions/1/approvers/approve  { userKeyId: X, signature: ..., approved: true }

# Step 3: Creator immediately lowers threshold to 1 (no status check blocks this)
PATCH /transactions/1/approvers/2  { threshold: 1 }
# → HTTP 200, threshold updated, emitTransactionUpdate fires
# → Chain service re-evaluates: 1 approval >= threshold 1 → moves to WAITING_FOR_EXECUTION
```

**Approver removal after rejection (Path B):**

```
# Step 1: Creator creates transaction with single required approver (User A, id=5)
POST /transactions/1/approvers  { approversArray: [{ userId: 5 }] }

# Step 2: User A rejects
POST /transactions/1/approvers/approve  { userKeyId: X, signature: ..., approved: false }

# Step 3: Creator removes User A (no status check)
DELETE /transactions/1/approvers/1
# → HTTP 200, approver deleted, emitTransactionStatusUpdate fires

# Step 4: Creator adds colluding User B
POST /transactions/1/approvers  { approversArray: [{ userId: 99 }] }

# Step 5: User B approves → transaction executes
```

```typescript
  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
