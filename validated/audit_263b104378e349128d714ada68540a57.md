## Audit Report

## Title
IDOR in Approver Removal: Cross-Transaction Approver Deletion via Missing Ownership Binding

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that the approver actually belongs to that transaction. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two sequential calls:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts, lines 102–113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
``` [1](#0-0) 

Step 1 (`getCreatorsTransaction`) validates that the authenticated user is the creator of the transaction at `:transactionId`. Step 2 (`removeTransactionApprover`) deletes the approver at `:id` with no check that `:id` belongs to `:transactionId`.

The service-level `removeTransactionApprover` accepts only the approver `id` and performs no transaction-ownership check:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts, lines 533–544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

The `updateTransactionApprover` path in the same service correctly performs this check — it fetches the root node and explicitly asserts `rootNode.transactionId !== transactionId` before proceeding:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts, lines 386–391
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The delete path is missing this exact guard.

## Impact Explanation

An attacker who is the creator of any transaction (T1) can delete approvers belonging to any other transaction (T2) they do not own. This directly undermines the multi-signature approval workflow: by removing required approvers from T2, the attacker can reduce or eliminate the approval threshold for a transaction they have no legitimate access to, potentially allowing T2 to proceed to execution without the intended oversight.

## Likelihood Explanation

The route is authenticated (requires a valid JWT via `JwtBlackListAuthGuard` + `JwtAuthGuard` + `VerifiedUserGuard`), so unauthenticated users cannot exploit it. However, any verified user who has ever created a transaction — a normal, expected action — satisfies the only authorization check. The approver `id` values are sequential integers, making enumeration straightforward. The exploit requires a single crafted HTTP DELETE request with no special tooling. [4](#0-3) 

## Recommendation

Pass `transactionId` into `removeTransactionApprover` and, after fetching the approver, resolve its root node and assert that `rootNode.transactionId === transactionId` before calling `removeNode`. This mirrors the guard already present in `updateTransactionApprover`:

```typescript
// approvers.service.ts — removeTransactionApprover (proposed fix)
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Update the controller call accordingly:
```typescript
await this.approversService.removeTransactionApprover(id, transactionId);
```

## Proof of Concept

**Setup:**
- Attacker (User A) is the creator of Transaction `T1` (id = 1).
- Victim (User B) is the creator of Transaction `T2` (id = 2), which has an approver record with id = `99`.

**Exploit:**
```http
DELETE /transactions/1/approvers/99 HTTP/1.1
Authorization: Bearer <User A's valid JWT>
```

**Execution trace:**
1. `getCreatorsTransaction(1, userA)` — passes; User A is creator of T1. [5](#0-4) 
2. `removeTransactionApprover(99)` — fetches approver 99 (belongs to T2), calls `removeNode(99)`, soft-deletes it and its entire subtree. No ownership check is performed. [6](#0-5) 
3. Approver 99 (and any child approvers in its tree) is permanently removed from T2 without User B's knowledge or consent. [7](#0-6)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],

    const transaction = await (entityManager

      : this.dataSource.manager.findOne(Transaction, find));
