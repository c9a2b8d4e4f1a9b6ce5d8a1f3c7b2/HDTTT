Audit Report

## Title
Any Authenticated User Can Post Comments on Transactions They Have No Access To

## Summary
`CommentsController` applies only authentication-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) and performs no per-transaction authorization check before allowing a user to create or read comments. The gap is self-documented with a `//TODO` in the source. The rest of the application enforces a strict participant-based access model via `TransactionsService.verifyAccess()`, but this is never called from the comments path.

## Finding Description

`CommentsController` at [1](#0-0)  applies only class-level authentication guards. The `createComment` handler carries an explicit self-documented acknowledgment of the missing check: [2](#0-1) 

`CommentsService.createComment()` performs no authorization check — it directly creates and persists the comment for any caller: [3](#0-2) 

The `getComments` and `getCommentById` endpoints are equally unguarded at the transaction-participant level: [4](#0-3) 

By contrast, `TransactionsService.verifyAccess()` enforces a strict participant model (creator, signer, observer, approver) and is correctly used in `getTransactionWithVerifiedAccess()` and `importSignatures()`: [5](#0-4) [6](#0-5) 

This check is entirely absent from the comments path.

## Impact Explanation

**Unauthorized write:** Any verified user can `POST /transactions/:transactionId/comments` with an arbitrary `transactionId` and inject comments into the audit trail of a transaction they have no relationship to. This corrupts the integrity of the collaborative workflow record for sensitive multi-signature operations.

**Unauthorized read:** Any verified user can `GET /transactions/:transactionId/comments` and retrieve the full comment history of any transaction, leaking internal deliberation, key management decisions, and operational context that should be restricted to transaction participants.

## Likelihood Explanation

Exploitation requires only a valid JWT for any registered, verified organization user. Transaction IDs are sequential integers, trivially enumerable by incrementing. No elevated privileges, race conditions, or special tooling are required. A single authenticated HTTP `POST` or `GET` to the comments endpoint suffices.

## Recommendation

Inject `TransactionsService` into `CommentsService` (or `CommentsController`) and call `getTransactionWithVerifiedAccess()` (or an equivalent `verifyAccess()` check) before processing any comment operation. For `createComment`, this ensures only participants can write. For `getComments` and `getCommentById`, the same check should gate read access. The `//TODO` comment at line 22 of `comments.controller.ts` already acknowledges this requirement — it must be implemented, not left open.

## Proof of Concept

1. Register and verify two user accounts: **UserA** (creator of transaction `id=5`) and **UserB** (no relationship to transaction `id=5`).
2. Authenticate **UserB** and obtain a valid JWT.
3. Send the following request as **UserB**:

```http
POST /transactions/5/comments
Authorization: Bearer <UserB_JWT>
Content-Type: application/json

{ "message": "Injected comment by unauthorized user" }
```

4. The server returns `201 Created` and the comment is persisted, associated with transaction `5` and **UserB**.
5. Confirm unauthorized read:

```http
GET /transactions/5/comments
Authorization: Bearer <UserB_JWT>
```

6. The server returns all comments for transaction `5`, including any sensitive deliberation by legitimate participants.

No access check is evaluated at any point in the `createComment` or `getComments` call chain, as confirmed by the source at [3](#0-2)  and the self-documented `//TODO` at [7](#0-6) .

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
