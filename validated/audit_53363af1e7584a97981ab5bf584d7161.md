# Audit Report

## Title
Approver Can Reject a Transaction Already in `WAITING_FOR_EXECUTION` State, Permanently Blocking Execution

## Summary
The `approveTransaction` function in `approvers.service.ts` permits a rejection (`approved: false`) when a transaction is already in `WAITING_FOR_EXECUTION` — the state where all required signatures are collected and the transaction is queued for Hedera network submission. An approver who has not yet cast their vote can submit a rejection after the approval threshold is already met, moving the transaction to the terminal `REJECTED` state and permanently preventing execution.

## Finding Description

**Root cause — missing state guard for rejections in `WAITING_FOR_EXECUTION`:**

In `approvers.service.ts`, the only status guard before recording an approver's choice is:

```typescript
if (
  transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
  transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
)
  throw new BadRequestException(ErrorCodes.TNRA);
``` [1](#0-0) 

This guard applies identically to both `approved: true` and `approved: false`. There is no additional check that blocks a rejection once the transaction has already advanced to `WAITING_FOR_EXECUTION`.

**The "already voted" guard does not protect against this:**

```typescript
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
``` [2](#0-1) 

This only blocks an approver who has **already** submitted a choice (has a `signature`). An approver who has not yet voted — the normal case when a threshold (e.g., 2-of-3) is met before all approvers respond — has no signature and passes this check freely.

**Rejection unconditionally triggers a terminal status update:**

After the approver record is updated with `approved: false`, the code at line 614 unconditionally emits `emitTransactionStatusUpdate` whenever `!dto.approved`:

```typescript
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
}
``` [3](#0-2) 

This triggers the receiver/consumer service to re-evaluate the transaction status. The receiver service maps the resulting `REJECTED` status to terminal notification types (`TRANSACTION_APPROVAL_REJECTION`, `TRANSACTION_INDICATOR_REJECTED`): [5](#0-4) 

**The front-end exposes this as a reachable flow:**

`isApprovableStatus` explicitly returns `true` for `WAITING_FOR_EXECUTION`:

```typescript
export const isApprovableStatus = (status: TransactionStatus | null | undefined): boolean =>
  [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    status as TransactionStatus,
  );
``` [4](#0-3) 

`ApproveTransactionController.vue` renders a "Reject transaction" button for any transaction in an approvable status, and the `performApprove` handler signs and submits the rejection via the standard REST API: [6](#0-5) 

## Impact Explanation
A transaction that has satisfied its approval threshold and is queued for Hedera network submission can be permanently killed by any single approver who has not yet cast their vote. The transaction moves to the terminal `REJECTED` state. The `approveTransaction` status guard then throws `ErrorCodes.TNRA` for any subsequent approval attempt on a `REJECTED` transaction, making the state irrecoverable. The transaction creator, signers, and observers lose the ability to execute the transaction. In an organization managing time-sensitive Hedera operations (e.g., fee schedule updates, token operations), this causes irreversible loss of the intended on-chain action.

## Likelihood Explanation
The attacker must be a legitimately assigned approver for the target transaction — a role granted by the transaction creator to any organization member. In any threshold approval scheme (e.g., 2-of-3), at least one approver will routinely not have voted by the time the threshold is met. That approver can intentionally or accidentally submit a rejection after `WAITING_FOR_EXECUTION` is reached. No admin keys, no leaked credentials, and no privileged access beyond the normal approver role are required. The attack path is directly exposed through the standard REST API endpoint and the front-end UI.

## Recommendation
Add an explicit guard in `approveTransaction` that blocks rejections (`approved: false`) when the transaction is already in `WAITING_FOR_EXECUTION`. Only approvals should be permitted at that stage:

```typescript
if (
  !dto.approved &&
  transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
) {
  throw new BadRequestException(ErrorCodes.TNRA);
}
```

This should be inserted immediately after the existing status guard at line 588 in `approvers.service.ts`. Alternatively, restrict the approvable status set for rejections to `WAITING_FOR_SIGNATURES` only, and update `isApprovableStatus` in the front-end to reflect the same distinction between approval and rejection eligibility.

## Proof of Concept

1. Creator creates a transaction with a 2-of-3 approver scheme (approvers A, B, C).
2. Approvers A and B submit `approved: true`. The system promotes the transaction to `WAITING_FOR_EXECUTION`.
3. Approver C (who has not yet voted) calls:
   ```
   POST /transactions/{id}/approvers/approve
   { "approved": false, "userKeyId": <C's key id>, "signature": "<valid sig>" }
   ```
4. The status guard at lines 584–588 passes (status is `WAITING_FOR_EXECUTION`).
5. The "already voted" guard at line 563 passes (C has no prior `signature`).
6. C's approver record is updated with `approved: false`.
7. `emitTransactionStatusUpdate` is called unconditionally (line 614–615).
8. The receiver/consumer service processes the event, evaluates the rejection, and transitions the transaction to `REJECTED`.
9. Any subsequent attempt to approve or execute the transaction receives `ErrorCodes.TNRA` — the transaction is permanently blocked.
