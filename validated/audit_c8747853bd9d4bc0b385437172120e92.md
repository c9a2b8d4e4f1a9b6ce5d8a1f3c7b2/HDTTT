Audit Report

## Title
Lack of Access Control on Transaction Comment Endpoints Allows Any Verified User to Read and Write Comments on Arbitrary Transactions

## Summary
`CommentsController` exposes three endpoints protected only by JWT authentication and email-verification guards. No check is performed to confirm the requesting user is a participant (creator, signer, approver, or observer) of the target transaction. Any verified user can post comments on and read comments from any transaction in the system. The gap is explicitly acknowledged in the source code with a `//TODO` comment.

## Finding Description

**Controller — missing transaction-level authorization:**

`CommentsController` applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level. These confirm a valid, non-blacklisted JWT and a verified account; they say nothing about the caller's relationship to the target transaction. [1](#0-0) 

The `POST` handler carries an explicit developer acknowledgement that the check is missing: [2](#0-1) 

The `GET` handlers return all comments for any caller-supplied `transactionId` with no per-user filtering: [3](#0-2) 

**Service layer — no access check:**

`CommentsService.createComment` creates and persists the comment directly without calling any access-verification helper: [4](#0-3) 

`CommentsService.getTransactionComments` returns every comment for the supplied `transactionId` without checking whether the caller is a participant: [5](#0-4) 

**Correct pattern used elsewhere:**

`TransactionsService.verifyAccess` is the intended authorization helper — it checks whether the user is a creator, signer, approver, or observer, and returns `false` otherwise: [6](#0-5) 

`TransactionsService.importSignatures` demonstrates the correct pattern: it calls `verifyAccess` before touching any transaction data and throws `ErrorCodes.TNF` if the user has no relationship to the transaction: [7](#0-6) 

The `verifyAccess` logic is tested and confirmed to return `false` for users with no relationship to the transaction: [8](#0-7) 

## Impact Explanation

- **Unauthorized state mutation**: Any verified user can inject comments into transactions they have no relationship to. Comment threads are visible to legitimate participants (creators, signers, approvers, observers); injected comments can mislead approvers or signers and corrupt the audit trail.
- **Cross-user information disclosure**: Comments may contain sensitive operational details (key management decisions, approval rationale). Any verified user can read them for any transaction ID, breaking confidentiality between unrelated users.

## Likelihood Explanation

- Precondition: attacker must hold a valid, verified account on the organization server — a low bar for any insider or compromised account.
- Transaction IDs are sequential integers (auto-increment primary keys), making enumeration trivial.
- No rate-limiting specific to comment endpoints was observed.
- The missing check is acknowledged in the source code itself (`//TODO need some sort of guard or check to ensure user can comment here`), confirming this is not an intentional design decision.

## Recommendation

Apply the same `verifyAccess` pattern already used in `importSignatures` to all three comment endpoints. Concretely:

1. Inject `TransactionsService` into `CommentsService` (or pass the resolved `Transaction` entity from the controller).
2. Before `createComment`, `getTransactionComments`, and `getTransactionCommentById`, fetch the transaction with its `creatorKey`, `observers`, `signers`, and `approvers` relations and call `transactionsService.verifyAccess(transaction, user)`.
3. Throw `UnauthorizedException` (or `BadRequestException(ErrorCodes.TNF)` to avoid leaking existence) if `verifyAccess` returns `false`.
4. Remove the `//TODO` comment once the guard is in place.

## Proof of Concept

```
# 1. Register and verify an account (attacker@example.com) — normal user, no special role.

# 2. Obtain a JWT for the attacker account.
TOKEN=$(curl -s -X POST https://<org-server>/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"attacker@example.com","password":"..."}' | jq -r .accessToken)

# 3. Read all comments on transaction ID 1 (information disclosure):
curl -s -X GET https://<org-server>/transactions/1/comments \
  -H "Authorization: Bearer $TOKEN"
# Returns all comments for transaction 1, regardless of whether the attacker
# is a creator, signer, approver, or observer.

# 4. Inject a comment into transaction ID 1 (unauthorized state mutation):
curl -s -X POST https://<org-server>/transactions/1/comments \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d '{"message":"Approved — please proceed with signing."}'
# Comment is persisted and visible to all legitimate participants of transaction 1.

# Repeat steps 3–4 for IDs 2, 3, 4, … to enumerate all transactions.
```

```typescript
  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
```
