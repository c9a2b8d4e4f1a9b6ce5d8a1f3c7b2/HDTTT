### Title
Cross-Transaction Authorization Bypass: Any Transaction Creator Can Delete Approvers from Transactions They Do Not Own

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies only that the authenticated user is the creator of the `transactionId` in the URL, but then calls `removeTransactionApprover(id)` with no check that the approver record being deleted actually belongs to that transaction. Any authenticated user who has created at least one transaction can supply their own `transactionId` to pass the ownership check, then supply an arbitrary approver `id` from a completely different transaction to delete it — bypassing the approval requirements of transactions they do not own.

### Finding Description

**Root cause — controller does not bind the approver to the URL transaction:** [1](#0-0) 

```
@Delete('/:id')
async removeTransactionApprover(@GetUser() user, @Param('transactionId') transactionId, @Param('id') id) {
    await this.approversService.getCreatorsTransaction(transactionId, user);  // ← checks ownership of URL transactionId only
    await this.approversService.removeTransactionApprover(id);                // ← deletes approver by id with NO cross-check
    return true;
}
```

`getCreatorsTransaction` only verifies the caller owns `transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches and deletes the approver by its own `id` with no verification that `approver.transactionId === transactionId`: [3](#0-2) 

Contrast this with `updateTransactionApprover`, which correctly validates cross-ownership: [4](#0-3) 

**Exploit path:**

1. Attacker (authenticated user) creates Transaction A — they are its creator.
2. Attacker discovers the `id` of an approver record belonging to Transaction B (owned by another user). Approver IDs are sequential integers; the attacker can enumerate them or observe them via the `GET /transactions/:id/approvers` endpoint for any transaction they have read access to.
3. Attacker calls `DELETE /transactions/{Transaction_A_id}/approvers/{approver_B_id}`.
4. `getCreatorsTransaction(Transaction_A_id, attacker)` passes — attacker owns Transaction A.
5. `removeTransactionApprover(approver_B_id)` soft-deletes the approver from Transaction B with no ownership check.

**Threshold manipulation consequence:**

The `isApproved` logic evaluates the live approver tree: [5](#0-4) 

If Transaction B has a threshold-based tree (e.g., threshold = 2, children = [Approver1 ✓, Approver2 ✓, Approver3 ✗]) and the attacker deletes Approver3, the tree now has 2 children both approved. The condition `approvals.length (2) >= threshold (2)` evaluates to `true` — the transaction is now considered approved with one fewer approval than the creator required. The `removeNode` SQL does not adjust the parent threshold: [6](#0-5) 

Additionally, `removeTransactionApprover` emits `emitTransactionStatusUpdate` using `approver.transactionId` (Transaction B's ID), which triggers the chain service to re-evaluate Transaction B's status immediately after the deletion. [7](#0-6) 

### Impact Explanation

An attacker who is any authenticated organization member (no admin role required — any user can create a transaction) can:

- Delete approvers from any other user's transaction, reducing the effective approval count required.
- Force a transaction that was blocked pending a required approval to appear fully approved, potentially triggering its execution on the Hedera network without the intended governance sign-off.
- Permanently corrupt the approval tree of any in-flight transaction (`WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`), since `getCreatorsTransaction` has no status guard. [2](#0-1) 

### Likelihood Explanation

- **Attacker precondition**: Must be an authenticated, verified organization member. No admin or special role required — creating a transaction is a standard user action.
- **Discovery of approver IDs**: Approver IDs are auto-incremented integers. Any approver the attacker can read (e.g., from their own transactions or from transactions they are an observer/signer of) leaks the ID space. Sequential enumeration is trivial.
- **No rate limiting or anomaly detection** is visible in the controller or service layer for this endpoint.

### Recommendation

Inside `removeTransactionApprover` (service), after fetching the approver, assert that it belongs to the transaction the caller was authorized against:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // Resolve the root node to get the owning transactionId
    const root = await this.getRootNodeFromNode(approver.id);
    if (!root || root.transactionId !== transactionId)
        throw new UnauthorizedException('Approver does not belong to this transaction');

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: root.transactionId }]);
    return result;
}
```

This mirrors the pattern already correctly implemented in `updateTransactionApprover`. [4](#0-3) 

### Proof of Concept

**Setup:**
- User A creates Transaction A (they are the creator).
- User B creates Transaction B with a threshold approver tree: threshold = 2, approvers = [User C, User D, User E]. Users C and D have already approved (signatures stored). User E has not.

**Attack:**
```
DELETE /transactions/{Transaction_A_id}/approvers/{approver_E_id}
Authorization: Bearer <User_A_JWT>
```

**Result:**
1. `getCreatorsTransaction(Transaction_A_id, UserA)` → passes (User A owns Transaction A).
2. `removeTransactionApprover(approver_E_id)` → soft-deletes User E's approver record from Transaction B.
3. Transaction B's approver tree now has 2 children (C ✓, D ✓) with threshold 2 → `isApproved` returns `true`.
4. `emitTransactionStatusUpdate` fires for Transaction B → chain service re-evaluates and may advance Transaction B to `WAITING_FOR_EXECUTION` or execute it.

**Expected (correct) behavior:** `403 Unauthorized` — approver does not belong to Transaction A.
**Actual behavior:** `200 OK` — approver deleted, Transaction B's approval gate bypassed.

```typescript
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
```

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
