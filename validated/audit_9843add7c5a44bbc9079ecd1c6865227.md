Audit Report

## Title
Unbounded Full-Table Scan with Per-Row Async Work in `getTransactionsToSign` Causes Server-Side DoS

## Summary
`GET /transactions/sign` fetches every non-terminal transaction from the database with no row limit, then performs a sequential async call (`userKeysToSign`) per row before applying pagination in memory. A malicious authenticated user can inflate the active-transaction count to force unbounded server-side work on every request.

## Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues a `repo.find()` with no `take` or `skip` clause:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no take / skip
});
``` [1](#0-0) 

It then iterates over every returned row and sequentially awaits an async call per transaction:

```ts
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

Pagination is applied only after the full iteration completes:

```ts
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

The controller explicitly notes the endpoint is **"NO LONGER USED BY FRONT-END"** but it remains fully reachable at `GET /transactions/sign`: [4](#0-3) 

The controller class is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no rate limit or transaction-count cap: [5](#0-4) 

A secondary surface exists at `POST /transactions/signers` (`uploadSignatureMap`) and `POST /transactions/signatures/import` (`importSignatures`). Both accept an unbounded `UploadSignatureMapDto[]` array with no `@ArrayMaxSize()` constraint on the DTO: [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation

- **Server resource exhaustion**: Each request to `GET /transactions/sign` loads all active (non-terminal) transactions into the Node.js heap and fires one sequential async call per row involving mirror-node queries and cryptographic key-tree resolution (`userKeysRequiredToSign`). With thousands of active transactions, this saturates the event loop, exhausts DB connection pool slots, and spikes memory.
- **Cascading service degradation**: The API, Chain, and Notifications microservices share infrastructure. A saturated API event loop delays all other users' requests.
- **Amplification via transaction creation**: Any verified user can create transactions via `POST /transactions`, directly inflating the dataset that every subsequent `GET /transactions/sign` call must process in full.
- **Secondary amplification**: Submitting a large unbounded array to `POST /transactions/signers` or `POST /transactions/signatures/import` triggers per-element cryptographic validation with no array size cap.

## Likelihood Explanation

- **Attacker precondition**: A valid JWT for any verified organization user — achievable by registering a normal account.
- **Attack path**: register → verify → create N transactions in a loop → repeatedly call `GET /transactions/sign` to trigger the unbounded scan.
- No privileged access, no leaked secrets, and no physical access required.
- The endpoint is still live despite the "NO LONGER USED BY FRONT-END" comment, meaning it receives no front-end traffic that would reveal degradation before an attacker exploits it.

## Recommendation

1. **`getTransactionsToSign`**: Rewrite to push the key-matching logic into the database query (e.g., join on `transaction_signer` / `user_key` tables) so that only relevant rows are fetched, and apply `take`/`skip` at the DB level before any per-row async work.
2. **If the endpoint is truly unused**: Remove or disable `GET /transactions/sign` entirely, as the comment already acknowledges it is no longer used by the front-end.
3. **`UploadSignatureMapDto[]` endpoints**: Add `@ArrayMaxSize(N)` (e.g., 100) to the array body parameter on both `POST /transactions/signers` and `POST /transactions/signatures/import` to cap the number of elements processed per request.
4. **Rate limiting**: Apply a rate-limiting guard (e.g., NestJS `ThrottlerGuard`) to all transaction mutation and query endpoints.

## Proof of Concept

```
# 1. Register and verify a user account, obtain JWT token.

# 2. Create N transactions (e.g., 10,000) in a loop:
for i in $(seq 1 10000); do
  curl -s -X POST https://<host>/transactions \
    -H "Authorization: Bearer <JWT>" \
    -H "Content-Type: application/json" \
    -d '{ ...valid CreateTransactionDto... }'
done

# 3. Repeatedly trigger the unbounded scan:
while true; do
  curl -s "https://<host>/transactions/sign?page=1&limit=10" \
    -H "Authorization: Bearer <JWT>"
done
```

Each iteration of step 3 causes the server to load all N active transactions from the database and call `userKeysToSign` (mirror-node + crypto) sequentially for each one before returning the first page of 10 results, exhausting CPU, memory, and DB connections.
