### Title
`getTransactionsToSign` Fetches Entire Transaction Table Without Limit, Enabling Authenticated DoS via Resource Exhaustion

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction from the database in a single unbounded query, then iterates over all of them performing expensive per-row operations (SDK deserialization, signature-key computation, database lookups) before applying pagination. Any authenticated user can trigger this endpoint to exhaust server memory, CPU, and database connections, degrading or crashing the API for all users.

### Finding Description

**Root cause — unbounded `repo.find()` with no `take` clause:**

In `getTransactionsToSign` (lines 295–299), the query fetches every transaction whose status is not in the five terminal states, with no row limit:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip`
});
``` [1](#0-0) 

**Expensive per-row loop before pagination:**

The code then iterates over every returned row and calls `userKeysToSign` for each one. That function deserializes the raw transaction bytes via the Hedera SDK, calls `transactionSignatureService.computeSignatureKey` (which may itself issue mirror-node network requests), and issues a `UserKey` database query:

```ts
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

**Pagination applied only after full scan:**

The `offset`/`limit` values from `PaginationParams` are only used to slice the already-computed in-memory array:

```ts
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

**Contrast with other endpoints that paginate correctly:**

`getHistoryTransactions` and `getTransactionsToApprove` both pass `skip`/`take` directly into the TypeORM query, so the database enforces the page size before any data is transferred or processed: [4](#0-3) 

**Exposed HTTP endpoint:**

The endpoint is registered at `GET /transactions/sign` and is reachable by any authenticated, verified user: [5](#0-4) 

The `PaginationParams` decorator enforces `size ≤ 100`, but this limit is never applied to the database query — only to the final slice: [6](#0-5) 

### Impact Explanation

With N active (non-terminal) transactions in the system, a single request to `GET /transactions/sign?page=1&size=1` causes the server to:
1. Load all N transaction rows (including `transactionBytes` blobs) into memory.
2. Deserialize each blob via the Hedera SDK.
3. Issue up to N additional database queries for `UserKey` lookups.
4. Optionally issue N mirror-node HTTP calls inside `computeSignatureKey`.

At scale (thousands of active transactions, which is realistic for an organization managing Hedera council operations), this exhausts heap memory, saturates the database connection pool, and blocks the Node.js event loop, making the entire API unresponsive for all users. The attacker pays only the cost of one HTTP request per attack cycle.

### Likelihood Explanation

- **Precondition**: valid JWT for any non-admin organization member — the lowest privilege level in the system.
- **Trigger**: a single `GET /transactions/sign?page=1&size=1` request.
- **No rate limiting** is visible on this endpoint beyond standard auth guards.
- The comment `/* NO LONGER USED BY FRONT-END */` in the controller confirms the endpoint is still live but no longer monitored for abuse.

### Recommendation

Apply the same pattern used by `getHistoryTransactions`: push `skip`/`take` into the TypeORM query so the database enforces the page size before any rows are transferred or processed.

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  skip: offset,
  take: limit,
});
```

Because `userKeysToSign` must be called per row to determine eligibility, the total count cannot be computed cheaply in the same pass. Accept that `totalItems` will reflect only the current page's eligible count, or maintain a separate count query. Alternatively, push the signing-key logic into a SQL subquery so filtering happens in the database.

If the endpoint is truly unused by the front-end, consider removing it entirely to eliminate the attack surface.

### Proof of Concept

1. Register or obtain credentials for any non-admin organization user.
2. Ensure the organization has a large number of active (non-terminal) transactions (e.g., ≥ 10 000 rows in `WAITING_FOR_SIGNATURES` status).
3. Issue:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <valid_jwt>
   ```
4. Observe: the server loads all N transaction rows, deserializes each blob, and issues N `UserKey` queries before returning a single result. Memory and CPU spike proportionally to N; repeated requests cause OOM or connection-pool exhaustion, returning 500 errors to all concurrent users.
