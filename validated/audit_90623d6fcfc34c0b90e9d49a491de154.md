### Title
Silent Error Swallowing in Transaction Scheduler Callbacks Leaves Transactions Permanently Stuck

### Summary
In `TransactionSchedulerService`, async callbacks registered via `setTimeout` catch all errors but only call `console.log(error)` without updating the transaction status or propagating the failure. When `computeSignatureKey` or `smartCollate` throws an unexpected error inside `collateAndExecute` or `collateGroupAndExecute`, the transaction is silently abandoned in `WAITING_FOR_EXECUTION` status — the execution timeout is never registered, the transaction is never submitted to Hedera, and users receive no notification of the failure.

### Finding Description

In `collateAndExecute`, the callback fires 10 seconds before the transaction's `validStart`. It calls `computeSignatureKey` (which performs mirror node HTTP calls and database queries) and `smartCollate` (which deserializes SDK transaction bytes). If either throws an unexpected error, the `catch` block only logs to stdout: [1](#0-0) 

The `catch` at line 319–321 swallows the error:

```typescript
} catch (error) {
  console.log(error);   // ← error silently dropped
} finally {
  this.schedulerRegistry.deleteTimeout(name);
}
```

Because `addExecutionTimeout` (line 318) is never reached, no execution timeout is ever registered. The transaction remains in `WAITING_FOR_EXECUTION` indefinitely until `handleExpiredTransactions` marks it `EXPIRED` — not `FAILED` — giving users no indication of what went wrong.

The identical pattern exists in `collateGroupAndExecute` for grouped transactions: [2](#0-1) 

And in `addExecutionTimeout` and `addGroupExecutionTimeout`, where unexpected errors from `executeService.executeTransaction` / `executeService.executeTransactionGroup` are also silently swallowed: [3](#0-2) [4](#0-3) 

The `collateAndExecute` and `collateGroupAndExecute` cases are the most severe: the `smartCollateFailed` path (lines 234–257 / 287–309) correctly updates status to `FAILED` and emits a notification, but this only handles the `null` return from `smartCollate`. Any thrown exception bypasses this path entirely.

### Impact Explanation

A transaction in `WAITING_FOR_EXECUTION` that silently fails the collation step is never submitted to the Hedera network. It will eventually be marked `EXPIRED` by `handleExpiredTransactions`, which runs every 10 seconds and transitions `WAITING_FOR_EXECUTION` transactions past their valid start to `EXPIRED`: [5](#0-4) 

The result is:
- The transaction is silently dropped — never executed on-chain.
- The final status is `EXPIRED` rather than `FAILED`, masking the true cause.
- No notification is emitted for the actual failure.
- For grouped/atomic transactions, all transactions in the group are silently abandoned.
- Operators have no visibility into the failure beyond a `console.log` that may be lost in log noise.

### Likelihood Explanation

`computeSignatureKey` performs mirror node HTTP calls and database queries: [6](#0-5) 

These can throw due to network timeouts, mirror node unavailability, database connection errors, or malformed data. `smartCollate` deserializes raw transaction bytes via the Hiero SDK, which can throw on corrupted or unexpected byte sequences. Both failure modes are realistic in production environments, making this a moderate-to-high likelihood issue.

### Recommendation

In the `catch` block of each affected callback, update the transaction(s) to `FAILED` status and emit a status update notification, mirroring the existing `smartCollateFailed` handling pattern already present in the same functions:

```typescript
} catch (error) {
  this.logger.error('Collation failed for transaction ' + transaction.id, error);
  const result = await this.transactionRepo
    .createQueryBuilder()
    .update(Transaction)
    .set({ status: TransactionStatus.FAILED, executedAt: new Date() })
    .where('id = :id AND status = :currentStatus', {
      id: transaction.id,
      currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
    })
    .returning('id')
    .execute();
  if (result.raw.length > 0) {
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      result.raw.map(row => ({ entityId: row.id })),
    );
  }
}
```

Replace `console.log` with the NestJS `Logger` throughout, and apply the equivalent group-aware version in `collateGroupAndExecute`.

### Proof of Concept

1. A transaction is created with sufficient signatures and transitions to `WAITING_FOR_EXECUTION`.
2. `prepareTransactions` calls `collateAndExecute`, which schedules a `setTimeout` callback 10 seconds before `validStart`.
3. When the callback fires, `computeSignatureKey` throws (e.g., mirror node is unreachable).
4. The `catch` block at line 319 executes `console.log(error)` and returns.
5. `addExecutionTimeout` is never called — no execution timeout is registered.
6. The transaction remains in `WAITING_FOR_EXECUTION`.
7. Three minutes after `validStart`, `handleExpiredTransactions` marks it `EXPIRED`.
8. The user sees the transaction as `EXPIRED` with no indication of the actual failure, and the transaction was never submitted to Hedera. [7](#0-6)

  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },
            });
            // All the transactions for the group are now pulled. If there is an issue validating for even one
            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);
        }
      }
    }
  }

  addGroupExecutionTimeout(transactionGroup: TransactionGroup) {
    const name = `group_execution_timeout_${transactionGroup.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart =
      transactionGroup.groupItems[0].transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransactionGroup(transactionGroup);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }

  addExecutionTimeout(transaction: Transaction) {
    const name = `execution_timeout_${transaction.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    if (transaction.isManual) return;

    const timeToValidStart = transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransaction(transaction);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }
