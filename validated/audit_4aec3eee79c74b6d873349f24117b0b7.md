Audit Report

## Title
Debounce Starvation in `DebouncedNotificationBatcher` Causes Silent In-App Notification Loss

## Summary
In `DebouncedNotificationBatcher.add()`, the `batchKey` TTL is set only once (on first insert) while the `flushKey` TTL is renewed on every subsequent call. A sustained stream of messages keeps `flushKey` alive past `maxFlushMS`, at which point `batchKey` expires. The resulting keyevent fires `flush()`, which calls `lrange` on the already-expired key, receives an empty list, and silently returns — permanently discarding all queued messages with no error, retry, or log.

## Finding Description

In `DebouncedNotificationBatcher.add()`:

- **`batchKey` TTL** is set to `maxFlushMS` only when `length === 1` (first message). It is never renewed. [1](#0-0) 

- **`flushKey` TTL** is reset to `delayMs` on every subsequent `add()` call. [2](#0-1) 

When `batchKey` expires, the Redis keyevent subscription fires `processExpiration()` → `flush()`: [3](#0-2) 

Inside `flush()`, `lrange(batchKey, 0, -1)` is called on the already-expired key. Redis returns an empty list. The guard at lines 102–104 silently returns with no error, no retry, and no log — all accumulated messages are permanently lost: [4](#0-3) 

The `maxBatchSize` guard only protects against a single burst exceeding the batch limit; it does not protect against a sustained low-rate stream that keeps `flushKey` alive past `maxFlushMS`: [5](#0-4) 

## Impact Explanation

The `WebsocketGateway` instantiates the batcher with `delayMs=500`, `maxBatchSize=200`, `maxFlushMS=2000`, using `userId` as the group key for per-user in-app notifications: [6](#0-5) 

Dropped messages mean users never receive in-app WebSocket alerts for transaction status transitions (`WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELLED`): [7](#0-6) 

In a multi-signature workflow, suppressed signing notifications can stall transactions past their `validStart` window, causing them to expire without execution. Note: email notifications are handled via a separate NATS pipeline in `ReceiverService` and are not affected by this batcher.

## Likelihood Explanation

Any authenticated user can create transactions. The `TransactionSchedulerService` runs cron jobs as frequently as every 10 seconds, automatically triggering status transitions: [8](#0-7) [9](#0-8) 

With `maxBatchSize=200` and `delayMs=500ms`, an attacker needs only to sustain ≥1 notification event per 500ms for a target user's group key for >2000ms — well below the 200-message burst threshold. This keeps `flushKey` perpetually renewed while `batchKey` silently expires.

## Recommendation

1. **Renew `batchKey` TTL on every `add()` call**, not just the first. Replace the `if (length === 1)` guard with an unconditional `pexpire(batchKey, maxFlushMS)` after each `rpush`. This ensures `batchKey` always outlives `flushKey`.

2. **Add a log and/or alert** in `flush()` when `messages.length === 0` after a `batchKey`-triggered expiration, to make silent drops observable.

3. **Consider using `GETDEL` or a Lua script** to atomically retrieve-and-delete `batchKey` contents, preventing the race between expiry and retrieval.

## Proof of Concept

```
Timeline (delayMs=500ms, maxFlushMS=2000ms, maxBatchSize=200):

t=0ms:    add(msg1) → rpush batchKey [msg1], pexpire batchKey 2000ms, set flushKey PX 500ms
t=400ms:  add(msg2) → rpush batchKey [msg1,msg2], pexpire flushKey 500ms  ← batchKey NOT renewed
t=800ms:  add(msg3) → rpush batchKey [msg1,msg2,msg3], pexpire flushKey 500ms
t=1200ms: add(msg4) → ... pexpire flushKey 500ms
t=1600ms: add(msg5) → ... pexpire flushKey 500ms
t=2000ms: *** batchKey EXPIRES *** → keyevent fires → processExpiration() → flush()
           lrange(batchKey) → [] (key is gone)
           messages.length === 0 → silent return
           msg1..msg5 permanently lost, flushKey still alive
t=2100ms: add(msg6) → rpush batchKey [msg6], pexpire batchKey 2000ms (new cycle)
           flushKey renewed again — cycle repeats indefinitely
```
