Audit Report

## Title
Authenticated User Can Register Any Public Key Without Proof-of-Ownership, Enabling Key Squatting and Transaction Workflow DoS

## Summary
The `uploadKey` endpoint in `user-keys.service.ts` accepts any syntactically valid Hedera public key from any authenticated, verified organization member without requiring cryptographic proof that the submitter controls the corresponding private key. A verified attacker can preemptively register a victim's known public key under their own account. Once squatted, the legitimate owner receives `ErrorCodes.PU` on every registration attempt and is permanently locked out of the organization's signing and approval workflows under their actual key.

## Finding Description

**Root cause — `uploadKey` in `user-keys.service.ts`:**

The only checks performed are:
1. The submitted string must parse as a valid Hedera public key (`@IsHederaPublicKey()` in the DTO).
2. The key must not already be registered by a *different* user. [1](#0-0) 

No challenge-response or signature proof is required to demonstrate control of the private key.

**The DTO only validates format, not ownership:** [2](#0-1) 

**The controller ignores the `userId` path parameter entirely for POST — the key is always registered for the JWT-authenticated caller:** [3](#0-2) 

**The e2e test explicitly confirms the `userId` path param is ignored:** [4](#0-3) 

The test passes `/123123123/keys` and `/5134/keys` but the key is registered for the authenticated user regardless.

**Removal is not available to the victim.** The user-facing delete endpoint enforces ownership: [5](#0-4) 

The internal `removeKey` (no ownership check) is not exposed through the controller: [6](#0-5) 

**A skipped test in `approvers.service.spec.ts` confirms the missing key-ownership check during approval was never enforced:** [7](#0-6) 

*(Note: The report's citation [7] misattributes this skipped test to the user-keys e2e spec; the skipped test in `user-keys.e2e-spec.ts` is about max-key limits, not signature ownership. The actual skipped ownership test is in `approvers.service.spec.ts`. This does not affect the validity of the vulnerability.)*

## Impact Explanation

- **Key squatting / permanent registration DoS**: The victim's real Hedera public key is permanently claimed by the attacker. The victim cannot register it and therefore cannot be added as a signer or approver using their on-chain identity.
- **Transaction workflow freeze**: If a transaction creator adds the attacker's user ID as an approver (expecting the squatted key to be the victim's), the attacker cannot produce a valid signature (they lack the private key), and the victim cannot sign either. The transaction is stuck in `WAITING_FOR_SIGNATURES` indefinitely.
- **Impersonation of key identity**: The attacker's account appears to "own" a public key associated with a known Hedera identity, corrupting the organization's key-to-user mapping.

## Likelihood Explanation

The attacker only needs to be a verified organization member — no admin or privileged role is required. Hedera public keys are public by design and trivially discoverable from mirror node APIs or from `GET /user/:userId/keys` (accessible to any verified member). The attack requires a single API call and is executable the moment a new member is invited but before they complete key registration.

## Recommendation

Implement a cryptographic proof-of-ownership challenge before accepting key registration:

1. **Challenge-response**: Issue a server-generated nonce to the client; require the client to sign it with the private key corresponding to the submitted public key and return the signature alongside the registration request.
2. **Signature verification**: In `uploadKey`, verify the submitted signature against the nonce using the submitted public key before persisting the record.
3. **Nonce expiry**: Expire nonces after a short TTL (e.g., 5 minutes) to prevent replay attacks.

This ensures only the holder of the private key can register the corresponding public key.

## Proof of Concept

```
# Step 1: Attacker (verified member) discovers victim's public key
GET /user/<victim_id>/keys
# → returns victim's publicKey: "302a300506032b6570032100<victim_hex>"

# Step 2: Attacker registers victim's key under their own account
POST /user/any/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json
{ "publicKey": "302a300506032b6570032100<victim_hex>" }
# → HTTP 201: key stored under attacker's userId

# Step 3: Victim attempts to register their own key
POST /user/<victim_id>/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json
{ "publicKey": "302a300506032b6570032100<victim_hex>" }
# → HTTP 400: ErrorCodes.PU ("Public key in use")
# Victim is permanently locked out of registering their own key.
```

The squatted key cannot be removed by the victim because `removeUserKey` enforces `userKey.userId !== user.id`, and the internal `removeKey` (no ownership check) is not exposed via the controller.
