### Title
Missing User Key Ownership Check in `importSignatures` Allows Injection of Unregistered Signatures into Transaction Bytes

### Summary

The `POST /transactions/signatures/import` endpoint in `transactions.service.ts` does not verify that the public keys present in the submitted signature map belong to the authenticated user. The parallel signing endpoint (`uploadSignatureMaps` in `signers.service.ts`) enforces this ownership check explicitly, but `importSignatures` skips it entirely, only performing a cryptographic validity check. Any authenticated user with transaction access can inject signatures from keys they control — including keys never registered in the system — directly into the stored `transactionBytes`, potentially advancing the transaction state machine without proper authorization records.

### Finding Description

**Root cause — missing ownership check in `importSignatures`:**

In `signers.service.ts`, `uploadSignatureMaps` builds a `userKeyMap` exclusively from `user.keys` (keys registered to the authenticated user) and throws `ErrorCodes.PNY` if any public key in the submitted signature map is absent from that map: [1](#0-0) [2](#0-1) 

In contrast, `importSignatures` in `transactions.service.ts` only calls `validateSignature` (cryptographic correctness) and `verifyAccess` (transaction-level access), then directly calls `sdkTransaction.addSignature` and persists the updated bytes — with no check that the public keys belong to the caller: [3](#0-2) 

The controller exposes this endpoint without `OnlyOwnerKey` or `HasKeyGuard` restrictions beyond the base JWT guards: [4](#0-3) 

Compare this to `createTransaction`, which explicitly applies `@OnlyOwnerKey<CreateTransactionDto>('creatorKeyId')` to enforce key ownership: [5](#0-4) 

**Exploit flow:**

1. Attacker is a legitimate organization user with any access role (observer, approver, or signer) on a target transaction.
2. Attacker generates a key pair that is **not** registered in the system under their account.
3. Attacker signs the raw `transactionBytes` with that key pair locally.
4. Attacker submits `POST /transactions/signatures/import` with the crafted `signatureMap`.
5. `validateSignature` passes (signature is cryptographically valid).
6. `verifyAccess` passes (attacker has transaction access).
7. The signature is added to `transaction.transactionBytes` and persisted — no `TransactionSigner` record is created.
8. If the injected key satisfies a required signing slot, `processTransactionStatus` may advance the transaction to `WAITING_FOR_EXECUTION`.

The `uploadSignatureMaps` path that correctly enforces ownership: [6](#0-5) 

### Impact Explanation

- **Audit trail bypass**: No `TransactionSigner` record is created, so the system's record of who signed is incomplete or incorrect.
- **Authorization model bypass**: A user who is only an observer (not a designated signer) can inject a valid signature from a key they control, potentially satisfying a required signing slot and advancing the transaction to `WAITING_FOR_EXECUTION` without being a registered signer.
- **State machine manipulation**: If the injected signature satisfies the threshold, the transaction can be submitted to the Hedera network with a signature from an unregistered, untracked key, violating the multi-sig governance invariant the system is designed to enforce.

### Likelihood Explanation

- Attacker preconditions: authenticated organization user with any access role on the target transaction (observer, approver, or signer). No privileged access required.
- The attacker must control a key pair whose public key is in the transaction's required signing key list. This is realistic when the attacker is the owner of a Hedera account involved in the transaction (e.g., the fee payer or an account being updated).
- The endpoint is reachable via standard REST API with a valid JWT token.
- No special tooling is needed — a standard HTTP client suffices.

### Recommendation

Add the same user key ownership check to `importSignatures` that `uploadSignatureMaps` enforces. Before calling `sdkTransaction.addSignature`, verify that every public key in the submitted signature map is present in `user.keys`:

```typescript
// In importSignatures, after loading user keys:
await attachKeys(user, this.entityManager);
const userKeyMap = new Map(user.keys.map(k => [k.publicKey, k]));

for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  if (!userKeyMap.has(raw) && !userKeyMap.has(publicKey.toStringDer())) {
    throw new BadRequestException(ErrorCodes.PNY);
  }
  sdkTransaction.addSignature(publicKey, map);
}
```

This mirrors the check already present in `signers.service.ts`: [2](#0-1) 

### Proof of Concept

**Setup**: Two users — `observer` (has observer role on transaction T) and `signer` (designated signer). Transaction T requires a signature from a key `K` that `observer` controls locally but has not registered in the system.

**Steps**:
1. `observer` logs in and obtains a JWT token.
2. `observer` generates key pair `K` locally (not uploaded via `POST /user/:userId/keys`).
3. `observer` signs transaction T's bytes with `K`.
4. `observer` sends:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <observer_jwt>
   [{ "id": <T_id>, "signatureMap": <formatted_map_with_K_signature> }]
   ```
5. Server responds `201` — no `TransactionSigner` record is created, but `transaction.transactionBytes` now contains `K`'s signature.
6. If `K` was the last required signing key, the transaction status advances to `WAITING_FOR_EXECUTION` and is submitted to the Hedera network.

**Expected (correct) behavior**: Server should return `400` with `ErrorCodes.PNY` because `K` is not in `observer`'s registered keys.

**Actual behavior**: Server accepts the signature, updates `transactionBytes`, and may advance the transaction state — bypassing the multi-sig authorization model.
