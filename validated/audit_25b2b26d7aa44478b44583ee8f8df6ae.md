Audit Report

## Title
Concurrent Signature Uploads Cause Silent Signature Loss via Blind `transactionBytes` Overwrite (TOCTOU Race Condition)

## Summary
`SignersService.uploadSignatureMaps` follows a read-modify-write pattern on `transactionBytes` with no optimistic-locking guard on the write. Two concurrent signers reading the same snapshot and each writing back only their own signature will cause the last writer to silently discard the first writer's cryptographic contribution. The stored bytes then diverge from the `TransactionSigner` table, and `processTransactionStatus` — which evaluates the actual bytes — will never promote the transaction to `WAITING_FOR_EXECUTION`.

## Finding Description

**Step 1 — Unguarded read (`loadTransactionData`)**

Both concurrent requests load the same `transactionBytes` snapshot via an unguarded `find`: [1](#0-0) 

**Step 2 — Independent in-memory mutation (`processTransactionSignatures`)**

Each request independently deserialises the snapshot and calls `addSignature`, producing a new blob that contains only its own signer's key: [2](#0-1) 

**Step 3 — Blind overwrite (`bulkUpdateTransactions`)**

The mutated bytes are written back with a plain `UPDATE … SET "transactionBytes" = CASE id … END WHERE id = ANY(…)`. There is no `WHERE "transactionBytes" = <expected_value>` predicate and no distributed lock: [3](#0-2) 

Race window:
```
T0  Request-A reads bytes_0
T0  Request-B reads bytes_0
T1  Request-A adds sig_A  → bytes_A  (sig_A only)
T1  Request-B adds sig_B  → bytes_B  (sig_B only)
T2  Request-A writes bytes_A to DB
T3  Request-B writes bytes_B to DB   ← overwrites sig_A silently
```

Final DB state: `transactionBytes` contains only `sig_B`. The `TransactionSigner` row for User A exists (inserted inside the DB transaction at `persistSignatureChanges`), but User A's cryptographic signature is absent from the stored bytes. [4](#0-3) 

Contrast with the status-update path, which correctly uses an optimistic-lock `WHERE id IN (:...ids) AND status = :oldStatus` guard: [5](#0-4) 

And the execution path, which uses `WHERE id = :id AND status = :currentStatus`: [6](#0-5) 

No equivalent guard exists for `transactionBytes`.

## Impact Explanation

`processTransactionStatus` determines whether a transaction can advance by calling `hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey)`, which inspects the actual bytes stored in the database, not the `TransactionSigner` table: [7](#0-6) 

After the overwrite, the bytes no longer contain the lost signer's key. Even though the `TransactionSigner` record shows the user signed, the scheduler will never promote the transaction to `WAITING_FOR_EXECUTION` because the cryptographic threshold is not met in the bytes. The transaction is permanently stuck unless the affected signer re-uploads — and even then, the same race can recur.

A malicious co-signer can deliberately time their upload to race with a target signer, repeatedly nullifying that signer's contribution and blocking execution indefinitely.

## Likelihood Explanation

Multi-user concurrent signing is the primary design goal of the organization workflow. The system explicitly supports multiple signers and the API accepts batch signature maps. Two users signing the same transaction at nearly the same time is a routine, expected event — not an edge case.

No distributed lock (such as the `@MurLock` used in `executeTransaction` and `executeTransactionGroup`) protects this path: [8](#0-7) [9](#0-8) 

The `uploadSignatureMaps` entry point has no equivalent protection: [10](#0-9) 

## Recommendation

Apply one of the following mitigations:

1. **Optimistic locking on `transactionBytes`**: Change `bulkUpdateTransactions` to include a `WHERE "transactionBytes" = $expected` predicate. If the update affects 0 rows, re-read the current bytes, merge the new signatures into them, and retry.

2. **Pessimistic locking**: Use `SELECT … FOR UPDATE` inside the DB transaction in `persistSignatureChanges` to lock the row before reading and writing `transactionBytes`, preventing concurrent writers from interleaving.

3. **Distributed lock**: Apply `@MurLock` (already used on `executeTransaction` and `executeTransactionGroup`) keyed on `transaction.id` around the read-modify-write cycle in `uploadSignatureMaps`.

The merge-on-retry approach (option 1) is the most scalable: re-read the current bytes, call `addSignature` on top of them, and write back only if the bytes have not changed since the re-read.

## Proof of Concept

```
1. Create a transaction T requiring signatures from User A and User B.
2. Both users sign locally and submit POST /transactions/:id/signers simultaneously.
3. Both requests execute loadTransactionData and read the same bytes_0.
4. Request-A calls processTransactionSignatures → bytes_A (contains sig_A only).
5. Request-B calls processTransactionSignatures → bytes_B (contains sig_B only).
6. Request-A commits: DB transactionBytes = bytes_A.
7. Request-B commits: DB transactionBytes = bytes_B  ← sig_A is gone.
8. TransactionSigner table now has rows for both User A and User B.
9. processTransactionStatus deserialises bytes_B, calls hasValidSignatureKey:
   sdkTransaction._signerPublicKeys contains only sig_B.
   Threshold not met → status stays WAITING_FOR_SIGNATURES.
10. Transaction never advances to WAITING_FOR_EXECUTION.
    User A must re-upload; the race can repeat indefinitely.
```
