Audit Report

## Title
IDOR in `removeTransactionApprover`: Creator Check on Wrong Resource Allows Any Approver Deletion

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` with no check that `:id` belongs to `:transactionId`. An authenticated attacker who owns any one transaction can delete approvers from any other transaction in the system.

## Finding Description
In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```
Line 108: await this.approversService.getCreatorsTransaction(transactionId, user);
Line 109: await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

The authorization check (line 108) validates that `user` is the creator of the transaction at `:transactionId`. The destructive action (line 109) operates on `:id`, which is a completely independent URL parameter. The service method `removeTransactionApprover` only checks that the approver exists, then soft-deletes it and its entire subtree:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    ...
}
``` [2](#0-1) 

There is no cross-check that the approver identified by `:id` belongs to the transaction identified by `:transactionId`. This is in direct contrast to `updateTransactionApprover`, which correctly resolves the root node of the approver and validates it matches the URL parameter before proceeding:

```typescript
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The `removeTransactionApprover` path has no equivalent guard.

## Impact Explanation
An attacker who is the creator of any transaction can:
1. Delete any approver record from any other user's transaction by supplying their own `transactionId` and a victim's approver `id`.
2. The `removeNode` function performs a recursive soft-delete of the entire approver subtree rooted at the targeted `id`, so a single request can collapse an entire threshold tree.
3. This can reduce the required approval count on a pending multi-signature transaction, allowing it to advance to execution without the intended approvals — directly undermining the multi-signature trust model.
4. The deletion is silent; the victim transaction's creator receives no indication that their approver structure was tampered with. [4](#0-3) 

## Likelihood Explanation
- The attacker only needs a valid authenticated session and to be the creator of any one transaction — a normal user capability.
- `TransactionApprover` IDs are sequential auto-increment integers in PostgreSQL, making enumeration trivial with a simple loop.
- The attack is a single HTTP DELETE request with no special privileges, leaked credentials, or physical access required.
- No rate limiting or anomaly detection is described for this endpoint.

## Recommendation
Apply the same cross-check used in `updateTransactionApprover` to the delete path. In `removeTransactionApprover` (service), before calling `removeNode`, resolve the root node of the target approver and verify its `transactionId` matches the `transactionId` URL parameter, then call `getCreatorsTransaction` on that resolved `transactionId` (not the URL parameter):

```typescript
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const rootNode = await this.getRootNodeFromNode(approver.id);
    if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

    // Cross-check: approver must belong to the transaction in the URL
    if (rootNode.transactionId !== transactionId)
        throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

    // Authorize against the actual owning transaction, not the attacker-controlled URL param
    await this.getCreatorsTransaction(rootNode.transactionId, user);

    const result = await this.removeNode(approver.id);
    ...
}
``` [5](#0-4) 

## Proof of Concept
1. Attacker (Alice) creates Transaction A and notes her `transactionId_A`.
2. Alice enumerates approver IDs (e.g., `id=42`) belonging to Victim Bob's Transaction B.
3. Alice sends:
   ```
   DELETE /transactions/{transactionId_A}/approvers/42
   Authorization: Bearer <alice_token>
   ```
4. Line 108 passes: Alice is the creator of `transactionId_A`. ✓
5. Line 109 executes: approver `id=42` (belonging to Bob's Transaction B) is soft-deleted along with its entire subtree.
6. Bob's Transaction B now has a corrupted or collapsed approver tree, potentially making it executable with fewer approvals than intended. [6](#0-5)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
