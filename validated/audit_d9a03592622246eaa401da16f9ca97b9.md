Audit Report

## Title
IDOR in `removeTransactionApprover`: Approver `id` Not Validated Against `transactionId`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller is the creator of `:transactionId`, but then deletes the approver identified by `:id` without checking that `:id` actually belongs to `:transactionId`. Any authenticated user who is the creator of **any** transaction can delete approvers belonging to **any other** transaction.

## Finding Description

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent service calls:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts  lines 107-109
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
``` [1](#0-0) 

`getCreatorsTransaction` only verifies `transaction.creatorKey?.userId !== user.id` for the supplied `transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by `id` alone and deletes it — no check that `approver.transactionId === transactionId`: [3](#0-2) 

By contrast, `updateTransactionApprover` **does** perform this cross-check correctly:

```typescript
// approvers.service.ts lines 390-391
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [4](#0-3) 

The missing equivalent check in the delete path is the root cause.

## Impact Explanation
An attacker who is the creator of any transaction can permanently soft-delete approvers from any other transaction in the system. This can:
- Remove required approvers from a transaction they do not own, bypassing the multi-party approval workflow.
- Cause a transaction that requires N-of-M approvals to execute with fewer approvers than intended, or to become immediately executable if all approvers are removed.

## Likelihood Explanation
The attacker only needs to be a legitimate user who has created at least one transaction (a normal workflow action). Approver IDs are sequential integers, making enumeration trivial. No special privileges, leaked credentials, or physical access are required. The attack is a single authenticated HTTP request.

## Recommendation
Inside `removeTransactionApprover` (service or controller), after fetching the approver, verify that its `transactionId` matches the URL parameter before proceeding with deletion — mirroring the check already present in `updateTransactionApprover`:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Get root node to handle nested approvers
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller call site accordingly.

## Proof of Concept

**Setup:**
- User A is the creator of Transaction 1 (ID = 1).
- User B is the creator of Transaction 2 (ID = 2), which has an approver with ID = 99.

**Attack:**
```http
DELETE /transactions/1/approvers/99
Authorization: Bearer <User A's JWT>
```

**Execution trace:**
1. `JwtBlackListAuthGuard` / `JwtAuthGuard` / `VerifiedUserGuard` pass — User A is a valid, verified user.
2. `getCreatorsTransaction(1, userA)` — passes, User A is the creator of Transaction 1.
3. `removeTransactionApprover(99)` — fetches approver 99 (belongs to Transaction 2), finds it exists, calls `removeNode(99)`, soft-deletes it.
4. Approver 99 on Transaction 2 is permanently removed. User B's transaction approval structure is silently corrupted. [1](#0-0) [3](#0-2)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
