### Title
Transaction Creator Can Delete Approvers Belonging to Any Transaction via Mismatched Authorization Scope

### Summary
`DELETE /transactions/:transactionId/approvers/:id` verifies the caller is the creator of `:transactionId`, but then unconditionally deletes the approver identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other user's transaction by supplying their own `transactionId` and a victim's `approver.id`.

### Finding Description

**Vulnerability class**: Authorization bypass (cross-resource scope mismatch)

**Root cause — controller** (`back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, lines 102–113):

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver `id` with no scope check
  return true;
}
``` [1](#0-0) 

**Root cause — service** (`back-end/apps/api/src/transactions/approvers/approvers.service.ts`, lines 533–544):

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

`removeTransactionApprover` accepts only an `id` and performs no check that `approver.transactionId === transactionId`. The `getCreatorsTransaction` guard in the controller only proves the caller owns the URL's `transactionId`; it says nothing about the approver record being deleted.

By contrast, `updateTransactionApprover` correctly validates the scope at line 390:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The delete path is missing this equivalent guard.

**Exploit flow**:
1. Attacker authenticates as a normal user and creates any transaction → obtains `attackerTxId`.
2. Attacker learns a victim approver ID (`victimApproverId`) — sequential integers, or obtained by reading approvers of any terminal-status transaction (which `verifyAccess` exposes to all users per lines 789–798 of `transactions.service.ts`).
3. Attacker sends: `DELETE /transactions/{attackerTxId}/approvers/{victimApproverId}`.
4. `getCreatorsTransaction(attackerTxId, attacker)` passes — attacker owns `attackerTxId`.
5. `removeTransactionApprover(victimApproverId)` deletes the approver record belonging to the victim's transaction. [4](#0-3) 

### Impact Explanation

An attacker can silently remove required approvers from any transaction they do not own. This bypasses the multi-signature approval workflow: a transaction that required N approvals can be reduced to fewer, allowing it to proceed to execution without the intended governance controls. This is an unauthorized state mutation affecting transaction integrity for all organization users.

### Likelihood Explanation

Any authenticated, non-admin user can trigger this. No special privileges are required beyond having created one transaction. Approver IDs are sequential integers and are also discoverable via the GET approvers endpoint for transactions the attacker has legitimate access to (including all terminal-status transactions). The attack requires no race condition or timing dependency.

### Recommendation

Add a scope check in `removeTransactionApprover` (or in the controller before calling it) that asserts the approver being deleted belongs to the `transactionId` in the URL, mirroring the guard already present in `updateTransactionApprover`:

```typescript
// In removeTransactionApprover or the controller:
const rootNode = await this.getRootNodeFromNode(approver.id);
if (rootNode?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

### Proof of Concept

**Preconditions**: Two authenticated users — Alice (victim, owns transaction `txB` with approver `approverId=42`) and Bob (attacker, owns transaction `txA` with id `txAId=7`).

**Steps**:
```
# Bob sends:
DELETE /transactions/7/approvers/42
Authorization: Bearer <bob_jwt>

# Server response: 200 true
# Approver 42 (belonging to Alice's transaction txB) is now deleted.
```

**Expected**: 401/403 — approver 42 does not belong to transaction 7.
**Actual**: 200 — approver 42 is deleted from Alice's transaction, corrupting her approval workflow.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
