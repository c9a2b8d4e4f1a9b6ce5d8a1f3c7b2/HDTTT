Audit Report

## Title
`archiveTransaction` Bypasses Status Guard for Manual Transactions, Allowing State Corruption of Terminal Transactions

## Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` uses a boolean condition that completely skips the status validity check whenever `transaction.isManual` is `true`. This allows any authenticated creator of a manual transaction to archive it regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, and `REJECTED` — by calling the API endpoint directly.

## Finding Description

**Root cause:**

The guard at lines 711–718 of `transactions.service.ts` reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard throws only when **both** conditions are simultaneously true: the status is outside the allowed set **AND** the transaction is not manual. For any `isManual = true` transaction, `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false` — the guard never fires, and `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes unconditionally. [2](#0-1) 

The REST endpoint `PATCH /transactions/archive/:id` is exposed without any additional status guard at the controller level: [3](#0-2) 

**Frontend guard does not protect the backend:**

The frontend correctly restricts the archive button to in-progress transactions:

```typescript
const canArchive = computed(() => {
  const isManual = props.organizationTransaction?.isManual;
  return isManual && isCreator.value && transactionIsInProgress.value;
});
``` [4](#0-3) 

`transactionIsInProgress` is defined as: [5](#0-4) 

This is a client-side check only. The REST endpoint is reachable directly.

**Test gap confirms the bug:**

The existing unit test for `archiveTransaction` only covers the non-manual case (`isManual` is absent/falsy) for the rejection path, and the happy-path test uses `isManual: true` with `WAITING_FOR_EXECUTION` — a valid in-progress state. No test covers `isManual: true` with a terminal status: [6](#0-5) 

## Impact Explanation

A transaction creator can overwrite the status of any of their manual transactions — including `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, or `REJECTED` — to `ARCHIVED`. The `TransactionStatus` enum confirms these are all defined terminal states: [7](#0-6) 

The service itself recognizes these as terminal via `terminalStatuses`: [8](#0-7) 

Consequences:
- **Audit trail corruption**: An `EXECUTED` transaction (with an on-chain record) can be relabeled `ARCHIVED`, hiding its true outcome from other organization members and auditors.
- **Concealment of failures**: A `FAILED` or `CANCELED` transaction can be archived, masking the failure from the organization's history view.
- **State machine invariant violation**: Terminal states are supposed to be immutable; this allows a user-controlled state regression.

## Likelihood Explanation

The attacker only needs to be an authenticated organization user who created at least one manual transaction — a normal, unprivileged role. The exploit requires a single direct HTTP `PATCH` request to a documented endpoint. No special tooling, timing, or race condition is needed.

## Recommendation

The condition should be restructured so that the status check applies to **all** transactions, and the `isManual` flag only expands (not eliminates) the set of allowed statuses. For example:

```typescript
const allowedStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!allowedStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions genuinely need a broader set of archiveable statuses, those statuses should be explicitly enumerated rather than bypassing the guard entirely. A corresponding unit test should be added for `isManual: true` with each terminal status to prevent regression.

## Proof of Concept

1. Authenticate as a normal user (creator of a manual transaction).
2. Create a manual transaction via `POST /transactions` with `isManual: true`.
3. Wait for or simulate the transaction reaching `EXECUTED` (or any terminal status).
4. Send directly: `PATCH /transactions/archive/<id>` with valid session credentials.
5. The backend calls `archiveTransaction`; `!transaction.isManual` is `false`; the guard is skipped; `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes unconditionally.
6. The transaction status is now `ARCHIVED` despite having been in a terminal state.
