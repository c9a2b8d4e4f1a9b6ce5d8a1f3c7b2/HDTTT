### Title
Creator Can Archive a Terminal-State Manual Transaction, Corrupting the Audit Trail

### Summary
`archiveTransaction` in `transactions.service.ts` uses a logically flawed guard that short-circuits the status check whenever `transaction.isManual === true`. This allows the creator of any manual transaction to force it into `ARCHIVED` status regardless of its current state ŌĆö including `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, or `REJECTED`. The system explicitly intends to block this (a test asserts a 400 for executed transactions), but the test only covers non-manual transactions, leaving the manual-transaction path unguarded.

### Finding Description

**Root cause ŌĆö flawed boolean guard in `archiveTransaction`:** [1](#0-0) 

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ŌåÉ short-circuits the entire status check
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** conditions are true:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- AND `isManual` is `false`

When `isManual === true`, `!transaction.isManual` is `false`, so the entire `&&` expression is `false` and no error is thrown ŌĆö regardless of the current status. The function then unconditionally overwrites the status: [2](#0-1) 

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

**Exploit path:**
1. Creator submits a manual transaction (`isManual: true`) via `POST /transactions`.
2. Transaction is signed, reaches `WAITING_FOR_EXECUTION`, and is executed on the Hedera network ŌåÆ status becomes `EXECUTED`.
3. Creator calls `PATCH /transactions/archive/:id`.
4. The guard passes (because `isManual === true`), and the status is overwritten to `ARCHIVED`.

The same path works for `FAILED`, `EXPIRED`, `CANCELED`, and `REJECTED` manual transactions.

**The test that proves intent but misses the case:** [3](#0-2) 

The test sets a transaction to `EXECUTED` and expects a 400 ŌĆö but the transaction used (`addedTransactions.userTransactions[0]`) is not a manual transaction, so the `isManual` bypass is never exercised.

**The controller exposes this directly:** [4](#0-3) 

### Impact Explanation

- **Audit trail corruption**: An `EXECUTED` transaction (one that has already been submitted to and confirmed by the Hedera network) can have its status silently overwritten to `ARCHIVED`. Observers, signers, and admins who rely on the status field to determine whether a transaction was executed lose that information permanently.
- **Irreversibility**: There is no "unarchive" endpoint. Once overwritten, the `EXECUTED` status cannot be restored through normal API flows.
- **Notification pollution**: `emitTransactionStatusUpdate` fires after the overwrite, broadcasting the false `ARCHIVED` status to all subscribers. [5](#0-4) 

### Likelihood Explanation

- The attacker only needs to be the **creator** of a manual transaction ŌĆö a standard, unprivileged role available to any verified user.
- The endpoint `PATCH /transactions/archive/:id` is publicly reachable (JWT-authenticated, no admin role required).
- Creating a manual transaction is a normal workflow explicitly supported by the UI. [6](#0-5) 

### Recommendation

Replace the `&&` with `||` so that the status check is enforced independently of `isManual`, and explicitly enumerate which statuses are archiveable:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!archivableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions are intended to be archiveable from additional states, those states must be explicitly listed ŌĆö not implicitly allowed by bypassing the check entirely.

### Proof of Concept

1. Authenticate as a verified user.
2. Create a manual transaction:
   ```
   POST /transactions  { ..., isManual: true }
   ```
3. Upload a valid signature so the transaction reaches `WAITING_FOR_EXECUTION`.
4. Trigger execution:
   ```
   PATCH /transactions/execute/:id
   ```
   The transaction is submitted to Hedera; status becomes `EXECUTED`.
5. Immediately call:
   ```
   PATCH /transactions/archive/:id
   ```
6. **Expected (correct) behavior**: 400 error ŌĆö transaction is already executed.
7. **Actual behavior**: 200 OK ŌĆö status is overwritten to `ARCHIVED`. The `EXECUTED` record is permanently lost. [7](#0-6)
