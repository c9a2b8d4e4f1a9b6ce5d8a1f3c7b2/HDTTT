Audit Report

## Title
Missing Key Ownership Verification in `importSignatures` Allows Injection of Signatures from Keys Not Owned by the Authenticated User

## Summary

`POST /transactions/signatures/import` validates cryptographic correctness of submitted signatures but never verifies that the public keys in the signature map are registered to the authenticated user. Every other signature-submission path enforces key ownership via a `userKeyMap` lookup that throws `ErrorCodes.PNY` on mismatch. The import path skips this check entirely, allowing any user with `verifyAccess` to a transaction to inject a cryptographically valid signature produced by another user's private key directly into the persisted transaction bytes.

## Finding Description

**Guarded path — `POST /{id}/signers` → `signers.service.ts` → `processTransactionSignatures`**

For every public key in the submitted signature map, the service performs an ownership lookup against the calling user's registered keys: [1](#0-0) 

If the key is not found in `userKeyMap`, the call throws `ErrorCodes.PNY` and the signature is rejected.

**Unguarded path — `POST /transactions/signatures/import` → `transactions.controller.ts`**

The controller has no `@OnlyOwnerKey` decorator and passes the body directly to `importSignatures`: [2](#0-1) 

Inside `importSignatures` in `transactions.service.ts`, after `verifyAccess` and status/expiry checks, the only gate is `validateSignature`, which is a pure cryptographic check. There is no `userKeyMap` lookup: [3](#0-2) 

Any public key whose signature bytes are cryptographically valid over the transaction body is accepted and written into `transaction.transactionBytes` without consulting the database to confirm the key belongs to the caller.

**No `TransactionSigner` record is created**, so the injection leaves no audit trail linking the action to the attacker.

**Skipped test confirms a known gap on the approve path**

A test that should enforce key-ownership on the `approveTransaction` path is explicitly skipped: [4](#0-3) 

## Impact Explanation

An authenticated user who has `verifyAccess` to a transaction (observer, approver, or signer role) and who possesses a cryptographically valid signature produced by another user's private key can:

1. Call `POST /transactions/signatures/import` with that foreign signature.
2. The backend injects it directly into `transaction.transactionBytes` and persists it via the batch query builder.
3. No `TransactionSigner` record is created, so there is no audit trail.
4. If the injected signature satisfies the remaining threshold of a `KeyList` or `ThresholdKey`, the transaction can advance to `WAITING_FOR_EXECUTION` without the legitimate key owner having explicitly submitted their signature through the authorized path.

This breaks the system's invariant that "a signature is recorded only when the key owner explicitly submits it."

## Likelihood Explanation

The attacker preconditions are realistic:

- **Obtain a valid signature**: Hedera transaction files (`.tx2`) are routinely shared between organization members as part of the offline signing workflow. The frontend's `importSignaturesFromV2File` function is the primary consumer of this endpoint and reads signatures from files that may be shared over chat or email. [5](#0-4) 

- **Have access to the transaction**: Any observer, approver, or signer passes `verifyAccess`.
- **No cryptographic break required**: The attacker uses a legitimately produced signature; they only need to possess the bytes.

## Recommendation

Add a `userKeyMap` ownership check inside `importSignatures` in `transactions.service.ts`, mirroring the logic already present in `signers.service.ts` `processTransactionSignatures`:

```typescript
// After validateSignature succeeds, before addSignature:
for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  const owned = user.keys.some(
    k => k.publicKey === raw || k.publicKey === publicKey.toStringDer()
  );
  if (!owned) throw new BadRequestException(ErrorCodes.PNY);
  sdkTransaction.addSignature(publicKey, map);
}
```

This ensures the import path enforces the same ownership invariant as the `POST /{id}/signers` path. Additionally, the skipped test in `approvers.service.spec.ts` at line 1350 should be investigated and re-enabled.

## Proof of Concept

1. User A (legitimate signer) signs a transaction offline and shares the `.tx2` file with User B (an observer on the same transaction).
2. User B extracts the signature bytes from the file.
3. User B calls:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <User B's JWT>
   [{ "id": <transactionId>, "signatureMap": <User A's signature map> }]
   ```
4. The server calls `verifyAccess` → passes (User B is an observer).
5. The server calls `validateSignature` → passes (the signature is cryptographically valid over the transaction body).
6. `sdkTransaction.addSignature(publicKey, map)` is called with User A's public key — no ownership check.
7. The updated `transactionBytes` (now containing User A's signature) is persisted to the database.
8. No `TransactionSigner` row is inserted for User A; the action is attributed to no one in the audit log.
9. If User A's signature was the last required signature, the transaction status advances to `WAITING_FOR_EXECUTION` without User A's explicit consent through the authorized path. [6](#0-5)
