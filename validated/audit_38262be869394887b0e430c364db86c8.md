### Title
JWT Token Not Invalidated on Password Change Allows Persistent Unauthorized Access

### Summary
The `PATCH /auth/change-password` endpoint in the back-end API service changes a user's password but does not blacklist the current JWT token used to authenticate the request. As a result, any previously issued JWT tokens for that user remain valid for their full configured lifetime even after a password change. An attacker who has obtained a victim's JWT token cannot be evicted by the victim changing their password.

### Finding Description
The `changePassword` handler in `AuthController` only delegates to `authService.changePassword()` and returns, with no call to `blacklistService.blacklistToken()`:

```
// back-end/apps/api/src/auth/auth.controller.ts, lines 115-119
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto): Promise<void> {
  return this.authService.changePassword(user, dto);
}
```

Compare this to the `logout` and `setPassword` endpoints, which explicitly blacklist the token after the sensitive operation:

- `POST /auth/logout` (line 103): `await this.blacklistService.blacklistToken(extractJwtAuth(req));`
- `PATCH /set-password` (line 174): `await this.blacklistService.blacklistToken(extractJwtOtp(req));`
- `POST /verify-reset` (line 157): `await this.blacklistService.blacklistToken(extractJwtOtp(req));`

The `BlacklistService` stores invalidated tokens in Redis with a TTL equal to `JWT_EXPIRATION` days. The guard `JwtBlackListAuthGuard` checks this blacklist on every protected request. The mechanism works correctly everywhere it is applied — it is simply absent from `changePassword`.

The `authService.changePassword()` itself only validates the old password and calls `usersService.setPassword()`. No token invalidation occurs anywhere in that call chain.

**Exploit path:**
1. Attacker obtains victim's JWT (e.g., via network interception, XSS in a browser context, or a compromised device).
2. Victim detects the compromise and changes their password via `PATCH /auth/change-password` using their own current token.
3. The victim's token is blacklisted only if they also call `POST /auth/logout` — but the attacker's copy of the token is never blacklisted.
4. The attacker continues to use the stolen token for the full `JWT_EXPIRATION` window (configured in days) to perform any authenticated action: creating, signing, approving, and executing Hedera transactions; managing organization keys; reading all transaction history.

### Impact Explanation
An attacker holding a stolen JWT retains full authenticated access to the victim's organization account for the entire token lifetime after the victim changes their password. Concretely, the attacker can:
- Create and submit Hedera transactions (token transfers, account updates, file operations) on behalf of the victim.
- Sign multi-signature transactions, potentially meeting thresholds and triggering irreversible on-chain execution.
- Read all transaction history and key metadata.

The victim has no in-product mechanism to force-invalidate the attacker's copy of the token short of waiting for natural expiry.

### Likelihood Explanation
Token theft is a realistic, no-privilege attack vector. The front-end stores the JWT in `sessionStorage` (accessible to any same-origin JavaScript) and transmits it as a Bearer token. Any XSS, compromised browser extension, or network-level interception yields a usable token. The attacker needs no privileged access — only a valid JWT string. The window of exploitation equals `JWT_EXPIRATION` days, which is a configurable multi-day value.

### Recommendation
Blacklist the current request's JWT immediately after a successful password change, mirroring the pattern already used by `logout` and `setPassword`:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto, @Req() req: Request): Promise<void> {
  await this.authService.changePassword(user, dto);
  await this.blacklistService.blacklistToken(extractJwtAuth(req));
}
```

This ensures the token used to authenticate the password-change request is immediately invalidated, forcing re-authentication with the new password and evicting any attacker holding a copy of the old token.

### Proof of Concept
1. Authenticate as `victim@example.com` → receive `tokenA`.
2. Simulate attacker obtaining `tokenA` (copy it).
3. As victim, call `PATCH /auth/change-password` with `Authorization: Bearer tokenA`, body `{ oldPassword, newPassword }` → HTTP 200.
4. As attacker, call `GET /transactions/history` with `Authorization: Bearer tokenA` → HTTP 200 (still authorized).
5. For comparison, call `POST /auth/logout` with `Authorization: Bearer tokenA` → HTTP 200, then repeat step 4 → HTTP 401.

Step 4 demonstrates that the password change alone does not invalidate `tokenA`. The attacker retains full access.

  async blacklistToken(jwt: string) {
    const expirationDays = this.configService.get<number>('JWT_EXPIRATION');
    const expirationSeconds = Number(expirationDays) * 24 * 60 * 60;

    await this.client.set(jwt, this.BLACKLISTED, 'EX', expirationSeconds);
  }

  async isTokenBlacklisted(jwt: string) {
    const data = await this.client.get(jwt);
    return data === this.BLACKLISTED;
  }

  async changePassword(user: User, { oldPassword, newPassword }: ChangePasswordDto): Promise<void> {
    if (oldPassword === newPassword) throw new BadRequestException(ErrorCodes.NPMOP);

    const { correct } = await this.dualCompareHash(oldPassword, user.password);
    if (!correct) throw new BadRequestException(ErrorCodes.INOP);

    if (user.status === UserStatus.NEW && user.keys.length === 0) {
      emitUserStatusUpdateNotifications(this.notificationsPublisher, { entityId: user.id, additionalData: { username: user.email } });
    }

    await this.usersService.setPassword(user, newPassword);
  }
