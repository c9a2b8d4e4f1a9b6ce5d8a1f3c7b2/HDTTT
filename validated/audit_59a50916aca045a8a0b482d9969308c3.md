Audit Report

## Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Injection of Invalid Signatures, Causing Transaction Denial of Service

## Summary

The `processTransactionSignatures` method in `signers.service.ts` accepts and persists signature bytes into a transaction's stored `transactionBytes` without cryptographically verifying that the submitted signatures are valid over the transaction body. A parallel code path (`importSignatures`) performs this verification correctly. An authenticated organization member with a registered public key that is a required signer can inject garbage signature bytes, corrupting the stored transaction and causing it to fail when the chain service submits it to the Hedera network.

## Finding Description

**Path 1 — `POST /transactions/signatures/import` → `importSignatures`**

`importSignatures` in `transactions.service.ts` calls `validateSignature` before accepting any signature: [1](#0-0) 

`validateSignature` in `transaction.ts` performs a real cryptographic check — it calls `publicKey.verify(bodyBytes, signature)` and throws on failure: [2](#0-1) 

**Path 2 — `POST /transactions/:transactionId/signers` → `uploadSignatureMaps` → `processTransactionSignatures`**

`processTransactionSignatures` in `signers.service.ts` only verifies that the public key belongs to the authenticated user, then immediately calls `addSignature` with no cryptographic verification of the signature bytes: [3](#0-2) 

The Hedera SDK's `addSignature` is a simple insertion — it does not verify the signature bytes. It does, however, add the public key to `_signerPublicKeys`.

After `addSignature`, if the resulting bytes differ from the stored bytes, the corrupted bytes are persisted to the database: [4](#0-3) [5](#0-4) 

**The `IsSignatureMap` decorator** only validates structural format (valid account IDs, transaction IDs, non-empty byte arrays) — it performs no cryptographic verification: [6](#0-5) 

**The execution-time `hasValidSignatureKey` check** in `execute.service.ts` only checks that the required public key strings are present in `_signerPublicKeys` — it does not verify that the associated signature bytes are cryptographically valid: [7](#0-6) [8](#0-7) 

Because `addSignature` adds the key to `_signerPublicKeys` regardless of signature validity, this check passes even when garbage bytes are stored, and the transaction proceeds to submission on the Hedera network where it is rejected.

## Impact Explanation

An attacker who is an authenticated organization member with a registered public key that is a **required signer** for a transaction can:

1. Call `POST /transactions/:transactionId/signers` with their valid public key but with garbage/invalid signature bytes.
2. The server accepts the request, calls `addSignature` with the invalid bytes, detects `isSameBytes = false`, and persists the corrupted `transactionBytes` to the database. A `TransactionSigner` record is also inserted.
3. The `hasValidSignatureKey` check at execution time passes because the attacker's public key is now present in `_signerPublicKeys`.
4. When the chain service submits the transaction to the Hedera network, the network rejects it with `INVALID_SIGNATURE` because the stored signature bytes are cryptographically invalid.
5. The transaction is marked `FAILED`.

If the attacker's key is not a required signer, the garbage bytes are still stored but do not affect the threshold check — the impact is limited to storage corruption rather than execution failure.

## Likelihood Explanation

**Medium.** The attacker must be an authenticated organization member with a registered `UserKey` that is a required signer for the target transaction. This is not an unauthenticated attack. However, insider threats, compromised accounts, or malicious members of an organization are realistic adversaries in a multi-user transaction coordination system. No special administrative privileges are required — any signer-eligible user can exploit this.

## Recommendation

Add cryptographic signature verification in `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`. Specifically, call `validateSignature(sdkTransaction, map)` and reject the request if it throws. The `validateSignature` utility already exists at: [9](#0-8) 

The fix should be applied inside `processTransactionSignatures` in `signers.service.ts` before the `addSignature` call at line 251: [10](#0-9)

```typescript
export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```

```typescript
  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
```
