Audit Report

## Title
Transaction Creator Can Mutate Approver Structure While Transaction Is Actively Being Signed, Invalidating Completed Approvals

## Summary
The transaction creator can call `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` at any time — including while a transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status — because none of these mutation paths check the transaction's current status before applying changes. A malicious creator can raise the approval threshold, replace a signed approver (which explicitly clears their signature), or add new required approvers after other users have already completed their approvals, permanently blocking execution or invalidating work already done by approvers.

## Finding Description

**Root cause:** All three mutation paths gate access only on `getCreatorsTransaction`, which verifies the caller is the creator but performs **no check on the transaction's current status**.

**`createTransactionApprovers` — no status check:**

`getCreatorsTransaction` is the only guard called before the approver creation logic proceeds. [1](#0-0) 

**`updateTransactionApprover` — no status check, and explicitly clears signatures:**

`getCreatorsTransaction` is the only authorization check. When a `userId` replacement is performed, the service explicitly sets `signature: undefined`, `userKeyId: undefined`, and `approved: undefined`, erasing any completed approval. [2](#0-1) [3](#0-2) 

Threshold raises on tree nodes are also unrestricted: [4](#0-3) 

**`removeTransactionApprover` (controller) — no status check:**

Only `getCreatorsTransaction` is called before the approver is deleted. [5](#0-4) 

**`getCreatorsTransaction` itself — only checks ownership, not status:** [6](#0-5) 

**Contrast with `approveTransaction`**, which correctly enforces a status guard before allowing an approver to act: [7](#0-6) 

The asymmetry is the root cause: approvers are restricted by status, but the creator's mutation endpoints are not.

## Impact Explanation

**Scenario A — Threshold raise after approval threshold is met:**
1. Creator creates a transaction with `threshold=1`, approvers `[Alice, Bob]`.
2. Alice approves. The 1-of-2 threshold is satisfied.
3. Creator calls `PATCH /transactions/:id/approvers/:treeId` with `{ threshold: 2 }`.
4. Now 2-of-2 approvals are required. Only Alice has signed. The transaction is stuck in `WAITING_FOR_SIGNATURES` indefinitely.
5. The creator can repeat this pattern to permanently block execution.

**Scenario B — Approver replacement to invalidate a completed signature:**
1. Creator creates a transaction with `threshold=1`, approver `[Alice]`.
2. Alice reviews and approves (signs).
3. Creator calls `PATCH /transactions/:id/approvers/:aliceApproverId` with `{ userId: Charlie }`.
4. The service sets `signature: undefined, approved: undefined` on the approver record, erasing Alice's completed approval. [8](#0-7) 
5. Charlie has not signed. The transaction reverts to needing a new approval.

**Scenario C — Add new required approvers after existing approvers have signed:**
1. Creator creates a transaction with `threshold=1`, approver `[Alice]`.
2. Alice approves.
3. Creator calls `POST /transactions/:id/approvers` to add a new root-level approver `Bob`.
4. Now both Alice's and Bob's approvals are independently required. Bob has not signed. Execution is blocked.

The impact is **permanent denial of execution** for a transaction that legitimate approvers have already completed their obligations on, and **invalidation of approver work**.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The attack requires only:
1. Creating a transaction (normal user action).
2. Waiting for approvers to sign (observable via the notification/status system).
3. Calling a standard `PATCH` or `POST` endpoint with a valid JWT.

No leaked credentials, no admin access, and no cryptographic break are required. The API endpoints are fully reachable by any authenticated user who is the creator of a transaction. The attack is deterministic and repeatable.

## Recommendation

Add a status guard at the beginning of `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` (or inside `getCreatorsTransaction` as a shared check). The guard should reject mutations when the transaction's status is `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any terminal state. This mirrors the existing guard already present in `approveTransaction`: [7](#0-6) 

Specifically, approver structure mutations should only be permitted when the transaction is in `WAITING_FOR_SIGNATURES` status **before any approver has signed**, or ideally only in a dedicated `DRAFT`/`NEW` status if one exists.

## Proof of Concept

```
# Step 1: Creator creates transaction with threshold=1, approver Alice
POST /transactions/:id/approvers
{ "approversArray": [{ "userId": <alice_id> }] }

# Step 2: Alice approves (signs) — transaction threshold is now met
POST /transactions/:id/approvers/approve
{ "userKeyId": <key_id>, "signature": "<sig>", "approved": true }

# Step 3: Creator replaces Alice with Charlie, clearing Alice's signature
PATCH /transactions/:id/approvers/:alice_approver_id
{ "userId": <charlie_id> }
# Result: approver record now has signature=null, approved=null
# Transaction reverts to WAITING_FOR_SIGNATURES with no valid approvals

# Step 4 (alternative): Creator raises threshold after Alice signed
PATCH /transactions/:id/approvers/:tree_node_id
{ "threshold": 2 }
# Result: 2-of-N now required; only 1 signature exists; execution permanently blocked
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
