### Title
`getSignaturesByTransactionId` Endpoint Exposes Signer Data Without Access Control

### Summary
The `GET /transactions/:transactionId/signers` endpoint in `SignersController` performs no authorization check to verify that the requesting user has any relationship to the queried transaction. Any authenticated, verified user can enumerate signers and their associated public keys for every transaction in the system, regardless of whether they are a creator, observer, signer, or approver of that transaction. This is the direct analog of the external report: a sensitive retrieval function that authenticates the caller but does not verify the caller is *authorized* to access the resource.

### Finding Description

**Root cause:**

In `back-end/apps/api/src/transactions/signers/signers.controller.ts`, the `getSignaturesByTransactionId` handler passes the caller-supplied `transactionId` URL parameter directly to the service with no ownership or membership check:

```typescript
@Get()
@HttpCode(200)
getSignaturesByTransactionId(
  @Param('transactionId', ParseIntPipe) transactionId: number,
): Promise<TransactionSigner[]> {
  return this.signaturesService.getSignaturesByTransactionId(transactionId, true);
}
``` [1](#0-0) 

The service method performs no access check — it queries all signers for the given transaction ID and eagerly loads the `userKey` relation (which includes public key material):

```typescript
getSignaturesByTransactionId(
  transactionId: number,
  withDeleted: boolean = false,
): Promise<TransactionSigner[]> {
  return this.repo.find({
    where: { transaction: { id: transactionId } },
    relations: { userKey: true },
    withDeleted,
  });
}
``` [2](#0-1) 

**Contrast with correct patterns in the same codebase:**

Every other sensitive transaction data retrieval in the codebase performs an explicit access check:

- `getVerifiedApproversByTransactionId` checks creator, observer, signer, and approver membership before returning approver data. [3](#0-2) 

- `importSignatures` calls `verifyAccess(transaction, user)` before processing any signature data. [4](#0-3) 

- `verifyAccess` itself enforces that the user must be a creator, observer, signer, or approver of the transaction (for non-terminal statuses). [5](#0-4) 

The `getSignaturesByTransactionId` endpoint is the only sensitive transaction data endpoint that skips this check entirely.

**Exploit path:**

1. Attacker registers and verifies an account (standard user, no admin).
2. Attacker calls `GET /transactions/1/signers`, `GET /transactions/2/signers`, … iterating over sequential integer IDs.
3. For each transaction, the server returns all `TransactionSigner` records with the `userKey` relation loaded — revealing which user keys signed which transactions.
4. The `withDeleted: true` flag means even soft-deleted signer records are returned, exposing historical signing activity. [6](#0-5) 

### Impact Explanation

A malicious authenticated user can enumerate the full signing history of every transaction in the organization — including transactions they have no legitimate access to. The response includes `userKey` data (public keys and key IDs) tied to specific users and transactions. This breaks the cross-tenant isolation model of the multi-user tool: a regular employee can learn which executives or council members signed which transactions, reconstruct signing timelines, and correlate public keys to user identities across the entire transaction history.

### Likelihood Explanation

The attack requires only a valid, verified account — the lowest privilege level in the system. Transaction IDs are sequential integers, making enumeration trivial. No special tooling, timing, or cryptographic capability is needed. The endpoint is a standard REST GET request.

### Recommendation

Add an access check in `getSignaturesByTransactionId` (or in the controller handler) that verifies the requesting user is a participant in the transaction before returning signer data, consistent with the pattern used by `getVerifiedApproversByTransactionId` and `verifyAccess`. For non-terminal transactions, restrict access to creators, observers, signers, and approvers. For terminal transactions, the existing codebase already grants broad read access via `verifyAccess`, so that policy can be preserved if desired.

### Proof of Concept

**Preconditions:** Two accounts exist — `victim` (creator of transaction ID 5) and `attacker` (no relationship to transaction 5).

**Steps:**
1. Authenticate as `attacker`, obtain JWT.
2. Send: `GET /transactions/5/signers` with `Authorization: Bearer <attacker_jwt>`
3. Server responds `200 OK` with the full list of `TransactionSigner` records for transaction 5, including `userKey` (public key, key ID) for every user who signed it.
4. Repeat for all transaction IDs to enumerate the entire organization's signing history.

**Expected (correct) behavior:** `401 Unauthorized` — attacker has no relationship to transaction 5.

**Actual behavior:** `200 OK` with full signer and public key data. [7](#0-6) [1](#0-0)
