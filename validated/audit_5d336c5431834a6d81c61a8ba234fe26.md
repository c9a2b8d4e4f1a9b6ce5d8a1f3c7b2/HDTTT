Audit Report

## Title
Unbounded Loop Over Dynamic Transaction Array in `getTransactionsToSign` Causes Server-Side DoS

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every result performing per-transaction async operations (`SDKTransaction.fromBytes`, `computeSignatureKey`) before applying pagination. A separately acknowledged missing input-size guard exists in `importSignatures`. Both paths are reachable by any authenticated user.

## Finding Description

**Root cause 1 — `getTransactionsToSign` (primary)**

The `whereForUser` clause at lines 266–277 filters only by `status` — it does **not** scope the query to the requesting user's transactions. The `repo.find()` call at lines 295–299 carries no `take` limit, loading the entire active transaction table into the Node.js process: [1](#0-0) 

The subsequent loop at lines 301–309 calls `userKeysToSign` for every row: [2](#0-1) 

Each call to `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs:
- `SDKTransaction.fromBytes` (CPU deserialization of transaction bytes)
- `transactionSignatureService.computeSignatureKey` (a database-backed computation) [3](#0-2) 

Pagination is applied only **after** all N operations complete: [4](#0-3) 

By contrast, every other paginated method in the same service (`getTransactions`, `getHistoryTransactions`) correctly passes `skip`/`take` to the ORM: [5](#0-4) [6](#0-5) 

**Root cause 2 — `importSignatures` (secondary, developer-acknowledged)**

The developer comment at line 575 explicitly acknowledges the missing guard: [7](#0-6) 

The function accepts `dto: UploadSignatureMapDto[]` with no enforced maximum length. For each element the loop at line 525 calls `verifyAccess` (which internally calls `userKeysToSign`), `SDKTransaction.fromBytes`, and `validateSignature` — all proportional to the caller-controlled array size: [8](#0-7) 

The `BATCH_SIZE = 500` at line 576 governs only the SQL bulk-update step; it does **not** limit the number of elements processed in the loop above it.

## Impact Explanation

- **`getTransactionsToSign`**: A single API call loads the entire active transaction table into the Node.js heap and fires N sequential async operations against the database. In an organization with thousands of staged transactions, repeated calls from one or more authenticated users can saturate the database connection pool, spike server memory, and render the endpoint unresponsive for all users.
- **`importSignatures`**: A caller submitting an array of thousands of IDs forces the server to perform thousands of cryptographic deserialization and signature-verification operations in a single synchronous request loop, causing CPU exhaustion.

## Likelihood Explanation

- Any registered user with at least one key can call `getTransactionsToSign`. The `whereForUser` clause does not filter by user, so the query always scans the full active transaction table regardless of the caller's actual involvement.
- `importSignatures` accepts a plain JSON array; no schema-level `@ArrayMaxSize` or equivalent decorator is enforced on `UploadSignatureMapDto[]`.
- No rate limiting or input-size guard is present in the service layer for either path.

## Recommendation

**`getTransactionsToSign`:**
- Add `take: limit` and `skip: offset` to the `repo.find()` call, mirroring the pattern used in `getTransactions` and `getHistoryTransactions`.
- Pre-filter the query to only return transactions where the user's public keys appear in `publicKeys` (the stored key list on the transaction), eliminating the need for a full-table scan.

**`importSignatures`:**
- Enforce a maximum array length on `UploadSignatureMapDto[]` at the DTO/controller level (e.g., NestJS `@ArrayMaxSize(N)` decorator).
- Reject requests exceeding the limit with a `400 Bad Request` before any processing begins.

## Proof of Concept

**`getTransactionsToSign`:**
1. Register an authenticated user with at least one key.
2. Ensure the organization has a large number of active (non-terminal) transactions (e.g., 10,000 rows in `WAITING_FOR_SIGNATURES` status).
3. Send repeated `GET /transactions/sign` requests as the authenticated user.
4. Observe: each request triggers a full-table `SELECT` with no `LIMIT`, followed by 10,000 sequential calls to `computeSignatureKey` (each involving a DB round-trip). Server CPU and DB connection pool are exhausted; subsequent requests time out.

**`importSignatures`:**
1. Authenticate as any user.
2. Send `POST /transactions/signatures` with a JSON body containing an array of thousands of valid transaction IDs.
3. Observe: the server iterates over all elements, calling `SDKTransaction.fromBytes` and `validateSignature` for each, causing CPU saturation for the duration of the request.
