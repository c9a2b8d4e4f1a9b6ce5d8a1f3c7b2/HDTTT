### Title
Unauthorized Approver Deletion via Missing Ownership Validation in `removeTransactionApprover`

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the calling user is the creator of the transaction identified by `:transactionId`, but never validates that the approver identified by `:id` actually belongs to that transaction. Any authenticated user who has created at least one transaction can delete approvers from any other user's transaction by supplying their own `transactionId` and an arbitrary approver `id`.

### Finding Description

**Vulnerability class**: Invalid Validation / Missing entity-ownership check — the same class as the Lavarage H-02 finding.

**Root cause** — two-step authorization that checks the wrong object:

In `approvers.controller.ts`, the delete handler is:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver id, no ownership check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the URL-supplied `transactionId`. It says nothing about whether approver `id` belongs to that transaction.

`removeTransactionApprover` in the service then deletes whatever approver matches `id`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

There is no check that `approver.transactionId === transactionId`. Compare this with `updateTransactionApprover`, which **does** perform this check:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The update path is protected; the delete path is not. This is the exact same pattern as Lavarage H-02: existence is checked (`approver` must exist), but ownership is not.

### Impact Explanation

An attacker can silently remove approvers from any transaction in the system:

- **Approval workflow bypass**: If a transaction requires a threshold of approvers before it can proceed, removing approvers can reduce or eliminate that requirement, allowing the transaction to execute without proper authorization.
- **Unauthorized state mutation**: The attacker modifies another user's transaction state without being the creator of that transaction.
- **Irreversible disruption**: Soft-deleted approvers (via `removeNode`) cannot be trivially restored; the approval tree is permanently altered.

Severity: **High** — unauthorized state change affecting the integrity of the multi-signature approval workflow.

### Likelihood Explanation

- **Precondition**: The attacker must be an authenticated, verified user who has created at least one transaction (to pass the `getCreatorsTransaction` check). This is a normal user capability.
- **Discovery**: Approver IDs are sequential integers assigned by the database. An attacker can enumerate them or observe IDs from their own approvers.
- **No special privilege required**: No admin, operator, or leaked credentials needed.
- **Realistic**: Any malicious organization member can exploit this against any other member's pending transactions.

### Recommendation

In `removeTransactionApprover` (service), after fetching the approver, verify that its root transaction matches the `transactionId` parameter — exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Add this ownership check (mirrors updateTransactionApprover)
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  ...
}
```

Pass `transactionId` from the controller into the service call. This mirrors the protection already present in `updateTransactionApprover`. [2](#0-1) 

### Proof of Concept

**Setup**: Two users, Alice (attacker) and Bob (victim), both authenticated in the same organization.

1. Bob creates Transaction B (`id=99`) and adds an approver (`approver id=42`, `userId=Carol`).
2. Alice creates Transaction A (`id=7`) — she is its creator.
3. Alice sends:
   ```
   DELETE /transactions/7/approvers/42
   Authorization: Bearer <Alice's JWT>
   ```
4. Server executes:
   - `getCreatorsTransaction(7, Alice)` → passes (Alice owns tx 7).
   - `removeTransactionApprover(42)` → deletes approver 42 (Carol on tx 99) with no ownership check.
5. **Result**: Carol is no longer an approver on Bob's Transaction B. Bob's approval threshold is silently broken. Transaction B may now proceed without Carol's required approval.
