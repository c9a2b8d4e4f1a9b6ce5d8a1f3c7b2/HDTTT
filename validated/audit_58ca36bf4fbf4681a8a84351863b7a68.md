Audit Report

## Title
Unbounded `groupItems` Array in `POST /transaction-groups` Enables Authenticated DoS via Resource Exhaustion

## Summary
`CreateTransactionGroupDto` imposes no upper bound on the `groupItems` array. A single authenticated HTTP request with thousands of items causes the server to spawn N concurrent CPU-heavy cryptographic operations via `Promise.all`, exhausting the Node.js event loop, memory, and PostgreSQL connection pool. The developers themselves flagged the missing limit in a comment on the analogous `importSignatures` path.

## Finding Description

**Root cause — no `@ArrayMaxSize()` on `groupItems`:**

`CreateTransactionGroupDto` accepts an unbounded array with no size constraint:

```typescript
@IsArray()
@IsNotEmpty()
@ValidateNested({ each: true })
@Type(() => CreateTransactionGroupItemDto)
groupItems: CreateTransactionGroupItemDto[];   // ← no @ArrayMaxSize()
``` [1](#0-0) 

**Exploit path — `Promise.all` over N concurrent heavy operations:**

`createTransactionGroup` maps all N items to transaction DTOs and passes them to `createTransactions`: [2](#0-1) 

`createTransactions` fans out all N validations **concurrently** via `Promise.all`:

```typescript
const validatedData = await Promise.all(
  dtos.map(dto => this.validateAndPrepareTransaction(dto, user, client)),
);
``` [3](#0-2) 

Each `validateAndPrepareTransaction` call performs:
- `publicKey.verify(dto.transactionBytes, dto.signature)` — elliptic-curve signature verification
- `SDKTransaction.fromBytes(dto.transactionBytes)` — protobuf deserialization
- `isTransactionBodyOverMaxSize(sdkTransaction)` — second deserialization pass [4](#0-3) 

All N results are then saved inside a single database transaction, holding a DB connection for the full duration. [5](#0-4) 

**Developer acknowledgment of the missing limit** on the analogous `importSignatures` path:

```typescript
//Added a batch mechanism, probably should limit this on the api side of things
const BATCH_SIZE = 500;
``` [6](#0-5) 

The same unbounded-array pattern exists on `POST /transactions/signatures/import` (`importSignatures`), which accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array-size cap. [7](#0-6) 

## Impact Explanation
A single authenticated user (no admin role required) can submit one HTTP request to `POST /transaction-groups` with thousands of `groupItems`. The server spawns N concurrent CPU-heavy crypto operations via `Promise.all`, exhausting the Node.js event loop and available memory. The long-running database transaction also holds a PostgreSQL connection for the full duration, starving the connection pool. The result is severe response-time degradation or an OOM crash affecting all concurrent users of the API service. [8](#0-7) 

## Likelihood Explanation
The attacker precondition is a valid user account — the lowest privilege level in the system. The endpoint is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no rate-limiting or array-size enforcement is present. No special tooling is required; a single crafted `POST` request is sufficient. Even within the default Express 100 KB body limit, ~100–150 items each requiring elliptic-curve verification is sufficient to cause measurable latency spikes. [9](#0-8) 

## Recommendation
1. Add `@ArrayMaxSize(N)` (e.g., `N = 50`) to `groupItems` in `CreateTransactionGroupDto`.
2. Apply the same constraint to `UploadSignatureMapDto[]` on the `importSignatures` and `uploadSignatureMaps` endpoints.
3. Process items in sequential batches (as the developer comment on line 575 already suggests) rather than a single unbounded `Promise.all`.
4. Consider adding API-level rate limiting per user on mutation endpoints.

## Proof of Concept
```http
POST /transaction-groups HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "description": "dos",
  "atomic": false,
  "sequential": false,
  "groupItems": [
    { "transaction": { /* valid CreateTransactionDto */ } },
    // ... repeated 500+ times
  ]
}
```
The server will invoke `validateAndPrepareTransaction` 500+ times concurrently via `Promise.all`, each performing EC signature verification and protobuf deserialization, blocking the event loop and holding a DB connection until all complete or the process OOMs. [10](#0-9)
