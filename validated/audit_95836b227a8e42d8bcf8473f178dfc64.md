Audit Report

## Title
Any Authenticated User Can Spoof a Privileged Fee Payer Account to Bypass the 6 KB Transaction Size Limit

## Summary
The backend's transaction size validation in `validateAndPrepareTransaction` derives the allowed size limit from the fee payer account embedded in the attacker-controlled `TransactionId`. Because the backend never verifies that the submitting user actually controls the fee payer account, any authenticated user can set the fee payer to a privileged governance account (e.g., `0.0.2` or any account in `0.0.42–0.0.799`) and have their transaction accepted at the 128 KB limit instead of the 6 KB limit.

## Finding Description

**Root cause — fee payer is attacker-controlled and never ownership-verified**

`validateAndPrepareTransaction` performs two checks before the size gate:

1. `creatorKeyId` belongs to the authenticated user's registered keys.
2. The user's public key signed `dto.transactionBytes`.

Neither check verifies that the fee payer account embedded in the `TransactionId` is owned or controlled by the submitting user. [1](#0-0) 

The size check at line 929 calls `isTransactionBodyOverMaxSize`, which reads the fee payer directly from `sdkTransaction.transactionId.accountId`: [2](#0-1) 

`isTransactionBodyOverMaxSize` is defined as: [3](#0-2) 

`getMaxTransactionSizeForTransaction` derives the limit from the fee payer extracted from the SDK transaction itself: [4](#0-3) 

`isPrivilegedFeePayer` grants 128 KB to accounts `0.0.2` and `0.0.42–0.0.799`: [5](#0-4) 

The `TransactionId` is fully attacker-controlled. Any Hedera SDK call such as `TransactionId.withValidStart(AccountId.fromString('0.0.2'), validStart)` sets the fee payer to the treasury account without requiring any cryptographic proof of ownership.

## Impact Explanation

- **Storage exhaustion / resource abuse**: Any authenticated user can store transactions up to 128 KB instead of 6 KB — a 21× amplification factor per transaction. Repeated submissions fill the PostgreSQL `transactionBytes` / `unsignedTransactionBytes` `bytea` columns far beyond the intended quota.
- **Integrity violation**: The 6 KB limit is a security invariant protecting the backend from oversized payloads. Spoofing a privileged fee payer silently nullifies it for any user.
- **Downstream chain-service waste**: The chain service will attempt to execute these transactions; they will fail on the Hedera network (the real fee payer `0.0.2` never signed), but the backend has already consumed storage and processing resources.
- No fund loss, but permanent database bloat and potential denial-of-service against the organization backend.

## Likelihood Explanation

- **Attacker precondition**: Only a valid authenticated account on the organization backend — no admin, no special role required.
- **Effort**: Trivial. The Hedera SDK makes it a one-liner to set an arbitrary fee payer in a `TransactionId`. No cryptographic material for the spoofed account is needed.
- **Detectability**: The backend logs no warning when a non-privileged user submits a transaction with a privileged fee payer. The attack is silent.
- **Realistic scenario**: Any disgruntled or malicious organization member can execute this immediately after registration.

## Recommendation

Before the size check in `validateAndPrepareTransaction`, verify that the fee payer account in the `TransactionId` matches an account registered to the submitting user, or alternatively, ignore the fee payer when computing the size limit for user-submitted transactions and always apply the standard 6 KB limit unless the user has been explicitly granted privileged status by an administrator.

Concretely, in `validateAndPrepareTransaction`: [2](#0-1) 

Replace the call to `isTransactionBodyOverMaxSize(sdkTransaction)` with a version that either:
- Always uses `MAX_TRANSACTION_BYTE_SIZE` (6 KB) regardless of fee payer, or
- Checks whether the fee payer account is registered to the authenticated user before granting the elevated limit.

## Proof of Concept

```typescript
import {
  FileUpdateTransaction,
  TransactionId,
  AccountId,
  Timestamp,
  PrivateKey,
} from '@hiero-ledger/sdk';

// Step 1: Attacker has a registered key pair on the backend
const attackerKey = PrivateKey.generateED25519();

// Step 2: Construct a transaction with fee payer = 0.0.2 (treasury)
const spoofedFeePayer = AccountId.fromString('0.0.2');
const validStart = Timestamp.fromDate(new Date());
const txId = TransactionId.withValidStart(spoofedFeePayer, validStart);

const tx = new FileUpdateTransaction()
  .setTransactionId(txId)
  .setFileId('0.0.111')
  .setContents(Buffer.alloc(100_000)) // ~100 KB payload
  .setNodeAccountIds([AccountId.fromString('0.0.3')])
  .freeze();

// Step 3: Sign with attacker's own key (no 0.0.2 key needed)
const txBytes = tx.toBytes();
const signature = attackerKey.sign(txBytes);

// Step 4: POST /transactions with creatorKeyId = attacker's registered key ID
// Backend checks:
//   - creatorKeyId belongs to attacker: PASS
//   - signature valid: PASS (attacker signed with their own key)
//   - isTransactionBodyOverMaxSize: reads fee payer 0.0.2 → limit = 131072 → 100 KB < 128 KB → PASS
// Result: 100 KB transaction persisted to database
```

export function isPrivilegedFeePayer(accountId: AccountId | null | undefined): boolean {
  if (!accountId) return false;
  if (accountId.shard.toNumber() !== 0 || accountId.realm.toNumber() !== 0) return false;
  const num = accountId.num.toNumber();
  return num === 2 || (num >= 42 && num <= 799);
}
