Audit Report

## Title
Missing User-Based Authorization on `getTransactionGroups` Endpoint Exposes All Organization Transaction Groups

## Summary
The `GET /transaction-groups` endpoint in `TransactionGroupsController` is explicitly marked "TESTING ONLY" in source code but is deployed with no user-scoping or admin restriction. Any authenticated, verified organization member can enumerate every transaction group in the organization regardless of their relationship to those groups.

## Finding Description

The `getTransactionGroups()` handler at line 55 of `transaction-groups.controller.ts` carries only the controller-level guard stack (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) applied at line 27, which verify authentication but impose no ownership or role restriction. [1](#0-0) 

The handler accepts no `@GetUser()` parameter and passes no user context to the service layer. The service method performs an unfiltered database query returning every `TransactionGroup` record: [2](#0-1) 

Every other sensitive endpoint in the same controller correctly accepts `@GetUser() user: User` and passes it to the service for ownership verification: [3](#0-2) [4](#0-3) 

The `AdminGuard` exists and is applied to sensitive admin-only operations elsewhere (e.g., `PATCH /users/:id`, `DELETE /users/:id`, `POST /auth/signup`) but is absent here: [5](#0-4) [6](#0-5) 

The `TransactionGroupDto` serializer exposes `id`, `description`, `atomic`, `sequential`, `createdAt`, `groupValidTime`, and `groupItems` for every group returned: [7](#0-6) 

## Impact Explanation

Any authenticated organization member can issue `GET /transaction-groups` and receive the full list of every transaction group ever created in the organization. The response includes group IDs, human-readable descriptions, atomic/sequential flags, and creation timestamps — organizational metadata that may reveal business-sensitive information (e.g., batch payment groupings, operational cadence). The exposed group IDs can also be used to probe `GET /transaction-groups/:id`, which does enforce user-based authorization but still represents an unnecessary attack surface expansion. The developer comment `/* TESTING ONLY */` confirms the intent was never to expose this endpoint to all users in production.

## Likelihood Explanation

The route is a standard REST `GET` on the root path of the `transaction-groups` controller. The NestJS Swagger integration (`@ApiTags('Transaction Groups')`) makes it discoverable to any user who browses the API documentation. No obscure path, parameter, or header is required — a single authenticated HTTP request suffices.

## Recommendation

Either:
1. **Remove the endpoint entirely**, as the developer comment explicitly marks it "TESTING ONLY" and it has no documented production use case.
2. **Restrict it to admins** by adding `@UseGuards(AdminGuard)` at the handler level, consistent with how other privileged operations are guarded.
3. **Scope it to the requesting user** by adding `@GetUser() user: User` and filtering the query to groups the user is a creator, signer, approver, or observer of — consistent with how `getTransactionGroup` (singular) is implemented.

```http
Host: <org-api-host>
Authorization: Bearer <any_valid_user_jwt>
```

Any authenticated user receives all groups regardless of their role or relationship to those groups. The `/* TESTING ONLY */` comment at line 52 of the controller confirms this was never intended for production exposure. [8](#0-7)

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class TransactionGroupsController {
  constructor(private readonly transactionGroupsService: TransactionGroupsService) {}

  /* Submit a transaction group */
  @ApiOperation({
    summary: 'Create a transaction group',
    description:
      'Create a transaction group for the organization. ' +
      'The group contains group items that each point to a transaction ' +
      'that the organization is to approve, sign, and execute.',
  })
  @ApiResponse({
    status: 201,
    type: TransactionGroupDto,
  })
  @Post()
  @Serialize(TransactionGroupDto)
  createTransactionGroup(
    @GetUser() user: User,
    @Body() dto: CreateTransactionGroupDto,
  ): Promise<TransactionGroup> {
    return this.transactionGroupsService.createTransactionGroup(user, dto);
  }

  /* TESTING ONLY: Get all transactions groups */
  @Get()
  @Serialize(TransactionGroupDto)
  getTransactionGroups(): Promise<TransactionGroup[]> {
    return this.transactionGroupsService.getTransactionGroups();
  }
```
