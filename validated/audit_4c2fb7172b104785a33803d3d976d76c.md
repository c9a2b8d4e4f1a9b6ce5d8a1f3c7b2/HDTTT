## Audit Report

## Title
Unbounded Iteration Over User-Controlled Arrays Causes Application-Layer DoS

## Summary
Two confirmed entry points allow any authenticated, verified user to trigger unbounded server-side processing — sequential async DB queries per transaction in `getTransactionsToSign`, and per-element cryptographic deserialization in `importSignatures` — with no input size cap enforced at the API layer. Two additional supporting vectors exist in `createTransactionGroup` and `createTransactionObservers`.

## Finding Description

**Vector 1 — `GET /transactions/sign` (`getTransactionsToSign`)**

`getTransactionsToSign` in `transactions.service.ts` issues a `repo.find()` at line 295 with no `take` (LIMIT) clause, fetching every non-terminal transaction in the database. It then iterates over the full result set in a sequential `for` loop at line 301, awaiting one `userKeysToSign` call per transaction (which itself performs async DB queries). Pagination is applied only at line 313 on the already-fully-processed in-memory result. [1](#0-0) 

The controller exposes this as `GET /sign` and is annotated `/* NO LONGER USED BY FRONT-END */` but remains fully reachable. [2](#0-1) 

**Vector 2 — `POST /transactions/signatures/import` (`importSignatures`)**

`importSignatures` accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no `@ArrayMaxSize()` constraint on the outer array. The `UploadSignatureMapDto` class itself only validates individual field types. [3](#0-2) 

The controller passes the array directly to the service with no size check. [4](#0-3) 

Inside `importSignatures`, for each element the service calls `SDKTransaction.fromBytes()` (SDK deserialization) and `validateSignature` (cryptographic verification) in a sequential `for` loop at line 525. The developer explicitly acknowledged the missing API-side limit in the comment at line 575: [5](#0-4) 

Note: the DB writes are partially mitigated by a `BATCH_SIZE = 500` mechanism, but the CPU-intensive cryptographic work per element is entirely unbounded.

**Supporting — `POST /transaction-groups` (`createTransactionGroup`)**

`CreateTransactionGroupDto.groupItems` has `@IsArray()` and `@IsNotEmpty()` but no `@ArrayMaxSize()`. [6](#0-5) 

`createTransactions` fans out to `Promise.all` over `validateAndPrepareTransaction` for every item, each performing SDK parse, signature verification, and a network call. [7](#0-6) 

**Supporting — `POST /transactions/:id/observers` (`createTransactionObservers`)**

`CreateTransactionObserversDto.userIds` has `@ArrayMinSize(1)` but no `@ArrayMaxSize()`. [8](#0-7) 

The service iterates over every `userId` in a `for` loop before a single bulk save. [9](#0-8) 

## Impact Explanation

- **Vector 1**: A single authenticated call to `GET /transactions/sign` when the system holds thousands of non-terminal transactions causes the API process to execute thousands of sequential async DB queries, blocking the Node.js event loop and exhausting the DB connection pool for all concurrent users. The endpoint is deprecated from the front-end but remains fully reachable via the API.
- **Vector 2**: A single `POST /transactions/signatures/import` request with an array of thousands of elements triggers thousands of `SDKTransaction.fromBytes` deserializations and `validateSignature` cryptographic operations in one request, saturating the CPU and blocking the event loop.
- Both vectors are reachable by any authenticated, verified user — no admin role is required.

## Likelihood Explanation

Any registered user can authenticate and call these endpoints. Vector 2 requires only a valid JWT and a large JSON body; no special on-chain state is needed. Vector 1 is amplified naturally as the platform accumulates transactions. The developer comment at line 575 of `transactions.service.ts` confirms the missing limit was a known, unaddressed gap. [10](#0-9) 

## Recommendation

1. **Vector 1**: Add a `take: limit` clause to the `repo.find()` call in `getTransactionsToSign` and restructure the query to filter at the DB level (e.g., via a subquery or join) rather than post-processing in memory. Alternatively, deprecate and remove the endpoint entirely since it is no longer used by the front-end.
2. **Vector 2**: Add `@ArrayMaxSize(N)` to the outer array accepted by `importSignatures` (the developer comment at line 575 explicitly calls this out). A reasonable cap (e.g., 100–500) should be enforced.
3. **Supporting vectors**: Add `@ArrayMaxSize(N)` to `CreateTransactionGroupDto.groupItems` and `CreateTransactionObserversDto.userIds`.

## Proof of Concept

**Vector 1:**
```
GET /transactions/sign?page=1&size=10
Authorization: Bearer <valid_jwt>
```
With thousands of non-terminal transactions in the DB, this request will execute thousands of sequential async DB queries before returning, blocking the event loop.

**Vector 2:**
```
POST /transactions/signatures/import
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 2, "signatureMap": { ... } },
  ... (thousands of entries)
]
```
Each entry triggers `SDKTransaction.fromBytes` + `validateSignature` with no server-side size cap, exhausting CPU for the duration of the request.
