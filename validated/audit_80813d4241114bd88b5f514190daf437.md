Audit Report

## Title
`updateTransactionApprover` Allows Duplicate `userId` Entries in Approver List, Enabling Threshold Bypass

## Summary
The `updateTransactionApprover` function in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` does not check for duplicate `userId` values when updating an approver's user. This allows a transaction creator to overwrite one approver's `userId` with a `userId` already present in the same transaction's approver list. Because `approveTransaction` bulk-updates all approver records matching the acting user's ID, a single user's approval can satisfy multiple approver slots, bypassing the configured threshold.

## Finding Description

**Root cause — missing duplicate check in `updateTransactionApprover`:**

`createTransactionApprovers` correctly guards against duplicates at line 250:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
``` [1](#0-0) 

The `isNode()` helper (lines 647–666) counts existing `TransactionApprover` rows matching the same `listId`, `userId`, `threshold`, and `transactionId`, returning `true` only when a duplicate user-type approver is found. [2](#0-1) 

However, the `userId` branch of `updateTransactionApprover` (lines 489–517) only verifies that the target user exists and that the approver being updated is not a tree node. It never calls `isNode()` or performs any equivalent check to detect whether the target `userId` is already present as another approver for the same transaction:

```typescript
} else if (typeof dto.userId === 'number') {
  if (typeof approver.threshold === 'number') throw new Error(this.APPROVER_IS_TREE);
  const userCount = await transactionalEntityManager.count(User, { where: { id: dto.userId } });
  if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dto.userId));
  if (approver.userId !== dto.userId) {
    // ← no isNode() / duplicate check here
    await transactionalEntityManager.update(TransactionApprover, approver.id, data);
    ...
  }
}
``` [3](#0-2) 

There is also no database-level unique constraint on `(transactionId, userId)` in the `TransactionApprover` entity — only non-unique `@Index` decorators exist: [4](#0-3) 

**Exploit path:**

1. Attacker (authenticated, verified user) creates a transaction and adds two approvers: User A (`approverId=10`) and User B (`approverId=11`), with a threshold of 2.
2. Attacker calls `PATCH /transactions/:txId/approvers/11` with body `{ "userId": <User A's id> }`.
3. No duplicate check fires. The DB now contains two `TransactionApprover` rows for the same `transactionId`, both with `userId = User A`.
4. User A calls `POST /transactions/:txId/approvers/approve`. `approveTransaction` filters `userApprovers = approvers.filter(a => a.userId === user.id)`, collecting both duplicate rows: [5](#0-4) 
5. Both rows are bulk-updated with User A's signature via `.whereInIds(userApprovers.map(a => a.id)).execute()`: [6](#0-5) 
6. Both approver records now carry User A's signature. The threshold of 2 is satisfied by a single unique approver.

The controller endpoint that exposes this is `PATCH /:id` in `approvers.controller.ts`: [7](#0-6) 

## Impact Explanation
A malicious transaction creator can reduce the effective approval threshold to 1 regardless of the configured value. In an organizational workflow where multi-party approval is the security control preventing unauthorized Hedera transactions (e.g., large HBAR transfers, account updates), this completely undermines the approval model. The attacker controls which transaction gets executed and can self-approve it by duplicating their own `userId` across all approver slots.

## Likelihood Explanation
Any authenticated, verified user who creates a transaction has the authority to call `updateTransactionApprover`. No admin keys, leaked credentials, or privileged roles are required. The attack requires only two sequential API calls (create approvers, then update one approver's `userId`) and is fully deterministic.

## Recommendation
Add a duplicate-user check inside the `userId` branch of `updateTransactionApprover`, mirroring the guard already present in `createTransactionApprovers`. Specifically, before performing the update, call `isNode()` (or an equivalent query) to verify that no other active `TransactionApprover` row for the same transaction already holds the target `userId`. Additionally, consider adding a database-level partial unique index on `(transactionId, userId)` where `deletedAt IS NULL` to enforce this invariant at the persistence layer.

## Proof of Concept

```
# Step 1: Create transaction with two approvers (threshold = 2)
POST /transactions/{txId}/approvers
Body: { "approversArray": [{ "userId": 10 }, { "userId": 11 }] }

# Step 2: Update approver 11 to use User A's userId (no duplicate check fires)
PATCH /transactions/{txId}/approvers/11
Body: { "userId": 10 }

# DB state: two TransactionApprover rows, both with userId=10, transactionId=txId

# Step 3: User A approves — both rows are updated, threshold of 2 is satisfied
POST /transactions/{txId}/approvers/approve
Body: { "userKeyId": ..., "signature": ..., "approved": true }
```

The relevant bulk-update in `approveTransaction` at lines 600–609 applies the signature to all rows matching `userId = User A`, causing both duplicate slots to be marked as approved simultaneously. [8](#0-7)
