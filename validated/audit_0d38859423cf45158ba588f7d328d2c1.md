## Audit Report

## Title
`importSignatures` Omits `processTransactionStatus` Call, Delaying Transition to `WAITING_FOR_EXECUTION`

## Summary
`importSignatures` in `transactions.service.ts` persists updated signature bytes to the database but never calls `processTransactionStatus` to evaluate whether the accumulated signatures now satisfy the required key structure and advance the transaction's `status` to `WAITING_FOR_EXECUTION`. The parallel code path `uploadSignatureMaps` in `signers.service.ts` correctly calls `processTransactionStatus` after persisting signatures. Because the chain service's execution path (`collateAndExecute`) is gated on `status === WAITING_FOR_EXECUTION`, a transaction that receives its final required signatures through `importSignatures` is not immediately eligible for execution and must wait for the periodic scheduler to correct the status.

## Finding Description

**Root cause — missing `processTransactionStatus` call in `importSignatures`**

`importSignatures` (lines 493–626 of `transactions.service.ts`) performs the following steps:
1. Validates each signature map against the stored `transactionBytes`.
2. Calls `sdkTransaction.addSignature(publicKey, map)` for each valid key.
3. Persists the updated `transactionBytes` to the database via a bulk `UPDATE`.
4. Emits `emitTransactionStatusUpdate` — a NATS/WebSocket notification to clients.

What it never does is call `processTransactionStatus`, which evaluates whether the accumulated signatures satisfy the required key structure and, if so, writes `status = WAITING_FOR_EXECUTION` to the database. [1](#0-0) 

**Contrast with the correct path — `uploadSignatureMaps`**

`uploadSignatureMaps` in `signers.service.ts` follows the same signature-persistence steps but then explicitly calls `updateStatusesAndNotify` (line 118), which internally calls `processTransactionStatus` (line 429) and writes the new status to the database before emitting any notification. [2](#0-1) [3](#0-2) 

**Why `emitTransactionStatusUpdate` does not compensate**

`emitTransactionStatusUpdate` at line 616 of `transactions.service.ts` is a NATS publish that tells the notification service to push a WebSocket event to connected clients. It does not trigger `processTransactionStatus` or update the `status` column in the database. [7](#0-6) 

**Chain service execution dependency on `WAITING_FOR_EXECUTION`**

`prepareTransactions` in the chain service only calls `collateAndExecute` for transactions whose in-memory `status === WAITING_FOR_EXECUTION`. Since `processTransactionStatus` updates the DB but not the in-memory object, a transaction promoted in one scheduler run is only picked up for execution on the next run. [4](#0-3) 

The most frequent scheduler covering the critical near-term window runs every 10 seconds: [5](#0-4) 

`processTransactionStatus` itself only updates the DB, not the in-memory transaction objects returned to `prepareTransactions`: [8](#0-7) 

## Impact Explanation

A transaction that receives its final required signatures via `importSignatures` remains in `WAITING_FOR_SIGNATURES` in the database. The chain service does not schedule it for execution until the periodic scheduler runs and corrects the status (up to 10 seconds), and then runs again to pick it up for `collateAndExecute` (another 10 seconds). This introduces a mandatory delay of up to ~20 seconds before execution begins. If the final signatures arrive within the last ~20 seconds of Hedera's 120-second validity window, the transaction will be submitted after the network's deadline and rejected as expired — a permanent, unrecoverable loss of the intended on-chain operation. `importSignatures` returns `{ id }` (success) for every updated transaction regardless, so no error is surfaced to the caller.

## Likelihood Explanation

Any authenticated user with access to a transaction can reach `importSignatures`. The timing window is realistic: Hedera transactions have a fixed 120-second validity window from `validStart`. If a user submits the final signatures within the last ~20 seconds of that window via `importSignatures` (rather than `uploadSignatureMaps`), the transaction will expire before the scheduler can promote and execute it. This is a plausible scenario in time-sensitive multi-party signing workflows, particularly when `importSignatures` is used as a bulk import path for externally collected signatures.

## Recommendation

After the bulk `UPDATE` of `transactionBytes` succeeds, call `processTransactionStatus` for all successfully updated transactions, exactly as `updateStatusesAndNotify` does in `signers.service.ts`. This requires injecting `TransactionSignatureService` and a `Repository<Transaction>` into `TransactionsService` (or delegating to a shared helper). The call should occur before `emitTransactionStatusUpdate` so that the notification reflects the correct, up-to-date status.

## Proof of Concept

1. Create a multi-sig transaction requiring two keys; leave it in `WAITING_FOR_SIGNATURES`.
2. Collect the second (final) required signature externally and call `POST /transactions/importSignatures` with the signature map, timed so that `validStart` is approximately 20 seconds away.
3. Observe that the DB `status` column remains `WAITING_FOR_SIGNATURES` immediately after the call returns `{ id }`.
4. The 10-second scheduler runs, calls `processTransactionStatus`, and updates the DB to `WAITING_FOR_EXECUTION` — but `prepareTransactions` receives the pre-update in-memory objects and does not call `collateAndExecute`.
5. The next scheduler run (another 10 seconds) fetches the transaction with `WAITING_FOR_EXECUTION` and calls `collateAndExecute`, which sets a timeout for `validStart - 10 seconds`. If `validStart` has already passed, the timeout fires immediately, but `addExecutionTimeout` fires at `validStart + 5 seconds` — which may now be past Hedera's 120-second validity window.
6. Hedera rejects the transaction as `TRANSACTION_EXPIRED`; the chain service marks it `FAILED`. The user received a success response from `importSignatures` with no indication of failure.
