Audit Report

## Title
Transaction Creator Can Manipulate Approver Tree After Approvals Are Submitted, Bypassing Multi-Approval Enforcement

## Summary
The `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` operations gate access solely on creator identity via `getCreatorsTransaction`, with no check on the transaction's current status. A transaction creator can therefore remove rejecting approvers, lower approval thresholds, or inject pre-approved approver nodes at any point during the active approval window, retroactively satisfying the threshold and forcing the transaction through without the required quorum.

## Finding Description

**Root cause — `getCreatorsTransaction` has no status guard**

`getCreatorsTransaction` is the sole authorization gate for all three mutating approver operations: [1](#0-0) 

It checks only that the transaction exists and that `transaction.creatorKey?.userId === user.id`. There is no check on `transaction.status`.

This function is invoked by:

1. `createTransactionApprovers` — line 239: [2](#0-1) 

2. `updateTransactionApprover` — line 394: [3](#0-2) 

3. The controller's `removeTransactionApprover` — line 108: [4](#0-3) 

The service-level `removeTransactionApprover` also performs no status check: [5](#0-4) 

`updateTransactionApprover` allows threshold reduction with no status guard: [6](#0-5) 

**Contrast with `approveTransaction`**, which correctly enforces a status check before accepting an approval: [7](#0-6) 

The approval submission path is guarded; the approver-tree mutation path is not.

**Additional amplifier — `createTransactionApprovers` inherits existing approval state**

When a new approver node is created for a user who already has an approver record on the same transaction, the new node inherits the existing `signature`, `userKeyId`, and `approved` values: [8](#0-7) 

This means the creator can inject a duplicate approver node for a user who already approved, and the new node will be born with `approved: true`, directly inflating the satisfied-approval count without any additional action from that user.

**Status re-evaluation path**

After `removeTransactionApprover` and `createTransactionApprovers`, `emitTransactionStatusUpdate` is fired: [9](#0-8) [10](#0-9) 

Independently, the chain scheduler runs `processTransactionStatus` every 10 seconds against all `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` transactions: [11](#0-10) 

Either path will re-evaluate the now-mutated approver tree and advance the transaction if the (lowered) threshold is satisfied.

## Impact Explanation

The multi-approval mechanism is the primary trust control for high-value or sensitive Hedera transactions in organization mode. Bypassing it allows a single user (the creator) to execute any transaction unilaterally, regardless of the approval policy configured. The approval quorum can be reduced to 1 at will, rejections can be silently erased, and pre-approved duplicate nodes can be injected — all with no audit trail distinguishing a manipulated approval from a legitimate one. This is a critical integrity failure of the governance model.

## Likelihood Explanation

The attacker is the transaction creator — a normal, unprivileged user role reachable by any registered organization member. No leaked credentials, admin keys, or race conditions are required. The three API endpoints (`POST`, `PATCH`, `DELETE` on `/transactions/:id/approvers`) are standard REST calls. The attack is deterministic and repeatable on every transaction the attacker creates. [12](#0-11) 

## Recommendation

Add a status guard inside `getCreatorsTransaction` (or as a dedicated check at the top of each mutating operation) that rejects modifications when the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`:

```typescript
if (
  transaction.status === TransactionStatus.WAITING_FOR_SIGNATURES ||
  transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
) {
  throw new BadRequestException(
    'Approver tree cannot be modified after the transaction has entered the approval window',
  );
}
```

This mirrors the guard already present in `approveTransaction`. [7](#0-6) 

Additionally, the `createTransactionApprovers` approval-state inheritance (lines 318–329) should be removed or gated behind the same status check, as it enables instant approval inflation. [8](#0-7) 

## Proof of Concept

1. Creator creates transaction T with approver tree: root node threshold=3, leaf approvers [A, B, C, D, E]. Transaction enters `WAITING_FOR_SIGNATURES`.
2. Approvers A and B submit valid approvals (`approved=true`). Approver C submits a rejection (`approved=false`). Current satisfied count: 2 of 3.
3. Creator calls `DELETE /transactions/T/approvers/<C_node_id>`. The controller calls `getCreatorsTransaction` (no status check passes), then `removeTransactionApprover` (no status check). C's rejection is soft-deleted.
4. Creator calls `PATCH /transactions/T/approvers/<root_node_id>` with body `{ "threshold": 2 }`. `updateTransactionApprover` calls `getCreatorsTransaction` (no status check passes), then updates the threshold to 2.
5. Within 10 seconds, the chain scheduler's `processTransactionStatus` re-evaluates T. The tree now has threshold=2 and two `approved=true` leaf nodes — threshold satisfied.
6. Transaction advances to execution with only 2 approvals instead of the originally required 3.

**Variant (approval inflation via `createTransactionApprovers`):**

After step 2, creator calls `POST /transactions/T/approvers` with `{ "approversArray": [{ "userId": <A_id> }] }`. Because A already has an `approved=true` record, the new node is created with `approved=true` inherited. The tree now has three `approved=true` entries, satisfying the original threshold=3 without any additional approver action.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```

```typescript
  async updateTransactions(from: Date, to?: Date) {
    //Get the transaction, creatorKey, groupItem, and group. We need the group info upfront
    //in order to determine if the group needs to be processed together
    const transactions = await this.transactionRepo.find({
      where: {
        status: In([
          TransactionStatus.WAITING_FOR_SIGNATURES,
          TransactionStatus.WAITING_FOR_EXECUTION,
        ]),
        validStart: to ? Between(from, to) : MoreThan(from),
      },
      relations: {
        creatorKey: true,
        groupItem: {
          group: true,
        },
      },
      order: {
        validStart: 'ASC',
      },
    });

    const results = await processTransactionStatus(this.transactionRepo, this.transactionSignatureService, transactions);

    if (results.size > 0) {
      const events = Array.from(results.keys(), id => ({ entityId: id }));
      emitTransactionStatusUpdate(this.notificationsPublisher, events);
    }
```
