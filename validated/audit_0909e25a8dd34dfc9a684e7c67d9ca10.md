### Title
Duplicate Transaction IDs in `uploadSignatureMaps` Array Cause Unbounded Duplicate `TransactionSigner` Row Insertion

### Summary
The `POST /transactions/signers` endpoint accepts an array of `UploadSignatureMapDto` objects and passes them directly to `uploadSignatureMaps`. No deduplication is performed on the array before processing. Because `existingSignerIds` is loaded once before any inserts, and `bulkInsertSigners` uses a plain `INSERT` with no `ON CONFLICT` clause, submitting N copies of the same `{id, signatureMap}` entry causes N duplicate rows to be inserted into `transaction_signer` for the same `(transactionId, userKeyId)` pair. The `TransactionSigner` entity declares only a non-unique `@Index` on `(transactionId, userKeyId)`, so the database enforces no uniqueness constraint.

### Finding Description

**Entry point** — `POST /transactions/signers`, handled by `SignersController.uploadSignatureMap`: [1](#0-0) 

The body is an array `UploadSignatureMapDto[]`. No uniqueness check on `id` values is performed before the array is forwarded to `uploadSignatureMaps`.

**Step 1 — `loadTransactionData`** extracts `transactionIds = dto.map(item => item.id)`. With N duplicates of `id=1`, this produces `[1,1,1,…]`. The DB query uses `In(transactionIds)`, which deduplicates at the SQL level and returns one `Transaction` row. The `signersByTransaction` map is also built once from the pre-existing DB state. [2](#0-1) 

**Step 2 — `validateAndProcessSignatures`** runs `Promise.all(dto.map(...))`, processing every array entry independently. Each duplicate entry calls `processTransactionSignatures` with the same `existingSignerIds` set (loaded before any inserts). Because no new signer has been persisted yet, the check `existingSignerIds.has(userKey.id)` returns `false` for every duplicate, so every call appends the same `UserKey` to its local `userKeys` array. [3](#0-2) 

**Step 3 — `persistSignatureChanges`** iterates over all N results and pushes `{userId, transactionId, userKeyId}` to `signersToInsert` for each one, producing N identical records. [4](#0-3) 

**Step 4 — `bulkInsertSigners`** executes a plain `INSERT … VALUES (…)` with no `ON CONFLICT DO NOTHING` or `orIgnore()`, inserting all N records verbatim. [5](#0-4) 

**No unique constraint on the entity** — `TransactionSigner` uses `@Index` (non-unique) on `(transactionId, userKeyId)`, so the database accepts all N duplicate rows without error. [6](#0-5) 

### Impact Explanation

1. **Database integrity corruption**: The `transaction_signer` table accumulates arbitrarily many duplicate rows for the same `(transactionId, userKeyId)` pair per request. Any downstream query that counts or aggregates signer rows (e.g., for audit logs, reporting, or status checks) will produce incorrect results.

2. **Storage exhaustion / DoS**: A single authenticated API call with an array of 10,000 identical entries inserts 10,000 rows. Repeated calls can exhaust PostgreSQL storage or degrade query performance across the entire backend, affecting all users.

3. **Potential transaction status manipulation**: If `processTransactionStatus` (called in `updateStatusesAndNotify`) counts `transaction_signer` rows to determine signing completion rather than verifying unique cryptographic keys, an attacker could artificially inflate the signer count to advance a transaction's status prematurely. The exact behavior depends on the `TransactionSignatureService` implementation, which was not fully reachable in this analysis.

### Likelihood Explanation

Any authenticated, non-privileged user with a valid JWT can reach `POST /transactions/signers`. The attacker only needs:
- A valid account on the organization server
- Knowledge of any transaction ID they have access to (observer, signer, or creator)
- A valid signature map for that transaction

No admin privileges, leaked secrets, or internal access are required. The attack is a single crafted HTTP request.

### Recommendation

1. **Deduplicate the `dto` array by `id` before processing** in `uploadSignatureMaps`:
   ```typescript
   const deduplicatedDto = Array.from(new Map(dto.map(item => [item.id, item])).values());
   ```

2. **Add `ON CONFLICT DO NOTHING`** (TypeORM: `.orIgnore()`) to `bulkInsertSigners` as a defense-in-depth measure:
   ```typescript
   manager.createQueryBuilder()
     .insert()
     .into(TransactionSigner)
     .values(signersToInsert)
     .orIgnore()   // <-- add this
     .returning('*')
     .execute();
   ```

3. **Add a database-level UNIQUE constraint** on `(transactionId, userKeyId)` in `TransactionSigner` by replacing `@Index` with `@Index({ unique: true })` or adding `@Unique(['transactionId', 'userKeyId'])`.

### Proof of Concept

**Preconditions**: Attacker is an authenticated user with access to transaction ID `42` (e.g., as a signer), and holds a valid signature map for it.

**Request**:
```http
POST /transactions/signers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 42, "signatureMap": { ... } },
  { "id": 42, "signatureMap": { ... } },
  { "id": 42, "signatureMap": { ... } },
  ... (repeated 1000 times)
]
```

**Expected outcome without fix**:
- `loadTransactionData` fetches one `Transaction` row for ID 42.
- `validateAndProcessSignatures` processes all 1000 entries; each passes the `existingSignerIds` check (none inserted yet).
- `persistSignatureChanges` builds `signersToInsert` with 1000 identical `{userId, transactionId: 42, userKeyId}` records.
- `bulkInsertSigners` inserts all 1000 rows into `transaction_signer`.
- Querying `SELECT COUNT(*) FROM transaction_signer WHERE "transactionId" = 42` returns 1000 instead of 1.
- Repeating the request N times produces N×1000 rows, exhausting storage. [7](#0-6)

    dto: UploadSignatureMapDto[],
    user: User,
  ): Promise<{ signers: TransactionSigner[]; notificationReceiverIds: number[] }> {

    const { transactionMap, signersByTransaction } = await this.loadTransactionData(dto);

    return {
      signers: Array.from(signers),
      notificationReceiverIds: notificationsToDismiss,
    };
  }

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
