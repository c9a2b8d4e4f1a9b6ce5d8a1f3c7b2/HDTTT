Audit Report

## Title
Any Transaction Creator Can Delete Approvers from Transactions They Do Not Own (IDOR → Approval Workflow DoS)

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller owns `:transactionId`, but then deletes the approver record identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who owns at least one transaction can permanently remove approvers from any other transaction in the system.

## Finding Description

**Root cause — missing cross-object ownership check in `removeTransactionApprover`**

The controller performs two independent, uncorrelated checks:

```typescript
// approvers.controller.ts lines 108-109
await this.approversService.getCreatorsTransaction(transactionId, user); // ① verifies user owns transactionId
await this.approversService.removeTransactionApprover(id);               // ② deletes approver by id — no transactionId filter
``` [1](#0-0) 

`removeTransactionApprover` fetches the approver purely by its primary key and soft-deletes it with no `transactionId` filter:

```typescript
// approvers.service.ts lines 534-543
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);  // no transactionId filter
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

`getTransactionApproverById` queries only on `id`:

```typescript
// approvers.service.ts lines 72-74
const find: FindOneOptions<TransactionApprover> = {
  relations: ['approvers'],
  where: { id },
};
``` [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly validates cross-ownership before acting:

```typescript
// approvers.service.ts lines 386-394
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [4](#0-3) 

The delete path has no equivalent guard. The `updateTransactionApprover` path walks up to the root node and confirms `rootNode.transactionId === transactionId`; `removeTransactionApprover` skips this entirely.

## Impact Explanation
An attacker with no privileged access can permanently remove required approvers from any transaction in the system. `removeNode` executes a recursive CTE soft-delete that cascades through the entire approver subtree rooted at the targeted node. [5](#0-4) 

If the deleted approver was the sole approver, or if the remaining approvers cannot satisfy the threshold after deletion, the transaction can never reach `WAITING_FOR_EXECUTION` and will expire or be permanently stuck. This is an unrecoverable DoS on targeted transactions' approval workflows.

## Likelihood Explanation
- **Precondition**: attacker must be an authenticated, verified user who has created at least one transaction — a normal product action.
- **Approver ID discovery**: approver IDs are auto-incremented integers. An attacker who is legitimately an observer, signer, or approver of any transaction can read approver IDs from `GET /transactions/:id/approvers`. Sequential enumeration is also trivial since IDs are integers. [6](#0-5) 
- **No race condition required**: this is a direct, synchronous API call with no timing dependency.

Likelihood is **high**.

## Recommendation
Mirror the cross-ownership guard already present in `updateTransactionApprover`. In `removeTransactionApprover` (or in the controller before calling it), walk up to the root node and assert that `rootNode.transactionId === transactionId` before proceeding with deletion:

```typescript
// In removeTransactionApprover or the controller:
const approver = await this.getTransactionApproverById(id);
if (!approver) throw new BadRequestException(ErrorCodes.ANF);

const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId)
  throw new UnauthorizedException('Approver does not belong to the specified transaction');

// then proceed with removeNode
``` [7](#0-6) 

This is the exact pattern used in `updateTransactionApprover` at lines 386–394 and should be applied consistently to the delete path. [4](#0-3) 

## Proof of Concept

1. Bob registers an account and creates **Transaction 1**. Bob is now the creator of Transaction 1.
2. Bob (or any user with observer/approver access) calls `GET /transactions/2/approvers` to retrieve approver IDs for **Transaction 2**, or enumerates integer IDs sequentially. [6](#0-5) 
3. Bob sends: `DELETE /transactions/1/approvers/N` where `N` is an approver ID belonging to Transaction 2.
4. `getCreatorsTransaction(1, Bob)` passes — Bob owns Transaction 1. [8](#0-7) 
5. `removeTransactionApprover(N)` fetches approver `N` by primary key only, finds it (it belongs to Transaction 2), and calls `removeNode(N.id)`, which recursively soft-deletes approver `N` and its entire subtree. [2](#0-1) 
6. Transaction 2's approval threshold can no longer be met; it is permanently blocked from reaching `WAITING_FOR_EXECUTION`.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
