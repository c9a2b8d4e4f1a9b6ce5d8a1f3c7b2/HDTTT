### Title
Unbounded Iteration in `getTransactionsToSign` Enables DoS via Mass Transaction Creation

### Summary
The `GET /transactions/sign` endpoint calls `TransactionsService::getTransactionsToSign`, which fetches **all** non-terminal transactions from the database with no row limit, then performs a per-transaction `computeSignatureKey` call (involving SDK deserialization and mirror-node I/O) for every record. Because any authenticated organization user with at least one registered key can create transactions without a per-user cap, a malicious user can flood the database with `WAITING_FOR_SIGNATURES` transactions, causing every subsequent call to `GET /transactions/sign` to exhaust server memory and time out.

### Finding Description

**Root cause — unbounded `repo.find()` with per-row async work:**

In `getTransactionsToSign`, the database query at lines 295–299 omits the `take` (LIMIT) clause that every other list endpoint uses:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / no `skip`
});
```

The function then iterates the entire result set and awaits an expensive async call per row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` → `transactionSignatureService.computeSignatureKey`, which deserializes raw transaction bytes via the Hedera SDK and issues mirror-node HTTP calls for each account involved.

Pagination is applied only **after** the full scan:
```typescript
items: result.slice(offset, offset + limit),
```
so requesting `page=1&size=10` still forces the server to process every transaction in the table.

**Contrast with correctly-paginated sibling endpoints:**

`getHistoryTransactions` and `getTransactionsToApprove` both pass `skip: offset, take: limit` to the ORM query, pushing pagination to the database layer.

**Attacker-controlled entry point — unrestricted transaction creation:**

`POST /transactions` is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`, and `HasKeyGuard`. Any verified organization user who has registered at least one key can call it. There is no per-user transaction count limit and no minimum cost. The global IP-based throttler (`GLOBAL_MINUTE_LIMIT` / `GLOBAL_SECOND_LIMIT`) slows bulk creation but does not prevent an attacker from accumulating tens of thousands of transactions over hours or days.

Each created transaction starts in `WAITING_FOR_SIGNATURES` status, which is explicitly excluded from the terminal-status filter in `getTransactionsToSign`, so every attacker-created transaction is included in every subsequent full scan.

### Impact Explanation

When the attacker has seeded enough transactions, any call to `GET /transactions/sign` by any user causes:

1. A full-table scan loading all non-terminal transactions (and their `groupItem` relations) into Node.js heap.
2. A sequential chain of awaited mirror-node HTTP calls — one per transaction — blocking the event loop and consuming connection-pool slots.
3. Server-side timeout or OOM crash, making the endpoint permanently unavailable until the attacker's transactions are purged.

Because the endpoint is still reachable (the "NO LONGER USED BY FRONT-END" comment does not remove the route), any API client or integration that calls it is affected. Collateral damage extends to the shared NestJS process: a hung or OOM'd worker degrades all other endpoints served by the same instance.

### Likelihood Explanation

- **Attacker preconditions:** A valid organization account with one registered key — the minimum required to join an organization.
- **No privileged access needed:** Standard user role, no admin rights.
- **Effort:** Scripted `POST /transactions` loop over hours; the global rate limiter (default values are operator-configurable and set to `99999` in CI) does not prevent slow accumulation.
- **Detectability:** Low — transaction creation is a normal workflow action; bulk creation blends into legitimate usage patterns.

### Recommendation

1. **Apply database-level pagination in `getTransactionsToSign`** — add `take: limit` and `skip: offset` to the `repo.find()` call, mirroring `getHistoryTransactions`. Accept that `totalItems` will be approximate or require a separate `COUNT` query.
2. **Enforce a per-user active-transaction cap** in `createTransaction` — reject creation when the requesting user already has more than a configurable threshold of non-terminal transactions.
3. **Consider removing or rate-limiting the `/sign` endpoint** — the controller comment already notes it is "NO LONGER USED BY FRONT-END"; if unused, remove the route entirely to eliminate the attack surface.

### Proof of Concept

**Step 1 — Seed transactions as a malicious authenticated user:**
```
for i in $(seq 1 10000); do
  curl -s -X POST https://target/transactions \
    -H "Authorization: Bearer $ATTACKER_JWT" \
    -H "Content-Type: application/json" \
    -d '{"name":"x","transactionBytes":"<valid_hex>","mirrorNetwork":"testnet",
         "signature":"<valid_sig>","creatorKeyId":<attacker_key_id>}'
done
```
Each request creates one `WAITING_FOR_SIGNATURES` transaction. Spread over time to stay under the per-minute throttle.

**Step 2 — Trigger the DoS as any user (including the attacker):**
```
curl -s "https://target/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer $ANY_JWT"
```

**Expected outcome:** The server loads all N transactions, issues N sequential `computeSignatureKey` calls (each involving SDK deserialization + mirror-node I/O), exhausts the request timeout or Node.js heap, and returns a 500 or connection reset. Repeated calls keep the worker busy, degrading the entire API.
