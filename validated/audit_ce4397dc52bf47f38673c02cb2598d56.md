### Title
`Math.random()` Used for Temporary Password Generation Produces Predictable Credentials

### Summary
`AuthService.generatePassword()` in the backend API uses `Math.random()` — a non-cryptographically-secure PRNG — to produce temporary passwords that are emailed to newly registered users. Because Node.js's `Math.random()` (V8 xorshift128+) is deterministic and its internal state can be recovered from observable outputs, an attacker can predict the temporary password of a newly registered user and authenticate as that user before they log in and change it.

### Finding Description
In `back-end/apps/api/src/auth/auth.service.ts`, the `generatePassword()` method generates a temporary credential using `Math.random()`:

```typescript
private generatePassword() {
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () => String.fromCharCode(97 + Math.floor(Math.random() * 26))).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
``` [1](#0-0) 

This password is then set as the user's actual account credential and emailed to them: [2](#0-1) 

`Math.random()` in Node.js uses V8's xorshift128+ algorithm. Its 128-bit internal state can be fully recovered by observing as few as ~3–5 consecutive floating-point outputs from the same process. Any endpoint or log that leaks `Math.random()` outputs (e.g., other generated values in the same process) allows an attacker to reconstruct the PRNG state and predict all future outputs, including the next temporary password.

The attack path:
1. Attacker observes enough `Math.random()` outputs from the server process (e.g., via any other feature that exposes random values, or by timing analysis).
2. Attacker reconstructs the V8 PRNG state using known techniques (e.g., `v8-randomness-predictor`).
3. Admin registers a new user (or attacker triggers re-registration of an existing `NEW`-status user).
4. Attacker predicts the 10-character temporary password (`[a-z]{5}-[a-z]{5}`, only 26^10 ≈ 1.4×10^14 space, but PRNG prediction collapses this to 1 candidate).
5. Attacker logs in as the victim user before they do, gaining full account access.

### Impact Explanation
Full account takeover of any newly registered user before they change their temporary password. The attacker gains authenticated access to the organization's transaction management system, including the ability to observe, approve, or manipulate multi-signature Hedera transactions depending on the victim's role.

### Likelihood Explanation
Moderate. The attack requires the ability to observe `Math.random()` outputs from the same Node.js process. This is feasible if any other API endpoint or log exposes random-derived values. The temporary password window (time between registration email and first login) may be hours or days, giving the attacker ample time. The password space without PRNG prediction is 26^10 but with state recovery it collapses to a single value.

### Recommendation
Replace `Math.random()` with Node.js's `crypto.randomBytes()` for all security-sensitive credential generation:

```typescript
import { randomBytes } from 'crypto';

private generatePassword(): string {
  const letters = 'abcdefghijklmnopqrstuvwxyz';
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () => letters[randomBytes(1)[0] % 26]).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
```

Alternatively, use `crypto.randomUUID()` or a dedicated library like `nanoid` with a cryptographic source. The frontend already correctly uses `randomUUID` from `crypto` for user IDs. [3](#0-2) 

### Proof of Concept
1. Deploy the backend and register any user to seed the PRNG state.
2. Use a tool such as [`v8-randomness-predictor`](https://github.com/nicowillis/v8-randomness-predictor) to recover the V8 xorshift128+ state from observed `Math.random()` outputs in the same process.
3. Simulate the next 10 calls to `Math.random()` locally to reconstruct the `getRandomLetters(5)-getRandomLetters(5)` output.
4. Trigger admin registration of a target email via `POST /auth/signup` (admin-only endpoint).
5. Immediately attempt login with the predicted password at `POST /auth/login` before the victim user does.
6. Expected result: successful authentication as the victim user.
