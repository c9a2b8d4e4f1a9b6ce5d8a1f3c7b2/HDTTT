Audit Report

## Title
Any Transaction Creator Can Delete Approvers Belonging to Other Transactions via Missing Cross-Ownership Check

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `:transactionId` but deletes the approver identified by `:id` with no check that `:id` belongs to `:transactionId`. An authenticated user who owns any transaction can delete approvers from any other transaction in the system.

## Finding Description

In `approvers.controller.ts`, the handler performs two fully decoupled operations:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts lines 102-113
await this.approversService.getCreatorsTransaction(transactionId, user); // gates on transactionId
await this.approversService.removeTransactionApprover(id);               // deletes by id, no cross-check
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller owns `transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by `id` and deletes it — with no assertion that `approver.transactionId == transactionId` (or that the approver's root node belongs to `transactionId`): [3](#0-2) 

By contrast, `updateTransactionApprover` **does** perform this cross-ownership check correctly — it fetches the root node and asserts `rootNode.transactionId !== transactionId` before proceeding: [4](#0-3) 

The check is present in the update path but was simply omitted from the delete path.

## Impact Explanation
`removeNode` performs a recursive soft-delete of the entire approver subtree rooted at the targeted ID: [5](#0-4) 

An attacker can:
1. Own any one transaction (no admin privilege required).
2. Enumerate approver IDs (sequential auto-increment integers).
3. Call `DELETE /transactions/{own_transactionId}/approvers/{victim_approverId}`.
4. The system confirms ownership of `own_transactionId` (passes), then deletes `victim_approverId` and its entire child tree without further checks.

This allows silently dismantling the approval workflow of any transaction in the organization — removing required approvers, collapsing threshold trees, or deleting entire approval trees — potentially reducing a multi-party approval requirement to zero, enabling unauthorized execution.

## Likelihood Explanation
- **Precondition**: a valid, verified organization account — no elevated privilege required.
- **Approver ID enumeration**: IDs are sequential integers; no brute-force difficulty.
- **Attack surface**: standard REST `DELETE` call, no rate-limiting specific to this action.
- **Reachability**: any normal user in Organization Mode who is the creator of at least one transaction.

## Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), add the same cross-ownership check that `updateTransactionApprover` already performs:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root node.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.

This mirrors the existing pattern at `approvers.service.ts` lines 386–391 and closes the gap.

## Proof of Concept
```
# Attacker owns transaction 1 (transactionId=1).
# Victim's approver tree is rooted at approver id=99, belonging to transaction 2.

DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>

# Flow:
# 1. getCreatorsTransaction(1, attacker) → passes (attacker owns tx 1)
# 2. removeTransactionApprover(99)
#    → getTransactionApproverById(99) → returns approver (no tx check)
#    → removeNode(99) → soft-deletes approver 99 and all its children
# Result: victim transaction 2's entire approval tree is destroyed.
```

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
