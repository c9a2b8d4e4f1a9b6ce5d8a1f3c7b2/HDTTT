Audit Report

## Title
Unbounded In-Memory Iteration Over All Active Transactions in `getTransactionsToSign` Enables Authenticated Resource Exhaustion DoS

## Summary
`getTransactionsToSign` in `TransactionsService` fetches all non-terminal transactions system-wide without any user-scoping filter, then iterates over every one performing per-transaction CPU deserialization and signature key computation. An authenticated user can pre-populate the system with many long-lived transactions, causing every subsequent call to this endpoint to consume unbounded server CPU, degrading service for all users.

## Finding Description

**Missing User Filter at DB Level**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` constructs a `whereForUser` clause that filters only by transaction status, not by the requesting user: [1](#0-0) 

Compare this to `getTransactions`, which correctly scopes to the user via `signers`, `observers`, and `creatorKey` relations: [2](#0-1) 

**Unbounded Loop with Per-Iteration CPU Work**

After loading all N active transactions, the function iterates over every one: [3](#0-2) 

Each call to `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU deserialization
2. `transactionSignatureService.computeSignatureKey(transaction)` — key tree traversal with potential mirror-node lookups (cached but not free) [4](#0-3) 

**Correction to Original Claim — No DB Round-Trip Per Iteration**

The original report claims a `entityManager.find(UserKey, ...)` DB query per iteration. This is inaccurate. Because `userKeysRequiredToSign` passes `user.keys` as the `userKeys` argument, `keysRequiredToSign` takes the in-memory filter branch, not the DB query branch: [5](#0-4) 

The per-iteration cost is CPU deserialization and cached mirror-node lookups, not unbounded DB queries.

**Pagination Applied After Full Scan**

Pagination parameters are accepted but applied only after the full in-memory scan completes: [6](#0-5) 

**Endpoint Status**

The endpoint is explicitly marked as no longer used by the front-end: [7](#0-6) 

It remains accessible via direct API calls.

## Impact Explanation

Every call to `GET /transactions/sign` triggers O(N) CPU deserialization operations where N is the total number of active transactions system-wide. An attacker who pre-creates a large number of long-lived transactions causes sustained high CPU usage on the backend API server and response timeouts for all users attempting to retrieve their signing queue.

## Likelihood Explanation

- **Precondition**: A valid registered account — no admin privileges required.
- **Cost**: Creating each transaction requires a valid Hedera SDK transaction object signed with the user's key. This is computationally cheap and scriptable.
- **Persistence**: Transactions with future `validStart` remain in `WAITING_FOR_SIGNATURES` status until expiry.
- **Reduced surface**: The endpoint is marked `NO LONGER USED BY FRONT-END`, meaning exploitation requires direct API access rather than normal application usage. This meaningfully reduces likelihood in practice.
- **No rate limiting** on transaction creation is visible in the reviewed code paths.

## Recommendation

1. Apply a user-scoping filter at the DB level before fetching transactions, similar to how `getTransactions` scopes by `signers.userId`, `observers.userId`, or `creatorKey.userId`.
2. Apply `skip`/`take` (pagination) at the DB query level rather than after the full in-memory scan.
3. Consider removing or formally deprecating the endpoint if it is confirmed to be unused by the front-end.
4. Add per-user transaction creation rate limiting or a cap on active transactions per user.

## Proof of Concept

1. Register an authenticated account on the backend.
2. Script the creation of thousands of transactions with `validStart` set 7 days in the future, each with valid Hedera SDK transaction bytes signed with the user's key.
3. Call `GET /transactions/sign?page=1&size=10` as any user.
4. Observe that the server must deserialize and compute signature keys for all N active transactions before returning the first page, causing O(N) CPU work regardless of the requested page size.
