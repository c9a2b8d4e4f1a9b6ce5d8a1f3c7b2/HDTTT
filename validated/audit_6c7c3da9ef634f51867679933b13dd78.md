### Title
Unbounded Linear Scan in `getTransactionsToSign` Enables Authenticated DoS via Transaction Flooding

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions system-wide with no user-scoped DB filter and no DB-level limit, then performs a sequential per-transaction DB query (`userKeysToSign`) for every row returned. Any authenticated user can create transactions in O(1); the victim's endpoint degrades O(n) in sequential DB round-trips where n is the total number of non-terminal transactions in the system.

### Finding Description

**Root cause — missing user filter and missing DB-level limit:** [1](#0-0) 

```typescript
// whereForUser filters ONLY by status — no user-scoped predicate
const whereForUser: FindOptionsWhere<Transaction> = {
  ...where,
  status: Not(In([EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED])),
};

// No `take`, no `skip`, no user filter → fetches every non-terminal tx
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});

// O(n) sequential DB queries — one per transaction
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Contrast with `getTransactions`, which correctly scopes to the requesting user: [2](#0-1) 

`getTransactionsToSign` has no equivalent user predicate.

**Per-transaction DB cost** — each `userKeysToSign` call resolves through `keysRequiredToSign`, which issues a `find(UserKey, { where: { publicKey: In(flatPublicKeys) } })` against the database: [3](#0-2) 

**Exploit flow:**
1. Attacker registers as a normal user (no privileged access required).
2. Attacker calls `POST /transactions` N times — each creation is O(1) and there is no apparent rate limit or per-user transaction cap.
3. Victim calls `GET /transactions/to-sign`.
4. Server issues one unbounded `SELECT` returning all N non-terminal transactions.
5. Server then issues N sequential `SELECT` queries via `userKeysToSign`.
6. Response latency grows linearly with N; at sufficient scale the request times out.

### Impact Explanation
The `getTransactionsToSign` endpoint is the primary mechanism by which users discover which transactions require their signature. Making it time out or severely degrade prevents victims from participating in multi-signature workflows — a core function of the platform. Additionally, O(n) sequential DB queries per request can exhaust the database connection pool, degrading service for all users on the shared backend.

### Likelihood Explanation
The attacker requires only a valid user account — the same precondition as any normal user. Transaction creation is a standard, unrestricted protocol flow. The asymmetry is identical to the referenced report: attacker cost is O(1) per transaction created; victim cost is O(n) per `getTransactionsToSign` call. No privileged keys, no leaked credentials, and no physical access are needed.

### Recommendation
1. **Add a user-scoped DB filter** to `getTransactionsToSign` analogous to the one already present in `getTransactions` (filter by `signers.userId`, `observers.userId`, or `creatorKey.userId`).
2. **Move pagination to the DB layer** — add `take: limit` and `skip: offset` to the `repo.find` call instead of slicing the in-memory result after fetching all rows.
3. **Consider rate-limiting** transaction creation per authenticated user to bound the attacker's ability to grow the global transaction set.
