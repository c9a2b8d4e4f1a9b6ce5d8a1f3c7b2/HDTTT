Audit Report

## Title
`COUNCIL_ACCOUNTS` Membership Check Always Returns False — Admin Key Bypass for Council-Initiated `NodeDeleteTransaction` Is Completely Ineffective

## Summary

`COUNCIL_ACCOUNTS` is declared as a JavaScript **array**, but every membership check uses the `in` operator, which tests for **index keys** (`'0'`, `'1'`, `'2'`), not element values. As a result, `isCouncilAccount` is always `false` for every real council account ID, and the admin key is unconditionally required for every `NodeDeleteTransaction`. The unit tests that are supposed to validate this protection are broken in the same way, providing false confidence.

## Finding Description

**Root cause — wrong operator on wrong data structure**

`COUNCIL_ACCOUNTS` is declared as an array in two places:

- Back-end constants: [1](#0-0) 

- Front-end models: [2](#0-1) 

The membership check uses `in` in three separate locations:

1. Back-end model (`NodeDeleteTransactionModel.getNodeId`): [3](#0-2) 

2. Back-end service (`TransactionSignatureService.addNodeKeys`): [4](#0-3) 

3. Front-end model (`NodeDeleteTransactionModel.getNodeKeys`): [5](#0-4) 

In JavaScript/TypeScript, `value in array` checks whether `value` is a **property name** of the array object — i.e., a numeric index string (`'0'`, `'1'`, `'2'`) or `'length'`. It never checks element values. Therefore:

```js
'0.0.2'  in ['0.0.2', '0.0.50', '0.0.55']  // → false  (not an index)
'0.0.50' in ['0.0.2', '0.0.50', '0.0.55']  // → false
'0.0.55' in ['0.0.2', '0.0.50', '0.0.55']  // → false
```

`isCouncilAccount` is **always `false`**, so the guard branch that skips the admin key requirement is **never entered**.

**Consequence in the back-end service**

Because `isCouncilAccount` is always `false`, `signatureKey.push(nodeInfo.admin_key)` is always executed for `NodeDeleteTransaction`, regardless of who the fee payer is: [6](#0-5) 

**The tests give false confidence**

Both test files use `Object.keys(COUNCIL_ACCOUNTS)[0]` to obtain a "council account ID": [7](#0-6) [8](#0-7) 

`Object.keys` on an array returns its **index strings** (`['0', '1', '2']`), so `Object.keys(COUNCIL_ACCOUNTS)[0]` yields `'0'`, not `'0.0.2'`. The expression `'0' in ['0.0.2', '0.0.50', '0.0.55']` is `true` because `'0'` is a valid array index — so the tests pass while testing a completely different (and meaningless) input. The actual council account IDs are never tested.

## Impact Explanation

The intended design is: when a privileged council account (`0.0.2`, `0.0.50`, `0.0.55`) is the fee payer of a `NodeDeleteTransaction`, its signature already satisfies the authorization requirement and the node admin key is not needed. Because the check is always `false`, the tool always demands the admin key signature.

Concrete effects:
1. **Governance operation lock**: Any council-initiated `NodeDeleteTransaction` submitted through the tool is permanently stuck in "waiting for signatures" unless the admin key holder co-signs. If the admin key holder is unavailable or adversarial, the operation cannot complete through the tool.
2. **Unintended veto power**: The admin key holder gains de-facto veto power over council node deletions — a privilege the design explicitly intended to exclude.
3. **Integrity failure in the trust model**: The tool's computed required-signature set diverges from what the Hedera network actually requires, breaking the correctness guarantee of the signature workflow.

## Likelihood Explanation

Every `NodeDeleteTransaction` where the fee payer is a council account triggers this bug — there are no preconditions to satisfy. The broken tests mean the defect has survived code review and CI. Any organization using the tool for council-governed node management will encounter this on the first such transaction.

## Recommendation

Replace the `in` operator with `Array.prototype.includes()` in all three locations:

```ts
// WRONG — tests array index keys, not values
const isCouncilAccount = payerId.toString() in COUNCIL_ACCOUNTS;

// CORRECT — tests element values
const isCouncilAccount = COUNCIL_ACCOUNTS.includes(payerId.toString());
```

Fix all three affected files:
- `back-end/libs/common/src/transaction-signature/model/node-delete-transaction.model.ts` line 18
- `back-end/libs/common/src/transaction-signature/transaction-signature.service.ts` line 200
- `front-end/src/renderer/utils/transactionSignatureModels/node-delete-transaction.model.ts` line 22

Fix the tests to use actual council account values instead of `Object.keys(COUNCIL_ACCOUNTS)[0]`:

```ts
// WRONG — yields '0' (an array index), not '0.0.2'
const councilAccountId = Object.keys(COUNCIL_ACCOUNTS)[0];

// CORRECT — yields '0.0.2'
const councilAccountId = COUNCIL_ACCOUNTS[0];
```

## Proof of Concept

```ts
import { COUNCIL_ACCOUNTS } from '@app/common/constants';
// COUNCIL_ACCOUNTS = ['0.0.2', '0.0.50', '0.0.55']

// Demonstrates the bug:
console.log('0.0.2'  in COUNCIL_ACCOUNTS); // false — '0.0.2' is not an index
console.log('0.0.50' in COUNCIL_ACCOUNTS); // false
console.log('0.0.55' in COUNCIL_ACCOUNTS); // false
console.log('0'      in COUNCIL_ACCOUNTS); // true  — '0' IS an index (what the test uses)
console.log('1'      in COUNCIL_ACCOUNTS); // true
console.log('2'      in COUNCIL_ACCOUNTS); // true

// With the fix:
console.log(COUNCIL_ACCOUNTS.includes('0.0.2'));  // true
console.log(COUNCIL_ACCOUNTS.includes('0.0.50')); // true
console.log(COUNCIL_ACCOUNTS.includes('0.0.55')); // true
```

A council account (e.g., `0.0.2`) initiating a `NodeDeleteTransaction` through the tool will always have `isCouncilAccount === false`, causing `signatureKey.push(nodeInfo.admin_key)` to execute unconditionally, requiring the admin key co-signature that the protocol does not actually mandate for council-initiated deletions.
