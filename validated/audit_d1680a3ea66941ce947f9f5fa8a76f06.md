### Title
Transaction Creator Can Add Themselves as an Approver to Self-Approve Their Own Transactions

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` enforces that only the transaction creator can add approvers, but it does not prevent the creator from adding their own `userId` to the approver list. Once added, the creator can call `approveTransaction` as a valid approver, completely bypassing the multi-party approval workflow that the system is designed to enforce.

### Finding Description

**Vulnerability class:** Authorization bypass — a separation-of-duties restriction is not enforced, analogous to the external report's self-referral bypass.

**Root cause:**

In `createTransactionApprovers`, the only identity check performed is that the caller is the transaction creator (line 239). When iterating over the submitted `approversArray`, the code verifies only that the target `userId` exists in the `User` table (lines 271–276). There is **no check** that `dtoApprover.userId !== user.id` (i.e., that the creator is not adding themselves). [1](#0-0) [2](#0-1) 

Once the creator is registered as an approver, `approveTransaction` filters approvers by `userId === user.id`, finds the creator's own record, and proceeds to accept the approval: [3](#0-2) 

**Exploit path:**

1. Attacker (a verified organization user) creates a transaction via `POST /transactions`.
2. Attacker calls `POST /transactions/{id}/approvers` with `approversArray: [{ userId: <own_user_id> }]`. The `getCreatorsTransaction` check passes because the attacker is the creator. The user-existence check passes because the attacker's own account exists.
3. Attacker calls `POST /transactions/{id}/approvers/approve` with a valid signature. The `userApprovers` filter returns the attacker's own approver record, the approval is recorded, and the transaction status advances.

The controller endpoint is accessible to any authenticated, verified user: [4](#0-3) [5](#0-4) 

### Impact Explanation

The approval system is the organization's primary multi-party authorization control for Hedera transactions. A creator who self-approves can advance a transaction to `WAITING_FOR_EXECUTION` (or satisfy a threshold) without any independent review. This allows a single malicious or compromised user to unilaterally authorize transactions — such as account updates, token transfers, or file operations on the Hedera network — that were supposed to require a second party's sign-off.

### Likelihood Explanation

Any authenticated, verified organization user who creates a transaction can exploit this immediately with two standard API calls. No special privileges, leaked credentials, or external dependencies are required. The attacker is operating entirely within normal product flows.

### Recommendation

In `createTransactionApprovers`, after confirming the caller is the creator, add an explicit check that none of the submitted `userId` values equal the creator's own `user.id`:

```typescript
if (dtoApprover.userId === user.id)
  throw new BadRequestException('The transaction creator cannot be added as an approver');
```

Apply the same guard in `updateTransactionApprover` when `dto.userId` is being set, to prevent the creator from reassigning an existing approver slot to themselves. [6](#0-5) 

### Proof of Concept

**Preconditions:** Two organization accounts exist — `creator` (verified) and `admin`. The organization requires approvals before execution.

1. `creator` authenticates and obtains a JWT.
2. `creator` submits `POST /transactions` → receives `{ id: 42 }`.
3. `creator` submits:
   ```
   POST /transactions/42/approvers
   Authorization: Bearer <creator_jwt>
   { "approversArray": [{ "userId": <creator_user_id> }] }
   ```
   Response: `201 Created` — creator is now the sole approver.
4. `creator` signs the transaction body and submits:
   ```
   POST /transactions/42/approvers/approve
   Authorization: Bearer <creator_jwt>
   { "userKeyId": <creator_key_id>, "signature": "<sig>", "approved": true }
   ```
   Response: `200 true` — transaction is self-approved and advances to execution without any independent review.
