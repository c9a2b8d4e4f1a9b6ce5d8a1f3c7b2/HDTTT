Audit Report

## Title
`removeTransaction()` Lacks Status Guard, Allowing Creators to Overwrite Terminal Transaction States

## Summary
`TransactionsService.removeTransaction()` performs no status check before forcibly overwriting a transaction's `status` to `CANCELED` and soft-deleting the record. Any authenticated, verified user who is the creator of a transaction can call `DELETE /transactions/:id` to corrupt the audit trail of an already-finalized transaction (`EXECUTED`, `FAILED`, `EXPIRED`, `REJECTED`, `ARCHIVED`).

## Finding Description

**Root cause — missing state guard in `removeTransaction()`:**

`cancelTransactionWithOutcome()` correctly enforces a `cancelableStatuses` allowlist before any mutation:

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [1](#0-0) 

It also uses a conditional `WHERE status IN (...)` clause in the UPDATE query for race safety: [2](#0-1) 

`removeTransaction()` performs no equivalent check. After verifying creator ownership via `getTransactionForCreator`, it unconditionally:
1. Overwrites `status` → `CANCELED` (line 633)
2. Soft-deletes the record (line 634)
3. Emits a `TRANSACTION_STATUS_UPDATE` notification with the corrupted state (lines 639–648) [3](#0-2) 

The `terminalStatuses` list is defined in the service but never consulted by `removeTransaction()`: [4](#0-3) 

**Exposed endpoint:**

`DELETE /transactions/:id` is a standard authenticated REST endpoint — no elevated role is required beyond being the transaction creator and a verified user: [5](#0-4) 

The controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) apply to all endpoints on the controller and provide no status-based protection: [6](#0-5) 

## Impact Explanation

A creator can call `DELETE /transactions/:id` on a transaction whose status is `EXECUTED`, `FAILED`, `EXPIRED`, `REJECTED`, or `ARCHIVED`. The result:

- The database record's `status` is permanently overwritten from its true terminal value to `CANCELED`, corrupting the audit trail.
- The record is soft-deleted (`deletedAt` set), hiding it from normal queries for all participants (observers, signers).
- A `TRANSACTION_STATUS_UPDATE` notification is broadcast to all subscribers with the false `CANCELED` status, propagating the corrupted state system-wide.
- Observers and signers — who have a legitimate interest in the true outcome — lose visibility into whether the transaction was actually executed on the Hedera network.

The on-chain Hedera state is unaffected, but the off-chain record of what happened is permanently falsified. This constitutes an unauthorized state mutation and audit-trail integrity violation.

## Likelihood Explanation

- **Attacker profile:** Any authenticated, verified user who created a transaction. No admin or privileged role is required.
- **Precondition:** The attacker must be the creator of the target transaction (enforced by `getTransactionForCreator`).
- **Trigger:** A single `DELETE /transactions/:id` HTTP request after the transaction reaches a terminal state.
- **Detectability:** Low — the soft-delete hides the record from normal queries; the notification system actively propagates the false state, making it appear legitimate.

## Recommendation

Add a status guard to `removeTransaction()` that mirrors the pattern used in `cancelTransactionWithOutcome()`. Before performing any mutation, check whether the transaction's current status is in a set of removable (non-terminal) statuses, and throw a `BadRequestException` if it is not. The existing `terminalStatuses` list defined in the service can be used directly for this check. The guard should be applied to both the `softRemove=true` and `softRemove=false` branches, since hard-deletion of a finalized transaction is equally problematic from an audit perspective.

## Proof of Concept

1. Authenticated user (creator) creates a transaction — it reaches `EXECUTED` status after being processed on-chain.
2. Attacker (the same creator) sends: `DELETE /transactions/{id}` with a valid JWT.
3. `deleteTransaction` in the controller calls `removeTransaction(id, user, true)`.
4. `removeTransaction` calls `getTransactionForCreator` — succeeds because the caller is the creator.
5. No status check is performed.
6. `repo.update(transaction.id, { status: TransactionStatus.CANCELED })` overwrites `EXECUTED` → `CANCELED`.
7. `repo.softRemove(transaction)` sets `deletedAt`, hiding the record.
8. `emitTransactionStatusUpdate` broadcasts `CANCELED` to all subscribers.
9. The audit trail now shows the transaction was canceled, not executed — permanently. [7](#0-6)

  async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (softRemove) {
      await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
      await this.repo.softRemove(transaction);
    } else {
      await this.repo.remove(transaction);
    }

    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
```
