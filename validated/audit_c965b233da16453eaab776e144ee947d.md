### Title
Unbounded Transaction Iteration in `getTransactionsToSign` Enables Authenticated DoS for All Users

### Summary
The `getTransactionsToSign` function fetches **all** non-terminal transactions from the database without any database-level row limit, then iterates over every row with a sequential async database call per transaction. Any authenticated user can create an unbounded number of transactions to inflate this global list, causing the endpoint to time out or severely degrade for every user in the system — directly preventing participation in the multi-signature workflow.

### Finding Description

**Root cause — unbounded fetch + sequential per-row async work:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` builds a `whereForUser` filter that only excludes terminal-status transactions. It does **not** scope the query to the requesting user:

```typescript
// lines 266-277
const whereForUser: FindOptionsWhere<Transaction> = {
  ...where,
  status: Not(
    In([
      TransactionStatus.EXECUTED,
      TransactionStatus.FAILED,
      TransactionStatus.EXPIRED,
      TransactionStatus.CANCELED,
      TransactionStatus.ARCHIVED,
    ]),
  ),
};
```

The `repo.find` call at line 295 has no `take` parameter — it loads every matching row:

```typescript
// lines 295-299
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

The code then iterates over the full result set, issuing one `await userKeysToSign(...)` database call per transaction:

```typescript
// lines 301-309
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

Pagination is applied **in-memory after** the full iteration (line 313: `result.slice(offset, offset + limit)`), so the database-level page size cap in `PaginationParams` (max 100 rows) provides no protection here.

**Attacker-controlled entry path:**

Any authenticated, verified user can create transactions via `POST /transactions`. There is no minimum cost, no per-user transaction count cap, and no rate limit scoped to transaction creation (only a global IP-based throttler exists, which is trivially bypassed with multiple IPs or slow drip creation). The transaction creation endpoint is:

```
POST /transactions
```

An attacker creates N transactions (with any valid Hedera transaction bytes). These immediately appear in the global non-terminal pool. Every subsequent call to `GET /transactions/sign` by **any** user now triggers N sequential async DB calls. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

- The `getTransactionsToSign` endpoint becomes progressively slower as the attacker-created transaction count grows, eventually timing out entirely.
- Because the query is **not scoped to the requesting user**, every user in the organization is affected simultaneously — a single attacker degrades the endpoint for all signers.
- Signers can no longer discover which transactions require their signature, permanently blocking the multi-signature workflow until the attacker's transactions expire or are manually cleaned up.
- The attacker's transactions remain in the non-terminal pool until their `validStart` expires on the Hedera network (up to 3 minutes per transaction), but the attacker can continuously create new ones to maintain the DoS. [4](#0-3) 

### Likelihood Explanation

- **Preconditions**: Only a valid authenticated account is required — no admin, no special role, no leaked secrets.
- **Cost**: Creating a transaction requires only a valid Hedera transaction byte payload (easily generated with the Hedera SDK) and a registered user key. The attacker does not need to pay any on-chain fees to create the backend record.
- **Throttler bypass**: The IP throttler (`ip-throttler.module.ts`) applies a global per-IP rate limit, but the attacker can distribute requests across multiple IPs, use a VPN, or simply drip-create transactions over time. There is no per-user transaction creation limit.
- **Persistence**: Each created transaction stays in the pool until it expires, giving the attacker a sustained window of impact per batch. [5](#0-4) 

### Recommendation

1. **Push the user-scoping filter into the database query.** Join `transaction_signer` or use a subquery to only fetch transactions where the requesting user's keys are actually required, rather than fetching all non-terminal transactions and filtering in application memory.

2. **Apply a hard `take` limit at the database level** in `getTransactionsToSign`, consistent with the pagination cap already enforced in `PaginationParams` (max 100).

3. **Add a per-user transaction creation rate limit** (separate from the global IP throttler) to prevent a single account from flooding the transaction pool.

### Proof of Concept

```
1. Attacker registers and verifies an account on the organization backend.
2. Attacker generates a valid frozen Hedera AccountCreateTransaction byte payload
   (no on-chain submission needed — just the serialized bytes).
3. Attacker loops, calling POST /transactions with that payload N times
   (e.g., N = 5,000, spread across multiple IPs to avoid the IP throttler).
4. Victim (any signer) calls GET /transactions/sign?page=1&size=10.
5. Backend executes:
     SELECT * FROM transaction WHERE status NOT IN (terminal statuses)
   → returns all N attacker transactions plus any legitimate ones.
6. Backend then awaits userKeysToSign(...) for each of the N rows sequentially.
7. The HTTP request times out before completing; the victim receives no response
   and cannot determine which transactions require their signature.
8. Attacker repeats step 3 as transactions expire to maintain the DoS.
``` [6](#0-5)

  /* Get the transactions that a user needs to sign */
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
