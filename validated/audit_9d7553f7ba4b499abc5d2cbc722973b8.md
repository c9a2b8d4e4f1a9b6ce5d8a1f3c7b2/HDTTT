Audit Report

## Title
Unbounded Full-Table Scan with Per-Transaction Async Work in `getTransactionsToSign` Causes Server Resource Exhaustion

## Summary
`GET /transactions/sign` loads every non-terminal transaction from the database without any SQL-level `LIMIT`, then performs per-transaction cryptographic computation and database/cache lookups before applying pagination in memory. Any verified user can trigger this endpoint, causing unbounded memory consumption and proportional database load.

## Finding Description

**Root cause — `getTransactionsToSign` in `transactions.service.ts`:**

The `repo.find()` call at line 295 has no `take` option, causing TypeORM to emit a `SELECT` with no `LIMIT`. Every active (non-terminal) transaction is loaded into Node.js heap memory. [1](#0-0) 

For each of the N loaded transactions, `userKeysToSign` is called in a sequential `for...of` loop: [2](#0-1) 

Pagination is applied only after the full scan completes: [3](#0-2) 

**Correction to the original report's DB query claim:** The report states that `entityManager.find(UserKey, { where: { publicKey: In(flatPublicKeys) } })` is called per transaction. This is **inaccurate**. `attachKeys` is called once before the loop (line 285), populating `user.keys`. Inside `userKeysRequiredToSign`, `attachKeys` is a no-op on subsequent calls since `user.keys` is already populated. `keysRequiredToSign` is then called with `user.keys` as the `userKeys` parameter, which takes the in-memory filter path — no DB query for `UserKey` per transaction. [4](#0-3) [5](#0-4) 

**The actual per-transaction DB/network cost** comes from `computeSignatureKey`, which calls `accountCacheService.getAccountInfoForTransaction()` for the fee payer, signing accounts, receiver accounts, and node accounts of each transaction. Each call performs a `dataSource.manager.findOne(CachedAccount, ...)` and potentially a mirror node HTTP fetch: [6](#0-5) [7](#0-6) 

**Exposed endpoint** — still registered and live, guarded only by standard JWT + email-verification guards: [8](#0-7) [9](#0-8) 

## Impact Explanation

A single HTTP request from any verified user causes the API process to:
1. Load the entire active transaction table into heap (memory exhaustion proportional to active transaction count)
2. Execute `computeSignatureKey` per transaction — each involving deserialization of transaction bytes, multiple `findOne` DB queries against the `CachedAccount` table, and potential mirror node HTTP calls
3. Block the Node.js event loop with sequential `await` calls across all N transactions before returning any response

At scale, a single request can exhaust Node.js heap and starve the database connection pool. Repeated requests compound the effect and can render the API unresponsive for all users.

## Likelihood Explanation

- **Attacker precondition:** valid account + email verification — achievable by any self-registering user.
- **Attack path:** `GET /transactions/sign` — a single authenticated HTTP request.
- **No rate limiting** or per-request resource cap is visible on this endpoint.
- The `/* NO LONGER USED BY FRONT-END */` comment confirms the endpoint is abandoned but remains fully reachable. [10](#0-9) 

## Recommendation

1. **Add SQL-level pagination** to the `repo.find()` call by passing `take: limit` and `skip: offset`, mirroring the pattern used in `getHistoryTransactions`: [11](#0-10) 
2. **Remove or disable the endpoint** entirely if it is confirmed to be unused by the front-end, eliminating the attack surface.
3. Consider adding rate limiting middleware on resource-intensive endpoints.

## Proof of Concept

```
# Authenticated request as any verified user
GET /transactions/sign?page=1&size=10
Authorization: Bearer <valid_jwt>
```

With N active transactions in the database, the server will:
1. Execute `SELECT * FROM transaction WHERE status NOT IN (...)` — no LIMIT
2. For each of the N rows, call `computeSignatureKey` which issues multiple `findOne` queries against `CachedAccount` and potentially mirror node HTTP requests
3. Return only 10 items after all N rows have been processed

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,

  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }

  async getAccountInfoForTransaction(
    transaction: Transaction,
    account: string,
    isReceiver: boolean = false,
  ): Promise<AccountInfoParsed | null> {
    if (!transaction || !transaction.mirrorNetwork) {
      return null;
    }

    this.validateAccount(account);

    const mirrorNetwork = transaction.mirrorNetwork;

    // Get cached data
    const cached = await this.dataSource.manager.findOne(CachedAccount, {
      where: { account, mirrorNetwork },
    });

    if (this.hasCompleteData(cached) && isFresh(cached.updatedAt, this.cacheTtlMs)) {
      // Link to transaction even if using cache
      await this.linkTransactionToAccount(transaction.id, cached.id, isReceiver);
      return this.parseCachedAccount(cached);
    }

    // Cache is stale or doesn't exist - fetch new data
    this.logger.debug(`Fetching account ${account} from mirror node (cache ${cached ? 'stale' : 'missing'})`);

    // Try to claim the account for refresh, create the account row if none exists
    const { data: claimedAccount, claimed } = await this.tryClaimAccountRefresh(account, mirrorNetwork);

    if (!claimed) {
      // Link to transaction
      await this.linkTransactionToAccount(transaction.id, claimedAccount.id, isReceiver);

      if (this.hasCompleteData(claimedAccount)) {
        return this.parseCachedAccount(claimedAccount);
      }

      // No cached data
      // This should never happen
      return null;
    }

    const { data } = await this.performRefreshForClaimedAccount(claimedAccount, transaction.id, isReceiver);
    return data;
  }
