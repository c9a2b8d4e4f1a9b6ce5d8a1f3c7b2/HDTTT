### Title
Missing Method Overrides in `TransactionBaseModel` Subclasses Cause Incomplete Signature Audits

### Summary

`TransactionBaseModel` defines `getSigningAccounts()`, `getReceiverAccounts()`, and `getNewKeys()` with default empty/null return values. Multiple concrete subclasses — `FileAppendTransactionModel`, `FreezeTransactionModel`, `SystemDeleteTransactionModel`, `SystemUndeleteTransactionModel`, and `AccountCreateTransactionModel` — inherit these defaults without overriding them. The central `computeSignatureKey()` method calls these functions to build the `SignatureAudit` used to determine which keys must sign a transaction. Because the overrides are missing, the audit silently omits required signers, mirroring the exact pattern of the Solidity `Certifier`/`SimpleCertifier` bug.

### Finding Description

**Root cause — base class with silent defaults:**

`TransactionBaseModel` (both back-end and front-end copies) defines non-abstract methods that return empty collections: [1](#0-0) 

These are not abstract — TypeScript does not force subclasses to override them. Any subclass that forgets to override silently inherits the empty-return behavior.

**Subclasses that never override (back-end):**

`FileAppendTransactionModel`, `FreezeTransactionModel`, `SystemDeleteTransactionModel`, `SystemUndeleteTransactionModel`, and `AccountCreateTransactionModel` each contain only a `TRANSACTION_TYPE` constant and nothing else: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The same pattern is repeated in the front-end: [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9) 

**Contrast with correctly implemented subclasses:**

`TransferTransactionModel` and `AccountAllowanceApproveTransactionModel` both properly override `getSigningAccounts()` (and `getReceiverAccounts()` for transfers), proving the override pattern is understood and intentional for those types: [11](#0-10) [12](#0-11) 

**The consumer — `computeSignatureKey()` (front-end analog of `MasterCertifier.getData()`):**

The front-end `computeSignatureKey()` calls all three methods and uses their results to build the `SignatureAudit`: [13](#0-12) [14](#0-13) 

For any of the five affected transaction types, `accounts` is always an empty `Set`, so the loop at lines 110–125 never executes, and no signing-account keys are added to `signatureKeys`.

**The factory routes all transaction types through these models:** [15](#0-14) 

Every `FileAppendTransaction`, `FreezeTransaction`, `SystemDeleteTransaction`, `SystemUndeleteTransaction`, and `AccountCreateTransaction` deserialized by the system is wrapped in a model that will silently return empty signing sets.

### Impact Explanation

For a `FileAppendTransaction`, the Hedera network requires the file's WACL (write access control list) key to sign. Because `FileAppendTransactionModel.getSigningAccounts()` returns an empty set, `computeSignatureKey()` produces a `SignatureAudit` that contains only the fee-payer key. In the organization multi-signature workflow, the system uses this audit to determine when enough approvers have signed. With an incomplete audit, the transaction is marked "ready to submit" while missing the WACL key signature. The Hedera network then rejects the submitted transaction. The fee payer loses the transaction fee, and the intended file operation silently fails. The same logic applies to `FreezeTransaction` (requires freeze-admin key), `SystemDeleteTransaction`, and `SystemUndeleteTransaction` (require system-admin keys). For `AccountCreateTransaction`, `getNewKeys()` returning `[]` means the new account's initial key is never surfaced in the audit, so the UI cannot warn that the key must be accessible.
