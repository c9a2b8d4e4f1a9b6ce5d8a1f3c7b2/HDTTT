### Title
Batch Transaction Creation Uses Only First DTO's `mirrorNetwork` for Node Validation of All Transactions

### Summary
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `createTransactions` method creates a single Hedera `Client` from `dtos[0].mirrorNetwork` and then uses that same client to validate the node account IDs of **every** DTO in the batch — including those with a different `mirrorNetwork`. This means a transaction targeting mainnet can be node-validated against testnet's node list (or vice versa), bypassing the `TNVN` guard.

### Finding Description
`createTransactions` is the core batch-creation path (the singular `createTransaction` delegates to it): [1](#0-0) 

```
const client = await getClientFromNetwork(dtos[0].mirrorNetwork);   // ← only first DTO's network
...
const validatedData = await Promise.all(
  dtos.map(dto => this.validateAndPrepareTransaction(dto, user, client)),  // same client for ALL
);
```

Inside `validateAndPrepareTransaction`, the node check is: [2](#0-1) 

```
const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
  throw new BadRequestException(ErrorCodes.TNVN);
}
```

`allowedNodes` is derived from the single shared `client`, which was built from `dtos[0].mirrorNetwork`. Each DTO's own `mirrorNetwork` is saved to the database independently: [3](#0-2) 

```
mirrorNetwork: data.mirrorNetwork,   // ← each DTO's own value, not validated against client
```

`getClientFromNetwork` resolves the node set from the network's address book: [4](#0-3) 

Mainnet and testnet have entirely different node account ID sets. A transaction submitted with `mirrorNetwork='mainnet'` but node IDs that are only valid on testnet will pass the TNVN check if `dtos[0]` is a testnet transaction, because the shared client holds testnet's node list.

### Impact Explanation
An authenticated user can submit a batch where `dtos[0]` is a legitimate testnet transaction and subsequent DTOs are mainnet transactions containing node account IDs that do not exist on mainnet. Those mainnet transactions bypass the `TNVN` validation, get stored with `mirrorNetwork='mainnet'`, and will fail at execution time — but they are accepted and stored as `WAITING_FOR_SIGNATURES`, consuming system resources, triggering notifications, and potentially confusing multi-sig workflows. The error code `TNVN` exists precisely to prevent this class of invalid transaction from entering the system. [5](#0-4) 

### Likelihood Explanation
The batch endpoint is a documented, authenticated API path. Any registered organization user can craft a multi-DTO POST request. No special privileges are required beyond a valid session. The attack requires knowledge of the batch API shape, which is standard REST.

### Recommendation
Create a separate `Client` per unique `mirrorNetwork` value in the batch, or enforce that all DTOs in a single batch share the same `mirrorNetwork`. A simple guard:

```typescript
const networks = new Set(dtos.map(d => d.mirrorNetwork));
if (networks.size > 1) throw new BadRequestException('All transactions in a batch must share the same mirrorNetwork');
const client = await getClientFromNetwork(dtos[0].mirrorNetwork);
```

Alternatively, call `validateAndPrepareTransaction` with a per-DTO client.

### Proof of Concept
1. Register as an organization user and obtain a JWT.
2. Build two SDK transactions:
   - `tx_A`: a valid `AccountCreateTransaction` frozen for **testnet** with a testnet node ID (e.g. `0.0.3`).
   - `tx_B`: a valid `AccountCreateTransaction` frozen for **mainnet** but with node ID `0.0.3` (which does not exist on mainnet's address book).
3. POST to `/transactions` with:
   ```json
   [
     { "mirrorNetwork": "testnet", "transactionBytes": "<tx_A bytes>", ... },
     { "mirrorNetwork": "mainnet", "transactionBytes": "<tx_B bytes>", ... }
   ]
   ```
4. Observe that both transactions are accepted with HTTP 201 and stored with status `WAITING_FOR_SIGNATURES`, even though `tx_B` contains a node ID that is invalid on mainnet — the TNVN check was performed against testnet's node list. [6](#0-5)

      // Batch schedule reminders
      const reminderPromises = savedTransactions

          if (!dto.reminderMillisecondsBefore) return null;

          const remindAt = new Date(tx.validStart.getTime() - dto.reminderMillisecondsBefore);
          return this.schedulerService.addReminder(
            getTransactionSignReminderKey(tx.id),
            remindAt,

        .filter(Boolean);

      await Promise.all(reminderPromises);

export const getClientFromNetwork = async (
  mirrorNetwork: string | string[],
  ledgerId?: string | LedgerId,
) => {
  if (!Array.isArray(mirrorNetwork)) {
    mirrorNetwork = [mirrorNetwork];
  }

  mirrorNetwork = mirrorNetwork.map(network => network.toLocaleLowerCase());
  if ([MAINNET, TESTNET, PREVIEWNET].includes(mirrorNetwork[0])) {
    return Client.forName(mirrorNetwork[0]);
  }

  if (mirrorNetwork[0] === LOCAL_NODE) {
    return Client.forNetwork(getLocalClientNetwork(process.env.NODE_ENV)).setMirrorNetwork(
      MirrorNetworkGRPC.LOCAL_NODE,
    );
  }

  const client = Client.forNetwork({}).setMirrorNetwork(
    MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0]),
  );

  const nodeAddressBook = await new AddressBookQuery()
    .setFileId(FileId.ADDRESS_BOOK)
    .execute(client);

  client.setNetworkFromAddressBook(nodeAddressBook);

  if (ledgerId) {
    client.setLedgerId(ledgerId);
  }

  return client;
};
