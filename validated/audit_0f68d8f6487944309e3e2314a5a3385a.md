Audit Report

## Title
TOCTOU Race Condition in `archiveTransaction` Allows Overwriting Terminal Transaction Status

## Summary
`archiveTransaction` in `TransactionsService` performs a status check on a stale in-memory snapshot and then issues an unconditional `UPDATE` with no status guard in the `WHERE` clause. The Chain Service scheduler can concurrently execute the same transaction and set its status to `EXECUTED` or `FAILED`. The archive write wins the race and overwrites the terminal status, permanently corrupting the on-chain audit record.

## Finding Description

**Step 1 — Read (CHECK):**

`archiveTransaction` fetches the transaction and checks its status in memory: [1](#0-0) 

The guard passes if `status` is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`, or if `isManual` is `true`. This snapshot is immediately stale after the `await` returns.

**Step 2 — Write (ACT) — no status guard:** [2](#0-1) 

The generated SQL is effectively `UPDATE transaction SET status = 'ARCHIVED' WHERE id = $1`. There is no `AND status IN (...)` predicate. Any status present in the database at write time is unconditionally overwritten.

**The concurrent actor — Chain Service scheduler:**

`handleTransactionsBetweenNowAndAfterThreeMinutes` fires every 10 seconds: [3](#0-2) 

It calls `prepareTransactions` → `collateAndExecute` → `addExecutionTimeout` → `executeService.executeTransaction`. The `_executeTransaction` path in `ExecuteService` writes `EXECUTED` or `FAILED` to the row — but it does so with a proper status guard: [4](#0-3) 

The Chain Service write is safe. The API's archive write is not.

**Race window:**
```
T0  API:   SELECT status = WAITING_FOR_EXECUTION  ← passes guard
T1  Chain: UPDATE status = EXECUTED               ← guarded, succeeds
T2  API:   UPDATE status = ARCHIVED               ← no guard, overwrites EXECUTED
```

**Contrast with the correctly guarded `cancelTransactionWithOutcome`:** [5](#0-4) 

The `.andWhere('status IN (:...statuses)', ...)` predicate makes the write a no-op if the status changed between read and write. `archiveTransaction` has no equivalent guard.

**Same structural defect in `removeTransaction`:** [6](#0-5) 

`repo.update(transaction.id, { status: TransactionStatus.CANCELED })` also carries no status guard and no prior status check, making it even more permissive.

## Impact Explanation

A transaction that was successfully submitted to and executed on the Hedera network will have its database `status` permanently set to `ARCHIVED` instead of `EXECUTED`. Downstream effects:

- The organization's audit log shows the transaction as archived (never executed), breaking compliance records.
- Any UI or API consumer checking `status === EXECUTED` to confirm on-chain settlement receives a false negative.
- The `executedAt` timestamp and `statusCode` columns written by the Chain Service at T1 remain in the row alongside the wrong `status`, creating an internally inconsistent record that is difficult to detect automatically.

## Likelihood Explanation

The `handleTransactionsBetweenNowAndAfterThreeMinutes` cron fires every 10 seconds and `addExecutionTimeout` schedules execution 5 seconds after `validStart`: [7](#0-6) 

A user who clicks "Archive" on a transaction near its `validStart` — a natural action when they decide not to execute it — races directly against the scheduler. No special attacker capability is required; the creator of any transaction can trigger this with a normal UI action. In a multi-pod deployment (multiple Chain Service replicas), the execution window is wider and the probability increases.

## Recommendation

Replace the unconditional `repo.update` in `archiveTransaction` with a guarded query builder call, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  // Re-read and surface the correct error (already archived, or moved to terminal status)
  const latest = await this.getTransactionForCreator(id, user);
  throw new BadRequestException(/* appropriate error code */);
}
```

Apply the same fix to `removeTransaction` (line 633 in `transactions.service.ts`), adding a status guard before the `CANCELED` update.

## Proof of Concept

1. Create a non-manual transaction; wait until its `validStart` is approximately 5–15 seconds in the future (within the scheduler's 3-minute window).
2. Confirm the transaction is in `WAITING_FOR_EXECUTION` status.
3. Immediately call `PATCH /transactions/archive/:id` as the creator.
4. The API thread reads `status = WAITING_FOR_EXECUTION`, passes the guard, and awaits the `repo.update` call.
5. Concurrently (within the same 10-second cron tick), the Chain Service scheduler picks up the transaction, calls `collateAndExecute` → `addExecutionTimeout` → `executeTransaction`, and writes `status = EXECUTED` with `executedAt` and `statusCode` to the row (guarded write succeeds because status was still `WAITING_FOR_EXECUTION`).
6. The API's unguarded `repo.update` then writes `status = ARCHIVED`, overwriting `EXECUTED`.
7. Query the database: `status = ARCHIVED`, but `executedAt` and `statusCode` are populated — an internally inconsistent record. The Hedera network shows the transaction as executed; the application shows it as archived.
