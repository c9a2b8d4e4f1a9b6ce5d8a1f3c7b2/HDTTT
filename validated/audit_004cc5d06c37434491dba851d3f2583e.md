### Title
TOCTOU Race Condition in `approveTransaction` Allows Approver to Overwrite Their Own Approval Decision

### Summary
In `ApproversService.approveTransaction()`, the duplicate-approval guard reads approver state from the database and checks it before the write, but the subsequent `UPDATE` carries no conditional clause (`WHERE signature IS NULL`). Two concurrent HTTP requests from the same approver both pass the guard on the stale snapshot and both execute the `UPDATE`, with the last write winning. This lets a malicious approver flip their recorded vote (e.g., from `approved=true` to `approved=false`) after it has already been committed, corrupting the approval state of a transaction.

### Finding Description

**Root cause — non-atomic check-then-act**

`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` follows this sequence:

1. **Read** all approvers for the transaction (line 553).
2. **Check** whether the caller has already approved — `if (userApprovers.every(a => a.signature))` (line 563).
3. Perform several async awaits (attach keys, re-fetch transaction, verify signature).
4. **Write** the approval unconditionally — `UPDATE TransactionApprover SET … WHERE id IN (…)` (lines 599–610). [1](#0-0) 

The guard at step 2 is evaluated against a snapshot that is already stale by the time step 4 executes. The `UPDATE` at step 4 contains no `AND signature IS NULL` predicate to make the write conditional on the approval not yet being set. [2](#0-1) 

**Exploit flow**

```
Attacker (legitimate approver) sends two concurrent POST requests:
  Request A → { approved: true,  userKeyId: X, signature: sigA }
  Request B → { approved: false, userKeyId: Y, signature: sigB }

Timeline:
  A reads approvers  → signature = null  → passes guard
  B reads approvers  → signature = null  → passes guard
  A writes DB        → approved=true,  signature=sigA
  B writes DB        → approved=false, signature=sigB   ← overwrites A
```

The final DB state reflects `approved=false` even though the approver had already committed `approved=true`. The notification logic at lines 614–618 also operates on the stale `userApprovers` snapshot, so the downstream status event is also incorrect. [3](#0-2) 

### Impact Explanation

The approval workflow is a trust-enforcement mechanism: a transaction cannot proceed to execution unless the required threshold of approvers have set `approved=true`. By racing two concurrent requests, a malicious approver can:

- **Flip their vote** from `approved=true` to `approved=false` after the fact, blocking a transaction that had already satisfied the approval threshold.
- **Corrupt the recorded key/signature** (`userKeyId`, `signature`) associated with their approval, breaking audit integrity.
- **Emit a misleading status event** to the notification bus because the notification branch reads the pre-race stale `userApprovers` array.

This is an unauthorized state change that violates the core invariant: *once an approver has submitted their decision, it must be immutable*.

### Likelihood Explanation

The attacker is a legitimate approver — a role reachable by any registered user who was added to a transaction's approver list. No privileged credentials are required. Triggering the race requires only two near-simultaneous HTTP requests, achievable with any scripting tool (`curl`, `axios`, `fetch`). The window is wide because `approveTransaction` performs multiple sequential `await` calls between the guard read and the final write (attach keys, find transaction, verify signature), giving ample time for a second request to pass the guard.

### Recommendation

Make the `UPDATE` conditional on the approval not yet being set by adding `AND signature IS NULL` (or `AND approved IS NULL`) to the `WHERE` clause:

```typescript
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
  .whereInIds(userApprovers.map(a => a.id))
  .andWhere('signature IS NULL')   // ← atomic guard
  .execute();
```

If `affected === 0`, the approval was already recorded by a concurrent request and the handler should return an appropriate error (`ErrorCodes.TAP`). This mirrors the pattern already used correctly in `cancelTransactionWithOutcome` and `cancelTransactionGroup`, where the `UPDATE` includes `AND status IN (…)` to make the write atomic. [4](#0-3) 

### Proof of Concept

```typescript
// Attacker is a registered approver for transaction ID 42.
// Send two concurrent requests with conflicting decisions:

const base = 'https://api.example.com/transactions/42/approvers/approve';
const headers = { Authorization: `Bearer ${attackerJwt}`, 'Content-Type': 'application/json' };

await Promise.all([
  fetch(base, { method: 'POST', headers,
    body: JSON.stringify({ approved: true,  userKeyId: 1, signature: sigA }) }),
  fetch(base, { method: 'POST', headers,
    body: JSON.stringify({ approved: false, userKeyId: 2, signature: sigB }) }),
]);

// Expected: second request rejected with TAP error (already approved).
// Actual:   both succeed; DB row reflects whichever write landed last.
// Verify by querying the approver record — approved may be false despite
// the attacker having submitted approved=true first.
```

**Expected outcome**: The transaction's approval state is corrupted. If the threshold was met by the `approved=true` write, the subsequent `approved=false` write silently reverts it, stalling or blocking the transaction's progression to execution.
