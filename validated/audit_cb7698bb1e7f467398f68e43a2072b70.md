Audit Report

## Title
Decrypted Private Key Persists in Renderer DOM Without Timeout and Is Accessible Without Re-Authentication During Cached Password Window

## Summary
Once a user authenticates with their application password for any operation, the password is cached in the Pinia store for a 10-minute sliding window. During this window, clicking the "show private key" eye icon in `Settings > Keys` (`KeysTab.vue`) decrypts and renders the raw private key in the DOM **without prompting for the password again**. The decrypted key is stored in a Vue reactive ref (`decryptedKeys`) with **no automatic expiry** — it persists in the DOM until the user manually hides it. A process running as the same OS user (e.g., malware) can exploit this via OS accessibility APIs to programmatically trigger decryption and read the key from the DOM without any user interaction.

## Finding Description

**Root Cause 1 — 10-minute sliding password cache (`storeUser.ts`):**

`passwordStoreDuration` is initialized to 10 minutes: [1](#0-0) 

`getPassword()` returns the cached password and resets the expiry on every call: [2](#0-1) 

**Root Cause 2 — `isLoggedInWithValidPassword()` only checks timestamp (`userStoreHelpers.ts`):**

The expiry check is a simple timestamp comparison with no additional factor: [3](#0-2) 

**Root Cause 3 — `usePersonalPassword.getPassword()` silently returns cached password:**

If `personalPassword` is truthy (i.e., cached and not expired), the modal is never opened and the password is returned directly to the caller: [4](#0-3) 

**Root Cause 4 — `KeysTab.vue` `handleShowPrivateKey` → `decrypt()` uses cached password with no additional auth gate:**

`handleShowPrivateKey` calls `decrypt()` directly: [5](#0-4) 

`decrypt()` calls `getPassword()` and, if the password is cached, proceeds immediately to `decryptPrivateKey` and pushes the result into `decryptedKeys` with no timeout: [6](#0-5) 

**Root Cause 5 — Decrypted key rendered in DOM indefinitely:**

The template renders the raw private key string from `decryptedKeys` in a `<span>` with a stable `data-testid`: [7](#0-6) 

The only removal path is the manual `handleHideDecryptedKey` click handler — there is no `setTimeout` or reactive watcher that clears `decryptedKeys` automatically. The eye icon that triggers decryption also carries a stable `data-testid`: [8](#0-7) 

## Impact Explanation
An attacker who obtains the raw ED25519 or ECDSA private key can sign arbitrary Hedera transactions — HBAR transfers, token operations, account updates — and drain all associated assets. The compromise is irreversible: private keys cannot be rotated without the account owner's cooperation, and any funds transferred are unrecoverable. Impact is **critical**.

## Likelihood Explanation
The attack requires: (a) malware running on the victim's machine as the same OS user (achievable remotely via phishing or drive-by download, no physical access required), and (b) the user to have performed any password-requiring operation within the past 10 minutes (a normal usage pattern — signing a single transaction satisfies this). On Windows, the UI Automation API, and on Linux, AT-SPI, allow same-user processes to enumerate and interact with Electron window elements without elevated privileges. The stable `data-testid="span-show-modal-{index}"` and `data-testid="span-private-key-{index}"` attributes make programmatic targeting trivial. Likelihood is **medium-high** for users actively using the application.

## Recommendation

1. **Require re-authentication for private key reveal.** `handleShowPrivateKey` should always open the password modal regardless of cache state. A dedicated, non-cached code path should be used for this sensitive operation.

2. **Auto-clear decrypted keys from the DOM.** After pushing to `decryptedKeys`, start a short `setTimeout` (e.g., 30–60 seconds) that calls `handleHideDecryptedKey` automatically. This limits the window during which the key is readable from the DOM.

3. **Reduce or eliminate the password cache for high-sensitivity operations.** Consider not caching the password at all for operations that expose raw key material, or use a separate, shorter-lived cache specifically for key reveal.

4. **Remove or obfuscate `data-testid` attributes in production builds** for elements that expose sensitive key material, to raise the bar for automated targeting.

## Proof of Concept

1. User opens the application and signs a transaction → `setPassword()` is called in `storeUser.ts`, caching the password with a 10-minute expiry.
2. Malware (same OS user, no elevated privileges) uses Windows UI Automation or Linux AT-SPI to locate the Electron window and find the element with `data-testid="span-show-modal-0"`.
3. Malware programmatically fires a click event on that element.
4. `handleShowPrivateKey` fires → `decrypt()` → `getPassword()` returns the cached password (no modal shown) → IPC call to `decryptPrivateKey` in the main process → raw private key returned to renderer → pushed into `decryptedKeys.value`.
5. Vue renders the private key into `span[data-testid="span-private-key-0"]`.
6. Malware reads the text content of that span via the accessibility tree, or captures a screenshot.
7. Raw private key is exfiltrated → complete Hedera account compromise.

export const isLoggedInWithValidPassword = (
  user: PersonalUser | null,
): user is LoggedInUserWithPassword => {
  const hasPassword = isLoggedInWithPassword(user);
  if (!hasPassword) return false;

  const isExpired = user.passwordExpiresAt && new Date(user.passwordExpiresAt) < new Date();

  return !isExpired;
};
