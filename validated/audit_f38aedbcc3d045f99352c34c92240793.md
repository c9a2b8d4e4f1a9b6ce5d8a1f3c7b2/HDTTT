Audit Report

## Title
Any Authenticated User Can Squat Another User's Public Key, Permanently Blocking Them From the Organization Signing Workflow

## Summary
The `POST /user/:userId?/keys` endpoint registers a public key under the JWT-authenticated caller's account with no cryptographic proof that the caller controls the corresponding private key. Because uniqueness is enforced at the application layer by `publicKey` lookup, any verified organization user who registers a victim's public key first permanently prevents the victim from registering that key under their own account.

## Finding Description

**Root cause — no proof-of-possession in `UploadUserKeyDto`**

`UploadUserKeyDto` accepts only `publicKey`, `mnemonicHash`, and `index`. There is no challenge-response, signature-over-nonce, or any other mechanism to prove the submitter controls the private key. [1](#0-0) 

**Root cause — controller ignores `:userId` path parameter for POST**

The controller is mounted at `user/:userId?/keys`, but `uploadKey` binds only `@GetUser()` (the JWT identity). The path parameter `userId` is silently ignored for writes. [2](#0-1) 

**Root cause — `uploadKey()` creates the record under the caller, then blocks all other users**

`UserKeysService.uploadKey()` does a `findOne` by `publicKey`. If no row exists, it creates one owned by `user` (the JWT caller). If a row already exists and `userKey.userId !== user.id`, it throws `BadRequestException(ErrorCodes.PU)` — permanently blocking any other user from registering that key. [3](#0-2) 

**Exploit flow**

1. Victim generates an ED25519 key pair. The public key is visible on the Hedera network (mirror node, transaction bytes, multi-sig coordination).
2. Attacker (any verified org user) sends:
   ```
   POST /user/<any_id>/keys
   Authorization: Bearer <attacker_jwt>
   { "publicKey": "<victim_pubkey>" }
   ```
3. Server creates a `UserKey` row linking `victim_pubkey` → attacker's `userId`.
4. Victim later calls the same endpoint. The service finds the existing row, evaluates `userKey.userId !== user.id` → `true`, and throws `BadRequestException(ErrorCodes.PU)`.
5. Victim is permanently blocked from registering their own key.

**E2E test confirms the squatting behavior**

The test suite explicitly verifies that a second user cannot register a key already registered by another user (HTTP 400), and separately confirms that the `:userId` path parameter is completely ignored for POST operations. [4](#0-3) [5](#0-4) 

## Impact Explanation

- **Permanent DoS on key registration**: Once squatted, the victim can never register `pubKey_V` under their own account. Recovery requires manual admin intervention (direct DB deletion of the attacker's `UserKey` row).
- **Exclusion from multi-sig workflows**: The `UserKey` entity is the authoritative link between a Hedera public key and an organization user. With the key registered under the attacker's account, the victim's signing authority is attributed to the wrong user, breaking the integrity of the multi-signature transaction workflow. [6](#0-5) 

## Likelihood Explanation

- **Attacker precondition**: Any verified organization account. No admin or privileged role required — only `VerifiedUserGuard` is enforced. [7](#0-6) 
- **Information required**: The victim's public key — which is inherently public and visible on the Hedera mirror node or in any transaction the victim has signed.
- **Attack complexity**: A single authenticated HTTP POST. No race condition or timing dependency; the attacker simply needs to register the key before the victim does.
- **Detection difficulty**: The attack is indistinguishable from a normal key upload at the API level.

## Recommendation

Require cryptographic proof-of-possession before accepting a public key registration:

1. **Challenge-response**: Issue a server-generated nonce to the client; require the client to submit a signature over that nonce using the private key corresponding to the submitted `publicKey`. Verify the signature server-side before creating the `UserKey` record.
2. **Alternatively**, add a `signature` field to `UploadUserKeyDto` containing a signature over a deterministic message (e.g., `userId || publicKey || timestamp`) and validate it in `uploadKey()` using the Hedera SDK's `PublicKey.verify()` before persisting the record.

This ensures only the holder of the private key can register the corresponding public key, eliminating the squatting attack entirely.

## Proof of Concept

```http
# Step 1 — Attacker registers victim's public key
POST /user/999/keys HTTP/1.1
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "publicKey": "<victim_ed25519_pubkey_hex>" }

# Response: 201 Created — key is now owned by attacker

# Step 2 — Victim attempts to register their own key
POST /user/2/keys HTTP/1.1
Authorization: Bearer <victim_jwt>
Content-Type: application/json

{ "publicKey": "<victim_ed25519_pubkey_hex>" }

# Response: 400 Bad Request — ErrorCodes.PU ("Public key in use")
# Victim is permanently locked out of registering their own key
```

The `:userId` path parameter in both requests is irrelevant — the server always uses the JWT identity, as confirmed by the e2e test at line 78–92 of `user-keys.e2e-spec.ts`. [8](#0-7)
