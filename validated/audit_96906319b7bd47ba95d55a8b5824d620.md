# Audit Report

## Title
Cross-Network Signature Replay via Missing Ledger-ID Binding in Signature Verification

## Summary
The signature verification logic in `validateSignature` and `processTransactionSignatures` performs only a cryptographic check (`publicKey.verify(bodyBytes, signature)`) with no binding to a specific Hedera network (ledger ID). Because the Hedera `TransactionBody` protobuf contains no chain/ledger ID, and because testnet and mainnet share overlapping node account IDs (`0.0.3`â€“`0.0.9`), a signature produced for a testnet transaction is cryptographically identical to a signature for a mainnet transaction with the same bytes. A malicious organization member can replay testnet signatures on mainnet transactions, satisfying signing thresholds without the signers' explicit consent for mainnet.

## Finding Description

**Root cause â€” `validateSignature` is network-blind**

`validateSignature` in `back-end/libs/common/src/utils/sdk/transaction.ts` verifies each signature purely by:

```ts
const signatureValid = publicKey.verify(bodyBytes, signature);
``` [1](#0-0) 

`bodyBytes` is the raw `TransactionBody` protobuf. The Hedera `TransactionBody` proto contains payer account, node account IDs, valid start, and transaction content â€” but **no ledger ID or chain ID**. A signature over these bytes is therefore valid on any Hedera network that presents the same bytes.

**`importSignatures` accepts any cryptographically valid signature with no network check**

`importSignatures` in `back-end/apps/api/src/transactions/transactions.service.ts` calls `validateSignature` and then `sdkTransaction.addSignature(publicKey, map)` for every key that passes. The only guards are: user access, transaction status, and expiry. The stored `mirrorNetwork` field is collected into the `updates` map but is **never compared** against the network for which the signature was actually produced. [2](#0-1) 

**`processTransactionSignatures` (signer upload path) has the same gap**

`processTransactionSignatures` in `back-end/apps/api/src/transactions/signers/signers.service.ts` calls `sdkTransaction.addSignature(publicKey, map)` with no network validation. The word `mirrorNetwork` does not appear anywhere in this file. [3](#0-2) 

**`mirrorNetwork` is metadata only â€” never cryptographically bound**

When a transaction is created, `mirrorNetwork` is taken verbatim from the client DTO and stored as a plain metadata column. It is never embedded in the signed bytes and is never checked during signature ingestion. [4](#0-3) 

**Node validation at creation time does not prevent the attack**

At transaction creation, `isTransactionValidForNodes` checks that the transaction's node account IDs belong to the specified network's allowed nodes. However, Hedera testnet and mainnet **share overlapping node account IDs** (`0.0.3`â€“`0.0.9`). A transaction targeting node `0.0.3` passes node validation for **both** networks, making identical transaction bytes possible across networks. [5](#0-4) 

## Impact Explanation

A malicious organization member (Bob) who can:
1. Read the testnet transaction bytes (stored in the DB, retrievable via the API for any transaction Bob has access to), and
2. Create a mainnet transaction with identical bytes (same payer account, same valid-start, same node IDs â€” all of which overlap between networks for accounts `0.0.2`â€“`0.0.9`),

can POST Alice's testnet signature map to the mainnet transaction's `/transactions/{id}/signers` endpoint or call `importSignatures`. The backend accepts it because `validateSignature` only checks `publicKey.verify(bodyBytes, sig)` â€” which passes identically on both networks.

If the replayed signature satisfies the required signing threshold, the chain service will execute the mainnet transaction (`TransferTransaction`, `FileUpdateTransaction` on a critical system file, `FreezeTransaction`, `NodeUpdateTransaction`, etc.) without Alice ever consenting to sign for mainnet. [6](#0-5) 

## Likelihood Explanation

- **Attacker profile**: malicious registered organization member â€” no admin/privileged keys required.
- **Preconditions**: (a) the organization tests transactions on testnet before mainnet (standard practice); (b) council/shared accounts (`0.0.2`, `0.0.3`, etc.) are used as payers on both networks; (c) the attacker can create or observe a mainnet transaction whose bytes match a testnet transaction that has already been signed.
- **Barrier**: the valid-start timestamp must match. This is achievable by copying the testnet transaction bytes directly into a new mainnet transaction record â€” a one-API-call operation for any organization member who can create transactions. The node validation at creation time does not block this because nodes `0.0.3`â€“`0.0.9` are valid on both testnet and mainnet. [7](#0-6) 

## Recommendation

1. **Embed the network identifier in the signed payload**: When a transaction is frozen and stored, include the `mirrorNetwork` value in the data that is signed (e.g., as a memo or as part of a wrapper structure). Signers must sign over `(transactionBytes || networkId)` so that a testnet signature is cryptographically invalid for mainnet.

2. **Validate network at signature ingestion**: In both `importSignatures` and `processTransactionSignatures`, after loading the stored transaction, compare `transaction.mirrorNetwork` against the network context of the incoming signature map. Since the signature map itself carries no network tag, the practical fix is to require the caller to supply the expected network and reject any submission where the stored `mirrorNetwork` does not match.

3. **Short-term mitigation**: At minimum, add a check in `importSignatures` and `processTransactionSignatures` that the submitted signature map's transaction ID (payer + valid-start) matches the stored transaction's `transactionId` field **and** that the stored `mirrorNetwork` is passed through to a network-aware verification step.

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }

export const getClientFromNetwork = async (
  mirrorNetwork: string | string[],
  ledgerId?: string | LedgerId,
) => {
  if (!Array.isArray(mirrorNetwork)) {
    mirrorNetwork = [mirrorNetwork];
  }

  mirrorNetwork = mirrorNetwork.map(network => network.toLocaleLowerCase());
  if ([MAINNET, TESTNET, PREVIEWNET].includes(mirrorNetwork[0])) {
    return Client.forName(mirrorNetwork[0]);
  }

  if (mirrorNetwork[0] === LOCAL_NODE) {
    return Client.forNetwork(getLocalClientNetwork(process.env.NODE_ENV)).setMirrorNetwork(
      MirrorNetworkGRPC.LOCAL_NODE,
    );
  }

  const client = Client.forNetwork({}).setMirrorNetwork(
    MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0]),
  );

  const nodeAddressBook = await new AddressBookQuery()
    .setFileId(FileId.ADDRESS_BOOK)
    .execute(client);

  client.setNetworkFromAddressBook(nodeAddressBook);

  if (ledgerId) {
    client.setLedgerId(ledgerId);
  }

  return client;
};
