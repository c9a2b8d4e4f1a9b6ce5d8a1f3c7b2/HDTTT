Audit Report

## Title
Cryptographically Weak PRNG (`Math.random()`) Used to Generate Temporary User Passwords

## Summary
`AuthService.generatePassword()` in `back-end/apps/api/src/auth/auth.service.ts` uses `Math.random()` — a non-cryptographic PRNG (V8 xorshift128+) — to produce the temporary password sent to every newly admin-registered user. Because `Math.random()` is not seeded from a cryptographically secure entropy source and its full 128-bit state is reconstructible from a small number of observed outputs, an attacker can predict temporary passwords and take over newly created accounts before the legitimate user changes their credential.

## Finding Description

**Root cause — weak PRNG for credential generation:**

`generatePassword()` at lines 147–154 of `back-end/apps/api/src/auth/auth.service.ts` builds a temporary password exclusively from `Math.random()`:

```ts
private generatePassword() {
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () =>
      String.fromCharCode(97 + Math.floor(Math.random() * 26))
    ).join('');

  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
``` [1](#0-0) 

This password is the **sole initial credential** for every user registered by an admin. It is hashed and emailed to the user:

```ts
const tempPassword = this.generatePassword();
// ...
emitUserRegistrationEmail(this.notificationsPublisher, [{
  email: user.email,
  additionalData: { url, tempPassword, downloadUrl }
}]);
``` [2](#0-1) 

**Password space is small:** The output is 10 lowercase ASCII letters (5+5), yielding 26^10 ≈ 1.4 × 10^14 combinations. Against a bcrypt/argon2 hash this is not trivially brute-forced offline, but the PRNG weakness reduces the effective search space dramatically.

**PRNG state-reconstruction attack surface:**

The same Node.js process exposes `Math.random()` outputs through two additional production code paths whose timing is externally observable:

- `cache-management.service.ts` line 58: `const jitterMs = Math.random() * 2000;` — a 0–2 s startup jitter on a cron job that fires every 30 seconds. [3](#0-2) 

- `mirror-node.client.ts` line 147: `const jitter = exponentialDelay * 0.25 * (Math.random() * 2 - 1);` — retry-backoff jitter observable through response latency on mirror-node-dependent endpoints. [5](#0-4) 

V8's xorshift128+ PRNG state (128 bits) can be fully reconstructed from as few as two consecutive 64-bit `Math.random()` outputs. Once the state is known, all past and future outputs — including those used to build `tempPassword` — are deterministic.

## Impact Explanation

A successful attack yields **full account takeover** of a newly registered user. The attacker gains a valid JWT session, can change the user's password (locking out the legitimate user), and can access all organization transactions, keys, and signing workflows that the compromised account is authorized for. Because the temporary password is the only credential until the user performs a first login and password change, the window of exposure exists for every new registration.

## Likelihood Explanation

The attack requires: (a) ability to send crafted requests to the API to induce mirror-node retries, (b) ability to measure response latencies with millisecond precision, and (c) knowledge of when a new user is registered. Conditions (a) and (b) are achievable by any unauthenticated external attacker with network access to the API. Condition (c) is achievable by a malicious existing organization member who can observe the user list. The PRNG state-reconstruction technique against V8 xorshift128+ is publicly documented and tooled. Likelihood is **medium**.

## Recommendation

Replace `Math.random()` in `generatePassword()` with Node.js's cryptographically secure `crypto.randomBytes()` or `crypto.getRandomValues()`:

```ts
import { randomBytes } from 'crypto';

private generatePassword(): string {
  const getRandomLetters = (length: number): string => {
    const bytes = randomBytes(length);
    return Array.from(bytes, b => String.fromCharCode(97 + (b % 26))).join('');
  };
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
```

This ensures the temporary password is derived from OS-level entropy and is not predictable even with full knowledge of the process's `Math.random()` state. Additionally, consider increasing password length and character set to raise the brute-force cost independently of the PRNG choice.

## Proof of Concept

1. Attacker sends repeated requests to a mirror-node-dependent API endpoint, causing retries and measuring response latencies to extract `Math.random()` jitter values from `mirror-node.client.ts` line 147.
2. Using a published V8 xorshift128+ state-recovery tool (e.g., `v8-randomness-predictor`), attacker reconstructs the full 128-bit PRNG state from two or more observed float values.
3. Attacker (as an existing org member) monitors the user list for a newly registered account.
4. Attacker replays the PRNG forward through the exact call sequence that `generatePassword()` will use for the next registration event, computing the resulting `tempPassword`.
5. Attacker calls `POST /auth/login` with the new user's email and the predicted `tempPassword`, obtaining a valid JWT before the legitimate user logs in and changes their password.
