### Title
Stale Pre-Update Data in `approveTransaction` Causes `emitTransactionStatusUpdate` to Never Fire on Approval

### Summary
In `approvers.service.ts`, the `approveTransaction` function fetches `userApprovers` from the database **before** persisting the new approval, then uses that stale snapshot in the post-update notification routing condition. Because the old records still carry `approved = null`, the expression `userApprovers.every(a => a.approved)` is always `false` when `dto.approved = true`. As a result, `emitTransactionStatusUpdate` is never emitted on an approval action — only `emitTransactionUpdate` is — regardless of whether all required approvals have been collected.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` first loads the approver rows: [1](#0-0) 

It then persists the new `approved` value to the database: [2](#0-1) 

After the write, the notification branch is chosen using the **pre-update** `userApprovers` snapshot: [3](#0-2) 

The guard at line 563 guarantees that at least one approver in `userApprovers` still has `approved = null` (otherwise the call would have been rejected as already-approved). Therefore:

- `!dto.approved` is `false` when the user is approving.
- `userApprovers.every(a => a.approved)` evaluates against stale `null` values → always `false`.
- The combined condition is always `false` when `dto.approved = true`.

`emitTransactionStatusUpdate` is therefore **never** reached on an approval; `emitTransactionUpdate` is always called instead. This is the direct analog of the Maia bug: a flag/condition that always resolves to the same branch regardless of user intent, due to an incorrect operand (stale data instead of post-update data).

The guard that prevents double-approval: [4](#0-3) 

confirms that `userApprovers` always contains at least one record with `approved = null` at the point of the stale check, making the bug deterministic.

### Impact Explanation
`emitTransactionStatusUpdate` is the event that signals downstream consumers (the Chain service) to re-evaluate whether a transaction has cleared all approval gates and should advance to execution. Because this event is never emitted when a user approves, the transaction can remain permanently stuck in `WAITING_FOR_EXECUTION` even after every required approver has signed — unless an out-of-band trigger (e.g., a separate approver removal, a manual re-submission) happens to fire `emitTransactionStatusUpdate` for the same transaction. This constitutes a potential permanent freeze of user funds/transactions with no in-protocol recovery path through the normal approval workflow.

### Likelihood Explanation
Every approval action by every user on every transaction hits this code path. No special attacker capability is required — any legitimate approver exercising the normal approval workflow triggers the bug. The condition is deterministically wrong: the stale-data check can never return `true` for a first-time approval, so the wrong event is always emitted.

### Recommendation
Re-fetch the approver state **after** the database update, or mutate the in-memory `userApprovers` array to reflect the newly written values before evaluating the notification condition:

```typescript
// After the DB update, reflect the new approved value in memory
const updatedUserApprovers = userApprovers.map(a => ({ ...a, approved: dto.approved }));

if (!dto.approved || updatedUserApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

### Proof of Concept

1. Create a transaction requiring one approver (User A).
2. User A calls `POST /transactions/:id/approvers/approve` with `{ approved: true, ... }`.
3. Inside `approveTransaction`:
   - `userApprovers` is loaded: `[{ userId: A, approved: null, signature: null }]`.
   - The DB update sets `approved = true` for that record.
   - Condition evaluated: `!true || [{ approved: null }].every(a => a.approved)` → `false || false` → `false`.
   - `emitTransactionUpdate` fires; `emitTransactionStatusUpdate` does **not** fire.
4. The Chain service never receives the status-update signal; the transaction remains in `WAITING_FOR_EXECUTION` indefinitely despite all approvals being present in the database. [5](#0-4)

```typescript
  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }
```
