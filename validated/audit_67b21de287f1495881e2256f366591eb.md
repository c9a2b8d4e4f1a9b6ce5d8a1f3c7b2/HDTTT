### Title
Unauthenticated Public Key Ownership Claim Allows Key Squatting in `uploadKey()`

### Summary
Any authenticated user in the organization can register an arbitrary Hedera public key as their own without proving possession of the corresponding private key. Because `uploadKey()` performs no cryptographic ownership verification, a malicious user who observes a victim's public key (e.g., from on-chain activity or API responses) can register it first, permanently blocking the legitimate owner from associating that key with their account.

### Finding Description
The root cause is in `uploadKey()` in `back-end/apps/api/src/user-keys/user-keys.service.ts`.

When a new public key is submitted, the service checks only whether the key already exists in the database:

```
// line 42-45
let userKey = await this.repo.findOne({
  where: { publicKey: dto.publicKey },
  withDeleted: true,
});
```

If the key is not yet registered, it is unconditionally assigned to the caller:

```
// line 57-60
} else {
  userKey = await this.repo.create(dto);
  userKey.user = user;
}
```

The only guard against cross-user conflict is the check at line 52:

```
if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
  throw new BadRequestException(ErrorCodes.PU);
}
```

This check fires only **after** a key is already registered. It does not prevent a race to register first. There is no challenge-response, no signed message, and no proof that the submitting user controls the private key corresponding to `dto.publicKey`. Any authenticated user can submit any public key string and claim ownership.

The controller endpoint `POST /user/:userId?/keys` is protected only by JWT authentication (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) — meaning any verified organization member is a valid attacker. [1](#0-0) [2](#0-1) 

### Impact Explanation
- A victim's Hedera public key, once squatted by an attacker, cannot be registered by the legitimate owner — the `ErrorCodes.PU` check at line 52 will permanently reject their attempt.
- In Organization Mode, `UserKey` records are the mechanism by which users are associated with multi-signature transaction signing. A user whose key is squatted cannot participate in threshold-signature workflows, effectively locking them out of the core product function.
- The attacker gains no signing capability (they lack the private key), but the denial-of-service to the victim is permanent without admin intervention. [3](#0-2) 

### Likelihood Explanation
- **Attacker precondition**: Must be a verified organization member (no admin privileges required). Any user who has completed the sign-up flow qualifies.
- **Key discovery**: Hedera public keys are not secret — they appear in account info queries on the Hedera mirror node, in transaction records, and may be visible in API responses from `getUserKeysRestricted()` (line 69-81), which returns `publicKey` for any `userId`.
- **Timing**: Unlike blockchain mempool front-running, this is a simple first-write-wins race with no time pressure. The attacker can register a key at any time before the victim does. [4](#0-3) 

### Recommendation
Require the caller to prove possession of the private key before registering a public key. The standard approach is a **challenge-response**:

1. Issue a server-generated random nonce tied to the requesting user's session.
2. Require the client to sign the nonce with the private key corresponding to `dto.publicKey`.
3. Verify the signature server-side using the submitted `publicKey` before persisting the record.

This ensures only the holder of the private key can register the corresponding public key, directly mirroring the fix recommended in the external SSV report.

### Proof of Concept

**Setup**: Two verified organization accounts — `victim` and `attacker`. The victim's Hedera public key is `302a300506032b6570032100<victim_pubkey_hex>`.

1. Attacker queries `GET /user/<victim_userId>/keys` — the response includes the victim's `publicKey` (visible to any authenticated user via `getUserKeysRestricted`).
2. Attacker sends:
   ```
   POST /user/keys
   Authorization: Bearer <attacker_jwt>
   { "publicKey": "302a300506032b6570032100<victim_pubkey_hex>" }
   ```
3. Server executes the `else` branch at line 57-60, creates a `UserKey` record owned by the attacker.
4. Victim later sends the same `POST /user/keys` request with their own JWT.
5. Server finds the existing record at line 42-45, evaluates `userKey.userId !== user.id` as `true` at line 52, and throws `ErrorCodes.PU`.
6. Victim is permanently blocked from registering their own key. [5](#0-4)
