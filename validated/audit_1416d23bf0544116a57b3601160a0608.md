Audit Report

## Title
Unsigned Electron Binaries and Organization-Server-Controlled Auto-Update URL Enable Malicious Binary Installation

## Summary
The Hedera Transaction Tool distributes Electron desktop binaries with macOS notarization explicitly disabled and Windows code signing unimplemented. The auto-updater's feed URL is derived directly from the connected organization server's version-check API response, with no independent verification. A malicious or compromised organization server can redirect the updater to an attacker-controlled host, serve a crafted `latest.yml` and backdoored binary, and have it installed on the victim's machine.

## Finding Description

**Root Cause 1 — Unsigned distributed binaries:**

`front-end/electron-builder.yml` line 36 explicitly sets `notarize: false` for macOS builds: [1](#0-0) 

`front-end/RELEASE.md` line 3 shows Windows code signing is still unimplemented ("TBD"): [2](#0-1) 

**Root Cause 2 — Auto-updater feed URL sourced from organization server:**

The `updateUrl` is populated directly from the organization server's version-check API response into `organizationUpdateUrls` in `front-end/src/renderer/stores/versionState.ts`: [3](#0-2) 

The `updateUrl` is then computed globally from `organizationUpdateUrls` and passed to `ElectronUpdaterService.initialize()`: [4](#0-3) 

`ElectronUpdaterService.initialize()` sets the feed URL with `provider: 'generic'` to whatever URL the org server returned: [5](#0-4) 

The `provider: 'generic'` mode fetches a `latest.yml` from the attacker-controlled URL and verifies only the SHA512 hash declared in that same `latest.yml`. Since the attacker controls both the binary and the `latest.yml`, this hash check provides no security guarantee. [6](#0-5) 

On the server side, the `updateUrl` is constructed from the `FRONTEND_REPO_URL` environment variable, which any organization server operator can freely set to any arbitrary URL: [7](#0-6) 

**Exploit flow:**
1. Attacker operates or compromises an organization server and sets `FRONTEND_REPO_URL` to an attacker-controlled host (e.g., `https://attacker.com/releases`).
2. The server's version-check response returns `updateUrl: "https://attacker.com/releases/download/v9.9.9/"`.
3. The app stores this in `organizationUpdateUrls` and computes it as the global `updateUrl`.
4. When the app triggers an update check, `ElectronUpdaterService` sets the feed URL to the attacker's server.
5. The attacker's server serves a crafted `latest.yml` (with a version number higher than the current app) and a backdoored binary whose SHA512 matches the `latest.yml`.
6. `electron-updater` downloads and installs the backdoored binary.
7. On next launch, the malicious binary runs with full access to the local SQLite database containing encrypted Hedera private keys.

## Impact Explanation
The Electron app manages BIP-39 mnemonics and Hedera private keys in a local SQLite database encrypted with `argon2`. A backdoored binary has full filesystem access to this database and can exfiltrate keys or mnemonics, leading to complete compromise of all Hedera accounts managed by the tool. [8](#0-7) 

## Likelihood Explanation
The attacker only needs to operate a reachable organization server — this is an explicitly supported workflow (any user can add an organization server via the UI). No privileged access to the victim's machine or the legitimate GitHub account is required. The update flow is triggered when the server reports a newer version, requiring only the user to click the "Install Update" button in the normal app UI. The absence of code signing means the OS provides no warning that the binary is from an unknown publisher. [9](#0-8) 

## Recommendation
1. **Pin the update URL client-side**: Hardcode the allowed update URL base (e.g., `https://github.com/hashgraph/hedera-transaction-tool/releases`) in the Electron app and reject any `updateUrl` from an organization server that does not match this allowlist. The org server should only be trusted to report version numbers, not download locations.
2. **Enable macOS notarization**: Set `notarize: true` and configure Apple notarization credentials in the CI/CD pipeline.
3. **Implement Windows code signing**: Complete the "TBD" step in `RELEASE.md` by configuring a valid Windows code signing certificate.
4. **Use `provider: github`** in the auto-updater (matching the `publish` config in `electron-builder.yml`) instead of `provider: 'generic'`, so that `electron-updater` can leverage GitHub's release infrastructure and its own signature verification.

## Proof of Concept
1. Deploy a NestJS organization server with `FRONTEND_REPO_URL=https://attacker.com/releases` and `LATEST_SUPPORTED_FRONTEND_VERSION=9.9.9`.
2. Connect the Hedera Transaction Tool to this server.
3. The app calls `POST /users/version-check` and receives `updateUrl: "https://attacker.com/releases/download/v9.9.9/"`.
4. This URL is stored in `organizationUpdateUrls` and passed to `ElectronUpdaterService.initialize()` with `provider: 'generic'`.
5. Host `https://attacker.com/releases/download/v9.9.9/latest.yml` with a version of `9.9.9` and the SHA512 of a backdoored binary.
6. Host the backdoored binary at the path referenced in `latest.yml`.
7. The app downloads and installs the backdoored binary. On next launch, the malicious binary has full access to the local SQLite database containing encrypted private keys. [10](#0-9) [11](#0-10)

```typescript
  private getUpdateUrl(): string | null {
    const repoUrl = this.configService.get<string>('FRONTEND_REPO_URL');
    const latestVersion = this.configService.get<string>('LATEST_SUPPORTED_FRONTEND_VERSION');

    if (!repoUrl || !latestVersion) {
      return null;
    }

    const cleanLatest = semver.clean(latestVersion);
    if (!cleanLatest) {
      return null;
    }

    const baseUrl = repoUrl.replace(/\/+$/, '');

    return `${baseUrl}/download/v${cleanLatest}/`;
  }
```

```typescript
export async function hash(data: string, usePseudoSalt = false): Promise<string> {
  let pseudoSalt: Buffer | undefined;
  if (usePseudoSalt) {
    const paddedData = data.padEnd(16, 'x');
    pseudoSalt = Buffer.from(paddedData.slice(0, 16));
  }
  return await argon2.hash(data, {
    salt: pseudoSalt,
  });
```
