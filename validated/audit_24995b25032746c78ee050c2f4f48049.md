### Title
Duplicate `userId` Entries in `createTransactionObservers` Input Array Bypass Deduplication Check, Creating Multiple Observer Records

### Summary
`ObserversService.createTransactionObservers` checks for duplicate observers only against the **already-persisted** database state, not against the **input array itself**. Submitting the same `userId` multiple times in a single `POST /transactions/:id/observers` request bypasses the guard and inserts multiple `TransactionObserver` rows for the same user, corrupting the observer list and causing duplicate notifications.

### Finding Description
In `back-end/apps/api/src/transactions/observers/observers.service.ts`, the deduplication guard at lines 49–53 is:

```typescript
for (const userId of dto.userIds) {
  if (!transaction.observers.some(o => o.userId === userId)) {
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);
  }
}
```

`transaction.observers` is the snapshot loaded from the database **before** the loop begins. [1](#0-0) 

When `dto.userIds` contains `[5, 5]`, both iterations evaluate `transaction.observers.some(o => o.userId === 5)` against the same unchanged DB snapshot. Both return `false`, so two `TransactionObserver` objects for `userId = 5` are pushed into the local `observers` array and then persisted together via `this.repo.save(observers)`. [2](#0-1) 

The DTO `CreateTransactionObserversDto` applies only `@IsArray()`, `@IsNumber({}, { each: true })`, and `@ArrayMinSize(1)` — no `@ArrayUnique()` or equivalent constraint. [3](#0-2) 

The existing unit test "should not add an observer if already added" only covers the case where the userId is **already in the database**, not the case where the same userId appears **twice in the same request**. [4](#0-3) 

### Impact Explanation
- **Data integrity**: Multiple `TransactionObserver` rows are created for the same `(transactionId, userId)` pair. Every downstream query that returns `transaction.observers` will include duplicates, corrupting the observer list visible to the creator and to the observer themselves.
- **Duplicate notifications**: `emitTransactionUpdate` is called once per save, but every subsequent status-change event iterates over `transaction.observers`. A user appearing N times receives N copies of every notification, constituting notification spam proportional to the number of duplicates submitted.
- **Batch failure (if unique DB constraint exists)**: If the schema enforces a unique constraint on `(transactionId, userId)`, the entire `repo.save(observers)` call fails, meaning **all** valid userIds in the same request are also rejected — a self-inflicted denial of observer assignment.

### Likelihood Explanation
Any authenticated, verified user can create a transaction and is then the sole party authorized to add observers to it. No elevated privilege is required beyond holding a valid JWT. The attacker path is: register → verify → create transaction → POST `/transactions/:id/observers` with `{ "userIds": [X, X, X, ...] }`. This is a direct API call with no special tooling.

### Recommendation
Add an intra-request uniqueness check before the loop, or deduplicate the input array at the DTO level:

**Option A — DTO-level (preferred):** Add `@ArrayUnique()` from `class-validator` to `CreateTransactionObserversDto.userIds`.

**Option B — Service-level:** Deduplicate the input before iterating:
```typescript
const uniqueUserIds = [...new Set(dto.userIds)];
for (const userId of uniqueUserIds) { ... }
```

### Proof of Concept
**Preconditions:** Two valid accounts — `creator` (JWT: `TOKEN_A`) and `target` (userId: `42`).

1. Creator creates a transaction, receives `transactionId: 7`.
2. Creator sends:
```http
POST /transactions/7/observers
Authorization: Bearer TOKEN_A
Content-Type: application/json

{ "userIds": [42, 42, 42] }
```
3. **Expected (correct):** One `TransactionObserver` row for `(transactionId=7, userId=42)`.
4. **Actual (vulnerable):** Three `TransactionObserver` rows for `(transactionId=7, userId=42)` are inserted (or the entire save fails with a constraint error, rejecting all observers in the batch).
5. Subsequent `GET /transactions/7/observers` returns user 42 listed three times; every transaction update triggers three notification events for user 42. [5](#0-4) [6](#0-5)
