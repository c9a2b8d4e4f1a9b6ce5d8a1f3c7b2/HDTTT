Audit Report

## Title
Unauthenticated-Scope Data Exposure via Unguarded `getTransactionGroups` Endpoint and Missing Authorization on Comment Routes

## Summary
Two endpoints in the NestJS API service expose organization-wide data to any authenticated user with no ownership or role checks. `GET /transaction-groups` returns every transaction group in the database with no user-scoping. The `CommentsController` allows any authenticated user to post to or read comments on any transaction they have no role in.

## Finding Description

**Issue 1 — `GET /transaction-groups` returns all groups to any authenticated user**

The `TransactionGroupsController` is guarded only by `JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard` at the class level — authentication only, no authorization. [1](#0-0) 

The `getTransactionGroups` handler, explicitly marked `TESTING ONLY`, passes no user context and calls the service with no filter: [2](#0-1) 

The service performs a bare `find` with no `where` clause, returning every `TransactionGroup` row in the database: [3](#0-2) 

By contrast, the singular `getTransactionGroup` correctly scopes results to the requesting user via `getTransactionGroupItemsQuery` and throws `UnauthorizedException` when the user has no items: [4](#0-3) 

---

**Issue 2 — `CommentsController` has no authorization on `createComment`, `getComments`, or `getCommentById`**

The controller carries only class-level authentication guards. The `createComment` handler has a developer TODO explicitly acknowledging the missing check: [5](#0-4) 

`getComments` and `getCommentById` expose comments on any transaction to any authenticated user with no ownership or participation check: [6](#0-5) 

The service performs bare queries with no user scoping: [7](#0-6) 

## Impact Explanation

- **Issue 1:** Any authenticated organization member can enumerate the IDs, metadata, and structure of every transaction group in the system — including those belonging to other users. Transaction groups contain financial transaction bytes, signing keys, and execution status. This is a full horizontal privilege escalation for read access across all groups.
- **Issue 2:** Any authenticated user can inject comments into transactions they have no role in (creator, signer, approver, observer), and read comments on any transaction. This pollutes audit trails and leaks transaction discussion data across organizational boundaries.

## Likelihood Explanation

Both endpoints are reachable with a standard authenticated HTTP request — no special tooling required. Any user who has registered and obtained a JWT can immediately call `GET /transaction-groups` or `POST /transactions/:id/comments`. The `TESTING ONLY` comment on Issue 1 confirms the endpoint was left in accidentally, making accidental or deliberate exploitation straightforward.

## Recommendation

**Issue 1:**
- Remove the `getTransactionGroups` endpoint entirely, or restrict it to admin-only access with a role guard.
- If a listing endpoint is needed, scope it to the requesting user using the same `getTransactionGroupItemsQuery` pattern already used in `getTransactionGroup`.

**Issue 2:**
- Before allowing `createComment`, verify the requesting user is a creator, signer, approver, or observer of the target transaction. Throw `UnauthorizedException` otherwise.
- Apply the same participation check to `getComments` and `getCommentById`.

  async getTransactionGroup(user: User, id: number, full?: boolean): Promise<TransactionGroup> {
    const group = await this.dataSource.manager.findOne(TransactionGroup, {
      where: { id },
    });

    if (!group) {
      throw new BadRequestException(ErrorCodes.TNF);
    }

    const query = getTransactionGroupItemsQuery(this.sqlBuilder, id, user);

    const rows = await this.dataSource.manager.query(
      query.text,
      query.values,
    );

    group.groupItems = rows.map(row => {
      const creator = this.dataSource.manager.create(User, {
        id: row.tx_creator_key_user_id,
        email: row.tx_creator_email,
      });

      const creatorKey = this.dataSource.manager.create(UserKey, {
        id: row.tx_creator_key_id,
        userId: row.tx_creator_key_user_id,
        user: creator,
      });

      const transaction = this.dataSource.manager.create(Transaction, {
        id: row.tx_id,
        name: row.tx_name,
        type: row.tx_type,
        description: row.tx_description,

        transactionHash: row.tx_transaction_hash,

        unsignedTransactionBytes: row.tx_unsigned_transaction_bytes,
