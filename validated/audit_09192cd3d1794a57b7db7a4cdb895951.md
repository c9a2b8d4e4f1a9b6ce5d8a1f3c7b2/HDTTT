### Title
Missing Authorization Check in `removeTransactionGroup` Allows Any Authenticated User to Delete Any Transaction Group

### Summary
The `removeTransactionGroup` function in `transaction-groups.service.ts` fetches and destroys a transaction group by ID without verifying that the requesting user owns or has any relationship to that group. Any authenticated user can permanently delete another user's transaction group and all its associated transactions by supplying the target group's integer ID.

### Finding Description
**Vulnerability class:** Authorization bypass (missing ownership check on a destructive operation — direct analog to H-02's missing `_cardId < numberOfCards` bounds check).

In `back-end/apps/api/src/transactions/groups/transaction-groups.service.ts`, `removeTransactionGroup` (lines 172–198) fetches the group by ID with no user-ownership filter, then unconditionally removes all group items and the group itself: [1](#0-0) 

```typescript
async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
    // ← NO check: is `user` the creator/owner of this group?
    if (!group) { throw new Error('group not found'); }
    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, { ... });
    for (const groupItem of groupItems) {
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(transactionId, user, false);
    }
    await this.dataSource.manager.remove(TransactionGroup, group);
    return true;
}
```

The controller exposes this as `DELETE /transaction-groups/:id`, guarded only by JWT authentication — no ownership guard: [2](#0-1) 

Contrast with `cancelTransactionGroup`, which correctly calls `getTransactionGroup(user, groupId, false)` first (which runs a user-scoped SQL query) and then explicitly asserts `allOwnedByUser`: [3](#0-2) 

`removeTransactionGroup` has neither of these safeguards.

The `getTransactionGroup` function used by `cancelTransactionGroup` enforces access via `getTransactionGroupItemsQuery`, which builds a WHERE clause scoped to the requesting user's keys, observer status, signer status, and approver status: [4](#0-3) 

`removeTransactionGroup` bypasses this entirely.

### Impact Explanation
Any authenticated user (no admin role required) can:
1. Enumerate group IDs (sequential integers starting from 1).
2. Issue `DELETE /transaction-groups/<victim_group_id>` with their own valid JWT.
3. Permanently destroy the victim's transaction group, all `TransactionGroupItem` records, and all associated `Transaction` records.

This is unrecoverable data destruction of another user's transaction history, signatures, and execution state — a direct integrity failure.

### Likelihood Explanation
- **Attacker precondition:** A valid JWT token — any registered user qualifies.
- **Group ID discovery:** IDs are sequential integers; an attacker can enumerate them trivially or observe IDs from their own groups.
- **No rate limiting or ownership guard** blocks the `DELETE` endpoint.
- The endpoint is publicly documented via `@ApiTags('Transaction Groups')`.

### Recommendation
Mirror the pattern already used in `cancelTransactionGroup`: call `getTransactionGroup(user, id, false)` first (which throws `UnauthorizedException` if the user has no access to any transaction in the group), then assert that the user is the creator of all transactions before proceeding with deletion:

```typescript
async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    // Reuse existing access-controlled fetch — throws if user has no access
    const group = await this.getTransactionGroup(user, id, false);

    const allOwnedByUser = group.groupItems.every(
      item => item.transaction?.creatorKey?.userId === user.id,
    );
    if (!allOwnedByUser) {
      throw new UnauthorizedException('Only the creator can remove a transaction group.');
    }
    // ... proceed with deletion
}
```

### Proof of Concept
1. **User A** creates a transaction group → server returns `{ id: 5, ... }`.
2. **User B** (any authenticated user, unrelated to the group) sends:
   ```
   DELETE /transaction-groups/5
   Authorization: Bearer <userB_jwt>
   ```
3. Server calls `removeTransactionGroup(userB, 5)`.
4. `findOneBy(TransactionGroup, { id: 5 })` succeeds — no user filter.
5. All `TransactionGroupItem` records for group 5 are removed.
6. All associated `Transaction` records are removed.
7. The `TransactionGroup` record is removed.
8. **User A's group is permanently destroyed.** User B receives `true`. [1](#0-0) [2](#0-1)
