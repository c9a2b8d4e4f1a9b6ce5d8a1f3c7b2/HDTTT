### Title
Stale Pre-Update `userApprovers` State Used to Determine Notification Event Type in `approveTransaction`, Causing Missed Status Transitions

### Summary

In `approvers.service.ts`, the `approveTransaction` function fetches `userApprovers` from the database **before** applying the DB update, then uses those stale pre-update values to decide whether to emit `emitTransactionStatusUpdate` or `emitTransactionUpdate`. Because the stale snapshot is used, the condition `userApprovers.every(a => a.approved)` is almost always `false` when a user approves for the first time, causing the wrong event to be emitted and the transaction status to not be immediately re-evaluated.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction` function:

1. Fetches `userApprovers` at line 553 — these are the **pre-update** records with `approved = undefined/false`. [1](#0-0) 

2. Applies the DB update at lines 599–610, setting `approved = dto.approved` for all `userApprovers`. [2](#0-1) 

3. After the update, decides which event to emit using the **stale** `userApprovers` snapshot: [3](#0-2) 

The condition `userApprovers.every(a => a.approved)` evaluates the **old** values (before the update). Since the guard at line 563 already ensures at least one approver record lacks a signature, `userApprovers.every(a => a.approved)` is always `false` for a first-time approval. This means:

- When `dto.approved = true`: `!dto.approved` is `false`, and `userApprovers.every(a => a.approved)` is `false` (stale) → `emitTransactionUpdate` is called.
- When `dto.approved = false`: `!dto.approved` is `true` → `emitTransactionStatusUpdate` is called correctly.

The two emit functions publish to different NATS subjects (`TRANSACTION_UPDATE` vs `TRANSACTION_STATUS_UPDATE`): [4](#0-3) 

`emitTransactionStatusUpdate` triggers `processTransactionStatusUpdateNotifications` in the notifications service, which re-evaluates the transaction status and sends approval/execution notifications. `emitTransactionUpdate` does not trigger this re-evaluation. [5](#0-4) 

The chain service's scheduler (`updateTransactions`) does eventually re-evaluate transaction statuses on its own schedule: [6](#0-5) 

However, this is a periodic background job, not an immediate response to the approval event.

### Impact Explanation

Every time a legitimate approver calls `POST /transactions/:transactionId/approvers/approve` with `approved: true`, the transaction status is **not immediately re-evaluated**. The system relies on the background scheduler to eventually pick it up. In the worst case:

- A transaction with a `validStart` close to its expiry window receives its final approval.
- The wrong event (`emitTransactionUpdate`) is emitted; the chain service does not immediately attempt execution.
- The scheduler's next run occurs after the `validStart` window has passed.
- The transaction expires (`EXPIRED` status) without ever being executed, even though all required approvals were in place.

This is a state transition integrity failure: the post-update state is never used to drive the notification/execution pipeline, causing the system to act on stale data — the direct analog of the external report's `_limitPrice` being set from `_trade.tpPrice` instead of the current `_price`.

### Likelihood Explanation

This is triggered on **every** approval call where `dto.approved = true` and the approver has not previously signed. This is the normal, expected flow for any approver. No special attacker capability is required — any legitimate approver of any transaction triggers this path. The only mitigating factor is the background scheduler, which bounds the delay but does not eliminate the window for expiry-based failures.

### Recommendation

Replace the stale-data check with a check based on `dto.approved` directly, since the update unconditionally sets all `userApprovers` to `approved = dto.approved`:

```typescript
// Current (buggy): uses stale pre-update snapshot
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}

// Fixed: always emit status update so the pipeline re-evaluates immediately
emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
```

If the intent is to distinguish between "status may have changed" and "status definitely changed," re-fetch the approvers after the update and call `processTransactionStatus` (as the signers service does at lines 421–464) to determine the correct event. [7](#0-6) 

### Proof of Concept

1. Create a transaction with a single approver (User A).
2. User A calls `POST /transactions/1/approvers/approve` with `{ approved: true, userKeyId: X, signature: <valid> }`.
3. In `approveTransaction`:
   - Line 553: `userApprovers = [{ userId: A, approved: undefined, signature: undefined }]`
   - Lines 599–610: DB update sets `approved = true` for the record.
   - Line 614: `!dto.approved` → `false`; `userApprovers.every(a => a.approved)` → `false` (stale `undefined`).
   - Line 617: `emitTransactionUpdate` is called — **not** `emitTransactionStatusUpdate`.
4. The notifications service receives a `TRANSACTION_UPDATE` event, not a `TRANSACTION_STATUS_UPDATE` event. The transaction status is not re-evaluated.
5. The transaction remains in `WAITING_FOR_SIGNATURES` until the chain service's background scheduler runs.
6. If the transaction's `validStart` expires before the scheduler runs, the transaction expires without execution despite having all required approvals. [8](#0-7)

  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }

  private async updateStatusesAndNotify(
    transactionsToProcess: Array<{ id: number; transaction: Transaction }>
  ) {
    if (transactionsToProcess.length === 0) return;

    // Process statuses in bulk
    let statusMap: Map<number, TransactionStatus>;
    try {
      statusMap = await processTransactionStatus(
        this.txRepo,
        this.transactionSignatureService,
        transactionsToProcess.map(t => t.transaction)
      );
    } catch (err) {
      console.error('Bulk status processing failed:', err);
      statusMap = new Map();
    }

    // Separate new statuses from unchanged
    const newStatusResults: number[] = [];
    const unchangedResults: number[] = [];

    for (const { id } of transactionsToProcess) {
      if (statusMap.has(id)) {
        newStatusResults.push(id);
      } else {
        unchangedResults.push(id);
      }
    }

    // Emit notifications
    if (newStatusResults.length > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        newStatusResults.map(id => ({ entityId: id }))
      );
    }
    if (unchangedResults.length > 0) {
      emitTransactionUpdate(
        this.notificationsPublisher,
        unchangedResults.map(id => ({ entityId: id }))
      );
    }
  }
