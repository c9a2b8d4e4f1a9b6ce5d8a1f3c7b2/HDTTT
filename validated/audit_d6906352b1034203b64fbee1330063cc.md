### Title
Threshold Value in `CreateTransactionApproverDto` Lacks Lower-Bound Validation Present on the Update Path, Allowing Negative Threshold Persistence

### Summary
The `UpdateTransactionApproverDto` enforces `@Min(1)` on the `threshold` field, preventing zero or negative values during updates. The `CreateTransactionApproverDto` used during initial approver creation has no such constraint, and the service-level guard only catches `threshold === 0` and `threshold > approvers.length`, leaving negative threshold values (e.g., `-1`) able to pass all validation and be persisted to the database. Any authenticated transaction creator can exploit this to store an invalid approver-tree threshold.

---

### Finding Description

**Root cause — DTO validation asymmetry:**

`UpdateTransactionApproverDto` enforces a minimum of 1:

```typescript
// back-end/apps/api/src/transactions/dto/update-transaction-approver.dto.ts
@IsNumber()
@Min(1)          // ← lower-bound enforced on update
@IsOptional()
threshold?: number;
```

`CreateTransactionApproverDto` has no such constraint:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
@IsNumber()
@IsOptional()
threshold?: number;   // ← no @Min(1) on creation
```

**Service-level guard is incomplete:**

Inside `createTransactionApprovers`, the only threshold guard is:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts  lines 303-307
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
```

For `threshold = -1` with `approvers = [userA, userB]`:
- `-1 > 2` → `false`
- `-1 === 0` → `false`

Both conditions are false; the guard is bypassed and the negative threshold is written to the database via:

```typescript
// lines 309-316
const data: DeepPartial<TransactionApprover> = {
  ...
  threshold: dtoApprover.threshold && dtoApprover.approvers ? dtoApprover.threshold : null,
  ...
};
await transactionalEntityManager.insert(TransactionApprover, approver);
```

The `validateApprover` helper also does not catch this case — it only rejects the case where **all** fields are simultaneously null/NaN/zero/empty:

```typescript
// lines 692-700
private validateApprover(approver: CreateTransactionApproverDto): void {
  if (
    (approver.listId === null || isNaN(approver.listId)) &&
    (approver.threshold === null || isNaN(approver.threshold) || approver.threshold === 0) &&
    ...
  )
    throw new BadRequestException(this.CANNOT_CREATE_EMPTY_APPROVER);
}
```

A payload of `{ threshold: -1, approvers: [{userId: X}, {userId: Y}] }` passes every check.

---

### Impact Explanation

A negative threshold is an invalid state for an approver tree. The `@Min(1)` guard on the update path exists precisely because a threshold below 1 is semantically meaningless (or dangerous) for the approval-counting logic. Persisting `threshold = -1` to the `transaction_approver` table means:

- Any downstream logic that evaluates `approvedCount >= threshold` will treat the condition as trivially satisfied (any non-negative count satisfies `>= -1`), potentially allowing a transaction to advance to `WAITING_FOR_EXECUTION` without the required approvals.
- The approver tree is in a permanently corrupt state that cannot be corrected via the update path (which would reject `threshold < 1`), creating an unrecoverable integrity failure for that transaction's approval workflow.

---

### Likelihood Explanation

The attacker precondition is minimal: a valid authenticated session and ownership of at least one transaction (i.e., being its creator). The `getCreatorsTransaction` check only verifies creator identity, not any elevated role. Any normal user of the organization mode can reach this code path by:

1. Creating a transaction.
2. Calling the `POST /transactions/:id/approvers` endpoint with a crafted body containing `threshold: -1`.

No admin keys, leaked secrets, or internal access are required.

---

### Recommendation

Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
import { IsArray, IsNumber, IsOptional, ValidateNested, ArrayMinSize, Min } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @IsOptional()
  listId?: number;

  @IsNumber()
  @Min(1)          // ← add this
  @IsOptional()
  threshold?: number;
  ...
}
```

Additionally, update the service-level guard to use `dto.threshold < 1` instead of `dto.threshold === 0` to provide defence-in-depth:

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold < 1)
)
```

### Proof of Concept

**Preconditions:** Authenticated as a normal organization user who is the creator of transaction ID `42`.

**Request:**
```http
POST /transactions/42/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 10 },
        { "userId": 11 }
      ]
    }
  ]
}
```

**Expected (correct) behaviour:** HTTP 400 — threshold must be ≥ 1.

**Actual behaviour:** HTTP 201 — a `TransactionApprover` row with `threshold = -1` is inserted into the database, bypassing the `@Min(1)` guard that would have rejected the same value on the update path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
