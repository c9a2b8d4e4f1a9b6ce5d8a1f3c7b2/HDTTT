Audit Report

## Title
Cross-Transaction Authorization Bypass: Any Transaction Creator Can Delete Approvers from Transactions They Do Not Own

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies only that the authenticated user is the creator of the `transactionId` in the URL, then calls `removeTransactionApprover(id)` with no check that the approver record being deleted actually belongs to that transaction. An attacker who has created at least one transaction can supply their own `transactionId` to pass the ownership check, then supply an arbitrary approver `id` from a completely different transaction to delete it.

## Finding Description

**Root cause — controller does not bind the approver to the URL transaction:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler:

```typescript
@Delete('/:id')
async removeTransactionApprover(@GetUser() user, @Param('transactionId') transactionId, @Param('id') id) {
    await this.approversService.getCreatorsTransaction(transactionId, user);  // checks ownership of URL transactionId only
    await this.approversService.removeTransactionApprover(id);                // deletes approver by id with NO cross-check
    return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the caller owns `transactionId` — it does not inspect the approver record at all: [2](#0-1) 

`removeTransactionApprover` then fetches and deletes the approver by its own `id` with no verification that `approver.transactionId === transactionId`: [3](#0-2) 

The `removeNode` SQL performs a recursive soft-delete of the entire subtree rooted at the given `id`, with no ownership guard: [4](#0-3) 

**Contrast with `updateTransactionApprover`**, which correctly validates cross-ownership by walking up to the root node and comparing `rootNode.transactionId !== transactionId` before proceeding: [5](#0-4) 

**Exploit path:**
1. Attacker (authenticated user) creates Transaction A — they are its creator.
2. Attacker discovers the `id` of an approver record belonging to Transaction B (owned by another user). Approver IDs are auto-incremented integers; the attacker can enumerate them or observe them via `GET /transactions/:id/approvers` for any transaction they have read access to (the `getVerifiedApproversByTransactionId` endpoint exposes approver `id` fields in the serialized DTO). [6](#0-5) 
3. Attacker calls `DELETE /transactions/{Transaction_A_id}/approvers/{approver_B_id}`.
4. `getCreatorsTransaction(Transaction_A_id, attacker)` passes — attacker owns Transaction A.
5. `removeTransactionApprover(approver_B_id)` soft-deletes the approver (and its entire subtree) from Transaction B with no ownership check.
6. After deletion, `emitTransactionStatusUpdate` is fired using `approver.transactionId` (Transaction B's ID), immediately triggering re-evaluation of Transaction B's approval status. [7](#0-6) 

## Impact Explanation

The `isApproved` utility evaluates the live approver tree recursively:

```typescript
const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
if (approvals.length >= (approver.threshold || approvals.length)) {
    return true;
}
``` [8](#0-7) 

If Transaction B has a threshold-based tree (e.g., threshold = 2, children = [Approver1 ✓, Approver2 ✓, Approver3 ✗]) and the attacker deletes Approver3, the tree now has 2 children both approved. The condition `approvals.length (2) >= threshold (2)` evaluates to `true` — the transaction is now considered approved with one fewer approval than the creator required.

An attacker who is any authenticated organization member (no admin role required) can:
- Delete approvers from any other user's transaction, reducing the effective approval count required.
- Force a transaction that was blocked pending a required approval to appear fully approved, potentially triggering its execution on the Hedera network without the intended governance sign-off.
- Permanently corrupt the approval tree of any in-flight transaction (`WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`), since `getCreatorsTransaction` has no status guard. [9](#0-8) 

## Likelihood Explanation

- **Attacker precondition**: Must be an authenticated, verified organization member. No admin or special role required — creating a transaction is a standard user action.
- **Discovery of approver IDs**: Approver IDs are auto-incremented integers exposed in the `TransactionApproverDto` response. Any approver the attacker can read (e.g., from their own transactions or from transactions they are an observer/signer of) leaks the ID space. Sequential enumeration is trivial.
- **No rate limiting or anomaly detection** is visible in the controller or service layer for this endpoint. [10](#0-9) 

## Recommendation

Apply the same cross-ownership check that `updateTransactionApprover` already performs. In `removeTransactionApprover` (or in the controller before calling it), after fetching the approver by `id`, walk up to its root node using `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId`. If they differ, throw an `UnauthorizedException`. This mirrors the existing pattern:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [5](#0-4) 

## Proof of Concept

```
# Setup:
# - Attacker is authenticated as user A, creator of Transaction 1
# - Transaction 2 is owned by user B, has approver tree:
#     root (threshold=2): [Approver id=10 (approved), Approver id=11 (approved), Approver id=12 (pending)]

# Step 1: Attacker observes approver id=12 via any readable transaction or enumeration

# Step 2: Attacker sends DELETE using their own transactionId=1 but victim's approver id=12
DELETE /transactions/1/approvers/12
Authorization: Bearer <attacker_jwt>

# Result:
# - getCreatorsTransaction(1, attacker) → PASSES (attacker owns Transaction 1)
# - removeTransactionApprover(12) → soft-deletes Approver 12 from Transaction 2
# - emitTransactionStatusUpdate fires for Transaction 2
# - Transaction 2's approval tree now has 2/2 approvals → evaluated as APPROVED
# - Transaction 2 may proceed to execution without the required third approval
```

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
