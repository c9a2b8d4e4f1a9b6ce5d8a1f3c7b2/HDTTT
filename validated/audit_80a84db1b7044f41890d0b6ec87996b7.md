Audit Report

## Title
Unbounded Recursive Approver Tree Creation Enables Resource Exhaustion DoS

## Summary
The `createTransactionApprovers` endpoint accepts an arbitrarily deeply nested `approversArray` payload. The `CreateTransactionApproverDto` DTO has no nesting depth cap, and the inner `createApprover` helper in `approvers.service.ts` recurses without a depth guard, issuing multiple database queries per nesting level. With a connection pool of 3, a small number of concurrent deeply-nested requests exhausts the database connection pool and stalls all other users.

## Finding Description

**Root cause 1 — unbounded self-referential DTO:**

`CreateTransactionApproverDto` in `back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts` is self-referential with no `@ArrayMaxSize` or depth constraint:

```ts
@IsArray()
@ArrayMinSize(1)
@IsOptional()
@ValidateNested({ each: true })
@Type(() => CreateTransactionApproverDto)
approvers?: CreateTransactionApproverDto[];
``` [1](#0-0) 

**Root cause 2 — unbounded recursive service handler:**

Inside `createTransactionApprovers`, the inner `createApprover` function calls itself for every nested child with no depth counter or guard:

```ts
await createApprover({ ...nestedDtoApprover, listId: approver.id });
``` [2](#0-1) 

Each recursive invocation issues at minimum 3–4 database queries (duplicate check via `isNode`, parent existence check, root-node traversal via recursive SQL CTE, user existence check). [3](#0-2) 

**Root cause 3 — `validateApprover` does not check depth:**

The only structural validation is an emptiness check: [4](#0-3) 

**Mitigating factors that invalidate the "process crash" claim:**

- `createApprover` is `async` and uses `await` at each recursive call; V8 unwinds the call stack at every `await`, so a `RangeError: Maximum call stack size exceeded` from this specific path is not the actual crash vector.
- Even if a `RangeError` were thrown (e.g., during synchronous `class-transformer` deserialization), it is a catchable JS error; the outer `try/catch` at line 243 re-throws it as `BadRequestException`, and the process does not crash. [5](#0-4) 

**Actual attack surface — DB connection pool exhaustion:**

The global IP throttle (`GLOBAL_MINUTE_LIMIT: 10000`, `GLOBAL_SECOND_LIMIT: 1000`) is far too permissive to block this. [6](#0-5) 

The body size limit is 2 MB, which comfortably accommodates thousands of nesting levels. [7](#0-6) 

The database connection pool is configured at `POSTGRES_MAX_POOL_SIZE: 3`. [8](#0-7) 

The controller endpoint carries no per-endpoint throttle beyond the global IP guard: [9](#0-8) 

## Impact Explanation
An authenticated user who owns any transaction can send a single POST request with a payload whose `approversArray[0].approvers[0].approvers[0]...` chain is hundreds of levels deep (well within the 2 MB body limit). Each level triggers 3–4 synchronous DB queries inside a TypeORM transaction, holding a connection from the pool for the entire duration. With `POSTGRES_MAX_POOL_SIZE: 3`, two or three concurrent such requests exhaust the pool, causing all other DB-dependent API calls to queue indefinitely. This is an effective denial of service for all users. The process itself does not crash (contrary to the original claim), but the service becomes unresponsive until the long-running transactions complete or time out.

## Likelihood Explanation
The precondition is a valid JWT and ownership of any transaction — reachable by any registered user. Crafting a deeply nested JSON object is trivial. The global throttle (10,000 req/min) does not prevent a single slow, resource-intensive request. No per-endpoint rate limit, no depth guard, and no payload-structure validation exist between the HTTP boundary and the recursive DB-query loop.

## Recommendation
1. **Enforce a maximum nesting depth in the DTO** — add a custom `@MaxDepth(n)` validator or flatten the DTO to disallow nesting beyond a configurable limit (e.g., 5 levels).
2. **Add a depth counter to `createApprover`** — pass a `depth` parameter and throw `BadRequestException` when it exceeds the allowed maximum.
3. **Reduce the global throttle or add a per-endpoint throttle** on `POST /transactions/:transactionId/approvers` to limit how many times a single IP/user can call this endpoint per minute.
4. **Increase `POSTGRES_MAX_POOL_SIZE`** or set a query timeout so that a long-running transaction does not hold a connection indefinitely.

## Proof of Concept
```http
POST /transactions/1/approvers HTTP/1.1

Content-Type: application/json

{

Sending 3 such requests concurrently exhausts the `POSTGRES_MAX_POOL_SIZE: 3` pool, stalling all other API requests that require a DB connection.
