Audit Report

## Title
DoS: `getTransactionsToSign()` Fetches All Non-Terminal Transactions Without DB-Level Pagination, Enabling Unbounded Resource Exhaustion

## Summary

`TransactionsService.getTransactionsToSign()` issues an unbounded `repo.find()` with no `skip`/`take` limit, loading every non-terminal transaction from the database into memory. It then iterates over all of them, calling `userKeysToSign()` per transaction — which deserializes transaction bytes via the Hedera SDK and invokes `computeSignatureKey()` — before applying pagination in memory via `result.slice()`. An authenticated user can accumulate transactions in non-terminal status, causing each call to `GET /transactions/sign` to consume unbounded memory, CPU, and time.

## Finding Description

**Root cause — unbounded DB fetch followed by per-item async work:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` accepts `Pagination` parameters but never passes `skip`/`take` to the database query:

```typescript
// lines 295-299: no skip/take — fetches ALL non-terminal transactions
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then iterates over every result, calling `userKeysToSign()` per transaction:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { console.log(error); }
}
``` [2](#0-1) 

Pagination is applied only after the full iteration:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

**Per-iteration cost of `userKeysToSign()`:**

`userKeysToSign()` delegates to `userKeysRequiredToSign()`, which calls `keysRequiredToSign()`. Inside `keysRequiredToSign()`:

1. `SDKTransaction.fromBytes(transaction.transactionBytes)` is called — CPU-intensive deserialization of the full transaction blob.
2. `transactionSignatureService.computeSignatureKey(transaction, showAll)` is called — potentially involves mirror node or DB lookups. [4](#0-3) 

`userKeysRequiredToSign()` also calls `attachKeys(user, entityManager)` on every iteration, which may issue a DB query if keys are not yet cached. [5](#0-4) 

**Exposed endpoint:**

`GET /transactions/sign` is registered in the controller with only JWT guards — no per-endpoint rate limiting, no per-user transaction count cap: [6](#0-5) 

The `whereForUser` filter excludes only `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED` — leaving `NEW`, `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, and `REJECTED` in scope: [7](#0-6) 

**Contrast with properly paginated sibling methods:**

`getTransactionsToApprove()` passes `skip: offset, take: limit` directly to the DB query — `getTransactionsToSign()` is the sole outlier: [8](#0-7) 

**Global throttler does not mitigate this:**

An `IpThrottlerModule` exists with global per-IP rate limits, but these are coarse global limits, not endpoint-specific. A single authenticated user from a single IP can still accumulate thousands of transactions and trigger the unbounded query repeatedly within those limits. [9](#0-8) 

## Impact Explanation

- **Memory exhaustion:** All matching transaction rows (including their `transactionBytes` blobs) are loaded into the Node.js heap simultaneously. With thousands of transactions, this causes OOM or severe GC pressure.
- **CPU exhaustion:** `SDKTransaction.fromBytes()` is called once per transaction per request. Concurrent requests multiply this linearly.
- **Latency amplification:** `computeSignatureKey()` is called per transaction and may involve mirror node or DB lookups, serializing the entire response behind N async calls.
- **Service-wide degradation:** The API service is a single-process NestJS application. A sustained attack degrades or crashes the API for all users, not just the attacker.

## Likelihood Explanation

- **Attacker preconditions:** Only a valid JWT (registered, verified user with at least one key). No admin access required.
- **Attack steps:** Create many transactions via `POST /transactions` (each with a unique `transactionId`), then repeatedly call `GET /transactions/sign`. Transactions remain in `WAITING_FOR_SIGNATURES` until their `validStart` expires.
- **No per-endpoint rate limiting** is applied to `GET /transactions/sign` at the controller level.
- The endpoint is still live and reachable despite the comment `/* NO LONGER USED BY FRONT-END */`. [10](#0-9) 
- The attack is self-amplifying: more transactions → slower responses → easier to sustain.

## Recommendation

1. **Push pagination to the DB layer** in `getTransactionsToSign()`, mirroring the pattern used in `getTransactionsToApprove()`: pass `skip: offset, take: limit` to the `repo.find()` call. Because the filtering for `keysToSign.length > 0` happens in application logic, a two-phase approach is needed (e.g., fetch a larger batch, filter, repeat until page is filled, or pre-filter via a subquery).
2. **Add per-endpoint or per-user rate limiting** to `GET /transactions/sign` and `POST /transactions` using the existing `UserThrottlerGuard`. [11](#0-10) 
3. **Consider caching** the result of `computeSignatureKey()` per transaction to avoid redundant mirror node/DB calls across concurrent requests.

## Proof of Concept

```
# Step 1: Register and authenticate as a normal user, obtain JWT
TOKEN="<jwt>"

# Step 2: Create N transactions (each with a unique transactionId, validStart in the future)
for i in $(seq 1 5000); do
  curl -s -X POST https://<api>/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"transactionBytes":"<bytes>","creatorKeyId":<keyId>,...}'
done

# Step 3: Repeatedly call the unbounded endpoint
while true; do
  curl -s "https://<api>/transactions/sign?page=1&limit=10" \
    -H "Authorization: Bearer $TOKEN"
done
```

Each call to `GET /transactions/sign` in Step 3 will load all 5,000 transaction rows into memory, deserialize each via `SDKTransaction.fromBytes()`, and call `computeSignatureKey()` per transaction before returning 10 results. Concurrent callers multiply the resource consumption proportionally, degrading the API for all users.

@Injectable()

    if (!user) {
