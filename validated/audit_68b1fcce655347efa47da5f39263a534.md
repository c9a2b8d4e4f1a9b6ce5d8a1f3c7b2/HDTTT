### Title
Cross-Transaction Approver Deletion: Any Transaction Creator Can Remove Approvers from Transactions They Do Not Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the authenticated user is the creator of the transaction identified by the URL parameter `transactionId`, but then deletes the approver identified by the separate URL parameter `id` without verifying that the approver actually belongs to that same transaction. Any user who has created at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` in the URL and a victim transaction's approver `id`.

### Finding Description

**Vulnerability class:** Authorization bypass — authorization check is performed on one entity (`transactionId` from URL) while the destructive action is performed on a different entity (`approver.id`) with no cross-ownership validation.

**Root cause — `approvers.controller.ts` lines 102–113:**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no cross-check
  return true;
}

### Impact Explanation

An attacker who is the creator of **any** transaction (even a trivially created one) can:

1. Enumerate or guess approver IDs belonging to a victim's transaction.
2. Call `DELETE /transactions/{attacker_tx_id}/approvers/{victim_approver_id}`.
3. The `getCreatorsTransaction` check passes (attacker owns `attacker_tx_id`).
4. `removeTransactionApprover(victim_approver_id)` deletes the approver — and its entire child subtree — from the victim's transaction.

This allows an attacker to **strip all approval requirements from any transaction**, causing a transaction that requires multi-party approval to become executable without those approvals. In an organization context where transactions control Hedera account operations (transfers, key changes, file updates), this is a direct integrity failure in the trust model.

### Likelihood Explanation

- **Attacker preconditions:** Authenticated user with no admin privileges; must have created at least one transaction (trivially achievable by any registered user).
- **Knowledge required:** The approver `id` of the target transaction. IDs are sequential integers and can be enumerated via the `GET /transactions/:transactionId/approvers` endpoint (accessible to any participant of a transaction) or by brute-force.
- **No privileged access required.** Any regular organization member can exploit this.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver's root transaction matches the `transactionId` supplied in the URL, exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Add this cross-ownership check (mirrors updateTransactionApprover lines 386-391)
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, and update the controller signature accordingly.

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (attacker-owned).
- User B creates Transaction 2 with a required approver (approver record ID = 42).

**Exploit:**
```http
DELETE /transactions/1/approvers/42
Authorization: Bearer <User A's JWT>
```

**Expected (correct) behavior:** 403 Forbidden — User A does not own Transaction 2.

**Actual behavior:** 200 OK — `getCreatorsTransaction(1, UserA)` passes; `removeTransactionApprover(42)` deletes approver 42 (belonging to Transaction 2) and all its children with no further checks.

**Result:** Transaction 2's approval requirement is silently removed. The transaction can now proceed to execution without the required approvals, violating the multi-signature workflow integrity.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
