### Title
Transaction Creator Can Modify Approval Threshold on In-Progress Transactions, Bypassing Multi-Signature Requirements

### Summary
The `updateTransactionApprover` endpoint allows the transaction creator to change the approval `threshold` (and restructure the approver tree) on transactions that are already in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status. No transaction-status guard exists in `getCreatorsTransaction` or `updateTransactionApprover`. A malicious creator can lower the threshold after partial approvals have been collected, causing the transaction to satisfy the approval requirement with fewer signatures than the approvers originally agreed to.

### Finding Description

**Root cause — missing status guard in `updateTransactionApprover`**

`approvers.service.ts` `updateTransactionApprover` (lines 367–531) performs four checks before mutating an approver record:

1. DTO has exactly one field.
2. Approver exists.
3. Root node's `transactionId` matches the URL param.
4. Caller is the transaction creator (`getCreatorsTransaction`). [1](#0-0) 

`getCreatorsTransaction` only verifies ownership — it never inspects `transaction.status`: [2](#0-1) 

The threshold-update branch (lines 467–488) then writes the new value unconditionally: [3](#0-2) 

The existing unit-test fixture for `updateTransactionApprover` explicitly uses `status: TransactionStatus.WAITING_FOR_EXECUTION`, confirming the update succeeds in that state: [4](#0-3) 

The HTTP surface is `PATCH /transactions/:transactionId/approvers/:id`, accessible to any authenticated creator: [5](#0-4) 

**Secondary vector — automatic threshold reduction on approver removal**

`removeTransactionApprover` (lines 534–544) also has no status guard. When a child approver is detached via `listId: null`, the parent's threshold is silently reduced if it would otherwise exceed the remaining child count: [6](#0-5) 

This gives the creator a second path to lower the effective approval bar after signatures have been collected.

### Impact Explanation

The multi-signature approval model is the primary trust boundary for organization-mode transactions. Approvers consent to sign under a specific threshold (e.g., 3-of-5). If the creator lowers the threshold to 2-of-5 after two approvals are already recorded, the scheduler's next `processTransactionStatus` pass will find the approval condition satisfied and advance the transaction to `WAITING_FOR_EXECUTION` or trigger submission — with fewer approvals than the approvers agreed to. This constitutes an unauthorized state change: the creator unilaterally overrides the multi-party governance rule, defeating the purpose of the approval workflow.

### Likelihood Explanation

The creator is a normal authenticated user with no elevated privileges. The attack requires only:
1. Create a transaction with a threshold > 1.
2. Wait for at least one approver to approve.
3. Call `PATCH /transactions/:id/approvers/:approverId` with `{ "threshold": 1 }`.

No leaked credentials, no admin access, and no race condition are required. The endpoint is documented and reachable in production.

### Recommendation

Add a transaction-status guard at the top of `updateTransactionApprover` (and `removeTransactionApprover`) that rejects modifications once the transaction has left the `NEW` status:

```typescript
const tx = await this.getCreatorsTransaction(transactionId, user, transactionalEntityManager);
if (tx.status !== TransactionStatus.NEW) {
  throw new BadRequestException('Approver structure cannot be modified after the transaction has been submitted');
}
```

This mirrors the pattern already used in `importSignatures` and `approveTransaction`, which both gate on `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION`. [7](#0-6) 

### Proof of Concept

1. Creator (Alice) creates a transaction with a 3-of-5 approver threshold.
2. Approvers Bob and Carol submit approvals via `POST /transactions/1/approvers/approve`.
3. Alice calls:
   ```
   PATCH /transactions/1/approvers/<tree-approver-id>
   Authorization: Bearer <alice-token>
   Content-Type: application/json

   { "threshold": 2 }
   ```
4. `updateTransactionApprover` accepts the request (no status check), writes `threshold = 2` to the `transaction_approver` row.
5. The chain-service scheduler's next `processTransactionStatus` run evaluates the approval tree, finds 2-of-2 approvals satisfy the new threshold, and advances the transaction to `WAITING_FOR_EXECUTION` — bypassing the original 3-of-5 requirement.
