# Audit Report

## Title
Unguarded Subtraction in Approver Tree Threshold Update Enables Permanent Approval Freeze

## Summary
In `updateTransactionApprover`, when a child approver is detached from its parent (`dto.listId = null`), the code reads the parent's `approvers` relation as an in-memory snapshot and computes `parent.approvers.length - 1` without row-level locking. Under the default PostgreSQL `READ COMMITTED` isolation level, two concurrent requests from the same transaction creator can both read the same stale snapshot, each compute an incorrect child count, and both skip the soft-delete and threshold-adjustment branches — leaving the parent node with 0 children but a non-zero threshold. The approval requirement becomes permanently unsatisfiable.

## Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, inside `updateTransactionApprover`, the parent node is fetched with its `approvers` relation as a plain ORM snapshot: [1](#0-0) 

The child count is then derived from that snapshot: [2](#0-1) 

The enclosing `dataSource.transaction()` uses the default PostgreSQL `READ COMMITTED` isolation level and no `SELECT FOR UPDATE` / pessimistic lock: [3](#0-2) 

Because the parent row is never locked before reading, two concurrent transactions can both observe the same `parent.approvers` snapshot before either commits. The child-detach `UPDATE` at line 409 is committed independently per transaction, so neither transaction sees the other's write when it evaluates the branching logic.

A secondary issue: there is no lower-bound guard on `parent.approvers.length`. If it is already 0 (reachable via the race itself or another bug), `newParentApproversLength` becomes `-1`, which satisfies `< parent.threshold` and writes `-1` as the stored threshold — an invalid sentinel value. [4](#0-3) 

## Impact Explanation
A parent approver node left with `threshold = 1` and zero children can never satisfy its approval requirement. The transaction is permanently blocked from reaching `WAITING_FOR_EXECUTION` status. Because the parent node is not soft-deleted, any attempt to recreate the approver tree is complicated by the orphaned parent record still present in the database. The transaction must be manually cancelled, discarding all work already performed by signers.

## Likelihood Explanation
The attacker is the transaction creator — a normal, unprivileged user with full legitimate access to `PATCH /transactions/:id/approvers/:approverId`. [5](#0-4) 

Triggering the race requires only two concurrent HTTP requests targeting two different children of the same parent node. No special timing precision is needed: the race window spans the entire duration of both database transactions, which execute the same read-then-write pattern under `READ COMMITTED`. This is trivially achievable via a script or browser DevTools.

## Recommendation

1. **Pessimistic row-level lock on the parent.** Replace the plain `findOne` with a `SELECT ... FOR UPDATE` query so that the second concurrent transaction blocks until the first commits:
   ```typescript
   const parent = await transactionalEntityManager
     .createQueryBuilder(TransactionApprover, 'p')
     .setLock('pessimistic_write')
     .leftJoinAndSelect('p.approvers', 'approvers')
     .where('p.id = :id', { id: approver.listId })
     .getOne();
   ```

2. **Re-query the live child count after acquiring the lock** rather than relying on the pre-loaded `approvers` relation snapshot. Use a `COUNT(*)` query inside the same transaction after the lock is held.

3. **Add a lower-bound guard** before the subtraction:
   ```typescript
   const newParentApproversLength = Math.max(0, parent.approvers.length - 1);
   ```

4. **Consider `SERIALIZABLE` isolation** for the entire `updateTransactionApprover` transaction if multiple concurrent structural mutations to the same approver tree are possible.

## Proof of Concept

**Setup:** Create a transaction with an approver tree: one parent node (`threshold = 1`) with two leaf children A and B.

**Attack:**
```bash
# Send both requests simultaneously (e.g., via Promise.all or parallel curl)
curl -X PATCH /transactions/1/approvers/<A_id> \
  -H "Authorization: Bearer <creator_token>" \
  -d '{"listId": null}' &

curl -X PATCH /transactions/1/approvers/<B_id> \
  -H "Authorization: Bearer <creator_token>" \
  -d '{"listId": null}' &

wait
```

**Expected (correct) behaviour:** The first request detaches A and reduces the parent's threshold to 0 children → soft-deletes the parent. The second request finds the parent already deleted and handles accordingly.

**Actual (buggy) behaviour under `READ COMMITTED`:**
1. Request-1 reads `parent.approvers = [A, B]`, computes `newParentApproversLength = 1`.
2. Request-2 reads `parent.approvers = [A, B]`, computes `newParentApproversLength = 1`.
3. Both evaluate `1 === 0` → false; `1 < 1` → false. Neither soft-deletes nor adjusts the threshold.
4. Both commit their child `UPDATE`. Parent now has 0 children, `threshold = 1`.
5. The transaction's approval requirement is permanently unsatisfiable. [6](#0-5)
