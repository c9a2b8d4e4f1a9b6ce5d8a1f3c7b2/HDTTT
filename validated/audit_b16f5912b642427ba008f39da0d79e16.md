### Title
Missing `@Min(1)` Constraint on `threshold` in `CreateTransactionApproverDto` Allows Zero-Threshold Approver Groups, Potentially Bypassing Multi-Signature Approval

### Summary
`CreateTransactionApproverDto` accepts `threshold: 0` because it lacks a `@Min(1)` constraint, while the sibling `UpdateTransactionApproverDto` correctly enforces `@Min(1)`. Any authenticated user can create a transaction approver group with a threshold of zero, which — depending on how the approval-count check is evaluated — can cause the approval gate to be trivially satisfied (0-of-N required), effectively bypassing the multi-signature workflow that is the core security control of the organizational mode.

### Finding Description

**Root cause — missing lower-bound constraint on `threshold` at creation time**

`CreateTransactionApproverDto` declares `threshold` as:

```typescript
@IsNumber()
@IsOptional()
threshold?: number;
``` [1](#0-0) 

There is no `@Min(1)` guard. By contrast, `UpdateTransactionApproverDto` — which modifies the same field on the same entity — explicitly enforces the lower bound:

```typescript
@IsNumber()
@Min(1)
@IsOptional()
threshold?: number;
``` [2](#0-1) 

The developer's intent is unambiguous: `@Min(1)` was deliberately added to the update path but was omitted from the create path.

**Exploit flow**

1. Attacker authenticates as a normal user (no privilege required).
2. Attacker creates a transaction and then POSTs to the approvers endpoint with a body such as:
   ```json
   { "approversArray": [{ "threshold": 0, "approvers": [{"userId": 5}, {"userId": 6}] }] }
   ```
3. The NestJS validation pipeline accepts the payload because `CreateTransactionApproverDto` has no `@Min(1)` on `threshold`.
4. A threshold-0 approver group is persisted to the database.
5. When the approval-count check runs (e.g., `approvedCount >= threshold`), the condition `N >= 0` is always true, so the transaction is treated as fully approved with zero actual approvals.

**Entry point**

The `CreateTransactionApproversArrayDto` wrapper is the validated DTO accepted by the approvers controller endpoint: [3](#0-2) 

### Impact Explanation

The multi-signature approval workflow is the primary organizational security control: transactions require a configurable number of approvers before they can be executed. A threshold of 0 collapses this gate entirely — the transaction is considered approved the moment it is created, regardless of how many approvers are listed. This allows an attacker (or a malicious insider with normal user access) to create transactions that execute without any peer review or approval, defeating the purpose of the organizational mode.

**Severity: High** — unauthorized state change / privilege escalation via bypassed approval gate.

### Likelihood Explanation

Any authenticated user who can create transactions and set approvers can trigger this. No admin keys, no leaked credentials, and no special network access are required. The API endpoint is reachable from the standard frontend or directly via HTTP. The only precondition is a valid session token, which any registered user possesses.

**Likelihood: Medium** — requires a deliberate crafted request (not a normal UI flow), but trivially achievable by any authenticated user or malicious API client.

### Recommendation

Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
@IsNumber()
@Min(1)          // ← add this
@IsOptional()
threshold?: number;
```

Additionally, add a server-side guard in the approver-creation service that rejects any persisted threshold value less than 1, as defense-in-depth against future DTO changes.

### Proof of Concept

**Request (no special privileges — only a valid session cookie/token):**
```http
POST /transactions/{id}/approvers HTTP/1.1
Authorization: Bearer <normal-user-token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 0,
      "approvers": [
        { "userId": 5 },
        { "userId": 6 }
      ]
    }
  ]
}
```

**Expected (correct) behavior:** HTTP 400 — `threshold` must be ≥ 1.

**Actual behavior:** HTTP 201 — approver group with `threshold: 0` is persisted. Subsequent approval-count evaluation (`approvedCount >= 0`) is always true, so the transaction proceeds to execution without any approver signing off.
