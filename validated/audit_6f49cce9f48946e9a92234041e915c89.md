Audit Report

## Title
Mutable Approver Structure Allows Transaction Creator to Nullify Rejections After Approvals Are Submitted

## Summary
`updateTransactionApprover()` and `removeTransactionApprover()` in `approvers.service.ts` allow the transaction creator to modify the approver tree — including swapping approver identities, lowering thresholds, and removing approvers — at any time, with no check on whether approvals or rejections have already been submitted. A malicious creator can exploit this to erase rejections and force a transaction through the approval workflow.

## Finding Description

**Root cause**: The sole authorization guard used before any structural mutation is `getCreatorsTransaction()`, which verifies only that the caller is the transaction creator. It performs no check on the transaction's current status or whether any approvers have already voted.

`getCreatorsTransaction()` fetches the transaction and checks `creatorKey.userId`, but returns the transaction unconditionally if the caller matches — no status gate: [1](#0-0) 

In `updateTransactionApprover()`, `getCreatorsTransaction()` is the only guard before mutations proceed: [2](#0-1) 

When a `userId` swap is performed, the existing approval state is **explicitly cleared** — `signature`, `userKeyId`, and `approved` are all set to `undefined`: [3](#0-2) 

The threshold can be lowered at any time with no status check: [4](#0-3) 

`removeTransactionApprover()` in the service has no status check whatsoever: [5](#0-4) 

In the controller, `getCreatorsTransaction()` is the only guard before the delete is executed: [6](#0-5) 

**Exploit flow**:

1. Creator creates a transaction and sets up approvers: User A, User B, User C with threshold = 2.
2. User A approves (`approved = true`). User B rejects (`approved = false`).
3. Creator calls `PATCH /transactions/:txId/approvers/:approverId` with `{ userId: userD }` to replace User B. The update clears User B's rejection — the record now has `signature = undefined`, `approved = undefined`.
4. User D approves.
5. Threshold of 2 is now met with User A and User D. User B's rejection is silently erased.

Alternatively, the creator calls `DELETE /transactions/:txId/approvers/:approverId` to remove User B entirely, reducing the approver pool and potentially meeting the threshold with only the remaining approvals.

## Impact Explanation

The approval system is the primary governance mechanism preventing a transaction creator from unilaterally executing transactions that require organizational consensus. By nullifying rejections through approver swaps or removals, the creator can force any transaction through regardless of how many approvers reject it.

Concrete impact:
- A creator can execute a transaction that was explicitly rejected by designated approvers.
- The rejection history is silently erased — no audit trail of the removed approver's decision remains.
- Organizational multi-party approval policies are rendered unenforceable.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges. The attack path requires only standard API calls (`PATCH` or `DELETE` on approver endpoints) that the creator is already authorized to make. No leaked credentials, no admin access, and no race conditions are required. The creator can observe rejections in real time via the notification system and respond immediately.

## Recommendation

Add a transaction status check at the start of both `updateTransactionApprover()` and `removeTransactionApprover()` (and in the controller's delete handler). Structural mutations to the approver tree should be rejected if the transaction is in any status other than `NEW` (i.e., before any approver has had the opportunity to vote). Specifically:

- In `getCreatorsTransaction()` or at the call sites, assert that `transaction.status === TransactionStatus.NEW` before allowing any approver tree modification.
- Alternatively, check whether any approver in the tree already has a non-null `approved` or `signature` field, and reject the mutation if so.

## Proof of Concept

```

POST /transactions/1/approvers
{ "approversArray": [{ "userId": A }, { "userId": B }, { "userId": C }] }

# Step 2: User B rejects
POST /transactions/1/approvers/approve
{ "userKeyId": B_key, "signature": "...", "approved": false }

# Step 3: Creator swaps User B for User D — clears B's rejection
PATCH /transactions/1/approvers/<B_approver_id>
{ "userId": D }
# Result: approved=undefined, signature=undefined for that record

# Step 4: User D approves
POST /transactions/1/approvers/approve
{ "userKeyId": D_key, "signature": "...", "approved": true }

# Step 5: Threshold met (A + D), transaction proceeds despite B's rejection
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
