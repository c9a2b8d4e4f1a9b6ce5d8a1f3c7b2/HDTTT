Audit Report

## Title
Unbounded In-Memory Iteration in `getTransactionsToSign` Causes Server-Side DoS

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no `LIMIT` clause, then performs a sequential async cryptographic/DB operation per transaction (`userKeysToSign`), and only applies pagination in memory after the full scan completes. Any authenticated user calling `GET /transactions/sign` triggers this full-table scan, causing CPU, memory, and DB resource consumption that scales linearly with the number of active transactions on the platform.

## Finding Description

**1. Missing DB-level pagination — confirmed:**

The `repo.find()` call at line 295 has no `take` parameter and no user-scoping filter. The `whereForUser` condition only excludes terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`) — it does not filter by the requesting user. This causes a full scan of all active transactions across the entire platform on every request. [1](#0-0) 

**2. Unbounded per-transaction async work — confirmed:**

The code then iterates over every returned transaction and calls `userKeysToSign` sequentially for each one: [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign` → `keysRequiredToSign`, which calls `transactionSignatureService.computeSignatureKey` (deserializes transaction bytes, performs mirror node account lookups via `AccountCacheService`) and then `entityManager.find(UserKey, ...)` — one DB query per transaction. [3](#0-2) [4](#0-3) [5](#0-4) 

**3. In-memory pagination applied after full scan — confirmed:**

The `limit`/`offset` from the `Pagination` parameter are never passed to the DB query; they are only used to slice the already-fully-loaded result array: [6](#0-5) 

**4. Developer comment acknowledging missing API-side limit in `importSignatures` — confirmed:** [7](#0-6) 

**5. Endpoint is still live despite front-end deprecation:**

The controller comment marks the endpoint as `NO LONGER USED BY FRONT-END`, but the route `GET /transactions/sign` remains fully accessible to any authenticated user: [8](#0-7) 

## Impact Explanation

- Every call to `GET /transactions/sign` by any authenticated user triggers a full table scan of all non-terminal transactions across the entire platform, followed by N sequential async operations (mirror node HTTP call + DB query per transaction).
- Memory usage in the Node.js process scales linearly with the number of active transactions loaded.
- CPU and DB connection time scale linearly with N per request.
- With a sufficiently large transaction backlog, the endpoint times out, the Node.js event loop stalls, and other API endpoints become unresponsive — full service degradation for all users.
- The `totalItems` returned reflects only the user's matching subset after the full scan, not the true total, compounding the logic error.

## Likelihood Explanation

- **No privilege required**: any authenticated, verified user can call this endpoint.
- **Natural growth**: the platform accumulates active transactions in normal operation; the vulnerability worsens over time without any attacker action.
- **Attacker acceleration**: a malicious authenticated user can create transactions via `POST /transactions` or `POST /transactions/group` to inflate the active transaction count. The `CreateTransactionGroupDto.groupItems` array has no `@ArrayMaxSize` constraint, enabling bulk creation in a single request. [9](#0-8) 

## Recommendation

1. **Push pagination to the DB layer**: Add `take: limit` and `skip: offset` to the `repo.find()` call in `getTransactionsToSign`, and add a user-scoping join/subquery so only transactions relevant to the requesting user are fetched.
2. **Scope the query to the requesting user**: The current `whereForUser` filter does not restrict results to the requesting user's transactions. A join against `transaction_signer`, `transaction_observer`, or `transaction_approver` scoped to `user.id` should be added, similar to how `getTransactionsToApprove` uses a correlated subquery.
3. **Add an `@ArrayMaxSize` constraint** on `CreateTransactionGroupDto.groupItems` to prevent bulk transaction creation abuse.
4. **Consider removing or rate-limiting the deprecated endpoint** since it is no longer used by the front-end.

## Proof of Concept

```
# Authenticated as any verified user:
GET /transactions/sign?page=1&size=10

# With N active transactions in the DB, this triggers:
# 1. SELECT * FROM transaction WHERE status NOT IN (...) -- no LIMIT
# 2. For each of N transactions:
#    a. computeSignatureKey(tx) -- deserialize bytes + mirror node HTTP call
#    b. entityManager.find(UserKey, ...) -- DB query
# 3. result.slice(0, 10) -- pagination applied in memory only
#
# Repeat concurrently from multiple sessions to exhaust the event loop.
```

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
```

```typescript
  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
