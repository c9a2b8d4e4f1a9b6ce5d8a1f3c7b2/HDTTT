Audit Report

## Title
AES-256-ECB Without Authentication in Legacy Mnemonic Decryption Allows Undetected Ciphertext Tampering

## Summary
`decryptLegacyMnemonic` in `front-end/src/main/services/localUser/dataMigration.ts` decrypts the legacy `recovery.aes` file using AES-256-ECB — a mode with no IV, no authentication tag, and no MAC — combined with a hardcoded PBKDF2 salt. An attacker with user-level filesystem access can silently tamper with the ciphertext before migration. Decryption will succeed without any integrity error, and the corrupted or attacker-influenced mnemonic will be imported into the wallet. The primary `encrypt`/`decrypt` path in `front-end/src/main/utils/crypto.ts` already uses AES-256-GCM, but this legacy migration path was never upgraded.

## Finding Description
In `front-end/src/main/services/localUser/dataMigration.ts`, `decryptLegacyMnemonic` reads the legacy encrypted recovery file and decrypts it using AES-256-ECB:

```ts
// line 108
const salt = Buffer.from([1, 2, 3, 4, 5, 6, 7, 8]); // hardcoded static salt
const key = crypto.pbkdf2Sync(password, salt, iterationCount, keyLength, 'sha256');
// ...
const decipher = crypto.createDecipheriv('aes-256-ecb', key, null);
const decrypted = Buffer.concat([decipher.update(data), decipher.final()]);
return decrypted.toString('utf-8');
``` [6](#0-5) 

ECB mode encrypts each 16-byte block independently with no chaining, no IV, and no authentication tag. This means:

- **No integrity check**: Any modification to the ciphertext file is silently accepted as long as PKCS7 padding in the final block remains valid. An attacker who only rearranges or substitutes non-final blocks will not trigger a padding error.
- **Block-level malleability**: An attacker can rearrange, duplicate, or substitute 16-byte ciphertext blocks. Each block decrypts independently, so the output is a deterministic transformation of the attacker's chosen block arrangement.
- **Hardcoded salt compounds the issue**: The PBKDF2 salt `[1, 2, 3, 4, 5, 6, 7, 8]` is identical for every user, enabling precomputed rainbow tables against common passwords and lowering the bar for full plaintext-recovery attacks.

This function is called from `decryptMigrationMnemonic` when `isLegacy = true`: [3](#0-2) 

The result is split by spaces and returned as the mnemonic word list, which is then used to derive Hedera private keys.

By contrast, the current production encryption path correctly uses AES-256-GCM with a random salt and authentication tag: [4](#0-3) 

## Impact Explanation
If an attacker tampers with `recovery.aes` before the user runs the migration wizard:

- **Corrupted mnemonic import**: Rearranging ECB blocks (excluding the final padding block) produces a deterministically corrupted plaintext. The user imports a mnemonic that does not correspond to their actual keys, permanently losing access to their Hedera wallet funds.
- **Partial block substitution**: If the attacker knows the ECB-encrypted form of specific BIP39 words (obtainable by encrypting known words with the same static salt and a guessed or brute-forced password), they can substitute individual 16-byte blocks to inject attacker-controlled words into the mnemonic, potentially redirecting derived keys to an attacker-controlled wallet.
- **Silent failure**: Because there is no authentication tag or MAC, neither the application nor the user receives any error. The migration completes "successfully" with a corrupted or attacker-influenced seed.

## Likelihood Explanation
The `recovery.aes` file lives on the local filesystem at a predictable path (`~/Documents/TransactionTools/Files/.System/recovery.aes`): [5](#0-4) 

For a desktop Electron wallet app, malware with user-level filesystem access is a well-established and realistic threat model. No privileged OS access is required — only read/write access to the user's Documents folder, which any process running as the user possesses. The migration wizard is a one-time flow, but users who have not yet migrated from the old tool are exposed for the entire window between installation and migration.

## Recommendation
1. **Replace AES-256-ECB with AES-256-GCM** in `decryptLegacyMnemonic`. Since this is a *decryption-only* legacy path (the old tool produced the ciphertext), the fix requires re-encrypting the file with GCM after successful decryption, or at minimum verifying a HMAC over the ciphertext before decryption.
2. **Replace the hardcoded salt** with a randomly generated salt stored alongside the ciphertext (prepended to the file), matching the pattern already used in `decryptMnemonic`.
3. After successful legacy decryption and mnemonic import, immediately re-encrypt the mnemonic using the current `decryptMnemonic` format (AES-256-GCM with Argon2id key derivation and random salt), so the `recovery.aes` file is replaced with an authenticated ciphertext.

## Proof of Concept
```python
# Attacker script: rearrange ECB blocks in recovery.aes before migration
import os

RECOVERY_PATH = os.path.expanduser(
    "~/Documents/TransactionTools/Files/.System/recovery.aes"
)
BLOCK_SIZE = 16

with open(RECOVERY_PATH, "rb") as f:
    data = f.read()

# Split into 16-byte ECB blocks
blocks = [data[i:i+BLOCK_SIZE] for i in range(0, len(data), BLOCK_SIZE)]

# Rearrange: swap first two word-blocks (each BIP39 word ~11 chars, fits in 1 block)
# Keep the final block (padding) intact to avoid padding errors
blocks[0], blocks[1] = blocks[1], blocks[0]

tampered = b"".join(blocks)
with open(RECOVERY_PATH, "wb") as f:
    f.write(tampered)

# Result: decryptLegacyMnemonic() returns no error, but the first two
# mnemonic words are swapped → wrong seed → wrong private keys imported.
```

When the user subsequently runs the migration wizard, `decryptLegacyMnemonic` decrypts the tampered file without error, `decryptMigrationMnemonic` splits the corrupted plaintext by spaces and returns it as the mnemonic word list, and the wallet is initialized with an incorrect seed — with no indication to the user that anything went wrong.

export function encrypt(data: string, password: string) {
  const iv = crypto.randomBytes(16);
  const salt = crypto.randomBytes(64);

  const key = deriveKey(password, salt);

  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([cipher.update(data, 'utf8'), cipher.final()]);

  const tag = cipher.getAuthTag();

  return Buffer.concat([salt, iv, tag, encrypted]).toString('base64');
}
