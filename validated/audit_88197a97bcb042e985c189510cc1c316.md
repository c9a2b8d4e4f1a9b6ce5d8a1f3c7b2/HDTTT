Audit Report

## Title
Missing Access Control on Transaction Comments Endpoints Allows Any Authenticated User to Post and Read Comments on Any Transaction

## Summary
`CommentsController` applies only authentication guards at the class level and performs no per-transaction ownership or membership check. Any authenticated, verified organization user can `POST` a comment to any transaction they have no relationship to, and `GET` all comments from any transaction. The developers explicitly flagged this gap with a `//TODO` comment directly in the controller.

## Finding Description

**Root cause — `CommentsController`:**

The controller at `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level. These guards only verify that the caller is a valid, non-blacklisted, verified organization user. They do not verify that the user is the creator, signer, observer, or approver of the target transaction. [1](#0-0) 

**`createComment` — unauthorized state mutation:**

The developer-acknowledged `//TODO need some sort of guard or check to ensure user can comment here` comment at line 22 confirms no access control exists. The method receives `user` and `transactionId` and passes them directly to `CommentsService.createComment` without any relationship check. [2](#0-1) 

`CommentsService.createComment` performs no ownership or membership verification — it creates and saves the comment unconditionally. [3](#0-2) 

**`getComments` and `getCommentById` — unauthorized information disclosure:**

Neither endpoint receives a `user` parameter. There is no access check at the controller or service layer. [4](#0-3) [5](#0-4) 

**Contrast with other controllers that do enforce per-transaction access:**

`ObserversService.getTransactionObserversByTransactionId` explicitly throws `UnauthorizedException` if the requesting user is not the creator, a signer, an observer, or an approver of the transaction. [6](#0-5) 

`ApproversService.getVerifiedApproversByTransactionId` applies the identical check. [7](#0-6) 

The comments endpoints have no equivalent check.

## Impact Explanation

1. **Unauthorized state mutation**: Any authenticated user can `POST /transactions/:transactionId/comments` with an arbitrary `transactionId` and inject comments into transaction workflows they have no legitimate role in — enabling misleading, disruptive, or harassing content injection.
2. **Cross-tenant information disclosure**: Transaction comments may contain sensitive organizational decisions, key coordination details, or approval rationale. Any authenticated user can enumerate all comments on any transaction by iterating `transactionId` values, bypassing the access control that protects all other transaction sub-resources (observers, approvers, signers, transaction bytes).

## Likelihood Explanation

The attacker precondition is only a valid, verified organization account — no admin privileges, no leaked secrets. The exploit is a direct HTTP `POST` or `GET` to a documented REST endpoint. The developer `//TODO` comment at line 22 confirms the gap is known and unmitigated. Any malicious insider or compromised low-privilege account can exploit this immediately with no special tooling.

## Recommendation

Apply the same relationship check used by `ObserversService` and `ApproversService` to the comments endpoints. Before allowing `createComment`, `getComments`, or `getCommentById`, verify that the requesting user is the transaction creator, a signer, an observer, or an approver of the target transaction. This check should be implemented in `CommentsService` (mirroring `getVerifiedApproversByTransactionId`) and invoked from the controller, or encapsulated in a dedicated NestJS guard. Remove the `//TODO` comment once the check is in place.

## Proof of Concept

```
# Step 1: Authenticate as any valid organization user (User A, no relationship to transaction #42)
POST /auth/login
{ "email": "userA@org.com", "password": "..." }
→ returns JWT token

# Step 2: Post a comment on a transaction User A has no role in
POST /transactions/42/comments
Authorization: Bearer <userA_jwt>
{ "text": "Injected comment on transaction 42" }
→ HTTP 201 Created — comment is persisted with no access check

# Step 3: Read all comments on any transaction
GET /transactions/42/comments
Authorization: Bearer <userA_jwt>
→ HTTP 200 OK — returns all comments with no access check

# Step 4: Read a specific comment by id (no transactionId scoping needed)
GET /transactions/42/comments/7
Authorization: Bearer <userA_jwt>
→ HTTP 200 OK — returns comment regardless of user relationship
```

The `//TODO` at line 22 of `comments.controller.ts` and the absence of any user parameter in `getComments`/`getCommentById` confirm there is no server-side gate to bypass. [8](#0-7)
