## Title
Transaction Creator Can Add Themselves as Approver and Self-Approve, Bypassing Multi-Party Governance

## Summary
`createTransactionApprovers` in `approvers.service.ts` restricts approver assignment to the transaction creator but performs no check preventing the creator from designating their own `userId` as an approver. `approveTransaction` then verifies only that the caller is listed as an approver and has not already approved — it never checks whether the caller is also the transaction creator. Together, these two missing guards allow a creator to unilaterally satisfy the approval requirement for their own transaction.

## Finding Description

**Root cause — `createTransactionApprovers`**

The function opens with `getCreatorsTransaction`, which confirms the caller is the creator: [1](#0-0) 

Inside the loop, the only validation performed on `dtoApprover.userId` is an existence check against the `User` table: [2](#0-1) 

There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. The creator's own `user.id` passes this check without issue.

**Root cause — `approveTransaction`**

The function filters approvers by `user.id` and checks only that the list is non-empty and that the user has not already signed: [3](#0-2) 

There is no check comparing `user.id` against `transaction.creatorKey.userId`. Because the creator was able to insert themselves as an approver in the previous step, this check passes and the approval is recorded.

**End-to-end exploit path**:
1. Attacker creates a transaction via `POST /transactions`.
2. Attacker calls `POST /transactions/:id/approvers` with `approversArray: [{ userId: <own user id> }]`. `getCreatorsTransaction` passes (attacker is the creator). No exclusion check fires.
3. Attacker calls `POST /transactions/:id/approvers/approve` with a valid signature. `approveTransaction` finds the attacker in the approver list, verifies the cryptographic signature, and records approval.
4. If the attacker is the sole required approver or satisfies the threshold, the transaction advances — without any other organization member having reviewed or approved it.

## Impact Explanation

The approval system is the organizational governance control for sensitive Hedera transactions. Self-approval collapses a multi-party control into a single-party action. A single malicious or compromised authenticated user can execute arbitrary Hedera transactions — including `AccountUpdateTransaction` (key rotation), `NodeUpdateTransaction`, or HBAR/token transfers — without peer review, defeating the entire purpose of the approval workflow. [4](#0-3) 

## Likelihood Explanation

Any authenticated, verified organization member who can create transactions can trigger this. No privileged access, leaked credentials, or external dependencies are required — only a valid JWT and a registered key. The steps follow entirely normal API flows and require no special tooling. The attack is fully self-contained. [5](#0-4) 

## Recommendation

**In `createTransactionApprovers`**: After confirming the user exists, add an explicit exclusion check:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
if (dtoApprover.userId === user.id)
  throw new Error('Transaction creator cannot be designated as an approver');
``` [2](#0-1) 

**In `approveTransaction`**: After fetching the transaction, add a creator-identity check:

```typescript
if (transaction.creatorKey?.userId === user.id)
  throw new UnauthorizedException('Transaction creator cannot approve their own transaction');
``` [6](#0-5) 

Both fixes are required; patching only one still leaves the other path exploitable.
