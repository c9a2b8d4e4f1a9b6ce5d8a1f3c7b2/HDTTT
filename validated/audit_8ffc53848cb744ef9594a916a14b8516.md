# Audit Report

## Title
Missing Transaction Status Check in `createTransactionApprovers` and `updateTransactionApprover` Allows Approver Tree Modification on Terminal-State Transactions

## Summary
`createTransactionApprovers` and `updateTransactionApprover` in `approvers.service.ts` delegate their authorization entirely to `getCreatorsTransaction`, which only verifies transaction existence and creator ownership. No status guard is present. A transaction creator can therefore add or restructure approvers on transactions in any state — including `EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, and `ARCHIVED` — corrupting the audit trail and, for `createTransactionApprovers`, triggering a NATS-based status recalculation on the modified transaction.

## Finding Description

**`getCreatorsTransaction` performs no status check.**
The function fetches the transaction, asserts it exists, and asserts `creatorKey.userId === user.id`. It returns the transaction unconditionally regardless of its `status` field. [1](#0-0) 

**`createTransactionApprovers` relies solely on this check.**
The only authorization gate is the `getCreatorsTransaction` call at line 239. After that, the function proceeds to insert `TransactionApprover` rows for any transaction status, then unconditionally calls `emitTransactionStatusUpdate`, publishing a NATS event that triggers status recalculation on the now-modified transaction. [2](#0-1) [3](#0-2) 

**`updateTransactionApprover` has the same gap.**
The update path calls `getCreatorsTransaction` at line 394 with no status validation, allowing structural changes (re-parenting nodes, changing thresholds, swapping users) on transactions in any state. [4](#0-3) 

**Contrast with `approveTransaction`, which correctly guards status.**
The approval action explicitly rejects anything outside `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` before proceeding. [5](#0-4) 

**Pre-population of approval state on new approver nodes.**
When a new approver node is created for a user who already has an existing approver record on the same transaction, the new node inherits the existing `signature`, `userKeyId`, and `approved` values. On a `WAITING_FOR_EXECUTION` transaction this means a creator can inject a structurally new approver node that arrives pre-approved, silently satisfying threshold requirements without the designated user performing any action. [6](#0-5) 

## Impact Explanation

1. **Audit-trail corruption on terminal transactions.** Approver records can be inserted or restructured after a transaction has reached `EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, or `ARCHIVED` status. The historical record of who approved what becomes unreliable.

2. **Forced status regression on `WAITING_FOR_EXECUTION` transactions.** Adding an unapproved approver node to a transaction that has already gathered all required approvals triggers `emitTransactionStatusUpdate` (line 358). The chain service recalculates status and can move the transaction back to `WAITING_FOR_SIGNATURES`, indefinitely stalling execution until the newly injected approver acts. [7](#0-6) 

3. **Phantom pre-approved nodes.** Because the code copies existing approval state into newly created approver nodes (lines 325–329), a creator can manufacture approver nodes that appear already-approved without the target user performing any signing action, silently satisfying threshold counts. [8](#0-7) 

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The endpoint `POST /transactions/:transactionId/approvers` is reachable by any verified user for their own transactions. No special tooling is required; a standard HTTP client suffices. The condition (a transaction in `WAITING_FOR_EXECUTION` or a terminal state) arises in normal operation.

## Recommendation

Add a status guard at the top of both `createTransactionApprovers` and `updateTransactionApprover`, immediately after the `getCreatorsTransaction` call. The guard should reject any transaction whose status is not in the set of mutable states (e.g., `NEW`, `WAITING_FOR_SIGNATURES`). Example:

```typescript
const transaction = await this.getCreatorsTransaction(transactionId, user);

const MUTABLE_STATUSES = [TransactionStatus.NEW, TransactionStatus.WAITING_FOR_SIGNATURES];
if (!MUTABLE_STATUSES.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP); // Operation not permitted in this state
}
```

This mirrors the pattern already used correctly in `approveTransaction`. [5](#0-4) 

## Proof of Concept

**Scenario: Injecting a pre-approved phantom node on a `WAITING_FOR_EXECUTION` transaction**

Prerequisites:
- User A is the creator of transaction T (status: `WAITING_FOR_EXECUTION`).
- User A (userId=1) already has an existing approver record on transaction T with `approved=true`, `signature=<sig>`, `userKeyId=<keyId>`.

**Steps:**

1. User A sends:
```http
POST /transactions/T/approvers
Authorization: Bearer <user_A_token>
Content-Type: application/json

{
  "approversArray": [
    { "userId": 1 }
  ]
}
```

2. `createTransactionApprovers` calls `getCreatorsTransaction(T, userA)` — passes (no status check).

3. The inner `createApprover` function queries existing approver records for userId=1 on transaction T (lines 319–323), finds the existing approved record, and copies `signature`, `userKeyId`, `approved=true` into the new node (lines 326–328).

4. A new `TransactionApprover` row is inserted with `approved=true` and a valid signature — without User A performing any new signing action.

5. `emitTransactionStatusUpdate` fires (line 358), triggering a status recalculation that now counts this phantom node toward threshold satisfaction.

**Scenario: Corrupting a terminal transaction's audit trail**

1. Transaction T has status `EXECUTED`.
2. User A (creator) sends `POST /transactions/T/approvers` with any valid approver payload.
3. No status check exists; new `TransactionApprover` rows are inserted into the database for an already-executed transaction, corrupting the historical approval record. [2](#0-1) [1](#0-0)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
