Audit Report

## Title
Any Authenticated User Can Post and Read Comments on Transactions They Have No Access To

## Summary
`CommentsController` applies only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) at the controller level. None of the three routes — `POST`, `GET`, and `GET /:id` — perform the transaction membership check (creator / observer / signer / approver) that every other transaction sub-resource enforces. The developer acknowledged this gap with an unresolved TODO comment directly above the `POST` handler. Any verified organization user can write comments to, and read comments from, any transaction by ID.

## Finding Description

**The missing check in `CommentsController`**

The controller at `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only three guards at the class level: [1](#0-0) 

The `POST` handler carries an explicit, unresolved TODO acknowledging the missing access control: [2](#0-1) 

The `GET` endpoints also pass no `user` argument and perform no membership check: [3](#0-2) 

**`CommentsService` performs no membership check**

`createComment` directly creates and persists the comment with no verification that the requesting user has any relationship to the target transaction: [4](#0-3) 

`getTransactionComments` and `getTransactionCommentById` similarly perform no access check: [5](#0-4) 

**Invariant enforced everywhere else**

`TransactionsService.verifyAccess` checks creator, observer, signer, and approver membership before granting access: [6](#0-5) 

`TransactionsService.getTransactionWithVerifiedAccess` calls `verifyAccess` and throws `UnauthorizedException` on failure: [7](#0-6) 

`ObserversService.getTransactionObserversByTransactionId` throws `UnauthorizedException` for non-participants: [8](#0-7) 

`ApproversService.getVerifiedApproversByTransactionId` applies the same pattern: [9](#0-8) 

The comments endpoints are the only transaction sub-resource that omits this check entirely.

## Impact Explanation

- **Unauthorized state mutation**: Any verified user can inject comments into any transaction they are not a party to, corrupting the audit trail and collaboration history of private multi-party transactions.
- **Information disclosure**: `GET /transactions/:transactionId/comments` returns all comments for a transaction with no membership check, leaking internal deliberation content (e.g., signing decisions, approver notes) to unauthorized users.
- **Transaction existence oracle**: A `201` response to `POST` confirms the transaction ID exists, enabling enumeration of all live transaction IDs by any authenticated user.

## Likelihood Explanation

- **Precondition**: Only a valid, verified organization account is required — no elevated privilege.
- **Enumeration**: Transaction IDs are sequential integers starting from 1, making brute-force trivial.
- **No rate limiting**: No rate-limiting specific to this endpoint is present in the controller or service.
- **Known gap**: The TODO comment at line 22 confirms the gap was known at the time of shipping and was never resolved. [10](#0-9) 

## Recommendation

Apply the same membership check used by all other transaction sub-resources. The fix should be applied in `CommentsService` for all three operations:

1. In `createComment`, `getTransactionComments`, and `getTransactionCommentById`, call the equivalent of `TransactionsService.verifyAccess` (or inline the same creator/observer/signer/approver check) before proceeding.
2. Pass the authenticated `user` into the `GET` handlers so the service can perform the check.
3. Remove the TODO comment once the guard is in place.

Alternatively, a dedicated `CommentAccessGuard` mirroring the logic in `verifyAccess` can be applied at the route level, consistent with the guard-based pattern used elsewhere.

## Proof of Concept

```
# Step 1 – Attacker registers and obtains a JWT
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2 – Enumerate a transaction ID (sequential integers)
# Step 3 – Write a comment to a transaction the attacker has no access to
POST /transactions/42/comments
Authorization: Bearer <JWT>
{ "message": "Injected comment" }
→ HTTP 201 Created  ← comment persisted, no membership check performed

# Step 4 – Read all comments on the same transaction
GET /transactions/42/comments
Authorization: Bearer <JWT>
→ HTTP 200 [ { "message": "Injected comment" }, { "message": "<legitimate participant comment>" } ]
← leaks comment content from legitimate participants
```

All three guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) pass because the attacker holds a valid, non-blacklisted JWT for a verified account. `CommentsService.createComment` writes the record unconditionally. [4](#0-3)

```typescript
  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
```
