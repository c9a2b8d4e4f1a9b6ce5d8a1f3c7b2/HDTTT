### Title
Any Authenticated Transaction Creator Can Delete Approvers Belonging to Transactions They Do Not Own via IDOR in `removeTransactionApprover`

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of the transaction identified by the URL's `:transactionId`, but then passes the unrelated `:id` (approver ID) directly to `removeTransactionApprover`, which performs no cross-ownership check. An attacker who owns any transaction can supply their own `transactionId` in the URL while targeting an approver ID from a completely different transaction, deleting it without authorization.

### Finding Description

**Root cause — missing cross-ownership check in `removeTransactionApprover`**

In `approvers.controller.ts`, the `DELETE /:id` handler performs two independent operations:

```
await this.approversService.getCreatorsTransaction(transactionId, user);  // line 108
await this.approversService.removeTransactionApprover(id);                // line 109
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the caller owns `transactionId` (the URL parameter). It says nothing about the approver identified by `id`.

`removeTransactionApprover` in `approvers.service.ts` then fetches the approver by `id` alone and deletes it — with no check that the approver belongs to `transactionId` or that the caller owns the transaction the approver actually belongs to:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly validates cross-ownership:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The delete path has no equivalent guard.

**Exploit path:**

1. Attacker (user A) creates any transaction T_A. This satisfies `getCreatorsTransaction(T_A, userA)`.
2. Attacker discovers approver ID `X` belonging to victim's transaction T_B (approver IDs are sequential integers, discoverable by enumeration or by being an observer/signer on T_B).
3. Attacker sends: `DELETE /transactions/T_A/approvers/X`
4. Server checks: "Is user A the creator of T_A?" → Yes. Passes.
5. Server calls `removeTransactionApprover(X)` → fetches approver X (which belongs to T_B), deletes it. No further check.
6. Approver X is permanently soft-deleted from T_B without the owner of T_B being involved.

### Impact Explanation

An attacker can unilaterally remove required approvers from any transaction in the organization, regardless of who created it. This directly undermines the multi-signature approval workflow — the core security invariant of the system. Consequences include:

- Reducing a threshold-based approver tree below its required threshold, potentially allowing a transaction to proceed with fewer approvals than intended.
- Completely removing all approvers from a transaction, bypassing the approval gate entirely.
- Sabotaging in-flight transactions belonging to other users or admins.

This is an unauthorized state change with direct impact on transaction integrity and financial authorization controls.

### Likelihood Explanation

- Precondition: attacker must be an authenticated, verified organization member who has created at least one transaction (a normal user capability).
- Approver IDs are sequential integers, trivially enumerable.
- No rate limiting or anomaly detection is needed to exploit this — a single crafted HTTP DELETE request is sufficient.
- No privileged access, leaked credentials, or social engineering required.

### Recommendation

In `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the `transactionId` supplied in the URL. The simplest fix mirrors what `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the claimed transaction
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  // Verify the caller owns that transaction
  await this.getCreatorsTransaction(transactionId, user);

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
}
```

The controller's separate `getCreatorsTransaction` call at line 108 can then be removed since the service enforces it.

### Proof of Concept

```
# Setup: User A owns transaction 10 (T_A). User B owns transaction 20 (T_B).
# T_B has approver with id=55 (userId=3, required for B's threshold).

# Step 1: User A authenticates and obtains JWT token_A.

# Step 2: User A sends:
DELETE /transactions/10/approvers/55
Authorization: Bearer token_A

# Result: HTTP 200 true
# Approver 55 (belonging to T_B) is soft-deleted.
# T_B's approval threshold is now broken without User B's knowledge or consent.
