### Title
Creator of Any Transaction Can Delete Approvers Belonging to Other Transactions via Mismatched Authorization Check

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system, disrupting approval workflows.

### Finding Description

**Root Cause — Controller (`approvers.controller.ts`, lines 102–113):**

The `removeTransactionApprover` handler performs two sequential calls:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no relationship check
  return true;
}
``` [1](#0-0) 

Step 1 verifies the user is the creator of the transaction at `:transactionId`. Step 2 deletes the approver at `:id`. There is **no check that the approver with `:id` belongs to the transaction with `:transactionId`**.

**Service (`approvers.service.ts`, lines 534–544):**

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

The service method accepts only an `id`, fetches the approver, and removes it — with no ownership or cross-transaction relationship validation.

**Contrast with `updateTransactionApprover` (lines 386–394), which correctly validates:**

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The update path verifies the approver's root transaction matches the URL parameter and that the user is the creator of *that* transaction. The delete path skips both checks entirely.

### Impact Explanation

Any authenticated, verified user who has created at least one transaction can delete approvers from **any** transaction in the system. Concretely:

- An attacker can remove required approvers from a victim's transaction, causing it to bypass the intended approval threshold and proceed to execution without proper authorization.
- An attacker can remove all approvers from a transaction, permanently blocking it from reaching `WAITING_FOR_EXECUTION` status if approvers are required.
- This constitutes unauthorized state mutation of another user's transaction — a direct integrity failure in the multi-signature orchestration trust model.

### Likelihood Explanation

**High.** The attacker preconditions are minimal:
- Must be a registered, verified user (standard account creation flow).
- Must have created at least one transaction (to pass the `getCreatorsTransaction` check on their own transaction).
- No admin access, no leaked credentials, no privileged role required.
- The exploit is a single HTTP DELETE request with a crafted URL.

### Recommendation

In the service's `removeTransactionApprover` method, add the same cross-transaction relationship check that `updateTransactionApprover` already performs:

```typescript
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  // Verify the user is the creator of that transaction
  await this.getCreatorsTransaction(rootNode.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Update the controller to pass `transactionId` and `user` to the service method, and remove the now-redundant `getCreatorsTransaction` call from the controller.

### Proof of Concept

**Setup:**
- Attacker is a registered, verified user and creator of Transaction A (ID = 1).
- Victim is the creator of Transaction B (ID = 2), which has an approver record with ID = 99.

**Exploit:**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>
```

**Execution trace:**
1. Controller calls `getCreatorsTransaction(1, attacker)` → passes (attacker is creator of transaction 1). [4](#0-3) 
2. Controller calls `removeTransactionApprover(99)` → service fetches approver 99 (belongs to transaction 2) and deletes it. [5](#0-4) 
3. Approver 99 is permanently removed from Transaction B. Victim's approval workflow is corrupted.

**Expected result:** HTTP 200, approver 99 deleted from transaction 2 without the attacker having any relationship to transaction 2.
