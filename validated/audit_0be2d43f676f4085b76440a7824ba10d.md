### Title
Use of Non-Cryptographically Secure `Math.random()` for Temporary Password Generation

### Summary
The `generatePassword()` method in the back-end API's `AuthService` uses JavaScript's `Math.random()` — a non-cryptographically secure pseudo-random number generator (PRNG) — to produce temporary passwords that are emailed to newly registered users. This is a direct analog to KZG-06, where a security-sensitive secret was generated with insufficient entropy.

### Finding Description
The `generatePassword()` private method in `back-end/apps/api/src/auth/auth.service.ts` constructs a temporary password of the form `xxxxx-xxxxx` (10 lowercase ASCII letters separated by a hyphen) using `Math.random()`:

```typescript
private generatePassword() {
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () => String.fromCharCode(97 + Math.floor(Math.random() * 26))).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
``` [1](#0-0) 

This password is then passed directly to `usersService.createUser()` or used to update an existing `UserStatus.NEW` account, and is emailed to the new user as their initial credential: [2](#0-1) 

`Math.random()` in Node.js (V8 engine) uses the **xorshift128+** algorithm, which is **not a CSPRNG**. Its internal 128-bit state can be recovered from a small number of observed outputs, and its output is predictable given knowledge of the seed or prior outputs. The password alphabet is also restricted to 26 lowercase letters, yielding a theoretical maximum of 26^10 ≈ 1.4 × 10^14 combinations — but the effective entropy is far lower when the PRNG state is recoverable.

The `signUp` endpoint is admin-only and protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `AdminGuard`: [3](#0-2) 

However, the temporary password is the sole credential protecting the new account until the user logs in and changes it via `change-password`.

### Impact Explanation
An attacker who can observe or infer the approximate time of account creation (e.g., via a registration notification email, a public user directory, or a side-channel) can attempt to reconstruct the V8 PRNG state and predict the temporary password. With the predicted password, the attacker can log in as the newly created user before that user has a chance to change their password, achieving full account takeover. The new user's account may have access to sensitive organization transactions, signing keys, and approval workflows.

### Likelihood Explanation
Likelihood is **Low-Medium**. Exploiting V8's xorshift128+ requires either observing prior `Math.random()` outputs from the same process (possible if any other endpoint leaks random values) or brute-forcing the reduced keyspace. The window of opportunity is the time between account creation and the user's first login. In practice, users may not log in immediately after receiving the registration email, extending this window. The attack is non-trivial but has been demonstrated in research contexts against Node.js applications.

### Recommendation
Replace `Math.random()` with Node.js's built-in CSPRNG. Use `crypto.randomBytes()` from the `node:crypto` module to generate the temporary password:

```typescript
import { randomBytes } from 'node:crypto';

private generatePassword(): string {
  const alphabet = 'abcdefghijklmnopqrstuvwxyz';
  const getRandomLetters = (length: number) =>
    Array.from(
      { length },
      (_, i) => alphabet[randomBytes(length)[i] % alphabet.length],
    ).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
```

Additionally, consider increasing the password length and character set (uppercase, digits, symbols) to raise the brute-force cost independently of the PRNG quality.

### Proof of Concept

1. Admin calls `POST /auth/signup` with a target email. The server calls `generatePassword()` using `Math.random()`.
2. The attacker, who has access to any other endpoint that exposes `Math.random()` output from the same Node.js process (e.g., any randomized response ordering, jitter values, or similar), collects enough outputs to reconstruct the xorshift128+ state using a known V8 PRNG state-recovery tool.
3. The attacker replays the PRNG forward from the recovered state to predict the 10 characters generated for the temporary password.
4. The attacker calls `POST /auth/login` with the victim's email and the predicted password before the victim logs in.
5. The attacker receives a valid JWT and gains full access to the victim's organization account, including transaction approvals and key management.
