Audit Report

## Title
Malicious Registered User Can Trick Co-Signers Into Unknowingly Authorizing Mainnet Transactions Labeled as Testnet

## Summary
The `mirrorNetwork` field in `CreateTransactionDto` is user-supplied and is not cryptographically bound to `transactionBytes`. Because Hedera mainnet and testnet share overlapping node account IDs (0.0.3–0.0.9), a malicious registered user can submit a real mainnet transaction with `mirrorNetwork: 'testnet'`, pass all server-side validation, and collect co-signer signatures from users who believe they are signing a harmless testnet transaction.

## Finding Description

**Root cause — `mirrorNetwork` is not signed and not cryptographically bound to `transactionBytes`.**

In `validateAndPrepareTransaction`, the signature verification covers only `dto.transactionBytes`:

```ts
const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
``` [1](#0-0) 

The `mirrorNetwork` value is accepted from user input with no binding to the signed payload.

The client used for node validation is built directly from the attacker-controlled `dtos[0].mirrorNetwork`:

```ts
const client = await getClientFromNetwork(dtos[0].mirrorNetwork);
``` [2](#0-1) 

`getClientFromNetwork` simply calls `Client.forName(mirrorNetwork[0])` for the named networks:

```ts
if ([MAINNET, TESTNET, PREVIEWNET].includes(mirrorNetwork[0])) {
  return Client.forName(mirrorNetwork[0]);
}
``` [3](#0-2) 

The node-validation guard then checks whether every node ID in the transaction is present in the testnet client's node set:

```ts
const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
  throw new BadRequestException(ErrorCodes.TNVN);
}
``` [4](#0-3) 

`isTransactionValidForNodes` performs a pure string comparison of account IDs:

```ts
return txNodeIds.every((id) => allowedNodeAccountIds.has(id));
``` [5](#0-4) 

**Why the guard fails:** Hedera mainnet and testnet both use node account IDs 0.0.3–0.0.9. A mainnet transaction constructed with only those node IDs passes testnet node validation because the account ID strings are identical on both networks. The check compares account ID strings, not network endpoints or ledger IDs.

The stored `mirrorNetwork` is taken directly from the attacker-controlled DTO field, not derived from the validated client:

```ts
mirrorNetwork: data.mirrorNetwork,
``` [6](#0-5) 

This value is then persisted to the database: [7](#0-6) 

The `TransactionDto` exposes `transactionBytes` as hex in API responses, making the fully-signed bytes retrievable by the attacker: [8](#0-7) 

## Impact Explanation

Co-signers unknowingly authorize real mainnet transactions. Depending on the transaction type, this enables:

- Unauthorized HBAR transfers on mainnet, draining accounts that require multi-party approval.
- Unauthorized `AccountUpdateTransaction` replacing account keys on mainnet.
- Unauthorized `FileUpdateTransaction` modifying Hedera system files (fee schedules, address books) — the tool is explicitly used for these high-stakes operations by the Hedera Council.

The Hedera Transaction Tool's stated purpose is managing high-stakes network operations for the Hedera Council. A successful attack against council members signing what they believe is a testnet transaction constitutes a critical integrity failure.

## Likelihood Explanation

The attacker requires only a registered organization user account — no admin, operator, or governance keys. The attack requires no cryptographic breaks, no race conditions, and no special timing. It is fully deterministic and repeatable. The only prerequisite is that the attacker's key is enrolled in the organization, which is a normal onboarding step.

## Recommendation

**Bind `mirrorNetwork` to the transaction bytes cryptographically or derive it from the transaction itself.**

1. **Ledger ID check (preferred):** After parsing the transaction bytes and freezing with the claimed-network client, compare the transaction's embedded `_ledgerId` against the expected ledger ID for the claimed `mirrorNetwork`. Reject if they do not match. Hedera mainnet has ledger ID `0x00`, testnet `0x01`, previewnet `0x02`.

2. **Exclusive node-set check:** Reject any transaction whose node IDs are a subset of more than one named network's node set. Since 0.0.3–0.0.9 appear on both mainnet and testnet, a transaction using only those IDs should be rejected when the claimed network is testnet (or mainnet), unless the ledger ID unambiguously identifies the network.

3. **Include `mirrorNetwork` in the signed payload:** Require the creator to sign a payload that includes both `transactionBytes` and `mirrorNetwork` (e.g., `sign(hash(transactionBytes || mirrorNetwork))`), so the declared network cannot be altered after signing.

## Proof of Concept

```python
# Pseudocode — no mainnet funds required to demonstrate the bypass
from hedera_sdk import TransferTransaction, AccountId, TransactionId, PrivateKey, Client

attacker_key = PrivateKey.generate()
mainnet_client = Client.for_mainnet()

# Step 1: Build a real mainnet TransferTransaction using only shared node IDs
tx = (TransferTransaction()
      .set_transaction_id(TransactionId.generate(AccountId(0, 0, VICTIM_ACCOUNT)))
      .add_hbar_transfer(AccountId(0, 0, VICTIM_ACCOUNT), -100_000_000)  # -1 HBAR
      .add_hbar_transfer(AccountId(0, 0, ATTACKER_ACCOUNT), +100_000_000)
      .set_node_account_ids([AccountId(0,0,3), AccountId(0,0,4), AccountId(0,0,5)])
      .freeze_with(mainnet_client))

tx_bytes = tx.to_bytes()
signature = attacker_key.sign(tx_bytes)

# Step 2: POST to the tool's API with mirrorNetwork='testnet'
payload = {
    "name": "Routine testnet transfer",
    "description": "Testnet maintenance",
    "transactionBytes": tx_bytes.hex(),
    "creatorKeyId": ATTACKER_KEY_ID,
    "signature": signature.hex(),
    "mirrorNetwork": "testnet",   # <-- lie: this is actually a mainnet transaction
}
# POST /transactions → 201 Created
# Server validates: signature ✓, nodes 0.0.3-0.0.5 ∈ testnet node set ✓, not expired ✓
# Transaction stored as mirrorNetwork='testnet'

# Step 3: Co-signers see "testnet" label and sign
# Step 4: Attacker retrieves fully-signed bytes from GET /transactions/:id
# Step 5: Attacker submits signed bytes directly to mainnet gRPC — transaction executes
```

The node IDs 0.0.3, 0.0.4, 0.0.5 are present in both `Client.forTestnet().network` and `Client.forMainnet().network`, so `isTransactionValidForNodes` returns `true` for a testnet client even though the transaction targets mainnet. [9](#0-8)

export const isTransactionValidForNodes = (
  sdkTransaction: SDKTransaction,
  allowedNodeAccountIds: Set<string>
): boolean  => {
  const nodeAccountIds = (sdkTransaction as any)._nodeAccountIds;
  const txNodeIds: string[] = [];
  if (
    nodeAccountIds &&
    typeof nodeAccountIds.length === 'number' &&
    typeof nodeAccountIds.get === 'function'
  ) {
    for (let i = 0; i < nodeAccountIds.length; i++) {
      const id = nodeAccountIds.get(i);
      const accountId =
        id instanceof AccountId ? id : AccountId.fromString(String(id));
      txNodeIds.push(accountId.toString());
    }
  }

  return txNodeIds.every((id) => allowedNodeAccountIds.has(id));
};
```
