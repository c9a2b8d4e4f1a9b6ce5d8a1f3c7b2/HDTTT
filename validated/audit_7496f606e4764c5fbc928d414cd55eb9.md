### Title
Any Verified User Can Post Comments on Transactions They Have No Access To

### Summary
The `CommentsController` exposes `POST /transactions/:transactionId/comments` protected only by `VerifiedUserGuard`, but the actual operation requires the caller to have a relationship to the transaction (creator, signer, observer, or approver). No such check exists. The code itself acknowledges this with a `//TODO` comment. This is the direct analog of the external report: the guard on the entry point (`VerifiedUserGuard`) does not match the permission actually required to perform the action.

### Finding Description
`CommentsController` is decorated with `@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)` at the class level. The `createComment` handler adds no further guard or service-level ownership check:

```ts
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
``` [1](#0-0) 

Compare this with every other transaction-scoped operation in the codebase. `ObserversService.createTransactionObservers` explicitly checks `transaction.creatorKey?.userId !== user.id` before proceeding. [2](#0-1) 

`ApproversService.getCreatorsTransaction` similarly enforces `transaction.creatorKey?.userId !== user.id`. [3](#0-2) 

`TransactionsService.verifyAccess` defines the canonical access model: a user must be a creator, signer, observer, or approver to interact with a transaction. [4](#0-3) 

The `createComment` path skips all of this. Additionally, `getComments` and `getCommentById` carry no access check at all — any verified user can read comments on any transaction. [5](#0-4) 

### Impact Explanation
A verified organization user who has no relationship to a transaction (not creator, signer, observer, or approver) can:
1. Post arbitrary comments on any transaction in the system.
2. Read all comments on any transaction, leaking potentially sensitive discussion or metadata.

This breaks the transaction visibility model enforced everywhere else in the codebase and constitutes unauthorized write access to transaction data.

### Likelihood Explanation
Any verified user account in the organization can exploit this immediately via a direct HTTP request to `POST /transactions/:id/comments`. No special privileges, race conditions, or complex setup are required. The `transactionId` is an integer that can be enumerated. The developer-left `//TODO` comment confirms the gap was known but unresolved. [6](#0-5) 

### Recommendation
In `CommentsService.createComment`, add an access check equivalent to `verifyAccess` before persisting the comment:

```ts
const hasAccess = await this.transactionsService.verifyAccess(transaction, user);
if (!hasAccess) throw new UnauthorizedException("You don't have permission to comment on this transaction");
```

Apply the same guard to `getComments` and `getCommentById` so comment reads are also scoped to users with transaction access.

### Proof of Concept
1. Register two users: `admin@org.com` (admin) and `attacker@org.com` (verified, non-admin).
2. Admin creates a transaction. `attacker` is not added as signer, observer, or approver.
3. `attacker` sends:
   ```
   POST /transactions/1/comments
   Authorization: Bearer <attacker_jwt>
   { "message": "injected comment" }
   ```
4. The request succeeds with HTTP 201. The comment is persisted on a transaction the attacker has no legitimate access to.
5. `attacker` also sends `GET /transactions/1/comments` and receives all comments, leaking internal discussion. [7](#0-6)
