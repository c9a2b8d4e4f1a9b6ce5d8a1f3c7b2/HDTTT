Audit Report

## Title
Any Verified User Can Enumerate All Transaction Group Metadata via Unguarded "Testing Only" Endpoint

## Summary
`GET /transaction-groups` is explicitly marked "TESTING ONLY" in source but is deployed with no `AdminGuard`. Any verified user can retrieve the `id`, `description`, `atomic`, `sequential`, `createdAt`, and `groupValidTime` of every transaction group in the organization's database.

## Finding Description

The controller applies class-level guards that enforce only JWT validity and account verification: [1](#0-0) 

The `getTransactionGroups` handler is explicitly commented "TESTING ONLY" but carries no additional guard — no `AdminGuard`, no ownership filter, and no user context passed to the service: [2](#0-1) 

The service performs an unfiltered `find()` with no `relations` option: [3](#0-2) 

Because the `TransactionGroup` entity declares `groupItems` as a lazy `@OneToMany` with no `eager: true`: [4](#0-3) 

...the response contains only the group's own columns: `id`, `description`, `atomic`, `sequential`, `createdAt`, `groupValidTime`. Transaction bytes, signer lists, and approver structures are **not** returned by this endpoint. The `AdminGuard` exists and is used elsewhere (e.g., `PATCH /auth/elevate-admin`) to gate organization-wide reads: [5](#0-4) 

Its absence here means the intended privilege boundary is absent.

## Impact Explanation
Any verified user can enumerate every transaction group's ID, description, atomic/sequential flags, and creation timestamp — regardless of whether they are a participant in those groups. The `description` field may contain sensitive business context about financial operations. The single-group fetch (`GET /transaction-groups/:id`) enforces user-scoped access via `getTransactionGroupItemsQuery` and throws `UnauthorizedException` if the user has no role in the group: [6](#0-5) 

The bulk-list endpoint bypasses this check entirely. Note: this application is single-organization (not multi-tenant), so the impact is intra-organization information disclosure, not cross-tenant leakage.

## Likelihood Explanation
High. The attacker requires only a valid verified-user account obtainable through the normal registration flow. No privileged credentials, no special network access, and no obscure parameters are needed. The "TESTING ONLY" comment confirms developer awareness that this endpoint should not be generally accessible.

## Recommendation
Either remove the endpoint entirely (as the comment implies it should be), or add `@UseGuards(AdminGuard)` at the handler level and import `AdminGuard` in the controller. The per-group endpoint already demonstrates the correct pattern of passing `user` to the service for ownership enforcement.

## Proof of Concept
```

# 2. Authenticate to obtain a JWT
POST /auth/login  →  { "accessToken": "<jwt>" }

# 3. Call the unguarded endpoint

Authorization: Bearer <jwt>

# 4. Response: all transaction groups in the database
[
  { "id": 1, "description": "Q1 Payroll Batch", "atomic": true, "sequential": false, "createdAt": "..." },
  { "id": 2, "description": "Treasury Rebalance", "atomic": false, "sequential": true, "createdAt": "..." },
  ...
]
```
