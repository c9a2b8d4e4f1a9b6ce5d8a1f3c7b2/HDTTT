### Title
Mutable Approval Threshold After Signatures Submitted Allows Creator to Bypass Multi-Sig Approval Requirements

### Summary
The `updateTransactionApprover` function in `approvers.service.ts` allows the transaction creator to modify the `threshold` of an approver tree node at any time, with no check on the transaction's current status. Because approval status is evaluated against the live `threshold` value at query time, a malicious creator can lower the threshold after partial approvals have been collected, causing the transaction to pass with fewer approvals than originally required — directly analogous to the `totalVotingPower` snapshot issue in the external report.

### Finding Description

**Root cause:** `updateTransactionApprover` verifies only that the caller is the transaction creator; it never checks whether the transaction is still in a pre-approval state. [1](#0-0) 

The only guard is `getCreatorsTransaction`, which checks ownership but not status: [2](#0-1) 

Contrast this with `approveTransaction`, which correctly rejects calls when the transaction is not in an actionable state: [3](#0-2) 

The threshold update path has no equivalent guard: [4](#0-3) 

Approval status is evaluated against the **current** `threshold` value at read time, not the value that existed when approvers submitted their signatures: [5](#0-4) 

The `TransactionApprover` entity stores `threshold` as a plain mutable column with no historical record: [6](#0-5) 

**Exploit flow:**

1. Malicious creator creates a transaction and sets a threshold-based approver tree requiring 3-of-3 approvals.
2. Two of the three approvers submit their approvals (`approved = true`).
3. Before the third approver acts, the creator calls `PATCH /transactions/:id/approvers/:nodeId` with `{ threshold: 1 }`.
4. The threshold node is updated in the database to `1`.
5. The approval check now reads `approvals.length (2) >= threshold (1)` → `true`.
6. The transaction advances to execution with only 2 of the originally required 3 approvals.

The same path also allows the creator to raise the threshold after approvals are collected, permanently blocking a transaction that legitimately met its original requirement — a denial-of-service against the approval flow.

Additionally, `createTransactionApprovers` carries the same missing status check, allowing the creator to add new approver nodes to a transaction that is already in `WAITING_FOR_EXECUTION` status: [7](#0-6) 

### Impact Explanation

The multi-sig approval model's core invariant is that a transaction cannot proceed without the required number of independent approvals. A creator who can silently lower the threshold after partial approvals have been collected can unilaterally satisfy the approval requirement, rendering the entire approver system ineffective. This is an unauthorized state change that bypasses the protocol's trust model without requiring any privileged credentials beyond being the transaction creator (a role reachable by any authenticated user).

### Likelihood Explanation

Any authenticated user can create a transaction and become its creator. No admin keys or leaked secrets are required. The attack requires only two API calls in sequence: one to wait for partial approvals, and one `PATCH` to the approver endpoint. The window is open for the entire lifetime of the transaction while it is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status.

### Recommendation

Add a transaction status check at the start of `updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover` — mirroring the guard already present in `approveTransaction`:

```typescript
if (
  transaction.status !== TransactionStatus.NEW
) {
  throw new BadRequestException('Approver structure cannot be modified after signing has begun');
}
```

The threshold and approver set should be considered immutable once the transaction leaves `NEW` status, so that the denominator used for approval evaluation is effectively snapshotted at the time the transaction enters the signing phase.

### Proof of Concept

1. Creator (User A) creates a transaction and sets approvers: `{ threshold: 3, approvers: [User B, User C, User D] }`.
2. User B and User C call `POST /transactions/:id/approve` → both records have `approved = true`.
3. Creator calls `PATCH /transactions/:id/approvers/:thresholdNodeId` with body `{ "threshold": 1 }`.
4. Server executes `UPDATE transaction_approver SET threshold = 1 WHERE id = :thresholdNodeId` with no status guard.
5. The next status evaluation reads `approvals (2) >= threshold (1)` → transaction is marked ready for execution.
6. Transaction executes with only 2 of the originally required 3 approvals, bypassing User D's veto. [8](#0-7)

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }
