### Title
Creator Can Execute Individual Transactions from an Atomic Group, Breaking Atomicity Guarantee

### Summary
The `PATCH /transactions/execute/:id` API endpoint in `TransactionsService.executeTransaction` allows the creator of a `isManual` transaction to execute it individually on Hedera, even when that transaction belongs to an atomic `TransactionGroup`. No group-membership check exists at the execution gate. Separately, `ExecuteService.executeTransactionGroup` silently drops any group member not in `WAITING_FOR_EXECUTION` status before submitting the rest, so the scheduler proceeds with a partial set — permanently breaking the atomicity invariant that all members of an atomic group must execute together or not at all.

### Finding Description

**Entry point — `PATCH /transactions/execute/:id`** [1](#0-0) 

`executeTransaction` performs exactly two checks before calling `executeService.executeTransaction(transaction)`:
1. Is the caller the creator? (`getTransactionForCreator`)
2. Is the transaction flagged `isManual`?

There is no check for whether the transaction belongs to an atomic or sequential `TransactionGroup`. The `TransactionGroup` entity explicitly carries `atomic` and `sequential` flags: [2](#0-1) 

**Silent partial-execution in `executeTransactionGroup`**

When the scheduler later calls `executeTransactionGroup` for the group, it filters the group items: [3](#0-2) 

Any transaction already moved to `EXECUTED` (because the creator submitted it individually) is silently dropped. The remaining transactions are then submitted to Hedera without the already-executed member. There is no guard that aborts execution when the surviving set is smaller than the full group for an `atomic: true` group.

**Scheduler path that triggers group execution** [4](#0-3) 

The scheduler calls `collateGroupAndExecute` → `addGroupExecutionTimeout` → `executeTransactionGroup` without re-verifying that all group members are still in `WAITING_FOR_EXECUTION`.

**`isManual` is a valid per-transaction flag**

The `CreateTransactionGroupDto` accepts individual transaction DTOs, each of which can carry `isManual: true` through `validateAndPrepareTransaction`: [5](#0-4) 

**Exploit flow**:
1. Creator submits an atomic `TransactionGroup` (A, B, C) with transaction A having `isManual: true`.
2. Organization members sign all three transactions expecting atomic execution.
3. Creator calls `PATCH /transactions/execute/A` after A's `validStart` passes.
4. `TransactionsService.executeTransaction` passes both checks (creator + isManual); `ExecuteService.executeTransaction` submits A to Hedera; A's DB status becomes `EXECUTED`.
5. The chain-service scheduler picks up B and C (still `WAITING_FOR_EXECUTION`), calls `executeTransactionGroup`.
6. The filter at line 65–67 of `execute.service.ts` drops A; B and C are submitted without A.
7. Atomicity is broken: A executed at a different time and independently of B and C.

### Impact Explanation

The `atomic` flag on a `TransactionGroup` is the protocol's guarantee that all member transactions execute together or not at all. A malicious creator can violate this guarantee by individually executing one member before the group fires. In an organization context, other signers approved the full group under the assumption of atomicity. Selective execution can exploit intermediate on-chain state (e.g., an account key update in A that changes authorization for B and C, or a token transfer in A that the creator exploits before B and C settle). The `Proposal.executed` analog here is that the group's DB records show mixed statuses (`EXECUTED` for A, `EXECUTED` for B and C later) with no indication that atomicity was violated.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The only precondition is that at least one transaction in the atomic group is created with `isManual: true`, which is a standard product feature. The API endpoint `PATCH /transactions/execute/:id` is publicly documented and reachable by any authenticated user who is the creator. No special tooling is required beyond a standard HTTP client.

### Recommendation

1. **In `TransactionsService.executeTransaction`**: Before calling `executeService.executeTransaction`, load the transaction's `groupItem` relation and check whether it belongs to an atomic or sequential group. If it does, reject the individual execution with an appropriate error (e.g., `"Cannot manually execute a transaction that belongs to an atomic/sequential group"`).

2. **In `ExecuteService.executeTransactionGroup`**: For groups with `atomic: true`, abort execution entirely if the filtered set of `WAITING_FOR_EXECUTION` transactions is smaller than the full group's item count, rather than silently proceeding with a partial set.

### Proof of Concept

**Preconditions**: Authenticated organization user (the creator), backend running.

**Steps**:

```
# 1. Create an atomic group with isManual: true on transaction A
POST /transaction-groups
{
  "description": "atomic-group",
  "atomic": true,
  "sequential": false,
  "groupItems": [
    { "seq": 1, "transaction": { ..., "isManual": true } },   // Transaction A
    { "seq": 2, "transaction": { ..., "isManual": false } },  // Transaction B
    { "seq": 3, "transaction": { ..., "isManual": false } }   // Transaction C
  ]
}
# → returns group with transaction IDs, e.g. A=101, B=102, C=103

# 2. Wait for all three to reach WAITING_FOR_EXECUTION status
#    (other signers sign; scheduler promotes status)

# 3. Creator individually executes A after validStart passes
PATCH /transactions/execute/101
Authorization: Bearer <creator-jwt>
# → 200 OK; A is now EXECUTED on Hedera and in DB

# 4. Scheduler fires executeTransactionGroup for the group
#    → filter drops A (status=EXECUTED)
#    → B and C are submitted to Hedera without A
#    → atomicity broken; A executed at a different time than B+C
```

**Expected (correct) behavior**: Step 3 should return 400 with "Cannot manually execute a transaction belonging to an atomic group."

**Observed behavior**: Step 3 succeeds; the group later executes B and C without A, violating the atomicity invariant.

  async executeTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }
