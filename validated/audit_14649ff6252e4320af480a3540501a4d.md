Audit Report

## Title
Creator Can Manipulate Approver List During Active Approval Phase, Bypassing Organizational Approval Requirements

## Summary
The transaction creator can add or remove approvers from a transaction at any time — including after the transaction has entered `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status — because `getCreatorsTransaction`, the shared authorization gate for all approver mutations, enforces only creator identity and never checks the transaction's current status. A malicious creator can remove existing approvers, add themselves as the sole approver, and self-approve, fully circumventing the organizational multi-party approval process.

## Finding Description

**Root cause — `getCreatorsTransaction` has no status guard**

`getCreatorsTransaction` is the shared authorization gate called before every approver mutation. It fetches the transaction, verifies the caller is the creator, and returns — with no check on `transaction.status`: [1](#0-0) 

**Approver removal path — no status guard**

The `DELETE /:id` controller calls `getCreatorsTransaction` (identity only) then `removeTransactionApprover`: [2](#0-1) 

`removeTransactionApprover` calls `removeNode` — a raw SQL recursive soft-delete — with no status check: [3](#0-2) 

**Approver creation path — no status guard**

`createTransactionApprovers` calls only `getCreatorsTransaction` before inserting new approvers, with no status check: [4](#0-3) 

**Secondary issue — `removeNode` does not adjust parent threshold**

`removeNode` soft-deletes a node and all its descendants but never decrements the parent's `threshold` column: [5](#0-4) 

By contrast, `updateTransactionApprover` (the `listId=null` detach path) correctly adjusts the parent threshold when a child is removed: [6](#0-5) 

This means a `DELETE` on a child approver can leave the parent with `threshold=2` and only one remaining child, making the threshold permanently unreachable.

**`approveTransaction` does have a status check**, but it is only reached after the approver list has already been tampered with: [7](#0-6) 

## Impact Explanation
The organizational approval workflow — the primary trust mechanism in Organization Mode — is completely bypassable by the transaction creator. Any transaction requiring multi-party sign-off can be self-approved by the creator after removing legitimate approvers, enabling unauthorized movement of funds or unauthorized account/file changes on Hedera. This is a critical integrity failure in the system's trust model.

## Likelihood Explanation
The attacker is a normal authenticated user (the transaction creator) abusing documented, reachable API endpoints (`DELETE /transactions/:transactionId/approvers/:id`, `POST /transactions/:transactionId/approvers`, `POST /transactions/:transactionId/approvers/approve`). No privileged credentials, leaked secrets, or cryptographic breaks are required. Any creator who faces rejection — or who simply wants to bypass the approval requirement — can trivially execute this sequence.

## Recommendation

1. **Add a status guard in `getCreatorsTransaction`** (or in each mutation method individually). Approver mutations should be rejected if the transaction is not in `WAITING_FOR_SIGNATURES` draft/setup state. Specifically, mutations should be blocked once the transaction has entered `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

2. **Fix `removeNode` to adjust the parent threshold.** After soft-deleting a child, fetch the parent and decrement its `threshold` if the remaining child count falls below it (mirroring the logic already present in `updateTransactionApprover`).

## Proof of Concept

1. Creator submits a transaction and sets approvers `[A, B]` with `threshold=2`. Transaction enters `WAITING_FOR_SIGNATURES`.
2. Creator calls `DELETE /transactions/:txId/approvers/:A_id` — succeeds because `getCreatorsTransaction` only checks identity (line 640), not status.
3. Creator calls `DELETE /transactions/:txId/approvers/:B_id` — same result.
4. Creator calls `POST /transactions/:txId/approvers` with `{ userId: creatorId }` — `createTransactionApprovers` calls `getCreatorsTransaction` (line 239), which again only checks identity; insertion succeeds.
5. Creator calls `POST /transactions/:txId/approvers/approve` — `approveTransaction` finds the creator in the approver list (line 556), the transaction is still in `WAITING_FOR_SIGNATURES` (status check at line 584 passes), and the approval is recorded.
6. Transaction proceeds toward execution with zero genuine third-party approvals.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
