Audit Report

## Title
No Upper-Bound Validation on `validStart` Allows Transactions to Be Permanently Stuck in Unrecoverable State

## Summary
`validateAndPrepareTransaction` in `transactions.service.ts` rejects already-expired transactions via `isExpired()` but applies no upper-bound check on how far in the future `validStart` may be. An authenticated user can submit transaction bytes with a `validStart` set decades ahead. No execution path ever reaches such a transaction, the expiry cron never fires for it, and no automated recovery exists — leaving the record permanently in `WAITING_FOR_SIGNATURES` (or at most `WAITING_FOR_EXECUTION`) with no automated resolution.

## Finding Description

**Root cause — missing upper-bound check at ingestion:**

`validateAndPrepareTransaction` applies only one temporal guard: [1](#0-0) 

`isExpired` is defined as: [2](#0-1) 

It only checks `now >= validStart + duration` (already past). A transaction with `validStart` = year 2076 passes trivially. `validStart` is then stored verbatim from the user-supplied bytes: [3](#0-2) 

**Why the execution path never reaches it:**

`prepareTransactions` is called from two crons. `handleTransactionsBetweenNowAndAfterThreeMinutes` queries only `Between(threeMinutesBefore, threeMinutesLater)`: [4](#0-3) 

`handleInitialTransactionStatusUpdate` queries `MoreThan(threeMinutesBefore)` (which does include far-future transactions), but `prepareTransactions` then filters via `isValidStartExecutable`: [5](#0-4) 

A transaction 50 years in the future fails `time <= now`, so it is never passed to `collateAndExecute` or `addExecutionTimeout`.

`handleTransactionsAfterOneWeek` does query far-future transactions (via `MoreThan(getOneWeekLater())`), but it only calls `updateTransactions` — it never calls `prepareTransactions`: [6](#0-5) 

**Why the expiry cron never fires:**

`handleExpiredTransactions` marks a transaction expired only when `validStart < getThreeMinutesBefore()`: [7](#0-6) 

A transaction with `validStart` decades in the future never satisfies this condition. It remains in `WAITING_FOR_SIGNATURES` (or `WAITING_FOR_EXECUTION` after enough signatures are collected) indefinitely.

**DTO has no lower-bound constraint on `reminderMillisecondsBefore`:** [8](#0-7) 

Only `@IsNumber()` is present — no `@Min(0)`. A negative value causes `remindAt = validStart.getTime() - (negative) = validStart.getTime() + |value|`, scheduling a reminder arbitrarily far in the future and consuming a scheduler slot indefinitely: [9](#0-8) 

## Impact Explanation

Any authenticated organization user can submit a transaction whose `validStart` is decades in the future. The record is permanently stored in `WAITING_FOR_SIGNATURES`. No automated process ever executes or expires it. If the transaction belongs to a sequential `TransactionGroup`, the entire group's execution is blocked indefinitely. Recovery requires direct database intervention by an administrator. This constitutes permanent, unrecoverable corruption of transaction state for the affected group.

## Likelihood Explanation

Low. Requires an authenticated organization user — no admin or operator role needed — to either deliberately or accidentally encode a far-future `validStart` in the Hedera SDK transaction bytes before submission. The Hedera SDK imposes no upper bound on `validStart`, making the crafted input trivially constructable.

## Recommendation

1. **Add an upper-bound check in `validateAndPrepareTransaction`**: After parsing the SDK transaction, reject any `validStart` that exceeds a reasonable horizon (e.g., `now + 1 year` or a configurable constant). Example:
   ```typescript
   const MAX_FUTURE_MS = 365 * 24 * 60 * 60 * 1_000; // 1 year
   const validStartMs = sdkTransaction.transactionId.validStart.toDate().getTime();
   if (validStartMs > Date.now() + MAX_FUTURE_MS) {
     throw new BadRequestException(ErrorCodes.TVS); // Transaction Valid Start too far in future
   }
   ```

2. **Add `@Min(1)` to `reminderMillisecondsBefore`** in `CreateTransactionDto` to prevent negative reminder offsets from scheduling reminders beyond `validStart`.

3. **Consider adding an upper-bound to `cutoffAt`** as well, to prevent unbounded future dates from being stored.

## Proof of Concept

1. Using the Hedera SDK, construct any transaction (e.g., `AccountCreateTransaction`) and set its `TransactionId` with `validStart` = `Timestamp.fromDate(new Date('2076-01-01'))`.
2. Freeze and sign the transaction bytes with a valid organization key.
3. POST to `POST /transactions` with the crafted bytes, a valid `creatorKeyId`, and a valid `signature`.
4. `isExpired` returns `false` (2076 is not in the past). No other temporal check fires. The transaction is saved with `status = WAITING_FOR_SIGNATURES` and `validStart = 2076-01-01`.
5. Observe that `handleExpiredTransactions` (runs every 10 seconds) never marks it expired — `validStart < threeMinutesBefore` is never true.
6. Observe that no execution cron ever calls `collateAndExecute` for it — `isValidStartExecutable` returns `false` for a 2076 timestamp.
7. The transaction remains permanently in `WAITING_FOR_SIGNATURES`. If it is part of a sequential group, the entire group is blocked.

export const isExpired = (transaction: SDKTransaction) => {
  if (!transaction.transactionId?.validStart) {
    return true;
  }

  const validStart = transaction.transactionId.validStart.toDate();
  const duration = transaction.transactionValidDuration;

  return new Date().getTime() >= validStart.getTime() + duration * 1_000;
};
