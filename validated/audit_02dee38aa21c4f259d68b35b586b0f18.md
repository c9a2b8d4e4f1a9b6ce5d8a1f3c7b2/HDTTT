Audit Report

## Title
`isNode` Duplicate-Check Bypass for Threshold Approver Nodes Due to Incorrect Boolean Logic

## Summary
The `isNode` function in `approvers.service.ts` unconditionally returns `false` for threshold (tree) approver nodes due to an extraneous `&& typeof approver.userId === 'number'` condition appended to its return expression. Because threshold nodes have no `userId`, the duplicate-existence guard is silently bypassed, allowing a transaction creator to insert unlimited identical threshold nodes into the approval tree.

## Finding Description

The root cause is in `isNode` at line 665:

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';
``` [1](#0-0) 

A threshold/tree approver node carries no `userId` — the field is `null` or `undefined`. Therefore `typeof approver.userId === 'number'` evaluates to `false` for every such node, short-circuiting the entire expression to `false` regardless of the value of `count`. Even when `count > 0` (a duplicate already exists in the database), `isNode` returns `false`.

This function is the sole duplicate-existence guard inside `createTransactionApprovers`:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
``` [2](#0-1) 

Because `isNode` always returns `false` for threshold nodes, `APPROVER_ALREADY_EXISTS` is never thrown for them, and the duplicate is silently inserted. The existing unit test for `isNode` only exercises the user-node path (`userId: 1`) and does not cover the threshold-node case, so the defect is not caught by the test suite. [3](#0-2) 

## Impact Explanation

A transaction creator can call `POST /transactions/:transactionId/approvers` repeatedly with the same threshold-node DTO (no `userId`, a `threshold` value, and child approvers). Each call bypasses the duplicate check and inserts an additional identical threshold node into the `transaction_approver` table. Consequences:

1. `getTreeStructure` reconstructs the approval tree from the flat database list and produces duplicate branches for every repeated call.
2. All participants receive a corrupted approval tree via `getApproversByTransactionId`.
3. Approval-counting logic that traverses the tree may produce incorrect threshold-satisfaction results — potentially allowing a transaction to be marked ready for execution with fewer real approvals than required, or permanently blocking execution due to unsatisfiable duplicated thresholds.

## Likelihood Explanation

The attacker is the transaction creator — a normal, authenticated user with no special privileges. The exploit path is a standard API call (`POST /transactions/:transactionId/approvers`) with a threshold-only approver DTO (no `userId`). The vulnerable condition is trivially reachable, requires no race condition, no timing dependency, and no leaked credentials. Any transaction creator can trigger it on any transaction they own.

## Recommendation

Remove the extraneous `&& typeof approver.userId === 'number'` condition from the return expression in `isNode`. The function's sole purpose is to report whether a matching record already exists; the `userId` type check is irrelevant to that purpose and breaks the guard for all non-user nodes.

**Fix:**
```typescript
// Before (line 665)
return count > 0 && typeof approver.userId === 'number';

// After
return count > 0;
``` [1](#0-0) 

Additionally, add a unit test for `isNode` with a threshold-node input (no `userId`) to prevent regression.

## Proof of Concept

1. Authenticate as a user who is the creator of transaction ID `42`.
2. Send the following request twice (or more):

```http
POST /transactions/42/approvers
Content-Type: application/json
Authorization: Bearer <creator_token>

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        { "userId": 7 }
      ]
    }
  ]
}
```

3. On the first call, the threshold node is inserted normally.
4. On the second call, `isNode` is invoked with `dtoApprover = { threshold: 1, approvers: [...] }` (no `userId`). The database `count` returns `1` (the record from step 3), but `typeof approver.userId === 'number'` is `false`, so `isNode` returns `false`. The guard does not throw, and a second identical threshold node is inserted.
5. Retrieve the approver tree via `GET /transactions/42/approvers` and observe duplicate threshold branches in the response.
