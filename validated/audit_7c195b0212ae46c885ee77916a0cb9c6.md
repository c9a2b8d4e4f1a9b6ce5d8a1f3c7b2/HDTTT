### Title
Stub Implementation of `getTransactionApproversForTransactions` Silently Returns Empty Approver List in Production

### Summary
`TransactionsService.getTransactionApproversForTransactions` in `back-end/apps/api/src/transactions/transactions.service.ts` is a stub that unconditionally returns an empty array (`[]`) for any non-empty input, with an inline comment acknowledging it is not yet implemented. Any caller relying on this method to retrieve approver data for a batch of transactions will silently receive no approvers, bypassing approver-dependent logic (access control, notification dispatch, signing-requirement checks) for those code paths.

### Finding Description
At lines 825–834 of `transactions.service.ts`, the method `getTransactionApproversForTransactions` is declared with a real signature but contains only a stub body:

```typescript
async getTransactionApproversForTransactions(
  transactionIds: number[],
): Promise<TransactionApprover[]> {
  if (!transactionIds.length) {
    return [];
  }

  //To be implemented when approver functionality is added.
  return [];
}
``` [1](#0-0) 

Regardless of how many valid `transactionIds` are supplied, the function always returns `[]`. The comment explicitly states the real logic has not been written. This is structurally identical to the external report's pattern: a production function replaced by a stub that returns a hardcoded safe/empty value.

By contrast, the single-transaction path `attachTransactionApprovers` (lines 779–784) correctly queries the database via `approversService.getApproversByTransactionId`: [2](#0-1) 

The batch path has no equivalent real implementation.

### Impact Explanation
Any feature that calls `getTransactionApproversForTransactions` to determine approver membership across multiple transactions will behave as if no approvers exist:

- **Access control bypass**: If a caller uses the returned list to decide whether a user is an approver (and therefore permitted to view or act on a transaction), the empty result means no user is ever recognized as an approver through this path.
- **Notification suppression**: Approvers will never receive batch-triggered notifications because the notification pipeline will see an empty approver set.
- **Signing/approval requirement bypass**: Any batch workflow that checks whether required approvals have been collected will treat every transaction as having zero approvers, potentially allowing premature execution or status transitions.

### Likelihood Explanation
The method is part of the public `TransactionsService` API and is reachable through normal authenticated API calls. The stub is in production code (not a test file), so it is active in any deployed environment. The inline comment confirms the omission is known but unresolved, meaning it has persisted through at least one release cycle.

### Recommendation
Replace the stub body with a real database query analogous to the working implementations for signers and observers:

```typescript
async getTransactionApproversForTransactions(
  transactionIds: number[],
): Promise<TransactionApprover[]> {
  if (!transactionIds.length) return [];

  return this.entityManager.find(TransactionApprover, {
    where: { transactionId: In(transactionIds), deletedAt: IsNull() },
    relations: ['user'],
  });
}
```

Ensure test coverage covers the batch approver path, not only the single-transaction path.

### Proof of Concept
1. Create a transaction with one or more approvers via the single-transaction path (which works correctly through `attachTransactionApprovers`).
2. Invoke any code path that calls `getTransactionApproversForTransactions` with the transaction IDs.
3. Observe that the returned list is always `[]`, regardless of the approvers stored in the database.
4. Any authorization or notification logic downstream of this call will behave as if the transaction has no approvers. [1](#0-0)
