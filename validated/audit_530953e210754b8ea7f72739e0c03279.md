Audit Report

## Title
JWT Token Not Blacklisted After Password Change Allows Continued Use of Compromised Tokens

## Summary
The `BlacklistService` JWT denylist is applied to `logout`, `verifyOtp`, and `setPassword`, but is entirely absent from the `changePassword` endpoint. A stolen JWT remains fully valid for its entire configured lifetime even after the legitimate user changes their password.

## Finding Description

`BlacklistService` provides `blacklistToken` and `isTokenBlacklisted` backed by Redis: [1](#0-0) 

`logout` correctly blacklists the current auth token immediately after the operation: [2](#0-1) 

The OTP flow also correctly blacklists tokens in both `verifyOtp` and `setPassword`: [3](#0-2) [4](#0-3) 

However, `changePassword` only delegates to `authService.changePassword` and never calls `blacklistToken`: [5](#0-4) 

The underlying service method also performs no token invalidation — it only validates the old password and updates the hash: [6](#0-5) 

JWTs are signed with an expiry of `JWT_EXPIRATION` days, and the blacklist TTL uses the same value — meaning a non-blacklisted token lives for the full window: [7](#0-6) [8](#0-7) 

The `JwtBlackListAuthGuard` guards `changePassword` itself (so the request must carry a valid, non-blacklisted token to reach the endpoint), but the token used to authenticate the request is never blacklisted upon success: [5](#0-4) 

## Impact Explanation
An attacker who obtains a valid JWT (via XSS, logging, network interception, or insider access) retains full API access to all endpoints guarded by `JwtBlackListAuthGuard` + `JwtAuthGuard` for the entire remaining token lifetime, even after the legitimate user changes their password. The password change — the standard remediation action for a compromised account — provides zero revocation of the stolen credential. [9](#0-8) 

## Likelihood Explanation
This is a low-complexity, realistic attack path. Token theft is a common threat vector. Users routinely change passwords specifically to lock out an attacker. The gap between user intent ("I changed my password to revoke access") and actual system behavior ("the old token still works") is concrete and exploitable without any special privileges or tooling.

## Recommendation
In `AuthController.changePassword`, add a call to `blacklistService.blacklistToken(extractJwtAuth(req))` after a successful password change, mirroring the pattern already used in `logout`. The `@Req() req: Request` parameter must also be added to the handler signature to extract the token. Example:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(
  @GetUser() user: User,
  @Body() dto: ChangePasswordDto,
  @Req() req: Request,
): Promise<void> {
  await this.authService.changePassword(user, dto);
  await this.blacklistService.blacklistToken(extractJwtAuth(req));
}
```

## Proof of Concept

1. Attacker obtains victim's JWT (e.g., via token leakage): `TOKEN_A`
2. Victim suspects compromise and calls `PATCH /auth/change-password` with `TOKEN_A` — succeeds with HTTP 200
3. Attacker immediately uses `TOKEN_A` on any protected endpoint (e.g., `GET /transactions/history`)
4. `JwtBlackListAuthGuard` checks Redis — `TOKEN_A` is **not** in the blacklist (never added by `changePassword`)
5. `JwtAuthGuard` validates the JWT signature and expiry — still valid
6. Request succeeds; attacker retains full access for the remaining `JWT_EXPIRATION` days

Contrast with `logout`: calling `POST /auth/logout` with `TOKEN_A` would add it to Redis, and step 4 would return 401. The fix for `changePassword` is identical in structure. [10](#0-9)

  async blacklistToken(jwt: string) {
    const expirationDays = this.configService.get<number>('JWT_EXPIRATION');
    const expirationSeconds = Number(expirationDays) * 24 * 60 * 60;

    await this.client.set(jwt, this.BLACKLISTED, 'EX', expirationSeconds);
  }

  async isTokenBlacklisted(jwt: string) {
    const data = await this.client.get(jwt);
    return data === this.BLACKLISTED;
  }
