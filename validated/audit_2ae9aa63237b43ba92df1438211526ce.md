### Title
Duplicate Signer Entry Allowed Due to Missing Unique Constraint on `TransactionSigner`

### Summary
The `TransactionSigner` entity defines only a plain (non-unique) composite index on `(transactionId, userKeyId)`. Unlike `TransactionObserver`, which enforces `{ unique: true }` on its composite index, `TransactionSigner` has no database-level or application-level guard against inserting the same `(transactionId, userKeyId)` pair more than once. A malicious authenticated user can submit repeated signing requests for the same transaction with the same key, inflating the recorded signer count and potentially satisfying multi-signature thresholds without the required number of distinct keys.

### Finding Description

**Root cause — entity definition:**

In `back-end/libs/common/src/database/entities/transaction-signer.entity.ts`, line 15:

```ts
@Index(['transactionId', 'userKeyId'])   // ← plain index, NOT unique
export class TransactionSigner { ... }
```

Compare with `back-end/libs/common/src/database/entities/transaction-observer.entity.ts`, line 20:

```ts
@Index(['userId', 'transactionId'], { unique: true })  // ← correctly unique
export class TransactionObserver { ... }
```

The `TransactionApprover` entity (`transaction-approver.entity.ts`) similarly carries only `@Index(['transactionId'])` and `@Index(['userId'])` as separate, non-unique indexes — no composite uniqueness guard on `(transactionId, userId)` or `(transactionId, userKeyId)`.

**Exploit path:**

1. Authenticated user U uploads a public key → receives `userKeyId = K`.
2. U calls the sign endpoint for `transactionId = T` with key `K`.
3. U immediately calls the same sign endpoint again for `(T, K)`.
4. Because no unique constraint exists at the DB level and (based on the entity alone) no application-level duplicate check is enforced before insert, both rows are persisted.
5. Any downstream logic that counts `SELECT COUNT(*) FROM transaction_signer WHERE transactionId = T` now returns 2 for a single real key.
6. If the transaction requires a threshold of N distinct signers, U can reach that threshold by replaying a single key N times.

### Impact Explanation

- **Multi-signature threshold bypass**: A transaction requiring N distinct signers can be satisfied by one user replaying the same key N times. This breaks the core security invariant of the multi-sig workflow.
- **Accounting corruption**: Signer counts, audit logs, and any UI display of "who signed" become incorrect, undermining the integrity of the approval record.
- **Severity: High** — directly violates the trust model of the multi-signature system with no privileged access required.

### Likelihood Explanation

- Any authenticated user with a valid account and at least one registered key can trigger this.
- No admin or special role is needed.
- The attack requires only repeated HTTP calls to the sign endpoint — trivially scriptable.

### Recommendation

1. Add `{ unique: true }` to the composite index on `TransactionSigner`:

```ts
@Index(['transactionId', 'userKeyId'], { unique: true })
export class TransactionSigner { ... }
```

2. Add a corresponding unique constraint on `TransactionApprover` for `(transactionId, userKeyId)` (leaf approvers) to prevent the same issue there.

3. As a defense-in-depth measure, add an application-level existence check in the signers service before inserting a new `TransactionSigner` row, and return a `409 Conflict` if the pair already exists.

### Proof of Concept

```
# Step 1: Authenticate as user U, obtain JWT
POST /auth/login  →  { token: "JWT" }

# Step 2: Register a key, obtain userKeyId = 42
POST /user-keys  →  { id: 42, publicKey: "..." }

# Step 3: Sign transaction T=7 with key 42 — first time (succeeds)
POST /transactions/7/signers  body: { userKeyId: 42, signature: "..." }
→ 201 Created  { id: 1, transactionId: 7, userKeyId: 42 }

# Step 4: Sign transaction T=7 with key 42 — second time (also succeeds, no rejection)
POST /transactions/7/signers  body: { userKeyId: 42, signature: "..." }
→ 201 Created  { id: 2, transactionId: 7, userKeyId: 42 }

# Step 5: Query signer count — returns 2 for one real key
GET /transactions/7/signers  →  [ {id:1,...}, {id:2,...} ]

# If threshold = 2, transaction is now considered fully signed by a single actor.
```
