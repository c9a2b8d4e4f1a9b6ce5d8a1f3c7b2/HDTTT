# Audit Report

## Title
`removeTransactionApprover` Authorization Check Decoupled from Deleted Resource — Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId` but then deletes the approver identified by `:id` with no check that the approver actually belongs to that transaction. An authenticated user who owns any transaction can therefore delete approvers from transactions they did not create.

## Finding Description

**Root cause — missing cross-validation in the delete path**

In the controller, the DELETE handler makes two independent calls: [1](#0-0) 

`getCreatorsTransaction` only confirms the caller created the transaction in the URL parameter. `removeTransactionApprover` then fetches and deletes the approver by its own `id` with no assertion that it belongs to that transaction: [2](#0-1) 

`removeNode` is invoked as `this.removeNode(approver.id)` — `transactionId` is never passed, so no cross-validation is possible inside it either.

**Contrast with the update path — which does the check correctly**

`updateTransactionApprover` resolves the root node of the approver and explicitly verifies it belongs to the URL's `transactionId` before proceeding: [3](#0-2) 

The delete path has no equivalent guard. The authorization check is applied to one resource (`:transactionId` in the URL) while the destructive action targets a different resource (approver `:id`) without cross-validating the two.

## Impact Explanation
An authenticated attacker with no special privileges beyond having created at least one transaction can silently delete approver nodes from any transaction in the organization. This can:

- Remove required approvers, allowing a transaction to proceed without the intended oversight.
- Destroy the entire approver tree of a transaction, making it permanently un-approvable.
- Undermine the multi-signature governance model that is the core security property of the system.

**Severity: High** — direct unauthorized state mutation of another user's transaction approval structure.

## Likelihood Explanation
Any registered, verified user can create a transaction (the minimum precondition). Approver IDs are auto-incremented integers exposed in API responses, making enumeration trivial. No brute-force or cryptographic attack is required. The attacker only needs one approver ID from a target transaction, obtainable from normal API responses or sequential enumeration.

## Recommendation
In `removeTransactionApprover` (or in the controller before calling it), add the same cross-validation that `updateTransactionApprover` already performs:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root node.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.
3. Only then proceed with deletion.

This mirrors the existing guard at lines 386–394 of `approvers.service.ts` and closes the gap between the two mutation paths.

## Proof of Concept

1. **Setup:** User A creates Transaction 1 (they are its creator). User B creates Transaction 2 with approver nodes; one approver has `id = 42`.
2. **Attack:** User A sends:
   ```
   DELETE /transactions/1/approvers/42
   Authorization: Bearer <User A's token>
   ```
3. **Step-by-step execution:**
   - `getCreatorsTransaction(1, userA)` — passes, User A is the creator of Transaction 1.
   - `removeTransactionApprover(42)` — fetches approver 42 (belongs to Transaction 2), calls `removeNode(42)`, soft-deletes it. No `transactionId` check is performed.
4. **Result:** Approver 42 from Transaction 2 is deleted. Transaction 2's approval tree is corrupted or destroyed without User B's knowledge. [4](#0-3) [2](#0-1)
