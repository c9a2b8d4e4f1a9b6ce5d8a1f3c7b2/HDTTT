### Title
Missing `@Min(1)` Constraint on `threshold` in `CreateTransactionApproverDto` Allows Zero-Threshold Approval Bypass

### Summary

`CreateTransactionApproverDto` accepts `threshold: 0` because it lacks the `@Min(1)` decorator that is correctly enforced in `UpdateTransactionApproverDto`. Any authenticated user who can create a transaction can set `threshold=0` on an approver list at creation time, potentially causing the approval gate to be satisfied with zero actual approver signatures — bypassing the multi-party authorization model entirely.

### Finding Description

The external report's vulnerability class — missing bounds validation on numeric setter/initializer parameters — maps directly to an inconsistency between the create and update DTOs for transaction approvers.

**`CreateTransactionApproverDto`** (create path) — no minimum bound on `threshold`: [1](#0-0) 

```ts
@IsNumber()
@IsOptional()
threshold?: number;   // ← no @Min(1)
```

**`UpdateTransactionApproverDto`** (update path) — correctly enforces `@Min(1)`: [2](#0-1) 

```ts
@IsNumber()
@Min(1)              // ← present here, absent in create DTO
@IsOptional()
threshold?: number;
```

The global `ValidationPipe` with `whitelist: true` is applied in `setupApp`: [3](#0-2) 

This pipe enforces DTO decorators strictly, so the absence of `@Min(1)` in the create DTO means `threshold: 0` (or any negative integer) passes validation and reaches the service layer unchecked.

Additional missing `@IsNotEmpty()` constraints exist on `name`/`description` in `CreateTransactionDto` and `message` in `CreateCommentDto`, but those are data-integrity issues rather than security-critical: [4](#0-3) [5](#0-4) 

### Impact Explanation

If the approver service evaluates a threshold-based approval check as "approvals received ≥ threshold," then `threshold=0` means the condition is satisfied immediately with zero approver signatures. This would allow a transaction creator to self-approve any transaction that is supposed to require multi-party sign-off, undermining the organization's authorization workflow. The `CreateTransactionApproverDto` is used in the nested approver tree structure, so the bypass can be embedded at any level of the approver hierarchy. [6](#0-5) 

### Likelihood Explanation

Any authenticated user who can submit a `POST` request to create a transaction with approvers can trigger this. No privileged access is required — only a valid session. The attacker simply sets `threshold: 0` in the `approversArray` payload. The inconsistency with the update DTO strongly indicates the `@Min(1)` constraint was intentional and was accidentally omitted from the create path.

### Recommendation

Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```ts
import { IsArray, IsNumber, IsOptional, ValidateNested, ArrayMinSize, Min } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @Min(1)          // ← add this
  @IsOptional()
  threshold?: number;
  // ...
}
```

Also add `@IsNotEmpty()` to `name` and `description` in `CreateTransactionDto` and `message` in `CreateCommentDto` to prevent empty-string persistence.

### Proof of Concept

1. Authenticate as any normal user and obtain a session token.
2. Create a transaction with an approver list where `threshold: 0`:

```json
POST /transactions
{
  "name": "test",
  "description": "test",
  "transactionBytes": "<valid_bytes>",
  "creatorKeyId": 1,
  "signature": "<valid_sig>",
  "mirrorNetwork": "testnet"
}

POST /transactions/{id}/approvers
{
  "approversArray": [
    {
      "threshold": 0,
      "approvers": [
        { "userId": 2 },
        { "userId": 3 }
      ]
    }
  ]
}
```

3. The `ValidationPipe` accepts `threshold: 0` because `CreateTransactionApproverDto` has no `@Min(1)`.
4. The approver service receives `threshold=0`; if it evaluates `approvedCount >= threshold`, the condition is `0 >= 0 = true` immediately, and the transaction proceeds without any approver action.
