Audit Report

## Title
`removeTransactionApprover` Lacks Cross-Transaction Ownership Verification, Enabling IDOR to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller as the creator of `:transactionId`, then passes the unrelated `:id` (approver ID) directly to `removeTransactionApprover()`, which never verifies that the approver belongs to the authorized transaction. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction by supplying their own `transactionId` for the authorization check and a victim's approver `id` for the deletion.

## Finding Description

**Root cause — controller authorization is decoupled from service action:**

In `approvers.controller.ts`, the `DELETE` handler at lines 102–113 authorizes the caller against `transactionId`, then immediately calls the service with the unrelated `id`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← authorizes transactionId
  await this.approversService.removeTransactionApprover(id);               // ← acts on id, no cross-check
  return true;
}
``` [1](#0-0) 

The service `removeTransactionApprover` (lines 534–544) accepts only the approver `id` and performs no check that the approver belongs to the transaction that was just authorized:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-transaction check before acting (lines 389–394):

```typescript
/* Verifies that the root transaction is the same as the param */
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

/* Verifies that the user is the creator of the transaction */
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The `updateTransactionApprover` path calls `getRootNodeFromNode(approver.id)` to find the root of the approver tree, then asserts `rootNode.transactionId === transactionId` before proceeding. The `removeTransactionApprover` path performs no equivalent check — the `transactionId` route parameter is used solely for the `getCreatorsTransaction` ownership check and is then discarded. [4](#0-3) 

Additionally, `removeNode` performs a recursive soft-delete of the target node and all its children via a CTE query, meaning a single crafted request can wipe an entire approver subtree belonging to a victim transaction. [5](#0-4) 

## Impact Explanation

An attacker who is the creator of any transaction (even a trivial one they created themselves) can:
1. Enumerate or guess approver IDs belonging to other transactions (IDs are sequential auto-increment integers).
2. Call `DELETE /transactions/{own_tx_id}/approvers/{victim_approver_id}`.
3. The `getCreatorsTransaction` check passes (they own `own_tx_id`), and the victim's approver record — along with its entire child subtree — is permanently soft-deleted.

This allows unauthorized removal of required approvers from any pending multi-signature transaction in the organization, bypassing the approval workflow entirely. It can allow transactions to proceed without the required approvals, or permanently disrupt the approval tree of critical transactions.

## Likelihood Explanation

- **Precondition**: The attacker must be an authenticated, verified organization user who has created at least one transaction. This is a normal user role with no elevated privileges.
- **Approver IDs** are sequential integers (auto-increment primary keys), making enumeration trivial.
- The attack requires a single crafted HTTP DELETE request with a mismatched `transactionId`/`id` pair.
- No rate limiting or anomaly detection is needed to bypass; the authorization logic itself is structurally broken.

## Recommendation

Apply the same cross-transaction ownership check that exists in `updateTransactionApprover` to `removeTransactionApprover`. Specifically, in the service method:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root of the approver tree.
2. Assert that `rootNode.transactionId === transactionId` (the route parameter), throwing `UnauthorizedException` if they differ.
3. Then call `getCreatorsTransaction(rootNode.transactionId, user)` to verify the caller owns that transaction.

This mirrors the pattern already implemented in `updateTransactionApprover` at lines 385–394. [4](#0-3) 

## Proof of Concept

**Setup:**
- Attacker (User A) is the creator of `Transaction #1` (their own transaction).
- Victim (User B) is the creator of `Transaction #2`, which has a required approver with `id = 99`.

**Attack:**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer <User A's JWT>
```

**Execution trace:**
1. Controller extracts `transactionId = 1`, `id = 99`.
2. `getCreatorsTransaction(1, userA)` — passes, User A owns Transaction #1.
3. `removeTransactionApprover(99)` — fetches approver with `id = 99` (belongs to Transaction #2), finds it exists, calls `removeNode(99)`, which recursively soft-deletes approver 99 and all its children.
4. Response: `true`.

Approver `id = 99` (and its subtree) belonging to Transaction #2 is now permanently soft-deleted, with no authorization check ever performed against Transaction #2 or User B. [6](#0-5) [2](#0-1)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
