### Title
Any Transaction Creator Can Delete Approvers from Transactions They Do Not Own via IDOR in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` for the authorization check while targeting a foreign approver `id`.

### Finding Description

**Root Cause — Authorization/Action Object Mismatch**

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```typescript
// Step 1: authorization — checks user is creator of :transactionId
await this.approversService.getCreatorsTransaction(transactionId, user);

// Step 2: action — deletes approver by :id, no cross-check performed
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

The service method `removeTransactionApprover` fetches the approver by `id` and soft-deletes it without ever verifying that `approver.transactionId` matches the `transactionId` that was used for the authorization check:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

**Contrast with the correctly-guarded `updateTransactionApprover`**

The `updateTransactionApprover` service method explicitly cross-checks that the approver's root transaction matches the URL parameter before proceeding:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The `removeTransactionApprover` path has no equivalent guard, making it the weaker of the two paths — directly analogous to the external report's `referralMint` vs `paidMint` pattern where a user selects the less-restricted path to bypass business logic.

**Exploit Flow**

1. Attacker (User A) creates transaction `T_A` — they are its creator.
2. Victim (User B) creates transaction `T_B` and adds approver record with database `id = 42`.
3. Attacker calls `DELETE /transactions/T_A/approvers/42`.
4. `getCreatorsTransaction(T_A, userA)` passes — attacker is creator of `T_A`. ✓
5. `removeTransactionApprover(42)` executes — approver 42 (belonging to `T_B`) is soft-deleted with no ownership check. ✓ [4](#0-3) 

### Impact Explanation

An attacker can silently remove required approvers from any transaction in the organization. In the multi-signature workflow, approvers gate whether a transaction advances to `WAITING_FOR_EXECUTION`. Removing an approver can:

- Reduce the effective threshold of a `ThresholdKey` approval tree, allowing a transaction to execute with fewer approvals than intended.
- Completely remove a mandatory approver, bypassing the organization's governance controls.
- Disrupt or sabotage another user's transaction workflow without their knowledge.

This is a direct unauthorized state change affecting the integrity of the multi-signature trust model. [5](#0-4) 

### Likelihood Explanation

- **Precondition**: The attacker must be an authenticated, verified organization member who has created at least one transaction. This is the normal state for any active user.
- **No privileged access required**: No admin role, no leaked credentials.
- **Approver IDs are sequential integers** (`PrimaryGeneratedColumn`), trivially enumerable by an attacker iterating `id` values.
- The attack requires a single crafted HTTP DELETE request. [1](#0-0) 

### Recommendation

In the `removeTransactionApprover` controller handler, after the creator check passes, verify that the targeted approver's root transaction matches the authorized `transactionId` before deletion. The simplest fix mirrors the pattern already used in `updateTransactionApprover`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);

  // ADD: verify the approver belongs to the authorized transaction
  const rootNode = await this.approversService.getRootNodeFromNode(id);
  if (!rootNode || rootNode.transactionId !== transactionId) {
    throw new UnauthorizedException('Approver does not belong to this transaction');
  }

  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and enforce the ownership check inside the service, consistent with how `updateTransactionApprover` handles it. [6](#0-5) 

### Proof of Concept

**Setup**:
- User A: authenticated, verified, creator of transaction `T_A` (id=1).
- User B: authenticated, verified, creator of transaction `T_B` (id=2) with approver record id=7.

**Attack**:
```http
DELETE /transactions/1/approvers/7
Authorization: Bearer <User_A_JWT>
```

**Expected (correct) behavior**: 403/401 — approver 7 does not belong to transaction 1.

**Actual behavior**: 200 OK — approver 7 is soft-deleted from `T_B`'s approval tree, bypassing User B's governance controls.

**Verification**: Query `transaction_approver` table — record with `id=7` now has `deletedAt` set, and `T_B`'s approval threshold is silently broken. [7](#0-6)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
