### Title
`cutoffAt` Deadline Not Enforced in Signature Upload Paths, Allowing Post-Deadline Signing

### Summary
The `cutoffAt` field on a transaction is documented as "The timestamp at which the transaction can no longer be signed by the signers." However, neither `uploadSignatureMaps` (the primary signing endpoint) nor `importSignatures` checks `cutoffAt` against the current time. Any authenticated signer can upload a valid signature after the creator's intended deadline, potentially pushing the transaction from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` and causing unintended on-chain execution.

### Finding Description

The `Transaction` entity stores a `cutoffAt` field: [1](#0-0) 

The database documentation confirms its purpose: [2](#0-1) 

**Path 1 — `POST /transactions/:id/signers` (`uploadSignatureMaps`)**

The entire validation before accepting a signature is in `validateTransactionStatus`: [3](#0-2) 

It checks only:
1. Status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`
2. The Hedera SDK transaction has not expired (`validStart + transactionValidDuration`, typically 180 s)

There is **no check** of `transaction.cutoffAt` against `Date.now()`. The `transaction` object loaded in `loadTransactionData` does contain `cutoffAt` (it is a column on the entity), but it is never read in the validation path. [4](#0-3) 

**Path 2 — `POST /transactions/signatures/import` (`importSignatures`)**

The same omission exists in the import path: [5](#0-4) 

Only status and SDK-level expiry are checked; `cutoffAt` is ignored.

**Consequence of a successful late signature**

After signatures are persisted, `processTransactionStatus` is called, which can transition the transaction to `WAITING_FOR_EXECUTION`: [6](#0-5) 

Once in `WAITING_FOR_EXECUTION`, the chain scheduler picks it up and submits it to the Hedera network: [7](#0-6) 

### Impact Explanation
A signer who deliberately delays their signature upload until after `cutoffAt` can unilaterally complete the required signing threshold and trigger on-chain execution of a transaction the creator believed was closed for signing. Depending on the transaction type (HBAR transfer, account update, file update, etc.) this can result in irreversible financial or state changes that the creator did not intend to allow after the cutoff.

### Likelihood Explanation
The attacker is any authenticated user who holds a required signer key on the transaction — a normal, non-privileged role. The exploit requires only a standard `POST` to `/transactions/:id/signers` with a valid signature map submitted after `cutoffAt`. No special tooling or elevated access is needed. The window of opportunity exists for the full remaining Hedera SDK validity window (up to 180 seconds after `validStart`), which can be well after `cutoffAt`.

### Recommendation
Add a `cutoffAt` guard inside `validateTransactionStatus` in `signers.service.ts` and the equivalent block in `importSignatures` in `transactions.service.ts`:

```typescript
// In validateTransactionStatus (signers.service.ts)
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  return ErrorCodes.TC; // or a new "Transaction cutoff passed" error code
}
```

Apply the same guard in `importSignatures` immediately after the status check at line 539 of `transactions.service.ts`.

### Proof of Concept

1. Creator creates a transaction with `cutoffAt = T+5min` and `validStart = T+10min`.
2. Required signers are Alice and Bob.
3. Alice signs immediately (before `cutoffAt`). Transaction remains `WAITING_FOR_SIGNATURES` (needs Bob too).
4. `cutoffAt` passes. Creator believes signing is now closed.
5. Bob calls `POST /transactions/:id/signers` with his valid signature map at `T+6min` (after `cutoffAt`, but before `validStart + 180s`).
6. `validateTransactionStatus` passes: status is `WAITING_FOR_SIGNATURES`, SDK transaction is not expired.
7. Bob's signature is persisted; `processTransactionStatus` transitions the transaction to `WAITING_FOR_EXECUTION`.
8. The chain scheduler executes the transaction on the Hedera network — after the creator's intended cutoff. [3](#0-2) [5](#0-4)

```typescript
  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
```

```typescript
  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
```
