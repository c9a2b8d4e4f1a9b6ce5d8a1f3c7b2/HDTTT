### Title
Transaction Creator Can Modify Approver Structure After Signing Has Started, Bypassing Multi-Approval Policy

### Summary
The `updateTransactionApprover` function in `ApproversService` allows the transaction creator to modify the approver list (change `userId`, `threshold`, or `listId`) at any transaction status, including `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION`. There is no status guard preventing structural changes after signing has begun. A malicious creator can reduce the required threshold or replace approvers after legitimate approvers have already signed, effectively circumventing the multi-approval policy the transaction was created with.

### Finding Description

The root cause is in `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, in `updateTransactionApprover` (lines 367–531). The function verifies only that the caller is the transaction creator via `getCreatorsTransaction` (line 394), but performs **no check on the transaction's current status** before mutating the approver structure. [1](#0-0) 

`getCreatorsTransaction` itself only validates creator ownership, not status: [2](#0-1) 

The threshold update path (lines 467–488) allows the creator to lower the threshold to any value ≥ 1 and ≤ current approver count, with no status guard: [3](#0-2) 

The userId update path (lines 489–517) allows replacing an approver who has not yet signed with a colluding user, clearing any existing signature in the process: [4](#0-3) 

The unit tests confirm this is the intended behavior — the test fixture uses `TransactionStatus.WAITING_FOR_EXECUTION` and the update succeeds without any status rejection: [5](#0-4) 

The controller exposes this via an authenticated `PATCH /transactions/:transactionId/approvers/:id` endpoint, guarded only by JWT and verified-user guards — no admin guard, no status guard: [6](#0-5) 

**Exploit flow:**
1. Attacker (as a normal authenticated user) creates a transaction and sets up an approver structure requiring threshold=3 (3-of-5 approvals).
2. Two legitimate approvers sign the transaction; it moves to `WAITING_FOR_SIGNATURES`.
3. Attacker calls `PATCH /transactions/:id/approvers/:treeId` with `{ "threshold": 1 }`.
4. The threshold is reduced to 1. The existing single approval already satisfies the new threshold.
5. The chain service evaluates the approver tree, finds the threshold met, and proceeds to execution — without the intended 3-of-5 approval.

Similarly, the attacker can replace an unsigned approver slot with a colluding user's ID (clearing the signature field), then have that colluding user sign, bypassing the original approver's participation requirement.

### Impact Explanation

A transaction creator can unilaterally reduce the approval threshold or swap in colluding approvers after signing has started. This breaks the multi-approval integrity guarantee that the approver system is designed to enforce. In the context of this tool — which is used to manage Hedera governance transactions (system file updates, node operations, account key rotations) — bypassing the multi-approval policy means a single actor can push through high-impact Hedera network transactions that were supposed to require organizational consensus. [7](#0-6) 

### Likelihood Explanation

The attacker precondition is only that they are an authenticated user who created a transaction — a role reachable by any registered user with no privileged access. The attack requires no leaked credentials, no admin keys, and no external dependencies. The `PATCH` endpoint is publicly documented and reachable via the standard REST API. The only timing requirement is that the creator acts before the chain service executes the transaction.

### Recommendation

Add a transaction status check at the top of `updateTransactionApprover` (and `createTransactionApprovers`) that rejects modifications when the transaction is in any status other than `NEW` (or a designated "draft" state). For example:

```typescript
if (transaction.status !== TransactionStatus.NEW) {
  throw new BadRequestException('Approver structure cannot be modified after signing has started');
}
```

This mirrors the pattern already used in `approveTransaction`, which correctly checks status before accepting a signature: [8](#0-7) 

### Proof of Concept

**Preconditions:** Two registered users (Alice = creator, Bob and Carol = approvers), a transaction in `WAITING_FOR_SIGNATURES` with threshold=2.

1. Alice creates a transaction via `POST /transactions` and adds Bob and Carol as approvers with threshold=2 via `POST /transactions/:id/approvers`.
2. Bob signs via `POST /transactions/:id/approvers/approve` — transaction now has 1 of 2 required approvals.
3. Alice sends:
   ```
   PATCH /transactions/:id/approvers/:treeNodeId
   Authorization: Bearer <alice_jwt>
   { "threshold": 1 }
   ```
4. Server responds 200 OK. Threshold is now 1.
5. The chain service evaluates the approver tree: threshold=1, Bob has approved → condition met.
6. Transaction proceeds to execution without Carol's approval.

**Expected (correct) behavior:** Step 3 should return 400 Bad Request because the transaction is no longer in a modifiable state.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
