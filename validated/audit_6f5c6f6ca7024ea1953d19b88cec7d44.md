### Title
Transaction Creator Can Modify Approval Threshold and Approvers After Votes Are Cast, Bypassing Multi-Signature Governance

### Summary
The Hedera Transaction Tool's approver system allows a transaction creator to modify the approval threshold, remove approvers, or replace approvers at any point during a transaction's lifecycle — including after approvers have already cast their votes. Because neither `updateTransactionApprover` nor `removeTransactionApprover` enforce any transaction-status guard, a malicious creator can retroactively lower the threshold to match already-received approvals, or purge rejecting approvers, causing a transaction to advance to execution despite failing the originally intended approval requirement.

### Finding Description

**Root cause:** `ApproversService.updateTransactionApprover` and `ApproversService.removeTransactionApprover` (and the controller endpoints that call them) only verify that the requesting user is the transaction creator via `getCreatorsTransaction`. They perform no check on the current `TransactionStatus`.

`getCreatorsTransaction` (the sole authorization gate for both mutation paths): [1](#0-0) 

It checks only creator ownership — never `transaction.status`.

The `PATCH /:id` controller handler delegates directly to `updateTransactionApprover`: [2](#0-1) 

Inside `updateTransactionApprover`, the threshold branch updates the stored threshold with no status guard: [3](#0-2) 

The `DELETE /:id` controller handler also only calls `getCreatorsTransaction` before removing the approver: [4](#0-3) 

`removeTransactionApprover` itself performs no status check: [5](#0-4) 

Additionally, when an approver is moved out of a tree (`listId → null`), the parent's threshold is silently auto-reduced if it would otherwise exceed the remaining child count: [6](#0-5) 

This means removing a rejecting approver from a threshold tree can automatically lower the required approval count.

**Exploit flow:**

1. Creator Alice creates a transaction and sets up a threshold approver tree: threshold = 3, approvers = [Bob, Carol, Dave, Eve, Frank].
2. Bob and Carol approve; Dave, Eve, and Frank reject.
3. Alice sends `PATCH /transactions/{id}/approvers/{threshold-node-id}` with body `{"threshold": 2}`.
4. The threshold is now 2; Bob and Carol's existing approvals satisfy it.
5. The chain service processes the transaction as fully approved and submits it to Hedera — despite the original governance intent requiring 3 approvals.

Alternatively, Alice can `DELETE` Dave, Eve, and Frank's approver records (removing the rejections entirely), leaving only Bob and Carol, and then lower the threshold to 2.

The `userId` update path also resets the `approved`, `signature`, and `userKeyId` fields of the replaced approver: [7](#0-6) 

This lets Alice swap a rejecting approver for a new user, erasing the rejection.

### Impact Explanation

The approval/threshold system is the primary governance control preventing a single creator from unilaterally executing a transaction that requires organizational consensus. Bypassing it allows:

- A transaction creator to execute any transaction they created, regardless of how many approvers rejected it.
- Complete circumvention of multi-party authorization workflows — the core security guarantee of Organization Mode.
- Irreversible on-chain effects (e.g., account key changes, fund transfers, node operations) that cannot be undone once submitted to Hedera.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The attack requires only standard API calls (`PATCH` or `DELETE` on approver endpoints) that are already part of the documented workflow. No leaked credentials, no admin access, and no race conditions are required. Any creator who faces a rejection can trivially execute this manipulation.

### Recommendation

Add a transaction-status guard at the start of both `updateTransactionApprover` and `removeTransactionApprover` (and/or in `getCreatorsTransaction` when called from these paths). Reject modifications when the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status and at least one approver has already cast a vote. A minimal fix:

```typescript
// In updateTransactionApprover and removeTransactionApprover,
// after fetching the transaction:
if (
  transaction.status === TransactionStatus.WAITING_FOR_SIGNATURES ||
  transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
) {
  const hasVotes = approvers.some(a => a.approved !== null || a.signature !== null);
  if (hasVotes) throw new BadRequestException('Cannot modify approvers after voting has begun');
}
```

Alternatively, lock the approver structure entirely once the transaction leaves `NEW` status, consistent with how `transactionBytes` is treated as immutable after creation.

### Proof of Concept

**Preconditions:** Attacker is a registered, verified user (Alice) in an organization. Bob and Carol are also registered users.

1. Alice creates a transaction via `POST /transactions` → receives `transactionId = 42`.
2. Alice creates a threshold approver tree via `POST /transactions/42/approvers`:
   ```json
   { "approversArray": [{ "threshold": 3, "approvers": [
       {"userId": 2}, {"userId": 3}, {"userId": 4}
   ]}] }
   ```
   Threshold node gets `id = 10`.
3. Bob (userId=2) approves via `POST /transactions/42/approvers/approve`.
4. Carol (userId=3) approves via `POST /transactions/42/approvers/approve`.
5. Dave (userId=4) rejects via `POST /transactions/42/approvers/approve` with `approved: false`.
6. Alice sends:
   ```
   PATCH /transactions/42/approvers/10
   Body: { "threshold": 2 }
   ```
   **Expected:** Error — voting has already occurred.
   **Actual:** HTTP 200; threshold updated to 2.
7. The chain service now sees 2 approvals satisfying threshold=2 and advances the transaction to `WAITING_FOR_EXECUTION`, then submits it to Hedera — bypassing Dave's rejection and the original 3-of-3 requirement.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
