### Title
Transaction Creator Can Manipulate Approval Outcome by Modifying Approver Structure After Votes Are Cast

### Summary
The transaction approval workflow in the Hedera Transaction Tool allows the transaction creator to add, remove, or update approvers (including lowering thresholds) at any point during the transaction lifecycle — including after approvers have already cast their approval or rejection. Because `getCreatorsTransaction` performs no transaction-status check, a malicious creator can nullify a rejection by deleting the rejecting approver, or lower the required threshold after partial approvals are in, forcing a transaction through without the originally required consent.

### Finding Description

**Vulnerability class:** State transition — missing status guard on approver mutation endpoints.

**Root cause:** `getCreatorsTransaction` is the sole authorization gate used by all three approver-mutation paths. It checks only that the caller is the transaction creator; it never inspects the transaction's current status. [1](#0-0) 

This function is called unconditionally by:

1. `createTransactionApprovers` — adds new approvers [2](#0-1) 

2. `updateTransactionApprover` — changes threshold or reassigns approver user [3](#0-2) 

3. `removeTransactionApprover` — called from the controller after `getCreatorsTransaction` [4](#0-3) 

The `removeTransactionApprover` service method itself performs no authorization or status check at all: [5](#0-4) 

**Exploit paths:**

**Path A — Nullify a rejection (remove rejecting approver):**
- Transaction has 3 approvers, threshold = 2.
- Approvers A and B approve (`approved=true`), approver C rejects (`approved=false`).
- Creator calls `DELETE /transactions/:id/approvers/:cId`.
- Approver C's record is soft-deleted. The approval tree now shows 2-of-2 approved.
- The chain service's `processTransactionStatus` re-evaluates and advances the transaction to `WAITING_FOR_EXECUTION`.

**Path B — Lower threshold after partial approval:**
- Transaction has 3 approvers under a tree node with threshold = 3 (unanimous required).
- Approvers A and B approve; C has not yet.
- Creator calls `PATCH /transactions/:id/approvers/:treeId` with `{ threshold: 2 }`.
- The threshold update is accepted with no status check: [6](#0-5) 
- The tree now requires only 2-of-3; the transaction advances without C's approval.

**Path C — Add a new approver who inherits an existing approval:**
When a new approver is added for a `userId` that already has an approval record for the transaction, the existing `signature`, `userKeyId`, and `approved` values are copied into the new approver record: [7](#0-6) 
This allows a creator to re-insert a previously removed approver with their approval pre-populated, or to restructure the tree to manufacture a passing state.

### Impact Explanation

A transaction creator (a normal authenticated user) can unilaterally override the approval decision of other organization members after those decisions have been recorded. This breaks the core integrity guarantee of the multi-approver workflow: that a transaction cannot proceed without the required number of independent approvals. Concretely:

- A rejected transaction can be forced into `WAITING_FOR_EXECUTION` by removing the rejecting approver.
- A transaction requiring unanimous consent can be executed with a subset of approvals by lowering the threshold.
- This can result in unauthorized Hedera transactions being submitted on behalf of the organization (fund transfers, account updates, file changes, etc.).

### Likelihood Explanation

- **Attacker profile:** The transaction creator — a normal authenticated user with no elevated privileges.
- **Preconditions:** The attacker must have created the transaction (standard workflow) and at least one approver must have cast a rejection or the threshold must be higher than the number of approvals received.
- **Effort:** Two standard API calls (one to approve as a colluding approver, one to delete the rejecting approver). No cryptographic bypass, no race condition, no special tooling required.
- **Detection difficulty:** The soft-delete of an approver record leaves a `deletedAt` timestamp, but there is no audit log or alert triggered when an approver is removed after votes are cast.

### Recommendation

Add a transaction-status guard to all three approver-mutation paths. Approver structure modifications should only be permitted when the transaction is in a pre-approval state. Since `NEW` is currently marked unused and transactions begin at `WAITING_FOR_SIGNATURES`, the practical fix is to reject mutations once any approver has recorded a signature:

In `getCreatorsTransaction`, or as a dedicated guard called before each mutation, add:

```typescript
// Reject structural changes if any approver has already voted
const existingApprovers = await this.getApproversByTransactionId(transactionId, undefined, entityManager);
const anyVoteCast = existingApprovers.some(a => a.signature !== null && a.signature !== undefined);
if (anyVoteCast) {
  throw new BadRequestException('Cannot modify approver structure after votes have been cast');
}
```

Alternatively, lock the approver structure once the transaction reaches `WAITING_FOR_EXECUTION` status, which is the state analogous to the "finalizing" state in the external report.

### Proof of Concept

**Setup:**
- Organization with creator (User A) and two approvers (User B, User C).
- Creator creates a transaction with approver tree: threshold=2, approvers=[B, C].
- Transaction is in `WAITING_FOR_SIGNATURES`.

**Steps:**
1. User B calls `POST /transactions/1/approvers/approve` with `approved=true` and a valid signature. ✓
2. User C calls `POST /transactions/1/approvers/approve` with `approved=false` (rejection). ✓
3. Transaction remains in `WAITING_FOR_SIGNATURES` (threshold not met due to rejection).
4. User A (creator) calls `DELETE /transactions/1/approvers/<C_approver_id>`.
   - `getCreatorsTransaction` passes (A is creator, no status check).
   - `removeNode` soft-deletes C's approver record.
5. Chain service's `processTransactionStatus` runs (every 10 seconds): [8](#0-7) 
6. With only approver B remaining (approved=true), threshold=2 is now impossible to evaluate correctly — or if the tree node is also removed, the transaction is re-evaluated with no blocking rejection.
7. Transaction advances to `WAITING_FOR_EXECUTION` and is submitted to the Hedera network without User C's consent.

**Expected (correct) behavior:** `DELETE /transactions/1/approvers/:id` should return an error when any approver has already cast a vote.

**Actual behavior:** The deletion succeeds, the rejection is erased, and the transaction proceeds.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
