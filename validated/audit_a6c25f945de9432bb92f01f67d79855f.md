Audit Report

## Title
Stale Approval State Carried Over When Approver Is Re-Added to Transaction Approver Tree

## Summary
In `createTransactionApprovers`, when a new approver node is created for a user who already has a non-deleted approver record on the same transaction, the code unconditionally copies the prior `signature`, `userKeyId`, and `approved` fields into the new record. The duplicate-check guard (`isNode`) only blocks re-adding a user with the **exact same** `listId`, so adding the same user under a different parent node bypasses the guard entirely. Because `getCreatorsTransaction` performs no transaction-status check, the creator can restructure the approver tree at any point — including after approvals have already been collected — causing stale approvals to satisfy a threshold that was never legitimately met.

## Finding Description

**Root cause 1 — Stale approval copy (lines 318–329):**

Inside `createTransactionApprovers` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, after all validation passes, the code queries for any existing non-deleted approver records for the same `userId` on the same `transactionId` and copies their approval state verbatim into the new record:

```typescript
if (typeof dtoApprover.userId === 'number') {
  const userApproverRecords = await this.getApproversByTransactionId(
    transactionId,
    dtoApprover.userId,
    transactionalEntityManager,
  );

  if (userApproverRecords.length > 0) {
    data.signature = userApproverRecords[0].signature;
    data.userKeyId = userApproverRecords[0].userKeyId;
    data.approved = userApproverRecords[0].approved;
  }
}

**Root cause 2 — `isNode` only checks exact `listId` match (lines 647–666):**

The duplicate-check guard matches on `listId` exactly. A root-level approver record for User A has `listId = null`; a new child record under a threshold node has `listId = thresholdNodeId`. These are different values, so `isNode` returns `false` and the guard does not fire:

```typescript

    userId: typeof approver.userId === 'number' ? approver.userId : null,

**Root cause 3 — `getCreatorsTransaction` has no status check (lines 624–644):**

The only authorization gate for approver creation checks only that the transaction exists and that the caller is its creator. No check is made on `transaction.status`, so the creator can add new approver nodes at any lifecycle stage, including after approvals have been collected:

```typescript
async getCreatorsTransaction(transactionId, user, entityManager?) {

  if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

  return transaction;

**`getApproversByTransactionId` traverses the full tree (lines 85–108):**

The recursive CTE used to find existing approver records for a user traverses the entire approver tree, so it will find User A's root-level record even when called during the creation of a child node: [4](#0-3) 

## Impact Explanation

A malicious transaction creator can bypass the intended multi-approver threshold by restructuring the approver tree after partial approvals are collected. Stale approvals from a prior tree position are silently counted in the new position, allowing the transaction to advance to `WAITING_FOR_EXECUTION` without the required number of genuine approvals under the new structure. This is a direct integrity failure in the approval/governance workflow: the on-chain transaction can be submitted with fewer real approvals than the policy requires.

Additionally, `emitTransactionStatusUpdate` is called immediately after `createTransactionApprovers` completes (line 358), which triggers the downstream status-evaluation pipeline with the already-stale approvals in place. [5](#0-4) 

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The exploit requires only standard API calls (`POST /transactions/:id/approvers`) that are already available to the creator. The creator has a direct incentive to push their transaction through approval. No external access, leaked credentials, or privileged roles are needed.

## Recommendation

1. **Remove the stale-approval copy block entirely (lines 318–329).** There is no legitimate reason to inherit approval state when creating a new approver node. A new node should always start with `signature`, `userKeyId`, and `approved` as `null`/`undefined`.

2. **Add a transaction-status guard in `getCreatorsTransaction` (or at the call site in `createTransactionApprovers`).** Approver-tree modifications should be rejected once the transaction has moved past `NEW` or `WAITING_FOR_SIGNATURES` (or whatever the intended editable window is). For example:
   ```typescript
   if (![TransactionStatus.NEW, TransactionStatus.WAITING_FOR_SIGNATURES].includes(transaction.status))
     throw new BadRequestException('Cannot modify approvers after approvals have started');
   ```

3. **Optionally strengthen `isNode`** to check for duplicate `userId` across the entire transaction tree (regardless of `listId`), or at minimum document clearly that the same user may appear in multiple positions and that approval state must never be inherited.

## Proof of Concept

1. Creator creates transaction T with root-level approvers `[A, B]`, threshold 2-of-2.
2. User A calls `POST /transactions/T/approvers/approve` — their `signature`, `userKeyId`, `approved=true` are stored in their root-level `TransactionApprover` record (`listId = null`).
3. Creator calls `POST /transactions/T/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": A }, { "userId": C }] }] }
   ```
4. `isNode` is called for the new child node for User A with `listId = <new threshold node id>`. The existing record for A has `listId = null`. The `where` clause does not match → `isNode` returns `false` → no duplicate error.
5. `getApproversByTransactionId(T, A.userId)` finds A's existing root-level record (`approved=true`, `signature=<sig>`, `userKeyId=<kid>`).
6. Lines 325–329 copy `signature`, `userKeyId`, `approved=true` into the new child record.
7. The new threshold node (1-of-2) now has 1 approval satisfied by the stale vote — without User A re-approving under the new structure.
8. `emitTransactionStatusUpdate` fires; the status evaluator sees the threshold node satisfied and advances the transaction to `WAITING_FOR_EXECUTION` without User B ever approving. [1](#0-0) [2](#0-1) [3](#0-2)
