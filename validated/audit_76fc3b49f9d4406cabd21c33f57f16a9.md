Audit Report

## Title
Transaction Creator Can Add Themselves as Approver and Self-Approve Their Own Transaction

## Summary
`ApproversService.createTransactionApprovers` permits the transaction creator to register their own `userId` as an approver because the only per-entry check is that the user exists in the database. `approveTransaction` then accepts the creator's approval because it only verifies the caller appears in the approvers list. Together, these two missing guards allow a creator to unilaterally satisfy the approval workflow without any independent review.

## Finding Description

**Root cause 1 — `createTransactionApprovers` (no self-approver guard):**

After confirming the caller is the creator via `getCreatorsTransaction` (line 239), the function iterates over `dto.approversArray`. For each entry carrying a `userId`, the only validation is a database existence check: [1](#0-0) 

There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. A creator who submits their own `userId` passes every check and is inserted as a valid approver.

**Root cause 2 — `approveTransaction` (no creator-is-approver guard):**

`approveTransaction` calls `getVerifiedApproversByTransactionId`, which grants the creator unconditional read access (lines 142–149). It then filters the approvers list for the caller's `userId`: [2](#0-1) 

If the creator previously added themselves, `userApprovers.length > 0` is satisfied and the approval is recorded. There is no check that `user.id !== transaction.creatorKey.userId`.

**Controller exposure:**

Both endpoints are guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — the default for any authenticated, verified user: [3](#0-2) 

## Impact Explanation
The approval workflow is the primary governance control in Organization mode. It is designed to require a party other than the creator to authorize sensitive transactions (e.g., large HBAR transfers, `ACCOUNT_UPDATE`, `NODE_CREATE`, `NODE_UPDATE`). A creator who self-approves defeats this control entirely, enabling unilateral execution of any transaction type the system supports without any independent review.

## Likelihood Explanation
The attacker only needs a valid organization account and the ability to create transactions — the default capability of any authenticated user. No special privileges, leaked credentials, or race conditions are required. The two API calls (`POST /transactions/:id/approvers` and `POST /transactions/:id/approvers/approve`) are standard REST operations. Any malicious or compromised creator can exploit this on every transaction they create.

## Recommendation
1. **In `createTransactionApprovers`**: After the user-existence check (line 276), add:
   ```ts
   if (dtoApprover.userId === user.id)
     throw new Error('Creator cannot add themselves as an approver');
   ```
   Apply the same guard recursively for nested `approvers` entries.

2. **In `approveTransaction`**: After fetching the transaction, add:
   ```ts
   if (transaction.creatorKey?.userId === user.id)
     throw new UnauthorizedException('Creator cannot approve their own transaction');
   ```
   This provides defense-in-depth even if the first guard is bypassed.

3. **In `updateTransactionApprover`**: Apply the same self-approver guard when a `userId` update is processed (lines 489–517), since the update path also only checks user existence. [4](#0-3) 

## Proof of Concept

```
# Step 1 – Creator (userId = X) creates a transaction (transactionId = T)
POST /transactions
Authorization: Bearer <creator_jwt>
{ ...transaction payload... }

# Step 2 – Creator adds themselves as the sole approver
POST /transactions/T/approvers
Authorization: Bearer <creator_jwt>
{ "approversArray": [{ "userId": X }] }
# → 201 Created; createTransactionApprovers only checks user X exists — passes.

# Step 3 – Creator approves their own transaction
POST /transactions/T/approvers/approve
Authorization: Bearer <creator_jwt>
{ "userKeyId": <keyId>, "signature": <validSig>, "approved": true }
# → approveTransaction finds userId X in approvers list → records approval.

# Result: approval threshold satisfied; transaction advances to
# WAITING_FOR_EXECUTION with no independent party having reviewed it.
```

@ApiTags('Transaction Approvers')
@Controller('transactions/:transactionId?/approvers')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
@Serialize(TransactionApproverDto)
export class ApproversController {
  constructor(private approversService: ApproversService) {}

  /* Create transaction approvers for the given transaction id with the user ids */
  @ApiOperation({
    summary: 'Creates transaction approvers',
    description: 'Create transaction approvers for the given transaction with the provided data.',
  })
  @ApiResponse({
    status: 201,
    type: [TransactionApproverDto],
  })
  @Post()
  createTransactionApprovers(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Body() body: CreateTransactionApproversArrayDto,
  ): Promise<TransactionApprover[]> {
    return this.approversService.createTransactionApprovers(user, transactionId, body);
  }

  /* Approves a transaction */
  @ApiOperation({
    summary: 'Approves a transaction',
    description: 'Approves the transaction with the given transaction id.',
  })
  @ApiResponse({
    status: 200,
    type: Boolean,
  })
  @Post('/approve')
  @OnlyOwnerKey<ApproverChoiceDto>('userKeyId')
  approveTransaction(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Body() body: ApproverChoiceDto,
  ): Promise<boolean> {
    return this.approversService.approveTransaction(body, transactionId, user);
  }
