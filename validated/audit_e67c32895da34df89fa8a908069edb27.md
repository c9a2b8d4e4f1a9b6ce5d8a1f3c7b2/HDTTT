### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Cross-Transaction Signature Replay

### Summary
The `POST /transactions/:transactionId/signers` endpoint (`SignersService.uploadSignatureMaps`) verifies that submitted public keys belong to the authenticated user but does **not** cryptographically verify that the signature bytes in the submitted map are valid for the target transaction. An authenticated user can replay a signature map from any transaction they previously signed onto a different transaction, corrupting its bytes, being falsely recorded as a signer, and potentially advancing it to `WAITING_FOR_EXECUTION` with an invalid signature that will be rejected by the Hedera network.

### Finding Description

**Root cause — missing `validateSignature` call in `processTransactionSignatures`:**

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, `processTransactionSignatures` (lines 217–267) iterates over the submitted `SignatureMap`, checks that each public key is registered to the calling user via `userKeyMap`, then unconditionally calls `sdkTransaction.addSignature(publicKey, map)`:

```typescript
// Look up key (raw first, then DER)
let userKey = userKeyMap.get(raw);
if (!userKey) {
  userKey = userKeyMap.get(publicKey.toStringDer());
}
if (!userKey) throw new Error(ErrorCodes.PNY);   // ← only ownership check

// Only add the signature once per unique key
sdkTransaction = sdkTransaction.addSignature(publicKey, map); // ← no crypto verify
``` [1](#0-0) 

The Hedera SDK's `addSignature` does **not** verify the signature bytes; it blindly appends them to the transaction body. Cryptographic verification only happens at the Hedera network level when the transaction is submitted.

**Contrast — `importSignatures` does verify:**

The sibling endpoint `POST /transactions/signatures/import` (`TransactionsService.importSignatures`, lines 545–549) calls `validateSignature` before adding any signature:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);
``` [2](#0-1) 

`validateSignature` (lines 213–248 of `back-end/libs/common/src/utils/sdk/transaction.ts`) explicitly calls `publicKey.verify(bodyBytes, signature)` and throws on failure:

```typescript
const signatureValid = publicKey.verify(bodyBytes, signature);
if (signatureValid) {
  signerPublicKeys.push(publicKey);
} else {
  throw new Error('Invalid signature');
}
``` [3](#0-2) 

The `/signers` endpoint has no equivalent call.

**Exploit flow:**

1. Attacker (legitimate org user) signs Transaction A (e.g., an `AccountCreateTransaction`) with their private key and submits the signature map normally.
2. Attacker extracts the `SignatureMap` from Transaction A (the map contains their public key → signature bytes over Transaction A's body).
3. Attacker submits that same `SignatureMap` to `POST /transactions/:transactionId_B/signers` targeting Transaction B (a different transaction, e.g., a `FreezeTransaction` or `FileUpdateTransaction` they are required to sign).
4. `processTransactionSignatures` confirms the public key is in `userKeyMap` → passes the ownership check.
5. `addSignature` appends the replayed (invalid) bytes to Transaction B's body.
6. The attacker is inserted into `TransactionSigner` as a signer of Transaction B.
7. `processTransactionStatus` is called; if the signing threshold is now met, Transaction B advances to `WAITING_FOR_EXECUTION`.
8. The chain service submits Transaction B to Hedera → network returns `INVALID_SIGNATURE` → Transaction B is marked `FAILED`.

### Impact Explanation

- **Transaction corruption**: Transaction B's stored `transactionBytes` are overwritten with bytes containing an invalid signature, making it permanently unexecutable on-chain.
- **False signer record**: The attacker is recorded as having signed Transaction B without ever producing a valid signature for it, corrupting the audit trail.
- **Premature status advancement**: If the replayed signature satisfies the threshold, the transaction is moved to `WAITING_FOR_EXECUTION` and submitted to Hedera, where it fails. Legitimate signers cannot recover the transaction.
- **Denial of service on targeted transactions**: Any transaction the attacker has access to sign can be permanently broken.

### Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated, verified org user with at least one registered public key — no admin or privileged role required.
- **Signature material is freely available**: The attacker obtains a valid signature map simply by signing any transaction they are legitimately involved in. The Hedera SDK makes it trivial to extract a `SignatureMap` from a signed transaction.
- **No rate limiting or anomaly detection** is present on the `/signers` endpoint.
- The asymmetry between `/signers` (no crypto verify) and `/signatures/import` (crypto verify) suggests this was an oversight rather than intentional design.

### Recommendation

Add a cryptographic signature verification step inside `processTransactionSignatures`, mirroring what `importSignatures` already does. Before calling `addSignature`, verify each signature byte against the target transaction's body:

```typescript
// After confirming the key belongs to the user:
const { data: _, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMP); // Invalid signature, not matching public key
```

Alternatively, extract the shared verification logic into a common helper and call it from both `processTransactionSignatures` and `importSignatures`.

### Proof of Concept

**Setup**: Two transactions exist in the system — Transaction A (ID=1) and Transaction B (ID=2). The attacker's public key is registered and is required to sign both.

**Steps**:
```
1. Sign Transaction A locally:
   sdkTxA = AccountCreateTransaction.fromBytes(txA.transactionBytes)
   await sdkTxA.sign(attackerPrivateKey)
   signatureMapA = sdkTxA.getSignatures()   // valid sig over txA body

2. POST /transactions/1/signers  { signatureMap: signatureMapA }
   → 201 OK (legitimate signing of Transaction A)

3. POST /transactions/2/signers  { signatureMap: signatureMapA }
   → 201 OK  ← BUG: replayed sig from txA accepted for txB

4. Query DB: TransactionSigner row exists for (transactionId=2, userKeyId=attacker)
   Transaction B's transactionBytes now contain invalid signature bytes.

5. If threshold met → Transaction B status = WAITING_FOR_EXECUTION
   Chain service submits to Hedera → INVALID_SIGNATURE → status = FAILED
```

**Expected (correct) behavior**: Step 3 should return `400` with `ErrorCodes.ISNMP` because the signature bytes in `signatureMapA` do not verify against Transaction B's body bytes.
