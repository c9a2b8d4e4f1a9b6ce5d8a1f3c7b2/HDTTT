### Title
Creator Can Force Execution of a Transaction Still Awaiting Signatures, Causing Premature Failure

### Summary
`TransactionsService.executeTransaction()` in the API service does not verify that the transaction has reached `WAITING_FOR_EXECUTION` status before submitting it to the Hedera network. A transaction creator who set `isManual = true` can call the execute endpoint while the transaction is still in `WAITING_FOR_SIGNATURES` status — before co-signers have provided their signatures — causing the Hedera network to reject it and permanently mark it `FAILED`, destroying the multi-signature workflow.

### Finding Description

**Root cause — missing status precondition check**

`TransactionsService.executeTransaction()` enforces only two conditions before dispatching:

1. `transaction.isManual === true`
2. `transaction.validStart` timing (past vs. future)

It never checks `transaction.status`. [1](#0-0) 

```
async executeTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);   // only isManual checked
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      ...
    } else {
      await this.executeService.executeTransaction(transaction);  // no status guard
    }
    return true;
  }
```

`getTransactionForCreator` only verifies the caller is the creator; it does not inspect `status`. [2](#0-1) 

By contrast, the scheduler pipeline that handles automatic execution explicitly filters for `WAITING_FOR_EXECUTION` before dispatching: [3](#0-2) 

The API path skips this guard entirely.

**Exploit path**

1. Attacker (creator) creates a transaction requiring N signatures (`isManual: true`).
2. Transaction enters `WAITING_FOR_SIGNATURES`; co-signers have not yet signed.
3. Creator calls `POST /transactions/:id/execute` (the endpoint backed by `executeTransaction`).
4. The only checks pass: `isManual === true`, `validStart` is in the past.
5. `executeService.executeTransaction(transaction)` submits the under-signed transaction bytes to the Hedera network.
6. Hedera rejects it (e.g., `INVALID_SIGNATURE`).
7. The transaction is permanently set to `FAILED`; co-signers' pending work is destroyed.

The unit test suite confirms the expected guard is absent — the test that should enforce key-ownership on the approve path is explicitly skipped, and no status-check test exists for `executeTransaction` with `WAITING_FOR_SIGNATURES`: [4](#0-3) 

(All passing tests supply `status: TransactionStatus.WAITING_FOR_EXECUTION`; no test covers the `WAITING_FOR_SIGNATURES` case.)

### Impact Explanation

A transaction creator can unilaterally abort any multi-signature workflow at any time by forcing premature execution. The resulting `FAILED` terminal state is unrecoverable — the transaction cannot be re-signed or re-executed. This permanently corrupts the state of any in-flight organization transaction and can block time-sensitive Hedera operations (account updates, file operations, token management) that depend on multi-party signing.

### Likelihood Explanation

The attacker needs only a valid authenticated session and creator ownership of a manual transaction — both are normal, unprivileged conditions for any registered user. No special role, leaked credential, or race condition is required. The API endpoint is reachable by any creator of a `isManual` transaction.

### Recommendation

Add an explicit status guard at the top of `executeTransaction` before any dispatch logic:

```typescript
async executeTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    // ADD: reject if signatures are still being collected
    if (transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION) {
      throw new BadRequestException(ErrorCodes.OTIP); // or a dedicated error code
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      ...
    } else {
      await this.executeService.executeTransaction(transaction);
    }
    return true;
  }
```

This mirrors the guard already present in `archiveTransaction`: [5](#0-4) 

### Proof of Concept

**Preconditions**: Two registered users, User A (creator) and User B (required co-signer). Transaction requires both keys.

1. User A creates a transaction: `POST /transactions` with `isManual: true`, requiring User B's signature.
2. Server responds with `status: WAITING_FOR_SIGNATURES`. User B has not signed.
3. User A immediately calls: `POST /transactions/:id/execute`.
4. `executeTransaction` passes both checks (`isManual === true`, `validStart` in the past).
5. `executeService.executeTransaction(transaction)` submits the transaction with only User A's signature.
6. Hedera network returns `INVALID_SIGNATURE`.
7. Transaction status is set to `FAILED` — permanently terminal.
8. User B's pending signature action is now meaningless; the transaction cannot be recovered.

**Expected result (correct behavior)**: Step 3 should return `400 Bad Request` because `status !== WAITING_FOR_EXECUTION`.
**Actual result**: The transaction is submitted and fails on-chain.
