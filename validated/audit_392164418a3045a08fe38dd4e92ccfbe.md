### Title
Any Transaction Creator Can Remove Approvers from Another User's Transaction via Mismatched Authorization Check

### Summary
In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `removeTransactionApprover` endpoint verifies that the caller is the creator of the `transactionId` URL parameter, but then removes the approver identified by the separate `id` parameter without verifying that approver actually belongs to that transaction. Any authenticated user who owns at least one transaction can exploit this to remove approvers from any other user's transaction.

### Finding Description

The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController` performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the transaction identified by `transactionId`.
2. `removeTransactionApprover(id)` — removes the approver identified by `id` with **no check** that this approver belongs to `transactionId`. [1](#0-0) 

The service-level `removeTransactionApprover` method simply fetches the approver by `id` and deletes it, with no cross-reference to the `transactionId` that was used for the authorization check: [2](#0-1) 

By contrast, the `updateTransactionApprover` service method correctly validates that the approver's root transaction matches the `transactionId` parameter before proceeding: [3](#0-2) 

The `getCreatorsTransaction` check only enforces ownership of the `transactionId` parameter — it says nothing about the `id` (approver) parameter: [4](#0-3) 

**Exploit flow:**
1. Attacker (Alice) creates Transaction A — she is its creator.
2. Victim (Bob) creates Transaction B and adds approvers to it (e.g., approver with `id = 99`).
3. Alice calls `DELETE /transactions/{A.id}/approvers/99`.
4. `getCreatorsTransaction(A.id, alice)` passes — Alice is the creator of Transaction A.
5. `removeTransactionApprover(99)` deletes Bob's approver from Transaction B without any ownership check.

### Impact Explanation

An attacker who is the creator of any transaction can silently remove approvers from any other user's transaction. This breaks the multi-signature approval workflow — a core security invariant of the platform. Removing required approvers can:
- Allow a transaction to advance past the approval gate with fewer approvals than intended.
- Permanently corrupt the approval tree of another user's transaction, causing it to become unexecutable or to execute without required consent. [5](#0-4) 

### Likelihood Explanation

The attacker precondition is minimal: any registered, verified user who has created at least one transaction. No admin access, no leaked credentials, and no privileged role is required. The attack is a single crafted HTTP DELETE request with a valid JWT token. [6](#0-5) 

### Recommendation

In `removeTransactionApprover` (controller), after verifying the caller is the creator of `transactionId`, verify that the approver `id` actually belongs to that `transactionId` before deleting it. Mirror the pattern used in `updateTransactionApprover`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);

  // Add: verify the approver belongs to this transaction
  const approver = await this.approversService.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const rootNode = await this.approversService.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

Alternatively, move the full ownership validation into `removeTransactionApprover` in the service layer (as `updateTransactionApprover` already does), so the check cannot be bypassed regardless of how the service is called. [3](#0-2) 

### Proof of Concept

**Preconditions:**
- Alice is a verified user and creator of Transaction `txA` (id = 1).
- Bob is a verified user and creator of Transaction `txB` (id = 2), which has an approver with `id = 99`.

**Steps:**
1. Alice authenticates and obtains a JWT token.
2. Alice sends:
   ```
   DELETE /transactions/1/approvers/99
   Authorization: Bearer <alice_jwt>
   ```
3. Server calls `getCreatorsTransaction(1, alice)` → passes (Alice owns tx 1).
4. Server calls `removeTransactionApprover(99)` → deletes approver 99 from Bob's tx 2.
5. Response: `200 true`.

**Expected (correct) behavior:** 403/401 — approver 99 does not belong to transaction 1.
**Actual behavior:** Approver 99 is deleted from Bob's transaction without authorization. [1](#0-0)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
