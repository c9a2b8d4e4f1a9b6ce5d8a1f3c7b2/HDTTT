Audit Report

## Title
Persistent JWT Sessions After Password Change Allow Continued Unauthorized Account Access

## Summary
The `changePassword` endpoint in `AuthController` updates the user's password hash but never blacklists the caller's current JWT. An attacker holding a stolen JWT retains full authenticated access for up to 365 days — the default `JWT_EXPIRATION` — even after the victim changes their password.

## Finding Description

**Root cause:** `changePassword` does not invoke `blacklistService.blacklistToken(...)` and does not accept `@Req() req` to extract the token.

`auth.controller.ts` lines 115–119: [1](#0-0) 

Compare with `logout` at lines 102–104, which correctly blacklists the token: [2](#0-1) 

`auth.service.ts` `changePassword` (lines 84–95) only verifies the old password, optionally emits a status notification, and updates the password hash — no session invalidation occurs: [3](#0-2) 

`BlacklistService.blacklistToken` stores the raw JWT string in Redis with a TTL of `JWT_EXPIRATION` days. It is called in `logout` (line 103) and in both OTP-flow endpoints (`verifyOtp` line 157, `setPassword` line 174), but is absent from the `changePassword` path: [4](#0-3) 

The default `JWT_EXPIRATION` is **365 days** in both the example environment file and the production Helm values: [5](#0-4) [6](#0-5) 

The same gap exists in `setPassword` (post-OTP reset): the OTP JWT is blacklisted, but any pre-existing auth JWTs for the account are not invalidated: [7](#0-6) 

## Impact Explanation
An attacker holding a stolen JWT retains full authenticated access to the victim's organization account — including viewing sensitive transaction data, submitting signatures, approving transactions, and managing keys — even after the victim performs the expected remediation action (password change). The exposure window is up to 365 days. The victim has no in-product mechanism to force-expire other sessions; only an explicit `POST /auth/logout` from the attacker's own session would close it, which an attacker will not perform.

## Likelihood Explanation
The front-end is an Electron application whose renderer process handles JWTs and communicates with the back-end over HTTP. JWT theft is realistic via XSS in the renderer, a compromised shared machine, or a logged network session. Once a JWT is obtained, continued access after a victim's password change is guaranteed by the missing blacklist call — no further attacker action is required.

## Recommendation
In `AuthController.changePassword`, inject `@Req() req: Request` and call `await this.blacklistService.blacklistToken(extractJwtAuth(req))` after the password is successfully changed, mirroring the pattern used in `logout`. Apply the same fix to `setPassword` for any pre-existing auth JWTs (not just the OTP JWT). Consider also issuing a fresh auth token to the legitimate user after the password change so their own session is not disrupted.

## Proof of Concept
1. Victim logs in via `POST /auth/login` → receives `JWT_A`.
2. Attacker obtains `JWT_A` (e.g., via XSS on the Electron renderer or a shared device).
3. Victim detects suspicious activity and calls `PATCH /auth/change-password` with their old and new passwords using `JWT_A` (or any valid session token). The password hash in the database is updated.
4. No call to `blacklistService.blacklistToken(...)` is made; `JWT_A` is not added to the Redis blacklist.
5. Attacker sends any authenticated request (e.g., `GET /transactions`) with `JWT_A`. `JwtBlackListAuthGuard` checks Redis — `JWT_A` is not blacklisted — and `JwtAuthGuard` verifies the signature and expiry — still valid. The request succeeds.
6. Attacker retains full access for up to 365 days.

```typescript
  async changePassword(user: User, { oldPassword, newPassword }: ChangePasswordDto): Promise<void> {
    if (oldPassword === newPassword) throw new BadRequestException(ErrorCodes.NPMOP);

    const { correct } = await this.dualCompareHash(oldPassword, user.password);
    if (!correct) throw new BadRequestException(ErrorCodes.INOP);

    if (user.status === UserStatus.NEW && user.keys.length === 0) {
      emitUserStatusUpdateNotifications(this.notificationsPublisher, { entityId: user.id, additionalData: { username: user.email } });
    }

    await this.usersService.setPassword(user, newPassword);
  }
```

```typescript
  async blacklistToken(jwt: string) {
    const expirationDays = this.configService.get<number>('JWT_EXPIRATION');
    const expirationSeconds = Number(expirationDays) * 24 * 60 * 60;

    await this.client.set(jwt, this.BLACKLISTED, 'EX', expirationSeconds);
  }
```
