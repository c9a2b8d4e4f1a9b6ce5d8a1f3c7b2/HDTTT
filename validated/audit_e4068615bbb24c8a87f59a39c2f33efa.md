Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Authenticated DoS via Resource Exhaustion

## Summary
`GET /transactions/sign` is backed by `getTransactionsToSign`, which fetches every non-terminal transaction from the database with no row limit, then performs an expensive per-row async loop (transaction deserialization + mirror-node network call) before applying pagination in memory. A single authenticated request can exhaust memory, CPU, and mirror-node connections as the transaction table grows through normal platform use.

## Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues an unlimited `repo.find` call with no `take` (SQL `LIMIT`):

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The full result set is then iterated sequentially:

```ts
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

Each iteration calls `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign`: [3](#0-2) 

Inside `keysRequiredToSign`, per transaction:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — synchronous CPU-bound deserialization of raw transaction bytes
2. `transactionSignatureService.computeSignatureKey(transaction, showAll)` — potentially a mirror-node network call [4](#0-3) 

Pagination is applied only **after** all N transactions have been fully processed:

```ts
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [5](#0-4) 

The endpoint is registered and protected only by JWT + verified-user guards, with a comment confirming it is abandoned but not removed: [6](#0-5) 

## Impact Explanation

- **Memory exhaustion:** All non-terminal transaction rows (including `transactionBytes` blobs) are loaded into the Node.js heap simultaneously with no bound.
- **CPU exhaustion:** `SDKTransaction.fromBytes` is synchronous CPU work executed N times per request, blocking the Node.js event loop.
- **Mirror-node connection exhaustion:** `computeSignatureKey` is awaited sequentially per transaction, potentially issuing N external network calls per request.
- **Service-wide degradation:** Because Node.js is single-threaded, a blocked event loop degrades all concurrent API users, not just the attacker.

## Likelihood Explanation

- Any registered, verified user can reach this endpoint — no admin or privileged role is required.
- Transaction accumulation is a natural consequence of normal platform use; no adversarial seeding is required.
- The endpoint remains registered in the router despite being marked as unused by the front-end, making it reachable by any HTTP client.
- A single crafted request is sufficient; no sustained flood is needed.

## Recommendation

1. **Apply DB-level pagination before processing:** Pass `take` and `skip` to `repo.find` so the ORM issues a `LIMIT`/`OFFSET` SQL query, preventing unbounded row loading.
2. **Restructure the query:** Filter at the DB level for transactions where the user's keys are required signers, rather than loading all non-terminal transactions and filtering in application memory.
3. **Remove or gate the endpoint:** Since the endpoint is explicitly marked `/* NO LONGER USED BY FRONT-END */`, consider removing it entirely or protecting it with an admin-only guard until it is properly refactored.
4. **Add rate limiting:** Apply per-user rate limiting on this endpoint as a defense-in-depth measure.

## Proof of Concept

1. Register and verify an account via the normal sign-up flow.
2. Ensure the platform has accumulated a significant number of non-terminal transactions (this occurs naturally through normal use).
3. Send a single authenticated request:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <valid_jwt>
   ```
4. The server loads **all** non-terminal transaction rows into memory, then sequentially calls `SDKTransaction.fromBytes` and `computeSignatureKey` for each row before slicing to 1 result.
5. On a large dataset, this causes request timeouts, OOM, or mirror-node connection exhaustion for all concurrent users. The `size=1` parameter is irrelevant because the full scan occurs before slicing.
