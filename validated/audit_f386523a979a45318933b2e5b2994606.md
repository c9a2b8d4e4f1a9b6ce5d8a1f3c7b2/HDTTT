Audit Report

## Title
Missing Unique Constraint on `TransactionSigner` Allows Duplicate Signer Records via TOCTOU Race Condition in `uploadSignatureMaps`

## Summary
The `TransactionSigner` entity carries only a plain (non-unique) composite index on `(transactionId, userKeyId)`. The duplicate-prevention check in `uploadSignatureMaps` is performed against an in-memory snapshot loaded **before** any database transaction is opened. Two concurrent `POST /transactions/:transactionId/signers` requests from the same authenticated user can both pass the in-memory guard and each insert a separate `TransactionSigner` row for the same `(transactionId, userKeyId)` pair, corrupting the signing state of the transaction.

## Finding Description

**1. No unique constraint on `TransactionSigner`**

The entity decorator uses a plain `@Index` with no `{ unique: true }` option:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← plain index, NOT unique
export class TransactionSigner { ... }
``` [1](#0-0) 

The initial migration creates the table with no `UNIQUE` constraint: [2](#0-1) 

The later migration adds only a plain index — still not unique:

```sql
CREATE INDEX "IDX_f8cbac3e0cae3730a762e4905e"
  ON "transaction_signer" ("transactionId", "userKeyId")
``` [3](#0-2) 

By contrast, `TransactionObserver` correctly uses `{ unique: true }` on its composite index, showing the developer was aware of the pattern: [4](#0-3) 

**2. TOCTOU window in `uploadSignatureMaps`**

Step 1 — existing signers are loaded into an in-memory `Set` *outside* any database transaction: [5](#0-4) 

Step 2 — the duplicate guard is evaluated against that stale in-memory snapshot: [6](#0-5) 

Step 3 — the insert is a plain `INSERT` with no `ON CONFLICT` clause: [7](#0-6) 

The database transaction wrapping the insert is opened only inside `persistSignatureChanges`, long after the snapshot was taken: [8](#0-7) 

**Exploit flow:**
1. Attacker (authenticated user with key `K`) sends two concurrent `POST /transactions/:transactionId/signers` requests.
2. Both requests call `loadTransactionData` simultaneously; both read `existingSigners` and find no record for key `K` → both build an empty `existingSignerIds` set.
3. Both pass the `!existingSignerIds.has(userKey.id)` guard and queue key `K` for insertion.
4. Both execute `bulkInsertSigners` inside separate database transactions; since there is no unique constraint, both inserts succeed.
5. The `transaction_signer` table now contains two rows with identical `(transactionId, userKeyId)`.

## Impact Explanation

Duplicate `TransactionSigner` rows corrupt the authoritative record of who has signed a transaction:

- **Inflated signer count**: Any logic that counts distinct signers from the `TransactionSigner` table (e.g., threshold checks, audit queries) will double-count the attacker's key, potentially allowing a transaction to be considered "fully signed" with fewer real unique signers than required.
- **Audit trail corruption**: The signing history becomes unreliable; the same user/key appears multiple times, undermining non-repudiation guarantees.
- **Cascading state machine errors**: `processTransactionStatus` is invoked after every upload and drives the `WAITING_FOR_SIGNATURES → WAITING_FOR_EXECUTION` transition; if it relies on signer-record counts rather than pure cryptographic verification, a transaction could advance prematurely. [9](#0-8) 

## Likelihood Explanation

- **Attacker preconditions**: A valid, authenticated organization user with at least one registered key — no elevated privileges required.
- **Trigger**: Two concurrent HTTP requests to `POST /transactions/:transactionId/signers`. Trivially achievable from any HTTP client (e.g., two parallel `fetch` calls, or a script).
- **Reliability**: The race window spans the entire time between `loadTransactionData` and the commit of `persistSignatureChanges`, which includes signature validation and SDK processing — a window of tens to hundreds of milliseconds, making the race reliably winnable. [10](#0-9) 

## Recommendation

Apply **both** of the following fixes — neither alone is sufficient:

1. **Add a unique constraint at the database level** on `(transactionId, userKeyId)` in the `TransactionSigner` entity:
   ```typescript
   @Entity()
   @Index(['transactionId', 'userKeyId'], { unique: true })
   export class TransactionSigner { ... }
   ```
   And generate a corresponding migration with `CREATE UNIQUE INDEX`.

2. **Add `ON CONFLICT DO NOTHING` (or `orIgnore()`) to the insert** in `bulkInsertSigners` so that any race-condition duplicate is silently discarded rather than causing an error:
   ```typescript
   await manager
     .createQueryBuilder()
     .insert()
     .into(TransactionSigner)
     .values(signersToInsert)
     .orIgnore()          // ← idempotent insert
     .returning('*')
     .execute();
   ```

  [TARGET_TX_ID, USER_KEY_ID]
