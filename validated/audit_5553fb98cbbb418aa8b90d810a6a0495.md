Audit Report

## Title
Unauthorized Cross-Transaction Approver Deletion via Missing Ownership Validation in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who created at least one transaction can delete approvers from any other transaction by supplying their own `transactionId` for the ownership check and a foreign approver `id` as the deletion target.

## Finding Description

**Root cause — controller decouples ownership check from deletion target:**

In `approvers.controller.ts` lines 102–113, `removeTransactionApprover`:
1. Calls `getCreatorsTransaction(transactionId, user)` — verifies the caller owns `:transactionId`.
2. Calls `removeTransactionApprover(id)` — deletes the approver node identified by `:id` with no further validation. [1](#0-0) 

The service function `removeTransactionApprover` fetches the approver by its own primary key and deletes it — `transactionId` is never consulted: [2](#0-1) 

**Contrast with `updateTransactionApprover` — which performs the check correctly:**

The update path walks up to the root node and explicitly asserts `rootNode.transactionId === transactionId` before proceeding: [3](#0-2) 

This is the exact asymmetry: `updateTransactionApprover` enforces the ownership invariant; `removeTransactionApprover` does not.

## Impact Explanation
An attacker can silently remove any approver record from any transaction in the system. Removing approvers from a transaction that requires a threshold of approvals before execution can allow the transaction to advance to `WAITING_FOR_EXECUTION` or be executed without the intended governance controls. This constitutes unauthorized state mutation and a bypass of the multi-signature approval workflow, which is a core security invariant of the system.

## Likelihood Explanation
- **Precondition:** attacker must be an authenticated, verified user who has created at least one transaction — a low barrier.
- **Approver ID discovery:** approver IDs are sequential database integers. They can be enumerated, or obtained legitimately if the attacker is also an approver/observer on the target transaction via `GET /transactions/:transactionId/approvers`.
- **No privileged role required.** The attack is fully reachable by any normal user.

## Recommendation
In the `removeTransactionApprover` service method, after fetching the approver, walk up to its root node (using the existing `getRootNodeFromNode` helper) and assert that `rootNode.transactionId === transactionId` before proceeding with deletion — mirroring the pattern already used in `updateTransactionApprover`:

```typescript
// approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, consistent with how `updateTransactionApprover` already receives it. [4](#0-3) 

## Proof of Concept

1. **Setup:** User A is the creator of Transaction 1. Transaction 2 belongs to User B and has an approver record with `id = 42`.
2. **Discovery:** User A obtains approver ID 42 via sequential enumeration, or by calling `GET /transactions/2/approvers` if they are an observer/approver on Transaction 2.
3. **Attack request:**
   ```
   DELETE /transactions/1/approvers/42
   Authorization: Bearer <User A's token>
   ```
4. **Execution flow:**
   - `getCreatorsTransaction(1, userA)` — passes, User A owns Transaction 1.
   - `removeTransactionApprover(42)` — fetches approver 42 (belonging to Transaction 2) by primary key only, then soft-deletes it. No cross-transaction check is performed.
5. **Result:** Approver 42 from Transaction 2 is deleted. User B's transaction now has a reduced approval requirement, potentially allowing it to advance without the intended governance controls. [2](#0-1)
