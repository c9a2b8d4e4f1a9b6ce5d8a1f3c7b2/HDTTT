Audit Report

## Title
OTP Verification Endpoint (`/auth/verify-reset`) Has No Per-Attempt Rate Limiting, Enabling Brute-Force of Password Reset Codes

## Summary
`POST /auth/verify-reset` carries only a global IP-based throttle (10,000 req/min per IP) and no per-OTP-attempt lockout. Because the OTP JWT is never blacklisted on a failed guess, an attacker can exhaust the 8-digit TOTP space against a victim's account within the JWT validity window, then call `/auth/set-password` to permanently take over the account.

## Finding Description

**Primary: No per-attempt throttle on `/auth/verify-reset`**

The endpoint is decorated with only `JwtBlackListOtpGuard` and `OtpJwtAuthGuard`: [1](#0-0) 

Compare to `/auth/reset-password`, which adds `EmailThrottlerGuard`: [2](#0-1) 

A global `IpThrottlerGuard` is registered as `APP_GUARD` and does apply to this route, but its production limits are `GLOBAL_MINUTE_LIMIT: 10000` and `GLOBAL_SECOND_LIMIT: 1000` — far too permissive to prevent brute-force: [3](#0-2) [4](#0-3) 

**OTP JWT is never blacklisted on a failed guess**

In `verifyOtp`, the service throws `UnauthorizedException` before the controller can call `blacklistToken`. The blacklist call is only reached on success: [5](#0-4) [6](#0-5) 

This means the same OTP JWT can be reused for unlimited guesses until it expires.

**TOTP configured with `window: 20`** [7](#0-6) 

With `step: 60` and `window: 20`, `otplib` accepts codes from ±20 steps, meaning 41 distinct 8-digit codes are simultaneously valid, spanning a ~41-minute window. The OTP JWT itself expires after `OTP_EXPIRATION` minutes (20 in production): [8](#0-7) 

**Secondary: Login guard order allows bcrypt/argon2 work before rate-limit check**

`LocalAuthGuard` (which triggers the expensive password comparison) is listed before `EmailThrottlerGuard`: [9](#0-8) 

NestJS executes guards in declaration order, so the cryptographic comparison runs before the rate limit is ever evaluated. Additionally, `EmailThrottlerGuard` keys on `req.body.email`: [10](#0-9) 

An attacker rotating email addresses bypasses the per-email throttle while still forcing bcrypt/argon2 work on every request.

## Impact Explanation

A successful brute-force of `/auth/verify-reset` yields a `verified: true` OTP JWT. The attacker then calls `/auth/set-password` to permanently set a new password for the victim: [11](#0-10) 

**Feasibility**: With 41 valid codes out of 10^8 and 10,000 requests/minute allowed per IP, an attacker gets ~200,000 attempts per 20-minute JWT window. That yields roughly an 8% success probability per JWT. With multiple source IPs (bypassing the IP throttler entirely), the attack becomes near-certain within a few JWT windows. The attacker can obtain fresh OTP JWTs by repeatedly calling `/auth/reset-password` (throttled at 3/min per email, but trivially worked around by waiting).

## Likelihood Explanation

The attack requires only the victim's email address (the login username) and the ability to send HTTP requests. No privileged access is needed. The attacker initiates the entire flow themselves via the public `/auth/reset-password` endpoint. The extended TOTP window (`window: 20`) and the absence of per-attempt lockout make this reliably exploitable.

## Recommendation

1. **Blacklist the OTP JWT on every failed verification attempt**, not only on success. This forces the attacker to obtain a new JWT (and thus a new OTP) for each guess, making brute-force impractical.
2. **Add a strict per-OTP-JWT attempt counter** (e.g., max 5 guesses) stored in Redis, keyed on the JWT `jti` claim. Exceed the limit → blacklist the token immediately.
3. **Add `EmailThrottlerGuard` to `/auth/verify-reset`** (or a dedicated low-limit throttler keyed on the OTP JWT or user identity).
4. **Reduce `window`** from 20 to 1 or 2. A window of 20 is unusually large and serves no legitimate UX purpose that a smaller window cannot cover.
5. **Fix the login guard order**: move `EmailThrottlerGuard` before `LocalAuthGuard` so rate limiting is enforced before the expensive cryptographic comparison.

## Proof of Concept

```bash
# Step 1: Trigger password reset for victim — attacker receives OTP JWT
OTP_JWT=$(curl -s -X POST https://target/auth/reset-password \
  -H 'Content-Type: application/json' \
  -d '{"email":"victim@example.com"}' | jq -r .token)

# Step 2: Brute-force the 8-digit OTP
# The same OTP_JWT is reused on every failed attempt (never blacklisted on failure)
for CODE in $(seq -w 00000000 99999999); do
  RESP=$(curl -s -o /dev/null -w "%{http_code}" -X POST https://target/auth/verify-reset \
    -H 'Content-Type: application/json' \
    -H "otp: $OTP_JWT" \
    -d "{\"token\":\"$CODE\"}")
  if [ "$RESP" = "200" ]; then
    echo "Valid OTP found: $CODE"
    break
  fi
done

# Step 3: Use the verified JWT to set a new password
curl -s -X PATCH https://target/auth/set-password \
  -H 'Content-Type: application/json' \
  -H "otp: $VERIFIED_JWT" \
  -d '{"password":"AttackerControlledPassword1!"}'
```

The loop can be parallelised across multiple IPs to bypass the global IP throttler. With `window: 20`, the valid code remains stable for ~41 minutes, giving ample time to complete the search.
