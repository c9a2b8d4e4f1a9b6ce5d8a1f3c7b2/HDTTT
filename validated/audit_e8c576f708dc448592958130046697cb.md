### Title
`archiveTransaction` Bypasses Status Validation for Manual Transactions, Allowing Terminal-State Mutation

### Summary

The `archiveTransaction` function in `transactions.service.ts` contains a flawed guard condition that short-circuits the status check whenever `transaction.isManual` is `true`. This allows any authenticated creator to archive their own manual transaction regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED` — overwriting an immutable terminal state with `ARCHIVED` and corrupting the organization's audit trail.

### Finding Description

**Root cause — `back-end/apps/api/src/transactions/transactions.service.ts` lines 708–733:**

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual          // ← short-circuit: when isManual=true this whole guard is false
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
``` [1](#0-0) 

The guard is logically equivalent to:

```
throw only if (status NOT in [WFS, WFE]) AND (NOT isManual)
```

When `isManual = true`, the right-hand operand of `AND` is always `false`, so the entire condition is `false` and the guard never fires. The status of the transaction is **never validated** for manual transactions. Any status — including `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED` — passes through to the `repo.update` call that stamps `ARCHIVED` onto the record.

**Contrast with the correct guard used elsewhere.** The `cancelTransaction` path correctly enforces status:

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [2](#0-1) 

**Test gap.** The existing unit test for the "not archiveable" case uses a transaction with `isManual` unset (falsy), so it never exercises the `isManual = true` + terminal-status combination:

```typescript
it('should throw if transaction status is not archiveable', async () => {
  const transaction = {
    creatorKey: { userId: 1 },
    status: TransactionStatus.CANCELED,   // isManual not set → falsy
  };
  ...
  await expect(service.archiveTransaction(123, { id: 1 } as User)).rejects.toThrow(ErrorCodes.OMTIP);
});
``` [3](#0-2) 

The passing test for the success path uses `isManual: true` only with `WAITING_FOR_EXECUTION`, never with a terminal status: [4](#0-3) 

**API surface.** The archive endpoint is reachable by any verified, authenticated user via `PATCH /transactions/archive/:id`, protected only by JWT and email-verification guards — no additional status guard at the controller layer: [5](#0-4) 

### Impact Explanation

A malicious (or mistaken) transaction creator can:

1. Overwrite `EXECUTED` → `ARCHIVED`, erasing on-chain execution evidence from the organization's shared view.
2. Overwrite `FAILED` → `ARCHIVED`, hiding a failed Hedera submission and preventing observers/approvers from acting on the failure.
3. Overwrite `CANCELED` → `ARCHIVED`, obscuring the fact
