# Audit Report

## Title
Threshold Invariant Broken After Approver Removal Permanently Locks Transaction Approval

## Summary
`ApproversService.removeTransactionApprover` soft-deletes a child approver node via `removeNode` but never reads or adjusts the parent node's `threshold` column. When the removed child was the only approver that could satisfy the threshold, the parent node is left with `threshold > remaining_children_count`, making the approval condition permanently unsatisfiable. The transaction is frozen in a non-terminal status indefinitely.

## Finding Description

**Root Cause:**

`removeTransactionApprover` fetches the approver and delegates entirely to `removeNode`, which performs a recursive soft-delete of the target and all its descendants. Neither function reads the parent node or adjusts its `threshold`.

```typescript
// approvers.service.ts lines 534–544
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);   // ← no parent threshold fix
    emitTransactionStatusUpdate(...);
    return result;
}
``` [1](#0-0) 

`removeNode` only soft-deletes the target subtree: [2](#0-1) 

**Contrast with the correct path:**

`updateTransactionApprover` (the "detach child" path, `dto.listId = null`) explicitly re-reads the parent's child count and lowers the threshold when it would exceed the new count:

```typescript
// lines 417–428
if (parent) {
    const newParentApproversLength = parent.approvers.length - 1;
    if (newParentApproversLength === 0) {
        await transactionalEntityManager.softRemove(TransactionApprover, parent);
    } else if (newParentApproversLength < parent.threshold) {
        await transactionalEntityManager.update(TransactionApprover, parent.id, {
            threshold: newParentApproversLength,
        });
    }
}
``` [3](#0-2) 

`removeTransactionApprover` has no equivalent adjustment.

**Exploit Flow:**

1. Transaction creator creates a threshold tree: parent node with `threshold = 2`, children = [User A, User B].
2. Creator calls `DELETE /transactions/:txId/approvers/:approverIdA`.
3. Controller calls `getCreatorsTransaction` (authorization passes), then `removeTransactionApprover(approverIdA)`.
4. `removeNode` soft-deletes User A's row. Parent row still holds `threshold = 2`.
5. Only User B remains. The approval condition now requires 2 approvals from 1 possible approver.
6. User B approves: `approvals.length = 1 < threshold = 2` → `isApproved` returns `null`.
7. The transaction stays in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely; no combination of approvals can satisfy `threshold = 2` with one approver.

The `isApproved` evaluation confirms the permanent lock:

```typescript
// front-end/src/renderer/utils/sdk/index.ts lines 354–357
if (approvals.length >= (approver.threshold || approvals.length)) {
    return true;
}
return rejections.length >= (approver.threshold || rejections.length) ? false : null;
``` [6](#0-5) 

## Impact Explanation

A transaction is permanently locked in a non-terminal status. No approver can ever satisfy the threshold condition, so the chain service will never advance the transaction to `EXECUTED`. The creator's funds/operations tied to that transaction are frozen. This matches the RESEARCHER.md impact category: *"Permanent lock, freeze, or unrecoverable corruption of user/project state."*

The state is unrecoverable through normal product flows: the creator cannot re-add the removed approver to restore the count (the threshold is already stale), and there is no API endpoint to directly lower the parent threshold without going through `updateTransactionApprover` with `dto.threshold`, which itself validates `threshold <= approvers.length` — but the stale threshold already violates this invariant, so that path would also fail.

## Likelihood Explanation

The trigger path is a normal, documented product workflow: the transaction creator removes an approver via the approvers API. No privileged access beyond being the transaction creator is required. The controller enforces creator-only access: [4](#0-3) 

The inconsistency is easy to reach accidentally (creator removes one approver without realizing the threshold is now unsatisfiable) or intentionally (creator deliberately locks a shared transaction). The `updateTransactionApprover` path already demonstrates the developers are aware the threshold must be adjusted when child count drops — the fix exists in one code path but is absent in the other. [7](#0-6) 

## Recommendation

In `removeTransactionApprover`, after calling `removeNode`, fetch the parent node (if any) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // NEW: adjust parent threshold if this is a child node
    if (approver.listId) {
        const parent = await this.repo.findOne({
            relations: ['approvers'],
            where: { id: approver.listId },
        });
        if (parent) {
            const newCount = parent.approvers.filter(a => a.id !== approver.id).length;
            if (newCount === 0) {
                await this.repo.softRemove(parent);
            } else if (newCount < parent.threshold) {
                await this.repo.update(parent.id, { threshold: newCount });
            }
        }
    }

    await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
}
```

This mirrors the existing fix in `updateTransactionApprover` and restores the invariant `threshold ≤ remaining_children_count` after every removal.

## Proof of Concept

**Setup:**
1. Create a transaction (creator = User C).
2. As User C, `POST /transactions/:txId/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 2, "approvers": [{ "userId": 1 }, { "userId": 2 }] }] }
   ```
   This creates a parent node (id=P) with `threshold=2` and two children (id=A for User 1, id=B for User 2).

**Trigger:**
3. As User C, `DELETE /transactions/:txId/approvers/A` — removes User 1's child node.

**Verify broken state:**
4. Query the database: parent node P still has `threshold=2`, but only child B remains.
   ```sql
   SELECT id, threshold FROM transaction_approver WHERE id = P;
   -- threshold = 2  ← stale
   SELECT COUNT(*) FROM transaction_approver WHERE "listId" = P AND "deletedAt" IS NULL;
   -- count = 1  ← only User 2 remains
   ```

**Demonstrate lock:**
5. As User 2, `POST /transactions/:txId/approvers/approve` with a valid signature.
6. Observe: `approvals.length = 1 < threshold = 2` → `isApproved` returns `null`.
7. Transaction remains in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely. No further approvals are possible (only 1 approver exists), and the transaction can never reach `EXECUTED`.

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
