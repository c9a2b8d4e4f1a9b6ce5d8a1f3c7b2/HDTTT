Audit Report

## Title
Critically Insufficient PBKDF2 Iteration Count (2560) Enables Offline Brute-Force of Encrypted Hedera Private Keys

## Summary
The `deriveKey` function in `front-end/src/main/utils/crypto.ts` uses only **2560 PBKDF2-SHA512 iterations** to derive the AES-256-GCM encryption key for Hedera private keys stored in the local SQLite database. This is ~82× below OWASP's current minimum recommendation of 210,000 iterations for PBKDF2-SHA512. An attacker who obtains the SQLite database file can perform an offline brute-force attack at ~82× the speed of a properly configured KDF, recovering raw Hedera private keys.

## Finding Description

**Root cause — hardcoded, critically weak KDF parameter:**

`front-end/src/main/utils/crypto.ts`, lines 5–9:
```typescript
export function deriveKey(password: string, salt: Buffer) {
  const iterations = 2560;   // no rationale, no documentation
  const keyLength = 32;
  return crypto.pbkdf2Sync(password, salt, iterations, keyLength, 'sha512');
}
``` [1](#0-0) 

The `encrypt` function constructs a fully self-contained ciphertext blob as `[64-byte salt | 16-byte IV | 16-byte GCM tag | ciphertext]`, base64-encoded: [2](#0-1) 

**Production code path — private key encryption:**

`front-end/src/main/services/localUser/keyPairs.ts` imports and uses `encrypt`/`decrypt` directly: [3](#0-2) [4](#0-3) [5](#0-4) 

The same weak `deriveKey` is also used in `organizationCredentials.ts` to encrypt organization passwords: [6](#0-5) [7](#0-6) 

**Contrast with the migration path:**

`front-end/src/main/services/localUser/dataMigration.ts` uses Argon2id with `memoryCost: 262144` (256 MB) and `timeCost: 3` for the same purpose — demonstrating the team is aware of proper KDF hardening but did not apply it to the primary key-storage path: [8](#0-7) 

## Impact Explanation

- **Direct asset theft:** Recovery of the Hedera ED25519 or ECDSA private key allows the attacker to sign arbitrary transactions, transferring all HBAR and tokens from the account with no recourse.
- **Permanent compromise:** Once the private key is recovered, the account is permanently compromised unless the user rotates keys on-chain before the attacker acts.
- **Scope:** Affects every user who stores keys via the password path (i.e., users who do not use the OS keychain). The `useKeychain` flag is conditional and not universally the default. [9](#0-8) 

## Likelihood Explanation

- **Attacker precondition:** Obtain the SQLite database file. For a desktop Electron application, this is realistic via: (a) malware/RAT running as the same OS user — no elevated privileges required; (b) unencrypted cloud or Time Machine backup; (c) physical access to an unlocked machine.
- **No online rate-limiting applies:** The attack is fully offline against the self-contained ciphertext blob.
- **Speed:** At 2560 iterations, a modern GPU (RTX 4090, ~1 billion SHA-512 ops/sec) yields approximately **390,000 password guesses/second**. A 10-million-entry common-password dictionary is exhausted in ~25 seconds. An 8-character alphanumeric brute-force completes in days.
- **No privileged access required:** The SQLite database file resides in the Electron app's user-data directory, readable by any process running as the same OS user. [1](#0-0) 

## Recommendation

1. **Increase PBKDF2 iterations** to at least 210,000 (OWASP minimum for PBKDF2-SHA512), or preferably migrate to Argon2id (already used in `dataMigration.ts`) with `memoryCost: 262144` and `timeCost: 3`.
2. **Re-encrypt existing stored keys** on next user login using the updated KDF parameters.
3. **Document the KDF parameters** with a rationale and link to the relevant OWASP guidance.
4. Apply the same fix to `organizationCredentials.ts`, which uses the same `encrypt`/`decrypt` functions. [8](#0-7) 

## Proof of Concept

```python
# Offline brute-force skeleton against the stored ciphertext blob
import base64, hashlib, hmac
from Crypto.Cipher import AES

# 1. Extract blob from SQLite KeyPair.private_key column
blob = base64.b64decode("<base64_private_key_from_db>")
salt       = blob[0:64]
iv         = blob[64:80]
tag        = blob[80:96]
ciphertext = blob[96:]

# 2. For each password candidate:
for password in open("rockyou.txt"):
    password = password.strip().encode()
    # Only 2560 iterations — trivially fast per guess
    key = hashlib.pbkdf2_hmac('sha512', password, salt, 2560, dklen=32)
    try:
        cipher = AES.new(key, AES.MODE_GCM, nonce=iv)
        cipher.update(b"")
        plaintext = cipher.decrypt_and_verify(ciphertext, tag)
        print(f"[+] Password found: {password.decode()}")
        print(f"[+] Private key: {plaintext.decode()}")
        break
    except Exception:
        pass
```

The self-contained blob format (`salt | iv | tag | ciphertext`) means no server interaction is needed. The 2560-iteration count makes each guess computationally trivial, enabling high-throughput GPU-accelerated attacks. [10](#0-9)

export function encrypt(data: string, password: string) {
  const iv = crypto.randomBytes(16);
  const salt = crypto.randomBytes(64);

  const key = deriveKey(password, salt);

  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([cipher.update(data, 'utf8'), cipher.final()]);

  const tag = cipher.getAuthTag();

  return Buffer.concat([salt, iv, tag, encrypted]).toString('base64');
}

export function decrypt(data: string, password: string) {
  const bData = Buffer.from(data, 'base64');

  const salt = bData.subarray(0, 64);
  const iv = bData.subarray(64, 80);
  const tag = bData.subarray(80, 96);
  const text = bData.subarray(96).toString('base64');

  const key = deriveKey(password, salt);

  const decipher = crypto.createDecipheriv('aes-256-gcm', key, iv);
  decipher.setAuthTag(tag);

  const decrypted = decipher.update(text, 'base64', 'utf8') + decipher.final('utf8');

  return decrypted;
}
