Audit Report

## Title
Any Authenticated User Can Post and Read Comments on Arbitrary Transactions Due to Missing Authorization Check

## Summary
The `POST /transactions/:transactionId/comments` and `GET /transactions/:transactionId/comments` endpoints in `CommentsController` perform no authorization check to verify that the requesting user has any relationship to the target transaction. Any verified, authenticated user can write and read comments on any transaction in the system. The developer explicitly acknowledged this gap with a `//TODO` comment in the controller.

## Finding Description

**Root Cause**

The `createComment` handler in `CommentsController` applies only the standard authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). These guards confirm the user is logged in and verified, but perform no relationship check between the user and the target transaction. [1](#0-0) 

The developer explicitly flagged this gap at line 22: [2](#0-1) 

The underlying `CommentsService.createComment` performs zero authorization checks — it does not verify that the transaction exists, that the user is a creator/signer/approver/observer, or that the transaction is in a state that permits comments: [3](#0-2) 

**Contrast with other services**

Every other transaction sub-resource enforces relationship-based access. `ObserversService.getTransactionObserversByTransactionId` throws `UnauthorizedException` if the user is not a creator, signer, approver, or observer: [4](#0-3) 

`ApproversService.getVerifiedApproversByTransactionId` applies the identical pattern: [5](#0-4) 

`CommentsService` has no equivalent check.

**Read path is also unprotected**

The `GET /transactions/:transactionId/comments` endpoint also performs no transaction-level authorization — the user object is not even passed to the service: [6](#0-5) 

## Impact Explanation

- **Unauthorized state modification**: Any verified user can inject comments into transactions they have no business relationship with, polluting the audit trail and potentially confusing legitimate participants.
- **Information disclosure**: Any verified user can read all comments on any transaction, including comments that may contain sensitive operational details (key decisions, financial amounts, internal notes).
- **Integrity violation**: The comment history — which is part of the transaction audit record stored in `transaction_comment` — can be corrupted by arbitrary users. [7](#0-6) 

## Likelihood Explanation

Exploitation requires only a valid, verified account obtained through the standard registration flow. Transaction IDs are sequential integers (primary key auto-increment), making enumeration trivial. No special tooling is needed — a single HTTP request with a valid JWT is sufficient. The attack is fully reachable by any malicious normal user.

## Recommendation

Implement a relationship check in `CommentsService.createComment` and `CommentsService.getTransactionComments` mirroring the pattern already used in `ObserversService` and `ApproversService`:

1. Fetch the target transaction with its `creatorKey`, `observers`, `signers`, and `approvers` relations.
2. Throw `BadRequestException` if the transaction does not exist.
3. Throw `UnauthorizedException` if the requesting user is not the creator, a signer, an approver, or an observer of the transaction (unless the transaction is in a terminal status such as `EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, or `ARCHIVED`, where visibility may be broadened per existing policy).
4. Remove the `//TODO` comment once the guard is in place.

## Proof of Concept

```
# Step 1: Register and verify an account (standard flow)
# Step 2: Obtain a JWT token

# Step 3: Post a comment to a transaction the attacker has no relationship with
POST /transactions/42/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "message": "Injected comment on victim transaction" }

# Expected (current): 201 Created — comment persisted to transaction_comment table
# Expected (fixed):   401 Unauthorized

# Step 4: Read all comments on any transaction
GET /transactions/42/comments
Authorization: Bearer <attacker_jwt>

# Expected (current): 200 OK — full comment history returned
# Expected (fixed):   401 Unauthorized
```

The `//TODO` at `comments.controller.ts` line 22 confirms the developer was aware this check was missing and deferred it. [8](#0-7)
