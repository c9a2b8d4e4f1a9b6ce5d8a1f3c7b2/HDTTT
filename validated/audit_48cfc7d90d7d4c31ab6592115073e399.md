Audit Report

## Title
Incorrect Logical Operator in `archiveTransaction` Guard Allows Archiving of Terminal-State Transactions

## Summary
The `archiveTransaction` method in `transactions.service.ts` uses `&&` in its guard condition, causing the status check to be completely bypassed for any transaction where `isManual` is `true`. This allows a transaction creator to transition a manual transaction from any terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `REJECTED`) to `ARCHIVED`, corrupting the immutable audit record.

## Finding Description
The guard at lines 711–718 of `back-end/apps/api/src/transactions/transactions.service.ts`:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

throws only when **both** conditions are simultaneously true. When `transaction.isManual === true`, the second operand `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false` regardless of the status. The guard is completely bypassed, and execution falls through unconditionally to:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

The `terminalStatuses` array defined in the same service explicitly enumerates states that must not be further mutated, yet `archiveTransaction` never consults it for manual transactions: [3](#0-2) 

The test suite for `archiveTransaction` in `transactions.service.spec.ts` contains only 3 test points and does not cover the `isManual: true` + terminal status combination, leaving the bypass silently untested. [4](#0-3) 

## Impact Explanation
A transaction creator can overwrite the status of any finalized manual transaction (e.g., `EXECUTED`, `FAILED`) to `ARCHIVED`. This:
- Corrupts the audit trail: `ARCHIVED` is filtered differently from `EXECUTED`/`FAILED` in history queries, effectively hiding the transaction's true outcome.
- Breaks immutability guarantees: terminal states are explicitly defined as non-mutable in `terminalStatuses`, but this contract is violated for manual transactions.
- Enables a user to conceal an executed or failed transaction from history views by re-labeling it as `ARCHIVED`.

## Likelihood Explanation
The attacker is the transaction creator — a normal, unprivileged authenticated user. The preconditions are trivial:
1. Create a transaction with `isManual: true` (a standard, documented feature).
2. Wait for it to reach any terminal state (e.g., `EXECUTED`, `CANCELED`).
3. Call `PATCH /transactions/archive/:id` with a valid auth token.

No elevated privileges, leaked credentials, or race conditions are required.

## Recommendation
The guard must be restructured so that terminal-state transactions are always blocked from archiving, regardless of `isManual`. The simplest correct fix:

```typescript
// Block archiving of any terminal-state transaction
if (this.terminalStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}

// For non-manual transactions, also require an active status
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Alternatively, combine into a single expression using `||` to enforce the terminal-state check unconditionally. Add test cases covering `isManual: true` with each terminal status to prevent regression.

## Proof of Concept
1. Authenticate as a normal user and create a transaction with `isManual: true`.
2. Allow or simulate the transaction reaching `EXECUTED` status.
3. Send `PATCH /transactions/archive/<id>` with a valid JWT.
4. Observe the response returns `200 OK` and the transaction status is now `ARCHIVED` — despite having been in a terminal state — confirmed by querying the transaction record.
