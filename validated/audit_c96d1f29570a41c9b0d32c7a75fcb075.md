Audit Report

## Title
Cross-Transaction Approver Removal via IDOR in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId`, but then deletes the approver record identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who has created at least one transaction can silently delete approvers from transactions they do not own.

## Finding Description

**Root cause — decoupled ownership check and object mutation**

The controller performs two independent, unlinked operations:

```
back-end/apps/api/src/transactions/approvers/approvers.controller.ts
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller owns `transactionId`. It says nothing about whether approver `id` belongs to that transaction.

`removeTransactionApprover` then fetches the approver purely by its primary key and deletes it: [2](#0-1) 

`getTransactionApproverById` queries only on `id`, with no `transactionId` filter: [3](#0-2) 

**The fix already exists in the sibling method — but was not applied here.**

`updateTransactionApprover` correctly performs the cross-check that `removeTransactionApprover` omits: it walks up to the root node and asserts `rootNode.transactionId === transactionId` before proceeding: [4](#0-3) 

The delete path skips this step entirely, creating the IDOR.

**Exploit flow**

1. Attacker registers as a normal user and creates any transaction (becomes its creator, obtaining a valid `attacker_own_tx_id`).
2. Attacker enumerates or guesses the integer `id` of an approver belonging to a victim's transaction (IDs are sequential auto-increment integers).
3. Attacker sends:
   ```
   DELETE /transactions/{attacker_own_tx_id}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction(attacker_own_tx_id, attacker)` passes — attacker owns that transaction.
5. `removeTransactionApprover(victim_approver_id)` executes — deletes the victim's approver record with no further check.
6. The victim's transaction now has fewer (or zero) required approvers; the chain service may advance it to `WAITING_FOR_EXECUTION` or `EXECUTED` without the intended approval gate.

## Impact Explanation

- **Unauthorized state mutation**: Any registered user can remove approvers from any other user's transaction, destroying the multi-signature approval requirement.
- **Approval gate bypass**: Once required approvers are removed, the chain service can execute a Hedera transaction that was never properly approved, resulting in unauthorized on-chain actions (fund transfers, account updates, node operations, etc.).
- **No asset recovery**: Hedera transactions are irreversible once submitted to the network.

## Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (any registered, verified user). No admin or privileged role required.
- **Enumeration**: `TransactionApprover` IDs are sequential integers. An attacker can trivially iterate over a small range to find valid approver IDs belonging to other transactions.
- **No rate limiting** is observed on the delete endpoint, making bulk enumeration practical.
- The attack is silent — the victim receives a status-update notification but no indication that an approver was removed by an unauthorized party.

## Recommendation

Apply the same cross-check already present in `updateTransactionApprover` to the delete path. In `removeTransactionApprover` (or in the controller before calling it), walk up to the root node of the approver and assert that `rootNode.transactionId === transactionId`:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // Cross-check: ensure the approver belongs to the expected transaction
    const rootNode = await this.getRootNodeFromNode(approver.id);
    if (!rootNode || rootNode.transactionId !== transactionId)
        throw new UnauthorizedException('Approver does not belong to this transaction');

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
    return result;
}
```

Pass `transactionId` from the controller call site: [5](#0-4) 

## Proof of Concept

```
# Step 1 — Attacker creates their own transaction
POST /transactions
Authorization: Bearer <attacker_jwt>
→ { "id": 42 }   # attacker_own_tx_id

# Step 2 — Attacker enumerates approver IDs (sequential integers)
# Victim's transaction has approver id=99 belonging to transaction id=7

# Step 3 — Attacker deletes victim's approver using their own tx as the ownership anchor
DELETE /transactions/42/approvers/99
Authorization: Bearer <attacker_jwt>
→ HTTP 200 true

# Result: approver id=99 (belonging to victim's transaction id=7) is now soft-deleted.
# Victim's transaction may proceed to execution without required approval.
```
