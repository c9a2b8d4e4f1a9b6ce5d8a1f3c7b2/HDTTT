### Title
Unbounded Full-Table Scan with Per-Row Async Work in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in `TransactionsService` fetches **every** non-terminal transaction from the database with no `LIMIT`, then executes a costly async cryptographic computation (`userKeysToSign`) for each row in a sequential loop. Pagination is applied only after the full scan completes in memory. As the number of active transactions grows — whether organically or through attacker-created transactions — each call to `GET /transactions/sign` consumes unbounded CPU, memory, and database resources, eventually causing severe service degradation or complete unavailability.

### Finding Description

**Root cause — no database-level limit on the fetch:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` calls `this.repo.find()` with no `take` option:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / LIMIT
});
``` [1](#0-0) 

This loads every transaction whose status is not in `[EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED]` — i.e., every active transaction in the entire system — into memory.

**Per-row expensive async work:**

For each fetched transaction, `userKeysToSign` is awaited sequentially:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

`userKeysToSign` calls `userKeysRequiredToSign`, which in turn calls `keysRequiredToSign`. Each invocation:
1. Deserializes transaction bytes via `SDKTransaction.fromBytes`
2. Calls `transactionSignatureService.computeSignatureKey()` — which makes async calls to `AccountCacheService` and `NodeCacheService` (potentially hitting mirror nodes)
3. Performs key filtering [3](#0-2) 

**Pagination applied after full scan:**

The `PaginationParams` decorator enforces `size <= 100`, but this only limits the final slice. The full scan and per-row work always runs over all N active transactions regardless of the requested page size:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // pagination applied here, after full scan
  ...
};
``` [4](#0-3) [5](#0-4) 

**Attacker entry point — the HTTP endpoint:**

The endpoint is still live and reachable by any authenticated, verified user:

```typescript
@Get('/sign')
getTransactionsToSign(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
  ...
) {
  return this.transactionsService.getTransactionsToSign(user, paginationParams, sort, filter);
}
``` [6](#0-5) 

**Transaction creation is available to any authenticated user with a key:**

An attacker can register, add a key, and call `POST /transactions` repeatedly to create thousands of transactions in `WAITING_FOR_SIGNATURES` status (non-terminal), inflating the scan size permanently. [7](#0-6) 

### Impact Explanation

Each call to `GET /transactions/sign` with N active transactions in the system:
- Loads all N transaction rows (with `groupItem` relation) into Node.js heap
- Runs N sequential async operations involving SDK deserialization and potential mirror-node cache lookups
- Holds a DB connection for the duration

With thousands of active transactions, a single request can take tens of seconds and consume hundreds of MB of memory. Multiple concurrent requests from one or more users can exhaust the Node.js event loop, DB connection pool, and heap, causing the API service to become unresponsive for all users. This constitutes a **service availability** impact — all users lose access to the platform.

### Likelihood Explanation

- **Attacker preconditions:** Only a valid account (registration + email verification) and at least one registered user key are required. No admin or privileged access is needed.
- **Attack execution:** Create a large number of transactions via `POST /transactions`, then repeatedly call `GET /transactions/sign?page=1&size=1`. Each call triggers the full unbounded scan.
- **Organic amplification:** Even without a deliberate attacker, a large organization with many concurrent active transactions will naturally degrade this endpoint over time.
- The endpoint comment says "NO LONGER USED BY FRONT-END" but it remains fully active and unauthenticated callers are rejected with 401, meaning any authenticated user can reach it.

### Recommendation

Apply the database-level `take` limit **before** the per-row async work. The correct fix requires restructuring the query to push the signing-key check into the database or a pre-filter step:

1. **Short-term:** Add a hard `take` cap to the `repo.find()` call (e.g., `take: limit * SAFETY_FACTOR`) to bound the scan, accepting that `totalItems` will be approximate.
2. **Proper fix:** Move the signing-eligibility check into the SQL query (similar to how `getTransactionsToApprove` uses a recursive CTE to filter at the DB level), so that only the page-sized result set is returned and no per-row application-level loop is needed.
3. **Rate limiting:** Apply per-user rate limiting on this endpoint as a defense-in-depth measure.

### Proof of Concept

1. Register a user account and obtain a JWT token.
2. Register a user key for the account.
3. Create 10,000 transactions via `POST /transactions` (each in `WAITING_FOR_SIGNATURES` status).
4. Call `GET /transactions/sign?page=1&size=1` with the JWT token.
5. **Observed:** The server fetches all 10,000 transactions from the DB, runs `userKeysToSign` 10,000 times sequentially (each involving SDK deserialization + async cache lookups), and returns after an arbitrarily long delay.
6. Send 10–20 concurrent requests to the same endpoint.
7. **Observed:** API response times spike to tens of seconds or the process OOMs/crashes, making the service unavailable to all users.

  transaction: Transaction,

  if (!transaction) return [];

  /* Deserialize the transaction */

  // flatten the key list to an array of public keys
