### Title
Any Authenticated User Can Enumerate All Transaction Groups via Unguarded "Testing Only" Endpoint

### Summary
The `GET /transaction-groups` endpoint in `TransactionGroupsController` is explicitly marked "TESTING ONLY" in source code but is deployed in production with no user-scoping authorization. Any authenticated, verified user can call it and receive the full list of every transaction group in the system, bypassing the per-user access controls enforced on every other transaction-related endpoint. This is a direct analog to the external report's pattern: a function intended for a restricted caller (internal testing) that lacks the required access control check, allowing any caller to extract data they should not see.

### Finding Description

**Root cause — missing authorization on a deliberately unrestricted endpoint:** [1](#0-0) 

```typescript
/* TESTING ONLY: Get all transactions groups */
@Get()
@Serialize(TransactionGroupDto)
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroups();
}
```

The comment "TESTING ONLY" signals that this route was never intended for production use by arbitrary users. However, it is registered under the same `@Controller('transaction-groups')` class that carries only the standard authentication guards: [2](#0-1) 

```typescript
@Controller('transaction-groups')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class TransactionGroupsController {
```

`JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` only verify that the caller holds a valid, non-blacklisted JWT and that the account status is `NONE` (i.e., verified). They perform **no ownership or membership check**. [3](#0-2) 

Every other endpoint in the same controller correctly scopes results to the requesting user: [4](#0-3) 

```typescript
getTransactionGroup(
  @GetUser() user: User,
  @Param('id', ParseIntPipe) groupId: number,
  ...
): Promise<TransactionGroup> {
  return this.transactionGroupsService.getTransactionGroup(user, groupId, full ?? true);
}
```

The `getTransactionGroups()` handler takes **no `user` parameter** and passes nothing to the service, so the service returns every group in the database unconditionally.

**Exploit path:**
1. Attacker self-registers (or is any existing verified user).
2. Attacker authenticates via `POST /auth/login` and obtains a JWT.
3. Attacker sends `GET /transaction-groups` with the JWT in the `Authorization` header.
4. The server returns the complete list of all `TransactionGroup` records, including groups belonging to other users and containing their pending/executed transaction metadata.

### Impact Explanation

Transaction groups aggregate multiple sensitive `Transaction` records (Hedera account operations, token transfers, file updates, system operations). Full enumeration of all groups leaks:
- Which transactions exist and their statuses
- Which users are involved in which groups
- Organizational workflow structure and timing

This is an **unauthorized cross-tenant data disclosure**. A malicious insider or any compromised account can silently harvest the entire organizational transaction history without triggering any per-resource access check.

### Likelihood Explanation

The endpoint is reachable by any user who can complete the standard registration and login flow — no elevated privilege is required. The route is `GET /transaction-groups` with no additional path segment, making it trivially discoverable via API documentation (Swagger is enabled) or simple enumeration. The attacker needs only a valid JWT, which is obtained through the normal login process.

### Recommendation

Remove the endpoint entirely from production code, or — if the listing capability is genuinely needed — add explicit user-scoping:

```typescript
// Option 1: Remove the route (preferred)
// Delete the getTransactionGroups() handler.

// Option 2: Scope to the requesting user
@Get()
@Serialize(TransactionGroupDto)
getTransactionGroups(@GetUser() user: User): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroupsForUser(user);
}
```

Additionally, add an `AdminGuard` if the full-list view is only needed for administrative purposes.

### Proof of Concept

```bash
# Step 1: Login as any verified user
TOKEN=$(curl -s -X POST http://<host>:3000/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"attacker@example.com","password":"password"}' \
  | jq -r '.accessToken')

# Step 2: Enumerate ALL transaction groups in the system
curl -s -X GET http://<host>:3000/transaction-groups \
  -H "Authorization: Bearer $TOKEN"

# Expected result: full JSON array of every TransactionGroup record,
# including groups owned by other users, with their transaction metadata.
```

The response will contain transaction groups that the attacker did not create and was never added to as a member, demonstrating the complete bypass of the per-user access control enforced on `GET /transaction-groups/:id`. [1](#0-0)
