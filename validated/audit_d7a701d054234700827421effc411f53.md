## Audit Report

## Title
`removeTransactionApprover` Does Not Re-Validate Parent Threshold After Child Deletion, Enabling Approval Deadlock

## Summary
`removeTransactionApprover` in `approvers.service.ts` soft-deletes a node and all its descendants via `removeNode`, but never adjusts the parent threshold node's `threshold` value. This leaves the parent in a state where `threshold > remaining_children`, making the transaction permanently unapprovable without creator intervention. The asymmetry with `updateTransactionApprover` (which does perform this adjustment) confirms the omission is unintentional.

## Finding Description

**Root cause — constraint enforced at creation and on detach, but not on delete.**

At creation time, `createTransactionApprovers` enforces `threshold ≤ approvers.length`: [1](#0-0) 

When a child is *detached* via `updateTransactionApprover` (`listId: null`), the parent threshold is correctly re-evaluated: [2](#0-1) 

However, `removeTransactionApprover` takes a completely different path — it fetches the approver and calls `removeNode`, which issues a recursive SQL soft-delete with **no parent threshold adjustment**: [3](#0-2) 

`removeNode` only soft-deletes the target and its descendants; the parent row is never touched: [4](#0-3) 

The controller exposes this endpoint to any transaction creator with no transaction-status gate: [5](#0-4) 

**Secondary issue:** `removeTransactionApprover` emits `emitTransactionStatusUpdate` using `approver.transactionId`. For child nodes (non-root), `transactionId` is `null`, so the status-update event fires with `entityId: null`, silently failing to notify the correct transaction. [6](#0-5) 

## Impact Explanation

After a child leaf node is deleted, the parent threshold node retains its original `threshold` value while its child count decreases. The `isApproved` utility (frontend) and any backend equivalent evaluate:

```typescript
approvals.length >= (approver.threshold || approvals.length)
``` [7](#0-6) 

With `threshold=2` and only 1 remaining child, this condition can never be satisfied. The transaction is stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. Approvers who have already signed cannot recover the transaction on their own — only the creator can remediate by updating the threshold via `updateTransactionApprover`. If the creator is the one who triggered the deletion (intentionally or not), the transaction may remain frozen indefinitely.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user exercising a documented API endpoint (`DELETE /transactions/:transactionId/approvers/:id`). No elevated privileges, leaked credentials, or external dependencies are required. The only precondition is a threshold approver tree, which is a standard workflow. The `getCreatorsTransaction` check confirms identity but imposes no status gate, so the deletion is permitted at any point in the transaction lifecycle.

## Recommendation

In `removeTransactionApprover`, after calling `removeNode`, fetch the deleted node's parent (via `listId`) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If the parent has no remaining children after deletion → soft-remove the parent.
- If `newChildCount < parent.threshold` → update `parent.threshold = newChildCount`.

Additionally, fix the `emitTransactionStatusUpdate` call to resolve the correct `transactionId` by traversing to the root node (via `getRootNodeFromNode`) before emitting, so child-node deletions correctly notify the owning transaction.

## Proof of Concept

1. Creator creates transaction `T` and sets up a threshold approver tree: parent node `P` with `threshold=2`, children `[User A (id=10), User B (id=11)]`.
2. User A signs and approves. `P.threshold` remains `2`.
3. Creator calls `DELETE /transactions/T/approvers/10` (User A's leaf node).
4. `removeNode(10)` soft-deletes row `id=10`. Parent row `P` is untouched: `threshold=2`, but now has only 1 child (User B).
5. User B signs and approves. Backend evaluates `approvals.length (1) >= threshold (2)` → `false`.
6. Transaction `T` is permanently stuck — it can never reach the approved state without the creator manually correcting the threshold via `PATCH /transactions/T/approvers/P` with `{ threshold: 1 }`.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);
