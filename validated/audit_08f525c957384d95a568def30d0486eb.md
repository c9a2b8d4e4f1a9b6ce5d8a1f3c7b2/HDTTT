### Title
TOCTOU Race Condition in `archiveTransaction` Allows Overwriting Terminal Transaction Status

### Summary
`archiveTransaction` in `TransactionsService` performs a status check on a stale in-memory snapshot and then issues an unconditional `UPDATE` with no status guard in the `WHERE` clause. Concurrently, the Chain Service scheduler can execute the same transaction and set its status to `EXECUTED` or `FAILED`. The archive write wins the race and overwrites the terminal status, permanently corrupting the on-chain audit record.

### Finding Description

`archiveTransaction` follows a classic check-then-act pattern without atomicity:

**Step 1 — Read (CHECK):** [1](#0-0) 

The status is read from the database into a local variable. The guard verifies the status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

**Step 2 — Write (ACT) — no status guard:** [2](#0-1) 

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

The `WHERE` clause is `id = :id` only. There is no `AND status IN (...)` predicate. Any status present in the database at write time is unconditionally overwritten.

**The concurrent actor — Chain Service scheduler:**

The `handleTransactionsBetweenNowAndAfterThreeMinutes` cron fires every 10 seconds and calls `prepareTransactions`, which schedules `collateAndExecute` → `addExecutionTimeout` → `executeService.executeTransaction`. That path updates the row to `EXECUTED` or `FAILED`: [3](#0-2) 

**Race window:**

```
T0  API:   SELECT status = WAITING_FOR_EXECUTION  ← passes guard
T1  Chain: UPDATE status = EXECUTED               ← transaction submitted to Hedera
T2  API:   UPDATE status = ARCHIVED               ← overwrites EXECUTED
```

At T2 the database record says `ARCHIVED` while the Hedera network has the transaction as `EXECUTED`. The `executedAt` and `statusCode` columns written by the Chain Service at T1 remain in the row, but the `status` column is wrong.

**Contrast with the correctly guarded `cancelTransactionWithOutcome`:** [4](#0-3) 

That function adds `.andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })` so the write is a no-op if the status changed between read and write. `archiveTransaction` has no equivalent guard.

The same structural defect exists in `removeTransaction`: [5](#0-4) 

`repo.update(transaction.id, { status: TransactionStatus.CANCELED })` also carries no status guard.

### Impact Explanation

A transaction that was successfully submitted to and executed on the Hedera network will have its database status permanently set to `ARCHIVED` instead of `EXECUTED`. Downstream effects:

- The organization's audit log shows the transaction as archived (never executed), breaking compliance records.
- Any UI or API consumer that checks `status === EXECUTED` to confirm on-chain settlement will receive a false negative.
- The `executedAt` timestamp and `statusCode` remain in the row alongside the wrong `status`, creating an internally inconsistent record that is difficult to detect automatically.

### Likelihood Explanation

The Chain Service cron `handleTransactionsBetweenNowAndAfterThreeMinutes` runs every 10 seconds and schedules execution 5 seconds after `validStart`. [6](#0-5) 

A user who clicks "Archive" on a manual transaction near its `validStart` — a natural action when they decide not to execute it — races directly against the scheduler. In a multi-pod deployment (multiple Chain Service replicas), the execution window is wider and the probability increases. No special attacker capability is required; the creator of any manual transaction can trigger this with a normal UI action.

### Recommendation

Add a status guard to the `UPDATE` in `archiveTransaction`, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
// Replace:
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });

// With:
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];
const result = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  .execute();

if (!result.affected || result.affected === 0) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Apply the same fix to `removeTransaction`. This makes the write atomic with respect to the status check, eliminating the TOCTOU window.

### Proof of Concept

1. Create a manual transaction with a `validStart` 15 seconds in the future.
2. Wait until ~5 seconds before `validStart` (the Chain Service collate window begins at `validStart - 10s`).
3. Send `PATCH /transactions/archive/:id` at the exact moment the Chain Service fires `executeTransaction`.
4. Observe: the Hedera network receipt confirms execution; the database row has `status = ARCHIVED`, `executedAt` set, and `statusCode` set — an internally contradictory state.

The race is reliably reproducible by inserting a short artificial delay (e.g., `await sleep(200)`) between the status read and the `repo.update` call in a test environment, then triggering execution concurrently.
