### Title
Non-Active Users Can Be Added as Transaction Approvers Without Status Validation

### Summary

The `createTransactionApprovers` function in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` and the `createTransactionObservers` function in `back-end/apps/api/src/transactions/observers/observers.service.ts` do not validate that target users hold an active status (`UserStatus.NONE`) before assigning them as approvers or observers. This is the direct analog of the external report: just as `execute_redelegate_proxy` allowed redelegation to non-active validators, these functions allow assignment of non-active users to transaction roles.

### Finding Description

The system defines three user statuses in `back-end/libs/common/src/database/entities/user.entity.ts`:

```typescript
export enum UserStatus {
  NEW = 'NEW',             // created, password not yet set
  RESET_PASSWORD = 'RESET_PASSWORD', // in password-reset flow
  NONE = 'NONE',           // fully active/verified
}
```

`UserStatus.NONE` is the only "active" state. The `VerifiedUserGuard` enforces this for the *requesting* user:

```typescript
return user.status === UserStatus.NONE;
```

However, when the transaction creator adds approvers, `createTransactionApprovers` only checks that the target user **exists**, not that they are active:

```typescript
// approvers.service.ts lines 271-277
const userCount = await transactionalEntityManager.count(User, {
  where: { id: dtoApprover.userId },
});
if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
```

Similarly, `createTransactionObservers` performs no existence or status check at all on target user IDs:

```typescript
// observers.service.ts lines 49-53
for (const userId of dto.userIds) {
  if (!transaction.observers.some(o => o.userId === userId)) {
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);
  }
}
```

A user with `UserStatus.NEW` or `UserStatus.RESET_PASSWORD` cannot authenticate through the API (blocked by `VerifiedUserGuard`), and therefore cannot call `approveTransaction`. If such a user is added as a required approver in a threshold group, the transaction is permanently stuck waiting for their approval.

### Impact Explanation

If a non-active user is added as a required approver (e.g., as the sole member of a threshold-1 approver group), the transaction can never reach `WAITING_FOR_EXECUTION` or be executed. The approval workflow is permanently blocked for that transaction. The transaction creator cannot remove the approver after the fact without admin intervention, and the non-active user cannot authenticate to approve. This constitutes a Denial-of-Service on the transaction approval lifecycle.

### Likelihood Explanation

The transaction creator must be authenticated (active), but they can supply any integer `userId` in the approver DTO. A creator could accidentally add a newly-invited colleague (status `NEW`, who hasn't set their password yet) as a required approver. This is a realistic operational mistake, not a purely theoretical one, especially in organizations that invite users before they complete onboarding.

### Recommendation

In `createTransactionApprovers`, extend the user existence check to also validate `UserStatus.NONE`:

```typescript
const userCount = await transactionalEntityManager.count(User, {
  where: { id: dtoApprover.userId, status: UserStatus.NONE },
});
if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
```

In `createTransactionObservers`, add an analogous check for each `userId` in `dto.userIds` before creating the observer record.

### Proof of Concept

1. Register a new user (e.g., `newuser@example.com`). The user is created with `status: UserStatus.NEW` and has not set a password — they cannot authenticate.
2. As an authenticated admin/creator, create a transaction.
3. POST to `/transactions/:id/approvers` with body:
   ```json
   { "approversArray": [{ "userId": <newuser_id>, "threshold": null }] }
   ```
4. The request succeeds — the non-active user is now a required approver.
5. The transaction enters `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` but can never advance: the non-active user cannot call `POST /transactions/:id/approvers/approve` because `VerifiedUserGuard` rejects their token.
6. The transaction is permanently blocked.
