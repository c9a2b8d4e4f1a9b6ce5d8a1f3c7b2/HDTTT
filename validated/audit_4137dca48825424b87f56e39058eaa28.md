### Title
Missing Lower-Bound Check on Approver Threshold Allows Negative Threshold Values

### Summary
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the threshold validation for transaction approver trees checks that the threshold is not zero and not greater than the number of approvers, but never rejects negative values. This is the direct analog of M-06: the upper bound is enforced, the zero case is explicitly blocked, but the lower bound (`threshold > 0`) is absent — so any negative integer passes the guard and is persisted to the database.

### Finding Description
The root cause exists in two places in `approvers.service.ts`.

**Path 1 — `createTransactionApprovers`:** [1](#0-0) 

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
```

**Path 2 — `updateTransactionApprover`:** [2](#0-1) 

```typescript
if (
  approver.approvers &&
  (dto.threshold > approver.approvers.length || dto.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(approver.approvers.length));
```

For a submitted `threshold = -1`:
- `-1 > approvers.length` → `false` → upper-bound check passes.
- `-1 === 0` → `false` → zero check passes.

The value is then written to the `TransactionApprover` row via `transactionalEntityManager.insert` / `transactionalEntityManager.update`. [3](#0-2) 

The error message itself reveals the intent: `"Threshold must be less or equal to the number of approvers (${total}) and not 0"` — the developer intended to block 0 but forgot that any integer below 0 is equally invalid. [4](#0-3) 

### Impact Explanation
The `threshold` field stored in `transaction_approver` is consumed by `TransactionSignatureService` to decide whether an approver tree is satisfied. A negative threshold (e.g., `-1`) means the required count of approvals is permanently below any realistic approval count, so the tree is treated as satisfied with zero approvals. A transaction creator can therefore construct an approver tree that appears to require multi-party sign-off but is silently auto-satisfied, bypassing the approval gate entirely and allowing the transaction to advance to execution without any approver action.

### Likelihood Explanation
The attacker must be an authenticated user who is the creator of a transaction — a normal, non-privileged role reachable by any registered user. The exploit requires only a single crafted API call to the `POST /transactions/:id/approvers` or `PATCH /transactions/:id/approvers/:approverId` endpoint with `"threshold": -1`. No admin keys, no leaked secrets, and no race conditions are required.

### Recommendation
Replace the equality check `=== 0` with a proper lower-bound check in both locations:

```typescript
// Before (broken)
dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0

// After (correct)
dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold < 1
```

Apply the same fix to the `updateTransactionApprover` path at line 475. Additionally, add a `@Min(1)` decorator to the `threshold` field in `create-transaction-approver.dto.ts` and `update-transaction-approver.dto.ts` so the constraint is enforced at the HTTP boundary before reaching service logic.

### Proof of Concept

1. Authenticate as a normal user and create a transaction (obtain `transactionId`).
2. POST to `/transactions/{transactionId}/approvers` with body:
   ```json
   {
     "approversArray": [{
       "threshold": -1,
       "approvers": [
         { "userId": 2 },
         { "userId": 3 }
       ]
     }]
   }
   ```
3. The service evaluates `-1 > 2` → false and `-1 === 0` → false; both guards pass.
4. A `TransactionApprover` row is inserted with `threshold = -1`.
5. The `TransactionSignatureService` evaluates the tree: the number of actual approvals (0) satisfies the threshold (-1), so the approver tree is marked as satisfied without any approver acting.
6. The transaction proceeds to execution with zero real approvals collected. [5](#0-4) [6](#0-5)
