### Title
Race Condition in `approveTransaction` Allows Replaced Approver to Corrupt Organizational Approval State

### Summary
`approveTransaction` in `approvers.service.ts` reads the approver list and performs its "already approved" guard **outside** a database transaction, then writes the approval **inside** a separate database transaction using stale IDs. If the transaction creator replaces approver Bob with Charlie between Bob's read and Bob's write, Bob's signature and `approved=true` are committed to Charlie's approver record — corrupting the organizational approval workflow and allowing a transaction to advance without Charlie's actual consent.

### Finding Description

**Root cause — TOCTOU between read and write in `approveTransaction`:** [1](#0-0) 

The approver list is fetched at line 553 and the "already approved" guard fires at line 563 — both **outside** any database transaction. [2](#0-1) 

The write at lines 599–610 opens a **separate** DB transaction and updates rows using `whereInIds(userApprovers.map(a => a.id))` — the IDs captured from the stale read. There is no re-validation of `userId` ownership inside the write transaction.

**Concurrent mutation path — `updateTransactionApprover` changes `userId` with no transaction-status guard:** [3](#0-2) 

When the creator replaces Bob with Charlie, the record's `userId`, `signature`, and `approved` are cleared — but the record's primary key (`id`) is unchanged. `updateTransactionApprover` contains no check that the transaction is in a terminal or locked status, so this mutation can happen at any time while the transaction is `WAITING_FOR_SIGNATURES`.

**Race window:**

```
Bob calls approveTransaction
  → line 553: reads approver record {id:5, userId:Bob, signature:null}
  → line 563: passes guard (no signature yet)
  ← [PAUSE]

Creator calls updateTransactionApprover({userId: Charlie})
  → DB transaction commits: record {id:5, userId:Charlie, signature:null, approved:null}

Bob's request resumes
  → line 599: opens DB transaction
  → whereInIds([5]) — same ID, now owned by Charlie
  → writes {userKeyId:Bob's key, signature:Bob's sig, approved:true}
  → commits

Result: record {id:5, userId:Charlie, signature:Bob's sig, approved:true}
```

### Impact Explanation

Charlie's approver record now carries Bob's cryptographic signature and `approved=true` without Charlie ever acting. The organizational approval workflow — which is the tool's mechanism for multi-party authorization of Hedera transactions — is corrupted:

- The system considers Charlie's approval slot satisfied.
- If the approval threshold is met by this corrupted state, the transaction advances to `WAITING_FOR_EXECUTION` without Charlie's actual consent.
- The creator's intent (requiring Charlie's approval) is silently bypassed. [4](#0-3) 

The notification and status-update events fire based on the now-corrupted `approved` values, propagating the false state to all observers.

### Likelihood Explanation

- Bob is a **normal authenticated user** — no privileged access required.
- Bob only needs to submit his approval request while the creator's `updateTransactionApprover` call is in-flight. In a multi-user organization, Bob may be notified (via WebSocket/email) that he is being replaced and can act immediately.
- The race window spans a full async I/O round-trip (DB read → network → DB write), making it wide enough to exploit deliberately or accidentally under load.
- No blockchain front-running is required; this is a standard HTTP API race against a PostgreSQL backend.

### Recommendation

Wrap the entire `approveTransaction` flow — read, guard, and write — in a **single serializable database transaction**, and re-validate that the approver record's `userId` still matches the requesting user inside that transaction before committing:

```typescript
await this.dataSource.transaction('SERIALIZABLE', async em => {
  const userApprovers = (await this.getVerifiedApproversByTransactionId(transactionId, user, em))
    .filter(a => a.userId === user.id);

  if (userApprovers.length === 0) throw new UnauthorizedException(...);
  if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

  // ... verify signature ...

  await em.createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId, signature, approved })
    .where('id IN (:...ids) AND "userId" = :userId', {
      ids: userApprovers.map(a => a.id),
      userId: user.id,
    })
    .execute();
});
```

Adding `AND "userId" = :userId` to the `WHERE` clause ensures that if the record was reassigned between the read and the write, the update affects zero rows and the operation fails safely.

Additionally, `updateTransactionApprover` should reject approver changes once the transaction has left `NEW` status, or at minimum re-check and clear any in-progress approval state atomically.

### Proof of Concept

1. Creator creates transaction T with approver Bob (approver record `id=5`).
2. Bob sends `POST /transactions/T/approvers/approve` with `{userKeyId: BobKey, signature: BobSig, approved: true}`.
3. Before the server commits Bob's write, creator sends `PATCH /transactions/T/approvers/5` with `{userId: CharlieId}`.
4. Creator's request commits first: record `id=5` → `{userId: Charlie, signature: null, approved: null}`.
5. Bob's request commits: record `id=5` → `{userId: Charlie, signature: BobSig, approved: true}`.
6. Query the DB: `SELECT * FROM transaction_approver WHERE id = 5` — observe `userId=Charlie`, `approved=true`, `signature=<Bob's bytes>`.
7. The approval threshold is satisfied; the transaction advances without Charlie's consent.
