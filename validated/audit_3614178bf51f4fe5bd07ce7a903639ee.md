Audit Report

## Title
IDOR in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without confirming that approver belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Controller handler** — `removeTransactionApprover` in `approvers.controller.ts`: [1](#0-0) 

The two service calls are decoupled: `getCreatorsTransaction` validates ownership of `:transactionId`, but `removeTransactionApprover` acts on `:id` with no cross-reference.

**`getCreatorsTransaction`** only checks that the caller's user ID matches the creator of the transaction referenced by the URL's `:transactionId` parameter — it performs no check on the approver: [2](#0-1) 

**`removeTransactionApprover` (service)** fetches the approver by `:id` and soft-deletes it with no assertion that `approver.transactionId === transactionId`: [3](#0-2) 

**`removeNode`** performs a recursive soft-delete purely by approver ID, with no transaction ownership guard: [4](#0-3) 

**The fix already exists in `updateTransactionApprover`**, which correctly resolves the root node and cross-checks its `transactionId` against the URL parameter before proceeding: [5](#0-4) 

The delete path simply omits this guard entirely.

## Impact Explanation
- **Bypass multi-sig approval gates**: An attacker can soft-delete all required approvers from a victim's transaction, causing the approval threshold to be unmet and preventing execution or corrupting approval state.
- **Sabotage organizational workflows**: In an organization context, an attacker can silently strip approvers from any pending transaction, causing it to stall or execute without the intended oversight.
- **Unauthorized state mutation**: The attacker modifies another user's transaction state without any authorization over that transaction.

## Likelihood Explanation
- **Attacker precondition**: Only requires a valid authenticated account and the ability to create one transaction — a standard, unprivileged product flow.
- **Approver ID discovery**: IDs are auto-incremented integers; an attacker can enumerate them trivially or observe IDs from their own transactions to estimate the range.
- The attack is silent — the victim receives a status-update notification but no indication that an unauthorized deletion occurred.

## Recommendation
In `removeTransactionApprover` (service), after fetching the approver, resolve its root node and assert that `rootNode.transactionId === transactionId` before calling `removeNode` — mirroring the guard already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the authorized transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

The controller must also pass `transactionId` to the service call:
```typescript
await this.approversService.removeTransactionApprover(id, transactionId);
```

## Proof of Concept

**Setup:**
- Attacker creates `Transaction A` (attacker is creator).
- Victim has `Transaction B` with approver node `id = 42` (a required multi-sig approver).

**Attack:**
```
DELETE /transactions/<A_id>/approvers/42
Authorization: Bearer <attacker_token>
```

**Execution flow:**
1. `getCreatorsTransaction(A_id, attacker)` — passes, attacker owns Transaction A.
2. `removeTransactionApprover(42)` — fetches approver 42 (belongs to Transaction B), calls `removeNode(42)`, soft-deletes it. No cross-check is performed.

**Result:** Approver 42 is deleted from Transaction B. The victim's multi-sig requirement is silently bypassed.

          from transaction_approver

              from transaction_approver, approversToDelete

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
