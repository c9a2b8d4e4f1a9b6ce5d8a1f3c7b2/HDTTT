### Title
Unbounded Loop Over Dynamic Transaction Array in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** matching transactions from the database without any row limit, then iterates over every result performing an async database operation (`userKeysToSign`) per transaction before applying pagination. A normal authenticated user who accumulates a large number of transactions (as creator, signer, or observer) can trigger repeated calls to this endpoint, exhausting server CPU, memory, and database connection pool resources. A separate but related issue exists in `importSignatures`, where the developer explicitly acknowledged a missing input-size limit.

### Finding Description

**Root cause 1 — `getTransactionsToSign` (primary)** [1](#0-0) 

```
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});                                          // ← no `take` limit

for (const transaction of transactions) {   // ← unbounded loop
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    ...
  }
}

return {
  items: result.slice(offset, offset + limit),  // pagination applied AFTER full scan
  ...
};
```

Every other paginated query in the service passes `skip`/`take` to the ORM. This one does not. The full result set is loaded into memory, then each element triggers `userKeysToSign` → `userKeysRequiredToSign`, which itself performs additional database queries (key lookups, signature checks). Pagination is applied only after all N async operations complete. [2](#0-1) 

**Root cause 2 — `importSignatures` (secondary, developer-acknowledged)** [3](#0-2) 

The comment at line 575 reads: *"Added a batch mechanism, probably should limit this on the api side of things."* The function accepts `dto: UploadSignatureMapDto[]` with no enforced maximum length. For each element it deserializes transaction bytes (`SDKTransaction.fromBytes`), validates cryptographic signatures, and builds a dynamic SQL `CASE` statement — all proportional to the caller-controlled array size. [4](#0-3) 

### Impact Explanation

- **`getTransactionsToSign`**: In an organization with thousands of staged transactions, a single API call loads all of them into the Node.js process and fires N sequential async DB queries. Repeated calls from one or more authenticated users can saturate the database connection pool, spike server memory, and make the endpoint unresponsive for all users (service unavailability).
- **`importSignatures`**: A caller submitting an array of thousands of IDs forces the server to perform thousands of cryptographic deserialization and signature-verification operations in a single request, causing CPU exhaustion.

Both paths are reachable by any authenticated user with no elevated privileges.

### Likelihood Explanation

- Any registered user can call `getTransactionsToSign`. In organization mode, a user can be added as observer or signer to an arbitrary number of transactions by other users, growing the array without the victim user's control.
- `importSignatures` accepts a plain JSON array; no schema-level maximum is enforced. A malicious user submits a crafted request with a large array.
- No rate limiting or input-size guard is visible in the service layer for either path.

### Recommendation

1. **`getTransactionsToSign`**: Apply a database-level limit before the loop. Fetch only the page-relevant rows, or cap the query with a configurable `MAX_TRANSACTIONS_TO_SCAN` constant and return an error if exceeded.

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: MAX_SCAN_LIMIT,   // add this
});
```

2. **`importSignatures`**: Enforce a maximum array length at the DTO validation layer (e.g., `@ArrayMaxSize(100)` via `class-validator`) and at the controller before delegating to the service. The developer comment at line 575 already flags this as a known gap. [5](#0-4) 

### Proof of Concept

**`getTransactionsToSign` DoS:**

1. Register as a normal user (User A).
2. Have another user (or the attacker themselves via multiple accounts) create or assign User A as observer/signer on 10,000 transactions.
3. Call `GET /transactions/sign` (the endpoint backed by `getTransactionsToSign`) repeatedly in a loop.
4. Each call causes the server to: (a) load all 10,000 transaction rows into memory, (b) execute `userKeysToSign` (DB query) for each of the 10,000 rows sequentially.
5. Observe: server response times degrade for all users; database connection pool is exhausted; Node.js heap grows proportionally to the number of transactions.

**`importSignatures` DoS:**

1. Register as a normal user.
2. Craft a POST request to the signature import endpoint with a JSON body containing an array of 5,000 arbitrary transaction IDs.
3. The server iterates over all 5,000 entries, calling `SDKTransaction.fromBytes` and `validateSignature` for each valid transaction found.
4. Observe: CPU spikes to 100% for the duration of the request; concurrent requests are starved.
