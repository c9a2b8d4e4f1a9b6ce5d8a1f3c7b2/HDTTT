Audit Report

## Title
XSS via Unsanitized `transactionMemo` and `description` Rendered with `v-html` in `CreateTransactionGroup.vue`

## Summary
`CreateTransactionGroup.vue` renders `groupItem.description` and `transactionMemo` (extracted from raw transaction bytes) directly into the DOM via Vue's `v-html` directive without any sanitization. Both values are attacker-controlled in an organization context. This enables HTML/CSS injection for UI deception and, in development mode (no CSP), JavaScript execution in the renderer process.

## Finding Description

The `v-html` binding at lines 421–429 of `CreateTransactionGroup.vue` injects one of four values as raw HTML: [1](#0-0) 

The two attacker-controlled branches are:
- `groupItem.description` — populated from organization backend data
- `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` — a free-text field (up to 100 chars) embedded in serialized transaction bytes by any organization member

Neither value is HTML-escaped before being passed to `v-html`. Vue's `v-html` bypasses its normal text-escaping and injects the string as raw HTML.

The `makeTransfer` branch calls `formatHbarTransfers`, which constructs its output from SDK-typed `AccountId` and `Hbar` objects — not from free-text fields — so that path does not introduce attacker-controlled injection. [2](#0-1) 

Importantly, the **confirm/sign modal** in `TransactionGroupProcessor.vue` renders the same fields using safe mustache interpolation (`{{ }}`), not `v-html`: [3](#0-2) 

So the injection surface is the group-building list view, not the final confirmation step.

## Impact Explanation

**UI deception:** An attacker crafts a `transactionMemo` containing HTML such as `<span style="display:none">0.0.9999 → 1000 HBAR</span><span>0.0.1234 → 0.001 HBAR</span>`. The victim sees fabricated transfer details in the group item list and may proceed to sign a transaction they did not intend to authorize. The confirm modal shows the real data via safe interpolation, but a user who trusts the list view may not scrutinize the modal.

**JavaScript execution (development mode only):** In production, a `Content-Security-Policy: script-src 'self'` header is applied via `onHeadersReceived`, which blocks both inline `<script>` tags and inline event handlers (`onerror`, `onclick`, etc.): [4](#0-3) 

This CSP is **not applied in development mode** (`if (!is.dev)`), so JavaScript injection executes in dev builds.

**Electron renderer escalation is limited:** Even if JavaScript executes, the Electron window is configured with `nodeIntegration: false` and `contextIsolation: true`: [5](#0-4) 

The preload uses `contextBridge` correctly: [6](#0-5) 

This means injected renderer-world JavaScript cannot directly access Node.js APIs or IPC. The RCE escalation claim is not achievable under the current configuration. However, `sandbox: false` means the renderer process retains OS-level privileges, so any future weakening of `contextIsolation` would immediately enable full Node.js access.

## Likelihood Explanation

Any authenticated organization member can set an arbitrary `transactionMemo` (the only constraint is a client-side `maxlength="100"`). The malicious bytes are stored in the backend and fetched by any other member who opens the Create Transaction Group page. No privileged role is required. The attack is passive — the victim only needs to open the page.

For personal (non-organization) use, the user controls their own `description` and `transactionMemo`, making this self-XSS, which is out of scope per `SECURITY.md`.

## Recommendation

Replace `v-html` with safe text interpolation (`{{ }}`) for all user-controlled fields. The `makeTransfer` path (which uses `v-html` for structured SDK data) should also be refactored to use a component-based approach rather than raw HTML injection. If HTML rendering is genuinely required for the transfer display, sanitize the output with a library such as DOMPurify before passing it to `v-html`.

## Proof of Concept

1. As an organization member, create a transaction with `transactionMemo` set to:
   ```
   <b style="color:red">0.0.9999 → 1000 HBAR</b>
   ```
2. Add this transaction to a group and share it with the organization.
3. A second organization member opens the Create Transaction Group page.
4. The group item list renders the bold red text as HTML, not as the literal string.
5. In development mode (no CSP), replace the payload with `<img src=x onerror="alert(document.cookie)">` to confirm JavaScript execution.

```typescript
export function formatHbarTransfers(transfers: Transfer[]): string {
  if (transfers.length === 0) {
    return 'No transfers';
  }

  if (transfers.length === 1) {
    const amount = transfers[0].amount;
    if (amount.isNegative()) {
      return 'Missing receiver';
    } else {
      return 'Missing sender';
    }
  }

  if (transfers.length === 2) {
    // the JS SDK sorts the order of the transfers by account ID. We don't want this. We want the sender to be on the
    // left and the receiver to be on the right. So we need to check if the amount is negative or positive and then
    // arrange the transfers accordingly
    let sender = transfers[0];
    let receiver = transfers[1];
    if (receiver.amount.isNegative()) {
      sender = transfers[1];
      receiver = transfers[0];
    }
    return `${sender.accountId} --> ${stringifyHbarWithFont(receiver.amount)} --> ${receiver.accountId}`;
  }

  return 'Multiple transfers';
```
