### Title
Inconsistency Between Create and Update Paths for `threshold` Validation in `ApproversService` Allows Negative Threshold to Be Persisted

### Summary
`UpdateTransactionApproverDto` enforces `@Min(1)` on the `threshold` field, preventing zero or negative values at the DTO layer. `CreateTransactionApproverDto` has no such constraint, and the service-level guard in `createTransactionApprovers` only rejects `threshold === 0`, silently allowing negative threshold values (e.g., `-1`) to be written to the database. This is a direct analog of the reported M-07 pattern: the setter (update path) has a validation the constructor (create path) lacks.

### Finding Description

**Root cause — DTO-level inconsistency:**

`UpdateTransactionApproverDto` (update path): [1](#0-0) 

```typescript
@IsNumber()
@Min(1)          // ← enforces threshold ≥ 1
@IsOptional()
threshold?: number;
```

`CreateTransactionApproverDto` (create path): [2](#0-1) 

```typescript
@IsNumber()
@IsOptional()
threshold?: number;   // ← no @Min(1), accepts any integer including negatives
```

**Root cause — service-level guard is incomplete:**

The only threshold range check in `createTransactionApprovers` is: [3](#0-2) 

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(...));
```

The condition `dtoApprover.threshold === 0` catches zero but **not negative values**. For `threshold = -1` with `approvers = [{userId: 1}]`:
- `-1 > 1` → `false`
- `-1 === 0` → `false`

Both branches are false, so the check passes and the record is inserted with `threshold = -1`. [4](#0-3) 

The update path, by contrast, is protected at the DTO layer by `@Min(1)` before the request even reaches the service. [5](#0-4) 

### Impact Explanation

A negative threshold stored in `transaction_approver.threshold` corrupts the approval-tree invariant. The threshold value is used in comparisons such as: [6](#0-5) 

```typescript
} else if (newParentApproversLength < parent.threshold) {
  await transactionalEntityManager.update(TransactionApprover, parent.id, {
    threshold: newParentApproversLength,
  });
}
```

With `threshold = -1`, `newParentApproversLength < -1` is always `false` for any non-negative count, so the auto-correction branch never fires. Downstream approval-evaluation logic that checks `approvals >= threshold` would trivially satisfy the condition for any number of approvals (including zero), effectively bypassing the multi-party approval requirement for that tree node. This constitutes an unauthorized state change: a transaction requiring N-of-M approval can be made to appear approved without collecting the required signatures.

### Likelihood Explanation

Any authenticated user who is the creator of a transaction can call the `POST /transactions/:id/approvers` endpoint with a crafted body such as:

```json
{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [{ "userId": 1 }, { "userId": 2 }]
    }
  ]
}
```

No privileged access is required beyond being the transaction creator, which is a normal user role. The NestJS global `ValidationPipe` will not reject the request because `CreateTransactionApproverDto.threshold` carries no `@Min` constraint. [7](#0-6) 

### Recommendation

Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
@IsNumber()
@Min(1)          // add this
@IsOptional()
threshold?: number;
```

Additionally, update the service-level guard to reject `threshold < 1` (not just `=== 0`) as a defense-in-depth measure: [3](#0-2) 

```typescript
(dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold < 1)
```

### Proof of Concept

1. Authenticate as a normal user and create a transaction (obtain `transactionId`).
2. POST to `/transactions/:transactionId/approvers` with:
   ```json
   {
     "approversArray": [
       {
         "threshold": -1,
         "approvers": [{ "userId": 1 }, { "userId": 2 }]
       }
     ]
   }
   ```
3. The request passes DTO validation (no `@Min` on `threshold` in `CreateTransactionApproverDto`) and passes the service check (`-1 > 2` is false, `-1 === 0` is false).
4. A `TransactionApprover` row is inserted with `threshold = -1`.
5. Any downstream approval-count comparison of the form `approvedCount >= threshold` evaluates to `true` for `approvedCount = 0`, bypassing the intended multi-party approval gate.
