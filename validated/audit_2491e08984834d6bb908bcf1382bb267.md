### Title
Unbounded Full-Table Scan with Per-Row Async Crypto in `getTransactionsToSign` Causes Application-Layer DoS

### Summary
`getTransactionsToSign` in the API service fetches **every** non-terminal transaction from the database without any row limit, then performs a per-row async cryptographic operation and a database query for each one. Any authenticated user can trigger this endpoint repeatedly, causing CPU, memory, and database exhaustion that grows linearly with the total number of active transactions in the system.

### Finding Description

**Root cause — no `take` on the database query:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` calls `this.repo.find()` with no `take` (limit) option:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no take / limit
});
``` [1](#0-0) 

This loads **all** rows matching the status filter (every transaction not in EXECUTED / FAILED / EXPIRED / CANCELED / ARCHIVED) into application memory in a single query.

**Per-row async work:**

For every transaction returned, the code calls `userKeysToSign`, which:
1. Deserializes raw transaction bytes via `SDKTransaction.fromBytes` (CPU-bound)
2. Calls `transactionSignatureService.computeSignatureKey` (async, potentially mirror-node or crypto work)
3. Issues a `entityManager.find(UserKey, ...)` database query

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign` → `keysRequiredToSign`, which performs the deserialization and DB lookup per call: [3](#0-2) 

**Pagination is applied only after all work is done:**

The `PaginationParams` decorator correctly caps `size` at 100: [4](#0-3) 

But the pagination slice happens **after** the full scan and per-row processing:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ← sliced after full scan
  ...
};
``` [5](#0-4) 

**The endpoint is still publicly reachable:**

The controller exposes `GET /transactions/sign` to any authenticated user. The comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end dropped it, but the route remains live: [6](#0-5) 

### Impact Explanation

With N active transactions in the system, a single request to `GET /transactions/sign` causes:
- One unbounded `SELECT` loading all N transaction rows (including `transactionBytes` blobs) into memory
- N sequential async operations: byte deserialization + crypto key computation + DB query each

An attacker (any authenticated user) sending repeated requests drives CPU, heap, and DB connection usage to O(N) per request. As the platform accumulates transactions over time, each request becomes more expensive. At sufficient scale this exhausts the Node.js event loop, causes OOM, or saturates the PostgreSQL connection pool, making the API unavailable to all users.

### Likelihood Explanation

- **Attacker precondition**: valid session token only — no admin, no special role
- **Trigger**: a single HTTP GET to `/transactions/sign?page=1&size=1`
- **Amplification**: the attacker pays only for the HTTP request; the server pays O(N) per request
- **Persistence**: the endpoint is permanently exposed; the front-end no longer uses it, so there is no legitimate traffic to distinguish from abuse
- **No rate-limit evidence found** in the codebase that would block repeated calls to this specific endpoint

### Recommendation

1. **Apply the database limit before processing**, not after:
   ```typescript
   const transactions = await this.repo.find({
     where: whereForUser,
     relations: ['groupItem'],
     order,
     take: limit,   // ← add this
     skip: offset,
   });
   ```
   This requires moving the signing-eligibility check into the SQL query (as done for `getTransactionsToApprove` with the recursive CTE) so the DB can filter and paginate correctly.

2. **Remove or gate the unused endpoint.** Since the front-end no longer calls `/transactions/sign`, either delete the route or protect it with an admin guard until it is properly refactored.

3. **Add a hard server-side cap** on the number of transactions processed per request, independent of the pagination parameter.

### Proof of Concept

**Precondition**: authenticated user account; system has accumulated ≥ 1 000 active (non-terminal) transactions (realistic in any production multi-org deployment).

**Steps**:
1. Obtain a valid JWT via `POST /auth/login`.
2. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <token>
   ```
3. Each request causes the server to:
   - Load all N transaction rows (with `transactionBytes` blobs) from PostgreSQL
   - Deserialize each blob and compute signature keys N times
   - Issue N additional `UserKey` queries to PostgreSQL
4. Observe: API response latency increases proportionally with N; under sustained load the Node.js process heap grows unboundedly and the PostgreSQL connection pool saturates, causing 503 errors for all users.

**File:** back-end/libs/common/src/utils/transaction/index.ts (L20-94)
```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
```
