Audit Report

## Title
IDOR in `removeTransactionApprover`: Any Authenticated User Can Delete Approvers from Any Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint validates that the caller owns the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that this approver belongs to `:transactionId`. An attacker who owns any one transaction can delete approvers from any other transaction by supplying their own `transactionId` (to pass the creator check) and an arbitrary victim approver `id`.

## Finding Description

The controller handler performs two independent, uncorrelated operations:

```typescript
// approvers.controller.ts, lines 102–113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only validates that `transaction.creatorKey.userId === user.id` for the supplied `transactionId` — it has no knowledge of the approver `id`: [2](#0-1) 

`removeTransactionApprover` fetches the approver by its own `id` and immediately soft-deletes it with no check that `approver.transactionId` matches the `:transactionId` URL parameter: [3](#0-2) 

`getTransactionApproverById` performs a plain lookup by `id` only, with no ownership filter: [4](#0-3) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-ownership check before acting:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [5](#0-4) 

The `removeTransactionApprover` path is entirely missing this guard.

## Impact Explanation

1. **Bypass approval requirements** — an attacker can delete all approvers from a victim's transaction, potentially allowing it to proceed to `WAITING_FOR_EXECUTION` without the intended governance threshold being met.
2. **Recursive subtree destruction** — `removeNode` performs a recursive SQL soft-delete of the entire approver subtree rooted at the given `id`, so a single request can wipe a multi-level approval structure in one call.
3. **Unauthorized state transitions** — `emitTransactionStatusUpdate` is called after deletion, which can trigger downstream state changes in the Chain Service based on the now-corrupted approver state. [6](#0-5) [7](#0-6) 

## Likelihood Explanation

- **Attacker preconditions**: Only a valid authenticated account with a verified JWT. No admin or privileged role is required.
- **Approver ID discovery**: Approver IDs are sequential integers. An attacker can observe IDs returned from their own transactions via `POST /transactions/:id/approvers` or `GET /transactions/:id/approvers`, then enumerate adjacent IDs belonging to other transactions.
- **Attack execution**: A single HTTP `DELETE` request — `DELETE /transactions/{attacker_transactionId}/approvers/{victim_approver_id}` — is sufficient to exploit this.

## Recommendation

In `removeTransactionApprover` (service), add the same cross-ownership check that `updateTransactionApprover` already performs before calling `removeNode`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Traverse to root and verify it belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover`. This mirrors the existing pattern in `updateTransactionApprover`. [8](#0-7) 

## Proof of Concept

**Setup:**
- Attacker controls `userA` (authenticated).
- Victim `userB` owns `transactionId=99` with approver `id=42`.

**Attack:**
```http
DELETE /transactions/1/approvers/42
Authorization: Bearer <userA_jwt>
```

Where `transactionId=1` is any transaction owned by `userA`.

**Execution trace:**
1. `getCreatorsTransaction(1, userA)` — passes, because `userA` owns transaction `1`.
2. `removeTransactionApprover(42)` — fetches approver `42` (which belongs to transaction `99`), calls `removeNode(42)`, recursively soft-deletes the entire subtree, and emits a status update for transaction `99`.
3. Response: `true` — no error, no indication to the victim.

**Result:** Approver `42` and its entire subtree are deleted from `userB`'s transaction `99` without `userB`'s knowledge or consent.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager

      : this.dataSource.manager.findOne(Transaction, find));
