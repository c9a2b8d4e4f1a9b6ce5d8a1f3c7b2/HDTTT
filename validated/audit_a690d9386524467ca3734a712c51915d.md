Audit Report

## Title
Unbounded In-Memory Load in `getTransactionsToSign` Enables OOM via Authenticated Transaction Accumulation

## Summary
`TransactionsService.getTransactionsToSign` issues an unbounded `SELECT` against the entire non-terminal transaction table, loading every matching row — including `transactionBytes` blobs — into the Node.js heap before applying pagination in-memory. Because the query is not scoped to the requesting user, and because `CreateTransactionGroupDto` imposes no upper bound on `groupItems`, an authenticated user can accumulate a large number of non-terminal transactions and trigger heap exhaustion with a single `GET /transactions/sign` request.

## Finding Description

**Root cause — missing DB-level `take` in `getTransactionsToSign`:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `find` call at lines 295–299 carries no `take` clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The `limit` and `offset` values from `@PaginationParams()` are only applied after the full result set is already resident in memory:

```typescript
items: result.slice(offset, offset + limit),
``` [2](#0-1) 

**`whereForUser` is not scoped to the requesting user:**

The filter at lines 266–277 only excludes terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does not filter by `creatorKeyId`, `userId`, or any user-specific column, so the query returns every non-terminal transaction from every user in the database. [3](#0-2) 

**Endpoint is still active:**

Despite the comment `/* NO LONGER USED BY FRONT-END */`, the route `GET /sign` is registered and protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — any verified organisation user can call it. [4](#0-3) 

**Transaction accumulation amplifier — unbounded `groupItems`:**

`CreateTransactionGroupDto` has no `@ArrayMaxSize` decorator on `groupItems`, so a single `POST /transaction-groups` request can create an arbitrary number of transactions (bounded only by the HTTP body limit). [5](#0-4) 

Each transaction persists a `transactionBytes` blob of up to 6 144 bytes for normal payers and up to 131 072 bytes for privileged payers (accounts 0.0.2 and 0.0.42–0.0.799). [6](#0-5) 

**Per-user rate limit:**

The `UserThrottlerModule` caps authenticated users at 100 requests per minute and 10 per second. [7](#0-6) 

**`PaginationParams` enforces a maximum page size of 100**, but this cap is irrelevant because the full table scan occurs before any slicing. [8](#0-7) 

## Impact Explanation

A single `GET /transactions/sign` request causes the API to issue an unbounded `SELECT … FROM transaction` with no `LIMIT`, loading every non-terminal transaction row — including binary `transactionBytes` and `unsignedTransactionBytes` columns — into the Node.js heap. With sufficient accumulated rows, the process exhausts available heap and crashes, making the entire API unavailable to all users. Because the query is global (not scoped to the requesting user), one attacker's accumulated data affects every concurrent request.

## Likelihood Explanation

The attacker requires only a valid, verified organisation account — no admin or privileged role. The `POST /transaction-groups` endpoint multiplies the creation rate: a single request with a large `groupItems` array (no server-side count cap) creates many non-terminal transactions while consuming only one rate-limit token. Transactions whose `validStart` is set slightly in the future remain in a non-terminal status until the chain service polls and marks them expired, providing a sustained accumulation window. The vulnerable `GET /sign` endpoint is reachable by any verified user.

## Recommendation

1. **Add a DB-level `take` to `getTransactionsToSign`** — pass `take: limit` and `skip: offset` directly into the `find` call, and use `findAndCount` to obtain the total. This is the same pattern already used in `getHistoryTransactions` and `getTransactionsToApprove`. [9](#0-8) 

2. **Scope `whereForUser` to the requesting user** — add a join or sub-query condition so only transactions where the user holds a relevant signing key are returned at the database level.

3. **Add `@ArrayMaxSize` to `groupItems`** in `CreateTransactionGroupDto` to cap the number of transactions that can be created in a single group request. [5](#0-4) 

4. **Consider removing or formally deprecating `GET /sign`** since it is marked "NO LONGER USED BY FRONT-END" and the replacement endpoint (`GET /transaction-nodes?collection=READY_TO_SIGN`) already exists and uses a properly scoped SQL query.

## Proof of Concept

```
# Step 1 – Accumulate non-terminal transactions
for i in $(seq 1 N); do
  curl -X POST https://<host>/transaction-groups \
    -H "Authorization: Bearer <verified_user_token>" \
    -H "Content-Type: application/json" \
    -d '{"description":"x","groupItems":[<large array of valid signed transactions>]}'
done

# Step 2 – Trigger unbounded full-table scan into heap
curl -X GET "https://<host>/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer <verified_user_token>"
# → API process exhausts heap and crashes (OOM)
```

The `find` call at line 295 of `transactions.service.ts` issues `SELECT … FROM transaction WHERE status NOT IN (…)` with no `LIMIT`, loading all accumulated rows into the Node.js process heap before `result.slice(offset, offset + limit)` is applied at line 313. [10](#0-9)
