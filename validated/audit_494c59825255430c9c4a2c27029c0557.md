Audit Report

## Title
Unbounded In-Memory Loop in `getTransactionsToSign` Causes Server-Side DoS

## Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every one sequentially calling the async `userKeysToSign` per transaction. Pagination is applied only after the full in-memory scan. As the number of active transactions grows, each API call to `GET /transactions/sign` becomes proportionally more expensive, eventually causing request timeouts and server resource exhaustion.

## Finding Description

**Root cause ŌĆö no `take` limit on the DB query (lines 295ŌĆō299):**

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ŌåÉ no `take` / `skip` here
});
```

Compare with `getTransactions` (lines 175ŌĆō181) and `getHistoryTransactions` (lines 227ŌĆō236), both of which correctly pass `skip: offset, take: limit` to the same `repo.find()` call.

**Unbounded sequential async loop (lines 301ŌĆō309):**

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

Each `userKeysToSign` call delegates to `userKeysRequiredToSign` ŌåÆ `keysRequiredToSign`, which deserializes the transaction bytes (`SDKTransaction.fromBytes`) and calls `transactionSignatureService.computeSignatureKey(transaction)` ŌĆö a non-trivial async operation per transaction.

**Pagination applied only after the full scan (lines 311ŌĆō316):**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ŌåÉ too late
  page,
  size,
};
```

**No throttle guard on the endpoint:** The `TransactionsController` class applies only `JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard` at the class level. The `/sign` route carries no `@Throttle` decorator, unlike other sensitive endpoints (e.g., `remindSigners` which explicitly sets `@Throttle({ 'global-minute': { limit: 1, ttl: seconds(60) } })`).

**Endpoint is still live:** Although the controller comment reads `/* NO LONGER USED BY FRONT-END */`, the route `GET /transactions/sign` is still registered and reachable by any authenticated API client.

## Impact Explanation
Any authenticated, verified user can trigger this endpoint. If an attacker (or even normal usage) accumulates a large number of non-terminal transactions (`WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `NEW`, `REJECTED`), every call to `GET /transactions/sign` will:
- Load all of them into memory from the database
- Perform one async `computeSignatureKey` call per transaction (CPU + possible DB I/O via `entityManager.find(UserKey, ...)`)
- Block the Node.js event loop for the duration of the sequential loop

This causes severe response latency degradation and can exhaust server memory and the database connection pool, effectively making the endpoint ŌĆö and potentially the entire API process ŌĆö unavailable for all users.

## Likelihood Explanation
The attacker only needs a valid authenticated account with at least one registered key (to pass the `user.keys.length === 0` early-exit check). Creating transactions is a normal product workflow. There is no rate limit or cap on the number of active transactions visible to this endpoint. A single malicious user creating thousands of transactions in `WAITING_FOR_SIGNATURES` status is sufficient to trigger the condition. No privileged access is required.

## Recommendation
1. **Remove the endpoint entirely.** The controller already documents it as `/* NO LONGER USED BY FRONT-END */`. Removing it eliminates the attack surface with no functional regression.
2. **If the endpoint must be retained**, restructure the query to push the key-matching logic into the database (e.g., join `transaction` with `user_key` on `publicKeys` column) so that only the page-sized result set is fetched and processed, mirroring the pattern used by `getTransactions` and `getHistoryTransactions`.
3. **Apply a throttle guard** (e.g., `@Throttle({ 'user-minute': { limit: 5, ttl: seconds(60) } })`) as a defence-in-depth measure regardless of which fix is chosen.
