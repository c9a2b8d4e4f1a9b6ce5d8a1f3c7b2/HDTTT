Audit Report

## Title
Any Authenticated User Can Enumerate All Organization Transaction Groups via Missing Authorization on `GET /transaction-groups`

## Summary
The `getTransactionGroups` handler in `TransactionGroupsController` is explicitly marked `/* TESTING ONLY */` but is live in production with no ownership scoping or admin restriction. Any verified authenticated user can call `GET /transaction-groups` and receive every `TransactionGroup` row in the database, regardless of who created it.

## Finding Description

**Controller — no user parameter, no admin guard:**

The handler at line 52–57 carries no `@GetUser()` parameter and applies no `AdminGuard`. The class-level guard stack at line 27 (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only verifies that the caller holds a valid, non-blacklisted JWT with a verified account — it does not restrict to admins or to the caller's own data. [1](#0-0) [2](#0-1) 

**Service — unfiltered `SELECT *`:**

`getTransactionGroups()` issues `this.dataSource.manager.find(TransactionGroup)` with no `where` clause, returning every row in the `transaction_group` table. [3](#0-2) 

**Contrast with the properly-guarded sibling `getTransactionGroup`:**

The single-group endpoint accepts `@GetUser() user: User`, passes the caller's identity into a user-scoped SQL query, and throws `UnauthorizedException` when the caller has no relation to the group's transactions. [4](#0-3) [5](#0-4) 

The unscoped `getTransactionGroups` path bypasses all of that logic entirely.

**Serialized fields exposed via `TransactionGroupDto`:**

`id`, `description`, `atomic`, `sequential`, `createdAt`, `groupValidTime`, and `groupItems` are all `@Expose()`-decorated and returned to the caller. [6](#0-5) 

## Impact Explanation
Any authenticated user gains full enumeration of all transaction groups created by all other users in the organization. Exposed data includes group IDs, descriptions (which may contain sensitive business context), `atomic`/`sequential` execution flags, and timestamps. In a multi-user organization handling Hedera network transactions (account management, HBAR transfers, file operations), leaking the existence and structure of other users' transaction groups constitutes a cross-tenant confidentiality breach. Group IDs can also be used as oracle input for further targeted probing of other endpoints.

## Likelihood Explanation
Exploitation requires only a valid, verified user account — the lowest possible privilege level in the system. The endpoint is reachable over standard HTTPS with no additional preconditions. The `/* TESTING ONLY */` comment confirms the endpoint was never intended for production use, making it likely to be overlooked in security reviews. [7](#0-6) 

## Recommendation
Either remove the endpoint entirely (as the comment implies it should not be in production), or restrict it to admin-only access by adding an `AdminGuard` at the method level, or scope the query to the authenticated caller's own groups by accepting `@GetUser() user: User` and filtering the `find()` call with a `where` clause tied to the user's identity — consistent with how `getTransactionGroup` is implemented.

## Proof of Concept
```
# 1. Obtain a valid JWT for any verified user account (normal registration flow)
# 2. Call the unguarded listing endpoint:
curl -H "Authorization: Bearer <JWT>" https://<host>/transaction-groups

# Response: full list of every TransactionGroup in the database, e.g.:
[
  { "id": 1, "description": "Q1 HBAR payroll batch", "atomic": true, "sequential": false, "createdAt": "...", ... },
  { "id": 2, "description": "Treasury rebalance - confidential", "atomic": false, "sequential": true, ... },
  ...
]

# 3. Use returned IDs for further targeted probing of /transaction-groups/:id
```
