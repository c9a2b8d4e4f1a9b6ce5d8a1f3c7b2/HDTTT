### Title
Any Authenticated User Can Access All Historical Transactions Without Ownership or Role Restriction

### Summary
The `GET /transactions/history` endpoint in `TransactionsController` is accessible to any authenticated, verified user and returns **all** historical transactions from all users in the system. Unlike the `GET /transactions` endpoint which filters results by the requesting user's involvement, `getHistoryTransactions` performs no user-scoped filtering whatsoever. This is a direct analog to the external report's missing `onlyOwner` restriction: a sensitive data-retrieval function that should be restricted to the transaction's participants (or admins) is instead open to every authenticated user.

### Finding Description
In `back-end/apps/api/src/transactions/transactions.controller.ts`, the controller-level guard chain is `JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard` — requiring only a valid, non-blacklisted JWT from a verified user. [1](#0-0) 

The `getHistoryTransactions` handler at line 142 does **not** inject `@GetUser()` and passes no user context to the service: [2](#0-1) 

In `TransactionsService.getHistoryTransactions`, the query has no `userId` predicate — it fetches every transaction whose status is in the terminal set (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`): [3](#0-2) 

Contrast this with `getTransactions`, which explicitly scopes results to the requesting user via signers, observers, creator key, and approver sub-queries: [4](#0-3) 

A second, compounding instance exists in `TransactionGroupsController`. The `GET /transaction-groups` endpoint is explicitly commented `/* TESTING ONLY */`, carries no `AdminGuard`, and returns every group in the database to any authenticated user: [5](#0-4) 

### Impact Explanation
Any authenticated user (including a newly registered, non-admin account) can enumerate the complete history of every transaction ever processed by the organization: transaction bytes, hashes, mirror network, valid-start timestamps, creator key references, and group membership. Transaction bytes may encode sensitive Hedera account or file operations. This constitutes unauthorized cross-tenant data disclosure and breaks the confidentiality model enforced everywhere else in the API.

### Likelihood Explanation
The attacker precondition is only a valid JWT — achievable by registering a normal user account (admins can create accounts via `POST /auth/signup`). The endpoint is a standard REST `GET` with pagination, trivially discoverable via the Swagger docs the API exposes. No brute-force or cryptographic attack is required.

### Recommendation
1. Inject `@GetUser() user: User` into `getHistoryTransactions` and pass it to the service.
2. In `TransactionsService.getHistoryTransactions`, add the same user-scoping `WHERE` clause used in `getTransactions` (filter by `creatorKey.userId`, `signers.userId`, `observers.userId`, or approver sub-query).
3. Either add `@UseGuards(AdminGuard)` to the `GET /transaction-groups` endpoint or remove it entirely if it is only a testing artifact.

### Proof of Concept
```
# 1. Register / obtain a JWT for a normal (non-admin) user
POST /auth/login  { "email": "attacker@example.com", "password": "..." }
# → { "accessToken": "<JWT>" }

# 2. Enumerate all historical transactions across all users
GET /transactions/history?page=1&limit=100
Authorization: Bearer <JWT>

# → Returns paginated list of ALL executed/failed/expired/canceled/archived
#   transactions from every user in the organization, with full transaction bytes.

# 3. (Bonus) Enumerate all transaction groups
GET /transaction-groups
Authorization: Bearer <JWT>
# → Returns all groups, including those belonging to other users.
```
