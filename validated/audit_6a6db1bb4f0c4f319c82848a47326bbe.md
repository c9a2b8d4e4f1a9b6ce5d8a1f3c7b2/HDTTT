### Title
`TransactionsController::shouldSignTransaction` Processes Non-Existent Transaction Without Existence Check

### Summary
The `GET /transactions/sign/:transactionId` endpoint in `TransactionsController` retrieves a transaction by ID and immediately passes the result to `userKeysToSign` without verifying the transaction exists. Since `getTransactionById` returns `null` for unknown IDs, any authenticated user can trigger an unhandled null-dereference (500 Internal Server Error) by supplying a non-existent transaction ID. The sibling endpoint `shouldApproveTransaction` correctly guards against this with an explicit null check, making the omission in `shouldSignTransaction` a clear inconsistency.

### Finding Description

**Root cause — missing existence check before processing:**

In `back-end/apps/api/src/transactions/transactions.controller.ts` lines 189–196:

```typescript
@Get('/sign/:transactionId')
async shouldSignTransaction(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
): Promise<number[]> {
  const transaction = await this.transactionsService.getTransactionById(transactionId);
  return this.transactionsService.userKeysToSign(transaction, user);  // null passed here
}
```

`getTransactionById` explicitly returns `null` for unknown IDs:

```typescript
// transactions.service.ts line 111
async getTransactionById(id: number | TransactionId): Promise<Transaction> {
  if (!id) return null;
  ...
  if (!transactions.length) return null;
  ...
}
```

The null value is then forwarded directly into `userKeysToSign`:

```typescript
// transactions.service.ts line 875-877
async userKeysToSign(transaction: Transaction, user: User, showAll: boolean = false) {
  return userKeysRequiredToSign(transaction, user, this.transactionSignatureService, this.entityManager, showAll);
}
```

`userKeysToSign` performs no null guard. `userKeysRequiredToSign` will attempt to access properties on the null `transaction` object (e.g., `transaction.transactionBytes`, `transaction.signers`), causing an unhandled `TypeError` that surfaces as a 500 Internal Server Error.

**Contrast with the correctly guarded sibling:**

`shouldApproveTransaction` in the service layer explicitly checks for null before proceeding:

```typescript
// transactions.service.ts lines 851-855
async shouldApproveTransaction(transactionId: number, user: User) {
  const transaction = await this.getTransactionById(transactionId);
  if (!transaction) {
    throw new BadRequestException(ErrorCodes.TNF);
  }
```

Every other consumer of `getTransactionById` or similar null-returning lookups also guards: `attachTransactionSigners`, `attachTransactionApprovers`, `verifyAccess`, and `getTransactionForCreator` all throw `BadRequestException(ErrorCodes.TNF)` when the entity is null. `shouldSignTransaction` is the sole exception.

### Impact Explanation

Any authenticated user (no special privileges required) can send `GET /transactions/sign/<arbitrary_id>` with any integer that does not correspond to a real transaction. The server will attempt to dereference `null.transactionBytes` (or similar), throw an unhandled `TypeError`, and return HTTP 500. This constitutes:

1. **Incorrect behavior / integrity failure**: The endpoint should return HTTP 400 (`TNF` error code) for non-existent transactions, consistent with every other endpoint in the service. Instead it crashes.
2. **Authenticated DoS surface**: Any valid session token is sufficient to repeatedly trigger 500 errors on this endpoint, degrading service reliability and polluting error logs.

### Likelihood Explanation

The endpoint is reachable by any authenticated user — no admin role, no ownership of the transaction, and no special precondition beyond a valid JWT. The `ParseIntPipe` accepts any integer, so supplying a non-existent ID is trivial. The bug is exercised on every request for a transaction ID that does not exist in the database.

### Recommendation

Add an existence check in `shouldSignTransaction` immediately after `getTransactionById`, mirroring the pattern used in `shouldApproveTransaction`:

```diff
@Get('/sign/:transactionId')
async shouldSignTransaction(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
): Promise<number[]> {
  const transaction = await this.transactionsService.getTransactionById(transactionId);
+ if (!transaction) {
+   throw new BadRequestException(ErrorCodes.TNF);
+ }
  return this.transactionsService.userKeysToSign(transaction, user);
}
```

Alternatively, move the null guard into `userKeysToSign` itself so all callers are protected by default.

### Proof of Concept

1. Authenticate as any valid user and obtain a JWT.
2. Send: `GET /transactions/sign/999999999` (a transaction ID that does not exist).
3. `getTransactionById(999999999)` returns `null`.
4. `userKeysToSign(null, user)` → `userKeysRequiredToSign(null, ...)` → TypeError on first property access of `null`.
5. NestJS catches the unhandled exception and returns HTTP 500 instead of HTTP 400.

**Expected:** HTTP 400 with `ErrorCodes.TNF` (as returned by `shouldApproveTransaction` under identical conditions).
**Actual:** HTTP 500 Internal Server Error.
