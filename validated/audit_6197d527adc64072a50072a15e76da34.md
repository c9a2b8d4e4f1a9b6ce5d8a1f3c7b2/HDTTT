Audit Report

## Title
`removeTransactionApprover` Emits Status Update with `null` entityId for Child Approvers, Silently Dropping Notification

## Summary
In `removeTransactionApprover`, the notification event is emitted using `approver.transactionId` as the `entityId`. For child approvers (those nested under a threshold tree, where `listId` is set and `transactionId` is `null` by design), this causes `emitTransactionStatusUpdate` to be called with `{ entityId: null }`. The notification service silently skips this event, meaning no status-update notifications are dispatched after a child approver is removed.

## Finding Description
The root cause is in `removeTransactionApprover` at line 541 of `back-end/apps/api/src/transactions/approvers/approvers.service.ts`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
    return result;
}
``` [1](#0-0) 

The `TransactionApprover` entity explicitly documents this invariant: child approvers (those with `listId` set) always have `transactionId = null`. [2](#0-1) 

This is confirmed by the test fixtures in `approvers.service.spec.ts`, where child approvers have `transactionId: null` and `listId` pointing to their parent: [3](#0-2) 

The emitted event `{ entityId: null }` reaches `processTransactionStatusUpdateNotifications` in `back-end/apps/notifications/src/receiver/receiver.service.ts`. The `transactionMap` is built from `events.map(e => e.entityId)`, so `null` is used as a key. `transactionMap.get(null)` returns `undefined`, and the event is silently skipped: [4](#0-3) 

The controller at `back-end/apps/api/src/transactions/approvers/approvers.controller.ts` has the correct `transactionId` available as a route parameter but does not forward it to the service method: [5](#0-4) 

The service already provides `getRootNodeFromNode`, which traverses the tree upward to find the root node (and its `transactionId`) from any child node ‚Äî this is the correct resolution path: [6](#0-5) 

## Impact Explanation
When a child approver is removed from a nested approval tree:
1. `emitTransactionStatusUpdate` fires with `entityId: null`.
2. The notification service silently skips the event ‚Äî no in-app or email notifications are sent to signers, observers, or the creator.
3. The transaction's approval status is not re-evaluated in real time by `processTransactionStatus`.
4. If removing the child approver causes the threshold to be automatically reduced (as done in `updateTransactionApprover` lines 423‚Äì427 when children are removed), the transaction may now satisfy its approval requirements but will not advance to `WAITING_FOR_EXECUTION` until the periodic scheduler (`transaction-scheduler.service.ts`) runs its next cycle. [7](#0-6) 

The periodic scheduler does eventually re-evaluate statuses, so the "permanently stuck" characterization is overstated ‚Äî but the real-time notification failure is confirmed and constitutes a state-update integrity gap.

## Likelihood Explanation
Any authenticated user who is the creator of a transaction can set up a nested approval tree and then delete a child approver via the `DELETE /transactions/:transactionId/approvers/:id` endpoint. No privileged access is required. The precondition ‚Äî a transaction with at least one child approver (i.e., a threshold tree with nested approvers) ‚Äî is a standard supported use case of the application.

## Recommendation
Pass the `transactionId` from the controller into `removeTransactionApprover`, or resolve the root transaction ID inside the service using the existing `getRootNodeFromNode` method before emitting the notification:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
    return result;
}
```

And update the controller call site accordingly:

```typescript
await this.approversService.removeTransactionApprover(id, transactionId);
```

Alternatively, resolve the root node inside the service (as `updateTransactionApprover` does) to avoid relying on the caller to supply the correct ID.

## Proof of Concept
1. Authenticate as a user and create a transaction.
2. Add a threshold approver tree: a root threshold node with two child user-approvers (e.g., threshold=2, children=[userA, userB]).
3. The root node has `transactionId = <txId>`, `listId = null`. Each child has `transactionId = null`, `listId = <rootNodeId>`.
4. Call `DELETE /transactions/<txId>/approvers/<childNodeId>` targeting one of the child approvers.
5. The service fetches the child approver, which has `transactionId = null`.
6. `emitTransactionStatusUpdate` is called with `[{ entityId: null }]`.
7. In `processTransactionStatusUpdateNotifications`, `transactionMap.get(null)` returns `undefined`; the `console.warn` fires and the loop `continue`s.
8. No notifications are dispatched; no real-time status re-evaluation occurs for the transaction.

  async getRootNodeFromNode(
    id: number,
    entityManager?: EntityManager,
  ): Promise<TransactionApprover | null> {
    if (!id || typeof id !== 'number') return null;

    return (
      await (entityManager || this.repo).query(
        `
        with recursive approverList as
          (
            select * from transaction_approver 
            where "id" = $1
              union all 
                select approver.* from transaction_approver as approver
                join approverList on approverList."listId" = approver."id"
          )
        select * from approverList
        where "listId" is null
        `,
        [id],
      )
    )[0];
  }
