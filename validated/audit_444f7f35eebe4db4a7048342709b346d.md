Audit Report

## Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Submission of Invalid Signatures to Advance Transaction State

## Summary
The `uploadSignatureMaps` endpoint in `signers.service.ts` verifies that submitted public keys belong to the authenticated user but never cryptographically verifies that the signature bytes are valid for the transaction body. The parallel `importSignatures` endpoint in `transactions.service.ts` explicitly calls `validateSignature` before accepting signatures. An authenticated required-signer can submit their own public key paired with arbitrary/invalid signature bytes, get recorded as having signed, advance the transaction to `WAITING_FOR_EXECUTION`, and cause it to fail permanently at the Hedera network.

## Finding Description

**Root cause — missing `validateSignature` call in `processTransactionSignatures`:**

In `signers.service.ts`, the private method `processTransactionSignatures` (lines 217–267) iterates over the submitted `SignatureMap`, checks that each public key belongs to the authenticated user via `userKeyMap` lookup, and then unconditionally calls `sdkTransaction.addSignature(publicKey, map)`:

```typescript
let userKey = userKeyMap.get(raw);
if (!userKey) {
  userKey = userKeyMap.get(publicKey.toStringDer());
}
if (!userKey) throw new Error(ErrorCodes.PNY);

// Only add the signature once per unique key
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
``` [1](#0-0) 

There is no call to `validateSignature` or any equivalent cryptographic check on the signature bytes before they are added to the transaction.

**Contrast with `importSignatures`:**

The `importSignatures` method in `transactions.service.ts` (lines 545–553) explicitly validates the signature bytes before calling `addSignature`:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);

for (const publicKey of publicKeys) {
  sdkTransaction.addSignature(publicKey, map);
}
``` [2](#0-1) 

The existence of this explicit `validateSignature` call confirms that `addSignature` in the Hedera SDK does **not** perform cryptographic verification internally — otherwise the call would be redundant. The `uploadSignatureMaps` path is therefore missing this guard entirely.

**How `processTransactionStatus` is fooled:**

`processTransactionStatus` reads `sdkTransaction._signerPublicKeys` from the stored transaction bytes and passes them to `hasValidSignatureKey`. Since `addSignature` adds the public key to `_signerPublicKeys` regardless of whether the signature bytes are valid, the threshold check passes even with garbage bytes:

```typescript
const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey
);
``` [3](#0-2) 

`hasValidSignatureKey` only checks whether the public key string is present in `_signerPublicKeys` — it performs no cryptographic verification of the associated signature bytes:

```typescript
} else if (key instanceof PublicKey) {
  return publicKeys.includes(key.toStringRaw());
}
``` [4](#0-3) 

**`isSameBytes` does not protect against this:**

After `addSignature` embeds the garbage bytes into the transaction's signature map, the serialized bytes differ from the original. `isSameBytes` is `false`, so the corrupted `transactionBytes` are persisted to the database:

```typescript
const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
  transaction.transactionBytes
);
``` [5](#0-4) 

```typescript
if (!isSameBytes) {
  transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
  transactionsToUpdate.push({ id, transactionBytes: transaction.transactionBytes });
}
``` [6](#0-5) 

**No access control on transaction ID in `loadTransactionData`:**

`loadTransactionData` fetches any transaction by ID without verifying the requesting user has any relationship to it:

```typescript
const transactions = await this.dataSource.manager.find(Transaction, {
  where: { id: In(transactionIds) },
});
``` [7](#0-6) 

The only constraint is that the submitted public key must belong to the authenticated user (via `userKeyMap`), meaning the attacker's key must be a required signer for the target transaction — a realistic insider scenario.

**The same gap exists in `execute.service.ts`:**

Even at execution time, `getValidatedSDKTransaction` only checks `_signerPublicKeys` presence, not signature byte validity:

```typescript
if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
``` [8](#0-7) 

The corrupted transaction passes this check and is submitted to the Hedera network, which then rejects it.

## Impact Explanation

- **Direct loss of funds**: The fee payer's HBAR is consumed when the Hedera network processes and rejects the transaction. The fee is non-refundable.
- **Permanent transaction destruction**: Once submitted and failed, the transaction is marked `FAILED` and cannot be re-executed. High-value operations (`NodeUpdateTransaction`, `AccountUpdateTransaction`, treasury transfers) are permanently lost.
- **Integrity failure in multi-sig workflow**: The attacker is recorded as a `TransactionSigner`, suppressing further signing reminders and deceiving other participants into believing the transaction is legitimately signed.

## Likelihood Explanation

- **Attacker preconditions**: A registered, authenticated user account whose public key is listed as a required signer for the target transaction (i.e., they were added as a signer by the transaction creator). This is a realistic insider or compromised-account scenario.
- **No special privileges required**: Standard JWT authentication is sufficient. The `uploadSignatureMap` endpoint is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. [9](#0-8) 

- **Ease of exploitation**: Constructing a `SignatureMap` with a valid public key and invalid signature bytes is trivial using the Hedera SDK. The `IsSignatureMap` decorator only validates structure (account ID format, transaction ID format, non-empty bytes) — it does not verify cryptographic correctness. [10](#0-9) 

## Recommendation

Add a `validateSignature` call in `processTransactionSignatures` before calling `sdkTransaction.addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const { data: validatedKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

This ensures that only cryptographically valid signatures are accepted through the `uploadSignatureMaps` path, consistent with the existing guard in `importSignatures`.

## Proof of Concept

1. Attacker is an authenticated user whose public key `PK_attacker` is a required signer for transaction ID `T`.
2. Attacker constructs a `SignatureMap` containing `PK_attacker` mapped to 64 bytes of `0xFF` (garbage).
3. Attacker calls `POST /transactions/signers` with `[{ id: T, signatureMap: <garbage_map> }]`.
4. `processTransactionSignatures` passes the `userKeyMap` check (key belongs to the user).
5. `sdkTransaction.addSignature(PK_attacker, garbage_map)` embeds the garbage bytes; `_signerPublicKeys` now includes `PK_attacker`; serialized bytes differ → `isSameBytes = false`.
6. Corrupted `transactionBytes` are persisted; attacker is inserted as a `TransactionSigner`.
7. `processTransactionStatus` reads `_signerPublicKeys` from the corrupted bytes, finds `PK_attacker` present, `hasValidSignatureKey` returns `true`, transaction advances to `WAITING_FOR_EXECUTION`.
8. The chain service submits the transaction to the Hedera network. The network rejects it due to the invalid signature. The transaction is permanently marked `FAILED`. Fee payer's HBAR is burned.
