### Title
Whole-Group Execution Blocked by Single Transaction Validation Failure in `executeTransactionGroup`

### Summary
`executeTransactionGroup` in `ExecuteService` performs a pre-execution validation loop over all group items before submitting any of them to Hedera. If any single transaction fails validation (e.g., its DB status changed to `CANCELED` or `EXPIRED` after the group was loaded into memory), the function throws immediately, permanently blocking all other valid transactions in the group from ever executing. This is the direct analog of the `_release_all` push-payment DoS: a batch push-execution mechanism that fails atomically when any one item fails, even when the remaining items are independently valid.

### Finding Description

**Root cause — stale in-memory filter + DB re-fetch mismatch:**

`executeTransactionGroup` first filters group items using the in-memory `transaction.status` field: [1](#0-0) 

This filter uses the status that was loaded when `prepareTransactions` called `transactionGroupRepo.findOne` (inside `collateGroupAndExecute`). That object is cached in the scheduler timeout closure and is never refreshed. [2](#0-1) 

The timeout is registered once and never re-loaded: [3](#0-2) 

After the filter, `getValidatedSDKTransaction` is called for each item, which **re-fetches the status from the DB**: [4](#0-3) 

If the DB status has changed (e.g., to `CANCELED`, `EXPIRED`, `REJECTED`) between the in-memory load and this DB re-fetch, `validateTransactionStatus` throws. That throw is immediately re-thrown, aborting the entire group: [5](#0-4) 

The caller (`addGroupExecutionTimeout`) catches the error, logs it, and deletes the timeout — with no retry and no status update for the remaining valid transactions: [6](#0-5) 

The remaining valid transactions are left in `WAITING_FOR_EXECUTION` indefinitely until the expiry cron marks them `EXPIRED`.

**Exploit path (malicious organization member):**

1. Attacker is a member of an organization and participates in a `TransactionGroup` with N transactions (e.g., a batch payroll group).
2. The chain service cron runs, loads the group into memory, and registers the execution timeout.
3. Before the timeout fires, the attacker cancels one transaction in the group via the cancel API endpoint (a normal user action).
4. The execution timeout fires; `executeTransactionGroup` is called with the stale in-memory object.
5. The in-memory filter passes (stale status = `WAITING_FOR_EXECUTION`).
6. `validateTransactionStatus` re-fetches from DB, finds `CANCELED`, throws.
7. All N-1 remaining valid transactions are permanently blocked from executing.
8. They expire without ever being submitted to Hedera.

The same outcome occurs naturally if the expiry cron races with the execution timeout (e.g., chain service is briefly delayed), making this also a reliability failure independent of malicious intent.

### Impact Explanation

All valid transactions in a group are permanently prevented from executing within their valid window. They are left in `WAITING_FOR_EXECUTION` with no recovery path, eventually expiring. In an organization context, this means batch operations (payroll, multi-account updates, scheduled transfers) silently fail in their entirety because of one member's action or a timing race. The state corruption is unrecoverable without manual intervention.

### Likelihood Explanation

The TOCTOU window exists for every group execution: the group is loaded once into memory and the stale object is used until the timeout fires. Any state change to any group item during that window (cancel, expiry race) triggers the DoS. A malicious organization member can deliberately cancel one transaction just before execution to block the entire group. No privileged access is required — canceling one's own transaction is a normal user action.

### Recommendation

Replace the fail-fast throw in the validation loop with per-item error handling. If a transaction fails validation, mark only that transaction as `FAILED` (or skip it), and continue validating and executing the remaining items. This mirrors the pull-payment pattern recommended in the external report: each item is handled independently, so one failure does not block others.

Specifically, in `executeTransactionGroup`, change the catch block from:

```typescript
} catch (error) {
  throw new Error(`Transaction Group cannot be submitted...`);
}
```

to collecting per-item failures, updating only the failed transaction's status in the DB, and proceeding with the remaining valid transactions.

Additionally, re-fetch the group's transaction statuses from the DB immediately before the validation loop (rather than relying on the stale in-memory object) to eliminate the TOCTOU window entirely.

### Proof of Concept

**Setup:** Organization with a `TransactionGroup` containing 3 transactions (IDs: 101, 102, 103), all in `WAITING_FOR_EXECUTION`.

**Steps:**
1. Chain service cron runs, loads the group into memory with all three transactions showing `WAITING_FOR_EXECUTION`.
2. Execution timeout is registered (fires at `validStart + 5s`).
3. Attacker calls the cancel API for transaction 101. DB status → `CANCELED`. In-memory object still shows `WAITING_FOR_EXECUTION`.
4. Execution timeout fires. `executeTransactionGroup` is called.
5. In-memory filter: all three pass (stale status).
6. Validation loop hits transaction 101: `validateTransactionStatus` fetches DB → `CANCELED` → throws `"Transaction has been canceled."`.
7. Outer catch re-throws: `"Transaction Group cannot be submitted. Error validating transaction 101: Transaction has been canceled."`.
8. `addGroupExecutionTimeout` catches, logs, deletes timeout. No DB update for 102 or 103.
9. Transactions 102 and 103 remain in `WAITING_FOR_EXECUTION` until the expiry cron marks them `EXPIRED`. They are never submitted to Hedera. [7](#0-6) [8](#0-7) [6](#0-5)

  private async getValidatedSDKTransaction(
    transaction: Transaction,
  ): Promise<SDKTransaction> {
    /* Throws an error if the transaction is not found or in incorrect state */
    if (!transaction) throw new Error('Transaction not found');

    await this.validateTransactionStatus(transaction);

    /* Gets the SDK transaction from the transaction body */
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Gets the signature key */
    const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);

    /* Checks if the transaction has valid signatureKey */
    if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
      throw new Error('Transaction has invalid signature.');

    return sdkTransaction;
  }
