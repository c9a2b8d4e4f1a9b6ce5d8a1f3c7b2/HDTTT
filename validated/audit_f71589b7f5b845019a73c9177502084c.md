### Title
`getTransactionsToSign()` Fetches and Iterates All Non-Terminal Transactions Without Bound, Enabling DoS via Transaction Flooding

### Summary
`getTransactionsToSign()` in `transactions.service.ts` fetches every non-terminal transaction from the database without a `take` limit, then iterates over all of them in-memory calling `userKeysToSign()` per entry before applying pagination. Any authenticated, verified user can create an unbounded number of transactions. By flooding the system with transactions, an attacker causes the `GET /transactions/sign` endpoint to exhaust server CPU, memory, and database resources, degrading or permanently blocking the signing workflow for all organization users.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` issues a database query with no `take` (row limit):

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take: limit` here
});
``` [1](#0-0) 

It then iterates over every returned row, invoking `userKeysToSign()` — which internally calls `TransactionSignatureService.computeSignatureKey()`, a non-trivial cryptographic key-tree traversal — for each transaction:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Pagination is applied only **after** the full in-memory scan:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

This contrasts directly with `getTransactions()` and `getTransactionsToApprove()`, both of which correctly pass `take: limit` to the database query and never iterate all rows in application code: [4](#0-3) [5](#0-4) 

Transaction creation (`createTransaction` / `createTransactions`) enforces no per-user cap or rate limit. The only uniqueness check is on `transactionId` (the Hedera-level ID, which embeds a timestamp and is trivially varied): [6](#0-5) 

The `CreateTransactionObserversDto` similarly has no `@ArrayMaxSize` constraint, only `@ArrayMinSize(1)`, meaning a single POST can add an arbitrarily large `userIds` array: [7](#0-6) 

### Impact Explanation

Every call to `GET /transactions/sign` triggers a full table scan of all non-terminal transactions followed by one `computeSignatureKey()` call per row. With N attacker-created transactions:

- **Database**: one unbounded `SELECT` loading all rows and their `groupItem` relations into memory.
- **CPU/Memory**: N sequential async calls to `userKeysToSign`, each performing key-tree traversal.
- **Result**: request timeouts, Node.js event-loop starvation, and OOM conditions under sufficient load.

This blocks the core signing workflow for every organization user, constituting a **High-impact** service availability failure. Funds/transactions cannot be signed or progressed.

### Likelihood Explanation

**Medium.** The attacker must be a registered and verified organization user (`VerifiedUserGuard` is enforced on the controller). However, self-registration is a standard product flow. Once verified, the attacker can script thousands of `POST /transactions` calls with unique Hedera transaction IDs (timestamp-based, trivially varied). No rate limiting, per-user transaction cap, or server-side guard prevents this.

### Recommendation

1. **Apply DB-level pagination before the loop.** Pre-filter candidates at the query level using a subquery or join on `transaction_signer`/`publicKeys` to narrow the set before loading into memory, then apply `take: limit` and `skip: offset`.
2. **Add a per-user transaction creation rate limit** (e.g., via NestJS `ThrottlerGuard`) on `POST /transactions`.
3. **Add `@ArrayMaxSize`** to `CreateTransactionObserversDto.userIds` to prevent a single request from adding an unbounded number of observers.

### Proof of Concept

1. Register and verify a user account via the standard organization flow.
2. Script a loop that sends `POST /transactions` with a unique `validStart` timestamp per request (making each `transactionId` unique), e.g.:
   ```
   for i in {1..10000}; do
     curl -X POST http://<host>/transactions \
       -H "Authorization: Bearer <token>" \
       -d '{"transactionBytes":"...","creatorKeyId":1,"mirrorNetwork":"testnet","validStart":"<now+i_seconds>",...}'
   done
   ```
3. After flooding, call `GET /transactions/sign?page=1&size=10` as any user.
4. Observe: the server fetches all 10,000+ rows, calls `userKeysToSign()` 10,000+ times sequentially, and the request times out or the Node.js process exhausts memory — blocking the signing endpoint for all users. [8](#0-7)

  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
