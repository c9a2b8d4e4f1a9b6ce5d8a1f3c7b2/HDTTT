Audit Report

## Title
Sequential Transaction Group Execution Ignores `seq` Ordering Field, Using `validStart` Instead

## Summary
The `TransactionGroupItem.seq` field is the documented mechanism for defining execution order within a sequential transaction group. However, `prepareTransactions` loads group items ordered by `transaction.validStart ASC` and completely ignores `seq`. The sequential execution path in `executeTransactionGroup` then consumes items in that `validStart`-derived order. The UI and API display items ordered by `seq`. Any group where `seq` order diverges from `validStart` order will execute in a different sequence than what signers reviewed and approved.

## Finding Description

**Root cause — ordering metadata is discarded on load:**

In `prepareTransactions`, the TypeORM query fetches the group ordered by `transaction.validStart ASC`: [1](#0-0) 

The `seq` column on `TransactionGroupItem` — documented as "Sequence of transaction execution within the transaction group" — is never referenced in this query or anywhere downstream in the execution path. [2](#0-1) [3](#0-2) 

**Execution path — order is consumed directly:**

`executeTransactionGroup` iterates `transactionGroup.groupItems` in the order they were loaded. For sequential groups it executes them one-by-one in that order: [4](#0-3) 

`collateGroupAndExecute` anchors its collation timing to `groupItems[0].transaction.validStart`, assuming the first item by `validStart` is the first to execute: [5](#0-4) 

`addGroupExecutionTimeout` does the same: [6](#0-5) 

**UI displays `seq` order, execution uses `validStart` order:**

The API query for group items displayed to signers orders by `seq`: [7](#0-6) 

This creates a confirmed, persistent divergence between the order signers review and the order the chain service executes.

## Impact Explanation

For sequential groups, execution order is semantically critical. Dependent operations — `FileCreate` → `FileAppend`, token association → token transfer, account creation → HBAR transfer — will fail or produce incorrect on-chain state if executed out of order. Because Hedera transactions expire at `validStart + 180 s`, failed transactions cannot be retried once the window closes. This constitutes an unrecoverable corruption of the intended transaction sequence and potential financial loss for the organization.

## Likelihood Explanation

Any authenticated organization user can create a transaction group. No privileged access is required. The divergence between `seq` and `validStart` order can arise naturally (a creator sets `validStart` values for scheduling reasons that do not match the intended `seq` order) or deliberately (a malicious creator assigns `seq` values that contradict `validStart` order to deceive co-signers). The `seq` field is surfaced in the UI and API response ordered by `seq`, so signers have no reliable way to detect that the actual execution order will differ.

## Recommendation

Replace the `validStart ASC` ordering in `prepareTransactions` with `seq ASC` ordering:

```ts
order: {
  groupItems: {
    seq: 'ASC',   // was: transaction: { validStart: 'ASC' }
  },
},
``` [1](#0-0) 

Additionally, update `collateGroupAndExecute` and `addGroupExecutionTimeout` to derive their timing anchor from the item with the minimum `validStart` (which, after correct `seq`-based ordering, should be `groupItems[0]`) rather than blindly using index 0. Enforce at the API layer that `seq` values must be assigned in ascending `validStart` order, or remove `seq` as a user-controlled field and derive it from `validStart` at creation time.

## Proof of Concept

1. Authenticated user creates a sequential group via `POST /transaction-groups` with `sequential: true` and two items:
   - Item A: `seq=0`, `validStart = T + 10s` (e.g., a `FileCreate`)
   - Item B: `seq=1`, `validStart = T + 5s` (e.g., a `FileAppend` depending on A)
2. Co-signers view the group via the UI. `getTransactionGroupItemsQuery` orders by `seq`, so they see Item A first, Item B second — the intended dependency order.
3. Co-signers sign both transactions believing execution will follow `seq` order.
4. The chain service runs `prepareTransactions`. The `findOne` query orders by `validStart ASC`, loading Item B (T+5s) first, Item A (T+10s) second.
5. `executeTransactionGroup` executes Item B (`FileAppend`) before Item A (`FileCreate`). The `FileAppend` fails because the file does not yet exist.
6. Both transactions expire within their 180-second windows. The operation is unrecoverable. [8](#0-7) [4](#0-3)
