### Title
Weak Password Policy Allows Trivially Guessable Passwords in Both Organization Backend and Local Mode

### Summary
The organization backend's `ChangePasswordDto` and `NewPasswordDto` enforce only an 8-character minimum with zero complexity requirements, meaning passwords like `aaaaaaaa` or `00000000` pass server-side validation. Separately, the local-mode `changePassword` IPC handler in the Electron main process performs no password strength validation whatsoever — the only check is a client-side UI gate that can be bypassed by calling the IPC channel directly from the renderer's JavaScript context. Because the password also serves as the decryption key for stored Hedera private keys, a weak password directly threatens key material.

### Finding Description

**Organization backend (server-side):**

`ChangePasswordDto` and `NewPasswordDto` use `class-validator`'s `IsStrongPassword` decorator, but all complexity requirements are explicitly zeroed out:

```ts
@IsStrongPassword(
  {
    minLength: 8,
    minLowercase: 0,
    minNumbers: 0,
    minSymbols: 0,
    minUppercase: 0,
  },
  { message: 'Password is too weak, must contain at least 8 characters.' },
)
``` [1](#0-0) [2](#0-1) 

This means any 8-character string — `aaaaaaaa`, `00000000`, `abcdefgh` — passes the server-side DTO validation and is accepted by `PATCH /auth/change-password` and `PATCH /auth/set-password`. [3](#0-2) [4](#0-3) 

**Local mode (Electron main process):**

The `changePassword` function in the main process has no password strength check at all. It only verifies the old password is correct, then immediately persists the new password:

```ts
export const changePassword = async (userId: string, oldPassword: string, newPassword: string) => {
  const isOldCorrect = await comparePasswords(userId, oldPassword);
  if (isOldCorrect) {
    await updatePassword(userId, newPassword);  // no strength check
  } else {
    throw new Error('Incorrect current password');
  }
  await changeDecryptionPassword(userId, oldPassword, newPassword);
};
``` [5](#0-4) 

The UI in `ProfileTab.vue` disables the submit button when `isPasswordStrong` fails, but this is purely a renderer-side guard. The IPC channel `localUser:changePassword` is registered unconditionally in the main process: [6](#0-5) 

Any code running in the renderer context (e.g., via the DevTools console, a compromised renderer, or a malicious deep-link handler) can invoke `window.electronAPI.local.localUser.changePassword(userId, currentPassword, 'aaa')` directly, bypassing the UI check entirely. [7](#0-6) 

Furthermore, even the front-end `isPasswordStrong` function only checks for 10 characters minimum with no complexity requirements at all (no uppercase, lowercase, numbers, or symbols required): [8](#0-7) 

### Impact Explanation

The password in this application serves a dual purpose:
1. It authenticates the user at login.
2. It is used as the decryption key for stored Hedera private key pairs (passed directly to `changeDecryptionPassword`). [5](#0-4) 

A weak password (e.g., `aaaaaaaa`) means:
- **Organization mode**: An attacker who can reach the organization API can brute-force the login endpoint (`POST /auth/login`) with a small dictionary. Upon success, they obtain a JWT and can sign or approve Hedera transactions on behalf of the victim.
- **Local mode**: If the encrypted SQLite database file is exfiltrated (e.g., via backup, sync, or physical access), the weak password makes offline brute-force of the encrypted private keys trivial, leading to full private key compromise and unauthorized Hedera transaction signing.

### Likelihood Explanation

- **Organization mode**: Any authenticated user can set a weak password through the normal UI or by directly calling `PATCH /auth/change-password`. No privilege escalation is required. The endpoint is reachable by any valid organization member.
- **Local mode**: The IPC bypass requires code execution in the renderer process, which is a higher bar, but the weak front-end policy (10 chars, no complexity) is reachable by any local user through the normal UI.

### Recommendation

1. **Backend DTOs**: Set meaningful complexity requirements in `IsStrongPassword`. At minimum: `minLength: 12`, `minLowercase: 1`, `minUppercase: 1`, `minNumbers: 1`, `minSymbols: 1`. Apply consistently to both `ChangePasswordDto` and `NewPasswordDto`. [1](#0-0) 

2. **Local mode main process**: Add password strength validation inside the `changePassword` (and `register`) functions in `front-end/src/main/services/localUser/auth.ts` before persisting the password, so the check cannot be bypassed by calling the IPC channel directly. [5](#0-4) 

3. **Front-end validator**: Extend `isPasswordStrong` to require at least one uppercase letter, one number, and one symbol in addition to the length check. [8](#0-7) 

### Proof of Concept

**Organization backend (no privilege required beyond a valid account):**

```http
PATCH /auth/change-password
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "oldPassword": "<current_password>",
  "newPassword": "aaaaaaaa"
}
```

Expected result: HTTP 200, password changed to `aaaaaaaa`. The server accepts it because `IsStrongPassword` with all complexity minimums at 0 passes any 8-character string.

**Local mode IPC bypass (from Electron renderer DevTools console):**

```js
await window.electronAPI.local.localUser.changePassword(
  '<userId>',
  '<currentPassword>',
  'aaa'   // 3 characters, no complexity — accepted by main process
);
```

Expected result: Password is changed to `aaa` with no error, because `front-end/src/main/services/localUser/auth.ts:changePassword` performs no strength check. [5](#0-4)

export function isPasswordStrong(password: string): {
  length: boolean;
  result: boolean;
} {
  const validationRegex = [
    /.{10,}/, // min 10 letters
  ];

  return {
    length: validationRegex[0].test(password),
    result: validationRegex.every(regex => regex.test(password)),
  };
}
