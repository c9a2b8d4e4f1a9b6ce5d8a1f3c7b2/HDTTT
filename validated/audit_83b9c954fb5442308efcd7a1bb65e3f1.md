### Title
Missing `setContentProtection` on Electron BrowserWindow Allows Screen Capture of Decrypted Private Keys and Recovery Phrases

### Summary
The Hedera Transaction Tool Electron desktop application creates its main window without calling `setContentProtection(true)`, which is the Electron equivalent of Android's `FLAG_SECURE`. Any process running on the same machine (e.g., malicious screen-recording software) can capture the application window at the moment sensitive cryptographic material — decrypted private keys and recovery phrase words — is rendered in plaintext on screen. No physical access to the device is required.

### Finding Description
**Root cause:** `front-end/src/main/windows/mainWindow.ts`, `createWindow()`, lines 22–37. The `BrowserWindow` is instantiated with `webPreferences` but `setContentProtection(true)` is never called anywhere in the codebase (confirmed: zero matches for `setContentProtection` across the entire repository).

```
const mainWindow = new BrowserWindow({
  ...
  webPreferences: {
    preload,
    nodeIntegration: false,
    contextIsolation: true,
    sandbox: false,
    session,
  },
  show: false,
  // ← setContentProtection(true) never called
});
```

**Sensitive surfaces exposed:**

1. **Decrypted private keys** — `KeysTab.vue` lines 350–374: when the user clicks the eye icon, the raw private key string is rendered directly in a `<span>` element in the DOM and is visible on screen.

2. **Recovery phrase words** — `Import.vue` lines 113–124: the component renders all mnemonic words with `visible-initially` prop, meaning all 24 words are visible in plaintext on screen during import/verification flows.

3. **Private key during account setup** — `KeyPairs.vue` lines 272–283: the private key is toggled visible via `privateKeyHidden` state and rendered in a `<span>`.

**Exploit path:** A malicious process running as the same OS user (e.g., a trojanized application, a screen recorder, or spyware) uses OS-level screen capture APIs — `BitBlt`/`PrintWindow` on Windows, `CGWindowListCreateImage` on macOS — to capture the Electron window at the moment the user reveals their private key or enters their recovery phrase. Because `setContentProtection(true)` is absent, the OS does not exclude the window from capture.

### Impact Explanation
A successful capture yields the raw ED25519 or ECDSA private key, or the full BIP-39 mnemonic. Either is sufficient for complete, irreversible wallet compromise: the attacker can reconstruct the private key, sign arbitrary Hedera transactions, and drain all associated accounts. Impact is critical — direct theft of cryptographic assets.

### Likelihood Explanation
Likelihood is medium. The attacker must have already installed malicious software on the victim's machine (no physical access required — this is achievable via trojanized downloads, supply-chain attacks, or social engineering). The timing window is non-trivial: the attacker's process must capture the screen while the key is visible. However, a persistent screen recorder running in the background trivially solves the timing problem. This is a realistic, documented threat class for desktop crypto wallets.

### Recommendation
In `front-end/src/main/windows/mainWindow.ts`, call `setContentProtection(true)` immediately after the window is created:

```ts
const mainWindow = new BrowserWindow({ ... });
mainWindow.setContentProtection(true);
```

On Windows this invokes `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)`; on macOS it sets `NSWindowSharingNone`. Note: on Linux this is a no-op and additional OS-level mitigations (e.g., Wayland compositor flags) should be documented. Also update developer documentation to list screen capture as a threat surface for any future sensitive UI additions.

### Proof of Concept

1. Build and launch the Hedera Transaction Tool desktop application.
2. On a second terminal (same OS user), run a minimal screen capture loop using the platform API (e.g., on macOS: `screencapture -x /tmp/cap.png` in a loop; on Windows: a `BitBlt`-based recorder).
3. In the application, navigate to **Settings → Keys**, click the eye icon next to any key pair, and enter the application password to decrypt.
4. Observe that `/tmp/cap.png` (or equivalent) contains the plaintext private key string rendered in the `<span data-testid="span-private-key-N">` element.
5. Repeat during a recovery phrase import flow (`Import.vue`) — all 24 mnemonic words are captured.
6. Expected result with fix: after `mainWindow.setContentProtection(true)`, the captured image shows a black/blank rectangle where the application window was.
