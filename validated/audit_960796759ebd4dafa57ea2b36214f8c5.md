### Title
Any Verified User Can Create and Read Comments on Transactions They Have No Access To

### Summary
`CommentsController.createComment()` in `back-end/apps/api/src/transactions/comments/comments.controller.ts` lacks transaction-level access control. Any authenticated and verified user can post comments on any transaction by ID, regardless of whether they are a creator, signer, approver, or observer of that transaction. The developer explicitly acknowledged this gap with a `TODO` comment in the source. The read endpoints `getComments()` and `getCommentById()` similarly perform no user-to-transaction access verification.

### Finding Description

The `CommentsController` applies only the standard authentication stack at the class level (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). No additional guard or service-level check restricts which transactions a user may interact with. [1](#0-0) 

The `createComment` handler accepts any `transactionId` from the URL and immediately delegates to the service with no ownership or membership check: [2](#0-1) 

The developer's own `TODO` comment on line 22 explicitly states: *"need some sort of guard or check to ensure user can comment here"*, confirming the control is intentionally absent and not yet implemented.

The two read endpoints compound the issue — neither accepts a `@GetUser()` parameter nor performs any access check: [3](#0-2) 

By contrast, every other transaction-scoped endpoint in the codebase (e.g., `getTransaction`, `getTransactionApproversByTransactionId`, `getTransactionWithVerifiedAccess`) enforces that the requesting user is a creator, signer, approver, or observer before returning data. [4](#0-3) 

### Impact Explanation

**Unauthorized state mutation**: A verified attacker can `POST /transactions/{anyId}/comments` with an arbitrary `transactionId` and inject comments into transactions they have no legitimate relationship with. This pollutes the comment history of sensitive multi-signature transactions and can be used to harass or mislead legitimate participants.

**Cross-tenant data exposure**: `GET /transactions/{anyId}/comments` and `GET /transactions/{anyId}/comments/{commentId}` return comment content for any transaction. Comments may contain sensitive coordination details (key references, signing instructions, business context) that should be visible only to transaction participants.

**Transaction ID enumeration**: Successful comment creation or retrieval on a given `transactionId` confirms that transaction exists, enabling systematic enumeration of all transaction IDs in the system.

### Likelihood Explanation

Exploitation requires only a valid, verified user account — the baseline attacker profile per `RESEARCHER.md`. No admin credentials, leaked secrets, or privileged access are needed. The attacker simply registers, gets verified, and issues HTTP requests with sequential or guessed `transactionId` values. The attack is trivially scriptable.

### Recommendation

Add a transaction membership check before allowing comment creation or retrieval. The existing `TransactionsService.getTransactionWithVerifiedAccess()` already implements this logic and should be called (or an equivalent guard applied) at the start of each `CommentsController` handler to verify the requesting user is a creator, signer, approver, or observer of the target transaction.

### Proof of Concept

1. Register two users: `attacker` (verified) and `victim` (verified).
2. As `victim`, create a transaction (receives `transactionId = N`). `attacker` is not added as signer, approver, or observer.
3. As `attacker`, send:
   ```
   POST /transactions/N/comments
   Authorization: Bearer <attacker_jwt>
   { "content": "attacker comment" }
   ```
4. Observe HTTP 201 — comment created on a transaction the attacker has no access to.
5. As `attacker`, send:
   ```
   GET /transactions/N/comments
   Authorization: Bearer <attacker_jwt>
   ```
6. Observe HTTP 200 — full comment list returned, including any comments posted by `victim` and other participants.
