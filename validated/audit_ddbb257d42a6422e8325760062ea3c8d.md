### Title
Pending Transactions Become Permanently Unmanageable When Creator User Is Removed

### Summary
When an admin removes a user via `removeUser()`, all pending transactions where that user is the `creatorKey` owner become permanently stuck in non-terminal states (`WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`). No other user — including admins — can cancel or manage these transactions, because every management path enforces a creator-only check that the deleted user can no longer satisfy.

---

### Finding Description

**Vulnerability class:** State transition / authorization — orphaned state after entity removal.

`removeUser()` in `users.service.ts` soft-deletes the user and all their keys without inspecting whether any pending transactions reference those keys as `creatorKey`: [1](#0-0) 

After deletion the user cannot authenticate, so they can never call any API endpoint. Every destructive or state-changing operation on a transaction — `cancelTransaction`, `archiveTransaction`, `removeTransaction`, `executeTransaction` — routes through `getTransactionForCreator`: [2](#0-1) 

That helper throws `UnauthorizedException` for anyone whose `id` does not match `transaction.creatorKey?.userId`. There is no admin bypass anywhere in `TransactionsService`.

The `Transaction` entity stores `creatorKeyId` as a non-nullable foreign key pointing to `UserKey`: [3](#0-2) 

`UserKey` uses a soft-delete column: [4](#0-3) 

After `removeUser` runs `softDelete(UserKey, { userId: id })` followed by `softRemove(user)`, the `creatorKeyId` column on every affected `Transaction` row still holds the old value, but the owning user account is gone. The transactions remain in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` with no reachable code path to move them to a terminal state.

The same root cause applies when a user deletes their own key via `removeUserKey`: [5](#0-4) 

No check is performed to see whether the key being removed is the `creatorKey` of any live transaction before the soft-remove is issued.

---

### Impact Explanation

Pending transactions whose creator user (or creator key) has been removed are permanently orphaned:

- They cannot be cancelled — `cancelTransaction` → `getTransactionForCreator` enforces creator-only access.
- They cannot be archived — `archiveTransaction` has the same guard.
- They cannot be removed — `removeTransaction` has the same guard.
- They remain visible in the system in non-terminal states, polluting dashboards and notification queues indefinitely.
- The transaction scheduler (`chain` service) may repeatedly attempt to process or remind on these transactions, generating spurious errors.

---

### Likelihood Explanation

- Admin user removal is a routine administrative action with no friction preventing it while transactions are in flight.
- Users routinely rotate or delete keys; no UI or API warning exists about pending transactions tied to a key.
- The window between transaction creation and expiry on Hedera can be minutes to hours (depending on `validStart`), making concurrent removal realistic.

---

### Recommendation

1. **In `removeUser`**: before soft-deleting the user, query for any transactions in non-terminal statuses where `creatorKey.userId = id`. Either reject the removal with an error, or automatically cancel/archive those transactions within the same operation.

2. **In `removeUserKey`**: before soft-deleting the key, query for live transactions where `creatorKeyId = id`. Reject the removal or cancel the affected transactions.

3. **Add an admin override**: introduce an admin-only path in `TransactionsService` (e.g., `cancelTransactionAsAdmin`) that bypasses the creator check, so orphaned transactions can be recovered without requiring the original creator.

---

### Proof of Concept

1. Admin creates user **Alice** (id=5). Alice uploads key K (id=10).
2. Alice creates a transaction T referencing `creatorKeyId=10`; T enters `WAITING_FOR_SIGNATURES`.
3. Admin calls `DELETE /users/5` → `removeUser(5)`:
   - `softDelete(UserKey, { userId: 5 })` marks key K deleted.
   - `softRemove(user)` marks Alice deleted.
4. Transaction T still has `creatorKeyId=10`, `status=WAITING_FOR_SIGNATURES`.
5. Any user (including admin) calls `POST /transactions/{T.id}/cancel`:
   - `getTransactionForCreator` loads T with `creatorKey` relation.
   - `transaction.creatorKey?.userId` returns `5` (soft-deleted key is still loaded by TypeORM relation).
   - Caller's `user.id` is not `5` → `UnauthorizedException` thrown.
6. Transaction T is permanently stuck. No API endpoint can move it to a terminal state. [6](#0-5) [7](#0-6) [2](#0-1)

```typescript
  // Remove a user from the organization.
  // This is a soft delete, meaning the deletedTimestamp will be set.
  // Also soft-deletes all keys belonging to the user.
  async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });

    if (!user) {
      throw new BadRequestException(ErrorCodes.UNF);
    }

    // Soft-delete all user keys first
    await this.repo.manager.softDelete(UserKey, { userId: id });

    // Then soft-delete the user
    await this.repo.softRemove(user);

    return true;
  }
```

```typescript
  async getTransactionForCreator(id: number, user: User) {
    const transaction = await this.getTransactionById(id);

    if (!transaction) {
      throw new BadRequestException(ErrorCodes.TNF);
    }

    if (transaction.creatorKey?.userId !== user?.id) {
      throw new UnauthorizedException('Only the creator has access to this transaction');
    }

    return transaction;
  }
```
