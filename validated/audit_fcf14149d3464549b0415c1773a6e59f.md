### Title
Missing Authorization in `removeTransactionApprover` Allows Any Authenticated User to Delete Approvers from Any Transaction

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` accepts only an approver `id` and performs no caller-identity check. Every other approver mutation (`createTransactionApprovers`, `updateTransactionApprover`) enforces that the requesting user is the transaction creator via `getCreatorsTransaction`. The remove path omits this check entirely, so any authenticated organization member can delete any approver from any transaction they did not create, bypassing the approval gate that controls whether a transaction may be executed.

### Finding Description

**Root cause — missing `user` parameter and authorization call in `removeTransactionApprover`:** [1](#0-0) 

The function signature is `removeTransactionApprover(id: number)`. It fetches the approver, calls `removeNode`, and returns — no user identity is ever consulted.

**Contrast with `createTransactionApprovers`**, which immediately calls `getCreatorsTransaction` to enforce creator-only access: [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which also calls `getCreatorsTransaction` inside the transaction block: [3](#0-2) 

**`getCreatorsTransaction` is the intended authorization gate:** [4](#0-3) 

It verifies `transaction.creatorKey?.userId !== user.id` and throws `UnauthorizedException` on mismatch. `removeTransactionApprover` never invokes it.

**Exploit path:**
1. Attacker (any authenticated user) learns or enumerates a `TransactionApprover.id` belonging to a transaction they did not create. Approver IDs are sequential integers and are returned in API responses visible to approvers and observers of the same transaction.
2. Attacker sends `DELETE /transactions/:transactionId/approvers/:id` (or equivalent route) with their own valid JWT.
3. `removeTransactionApprover(id)` is called; no user check occurs; `removeNode` soft-deletes the entire approver subtree rooted at that node.
4. If all approvers are removed, the approval requirement is eliminated and the transaction can proceed to execution without the required approvals.

### Impact Explanation
An attacker who is any authenticated member of the organization can:
- Delete individual approvers or entire approver trees from transactions they do not own.
- Reduce or eliminate the approval threshold, allowing a transaction to advance to `WAITING_FOR_EXECUTION` or `EXECUTED` without the required approvals.
- This constitutes unauthorized state mutation of the transaction approval workflow — a critical integrity failure in the trust model that governs which Hedera transactions are authorized to execute.

### Likelihood Explanation
- **Precondition:** The attacker must be a registered, authenticated user in the organization — a normal user role, no privilege required.
- **Approver ID discovery:** IDs are sequential integers returned in API responses to any user who has read access to a transaction (observers, signers, approvers, creator). A user added as an approver to one transaction can observe IDs and attempt removal on others.
- **No rate-limit or secondary confirmation** is required for the delete operation.
- Likelihood is **high** for any multi-user organization deployment.

### Recommendation
Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` (or an equivalent inline check) before invoking `removeNode`, mirroring the pattern used in `createTransactionApprovers` and `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Enforce creator-only access, same as create/update paths
  await this.getCreatorsTransaction(approver.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Propagate the `user` argument from the controller handler down to this service call.

### Proof of Concept

**Setup:**
- Organization with two users: Alice (transaction creator) and Bob (any authenticated member).
- Alice creates a transaction with an approver tree requiring Bob's approval before execution.

**Attack steps:**
1. Bob authenticates and obtains a JWT.
2. Bob calls `GET /transactions/:id/approvers` (he is listed as an approver, so he has read access) and records the approver node IDs.
3. Bob sends:
   ```
   DELETE /transactions/:id/approvers/:approverId
   Authorization: Bearer <Bob's JWT>
   ```
4. `removeTransactionApprover(approverId)` executes with no user check; `removeNode` soft-deletes the approver subtree.
5. The transaction's approval requirement is now gone. The transaction status advances without Alice's intended approval gate.

**Expected (correct) behavior:** `UnauthorizedException` — only the transaction creator may remove approvers.
**Actual behavior:** `200 OK`, approver deleted.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
