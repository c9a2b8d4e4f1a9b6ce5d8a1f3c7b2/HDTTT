### Title
Missing Parent Threshold Validation in `removeTransactionApprover` Allows Unsatisfiable Approval State

### Summary

`removeTransactionApprover` deletes a child approver node without checking or updating the parent threshold node. This mirrors the external report's pattern exactly: validation enforced in one code path (`updateTransactionApprover`) is absent in the parallel path (`removeTransactionApprover`), allowing `parent.threshold > parent.approvers.length` — a state where the approval requirement can never be met.

### Finding Description

The system supports a hierarchical approver tree where a parent node holds a `threshold` (minimum approvals required) and a list of child `approvers`. The invariant `threshold <= approvers.length` is enforced at creation time and in `updateTransactionApprover`.

`updateTransactionApprover` (called with `dto.listId = null` to detach a child) explicitly handles the post-removal threshold: [1](#0-0) 

However, `removeTransactionApprover` simply calls `removeNode` — a recursive soft-delete — with no parent threshold adjustment: [2](#0-1) 

`removeNode` only marks the target node and all its descendants as deleted. It never touches the parent record: [3](#0-2) 

The controller exposes this as `DELETE /transactions/:transactionId/approvers/:id`, gated only by a creator check: [4](#0-3) 

### Impact Explanation

After removing a child whose deletion causes `parent.threshold > remaining_children`, the `isApproved` evaluation: [5](#0-4) 

can never return `true` for that subtree because `approvals.length` can never reach `approver.threshold`. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` with no recovery path short of the transaction expiring. All prior approver signatures are wasted and the intended multi-party workflow is broken.

### Likelihood Explanation

The transaction creator — a normal authenticated user — has direct access to `DELETE /transactions/:transactionId/approvers/:id`. No elevated privilege is required. The scenario arises naturally: a creator sets up a threshold group of N approvers with threshold = N, then removes one member (e.g., a user who left the organization), not realizing the threshold is now unsatisfiable. The system provides no warning or guard.

### Recommendation

In `removeTransactionApprover`, after soft-deleting the node, fetch the parent (if any) and apply the same threshold correction logic already present in `updateTransactionApprover`:

- If `parent.approvers.length - 1 === 0`, soft-delete the parent as well.
- Else if `parent.threshold > parent.approvers.length - 1`, update `parent.threshold` to `parent.approvers.length - 1`.

This mirrors the fix pattern from the external report: apply the same invariant check in the execution path that already exists in the modification path.

### Proof of Concept

1. Creator creates a transaction with a threshold approver group:
   ```
   POST /transactions/1/approvers
   { approversArray: [{ threshold: 2, approvers: [{ userId: 10 }, { userId: 11 }] }] }
   ```
   DB state: Parent (id=5, threshold=2), children UserA (id=6), UserB (id=7).

2. Creator removes UserA:
   ```
   DELETE /transactions/1/approvers/6
   ```
   `removeNode(6)` soft-deletes node 6. Parent node 5 still has `threshold = 2`.

3. DB state after deletion: Parent (id=5, threshold=2, active_children=[UserB]).

4. UserB approves — `approvals.length = 1`, `approver.threshold = 2` → `1 >= 2` is `false`.

5. The approval condition can never be satisfied. The transaction is permanently blocked. [6](#0-5) [1](#0-0)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
