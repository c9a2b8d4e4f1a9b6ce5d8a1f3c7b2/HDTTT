### Title
Missing Authorization Guard on Transaction Comment Creation Allows Any Authenticated User to Comment on Any Transaction

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` is explicitly marked with a `TODO` noting that a guard or check to ensure the user can comment is missing. No such check was ever implemented. Any verified, authenticated user can post comments on any transaction in the organization, regardless of whether they are a participant (creator, signer, observer, or approver) of that transaction.

### Finding Description
In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the `createComment` handler carries only the class-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`), which verify that the caller holds a valid, non-blacklisted JWT and has a verified account. There is no check that the calling user has any relationship to the target transaction. [1](#0-0) 

The developer explicitly acknowledged this gap with the comment on line 22: [2](#0-1) 

The underlying service method performs no membership check either — it blindly creates and persists the comment for whatever `transactionId` is supplied: [3](#0-2) 

The `GET` routes on the same controller (`getComments`, `getCommentById`) are similarly unguarded at the transaction level, meaning any authenticated user can also read all comments on any transaction. [4](#0-3) 

### Impact Explanation
- **Information disclosure**: An authenticated user with no relationship to a transaction can read all comments on it, which may contain sensitive operational details, key management decisions, or internal approval rationale.
- **Comment spam / integrity**: Any authenticated user can inject arbitrary comments into any transaction's audit trail, polluting the record and potentially misleading approvers or observers.
- **Privilege boundary violation**: The transaction comment thread is implicitly scoped to transaction participants; the missing guard breaks that boundary entirely.

### Likelihood Explanation
The attack path requires only a valid organization account (any verified user). The attacker needs only to know or enumerate a `transactionId` (integer IDs are sequential and trivially guessable). No elevated privilege, no special tooling — a single authenticated HTTP `POST` is sufficient.

### Recommendation
Add a guard or service-layer check inside `createComment` (and the `GET` routes) that verifies the requesting user is a participant of the target transaction — i.e., is the creator, an assigned signer, an observer, or an approver. If no such relationship exists, return `403 Forbidden`. This is the check the TODO on line 22 explicitly deferred.

### Proof of Concept
```
# 1. Authenticate as any verified org user → obtain JWT
POST /auth/login  { "email": "attacker@org.com", "password": "..." }
→ Set-Cookie: access_token=<JWT>

# 2. Post a comment on a transaction the attacker has no relation to
POST /transactions/42/comments
Cookie: access_token=<JWT>
Content-Type: application/json
{ "content": "Injected comment on transaction 42" }

# Expected: 403 Forbidden
# Actual:   201 Created — comment is persisted to transaction 42's thread
```

The `transactionId` path parameter is parsed with `ParseIntPipe` and passed directly to `CommentsService.createComment` with no membership validation, confirming the vulnerability is reachable with a single authenticated request. [5](#0-4)
