## Audit Report

## Title
Concurrent Signature Submissions Create Duplicate `TransactionSigner` Records via TOCTOU Race — No Unique Constraint Enforced

## Summary
The `uploadSignatureMaps` function in `SignersService` performs its deduplication check outside the database transaction. Two concurrent POST requests to `/transactions/:id/signers` carrying the same signature map will both pass the "already signed" check and both successfully insert a `TransactionSigner` row for the same `(transactionId, userKeyId)` pair. The `transaction_signer` table has no `UNIQUE` constraint on that pair, so the database silently accepts both inserts.

## Finding Description

**Step 1 — Non-unique index on `transaction_signer`.**

The `TransactionSigner` entity uses `@Index(['transactionId', 'userKeyId'])` without `unique: true`, creating only a performance index, not an integrity constraint. [1](#0-0) 

The migration DDL for `transaction_signer` confirms no `UNIQUE` constraint is defined on the column pair — only a `PRIMARY KEY` on `id`. [2](#0-1) 

**Step 2 — Deduplication read is outside the DB transaction.**

`loadTransactionData` reads all existing `TransactionSigner` rows and builds `signersByTransaction` (a `Map<number, Set<number>>`) before any write transaction is opened. [3](#0-2) 

**Step 3 — The "new signer" decision is made on the stale snapshot.**

`processTransactionSignatures` consults `existingSignerIds` (derived from the snapshot above) to decide whether to push a key into `userKeys`. This decision is still outside the DB transaction. [4](#0-3) 

**Step 4 — The DB transaction only covers the write phase.**

By the time `persistSignatureChanges` opens `this.dataSource.transaction(...)`, both concurrent requests have already decided `userKeys.length > 0`. Both enter the transaction and call `bulkInsertSigners`. [5](#0-4) 

**Step 5 — `bulkInsertSigners` performs a plain insert with no conflict handling.**

The insert uses TypeORM's `.insert().into(TransactionSigner).values(...)` with no `orIgnore()`, `orUpdate()`, or `ON CONFLICT DO NOTHING` clause. With no database-level unique constraint to reject the second insert, both rows are committed. [6](#0-5) 

## Impact Explanation
Duplicate `TransactionSigner` rows for the same `(transactionId, userKeyId)` pair corrupt the signing state of a transaction. Downstream logic that counts distinct signers (e.g., `processTransactionStatus`) may count the same key twice, causing a transaction to appear fully signed before it actually is, or producing incorrect threshold evaluations. It also pollutes audit trails and any UI that displays who has signed a transaction.

## Likelihood Explanation
The race window is the entire async execution path from `loadTransactionData` through `validateAndProcessSignatures` to `persistSignatureChanges` — a span that includes multiple `await` points and database round-trips. Any client (or automated tool) that submits the same signature map twice in rapid succession, or any retry logic on a slow network, can trigger this. No special privileges are required beyond being an authenticated user with a valid key for the transaction.

## Recommendation
Apply both a database-level constraint and an application-level guard:

1. **Add a `UNIQUE` constraint** on `(transactionId, userKeyId)` in the `transaction_signer` table via a new migration, and change the `@Index` decorator to `@Unique` in the entity.
2. **Use `INSERT ... ON CONFLICT DO NOTHING`** (or TypeORM's `.orIgnore()`) in `bulkInsertSigners` so that a race-induced duplicate is silently discarded rather than causing an error.
3. Optionally, move the `loadTransactionData` read **inside** the same database transaction as the write, using `SERIALIZABLE` isolation or a `SELECT ... FOR UPDATE` lock on the relevant rows to eliminate the TOCTOU window entirely.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
