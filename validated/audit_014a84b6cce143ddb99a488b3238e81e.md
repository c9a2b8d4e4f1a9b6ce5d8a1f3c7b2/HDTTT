### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Sabotage of Multi-Sig Transactions

### Summary
The `SignersService.uploadSignatureMaps` endpoint accepts a user-submitted `SignatureMap` and adds it to a transaction's bytes after only verifying that the public key belongs to the authenticated user. It never cryptographically verifies that the submitted signature bytes are actually valid for that public key. A required signer can submit their own registered public key paired with completely invalid/random signature bytes, causing the system to record a fraudulent "signed" state, advance the transaction to `WAITING_FOR_EXECUTION`, and ultimately cause permanent failure on the Hedera network.

### Finding Description

**Root cause:** `processTransactionSignatures` in `signers.service.ts` performs only an ownership check (is the public key registered to this user?) but skips cryptographic verification of the signature bytes before calling `sdkTransaction.addSignature`.

The vulnerable path is `POST /transactions/:transactionId/signers` → `SignersController.uploadSignatureMap` → `SignersService.uploadSignatureMaps` → `validateAndProcessSignatures` → `processTransactionSignatures`. [1](#0-0) 

The loop iterates over every public key in the attacker-supplied `signatureMap`. For each key it:
1. Checks the key exists in `userKeyMap` (built from `user.keys`) — ownership check only.
2. Immediately calls `sdkTransaction.addSignature(publicKey, map)` — no cryptographic check. [2](#0-1) 

Contrast this with `TransactionsService.importSignatures`, which explicitly calls `validateSignature` before `addSignature`: [3](#0-2) 

`validateSignature` performs `publicKey.verify(bodyBytes, signature)` and throws on failure: [4](#0-3) 

Because `addSignature` adds the public key to `sdkTransaction._signerPublicKeys` without verifying the bytes, the execution service's `hasValidSignatureKey` check will see the key as having signed: [5](#0-4) 

The transaction is then submitted to the Hedera network with an invalid signature and permanently fails.

### Impact Explanation

A required signer for a multi-sig organization transaction can:
1. Submit their registered public key with arbitrary/random signature bytes via `POST /transactions/:id/signers`.
2. The system records the key as having signed and updates `transactionBytes` with the invalid signature.
3. `processTransactionStatus` advances the transaction to `WAITING_FOR_EXECUTION` if threshold is met.
4. The chain service submits the transaction to Hedera; Hedera rejects it with an invalid signature error.
5. The transaction is permanently marked `FAILED`.

Once `FAILED`, the transaction cannot be re-signed or re-executed: [6](#0-5) 

**Concrete impact:** Permanent, irreversible sabotage of any multi-sig transaction by any required signer. This is especially severe for high-value or time-sensitive Hedera operations (e.g., `FreezeTransaction`, `FileUpdateTransaction`, `NodeUpdateTransaction`).

### Likelihood Explanation

- **Attacker preconditions:** Must be an authenticated organization user with a registered public key that is listed as a required signer for the target transaction. This is a normal, non-privileged role in the system.
- **Attack complexity:** Trivial. The attacker simply submits their valid public key paired with 64 bytes of zeros as the signature. No cryptographic knowledge required.
- **Detection:** The attack is silent until the Hedera network rejects the transaction. No server-side error is raised during signature upload.

### Recommendation

Add cryptographic signature verification inside `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const signatureBytes = /* extract from map for this publicKey */;
const bodyBytes = /* transaction body bytes */;
if (!publicKey.verify(bodyBytes, signatureBytes)) {
  throw new Error(ErrorCodes.ISNMPN); // Invalid signature
}
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
```

Alternatively, refactor to reuse the existing `validateSignature` utility: [7](#0-6) 

### Proof of Concept

1. Register as an organization user with public key `PK_A` (ECDSA).
2. Be assigned as a required signer on a multi-sig transaction `TX_1`.
3. Construct a `SignatureMap` containing `PK_A` mapped to 64 zero bytes as the signature.
4. `POST /transactions/TX_1/signers` with this map using a valid JWT.
5. Observe: server returns HTTP 201, `TransactionSigner` record is created for `PK_A`.
6. If `PK_A` was the last required key, the transaction advances to `WAITING_FOR_EXECUTION`.
7. The chain service submits the transaction to Hedera; Hedera returns `INVALID_SIGNATURE`.
8. Transaction status is permanently set to `FAILED`. [8](#0-7)

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
