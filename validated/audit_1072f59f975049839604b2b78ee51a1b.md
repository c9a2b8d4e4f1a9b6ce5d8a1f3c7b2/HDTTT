# Audit Report

## Title
`getTransactionsToSign()` Fetches All Active Transactions Without Pagination, Causing Unbounded Per-Transaction CPU and I/O Work on Every Request

## Summary
`getTransactionsToSign()` in `back-end/apps/api/src/transactions/transactions.service.ts` issues an unbounded `repo.find()` with no `take`/`skip`, loading every active transaction in the system into memory. It then serially deserializes each transaction's bytes and performs a mirror-node/account-cache lookup via `computeSignatureKey()` before applying pagination in memory. Any authenticated user can trigger this endpoint, causing CPU and I/O load proportional to the total number of active transactions per request.

## Finding Description

**Unbounded database query — root cause:**

`getTransactionsToSign()` issues a `repo.find()` with no `take` or `skip`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

This is the only paginated endpoint in the service that omits these constraints. `getHistoryTransactions()` correctly passes `skip: offset, take: limit`: [9](#0-8) 

And `getTransactionsToApprove()` also correctly passes `skip: offset, take: limit`: [10](#0-9) 

**O(N) per-transaction CPU and I/O work:**

After loading all rows, the function serially awaits `userKeysToSign()` for each transaction:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [3](#0-2) 

`userKeysToSign()` calls `userKeysRequiredToSign()` → `keysRequiredToSign()`, which per transaction:
1. Deserializes `transaction.transactionBytes` via `SDKTransaction.fromBytes()` — CPU-intensive protobuf decode
2. Calls `transactionSignatureService.computeSignatureKey()` — mirror-node/account-cache network I/O [7](#0-6) 

Pagination is only applied after the full loop completes:

```typescript
items: result.slice(offset, offset + limit),
``` [4](#0-3) 

**Endpoint is accessible to any verified user and is explicitly dead code:**

The controller exposes `GET /transactions/sign` to any verified (non-admin) user, and the comment at line 157 explicitly states it is no longer used by the front-end: [5](#0-4) 

**Rate limits are far too permissive to prevent this:**

The Helm values configure `GLOBAL_MINUTE_LIMIT: "10000"` and `GLOBAL_SECOND_LIMIT: "1000"`, which allow hundreds of concurrent requests before any throttling applies: [11](#0-10) 

## Impact Explanation

With N active transactions in the system, each call to `GET /transactions/sign`:
- Loads all N transaction rows (including `transactionBytes` blobs) into the API process heap
- Performs N protobuf deserializations (`SDKTransaction.fromBytes()`)
- Performs N mirror-node/account-cache lookups (`computeSignatureKey()`)

Concurrent requests multiply this load linearly. The API container is configured with a memory limit of 250Mi and a CPU limit of 250m: [12](#0-11) 

These tight resource limits mean that even a modest number of active transactions combined with a small number of concurrent requests is sufficient to exhaust the container's CPU and memory, causing OOM kills or CPU throttling that degrades or denies service for all API users.

## Likelihood Explanation

- **Attacker precondition**: A valid JWT for any verified user account. No admin access required.
- **Transaction seeding**: Any verified user can create transactions via `POST /transactions`. Each created transaction persists in an active (non-terminal) status until executed, expired, or canceled, naturally accumulating over time in a production deployment.
- **Trigger**: A single `GET /transactions/sign` request with any pagination parameters forces the full O(N) work.
- **Amplification**: The attacker can issue concurrent requests. The global rate limit of 1000 req/s is far too permissive to prevent this.
- **No operational justification**: The endpoint is explicitly marked as no longer used by the front-end, so there is no legitimate traffic to weigh against the risk.

## Recommendation

1. **Immediate**: Remove or disable the `GET /transactions/sign` endpoint entirely, since it is explicitly noted as no longer used by the front-end. This eliminates the attack surface with zero functional regression.
2. **If the endpoint must be retained**: Push `skip: offset, take: limit` into the `repo.find()` call, as is done in `getHistoryTransactions()` and `getTransactionsToApprove()`. Note that this changes the semantics of `totalItems` (it would need a separate `count()` query), but it bounds per-request work to a fixed page size.
3. **Defense in depth**: Apply a per-endpoint or per-user rate limit significantly tighter than the global 1000 req/s for any endpoint that performs per-row async work.

## Proof of Concept

**Setup:**
1. Register and verify a user account (standard user, no admin privileges).
2. Using the authenticated user, call `POST /transactions` repeatedly to create a large number of transactions (e.g., 500–1000). Each transaction will remain in `WAITING_FOR_SIGNATURES` status (active/non-terminal).

**Trigger:**
3. Issue concurrent `GET /transactions/sign?page=1&limit=10` requests from the attacker's account (e.g., 20–50 concurrent requests using a tool like `ab` or `k6`).

**Expected outcome:**
- Each request causes the server to load all N transaction rows into memory, deserialize each transaction's bytes, and call `computeSignatureKey()` for each one before returning 10 results.
- With 1000 active transactions and 50 concurrent requests, the API process must handle 50,000 protobuf deserializations and 50,000 mirror-node lookups simultaneously.
- Given the 250Mi memory and 250m CPU limits on the API container, this will cause CPU throttling and/or OOM termination, resulting in 503 errors for all API users.
