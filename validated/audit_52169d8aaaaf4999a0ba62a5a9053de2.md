Audit Report

## Title
TOCTOU Race Condition in `approveTransaction` Allows an Approver to Override Their Own Submitted Approval

## Summary
`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` reads the approver's current signature state, performs several sequential async database and cryptographic operations, and only then writes the new approval. Because the "already approved" guard uses a snapshot taken before those `await`s, two concurrent HTTP requests from the same authenticated approver can both pass the guard and write conflicting `approved` values, letting the user silently change a decision that the protocol treats as final.

## Finding Description

**Root cause — stale-read guard with async gap before write**

`approveTransaction` (line 547) follows this sequence:

1. **Read** — `getVerifiedApproversByTransactionId` fetches all approver rows and the result is stored in `userApprovers` (line 553–556). [6](#0-5) 

2. **Guard** — throws `TAP` only when *every* approver row for this user already has a `signature` (line 563). This check uses the stale in-memory snapshot. [2](#0-1) 

3. **Async gap** — `attachKeys` (line 566), `findOne(Transaction)` (line 575), `SDKTransaction.fromBytes` (line 590), and `verifyTransactionBodyWithoutNodeAccountIdSignature` (line 593–596) all `await` between the guard and the write, yielding the event loop. [3](#0-2) 

4. **Write** — `UPDATE transaction_approver SET userKeyId = ?, signature = ?, approved = ? WHERE id IN (...)` executes with no database lock, no `SELECT … FOR UPDATE`, and no optimistic-concurrency version check (lines 599–610). [4](#0-3) 

5. **Stale notification** — the notification branch at lines 614–618 evaluates `userApprovers.every(a => a.approved)` against the pre-write snapshot, where `a.approved` is still `undefined`, so the emitted event reflects the wrong state. [5](#0-4) 

**Exploit flow**

An authenticated approver sends two concurrent `POST /transactions/:id/approvers/approve` requests — one with `approved: true` and one with `approved: false`. Both execute step 1 and step 2 against the same database snapshot (no signature yet), both pass the guard, and both proceed through the async gap. Whichever write lands last wins, overwriting the first. The one-approval-per-user invariant is violated.

## Impact Explanation

A legitimate approver can flip their own recorded decision (`approved: true → false` or vice versa) after it has already been persisted, without triggering the duplicate-approval guard. In a threshold-approval workflow this can:

- Prevent a transaction from reaching `WAITING_FOR_EXECUTION` by retracting a previously cast approval.
- Force a transaction into execution by retracting a previously cast rejection.
- Cause the notification subsystem to emit a status event based on stale approval state, desynchronising observer UIs.

The attacker controls only their own approver record, so they cannot forge another user's signature. Impact is scoped to transactions where the attacker is a designated approver.

## Likelihood Explanation

Any authenticated user who is listed as an approver can trigger this with two simultaneous HTTP requests — no special tooling, no privileged credentials, no network position. The async gap spans at least two database round-trips and one cryptographic verification, giving a wide enough window for concurrent requests to interleave reliably.

## Recommendation

**Option A — Re-read inside a serializable transaction with a row-level lock:**
Wrap the entire read-guard-write sequence in a single `dataSource.transaction` block and issue a `SELECT … FOR UPDATE` (or TypeORM's `setLock('pessimistic_write')`) on the approver rows before the guard check. This prevents a second concurrent request from reading the same rows until the first transaction commits.

**Option B — Optimistic concurrency / unique constraint:**
Add a database-level unique constraint on `(transactionId, userId)` for the `signature` column (or a separate `approvedAt` timestamp), and use an `UPDATE … WHERE signature IS NULL` pattern. If the row was already updated by a concurrent request, the affected-row count will be 0 and the second request can be rejected.

**Option C — Application-level mutex (short-term mitigation):**
Use a per-`(transactionId, userId)` in-process lock (e.g., a `Map` of `Promise` chains or a Redis-based distributed lock) to serialize concurrent approval requests for the same user and transaction. This is less robust than a DB-level solution but can be applied quickly.

The notification branch at lines 614–618 should also be updated to re-read the post-write approver state rather than relying on the stale `userApprovers` snapshot. [5](#0-4) 

## Proof of Concept

```bash
# Approve with approved=true and approved=false simultaneously
curl --parallel \
  --request POST "https://<host>/transactions/<id>/approvers/approve" \
  --header "Authorization: Bearer <token>" \
  --json '{"userKeyId": <keyId>, "signature": "<sig>", "approved": true}' \
  --request POST "https://<host>/transactions/<id>/approvers/approve" \
  --header "Authorization: Bearer <token>" \
  --json '{"userKeyId": <keyId>, "signature": "<sig>", "approved": false}'
```

Both requests will:
1. Call `getVerifiedApproversByTransactionId` — both see `signature = null`, both pass the guard at line 563.
2. Race through `attachKeys`, `findOne`, `fromBytes`, `verifyTransactionBodyWithoutNodeAccountIdSignature`.
3. Both execute the `UPDATE` at lines 599–610. The second write overwrites the first, setting `approved` to whichever value arrived last — which the attacker can bias by adding a small delay to the preferred request.

The final `approved` value in the database will differ from the value the approver originally intended to be final, violating the one-approval-per-user invariant. [7](#0-6)

  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }
