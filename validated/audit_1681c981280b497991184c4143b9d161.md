### Title
Transaction Creator Can Delete Approvers From Any Transaction They Do Not Own

### Summary
The `removeTransactionApprover` endpoint in the approvers controller verifies only that the requesting user is the creator of the `transactionId` supplied in the URL path. It then deletes the approver identified by the separate `id` path parameter without verifying that this approver actually belongs to the supplied `transactionId`. Any authenticated user who has created at least one transaction can exploit this mismatch to silently remove approvers from transactions they do not own, bypassing the multi-signature approval workflow.

### Finding Description

**Root cause — missing cross-parameter ownership check**

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `DELETE /:id` handler performs two independent operations:

```
await this.approversService.getCreatorsTransaction(transactionId, user);  // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);                // deletes approver by id — no relation to transactionId
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction identified by `transactionId`. It says nothing about whether the approver identified by `id` belongs to that transaction.

`removeTransactionApprover` fetches the approver by its own primary key and soft-deletes it, with no check that `approver.transactionId === transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

**Exploit path**

1. Attacker (normal authenticated user) creates transaction `T_own` — they are now its creator.
2. Attacker discovers the numeric ID of an approver (`approver_id`) belonging to a victim's transaction `T_victim` (approver IDs are sequential integers; the GET endpoint for any transaction the attacker can observe leaks them).
3. Attacker sends:
   ```
   DELETE /transactions/{T_own}/approvers/{approver_id}
   ```
4. `getCreatorsTransaction(T_own, attacker)` passes — attacker owns `T_own`.
5. `removeTransactionApprover(approver_id)` executes — the approver from `T_victim` is soft-deleted with no further check.

### Impact Explanation

An attacker can unilaterally remove any approver from any transaction in the organization, regardless of who created it. This directly undermines the multi-signature approval workflow: required approvers can be stripped from a pending transaction, allowing it to proceed without the mandated signatures, or permanently disrupting the approval tree of a transaction that requires a threshold of approvers. In an organization relying on approval trees for financial or administrative Hedera transactions, this constitutes unauthorized state mutation with direct operational impact.

### Likelihood Explanation

The precondition is only that the attacker is an authenticated, verified organization user — the lowest privilege level in the system. Creating a transaction (to obtain a `transactionId` they own) is a standard product flow. Approver IDs are sequential integers discoverable by observing any transaction the attacker legitimately participates in. No admin credentials, leaked secrets, or special access are required.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the transaction whose ownership was checked:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  if (approver.transactionId !== transactionId)
    throw new BadRequestException(ErrorCodes.ANF); // or a dedicated error code
  ...
}
```

Pass `transactionId` from the controller into the service call so the binding is enforced at the data layer, not just at the authorization layer.

### Proof of Concept

**Setup:** Two users — Admin (creator of `T_victim` with approver `approver_id = 5`) and Attacker (creator of `T_own`).

**Step 1 — Attacker creates a transaction** (standard API flow, returns `T_own = 42`).

**Step 2 — Attacker discovers approver ID** by calling `GET /transactions/{any_transaction_attacker_can_see}/approvers` and noting the integer IDs returned.

**Step 3 — Attacker deletes victim's approver:**
```
DELETE /transactions/42/approvers/5
Authorization: Bearer <attacker_jwt>
```

**Expected (correct) response:** 401/403 — approver 5 does not belong to transaction 42.

**Actual response:** 200 `true` — approver 5 is soft-deleted from `T_victim`, bypassing the approval requirement. [1](#0-0) [2](#0-1)

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
