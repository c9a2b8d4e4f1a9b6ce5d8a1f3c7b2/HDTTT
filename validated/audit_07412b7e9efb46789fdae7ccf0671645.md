### Title
Malicious organization member can cause `createTransactions()` to fail entirely by pre-creating a conflicting transaction ID

### Summary
The `createTransactions()` function in `back-end/apps/api/src/transactions/transactions.service.ts` performs an all-or-nothing duplicate check across the entire batch of submitted transaction DTOs. If any single transaction ID in the batch already exists in the database with a non-terminal status, the entire batch is rejected with a `BadRequestException`. A malicious organization member who can observe or predict a transaction ID in a pending batch can race to create that transaction first, causing the entire batch to fail — directly analogous to the `pokeTokens()` griefing pattern in the external report.

### Finding Description

**Root cause — `createTransactions()`, lines 413–433:**

```typescript
// Batch check for existing transactions
const transactionIds = validatedData.map(v => v.transactionId);
const existing = await this.repo.find({
  where: {
    transactionId: In(transactionIds),
    status: Not(
      In([
        TransactionStatus.CANCELED,
        TransactionStatus.REJECTED,
        TransactionStatus.ARCHIVED,
      ]),
    ),
  },
  select: ['transactionId'],
});

if (existing.length > 0) {
  throw new BadRequestException(
    `Transactions already exist: ${existing.map(t => t.transactionId).join(', ')}`,
  );
}
``` [1](#0-0) 

There is no per-item handling. A single pre-existing transaction ID in the batch causes the entire `createTransactions()` call to throw, discarding all validation work and all other valid DTOs in the batch.

**Attack path:**

1. Attacker is a legitimate organization member (no privileged role required).
2. A batch of transactions is being prepared for submission. Hedera transaction IDs are `accountId@validStart` — the `accountId` of the creator is visible to all organization members, and the `validStart` timestamp is either observable from shared transaction bytes (which are distributed for signing before batch submission) or predictable within a narrow time window.
3. The attacker races to call the single-transaction creation endpoint with one of the IDs that will appear in the batch. This succeeds and stores the transaction with status `WAITING_FOR_SIGNATURES`.
4. The legitimate batch submission hits the check at line 429, finds the pre-existing ID, and throws `BadRequestException` for the **entire** batch.
5. The attacker can repeat this for every retry, indefinitely blocking batch creation. [2](#0-1) 

The single-item entry point `createTransaction()` delegates directly to `createTransactions()`, so the attacker uses the same normal API flow as any other user. [3](#0-2) 

### Impact Explanation

- All validation work (`validateAndPrepareTransaction` via `Promise.all`) is wasted on every failed attempt.
- The entire batch of legitimate transactions is blocked from being created.
- The attacker can sustain the DoS indefinitely at negligible cost (one API call per retry by the victim).
- No financial gain is required — pure griefing by a malicious insider.

### Likelihood Explanation

- **Precondition**: Attacker must be an organization member. This is an insider-threat scenario, but organization membership requires no privileged role — any invited user qualifies.
- **Transaction ID predictability**: Hedera transaction IDs embed the creator's account ID (known to all org members) and a `validStart` timestamp. Transaction bytes are shared with signers before execution, making the ID directly readable. Even without byte access, the timestamp is guessable within seconds.
- **Race window**: The window between batch preparation and submission is non-trivial in multi-signer workflows, giving the attacker ample time to act.

### Recommendation

Replace the all-or-nothing batch rejection with per-item handling. Collect duplicate IDs and either:
- Skip them and proceed with the non-duplicate items, returning a partial result with per-item status, or
- Return a structured error listing which IDs conflicted without aborting the entire batch.

This mirrors the pattern already used correctly in `cancelTransactionGroup()`, which tracks `canceled`, `alreadyCanceled`, and `failed` per item rather than throwing on the first conflict. [4](#0-3) 

### Proof of Concept

**Setup**: Two organization members, Alice (legitimate) and Bob (malicious). Alice prepares a batch of three transactions: `[tx1, tx2, tx3]`.

1. Bob observes the transaction bytes for `tx2` shared in the signing workflow and extracts its Hedera transaction ID: `0.0.5678@1700000000.000000000`.
2. Bob calls `POST /transactions` with `tx2`'s bytes. The server stores it with status `WAITING_FOR_SIGNATURES`. Response: `201 Created`.
3. Alice calls `POST /transactions/batch` (or equivalent) with `[tx1, tx2, tx3]`.
4. `createTransactions()` runs `validateAndPrepareTransaction` for all three (wasted work), then queries the DB and finds `tx2` already exists.
5. Line 429 throws: `BadRequestException: Transactions already exist: 0.0.5678@1700000000.000000000`.
6. Alice's entire batch fails. Bob repeats for `tx1` on Alice's next retry.

**Expected outcome**: Alice cannot successfully submit any batch that includes a transaction ID Bob has pre-registered, for as long as Bob's copy remains in a non-terminal status (`WAITING_FOR_SIGNATURES`).
