### Title
Unbounded Full-Table Scan with Per-Row Async Work in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then performs an async cryptographic key-computation call (`userKeysToSign`) for every row before applying pagination. As the number of active transactions grows, a single authenticated API request to `GET /transactions/sign` triggers O(N) database queries and cryptographic operations, exhausting server CPU and memory and degrading service for all users.

### Finding Description

**Root cause — missing `take` in the DB query:**

In `getTransactionsToSign`, the `repo.find()` call at line 295 has no `take` (SQL `LIMIT`) parameter, so it loads every non-terminal transaction in the system:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / no `skip`
});
``` [1](#0-0) 

**Unbounded per-row async work:**

The code then iterates over every returned row and calls `userKeysToSign` for each one. `userKeysToSign` internally calls `userKeysRequiredToSign`, which calls `attachKeys` (a DB query) and `transactionSignatureService.computeSignatureKey` (a potentially network-bound mirror-node call):

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Pagination is applied only **after** the full iteration:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

**Exposed endpoint:**

The endpoint `GET /transactions/sign` is live and reachable by any authenticated, verified user:

```typescript
@Get('/sign')
getTransactionsToSign(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
  ...
) {
  return this.transactionsService.getTransactionsToSign(user, paginationParams, sort, filter);
}
``` [4](#0-3) 

The comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end has moved away from this endpoint, but it remains fully live and unenforced.

**`userKeysToSign` cost per row:** [5](#0-4) 

Each call to `userKeysRequiredToSign` performs at minimum one DB round-trip (`attachKeys`) and one cryptographic/mirror-node call (`computeSignatureKey`). With N active transactions, a single request causes N sequential async operations.

**Throttling does not mitigate this:**

The `UserThrottlerGuard` limits to 100 requests/minute and 10 requests/second per user. [6](#0-5) 

Throttling limits request *frequency*, not per-request *work*. A single request with 10,000 active transactions still performs 10,000 DB + crypto operations before returning.

### Impact Explanation

As the organization accumulates active transactions (a normal operational outcome), each call to `GET /transactions/sign` consumes proportionally more server CPU, memory, and DB connection time. At sufficient scale:
- The NestJS event loop is blocked by sequential awaits across thousands of rows.
- The PostgreSQL connection pool is saturated.
- The API process may OOM or time out, causing HTTP 500/504 responses for all concurrent users.
- A single authenticated user can trigger this repeatedly within the throttle window (100 calls/minute × N rows of work each).

### Likelihood Explanation

**Attacker preconditions:** Any registered, verified organization member — no admin or privileged role required. The endpoint requires only a valid JWT (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`).

**Realistic trigger:** In an active organization with hundreds or thousands of pending multi-sig transactions (a normal operational state for the tool's target use case), a single `GET /transactions/sign` request already causes severe latency. No crafted input is needed; the vulnerability is triggered by normal system growth.

### Recommendation

Apply the database-level limit **before** the application-level loop. The correct fix is to push the key-eligibility check into the SQL query (as is already done for `getTransactionsToApprove` using a recursive CTE), or at minimum add a hard `take` cap on the initial `repo.find()` and restructure pagination to be DB-driven:

```typescript
// Option A: add take to the DB query and accept approximate pagination
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,
  skip: offset,
});

// Option B (preferred): push the user-key eligibility filter into SQL,
// mirroring the pattern used in getTransactionsToApprove (recursive CTE).
```

Also enforce a maximum page size cap in `PaginationParams` to prevent callers from requesting unbounded result sets.

### Proof of Concept

1. Register and verify a user account on the organization server.
2. Create (or have other users create) a large number of transactions in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status (e.g., 5,000 transactions — achievable via the `POST /transactions` endpoint or the bulk-create flow).
3. As the verified user, issue:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <valid_jwt>
   ```
4. Observe: despite requesting only 10 items, the server loads all 5,000 non-terminal transactions and calls `userKeysToSign` 5,000 times before returning. Response time scales linearly with N. At sufficient N, the request times out (HTTP 504) and concurrent requests to other endpoints are degraded or fail. [7](#0-6)

      transaction: Transaction;
      keysToSign: number[];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);

      items: result.slice(offset, offset + limit),

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
};
