## Audit Report

## Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Concurrent Signature Writes to Silently Corrupt Transaction Bytes

## Summary
`SignersService.uploadSignatureMaps` reads transaction state outside any database transaction, performs async signature processing on the snapshot, then writes updated bytes back with an unconditional `UPDATE` that carries no version guard and no status predicate. Two concurrent authenticated signers of the same transaction will each compute new bytes from the same original snapshot; the second writer's `UPDATE` silently overwrites the first writer's signatures. The same flaw exists in `importSignatures` in `transactions.service.ts`.

## Finding Description

**Root cause — read and write are not atomic**

`uploadSignatureMaps` executes in four distinct, non-atomic phases:

**Phase 1 — Read** (`loadTransactionData`, lines 131–133): fetches the transaction row with a plain `SELECT`, no `FOR UPDATE` / `SKIP LOCKED`. [1](#0-0) 

**Phase 2 — Check** (`validateTransactionStatus`, lines 201–215): inspects the in-memory snapshot's `status` field only. [2](#0-1) 

**Phase 3 — Process** (`processTransactionSignatures`, lines 223–266): deserializes the snapshot bytes and calls `addSignature` on them. [3](#0-2) 

**Phase 4 — Write** (`bulkUpdateTransactions`, lines 365–371): issues an unconditional `UPDATE … WHERE id = ANY(…)` with **no status predicate and no version/ETag column**. [4](#0-3) 

The DB transaction wrapping the writes (line 317) only makes the writes atomic with each other; the read happens entirely outside it. [5](#0-4) 

**Contrast with the protected path**

`processTransactionStatus` correctly uses optimistic locking (`AND status = :oldStatus`) so status transitions are safe. The `transactionBytes` column has no equivalent protection. [6](#0-5) 

**Status/bytes desynchronization**

`updateStatusesAndNotify` calls `processTransactionStatus` with the **in-memory** transaction objects (line 432), whose `transactionBytes` were updated at line 297. If User A's bytes+A are later overwritten in the DB by User B's bytes+B, the status evaluation was performed against bytes+A (in memory) and may have advanced the status to `WAITING_FOR_EXECUTION`, while the DB now stores bytes+B — only Key B's signature. [7](#0-6) [8](#0-7) 

**Same pattern in `importSignatures`**

`importSignatures` in `transactions.service.ts` reads without a lock (line 507–510), processes signatures on the snapshot (lines 542–555), then issues an unconditional bulk `UPDATE` (lines 595–601) with no status guard. [9](#0-8) [10](#0-9) 

## Impact Explanation

- **Permanent, silent signature loss**: Key A's signature is discarded from the stored bytes with no error, no log entry visible to users, and no automatic retry.
- **Status/bytes desynchronization**: The transaction status can advance to `WAITING_FOR_EXECUTION` (via the in-memory evaluation) while the stored bytes lack the required signatures, causing execution to fail or the transaction to be permanently stuck.
- **Integrity failure in the multi-signature trust model**: The core invariant — "a transaction in `WAITING_FOR_EXECUTION` has all required signatures in its stored bytes" — is violated without any privileged access.

## Likelihood Explanation

- **No privilege required**: Any two authenticated organization members who are both required signers of the same transaction can trigger this by signing simultaneously — a routine workflow in multi-signer organizations.
- **Wide race window**: The race window spans the entire async signature-processing duration (crypto operations, network I/O), making it reliably triggerable.
- **Realistic trigger**: The frontend's "Sign All" flow and automated signing scripts make concurrent submissions highly probable in practice.

## Recommendation

Move the read inside the same database transaction as the write, and acquire a row-level lock at read time:

```sql
SELECT ... FROM transaction WHERE id = ANY($1) FOR UPDATE
```

This ensures that concurrent requests for the same transaction are serialized. Alternatively, add an optimistic-lock column (e.g., `version INTEGER`) and include `AND version = :expectedVersion` in the `UPDATE` predicate, rejecting the write and returning a conflict error if the version has changed. Apply the same fix to `importSignatures` in `transactions.service.ts`.

## Proof of Concept

```
T=0  User A: loadTransactionData()  → reads bytes₀ (no sigs)
T=0  User B: loadTransactionData()  → reads bytes₀ (no sigs)
T=1  User A: processTransactionSignatures(bytes₀) → bytes_A (sig A)
T=1  User B: processTransactionSignatures(bytes₀) → bytes_B (sig B)
T=2  User A: bulkUpdateTransactions → DB: bytes_A
T=2  User B: bulkUpdateTransactions → DB: bytes_B  ← overwrites bytes_A
T=3  User A: processTransactionStatus(in-memory bytes_A) → status = WAITING_FOR_EXECUTION
             DB UPDATE status WHERE status = WAITING_FOR_SIGNATURES → succeeds

Result: DB holds status=WAITING_FOR_EXECUTION, transactionBytes=bytes_B (only sig B).
        Execution will fail: the bytes do not satisfy the required signature threshold.
```

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;

    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
