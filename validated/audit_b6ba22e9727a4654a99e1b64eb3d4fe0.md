Audit Report

## Title
Critically Weak PBKDF2 Iteration Count (2560) Enables Offline Brute-Force of Stored Hedera Private Keys

## Summary
The `deriveKey` function in `front-end/src/main/utils/crypto.ts` uses PBKDF2-SHA512 with only 2560 iterations to derive the AES-256-GCM key used to encrypt all stored Hedera private keys and organization credentials. This is approximately 82× below OWASP's current minimum recommendation of 210,000 iterations for PBKDF2-SHA512. The resulting ciphertext blobs are stored in a plain, unencrypted SQLite database at a predictable OS path. An attacker who obtains the database file — through cloud backup sync, malware, or OS-level backup — can brute-force the user's password at GPU speeds and recover all signing keys.

## Finding Description

**Root cause — weak KDF iteration count:**

`front-end/src/main/utils/crypto.ts`, `deriveKey`, line 6: [1](#0-0) 

The value `2560` is hardcoded. OWASP 2023 recommends ≥ 210,000 iterations for PBKDF2-SHA512. At 2560 iterations, a single modern GPU (e.g., RTX 4090) can test hundreds of millions of candidate passwords per second against this KDF.

**This KDF encrypts every stored private key:**

`front-end/src/main/services/localUser/keyPairs.ts`, `storeKeyPair`, line 76: [2](#0-1) 

**The same path encrypts organization credentials:**

`front-end/src/main/services/localUser/organizationCredentials.ts`, `encryptData`, line 301: [3](#0-2) 

**Database stored at a predictable, unprotected path:**

`front-end/src/main/db/prisma.ts`, `getDatabasePath`, line 11: [4](#0-3) 

The database is a plain SQLite file with no SQLCipher or OS-level encryption. The `KeyPair` table stores PBKDF2-encrypted private key blobs directly, readable by any process with filesystem access to the path.

**Note on correct Argon2id usage:** The application correctly uses Argon2id for password *authentication* hashing, but the *key derivation* for at-rest encryption of private keys uses the weak PBKDF2 path: [5](#0-4) 

## Impact Explanation
An attacker who obtains `database.db` can extract `private_key` blobs from the `KeyPair` table without any password. The known ciphertext format (`salt[64B] || iv[16B] || tag[16B] || ciphertext`) maps directly to hashcat mode 12100 (PBKDF2-SHA512). At 2560 iterations, brute-force speed on commodity hardware renders common passwords recoverable in hours. Recovered ED25519/ECDSA private keys can sign arbitrary Hedera `CryptoTransfer`, `TokenTransfer`, and `ScheduleSign` transactions, resulting in complete and irreversible theft of all HBAR and HTS tokens. Organization credentials encrypted with the same KDF are also exposed, enabling account takeover on the organization server.

## Likelihood Explanation
Database file acquisition does not require physical access and is realistic through multiple vectors:
- **Cloud backup sync**: macOS iCloud Drive, Windows OneDrive, and Google Drive Backup commonly sync `Application Support` / `AppData` directories by default, making the database file accessible to anyone with cloud account credentials.
- **OS-level backup**: Time Machine (macOS) and Windows Backup include the `userData` directory by default.
- **Malware**: A single file-read operation on the known, predictable path suffices — no privilege escalation required, as the file is readable by any process running as the same OS user.

Once the file is obtained, 2560 PBKDF2 iterations provide negligible resistance. Hashcat mode 12100 is a standard, well-optimized attack mode with publicly available GPU kernels.

## Recommendation
Increase the PBKDF2 iteration count in `deriveKey` to at least **210,000** (OWASP 2023 minimum for PBKDF2-SHA512), or migrate to **Argon2id** (already a dependency in the project) with parameters meeting OWASP recommendations (e.g., `m=19456`, `t=2`, `p=1`). A migration path for existing stored keys (re-encrypt on next successful login with the new KDF) should be implemented. Additionally, consider using Electron's `safeStorage` API (already used in the keychain path) as the default encryption mechanism where available, rather than the password-derived KDF path. [1](#0-0) 

## Proof of Concept
1. Locate `database.db` at the platform-specific `userData` path (e.g., `~/Library/Application Support/hedera-transaction-tool/database.db` on macOS).
2. Open the SQLite file with any SQLite client (no password required): `sqlite3 database.db "SELECT private_key FROM KeyPair;"`
3. Decode the base64 blob; extract `salt = blob[0:64]`, `iv = blob[64:80]`, `tag = blob[80:96]`, `ciphertext = blob[96:]`.
4. Run hashcat with mode 12100 (PBKDF2-SHA512), 2560 iterations, against a common password wordlist:
   ```
   hashcat -m 12100 -a 0 "<salt>:<ciphertext>" rockyou.txt
   ```
5. Use the recovered password to call `decrypt()` in `front-end/src/main/utils/crypto.ts` and obtain the raw ED25519/ECDSA private key.
6. Use the private key with the Hedera SDK to sign and submit `CryptoTransfer` transactions draining all associated accounts. [6](#0-5)

export function decrypt(data: string, password: string) {
  const bData = Buffer.from(data, 'base64');

  const salt = bData.subarray(0, 64);
  const iv = bData.subarray(64, 80);
  const tag = bData.subarray(80, 96);
  const text = bData.subarray(96).toString('base64');

  const key = deriveKey(password, salt);

  const decipher = crypto.createDecipheriv('aes-256-gcm', key, iv);
  decipher.setAuthTag(tag);

  const decrypted = decipher.update(text, 'base64', 'utf8') + decipher.final('utf8');

  return decrypted;
}
