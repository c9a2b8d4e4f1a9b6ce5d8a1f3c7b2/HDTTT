Audit Report

## Title
Bulk Approval in `ApproveAllController.vue` Executes Cryptographic Signatures Over All Group Transactions Without Displaying Individual Transaction Parameters

## Summary
The `ApproveAllController.vue` component allows an organization approver to cryptographically sign every transaction in a group with a single click. The only user-facing gate is a static generic confirmation prompt. No per-transaction parameters — recipient, amount, memo, or type-specific fields — are surfaced before signatures are produced and submitted. A malicious transaction creator (any authenticated organization member) can embed a harmful transaction inside a group alongside legitimate ones, and an approver using "Approve All" will unknowingly sign it.

## Finding Description

**Root cause — `ApproveAllController.vue`**

`handleApproveAll` (lines 57–111) performs the following silently:

1. Fetches the full group from the server via `getTransactionGroupById`.
2. Decrypts the approver's private key via `decryptPrivateKey`.
3. Iterates over every `groupItem`, deserializes `transactionBytes` via `Transaction.fromBytes`, computes a cryptographic signature via `getTransactionBodySignatureWithoutNodeAccountId`, and calls `sendApproverChoice` — with zero per-transaction field display. [1](#0-0) 

The only user-facing confirmation text is the static computed string: [2](#0-1) 

**What the group list page shows**

`TransactionGroupDetails.vue` renders a table with columns: Transaction ID, Transaction Type, Status, Valid Start, Actions. Recipient accounts, transfer amounts, memos, and other payload fields are never shown in this view. [3](#0-2) 

**Attacker-controlled entry point**

`CreateTransactionDto` accepts raw `transactionBytes` as a `Buffer` with no semantic validation of the payload — only a non-empty byte check via `@IsNotEmpty()` and `@TransformBuffer()`. [4](#0-3) 

`TransactionDto` serializes `transactionBytes` back to the client as a hex string, which `ApproveAllController.vue` decodes and signs verbatim. [5](#0-4) 

**Contrast with individual approval**

When an approver navigates to a single transaction's detail page, `TransactionDetails.vue` decodes the SDK transaction and renders type, transaction ID, valid start, fee payer, memo, and a transaction-specific component (recipient, amount, etc.) before the "Sign" button is reachable. [6](#0-5) 

The "Approve All" path entirely bypasses this display.

## Impact Explanation

An approver who uses "Approve All" provides a valid cryptographic approval signature over every transaction in the group, including any malicious one embedded by the creator. Once the approval threshold is met, the back-end chain service submits the transaction to the Hedera network. Concrete irreversible outcomes include:

- **Direct HBAR theft**: A `TransferTransaction` moving HBAR from an organization-controlled account to the attacker's account is approved and executed.
- **Token allowance escalation**: An `ApproveHbarAllowanceTransaction` granting the attacker unlimited spending rights over an organization account is silently approved.
- **Account key replacement**: An `AccountUpdateTransaction` replacing the account's key with the attacker's key is approved, permanently locking out the legitimate owner.

## Likelihood Explanation

The attacker must be an authenticated member of the target organization — a normal user with no elevated privileges, only the ability to create transactions. This is a realistic insider-threat scenario in any multi-user organization deployment. The "Approve All" button is a primary action on the group details page and is the natural path for approvers managing high-volume transaction groups. No special tooling is required beyond the standard API. [7](#0-6) 

## Recommendation

Before executing `handleApproveAll`, decode each `transactionBytes` using the Hedera SDK and render a per-transaction summary (type, recipient/target account, amount, memo, fee payer) in the confirmation modal — mirroring what `TransactionDetails.vue` already does for individual approvals. The approver must explicitly acknowledge each transaction's parameters, or at minimum scroll through a rendered list of decoded transaction details, before signatures are produced. [8](#0-7) 

## Proof of Concept

1. Attacker (authenticated org user) creates a transaction group via the API containing two items:
   - Item 1: A legitimate `TransferTransaction` (e.g., paying a vendor 1 HBAR).
   - Item 2: A malicious `TransferTransaction` sending 10,000 HBAR from the org account to the attacker's account, serialized to `transactionBytes` and submitted via `POST /transaction-groups`.
2. The group appears in the approver's queue. The group list table shows only Transaction ID, Type (`CRYPTO_TRANSFER`), Status, and Valid Start for both items — no amounts or recipients.
3. The approver clicks "Approve All". The `ActionController` shows the static prompt: *"Are you sure you want to approve all transactions?"*
4. The approver confirms and enters their personal password.
5. `handleApproveAll` iterates both items, calls `Transaction.fromBytes` on each, computes a signature, and calls `sendApproverChoice` for both — including the malicious transfer.
6. Once the approval threshold is met, the back-end submits the malicious `TransferTransaction` to the Hedera network. The 10,000 HBAR transfer executes and is irreversible. [9](#0-8)

const handleApproveAll = async (personalPassword: string | null): Promise<ActionReport | null> => {
  if (!isUserLoggedIn(user.personal)) {
    toastManager.error(`You must be logged in to ${action.value} transactions.`);
    return null;
  }
  assertIsLoggedInOrganization(user.selectedOrganization);
  if (props.groupOrId !== null) {
    const groupId = typeof props.groupOrId == 'number' ? props.groupOrId : props.groupOrId.id;
    try {
      let group: IGroup;
      if (typeof props.groupOrId === 'number') {
        const serverUrl = user.selectedOrganization.serverUrl;
        group = await getTransactionGroupById(serverUrl, props.groupOrId);
      } else {
        group = props.groupOrId;
      }

      progressText.value = `${props.approved ? 'Approving' : 'Rejecting'} ${group.groupItems.length} transactions`;

      const publicKey = user.selectedOrganization.userKeys[0].publicKey;
      const privateKeyRaw = await decryptPrivateKey(user.personal.id, personalPassword, publicKey);
      const privateKey = getPrivateKey(publicKey, privateKeyRaw);

      for (const item of group.groupItems) {
        if (await getUserShouldApprove(user.selectedOrganization.serverUrl, item.transaction.id)) {
          const transactionBytes = hexToUint8Array(item.transaction.transactionBytes);
          const transaction = Transaction.fromBytes(transactionBytes);
          const signature = getTransactionBodySignatureWithoutNodeAccountId(
            privateKey,
            transaction,
          );

          await sendApproverChoice(
            user.selectedOrganization.serverUrl,
            item.transaction.id,
            user.selectedOrganization.userKeys[0].id,
            signature,
            props.approved,
          );
        }
      }
      toastManager.success(`Transactions ${props.approved ? 'approved' : 'rejected'} successfully`);
    } catch (error) {
      toastManager.error(getErrorMessage(error, `Failed to ${action.value} transactions`));
    } finally {
      await invokeCallback(groupId);
      progressText.value = 'Loading group items…';
    }
  } else {
    // Bug
    toastManager.error(`Unable to ${action.value} transactions: group is not available`);
  }

  return null;
};
