Audit Report

## Title
`hasValidSignatureKey` Returns `true` for Empty `KeyList`, Bypassing Signature Validation When `computeSignatureKey` Yields No Keys

## Summary
`hasValidSignatureKey` unconditionally returns `true` when passed an empty `KeyList` with no threshold set. `computeSignatureKey` in `TransactionSignatureService` silently swallows all errors and skips adding keys when the account cache returns `null`. When both conditions coincide, the signature validation gates in `getValidatedSDKTransaction` and `processTransactionStatus` pass without verifying any actual signatures, allowing an unsigned or under-signed transaction to be promoted to `WAITING_FOR_EXECUTION` and submitted to the Hedera network.

## Finding Description

**Root cause — `hasValidSignatureKey` with empty `KeyList`:**

In `back-end/libs/common/src/utils/sdk/key.ts` lines 31–46:

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();   // [] when KeyList is empty
    let currentThreshold = 0;

    keys.forEach(key => {         // loop body never executes
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
    //     0                 >= (undefined    || 0          )
    //     0                 >= 0   →  true
  }
  ...
};
```

When `key` is an empty `KeyList` with no threshold, `key.threshold` is `undefined`, `keys.length` is `0`, so `key.threshold || keys.length` evaluates to `0`. The check `0 >= 0` is always `true`. [1](#0-0) 

**Trigger — `computeSignatureKey` silently drops keys on cache failure:**

`addFeePayerKey` (lines 82–99) catches all errors and returns without adding any key to `signatureKey`: [2](#0-1) 

`addSigningAccountKeys` (lines 104–122) has the same silent-catch pattern for every signing account: [3](#0-2) 

When `getAccountInfoForTransaction` returns `null`, neither method pushes any key, and `computeSignatureKey` returns a `KeyList` with zero items. This is explicitly documented in the test suite: [4](#0-3) 

**Validation gate — `getValidatedSDKTransaction` passes on empty `KeyList`:**

In `back-end/libs/common/src/execute/execute.service.ts` lines 216–220, the same `hasValidSignatureKey` call is used as the sole signature gate before submitting to Hedera: [5](#0-4) 

**Status promotion gate — `processTransactionStatus` promotes on empty `KeyList`:**

In `back-end/libs/common/src/utils/transaction/index.ts` lines 133–144, `hasValidSignatureKey` controls the transition to `WAITING_FOR_EXECUTION`. If it returns `true` and `smartCollate` returns non-null (which it does for any normally-sized transaction), the status is promoted: [6](#0-5) 

`smartCollate` only returns `null` when the transaction exceeds the maximum size limit; for any normal-sized transaction it returns the `sdkTransaction` unconditionally: [7](#0-6) 

## Impact Explanation
- Transactions are incorrectly promoted from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` without any signature check, corrupting the internal transaction state machine.
- The scheduler picks up these transactions and submits them to the Hedera network via `executeTransaction`, consuming scheduler resources and incurring Hedera network fees for the rejected submission.
- The Hedera network's own signature validation acts as a safety net, preventing actual unauthorized on-chain execution. This limits the severity — no funds can be moved — but the system's internal state becomes inconsistent and fees are wasted.

## Likelihood Explanation
- The trigger condition is reachable without privileged access: a newly created Hedera account may not yet be indexed by the Mirror Node, and any authenticated user can create a transaction with such an account as the fee payer.
- `addFeePayerKey` and `addSigningAccountKeys` silently swallow all errors, making the empty-`KeyList` outcome reachable under transient infrastructure conditions (Mirror Node lag, network partition, cache miss).
- The `hasValidSignatureKey` empty-`KeyList` bypass is deterministic once the empty list is produced — no probabilistic element.

## Recommendation
1. **Guard against empty `KeyList` in `hasValidSignatureKey`:** Add an explicit check at the top of the `KeyList` branch. If `keys.length === 0` and `key.threshold` is not set (or is `0`), return `false` rather than `true`. An empty key list with no threshold means "no keys are required," which should be treated as an unresolvable requirement, not a satisfied one.

```typescript
if (key instanceof KeyList) {
  const keys = key.toArray();
  const threshold = key.threshold ?? keys.length;
  if (threshold === 0) return false; // no keys required → treat as invalid
  ...
}
```

2. **Propagate cache failures instead of silently swallowing them:** In `addFeePayerKey` and `addSigningAccountKeys`, consider re-throwing the error (or returning a sentinel value) so that `computeSignatureKey` can surface the failure to its callers, preventing a silent empty-`KeyList` from being used as a valid signature requirement.

3. **Add a non-empty guard in `processTransactionStatus` and `getValidatedSDKTransaction`:** Before calling `hasValidSignatureKey`, verify that the returned `KeyList` is non-empty. If it is empty, treat the transaction as not yet signable rather than fully signed.

## Proof of Concept

1. Create a Hedera account (`0.0.X`) that is not yet indexed by the Mirror Node (e.g., immediately after creation, before Mirror Node propagation).
2. Submit a transaction to the tool with `0.0.X` as the fee payer, with no signatures attached.
3. A signature event is triggered, calling `processTransactionStatus`.
4. `computeSignatureKey` calls `addFeePayerKey` → `getAccountInfoForTransaction` returns `null` → no key is pushed → returns an empty `KeyList`.
5. `hasValidSignatureKey([], emptyKeyList)` → `0 >= 0` → `true`.
6. `smartCollate` is called with the empty `KeyList` → transaction is not over max size → returns non-null.
7. `newStatus` is set to `WAITING_FOR_EXECUTION`; the transaction is promoted in the database.
8. The scheduler calls `executeTransaction` → `getValidatedSDKTransaction` → same empty `KeyList` → same bypass → `sdkTransaction.execute(client)` is called.
9. Hedera rejects the transaction with `INVALID_SIGNATURE`, but the system has already recorded the incorrect state transition and consumed scheduler resources.

export async function smartCollate(
  transaction: Transaction,
  requiredKeys: KeyList,
): Promise<SDKTransaction | null> {
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // HIP-1300: isTransactionOverMaxSize is fee-payer aware. Privileged payers
  // (0.0.2, 0.0.42-0.0.799) can carry up to 128 KB, so transactions under that
  // limit pass through here untouched and signatures are NOT stripped.
  if (await isTransactionOverMaxSize(sdkTransaction)) {
    const publicKeys = computeShortenedPublicKeyList(
      sdkTransaction._signerPublicKeys,
      requiredKeys,
    );

    const signatureMap = sdkTransaction.getSignatures();
    sdkTransaction.removeAllSignatures();

    for (const key of publicKeys) {
      sdkTransaction.addSignature(key, signatureMap);
    }

    // If the transaction is still too large,
    // set it to failed with the TRANSACTION_OVERSIZE status code
    // update the transaction, emit the event, and delete the timeout
    if (await isTransactionOverMaxSize(sdkTransaction)) {
      return null;
    }
  }

  return sdkTransaction;
}
