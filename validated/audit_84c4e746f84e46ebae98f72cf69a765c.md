### Title
Creator Can Remove or Replace Rejecting Approvers to Bypass Transaction Approval Governance

### Summary
The transaction creator can add, remove, or update approvers on a transaction at any point in its lifecycle — including after approvers have already submitted rejections — because none of the three approver-mutation endpoints enforce a transaction-status guard. A malicious creator can exploit this to silently delete a rejecting approver's record, causing the status-recalculation engine to re-evaluate the approval tree without the rejection, and allowing the transaction to advance to `WAITING_FOR_EXECUTION` and ultimately be submitted to the Hedera network without the required consent.

### Finding Description

**Root cause — missing status guard on all three approver-mutation paths:**

**Path 1 — `DELETE /transactions/:transactionId/approvers/:id`** [1](#0-0) 

`getCreatorsTransaction` only verifies that the caller is the creator; it performs no check on `transaction.status`. After that, `removeTransactionApprover(id)` is called unconditionally. [2](#0-1) 

`removeTransactionApprover` fetches the approver and calls `removeNode` with no status guard whatsoever.

**Path 2 — `POST /transactions/:transactionId/approvers` (create)** [3](#0-2) 

Only `getCreatorsTransaction` is called before inserting new approvers; no status check.

**Path 3 — `PATCH /transactions/:transactionId/approvers/:id` (update userId)** [4](#0-3) 

When `dto.userId` is supplied, the existing approver's `userId`, `signature`, `userKeyId`, and `approved` fields are all cleared and replaced. This lets the creator swap out a rejecting approver for a compliant one, erasing the rejection.

**`getCreatorsTransaction` — the only guard, and it checks nothing about status:** [5](#0-4) 

**Status recalculation is triggered after every mutation:**

After each approver removal or update, `emitTransactionStatusUpdate` is published. The chain service's `processTransactionStatus` then re-evaluates whether the transaction should advance. With the rejecting approver gone, the recalculation sees a clean approval tree and can promote the transaction to `WAITING_FOR_EXECUTION`. [6](#0-5) 

**Approval stores a cryptographic signature over the *current* transaction bytes:** [7](#0-6) 

The signature is bound to the frozen transaction bytes, not to the approver-list configuration. Removing an approver record does not invalidate the bytes; it only removes the governance gate.

### Impact Explanation

The approval system is the sole governance control preventing a transaction from being submitted to the Hedera network without multi-party consent. A creator who receives a rejection can:

1. Delete the rejecting approver record (no status check blocks this).
2. Trigger a status recalculation that no longer sees the rejection.
3. Allow the transaction to advance to `WAITING_FOR_EXECUTION` and be executed on-chain.

Concrete example: an organization requires CFO approval before large HBAR transfers. The CFO rejects a suspicious transfer. The creator (malicious employee) calls `DELETE /transactions/:id/approvers/:cfoApproverId`. The rejection disappears, the status engine promotes the transaction, and the transfer executes on-chain — irreversibly, because Hedera transactions are final.

### Likelihood Explanation

- **Attacker profile**: any authenticated user who is the creator of a transaction — no admin or privileged role required.
- **Precondition**: the attacker must have created the transaction (a normal workflow step).
- **Trigger**: a single authenticated HTTP `DELETE` or `PATCH` request.
- **Detection difficulty**: the deletion leaves no audit trail of the original rejection in the approver table (soft-delete only marks `deletedAt`); the approver who rejected receives a status-update notification but no explicit "your rejection was removed" alert.

### Recommendation

Add a transaction-status guard to all three approver-mutation operations. Approver additions, removals, and updates should be rejected once the transaction has left the `NEW` or `WAITING_FOR_SIGNATURES` state (i.e., once any approver has cast a vote). A minimal fix:

```typescript
// In getCreatorsTransaction or at the top of each mutation handler:
const nonModifiableStatuses = [
  TransactionStatus.WAITING_FOR_EXECUTION,
  TransactionStatus.EXECUTED,
  TransactionStatus.CANCELED,
  TransactionStatus.EXPIRED,
  TransactionStatus.FAILED,
  TransactionStatus.ARCHIVED,
];
if (nonModifiableStatuses.includes(transaction.status)) {
  throw new BadRequestException('Approvers cannot be modified after approval voting has begun');
}
```

A stricter policy would also lock the approver list once *any* approver has set `approved` to any value (true or false), preventing mid-vote manipulation.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
