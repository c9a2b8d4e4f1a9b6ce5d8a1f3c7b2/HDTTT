### Title
Manual Transaction Creator Can Archive Transactions in Terminal States Due to Logical AND Error in Status Guard

### Summary
In `archiveTransaction` within `back-end/apps/api/src/transactions/transactions.service.ts`, the guard condition uses a logical AND (`&&`) between the status check and the `!transaction.isManual` check. Because both conditions must be true to throw, setting `isManual = true` short-circuits the status restriction entirely, allowing a transaction creator to archive their own transaction regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED`. This is a direct analog of the external report's uninitialized-value bypass: a default/flag value causes a prerequisite guard to silently pass.

### Finding Description

**Root cause — `archiveTransaction` guard logic:**

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual          // ← short-circuits the entire check when isManual=true
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
``` [1](#0-0) 

The guard is intended to block archiving unless the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. However, because the two sub-conditions are joined with `&&`, when `transaction.isManual === true` the expression `!transaction.isManual` evaluates to `false`, making the entire `if` condition `false` — the exception is never thrown, and the `repo.update` proceeds unconditionally.

**Contrast with the frontend guard**, which correctly enforces `transactionIsInProgress` (i.e., `NEW | WAITING_FOR_SIGNATURES | WAITING_FOR_EXECUTION`) even for manual transactions:

```typescript
const canArchive = computed(() => {
  const isManual = props.organizationTransaction?.isManual;
  return isManual && isCreator.value && transactionIsInProgress.value;  // status still checked
});
``` [2](#0-1) 

The backend does not replicate this status restriction for manual transactions, so a direct API call bypasses the frontend guard entirely.

**Exploit path:**
1. Attacker (transaction creator) creates a manual transaction (`isManual: true`) and submits it via `POST /transactions`.
2. The transaction progresses to `EXECUTED` (or `FAILED`, `EXPIRED`, `CANCELED`).
3. Attacker calls `PATCH /transactions/archive/:id` directly against the API.
4. The backend guard passes because `isManual = true` → `!isManual = false` → AND condition is false → no exception.
5. The transaction status is overwritten to `ARCHIVED`.

**Existing test gap:** The unit test for `archiveTransaction` only covers the case where `isManual` is falsy (unset), and the e2e test for "should not archive an already-executed transaction" uses a non-manual transaction. No test covers a manual transaction in a terminal state. [3](#0-2) [4](#0-3) 

### Impact Explanation

**Severity: High**

In a multi-signature transaction management tool, the terminal status of a transaction (`EXECUTED`, `FAILED`, `EXPIRED`) is the authoritative audit record for all participants — signers, observers, and approvers. By overwriting `EXECUTED` → `ARCHIVED`, a malicious creator can:

- **Erase execution evidence**: Other participants (signers, observers, approvers) lose the `EXECUTED` record and see only `ARCHIVED`, making it impossible to confirm whether a Hedera network transaction was submitted.
- **Hide failures**: A `FAILED` transaction can be silently archived, concealing errors from auditors and co-signers.
- **Corrupt group transaction history**: In transaction groups, individual item statuses feed into group-level reporting; corrupting one item's status corrupts the group view.

The `status` field is the sole source of truth for transaction lifecycle state in the PostgreSQL database. [5](#0-4) 

### Likelihood Explanation

**Likelihood: High**

- No privileged access is required — any authenticated user who created a manual transaction can exploit this.
- The exploit is a single direct HTTP `PATCH` request to a documented REST endpoint (`/transactions/archive/:id`).
- `isManual` is a standard field set at transaction creation time and is not restricted to any special role.
- The frontend guard is purely cosmetic; it is trivially bypassed by calling the API directly. [6](#0-5) 

### Recommendation

Replace the logical AND with a single, unconditional status allowlist check. The `isManual` flag should not exempt a transaction from the status restriction:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    )
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
```

Additionally, add a unit test covering `isManual: true` with a terminal status (e.g., `EXECUTED`) to confirm the guard rejects it.

### Proof of Concept

**Preconditions:** Attacker is an authenticated user with a valid session cookie/JWT.

1. Create a manual transaction:
   ```http
   POST /transactions
   { ..., "isManual": true }
   → { "id": 42, ... }
   ```

2. Wait for (or observe) the transaction reaching `EXECUTED` status (or any terminal state).

3. Call archive directly, bypassing the frontend:
   ```http
   PATCH /transactions/archive/42
   Cookie: <attacker session>
   → 200 true
   ```

4. Query the transaction:
   ```http
   GET /transactions/42
   → { "status": "ARCHIVED", ... }   // was EXECUTED
   ```

**Expected (correct) behavior:** Step 3 returns `400 Bad Request` with `ErrorCodes.OMTIP`.
**Actual behavior:** Step 3 returns `200 true` and the status is permanently overwritten to `ARCHIVED`. [7](#0-6)
