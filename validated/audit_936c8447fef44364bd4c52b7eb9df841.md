Audit Report

## Title
Unauthorized Approver Removal via IDOR in `removeTransactionApprover` Bypasses Approval Workflow

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId` but deletes the approver record identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who is the creator of any transaction can silently remove approvers from transactions they do not own.

## Finding Description

**Root cause â€” missing cross-ownership check in `removeTransactionApprover`**

The controller handler performs two independent, uncorrelated operations: [1](#0-0) 

Line 108 calls `getCreatorsTransaction(transactionId, user)`, which only verifies the authenticated user is the creator of the URL-supplied `transactionId`: [2](#0-1) 

It says nothing about whether approver `id` belongs to that transaction. Line 109 then calls `removeTransactionApprover(id)`, which fetches the approver by primary key and deletes it with no cross-check against the transaction in the URL: [3](#0-2) 

`removeNode` performs a recursive soft-delete of the entire approver subtree rooted at `id`, regardless of which transaction it belongs to: [4](#0-3) 

**The fix pattern already exists in the codebase.** `updateTransactionApprover` correctly resolves the root node and validates it belongs to the URL's `transactionId` before proceeding: [5](#0-4) 

This cross-check is entirely absent from the delete path.

## Impact Explanation
Approval records gate whether a transaction advances from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION`. Removing an approver node eliminates that gating requirement. An attacker can silently strip approval requirements from any in-progress transaction they do not own, allowing those transactions (account updates, transfers, key changes, file operations, etc.) to be submitted to the Hedera network without the intended human oversight. This is an unauthorized state change with direct operational impact on the integrity of the multi-signature workflow.

## Likelihood Explanation
- **Attacker precondition**: only a valid authenticated account. No admin or privileged role required.
- **Approver IDs**: auto-incremented integers, trivially enumerable by an attacker who creates their own approvers and observes the returned IDs.
- The attack is silent â€” the victim transaction's creator receives a status-update notification (via `emitTransactionStatusUpdate`) but no indication that an external party removed the approver.
- No rate limiting on the delete endpoint is visible in the codebase.

## Recommendation
In `removeTransactionApprover` (service), after fetching the approver, resolve its root node using `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId` (the value passed from the controller). This is exactly the pattern already used in `updateTransactionApprover` at lines 386â€“391. The controller should pass `transactionId` to the service method so the check can be performed before deletion.

## Proof of Concept
1. Attacker registers as a normal user and creates any transaction, becoming its creator (`transactionId = A`).
2. Attacker observes auto-incremented approver IDs from their own transactions to enumerate IDs belonging to victim transaction B (`approver id = X`).
3. Attacker calls `DELETE /transactions/A/approvers/X`.
4. `getCreatorsTransaction(A, attacker)` succeeds â€” attacker owns A.
5. `removeTransactionApprover(X)` soft-deletes approver X (and its entire subtree) from transaction B with no further check.
6. Transaction B's approval tree is reduced or eliminated; `processTransactionStatus` may now advance B to `WAITING_FOR_EXECUTION` without the required approvals.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
