Audit Report

## Title
Missing Authorization Validation in `createComment` Allows Any Authenticated User to Comment on Arbitrary Transactions

## Summary
`CommentsService.createComment` performs a database write using a caller-supplied `transactionId` without verifying that the requesting user has any relationship to the target transaction. Any authenticated, verified user can inject comments into transactions they did not create, observe, sign, or approve.

## Finding Description
The root cause is in `back-end/apps/api/src/transactions/comments/comments.service.ts`, `createComment` (lines 15–23):

```typescript
async createComment(
  user: User,
  transactionId: number,
  dto: CreateCommentDto,
): Promise<TransactionComment> {
  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);
}
``` [1](#0-0) 

There are two missing checks:

**1. No transaction existence check.** The function never queries the database to confirm a `Transaction` with `id = transactionId` exists. The `TransactionComment` entity declares a `@ManyToOne` FK to `Transaction`, so a save against a non-existent ID will produce an unhandled DB-level FK violation, leaking internal error details to the caller. [2](#0-1) 

**2. No authorization check.** The function never verifies that `user` is the creator, observer, signer, or approver of the target transaction. Every other analogous write operation enforces this. For example, `ObserversService.createTransactionObservers` explicitly checks existence and creator identity:

```typescript
const transaction = await this.entityManager.findOne(Transaction, {
  where: { id: transactionId }, ...
});
if (!transaction) throw new BadRequestException(ErrorCodes.TNF);
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException(...);
``` [3](#0-2) 

`TransactionsService.getTransactionWithVerifiedAccess` similarly enforces a full role check (creator, observer, signer, approver) before any access: [4](#0-3) 

Notably, the controller itself contains a developer TODO explicitly acknowledging this gap:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(...)
``` [5](#0-4) 

The only guards applied at the controller level are `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — none of which check transaction-level authorization. [6](#0-5) 

## Impact Explanation
- **Unauthorized state mutation**: Any authenticated, verified user can inject comments into any existing transaction in the system, regardless of whether they are the creator, observer, signer, or approver. This corrupts the comment history of arbitrary transactions.
- **Data integrity**: The `transactionId` FK constraint means saves against non-existent IDs produce unhandled DB errors that expose internal error details to the caller.
- **Confidentiality side-channel**: By observing whether the save succeeds (200) or fails (500 FK error), an attacker can enumerate valid transaction IDs. Note: per `SECURITY.md`, pure enumeration of existence is listed as out of scope, but the primary impact here is the unauthorized write, which is in scope.

## Likelihood Explanation
- **Attacker precondition**: A valid JWT for any registered, verified user — no elevated privilege required.
- **Attack surface**: The endpoint `POST /transactions/:transactionId/comments` is reachable via standard HTTP.
- **ID predictability**: `transactionId` is a sequential integer, trivially enumerable from the attacker's own transactions.
- **No additional mitigations**: No rate limiting, no transaction-scoped guard, and no service-level check beyond authentication.

## Recommendation
In `CommentsService.createComment`, before writing:

1. Query the `Transaction` entity by `transactionId` and throw `BadRequestException(ErrorCodes.TNF)` if not found.
2. Call the existing `TransactionsService.verifyAccess(transaction, user)` (or equivalent) and throw `UnauthorizedException` if the user is not the creator, observer, signer, or approver of that transaction.

This mirrors the pattern already established in `ObserversService.createTransactionObservers` and `TransactionsService.getTransactionWithVerifiedAccess`. [7](#0-6) 

## Proof of Concept

```
# Step 1: Authenticate as any registered, verified user (attacker)
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "token": "<attacker_jwt>" }

# Step 2: Obtain a victim's transactionId (e.g., from attacker's own transaction list,
#          sequential IDs make enumeration trivial)
# Assume victim's transactionId = 42

# Step 3: POST a comment to the victim's transaction
POST /transactions/42/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json
{ "message": "Injected comment by attacker" }

# Result: HTTP 201 Created
# The comment is now persisted in the database linked to transaction 42,
# which the attacker has no legitimate relationship to.
# No existence or authorization check is performed in createComment.
```

The `repo.save` call succeeds unconditionally for any valid `transactionId`, as confirmed by the service implementation at lines 20–23 of `comments.service.ts`. [8](#0-7)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
```
