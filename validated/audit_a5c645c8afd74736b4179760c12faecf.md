Audit Report

## Title
Cross-Transaction Approver Deletion: Any Transaction Creator Can Remove Approvers from Transactions They Do Not Own

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but then passes the approver `:id` directly to `removeTransactionApprover()` with no check that the approver belongs to that transaction. Any verified user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause — missing cross-ownership check in the delete path:**

In `approvers.controller.ts`, the handler performs two independent operations: [1](#0-0) 

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the transaction at `:transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver record identified by `:id`.

There is **no check** that the approver `id` belongs to `transactionId`. The service function `removeTransactionApprover` only verifies the approver exists, then unconditionally deletes it: [2](#0-1) 

The `transactionId` URL parameter is consumed only by the creator-check and is never used to scope the deletion.

**Contrast with `updateTransactionApprover`**, which correctly validates the root transaction before proceeding: [3](#0-2) 

The `removeTransactionApprover` path has no equivalent guard.

## Impact Explanation

An attacker can silently remove required approvers from any transaction they do not own. This directly undermines the multi-signature approval workflow: a transaction requiring N approvals can have its approvers stripped, potentially allowing it to proceed with fewer approvals than intended, or disrupting the workflow of other organizations' transactions. This constitutes unauthorized state mutation of another user's transaction data.

**Impact: High** — approval integrity is a core security invariant of the system.

## Likelihood Explanation

The attacker only needs to be a verified user and the creator of any one transaction. Both conditions are reachable by any normal registered user with no privileged access. Approver `id` values are sequential integers, trivially enumerable. No leaked secrets or admin keys are required.

**Likelihood: Medium** — requires a valid account and one self-owned transaction, both achievable by a normal user.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), add a root-transaction ownership check identical to the one already present in `updateTransactionApprover`. Specifically, after fetching the approver, resolve its root node via `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId`. If the assertion fails, throw an `UnauthorizedException`. This mirrors the existing guard at line 390 of `approvers.service.ts`. [2](#0-1) 

## Proof of Concept

1. Attacker registers and is verified (`UserStatus.NONE`).
2. Attacker creates their own transaction → becomes creator of `transactionId=1`.
3. Victim has a separate transaction `transactionId=2` with a required approver record `id=99`.
4. Attacker enumerates approver IDs (sequential integers) and identifies `id=99`.
5. Attacker sends:
   ```
   DELETE /transactions/1/approvers/99
   Authorization: Bearer <attacker_jwt>
   ```
6. `getCreatorsTransaction(1, attacker)` passes — attacker owns tx 1.
7. `removeTransactionApprover(99)` executes — approver 99 (belonging to tx 2) is soft-deleted with no further check.
8. Victim's transaction `transactionId=2` now has its required approver removed, bypassing the multi-signature approval workflow.
