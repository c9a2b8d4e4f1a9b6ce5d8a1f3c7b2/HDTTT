Audit Report

## Title
Creator Can Archive a Manual Transaction From Any Status, Including Terminal States

## Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a flawed guard condition. Because the status-range check and the `isManual` flag are combined with a logical `&&`, setting `isManual: true` on a transaction completely bypasses the status check. Any authenticated user who is the creator of a manual transaction can call `PATCH /transactions/archive/:id` and force the status to `ARCHIVED` regardless of the transaction's current state — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED`.

## Finding Description

**Root cause — `archiveTransaction`, lines 711–718:**

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← entire guard is short-circuited when isManual = true
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard is supposed to reject archive requests when the transaction is not in an archiveable state. However, the condition only throws when **both** sub-expressions are true:

| `status` not in allowed list | `!isManual` | throws? |
|---|---|---|
| true | true (non-manual) | ✓ |
| true | false (manual) | **✗ — bypassed** |
| false | true | ✗ |
| false | false | ✗ |

When `isManual` is `true`, `!transaction.isManual` evaluates to `false`, so the entire `&&` expression is `false` and the guard never fires. The unconditional update on line 720 then runs:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

The `TransactionStatus` enum defines the terminal states that can be illegitimately overwritten: [3](#0-2) 

The `getTransactionForCreator` helper performs no status check — it only verifies the caller is the creator: [4](#0-3) 

The existing unit test suite only covers a non-manual `CANCELED` transaction and a manual `WAITING_FOR_EXECUTION` transaction — the manual+terminal combination is entirely untested: [5](#0-4) 

The e2e test at lines 884–897 similarly only tests a non-manual `EXECUTED` transaction: [6](#0-5) 

## Impact Explanation

- **Audit-trail corruption:** Terminal states (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) represent the definitive outcome of a Hedera network transaction. Overwriting them with `ARCHIVED` permanently hides the true outcome from all other participants (signers, observers, approvers) who rely on the stored status. The codebase itself treats these as the canonical "HISTORY" states: [7](#0-6) 
- **Integrity failure in multi-signature workflows:** Signers and observers depend on accurate status to understand whether a transaction succeeded or failed. A creator can conceal a failed or expired transaction by archiving it, undermining the trust model of the multi-party signing workflow.
- **Irreversible in normal operation:** There is no API endpoint to un-archive a transaction; the corruption persists unless an administrator directly modifies the database.

## Likelihood Explanation

- **Preconditions:** The attacker must be an authenticated, verified user and the creator of a manual transaction. Both conditions are reachable by any normal user — `isManual: true` is a standard field in the create payload. [8](#0-7) 
- **No special knowledge required:** The flaw is triggered by a single `PATCH` request to a documented endpoint after a transaction reaches a terminal state.
- **Realistic scenario:** A user whose transaction failed or expired could archive it to hide the failure from co-signers or auditors.

## Recommendation

Replace the `&&` with an `||` (or restructure the guard) so that `isManual` does not bypass the status check entirely. The corrected intent should be: throw if the status is not in the allowed list, **regardless** of whether the transaction is manual. If manual transactions are intended to be archiveable from additional statuses (e.g., `WAITING_FOR_EXECUTION`), those statuses should be explicitly added to the allowed list rather than using `isManual` as a blanket bypass:

```typescript
// Option A: remove isManual bypass entirely
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}

// Option B: if manual transactions have a broader set of archiveable statuses, enumerate them explicitly
const archiveableStatuses = transaction.isManual
  ? [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION]
  : [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION];

if (!archiveableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

## Proof of Concept

1. Authenticate as any verified user (User A).
2. Create a transaction with `isManual: true` via `POST /transactions`.
3. Allow the transaction to reach `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED` status (or set it directly in a test environment).
4. As User A (the creator), call `PATCH /transactions/archive/<id>`.
5. Observe HTTP 200 response and confirm in the database that `status` is now `ARCHIVED`, overwriting the terminal state.

The guard at lines 711–718 evaluates `(true) && (false)` → `false` when `isManual = true`, so no exception is thrown and the `repo.update` at line 720 executes unconditionally. [9](#0-8)
