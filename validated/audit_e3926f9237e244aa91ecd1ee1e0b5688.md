### Title
Missing Whitelist Validation on `mirrorNetwork` Parameter Enables Server-Side Request Forgery (SSRF)

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any arbitrary string without restricting it to an allowed set of known networks. When a non-standard value is supplied, the server constructs a gRPC client targeting the attacker-controlled host and executes an `AddressBookQuery` against it, causing the server to make outbound network connections to arbitrary infrastructure. This is the direct analog of the external report's missing whitelist check on a user-supplied parameter.

### Finding Description

**Root cause — no whitelist on `mirrorNetwork`:**

`CreateTransactionDto` validates `mirrorNetwork` only as a non-empty string: [1](#0-0) 

No enum constraint, no allowed-values check, no URL validation.

**Code path — user value flows directly into outbound gRPC call:**

`createTransactions()` immediately passes the raw user-supplied value to `getClientFromNetwork()`: [2](#0-1) 

Inside `getClientFromNetwork()`, for any value that is not `mainnet`, `testnet`, `previewnet`, or `local-node`, the server builds a gRPC client pointing at the user-supplied host and executes a live `AddressBookQuery` against it: [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` converts the raw string into a gRPC endpoint with no sanitisation beyond appending `:443`: [4](#0-3) 

**Entry point — authenticated but unprivileged:**

The endpoint is guarded only by JWT authentication and email verification — no admin role required: [5](#0-4) 

### Impact Explanation

Any authenticated normal user can force the backend server to open an outbound gRPC TCP connection to an arbitrary host on port 443 (or any port the attacker embeds in the URL string, e.g., `"internal-svc:8443"`). Concrete consequences:

- **Internal network probing**: the attacker supplies RFC-1918 addresses or internal Kubernetes DNS names (e.g., `"10.0.0.5"`, `"postgres.default.svc.cluster.local"`) to map reachable internal services.
- **Cloud metadata exfiltration**: on AWS/GCP/Azure, `"169.254.169.254:443"` or equivalent metadata endpoints may be reachable.
- **Server IP disclosure**: the attacker's server observes the originating IP of the backend.
- **Persistent SSRF**: the `mirrorNetwork` value is stored in the database and later consumed by the chain service when executing the transaction, potentially triggering a second outbound connection.

### Likelihood Explanation

- Attacker precondition: a valid authenticated user account. Registration requires admin invitation, but a malicious insider or a user whose account was obtained through normal means satisfies this.
- The `RESEARCHER.md` explicitly lists "Malicious normal user abusing valid product/protocol flows" as an in-scope attacker profile.
- The attack requires a single crafted `POST /transactions` request — no special tooling, no race condition, no timing dependency.
- Custom mirror networks are a documented, supported feature of the application (the front-end exposes a "Custom" network option), so the code path is exercised in production.

### Recommendation

Validate `mirrorNetwork` against an explicit allowlist before passing it to `getClientFromNetwork()`. Add an `@IsEnum` or custom `@IsIn([...])` decorator to `CreateTransactionDto.mirrorNetwork`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction.dto.ts
@IsNotEmpty()
@IsString()
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

If custom mirror node URLs must be supported, apply strict URL validation (scheme, hostname format, no private IP ranges) before the value is used to construct a network client.

### Proof of Concept

1. Register or obtain a normal (non-admin) user account.
2. Obtain a JWT token via `POST /auth/login`.
3. Start a listener on an attacker-controlled server at `attacker.example.com:443`.
4. Submit the following request:

```http
POST /transactions HTTP/1.1
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "name": "ssrf-test",
  "description": "ssrf",
  "transactionBytes": "<valid hex-encoded SDK transaction bytes>",
  "creatorKeyId": <valid key id>,
  "signature": "<valid signature>",
  "mirrorNetwork": "attacker.example.com"
}
```

5. Observe an inbound gRPC connection from the backend server's IP to `attacker.example.com:443` — confirming the server made an outbound connection to the attacker-controlled host.

Substitute `"attacker.example.com"` with an internal IP (e.g., `"10.96.0.1"`) to probe Kubernetes cluster-internal services.
