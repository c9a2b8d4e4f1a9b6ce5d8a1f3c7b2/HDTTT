Audit Report

## Title
Any Authenticated User Can Post Comments on Any Transaction Due to Missing Participant Access Control

## Summary
The `createComment`, `getComments`, and `getCommentById` endpoints in `CommentsController` perform no participant relationship check between the requesting user and the target transaction. Any authenticated, verified user can post comments on — and read comments from — any transaction in the system. The codebase explicitly acknowledges this gap with a developer TODO comment.

## Finding Description
In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the controller applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level. [1](#0-0) 

The `POST /` handler carries an explicit developer acknowledgment that the access check is missing: [2](#0-1) 

`CommentsService.createComment` performs no participant validation — it directly creates and persists the comment: [3](#0-2) 

The `GET /` and `GET /:id` handlers similarly pass no user context to the service, making a participant check structurally impossible at the service layer: [4](#0-3) 

By contrast, the rest of the transaction subsystem enforces participant checks. `removeTransactionApprover` calls `getCreatorsTransaction` before proceeding: [5](#0-4) 

`updateTransactionObserver` and `removeTransactionObserver` call `getUpdateableObserver`, which enforces creator-only access: [6](#0-5) 

The system has a fully implemented `verifyAccess` method in `TransactionsService` that checks creator, signer, observer, and approver relationships — but it is never invoked by the comments subsystem: [7](#0-6) 

## Impact Explanation
1. **Unauthorized write**: Any authenticated user can post comments on any transaction they have no business relationship with, polluting the audit trail and disrupting legitimate participant workflows.
2. **Unauthorized read**: Any authenticated user can enumerate and read all comments on any transaction via `GET /transactions/:transactionId/comments`, potentially exposing sensitive operational details (signing instructions, key coordination notes, dispute context) scoped to specific participants.

## Likelihood Explanation
Exploitation requires only a valid organization account and a known transaction ID. Transaction IDs are sequential integers, making enumeration trivial. No elevated privileges, leaked credentials, or complex setup are required — only a standard authenticated HTTP request.

## Recommendation
Apply a participant check in `CommentsService` before allowing comment creation or retrieval. The existing `TransactionsService.getTransactionWithVerifiedAccess` method already implements the correct creator/signer/observer/approver check and should be called at the start of `createComment`, `getTransactionComments`, and `getTransactionCommentById`. The `getComments` and `getCommentById` controller handlers must also be updated to pass the authenticated `user` to the service so the check can be enforced.

## Proof of Concept
```
# Attacker has a valid JWT for user B, who has no relationship to transaction ID 5
# (created by user A)

# Unauthorized comment write
POST /transactions/5/comments
Authorization: Bearer <user_B_jwt>
Content-Type: application/json
{"message": "Attacker comment"}
# Returns 201 Created — comment is persisted

# Unauthorized comment read
GET /transactions/5/comments
Authorization: Bearer <user_B_jwt>
# Returns 200 with all comments on transaction 5
```
Transaction IDs can be enumerated by looping from `1` upward. No other precondition is needed.

```typescript
  private async getUpdateableObserver(id: number, user: User): Promise<TransactionObserver> {
    const observer = await this.repo.findOneBy({ id });

    if (!observer) throw new BadRequestException(ErrorCodes.ONF);

    const transaction = await this.entityManager.findOne(Transaction, {
      where: { id: observer.transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    });

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to update it');

    return observer;
```

```typescript
  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
```
