Audit Report

## Title
Single User Approval Satisfies Multiple Threshold Positions via Duplicate Approver Entries

## Summary
A transaction creator can insert the same `userId` under multiple distinct threshold-group nodes (different `listId` values). Because `approveTransaction` bulk-updates **every** `TransactionApprover` record belonging to the calling user in a single query, one approval call simultaneously satisfies all positions that user occupies — collapsing a multi-party threshold into a single-party approval.

## Finding Description

**Root cause 1 — `isNode` does not prevent cross-group duplicates**

`isNode` checks for an existing record matching `(listId, userId, transactionId)`: [1](#0-0) 

Because `listId` is part of the uniqueness key, inserting the same `userId` under two different parent nodes (different `listId` values) produces two records that each pass the check independently. The guard at line 250 therefore does not fire: [2](#0-1) 

**Root cause 2 — `approveTransaction` bulk-updates all positions for the user**

`getVerifiedApproversByTransactionId` calls `getApproversByTransactionId`, which uses a recursive CTE that returns every node in the entire approver tree: [3](#0-2) 

`approveTransaction` then filters that flat list to every record whose `userId` matches the caller, and updates them all in one query: [4](#0-3) 

The "already approved" guard at line 563 only throws if **every** record already has a signature — it does not prevent the first call from stamping multiple records simultaneously: [5](#0-4) 

## Impact Explanation
A malicious transaction creator (no admin role required) can craft an approver tree such as:

```
Root threshold: 2-of-2
  ├── Group A (threshold 1-of-1) → userId: Alice
  └── Group B (threshold 1-of-1) → userId: Alice   ← same user, different listId
```

Both leaf insertions pass `isNode` because they carry different `listId` values. When Alice calls `POST /transactions/:id/approvers/approve` once, `userApprovers` contains both records and `.whereInIds(userApprovers.map(a => a.id))` stamps both with her signature. Group A and Group B are each satisfied (1-of-1), so the root 2-of-2 threshold is met — with only Alice's single approval. Any transaction, including high-value transfers or administrative actions, can be approved by one colluding user, defeating the multi-party approval requirement entirely.

## Likelihood Explanation
The attacker must be a legitimate user with transaction-creation rights; no elevated privileges are needed. The `createTransactionApprovers` endpoint is available to any authenticated creator. The attack requires only crafting a specific JSON payload for the approver tree and then calling the approve endpoint once. No cryptographic weakness, race condition, or special timing is involved.

## Recommendation

1. **Enforce per-transaction `userId` uniqueness across all groups.** Before inserting a leaf node, query whether the `userId` already exists anywhere in the transaction's approver tree (not just under the same `listId`). A query against the recursive CTE in `getApproversByTransactionId` with a `userId` filter is sufficient.

2. **Limit `approveTransaction` to one record per call.** Instead of updating all `userApprovers` records at once, require the caller to specify which approver record they are acting on (e.g., pass the `approverId` in the request), and update only that single record. This prevents one signature from satisfying multiple positions.

3. **Add a unique constraint at the database level** on `(transactionId-root, userId)` across the full tree, or enforce it in the service layer as described above.

## Proof of Concept

```
# Step 1 – Creator builds the approver tree
POST /transactions/42/approvers
{
  "approversArray": [
    {
      "threshold": 2,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [{ "userId": <Alice_id> }]   // Group A, listId = X
        },
        {
          "threshold": 1,
          "approvers": [{ "userId": <Alice_id> }]   // Group B, listId = Y — passes isNode
        }
      ]
    }
  ]
}

# Step 2 – Alice approves once
POST /transactions/42/approvers/approve
{
  "userKeyId": <Alice_key_id>,
  "signature": "<valid_signature>",
  "approved": true
}

# Result: both Group A → Alice and Group B → Alice records are stamped.
# Root 2-of-2 threshold is satisfied. Transaction proceeds with only Alice's approval.
```

The `whereInIds` call at line 608 of `approvers.service.ts` updates both records in a single database statement, confirming the exploit requires exactly one API call after tree creation. [6](#0-5)
