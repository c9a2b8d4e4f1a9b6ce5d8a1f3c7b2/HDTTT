Audit Report

## Title
Unbounded O(n) Full-Table Scan in `getTransactionsToSign` Enables DoS via Transaction Flooding

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction in the system with no database-level row limit, then performs an expensive per-transaction computation (SDK deserialization + mirror-node account resolution + DB query) for each row before applying pagination. Any verified user can flood the `transaction` table with non-terminal transactions via `POST /transactions`, causing `GET /transactions/sign` to time out or exhaust server resources for all users.

## Finding Description

**Root cause — unbounded fetch with no `take` limit**

`getTransactionsToSign` issues a `repo.find()` with no `take` parameter: [1](#0-0) 

The `whereForUser` predicate excludes only five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). Notably, `REJECTED` is also a terminal status (present in `this.terminalStatuses`) but is **not** excluded here, widening the scan further. Critically, the predicate does **not** scope the query to the requesting user's transactions — it returns every matching transaction in the entire system: [2](#0-1) 

Compare this to `getTransactions`, which correctly applies `skip`/`take` at the database level: [3](#0-2) 

**O(n) per-transaction computation**

For every row returned, the service calls `userKeysToSign`, which delegates to `userKeysRequiredToSign` → `keysRequiredToSign`: [4](#0-3) 

Inside `keysRequiredToSign`, each iteration performs:
1. `SDKTransaction.fromBytes` — full deserialization of transaction bytes
2. `transactionSignatureService.computeSignatureKey` — which calls `accountCacheService.getAccountInfoForTransaction` (a DB lookup per account, and a mirror-node HTTP call on cache miss)
3. An `entityManager.find(UserKey, ...)` DB query [5](#0-4) 

The `AccountCacheService` does maintain a DB-backed cache with a configurable TTL (default 10 s), so repeated calls for the same fee-payer account may be served from cache: [6](#0-5) 

However, SDK deserialization and the `UserKey` DB query still execute unconditionally for every transaction row, regardless of caching.

**Pagination applied only after full scan** [7](#0-6) 

**No rate limiting on `POST /transactions`**

The only global guard on `POST /transactions` is an IP-based throttler applied at the application level: [8](#0-7) 

There is no `@Throttle` decorator on the `createTransaction` endpoint, no per-user transaction count cap, and no `HasKeyGuard` restriction beyond requiring at least one registered key: [9](#0-8) 

**Exposed endpoint**

`GET /transactions/sign` is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin role required. The comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is live but abandoned, meaning it receives no front-end-side mitigations: [10](#0-9) 

## Impact Explanation

An attacker who creates N non-terminal transactions causes every subsequent call to `GET /transactions/sign` by any user to:
- Load N rows from PostgreSQL (with `groupItem` relation)
- Perform N SDK deserializations (`SDKTransaction.fromBytes`)
- Issue up to N mirror-node HTTP calls (or N DB cache lookups)
- Issue N `UserKey` DB queries

At a few thousand transactions the endpoint will exceed typical HTTP timeout thresholds, rendering the "transactions to sign" workflow completely unavailable for all users. Server memory and connection-pool exhaustion are secondary effects. Because the endpoint is no longer used by the front-end, the degradation may go undetected until it affects other server resources.

## Likelihood Explanation

- **Attacker preconditions**: a valid account + at least one registered `UserKey` — both achievable through the normal self-registration flow.
- **Cost**: issuing N `POST /transactions` HTTP requests with valid (but otherwise arbitrary) `transactionBytes`. No privileged access required.
- **Persistence**: transactions remain in `WAITING_FOR_SIGNATURES` indefinitely if the required signers never sign; the attacker does not need to remain active after the initial flood.
- **Partial mitigations**: the IP throttler limits request rate from a single IP, but does not cap the total number of transactions a user may create. The account-info cache reduces mirror-node calls for repeated fee-payer accounts but does not eliminate the per-row SDK deserialization or `UserKey` DB queries.

## Recommendation

1. **Add a `take` limit to the database query** in `getTransactionsToSign` so the DB never returns more rows than the requested page size. Compute `totalItems` with a separate `count()` query scoped to the user.
2. **Scope `whereForUser` to the requesting user** (e.g., filter by `creatorKey.userId`, `signers.userId`, or `observers.userId`) to eliminate cross-user data leakage and reduce scan size.
3. **Add `REJECTED` to the excluded statuses** in `whereForUser` to match the `terminalStatuses` definition.
4. **Add a per-user transaction creation cap** or a `@Throttle` decorator on `POST /transactions` to limit how many transactions a single user can create within a time window.
5. **Consider removing or formally deprecating** the `GET /transactions/sign` endpoint if it is no longer used by the front-end.

## Proof of Concept

```
# 1. Register a user account and obtain a JWT token (normal self-registration flow)
# 2. Register at least one UserKey for the account
# 3. Flood the transaction table:
for i in $(seq 1 5000); do
  curl -s -X POST https://<host>/transactions \
    -H "Authorization: Bearer <jwt>" \
    -H "Content-Type: application/json" \
    -d '{"transactionBytes":"<valid_bytes>","creatorKeyId":<keyId>,"signature":"<sig>","mirrorNetwork":"testnet","name":"flood","isManual":false}' &
done
wait

# 4. Now any user calling GET /transactions/sign will trigger a full scan of 5000+ rows,
#    each requiring SDK deserialization + mirror-node resolution + DB query.
curl -v -H "Authorization: Bearer <any_valid_jwt>" \
  "https://<host>/transactions/sign?page=1&limit=10"
# Expected: request times out or server returns 5xx due to resource exhaustion
```
