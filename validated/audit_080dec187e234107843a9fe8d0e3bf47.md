Audit Report

## Title
Weak Password Policy Allows Trivially Guessable Passwords in Organization Backend and Local Mode

## Summary
The organization backend's `ChangePasswordDto` and `NewPasswordDto` use `@IsStrongPassword` with all complexity requirements explicitly zeroed out, accepting any 8-character string as a valid password. The local-mode `changePassword` function in the Electron main process performs no password strength validation whatsoever. Because the password also serves as the decryption key for stored Hedera private keys, a weak password directly threatens key material.

## Impact Explanation

The password serves a dual purpose:
1. It authenticates the user at login.
2. It is passed directly to `changeDecryptionPassword` and used as the decryption key for stored Hedera private key pairs. [3](#0-2) [10](#0-9) 

- **Organization mode**: An attacker who can reach the organization API can brute-force `POST /auth/login` with a small dictionary. Upon success, they obtain a JWT and can sign or approve Hedera transactions on behalf of the victim.
- **Local mode**: If the encrypted SQLite database file is exfiltrated, the weak password makes offline brute-force of the encrypted private keys trivial, leading to full private key compromise and unauthorized Hedera transaction signing.

## Likelihood Explanation

- **Organization mode**: Any authenticated user can set a weak password through the normal UI or by directly calling `PATCH /auth/change-password`. No privilege escalation is required. The endpoint is reachable by any valid organization member.
- **Local mode**: The lack of any server-side (main process) strength check means a user can set an arbitrarily weak password through the normal UI flow (the UI only enforces 10 characters, no complexity). The IPC bypass additionally allows setting a password shorter than 10 characters from the renderer context.

## Recommendation

**Organization backend**: Replace the zeroed-out `@IsStrongPassword` options with meaningful complexity requirements (e.g., `minLowercase: 1, minUppercase: 1, minNumbers: 1, minSymbols: 1, minLength: 12`) in both `ChangePasswordDto` and `NewPasswordDto`.

**Local mode**: Add a password strength check inside the `changePassword` function in `front-end/src/main/services/localUser/auth.ts` before calling `updatePassword`, mirroring (or exceeding) whatever policy is enforced in the UI. Similarly, add the same check to the `register` function. This ensures the main process enforces the policy regardless of how the IPC channel is invoked.

**Front-end**: Align `isPasswordStrong` in `front-end/src/renderer/utils/validator.ts` with the backend policy by adding complexity requirements (uppercase, lowercase, number, symbol) in addition to the length check.

## Proof of Concept

**Organization backend:**
```
PATCH /auth/change-password
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{ "oldPassword": "<current>", "newPassword": "aaaaaaaa" }
```
The DTO validation passes because `minLowercase: 0`, `minNumbers: 0`, `minSymbols: 0`, `minUppercase: 0` are all zero. The response is `200 OK` and the password is updated to `aaaaaaaa`.

**Local mode (IPC bypass from renderer/DevTools console):**
```js
// Obtain userId from app state, then:
await window.electronAPI.local.localUser.changePassword(
  userId,
  'currentCorrectPassword',
  'aaa'   // 3 characters, no complexity â€” accepted by main process
);
```
The main process `changePassword` function verifies only that `oldPassword` is correct, then calls `updatePassword(userId, 'aaa')` and `changeDecryptionPassword(userId, 'currentCorrectPassword', 'aaa')`, re-encrypting all stored private keys under the trivially weak new password.

export function isPasswordStrong(password: string): {
  length: boolean;
  result: boolean;
} {
  const validationRegex = [
    /.{10,}/, // min 10 letters
  ];

  return {
    length: validationRegex[0].test(password),
    result: validationRegex.every(regex => regex.test(password)),
  };
}

export const changeDecryptionPassword = async (
  userId: string,
  oldPassword: string,
  newPassword: string,
) => {
  const prisma = getPrismaClient();

  const keyPairs = await getKeyPairs(userId);

  for (let i = 0; i < keyPairs.length; i++) {
    const keyPair = keyPairs[i];
    const decryptedPrivateKey = decrypt(keyPair.private_key, oldPassword);
    const encryptedPrivateKey = encrypt(decryptedPrivateKey, newPassword);

    await prisma.keyPair.update({
      where: {
        id: keyPair.id,
        public_key: keyPair.public_key,
      },
      data: {
        private_key: encryptedPrivateKey,
      },
    });
  }

  return await getKeyPairs(userId);
};
