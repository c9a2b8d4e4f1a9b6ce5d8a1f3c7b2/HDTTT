Audit Report

## Title
Missing Transaction Ownership Cross-Check in `removeTransactionApprover` Allows Deletion of Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns the transaction at `:transactionId`, but never verifies that the approver at `:id` belongs to that same transaction. An authenticated user who owns any transaction can delete approvers from transactions they do not own.

## Finding Description
In `approvers.controller.ts`, the delete handler performs two sequential calls:

```typescript
// approvers.controller.ts lines 102–113
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

`getCreatorsTransaction` confirms the caller created the transaction at `:transactionId`. However, `removeTransactionApprover(id)` then deletes the approver row by `:id` alone:

```typescript
// approvers.service.ts lines 534–544
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(..., [{ entityId: approver.transactionId }]);
    return result;
}
``` [2](#0-1) 

No check is made that `approver.transactionId === transactionId`. The `transactionId` URL parameter is consumed only by the ownership check on the attacker's own transaction and is then discarded.

By contrast, `updateTransactionApprover` in the same service **correctly** performs this cross-check:

```typescript
// approvers.service.ts lines 386–394
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The delete path is missing the equivalent guard that the update path has. The `ROOT_TRANSACTION_NOT_SAME` error message and `getRootNodeFromNode` helper already exist in the service and are simply not called from the delete path.

## Impact Explanation
An attacker who is the creator of any transaction (even a trivial one they created themselves) can delete approvers from any other transaction in the system. Removing required approvers can:
- Cause a transaction to bypass its intended approval threshold.
- Silently eliminate a required signer from the approval tree, allowing a transaction to proceed without the intended authorization.
- Disrupt the approval workflow of victim transactions without the victim being able to prevent it.

Note that line 541 (`emitTransactionStatusUpdate(..., [{ entityId: approver.transactionId }])`) uses the approver's *actual* `transactionId` (the victim's), not the URL parameter — confirming the notification side-effect targets the victim transaction, not the attacker's. [4](#0-3) 

## Likelihood Explanation
Any registered, verified user can create a transaction, satisfying the ownership check on their own transaction. Approver IDs are sequential integers (auto-increment primary keys), making enumeration trivial. No special privilege is required beyond a valid account. The attack requires only two pieces of information: the attacker's own `transactionId` and a target `approverId` from another transaction.

## Recommendation
Apply the same cross-check that `updateTransactionApprover` already uses. In `removeTransactionApprover`, after fetching the approver, resolve its root node and assert that `rootNode.transactionId === transactionId` before calling `removeNode`:

```typescript
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const rootNode = await this.getRootNodeFromNode(approver.id);
    if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
    if (rootNode.transactionId !== transactionId)
        throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

    await this.getCreatorsTransaction(rootNode.transactionId, user);

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: rootNode.transactionId }]);
    return result;
}
```

The controller must be updated to pass `transactionId` and `user` to the service method, mirroring the pattern already used by `updateTransactionApprover`. [5](#0-4) 

## Proof of Concept

1. Attacker (user A) creates their own transaction → receives `transactionId = 1`.
2. Attacker discovers (via enumeration or the GET endpoint) that transaction owned by user B has an approver with `id = 42`.
3. Attacker sends:
   ```
   DELETE /transactions/1/approvers/42
   Authorization: Bearer <attacker_token>
   ```
4. Server calls `getCreatorsTransaction(1, userA)` → passes (attacker owns transaction 1).
5. Server calls `removeTransactionApprover(42)` → fetches approver 42 (belongs to transaction 2, owned by user B), calls `removeNode(42)`, soft-deletes it. No cross-check is performed.
6. Approver 42 is deleted from user B's transaction. User B's approval workflow is disrupted.
