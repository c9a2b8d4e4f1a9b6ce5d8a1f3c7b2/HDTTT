### Title
Persistent JWT Sessions After Password Change Allow Continued Unauthorized Account Access

### Summary
The `changePassword` endpoint in `AuthController` updates the user's password hash but does not invalidate existing JWT access tokens. An attacker who has obtained a victim's JWT retains full authenticated access to the victim's account for up to the full token lifetime (365 days in the default configuration) even after the victim changes their password. The `BlacklistService` infrastructure exists and is correctly used in `logout` and OTP flows, but is never invoked in the `changePassword` path.

### Finding Description

**Root cause — `changePassword` never blacklists the current session token.**

`auth.controller.ts` line 115–119 handles the password-change request:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto): Promise<void> {
    return this.authService.changePassword(user, dto);
}
``` [1](#0-0) 

The controller does not pass `req` to the service and does not call `blacklistService.blacklistToken(...)`. Compare this with the `logout` handler, which correctly blacklists the token:

```typescript
async logout(@Req() req: Request) {
    await this.blacklistService.blacklistToken(extractJwtAuth(req));
}
``` [2](#0-1) 

`auth.service.ts` `changePassword` only updates the password hash and emits a notification — no session invalidation occurs: [3](#0-2) 

The `BlacklistService.blacklistToken` method stores the raw JWT string in Redis with a TTL equal to `JWT_EXPIRATION` days. It is called in `logout` and in both OTP-flow endpoints (`verifyOtp`, `setPassword`) but is absent from `changePassword`: [4](#0-3) 

The default `JWT_EXPIRATION` is 365 days: [5](#0-4) 

**Exploit flow:**
1. Attacker obtains victim's JWT (e.g., via XSS on the Electron renderer, network interception, or a compromised shared device).
2. Victim detects suspicious activity and changes their password via `PATCH /auth/change-password` with their old and new passwords.
3. The password hash in the database is updated; no token is blacklisted.
4. Attacker's captured JWT passes `JwtBlackListAuthGuard` (not blacklisted) and `JwtAuthGuard` (not expired) — full access is retained.
5. Attacker continues to read, create, sign, and approve transactions on behalf of the victim for up to 365 days.

The same gap exists in the `setPassword` (post-OTP-reset) path: the OTP JWT is blacklisted, but any pre-existing auth JWTs for the account are not: [6](#0-5) 

### Impact Explanation
An attacker with a stolen JWT retains full authenticated access to the victim's organization account — including viewing sensitive transaction data, submitting signatures, approving transactions, and managing keys — even after the victim performs the expected remediation action (password change). The window of exposure is up to 365 days. The victim has no in-product mechanism to force-expire other sessions; only explicit `POST /auth/logout` from the attacker's session (which the attacker will not perform) would close it.

### Likelihood Explanation
JWT theft is a realistic precondition: the front-end is an Electron app whose renderer process handles JWTs and communicates with the back-end over HTTP. Any XSS in the renderer, a compromised shared machine, or a logged network session can yield a valid JWT. Once obtained, the attacker's continued access after a password change is guaranteed by the missing blacklist call — no further attacker action is required.

### Recommendation
In `AuthController.changePassword`, extract and blacklist the current request's JWT immediately after the password is successfully updated:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto, @Req() req: Request): Promise<void> {
    await this.authService.changePassword(user, dto);
    await this.blacklistService.blacklistToken(extractJwtAuth(req));
}
```

Apply the same fix to `setPassword` (post-OTP reset) for pre-existing auth tokens. For complete session revocation, consider storing a per-user `tokenVersion` in the database and embedding it in the JWT payload; incrementing it on password change invalidates all outstanding tokens without requiring individual blacklist entries.

### Proof of Concept

**Setup:** Two HTTP clients — Victim (V) and Attacker (A).

1. **V** logs in: `POST /auth/login` → receives `jwt_v1`.
2. **A** captures `jwt_v1` (assume prior theft).
3. **V** changes password: `PATCH /auth/change-password` with `Authorization: Bearer jwt_v1`, body `{oldPassword, newPassword}` → HTTP 200.
4. **A** calls `GET /transactions?page=1&size=10` with `Authorization: Bearer jwt_v1` → **HTTP 200**, full transaction list returned.
5. **A** calls `POST /transactions/{id}/signers` with `jwt_v1` → **HTTP 201**, signature accepted.

Expected (correct) behavior at step 4: HTTP 401 Unauthorized, because `jwt_v1` should have been blacklisted at step 3. [7](#0-6) [3](#0-2) [4](#0-3)

  async changePassword(user: User, { oldPassword, newPassword }: ChangePasswordDto): Promise<void> {
    if (oldPassword === newPassword) throw new BadRequestException(ErrorCodes.NPMOP);

    const { correct } = await this.dualCompareHash(oldPassword, user.password);
    if (!correct) throw new BadRequestException(ErrorCodes.INOP);

    if (user.status === UserStatus.NEW && user.keys.length === 0) {
      emitUserStatusUpdateNotifications(this.notificationsPublisher, { entityId: user.id, additionalData: { username: user.email } });
