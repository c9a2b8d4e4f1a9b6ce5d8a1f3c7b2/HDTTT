### Title
Transaction Approval Threshold Can Be Retroactively Lowered to Bypass Multi-Party Approval Requirements

### Summary

The `updateTransactionApprover` function in `approvers.service.ts` allows the transaction creator to reduce the `threshold` of an approver tree at any time, including after some (but not all) required approvers have already signed. Because no transaction-status guard exists on this mutation, a creator can retroactively satisfy the approval requirement with fewer approvals than were originally mandated — directly analogous to how `changeMaxGenerationId` in FootiumAcademy retroactively unlocked minting from older seasons.

### Finding Description

**Vulnerability class:** State-transition / retroactive validation bypass.

`updateTransactionApprover` is the entry point for modifying an existing approver node. The threshold-update branch (lines 467–488) performs only two checks before writing the new value to the database:

1. The approver must be a tree node (has a `threshold`, no `userId`).
2. The new threshold must be ≤ the current number of child approvers and ≠ 0. [1](#0-0) 

Neither check inspects the **current transaction status** nor whether any child approvers have already recorded a signature/approval. The only authorization gate is `getCreatorsTransaction`, which verifies the caller is the transaction creator — it does not restrict modifications based on lifecycle state. [2](#0-1) 

A parallel issue exists in `createTransactionApprovers`: when a user-approver is added and that user already has a signature record for the same transaction (e.g., from a previously soft-deleted approver entry), the existing `signature`, `userKeyId`, and `approved` values are silently copied into the new approver record. [3](#0-2) 

This behavior is confirmed by the test suite: [4](#0-3) 

### Impact Explanation

The approver system is the protocol's multi-party governance layer: a transaction creator designates a set of approvers (potentially with a threshold key-list tree) who must consent before the transaction advances to execution. If the creator can lower the threshold after partial approvals have been collected, the entire governance guarantee is void. Concretely:

- A creator sets threshold = N, designates N approvers.
- K < N approvers sign (meeting their individual obligations).
- Creator calls `PATCH /transactions/:id/approvers/:approverId` with `{ threshold: K }`.
- The approval requirement is now retroactively satisfied; the chain service recalculates status and the transaction proceeds to `WAITING_FOR_EXECUTION` or direct execution.

The secondary path (signature inheritance on re-add) allows the creator to remove a previously-approved approver, restructure the tree, re-add the same user, and have the new record inherit `approved: true` — again bypassing the original approval structure.

### Likelihood Explanation

The attack requires only that the caller be the transaction creator, which is a normal, non-privileged role. No external dependency, oracle, or admin key is needed. The API endpoint is reachable by any authenticated user who created a transaction. The window of exploitation is any time between the first approver signing and the transaction reaching a terminal status.

### Recommendation

1. **Guard threshold mutations on transaction status.** In `updateTransactionApprover`, before applying any structural change (threshold, listId, userId), verify that the transaction is still in `NEW` or `WAITING_FOR_SIGNATURES` status and that **no child approver has yet recorded a signature** (`approved IS NULL` for all descendants). Reject the mutation otherwise.

2. **Do not inherit signatures on approver re-creation.** In `createTransactionApprovers`, remove the block that copies `signature`, `userKeyId`, and `approved` from existing approver records. A freshly added approver should always start unsigned, regardless of prior history for that user on the same transaction.

3. **Freeze the approver tree once any approval is recorded.** Treat the first `approved = true` record as a lock on the tree structure, preventing any further structural mutations (threshold changes, node additions/removals) until the transaction reaches a terminal state.

### Proof of Concept

```
1. Creator (User C) creates Transaction T.
2. Creator calls POST /transactions/T/approvers with:
     { approversArray: [{ threshold: 2, approvers: [{ userId: A }, { userId: B }] }] }
   → Tree node (id=10, threshold=2) with children User A (id=11) and User B (id=12).

3. User A calls POST /transactions/T/approvers/approve with a valid signature.
   → approver id=11: approved=true, signature=<sigA>
   → approver id=12: approved=null  (User B has NOT signed)

4. Creator calls PATCH /transactions/T/approvers/10 with { threshold: 1 }.
   → updateTransactionApprover passes all checks (1 ≤ 2, node is a tree).
   → DB: threshold updated to 1.
   → emitTransactionUpdate fires → chain service re-evaluates approval state.

5. Chain service sees: threshold=1, approvals=1 (User A) → requirement satisfied.
   → Transaction advances to WAITING_FOR_EXECUTION / executes.
   → User B's approval was never obtained, defeating the 2-of-2 governance requirement.
``` [5](#0-4) [1](#0-0)

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
