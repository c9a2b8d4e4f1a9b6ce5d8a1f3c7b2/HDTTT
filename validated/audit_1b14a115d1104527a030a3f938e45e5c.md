### Title
Unchecked Execution Result in Sequential Transaction Group Allows Subsequent Transactions to Execute After Failure

### Summary
In `executeTransactionGroup`, the sequential execution loop calls `_executeTransaction` for each transaction and collects results, but never inspects the returned status. When a transaction in a sequential group fails on the Hedera network, the loop continues executing all remaining transactions unconditionally. This is the direct analog of the external report's "unchecked return value" pattern: the result of the execution call is collected but never acted upon to halt the sequence.

### Finding Description
**Root cause — `back-end/libs/common/src/execute/execute.service.ts` lines 86–91:**

```typescript
if (transactionGroup.sequential) {
  for (const { sdkTransaction, transaction } of transactions) {
    const delay = transaction.validStart.getTime() - Date.now();
    await sleep(delay);
    rawResults.push(await this._executeTransaction(transaction, sdkTransaction));
  }
}
```

`_executeTransaction` returns one of three values:
- `null` — duplicate or lost DB race (another pod won)
- `TransactionExecutedDto { status: FAILED, error: ... }` — Hedera rejected the transaction
- `TransactionExecutedDto { status: EXECUTED }` — success

The loop pushes every result into `rawResults` but **never reads it back**. There is no `break`, no early-return, and no conditional guard on the result before proceeding to the next iteration. A `FAILED` result is indistinguishable from a `EXECUTED` result in terms of loop control — both cause the next transaction to execute. [1](#0-0) 

`_executeTransaction` explicitly sets `transactionStatus = TransactionStatus.FAILED` and persists it to the database when the Hedera SDK throws, but returns the DTO rather than throwing — so the caller's loop never sees an exception and never stops. [2](#0-1) 

The `atomic` flag, which implies all-or-nothing semantics, is accepted via the API and used in the scheduler to route groups into `executeTransactionGroup`, but the word `atomic` does not appear anywhere inside `executeTransactionGroup`. Atomic groups are executed identically to plain parallel groups — no abort on failure. [3](#0-2) [4](#0-3) 

### Impact Explanation
A sequential transaction group is semantically ordered and dependency-bearing (e.g., tx1 creates a Hedera account, tx2 transfers HBAR to it; or tx1 sets an allowance, tx2 spends it). When tx1 fails on-chain:

1. The DB correctly records tx1 as `FAILED`.
2. The loop immediately submits tx2 to the Hedera network anyway.
3. tx2 may fail (wasted fees, confusing error) or — worse — succeed with unintended effects (e.g., HBAR sent to a wrong/non-existent account, or a state-dependent operation applied to stale state).

For `atomic` groups the impact is identical: a failure in any member transaction does not prevent the remaining transactions from being submitted. The all-or-nothing guarantee advertised by the `atomic` flag is never enforced.

This constitutes **permanent, unrecoverable state corruption** on the Hedera ledger: submitted transactions cannot be un-submitted, and HBAR transfers cannot be reversed.

### Likelihood Explanation
No privileged access is required. Any user who creates a sequential or atomic transaction group and experiences a network-level failure (insufficient fees, account not found, transaction expired, node rejection) will trigger this path automatically via the scheduler. In a multi-signature workflow, a malicious co-signer can deliberately cause tx1 to fail (e.g., by submitting a conflicting transaction to Hedera directly before the scheduler fires) while the remaining group members execute unguarded. The scheduler path is fully automated and requires no further user interaction once the group is submitted.

### Recommendation
In the sequential loop, inspect the result of `_executeTransaction` before proceeding:

```typescript
if (transactionGroup.sequential) {
  for (const { sdkTransaction, transaction } of transactions) {
    const delay = transaction.validStart.getTime() - Date.now();
    await sleep(delay);
    const result = await this._executeTransaction(transaction, sdkTransaction);
    rawResults.push(result);
    // Abort the sequence if the transaction failed
    if (result && result.status === TransactionStatus.FAILED) {
      break;
    }
  }
}
```

For `atomic` groups, implement an abort-on-failure check in the parallel path as well, and mark all remaining transactions as `FAILED` (or `CANCELED`) when any member fails.

### Proof of Concept
1. Create a sequential transaction group with two transactions: tx1 (e.g., a `CryptoTransfer` from an account with insufficient balance) and tx2 (e.g., a `CryptoTransfer` to a third party).
2. Submit the group through the API. The scheduler picks it up and calls `executeTransactionGroup`.
3. tx1 is submitted to Hedera and rejected (e.g., `INSUFFICIENT_ACCOUNT_BALANCE`). `_executeTransaction` returns `{ status: FAILED, error: "..." }`.
4. The loop does not inspect the return value. It immediately proceeds to tx2.
5. tx2 is submitted to Hedera and executes successfully, transferring HBAR even though the prerequisite tx1 failed.
6. The DB shows tx1 = `FAILED`, tx2 = `EXECUTED` — the sequential dependency was silently violated.

Relevant code path:
- Entry: `transaction-scheduler.service.ts` → `prepareTransactions` → `collateGroupAndExecute` → `executeService.executeTransactionGroup`
- Failure point: `execute.service.ts` lines 86–91 (sequential loop, no result check) [1](#0-0) [5](#0-4)

  private async _executeTransaction(
    transaction: Transaction,
    sdkTransaction: SDKTransaction,
  ): Promise<TransactionExecutedDto | null> {
    const client = await getClientFromNetwork(transaction.mirrorNetwork);

    const executedAt = new Date();
    let transactionStatus = TransactionStatus.EXECUTED;
    let transactionStatusCode = null;
    let isDuplicate = false;

    const result: TransactionExecutedDto = {
      status: transactionStatus,
    };

    try {
      const response = await sdkTransaction.execute(client);
      const receipt = await response.getReceipt(client);

      result.response = JSON.stringify(response.toJSON());
      result.receipt = JSON.stringify(receipt.toJSON());
      result.receiptBytes = Buffer.from(receipt.toBytes());
      transactionStatusCode = receipt.status._code || Status.Ok._code;
    } catch (error) {
      let message = 'Unknown error';
      let statusCode = null;

      if (error instanceof Error) {
        message = error.message;

        const status = (error as any).status;
        if (status?._code) {
          statusCode = status._code;
        } else {
          statusCode = getStatusCodeFromMessage(message);
        }
      }

      // Another pod already submitted this — don't touch the row, let the
      // successful pod win the update and emit the change
      if (statusCode === Status.DuplicateTransaction._code) {
        isDuplicate = true;
        this.logger.debug(
          `Duplicate transaction ${transaction.id} (txId=${sdkTransaction.transactionId}, statusCode=${statusCode}) detected; assuming it was successfully executed by another pod and skipping updates.`,
        );
      } else {
        transactionStatus = TransactionStatus.FAILED;
        transactionStatusCode = statusCode;
        result.error = message;
        this.logger.error(
          `Error executing transaction ${transaction.id} (txId=${sdkTransaction.transactionId}, statusCode=${statusCode}): ${message}`,
        );
      }
    } finally {
      client.close();
    }

    if (isDuplicate) return null;

    const updateResult = await this.transactionsRepo
      .createQueryBuilder()
      .update(Transaction)
      .set({ status: transactionStatus, executedAt, statusCode: transactionStatusCode })
      .where('id = :id AND status = :currentStatus', {
        id: transaction.id,
        currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
      })
      .returning('id')
      .execute();

    if (updateResult.raw.length === 0) return null;

    result.status = transactionStatus;
    return result;
  }
