### Title
Soft-Deleted (Admin-Removed) Users Can Bypass Removal by Re-Registering with the Same Email

### Summary
When an admin removes a user from the organization, `removeUser` performs only a soft delete (sets `deletedAt`). The `createUser` function explicitly detects soft-deleted records and **restores** them upon re-registration with the same email. This allows any removed user to immediately regain full organization access by simply re-registering, completely undermining the admin's removal decision — a direct analog to the Arbitrum M-04 pattern where a privileged removal action can be silently reversed.

### Finding Description

**Root cause — `createUser` restores soft-deleted accounts:**

In `back-end/apps/api/src/users/users.service.ts`, `createUser` queries with `withDeleted = true` and, upon finding a soft-deleted record, calls `updateUser` to clear `deletedAt` and reset status to `UserStatus.NEW`:

```typescript
// users.service.ts lines 32-47
async createUser(email: string, password: string): Promise<User> {
    let user = await this.getUser({ email }, true);   // withDeleted = true
    password = await this.getSaltedHash(password);

    if (user) {
      if (!user.deletedAt) throw new UnprocessableEntityException('Email already exists.');
      return this.updateUser(user, { email, password, status: UserStatus.NEW, deletedAt: null });
    }
    ...
}
``` [1](#0-0) 

**The removal action that is bypassed — `removeUser` is only a soft delete:**

```typescript
// users.service.ts lines 156-170
async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });
    await this.repo.manager.softDelete(UserKey, { userId: id });
    await this.repo.softRemove(user);
    return true;
}
``` [2](#0-1) 

**Exploit path:**
1. Admin calls `DELETE /users/:id` → `removeUser(id)` → user record gets `deletedAt` set (soft delete).
2. Removed user calls the registration endpoint (handled by `auth.controller.ts`) with the same email and any new password.
3. `createUser` finds the soft-deleted record via `withDeleted = true`, sees `deletedAt` is non-null, and restores the account: `{ deletedAt: null, status: UserStatus.NEW }`.
4. The user is fully reinstated in the organization — the admin's removal is silently reversed. [3](#0-2) 

### Impact Explanation
An admin's explicit decision to remove a user (e.g., due to a security incident, misconduct, or access revocation) is completely nullified. The reinstated user regains access to all organization transactions they were previously associated with as creator, signer, observer, or approver. No privileged access is required by the attacker — only knowledge of their own email address. This is a critical integrity failure in the user management trust model: the admin role's removal authority is rendered meaningless.

### Likelihood Explanation
Exploitation is trivial and requires zero privileged access. Any removed user who knows their own email (which they always do) can re-register immediately after removal. There is no rate limiting, admin approval gate, or blocklist check on the registration path for previously-removed accounts. The attack is deterministic and repeatable.

### Recommendation
In `createUser`, treat a soft-deleted (admin-removed) user record as permanently ineligible for self-service re-registration. Instead of restoring the account, throw an error (e.g., `UnprocessableEntityException('This account has been removed from the organization.')`). If re-onboarding a previously removed user is a legitimate use case, it should require an explicit admin action (e.g., a dedicated `reactivateUser` endpoint protected by `AdminGuard`), not a silent restore triggered by the removed user themselves.

### Proof of Concept

1. Admin registers `victim@org.com` → user exists with `deletedAt = null`.
2. Admin calls `DELETE /users/:id` → `removeUser` sets `deletedAt = <timestamp>` on the user and their keys.
3. Removed user calls `POST /auth/register` (or equivalent) with `{ email: "victim@org.com", password: "newpass123" }`.
4. `createUser` executes `this.getUser({ email }, true)` — finds the soft-deleted record.
5. Branch `if (user) { if (!user.deletedAt) throw ... ; return this.updateUser(user, { ..., deletedAt: null }); }` executes.
6. User record is restored: `deletedAt = null`, `status = UserStatus.NEW`.
7. Removed user logs in with `victim@org.com` / `newpass123` and has full organization access again. [3](#0-2)
