### Title
Unbounded Array Input in Batch Signature Endpoints Enables Authenticated Resource Exhaustion

### Summary
Two authenticated API endpoints — `POST /transactions/:transactionId/signers` and `POST /transactions/signatures/import` — accept an unbounded array of `UploadSignatureMapDto` objects with no array-size cap. A single crafted request containing thousands of entries forces the server to perform unbounded concurrent cryptographic operations, SDK deserialization, and database queries. The rate-limiter only counts requests, not payload cardinality, so it provides no protection. A developer comment in the codebase explicitly acknowledges the missing limit.

### Finding Description

**Vulnerability class**: Resource exhaustion via unbounded batch input (analogous to the external report's "extra transactions causing unnecessary computation").

**Entry point 1 — `POST /transactions/:transactionId/signers`**

The controller accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no `@ArrayMaxSize()` constraint: [1](#0-0) 

The service then fans out all items concurrently via `Promise.all()`: [2](#0-1) 

Each element triggers:
- `SDKTransaction.fromBytes()` (protobuf deserialization)
- Cryptographic signature validation (`addSignature`)
- Database reads and writes

**Entry point 2 — `POST /transactions/signatures/import`**

Same unbounded array, same missing constraint: [3](#0-2) 

Additionally, the service builds a SQL `CASE` string by linear concatenation over every element with no cap: [4](#0-3) 

The developer comment at line 575 explicitly acknowledges the missing limit: `//Added a batch mechanism, probably should limit this on the api side of things`.

**Entry point 3 — `POST /transaction-groups`**

`CreateTransactionGroupDto.groupItems` has no `@ArrayMaxSize()`: [5](#0-4) 

Each item is passed through `validateAndPrepareTransaction`, which performs signature verification, SDK deserialization, expiry check, size check, node validation, and hash computation — all concurrently via `Promise.all()`: [6](#0-5) 

**Why the throttler does not help**

The user-throttler limits to 100 requests/minute and 10 requests/second: [7](#0-6) 

A single request containing 10,000 array elements counts as one request and passes through the throttler entirely.

### Impact Explanation
**Medium** — A single authenticated request with a large array (e.g., 10,000 entries) forces the server to:
1. Spawn thousands of concurrent DB connections and queries, potentially exhausting the PostgreSQL connection pool.
2. Perform thousands of concurrent cryptographic operations (signature verification, `SDKTransaction.fromBytes`), saturating CPU.
3. For `importSignatures`, build a SQL `CASE` string proportional to input size, causing extreme memory allocation and potentially crashing the query builder or the DB.

This can degrade or crash the API service for all other users, blocking transaction signing and execution workflows.

### Likelihood Explanation
**Medium** — Any registered, verified user with a valid JWT token can reach these endpoints. No admin or special privilege is required. The attacker needs only to craft a JSON array body with many entries. The attack is repeatable and requires no prior knowledge of valid transaction IDs (invalid IDs are silently skipped, still consuming server resources for the lookup).

### Recommendation
Add `@ArrayMaxSize(N)` to the array fields at the DTO level. For example, in `UploadSignatureMapDto` when used as an array, enforce a reasonable maximum (e.g., 100 or 500). For `CreateTransactionGroupDto.groupItems`, add `@ArrayMaxSize(100)`. Additionally, enforce a maximum body size at the NestJS/HTTP layer to prevent oversized payloads from reaching business logic.

### Proof of Concept

Send the following request as any authenticated user:

```http
POST /transactions/signers HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1000.000000000": { "<pubkey_hex>": "<sig_hex>" } } } },
  { "id": 1, "signatureMap": { ... } },
  ... (repeat 10,000 times)
]
```

Expected outcome: The server spawns 10,000 concurrent `Promise.all` branches, each performing `SDKTransaction.fromBytes`, signature validation, and DB queries. CPU and DB connection pool are saturated. Legitimate requests time out or are rejected. The rate-limiter does not trigger because this is a single HTTP request.

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
