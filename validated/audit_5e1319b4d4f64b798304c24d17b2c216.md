### Title
Null Dereference in `processNotificationType` Causes Entire Signer-Reminder Batch to Silently Drop All Notifications

### Summary
`processNotificationType` in `receiver.service.ts` calls `entityManager.findOne()` to look up an existing `Notification` record, then immediately dereferences the result without a null guard. When no matching notification exists, the function throws a `TypeError`, which propagates uncaught through `handleSignerReminderNotifications` and aborts the entire `processSignerReminders` loop — causing every transaction's signer-reminder notification in the batch to be silently dropped.

### Finding Description

`processNotificationType` fetches an existing `Notification` row:

```typescript
const notification = await entityManager.findOne(Notification, {
  where: { entityId: transactionId, type: notificationType },
  relations: { notificationReceivers: true },
});
```

TypeORM's `findOne` returns `null` when no row matches. The very next use of `notification` is an unconditional property access:

```typescript
const receiversToUpdate = notification.notificationReceivers.filter(   // line 454
  nr => receiverIds.includes(nr.userId)
);
```

and again at line 479:

```typescript
const existingUserIds = new Set(
  notification.notificationReceivers.map(nr => nr.userId)   // line 479
);
```

There is no null check between the `findOne` call and these accesses.

`processNotificationType` is called from `handleSignerReminderNotifications`:

```typescript
const { newReceivers, updatedReceivers } = await this.processNotificationType(
  entityManager, transactionId, syncType, userIds, cache,   // line 832-838
);
```

`handleSignerReminderNotifications` has no try-catch. The call site in `processSignerReminders` wraps each transaction in a DB transaction but also has no per-iteration error handling:

```typescript
for (const { entityId: transactionId } of events) {
  ...
  await this.entityManager.transaction(async entityManager => {
    await this.handleSignerReminderNotifications(...);   // line 1152-1165
  });
}
// batch send happens only after the loop
await this.sendDeletionNotifications(deletionNotifications);
await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
await this.sendEmailNotifications(emailNotifications, emailReceiverIds);
```

A `TypeError` thrown by `processNotificationType` propagates out of the `entityManager.transaction` callback, out of the for-loop, and unwinds the entire `processSignerReminders` call before the batch-send lines are ever reached. All accumulated notification state is discarded.

### Impact Explanation

When a signer-reminder event batch contains even one transaction whose `Notification` row of the expected type does not exist in the database, the entire batch fails. No in-app indicator, no email reminder, and no WebSocket push is delivered for any transaction in that batch — including transactions that do have valid notification records. Users who are required to sign transactions receive no reminder, potentially causing transactions to expire unnoticed.

### Likelihood Explanation

The `TRANSACTION_INDICATOR_SIGN` notification is created by `processTransactionStatusUpdateNotifications`. If that earlier step failed (e.g., a transient DB error), was rolled back, or the notification row was deleted (e.g., by an admin or a migration), the reminder path will crash. Additionally, any transaction created before the notification system was deployed would have no pre-existing `Notification` row. Both scenarios are realistic in a production environment.

### Recommendation

Add a null guard immediately after the `findOne` call in `processNotificationType`. If `notification` is `null`, either return early with empty arrays or create the notification on the fly (matching the behaviour of `createNotificationWithReceivers`). Additionally, wrap each iteration of the for-loop in `processSignerReminders` with a try-catch so that a single failing transaction does not abort the entire batch — mirroring the per-transaction error handling already present in `handleTransactionStatusUpdateNotifications` (line 811-813).

### Proof of Concept

1. Create a transaction that reaches `WAITING_FOR_SIGNATURES` status but whose `Notification` row of type `TRANSACTION_INDICATOR_SIGN` is absent (e.g., delete it directly from the DB, or simulate a failed initial notification creation).
2. Trigger the signer-reminder scheduler so that `remindSigners` or `remindSignersManual` is called with a batch that includes this transaction's `entityId`.
3. `processSignerReminders` → `handleSignerReminderNotifications` → `processNotificationType` calls `entityManager.findOne(...)` → returns `null`.
4. Line 454: `null.notificationReceivers.filter(...)` → `TypeError: Cannot read properties of null`.
5. Exception propagates out of the `entityManager.transaction` callback and out of the for-loop.
6. `sendDeletionNotifications`, `sendInAppNotifications`, `sendEmailNotifications` are never called.
7. All signers across all transactions in the batch receive no reminder notification.
