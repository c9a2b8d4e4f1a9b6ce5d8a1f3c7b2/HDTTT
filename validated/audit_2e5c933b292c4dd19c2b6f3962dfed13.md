### Title
Transaction Creator Can Modify Approver Structure After Signatures Are Collected, Bypassing Multi-Sig Threshold

### Summary
The `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` endpoints in the API service perform no check on the transaction's current status before mutating the approver tree. A malicious transaction creator — any authenticated user — can lower the approval threshold or remove signed approvers after other parties have already committed their signatures, silently bypassing the intended multi-signature requirement.

### Finding Description

The root cause is that `getCreatorsTransaction`, the sole authorization gate used by all three approver-mutation paths, only verifies that the caller is the transaction creator. It never inspects `transaction.status`. [1](#0-0) 

All three mutation paths delegate to this function and add no status guard of their own:

- `createTransactionApprovers` — calls `getCreatorsTransaction` then proceeds unconditionally. [2](#0-1) 

- `removeTransactionApprover` (controller) — calls `getCreatorsTransaction` then removes the approver with no status check. [3](#0-2) 

- `updateTransactionApprover` — calls `getCreatorsTransaction` inside the DB transaction but never checks status. [4](#0-3) 

By contrast, `approveTransaction` — the path used by approvers — does enforce a status check (`WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`), confirming the pattern is intentional for approvers but was simply omitted for the creator's mutation paths. [5](#0-4) 

**Exploit flow:**

1. Creator submits a transaction with a threshold-3-of-3 approver tree (all three council members must sign).
2. Approver A and Approver B sign (`WAITING_FOR_SIGNATURES` status).
3. Creator calls `PATCH /transactions/:id/approvers/:nodeId` with `{ "threshold": 2 }` — no status check blocks this.
4. The threshold node is updated to 2-of-3 in the database.
5. The scheduler's `processTransactionStatus` re-evaluates the tree, finds 2 signatures satisfy the new threshold-2 requirement, and advances the transaction to `WAITING_FOR_EXECUTION`.
6. The transaction is executed on-chain without Approver C's signature.

The same primitive can be used to remove a signed approver from the tree, or to inject a new approver controlled by the creator after legitimate approvers have already signed.

### Impact Explanation

The Hedera Transaction Tool's core security guarantee is that a transaction cannot be submitted to the Hedera network unless the configured set of approvers has signed off. Bypassing the threshold check breaks this guarantee entirely. For Hedera Council use cases — where transactions may govern network-level parameters — this means a single compromised or malicious creator account can unilaterally execute transactions that were designed to require consensus from multiple independent parties.

### Likelihood Explanation

The attacker is any authenticated user who created a transaction. No admin credentials, no leaked keys, and no external access are required. The API endpoint is reachable over standard HTTP with a valid JWT. The window of opportunity is the entire `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` period, which can last hours or days in practice.

### Recommendation

Add a terminal/active-status guard inside `getCreatorsTransaction` (or at the top of each mutation method) that rejects modifications when the transaction is in `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any terminal state (`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, `ARCHIVED`). Approver-tree mutations should only be permitted while the transaction is in a draft/pending state before any signatures have been collected.

```typescript
// In getCreatorsTransaction or at the start of each mutation:
const nonModifiableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
  TransactionStatus.EXECUTED,
  TransactionStatus.FAILED,
  TransactionStatus.CANCELED,
  TransactionStatus.EXPIRED,
  TransactionStatus.ARCHIVED,
];
if (nonModifiableStatuses.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify approvers once signing has begun');
}
```

### Proof of Concept

**Preconditions:** Two registered users (Alice = creator, Bob, Carol = approvers). Transaction T requires threshold 2-of-2 (Bob and Carol).

1. Alice creates transaction T via `POST /transactions`.
2. Alice adds Bob and Carol as approvers with threshold 2 via `POST /transactions/T/approvers`.
3. Bob signs and submits approval via `POST /transactions/T/approvers/approve`. Transaction status becomes `WAITING_FOR_SIGNATURES`.
4. Alice sends `PATCH /transactions/T/approvers/<threshold-node-id>` with body `{ "threshold": 1 }`.
   - Expected: 400 error (transaction is in signing phase).
   - Actual: 200 OK — threshold is updated to 1-of-2.
5. The scheduler runs `processTransactionStatus`; Bob's single signature now satisfies threshold 1. Transaction advances to `WAITING_FOR_EXECUTION` and is submitted to Hedera without Carol's approval.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
