Audit Report

## Title
Unimplemented `getTransactionApproversForTransactions` Causes Approver Data to Always Be Empty in Transaction Group Details

## Summary
The `getTransactionApproversForTransactions` function in `transactions.service.ts` is an explicit stub that unconditionally returns an empty array. It is the sole data source for populating `transaction.approvers` when a transaction group is fetched with `full=true`. Every transaction inside a group therefore always reports zero approvers to the client, regardless of actual database state.

## Finding Description
The stub is located at: [1](#0-0) 

```ts
async getTransactionApproversForTransactions(
  transactionIds: number[],
): Promise<TransactionApprover[]> {
  if (!transactionIds.length) {
    return [];
  }
  //To be implemented when approver functionality is added.
  return [];
}
```

The call site in `getTransactionGroup` (invoked on every `GET /transactions/groups/:id?full=true` request) passes the result directly into the approver map: [2](#0-1) 

Because `transactionApprovers` is always `[]`, `approverMap` is always empty, and every `groupItem.transaction.approvers` is set to `[]`.

By contrast, the individual-transaction path correctly fetches approvers: [3](#0-2) 

The `approversService.getApproversByTransactionId`, `createTransactionApprovers`, and `approveTransaction` are all fully implemented and functional — only the batch-fetch path used by the group view is a dead stub.

## Impact Explanation
1. **Approval workflow invisible in group view.** Any front-end or API consumer that reads `transaction.approvers` from the group detail response will always see an empty list, regardless of how many approvers are assigned. Approvers cannot see their pending tasks through the group view.
2. **Potential approval-gate bypass.** If any client-side or downstream logic conditions execution readiness on `transaction.approvers.length > 0`, the stub causes every group transaction to appear approval-free, potentially allowing premature execution treatment before required approvals are collected.
3. **Structurally incorrect response.** The `approved` and `signature` fields on `TransactionApprover` records are never surfaced in the group context, making the group detail response untrustworthy for any consumer.

## Likelihood Explanation
This is triggered by any authenticated organization member calling `GET /transactions/groups/:id?full=true`. No special privileges are required. The stub is unconditional — there is no feature flag or configuration that bypasses it. Every invocation of `getTransactionGroup` with `full=true` hits this code path. [4](#0-3) 

## Recommendation
Implement `getTransactionApproversForTransactions` in `transactions.service.ts` analogously to the already-implemented `getTransactionObserversForTransactions`. The implementation should query `TransactionApprover` records filtered by `transactionId IN (transactionIds)`, including the necessary relations (e.g., `userKey`), and return the flat list for the caller to group by `transactionId`. The existing `approversService.getApproversByTransactionId` and `getTreeStructure` methods can serve as reference for the query shape. [5](#0-4) 

## Proof of Concept
1. Create a transaction group with at least one transaction that has approvers assigned via `POST /transactions/approvers`.
2. Call `GET /transactions/groups/:id?full=true` as any authenticated member.
3. Observe that every `groupItems[*].transaction.approvers` in the response is `[]`, despite approver records existing in the `transaction_approver` table.
4. Compare with `GET /transactions/:id` for the same transaction — the individual endpoint correctly returns the approver tree via `attachTransactionApprovers`. [1](#0-0)
