## Audit Report

## Title
`handleExpiredTransactions` Uses Hardcoded 3-Minute Window Instead of Actual `validDuration`, Causing Incorrect `FAILED` Status on Expired Transactions

## Summary
The transaction scheduler marks transactions as `EXPIRED` only when `validStart < now - 180s`, regardless of the actual `transactionValidDuration` embedded in the transaction bytes. Hedera's default `validDuration` is 120 seconds and the maximum is 180 seconds. Any transaction with `validDuration < 180s` will be submitted to Hedera after it has already expired on-network, receive a `TRANSACTION_EXPIRED` rejection, and be recorded as `FAILED` instead of `EXPIRED`.

## Finding Description

The `Transaction` entity stores only `validStart` as a database column; `validDuration` is not persisted separately and is only accessible by deserializing `transactionBytes`. [1](#0-0) 

The `isExpired` utility correctly computes expiry as `validStart + validDuration * 1000`: [2](#0-1) 

However, `handleExpiredTransactions` ignores `validDuration` entirely and uses a hardcoded 180-second cutoff: [3](#0-2) 

`isValidStartExecutable`, which gates whether the scheduler attempts execution, uses the same hardcoded 3-minute window: [4](#0-3) 

When Hedera rejects the submission with `TRANSACTION_EXPIRED`, `_executeTransaction` catches the error, resolves `statusCode = 4` via `getStatusCodeFromMessage`, and writes `status = FAILED`: [5](#0-4) [6](#0-5) 

The `isExpired` utility is only called at transaction creation time and during signature import — never inside the scheduler's expiry or execution path: [7](#0-6) 

## Impact Explanation

- Transactions with `validDuration < 180s` (including the Hedera SDK default of 120s) are submitted to the network after they have already expired. Hedera rejects them with `TRANSACTION_EXPIRED`.
- The final recorded status is `FAILED` (with `statusCode = 4`) instead of `EXPIRED`, corrupting the audit trail.
- The `generateTransactionExpiredContent` email notification path is never triggered for these transactions; instead they appear as execution failures. [8](#0-7) 
- Any downstream logic branching on `EXPIRED` vs `FAILED` (retry policies, user-facing error messages, analytics) receives incorrect data. [9](#0-8) 

## Likelihood Explanation

The API's `validateAndPrepareTransaction` does not enforce a minimum `validDuration` of 180 seconds; it only checks `isExpired` at submission time: [10](#0-9) 

The CSV import path explicitly accepts a user-supplied `transaction valid duration` field and defaults to 180 only if absent: [11](#0-10) [12](#0-11) 

Any authenticated user can submit a transaction with `validDuration` between Hedera's minimum (15s) and 179s. The cron runs every 10 seconds, so the incorrect state is observable on every polling cycle during the gap window. [13](#0-12) 

## Recommendation

1. **Persist `validDuration`**: Add a `validDuration` column to the `Transaction` entity, populated from `sdkTransaction.transactionValidDuration` during `validateAndPrepareTransaction`.
2. **Fix `handleExpiredTransactions`**: Change the expiry query to use `validStart + validDuration < now` (requires the persisted column).
3. **Fix `isValidStartExecutable`**: Replace the hardcoded 3-minute window with `validStart + validDuration >= now` (transaction not yet expired on-network).
4. **Short-term mitigation**: Use the minimum possible `validDuration` (15s) as the expiry cutoff in `handleExpiredTransactions` to reduce the gap, or enforce `validDuration = 180s` at the API layer.

## Proof of Concept

1. Authenticate as any user and POST to `/transactions` with `transactionBytes` encoding a transaction where `.setTransactionValidDuration(60)` is set and `validStart = now`.
2. Wait 61 seconds. The transaction is now expired on Hedera.
3. Observe that `handleExpiredTransactions` does NOT mark it `EXPIRED` (since `validStart = now - 61s > now - 180s`).
4. `isValidStartExecutable` returns `true` (since `now - 61s` is within the 3-minute window).
5. `prepareTransactions` calls `collateAndExecute` → `addExecutionTimeout`.
6. The execution timeout fires (negative delay → immediate), submits to Hedera, receives `TRANSACTION_EXPIRED`.
7. `_executeTransaction` sets `status = FAILED`, `statusCode = 4`.
8. Query the transaction: `status = "FAILED"`, `statusCode = 4` — not `EXPIRED`. [14](#0-13)

export const isExpired = (transaction: SDKTransaction) => {
  if (!transaction.transactionId?.validStart) {
    return true;
  }

  const validStart = transaction.transactionId.validStart.toDate();
  const duration = transaction.transactionValidDuration;

  return new Date().getTime() >= validStart.getTime() + duration * 1_000;
};

  private async validateAndPrepareTransaction(
    dto: CreateTransactionDto,
    user: User,
    client: Client,
  ) {
    const creatorKey = user.keys.find(key => key.id === dto.creatorKeyId);

    if (!creatorKey) {
      throw new BadRequestException(`Creator key ${dto.creatorKeyId} not found`);
    }

    const publicKey = PublicKey.fromString(creatorKey.publicKey);

    // Verify signature
    const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
    if (!validSignature) {
      throw new BadRequestException(ErrorCodes.SNMP);
    }

    // Parse SDK transaction
    const sdkTransaction = SDKTransaction.fromBytes(dto.transactionBytes);

    // Check the transaction is frozen, cannot require it to be frozen, breaks backwards compatibility
    if (!sdkTransaction.isFrozen()) {
      sdkTransaction.freezeWith(client);
    }

    // Check if expired
    if (isExpired(sdkTransaction)) {
      throw new BadRequestException(ErrorCodes.TE);
