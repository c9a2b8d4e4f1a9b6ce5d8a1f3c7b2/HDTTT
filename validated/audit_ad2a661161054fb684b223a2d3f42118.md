### Title
Unbounded Transaction Fetch and Per-Row DB Query in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction in the database with no row limit, then executes an additional async database call (`userKeysToSign`) for each row before applying pagination. Any authenticated user can trigger this endpoint, causing unbounded memory allocation, CPU consumption, and N+1 database queries proportional to the total number of active transactions in the system.

### Finding Description

**Root cause — no `take`/`skip` on the bulk fetch:** [1](#0-0) 

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

No `take` or `skip` is passed. TypeORM's `find` without `take` returns every matching row. The `whereForUser` filter only excludes five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`); all other transactions — across all users — are loaded into memory.

**N+1 async DB call per row:** [2](#0-1) 

```ts
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
```

`userKeysToSign` calls `userKeysRequiredToSign` → `keysRequiredToSign`, which performs additional database queries per transaction to resolve signature keys. [3](#0-2) 

**Pagination applied only after full processing:** [4](#0-3) 

```ts
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
```

The caller's `page=1&size=1` request still causes the server to load and process every active transaction before slicing.

**Endpoint is live and reachable by any authenticated user:** [5](#0-4) 

The comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end has moved on, but the route `GET /transactions/sign` remains deployed and protected only by standard user authentication — no admin role required.

### Impact Explanation

A single authenticated HTTP request to `GET /transactions/sign` causes the API process to:
1. Load all non-terminal transactions (potentially thousands) into Node.js heap memory.
2. Execute one or more database queries per transaction sequentially (N+1 pattern).
3. Hold a database connection pool slot for the entire duration.

With a large transaction table, this degrades or crashes the API service for all users. Repeated calls (even at the rate-limit ceiling of 100 req/min per user) sustain the exhaustion. The impact is **service unavailability / severe degradation** — a concrete, measurable outcome matching the RESEARCHER.md target category.

### Likelihood Explanation

- **Attacker precondition:** valid user account only (self-registration is a normal workflow).
- **No special knowledge required:** the endpoint path is standard REST convention and documented in the OpenAPI spec generated by NestJS decorators.
- **Rate limits exist** (`user-minute: 100`, `user-second: 10`) but do not prevent the attack — each individual request is already sufficient to exhaust resources if the transaction table is large, and 100 requests/minute sustains the load.
- The vulnerability worsens naturally over time as the platform accumulates transactions, requiring no attacker action to increase severity.

### Recommendation

Apply the database-level limit **before** the per-row processing loop, mirroring the pattern already used correctly in `getHistoryTransactions` and `getTransactionsToApprove`:

```ts
// Replace the unbounded find:
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // No take/skip → unbounded ← BUG
});

// With a cursor/keyset or two-phase approach:
// Phase 1: paginated candidate fetch (DB-side)
const candidates = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: SAFE_BATCH_LIMIT,   // e.g. 500
  skip: offset,
});
// Phase 2: filter by signing eligibility within the batch
```

Alternatively, push the signing-eligibility check into the SQL query (as the newer `getTransactionNodesQuery` path already does), eliminating the N+1 pattern entirely. [6](#0-5) 

### Proof of Concept

**Preconditions:** authenticated user account; system has ≥ 1 000 non-terminal transactions (achievable by any user creating transactions via the normal API).

**Steps:**
1. Obtain a valid JWT via `POST /auth/login`.
2. Send: `GET /transactions/sign?page=1&size=1` with the JWT.
3. Observe: the API process allocates memory proportional to the total non-terminal transaction count, executes N+1 DB queries (visible in DB slow-query logs), and response latency scales linearly with table size.
4. Repeat at the allowed rate (100 req/min) to sustain server degradation.

**Expected outcome:** API response times increase to seconds/timeouts for all concurrent users; database connection pool exhaustion causes cascading failures across other endpoints.

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
};
```
