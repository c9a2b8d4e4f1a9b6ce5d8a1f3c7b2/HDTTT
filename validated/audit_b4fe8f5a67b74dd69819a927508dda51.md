Audit Report

## Title
`archiveTransaction()` Bypasses Status Guard When `isManual` Is True, Allowing Terminal-State Transactions to Be Archived

## Summary
The `archiveTransaction()` function in `TransactionsService` uses a logically flawed `&&` guard that is fully bypassed when `transaction.isManual === true`. Any authenticated user who created a manual transaction can archive it after it has reached a terminal state (`EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, `REJECTED`), overwriting the recorded outcome and corrupting the application's audit trail.

## Finding Description

**Root cause — flawed boolean guard:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `archiveTransaction()` reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard throws only when **both** sub-conditions are true simultaneously:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- **AND** `isManual === false`

Consequently, the guard does **not** throw when **either** sub-condition is false — including when `isManual === true`, regardless of status. A manual transaction in `EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, or `REJECTED` state passes the guard and proceeds to line 720:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

The service itself defines `terminalStatuses` as `[EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED, REJECTED]`, confirming these are intended to be immutable final states: [3](#0-2) 

The `TransactionStatus` enum confirms all relevant statuses exist: [4](#0-3) 

**Exploit path:**
1. Authenticated user creates a manual transaction (`isManual: true`) via `POST /transactions`.
2. The transaction reaches a terminal state (e.g., `EXECUTED` after on-chain execution, or `CANCELED`).
3. The creator calls `PATCH /transactions/archive/:id`.
4. `archiveTransaction()` fetches the transaction via `getTransactionForCreator()` (ownership verified), evaluates the guard — since `isManual === true`, the `&&` short-circuits and the status check is skipped — and executes `repo.update({ id }, { status: TransactionStatus.ARCHIVED })`.
5. The transaction's status is now `ARCHIVED`, overwriting its true terminal state.

The controller endpoint is reachable by any authenticated, verified user who is the transaction creator: [5](#0-4) 

**Test coverage gap confirmed:**

The negative test only covers `isManual` unset (falsy) with `CANCELED` — which correctly throws. The positive test only covers `isManual: true` with `WAITING_FOR_EXECUTION` — a valid case. The combination of `isManual: true` + any terminal status is never tested: [6](#0-5) 

## Impact Explanation

A transaction creator can overwrite the recorded outcome of any of their manual transactions from a terminal state to `ARCHIVED`:

- An `EXECUTED` transaction (confirmed on-chain) can be re-labeled `ARCHIVED`, hiding that it was executed from observers, approvers, and audit logs within the application.
- A `FAILED` or `CANCELED` transaction can be re-labeled `ARCHIVED`, obscuring its true outcome.
- Notification and status-update events are emitted with the new `ARCHIVED` status via `emitTransactionStatusUpdate`, propagating the corrupted state to all subscribers. [7](#0-6) 

This is an unauthorized state transition that violates the application's state integrity invariant: terminal states should be immutable.

## Likelihood Explanation

- **Attacker profile**: Any authenticated, verified user who has created at least one manual transaction. No privileged role required.
- **Preconditions**: The attacker must be the creator of an `isManual: true` transaction that has reached a terminal state. Manual transactions are a supported, documented feature.
- **Trigger**: A single `PATCH /transactions/archive/:id` HTTP request. No race condition or timing dependency required.

## Recommendation

Replace the `&&` compound guard with two independent checks so that `isManual` cannot bypass the status check:

```typescript
// Option 1: explicit allowlist that covers both manual and non-manual cases
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!archivableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}

// Then separately enforce that only manual transactions can be archived
if (!transaction.isManual) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

This ensures that regardless of `isManual`, a transaction in a terminal state (`EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, `REJECTED`) can never be archived.

## Proof of Concept

```typescript
// Arrange: manual transaction already in EXECUTED state
const transaction = {
  id: 999,
  transactionId: '0.0.12345@1700000000.000000000',
  creatorKey: { userId: 1 },
  isManual: true,                          // <-- bypass key
  status: TransactionStatus.EXECUTED,      // <-- terminal state
  mirrorNetwork: 'testnet',
};

jest
  .spyOn(service, 'getTransactionForCreator')
  .mockResolvedValueOnce(transaction as Transaction);

// Act: call archiveTransaction — should throw but does NOT
const result = await service.archiveTransaction(999, { id: 1 } as User);

// Assert: status was overwritten from EXECUTED to ARCHIVED
expect(transactionsRepo.update).toHaveBeenCalledWith(
  { id: 999 },
  { status: TransactionStatus.ARCHIVED },
);
expect(result).toBe(true); // succeeds — terminal state silently overwritten
```

The guard at lines 711–718 evaluates `(!statusCheck && !isManual)`. With `isManual: true`, `!transaction.isManual` is `false`, making the entire `&&` expression `false`, so the `throw` is never reached. [8](#0-7)

  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&

      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
