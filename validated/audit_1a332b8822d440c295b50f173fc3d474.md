Audit Report

## Title
Transaction Creator Can Modify Approver Structure After Signing Has Started, Bypassing Multi-Approval Policy

## Summary
`updateTransactionApprover` in `ApproversService` allows the transaction creator to mutate the approver tree (threshold, userId, listId) at any transaction status, including `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION`. No status guard exists anywhere in the call path. A malicious creator can reduce the required threshold or swap in a colluding approver after legitimate approvers have already signed, bypassing the multi-approval policy the transaction was created with.

## Finding Description

The root cause is in `updateTransactionApprover` (lines 367–531 of `approvers.service.ts`). After verifying the approver exists and the caller is the creator, the function immediately proceeds to mutate the approver tree with no check on the transaction's current status. [1](#0-0) 

`getCreatorsTransaction`, called at line 394, only validates creator ownership — it fetches the transaction and checks `creatorKey.userId`, nothing more: [2](#0-1) 

The threshold update path (lines 467–488) allows the creator to lower the threshold to any value ≥ 1 and ≤ current approver count, with no status guard: [3](#0-2) 

The userId update path (lines 489–517) allows replacing an approver, clearing their existing `signature`, `userKeyId`, and `approved` fields in the process: [4](#0-3) 

The unit tests confirm this is the intended (but insecure) behavior — the test fixture explicitly uses `TransactionStatus.WAITING_FOR_EXECUTION` and the update succeeds without any status rejection: [5](#0-4) 

The controller exposes this via `PATCH /transactions/:transactionId/approvers/:id`, guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin guard, no status guard: [6](#0-5) [7](#0-6) 

## Impact Explanation

A transaction creator can unilaterally reduce the approval threshold or swap in colluding approvers after signing has started. This breaks the multi-approval integrity guarantee that the approver system is designed to enforce. In the context of this tool — which manages Hedera governance transactions (system file updates, node operations, account key rotations) — bypassing the multi-approval policy means a single actor can push through high-impact Hedera network transactions that were supposed to require organizational consensus.

## Likelihood Explanation

The attacker precondition is only that they are an authenticated user who created a transaction — a role reachable by any registered user with no privileged access. The attack requires no leaked credentials, no admin keys, and no external dependencies. The `PATCH` endpoint is reachable via the standard REST API. The only timing requirement is that the creator acts before the chain service executes the transaction.

## Recommendation

Add a status guard at the top of `updateTransactionApprover` (and similarly in `removeTransactionApprover`) that rejects modifications when the transaction is in `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any terminal state. Only `NEW` (or an equivalent draft status) should permit structural changes to the approver tree. The guard should be applied using the transaction object already fetched by `getCreatorsTransaction`, checking `transaction.status` before any mutation is performed.

## Proof of Concept

1. Attacker (authenticated user) creates a transaction and configures an approver tree with `threshold=3` (3-of-5 approvals required).
2. Two legitimate approvers call `POST /transactions/:id/approvers/approve` — the transaction moves to `WAITING_FOR_SIGNATURES`.
3. Attacker calls `PATCH /transactions/:id/approvers/:treeNodeId` with body `{ "threshold": 1 }`.
4. `updateTransactionApprover` fetches the transaction via `getCreatorsTransaction`, confirms the caller is the creator, and proceeds — no status check occurs.
5. The threshold is updated to `1` in the database. The existing single approval already satisfies the new threshold.
6. The chain scheduler (`updateTransactions` in `transaction-scheduler.service.ts`) evaluates the approver tree on its next tick, finds the threshold met, and advances the transaction to execution — without the intended 3-of-5 approval.

Alternatively, the attacker can call `PATCH` with `{ "userId": <colluding_user_id> }` to replace an unsigned approver slot, clearing any signature state, then have the colluding user sign — bypassing the original approver's participation requirement entirely. [8](#0-7)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
