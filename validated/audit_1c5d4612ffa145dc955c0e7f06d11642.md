### Title
TOCTOU Race Condition in `uploadKey` Allows Bypassing `MAX_USER_KEYS` Limit

### Summary
The `uploadKey` function in `UserKeysService` enforces a per-user key limit by reading the current key count from the database and comparing it to `MAX_USER_KEYS`. Because the check and the subsequent insert are not atomic, a user can send multiple concurrent `POST /user/:userId/keys` requests to bypass the limit and register more keys than allowed. This is the same vulnerability class as the archiver join limit bug: a limit check that reads committed state but does not account for in-flight concurrent operations.

### Finding Description

**Root cause — non-atomic check-then-insert:**

`attachKeys` loads the user's current keys from the database only if `user.keys` is empty: [1](#0-0) 

`uploadKey` then checks the loaded count against `MAX_USER_KEYS`: [2](#0-1) 

If the check passes, a new `UserKey` is created and persisted: [3](#0-2) 

There is no database-level constraint on the count of keys per user, and no pessimistic lock or atomic `INSERT … WHERE count < MAX` guard. Each concurrent HTTP request creates its own `user` object (populated from the JWT token with `user.keys` empty), calls `attachKeys` independently, reads the same snapshot count from the database, passes the check, and then saves a new key.

**Exploit path:**

1. Attacker authenticates as a normal user (no privileged access required).
2. User currently has `MAX_USER_KEYS - 1` keys.
3. Attacker fires N concurrent `POST /user/:userId/keys` requests, each with a distinct new public key.
4. All N requests call `attachKeys`, all read `MAX_USER_KEYS - 1` from the database simultaneously.
5. All N requests evaluate `MAX_USER_KEYS - 1 >= MAX_USER_KEYS` → `false`.
6. All N requests proceed to `repo.save(userKey)`.
7. Result: user now has `MAX_USER_KEYS - 1 + N` keys — exceeding the limit by `N - 1`.

The e2e test that validates the limit is explicitly **skipped** in the test suite: [4](#0-3) 

### Impact Explanation

A user can accumulate an unbounded number of registered public keys. These keys are used throughout the system for transaction signing, signature verification, and key-based access control. Consequences include:

- **Resource exhaustion**: Unbounded rows in the `user_key` table, degrading database performance for all users.
- **Signature engine abuse**: Functions such as `keysRequiredToSign` and `processTransactionSignatures` iterate over all user keys. An attacker with thousands of keys can cause CPU/memory exhaustion in the chain service during signature computation.
- **Business logic integrity**: The key limit is a deliberate invariant. Bypassing it undermines trust assumptions made elsewhere in the codebase (e.g., `attachKeys` is called in many places and its result is used for authorization decisions).

### Likelihood Explanation

- **Attacker preconditions**: Only a valid authenticated account is required — no admin or privileged role.
- **Exploitation complexity**: Sending concurrent HTTP requests is trivial with any HTTP client (e.g., `Promise.all` in Node.js, `asyncio.gather` in Python, or `curl` with `&`).
- **No rate limiting observed** in the controller or service that would prevent burst requests. [5](#0-4) 

### Recommendation

Replace the application-level read-check-insert pattern with an atomic database-level enforcement:

1. **Add a database-level `CHECK` constraint or a partial unique index** that limits the count of non-deleted keys per user to `MAX_USER_KEYS`.
2. **Alternatively**, use a pessimistic row-level lock (`SELECT … FOR UPDATE`) on the user row before reading the key count, ensuring only one concurrent request can pass the check at a time.
3. **Remove the `it.skip`** on the e2e test that validates the limit so regressions are caught automatically.

### Proof of Concept

```typescript
// Attacker is authenticated; user currently has MAX_USER_KEYS - 1 keys.
// Send N concurrent upload requests with distinct public keys:

const { PrivateKey } = require('@hiero-ledger/sdk');
const axios = require('axios');

const SERVER = 'http://localhost:3000';
const AUTH_TOKEN = '<attacker_jwt>';
const N = 5; // concurrent requests

const requests = Array.from({ length: N }, async () => {
  const key = PrivateKey.generateED25519();
  return axios.post(
    `${SERVER}/user/2/keys`,
    { publicKey: key.publicKey.toStringRaw() },
    { headers: { Authorization: `Bearer ${AUTH_TOKEN}` } },
  );
});

const results = await Promise.all(requests);
// Expected (with fix): only 1 succeeds, rest return 400 UMK.
// Actual (without fix): all N succeed → user has MAX_USER_KEYS - 1 + N keys.
console.log(results.map(r => r.status)); // [201, 201, 201, 201, 201]
```
