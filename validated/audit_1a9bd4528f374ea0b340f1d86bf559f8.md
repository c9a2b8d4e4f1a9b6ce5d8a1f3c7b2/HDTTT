### Title
Unbounded Array Iteration in `uploadSignatureMaps` Enables Authenticated DoS via Resource Exhaustion

### Summary
The `uploadSignatureMaps` function in `signers.service.ts` accepts an unbounded `UploadSignatureMapDto[]` array with no server-side size cap. It fans out all items concurrently via `Promise.all`, triggering expensive cryptographic operations and database queries for every element in a single request. A malicious authenticated user can submit one crafted request with thousands of entries to exhaust server CPU, memory, and the database connection pool, degrading or crashing the service for all users. The developers themselves flagged the analogous `importSignatures` path in `transactions.service.ts` with the comment "probably should limit this on the api side of things."

### Finding Description

**Root cause — no array size bound before concurrent fan-out**

`uploadSignatureMaps` in `signers.service.ts` receives the raw caller-supplied array and immediately fans it out with `Promise.all`: [1](#0-0) 

Inside `validateAndProcessSignatures`, every element of `dto` is mapped to an async closure and all closures are awaited simultaneously: [2](#0-1) 

Each closure performs:
- A database lookup (`transactionMap.get`)
- `SDKTransaction.fromBytes` (protobuf deserialization)
- Nested triple-loop over the `SignatureMap` to call `sdkTransaction.addSignature` per key
- A `UserKey` lookup per public key [3](#0-2) 

After all concurrent work completes, `persistSignatureChanges` issues bulk SQL writes proportional to the input size: [4](#0-3) 

The parallel `importSignatures` path in `transactions.service.ts` has the same structure and carries a developer comment acknowledging the missing limit: [5](#0-4) 

**No DTO-level guard found.** No `@ArrayMaxSize` or equivalent class-validator decorator was located on `UploadSignatureMapDto[]` in the searched codebase.

### Impact Explanation

A single authenticated HTTP request containing N transaction IDs causes:
- N concurrent `SDKTransaction.fromBytes` + signature-map iterations (CPU/memory proportional to N)
- N concurrent database reads plus bulk writes (DB connection pool exhaustion)
- Node.js event loop saturation, making the API unresponsive to all other users

Because the work is concurrent (`Promise.all`), the resource spike is immediate and proportional to N, not amortised. Sustained or repeated requests can crash the process or keep it in a degraded state, constituting a denial-of-service against the entire API service.

### Likelihood Explanation

Any registered, authenticated user who can reach the signature-upload endpoint can trigger this. No admin privilege, leaked credential, or physical access is required. The endpoint is part of the normal transaction-signing workflow, so the attacker path is reachable without any unusual precondition. The developer comment at line 575 of `transactions.service.ts` confirms the team is already aware the limit is missing.

### Recommendation

1. **Enforce a maximum array length at the DTO layer** using a class-validator decorator, e.g.:
   ```typescript
   @IsArray()
   @ArrayMaxSize(50)          // tune to operational needs
   uploadSignatureMaps: UploadSignatureMapDto[];
   ```
2. **Replace `Promise.all` with bounded concurrency** (e.g., process in chunks of a fixed size) so that even a large but legitimate batch cannot saturate the event loop.
3. **Apply the same fix to `importSignatures`** in `transactions.service.ts`, which the developers already flagged as needing an API-side limit.

### Proof of Concept

**Preconditions:** attacker holds a valid JWT for any normal user account.

**Steps:**
1. Obtain a valid JWT via the normal login flow.
2. Collect or fabricate a list of valid transaction IDs (or reuse the same ID thousands of times — the batch query will return the same row for each duplicate).
3. Send a single POST request to the signature-upload endpoint with a body containing an array of, e.g., 5 000 entries:
   ```json
   [
     { "id": 1, "signatureMap": { ... } },
     { "id": 1, "signatureMap": { ... } },
     ... (×5000)
   ]
   ```
4. The server immediately spawns 5 000 concurrent async tasks, each deserialising a Hedera transaction and iterating its signature map.
5. Observe: API response latency for all other users spikes; Node.js process memory and CPU climb sharply; database connection pool is exhausted; repeated requests keep the service in a degraded state.

**Expected outcome:** service-wide API degradation or crash from a single unauthenticated-to-authenticated request, with no privileged access required.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

  private async persistSignatureChanges(
    validationResults: any[],
    user: User,
  ) {
    const signers = new Set<TransactionSigner>();
    let notificationsToDismiss: number[] = [];

    // Prepare batched operations
    const transactionsToUpdate: { id: number; transactionBytes: Buffer }[] = [];
    const notificationsToUpdate: { userId: number; transactionId: number }[] = [];
    const signersToInsert: { userId: number; transactionId: number; userKeyId: number }[] = [];
    const transactionsToProcess: { id: number; transaction: Transaction }[] = [];

    for (const result of validationResults) {
      if (result.error) {
        console.error(`[TX ${result.id}] Validation failed: ${result.error}`);
        continue;
      }

      const { id, transaction, sdkTransaction, userKeys, isSameBytes } = result;

      // Skip if nothing to do - no signatures were added to the transaction
      // AND no new signers were inserted (the signature can be present on the transaction
      // if collated by an outside or 'offline' method)
      if (isSameBytes && userKeys.length === 0) continue;

      // Collect updates
      if (!isSameBytes) {
        transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
        transactionsToUpdate.push({ id, transactionBytes: transaction.transactionBytes });
      }

      // Collect inserts
      if (userKeys.length > 0) {
        const newSigners = userKeys.map(userKey => ({
          userId: user.id,
          transactionId: id,
          userKeyId: userKey.id,
        }));
        signersToInsert.push(...newSigners);
      }

      transactionsToProcess.push({ id, transaction });
      notificationsToUpdate.push({ userId: user.id, transactionId: transaction.id });
    }

    // Execute in single transaction
    try {
      await this.dataSource.transaction(async manager => {
        // Bulk update transactions
        if (transactionsToUpdate.length > 0) {
          await this.bulkUpdateTransactions(manager, transactionsToUpdate);
        }

        // Bulk update notifications
        if (notificationsToUpdate.length > 0) {
          const updatedNotificationReceivers = await this.bulkUpdateNotificationReceivers(manager, notificationsToUpdate);

          // To maintain backwards compatibility and multi-machine support, we send off a dismiss event.
          emitDismissedNotifications(
            this.notificationsPublisher,
            updatedNotificationReceivers,
          );

          notificationsToDismiss = updatedNotificationReceivers.map(nr => nr.id);
        }

        // Bulk insert signers
        if (signersToInsert.length > 0) {
          const results = await this.bulkInsertSigners(manager, signersToInsert);
          results.forEach(signer => signers.add(signer));
        }
      });
    } catch (err) {
      console.error('Database transaction failed:', err);
      throw new BadRequestException(ErrorCodes.FST);
    }

    return {
      transactionsToProcess,
      signers,
      notificationsToDismiss,
    };
  }
