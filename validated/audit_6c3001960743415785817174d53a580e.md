Audit Report

## Title
IP-Based Rate Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header

## Summary
The global `IpThrottlerGuard` â€” the sole rate-limiting `APP_GUARD` applied to every API endpoint â€” derives its throttle key directly from the raw `req.headers['x-forwarded-for']` value without any validation or sanitization. Because this header is fully attacker-controlled, any HTTP client can rotate arbitrary values to generate a fresh throttle key per request, completely bypassing the configured rate limits.

## Finding Description

**Root cause â€” `IpThrottlerGuard.getTracker()`:**

In `back-end/apps/api/src/guards/ip-throttler.guard.ts`, the guard unconditionally prefers the raw `x-forwarded-for` header over `req.ip`:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [1](#0-0) 

The NestJS throttler uses the return value of `getTracker()` as the Redis key for the rate-limit counter. A different string means a different counter, so rotating the header value resets the counter on every request.

**Global `APP_GUARD` registration:**

`IpThrottlerGuard` is registered as the global `APP_GUARD` in `back-end/apps/api/src/api.module.ts`, making it the primary rate-limiting layer for every controller: [2](#0-1) 

**`UserThrottlerModule` never imported:**

Only `IpThrottlerModule` and `EmailThrottlerModule` are imported. `UserThrottlerModule` is defined but never imported into `ApiModule`, and `UserThrottlerGuard` is never registered as a fallback: [3](#0-2) [4](#0-3) 

**`trust proxy` enabled in production:**

In `back-end/apps/api/src/main.ts`, production deployments call `app.enable('trust proxy')`, which makes Express populate `req.ip` correctly from the proxy chain â€” but the guard ignores `req.ip` whenever `x-forwarded-for` is present: [5](#0-4) 

**Unit tests confirm no validation:**

The unit tests in `back-end/apps/api/src/guards/ip-throttler.guard.spec.ts` confirm the guard accepts any arbitrary string in the header without validation: [6](#0-5) 

**`/notifications/remind-signers` per-IP `@Throttle` override equally bypassable:**

The `remindSigners` endpoint applies a stricter `@Throttle` override (`global-minute` limit of 1 per 60 seconds), but since `IpThrottlerGuard` is still the guard resolving the tracker key from the raw `x-forwarded-for` header, this override is equally bypassable: [7](#0-6) 

## Impact Explanation

An unauthenticated attacker can send an unlimited number of requests to any endpoint by cycling the `X-Forwarded-For` header (e.g., `10.0.0.1`, `10.0.0.2`, â€¦). This exhausts server CPU, PostgreSQL connection pool, and Redis write capacity, rendering the API unavailable or severely degraded for legitimate users. Sensitive endpoints such as `POST /auth/login` and `POST /auth/reset-password` become open to unlimited OTP-spray attacks as a secondary consequence (note: `/auth/login` also applies `EmailThrottlerGuard` keyed on email, which provides partial mitigation for per-account brute-force, but the IP-level guard is still fully bypassed). The `POST /notifications/remind-signers` endpoint, which triggers email delivery, becomes an open email-spam vector.

**Impact: 3/5**

## Likelihood Explanation

No privileges, credentials, or special network position are required. Any HTTP client (`curl`, Python `requests`, etc.) can set arbitrary headers. The bypass is trivial to script and requires no knowledge of application internals. The `X-Forwarded-For` header is a standard HTTP header accepted by every web framework by default.

**Likelihood: 4/5**

## Recommendation

1. **Do not trust `x-forwarded-for` directly.** When `trust proxy` is enabled, Express already populates `req.ip` with the correct client IP extracted from the proxy chain. Remove the `x-forwarded-for` fallback and use only `req.ip`:
   ```typescript
   const clientIp = req.ip;
   ```
2. **If `x-forwarded-for` must be used**, extract only the last untrusted hop (rightmost IP added by the trusted proxy), not the full raw header value which is attacker-controlled.
3. **Consider registering `UserThrottlerGuard`** as an additional guard for authenticated endpoints to provide per-user rate limiting as a defense-in-depth layer.

## Proof of Concept

```bash
# Each request uses a different X-Forwarded-For value, generating a fresh throttle key
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://api.example.com/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$i" \
    -H "X-Frontend-Version: <valid_version>" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
# All requests return 200/401 instead of 429 â€” rate limit is never triggered
```
