### Title
`processTransactionSignatures` in `signers.service.ts` Accepts Cryptographically Invalid Signatures Without Verification, Allowing Signature Forgery

### Summary
The `uploadSignatureMaps` endpoint (`POST /transactions/:transactionId/signers`) processes signatures through `processTransactionSignatures`, which calls `sdkTransaction.addSignature()` without first calling `validateSignature`. The parallel code path `importSignatures` in `transactions.service.ts` explicitly calls `validateSignature` before `addSignature`, demonstrating that `addSignature` does not itself verify cryptographic validity. A malicious authenticated user with a registered key can submit arbitrary garbage bytes as a signature, be recorded as a legitimate signer, and corrupt the transaction state permanently.

### Finding Description

**Root cause:** `processTransactionSignatures` in `signers.service.ts` (lines 217–267) iterates over the submitted `SignatureMap`, checks only that each public key belongs to the authenticated user (line 248), then unconditionally calls `sdkTransaction.addSignature(publicKey, map)` (line 251) without any cryptographic verification of the signature bytes. [1](#0-0) 

By contrast, `importSignatures` in `transactions.service.ts` (lines 545–553) explicitly calls `validateSignature` before `addSignature`: [2](#0-1) 

`validateSignature` is defined in `back-end/libs/common/src/utils/sdk/transaction.ts` and is the only place where cryptographic correctness of the signature bytes is checked. Its absence in `processTransactionSignatures` means the signature bytes are never verified against the transaction body and the claimed public key.

**Exploit flow:**
1. Attacker authenticates as a normal user who has a registered `UserKey` in the system.
2. Attacker constructs a `SignatureMap` where the signature bytes for their public key are arbitrary garbage (e.g., 64 zero bytes).
3. Attacker POSTs this to `POST /transactions/:transactionId/signers`.
4. `processTransactionSignatures` finds the public key in `userKeyMap` (line 244–248), passes the check, calls `addSignature` with the invalid bytes (line 251), and returns `userKeys` containing the attacker's key.
5. `persistSignatureChanges` inserts a `TransactionSigner` row for the attacker (lines 302–309) and updates `transactionBytes` with the corrupted signature embedded (lines 296–298).
6. `updateStatusesAndNotify` calls `processTransactionStatus`, which may advance the transaction to `WAITING_FOR_EXECUTION` if the attacker's key was the last required signer.
7. The chain service picks up the transaction and submits it to the Hedera network. The network rejects it with `INVALID_SIGNATURE`. The transaction is permanently marked `FAILED`. [3](#0-2) 

### Impact Explanation
- **Permanent transaction corruption:** Once the transaction is submitted with an invalid signature and rejected by Hedera, it is marked `FAILED` and cannot be re-executed. Legitimate signers cannot recover it.
- **Signature forgery / signer impersonation:** The attacker is recorded in the `TransactionSigner` table as having signed the transaction, which is false. This corrupts the audit trail and approval workflow.
- **Multi-sig threshold bypass:** In a multi-sig scenario, the attacker's fraudulent signer record can satisfy a threshold, causing the transaction to advance to execution prematurely and then fail permanently on-chain.

### Likelihood Explanation
Any authenticated user with at least one registered `UserKey` who has access to a transaction (i.e., is an approver, observer, or creator) can trigger this. No privileged access is required. The attacker only needs to know the transaction ID and their own registered public key. The `POST /transactions/:transactionId/signers` endpoint is a standard, documented API endpoint protected only by JWT authentication. [4](#0-3) 

### Recommendation
Add a `validateSignature` call inside `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const { data: validatedKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

This ensures that only cryptographically valid signatures (i.e., signatures that actually verify against the transaction body under the claimed public key) are accepted and persisted. [5](#0-4) 

### Proof of Concept

1. Authenticate as a normal user with a registered `UserKey` (public key `PK`).
2. Obtain a transaction ID `T` for a transaction in `WAITING_FOR_SIGNATURES` status where `PK` is a required signer.
3. Construct a `SignatureMap` with the correct structure (node account ID → transaction ID → `PK` → 64 zero bytes as the signature).
4. POST to `POST /transactions/T/signers` with this map.
5. Observe HTTP 201 response. Check the database: a `TransactionSigner` row exists for the attacker's key, and `transactionBytes` now contains the zero-byte signature.
6. If `PK` was the last required signer, the transaction advances to `WAITING_FOR_EXECUTION`.
7. The chain service submits the transaction to Hedera. Hedera returns `INVALID_SIGNATURE`. The transaction is permanently `FAILED`.

The contrast with the `importSignatures` path (which would reject the same garbage signature at step 4 with `ErrorCodes.ISNMPN`) confirms the missing check. [2](#0-1)

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
