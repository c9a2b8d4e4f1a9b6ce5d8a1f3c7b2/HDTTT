Audit Report

## Title
JWT Token Not Blacklisted After Password Change Allows Continued Use of Compromised Tokens

## Summary
The `changePassword` endpoint (`PATCH /auth/change-password`) successfully changes a user's password but never invalidates the current JWT. The `BlacklistService` is available in `AuthController` and is correctly used in `logout` and the OTP flow, but is silently omitted from `changePassword`, leaving the old token fully valid for its entire remaining lifetime.

## Finding Description

`BlacklistService` in `back-end/libs/common/src/blacklist/blacklist.service.ts` provides `blacklistToken` (writes to Redis with `JWT_EXPIRATION`-day TTL) and `isTokenBlacklisted` (reads from Redis): [1](#0-0) 

The `logout` endpoint correctly blacklists the bearer token immediately after the operation: [2](#0-1) 

The OTP flow also correctly blacklists the OTP token in both `verifyOtp` and `setPassword`: [3](#0-2) 

However, `changePassword` only delegates to `authService.changePassword` and never calls `blacklistService.blacklistToken`: [4](#0-3) 

The underlying `AuthService.changePassword` also performs no token invalidation — it only validates the old password and updates the hash: [5](#0-4) 

The JWT is signed with an expiry of `JWT_EXPIRATION` days, and the blacklist TTL uses the same value — meaning a non-blacklisted token remains valid for the full configured window: [6](#0-5) [7](#0-6) 

## Impact Explanation
An attacker who obtains a valid JWT (via XSS, logging, network interception, or insider access) retains full API access to all endpoints guarded by `JwtBlackListAuthGuard` + `JwtAuthGuard` even after the legitimate user changes their password. The password change — the standard remediation action for a compromised account — provides no actual revocation of the stolen credential. The token remains usable until its natural expiry. [8](#0-7) 

## Likelihood Explanation
This is a realistic, low-complexity attack path. Token theft is a common threat vector. Users routinely change passwords specifically to revoke access after suspecting compromise. The gap between user intent ("I changed my password to lock out the attacker") and actual behavior ("the old token still works") is concrete and exploitable without any special privileges. The e2e test suite for `change-password` does not assert that the old token is invalidated after the operation, confirming the gap is untested: [9](#0-8) 

## Recommendation
In `AuthController.changePassword`, add a call to `this.blacklistService.blacklistToken(extractJwtAuth(req))` after the password change succeeds, mirroring the pattern used in `logout`. The `@Req() req: Request` parameter must also be added to the handler signature to extract the token. Example:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(
  @GetUser() user: User,
  @Body() dto: ChangePasswordDto,
  @Req() req: Request,
): Promise<void> {
  await this.authService.changePassword(user, dto);
  await this.blacklistService.blacklistToken(extractJwtAuth(req));
}
```

An e2e test should also be added to assert that the old token is rejected (401) after a successful password change.

## Proof of Concept

1. Attacker obtains victim's JWT `TOKEN_A` (e.g., via XSS or log leakage).
2. Victim notices the compromise and calls `PATCH /auth/change-password` with their old and new passwords — receives HTTP 200.
3. Attacker immediately calls any protected endpoint (e.g., `GET /transactions/history`) with `Authorization: Bearer TOKEN_A`.
4. `JwtBlackListAuthGuard` checks Redis — `TOKEN_A` is **not** in the blacklist (it was never added by `changePassword`).
5. `JwtAuthGuard` validates the JWT signature and expiry — both still valid.
6. The attacker's request succeeds with HTTP 200, despite the victim having changed their password.
7. This continues until `TOKEN_A` naturally expires after `JWT_EXPIRATION` days.

  async blacklistToken(jwt: string) {
    const expirationDays = this.configService.get<number>('JWT_EXPIRATION');
    const expirationSeconds = Number(expirationDays) * 24 * 60 * 60;

    await this.client.set(jwt, this.BLACKLISTED, 'EX', expirationSeconds);
  }

  async isTokenBlacklisted(jwt: string) {
    const data = await this.client.get(jwt);
    return data === this.BLACKLISTED;
  }
