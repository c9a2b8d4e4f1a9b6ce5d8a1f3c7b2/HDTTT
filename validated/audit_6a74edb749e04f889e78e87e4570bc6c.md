### Title
Unbounded Loop with Per-Item Deserialization and DB Queries in `getTransactionsToSign` Causes Server-Side DoS

### Summary

`GET /transactions/sign` fetches every non-terminal transaction from the database with no row limit, then synchronously iterates over the entire result set calling `userKeysToSign` per transaction. Each call deserializes transaction bytes twice and issues async account-cache and database queries. As the transaction table grows, a single authenticated request can exhaust the Node.js event loop and degrade the API for all users.

### Finding Description

**Root cause — missing DB-level pagination in `getTransactionsToSign`**

In `back-end/apps/api/src/transactions/transactions.service.ts`, the function fetches all non-terminal transactions without any `take` / `skip` constraint: [1](#0-0) 

It then iterates over the entire result set, calling `userKeysToSign` for every row: [2](#0-1) 

Pagination is applied only **after** the loop, on the already-computed in-memory array: [3](#0-2) 

**Cost of each loop iteration**

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign`:

1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive binary deserialization.
2. `transactionSignatureService.computeSignatureKey(transaction)` — which itself calls `SDKTransaction.fromBytes` a **second time** and issues async `accountCacheService.getAccountInfoForTransaction` calls (network/cache round-trips) for every signing account.
3. `entityManager.find(UserKey, ...)` — a database query per transaction. [4](#0-3) [5](#0-4) 

**Contrast with correctly paginated endpoints**

`getHistoryTransactions` and `getTransactions` both pass `skip: offset, take: limit` directly to the ORM query, so the database enforces the page size: [6](#0-5) 

`getTransactionsToSign` is the only endpoint that omits this.

**The endpoint is still live**

Despite the comment "NO LONGER USED BY FRONT-END", the route is registered and reachable by any authenticated user: [7](#0-6) 

No throttle guard is applied to this route — only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`.

**Exploit path**

1. Attacker registers as a normal user and obtains a valid JWT.
2. Attacker (or any set of users over time) creates many transactions via `POST /transactions`. Each transaction persists in the `transaction` table with a non-terminal status.
3. Attacker calls `GET /transactions/sign` (a single HTTP request). The server fetches every non-terminal transaction and runs O(N) deserialization + async I/O operations sequentially.
4. With a large N, the request takes seconds to minutes, blocking the Node.js event loop and starving other requests.
5. Repeated calls (or concurrent calls from multiple users) cause sustained service degradation or timeout failures for all API consumers.

### Impact Explanation

The API service becomes unresponsive or extremely slow for all users. Because `userKeysToSign` is `await`-ed sequentially inside the loop, the event loop is occupied for the entire duration of the request. This is a permanent degradation path: as the platform accumulates transactions (normal operational growth), the endpoint becomes progressively slower with no self-correcting mechanism. Critical workflows — signing, approving, executing transactions — share the same NestJS process and are affected.

### Likelihood Explanation

Any verified, authenticated user can trigger this. No privileged role is required. Transaction creation is a core, intended user action (`POST /transactions`), so accumulating a large transaction table is a natural consequence of normal platform use, not an exotic attack. The endpoint requires no special parameters and is reachable via a standard `GET` request.

### Recommendation

Apply DB-level pagination inside `getTransactionsToSign` the same way `getHistoryTransactions` does — pass `take: limit` and `skip: offset` to the `repo.find()` call. Because the filtering logic (which transactions require the user's keys) cannot be expressed as a pure SQL predicate with the current architecture, the correct fix is to either:

1. Rewrite the query as a SQL join against `transaction_signer` / `public_keys` columns (as the newer `transaction-nodes` service does with `getTransactionNodesQuery`), so the database filters rows before they reach the application layer.
2. Or, if the in-memory loop must be kept, cap the fetch with a hard maximum row limit and document that `totalItems` is an approximation.

Additionally, if the endpoint is truly no longer used by the front-end, it should be removed or protected with a stricter guard to eliminate the attack surface entirely.

### Proof of Concept

1. Register and verify a user account; obtain a JWT.
2. Create N transactions (e.g., N = 10,000) via repeated `POST /transactions` calls with valid transaction bytes.
3. Issue `GET /transactions/sign?page=1&size=10` with the JWT.
4. Observe: the server performs N iterations of `SDKTransaction.fromBytes` + `computeSignatureKey` + DB query before returning the first 10 results.
5. Monitor API response times for other endpoints during step 3 — they degrade proportionally to N. [8](#0-7)

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
