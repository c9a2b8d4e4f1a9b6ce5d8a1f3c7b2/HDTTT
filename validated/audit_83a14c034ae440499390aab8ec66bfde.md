### Title
Any Authenticated User Can Corrupt Another User's Transaction Group via Missing Upfront Ownership Check in `removeTransactionGroup`

---

### Summary

`TransactionGroupsService::removeTransactionGroup` deletes `TransactionGroupItem` records before verifying that the requesting user owns the group or its transactions. Because the per-transaction ownership check (`getTransactionForCreator`) is called only *after* the join-table row is already removed, any authenticated user can iteratively strip group items from another user's transaction group, permanently corrupting its structure.

---

### Finding Description

In `back-end/apps/api/src/transactions/groups/transaction-groups.service.ts`, `removeTransactionGroup` performs the following sequence:

```typescript
async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id }); // ← no user filter
    if (!group) { throw new Error('group not found'); }

    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
      relations: { group: true },
      where: { group: { id: group.id } },                                            // ← no user filter
    });

    for (const groupItem of groupItems) {
      const transactionId = groupItem.transactionId;
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);         // ← DELETE happens first
      await this.transactionsService.removeTransaction(transactionId, user, false);  // ← ownership checked here
    }
    await this.dataSource.manager.remove(TransactionGroup, group);
    return true;
}
``` [1](#0-0) 

The group is fetched with no user-ownership filter, and the `TransactionGroupItem` row is permanently deleted *before* `removeTransaction` calls `getTransactionForCreator` to verify ownership. [2](#0-1) 

When an attacker (who does not own the group) calls `DELETE /transaction-groups/:id`:

1. The group is found (no ownership gate).
2. All group items are fetched.
3. The **first** `TransactionGroupItem` is deleted from the database.
4. `removeTransaction` throws `UnauthorizedException` because the attacker is not the creator.
5. The loop aborts — but the group item is already gone.

By repeating this call, the attacker can delete every group item one at a time, leaving the `TransactionGroup` record orphaned with no items.

The `DELETE /transaction-groups/:id` endpoint is reachable by any verified, authenticated user:

```typescript
@Controller('transaction-groups')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
``` [3](#0-2) 

```typescript
@Delete('/:id')

  @GetUser() user: User,
  @Param('id', ParseIntPipe) groupId: number,
): Promise<boolean> {
  return this.transactionGroupsService.removeTransactionGroup(user, groupId);
}
``` [4](#0-3) 

Note that the `GET /transaction-groups/:id` endpoint *does* filter by user (via `getTransactionGroupItemsQuery`), so it cannot be used to enumerate foreign group IDs. However, group IDs are sequential integers, making them trivially guessable. [5](#0-4) 

---

### Impact Explanation

An attacker can permanently destroy the structural integrity of any other user's transaction group by repeatedly calling `DELETE /transaction-groups/:id`. Each call removes one `TransactionGroupItem` row. After all items are removed, the group is an empty shell. The underlying `Transaction` records are not deleted (because `removeTransaction` throws before completing), but the group-to-transaction linkage is severed. This constitutes irreversible data corruption and denial of service against the victim's multi-signature workflow.

---

### Likelihood Explanation

- The endpoint requires only a valid JWT and verified-user status — no elevated privilege.
- Group IDs are auto-incremented integers, trivially enumerable by any authenticated user.
- No rate-limiting is applied to `DELETE /transaction-groups/:id`.
- The attack requires only repeated HTTP calls with no special knowledge beyond a valid session token.

---

### Recommendation

Add an upfront ownership check before any destructive operation. The simplest fix mirrors the pattern used in `cancelTransactionGroup`: verify that the requesting user is the creator of all transactions in the group before touching any database rows.

```typescript
async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
    if (!group) { throw new Error('group not found'); }

    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
      relations: { group: true, transaction: { creatorKey: true } },
      where: { group: { id: group.id } },
    });

+   // Verify ownership before any mutation
+   const allOwnedByUser = groupItems.every(
+     item => item.transaction?.creatorKey?.userId === user.id,
+   );
+   if (!allOwnedByUser) {
+     throw new UnauthorizedException('Only the creator can remove a transaction group.');
+   }

    for (const groupItem of groupItems) {
      const transactionId = groupItem.transactionId;
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(transactionId, user, false);
    }
    await this.dataSource.manager.remove(TransactionGroup, group);
    return true;
}
```

This is consistent with the ownership check already present in `cancelTransactionGroup`: [6](#0-5)

### Proof of Concept

```
# Attacker is authenticated as user B (valid JWT, verified status).
# Victim (user A) owns transaction group with id=42.

# Step 1: Attacker calls DELETE on victim's group
PATCH DELETE /transaction-groups/42
Authorization: Bearer <attacker_token>

# Server behavior:
# - Finds group 42 (no ownership check)
# - Finds all group items
# - Deletes TransactionGroupItem for first transaction (e.g., transactionId=7)
# - Calls removeTransaction(7, userB, false) → throws UnauthorizedException
# - Response: 401 Unauthorized

# Step 2: Attacker repeats the call
DELETE /transaction-groups/42
Authorization: Bearer <attacker_token>

# - Finds group 42 (still exists)
# - Finds remaining group items (one fewer than before)
# - Deletes next TransactionGroupItem
# - Throws again

# After N calls (N = number of group items), all TransactionGroupItem rows are gone.
# TransactionGroup record still exists but is permanently orphaned.
# Victim's multi-signature workflow is destroyed.
```
