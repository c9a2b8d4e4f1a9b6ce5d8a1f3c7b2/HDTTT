Audit Report

## Title
Unbounded Signature Map Processing in `IsSignatureMap` Transform Enables CPU-Exhaustion DoS

## Summary
The `IsSignatureMap` decorator's `Transform` function iterates over all attacker-supplied signature map entries in a triple-nested loop, invoking expensive Hedera SDK operations (`AccountId.fromString`, `TransactionId.fromString`, `PublicKey.fromString`, `signatureMap.addSignature`) for every entry with no size check. Within the 2 MB body limit, a single authenticated user can pack thousands of entries into one request, causing significant CPU consumption before the request is rejected.

## Finding Description

**Entry points:**
- `POST /transactions/:transactionId/signers` → `SignersController.uploadSignatureMap`
- `POST /transactions/signatures/import` → `TransactionsController.importSignatures`

Both call `transformAndValidateDto(UploadSignatureMapDto, body)`.

**Step 1 — `transformAndValidateDto` has no array-length cap.**

`plainToInstance` triggers the `@IsSignatureMap()` `Transform` for every item with no count check: [1](#0-0) 

**Step 2 — `IsSignatureMap` Transform iterates without size guards.**

The triple-nested loop calls expensive SDK operations on every entry: [2](#0-1) 

Critically, `isAccountId` and `isTransactionId` (used in the validation helpers at lines 41 and 45) already call `AccountId.fromString` and `TransactionId.fromString` internally: [3](#0-2) 

Then the innermost loop calls them *again* at lines 56–59, meaning each entry triggers at least two `AccountId.fromString` calls, two `TransactionId.fromString` calls, one `PublicKey.fromString` call, and one `signatureMap.addSignature` allocation — all before any size check.

**Step 3 — The only size constraint is the 2 MB body limit.** [4](#0-3) 

Within 2 MB, reusing a single node account ID and transaction ID, an attacker can pack approximately ~10,000 key-signature pairs (~200 bytes each). The Transform calls `PublicKey.fromString()` ~10,000 times and `signatureMap.addSignature` ~10,000 times per request, all before any business-logic rejection.

## Impact Explanation
A single authenticated user can send one crafted 2 MB request and saturate a CPU core for the duration of the Transform processing. With `GLOBAL_MINUTE_LIMIT` set to 10,000 in production configs, an attacker can sustain near-continuous CPU load on the API service, degrading or denying service to all other users. [5](#0-4) 

## Likelihood Explanation
Any registered user can exploit this. The attacker needs only:
1. A valid JWT (obtained via normal login).
2. A crafted JSON body with thousands of valid-format ED25519 public keys under a single node account ID and transaction ID key.
3. A script to generate and send the 2 MB payload.

The Transform runs before any transaction lookup or ownership check, so even a non-existent transaction ID causes full processing. [6](#0-5) [7](#0-6) 

## Recommendation
Add a size guard at the start of the `IsSignatureMap` Transform before any iteration begins. For example, enforce a maximum number of node account IDs, transaction IDs per node, and public keys per transaction ID. A reasonable cap (e.g., 10 nodes × 10 transaction IDs × 50 keys = 5,000 entries maximum) would eliminate the attack surface while accommodating legitimate use cases. Additionally, consider adding an array-length cap in `transformAndValidateDto` to limit the number of DTO items processed per request. [2](#0-1) 

## Proof of Concept
```
POST /transactions/1/signers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "0.0.3": {
    "0.0.2@1234567890.000000000": {
      "<ed25519_pubkey_1>": "<base64_signature>",
      "<ed25519_pubkey_2>": "<base64_signature>",
      ... (repeat ~10,000 times to fill ~2 MB)
    }
  }
}
```
The `IsSignatureMap` Transform processes all ~10,000 entries — calling `AccountId.fromString`, `TransactionId.fromString`, `PublicKey.fromString`, and `signatureMap.addSignature` for each — before the request is rejected. Repeating this request continuously saturates the Node.js event loop thread.

export async function transformAndValidateDto<T extends object>(
  dtoClass: new (...args: any[]) => T,
  payload: T | T[],
): Promise<T[]> {
  const items = Array.isArray(payload) ? payload : [payload];
  const instances = items.map(item => plainToInstance(dtoClass, item));
  await Promise.all(instances.map(instance => validateOrReject(instance)));
  return instances;
