### Title
Unauthorized Approver Removal via IDOR in `removeTransactionApprover` Bypasses Approval Workflow

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but then removes the approver record identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who is the creator of any transaction can remove approvers from any other transaction by supplying a foreign approver ID, silently eliminating approval requirements on transactions they do not own.

### Finding Description

**Root cause — missing ownership check in `removeTransactionApprover`**

In `approvers.controller.ts` the handler performs two independent operations:

```typescript
// approvers.controller.ts lines 102-113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ← removes approver by id, no ownership check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the authenticated user is the creator of the URL-supplied `transactionId`. It says nothing about whether approver `id` belongs to that transaction. [2](#0-1) 

`removeTransactionApprover` in the service then fetches the approver by its primary key and deletes it (and its entire subtree) with no cross-check against the transaction in the URL:

```typescript
// approvers.service.ts lines 533-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);          // ← deletes any approver tree
  emitTransactionStatusUpdate(...);
  return result;
}
``` [3](#0-2) 

`removeNode` performs a recursive soft-delete of the entire approver subtree rooted at `id`, regardless of which transaction it belongs to. [4](#0-3) 

**Exploit path**

1. Attacker registers as a normal user and creates any transaction (becomes its creator — `transactionId = A`).
2. Attacker enumerates or guesses approver IDs (sequential integers in PostgreSQL). Approver IDs for victim transaction B are `id = X`.
3. Attacker calls `DELETE /transactions/A/approvers/X`.
4. `getCreatorsTransaction(A, attacker)` succeeds — attacker owns A.
5. `removeTransactionApprover(X)` soft-deletes approver X (and its subtree) from transaction B without any further check.
6. Transaction B's approval tree is now reduced or eliminated; the chain service's `processTransactionStatus` may now advance B to `WAITING_FOR_EXECUTION` without the required approvals. [5](#0-4) 

### Impact Explanation

Approval records gate whether a transaction advances from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION`. Removing an approver node eliminates that gating requirement. An attacker can silently strip approval requirements from any in-progress transaction they do not own, allowing those transactions (account updates, transfers, key changes, file operations, etc.) to be submitted to the Hedera network without the intended human oversight. This is an unauthorized state change with direct operational impact on the integrity of the multi-signature workflow.

### Likelihood Explanation

- **Attacker precondition**: only a valid authenticated account (any registered user). No admin or privileged role required.
- **Approver IDs**: auto-incremented integers, trivially enumerable by an attacker who creates their own approvers and observes the returned IDs.
- **No rate limiting** on the delete endpoint is visible in the codebase.
- The attack is silent — the victim transaction's creator receives a status-update notification but no indication that an external party removed the approver.

### Recommendation

In `removeTransactionApprover` (service), after fetching the approver, verify it belongs to the transaction supplied by the caller before deleting:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve the root node to get the owning transactionId
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, and update the controller accordingly.

### Proof of Concept

**Setup**: Two organization users — Alice (creator of transaction A) and Bob (creator of transaction B with approver record `id = 42`).

**Steps**:
1. Alice authenticates and obtains a JWT.
2. Alice creates transaction A (any valid transaction bytes). Server returns `{ id: 99 }`.
3. Alice observes that approver IDs are sequential; she creates a test approver on her own transaction and sees `id = 41`, inferring Bob's approver is `id = 42`.
4. Alice sends:
   ```
   DELETE /transactions/99/approvers/42
   Authorization: Bearer <alice_jwt>
   ```
5. Server responds `200 true`.
6. Bob's transaction B now has approver `42` soft-deleted. `processTransactionStatus` re-evaluates B and, if the threshold is now met without that approver, advances B to `WAITING_FOR_EXECUTION`.

**Expected (correct) behavior**: 403 Unauthorized — approver 42 does not belong to transaction 99.
**Actual behavior**: 200 OK — approver 42 is deleted from transaction B. [1](#0-0) [3](#0-2)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }

```typescript
export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }

  if (updatesByStatus.size > 0) {
    await Promise.all(
      Array.from(updatesByStatus.values()).map(async ({ newStatus, oldStatus, ids }) => {
        const result = await transactionRepo
          .createQueryBuilder()
          .update(Transaction)
          .set({ status: newStatus })
          .where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
          .returning('id')
          .execute();

        for (const row of result.raw) {
          statusChanges.set(row.id, newStatus);
        }
      })
    );
  }

  return statusChanges;
