### Title
Unbounded Full-Table Scan with Per-Row Async DB Queries in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`GET /transactions/sign` fetches every active transaction in the system without any database-level limit, then iterates over the entire result set calling an async cryptographic key-resolution function (plus additional DB queries) per transaction before applying pagination. A single authenticated user with one registered key can trigger this endpoint repeatedly to exhaust server CPU, memory, and database connections, degrading or crashing the API service.

### Finding Description

**Root cause — no `take`/`skip` on the initial DB fetch:**

In `getTransactionsToSign`, the repository query has no row limit:

```ts
const transactions = await this.repo.find({
  where: whereForUser,   // only excludes terminal statuses
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip`
});
```

The `whereForUser` filter only excludes `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `ARCHIVED` transactions. Every other active transaction in the entire system is loaded into memory. [1](#0-0) 

**Unbounded loop with per-iteration async work:**

After loading the full result set, the code iterates over every transaction and calls `userKeysToSign` for each one:

```ts
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

Each `userKeysToSign` call chains through `userKeysRequiredToSign` → `keysRequiredToSign`, which:
1. Deserializes the transaction bytes (`SDKTransaction.fromBytes`)
2. Calls `transactionSignatureService.computeSignatureKey` (cryptographic key tree traversal)
3. Issues a `entityManager.find(UserKey, ...)` database query [3](#0-2) 

**Pagination applied only after full scan:**

The `offset`/`limit` from the request are applied only after the entire loop completes:

```ts
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [4](#0-3) 

**Endpoint is still live:**

The controller exposes this as `GET /transactions/sign` with a comment acknowledging it is no longer used by the front-end, but it remains fully accessible to any authenticated user: [5](#0-4) 

### Impact Explanation

With N active transactions in the system, a single request causes:
- One full-table DB scan loading all N transaction rows + their `groupItem` relations into memory
- N sequential async operations each involving: transaction byte deserialization, cryptographic key tree computation, and a DB query for `UserKey` records

At scale (e.g., 10,000 active transactions), a single request can hold the Node.js event loop for seconds, exhaust the PostgreSQL connection pool, and consume gigabytes of heap. Repeated requests from one attacker can render the API service unresponsive for all users.

### Likelihood Explanation

The attacker preconditions are minimal:
1. Register an account (normal user flow)
2. Get verified (`VerifiedUserGuard`)
3. Register at least one key (`HasKeyGuard` is NOT on this endpoint; the early-return check `user.keys.length === 0` is bypassed as long as the user has one key)

Any legitimate organization member satisfies all three conditions. The endpoint requires no special privilege. The attack is a single HTTP GET request and can be repeated in a tight loop.

### Recommendation

Apply database-level pagination before the loop, or replace the approach entirely with a SQL-side filter that identifies transactions requiring the user's keys without loading all rows into application memory. At minimum, add a `take` cap on the initial `repo.find()` call and push the key-matching logic into the query layer (as the newer `transaction-nodes` endpoint already does via `getTransactionNodesQuery`).

### Proof of Concept

1. Register and verify a user account; register one key.
2. Ensure the system has a large number of active (non-terminal) transactions.
3. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <token>
   ```
4. Observe: each request loads all active transactions from the DB and executes one async DB+crypto call per transaction, regardless of the requested page size of 10. Server response time grows linearly with the number of active transactions. Under sustained load, the API becomes unresponsive.

**File:** back-end/libs/common/src/utils/transaction/index.ts (L20-94)
```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
```
