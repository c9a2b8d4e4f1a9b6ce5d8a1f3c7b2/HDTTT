### Title
Race Condition in `archiveTransaction` Allows Unconditional Status Overwrite, Corrupting Execution Audit Trail

### Summary
`archiveTransaction` in `transactions.service.ts` performs a status check followed by an **unconditional** database update (`repo.update({ id }, { status: ARCHIVED })`). This creates a TOCTOU race window with the chain service's `_executeTransaction`, which makes a long-running external call to the Hedera network before writing its own status update. A transaction creator can call `PATCH /transactions/archive/:id` while the chain service is mid-execution, causing the DB to record `ARCHIVED` over a freshly written `EXECUTED` status — or preventing the execution status from ever being written — while the transaction is irreversibly committed on Hedera.

### Finding Description

**Root cause — unconditional update in `archiveTransaction`:**

`back-end/apps/api/src/transactions/transactions.service.ts` lines 708–720:
```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);  // READ

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) && !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED }); // UNCONDITIONAL WRITE
```

The update carries **no `WHERE status = ...` guard**. Any status present in the DB at write time is silently overwritten.

**Contrast with the chain service's conditional update:**

`back-end/libs/common/src/execute/execute.service.ts` lines 187–196:
```typescript
const updateResult = await this.transactionsRepo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: transactionStatus, executedAt, statusCode: transactionStatusCode })
  .where('id = :id AND status = :currentStatus', {
    id: transaction.id,
    currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
  })
  .returning('id')
  .execute();

if (updateResult.raw.length === 0) return null;  // silently aborts if status changed
```

The chain service correctly guards its write, but this means it **silently loses** if `archiveTransaction` runs first.

**Race window — the Hedera network call:**

`back-end/libs/common/src/execute/execute.service.ts` lines 143–150:
```typescript
const response = await sdkTransaction.execute(client);   // external call — seconds of latency
const receipt = await response.getReceipt(client);       // second external call
```

Between `validateTransactionStatus` (which re-reads the DB and sees `WAITING_FOR_EXECUTION`) and the final conditional `UPDATE`, there is a multi-second window during which `archiveTransaction` can read the same `WAITING_FOR_EXECUTION` status, pass its guard, and commit an unconditional `ARCHIVED` write.

**Two concrete race scenarios:**

*Scenario A — archive wins the write race:*
1. Chain service reads `WAITING_FOR_EXECUTION`, passes `validateTransactionStatus`.
2. Chain service calls `sdkTransaction.execute(client)` — Hedera call in flight.
3. Creator calls `PATCH /archive/:id` → reads `WAITING_FOR_EXECUTION` → passes guard → writes `ARCHIVED`.
4. Chain service receives Hedera receipt → tries conditional update `WHERE status = WAITING_FOR_EXECUTION` → 0 rows affected → returns `null`.
5. **Result:** Transaction executed on Hedera; DB shows `ARCHIVED`; no execution notification emitted.

*Scenario B — execution wins the write race, archive overwrites:*
1. Chain service executes on Hedera, writes `EXECUTED` to DB.
2. Creator's concurrent `archiveTransaction` (which read `WAITING_FOR_EXECUTION` before step 1 completed) now writes `ARCHIVED` unconditionally, overwriting `EXECUTED`.
3. **Result:** DB shows `ARCHIVED`; Hedera shows executed; execution notification was emitted but is now contradicted by an archive notification.

The `@MurLock` on `executeTransaction` (`back-end/libs/common/src/execute/execute.service.ts` line 41) only serialises concurrent `executeTransaction` calls for the same transaction ID. It does **not** block `archiveTransaction`, which holds no lock.

### Impact Explanation

- **Audit trail corruption:** The organization's DB permanently records `ARCHIVED` for a transaction that was irreversibly executed on Hedera. Signers, approvers, and observers cannot determine the true outcome.
- **Suppressed execution notifications:** In Scenario A, `emitTransactionStatusUpdate` is never called for the execution event (`_executeTransaction` returns `null`). Downstream services and users are never informed the transaction executed.
- **False state in Scenario B:** An `EXECUTED` row is silently overwritten with `ARCHIVED`, contradicting the already-emitted execution notification and leaving the system in an inconsistent state.
- **No financial double-spend** (Hedera rejects duplicate transaction IDs), but the integrity of the organizational workflow and compliance record is broken.

### Likelihood Explanation

- **Attacker profile:** Any authenticated user who is the creator of a transaction — no privileged role required.
- **Trigger:** The race window is the duration of the Hedera network round-trip (typically 2–10 seconds). A malicious creator can poll transaction status, detect the `WAITING_FOR_EXECUTION` transition, and immediately call `PATCH /archive/:id`. This is a deliberate, scriptable action.
- **No special tooling required:** A standard HTTP client suffices.

### Recommendation

Replace the unconditional update in `archiveTransaction` with a conditional one that only succeeds if the status has not yet changed:

```typescript
// Instead of:
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });

// Use:
const result = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id AND status IN (:...statuses)', {
    id,
    statuses: [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION],
  })
  .execute();

if (!result.affected || result.affected === 0) {
  throw new ConflictException('Transaction state changed; cannot archive.');
}
```

This mirrors the pattern already used correctly in `cancelTransactionWithOutcome` and `_executeTransaction`.

### Proof of Concept

**Preconditions:** Authenticated user is the creator of a non-manual transaction currently in `WAITING_FOR_EXECUTION`.

**Steps:**
1. Subscribe to WebSocket notifications to observe status events.
2. Poll `GET /transactions/:id` until `status === "WAITING_FOR_EXECUTION"`.
3. Immediately send `PATCH /transactions/archive/:id`.
4. Observe:
   - API returns `200 true`.
   - DB record shows `status = ARCHIVED`.
   - Hedera Mirror Node confirms the transaction was executed (query by `transactionId`).
   - No `EXECUTED` status-update notification was emitted (Scenario A), or an `EXECUTED` notification was emitted followed by an `ARCHIVED` notification for the same transaction (Scenario B).
