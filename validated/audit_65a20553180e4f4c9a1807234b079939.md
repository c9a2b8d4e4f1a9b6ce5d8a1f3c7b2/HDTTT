### Title
Transaction Creator Can Mutate Approver Structure While Transaction Is Actively Being Signed, Invalidating Completed Approvals

### Summary
The transaction creator can call `updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover` at any time — including while a transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status — because none of these mutation paths check the transaction's current status before applying changes. A malicious creator can raise the approval threshold, replace a signed approver (which explicitly clears their signature), or add new required approvers after other users have already completed their approvals, permanently blocking execution or invalidating work already done by approvers.

### Finding Description

**Root cause:** `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` all gate access only on `getCreatorsTransaction` (verifying the caller is the creator), but perform **no check on the transaction's current status**. There is no guard preventing these mutations when the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

**Code path — `createTransactionApprovers`:** [1](#0-0) 

Only `getCreatorsTransaction` is called. No status check follows. The creator can add new required approvers at any point.

**Code path — `updateTransactionApprover`:** [2](#0-1) 

Again, only `getCreatorsTransaction` is called. No status check. The creator can then:

- **Raise the threshold** on a tree approver node: [3](#0-2) 

- **Replace a signed approver with a new user**, which explicitly clears the existing signature and approval: [4](#0-3) 

**Code path — `removeTransactionApprover` (controller):** [5](#0-4) 

No status check before removal. A creator can remove an approver who has already signed.

**Contrast with `approveTransaction`**, which correctly checks status before allowing an approver to act: [6](#0-5) 

The asymmetry is the root cause: approvers are restricted by status, but the creator's mutation endpoints are not.

### Impact Explanation

**Scenario A — Threshold raise after approval threshold is met:**
1. Creator creates a transaction with a threshold tree: `threshold=1`, approvers `[Alice, Bob]`.
2. Alice approves. The 1-of-2 threshold is satisfied; the transaction should proceed to execution.
3. Before the chain service executes, the creator calls `PATCH /transactions/:id/approvers/:treeId` with `{ threshold: 2 }`.
4. Now 2-of-2 approvals are required. Only Alice has signed. The transaction is stuck in `WAITING_FOR_SIGNATURES` indefinitely unless Bob also approves.
5. The creator can repeat this pattern to permanently block execution.

**Scenario B — Approver replacement to invalidate a completed signature:**
1. Creator creates a transaction with `threshold=1`, approver `[Alice]`.
2. Alice reviews and approves (signs).
3. Creator calls `PATCH /transactions/:id/approvers/:aliceApproverId` with `{ userId: Charlie }`.
4. The service explicitly sets `signature: undefined, approved: undefined` on the approver record, erasing Alice's completed approval.
5. Charlie has not signed. The transaction reverts to needing a new approval.

**Scenario C — Add new required approvers after existing approvers have signed:**
1. Creator creates a transaction with `threshold=1`, approver `[Alice]`.
2. Alice approves.
3. Creator calls `POST /transactions/:id/approvers` to add a new root-level approver `Bob`.
4. Now both Alice's and Bob's approvals are independently required. Bob has not signed. Execution is blocked.

The impact is **permanent denial of execution** for a transaction that legitimate approvers have already completed their obligations on, and **invalidation of approver work** (wasted effort, potential gas/fee costs in integrations, and broken organizational workflows).

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The attack requires only:
1. Creating a transaction (normal user action).
2. Waiting for approvers to sign (observable via the notification/status system).
3. Calling a standard PATCH or POST endpoint with a valid JWT.

No leaked credentials, no admin access, and no cryptographic break are required. The API endpoints are fully reachable by any authenticated user who is the creator of a transaction. The attack is deterministic and repeatable.

### Recommendation

Add a status guard at the start of `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` (or in `getCreatorsTransaction`) that rejects mutations when the transaction is in any active signing state:

```typescript
const MUTABLE_STATUSES = [TransactionStatus.WAITING_FOR_SIGNATURES];

if (!MUTABLE_STATUSES.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify approvers: transaction is not in a mutable state');
}
```

Alternatively, restrict mutations to only the `NEW` or `DRAFT` status (before the transaction has been sent out for signing), which is the most conservative and correct invariant: approver structure should be frozen once signing begins.

### Proof of Concept

**Setup:**
- Creator (Mallory) creates transaction `T` with a threshold approver tree: `threshold=1`, approvers `[Alice, Bob]`.
- Alice is notified and reviews the transaction.

**Step 1 — Alice approves:**
```
POST /transactions/T/approvers/approve
Authorization: Bearer <alice-token>
{ "userKeyId": 1, "signature": "<valid-sig>", "approved": true }
```
Alice's approver record now has `signature` set. The 1-of-2 threshold is met.

**Step 2 — Mallory raises the threshold before execution:**
```
PATCH /transactions/T/approvers/<tree-approver-id>
Authorization: Bearer <mallory-token>
{ "threshold": 2 }
```
`updateTransactionApprover` calls `getCreatorsTransaction` (passes — Mallory is creator), performs no status check, and updates `threshold` to `2`.

**Step 3 — Result:**
The transaction now requires 2-of-2 approvals. Only Alice has signed. Bob has not. The transaction is stuck in `WAITING_FOR_SIGNATURES`. Mallory can repeat step 2 indefinitely to prevent execution even if Bob eventually signs (by adding more approvers via `POST`).

**Alternatively (Scenario B):**

**Step 2b — Mallory replaces Alice's approver slot:**
```
PATCH /transactions/T/approvers/<alice-approver-id>
Authorization: Bearer <mallory-token>
{ "userId": <charlie-id> }
```
The service sets `signature: undefined, approved: undefined` on Alice's record and assigns it to Charlie. Alice's completed approval is erased. Charlie has not signed. The transaction is back to `WAITING_FOR_SIGNATURES`.
