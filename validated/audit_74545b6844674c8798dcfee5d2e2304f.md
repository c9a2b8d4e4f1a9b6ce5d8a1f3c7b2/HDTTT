### Title
`archiveTransaction` Bypasses Status Guard for Manual Transactions, Allowing State Corruption from Terminal States

### Summary
The `archiveTransaction` method in `back-end/apps/api/src/transactions/transactions.service.ts` contains a flawed guard condition that uses a logical `&&` with `!transaction.isManual`. When `isManual` is `true`, the status check is short-circuited entirely, allowing any authenticated creator to archive a manual transaction regardless of its current status ŌĆö including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `REJECTED`. This is a direct analog of the external report's state-transition class: a state-mutating action executes in states where it must be forbidden.

### Finding Description

**Root cause ŌĆö flawed guard logic:**

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts, lines 711ŌĆō718
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ŌåÉ short-circuits the entire check when isManual=true
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}

await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

The guard only throws when **both** sub-conditions are true:
- `status NOT IN [WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- `isManual === false`

When `isManual === true`, the second operand is `false`, so the `&&` evaluates to `false` regardless of status. The throw is never reached. The unconditional `repo.update` on line 720 then overwrites the transaction's status to `ARCHIVED`.

**Reachable entry path:**

The endpoint is publicly reachable by any authenticated, verified user who is the creator of the transaction:

```
PATCH /transactions/archive/:id
ŌåÆ TransactionsController.archiveTransaction (line 265)
ŌåÆ TransactionsService.archiveTransaction (line 708)
```

`isManual` is a creator-controlled flag set at transaction creation time via `CreateTransactionDto.isManual`. Any user can create a manual transaction.

**Affected states:** `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `REJECTED`, and even `ARCHIVED` (re-archiving).

**Existing test gap:** The e2e test at `back-end/apps/api/test/spec/transaction.e2e-spec.ts` line 884 only tests archiving an `EXECUTED` non-manual transaction and correctly expects a 400. It does not test the `isManual: true` + terminal-status combination, so the bug is not caught.

### Impact Explanation

1. **Audit trail corruption:** A creator can archive an `EXECUTED` manual transaction, removing it from the execution history. The `ARCHIVED` status is treated as a terminal "sign-only" state, not an execution record, so the on-chain execution is effectively hidden from the backend's history view.
2. **Terminal state overwrite:** `CANCELED`, `REJECTED`, `FAILED`, and `EXPIRED` are all terminal states with distinct semantics. Overwriting any of them with `ARCHIVED` corrupts the integrity of the transaction lifecycle state machine.
3. **Notification side-effect:** `emitTransactionStatusUpdate` is called unconditionally after the update (line 721), broadcasting a false status change to all subscribers.

### Likelihood Explanation

- **Attacker preconditions:** Only requires a normal authenticated user account (no admin/privileged role). The attacker must be the creator of a manual transaction, which is a standard product workflow.
- **Trigger:** Create a transaction with `isManual: true`, wait for it to reach any terminal state (or execute it), then call `PATCH /transactions/archive/:id`.
- **No special tooling required:** A direct HTTP call to the archive endpoint suffices.

### Recommendation

Replace the compound `&&` guard with an explicit allowlist check that is independent of `isManual`:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  const archivableStatuses = [
    TransactionStatus.WAITING_FOR_SIGNATURES,
    TransactionStatus.WAITING_FOR_EXECUTION,
  ];

  if (!archivableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  // ...
}
```

The `isManual` flag controls whether a transaction requires manual submission; it must not gate the status-validity check for archiving.

### Proof of Concept

1. Authenticate as any verified user and obtain a JWT.
2. Create a manual transaction:
   ```
   POST /transactions
   { ..., isManual: true }
   ŌåÆ returns { id: 42, status: "WAITING FOR SIGNATURES", isManual: true }
   ```
3. Force the transaction into a terminal state (e.g., directly update DB to `EXECUTED`, or let the scheduler execute it):
   ```sql
   UPDATE transaction SET status = 'EXECUTED' WHERE id = 42;
   ```
4. Call the archive endpoint as the creator:
   ```
   PATCH /transactions/archive/42
   Authorization: Bearer <creator_jwt>
   ŌåÆ HTTP 200, returns true
   ```
5. Verify the transaction is now `ARCHIVED` despite having been `EXECUTED`:
   ```sql
   SELECT status FROM transaction WHERE id = 42;
   -- Returns: ARCHIVED   ŌåÉ terminal state overwritten
   ```
