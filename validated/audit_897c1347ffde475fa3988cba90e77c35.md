### Title
Stale In-Memory Transaction Status Causes Newly-Promoted Transactions to Be Permanently Missed by Execution Scheduler

### Summary
`updateTransactions()` computes a `results` map of status changes via `processTransactionStatus()` and persists them to the database, but never applies those changes back to the in-memory `transactions` array it returns. `prepareTransactions()` then evaluates `transaction.status` on the stale in-memory objects, causing transactions just promoted from `WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION` to be skipped for execution scheduling. In the worst case, when a transaction's `validStart` is within 10 seconds of the 3-minute expiry boundary, the transaction is permanently marked `EXPIRED` instead of being executed — an unrecoverable state.

### Finding Description

**Root cause — `updateTransactions` discards the computed status-change map:**

In `back-end/apps/chain/src/transaction-scheduler/transaction-scheduler.service.ts`, `updateTransactions()` calls `processTransactionStatus()`, which updates statuses in the database and returns a `Map<number, TransactionStatus>` of what changed. That map is used only to emit notifications, then discarded. The original `transactions` array — with its pre-update in-memory `.status` values — is returned unchanged. [1](#0-0) 

**Downstream consumer reads stale status:**

`prepareTransactions()` iterates the returned array and gates all execution scheduling on `transaction.status === TransactionStatus.WAITING_FOR_EXECUTION`. Because the in-memory status was never updated, any transaction promoted to `WAITING_FOR_EXECUTION` in this cycle still shows `WAITING_FOR_SIGNATURES` in memory and is silently skipped. [2](#0-1) 

**`processTransactionStatus` never mutates the passed-in objects:**

The function builds DB updates and returns a `Map`, but never writes back to the `transaction.status` fields of the objects in the array. [3](#0-2) 

**Callers that invoke `prepareTransactions`:**

Only two cron handlers call `prepareTransactions`: `handleInitialTransactionStatusUpdate` (fires once on startup) and `handleTransactionsBetweenNowAndAfterThreeMinutes` (fires every 10 seconds). Both pass the stale array. [4](#0-3) 

**Concurrent expiry cron closes the window permanently:**

`handleExpiredTransactions` runs every 10 seconds and marks any transaction with `validStart < threeMinutesBefore` as `EXPIRED`. If a transaction's `validStart` is within the last 10 seconds of the 3-minute execution window when it is promoted, the next cron tick will expire it before `prepareTransactions` can schedule it. [5](#0-4) 

### Impact Explanation

A fully-signed transaction whose `validStart` falls within the 10-second window before the 3-minute expiry boundary will be:
1. Promoted to `WAITING_FOR_EXECUTION` in the DB by `processTransactionStatus`.
2. Skipped by `prepareTransactions` (stale in-memory status still shows `WAITING_FOR_SIGNATURES`).
3. Marked `EXPIRED` by `handleExpiredTransactions` in the next 10-second tick.

The transaction is then permanently unexecutable — the Hedera network will reject it as expired, and the system has no retry path for `EXPIRED` transactions. This is an unrecoverable corruption of user state.

### Likelihood Explanation

The `handleTransactionsBetweenNowAndAfterThreeMinutes` and `handleExpiredTransactions` crons both fire every 10 seconds. Any transaction whose `validStart` is between 2 minutes 50 seconds and 3 minutes before the current time at the moment it first accumulates enough signatures is vulnerable. This is a realistic scenario for any organization running time-sensitive multi-signature workflows where the final signature arrives near the end of the validity window.

### Recommendation

After `processTransactionStatus` returns, apply the computed status changes back to the in-memory objects before returning from `updateTransactions`, so that `prepareTransactions` operates on current state:

```typescript
// In updateTransactions(), after processTransactionStatus:
const results = await processTransactionStatus(..., transactions);

// Apply status changes back to in-memory objects
for (const transaction of transactions) {
  const newStatus = results.get(transaction.id);
  if (newStatus !== undefined) {
    transaction.status = newStatus;
  }
}

if (results.size > 0) { ... }
return transactions;
```

### Proof of Concept

1. Create a multi-signature transaction with `validStart` set to approximately 2 minutes 52 seconds in the future.
2. Wait until `validStart` is 2 minutes 52 seconds in the past (i.e., 8 seconds before the 3-minute expiry boundary).
3. Submit the final required signature so the transaction becomes fully signed.
4. Observe: `handleTransactionsBetweenNowAndAfterThreeMinutes` fires, `processTransactionStatus` promotes the transaction to `WAITING_FOR_EXECUTION` in the DB, but `prepareTransactions` reads the stale in-memory status (`WAITING_FOR_SIGNATURES`) and skips scheduling.
5. 10 seconds later: `handleExpiredTransactions` fires and marks the transaction `EXPIRED` (`validStart` is now 3 minutes 2 seconds in the past).
6. The transaction is never executed despite having all required signatures. Its final state is `EXPIRED` with no recovery path.

  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },

            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);

export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }

  if (updatesByStatus.size > 0) {
    await Promise.all(
      Array.from(updatesByStatus.values()).map(async ({ newStatus, oldStatus, ids }) => {
        const result = await transactionRepo
          .createQueryBuilder()
          .update(Transaction)
          .set({ status: newStatus })
          .where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
          .returning('id')
          .execute();

        for (const row of result.raw) {
          statusChanges.set(row.id, newStatus);
        }
      })
    );
  }

  return statusChanges;
