### Title
Creator of Any Transaction Can Remove Approvers from Any Other Transaction, Bypassing Approval Requirements

### Summary

In `approvers.controller.ts`, the `removeTransactionApprover` endpoint verifies the requesting user is the creator of the `transactionId` supplied in the URL, but then deletes the approver identified by a separate `id` parameter without verifying that approver actually belongs to that transaction. Any authenticated user who has created at least one transaction can exploit this decoupling to remove approvers from any other transaction in the system, bypassing the intended approval threshold enforcement.

### Finding Description

The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the transaction identified by the URL's `transactionId`.
2. `removeTransactionApprover(id)` — soft-deletes the approver row identified by `id`. [1](#0-0) 

There is no check that the approver `id` belongs to `transactionId`. The service-level `removeTransactionApprover` only verifies the approver exists; it performs no ownership or transaction-membership validation: [2](#0-1) 

The underlying `removeNode` function executes a recursive SQL delete on the approver tree rooted at the supplied `id`, again with no transaction-membership guard: [3](#0-2) 

**Exploit path:**

1. Attacker registers as a normal user and creates any transaction (becomes its creator).
2. Attacker enumerates or brute-forces approver IDs (sequential integers) belonging to a target transaction they wish to manipulate.
3. Attacker calls `DELETE /transactions/{attacker_own_tx_id}/approvers/{target_approver_id}`.
4. The authorization check passes (attacker is creator of their own transaction).
5. The target approver is deleted from the target transaction.
6. Attacker repeats until all approvers of the target transaction are removed.
7. `emitTransactionStatusUpdate` fires, triggering re-evaluation of the target transaction's status. With no approvers remaining, the transaction may advance to `WAITING_FOR_EXECUTION` without any approval having been collected. [4](#0-3) 

### Impact Explanation

The approval tree is the sole organizational gate preventing unauthorized Hedera transactions from being submitted. Removing all approvers from a target transaction eliminates that gate. The transaction can then be scheduled for execution and submitted to the Hedera network without any required organizational approvals, resulting in unauthorized on-chain state changes (token transfers, account updates, file writes, etc.). This is a direct integrity failure in the multi-signature approval workflow.

### Likelihood Explanation

- **Precondition**: The attacker must be an authenticated user with the ability to create at least one transaction — a baseline capability for any organization member.
- **Approver ID discovery**: Approver IDs are sequential database integers. An attacker can enumerate them by creating their own approvers and observing the returned IDs, then probing nearby values. No privileged access is required.
- **No rate-limit or anomaly detection** is visible in the codebase for this endpoint.
- The attack is fully automatable via repeated HTTP DELETE requests.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the `transactionId` supplied in the URL:

```typescript
// In removeTransactionApprover or the controller:
const approver = await this.getTransactionApproverById(id);
if (!approver) throw new BadRequestException(ErrorCodes.ANF);

// Resolve the root node to get the owning transactionId
const root = await this.getRootNodeFromNode(approver.id);
if (root?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

Additionally, add a transaction-status guard to prevent approver removal once a transaction has advanced past `WAITING_FOR_SIGNATURES`.

### Proof of Concept

```
# Step 1: Attacker creates their own transaction → receives transaction id = 1
POST /transactions  { ... }

# Step 2: Target transaction id = 42 has approvers with ids 100, 101, 102
# (discovered by brute-force or side-channel)

# Step 3: Attacker removes each approver using their own transactionId in the URL
DELETE /transactions/1/approvers/100   → 200 OK (creator check passes for tx 1)
DELETE /transactions/1/approvers/101   → 200 OK
DELETE /transactions/1/approvers/102   → 200 OK

# Result: Transaction 42 now has zero approvers.
# emitTransactionStatusUpdate fires → status re-evaluated → may advance to WAITING_FOR_EXECUTION
# Transaction 42 can now be executed without any organizational approval.
```

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

    // notifyTransactionAction(this.notificationsService);
  }

  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
