### Title
Non-Atomic Duplicate Check in `createTransactions` Allows Concurrent Duplicate Transaction Records

### Summary
The `createTransactions` function in `transactions.service.ts` performs a duplicate-existence check outside the database write transaction, with no database-level unique constraint on `transactionId`. Two concurrent authenticated requests carrying identical transaction bytes will both pass the check and both insert, producing two active records for the same Hedera transaction ID. This is the TypeScript analog of the external report's "default state == initiated state" class: the guard that is supposed to prevent duplicate creation is structurally unable to do so under concurrency.

### Finding Description
**Root cause — non-atomic check-then-act with no DB uniqueness enforcement**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `createTransactions` first queries for existing records:

```typescript
// lines 415-433 — outside any DB transaction
const existing = await this.repo.find({
  where: {
    transactionId: In(transactionIds),
    status: Not(In([CANCELED, REJECTED, ARCHIVED])),
  },
  select: ['transactionId'],
});
if (existing.length > 0) throw new BadRequestException(...);
```

Then, in a separate step, it opens a DB transaction and inserts:

```typescript
// lines 436-462 — separate DB transaction
const savedTransactions = await this.entityManager.transaction(async (em) => {
  return await em.save(Transaction, transactions);
});
```

The `find` and the `save` are **not inside the same database transaction**. There is no `UNIQUE` constraint on the `transactionId` column in the schema (confirmed: the `Transaction` entity has no `@Unique()` decorator, and the migration creates the column as `"transactionId" character varying NOT NULL` with only a primary-key constraint). Two concurrent requests that arrive between the `find` and the `save` of each other will both see an empty result from the check and both proceed to insert.

**Exploit flow**
1. Authenticated user (or a script) fires two simultaneous `POST /transactions` requests with the same serialized Hedera transaction bytes.
2. Both requests execute `repo.find(...)` concurrently; neither finds an existing record.
3. Both proceed to `entityManager.transaction(...)` and both call `em.save(Transaction, ...)`.
4. Two rows with identical `transactionId` and status `WAITING FOR SIGNATURES` are committed.
5. Signers and observers see two separate transaction objects for the same underlying Hedera transaction.
6. Both can accumulate signatures independently; both can be submitted to the Hedera network.
7. The first submission succeeds; the second returns `DUPLICATE_TRANSACTION` (status code handled in `execute.service.ts` lines 168–185), but the duplicate DB record remains active and continues to consume signer attention and notification bandwidth.

### Impact Explanation
- **Duplicate active transaction records** for the same Hedera transaction ID persist in the database.
- **Signers are misled**: they may sign both copies, wasting key operations and generating spurious notifications.
- **Observers and approvers** receive duplicate notification events for the same logical transaction.
- **Execution confusion**: the chain service's scheduler (`updateTransactions`) will pick up both records and attempt to submit both; the second submission fails silently with `isDuplicate = true` and returns `null`, but the duplicate DB row is never cleaned up — it stays in `WAITING_FOR_EXECUTION` until it expires.
- No direct loss of Hedera funds, but operational integrity of the multi-signature workflow is broken.

### Likelihood Explanation
Any authenticated user can trigger this with two concurrent HTTP requests — no privileged access required. It can occur accidentally (rapid double-submit from the UI) or deliberately (scripted concurrent POST). The window between the `find` and the `save` is wide enough (network I/O, validation, `getClientFromNetwork`) to be reliably hit.

### Recommendation
Apply **both** of the following fixes:

1. **Database-level unique constraint** on `transactionId` (scoped to non-terminal statuses, or unconditionally with application-level handling of the constraint violation):
   ```sql
   CREATE UNIQUE INDEX uq_transaction_transactionId
     ON "transaction" ("transactionId")
     WHERE status NOT IN ('CANCELED', 'REJECTED', 'ARCHIVED');
   ```
   This makes the database the authoritative guard, regardless of application-level race conditions.

2. **Move the existence check inside the DB transaction** so the read and write are atomic:
   ```typescript
   const savedTransactions = await this.entityManager.transaction(async (em) => {
     const existing = await em.find(Transaction, { where: { transactionId: In(ids), status: Not(In([...])) } });
     if (existing.length > 0) throw new BadRequestException(...);
     return await em.save(Transaction, transactions);
   });
   ```
   Use `SERIALIZABLE` isolation or a `SELECT ... FOR UPDATE` / advisory lock if the DB partial-unique index is not added.
