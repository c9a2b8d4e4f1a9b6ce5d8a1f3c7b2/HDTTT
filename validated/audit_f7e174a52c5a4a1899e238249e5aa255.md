Audit Report

## Title
Unbounded Array Input in Batch Signature Endpoints Enables Authenticated DoS

## Summary
The `importSignatures` (`POST /transactions/signatures/import`) and `uploadSignatureMap` (`POST /transactions/:transactionId?/signers`) endpoints accept arrays of `UploadSignatureMapDto` objects with no enforced upper bound on array length. The shared `transformAndValidateDto` utility imposes no size limit, and the service layer performs per-item cryptographic operations and database work proportional to input size. A developer comment in the code explicitly acknowledges the missing API-side guard.

## Finding Description

**`transformAndValidateDto` — no array size check:**

The shared helper at `back-end/libs/common/src/dtos/index.ts` lines 13–21 normalises any input into an array and validates each item, but applies no upper bound on the number of items: [1](#0-0) 

**Both controllers pass the raw body directly through this helper:**

`TransactionsController.importSignatures` at `back-end/apps/api/src/transactions/transactions.controller.ts` lines 96–107: [2](#0-1) 

`SignersController.uploadSignatureMap` at `back-end/apps/api/src/transactions/signers/signers.controller.ts` lines 102–119: [3](#0-2) 

**Per-item cryptographic work in `importSignatures`:**

For every element in the array, `TransactionsService.importSignatures` calls `verifyAccess` (async DB traversal), `SDKTransaction.fromBytes` (protobuf deserialization), `isExpired` (timestamp check), `validateSignature` (cryptographic verification), and `sdkTransaction.addSignature` (cryptographic operation): [4](#0-3) 

**Developer-acknowledged missing guard:**

The `BATCH_SIZE = 500` constant at line 575–576 only controls how many rows are written per SQL statement — it does **not** cap the total number of items accepted from the caller. The comment on line 575 explicitly acknowledges the gap: [5](#0-4) 

**`uploadSignatureMaps` follows the same pattern:**

`SignersService.validateAndProcessSignatures` calls `Promise.all` over all DTO items simultaneously, spawning parallel cryptographic verifications with no input-count guard: [6](#0-5) 

**No throttle guards on the affected controllers:**

Both controllers carry only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. The IP and user throttler modules exist but are not applied to these endpoints — no `@Throttle` decorator or `ThrottlerGuard` is present in any transactions or signers controller file: [7](#0-6) [8](#0-7) 

## Impact Explanation
A single authenticated HTTP POST containing a large JSON array forces the server to deserialize N protobuf blobs, perform N cryptographic signature verifications, and issue batched SQL `CASE` statements proportional to N. Because `validateAndProcessSignatures` uses `Promise.all`, all items are processed concurrently, amplifying CPU and memory pressure. This can exhaust the Node.js event loop, saturate the PostgreSQL connection pool, and cause out-of-memory conditions, rendering the API unavailable for all other users for the duration of the request.

## Likelihood Explanation
Any email-verified user account is sufficient to reach both endpoints — no elevated role is required. The attack requires only a single HTTP POST with a large JSON array body, which is trivially scriptable. The developer's own comment at line 575 of `transactions.service.ts` ("probably should limit this on the api side of things") confirms awareness that the guard is absent. The throttler modules that exist operate on request count per time window, not on payload size or array length within a single request.

## Recommendation
1. **Enforce an array length cap at the controller layer** before calling `transformAndValidateDto`. Reject requests exceeding a configurable maximum (e.g., 100 items) with HTTP 400.
2. **Apply the existing throttler guards** (`UserThrottlerGuard` / `IpThrottlerGuard`) to the transactions and signers controllers.
3. **Configure an explicit HTTP body size limit** in `setup-app.ts` to prevent oversized payloads from reaching application logic.
4. **Process items sequentially or in bounded concurrency** rather than unbounded `Promise.all` in `validateAndProcessSignatures`.

## Proof of Concept
```http
POST /transactions/signatures/import HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1700000000.000000000": { "<pubkey_hex>": "<sig_hex>" } } } },
  { "id": 2, "signatureMap": { ... } },
  ... (repeated N times, bounded only by the HTTP body size limit)
]
```
Each item causes `SDKTransaction.fromBytes`, `validateSignature`, and `addSignature` to execute. With N items, the server performs N parallel cryptographic operations and N database accesses before any response is returned, blocking the event loop for the duration.

export async function transformAndValidateDto<T extends object>(
  dtoClass: new (...args: any[]) => T,
  payload: T | T[],
): Promise<T[]> {
  const items = Array.isArray(payload) ? payload : [payload];
  const instances = items.map(item => plainToInstance(dtoClass, item));
  await Promise.all(instances.map(instance => validateOrReject(instance)));
  return instances;
}
