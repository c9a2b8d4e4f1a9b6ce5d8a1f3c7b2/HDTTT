Audit Report

## Title
Unbounded `reminderMillisecondsBefore` Causes Immediate Reminder Notification Bypass via Arithmetic Underflow in `createTransactions`

## Summary
`CreateTransactionDto.reminderMillisecondsBefore` accepts any numeric value with no upper or lower bound validation. In `transactions.service.ts`, the expression `tx.validStart.getTime() - dto.reminderMillisecondsBefore` is computed without guards. Supplying a value larger than `validStart.getTime()` produces a negative Unix timestamp, which Redis's `PXAT` option treats as already-expired, causing the reminder notification to fire immediately upon transaction creation â€” bypassing the intended "remind N ms before valid start" semantics entirely.

## Finding Description

**Root cause â€” missing bounds on DTO field:**

`CreateTransactionDto` declares `reminderMillisecondsBefore` with only `@IsOptional()` and `@IsNumber()`. No `@Min(1)` or `@Max()` constraint exists: [1](#0-0) 

**Unchecked arithmetic in service:**

In `createTransactions`, the reminder date is computed directly from user-supplied input with no bounds check: [2](#0-1) 

If `dto.reminderMillisecondsBefore` > `tx.validStart.getTime()` (e.g., attacker sends `9007199254740991` â€” `Number.MAX_SAFE_INTEGER`), then `remindAt` becomes a `Date` with a large negative millisecond value â€” a timestamp far in the past.

**Scheduler stores the poisoned date in Redis:**

`SchedulerService.addReminder` passes `date.getTime()` directly as the Redis `PXAT` argument: [3](#0-2) 

Redis `PXAT` interprets the value as a Unix timestamp in milliseconds. A negative or past timestamp causes the key to expire immediately, firing the keyspace notification at once.

**Immediate notification dispatch:**

`ReminderHandlerService.handleTransactionReminder` is invoked by the keyspace event. It checks transaction status and deduplication, then calls `emitTransactionRemindSigners` â€” sending a "please sign" notification to all signers the moment the transaction is created, not at the intended reminder time: [4](#0-3) 

The deduplication check (lines 51â€“58) only prevents a *second* reminder for the same transaction, not the initial spurious one triggered by the crafted timestamp.

## Impact Explanation
Every signer assigned to any transaction created with a crafted `reminderMillisecondsBefore` receives an immediate, unsolicited "sign this transaction" notification. At scale, an attacker can create many transactions in rapid succession, generating a notification storm that degrades the notification system and desensitizes signers to legitimate reminders â€” undermining the integrity of the multi-signature workflow.

## Likelihood Explanation
Any authenticated organization member can reach `POST /transactions` with a crafted body. No privilege escalation is required. The API accepts arbitrary numeric values for `reminderMillisecondsBefore` because the DTO only enforces `@IsNumber()`. The attack requires no special tooling â€” a single crafted HTTP request suffices.

## Recommendation
Add a `@Min(1)` and a reasonable `@Max()` (e.g., `@Max(365 * 24 * 60 * 60 * 1000)` for one year) constraint to `reminderMillisecondsBefore` in `CreateTransactionDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction.dto.ts
@IsOptional()
@IsNumber()
@Min(1)
@Max(365 * 24 * 60 * 60 * 1000) // max 1 year in ms
reminderMillisecondsBefore?: number;
```

Additionally, add a server-side guard in `createTransactions` before computing `remindAt` to ensure the resulting timestamp is in the future:

```typescript
const remindAt = new Date(tx.validStart.getTime() - dto.reminderMillisecondsBefore);
if (remindAt.getTime() <= Date.now()) {
  // skip or throw â€” reminder time is already in the past
  return null;
}
```
