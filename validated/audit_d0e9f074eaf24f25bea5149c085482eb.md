Audit Report

## Title
Stale In-Memory Transaction Status Causes Newly-Promoted Transactions to Be Permanently Missed by Execution Scheduler

## Summary
`updateTransactions()` calls `processTransactionStatus()`, which persists status changes to the database but never writes them back to the in-memory `Transaction` objects it received. The stale array is returned to `prepareTransactions()`, which gates execution scheduling on the in-memory `.status` field. Transactions just promoted from `WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION` are silently skipped. A concurrent `handleExpiredTransactions` cron running every 10 seconds can then permanently expire those transactions before the next scheduling cycle has a chance to pick them up.

## Finding Description

**Root cause — `processTransactionStatus` never mutates in-memory objects:**

In `back-end/libs/common/src/utils/transaction/index.ts`, `processTransactionStatus` iterates the `transactions` array, computes a new status for each entry, issues bulk `UPDATE` queries against the database, and returns a `Map<number, TransactionStatus>` of what changed. At no point does it assign back to `transaction.status` on any object in the array. [1](#0-0) [2](#0-1) 

**`updateTransactions` discards the result map and returns the stale array:**

In `back-end/apps/chain/src/transaction-scheduler/transaction-scheduler.service.ts`, `updateTransactions()` calls `processTransactionStatus()`, uses the returned map only to emit notifications, then returns the original `transactions` array — whose `.status` fields still reflect the pre-update values. [3](#0-2) 

**`prepareTransactions` reads the stale `.status`:**

`prepareTransactions()` iterates the returned array and gates all execution scheduling on `transaction.status === TransactionStatus.WAITING_FOR_EXECUTION`. A transaction promoted to `WAITING_FOR_EXECUTION` in the DB during this same cycle still carries `WAITING_FOR_SIGNATURES` in memory and is silently skipped. [4](#0-3) 

**Both cron callers of `prepareTransactions` are affected:**

`handleInitialTransactionStatusUpdate` (fires once on startup) and `handleTransactionsBetweenNowAndAfterThreeMinutes` (fires every 10 seconds) both pass the stale array directly to `prepareTransactions`. [5](#0-4) [6](#0-5) 

**`handleExpiredTransactions` closes the window permanently:**

Running every 10 seconds, this cron expires any transaction — including those with `WAITING_FOR_EXECUTION` status — whose `validStart < (now − 3 minutes)`. Once a transaction is `EXPIRED`, no retry path exists. [7](#0-6) 

## Impact Explanation

A fully-signed transaction whose `validStart` falls within the final 10-second window before the 3-minute expiry boundary will:

1. Be promoted to `WAITING_FOR_EXECUTION` in the DB by `processTransactionStatus`.
2. Be skipped by `prepareTransactions` because the in-memory `.status` still reads `WAITING_FOR_SIGNATURES`.
3. Fall outside the `[now − 3m, now + 3m]` fetch window on the very next 10-second tick of `handleTransactionsBetweenNowAndAfterThreeMinutes`, so it is never re-fetched for scheduling.
4. Be marked `EXPIRED` by `handleExpiredTransactions` on that same next tick, since `validStart < (now − 3m)` is now true.

The transaction is then permanently unexecutable. The Hedera network will reject it as expired, and the system has no retry path for `EXPIRED` transactions. This constitutes unrecoverable corruption of user state for any multi-signature workflow where the final signature arrives near the end of the validity window.

## Likelihood Explanation

Both `handleTransactionsBetweenNowAndAfterThreeMinutes` and `handleExpiredTransactions` fire every 10 seconds. Any transaction whose `validStart` is between 2 minutes 50 seconds and 3 minutes before the current time at the moment it first accumulates enough signatures is vulnerable. This is a realistic scenario for organizations running time-sensitive multi-signature workflows where the final signature arrives near the end of the validity window — a common operational pattern.

## Recommendation

Apply the status changes from the `results` map back to the in-memory objects before returning from `updateTransactions`, so that `prepareTransactions` operates on current data:

```typescript
// In updateTransactions(), after processTransactionStatus returns:
const results = await processTransactionStatus(
  this.transactionRepo, this.transactionSignatureService, transactions
);

// Write computed statuses back to in-memory objects
for (const transaction of transactions) {
  if (results.has(transaction.id)) {
    transaction.status = results.get(transaction.id)!;
  }
}

if (results.size > 0) {
  const events = Array.from(results.keys(), id => ({ entityId: id }));
  emitTransactionStatusUpdate(this.notificationsPublisher, events);
}

return transactions;
```

This ensures `prepareTransactions` sees the post-update status for every transaction in the same cycle, eliminating the race condition with `handleExpiredTransactions`.

## Proof of Concept

**Setup:** A multi-signature transaction with `validStart = now − 2m55s` is in `WAITING_FOR_SIGNATURES` state. The final required signature is submitted at time T.

**Step 1 — T (10-second cron fires):**
- `handleTransactionsBetweenNowAndAfterThreeMinutes` calls `updateTransactions(now − 3m, now + 3m)`.
- The transaction is fetched (its `validStart` is within the window).
- `processTransactionStatus` determines all required signatures are present, sets `status = WAITING_FOR_EXECUTION` in the DB, and returns `Map { txId → WAITING_FOR_EXECUTION }`.
- `updateTransactions` emits a notification and returns the original array — `transaction.status` is still `WAITING_FOR_SIGNATURES` in memory.
- `prepareTransactions` evaluates `transaction.status === WAITING_FOR_EXECUTION` → `false` → skips the transaction. No execution is scheduled. [1](#0-0) [3](#0-2) 

**Step 2 — T+10s (next cron tick):**
- `handleTransactionsBetweenNowAndAfterThreeMinutes`: fetch window is now `[now − 3m, now + 3m] = [T − 2m50s, ...]`. Since `validStart = T − 2m55s < T − 2m50s`, the transaction is **outside the window** and is not fetched. No scheduling occurs.
- `handleExpiredTransactions`: `threeMinutesBefore = T + 10s − 3m = T − 2m50s`. Since `validStart = T − 2m55s < T − 2m50s`, the condition `validStart < threeMinutesBefore` is **true**. The transaction is bulk-updated to `EXPIRED`. [8](#0-7) 

**Result:** The transaction is permanently `EXPIRED` and unexecutable, despite having been fully signed and correctly promoted in the database.
