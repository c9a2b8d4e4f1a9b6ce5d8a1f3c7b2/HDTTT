### Title
Approver Structure Can Be Modified After Approvals Are Cast, Enabling Approval Double-Counting

### Summary

In `createTransactionApprovers`, when a user who has already approved a transaction is added as an approver again (at a different tree position), their existing approval state (`signature`, `userKeyId`, `approved: true`) is silently copied into the new approver record. Because there is no guard preventing approver additions or threshold modifications after approvals have been cast, a malicious transaction creator can retroactively satisfy an approval threshold using a single approver's vote counted multiple times across different tree nodes.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `createTransactionApprovers` function contains the following block: [1](#0-0) 

When a `userId` is provided, the code queries all existing approver records for that user on the same transaction and copies the first record's `signature`, `userKeyId`, and `approved` fields into the new approver entry. This is intended to propagate an existing approval to a newly added position in the approver tree.

The duplicate-prevention check (`isNode`) only blocks adding the same user to the **same parent node** (`listId`): [2](#0-1) 

It does **not** prevent adding the same user under a different `listId`. A user can therefore appear in multiple positions in the approver tree.

Critically, `createTransactionApprovers` delegates its authorization check to `getCreatorsTransaction`, which only verifies that the caller is the transaction creator — it performs **no check on transaction status**: [3](#0-2) 

Similarly, `updateTransactionApprover` allows the creator to change the threshold of any tree node at any time, also with no status guard: [4](#0-3) 

### Impact Explanation

A malicious transaction creator can bypass the multi-approval requirement:

1. Creator sets up a transaction with approvers `[A, B]` at threshold `2 of 2`.
2. User A approves — their record now has `approved: true`.
3. Creator calls `POST /transactions/:id/approvers` to add a new threshold sub-node (e.g., `threshold: 1`, children: `[A]`). Because `A`'s `listId` differs from the original, `isNode` does not block this. The new record for A is created with `approved: true` copied from the existing record.
4. Creator calls `PATCH /transactions/:id/approvers/:nodeId` to lower the root threshold to `2 of 3` (or restructure so that A's two positions satisfy the threshold).
5. The approval threshold is now satisfied by a single user's approval, counted twice across two tree nodes.

This allows a transaction creator to push a transaction to `WAITING_FOR_EXECUTION` without obtaining the required number of distinct approvals, undermining the entire multi-approval governance model.

### Likelihood Explanation

The creator role is not a privileged admin — any user who creates a transaction holds this role. In an organizational setting where the approval workflow is the primary security control, a creator who wants to bypass approvals (e.g., to execute an unauthorized fund transfer or account change) has a clear, low-complexity path to do so. The attack requires only standard API calls available to any authenticated creator.

### Recommendation

1. **Prevent approver structure changes after any approval has been cast.** In both `createTransactionApprovers` and `updateTransactionApprover`, check that no existing approver record for the transaction has a non-null `signature` (i.e., has already approved) before allowing structural changes:

```typescript
const existingApprovals = await transactionalEntityManager.count(TransactionApprover, {
  where: { transactionId, signature: Not(IsNull()) },
});
if (existingApprovals > 0) throw new Error('Cannot modify approvers after approvals have been cast');
```

2. **Do not copy approval state when adding a new approver.** Remove the block that propagates `signature`, `userKeyId`, and `approved` from an existing record to a newly created approver entry. A new approver position should always start in an unapproved state. [1](#0-0) 

3. **Enforce a transaction status guard** in `getCreatorsTransaction` or at the entry points of `createTransactionApprovers` and `updateTransactionApprover` to reject structural changes once the transaction has moved past `NEW` or `WAITING_FOR_SIGNATURES`.

### Proof of Concept

```
# Step 1: Creator creates transaction TX-1 with approvers A and B, threshold 2 of 2
POST /transactions/1/approvers
{ "approversArray": [{ "userId": A }, { "userId": B }] }
# Root threshold set to 2 via separate call

# Step 2: User A approves
POST /transactions/1/approvers/approve
{ "userKeyId": A_key, "signature": "...", "approved": true }
# TransactionApprover for A now has approved=true, signature=<sig>

# Step 3: Creator adds a new threshold sub-node containing A
POST /transactions/1/approvers
{ "approversArray": [{ "threshold": 1, "approvers": [{ "userId": A }] }] }
# isNode check passes (different listId)
# New approver record for A is created with approved=true (copied from existing)

# Step 4: Creator lowers root threshold to 1 of 3 (or 2 of 3 where A satisfies 2 slots)
PATCH /transactions/1/approvers/<root_node_id>
{ "threshold": 1 }

# Result: Transaction moves to WAITING_FOR_EXECUTION with only A's single approval
``` [5](#0-4)

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
