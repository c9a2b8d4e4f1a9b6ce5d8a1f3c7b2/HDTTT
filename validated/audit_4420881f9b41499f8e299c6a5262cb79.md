Audit Report

## Title
Transaction Creator Can Add Themselves as Approver and Self-Approve, Bypassing Multi-Party Governance

## Summary
`createTransactionApprovers` in `approvers.service.ts` restricts the caller to the transaction creator but places no restriction on which `userId` values may be submitted as approvers — including the creator's own `user.id`. `approveTransaction` in the same file then verifies only that the caller is listed as an approver and has not already signed, with no check that the approver is not also the creator. Together these two gaps allow a single authenticated user to unilaterally satisfy the approval requirement for their own transaction.

## Finding Description

**Root cause — `createTransactionApprovers`**

The function opens with `getCreatorsTransaction`, which confirms the caller is the creator: [1](#0-0) 

The only validation applied to `dtoApprover.userId` is a database existence check: [2](#0-1) 

There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. The creator's own `user.id` passes this check without issue, and the record is inserted: [3](#0-2) 

**Root cause — `approveTransaction`**

The function filters approvers by `user.id` and checks only two conditions — that the user is listed as an approver, and that they have not already signed: [4](#0-3) 

The transaction is fetched with its `creatorKey` relation: [5](#0-4) 

Yet `transaction.creatorKey?.userId` is never compared against `user.id` to block self-approval. The approval is recorded unconditionally once the signature is verified: [6](#0-5) 

## Impact Explanation

The approval workflow is the primary governance control in Organization Mode. Its purpose is to ensure that sensitive Hedera transactions — key rotations (`AccountUpdateTransaction`), node operations (`NodeUpdateTransaction`), HBAR/token transfers, system file changes — cannot be executed by a single actor. By self-approving, the creator collapses a multi-party control into a single-party action. If the creator is the sole required approver, or if the threshold is configured such that one approval suffices, the transaction advances to `WAITING_FOR_EXECUTION` or executes without any peer review.

## Likelihood Explanation

Any authenticated, verified organization member who can create transactions can trigger this. No privileged access, leaked credentials, or external dependencies are required. The attacker needs only a valid JWT and a registered key. All steps use normal, documented API flows (`POST /transactions`, `POST /transactions/:id/approvers`, `POST /transactions/:id/approvers/approve`) and require no special tooling.

## Recommendation

1. **In `createTransactionApprovers`**: After the user-existence check, add an explicit guard rejecting any `dtoApprover.userId` that equals `user.id`:
   ```ts
   if (dtoApprover.userId === user.id)
     throw new Error('The transaction creator cannot be designated as an approver');
   ```
   Apply the same check recursively for nested approvers.

2. **In `approveTransaction`**: After fetching the transaction, verify the approving user is not the creator:
   ```ts
   if (transaction.creatorKey?.userId === user.id)
     throw new UnauthorizedException('The transaction creator cannot approve their own transaction');
   ```

3. **In `updateTransactionApprover`**: Apply the same creator-exclusion check when `dto.userId` is updated, since the same self-assignment is possible through the update path. [7](#0-6) 

## Proof of Concept

```
# Step 1 – Create a transaction (attacker is creator)
POST /transactions
Authorization: Bearer <attacker_jwt>
→ { "id": 42, "status": "WAITING_FOR_SIGNATURES", ... }

# Step 2 – Add self as the sole approver
POST /transactions/42/approvers
Authorization: Bearer <attacker_jwt>
{ "approversArray": [{ "userId": <attacker_user_id> }] }
→ 201 Created  (no rejection — userId existence check passes, no creator-exclusion check)

# Step 3 – Self-approve with a valid signature
POST /transactions/42/approvers/approve
Authorization: Bearer <attacker_jwt>
{ "userKeyId": <attacker_key_id>, "signature": "<valid_sig>", "approved": true }
→ true  (approveTransaction finds attacker in approver list, verifies signature, records approval)

# Result: transaction advances to WAITING_FOR_EXECUTION / EXECUTED
# with zero review from any other organization member.
```
