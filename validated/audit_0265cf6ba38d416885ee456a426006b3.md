### Title
`verifyAccess` Unconditionally Returns `true` for Terminal-Status Transactions, Bypassing Per-User Authorization

### Summary
The `verifyAccess` function in `transactions.service.ts` immediately returns `true` for any transaction whose status is EXECUTED, EXPIRED, FAILED, CANCELED, or ARCHIVED — without checking whether the requesting user has any relationship to that transaction. This is the direct analog of the external `_exitMarket` report: a gating function that always signals "access granted" under a specific condition instead of enforcing the intended check. Any authenticated user can retrieve the full details of any historical transaction by supplying its ID.

### Finding Description
**Root cause — `verifyAccess` skips the user-relationship check for terminal statuses:** [1](#0-0) 

Lines 790–798 unconditionally `return true` the moment the transaction's status is in the terminal set. The user-relationship evaluation (creator, signer, observer, approver) that follows on lines 800–808 is never reached for those transactions.

**Call site — `getTransactionWithVerifiedAccess` relies on this gate:** [2](#0-1) 

The function fetches the full transaction record (including `transactionBytes`, `transactionHash`, signers, observers, comments, and creator key) and then calls `verifyAccess`. Because `verifyAccess` returns `true` for any terminal-status transaction regardless of the caller, the `UnauthorizedException` branch is never triggered for those records.

**Contrast with the list endpoint:** `getTransactions` (lines 150–217) correctly scopes results to transactions where the user is creator, signer, observer, or approver. The detail endpoint provides no equivalent scoping for historical transactions, creating an inconsistency identical in structure to the external report. [3](#0-2) 

### Impact Explanation
Any authenticated user (no elevated privilege required) can enumerate integer transaction IDs and call the transaction-detail endpoint. For every transaction in a terminal status — which includes the majority of all transactions in a live system — they receive:
- Full `transactionBytes` (the serialized Hedera transaction, including all node account IDs and memo fields)
- `transactionHash`
- The list of signers and their public keys
- The list of observers and their user IDs
- Creator key and associated user identity
- All comments

This constitutes unauthorized cross-tenant data disclosure. In an organizational deployment, one organization's members can read another organization's completed transaction history.

### Likelihood Explanation
The precondition is a valid JWT — i.e., any registered user. Transaction IDs are sequential integers, so enumeration requires no guessing. The terminal-status population grows monotonically; in any production system the vast majority of stored transactions are historical. No special timing, race condition, or privileged role is needed.

### Recommendation
Remove the early-return shortcut for terminal statuses in `verifyAccess` and apply the same user-relationship check unconditionally:

```typescript
async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
  if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

  const userKeysToSign = await this.userKeysToSign(transaction, user, true);

  return (
    userKeysToSign.length !== 0 ||
    transaction.creatorKey?.userId === user.id ||
    !!transaction.observers?.some(o => o.userId === user.id) ||
    !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
    !!transaction.approvers?.some(a => a.userId === user.id)
  );
}
```

If read access to terminal-status transactions is intentionally broader (e.g., for audit purposes), that policy must be explicit and scoped — for example, restricted to admin roles — rather than silently granted to every authenticated user.

### Proof of Concept
1. Register two accounts, User A and User B, in the same deployment.
2. As User A, create and execute a transaction (status transitions to EXECUTED).
3. Note the transaction's integer `id` (e.g., `42`).
4. As User B (no relationship to the transaction), call:
   ```
   GET /transactions/42
   Authorization: Bearer <User B's JWT>
   ```
5. Observe that the full transaction record — including `transactionBytes`, signers, observers, and creator identity — is returned with HTTP 200 instead of 401/403.
6. Repeat for any EXPIRED, FAILED, CANCELED, or ARCHIVED transaction ID to confirm the bypass applies to the entire terminal-status set.
