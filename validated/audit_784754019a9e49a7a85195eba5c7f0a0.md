### Title
TOCTOU Race Condition in `approveTransaction` Allows an Approver to Override Their Own Submitted Approval

### Summary
`approveTransaction` in `approvers.service.ts` reads the approver's current signature state, performs several sequential async database and cryptographic operations, and only then writes the new approval. Because the "already approved" guard uses a snapshot taken before those awaits, two concurrent HTTP requests from the same authenticated approver can both pass the guard and write conflicting `approved` values, letting the user silently change a decision that the protocol treats as final.

### Finding Description

**Root cause — stale-read guard with async gap before write**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` follows this sequence:

1. **Read** approver rows from the database. [1](#0-0) 

2. **Guard** — throw if the user has already signed every one of their approver records. [2](#0-1) 

3. **Multiple async awaits** — `attachKeys`, `findOne(Transaction)`, `SDKTransaction.fromBytes`, `verifyTransactionBodyWithoutNodeAccountIdSignature` — all happen between the guard and the write. [3](#0-2) 

4. **Write** — update the approver rows with the new `userKeyId`, `signature`, and `approved` value. [4](#0-3) 

There is no database-level lock, no `SELECT … FOR UPDATE`, and no optimistic-concurrency check between steps 2 and 4. Node.js's event loop yields at every `await`, so two in-flight requests interleave freely.

**Exploit flow**

An authenticated approver sends two HTTP `POST /transactions/:id/approvers/approve` requests concurrently — one with `approved: true` and one with `approved: false`. Both requests execute step 1 and step 2 against the same database snapshot (no signature yet), both pass the guard, and both proceed through the async gap. Whichever write lands last wins, overwriting the first. The protocol invariant — *an approval decision is final once submitted* — is violated.

The notification branch at lines 614–618 also uses the stale `userApprovers` snapshot, so the event emitted to downstream consumers may reflect the wrong state. [5](#0-4) 

**Structural parallel to the external report**

The external report's `_splitValue` bug: `_mintBatch` (external call) fires before `tokenValues[_tokenID]` is updated, so a reentrant call sees stale state and mints excess tokens. Here: the write (step 4) happens after multiple async yields, so a concurrent call sees the same stale "not yet signed" state and bypasses the one-approval-per-user invariant.

### Impact Explanation

A legitimate approver can flip their own recorded decision (`approved: true → false` or vice versa) after it has already been persisted, without triggering the duplicate-approval guard. In a threshold-approval workflow this can:

- Prevent a transaction from reaching `WAITING_FOR_EXECUTION` by retracting a previously cast approval.
- Force a transaction into execution by retracting a previously cast rejection.
- Cause the notification subsystem to emit a status event based on stale approval state, desynchronising observer UIs.

The attacker controls only their own approver record, so they cannot forge another user's signature. Impact is scoped to transactions where the attacker is a designated approver.

### Likelihood Explanation

Any authenticated user who is listed as an approver can trigger this with two simultaneous HTTP requests — no special tooling, no privileged credentials, no network position. The async gap spans at least two database round-trips and one cryptographic verification, giving a wide enough window for concurrent requests to interleave reliably. This is a standard race condition that is straightforward to reproduce with `curl --parallel` or any HTTP client that supports concurrent requests.

### Recommendation

Move the guard inside the same database transaction as the write, and use a pessimistic row-level lock so the check and the update are atomic:

```typescript
await this.dataSource.transaction(async em => {
  // Re-read with a row lock so no concurrent request can interleave
  const lockedApprovers = await em
    .createQueryBuilder(TransactionApprover, 'a')
    .setLock('pessimistic_write')
    .where('a.id IN (:...ids)', { ids: userApprovers.map(a => a.id) })
    .getMany();

  if (lockedApprovers.every(a => a.signature)) {
    throw new BadRequestException(ErrorCodes.TAP);
  }

  await em
    .createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .whereInIds(lockedApprovers.map(a => a.id))
    .execute();
});
```

All validation that does not require the lock (key lookup, signature verification) can remain outside the transaction, but the guard and the write must be atomic.

### Proof of Concept

**Preconditions**
- Attacker is an authenticated user who is a designated approver for transaction ID `T`.
- Transaction `T` is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status.
- Attacker has not yet submitted an approval.

**Steps**

1. Attacker prepares two valid `ApproverChoiceDto` payloads for transaction `T`: one with `approved: true` (R1) and one with `approved: false` (R2), both signed with the attacker's key.

2. Attacker fires both requests simultaneously:
   ```
   curl -X POST https://<server>/transactions/T/approvers/approve \
     -H "Authorization: Bearer <token>" \
     -d '{"userKeyId":K,"signature":"<sig>","approved":true}' &

   curl -X POST https://<server>/transactions/T/approvers/approve \
     -H "Authorization: Bearer <token>" \
     -d '{"userKeyId":K,"signature":"<sig>","approved":false}' &
   wait
   ```

3. Both requests pass the guard at line 563 (neither sees a signature yet).

4. Both proceed through the async gap (lines 565–596).

5. Both execute the `UPDATE` at lines 599–610. The last write wins.

**Expected (correct) outcome:** The second request is rejected with `ErrorCodes.TAP` ("transaction already approved").

**Actual outcome:** Both requests return HTTP 200. The approver's final recorded decision is whichever request's write landed last — which the attacker can influence by retrying until the desired outcome is stored, effectively allowing them to change a final approval decision.
