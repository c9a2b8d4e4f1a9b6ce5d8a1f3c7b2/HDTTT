Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated Resource Exhaustion

## Summary
`getTransactionsToSign()` in `transactions.service.ts` fetches every non-terminal transaction in the database without a `take` limit, then serially calls `userKeysToSign()` (which invokes `computeSignatureKey()` and hits the mirror-node cache layer) for each row before applying pagination. Any authenticated, verified user with at least one registered key can inflate the transaction table and trigger unbounded server-side work with a single `GET /transactions/sign` request.

## Finding Description

**Unbounded query — no `take` clause:**

`getTransactionsToSign()` builds a `whereForUser` filter that excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does not filter by the requesting user, and the `repo.find()` call carries no `take` limit:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

This loads every active transaction in the system — across all users — into the Node.js heap simultaneously.

**Per-row async work before pagination:**

The full result set is then iterated serially, calling `userKeysToSign()` for every row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

`userKeysToSign()` calls `computeSignatureKey()`, which deserializes `transactionBytes` via `SDKTransaction.fromBytes()` and calls `AccountCacheService.getAccountInfoForTransaction()` for each involved account. The cache has a 10-second TTL (`CACHE_STALE_THRESHOLD_MS`, default 10 000 ms); for novel accounts introduced by attacker-created transactions the cache will be cold, resulting in outbound mirror-node HTTP calls. [3](#0-2) [4](#0-3) 

**Pagination applied after full scan:**

```typescript
items: result.slice(offset, offset + limit),
``` [5](#0-4) 

A request for `page=1&size=1` still forces the server to process every active transaction.

**Endpoint still registered and reachable:**

`GET /transactions/sign` is a live, authenticated route. The "NO LONGER USED BY FRONT-END" comment does not remove it from the router. [6](#0-5) 

**No throttler guard on the transactions controller:**

The controller applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. No `UserThrottlerGuard` or `ThrottlerGuard` is present on the controller or the `getTransactionsToSign` handler. [7](#0-6) 

**No per-user transaction count cap found** in the service or controller.

## Impact Explanation

- **Heap exhaustion:** TypeORM loads all matching `Transaction` rows (with `groupItem` relation) into the Node.js heap at once. With thousands of rows this can exhaust available memory and crash the process.
- **CPU / event-loop exhaustion:** Each row triggers synchronous deserialization of `transactionBytes` (`SDKTransaction.fromBytes`) and one or more async cache lookups, blocking the event loop for the duration of the serial loop.
- **Mirror-node connection pressure:** For transactions referencing accounts not yet in the cache, `AccountCacheService` makes outbound HTTP calls to the mirror node. With many novel accounts this saturates the outbound HTTP pool and causes cascading timeouts for all callers.
- **System-wide impact:** Because `whereForUser` is not scoped to the requesting user, a single attacker inflating the table degrades the endpoint for every other user who calls it.

## Likelihood Explanation

- **Precondition:** A valid, verified account with at least one registered key — normal user privileges, no admin access required.
- **Attack steps:** Register an account, add a key, call `POST /transactions` in a loop to create thousands of `WAITING_FOR_SIGNATURES` transactions (each requires only valid transaction bytes and the creator key ID). Then call `GET /transactions/sign` once.
- **Barrier analysis:** The `POST /transactions` endpoint requires `HasKeyGuard` and `OnlyOwnerKey`, but both are satisfied by any user who has registered a key. No per-user transaction count cap exists. The `GET /transactions/sign` endpoint has no rate-limiting guard.
- **SECURITY.md scope check:** The exclusion "Impacts that only require DDoS" does not apply. This vulnerability exploits a specific code-level design flaw (missing `take` limit + post-scan pagination) and is triggered by a single request after table inflation — not by request flooding.

## Recommendation

1. **Push pagination into the database query.** The correct fix is to filter at the SQL level so only transactions relevant to the requesting user are fetched, and apply `take`/`skip` before loading rows into memory. The `getTransactionsToApprove` method in the same file demonstrates the correct pattern using a `createQueryBuilder` with `skip`/`take` and a user-scoped subquery. [8](#0-7) 

2. **Add a user-scoped filter.** The query should join against the user's keys and only return transactions whose required signers intersect with those keys, mirroring the approach in `getTransactionsToApprove`.

3. **Apply a throttler guard.** Add `UserThrottlerGuard` (already implemented in the codebase) to the `getTransactionsToSign` handler. [9](#0-8) 

4. **Remove or formally deprecate the endpoint.** If the endpoint is genuinely no longer used by any client, remove it from the router to eliminate the attack surface entirely.

## Proof of Concept

```
# 1. Authenticate and obtain a JWT for a normal verified user with a registered key.

# 2. Inflate the transaction table (repeat N times):
POST /transactions
Authorization: Bearer <token>
Content-Type: application/json
{
  "name": "flood-<i>",
  "transactionBytes": "<valid_hex_encoded_bytes>",
  "creatorKeyId": <user_key_id>,
  "signature": "<valid_signature>",
  "mirrorNetwork": "testnet"
}

# 3. Trigger the unbounded scan with a single request:
GET /transactions/sign?page=1&size=1
Authorization: Bearer <token>

# Expected result: server processes all N transactions in-process before
# returning a single item. With N in the thousands, memory and CPU usage
# spike proportionally; mirror-node calls are made for each novel account.
# Concurrent legitimate requests to the same endpoint are starved or fail.
```

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
