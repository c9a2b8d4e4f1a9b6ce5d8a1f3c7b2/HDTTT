### Title
`removeTransaction` Bypasses Status Invariant Enforced by `cancelTransaction`, Allowing Creators to Corrupt Executed/Failed Transaction State

---

### Summary

The system enforces a strict status invariant in `cancelTransaction`: only transactions in `NEW`, `WAITING_FOR_SIGNATURES`, or `WAITING_FOR_EXECUTION` states may be canceled. However, the `DELETE /transactions/:id` endpoint calls `removeTransaction`, which performs no such status check. Any transaction creator can invoke this endpoint against an already-`EXECUTED`, `FAILED`, `REJECTED`, `EXPIRED`, or `ARCHIVED` transaction, causing the system to overwrite its terminal status with `CANCELED` before soft-deleting it, and to emit a false `CANCELED` status notification to all observers.

---

### Finding Description

**Root cause — inconsistent invariant enforcement across two state-mutating paths:**

`cancelTransaction` / `cancelTransactionWithOutcome` explicitly guards against canceling terminal-state transactions: [1](#0-0) 

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
```

`cancelableStatuses` = `[NEW, WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`.

`removeTransaction`, called by `DELETE /transactions/:id`, performs **no status check**. It only verifies ownership via `getTransactionForCreator`, then unconditionally overwrites the status to `CANCELED` and soft-deletes: [2](#0-1) 

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);  // ownership only

    if (softRemove) {
      await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED }); // ← no status guard
      await this.repo.softRemove(transaction);
    }
    ...
    emitTransactionStatusUpdate(...);  // ← emits CANCELED notification
}
```

The endpoint is reachable by any authenticated, verified user for their own transactions: [3](#0-2) 

```typescript
@Delete('/:id')
deleteTransaction(@GetUser() user, @Param('id', ParseIntPipe) id: number): Promise<boolean> {
    return this.transactionsService.removeTransaction(id, user, true);
}
```

**Exploit path:**

1. Attacker (normal user) creates a transaction and it reaches `EXECUTED`, `FAILED`, `REJECTED`, `EXPIRED`, or `ARCHIVED` state.
2. Attacker sends `DELETE /transactions/:id`.
3. `removeTransaction` sets `status = CANCELED` on the record, then soft-deletes it.
4. A `CANCELED` status-update notification is emitted to all observers and signers of that transaction.

---

### Impact Explanation

- **Audit trail corruption**: Soft-deleted records retain the overwritten `CANCELED` status. Any audit query using `withDeleted: true` will show an `EXECUTED` (or `FAILED`/`REJECTED`) transaction as `CANCELED`, destroying the integrity of the historical record.
- **False status notifications**: `emitTransactionStatusUpdate` fires with `CANCELED` for a transaction that already completed execution on the Hedera network. Observers who rely on these notifications may incorrectly conclude the transaction did not execute and attempt to resubmit it, causing duplicate Hedera network submissions.
- **Invariant violation**: The system's own `cancelTransaction` path explicitly prohibits this state transition for a reason; `removeTransaction` silently bypasses that protection.

---

### Likelihood Explanation

- **No privileged access required**: Any verified user who created a transaction can trigger this.
- **Trivially reachable**: A single authenticated `DELETE /transactions/:id` HTTP request is sufficient.
- **No race condition or timing dependency**: The attacker simply waits for the transaction to reach a terminal state, then issues the delete.

---

### Recommendation

Add the same status guard to `removeTransaction` that exists in `cancelTransactionWithOutcome`. Before performing the soft-delete, check that the transaction is in a cancelable state, or — if deletion of terminal-state transactions must be supported — separate the status-overwrite from the soft-delete so that terminal statuses are preserved:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (softRemove) {
      // Only overwrite status to CANCELED if the transaction is not already in a terminal state
      if (this.cancelableStatuses.includes(transaction.status)) {
        await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
      }
      await this.repo.softRemove(transaction);
    } else {
      await this.repo.remove(transaction);
    }
    ...
}
```

Alternatively, extract the status-guard logic into a shared method (analogous to the modifier pattern suggested in the external report) and apply it to both `cancelTransactionWithOutcome` and `removeTransaction`.

---

### Proof of Concept

**Preconditions**: Attacker is a verified user; they own transaction with `id = 42` which is in `EXECUTED` state.

**Steps**:

1. Confirm current status:
   ```
   GET /transactions/42  →  { status: "EXECUTED" }
   ```

2. Attempt `cancelTransaction` (blocked by invariant):
   ```
   PATCH /transactions/cancel/42  →  400 Bad Request (ErrorCodes.OTIP)
   ```

3. Call `removeTransaction` (no status guard):
   ```
   DELETE /transactions/42  →  200 OK
   ```

4. **Result**: The DB record now has `status = CANCELED`, `deletedAt = <now>`. All observers received a `CANCELED` status-update notification for a transaction that successfully executed on the Hedera network. The audit trail is corrupted. [2](#0-1) [1](#0-0) [3](#0-2)

  async cancelTransactionWithOutcome(

    user: User,
  ): Promise<CancelTransactionOutcome> {

      return CancelTransactionOutcome.ALREADY_CANCELED;
