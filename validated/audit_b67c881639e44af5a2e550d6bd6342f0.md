### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated DoS via `GET /transactions/sign`

### Summary

`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then iterates over the entire result set in a sequential `for` loop, invoking the expensive `userKeysToSign` (which deserializes transaction bytes and makes multiple async account-cache/DB calls) for each row. Pagination is applied only after the full scan completes. Any authenticated user can call `GET /transactions/sign` to trigger this unbounded work, causing request timeouts and Node.js event-loop saturation that degrades the API for all users.

**Per-transaction cost inside `userKeysToSign` → `keysRequiredToSign`:**

Each iteration calls `computeSignatureKey`, which:
- Deserializes `transaction.transactionBytes` via `SDKTransaction.fromBytes()` (CPU + memory)
- Calls `accountCacheService.getAccountInfoForTransaction()` for the fee-payer account, each signing account, and each receiver account (multiple async I/O calls per transaction)
- Issues a `entityManager.find(UserKey, ...)` DB query [4](#0-3) [5](#0-4)

### Impact Explanation

- **Service degradation / DoS:** With N active (non-terminal) transactions in the system, every call to `GET /transactions/sign` forces the API worker to deserialize N transaction byte blobs, make O(N × accounts_per_tx) async cache/DB calls, and hold all N transaction objects in memory before returning a single page. Repeated calls from one or more authenticated users saturate the Node.js event loop and exhaust the DB connection pool, causing timeouts for all concurrent users.
- **Memory pressure:** All non-terminal transactions are loaded into the process heap simultaneously; large deployments can trigger OOM.
- **No rate-limit or cap:** The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`); every other transaction is included regardless of who created it.

### Likelihood Explanation

- **Attacker precondition:** A valid, verified user account — the minimum privilege level in the system. No admin role, no special key, no internal access required.
- **Trigger:** A single HTTP GET to `/transactions/sign` (with any pagination params) is sufficient. Repeated calls amplify the effect.
- **Realistic scenario:** An organization accumulates thousands of pending transactions (normal in a multi-user signing workflow). Any member can then hammer the endpoint to degrade the API for all other users. The load-test seed script itself targets 500 sign-transactions per user, confirming the scale is operationally realistic. [7](#0-6) 

### Recommendation

1. **Apply DB-level pagination before the loop.** Pass `take: limit` and `skip: offset` to `this.repo.find(...)` and filter by the user's own keys at the query level (join `transaction_signer` or `user_key`) so only the page-sized slice is fetched and processed.
2. **Alternatively, remove the endpoint** if it is confirmed unused by the front-end (the controller already documents "NO LONGER USED BY FRONT-END"). Removing it eliminates the attack surface entirely.
3. **Add a hard cap** on the maximum number of rows processed in any single request (e.g., `take: Math.min(limit, 100)`) as a defence-in-depth measure.

### Proof of Concept

**Preconditions:**
- A verified user account exists.
- The organization has ≥ 1,000 non-terminal transactions (achievable by any user who can create transactions, or by the attacker themselves creating transactions in bulk).

**Steps:**
1. Authenticate and obtain a JWT token.
2. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <token>
   ```
3. Each request causes the server to:
   - `SELECT * FROM transaction WHERE status NOT IN (...)` — returns all N rows
   - Loop N times, calling `SDKTransaction.fromBytes()` + `computeSignatureKey()` + `entityManager.find(UserKey)` per iteration
   - Return only 10 items from the result

**Expected outcome:** Response latency grows linearly with N. At ~1,000 active transactions, response times exceed typical HTTP timeouts (30 s). Concurrent requests from the same or different users exhaust the DB connection pool and block all other API operations.

export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
