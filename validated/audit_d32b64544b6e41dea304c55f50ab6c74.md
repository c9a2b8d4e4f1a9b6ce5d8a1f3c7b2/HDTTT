### Title
Approver State Mutations Permitted on Terminal-State Transactions

### Summary
The `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` operations in `approvers.service.ts` use `getCreatorsTransaction()` as their authorization gate. That function only verifies ownership — it never checks the transaction's status. As a result, a transaction creator can add, update, or delete approvers on transactions that are already in terminal states (`EXECUTED`, `CANCELED`, `EXPIRED`, `FAILED`, `ARCHIVED`), retroactively corrupting the approval audit trail that the multi-signature workflow depends on.

### Finding Description

**Root cause — `getCreatorsTransaction` has no status guard:** [1](#0-0) 

The function fetches the transaction and checks `creatorKey.userId === user.id`, then returns. There is no check against `TransactionStatus`.

**`createTransactionApprovers` delegates entirely to this gate:** [2](#0-1) 

After `getCreatorsTransaction` returns, the function proceeds to insert new `TransactionApprover` rows regardless of the transaction's current status.

**The test suite confirms this is reachable on EXPIRED transactions:** [3](#0-2) 

The fixture transaction has `status: TransactionStatus.EXPIRED` and the test asserts that `dataSource.manager.insert` is called and `emitTransactionStatusUpdate` fires — meaning the operation succeeds on a terminal-state transaction.

**`removeTransactionApprover` in the controller uses the same unguarded gate:** [4](#0-3) 

`getCreatorsTransaction` is called, then `removeTransactionApprover` soft-deletes the approver node with no status check.

**`updateTransactionApprover` has the same gap:** [5](#0-4) 

The only authorization check inside the transaction block is `getCreatorsTransaction`, which again does not inspect status.

**Contrast with `approveTransaction`, which does guard status correctly:** [6](#0-5) 

The approval action itself is protected, but the structural management of the approver tree is not.

### Impact Explanation

This is a multi-signature transaction management system where the approval tree is the authoritative record of who was required to authorize a transaction. Allowing a creator to mutate that tree after a transaction reaches a terminal state (`EXECUTED`, `CANCELED`, `EXPIRED`, `FAILED`, `ARCHIVED`) means:

1. **Audit trail erasure**: A creator can call `DELETE /transactions/:id/approvers/:approverId` on an already-executed transaction, removing the record of who was required to sign. This undermines the non-repudiation guarantee of the multi-sig workflow.
2. **Retroactive approval tree rewriting**: A creator can add new approvers or change existing ones on an executed transaction, making the historical record inconsistent with what was actually enforced at execution time.
3. **Spurious notifications**: `emitTransactionStatusUpdate` fires on every successful mutation, sending notifications to users about transactions that are already in terminal states.

### Likelihood Explanation

The attacker precondition is minimal: any authenticated, verified user who created at least one transaction. The exploit path is a standard authenticated API call (`POST /transactions/:id/approvers`, `DELETE /transactions/:id/approvers/:id`, or `PATCH /transactions/:id/approvers/:id`) with a terminal-state transaction ID. No privilege escalation, no leaked secrets, and no race condition is required.

### Recommendation

Add a status guard inside `getCreatorsTransaction` (or as a dedicated helper called at the top of each mutating approver operation) that rejects requests when the transaction is in a terminal state:

```typescript
const terminalStatuses = [
  TransactionStatus.EXECUTED,
  TransactionStatus.FAILED,
  TransactionStatus.EXPIRED,
  TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED,
];

if (terminalStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP); // or a dedicated error code
}
```

This mirrors the pattern already applied correctly in `approveTransaction` and in `cancelTransactionWithOutcome`. [7](#0-6) 

### Proof of Concept

**Preconditions:**
- Attacker is a registered, verified user.
- Attacker previously created transaction `T` (any type). `T` has since reached `EXECUTED` status.

**Steps:**

1. Authenticate and obtain a JWT token.
2. Send:
   ```
   POST /transactions/{T.id}/approvers
   Authorization: Bearer <token>
   Content-Type: application/json

   { "approversArray": [{ "userId": <any_valid_user_id> }] }
   ```
3. The server calls `getCreatorsTransaction(T.id, user)` — passes because the caller is the creator.
4. No status check occurs. The new `TransactionApprover` row is inserted into the database for the already-executed transaction.
5. `emitTransactionStatusUpdate` fires, sending a notification event for a terminal transaction.
6. Repeat with `DELETE /transactions/{T.id}/approvers/{approver_id}` to erase existing approver records from the executed transaction's history.

**Expected (correct) behavior:** `400 Bad Request` — transaction is not in a mutable state.
**Actual behavior:** `201 Created` — approver inserted; audit trail corrupted.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
