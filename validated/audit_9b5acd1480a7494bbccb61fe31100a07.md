### Title
Stale Approval State Carried Over When Approver Is Re-Added to Transaction Approver Tree

### Summary
In `createTransactionApprovers`, when a new approver node is created for a user who already has an existing (non-deleted) approver record on the same transaction, the code explicitly copies the prior `signature`, `userKeyId`, and `approved` fields into the new record. This means a previously collected approval vote is silently inherited by the new node without the user needing to re-approve. A malicious transaction creator can exploit this to restructure the approver tree after partial approvals have been collected, causing stale approvals to satisfy a threshold that was not legitimately met.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, inside `createTransactionApprovers`, after basic validation passes, the code queries for any existing approver records for the same `userId` on the same `transactionId` and copies their approval state into the new record: [1](#0-0) 

The duplicate-check guard (`isNode`) only prevents re-adding a user with the **exact same** `listId` and `transactionId` combination: [2](#0-1) 

Because `isNode` matches on `listId` exactly, adding User A as a **child of a threshold node** (`listId = thresholdNodeId`) does not conflict with an existing **root-level** approver record for User A (`listId = null`). The duplicate check passes, and the stale approval is copied into the new child record.

`getCreatorsTransaction` — the only authorization gate for approver creation — performs **no transaction-status check**, so the creator can restructure the approver tree at any point, including after approvals have already been collected: [3](#0-2) 

**Exploit path:**

1. Creator creates transaction T with root-level approvers `[A, B]`, threshold 2-of-2.
2. User A approves — their `signature`, `userKeyId`, `approved=true` are stored in their root-level `TransactionApprover` record.
3. Creator calls `POST /transactions/:id/approvers` to add a new threshold node (threshold 1-of-2) with children `[A, C]`.
4. `isNode` does not block this because the new child record has `listId = thresholdNodeId` (≠ null), so no duplicate is detected.
5. `getApproversByTransactionId(transactionId, A.userId)` finds A's existing root-level record (non-deleted, `approved=true`) and copies its approval state into the new child record (lines 325–329).
6. The threshold node now has 1-of-2 approvals satisfied by the stale vote — without User A re-approving under the new structure.
7. The overall approval requirement is satisfied without User B ever approving.

### Impact Explanation

A malicious transaction creator can bypass the intended multi-approver threshold by restructuring the approver tree after partial approvals are collected. Stale approvals from a prior tree position are silently counted in the new position, allowing the transaction to advance to `WAITING_FOR_EXECUTION` without the required number of genuine approvals. This is an integrity failure in the approval/governance workflow.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The exploit requires only standard API calls (`POST /transactions/:id/approvers`) that are already available to the creator. The creator has a direct incentive to push their transaction through approval. No external access or leaked credentials are needed.

### Recommendation

When creating a new approver node for a user, **do not carry over prior approval state**. The approval state should always start as `null`/`undefined` for a freshly created approver node, regardless of whether the user has previously approved the transaction in a different position. Remove lines 318–329 from `createTransactionApprovers`: [1](#0-0) 

Additionally, add a transaction-status guard in `getCreatorsTransaction` (or at the start of `createTransactionApprovers` / `updateTransactionApprover`) to prevent approver-tree modifications once any approval has been recorded or once the transaction has advanced past `NEW`/`WAITING_FOR_SIGNATURES`.

### Proof of Concept

```
# Step 1: Creator creates transaction T, adds approvers A and B (threshold 2/2)
POST /transactions/T/approvers
Body: { approversArray: [{ userId: A }, { userId: B }] }

# Step 2: User A approves
POST /transactions/T/approvers/approve  (as User A)
Body: { userKeyId: ..., signature: ..., approved: true }

# Step 3: Creator adds a threshold node with A and C as children (threshold 1/2)
POST /transactions/T/approvers
Body: { approversArray: [{ threshold: 1, approvers: [{ userId: A }, { userId: C }] }] }

# Result: The new child record for User A inherits approved=true from step 2.
# The threshold node (1/2) is now satisfied by the stale vote.
# Transaction advances without User B approving.
```

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
