### Title
Unbounded In-Memory Iteration in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** active transactions from the database without a `LIMIT` clause, then iterates over every result calling an async per-transaction cryptographic/DB operation (`userKeysToSign`), and only paginates the final result in memory. As the number of active transactions grows — either naturally or through attacker-created transactions — each call to this endpoint consumes unbounded CPU, memory, and database resources, eventually causing request timeouts and service degradation for all users.

### Finding Description

**Root cause — missing DB-level pagination:**

In `getTransactionsToSign`, the repository `find()` call has no `take` (SQL `LIMIT`) parameter: [1](#0-0) 

This fetches every transaction in `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` / `NEW` / `REJECTED` status across the entire platform — not scoped to the requesting user.

**Unbounded per-transaction async work:**

The code then loops over every returned transaction and calls `userKeysToSign` for each one: [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign` → `transactionSignatureService.computeSignatureKey`, which performs cryptographic key resolution and database lookups per transaction.

**In-memory pagination applied after full scan:**

Pagination is applied only after the full scan completes: [3](#0-2) 

The `limit`/`offset` from the `Pagination` parameter are never passed to the DB query, only used to slice the already-loaded result array.

**Self-acknowledged missing limit in `importSignatures`:**

A related pattern exists in `importSignatures`, where a developer comment explicitly notes the missing API-side limit: [4](#0-3) 

### Impact Explanation

- Every call to `GET /transactions/to-sign` by any authenticated user triggers a full table scan of all active transactions, followed by N async cryptographic/DB operations (one per transaction).
- Memory usage scales linearly with the number of active transactions loaded into the Node.js process.
- CPU and DB connection time scale linearly with N per request.
- With a large enough transaction backlog, the endpoint times out, the event loop stalls, and other API endpoints become unresponsive — full service degradation for all users.
- The `totalItems` returned is also incorrect (it reflects only the user's matching subset, not the full count), compounding the logic error.

### Likelihood Explanation

- **No privilege required**: any authenticated user can call this endpoint.
- **Natural growth**: the platform accumulates transactions in normal operation; the vulnerability worsens over time without any attacker action.
- **Attacker acceleration**: a malicious authenticated user can create large numbers of transactions (via `POST /transactions` or `POST /transactions/group`) to rapidly inflate the active transaction count, amplifying the per-request cost for all users.
- The `createTransactions` endpoint accepts a batch array of DTOs with no documented upper bound on array size, making bulk creation straightforward. [5](#0-4) 

### Recommendation

1. **Push pagination into the database query**: pass `take: limit` and `skip: offset` to the `repo.find()` call inside `getTransactionsToSign`, and restructure the query so the user-key filtering is done at the SQL level (e.g., join on `transaction_signer.userKeyId IN (user's key IDs)`) rather than in application code.
2. **Add an API-level cap** on the maximum number of items that can be requested in a single call (e.g., `limit = Math.min(limit, 100)`).
3. **Add a cap on batch input size** for `importSignatures` and `uploadSignatureMaps` to prevent unbounded processing loops over attacker-supplied arrays.

### Proof of Concept

**Preconditions**: attacker has a valid authenticated session (normal user account).

**Steps**:
1. Authenticate and obtain a JWT token.
2. Create a large number of transactions (e.g., 5,000–10,000) in `WAITING_FOR_SIGNATURES` status using repeated calls to `POST /transactions` or a single call to `POST /transactions/group` with a large `groupItems` array.
3. Call `GET /transactions/to-sign?page=1&limit=10` (any pagination parameters).
4. **Observed**: the server fetches all N active transactions from the DB, calls `userKeysToSign` N times, exhausts the Node.js heap or times out the HTTP request.
5. **Expected**: the endpoint should return in bounded time regardless of total transaction count.
