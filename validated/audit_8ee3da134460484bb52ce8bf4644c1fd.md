### Title
Cancelled Transactions Can Still Be Executed on Hedera Due to Stale In-Memory State in the Scheduler

### Summary
The Chain Service scheduler captures transaction objects in JavaScript closure at scheduling time and never re-fetches their status from the database before submitting them to the Hedera network. When a user cancels a transaction after the scheduler has already registered an execution timeout, the cancellation is written to the database but the in-flight timeout fires against the stale in-memory object, causing the transaction to be submitted to Hedera regardless. This is the direct analog of TRST-M-8: a "paused" (cancelled) state does not stop the dependent background process.

### Finding Description

**Vulnerability class:** State-transition inconsistency — a terminal state change (CANCELLED) is not propagated to already-scheduled in-process execution callbacks.

**Root cause — `collateGroupAndExecute`:**

`prepareTransactions` loads a `TransactionGroup` from the database and passes it to `collateGroupAndExecute`. That function captures the entire group object in a `setTimeout` closure scheduled to fire at `validStart − 10 s`. [1](#0-0) 

When the closure fires it iterates `transactionGroup.groupItems` — the **same in-memory object** loaded minutes earlier — calls `smartCollate`, then calls `addGroupExecutionTimeout(transactionGroup)` with no database re-fetch and no status guard. [2](#0-1) 

`addGroupExecutionTimeout` registers a second `setTimeout` at `validStart + 5 s` that calls `executeService.executeTransactionGroup(transactionGroup)` — again with the stale object.

**Root cause — `executeTransactionGroup` in-memory filter:**

The only status guard inside `executeTransactionGroup` is:

```typescript
transactionGroup.groupItems = transactionGroup.groupItems.filter(
  tx => tx.transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
);
``` [3](#0-2) 

`tx.transaction.status` is the value that was set when the group was originally loaded from the database — it is **never refreshed**. A cancellation written to the database after that point is invisible to this filter.

**Root cause — `collateAndExecute` (single transactions):**

The same pattern applies to individual transactions. The `transaction` object is captured in a closure, `addExecutionTimeout` is called with it, and `executeService.executeTransaction(transaction)` is invoked with no intervening DB status check. [4](#0-3) [5](#0-4) 

**Contrast with the one place a DB guard IS used:**

When `smartCollate` returns `null` (oversize), the code correctly uses `AND status = :currentStatus` in the UPDATE query to guard against stale state. No equivalent guard exists on the happy-path execution branch. [6](#0-5) 

### Impact Explanation

A user who cancels a transaction after the scheduler has registered its collation timeout (i.e., within the final ~15-second window before `validStart`) will have their transaction submitted to the Hedera network despite the explicit cancellation. For transactions that transfer HBAR or tokens, this means an irreversible on-chain operation executes against the user's intent. The Hedera network does not support transaction recall; once submitted and accepted, the effect is permanent.

Severity: **High** — direct, irreversible asset movement contrary to explicit user action.

### Likelihood Explanation

The race window is deterministic and predictable: the scheduler polls every 10 seconds and registers timeouts for all transactions whose `validStart` falls within a 6-minute window. Any cancellation that arrives after `collateAndExecute`/`collateGroupAndExecute` has registered its timeout but before `validStart + 5 s` will trigger the bug. A normal user who changes their mind in the last few minutes before a scheduled transaction fires hits this window routinely. No attacker privilege is required — the user's own cancellation action is the trigger. [7](#0-6) 

### Recommendation

Re-fetch the transaction status from the database immediately before submitting to Hedera in both `collateAndExecute` and `collateGroupAndExecute` callbacks, and abort if the status is no longer `WAITING_FOR_EXECUTION`. Specifically:

1. In the `collateAndExecute` callback, before calling `addExecutionTimeout`, query the DB for the current status and return early if it is not `WAITING_FOR_EXECUTION`.
2. In `executeTransactionGroup`, replace the in-memory filter with a fresh DB query (or add `AND status = 'WAITING FOR EXECUTION'` to the execution UPDATE) so that a cancellation written between scheduling and execution is respected.
3. Apply the same `AND status = :currentStatus` guard that already exists in the oversize-failure branch to the execution submission path.

### Proof of Concept

1. User creates a group transaction with `validStart` set 2 minutes in the future.
2. The Chain Service cron (`handleTransactionsBetweenNowAndAfterThreeMinutes`) fires, loads the group, calls `collateGroupAndExecute`, and registers a `setTimeout` for `validStart − 10 s`.
3. User calls `PATCH /transactions/cancel/:id` (or the group cancel endpoint). The database row is updated to `CANCELED`.
4. The `setTimeout` fires. The closure holds the original `transactionGroup` object whose `groupItems[*].transaction.status` is still `WAITING_FOR_EXECUTION` in memory.
5. `executeTransactionGroup` filters by in-memory status — all items pass. `getValidatedSDKTransaction` and `_executeTransaction` are called.
6. The Hedera SDK submits the transaction to the network. The on-chain transfer executes despite the user's cancellation. [8](#0-7) [9](#0-8)

  collateGroupAndExecute(transactionGroup: TransactionGroup) {
    const name = `smart_collate_group_timeout_${transactionGroup.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart =
      transactionGroup.groupItems[0].transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        let smartCollateFailed = false;
        for (const groupItem of transactionGroup.groupItems) {
          const transaction = groupItem.transaction;

          const requiredKeys = await this.transactionSignatureService.computeSignatureKey(transaction);

          const sdkTransaction = await smartCollate(transaction, requiredKeys);

          // If the transaction is still too large,
          // break out of the loop and update all transactions in the group to failed
          // with the TRANSACTION_OVERSIZE status code.
          // This should happen on the first transaction, if at all
          if (sdkTransaction === null) {
            smartCollateFailed = true;
            break;
          }

          //NOTE: the transactionBytes are set here but are not to be saved. Otherwise,
          // any signatures that were removed in order to make the transaction fit
          // would be lost.
          transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
        }

        if (smartCollateFailed) {
          const result = await this.transactionRepo
            .createQueryBuilder()
            .update(Transaction)
            .set({
              status: TransactionStatus.FAILED,
              executedAt: new Date(),
              statusCode: Status.TransactionOversize._code,
            })
            .where('id IN (:...ids) AND status = :currentStatus', {
              ids: transactionGroup.groupItems.map(gi => gi.transaction.id),
              currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
            })
            .returning('id')
            .execute();

          if (result.raw.length > 0) {
            emitTransactionStatusUpdate(
              this.notificationsPublisher,
              result.raw.map(row => ({ entityId: row.id })),
            );
          }
          return;
        }

        this.addGroupExecutionTimeout(transactionGroup);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart - 10 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);

  addExecutionTimeout(transaction: Transaction) {

    // first we need to validate all the transactions, as they all need to be valid before we can execute any of them
    for (const groupItem of transactionGroup.groupItems) {
      const transaction = groupItem.transaction;
      try {
        const sdkTransaction = await this.getValidatedSDKTransaction(transaction);
        transactions.push({ sdkTransaction, transaction });

        );
