Audit Report

## Title
Unbounded Sequential Per-Transaction Computation in `getTransactionsToSign` Causes Server Resource Exhaustion

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then sequentially awaits `userKeysToSign` for every transaction in a `for` loop. Each call invokes `userKeysRequiredToSign`, which performs expensive async operations (including `computeSignatureKey`). A single authenticated `GET /transactions/sign` request causes O(N) sequential async operations with no upper bound, exhausting server CPU, memory, and DB connection resources.

## Finding Description

**Root cause — unbounded DB fetch before the per-item loop:**

In `transactions.service.ts` lines 295–299, `repo.find` is called with no `take` or `skip`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip`
});
``` [1](#0-0) 

Pagination parameters are accepted by the function signature but are only used **after** all processing completes, to slice the already-built result array:

```typescript
items: result.slice(offset, offset + limit),
``` [2](#0-1) 

**Sequential per-transaction expensive operations:**

Lines 301–309 show a sequential `await` inside a `for` loop — each iteration calls `userKeysToSign`, which delegates to `userKeysRequiredToSign` (involving `computeSignatureKey`, an async Hedera key-resolution operation):

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [3](#0-2) 

`userKeysToSign` delegates directly to `userKeysRequiredToSign` with the `transactionSignatureService`: [4](#0-3) 

**Endpoint is still active and reachable:**

The controller registers `GET /sign` with only standard auth guards, despite the "NO LONGER USED BY FRONT-END" comment: [5](#0-4) 

## Impact Explanation
A single authenticated HTTP `GET /transactions/sign` request causes the NestJS server to:
1. Load every non-terminal transaction from the DB into memory (no row cap).
2. Sequentially await `computeSignatureKey` (Hedera key resolution) for each transaction.

With N = 10,000 transactions, this means ≥10,000 sequential async key-resolution operations in one request, holding the Node.js event loop and DB connection pool for the entire duration. This results in **service unavailability or severe degradation** for all concurrent users — request timeouts, DB connection pool starvation, and potential OOM crashes.

## Likelihood Explanation
Any authenticated user — with no special privileges — can trigger this with a single HTTP GET request. No crafted payload is required. The number of non-terminal transactions grows organically as the platform is used, making the vulnerability worse over time. Every registered user possesses a valid session token.

## Recommendation
1. **Apply DB-level pagination before the loop**: pass `take: limit` and `skip: offset` directly into `repo.find(...)` to cap the number of rows fetched per request.
2. **Pre-filter at the DB level**: push the "user has keys to sign" logic into a SQL/query-builder predicate so only relevant transactions are fetched, eliminating the need to iterate all non-terminal transactions.
3. **Remove or gate the unused endpoint**: since the endpoint is annotated as no longer used by the front-end, either remove it or protect it with a stricter rate-limit or admin-only guard.
4. **Parallelize with a concurrency cap** (short-term mitigation): replace the sequential `for...await` loop with `Promise.all` with a bounded concurrency limit (e.g., using `p-limit`) to reduce wall-clock time while a proper fix is implemented.

## Proof of Concept
```
GET /transactions/sign HTTP/1.1
Authorization: Bearer <any_valid_user_token>
```
With a large number of non-terminal transactions in the system, the server will sequentially process each one, blocking the event loop and exhausting DB connections for the duration of the request. No special privileges or crafted payload are required.
