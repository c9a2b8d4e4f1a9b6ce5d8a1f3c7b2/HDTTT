### Title
Transaction Creator Can Modify Approver Structure After Approvals Are Collected, Silently Invalidating Existing Signatures

### Summary

The `updateTransactionApprover` function in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` allows the transaction creator to change the `userId` of an approver (clearing their existing signature/approval) or increase the `threshold` of an approver tree, **without checking the current transaction status or whether approvals have already been collected**. This is the direct analog of the external report: a privileged actor modifies a bounding parameter without checking existing state that depends on it.

### Finding Description

The `getCreatorsTransaction` helper (lines 624–644) only verifies that the caller is the transaction creator. It performs no check on the transaction's current status. [1](#0-0) 

As a result, `updateTransactionApprover` can be called at any point in the transaction lifecycle — including when the transaction is in `WAITING_FOR_EXECUTION` status (meaning all required approvals have already been collected).

**Path 1 — `userId` replacement after approval:**
When `dto.userId` is provided, the service unconditionally clears the existing approver's `signature`, `userKeyId`, and `approved` fields and replaces the user, with no check on whether the old approver had already signed: [2](#0-1) 

**Path 2 — `threshold` increase after approvals are collected:**
When `dto.threshold` is provided, the only guard is that the new threshold does not exceed the total number of child approvers. There is no check on how many of those approvers have already approved: [3](#0-2) 

The same gap exists in `removeTransactionApprover` (lines 534–544), which also calls `getCreatorsTransaction` with no status guard and can remove an approver who has already signed. [4](#0-3) 

The controller exposes both operations over HTTP with no additional guards: [5](#0-4) 

### Impact Explanation

**Concrete scenario:**

1. Creator creates a transaction with an approver tree: `threshold=1`, `approvers=[UserA, UserB]`.
2. UserA calls `POST /approve` — their signature is stored, the transaction advances to `WAITING_FOR_EXECUTION`.
3. Creator calls `PATCH /transactions/:id/approvers/:approverAId` with `{ userId: UserC }`.
4. UserA's `signature`, `userKeyId`, and `approved` fields are silently set to `undefined` in the database. UserC is now the required approver but has not approved.
5. The transaction is now stuck: it cannot proceed without UserC's approval, but UserC has received no notification.
6. If the Hedera transaction's `validStart` deadline passes before UserC approves, the transaction expires and must be recreated from scratch.

The already-collected approval is silently discarded with no warning to UserA or UserC. The creator can repeat this indefinitely to prevent a transaction from ever reaching execution.

### Likelihood Explanation

The creator must call the `PATCH` endpoint intentionally. However:
- There is no system-level guard preventing this call on an already-approved transaction.
- The creator may do this accidentally (e.g., correcting a typo in the approver list after approvals have started arriving).
- In an organizational setting with many concurrent transactions, this is a realistic operational mistake.
- A malicious creator could use this to indefinitely delay a transaction they no longer wish to execute, without canceling it outright (which would be more visible).

### Recommendation

Add a transaction status check inside `getCreatorsTransaction` (or at the top of `updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover`) that rejects modifications when the transaction is not in `NEW` or `WAITING_FOR_SIGNATURES` status. Specifically, modifications to the approver structure should be forbidden once the transaction has reached `WAITING_FOR_EXECUTION` or any terminal state.

Additionally, when `userId` is changed, the system should check whether the existing approver has already signed and either reject the change or explicitly notify both the old and new approver.

### Proof of Concept

```
# 1. Creator creates transaction T with approver UserA (threshold=1)
POST /transactions/:id/approvers
Body: { approversArray: [{ userId: <UserA_id> }] }

# 2. UserA approves — transaction moves to WAITING_FOR_EXECUTION
POST /transactions/:id/approvers/approve
Body: { userKeyId: ..., signature: ..., approved: true }

# 3. Creator replaces UserA with UserC — UserA's signature is silently cleared
PATCH /transactions/:id/approvers/:approverAId
Body: { userId: <UserC_id> }

# Result: transaction is back to needing approval, UserC is unaware,
# UserA's approval is gone, and the transaction may expire.
```

The root cause is in `approvers.service.ts` at the `updateTransactionApprover` function: [6](#0-5) 

No status check is performed before any of the three mutation paths (`userId`, `threshold`, `listId`) are applied.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
