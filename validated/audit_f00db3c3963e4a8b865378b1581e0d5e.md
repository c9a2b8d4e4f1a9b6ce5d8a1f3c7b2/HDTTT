### Title
Missing State Validation in `removeUserKey` Causes Permanent Transaction Lock for Active Transactions

### Summary
When a user removes their own user key via `removeUserKey`, the system does not verify whether that key is currently referenced as a `creatorKey` on any active transactions. After the key is soft-deleted, TypeORM no longer loads it as a relation (no `withDeleted: true`), causing `transaction.creatorKey` to resolve to `null`. Every subsequent creator-only operation — cancel, archive, execute, remove — calls `getTransactionForCreator`, which unconditionally throws `UnauthorizedException` when `creatorKey` is null, permanently locking those transactions in their current state with no recovery path.

### Finding Description

**Root cause — missing pre-condition check in `removeUserKey`:**

`removeUserKey` soft-deletes a key after only verifying ownership. It never checks whether the key is currently used as a `creatorKey` on any non-terminal transaction. [1](#0-0) 

**Broken relation load in `getTransactionById`:**

The transaction fetch loads `creatorKey` as a standard relation with no `withDeleted: true`. Once the key is soft-deleted, TypeORM silently returns `null` for that relation. [2](#0-1) 

**Authorization gate that always fails when `creatorKey` is null:**

Every creator-only operation routes through `getTransactionForCreator`. With `creatorKey` null, `transaction.creatorKey?.userId` is `undefined`, which never equals `user.id`, so the check always throws. [3](#0-2) 

**All creator-only operations are blocked:**

`cancelTransaction`, `archiveTransaction`, `executeTransaction`, and `removeTransaction` all call `getTransactionForCreator` as their first step. [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

**Exploit flow:**

1. User creates one or more transactions using key ID `K`. Transactions enter `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.
2. User calls `DELETE /user-keys/{K}` (their own key). `removeUserKey` passes the ownership check and soft-deletes the key.
3. Any attempt to cancel, archive, execute, or remove those transactions now fails with `UnauthorizedException` because `creatorKey` resolves to `null`.
4. No admin-level cleanup path exists — `removeTransaction` also calls `getTransactionForCreator` and fails identically.
5. Transactions are permanently stuck in a non-terminal, non-actionable state.

### Impact Explanation

Active transactions whose `creatorKey` has been soft-deleted become permanently unmanageable. They cannot be canceled, archived, executed, or removed by anyone. They remain in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely, consuming scheduler resources and polluting the transaction list. This constitutes an unrecoverable corruption of transaction state — a permanent freeze reachable through a normal user action.

### Likelihood Explanation

Key rotation is a routine operation for any security-conscious user. A user who rotates or cleans up old keys without realizing those keys are still referenced as `creatorKey` on active transactions will silently trigger this condition. No special attacker capability is required — only a valid authenticated session and ownership of the key. The condition is also irreversible once triggered.

### Recommendation

In `removeUserKey` (and `removeKey`), before soft-deleting a key, query for active (non-terminal) transactions where `creatorKeyId = id`. If any exist, either:
- Reject the deletion with a descriptive error listing the affected transaction IDs, or
- Require the caller to first cancel/archive those transactions.

Additionally, load `creatorKey` with `withDeleted: true` in `getTransactionById` so that soft-deleted keys are still visible for authorization checks, preventing silent null-dereference.

### Proof of Concept

```
1. Authenticate as User A.
2. POST /transactions  →  create transaction T using creatorKeyId = K.
   Response: { id: T, status: "WAITING_FOR_SIGNATURES" }

3. DELETE /user-keys/K  →  soft-deletes key K.
   Response: 200 OK

4. DELETE /transactions/T  →  attempt to cancel transaction T.
   Response: 401 Unauthorized ("Only the creator has access to this transaction")

5. PATCH /transactions/T/cancel  →  same result.
   Response: 401 Unauthorized

6. Transaction T is now permanently stuck in WAITING_FOR_SIGNATURES
   with no recovery path available to any user or admin via the API.
```

  async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (softRemove) {
      await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
      await this.repo.softRemove(transaction);

      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }

  async cancelTransactionWithOutcome(
    id: number,
    user: User,
  ): Promise<CancelTransactionOutcome> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (transaction.status === TransactionStatus.CANCELED) {
      return CancelTransactionOutcome.ALREADY_CANCELED;
    }

    if (!this.cancelableStatuses.includes(transaction.status)) {
      throw new BadRequestException(ErrorCodes.OTIP);
    }

    const updateResult = await this.repo
      .createQueryBuilder()
      .update(Transaction)
      .set({ status: TransactionStatus.CANCELED })
      .where('id = :id', { id })
      .andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })
      .execute();

    if (updateResult.affected && updateResult.affected > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        [{
          entityId: id,
          additionalData: {
            transactionId: transaction.transactionId,
            network: transaction.mirrorNetwork,
          },
        }],
      );

      return CancelTransactionOutcome.CANCELED;
    }

    // Race-safe fallback: state changed between read and update, so re-check current status.
    const latestTransaction = await this.getTransactionForCreator(id, user);
    if (latestTransaction.status === TransactionStatus.CANCELED) {
      return CancelTransactionOutcome.ALREADY_CANCELED;
    }
    if (!this.cancelableStatuses.includes(latestTransaction.status)) {
      throw new BadRequestException(ErrorCodes.OTIP);
    }
    throw new ConflictException('Cancellation conflict');
  }

  async executeTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }

  async getTransactionForCreator(id: number, user: User) {
    const transaction = await this.getTransactionById(id);

    if (!transaction) {
      throw new BadRequestException(ErrorCodes.TNF);
    }

    if (transaction.creatorKey?.userId !== user?.id) {
