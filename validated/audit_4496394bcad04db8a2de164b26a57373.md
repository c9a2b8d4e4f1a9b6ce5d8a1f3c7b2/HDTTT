### Title
Unbounded Array in `uploadSignatureMaps` Allows Authenticated DoS via Resource Exhaustion

### Summary
The `POST /transactions/signers` endpoint accepts an unbounded array of `UploadSignatureMapDto` objects with no size cap enforced at any layer. Any authenticated user can submit an arbitrarily large array, causing the server to spawn unlimited parallel database queries and SDK deserialization operations via `Promise.all`, exhausting the Node.js event loop, database connection pool, and heap memory. This can render the API unavailable for all other users.

### Finding Description

**Root cause:** `CreateTransactionObserversDto` enforces `@ArrayMinSize(1)` but no `@ArrayMaxSize()`. More critically, the `uploadSignatureMap` controller endpoint accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array-level size constraint at all. [1](#0-0) 

The controller accepts the raw body array and passes it directly to the service: [2](#0-1) 

Inside `uploadSignatureMaps`, the service calls `loadTransactionData` (two `IN`-clause DB queries over all IDs), then immediately fans out into `Promise.all` over every element of the array: [3](#0-2) 

Each parallel branch in `validateAndProcessSignatures` performs:
- `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive deserialization
- Signature map iteration and `addSignature` calls
- Comparison of serialized transaction bytes [4](#0-3) 

After that, `persistSignatureChanges` runs a database transaction that bulk-updates transactions, notification receivers, and inserts signers — all proportional to the input array size. [5](#0-4) 

There is no guard, middleware, or DTO decorator that caps the number of elements in the submitted array.

The same pattern exists in `importSignatures` on the transactions controller, which also accepts `UploadSignatureMapDto[]` with no size limit: [6](#0-5) 

### Impact Explanation

A single malicious authenticated user can POST a JSON array with thousands of `{ id, signatureMap }` entries. The server will:
1. Issue two unbounded `IN (...)` SQL queries (potentially thousands of IDs)
2. Spawn thousands of concurrent `Promise.all` branches, each deserializing SDK transactions and processing signature maps
3. Open a database transaction that bulk-updates thousands of rows

This exhausts the PostgreSQL connection pool, saturates the Node.js event loop, and can OOM-crash the API process — causing complete service unavailability for all legitimate users. Recovery requires a process restart.

### Likelihood Explanation

Any user who has completed registration and JWT authentication can reach `POST /transactions/signers`. No admin or special role is required. The endpoint is designed for normal user workflow (uploading signatures for multiple transactions at once), so the array-accepting behavior is intentional and documented. An attacker needs only a valid JWT token and knowledge of any valid transaction IDs (which can be enumerated or guessed, or the attacker can create their own transactions). This is a low-effort, high-impact attack reachable by a malicious normal user.

### Recommendation

1. Add `@ArrayMaxSize(N)` to the `UploadSignatureMapDto[]` array at the controller or DTO level, where `N` is a practical business limit (e.g., 50–100 items per request).
2. Apply the same cap to `CreateTransactionObserversDto.userIds`.
3. Replace `Promise.all` with a concurrency-limited alternative (e.g., `p-limit`) for the signature processing fan-out, so even within the allowed limit, parallel DB pressure is bounded.
4. Add request-body size limits at the HTTP layer (NestJS `bodyParser` `limit` option) as a defense-in-depth measure.

### Proof of Concept

```
POST /transactions/signers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.2@1000.000000000": { "<pubkey_hex>": "<sig_hex>" } } } },
  { "id": 1, "signatureMap": { ... } },
  ... (repeated 10,000 times)
]
```

**Expected outcome:** The server spawns 10,000 parallel `Promise.all` branches, each performing SDK deserialization and DB queries. The PostgreSQL connection pool is exhausted, the Node.js heap grows unboundedly, and the API becomes unresponsive to all other requests. Legitimate users receive timeouts or 503 errors until the process is restarted.

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }

  private async persistSignatureChanges(
    validationResults: any[],
    user: User,
  ) {
    const signers = new Set<TransactionSigner>();
    let notificationsToDismiss: number[] = [];

    // Prepare batched operations
    const transactionsToUpdate: { id: number; transactionBytes: Buffer }[] = [];
    const notificationsToUpdate: { userId: number; transactionId: number }[] = [];
    const signersToInsert: { userId: number; transactionId: number; userKeyId: number }[] = [];
    const transactionsToProcess: { id: number; transaction: Transaction }[] = [];

    for (const result of validationResults) {
      if (result.error) {
        console.error(`[TX ${result.id}] Validation failed: ${result.error}`);
        continue;
      }

      const { id, transaction, sdkTransaction, userKeys, isSameBytes } = result;

      // Skip if nothing to do - no signatures were added to the transaction
      // AND no new signers were inserted (the signature can be present on the transaction
      // if collated by an outside or 'offline' method)
      if (isSameBytes && userKeys.length === 0) continue;

      // Collect updates
      if (!isSameBytes) {
        transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
        transactionsToUpdate.push({ id, transactionBytes: transaction.transactionBytes });
      }

      // Collect inserts
      if (userKeys.length > 0) {
        const newSigners = userKeys.map(userKey => ({
          userId: user.id,
          transactionId: id,
          userKeyId: userKey.id,
        }));
        signersToInsert.push(...newSigners);
      }

      transactionsToProcess.push({ id, transaction });
      notificationsToUpdate.push({ userId: user.id, transactionId: transaction.id });
    }

    // Execute in single transaction
    try {
      await this.dataSource.transaction(async manager => {
        // Bulk update transactions
        if (transactionsToUpdate.length > 0) {
          await this.bulkUpdateTransactions(manager, transactionsToUpdate);
        }

        // Bulk update notifications
        if (notificationsToUpdate.length > 0) {
          const updatedNotificationReceivers = await this.bulkUpdateNotificationReceivers(manager, notificationsToUpdate);

          // To maintain backwards compatibility and multi-machine support, we send off a dismiss event.
          emitDismissedNotifications(
            this.notificationsPublisher,
            updatedNotificationReceivers,
          );

          notificationsToDismiss = updatedNotificationReceivers.map(nr => nr.id);
        }

        // Bulk insert signers
        if (signersToInsert.length > 0) {
          const results = await this.bulkInsertSigners(manager, signersToInsert);
          results.forEach(signer => signers.add(signer));
        }
      });
    } catch (err) {
      console.error('Database transaction failed:', err);
      throw new BadRequestException(ErrorCodes.FST);
    }

    return {
      transactionsToProcess,
      signers,
      notificationsToDismiss,
    };
  }
