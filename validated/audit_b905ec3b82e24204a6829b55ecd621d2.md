Based on my analysis of the codebase, I found a valid analog to the M-2 vulnerability.

### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Progressive DoS for All API Users

### Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction from the database without any SQL `LIMIT`, then sequentially calls `userKeysToSign` (which triggers mirror-node API calls) for each row. Pagination is applied only after the full in-memory scan completes. As the number of active transactions grows through normal use, each invocation of this endpoint becomes progressively more expensive, eventually causing severe latency or service unavailability for all users sharing the API.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method issues an unbounded database query:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / LIMIT
});
```

It then iterates over every returned row with a sequential `await` inside the loop:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

Pagination is applied only after the full scan:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
```

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `TransactionSignatureService.computeSignatureKey`. That method makes one or more async calls to the mirror-node cache service (`AccountCacheService.getAccountInfoForTransaction`) per transaction. [1](#0-0) [2](#0-1) 

The `whereForUser` filter excludes only five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). Every other transaction — across all users and organizations — is loaded into memory and processed. [3](#0-2) 

### Impact Explanation

- **Memory exhaustion**: Loading thousands of `Transaction` rows (with `groupItem` relations) into the Node.js heap can exhaust server memory.
- **CPU/latency exhaustion**: The sequential `await` loop means each mirror-node round-trip blocks the next iteration. With N active transactions, response time grows as O(N × mirror-node latency).
- **Shared-service DoS**: The API is a single NestJS process. A slow or hung request occupies an event-loop tick and a DB connection, degrading all concurrent users.
- **Amplification by normal use**: Every new transaction created by any user permanently increases the cost of this endpoint for every other user who calls it.

### Likelihood Explanation

- The endpoint is reachable by any authenticated user — no elevated privilege is required.
- Normal platform operation (organizations creating many transactions) naturally grows the dataset over time.
- A malicious authenticated user can deliberately create large numbers of transactions (each with a unique `transactionId`) to accelerate the growth, since there is no per-user transaction creation cap visible in the service.
- The throttler modules present (`ip-throttler`, `user-throttler`) rate-limit request frequency but do not bound the per-request work, so they do not mitigate this issue.


### Recommendation

Apply the database-level `take` limit **before** the loop, not after:

```typescript
// Push the filter into SQL so only candidate transactions are fetched
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: SOME_REASONABLE_CAP,   // e.g. 500 or a configurable max
  skip: offset,
});
```

If exact pagination over the filtered result set is required, move the `userKeysToSign` check into a database-side subquery (similar to how `getTransactionsToApprove` uses a recursive CTE), so the DB engine can filter and paginate in one query instead of loading all rows into the application layer. [4](#0-3) 

### Proof of Concept

1. Register as a normal user and obtain a JWT.
2. Create N transactions (e.g., N = 5 000) via `POST /transactions`, each with a valid but distinct `transactionId`. All will remain in `WAITING_FOR_SIGNATURES` status.
3. Call `GET /transactions/sign` (the endpoint backed by `getTransactionsToSign`) with any pagination parameters.
4. Observe: the server loads all N transactions from PostgreSQL, then sequentially awaits `userKeysToSign` for each one. Response time grows linearly with N. At large N the request times out or the process OOMs, making the endpoint unavailable for all other users.

The root cause — fetching an unbounded result set and iterating it with per-row async I/O before applying pagination — is the direct analog of the Opyn Finance `userDepositIndex`/`userWithdrawIndex` unbounded-array iteration pattern described in M-2. [1](#0-0)

          TransactionStatus.EXPIRED,

          TransactionStatus.ARCHIVED,

        creatorKey: true,
        groupItem: true,
      },
      skip: offset,
      take: limit,
    };

    const [transactions, total] = await this.repo
      .createQueryBuilder()
      .setFindOptions(findOptions)
      .where(
        new Brackets(qb =>
          qb.where(whereForUser).andWhere(

              with recursive "approverList" as
                (
                  select * from "transaction_approver"

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
