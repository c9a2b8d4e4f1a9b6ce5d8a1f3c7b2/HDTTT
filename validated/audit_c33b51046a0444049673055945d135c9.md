Audit Report

## Title
Malicious Verified User Can Spam Identical-Looking Transactions to Confuse Approvers Into Approving Malicious Payloads

## Summary
Any verified user can create unlimited transactions with identical `name` and `description` fields but different (malicious) `transactionBytes`, then assign the same approvers as a legitimate transaction. Approvers see all assigned transactions in their approval queue with no way to distinguish real from fake by human-readable metadata alone, creating a confusion attack that can result in approvers cryptographically signing a malicious Hedera transaction body.

## Finding Description

**Root cause — no uniqueness on display metadata:**

`createTransactions` enforces uniqueness only on the Hedera `transactionId` (`payerAccountId@validStartTime`). There is no uniqueness or rate constraint on `name` or `description`. [1](#0-0) 

Since each new transaction uses a different `validStart` nanosecond, the attacker trivially bypasses this check by varying the Hedera transaction's valid-start timestamp. The `name` and `description` fields are free-form strings with no cross-transaction uniqueness enforcement. [2](#0-1) 

**Approver assignment — only creator of the *target* transaction is checked:**

`createTransactionApprovers` calls `getCreatorsTransaction` to verify the caller created the target transaction. It places no restriction on which user IDs may be designated as approvers — it only checks that the user exists. [3](#0-2) [4](#0-3) 

The attacker creates their own fake transactions and assigns any valid user IDs as approvers — this is entirely within the intended API contract.

**User enumeration — `GET /users` accessible to all verified users:** [5](#0-4) 

`getUsers` returns all organization users to any verified user (non-admin path returns `this.repo.find()` with no filter). [6](#0-5) 

**Approver queue — no creator filter:**

`getTransactionsToApprove` returns every transaction where the authenticated user appears as an approver with `approved IS NULL`, regardless of who created the transaction. [7](#0-6) 

**Approval records a cryptographic signature over the stored `transactionBytes`:**

When an approver calls `approveTransaction`, the service verifies the supplied signature against the stored `transactionBytes` and persists it. Approving the wrong transaction means the approver has cryptographically endorsed the malicious bytes. [8](#0-7) 

**Rate limiting — sufficient to create many decoy transactions:**

The per-user throttler allows 100 requests per minute and 10 per second. [9](#0-8) 

**Exploit flow:**

1. Attacker (verified user) calls `GET /users` to collect all user IDs.
2. Attacker observes or guesses that a legitimate transaction named "Monthly Payroll Transfer" is pending approval by users A and B.
3. Attacker creates N fake transactions with `name="Monthly Payroll Transfer"`, `description="..."`, but with `transactionBytes` encoding a transfer to the attacker's account. Each fake uses a slightly different `validStart` to satisfy the uniqueness check.
4. Attacker calls `POST /transactions/:fakeId/approvers` for each fake, assigning users A and B.
5. Users A and B now see N+1 identical-looking entries in their approval queue. A careless approver approves the wrong entry.
6. The malicious transaction accumulates the required approvals and, once signers also sign, is submitted to the Hedera network.

## Impact Explanation

An approver who approves the wrong transaction provides a valid cryptographic signature over malicious `transactionBytes`. If the transaction also collects the required signer signatures (which may overlap with approvers in many org configurations), the malicious Hedera transaction is executed on-chain — e.g., an unauthorized HBAR transfer or a file update with attacker-controlled content. This is a direct, irreversible on-chain action.

## Likelihood Explanation

The attacker only needs to be a verified member of the organization — the lowest privilege level above anonymous. No admin rights, no leaked secrets, and no cryptographic breaks are required. User IDs are freely enumerable via `GET /users`. The attack is scriptable within the existing rate limits (100 req/min). Organizations with high transaction volume are especially susceptible because approvers routinely process many similar-looking entries.

## Recommendation

1. **Enforce creator identity in the approval queue UI and API**: Surface the creator's identity prominently alongside each transaction in the approval queue so approvers can distinguish legitimate from attacker-created transactions.
2. **Add a `transactionHash` display field**: Expose the `transactionHash` in the approval queue so approvers can verify the bytes they are signing match the expected transaction.
3. **Restrict approver assignment to transactions the assigning user did not create alone**: Consider requiring that approvers can only be assigned by the creator, and that the creator's identity is verified against an org-level policy.
4. **Rate-limit transaction creation per user more aggressively**: Reduce the per-user transaction creation rate or add a per-user daily cap to limit the number of decoy transactions an attacker can create.
5. **Add a uniqueness constraint or warning on `(name, description, creatorKeyId)`**: Warn or block when a user creates multiple active transactions with identical display metadata.

## Proof of Concept

```bash
# Step 1: Enumerate users
curl -H "Authorization: Bearer $ATTACKER_TOKEN" https://api/users
# Returns all user IDs including target approvers A and B

# Step 2: Create N fake transactions with identical name/description but malicious bytes
for i in $(seq 1 10); do
  VALID_START=$(date +%s).$i  # vary nanoseconds to bypass uniqueness check
  TX_BYTES=$(build_malicious_transfer_tx $VALID_START)  # transfer to attacker account
  TX_ID=$(curl -X POST -H "Authorization: Bearer $ATTACKER_TOKEN" \
    -d "{\"name\":\"Monthly Payroll Transfer\",\"description\":\"...\",\"transactionBytes\":\"$TX_BYTES\",...}" \
    https://api/transactions | jq .id)

  # Step 3: Assign target approvers A and B to each fake transaction
  curl -X POST -H "Authorization: Bearer $ATTACKER_TOKEN" \
    -d "{\"approversArray\":[{\"userId\":$USER_A},{\"userId\":$USER_B}]}" \
    https://api/transactions/$TX_ID/approvers
done

# Step 4: Users A and B now see 10+ identical "Monthly Payroll Transfer" entries
# in GET /transactions/approve — one of which transfers funds to the attacker
```

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
