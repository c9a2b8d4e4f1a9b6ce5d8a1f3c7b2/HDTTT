Audit Report

## Title
`removeTransaction()` Lacks Status Guard, Allowing Creators to Overwrite Terminal Transaction States

## Summary
`TransactionsService.removeTransaction()` performs no status check before forcibly overwriting a transaction's `status` to `CANCELED` and soft-deleting the record. Any authenticated user who is the creator of a transaction can call `DELETE /transactions/:id` to corrupt the audit trail of an already-executed or otherwise finalized transaction.

## Finding Description

**Root cause â€” missing state guard in `removeTransaction()`:**

`cancelTransactionWithOutcome()` correctly enforces a status allowlist via `this.cancelableStatuses` (`NEW`, `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`): [1](#0-0) 

It also uses a conditional `WHERE status IN (...)` clause in the UPDATE query, making it race-safe: [2](#0-1) 

`removeTransaction()` performs no equivalent check. After verifying only creator ownership via `getTransactionForCreator`, it unconditionally:
1. Overwrites `status` â†’ `CANCELED` (line 633)
2. Soft-deletes the record (line 634)
3. Emits a `TRANSACTION_STATUS_UPDATE` notification with the corrupted state (lines 639â€“648) [3](#0-2) 

The service defines `terminalStatuses` (`EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, `ARCHIVED`, `REJECTED`) but never references this list inside `removeTransaction()`: [4](#0-3) 

**Exposed endpoint:**

The `DELETE /transactions/:id` controller endpoint calls `removeTransaction` with `softRemove: true` and requires only standard JWT authentication â€” no elevated role: [5](#0-4) 

The controller-level guards are `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` â€” all standard user-level guards: [6](#0-5) 

## Impact Explanation

A creator can call `DELETE /transactions/:id` on a transaction whose status is `EXECUTED`, `FAILED`, `EXPIRED`, `REJECTED`, or `ARCHIVED`. The result:

- The database record's `status` is overwritten from its true terminal value to `CANCELED`, permanently corrupting the audit trail.
- The record is soft-deleted (`deletedAt` set), hiding it from normal queries for all participants (observers, signers, approvers).
- A `TRANSACTION_STATUS_UPDATE` notification is broadcast to all subscribers with the false `CANCELED` status, propagating the corrupted state system-wide.
- Observers and signers â€” who have a legitimate interest in the true outcome â€” lose visibility into whether the transaction was actually executed on the Hedera network.

The on-chain Hedera state is unaffected, but the off-chain record of what happened is permanently falsified.

## Likelihood Explanation

- **Attacker profile:** Any authenticated, verified user who created a transaction. No admin or privileged role is required.
- **Precondition:** The attacker must be the creator of the target transaction (enforced by `getTransactionForCreator`).
- **Trigger:** A single `DELETE /transactions/:id` HTTP request after the transaction reaches a terminal state.
- **Detectability:** Low â€” the soft-delete hides the record from normal queries; the notification system actively propagates the false `CANCELED` state, making it appear legitimate.

## Recommendation

Add a status guard to `removeTransaction()` mirroring the pattern in `cancelTransactionWithOutcome()`. Before performing any mutation, check whether the transaction's current status is in a set of removable (non-terminal) statuses and throw a `BadRequestException` if not. For example:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  // Guard: prevent mutation of terminal states
  if (this.terminalStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OTIP);
  }

  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction);
  }
  // ...
}
```

The existing `terminalStatuses` array is already defined on the service and can be reused directly. [4](#0-3) 

## Proof of Concept

1. Authenticate as a verified user (User A) and create a transaction. Note the returned `id` (e.g., `42`).
2. Wait for the transaction to reach `EXECUTED` status (or any other terminal state).
3. Issue the following request as User A:
   ```
   DELETE /transactions/42
   Authorization: Bearer <User A's JWT>
   ```
4. Observe:
   - HTTP 200 response with `true`.
   - The database record for transaction `42` now has `status = CANCELED` and a non-null `deletedAt`.
   - All subscribers receive a `TRANSACTION_STATUS_UPDATE` notification reporting `CANCELED` for transaction `42`.
   - Observers and signers can no longer see the transaction in normal queries, and the true `EXECUTED` status is permanently lost from the off-chain record.
