### Title
Unbounded Loop in `getTransactionsToSign` Causes Server-Side DoS via Unbounded Database Fetch and Per-Transaction Mirror Node Calls

### Summary

`TransactionsService.getTransactionsToSign` fetches **all** active transactions from the database with no row limit, then iterates over every result calling `userKeysToSign` (which internally calls `computeSignatureKey`, involving async mirror node HTTP requests and cache claim/poll loops). Pagination is applied only **after** the full iteration. Any authenticated user can trigger this endpoint to cause unbounded server resource consumption that grows linearly with the total number of active transactions in the system.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method issues a database query with no `take` limit:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then iterates over **every** returned transaction, calling `userKeysToSign` per transaction:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Pagination is applied only **after** the full loop completes:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [3](#0-2) 

Each call to `userKeysToSign` → `computeSignatureKey` → `AccountCacheService.getAccountInfoForTransaction` may trigger a mirror node HTTP fetch and/or a cache claim/poll loop. The `tryClaimRefresh` helper polls with up to `maxAttempts = 20` iterations at `pollIntervalMs = 500ms` per attempt — up to **10 seconds of blocking per account lookup per transaction**. [4](#0-3) 

By contrast, `getTransactionsToApprove` correctly applies `skip: offset, take: limit` at the database level before any iteration, demonstrating the intended pattern. [5](#0-4) 

The `whereForUser` filter only excludes terminal statuses (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED), meaning all `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION` transactions are included — exactly the transactions that are most likely to accumulate in a busy organization. [6](#0-5) 

### Impact Explanation

As the number of active transactions grows (e.g., a large organization with hundreds of pending multi-sig transactions), each call to `GET /transactions/sign` causes the server to:

1. Load **all** active transactions from the database into memory.
2. For each transaction, perform async mirror node HTTP calls and cache operations.
3. Block the Node.js event loop for the duration of all sequential `await` calls.

This causes request timeouts, memory exhaustion, and degraded service for all users on the platform. A single authenticated user repeatedly calling this endpoint can sustain the DoS condition.

### Likelihood Explanation

- Any verified/authenticated user can call this endpoint — no special role required.
- The `PaginationParams` decorator enforces `size <= 100` on other endpoints, but `getTransactionsToSign` bypasses this by fetching all rows before slicing.
- The condition worsens naturally as the platform grows; no adversarial setup is needed beyond having many active transactions. [7](#0-6) 

### Recommendation

Apply the database-level row limit **before** the loop. Pre-filter at the SQL level using a subquery (similar to how `getTransactionsToApprove` uses a recursive CTE) to identify only transactions the user needs to sign, then apply `skip`/`take` to that filtered set. This avoids loading and processing all active transactions in memory.

Alternatively, add a hard cap (e.g., `take: 500`) to the initial `repo.find` call as an immediate mitigation, and track a cursor/offset so users can page through results across multiple requests.

### Proof of Concept

1. Create an organization with many active transactions in `WAITING_FOR_SIGNATURES` status (e.g., 500+).
2. As any authenticated user, repeatedly call `GET /transactions/sign?page=1&size=10`.
3. Observe that each request causes the server to load all 500+ transactions and perform per-transaction mirror node lookups, regardless of the requested page size of 10.
4. Server response time grows linearly with the number of active transactions; at sufficient scale, requests time out and the service becomes unavailable.
