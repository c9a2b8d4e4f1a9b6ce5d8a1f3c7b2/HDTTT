### Title
Docker Containers Run As Root and Multiple Services Exposed to LAN By Default

### Summary
All application Dockerfiles omit a `USER` instruction, causing every container to run with root privileges. Simultaneously, the production `docker-compose.prod.yaml` exposes NATS (4222), PostgreSQL (5432), pgAdmin (5050), and Redis (6380) to the local network without authentication, allowing any LAN actor to interact with these services directly.

### Finding Description

**Root runs as root (CBST-04 analog):**

None of the five Dockerfiles specify a non-root user:

- `back-end/apps/api/Dockerfile` — no `USER` instruction
- `back-end/apps/chain/Dockerfile` — no `USER` instruction
- `back-end/apps/notifications/Dockerfile` — no `USER` instruction
- `back-end/redis/Dockerfile` — no `USER` instruction
- `back-end/typeorm/Dockerfile` — no `USER` instruction

All containers default to running as `root` (UID 0). [1](#0-0) [2](#0-1) [3](#0-2) 

**Services exposed to LAN (CBST-06 analog):**

In both `docker-compose.yaml` (dev) and `docker-compose.prod.yaml` (production), the following services bind to all interfaces (`0.0.0.0`) by default:

| Service | Port binding | Authentication |
|---|---|---|
| NATS | `4222:4222` | None |
| PostgreSQL | `5432:5432` | Password only (trivial default `postgres`) |
| pgAdmin | `5050:80` | Default credentials (`admin@admin.com` / `pgadmin4`) |
| Redis | `6380:6379` | None configured | [4](#0-3) [5](#0-4) 

The example env files confirm the trivial default credentials shipped with the project: [6](#0-5) [7](#0-6) 

Additionally, the k6 load-testing Grafana container is configured with **anonymous admin access** (no password required at all): [8](#0-7) 

Note: The Helm chart (`charts/`) does correctly set `runAsNonRoot: true` and `runAsUser: 10001`, but this protection is absent from the Docker Compose deployment path used for local and production deployments. [9](#0-8) 

### Impact Explanation

**Running as root:** Any container escape or RCE vulnerability in the application grants the attacker root-level access inside the container. Combined with Docker socket mounts or kernel exploits, this can escalate to full host compromise.

**LAN-exposed services without authentication:**
- **NATS (4222):** Any LAN actor can publish arbitrary messages to NATS JetStream topics, injecting malicious transaction events or disrupting the chain scheduler and notification services.
- **PostgreSQL (5432):** With the trivial default password `postgres`, a LAN actor can read or modify all transaction records, user keys, and organization data.
- **pgAdmin (5050):** With default credentials `admin@admin.com` / `pgadmin4`, a LAN actor gains a full GUI to the database.
- **Redis (6380):** No authentication; a LAN actor can read cached session data or poison the cache.

### Likelihood Explanation

Any user on the same local network (office, shared hosting, cloud VPC without proper security groups) can reach these ports. The default credentials are documented in the repository's example env files, making exploitation trivial. The production compose file (`docker-compose.prod.yaml`) is the intended deployment path, making this a realistic production risk.

### Recommendation

1. **Add a non-root user to all Dockerfiles:**
```dockerfile
RUN addgroup -g 10001 appgroup && \
    adduser -u 10001 -G appgroup -s /sbin/nologin -D appuser
USER appuser
```

2. **Bind internal services to localhost only in docker-compose:**
```yaml
ports:
  - "127.0.0.1:4222:4222"   # NATS
  - "127.0.0.1:5432:5432"   # PostgreSQL
  - "127.0.0.1:5050:80"     # pgAdmin
  - "127.0.0.1:6380:6379"   # Redis
```

3. **Enable NATS authentication** using the `--user`/`--pass` or token flags.

4. **Enable Redis authentication** via `requirepass` in `redis.conf`.

5. **Replace trivial default credentials** in example env files with placeholder values that force operators to set their own.

6. **Remove anonymous admin access** from the Grafana k6 container (`GF_AUTH_ANONYMOUS_ENABLED=false`).

### Proof of Concept

On any machine on the same LAN as a deployed instance using the default configuration:

```bash
# Connect to NATS and subscribe to all subjects (no credentials needed)
nats sub --server nats://<host>:4222 ">"

# Connect to PostgreSQL with default credentials
psql -h <host> -U postgres -d postgres
# Password: postgres

# Access pgAdmin web UI
curl http://<host>:5050
# Login: admin@admin.com / pgadmin4

# Connect to Redis (no auth)
redis-cli -h <host> -p 6380 KEYS "*"
``` [4](#0-3) [1](#0-0)
