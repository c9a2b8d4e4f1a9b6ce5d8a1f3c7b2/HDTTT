**Key findings from the actual code:**

**Controller** (`approvers.controller.ts`, lines 103–113): The `removeTransactionApprover` handler **does** call `getCreatorsTransaction(transactionId, user)` before invoking the service — so the claim that the check is "structurally unreachable" is inaccurate at the controller level. [1](#0-0) 

**However**, the controller check only verifies the user is the creator of the `transactionId` in the URL path. It does **not** verify that the approver `id` in the path actually belongs to that `transactionId`. The service's `removeTransactionApprover` then deletes by `id` alone with no cross-reference check. [2](#0-1) 

This is a real IDOR: a user who is the creator of **any** transaction can supply their own `transactionId` to pass the guard, then supply a foreign approver `id` to delete an approver on a transaction they do not own.

Audit Report

## Title
IDOR in `removeTransactionApprover`: Missing Approver-to-Transaction Binding Allows Cross-Transaction Approver Deletion

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of the transaction identified by `:transactionId`, but never verifies that the approver identified by `:id` actually belongs to that same transaction. A user who is the creator of any transaction can pass the authorization guard using their own `transactionId` while supplying an approver `id` from a transaction they do not own, causing unauthorized deletion of that approver.

## Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the delete handler calls `getCreatorsTransaction(transactionId, user)` to confirm the caller owns the transaction at `:transactionId`: [3](#0-2) 

It then immediately calls `removeTransactionApprover(id)` with the approver `:id` from the URL, with no check that this approver belongs to `transactionId`.

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `removeTransactionApprover` accepts only `id: number`, performs a bare existence check, and deletes: [2](#0-1) 

By contrast, `updateTransactionApprover` explicitly fetches the root node and verifies `rootNode.transactionId === transactionId` before calling `getCreatorsTransaction`: [4](#0-3) 

The `getCreatorsTransaction` guard that enforces creator-only access: [5](#0-4) 

## Impact Explanation
Any user who is the creator of at least one transaction can:
1. Enumerate approver IDs from API responses (sequential integers).
2. Call `DELETE /transactions/<own_transactionId>/approvers/<foreign_approver_id>`.
3. The guard passes (they own `<own_transactionId>`), and the foreign approver is deleted without any ownership check.
4. This removes approvers from transactions they do not own, collapsing multi-party approval requirements and potentially allowing transactions to reach `WAITING_FOR_EXECUTION` without the required approvals.

## Likelihood Explanation
The precondition is that the attacker must be the creator of at least one transaction (not merely any registered user, as the original report states). Approver IDs are sequential integers returned in API responses. No privileged access beyond normal transaction creation is required. The attack is trivially scriptable.

## Recommendation
In the controller's delete handler, after `getCreatorsTransaction`, verify that the approver being deleted actually belongs to the authorized `transactionId`. The pattern already exists in `updateTransactionApprover`: fetch the root node of the approver and assert `rootNode.transactionId === transactionId` before proceeding with deletion. Alternatively, move this binding check into `removeTransactionApprover` in the service, accepting both `id` and `transactionId` (and optionally `user`) as parameters, mirroring the signature of `updateTransactionApprover`.

## Proof of Concept
1. Attacker (User A) creates Transaction T1 — they are its creator.
2. Victim (User B) creates Transaction T2 with approver record `id=42`.
3. Attacker calls: `DELETE /transactions/T1/approvers/42`
4. Controller calls `getCreatorsTransaction(T1, UserA)` → passes (UserA owns T1).
5. Controller calls `removeTransactionApprover(42)` → approver 42 (belonging to T2) is soft-deleted with no further check.
6. T2's approval workflow is now broken; the intended approver has been silently removed.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
