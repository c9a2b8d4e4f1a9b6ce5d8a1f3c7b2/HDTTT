### Title
Unbounded Loop with Per-Iteration Async Calls in `getTransactionsToSign` Causes Server Resource Exhaustion

---

### Summary

`TransactionsService.getTransactionsToSign` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every result calling `await this.userKeysToSign(transaction, user)` — an async operation that performs additional database queries — before applying pagination. Any authenticated user can trigger this endpoint, and as the number of active transactions grows the server performs O(N) sequential async calls per request, exhausting CPU, memory, and database connection pool resources.

---

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues an unbounded `repo.find` with no `take` clause: [1](#0-0) 

The result set — potentially thousands of rows — is then iterated with a sequential `await` inside the loop: [2](#0-1) 

Pagination (`limit`, `offset`) is only applied **after** all N iterations complete: [3](#0-2) 

`userKeysToSign` delegates to `userKeysRequiredToSign` → `TransactionSignatureService`, which performs additional database queries (and potentially mirror-node HTTP calls) to resolve key requirements per transaction: [4](#0-3) 

The `whereForUser` filter only excludes terminal-status transactions; it does **not** bound the result set size: [5](#0-4) 

This is the direct analog of the DyDx finding: an external/async call (`userKeysToSign`) is made once per element inside a loop whose bound is determined by external state (the number of active transactions in the database), with no cap enforced at the query level.

---

### Impact Explanation

- **CPU / event-loop saturation**: N sequential `await` calls block the NestJS worker for the full duration of the request.
- **Database connection exhaustion**: Each `userKeysToSign` call opens additional queries; with many concurrent requests the connection pool is drained.
- **Memory pressure**: All N transaction rows are loaded into heap before any result is returned.
- **Service unavailability**: Repeated calls from one or more users with many associated transactions can degrade or crash the API service for all users.

---

### Likelihood Explanation

- **No privilege required**: Any authenticated user (normal product flow) can call `GET /transactions/to-sign`.
- **Attacker-controlled loop bound**: A user can be added as a signer or observer to an arbitrarily large number of transactions, directly inflating N.
- **Amplification**: The endpoint is likely polled by the frontend on page load and on refresh, so a single malicious account with many transactions causes repeated O(N) work.
- **No existing guard**: There is no rate limit, no `take` cap on the initial query, and no concurrency throttle visible in the service.

---

### Recommendation

Apply the database-level `take` limit **before** the loop, not after:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  skip: offset,   // push pagination into the DB query
  take: limit,
});
```

Because `userKeysToSign` must be called to determine whether a transaction belongs in the result, a cursor-based approach is preferable: fetch a page of candidates, filter, and repeat until the page is full. Alternatively, push the key-matching logic into a single SQL query so the database does the filtering and the application loop is eliminated entirely.

---

### Proof of Concept

**Preconditions**: Attacker has a valid account. Attacker (or a colluding second account) creates or is added as a signer to a large number of transactions (e.g., 5 000) that remain in `WAITING_FOR_SIGNATURES` status.

**Trigger**:
```
GET /transactions/to-sign?page=1&size=10
Authorization: Bearer <attacker_token>
```

**Observed behaviour**:
1. `getTransactionsToSign` issues `repo.find` with no `take` → returns all 5 000 rows.
2. The `for` loop calls `await this.userKeysToSign(tx, user)` 5 000 times sequentially.
3. Each call issues additional DB queries via `TransactionSignatureService`.
4. The request holds a DB connection and CPU for the entire duration.
5. Repeated or concurrent calls exhaust the connection pool and degrade the API for all users.

**Expected outcome**: Server response time degrades linearly with N; under sustained load the API becomes unavailable. [6](#0-5)

```typescript
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
