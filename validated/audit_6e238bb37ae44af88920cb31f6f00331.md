Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign()` Causes Server Resource Exhaustion via Single Authenticated Request

## Summary
`TransactionsService.getTransactionsToSign()` fetches all active (non-terminal) transactions from the database with no row limit, then serially awaits `userKeysToSign()` for every row before applying pagination. Any authenticated, verified user can trigger `GET /transactions/sign` to force the server to load and process an unbounded number of transactions, exhausting memory, CPU, and database connections.

## Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` issues an unconstrained `find()` with no `take` parameter:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then serially awaits `userKeysToSign()` for every returned row:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Pagination is applied only after the full loop completes:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

Each `userKeysToSign()` call delegates to `userKeysRequiredToSign()`, which: (1) calls `attachKeys()` (DB query), (2) deserializes transaction bytes via `SDKTransaction.fromBytes()` (CPU), (3) calls `transactionSignatureService.computeSignatureKey()` (potentially expensive key-structure traversal), and (4) may issue additional `entityManager.find(UserKey, ...)` queries. [4](#0-3) 

The controller exposes this as `GET /transactions/sign`, guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin role required. The endpoint is still live despite the `/* NO LONGER USED BY FRONT-END */` comment: [5](#0-4) 

The class-level guard configuration confirms no elevated privilege is required: [6](#0-5) 

## Impact Explanation

With N active (non-terminal) transactions in the system, a single `GET /transactions/sign` request causes the API process to:

1. Load all N transaction rows (plus `groupItem` relations) into heap memory.
2. Serially await N async `userKeysToSign()` calls, each performing transaction deserialization, cryptographic key-structure traversal, and one or more database round-trips.

Because the loop is sequential and awaited, a single request holds the Node.js event loop busy proportional to N, blocking other requests from being processed. At scale this exhausts heap memory and saturates the event loop, causing request timeouts and degraded or unavailable service for all users. The condition worsens naturally as the organization accumulates active transactions over time.

**Impact: High** — complete API service degradation achievable by a single authenticated user.

## Likelihood Explanation

**Likelihood: Low-Medium.** The attacker must hold a valid, verified organization account (no admin privilege needed). The endpoint is reachable via the REST API despite the front-end no longer calling it. The attack requires only one HTTP request; no flooding or DDoS tooling is needed. The severity grows organically as the organization accumulates active transactions.

## Recommendation

1. **Push pagination into the database query**: Add `skip: offset` and `take: limit` to the `find()` call in `getTransactionsToSign()`. This requires restructuring the logic so that key-signing eligibility is determined at the query level (e.g., via a subquery or a pre-filter on the user's public keys stored in `publicKeys` column) rather than in application memory.
2. **Add a hard cap**: Even with pagination, enforce a maximum `limit` (e.g., 100) to prevent a caller from requesting an arbitrarily large page.
3. **Remove or gate the deprecated endpoint**: Since the front-end no longer uses `GET /transactions/sign`, consider removing it or protecting it with an admin-only guard until it is properly refactored.
4. **Parallelize with a concurrency limit**: If the in-memory loop must remain temporarily, replace the serial `for...await` with a bounded concurrent executor (e.g., `p-limit`) to reduce event-loop blocking.

## Proof of Concept

```
# Authenticated request as any verified user
GET /transactions/sign?page=1&limit=10
Authorization: Bearer <valid_jwt>
```

With a large number of active (non-terminal) transactions in the database, this single request causes the server to:
- Execute `SELECT * FROM transaction WHERE status NOT IN (...)` with no `LIMIT` clause, loading all rows into memory.
- Serially call `userKeysToSign()` N times, each involving `SDKTransaction.fromBytes()`, `computeSignatureKey()`, and `entityManager.find(UserKey, ...)`.
- Block the Node.js event loop for the entire duration, starving all concurrent requests of processing time.

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
};
