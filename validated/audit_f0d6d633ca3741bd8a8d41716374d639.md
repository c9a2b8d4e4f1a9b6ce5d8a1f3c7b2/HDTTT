Audit Report

## Title
IDOR in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` performs an ownership check on the URL-supplied `transactionId` but then passes the unrelated approver `:id` directly to `removeTransactionApprover(id)` with no cross-validation. Any authenticated user who is the creator of at least one transaction can delete approvers belonging to any other transaction in the system.

## Finding Description

The controller's `removeTransactionApprover` handler (lines 102–113) performs two independent operations without linking them:

1. It calls `getCreatorsTransaction(transactionId, user)` to verify the caller owns the transaction at `:transactionId`.
2. It then calls `removeTransactionApprover(id)` with the approver `:id` from the URL — with no check that this approver belongs to the just-authorized transaction. [1](#0-0) 

The service method `removeTransactionApprover` (lines 534–544) only verifies the approver record exists, then soft-deletes it. It never checks `approver.transactionId` against the authorized `transactionId`: [2](#0-1) 

The `getCreatorsTransaction` helper (lines 640–641) enforces `creatorKey.userId === user.id` only for the transaction in the URL path — it has no knowledge of which transaction the approver being deleted actually belongs to: [3](#0-2) 

By contrast, `updateTransactionApprover` in the same service correctly resolves the approver's root node and validates `rootNode.transactionId !== transactionId` before proceeding, demonstrating the intended pattern that `removeTransactionApprover` omits: [4](#0-3) 

## Impact Explanation
An attacker who is the creator of any transaction (even a trivial self-created one) can delete approvers from any other transaction in the organization. This directly undermines the multi-signature governance model: by removing required approvers from a victim transaction, the attacker can either prevent it from ever reaching the execution threshold or silently reduce the approval requirements, allowing a transaction to proceed without the intended oversight. This is an unauthorized, persistent state change with direct impact on transaction integrity and organizational governance.

## Likelihood Explanation
The precondition is minimal: the attacker only needs to be a registered, verified organization member who has created at least one transaction — a normal user action requiring no admin privileges, leaked credentials, or special roles. Approver IDs are sequential integers, making enumeration trivial. The endpoint is a standard authenticated REST call requiring no special tooling.

## Recommendation
In `removeTransactionApprover` (controller or service), after fetching the approver record, resolve its root node via `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId` before proceeding with deletion — exactly as `updateTransactionApprover` does at lines 386–391:

```typescript
// In removeTransactionApprover (service), add after fetching the approver:
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [5](#0-4) 

## Proof of Concept

1. Attacker (user A) creates Transaction T1 — they are now its creator.
2. Victim (user B) creates Transaction T2 with approver record ID = 99 (belonging to T2).
3. Attacker sends:
   ```
   DELETE /transactions/T1/approvers/99
   Authorization: Bearer <attacker_jwt>
   ```
4. Server calls `getCreatorsTransaction(T1, userA)` → passes (A owns T1).
5. Server calls `removeTransactionApprover(99)` → fetches approver 99 (which belongs to T2), finds it exists, and soft-deletes it.
6. Approver 99 is permanently removed from T2 without any authorization check against T2's ownership. [6](#0-5)
