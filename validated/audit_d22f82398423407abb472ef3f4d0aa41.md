## Audit Report

## Title
Any Authenticated User Can Register a Transaction with an Arbitrary Payer Account ID, Enabling `transactionId` Collision DoS

## Summary
The `POST /transactions` endpoint derives the `transactionId` (format: `payerAccountId@validStart`) directly from user-supplied transaction bytes without verifying that the payer account belongs to the submitting user. Because the uniqueness check rejects any second registration of the same `transactionId` in a non-terminal status, a malicious authenticated user can pre-register a transaction using a victim's payer account ID and a predicted `validStart`, permanently blocking the victim from registering that specific `(payer, validStart)` combination until the attacker's record reaches a terminal status.

## Finding Description

**Root cause — no payer account ownership check:**

In `validateAndPrepareTransaction`, the only identity check performed is that the `creatorKeyId` belongs to the submitting user and that the user's registered public key signed the raw transaction bytes: [1](#0-0) 

After that check passes, the `transactionId` is extracted verbatim from the parsed SDK transaction: [2](#0-1) 

There is no assertion that `sdkTransaction.transactionId.accountId` (the payer) matches any account owned by the submitting user. Any authenticated user can embed any Hedera account ID as the payer in their transaction bytes, sign those bytes with their own registered key, and submit successfully.

**Uniqueness enforcement that makes the collision fatal:**

In `createTransactions`, the service rejects any registration whose `transactionId` already exists in a non-terminal status: [3](#0-2) 

The first registration wins; all subsequent registrations with the same `transactionId` receive a `400 Bad Request` with message `Transactions already exist: <transactionId>`.

## Impact Explanation

- **Denial of Service on transaction registration**: The victim cannot register any transaction whose `(payer, validStart)` pair has been pre-claimed by the attacker. The error thrown is `Transactions already exist: 0.0.5000@T`.
- **Organization workflow disruption**: In multi-signature organization mode, a transaction that cannot be registered cannot collect signatures or be executed, stalling collaborative workflows.
- **Attacker's colliding record is harmless on-chain** (it will never execute because the real payer's key won't sign it), but the database slot is occupied until the attacker's record reaches a terminal status (`CANCELED`/`REJECTED`/`ARCHIVED`), after which the victim can retry.

## Likelihood Explanation

- **Preconditions**: Attacker must be a registered, verified user on the same organization server — no admin privilege required.
- **Payer account discoverability**: Hedera account IDs are public; an attacker who knows the victim's organization can enumerate likely payer accounts.
- **`validStart` predictability**: The Hedera SDK's `Timestamp.generate()` uses the current wall-clock time with nanosecond precision. Blind guessing across a ±1-second window (1 billion slots) is impractical without rate limiting. However, in organization mode where transaction preparation is collaborative and draft bytes are shared, the attacker can observe the exact `validStart` from the draft before the creator submits, making a targeted collision straightforward.
- **Realistic scenario**: Most impactful in organization mode where the attacker is a legitimate participant who can observe draft bytes before the creator submits.

## Recommendation

In `validateAndPrepareTransaction`, after parsing the SDK transaction, verify that `sdkTransaction.transactionId.accountId` corresponds to an account associated with the submitting user. Specifically:

1. Retrieve the set of Hedera account IDs linked to the submitting user's registered keys.
2. Assert that `sdkTransaction.transactionId.accountId.toString()` is a member of that set.
3. Throw a `BadRequestException` (e.g., `ErrorCodes.SNMP` or a new dedicated error code) if the check fails.

This ensures that only the legitimate owner of a payer account can register transactions under that account's ID, eliminating the collision vector entirely.
