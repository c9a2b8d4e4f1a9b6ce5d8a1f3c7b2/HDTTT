### Title
Any Authenticated User Can Post Comments on Arbitrary Transactions Due to Missing Authorization Check

### Summary
The `createComment` endpoint in `CommentsController` is protected by authentication guards but contains no authorization check to verify the requesting user has any relationship to the target transaction. The code itself acknowledges this gap with an explicit TODO comment. Any registered, verified user can post comments on any transaction in the system — including transactions they did not create, sign, observe, or approve.

### Finding Description
In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the `createComment` handler at line 22 is decorated with controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but carries no per-route authorization guard or service-layer ownership check:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
```

The developer-left TODO at line 22 explicitly acknowledges the missing control. Contrast this with other state-mutating endpoints in the same codebase that enforce ownership:

- `cancelTransaction` / `archiveTransaction` / `executeTransaction` delegate to service methods that verify the requesting user's relationship to the transaction.
- `createTransaction` additionally requires `HasKeyGuard`.
- `removeTransactionApprover` calls `this.approversService.getCreatorsTransaction(transactionId, user)` before mutating state.

No equivalent check exists in `CommentsService.createComment`. The attacker path is:

1. Register as a normal user (no admin or privileged role needed).
2. Obtain a valid JWT via `POST /auth/login`.
3. Issue `POST /transactions/:transactionId/comments` with any `transactionId` value and arbitrary comment body.
4. The comment is persisted against the target transaction with no ownership validation. [1](#0-0) 

### Impact Explanation
- **Unauthorized state mutation**: Any authenticated user can inject comments into transactions they have no legitimate access to, polluting the audit trail and comment history for all participants.
- **Information probing**: A successful `201` response confirms the existence of a transaction with a given ID, enabling enumeration of valid transaction IDs across the organization.
- **Integrity degradation**: Transaction comment threads — used by signers, approvers, and observers to coordinate — can be flooded or manipulated by any user in the organization, disrupting multi-party workflows.

### Likelihood Explanation
Exploitation requires only a valid user account (obtainable via the standard `POST /auth/signup` flow initiated by any admin, or by being an existing organization member). No privileged role, leaked credential, or special network position is needed. The attack is trivially scriptable against any known or enumerated transaction ID.

### Recommendation
Add an authorization check inside `createComment` (or in `CommentsService.createComment`) that verifies the requesting user is a creator, signer, observer, or approver of the target transaction — mirroring the pattern already used in `ApproversService.getCreatorsTransaction` and `TransactionsService.verifyAccess`. The existing `verifyAccess` method in `TransactionsService` already encodes this logic and can be reused:

```typescript
// In CommentsService.createComment or the controller:
const transaction = await this.transactionsService.getTransactionById(transactionId);
const hasAccess = await this.transactionsService.verifyAccess(transaction, user);
if (!hasAccess) throw new UnauthorizedException();
``` [2](#0-1) 

### Proof of Concept
1. Log in as User A (transaction creator): `POST /auth/login` → receive `tokenA`.
2. Create a transaction as User A: `POST /transactions` → receive `transactionId = 42`.
3. Log in as User B (unrelated user, not creator/signer/observer/approver of tx 42): `POST /auth/login` → receive `tokenB`.
4. As User B, post a comment: `POST /transactions/42/comments` with `Authorization: Bearer tokenB` and body `{"content": "attacker comment"}`.
5. **Expected (secure)**: `401 Unauthorized` or `403 Forbidden`.
6. **Actual**: `201 Created` — comment is persisted on transaction 42 under User B's identity, with no relationship to that transaction required. [3](#0-2)
