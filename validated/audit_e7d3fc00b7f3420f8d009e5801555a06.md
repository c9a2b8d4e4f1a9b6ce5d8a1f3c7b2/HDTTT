Audit Report

## Title
`removeTransactionApprover` Does Not Update Parent Threshold, Causing Approval DoS

## Summary
`removeTransactionApprover` in `ApproversService` delegates to `removeNode`, which recursively soft-deletes a node and all its descendants but never adjusts the parent tree node's `threshold` column. When a child approver is removed from a threshold group, the parent's `threshold` can exceed the count of remaining live children, making the approval requirement mathematically impossible to satisfy until the creator manually corrects the threshold via a separate API call.

## Finding Description

**File**: `back-end/apps/api/src/transactions/approvers/approvers.service.ts`

`removeTransactionApprover` fetches the target approver and immediately delegates to `removeNode`: [1](#0-0) 

`removeNode` performs a recursive SQL soft-delete (setting `deletedAt = now()`) on the target node and all its descendants. It never reads or modifies the parent node's `threshold` column: [2](#0-1) 

By contrast, `updateTransactionApprover` (the `dto.listId === null` branch) correctly handles this case: it loads the parent with its `approvers` relation, computes `newParentApproversLength = parent.approvers.length - 1`, and if `newParentApproversLength < parent.threshold`, it updates the parent's threshold to match: [3](#0-2) 

`removeTransactionApprover` has no equivalent adjustment. The controller only verifies the caller is the transaction creator before invoking `removeTransactionApprover`: [4](#0-3) 

The `TransactionApprover` entity stores `threshold` as an independent nullable integer column with no DB-level constraint tying it to the live child count: [5](#0-4) 

## Impact Explanation

After the deletion, the parent node's `threshold` exceeds the count of live children. The `isApproved` utility (used by both the front-end and back-end status logic) evaluates approval by checking `approvals.length >= approver.threshold`: [6](#0-5) 

With fewer live approvers than the threshold demands, this condition can never be `true`. The transaction remains stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. Recovery requires the creator to manually call `PATCH /transactions/{id}/approvers/{parentId}` with `{ threshold: N }` where `N` is the new live child count. The `updateTransactionApprover` threshold branch validates `dto.threshold <= approver.approvers.length` against the current DB state (which correctly reflects the reduced count after soft-delete), so the fix is possible — but only if the creator is aware of the inconsistency and takes deliberate action. There is no automatic correction or warning.

## Likelihood Explanation

The trigger is a routine, documented product action: a transaction creator removes one approver from a threshold group. No special privilege beyond being the authenticated transaction creator is required. The creator can trigger this accidentally (e.g., removing the wrong approver from a 2-of-2 group, leaving a 2-of-1 state) or intentionally. The API endpoint (`DELETE /transactions/:transactionId/approvers/:id`) is publicly documented and requires no attacker-controlled input beyond normal authenticated use.

## Recommendation

In `removeTransactionApprover`, after calling `removeNode`, fetch the deleted approver's parent (via `approver.listId`) with its `approvers` relation and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

1. If the parent has no remaining live children after deletion, soft-delete the parent node as well (matching the behavior in `updateTransactionApprover` at line 421–422).
2. Otherwise, if `newParentApproversLength < parent.threshold`, update the parent's `threshold` to `newParentApproversLength`.

This mirrors the existing, correct logic at lines 417–428 of `approvers.service.ts` and closes the inconsistency.

## Proof of Concept

1. Creator creates a transaction and adds a threshold approver tree:
   ```
   POST /transactions/1/approvers
   Body: { approversArray: [{ threshold: 2, approvers: [{ userId: A }, { userId: B }] }] }
   ```
   DB state: parent node `threshold=2`, children = [A, B].

2. Creator deletes child A:
   ```
   DELETE /transactions/1/approvers/{A.id}
   ```
   `removeNode(A.id)` soft-deletes A. Parent's `threshold` remains `2`.

3. DB state: parent node `threshold=2`, live children = [B] (count = 1).

4. User B approves. `isApproved` evaluates: `approvals.length (1) >= threshold (2)` → `false`. The approval requirement is never satisfied.

5. The transaction is stuck. The creator must manually call:
   ```
   PATCH /transactions/1/approvers/{parent.id}
   Body: { threshold: 1 }
   ```
   to restore a satisfiable state — but no system prompt or error guides them to do so.
