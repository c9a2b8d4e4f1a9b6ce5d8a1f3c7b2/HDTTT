### Title
Missing Cross-Transaction Scope Validation in `removeTransactionApprover` Enables Unauthorized Approver Deletion

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user is the creator of `:transactionId`, but never verifies that the approver record identified by `:id` actually belongs to that transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` and a victim's approver `id`.

### Finding Description
In `approvers.controller.ts` (lines 102–113), the `removeTransactionApprover` handler performs two sequential calls:

1. `this.approversService.getCreatorsTransaction(transactionId, user)` — confirms the caller is the creator of the transaction identified by the URL parameter `transactionId`.
2. `this.approversService.removeTransactionApprover(id)` — deletes the approver record identified by the URL parameter `id` with **no check** that this approver belongs to `transactionId`. [1](#0-0) 

The service method `removeTransactionApprover` simply fetches the approver by primary key and removes it: [2](#0-1) 

There is no check that `approver.transactionId === transactionId` (or that the approver's root node belongs to `transactionId`).

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
