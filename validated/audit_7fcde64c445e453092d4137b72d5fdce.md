## Audit Report

## Title
`isNode` Duplicate-Check Logic Silently Skips Threshold Approver Nodes, Allowing Unlimited Duplicate Threshold Nodes to Be Injected into the Approval Tree

## Summary
The `isNode` method in `approvers.service.ts` contains a faulty return expression that makes it unconditionally return `false` for every threshold-type approver node, regardless of whether a matching record already exists in the database. The sole duplicate-insertion guard in `createTransactionApprovers` relies entirely on `isNode`, so any authenticated transaction creator can call `POST /transactions/:id/approvers` repeatedly with the same threshold payload and insert an unbounded number of duplicate threshold nodes, permanently corrupting the approval tree.

## Finding Description

**Root cause — line 665 of `approvers.service.ts`:** [1](#0-0) 

The database query (lines 652–661) constructs a precise `WHERE` clause on `listId`, `userId`, `threshold`, and `transactionId`. When `count > 0`, the record already exists. The extra conjunct `&& typeof approver.userId === 'number'` is the defect:

| Approver type | `typeof approver.userId === 'number'` | `isNode` result when duplicate exists |
|---|---|---|
| User node (`userId` set) | `true` | `true` — duplicate blocked |
| Threshold node (`userId` not set) | `false` | **`false` — duplicate allowed** |

The guard that relies on `isNode`: [2](#0-1) 

Because `isNode` always returns `false` for threshold nodes, this guard never fires for them. The only test covering `isNode` passes a `userId`-bearing object and never exercises the threshold-node path: [3](#0-2) 

## Impact Explanation

**Approval tree corruption / permanent transaction DoS.**

`getTreeStructure` builds the tree by iterating all approver rows and linking children to parents by `listId`. [4](#0-3) 

Duplicate root threshold nodes (all with `listId = null`) each appear as independent roots. The approval evaluation must satisfy every root. If N identical threshold roots exist, N independent sets of approvals are required — a condition that can never be met because the same user cannot approve the same transaction twice (the `userApprovers.every(a => a.signature)` guard at line 563 blocks re-approval): [5](#0-4) 

The transaction is permanently locked in `WAITING_FOR_SIGNATURES` and can never be executed or cancelled through normal flows. Additionally, each duplicate threshold root spawns its own child user-approver rows (each with a distinct `id`), causing unbounded storage growth and degraded query performance for all recursive CTE queries.

## Likelihood Explanation

The attack requires only that the attacker be an authenticated user who has created a transaction — a capability available to every registered user of the system. No admin privileges, no leaked secrets, and no race conditions are required. The endpoint `POST /transactions/:id/approvers` is a standard REST call. The bug is deterministic and reproducible on every call. [6](#0-5) 

## Recommendation

Fix the return statement in `isNode` to remove the erroneous conjunct:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts, line 665
// Before (buggy):
return count > 0 && typeof approver.userId === 'number';

// After (correct):
return count > 0;
```

The database query already encodes all the specificity needed to distinguish node types. The `typeof approver.userId === 'number'` check is redundant and incorrect here. Additionally, add a unit test covering the threshold-node duplicate case to prevent regression.

## Proof of Concept

1. Register as a normal user and create any transaction (attacker becomes creator).
2. Call `POST /transactions/:id/approvers` with a threshold payload:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": 2 }] }] }
   ```
3. `createTransactionApprovers` calls `getCreatorsTransaction` — passes (attacker is creator).
4. `isNode` is called for the threshold node. `typeof approver.userId === 'number'` → `false`. Returns `false`.
5. The threshold node is inserted without any duplicate check.
6. Repeat step 2 N times. N identical threshold root nodes now exist for the same transaction, each with its own child user-approver subtree.
7. The transaction is permanently locked — it requires N independent sets of approvals that can never all be satisfied simultaneously, since a user cannot approve the same transaction twice.
