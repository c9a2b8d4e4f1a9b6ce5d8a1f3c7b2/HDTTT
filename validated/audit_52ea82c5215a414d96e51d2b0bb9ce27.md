Audit Report

## Title
Global IP Rate Limiter Bypass via Attacker-Controlled `X-Forwarded-For` Header Enables Brute Force on All API Endpoints

## Summary
`IpThrottlerGuard` — the sole globally-applied rate limiter — derives its throttle key from the raw `req.headers['x-forwarded-for']` value, which is fully attacker-controlled. By rotating arbitrary strings in this header, any unauthenticated client can bypass the rate limiter entirely. The `/auth/verify-reset` OTP endpoint carries no secondary throttler, leaving it open to unlimited OTP guessing once the IP throttler is bypassed.

## Finding Description

**Root cause — `IpThrottlerGuard.getTracker` trusts a client-supplied header:**

`getTracker` reads `req.headers['x-forwarded-for']` directly as the throttle key before falling back to `req.ip`. [1](#0-0) 

Because `req.headers['x-forwarded-for']` is the raw HTTP header set by the client — not a value resolved by Express's proxy-trust logic — an attacker simply sends `X-Forwarded-For: <arbitrary-string>` and the throttler counts that string as the "IP". Each request with a new fake IP string starts a fresh counter.

**Production amplifier — `trust proxy` is unconditionally enabled:** [2](#0-1) 

`app.enable('trust proxy')` with no hop count or allowlist tells Express to trust the entire `X-Forwarded-For` chain from any source. Critically, this does **not** fix the guard — the guard reads `req.headers['x-forwarded-for']` directly, bypassing Express's own `req.ip` resolution entirely.

**Global guard registration — bypass affects every route:** [3](#0-2) 

`IpThrottlerGuard` is the only rate limiter registered globally via `APP_GUARD`. The `EmailThrottlerGuard` is applied only to `/auth/signup`, `/auth/login`, and `/auth/reset-password`.

**Unprotected OTP endpoint — `/auth/verify-reset` has no secondary throttler:** [4](#0-3) 

The only guards on this endpoint are `JwtBlackListOtpGuard` (checks Redis blacklist) and `OtpJwtAuthGuard` (validates JWT signature). Neither limits the rate of OTP guesses. The OTP JWT is only blacklisted **after a successful verification**, so the attacker can reuse the same unverified OTP JWT for unlimited guesses.

**Wide TOTP window amplifies brute-force feasibility:** [5](#0-4) 

`window: 20` with `step: 60` means codes from ±20 steps (41 total steps) are accepted simultaneously. With an 8-digit OTP space of 10^8, 41 valid codes exist at any moment, giving a per-guess success probability of ~4.1×10^-7. With no rate limit, an attacker can enumerate the full space.

## Impact Explanation

**Account takeover via OTP brute force:** An attacker triggers a password reset for a target account via the public `/auth/reset-password` endpoint (which returns an OTP JWT). With the IP throttler bypassed by rotating `X-Forwarded-For` values, the attacker submits unlimited OTP guesses to `/auth/verify-reset`. A successful guess returns a `verified: true` OTP JWT, which can then be used to call `/auth/set-password` and take over the account. No privileged access is required — only the target's email address.

**Credential spraying on `/auth/login`:** While `EmailThrottlerGuard` on `/auth/login` limits per-email attempts, bypassing the IP throttler removes the only cross-account rate cap, enabling simultaneous credential spraying across many accounts.

**Resource exhaustion:** Any resource-intensive endpoint (transaction creation, key operations) can be flooded without IP-level throttling.

Impact: **4 / 5** — full account takeover achievable with no privileged access.

## Likelihood Explanation

Setting an arbitrary `X-Forwarded-For` header requires only a standard HTTP client (`curl`, Python `requests`, etc.). No authentication, no leaked secrets, and no internal network access are needed. The attacker only needs to know a target's email address to initiate the OTP flow. The attack is fully scriptable and deterministic.

Likelihood: **4 / 5** — trivially reachable by any external attacker.

## Recommendation

1. **Fix `getTracker`:** Remove the `X-Forwarded-For` header read entirely. Use only `req.ip`, which Express resolves correctly when `trust proxy` is configured with a specific hop count or CIDR allowlist (e.g., `app.set('trust proxy', 1)` for a single trusted reverse proxy).

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

2. **Scope `trust proxy`:** Replace `app.enable('trust proxy')` with `app.set('trust proxy', 1)` (or the exact number of trusted proxy hops / a CIDR allowlist) so Express correctly resolves `req.ip` to the real client IP.

3. **Add a secondary throttler to `/auth/verify-reset`:** Apply `EmailThrottlerGuard` (or a dedicated OTP-attempt throttler keyed on the OTP JWT's `jti`/`email` claim) to limit OTP guesses per identity regardless of IP.

4. **Reduce the TOTP window:** A `window: 20` is extremely permissive. Reduce it to `window: 1` or `window: 2` to minimize the number of simultaneously valid codes.

## Proof of Concept

```python
import requests
import itertools

TARGET = "https://example.com"
OTP_JWT = "<token from POST /auth/reset-password>"

# Rotate fake IPs to bypass IpThrottlerGuard
def fake_ip(n):
    return f"10.{(n>>16)&0xff}.{(n>>8)&0xff}.{n&0xff}"

for i, code in enumerate(f"{n:08d}" for n in range(100_000_000)):
    headers = {
        "X-Forwarded-For": fake_ip(i),  # new "IP" every request
        "otp": OTP_JWT,
    }
    r = requests.post(
        f"{TARGET}/auth/verify-reset",
        json={"token": code},
        headers=headers,
    )
    if r.status_code == 200:
        print(f"[+] Valid OTP: {code}")
        print(f"[+] Verified JWT: {r.json()['token']}")
        break
```

Each iteration presents a fresh fake IP to the throttler. Because `IpThrottlerGuard.getTracker` uses `req.headers['x-forwarded-for']` verbatim, every request is counted against a different key and the rate limit is never reached.
