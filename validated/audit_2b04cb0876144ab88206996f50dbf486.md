# Audit Report

## Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Invalid Signatures to Advance Transaction State

## Summary
The `uploadSignatureMaps` endpoint (`POST /transactions/:id/signers`) in `signers.service.ts` accepts and persists signature maps without verifying that the submitted signature bytes are cryptographically valid over the transaction body. An authenticated user can submit a garbage byte string paired with their registered public key; the server records it as a valid signing event, updates `transactionBytes`, potentially advances the transaction to `WAITING_FOR_EXECUTION`, and causes an irrecoverable `FAILED` state when the Hedera network rejects the invalid signature. The `importSignatures` path performs this check; the primary signing path does not.

## Finding Description

**Root cause ã `processTransactionSignatures` calls `addSignature` without verifying bytes:**

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, `processTransactionSignatures` (lines 217ã267) iterates the submitted `SignatureMap`, confirms the public key belongs to the authenticated user via `userKeyMap`, then unconditionally calls `sdkTransaction.addSignature(publicKey, map)` at line 251 without ever calling `publicKey.verify(bodyBytes, signature)`: [1](#0-0) 

The only checks performed are ownership (`userKeyMap.get(raw)` / `userKeyMap.get(publicKey.toStringDer())`), transaction status, and expiry. No cryptographic verification of the signature bytes occurs. [2](#0-1) 

**Contrast with `importSignatures` which DOES verify:**

`back-end/apps/api/src/transactions/transactions.service.ts` `importSignatures` (lines 546ã549) calls `validateSignature` before adding any signature: [3](#0-2) 

`validateSignature` in `back-end/libs/common/src/utils/sdk/transaction.ts` (lines 213ã248) explicitly calls `publicKey.verify(bodyBytes, signature)` and throws on failure: [4](#0-3) 

**Execution-path check is also bypassed:**

`back-end/libs/common/src/execute/execute.service.ts` `getValidatedSDKTransaction` (lines 219ã220) only checks that required public keys are *present* in `_signerPublicKeys`, not that their associated signature bytes are cryptographically valid: [5](#0-4) 

`_signerPublicKeys` is populated by `addSignature` regardless of whether the signature bytes are valid, so this check is bypassed by an invalid signature.

**`IsSignatureMap` decorator does not verify either:**

The DTO-level `IsSignatureMap` decorator only validates structural format (valid account ID, valid transaction ID, non-empty bytes) ã it performs no cryptographic check: [6](#0-5) 

## Impact Explanation

An authenticated user who is a required signer for a transaction can:

1. Submit a `POST /transactions/:id/signers` request containing their registered public key paired with arbitrary/invalid 64-byte signature bytes.
2. `processTransactionSignatures` passes the ownership check (public key is in `userKeyMap`), calls `sdkTransaction.addSignature(publicKey, map)`, and the public key is added to `_signerPublicKeys` without any cryptographic check.
3. `isSameBytes` evaluates to `false` (bytes changed), so `transactionBytes` is updated in the database with the invalid signature, and a `TransactionSigner` row is inserted.
4. `processTransactionStatus` may advance the transaction to `WAITING_FOR_EXECUTION` if the signing threshold is now met.
5. When the transaction is submitted to the Hedera network, it is rejected because the signature is cryptographically invalid.
6. The transaction is marked `FAILED` ã an irrecoverable terminal state.

The transaction is permanently destroyed. Any HBAR or token operations it was meant to perform are lost. A malicious co-signer on a multi-sig threshold-key transaction can unilaterally sabotage any transaction they are required to sign.

## Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated user with at least one registered `UserKey` in the system. No admin or privileged access required.
- **Attack complexity**: Trivial. The attacker crafts a `POST /transactions/:id/signers` request with their real public key but a random 64-byte hex string as the signature value.
- **Reachability**: The endpoint is a standard authenticated API endpoint, reachable by any registered organization user.
- **Detection**: No server-side error is raised; the request succeeds with HTTP 201. The failure only surfaces later when the Hedera network rejects the transaction.

## Recommendation

Add a `publicKey.verify(bodyBytes, signature)` call inside `processTransactionSignatures` before calling `sdkTransaction.addSignature`. The existing `validateSignature` utility in `back-end/libs/common/src/utils/sdk/transaction.ts` already implements this logic correctly and is already imported in the same codebase. The fix is to call `validateSignature(sdkTransaction, map)` (or an equivalent per-key check) at the top of `processTransactionSignatures`, mirroring the pattern used in `importSignatures`: [7](#0-6) 

## Proof of Concept

```
POST /transactions/42/signers
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "id": 42,
  "signatureMap": {
    "0.0.3": {
      "0.0.1234@1700000000.000000000": {
        "<attacker_registered_public_key_hex>": "<64_random_bytes_hex>"
      }
    }
  }
}
```

**Expected (secure) behavior**: HTTP 400, signature validation error.

**Actual behavior**: HTTP 201, `TransactionSigner` row inserted, `transactionBytes` updated with invalid signature, transaction potentially advanced to `WAITING_FOR_EXECUTION`, subsequently submitted to Hedera and rejected, transaction permanently marked `FAILED`.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
