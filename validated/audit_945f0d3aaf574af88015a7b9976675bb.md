Audit Report

## Title
`CreateTransactionApproverDto.threshold` Lacks Minimum Value Validation, Allowing Negative Threshold to Bypass Organizational Approval Workflow

## Summary
`CreateTransactionApproverDto` accepts a `threshold` field decorated only with `@IsNumber()` and `@IsOptional()` — no `@Min(1)` constraint. The service-level guard only rejects `threshold === 0` and `threshold > approvers.length`. A negative value (e.g., `-1`) passes both layers, is persisted to the database, and causes the approval tree to be evaluated as immediately satisfied because `approvals.length >= -1` is always `true`.

## Finding Description

**Root cause — DTO layer:**

`CreateTransactionApproverDto.threshold` has no lower-bound constraint:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
@IsNumber()
@IsOptional()
threshold?: number;   // ← no @Min(1)
``` [1](#0-0) 

By contrast, `UpdateTransactionApproverDto` correctly enforces `@Min(1)`:

```typescript
// back-end/apps/api/src/transactions/dto/update-transaction-approver.dto.ts
@IsNumber()
@Min(1)
@IsOptional()
threshold?: number;
``` [2](#0-1) 

The inconsistency confirms `@Min(1)` was intentionally added to the update path but omitted from the create path.

**Root cause — service layer:**

The threshold guard in `createTransactionApprovers` only rejects `=== 0` and `> approvers.length`:

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
``` [3](#0-2) 

For `threshold = -1` with 1 child approver: `-1 > 1` → `false`, `-1 === 0` → `false`. The check passes.

**Persistence confirmed:**

The threshold is stored using a JavaScript truthiness check:

```typescript
threshold:
  dtoApprover.threshold && dtoApprover.approvers ? dtoApprover.threshold : null,
``` [4](#0-3) 

In JavaScript, `-1` is truthy, so `dtoApprover.threshold && dtoApprover.approvers` evaluates to `dtoApprover.approvers` (truthy), and `threshold: -1` is written to the database.

**Impact on approval evaluation:**

The `isApproved` function evaluates tree nodes as:

```typescript
if (approvals.length >= (approver.threshold || approvals.length)) {
  return true;
}
``` [5](#0-4) 

With `threshold = -1` and `approvals.length = 0`: `-1` is truthy, so the expression becomes `0 >= -1` → `true`. The tree node is immediately evaluated as approved with zero actual approvals.

## Impact Explanation
The organizational approval workflow — the primary multi-party authorization control before a transaction is submitted to the Hedera network — is bypassed. A transaction creator can set `threshold: -1` on an approver tree, causing the approval gate to be satisfied with zero approvals. This enables unauthorized transaction progression under the appearance of a legitimately approved workflow, constituting an integrity failure in the trust model.

## Likelihood Explanation
Any authenticated user who can create transactions (a normal product flow) can trigger this. No privileged access, no leaked credentials, and no special tooling are required — a crafted JSON body to the documented `POST /transactions/:id/approvers` endpoint is sufficient. The inconsistency between the create and update DTOs strongly suggests an oversight rather than intentional design.

## Recommendation
Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring the existing constraint in `UpdateTransactionApproverDto`:

```typescript
import { IsArray, IsNumber, IsOptional, Min, ValidateNested, ArrayMinSize } from 'class-validator';

@IsNumber()
@Min(1)
@IsOptional()
threshold?: number;
``` [1](#0-0) 

Additionally, update the service-level guard to use `< 1` instead of `=== 0` as a defense-in-depth measure:

```typescript
dtoApprover.threshold < 1   // replaces: dtoApprover.threshold === 0
``` [3](#0-2) 

## Proof of Concept

```http
POST /transactions/1/approvers
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 99 }
      ]
    }
  ]
}
```

1. `CreateTransactionApproverDto` passes NestJS `ValidationPipe` — no `@Min` check exists.
2. `approvers.service.ts` passes the guard: `-1 > 1` is `false`, `-1 === 0` is `false`.
3. The truthiness check `dtoApprover.threshold && dtoApprover.approvers` evaluates to truthy (`-1` is truthy), so `threshold: -1` is inserted into the `transaction_approver` table.
4. `isApproved` evaluates `approvals.length >= (approver.threshold || approvals.length)` → `0 >= -1` → `true`.
5. The approval tree is immediately considered satisfied with zero approvals, bypassing the intended multi-party authorization requirement.
