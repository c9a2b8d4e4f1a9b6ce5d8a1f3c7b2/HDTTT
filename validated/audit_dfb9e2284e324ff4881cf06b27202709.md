Audit Report

## Title
IDOR in `removeTransactionApprover`: Any Authenticated User Can Delete Approvers from Any Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId`, but then deletes the approver identified by `:id` with no check that the approver actually belongs to `:transactionId`. An attacker who owns any one transaction can delete approvers from any other transaction by supplying their own `transactionId` and a victim's approver `id`.

## Finding Description

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent, uncorrelated operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id only
``` [1](#0-0) 

`getCreatorsTransaction` only validates that `transaction.creatorKey.userId === user.id` for the supplied `transactionId`: [2](#0-1) 

`removeTransactionApprover` in the service fetches the approver by `id` alone and immediately soft-deletes it and its entire subtree via `removeNode`, with no cross-check against `transactionId`: [3](#0-2) 

`getTransactionApproverById` performs a plain lookup by `id` with no transaction scope: [4](#0-3) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-ownership check before acting — it walks up to the root node and asserts `rootNode.transactionId === transactionId` before proceeding: [5](#0-4) 

The `removeTransactionApprover` path is entirely missing this guard.

## Impact Explanation

1. **Recursive subtree deletion**: `removeNode` performs a recursive SQL soft-delete of the entire approver subtree rooted at the target `id`. A single request can wipe a multi-level approval structure belonging to any transaction. [6](#0-5) 

2. **Bypass of approval governance**: Removing all approvers from a victim's transaction can allow it to proceed to `WAITING_FOR_EXECUTION` without the intended threshold of approvals being met.

3. **Unauthorized state transitions**: `emitTransactionStatusUpdate` is called after every deletion, potentially triggering downstream state changes in the Chain Service for the victim's transaction. [7](#0-6) 

## Likelihood Explanation

- **Attacker preconditions**: Only a valid authenticated JWT. No admin or elevated role required.
- **Approver ID discovery**: Approver IDs are sequential integers. An attacker can observe IDs from their own transactions via `GET /transactions/:id/approvers`, or enumerate adjacent IDs.
- **Attack execution**: A single HTTP `DELETE /transactions/{attacker_transactionId}/approvers/{victim_approver_id}` request is sufficient. The creator check passes because `attacker_transactionId` is legitimately owned; the deletion proceeds against the victim's approver.

## Recommendation

In `removeTransactionApprover` (service), add the same cross-ownership check that `updateTransactionApprover` already performs:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root of the approver tree.
2. Assert `rootNode.transactionId === transactionId` (the URL parameter); throw `UnauthorizedException` if they differ.
3. Then call `getCreatorsTransaction(rootNode.transactionId, user)` to confirm the caller owns that transaction.

This mirrors the existing pattern in `updateTransactionApprover` at lines 386–394 and closes the IDOR. [5](#0-4) 

## Proof of Concept

**Setup:**
- Attacker registers as `userA` and creates `transactionA`, adding an approver (ID = 42) to it.
- Victim `userB` creates `transactionB` with approver tree (root approver ID = 99).

**Attack:**
```http
DELETE /transactions/{transactionA_id}/approvers/99
Authorization: Bearer <userA_jwt>
```

**What happens:**
1. `getCreatorsTransaction(transactionA_id, userA)` → passes (userA owns transactionA).
2. `removeTransactionApprover(99)` → fetches approver 99 (belongs to transactionB), calls `removeNode(99)`, recursively soft-deletes approver 99 and all its children.
3. `emitTransactionStatusUpdate` fires for transactionB, potentially triggering a state change.

**Result:** userB's entire approver tree is silently deleted. No error is returned to the attacker.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
