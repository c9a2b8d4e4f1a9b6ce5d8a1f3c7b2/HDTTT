### Title
`extractSignatureRequirements` Calls `.toString()` on Potentially Null Return Value of `getFeePayerAccountId()`, Causing Unhandled TypeError and DoS of Signature Key Computation

### Summary
In `back-end/libs/common/src/transaction-signature/transaction-signature.service.ts`, the private method `extractSignatureRequirements` calls `transactionModel.getFeePayerAccountId().toString()` at line 71 without first checking whether the return value is `null`. The base model `getFeePayerAccountId()` is explicitly typed to return `AccountId | null` and returns `null` whenever `transactionId` or `transactionId.accountId` is absent. Calling `.toString()` on `null` throws a `TypeError`, crashing the entire `computeSignatureKey` pipeline and making signature key resolution unavailable for any transaction that lacks a fee payer account ID.

### Finding Description

**Root cause — `getFeePayerAccountId()` returns `null`:**

In `back-end/libs/common/src/transaction-signature/model/transaction-base.model.ts`:

```typescript
getFeePayerAccountId(): AccountId | null {
  const payerId = this.transaction.transactionId?.accountId;
  if (payerId) {
    return payerId;
  }
  return null;   // ← explicit null return path
}
``` [1](#0-0) 

The project's own unit tests confirm both null-return paths: [2](#0-1) 

**Vulnerable call site — no null guard before `.toString()`:**

In `back-end/libs/common/src/transaction-signature/transaction-signature.service.ts`, line 71:

```typescript
private extractSignatureRequirements(
  transactionModel: TransactionBaseModel<any>
): SignatureRequirements {
  return {
    feePayerAccount: transactionModel.getFeePayerAccountId().toString(), // ← null dereference
    ...
  };
}
``` [3](#0-2) 

When `getFeePayerAccountId()` returns `null`, JavaScript throws:
```
TypeError: Cannot read properties of null (reading 'toString')
```

This exception is **not caught** inside `extractSignatureRequirements`. It propagates up through `computeSignatureKey`: [4](#0-3) 

And further up through `keysRequiredToSign`: [5](#0-4) 

**Contrast with the frontend model**, which correctly guards the null before use:

```typescript
// front-end/src/renderer/utils/transactionSignatureModels/transaction.model.ts
const feePayerAccountId = this.getFeePayerAccountId();
...
if (feePayerAccountId) {   // ← null check present
  const accountInfo = await accountInfoCache.lookup(
    feePayerAccountId.toString(), ...
  );
}
``` [6](#0-5) 

The backend service omits this guard entirely.

### Impact Explanation

Any code path that calls `computeSignatureKey` on a transaction whose bytes deserialize to an SDK transaction without a `transactionId.accountId` will throw an unhandled `TypeError`. This breaks:

1. **Signature key resolution** — the backend cannot determine which keys are required to sign the transaction.
2. **Transaction status progression** — `keysRequiredToSign` (called during transaction processing) will throw, preventing the transaction from advancing through the `WAITING_FOR_SIGNATURES → READY_FOR_EXECUTION` state machine.
3. **Potential service crash** — depending on NestJS exception handling configuration, an unhandled synchronous throw inside an `async` service method can propagate to the caller as an unhandled rejection, causing the affected request to hang or the worker to crash.

### Likelihood Explanation

The Hedera SDK allows constructing a `Transaction` object without setting a `transactionId` (as demonstrated by the project's own test fixtures in `transaction-base.model.spec.ts`). If any such transaction bytes are stored in the database — whether through a missing validation gate at the API ingestion layer, a future code path, or a crafted import — the crash is deterministic and reproducible. The null-return path is not a theoretical edge case; it is explicitly tested and documented in the type signature.

### Recommendation

Add a null guard in `extractSignatureRequirements` before calling `.toString()`, mirroring the pattern already used correctly in the frontend model:

```typescript
private extractSignatureRequirements(
  transactionModel: TransactionBaseModel<any>
): SignatureRequirements {
  const feePayerAccountId = transactionModel.getFeePayerAccountId();
  if (!feePayerAccountId) {
    throw new Error('Transaction is missing a fee payer account ID');
    // or: return a sentinel / empty string and skip addFeePayerKey
  }
  return {
    feePayerAccount: feePayerAccountId.toString(),
    ...
  };
}
```

Alternatively, change the `SignatureRequirements.feePayerAccount` field type to `string | null` and guard the downstream `addFeePayerKey` call accordingly.

### Proof of Concept

1. Construct any Hedera SDK transaction without calling `setTransactionId()`:
   ```typescript
   const tx = new TransferTransaction(); // no transactionId set
   const bytes = tx.toBytes();
   ```
2. Store `bytes` as the `transactionBytes` of a `Transaction` entity in the backend database (bypassing or before any transactionId validation).
3. Call `computeSignatureKey(transaction)` on the backend service.
4. `SDKTransaction.fromBytes(bytes)` deserializes the transaction; `transactionId` is `undefined`.
5. `TransactionFactory.fromTransaction(sdkTransaction)` returns a model.
6. `extractSignatureRequirements` calls `model.getFeePayerAccountId()` → returns `null`.
7. `null.toString()` → `TypeError: Cannot read properties of null (reading 'toString')`.
8. The exception propagates uncaught through `computeSignatureKey` and `keysRequiredToSign`, crashing the request handler. [3](#0-2) [1](#0-0)
