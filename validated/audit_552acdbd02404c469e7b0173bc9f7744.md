### Title
Terminal-State Manual Transactions Can Be Re-Archived, Overwriting Irreversible Status

### Summary
The `archiveTransaction` function in `transactions.service.ts` contains a logically flawed guard condition that, when `isManual = true`, entirely bypasses the status check. This allows any transaction creator to call `PATCH /transactions/archive/:id` on a manual transaction that has already reached a terminal state (`EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, `REJECTED`), overwriting that terminal status with `ARCHIVED`. This is the direct analog of the external report: a function intended only for in-progress transactions silently accepts terminal-state inputs and mutates state that should be immutable.

### Finding Description

**Root cause — flawed boolean guard in `archiveTransaction`:** [1](#0-0) 

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual          // ← second operand of AND
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

The guard is a logical AND of two conditions:

| Condition | Meaning |
|---|---|
| `status NOT IN [WFS, WFE]` | Transaction is not in an active state |
| `!transaction.isManual` | Transaction is not manual |

When `isManual = true`, the second operand is `false`, making the entire AND expression `false`. The `throw` is never reached. The function proceeds unconditionally to `repo.update(... ARCHIVED)` regardless of the current status.

**Exploit flow:**

1. Attacker (normal authenticated user) creates a transaction with `isManual: true` via `POST /transactions`. [2](#0-1) 
2. The transaction progresses normally and reaches a terminal state — e.g., `EXECUTED` (set by the chain service after on-chain submission), or `CANCELED` (set by the creator or scheduler). [3](#0-2) 
3. Attacker calls `PATCH /transactions/archive/:id`. The controller delegates to `archiveTransaction` with no additional guards. [4](#0-3) 
4. Because `isManual = true`, the status guard is bypassed. The database row is updated: `status = ARCHIVED`. [5](#0-4) 
5. The transaction now appears as `ARCHIVED` to all observers, signers, and audit consumers — its true terminal state is permanently overwritten.

**Secondary path — `executeTransaction` also lacks a status check:** [6](#0-5) 

`executeTransaction` only checks `!transaction.isManual` and never validates the current status. If `validStart > Date.now()`, it writes `isManual = false` to a `CANCELED` or `EXECUTED` transaction. The downstream `executeService.validateTransactionStatus` does protect the actual Hedera submission path, but the `isManual` flag mutation on a terminal-state row is an unguarded state change. [7](#0-6) 

### Impact Explanation

The most severe case is `EXECUTED → ARCHIVED`:

- `EXECUTED` means the transaction was submitted to and confirmed by the Hedera network. This is an on-chain, irreversible fact.
- Overwriting it with `ARCHIVED` erases that fact from the application's state. Observers, signers, and audit consumers see `ARCHIVED` and have no record that the transaction was executed.
- The history endpoint includes `ARCHIVED` in its allowed statuses alongside `EXECUTED`, so the transaction remains visible but with a misleading status. [8](#0-7) 
- Notification routing and email templates are status-driven; an `ARCHIVED` status triggers different (or no) downstream notifications compared to `EXECUTED`. [9](#0-8) 

Other impactful cases: `CANCELED → ARCHIVED` (hides cancellation), `FAILED → ARCHIVED` (hides failure), `REJECTED → ARCHIVED` (hides rejection).

### Likelihood Explanation

- **Attacker profile**: Any authenticated, verified user who has created at least one manual transaction. No admin, operator, or privileged key is required.
- **Preconditions**: The attacker must be the creator of the transaction (`getTransactionForCreator` enforces this). Creating a manual transaction is a standard product flow available to all users.
- **Trigger**: A single HTTP `PATCH` request to `/transactions/archive/:id` after the transaction reaches a terminal state.
- **Detectability**: The API returns `200 OK` with no error, making the mutation silent.

### Recommendation

Fix the guard in `archiveTransaction` to enforce status independently of `isManual`. The intended semantics are: archive is only valid for in-progress manual transactions. The corrected guard:

```typescript
// Only allow archiving when the transaction is actively in-progress
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!archivableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
// Optionally retain the isManual check as an additional constraint:
if (!transaction.isManual) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Apply the same pattern to `executeTransaction`: add an explicit status check before the `isManual` branch to reject terminal-state transactions before any mutation occurs.

### Proof of Concept

```
# Step 1: Authenticate as a normal user and obtain JWT
POST /auth/login  { email, password }
→ { accessToken }

# Step 2: Create a manual transaction
POST /transactions
Authorization: Bearer <accessToken>
Body: { ..., isManual: true, ... }
→ { id: 42, status: "WAITING FOR SIGNATURES", isManual: true }

# Step 3: Wait for or trigger a terminal state
# e.g., let the scheduler execute it → status becomes "EXECUTED"
# or call PATCH /transactions/cancel/42 → status becomes "CANCELED"

# Step 4: Archive the terminal-state transaction
PATCH /transactions/archive/42
Authorization: Bearer <accessToken>
→ 200 OK, true

# Step 5: Verify the status has been overwritten
GET /transactions/42
→ { id: 42, status: "ARCHIVED" }   ← was "EXECUTED" or "CANCELED"
```

The terminal status is permanently overwritten. The on-chain execution record is hidden from all application-layer consumers.

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }

  private async validateTransactionStatus(transaction: Transaction) {
    const { status } = await this.transactionsRepo.findOne({
      where: { id: transaction.id },
      select: ['status'],
    });

    switch (status) {
      case TransactionStatus.NEW:
        throw new Error('Transaction is new and has not been signed yet.');
      case TransactionStatus.FAILED:
        throw new Error('Transaction has already been executed, but failed.');
      case TransactionStatus.EXECUTED:
        throw new Error('Transaction has already been executed.');
      case TransactionStatus.REJECTED:
        throw new Error('Transaction has already been rejected.');
      case TransactionStatus.EXPIRED:
        throw new Error('Transaction has been expired.');
      case TransactionStatus.CANCELED:
        throw new Error('Transaction has been canceled.');
      case TransactionStatus.ARCHIVED:
        throw new Error('Transaction is archived.');
    }
  }
