### Title
Hardcoded Private Keys and Weak Secrets Committed to Repository

### Summary
Multiple hardcoded Hedera private keys and cryptographically weak JWT/OTP secrets are committed directly to the repository across test utilities and automation constants. Any developer who clones or forks the repository gains immediate access to these credentials through the full git history.

### Finding Description
The repository contains hardcoded private keys and weak secrets in at least three distinct locations:

**1. `back-end/apps/api/test/utils/hederaUtils.ts` (lines 35–55)**
Five Hedera account private keys are hardcoded as string literals and exported as named constants used across the entire E2E test suite:

```ts
export const localnet2 = new HederaAccount()

**2. `automation/constants/transactionEnvironment.constants.ts` (line 4)**
The default localnet operator private key is hardcoded as an exported constant:

```ts
export const DEFAULT_LOCALNET_OPERATOR_KEY =
  '0x91132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137';
``` [2](#0-1) 

The same key also appears hardcoded as a fallback default in `automation/k6/helpers/create-complex-accounts.ts` (line 59): [3](#0-2) 

**3. `back-end/apps/api/.env.test` (lines 11, 16)**
A committed `.env` file contains trivially weak secrets for JWT signing and OTP generation:

```
JWT_SECRET=13123
OTP_SECRET=123
``` [4](#0-3) 

### Impact Explanation
- **Private keys**: Any party with repository access (or access to any fork/clone) can extract and use these private keys to sign Hedera transactions on behalf of the associated accounts. If any of these accounts are ever funded or reused outside of a fully isolated local-node environment, funds and account control are immediately compromised.
- **JWT_SECRET / OTP_SECRET**: If a developer accidentally runs the backend against `.env.test` in a staging or shared environment (a realistic mistake given the file is committed and named `.env.test` rather than `.env.example`), an attacker who knows these values (anyone with repo access) can forge arbitrary JWT tokens and bypass OTP authentication entirely, gaining full API access as any user.

**Impact: 5**

### Likelihood Explanation
The repository is structured as a monorepo with a public GitHub presence. The secrets exist in committed files (not just git history), making them trivially discoverable via a simple `grep` or GitHub code search. The `.env.test` file is particularly dangerous because it is a functional configuration file, not a documentation example, and developers routinely copy `.env.*` files as starting points.

**Likelihood: 3**

### Recommendation
1. **Rotate all exposed credentials**: Treat all hardcoded private keys and the JWT/OTP secrets as compromised. Rotate them immediately and verify none of the associated Hedera accounts hold real funds or are used in any non-isolated environment.
2. **Remove secrets from git history**: Use `git filter-repo` or BFG Repo Cleaner to purge the hardcoded values from all commits.
3. **Replace `.env.test` with `.env.test.example`**: Commit only a template with placeholder values (e.g., `JWT_SECRET=REPLACE_ME`). Add `.env.test` to `.gitignore`.
4. **Use environment variables for test keys**: Load localnet keys from environment variables with documented defaults in a `.env.test.example`, rather than hardcoding them in source files.
5. **Add secret scanning**: Enable GitHub secret scanning or add a pre-commit hook (e.g., `gitleaks`, `truffleHog`) to prevent future commits of raw key material.

### Proof of Concept
An attacker with read access to the repository can:

1. Clone the repository.
2. Extract the JWT secret directly from the committed file:
   ```bash
   grep JWT_SECRET back-end/apps/api/.env.test
   # JWT_SECRET=13123
   ```
3. Forge a valid JWT token signed with `13123` targeting any user ID.
4. Use the forged token to authenticate against the API as an arbitrary user, bypassing all authentication controls — if the backend is running with this `.env.test` configuration.

For the private keys:
```bash
grep -r "setPrivateKey\|OPERATOR_KEY" back-end/apps/api/test/utils/hederaUtils.ts automation/constants/
# Immediately yields 5+ raw private keys
```

These keys can be imported into any Hedera SDK client to sign and submit transactions on behalf of the associated accounts.
