### Title
Unbounded Loop in `getTransactionsToSign()` Enables Server-Side DoS via Per-Transaction Mirror Node I/O

### Summary
`getTransactionsToSign()` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then serially calls `userKeysToSign()` — which internally invokes `computeSignatureKey()` and makes async mirror-node HTTP calls — for each transaction before applying pagination. As the number of active transactions grows, a single authenticated request can exhaust server CPU, memory, and outbound I/O, making the API unresponsive for all users.

### Finding Description

**Root cause — no `take` limit on the database query:**

`getTransactionsToSign()` issues a `repo.find()` with no `take` parameter, loading every non-terminal transaction into memory: [1](#0-0) 

**Unbounded per-item async loop:**

Every loaded transaction is then processed sequentially through `userKeysToSign()`: [2](#0-1) 

**`userKeysToSign()` → `computeSignatureKey()` → mirror-node I/O per transaction:**

`userKeysToSign()` delegates to `userKeysRequiredToSign()` → `keysRequiredToSign()` → `transactionSignatureService.computeSignatureKey()`. That function makes one or more async HTTP calls to the Hedera mirror node (via `AccountCacheService.getAccountInfoForTransaction()`) for the fee-payer, signing accounts, and receiver accounts of each transaction: [3](#0-2) 

**Pagination applied only after the full loop completes:**

The `slice()` for pagination happens only after every transaction has been processed, so the `limit` parameter provides no protection against the unbounded work: [4](#0-3) 

**Endpoint is still live:**

The controller exposes this as `GET /transactions/sign` and is guarded only by JWT authentication — no admin role required. The comment "NO LONGER USED BY FRONT-END" confirms it is still reachable: [5](#0-4) 

### Impact Explanation

A single authenticated request to `GET /transactions/sign` when N active transactions exist causes:
- N synchronous DB row deserializations into memory
- Up to N × (number of signing/receiver accounts per transaction) outbound HTTP calls to the Hedera mirror node
- The Node.js event loop is blocked on awaited I/O for the entire duration

With a large enough N (hundreds to thousands of active transactions, which is realistic in an organization deployment), the request never completes within any reasonable timeout, the server's connection pool and memory are exhausted, and the API becomes unresponsive for all users — a full service outage.

### Likelihood Explanation

- **Attacker precondition:** Any valid JWT — a normal registered user.
- **Trigger:** Create a large number of transactions (a normal product workflow), then call `GET /transactions/sign`. No special privilege is needed to create transactions.
- **Amplification:** The endpoint is not rate-limited at the application layer (no evidence of rate limiting on this route), so the attacker can repeat the call continuously.
- **Realistic scenario:** In an organization with hundreds of pending multi-sig transactions (a stated use case of the tool), even legitimate users will trigger severe latency degradation on every call to this endpoint.

### Recommendation

Apply the database-level `take` limit **before** the loop, and push the per-transaction key-eligibility check into the SQL query (or a post-query filter on already-loaded data without additional I/O). Concretely:

1. Add `take: limit` and `skip: offset` to the `repo.find()` call so the DB returns only the page of rows needed.
2. Pre-filter by user key membership in SQL (as `getTransactions()` already does with `signers: { userId: user.id }`) to avoid loading transactions the user cannot sign.
3. If per-transaction mirror-node I/O is unavoidable, run it with bounded concurrency (`Promise.all` with a concurrency limiter) rather than serially, and enforce a hard cap on the number of transactions processed per request.

### Proof of Concept

1. Register a normal user account and obtain a JWT.
2. Create N = 500 transactions in `WAITING_FOR_SIGNATURES` status (all non-terminal, so none are filtered by the `whereForUser` clause).
3. Issue `GET /transactions/sign?page=1&limit=10` with the JWT.
4. Observe: the server fetches all 500 rows, calls `computeSignatureKey()` (with mirror-node I/O) for each, and the request either times out or consumes enough CPU/memory to degrade or crash the API process for all concurrent users.
5. Repeat the request in a loop to sustain the outage. [6](#0-5)

```typescript
  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
