### Title
Unbounded Recursive Depth in `createTransactionApprovers` Enables DoS via Approver Tree Nesting

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` uses a recursive inner function `createApprover` that calls itself for each nested child approver with no depth limit enforced. The `CreateTransactionApproverDto` allows arbitrary self-referential nesting via `approvers?: CreateTransactionApproverDto[]`. Any authenticated user who owns a transaction can submit a deeply nested approver tree payload that exhausts Node.js call stack memory and triggers thousands of database queries within a single transaction, crashing or severely degrading the API service.

### Finding Description

**Root cause — no depth limit on recursive `createApprover`:**

The inner function `createApprover` at line 245 of `approvers.service.ts` calls itself recursively at line 348 for every child in `dtoApprover.approvers`, with no depth counter or maximum depth check anywhere in the function or its callers. [1](#0-0) 

**DTO allows unlimited nesting:**

`CreateTransactionApproverDto` declares `approvers` as a self-referential array with no `@ArrayMaxSize` or depth constraint, meaning `class-validator` and `class-transformer` will happily deserialize and validate an arbitrarily deep JSON tree. [2](#0-1) 

**Only constraint is a 2 MB body size limit:**

The sole guard against oversized input is `app.use(json({ limit: '2mb' }))` in `setup-app.ts`. A chain of 1 000 nesting levels requires only ~30 KB of JSON (each level adds ~30 bytes), well within the 2 MB cap. [3](#0-2) 

**Per-level DB amplification:**

Each recursive invocation of `createApprover` issues up to five database round-trips: `isNode` (COUNT), `findOne` (parent lookup), `getRootNodeFromNode` (recursive SQL CTE), `count` (user existence), and `insert`. At 1 000 nesting levels this produces ~5 000 sequential DB queries inside a single open TypeORM transaction, holding a DB connection and transaction lock for the entire duration. [4](#0-3) 

**Entry point:**

`POST /transactions/:transactionId/approvers` is guarded only by JWT authentication and ownership of the target transaction — no admin role required. [5](#0-4) 

### Impact Explanation

A single crafted request with ~1 000 nesting levels (~30 KB body):
- Exhausts the Node.js call stack (V8 default ~10 000–15 000 frames), crashing the API worker process.
- Even below the crash threshold, holds a PostgreSQL transaction open for the duration of thousands of sequential queries, starving the connection pool and blocking all other requests that need a DB connection.
- Because the crash or hang occurs inside a NestJS request handler, it takes down the entire API process (no worker isolation), causing a full service outage until the process is restarted.

### Likelihood Explanation

The attacker precondition is minimal: register an account, create any transaction (a standard user workflow), then POST a crafted approver payload. No privileged role, no leaked credentials, and no special network access are required. The attack is a single HTTP request and is fully reproducible.

### Recommendation

1. **Enforce a maximum nesting depth** inside `createApprover` by passing a `depth` counter and throwing when it exceeds a small constant (e.g., 5):

```typescript
const createApprover = async (dtoApprover: CreateTransactionApproverDto, depth = 0) => {
  if (depth > MAX_APPROVER_DEPTH) throw new Error('Approver tree exceeds maximum depth');
  // ...
  await createApprover({ ...nestedDtoApprover, listId: approver.id }, depth + 1);
};
```

2. **Add `@ArrayMaxSize` to the DTO** to cap the fan-out at each level:

```typescript
@ArrayMaxSize(20)
approvers?: CreateTransactionApproverDto[];
```

3. **Add a `@Max` depth validator** or a custom class-validator decorator that rejects payloads whose nesting exceeds the allowed depth before the service layer is reached.

### Proof of Concept

**Attacker steps:**

1. Authenticate as a normal user and obtain a JWT.
2. Create any transaction to obtain a `transactionId`.
3. Craft a POST body with 1 000 levels of nesting:

```python
import json, requests

def make_nested(depth):
    if depth == 0:
        return {"userId": 1}
    return {"threshold": 1, "approvers": [make_nested(depth - 1)]}

payload = {"approversArray": [make_nested(1000)]}

requests.post(
    "https://<host>/transactions/1/approvers",
    json=payload,
    headers={"Authorization": "Bearer <jwt>"}
)
```

4. **Expected outcome:** The API process crashes with a `RangeError: Maximum call stack size exceeded` (or hangs for tens of seconds exhausting the DB connection pool), returning a 500 or timing out, and the service becomes unavailable until the process restarts. [6](#0-5) [7](#0-6)
