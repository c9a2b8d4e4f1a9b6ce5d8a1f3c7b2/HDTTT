### Title
Any Authenticated User Can Post Comments on Transactions They Have No Access To

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` enforces only authentication (JWT validity and user verification status) but performs no transaction-level authorization check. Any verified user in the organization can write a comment to any transaction, regardless of whether they are the creator, signer, observer, or approver of that transaction. The codebase itself acknowledges this gap with a `//TODO` comment at the exact location.

### Finding Description

The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level, which only confirm the caller holds a valid, non-blacklisted JWT and has a verified account status. [1](#0-0) 

The `createComment` handler passes the caller's user object and the attacker-supplied `transactionId` directly to `CommentsService.createComment()` with no further check: [2](#0-1) 

`CommentsService` never calls `verifyAccess()`, `getTransactionWithVerifiedAccess()`, or any equivalent. Compare this to every other sensitive transaction sub-resource in the codebase: `ApproversService.getVerifiedApproversByTransactionId()` and `ObserversService.getTransactionObserversByTransactionId()` both enforce the same multi-role membership check before returning data: [3](#0-2) 

`TransactionsService.verifyAccess()` defines the canonical access model — a user must be creator, signer, observer, approver, or hold a signing key for the transaction: [4](#0-3) 

The comments flow skips this check entirely. The developer acknowledged the gap inline: [5](#0-4) 

### Impact Explanation

A malicious authenticated user can:
1. **Write unauthorized comments** on any transaction in the organization — including sensitive multi-sig transactions they have no legitimate role in — causing confusion, social engineering of legitimate signers, or workflow disruption.
2. **Confirm transaction existence** by observing whether the comment is saved successfully (non-404 response), leaking organizational transaction IDs.
3. **Pollute the audit trail** of transactions with arbitrary content, undermining the integrity of the comment history that legitimate participants rely on.

This is an unauthorized state change (persistent write) to a resource the attacker has no business accessing.

### Likelihood Explanation

The attacker precondition is only a valid organization account — no admin role, no leaked secrets, no privileged access required. Any employee or contractor with a standard account can exploit this immediately via a single authenticated HTTP POST. The endpoint is reachable from the standard API surface documented in `comments-requests.http`. [6](#0-5) 

### Recommendation

In `CommentsService.createComment()`, verify the requesting user has access to the target transaction before persisting the comment. Inject `TransactionsService` (or reuse the `EntityManager` already available) and call `verifyAccess()` against the resolved transaction. Reject with `UnauthorizedException` if the check returns `false`, mirroring the pattern used in `ApproversService` and `ObserversService`.

### Proof of Concept

**Preconditions:** Two accounts exist — User A (creator of transaction ID 42, a sensitive pending multi-sig) and User B (no relationship to transaction 42).

1. User B authenticates: `POST /auth/login` → receives `accessToken`.
2. User B posts a comment: `POST /transactions/42/comments` with `Authorization: Bearer <accessToken>` and body `{"message": "Please sign urgently - CEO"}`.
3. The server responds `201 Created` and persists the comment linked to transaction 42.
4. User A and all legitimate signers of transaction 42 now see the attacker-injected comment in the transaction's comment thread, with no indication it came from an unauthorized party.

No admin privileges, no special tooling, and no knowledge of transaction internals beyond the numeric ID are required.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
