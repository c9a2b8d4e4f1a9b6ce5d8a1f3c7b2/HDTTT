Audit Report

## Title
Server-Side Request Forgery (SSRF) via User-Controlled `mirrorNetwork` in Back-End `MirrorNodeClient`

## Summary
The back-end `MirrorNodeClient` constructs outbound HTTP request URLs from a user-supplied `mirrorNetwork` string with no server-side allowlist or format validation. An authenticated user can supply an arbitrary hostname as `mirrorNetwork` in the transaction creation request, causing the NestJS server to issue HTTP requests to attacker-controlled or internal hosts.

## Finding Description

**Root cause — unvalidated URL construction:**

`MirrorNodeREST.fromBaseURL` in `back-end/libs/common/src/utils/mirrorNode/index.ts` has a `default` branch that blindly prepends `https://` to the caller-supplied string:

```ts
default:
    return `https://${mirrorNetwork}`;
``` [1](#0-0) 

**Sink — server-side HTTP request:**

`MirrorNodeClient.getMirrorNodeRESTURL` passes the result directly as the base URL for outbound HTTP requests:

```ts
private getMirrorNodeRESTURL(mirrorNetwork: string): string {
    return `${MirrorNodeREST.fromBaseURL(mirrorNetwork)}${this.endpointPrefix}`;
}
``` [2](#0-1) 

The constructed URL is then used in a live HTTP call: [3](#0-2) 

**Source — no allowlist in `CreateTransactionDto`:**

The `mirrorNetwork` field in the DTO only requires a non-empty string. There is no allowlist, regex, or URL validation:

```ts
@IsNotEmpty()
@IsString()
mirrorNetwork: string;
``` [4](#0-3) 

The `validateAndPrepareTransaction` method validates signature, expiry, size, and node IDs — but never validates `mirrorNetwork` before storing it: [5](#0-4) 

**Front-end source — arbitrary custom URL accepted:**

`NetworkSettings.vue` accepts any user-typed mirror node URL. The `formatMirrorNodeBaseURL` function strips the `https://` scheme prefix before storing, which is then re-prepended server-side by `MirrorNodeREST.fromBaseURL`: [6](#0-5) 

The front-end submits the stored value as `mirrorNetwork` in the transaction POST body: [7](#0-6) 

**Server-side call path confirmed:**

`NodeCacheService.fetchAndSaveNodeInfo` passes the stored `mirrorNetwork` directly to `mirrorNodeClient.fetchNodeInfo`, completing the SSRF call chain: [8](#0-7) 

## Impact Explanation

- **SSRF to internal network:** The server can be made to probe internal hosts unreachable from the public internet (e.g., cloud metadata endpoints at `169.254.169.254`, internal databases, Redis, RabbitMQ) by supplying their IP or hostname as `mirrorNetwork`.
- **Data exfiltration via request path:** The server appends account IDs and node IDs to the attacker-controlled URL (e.g., `/api/v1/accounts/{accountId}`), leaking these identifiers to the attacker's server in the HTTP request path.
- **Firewall bypass:** Requests originate from the trusted back-end server IP, bypassing network-level controls that block direct attacker access to internal services.
- **Retry amplification:** The `fetchWithRetry` logic retries up to 3 times with exponential backoff, meaning each trigger causes up to 3 outbound requests to the attacker-controlled host. [9](#0-8) 

## Likelihood Explanation

Any authenticated organization user can trigger this without any privileged role. The attack requires only the ability to submit a transaction — a core, normal product workflow. The front-end UI natively supports custom mirror node URLs. An attacker can also bypass the front-end entirely and POST directly to the `/transactions` API endpoint with an arbitrary `mirrorNetwork` value (e.g., `169.254.169.254`), since the back-end performs no allowlist check.

## Recommendation

1. **Allowlist on the server side:** In `CreateTransactionDto` or in `validateAndPrepareTransaction`, validate `mirrorNetwork` against the set of known network identifiers (`mainnet`, `testnet`, `previewnet`, `localnode`) and a strict URL format for custom networks (e.g., enforce HTTPS, reject private IP ranges and link-local addresses).
2. **Harden `MirrorNodeREST.fromBaseURL`:** In the `default` branch, parse the input as a URL and reject any hostname that resolves to a private, loopback, or link-local address before constructing the request URL.
3. **Egress filtering:** At the network level, restrict the back-end server's outbound HTTP connections to known mirror node hostnames or IP ranges.

## Proof of Concept

1. Authenticate as any verified organization user.
2. POST directly to the `/transactions` API endpoint:
   ```http
   POST /transactions
   Authorization: Bearer <valid_token>
   Content-Type: application/json

   {
     "name": "ssrf-test",
     "description": "ssrf",
     "transactionBytes": "<valid_hex>",
     "signature": "<valid_sig>",
     "creatorKeyId": <valid_key_id>,
     "mirrorNetwork": "169.254.169.254"
   }
   ```
3. The back-end stores `mirrorNetwork: "169.254.169.254"` in the database.
4. When `NodeCacheService` or `AccountCacheService` processes the transaction, it calls `MirrorNodeREST.fromBaseURL("169.254.169.254")`, which returns `"https://169.254.169.254"`.
5. The server issues `GET https://169.254.169.254/api/v1/accounts/{accountId}` — an outbound request from the back-end server to the cloud metadata endpoint.

      const shouldRetry = this.isRetryableError(error);

        this.logger.error(

      );

      return this.fetchWithRetry<T>(url, etag, attempt + 1);
    }
