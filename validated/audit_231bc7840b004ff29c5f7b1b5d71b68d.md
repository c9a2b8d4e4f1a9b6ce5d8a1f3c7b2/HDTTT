### Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Duplicate `TransactionSigner` Rows Due to Missing Unique Constraint

### Summary
The `SignersService.uploadSignatureMaps` function uses a read-then-write pattern (TOCTOU) to determine whether a signer record already exists before inserting a new one. Because the `transaction_signer` table has no unique constraint on `(transactionId, userKeyId)`, two concurrent signature-upload requests from the same user for the same transaction and key can both pass the "already signed" check and both successfully insert duplicate `TransactionSigner` rows. This is the direct analog of the external report's `tokenIdsToPoolIds` overwrite: a mapping-like structure (signer registry) can be written multiple times for the same logical key, corrupting the state that downstream status-computation logic depends on.

### Finding Description

**Root cause — missing unique constraint:**

The `TransactionSigner` entity declares only a non-unique composite index:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← NOT { unique: true }
export class TransactionSigner { ... }
``` [1](#0-0) 

The corresponding migration confirms this is a plain index, not a unique index:

```sql
CREATE INDEX "IDX_f8cbac3e0cae3730a762e4905e"
  ON "transaction_signer" ("transactionId", "userKeyId")
``` [2](#0-1) 

No `UNIQUE` constraint exists at the database level to prevent duplicate `(transactionId, userKeyId)` pairs.

**Root cause — TOCTOU in `uploadSignatureMaps`:**

The service performs three non-atomic steps:

**Step 1** — Read existing signers into an in-memory `Set`:
```typescript
const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
  where: { transactionId: In(transactionIds) },
  select: ['transactionId', 'userKeyId'],
});
// ...
signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
``` [3](#0-2) 

**Step 2** — Check the in-memory set to decide whether to insert:
```typescript
if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);
}
``` [4](#0-3) 

**Step 3** — Bulk-insert with no `ON CONFLICT` guard:
```typescript
const result = await manager
  .createQueryBuilder()
  .insert()
  .into(TransactionSigner)
  .values(signersToInsert)
  .returning('*')
  .execute();
``` [5](#0-4) 

If two concurrent requests both execute Step 1 before either completes Step 3, both see an empty set for the signer, both include the key in `signersToInsert`, and both successfully insert — creating two `TransactionSigner` rows with identical `(transactionId, userKeyId)`.

**Downstream corruption:**

`processTransactionStatus` is called immediately after insertion and reads the `TransactionSigner` table to determine whether a transaction has accumulated enough signatures to advance to `WAITING_FOR_EXECUTION`: [6](#0-5) 

If that function counts signer rows without deduplicating on `userKeyId`, duplicate rows inflate the apparent signer count, potentially advancing a transaction to `WAITING_FOR_EXECUTION` before the required threshold of *distinct* keys is actually met. Conversely, `getSignaturesByTransactionId` returns all rows including duplicates, corrupting the signer list exposed to callers. [7](#0-6) 

### Impact Explanation

- **Incorrect transaction status advancement**: A transaction requiring N distinct signers could be marked `WAITING_FOR_EXECUTION` with fewer than N real signers if duplicate rows inflate the count.
- **Corrupted signer audit trail**: `getSignaturesByTransactionId` returns duplicate entries, breaking any UI or downstream logic that relies on the signer list being a set.
- **Potential premature execution**: If the chain service picks up a transaction marked `WAITING_FOR_EXECUTION` that does not actually have sufficient signatures, execution will fail on-chain, permanently marking the transaction `FAILED` and preventing legitimate execution — effectively freezing the transaction.

### Likelihood Explanation

Any authenticated user can trigger this by submitting two concurrent `POST /transactions/:id/signers` requests with the same signature map. No privileged access is required. HTTP clients, browser double-clicks, or retry logic can all produce concurrent requests trivially. The race window is the entire duration of the DB read-to-write gap, which under normal load is tens to hundreds of milliseconds.

### Recommendation

1. **Add a unique constraint at the database level** (the only reliable fix):
   ```sql
   CREATE UNIQUE INDEX "UQ_transaction_signer_tx_key"
     ON "transaction_signer" ("transactionId", "userKeyId");
   ```
   And update the entity:
   ```typescript
   @Index(['transactionId', 'userKeyId'], { unique: true })
   ```

2. **Use `INSERT ... ON CONFLICT DO NOTHING`** in `bulkInsertSigners` so concurrent inserts are idempotent:
   ```typescript
   .orIgnore()   // TypeORM QueryBuilder method
   ```

3. **Deduplicate in `processTransactionStatus`** on `userKeyId` when counting signers, as a defense-in-depth measure.

### Proof of Concept

1. Register a user with key `K` and upload a transaction `T` requiring 1 signature.
2. Sign `T` with `K` locally to produce a valid `signatureMap`.
3. Fire two concurrent HTTP requests simultaneously:
   ```
   POST /transactions/T/signers  { signatureMap: <signed> }
   POST /transactions/T/signers  { signatureMap: <signed> }
   ```
4. Query the database:
   ```sql
   SELECT * FROM transaction_signer
   WHERE "transactionId" = T AND "userKeyId" = K_id;
   ```
   **Expected (correct)**: 1 row.
   **Actual (buggy)**: 2 rows with identical `(transactionId, userKeyId)`.
5. Observe that `processTransactionStatus` may now count 2 signers for a threshold that requires only 1, or that `getSignaturesByTransactionId` returns duplicate entries.
