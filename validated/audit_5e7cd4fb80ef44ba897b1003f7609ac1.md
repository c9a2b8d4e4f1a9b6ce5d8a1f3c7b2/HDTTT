Audit Report

## Title
Unbounded Loop with Per-Iteration CPU Work in `getTransactionsToSign` Enables Authenticated Resource Exhaustion

## Summary
`TransactionsService.getTransactionsToSign` fetches **all** non-terminal transactions system-wide without any row limit, then iterates over every one performing CPU-intensive deserialization and signature key computation per transaction. Pagination is applied only after the full loop completes. An authenticated user can trigger this endpoint to exhaust server CPU and memory, degrading service for all users.

## Finding Description

**Unbounded system-wide fetch — confirmed:**

In `getTransactionsToSign`, the repository `find()` call has no `take` limit:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // all non-terminal statuses, system-wide
  relations: ['groupItem'],
  order,
});                      // ← no .take / .limit
``` [1](#0-0) 

The `whereForUser` clause excludes only terminal statuses and is **not** scoped to the requesting user — it returns every active transaction in the system: [2](#0-1) 

**Per-iteration CPU-intensive work — confirmed:**

The full result set is iterated sequentially:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [3](#0-2) 

Each call to `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive deserialization of raw Hedera transaction bytes
2. `transactionSignatureService.computeSignatureKey(transaction)` — calls `SDKTransaction.fromBytes` again internally, then makes `accountCacheService.getAccountInfoForTransaction` calls (which hit the mirror node / DB on cache misses) for fee payer, signing accounts, and receiver accounts [4](#0-3) [5](#0-4) 

**Correction to the original claim:** The per-iteration `entityManager.find(UserKey, ...)` DB query does **not** occur in this call path. Because `attachKeys` is called once at line 285 before the loop and caches `user.keys` on the user object, subsequent `attachKeys` calls inside `userKeysRequiredToSign` are no-ops. `keysRequiredToSign` then takes the in-memory filter branch (since `user.keys` is passed as `userKeys`), avoiding the DB round-trip per iteration. [6](#0-5) [7](#0-6) 

**Pagination after full scan — confirmed:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ← sliced after full scan
  page,
  size,
};
``` [8](#0-7) 

**Endpoint still active despite deprecation:**

The controller registers the endpoint and explicitly comments it is no longer used by the front-end, but it remains fully accessible: [9](#0-8) 

The front-end has migrated to the optimized `GET /transaction-nodes?collection=READY_TO_SIGN` endpoint which uses a single SQL query scoped to the user and does not perform per-transaction deserialization. [10](#0-9) 

## Impact Explanation

- **CPU exhaustion**: Each loop iteration calls `SDKTransaction.fromBytes` twice (once in `keysRequiredToSign`, once inside `computeSignatureKey`) and performs cryptographic key list computation. With thousands of active transactions, a single API call saturates a CPU core for the full duration.
- **Memory pressure**: The entire active transaction set (including `transactionBytes` blobs and `groupItem` relations) is loaded into memory before any filtering or pagination occurs.
- **External service pressure**: `computeSignatureKey` calls `accountCacheService.getAccountInfoForTransaction` for each unique account in each transaction. Cache misses result in mirror node or DB queries, multiplied across all transactions in the loop.
- **Cascading effect**: Because the fetch is system-wide, every caller pays the full cost regardless of how many transactions they personally own. Multiple concurrent callers multiply the effect.

**Impact: Service degradation / unavailability for all users of the API service.**

## Likelihood Explanation

- **Attacker precondition**: A valid authenticated account with verified status (`VerifiedUserGuard` is enforced).
- **Attack path**: Register and verify → create N transactions via `POST /transactions` (no enforced creation limit) → repeatedly call `GET /transactions/sign`.
- **No special privilege required**: The endpoint is accessible to any verified authenticated user.
- **Amplification**: Even transactions created by other users contribute to the attacker's loop cost, since the fetch is system-wide.
- **No rate limiting**: No rate-limiting middleware is applied to this endpoint. [11](#0-10) 

## Recommendation

1. **Remove or disable the endpoint entirely** — it is already marked `/* NO LONGER USED BY FRONT-END */`. The replacement `GET /transaction-nodes?collection=READY_TO_SIGN` is already in production and handles this use case correctly via a single scoped SQL query.
2. **If the endpoint must be retained**, apply a `take: limit` on the initial `repo.find()` call and scope the query to transactions where the user is a signer (join against `transaction_signer`), mirroring the approach in `TransactionNodesService.getTransactionNodes`.
3. **Add rate limiting** to all authenticated endpoints that perform per-item async work.

## Proof of Concept

```
# 1. Register and verify an account
POST /auth/register  { email, password }
# (admin verifies the account)

# 2. Create N transactions (no server-side creation limit enforced)
for i in 1..N:
  POST /transactions  { transactionBytes: <valid_hedera_tx_bytes>, ... }

# 3. Repeatedly call the unbounded endpoint
while true:
  GET /transactions/sign?page=1&size=10
  # Server fetches ALL N transactions, deserializes each one,
  # calls computeSignatureKey on each, then slices to 10 items.
  # CPU and memory scale linearly with N regardless of page size.
```

With N = 5,000 transactions and concurrent callers, each request forces the server to deserialize 5,000 Hedera transaction byte blobs and compute signature keys for all of them before returning 10 results. The `size` query parameter provides no protection because pagination is applied after the full loop. [12](#0-11)

      transaction: Transaction;

      where: whereForUser,

    };

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
