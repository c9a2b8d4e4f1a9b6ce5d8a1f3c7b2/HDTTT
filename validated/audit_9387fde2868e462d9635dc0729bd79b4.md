Audit Report

## Title
Non-Atomic Duplicate Check in `createTransactions` Allows Concurrent Duplicate Transaction Records

## Summary
The `createTransactions` function in `back-end/apps/api/src/transactions/transactions.service.ts` performs a duplicate-existence check outside the database write transaction, and no `UNIQUE` constraint exists on the `transactionId` column in the `transaction` table. Two concurrent authenticated requests carrying identical transaction bytes will both pass the check and both insert, producing two active records for the same Hedera transaction ID.

## Finding Description

**Root cause — non-atomic check-then-act with no DB uniqueness enforcement**

In `createTransactions`, the duplicate check at lines 415–433 is a plain `repo.find()` call executed outside any database transaction: [1](#0-0) 

Immediately after, a separate `entityManager.transaction()` block performs the actual insert: [2](#0-1) 

The `find` and the `save` are in **two separate database transactions**. There is no serialization or locking between them.

The `Transaction` entity declares `transactionId` as a plain `@Column()` with no `@Unique()` decorator: [3](#0-2) 

The initial schema migration creates the column as `"transactionId" character varying NOT NULL` with only a primary-key constraint — no `UNIQUE` constraint: [4](#0-3) 

No subsequent migration adds a unique constraint on `transaction.transactionId`. The `UNIQUE` indexes added in later migrations (`1766811596315`, `1768289349311`) apply only to `transaction_cached_account` and `transaction_cached_node` composite keys, not to the `transaction` table's `transactionId` column. [5](#0-4) 

## Impact Explanation
- Two rows with identical `transactionId` and status `WAITING_FOR_SIGNATURES` are committed to the database.
- Signers receive duplicate notification events and may sign both copies, wasting key operations.
- Observers and approvers see two separate transaction objects for the same underlying Hedera transaction.
- The chain service scheduler will pick up both records and attempt to submit both to the Hedera network. The second submission fails with `DUPLICATE_TRANSACTION`, but the duplicate DB row is never cleaned up and remains active.
- No direct loss of Hedera funds, but the operational integrity of the multi-signature workflow is broken.

## Likelihood Explanation
Any authenticated user can trigger this with two concurrent HTTP requests — no privileged access is required. The window between the `find` and the `save` is wide (it spans network I/O, DTO validation, and `getClientFromNetwork` calls), making it reliably exploitable both accidentally (rapid double-submit from the UI) and deliberately (scripted concurrent POST). [6](#0-5) 

## Recommendation
Apply **one or both** of the following fixes:

1. **Database-level unique constraint**: Add a partial unique index on `(transactionId)` where `status NOT IN ('CANCELED', 'REJECTED', 'ARCHIVED')`, or a full unique index if the business logic permits. This is the only reliable defense under concurrency.

2. **Move the check inside the transaction with a lock**: Perform the `find` inside the same `entityManager.transaction()` block using a `SELECT ... FOR UPDATE` or `SERIALIZABLE` isolation level, so the check and the insert are atomic.

Without a DB-level constraint, any application-level guard is inherently racy.

## Proof of Concept

```bash
# Fire two concurrent POST /transactions requests with identical transaction bytes
curl -s -X POST https://<host>/transactions \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"transactionBytes":"<hex>", ...}' &

curl -s -X POST https://<host>/transactions \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"transactionBytes":"<hex>", ...}' &

wait
# Both requests return HTTP 201 with different `id` values but identical `transactionId`
# Confirmed by: SELECT id, "transactionId", status FROM transaction WHERE "transactionId" = '<id>';
# Returns two rows, both with status 'WAITING FOR SIGNATURES'
```
