All claims are verified against the actual source code. The vulnerability is real and in scope.

Audit Report

## Title
Any Authenticated User Can Enumerate All Transaction Groups via Missing Authorization on `getTransactionGroups()`

## Summary
The `GET /transaction-groups` endpoint, explicitly marked `/* TESTING ONLY */` in source, is live in production and returns every `TransactionGroup` row in the database to any authenticated user. No ownership or membership filter is applied at either the controller or service layer.

## Finding Description

The controller at `back-end/apps/api/src/transactions/groups/transaction-groups.controller.ts` lines 52–57 defines:

```typescript
/* TESTING ONLY: Get all transactions groups */
@Get()
@Serialize(TransactionGroupDto)
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroups();
}
``` [1](#0-0) 

No `@GetUser() user: User` parameter is present, so the authenticated user's identity is never forwarded to the service layer.

The service implementation at `back-end/apps/api/src/transactions/groups/transaction-groups.service.ts` lines 36–38 performs an unfiltered database scan:

```typescript
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.dataSource.manager.find(TransactionGroup);
}
``` [2](#0-1) 

`find(TransactionGroup)` with no `where` clause returns every row in the `transaction_group` table.

**Contrast with the correctly-guarded singular endpoint** at lines 67–75, which passes `user` to the service so membership can be enforced:

```typescript
getTransactionGroup(
  @GetUser() user: User,
  @Param('id', ParseIntPipe) groupId: number,
  ...
): Promise<TransactionGroup> {
  return this.transactionGroupsService.getTransactionGroup(user, groupId, full ?? true);
}
``` [3](#0-2) 

The class-level guard stack (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only authenticates the caller — it performs no authorization or ownership check. [4](#0-3) 

## Impact Explanation
Any authenticated user can retrieve all `TransactionGroup` records in the system, including groups they have no membership in. The exposed data includes group IDs, associated transaction IDs, group metadata, and execution sequencing — sufficient for targeted follow-on attacks (e.g., transaction cancellation, signature injection, observer manipulation via other endpoints). This constitutes cross-tenant data exposure, not merely user/tenant enumeration, so it is not excluded by the SECURITY.md out-of-scope clause covering "enumeration or confirmation of the existence of users or tenants."

## Likelihood Explanation
The precondition is simply holding a valid JWT for any registered, verified account — a normal user state. The route is a standard REST `GET` with no additional friction. The `/* TESTING ONLY */` comment confirms the endpoint was never intended to be production-accessible, making its presence an oversight. Exploitation requires no elevated privileges, no special tooling, and no race conditions.

## Recommendation
**Short-term:** Remove or disable the `getTransactionGroups()` endpoint entirely, consistent with its "TESTING ONLY" designation.

**If the endpoint must be retained** (e.g., for admin use), apply one of the following:
1. Add `@GetUser() user: User` to the controller method and pass it to a new service method that filters by group membership (mirroring the pattern used in `getTransactionGroup()`).
2. Gate the endpoint behind an admin-only guard (e.g., `AdminGuard`) so only privileged roles can call it.

## Proof of Concept

```bash
# 1. Obtain a valid JWT for any registered, verified user account
TOKEN="<valid_jwt>"

# 2. Call the unprotected listing endpoint
curl -H "Authorization: Bearer $TOKEN" \
     https://<host>/transaction-groups

# Expected (vulnerable) response: all TransactionGroup rows in the database,
# including groups the caller has no membership in.
```

The response will contain every `TransactionGroup` record, confirmed by the bare `find(TransactionGroup)` call in the service with no `where` clause. [2](#0-1)
