### Title
Signatures Can Be Submitted After `cutoffAt` Deadline Has Passed

### Summary
The `Transaction` entity defines a `cutoffAt` field — explicitly documented as "The timestamp at which the transaction can no longer be signed by the signers." However, the signing validation path in `SignersService.validateTransactionStatus()` never checks this field. As a result, any authorized signer can submit signatures via `POST /transactions/signers` after the `cutoffAt` deadline has passed, violating the transaction creator's intent to close the signing window.

### Finding Description
The `cutoffAt` column is stored on the `Transaction` entity and accepted as an optional field in `CreateTransactionDto`: [1](#0-0) [2](#0-1) 

The database documentation confirms its semantic meaning: [3](#0-2) 

When a signer calls `uploadSignatureMaps()`, the only validation performed before accepting signatures is inside `validateTransactionStatus()`: [4](#0-3) 

This method checks two things:
1. Whether the transaction status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.
2. Whether the Hedera SDK transaction has expired (via `isExpired()`, which computes `validStart + transactionValidDuration`).

There is **no check against `transaction.cutoffAt`**. The `cutoffAt` value is loaded from the database in `loadTransactionData()`: [5](#0-4) 

But it is never read or compared to `Date.now()` anywhere in the signing flow. The same gap exists in `transactions.service.ts`'s `importSignatures` path, which delegates to the same `validateTransactionStatus` logic.

### Impact Explanation
A transaction creator sets `cutoffAt` to enforce a signing deadline — after that point, no new signatures should be accepted. Because the check is absent, any signer can submit valid signatures after the deadline. If the late signature is the threshold-completing one, it will trigger `processTransactionStatus` to advance the transaction to `WAITING_FOR_EXECUTION`, causing the transaction to be submitted to the Hedera network after the creator intended to close the window. This is an integrity violation: the creator's access-control intent (time-bounded signing) is silently bypassed.

### Likelihood Explanation
The attacker precondition is simply being an authenticated user who is a designated signer on a transaction that has a `cutoffAt` set. No privilege escalation is required. The signer only needs to delay their signature submission past the deadline — a trivially reachable condition (network delay, intentional timing, etc.). The API endpoint is reachable by any authenticated organization member who is listed as a signer.

### Recommendation
Add a `cutoffAt` check inside `validateTransactionStatus()` in `signers.service.ts`:

```typescript
private validateTransactionStatus(transaction: Transaction): string | null {
  if (
    transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
    transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
  ) {
    return ErrorCodes.TNRS;
  }

  // Enforce cutoff deadline if set
  if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
    return ErrorCodes.TNRS; // or a dedicated ErrorCodes.TCO (Transaction Cutoff)
  }

  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
  if (isExpired(sdkTransaction)) {
    return ErrorCodes.TE;
  }

  return null;
}
```

Apply the same guard to the `importSignatures` path in `transactions.service.ts`.

### Proof of Concept

1. Organization admin creates a transaction with `cutoffAt` set to `T + 60s` (one minute from now).
2. A signer is notified but waits until `T + 90s` (after the cutoff).
3. The signer calls `POST /transactions/signers` with a valid `UploadSignatureMapDto` for the transaction.
4. `uploadSignatureMaps` → `validateAndProcessSignatures` → `validateTransactionStatus` is called.
5. The transaction status is still `WAITING_FOR_SIGNATURES` and `isExpired()` returns `false` (Hedera's `validStart + duration` window is typically 3 minutes).
6. **No cutoff check fires.** The signature is accepted, persisted via `bulkInsertSigners`, and if it completes the threshold, the transaction advances to `WAITING_FOR_EXECUTION` and is submitted to the Hedera network — past the creator's intended deadline. [6](#0-5) [7](#0-6)

```typescript
  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
```

```typescript
  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
```
