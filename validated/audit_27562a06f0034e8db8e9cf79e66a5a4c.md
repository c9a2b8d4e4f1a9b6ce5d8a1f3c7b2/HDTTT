### Title
Unbounded Array Input in `importSignatures` Endpoint Enables Authenticated Resource Exhaustion

### Summary
The `POST /transactions/signatures/import` endpoint accepts an array of `UploadSignatureMapDto` objects with no server-side size limit. The developer explicitly acknowledged this gap in a code comment. Any authenticated user can submit a single request containing thousands of items, each triggering CPU-intensive SDK deserialization and cryptographic signature validation, causing severe service degradation for all users.

### Finding Description
**Root cause:** No array-length constraint is enforced at the controller or service layer.

In `back-end/apps/api/src/transactions/transactions.controller.ts` lines 93–107, the endpoint accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no size guard:

```typescript
@Post('/signatures/import')
async importSignatures(
  @Body() body: UploadSignatureMapDto[] | UploadSignatureMapDto,
  @GetUser() user: User,
): Promise<SignatureImportResultDto[]> {
  const transformedSignatureMaps = await transformAndValidateDto(
    UploadSignatureMapDto,
    body
  );
  return this.transactionsService.importSignatures(transformedSignatureMaps, user);
}
```

`transformAndValidateDto` validates individual DTO field shapes but imposes no limit on array length.

In `back-end/apps/api/src/transactions/transactions.service.ts` line 575, the developer explicitly acknowledged the missing control:

```typescript
//Added a batch mechanism, probably should limit this on the api side of things
const BATCH_SIZE = 500;
```

For every element in the submitted array, the service executes (lines 542–553):
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — full SDK deserialization
2. `validateSignature(sdkTransaction, map)` — cryptographic signature verification
3. `sdkTransaction.addSignature(publicKey, map)` — signature attachment

After processing, a raw CASE SQL expression is built by string concatenation across all updates (lines 583–601), with no cap on the number of WHEN clauses beyond the internal 500-item batch loop — meaning an attacker submitting 50,000 items triggers 100 such batches sequentially.

**Exploit path:**
1. Attacker registers as a normal user (no admin role required).
2. Attacker sends `POST /transactions/signatures/import` with a JSON array of N items (e.g., N = 50,000), each containing a valid or invalid `id` and a crafted `signatureMap`.
3. The service iterates all N items: for each, it performs SDK deserialization and cryptographic validation.
4. CPU and memory are saturated; the API process becomes unresponsive for all concurrent users. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
Any authenticated user can render the API service severely degraded or temporarily unavailable for all other users with a single HTTP request. The operations triggered per array element (SDK deserialization + cryptographic verification) are computationally expensive. At 50,000 items, the server must perform 50,000 deserialization + verification cycles and 100 sequential bulk SQL updates, exhausting CPU, memory, and database connection time. This constitutes "severe degradation under realistic attacker input" as defined in RESEARCHER.md.

### Likelihood Explanation
The precondition is only a valid JWT token — any registered user qualifies. The developer comment at line 575 explicitly flags the missing limit as a known gap, confirming the control was never implemented. The attack requires a single crafted HTTP request with no special tooling beyond a standard HTTP client. No admin privilege, no leaked credentials, and no physical access are needed.

### Recommendation
Enforce a maximum array length at the controller level before delegating to the service. For example:

```typescript
if (Array.isArray(body) && body.length > 50) {
  throw new BadRequestException('Too many signature maps in a single request (max 50)');
}
```

Alternatively, apply a NestJS `ValidationPipe` with `@ArrayMaxSize(50)` on the DTO, or add a dedicated guard that rejects oversized payloads before any processing begins.

### Proof of Concept

**Preconditions:** Valid JWT for any registered user; knowledge of any transaction ID (even non-existent IDs are handled gracefully and still consume processing cycles for the lookup).

**Steps:**
1. Authenticate: `POST /auth/login` → obtain `accessToken`.
2. Construct a payload of 50,000 items:
   ```json
   [
     { "id": 1, "signatureMap": {} },
     { "id": 2, "signatureMap": {} },
     ... (repeat 49,998 more times)
   ]
   ```
3. Send:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <accessToken>
   Content-Type: application/json
   Body: <50,000-element array>
   ```
4. **Expected outcome:** The server iterates all 50,000 items performing SDK deserialization and signature validation for each, saturating CPU. Concurrent legitimate requests time out or receive 503 responses during processing.
