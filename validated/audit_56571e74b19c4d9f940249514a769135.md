1. `size < 0` guard (not `<= 0`) at line 18 of the decorator — confirmed.
2. `limit = size` (line 27), so `limit=0` flows downstream.
3. `take: limit` passed directly to TypeORM in `transactions.service.ts`, `signers.service.ts`, `user-keys.service.ts`, and `notification-receiver.service.ts` — all confirmed.

Audit Report

## Title
`PaginationParams` Decorator Accepts `size=0`, Forwarding `take:0` to TypeORM and Removing the SQL `LIMIT` Clause

## Summary
The `PaginationParams` decorator validates `size` with `size < 0` instead of `size <= 0`. This allows `size=0` to pass all guards, producing `limit=0`, which is forwarded directly to TypeORM's `take` option. TypeORM treats `take:0` as a falsy value and omits the `LIMIT` clause from the generated SQL, returning every matching row. The 100-item cap is completely bypassed.

## Finding Description

**Root cause — off-by-one in the boundary check:**

In `back-end/libs/common/src/decorators/pagination-params.decorator.ts` line 18:

```ts
if (isNaN(page) || page <= 0 || isNaN(size) || size < 0)
```

`page` is correctly rejected when `<= 0`. `size` is only rejected when `< 0`, so `size=0` passes. The upper-bound guard at line 22 (`size > 100`) also passes for `0`. The decorator then computes:

```ts
const limit = size;          // limit = 0
const offset = (page - 1) * limit;  // offset = 0
return { page, limit, size, offset };
``` [1](#0-0) 

**Propagation to TypeORM — `take:0` removes the SQL `LIMIT` clause:**

Every affected service passes `limit` directly to TypeORM's `take` option:

- `getTransactions` and `getHistoryTransactions` in `transactions.service.ts`: [2](#0-1) [3](#0-2) 

- `getTransactionsToApprove` in `transactions.service.ts`: [4](#0-3) 

- `getSignaturesByUser` in `signers.service.ts`: [5](#0-4) 

- `getUserKeys` in `user-keys.service.ts`: [6](#0-5) 

- `getReceivedNotifications` in `notification-receiver.service.ts` (via `getFindOptionsForNotifications`): [7](#0-6) 

TypeORM evaluates `take` as a falsy check; when `take` is `0`, it omits the `LIMIT` clause entirely, returning all rows matching the `WHERE` clause.

**Affected controllers (all decorated with `@PaginationParams()`):** [8](#0-7) [9](#0-8) [10](#0-9) 

**Note:** `getTransactionsToSign` is not exploitable via this path because it performs in-memory slicing (`result.slice(offset, offset + limit)`) — with `limit=0` this returns an empty array. [11](#0-10) 

## Impact Explanation
An authenticated user sends `GET /transactions?page=1&size=0` and receives every transaction row visible to them in a single response. The same applies to `/notifications`, `/user-keys`, `/transactions/history`, and `/transactions/approve`. This constitutes:
- **Data over-exposure**: the full dataset is returned instead of a bounded page, bypassing the explicit 100-item cap.
- **Server-side resource exhaustion**: an unbounded database query and full response serialization can exhaust memory and CPU, degrading service for all users. 

## Likelihood Explanation
- Requires only a valid session JWT — any registered user qualifies.
- The attack is a single HTTP GET with a trivially crafted query string (`?page=1&size=0`).
- No brute-force, timing, or cryptographic work is needed.
- The bug is in a shared decorator applied to every paginated endpoint, so the attack surface is wide. [12](#0-11) 

## Recommendation
Change the boundary check in `pagination-params.decorator.ts` from `size < 0` to `size <= 0`:

```ts
// Before (vulnerable)
if (isNaN(page) || page <= 0 || isNaN(size) || size < 0)

// After (fixed)
if (isNaN(page) || page <= 0 || isNaN(size) || size <= 0)
``` [13](#0-12) 

This ensures `size=0` is rejected with a `BadRequestException`, consistent with how `page=0` is already handled. No changes to the service layer are required.

## Proof of Concept
```
GET /transactions?page=1&size=0
Authorization: Bearer <valid_jwt>
```

**Decorator execution:**
- `page = 1`, `size = 0`
- Guard: `isNaN(1) || 1 <= 0 || isNaN(0) || 0 < 0` → all `false` → no exception thrown
- Guard: `0 > 100` → `false` → no exception thrown
- Returns `{ page: 1, limit: 0, size: 0, offset: 0 }`

**Service execution (`getTransactions`):**
- `findOptions = { ..., skip: 0, take: 0 }`
- TypeORM generates: `SELECT ... FROM "transaction" ... -- no LIMIT clause`
- All rows matching the user's visibility filter are returned in a single response. [14](#0-13)

export const PaginationParams = createParamDecorator((data, ctx: ExecutionContext): Pagination => {
  const req: Request = ctx.switchToHttp().getRequest();
  const page = parseInt(req.query.page as string);
  const size = parseInt(req.query.size as string);

  if (isNaN(page) || page <= 0 || isNaN(size) || size < 0) {
    throw new BadRequestException(ErrorCodes.IPP);
  }

  if (size > 100) {
    throw new BadRequestException(ErrorCodes.IPP);
  }

  // calculate pagination parameters
  const limit = size;
  const offset = (page - 1) * limit;
  return { page, limit, size, offset };
