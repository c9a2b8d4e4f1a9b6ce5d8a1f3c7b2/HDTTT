### Title
Unbounded Batch Array in Signature Import/Upload Endpoints Enables Authenticated Resource-Exhaustion DoS

### Summary
Two API endpoints accept an attacker-controlled array of `UploadSignatureMapDto` objects with no upper-bound validation on array length. Each element triggers CPU-intensive SDK deserialization, cryptographic signature validation, and database I/O. A single authenticated HTTP request carrying a large array can saturate the server's CPU, memory, and database connection pool, denying service to all other users. The developers themselves flagged the missing limit in a code comment.

### Finding Description

**Root cause — no array size cap on two batch endpoints:**

**Endpoint 1:** `POST /transactions/signatures/import` [1](#0-0) 

The controller accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` and passes the entire array directly to `TransactionsService.importSignatures`. [2](#0-1) 

Inside `importSignatures`, a `for...of` loop iterates over every element. Per element it calls `SDKTransaction.fromBytes()` (protobuf deserialization), `validateSignature()` (cryptographic verification), and `sdkTransaction.addSignature()`. The developer explicitly acknowledged the missing guard: [3](#0-2) 

**Endpoint 2:** `POST /transactions/:transactionId?/signers` [4](#0-3) 

This endpoint feeds the unbounded array into `SignersService.uploadSignatureMaps`, which calls `Promise.all()` over every element simultaneously: [5](#0-4) 

`Promise.all` fans out all N items concurrently, each performing `SDKTransaction.fromBytes()`, signature validation, and DB queries — multiplying the resource impact.

**No array size constraint exists on the DTO:** [6](#0-5) 

There is no `@ArrayMaxSize()`, `@MaxLength()`, or any equivalent decorator on the array parameter in either controller. The pagination decorator enforces a max of 100 for list queries, but no equivalent guard exists here: [7](#0-6) 

**Exploit flow:**
1. Attacker registers a normal user account (no privilege required).
2. Attacker sends a single `POST /transactions/signatures/import` with a JSON array of N (e.g., 10,000) `UploadSignatureMapDto` items, each referencing valid or invalid transaction IDs.
3. The server iterates all N items: each triggers `SDKTransaction.fromBytes()` (protobuf decode), `validateSignature()` (ECDSA/ED25519 verification), and a DB lookup.
4. For the `/signers` endpoint, `Promise.all` fans all N operations out concurrently, exhausting the DB connection pool instantly.
5. The server becomes unresponsive to all other users for the duration of processing.

### Impact Explanation
A single authenticated HTTP request can exhaust server CPU (cryptographic operations), heap memory (N deserialized SDK transaction objects held simultaneously), and PostgreSQL connection pool slots (N concurrent DB queries via `Promise.all`). This causes complete service unavailability for all organization users — no transactions can be signed, submitted, or monitored — for as long as the attacker repeats the request. The impact is **service unavailability / severe degradation** matching the RESEARCHER.md priority category of "Request/message patterns causing sustained crash or unbounded resource usage."

### Likelihood Explanation
Any registered, verified user can reach both endpoints — authentication is the only barrier. No special role, key, or privilege is required. The attack requires a single HTTP request with a crafted JSON body. The developer comment at line 575–576 (`//Added a batch mechanism, probably should limit this on the api side of things`) confirms the team is aware the limit is missing. Likelihood is **high**.

### Recommendation
1. Add an `@ArrayMaxSize(N)` class-validator decorator (e.g., `N = 100`) to the array parameter in both controllers, or validate array length at the top of each service method and throw `BadRequestException` if exceeded.
2. For `SignersService.uploadSignatureMaps`, replace the unbounded `Promise.all` with a chunked sequential or concurrency-limited executor (e.g., process in batches of 10).
3. Apply a global or per-route request body size limit in the NestJS/Express configuration to cap raw payload bytes as a secondary defense.

### Proof of Concept

**Target:** `POST /transactions/signatures/import` (or `/transactions/:id/signers`)

**Precondition:** Attacker holds a valid JWT for any registered, verified user account.

**Request:**
```http
POST /transactions/signatures/import HTTP/1.1
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1000.000000000": { "<pubkey_hex>": "<sig_hex>" } } } },
  { "id": 2, "signatureMap": { ... } },
  ... (repeat 10,000 times with arbitrary IDs)
]
```

**Expected outcome:** The server enters `importSignatures`, iterates all 10,000 items calling `SDKTransaction.fromBytes()` and `validateSignature()` per item (or fans them all out concurrently via `Promise.all` on the `/signers` endpoint), saturating CPU and DB connections. Concurrent legitimate requests time out or are rejected. The attacker can sustain the attack by repeating the request. [8](#0-7) [9](#0-8)
