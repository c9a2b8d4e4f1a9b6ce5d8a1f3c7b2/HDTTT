Audit Report

## Title
Server-Side Request Forgery (SSRF) via Unvalidated `mirrorNetwork` Field in Transaction Creation

## Summary
An authenticated, verified user can supply an arbitrary hostname or IP address as the `mirrorNetwork` field when creating a transaction. The backend immediately makes outbound gRPC and HTTP connections to the attacker-controlled host, and the value is persisted and reused on every subsequent transaction lifecycle event, causing repeated SSRF without further attacker interaction.

## Finding Description

**Root cause:** `CreateTransactionDto.mirrorNetwork` carries only `@IsNotEmpty()` and `@IsString()` decorators with no allowlist enforcement. [1](#0-0) 

**Exploit path 1 — gRPC connection at transaction creation time:**

`createTransactions()` immediately calls `getClientFromNetwork(dtos[0].mirrorNetwork)` before any other validation. [2](#0-1) 

Inside `getClientFromNetwork`, any value that is not `mainnet`, `testnet`, `previewnet`, or `local-node` falls through to the `default` branch, which constructs a gRPC client pointed at `<attacker-host>:443` and immediately executes an `AddressBookQuery` against it. [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` for an unknown value returns `[mirrorNetwork + ':443']` with no sanitisation. [4](#0-3) 

**Exploit path 2 — HTTP connection via `MirrorNodeClient`:**

`MirrorNodeREST.fromBaseURL` for an unknown value returns `https://<mirrorNetwork>` verbatim. [5](#0-4) 

`MirrorNodeClient.getMirrorNodeRESTURL` uses this to build the full URL, and `fetchAccountInfo` / `fetchNodeInfo` then issue real HTTP GET requests to it via Axios. [6](#0-5) [7](#0-6) 

**Persistence and repeated triggering:** The raw `mirrorNetwork` string is stored in the `Transaction` entity and reused by `ExecuteService._executeTransaction` on every execution attempt. [8](#0-7) [9](#0-8) 

## Impact Explanation
An authenticated user can force the backend to:
- Make TCP/gRPC connections to arbitrary hosts and ports (internal network port scanning via `AddressBookQuery` execution).
- Issue HTTP GET requests to cloud instance-metadata endpoints (e.g., `http://169.254.169.254/latest/meta-data/`) to retrieve IAM credentials or other secrets.
- Reach internal microservices, databases, or admin panels not exposed to the public internet.
- Exfiltrate data returned by internal services through error messages or timing side-channels.

Because the value is persisted and reused by the chain scheduler on every subsequent operation (scheduling, execution, account/node cache refresh), a single malicious submission causes repeated outbound connections without further attacker interaction. [10](#0-9) 

## Likelihood Explanation
The attacker must be a registered, verified organization user with at least one uploaded key pair — a normal user role reachable without any privileged access. Creating a valid signed transaction is straightforward using the Hedera SDK. No special tooling beyond a standard HTTP client is required to submit the crafted `mirrorNetwork` value. The barrier to exploitation is low for any insider or compromised account. [1](#0-0) 

## Recommendation
1. **Enforce an allowlist** on `mirrorNetwork` in `CreateTransactionDto` using a custom validator or `@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])`. If custom networks must be supported, validate the value against a server-side configuration list of approved hostnames rather than accepting arbitrary user input.
2. **Validate before connecting**: Move the `getClientFromNetwork` call to after all DTO validation is complete, and reject requests with non-allowlisted values before any network I/O occurs.
3. **Sanitize stored values**: Ensure that only allowlisted or admin-approved network identifiers are persisted to the `Transaction` entity.

## Proof of Concept

```http
POST /api/transactions HTTP/1.1
Authorization: Bearer <valid_user_jwt>
Content-Type: application/json

{
  "name": "SSRF Test",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<valid_hedera_tx_hex>",
  "creatorKeyId": 1,
  "signature": "<valid_signature_hex>",
  "mirrorNetwork": "169.254.169.254/latest/meta-data/iam/security-credentials/role"
}
```

Upon receipt, the backend calls `getClientFromNetwork("169.254.169.254/...")`, which constructs a gRPC client targeting `169.254.169.254/...:443` and executes an `AddressBookQuery` against it. Separately, `MirrorNodeREST.fromBaseURL` produces `https://169.254.169.254/latest/meta-data/iam/security-credentials/role`, and subsequent `fetchAccountInfo` calls issue HTTP GET requests to that URL via Axios. The `mirrorNetwork` value is then persisted and the SSRF is re-triggered on every scheduler cycle. [3](#0-2) [5](#0-4)

      throw error;
    }
  }
