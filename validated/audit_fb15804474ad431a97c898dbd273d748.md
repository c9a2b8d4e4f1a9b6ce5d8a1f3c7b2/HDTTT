Audit Report

## Title
Unbounded Array Input in `POST /transactions/signers` Enables Single-Request CPU/Memory Exhaustion

## Summary
The `uploadSignatureMap` endpoint accepts an array of `UploadSignatureMapDto` objects with no enforced maximum count. A single authenticated request containing a large array causes the server to concurrently process all items via `Promise.all`, performing unbounded in-memory deserialization and cryptographic validation. A developer comment in the codebase explicitly acknowledges the missing API-side limit on the related `importSignatures` path. Any verified user can trigger this without elevated privileges.

## Finding Description

**Entry point:** `POST /transactions/signers` — protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` (any verified user). [1](#0-0) 

The controller accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array size constraint: [2](#0-1) 

The `UploadSignatureMapDto` itself has no `@ArrayMaxSize()` or equivalent decorator: [3](#0-2) 

Inside `validateAndProcessSignatures`, all items are processed **concurrently** via `Promise.all`. Each item triggers `SDKTransaction.fromBytes` deserialization and cryptographic signature validation: [4](#0-3) 

**Correction to the original report's DB claim:** The DB queries are *not* spawned per-item. `loadTransactionData` issues two batched `In(transactionIds)` queries regardless of array size. The `Promise.all` concern is CPU and memory, not DB connections. [5](#0-4) 

A developer comment in the parallel `importSignatures` path explicitly acknowledges the missing limit: [6](#0-5) 

**Secondary finding — `CreateTransactionObserversDto`:** Only `@ArrayMinSize(1)` is enforced, no `@ArrayMaxSize()`. However, this endpoint is restricted to the transaction creator, not any verified user. [7](#0-6) 

**Partial mitigation present:** A 2MB body size limit is configured globally, which the original report omits. This bounds the maximum number of array items per request (roughly 2,000–10,000 items depending on signature map size), but does not eliminate the risk. [8](#0-7) 

## Impact Explanation

A single crafted POST request with an array of hundreds to low-thousands of entries (within the 2MB body limit) causes:
- **Memory exhaustion** — all `SDKTransaction.fromBytes` deserialized objects are held in memory simultaneously via `Promise.all`.
- **CPU saturation** — concurrent cryptographic signature validation across all items in a single request.
- **Event loop stall** — sustained CPU pressure degrades responsiveness for all concurrent users.

This is not a volumetric DDoS; a single authenticated HTTP request is sufficient to trigger the condition.

## Likelihood Explanation

- **Attacker precondition**: Any registered and verified user account — no admin or special role required.
- **Attack complexity**: Low. The attacker constructs one HTTP POST with a large JSON array body up to 2MB.
- The IP-based throttler counts requests, not payload item count, so a single large request is not blocked by rate limiting.
- The developer's own comment confirms awareness that the API-side limit is missing. [9](#0-8) 

## Recommendation

1. Add `@ArrayMaxSize(N)` to the array wrapper or enforce a count check at the controller level before passing to the service. A reasonable limit (e.g., 100–500) matches legitimate batch signing use cases.
2. Consider adding a `@Throttle()` decorator on the `uploadSignatureMap` endpoint scoped to the user identity (using the existing `UserThrottlerGuard`) to limit burst throughput per user.
3. Apply the same fix to `importSignatures` (`POST /transactions/signatures/import`), where the developer comment already acknowledges the gap.
4. Apply `@ArrayMaxSize()` to `CreateTransactionObserversDto.userIds` as a defense-in-depth measure.

## Proof of Concept

```http
POST /transactions/signers HTTP/1.1
Authorization: Bearer <valid_verified_user_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.100@1700000000.000000000": { "<pubkey_hex>": "<sig_hex>" } } } },
  { "id": 2, "signatureMap": { ... } },
  ... (repeated up to ~2MB body limit)
]
```

The server will invoke `Promise.all` over all N items simultaneously, each calling `SDKTransaction.fromBytes` and performing cryptographic validation, exhausting CPU and memory proportional to N within a single request lifecycle.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
