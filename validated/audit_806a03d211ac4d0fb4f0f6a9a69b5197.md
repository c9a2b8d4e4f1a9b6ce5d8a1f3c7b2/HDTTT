Audit Report

## Title
`isNode` Duplicate Check Bypassed for Threshold-Type Approver Nodes Allows Corrupted Approval Tree

## Summary
`ApproversService.isNode()` unconditionally returns `false` for any approver node that lacks a `userId` (i.e., threshold/list nodes), because the return expression short-circuits on `typeof approver.userId === 'number'`. A transaction creator can therefore insert duplicate threshold-node rows into `transaction_approver` with no error, and no database-level unique constraint exists to catch it. The resulting corrupted approval tree can make a transaction permanently un-approvable or cause incorrect approval-status computation.

## Finding Description

`isNode` is the sole application-level guard against duplicate approver nodes before insertion: [1](#0-0) 

The critical flaw is the final return expression at line 665:

```typescript
return count > 0 && typeof approver.userId === 'number';
```

For a threshold node, `approver.userId` is `undefined`, so `typeof approver.userId === 'number'` is always `false`. The entire expression evaluates to `false` regardless of whether `count > 0`, meaning `isNode` never signals a duplicate for threshold nodes.

The guard at the call site therefore never fires for threshold nodes: [2](#0-1) 

The `transaction_approver` table has no unique constraint on any combination of columns — only a serial primary key — so the database provides no safety net: [3](#0-2) 

By contrast, `transaction_observer` is created with a composite primary key `("userId", "transactionId")` that enforces uniqueness at the database level, but no equivalent protection exists for `transaction_approver`: [4](#0-3) 

The `TransactionApprover` entity itself carries only non-unique indexes on `transactionId` and `userId`, with no unique composite index: [5](#0-4) 

## Impact Explanation

When a duplicate threshold node is inserted, each copy receives its own set of child rows, because children are created with `listId: approver.id` pointing to the newly inserted parent's auto-generated `id`: [6](#0-5) 

The approval-status logic and the recursive CTE used in `getTransactionApproversById` traverse the full tree. With duplicate subtrees, the transaction may require approvals from both copies of the same logical group, making it impossible to reach the "approved" state even when all intended approvers have signed. Alternatively, if the duplicate subtrees have different thresholds, the effective approval requirement diverges silently from what the creator intended. [7](#0-6) 

## Likelihood Explanation

The `createTransactionApprovers` endpoint is restricted to the transaction creator via `getCreatorsTransaction`: [8](#0-7) 

Any authenticated user who creates a transaction can trigger this — no admin privilege is required. The creator can send two sequential `POST /transactions/:id/approvers` requests with an identical threshold-node payload. No special race condition or exploit chain is needed; the bug is reachable through normal API usage.

## Recommendation

1. **Fix the return expression in `isNode`** to simply `return count > 0;`. The `userId` type-check is incorrect and was never needed — the `find` query already encodes all the relevant fields (including `userId: null` for threshold nodes) to identify duplicates.

2. **Add a database-level unique constraint** on `transaction_approver` for root nodes (e.g., a partial unique index on `(transactionId, threshold)` where `listId IS NULL`) to provide a safety net independent of application logic.

## Proof of Concept

```
# Step 1: Creator authenticates and creates a transaction (transactionId = 1)

# Step 2: Creator sends first threshold-node payload
POST /transactions/1/approvers
Authorization: Bearer <creator_token>
Content-Type: application/json

{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{ "userId": 2 }]
  }]
}
# → 201 Created, inserts root node id=10 (threshold=1) and child id=11 (userId=2)

# Step 3: Creator sends identical payload again
POST /transactions/1/approvers
Authorization: Bearer <creator_token>
Content-Type: application/json

{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{ "userId": 2 }]
  }]
}
# → 201 Created (no error), inserts duplicate root node id=12 (threshold=1) and child id=13 (userId=2)

# Result: transaction_approver now contains two independent root threshold nodes
# for the same transaction. The approval tree is corrupted.
# User 2 must satisfy BOTH subtrees to approve, which is impossible since
# approveTransaction updates all records for that userId in one pass but
# the recursive CTE will still see two unsatisfied root nodes.
```
