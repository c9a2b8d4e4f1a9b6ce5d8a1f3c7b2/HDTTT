Audit Report

## Title
`cutoffAt` Deadline Not Enforced in Signature Upload Paths, Allowing Post-Deadline Signing

## Summary
The `cutoffAt` field on the `Transaction` entity is documented as the timestamp after which signers can no longer sign a transaction. Neither `uploadSignatureMaps` (via `validateTransactionStatus`) nor `importSignatures` checks `cutoffAt` against the current time. A signer can upload a valid signature after the creator's intended deadline, potentially advancing the transaction to `WAITING_FOR_EXECUTION` and triggering irreversible on-chain execution.

## Finding Description

**The `cutoffAt` field** is declared as a nullable column on the `Transaction` entity: [1](#0-0) 

The database documentation explicitly states its purpose: [2](#0-1) 

**Path 1 — `uploadSignatureMaps` → `validateTransactionStatus`**

The entire pre-acceptance validation in `SignersService` is: [3](#0-2) 

It checks only (1) that the status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`, and (2) that the Hedera SDK transaction has not expired via `isExpired()`. There is no comparison of `transaction.cutoffAt` against `Date.now()`. The `transaction` object loaded in `loadTransactionData` does contain `cutoffAt` (it is a column on the entity), but it is never read in this path. [4](#0-3) 

**Path 2 — `importSignatures`**

The same omission exists in `TransactionsService.importSignatures`: [5](#0-4) 

Only status and SDK-level expiry are checked; `cutoffAt` is never consulted.

**Consequence of a successful late signature**

After signatures are persisted, `processTransactionStatus` is called, which evaluates whether the signing threshold is met and can transition the transaction to `WAITING_FOR_EXECUTION`: [6](#0-5) 

Once in `WAITING_FOR_EXECUTION`, the chain scheduler's `prepareTransactions` picks it up and calls `collateAndExecute`, submitting it to the Hedera network: [7](#0-6) 

## Impact Explanation
A signer who deliberately delays their signature upload until after `cutoffAt` can unilaterally complete the required signing threshold and trigger on-chain execution of a transaction the creator believed was closed for signing. Depending on the transaction type (HBAR transfer, account update, file update, etc.) this results in irreversible financial or state changes that the creator did not intend to allow after the cutoff. The `cutoffAt` field is entirely decorative — it is stored, returned in API responses, and documented, but never enforced server-side.

## Likelihood Explanation
The attacker is any authenticated user who holds a required signer key on the transaction — a normal, non-privileged role. The exploit requires only a standard `POST` to the signers endpoint with a valid signature map submitted after `cutoffAt`. No special tooling or elevated access is needed. The window of opportunity exists for the full remaining Hedera SDK validity window (up to 180 seconds after `validStart`), which can be well after `cutoffAt`.

## Recommendation
In both `validateTransactionStatus` (`signers.service.ts`) and `importSignatures` (`transactions.service.ts`), add a guard immediately after the status check:

```typescript
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  // return/throw appropriate error, e.g. ErrorCodes.TC (Transaction Cutoff)
}
```

This should be applied consistently in both paths so that neither direct signature upload nor the import flow can bypass the deadline.

## Proof of Concept
1. Creator creates a transaction with `cutoffAt` set to `T+60s` and `validStart` set to `T+120s` (within the 180 s Hedera validity window).
2. At `T+61s` (after `cutoffAt` but before SDK expiry), a signer calls `POST /transactions/:id/signers` with a valid signature map.
3. `validateTransactionStatus` passes: status is `WAITING_FOR_SIGNATURES`, `isExpired()` returns `false` (validStart is still in the future).
4. The signature is persisted; `processTransactionStatus` detects the threshold is met and updates status to `WAITING_FOR_EXECUTION`.
5. The chain scheduler's `prepareTransactions` picks up the transaction and calls `collateAndExecute`, scheduling on-chain submission — despite the creator's `cutoffAt` deadline having passed.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }

  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },
            });
            // All the transactions for the group are now pulled. If there is an issue validating for even one
            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);
        }
      }
    }
  }
