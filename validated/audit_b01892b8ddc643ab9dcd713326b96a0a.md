After thoroughly examining the actual source code, here is my assessment:

**Vector 1 & 2** are confirmed. `getCreatorsTransaction` checks only existence and creator ownership — no status guard.

**Vector 3** is partially confirmed but materially mischaracterized. The claim that "any authenticated user" can delete any approver is **wrong**: the controller at line 108 calls `getCreatorsTransaction(transactionId, user)` before `removeTransactionApprover(id)`, enforcing creator-only access. Audit Report

## Title
Missing Transaction Status Validation in Approver/Observer Mutation Endpoints; Cross-Transaction Approver Deletion

## Summary
`createTransactionApprovers` and `createTransactionObservers` perform no status check before mutating terminal-state transactions. Additionally, the `removeTransactionApprover` controller verifies creator ownership of `transactionId` but never validates that the target approver `id` belongs to that transaction, enabling cross-transaction approver deletion.

## Finding Description

**Vector 1 — No status guard in `createTransactionApprovers`**

`ApproversService.createTransactionApprovers` delegates its only pre-condition to `getCreatorsTransaction`, which checks existence and creator ownership but never reads `transaction.status`: [1](#0-0) 

`getCreatorsTransaction` itself: [2](#0-1) 

No status guard exists between the ownership check and the `INSERT` into `transaction_approver`.

**Vector 2 — No status guard in `createTransactionObservers`**

`ObserversService.createTransactionObservers` checks existence and creator ownership, then immediately inserts observers with no status validation: [3](#0-2) 

**Vector 3 — Cross-transaction approver deletion (corrected characterization)**

The controller's `removeTransactionApprover` calls `getCreatorsTransaction(transactionId, user)` to verify the caller is creator of `transactionId`, then calls `removeTransactionApprover(id)` — but never verifies that the approver `id` belongs to `transactionId`: [4](#0-3) 

The service method itself accepts only `id` with no status or ownership check: [5](#0-4) 

**Note:** The original claim that "any authenticated user" can delete any approver is incorrect. The controller enforces creator-only access via `getCreatorsTransaction`. The actual bypass is cross-transaction: a creator of transaction A can delete approvers from transaction B.

**Contrast with functions that do check status**

`approveTransaction` guards on status before proceeding: [6](#0-5) 

`importSignatures` similarly guards: [7](#0-6) 

The terminal-status set used elsewhere: [8](#0-7) 

## Impact Explanation

**Vectors 1 & 2 (status bypass):**
- A transaction creator can add approvers or observers to a transaction already in `EXECUTED`, `CANCELED`, `EXPIRED`, `FAILED`, or `ARCHIVED` state.
- `emitTransactionStatusUpdate` fires after the insert, propagating the corrupted state to all connected clients.
- Approval-request notifications are sent to newly added approvers for transactions that can never be acted upon, corrupting the audit trail.
- `shouldApproveTransaction` does guard on terminal status and returns `false`, so the approval gate itself is not bypassed — but data and notification integrity are broken. [9](#0-8) 

**Vector 3 (cross-transaction approver deletion):**
- A creator of any transaction (even a dummy one they created themselves) can delete approvers from any other in-flight transaction by supplying their own `transactionId` and a target approver `id` from another transaction.
- Removing all approvers from a `WAITING_FOR_SIGNATURES` transaction eliminates the approval requirement, potentially allowing the transaction to advance without the intended governance controls.
- No status check means approvers can also be deleted from terminal-state transactions, further corrupting the audit trail.

## Likelihood Explanation

**Vectors 1 & 2:** Require the transaction creator to call the endpoint after the transaction reaches a terminal state. This can happen accidentally (race condition between execution and a UI action) or deliberately.

**Vector 3:** Requires the attacker to be the creator of at least one transaction (any transaction). Approver IDs are auto-incremented integers, making enumeration trivial. Any registered user who has ever created a transaction can exploit this.

## Recommendation

1. **Vectors 1 & 2:** Add a terminal-status guard inside `getCreatorsTransaction` (or at the call site in `createTransactionApprovers` and `createTransactionObservers`) that throws if `transaction.status` is in the terminal set (`EXECUTED`, `CANCELED`, `EXPIRED`, `FAILED`, `ARCHIVED`).

2. **Vector 3:** After calling `getCreatorsTransaction(transactionId, user)`, verify that the approver being deleted actually belongs to `transactionId` before calling `removeTransactionApprover(id)`. Also add a status guard to prevent deletion from terminal-state transactions.

## Proof of Concept

**Vectors 1 & 2:**
```
# 1. Creator creates a transaction (status: WAITING_FOR_SIGNATURES)
POST /transactions  →  { id: 42 }

# 2. Transaction reaches EXECUTED state (normal workflow)

# 3. Creator adds an approver to the already-executed transaction
POST /transactions/42/approvers
Authorization: Bearer <creator_jwt>
{ "approversArray": [{ "userId": 99 }] }

# Result: 201 Created — approver inserted, notification emitted, audit trail corrupted
```

**Vector 3:**
```
# Attacker is creator of transaction 1 (their own)
# Target: delete approver id=77 which belongs to transaction 99 (someone else's)

DELETE /transactions/1/approvers/77
Authorization: Bearer <attacker_jwt>

# Controller checks: is attacker creator of transaction 1? YES → passes
# Service removes approver 77 (which belongs to transaction 99) — no cross-check
# Result: 200 true — approver from transaction 99 deleted
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }

      where: { id: transactionId },

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)

    const observers: TransactionObserver[] = [];

    for (const userId of dto.userIds) {
      if (!transaction.observers.some(o => o.userId === userId)) {
        const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
        observers.push(observer);
      }
    }

      return [];
    }

      const result = await this.repo.save(observers);

      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);

      return result;
    } catch (error) {

  }
