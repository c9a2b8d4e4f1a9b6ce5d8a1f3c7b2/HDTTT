## Audit Report

## Title
`ApproversService.updateTransactionApprover`: Old parent threshold not corrected after approver re-parenting, permanently violating `threshold ≤ approvers.length`

## Summary
When `updateTransactionApprover` moves an approver to a new parent (`dto.listId` is a number), it validates and updates the new parent relationship but never reads or adjusts the **old** parent's `threshold`. If the old parent's `threshold` equalled its child count before the move, it is left with `threshold > approvers.length` after the move, making that subtree permanently unable to accumulate enough approvals. Any transaction gated on that subtree is irrecoverably locked.

## Finding Description

**Root cause — `approvers.service.ts`, lines 434–466**

The re-parenting branch only validates the new parent and writes the new `listId`: [1](#0-0) 

It never fetches the old parent (`approver.listId` before the update) or adjusts its threshold.

**Contrast with the `dto.listId === null` (detach) path**, which explicitly corrects the old parent: [2](#0-1) 

The detach path soft-deletes the old parent if it becomes childless, or lowers its threshold if `threshold > newChildCount`. The re-parent path has no equivalent.

**The invariant** `threshold ≤ approvers.length` is enforced only when a threshold is set directly via `dto.threshold`: [3](#0-2) 

There is no post-mutation check for the old parent after a re-parent operation.

**Exploit path:**
1. Transaction creator builds: `Parent (threshold=2)` → children `[A, B]`.
2. Creator calls `PATCH /transactions/{txId}/approvers/{A.id}` with `{ listId: <other_tree_id> }`.
3. `A` is moved. `Parent` now has `threshold=2` but only one child `[B]`.
4. `threshold > approvers.length` is silently violated; `Parent` can never accumulate 2 approvals.
5. The transaction is permanently unapprovable.

## Impact Explanation
A threshold node whose `threshold` exceeds its remaining child count can never satisfy its approval condition. Any transaction gated on that subtree is permanently locked in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` with no on-chain recourse. This constitutes **unrecoverable corruption of transaction state** — a category explicitly listed as high-value impact in `RESEARCHER.md`. [4](#0-3) 

## Likelihood Explanation
The only precondition is being the **transaction creator** — a normal, unprivileged product role reachable by any registered user. The trigger is a single valid `PATCH` API call with `{ listId: <id> }` on an approver that currently has a parent. No special knowledge, timing, or elevated access is required. [5](#0-4) 

## Recommendation
In the `typeof dto.listId === 'number'` branch, before writing the new `listId`, fetch the old parent (using `approver.listId`) and apply the same correction logic already present in the `dto.listId === null` branch:

- If the old parent's `approvers.length - 1 === 0`, soft-delete it.
- Else if `oldParent.threshold > oldParent.approvers.length - 1`, update `threshold` to `oldParent.approvers.length - 1`.

This mirrors the existing logic at lines 417–428 and closes the gap.

## Proof of Concept

**Setup:**
```
POST /transactions/{txId}/approvers
Body: { threshold: 2, approvers: [{ userId: 1 }, { userId: 2 }] }
```
This creates `Parent (threshold=2, children=[A(userId=1), B(userId=2)])`.

**Trigger:**
```
PATCH /transactions/{txId}/approvers/{A.id}
Body: { listId: <id of any other valid threshold node in the same transaction> }
```

**Result:**
- `A` is moved to the new parent.
- `Parent` now has `threshold=2` but only `[B]` as a child.
- `Parent.threshold (2) > Parent.approvers.length (1)` — invariant violated.
- No combination of approvals on `Parent`'s subtree can ever satisfy `threshold=2`.
- The transaction is permanently unapprovable with no recovery path.
