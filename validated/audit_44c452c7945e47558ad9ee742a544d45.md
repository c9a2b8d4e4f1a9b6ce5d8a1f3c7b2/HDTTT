### Title
Sequential Transaction Group Execution Permanently Blocked When Any Member Transaction Changes State (TOCTOU)

### Summary

`executeTransactionGroup` in `execute.service.ts` validates every transaction in a group against the live database before executing any of them. If any single transaction's status has changed since the group was loaded into memory (e.g., canceled by its creator or expired by the background cron), the entire group throws and is silently abandoned — no other transaction in the group is ever executed. This is a direct analog of the Notional M-9 pattern: a batch operation that reverts entirely when one item's state changes, blocking all remaining valid items.

### Finding Description

**Vulnerability class**: State-transition / TOCTOU batch processing.

**Code path**:

1. `handleTransactionsBetweenNowAndAfterThreeMinutes` (cron, every 10 s) calls `prepareTransactions`, which loads the full `TransactionGroup` from the DB — including all `groupItem.transaction` objects with their current statuses — and schedules `collateGroupAndExecute`. [1](#0-0) 

2. `collateGroupAndExecute` registers a `setTimeout` that fires at `validStart − 10 s`, then calls `addGroupExecutionTimeout`, which registers a second `setTimeout` that fires at `validStart + 5 s` and calls `executeService.executeTransactionGroup(transactionGroup)`. [2](#0-1) 

3. Inside `executeTransactionGroup`, the group items are first filtered using the **stale in-memory** `transaction.status` field: [3](#0-2) 

4. Then, for every item that passed the in-memory filter, `getValidatedSDKTransaction` is called, which immediately re-reads the status **from the database**: [4](#0-3) 

5. If the DB status is anything other than `WAITING_FOR_EXECUTION` (e.g., `CANCELED`, `EXPIRED`, `REJECTED`), `validateTransactionStatus` throws. That exception is caught and **re-thrown** as a group-level error: [5](#0-4) 

6. The group-level error propagates to `addGroupExecutionTimeout`'s callback, where it is silently swallowed with `console.log`: [6](#0-5) 

The timeout is deleted in the `finally` block and is never re-registered. All remaining valid transactions in the group are permanently skipped.

**TOCTOU window**: The group is loaded at cron time (T1). The execution callback fires at `validStart + 5 s` (T2). Any status change to any group member between T1 and T2 causes the entire group to fail.

**Natural trigger — expiry cron**: `handleExpiredTransactions` runs every 10 seconds and sets `WAITING_FOR_EXECUTION` transactions to `EXPIRED` when `validStart < now − 3 min`. [7](#0-6) 

If the scheduler is under load and the execution callback fires late (past the 3-minute window), the expiry cron can race ahead and expire one or more group members, triggering the group-level throw.

**Adversarial trigger**: A user who is the creator of any one transaction in a sequential group can cancel that transaction (via the `cancelTransaction` API) in the window between group scheduling and execution. The cancellation changes the DB status to `CANCELED`. When the execution callback fires, `validateTransactionStatus` detects the mismatch and throws, blocking all other transactions in the group from executing. [8](#0-7) 

### Impact Explanation

All transactions in a sequential group are permanently abandoned when any single member changes state. The group execution timeout is deleted and never rescheduled. Affected transactions remain in `WAITING_FOR_EXECUTION` until the expiry cron eventually marks them `EXPIRED` (up to 3 minutes later). Users lose the ability to execute their transactions with no error surfaced to them — only a `console.log` on the server side.

Severity: **Medium** — permanent loss of transaction execution for all group members, triggered by a single member's state change. No funds are directly stolen, but the integrity of the transaction lifecycle is broken.

### Likelihood Explanation

- **Natural race**: The expiry cron runs every 10 seconds against `WAITING_FOR_EXECUTION` transactions. Under scheduler load or clock skew, the execution callback can fire after the 3-minute expiry window, causing a natural collision.
- **Adversarial**: Any legitimate user who is the creator of one transaction in a group can deliberately cancel it in the scheduling window to block the entire group. No privileged access is required — only a valid user account with a transaction in the group.

### Recommendation

Mirror the fix recommended in the Notional report: instead of throwing and aborting the entire group when one transaction fails validation, **skip** the failing transaction and continue processing the rest. For sequential groups specifically:

```typescript
for (const groupItem of transactionGroup.groupItems) {
  const transaction = groupItem.transaction;
  try {
    const sdkTransaction = await this.getValidatedSDKTransaction(transaction);
    transactions.push({ sdkTransaction, transaction });
  } catch (error) {
-   throw new Error(
-     `Transaction Group cannot be submitted. Error validating transaction ${transaction.id}: ${error.message}`,
-   );
+   // Log and skip this transaction; do not abort the entire group
+   this.logger.warn(
+     `Skipping transaction ${transaction.id} in group ${transactionGroup.id}: ${error.message}`,
+   );
  }
}
```

For **atomic** groups (where all-or-nothing semantics are required), the current throw behavior is correct and should be preserved. The fix should branch on `transactionGroup.sequential` vs `transactionGroup.atomic`.

### Proof of Concept

1. User A and User B are both members of an organization. They create a **sequential** transaction group with two transactions: Tx-1 (creator: User A) and Tx-2 (creator: User B), both with `validStart = now + 30 s`.
2. The scheduler's 10-second cron picks up both transactions, marks them `WAITING_FOR_EXECUTION`, loads the group, and schedules `collateGroupAndExecute` (fires at `validStart − 10 s = now + 20 s`).
3. At `now + 15 s`, User A calls `DELETE /transactions/{tx1-id}` (cancel), changing Tx-1's DB status to `CANCELED`.
4. At `now + 20 s`, `collateGroupAndExecute` fires. The in-memory filter passes both transactions (both were `WAITING_FOR_EXECUTION` in memory at load time). `addGroupExecutionTimeout` is scheduled.
5. At `now + 35 s`, the execution callback fires and calls `executeTransactionGroup`. `validateTransactionStatus` re-reads Tx-1 from DB, finds `CANCELED`, and throws.
6. The throw is caught at line 341 with `console.log`. The timeout is deleted. Tx-2 (User B's valid transaction) is **never executed**.
7. Up to 3 minutes later, the expiry cron marks Tx-2 as `EXPIRED`. User B's transaction is permanently lost with no user-facing error. [9](#0-8) [6](#0-5)

  addGroupExecutionTimeout(transactionGroup: TransactionGroup) {
    const name = `group_execution_timeout_${transactionGroup.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart =
      transactionGroup.groupItems[0].transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransactionGroup(transactionGroup);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }

  private async validateTransactionStatus(transaction: Transaction) {
    const { status } = await this.transactionsRepo.findOne({
      where: { id: transaction.id },
      select: ['status'],
    });

    switch (status) {
      case TransactionStatus.NEW:
        throw new Error('Transaction is new and has not been signed yet.');
      case TransactionStatus.FAILED:
        throw new Error('Transaction has already been executed, but failed.');
      case TransactionStatus.EXECUTED:
        throw new Error('Transaction has already been executed.');
      case TransactionStatus.REJECTED:
        throw new Error('Transaction has already been rejected.');
      case TransactionStatus.EXPIRED:
        throw new Error('Transaction has been expired.');
      case TransactionStatus.CANCELED:
        throw new Error('Transaction has been canceled.');
      case TransactionStatus.ARCHIVED:
        throw new Error('Transaction is archived.');
    }
  }
