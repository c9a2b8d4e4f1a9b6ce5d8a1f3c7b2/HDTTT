# Audit Report

## Title
Manual Transaction Execution Allowed in `WAITING_FOR_SIGNATURES` State Due to Blocklist Status Guard

## Summary
`executeTransaction` in `transactions.service.ts` performs no transaction-status check before delegating to `ExecuteService`. The inner `validateTransactionStatus` guard uses a blocklist (throws on specific bad states) rather than an allowlist (only permits `WAITING_FOR_EXECUTION`). Because `WAITING_FOR_SIGNATURES` is absent from the blocklist, a transaction creator can force-submit a manual transaction to the Hedera network during the race window between the `transactionBytes` DB write and the status promotion write, producing a permanent state split: the network considers the transaction executed while the database eventually marks it `EXPIRED`.

## Finding Description

**Gap 1 — missing status assertion in `executeTransaction`:**

`back-end/apps/api/src/transactions/transactions.service.ts` L736–751 checks only `isManual`; it never asserts the transaction is in `WAITING_FOR_EXECUTION` before delegating to `executeService.executeTransaction`. [1](#0-0) 

**Gap 2 — blocklist in `validateTransactionStatus`:**

`back-end/libs/common/src/execute/execute.service.ts` L226–248 enumerates every terminal/invalid state explicitly but omits `WAITING_FOR_SIGNATURES`. The switch-case falls through without throwing, and execution proceeds to network submission. [2](#0-1) 

**Contrast with the correct pattern:**

`executeTransactionGroup` (L65–67) uses an allowlist filter — only `WAITING_FOR_EXECUTION` items are processed. The single-transaction path lacks an equivalent guard. [4](#0-3) 

**Race window:**

`processTransactionStatus` writes the updated `transactionBytes` (with all signatures) to the DB first, then separately writes the status promotion to `WAITING_FOR_EXECUTION`. Between these two writes, the row is in `WAITING_FOR_SIGNATURES` with fully-signed bytes — exactly the state the exploit requires. [5](#0-4) 

**DB update failure:**

`_executeTransaction` (L191–193) conditions its update on `status = WAITING_FOR_EXECUTION`. When the row is still `WAITING_FOR_SIGNATURES`, `updateResult.raw.length === 0` and the function returns `null` — no DB update, no notification, no record of execution. [7](#0-6) 

## Impact Explanation

- **Permanent state split:** A successfully executed Hedera transaction is recorded as `EXPIRED` in the database. All downstream logic (notifications, audit trails, UI status) reflects a false failure.
- **Double network submission:** The scheduler later promotes the row to `WAITING_FOR_EXECUTION` and re-submits. Hedera returns `DUPLICATE_TRANSACTION`; the duplicate-handling path (L168–172) returns `null` and skips the DB update, leaving the row in `WAITING_FOR_EXECUTION` until the valid-start window closes and it is marked `EXPIRED`. [8](#0-7) 
- **Integrity failure:** The organization's authoritative record of which transactions executed is corrupted for any transaction the creator races through this path.

## Likelihood Explanation

Preconditions are low-privilege: only the transaction's own creator is required. The race window (between the `transactionBytes` write and the status write in `processTransactionStatus`) is narrow but real and widens under load or in multi-pod deployments. The creator can observe when the last required signer uploads their signature (via notifications or polling) and immediately issue the `POST /transactions/:id/execute` call, making the timing attack practical rather than purely theoretical.

## Recommendation

Replace the blocklist in `validateTransactionStatus` with an allowlist:

```typescript
private async validateTransactionStatus(transaction: Transaction) {
  const { status } = await this.transactionsRepo.findOne({
    where: { id: transaction.id },
    select: ['status'],
  });

  if (status !== TransactionStatus.WAITING_FOR_EXECUTION) {
    throw new Error(`Transaction cannot be executed in status: ${status}`);
  }
}
```

This single change closes both gaps simultaneously: any status other than `WAITING_FOR_EXECUTION` — including `WAITING_FOR_SIGNATURES` — will throw before reaching network submission.

## Proof of Concept

1. Creator creates a manual transaction requiring two signers (A and B).
2. Signer A uploads their signature map via `POST /transactions/:id/sign`. Status remains `WAITING_FOR_SIGNATURES`.
3. Signer B uploads their signature map. The service writes the fully-signed `transactionBytes` to the DB. `processTransactionStatus` is called but has not yet committed the status update to `WAITING_FOR_EXECUTION`.
4. During this window, the creator calls `POST /transactions/:id/execute`.
5. `executeTransaction` passes the `isManual` check (L739).
6. `validateTransactionStatus` reads `WAITING_FOR_SIGNATURES` from the DB — not in the blocklist — returns without throwing (L232–247).
7. `hasValidSignatureKey` returns `true` because the fully-signed bytes are already in `transactionBytes` (L219).
8. `sdkTransaction.execute(client)` submits the transaction to the Hedera network. **Transaction is now live on-chain.**
9. `_executeTransaction` attempts the DB update with `status = WAITING_FOR_EXECUTION` (L191–193). The row is still `WAITING_FOR_SIGNATURES`, so `affected = 0`, function returns `null`. No DB update, no notification.
10. `processTransactionStatus` commits — status becomes `WAITING_FOR_EXECUTION`.
11. The scheduler picks up the row and re-submits. Hedera returns `DUPLICATE_TRANSACTION`. The duplicate path (L168–172) returns `null`, skipping the DB update.
12. After the 3-minute valid-start window closes, the scheduler marks the row `EXPIRED`. The transaction succeeded on-chain but the database permanently records it as expired.
