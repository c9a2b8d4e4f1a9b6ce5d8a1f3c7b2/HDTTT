Audit Report

## Title
Missing Transaction Existence and Authorization Check in `createComment` Allows Any Authenticated User to Post Comments on Arbitrary Transactions

## Summary
`createComment` in `CommentsService` accepts a caller-supplied `transactionId` and immediately persists a comment without verifying that the referenced transaction exists or that the requesting user has any relationship to it. The controller itself contains a developer `//TODO` comment explicitly acknowledging the missing guard. Every analogous write path in the same service layer performs both checks; `createComment` skips them entirely.

## Finding Description
`createComment` at lines 15–24 of `back-end/apps/api/src/transactions/comments/comments.service.ts`:

```typescript
async createComment(
  user: User,
  transactionId: number,
  dto: CreateCommentDto,
): Promise<TransactionComment> {
  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);   // no existence check, no authz check
}
``` [1](#0-0) 

The function performs no database lookup to confirm a `Transaction` row with `id = transactionId` exists, and performs no check that the calling user is a creator, signer, observer, or approver of that transaction.

The controller that invokes this function carries only authentication-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) and contains an explicit developer acknowledgement of the missing check:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(...)
``` [2](#0-1) 

`VerifiedUserGuard` only checks `user.status === UserStatus.NONE` — it enforces account verification, not transaction-level authorization. [3](#0-2) 

By contrast, every other write path in the same service layer performs both an existence check and an authorization check:

- `ObserversService.createTransactionObservers` fetches the transaction, throws `ErrorCodes.TNF` if absent, then asserts `transaction.creatorKey?.userId !== user.id`. [4](#0-3) 

- `ApproversService.createTransactionApprovers` delegates to `getCreatorsTransaction`, which throws `ErrorCodes.TNF` on a missing transaction and `UnauthorizedException` if the caller is not the creator. [5](#0-4) 

The `TransactionComment` entity declares `transaction` as a `@ManyToOne` relation with no nullable override, meaning the database enforces a foreign-key constraint. If `transactionId` references a non-existent row, the unhandled DB exception propagates as an unmasked 500 response. [6](#0-5) 

## Impact Explanation
- **Unauthorized state modification**: Any authenticated user can inject comments into transactions they have no legitimate relationship to, corrupting the audit trail and communication channel for every transaction in the system.
- **Integrity degradation**: Comment history used for compliance or coordination can be polluted by arbitrary users.
- **Information leakage via error responses**: Submitting a non-existent `transactionId` causes an unhandled DB exception to propagate as a 500 response, potentially leaking internal error details (stack traces, constraint names, table names).

## Likelihood Explanation
Exploitation requires only a valid session token — no elevated privileges, no leaked credentials, no special network position. Transaction IDs are sequential integers (`@PrimaryGeneratedColumn()`), making enumeration trivial. The attack surface is the normal REST API endpoint `POST /transactions/:transactionId/comments`, which is fully reachable by any authenticated user. [7](#0-6) 

## Recommendation
Add the same two-step guard used by every other write path in the service layer:

1. **Existence check**: Query the `Transaction` table for `id = transactionId`; throw `BadRequestException(ErrorCodes.TNF)` if absent.
2. **Authorization check**: Verify the calling user is the creator, a signer, an observer, or an approver of the transaction before allowing the comment to be persisted. The pattern already implemented in `ObserversService.getTransactionObserversByTransactionId` (lines 72–113) and `ApproversService.getVerifiedApproversByTransactionId` (lines 111–152) can be reused or extracted into a shared helper. [8](#0-7) 

## Proof of Concept
1. Authenticate as User A (any valid account with no relationship to Transaction T owned by User B).
2. Send: `POST /transactions/{T.id}/comments` with body `{ "message": "injected comment" }` and User A's JWT.
3. `createComment` is invoked with `transactionId = T.id`; no existence or authorization check is performed.
4. The comment is persisted and associated with Transaction T.
5. All legitimate participants of Transaction T (creator, signers, observers, approvers) now see a comment injected by an unauthorized party.
6. To trigger the information-leakage path, send the same request with a non-existent `transactionId`; the unhandled FK violation propagates as a 500 response with internal error details.

  async getTransactionObserversByTransactionId(
    transactionId: number,
    user: User,
  ): Promise<TransactionObserver[]> {
    const transaction = await this.entityManager.findOne(Transaction, {
      where: { id: transactionId },
      relations: ['creatorKey', 'observers', 'signers', 'signers.userKey'],
    });

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    const userKeysToSign = await userKeysRequiredToSign(
      transaction,
      user,
      this.transactionSignatureService,
      this.entityManager,
    );

    const approvers = await this.approversService.getApproversByTransactionId(transaction.id);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return transaction.observers;

    if (
      userKeysToSign.length === 0 &&
      transaction.creatorKey?.userId !== user.id &&
      !transaction.observers.some(o => o.userId === user.id) &&
      !transaction.signers.some(s => s.userKey?.userId === user.id) &&
      !approvers.some(a => a.userId === user.id)
    )
      throw new UnauthorizedException("You don't have permission to view this transaction");

    return transaction.observers;
  }
```

  async getCreatorsTransaction(
    transactionId: number,

    const find: FindOneOptions<Transaction> = {

      relations: ['creatorKey', 'creatorKey.user'],

      : this.dataSource.manager.findOne(Transaction, find));

      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
