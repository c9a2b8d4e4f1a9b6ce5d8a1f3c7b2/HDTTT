Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated Resource Exhaustion DoS

## Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction in the entire database without any SQL-level `LIMIT`, then performs a sequential per-transaction async cryptographic key-computation call (`userKeysToSign`) for each row before applying pagination in application memory. An authenticated attacker who pre-creates a large number of transactions causes every subsequent call to `GET /transactions/sign` to perform O(N) database queries and O(N) cryptographic operations, progressively degrading and eventually making the API server unresponsive — with a single HTTP request.

## Finding Description

**Root cause — unbounded `repo.find()` with no `take`:**

In `getTransactionsToSign`, the database query has no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / no `limit`
});
``` [1](#0-0) 

The `whereForUser` filter excludes only terminal statuses. Critically, it does **not** filter by the requesting user — every transaction in `NEW`, `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, and `REJECTED` states across **all users** is loaded into memory:

```typescript
const whereForUser: FindOptionsWhere<Transaction> = {
  ...where,
  status: Not(In([EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED])),
};
``` [2](#0-1) 

**Per-row async work:**

For every loaded transaction, `userKeysToSign` is called sequentially. This calls `userKeysRequiredToSign` → `keysRequiredToSign`, which deserializes the transaction bytes via `SDKTransaction.fromBytes()` and calls `transactionSignatureService.computeSignatureKey()`, which makes multiple async DB/cache lookups (fee payer key, signing account keys, receiver account keys, node keys): [3](#0-2) [4](#0-3) 

**Pagination applied after full scan:**

The `limit`/`offset` from the request is applied only after the full in-memory result is built:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [5](#0-4) 

So even a request for `page=1&size=1` causes the server to load and process every non-terminal transaction in the database.

**The endpoint remains registered and reachable:**

The controller comment says "NO LONGER USED BY FRONT-END" but the route is still active and protected only by JWT + verified-user guards — no additional throttle decorator is applied to this specific endpoint: [6](#0-5) 

**Global IP throttler does not prevent this:**

The `IpThrottlerGuard` limits the number of requests per IP per time window (default: 10,000/minute, 1,000/second), but a **single request** with N=100,000 transactions in the DB still causes the server to do O(N) work per call. The throttler limits request count, not per-request computational cost: [7](#0-6) 

**Transaction creation has no per-user cap:**

Any authenticated user with at least one registered key can create transactions. There is no limit on how many transactions a single user may create: [8](#0-7) 

## Impact Explanation

- **CPU exhaustion**: Each call to `GET /transactions/sign` triggers O(N) calls to `computeSignatureKey`, which involves `SDKTransaction.fromBytes()` deserialization and multiple async DB lookups per transaction.
- **Memory exhaustion**: All N transaction rows (including their `transactionBytes` blobs) are loaded into the Node.js heap simultaneously.
- **Cascading degradation**: As N grows, response latency increases linearly. At sufficient scale, the NestJS event loop stalls, causing all API endpoints to time out — not just `/transactions/sign`.
- **Cross-user impact**: Because the scan is not scoped to the requesting user's transactions, one attacker's transactions degrade the experience for all platform users.

## Likelihood Explanation

- **Attacker preconditions**: A registered, verified account with one uploaded key — the normal onboarding flow.
- **Attack cost**: Creating each transaction requires constructing a valid Hedera SDK transaction and signing it with the registered key. This is fully scriptable using the open-source Hedera SDK. The attacker's only cost is registration and key upload.
- **No rate limit on transaction creation per user**: `POST /transactions` is subject only to the global IP throttler (10,000/minute), which is designed for general abuse prevention, not for preventing bulk transaction creation by a single authenticated user.
- **Single-request amplification**: One HTTP request to `GET /transactions/sign` triggers O(N) server-side work. The attacker does not need to send many requests — they only need to have pre-created many transactions.

## Recommendation

1. **Push pagination to the database layer**: Add `take: limit` and `skip: offset` to the `repo.find()` call. This requires restructuring the query to filter by user-relevant keys at the SQL level (similar to how `getTransactionsToApprove` uses a subquery with `andWhere`), rather than filtering in application memory.
2. **Scope the query to the requesting user**: The `whereForUser` filter must include a user-scoping condition (e.g., joining on `signers.userId = :userId` or a similar subquery) so the DB only returns transactions relevant to the requesting user.
3. **Add a per-user transaction creation limit**: Implement a rate limit or hard cap on the number of active (non-terminal) transactions per user.
4. **Remove or disable the endpoint**: Since the controller comment already states "NO LONGER USED BY FRONT-END", the endpoint should be removed or explicitly disabled if it is no longer needed.

## Proof of Concept

```python
import requests, time

BASE = "https://<target>/api"
TOKEN = "<jwt_of_attacker>"

# Step 1: Create N transactions (scriptable with Hedera SDK)
# Each transaction needs a unique transactionId (account + timestamp)
for i in range(10_000):
    requests.post(f"{BASE}/transactions", json={
        "name": f"tx-{i}",
        "transactionBytes": "<hex_encoded_bytes_with_unique_id>",
        "creatorKeyId": 1,
        "signature": "<valid_sig>",
        "mirrorNetwork": "testnet"
    }, headers={"Authorization": f"Bearer {TOKEN}"})

# Step 2: Single request causes O(N) server-side work
start = time.time()
r = requests.get(f"{BASE}/transactions/sign?page=1&size=1",
                 headers={"Authorization": f"Bearer {TOKEN}"})
print(f"Response time: {time.time() - start:.1f}s, status: {r.status_code}")
# Expected: response time grows linearly with N; at large N, request times out
# and the event loop stall affects all other API endpoints
```

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
