Audit Report

## Title
Any Authenticated User Can Squat Another User's Public Key, Permanently Blocking Them From the Organization Signing Workflow

## Summary
The `POST /user/:userId?/keys` endpoint registers a public key under the **JWT-authenticated caller's** account with no cryptographic proof of private-key ownership. Because the `publicKey` column is looked up globally, any verified organization user who registers a victim's public key first permanently prevents the victim from registering that key under their own account, locking them out of the multi-signature transaction workflow.

## Finding Description

**Root cause — no proof-of-possession check**

`UploadUserKeyDto` accepts only `publicKey`, `mnemonicHash`, and `index`. No challenge-response or signature-over-identity is required. [1](#0-0) 

**Controller ignores the `:userId` path parameter for writes**

The controller is mounted at `user/:userId?/keys`, but `uploadKey` binds only `@GetUser()` (the JWT identity) — the path parameter is never read for the POST handler. [2](#0-1) 

**Service enforces global uniqueness, blocking the true owner**

`uploadKey()` queries `publicKey` across all users (`withDeleted: true`). If the row already exists and `userKey.userId !== user.id`, it throws `BadRequestException(ErrorCodes.PU)` — permanently blocking the legitimate owner. [3](#0-2) 

**Exploit flow**

1. Victim generates an ED25519 key pair. The public key is visible on the Hedera network (mirror node, transaction bytes, multi-sig coordination).
2. Attacker (any verified org user) calls `POST /user/<any_id>/keys` with `{ "publicKey": "<victim_pubKey>" }`. The `:userId` path param is ignored; the key is registered to the attacker.
3. Victim later calls `POST /user/<victim_id>/keys` with the same public key. The service finds the existing row, sees `userKey.userId !== victim.id`, and throws `400 ErrorCodes.PU`.
4. Victim is permanently blocked from registering their own key.

**E2E test confirms the blocking behavior** [4](#0-3) 

## Impact Explanation

**Permanent DoS on key registration**: The victim can never register `pubKey_V` under their own account. The only recovery path requires an admin to manually delete the attacker's `UserKey` row.

**Exclusion from multi-sig workflows**: `validateAndProcessSignatures` builds `userKeyMap` exclusively from the authenticated submitter's registered keys. Because the victim's key is not in their own key map, the victim cannot submit signatures for transactions requiring `pubKey_V`, even though they hold the private key. [5](#0-4) [6](#0-5) 

**Organizational integrity break**: The `UserKey` entity is the authoritative link between a Hedera public key and an organization user. Squatting corrupts this mapping, causing the system to attribute signing authority to the wrong user.

## Likelihood Explanation

- **Attacker precondition**: Any verified organization account. No admin or privileged role required.
- **Information required**: The victim's public key — which is inherently public. It appears in Hedera transaction bytes, mirror node account queries, and is shared during normal multi-sig coordination.
- **Attack complexity**: A single authenticated HTTP POST. No race condition or timing dependency is required; the attacker simply needs to register the key before the victim does.
- **Detection difficulty**: The attack is indistinguishable from a normal key upload at the API level.

## Recommendation

Require cryptographic proof of private-key ownership before registering a public key. The standard approach is a **challenge-response / proof-of-possession** flow:

1. Client requests a server-generated nonce tied to the session.
2. Client signs the nonce with the private key corresponding to the public key being registered.
3. Server verifies the signature against the submitted public key before creating the `UserKey` row.

This ensures only the holder of the private key can register the corresponding public key, eliminating the squatting attack entirely.

## Proof of Concept

```http
# Step 1 — Attacker registers victim's public key (victim has not yet registered it)
POST /user/999/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{
  "publicKey": "<victim_ed25519_public_key_hex>"
}
# Response: 201 Created — key is now owned by attacker

# Step 2 — Victim attempts to register their own key
POST /user/2/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json

{
  "publicKey": "<victim_ed25519_public_key_hex>"
}
# Response: 400 Bad Request — ErrorCodes.PU ("Public key in use")
# Victim is permanently locked out of their own key registration
```
