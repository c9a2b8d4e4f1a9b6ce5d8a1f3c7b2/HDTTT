### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Authenticated DoS via Single Request

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then iterates over the entire result set in-process — calling an async `userKeysToSign` (which involves mirror-node I/O) for each row — before applying pagination. Any authenticated, verified user can trigger this via `GET /transactions/sign`, and as the platform accumulates transactions the request cost grows without bound, eventually starving the Node.js event loop and degrading service for all users.

### Finding Description

**Root cause — no `take` on the database query:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues an unbounded `find()`:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / no `skip`
});
``` [1](#0-0) 

This returns every row whose status is not in `{EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED}`. On a live platform those rows accumulate indefinitely.

**Unbounded in-process loop with async I/O per iteration:**

The code then iterates over the full result set, awaiting an async call per transaction:

```ts
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which performs mirror-node network calls to resolve key structures. Each iteration is therefore an async I/O round-trip, not a cheap in-memory check.

**Pagination applied post-hoc:**

```ts
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

The `limit`/`offset` from the caller's `PaginationParams` are never passed to the database query; they are applied only after the full scan completes. Requesting `page=1&size=1` still forces the server to load and process every active transaction.

**Endpoint is still live:**

The controller registers the route and accepts any authenticated, verified user:

```ts
@Get('/sign')
getTransactionsToSign(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
  ...
)
``` [4](#0-3) 

The comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end has moved on, but the route is still registered and reachable.

### Impact Explanation

A single authenticated HTTP request to `GET /transactions/sign` forces the server to:
1. Load all non-terminal transactions into memory (unbounded DB read).
2. Perform one mirror-node I/O round-trip per transaction (unbounded async loop).

As the transaction table grows, response latency grows linearly. At sufficient scale (thousands of active transactions), the Node.js event loop is saturated by a single request, causing timeouts and HTTP 5xx errors for all concurrent users. Because the endpoint requires only a valid JWT (any verified user), the attacker precondition is minimal.

### Likelihood Explanation

Any registered, verified user of the organization backend can trigger this with a single `GET /transactions/sign` request. No special role, no admin access, no leaked credentials are required. The platform is designed for multi-user organizations that accumulate transactions over time, so the unbounded table naturally grows during normal operation. The attack is trivially repeatable in a tight loop to sustain degradation.

### Recommendation

Apply the database-level limit **before** the loop, mirroring the pattern already used in `getHistoryTransactions`:

```ts
// Pass take/skip to the DB query
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,
  skip: offset,
});
``` [5](#0-4) 

Because the eligibility check (`userKeysToSign`) cannot be pushed into SQL directly, the correct approach is to over-fetch a bounded window (e.g., `take: limit * N`) and filter in-process, or to move the key-eligibility logic into a SQL subquery so the database can filter before returning rows. Additionally, if the endpoint is truly unused by the front-end, it should be removed or protected behind an admin guard to eliminate the attack surface entirely.

### Proof of Concept

**Preconditions:** A valid JWT for any verified user; an organization with a large number of active (non-terminal) transactions.

**Steps:**
1. Authenticate as any verified user: `POST /auth/login` → obtain `access_token`.
2. Ensure the platform has accumulated N active transactions (normal usage over time, or by creating them via `POST /transactions`).
3. Issue: `GET /transactions/sign?page=1&size=1` with the JWT.
4. Observe: server loads all N transactions from the DB, then awaits `userKeysToSign` N times (each involving a mirror-node call). Response time scales as O(N × mirror-node-latency).
5. Repeat the request in a loop; the event loop remains saturated, causing timeouts for all other concurrent API consumers.

**Expected outcome:** With N = 10,000 active transactions and a 50 ms mirror-node RTT, a single request holds the event loop for ~500 seconds of cumulative async work, effectively rendering the API unavailable.
