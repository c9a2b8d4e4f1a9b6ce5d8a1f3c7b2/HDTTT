Audit Report

## Title
Missing Cryptographic Signature Verification in `processTransactionSignatures` Allows Authenticated User to Falsely Register as a Transaction Signer

## Summary
`SignersService.processTransactionSignatures` verifies that a submitted public key belongs to the authenticated user but never verifies that the accompanying signature bytes are cryptographically valid for that key. An authenticated user can submit a `SignatureMap` containing their own registered public key paired with arbitrary/garbage bytes. The system accepts it, persists the attacker as a `TransactionSigner`, and overwrites the stored `transactionBytes` with the corrupted version.

## Finding Description

**Root cause — `processTransactionSignatures` in `signers.service.ts`:**

The loop at lines 234–258 iterates over every public key in the submitted `SignatureMap`, checks ownership via `userKeyMap`, then unconditionally calls `sdkTransaction.addSignature(publicKey, map)`: [1](#0-0) 

The ownership check (`if (!userKey) throw new Error(ErrorCodes.PNY)`) is the only gate. There is no call to `PublicKey.verify()` or `validateSignature` before or after `addSignature`. The Hedera SDK's `addSignature` does not cryptographically verify signature bytes — it blindly appends whatever bytes are in the map.

**Contrast with `validateSignature`**, which does perform cryptographic verification using `publicKey.verify(bodyBytes, signature)` and throws on failure: [2](#0-1) 

This function is available and used elsewhere in the codebase but is never invoked in the `uploadSignatureMaps` → `processTransactionSignatures` path.

**Byte corruption and signer persistence:**

After `addSignature` appends the garbage bytes, `isSameBytes` is computed by comparing the resulting transaction bytes to the original: [3](#0-2) 

Because the bytes differ, `persistSignatureChanges` overwrites `transactionBytes` in the database and inserts the attacker as a `TransactionSigner`: [4](#0-3) 

**Exploit flow:**
1. Attacker authenticates (valid JWT) and has one registered public key `K`.
2. Attacker constructs a `SignatureMap` with key `K` paired with 64 bytes of garbage (e.g., all zeros).
3. Attacker POSTs to `POST /transactions/<id>/signers`.
4. Ownership check passes (`K` is in `userKeyMap`).
5. `addSignature(K, garbage)` appends invalid bytes → `isSameBytes = false`.
6. `userKeys` contains `K` → attacker inserted as `TransactionSigner`.
7. `transactionBytes` overwritten with the corrupted version.

## Impact Explanation
- **False signer accounting**: The `TransactionSigner` table records the attacker as having signed. If a threshold of signers is required, this false entry can make the system believe the threshold is met when it is not.
- **Transaction byte corruption**: All subsequent legitimate signers download and re-sign from corrupted bytes, propagating the corruption.
- **On-chain failure / DoS**: When the chain service submits the transaction, the Hedera network rejects it due to the invalid signature. The transaction cannot be recovered because the stored bytes are already corrupted.
- **Integrity break**: The trust model of the multi-signature workflow is broken — the system no longer accurately reflects who has actually signed.

## Likelihood Explanation
- **Preconditions**: Only a valid JWT and one registered public key — both are normal user-level capabilities requiring no privilege escalation.
- **Attack complexity**: Constructing a `SignatureMap` with a known public key and garbage signature bytes is trivial using the Hedera SDK or by crafting the protobuf directly.
- **No detection**: The system logs no error and returns a success response; the attack is silent.
- **Reachable endpoint**: `POST /transactions/:id/signers` is a standard API endpoint protected only by JWT authentication.

## Recommendation
Add a call to `validateSignature` (already implemented in `back-end/libs/common/src/utils/sdk/transaction.ts` at line 213) inside `processTransactionSignatures`, before calling `addSignature`, to cryptographically verify each signature byte array against the transaction body bytes using `publicKey.verify()`. If verification fails, throw a `BadRequestException` and reject the entire request. This mirrors the verification already performed in the `importSignatures` path. [5](#0-4)

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
