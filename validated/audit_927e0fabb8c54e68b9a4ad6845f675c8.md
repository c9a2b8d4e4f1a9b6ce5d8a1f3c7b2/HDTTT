### Title
Unbounded In-Memory Loop with Per-Iteration DB Queries in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then executes a separate DB query per transaction inside a loop to evaluate signing requirements. Pagination is applied only after the full in-memory scan. Any authenticated user can trigger this endpoint; as the number of active transactions in the system grows, each call consumes proportionally more server CPU, memory, and database connections, eventually causing request timeouts and degraded availability for all users.

### Finding Description

**Root cause — no DB-level limit before the loop:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

There is no `take` or `skip` on this query. Every non-terminal transaction in the system is loaded into memory.

**Per-transaction DB query inside the loop:** [2](#0-1) 

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

`userKeysToSign` delegates to `userKeysRequiredToSign` → `TransactionSignatureService.computeSignatureKey`, which performs additional database lookups per transaction. This is a classic N+1 query pattern with N unbounded.

**Fake pagination applied after full scan:** [3](#0-2) 

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  page,
  size,
};
```

The caller receives a paginated response, but the server has already paid the full cost of scanning every active transaction and issuing N DB queries.

**Contrast with sibling methods** that correctly apply `skip`/`take` at the DB level: [4](#0-3) 

### Impact Explanation

- **Server CPU/memory exhaustion**: loading thousands of transaction rows and their relations into the Node.js process heap.
- **Database connection starvation**: N sequential awaited queries per request hold a DB connection for the full duration.
- **Cascading availability degradation**: because the loop is synchronous-per-iteration (`await` inside `for`), a single request with a large transaction set blocks the event loop for an extended period, degrading response times for all concurrent users.
- **No self-healing**: the problem worsens monotonically as the organization accumulates transactions; there is no expiry of the load.

### Likelihood Explanation

- **Attacker precondition**: a valid authenticated session only — no admin role required.
- **Trigger**: call `GET /transactions/sign` (or equivalent endpoint backed by `getTransactionsToSign`). The cost is borne entirely by the server.
- **Amplification**: a malicious user can be added as an observer or signer to a large number of transactions by other users, or can create many transactions themselves (each requiring only a valid Hedera transaction body and a self-signed signature). Organic system growth achieves the same effect without any deliberate attack.
- **Repeatability**: the endpoint can be called in a tight loop; there is no rate-limiting visible in the service layer.

### Recommendation

1. **Apply DB-level pagination before the loop.** Pass `take: limit` and `skip: offset` to `this.repo.find(...)` so the database returns only the page of rows the caller needs.
2. **Push the user-key filter into the database.** Rewrite the query to join against the user's keys and the required-signers view so that only transactions actually requiring the user's signature are returned, eliminating the per-row `userKeysToSign` call entirely.
3. **Add an API-layer rate limit** on this endpoint to bound the request rate per authenticated user.

### Proof of Concept

1. Register as a normal user (no admin privileges).
2. Create or be added as a signer/observer to a large number of active (non-terminal) transactions — e.g., 10,000 transactions across the organization.
3. Issue repeated `GET /transactions/sign` requests.
4. Observe: each request causes the server to execute `SELECT * FROM transaction WHERE status NOT IN (...)` returning all 10,000 rows, followed by 10,000 sequential `userKeysToSign` DB calls.
5. Server response times degrade for all users; under sustained load the process exhausts its DB connection pool or Node.js heap, causing 503 errors or process crashes. [5](#0-4)

```typescript
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
