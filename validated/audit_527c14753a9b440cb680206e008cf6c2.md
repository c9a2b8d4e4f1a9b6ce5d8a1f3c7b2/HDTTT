Audit Report

## Title
Unbounded Full-Table Scan with Per-Row Async DB Work in `getTransactionsToSign` Enables Targeted Resource Exhaustion

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then performs a sequential async DB call per row to determine signing eligibility. Pagination is applied in-memory only after the full scan completes. An authenticated attacker can pre-populate the database with transactions requiring a victim's public key, causing the victim's subsequent requests to trigger O(N) DB queries, leading to extreme latency or connection pool exhaustion.

## Finding Description

**Root cause — unbounded query with per-row async work**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` (lines 295–313):

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // filters only on STATUS, not on user identity
  relations: ['groupItem'],
  order,
  // NO `take` / limit
});

for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user); // async DB call per row
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { console.log(error); }
}

return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit), // pagination applied AFTER full scan
  ...
};
``` [1](#0-0) 

The `whereForUser` predicate filters only on `status` (excluding terminal states), not on any user-specific column, meaning the query returns every non-terminal transaction in the entire system. [2](#0-1) 

Each `userKeysToSign` call delegates to `userKeysRequiredToSign`, which calls `keysRequiredToSign`, which in turn calls `transactionSignatureService.computeSignatureKey` and `entityManager.find(UserKey, ...)` — two additional async DB operations per transaction row. [3](#0-2) [4](#0-3) 

The `PaginationParams` decorator enforces a maximum page `size` of 100, but this only limits the final `result.slice()` — it does not limit the upstream `repo.find()` or the sequential loop. [5](#0-4) 

**Contrast with `getTransactionsToApprove`**

The sibling method `getTransactionsToApprove` correctly applies `skip`/`take` at the DB query level, pushing pagination into the database and avoiding the full-scan problem. [6](#0-5) 

**Endpoint still accessible**

Although the controller comment notes `/* NO LONGER USED BY FRONT-END */`, the `GET /transactions/sign` endpoint remains registered and reachable by any authenticated API client. [7](#0-6) 

## Impact Explanation

- **Victim's signing workflow is blocked**: The victim cannot reliably retrieve their list of transactions to sign, which is a core platform function.
- **Server-side resource exhaustion**: Each victim request triggers O(N) sequential async DB queries. With enough attacker-created transactions, this saturates the DB connection pool and degrades service for all users on the same server instance.
- **Persistent state**: Attacker-created transactions remain in the DB until they expire or are manually cleaned up. The attack is durable and does not require the attacker to remain active.
- **No victim-side recovery**: The victim cannot filter out attacker-created transactions because the full-table scan and per-row loop are server-side with no user-scoped DB predicate.

## Likelihood Explanation

- **Attacker preconditions**: Only a valid, non-privileged user account is required.
- **Cost to attacker**: Creating transactions is a free API operation (no on-chain fees for backend storage). An attacker can script thousands of requests with minimal effort.
- **Victim's public key**: Hedera public keys are publicly visible on-chain and via the mirror node API, making target selection trivial.
- **No existing mitigations**: There is no `take` limit on the `repo.find()` call, no per-user transaction creation rate limit visible in the codebase, and no cap on how many transactions can reference a given public key.

## Recommendation

1. **Push pagination to the database**: Add `take: limit` and `skip: offset` to the `repo.find()` options in `getTransactionsToSign`, mirroring the pattern already used in `getTransactionsToApprove`.
2. **Pre-filter by user keys at the DB level**: Add a SQL subquery or join that filters transactions to only those whose required signers include at least one of the requesting user's public keys, eliminating the need to load and check every non-terminal row.
3. **Consider removing or guarding the endpoint**: Since the front-end no longer uses `GET /transactions/sign`, consider deprecating or removing it, or adding an explicit authorization guard that limits its use.

## Proof of Concept

1. Attacker registers a normal user account.
2. Attacker obtains the victim's Hedera public key (publicly available via mirror node).
3. Attacker crafts N valid Hedera SDK transactions (e.g., `AccountUpdateTransaction` with the victim's key as the new key) and submits each via `POST /transactions` using their own `creatorKeyId`. These are stored in the backend DB.
4. Victim calls `GET /transactions/sign?page=1&size=10`.
5. Server executes `repo.find({ where: { status: Not(In([...terminalStatuses])) } })` — returns all N attacker rows plus any legitimate rows, with no limit.
6. Server calls `userKeysToSign(transaction, victim)` N times sequentially, each making 2+ async DB queries.
7. The request either times out at the HTTP layer, exhausts the DB connection pool, or returns after extreme delay — permanently degrading the victim's ability to view and sign transactions.
