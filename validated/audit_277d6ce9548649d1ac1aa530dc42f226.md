### Title
Unbounded Approvers Array in `POST /transactions/:transactionId/approvers` Enables Single-Request Resource Exhaustion DoS

### Summary
The `createTransactionApprovers` endpoint accepts a `CreateTransactionApproversArrayDto` whose `approversArray` field has no maximum size constraint. Each element of that array can itself contain a nested `approvers` array — also with no size or depth limit — which the service processes via an unbounded recursive function that issues a sequential database insert per node. A single crafted request from any authenticated user can trigger thousands of sequential DB writes and unbounded recursion, exhausting server resources and degrading or crashing the API service.

### Finding Description

**Root cause — missing `@ArrayMaxSize` at every level of the DTO:**

`CreateTransactionApproversArrayDto.approversArray` and the recursive `CreateTransactionApproverDto.approvers` field both lack any `@ArrayMaxSize` decorator. [1](#0-0) 

The outer array (`approversArray`) has only `@IsArray` and `@ValidateNested`. The inner recursive field (`approvers`) has only `@IsArray`, `@ArrayMinSize(1)`, and `@ValidateNested`. Neither has an upper bound.

**Service processes the tree with unbounded sequential recursion:**

The inner `createApprover` closure is called sequentially with `await` for every element of `dto.approversArray`, and then recursively for every element of each `dtoApprover.approvers`: [2](#0-1) 

Each recursive call issues a `transactionalEntityManager.insert(TransactionApprover, approver)` — a synchronous DB round-trip — before descending into children. The entire tree is processed inside a single database transaction, holding it open for the full duration.

**Controller exposes the endpoint to any authenticated user:** [3](#0-2) 

Guards are `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` only — no admin or owner-only restriction beyond what the service enforces internally.

### Impact Explanation

- **Flat attack**: Sending `approversArray` with N=10,000 leaf approvers causes N sequential DB inserts inside one open transaction, blocking the DB connection pool and the NestJS event loop for the duration.
- **Recursive attack**: A tree of depth D with branching factor B causes B^D recursive calls. At depth 13 / branching 10 the call count exceeds 10^13; in practice the Node.js call stack overflows (~10,000–15,000 frames), crashing the request handler and potentially the process.
- **Cascading effect**: Because the inserts run inside a single `dataSource.transaction`, a crash or timeout leaves the DB transaction open until the connection is forcibly closed, consuming a DB connection slot for the entire duration.
- **Result**: API service unavailability or severe degradation for all users.

### Likelihood Explanation

Any registered, verified user of the organization backend can reach this endpoint. The user only needs a valid JWT and a transaction ID they have access to (e.g., one they created). No admin credentials, no leaked secrets, and no special role are required. The crafted payload is trivially constructed — a deeply nested JSON object or a large flat array — and fits well within the default NestJS body-size limit (100 KB) while still triggering thousands of recursive calls.

### Recommendation

1. Add `@ArrayMaxSize(N)` to both `approversArray` in `CreateTransactionApproversArrayDto` and `approvers` in `CreateTransactionApproverDto`, where N is a reasonable business limit (e.g., 50–100).
2. Add an explicit maximum recursion depth guard inside `createApprover` (e.g., reject if `depth > 10`).
3. Consider replacing the sequential `await createApprover(...)` loop with a bulk insert after collecting all nodes in a single traversal pass, to reduce DB round-trips regardless of size.

### Proof of Concept

**Flat array attack (10,000 sequential inserts):**
```http
POST /transactions/1/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    { "userId": 2 },
    { "userId": 3 },
    ... (repeat 10,000 times)
  ]
}
```
Each entry triggers one `INSERT INTO transaction_approver` inside a single open DB transaction. The request holds the connection for the full duration, starving other requests.

**Recursive tree attack (stack overflow / exponential inserts):**
```http
POST /transactions/1/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{
      "threshold": 1,
      "approvers": [{
        "threshold": 1,
        "approvers": [{ "userId": 2 }, { "userId": 3 }]
      }, {
        "threshold": 1,
        "approvers": [{ "userId": 4 }, { "userId": 5 }]
      }]
    }, ...]
  }]
}
```
A tree of depth 14 with branching factor 2 fits in ~20 KB of JSON but triggers 2^14 = 16,384 recursive `createApprover` calls and DB inserts. Increasing depth to ~13,000 causes a Node.js stack overflow, crashing the request handler.
