Audit Report

## Title
Transaction Creator Can Delete Approvers From Any Transaction They Do Not Own

## Summary
The `DELETE /:id` handler in `ApproversController` verifies only that the requesting user is the creator of the `transactionId` supplied in the URL path, then unconditionally deletes the approver identified by the separate `id` path parameter. No check is made that the targeted approver actually belongs to the supplied `transactionId`. Any authenticated, verified user who has created at least one transaction can exploit this mismatch to silently remove approvers from transactions they do not own.

## Finding Description

**Root cause — missing cross-parameter ownership check in the DELETE handler**

In `approvers.controller.ts`, the `DELETE /:id` handler performs two independent operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // only checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no relation to transactionId
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction identified by `transactionId`. It says nothing about whether the approver identified by `id` belongs to that transaction. [2](#0-1) 

`removeTransactionApprover` fetches the approver by its own primary key and soft-deletes it (and its entire subtree via `removeNode`) with no check that `approver.transactionId === transactionId`: [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-check by fetching the root node of the approver and verifying `rootNode.transactionId !== transactionId` before proceeding: [4](#0-3) 

The fix pattern already exists in the codebase for `PATCH` but was never applied to `DELETE`.

## Impact Explanation
An attacker can unilaterally remove any approver (or entire approver subtree, because `removeNode` is recursive) from any transaction in the organization, regardless of who created it. This directly undermines the multi-signature approval workflow: required approvers can be stripped from a pending transaction, allowing it to proceed without the mandated signatures, or permanently disrupting the approval tree of a transaction that requires a threshold of approvers. In an organization relying on approval trees for financial or administrative Hedera transactions, this constitutes unauthorized state mutation with direct operational impact.

## Likelihood Explanation
The only precondition is that the attacker is an authenticated, verified organization user — the lowest privilege level in the system. Creating a transaction (to obtain a `transactionId` they own) is a standard product flow. Approver IDs are sequential integers discoverable by observing any transaction the attacker legitimately participates in (via the `GET /transactions/:transactionId/approvers` endpoint). No admin credentials, leaked secrets, or special access are required.

## Recommendation
In the `removeTransactionApprover` controller handler, after fetching the approver, verify that the approver's root node belongs to the same `transactionId` that was ownership-checked. The existing `getRootNodeFromNode` + `transactionId` comparison pattern used in `updateTransactionApprover` should be applied:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);

  // Add: verify the approver belongs to the same transaction
  const rootNode = await this.approversService.getRootNodeFromNode(id);
  if (!rootNode || rootNode.transactionId !== transactionId) {
    throw new UnauthorizedException('Approver does not belong to this transaction');
  }

  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

Alternatively, the check can be encapsulated inside `removeTransactionApprover` by accepting `transactionId` as a parameter and asserting the relationship before deletion.

## Proof of Concept

1. Attacker (normal authenticated user) creates transaction `T_own` — they are now its creator.
2. Attacker observes any transaction `T_victim` they legitimately participate in (e.g., as a signer or observer) and reads the numeric `approver_id` of one of its approvers from `GET /transactions/{T_victim}/approvers`.
3. Attacker sends:
   ```
   DELETE /transactions/{T_own}/approvers/{approver_id}
   ```
4. `getCreatorsTransaction(T_own, attacker)` passes — attacker owns `T_own`.
5. `removeTransactionApprover(approver_id)` executes — the approver (and its entire subtree via `removeNode`) from `T_victim` is soft-deleted with no further check.
6. `T_victim`'s approval tree is now permanently mutated; if the deleted approver was required for threshold satisfaction, the transaction's approval workflow is broken. [5](#0-4)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
