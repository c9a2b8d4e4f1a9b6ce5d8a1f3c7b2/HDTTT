Audit Report

## Title
Missing User Key Ownership Validation in `importSignatures` Allows Any Authorized User to Inject Foreign Signatures

## Summary
The `importSignatures` endpoint (`POST /transactions/signatures/import`) validates cryptographic correctness of submitted signatures but does not verify that the public keys in the signature map belong to the authenticated user. The parallel `uploadSignatureMaps` endpoint enforces this ownership check. Any user with read access to a transaction (including observers) can inject cryptographically valid signatures from keys they do not own, advancing the transaction status without proper authorization and leaving no audit trail.

## Finding Description

**Root cause — missing ownership check in `importSignatures`:**

`importSignatures` in `back-end/apps/api/src/transactions/transactions.service.ts` performs exactly three checks before accepting a signature map:

1. `verifyAccess` — confirms the user has any access to the transaction (creator, observer, signer, approver, or has keys to sign). [1](#0-0) 

2. Status check — transaction must be `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. [2](#0-1) 

3. `validateSignature` — confirms the signature bytes are cryptographically valid for the given public key. [3](#0-2) 

What is **absent** is any check that the public keys in the submitted signature map belong to the authenticated user. The function accepts any cryptographically valid `(publicKey, signature)` pair regardless of who owns that key.

**Contrast with `uploadSignatureMaps` in `signers.service.ts`:**

The normal signing path builds a `userKeyMap` from `user.keys`: [4](#0-3) 

Inside `processTransactionSignatures`, it throws `ErrorCodes.PNY` ("Public key not yours") if a submitted public key is not found in that map: [5](#0-4) 

`importSignatures` has no equivalent guard.

**Additional gap — no `TransactionSigner` record is created:**

`uploadSignatureMaps` persists a `TransactionSigner` row for every accepted key, creating an auditable record of who signed: [6](#0-5) 

`importSignatures` only updates `transactionBytes` directly via a raw SQL `CASE` statement and emits a status-update notification — no signer record is written: [7](#0-6) 

**`verifyAccess` explicitly grants access to observers:** [8](#0-7) 

## Impact Explanation

- **Unauthorized status advancement:** An observer (read-only role) can push a transaction from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` by injecting signatures from keys they do not own, bypassing the intended multi-party signing workflow.
- **Audit trail corruption:** Because no `TransactionSigner` rows are written by `importSignatures`, the system cannot determine which user submitted which signature. Compliance and forensic capabilities are broken for this code path.
- **Integrity of the signing model:** The invariant that "only the owner of a key may add that key's signature to a transaction" is violated on this code path, while being correctly enforced on the `uploadSignatureMaps` path.

## Likelihood Explanation

The attacker must be an authenticated user with any access to the target transaction (observer is sufficient, as confirmed by `verifyAccess` at line 805). They must also possess a cryptographically valid signature for the specific transaction body from a key they do not own. This second requirement is realistic because transaction files containing signatures are explicitly designed to be shared externally — the front-end `importSignatures` flow exists for this purpose. An observer who receives such a file can replay its signatures via this endpoint. No privileged access, no secret leakage, and no cryptographic break are required.

## Recommendation

1. **Add ownership validation to `importSignatures`:** Before calling `sdkTransaction.addSignature`, build a `userKeyMap` from `user.keys` (as done in `processTransactionSignatures` in `signers.service.ts`) and verify that each public key in the submitted signature map is present in that map. Throw an appropriate error (e.g., `ErrorCodes.PNY`) if any key is not owned by the authenticated user.

2. **Create `TransactionSigner` records:** After successfully merging signatures, persist a `TransactionSigner` row for each accepted key (linking `userId`, `transactionId`, and `userKeyId`), mirroring the behavior in `persistSignatureChanges` in `signers.service.ts`.

3. **Consider restricting the caller role:** Evaluate whether observers should be permitted to call `importSignatures` at all, or whether the endpoint should be restricted to users who have keys required to sign the transaction.

## Proof of Concept

1. User A (attacker) is added as an observer on a multi-signature transaction in `WAITING_FOR_SIGNATURES` status.
2. User B (legitimate signer) signs the transaction offline and shares the resulting transaction file (a supported workflow via the front-end `importSignatures` flow).
3. User A extracts the `SignatureMap` from the file, which contains cryptographically valid signatures from User B's keys.
4. User A calls `POST /transactions/signatures/import` with the extracted signature map.
5. `verifyAccess` passes (User A is an observer — line 805 of `transactions.service.ts`).
6. Status check passes (transaction is `WAITING_FOR_SIGNATURES`).
7. `validateSignature` passes (the signatures are cryptographically valid for the transaction body bytes).
8. The foreign signatures are merged into `transactionBytes` via `sdkTransaction.addSignature` (line 552) and persisted via the raw SQL `CASE` update (lines 584–601).
9. The transaction may advance to `WAITING_FOR_EXECUTION`.
10. No `TransactionSigner` record is created, so the audit log shows no signer for those keys. [9](#0-8)

  async importSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
  ): Promise<SignatureImportResultDto[]> {
    type UpdateRecord = {
      id: number;
      transactionBytes: Buffer;
      transactionId: string;
      network: string;
    };

    const ids = dto.map(d => d.id);

    // Single batch query for all transactions
    const transactions = await this.entityManager.find(Transaction, {
      where: { id: In(ids) },
      relations: ['creatorKey', 'approvers', 'signers', 'observers'],
    });

    if (transactions.length === 0) {
      return ids.map(id => ({
        id,
        error: new BadRequestException(ErrorCodes.TNF).message,
      }));
    }

    // Create a map for quick lookup
    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    const results = new Map<number, SignatureImportResultDto>();
    const updates = new Map<number, UpdateRecord>();

    for (const { id, signatureMap: map } of dto) {
      const transaction = transactionMap.get(id);

      try {
        /* Verify that the transaction exists and access is verified */
        if (!(await this.verifyAccess(transaction, user))) {
          throw new BadRequestException(ErrorCodes.TNF);
        }

        /* Checks if the transaction is canceled */
        if (
          transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
          transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
        )
          throw new BadRequestException(ErrorCodes.TNRS);

        /* Checks if the transaction is expired */
        const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
        if (isExpired(sdkTransaction)) throw new BadRequestException(ErrorCodes.TE);

        /* Validates the signatures */
        const { data: publicKeys, error } = safe<PublicKey[]>(
          validateSignature.bind(this, sdkTransaction, map),
        );
        if (error) throw new BadRequestException(ErrorCodes.ISNMPN);

        for (const publicKey of publicKeys) {
          sdkTransaction.addSignature(publicKey, map);
        }

        transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());

        results.set(id, { id });
        updates.set(id, {
          id,
          transactionBytes: transaction.transactionBytes,
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        });
      } catch (error) {
        results.set(id, {
          id,
          error:
            (error instanceof BadRequestException)
              ? error.message
              : 'An unexpected error occurred while importing the signatures',
        });
      }
    }

    //Added a batch mechanism, probably should limit this on the api side of things
    const BATCH_SIZE = 500;

    const updateArray = Array.from(updates.values());

    if (updateArray.length > 0) {
      for (let i = 0; i < updateArray.length; i += BATCH_SIZE) {
        const batch = updateArray.slice(i, i + BATCH_SIZE);

        let caseSQL = 'CASE id ';
        const params: any = {};

        batch.forEach((update, idx) => {
          caseSQL += `WHEN :id${idx} THEN :bytes${idx}::bytea `;
          params[`id${idx}`] = update.id;
          params[`bytes${idx}`] = update.transactionBytes;
        });
        caseSQL += 'END';

        try {
          await this.entityManager
            .createQueryBuilder()
            .update(Transaction)
            .set({ transactionBytes: () => caseSQL })
            .where('id IN (:...ids)', { ids: batch.map(u => u.id) })
            .setParameters(params)
            .execute();

          // mark each update in the batch as succeeded
          batch.forEach(u => results.set(u.id, { id: u.id }));
        } catch (err) {
          const SAVE_ERROR_PREFIX = 'An unexpected error occurred while saving the signatures';
          const message =
            err instanceof Error && err.message
              ? `${SAVE_ERROR_PREFIX}: ${err.message}`
              : SAVE_ERROR_PREFIX;

          batch.forEach(u => results.set(u.id, { id: u.id, error: message }));
        }
      }

      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        updateArray.map(r => ({
          entityId: r.id,
          additionalData: { transactionId: r.transactionId, network: r.network },
        })),
      );
    }

    return Array.from(results.values());
  }
