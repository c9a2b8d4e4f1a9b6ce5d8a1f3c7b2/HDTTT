### Title
`importSignatures` Accepts Signatures from Keys Not Belonging to the Authenticated User, Enabling Unauthorized Transaction Advancement

### Summary
The `importSignatures` endpoint in `TransactionsService` validates submitted signatures only for cryptographic correctness but never verifies that the public keys in the signature map are registered to the authenticated user. A parallel endpoint, `uploadSignatureMaps` in `SignersService`, correctly enforces key ownership. This inconsistency allows any user with access to a transaction (including observers and approvers who have no signing authority) to inject cryptographically valid signatures from keys they do not own, potentially advancing the transaction to `WAITING_FOR_EXECUTION` and triggering unauthorized execution.

### Finding Description

There are two distinct code paths for submitting signatures to a transaction:

**Path 1 — `SignersService.uploadSignatureMaps` (enforces key ownership):**

In `validateAndProcessSignatures`, a `userKeyMap` is built from `user.keys` (the authenticated user's registered keys). For every public key in the submitted signature map, the code checks:

```typescript
let userKey = userKeyMap.get(raw);
if (!userKey) {
  userKey = userKeyMap.get(publicKey.toStringDer());
}
if (!userKey) throw new Error(ErrorCodes.PNY);
```

If the key is not registered to the user, the request is rejected with `PNY` ("Public key Not Yours"). [1](#0-0) 

**Path 2 — `TransactionsService.importSignatures` (no key ownership check):**

The function calls `verifyAccess` (which grants access to creators, observers, signers, and approvers) and then calls `validateSignature`, which only verifies cryptographic validity:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);

for (const publicKey of publicKeys) {
  sdkTransaction.addSignature(publicKey, map);
}
```

There is no check that any of the `publicKeys` returned by `validateSignature` belong to the authenticated user. [2](#0-1) 

The `validateSignature` function itself only verifies that `publicKey.verify(bodyBytes, signature)` returns true — it is purely a cryptographic check with no identity binding: [3](#0-2) 

After the signatures are injected into `transactionBytes` and persisted, `emitTransactionStatusUpdate` is fired. The chain service then re-evaluates the transaction status via `processTransactionStatus`, which calls `computeSignatureKey` and `hasValidSignatureKey`. If the injected signatures satisfy the required key list, the transaction is promoted to `WAITING_FOR_EXECUTION` and scheduled for execution: [4](#0-3) 

### Impact Explanation

A malicious user who has access to a transaction as an **observer** or **approver** (roles that carry no signing authority) can obtain a cryptographically valid signature for a required key (e.g., from a signed transaction file shared in the external signing workflow) and submit it via `POST /transactions/signatures/import`. The server will accept it, update `transactionBytes`, and re-evaluate the transaction status. If the injected signature satisfies the threshold, the transaction advances to `WAITING_FOR_EXECUTION` and is executed on the Hedera network — without the legitimate key holder having authorized the action through the system's key-ownership model.

This breaks the integrity of the multi-signature orchestration model: the system's authorization invariant is that only users whose registered keys are required to sign can contribute signatures. `importSignatures` violates this invariant.

### Likelihood Explanation

The attacker must:
1. Have access to the target transaction (as observer, approver, past signer, or creator) — a realistic role in organizational workflows.
2. Possess a valid signature for a required key. This is realistic because the external signing flow (`SignTransactionFileModal.vue`) produces signed transaction files that are shared between parties. A malicious observer receiving such a file has everything needed.

No privileged access, no cryptographic break, and no leaked secrets are required. The attack is fully reachable via the documented API endpoint `POST /transactions/signatures/import`. [5](#0-4) 

### Recommendation

Add a key-ownership check inside `importSignatures`, mirroring the logic in `SignersService.processTransactionSignatures`. After `validateSignature` returns the list of public keys, verify that each key is registered to the authenticated user:

```typescript
await attachKeys(user, this.entityManager);
const userKeyMap = new Map(user.keys.map(k => [k.publicKey, k]));

for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  if (!userKeyMap.has(raw) && !userKeyMap.has(publicKey.toStringDer())) {
    throw new BadRequestException(ErrorCodes.PNY);
  }
  sdkTransaction.addSignature(publicKey, map);
}
```

This synchronizes the authorization model between the two signature submission paths. [6](#0-5) 

### Proof of Concept

**Setup:**
- User A is the creator of a transaction requiring key K (owned by User B).
- User C is an observer of the transaction.
- User B signs the transaction offline and shares the signed transaction file with User C (a normal workflow step).

**Attack:**
1. User C extracts the signature map from the signed transaction file (the file contains the raw bytes of the signed transaction; `SignatureMap._fromTransaction` is used client-side to extract it).
2. User C calls `POST /transactions/signatures/import` with:
   ```json
   [{ "id": <transaction_id>, "signatureMap": <extracted_map_with_key_B_signature> }]
   ```
3. The server calls `verifyAccess` — User C is an observer, so access is granted.
4. `validateSignature` verifies the signature is cryptographically valid for key K — it is, because User B actually signed it.
5. The signature is added to `transactionBytes` and persisted.
6. `emitTransactionStatusUpdate` fires; the chain service sees the transaction now has sufficient signatures and promotes it to `WAITING_FOR_EXECUTION`.
7. The transaction is executed on the Hedera network, driven by User C's API call, not by User B's explicit authorization through the system.

**Expected result without the bug:** The server should reject the request because key K is not registered to User C.

**Actual result:** The server accepts the signature and advances the transaction. [7](#0-6)

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
