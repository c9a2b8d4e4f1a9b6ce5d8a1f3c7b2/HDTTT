### Title
Missing Transaction-Level Authorization on Comment Endpoints Allows Any Verified User to Read and Write Comments on Arbitrary Transactions

### Summary
The `CommentsController` enforces only authentication (`JwtAuthGuard`) and account verification (`VerifiedUserGuard`) on its `POST` and `GET` endpoints, but performs no check that the requesting user has any relationship to the target transaction (creator, observer, signer, or approver). The codebase itself acknowledges this gap with an explicit `//TODO` comment. Any registered, verified user can post comments on — and read all comments from — any transaction in the system, including private ones they have no legitimate access to. This is the direct analog of the external report's vulnerability class: an authorization check that verifies only a surface-level identity attribute (authentication) rather than the caller's actual relationship to the protected resource.

---

### Finding Description

**Root cause — missing transaction-level access check in `CommentsController`**

The controller applies three guards at the class level:

```
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
```

`VerifiedUserGuard.canActivate` returns `user.status === UserStatus.NONE`, i.e., it passes for any user who has completed account setup. It does not inspect which transaction is being targeted. [1](#0-0) 

The `createComment` handler passes the caller's user object and the attacker-supplied `transactionId` directly to the service with no further check. The developer left an explicit acknowledgement of the missing control:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
``` [2](#0-1) 

`CommentsService.createComment` performs no access check; it creates and persists the comment unconditionally: [3](#0-2) 

The `GET` handler for listing comments is equally unguarded — it accepts a `transactionId` and returns all comments for it without verifying the caller's relationship to that transaction: [4](#0-3) [5](#0-4) 

**Contrast with the correct pattern used elsewhere**

Every other sensitive transaction endpoint calls `getTransactionWithVerifiedAccess`, which internally calls `verifyAccess`. That method checks whether the user is the creator, a signer, an observer, or an approver before granting access: [6](#0-5) [7](#0-6) 

The comments subsystem is the only area that skips this check entirely.

---

### Impact Explanation

1. **Unauthorized write — comment injection on private transactions.** A verified user with no relationship to a transaction can `POST /transactions/{id}/comments` with any `transactionId`. The comment is persisted and associated with that transaction. This pollutes the audit trail and can be used to inject misleading information into transactions belonging to other users.

2. **Unauthorized read — information disclosure.** A verified user can `GET /transactions/{id}/comments` for any `transactionId` and retrieve all comments ever posted on that transaction. Comments may contain sensitive operational details (e.g., signing instructions, key identifiers, business context) that the transaction's access control model was designed to keep private.

Both impacts are concrete state changes / data exposures with no cryptographic barrier.

---

### Likelihood Explanation

**Attacker precondition:** Must hold a valid JWT for a verified account (`UserStatus.NONE`). Account creation requires an admin to call `POST /auth/signup`, so the attacker must be a registered member of the organization — a realistic insider-threat or compromised-account scenario. [8](#0-7) 

**Exploit steps:** Once authenticated, the attacker needs only to enumerate integer `transactionId` values (sequential primary keys, trivially guessable) and issue standard HTTP requests. No special tooling, no privilege escalation, no cryptographic attack required.

**Likelihood: High.** The vulnerability is reachable by any registered user with a single HTTP request, and the codebase's own TODO comment confirms the gap was known but unresolved.

---

### Recommendation

Before executing `createComment` or `getTransactionComments`, verify the requesting user has an established relationship to the transaction using the existing `verifyAccess` / `getTransactionWithVerifiedAccess` pattern:

```typescript
// In CommentsService.createComment — add before persisting:
const transaction = await this.transactionsService.getTransactionWithVerifiedAccess(
  transactionId,
  user,
);
// throws UnauthorizedException if user has no access
```

Apply the same check in `getTransactionComments` by accepting the `user` parameter and delegating to `verifyAccess`. This mirrors the pattern already enforced in `TransactionsController`, `ApproversService.getVerifiedApproversByTransactionId`, and all other transaction-scoped endpoints. [9](#0-8) 

---

### Proof of Concept

**Setup:** Two users exist — `alice` (creator of transaction ID 42, a private transaction) and `bob` (no relationship to transaction 42).

**Step 1 — Bob authenticates:**
```
POST /auth/login
{ "email": "bob@org.com", "password": "..." }
→ 200 { "accessToken": "<BOB_JWT>" }
```

**Step 2 — Bob reads Alice's private transaction comments (unauthorized read):**
```
GET /transactions/42/comments
Authorization: Bearer <BOB_JWT>
→ 200 [ { "message": "Signing key is 0xABCD...", ... } ]
```
Bob receives all comments on a transaction he has no access to. `GET /transactions/42` with the same token would return `403 Unauthorized` via `getTransactionWithVerifiedAccess`, but the comments endpoint has no equivalent check.

**Step 3 — Bob injects a comment (unauthorized write):**
```
POST /transactions/42/comments
Authorization: Bearer <BOB_JWT>
Content-Type: application/json
{ "message": "Injected comment by unauthorized user" }
→ 201 { "id": 99, "message": "Injected comment by unauthorized user", ... }
```
The comment is persisted and associated with Alice's transaction. Alice and other participants will see it as if it came from a legitimate participant.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
