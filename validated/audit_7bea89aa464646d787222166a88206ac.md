## Title
Unauthorized Deletion of Transaction Approvers via IDOR in `removeTransactionApprover`

## Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of the transaction identified by the URL's `:transactionId` parameter, but then deletes the approver identified by `:id` **without verifying that approver belongs to that transaction**. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

The vulnerability class from the GoGoPool report is **authorization bypass via missing ownership cross-check on a resource operation**. In GoGoPool, `createMinipool` verified a state transition was valid but never checked that the caller was the original owner. Here, `removeTransactionApprover` verifies the caller is the creator of *some* transaction, but never checks that the approver being deleted belongs to *that* transaction.

**Root cause — controller:** [1](#0-0) 

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ✓ checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ✗ deletes approver `id` with no cross-check
  return true;
}
```

**Root cause — service:** [2](#0-1) 

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  // approver.transactionId is NEVER compared to the URL's transactionId
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

The notification even emits against `approver.transactionId` (the victim's transaction), not the attacker's `transactionId`, confirming the two are never reconciled.

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-check: [3](#0-2) 

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

/* Verifies that the root transaction is the same as the param */
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
```

The `update` path has this guard; the `delete` path does not.

**Attack path:**

1. Attacker creates any transaction (Transaction A) — they are now its creator.
2. Attacker enumerates or guesses the approver `id` belonging to Transaction B (owned by a victim). Approver IDs are sequential integers.
3. Attacker calls `DELETE /transactions/{A}/approvers/{approver_id_from_B}`.
4. `getCreatorsTransaction(A, attacker)` passes — attacker owns A.
5. `removeTransactionApprover(approver_id_from_B)` executes — the approver (and its entire subtree via `removeNode`) is soft-deleted from Transaction B.

## Impact Explanation

An attacker can silently remove any approver (or entire approval subtrees) from any transaction they do not own. This undermines the multi-signature approval workflow that is the core security guarantee of the platform. A transaction that required N-of-M approvals can have its approvers stripped, potentially allowing it to proceed to execution without the intended governance controls. [4](#0-3) 

The `removeNode` function recursively soft-deletes the entire approver subtree, so a single call can wipe out a complex threshold approval tree.

## Likelihood Explanation

- Requires only a valid authenticated session and having created at least one transaction (a low bar for any organization member).
- Approver IDs are sequential integers, making enumeration trivial.
- No rate limiting or anomaly detection is described for this endpoint.
- The attack is silent — the victim transaction's creator receives a status update notification but may not immediately notice an approver was removed.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), verify that the approver's root transaction matches the URL's `transactionId`, mirroring the guard already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId) {
    throw new UnauthorizedException('Approver does not belong to the specified transaction');
  }

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
}
``` [1](#0-0) 

## Proof of Concept

```
# Step 1: Attacker creates Transaction A (attacker is creator)
POST /transactions
{ ...valid transaction payload... }
→ { id: 100 }

# Step 2: Victim creates Transaction B with an approver (approver id = 42)
POST /transactions/200/approvers
{ approversArray: [{ userId: 5 }] }
→ [{ id: 42, transactionId: 200, userId: 5 }]

# Step 3: Attacker deletes victim's approver using their own transactionId in the URL
DELETE /transactions/100/approvers/42
Authorization: Bearer <attacker_token>

→ HTTP 200 true
# Approver id=42 (belonging to transaction 200) is now soft-deleted.
# Transaction 200's approval requirement is silently bypassed.
```

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

    // notifyTransactionAction(this.notificationsService);
  }
