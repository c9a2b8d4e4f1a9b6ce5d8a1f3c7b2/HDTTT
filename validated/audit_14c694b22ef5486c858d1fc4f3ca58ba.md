### Title
Missing Unique Database Constraint on `transaction_signer` Enables Duplicate Signer Insertion via Concurrent Requests (TOCTOU)

### Summary
The `transaction_signer` table has no unique constraint on `(transactionId, userKeyId)`. The application-level duplicate-signer check in `loadTransactionData` / `processTransactionSignatures` is a read-then-write pattern with no atomicity guarantee. Two concurrent `POST /transactions/:id/signers` requests from the same authenticated user can both pass the in-memory check and both insert duplicate `TransactionSigner` rows, corrupting the signer ledger for a transaction.

### Finding Description

**Root cause — no database-level uniqueness enforcement**

The `TransactionSigner` entity declares only a non-unique index:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← performance index, NOT unique
export class TransactionSigner { … }
``` [1](#0-0) 

The migration confirms no `UNIQUE` constraint exists on the table:

```sql
CREATE TABLE IF NOT EXISTS "transaction_signer" (
  "id" SERIAL NOT NULL, "transactionId" integer NOT NULL,
  "userKeyId" integer NOT NULL, "userId" integer NOT NULL, …
  CONSTRAINT "PK_…" PRIMARY KEY ("id")
)
``` [2](#0-1) 

**TOCTOU race in the application-level guard**

`uploadSignatureMaps` follows a read → validate → write pattern split across three separate async steps:

1. **Read** — `loadTransactionData` fetches existing signers at time T₁ and builds `signersByTransaction`. [3](#0-2) 

2. **Validate** — `processTransactionSignatures` uses the snapshot to decide which keys are "new":
   ```typescript
   if (!existingSignerIds.has(userKey.id)) {
     userKeys.push(userKey);
   }
   ``` [4](#0-3) 

3. **Write** — `bulkInsertSigners` inserts without any conflict-resolution clause:
   ```typescript
   await manager.createQueryBuilder()
     .insert().into(TransactionSigner).values(signersToInsert)
     .returning('*').execute();
   ``` [5](#0-4) 

Because steps 1–3 are not wrapped in a single serializable database transaction with a unique constraint, two concurrent requests that both read the same empty `signersByTransaction` snapshot will both classify the same key as "new" and both successfully insert a `TransactionSigner` row for the same `(transactionId, userKeyId)` pair.

**Exploit path**

```
Attacker (authenticated user, owns key K):
  Request A ──► loadTransactionData  (sees: no signer for K)
  Request B ──► loadTransactionData  (sees: no signer for K)
  Request A ──► processSignatures    (K is "new")
  Request B ──► processSignatures    (K is "new")
  Request A ──► bulkInsertSigners    (inserts row: txId=X, userKeyId=K) ✓
  Request B ──► bulkInsertSigners    (inserts row: txId=X, userKeyId=K) ✓  ← duplicate
```

### Impact Explanation

- **Duplicate `TransactionSigner` rows** for the same `(transactionId, userKeyId)` pair are persisted permanently.
- Any downstream logic that counts `TransactionSigner` records to evaluate whether a signing threshold is met (e.g., `processTransactionStatus`) will see an inflated count, potentially advancing a transaction to `WAITING_FOR_EXECUTION` before all *distinct* required signers have actually signed.
- Notification dismissal (`bulkUpdateNotificationReceivers`) is triggered for each duplicate, causing spurious notification state changes.
- The corrupted signer ledger is visible to all organization members and cannot be self-corrected by normal users.

### Likelihood Explanation

Any authenticated user who owns at least one key registered on a transaction can trigger this by issuing two HTTP POST requests to `POST /transactions/:id/signers` simultaneously (e.g., with `Promise.all` or two browser tabs). No privileged access, no leaked credentials, and no special network position are required. The race window is the entire async processing pipeline (~tens of milliseconds), which is wide enough to be reliably hit.

### Recommendation

Apply both layers of defense:

1. **Database constraint** — add a unique constraint at the schema level:
   ```typescript
   @Entity()
   @Index(['transactionId', 'userKeyId'], { unique: true })
   export class TransactionSigner { … }
   ```
   And in the migration:
   ```sql
   ALTER TABLE transaction_signer
     ADD CONSTRAINT UQ_signer_tx_key UNIQUE ("transactionId", "userKeyId");
   ```

2. **Conflict-safe insert** — use `ON CONFLICT DO NOTHING` (or TypeORM's `orIgnore()`) in `bulkInsertSigners` so concurrent inserts are idempotent rather than error-throwing:
   ```typescript
   await manager.createQueryBuilder()
     .insert().into(TransactionSigner).values(signersToInsert)
     .orIgnore()          // ← idempotent on duplicate
     .returning('*').execute();
   ```

### Proof of Concept

```typescript
// Attacker is an authenticated user with key K registered on transaction TX_ID

// Fire two concurrent requests

  fetch('/transactions/signers', { method: 'POST', body: JSON.stringify(payload), headers }),
  fetch('/transactions/signers', { method: 'POST', body: JSON.stringify(payload), headers }),
]);

// Both return HTTP 201
// DB now contains two rows: { transactionId: TX_ID, userKeyId: K_ID }
// SELECT COUNT(*) FROM transaction_signer WHERE "transactionId"=TX_ID AND "userKeyId"=K_ID

```

Expected outcome: two `TransactionSigner` rows with identical `(transactionId, userKeyId)` exist in the database, inflating the signer count and potentially triggering premature status transitions.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
