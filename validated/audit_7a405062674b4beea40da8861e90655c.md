Audit Report

## Title
Unbounded Recursive Approver Tree Enables Authenticated DoS via Excessive Database Operations

## Summary
The `POST /transactions/:id/approvers` endpoint accepts an arbitrarily large and deeply nested approver tree with no depth or breadth limit. The server-side handler processes this tree with a recursive async function that issues multiple sequential database queries per node, all inside a single open database transaction. Any verified user who has created a transaction can exploit this to exhaust the database connection pool and degrade server availability.

## Finding Description

**`CreateTransactionApproverDto` and `CreateTransactionApproversArrayDto` impose no upper bound on array size.**

`CreateTransactionApproverDto.approvers` carries only `@ArrayMinSize(1)` — no `@ArrayMaxSize`. `CreateTransactionApproversArrayDto.approversArray` carries no size constraint at all. A repository-wide grep for `ArrayMaxSize` returns zero matches. [1](#0-0) 

**The service method `createTransactionApprovers` defines an inner recursive async function `createApprover` that issues multiple sequential DB queries per node, all inside a single `dataSource.transaction(...)` call.**

For every node in the submitted tree, the following operations are performed sequentially:

1. `isNode(...)` — a `count` query against `transaction_approver`
2. `transactionalEntityManager.findOne(TransactionApprover, ...)` — parent lookup (when `listId` is set)
3. `getRootNodeFromNode(...)` — a recursive SQL CTE traversal query (when `listId` is set)
4. `transactionalEntityManager.count(User, ...)` — user existence check (when `userId` is set)
5. `getApproversByTransactionId(...)` — another recursive SQL CTE query (when `userId` is set)
6. `transactionalEntityManager.insert(TransactionApprover, ...)` — the actual write

Then it recurses into every child node. [2](#0-1) 

The `isNode` helper issues a `count` query per call: [3](#0-2) 

`getRootNodeFromNode` and `getApproversByTransactionId` each execute a recursive CTE against `transaction_approver`: [4](#0-3) [5](#0-4) 

**The same missing limit exists on the observer endpoint.** `CreateTransactionObserversDto.userIds` has `@ArrayMinSize(1)` but no `@ArrayMaxSize`, so a creator can add thousands of observers in one call, each triggering a DB insert and a NATS notification event. [6](#0-5) 

**The database connection pool is configured to 2–3 connections in all deployment configurations:** [7](#0-6) [8](#0-7) 

## Impact Explanation

A single malicious `POST /transactions/:id/approvers` request carrying a wide flat tree (e.g., `approversArray` with a root threshold node containing thousands of leaf `userId` entries) or a deeply nested chain triggers O(N) to O(N²) database round-trips within a single open database transaction, holding one of the 2–3 pooled PostgreSQL connections for the entire duration. With a small number of concurrent such requests, the connection pool is exhausted, causing all other API operations — including legitimate transaction signing and execution — to queue or fail with connection timeout errors. This constitutes a server-wide denial of service.

## Likelihood Explanation

Any registered and verified user can create a transaction and immediately become its creator, granting them the right to call `POST /transactions/:id/approvers`. The user-level throttler allows 100 requests per minute and 10 per second — sufficient to sustain the attack, since the damage is done within a single large request, not through request volume. [9](#0-8) 

The attack payload is trivially constructed as a JSON object. No admin privilege is required.

## Recommendation

1. **Add `@ArrayMaxSize(N)` to both `CreateTransactionApproverDto.approvers` and `CreateTransactionApproversArrayDto.approversArray`** with a reasonable bound (e.g., 50–100 entries per level).
2. **Add a maximum recursion depth check** at the start of `createApprover` (e.g., reject trees deeper than 5–10 levels).
3. **Add `@ArrayMaxSize(N)` to `CreateTransactionObserversDto.userIds`** (e.g., 100).
4. Consider pre-validating the total node count of the submitted tree before opening the database transaction.

## Proof of Concept

```http
POST /transactions/1/approvers
Authorization: Bearer <valid_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1000,
      "approvers": [
        { "userId": 1 },
        { "userId": 2 },
        ...
        { "userId": 1000 }
      ]
    }
  ]
}
```

This single request causes `createApprover` to be called 1001 times (1 root + 1000 leaves). Each leaf node triggers at minimum 3 sequential DB queries (`isNode` count, `count(User)`, `getApproversByTransactionId` CTE) plus 1 insert, totalling ~4000 DB operations inside one open transaction. With `POSTGRES_MAX_POOL_SIZE=3`, two concurrent such requests fully exhaust the connection pool.
