Audit Report

## Title
Any Verified User Can Enumerate All Organization User Keys via Unguarded `GET /user-keys` Endpoint

## Summary
`UserKeysAllController` exposes `GET /user-keys` protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. Any authenticated, verified organization member can paginate through every user's `UserKey` records — including `mnemonicHash` and `index` — fields that the ownership-aware sibling endpoint deliberately withholds from non-owners.

## Finding Description

**Missing `AdminGuard` on `UserKeysAllController`:**

The controller is decorated with only three guards, with no `AdminGuard`:

```ts
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class UserKeysAllController {
``` [1](#0-0) 

The single route handler calls `getUserKeys(paginationParams)` with no user context passed: [2](#0-1) 

**Service layer — no user filter, no field restriction:**

`UserKeysService.getUserKeys` issues an unrestricted `findAndCount` with no `where` clause and no `select` projection, returning every `UserKey` row in the database:

```ts
const [items, total] = await this.repo.findAndCount({
  take: limit,
  skip: offset,
});
``` [3](#0-2) 

**`UserKeyDto` exposes sensitive fields:**

`UserKeyDto` marks both `mnemonicHash` and `index` with `@Expose()`. The `SerializeInterceptor` uses `plainToInstance` with `excludeExtraneousValues: true`, meaning only `@Expose()`-decorated fields are emitted — and both sensitive fields pass through: [4](#0-3) [5](#0-4) 

**Contrast with the ownership-aware path:**

`getUserKeysRestricted`, used by `UserKeysController`, explicitly gates `mnemonicHash` and `index` behind an ownership check:

```ts
mnemonicHash: user.id === userId,
index: user.id === userId,
``` [6](#0-5) 

`UserKeysAllController` bypasses this entirely by calling `getUserKeys` instead.

**`UserKey` entity fields confirmed:**

`mnemonicHash` stores a hash of the user's HD wallet mnemonic; `index` stores the BIP-32 derivation path index. Both are nullable columns on the entity: [7](#0-6) 

## Impact Explanation

- **Cross-user sensitive data exposure**: Any verified user receives `mnemonicHash` and `index` for every registered key in the organization. `mnemonicHash` is HD wallet derivation metadata; leaking it organization-wide exposes the key infrastructure of all users, not just the attacker's own.
- **Privilege escalation of information**: A normal verified user gains visibility equivalent to an admin over the entire key registry, including fields the ownership-aware endpoint deliberately hides.
- **Hedera multi-sig exposure**: Public keys identify signers in multi-sig workflows; the full enumeration reveals the organization's signing topology.

## Likelihood Explanation

- **Precondition**: Attacker must be a verified organization member (requires an admin invite). This is a realistic insider-threat or compromised-account scenario.
- **Attack complexity**: A single HTTP GET with a valid JWT. No special tooling required.
- **Discoverability**: The endpoint is Swagger-documented via `@ApiTags('User Keys All')`, making it trivially discoverable. [8](#0-7) 

## Recommendation

Apply `AdminGuard` to `UserKeysAllController` so only administrators can access the unrestricted cross-user listing:

```ts
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard, AdminGuard)
export class UserKeysAllController { ... }
```

Additionally, add a `select` projection to `getUserKeys` in `UserKeysService` that excludes `mnemonicHash` and `index` (mirroring the pattern in `getUserKeysRestricted`), providing defense-in-depth even if the guard is ever misconfigured. [9](#0-8) 

## Proof of Concept

```
# 1. Obtain a JWT for any verified (non-admin) organization user
TOKEN="<verified_user_jwt>"

# 2. Call the unguarded endpoint
curl -H "Authorization: Bearer $TOKEN" \
     "https://<host>/user-keys?page=1&size=100"

# 3. Response includes mnemonicHash and index for ALL users:
# {
#   "totalItems": N,
#   "items": [
#     { "id": 1, "userId": 2, "mnemonicHash": "abc...", "index": 0, "publicKey": "..." },
#     { "id": 2, "userId": 3, "mnemonicHash": "def...", "index": 1, "publicKey": "..." },
#     ...
#   ]
# }

# 4. Paginate with increasing page numbers to collect all records
```

The `UserKeyDto` serializer confirms `mnemonicHash` and `index` are `@Expose()`-decorated and will appear in every page of results. [10](#0-9)
