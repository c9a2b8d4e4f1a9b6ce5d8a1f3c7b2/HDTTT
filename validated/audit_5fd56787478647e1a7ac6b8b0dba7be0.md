### Title
`removeTransactionApprover` Does Not Update Parent Threshold, Causing Permanent Approval DoS

### Summary
`removeTransactionApprover` in `ApproversService` calls `removeNode` to soft-delete an approver and its descendants, but never adjusts the parent tree node's `threshold` value. If a child approver is deleted from a threshold group, the parent's `threshold` can exceed the remaining number of children, making it mathematically impossible to ever satisfy the approval requirement. The transaction is permanently stuck and can never be executed.

### Finding Description

**Vulnerability class**: State transition / DoS — two coupled state variables (`threshold` and the count of live child approvers) become inconsistent through a reachable user action.

**Root cause**:

`removeTransactionApprover` fetches the target approver and delegates to `removeNode`: [1](#0-0) 

`removeNode` performs a recursive soft-delete of the target node and all its descendants, but **never touches the parent's `threshold` column**: [2](#0-1) 

By contrast, `updateTransactionApprover` (the `dto.listId = null` branch) **does** correctly adjust the parent threshold when a child is detached: [3](#0-2) 

`removeTransactionApprover` has no equivalent adjustment, leaving the parent's `threshold` stale.

**Exploit path**:

1. Creator creates a transaction and adds a threshold approver tree:
   ```
   POST /transactions/1/approvers
   { approversArray: [{ threshold: 2, approvers: [{ userId: A }, { userId: B }] }] }
   ```
   DB state: parent node `threshold=2`, children = [A, B].

2. Creator deletes child A:
   ```
   DELETE /transactions/1/approvers/{A.id}
   ```
   `removeNode(A.id)` soft-deletes A. Parent's `threshold` remains `2`.

3. DB state: parent node `threshold=2`, live children = [B] (count = 1).

4. Only user B can approve, but the system requires `threshold=2` approvals. The approval requirement can never be satisfied. The transaction is permanently stuck.

The controller only verifies the caller is the transaction creator before invoking `removeTransactionApprover`: [4](#0-3) 

No privilege beyond being the transaction creator is required.

### Impact Explanation

The transaction is permanently locked in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status. The approval threshold can never be met because fewer live approvers exist than the threshold demands. There is no recovery path exposed in the API — the creator cannot lower the threshold of the parent node to match the remaining children count because `updateTransactionApprover` with `dto.threshold` validates `dto.threshold <= approver.approvers.length`, but `approver.approvers` is loaded from the DB at query time and will correctly reflect the reduced count — however the creator would need to know to do this as a manual remediation step. If the creator is unaware, the transaction is permanently DoS'd. [5](#0-4) 

### Likelihood Explanation

The trigger is a normal, documented product flow: a transaction creator removes one approver from a threshold group. No attacker-controlled input is needed beyond being the authenticated creator of the transaction. The creator can trigger this accidentally (e.g., removing the wrong approver) or intentionally to sabotage their own transaction. The API endpoint is publicly documented. [6](#0-5) 

### Recommendation

In `removeTransactionApprover`, after calling `removeNode`, fetch the deleted approver's parent (via `listId`) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If the parent's remaining live children count drops to 0, soft-delete the parent.
- If the remaining count is less than the parent's `threshold`, update `threshold` to the remaining count.

This mirrors the existing correct logic at lines 417–428 of `approvers.service.ts`. [3](#0-2) 

### Proof of Concept

**Setup**:
1. Authenticate as user U (transaction creator).
2. `POST /transactions` → creates transaction T (id=1).
3. `POST /transactions/1/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 2, "approvers": [{ "userId": 10 }, { "userId": 11 }] }] }
   ```
   This creates parent node P (`threshold=2`) with children C1 (`userId=10`) and C2 (`userId=11`).

**Trigger**:

4. `DELETE /transactions/1/approvers/{C1.id}` — removes C1.
   - `removeNode(C1.id)` soft-deletes C1.
   - Parent P still has `threshold=2` in DB.
   - Only C2 remains (1 live child).

**Observe**:

5. User 11 calls `POST /transactions/1/approvers/approve` — records their approval.
6. The system evaluates: `threshold=2` required, only 1 approver approved → threshold never satisfied.
7. Transaction is permanently stuck; it can never advance to execution.

**Verify inconsistency**:
```sql
SELECT id, threshold FROM transaction_approver WHERE id = {P.id};
-- threshold = 2

SELECT COUNT(*) FROM transaction_approver 
WHERE "listId" = {P.id} AND "deletedAt" IS NULL;
-- count = 1
-- threshold (2) > live children (1) → unsatisfiable
```

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
