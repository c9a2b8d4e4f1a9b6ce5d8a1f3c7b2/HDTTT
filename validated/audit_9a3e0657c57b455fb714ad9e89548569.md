1. `processTransactionSignatures` in `signers.service.ts` calls `sdkTransaction.addSignature(publicKey, map)` with no prior `publicKey.verify(bodyBytes, signature)` call.

## Audit Report

## Title
Missing Cryptographic Signature Verification in `processTransactionSignatures` Allows Authenticated User to Corrupt Multi-Sig Transaction Bytes

## Summary
The `processTransactionSignatures` method in `signers.service.ts` accepts and stores signature bytes without cryptographically verifying them against the transaction body. An authenticated user can upload an invalid (non-verifiable) signature for their own registered public key, causing the transaction to be recorded as signed and potentially moved to execution, where it will fail on the Hedera network — wasting co-signers' legitimate signatures and potentially expiring the transaction window.

## Finding Description

**Root cause:** `processTransactionSignatures` in `back-end/apps/api/src/transactions/signers/signers.service.ts` (lines 217–266) only checks that the public key in the submitted `SignatureMap` belongs to the authenticated user via a `userKeyMap` lookup, then unconditionally calls `sdkTransaction.addSignature(publicKey, map)` — never calling `publicKey.verify(bodyBytes, signature)` to confirm the signature bytes are cryptographically valid. [1](#0-0) 

Contrast this with `importSignatures` in `transactions.service.ts`, which correctly calls `validateSignature(sdkTransaction, map)` — which internally calls `publicKey.verify(bodyBytes, signature)` — before accepting any signature: [2](#0-1) 

The `validateSignature` utility explicitly verifies each signature cryptographically: [3](#0-2) 

The two code paths diverge at the controller level: `POST /transactions/:id/signers` routes through `signers.service.ts` (no verification), while the `importSignatures` path routes through `transactions.service.ts` (with verification): [4](#0-3) 

Additionally, the skip guard at line 293 only skips processing when **both** `isSameBytes` is true **and** `userKeys.length === 0`. This means even if the Hedera SDK silently ignores an invalid signature (leaving bytes unchanged), the `TransactionSigner` record is still inserted as long as the user key was found in the map — falsely recording the user as a valid signer regardless. [5](#0-4) 

## Impact Explanation
An attacker who is an authenticated user with a registered public key can submit a `SignatureMap` containing their real public key paired with arbitrary/invalid signature bytes. The system will:
1. Accept the submission (key ownership check passes).
2. Embed the invalid signature bytes into the stored `transactionBytes`.
3. Insert a `TransactionSigner` record marking the user as having signed.
4. Potentially advance the transaction status to `WAITING_FOR_EXECUTION` if the threshold is met.
5. When the transaction is submitted to the Hedera network, it will be rejected due to invalid signatures, wasting all co-signers' legitimate signatures and potentially expiring the transaction window.

## Likelihood Explanation
The attack requires only a valid authenticated session and knowledge of one's own registered public key — both trivially available to any enrolled user. No elevated privileges, leaked credentials, or external dependencies are needed. The endpoint is a standard `POST` available to all verified users. Likelihood is **high**.

## Recommendation
Add a call to `validateSignature(sdkTransaction, map)` inside `processTransactionSignatures` before calling `sdkTransaction.addSignature(publicKey, map)`, mirroring the pattern already used in `importSignatures`. Replace the raw `addSignature` call with the verified public keys returned by `validateSignature`, as done at `transactions.service.ts` lines 551–553. [6](#0-5) 

## Proof of Concept
1. Register as a user and add a public key (e.g., Ed25519 key `PK`).
2. Obtain a transaction `T` in `WAITING_FOR_SIGNATURES` status that requires `PK` as a signer.
3. Construct a `SignatureMap` containing `PK` mapped to 64 bytes of random data (not a valid signature over `T`'s body bytes).
4. `POST /transactions/:id/signers` with the malformed `SignatureMap`.
5. Observe: the server returns HTTP 201, a `TransactionSigner` record is created for `PK`, and `T.transactionBytes` is updated to include the invalid signature bytes.
6. If `T` now has enough recorded signers to meet the threshold, its status advances to `WAITING_FOR_EXECUTION`.
7. When `T` is submitted to the Hedera network, it is rejected with `INVALID_SIGNATURE`, consuming all co-signers' valid signatures and potentially expiring the transaction.
