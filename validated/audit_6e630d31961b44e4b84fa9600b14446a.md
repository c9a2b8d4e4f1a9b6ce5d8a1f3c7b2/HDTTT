### Title
Missing `@Min(1)` Constraint on `threshold` in `CreateTransactionApproverDto` Allows Negative Threshold to Bypass Multi-Signature Approval Requirement

### Summary
`CreateTransactionApproverDto.threshold` accepts any integer including negative values because it lacks a `@Min(1)` decorator. The service-level guard only rejects `threshold === 0` and `threshold > approvers.length`, so a negative value (e.g., `-1`) passes all validation, is persisted to the database, and causes the approval-tree evaluation to treat the node as permanently satisfied — effectively bypassing the multi-signature approval gate for any transaction whose creator supplies a crafted payload.

### Finding Description

**Root cause — DTO missing lower-bound constraint**

`CreateTransactionApproverDto` decorates `threshold` with only `@IsNumber()` and `@IsOptional()`: [1](#0-0) 

Compare with `UpdateTransactionApproverDto`, which correctly adds `@Min(1)`: [2](#0-1) 

**Service-level guard does not catch negative values**

Inside `createTransactionApprovers`, the only threshold range check is: [3](#0-2) 

The condition `dtoApprover.threshold === 0` catches zero, and `dtoApprover.threshold > dtoApprover.approvers.length` catches values that are too high. A negative value such as `-1` satisfies neither branch (`-1 === 0` → false; `-1 > N` → false for any N ≥ 0), so execution continues.

**Negative threshold is persisted**

The assignment that stores the threshold uses a JavaScript truthiness check: [4](#0-3) 

In JavaScript, `-1` is truthy, so `dtoApprover.threshold && dtoApprover.approvers` evaluates to the approvers array (truthy) when threshold is `-1`, and the negative value is written to `transaction_approver.threshold`.

**Exploit path**

1. Attacker (any authenticated user) creates a transaction via `POST /transactions`.
2. Attacker calls `POST /transactions/:id/approvers` with:
   ```json
   { "approversArray": [{ "threshold": -1, "approvers": [{"userId": 2}, {"userId": 3}] }] }
   ```
3. The DTO passes `class-validator` (no `@Min` constraint) and passes the service guard (neither `=== 0` nor `> 2`).
4. A `transaction_approver` row is inserted with `threshold = -1`.
5. When the chain service evaluates approval status, it computes `approvedCount >= threshold`. With threshold `-1`, the condition `0 >= -1` is immediately true — the approval node is considered satisfied with zero actual approvals.
6. The transaction advances to execution without any approver signing.

### Impact Explanation
Any authenticated user who creates a transaction can unilaterally bypass the multi-signature approval requirement that the organization configured. Transactions that should require N-of-M approvals from designated approvers can be executed with zero approvals. This is a direct integrity failure in the trust model: the approval gate — the primary organizational control over transaction execution — is rendered ineffective.

### Likelihood Explanation
The attack requires only a valid authenticated session and the ability to create a transaction (a normal user capability). The crafted payload is a single integer field set to a negative number. No privileged access, leaked secrets, or special tooling are needed. Any user who can call `POST /transactions/:id/approvers` can trigger this.

### Recommendation
Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
import { IsArray, IsNumber, IsOptional, ValidateNested, ArrayMinSize, Min } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @IsOptional()
  listId?: number;

  @IsNumber()
  @Min(1)          // ← add this
  @IsOptional()
  threshold?: number;
  ...
}
```

Additionally, harden the service-level guard to use `threshold < 1` instead of `threshold === 0`:

```typescript
// approvers.service.ts line ~305
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold < 1)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
```

### Proof of Concept

**Preconditions**: attacker has a valid JWT (normal user account).

```bash
# 1. Create a transaction (normal flow)
POST /transactions
Authorization: Bearer <attacker_token>
{ "name": "...", "transactionBytes": "...", ... }
# → { "id": 42 }

# 2. Add approver tree with negative threshold
POST /transactions/42/approvers
Authorization: Bearer <attacker_token>
Content-Type: application/json
{
  "approversArray": [{
    "threshold": -1,
    "approvers": [
      { "userId": 10 },
      { "userId": 11 }
    ]
  }]
}
# → 201 Created (no validation error)

# 3. Verify DB state
SELECT threshold FROM transaction_approver WHERE "transactionId" = 42;
# → threshold = -1

# 4. Chain service evaluates: approvedCount (0) >= threshold (-1) → true
# Transaction proceeds to WAITING_FOR_EXECUTION / execution without any approver signing.
```

**Expected**: request rejected with `400 Bad Request` — threshold must be ≥ 1.
**Actual**: request accepted; negative threshold stored; approval gate bypassed.
