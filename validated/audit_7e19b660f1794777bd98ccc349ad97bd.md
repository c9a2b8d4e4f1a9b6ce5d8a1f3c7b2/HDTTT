Audit Report

## Title
Race Condition in `DebouncedNotificationBatcher.flush()` Allows Duplicate Notification Delivery and Message Loss

## Summary
`DebouncedNotificationBatcher` has two code paths that call `flush()`: a size-triggered path in `add()` (no lock) and an expiry-triggered path in `processExpiration()` (with a Redis NX lock). Because the lock only serialises concurrent `processExpiration()` calls and does not guard the `add()` → `flush()` path, both paths can execute `flush()` concurrently for the same `groupKey`. Additionally, `flush()` itself performs a non-atomic `LRANGE` + `DEL` sequence across two separate `await` calls, creating a window where messages can be read twice or silently dropped.

## Finding Description

**File:** `back-end/apps/notifications/src/utils/DebouncedNotificationBatcher.ts`

**Path 1 — size-triggered flush, no lock (`add()`, lines 75–77):**

When `rpush` returns a length ≥ `maxBatchSize`, `flush()` is called directly with no distributed lock acquired. [1](#0-0) 

**Path 2 — expiry-triggered flush, with lock (`processExpiration()`, lines 45–53):**

`processExpiration()` acquires a Redis `SET … NX` lock before calling `flush()`. However, the lock key is scoped to the expiry path only; it is never checked or acquired by `add()`. [2](#0-1) 

**Non-atomic read-then-delete in `flush()` (lines 101–108):**

`flush()` performs `LRANGE` (line 101) and `DEL` (line 107) as two separate `await` calls. The Node.js event loop yields at each `await`, creating two distinct race windows:
- **Duplicate read window:** A second concurrent `flush()` can call `LRANGE` before the first `flush()` has called `DEL`, causing both to read the same message set.
- **Message loss window:** A concurrent `add()` can `RPUSH` a new message into `batchKey` after `LRANGE` but before `DEL`; that message is then deleted but was never included in `parsedMessages`. [3](#0-2) 

**Concrete double-flush sequence:**
1. `add()` receives the Nth message (N ≥ `maxBatchSize`); `rpush` returns `length >= maxBatchSize`.
2. `add()` calls `flush()` → `await lrange(batchKey)` → event loop yields.
3. The flush key expires; Redis publishes a keyevent; the `subClient` message handler fires → `processExpiration()` → `await set(lockKey, NX)` → lock is free (never taken by `add()`) → acquired → calls `flush()` → `await lrange(batchKey)` → reads the **same** messages.
4. Both `flush()` invocations pass through the `messages.length === 0` guard (both see non-empty lists).
5. Both call `flushCallback(groupKey, parsedMessages)` → **duplicate notifications dispatched**.

The codebase's own comment at line 3 explicitly acknowledges the large-group scenario (>100 accounts) that makes `maxBatchSize` being reached concurrently with a debounce expiry a routine occurrence: [4](#0-3) 

## Impact Explanation
- **Duplicate notifications:** Every affected user receives the same signing/approval notification more than once for the same transaction, degrading trust and causing confusion in multi-signature workflows.
- **Silent message loss:** Messages pushed between `LRANGE` and `DEL` are permanently deleted without being dispatched. Affected signers never receive their notification, leaving transactions stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely.

## Likelihood Explanation
No attacker input is required. The race is triggered by normal high-throughput usage: creating a transaction group with >100 accounts (explicitly noted in the source comment) causes `maxBatchSize` to be reached at the same time a debounce timer fires. This is an expected production scenario, not an edge case.

## Recommendation
Replace the non-atomic `LRANGE` + `DEL` pair in `flush()` with a single atomic Redis Lua script:

```lua
local msgs = redis.call('LRANGE', KEYS[1], 0, -1)
redis.call('DEL', KEYS[1])
return msgs
```

This eliminates both the duplicate-read and message-loss windows in one step. Additionally, `add()` must acquire the same distributed lock before calling `flush()` — or the lock must be moved into `flush()` itself — so that the size-triggered and expiry-triggered paths are mutually exclusive for the same `groupKey`.

## Proof of Concept

```
T=0  add() → rpush(batchKey, msgN) → length = maxBatchSize
T=1  add() → flush() → await lrange(batchKey)  [yields event loop; returns [m1..mN]]
T=2  flushKey expires → subClient fires → processExpiration()
       → await set(lockKey, NX)  [lock free, acquired]
       → flush() → await lrange(batchKey)  [batchKey not yet deleted; returns [m1..mN]]
T=3  flush() #1 resumes → await del(batchKey) → await del(flushKey)
       → flushCallback(groupKey, [m1..mN])  ← first dispatch
T=4  flush() #2 resumes → await del(batchKey) [no-op, already deleted]
       → flushCallback(groupKey, [m1..mN])  ← duplicate dispatch
```

Any message pushed by a concurrent `add()` between T=1 and T=3 is deleted at T=3 but was not present in either `lrange` result → **silent message loss**.
