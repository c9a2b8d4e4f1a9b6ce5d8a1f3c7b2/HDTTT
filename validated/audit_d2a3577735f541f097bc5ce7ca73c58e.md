Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Progressive DoS for All API Users

## Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction from the database without any SQL `LIMIT`, then sequentially calls `userKeysToSign` (which triggers mirror-node API calls via `AccountCacheService`) for each row. Pagination is applied only after the full in-memory scan completes. As the number of active transactions grows through normal use, each invocation of this endpoint becomes progressively more expensive, eventually causing severe latency or service unavailability for all users sharing the API.

## Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method issues an unbounded database query at lines 295–299:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / LIMIT
});
``` [1](#0-0) 

It then iterates over every returned row with a sequential `await` inside the loop at lines 301–309:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Pagination is applied only after the full scan at lines 311–316:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  page,
  size,
};
``` [3](#0-2) 

The `whereForUser` filter at lines 266–277 excludes only five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`) and does **not** filter by the requesting user — every non-terminal transaction across all users and organizations is loaded into memory and processed. [4](#0-3) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `TransactionSignatureService.computeSignatureKey`. [9](#0-8) 

`computeSignatureKey` makes one or more async calls to `AccountCacheService.getAccountInfoForTransaction` per transaction (for fee payer, signing accounts, and receiver accounts). [10](#0-9) 

`getAccountInfoForTransaction` consults a cache but falls back to a live mirror-node fetch when the cache is stale or missing. [11](#0-10) 

The endpoint is exposed at `GET /transactions/sign` and is still live and accessible to any authenticated user, even though it is annotated as "NO LONGER USED BY FRONT-END". [12](#0-11) 

## Impact Explanation

- **Memory exhaustion**: Loading thousands of `Transaction` rows (with `groupItem` relations) into the Node.js heap can exhaust server memory.
- **CPU/latency exhaustion**: The sequential `await` loop means each mirror-node round-trip blocks the next iteration. With N active transactions, response time grows as O(N × mirror-node latency).
- **Shared-service DoS**: The API is a single NestJS process. A slow or hung request occupies an event-loop tick and a DB connection, degrading all concurrent users.
- **Cross-user amplification**: Because `whereForUser` does not filter by the requesting user's transactions, every transaction created by any user on the platform increases the cost of this endpoint for every other user who calls it.

## Likelihood Explanation

- The endpoint is reachable by any authenticated user — no elevated privilege is required.
- Normal platform operation (organizations creating many transactions) naturally grows the dataset over time.
- A malicious authenticated user can deliberately create large numbers of transactions to accelerate the growth, since there is no per-user transaction creation cap visible in the service.
- The throttler modules present (`ip-throttler`, `user-throttler`) rate-limit request frequency but do not bound the per-request work, so they do not mitigate this issue.
- The `AccountCacheService` caching (`CACHE_STALE_THRESHOLD_MS`, default 10 s) reduces but does not eliminate mirror-node calls, since cache entries expire and new accounts may not be cached at all.

## Recommendation

1. **Push pagination into the database query**: Add `take: limit` and `skip: offset` to the `repo.find()` call. This requires a two-pass approach — first query the DB for candidate transactions (filtered by user membership), then call `userKeysToSign` only on the page-sized result set.
2. **Filter by user at the DB level**: Add a join/subquery so only transactions where the requesting user's keys are relevant are returned, rather than loading all non-terminal transactions across all users.
3. **Parallelize `userKeysToSign` calls**: Replace the sequential `await` loop with `Promise.all` or a bounded concurrency pool (e.g., `p-limit`) to reduce wall-clock latency.
4. **Consider removing the endpoint**: The controller already notes it is "NO LONGER USED BY FRONT-END". If it is truly unused, removing it eliminates the attack surface entirely.

## Proof of Concept

1. Authenticate as any verified user and obtain a JWT.
2. Create a large number of transactions (e.g., 10,000) via `POST /transactions`, each with a unique `transactionId` and status `NEW` or `WAITING_FOR_SIGNATURES`.
3. Issue a single `GET /transactions/sign?page=1&size=10` request.
4. Observe that the server loads all N transactions from the database, sequentially calls `userKeysToSign` (and potentially the mirror node) for each, and only then slices the result to 10 items.
5. Response time grows linearly with N. With sufficient transactions, the request times out or the server becomes unresponsive for all concurrent users.

  async getAccountInfoForTransaction(
    transaction: Transaction,
    account: string,
    isReceiver: boolean = false,
  ): Promise<AccountInfoParsed | null> {
    if (!transaction || !transaction.mirrorNetwork) {
      return null;
    }

    this.validateAccount(account);

    const mirrorNetwork = transaction.mirrorNetwork;

    // Get cached data
    const cached = await this.dataSource.manager.findOne(CachedAccount, {
      where: { account, mirrorNetwork },
    });

    if (this.hasCompleteData(cached) && isFresh(cached.updatedAt, this.cacheTtlMs)) {
      // Link to transaction even if using cache
      await this.linkTransactionToAccount(transaction.id, cached.id, isReceiver);
      return this.parseCachedAccount(cached);
    }

    // Cache is stale or doesn't exist - fetch new data
    this.logger.debug(`Fetching account ${account} from mirror node (cache ${cached ? 'stale' : 'missing'})`);

    // Try to claim the account for refresh, create the account row if none exists
    const { data: claimedAccount, claimed } = await this.tryClaimAccountRefresh(account, mirrorNetwork);

    if (!claimed) {
      // Link to transaction
      await this.linkTransactionToAccount(transaction.id, claimedAccount.id, isReceiver);

      if (this.hasCompleteData(claimedAccount)) {
        return this.parseCachedAccount(claimedAccount);
      }

      // No cached data
      // This should never happen
      return null;
    }

    const { data } = await this.performRefreshForClaimedAccount(claimedAccount, transaction.id, isReceiver);
    return data;
  }
