Audit Report

## Title
`archiveTransaction` Bypasses Terminal-State Guard for Manual Transactions, Allowing Unauthorized State Transitions

## Summary

The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` uses a compound `&&` condition that completely skips the status check whenever `transaction.isManual` is `true`. Any authenticated user who is the creator of a manual transaction can move it from any terminal state (`CANCELED`, `EXECUTED`, `REJECTED`, `EXPIRED`, `FAILED`) to `ARCHIVED`, violating the service's own invariant that terminal states are permanent.

## Finding Description

**Root cause — `archiveTransaction` (lines 711–718):**

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← false when isManual=true → entire guard is false
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** sub-expressions are `true`:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`, **AND**
- `isManual` is `false`.

When `isManual = true`, `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false` regardless of `transaction.status`. The guard never fires, and `repo.update` at line 720 executes unconditionally. [1](#0-0) 

**Contrast with the correctly guarded path — `cancelTransactionWithOutcome` (lines 669–671):**

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);   // always enforced, no isManual bypass
}
```

`cancelTransactionWithOutcome` validates status unconditionally, independent of any other field. `archiveTransaction` does not — the same invariant is enforced inconsistently across two operations that both mutate transaction state. [2](#0-1) 

**Terminal states defined by the service (lines 98–105):**

```typescript
private readonly terminalStatuses = [
  TransactionStatus.EXECUTED,
  TransactionStatus.EXPIRED,
  TransactionStatus.FAILED,
  TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED,
  TransactionStatus.REJECTED,
];
``` [3](#0-2) 

**Test suite confirms the gap:** The only `archiveTransaction` test that checks a terminal status uses a transaction where `isManual` is `undefined` (falsy). No test covers `isManual = true` with a terminal status, so the bypass is untested. [4](#0-3) 

## Impact Explanation

- **Audit-trail corruption**: An `EXECUTED` transaction (confirmed on the Hedera network) can be relabeled `ARCHIVED` by its creator, hiding the fact that it was submitted and confirmed.
- **Rejection concealment**: A `REJECTED` transaction can be relabeled `ARCHIVED` by the creator, erasing the rejection record from the workflow history.
- **Invariant violation**: The service's own `terminalStatuses` list declares these states as permanent; `archiveTransaction` silently violates that contract for any manual transaction. [5](#0-4) 

## Likelihood Explanation

- **No privileged access required**: Any verified user can create a manual transaction — `isManual: true` is a standard creation-time field accepted by `POST /transactions`.
- **Reachable via standard API**: `PATCH /transactions/archive/:id` is a documented, authenticated endpoint. [6](#0-5) 

- **Low complexity**: The attacker only needs to know their own transaction ID (returned at creation time) and call one endpoint. [7](#0-6) 

## Recommendation

Replace the compound `&&` condition with an explicit status allowlist that applies unconditionally, regardless of `isManual`:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  const archivableStatuses = [
    TransactionStatus.WAITING_FOR_SIGNATURES,
    TransactionStatus.WAITING_FOR_EXECUTION,
  ];

  if (!archivableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  // ...
}
```

This mirrors the pattern used in `cancelTransactionWithOutcome` and ensures the status guard is always enforced, independent of `isManual`. [8](#0-7) 

## Proof of Concept

1. Attacker (normal authenticated user) creates a manual transaction via `POST /transactions` with `isManual: true`. The response returns the transaction `id`.
2. The approval workflow or an admin moves the transaction to `REJECTED` (or the system moves it to `EXECUTED`, `CANCELED`, `EXPIRED`, or `FAILED`).
3. Attacker calls `PATCH /transactions/archive/:id`.
4. `getTransactionForCreator` succeeds — the attacker is the creator.
5. The guard at lines 711–718 evaluates: `(status NOT IN allowlist) && !true` → `(true) && (false)` → `false`. The exception is never thrown.
6. `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` at line 720 executes unconditionally.
7. The transaction's terminal status is overwritten with `ARCHIVED`, corrupting the audit trail. [1](#0-0)
