Audit Report

## Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Sabotage of Multi-Sig Transactions

## Summary
`SignersService.processTransactionSignatures` accepts attacker-supplied `SignatureMap` bytes and calls `sdkTransaction.addSignature` after only verifying public key ownership. It never calls `validateSignature` to cryptographically verify that the submitted bytes are valid for the key. A required signer can submit their registered public key paired with arbitrary/random signature bytes, causing the system to record a fraudulent signed state, potentially advance the transaction to `WAITING_FOR_EXECUTION`, and ultimately cause permanent failure when Hedera rejects the invalid signature.

## Finding Description

**Root cause:** `processTransactionSignatures` in `signers.service.ts` performs only an ownership check but skips cryptographic verification before calling `addSignature`.

The vulnerable path is:
`POST /transactions/:transactionId/signers` → `SignersController.uploadSignatureMap` → `SignersService.uploadSignatureMaps` → `validateAndProcessSignatures` → `processTransactionSignatures`

In `processTransactionSignatures`, the loop over the signature map:
1. Checks the key exists in `userKeyMap` (built from `user.keys`) — ownership check only.
2. Immediately calls `sdkTransaction.addSignature(publicKey, map)` — no cryptographic check. [1](#0-0) 

Contrast this with `TransactionsService.importSignatures`, which explicitly calls `validateSignature` before `addSignature`: [2](#0-1) 

`validateSignature` performs `publicKey.verify(bodyBytes, signature)` and throws on failure: [3](#0-2) 

Because `addSignature` adds the public key to `sdkTransaction._signerPublicKeys` without verifying the bytes, the `hasValidSignatureKey` check in both `processTransactionStatus` and `ExecuteService.getValidatedSDKTransaction` will see the key as having signed: [4](#0-3) [5](#0-4) 

`hasValidSignatureKey` only checks key presence in `_signerPublicKeys` — it does not re-verify the signature bytes. The transaction is then submitted to the Hedera network with an invalid signature and permanently fails.

Once `FAILED`, the transaction cannot be re-executed: [6](#0-5) 

## Impact Explanation

A required signer for a multi-sig organization transaction can:
1. Submit their registered public key with arbitrary/random signature bytes via `POST /transactions/:id/signers`.
2. The system records the key as having signed and updates `transactionBytes` with the invalid signature.
3. `processTransactionStatus` advances the transaction to `WAITING_FOR_EXECUTION` if the threshold is met.
4. The chain service submits the transaction to Hedera; Hedera rejects it with an invalid signature error.
5. The transaction is permanently marked `FAILED` and cannot be re-signed or re-executed.

This is especially severe for high-value or time-sensitive Hedera operations (e.g., `FreezeTransaction`, `FileUpdateTransaction`, `NodeUpdateTransaction`).

## Likelihood Explanation

- **Attacker preconditions:** Must be an authenticated organization user with a registered public key listed as a required signer for the target transaction. This is a normal, non-privileged role.
- **Attack complexity:** Trivial. The attacker submits their valid public key paired with 64 bytes of zeros as the signature. No cryptographic knowledge required.
- **Detection:** The attack is silent until the Hedera network rejects the transaction. No server-side error is raised during signature upload.

## Recommendation

In `processTransactionSignatures`, call `validateSignature` before `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// Before calling addSignature, verify the signature bytes are cryptographically valid
const { data: validPublicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);

sdkTransaction = sdkTransaction.addSignature(publicKey, map);
```

The `validateSignature` utility already exists at `back-end/libs/common/src/utils/sdk/transaction.ts` and is imported in `transactions.service.ts`. It should be reused here. [7](#0-6) 

## Proof of Concept

```
1. Attacker (authenticated user with registered key `pubKeyA`) targets transaction ID 42,
   which requires `pubKeyA` as a required signer.

2. Attacker crafts a POST /transactions/42/signers body:
   {
     "id": 42,
     "signatureMap": {
       "0.0.3": {
         "<transactionId>": {
           "<pubKeyA_DER>": "<base64(64 zero bytes)>"
         }
       }
     }
   }

3. Server executes processTransactionSignatures:
   - userKeyMap.get(pubKeyA_raw) → found (ownership check passes)
   - sdkTransaction.addSignature(pubKeyA, map) → pubKeyA added to _signerPublicKeys
   - No validateSignature call → invalid bytes accepted

4. processTransactionStatus reads _signerPublicKeys, sees pubKeyA present,
   hasValidSignatureKey returns true → status set to WAITING_FOR_EXECUTION.

5. Chain service calls executeTransaction → hasValidSignatureKey passes again
   (still only checks key presence) → submits to Hedera.

6. Hedera returns INVALID_SIGNATURE → transaction status set to FAILED permanently.
```

```typescript
export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```
