Audit Report

## Title
Any Authenticated Transaction Creator Can Delete Approvers Belonging to Other Users' Transactions (IDOR)

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but never verifies that the approver identified by `:id` actually belongs to that transaction. An attacker who owns any transaction can supply their own `transactionId` to pass the ownership check, then supply an arbitrary approver `id` from a completely different transaction to delete it.

## Finding Description

**Root cause:** In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent, uncorrelated operations:

```typescript
// approvers.controller.ts lines 108-109
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of :transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver :id — no cross-check
``` [1](#0-0) 

There is no assertion that the approver `id` belongs to `transactionId`. The service method `removeTransactionApprover` only checks that the approver row exists, then deletes it unconditionally:

```typescript
// approvers.service.ts lines 534-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-transaction validation that `removeTransactionApprover` omits:

```typescript
// approvers.service.ts lines 386-394
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

if (rootNode.transactionId !== transactionId)          // ← cross-check present here
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The `DELETE` handler is missing exactly this pattern.

## Impact Explanation

Removing an approver from a transaction that requires multi-party approval can:
- Bypass the approval threshold, allowing a transaction to proceed to execution without the required signatures.
- Permanently corrupt the approval workflow of any transaction in the system.
- Silently remove a required approver without the transaction creator's knowledge, undermining multi-signature security guarantees.

This is an unauthorized, persistent state change with direct impact on transaction integrity.

## Likelihood Explanation

Any registered, verified user who has created at least one transaction can exploit this. No admin or privileged access is required. The attacker only needs a valid JWT token. Approver IDs are sequential integers (auto-increment primary keys), making enumeration trivial. The attack requires only a single crafted HTTP request.

## Recommendation

In the `removeTransactionApprover` controller handler, after fetching the approver, verify that its root transaction matches the URL `:transactionId` parameter — exactly as `updateTransactionApprover` does. Concretely, call `getRootNodeFromNode(id)` and assert `rootNode.transactionId === transactionId` before proceeding with deletion. The fix should be applied in `approvers.service.ts` inside `removeTransactionApprover`, or as an additional guard in the controller handler.

```typescript
// Recommended fix pattern (mirrors updateTransactionApprover)
await this.approversService.getCreatorsTransaction(transactionId, user);
const rootNode = await this.approversService.getRootNodeFromNode(id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
await this.approversService.removeTransactionApprover(id);
```

## Proof of Concept

1. Attacker (User A) registers and creates **Transaction A** — they are its creator.
2. Victim (User B) creates **Transaction B** with a required approver (approver ID = `42`).
3. Attacker enumerates approver IDs (sequential integers) and identifies approver `42` belonging to Transaction B.
4. Attacker sends:
   ```
   DELETE /transactions/{A_id}/approvers/42
   Authorization: Bearer <attacker_jwt>
   ```
5. `getCreatorsTransaction(A_id, attacker)` passes — attacker owns Transaction A.
6. `removeTransactionApprover(42)` executes — approver `42` (belonging to Transaction B) is soft-deleted with no further validation.
7. Transaction B's approval workflow is now corrupted without the victim's knowledge. [1](#0-0) [2](#0-1)
