### Title
Transaction Creator Can Add Themselves as Approver and Self-Approve Their Own Transaction

### Summary
The `createTransactionApprovers` function in `ApproversService` does not check whether the `userId` being added as an approver matches the transaction creator's own user ID. Because `approveTransaction` only checks that the caller appears in the approvers list, a creator can add themselves as the sole approver and then immediately approve their own transaction, completely bypassing the intended separation-of-duties approval workflow.

### Finding Description

**Root cause — `createTransactionApprovers` (no self-approver guard):**

`createTransactionApprovers` first verifies the caller is the creator via `getCreatorsTransaction` (line 239), then iterates over the submitted `approversArray`. For each entry with a `userId`, it only checks that the user exists in the database (lines 271–276). There is no check that `dtoApprover.userId !== user.id` (the creator's own ID). [1](#0-0) 

**Root cause — `approveTransaction` (no creator-is-approver guard):**

`approveTransaction` calls `getVerifiedApproversByTransactionId`, which grants access to the creator unconditionally (line 144). It then filters `approvers` for entries where `a.userId === user.id` (line 556). If the creator previously added themselves, `userApprovers.length > 0` passes and the approval proceeds. There is no check that the approving user is not the transaction creator. [2](#0-1) 

**Access path:**

The controller exposes both operations to any authenticated, verified user: [3](#0-2) 

**Exploit flow:**

1. Attacker (creator, user ID = `X`) creates a transaction that is configured to require approvals.
2. Attacker calls `POST /transactions/:id/approvers` with body `{ "approversArray": [{ "userId": X }] }`. `createTransactionApprovers` accepts this because it only verifies the caller is the creator and that user `X` exists — both true.
3. Attacker calls `POST /transactions/:id/approvers/approve` with a valid signature. `approveTransaction` finds the attacker in the approvers list and records the approval.
4. The approval threshold is satisfied; the transaction advances to `WAITING_FOR_EXECUTION` without any independent party having reviewed it.

### Impact Explanation
The approval workflow is the primary governance control in Organization mode. It is intended to require a party other than the creator to authorize sensitive transactions (e.g., large HBAR transfers, account key changes, node operations). A creator who self-approves defeats this control entirely, allowing unilateral execution of any transaction type the system supports — including privileged operations such as `ACCOUNT_UPDATE`, `NODE_CREATE`, and `NODE_UPDATE`. [4](#0-3) 

### Likelihood Explanation
The attacker only needs a valid organization account and the ability to create transactions — the default capability of any authenticated user. No special privileges, leaked credentials, or race conditions are required. The two API calls are standard REST operations documented in the Swagger spec. Any malicious or compromised creator can exploit this on every transaction they create.

### Recommendation
In `createTransactionApprovers`, after verifying the caller is the creator, reject any `dtoApprover.userId` that equals `user.id`:

```typescript
if (typeof dtoApprover.userId === 'number' && dtoApprover.userId === user.id) {
  throw new Error('Transaction creator cannot be added as an approver');
}
```

Additionally, add a symmetric guard in `approveTransaction` to reject approval when the caller is the transaction creator:

```typescript
if (transaction.creatorKey?.userId === user.id) {
  throw new UnauthorizedException('Transaction creator cannot approve their own transaction');
}
``` [5](#0-4) 

### Proof of Concept

**Preconditions:** Attacker has a valid JWT for user ID `42` in an organization. A transaction with ID `7` exists and was created by user `42`.

```http
# Step 1 – Add self as approver
POST /transactions/7/approvers
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{
  "approversArray": [{ "userId": 42 }]
}

# Expected response: 201 Created — approver record inserted with userId=42

# Step 2 – Self-approve
POST /transactions/7/approvers/approve
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{
  "userKeyId": <attacker_key_id>,
  "signature": "<valid_signature_over_transaction_body>",
  "approved": true
}

# Expected response: 200 true
# Transaction status advances to WAITING_FOR_EXECUTION with no independent approval
```

The `getCreatorsTransaction` guard at line 239 passes (caller is creator). The `isNode` duplicate check at line 250 passes (first insertion). The `userApprovers` filter at line 556 returns the self-added record. The approval is recorded and the transaction proceeds. [6](#0-5) [7](#0-6)
