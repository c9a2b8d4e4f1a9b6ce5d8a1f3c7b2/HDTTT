Audit Report

## Title
Stale In-Memory State After DB Write Causes `emitTransactionStatusUpdate` to Never Fire on the Approval Path

## Summary
In `approveTransaction` within `approvers.service.ts`, the `userApprovers` array is loaded from the database before the approval write. After the write, the in-memory objects are never refreshed. The post-write guard condition that decides whether to emit `emitTransactionStatusUpdate` or `emitTransactionUpdate` reads the stale in-memory `approved` field. Because the stale value is always `null/undefined` for a first-time approval, the condition always evaluates to `false` on the approval path, so `emitTransactionStatusUpdate` is never emitted when a user approves — only `emitTransactionUpdate` is.

## Finding Description

**File:** `back-end/apps/api/src/transactions/approvers/approvers.service.ts`

**Step 1 — Pre-write load of `userApprovers`**

`userApprovers` is populated from the database before any write occurs: [1](#0-0) 

**Step 2 — DB write via `createQueryBuilder().update().execute()`**

The approval state is written to the database using a raw query builder. TypeORM's `execute()` on a query builder does **not** update the in-memory entity objects — `userApprovers[i].approved` remains `null/undefined` after this call: [2](#0-1) 

**Step 3 — Stale read in the post-write guard**

Immediately after the write, the code reads the same stale in-memory array to decide which event to emit: [3](#0-2) 

**Condition analysis:**

| `dto.approved` | `!dto.approved` | `userApprovers.every(a => a.approved)` (stale) | Combined | Event emitted |
|---|---|---|---|---|
| `true` (approve) | `false` | `false` (stale null/undefined) | `false` | `emitTransactionUpdate` ← **wrong** |
| `false` (reject) | `true` | (short-circuits) | `true` | `emitTransactionStatusUpdate` ← correct |

The pre-write guard at line 563 (`if (userApprovers.every(a => a.signature))`) ensures that at least one approver record has no signature yet, meaning the stale `approved` values are always `null/undefined` for the records being updated. Therefore `every(a => a.approved)` is deterministically `false` on every first-time approval. [4](#0-3) 

**Test suite confirms the bug silently:**

The existing test calls `approveTransaction` twice — once with `approved: true` and once with `approved: false` — then asserts `emitTransactionStatusUpdate` was called. The assertion passes because the **rejection** call satisfies it; the approval call's incorrect behavior is never isolated or caught: [5](#0-4) 

## Impact Explanation

`emitTransactionStatusUpdate` publishes to the `TRANSACTION_STATUS_UPDATE` NATS subject. The notifications service's `processTransactionStatusUpdateNotifications` handler processes this event to re-evaluate transaction status, send in-app/email notifications, and push real-time client updates: [6](#0-5) 

Without this event on the approval path, the chain service's `processTransactionStatus` is not triggered immediately after approval. The scheduler provides fallback coverage, but with delays that scale with `validStart` distance: [7](#0-6) 

| `validStart` distance | Scheduler interval | Max delay |
|---|---|---|
| < 3 minutes | 10 seconds | ~10 s |
| 3–10 minutes | 30 seconds | ~30 s |
| 10 min – 1 hour | 10 minutes | ~10 min |
| 1 hour – 1 day | 30 minutes | ~30 min |
| 1 day – 1 week | daily at midnight | ~24 hours |
| > 1 week | daily at midnight | ~24 hours |

For transactions with `validStart` more than an hour in the future — a common case for pre-scheduled multi-party transactions — the approval event is silently swallowed and the transaction will not advance to `WAITING_FOR_EXECUTION` for up to 30 minutes or 24 hours. Approvers and observers receive no real-time notification that the approval threshold has been met.

## Likelihood Explanation

This code path is exercised on every approval action. The stale-read condition is deterministic: for any first-time approval, `userApprovers[i].approved` is `null/undefined` before the write, so the guard always misfires. No special setup is required; any normal user performing a normal approval triggers the bug.

## Recommendation

After the `createQueryBuilder().update().execute()` call, mutate the in-memory objects to reflect the written state before evaluating the guard condition:

```typescript
// After the DB write:
userApprovers.forEach(a => {
  a.userKeyId = dto.userKeyId;
  a.signature  = dto.signature;
  a.approved   = dto.approved;
});

if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

Alternatively, simplify the condition entirely: since any approval or rejection should trigger a status re-evaluation (the chain service will determine whether the status actually changes), always emit `emitTransactionStatusUpdate` unconditionally after a successful write.

Additionally, the test at line 1271–1278 should be split into two isolated cases — one for `approved: true` and one for `approved: false` — each asserting the correct event independently.

## Proof of Concept

1. Create a transaction with `validStart` set 2 hours in the future and configure one required approver.
2. The approver calls `POST /transactions/:id/approvers/approve` with `approved: true`.
3. The DB write succeeds — `TransactionApprover.approved` is `true` in the database.
4. Observe via NATS monitoring: only `TRANSACTION_UPDATE` is published; `TRANSACTION_STATUS_UPDATE` is **not** published.
5. The transaction remains in `WAITING_FOR_SIGNATURES` (or does not advance to `WAITING_FOR_EXECUTION`) until the 30-minute scheduler bucket fires.
6. Repeat with `approved: false` (rejection): `TRANSACTION_STATUS_UPDATE` **is** published immediately, confirming the asymmetry. [8](#0-7)

    const ctx = await this.prepareEventContext(events, true);
    if (!ctx) return;

    const {
      cache,
      keyCache,
      transactionMap,
      approversMap,
      deletionNotifications,
      inAppNotifications,
      emailNotifications,
      inAppReceiverIds,
      emailReceiverIds,
      affectedUsers,
    } = ctx;

    // Process each event
    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping status-update notifications`);
        continue;
      }
      const approvers = approversMap.get(transactionId) || [];

      if (transaction.deletedAt && transaction.status !== TransactionStatus.CANCELED) {
        console.error(
          `Soft-deleted transaction ${transactionId} has unexpected status: ${transaction.status} (expected CANCELED)`
        );
        transaction.status = TransactionStatus.CANCELED;
      }

      const syncType = this.getInAppNotificationType(transaction.status);
      const emailType = this.getEmailNotificationType(transaction.status);

      // Single transaction for both notification types
      await this.entityManager.transaction(async entityManager => {
        await this.handleTransactionStatusUpdateNotifications(
          entityManager,
          transaction,
          approvers,
          syncType,
          emailType,
          cache,
          keyCache,
          deletionNotifications,
          inAppNotifications,
          inAppReceiverIds,
          emailNotifications,
          emailReceiverIds,
          affectedUsers,
          transactionId,
        );
      });
    }

    // Send all notifications in batch
    await this.sendDeletionNotifications(deletionNotifications);
    await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
    await this.sendEmailNotifications(emailNotifications, emailReceiverIds);
    await this.sendNotifyClients(affectedUsers, TRANSACTION_EVENT_TYPE.STATUS_UPDATE);
  }
