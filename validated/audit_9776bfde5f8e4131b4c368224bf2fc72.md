### Title
Any Authenticated User Can Enumerate All Transaction Groups via Missing Authorization on `getTransactionGroups()`

### Summary
The `GET /transaction-groups` endpoint in `TransactionGroupsController` is explicitly marked "TESTING ONLY" in source code but remains live in production. It lacks user-scoped access control and returns **all** transaction groups in the system to any authenticated user. This directly contradicts the per-user authorization enforced by the adjacent `getTransactionGroup()` (singular) endpoint, which correctly gates access by the requesting user.

### Finding Description

**Root cause:** `getTransactionGroups()` at line 52–57 of `back-end/apps/api/src/transactions/groups/transaction-groups.controller.ts` does not pass the authenticated user to the service layer, so no ownership or membership filter is applied.

```
/* TESTING ONLY: Get all transactions groups */
@Get()
@Serialize(TransactionGroupDto)
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroups();
}
``` [1](#0-0) 

The class-level guard stack (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only verifies that the caller is a valid, verified JWT holder — it performs **no authorization** check. [2](#0-1) 

**Contrast with the correctly-guarded singular endpoint**, which passes `user` to the service so it can enforce membership:

```typescript
getTransactionGroup(
  @GetUser() user: User,
  @Param('id', ParseIntPipe) groupId: number,
  ...
): Promise<TransactionGroup> {
  return this.transactionGroupsService.getTransactionGroup(user, groupId, full ?? true);
}
``` [3](#0-2) 

The inconsistency is structural: the bulk listing endpoint omits the user parameter entirely, so the service has no basis on which to filter results.

**Exploit path:**
1. Attacker registers or obtains any valid organization account (no admin role required).
2. Attacker sends `GET /transaction-groups` with a valid JWT.
3. The endpoint returns every `TransactionGroup` row in the database, including groups the attacker has no membership in.

### Impact Explanation
- **Cross-tenant data exposure:** An attacker can enumerate all transaction groups belonging to other users/teams within the organization, revealing group membership, associated transaction IDs, group metadata, and execution sequencing.
- **Reconnaissance for further attacks:** Knowing all transaction IDs and group structures allows a low-privilege user to target specific transactions for cancellation, signature injection, or observer manipulation via other endpoints.
- **Severity:** Medium-High — no privileged role is required; any verified user can trigger this.

### Likelihood Explanation
The endpoint is reachable by any authenticated user with a valid JWT. The attacker precondition is simply having a registered account in the organization, which is a normal user state. The route is a standard REST `GET` with no additional friction. The "TESTING ONLY" comment confirms the endpoint was never intended to be production-accessible, making its presence an oversight rather than a design choice.

### Recommendation
1. **Remove the endpoint entirely** — the comment explicitly marks it as testing-only; it should not exist in production builds.
2. If a listing endpoint is genuinely needed, pass the authenticated user to the service and filter results to groups the user is a creator of or a member of, mirroring the pattern used by `getTransactionGroup()`.
3. Add an `AdminGuard` at minimum if a global listing is required for administrative purposes.

### Proof of Concept

**Preconditions:** Two accounts exist — `victim` (creator of a transaction group) and `attacker` (a regular verified user with no group membership).

```
# Step 1 – Attacker logs in
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<attacker_jwt>" }

# Step 2 – Attacker lists ALL transaction groups
GET /transaction-groups
Authorization: Bearer <attacker_jwt>

# Expected (correct) response: [] or only attacker's own groups
# Actual response: full list of every TransactionGroup in the DB,
#                  including victim's groups with their transaction IDs,
#                  group items, and sequencing metadata
```

The attacker receives `TransactionGroupDto` objects for groups they have no legitimate access to, confirmed by the fact that `getTransactionGroups()` calls `this.transactionGroupsService.getTransactionGroups()` with no user argument. [4](#0-3)
