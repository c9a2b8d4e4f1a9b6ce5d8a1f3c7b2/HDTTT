Audit Report

## Title
Denial of Service via Unbounded Full-Table Scan in `getTransactionsToSign`

## Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction from the database with no row limit, then performs a sequential async `userKeysToSign()` DB call per row before applying pagination in memory. The endpoint remains live and is accessible to any verified user, making it trivially exploitable for resource exhaustion. A secondary surface in `importSignatures` accepts an unbounded array with no enforced size cap.

## Finding Description

**Root cause — no DB-level pagination in `getTransactionsToSign`:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `repo.find()` call at lines 295–299 carries no `take` or `skip` constraint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` or `skip`
});
```

TypeORM therefore issues a `SELECT … FROM transaction` with no `LIMIT`. Every matching non-terminal row is loaded into the Node.js heap. Then `userKeysToSign()` — which itself performs async DB work — is awaited sequentially for each row at lines 301–308. Pagination is applied only after the full scan at line 313:

```typescript
items: result.slice(offset, offset + limit),
``` [1](#0-0) 

**Exposed endpoint still live:**

The route `GET /transactions/sign` is registered and active. The controller comment explicitly acknowledges it is `/* NO LONGER USED BY FRONT-END */` but the route is not removed or disabled. It is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — any verified organization user can call it. [2](#0-1) 

**Contrast with other methods that use DB-level pagination:**

`getTransactions` and `getHistoryTransactions` both pass `skip: offset, take: limit` inside their `FindManyOptions`, delegating pagination to the database engine. [3](#0-2) [4](#0-3) 

**Secondary surface — `importSignatures` has no array-size cap (developer-acknowledged):**

The developer left a comment at line 575 acknowledging the missing limit:

```typescript
//Added a batch mechanism, probably should limit this on the api side of things
const BATCH_SIZE = 500;
```

The `POST /transactions/signatures/import` endpoint accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no enforced upper bound on array length. [5](#0-4) [6](#0-5) 

## Impact Explanation

With N pending (non-terminal) transactions in the database, each call to `GET /transactions/sign`:

1. Loads all N transaction rows into Node.js heap memory via a single unbounded `SELECT`.
2. Executes N sequential async `userKeysToSign()` calls, each of which performs its own DB round-trips via `userKeysRequiredToSign` → `keysRequiredToSign`.

This is O(N) memory and O(N) DB queries per request. At scale this causes Node.js OOM crashes or request timeouts, making the endpoint — and potentially the entire API process — unavailable to all users.

For `importSignatures`, a single authenticated user can submit an arbitrarily large array, triggering O(N) cryptographic operations (`SDKTransaction.fromBytes`, `validateSignature`, `addSignature`) in one request. [7](#0-6) 

## Likelihood Explanation

- **Precondition:** A valid JWT for any verified organization user — no admin role required.
- **Trigger:** A single `GET /transactions/sign` HTTP request. No special tooling needed.
- **Organic trigger:** Even without a malicious actor, a busy organization accumulating thousands of pending transactions will hit this naturally.
- **Transaction creation cost:** Creating transactions requires valid Hedera transaction bytes and a creator key signature, but a determined attacker with a registered key can automate this at low cost.

## Recommendation

1. **`getTransactionsToSign`:** Add `take: limit` and `skip: offset` to the `repo.find()` call. Because the filtering on `keysToSign.length > 0` is done in application code, the correct fix is to push the key-matching logic into the database query (e.g., a subquery or join on `transaction_signer` filtered by the user's key IDs), enabling true DB-level pagination. Alternatively, if the endpoint is truly unused by the front-end, remove it entirely.
2. **`importSignatures`:** Enforce a maximum array length (e.g., 50–100 items) at the controller level using a class-validator `@ArrayMaxSize()` decorator or a manual guard before processing.

## Proof of Concept

```
# Attacker registers as a verified user and obtains a JWT.
# Attacker (or organic growth) accumulates N non-terminal transactions.
# Single request triggers full-table scan + N async DB calls:

GET /transactions/sign?page=1&size=10
Authorization: Bearer <valid_jwt>

# With N = 10,000 pending transactions:
# - DB returns 10,000 rows into heap
# - 10,000 sequential userKeysToSign() calls execute
# - Server memory spikes; request times out or OOM kills the process
# - All concurrent users are denied service
```

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
