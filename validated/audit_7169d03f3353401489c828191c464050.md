### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated-User DoS Against All API Consumers

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction in the database with no row limit, then performs an expensive per-row async operation (mirror-node HTTP call + transaction-bytes deserialization) before applying pagination in memory. Any authenticated user can pre-populate the database with thousands of long-lived transactions, causing every subsequent call to `GET /transactions/sign` by any user to time out or consume unbounded server resources — a direct analog to the external report's unbounded-iteration DoS pattern.

### Finding Description

**Root cause — missing `take` on the repository query:**

In `getTransactionsToSign`, the TypeORM `find` call at line 295 carries no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters by status, NOT by user
  relations: ['groupItem'],
  order,
});
```

`whereForUser` excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope the query to the requesting user's transactions. Every active transaction in the entire system is returned.

The result set is then iterated synchronously, one transaction at a time, with a costly async call per row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
```

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` → `computeSignatureKey` which:
1. Deserializes raw transaction bytes (`SDKTransaction.fromBytes`)
2. Makes one or more outbound mirror-node HTTP calls (`getAccountInfoForTransaction`) to resolve fee-payer and signing-account keys
3. Issues a database query for matching `UserKey` rows

Pagination is applied only **after** all rows are processed:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // post-hoc slice
  ...
};
```

**Attack path:**

1. Attacker registers a normal user account and obtains a verified JWT (no privilege required).
2. Attacker calls `POST /transactions` repeatedly, each time with a unique `transactionId` (different `validStart` timestamp, e.g. far in the future) and a valid `creatorKeyId`. There is no per-user transaction count limit or rate limit on this endpoint.
3. Each created transaction enters `WAITING_FOR_SIGNATURES` status and remains there until its `validStart` passes the 3-minute expiry window. By using `validStart` values days or weeks in the future, the attacker keeps thousands of rows permanently active.
4. Any user (including legitimate users) calling `GET /transactions/sign` now triggers: a full-table scan returning all N attacker-created rows + N mirror-node HTTP calls + N deserialization operations. The request either times out or monopolises the Node.js event loop for the duration.
5. The endpoint is still a live, registered route despite the comment `/* NO LONGER USED BY FRONT-END */`; it is reachable by any API client.

### Impact Explanation

- **Service degradation / unavailability**: With O(N) mirror-node HTTP calls per request, response time grows linearly with the number of attacker-created transactions. At a few thousand rows the endpoint becomes effectively unusable for all users.
- **Scope is global**: Because the DB query is not scoped to the requesting user, a single attacker's transactions degrade the endpoint for every user in the organization.
- **Persistent**: Transactions with far-future `validStart` values remain active indefinitely, so the attacker does not need to keep sending requests — a one-time batch creation is sufficient.

### Likelihood Explanation

- Attacker precondition: a valid, verified user account — the lowest privilege level in the system.
- No special tooling required: a simple loop calling `POST /transactions` with incrementing timestamps.
- The `createTransaction` endpoint has no observable rate limit or per-user quota in the reviewed code.
- The attack is silent and persistent; it does not require the attacker to remain online after the initial batch.

### Recommendation

1. **Apply `take: limit` at the database level** in `getTransactionsToSign`, scoping the query to the requesting user's keys (join on `signers.userId` or `creatorKey.userId`) and enforcing the pagination limit before the per-row async work begins — matching the pattern already used correctly in `getTransactions` and `getTransactionsToApprove`.
2. **Enforce a per-user active-transaction quota** in `createTransaction` / `createTransactions` to bound the number of non-terminal transactions any single user can hold.
3. **Consider removing or formally deprecating** the `GET /transactions/sign` endpoint if it is no longer used by the front-end, to reduce attack surface.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
