## Audit Report

## Title
Non-Verified User Can Be Set as Transaction Approver, Causing Transactions to Become Stuck

## Summary
`createTransactionApprovers` and `updateTransactionApprover` in `approvers.service.ts` validate that a designated approver *exists* but do not validate that the user is *verified* (`status === UserStatus.NONE`). Because `VerifiedUserGuard` blocks any user whose status is not `NONE` from calling `POST /approve`, a transaction can be permanently stuck in `WAITING_FOR_SIGNATURES` if a non-verified user is set as a required approver.

## Finding Description

**Vulnerability class:** Incomplete prerequisite validation (state transition)

The system enforces a two-tier user status model defined in the `User` entity:

- `UserStatus.NEW` — default status; user exists but has not completed verification
- `UserStatus.NONE` — fully verified; can access protected endpoints [1](#0-0) [2](#0-1) 

`VerifiedUserGuard` enforces this at the controller level by returning `true` only when `user.status === UserStatus.NONE`: [3](#0-2) 

`ApproversController` applies this guard globally to all its routes via the class-level `@UseGuards` decorator, including `POST /approve`: [4](#0-3) [5](#0-4) 

However, when `createTransactionApprovers` adds a user as an approver, it only checks that the user *exists* (count > 0), not that the user's status is `NONE`: [6](#0-5) 

The same incomplete check exists in `updateTransactionApprover` when reassigning an approver to a different user: [7](#0-6) 

## Impact Explanation

A transaction requiring approval from a non-verified user (`UserStatus.NEW` or `UserStatus.RESET_PASSWORD`) will remain stuck in `WAITING_FOR_SIGNATURES` indefinitely. The non-verified approver is blocked by `VerifiedUserGuard` with HTTP 403 when attempting `POST /transactions/:id/approvers/approve`, so they can never fulfill their approver role. If the Hedera transaction's `validStart` window expires before the misconfiguration is detected and corrected, the transaction expires without execution. Recovery requires the creator to manually detect the issue, remove the broken approver, and re-add a verified one — a non-obvious recovery path. [8](#0-7) 

## Likelihood Explanation

This is realistic in organization mode: an admin invites a new user, the user registers (defaulting to `UserStatus.NEW`), and a transaction creator immediately adds them as an approver before they complete account setup. The creator receives no warning that the designated approver cannot yet act. The test suite explicitly creates a `userNew` with `UserStatus.NEW` and confirms it is blocked from all verified-user endpoints: [9](#0-8) 

## Recommendation

Add a `status` filter to the user existence checks in both `createTransactionApprovers` and `updateTransactionApprover` so that only verified users (`UserStatus.NONE`) can be designated as approvers.

In `createTransactionApprovers`:
```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
const userCount = await transactionalEntityManager.count(User, {
  where: { id: dtoApprover.userId, status: UserStatus.NONE },
});
if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
```

Apply the same fix to the equivalent block in `updateTransactionApprover` (lines 494–497). A dedicated error message such as `USER_NOT_VERIFIED` would improve debuggability over reusing `USER_NOT_FOUND`.

## Proof of Concept

1. Register a new user (`userNew`) — their status defaults to `UserStatus.NEW`.
2. As a verified transaction creator, call `POST /transactions/:id/approvers` with `{ userId: userNew.id }`. The request succeeds because only existence is checked.
3. The transaction enters `WAITING_FOR_SIGNATURES` requiring `userNew`'s approval.
4. `userNew` calls `POST /transactions/:id/approvers/approve`. `VerifiedUserGuard` evaluates `user.status === UserStatus.NONE` → `false` → HTTP 403.
5. The transaction remains permanently stuck in `WAITING_FOR_SIGNATURES` until the creator manually removes and replaces the approver. [10](#0-9) [6](#0-5) [7](#0-6)
