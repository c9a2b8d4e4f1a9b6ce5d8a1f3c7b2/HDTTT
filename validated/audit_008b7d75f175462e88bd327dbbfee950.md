### Title
`FileDeleteTransaction` Missing from Signature Model Lookup Table Causes Permanent Transaction Lock

### Summary
Both the back-end and front-end `TransactionFactory` classes maintain a lookup table mapping transaction type strings to signature model constructors. `FileDeleteTransaction` is recognized as a valid type throughout the system (present in the `TransactionType` enum and handled in `getTransactionType()`), but is absent from both lookup tables. When a `FileDeleteTransaction` is submitted to the organization workflow, the factory throws an unhandled error during signature key computation, permanently locking the transaction in `WAITING_FOR_SIGNATURES` state with no recovery path.

---

### Finding Description

**Root cause — incomplete lookup table, analog to the external report:**

The `getTransactionType()` function in both back-end and front-end explicitly handles `FileDeleteTransaction`: [1](#0-0) 

```
} else if (transaction instanceof FileDeleteTransaction) {
  transactionType = "File Delete Transaction";
```

With `short = true` (the flag used by both factories), this returns `"FileDeleteTransaction"`.

`getTransactionTypeEnumValue()` also maps it: [2](#0-1) 

And the `TransactionType` entity enum explicitly includes it: [3](#0-2) 

**However**, the back-end `TRANSACTION_MODEL_MAP` has 14 entries and `FileDeleteTransaction` is absent: [4](#0-3) 

When `fromTransaction()` is called with a `FileDeleteTransaction`, `TRANSACTION_MODEL_MAP.get("FileDeleteTransaction")` returns `undefined`, and the factory throws: [5](#0-4) 

The same gap exists in the front-end factory: [6](#0-5) 

**Exploit path:**

`TransactionFactory.fromTransaction()` is called unconditionally inside `TransactionSignatureService.computeSignatureKey()`: [7](#0-6) 

The thrown error is not caught in `computeSignatureKey` or in `keysRequiredToSign`: [8](#0-7) 

The error propagates to the API handler, returning HTTP 500. The transaction remains in `WAITING_FOR_SIGNATURES` indefinitely because the system can never successfully compute required signers.

---

### Impact Explanation

Any `FileDeleteTransaction` submitted to the organization workflow is permanently locked in `WAITING_FOR_SIGNATURES`. The system cannot compute required signers, cannot advance the transaction to `WAITING_FOR_EXECUTION`, and cannot execute or cancel it through normal flows. This is an unrecoverable corruption of transaction state — a direct match to the RESEARCHER.md impact category "Permanent lock, freeze, or unrecoverable corruption of user/project state."

---

### Likelihood Explanation

The back-end API accepts raw `transactionBytes` from any authenticated user. `FILE_DELETE` is an explicitly declared `TransactionType` in the database entity, signaling it is an intended supported type. Any organization member who submits a `FileDeleteTransaction` (e.g., to delete a Hedera file via the multi-sig workflow) will trigger this path. No privilege escalation or leaked credentials are required — only a valid organization account.

---

### Recommendation

Add a `FileDeleteTransactionModel` class (analogous to the existing `FileUpdateTransactionModel`) to both the back-end and front-end lookup tables:

**Back-end** (`back-end/libs/common/src/transaction-signature/model/transaction-factory.ts`): add `['FileDeleteTransaction', FileDeleteTransactionModel]` to `TRANSACTION_MODEL_MAP`.

**Front-end** (`front-end/src/renderer/utils/transactionSignatureModels/transaction-factory.ts`): add `FileDeleteTransaction: FileDeleteTransactionModel` to `transactionModelMap`.

The model's `getSigningAccounts()` should return the file's admin key account ID (the account that owns the file being deleted), consistent with how Hedera's `FileDeleteTransaction` requires the file's key to sign.

---

### Proof of Concept

1. Authenticate as an organization member via the back-end API.
2. Construct a `FileDeleteTransaction` using the Hedera SDK targeting any file ID, freeze it, and serialize to bytes.
3. Submit the transaction bytes to the back-end `POST /transactions` endpoint. The system accepts it because `FILE_DELETE` is a valid `TransactionType`.
4. The transaction enters `WAITING_FOR_SIGNATURES` state.
5. Any subsequent call that invokes `computeSignatureKey` for this transaction (e.g., fetching required signers, checking signability) triggers `TransactionFactory.fromTransaction()` → `TRANSACTION_MODEL_MAP.get("FileDeleteTransaction")` returns `undefined` → throws `Error: No transaction model registered for type: FileDeleteTransaction`.
6. The transaction is permanently stuck; no signer can be computed, no execution path exists, and the state cannot advance.
