### Title
Bulk Approval in `ApproveAllController.vue` Executes Cryptographic Signatures Over All Group Transactions Without Displaying Individual Transaction Parameters

### Summary
The `ApproveAllController.vue` component allows an approver in Organization Mode to cryptographically sign and approve every transaction in a group with a single click, presenting only a generic "Are you sure you want to approve all transactions?" prompt. No individual transaction parameters — recipient account, transfer amount, memo, or transaction type specifics — are shown before the signatures are produced and submitted. A malicious transaction creator (a normal organization user) can embed a harmful transaction (e.g., a large HBAR transfer to an attacker-controlled account) inside a group alongside legitimate transactions, and an approver using "Approve All" will unknowingly sign it.

### Finding Description

**Root cause — `ApproveAllController.vue`**

The `handleApproveAll` handler (lines 57–111) performs the following steps silently, with no per-transaction display:

1. Fetches the full group from the server (`getTransactionGroupById`).
2. Decrypts the approver's private key.
3. Iterates over every `groupItem`, deserializes `transactionBytes`, computes a cryptographic signature over the transaction body, and calls `sendApproverChoice` — all without surfacing any transaction-specific field to the user. [1](#0-0) 

The only user-facing confirmation text is the static string: [2](#0-1) 

**What the group list page shows**

`TransactionGroupDetails.vue` renders a table with columns: Transaction ID, Transaction Type, Status, Valid Start. Recipient accounts, transfer amounts, memos, and other payload fields are never shown in this view; they are only accessible by navigating into each individual transaction's detail page. [3](#0-2) 

**Attacker-controlled entry point**

Any authenticated organization user can create a transaction group via the API (`POST /transaction-groups`) and supply arbitrary `transactionBytes` for each item. The `CreateTransactionDto` accepts raw `transactionBytes` as a `Buffer` with no semantic validation of the payload beyond byte-level integrity. [4](#0-3) 

The `TransactionDto` serializes `transactionBytes` back to the client as a hex string, which `ApproveAllController.vue` decodes and signs verbatim. [5](#0-4) 

**Contrast with individual approval**

When an approver navigates to a single transaction's detail page, `TransactionDetails.vue` decodes the SDK transaction and renders type, transaction ID, valid start, fee payer, memo, and a transaction-specific component (recipient, amount, etc.) before the "Sign" button is reachable. [6](#0-5) 

The "Approve All" path entirely bypasses this display.

### Impact Explanation

An approver who uses "Approve All" unknowingly provides a valid cryptographic approval signature over every transaction in the group, including any malicious one. Once the approval threshold is met, the back-end chain service submits the transaction to the Hedera network. Concrete outcomes:

- **Direct HBAR theft**: A `TransferTransaction` moving HBAR from an organization-controlled account to the attacker's account is approved and executed.
- **Token allowance escalation**: An `ApproveHbarAllowanceTransaction` granting the attacker unlimited spending rights over an organization account is approved silently.
- **Account key replacement**: An `AccountUpdateTransaction` replacing the account's key with the attacker's key is approved, permanently locking out the legitimate owner.

All of these are irreversible on-chain once executed.

### Likelihood Explanation

The attacker must be an authenticated member of the target organization — a normal user with no elevated privileges, only the ability to create transactions. This is a realistic insider-threat scenario in any multi-user organization deployment. The "Approve All" button is a primary action on the group details page and is the natural path for approvers managing high-volume transaction groups. No special tooling is required beyond the standard API documented in `front-end/docs/api.md`. [7](#0-6) 

### Recommendation

Before executing any signature loop in `handleApproveAll`, render a per-transaction summary modal that decodes each `transactionBytes` via `Transaction.fromBytes` and displays at minimum: transaction type, recipient/target account, transfer amount or allowance, and transaction memo. Require explicit per-transaction acknowledgement, or at minimum a scrollable list of decoded summaries that the approver must review before the single "Confirm All" action is enabled. This mirrors the detail already available in `TransactionDetails.vue` and closes the gap introduced by the bulk path.

### Proof of Concept

1. Attacker authenticates to the organization server and obtains a valid JWT.
2. Attacker constructs two transactions using the Hiero SDK:
   - **TX-A** (legitimate): a routine `AccountUpdateTransaction` with innocuous parameters.
   - **TX-B** (malicious): a `TransferTransaction` sending 1000 HBAR from an organization account (`0.0.X`) to the attacker's account (`0.0.ATTACKER`).
3. Attacker submits a transaction group via `POST /transaction-groups` containing both TX-A and TX-B, adds the victim approver as an approver for both.
4. Victim approver receives a notification, navigates to the group details page, sees only "Account Update" and "Transfer" in the type column with no amounts or recipients visible.
5. Victim clicks "Approve All" → confirms the generic dialog → enters their application password.
6. `handleApproveAll` iterates both items, signs TX-B's body with the victim's private key, and calls `sendApproverChoice` with `approved: true`.
7. Once the approval threshold is satisfied, the chain service executes TX-B on the Hedera network, transferring 1000 HBAR to the attacker's account. [8](#0-7)

  if (!isUserLoggedIn(user.personal)) {

    try {
      let group: IGroup;
      if (typeof props.groupOrId === 'number') {
        const serverUrl = user.selectedOrganization.serverUrl;
        group = await getTransactionGroupById(serverUrl, props.groupOrId);
      } else {
        group = props.groupOrId;
      }

      progressText.value = `${props.approved ? 'Approving' : 'Rejecting'} ${group.groupItems.length} transactions`;

      const publicKey = user.selectedOrganization.userKeys[0].publicKey;
