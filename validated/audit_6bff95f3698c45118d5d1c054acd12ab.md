### Title
OTP Verification Endpoint (`/auth/verify-reset`) Has No Rate Limiting, Enabling Brute-Force of Password Reset Codes

### Summary
The `/auth/verify-reset` endpoint, which verifies an 8-digit TOTP for password reset, has no rate limiting of any kind. An attacker who triggers a password reset for a victim can brute-force the OTP without restriction, ultimately taking over the victim's account.

### Finding Description

The `POST /auth/verify-reset` endpoint is protected only by `JwtBlackListOtpGuard` and `OtpJwtAuthGuard` — neither of which applies any rate limiting: [1](#0-0) 

Compare this to `/auth/reset-password`, which at least applies `EmailThrottlerGuard`: [2](#0-1) 

The OTP is an 8-digit TOTP generated with `otplib`, configured with a 60-second step and a `window` of 20, meaning the same code is accepted across ±20 windows (up to ~20 minutes of validity): [3](#0-2) 

The verification itself is a simple `totp.check()` call with no attempt counter or lockout: [4](#0-3) 

An attacker who calls `POST /auth/reset-password` for a victim's email receives a valid OTP JWT. They can then call `POST /auth/verify-reset` with that JWT and cycle through OTP guesses at full network speed. With `window: 20`, the same 8-digit code is valid for an extended period, giving the attacker a large time budget.

A secondary, related issue exists on `/auth/login`: `LocalAuthGuard` (which triggers the expensive `bcrypt`/`argon2` password comparison) is listed **before** `EmailThrottlerGuard` in the guard chain: [5](#0-4) 

In NestJS, guards execute in declaration order, so the expensive cryptographic work occurs before the rate limit is ever checked. Furthermore, `EmailThrottlerGuard` keys on the email address from the request body: [6](#0-5) 

An attacker using different email addresses per request bypasses the per-email throttle entirely while still forcing the server to perform bcrypt/argon2 comparisons.

### Impact Explanation
For the OTP brute-force path: an attacker can take over any user account whose password reset they trigger. With no rate limiting on `/auth/verify-reset`, the 8-digit space (100,000,000 values) is reachable within the OTP validity window at typical HTTP request rates. A successful guess yields a `verified: true` OTP JWT, which then allows the attacker to call `/auth/set-password` and permanently set a new password for the victim. [7](#0-6) 

### Likelihood Explanation
The attack requires only knowledge of a victim's email address (which is the username for the system) and the ability to send HTTP requests. No privileged access is needed. The attacker initiates the flow themselves via the public `/auth/reset-password` endpoint. The extended TOTP window (`window: 20`) gives a large time budget for brute-forcing.

### Recommendation

1. **Add rate limiting to `/auth/verify-reset`**: Apply `EmailThrottlerGuard` (or a stricter IP-based guard) to this endpoint, and increment the counter on every failed OTP attempt — not just on success.
2. **Reduce the TOTP window**: `window: 20` is very permissive. Reducing it to 1–2 significantly shrinks the brute-force window.
3. **Fix guard ordering on `/auth/login`**: Move `EmailThrottlerGuard` before `LocalAuthGuard` so rate limiting is enforced before the expensive bcrypt/argon2 comparison runs.
4. **Consider IP-based rate limiting** in addition to email-based, since the current `EmailThrottlerGuard` is trivially bypassed by rotating email addresses.

### Proof of Concept

```
# Step 1: Trigger password reset for victim — attacker receives OTP JWT in response
POST /auth/reset-password
{ "email": "victim@example.com" }
→ 200 OK  (EmailThrottlerGuard limits this, but attacker only needs to do it once)

# Step 2: Brute-force OTP with no rate limiting
# The OTP JWT from step 1 is placed in the 'otp' header.
# Loop through all 8-digit values:
for code in 00000000..99999999:
    POST /auth/verify-reset
    Headers: otp: <jwt_from_step1>
    Body: { "token": "<code>" }
    → 401 Unauthorized (wrong code) — no throttle, no lockout
    → 200 OK (correct code) → attacker now holds a verified OTP JWT

# Step 3: Set a new password for the victim
PATCH /auth/set-password
Headers: otp: <verified_jwt>
Body: { "password": "attacker_controlled_password" }
→ 200 OK — account taken over
``` [8](#0-7)

    const secret = this.getOtpSecret(user.email);

    if (!totp.check(token, secret)) throw new UnauthorizedException('Incorrect token');

    try {
      await this.usersService.updateUser(user, { status: UserStatus.NEW });
      const token = this.getOtpToken({ email: user.email, verified: true });
      return { token };
    } catch {
      throw new InternalServerErrorException('Error while updating user status');
    }
  }
