### Title
`removeTransactionApprover` Does Not Adjust Parent Threshold, Violating `threshold <= children.length` Invariant

### Summary

The `ApproversService.removeTransactionApprover()` function soft-deletes a child approver node from a threshold tree without updating the parent node's `threshold` value. This is the direct analog of the reported MultiSig bug: a removal operation reduces the effective member count without adjusting the required threshold, breaking the invariant `threshold <= children.length`. The `updateTransactionApprover()` function correctly handles this adjustment when detaching a child, but `removeTransactionApprover()` does not, leaving the parent tree node in a state where its approval condition can never be satisfied.

### Finding Description

The system enforces the invariant `threshold <= approvers.length` at creation time and during `updateTransactionApprover` (detach path). However, `removeTransactionApprover` bypasses this adjustment entirely.

**Creation-time enforcement** (enforced correctly): [1](#0-0) 

**`updateTransactionApprover` detach path** (correctly adjusts parent threshold when a child is detached via `listId=null`): [2](#0-1) 

**`removeTransactionApprover`** (the vulnerable path — no parent threshold adjustment): [3](#0-2) 

`removeNode()` only soft-deletes the target node and its subtree recursively. It never touches the parent: [4](#0-3) 

The controller's `DELETE /:id` endpoint accepts any approver `id` — not just root nodes — and calls `removeTransactionApprover` directly after verifying the caller is the transaction creator: [5](#0-4) 

**Exploit flow**:
1. Creator creates a transaction with a threshold tree: `Parent { threshold: 3, children: [A, B, C] }`.
2. Creator calls `DELETE /transactions/:txId/approvers/:A.id`.
3. `removeTransactionApprover(A.id)` → `removeNode(A.id)` soft-deletes A.
4. DB state: `Parent { threshold: 3, children: [B, C] }` — threshold (3) > children count (2).
5. The `isApproved` logic (front-end) and the chain service's approval check will never see 3 approvals from 2 remaining approvers.

The `isApproved` utility confirms the threshold is used directly against the count of approved children: [6](#0-5) 

### Impact Explanation

After the removal, the parent threshold node permanently requires more approvals than there are approvers. The transaction can never accumulate enough approvals to advance from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION`. This constitutes a **permanent lock of the transaction's approval flow** — the transaction is frozen in an unresolvable state unless the creator manually corrects the threshold via a separate `PATCH` call. If the creator does not realize the invariant is broken, the transaction is effectively dead.

**Severity: Medium** — state invariant violation causing a permanent approval deadlock on the affected transaction.

### Likelihood Explanation

The trigger is a normal, authenticated workflow available to any transaction creator. No privileged access, leaked credentials, or external dependencies are required. The creator only needs to:
- Create a transaction with a threshold tree where `threshold == children.length` (e.g., 3-of-3).
- Call `DELETE /transactions/:txId/approvers/:childId` for any one child.

This is a realistic accidental or intentional action within the normal product workflow.

### Recommendation

In `removeTransactionApprover`, after soft-deleting the target node, fetch the parent (if any) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id); // includes relations: ['approvers']
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // NEW: adjust parent threshold before deletion
  if (approver.listId) {
    const parent = await this.repo.findOne(TransactionApprover, {
      relations: ['approvers'],
      where: { id: approver.listId },
    });
    if (parent) {
      const newChildCount = parent.approvers.length - 1;
      if (newChildCount === 0) {
        await this.repo.softRemove(parent);
      } else if (newChildCount < parent.threshold) {
        await this.repo.update(parent.id, { threshold: newChildCount });
      }
    }
  }

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
}
```

This mirrors the existing fix already applied in `updateTransactionApprover` at lines 417–428. [2](#0-1) 

### Proof of Concept

**Setup**:
1. Authenticate as a transaction creator (User A).
2. Create a transaction (txId = 1).
3. `POST /transactions/1/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 3, "approvers": [{"userId":2},{"userId":3},{"userId":4}] }] }
   ```
   This creates: `Parent { id: 10, threshold: 3, children: [id:11, id:12, id:13] }`.

**Trigger**:
4. `DELETE /transactions/1/approvers/11` (remove child approver id=11).

**Observed state**:
- `transaction_approver` table: `id=10` still has `threshold=3`, but only children `id=12` and `id=13` remain (non-deleted).
- `threshold (3) > children.length (2)` — invariant violated.

**Result**:
- Users 3 and 4 both approve → `approvals.length = 2`, but `approver.threshold = 3` → `isApproved` returns `null` (neither approved nor rejected).
- Transaction is permanently stuck in `WAITING_FOR_SIGNATURES` with no path to `WAITING_FOR_EXECUTION`.

              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"

      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
