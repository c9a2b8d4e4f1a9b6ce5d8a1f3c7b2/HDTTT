Audit Report

## Title
`getTransactionsToSign` Performs Unbounded System-Wide DB Fetch with O(N) Per-Transaction Work, Enabling Authenticated Resource Exhaustion

## Summary

`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction in the system without a row limit, then performs CPU-intensive deserialization and multiple async cache-DB calls per transaction before applying pagination. An authenticated user who creates a large number of transactions can cause this endpoint to exhaust server resources for all callers.

## Finding Description

**Unbounded fetch — confirmed:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–299, the query has no `take` limit:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // status filter only — no user scope, no LIMIT
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The `whereForUser` filter excludes only five terminal statuses and is not scoped to the requesting user, so the query returns every active transaction in the system. [2](#0-1) 

**Pagination applied after full in-memory scan — confirmed:** [3](#0-2) 

**Per-transaction work — confirmed:**

For every fetched transaction, `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` is called: [4](#0-3) 

Inside `keysRequiredToSign`, each iteration performs:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU deserialization
2. `transactionSignatureService.computeSignatureKey(transaction)` — calls `accountCacheService.getAccountInfoForTransaction` per signing account, which issues a `findOne(CachedAccount, ...)` against the cache DB and an idempotent `INSERT INTO TransactionCachedAccount` per account [5](#0-4) [6](#0-5) 

**Correction to the original report:** The claim that `entityManager.find(UserKey, ...)` is called per transaction is inaccurate. When `userKeys` is provided (as it is from `userKeysRequiredToSign`), `keysRequiredToSign` filters the user's keys in memory with no DB call. [7](#0-6) 

Similarly, `attachKeys` is idempotent — it only queries the DB when `user.keys` is empty, and since it is called once before the loop, subsequent calls inside the loop are no-ops. [8](#0-7) 

**No rate limiting on the transactions endpoint — confirmed:**

The throttler guards (`UserThrottlerGuard`, `IpThrottlerGuard`, `EmailThrottlerGuard`) are only applied to auth endpoints. The `TransactionsController` and `POST /transactions` carry no throttler guard. [9](#0-8) 

**No per-user transaction cap — confirmed:**

`createTransactions` validates transaction bytes, checks for duplicates by `transactionId`, and enforces no per-user count limit. [10](#0-9) 

## Impact Explanation

- **Service unavailability for all users:** `GET /transactions/sign` becomes unresponsive for every authenticated caller, not just the attacker. The endpoint is the core signing workflow.
- **Cache DB pressure:** Each request spawns O(N) sequential async calls against the cache DB (`CachedAccount` lookups and `TransactionCachedAccount` inserts). Under concurrent load this saturates the connection pool.
- **CPU exhaustion:** Deserializing N transaction byte blobs sequentially on the Node.js event loop blocks the request thread.
- **Durable degradation:** Attacker-created transactions persist in `WAITING_FOR_SIGNATURES` indefinitely if `validStart` is set far in the future, requiring admin intervention to clean up.

## Likelihood Explanation

- Only a valid user account is required — the lowest privilege level.
- `POST /transactions` is a normal product flow with no observed rate limit or per-user cap.
- The attack is cheap: creating thousands of transactions costs only API calls, not on-chain fees.
- The vulnerable endpoint, while marked `/* NO LONGER USED BY FRONT-END */`, remains a live, authenticated API route accessible to any API client. [11](#0-10) 

## Recommendation

1. **Push pagination to the DB layer:** Add `take: limit` and `skip: offset` to the `repo.find` call, and restructure the logic so the user-key check is done in a single query or batched, rather than per-row in application code.
2. **Scope the query to the requesting user:** Pre-filter transactions to those where the user's public keys appear in `publicKeys` or related signing structures, eliminating irrelevant rows before deserialization.
3. **Add a per-user transaction creation cap** or rate-limit `POST /transactions` with the existing `UserThrottlerGuard`.
4. **Batch `computeSignatureKey` calls** or cache results within a single request to avoid redundant cache-DB round-trips for transactions sharing the same fee payer account.

## Proof of Concept

```
# Step 1: Authenticate as a normal user and obtain JWT
POST /auth/login  →  { token }

# Step 2: Create N transactions with validStart far in the future
for i in 1..5000:
    POST /transactions
    Body: { transactionBytes: <valid_bytes_with_future_validStart>, creatorKeyId: <id>, ... }

# Step 3: Any user calls the signing endpoint
GET /transactions/sign?page=1&size=10

# Result: server fetches all 5000+ transactions, deserializes each one,
# issues multiple cache-DB queries per transaction, then slices to 10 items.
# Under load, the request times out or the DB connection pool is exhausted,
# returning errors to all concurrent callers.
```
