Audit Report

## Title
Cross-Transaction Approver Deletion via Missing Ownership Verification in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user owns the `transactionId` in the URL, but then deletes the approver looked up solely by its `id` — with no check that the approver actually belongs to that transaction. Any authenticated user who has created at least one transaction can delete approvers from any other transaction by supplying an arbitrary approver `id`.

## Finding Description

**Entry point:** `DELETE /transactions/:transactionId/approvers/:id`

In `approvers.controller.ts`, the handler performs two completely decoupled operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // only proves user owns transactionId from URL
await this.approversService.removeTransactionApprover(id);               // deletes approver by id alone
``` [1](#0-0) 

`removeTransactionApprover` fetches the approver by `id` with no `transactionId` filter and immediately soft-deletes it: [2](#0-1) 

`getTransactionApproverById` queries only on `id`:

```typescript
const find: FindOneOptions<TransactionApprover> = {
  relations: ['approvers'],
  where: { id },
};
``` [3](#0-2) 

`removeNode` then recursively soft-deletes the target node and every descendant via a CTE `UPDATE`: [4](#0-3) 

`getCreatorsTransaction` only verifies the caller created the transaction identified by `transactionId` in the URL — it says nothing about whether the approver `id` in the path belongs to that transaction: [5](#0-4) 

The fix is already demonstrated in the codebase's own `updateTransactionApprover`, which correctly resolves the root node and checks `rootNode.transactionId !== transactionId` before proceeding: [6](#0-5) 

This cross-check is entirely absent from the delete path.

## Impact Explanation

A malicious transaction creator can permanently soft-delete any `TransactionApprover` node (and its entire subtree) from any other user's transaction. Because `TransactionApprover` nodes represent the multi-signature approval tree for Hedera transactions:

- Required signers are eliminated from another organization's transaction, allowing it to proceed without the intended approvals.
- The soft-delete is not reversed by normal flows, making the corruption permanent.
- If a parent node's child count drops below its `threshold`, the transaction can never reach the required approval count, permanently blocking execution.

## Likelihood Explanation

- **Attacker precondition:** Any authenticated user who has created at least one transaction. No admin or privileged role required.
- **Target discovery:** `TransactionApprover` IDs are sequential auto-incremented integers (`@PrimaryGeneratedColumn()`), making them trivially enumerable by incrementing the `:id` path parameter. [7](#0-6) 
- **Trigger:** A single crafted `DELETE` request with a valid `transactionId` the attacker owns and an arbitrary target approver `id`.
- **Detection difficulty:** The operation appears as a normal approver deletion in logs; the only anomaly is the mismatch between `transactionId` in the URL and the actual transaction the approver belongs to.

## Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and verify its `transactionId` matches the `transactionId` from the URL — exactly as `updateTransactionApprover` already does:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to the specified transaction');
}
```

This mirrors the existing guard at: [6](#0-5) 

## Proof of Concept

1. Attacker (user A) creates transaction `T_A` — this satisfies `getCreatorsTransaction`.
2. Attacker enumerates approver IDs (sequential integers) and identifies approver `id=42` belonging to victim's transaction `T_B`.
3. Attacker sends:
   ```
   DELETE /transactions/{T_A}/approvers/42
   ```
4. `getCreatorsTransaction(T_A, userA)` passes — user A owns `T_A`.
5. `removeTransactionApprover(42)` fetches approver 42 (which belongs to `T_B`) with no ownership check and soft-deletes it along with its entire subtree.
6. The approval tree of `T_B` is permanently corrupted without the victim's knowledge.
