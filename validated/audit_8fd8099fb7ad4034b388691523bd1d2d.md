# Audit Report

## Title
Missing Cryptographic Signature Validation and Access Control in `uploadSignatureMaps` Allows Any Authenticated User to Corrupt Transaction Bytes

## Summary
The `uploadSignatureMaps` function in `signers.service.ts` accepts caller-supplied `SignatureMap` data, verifies only that each submitted public key belongs to the authenticated user, and then unconditionally appends the (potentially invalid) signature bytes to the stored transaction via `addSignature`. No cryptographic verification of the signature bytes is performed, and no access-control check confirms the submitting user has any relationship to the target transaction. Any authenticated, verified user can corrupt the `transactionBytes` of any in-flight transaction, causing it to be permanently rejected by the Hedera network.

## Finding Description

**Root cause 1 — Missing cryptographic validation:**

`processTransactionSignatures` in `signers.service.ts` iterates the caller-supplied `SignatureMap`, confirms each public key belongs to the authenticated user via `userKeyMap`, and then calls `sdkTransaction.addSignature(publicKey, map)` with no cryptographic check on the signature bytes: [1](#0-0) 

The Hedera SDK's `addSignature` appends the raw bytes to the inner signed transactions without performing `publicKey.verify(bodyBytes, signature)`. That verification is left to the network at submission time.

The `validateSignature` utility in `transaction.ts` performs exactly this check and throws on failure: [2](#0-1) 

The `importSignatures` path in `transactions.service.ts` calls `validateSignature` before any `addSignature`: [3](#0-2) 

The `/signers` path has no equivalent call.

**Root cause 2 — Missing access control:**

`loadTransactionData` fetches transactions by ID with no user filter: [4](#0-3) 

`validateAndProcessSignatures` performs no `verifyAccess` call at any point: [5](#0-4) 

By contrast, `importSignatures` gates every transaction on `verifyAccess` before proceeding: [6](#0-5) 

**Persistence of corrupted bytes:**

When `addSignature` appends invalid bytes, the resulting serialized transaction differs from the original, so `isSameBytes` is `false`. `persistSignatureChanges` then writes the corrupted bytes to the database: [7](#0-6) 

**Execution-time failure:**

`getValidatedSDKTransaction` in `execute.service.ts` checks `hasValidSignatureKey` against `_signerPublicKeys`. Because `addSignature` adds the public key to `_signerPublicKeys` regardless of whether the signature bytes are valid, this check passes. The transaction is then submitted to the Hedera network, which performs the actual cryptographic verification and rejects it, setting the status to `FAILED`: [8](#0-7) 

The endpoint is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no role or ownership check: [9](#0-8) 

## Impact Explanation

- **Permanent DoS on any in-flight transaction**: Once corrupted `transactionBytes` are persisted, the transaction cannot be recovered. `FAILED` is a terminal status in `terminalStatuses`.
- **False signer records**: The attacker is recorded as a `TransactionSigner`, corrupting audit trails and potentially satisfying threshold counts prematurely.
- **No asset recovery path**: Hedera transactions are immutable once submitted; a failed transaction cannot be re-executed with the same transaction ID.
- **Scope**: Every transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status is reachable by any authenticated user, since transaction IDs are sequential integers and no ownership filter is applied.

## Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (registered, verified account) and at least one registered public key. No admin keys, no leaked credentials, no privileged role required.
- **Transaction ID enumeration**: IDs are sequential integers, trivially enumerable.
- **No rate limiting or anomaly detection** is visible on this endpoint.
- The attack is fully automated and leaves no obvious trace beyond the corrupted bytes and a spurious `TransactionSigner` record.

## Recommendation

1. **Add cryptographic signature validation** in `processTransactionSignatures` by calling `validateSignature(sdkTransaction, map)` before any `addSignature` call, mirroring the pattern already used in `importSignatures`.
2. **Add access control** in `validateAndProcessSignatures` (or `loadTransactionData`) by calling `verifyAccess(transaction, user)` for each transaction before processing, mirroring the pattern in `importSignatures`.

## Proof of Concept

```
1. Register a user account and obtain a JWT (standard registration flow).
2. Register any public key (e.g., a freshly generated Ed25519 key) to the account.
3. Enumerate transaction IDs (e.g., id=1, 2, 3, ...) to find a transaction in
   WAITING_FOR_SIGNATURES or WAITING_FOR_EXECUTION status.
4. Construct a SignatureMap where:
     nodeAccountId  → transactionId → <attacker's registered DER public key>
                                        → <64 zero bytes (invalid signature)>
5. POST to /transactions/signers with body:
     [{ "id": <target_id>, "signatureMap": <constructed map> }]
   using the attacker's JWT.
6. Observe HTTP 201 response with a TransactionSigner record for the attacker.
7. Observe in the database that transactionBytes for <target_id> now contains
   the invalid signature appended by addSignature.
8. When the chain service attempts execution, hasValidSignatureKey passes
   (attacker's key is in _signerPublicKeys), the transaction is submitted,
   the Hedera network rejects it (INVALID_SIGNATURE), and the status is
   permanently set to FAILED.
```

  private async getValidatedSDKTransaction(
    transaction: Transaction,
  ): Promise<SDKTransaction> {
    /* Throws an error if the transaction is not found or in incorrect state */
    if (!transaction) throw new Error('Transaction not found');

    await this.validateTransactionStatus(transaction);

    /* Gets the SDK transaction from the transaction body */
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Gets the signature key */
    const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);

    /* Checks if the transaction has valid signatureKey */
    if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
      throw new Error('Transaction has invalid signature.');

    return sdkTransaction;
  }
