Audit Report

## Title
Decrypted Private Key Persists in Renderer DOM Without Timeout and Is Accessible Without Re-Authentication During Cached Password Window

## Summary
Once a user authenticates with their password for any operation, the password is cached in the Pinia store for a 10-minute sliding window. During this window, clicking the "show private key" eye icon in `Settings > Keys` decrypts and renders the raw private key in the DOM **without prompting for the password again**. The decrypted key is stored in a Vue reactive ref with **no expiry timeout** — it remains in the DOM until the user manually hides it. A same-user process (e.g., malware installed via phishing) can exploit this via OS accessibility APIs or screenshot capture to steal the private key without physical access.

## Finding Description

**Root cause 1 — 10-minute sliding password cache:**

`storeUser.ts` initializes `passwordStoreDuration` to 10 minutes: [1](#0-0) 

`getPassword()` returns the cached password and resets the expiry on every call (sliding window): [2](#0-1) 

`isLoggedInWithValidPassword()` in `userStoreHelpers.ts` only checks whether `passwordExpiresAt` is in the past: [3](#0-2) 

**Root cause 2 — `usePersonalPassword.getPassword()` bypasses the modal when password is cached:**

If `user.getPassword()` returns a non-null value, the function returns it immediately without opening the password modal: [4](#0-3) 

**Root cause 3 — `KeysTab.vue` `handleShowPrivateKey` → `decrypt()` uses cached password with no additional auth gate:** [5](#0-4) 

`decrypt()` calls `getPassword()` and, if the password is cached, proceeds directly to decrypt and push the raw key into `decryptedKeys`: [6](#0-5) 

**Root cause 4 — Decrypted key rendered in DOM with no timeout:**

`decryptedKeys` is a plain reactive ref with no expiry mechanism: [7](#0-6) 

The template renders the raw private key string directly from `decryptedKeys` and keeps it there until the user manually clicks the hide icon. The eye icon and private key span both carry predictable `data-testid` attributes: [8](#0-7) 

## Impact Explanation
An attacker who obtains the raw ED25519 or ECDSA private key can sign arbitrary Hedera transactions — HBAR transfers, token operations, account updates — and drain all assets. The impact is **critical**: direct, irreversible theft of funds and full account takeover. No password or further credential is needed after the key is exfiltrated.

## Likelihood Explanation
Requires: (a) malware installed on the victim's machine (achievable remotely via phishing or drive-by download, no physical access needed), and (b) the user to have authenticated within the past 10 minutes (a normal usage pattern — any signing operation triggers this). On Windows and Linux, same-user processes can access the accessibility tree and capture screenshots without elevated privileges. The `data-testid` attributes (`span-show-modal-${index}` and `span-private-key-${index}`) make programmatic targeting trivial. Likelihood is **medium-high** for users who are actively using the application.

## Recommendation

1. **Require re-authentication before revealing a private key.** The `decrypt()` function in `KeysTab.vue` should always prompt for the password regardless of the cache state. This is a high-sensitivity operation that warrants a dedicated auth gate, separate from the general-purpose password cache.

2. **Auto-clear decrypted keys from the DOM.** After pushing a decrypted key into `decryptedKeys`, start a short `setTimeout` (e.g., 30–60 seconds) that calls `handleHideDecryptedKey` automatically. This limits the window during which the key is readable in the DOM.

3. **Reduce or eliminate the sliding window for the password cache.** The current design resets the 10-minute timer on every `getPassword()` call, meaning the password can remain cached indefinitely during active use. Consider a fixed (non-sliding) expiry, or a shorter duration for sensitive operations.

4. **Remove or obfuscate `data-testid` attributes in production builds** for elements that expose sensitive key material, to reduce the ease of programmatic targeting.

## Proof of Concept

1. User opens the app and signs a transaction → `setPassword()` is called, `passwordExpiresAt` is set to `now + 10 min`.
2. Malware (same OS user, no elevated privileges) uses Windows UI Automation or Linux AT-SPI to enumerate the Electron window.
3. Malware locates the element with `data-testid="span-show-modal-0"` and programmatically fires a click event.
4. `handleShowPrivateKey` fires → `decrypt()` calls `getPassword()` → `isLoggedInWithValidPassword()` returns `true` → cached password returned → IPC call to `decryptPrivateKey` in main process → raw private key pushed into `decryptedKeys.value`.
5. Vue renders the key into `span[data-testid="span-private-key-0"]`.
6. Malware reads the DOM text content via the accessibility tree, or captures a screenshot.
7. Raw private key is exfiltrated → complete Hedera account compromise.

export const isLoggedInWithValidPassword = (
  user: PersonalUser | null,
): user is LoggedInUserWithPassword => {
  const hasPassword = isLoggedInWithPassword(user);
  if (!hasPassword) return false;

  const isExpired = user.passwordExpiresAt && new Date(user.passwordExpiresAt) < new Date();

  return !isExpired;
};
