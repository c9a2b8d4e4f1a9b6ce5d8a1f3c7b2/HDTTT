### Title
User Key Soft-Deletion Without Checking Active Approver References Permanently Disrupts Pending Transaction Approval Workflows

### Summary
When a `UserKey` is soft-deleted via `removeUserKey` (user self-service) or `removeUser` (admin), the system performs no check to determine whether that key is still referenced as a required approver in pending `TransactionApprover` records. A malicious approver can intentionally delete their own key after being assigned to a transaction, causing the transaction to be permanently stuck in `WAITING_FOR_SIGNATURES` state because `approveTransaction` silently returns `false` when the approver's keys are gone.

### Finding Description

**Root cause — no reference check on key deletion:**

`removeUserKey` in `back-end/apps/api/src/user-keys/user-keys.service.ts` soft-removes the key with no query against `TransactionApprover`: [1](#0-0) 

`removeUser` in `back-end/apps/api/src/users/users.service.ts` bulk-soft-deletes all keys for a user, again with no check for pending approvals: [2](#0-1) 

**How the approval flow breaks after key deletion:**

`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` calls `attachKeys` to load the user's active (non-soft-deleted) keys. If the key was soft-deleted, `user.keys` is empty and the function silently returns `false` at line 567 — no error, no state change, no notification: [3](#0-2) 

The `TransactionApprover` row with the deleted user's `userId` remains in the database with `signature = null` and `approved = null`. The transaction cannot advance past `WAITING_FOR_SIGNATURES` because the required approver can never sign.

**The `UserKey` entity confirms the relationship:** [4](#0-3) 

The `approvedTransactions` one-to-many relation shows that a single `UserKey` is the designated signing key for potentially many `TransactionApprover` records.

**Notification service confirms soft-deleted keys are excluded from required signers:** [5](#0-4) 

This confirms the system intentionally excludes soft-deleted keys from signing requirements — but does not cascade that exclusion back to unblock the stuck `TransactionApprover` record.

### Impact Explanation

A transaction in `WAITING_FOR_SIGNATURES` that requires approval from a user whose key has been soft-deleted is permanently stuck. No signature can be recorded for that approver slot. If the stuck approver is the sole required approver, or if a threshold cannot be met without them, the transaction can never be executed or cancelled through normal flow. This is a permanent DoS on the transaction workflow for all parties (creator, other approvers, observers) waiting on that transaction.

### Likelihood Explanation

The attack requires only a normal authenticated user who has been added as a transaction approver. The user calls `DELETE /users/:userId/keys/:keyId` — a standard self-service endpoint — after being assigned as an approver. No admin access, no leaked credentials, and no special timing is required. The attacker simply deletes their own key. This is a realistic, low-effort action by a malicious or uncooperative approver.

### Recommendation

Before soft-deleting a `UserKey`, query `TransactionApprover` for any records where `userId` matches the key's owner AND `signature IS NULL` AND the parent transaction is in an active status (`WAITING_FOR_SIGNATURES`). If any such records exist, either:
1. Reject the deletion with an error explaining the key is required for pending approvals, or
2. Automatically soft-delete the corresponding `TransactionApprover` records and notify the transaction creator so they can reassign the approver.

Apply the same guard in `removeUser` before bulk-deleting all keys for a user.

### Proof of Concept

1. Admin creates a transaction `T` requiring approval from User A (key `K`).
2. Admin adds User A as a `TransactionApprover` for `T` via `POST /transactions/:id/approvers`.
3. User A (malicious) calls `DELETE /users/:userId/keys/:keyId` to soft-delete key `K`. The call succeeds — `removeUserKey` performs no check against `TransactionApprover`.
4. User A attempts to call `POST /transactions/:id/approvers/approve` with `userKeyId = K`. `attachKeys` returns an empty array (soft-deleted key excluded). `approveTransaction` hits line 567 and returns `false` silently.
5. Transaction `T` remains in `WAITING_FOR_SIGNATURES` indefinitely. The `TransactionApprover` row for User A still exists with `signature = null`, preventing the threshold from being met.
6. The transaction creator must manually call `DELETE /transactions/:id/approvers/:approverId` to remove User A and add a replacement — an out-of-band remediation step not enforced by the protocol.

  async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });

    if (!user) {
      throw new BadRequestException(ErrorCodes.UNF);
    }

    // Soft-delete all user keys first
    await this.repo.manager.softDelete(UserKey, { userId: id });

    // Then soft-delete the user
    await this.repo.softRemove(user);

    return true;
  }
