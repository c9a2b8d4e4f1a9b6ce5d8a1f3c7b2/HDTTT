Audit Report

## Title
Any Authenticated User Can Squat Another User's Public Key, Permanently Blocking Them From the Organization Signing Workflow

## Summary
The `POST /user/:userId?/keys` endpoint registers a public key under the JWT-authenticated caller's account with no cryptographic proof that the caller controls the corresponding private key. Because the `publicKey` column is looked up globally, any verified organization user who registers a victim's public key first permanently prevents the victim from registering that key under their own account.

## Finding Description

**Root cause**

`UploadUserKeyDto` accepts only `publicKey`, `mnemonicHash`, and `index`. No challenge-response or proof-of-possession is required. [1](#0-0) 

The controller is declared as `@Controller('user/:userId?/keys')` but the `uploadKey` handler binds only `@GetUser()` (the JWT identity) — the `:userId` path parameter is entirely ignored for POST. [2](#0-1) 

`UserKeysService.uploadKey()` searches for the submitted `publicKey` globally (no `userId` filter). If the found row belongs to a different user, it throws `BadRequestException(ErrorCodes.PU)`. If no row exists, it creates one owned by the **caller**, not the key's true owner. [3](#0-2) 

**Exploit flow**

1. Victim generates an ED25519 key pair. The public key is visible on the Hedera network (mirror node, transaction bytes, multi-sig coordination).
2. Attacker (any verified organization user) calls `POST /user/<any_id>/keys` with `{ "publicKey": "<victim_pubKey>" }`. The `:userId` path parameter is silently ignored.
3. The server creates a `UserKey` row linking `victim_pubKey` to the **attacker's** `userId`.
4. Victim later calls `POST /user/<victim_id>/keys` with the same public key. The service finds the existing row, sees `userKey.userId !== victim.id`, and throws `BadRequestException(ErrorCodes.PU)`.
5. Victim is permanently blocked from registering their own key.

The e2e test suite explicitly confirms both behaviors:

- That the `:userId` path parameter is ignored and keys always register to the authenticated caller: [4](#0-3) 

- That a key registered by one user cannot be registered by another: [5](#0-4) 

## Impact Explanation

**Permanent DoS on key registration:** Once the attacker squats the victim's public key, the victim can never register it under their own account. The only recovery path requires an admin to manually delete the attacker's `UserKey` row.

**Exclusion from multi-sig workflows:** `SignersService.validateAndProcessSignatures` builds `userKeyMap` exclusively from the authenticated submitter's registered keys (`user.keys`). Because the victim's key is not in their own key map, the victim cannot submit signatures for any transaction requiring that key, even though they hold the private key. [6](#0-5) [7](#0-6) 

**Organizational integrity break:** The `UserKey` entity is the authoritative link between a Hedera public key and an organization user. Squatting corrupts this mapping, causing the system to attribute signing authority to the wrong user.

## Likelihood Explanation

- **Attacker precondition:** Any verified organization account. No admin or privileged role required.
- **Information required:** The victim's public key — which is inherently public. It appears in Hedera transaction bytes, mirror node account queries, and is shared during normal multi-sig coordination.
- **Attack complexity:** A single authenticated HTTP POST. No race condition or timing dependency is required; the attacker simply needs to register the key before the victim does.
- **Detection difficulty:** The attack is indistinguishable from a normal key upload at the API level.

## Recommendation

Require cryptographic proof of key ownership at upload time. The standard approach is a **proof-of-possession (PoP) challenge**:

1. The server issues a short-lived, user-specific challenge (e.g., a signed nonce tied to the user's JWT subject and a timestamp).
2. The client signs the challenge with the private key corresponding to the submitted `publicKey`.
3. The server verifies the signature using the submitted `publicKey` before creating the `UserKey` row.

This ensures only the holder of the private key can register the corresponding public key, regardless of which user account is authenticated.

Additionally, the `:userId` path parameter in `POST /user/:userId?/keys` should either be removed (since it is already ignored) or validated to match the authenticated user's ID to prevent confusion.

## Proof of Concept

```http
# Step 1: Attacker registers victim's public key
POST /user/999/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{
  "publicKey": "<victim_ed25519_public_key_raw>"
}
# Response: 201 Created — key is now owned by attacker

# Step 2: Victim attempts to register their own key
POST /user/<victim_id>/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json

{
  "publicKey": "<victim_ed25519_public_key_raw>"
}
# Response: 400 Bad Request — ErrorCodes.PU ("Public key in use")
# Victim is now permanently locked out of registering their own key
```

The service-level unit test at `user-keys.service.spec.ts` lines 91–105 directly confirms the blocking behavior when `userKey.userId !== user.id`. [8](#0-7)
