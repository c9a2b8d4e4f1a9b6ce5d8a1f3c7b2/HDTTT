## Audit Report

## Title
Unbounded Per-Transaction Async Loop in `getTransactionsToSign` Enables Authenticated DoS

## Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches all non-terminal transactions from the database without any row limit, then performs sequential async work per transaction before applying pagination. An authenticated user can accumulate transactions to degrade this endpoint for all users.

## Finding Description

**Unbounded database fetch (line 295):**

`this.repo.find` is called with no `take` or `limit` clause, returning every non-terminal transaction matching the filter: [1](#0-0) 

The `whereForUser` filter only excludes terminal statuses (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED) and does not scope results to the requesting user's transactions: [2](#0-1) 

**Sequential async loop over the full result set (lines 301–309):**

Every transaction in the unbounded result set is processed one-by-one with `await`: [3](#0-2) 

**Per-transaction call chain:**

`userKeysToSign` delegates to `userKeysRequiredToSign`: [4](#0-3) 

Which calls `keysRequiredToSign` → `transactionSignatureService.computeSignatureKey`: [5](#0-4) 

`computeSignatureKey` issues per-account calls to `accountCacheService.getAccountInfoForTransaction` for the fee payer, all signing accounts, and all receiver accounts: [6](#0-5) 

Each `getAccountInfoForTransaction` call performs a DB lookup (`dataSource.manager.findOne`) and, when the cache is stale or missing, an outbound mirror-node HTTP request, plus a `linkTransactionToAccount` DB insert: [7](#0-6) 

**Pagination applied after the full loop (line 313):** [8](#0-7) 

**Partial mitigation — `AccountCacheService`:**

The cache has a configurable TTL (`CACHE_STALE_THRESHOLD_MS`, default 10 s). Repeated lookups for the same account within the TTL window skip the mirror-node HTTP call. However, the DB `findOne` and `linkTransactionToAccount` insert still execute per account per transaction, and the sequential O(N) loop itself is not eliminated. [9](#0-8) 

**Endpoint status:**

The controller marks this endpoint `/* NO LONGER USED BY FRONT-END */`, which reduces real-world exposure but does not remove the route — it remains accessible to any authenticated API caller: [10](#0-9) 

## Impact Explanation

- Any authenticated user's non-terminal transactions contribute to the load for **all** users querying `GET /transactions/sign`, because the fetch is not scoped to the requesting user.
- Response time grows linearly with the number of non-terminal transactions in the database. Each transaction incurs at minimum one DB query and one DB insert; for transactions with multiple signing/receiver accounts, this multiplies.
- The sequential (non-parallel) loop means a single slow DB or mirror-node response per transaction compounds across the entire dataset.
- Accumulated transactions persist until expiry, requiring no sustained attacker effort after the initial creation burst.
- Although the front-end no longer calls this endpoint, it remains a live, authenticated API route.

## Likelihood Explanation

- Any verified user can call `POST /transactions` — no admin role is required.
- Transactions stored only in the back-end database impose no Hedera network cost on the attacker.
- The degradation is gradual and indistinguishable from legitimate usage growth until the endpoint times out.
- The endpoint being deprecated reduces the likelihood of real-world exploitation, but does not eliminate it for direct API consumers or integrations.

## Recommendation

1. **Apply database-level pagination before the async loop.** Pre-filter candidates in SQL (e.g., join against `transaction_signer` or `user_key` tables) so only transactions plausibly requiring the requesting user's signature are fetched, with a `take` limit matching the requested page size.
2. **Scope the fetch to the requesting user.** Add a user-specific join/subquery to `whereForUser`, analogous to the recursive CTE used in `getTransactionsToApprove`. [11](#0-10) 

3. **Remove or formally deprecate the endpoint** if it is confirmed unused, to eliminate the attack surface entirely.
4. **Add a hard cap** (e.g., `take: MAX_SAFE_LIMIT`) as a defensive backstop even after the above fixes.

## Proof of Concept

```
# 1. Register/authenticate as a normal user and obtain a JWT.

# 2. Create N transactions (e.g., AccountUpdateTransaction targeting accounts
#    with many signing accounts) via POST /transactions.
#    Set validStart far in the future so they remain WAITING_FOR_SIGNATURES.

for i in $(seq 1 500); do
  curl -s -X POST https://<host>/transactions \
    -H "Authorization: Bearer <jwt>" \
    -H "Content-Type: application/json" \
    -d '{"transactionBytes": "<hex_of_account_update_tx>", ...}'
done

# 3. Any user (including legitimate users) now calls:
curl -w "\nTime: %{time_total}s\n" \
  "https://<host>/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer <any_valid_jwt>"

# Expected: response time grows linearly with N.
# At sufficient N, the request times out at the HTTP gateway,
# making the endpoint unavailable for all users.
```
