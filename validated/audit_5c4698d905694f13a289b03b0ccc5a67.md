Audit Report

## Title
Missing Ownership Check in `removeTransactionGroup` Allows Any Authenticated User to Delete Any Transaction Group (IDOR)

## Summary
The `removeTransactionGroup` method in `TransactionGroupsService` performs no authorization check to verify that the requesting user owns the transaction group before deleting it and all its associated transactions. Any authenticated, verified user can permanently destroy any other user's transaction group by supplying its integer ID.

## Finding Description

In `transaction-groups.service.ts`, `removeTransactionGroup` (lines 172–198) fetches the group by ID and immediately proceeds to delete all group items and their transactions without ever verifying that the caller is the creator or owner: [1](#0-0) 

The `user` parameter is accepted but is only forwarded to `removeTransaction` — it is never used to gate access to the group itself.

Contrast this with `cancelTransactionGroup`, which explicitly enforces ownership before proceeding: [2](#0-1) 

And `getTransactionGroup`, which uses a user-scoped SQL query and throws `UnauthorizedException` if the user has no visibility into the group: [3](#0-2) 

The `DELETE /:id` route on the controller is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — all of which verify that a valid, non-blacklisted, verified JWT is present, but none of which check resource ownership: [4](#0-3) 

## Impact Explanation
A malicious authenticated user can permanently delete any transaction group in the system — including all `TransactionGroupItem` records and their linked `Transaction` records — by issuing `DELETE /transaction-groups/{id}` with an arbitrary group ID. This results in irreversible data loss for other organizations or users, destruction of in-progress multi-signature workflows, and potential financial impact if transactions that were pending execution are wiped.

## Likelihood Explanation
Group IDs are sequential integers (TypeORM default primary key). An attacker with a valid account needs only to iterate IDs starting from 1 to enumerate and destroy all groups. No special privilege, leaked credential, or physical access is required — only a valid JWT obtained through normal registration.

## Recommendation
Add an ownership check at the start of `removeTransactionGroup`, mirroring the pattern already used in `cancelTransactionGroup`. Specifically, before deleting, verify that the requesting user is the creator of all transactions in the group. The simplest approach is to reuse `getTransactionGroup(user, id, false)` (which already enforces user-scoped visibility via the SQL query) and then assert ownership, or add a direct DB query that joins `TransactionGroup → TransactionGroupItem → Transaction → UserKey` and filters by `user_key.user_id = user.id`.

## Proof of Concept

```
# Attacker (user B) obtains a valid JWT via normal login
POST /auth/login  →  { "token": "<attacker_jwt>" }

# Attacker enumerates group IDs (sequential integers)
DELETE /transaction-groups/1
Authorization: Bearer <attacker_jwt>

# Response: HTTP 200, true
# Result: Victim's transaction group 1, all its group items,
#         and all linked transactions are permanently deleted.
```

No special tooling is required. The attack is repeatable for every group ID in the system.

```typescript
  async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
    if (!group) {
      throw new Error('group not found');
    }
    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
      relations: {
        group: true,
      },
      where: {
        group: {
          id: group.id,
        },
      },
    });
    for (const groupItem of groupItems) {
      const transactionId = groupItem.transactionId;
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(transactionId, user, false);
    }

    await this.dataSource.manager.remove(TransactionGroup, group);

    emitTransactionUpdate(this.notificationsPublisher, groupItems.map(gi => ({ entityId: gi.transactionId })));

    return true;
  }
```
