### Title
Unverified Signature Bytes in `uploadSignatureMaps` Allow False Signer Registration and Premature Status Advancement

### Summary

The `uploadSignatureMaps` endpoint (`POST /transactions/:transactionId/signers`) accepts attacker-controlled signature bytes without cryptographically verifying that those bytes are a valid signature over the actual transaction body. Any authenticated user whose public key is a required signer can submit their valid public key paired with arbitrary/invalid signature bytes, be recorded as a legitimate signer in the database, and potentially advance the transaction status to `WAITING_FOR_EXECUTION` — causing the transaction to fail on the Hedera network when executed.

### Finding Description

**Root cause — missing cryptographic verification in `processTransactionSignatures`:**

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, the `processTransactionSignatures` private method processes an attacker-supplied `SignatureMap`: [1](#0-0) 

The method:
1. Confirms the public key belongs to the authenticated user (`userKeyMap.get(raw)`) — line 244–248.
2. Calls `sdkTransaction.addSignature(publicKey, map)` — line 251 — which inserts the raw signature bytes into the transaction structure.
3. **Never calls `publicKey.verify(bodyBytes, signature)` or any equivalent.**

The signer record is then unconditionally inserted into the database as long as `userKeys.length > 0`, even when `isSameBytes = true` (i.e., even when the SDK silently rejected the invalid bytes): [2](#0-1) 

**Contrast with the correct path — `importSignatures`:**

`transactions.service.ts` has a separate import path that correctly calls `validateSignature` before `addSignature`: [3](#0-2) 

`validateSignature` performs explicit cryptographic verification: [4](#0-3) 

The `uploadSignatureMaps` path has no equivalent call.

**Status advancement path:**

After `persistSignatureChanges`, `updateStatusesAndNotify` calls `processTransactionStatus`, which inspects `_signerPublicKeys` on the SDK transaction object. If the SDK's `addSignature` did insert the invalid bytes (SDK does not always reject them), the public key is added to `_signerPublicKeys`, and the status check may conclude all required keys have signed, advancing the transaction to `WAITING_FOR_EXECUTION`: [5](#0-4) 

The execution service then submits the transaction to the Hedera network with an invalid signature, causing it to fail: [6](#0-5) 

### Impact Explanation

1. **False signer registration**: An attacker whose key is a required signer submits their valid public key with garbage signature bytes. They are recorded as a `TransactionSigner` in the database without having produced a valid cryptographic signature.
2. **Premature status advancement**: If the attacker is the last required signer, the transaction status advances to `WAITING_FOR_EXECUTION`, locking out further legitimate signing.
3. **Forced transaction failure**: The transaction is submitted to the Hedera network with an invalid (or missing) signature and is rejected by the network, permanently failing the transaction and wasting any associated fees.
4. **Integrity corruption**: Other participants (approvers, observers) see the transaction as fully signed when it is not, breaking the trust model of the multi-signature workflow.

### Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated organization user whose public key appears in the transaction's required key list. No admin, governance, or privileged access is required.
- **Attack surface**: The `POST /transactions/:transactionId/signers` endpoint is a standard, documented API endpoint reachable by any verified user.
- **Ease**: A single crafted HTTP request with a valid `signatureMap` structure (correct node account ID, transaction ID, and the attacker's own DER public key) but with arbitrary bytes as the signature value is sufficient. The `IsSignatureMap` decorator only validates structural format, not cryptographic correctness. [7](#0-6) 

### Recommendation

Add cryptographic signature verification inside `processTransactionSignatures`, mirroring the `validateSignature` call already present in `importSignatures`. Before calling `sdkTransaction.addSignature(publicKey, map)`, verify each signature byte array against the transaction body bytes using `publicKey.verify(bodyBytes, signatureBytes)`. Reject the entire upload if any signature fails verification, consistent with the behavior in `validateSignature` at `back-end/libs/common/src/utils/sdk/transaction.ts` lines 213–248.

### Proof of Concept

1. Authenticate as a user (User A) whose public key is a required signer on Transaction T.
2. Construct a `SignatureMap` JSON body with the correct node account ID and transaction ID, User A's DER public key, but with `"deadbeefdeadbeef..."` (arbitrary hex) as the signature value.
3. `POST /transactions/T/signers` with this body.
4. Observe: HTTP 201 returned; a `TransactionSigner` row for User A is inserted; `transaction.transactionBytes` is updated (or unchanged); `processTransactionStatus` runs.
5. If User A was the last required signer, the transaction status advances to `WAITING_FOR_EXECUTION`.
6. The chain service picks up the transaction and submits it to the Hedera network; the network rejects it with `INVALID_SIGNATURE`, permanently failing the transaction.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
