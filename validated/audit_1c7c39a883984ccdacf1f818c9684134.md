### Title
Transaction Creator Can Manipulate Approver Structure After Approvals Are Collected, Bypassing Rejections

### Summary
The approver management endpoints (`POST`, `PATCH`, `DELETE` on `/transactions/:transactionId/approvers`) perform no check on the transaction's current status before modifying the approver tree. A transaction creator can observe which approvers have approved or rejected, then add, remove, or replace approvers — including those who rejected — while the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status. This allows the creator to unilaterally override the outcome of the approval process.

### Finding Description

**Root cause:** `createTransactionApprovers`, `updateTransactionApprover`, and the controller-level `removeTransactionApprover` all gate on a single check — `getCreatorsTransaction` — which only verifies the caller is the transaction creator. None of these paths check the transaction's `status` field before mutating the approver tree.

`getCreatorsTransaction` (the sole guard):


It fetches the transaction and checks `creatorKey.userId === user.id`. No status check. [1](#0-0) 

`createTransactionApprovers` — only calls `getCreatorsTransaction`, then inserts new approvers unconditionally: [2](#0-1) 

`updateTransactionApprover` — only calls `getCreatorsTransaction`, then updates userId/threshold/listId unconditionally: [3](#0-2) 

Controller `removeTransactionApprover` — calls `getCreatorsTransaction` then `removeTransactionApprover` with no status check: [4](#0-3) 

The `approveTransaction` function correctly checks that the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` before accepting a vote, but the approver-structure mutation paths have no equivalent guard: [5](#0-4) 

**Exploit flow:**

1. Creator creates a transaction with approvers A, B, C and threshold 2-of-3. Transaction enters `WAITING_FOR_SIGNATURES`.
2. A approves. B rejects (`approved = false`, `signature` set).
3. Creator observes B's rejection via `GET /transactions/:id/approvers` (visible to creator).
4. Creator calls `DELETE /transactions/:id/approvers/:B_id` — B is soft-deleted, rejection erased.
5. Creator calls `POST /transactions/:id/approvers` to add D (a cooperative user).
6. D approves. Threshold 2-of-3 is now met (A + D). Transaction proceeds.

Alternatively, the creator can call `PATCH /transactions/:id/approvers/:B_id` with `{ userId: D_id }`, which replaces B with D and clears B's `signature`/`approved` fields: [6](#0-5) 

The creator can also lower the threshold via `PATCH` with `{ threshold: N }` to reduce the number of approvals required after seeing partial rejections: [7](#0-6) 

### Impact Explanation

The multi-approver workflow is the protocol's primary organizational control mechanism. Bypassing it allows the transaction creator to execute transactions that the designated approvers have explicitly rejected. In an organizational context where approvers represent compliance, finance, or security reviewers, this completely nullifies the approval gate. Any transaction — including account updates, key changes, or fund transfers on Hedera — can be forced through by the creator regardless of approver decisions.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The attack requires only standard API calls available to any creator. The approval state (who approved, who rejected) is visible to the creator via the existing `GET /transactions/:id/approvers` endpoint. No special tooling, timing precision, or cryptographic capability is needed. The window of opportunity is the entire duration the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

### Recommendation

In `createTransactionApprovers`, `updateTransactionApprover`, and the controller's `removeTransactionApprover` handler, add a status check immediately after `getCreatorsTransaction` returns. Reject the mutation if the transaction status is anything other than `NEW`:

```typescript
const transaction = await this.getCreatorsTransaction(transactionId, user);
if (transaction.status !== TransactionStatus.NEW) {
  throw new BadRequestException('Approver structure cannot be modified after approval has started');
}
```

This mirrors the intent of the `approveTransaction` guard and locks the approver tree once the transaction leaves the `NEW` state.

### Proof of Concept

**Setup:** Transaction T with approvers [A, B, C], threshold 2-of-3, status `WAITING_FOR_SIGNATURES`.

**Step 1 — A approves:**
```
POST /transactions/T/approvers/approve
{ userKeyId: A_key, signature: "...", approved: true }
→ 200 OK
```

**Step 2 — B rejects:**
```
POST /transactions/T/approvers/approve
{ userKeyId: B_key, signature: "...", approved: false }
→ 200 OK
```

**Step 3 — Creator removes B (no status check blocks this):**
```
DELETE /transactions/T/approvers/:B_approver_id
Authorization: Bearer <creator_jwt>
→ 200 OK  (B's rejection is erased)
```

**Step 4 — Creator adds D:**
```
POST /transactions/T/approvers
{ approversArray: [{ userId: D_id }] }
→ 201 Created
```

**Step 5 — D approves:**
```
POST /transactions/T/approvers/approve
{ userKeyId: D_key, signature: "...", approved: true }
→ 200 OK
```

**Result:** Transaction T advances with approvals from A and D. B's rejection is permanently erased. The approval threshold is satisfied despite an explicit rejection from a designated approver.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
