Audit Report

## Title
Excessively Large TOTP `window` Value Allows Extended OTP Validity in Password Reset Flow

## Summary
The TOTP configuration in `auth.service.ts` sets `window: 20` with `step: 60`, making every OTP code valid for approximately **41 minutes** instead of the typical ~3 minutes. An attacker who intercepts a password-reset OTP email has a vastly extended window to exploit it. The report's secondary claim — that the same OTP can be submitted multiple times — is **partially inaccurate**: the OTP JWT is blacklisted after first use, but the extended window still enables a realistic attack via a fresh JWT request.

## Finding Description

In `back-end/apps/api/src/auth/auth.service.ts`, the global TOTP options are set at module load time:

```typescript
totp.options = {
  digits: 8,
  step: 60,
  window: 20,   // accepts codes ±20 steps from current time
};
``` [1](#0-0) 

With `step: 60` and `window: 20`, otplib accepts any TOTP code valid within ±20 steps of the current time: **41 steps × 60 seconds = 2,460 seconds ≈ 41 minutes** of total validity per code.

`verifyOtp()` calls `totp.check(token, secret)` which uses the global `window: 20`: [2](#0-1) 

A secondary indicator is the dead-code `expires` variable in `getOtpToken`: [3](#0-2) 

The `expires` value (`step × window = 1200 seconds`) is computed but never passed to `jwtService.sign()`. The JWT expiry is instead driven by the `OTP_EXPIRATION` config variable. The TOTP code itself, however, remains valid for the full 41-minute window regardless of the JWT.

**Correction to the report's OTP reuse claim:** The `/verify-reset` controller **does** blacklist the OTP JWT token after successful verification: [4](#0-3) 

This is enforced by `JwtBlackListOtpGuard` on the endpoint: [5](#0-4) 

The e2e test confirms that a second attempt with the same OTP JWT returns 401: [6](#0-5) 

So direct reuse of the same JWT+OTP pair is blocked. However, the extended window still enables the following attack: an attacker who intercepts the 8-digit OTP code can call `/reset-password` again to obtain a **fresh, non-blacklisted JWT**, then submit the intercepted code with the new JWT — all within the 41-minute window. The TOTP secret is deterministic (`OTP_SECRET + email`), so the same code remains valid across multiple JWT issuances within the window. [7](#0-6) 

## Impact Explanation

- **Password account takeover**: An attacker who intercepts the OTP email has ~41 minutes to obtain a fresh JWT (via `/reset-password`) and submit the intercepted code, resetting the victim's password.
- **Severity**: High — successful exploitation leads to full account takeover.
- The `EmailThrottlerGuard` on `/reset-password` throttles by email address but does not prevent an attacker from making a small number of fresh JWT requests within the 41-minute window. [8](#0-7) 

## Likelihood Explanation

- The attacker only needs to intercept the OTP email (compromised mail relay, email forwarding rule, or malicious insider).
- No privileged application access is required.
- The 41-minute window is long enough to make manual exploitation trivial.
- The attack requires only two HTTP requests: one to `/reset-password` (to get a fresh JWT) and one to `/verify-reset` (to submit the intercepted code).

## Recommendation

1. **Reduce `window` to 1** (the otplib default), which accepts codes ±1 step = 3 minutes of total validity — sufficient for legitimate users and far less exploitable.
2. **Fix the dead-code `expires` variable** in `getOtpToken`: either remove it or use it to drive the JWT expiry instead of the `OTP_EXPIRATION` config, ensuring the JWT and TOTP lifetimes are consistent.
3. Consider storing a per-user OTP nonce in the database and invalidating it on first use, so the 8-digit code itself cannot be reused even with a fresh JWT.

## Proof of Concept

```
# Step 1: Victim requests password reset
POST /auth/reset-password { "email": "victim@example.com" }
→ OTP email sent to victim (code: 12345678), JWT_1 returned

# Step 2: Attacker intercepts email, obtains code 12345678

# Step 3: Attacker requests a fresh JWT (within 41-minute window)
POST /auth/reset-password { "email": "victim@example.com" }
→ JWT_2 returned (not blacklisted)

# Step 4: Attacker submits intercepted code with fresh JWT
POST /auth/verify-reset
  otp: JWT_2
  body: { "token": "12345678" }
→ 200 OK, verified JWT returned

# Step 5: Attacker sets new password
PATCH /auth/set-password
  otp: <verified JWT>
  body: { "password": "attacker_password" }
→ 200 OK — account taken over
```
