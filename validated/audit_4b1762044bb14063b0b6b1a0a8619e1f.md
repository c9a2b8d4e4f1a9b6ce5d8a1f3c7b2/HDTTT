### Title
`removeTransactionApprover` Authorization Check Decoupled from Deleted Resource — Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Because `updateTransactionApprover` correctly cross-validates the approver-to-transaction link while `removeTransactionApprover` does not, an attacker who owns any transaction can delete approvers from transactions they did not create, disrupting approval workflows across the system.

### Finding Description

**Root cause — missing cross-validation in the delete path**

In `approvers.controller.ts`, the DELETE handler performs two independent calls:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns :transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver :id — no link check
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller created the transaction in the URL parameter. `removeTransactionApprover` then fetches and deletes the approver by its own `id` with no assertion that it belongs to that transaction:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);   // no transactionId check
    ...
}
``` [2](#0-1) 

**Contrast with the update path — which does the check correctly**

`updateTransactionApprover` resolves the root node of the approver and explicitly verifies it belongs to the URL's `transactionId` before proceeding:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The delete path has no equivalent guard. The restriction (creator check) is applied to one resource (`transactionId` in the URL) while the destructive action targets a different resource (approver `id`) without cross-validating the two.

**Exploit flow**

1. Attacker (User A) creates Transaction 1 — they are its creator.
2. Victim (User B) creates Transaction 2 with approver nodes (IDs are sequential integers, trivially enumerable).
3. Attacker calls: `DELETE /transactions/1/approvers/{approver_id_from_tx2}`
4. `getCreatorsTransaction(1, userA)` passes — User A is the creator of Transaction 1.
5. `removeTransactionApprover(approver_id_from_tx2)` executes — the approver from Transaction 2 is soft-deleted with no further check.
6. Transaction 2's approval tree is now corrupted or destroyed.

### Impact Explanation

An authenticated attacker with no special privileges beyond having created at least one transaction can silently delete approver nodes from any transaction in the organization. This can:

- Remove required approvers, allowing a transaction to proceed without the intended oversight.
- Destroy the entire approver tree of a transaction, making it permanently un-approvable (locked state).
- Undermine the multi-signature governance model that is the core security property of the system.

**Severity: High** — direct unauthorized state mutation of another user's transaction approval structure.

### Likelihood Explanation

Any registered, verified user can create a transaction (the minimum precondition). Approver IDs are auto-incremented integers exposed in API responses, making enumeration trivial. No brute-force or cryptographic attack is required. The attacker only needs to know one approver ID from a target transaction, which is obtainable by observing normal API responses or by sequential enumeration.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), resolve the root transaction of the approver being deleted and assert it matches the `transactionId` URL parameter, mirroring the guard already present in `updateTransactionApprover`:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const rootNode = await this.getRootNodeFromNode(approver.id);
    if (!rootNode || rootNode.transactionId !== transactionId)
        throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

    await this.getCreatorsTransaction(rootNode.transactionId, user);
    await this.removeNode(approver.id);
    ...
}
```

This ensures the authorization check and the deleted resource are always bound to the same transaction, closing the decoupled-check gap.

### Proof of Concept

**Preconditions:** Two registered users (Alice, Bob) in the same organization. Alice has created Transaction 1. Bob has created Transaction 2 with an approver node (approver ID = 42, obtained from a prior GET response).

**Steps:**

```
# Step 1 — Alice creates any transaction (Transaction 1, id=1)
POST /transactions  [Alice's JWT]
→ 201 { id: 1, ... }

# Step 2 — Bob creates Transaction 2 with an approver
POST /transactions  [Bob's JWT]
→ 201 { id: 2, ... }

POST /transactions/2/approvers  [Bob's JWT]
Body: { approversArray: [{ userId: 3 }] }
→ 201 [{ id: 42, transactionId: 2, userId: 3 }]

# Step 3 — Alice deletes Bob's approver using her own transactionId in the URL
DELETE /transactions/1/approvers/42  [Alice's JWT]
→ 200 true   ← succeeds; approver 42 from Transaction 2 is now deleted

# Step 4 — Verify Transaction 2's approver is gone
GET /transactions/2/approvers  [Bob's JWT]
→ 200 []   ← approver tree destroyed
```

**Expected result:** 401 Unauthorized at Step 3.
**Actual result:** 200 OK — approver from a foreign transaction is deleted. [1](#0-0) [2](#0-1) [3](#0-2)
