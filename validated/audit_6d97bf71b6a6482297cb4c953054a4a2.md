### Title
Balance Validation Check Bypassed in Organization Mode for HBAR Transfer Transactions

### Summary
In `TransferHbar.vue`, the guard that prevents submitting a transfer whose debit amount exceeds the sender's on-chain balance is conditioned on `user.selectedOrganization === null`. When a user operates in Organization mode the check is silently dropped, and neither `preCreateAssert` nor the back-end `validateAndPrepareTransaction` compensates for the missing guard. A malicious organization member can therefore craft and submit a Transfer HBAR transaction whose debit amount exceeds the sender account's actual balance, causing the transaction to fail on-chain after all required co-signers have already spent time reviewing and signing it, and after the fee-payer account has been charged the Hedera network fee.

### Finding Description

**Root cause — client-side guard is mode-gated:**

In `front-end/src/renderer/components/Transaction/Create/TransferHbar/TransferHbar.vue` lines 45–51, the `createDisabled` computed property reads:

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    (user.selectedOrganization === null && anyTransfersExceedingBalance.value)  // ← only in personal mode
  );
});
``` [1](#0-0) 

`anyTransfersExceedingBalance` (lines 66–81) correctly computes whether any debit leg exceeds the mirror-node balance for that account: [2](#0-1) 

But the entire expression is short-circuited to `false` whenever `user.selectedOrganization !== null`, so the button is never disabled for an over-balance transfer in Organization mode.

**`preCreateAssert` does not fill the gap:**

The `preCreateAssert` function (lines 101–113) only validates that the net sum of all transfers equals zero, that the adjustment count is within bounds, and nothing else. There is no per-account balance check here either. [3](#0-2) 

**Back-end `validateAndPrepareTransaction` does not fill the gap:**

The server-side validation in `back-end/apps/api/src/transactions/transactions.service.ts` (lines 896–979) checks signature validity, expiry, size, and node membership — but performs no balance check against the Hedera mirror node or any other source. [4](#0-3) 

**Exploit flow:**

1. Attacker is a legitimate organization member (no privileged keys required).
2. Attacker opens the Transfer HBAR form in Organization mode.
3. Attacker sets a debit of, e.g., 1,000,000 HBAR from an account holding 1 HBAR. The "Sign & Submit" button remains enabled because `anyTransfersExceedingBalance` is not evaluated.
4. Attacker submits the transaction to the back-end. It passes all server-side checks.
5. All required co-signers receive notifications, review, and sign the transaction.
6. The `chain` service submits the signed transaction to the Hedera network.
7. Hedera rejects it with `INSUFFICIENT_ACCOUNT_BALANCE`; the fee-payer account is charged the network fee.
8. The transaction is recorded as `FAILED` in the back-end.

### Impact Explanation

- **Fee drain:** Every failed submission charges the fee-payer account a Hedera network fee. An attacker who repeatedly creates such transactions can drain the organization's fee-payer account of small but cumulative amounts.
- **Workflow disruption / DoS:** Every invalid transaction triggers the full multi-signature collection workflow. Co-signers receive notifications, spend time reviewing, and sign transactions that are guaranteed to fail. At scale this degrades the organization's ability to process legitimate transactions.
- **Trust erosion:** Repeated failed transactions attributed to a member's submissions can corrupt audit logs and erode trust in the platform's integrity checks.

The Hedera network itself prevents actual fund theft (it enforces balance at the protocol layer), so the severity is **Medium** — real but bounded impact.

### Likelihood Explanation

Any authenticated organization member can trigger this with zero special privileges and zero cryptographic capability beyond their normal membership. The attacker-controlled entry point is the standard Transfer HBAR UI form. No social engineering or insider access beyond normal membership is required. Likelihood is **High**.

### Recommendation

Remove the `user.selectedOrganization === null` condition so the balance guard applies in both personal and organization modes:

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    anyTransfersExceedingBalance.value   // apply regardless of mode
  );
});
```

Additionally, add the same check to `preCreateAssert` so it is enforced even if the reactive guard is bypassed:

```js
if (anyTransfersExceedingBalance.value) {
  throw new Error('Transfer amount exceeds sender account balance');
}
```

### Proof of Concept

1. Log in to the Hedera Transaction Tool and join an organization.
2. Navigate to **Transactions → Create → Transfer HBAR**.
3. Add a "From" entry for any account (e.g., `0.0.1234`) with a balance of 1 HBAR, and set the debit amount to 1,000,000 HBAR.
4. Add a matching "To" entry so the net sum is zero.
5. Observe that the **Sign & Submit** button is **enabled** (it would be disabled in personal mode for the same inputs).
6. Click **Sign & Submit**. The transaction is accepted by the back-end, co-signers are notified, and after signing the `chain` service submits it to Hedera.
7. The Hedera network returns `INSUFFICIENT_ACCOUNT_BALANCE`; the fee-payer is charged; the transaction is marked `FAILED`.

const anyTransfersExceedingBalance = computed(() => {
  let result = false;
  for (const transfer of data.transfers) {
    if (transfer.amount.isNegative()) {
      const accountInfo = accountInfos.value[transfer.accountId.toString()];
      if (
        accountInfo &&
        transfer.amount.negated().toBigNumber().isGreaterThan(accountInfo.balance.toBigNumber())
      ) {
        result = true;
        break;
      }
    }
  }
  return result;
});
