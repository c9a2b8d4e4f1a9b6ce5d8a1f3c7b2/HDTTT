Audit Report

## Title
Unbounded Recursive Iteration in `createTransactionApprovers` Enables Authenticated DoS via Exhaustive DB Operations

## Summary
The `POST /transactions/:transactionId/approvers` endpoint accepts an arbitrarily large, deeply nested `approversArray` with no maximum-size constraint. The internal `createApprover` function is recursive and executes up to 6 sequential database queries per node, all inside a single held database transaction. An authenticated transaction creator can craft a wide or deeply nested approver tree that triggers thousands of DB operations in one request, exhausting database connections and degrading service for all users.

## Finding Description

**Root cause — missing `@ArrayMaxSize` on both DTO fields:**

`CreateTransactionApproverDto` enforces only `@ArrayMinSize(1)` on the recursive `approvers` field but imposes no upper bound. `CreateTransactionApproversArrayDto` has no size constraint at all on `approversArray`. [1](#0-0) 

**Unbounded recursive loop with per-node DB I/O:**

Inside `createTransactionApprovers`, the inner async function `createApprover` (defined at line 245) is called once per root entry in `dto.approversArray` and recursively for every element of `dtoApprover.approvers`. Each invocation performs:

1. `isNode` — `count(TransactionApprover)` query (line 250)
2. `findOne(TransactionApprover)` — parent lookup (line 255)
3. `getRootNodeFromNode` — recursive CTE traversal query (line 262)
4. `count(User)` — user existence check (line 272)
5. `getApproversByTransactionId` — recursive CTE query (line 319)
6. `insert(TransactionApprover)` — DB write (line 336) [2](#0-1) 

All of this runs inside a single `dataSource.transaction(...)` block, holding a DB connection open for the entire duration. [3](#0-2) 

The outer loop that starts the recursion: [4](#0-3) 

**Body size is the only payload guard — and it is insufficient:**

The only server-wide protection against large payloads is a 2 MB JSON body limit set in `setup-app.ts`: [5](#0-4) 

A 2 MB payload can encode thousands of nested approver nodes (each node is a small JSON object), producing a proportionally large number of DB operations. A global IP-based throttler exists (`IpThrottlerModule`) but it limits request *frequency*, not per-request *complexity*; a single crafted request is sufficient to trigger the issue. [6](#0-5) 

**Secondary surface — observers endpoint:**

`CreateTransactionObserversDto` also lacks `@ArrayMaxSize`. The loop at line 49 of `observers.service.ts` iterates over every supplied `userId` in memory before a single batch `repo.save`. This is less severe than the approvers path (no per-item DB queries inside a held transaction) but still allows a single request to create an arbitrarily large in-memory array. [7](#0-6) [8](#0-7) 

## Impact Explanation
A single crafted request can hold a PostgreSQL connection open for an extended period while executing hundreds or thousands of recursive CTE queries and inserts. With a small number of concurrent such requests, the database connection pool is exhausted, causing all other API operations (authentication, transaction reads, signature submissions) to queue and time out. This constitutes a full service denial for all organization users. The impact persists until the attacker stops sending requests; no admin action can recover the pool while the long transaction is running.

## Likelihood Explanation
The attacker must be an authenticated user who is the creator of at least one transaction — a condition reachable by any registered organization member. No elevated privilege is required. The attack requires only a single HTTP request with a crafted JSON body, which any HTTP client can produce. The global throttler limits request frequency but does not prevent a single large request from causing the described DB load. The attack is therefore trivially repeatable within the throttle window.

## Recommendation

1. **Add `@ArrayMaxSize` to both DTO fields** — apply a reasonable upper bound (e.g., 20–50) to `approversArray` in `CreateTransactionApproversArrayDto` and to `approvers` in `CreateTransactionApproverDto`.
2. **Enforce a maximum recursion depth** — track nesting depth during `createApprover` recursion and reject inputs that exceed a defined limit (e.g., 5 levels).
3. **Pre-validate the full tree size before opening the DB transaction** — count total nodes in the DTO tree before entering `dataSource.transaction(...)` and reject oversized inputs immediately.
4. **Consider per-endpoint rate limiting** on the `POST /transactions/:transactionId/approvers` route in addition to the global throttler.
