### Title
Missing Notification Emission After Sensitive State-Changing Actions

### Summary
Several sensitive state-changing functions in the backend API do not emit any notification events after completing their operations. The notification infrastructure (`NatsPublisherService`) is already injected and used in the same services for other operations, but is absent from critical privilege-escalation and key-management actions. This prevents off-chain clients (the Electron desktop app) from tracking important state changes and leaves an incomplete audit trail.

### Finding Description

**1. `elevateAdmin()` in `auth.service.ts` — no notification after admin privilege escalation** [1](#0-0) 

The function elevates any user to admin by calling `updateUserById` with `{ admin: true }`, but emits no notification. The `NatsPublisherService` is already injected in `AuthService` and `emitUserStatusUpdateNotifications` is already called in `changePassword()` for user status updates: [2](#0-1) 

Yet the equally (or more) sensitive `elevateAdmin()` path is silent.

**2. `removeUser()` in `users.service.ts` — no notification after user removal** [3](#0-2) 

Soft-deleting a user (and all their keys) emits no notification. Other users with pending transactions that required the deleted user's keys are not informed.

**3. `removeUserKey()` / `uploadKey()` / `updateMnemonicHash()` in `user-keys.service.ts` — no notification after key management operations** [4](#0-3) [5](#0-4) [6](#0-5) 

Uploading, removing, or updating the mnemonic hash of a cryptographic key — all sensitive operations — emit no notification. By contrast, every transaction-level state change (signing, approving, canceling, archiving) correctly emits a notification: [7](#0-6) 

**4. Developer intent confirmed by commented-out notification in `removeNode()`** [8](#0-7) 

The comment `// notifyTransactionAction(this.notificationsService);` shows the developer intended to emit a notification from `removeNode()` but left it commented out.

### Impact Explanation

- **Admin privilege escalation is invisible**: When `elevateAdmin()` is called, no other admin or the affected user receives any notification. The Electron desktop client has no way to detect that a user's privilege level changed without polling.
- **Key removal is silent**: Removing a user key that is referenced by pending transactions leaves those transactions in an ambiguous state with no notification to involved parties.
- **Incomplete audit trail**: The notification/event system is the primary mechanism for off-chain clients to track state changes. Missing emissions for these sensitive actions means the audit trail is incomplete and the UI can display stale data.

### Likelihood Explanation

These are routine administrative operations (user management, key management, admin elevation) that are exercised regularly in a multi-user organizational workflow. The notification infrastructure is already wired into the same services, so the omission is a consistent pattern rather than an isolated oversight.

### Recommendation

Emit appropriate NATS notification events after each sensitive state-changing action:

- In `AuthService.elevateAdmin()`: call `emitUserStatusUpdateNotifications` (already used in `changePassword()`).
- In `UsersService.removeUser()`: emit a user-removal notification.
- In `UserKeysService.uploadKey()`, `removeUserKey()`, and `updateMnemonicHash()`: emit a key-change notification so that clients and pending-transaction participants can react.
- Remove the comment from `removeNode()` or confirm the notification is covered by all callers.

### Proof of Concept

1. Admin A calls `POST /auth/elevate-admin` with `{ userId: X }`.
2. `AuthService.elevateAdmin()` calls `usersService.updateUserById(userId, { admin: true })` and returns.
3. No NATS message is published. The Electron desktop client for Admin B, User X, or any other user receives no WebSocket push and has no way to know the privilege change occurred without a full page reload/poll.
4. Contrast with `cancelTransaction()`, which always calls `emitTransactionStatusUpdate` after the DB update — the pattern is established but not applied to admin/key management actions. [1](#0-0) [9](#0-8)

  async updateMnemonicHash(
    user: User,
    id: number,
    dto: UpdateUserKeyMnemonicHashDto,
  ): Promise<UserKey> {
    const userKey = await this.getUserKey({ id });

    if (!userKey) {
      throw new BadRequestException(ErrorCodes.KNF);
    }

    if (userKey.userId !== user.id) {
      throw new BadRequestException(ErrorCodes.PNY);
    }

    await this.repo.update({ id }, { mnemonicHash: dto.mnemonicHash, index: dto.index });
    userKey.mnemonicHash = dto.mnemonicHash;
    userKey.index = dto.index || userKey.index;

    return userKey;
  }
