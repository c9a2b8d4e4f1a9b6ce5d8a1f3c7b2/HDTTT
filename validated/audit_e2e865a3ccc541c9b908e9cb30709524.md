Audit Report

## Title
Transaction Creator Can Modify Approval Rules Mid-Lifecycle, Bypassing Multi-Party Governance

## Summary
`updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover` in `ApproversService` authorize callers only by verifying they are the transaction creator. None of these mutation paths check the transaction's current `status`. A transaction creator can lower the approval threshold, replace a rejecting approver (clearing their recorded decision), or delete approvers entirely — at any point during the transaction lifecycle, including while the transaction is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

## Finding Description

**Root cause:** `getCreatorsTransaction` — the sole authorization check used by all three mutation paths — only verifies creator identity. It performs no status guard. [1](#0-0) 

`createTransactionApprovers` calls only `getCreatorsTransaction` before mutating: [2](#0-1) 

`updateTransactionApprover` calls only `getCreatorsTransaction` before mutating: [3](#0-2) 

The controller's `removeTransactionApprover` handler calls `getCreatorsTransaction` then immediately removes — no status check: [4](#0-3) 

By contrast, `approveTransaction` correctly enforces a status guard: [5](#0-4) 

**Threshold mutation path:** The creator can lower the threshold of any tree node at any time with no status check: [6](#0-5) 

**Approver replacement path:** Replacing a `userId` unconditionally clears `userKeyId`, `signature`, and `approved` — nullifying any prior approval or rejection: [7](#0-6) 

## Impact Explanation

The approval system is the primary governance gate for multi-party Hedera transaction authorization. Without a status guard on mutation paths, a transaction creator can:

1. **Lower the threshold after partial approval** — configure a 3-of-5 requirement, collect 2 approvals, then `PATCH /transactions/:id/approvers/:nodeId` with `{"threshold": 2}`, satisfying the requirement with fewer approvers than originally mandated.
2. **Nullify a rejection** — if an approver submits `approved: false`, the creator can `PATCH` that approver's `userId` to a cooperative user, which clears `signature`, `approved`, and `userKeyId`, erasing the rejection entirely.
3. **Remove all approvers** — `DELETE /transactions/:id/approvers/:id` for every approver node eliminates the approval requirement while the transaction is live.

This breaks the integrity of multi-party approval for high-value Hedera network operations (system file updates, node management, etc.).

## Likelihood Explanation

The attacker is the transaction creator — any authenticated, verified user who created the transaction. No special privileges, cryptographic material, or admin access are required. The attack requires only standard API calls to `PATCH /transactions/:transactionId/approvers/:id` or `DELETE /transactions/:transactionId/approvers/:id`, which are reachable by any such user.

## Recommendation

Add a status guard at the start of `createTransactionApprovers`, `updateTransactionApprover`, and in the controller's `removeTransactionApprover` handler (or inside `removeTransactionApprover` in the service). After calling `getCreatorsTransaction`, check that the returned transaction's `status` is `WAITING_FOR_SIGNATURES` (or whichever statuses are considered mutable by design) and throw a `BadRequestException` otherwise — mirroring the pattern already used in `approveTransaction` at lines 583–588.

## Proof of Concept

1. Creator calls `POST /transactions/:id/approvers` to set up a 3-of-5 approval tree.
2. Two approvers call `POST /transactions/:id/approvers/approve` — two approvals are recorded.
3. A third approver calls `POST /transactions/:id/approvers/approve` with `approved: false` — a rejection is recorded.
4. Creator calls `PATCH /transactions/:id/approvers/:treeNodeId` with `{"threshold": 2}` — the threshold is lowered to 2, and the transaction now satisfies its own approval requirement despite the rejection.
5. Alternatively at step 4, creator calls `PATCH /transactions/:id/approvers/:rejectingApproverNodeId` with `{"userId": <cooperative_user_id>}` — the rejection record (`approved`, `signature`, `userKeyId`) is cleared and replaced with a new, unsigned approver slot.
6. Alternatively, creator calls `DELETE /transactions/:id/approvers/:id` for each approver node, removing the approval requirement entirely.

In all three cases, the mutation succeeds because `getCreatorsTransaction` only checks creator identity and no status guard exists on these paths.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
```
