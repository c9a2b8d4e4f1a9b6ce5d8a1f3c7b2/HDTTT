### Title
Transaction Creator Can Modify Approval Threshold Mid-Flight, Bypassing Multi-Party Governance

### Summary
The `ApproversService` allows a transaction creator to update or remove approvers — including lowering the approval threshold — on a transaction that is already in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` state. Because `getCreatorsTransaction` only verifies creator identity and never checks transaction status, a malicious creator can reduce the required approval count to a value already satisfied by collected approvals, causing the transaction to advance to execution without the originally intended governance quorum.

### Finding Description

**Vulnerability class:** State-transition / authorization bypass (direct analog to the `deployerCut` no-timelock issue).

**Root cause — `getCreatorsTransaction` has no status guard:** [1](#0-0) 

The function only checks `transaction.creatorKey?.userId !== user.id`. It never checks whether the transaction is in a terminal or in-progress state. Every mutation path calls this function as its sole authorization gate:

- `createTransactionApprovers` (line 239) — adds new approvers
- `updateTransactionApprover` (line 394) — changes threshold or swaps a user
- `removeTransactionApprover` via controller (line 108 of controller) — deletes an approver node [2](#0-1) [3](#0-2) 

**Exploit path:**

1. Creator creates a transaction and sets a threshold approver tree requiring 3-of-3 approvals.
2. Two of the three designated approvers submit their approvals. Each call to `approveTransaction` triggers `emitTransactionStatusUpdate`, which re-evaluates whether the threshold is now satisfied.
3. Before the third approver acts, the creator calls `PATCH /transactions/:txId/approvers/:approverId` with `{ "threshold": 2 }`.
4. `updateTransactionApprover` accepts the change — no status check prevents it. [4](#0-3) 

5. The threshold is now 2, which is already satisfied by the two existing approvals. The next `emitTransactionStatusUpdate` (fired at line 524) causes the receiver service to re-evaluate and advance the transaction to `WAITING_FOR_EXECUTION` — or directly to execution — without the third approval. [5](#0-4) 

An even simpler variant: the creator removes one of the three required approvers (reducing the denominator), which automatically lowers the effective threshold via the auto-adjust logic at lines 423–428, again satisfying the threshold with fewer approvals than originally required. [6](#0-5) 

### Impact Explanation

The approver system is the organizational governance layer that prevents a single creator from unilaterally executing sensitive Hedera transactions (system file updates, large HBAR transfers, key rotations, node updates). By silently lowering the threshold mid-flight, the creator bypasses the quorum that other organization members believed was in place. Approvers who have not yet voted are deceived: they were told their approval was required, but the transaction executes without it. This is a direct integrity failure in the trust model — the exact harm class the external report describes.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond having created the transaction. The attack requires only a standard `PATCH` API call to an endpoint that is always reachable. No race condition or timing precision is needed; the creator can lower the threshold at any point while the transaction is in `WAITING_FOR_SIGNATURES`. The precondition (being the creator of a transaction with approvers) is a routine workflow state.

### Recommendation

Add a status guard inside `getCreatorsTransaction` (or at the top of each mutation method) that rejects modifications when the transaction is already in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`:

```typescript
// In getCreatorsTransaction or at the start of each mutating method:
if (
  transaction.status === TransactionStatus.WAITING_FOR_SIGNATURES ||
  transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
) {
  throw new BadRequestException('Cannot modify approvers on an in-progress transaction');
}
```

Apply the same guard to `removeTransactionApprover` in the controller before delegating to the service. Approver structure should only be modifiable while the transaction is in `NEW` (draft) status, analogous to requiring a timelock before a parameter change takes effect.

### Proof of Concept

```
# Setup: creator creates transaction T with threshold approver tree {threshold:3, approvers:[A,B,C]}
POST /transactions/:txId/approvers  →  tree node id = 10

# A and B approve
POST /transactions/:txId/approvers/approve  (user A)
POST /transactions/:txId/approvers/approve  (user B)

# Transaction is still WAITING_FOR_SIGNATURES (needs 3-of-3)

# Creator lowers threshold to 2 — no status check blocks this
PATCH /transactions/:txId/approvers/10
Body: { "threshold": 2 }
→ HTTP 200 OK

# emitTransactionStatusUpdate fires; 2-of-2 collected approvals now satisfy threshold=2
# Transaction advances to WAITING_FOR_EXECUTION without C's approval
# C's approval was contractually required but silently bypassed
```

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
