### Title
Stale Approval Signature Carried Over When Re-Adding an Approver at a Different Tree Position, Allowing a Single Signature to Satisfy Multiple Approval Slots

### Summary

In `createTransactionApprovers`, when a user is added as an approver at a position that does not already exist in the tree, the code fetches any existing active approver records for that user on the same transaction and copies their `signature`, `userKeyId`, and `approved` fields into the new record. Because the duplicate-check (`isNode`) is position-specific (it matches on `listId` + `userId` + `transactionId`), a malicious transaction creator can remove an approver who has already signed and re-add that same user at one or more different positions in the tree, causing the single genuine signature to appear at multiple nodes simultaneously. This allows the creator to satisfy a multi-party approval threshold with fewer genuine approvals than required.

### Finding Description

**Root cause — `approvers.service.ts` lines 318–329:**

```typescript
if (typeof dtoApprover.userId === 'number') {
  const userApproverRecords = await this.getApproversByTransactionId(
    transactionId,
    dtoApprover.userId,
    transactionalEntityManager,
  );

  if (userApproverRecords.length > 0) {
    data.signature = userApproverRecords[0].signature;
    data.userKeyId = userApproverRecords[0].userKeyId;
    data.approved = userApproverRecords[0].approved;
  }
}
``` [1](#0-0) 

`getApproversByTransactionId` returns all **active** (non-soft-deleted) approver rows for the given `(transactionId, userId)` pair, regardless of their position in the tree. [2](#0-1) 

The duplicate guard `isNode` only blocks re-adding a user at the **exact same position** (same `listId`, same `transactionId`): [3](#0-2) 

Because `isNode` is position-specific, adding the same user at a different `listId` passes the guard, reaches the signature-copy block, and inserts a new row with the old signature already populated.

**Exploit flow:**

1. Creator creates transaction T with a threshold-2 approver tree: root threshold node → [Alice, Bob].
2. Alice calls `POST /transactions/T/approvers/approve` → her row gets `signature=S`, `approved=true`.
3. Creator calls `DELETE /transactions/T/approvers/<alice-node-id>` → Alice's row is soft-deleted.
4. Creator calls `POST /transactions/T/approvers` twice, adding Alice as a child of two **different** threshold nodes (e.g., `listId=nodeX` and `listId=nodeY`).
5. Both calls pass `isNode` (different positions), both reach the signature-copy block, and both new rows are inserted with `signature=S`, `approved=true`.
6. Alice now occupies two positions in the tree, both pre-approved, using only one genuine signature.
7. The threshold evaluation counts both, and the transaction advances to `WAITING_FOR_EXECUTION` without Bob ever signing. [4](#0-3) 

### Impact Explanation

A malicious transaction creator can bypass the multi-party approval threshold by duplicating a single approver's signature across multiple tree positions. The transaction is submitted to the Hedera network as if all required parties approved it, when in reality only a subset did. This undermines the entire purpose of the multi-signature orchestration model and can result in unauthorized Hedera transactions being executed (e.g., unauthorized token transfers, account updates, or system file changes).

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The required API calls (`POST /approvers`, `DELETE /approvers/:id`) are standard product flows available to any creator. The only precondition is that at least one approver has already signed. No cryptographic break, no admin access, and no external dependency is required.

### Recommendation

When creating a new approver record for a user who already has active approver records on the same transaction, **do not carry over the old signature, userKeyId, or approved fields**. The new approver record should always start in an unsigned state:

```typescript
const data: DeepPartial<TransactionApprover> = {
  transactionId: ...,
  listId: dtoApprover.listId,
  threshold: ...,
  userId: dtoApprover.userId,
  // Do NOT copy signature/userKeyId/approved from existing records
};
```

If the intent is to preserve a signature when restructuring the tree (e.g., moving a node), that should be an explicit, audited operation — not an implicit side-effect of `createTransactionApprovers`.

### Proof of Concept

**Preconditions:** Creator (C) owns transaction T. Transaction T has a threshold-2 approver tree with Alice (userId=2) and Bob (userId=3) as children of root threshold node (id=10). Alice has already approved.

```
# Step 1 – Alice approves
POST /transactions/T/approvers/approve
{ "userKeyId": 5, "signature": "<valid-sig>", "approved": true }
# Alice's approver row (id=20) now has signature set.

# Step 2 – Creator removes Alice
DELETE /transactions/T/approvers/20
# Row 20 is soft-deleted.

# Step 3 – Creator adds a new threshold node (id=30) as child of root
POST /transactions/T/approvers
{ "approversArray": [{ "listId": 10, "threshold": 1, "approvers": [{ "userId": 2 }] }] }
# isNode: listId=30, userId=2 → no existing row → passes.
# getApproversByTransactionId(T, 2) → returns [] (Alice's row is soft-deleted).
# New row inserted with userId=2, listId=30, signature=null.

# Step 4 – Creator adds Alice again under a SECOND new threshold node (id=40)
POST /transactions/T/approvers
{ "approversArray": [{ "listId": 10, "threshold": 1, "approvers": [{ "userId": 2 }] }] }
# isNode: listId=40, userId=2 → no existing row → passes.
# getApproversByTransactionId(T, 2) → returns Alice's row from step 3 (active, signature=null).
# New row inserted with userId=2, listId=40, signature=null.
```

For the signature carry-over to work, Alice must still be **active** (not soft-deleted) when the second `createTransactionApprovers` call is made. The precise sequence is:

```
# Step 1 – Alice approves (row id=20, active, signature=S)
# Step 2 – Creator adds Alice at listId=30 (isNode passes; getApproversByTransactionId returns row 20 with S)
#           → new row id=21 inserted with signature=S, approved=true
# Step 3 – Creator adds Alice at listId=40 (isNode passes; getApproversByTransactionId returns rows 20 AND 21)
#           → new row id=22 inserted with signature=S, approved=true
# Result: Alice has 3 active approver rows (20, 21, 22), all with approved=true.
#         Threshold evaluation counts all three; transaction advances without Bob signing.
``` [5](#0-4) [3](#0-2)

  async getApproversByTransactionId(
    transactionId: number,
    userId?: number,
    entityManager?: EntityManager,
  ): Promise<TransactionApprover[]> {
    if (typeof transactionId !== 'number' || (userId && typeof userId !== 'number')) return null;

    return (entityManager || this.repo).query(
      `
      with recursive approverList as
        (
          select * from transaction_approver 
          where "transactionId" = $1
            union all
              select approver.* from transaction_approver as approver
              join approverList on approverList."id" = approver."listId"
        )
      select * from approverList
      where approverList."deletedAt" is null
        ${userId ? 'and approverList."userId" = $2' : ''}
      `,
      userId ? [transactionId, userId] : [transactionId],
    );
  }
