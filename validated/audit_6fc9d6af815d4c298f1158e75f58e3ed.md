Audit Report

## Title
`removeTransaction` Lacks Status Guard, Allowing Creators to Overwrite Terminal Transaction States and Corrupt Audit Trail

## Summary
`removeTransaction` performs no status validation before acting, allowing a transaction creator to soft-delete (overwriting status to `CANCELED`) or hard-delete any transaction regardless of its current state — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, and `REJECTED`. This directly contradicts the status machine enforced by `cancelTransactionWithOutcome`, which gates cancellation behind a strict `cancelableStatuses` allowlist. For group transactions, `removeTransactionGroup` calls `removeTransaction` with `softRemove=false`, permanently hard-deleting all group transactions with no status pre-check.

## Finding Description

`cancelTransactionWithOutcome` enforces a strict status guard before acting:

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [1](#0-0) 

The `cancelableStatuses` allowlist is defined as only `NEW`, `WAITING_FOR_SIGNATURES`, and `WAITING_FOR_EXECUTION`: [2](#0-1) 

The `terminalStatuses` list (`EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, `ARCHIVED`, `REJECTED`) is explicitly defined but is **never consulted** inside `removeTransaction`: [3](#0-2) 

`removeTransaction` calls only `getTransactionForCreator` (ownership check only), then unconditionally overwrites status and removes the record:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user); // ownership only, no status check

    if (softRemove) {
      await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED }); // overwrites any state
      await this.repo.softRemove(transaction);
    } else {
      await this.repo.remove(transaction); // hard delete, no state check
    }
``` [4](#0-3) 

`getTransactionForCreator` confirms it only validates ownership, not status: [5](#0-4) 

The `DELETE /:id` controller endpoint routes directly to `removeTransaction` with `softRemove=true`:

```typescript
@Delete('/:id')
deleteTransaction(@GetUser() user, @Param('id', ParseIntPipe) id: number): Promise<boolean> {
    return this.transactionsService.removeTransaction(id, user, true);
}
``` [6](#0-5) 

For group transactions, `removeTransactionGroup` calls `removeTransaction` with `softRemove=false` (hard delete) for every item in the group, with no status pre-check:

```typescript
for (const groupItem of groupItems) {
    const transactionId = groupItem.transactionId;
    await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
    await this.transactionsService.removeTransaction(transactionId, user, false); // hard delete, no status guard
}
``` [7](#0-6) 

The `DELETE /:id` group endpoint routes directly to `removeTransactionGroup`: [8](#0-7) 

## Impact Explanation

- **Audit trail corruption**: An `EXECUTED` transaction — one that has already been submitted to the Hedera network and produced a receipt — can have its database status overwritten to `CANCELED` and be soft-deleted. Any downstream system, compliance check, or UI query relying on the database record will see a false state.
- **Permanent record destruction**: Via `removeTransactionGroup`, a creator can permanently hard-delete `EXECUTED` transactions from the database, making the on-chain execution unreconcilable with the application's own records.
- **State invariant violation**: The system's own `cancelableStatuses` allowlist (enforced in `cancelTransactionWithOutcome`) is completely bypassed by `removeTransaction`, breaking the intended state machine.

## Likelihood Explanation

Any authenticated user who created a transaction can trigger this with a standard `DELETE` HTTP request — no admin keys, no special privileges, no race condition required. The attacker precondition is simply: be the creator of a transaction that has reached a terminal state. This is a normal, reachable user flow.

## Recommendation

Add a status guard at the top of `removeTransaction` in `back-end/apps/api/src/transactions/transactions.service.ts`, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    const removableStatuses = [
      TransactionStatus.NEW,
      TransactionStatus.WAITING_FOR_SIGNATURES,
      TransactionStatus.WAITING_FOR_EXECUTION,
      TransactionStatus.CANCELED,
      TransactionStatus.REJECTED,
    ];

    if (!removableStatuses.includes(transaction.status)) {
      throw new BadRequestException(ErrorCodes.OTIP); // or a dedicated error code
    }
    // ... rest of function
}
```

For `removeTransactionGroup` in `back-end/apps/api/src/transactions/groups/transaction-groups.service.ts`, add an equivalent pre-check before iterating group items, or rely on the guard added to `removeTransaction` and handle the resulting exception gracefully per item.

## Proof of Concept

1. Creator submits a transaction; it reaches `EXECUTED` status (confirmed on-chain).
2. Creator sends: `DELETE /transactions/:id` with a valid JWT.
3. `deleteTransaction` controller calls `removeTransaction(id, user, true)`.
4. `removeTransaction` calls `getTransactionForCreator` — passes (user is creator).
5. No status check occurs. `repo.update(transaction.id, { status: TransactionStatus.CANCELED })` overwrites `EXECUTED` → `CANCELED`.
6. `repo.softRemove(transaction)` soft-deletes the record.
7. The database now shows the transaction as `CANCELED` and soft-deleted; the on-chain `EXECUTED` state is unreconcilable.

**Group variant (hard delete):**
1. Creator creates a group of transactions; all reach `EXECUTED`.
2. Creator sends: `DELETE /transaction-groups/:id`.
3. `removeTransactionGroup` iterates each group item and calls `removeTransaction(transactionId, user, false)`.
4. Each transaction is permanently hard-deleted from the database with no status check.
