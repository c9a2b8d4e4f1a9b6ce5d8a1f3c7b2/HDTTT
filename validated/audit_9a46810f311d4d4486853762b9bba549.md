Audit Report

## Title
Non-Active Users Can Be Added as Transaction Approvers Without Status Validation

## Summary
`createTransactionApprovers` in `approvers.service.ts` and `createTransactionObservers` in `observers.service.ts` do not validate that target users hold `UserStatus.NONE` before assigning them to transaction roles. A non-active user (status `NEW` or `RESET_PASSWORD`) added as a required approver cannot authenticate to fulfill the approval, stalling the transaction's approval workflow until the approver record is manually removed.

## Finding Description

The `UserStatus` enum defines three states: [1](#0-0) 

`VerifiedUserGuard` enforces that only `UserStatus.NONE` users can make authenticated API calls: [2](#0-1) 

However, `createTransactionApprovers` only checks that the target user **exists** by ID — it does not check `status`: [3](#0-2) 

The same gap exists in `updateTransactionApprover` when changing a `userId`: [4](#0-3) 

`createTransactionObservers` performs no existence or status check at all on the supplied `userIds`: [5](#0-4) 

## Impact Explanation

If a user with `UserStatus.NEW` or `UserStatus.RESET_PASSWORD` is added as a required approver in a threshold group, they cannot call `approveTransaction` because `VerifiedUserGuard` will reject their JWT. The transaction remains stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` until the creator explicitly removes the approver record via `removeTransactionApprover`. The creator must know to do this; there is no system-level warning or automatic remediation. For observers the impact is lower — a non-active observer cannot view the transaction through the API, but the transaction itself is not blocked.

## Likelihood Explanation

The transaction creator must be authenticated (`UserStatus.NONE`) to invoke `createTransactionApprovers`, but they can supply any integer `userId` in the DTO. A realistic scenario: an admin invites a new colleague (status `NEW`, password not yet set) and the creator immediately adds them as a required approver before onboarding is complete. This is an ordinary operational mistake, not a contrived attack.

## Recommendation

In `createTransactionApprovers` and `updateTransactionApprover` (the `dto.userId` branch), extend the user existence check to also assert `status = 'NONE'`:

```typescript
// approvers.service.ts – createTransactionApprovers, ~line 272
const userCount = await transactionalEntityManager.count(User, {
  where: { id: dtoApprover.userId, status: UserStatus.NONE },
});
if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
```

Apply the same fix to the `updateTransactionApprover` path at line 494. [4](#0-3) 

For `createTransactionObservers`, add an existence + status check before creating each observer record: [5](#0-4) 

## Proof of Concept

1. Register two users: `creator` (completes onboarding → `UserStatus.NONE`) and `newUser` (invited but has not set a password → `UserStatus.NEW`).
2. As `creator`, create a transaction and note its `transactionId`.
3. As `creator`, call `POST /transactions/:transactionId/approvers` with body:
   ```json
   { "approversArray": [{ "userId": <newUser.id> }] }
   ```
   The request succeeds — `createTransactionApprovers` counts the user by ID only and finds 1 row. [6](#0-5) 
4. Attempt to authenticate as `newUser`. `VerifiedUserGuard` returns `false` because `user.status !== UserStatus.NONE`. [7](#0-6) 
5. `newUser` cannot call `POST /transactions/:transactionId/approvers/approve`. The transaction remains in its current status indefinitely until `creator` manually removes the approver record.
