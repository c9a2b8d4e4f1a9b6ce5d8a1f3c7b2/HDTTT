Audit Report

## Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header Enables Brute Force on `/auth/login`

## Summary
`IpThrottlerGuard.getTracker()` derives its throttle-bucket key from the raw, client-supplied `X-Forwarded-For` header before falling back to `req.ip`. Because this header is fully attacker-controlled, an adversary can rotate it on every request to land in a fresh bucket, completely nullifying the IP-based rate limit across all routes. On `/auth/login` and `/auth/reset-password`, only the per-email `EmailThrottlerGuard` then remains as a brake, which is insufficient to prevent credential brute force or email flooding.

## Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`**

In `back-end/apps/api/src/guards/ip-throttler.guard.ts`:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
```

`req.headers['x-forwarded-for']` is the raw HTTP header sent by the client with no validation, sanitization, or proxy-trust check. It is used as the throttle-bucket key with higher priority than `req.ip`. An attacker who sends a unique value (e.g., a random UUID) in `X-Forwarded-For` on every request will always land in a new bucket; the counter never reaches the configured limit. [1](#0-0) 

**Global registration — every route is affected**

`IpThrottlerGuard` is registered as an `APP_GUARD` in `api.module.ts`, making it the sole IP-level protection for the entire API surface. [2](#0-1) 

**Attack surface — `/auth/login`**

`/auth/login` is decorated with `@UseGuards(LocalAuthGuard, EmailThrottlerGuard)`. With the IP guard bypassed, only `EmailThrottlerGuard` (per-email bucket) remains. [3](#0-2) 

**`EmailThrottlerGuard` throttles per email, not per attacker**

The tracker key is `req.body.email`. An attacker targeting N accounts gets `ANONYMOUS_MINUTE_LIMIT` attempts per minute **per account**, with no cross-account ceiling. [4](#0-3) 

**Throttle limits are env-var-only, no hard floor**

`ANONYMOUS_MINUTE_LIMIT` and `ANONYMOUS_FIVE_SECOND_LIMIT` are read directly from environment variables with no enforced minimum in code. [5](#0-4) 

**Unauthenticated `/auth/reset-password` also exposed**

`/auth/reset-password` carries only `EmailThrottlerGuard` and no authentication guard. With the IP guard bypassed and a list of known emails, an attacker can trigger OTP emails to arbitrary users at the per-email rate, causing email flooding. [6](#0-5) 

## Impact Explanation
With the IP throttle nullified, an attacker can attempt `ANONYMOUS_MINUTE_LIMIT` passwords per minute against every known account in parallel. There is no cross-account rate ceiling and no CAPTCHA. Weak or reused passwords are recoverable through sustained credential stuffing. A compromised account gives the attacker access to the organization's Hedera transaction signing workflow, including the ability to approve or submit transactions on behalf of the victim.

## Likelihood Explanation
The bypass requires only a standard HTTP client capable of setting custom headers — no special tooling, no privileged access, no cryptographic break. The `X-Forwarded-For` header is universally supported. The attacker needs a list of registered email addresses (obtainable via the `/auth/reset-password` endpoint, which processes requests for existing users, or via other enumeration). Likelihood is high.

## Recommendation
1. **Do not trust `X-Forwarded-For` directly.** If the application sits behind a trusted reverse proxy, configure Express's `trust proxy` setting and rely solely on `req.ip` (which Express will correctly populate from the last trusted proxy hop). Remove the `req.headers['x-forwarded-for']` branch from `IpThrottlerGuard.getTracker()`.
2. **If `X-Forwarded-For` must be used** (e.g., multi-hop proxy chains), extract only the **last** IP in the comma-separated list that was appended by a trusted proxy, not the first (which is client-supplied).
3. **Add a hard floor** for `ANONYMOUS_MINUTE_LIMIT` and `ANONYMOUS_FIVE_SECOND_LIMIT` in the Joi validation schema in `api.module.ts` (e.g., `Joi.number().min(1).max(10).required()`).
4. Consider adding account lockout or progressive delay after N failed login attempts per email, independent of the throttle mechanism.

## Proof of Concept
```
# Rotate X-Forwarded-For on every request to bypass IP throttle
for i in $(seq 1 1000); do
  curl -s -X POST https://target/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: $(cat /proc/sys/kernel/random/uuid)" \
    -d '{"email":"victim@example.com","password":"attempt'$i'"}'
done
```

Each request lands in a unique throttle bucket. The `IpThrottlerGuard` counter for any single bucket never reaches `GLOBAL_MINUTE_LIMIT`. Only `EmailThrottlerGuard`'s per-email limit (`ANONYMOUS_MINUTE_LIMIT`) applies, and it resets every 60 seconds, allowing sustained brute force across all targeted accounts in parallel.
