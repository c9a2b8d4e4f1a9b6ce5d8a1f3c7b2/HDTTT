### Title
Any Authenticated User Can Enumerate All Organization Transaction Groups via Missing Authorization on `GET /transaction-groups`

### Summary
The `getTransactionGroups` endpoint in `TransactionGroupsController` is explicitly marked "TESTING ONLY" in code comments but remains live in production with no user-scoping or admin restriction. Any verified authenticated user can call `GET /transaction-groups` and receive every `TransactionGroup` record in the database, regardless of ownership. This is a direct analog to the external report's pattern: a sensitive operation lacks the required access control, allowing any user to access data that should be restricted.

### Finding Description
**Root cause — controller:** [1](#0-0) 

The handler carries no `@GetUser()` parameter and applies no `AdminGuard` or ownership check. The class-level guard stack (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only verifies that the caller holds a valid, non-blacklisted JWT and has a verified account status — it does not restrict to admins or to the caller's own data. [2](#0-1) 

**Root cause — service:** [3](#0-2) 

`this.dataSource.manager.find(TransactionGroup)` issues an unfiltered `SELECT * FROM transaction_group`, returning every row in the table.

**Contrast with the properly-guarded sibling endpoint** `getTransactionGroup(user, id)`, which passes the caller's identity into a user-scoped SQL query and throws `UnauthorizedException` when the caller has no relation to the group: [4](#0-3) [5](#0-4) 

The unscoped `getTransactionGroups` path bypasses all of that logic entirely.

**Exploit flow:**
1. Attacker registers a normal user account (or uses any existing verified account).
2. Attacker sends `GET /transaction-groups` with a valid JWT.
3. Server returns all `TransactionGroup` rows for every user in the organization — including group IDs, descriptions, `atomic`/`sequential` flags, and any other fields exposed by `TransactionGroupDto`.
4. Attacker iterates the returned group IDs and calls `GET /transaction-groups/:id` for each one. Because `getTransactionGroup` checks whether the caller has any relation to the group's transactions, this second call will be blocked — but the listing endpoint already leaked the group metadata.

### Impact Explanation
Any authenticated user gains full enumeration of all transaction groups created by all other users in the organization. Exposed data includes group IDs, descriptions, and structural metadata (atomic/sequential execution flags). Group IDs can be used as oracle input for further targeted probing. In a multi-user organization handling sensitive Hedera network transactions (account management, HBAR transfers, file operations), leaking the existence and structure of other users' transaction groups constitutes a cross-tenant confidentiality breach.

### Likelihood Explanation
Exploitation requires only a valid, verified user account — the lowest possible privilege level in the system. The endpoint is reachable over standard HTTPS with no additional preconditions. The comment `/* TESTING ONLY */` confirms the endpoint was never intended for production use, making it likely to be overlooked in security reviews. Any authenticated user, including a malicious insider or a compromised low-privilege account, can trigger this immediately. [1](#0-0) 

### Recommendation
Either remove the endpoint entirely (it is self-described as "TESTING ONLY") or, if a listing endpoint is required for legitimate use, restrict it to admins and add user-scoping:

```typescript
// Option A: Remove the endpoint entirely.

// Option B: Restrict to admins only
@Get()
@UseGuards(AdminGuard)
@Serialize(TransactionGroupDto)
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroups();
}

// Option C: Scope to the requesting user
@Get()
@Serialize(TransactionGroupDto)
getTransactionGroups(@GetUser() user: User): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroupsForUser(user);
}
```

The `AdminGuard` is already implemented and used elsewhere in the codebase: [6](#0-5) 

### Proof of Concept
```
# Step 1: Authenticate as any verified user
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Call the unguarded listing endpoint
GET /transaction-groups
Authorization: Bearer <JWT>

# Expected (vulnerable) response:
HTTP 200
[
  { "id": 1, "description": "Finance Q1 Transfers", "atomic": true, ... },
  { "id": 2, "description": "HR Payroll Batch",     "atomic": false, ... },
  ...  ← all groups from all users
]
```

The attacker receives the full list of every transaction group in the organization without any ownership relationship to those groups.
