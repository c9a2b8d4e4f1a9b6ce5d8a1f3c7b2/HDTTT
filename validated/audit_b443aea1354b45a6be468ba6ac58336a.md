Audit Report

## Title
Unbounded Transaction Fetch in `getTransactionsToSign` Enables Authenticated DoS via Memory and CPU Exhaustion

## Summary
`TransactionsService.getTransactionsToSign` fetches every non-terminal transaction from the database without a `take` (row limit) clause, then iterates over the full result set calling `userKeysToSign` per row — which performs SDK deserialization on each row — before applying in-memory pagination. Any verified user can trigger `GET /transactions/sign` to force the server to load and process an unbounded number of rows, exhausting memory and CPU. The existing rate limiter (100 req/min) does not bound the per-request cost, which scales with total transaction count.

## Finding Description

**Root cause — missing DB-level limit in `getTransactionsToSign`**

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `repo.find` call has no `take` or `skip` parameter:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no take / no skip
});
``` [1](#0-0) 

Every non-terminal `Transaction` row (with `groupItem` relation) is loaded into the Node.js heap. The loop then calls `userKeysToSign` for each row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

`userKeysToSign` delegates to `keysRequiredToSign`, which performs `SDKTransaction.fromBytes(transaction.transactionBytes)` (full SDK deserialization) and a `transactionSignatureService.computeSignatureKey` call for every row: [3](#0-2) 

**Pagination guard does not protect the DB query**

`PaginationParams` caps `size` at 100 and throws `BadRequestException(ErrorCodes.IPP)` for larger values: [4](#0-3) 

But this limit is applied only to the final `result.slice(offset, offset + limit)`: [5](#0-4) 

It has no effect on how many rows are fetched from the database or how many iterations of the `userKeysToSign` loop execute.

**Contrast with other paginated methods**

Every other paginated method in the same service correctly passes `skip: offset, take: limit` to the ORM:
- `getTransactions` — `skip: offset, take: limit`
- `getHistoryTransactions` — `skip: offset, take: limit`
- `getTransactionsToApprove` — `skip: offset, take: limit` [6](#0-5) [7](#0-6) 

Only `getTransactionsToSign` omits these.

**Attacker-controlled entry point**

The endpoint is live and reachable by any verified user, and is explicitly noted as no longer used by the front-end (meaning it receives no front-end-side safeguards): [8](#0-7) 

**Rate limiter does not bound per-request cost**

A `UserThrottlerModule` exists with limits of 100 req/min and 10 req/sec: [9](#0-8) 

However, the throttler bounds request *frequency*, not per-request *cost*. A single request with N transactions in the database costs O(N) memory and O(N) CPU regardless of the rate limit. With thousands of accumulated transactions, even one request per minute is sufficient to cause sustained resource exhaustion.

## Impact Explanation

- **Memory exhaustion**: All non-terminal `Transaction` rows (with `groupItem` relation) are loaded into the Node.js heap per request. With thousands of rows, a single request can consume hundreds of MB.
- **CPU exhaustion**: `keysRequiredToSign` performs `SDKTransaction.fromBytes(transaction.transactionBytes)` and `computeSignatureKey` once per row in a sequential `for` loop. Per-request CPU cost is proportional to total transaction count.
- **Service degradation / crash**: Sustained calls cause the NestJS API process to OOM-crash or become unresponsive, denying service to all organization users — including legitimate signers and approvers who depend on the API for multi-sig workflows.

## Likelihood Explanation

- **Precondition**: A valid, verified organization account. Registration is a normal product flow; no privileged access is required.
- **Amplification**: The attacker creates transactions (valid product flow, no per-user cap found in the codebase) to grow the unbounded list, then polls the endpoint. The cost to the attacker is low (one HTTP request per attack cycle); the cost to the server scales with total transaction count.
- **Rate limiter insufficient**: The throttler (100 req/min) does not bound the per-request DB and CPU cost. A single request against a large transaction table is sufficient to cause significant degradation.
- **Endpoint still exposed**: The `/* NO LONGER USED BY FRONT-END */` comment confirms the endpoint is a dead code path from the product perspective, meaning it is unlikely to receive monitoring attention or be proactively hardened.

## Recommendation

1. **Add `take: limit, skip: offset` to the `repo.find` call in `getTransactionsToSign`**, consistent with all other paginated methods in the same service. This bounds the DB query to the requested page size.
2. Because `userKeysToSign` filtering must happen in-application (it cannot be pushed to SQL without significant refactoring), consider a two-pass approach: fetch a bounded window of candidate rows, filter, and if the page is not full, fetch the next window — or pre-filter at the DB level using a subquery similar to `getTransactionsToApprove`.
3. Consider removing or formally deprecating the `GET /transactions/sign` endpoint if it is confirmed to be unused by the front-end, to reduce attack surface.

## Proof of Concept

```
# Step 1: Register and verify a user account (normal product flow)
# Step 2: Create N transactions via POST /transactions (no per-user cap enforced)
for i in $(seq 1 5000); do
  curl -s -X POST http://api/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"name":"tx'$i'","transactionBytes":"...","creatorKeyId":1,...}'
done

# Step 3: Issue a single GET /transactions/sign request
# The server loads all N rows, deserializes each transaction via SDKTransaction.fromBytes,
# and calls computeSignatureKey per row before slicing to page size=1.
curl -s "http://api/transactions/sign?page=1&size=1" \
  -H "Authorization: Bearer $TOKEN"

# Result: Server processes O(N) rows and O(N) SDK deserializations for a page-1 response.
# Repeated at the throttle limit (100/min), this sustains continuous resource exhaustion.
```
