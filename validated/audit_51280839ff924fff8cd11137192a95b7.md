Audit Report

## Title
Sequential Transaction Group Execution Permanently Blocked When Any Member Transaction Changes State (TOCTOU)

## Summary
`executeTransactionGroup` in `execute.service.ts` validates every transaction in a group against the live database before executing any of them. If any single transaction's status has changed since the group was loaded into memory (e.g., canceled by its creator or expired by the background cron), the entire group throws and is silently abandoned — no other transaction in the group is ever executed.

## Finding Description

**Vulnerability class**: State-transition / TOCTOU batch processing.

**Verified code path**:

**Step 1** — `handleTransactionsBetweenNowAndAfterThreeMinutes` (cron, every 10 s) calls `prepareTransactions`, which loads the full `TransactionGroup` from the DB — including all `groupItem.transaction` objects with their current statuses — and schedules `collateGroupAndExecute`. [1](#0-0) 

**Step 2** — `collateGroupAndExecute` registers a `setTimeout` that fires at `validStart − 10 s`, then calls `addGroupExecutionTimeout`, which registers a second `setTimeout` that fires at `validStart + 5 s` and calls `executeService.executeTransactionGroup(transactionGroup)`. [2](#0-1) 

**Step 3** — Inside `executeTransactionGroup`, the group items are first filtered using the **stale in-memory** `transaction.status` field: [3](#0-2) 

**Step 4** — For every item that passed the in-memory filter, `getValidatedSDKTransaction` is called, which immediately re-reads the status **from the database** via `validateTransactionStatus`: [4](#0-3) 

**Step 5** — If the DB status is anything other than `WAITING_FOR_EXECUTION` (e.g., `CANCELED`, `EXPIRED`, `REJECTED`), `validateTransactionStatus` throws. That exception is caught and **re-thrown** as a group-level error, aborting the entire loop: [5](#0-4) 

**Step 6** — The group-level error propagates to `addGroupExecutionTimeout`'s callback, where it is silently swallowed with `console.log`. The timeout is deleted in the `finally` block and is never re-registered: [6](#0-5) 

**TOCTOU window**: The group is loaded at cron time (T1). The execution callback fires at `validStart + 5 s` (T2). Any status change to any group member between T1 and T2 causes the entire group to fail.

**Natural trigger — expiry cron**: `handleExpiredTransactions` runs every 10 seconds and sets `WAITING_FOR_EXECUTION` transactions to `EXPIRED` when `validStart < now − 3 min`. Under scheduler load or clock skew, the execution callback can fire after the 3-minute window, causing the expiry cron to race ahead and expire one or more group members, triggering the group-level throw. [7](#0-6) 

**Adversarial trigger**: A user who is the creator of any one transaction in a sequential group can cancel that transaction in the window between group scheduling and execution. The cancellation changes the DB status to `CANCELED`. When the execution callback fires, `validateTransactionStatus` detects the mismatch and throws, blocking all other transactions in the group from executing.

## Impact Explanation
All transactions in a sequential group are permanently abandoned when any single member changes state. The group execution timeout is deleted and never rescheduled. Affected transactions remain in `WAITING_FOR_EXECUTION` until the expiry cron eventually marks them `EXPIRED` (up to 3 minutes later). Users lose the ability to execute their transactions with no error surfaced to them — only a `console.log` on the server side. In a multi-user group scenario, one user canceling their own transaction silently blocks all other users' transactions from executing.

**Severity: Medium** — permanent loss of transaction execution for all group members, triggered by a single member's state change. No funds are directly stolen, but the integrity of the transaction lifecycle is broken.

## Likelihood Explanation
- **Natural race**: The expiry cron runs every 10 seconds against `WAITING_FOR_EXECUTION` transactions. Under scheduler load or clock skew, the execution callback can fire after the 3-minute expiry window, causing a natural collision.
- **Adversarial**: Any legitimate user who is the creator of one transaction in a group can deliberately cancel it in the scheduling window to block the entire group. No privileged access is required — only a valid user account with a transaction in the group. Canceling a transaction is a normal in-app workflow, so this is not excluded by the SECURITY.md out-of-scope rules.

## Recommendation
1. **Per-item error handling**: In the validation loop inside `executeTransactionGroup`, catch per-transaction errors and skip (or individually mark as failed) the offending transaction rather than re-throwing a group-level error. Only abort the group if the group is explicitly `atomic`.
2. **Re-fetch before execution**: Re-query each transaction's status from the DB immediately before the execution loop rather than relying on the stale in-memory snapshot loaded at cron time.
3. **Surface errors to users**: Replace the silent `console.log` swallow in `addGroupExecutionTimeout` with proper status updates (e.g., emit a `FAILED` status event) so users are notified when their transactions are not executed.
4. **Distinguish sequential vs. atomic semantics**: For `sequential` groups (as opposed to `atomic`), a single member's cancellation or expiry should not block the remaining valid members.

## Proof of Concept
1. User A and User B each have one transaction in a `sequential` `TransactionGroup` with `validStart` 2 minutes in the future.
2. The `handleTransactionsBetweenNowAndAfterThreeMinutes` cron fires, loads the group (both transactions in `WAITING_FOR_EXECUTION`), and schedules `collateGroupAndExecute`.
3. User A calls the `cancelTransaction` API on their transaction. The DB status changes to `CANCELED`.
4. The `collateGroupAndExecute` timeout fires at `validStart − 10 s`. The in-memory filter at line 65–67 of `execute.service.ts` still passes User A's transaction (stale in-memory status is `WAITING_FOR_EXECUTION`).
5. `getValidatedSDKTransaction` re-reads User A's status from the DB → `CANCELED` → `validateTransactionStatus` throws `"Transaction has been canceled."`.
6. The catch block at line 76–80 of `execute.service.ts` re-throws: `"Transaction Group cannot be submitted. Error validating transaction <A.id>: Transaction has been canceled."`.
7. `addGroupExecutionTimeout`'s callback catches this at line 341–342 of `transaction-scheduler.service.ts`, logs it, and deletes the timeout in `finally`.
8. User B's transaction is never executed. It remains in `WAITING_FOR_EXECUTION` until `handleExpiredTransactions` marks it `EXPIRED` up to 3 minutes later. No notification is sent to User B.

  addGroupExecutionTimeout(transactionGroup: TransactionGroup) {
    const name = `group_execution_timeout_${transactionGroup.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart =
      transactionGroup.groupItems[0].transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransactionGroup(transactionGroup);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }

  private async validateTransactionStatus(transaction: Transaction) {
    const { status } = await this.transactionsRepo.findOne({
      where: { id: transaction.id },
      select: ['status'],
    });

    switch (status) {
      case TransactionStatus.NEW:
        throw new Error('Transaction is new and has not been signed yet.');
      case TransactionStatus.FAILED:
        throw new Error('Transaction has already been executed, but failed.');
      case TransactionStatus.EXECUTED:
        throw new Error('Transaction has already been executed.');
      case TransactionStatus.REJECTED:
        throw new Error('Transaction has already been rejected.');
      case TransactionStatus.EXPIRED:
        throw new Error('Transaction has been expired.');
      case TransactionStatus.CANCELED:
        throw new Error('Transaction has been canceled.');
      case TransactionStatus.ARCHIVED:
        throw new Error('Transaction is archived.');
    }
  }
