### Title
Plaintext Password and Recovery Phrase Words Stored in Pinia Renderer Store

### Summary
The Electron front-end stores the user's plaintext application password and plaintext mnemonic recovery phrase words in the Vue/Pinia reactive store (`storeUser`). Both secrets are accessible from the renderer process's reactive state — directly analogous to the reported `wallet.walletPassword` / `wallet.identities[0].secretKey` exposure in the Redux store.

### Finding Description

**Secret 1 — Plaintext password in Pinia store**

The `LoggedInUserWithPassword` type explicitly carries a `password: string` field: [1](#0-0) 

`setPassword` writes the plaintext password directly into the reactive `personal` ref: [2](#0-1) 

`getPassword` returns it as a plain string and extends its TTL on every read: [3](#0-2) 

The `personal` ref is exported from the store and is therefore part of the observable Pinia state: [4](#0-3) 

**Secret 2 — Plaintext recovery phrase words in Pinia store**

The `RecoveryPhrase` type stores the raw word list: [5](#0-4) 

`createRecoveryPhrase` embeds the plaintext `words` array into the object that is placed in the store: [6](#0-5) 

`setRecoveryPhrase` writes this object (including `words`) into the reactive `recoveryPhrase` ref: [7](#0-6) 

The `recoveryPhrase` ref is also exported from the store: [8](#0-7) 

### Impact Explanation
The application password is the sole protection for all locally stored private keys (AES-256-GCM encrypted with the password via `encrypt()` in `front-end/src/main/utils/crypto.ts`). The recovery phrase words allow full re-derivation of every HD key pair. An attacker who can read the renderer process memory — via Vue DevTools, a malicious browser extension injected into the Electron renderer, a compromised npm dependency with renderer access, or a Spectre/Meltdown-class side-channel — can extract both secrets and use them to steal all funds controlled by the wallet. [9](#0-8) 

### Likelihood Explanation
Electron renderer processes share the Chromium renderer attack surface. Malicious npm packages with renderer-side code execution, prototype-pollution gadgets, or XSS in any loaded content can reach the Pinia store. Vue DevTools (enabled in development builds) exposes the full store state in the browser console. The password is kept alive for up to 10 minutes by default (`passwordStoreDuration = 10 * 60 * 1_000`), and the recovery phrase persists for the entire session once set. [10](#0-9) 

### Recommendation
- **Short term:** Do not store the plaintext password or recovery phrase words in the Pinia store. Store only a transient, scoped reference (e.g., pass the password as a function argument directly to the IPC call that needs it, then discard it). If caching is required, keep it only in a closure or a `WeakRef` that is not part of the reactive state graph.
- **Long term:** Move all key-decryption operations exclusively to the Electron Main process (already partially done via IPC). The renderer should never hold the plaintext password or mnemonic words; it should only trigger a Main-process operation that decrypts, uses, and immediately zeroes the secret.

### Proof of Concept
In a running development build, open the Electron DevTools console in the renderer window and execute:

```js
// Access the Pinia store directly from the renderer
const store = window.__pinia.state.value.user;

// Plaintext password (valid within the passwordStoreDuration window)
console.log(store.personal.password);

// Plaintext recovery phrase words
console.log(store.recoveryPhrase?.words);
```

Both values are readable as plaintext strings from the reactive Pinia state, mirroring the `wallet.walletPassword` / `wallet.identities[0].secretKey` exposure described in the external report.

export const createRecoveryPhrase = async (words: string[]): Promise<RecoveryPhrase> => {
  try {
    const mnemonic = await Mnemonic.fromWords(words);
    const hash = await hashData(getRecoveryPhraseHashValue(words), true);

    return {
      mnemonic,
      words,
      hash,
    };
  } catch {
    throw Error('Invalid recovery phrase');
  }
};

export function encrypt(data: string, password: string) {
  const iv = crypto.randomBytes(16);
  const salt = crypto.randomBytes(64);

  const key = deriveKey(password, salt);

  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([cipher.update(data, 'utf8'), cipher.final()]);

  const tag = cipher.getAuthTag();

  return Buffer.concat([salt, iv, tag, encrypted]).toString('base64');
}
