Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated-User DoS Against All API Consumers

## Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction in the database with no SQL `LIMIT`, then performs an expensive per-row async operation (transaction-bytes deserialization + potential mirror-node HTTP call) before applying pagination in memory. Any authenticated user can pre-populate the database with long-lived transactions, causing every subsequent call to `GET /transactions/sign` to consume unbounded server resources.

## Finding Description

**Root cause — missing `take` on the repository query:**

The `repo.find()` call at line 295 carries no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // filters only by status, NOT by requesting user
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

`whereForUser` excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope the query to the requesting user's transactions — every active transaction in the entire system is returned. [2](#0-1) 

The result set is then iterated one transaction at a time with a costly async call per row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [3](#0-2) 

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` → `computeSignatureKey`, which:
1. Deserializes raw transaction bytes via `SDKTransaction.fromBytes(transaction.transactionBytes)`
2. Calls `accountCacheService.getAccountInfoForTransaction()` — which, on a cache miss, makes an outbound mirror-node HTTP call [4](#0-3) [5](#0-4) 

Pagination is applied only **after** all rows are processed:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // post-hoc slice
  ...
};
``` [6](#0-5) 

Compare this to `getTransactionsToApprove`, which correctly applies `skip`/`take` at the database level before any per-row processing: [7](#0-6) 

The endpoint is still a live, registered route despite the `/* NO LONGER USED BY FRONT-END */` comment: [8](#0-7) 

**No per-user rate limit on `POST /transactions`:**

The `UserThrottlerModule` is defined but is **not** imported in `api.module.ts`. Only `IpThrottlerModule` and `EmailThrottlerModule` are imported. The global `APP_GUARD` registers only `IpThrottlerGuard`, not `UserThrottlerGuard`. An attacker can create transactions up to the IP-rate limit, or spread creation across time/IPs. [9](#0-8) 

## Impact Explanation

- **Service degradation / unavailability**: With O(N) deserialization operations and O(N) potential mirror-node HTTP calls per request, response time grows linearly with the number of attacker-created transactions. At a few thousand rows the endpoint becomes effectively unusable for all users.
- **Scope is global**: Because the DB query is not scoped to the requesting user, a single attacker's transactions degrade the endpoint for every user in the organization.
- **Persistent**: Transactions with far-future `validStart` values remain in `WAITING_FOR_SIGNATURES` status indefinitely, so the attacker does not need to remain online after the initial batch creation.
- **Note on cache mitigation**: `AccountCacheService` does consult a DB-backed cache before making outbound HTTP calls. However, the deserialization (`SDKTransaction.fromBytes`) still occurs for every row regardless of cache state, and cache misses (e.g., unique fee-payer accounts per transaction) still trigger HTTP calls.

## Likelihood Explanation

- Attacker precondition: a valid, verified user account — the lowest privilege level in the system.
- No special tooling required: a simple loop calling `POST /transactions` with incrementing `validStart` timestamps.
- The `createTransaction` endpoint has no per-user rate limit (only IP-based throttling via `IpThrottlerGuard`).
- The attack is silent and persistent; it does not require the attacker to remain online after the initial batch.

## Recommendation

1. **Apply SQL-level pagination before per-row processing** in `getTransactionsToSign`, analogous to how `getTransactionsToApprove` uses `skip`/`take` in the `FindManyOptions`. This is the primary fix.
2. **Scope the query to the requesting user** where possible (e.g., filter by `creatorKey.userId` or by observer/signer relationships) to reduce the candidate set.
3. **Add a per-user rate limit** on `POST /transactions` by importing `UserThrottlerModule` and applying `UserThrottlerGuard` to the `createTransaction` endpoint.
4. **Consider removing or disabling** the `GET /transactions/sign` endpoint entirely if it is confirmed to be unused by the front-end, to reduce attack surface.

## Proof of Concept

```
# Step 1: Obtain a verified JWT for a normal user account
POST /auth/login  →  { accessToken: "..." }

# Step 2: Create N transactions with far-future validStart values
# (repeat with incrementing timestamps to avoid duplicate transactionId rejection)
for i in $(seq 1 5000); do
  curl -X POST https://target/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -d '{"transactionBytes":"<valid_hex>","creatorKeyId":1,"mirrorNetwork":"testnet",
         "validStart":"2099-01-01T00:00:0'$i'.000Z","name":"flood","signature":"<sig>"}'
done

# Step 3: Any user (including legitimate users) calling GET /transactions/sign
# now triggers a full-table scan of all N rows + N deserialization operations
# + up to N mirror-node HTTP calls before any pagination is applied.
GET /transactions/sign?page=1&size=10
# → Server processes all 5000 rows before returning 10 items
```
