### Title
`cutoffAt` Field Stored But Never Enforced — Signers Can Submit Signatures Past the Creator-Defined Deadline

### Summary
The `Transaction` entity exposes a `cutoffAt` field, documented as "The timestamp at which the transaction can no longer be signed by the signers." This field is accepted in `CreateTransactionDto`, persisted to the database, and surfaced in API responses. However, the signing validation path (`SignersService.validateTransactionStatus`) never checks `cutoffAt`, so any authorized signer can submit valid signatures after the cutoff deadline has passed, violating the creator's explicit intent.

### Finding Description

**Root cause — field stored, never enforced:**

The `cutoffAt` column is declared on the `Transaction` entity: [1](#0-0) 

It is accepted and transformed in `CreateTransactionDto` (confirmed by the DTO spec test): [2](#0-1) 

The database documentation explicitly states its purpose: [3](#0-2) 

**The signing validation path that should enforce it:**

`SignersService.validateTransactionStatus` is the gatekeeper before a signature is accepted. It checks only two conditions — whether the transaction is `CANCELED`, and whether `isExpired()` returns true (which checks the Hedera SDK validity window, not `cutoffAt`): [4](#0-3) 

There is no branch in `validateTransactionStatus`, no guard, and no middleware anywhere in the signing flow that reads `cutoffAt` and rejects a signature when `Date.now() > transaction.cutoffAt`. The field is set (analogous to `stopped = true` in the external report) but the check that would enforce it (analogous to `isClosed()`) is simply absent.

**Exploit path:**

1. Creator calls `POST /transactions` with `cutoffAt` set to a near-future timestamp (e.g., 60 seconds from now), intending to close the signing window early.
2. The value is stored in the database.
3. After `cutoffAt` elapses, an authorized signer calls `POST /transactions/:id/signers` with a valid `signatureMap`.
4. `validateTransactionStatus` checks only CANCELED status and `isExpired()` — both pass — and the signature is accepted.
5. The creator's cutoff deadline is silently ignored.

### Impact Explanation

The `cutoffAt` field is the only mechanism by which a transaction creator can enforce a signing deadline shorter than the Hedera network's own validity window. Because it is never checked server-side, the invariant "no new signatures after `cutoffAt`" is permanently broken. Any signer who is still authorized (i.e., the transaction is in `WAITING_FOR_SIGNATURES` status) can add their signature at any time, regardless of the cutoff. This is a state-transition integrity failure: the creator's declared policy is stored but never enforced.

### Likelihood Explanation

Likelihood is high. No privilege escalation is required — any user who is a legitimate signer on the transaction can exploit this by simply submitting a signature after the `cutoffAt` timestamp. The API endpoint `POST /transactions/:id/signers` is reachable by any authenticated organization member who holds a required key. The attacker needs only to wait for the cutoff to pass and then submit a normal signing request.

### Recommendation

In `SignersService.validateTransactionStatus` (or immediately before the signature is persisted), add an explicit check:

```typescript
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  return ErrorCodes.TC; // Transaction cutoff exceeded
}
```

This mirrors the existing `isExpired()` check and closes the gap between the stored field and its intended enforcement.

### Proof of Concept

1. Authenticate as a creator and call `POST /transactions` with body including `cutoffAt: "<now + 30 seconds>"`.
2. Confirm the transaction is created with `status = WAITING_FOR_SIGNATURES`.
3. Wait 31 seconds (past `cutoffAt`).
4. Authenticate as an authorized signer and call `POST /transactions/:id/signers` with a valid `signatureMap`.
5. **Expected (per documentation):** `400` error — cutoff has passed.
6. **Actual:** `200` — signature is accepted; `cutoffAt` is never read by the server during signature validation. [1](#0-0) [5](#0-4)
