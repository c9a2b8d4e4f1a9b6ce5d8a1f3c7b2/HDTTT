### Title
Recovery Phrase Validation Inconsistency Allows Silent Import of Non-Restorable 12-Word Mnemonics

### Summary

`validateMnemonic` accepts any word count that the Hedera SDK's `Mnemonic.fromWords()` accepts (including standard BIP-39 12-word mnemonics), while `restorePrivateKey` hard-codes a `words.length !== 24` guard. A user who pastes a valid 12-word BIP-39 mnemonic into the Import component receives no error — the phrase is stored — but every subsequent attempt to derive a key from it silently throws "Invalid Recovery Phrase". The user believes their keys are backed up, but they are not recoverable.

### Finding Description

**Root cause — split validation logic:**

`validateMnemonic` delegates entirely to the SDK with no word-count check: [1](#0-0) 

`restorePrivateKey` enforces exactly 24 words, but only at key-derivation time: [2](#0-1) 

**Trigger path — paste in `Import.vue`:**

The paste handler splits clipboard text, filters empty tokens, and slices to `WORD_COUNT` (24). If the user pastes a 12-word phrase, `mnemonic` is a 12-element array: [3](#0-2) 

`validateMnemonic(12_words)` returns `true` (the Hedera SDK's `Mnemonic.fromWords` accepts standard BIP-39 12-word mnemonics). The watcher then calls `user.setRecoveryPhrase(12_words)`: [4](#0-3) 

`setRecoveryPhrase` calls `createRecoveryPhrase`, which again calls `Mnemonic.fromWords` with no word-count guard — succeeds, stores the 12-word phrase: [5](#0-4) 

**Second trigger path — data migration:**

`decryptMigrationMnemonic` returns `words.split(' ')` with no length validation. A legacy tool that stored a 12-word mnemonic produces a 12-element array, which is passed directly to `createRecoveryPhrase`: [6](#0-5) [7](#0-6) 

**Failure at key restoration:**

Every code path that derives a private key — `RestoreStep.vue`, `RestoreMissingKeys.vue`, `BeginKeysImport.vue`, `SetupPersonal.vue` — calls `restorePrivateKey`, which throws immediately for any non-24-word phrase: [2](#0-1) 

### Impact Explanation

A user who imports a valid 12-word BIP-39 mnemonic (e.g., from another wallet, or migrated from the legacy Hedera Transaction Tools desktop app) sees no error during import. The phrase is stored and displayed as valid. When the user later attempts to restore or derive any key pair, every call to `restorePrivateKey` throws "Invalid Recovery Phrase". The user has no working key backup. If the original private key is lost, the funds/account are permanently unrecoverable. This maps to **permanent lock/freeze of user state** caused by a reachable user action.

### Likelihood Explanation

The paste handler in `Import.vue` is an explicitly supported workflow. BIP-39 12-word mnemonics are the default for many wallets (MetaMask, Trust Wallet, etc.). A user migrating from such a wallet, or from the legacy Hedera Transaction Tools app (which the migration feature explicitly targets), will naturally attempt to paste their existing 12-word phrase. No privilege is required; the attacker is the user themselves, acting within normal product flows.

### Recommendation

Add a word-count check inside `validateMnemonic` (or as a separate pre-check before calling it) that rejects any phrase that is not exactly 24 words:

```ts
export const validateMnemonic = async (words: string[]) => {
  if (words.length !== 24) return false;
  try {
    words = words.map(w => w.toLocaleLowerCase());
    await Mnemonic.fromWords(words);
    return true;
  } catch {
    return false;
  }
};
```

Apply the same guard inside `decryptMigrationMnemonic` before returning the word array, and surface a clear error to the user if the decrypted legacy mnemonic does not contain exactly 24 words.

### Proof of Concept

1. Open the Hedera Transaction Tool desktop app.
2. Navigate to any recovery phrase import screen (Account Setup → Import, Settings → Restore Key, or Restore Missing Keys).
3. Copy a valid 12-word BIP-39 mnemonic to the clipboard (e.g., generate one with any standard BIP-39 tool).
4. Click into the first word field and paste (`Ctrl+V`).
5. Observe: the 12 words are accepted, `isMnemonicValid` becomes `true`, and `user.recoveryPhrase` is set with 12 words. No error is shown.
6. Proceed to the key restoration step (e.g., click "Continue" on the Restore Key page).
7. Observe: `restorePrivateKey` is called with the 12-word array, immediately throws `Error('Invalid Recovery Phrase')`, and no key is derived.
8. The user's recovery phrase is stored in the app state as "valid" but is permanently unusable for key derivation.

export async function decryptMigrationMnemonic(password: string): Promise<string[] | null> {
  const content = await fs.promises.readFile(getPropertiesPath(), { encoding: 'utf-8', flag: 'r' });
  const parsedContent = parseUserProperties(content);

  const token = parsedContent.hash;
  if (!token) throw Error('No hash found at location');

  const isLegacy = parsedContent.legacy;

  let words: string | null;
  if (isLegacy) {
    words = await decryptLegacyMnemonic(getMnemonicPath(), password);
  } else {
    words = await decryptMnemonic(getMnemonicPath(), token, password);
  }

  if (words) return words.split(' ');

  return null;
}
