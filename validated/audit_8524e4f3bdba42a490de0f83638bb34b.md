Audit Report

## Title
Any Authenticated User Can Delete Approvers from Any Transaction They Do Not Own (IDOR)

## Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. The two checks are fully decoupled, enabling any authenticated user to destroy the approval tree of any transaction in the system.

## Finding Description

**Root cause — authorization check and mutation target are decoupled**

In `approvers.controller.ts`, the delete handler is: [1](#0-0) 

```typescript
@Delete('/:id')
async removeTransactionApprover(...) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
  return true;
}
```

`getCreatorsTransaction` only confirms the caller owns `transactionId`: [2](#0-1) 

`removeTransactionApprover(id)` fetches the approver by `id` and deletes it — it never verifies the approver's `transactionId` matches the route param: [3](#0-2) 

`removeNode` then performs a recursive CTE soft-delete of the entire subtree rooted at that approver: [4](#0-3) 

**The fix already exists in the sibling handler.** `updateTransactionApprover` correctly performs the cross-check — it walks up to the root node and verifies `rootNode.transactionId === transactionId` before proceeding: [5](#0-4) 

The delete handler simply omits this same guard.

## Impact Explanation

- **Unauthorized state mutation**: Any authenticated user can delete any approver record from any transaction in the system, regardless of ownership.
- **Approval workflow destruction**: Deleting a root approver node triggers `removeNode`, which recursively soft-deletes the entire approval tree via the CTE query. A single request can wipe all approvers for a transaction.
- **Multi-signature bypass**: If a transaction requires M-of-N approvals and an attacker deletes enough approver nodes, the threshold requirement is undermined or the transaction is left in an inconsistent state.
- **Permanent disruption**: Soft-deleted approvers are not automatically restored. The transaction creator must manually recreate the entire approval tree.

## Likelihood Explanation

- **Precondition**: The attacker only needs a valid authenticated account — no admin role, no special privilege.
- **Enumeration**: Approver IDs are auto-incremented integers (`id` column on `transaction_approver`). An attacker can trivially enumerate valid IDs by observing their own approver IDs or probing sequentially.
- **No rate limiting**: Nothing in the controller or guards prevents bulk deletion across many transactions in rapid succession.
- **Standard IDOR pattern**: This is a textbook Insecure Direct Object Reference exploitable by any malicious registered user.

## Recommendation

Apply the same cross-check already present in `updateTransactionApprover`. In `removeTransactionApprover` (or in the controller before calling it), walk up to the root node and assert its `transactionId` matches the route param:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

This mirrors the guard already implemented in `updateTransactionApprover` at lines 386–394 of `approvers.service.ts`.

## Proof of Concept

1. Attacker registers an account and creates transaction A (`id=1`). This satisfies `getCreatorsTransaction`.
2. Attacker enumerates or guesses the integer `id` of an approver belonging to victim's transaction B (`id=2`). Because IDs are sequential, enumeration is trivial (e.g., attacker's own approver is `id=10`, so victim's approvers are nearby).
3. Attacker sends:
   ```
   DELETE /transactions/1/approvers/{victim_approver_id}
   Authorization: Bearer <attacker_token>
   ```
4. Controller calls `getCreatorsTransaction(1, attacker)` — passes, attacker owns transaction 1.
5. Controller calls `removeTransactionApprover(victim_approver_id)` — fetches the approver, finds it exists, calls `removeNode`, which recursively soft-deletes the entire approval subtree rooted at that ID.
6. Victim's approval workflow for transaction B is now corrupted or entirely destroyed. No cross-check was ever performed.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
