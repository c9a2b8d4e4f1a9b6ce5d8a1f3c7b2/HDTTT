### Title
Threshold Invariant Broken After Approver Removal Permanently Locks Transaction Approval

### Summary
In `ApproversService.removeTransactionApprover`, when a child approver node is soft-deleted from a threshold tree, the parent node's `threshold` value is never adjusted. This leaves the database in a state where `threshold > remaining_children_count`, making the approval condition permanently unsatisfiable and locking the transaction in a non-terminal status indefinitely. The analog to the external report is exact: a parameter decrease (number of approvers) causes a stored invariant (threshold) to become inconsistent with the new reality, permanently blocking a user operation.

### Finding Description

**Root Cause:**

`removeTransactionApprover` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` calls `removeNode` to soft-delete an approver and its subtree, but never reads or adjusts the parent node's `threshold` column. [1](#0-0) 

`removeNode` performs a recursive soft-delete of the target node and all its descendants, but touches no other rows: [2](#0-1) 

**Contrast with the correct path:**

`updateTransactionApprover` (the "detach child" path, `dto.listId = null`) explicitly re-reads the parent's child count and lowers the threshold when it would exceed the new count: [3](#0-2) 

`removeTransactionApprover` has no equivalent adjustment.

**Exploit Flow:**

1. Transaction creator creates a threshold tree: parent node with `threshold = 2`, children = [User A, User B].
2. Creator calls `DELETE /transactions/:txId/approvers/:approverIdA` → `removeTransactionApprover(approverIdA)`.
3. `removeNode` soft-deletes User A's row. Parent row still holds `threshold = 2`.
4. Only User B remains. The approval condition now requires 2 approvals from 1 possible approver — permanently unsatisfiable.
5. The transaction stays in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely; no combination of approvals can satisfy `threshold = 2` with one approver.

### Impact Explanation

A transaction is permanently locked in a non-terminal status. No approver can ever satisfy the threshold condition, so the chain service will never advance the transaction to `EXECUTED`. The creator's funds/operations tied to that transaction are frozen. This matches the RESEARCHER.md impact category: *"Permanent lock, freeze, or unrecoverable corruption of user/project state."* [4](#0-3) 

### Likelihood Explanation

The trigger path is a normal, documented product workflow: the transaction creator removes an approver via the approvers API. No privileged access beyond being the transaction creator is required. The `removeTransactionApprover` service method itself contains no authorization guard: [5](#0-4) 

The inconsistency is easy to reach accidentally (creator removes one approver without realizing the threshold is now unsatisfiable) or intentionally (creator deliberately locks a shared transaction). The `updateTransactionApprover` path already demonstrates the developers are aware the threshold must be adjusted when child count drops — the fix exists in one code path but is absent in the other. [6](#0-5) 

### Recommendation

In `removeTransactionApprover`, after calling `removeNode`, load the deleted approver's parent (if any) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If `newChildCount === 0`: soft-delete the parent node as well (it is now an empty threshold node).
- If `newChildCount < parent.threshold`: set `parent.threshold = newChildCount`.

Wrap the removal and the threshold update in a single database transaction to prevent partial state.

### Proof of Concept

**Setup:**
```
POST /transactions/:txId/approvers
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [{ "userId": 10 }, { "userId": 11 }]
  }]
}
```
Parent row: `id=5, threshold=2`. Children: `id=6 (userId=10)`, `id=7 (userId=11)`.

**Trigger:**
```
DELETE /transactions/:txId/approvers/6
```

**Resulting DB state:**
- Row `id=6` → `deletedAt = now()` (soft-deleted).
- Row `id=5` → `threshold = 2` (unchanged).
- Row `id=7` → still active.

**Observed:** Only User 11 can approve. Threshold requires 2. Approval condition is `2 of 1` — permanently unsatisfiable. Transaction never advances past `WAITING_FOR_SIGNATURES`.

**Expected:** `threshold` on row `id=5` should be automatically reduced to `1` (matching the `updateTransactionApprover` path at lines 423–427). [7](#0-6)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);
