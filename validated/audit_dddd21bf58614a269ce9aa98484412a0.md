## Title
Single-User Approval Satisfies Multi-Party Threshold via Duplicate `userId` Across Approver Tree Branches

## Summary
The `isNode` duplicate-check in `approvers.service.ts` is scoped to a single parent node (`listId`), not globally across the tree. This allows the same `userId` to be inserted at multiple distinct positions in the approval tree. Because `approveTransaction` then bulk-updates **all** `TransactionApprover` rows belonging to that user in a single `UPDATE … WHERE id IN (…)`, one approval action simultaneously satisfies every branch the user inhabits, collapsing a multi-party threshold requirement to a single-user action.

## Finding Description

**Root cause — `isNode` check is scoped to a single parent:**

`isNode` builds its `WHERE` clause using both `listId` and `transactionId`:

```typescript
where: {
  listId: typeof approver.listId === 'number' ? approver.listId : null,
  userId: typeof approver.userId === 'number' ? approver.userId : null,
  threshold: ...,
  transactionId: typeof approver.listId === 'number' ? null : transactionId,
},
``` [1](#0-0) 

Because `listId` is part of the uniqueness key, inserting `userA` with `listId=null` (root level) and `userA` with `listId=branchA.id` (child level) produce two different WHERE conditions. Both pass the count check, and both insertions succeed.

**Insertion path — no global uniqueness check:**

`createTransactionApprovers` calls `isNode` before each insert. Because `isNode` is parent-scoped, a creator can submit a tree such as:

```
root (threshold=2)
├── branchA (threshold=1)
│   ├── userA   ← listId=branchA.id  → isNode checks {listId=branchA.id, userId=userA} → 0 rows → allowed
│   └── userB
└── userA       ← listId=null        → isNode checks {listId=null, userId=userA, transactionId=X} → 0 rows → allowed
``` [2](#0-1) 

**Approval path — bulk-updates every record for the user:**

`approveTransaction` calls `getVerifiedApproversByTransactionId`, which uses a recursive CTE to return the full flattened list of all approvers for the transaction: [3](#0-2) 

It then filters by `userId` and bulk-updates all matching rows:

```typescript
const userApprovers = approvers.filter(a => a.userId === user.id);
// ...
.whereInIds(userApprovers.map(a => a.id))
.execute();
``` [4](#0-3) 

**Additional amplifier — retroactive approval propagation:**

During `createTransactionApprovers`, when a `userId` is inserted, the code explicitly copies the existing approval state from any prior record for that user:

```typescript
if (userApproverRecords.length > 0) {
  data.signature = userApproverRecords[0].signature;
  data.userKeyId = userApproverRecords[0].userKeyId;
  data.approved = userApproverRecords[0].approved;
}
``` [5](#0-4) 

This means a creator can also add a user who has **already approved** to a new branch after the fact, and the new record is immediately born with `approved=true`, retroactively satisfying that branch without any further action from the user.

## Impact Explanation

The multi-party approval governance model is completely bypassed. A transaction creator can construct an approver tree that visually advertises an N-of-M threshold requirement but is silently satisfiable by a single user's one-time approval action. Any auditor inspecting the tree structure would see a legitimate-looking threshold that is deterministically defeatable. The transaction proceeds to execution with a fraudulently satisfied approval state.

## Likelihood Explanation

The attacker must be the transaction creator — a normal, unprivileged role reachable by any registered user. No admin keys, leaked secrets, or race conditions are required. The creator has full, legitimate API access to `POST /transactions/:id/approvers` and can craft the malicious tree in a single request. The `isNode` bypass is deterministic and requires no timing or probabilistic exploitation.

## Recommendation

1. **Enforce global `userId` uniqueness per transaction tree.** In `isNode`, remove `listId` from the uniqueness key and instead check whether the `userId` already exists anywhere in the transaction's approver tree (across all branches). A query against the recursive CTE in `getApproversByTransactionId` with a `userId` filter is sufficient.

2. **Remove the approval-state copy-forward logic** at lines 325–329 of `createTransactionApprovers`, or at minimum gate it behind an explicit policy decision, since it allows retroactive satisfaction of newly added branches.

3. **Apply the same global uniqueness check in `updateTransactionApprover`** when a `userId` is changed, to prevent the same bypass via the update path.

## Proof of Concept

```
POST /transactions/42/approvers
{
  "approversArray": [
    {
      "threshold": 2,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            { "userId": 7 },   // userA in branchA — isNode: {listId=branchA.id, userId=7} → 0 → OK
            { "userId": 8 }    // userB
          ]
        },
        { "userId": 7 }        // userA at root — isNode: {listId=root.id, userId=7} → 0 → OK
      ]
    }
  ]
}
```

userA (id=7) now has two `TransactionApprover` rows. userA calls:

```
POST /transactions/42/approvers/approve
{ "userKeyId": ..., "signature": ..., "approved": true }
```

`approveTransaction` executes:
```sql
UPDATE transaction_approver
SET approved = true, signature = ..., userKeyId = ...
WHERE id IN (<branchA-userA-row-id>, <root-userA-row-id>)
```

Both rows are now `approved=true`. The threshold evaluator sees:
- `branchA`: 1 approved child ≥ threshold 1 → **satisfied**
- Root: 2 approved children (branchA + userA-at-root) ≥ threshold 2 → **satisfied**

The entire tree is satisfied by userA's single approval action, despite the tree advertising a 2-of-N multi-party requirement. [6](#0-5)
