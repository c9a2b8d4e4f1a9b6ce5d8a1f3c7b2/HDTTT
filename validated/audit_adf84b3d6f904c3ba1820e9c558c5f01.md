### Title
Any Authenticated User Can Register a Transaction with an Arbitrary Payer Account ID, Enabling transactionId Collision DoS

### Summary
The `POST /transactions` endpoint derives the `transactionId` (format: `payerAccountId@validStart`) directly from user-supplied transaction bytes without verifying that the payer account belongs to the submitting user. Because the uniqueness check rejects any second registration of the same `transactionId`, a malicious authenticated user can pre-register a transaction using a victim's payer account ID and a predicted `validStart`, permanently blocking the victim from registering that specific `(payer, validStart)` combination.

### Finding Description

**Root cause — no payer account ownership check:**

In `validateAndPrepareTransaction`, the only identity check is that the creator's registered public key signed the raw transaction bytes: [1](#0-0) 

After that check passes, the `transactionId` is extracted verbatim from the parsed SDK transaction: [2](#0-1) 

There is no assertion that `sdkTransaction.transactionId.accountId` (the payer) matches any account owned by the submitting user. Any authenticated user can embed any Hedera account ID as the payer in their transaction bytes, sign those bytes with their own registered key, and submit successfully.

**Uniqueness enforcement that makes the collision fatal:**

In `createTransactions`, the service rejects any registration whose `transactionId` already exists in a non-terminal status: [3](#0-2) 

The first registration wins; all subsequent registrations with the same `transactionId` receive a `400 Bad Request`.

**End-to-end exploit path:**

1. Attacker (authenticated user) learns victim's Hedera payer account ID (e.g., `0.0.5000`) — account IDs are public on Hedera.
2. Attacker constructs a valid Hedera SDK transaction (e.g., `AccountCreateTransaction`) with:
   - `transactionId.accountId = 0.0.5000` (victim's payer)
   - `validStart = T` (a timestamp the victim is expected to use, or a sweep of timestamps)
3. Attacker signs the bytes with their own registered key and calls `POST /transactions`.
4. The API validates the attacker's signature (passes), extracts `transactionId = 0.0.5000@T`, finds no existing record, and persists the record.
5. When the victim later calls `POST /transactions` with their own transaction bytes containing `0.0.5000@T`, the uniqueness check fires and returns `400 Transactions already exist: 0.0.5000@T`.

### Impact Explanation

- **Denial of Service on transaction registration**: The victim cannot register any transaction whose `(payer, validStart)` pair has been pre-claimed by the attacker.
- **Organization workflow disruption**: In multi-signature organization mode, a transaction that cannot be registered cannot collect signatures or be executed, stalling collaborative workflows.
- **Attacker's colliding record is harmless on-chain** (it will never execute because the real payer's key won't sign it), but the database slot is permanently occupied until the attacker's record reaches a terminal status (CANCELED/REJECTED/ARCHIVED), after which the victim can retry.

### Likelihood Explanation

- **Preconditions**: Attacker must be a registered, verified user on the same organization server — no admin privilege required.
- **Payer account discoverability**: Hedera account IDs are public; an attacker who knows the victim's organization can enumerate likely payer accounts.
- **validStart predictability**: The Hedera SDK's `Timestamp.generate()` uses the current wall-clock time. An attacker who submits immediately after observing the victim's intent (e.g., via a shared draft or side-channel) can match the timestamp. Nanosecond precision makes blind guessing hard, but a targeted sweep of a ±1-second window (1 billion slots) is impractical without rate limiting — however, even a single successful collision is sufficient to block one specific transaction.
- **Realistic scenario**: Most impactful in organization mode where transaction preparation is collaborative and the attacker is a legitimate participant who can observe draft bytes before the creator submits.

### Recommendation

Add a validation step inside `validateAndPrepareTransaction` that asserts the payer account embedded in the transaction bytes (`sdkTransaction.transactionId.accountId`) corresponds to a Hedera account whose key is registered to the submitting user. Concretely:

```typescript
// After extracting sdkTransaction:
const payerAccountId = sdkTransaction.transactionId.accountId.toString();
const userOwnsPayerAccount = user.keys.some(
  key => /* key is linked to payerAccountId via CachedAccount */
);
if (!userOwnsPayerAccount) {
  throw new BadRequestException(ErrorCodes.PNY); // payer not yours
}
```

Alternatively, enforce that the payer account ID in the transaction bytes must match the account associated with the `creatorKeyId` supplied in the DTO, preventing any cross-account payer spoofing.

  private async validateAndPrepareTransaction(
    dto: CreateTransactionDto,
    user: User,
    client: Client,
  ) {
    const creatorKey = user.keys.find(key => key.id === dto.creatorKeyId);

    if (!creatorKey) {
      throw new BadRequestException(`Creator key ${dto.creatorKeyId} not found`);
    }

    const publicKey = PublicKey.fromString(creatorKey.publicKey);

    // Verify signature
    const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
    if (!validSignature) {
      throw new BadRequestException(ErrorCodes.SNMP);
    }

    // Parse SDK transaction
    const sdkTransaction = SDKTransaction.fromBytes(dto.transactionBytes);

    // Check the transaction is frozen, cannot require it to be frozen, breaks backwards compatibility
    if (!sdkTransaction.isFrozen()) {
      sdkTransaction.freezeWith(client);
    }

    // Check if expired
    if (isExpired(sdkTransaction)) {
      throw new BadRequestException(ErrorCodes.TE);
    }

    // Check size
    if (isTransactionBodyOverMaxSize(sdkTransaction)) {
      throw new BadRequestException(ErrorCodes.TOS);
    }

    // Check nodes
    const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
    if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
      throw new BadRequestException(ErrorCodes.TNVN);
    }

    const transactionHash = await sdkTransaction.getTransactionHash();
    const transactionType = getTransactionTypeEnumValue(sdkTransaction);

    // Extract new keys if applicable
    let publicKeys: string[] | null = null;
    try {
      let keyToExtract: Key | null = null;

      if (transactionType === TransactionType.ACCOUNT_UPDATE) {
        keyToExtract = (sdkTransaction as AccountUpdateTransaction).key;
      } else if (transactionType === TransactionType.NODE_UPDATE) {
        keyToExtract = (sdkTransaction as NodeUpdateTransaction).adminKey;
      } else if (transactionType === TransactionType.NODE_CREATE) {
        keyToExtract = (sdkTransaction as NodeCreateTransaction).adminKey;
      }

      if (keyToExtract) {
        publicKeys = flattenKeyList(keyToExtract).map(pk => pk.toStringRaw());
      }
    } catch (error) {
      // Log but don't fail - publicKeys will remain null
      console.error(`Failed to extract public keys from transaction ${sdkTransaction.transactionId}:`, error);
    }

    return {
      name: dto.name,
      type: transactionType,
      description: dto.description,
      transactionId: sdkTransaction.transactionId.toString(),
      transactionHash: encodeUint8Array(transactionHash),
      transactionBytes: sdkTransaction.toBytes(),
      unsignedTransactionBytes: sdkTransaction.toBytes(),
      creatorKeyId: dto.creatorKeyId,
      signature: dto.signature,
      mirrorNetwork: dto.mirrorNetwork,
      validStart: sdkTransaction.transactionId.validStart.toDate(),
      isManual: dto.isManual,
      cutoffAt: dto.cutoffAt,
      publicKeys,
    };
  }
