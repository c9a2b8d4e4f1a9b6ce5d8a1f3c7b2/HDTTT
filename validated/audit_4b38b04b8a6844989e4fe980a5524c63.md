### Title
`COUNCIL_ACCOUNTS` Hardcoded List Omits the Full HIP-1300 Privileged Payer Range, Causing Incorrect Transaction Size Enforcement

### Summary
The code's own inline comment explicitly states that HIP-1300 defines privileged governance fee payers as accounts `0.0.2` and the range `0.0.42–0.0.799`. However, the hardcoded `COUNCIL_ACCOUNTS` constant only contains three specific accounts (`0.0.2`, `0.0.50`, `0.0.55`), omitting the vast majority of the HIP-1300-specified range. This is a direct analog to the external report: a hardcoded default value in code diverges from the protocol specification it is supposed to implement.

### Finding Description
In `front-end/src/shared/constants/index.ts`, the comment directly references HIP-1300 and states the privileged range:

```
// HIP-1300: privileged governance fee payers (0.0.2 and 0.0.42-0.0.799) get an
// increased transaction size limit of 128 KB to accommodate council signatures.
export const TRANSACTION_MAX_PRIVILEGED_SIZE = 131_072; // in bytes (128 KB)
``` [1](#0-0) 

However, the `COUNCIL_ACCOUNTS` constant used to determine which payers are privileged is defined as:

```ts
export const COUNCIL_ACCOUNTS = ['0.0.2', '0.0.50', '0.0.55'];
``` [2](#0-1) 

The privileged-payer logic in both `front-end/src/renderer/utils/sdk/privilegedPayer.ts` and `back-end/libs/common/src/utils/sdk/privileged-payer.ts` consults `COUNCIL_ACCOUNTS` to decide whether to apply `TRANSACTION_MAX_PRIVILEGED_SIZE` (131,072 bytes) or the standard `TRANSACTION_MAX_SIZE` (6,144 bytes). [3](#0-2) [4](#0-3) 

HIP-1300 grants the 128 KB limit to all accounts in `0.0.42–0.0.799` (758 accounts). The code only recognises 3. Every council account in that range except `0.0.50` and `0.0.55` is silently treated as a non-privileged payer.

### Impact Explanation
Any council account in `0.0.42–0.0.799` (other than `0.0.50` and `0.0.55`) that attempts to submit a governance transaction larger than 6,144 bytes through this tool will have its transaction incorrectly rejected or truncated at the client/server validation layer, even though the Hedera network itself would accept it. This breaks the core governance workflow the tool is designed to support — specifically council-scale transactions with many nested threshold signatures (as acknowledged in test scenario 11.6 and the `TRANSACTION_MAX_PRIVILEGED_SIZE` comment). [5](#0-4) 

### Likelihood Explanation
The mismatch is structural and permanent until corrected. Any council account holder using an account ID in `0.0.42–0.0.799` (excluding `0.0.50`/`0.0.55`) will hit this on every large governance transaction. No attacker action is required — the bug is triggered by normal, legitimate use of the tool by council members.

### Recommendation
Replace the hardcoded three-element list with a range check that matches the HIP-1300 specification:

```ts
// back-end/libs/common/src/constants/index.ts
// Remove COUNCIL_ACCOUNTS list and replace privileged-payer check with:
export function isPrivilegedAccount(accountId: string): boolean {
  // HIP-1300: 0.0.2 and 0.0.42–0.0.799
  const num = parseInt(accountId.split('.')[2], 10);
  return num === 2 || (num >= 42 && num <= 799);
}
```

Apply the same fix symmetrically in `front-end/src/renderer/utils/sdk/privilegedPayer.ts` and `back-end/libs/common/src/utils/sdk/privileged-payer.ts`.

### Proof of Concept
1. Use the tool as a council account holder with account ID `0.0.100` (a system account within the HIP-1300 range).
2. Attempt to submit a governance transaction (e.g., a system file update or a council-scale account update with 57-user nested threshold key — see test scenario 11.6.1).
3. The tool enforces the 6,144-byte limit because `COUNCIL_ACCOUNTS` does not include `0.0.100`.
4. The transaction is rejected at the client validation layer despite being valid on the Hedera network.
5. Changing `COUNCIL_ACCOUNTS` to `['0.0.2', '0.0.100']` (or implementing the range check) resolves the issue. [6](#0-5) [7](#0-6)
