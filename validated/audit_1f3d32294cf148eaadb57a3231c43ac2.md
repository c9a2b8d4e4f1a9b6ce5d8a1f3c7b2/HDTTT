Audit Report

## Title
Missing `@Min(1)` on `threshold` in `CreateTransactionApproverDto` Allows Negative Threshold to Bypass Multi-Signature Approval Requirement

## Summary

`CreateTransactionApproverDto` omits the `@Min(1)` decorator on its `threshold` field, while `UpdateTransactionApproverDto` correctly enforces it. The service-level guard in `approvers.service.ts` only rejects `threshold === 0` and `threshold > approvers.length`, leaving negative values unchecked. A negative threshold (e.g., `-1`) is truthy in JavaScript and therefore passes the persistence assignment, gets written to the database, and causes the frontend `isApproved` approval-count comparison to be trivially satisfied with zero actual approvals.

## Finding Description

**DTO-level gap — `CreateTransactionApproverDto`:**

The `threshold` field carries only `@IsNumber()` and `@IsOptional()`. No `@Min(1)` is present. [1](#0-0) 

**Contrast with `UpdateTransactionApproverDto`:**

The update path correctly applies `@Min(1)`, demonstrating the constraint is known and intentional — it was simply omitted from the create path. [2](#0-1) 

**Service-level guard is incomplete:**

The condition `dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0` does not catch negative values. For `threshold = -1` with two child approvers: `-1 > 2` → `false`, `-1 === 0` → `false`. No exception is thrown. [3](#0-2) 

**Negative threshold is persisted:**

The persistence assignment `dtoApprover.threshold && dtoApprover.approvers ? dtoApprover.threshold : null` evaluates `-1` as truthy in JavaScript, so the negative value is written to the database rather than coerced to `null`. [4](#0-3) 

**`validateApprover` also misses negative values:**

The private validator only checks `approver.threshold === 0`, not `approver.threshold < 0`. [5](#0-4) 

## Impact Explanation

The `isApproved` utility used to evaluate whether a threshold-based approver tree is satisfied reads:

```typescript
if (approvals.length >= (approver.threshold || approvals.length)) {
  return true;
}
``` [6](#0-5) 

Because `-1` is truthy in JavaScript, `approver.threshold || approvals.length` evaluates to `-1`. The condition `approvals.length >= -1` is always `true` for any non-negative count, including zero. A transaction whose approver tree has a negative threshold is therefore considered approved with **zero actual approvals**, completely defeating the multi-signature workflow.

The `threshold` column on the `TransactionApprover` entity has no database-level constraint either, so the invalid value persists without any further rejection. [7](#0-6) 

## Likelihood Explanation

The API endpoint is authenticated (requires a valid JWT), so anonymous exploitation is not possible. However, any authenticated organization user who is the **creator** of a transaction can submit `POST /transactions/:id/approvers` with a crafted negative `threshold`. The front-end UI constrains the select widget to values `≥ 1` (options are generated from `Array.from(Array(length).keys())` yielding `num + 1`), but the back-end performs no equivalent enforcement at the DTO layer, making a direct API call the trivial attack path. [8](#0-7) 

## Recommendation

1. Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`, mirroring `UpdateTransactionApproverDto`.
2. Update the service-level guard from `dtoApprover.threshold === 0` to `dtoApprover.threshold <= 0` (or `< 1`) in both `createTransactionApprovers` and `updateTransactionApprover`.
3. Add a `CHECK (threshold >= 1)` database constraint on the `threshold` column as a defense-in-depth measure.

## Proof of Concept

```http
POST /transactions/42/approvers
Authorization: Bearer <valid-creator-jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 10 },
        { "userId": 11 }
      ]
    }
  ]
}
```

- `-1 > 2` → `false`; `-1 === 0` → `false` → guard passes, no exception thrown.
- `-1` is truthy → `threshold: -1` is written to the database.
- `isApproved` evaluates `approvals.length >= -1` → `true` with zero approvals.
- The transaction proceeds as fully approved without any approver having acted.
