## Audit Report

## Title
Approver Modification Operations Lack Transaction Status Checks, Allowing Creator to Bypass Multi-Party Approval Requirements

## Summary
The three approver-management endpoints (`createTransactionApprovers`, `updateTransactionApprover`, `removeTransactionApprover`) enforce only a creator-identity check via `getCreatorsTransaction` but perform no transaction-status check. A malicious transaction creator can add, remove, or lower the threshold of approvers on a transaction already in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` state, effectively bypassing the multi-party approval requirement the approver system is designed to enforce.

## Finding Description

`approveTransaction` correctly guards against acting on a transaction in the wrong state:

```typescript
// approvers.service.ts lines 584–588
if (
  transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
  transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
)
  throw new BadRequestException(ErrorCodes.TNRA);
``` [1](#0-0) 

However, the three approver-management operations contain no equivalent status guard:

**`createTransactionApprovers`** — calls only `getCreatorsTransaction` (creator identity check), then proceeds to insert new approvers with no status check: [2](#0-1) 

**`updateTransactionApprover`** — calls `getCreatorsTransaction` inside the transaction block, then allows threshold or userId changes with no status check: [3](#0-2) 

**`removeTransactionApprover`** (controller) — calls `getCreatorsTransaction` then immediately removes the approver with no status check: [4](#0-3) 

The service-level `removeTransactionApprover` itself has no authorization or status check at all: [5](#0-4) 

`getCreatorsTransaction` only verifies creator identity — it performs no status check: [6](#0-5) 

After each approver modification, `emitTransactionStatusUpdate` is called, which triggers a status recalculation. If the approval threshold is now satisfied by the manipulated approver set, the transaction automatically advances to `WAITING_FOR_EXECUTION`. [7](#0-6) 

## Impact Explanation

A transaction creator can:
1. Create a transaction requiring N approvals (e.g., threshold of 3 out of 5 approvers).
2. Wait for N-1 approvers to sign (e.g., 2 of 3 required approvals collected; transaction remains in `WAITING_FOR_SIGNATURES`).
3. Call `PATCH /transactions/:transactionId/approvers/:id` to lower the threshold from 3 to 2, **or** call `DELETE /transactions/:transactionId/approvers/:id` to remove the remaining unapproved approver.
4. `emitTransactionStatusUpdate` fires, the status engine recalculates, and the transaction advances to `WAITING_FOR_EXECUTION` with fewer legitimate approvals than originally required.

This completely undermines the multi-party approval invariant: the creator can unilaterally reduce the approval bar mid-flight, making the approver system ineffective as a trust control.

## Likelihood Explanation

The attacker is a **legitimate, authenticated transaction creator** — no privilege escalation or credential theft is required. The attack path uses only documented API endpoints (`POST`, `PATCH`, and `DELETE` on `/transactions/:transactionId/approvers`) with a valid JWT. Any organization user who creates transactions can exploit this. The only precondition is that the attacker is the creator of the transaction, which is by definition true for any transaction they submitted.

## Recommendation

Add a transaction-status guard inside `getCreatorsTransaction`, or at the start of each of the three mutating service methods, that rejects the operation when the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` state. For example:

```typescript
// In getCreatorsTransaction or at the top of each mutating method:
const modifiableStatuses = [TransactionStatus.NEW];
if (!modifiableStatuses.includes(transaction.status)) {
  throw new BadRequestException('Approvers cannot be modified once the transaction is in progress');
}
```

This mirrors the guard already present in `approveTransaction` and in `cancelTransactionWithOutcome`. [8](#0-7) 

## Proof of Concept

```
1. Creator authenticates and creates a transaction with a 3-of-3 approver requirement.
   POST /transactions  →  { id: 42, status: "WAITING_FOR_SIGNATURES" }

2. Approvers A and B sign (2 of 3 collected).
   POST /transactions/42/approvers/approve  (×2)

3. Creator calls PATCH to lower the threshold of the approver tree node from 3 to 2:
   PATCH /transactions/42/approvers/7
   Body: { "threshold": 2 }
   → 200 OK  (no status check blocks this)

4. emitTransactionStatusUpdate fires; the status engine sees 2 approvals satisfy
   the new threshold of 2 and advances the transaction:
   Transaction 42 status → "WAITING_FOR_EXECUTION"

5. The transaction proceeds to execution with only 2 of the originally required
   3 approvals — the multi-party approval requirement has been bypassed.
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
