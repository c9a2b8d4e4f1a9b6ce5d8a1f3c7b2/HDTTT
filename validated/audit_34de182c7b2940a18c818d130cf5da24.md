Audit Report

## Title
Missing Authorization Check Allows Any Authenticated User to Post Comments on Any Transaction

## Summary
`CommentsController.createComment` applies only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) and performs no authorization check to verify the requesting user has any relationship to the target transaction. Any verified, authenticated user can inject comments into any transaction's thread.

## Finding Description
**Root cause:** The `@Post()` handler in `CommentsController` is protected only by class-level authentication guards. No guard, decorator, or service-layer check verifies that the caller is a creator, approver, signer, or observer of the target transaction.

The controller applies only authentication at the class level: [1](#0-0) 

The `// TODO` on line 22 explicitly acknowledges the missing authorization check: [2](#0-1) 

The service layer performs no relationship check — it directly creates and persists the comment with the caller-supplied `transactionId`: [3](#0-2) 

By contrast, `ApproversController.removeTransactionApprover` calls `getCreatorsTransaction` before any mutation: [4](#0-3) 

And `getCreatorsTransaction` in `ApproversService` enforces ownership by checking `creatorKey.userId`: [5](#0-4) 

No equivalent check exists anywhere in the comments flow.

## Impact Explanation
An attacker with a valid account can inject arbitrary comments into any transaction's discussion thread, including high-value multi-signature transactions awaiting approval. Approvers reading the comment thread may be misled by fabricated status updates, false urgency claims, or spoofed instructions, causing them to approve or reject transactions based on attacker-controlled information. This is an unauthorized write path that can directly influence approval decisions and constitutes a cross-tenant data integrity failure.

## Likelihood Explanation
The only precondition is a valid authenticated account — the lowest possible privilege level in the system. Transaction IDs are sequential integers (`@PrimaryGeneratedColumn()` on `TransactionComment.id`), making enumeration trivial. The `// TODO` comment confirms the gap is known and unaddressed. Any registered user can exploit this immediately with a single HTTP `POST` request. [6](#0-5) 

## Recommendation
Before persisting a comment, verify the requesting user is a participant in the target transaction. The existing `verifyAccess` method in `TransactionsService` already encodes this logic (checks creator, observers, signers, approvers): [7](#0-6) 

Call `verifyAccess` (or an equivalent check) inside `CommentsService.createComment` before calling `this.repo.save(comment)`, and throw `UnauthorizedException` if the user has no relationship to the transaction.

## Proof of Concept
1. Register two accounts: `victim` (creator of transaction ID 5) and `attacker` (no relationship to any transaction).
2. Authenticate as `attacker` and obtain a valid JWT.
3. Send:
   ```
   POST /transactions/5/comments
   Authorization: Bearer <attacker_jwt>
   Content-Type: application/json

   { "message": "Approved by admin — please sign immediately." }
   ```
4. The comment is persisted and returned with HTTP 201. It is now visible to all participants of transaction 5 via `GET /transactions/5/comments`, with no indication it came from an unauthorized party.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
```
