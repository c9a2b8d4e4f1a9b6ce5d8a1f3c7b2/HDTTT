### Title
Unbounded Array Input to `POST /transactions/signers` Enables Authenticated DoS via Resource Exhaustion

### Summary
The `uploadSignatureMaps` endpoint in `signers.service.ts` and the `importSignatures` endpoint in `transactions.service.ts` both accept an attacker-controlled array of arbitrary size with no `@ArrayMaxSize()` constraint on the DTO. Each element triggers CPU-intensive operations (transaction byte deserialization, signature validation, cryptographic processing) and multiple database queries. A single authenticated user can submit a massive array to exhaust server CPU, memory, and database connections, causing denial of service for all other users.

### Finding Description

**Root cause:** The `UploadSignatureMapDto[]` array accepted by both `POST /transactions/signers` (via `SignersController`) and `POST /transactions/signers` (via `TransactionsController::importSignatures`) has no upper-bound validation.

In `signers.controller.ts`, the endpoint accepts the body directly: [1](#0-0) 

The `uploadSignatureMaps` method in `signers.service.ts` then calls `validateAndProcessSignatures`, which fans out into `Promise.all` over every element of the unbounded array: [2](#0-1) 

Each element triggers `processTransactionSignatures`, which deserializes transaction bytes and performs cryptographic signature processing: [3](#0-2) 

The same pattern exists in `transactions.service.ts::importSignatures`, where the developer explicitly acknowledged the missing limit with a comment: [4](#0-3) 

The DTO has no `@ArrayMaxSize()` constraint: [5](#0-4) 

Similarly, `CreateTransactionApproversArrayDto` and `CreateTransactionGroupDto` both accept unbounded arrays with no max-size guard: [6](#0-5) [7](#0-6) 

### Impact Explanation
An authenticated attacker submits a single `POST /transactions/signers` request with an array of thousands of entries (referencing valid or invalid transaction IDs). The server fans out `Promise.all` over all entries simultaneously, each performing:
- `SDKTransaction.fromBytes()` — CPU-intensive deserialization
- Cryptographic signature validation
- Database lookups (`entityManager.find`)

This exhausts the Node.js event loop, available memory, and the PostgreSQL connection pool, causing the API service to become unresponsive for all legitimate users. The attack is repeatable and requires only a valid JWT token (any registered user).

### Likelihood Explanation
Any registered organization user can reach this endpoint — authentication is required but no elevated privilege is needed. The attacker only needs one valid account. The array can contain IDs for non-existent transactions (the service still processes each entry before returning an error per item). No rate limiting or payload size cap is visible in the controller or DTO layer.

### Recommendation
1. **Short term:** Add `@ArrayMaxSize(N)` to `UploadSignatureMapDto[]` at the controller/DTO level (e.g., `N = 100` to match the pagination cap already enforced elsewhere in the system). Apply the same to `CreateTransactionApproversArrayDto.approversArray` and `CreateTransactionGroupDto.groupItems`.
2. **Long term:** Replace `Promise.all` fan-out with a bounded concurrency primitive (e.g., `p-limit`) and enforce a global request body size limit at the NestJS application level (`app.use(express.json({ limit: '1mb' }))`).

### Proof of Concept

```
POST /transactions/signers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": {} },
  { "id": 2, "signatureMap": {} },
  ... (repeat 10,000 times with arbitrary IDs)
]
```

The server will fan out `Promise.all` over 10,000 concurrent async operations, each calling `SDKTransaction.fromBytes()` and hitting the database. The Node.js process will spike to 100% CPU and the PostgreSQL connection pool will be saturated, causing all concurrent legitimate requests to time out.

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);

            map,
            userKeyMap,

            id,
            transaction,
            sdkTransaction,

          };

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
