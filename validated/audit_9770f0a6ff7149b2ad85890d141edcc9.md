Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side Resource Exhaustion (DoS)

## Summary
`TransactionsService.getTransactionsToSign` fetches every non-terminal transaction from the database with no row limit, then performs an expensive per-transaction sequential async signature-key computation for each result. Any authenticated, verified user can trigger `GET /transactions/sign` to exhaust server memory and CPU. Pagination is applied only after the full in-memory scan, providing no protection.

## Finding Description

**Root cause — no database-level limit on the query:**

`getTransactionsToSign` issues a `repo.find` call with no `take` parameter, loading every non-terminal transaction into Node.js heap memory in a single query: [1](#0-0) 

Note also that `TransactionStatus.REJECTED` is present in `terminalStatuses` (line 104) but is **not** excluded from this query's `Not(In([...]))` filter (lines 268–276), meaning rejected transactions are also loaded — making the scan larger than the report describes.

**Unbounded sequential loop with per-iteration async DB work:**

The full result set is iterated sequentially, calling `userKeysToSign` (which internally invokes `TransactionSignatureService.computeSignatureKey` and additional entity-manager queries) for every transaction: [2](#0-1) 

**Pagination applied only after full scan:**

The `offset`/`limit` slice is applied to the already-computed in-memory array, not to the database query: [3](#0-2) 

**Contrast with the correctly paginated sibling method:**

`getTransactionsToApprove` correctly passes `skip: offset, take: limit` to the database query, so only the requested page is loaded: [4](#0-3) 

**Exposed HTTP endpoint:**

The route is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin role is required: [5](#0-4) [6](#0-5) 

The comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end has moved on, but the route remains live and unprotected.

## Impact Explanation

As the number of active (non-terminal) transactions grows, each call to `GET /transactions/sign` forces the API process to:

1. Load the entire active (and rejected) transaction table into Node.js heap memory.
2. Execute N sequential async DB round-trips — one `computeSignatureKey` call per transaction.

A single authenticated user repeatedly calling this endpoint can saturate the API server's memory and CPU, causing request timeouts for all other users and potentially crashing the NestJS process. The cost per request scales linearly with the total transaction count, a quantity that only grows over the platform's lifetime, so severity worsens automatically without any attacker escalation.

## Likelihood Explanation

- **Attacker precondition:** A valid JWT for any non-admin verified user. Registration is a normal product flow.
- **Trigger:** A single HTTP `GET /transactions/sign` (or a low-rate loop of such requests).
- **No special knowledge required:** The endpoint is documented in the Swagger spec and follows the standard REST pattern of the API.
- **Amplification:** The cost per request grows automatically as the organization accumulates transactions; no attacker action is needed to increase severity over time.

This is **not** a network-level DDoS (which the SECURITY.md excludes). It is an application-level algorithmic complexity flaw — a single request from a single authenticated user causes the server to perform unbounded work due to a missing `take` clause and a sequential async loop. The SECURITY.md exclusion for "Impacts that only require DDoS" does not apply here.

## Recommendation

1. **Push pagination into the database query** in `getTransactionsToSign`, analogous to how `getTransactionsToApprove` already does it — pass `skip: offset, take: limit` to `repo.find`.
2. **Add `TransactionStatus.REJECTED` to the excluded statuses** in the `whereForUser` filter to match the intent of `terminalStatuses`.
3. **Consider removing or rate-limiting the endpoint** given the `/* NO LONGER USED BY FRONT-END */` comment; if it is truly unused, disabling it eliminates the attack surface entirely.
4. **Apply a hard server-side cap** (e.g., `take: Math.min(limit, 100)`) to prevent any single request from loading an unbounded number of rows, regardless of client-supplied pagination parameters.

## Proof of Concept

```
# 1. Obtain a JWT for any verified (non-admin) user via normal registration/login.
# 2. Ensure the database has a large number of non-terminal transactions
#    (grows naturally over platform lifetime).
# 3. Issue the request:

GET /transactions/sign?page=1&limit=10
Authorization: Bearer <valid_jwt>

# The server will:
#   a. SELECT * FROM transaction WHERE status NOT IN (...) -- no LIMIT clause
#   b. For each of N rows, await userKeysToSign() → computeSignatureKey() → DB round-trip
#   c. Only then slice [0:10] and return
#
# Repeating this request in a loop (or even once with N large enough)
# exhausts Node.js heap and CPU, degrading or crashing the API for all users.
```
