# Audit Report

## Title
Missing `verifyAccess` Check in `SignersService.uploadSignatureMaps` Allows Any Authenticated User to Inject Signatures Into Arbitrary Transactions

## Summary
`SignersService.uploadSignatureMaps` (called by `POST /transactions/:transactionId/signers`) processes signature submissions for any active transaction without verifying that the authenticated user has any organizational relationship (creator, signer, observer, or approver) to the target transaction. The sibling endpoint `TransactionsService.importSignatures` (`POST /transactions/signatures/import`) correctly calls `verifyAccess` before processing; `uploadSignatureMaps` does not.

## Finding Description

**Root cause — missing `verifyAccess` call in `uploadSignatureMaps`:**

`SignersService.uploadSignatureMaps` delegates to `validateAndProcessSignatures`, which only checks:
1. Transaction exists in the database.
2. Transaction status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.
3. Every public key in the submitted signature map belongs to the authenticated user's registered keys (`userKeyMap`). [1](#0-0) 

There is no check that the user is a creator, signer, observer, or approver of the transaction. Critically, `loadTransactionData` fetches transactions **without** loading the relations (`creatorKey`, `approvers`, `signers`, `observers`) that `verifyAccess` requires: [2](#0-1) 

**Contrast with `importSignatures`:**

`TransactionsService.importSignatures` loads transactions with all required relations and explicitly calls `this.verifyAccess(transaction, user)` before any processing: [3](#0-2) 

`verifyAccess` enforces the organizational access model — it returns `false` (and the caller throws) if the user is not a creator, observer, signer, approver, or key-holder for the transaction: [4](#0-3) 

The test suite for `importSignatures` explicitly covers the access-denied case: [5](#0-4) 

No equivalent test exists for `uploadSignatureMaps`. The documented test scenario `11.5.3` ("Import blocked when user lacks transaction visibility") applies only to `importSignatures`, not to `uploadSignatureMaps`: [6](#0-5) 

## Impact Explanation

An authenticated organization member who has a registered key that is part of a transaction's required key structure can:

1. Submit a valid cryptographic signature for any active transaction (status `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`) by guessing or enumerating sequential integer transaction IDs.
2. Have their signature persisted into the transaction's `transactionBytes` in the database via `bulkUpdateTransactions`.
3. Be recorded as a `TransactionSigner` in the database via `bulkInsertSigners`.
4. Potentially advance the transaction's status (e.g., from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION`) if their key was the last required threshold key, triggering `processTransactionStatus` and status-change notifications. [7](#0-6) 

This violates the organizational access control model and can cause transactions to advance to execution without the creator's awareness of who signed.

## Likelihood Explanation

- The attacker must be an authenticated, verified organization member (guarded by `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`).
- The attacker must have a registered key that is part of the target transaction's required key structure.
- Transaction IDs are sequential integers, making enumeration trivial once an attacker is authenticated.
- The constraint that the submitted key must be in the user's own `userKeyMap` limits the attack to users whose keys are genuinely part of the transaction's key structure — but this is precisely the class of users who could cause real harm (advancing status, polluting signer records).

## Recommendation

Add a `verifyAccess` call inside `validateAndProcessSignatures`, mirroring the pattern in `importSignatures`. This requires loading transactions with the necessary relations in `loadTransactionData`:

```typescript
// In loadTransactionData, load with relations:
const transactions = await this.dataSource.manager.find(Transaction, {
  where: { id: In(transactionIds) },
  relations: ['creatorKey', 'approvers', 'signers', 'observers'],
});

// In validateAndProcessSignatures, before processing each transaction:
if (!(await this.transactionsService.verifyAccess(transaction, user))) {
  return { id, error: ErrorCodes.TNF };
}
``` [8](#0-7) 

## Proof of Concept

1. Authenticate as User C (an organization member with a registered key `K` that is part of Transaction T's required key structure, but who is not a creator/signer/observer/approver of T).
2. Obtain Transaction T's integer ID (e.g., by enumeration or observation).
3. Sign the transaction bytes locally with the private key corresponding to `K`.
4. POST to `POST /transactions/T/signers` with the resulting signature map.
5. Observe HTTP 201 response — the signature is accepted, persisted to `transactionBytes`, and User C is recorded as a `TransactionSigner` for Transaction T, potentially advancing its status. [9](#0-8) [10](#0-9)

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
