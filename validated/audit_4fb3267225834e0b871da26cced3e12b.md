### Title
`computeSignatureKey` Skips Node Admin Key for Node ID 0 Due to Falsy Check

### Summary

In `back-end/libs/common/src/transaction-signature/transaction-signature.service.ts`, the `computeSignatureKey` method guards the `addNodeKeys` call with `if (requirements.nodeId)`. Because JavaScript treats `0` as falsy, any `NodeUpdateTransaction` or `NodeDeleteTransaction` targeting consensus node ID `0` silently skips the entire node-key resolution path. The required admin key is never added to the signature list, so the system incorrectly computes the required signers for that node.

### Finding Description

The root cause is at line 55 of `transaction-signature.service.ts`:

```typescript
if (requirements.nodeId) {
  await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
}
```

`requirements.nodeId` is typed `number | null`. When `getNodeId()` returns `0` (a valid Hedera consensus node ID), the expression `if (0)` evaluates to `false`, so `addNodeKeys` is never invoked. The admin key for node 0 is therefore never pushed onto `signatureKey`. [1](#0-0) 

The correct guard is `if (requirements.nodeId !== null)`, which is exactly how the front-end model handles the same field:

```typescript
if (nodeId == null || Number.isNaN(nodeId)) {
  return null;
}
``` [2](#0-1) 

The `getNodeId()` implementations in both `NodeUpdateTransactionModel` and `NodeDeleteTransactionModel` return a plain `number` (via `.toNumber()`) or `null`. Returning `0` for node 0 is entirely valid. [3](#0-2) 

### Impact Explanation

For any `NodeUpdateTransaction` or `NodeDeleteTransaction` whose target node has ID `0`:

1. `computeSignatureKey` returns a `KeyList` that omits the node admin key entirely.
2. `keysRequiredToSign` (which calls `computeSignatureKey`) therefore never identifies the admin key as required.
3. The system marks the transaction as fully signed and ready to execute without the admin key signature.
4. When the chain service submits the transaction to the Hedera network, it is rejected because the mandatory admin key signature is absent.
5. The transaction is recorded as failed/expired despite appearing valid inside the tool, and any associated fees are wasted.

Additionally, signers and approvers who hold the node-0 admin key are never notified, breaking the multi-signature workflow for that node. [4](#0-3) 

### Likelihood Explanation

Hedera consensus node IDs are zero-indexed integers. Node `0` is the first node in any Hedera network (mainnet, testnet, previewnet). Any authenticated user of the tool can draft a `NodeUpdateTransaction` or `NodeDeleteTransaction` and set `nodeId = 0`. No privileged access is required beyond having a valid account on the network. The trigger is deterministic and reproducible. [5](#0-4) 

### Recommendation

Replace the falsy guard with an explicit null check:

```typescript
// Before (buggy — skips node 0)
if (requirements.nodeId) {
  await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
}

// After (correct)
if (requirements.nodeId !== null) {
  await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
}
```

This mirrors the pattern already used correctly in the front-end model (`nodeId == null` check) and in `keysRequiredToSign` (`if (!transaction) return []`). [1](#0-0) 

### Proof of Concept

1. Authenticate as any normal user in the Hedera Transaction Tool.
2. Create a `NodeUpdateTransaction` with `nodeId = 0` and any field change (e.g., update `description`).
3. Submit the transaction to the API (`POST /transactions`).
4. Call `GET /transactions/:id/signers` — the response will not include the node-0 admin key holder as a required signer.
5. Approve and execute the transaction. The chain service submits it to the Hedera network; the network rejects it with `INVALID_SIGNATURE` because the admin key never signed.
6. Repeat with `NodeDeleteTransaction` targeting node `0` — identical outcome.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
