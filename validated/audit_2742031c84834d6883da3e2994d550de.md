Audit Report

## Title
Unbounded Iteration in `getTransactionsToSign` Enables DoS via Mass Transaction Creation

## Summary
`TransactionsService::getTransactionsToSign` fetches all non-terminal transactions from the database with no row limit, then performs a per-transaction `computeSignatureKey` call (involving SDK deserialization and potential mirror-node I/O) for every record. Because any authenticated organization user with at least one registered key can create transactions without a per-user cap, a malicious user can accumulate `WAITING_FOR_SIGNATURES` transactions, causing every subsequent call to `GET /transactions/sign` to exhaust server memory and time out.

## Finding Description

**Root cause — unbounded `repo.find()` with per-row async work:**

In `getTransactionsToSign`, the database query omits the `take`/`skip` clause that every other list endpoint uses:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / no `skip`
});
``` [1](#0-0) 

The function then iterates the entire result set and awaits an expensive async call per row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign` → `keysRequiredToSign` → `transactionSignatureService.computeSignatureKey`, which calls `SDKTransaction.fromBytes` (SDK deserialization) and `accountCacheService.getAccountInfoForTransaction` (which may issue mirror-node HTTP calls for each account involved). [3](#0-2) [4](#0-3) [5](#0-4) 

Pagination is applied only **after** the full scan:

```typescript
items: result.slice(offset, offset + limit),
``` [6](#0-5) 

So requesting `page=1&size=10` still forces the server to process every transaction in the table.

**Contrast with correctly-paginated sibling endpoints:**

`getHistoryTransactions` passes `skip: offset, take: limit` to the ORM query, pushing pagination to the database layer: [7](#0-6) 

`getTransactionsToApprove` does the same: [8](#0-7) 

**Attacker-controlled entry point — unrestricted transaction creation:**

`POST /transactions` is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`, and `HasKeyGuard`. Any verified organization user who has registered at least one key can call it with no per-user transaction count limit. [9](#0-8) 

The `GET /sign` route is still registered and active despite the "NO LONGER USED BY FRONT-END" comment: [10](#0-9) 

## Impact Explanation

When the attacker has seeded enough transactions, any call to `GET /transactions/sign` by any user causes:

1. A full-table scan loading all non-terminal transactions (and their `groupItem` relations) into Node.js heap.
2. A sequential chain of awaited calls — `SDKTransaction.fromBytes` + mirror-node account lookups — one per transaction, blocking the event loop and consuming connection-pool slots.
3. Server-side timeout or OOM crash, making the endpoint permanently unavailable until the attacker's transactions are purged.

Collateral damage extends to the shared NestJS process: a hung or OOM'd worker degrades all other endpoints served by the same instance.

## Likelihood Explanation

- **Attacker preconditions:** A valid organization account with one registered key — the minimum required to join an organization.
- **No privileged access needed:** Standard user role, no admin rights.
- **Effort:** Scripted `POST /transactions` loop over hours; the global rate limiter does not prevent slow accumulation.
- **Detectability:** Low — transaction creation is a normal workflow action; bulk creation blends into legitimate usage patterns.
- **Persistence:** Each created transaction starts in `WAITING_FOR_SIGNATURES` status, which is explicitly excluded from the terminal-status filter in `getTransactionsToSign`, so every attacker-created transaction is included in every subsequent full scan.

## Recommendation

Apply DB-level pagination to `getTransactionsToSign` before the per-row async loop, mirroring the pattern used by `getHistoryTransactions` and `getTransactionsToApprove`. Since the filtering for "keys to sign" requires per-row async evaluation (which cannot be pushed into SQL), a practical fix is to:

1. Add a hard cap on the number of rows fetched (e.g., `take: MAX_SAFE_LIMIT`) to bound memory and I/O.
2. Alternatively, restructure the query to pre-filter eligible transactions at the DB layer (e.g., joining on `transaction_signer` for the user's keys) before applying `take`/`skip`, so the async work is only done on the paginated slice.
3. Consider adding a per-user transaction creation rate limit or a maximum active-transaction count per user.

## Proof of Concept

```
# Step 1: Authenticate as a standard org user with one registered key
TOKEN=$(curl -s -X POST .../auth/login -d '{"email":"attacker@org.com","password":"..."}' | jq -r .token)

# Step 2: Flood the database with WAITING_FOR_SIGNATURES transactions (scripted loop)
for i in $(seq 1 50000); do
  curl -s -X POST .../transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"name":"tx'$i'","transactionBytes":"<valid_hex>","creatorKeyId":1,"mirrorNetwork":"testnet","signature":"<valid_sig>"}' &
done
wait

# Step 3: Trigger the full-table scan — a single request now processes all 50,000 rows
curl -s "http://.../transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer $TOKEN"
# → Server hangs / OOM / timeout; all users' requests to this endpoint are blocked
```

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
