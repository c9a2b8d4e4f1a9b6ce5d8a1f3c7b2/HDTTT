Audit Report

## Title
Concurrent Signature Uploads Silently Overwrite Each Other's `transactionBytes`, Causing Integrity Failure

## Summary
`uploadSignatureMaps` in `signers.service.ts` and `importSignatures` in `transactions.service.ts` both follow a read-modify-write pattern on `transactionBytes` with no optimistic-concurrency control. When two signers submit signatures concurrently, the second write silently overwrites the first, discarding one signer's cryptographic contribution. The system records both `TransactionSigner` rows as successful, creating a false belief that both have signed while the stored bytes only contain one signature.

## Finding Description

**Root cause — unconditional byte overwrite in `uploadSignatureMaps`**

`loadTransactionData` fetches a snapshot of `transactionBytes` at time T₀ with a plain `find`: [1](#0-0) 

`processTransactionSignatures` merges the caller's signatures into that in-memory snapshot: [2](#0-1) 

`bulkUpdateTransactions` then writes the result back with a bare `UPDATE … WHERE id = ANY(…)` — no `AND "transactionBytes" = <expected>` optimistic-lock, no row-level lock: [3](#0-2) 

**The identical pattern in `importSignatures`**

`importSignatures` reads `transactionBytes`, merges signatures in memory, then writes back with a bare `UPDATE … WHERE id IN (:...ids)`: [4](#0-3) 

**Contrast with correctly guarded paths**

`_executeTransaction` uses `AND status = :currentStatus` to detect concurrent changes: [5](#0-4) 

`processTransactionStatus` uses `AND status = :oldStatus` on every bulk status update: [6](#0-5) 

`executeTransaction` is additionally protected by a distributed `@MurLock`: [7](#0-6) 

`uploadSignatureMaps` has no equivalent protection.

**Why `processTransactionStatus` cannot recover**

After the overwrite, `processTransactionStatus` deserializes `transactionBytes` from the DB and checks `sdkTransaction._signerPublicKeys` against `computeSignatureKey`. Because the DB bytes only contain one signer's signature, `hasValidSignatureKey` returns false and the transaction is never promoted to `WAITING_FOR_EXECUTION`: [8](#0-7) 

## Impact Explanation

- **Silent integrity failure**: both signers receive a success response and both `TransactionSigner` rows exist, yet the stored bytes are cryptographically incomplete.
- **Transaction stuck in `WAITING_FOR_SIGNATURES`**: the scheduler's periodic `processTransactionStatus` reads the DB bytes and will not promote the transaction.
- **Recovery requires manual re-signing**: the losing signer must re-submit. The skip condition `if (isSameBytes && userKeys.length === 0) continue` at line 293 does allow a byte-only update (no new signer row) when `isSameBytes = false`, so re-signing is technically possible. However, since the system already reported success to the losing signer, they have no indication that re-signing is needed. If the transaction's `validStart` window expires before re-signing occurs, the transaction is permanently lost. [9](#0-8) 

## Likelihood Explanation

- Multi-signer organization transactions are the primary use-case; concurrent signing is expected and actively encouraged by the UI ("Sign All" automation).
- `validateAndProcessSignatures` is async and involves mirror-node lookups, widening the race window significantly. [10](#0-9) 
- No special attacker capability is required — two legitimate users signing at the same time is sufficient.
- The k6 load-test suite explicitly exercises concurrent API calls, confirming this scenario is anticipated.

## Recommendation

Apply optimistic concurrency control to the `transactionBytes` update. The simplest approach is to add a version column (or use the existing `updatedAt` timestamp) and include it in the `WHERE` clause:

```sql
UPDATE transaction
SET "transactionBytes" = :newBytes, "updatedAt" = NOW()
WHERE id = :id AND "transactionBytes" = :expectedBytes
```

If the update affects 0 rows, re-fetch the latest bytes, re-merge the signature, and retry (up to N times). Alternatively, acquire a row-level advisory lock (`SELECT … FOR UPDATE`) at the start of `loadTransactionData` within a database transaction to serialize concurrent writes for the same transaction ID.

The same fix must be applied to `importSignatures` in `transactions.service.ts`. [11](#0-10) [12](#0-11) 

## Proof of Concept

1. Create transaction T requiring signatures from keys A and B.
2. Simultaneously POST to `/transactions/signers` from two sessions — one for key A, one for key B.
3. Both requests execute `loadTransactionData` and receive the same `transactionBytes` snapshot (no signatures).
4. Request A merges A's signature → `bytes_A`; Request B merges B's signature → `bytes_B`.
5. Request A commits: DB holds `bytes_A`.
6. Request B commits: DB holds `bytes_B` — A's signature is gone.
7. Both `TransactionSigner` rows exist; both callers received HTTP 200.
8. `processTransactionStatus` deserializes `bytes_B`, finds only B's key in `_signerPublicKeys`, `hasValidSignatureKey` returns false, transaction remains `WAITING_FOR_SIGNATURES`.
9. Unless A re-signs (with no UI prompt to do so), the transaction expires and is permanently lost.
