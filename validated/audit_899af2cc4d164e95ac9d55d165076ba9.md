### Title
Cross-Network Transaction Replay via Unsigned `mirrorNetwork` Field

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` is user-supplied and is not included in the data the creator cryptographically signs. The creator's signature covers only `dto.transactionBytes`, while `mirrorNetwork` is a separate metadata field that determines which Hedera network the transaction is executed on. Because Hedera transactions contain no embedded network identifier and node account IDs (e.g., `0.0.3`, `0.0.4`) are identical across mainnet, testnet, and previewnet, the same signed transaction bytes pass all server-side validation checks regardless of which `mirrorNetwork` value is supplied.

### Finding Description

**Root cause â€” unsigned network discriminator:**

In `validateAndPrepareTransaction`, the server verifies the creator's signature over `dto.transactionBytes` only:

```typescript
const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
``` [1](#0-0) 

The `mirrorNetwork` value is accepted from the request body and stored verbatim without being covered by the signature:

```typescript
mirrorNetwork: dto.mirrorNetwork,
``` [2](#0-1) 

**Node validation does not prevent cross-network replay:**

The server validates that the transaction's node account IDs are present in the client built from `dto.mirrorNetwork`:

```typescript
const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
  throw new BadRequestException(ErrorCodes.TNVN);
}
``` [3](#0-2) 

However, Hedera mainnet, testnet, and previewnet all use the same node account IDs (`0.0.3`, `0.0.4`, `0.0.5`, â€¦). A transaction frozen against testnet node `0.0.3` passes node validation for mainnet as well, so this check provides no cross-network isolation.

**`transactionId` uniqueness check is bypassable:**

The server prevents duplicate submissions of the same `transactionId`, but only for non-terminal statuses:

```typescript
status: Not(In([TransactionStatus.CANCELED, TransactionStatus.REJECTED, TransactionStatus.ARCHIVED])),
``` [4](#0-3) 

Once the original transaction is canceled, the same `transactionId` (and the same signed bytes) can be resubmitted with a different `mirrorNetwork`.

**Exploit path:**

1. Authenticated creator builds a transaction (e.g., `AccountUpdateTransaction` changing account keys) using testnet node IDs.
2. Creator signs the raw bytes: `signature = privateKey.sign(transactionBytes)`.
3. Creator submits `POST /transactions` with `{transactionBytes, signature, mirrorNetwork: 'testnet', creatorKeyId: X}` â€” accepted, stored as testnet transaction.
4. Creator cancels the testnet transaction (`PATCH /transactions/cancel/{id}`).
5. Creator resubmits the **identical** `{transactionBytes, signature}` with `mirrorNetwork: 'mainnet'`.
6. Server re-verifies the signature (passes â€” bytes unchanged), re-validates nodes (passes â€” same IDs), and stores a new mainnet transaction.
7. The transaction is now queued for mainnet execution with the creator's valid signature already attached.

In an organization context with multiple required signers, the creator can present the transaction as a testnet operation to collect co-signer approvals, cancel it, then replay it on mainnet. Co-signers who upload their own signatures to the new mainnet transaction (believing it is still testnet) complete the required threshold.

### Impact Explanation

A malicious creator can cause a Hedera transaction to execute on a network other than the one
