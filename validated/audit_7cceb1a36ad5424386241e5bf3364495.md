### Title
Hardcoded Static Salt in Legacy Mnemonic PBKDF2 Key Derivation Enables Precomputed Password Attacks

### Summary
`decryptLegacyMnemonic` in `front-end/src/main/services/localUser/dataMigration.ts` uses a hardcoded 8-byte salt `[1, 2, 3, 4, 5, 6, 7, 8]` for PBKDF2 key derivation when decrypting the legacy `recovery.aes` mnemonic file. Because the salt is static and publicly known from the open-source repository, an attacker who obtains the encrypted file can precompute password candidates against this exact salt, bypassing the per-user uniqueness that salts are designed to provide. The derived key is then used with AES-256-ECB, a deterministic cipher mode with no IV, compounding the cryptographic weakness.

### Finding Description
In `front-end/src/main/services/localUser/dataMigration.ts`, the function `decryptLegacyMnemonic` (lines 99–119) derives an AES key from the user's password using PBKDF2:

```typescript
const salt = Buffer.from([1, 2, 3, 4, 5, 6, 7, 8]);  // line 108 — hardcoded
const key = crypto.pbkdf2Sync(password, salt, iterationCount, keyLength, 'sha256');
// ...
const decipher = crypto.createDecipheriv('aes-256-ecb', key, null);  // line 113 — ECB, no IV
```

The salt is identical for every user who migrated from the legacy Hedera Transaction Tool. Since the source code is public, the salt value `[1,2,3,4,5,6,7,8]` is globally known. This allows an attacker to:

1. Obtain the user's `recovery.aes` file (located at `<Documents>/TransactionTools/Files/.System/recovery.aes`).
2. Precompute a dictionary/rainbow table of `PBKDF2(candidate_password, [1,2,3,4,5,6,7,8], 65536, 32, sha256)` values — all users share the same salt, so one table serves all targets.
3. Decrypt the file using AES-256-ECB with each candidate key until plaintext is recovered.

The call chain is: `decryptMigrationMnemonic` (line 174) → `decryptLegacyMnemonic` (line 174), triggered during the data migration flow for legacy users. [1](#0-0) [2](#0-1) 

### Impact Explanation
Recovery of the mnemonic phrase grants complete, irrecoverable access to all Hedera accounts derived from it. The attacker can transfer all funds, replace keys, and permanently lock the legitimate owner out. The static salt eliminates the per-user cost of brute-forcing: a single precomputed table is reusable against every legacy-migrated user. AES-256-ECB further leaks block-level plaintext patterns and is deterministic, making ciphertext comparison trivial.

**Impact: High** — full wallet compromise for any legacy-migrated user whose `recovery.aes` file is obtained.

### Likelihood Explanation
The attacker must obtain the `recovery.aes` file from the user's local filesystem. For a desktop Electron application this is a realistic threat: malware, physical access, cloud backup exposure (e.g., iCloud/OneDrive syncing the Documents folder), or a compromised backup are all common vectors. No privileged application access is required — only filesystem read access to the Documents directory. The salt and algorithm are fully disclosed in the public repository.

**Likelihood: Medium** — filesystem access is a realistic precondition for a desktop app; the static salt then makes password cracking significantly cheaper than with a random per-user salt.

### Recommendation
1. **Replace the hardcoded salt**: The legacy path cannot retroactively change the salt for already-encrypted files (it must match what the old tool used). Document this as a known limitation of the migration path and warn users that migration from legacy files carries elevated risk if the `recovery.aes` file is exposed.
2. **Replace AES-256-ECB**: If the legacy format permits, switch to AES-256-GCM with a random IV (as already done in `decryptMnemonic`). If the format is fixed by the old tool, document the ECB limitation explicitly.
3. **Post-migration re-encryption**: After a successful migration, immediately re-encrypt the recovered mnemonic using the modern path (Argon2id + AES-256-GCM with a random salt/IV) and delete or warn about the legacy file.
4. **Increase PBKDF2 iterations**: 65,536 iterations of SHA-256 is below current NIST SP 800-132 recommendations (≥210,000 for SHA-256). The modern path already uses Argon2id, which is preferable.

### Proof of Concept

```python
import hashlib, hmac, struct
from Crypto.Cipher import AES
from Crypto.Util.Padding import unpad

# Publicly known from source code
STATIC_SALT = bytes([1, 2, 3, 4, 5, 6, 7, 8])
ITERATIONS  = 65536
KEY_LEN     = 32

# Attacker obtains recovery.aes from Documents/TransactionTools/Files/.System/
with open('recovery.aes', 'rb') as f:
    ciphertext = f.read()

# Precomputed table or online dictionary attack — same salt for ALL users
for candidate in ['password123', 'hedera2024', 'mypassword', ...]:
    key = hashlib.pbkdf2_hmac('sha256', candidate.encode(), STATIC_SALT, ITERATIONS, KEY_LEN)
    try:
        cipher = AES.new(key, AES.MODE_ECB)
        plaintext = unpad(cipher.decrypt(ciphertext), 16)
        print(f"[+] Password found: {candidate}")
        print(f"[+] Mnemonic: {plaintext.decode('utf-8')}")
        break
    except Exception:
        continue
```

Because the salt is identical for every legacy-migrated user, the PBKDF2 outputs for a given password candidate are the same across all targets. A single precomputed table of common passwords against `STATIC_SALT` can be applied to any stolen `recovery.aes` file, recovering the 24-word mnemonic and granting full wallet access.
