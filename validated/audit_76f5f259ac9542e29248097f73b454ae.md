### Title
Stale `seq` Assignment via `groupItems.length` Causes Wrong-Item Mutation After Group Item Removal

### Summary
In `BaseGroupHandler.vue`, the `seq` field for a newly added group item is assigned as `transactionGroup.groupItems.length.toString()`. When a group item is removed via `removeGroupItem`, the array shrinks but the `seq` values of surviving items are not updated. Adding a new item then produces a duplicate `seq` value in the in-memory store. Because `editGroupItem` uses `seq` as a direct array index, subsequent edit operations silently target the wrong transaction in the group — a concrete state-integrity failure in a transaction management tool.

### Finding Description

**Root cause — `seq` derived from mutable array length:**

In `BaseGroupHandler.vue`, when `action === 'add'`, the new item's `seq` is set to the current array length: [1](#0-0) 

**Removal shrinks the array but leaves surviving `seq` values stale:**

`removeGroupItem` splices the array without renumbering the remaining items' `seq` fields: [2](#0-1) 

**`editGroupItem` treats `seq` as an array index:** [3](#0-2) 

**Concrete collision scenario:**

| Step | Action | In-memory `groupItems` |
|---|---|---|
| 1 | Add A, B, C | `[A(seq='0'), B(seq='1'), C(seq='2')]`, length=3 |
| 2 | `removeGroupItem(1)` (remove B) | `[A(seq='0'), C(seq='2')]`, length=2 |
| 3 | Add D → `seq = length.toString() = '2'` | `[A(seq='0'), C(seq='2'), D(seq='2')]`, length=3 |

Now C (at array index 1) and D (at array index 2) both carry `seq='2'`. When the user navigates to edit C, `editGroupItem` parses `seq='2'` → `editIndex=2` → mutates D instead of C. C is permanently un-editable; every edit attempt on it silently overwrites D. [4](#0-3) 

The SQLite schema enforces a unique constraint on `(transaction_group_id, seq)`: [5](#0-4) 

The standalone `addGroupItem` service function passes `groupItem.seq` directly to Prisma, so a duplicate `seq` on the same group would also trigger a unique-constraint error on that code path: [6](#0-5) 

### Impact Explanation

The Hedera Transaction Tool is used by Hedera Council members to construct, sign, and submit high-value network transactions. A group item whose `seq` has been shadowed by a newly added item:

1. **Cannot be edited** — every edit silently mutates the wrong transaction.
2. **Carries stale transaction parameters** into the signing and submission pipeline.
3. In organization mode, observers and approvers are looked up by `seq` as an array index: [7](#0-6) 

Wrong observers/approvers are therefore uploaded for the affected transaction, corrupting the multi-signature workflow. Incorrect transactions can be submitted to the Hedera network.

### Likelihood Explanation

The trigger path — add items to a group, remove one, add another, then edit a surviving item — is a routine editing workflow explicitly supported by the UI. No special privileges, external access, or race conditions are required. Any user building a transaction group with more than one item who performs a remove-then-add sequence will encounter this silently.

### Recommendation

Do not derive `seq` from `groupItems.length`. Instead, maintain a monotonically increasing counter (e.g., `nextSeq`) in the store, analogous to the `Vesting.next_claim_id` fix described in the external report. Increment it on every `addGroupItem` call and never decrement it on removal. Additionally, after any removal, renumber the `seq` fields of all surviving items to match their current array indices so that `editGroupItem`'s index-based lookup remains consistent.

### Proof of Concept

1. Open the Create Transaction Group UI.
2. Add three transactions: A, B, C. Observe `seq` values `'0'`, `'1'`, `'2'` in the store.
3. Remove transaction B (index 1). Array becomes `[A(seq='0'), C(seq='2')]`, length = 2.
4. Add a fourth transaction D. `buildActionData` computes `seq = groupItems.length.toString() = '2'`. Array becomes `[A(seq='0'), C(seq='2'), D(seq='2')]`.
5. Navigate to edit transaction C (the UI passes `seq='2'` via route params).
6. `editGroupItem` parses `seq='2'` → `editIndex=2` → replaces `groupItems[2]` (D) with the edited data. C is unchanged. The edit was silently applied to the wrong transaction. [8](#0-7) [3](#0-2)
