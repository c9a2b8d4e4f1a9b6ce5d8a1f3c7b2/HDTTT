# Audit Report

## Title
TOCTOU Race Condition in `approveTransaction` Allows Duplicate Vote Submission and Approval State Corruption

## Summary
`approveTransaction` in `approvers.service.ts` performs its duplicate-approval check (the `ErrorCodes.TAP` guard) outside of any database transaction and without a `WHERE signature IS NULL` guard on the subsequent `UPDATE`. Two concurrent HTTP requests from the same authenticated approver can both pass the guard and both write conflicting approval values, allowing an approver to effectively change their vote after it has been cast and corrupt the multi-signature approval state.

## Finding Description

**Root cause — TOCTOU between check and write**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` follows this sequence:

**Step 1 — Read** approver records outside any DB transaction: [1](#0-0) 

**Step 2 — Check** whether the user has already voted, also outside any DB transaction: [2](#0-1) 

**Step 3 — Write** the approval decision inside a DB transaction, but with **no** `WHERE signature IS NULL` condition: [3](#0-2) 

Because steps 1–2 are not inside the same serializable transaction as step 3, two concurrent requests from the same user can both read `signature = null`, both pass the guard at line 563, and both execute the `UPDATE`. The last write wins unconditionally.

**No unique constraint prevents duplicate writes**

The `transaction_approver` table is created with only a primary key on `id` — there is no unique constraint on `(transactionId, userId)`: [4](#0-3) 

**Endpoint is reachable by any authenticated approver**

The `/approve` endpoint is protected only by JWT and `VerifiedUserGuard` — no elevated privilege is required: [5](#0-4) 

## Impact Explanation

- **Vote manipulation**: An approver can race `approved: true` and `approved: false` simultaneously. The last write wins, allowing them to change their vote after the fact, bypassing the `ErrorCodes.TAP` guard entirely.
- **Inconsistent approval state**: If Request A's `approved: true` causes the chain service to advance the transaction to `WAITING_FOR_EXECUTION`, and Request B's `approved: false` then overwrites the approver record, the system is left in a state where the transaction is queued for execution but the approver record shows a rejection. This violates the core multi-signature integrity invariant.
- **Threshold manipulation**: In a 1-of-1 or N-of-N threshold scenario, a single malicious approver can approve to trigger advancement and then immediately overwrite with a rejection, corrupting the approval audit trail.

## Likelihood Explanation

The attacker is a legitimate approver — no privilege escalation is required. Sending two concurrent HTTP requests is trivial from any HTTP client (e.g., `Promise.all([fetch(...), fetch(...)])`). The race window spans the entire async execution path from the read at line 553 to the write at line 609, which in a Node.js/NestJS service under normal load is reliably exploitable. No special network conditions or timing precision is needed.

## Recommendation

1. **Wrap the entire check-then-write sequence in a single serializable database transaction.** Move the `getVerifiedApproversByTransactionId` call and the `ErrorCodes.TAP` guard inside the `this.dataSource.transaction(...)` block at line 599.

2. **Add a `WHERE signature IS NULL` guard to the UPDATE query**, so that even if two requests race past the check, only the first one that finds `signature = null` will actually write:

```typescript
.update(TransactionApprover)
.set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
.whereInIds(userApprovers.map(a => a.id))
.andWhere('signature IS NULL')   // <-- add this
.execute();
```

3. **Add a unique constraint on `(transactionId, userId)` in `transaction_approver`** (where `userId IS NOT NULL`) as a defense-in-depth measure to prevent duplicate approver rows for the same user on the same transaction.

## Proof of Concept

```typescript

// Both requests are fired concurrently before either completes.

const [resA, resB] = await Promise.all([

    method: 'POST',

    body: JSON.stringify({ userKeyId: 1, signature: sigA, approved: true }),
  }),
  fetch('/transactions/42/approvers/approve', {
    method: 'POST',
    headers: { Authorization: `Bearer ${aliceToken}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ userKeyId: 1, signature: sigB, approved: false }),
  }),
]);

// Both return HTTP 200.
// t=1: Request A writes approved=true, signature=sigA → chain service may advance to WAITING_FOR_EXECUTION
// t=2: Request B writes approved=false, signature=sigB → DB now shows rejection
```
