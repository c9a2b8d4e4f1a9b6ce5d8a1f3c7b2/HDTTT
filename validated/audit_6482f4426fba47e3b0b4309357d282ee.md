Audit Report

## Title
Any Transaction Creator Can Remove Approvers From Arbitrary Transactions Due to Missing Cross-Reference Check

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `:transactionId` but deletes the approver record identified by `:id` without verifying that the approver belongs to `:transactionId`. An authenticated user who is the creator of any one transaction can remove approvers from any other transaction in the system.

## Finding Description

**Controller — decoupled authorization and operation:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent steps: [1](#0-0) 

Step 1 (`getCreatorsTransaction`) verifies the caller owns `:transactionId`. Step 2 (`removeTransactionApprover(id)`) deletes the approver by `:id`. There is no assertion that the approver at `:id` belongs to `:transactionId`.

**Service — no ownership check:**

`removeTransactionApprover` only checks existence, then deletes and emits a notification against `approver.transactionId` — which may be a completely different transaction than the one that was authorized: [2](#0-1) 

The notification firing against `approver.transactionId` (line 541) rather than the authorized `transactionId` confirms the operation targets the wrong transaction.

**Contrast with the correct pattern in `updateTransactionApprover`:**

The `PATCH /:id` path correctly resolves the root node of the approver, asserts `rootNode.transactionId === transactionId`, and only then calls `getCreatorsTransaction`: [3](#0-2) 

The `DELETE` path omits this entire cross-reference check.

## Impact Explanation

- Required approvers are silently removed from victim transactions, allowing those transactions to proceed without the intended approval threshold — bypassing the multi-party authorization model entirely.
- An attacker can reduce a threshold-based approver tree to zero approvers, making a transaction auto-approvable or permanently broken.
- The operation leaves no audit trail pointing to the attacker's transaction; the notification event references the victim's transaction ID.

## Likelihood Explanation

- **Attacker preconditions:** Must be an authenticated, verified organization user and the creator of at least one transaction — a normal user role with no elevated privileges.
- **Attack complexity:** Trivial — a single crafted HTTP `DELETE` request with a mismatched `:transactionId` and `:id`.
- **Discoverability:** Approver IDs are returned by `GET /transactions/:id/approvers` (`getVerifiedApproversByTransactionId`), accessible to any user with transaction visibility. IDs are sequential integers, making enumeration straightforward. [4](#0-3) 

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and assert that its `transactionId` matches the authorized `transactionId`, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-reference check (mirrors updateTransactionApprover lines 386-391)
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller: [5](#0-4) 

## Proof of Concept

1. Attacker (User A) creates Transaction A — they are its creator.
2. Victim (User B) creates Transaction B with approver records. Attacker obtains an approver ID from Transaction B via `GET /transactions/:B/approvers` (accessible to any verified user with visibility of Transaction B).
3. Attacker sends:
   ```
   DELETE /transactions/:A/approvers/:approverB_id
   ```
4. `getCreatorsTransaction(A, userA)` passes — attacker is creator of A.
5. `removeTransactionApprover(approverB_id)` deletes the approver from Transaction B with no further checks.
6. Transaction B now has one fewer required approver, potentially falling below its approval threshold. [2](#0-1)
