### Title
Lack of Access Control on Transaction Comment Endpoints Allows Any Verified User to Read and Write Comments on Arbitrary Transactions

### Summary
The `CommentsController` exposes three endpoints (`POST /transactions/:transactionId/comments`, `GET /transactions/:transactionId/comments`, `GET /transactions/:transactionId/comments/:id`) that are protected only by JWT authentication and email-verification guards. There is no check that the requesting user is actually a participant (creator, signer, approver, or observer) of the target transaction. Any verified user can post comments on, and read comments from, any transaction in the system — including transactions they have no legitimate relationship to. The code itself acknowledges this gap with an explicit `//TODO` comment.

### Finding Description

**Root cause — missing transaction-level authorization in `CommentsController`:**

`back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only three guards at the controller level: `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. These confirm the caller holds a valid, non-blacklisted JWT and has a verified account. They say nothing about whether the caller is related to the target transaction. [1](#0-0) 

The `POST` handler carries an explicit developer acknowledgement that the check is missing:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(...)
``` [2](#0-1) 

The `GET` handlers have no per-user filtering at all — they return all comments for any `transactionId` supplied by the caller: [3](#0-2) 

**Service layer confirms no access check:**

`CommentsService.createComment` creates and persists the comment directly without calling any access-verification helper: [4](#0-3) 

**Contrast with the correct pattern used elsewhere:**

`TransactionsService.importSignatures` shows the intended pattern — it calls `verifyAccess(transaction, user)` before touching any transaction data, and throws `ErrorCodes.TNF` if the user is not a creator, signer, approver, or observer: [5](#0-4) 

The `verifyAccess` logic is also tested to confirm it returns `false` for users with no relationship to the transaction: [6](#0-5) 

**Exploit path:**

1. Attacker registers and is verified on the organization server (normal user, no special role).
2. Attacker enumerates transaction IDs (sequential integers starting from 1).
3. `GET /transactions/{id}/comments` — returns all comments for that transaction, leaking any sensitive discussion between legitimate participants.
4. `POST /transactions/{id}/comments` — injects arbitrary comment content into any transaction's comment thread, polluting audit trails and potentially misleading approvers or signers.

### Impact Explanation

- **Unauthorized state mutation**: Any verified user can inject comments into transactions they have no relationship to. Comment threads are part of the transaction audit trail and are visible to legitimate participants; injected comments can mislead approvers or signers.
- **Cross-tenant information disclosure**: Comments on a transaction may contain sensitive operational details (e.g., key management decisions, approval rationale). Any verified user can read them for any transaction ID, breaking the confidentiality boundary between unrelated users.

### Likelihood Explanation

- Precondition: attacker must hold a valid, verified account on the organization server. This is a low bar — account creation is admin-controlled, but any insider or compromised account qualifies.
- Transaction IDs are sequential integers, making enumeration trivial.
- No rate-limiting specific to comment endpoints was observed.
- The missing check is acknowledged in the source code itself, confirming it is not an intentional design decision.

### Recommendation

Apply the same `verifyAccess` pattern used in `TransactionsService.importSignatures` inside `CommentsService.createComment`, `getTransactionComments`, and `getTransactionCommentById`. Specifically:

1. Load the `Transaction` entity for the given `transactionId`.
2. Call `transactionsService.verifyAccess(transaction, user)` (or an equivalent inline check against `creatorKey.userId`, `signers`, `approvers`, `observers`).
3. Throw `UnauthorizedException` or `BadRequestException(ErrorCodes.TNF)` if the user is not a participant.
4. Apply this check to both read (`GET`) and write (`POST`) endpoints.

Remove the `//TODO` comment once the guard is in place.

### Proof of Concept

```
# 1. Obtain a valid JWT for a verified user who is NOT a participant of transaction #42
POST /auth/login  { "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# 2. Read all comments on transaction #42 (cross-tenant disclosure)
GET /transactions/42/comments
Authorization: Bearer <JWT>
→ 200 OK  [ { "content": "Approved — key rotation for account 0.0.5001", ... } ]

# 3. Inject a comment into transaction #42 (unauthorized state mutation)
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "content": "Do NOT approve — this transaction is fraudulent." }
→ 201 Created  { "id": 99, "transactionId": 42, "content": "Do NOT approve...", ... }

# Legitimate approvers now see the injected comment in their workflow.
```
