**Audit Report**

## Title
Missing Array Size Limit on Bulk Signature Endpoints Enables Authenticated DoS via Resource Exhaustion

## Summary
Both `POST /transactions/signatures/import` and `POST /transactions/:transactionId/signers` accept an unbounded array of `UploadSignatureMapDto` objects. No maximum-size guard exists at the DTO, controller, or service layer. Any authenticated user can submit a single request with an arbitrarily large array, forcing proportional cryptographic operations, protobuf deserializations, and heavy relational database queries, exhausting CPU, memory, and DB connection resources.

## Finding Description

**Root cause — no array-level size constraint on `UploadSignatureMapDto`**

The DTO carries only field-level validators (`@IsDefined`, `@IsNumber`, `@IsSignatureMap`) with no `@ArrayMaxSize` or equivalent guard on the wrapping array: [1](#0-0) 

**Both controllers accept unbounded arrays**

`importSignatures` in `TransactionsController`: [2](#0-1) 

`uploadSignatureMap` in `SignersController`: [3](#0-2) 

**Exploit path in `importSignatures`**

For every element in the unbounded array, the service issues a single `IN (…)` query loading all matching `Transaction` rows **with four eager relations** (`creatorKey`, `approvers`, `signers`, `observers`): [4](#0-3) 

It then loops over every entry calling `SDKTransaction.fromBytes()` (protobuf decode), `validateSignature()` (ECDSA/ED25519 verification), and `addSignature()`: [5](#0-4) 

Finally, it builds a raw SQL `CASE … END` bulk-update statement whose size grows linearly with the input array. The developer comment at line 575 explicitly acknowledges the missing limit: [6](#0-5) 

**Same unbounded pattern in `uploadSignatureMaps`**

`loadTransactionData` issues two `IN (…)` queries with no cap on the ID list: [7](#0-6) 

`validateAndProcessSignatures` then runs `Promise.all` over the entire unbounded array, calling `SDKTransaction.fromBytes()` and `addSignature()` for each entry concurrently: [8](#0-7) 

## Impact Explanation

A single crafted POST request with N entries causes:
- One massive relational DB query (`IN` with N IDs, four JOINs for `importSignatures`) — potential DB OOM / connection exhaustion.
- N protobuf deserializations and cryptographic signature verifications — CPU spike proportional to N.
- A raw SQL `CASE` statement with up to N branches (batched at 500, but no cap on total input) — potential query-planner failure or DB timeout.

The API process can be rendered unresponsive for all other users for the duration of the request, constituting a service availability denial for the entire organization backend.

## Likelihood Explanation

Any user who has completed registration (`VerifiedUserGuard`) can reach both endpoints — no admin or privileged role is required. The global `IpThrottlerGuard` limits the number of requests per IP per second/minute, but does **not** limit the size of individual request bodies, so a single oversized request bypasses the throttle entirely: [9](#0-8) 

The attacker only needs a valid JWT. The attack is a single HTTP request, requires no special tooling, and is repeatable.

## Recommendation

1. Add `@ArrayMaxSize(N)` (e.g., N = 100 or 250) to a wrapper DTO used at both endpoints, or enforce the limit inside `transformAndValidateDto`.
2. Enforce a maximum request body size at the HTTP layer (NestJS `bodyParser` `limit` option) to prevent oversized payloads from reaching application code.
3. Consider applying a per-user throttle guard (a `UserThrottlerGuard` already exists in the codebase) to these two POST endpoints in addition to the global IP throttle.

## Proof of Concept

```bash
# Generate a large array of entries (10,000 items with arbitrary IDs)
python3 -c "
import json, sys
entries = [{'id': i, 'signatureMap': {'0.0.3': {'0.0.1@0.000000000': {'302a300506032b6570032100' + 'aa'*32: '0x' + 'bb'*64}}}} for i in range(1, 10001)]
print(json.dumps(entries))
" > payload.json

curl -X POST https://<host>/transactions/signatures/import \
  -H "Authorization: Bearer <valid_jwt>" \
  -H "Content-Type: application/json" \
  -d @payload.json
```

This single request forces the server to:
1. Execute `SELECT … WHERE id IN (1, 2, …, 10000)` with four JOINs.
2. Deserialize and cryptographically process up to 10,000 entries.
3. Build and execute raw SQL `CASE` statements with up to 500 branches each across 20 batches.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
