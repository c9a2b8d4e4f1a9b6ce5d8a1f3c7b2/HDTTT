### Title
Transaction Creator Can Add Themselves as Approver, Bypassing Multi-Party Approval Workflow

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` verifies the caller is the transaction creator but contains no check preventing the creator from designating their own `userId` as an approver. This allows a creator to unilaterally approve their own transaction, defeating the multi-party authorization control the approval system is designed to enforce.

### Finding Description
The vulnerability class is **missing self-referential input validation in an authorization assignment function** — the direct analog to the Amp.sol `authorizeOperatorByPartition` issue where `msg.sender` could authorize itself.

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `createTransactionApprovers` first confirms the caller is the transaction creator via `getCreatorsTransaction`, then iterates over the submitted `approversArray` and inserts each entry. When a `userId` is provided, the only check performed is that the user exists in the database: [1](#0-0) 

There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. The same gap exists in `updateTransactionApprover`, which also accepts an arbitrary `dto.userId` and updates the approver record without checking whether the new user is the creator: [2](#0-1) 

**Exploit path:**
1. Attacker (authenticated, verified user) creates a transaction — they become the creator.
2. Attacker calls `POST /transactions/:id/approvers` with `approversArray: [{ userId: <own id> }]`.
3. No server-side check rejects this; the attacker is inserted as the sole approver.
4. Attacker calls `POST /transactions/:id/approvers/approve` with a valid signature.
5. `approveTransaction` finds the attacker in the approver list, validates the signature, and marks the approval — the transaction advances without any other party's consent. [3](#0-2) 

The controller exposes both endpoints to any verified user with no additional guard: [4](#0-3) 

### Impact Explanation
The approval system exists to enforce organizational multi-party authorization before a Hedera transaction is submitted to the network. A creator who self-assigns as approver can unilaterally satisfy the approval requirement, causing the transaction to reach `WAITING_FOR_EXECUTION` or `EXECUTED` status without any independent review. This is an **unauthorized state transition / privilege escalation**: the creator gains the approver role they are not supposed to hold, bypassing the trust model the system is built on.

### Likelihood Explanation
Any authenticated, verified user can create transactions and set approvers — no privileged access is required. The API endpoint is reachable over standard HTTP with a valid JWT. The attacker only needs their own account credentials, which they legitimately possess. The steps are deterministic and require no race condition or timing dependency.

### Recommendation
In `createTransactionApprovers`, after confirming the user exists, add:

```typescript
if (dtoApprover.userId === user.id)
  throw new Error('Creator cannot add themselves as an approver');
```

Apply the same guard in `updateTransactionApprover` when `dto.userId` is being set:

```typescript
if (dto.userId === user.id)
  throw new Error('Creator cannot assign themselves as an approver');
``` [5](#0-4) 

### Proof of Concept
```
1. Register and verify two accounts: creatorUser (id=10), otherUser (id=20).
2. As creatorUser, POST /transactions  → returns { id: 5 }
3. As creatorUser, POST /transactions/5/approvers
   Body: { "approversArray": [{ "userId": 10 }] }
   → 201 Created; approver record inserted with userId=10 (the creator)
4. As creatorUser, POST /transactions/5/approvers/approve
   Body: { "userKeyId": <key_id>, "signature": <valid_sig>, "approved": true }
   → 200 true; transaction status advances
5. Observe: transaction moved toward execution with zero independent approvals.
   otherUser was never involved.
```

  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,

  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
