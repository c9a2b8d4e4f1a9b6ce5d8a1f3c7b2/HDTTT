### Title
`removeTransaction` Allows Creator to Soft-Delete Transactions in Terminal States, Corrupting Audit Trail

### Summary
The `removeTransaction` function in `TransactionsService` performs no check on the current transaction status before overwriting it to `CANCELED` and soft-deleting the record. This allows the transaction creator to call `DELETE /transactions/:id` on transactions that are already in terminal states (`EXECUTED`, `FAILED`, `EXPIRED`, `ARCHIVED`), silently corrupting the audit trail and erasing visibility for all other participants (signers, observers, approvers).

### Finding Description
`removeTransaction` in `back-end/apps/api/src/transactions/transactions.service.ts` (lines 628–651) only verifies that the caller is the transaction creator via `getTransactionForCreator`, then unconditionally overwrites the status to `CANCELED` and calls `softRemove`:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);  // ownership only, no status check

    if (softRemove) {
      await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
      await this.repo.softRemove(transaction);  // sets deletedAt, hides from normal queries
    }
    ...
}
```

This is exposed directly via the `DELETE /:id` endpoint in `transactions.controller.ts` (lines 313–316):

```typescript
@Delete('/:id')
deleteTransaction(@GetUser() user, @Param('id', ParseIntPipe) id: number): Promise<boolean> {
    return this.transactionsService.removeTransaction(id, user, true);
}
```

By contrast, `cancelTransactionWithOutcome` (lines 659–704) correctly enforces a status guard:

```typescript
if (!this.cancelableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OTIP);
}
```

`removeTransaction` has no equivalent guard. The `getTransactionForCreator` helper (mirrored in `approvers.service.ts` as `getCreatorsTransaction`, lines 624–644) only checks ownership — it never inspects `transaction.status`.

The state machine defines clear terminal states (`EXECUTED`, `FAILED`, `EXPIRED`, `ARCHIVED`) that are supposed to be immutable from the application layer. `removeTransaction` violates this invariant.

### Impact Explanation
A malicious transaction creator can:
1. Create a transaction, collect signatures, and have it executed on the Hedera network.
2. Immediately call `DELETE /transactions/:id` on the now-`EXECUTED` transaction.
3. The backend overwrites the status to `CANCELED` and soft-deletes the record (`deletedAt` is set).
4. All other participants — signers, observers, approvers — lose visibility into the transaction. It disappears from their dashboards and history.
5. The backend audit trail now shows `CANCELED` (or nothing, due to soft-delete) instead of `EXECUTED`.

This is an integrity failure: the on-chain execution is immutable, but the off-chain coordination record — which is the authoritative source of truth for the organization — is silently corrupted. This enables a creator to deny or obscure that a transaction was executed, undermining the multi-party accountability model that is the core purpose of the tool.

### Likelihood Explanation
The attack requires only that the attacker be the creator of a transaction — a normal, unprivileged role any registered user can hold. No special access, leaked credentials, or privileged keys are needed. The endpoint is a standard authenticated `DELETE` call. Any creator who wishes to hide an executed transaction can do so trivially.

### Recommendation
Add a status guard at the start of `removeTransaction` that rejects the operation if the transaction is already in a terminal state:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    const terminalStatuses = [
        TransactionStatus.EXECUTED,
        TransactionStatus.FAILED,
        TransactionStatus.EXPIRED,
        TransactionStatus.ARCHIVED,
    ];
    if (terminalStatuses.includes(transaction.status)) {
        throw new BadRequestException('Cannot delete a transaction that has already reached a terminal state.');
    }
    ...
}
```

This mirrors the pattern already used in `cancelTransactionWithOutcome`.

### Proof of Concept

1. Authenticate as a normal user (User A) and create a transaction via `POST /transactions`.
2. Collect required signatures; the chain service executes the transaction — status becomes `EXECUTED`.
3. As User A (the creator), send: `DELETE /transactions/:id`.
4. `removeTransaction` is called with `softRemove: true`.
5. No status check occurs. The record is updated to `status: CANCELED` and `softRemove` sets `deletedAt`.
6. Other participants (signers, observers) query `GET /transactions/:id` — the record is gone (soft-deleted) or shows `CANCELED`.
7. The audit trail no longer reflects that the transaction was executed.
