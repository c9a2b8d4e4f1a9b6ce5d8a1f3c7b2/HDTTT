### Title
Old Parent Threshold Not Adjusted When Approver Is Re-Parented, Violating `threshold ≤ approvers.length` Invariant

### Summary

In `approvers.service.ts`, the `updateTransactionApprover` function handles two distinct cases when `dto.listId` is provided: detaching a child to root (`listId = null`) and re-parenting a child to a new parent (`listId = <number>`). The `null` path correctly adjusts the old parent's threshold when the child count drops below it. The re-parenting path does not touch the old parent at all, leaving it with `threshold > approvers.length` — an impossible-to-satisfy state that permanently blocks transaction approval for that subtree.

### Finding Description

**Invariant**: For every threshold-type `TransactionApprover` node, `threshold ≤ approvers.length` must hold, because the approval check requires that many children to approve.

**Path 1 — `dto.listId === null` (lines 398–431)**: The code correctly fetches the old parent with its `approvers` relation and adjusts its threshold downward if the remaining child count falls below it:

```typescript
} else if (newParentApproversLength < parent.threshold) {
  await transactionalEntityManager.update(TransactionApprover, parent.id, {
    threshold: newParentApproversLength,
  });
}
``` [1](#0-0) 

**Path 2 — `dto.listId = <number>` (lines 434–466)**: The code finds the *new* parent and validates it, but never fetches or adjusts the *old* parent. After the update, the old parent has lost one child but its `threshold` is unchanged:

```typescript
await transactionalEntityManager.update(TransactionApprover, approver.id, {
  listId: dto.listId,
  transactionId: null,
});
// old parent's threshold is never touched
``` [2](#0-1) 

The asymmetry is exact: one setter path enforces the invariant, the other does not — mirroring the `_setMinThreshold` / `_setTargetThreshold` asymmetry in the external report.

### Impact Explanation

After the re-parenting call, the old parent node has `threshold = N` but only `N-1` children. Because `threshold > approvers.length`, the approval condition for that subtree can never be satisfied regardless of how many users approve. Any transaction whose approver tree contains this node is permanently blocked from reaching the approved state, causing a denial-of-service on transaction execution for the affected transaction.

### Likelihood Explanation

The attacker only needs to be the **creator of a transaction** — a normal authenticated user role with no elevated privileges. The exploit requires two sequential API calls:

1. Create a threshold approver tree where a parent has `threshold = 2` and exactly 2 children.
2. Call `PATCH /transactions/:transactionId/approvers/:childId` with `{ "listId": <other_parent_id> }` to move one child to a different parent.

After step 2, the old parent has `threshold = 2` and `1` child. This is a reachable, low-privilege, deterministic exploit path.

### Recommendation

In the `dto.listId = <number>` branch, fetch the old parent (using `approver.listId`) with its `approvers` relation and apply the same threshold-adjustment logic used in the `dto.listId === null` branch:

```typescript
// After updating approver.listId to dto.listId, handle the old parent:
if (approver.listId !== null) {
  const oldParent = await transactionalEntityManager.findOne(TransactionApprover, {
    relations: ['approvers'],
    where: { id: approver.listId },
  });
  if (oldParent) {
    const newOldParentCount = oldParent.approvers.length - 1;
    if (newOldParentCount === 0) {
      await transactionalEntityManager.softRemove(TransactionApprover, oldParent);
    } else if (newOldParentCount < oldParent.threshold) {
      await transactionalEntityManager.update(TransactionApprover, oldParent.id, {
        threshold: newOldParentCount,
      });
    }
  }
}
```

### Proof of Concept

1. Authenticate as a normal user and create a transaction.
2. Create a threshold approver tree: root node with `threshold = 2`, children A and B.
3. Create a second tree node X (any valid parent with a threshold).
4. Call `PATCH /transactions/:txId/approvers/:A_id` with body `{ "listId": X.id }`.
5. Query the approver tree: the original root node now has `threshold = 2` but only child B remains (`approvers.length = 1`).
6. Attempt to get the transaction approved: the root node's threshold can never be satisfied, permanently blocking approval. [3](#0-2)
