Audit Report

## Title
Incomplete State Reset in `updateTransactionApprover` Allows Stale Signature to Permanently Block Replacement Approver

## Summary
When a transaction creator replaces an approver's `userId` via `updateTransactionApprover`, the code attempts to clear `signature`, `userKeyId`, and `approved` by setting them to `undefined` in the TypeORM update payload. TypeORM's `EntityManager.update()` silently omits fields set to `undefined` from the generated SQL `UPDATE` statement. As a result, the previous approver's signature data persists in the database under the new user's identity, and the replacement approver is permanently blocked from submitting their own approval.

## Finding Description

**Root cause — TypeORM skips `undefined` in updates:**

In `updateTransactionApprover`, when a `userId` change is detected, the code constructs the following update payload: [1](#0-0) 

```typescript
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: undefined,   // ← TypeORM SKIPS this
  signature: undefined,   // ← TypeORM SKIPS this
  approved: undefined,    // ← TypeORM SKIPS this
};
await transactionalEntityManager.update(TransactionApprover, approver.id, data);
```

TypeORM's `EntityManager.update()` only includes fields with non-`undefined` values in the generated SQL. The effective SQL executed is:

```sql
UPDATE transaction_approver SET "userId" = $1 WHERE id = $2
-- signature, userKeyId, approved are NOT touched
```

The correct way to clear a nullable column is to pass `null`, not `undefined`.

**Entity confirms columns are nullable:**

All three fields are declared `nullable: true` in the entity, so setting them to `null` via SQL is valid: [2](#0-1) 

**Downstream check that blocks the replacement approver:**

In `approveTransaction`, the guard against double-approval reads: [3](#0-2) 

```typescript
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

Because `signature` was never cleared in the database, this check evaluates to `true` for the replacement approver (User B), and the `BadRequestException` is thrown unconditionally every time User B attempts to approve.

**Why the unit test does not catch this:**

The test at lines 808–813 of `approvers.service.spec.ts` only asserts that `dataSource.manager.update` was *called* with the `undefined` values — it does not test actual TypeORM persistence behavior, because the manager is fully mocked: [4](#0-3) 

## Impact Explanation

A transaction requiring approval becomes permanently stuck after an approver replacement where the original approver had already signed:

- User B calls `POST /transactions/{id}/approvers/approve` and receives `ErrorCodes.TAP` every time.
- The transaction cannot advance past the approval gate.
- No API endpoint exists for the creator to directly clear the stale signature.
- If the transaction's `validStart` window expires while stuck, it transitions to `EXPIRED` status and can never be executed.

This constitutes **permanent lock / unrecoverable corruption of transaction state**.

## Likelihood Explanation

The scenario is realistic in normal organization workflows:

1. A transaction is created with approver User A.
2. User A approves (writes `signature`, `userKeyId`, `approved` to DB).
3. The creator decides to replace User A with User B (e.g., wrong person assigned, User A unavailable).
4. The creator calls `PATCH /transactions/{id}/approvers/{id}` with `{ userId: UserB_id }` — a documented, supported API operation.
5. The bug triggers silently; no error is returned, the API responds with success.
6. User B is permanently blocked from approving.

No privileged access is required beyond being the transaction creator, a role reachable by any authenticated organization member.

## Recommendation

Replace `undefined` with `null` for the three fields in the update payload so TypeORM includes them in the SQL `UPDATE` statement:

```typescript
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: null,   // explicitly sets column to NULL in DB
  signature: null,   // explicitly sets column to NULL in DB
  approved: null,    // explicitly sets column to NULL in DB
};
```

Additionally, add an integration-level test (or a test that inspects the actual SQL/query arguments with `null` rather than `undefined`) to prevent regression.

## Proof of Concept

1. Create a transaction requiring one approver; assign User A.
2. User A calls `POST /transactions/1/approvers/approve` with a valid signature → succeeds; DB row now has `signature = <bytes>`, `userKeyId = <id>`, `approved = true`.
3. Creator calls `PATCH /transactions/1/approvers/1` with body `{ "userId": <UserB_id> }` → HTTP 200 returned, no error.
4. Inspect DB: `SELECT "userId","signature","userKeyId","approved" FROM transaction_approver WHERE id = 1;` — `userId` is now User B's ID, but `signature`, `userKeyId`, and `approved` still hold User A's values.
5. User B calls `POST /transactions/1/approvers/approve` → receives `400 Bad Request` with `ErrorCodes.TAP` ("Transaction Already Processed") on every attempt.
6. The transaction is permanently stuck in the approval gate.
