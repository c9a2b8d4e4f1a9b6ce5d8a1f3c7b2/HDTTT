# Audit Report

## Title
Balance Validation Bypassed in Organization Mode for HBAR Transfer Transactions

## Summary
In `TransferHbar.vue`, the `createDisabled` computed property gates the balance-insufficiency check behind `user.selectedOrganization === null`, meaning the check is entirely skipped when an organization is selected. Any authenticated organization-mode user can create and submit an HBAR transfer whose debit amount exceeds the sender's on-chain balance. The Hedera network rejects the transaction at execution time, but the payer account is charged the transaction fee and the organization backend records the transaction as submitted, creating a state divergence.

## Finding Description

**Root cause — `TransferHbar.vue` lines 45–51:**

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    (user.selectedOrganization === null && anyTransfersExceedingBalance.value) // ← skipped in org mode
  );
});
``` [1](#0-0) 

The short-circuit `user.selectedOrganization === null &&` means `anyTransfersExceedingBalance` is never consulted when an organization is active. The submit button is enabled regardless of whether the debit amount exceeds the sender's balance.

**`anyTransfersExceedingBalance` correctly detects the problem but is never consulted in org mode:**

```js
const anyTransfersExceedingBalance = computed(() => {
  let result = false;
  for (const transfer of data.transfers) {
    if (transfer.amount.isNegative()) {
      const accountInfo = accountInfos.value[transfer.accountId.toString()];
      if (accountInfo &&
          transfer.amount.negated().toBigNumber().isGreaterThan(accountInfo.balance.toBigNumber())) {
        result = true; break;
      }
    }
  }
  return result;
});
``` [2](#0-1) 

**`preCreateAssert` also has no balance check**, so even the assertion path executed just before transaction creation does not catch this:

```js
const preCreateAssert = () => {
  if (totalBalanceAdjustments.value > 10) { throw ... }
  if (totalBalanceAdjustments.value === 0) { throw ... }
  if (!totalBalance.value.toBigNumber().isEqualTo(0)) { throw ... }
  // ← no anyTransfersExceedingBalance check
};
``` [3](#0-2) 

**`TransferHbarFormData.vue` shows a display-only error message** (`errorMessage`) and a warning icon, but neither disables the submit button nor prevents the transfer from being added to the list: [4](#0-3) 

**`TransferCard.vue`'s "Add Transfer" button** is not disabled when `amountExceedsBalance` is true — it only shows a warning label — so the transfer entry can be added to the list regardless of mode: [5](#0-4) 

**The organization submission path** (`OrganizationRequestHandler.vue`) signs and submits the transaction to the backend with no balance validation: [6](#0-5) [7](#0-6) 

## Impact Explanation

- **Transaction fee loss**: The payer account is charged the Hedera network fee for every failed transaction. In a multi-approver workflow this can be triggered repeatedly by any org-mode user.
- **System state inconsistency**: The organization backend records the transaction as submitted while the network records it as failed, diverging the two states.
- **Approver time waste / griefing**: A malicious or careless organization member can force all required approvers to sign a transaction that is guaranteed to fail on-chain, wasting their time and consuming their credentials.
- No on-chain fund loss for the intended transfer (the network rejects it), but fee loss and state divergence are concrete and measurable.

## Likelihood Explanation

Any authenticated organization-mode user can trigger this without special privileges. The condition is reachable in normal product usage: select an organization, open the Transfer HBAR form, enter a debit amount larger than the sender's balance, and click submit. The UI shows a warning icon and error text but does not block submission in org mode. [8](#0-7) 

## Recommendation

Remove the `user.selectedOrganization === null &&` short-circuit from `createDisabled` so the balance check applies in both personal and organization modes:

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    anyTransfersExceedingBalance.value  // ← apply in all modes
  );
});
```

Additionally, add the same check to `preCreateAssert` as a defense-in-depth measure:

```js
if (anyTransfersExceedingBalance.value) {
  throw new Error('One or more sender accounts have insufficient balance');
}
```

## Proof of Concept

1. Log in and select any organization (`user.selectedOrganization !== null`).
2. Navigate to **Create Transaction → Transfer HBAR**.
3. In the "From" card, enter an account whose on-chain balance is, e.g., 0 HBAR, and enter a transfer amount of 100 HBAR. Click **Add Transfer**.
4. In the "To" card, enter any valid recipient account and 100 HBAR. Click **Add Transfer**.
5. Observe: the "Sign & Submit" button is **enabled** despite the sender having 0 HBAR. The warning icon and "Insufficient balance for transfer" text appear in the UI but do not block submission.
6. Click **Sign & Submit**. The transaction is signed and submitted to the organization backend via `OrganizationRequestHandler.vue → submit()`.
7. Approvers sign the transaction without any blocking warning.
8. The Hedera network executes the transaction, returns `INSUFFICIENT_ACCOUNT_BALANCE`, and charges the payer account the transaction fee.
9. The organization backend records the transaction as submitted; the on-chain result is a failure. [1](#0-0) [9](#0-8)

  try {
    assertIsLoggedInOrganization(user.selectedOrganization);
    if (!request.value) throw new Error('Request is required to sign');

      request.value?.description || '',

      request.value.submitManually,
      request.value.reminderMillisecondsBefore,
