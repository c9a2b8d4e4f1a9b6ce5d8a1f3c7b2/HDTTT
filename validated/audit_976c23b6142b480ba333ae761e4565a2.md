### Title
Missing Unique Constraint on `TransactionSigner` Allows Duplicate Signer Records via Race Condition

### Summary
The `TransactionSigner` entity lacks a unique database constraint on `(transactionId, userKeyId)`, and the application-level duplicate check in `SignersService.uploadSignatureMaps` is performed outside the database transaction (TOCTOU). Two concurrent `POST /transactions/:id/signers` requests from the same user with the same signature map can both pass the in-memory duplicate check and both insert identical signer rows, corrupting the signer ledger for a transaction.

### Finding Description

**Root cause — missing unique constraint:**

`TransactionSigner` declares only a plain (non-unique) composite index: [1](#0-0) 

Compare with `TransactionObserver`, which correctly enforces uniqueness: [2](#0-1) 

**Root cause — TOCTOU in the duplicate check:**

`uploadSignatureMaps` loads existing signers into an in-memory `Set` in `loadTransactionData`, outside any database transaction: [3](#0-2) 

The "new signer" decision is made in `processTransactionSignatures` by checking that in-memory set: [4](#0-3) 

The actual database insert happens later, inside a separate transaction in `persistSignatureChanges` → `bulkInsertSigners`, with no conflict-handling clause: [5](#0-4) 

**Exploit flow:**

1. Attacker (authenticated user) holds a key that is required to sign transaction T.
2. Attacker sends two identical `POST /transactions/T/signers` requests concurrently.
3. Both requests execute `loadTransactionData` simultaneously; both see an empty existing-signer set for key K.
4. Both pass the `!existingSignerIds.has(userKey.id)` check and add key K to `signersToInsert`.
5. Both execute `bulkInsertSigners`; the database accepts both inserts because there is no unique constraint.
6. Two `TransactionSigner` rows with identical `(transactionId, userKeyId)` now exist.

### Impact Explanation

Duplicate `TransactionSigner` rows corrupt the authoritative signer ledger for a transaction. Any downstream logic that counts or iterates signer records — including `processTransactionStatus` and the notification/dismissal pipeline — will observe inflated signer counts. In a multi-signature threshold workflow this can cause a transaction to be advanced to `WAITING_FOR_EXECUTION` (or treated as fully signed) before the required number of *distinct* keys have actually signed, breaking the core multi-sig integrity guarantee. The audit trail is also permanently corrupted, as the duplicate rows persist even after the transaction is executed. [6](#0-5) 

### Likelihood Explanation

Any authenticated user who is a required signer on a transaction can trigger this with two simultaneous HTTP requests — no elevated privileges, no leaked secrets, no special tooling beyond a basic HTTP client. The window is wide because the check-then-insert gap spans multiple async I/O operations (DB reads, SDK signature processing, then a separate DB write transaction). Under normal network latency the race is reliably winnable.

### Recommendation

1. **Add a unique database constraint** on `(transactionId, userKeyId)` in `TransactionSigner`:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'], { unique: true })
export class TransactionSigner { ... }
```

2. **Use an upsert / `INSERT ... ON CONFLICT DO NOTHING`** in `bulkInsertSigners` so concurrent inserts are idempotent:

```typescript
await manager
  .createQueryBuilder()
  .insert()
  .into(TransactionSigner)
  .values(signersToInsert)
  .orIgnore()   // ON CONFLICT DO NOTHING
  .returning('*')
  .execute();
```

Both fixes are needed: the constraint is the safety net; the `orIgnore` prevents spurious errors under concurrency.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
