### Title
Unbounded Database Fetch in `getTransactionsToSign` Causes Uncontrolled Server-Side Resource Consumption

### Summary
The `getTransactionsToSign` method in `TransactionsService` fetches **all** non-terminal transactions from the database with no `take` (LIMIT) clause, then iterates over every row calling `userKeysToSign` per transaction before applying pagination in memory. Any authenticated user can trigger this path with a single request, causing memory and CPU consumption proportional to the total number of non-terminal transactions in the system — a direct analog to the uncontrolled resource consumption class described in the jose2go report.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` function accepts pagination parameters but applies them only **after** an unbounded database fetch and a per-row async loop:

```typescript
// Line 295 — no `take:` field → fetches every non-terminal transaction
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});

// Line 301 — O(n) async loop over the full result set
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}

// Line 313 — pagination applied in memory, after all work is done
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [1](#0-0) 

Contrast this with `getTransactions` and `getHistoryTransactions`, which correctly pass `skip` and `take` directly into the `FindManyOptions`, delegating pagination to the database engine. [2](#0-1) 

The `whereForUser` filter only excludes terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`, `REJECTED`). It does **not** restrict to transactions the requesting user is involved in at the SQL level — it fetches the full non-terminal set and filters in the application loop. [3](#0-2) 

The endpoint is protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`, meaning any verified user can reach it. The `UserThrottlerGuard` allows up to 10 requests/second per user. [4](#0-3) 

### Impact Explanation

Each call to this endpoint materializes the entire non-terminal transaction table into Node.js heap memory, loads the `groupItem` relation for every row, and then executes one `userKeysToSign` async call per row. As the system accumulates transactions over time (normal operation), a single authenticated request can:

- Exhaust heap memory on the API process, triggering OOM or GC pressure.
- Saturate the PostgreSQL connection pool with a large unbounded query.
- Block the event loop during the synchronous slice of the result set.

This degrades or crashes the API service for all users — a denial-of-service impact without requiring any privileged access.

### Likelihood Explanation

- **Precondition**: Valid authenticated account (any verified user; no admin role required).
- **Trigger**: A single `GET` request to the transactions-to-sign endpoint.
- **Amplification**: The cost scales with the total number of non-terminal transactions in the system, which grows naturally over time without attacker intervention.
- **Rate limit**: 10 req/s per user is sufficient to sustain continuous pressure once the transaction count is large.

Likelihood is **low-medium**: the impact is severe only when the transaction table is large, but no special crafting is needed beyond a valid session token.

### Recommendation

Apply the database-level `take` limit **before** the fetch, mirroring the pattern used in `getTransactions`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // push pagination into SQL
  skip: offset,
});
```

Because `userKeysToSign` filters further in memory, the total count returned may be less than `limit`. If an accurate `totalItems` count is required, issue a separate `COUNT` query with the same `where` clause rather than fetching all rows.

### Proof of Concept

1. Register and verify a normal user account.
2. Ensure the system has accumulated a large number of non-terminal transactions (e.g., thousands created by other users over normal operation).
3. Send a single authenticated request:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <valid_jwt>
   ```
4. Observe: the API process fetches all non-terminal transactions into memory, iterates over each with `userKeysToSign`, and only then slices to 10 items. Memory and CPU spike proportionally to the total transaction count. Repeating at 10 req/s (within the user throttle limit) sustains the pressure. [5](#0-4) [6](#0-5)

    user: User,

    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }

export const PaginationParams = createParamDecorator((data, ctx: ExecutionContext): Pagination => {
  const req: Request = ctx.switchToHttp().getRequest();
  const page = parseInt(req.query.page as string);
  const size = parseInt(req.query.size as string);

  if (isNaN(page) || page <= 0 || isNaN(size) || size < 0) {
    throw new BadRequestException(ErrorCodes.IPP);
  }

  if (size > 100) {
    throw new BadRequestException(ErrorCodes.IPP);
  }

  // calculate pagination parameters
  const limit = size;
  const offset = (page - 1) * limit;
  return { page, limit, size, offset };
