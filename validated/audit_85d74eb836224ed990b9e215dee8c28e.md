Audit Report

## Title
Missing User-Scoped Access Control in `getHistoryTransactions` Exposes All Users' Terminal Transactions

## Summary
The `GET /transactions/history` endpoint returns all terminal-state transactions from every user in the organization to any authenticated requester. No per-user filter is applied. Every other data-retrieval endpoint in the controller correctly scopes results to the requesting user.

## Finding Description

**Controller — no `@GetUser()` decorator:**

`getHistoryTransactions` in `transactions.controller.ts` (lines 144–154) accepts no `user` parameter and passes none to the service: [1](#0-0) 

**Service — no user filter applied:**

`getHistoryTransactions` in `transactions.service.ts` (lines 220–249) accepts no `User` argument. Its `findOptions.where` clause filters only on `status` and any client-supplied `filter` params — never on the requesting user's identity: [2](#0-1) 

**Contrast with the correctly-scoped sibling:**

`getTransactions` (lines 150–217) receives `user: User` and restricts results to transactions where the user is creator, signer, observer, or approver via a multi-branch `WHERE` clause and a recursive approver subquery: [3](#0-2) 

The controller's `getTransactions` handler uses `@GetUser()` and passes the user through: [4](#0-3) 

**Additional aggravating factor — `verifyAccess` bypasses checks for terminal transactions:**

`verifyAccess` unconditionally returns `true` for any transaction in a terminal status (EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED), meaning even the `GET /:id` endpoint imposes no ownership check on historical records: [5](#0-4) 

**Front-end usage note:**

The `/* NO LONGER USED BY FRONT-END */` comment appears on `getTransactions`, `getTransactionsToSign`, and `getTransactionsToApprove` — but **not** on `getHistoryTransactions`, confirming the history endpoint is the active primary history view: [6](#0-5) 

## Impact Explanation

Any authenticated organization member can issue `GET /transactions/history` (with optional pagination, sort, and filter query parameters) and receive the complete set of terminal-state transactions for every other user in the organization. Exposed fields include transaction IDs, types, amounts, counterparties, creator key associations (linking public keys to users), raw `transactionBytes` (the signed Hedera payload), mirror network, and valid-start timestamps. This is a cross-tenant data exposure of sensitive financial records requiring no special privilege.

## Likelihood Explanation

- **Precondition**: a valid JWT for any registered organization account.
- **Attack path**: a single `GET /transactions/history` HTTP request — no special parameters, no privilege escalation.
- **Guards present**: `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard` — identical to every other transaction endpoint; none perform per-user data scoping.
- The endpoint is actively used by the front-end (no deprecation comment), maximizing real-world exposure surface.

## Recommendation

Pass the authenticated user into `getHistoryTransactions` and apply the same user-scoping logic used in `getTransactions`:

1. Add `@GetUser() user: User` to the `getHistoryTransactions` controller handler and forward it to the service.
2. Add a `user: User` parameter to `TransactionsService.getHistoryTransactions`.
3. Extend the `findOptions.where` clause to restrict results to transactions where `creatorKey.userId`, `signers.userId`, `observers.userId`, or the approver subquery matches `user.id` — mirroring the pattern in `getTransactions`.
4. Separately, reconsider the unconditional `return true` in `verifyAccess` for terminal statuses, as it also allows any authenticated user to fetch any individual historical transaction by ID if they can guess or enumerate it.
