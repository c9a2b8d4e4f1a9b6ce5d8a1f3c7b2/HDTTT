### Title
Any Transaction Creator Can Remove Approvers from Transactions They Do Not Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of `:transactionId`, but then removes the approver identified by `:id` without checking that this approver actually belongs to `:transactionId`. Any authenticated user who owns at least one transaction can exploit this to silently strip all approvers from any other user's transaction, bypassing the approval gate and allowing that transaction to proceed to execution without required sign-off.

### Finding Description
In `approvers.controller.ts`, the delete handler performs two sequential calls:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

`getCreatorsTransaction` only verifies that the authenticated user is the creator of the URL-supplied `transactionId`. It says nothing about the approver `id`. [2](#0-1) 

`removeTransactionApprover` then fetches the approver purely by its own primary key and deletes it, with no check that it belongs to `transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [3](#0-2) 

`getTransactionApproverById` fetches by `id` alone, with no `transactionId` filter: [4](#0-3) 

**Exploit flow:**
1. Attacker (normal authenticated user) creates their own transaction `T_attacker` — this satisfies the creator check.
2. Attacker discovers `approver_id` belonging to victim's transaction `T_victim` (approver IDs are sequential integers; the attacker can enumerate them or observe them as a legitimate approver on another transaction).
3. Attacker sends: `DELETE /transactions/T_attacker/approvers/{approver_id_from_T_victim}`.
4. `getCreatorsTransaction(T_attacker, attacker)` passes — attacker owns `T_attacker`.
5. `removeTransactionApprover(approver_id_from_T_victim)` executes — the approver from `T_victim` is deleted with no further check.
6. Repeat for all approvers on `T_victim` to fully strip its approval requirement.

Note: `updateTransactionApprover` does **not** share this flaw — it correctly verifies `rootNode.transactionId !== transactionId` before acting. [5](#0-4) 

### Impact Explanation
Removing all approvers from a victim's transaction eliminates the approval gate. The transaction's status can then advance to `WAITING_FOR_EXECUTION` and be submitted to the Hedera network without the required organizational sign-off. Depending on the transaction type (HBAR transfer, account update, file update, etc.), this can result in unauthorized fund movement or unauthorized state changes on the Hedera ledger. The impact is unauthorized state mutation / unauthorized asset movement — the direct analog to the external report's "drain vault without conditions."

### Likelihood Explanation
The attacker requires only a valid authenticated account in the organization (no admin role, no leaked credentials). Approver IDs are sequential integers, making enumeration trivial. The attack requires no special timing or race condition. Any organization member who can create a transaction can exploit this immediately.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the `transactionId` supplied in the URL. The simplest fix is to add a `transactionId` membership check in the service, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the expected transaction
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

### Proof of Concept
**Preconditions:** Two organization users, Alice (victim) and Bob (attacker), both authenticated.

1. Alice creates transaction `T_alice` (id=10) and adds approver records (ids=5, 6, 7) requiring 2-of-3 approval before execution.
2. Bob creates any transaction `T_bob` (id=99) — this is only needed to pass the creator check.
3. Bob sends:
   - `DELETE /transactions/99/approvers/5` → 200 OK, approver 5 deleted from Alice's transaction.
   - `DELETE /transactions/99/approvers/6` → 200 OK, approver 6 deleted.
   - `DELETE /transactions/99/approvers/7` → 200 OK, approver 7 deleted.
4. Alice's transaction `T_alice` now has zero approvers. The approval gate is gone; the transaction can proceed to execution without any organizational approval, submitting an arbitrary Hedera transaction (e.g., HBAR transfer) to the network.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
