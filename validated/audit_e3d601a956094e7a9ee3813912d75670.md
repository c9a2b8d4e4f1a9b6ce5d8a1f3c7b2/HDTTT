### Title
Cross-Transaction Approver Deletion via Missing Ownership Binding in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId` but never verifies that the approver record `:id` actually belongs to that transaction. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system, bypassing the approval workflow entirely.

### Finding Description

**Root cause — two-step authorization with a broken binding check:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```
await this.approversService.getCreatorsTransaction(transactionId, user);  // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);                // deletes approver by id — no binding check
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction named in the URL path: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by its own `id` and deletes it with no check that `approver.transactionId` matches the `transactionId` from the URL: [3](#0-2) 

**Exploit flow:**

1. Attacker registers and creates any transaction (Transaction A). They are now its creator.
2. Victim creates Transaction B and assigns approvers (e.g., approver record with `id = 42`, `transactionId = B`).
3. Attacker calls:
   ```
   DELETE /transactions/{A_id}/approvers/42
   ```
4. `getCreatorsTransaction(A_id, attacker)` passes — attacker IS the creator of A.
5. `removeTransactionApprover(42)` fetches approver 42, finds it exists, and deletes it — no check that `approver.transactionId === A_id`.
6. Approver 42 is permanently removed from Transaction B.

Approver IDs are sequential integers, making enumeration trivial.

### Impact Explanation

An attacker can silently delete any approver from any transaction they do not own. This:
- Removes required approval gates, potentially allowing a transaction to advance to `WAITING_FOR_EXECUTION` or be submitted to the Hedera network without the intended approvals.
- Corrupts the integrity of the multi-signature/approval workflow, which is the core security model of the tool.
- Is a permanent state change — deleted approvers are soft-removed and the approval record is gone.

Severity: **Critical** — unauthorized state mutation that directly undermines the trust model of the approval system.

### Likelihood Explanation

- **Preconditions**: The attacker only needs a valid account and to have created one transaction (a normal user action).
- **No privileged access required**: Standard authenticated user.
- **Enumeration**: Approver IDs are auto-incremented integers; an attacker can iterate `id` values starting from 1.
- **No rate limiting** on this endpoint is visible in the codebase.

Likelihood: **High**.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the fetched approver belongs to the transaction specified in the URL:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // ADD THIS CHECK:
  if (approver.transactionId !== transactionId) {
    throw new UnauthorizedException('Approver does not belong to this transaction');
  }

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, mirroring the pattern already used by `getCreatorsTransaction`.

### Proof of Concept

**Setup:**
- User A (attacker) creates Transaction 1 (`id=1`). Attacker is its creator.
- User B (victim) creates Transaction 2 (`id=2`) with an approver record (`id=5`, `transactionId=2`).

**Attack request (attacker's session):**
```http
DELETE /transactions/1/approvers/5
Authorization: Bearer <attacker_jwt>
```

**Server execution trace:**
1. `getCreatorsTransaction(1, attacker)` → passes, attacker owns Transaction 1. [4](#0-3) 
2. `removeTransactionApprover(5)` → fetches approver `id=5`, it exists, deletes it. No `transactionId` binding check. [5](#0-4) 

**Result:** Approver 5 is deleted from Transaction 2 (victim's transaction). Transaction 2's approval requirement is silently bypassed. Response: `HTTP 200 true`.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
