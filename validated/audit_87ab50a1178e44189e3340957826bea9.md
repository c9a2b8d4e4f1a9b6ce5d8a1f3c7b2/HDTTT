Audit Report

## Title
TOCTOU Race Condition in `uploadKey` Allows Bypassing `MAX_USER_KEYS` Limit

## Summary
The `uploadKey` function in `UserKeysService` enforces a per-user key limit by reading the current key count from the database and comparing it to `MAX_USER_KEYS` (500). Because the check and the subsequent insert are not atomic and no database-level constraint exists, a user can send multiple concurrent `POST /user/:userId/keys` requests to bypass the limit and register more keys than allowed.

## Finding Description

**Root cause — non-atomic check-then-insert:**

`attachKeys` loads the user's current keys from the database only when `user.keys` is empty, which is always the case for a freshly-decoded JWT user object: [1](#0-0) 

`uploadKey` then checks the loaded count against `MAX_USER_KEYS`: [2](#0-1) 

If the check passes, a new `UserKey` is created and persisted: [3](#0-2) 

There is no database-level constraint on the count of keys per user. The `UserKey` entity defines only a non-unique index on `publicKey` — no `CHECK` constraint or unique partial index guards the per-user count: [4](#0-3) 

Each concurrent HTTP request creates its own `user` object with `keys` empty, calls `attachKeys` independently, reads the same committed snapshot count from the database, passes the check, and then saves a new key.

The e2e test that validates the limit is explicitly skipped: [5](#0-4) 

**Exploit path:**

1. Attacker authenticates as a normal user (no privileged access required).
2. Attacker fires N concurrent `POST /user/:userId/keys` requests, each with a distinct new public key.
3. All N requests call `attachKeys`, all read the same key count from the database simultaneously.
4. All N requests evaluate `count >= MAX_USER_KEYS` → `false`.
5. All N requests proceed to `repo.save(userKey)`.
6. Result: user accumulates far more than `MAX_USER_KEYS` keys.

## Impact Explanation

- **Resource exhaustion**: Unbounded rows in the `user_key` table degrade database performance for all users.
- **Signature engine abuse**: Functions such as `keysRequiredToSign` iterate over all user keys. An attacker with thousands of keys can cause CPU/memory exhaustion in the chain service during signature computation. [6](#0-5) 
- **Business logic integrity**: The key limit (`MAX_USER_KEYS = 500`) is a deliberate invariant. Bypassing it undermines trust assumptions made throughout the codebase wherever `attachKeys` results are used for authorization decisions. [7](#0-6) 

## Likelihood Explanation

- **Attacker preconditions**: Only a valid authenticated account is required — no admin or privileged role.
- **Exploitation complexity**: Sending concurrent HTTP requests is trivial with any HTTP client (e.g., `Promise.all` in Node.js, `asyncio.gather` in Python, or `curl` with `&`).
- **No rate limiting** is present in the controller or service layer. The only guards are `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`, none of which throttle request rate. [8](#0-7) 

## Recommendation

Apply one or more of the following mitigations:

1. **Database-level constraint**: Add a `CHECK` constraint or enforce the limit via a trigger at the PostgreSQL level so no application-layer race can bypass it.
2. **Pessimistic locking**: Wrap the `attachKeys` + count check + insert in a serializable transaction with a `SELECT … FOR UPDATE` on the user row, preventing concurrent inserts from reading stale counts.
3. **Atomic count-guarded insert**: Replace the read-check-insert pattern with a single `INSERT … WHERE (SELECT COUNT(*) FROM user_key WHERE "userId" = $1 AND "deletedAt" IS NULL) < $MAX` query.
4. **Re-enable and fix the skipped e2e test**: The skipped test at line 192 of `user-keys.e2e-spec.ts` should be restored and extended to cover concurrent requests.

## Proof of Concept

```typescript
// Node.js PoC — fires MAX_USER_KEYS + 10 concurrent POST requests
const axios = require('axios');
const { PrivateKey } = require('@hashgraph/sdk');

const TOKEN = '<valid_jwt>';
const BASE  = 'http://localhost:3000';
const N     = 510; // exceeds MAX_USER_KEYS (500)

const requests = Array.from({ length: N }, () => {
  const publicKey = PrivateKey.generateED25519().publicKey.toStringRaw();
  return axios.post(
    `${BASE}/user/2/keys`,
    { publicKey },
    { headers: { Authorization: `Bearer ${TOKEN}` } }
  ).catch(e => e.response);
});

Promise.all(requests).then(results => {
  const created = results.filter(r => r.status === 201).length;
  console.log(`Keys created: ${created}`); // expect > 500
});
```

All N requests read the same committed key count before any insert completes, all pass the `>= MAX_USER_KEYS` guard, and all succeed — leaving the user with `N` keys instead of the intended maximum of 500.
