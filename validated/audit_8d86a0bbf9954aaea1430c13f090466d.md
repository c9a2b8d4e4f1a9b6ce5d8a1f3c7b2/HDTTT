# Audit Report

## Title
Unbounded Full-Table Scan and Per-Row Async Processing in `getTransactionsToSign` Causes Server-Side DoS

## Summary
`TransactionsService.getTransactionsToSign` fetches every non-terminal transaction from the database with no row limit, then iterates over all of them calling the expensive `userKeysToSign` (which deserializes transaction bytes and makes mirror-node network calls) for each row. Pagination is applied only after the full scan completes. Any authenticated, verified user can trigger this endpoint; as the transaction table grows, the server exhausts memory, CPU, and open connections.

## Finding Description

**Root cause — missing `take` on the database query:**

The `repo.find()` call in `getTransactionsToSign` has no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`), so every `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION` transaction in the entire organization is loaded into the Node.js process heap. [2](#0-1) 

**Unbounded per-row async loop:**

Every loaded transaction is then processed individually in a sequential `for...of` loop:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [3](#0-2) 

Each iteration calls `userKeysToSign`, which internally calls `computeSignatureKey`. `computeSignatureKey` performs:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive protobuf deserialization
2. `accountCacheService.getAccountInfoForTransaction(...)` — outbound mirror-node HTTP call per signing account
3. Optionally `nodeCacheService.getNodeInfoForTransaction(...)` — another outbound call [4](#0-3) 

**Pagination applied only after the full scan:**

The `PaginationParams` decorator enforces a maximum page size of 100 on the *request parameter*, but this has no effect on the database query or the loop:

```typescript
if (size > 100) {
  throw new BadRequestException(ErrorCodes.IPP);
}
``` [5](#0-4) 

The slice is applied only after all rows are processed:
```typescript
items: result.slice(offset, offset + limit),
``` [6](#0-5) 

**Attacker-controlled entry path:**

The endpoint is reachable by any authenticated, verified user — no admin role required. The controller exposes it under `GET /transactions/sign` and is still active (even though the comment notes it is "NO LONGER USED BY FRONT-END"): [7](#0-6) 

## Impact Explanation

- **Memory exhaustion**: Loading thousands of transaction rows (each containing serialized transaction bytes) into the Node.js heap can trigger OOM kills.
- **CPU exhaustion**: Protobuf deserialization of every transaction byte blob on every request saturates the event loop.
- **Connection pool exhaustion**: Outbound mirror-node HTTP calls per transaction row can exhaust the outbound connection pool, blocking other services.
- **Cascading denial of service**: Because the loop is sequential and awaited, a single slow mirror-node response stalls the entire request, holding the HTTP connection open and blocking NestJS worker threads.
- All other API users are affected while the server is degraded.

## Likelihood Explanation

Any authenticated organization member can trigger this. The load-test seed data explicitly demonstrates 500 active transactions as a realistic and intended scale: [8](#0-7) 

At 500 active transactions, each request to `GET /transactions/sign` triggers 500 deserialization operations and up to 500+ outbound mirror-node calls. No special privilege or exploit is required beyond a valid JWT and verified status.

## Recommendation

1. **Add a `take` clause to the database query** in `getTransactionsToSign` to limit the number of rows fetched from the database to a reasonable maximum (e.g., the requested page size or a hard cap).
2. **Push filtering into the database**: Rather than loading all transactions and filtering in-memory via `userKeysToSign`, use a database-side join or subquery to pre-filter transactions that require the user's keys before loading them.
3. **Consider deprecating or removing the endpoint entirely**, given the controller comment already states it is "NO LONGER USED BY FRONT-END". The replacement `/transaction-nodes?collection=READY_TO_SIGN` endpoint uses a more efficient query pattern.
4. **Add rate limiting** on the `GET /transactions/sign` endpoint as a defense-in-depth measure.

## Proof of Concept

1. Register and verify an account in the organization.
2. Create N transactions (e.g., N = 500) via `POST /transactions` using the verified account's key, leaving them in `WAITING_FOR_SIGNATURES` status.
3. Repeatedly call `GET /transactions/sign?page=1&size=1` with the valid JWT.
4. Each call causes the server to: (a) load all N transaction rows from the database, (b) deserialize all N transaction byte blobs, (c) make up to N+ outbound mirror-node HTTP calls sequentially.
5. Observe server CPU, memory, and response latency degrade proportionally to N. With N = 500 (the seeded load-test volume), each request performs 500 deserialization operations and 500+ network calls before returning a single result.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
