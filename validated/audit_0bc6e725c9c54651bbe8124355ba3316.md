Audit Report

## Title
Cross-Transaction Approver Deletion via IDOR in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authenticates the caller as the creator of `:transactionId` but performs no cross-check to confirm that the approver record `:id` belongs to that transaction. Any verified organization member who owns at least one transaction can delete approvers from any other transaction by supplying their own `transactionId` for the ownership check and an arbitrary approver `id` for the deletion.

## Finding Description

**Root cause — two independent lookups with no binding check:**

In `approvers.controller.ts`, the delete handler executes two sequential service calls:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // verifies user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no ownership check
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of `:transactionId`; it says nothing about whether the approver record `:id` belongs to that transaction. [2](#0-1) 

`removeTransactionApprover` then fetches the approver by its own primary key and deletes it unconditionally — there is no assertion that `approver.transactionId === transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // no transactionId check
  emitTransactionStatusUpdate(...);
  return result;
}
``` [3](#0-2) 

`removeNode` performs a recursive soft-delete of the entire subtree rooted at the given node, again with no transaction ownership guard: [4](#0-3) 

**Clear inconsistency with the update path:**

`updateTransactionApprover` explicitly walks up to the root node and asserts `rootNode.transactionId !== transactionId` before proceeding, making the omission in the delete path a deliberate divergence rather than an oversight in design: [5](#0-4) 

## Impact Explanation

Deleting an approver from another user's transaction directly undermines the multi-signature approval workflow:

- If the deleted approver was the sole required approver, the transaction can advance to execution with zero approvals.
- If the deleted node was an interior node of a threshold tree, `removeNode` recursively soft-deletes the entire subtree, silently eliminating multiple required signers at once.
- The attacker leaves no obvious trace on their own transaction (`txA`), making forensic attribution harder.

This constitutes an unauthorized, persistent state change against another user's transaction with direct integrity impact on the approval/execution pipeline.

## Likelihood Explanation

- The attacker must be a registered, verified member of the same organization — a low but realistic bar (any internal user qualifies, no admin/owner role required).
- Approver IDs are sequential database integers, making enumeration straightforward with a small number of probe requests.
- The attack requires exactly one authenticated HTTP request.
- No special tooling or knowledge of cryptographic material is needed.

## Recommendation

In `removeTransactionApprover` (service), after fetching the approver, resolve its root node and assert that the root's `transactionId` matches the `transactionId` parameter — exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Mirror the check already present in updateTransactionApprover
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller to the service method, and update the controller call accordingly.

## Proof of Concept

**Setup:**
- Alice (attacker) is a verified org member; she created `txA` (id = 10).
- Bob (victim) created `txB` (id = 20), which has a required approver record with `id = 99` (a root node with `transactionId = 20`).

**Attack request:**
```http
DELETE /transactions/10/approvers/99
Authorization: Bearer <alice-jwt>
```

**Execution trace:**
1. `getCreatorsTransaction(10, alice)` — passes; Alice owns `txA`.
2. `removeTransactionApprover(99)` — fetches approver `99` (belongs to `txB`), calls `removeNode(99)`.
3. `removeNode(99)` recursively soft-deletes approver `99` and all its children.
4. Bob's transaction `txB` now has its approver tree partially or fully destroyed, potentially allowing execution without any approval.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1

              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"

      set "deletedAt" = now()

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));
