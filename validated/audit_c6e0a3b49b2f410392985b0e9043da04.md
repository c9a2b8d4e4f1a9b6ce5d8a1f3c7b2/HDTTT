Audit Report

## Title
`GET /user-keys` Returns All Users' Keys Including `mnemonicHash` and `index` to Any Verified User

## Summary
`UserKeysAllController.getUserKeys` at `GET /user-keys` performs a completely unfiltered database query returning every row in the `user_key` table — including `mnemonicHash` and `index` — to any authenticated, verified organization user. No admin role or ownership check is enforced.

## Finding Description
The controller at `back-end/apps/api/src/user-keys/user-keys-all.controller.ts` accepts no user identity parameter and applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`: [1](#0-0) 

It delegates to `UserKeysService.getUserKeys`, which issues a bare `findAndCount` with no `where` clause and no `select` projection: [2](#0-1) 

The serializer `UserKeyDto` marks every field — including `mnemonicHash` and `index` — with `@Expose()`, so all columns are passed through to the response: [3](#0-2) 

The correctly-scoped sibling `getUserKeysRestricted` demonstrates the developers' intent: `mnemonicHash` and `index` are only returned when `user.id === userId`: [4](#0-3) 

That restriction is entirely absent from `getUserKeys`.

## Impact Explanation
Any verified user receives the full paginated dump of every `user_key` row in the organization. Because `UserKeyDto` exposes `mnemonicHash` and `index`, every caller receives:

- The `mnemonicHash` (hash of every other user's mnemonic recovery phrase) — useful for offline dictionary/rainbow attacks against weak mnemonics and for confirming whether two users share the same recovery phrase.
- The `index` (BIP-32 derivation index) — narrows the key-derivation search space for an attacker.
- The `publicKey` of every registered key — enables complete key-to-user mapping across the organization.

This directly contradicts the access model enforced by `getUserKeysRestricted`, where those fields are withheld for keys belonging to other users.

## Likelihood Explanation
The endpoint requires no elevated role — only a valid JWT and `VerifiedUserGuard` (non-new account status). The front-end calls it unconditionally via `getAllUserKeys` on every contact-list load: [5](#0-4) [6](#0-5) 

Any verified user — including a newly onboarded or compromised account — can exploit this immediately with a single HTTP request.

## Recommendation
Replace the unfiltered `getUserKeys` call in `UserKeysAllController` with a query that either:

1. Restricts results to the calling user's own keys (using `@GetUser()` and a `where: { userId: user.id }` clause), or
2. Applies the same `select` projection as `getUserKeysRestricted` — omitting `mnemonicHash` and `index` for rows not owned by the caller — if cross-user public-key enumeration is an intentional feature.

Additionally, add `AdminGuard` if the intent is to expose all keys only to administrators.

## Proof of Concept
```
GET /user-keys?page=1&size=100 HTTP/1.1
Authorization: Bearer <any_verified_user_jwt>
```

Response includes every `user_key` row:
```json
{
  "totalItems": N,
  "items": [
    {
      "id": 1,
      "userId": 42,
      "mnemonicHash": "<hash_of_user_42s_mnemonic>",
      "index": 0,
      "publicKey": "302a...",
      "deletedAt": null
    },
    ...
  ]
}
```

The caller need not own any of the returned keys. Iterating pages exhausts the entire `user_key` table.

export const getAllUserKeys = async (organizationServerUrl: string): Promise<IUserKey[]> => {
  let page = 1;
  const size = 100;
  let totalItems = 0;
  const allUserKeys: IUserKey[] = [];

  do {
    const { data, error } = await safeAwait(
      getUserKeysPaginated(organizationServerUrl, page, size),
    );
    if (data) {
      totalItems = data.totalItems;
      allUserKeys.push(...data.items);
      page++;
    }

    if (error) {
      break;
    }
  } while (allUserKeys.length < totalItems);

  return allUserKeys;
