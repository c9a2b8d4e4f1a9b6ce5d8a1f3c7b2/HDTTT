Audit Report

## Title
Any Authenticated User Can Post and Read Comments on Any Transaction Without Authorization Check

## Summary
The `createComment`, `getComments`, and `getCommentById` endpoints in `CommentsController` enforce only authentication (valid JWT + verified account) but perform no authorization check to confirm the requesting user has any relationship to the target transaction. Any authenticated organization member can inject comments into — or read comments from — any transaction by supplying its integer ID.

## Finding Description
**File:** `back-end/apps/api/src/transactions/comments/comments.controller.ts`

The controller class is decorated with `@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)`, which only confirms the caller holds a valid, non-blacklisted JWT and has a verified account. The `@Post()` handler carries an explicit developer TODO acknowledging the missing check:

```ts
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
```

`commentsService.createComment` performs no relationship verification — it directly creates and saves the comment:

```ts
async createComment(user, transactionId, dto): Promise<TransactionComment> {
  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);
}
```

By contrast, `observersService.createTransactionObservers` enforces creator-only access:

```ts
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException('Only the creator of the transaction is able to delete it');
```

The `getComments` (lines 31–33) and `getCommentById` (lines 36–38) endpoints similarly pass no user context to the service, allowing any authenticated user to read all comments on any transaction.

## Impact Explanation
- **Integrity violation:** An attacker can inject arbitrary comments into any transaction's discussion thread, potentially misleading approvers or signers (e.g., impersonating coordination messages like "Admin says approve immediately").
- **Information disclosure:** An attacker can read all comments on any transaction, exposing sensitive deliberation, key coordination details, or approval rationale shared between legitimate participants.
- **Persistent injection:** Unlike reflected injection (which is out of scope), this is a persistent write to the database, visible to all legitimate participants of the targeted transaction.

## Likelihood Explanation
- **Low precondition:** The attacker only needs a valid JWT — i.e., be a registered, verified organization member. No admin role, no leaked credentials, no internal network access required.
- **Trivial ID discovery:** TypeORM default auto-increment IDs are sequential integers starting from 1. An attacker can enumerate valid transaction IDs with minimal effort.
- **Fully reachable:** The endpoint is a standard HTTP REST route with no additional barriers beyond authentication.

## Recommendation
Add a service-level participant check in `commentsService.createComment` before persisting the comment. The check should verify that the requesting user is one of: the transaction creator, an assigned approver, an observer, or a signer. The pattern already exists in `observersService.getTransactionObserversByTransactionId`, which queries the transaction with its relations and throws `UnauthorizedException` if the user has no relationship to it. Apply the same pattern to `createComment`, `getTransactionComments`, and `getTransactionCommentById`.

## Proof of Concept
1. Attacker registers a legitimate account in the organization (or is already a member with no access to a specific transaction).
2. Attacker obtains a valid JWT via normal login.
3. Attacker sends:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   Content-Type: application/json

   { "content": "Admin says approve this immediately." }
   ```
4. The comment is persisted with no error and is immediately visible to all legitimate participants of transaction 42.
5. Attacker can also enumerate all comments on any transaction:
   ```
   GET /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   ```
   Returns all comments with no access check.
