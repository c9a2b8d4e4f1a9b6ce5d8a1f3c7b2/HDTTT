Audit Report

## Title
Wrong Preference Field Checked in `filterReceiversByPreferenceForType` Causes Incorrect Email Notification Delivery

## Summary
`filterReceiversByPreferenceForType` in `back-end/apps/notifications/src/receiver/receiver.service.ts` always evaluates `preference.inApp` regardless of the delivery channel. Inside `handleUserRegisteredNotifications`, this function is called twice with **identical arguments** — once intended for in-app filtering and once for email filtering — producing two identical sets. As a result, `isEmailSent` is set based on the in-app opt-in flag instead of the email opt-in flag, causing email notifications to be sent to admins who opted out and silently dropped for admins who opted in.

## Finding Description

**Root cause — `filterReceiversByPreferenceForType` always reads `preference.inApp`:**

The function accepts no channel parameter and unconditionally checks `preference.inApp`:

```typescript
// receiver.service.ts lines 286–292
const preference = user.notificationPreferences?.find(
  p => p.type === notificationType
);

if (preference ? preference.inApp : true) {   // ← always inApp, never email
  result.push(id);
}
``` [1](#0-0) 

The `NotificationPreferences` entity exposes a distinct `email` boolean column that is never read here: [2](#0-1) 

**Call site — `handleUserRegisteredNotifications` calls it twice with identical arguments:**

```typescript
// receiver.service.ts lines 904–917
const inAppReceiverUserIds = await this.filterReceiversByPreferenceForType(
  entityManager, NotificationType.USER_REGISTERED, adminUserIds, cache,
);

const emailReceiverUserIds = await this.filterReceiversByPreferenceForType(
  entityManager, NotificationType.USER_REGISTERED, adminUserIds, cache,
  // ↑ identical call — emailReceiverUserIds === inAppReceiverUserIds always
);
``` [3](#0-2) 

**Downstream — `isEmailSent` is set from the wrong set:**

```typescript
// receiver.service.ts lines 942–943
isInAppNotified: inAppReceiverUserIds.includes(adminUserId) ? false : null,
isEmailSent:     emailReceiverUserIds.includes(adminUserId) ? false : null,
// ↑ emailReceiverUserIds was filtered by preference.inApp, not preference.email
``` [4](#0-3) 

## Impact Explanation

| Admin preference state | Expected behaviour | Actual behaviour |
|---|---|---|
| `inApp: false`, `email: true` | No in-app; receives email | `isEmailSent: null` → **email silently dropped** |
| `inApp: true`, `email: false` | Receives in-app; no email | `isEmailSent: false` → **unwanted email sent** |
| `inApp: true`, `email: true` | Both channels | Both channels (correct by coincidence) |
| `inApp: false`, `email: false` | Neither channel | Neither channel (correct by coincidence) |

Admins who explicitly opted out of email receive it anyway (spam/privacy violation). Admins who opted into email but not in-app silently miss every `USER_REGISTERED` notification.

## Likelihood Explanation

This fires on every user registration event routed through `processUserRegisteredNotifications`. No attacker action is required; any admin whose `inApp` and `email` preferences differ will be affected on every registration. The divergence is a supported, documented user-configurable state (the `PATCH /notification-preferences` endpoint accepts independent `email` and `inApp` booleans). [5](#0-4) 

## Recommendation

Add a `channel` parameter to `filterReceiversByPreferenceForType` (or create a sibling function) so the correct preference field is read per call:

```typescript
private async filterReceiversByPreferenceForType(
  entityManager: EntityManager,
  notificationType: NotificationType,
  userIds: Set<number>,
  cache: Map<number, User>,
  channel: 'inApp' | 'email' = 'inApp',   // ← add this
): Promise<number[]> {
  await this.loadUsersWithPreferences(entityManager, Array.from(userIds), cache);
  const result: number[] = [];
  for (const id of userIds) {
    const user = cache.get(id);
    if (!user) continue;
    const preference = user.notificationPreferences?.find(
      p => p.type === notificationType
    );
    if (preference ? preference[channel] : true) {   // ← use channel
      result.push(id);
    }
  }
  return result;
}
```

Then update the two call sites in `handleUserRegisteredNotifications`:

```typescript
const inAppReceiverUserIds = await this.filterReceiversByPreferenceForType(
  entityManager, NotificationType.USER_REGISTERED, adminUserIds, cache, 'inApp',
);
const emailReceiverUserIds = await this.filterReceiversByPreferenceForType(
  entityManager, NotificationType.USER_REGISTERED, adminUserIds, cache, 'email',
);
```

## Proof of Concept

1. Create admin user A with preferences `{ type: USER_REGISTERED, inApp: false, email: true }`.
2. Register a new user, triggering `processUserRegisteredNotifications`.
3. Observe that `handleUserRegisteredNotifications` calls `filterReceiversByPreferenceForType` twice with identical arguments.
4. Both calls return `[]` for admin A (because `preference.inApp === false`).
5. Admin A is excluded from `allReceiverIds` entirely — no `NotificationReceiver` row is created, and no email is sent, despite `email: true`.

Conversely, set `{ inApp: true, email: false }` for admin B:

1. Both calls return `[B]` (because `preference.inApp === true`).
2. Admin B's receiver row is created with `isEmailSent: false`, and the email delivery pipeline sends an email — despite `email: false`.
