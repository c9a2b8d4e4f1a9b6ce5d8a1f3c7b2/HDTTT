### Title
TOCTOU in `approveTransaction` Allows an Approver to Bypass the "Already Approved" Guard and Overwrite Their Approval Decision

### Summary
In `approvers.service.ts`, the `approveTransaction` function reads the approver state (including whether the user has already submitted a signature) outside of any database transaction, then performs the actual write inside a separate database transaction. A user who is a legitimate approver can race two concurrent HTTP requests to bypass the "already approved" guard and overwrite their previously committed approval decision ŌĆö including flipping from `approved: true` to `approved: false` ŌĆö after the guard would normally block them.

The guard at line 563 uses `userApprovers` that was fetched at line 553 ŌĆö a point-in-time snapshot taken **before** the write transaction opens. There is no re-read of the approver state inside the write transaction, and no optimistic-lock or conditional `WHERE signature IS NULL` clause on the `UPDATE`.

**Exploit flow:**

1. Attacker (a legitimate approver) sends **Request A** (`approved: true`) and **Request B** (`approved: false`) concurrently.
2. Both requests execute line 553 before either commits: both see `signature = NULL` ŌåÆ both pass the guard at line 563.
3. Request A commits first: approver row is updated with `approved = true, signature = <sigA>`.
4. Request B commits second: the unconditional `UPDATE ŌĆ” WHERE id IN (ŌĆ”)` overwrites the row with `approved = false, signature = <sigB>`.
5. Final DB state: `approved = false` ŌĆö the user has effectively reversed their approval after the system believed it was locked in.

The same pattern exists in `uploadSignatureMaps` in `signers.service.ts` (lines 99ŌĆō124): `loadTransactionData` reads the transaction status and existing signers at time T1 (outside any DB transaction), `validateTransactionStatus` checks the stale status, and `persistSignatureChanges` writes at time T2. A transaction that is canceled between T1 and T2 will still have signatures persisted against it. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

In a multi-signature approval workflow, the integrity of each approver's decision is a core invariant. By racing two concurrent requests, a legitimate approver can:

- **Flip `approved: true` ŌåÆ `approved: false`** after the system has already recorded their approval, potentially preventing a transaction from reaching its approval threshold and blocking execution.
- **Flip `approved: false` ŌåÆ `approved: true`** after the system has already recorded their rejection, potentially pushing a transaction past its approval threshold when it should have been blocked.
- **Overwrite the stored signature** with a signature from a different key, corrupting the audit trail.

The notification logic at line 614 also uses the stale `userApprovers` snapshot, so the wrong notification type (`emitTransactionStatusUpdate` vs. `emitTransactionUpdate`) may be emitted, causing downstream state inconsistency in connected clients. [4](#0-3) 

### Likelihood Explanation

The attacker is a **normal authenticated user** who is already a designated approver ŌĆö no privileged access is required. Sending two HTTP POST requests to `/transactions/:id/approvers/approve` concurrently is trivially achievable from any HTTP client (e.g., `Promise.all` in a browser, two parallel `curl` calls). The window between the guard read (line 553) and the write commit (lines 599ŌĆō610) includes multiple async DB calls (`attachKeys`, `findOne` for the transaction, signature verification), making the race window wide enough to be reliably exploitable.
