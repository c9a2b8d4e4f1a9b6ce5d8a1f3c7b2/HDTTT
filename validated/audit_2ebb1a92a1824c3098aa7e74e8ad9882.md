# Audit Report

## Title
TOCTOU Race Condition in `approveTransaction` Allows Approver to Overwrite Their Own Approval Decision

## Summary
`ApproversService.approveTransaction()` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` performs a non-atomic check-then-act: it reads approver state, checks for a prior signature, then writes the approval — but the write carries no conditional guard (`AND signature IS NULL`). Two concurrent requests from the same approver both pass the stale-snapshot guard and both execute the unconditional `UPDATE`, with the last write winning. This allows a malicious approver to flip their recorded vote after it has already been committed.

## Finding Description
`approveTransaction` follows this sequence:

1. **Read** — line 553: `getVerifiedApproversByTransactionId` fetches all approvers into `userApprovers` (a plain in-memory snapshot, no DB lock).
2. **Guard** — line 563: `if (userApprovers.every(a => a.signature)) throw BadRequestException(TAP)` — evaluated against the stale snapshot.
3. **Wide async gap** — lines 566–596: `attachKeys`, `findOne` (transaction re-fetch), `verifyTransactionBodyWithoutNodeAccountIdSignature` — all `await`ed sequentially, giving ample time for a second request to pass the same guard.
4. **Unconditional write** — lines 599–610: `.whereInIds(userApprovers.map(a => a.id))` with no `AND signature IS NULL` predicate. Any concurrent request that passed the guard will overwrite whatever was written by the first request.

The notification branch at lines 614–618 also reads `userApprovers.every(a => a.approved)` from the pre-race snapshot, so the emitted status event reflects the stale state, not the actual DB state after the race.

## Impact Explanation
The approval workflow is the trust-enforcement gate before a transaction can be executed. Exploiting this race allows a malicious approver to:

- **Flip their vote** from `approved=true` to `approved=false` after the threshold was already satisfied, blocking execution of a transaction that legitimately cleared approval.
- **Corrupt the recorded key/signature** (`userKeyId`, `signature`) associated with their approval, breaking audit integrity.
- **Emit a misleading status event** to the notification bus because the notification branch reads the pre-race stale `userApprovers` array.

## Likelihood Explanation
The attacker is a legitimate approver — any registered user added to a transaction's approver list. No elevated privileges are required. The attack requires only two near-simultaneous HTTP requests, trivially achievable with `curl`, `axios`, or any scripting tool. The race window is wide because three sequential `await` calls (lines 566–596) separate the guard from the write, giving a second concurrent request ample time to pass the guard before the first write completes.

## Recommendation
Make the write conditional and atomic:

1. **Add `AND signature IS NULL` to the UPDATE predicate** so the write is a no-op if a signature was already committed:
   ```ts
   .where('id IN (:...ids) AND signature IS NULL', { ids: userApprovers.map(a => a.id) })
   ```
2. **Move the guard inside the same DB transaction as the write**, using a `SELECT ... FOR UPDATE` (pessimistic lock) so the read and write are atomic:
   ```ts
   await this.dataSource.transaction(async em => {
     const rows = await em.getRepository(TransactionApprover)
       .createQueryBuilder('a')
       .setLock('pessimistic_write')
       .whereInIds(userApprovers.map(a => a.id))
       .getMany();
     if (rows.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
     // ... perform write
   });
   ```
3. **Re-fetch `userApprovers` after the write** before evaluating the notification branch, so the emitted event reflects actual DB state.

## Proof of Concept
```
Attacker (legitimate approver) sends two concurrent POST /transactions/:id/approvers/approve:

  Request A → { approved: true,  userKeyId: K, signature: validSig }
  Request B → { approved: false, userKeyId: K, signature: validSig }

Timeline:
  A: line 553 — reads userApprovers, signature = null
  B: line 553 — reads userApprovers, signature = null   (same snapshot)
  A: line 563 — guard passes (signature is null)
  B: line 563 — guard passes (signature is null)
  A: lines 566–596 — awaits attachKeys, findOne, verifySignature
  B: lines 566–596 — awaits attachKeys, findOne, verifySignature
  A: lines 599–610 — UPDATE SET approved=true, signature=validSig WHERE id IN (...)
  B: lines 599–610 — UPDATE SET approved=false, signature=validSig WHERE id IN (...)  ← overwrites A

Final DB state: approved=false, even though the approver committed approved=true.
The approval threshold that was satisfied by A's write is now broken.
```
