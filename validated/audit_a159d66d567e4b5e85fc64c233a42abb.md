### Title
Cross-Transaction Approver Deletion via IDOR in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of `:transactionId`, but never validates that the approver `:id` actually belongs to that transaction. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system, bypassing multi-signature approval requirements.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the delete handler at line 102–113 performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the URL-supplied `transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver record with the URL-supplied `id`. [1](#0-0) 

There is no check that the approver `id` belongs to `transactionId`. The service method `removeTransactionApprover` receives only the approver `id` and performs no ownership or transaction-membership validation: [2](#0-1) 

By contrast, the `updateTransactionApprover` service method correctly validates cross-ownership by checking `rootNode.transactionId !== transactionId` before proceeding: [3](#0-2) 

This check is entirely absent in the delete path.

**Exploit flow:**
1. Attacker registers as a normal user and creates any transaction (becomes its creator, obtaining a valid `transactionId_A`).
2. Attacker enumerates approver IDs on victim's transaction (IDs are sequential integers; the GET endpoint `getVerifiedApproversByTransactionId` exposes them to any participant, and IDs can also be brute-forced).
3. Attacker sends:
   ```
   DELETE /transactions/{transactionId_A}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction(transactionId_A, attacker)` passes — attacker is the creator of transaction A.
5. `removeTransactionApprover(victim_approver_id)` executes with no further checks, soft-deleting the victim's approver record.

### Impact Explanation

Removing an approver from a transaction eliminates that party's required approval from the multi-signature workflow. If the deleted approver was the sole or threshold-completing approver, the transaction can proceed to execution without the intended authorization. This constitutes an unauthorized state change that directly undermines the integrity of the organization-mode approval model — the core security guarantee of the product.

### Likelihood Explanation

Any authenticated, verified user who has created at least one transaction can exploit this. No elevated privileges are required. Approver IDs are sequential integers and are disclosed through the normal GET approvers endpoint to transaction participants. The attack requires only a valid JWT and knowledge of a target approver ID, both of which are realistic for a malicious organization member.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted belongs to the `transactionId` supplied in the URL. The simplest fix mirrors the pattern already used in `updateTransactionApprover`:

```typescript
// In removeTransactionApprover or the controller:
const rootNode = await this.getRootNodeFromNode(approver.id, entityManager);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

Alternatively, scope the `getTransactionApproverById` query to also filter by `transactionId` so a mismatched ID simply returns `null`.

### Proof of Concept

**Preconditions:** Two users, Alice (creator of transaction 1) and Bob (creator of transaction 2). Bob is an approver on transaction 1 (approver record ID = 5).

**Steps:**
1. Alice authenticates and obtains a JWT.
2. Alice sends:
   ```
   DELETE /transactions/2/approvers/5
   Authorization: Bearer <alice_jwt>
   ```
3. `getCreatorsTransaction(2, alice)` succeeds — Alice is the creator of transaction 2.
4. `removeTransactionApprover(5)` executes — approver record 5 (Bob on transaction 1) is soft-deleted.
5. Transaction 1 now proceeds without Bob's required approval.

**Expected (correct) behavior:** The server should reject the request with `401 Unauthorized` because approver 5 does not belong to transaction 2.

**Actual behavior:** The approver is deleted. [4](#0-3)

  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);
