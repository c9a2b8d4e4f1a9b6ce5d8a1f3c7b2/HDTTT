Audit Report

## Title
`hasValidSignatureKey` and `ableToSign` Return `true` for Empty `KeyList`, Bypassing Signature Validation

## Summary
Both `hasValidSignatureKey` (back-end) and `ableToSign` (front-end) recursively evaluate a `KeyList` to determine whether sufficient signatures exist. Neither function guards against an **empty `KeyList`** (zero keys, no threshold). When an empty `KeyList` is encountered as a nested entry, the expression `0 >= (null || 0)` evaluates to `true`, causing the function to report the signature requirement as satisfied with zero actual signatures collected.

## Finding Description

**Root cause — `hasValidSignatureKey`:**

The function is confirmed at `back-end/libs/common/src/utils/sdk/key.ts` lines 31–46:

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();          // [] for an empty KeyList
    let currentThreshold = 0;

    keys.forEach(key => {                // loop never executes
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
    //     0                >= (null          || 0          )
    //     0                >= 0   → TRUE  ← false positive
  }
  ...
};
``` [1](#0-0) 

When `key` is an empty `KeyList`: `key.toArray()` returns `[]`, the `forEach` never runs, `currentThreshold` stays `0`, `key.threshold` is `null`, and `keys.length` is `0`. The expression `0 >= (null || 0)` → `0 >= 0` → `true`.

**Identical logic in `ableToSign`:**

Confirmed at `front-end/src/renderer/utils/sdk/index.ts` lines 109–126: [2](#0-1) 

The same `currentThreshold >= (key.threshold || keys.length)` expression is present with no empty-list guard.

**Critical call site 1 — transaction execution gate:**

Confirmed at `back-end/libs/common/src/execute/execute.service.ts` line 219:

```typescript
if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
``` [3](#0-2) 

This is the sole signature check before the transaction is submitted to the Hedera network.

**Critical call site 2 — status promotion:**

Confirmed at `back-end/libs/common/src/utils/transaction/index.ts` lines 134–144:

```typescript
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey
);
if (isAbleToSign) {
  const collatedTx = await smartCollate(transaction, signatureKey);
  if (collatedTx !== null) {
    newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
  }
}
``` [4](#0-3) 

**How the empty `KeyList` enters `signatureKey`:**

`TransactionSignatureService.computeSignatureKey` pushes each account's on-chain key directly into the outer `signatureKey` `KeyList` via `signatureKey.push(accountInfo.key)`. [5](#0-4) 

If any participating account's mirror-node-fetched key is itself an empty `KeyList` (e.g., an immutable account whose protobuf key encodes as an empty `keyList`), that empty `KeyList` is pushed as a nested entry into `signatureKey`. When `hasValidSignatureKey` recurses into it, it returns `true` with zero signatures present, which then increments `currentThreshold` in the parent call, potentially satisfying the parent threshold check as well.

**Note — `isKeyListValid` is not used in the validation path:**

The front-end does define `isKeyListValid` which explicitly rejects empty key lists (`keys.length === 0` → `return false`), but this function is only called from `ComplexKeyModal.vue` for UI-level key construction validation. It is never invoked inside `ableToSign` or `hasValidSignatureKey`. [6](#0-5) 

## Impact Explanation

A transaction that has collected **zero signatures** for an account whose on-chain key resolves to an empty `KeyList` will pass the `hasValidSignatureKey` gate in `execute.service.ts` and be submitted to the Hedera network. Simultaneously, `processTransactionStatus` will promote the transaction to `WAITING_FOR_EXECUTION`, corrupting the tool's internal state. Even if the Hedera network rejects the submission, the multi-signature coordination workflow is bypassed: the transaction is incorrectly marked ready for execution without the required authorizations being collected.

## Likelihood Explanation

Hedera accounts can be created with arbitrary `KeyList` structures. Immutable accounts (accounts intentionally created without a controlling key) are represented on-chain with an empty `KeyList` in the protobuf encoding. A user can:
1. Use an immutable account (or an account whose key contains a nested empty `KeyList`) as a fee payer or required signer in a transaction submitted through the tool.
2. The mirror-node-fetched key propagates the empty `KeyList` into `signatureKey` via `computeSignatureKey`.
3. `hasValidSignatureKey` returns `true` with zero signatures, bypassing the check.

No admin keys, leaked credentials, or out-of-scope capabilities are required.

## Recommendation

Add an explicit guard at the top of both `hasValidSignatureKey` and `ableToSign` to return `false` immediately when the `KeyList` is empty:

```typescript
if (key instanceof KeyList) {
  const keys = key.toArray();
  if (keys.length === 0) return false;  // ← add this guard
  ...
}
```

This mirrors the logic already present in `isKeyListValid` and `computeShortenedPublicKeyList` (which checks `threshold ? threshold : keyList.toArray().length` and handles the empty case implicitly via `result.length + secondary.length >= threshold` being `0 >= 0` — though that function also benefits from the same fix). [7](#0-6)
