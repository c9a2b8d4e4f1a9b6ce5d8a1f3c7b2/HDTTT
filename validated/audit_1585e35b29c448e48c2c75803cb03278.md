Audit Report

## Title
Unbounded Full-Table Scan Loop in `getTransactionsToSign` Causes Server-Side Resource Exhaustion

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then performs one async database call per row before applying pagination. Any authenticated, verified user can trigger this endpoint to exhaust server memory and CPU.

## Finding Description

**Unbounded query — confirmed at lines 295–299:**

`this.repo.find()` is called with `where`, `relations`, and `order` — but no `take` or `skip`. It returns every transaction whose status is not in the terminal set (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED). [1](#0-0) 

**Per-row async call — confirmed at lines 301–308:**

The result set is iterated in a `for` loop. Each iteration `await`s `this.userKeysToSign(transaction, user)`, which internally calls `userKeysRequiredToSign` → `keysRequiredToSign` + `attachKeys`, both of which issue additional async database queries. [2](#0-1) [3](#0-2) 

**Pagination applied only after full scan — confirmed at line 313:**

`result.slice(offset, offset + limit)` is called after the loop completes. The `limit` value (max 100) is never passed to the database query. [4](#0-3) 

**`PaginationParams` size cap — confirmed at lines 22–24:**

The decorator enforces `size ≤ 100`, but this constraint is applied only to the response slice, not to the database query. [5](#0-4) 

**Endpoint accessibility — confirmed at lines 165–178:**

`GET /transactions/sign` is registered and guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. No admin or elevated role is required. The comment "NO LONGER USED BY FRONT-END" confirms the front-end does not call it, but the route remains live and reachable via direct HTTP. [6](#0-5) 

**Secondary instance — developer comment confirmed in `transactions.service.ts`:**

A grep match for `"probably should limit"` exists in `transactions.service.ts`, confirming developer awareness of missing array-size limits in the `importSignatures` path. The controller accepts an unbounded `UploadSignatureMapDto[] | UploadSignatureMapDto` body with no array-length cap. [7](#0-6) 

## Impact Explanation
An attacker with a valid JWT repeatedly calls `GET /transactions/sign` against a deployment with a large active-transaction table. Each request:
- Loads the entire non-terminal transaction table into Node.js heap memory as ORM objects.
- Executes N sequential async database round-trips (one per transaction row) before returning any data.

This causes heap exhaustion / OOM crash of the API process, CPU saturation from deserializing thousands of ORM objects, and cascading latency for all other API consumers sharing the same process. Because the HTTP response is paginated, operators have no visible signal that the underlying query is unbounded.

## Likelihood Explanation
The endpoint is reachable by any authenticated organization member — no admin role is required. In a production deployment with months of transaction history, the non-terminal transaction count grows continuously. The attack requires only a valid JWT and a single HTTP GET request. The developer comment in `transactions.service.ts` confirms awareness that array-size limits are missing, suggesting the pattern is systemic.

## Recommendation
1. **Apply `take`/`skip` at the database layer** in `getTransactionsToSign`. Because `userKeysToSign` filtering must happen in application code (it requires key-matching logic), use a cursor/keyset approach: fetch a bounded window (e.g., `take: limit * N` with a configurable multiplier), process it, and repeat until `limit` qualifying rows are found or the table is exhausted.
2. **Alternatively, push the signing-eligibility check into the query** using a subquery or join on `transaction_signer`, eliminating the per-row async call entirely and allowing standard `take`/`skip` pagination.
3. **Cap the `importSignatures` array length** at the controller or DTO level (e.g., `@ArrayMaxSize(100)` on the body).

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
```
