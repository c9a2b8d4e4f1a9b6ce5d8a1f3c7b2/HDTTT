# Audit Report

## Title
Malicious Organization Member Can Permanently Block Victim's Public Key Registration via Pre-emption DoS

## Summary
Any authenticated organization member can register an arbitrary public key — including one belonging to another user — before that user does. Because `uploadKey` enforces effective per-key uniqueness at the service layer with no proof-of-possession, the victim's subsequent registration attempt is permanently rejected with `ErrorCodes.PU` ("Public key already used"), blocking them from all multi-signature workflows on the server.

## Finding Description

**Root cause — no proof-of-possession on key registration.**

The `UserKey` entity indexes `publicKey` for fast lookup: [1](#0-0) 

`uploadKey` in `user-keys.service.ts` performs a `findOne` by `publicKey` (including soft-deleted rows) and rejects the upload if the found record belongs to a different user: [2](#0-1) 

There is no challenge-response or signature verification to prove the caller controls the private key corresponding to the submitted `publicKey`. Any valid string that passes the `IsHederaPublicKey` format validator is accepted.

The controller resolves the acting user from the JWT, not the URL parameter, so the attacker posts to their own endpoint: [3](#0-2) 

**Attack path:**
1. Attacker (any non-admin org member) obtains the victim's Hedera public key (publicly visible on the mirror node, or via `GET /user/:victimId/keys`).
2. Attacker sends `POST /user/<attacker_id>/keys` with `{ publicKey: <victim_public_key> }`.
3. `uploadKey` finds no existing record → creates `UserKey` row with `userId = attacker.id`, `publicKey = victim_public_key`.
4. Victim sends `POST /user/<victim_id>/keys` with their own public key.
5. `uploadKey` finds the attacker's row → evaluates `userKey.userId !== user.id` → `true` → throws `BadRequestException(ErrorCodes.PU)`.
6. Victim is permanently blocked.

**Why admin remediation does not mitigate:**

Both `removeKey` and `removeUserKey` perform a soft-delete (`softRemove`), setting `deletedAt`: [4](#0-3) 

The `findOne` in `uploadKey` uses `withDeleted: true`, so soft-deleted rows are still found. After admin deletion, the attacker re-registers the same key in one request, instantly restoring the block. There is no rate-limiting specific to this endpoint beyond the global throttler.

## Impact Explanation
A victim who cannot register their public key:
- Cannot be added as a signer or approver to any organization transaction.
- Cannot sign transactions already assigned to them (their `UserKey` record does not exist under their `userId`).
- Is effectively excluded from all multi-signature workflows on the organization server.

This is a **permanent, targeted denial of service** against a specific user's ability to participate in the organization's core function, achievable by any peer member with no elevated privileges and re-applicable immediately after any admin cleanup.

## Likelihood Explanation
- **Attacker preconditions:** A valid organization account (non-admin). Any invited member qualifies.
- **Key discovery:** Victim public keys are publicly visible on the Hedera mirror node and returned by `GET /user/:id/keys`.
- **Effort:** A single authenticated HTTP POST. No cryptographic capability required.
- **Persistence:** The attack re-applies immediately after any admin soft-delete, making it a sustained DoS with no meaningful barrier.

Likelihood is **high**.

## Recommendation
Implement proof-of-possession before accepting a public key registration. The standard approach is a challenge-response: the server issues a short-lived nonce, the client signs it with the private key corresponding to the submitted public key, and the server verifies the signature before creating the `UserKey` record. This ensures only the holder of the private key can register the corresponding public key, eliminating the pre-emption attack entirely.

Additionally, consider removing `withDeleted: true` from the `findOne` in `uploadKey` or, at minimum, not counting soft-deleted rows owned by a different user as a conflict.
