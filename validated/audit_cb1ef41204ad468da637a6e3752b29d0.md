### Title
`archiveTransaction` Guard Bypassed by `isManual` Flag Allows Overwriting Terminal Transaction States

### Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a logically flawed guard condition that uses `isManual=true` as a complete bypass for the status check. Any authenticated transaction creator who set `isManual: true` at creation time can call `PATCH /transactions/archive/:id` on a transaction in any terminal state — including `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED` — and overwrite it with `ARCHIVED`. This corrupts the transaction audit trail and broadcasts false status notifications to all observers and signers.

### Finding Description
The guard at lines 711–718 of `transactions.service.ts` reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The condition throws only when **both** sub-expressions are true: (1) status is not in the allowed list **AND** (2) `isManual` is `false`. When `isManual=true`, the entire status check short-circuits to `false` and no exception is raised — regardless of the current status.

`isManual` is a user-controlled optional boolean accepted directly from the request body in `CreateTransactionDto`:

```typescript
@IsOptional()
@IsBoolean()
isManual?: boolean;
```

It is stored verbatim on the `Transaction` entity and never reset after execution.

**Exploit path:**
1. Attacker (normal authenticated user) calls `POST /transactions` with `{ ..., isManual: true }` → transaction created with `status = WAITING_FOR_SIGNATURES`, `isManual = true`.
2. Signatures are collected; chain service executes the transaction on the Hedera network and sets `status = EXECUTED`. `isManual` remains `true`.
3. Attacker calls `PATCH /transactions/archive/:id`.
4. Guard evaluates: `!includes(EXECUTED) && !true` → `true && false` → `false` → **no throw**.
5. `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes unconditionally.
6. `emitTransactionStatusUpdate` fires, broadcasting `ARCHIVED` to every observer and signer.

The same path applies to `FAILED`, `EXPIRED`, and `CANCELED` statuses whenever `isManual=true`.

The existing e2e test that asserts "should not archive a transaction if it's already executed" (lines 884–897) only exercises a **non-manual** transaction; it does not cover the `isManual=true` + `EXECUTED` combination, leaving the bypass untested. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
- **Audit trail corruption**: An `EXECUTED` transaction's status is permanently overwritten to `ARCHIVED` in PostgreSQL. The true outcome of the Hedera network operation is erased from the application's record.
- **False notifications**: All observers and signers receive an `ARCHIVED` status-update notification for a transaction that was actually executed, making it impossible to distinguish executed from archived without querying the Hedera mirror node directly.
- **Terminal-state masking**: A `FAILED` transaction can be archived, hiding the failure from all participants. A `CANCELED` transaction can be re-archived, producing duplicate status events.
- **State machine violation**: The `ARCHIVED` status is defined as a terminal history state alongside `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED`. Allowing one terminal state to overwrite another breaks the invariant that terminal states are immutable. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
- Any verified, authenticated user can set `isManual: true` at transaction creation — no privileged role is required.
- The exploit requires exactly two API calls: `POST /transactions` (with `isManual: true`) and `PATCH /transactions/archive/:id` after execution.
- The `getTransactionForCreator` check only verifies ownership, not status eligibility, so the attacker needs only to be the creator of the transaction.
- The test suite does not cover this path, so the bypass has not been caught by automated testing. [6](#0-5) 

### Recommendation
Remove the `isManual` bypass from the guard. The status check must be unconditional:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions legitimately need to be archiveable in additional non-terminal states, those states must be explicitly enumerated rather than bypassing the check entirely. Add unit and e2e test cases covering `isManual=true` with `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED` statuses to assert that archiving is rejected. [7](#0-6) 

### Proof of Concept

**Step 1 — Create a manual transaction:**
```
POST /transactions
Authorization: Bearer <user_token>
{ "name": "poc", "description": "poc", "transactionBytes": "<hex>",
  "signature": "<hex>", "creatorKeyId": <id>, "mirrorNetwork": "testnet",
  "isManual": true }
→ 201 { "id": 123, "status": "WAITING FOR SIGNATURES", "isManual": true }
```

**Step 2 — Collect signatures and trigger execution; chain service sets status to EXECUTED.**

**Step 3 — Archive the executed transaction:**
```
PATCH /transactions/archive/123
Authorization: Bearer <user_token>
→ 200 true
```

**Step 4 — Verify state corruption:**
```
GET /transactions/123
→ { "id": 123, "status": "ARCHIVED" }   ← was EXECUTED
```

All observers and signers receive an `ARCHIVED` notification. The executed transaction no longer appears in the `EXECUTED` history; it is silently reclassified as `ARCHIVED`. [8](#0-7) [9](#0-8)

  /* Archive the transaction if the transaction is sign only. */
  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
