### Title
Unbounded Full-Table Scan with Per-Row External I/O in `getTransactionsToSign` Causes Authenticated DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **every** non-terminal transaction from the database with no `take` limit, then sequentially calls `userKeysToSign` (which invokes `computeSignatureKey` → mirror-node/cache I/O) for each row before applying pagination. A single authenticated API request against a system with many pending transactions causes O(N × M) async work, where N is the number of non-terminal transactions and M is the number of accounts per transaction. This can exhaust server resources and make the service unresponsive.

### Finding Description

**Root cause — no database-level limit on the fetch:**

In `getTransactionsToSign`, the repository query at line 295 omits `take`/`skip`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` or `skip`
});
``` [1](#0-0) 

This loads **all** rows whose status is not in `[EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED]` — i.e., every active transaction in the system — regardless of the `page`/`limit` the caller requested.

**Per-row expensive async work:**

For every transaction returned, the code calls `userKeysToSign`:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign` → `keysRequiredToSign` → `transactionSignatureService.computeSignatureKey`, which makes multiple sequential async calls to the mirror-node cache per transaction (fee-payer key, signing-account keys, receiver-account keys, node keys): [3](#0-2) 

Each of those calls hits `accountCacheService.getAccountInfoForTransaction` or `nodeCacheService.getNodeInfoForTransaction`: [4](#0-3) 

**Pagination applied after full scan:**

The `result.slice(offset, offset + limit)` at line 313 is applied only after the entire loop completes, so the caller's `page=1&size=10` parameter provides zero protection against the full scan: [5](#0-4) 

**Exposed endpoint:**

The endpoint `GET /transactions/sign` is still registered and reachable by any authenticated user (the comment "NO LONGER USED BY FRONT-END" does not remove it): [6](#0-5) 

### Impact Explanation

A single authenticated HTTP request to `GET /transactions/sign` causes the API process to:
1. Load the entire non-terminal transaction table into memory.
2. Deserialize every transaction's bytes (`SDKTransaction.fromBytes`).
3. Perform multiple async I/O calls (mirror-node cache) per transaction.

In an organization with thousands of pending transactions, this single request can consume gigabytes of memory and saturate the event loop for seconds to minutes, making the API unresponsive to all other users. Concurrent calls (even from the same user) compound the effect. The result is **service unavailability** for the entire organization.

### Likelihood Explanation

Any verified, authenticated user can trigger this with a single `GET /transactions/sign` request — no special privileges, no leaked credentials, no coordination required. The number of non-terminal transactions grows naturally with platform usage, so the impact worsens over time without any attacker action. An attacker can also deliberately create many transactions to amplify the effect, since transaction creation is a normal user flow.

### Recommendation

Apply the database-level `take`/`skip` **before** the per-row processing, and push the signing-key check into the database query (e.g., via a subquery or a pre-computed `signerPublicKeys` column) rather than doing it in application code for every row. If the per-row check must remain in application code, enforce a hard cap on the number of rows fetched and document that the endpoint cannot support arbitrary dataset sizes.

### Proof of Concept

1. Authenticate as any verified user and obtain a JWT.
2. Ensure the system has a large number of non-terminal transactions (e.g., 10 000 rows in `WAITING_FOR_SIGNATURES` status).
3. Issue a single request:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <token>
   ```
4. Observe that the server fetches all 10 000 rows, calls `computeSignatureKey` (with mirror-node I/O) for each, and takes O(N) time to respond — regardless of the requested page size of 10.
5. Issue the same request concurrently from multiple tabs/scripts; the API becomes unresponsive for all users.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
