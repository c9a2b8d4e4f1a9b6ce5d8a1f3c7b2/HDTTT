Audit Report

## Title
Global IP Rate Limiter Bypassed via Attacker-Controlled `X-Forwarded-For` Header, Enabling Unlimited Brute-Force on Login

## Summary
`IpThrottlerGuard`, the sole global rate limiter, derives its tracking key from the attacker-supplied `X-Forwarded-For` header with no validation. By rotating spoofed IP values in this header, an unauthenticated attacker can assign each request to a fresh rate-limit bucket, completely nullifying the configured limits. The per-email `EmailThrottlerGuard` on `POST /auth/login` is ordered after `LocalAuthGuard` and never executes on failed credential attempts, leaving the IP throttler as the only protection — which is trivially bypassed.

## Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`:**

The guard unconditionally prefers the client-controlled `x-forwarded-for` header over the socket-level `req.ip`: [1](#0-0) 

The returned value becomes the Redis key used to count requests. No validation, stripping, or allowlisting of this header is performed.

**Global registration — `ApiModule`:**

`IpThrottlerGuard` is registered as `APP_GUARD`, making it the first guard executed on every incoming request: [2](#0-1) 

**Why `EmailThrottlerGuard` does not compensate on `/auth/login`:**

The login route is decorated as:

```typescript
@UseGuards(LocalAuthGuard, EmailThrottlerGuard)
``` [3](#0-2) 

NestJS executes guards in declaration order. `LocalAuthGuard` invokes the Passport local strategy and throws `UnauthorizedException` on wrong credentials — before `EmailThrottlerGuard` ever runs. The email-based throttle therefore only fires on *successful* logins, providing zero protection against failed credential attempts.

**`POST /reset-password` is also affected:**

This endpoint uses only `EmailThrottlerGuard` with no `LocalAuthGuard` before it: [4](#0-3) 

The global IP throttler is the only unauthenticated-path guard here, and it is bypassed the same way.

## Impact Explanation

- **Credential brute-force**: An attacker can submit an unlimited number of password guesses against any known user email. Bcrypt/argon2 hashing is intentionally slow; at scale this also exhausts server CPU.
- **OTP email flooding**: Spoofing `X-Forwarded-For` on `POST /reset-password` bypasses the IP throttler, allowing unlimited OTP emails to any user's inbox.
- **Severity**: High — direct path to account compromise via brute-force; no privileges required.

## Likelihood Explanation

- **Attacker preconditions**: None. The endpoint is public; the `X-Forwarded-For` header requires no special tooling — a single `curl -H` flag suffices.
- **Detectability**: The attack is indistinguishable from normal traffic at the application layer because each request appears to originate from a different IP.
- **Realistic scenario**: Any attacker with a list of registered email addresses can run an automated credential-stuffing or dictionary attack at arbitrary speed.

## Recommendation

1. **Do not trust `X-Forwarded-For` unconditionally.** If the application sits behind a known, trusted reverse proxy, extract only the first IP from the header and validate it against a trusted proxy allowlist. If no trusted proxy is configured, use `req.socket.remoteAddress` (i.e., `req.ip` with `trust proxy` disabled) exclusively.

2. **Reorder guards on `POST /auth/login`.** Place `EmailThrottlerGuard` *before* `LocalAuthGuard` so that the per-email rate limit is enforced regardless of whether credentials are correct:
   ```typescript
   @UseGuards(EmailThrottlerGuard, LocalAuthGuard)
   ```

3. **Add a second, independent throttle on `POST /reset-password`** that is not solely IP-based (e.g., per-email limit already present via `EmailThrottlerGuard`, but ensure the IP throttler cannot be bypassed as described above).

## Proof of Concept

```bash
# Iterate through password candidates, rotating X-Forwarded-For each time
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<target>/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.$((i/256)).$((i%256))" \
    -d '{"email":"victim@example.com","password":"candidate'$i'"}'
done
```

Each request is assigned to a unique Redis bucket derived from the spoofed IP. No bucket ever reaches the configured limit. `LocalAuthGuard` performs a full bcrypt/argon2 comparison on every attempt with no server-side enforcement stopping the loop.
