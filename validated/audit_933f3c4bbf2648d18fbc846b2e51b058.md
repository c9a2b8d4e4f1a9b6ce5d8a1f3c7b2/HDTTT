### Title
Sequential Transaction Group Ordering Not Enforced at Execution Time

### Summary

The `executeTransactionGroup` function silently drops group members that are not yet in `WAITING_FOR_EXECUTION` status before submitting the remaining ones. For sequential groups this means a later transaction can be submitted to the Hedera network before an earlier, still-unsigned transaction, breaking the causal dependency the group was designed to enforce.

### Finding Description

**Vulnerability class:** State-transition ordering (direct analog of the external report's message-ordering issue).

**Root cause — `execute.service.ts` lines 65-67:**

```typescript
transactionGroup.groupItems = transactionGroup.groupItems.filter(
  tx => tx.transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
);
``` [1](#0-0) 

After this filter the function iterates the survivors in `validStart ASC` order and, for `sequential` groups, awaits each one before starting the next:

```typescript
if (transactionGroup.sequential) {
  for (const { sdkTransaction, transaction } of transactions) {
    const delay = transaction.validStart.getTime() - Date.now();
    await sleep(delay);
    rawResults.push(await this._executeTransaction(transaction, sdkTransaction));
  }
}
``` [2](#0-1) 

The sequential loop is correct *among the survivors*, but there is no guard that verifies every preceding item (by `seq` or `validStart`) is also `WAITING_FOR_EXECUTION` before any item is submitted. If item N-1 is still `WAITING_FOR_SIGNATURES` it is silently removed, and item N is executed alone.

**Trigger path — `transaction-scheduler.service.ts` `prepareTransactions`:**

The scheduler fires `collateGroupAndExecute` as soon as it finds *any* group member that is `WAITING_FOR_EXECUTION` and whose `validStart` is executable:

```typescript
if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
  if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
    if (!processedGroupIds.has(transaction.groupItem.groupId)) {
      processedGroupIds.add(transaction.groupItem.groupId);
      ...
      this.collateGroupAndExecute(transactionGroup);
    }
  }
``` [3](#0-2) 

There is no check that *all* members of the sequential group have reached `WAITING_FOR_EXECUTION` before the group is handed to `collateGroupAndExecute` / `executeTransactionGroup`.

**Concrete scenario:**

| Item | seq | validStart | Signers required | Status when scheduler fires |
|------|-----|-----------|------------------|-----------------------------|
| Tx A | 1 | T | Alice + Bob | `WAITING_FOR_SIGNATURES` (Bob hasn't signed) |
| Tx B | 2 | T+1 s | Alice only | `WAITING_FOR_EXECUTION` |

1. Alice signs both; Bob has not yet signed Tx A.
2. The every-10-second cron finds Tx B in `WAITING_FOR_EXECUTION` with an executable `validStart`.
3. `prepareTransactions` triggers `collateGroupAndExecute` for the group.
4. `executeTransactionGroup` filters Tx A out (still `WAITING_FOR_SIGNATURES`).
5. Tx B is submitted to Hedera **before** Tx A.
6. If Tx B depends on the result of Tx A (e.g., a transfer to an account that Tx A was supposed to create), Tx B fails on-chain.

### Impact Explanation

Any sequential transaction group whose members have heterogeneous signature requirements can have its ordering guarantee silently violated. Dependent operations — account creation followed by a transfer, file creation followed by a file append, node creation followed by node update — will fail on-chain with no clear error surfaced to the user, because the scheduler does not report the ordering violation; it simply executes whatever is ready.

### Likelihood Explanation

Organization mode is explicitly designed for multi-signer workflows where different transactions in a group can have different threshold-key requirements. A group creator routinely assigns different signers to different items. The 10-second polling interval means the window is narrow but recurring; any time signatures arrive for a later item before an earlier item is fully signed, the race is entered. This is a normal operational pattern, not an edge case.

### Recommendation

In `executeTransactionGroup`, before executing any item of a sequential group, verify that **all** items are in `WAITING_FOR_EXECUTION`. If any predecessor is not ready, abort execution for the entire group and let the scheduler retry on the next cycle:

```typescript
if (transactionGroup.sequential) {
  const allReady = transactionGroup.groupItems.every(
    gi => gi.transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
  );
  if (!allReady) {
    this.logger.warn(`Sequential group ${transactionGroup.id} skipped: not all items are ready.`);
    return { transactions: [] };
  }
}
```

Alternatively, enforce the check in `prepareTransactions` before calling `collateGroupAndExecute`, so the group is never handed to the executor until every member has reached `WAITING_FOR_EXECUTION`.

### Proof of Concept

1. Create an organization sequential group with two transactions:
   - Tx A (`seq=1`): requires two signers (e.g., threshold 2-of-2).
   - Tx B (`seq=2`): requires one signer (threshold 1-of-1).
2. Have signer 1 sign both Tx A and Tx B. Do **not** have signer 2 sign Tx A yet.
3. Wait for the 10-second cron (`handleTransactionsBetweenNowAndAfterThreeMinutes`) to fire after the `validStart` of Tx B passes.
4. Observe that `processTransactionStatus` promotes Tx B to `WAITING_FOR_EXECUTION` while Tx A remains `WAITING_FOR_SIGNATURES`.
5. `prepareTransactions` detects Tx B as executable and calls `collateGroupAndExecute`.
6. `executeTransactionGroup` filters Tx A out and submits Tx B to Hedera.
7. If Tx B depends on Tx A (e.g., a transfer to an account Tx A was to create), the Hedera network returns `INVALID_ACCOUNT_ID` or equivalent, and Tx B is marked `FAILED`.
8. Signer 2 later signs Tx A; Tx A is submitted and succeeds — but Tx B has already failed permanently. [4](#0-3) [5](#0-4)

```typescript
  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },
            });
            // All the transactions for the group are now pulled. If there is an issue validating for even one
            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);
        }
      }
    }
  }
