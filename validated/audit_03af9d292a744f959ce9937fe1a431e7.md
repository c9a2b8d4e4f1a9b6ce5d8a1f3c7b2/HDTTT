### Title
Authorization Bypass via Missing Key-Ownership Check in `importSignatures` — Parallel Signature-Upload Path Omits the Enforcement Present in the Primary Path

### Summary

The codebase exposes two parallel HTTP endpoints for uploading signatures onto a stored transaction. The primary path (`POST /transactions/:id/signers`) enforces that every public key in the submitted signature map belongs to the authenticated user. The secondary path (`POST /transactions/signatures/import`) performs only a cryptographic validity check and omits the key-ownership guard entirely. Any authenticated user who has read access to a transaction can therefore inject cryptographically valid signatures from keys they do not own, advancing the transaction's signing state without the key owners' explicit participation through the normal workflow.

### Finding Description

**Primary path — key ownership enforced**

`SignersController.uploadSignatureMap` → `SignersService.uploadSignatureMaps` → `processTransactionSignatures`:

```
// signers.service.ts lines 244-248
let userKey = userKeyMap.get(raw);
if (!userKey) {
  userKey = userKeyMap.get(publicKey.toStringDer());
}
if (!userKey) throw new Error(ErrorCodes.PNY);   // ← ownership enforced
```

`userKeyMap` is built exclusively from `user.keys` (the authenticated user's registered keys), so any public key not owned by the caller is rejected. [1](#0-0) [2](#0-1) 

**Secondary path — key ownership absent**

`TransactionsController.importSignatures` → `TransactionsService.importSignatures`:

```
// transactions.service.ts lines 545-553
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),   // ← only crypto check
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);

for (const publicKey of publicKeys) {
  sdkTransaction.addSignature(publicKey, map);          // ← no ownership check
}
```

`validateSignature` verifies that the supplied bytes are a valid signature over the transaction body for the given public key. It does not verify that the public key is registered to the calling user. Any key whose signature is cryptographically valid is accepted and written into the stored transaction bytes. [3](#0-2) 

The endpoint is publicly reachable by any authenticated, verified user: [4](#0-3) 

The only gate before the signature injection is `verifyAccess`, which grants entry to creators, signers, observers, and approvers — a much broader population than "users who own the keys being submitted." [5](#0-4) 

**Root cause — structural mismatch between two implementations of the same operation**

This is the direct analog of the external report: two code paths implement "upload signatures for a transaction," but only one carries the critical authorization check. The omission in `importSignatures` is not compensated by any other guard.

### Impact Explanation

An attacker who is an observer (or any other role that passes `verifyAccess`) on a transaction can:

1. Obtain a valid signature over the transaction body for a key they do not own (e.g., from a `.tx2` export file shared over email or a shared folder, or from any other out-of-band channel).
2. Call `POST /transactions/signatures/import` with that signature map.
3. The backend writes the foreign signature into the stored `transactionBytes` without creating a `TransactionSigner` record, so the injection is invisible in the signing audit trail.
4. If the injected signature satisfies the remaining threshold, the transaction advances to `WAITING_FOR_EXECUTION` and can be submitted to the Hedera network without the key owner having explicitly approved through the normal workflow.

The result is unauthorized state advancement of a multi-party transaction and circumvention of the signing-approval workflow that the platform is designed to enforce. [6](#0-5) 

### Likelihood Explanation

- The attacker must be an authenticated, verified user with `verifyAccess` on the target transaction (observer role qualifies).
- Valid signature bytes for a foreign key are obtainable without breaking cryptography: the `.tx2` export/import workflow is a first-class feature of the application, and exported files contain raw signatures.
- The endpoint requires no elevated privilege and is reachable from the standard frontend client or any HTTP client.
- No rate-limiting or additional confirmation step is applied to `importSignatures`.

### Recommendation

Add the same key-ownership check to `importSignatures` that exists in `processTransactionSignatures`. Before calling `sdkTransaction.addSignature`, verify that each public key in the submitted map is registered to the authenticated user:

```typescript
// In importSignatures, after validateSignature succeeds:
await attachKeys(user, this.entityManager);
const userKeySet = new Set(user.keys.map(k => k.publicKey));

for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  if (!userKeySet.has(raw) && !userKeySet.has(publicKey.toStringDer())) {
    throw new BadRequestException(ErrorCodes.PNY);
  }
  sdkTransaction.addSignature(publicKey, map);
}
```

Alternatively, consolidate both paths into a single shared service method that always enforces ownership, eliminating the structural duplication that caused the divergence.

### Proof of Concept

**Preconditions:**
- Alice (attacker) is an observer on transaction ID `42`, which requires Bob's signature.
- Alice has obtained Bob's exported `.tx2` file (shared via email for offline signing).

**Steps:**

1. Alice parses the `.tx2` file to extract Bob's `SignatureMap`.
2. Alice formats the map and calls:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <alice_jwt>
   Content-Type: application/json

   [{ "id": 42, "signatureMap": { "<nodeId>": { "<txId>": { "<bob_pubkey>": "<bob_sig_hex>" } } } }]
   ```
3. The server calls `validateSignature` — Bob's signature is cryptographically valid → passes.
4. `verifyAccess(transaction, alice)` → Alice is an observer → passes.
5. No key-ownership check is performed.
6. `sdkTransaction.addSignature(bobPublicKey, map)` executes; the updated bytes (now containing Bob's signature) are written to the database.
7. No `TransactionSigner` record is created for Bob; the audit trail shows no sign of the injection.
8. If this was the last required signature, the transaction status advances and can be executed — without Bob having signed through the normal workflow.

**Expected (correct) behavior:** Step 5 should throw `ErrorCodes.PNY` because Bob's public key is not in Alice's registered key set. [7](#0-6) [8](#0-7)

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
