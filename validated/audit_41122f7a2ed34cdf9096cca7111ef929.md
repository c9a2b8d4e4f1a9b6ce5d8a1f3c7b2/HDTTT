Audit Report

## Title
Any Verified User Can Create and Read Comments on Transactions They Have No Access To

## Summary
`CommentsController` in `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only authentication guards at the class level and performs no transaction-level authorization. Any verified user can `POST` comments to any transaction by ID and `GET` all comments for any transaction, regardless of whether they are a creator, signer, approver, or observer of that transaction. The developer explicitly acknowledged the missing control with a `TODO` comment in the source.

## Finding Description

**Root cause:** The controller applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — these verify identity, not transaction membership. [1](#0-0) 

The `createComment` handler accepts any `transactionId` from the URL and delegates directly to the service with no ownership or membership check. The developer's own `TODO` on line 22 explicitly states: *"need some sort of guard or check to ensure user can comment here"*, confirming the control is intentionally absent and not yet implemented. [2](#0-1) 

The two read endpoints similarly perform no user-to-transaction access verification: [3](#0-2) 

At the service layer, `createComment` blindly saves the comment, `getTransactionComments` queries only by `transactionId`, and `getTransactionCommentById` queries only by `id` — no user filter at any level: [4](#0-3) 

**Contrast with correct pattern:** Every other transaction-scoped endpoint calls `getTransactionWithVerifiedAccess`, which enforces creator/signer/approver/observer membership and throws `UnauthorizedException` on failure: [5](#0-4) [6](#0-5) 

## Impact Explanation

**Unauthorized state mutation:** A verified attacker can `POST /transactions/{anyId}/comments` with an arbitrary `transactionId` and inject comments into transactions they have no legitimate relationship with. This pollutes the comment history of sensitive multi-signature transactions and can be used to mislead or harass legitimate participants.

**Cross-tenant data exposure:** `GET /transactions/{anyId}/comments` and `GET /transactions/{anyId}/comments/{commentId}` return comment content for any transaction. Comments may contain sensitive coordination details (key references, signing instructions, business context) that should be visible only to transaction participants.

## Likelihood Explanation

Exploitation requires only a valid, verified user account — the baseline attacker profile per `RESEARCHER.md`. No admin credentials, leaked secrets, or privileged access are needed. The attacker registers, gets verified, and issues HTTP requests with sequential or known `transactionId` values. The attack is trivially scriptable and leaves no obvious trace distinguishing it from legitimate comment activity.

## Recommendation

In `createComment`, `getComments`, and `getCommentById`, verify that the requesting user is a creator, signer, approver, or observer of the target transaction before proceeding. The existing `TransactionsService.getTransactionWithVerifiedAccess` (or `verifyAccess`) already implements this logic and should be reused. Inject `TransactionsService` into `CommentsService` (or `CommentsController`) and call it before any comment operation, throwing `UnauthorizedException` if access is denied.

## Proof of Concept

1. Register two accounts: `attacker@example.com` and `victim@example.com`. Get both verified.
2. As `victim`, create a transaction (returns `transactionId = 42`). The attacker has no relationship to this transaction.
3. As `attacker`, obtain a valid JWT via login.
4. Issue:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   Content-Type: application/json

   { "message": "Injected comment" }
   ```
   **Expected (correct):** `401 Unauthorized`
   **Actual:** `201 Created` — comment is persisted against transaction 42.
5. Issue:
   ```
   GET /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   ```
   **Expected (correct):** `401 Unauthorized`
   **Actual:** `200 OK` — all comments for transaction 42 are returned, including any sensitive coordination content left by legitimate participants.

```typescript
      .createQueryBuilder('comment')
      .where('comment.transactionId = :transactionId', { transactionId })
```

```typescript
  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
```
