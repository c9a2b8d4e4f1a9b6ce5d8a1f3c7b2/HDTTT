Audit Report

## Title
Race Condition in `approveTransaction` Allows Revoked Approver to Cast a Binding Vote Before Soft-Delete Commits

## Summary
`approveTransaction` in `approvers.service.ts` performs a non-atomic read-check-write sequence. A malicious approver who detects that their removal is in flight can submit an approval that passes all in-memory checks and is written to a still-live DB record before the soft-delete commits. If that approval satisfies the threshold, the transaction advances to `WAITING_FOR_EXECUTION` and the chain scheduler registers an execution timeout that is never cancelled by the subsequent removal event.

## Finding Description

**Vulnerability class:** TOCTOU / race condition in permission revocation.

**Phase 1 — Read (outside any DB transaction):**
`approveTransaction` calls `getVerifiedApproversByTransactionId` at line 553, which delegates to `getApproversByTransactionId`. That function issues a raw recursive CTE filtered by `deletedAt IS NULL`: [1](#0-0) 

This read is not wrapped in any DB transaction, so it represents a point-in-time snapshot.

**Phase 2 — Check (in-memory, on the Phase 1 snapshot):** [2](#0-1) 

**Phase 3 — Write (inside a DB transaction, but without re-checking `deletedAt`):** [3](#0-2) 

`whereInIds` generates `WHERE id IN (...)` with no `deletedAt IS NULL` guard. If the soft-delete committed between Phase 1 and Phase 3, the approval is written to the now-soft-deleted row. If Phase 3 commits *before* the soft-delete, the approval lands on a live row.

**Controller-level non-atomicity in the removal path:** [4](#0-3) 

`getCreatorsTransaction` (auth check) and `removeTransactionApprover` (soft-delete) are two separate `await` calls with no locking between them, widening the race window.

**Service-level removal:** [5](#0-4) 

**Critical sub-case — approval written before soft-delete:**
If User B's Phase 3 write commits before Alice's `removeNode` call, the approval record has `deletedAt = NULL` and `approved = true`. The `emitTransactionStatusUpdate` at line 615 fires: [6](#0-5) 

The chain scheduler's `processTransactionStatus` evaluates the threshold against live approvers (those with `deletedAt IS NULL`). If the threshold is met, the transaction advances to `WAITING_FOR_EXECUTION` and `collateAndExecute` registers a named `setTimeout`: [7](#0-6) 

Alice's subsequent soft-delete fires a second `emitTransactionStatusUpdate` (line 541), but the named timeout is already registered. The guard at line 274 (`if (this.schedulerRegistry.doesExist('timeout', name)) return`) prevents a second registration but does **not** cancel the existing one. The execution timeout fires at `validStart + 5 s`: [8](#0-7) 

## Impact Explanation

A malicious approver can cast a threshold-completing vote in the window between the creator initiating their removal and the soft-delete committing. If the approval lands on a live record and satisfies the threshold, the transaction advances to `WAITING_FOR_EXECUTION`. The chain scheduler's execution timeout is registered immediately and is not cancelled by the subsequent removal event, meaning the transaction can be submitted to the Hedera network with the revoked approver's vote intact. The result is an unauthorized transaction execution approved by a party the creator explicitly revoked.

## Likelihood Explanation

The attacker must be a legitimate approver — a normal user role. The race window is narrow (milliseconds between two DB operations), but it is deterministically exploitable: the attacker can monitor the NATS notification stream (`emitTransactionUpdate` / `emitTransactionStatusUpdate`) or poll `GET /transactions/:id/approvers` to detect when a removal is in flight, then immediately submit `POST /transactions/:id/approvers/approve`. No cryptographic break or elevated privilege is required. The attack is a standard TOCTOU against a REST API with no request serialization or optimistic locking.

## Recommendation

1. **Re-check `deletedAt` inside the write transaction.** Replace the bare `whereInIds` at line 608 with a query that adds `AND "deletedAt" IS NULL`, so an approval targeting a concurrently soft-deleted row is a no-op:
   ```ts
   .whereInIds(userApprovers.map(a => a.id))
   .andWhere('"deletedAt" IS NULL')
   ```
2. **Atomize the removal controller.** Wrap `getCreatorsTransaction` and `removeTransactionApprover` in a single `dataSource.transaction(...)` call so the auth check and the soft-delete are serializable.
3. **Cancel the execution timeout on status downgrade.** When `processTransactionStatus` downgrades a transaction from `WAITING_FOR_EXECUTION` back to `WAITING_FOR_SIGNATURES`, the scheduler should attempt to delete the named timeout (`smart_collate_timeout_{id}` / `execution_timeout_{id}`) if it exists.

## Proof of Concept

```
1. Alice creates transaction T with approver threshold = 1, adds User B as the sole approver.
2. Alice sends DELETE /transactions/T/approvers/{B_approver_id}.
   → Controller calls getCreatorsTransaction (auth check) — completes.
   → Controller is about to call removeTransactionApprover (soft-delete).
3. User B, monitoring the NATS stream, detects the pending removal and immediately sends
   POST /transactions/T/approvers/approve with a valid signature.
4. User B's request reaches Phase 1 (line 553) BEFORE the soft-delete commits.
   → getApproversByTransactionId returns User B (deletedAt IS NULL).
5. User B's Phase 2 checks pass (lines 559–563): user is an approver, no prior signature.
6. Alice's removeNode commits → B's record gets deletedAt = now().
7. User B's Phase 3 write (lines 599–610) executes whereInIds([B_id]) with no deletedAt guard
   → approval is written to the now-soft-deleted record.
   [OR, if Phase 3 commits before step 6: approval lands on a live record, which is the
    critical sub-case that advances the transaction to WAITING_FOR_EXECUTION.]
8. emitTransactionStatusUpdate fires (line 615).
9. Chain scheduler evaluates threshold → transaction advances to WAITING_FOR_EXECUTION.
10. collateAndExecute registers smart_collate_timeout_T.
11. Alice's removal fires a second emitTransactionStatusUpdate (line 541).
12. Chain scheduler re-evaluates, but smart_collate_timeout_T already exists (line 274 guard).
13. Timeout fires at validStart + 5 s → executeService.executeTransaction(T) submits to Hedera.
```

  addExecutionTimeout(transaction: Transaction) {
    const name = `execution_timeout_${transaction.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    if (transaction.isManual) return;

    const timeToValidStart = transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransaction(transaction);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }
