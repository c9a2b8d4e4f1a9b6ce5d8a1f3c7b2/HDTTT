Audit Report

## Title
TOCTOU State Transition in `archiveTransaction` Allows Creator to Overwrite Terminal Transaction Status

## Summary
`archiveTransaction` in `transactions.service.ts` performs a non-atomic read-check-write sequence. The status check at read time is not re-validated at write time, allowing the chain service's scheduler to transition a transaction to `EXECUTED` between the two operations. The creator's subsequent unconditional `UPDATE` then overwrites `EXECUTED` with `ARCHIVED`, corrupting the system's record of a transaction already submitted to the Hedera network.

## Finding Description

**Root cause — `archiveTransaction` (lines 708–733):**

The function reads the transaction, validates the status, then issues an unconditional update:

```
Step 1 (READ):  transaction = await this.getTransactionForCreator(id, user);   // line 709
Step 2 (CHECK): if status NOT IN [WFS, WFE] AND NOT isManual → throw           // lines 711–718
Step 3 (WRITE): await this.repo.update({ id }, { status: ARCHIVED });          // line 720
``` [1](#0-0) 

The `UPDATE` at step 3 uses only `WHERE id = :id`. There is no `AND status IN (...)` guard. Any status change occurring between step 1 and step 3 is silently overwritten.

**Contrast with the correctly-guarded `cancelTransactionWithOutcome` (lines 673–679):**

`cancelTransaction` uses `.andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })`, so a concurrent status change causes the UPDATE to affect 0 rows, which is then detected and handled with a re-read and appropriate error. [2](#0-1) 

**Race window — chain service scheduler:**

The chain service's `handleTransactionsBetweenNowAndAfterThreeMinutes` cron runs every 10 seconds and calls `updateTransactions`, which calls `processTransactionStatus` to transition `WAITING_FOR_EXECUTION` transactions to `EXECUTED`. [3](#0-2) 

The `handleExpiredTransactions` cron also runs every 10 seconds and uses a conditional `WHERE status IN (...)` guard, confirming the project's own pattern for safe concurrent updates — a pattern `archiveTransaction` omits. [4](#0-3) 

**Additional bypass via `isManual` flag:**

The status check logic (`!transaction.isManual` in the guard condition) means that if `isManual` is `true`, the status check is bypassed entirely — a transaction already in `EXECUTED` status could be archived without any race at all. This compounds the severity. [5](#0-4) 

## Impact Explanation

A transaction creator can cause the database to record `ARCHIVED` for a transaction that was already submitted to and executed on the Hedera network:

- **Integrity failure**: The system's state diverges from the Hedera ledger. The transaction was executed on-chain but the organization's tool shows it as archived.
- **Deception of co-participants**: Signers, observers, and approvers who rely on the tool to track transaction outcomes see `ARCHIVED` instead of `EXECUTED`, hiding the fact that the transaction was executed.
- **Audit trail corruption**: Historical records no longer accurately reflect what happened on-chain.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The race window is the time between `getTransactionForCreator` (line 709) and `repo.update` (line 720). The chain service scheduler runs every 10 seconds. A creator who monitors when their transaction reaches `WAITING_FOR_EXECUTION` and immediately calls `PATCH /transactions/archive/:id` has a realistic chance of winning the race, especially under load. No special tooling is required beyond a standard HTTP client. The `isManual` bypass requires no race at all.

## Recommendation

Apply the same conditional-WHERE pattern used in `cancelTransactionWithOutcome` to `archiveTransaction`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  // Re-read and throw appropriate error
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Additionally, the `isManual` bypass in the status guard (lines 711–718) should be reviewed to ensure it cannot be used to archive a transaction that is already in a terminal state (`EXECUTED`, `FAILED`, etc.).

## Proof of Concept

1. Creator submits a transaction; it reaches `WAITING_FOR_EXECUTION` status.
2. Creator monitors the transaction status (via polling the API).
3. The chain service scheduler (`handleTransactionsBetweenNowAndAfterThreeMinutes`, every 10 seconds) picks up the transaction and begins executing it on Hedera — transitioning status to `EXECUTED`.
4. Between step 1 (the `getTransactionForCreator` read at line 709, which sees `WAITING_FOR_EXECUTION`) and step 3 (the `repo.update` at line 720), the creator sends `PATCH /transactions/archive/:id`.
5. The status check at lines 711–718 passes (status was `WAITING_FOR_EXECUTION` at read time).
6. `repo.update({ id }, { status: ARCHIVED })` executes with no status guard, overwriting `EXECUTED` with `ARCHIVED`.
7. The database now records `ARCHIVED` for a transaction that was submitted to and executed on the Hedera network. [6](#0-5) [7](#0-6)

  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }

      .createQueryBuilder()
      .update(Transaction)
      .set({ status: TransactionStatus.EXPIRED })
      .where('status IN (:...statuses) AND validStart < :before', {
        statuses: [
          TransactionStatus.NEW,
          TransactionStatus.REJECTED,
          TransactionStatus.WAITING_FOR_EXECUTION,
          TransactionStatus.WAITING_FOR_SIGNATURES,
        ],
        before: this.getThreeMinutesBefore(),
      })
      .returning(['id'])
      .execute();

    if (result.raw.length > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        result.raw.map(t => ({
          entityId: t.id,
        })),
      );
    }
  }

  async updateTransactions(from: Date, to?: Date) {
    //Get the transaction, creatorKey, groupItem, and group. We need the group info upfront
    //in order to determine if the group needs to be processed together
    const transactions = await this.transactionRepo.find({
      where: {
        status: In([
          TransactionStatus.WAITING_FOR_SIGNATURES,
          TransactionStatus.WAITING_FOR_EXECUTION,
        ]),
        validStart: to ? Between(from, to) : MoreThan(from),
      },
      relations: {
        creatorKey: true,
        groupItem: {
          group: true,
        },
      },
      order: {
        validStart: 'ASC',
      },
    });

    const results = await processTransactionStatus(this.transactionRepo, this.transactionSignatureService, transactions);

    if (results.size > 0) {
      const events = Array.from(results.keys(), id => ({ entityId: id }));
      emitTransactionStatusUpdate(this.notificationsPublisher, events);
    }

    return transactions;
  }
