Audit Report

## Title
Any Authenticated User Can Post Comments on Any Transaction Due to Missing Access Control

## Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` applies only authentication guards and performs no ownership or membership check on the target transaction. Any verified, authenticated user can inject comments into any transaction in the system, regardless of their relationship to it. The developer explicitly acknowledged this gap with a `TODO` comment in the source.

## Finding Description
In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the `createComment` handler is protected at the class level by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` only. [1](#0-0) 

These guards verify only that the caller holds a valid, non-blacklisted JWT and has a verified account. The `createComment` handler itself carries an explicit developer acknowledgment of the missing check: [2](#0-1) 

The service layer (`CommentsService.createComment`) performs no further authorization — it directly creates and persists the comment with the supplied `transactionId` and authenticated user, with no check that the user is a creator, approver, observer, or signer of that transaction: [3](#0-2) 

By contrast, every other sensitive write endpoint in the same module enforces ownership or membership. `ApproversService.createTransactionApprovers` calls `getCreatorsTransaction(transactionId, user)` as its first action, which throws `UnauthorizedException` if the caller is not the transaction creator: [4](#0-3) [5](#0-4) 

`ObserversService.createTransactionObservers` similarly receives the `user` parameter for a membership check: [6](#0-5) 

The comments endpoint skips this pattern entirely.

## Impact Explanation
- **Unauthorized write access**: Any authenticated user can inject comments into transactions they have no business relationship with, constituting a cross-tenant integrity violation.
- **Workflow disruption**: Malicious or misleading comments can be injected into the approval/signing workflow of any transaction, potentially confusing approvers or signers.
- **Spam/harassment**: An attacker can flood any transaction's comment thread, degrading the usability of the collaboration layer.
- **Severity: Medium** — no direct asset theft, but unauthorized state mutation across organizational transaction boundaries.

## Likelihood Explanation
- **Preconditions**: Only a valid, verified account is required — the lowest privilege level in the system.
- **Exploit complexity**: Trivial. The attacker needs only a valid JWT and any transaction ID. Transaction IDs are sequential integers and are also exposed through `GET /transactions`.
- **No endpoint-specific rate limiting** beyond the global throttler.
- **Likelihood: High** — reachable by any registered user with zero additional privilege.

## Recommendation
Add a membership/ownership check in `CommentsService.createComment` (or via a dedicated guard) before persisting the comment. The check should verify that the authenticated user is one of: the transaction creator, an approver, an observer, or a signer of the target transaction. The existing `getCreatorsTransaction` pattern in `ApproversService` and the membership check pattern in `ObserversService` can serve as direct references for implementing this. [5](#0-4) 

## Proof of Concept

```
# 1. Register and verify two accounts: victim_user and attacker_user
# 2. victim_user creates a transaction → receives transactionId (e.g., 42)
# 3. attacker_user authenticates and obtains a JWT
# 4. attacker_user posts a comment on victim_user's transaction:

POST /transactions/42/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "message": "Injected comment by unauthorized user" }

# 5. Response: 201 Created — comment is persisted with no error.
# 6. The comment now appears in victim_user's transaction comment thread.
```

The endpoint at `back-end/apps/api/src/transactions/comments/comments.controller.ts` line 21–29 accepts the request, and `CommentsService.createComment` at lines 15–24 saves it without any authorization check. [2](#0-1) [3](#0-2)

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
