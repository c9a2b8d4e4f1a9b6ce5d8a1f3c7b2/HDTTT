## Audit Report

## Title
Missing Bounds Validation on `reminderMillisecondsBefore` Causes DB/API State Inconsistency via Invalid Redis `PXAT` Argument

## Summary
`CreateTransactionDto.reminderMillisecondsBefore` accepts any finite number without a minimum or maximum bound. Supplying a value such as `1e308` causes the arithmetic `tx.validStart.getTime() - dto.reminderMillisecondsBefore` to produce `-Infinity`, yielding an Invalid Date whose `.getTime()` is `NaN`. The `SchedulerService.addReminder` call passes `NaN` to Redis as a `PXAT` argument, which Redis rejects with an error. Because the database transaction commits before the reminder is scheduled, the transaction record is permanently persisted while the API returns HTTP 400 to the caller. The duplicate-transaction guard then prevents any retry, leaving the transaction in an orphaned `WAITING_FOR_SIGNATURES` state.

## Finding Description

**Root cause — no bounds on `reminderMillisecondsBefore`:**

`CreateTransactionDto` declares the field with only `@IsOptional()` and `@IsNumber()`: [1](#0-0) 

`@IsNumber()` from `class-validator` accepts any finite JavaScript number (including `1e308`). No `@Min()` or `@Max()` decorator is present.

**Arithmetic overflow to `-Infinity`:**

In `createTransactions`, after the DB commit, the reminder timestamp is computed as: [2](#0-1) 

`tx.validStart.getTime()` is a normal Unix millisecond timestamp (~`1.7e12`). Subtracting `1e308` produces `-Infinity`. `new Date(-Infinity)` is an Invalid Date; calling `.getTime()` on it returns `NaN`.

**Redis rejects `NaN` as `PXAT`:**

`SchedulerService.addReminder` passes the result directly to Redis: [3](#0-2) 

ioredis serialises `NaN` to the string `"NaN"` before sending. Redis responds with `ERR value is not an integer or out of range`, causing the promise to reject.

**DB commits before reminder scheduling:**

The `entityManager.transaction` block (lines 436–462) commits the record with status `WAITING_FOR_SIGNATURES` before `Promise.all(reminderPromises)` is awaited: [4](#0-3) [5](#0-4) 

**Error is caught and re-thrown as HTTP 400:**

The outer catch block wraps the Redis error as a `BadRequestException`: [6](#0-5) 

**Duplicate guard blocks retry:**

On a subsequent attempt with the same transaction bytes, the guard finds the already-saved record (status `WAITING_FOR_SIGNATURES` is not in the excluded set `[CANCELED, REJECTED, ARCHIVED]`) and throws: [7](#0-6) 

## Impact Explanation
An authenticated user who submits `reminderMillisecondsBefore: 1e308` will have their transaction permanently committed to the database while receiving an HTTP 400 response. The transaction is inaccessible for re-submission via the normal creation flow. The record sits in `WAITING_FOR_SIGNATURES` indefinitely, consuming storage and potentially triggering downstream logic (e.g., notification pipelines, status-update events) without the user's awareness.

## Likelihood Explanation
Any authenticated user can trigger this by crafting a single API request. No special privileges, race conditions, or external dependencies are required. The value `1e308` is a valid JSON number and passes all existing DTO validation. The attack is trivially reproducible and requires no prior knowledge beyond the API schema.

## Recommendation
Add a `@Min(1)` and a reasonable `@Max` (e.g., one week in milliseconds: `604_800_000`) to `reminderMillisecondsBefore` in `CreateTransactionDto`:

```typescript
@IsOptional()
@IsNumber()
@Min(1)
@Max(604_800_000)
reminderMillisecondsBefore?: number;
```

Additionally, add a guard in `createTransactions` before calling `addReminder` to validate that `remindAt` is a valid, future `Date` (i.e., `!isNaN(remindAt.getTime())`), and consider moving the reminder scheduling inside the DB transaction or using a compensating rollback if scheduling fails.

## Proof of Concept

```http
POST /transactions HTTP/1.1
Content-Type: application/json
Authorization: Bearer <valid_token>

{
  "name": "Test",
  "description": "PoC",
  "transactionBytes": "<valid_hex>",
  "signature": "<valid_hex>",
  "creatorKeyId": 1,
  "mirrorNetwork": "testnet",
  "reminderMillisecondsBefore": 1e308
}
```

**Expected (correct) behaviour:** Validation rejects the request before any DB write.

**Actual behaviour:**
1. `@IsNumber()` passes — `1e308` is a finite JS number.
2. DB transaction commits; record saved with status `WAITING_FOR_SIGNATURES`.
3. `new Date(validStart.getTime() - 1e308)` → `new Date(-Infinity)` → Invalid Date.
4. `invalidDate.getTime()` → `NaN` → passed to `pubClient.set(..., 'PXAT', NaN)`.
5. Redis returns `ERR value is not an integer or out of range`.
6. Outer catch re-throws as `BadRequestException` → HTTP 400 returned to caller.
7. Retry attempt blocked: `Transactions already exist: <transactionId>`.
