Audit Report

## Title
IDOR in `removeTransactionApprover`: Creator Check on Wrong Resource Allows Any Approver Deletion

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user is the creator of `:transactionId`, then deletes the approver identified by `:id` with no cross-check that `:id` actually belongs to `:transactionId`. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system.

## Finding Description
In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two sequential, unrelated operations: [1](#0-0) 

Line 108 validates that the caller is the creator of `:transactionId`. Line 109 deletes the approver identified by `:id`. There is no check that `:id` belongs to `:transactionId`. The service method `removeTransactionApprover` accepts only the approver `id` and performs no ownership or transaction-membership validation: [2](#0-1) 

The `updateTransactionApprover` service method demonstrates the correct pattern — it fetches the root node of the approver and explicitly verifies that `rootNode.transactionId === transactionId` before proceeding: [3](#0-2) 

`removeTransactionApprover` has no equivalent guard, making the authorization check entirely bypassable.

## Impact Explanation
An attacker who is the creator of any transaction can:
1. Delete any approver record from any other user's transaction by supplying their own `transactionId` and a victim's approver `id`.
2. Collapse the threshold structure of a pending multi-signature transaction, potentially allowing it to advance to execution with fewer approvals than intended.
3. Silently corrupt another organization member's transaction workflow without any indication to the victim.

This directly undermines the multi-signature trust model, which is the primary security invariant of the system.

## Likelihood Explanation
- The attacker only needs a valid authenticated session and to be the creator of any one transaction — a normal user capability.
- Approver IDs are sequential auto-increment integers in PostgreSQL (`id` column on `transaction_approver`), making enumeration trivial via a simple loop.
- No special privileges, leaked credentials, or physical access are required.
- The attack is a single HTTP DELETE request.

## Recommendation
In the `removeTransactionApprover` controller handler, add a cross-check between the approver's root transaction and the URL `:transactionId` parameter before deletion, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
// After fetching the approver, resolve its root node:
const rootNode = await this.approversService.getRootNodeFromNode(id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
// Then verify creator and delete:
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
```

Alternatively, refactor `removeTransactionApprover` in the service to accept and enforce `transactionId` as a required parameter, consistent with how `updateTransactionApprover` is implemented. [4](#0-3) 

## Proof of Concept
1. Attacker (Alice) authenticates and creates Transaction T1. She is the creator of T1.
2. Alice enumerates approver IDs (e.g., `id = 42`) belonging to Bob's Transaction T2 by incrementing integers.
3. Alice sends:
   ```
   DELETE /transactions/{T1_id}/approvers/42
   Authorization: Bearer <alice_token>
   ```
4. The server checks: is Alice the creator of T1? **Yes** → passes.
5. The server calls `removeTransactionApprover(42)` — no check that approver 42 belongs to T1.
6. Approver 42 (belonging to T2) is soft-deleted. Bob's transaction T2 now has a corrupted or reduced approver set, potentially allowing it to execute with fewer approvals than intended.

  async getRootNodeFromNode(
    id: number,
    entityManager?: EntityManager,
  ): Promise<TransactionApprover | null> {
    if (!id || typeof id !== 'number') return null;

    return (
      await (entityManager || this.repo).query(
        `
        with recursive approverList as
          (
            select * from transaction_approver 
            where "id" = $1
              union all 
                select approver.* from transaction_approver as approver
                join approverList on approverList."listId" = approver."id"
          )
        select * from approverList
        where "listId" is null
        `,
        [id],
      )
    )[0];
  }
