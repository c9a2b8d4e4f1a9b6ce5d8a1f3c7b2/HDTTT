### Title
Unbounded Recursive Approver Tree Creation Enables Single-Request DoS via Database Exhaustion

### Summary
The `POST /transactions/:id/approvers` endpoint accepts an arbitrarily large and arbitrarily deep nested approver tree with no size or depth constraints. The service processes each node recursively, executing multiple synchronous database queries per node inside a single database transaction. A single crafted request from any authenticated user who has created a transaction can exhaust the database connection pool and degrade or crash the API service.

### Finding Description

**Root cause — no array size or depth limit on the DTO:**

`CreateTransactionApproverDto` and `CreateTransactionApproversArrayDto` enforce only `ArrayMinSize(1)` and no `ArrayMaxSize`: [1](#0-0) 

The `approvers` field inside `CreateTransactionApproverDto` is itself typed as `CreateTransactionApproverDto[]`, making the structure arbitrarily deep with no depth guard.

**Root cause — unbounded recursive processing with per-node DB queries:**

`createTransactionApprovers` calls the inner `createApprover` function for every element of `dto.approversArray`, and `createApprover` calls itself recursively for every nested child: [2](#0-1) 

Each recursive invocation of `createApprover` executes up to **five** sequential database round-trips:
1. `isNode` → `count(TransactionApprover)` [3](#0-2) 
2. `getRootNodeFromNode` → recursive CTE query [4](#0-3) 
3. `count(User)` for userId validation [5](#0-4) 
4. `insert(TransactionApprover)` [6](#0-5) 
5. `getApproversByTransactionId` → another recursive CTE (for userId nodes) [7](#0-6) 

All of this runs inside a single `dataSource.transaction(...)` block, holding a database connection open for the entire duration: [8](#0-7) 

**Secondary surface — unbounded observer array:**

`CreateTransactionObserversDto` also has no `ArrayMaxSize`, and `createTransactionObservers` iterates over every `userId` with an O(n×m) inner `.some()` scan against existing observers before a bulk save: [9](#0-8) [10](#0-9) 

### Impact Explanation

A single HTTP request containing a deeply nested approver tree with N total nodes triggers O(5N) database queries, all held within one open database transaction. With N = 200 nodes (easily within a 100 KB JSON body), this produces ~1,000 sequential DB queries per request. Concurrent requests of this kind will exhaust the PostgreSQL connection pool, causing all other API operations to queue or time out. This constitutes a full API denial-of-service achievable by any authenticated user with no privileged role.

### Likelihood Explanation

The attacker preconditions are minimal:
1. Register a normal user account (open registration flow).
2. Create any transaction (normal product flow).
3. POST a crafted deeply-nested `approversArray` JSON body to `POST /transactions/:id/approvers`.

No admin keys, no leaked credentials, and no internal network access are required. The endpoint is reachable by any authenticated organization member who has created at least one transaction.

### Recommendation

1. **Add `ArrayMaxSize` to both DTOs** — enforce a hard cap (e.g., 50) on `approversArray` in `CreateTransactionApproversArrayDto` and on `userIds` in `CreateTransactionObserversDto`.
2. **Add `ArrayMaxSize` on nested `approvers`** inside `CreateTransactionApproverDto` to cap the fan-out at each level.
3. **Add a maximum recursion depth guard** inside `createApprover` — pass a `depth` counter and throw if it exceeds a configured limit (e.g., 5).
4. **Batch or limit DB queries** — replace the per-node sequential queries with set-based pre-validation before entering the recursive loop.

### Proof of Concept

**Attacker-controlled request** (authenticated as the transaction creator):

```
POST /transactions/1/approvers
Authorization: Bearer <valid_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                ... (repeat nesting 50+ levels deep, each level with multiple children)
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**Expected outcome:** The server executes hundreds to thousands of sequential DB queries within a single held database transaction. Concurrent requests of this shape exhaust the PostgreSQL connection pool. All other API requests begin timing out, resulting in full service unavailability for all users.

              where: { id: dtoApprover.listId },

            /* Check if the root transaction is the same */

              dtoApprover.listId,
              transactionalEntityManager,

            if (root?.transactionId !== transactionId)
              throw new Error(this.ROOT_TRANSACTION_NOT_SAME);
          }

          /* Check if the user exists */
          if (typeof dtoApprover.userId === 'number') {
            const userCount = await transactionalEntityManager.count(User, {
              where: { id: dtoApprover.userId },

            if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
          }

          /* Check if there are sub approvers */
          if (
            typeof dtoApprover.userId === 'number' &&
            dtoApprover.approvers &&
            dtoApprover.approvers.length > 0

          /* Check if the approver has threshold when there are children */
          if (
            dtoApprover.approvers &&
            dtoApprover.approvers.length > 0 &&
            (dtoApprover.threshold === null || isNaN(dtoApprover.threshold))

          /* Check if the approver has children when there is threshold */
          if (

          if (
            dtoApprover.approvers &&
            (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)

              data.userKeyId = userApproverRecords[0].userKeyId;
