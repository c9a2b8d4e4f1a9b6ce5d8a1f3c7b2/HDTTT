Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Memory and CPU Exhaustion for Any Authenticated User

## Summary
`TransactionsService::getTransactionsToSign()` fetches every non-terminal transaction from the database with no row limit, then iterates the entire result set in application memory — issuing additional database calls per row via `userKeysToSign()` — before applying pagination in memory. Any registered, verified user can trigger this endpoint with a single HTTP request, causing resource consumption proportional to the total number of active transactions in the system.

## Finding Description

**Root cause — unbounded `find()` with no `take`/`skip`:**

`getTransactionsToSign` issues a `repo.find()` with only a status filter and no pagination limit: [1](#0-0) 

Compare this to `getHistoryTransactions` and `getTransactionsToApprove`, which correctly pass `skip: offset, take: limit` to the query: [2](#0-1) 

**N+1 query pattern — `userKeysToSign()` called per transaction:**

Every row returned by the unbounded query is iterated, and `userKeysToSign()` is called for each one: [3](#0-2) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `attachKeys(user, entityManager)` and `transactionSignatureService.computeSignatureKey(transaction, showAll)` per invocation — both of which involve additional I/O: [4](#0-3) [5](#0-4) 

**In-memory pagination applied after full scan:**

The `offset`/`limit` slice is applied only after the entire result set has been loaded and processed: [6](#0-5) 

**Endpoint accessible to any verified, authenticated user:**

The route `GET /transactions/sign` is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin or elevated role is required: [7](#0-6) [8](#0-7) 

**Secondary finding — `getUsers` also unbounded:**

`UsersService::getUsers()` issues an unbounded `repo.find()` with no pagination for both admin and non-admin callers: [9](#0-8) 

## Impact Explanation

For each request to `GET /transactions/sign`, the Node.js process:
1. Loads all N non-terminal transaction rows into heap memory.
2. Executes at least one additional I/O call per row via `computeSignatureKey` (and `attachKeys` on first call).

With a large transaction table, a single request can exhaust heap memory (OOM kill) or saturate the database connection pool. Repeated requests from a single authenticated user can sustain the degradation, making the API unavailable for all users. Even without a malicious actor, organic transaction accumulation in production will cause this to degrade over time.

## Likelihood Explanation

- **Attacker precondition:** any registered, verified user account — a normal product onboarding flow.
- **Attack complexity:** a single `GET /transactions/sign` request with default pagination parameters is sufficient; no special tooling or elevated privileges are required.
- **Natural growth risk:** even without a malicious actor, a production deployment accumulates non-terminal transactions over time, making this a reliability issue that degrades organically.
- **Note:** the controller comment marks this endpoint as "NO LONGER USED BY FRONT-END," but it remains live and reachable via the API. [10](#0-9) 

## Recommendation

1. **Push pagination to the database layer.** Add `take: limit` and `skip: offset` to the `repo.find()` call in `getTransactionsToSign`, mirroring the pattern used in `getHistoryTransactions` and `getTransactionsToApprove`.
2. **Eliminate the N+1 pattern.** Batch-fetch the signature key data for all transactions in a single query rather than calling `userKeysToSign()` per row. Consider pre-fetching `publicKeys` stored on the transaction entity to avoid per-row `computeSignatureKey` calls where possible.
3. **Add a hard server-side cap.** Enforce a maximum `limit` value (e.g., 100) in the `PaginationParams` decorator or middleware to prevent callers from requesting arbitrarily large pages.
4. **Paginate `getUsers`.** Apply `take`/`skip` to the `repo.find()` calls in `UsersService::getUsers()`.
5. **Consider deprecating or removing the endpoint** if it is confirmed to be unused by all consumers, to reduce attack surface.

## Proof of Concept

```
# 1. Obtain a valid JWT for any registered, verified user account.
# 2. Issue a single GET request to the sign endpoint:

curl -H "Authorization: Bearer <JWT>" \
     "https://<api-host>/transactions/sign?page=1&limit=10"

# With N non-terminal transactions in the database, the server will:
# - Execute SELECT * FROM transaction WHERE status NOT IN (...) [no LIMIT]
# - Call computeSignatureKey() for each of the N rows in a sequential loop
# - Hold all N transaction objects in heap until the response is sent
#
# Repeating this request in a loop sustains heap pressure and DB load:

while true; do
  curl -s -o /dev/null -H "Authorization: Bearer <JWT>" \
       "https://<api-host>/transactions/sign?page=1&limit=10"
done
```

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
