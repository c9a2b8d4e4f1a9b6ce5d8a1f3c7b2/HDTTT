### Title
Server-Side Request Forgery (SSRF) via Unvalidated `mirrorNetwork` Field in Transaction Creation

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` is only validated as a non-empty string with no allowlist enforcement. Any authenticated, verified user can supply an arbitrary hostname or IP address as `mirrorNetwork`, causing the backend to make outbound gRPC and HTTP connections to attacker-controlled or internal infrastructure. The value is also persisted to the database and reused by the chain service on every subsequent transaction lifecycle event.

### Finding Description

**Root cause:** `CreateTransactionDto.mirrorNetwork` carries only `@IsNotEmpty()` and `@IsString()` decorators — no allowlist against known networks. [1](#0-0) 

**Exploit path 1 — gRPC connection at transaction creation time:**

`createTransactions()` immediately calls `getClientFromNetwork(dtos[0].mirrorNetwork)`. [2](#0-1) 

Inside `getClientFromNetwork`, any value that is not `mainnet`, `testnet`, `previewnet`, or `local-node` falls through to the `default` branch, which constructs a gRPC client pointed at `<attacker-host>:443` and immediately executes an `AddressBookQuery` against it. [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` for an unknown value returns `[mirrorNetwork + ':443']` with no sanitisation. [4](#0-3) 

**Exploit path 2 — HTTP connection via `MirrorNodeClient`:**

`MirrorNodeREST.fromBaseURL` for an unknown value returns `https://<mirrorNetwork>` verbatim. [5](#0-4) 

`MirrorNodeClient.getMirrorNodeRESTURL` uses this to build the full URL, and `fetchAccountInfo` / `fetchNodeInfo` then issue real HTTP GET requests to it via Axios. [6](#0-5) [7](#0-6) 

**Persistence:** The raw `mirrorNetwork` string is stored in the `Transaction` entity and reused by the chain service on every subsequent operation (scheduling, execution, account/node cache refresh), so a single malicious submission causes repeated outbound connections. [8](#0-7) 

### Impact Explanation
An authenticated user can force the backend to:
- Make TCP connections to arbitrary hosts and ports (internal network port scan).
- Issue HTTP GET requests to cloud instance-metadata endpoints (e.g., `http://169.254.169.254/latest/meta-data/`) to retrieve IAM credentials or other secrets.
- Reach internal microservices, databases, or admin panels not exposed to the public internet.
- Exfiltrate data returned by internal services through error messages or timing side-channels.

Because the value is persisted and reused, the SSRF is triggered repeatedly without further attacker interaction.

### Likelihood Explanation
The attacker must be a registered, verified organization user with at least one uploaded key pair — a normal user role reachable without any privileged access. Creating a valid signed transaction is straightforward using the Hedera SDK. No special tooling beyond a standard HTTP client is required to submit the crafted `mirrorNetwork` value.

### Recommendation
Enforce an allowlist on `mirrorNetwork` at the DTO layer. Reject any value that is not one of the four known network identifiers (`mainnet`, `testnet`, `previewnet`, `local-node`) or, if custom networks must be supported, validate the value against a server-side configured list of permitted hostnames/CIDRs before any outbound connection is made.

```typescript
// In CreateTransactionDto
@IsNotEmpty()
@IsString()
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

If custom mirror nodes are a required feature, move the allowlist to server configuration and validate against it before calling `getClientFromNetwork` or `MirrorNodeREST.fromBaseURL`.

### Proof of Concept

1. Register and verify an account on the organization backend.
2. Upload a valid ED25519 public key via `POST /user/:userId/keys`.
3. Construct a minimal valid Hedera `AccountCreateTransaction`, sign it with the corresponding private key.
4. Submit:

```http
POST /transactions
Authorization: Bearer <jwt>
Content-Type: application/json

{
  "name": "ssrf-test",
  "description": "ssrf",
  "transactionBytes": "<hex-encoded signed tx bytes>",
  "creatorKeyId": <your key id>,
  "signature": "<hex-encoded signature>",
  "mirrorNetwork": "169.254.169.254"
}
```

5. The backend calls `getClientFromNetwork("169.254.169.254")`, which constructs a gRPC client targeting `169.254.169.254:443` and executes `AddressBookQuery` against it.
6. Subsequently, `MirrorNodeClient` issues `GET https://169.254.169.254/api/v1/accounts/<accountId>` — reaching the AWS instance-metadata service.
7. Observe outbound connections in network logs or via a controlled listener (e.g., `requestbin`, `interactsh`) confirming the SSRF.
