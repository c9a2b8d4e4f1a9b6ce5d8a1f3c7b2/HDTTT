# Audit Report

## Title
Cross-Transaction Approver Deletion via Missing Ownership Binding in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `:transactionId` but then deletes the approver identified by `:id` without verifying that the approver belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause:** The controller performs authorization against `transactionId` (URL param), but the service method that executes the deletion operates only on the approver `id` (a separate URL param) with no cross-reference check.

**Controller** — `approvers.controller.ts` lines 102–113: [1](#0-0) 

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // ← deletes approver `id` with no binding check
```

**Service** — `approvers.service.ts` lines 533–544: [2](#0-1) 

`removeTransactionApprover` receives only `id`, fetches the approver, and calls `removeNode(approver.id)`. The `transactionId` URL parameter is used solely for the `getCreatorsTransaction` guard and is then discarded — it is never passed into the service method.

**Contrast with `updateTransactionApprover`**, which correctly validates the binding at lines 386–394: [3](#0-2) 

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
```

The update path is protected; the delete path is not.

## Impact Explanation

An attacker can remove approvers from any transaction they do not own, bypassing the multi-signature approval workflow:

- Approval requirements for high-value Hedera transactions (account updates, token operations, etc.) can be silently stripped.
- A transaction requiring N-of-M approvals can be reduced to zero required approvals, allowing it to proceed to execution without the intended governance controls.
- The victim transaction creator receives no indication that their approver configuration was tampered with.

## Likelihood Explanation

Attacker preconditions (no elevated privilege required):
1. **Valid JWT** — any registered, verified user.
2. **Creator of at least one transaction** — trivially satisfied by creating any transaction to pass the `getCreatorsTransaction` guard.
3. **Knowledge of a target approver `id`** — approver IDs are sequential integers, fully enumerable by brute-force or by observing one's own approver IDs.

The endpoint is reachable by any authenticated user with no admin role, no leaked credentials, and no physical access required.

## Recommendation

Mirror the binding check already present in `updateTransactionApprover`. In `removeTransactionApprover`, after fetching the approver, walk up to its root node and assert that `rootNode.transactionId === transactionId` before proceeding with deletion:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Binding check — mirrors updateTransactionApprover
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: rootNode.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, and remove the separate `getCreatorsTransaction` call from the controller (the service already calls it in the update path; the same pattern should be followed here, or keep it in the controller but add the binding check in the service).

## Proof of Concept

1. Attacker (user A) creates transaction `T_A` — this satisfies `getCreatorsTransaction`.
2. Attacker observes or enumerates approver IDs (e.g., their own approver has `id=5`, so victim approver IDs near `id=6`, `id=7`, etc. are candidates).
3. Attacker sends:
   ```
   DELETE /transactions/{T_A}/approvers/{victim_approver_id}
   ```
   where `victim_approver_id` belongs to transaction `T_B` owned by user B.
4. The server calls `getCreatorsTransaction(T_A, userA)` — passes, because A owns `T_A`.
5. The server calls `removeTransactionApprover(victim_approver_id)` — no binding check, approver is soft-deleted.
6. Transaction `T_B`'s approval requirement is silently reduced without user B's knowledge.
