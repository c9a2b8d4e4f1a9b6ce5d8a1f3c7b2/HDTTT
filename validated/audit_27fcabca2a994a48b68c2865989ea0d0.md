Audit Report

## Title
`uploadKey()` Index Immutability Bypass via JavaScript Falsy Check on `index === 0`

## Summary
In `back-end/apps/api/src/user-keys/user-keys.service.ts`, the `uploadKey()` function uses a JavaScript truthy check (`userKey.index && ...`) to guard against re-uploading a key with a different `index`. When the stored `index` is `0` — a valid BIP-32 derivation value accepted by the DTO and entity — the guard silently evaluates to `false`, allowing an authenticated user to overwrite their own key's `index` (and `mnemonicHash`) unconditionally.

## Finding Description
The guard at line 52 reads:

```typescript
if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
  throw new BadRequestException(ErrorCodes.PU);
}
// Set the hash and/or index (only if the current value is null)
Object.assign(userKey, dto);
``` [1](#0-0) 

The sub-expression `(userKey.index && userKey.index !== dto.index)` treats `userKey.index` as a boolean. When `userKey.index === 0`, JavaScript evaluates `0` as falsy, causing the entire sub-expression to short-circuit to `false`. The outer `||` then reduces to `userKey.userId !== user.id`, which is `false` for the key's legitimate owner. No exception is thrown, and `Object.assign(userKey, dto)` overwrites both `index` and `mnemonicHash` with attacker-supplied values.

The `UserKey` entity declares `index` as a nullable integer column with no minimum-value constraint:

```typescript
@Column({ nullable: true })
index: number;
``` [2](#0-1) 

The `UploadUserKeyDto` similarly applies only `@IsNumber()` with no `@Min(1)`, so `index: 0` passes all validation:

```typescript
@IsNumber()
@IsNotEmpty()
index?: number;
``` [3](#0-2) 

The code comment on line 55 explicitly states the intent — *"Set the hash and/or index (only if the current value is null)"* — but the implementation does not enforce this; `Object.assign` always overwrites regardless of the current value.

## Impact Explanation
- **Index corruption**: `index` records the BIP-32 derivation slot used to produce the key pair. Overwriting it breaks the linkage between the stored public key and the derivation path the front-end uses to reconstruct the private key, causing signature verification to fail for any transaction requiring this key.
- **`mnemonicHash` overwrite**: The hash is used by downstream account-setup checks. Replacing it with an arbitrary value can corrupt account-setup state.
- **Permanent state corruption**: Once overwritten, the original `index=0` value is gone with no recovery path without admin intervention.

## Likelihood Explanation
- Requires only a valid authenticated session — no admin or privileged role needed.
- `index: 0` is a valid BIP-32 derivation value, accepted by both the DTO (`@IsNumber()`, no `@Min(1)`) and the entity (`nullable: true`).
- The API endpoint is reachable via normal product flow (`POST /users/:id/keys`).
- The existing e2e test at line 126–142 never exercises `index = 0` because `generatePrivateKey()` hardcodes `index = 1`, leaving the vulnerable case undetected:

```typescript
const index = 1;  // always 1, never 0
``` [4](#0-3) [5](#0-4) 

## Recommendation
Replace the falsy check with an explicit `null`/`undefined` check:

```typescript
// Before (buggy):
if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {

// After (correct):
if (
  userKey.userId !== user.id ||
  (userKey.index != null && userKey.index !== dto.index)
) {
```

Additionally, enforce the comment's stated intent by only assigning fields that are currently `null`/`undefined`, rather than using `Object.assign` unconditionally:

```typescript
if (userKey.mnemonicHash == null) userKey.mnemonicHash = dto.mnemonicHash;
if (userKey.index == null) userKey.index = dto.index;
```

Add a dedicated e2e test case that registers a key with `index: 0` and then attempts to re-upload it with a different index, asserting a `400` response.

## Proof of Concept
1. Authenticated user registers key with `index: 0`:
   ```
   POST /user/2/keys
   { "publicKey": "X", "mnemonicHash": "H1", "index": 0 }
   → 201 Created; stored: index=0, mnemonicHash=H1
   ```
2. Same user re-submits with a different index:
   ```
   POST /user/2/keys
   { "publicKey": "X", "mnemonicHash": "H2", "index": 99 }
   ```
3. Guard evaluation: `(0 && 0 !== 99)` → `0` (falsy) → no exception thrown.
4. `Object.assign(userKey, dto)` executes: backend now stores `index=99, mnemonicHash=H2`.
5. The original derivation path (`index=0`) is permanently lost. [6](#0-5)
