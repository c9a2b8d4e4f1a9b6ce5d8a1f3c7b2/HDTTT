### Title
Transaction Creator Can Add Themselves as an Approver, Bypassing the Multi-Party Approval Workflow

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` restricts the caller to the transaction creator but imposes no check preventing the creator from designating their own `userId` as an approver. This mirrors the external report's pattern: a privileged actor who should be excluded from a specific role can still assume it, collapsing the intended separation of duties. The creator can add themselves as an approver and then immediately self-approve the transaction, bypassing the entire multi-party approval mechanism.

### Finding Description

**Root cause — missing self-exclusion check in `createTransactionApprovers`:**

`createTransactionApprovers` first verifies the caller is the transaction creator: [1](#0-0) 

It then validates each approver DTO. The only user-existence check is: [2](#0-1) 

There is **no check** that `dtoApprover.userId !== user.id`. The creator can freely pass their own `userId` in the `approversArray` payload.

The same gap exists in `updateTransactionApprover`: after verifying the caller is the creator, it allows changing an existing approver's `userId` to any value, again without excluding the creator's own id: [3](#0-2) 

**Exploit flow:**

1. Authenticated user (creator) calls `POST /transactions` → creates transaction T.
2. Creator calls `POST /transactions/T/approvers` with body `{ "approversArray": [{ "userId": <creator_id> }] }`.
3. `createTransactionApprovers` passes `getCreatorsTransaction` (creator check ✓), passes user-existence check (creator exists ✓), and inserts the creator as the sole approver.
4. Creator calls `POST /transactions/T/approvers/approve` with their own key signature.
5. `approveTransaction` finds the creator in `userApprovers`, verifies the signature, and marks the transaction as approved.

The controller exposes both endpoints to any verified JWT holder with no additional guard: [4](#0-3) [5](#0-4) 

### Impact Explanation
The approver system is the protocol's multi-party check-and-balance layer: independent parties must approve a transaction before it can be executed. A creator who self-approves collapses this to a single-party action, defeating the governance model entirely. Any transaction — including high-value transfers or account updates — can be unilaterally approved by its creator without any independent oversight.

### Likelihood Explanation
The attacker is a normal authenticated user (no admin, no leaked secrets). The exploit requires only two standard API calls after transaction creation. The path is deterministic and requires no race condition or timing dependency. Any organization member who can create transactions can exploit this immediately.

### Recommendation
Add an explicit self-exclusion check inside `createTransactionApprovers` before inserting each approver:

```typescript
if (typeof dtoApprover.userId === 'number' && dtoApprover.userId === user.id) {
  throw new Error('The transaction creator cannot be added as an approver');
}
```

Apply the same guard in `updateTransactionApprover` when `dto.userId` is being set:

```typescript
if (typeof dto.userId === 'number' && dto.userId === user.id) {
  throw new Error('The transaction creator cannot be set as an approver');
}
```

These checks should be placed at:
- `approvers.service.ts` line ~277 (inside `createApprover`, after the user-existence check)
- `approvers.service.ts` line ~497 (inside the `dto.userId` branch of `updateTransactionApprover`)

### Proof of Concept

**Preconditions:** Valid JWT for user with `id = 42`, who is the creator of transaction `id = 7`.

**Step 1 — Self-add as approver:**
```
POST /transactions/7/approvers
Authorization: Bearer <creator_jwt>
Content-Type: application/json

{ "approversArray": [{ "userId": 42 }] }
```
Response: `201 Created` — creator is now the sole approver.

**Step 2 — Self-approve:**
```
POST /transactions/7/approvers/approve
Authorization: Bearer <creator_jwt>
Content-Type: application/json

{ "userKeyId": <creator_key_id>, "signature": "<valid_sig>", "approved": true }
```
Response: `200 true` — transaction is approved by its own creator, bypassing all independent oversight.
