## Audit Report

## Title
Creator Can Remove or Replace Rejecting Approvers to Bypass Transaction Approval Governance

## Summary
The transaction creator can add, remove, or update approvers on a transaction at any point in its lifecycle — including after approvers have already submitted rejections — because none of the three approver-mutation endpoints enforce a transaction-status guard. A malicious creator can silently delete a rejecting approver's record, erasing the rejection from the governance layer and allowing the transaction to proceed without the required multi-party consent.

## Finding Description

**Root cause — missing status guard on all three approver-mutation paths:**

**Path 1 — `DELETE /transactions/:transactionId/approvers/:id`**

In `approvers.controller.ts`, the handler calls only `getCreatorsTransaction` (creator identity check) and then unconditionally calls `removeTransactionApprover`:

```typescript
@Delete('/:id')
async removeTransactionApprover(user, transactionId, id) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
``` [1](#0-0) 

`removeTransactionApprover` fetches the approver, calls `removeNode` (a recursive soft-delete), and then emits `emitTransactionStatusUpdate` — with no status guard at any point: [2](#0-1) 

**Path 2 — `POST /transactions/:transactionId/approvers` (create)**

`createTransactionApprovers` calls `getCreatorsTransaction` and proceeds to insert new approvers with no status check: [3](#0-2) 

**Path 3 — `PATCH /transactions/:transactionId/approvers/:id` (update userId)**

When `dto.userId` is supplied and differs from the current approver's `userId`, the existing approver's `userId`, `userKeyId`, `signature`, and `approved` fields are all cleared and replaced — erasing the rejection:

```typescript
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: undefined,
  signature: undefined,
  approved: undefined,
};
``` [4](#0-3) 

**`getCreatorsTransaction` — the only guard, checks nothing about status:** [5](#0-4) 

**Status recalculation is triggered after every mutation:**

After each approver removal, `emitTransactionStatusUpdate` is published. The `processTransactionStatus` function — called by the chain service's cron jobs — re-evaluates whether the transaction should advance. Critically, `processTransactionStatus` evaluates only cryptographic signatures on the transaction bytes; it does **not** inspect the `approved` flag on approver records. With the rejecting approver's record soft-deleted, the governance record is clean, and the transaction can advance to `WAITING_FOR_EXECUTION` once sufficient cryptographic signatures are present: [6](#0-5) 

**Approval stores a cryptographic signature over the current transaction bytes:**

The signature is bound to the frozen transaction bytes, not to the approver-list configuration. Removing an approver record does not invalidate the bytes; it only removes the governance gate. [7](#0-6) 

## Impact Explanation

The approval system is the sole governance control preventing a transaction from being submitted to the Hedera network without multi-party consent. A creator who receives a rejection can:

1. Call `DELETE /transactions/:id/approvers/:cfoApproverId` — no status check blocks this.
2. The rejection record is soft-deleted; the governance record no longer shows the rejection.
3. The creator can add a new compliant approver (or the transaction already has sufficient cryptographic signatures).
4. The transaction advances to `WAITING_FOR_EXECUTION` and is executed on-chain — irreversibly, because Hedera transactions are final.

Concrete example: an organization requires CFO approval before large HBAR transfers. The CFO rejects a suspicious transfer. The creator (malicious employee) calls `DELETE /transactions/:id/approvers/:cfoApproverId`. The rejection disappears, and the transfer can execute on-chain without the CFO's consent.

## Likelihood Explanation

- **Attacker profile**: any authenticated user who is the creator of a transaction — no admin or privileged role required.
- **Precondition**: the attacker must have created the transaction (a normal workflow step).
- **Trigger**: a single authenticated HTTP `DELETE` or `PATCH` request.
- **Detection difficulty**: the deletion only sets `deletedAt` on the approver record; the approver who rejected receives a status-update notification but no explicit "your rejection was removed" alert. There is no audit trail of the original rejection in the active approver table.

## Recommendation

Add a transaction-status guard to all three approver-mutation paths. Before allowing any mutation, verify that the transaction is still in a mutable state (e.g., `NEW`). Reject mutations if the transaction is in `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any terminal state.

Concretely, in `getCreatorsTransaction` (or in each mutation handler), add:

```typescript
const allowedStatuses = [TransactionStatus.NEW];
if (!allowedStatuses.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify approvers: transaction is no longer in a mutable state.');
}
```

Apply this guard consistently to:
- `removeTransactionApprover` (DELETE path) — `approvers.controller.ts` line 108
- `createTransactionApprovers` (POST path) — `approvers.service.ts` line 239
- `updateTransactionApprover` (PATCH path) — `approvers.service.ts` line 394

Additionally, consider emitting an explicit notification to the affected approver when their approver record is removed, to provide an audit trail.

export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }
