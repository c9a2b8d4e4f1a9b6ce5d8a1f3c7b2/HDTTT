### Title
Stale Pre-Update State Checked in `approveTransaction` Causes Incorrect Notification Routing on Every Approval

### Summary
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction` function decides which notification event to emit by evaluating `userApprovers.every(a => a.approved)` on the **in-memory, pre-update snapshot** of approver records — not on the post-update database state. Because the guard at line 563 guarantees that at least one approver record has no signature (and therefore `approved=null`) before the update, this stale check is **always `false`** when `dto.approved=true`. The result is that every approval action unconditionally emits the wrong event (`emitTransactionUpdate` instead of `emitTransactionStatusUpdate`), suppressing all in-app indicator updates and email notifications to transaction participants.

### Finding Description

**Root cause — stale variable checked after DB write:**

`userApprovers` is populated at line 553 from a database query, then the DB is updated at lines 599–610, but the in-memory array is never refreshed. [1](#0-0) [2](#0-1) 

At line 614, the branching condition is:

```typescript
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

**Why the stale check is always `false` for approvals:**

The guard at line 563 throws `TAP` (already approved) if `userApprovers.every(a => a.signature)` is true. Because `signature` and `approved` are always written together in the same `.set({...})` call, any record with `approved=true` also has `signature` set. Therefore, if all records already had `approved=true`, the guard would have thrown and execution would never reach line 614. [3](#0-2) 

This means: when `dto.approved=true`, `userApprovers.every(a => a.approved)` on the stale snapshot is **always `false`** (at least one record has `approved=null`). The condition collapses to `false || false = false`, so `emitTransactionUpdate` is **always** emitted for every approval.

**Downstream effect — two fundamentally different notification pipelines:**

`emitTransactionStatusUpdate` → `TRANSACTION_STATUS_UPDATE` → `processTransactionStatusUpdateNotifications`:
- Deletes and recreates in-app indicator notifications
- Sends email notifications (`TRANSACTION_READY_FOR_EXECUTION`, `TRANSACTION_WAITING_FOR_SIGNATURES`, etc.)
- Emits `sendNotifyClients` with `STATUS_UPDATE` event type [4](#0-3) 

`emitTransactionUpdate` → `TRANSACTION_UPDATE` → `processTransactionUpdateNotifications`:
- Only calls `sendNotifyClients` with `UPDATE` event type
- No in-app indicator creation, no email dispatch [5](#0-4) 

The two handlers are wired separately in the consumer: [6](#0-5) 

### Impact Explanation

Every time an approver submits an approval (`dto.approved=true`), the system:
1. **Skips all in-app notification indicator updates** — participants never see the updated `TRANSACTION_INDICATOR_EXECUTABLE` or `TRANSACTION_INDICATOR_SIGN` badge in the UI.
2. **Skips all email notifications** — the creator, observers, required signers, and other approvers do not receive `TRANSACTION_READY_FOR_EXECUTION` or `TRANSACTION_WAITING_FOR_SIGNATURES` emails.
3. **Emits only a lightweight WebSocket `UPDATE` event** instead of the `STATUS_UPDATE` event that drives the full notification pipeline.

This is a deterministic, 100%-reproducible integrity failure in the notification system for every approval action. Participants are left unaware that a transaction has progressed to a state requiring their action (e.g., signing), which can stall multi-party transaction workflows indefinitely. [7](#0-6) 

### Likelihood Explanation

This is triggered by the normal, unprivileged approval flow. Any authenticated user who is a designated approver on any transaction can trigger it simply by submitting an approval via `POST /transactions/:id/approvers/approve`. No special privileges, no race conditions, no crafted input required. The bug fires on every approval, not just edge cases. [8](#0-7) 

### Recommendation

After the DB update at lines 599–610, re-fetch the approver records from the database to get the post-update state before evaluating the notification routing condition. Alternatively, since the update sets all `userApprovers` to `approved=dto.approved`, the condition can be derived directly from `dto.approved` and the count of the user's approver records without a re-fetch:

```typescript
// After the DB update, always emit status update when approving
// (rejection already correctly emits status update via !dto.approved)
if (!dto.approved || true /* all user approvers now have approved=dto.approved */) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
}
```

Or more precisely, re-fetch `userApprovers` after the update and check the live state:

```typescript
const updatedApprovers = await this.getVerifiedApproversByTransactionId(transactionId, user);
const updatedUserApprovers = updatedApprovers.filter(a => a.userId === user.id);
if (!dto.approved || updatedUserApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(...);
} else {
  emitTransactionUpdate(...);
}
```

### Proof of Concept

1. Create a transaction requiring approval (status `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`).
2. As an approver, call `POST /transactions/:id/approvers/approve` with `approved: true`.
3. Observe that `emitTransactionUpdate` is called (not `emitTransactionStatusUpdate`) — confirmed by the fact that `processTransactionUpdateNotifications` runs instead of `processTransactionStatusUpdateNotifications`.
4. Verify that no in-app notification indicators are created/updated and no emails are dispatched to the creator, observers, or required signers.
5. Repeat for any number of approvers — the bug fires on every approval without exception. [9](#0-8)

  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }

  async processTransactionStatusUpdateNotifications(events: NotificationEventDto[]) {
    const ctx = await this.prepareEventContext(events, true);
    if (!ctx) return;

    const {
      cache,
      keyCache,
      transactionMap,
      approversMap,
      deletionNotifications,
      inAppNotifications,
      emailNotifications,
      inAppReceiverIds,
      emailReceiverIds,
      affectedUsers,
    } = ctx;

    // Process each event
    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping status-update notifications`);
        continue;
      }
      const approvers = approversMap.get(transactionId) || [];

      if (transaction.deletedAt && transaction.status !== TransactionStatus.CANCELED) {
        console.error(
          `Soft-deleted transaction ${transactionId} has unexpected status: ${transaction.status} (expected CANCELED)`
        );
        transaction.status = TransactionStatus.CANCELED;
      }

      const syncType = this.getInAppNotificationType(transaction.status);
      const emailType = this.getEmailNotificationType(transaction.status);

      // Single transaction for both notification types
      await this.entityManager.transaction(async entityManager => {
        await this.handleTransactionStatusUpdateNotifications(
          entityManager,
          transaction,
          approvers,
          syncType,
          emailType,
          cache,
          keyCache,
          deletionNotifications,
          inAppNotifications,
          inAppReceiverIds,
          emailNotifications,
          emailReceiverIds,
          affectedUsers,
          transactionId,
        );
      });
    }

    // Send all notifications in batch
    await this.sendDeletionNotifications(deletionNotifications);
    await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
    await this.sendEmailNotifications(emailNotifications, emailReceiverIds);
    await this.sendNotifyClients(affectedUsers, TRANSACTION_EVENT_TYPE.STATUS_UPDATE);
  }

  async processTransactionUpdateNotifications(events: NotificationEventDto[]) {
    const ctx = await this.prepareEventContext(events);
    if (!ctx) return;

    const {
      keyCache,
      transactionMap,
      approversMap,
      affectedUsers,
    } = ctx;

    // Process each event
    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) continue;
      const approvers = approversMap.get(transactionId) || [];

      const syncType = this.getInAppNotificationType(transaction.status);

      if (syncType) {
        const receiverIds = await this.getNotificationReceiverIds(this.entityManager, transaction, syncType, approvers, keyCache);
        const groupId = transaction.groupItem?.groupId;
        receiverIds.forEach(id => {
          this.addAffectedUser(affectedUsers, id, transactionId, groupId);
        });
      }
    }

    await this.sendNotifyClients(affectedUsers, TRANSACTION_EVENT_TYPE.UPDATE);
  }
