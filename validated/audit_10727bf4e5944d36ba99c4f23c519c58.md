Audit Report

## Title
User Key Soft-Deletion Without Checking Active Approver References Disrupts Pending Transaction Approval Workflows

## Summary
When a `UserKey` is soft-deleted via `removeUserKey` (user self-service) or `removeUser` (admin), no check is performed against pending `TransactionApprover` records. A user assigned as an approver can delete their own key after assignment, causing `approveTransaction` to silently return `false` or fail, leaving the `TransactionApprover` record permanently unsigned until the creator intervenes.

## Finding Description

**Root cause — no reference check on key deletion:**

`removeUserKey` in `back-end/apps/api/src/user-keys/user-keys.service.ts` soft-removes the key with no query against `TransactionApprover`: [1](#0-0) 

`removeUser` in `back-end/apps/api/src/users/users.service.ts` bulk-soft-deletes all keys for a user, again with no check for pending approvals: [2](#0-1) 

**How the approval flow breaks after key deletion:**

`attachKeys` in `back-end/libs/common/src/utils/user/index.ts` queries `UserKey` without `withDeleted: true`, so soft-deleted keys are excluded from `user.keys`: [3](#0-2) 

`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` calls `attachKeys` and silently returns `false` at line 567 if all of the user's keys are soft-deleted: [4](#0-3) 

If the user has other active keys but the specific `dto.userKeyId` was deleted, `signatureKey` resolves to `undefined` at line 569, and `PublicKey.fromString(undefined)` at line 572 will throw, resulting in a `400` error with no state change.

In both cases, the `TransactionApprover` row with `signature = null` and `approved = null` remains in the database and cannot be advanced.

**The `UserKey` entity confirms the relationship:** [5](#0-4) 

The `approvedTransactions` one-to-many relation shows a single `UserKey` is the designated signing key for potentially many `TransactionApprover` records.

**Notification service confirms soft-deleted keys are excluded from required signers:** [6](#0-5) 

The system intentionally excludes soft-deleted keys from signing requirements but does not cascade that exclusion back to unblock the stuck `TransactionApprover` record.

## Impact Explanation

A transaction in `WAITING_FOR_SIGNATURES` that requires approval from a user whose key has been soft-deleted is stuck: no signature can be recorded for that approver slot. The `TransactionApprover` row remains with `signature = null` and `approved = null`. The creator must manually intervene — either by removing the stuck approver via `DELETE /transactions/:transactionId/approvers/:id`, replacing them via `PATCH`, or cancelling the transaction via `PATCH /transactions/cancel/:id`. Until the creator takes action, all parties waiting on that transaction are blocked. The impact is **not permanent** (the creator has remediation paths), but it is a real, exploitable workflow disruption that requires out-of-band coordination to resolve.

## Likelihood Explanation

The attack requires only a normal authenticated user who has been added as a transaction approver. The user calls `DELETE /users/:userId/keys/:keyId` — a standard self-service endpoint — after being assigned as an approver. No admin access, no leaked credentials, and no special timing is required. The attacker simply deletes their own key. This is a realistic, low-effort action by a malicious or uncooperative approver.

## Recommendation

1. **Guard key deletion against active approver references:** Before soft-deleting a `UserKey`, query `TransactionApprover` for any records where `userKeyId = :keyId` (or `userId = :userId` for user deletion) and `signature IS NULL` in transactions with status `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. Reject the deletion or warn the user if such records exist.

2. **Cascade the exclusion:** When a key is soft-deleted, automatically null out or reassign the corresponding `TransactionApprover.userKeyId` fields so the approver can re-sign with another key, or notify the transaction creator that an approver's key is no longer valid.

3. **Add a guard in `approveTransaction`:** Instead of silently returning `false` when `user.keys.length === 0`, return a descriptive error so the creator and approver are informed of the problem.

## Proof of Concept

1. User A creates a transaction requiring approval from User B.
2. User B is added as a `TransactionApprover` (record inserted with `userId = B.id`, `signature = null`, `approved = null`).
3. User B calls `DELETE /users/B/keys/:keyId`, triggering `removeUserKey` which calls `this.repo.softRemove(userKey)` with no check against `TransactionApprover`.
4. User B attempts to call `POST /transactions/:id/approvers/approve`. `attachKeys` loads `user.keys = []` (soft-deleted key excluded). Line 567 evaluates `user.keys.length === 0` → `return false`. No error, no state change.
5. The `TransactionApprover` row for User B remains with `signature = null`, `approved = null`. The transaction cannot advance past `WAITING_FOR_SIGNATURES` without creator intervention to remove or replace User B as an approver.
