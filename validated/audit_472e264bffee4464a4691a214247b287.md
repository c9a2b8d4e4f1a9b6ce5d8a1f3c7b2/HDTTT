Audit Report

## Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header Enables Unbounded Transaction and Group Creation

## Summary
`IpThrottlerGuard` — the only global rate-limiting guard — uses the raw, unvalidated `x-forwarded-for` request header as its Redis throttle key. An authenticated user can rotate this header on every request to obtain a fresh throttle bucket each time, completely bypassing the IP-based limit. The `UserThrottlerModule` and `UserThrottlerGuard` exist in the codebase but are never wired into the application, leaving `POST /transactions` and `POST /transaction-groups` with no effective per-user rate limit.

## Finding Description

**Root cause — `IpThrottlerGuard` trusts attacker-controlled header:**

`getTracker` in `IpThrottlerGuard` uses the full, unvalidated `x-forwarded-for` header string as the Redis throttle key: [1](#0-0) 

An attacker who sends `X-Forwarded-For: <rotating-value>` on each request gets a fresh throttle bucket every time. In a typical reverse-proxy setup (nginx, traefik), the proxy *appends* the real IP to an existing `x-forwarded-for` header rather than replacing it, so the full header value (and thus the Redis key) still changes with each rotation of the attacker-supplied prefix.

**`IpThrottlerGuard` is the sole `APP_GUARD` for rate limiting:**

`api.module.ts` registers `IpThrottlerGuard` as the only rate-limiting `APP_GUARD`: [2](#0-1) 

**`UserThrottlerModule` is defined but never imported:**

`UserThrottlerModule` defines per-user throttle limits (100 req/min, 10 req/sec) backed by Redis: [3](#0-2) 

However, it is absent from the `imports` array in `api.module.ts`, which only imports `IpThrottlerModule` and `EmailThrottlerModule`: [4](#0-3) 

**`UserThrottlerGuard` is defined but never applied:**

`UserThrottlerGuard` exists and correctly tracks by `req.user.id`: [5](#0-4) 

But it is not applied to any controller or route. Neither `POST /transactions`: [6](#0-5) 

nor `POST /transaction-groups`: [7](#0-6) 

uses `UserThrottlerGuard`. Both controllers only apply `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`.

## Impact Explanation

- **Database exhaustion:** Unlimited rows inserted into the `transaction` table, consuming PostgreSQL storage and degrading query performance for all users.
- **Notification spam / WebSocket flooding:** Every created transaction emits a NATS event fanned out to all connected WebSocket clients via the Notifications service.
- **Redis job exhaustion:** If `reminderMillisecondsBefore` is supplied, a Redis scheduler job is created per transaction, exhausting Redis memory.
- **Service degradation for all users:** Legitimate users experience slow queries, missed notifications, and UI unresponsiveness.

## Likelihood Explanation

- Precondition: a valid authenticated account — achievable by any registered user.
- Exploit technique: setting an HTTP header — trivially scriptable with `curl` or any HTTP client.
- No privileged access, no cryptographic break, no physical access required.
- Trusting `x-forwarded-for` without validation is a well-known bypass technique; the code explicitly prefers that header over `req.ip`.

## Recommendation

1. **Fix `IpThrottlerGuard`:** Extract only the *last* (rightmost) IP from the `x-forwarded-for` header (the one appended by the trusted proxy), or rely solely on `req.ip` which NestJS/Express populates from the socket address when `trust proxy` is configured correctly.
2. **Wire `UserThrottlerModule`:** Add `UserThrottlerModule` to the `imports` array in `api.module.ts`.
3. **Apply `UserThrottlerGuard`:** Decorate `createTransaction` and `createTransactionGroup` (and other resource-creation endpoints) with `@UseGuards(UserThrottlerGuard)` so that per-user limits are enforced independently of IP.

## Proof of Concept

```bash
JWT="<valid_jwt>"
KEY_ID=<registered_key_id>
COUNTER=0

while true; do
  COUNTER=$((COUNTER + 1))
  FAKE_IP="10.0.$((COUNTER / 256)).$((COUNTER % 256))"

  curl -s -o /dev/null -X POST https://<host>/transactions \
    -H "Authorization: Bearer $JWT" \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: $FAKE_IP" \
    -d "{\"creatorKeyId\": $KEY_ID, ...}"
done
```

Each iteration uses a fresh `X-Forwarded-For` value, producing a new Redis throttle bucket. The IP throttle never triggers, and with no `UserThrottlerGuard` in place, transactions are created without bound.
