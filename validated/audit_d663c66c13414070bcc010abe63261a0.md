Audit Report

## Title
Missing Input Validation on CSV-Supplied `txValidDuration` and `transactionFee` Causes Entire Transaction Groups to Fail on Submission

## Summary
In `ImportCSVController.vue`, the `txValidDuration` and `transactionFee` fields are read verbatim from a user-supplied CSV file and passed directly to `setTransactionValidDuration()` and `setMaxTransactionFee()` with no bounds or sanity checks. A malicious or malformed CSV can set these to values that cause every transaction in the imported group to be rejected by the Hedera network, wasting the user's (and all co-signers') signing effort and making the entire batch unexecutable.

## Finding Description

In `handleImportCsv()`, both parameters are captured from CSV rows without any validation: [1](#0-0) 

They are then used directly when constructing each `TransferTransaction`: [2](#0-1) 

**`txValidDuration` attack paths:**
- `transaction valid duration, 0` → `"0"` is truthy in JavaScript, so the fallback `180` is **never used**; `Number.parseInt("0")` = `0` is passed directly. Hedera enforces the range `[1, 180]` seconds and rejects `0` with `INVALID_TRANSACTION_DURATION`.
- `transaction valid duration, 99999` → duration exceeds the 180-second maximum; Hedera rejects with `INVALID_TRANSACTION_DURATION`.
- `transaction valid duration, abc` → `Number.parseInt("abc")` = `NaN`; passed to the SDK, which may throw or produce a malformed transaction body.

**`transactionFee` attack paths:**
- `transaction fee, 0` → `"0"` is truthy; `new Hbar("0", HbarUnit.Tinybar)` = 0 ℏ max fee. Hedera rejects every transaction with `INSUFFICIENT_TX_FEE`.
- `transaction fee, -1` → `new Hbar("-1", HbarUnit.Tinybar)` = negative Hbar; the SDK may throw or produce a malformed transaction.

The normal UI path (`BaseTransaction.vue`) does enforce a `maxTransactionFee > 0` guard: [3](#0-2) 

The UI also disables the submit button when the fee is ≤ 0: [4](#0-3) 

The CSV import path in `handleImportCsv()` bypasses both guards entirely.

## Impact Explanation
Every transaction in the imported group is built with the invalid parameter. After the user reviews and signs the group, all submissions to the Hedera network fail. The user receives no clear error at import time. In an organizational multi-signature workflow, this also wastes the signing effort of all co-signers who approved the group before execution is attempted, and the entire batch must be discarded and recreated.

## Likelihood Explanation
The attack requires delivering a crafted CSV to a user who then imports it via the normal "Import CSV" workflow. This is a realistic social-engineering vector (e.g., a malicious file sent by email or shared via a collaboration tool). The feature is explicitly designed to accept external CSV files, so no privileged access is required — only the ability to influence which file the user imports.

## Recommendation
Add explicit validation of `txValidDuration` and `transactionFee` immediately after they are parsed from the CSV, before any `TransferTransaction` is constructed. Specifically:

1. **`txValidDuration`**: Parse with `Number.parseInt`, then verify the result is a finite integer in the range `[1, 180]`. If invalid, call `toastManager.error(...)` and return `null` (matching the pattern already used for invalid account IDs at lines 72–77 and 84–89).
2. **`transactionFee`**: Parse the value and verify it is a finite number strictly greater than `0` before constructing `new Hbar(transactionFee, HbarUnit.Tinybar)`. If invalid, call `toastManager.error(...)` and return `null`.

These checks should mirror the guard already present in `basePreCreateAssert()` in `BaseTransaction.vue`.

## Proof of Concept
Craft a CSV file with the following header rows:

```
transaction fee, 0
transaction valid duration, 0
sender account, 0.0.1234
fee payer account, 0.0.1234
sending time, 12:00:00
0.0.5678, 100000000, 2024-01-01
```

Import this file via the "Import CSV" workflow. The resulting `TransferTransaction` will be constructed with:
- `setTransactionValidDuration(0)` — because `"0"` is truthy, bypassing the `180` fallback
- `setMaxTransactionFee(new Hbar("0", HbarUnit.Tinybar))` — zero max fee

After signing and submitting, the Hedera network rejects the transaction with `INVALID_TRANSACTION_DURATION` (or `INSUFFICIENT_TX_FEE` for the fee case). No error is shown to the user at import time.
