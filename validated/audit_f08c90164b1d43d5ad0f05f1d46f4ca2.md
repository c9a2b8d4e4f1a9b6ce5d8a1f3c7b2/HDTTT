### Title
Unauthenticated Signature Bytes Accepted in `uploadSignatureMaps` — Cryptographic Verification Absent

### Summary
`SignersService.uploadSignatureMaps()` records a user as a transaction signer and persists modified transaction bytes without ever cryptographically verifying that the submitted signature bytes are valid for the associated public key. A registered user can submit any arbitrary bytes as a "signature" for their own public key, falsely mark themselves as a signer, and corrupt the stored transaction bytes — potentially advancing the transaction to `WAITING_FOR_EXECUTION` with an invalid signature that Hedera will reject.

### Finding Description

There are two distinct code paths for submitting signatures:

**Path 1 — `TransactionsService.importSignatures()`** (`back-end/apps/api/src/transactions/transactions.service.ts`):

This path calls `validateSignature()` which performs real cryptographic verification via `publicKey.verify(bodyBytes, signature)` and throws if the signature is invalid. [1](#0-0) [2](#0-1) 

**Path 2 — `SignersService.uploadSignatureMaps()`** (`back-end/apps/api/src/transactions/signers/signers.service.ts`):

This path calls `processTransactionSignatures()`, which:
1. Checks that the public key in the submitted map belongs to the authenticated user (`userKeyMap.get(raw)`) — an ownership check only.
2. Calls `sdkTransaction.addSignature(publicKey, map)` — the Hedera SDK adds the bytes to the transaction without verifying them cryptographically.
3. Pushes the `userKey` into `userKeys[]` — recording the user as a signer.
4. Computes `isSameBytes` — only checks whether bytes changed, not whether the signature is valid. [3](#0-2) 

In `persistSignatureChanges()`, the condition `if (isSameBytes && userKeys.length === 0) continue;` means that as long as the user's key was found (even with garbage signature bytes), the record is persisted and transaction bytes are updated. [4](#0-3) 

The `IsSignatureMap` DTO decorator validates structure (account ID format, transaction ID format, public key format, non-empty decoded bytes) but performs **no cryptographic verification**. [5](#0-4) 

**Exploit flow:**
1. Attacker is an authenticated user with a registered public key (e.g., `pubKeyA`).
2. Attacker constructs a `signatureMap` containing `pubKeyA` → 64 bytes of random data (valid hex, non-empty, passes DTO validation).
3. POST to `POST /transactions/:id/signers` with this payload.
4. `processTransactionSignatures` finds `pubKeyA` in `userKeyMap` → ownership check passes.
5. `addSignature` appends the invalid bytes to the transaction → `isSameBytes = false`.
6. `userKeys = [userKey]` → user is recorded as a signer.
7. `persistSignatureChanges` inserts a `TransactionSigner` row and updates `transactionBytes` with the corrupted bytes.
8. `updateStatusesAndNotify` calls `processTransactionStatus` — if the threshold is now met, the transaction advances to `WAITING_FOR_EXECUTION`.

### Impact Explanation

- **False signer registration**: The attacker is permanently recorded as having signed a transaction without possessing the private key.
- **Transaction byte corruption**: The stored `transactionBytes` are overwritten with an invalid signature. Any subsequent legitimate signers who call `addSignature` will work on corrupted bytes.
- **Transaction execution failure**: If the threshold is met (including the fake signature), the chain service submits the transaction to Hedera, which rejects it with `INVALID_SIGNATURE`. The transaction may become permanently stuck or fail, causing loss of the transaction's intended effect (e.g., a fund transfer or account update that never executes).
- **Multi-sig integrity broken**: In a threshold key scenario (e.g., 2-of-3), the attacker can falsely satisfy their required slot, preventing the transaction from ever succeeding even if all other legitimate signers sign correctly.

### Likelihood Explanation

- **Preconditions**: Attacker must be an authenticated organization user with at least one registered public key. No admin or privileged access is required.
- **Effort**: Trivial — construct a hex-encoded 64-byte random string as the signature value in the JSON body.
- **Detectability**: The API returns HTTP 201 success, giving no indication of the invalid signature.
- **Reachability**: The endpoint `POST /transactions/:id/signers` is protected only by JWT auth and `VerifiedUserGuard`, both of which a normal user satisfies. [6](#0-5) 

### Recommendation

Add cryptographic signature verification in `processTransactionSignatures()` before calling `addSignature`, mirroring the logic already present in `validateSignature()`:

```typescript
// In processTransactionSignatures, after finding userKey:
const row = nodeAccountIdRow[nodeAccountId];
const col = transactionIdCol[transactionId];
const bodyBytes = sdkTransaction._signedTransactions.get(col * rowLength + row).bodyBytes;
const signatureBytes = txMap.get(publicKey); // get the raw bytes
if (!publicKey.verify(bodyBytes, signatureBytes)) {
  throw new Error(ErrorCodes.ISNMPN); // or a dedicated error code
}
```

Alternatively, refactor `validateSignature()` from `transactions.service.ts` into a shared utility and call it from both `importSignatures` and `uploadSignatureMaps` before any state mutation.

### Proof of Concept

1. Register as a user, add a key pair (keep only the public key registered; discard the private key or use a different one).
2. Create or identify a transaction in `WAITING_FOR_SIGNATURES` status where your key is a required signer.
3. Construct the request body:
```json
{
  "signatureMap": {
    "0.0.3": {
      "0.0.2@1234567890.000000000": {
        "<your_registered_public_key_DER>": "aabbccddeeff00112233445566778899aabbccddeeff00112233445566778899aabbccddeeff00112233445566778899aabbccddeeff00112233445566778899"
      }
    }
  }
}
```
4. `POST /transactions/:id/signers` with a valid JWT.
5. Observe HTTP 201 response. Query `GET /transactions/:id/signers` — your key appears as a signer.
6. Query the transaction — `transactionBytes` now contains the invalid signature bytes.
7. If threshold is met, the transaction advances to `WAITING_FOR_EXECUTION` and the chain service will submit it to Hedera, which returns `INVALID_SIGNATURE`.

```typescript
export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```

```typescript
  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
```
