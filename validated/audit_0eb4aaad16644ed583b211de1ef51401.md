### Title
Any Authenticated User Can Post and Read Comments on Any Transaction Without Transaction-Level Authorization

### Summary
The `createComment` endpoint in `CommentsController` performs only JWT authentication and account-status checks, but applies no transaction-membership authorization. Any active organization user can post comments on — and read comments from — any transaction they have no relationship to (not creator, signer, observer, or approver). The developer explicitly flagged this gap with a TODO comment in the source. The rest of the codebase enforces strict per-transaction access control, making this an isolated but concrete authorization bypass.

### Finding Description
**Root cause:** `CommentsController` applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level. `VerifiedUserGuard` only checks `user.status === UserStatus.NONE` — i.e., that the account is active — and performs no transaction-membership check. [1](#0-0) 

The developer acknowledged the missing check inline:

```
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(...)
```

`CommentsService.createComment` then directly persists the comment with no membership verification: [2](#0-1) 

The `getComments` and `getCommentById` endpoints on the same controller also carry no transaction-access check, exposing comment content to any authenticated user: [3](#0-2) 

**Contrast with the rest of the codebase:** Every other sensitive transaction sub-resource enforces membership. `TransactionsService.verifyAccess` checks creator, signer, observer, and approver relationships before granting access: [4](#0-3) 

`ObserversService.createTransactionObservers` explicitly rejects non-creators: [5](#0-4) 

`ApproversService.getVerifiedApproversByTransactionId` throws `UnauthorizedException` for users with no relationship to the transaction: [6](#0-5) 

The comments subsystem is the only area where this pattern is absent.

**Exploit flow:**
1. Attacker registers as a normal user (admin-created account, `UserStatus.NONE`).
2. Attacker has no relationship to transaction ID `N` (not creator, signer, observer, or approver).
3. Attacker sends `POST /transactions/N/comments` with a valid JWT and any `{ "message": "..." }` body.
4. `CommentsService.createComment` persists the comment without any membership check.
5. Attacker also sends `GET /transactions/N/comments` to read all comments on the transaction.

### Impact Explanation
- **Unauthorized state mutation:** Any active user can inject comments into any transaction, polluting the audit trail and communication channel for transactions they have no legitimate role in.
- **Cross-tenant data exposure:** Any active user can enumerate and read all comments on any transaction, leaking potentially sensitive operational details (e.g., signing instructions, key coordination notes) that are scoped to specific participants.
- The `TransactionComment` entity stores `message`, `userId`, `transactionId`, and `createdAt` — all of which are exposed. [7](#0-6) 

### Likelihood Explanation
**High.** The attacker precondition is only a valid active-user account (`UserStatus.NONE`), which is the normal state for any organization member. No admin privileges, no leaked secrets, and no special network access are required. The endpoint is reachable over the standard REST API. The developer's own TODO comment confirms the gap was known and left unresolved.

### Recommendation
In `CommentsService.createComment`, load the target transaction with its relations and call the existing `verifyAccess` logic (or an equivalent membership check) before persisting the comment. Reject the request with `UnauthorizedException` if the user is not the creator, a signer, an observer, or an approver of the transaction. Apply the same check to `getTransactionComments` to prevent unauthorized comment enumeration.

```typescript
// Example fix in CommentsService.createComment
const transaction = await this.transactionRepo.findOne({
  where: { id: transactionId },
  relations: ['creatorKey', 'observers', 'signers', 'signers.userKey', 'approvers'],
});
if (!transaction) throw new BadRequestException(ErrorCodes.TNF);
const hasAccess = await this.transactionsService.verifyAccess(transaction, user);
if (!hasAccess) throw new UnauthorizedException("You don't have permission to comment on this transaction");
```

### Proof of Concept
**Preconditions:** Two users exist — User A (creator of transaction `T`) and User B (no relationship to `T`). Both have `UserStatus.NONE`.

1. User B authenticates: `POST /auth/login` → receives `accessToken`.
2. User B posts a comment: `POST /transactions/T/comments` with header `Authorization: Bearer <accessToken>` and body `{"message": "injected comment"}`.
3. **Expected (correct) behavior:** `401 Unauthorized`.
4. **Actual behavior:** `201 Created` — comment is persisted to `transaction_comment` with `transactionId = T` and `userId = B`.
5. User B reads all comments: `GET /transactions/T/comments` → returns all comments including those from legitimate participants. [8](#0-7) [2](#0-1)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
