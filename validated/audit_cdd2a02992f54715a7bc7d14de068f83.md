# Audit Report

## Title
Missing Proof-of-Ownership on Public Key Registration Enables Key Squatting and Signer DoS

## Summary
`UserKeysService.uploadKey()` associates any valid Hedera public key with an authenticated user's account without requiring cryptographic proof that the submitter controls the corresponding private key. A registered attacker can pre-register a victim's known public key, permanently blocking the legitimate owner from registering it and preventing them from participating in any multi-signature workflow on the platform.

## Finding Description
The root cause is in `UserKeysService.uploadKey()` in `back-end/apps/api/src/user-keys/user-keys.service.ts`. The only checks performed are:

1. The public key is a valid Hedera key format (`@IsHederaPublicKey()` in `UploadUserKeyDto`).
2. The key is not already registered to a **different** user — if it is, `ErrorCodes.PU` is thrown.

No challenge-response, no signature-over-nonce, and no proof of private key possession is required. [6](#0-5) 

The DTO confirms there is no signature or proof field: [2](#0-1) 

The controller is protected by JWT authentication, so the attacker must be a registered user — but any registered user can submit any public key string: [7](#0-6) 

Once the attacker's account owns the key record, the uniqueness guard in `uploadKey` permanently rejects any subsequent attempt by the legitimate owner to register the same key: [3](#0-2) 

The `OnlyOwnerKey` interceptor relies on `user.keys.map(key => key.id)`, which now includes the squatted key under the attacker's account: [8](#0-7) 

## Impact Explanation
**Primary impact — Key Squatting / Permanent DoS on Signer:**
An attacker registers a target's known Hedera public key before the victim does. The victim's subsequent registration attempt is rejected with `ErrorCodes.PU`. The victim can no longer participate in any multi-signature transaction workflow on the platform that requires their key, because the platform has no record of them owning it. The attacker also cannot sign (they lack the private key), so any transaction requiring that key is permanently stalled on the platform.

**Secondary impact — False identity in non-cryptographic access control:**
The attacker's account holds a `userKeyId` pointing to the victim's public key. In `approveTransaction`, the key lookup `user.keys.find(key => key.id === dto.userKeyId)` resolves to the victim's public key under the attacker's account. While `verifyTransactionBodyWithoutNodeAccountIdSignature` would reject a forged signature (the attacker lacks the private key), the attacker appears as the registered owner of that key in all non-cryptographic access control paths, including observer checks and approver list lookups via `getVerifiedApproversByTransactionId`. [9](#0-8) 

## Likelihood Explanation
- **Attacker precondition**: Only a valid registered account is needed. No admin access, no leaked secrets.
- **Target information**: Hedera public keys are routinely shared — they are the public half of a key pair, visible on-chain and in transaction metadata.
- **Exploit window**: The race is won by whoever registers the key first. An attacker monitoring the Hedera mirror node for new public keys can pre-register them before legitimate users onboard to the platform.
- **No rate-limit or anomaly detection** on the key upload endpoint is present in the codebase.

## Recommendation
Implement a proof-of-possession challenge before accepting a public key registration:

1. **Server-side nonce**: When a user initiates key registration, the server issues a short-lived nonce tied to the user's session.
2. **Client-side signature**: The client signs the nonce with the private key corresponding to the public key being registered.
3. **Server-side verification**: The server verifies the signature against the submitted public key before persisting the association. The existing `verifyTransactionBodyWithoutNodeAccountIdSignature` utility or a direct `PublicKey.verify()` call can be reused for this purpose.

This ensures only the holder of the private key can register the corresponding public key.

## Proof of Concept
```
1. Attacker (registered user A) observes victim's public key `PK_victim` on-chain or in transaction metadata.

2. Attacker sends:
   POST /user/keys
   Authorization: Bearer <attacker_jwt>
   { "publicKey": "<PK_victim>" }
   → 201 Created. Key is now associated with attacker's account.

3. Victim (registered user B) attempts to register their own key:
   POST /user/keys
   Authorization: Bearer <victim_jwt>
   { "publicKey": "<PK_victim>" }
   → 400 Bad Request: ErrorCodes.PU ("Public key in use")

4. Victim can no longer register PK_victim on the platform.
   Any multi-sig transaction requiring PK_victim is permanently stalled:
   - Attacker holds the key record but cannot produce a valid signature.
   - Victim cannot register the key to sign legitimately.
```
