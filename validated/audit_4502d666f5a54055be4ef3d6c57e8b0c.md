## Audit Report

## Title
`archiveTransaction` Allows Unchecked State Transition to `ARCHIVED` for Manual Transactions

## Summary
A logical flaw in the `archiveTransaction` guard in `transactions.service.ts` causes the status validation to be completely skipped for any transaction where `isManual === true`. This allows a transaction creator to force a manual transaction that is already in a terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) into the `ARCHIVED` state, corrupting the audit trail.

## Finding Description
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `archiveTransaction` method contains the following guard:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The two sub-expressions are joined with `&&`. When `transaction.isManual === true`, the right-hand operand `!transaction.isManual` evaluates to `false`, making the entire condition `false` regardless of the transaction's current `status`. The exception is never thrown for manual transactions.

The subsequent database write:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

contains no status guard in its `WHERE` clause, unconditionally overwriting whatever status the transaction currently holds.

The service itself defines `terminalStatuses` as `[EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED, REJECTED]`, confirming these states are intended to be immutable post-transition.

The existing e2e test `"(PATCH) should not archive a transaction if it's already executed"` uses a non-manual seed transaction (`addedTransactions.userTransactions[0]`, seeded without `isManual: true`), so it correctly rejects the request — but it does not exercise the manual-transaction code path and therefore does not catch this bypass.

## Impact Explanation
A transaction creator can call `PATCH /transactions/archive/:id` on any manual transaction that has already reached `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`. The `status` column is unconditionally overwritten to `ARCHIVED`, which:

- Erases the `EXECUTED` or `FAILED` outcome from the transaction record, corrupting the on-chain audit trail visible to all organization members.
- Breaks downstream logic and reporting that distinguishes `EXECUTED` from `ARCHIVED` (e.g., the `HISTORY` status collection and `getHistoryStatusWhere` filtering).
- Allows a creator to retroactively relabel a failed or executed transaction as merely "archived," hiding its true outcome.

## Likelihood Explanation
The attacker must be the creator of a manual transaction — a legitimate, non-privileged role available to any organization member with at least one registered key. No special permissions beyond transaction creation are required. The archive endpoint is a standard REST `PATCH` call, and the bypass is triggered simply by having `isManual: true` on the transaction record, which is a normal, user-controlled flag set at creation time.

## Recommendation
Replace the `&&` conjunction with `||` so that the guard rejects the request when the status is not archiveable **regardless** of `isManual`:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Additionally, add a status guard to the `repo.update` `WHERE` clause (as is already done in `cancelTransactionWithOutcome`) to prevent race-condition overwrites:

```typescript
await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', {
    statuses: [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION],
  })
  .execute();
```

Add a dedicated unit/e2e test covering a manual transaction in each terminal state to prevent regression.

## Proof of Concept
1. Authenticate as any organization member with a registered key.
2. Create a manual transaction: `POST /transactions` with `{ ..., isManual: true }`. The transaction is created with status `WAITING_FOR_SIGNATURES`.
3. Simulate or wait for the transaction to reach `EXECUTED` status (e.g., via `executeTransaction`, or directly update the DB row for testing).
4. Call `PATCH /transactions/archive/<id>` as the creator.
5. **Expected**: `400 Bad Request` with `ErrorCodes.OMTIP`.
6. **Actual**: `200 OK`. The database row now shows `status = ARCHIVED`, erasing the `EXECUTED` record.

The bypass is triggered because at step 4, `transaction.isManual === true` causes `!transaction.isManual` to be `false`, collapsing the entire `&&` guard to `false`, and execution falls through to the unconditional `repo.update`.
