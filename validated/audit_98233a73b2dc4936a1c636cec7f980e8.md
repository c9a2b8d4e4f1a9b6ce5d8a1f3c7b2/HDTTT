Audit Report

## Title
Cross-Transaction Approver Removal via Missing Ownership Validation in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user is the creator of `:transactionId`, but then deletes the approver identified by `:id` without confirming that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can remove approvers from any other transaction in the system.

## Finding Description

**Controller** — `approvers.controller.ts`, lines 102–113:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of URL :transactionId only
  await this.approversService.removeTransactionApprover(id);               // deletes approver :id with NO cross-check
  return true;
}
``` [1](#0-0) 

**Service** — `approvers.service.ts`, lines 534–544:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id); // fetches by id only
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);          // deletes without ownership check
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

**Contrast with the correctly-protected `updateTransactionApprover`**, which explicitly validates the approver's root transaction matches the URL parameter before proceeding:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The `removeTransactionApprover` path has no equivalent guard. The `getCreatorsTransaction` call in the controller only validates the URL's `:transactionId`, not the approver's actual owning transaction.

## Impact Explanation
An attacker who removes all approvers from a victim's transaction eliminates the approval gate entirely. The transaction can then proceed to `WAITING_FOR_EXECUTION` and be submitted to the Hedera network without any approver having signed off. For organizations using the approval workflow to enforce governance (e.g., multi-party authorization for large HBAR transfers, account key changes, or node operations), this completely bypasses that control — a transaction executes without the required human approvals.

## Likelihood Explanation
**Attacker preconditions (no privilege required beyond a normal account):**
- Must be a registered user of the organization (standard authenticated user).
- Must have created at least one transaction (to pass `getCreatorsTransaction` on their own transaction).
- Must know a target approver's integer `id`. Approver IDs are sequential database integers; an attacker can enumerate them by creating their own approvers and observing the returned IDs, or by calling `GET /transactions/:id/approvers` on any transaction they have read access to.

The attack requires only two API calls and is fully deterministic. No race condition, no cryptographic break, no privileged role needed.

## Recommendation
In `removeTransactionApprover` (service), after fetching the approver, resolve its root node and verify that `rootNode.transactionId === transactionId` (the URL parameter), mirroring the guard already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

  // Cross-check: approver must belong to the transaction in the URL
  if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  // Re-verify creator ownership against the approver's actual transaction
  await this.getCreatorsTransaction(rootNode.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: rootNode.transactionId }]);
  return result;
}
```

Update the controller to pass `transactionId` and `user` to the service method accordingly.

## Proof of Concept

**Setup:**
- Attacker is a registered user with `userId = 2`.
- Attacker owns `transactionId = 10` (created by them).
- Victim owns `transactionId = 20`, which has an approver with `id = 55`.

**Attack:**
```
DELETE /transactions/10/approvers/55
Authorization: Bearer <attacker_jwt>
```

**Execution trace:**
1. `getCreatorsTransaction(10, attacker)` — passes, attacker owns transaction 10.
2. `removeTransactionApprover(55)` — fetches approver 55 (belongs to transaction 20), calls `removeNode(55)`, soft-deletes it. No cross-check is performed.

**Result:** Approver 55 is deleted from victim's transaction 20. If this was the last approver, the approval gate is gone and the transaction can proceed to execution without any approval.
