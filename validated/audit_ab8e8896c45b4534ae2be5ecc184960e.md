Audit Report

## Title
`cutoffAt` Signing Deadline Defined and Documented But Never Enforced Server-Side

## Summary
The `Transaction` entity defines a `cutoffAt` field explicitly documented as "The timestamp at which the transaction can no longer be signed by the signers." The field is accepted via `CreateTransactionDto`, persisted to the database, and returned in API responses. However, neither of the two server-side signature submission paths — `SignersService.uploadSignatureMaps()` nor `TransactionsService.importSignatures()` — checks whether the current time has passed `cutoffAt` before accepting and persisting a signature. The creator-defined signing deadline is entirely unenforced.

## Finding Description

**The field is defined and documented:**

`cutoffAt` is a nullable `Date` column on the `Transaction` entity: [1](#0-0) 

The database documentation explicitly states its purpose: [2](#0-1) 

It is accepted in `CreateTransactionDto`: [3](#0-2) 

And persisted during transaction creation: [4](#0-3) 

**Path 1 — `POST /transactions/:transactionId/signers` via `SignersService.uploadSignatureMaps()`:**

The controller delegates to `SignersService.uploadSignatureMaps()`: [5](#0-4) 

Inside `uploadSignatureMaps()`, per-transaction validation is performed by `validateTransactionStatus()`: [6](#0-5) 

`validateTransactionStatus()` checks only the status enum and whether the Hedera SDK transaction bytes are expired. It never reads `transaction.cutoffAt`: [7](#0-6) 

**Path 2 — `POST /transactions/signatures/import` via `TransactionsService.importSignatures()`:**

The same pattern applies — status check and `isExpired()` check, but no `cutoffAt` check: [8](#0-7) 

## Impact Explanation
A transaction creator sets `cutoffAt` to define a signing window — for example, to ensure only signatures collected before a governance deadline are counted. After `cutoffAt` passes, any signer who still has access to the transaction (i.e., it has not yet expired per the Hedera SDK's validity window, or is a long-lived/manual transaction) can submit a valid signature that will be accepted, persisted, incorporated into `transactionBytes`, counted toward the required threshold, and can cause the transaction to advance to `WAITING_FOR_EXECUTION` — all after the deadline the creator explicitly set. This violates the creator's documented access-control intent.

## Likelihood Explanation
The attack requires no privilege escalation. Any authenticated user who is a designated signer on a transaction can exploit this by simply calling `POST /transactions/:id/signers` after `cutoffAt` has passed. The transaction must still be in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status and not yet Hedera-expired. This is a realistic window: `cutoffAt` is typically set well before `validStart`, meaning the Hedera expiry has not yet triggered. The attacker needs only a valid JWT and a key registered on the transaction.

## Recommendation
Add a `cutoffAt` enforcement check in both signature submission paths, immediately after the status check and before any signature processing:

**In `SignersService.validateTransactionStatus()`** (`back-end/apps/api/src/transactions/signers/signers.service.ts`, after line 207):
```typescript
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  return ErrorCodes.TCO; // Transaction cutoff exceeded
}
```

**In `TransactionsService.importSignatures()`** (`back-end/apps/api/src/transactions/transactions.service.ts`, after line 539):
```typescript
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  throw new BadRequestException(ErrorCodes.TCO);
}
```

A corresponding error code (e.g., `TCO` — Transaction Cutoff Exceeded) should be added to the `ErrorCodes` enum. Tests should be added to `signers.service.spec.ts` and `transactions.service.spec.ts` covering the cutoff enforcement path.

## Proof of Concept
1. Creator calls `POST /transactions` with `cutoffAt` set to `T+5min` and `validStart` set to `T+30min`.
2. Wait until `T+6min` (past `cutoffAt`, but the Hedera transaction is not yet expired).
3. Signer calls `POST /transactions/:id/signers` with a valid `signatureMap`.
4. The server executes `validateTransactionStatus()`, which checks only status enum and `isExpired()` — both pass.
5. The signature is accepted, persisted as a `TransactionSigner` record, merged into `transactionBytes`, and the transaction may advance to `WAITING_FOR_EXECUTION` — all after the creator's declared deadline.

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
