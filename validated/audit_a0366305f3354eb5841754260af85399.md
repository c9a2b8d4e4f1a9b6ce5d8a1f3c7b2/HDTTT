### Title
Unbounded In-Memory Loop in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every one of them in a sequential `for` loop, calling the async `userKeysToSign` per transaction. Pagination is applied only after the full in-memory scan. As the number of active transactions grows, each API call to this endpoint becomes proportionally more expensive, eventually causing request timeouts and server resource exhaustion.

### Finding Description

**Root cause — no `take` limit on the DB query:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip` here
});
```

**Unbounded sequential async loop:** [2](#0-1) 

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

Each `userKeysToSign` call invokes `userKeysRequiredToSign` → `keysRequiredToSign`, which deserializes the transaction bytes and calls `transactionSignatureService.computeSignatureKey(transaction)` — a non-trivial async operation per transaction. [3](#0-2) 

**Pagination is applied only after the full scan:** [4](#0-3) 

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ← pagination here, too late
  page,
  size,
};
```

This is the direct analog to the Solidity `isParticipating` for-loop: instead of gas exhaustion, the cost is server CPU/memory/time, and it grows linearly with the number of active transactions.

### Impact Explanation

Any authenticated user can trigger this endpoint. If an attacker (or even normal usage) accumulates a large number of non-terminal transactions (`WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `NEW`), every call to `GET /transactions/sign` will:
- Load all of them into memory
- Perform one async `computeSignatureKey` call per transaction (CPU + possible DB I/O)
- Block the Node.js event loop for the duration

This causes severe response latency degradation and can exhaust server memory/connection pool, effectively making the endpoint unavailable for all users — a service-level DoS.

### Likelihood Explanation

The attacker only needs a valid authenticated account. Creating transactions is a normal product flow. There is no rate limit or cap on the number of active transactions visible to the endpoint. A single malicious user creating thousands of transactions in `WAITING_FOR_SIGNATURES` status is sufficient to trigger the condition. No privileged access is required.

### Recommendation

Apply the database-level `take`/`skip` pagination **before** the loop, and push the signing-eligibility check into the SQL query (as is already done in `getTransactionNodesQuery` and `getTransactionsToApprove`). The pattern used in `getTransactionsToApprove` — which uses a recursive CTE inside the WHERE clause — should be the model: [5](#0-4) 

Alternatively, pre-filter at the DB level using the user's public keys so only transactions that actually require the user's signature are returned, eliminating the need for the in-memory loop entirely.

### Proof of Concept

1. Authenticate as a normal user (User A) with a valid key pair.
2. Create N transactions (e.g., N = 10,000) in `WAITING_FOR_SIGNATURES` status, each requiring User A's key.
3. As any authenticated user, call `GET /api/transactions/sign?page=1&limit=10`.
4. Observe: the server must load all N transactions, deserialize each one, and call `computeSignatureKey` N times before returning 10 results.
5. Response time grows linearly with N; at sufficient scale the request times out and the server becomes unresponsive to other requests. [6](#0-5)

  async getTransactionsToSign(

    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,

        console.log(error);
      }
    }

      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
