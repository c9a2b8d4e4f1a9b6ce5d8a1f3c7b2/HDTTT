### Title
`isNode()` Duplicate Guard Bypassed for Threshold-Based Approver Trees, Allowing Repeated Insertion

---

### Summary

`ApproversService.isNode()` is the sole guard preventing duplicate approver nodes from being inserted into a transaction's approval tree. Its return expression unconditionally short-circuits to `false` for every threshold-based (tree) approver, regardless of whether an identical record already exists in the database. A transaction creator can therefore call `POST /transactions/:id/approvers` repeatedly with the same tree-node payload and insert it an unlimited number of times, corrupting the approval tree and breaking all downstream approval-status logic.

---

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `createTransactionApprovers` calls `isNode` before inserting each new approver: [1](#0-0) 

`isNode` queries the database for a matching record and then returns: [2](#0-1) 

The second operand — `typeof approver.userId === 'number'` — is always `false` for a threshold-based tree node (which carries `threshold` + child `approvers` but no `userId`). Therefore `isNode` always returns `false` for tree nodes, even when `count > 0` confirms an identical record already exists. The guard at line 250–251 is never triggered for tree nodes.

There is no database-level unique constraint on `transaction_approver` to serve as a backstop: [3](#0-2) 

The entity definition likewise carries no `@Unique` decorator: [4](#0-3) 

The recursive tree-building query and `getTreeStructure` both iterate over every non-deleted row for a transaction: [5](#0-4) [6](#0-5) 

Duplicate root-level tree nodes will each appear as independent roots, and duplicate child nodes will each be attached to the parent, producing a structurally invalid tree.

---

### Impact Explanation

- **Approval-status corruption**: The threshold-satisfaction check counts child approvers per parent. Duplicate parent nodes each independently demand their threshold to be met, so a transaction may never reach `WAITING_FOR_EXECUTION` even after all legitimate signers have approved.
- **Removal logic breakage**: `removeNode` adjusts or soft-deletes a parent when its child count drops below threshold. With duplicates, the child-count arithmetic is wrong, leaving orphaned or incorrectly-thresholded nodes.
- **Notification/status event spam**: `emitTransactionStatusUpdate` is called after every successful `createTransactionApprovers` invocation; repeated calls produce spurious status events. [7](#0-6) 

---

### Likelihood Explanation

The endpoint is restricted to the transaction creator: [8](#0-7) 

Any authenticated user can create a transaction and is therefore its creator. A malicious or compromised creator can exploit this without any special privilege. The API is reachable over standard HTTPS with a valid JWT, making the attack straightforward.

---

### Recommendation

Fix the return expression in `isNode` to return `true` whenever a matching record exists, regardless of whether the approver is a user-leaf or a tree-node:

```typescript
// current (broken for tree nodes):
return count > 0 && typeof approver.userId === 'number';

// fixed:
return count > 0;
```

Additionally, add a composite unique index at the database level (e.g., on `(transactionId, userId, threshold, listId)` where applicable) so the constraint is enforced even if application logic is bypassed. [9](#0-8) 

---

### Proof of Concept

1. Authenticate as any user (User A) and create a transaction → receive `transactionId = 42`.
2. `POST /transactions/42/approvers` with body:
   ```json
   {
     "approversArray": [{
       "threshold": 1,
       "approvers": [{ "userId": 99 }]
     }]
   }
   ```
3. Repeat step 2 ten times with the identical payload.
4. `GET /transactions/42/approvers` — observe ten identical root-level threshold nodes, each with a child `userId: 99`.
5. User 99 approves the transaction. The approval is written to all ten child records, but each of the ten duplicate root nodes independently evaluates its threshold. Depending on how the status-update logic aggregates roots, the transaction either never advances or advances incorrectly.
6. Attempting to remove any one duplicate root triggers `removeNode`, which soft-deletes the parent and adjusts thresholds based on a now-incorrect child count, leaving the remaining nine duplicates in an inconsistent state. [1](#0-0) [2](#0-1)

  /* Get the tree structure of the approvers */
  getTreeStructure(approvers: TransactionApprover[]): TransactionApprover[] {
    const approverMap = new Map(approvers.map(approver => [approver.id, { ...approver }]));

    approverMap.forEach(approver => {
      if (approver.listId) {
        const parentApprover = approverMap.get(approver.listId);
        if (parentApprover) {
          if (!parentApprover.approvers) {
            parentApprover.approvers = [];
          }
          parentApprover.approvers.push(approver);
        }
      }
    });

    const rootApprovers = Array.from(approverMap.values()).filter(
      approver => approver.listId === null,
    );

    return rootApprovers;
  }
