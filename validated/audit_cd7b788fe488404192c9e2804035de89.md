Audit Report

## Title
Missing Cryptographic Signature Validation in `uploadSignatureMaps` Allows Forged Signer Registration and Transaction Corruption

## Summary
The `processTransactionSignatures` function in `signers.service.ts` accepts and persists `SignatureMap` payloads without verifying that the submitted signature bytes are cryptographically valid over the transaction body. An authenticated user with any registered public key can submit a well-formed but cryptographically invalid signature, be recorded as a legitimate signer in `TransactionSigner`, and corrupt the stored `transactionBytes` — potentially causing the transaction to fail irreversibly on the Hedera network.

## Finding Description

Two distinct signature upload paths exist in the codebase, and they apply different levels of validation:

**Path 1 — `importSignatures` (safe):**

`transactions.service.ts` calls `validateSignature(sdkTransaction, map)` before invoking `addSignature`. The `validateSignature` utility iterates every `(publicKey, signature)` pair in the map and calls `publicKey.verify(bodyBytes, signature)`, throwing `'Invalid signature'` on any failure. [1](#0-0) [2](#0-1) 

**Path 2 — `uploadSignatureMaps` → `processTransactionSignatures` (vulnerable):**

`signers.service.ts` only checks whether the public key in the submitted map belongs to the authenticated user (`userKeyMap.get(raw)`), then immediately calls `sdkTransaction.addSignature(publicKey, map)` with no cryptographic check on the signature bytes. [3](#0-2) 

The Hedera SDK's `addSignature` is a byte-insertion method — it does not verify that the signature is valid over the transaction body. The `validateSignature` utility that performs `publicKey.verify(bodyBytes, signature)` is never called in this path.

The `isSameBytes` comparison at lines 262–264 does **not** protect against this. It only detects whether the transaction bytes changed after `addSignature`. If the attacker submits invalid-but-different bytes, `isSameBytes` evaluates to `false`, and the corrupted bytes are unconditionally persisted. [4](#0-3) 

The controller entry point is `POST /transactions/:transactionId?/signers`, guarded only by JWT authentication and user verification — no additional privilege is required. [5](#0-4) 

## Impact Explanation

1. **False signer registration**: The attacker is inserted into the `TransactionSigner` table as a legitimate signer despite never producing a valid cryptographic signature. This breaks the integrity of the multi-signature approval workflow. [6](#0-5) 

2. **Transaction byte corruption**: The stored `transactionBytes` are overwritten with bytes containing an invalid signature, permanently corrupting the transaction record. [7](#0-6) 

3. **Execution failure / DoS on a transaction**: If the corrupted transaction advances to `WAITING_FOR_EXECUTION` because the required signer count is met, the chain service will submit it to Hedera, which will reject it due to the invalid signature. The transaction cannot be recovered. [8](#0-7) 

## Likelihood Explanation

The attacker must be an authenticated user with at least one registered public key — a normal, non-privileged role. No special access beyond a valid JWT and a registered key is required. The attacker constructs a `SignatureMap` containing their registered public key paired with arbitrary bytes of the correct length and encoding to pass DTO validation. This is a straightforward, low-effort manipulation of a standard API call.

## Recommendation

Add a call to `validateSignature(sdkTransaction, map)` inside `processTransactionSignatures`, mirroring the pattern already used in `importSignatures`. Specifically, before calling `sdkTransaction.addSignature(publicKey, map)` at line 251 of `signers.service.ts`, invoke `validateSignature` and reject the entire request if any signature fails verification. The utility already exists and is already imported in the common library. [2](#0-1) [9](#0-8) 

## Proof of Concept

1. Authenticate as any user with a registered public key (e.g., `pubKeyHex`).
2. Obtain a transaction in `WAITING_FOR_SIGNATURES` status.
3. Construct a `SignatureMap` where the entry for `pubKeyHex` maps to 64 bytes of arbitrary data (e.g., all zeros) — valid encoding, invalid cryptographic signature.
4. `POST /transactions/{id}/signers` with the crafted payload.
5. **Observed**: The server accepts the request, calls `sdkTransaction.addSignature(publicKey, map)` without verification, `isSameBytes` is `false` (bytes changed), the corrupted `transactionBytes` are written to the database, and a new `TransactionSigner` row is inserted for the attacker's key.
6. **Expected**: The server should call `validateSignature` first, detect `publicKey.verify(bodyBytes, fakeSignature) === false`, and reject the request with an error — exactly as `importSignatures` does. [1](#0-0) [10](#0-9)

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
