Audit Report

## Title
Null Dereference in `processNotificationType` Causes Entire Signer-Reminder Batch to Silently Drop All Notifications

## Summary
`processNotificationType` in `receiver.service.ts` unconditionally dereferences the result of `entityManager.findOne()` without a null guard. When no matching `Notification` row exists, a `TypeError` is thrown, propagates uncaught through `handleSignerReminderNotifications` and out of the `processSignerReminders` loop, aborting the entire batch before any notifications are sent.

## Finding Description

In `processNotificationType`, TypeORM's `findOne` is called and may return `null`: [1](#0-0) 

Immediately after, `notification` is dereferenced without any null check at line 454: [2](#0-1) 

And again at line 478–480: [3](#0-2) 

`handleSignerReminderNotifications` calls `processNotificationType` with no try-catch: [4](#0-3) 

`processSignerReminders` wraps each iteration in a DB transaction but has no per-iteration error handling: [5](#0-4) 

The batch-send calls only execute after the loop completes successfully: [6](#0-5) 

A `TypeError` from any single iteration unwinds the entire `processSignerReminders` call stack before those lines are ever reached, discarding all accumulated notification state.

Note the contrast with `handleTransactionStatusUpdateNotifications`, which wraps its body in a try-catch and logs per-transaction errors — `handleSignerReminderNotifications` has no equivalent protection. [7](#0-6) 

## Impact Explanation

When a signer-reminder event batch contains even one transaction whose `Notification` row of the expected type does not exist, the entire batch fails. No in-app indicator, no email reminder, and no WebSocket push is delivered for any transaction in the batch — including transactions that do have valid notification records. Users required to sign transactions receive no reminder, potentially causing transactions to expire unnoticed.

## Likelihood Explanation

The `TRANSACTION_INDICATOR_SIGN` notification row is created by `processTransactionStatusUpdateNotifications`. If that earlier step failed (transient DB error, rollback, or migration), or if the row was deleted by an admin or cleanup script, the reminder path will crash. Any transaction created before the notification system was deployed would also have no pre-existing `Notification` row. Both scenarios are realistic in production.

## Recommendation

Add a null guard immediately after the `findOne` call in `processNotificationType`. If `notification` is `null`, the function should either return early with empty arrays (treating it as a no-op) or create the missing notification record before proceeding:

```typescript
if (!notification) {
  // No existing notification record; nothing to update or extend.
  return { newReceivers: [], updatedReceivers: [] };
}
```

Additionally, add per-iteration error handling in `processSignerReminders` (mirroring the try-catch pattern already used in `handleTransactionStatusUpdateNotifications`) so that a single failing transaction does not abort the entire batch:

```typescript
for (const { entityId: transactionId } of events) {
  try {
    await this.entityManager.transaction(async entityManager => {
      await this.handleSignerReminderNotifications(...);
    });
  } catch (error) {
    console.error(`Error processing signer reminder for transaction ${transactionId}:`, error);
  }
}
```

## Proof of Concept

1. Create a transaction in `WAITING_FOR_SIGNATURES` status.
2. Ensure no `Notification` row of type `TRANSACTION_INDICATOR_SIGN` exists for that transaction (e.g., delete it directly in the DB, or trigger the reminder before `processTransactionStatusUpdateNotifications` has run).
3. Trigger `remindSigners` or `remindSignersManual` with a batch that includes this transaction alongside other valid transactions.
4. `processNotificationType` calls `findOne` → returns `null`.
5. Line 454 executes `null.notificationReceivers.filter(...)` → `TypeError: Cannot read properties of null`.
6. The error propagates through `handleSignerReminderNotifications` (no catch), through the `entityManager.transaction` callback, out of the `for` loop, and unwinds `processSignerReminders`.
7. Lines 1169–1171 (`sendDeletionNotifications`, `sendInAppNotifications`, `sendEmailNotifications`) are never reached.
8. All signer-reminder notifications for every transaction in the batch are silently dropped.
