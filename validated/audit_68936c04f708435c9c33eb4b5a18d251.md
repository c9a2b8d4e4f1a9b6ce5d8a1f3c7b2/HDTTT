Audit Report

## Title
Authorization Bypass in `verifyAccess` Grants Any Authenticated User Full Read Access to All Terminal-Status Transactions

## Summary
The `verifyAccess` method in `TransactionsService` unconditionally returns `true` for any transaction whose status is EXECUTED, EXPIRED, FAILED, CANCELED, or ARCHIVED — with no check that the requesting user has any relationship to that transaction. Because `GET /transactions/:id` relies solely on this function for authorization, any authenticated user can read the full details of any other organization's completed transaction by its integer ID. A second, independent exposure exists on `GET /transactions/history`, which returns all terminal-status transactions across all users with no ownership filter.

## Finding Description

**Root cause — `verifyAccess` skips identity checks for terminal transactions:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, lines 789–798:

```typescript
if (
  [
    TransactionStatus.EXECUTED,
    TransactionStatus.EXPIRED,
    TransactionStatus.FAILED,
    TransactionStatus.CANCELED,
    TransactionStatus.ARCHIVED,
  ].includes(transaction.status)
)
  return true;   // ← no user identity check whatsoever
``` [1](#0-0) 

For non-terminal transactions the function correctly checks creator, observer, signer, and approver membership. That entire block is skipped for terminal transactions.

**`GET /transactions/:id` call chain:**

`getTransactionWithVerifiedAccess` (lines 754–763) fetches the transaction with no user filter, then calls `verifyAccess` as the sole authorization gate: [2](#0-1) 

The controller wires `GET /transactions/:id` directly to this method: [3](#0-2) 

`getTransactionById` (lines 110–147) performs no user-scoped filtering; it fetches by raw integer ID: [4](#0-3) 

**Second exposure — `GET /transactions/history` has no user filter:**

The controller handler accepts no `@GetUser()` parameter and passes no user context to the service: [5](#0-4) 

`getHistoryTransactions` in the service has no `user` parameter and applies no ownership filter — it returns all terminal-status transactions across every user and organization: [6](#0-5) 

## Impact Explanation
Any authenticated user can:

1. Call `GET /transactions/history` to enumerate all integer IDs of terminal-status transactions across every organization (no ownership filter applied).
2. Call `GET /transactions/:id` for each ID and receive the full `TransactionFullDto`, which includes `transactionBytes` (raw serialized Hedera transaction body), `publicKeys` (public keys of all required signers), `signature`, `mirrorNetwork`, creator key metadata, observer list, signer list, and approver tree.

This constitutes cross-tenant data exfiltration. In a multi-organization deployment, one organization's completed, failed, canceled, or archived transaction data — including the public keys of its members and the exact bytes of every executed Hedera transaction — is permanently readable by any other authenticated user.

## Likelihood Explanation
- **Attacker precondition:** a valid JWT for any account on the platform. No admin role, no leaked credentials, no privileged access required.
- **Effort:** trivial. The history endpoint paginates all terminal transactions; IDs are sequential integers. A single authenticated HTTP session is sufficient.
- **Detection:** the request is indistinguishable from a legitimate user viewing their own history.
- **Prevalence:** every transaction that has ever reached a terminal status is permanently exposed.

## Recommendation

**Fix `verifyAccess`:** Remove the unconditional early return for terminal statuses. Apply the same creator/observer/signer/approver membership check regardless of transaction status:

```typescript
async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
  if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

  const userKeysToSign = await this.userKeysToSign(transaction, user, true);

  return (
    userKeysToSign.length !== 0 ||
    transaction.creatorKey?.userId === user.id ||
    !!transaction.observers?.some(o => o.userId === user.id) ||
    !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
    !!transaction.approvers?.some(a => a.userId === user.id)
  );
}
```

**Fix `getHistoryTransactions`:** Add a `user: User` parameter and apply the same ownership filter used in `getTransactions` (creator, observer, signer, approver membership) so that the history endpoint only returns transactions the requesting user has a relationship to.

## Proof of Concept

```
# Step 1: Authenticate as any valid user (User A, Org A)
POST /auth/login  →  { "accessToken": "<jwt_a>" }

# Step 2: Enumerate all terminal-status transaction IDs across all organizations
GET /transactions/history?page=1&limit=100
Authorization: Bearer <jwt_a>
→ Returns paginated list of ALL terminal transactions (no ownership filter)
→ Extract integer IDs: [1, 2, 3, ..., N]

# Step 3: Fetch full details of a transaction belonging to Org B
GET /transactions/42
Authorization: Bearer <jwt_a>
→ HTTP 200 OK
→ Returns TransactionFullDto:
  {
    "id": 42,
    "transactionBytes": "<raw Hedera tx bytes>",
    "publicKeys": ["<org B member public key>", ...],
    "signature": "...",
    "mirrorNetwork": "...",
    "creatorKey": { "userId": <org B user id>, ... },
    "observers": [...],
    "signers": [...],
    "approvers": [...]
  }
```

`verifyAccess` returns `true` at line 798 because the transaction status is EXECUTED, without ever inspecting `user.id`. The full entity is returned to the attacker.
