Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Per-Request Server Resource Exhaustion

## Summary
`GET /transactions/sign` loads every non-terminal transaction from the database with no row limit, then serially calls `userKeysToSign` (which invokes `computeSignatureKey` and may trigger outbound mirror-node HTTP calls) on every row. Pagination parameters are accepted but only applied to the in-memory result slice. Any authenticated, verified user can trigger this path.

## Finding Description

**Missing `take`/`skip` in the database query**

`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` accepts `Pagination` parameters but never passes `take`/`skip` to the repository query: [1](#0-0) 

Every other list endpoint passes `skip: offset, take: limit` to the query. For example, `getTransactions`: [2](#0-1) 

And `getHistoryTransactions`: [3](#0-2) 

And `getTransactionsToApprove`: [4](#0-3) 

**Unbounded sequential async work per row**

After loading the full result set, the code iterates over every row and serially awaits `userKeysToSign`: [5](#0-4) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `keysRequiredToSign`, which calls `transactionSignatureService.computeSignatureKey`: [6](#0-5) 

`computeSignatureKey` deserializes transaction bytes and calls `accountCacheService.getAccountInfoForTransaction` for the fee-payer, signing accounts, receiver accounts, and node account: [7](#0-6) 

`getAccountInfoForTransaction` consults a local DB cache, but when the cache is stale or missing it calls `mirrorNodeClient.fetchAccountInfo`, which is an outbound HTTP call to the Hedera mirror node: [8](#0-7) 

**Pagination silently ignored at the DB level**

The `page`/`size` query parameters are accepted and destructured, but only applied to the in-memory slice after all work is done: [9](#0-8) 

**Endpoint remains publicly routed**

The controller comment explicitly notes the endpoint is no longer used by the front-end, yet it remains live and accessible to any authenticated, verified user: [10](#0-9) 

## Impact Explanation
A single authenticated request causes the server to load all non-terminal transactions from the database and serially call `computeSignatureKey` on each one. Each call deserializes transaction bytes and may make multiple outbound HTTP calls to the Hedera mirror node (fee-payer, signing accounts, receiver accounts, node account). With a large number of active transactions, this blocks the Node.js event loop for the duration of the request, starving other requests and degrading or hanging the service for all users of the organization. Because pagination is silently ignored at the database level, there is no client-side workaround.

## Likelihood Explanation
Any registered, verified user with at least one key can reach this endpoint — no admin or privileged role is required. Transaction creation is a core workflow, so a large number of active transactions is a realistic steady-state condition in production. The attacker does not need to flood the server; a single request against a sufficiently populated database is sufficient to cause a timeout or event-loop stall.

## Recommendation
1. Pass `take: limit` and `skip: offset` to the `this.repo.find(...)` call in `getTransactionsToSign`, consistent with every other paginated endpoint.
2. Because `userKeysToSign` must be called per-row to filter results, consider pre-filtering at the DB level (e.g., joining on `transaction_signer` for the user's keys) to reduce the result set before the per-row async work.
3. If the endpoint is genuinely no longer used by any client, remove it from the router entirely to eliminate the attack surface.

## Proof of Concept
1. Register and verify a user account.
2. Call `POST /transactions` N times to create N transactions with status `WAITING_FOR_SIGNATURES`.
3. Call `GET /transactions/sign?page=1&size=10`.
4. The server loads all N transactions from the database (no `take` limit), then calls `computeSignatureKey` N times sequentially, each potentially making mirror-node HTTP calls. The `size=10` parameter only slices the final in-memory array at line 313 — it has no effect on the DB load or the loop.
5. With a sufficiently large N, the request hangs the Node.js event loop for its duration, blocking all other in-flight requests.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }

  async getAccountInfoForTransaction(
    transaction: Transaction,
    account: string,
    isReceiver: boolean = false,
  ): Promise<AccountInfoParsed | null> {
    if (!transaction || !transaction.mirrorNetwork) {
      return null;
    }

    this.validateAccount(account);

    const mirrorNetwork = transaction.mirrorNetwork;

    // Get cached data
    const cached = await this.dataSource.manager.findOne(CachedAccount, {
      where: { account, mirrorNetwork },
    });

    if (this.hasCompleteData(cached) && isFresh(cached.updatedAt, this.cacheTtlMs)) {
      // Link to transaction even if using cache
      await this.linkTransactionToAccount(transaction.id, cached.id, isReceiver);
      return this.parseCachedAccount(cached);
    }

    // Cache is stale or doesn't exist - fetch new data
    this.logger.debug(`Fetching account ${account} from mirror node (cache ${cached ? 'stale' : 'missing'})`);

    // Try to claim the account for refresh, create the account row if none exists
    const { data: claimedAccount, claimed } = await this.tryClaimAccountRefresh(account, mirrorNetwork);

    if (!claimed) {
      // Link to transaction
      await this.linkTransactionToAccount(transaction.id, claimedAccount.id, isReceiver);

      if (this.hasCompleteData(claimedAccount)) {
        return this.parseCachedAccount(claimedAccount);
      }

      // No cached data
      // This should never happen
      return null;
    }

    const { data } = await this.performRefreshForClaimedAccount(claimedAccount, transaction.id, isReceiver);
    return data;
  }
