Audit Report

## Title
Any Authenticated User Can Post Comments on Any Transaction Without Being a Participant

## Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` enforces only authentication (JWT validity + verified user status) but performs no authorization check to confirm the requesting user is a participant (creator, signer, observer, or approver) of the target transaction. The developer explicitly acknowledged this gap with a `//TODO` comment. Any verified organization user can inject comments into any transaction they have no legitimate relationship with.

## Finding Description

**Root cause:** `CommentsController.createComment` delegates directly to `CommentsService.createComment` with no participant membership check. The controller applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level: [1](#0-0) 

The `@Post()` handler accepts `transactionId` from the URL and `user` from the JWT, then calls the service with no further authorization: [2](#0-1) 

The developer-left TODO on line 22 explicitly acknowledges the missing guard: [3](#0-2) 

`CommentsService.createComment` performs no ownership or membership check — it blindly creates and persists the comment: [4](#0-3) 

**Contrast with other sub-resources:** Other transaction sub-resources enforce participant-level authorization. `TransactionsService.verifyAccess` checks creator, signer, observer, and approver membership before allowing access: [5](#0-4) 

`ObserversService.getTransactionObserversByTransactionId` and `ApproversService.getVerifiedApproversByTransactionId` both throw `UnauthorizedException` when the requesting user is not a participant: [6](#0-5) [7](#0-6) 

The `comments` sub-resource is the only one that skips this pattern entirely.

## Impact Explanation
- **Unauthorized state modification:** Any verified user can inject arbitrary comments into transactions they have no legitimate relationship with, polluting the audit trail and potentially misleading approvers or signers.
- **Cross-team integrity break:** In a multi-team organization, a user from one team can comment on another team's sensitive or confidential transactions.
- **Workflow interference:** Misleading or spam comments on a pending transaction could influence approval decisions by other participants.

## Likelihood Explanation
- The attacker only needs to be a registered, verified organization user — a realistic baseline for a multi-user organization tool.
- Transaction IDs are sequential auto-incremented integers (`@PrimaryGeneratedColumn()` on `TransactionComment` and `Transaction` entities), making them trivially enumerable.
- No rate-limiting or ownership check stands in the way.
- Exploitation requires only a valid JWT and a single HTTP `POST` request. [8](#0-7) 

## Recommendation
Apply the same participant-membership check used by other sub-resources before persisting a comment. Specifically, inject `TransactionsService` (or a shared helper) into `CommentsService` and call `verifyAccess` (or `getTransactionWithVerifiedAccess`) at the start of `createComment`. If the user is not a participant, throw `UnauthorizedException`. This mirrors the pattern already enforced in `ObserversService` and `ApproversService`.

## Proof of Concept

```
# 1. Attacker registers and is verified as a normal org user, obtains JWT.
# 2. Enumerate or guess a valid transactionId (sequential integers).
# 3. POST a comment to a transaction the attacker has no role in:

POST /transactions/42/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "message": "Approve this — it looks fine." }

# 4. HTTP 201 Created — comment is persisted against transaction 42
#    with no rejection, no ownership check, no error.
```

The comment is now visible to all legitimate participants of transaction 42 (creators, signers, approvers, observers) and appears in the audit trail as if posted by a legitimate participant.

```typescript
  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
```
