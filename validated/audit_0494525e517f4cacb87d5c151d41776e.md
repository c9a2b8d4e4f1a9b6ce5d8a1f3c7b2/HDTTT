## Audit Report

## Title
Unbounded Loop with Per-Iteration Async DB Calls in `getTransactionsToSign` Causes Server Resource Exhaustion

## Summary
`TransactionsService.getTransactionsToSign` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every result performing a sequential `await this.userKeysToSign(transaction, user)` — an async operation that issues additional database queries — before applying pagination. Any authenticated user can trigger this endpoint, and as the number of active transactions grows the server performs O(N) sequential async calls per request, exhausting CPU, memory, and database connection pool resources.

## Finding Description

**Claim 1 — Unbounded `repo.find` with no `take` clause: CONFIRMED.** [1](#0-0) 

The `repo.find` call at lines 295–299 has no `take` property, meaning it loads every non-terminal transaction in the database into memory in a single query.

**Claim 2 — Sequential `await` inside the loop: CONFIRMED.** [2](#0-1) 

For every transaction returned, `await this.userKeysToSign(transaction, user)` is called sequentially at line 304. Each call delegates to `userKeysRequiredToSign` via `TransactionSignatureService`, which performs additional database queries per transaction.

**Claim 3 — Pagination applied only after all N iterations: CONFIRMED.** [3](#0-2) 

`result.slice(offset, offset + limit)` at line 313 is applied only after the full loop completes, meaning all N async DB calls are made regardless of the requested page size.

**Claim 4 — `userKeysToSign` delegates to `userKeysRequiredToSign`: CONFIRMED.** [4](#0-3) 

**Claim 5 — `whereForUser` does not bound the result set size: CONFIRMED.** [5](#0-4) 

The filter only excludes terminal-status transactions (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). There is no `take` cap, no user-scoped pre-filter, and no concurrency throttle.

**Contrast with other methods:** `getTransactions` and `getHistoryTransactions` both apply `skip`/`take` at the query level before returning results. [6](#0-5) 

## Impact Explanation
- **CPU / event-loop saturation:** N sequential `await` calls hold the NestJS event loop for the full duration of the request.
- **Database connection exhaustion:** Each `userKeysToSign` call opens additional queries; concurrent requests drain the connection pool.
- **Memory pressure:** All N transaction rows are loaded into heap before any result is returned.
- **Service unavailability:** Repeated calls from one or more users with many associated transactions can degrade or crash the API for all users.

## Likelihood Explanation
- **No elevated privilege required:** Any authenticated user can call `GET /transactions/to-sign`.
- **Attacker-controlled loop bound:** A user can be added as a signer or observer to an arbitrarily large number of transactions, directly inflating N.
- **Amplification:** The endpoint is likely polled by the frontend on page load and refresh, so a single account with many transactions causes repeated O(N) work.
- **No existing guard:** There is no rate limit, no `take` cap on the initial query, and no concurrency throttle in the service.

## Recommendation
1. **Push pagination into the query:** Apply `take` and `skip` at the `repo.find` level, not after the loop. This requires pre-filtering at the DB level (e.g., joining on `transaction_signer` for the user's keys) rather than post-filtering in application code.
2. **Batch the key-resolution query:** Replace the per-transaction `userKeysToSign` loop with a single batched query that resolves signing requirements for all transactions at once.
3. **Add a hard `take` cap:** Even before a full refactor, add a server-side maximum (e.g., `take: Math.min(limit, MAX_PAGE_SIZE)`) to the `repo.find` call to prevent unbounded loads.
4. **Rate-limit the endpoint:** Apply a per-user rate limit on `GET /transactions/to-sign`.

## Proof of Concept
1. Authenticate as user A.
2. Create or be added as a signer to a large number (e.g., 10,000) of non-terminal transactions.
3. Call `GET /transactions/to-sign?page=1&limit=10`.
4. Observe: the server executes one unbounded `SELECT` loading all 10,000 rows, then performs 10,000 sequential async DB calls before returning 10 results. CPU, memory, and DB connections spike for the full duration. Repeating the call concurrently amplifies the effect.
