Audit Report

## Title
`isApproved` Rejection Condition Uses Wrong Threshold Comparison, Causing Premature "Rejected" Display for Small-Threshold Approver Groups

## Summary
The `isApproved` function in `front-end/src/renderer/utils/sdk/index.ts` uses an incorrect mathematical condition to determine definitive rejection of a threshold approver group. When `rejections.length >= threshold`, the function returns `false` (rejected), but the correct condition for definitive rejection is `(total_approvers - rejections.length) < threshold`. For any group where `threshold < total_approvers` (e.g., 1-of-3), a single rejection causes the entire group to be displayed as definitively rejected in the UI, even though remaining approvers could still satisfy the threshold.

## Finding Description

The root cause is at line 357 of `front-end/src/renderer/utils/sdk/index.ts`:

```typescript
return rejections.length >= (approver.threshold || rejections.length) ? false : null;
``` [1](#0-0) 

**Concrete trace — 1-of-3 threshold group, Approver A rejects, B and C pending:**

- `approvals.length = 0`, `rejections.length = 1`, `approver.threshold = 1`, `approver.approvers.length = 3`
- Line 354: `0 >= (1 || 0)` → `false` — not yet approved (correct)
- Line 357: `1 >= (1 || 1)` → `true` → returns `false` — **definitively rejected (wrong)**

The correct rejection condition is impossibility of reaching the threshold:
```
(approver.approvers.length - rejections.length) < approver.threshold
// (3 - 1) < 1 → 2 < 1 → false → should return null (still pending)
```

The `ITransactionApprover` interface confirms `threshold` is an optional number with no lower bound enforced on the frontend: [2](#0-1) 

The backend explicitly validates and accepts `threshold >= 1` and `threshold <= approvers.length`, making threshold=1 with N>1 approvers a fully valid server-accepted configuration: [3](#0-2) 

The incorrect `false` return propagates into two UI components:

**`ReadOnlyApproversList.vue`** applies `bg-danger` (red badge) to the approver group item: [4](#0-3) 

**`ApproverStructureStatus.vue`** renders a red X icon (`bi-x-lg text-danger`) on the threshold group header: [5](#0-4) 

## Impact Explanation

Any organization using a threshold approver group where `threshold < total_approvers` is affected. When a single approver in such a group rejects, the UI immediately displays the entire approval group with a red badge and X icon — a terminal "rejected" visual state. Other approvers who have not yet voted see this state and are likely to conclude no further action is needed, effectively soft-locking the multi-party approval workflow. The backend has not actually rejected the transaction; the actual transaction status remains pending, but the UI presents it as rejected. This materially disrupts the core approval workflow of the platform.

## Likelihood Explanation

The actor is a legitimate member of a threshold approver group — no elevated privileges are required. The exploit requires only submitting a rejection via the standard approval endpoint with `approved: false`, which is a normal in-product action. Threshold groups with small thresholds (e.g., 1-of-N) are a natural and common configuration for organizations that want any one of several designated approvers to unblock a transaction. The backend test suite confirms threshold=1 with multiple approvers is a tested, accepted configuration: [6](#0-5) 

## Recommendation

Replace the rejection condition on line 357 with the mathematically correct impossibility check:

```typescript
// Before (incorrect):
return rejections.length >= (approver.threshold || rejections.length) ? false : null;

// After (correct):
const remaining = approver.approvers.length - rejections.length;
return remaining < (approver.threshold || approver.approvers.length) ? false : null;
```

This ensures the group is only marked as definitively rejected when it is impossible for the remaining pending approvers to satisfy the threshold.

## Proof of Concept

Given a threshold approver group with `threshold = 1` and three child approvers `[A, B, C]`:

1. Approver A submits a rejection (`approved: false`) via the normal workflow.
2. The frontend calls `isApproved` on the group node.
3. `approvals = []`, `rejections = [A]`.
4. Line 354: `0 >= 1` → `false` (not approved).
5. Line 357: `1 >= (1 || 1)` → `1 >= 1` → `true` → returns `false`.
6. `ReadOnlyApproversList.vue` renders the group with `bg-danger` class.
7. `ApproverStructureStatus.vue` renders `bi-x-lg text-danger` on the group header.
8. Approvers B and C, who have not voted, see a red "rejected" badge on the group and conclude the transaction has been rejected — even though either of them could approve it and satisfy the 1-of-3 threshold.
9. The backend transaction remains in a pending state, but the UI presents it as rejected, disrupting the approval workflow. [7](#0-6)

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
