### Title
Unbounded Full-Table Scan with Per-Row Async Work in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`GET /transactions/sign` fetches every non-terminal transaction from the database with no row limit, then calls `userKeysToSign()` asynchronously for each row before applying pagination in memory. A malicious authenticated user can inflate the active-transaction count by creating many transactions, forcing the server to load and process an unbounded dataset on every request, exhausting CPU, memory, and database connections.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues an unbounded `repo.find()` with no `take` clause:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip`
});
```

It then iterates over every returned row and awaits an async call per transaction:

```ts
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Pagination is applied only after the full iteration:

```ts
items: result.slice(offset, offset + limit),
```

`userKeysToSign` involves mirror-node queries and cryptographic key-tree resolution, making each iteration expensive. The endpoint is exposed at `GET /transactions/sign` and is guarded only by JWT authentication and `VerifiedUserGuard` — no rate limit or transaction-count cap is enforced.

The controller itself notes the endpoint is **"NO LONGER USED BY FRONT-END"** but remains fully reachable: [1](#0-0)

A secondary surface exists at `POST /transactions/signers` (`uploadSignatureMap`) and `POST /transactions/signatures/import` (`importSignatures`), both of which accept an unbounded `UploadSignatureMapDto[]` array with no `@ArrayMaxSize()` constraint on the DTO, and process every element with cryptographic validation: [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation

- **Server resource exhaustion**: Each request to `GET /transactions/sign` loads all active transactions into Node.js heap and fires one async mirror-node + crypto call per row. With thousands of active transactions, this saturates the event loop, exhausts DB connection pool slots, and spikes memory.
- **Cascading service degradation**: The API, Chain, and Notifications microservices share infrastructure. A saturated API event loop delays all other users' requests.
- **Amplification via transaction creation**: Any verified user can create transactions (`POST /transactions`), directly inflating the dataset that every subsequent `GET /transactions/sign` call must process.

### Likelihood Explanation

- Attacker precondition: a valid JWT for any verified organization user — achievable by registering a normal account.
- Attack path: register → verify → create N transactions in a loop → repeatedly call `GET /transactions/sign` to trigger the unbounded scan.
- No privileged access, no leaked secrets, no physical access required.
- The endpoint is still live despite the "NO LONGER USED BY FRONT-END" comment, meaning it receives no front-end traffic that would reveal the degradation before an attacker exploits it.

### Recommendation

1. **Push pagination into the database query** in `getTransactionsToSign`: add `take: limit` and `skip: offset` to the `repo.find()` call and remove the in-memory `result.slice()`. This requires restructuring the query so the `userKeysToSign` filter is expressed as a SQL predicate (similar to the `getTransactionsToApprove` recursive CTE pattern already used in the same file).
2. **Add `@ArrayMaxSize(N)` to `UploadSignatureMapDto[]`** inputs on both `uploadSignatureMap` and `importSignatures` endpoints to cap the number of items processed per request.
3. **Apply rate limiting** (e.g., NestJS `ThrottlerGuard`) to the `GET /transactions/sign`, `POST /transactions/signers`, and `POST /transactions/signatures/import` endpoints.
4. **Consider removing or disabling** `GET /transactions/sign` entirely if it is confirmed to be unused by the front-end.

### Proof of Concept

1. Register and verify a user account on the organization backend.
2. Using the authenticated user, create 10,000 transactions via `POST /transactions` (each with a valid `transactionBytes` and `creatorKeyId`).
3. Issue repeated `GET /transactions/sign?page=1&size=10` requests.
4. Observe: each request triggers a full DB scan of all 10,000 rows followed by 10,000 sequential `userKeysToSign()` async calls, causing response times to grow linearly with transaction count and eventually timing out or crashing the API process under concurrent load. [7](#0-6)

  /* Get the transactions that a user needs to sign */
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
