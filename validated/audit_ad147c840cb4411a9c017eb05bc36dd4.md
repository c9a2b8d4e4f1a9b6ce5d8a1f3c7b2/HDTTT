Audit Report

## Title
Threshold Value in `CreateTransactionApproverDto` Lacks Lower-Bound Validation Present on the Update Path, Allowing Negative Threshold Persistence

## Summary
The `CreateTransactionApproverDto` omits the `@Min(1)` decorator present on `UpdateTransactionApproverDto`, and the service-level guard in `createTransactionApprovers` only rejects `threshold === 0` and `threshold > approvers.length`. A negative threshold (e.g., `-1`) is truthy in JavaScript, bypasses every check, and is written to the `transaction_approver` table.

## Finding Description

**DTO asymmetry — confirmed:**

`CreateTransactionApproverDto` has no lower-bound constraint on `threshold`: [1](#0-0) 

`UpdateTransactionApproverDto` enforces `@Min(1)`: [2](#0-1) 

**Service guard is incomplete — confirmed:**

The only threshold guard in `createTransactionApprovers` is: [3](#0-2) 

For `threshold = -1`, `approvers = [userA, userB]`:
- `-1 > 2` → `false`
- `-1 === 0` → `false`

Both branches are false; the guard is bypassed entirely.

**`validateApprover` also misses negative values — confirmed:** [4](#0-3) 

The condition only rejects `threshold === 0`; `-1` passes.

**Persistence — confirmed:** [5](#0-4) 

In JavaScript, `-1` is truthy (only `0` is falsy among numbers), so `dtoApprover.threshold && dtoApprover.approvers` evaluates to the approvers array (truthy), and `threshold: -1` is written to the database.

**`isNode` duplicate-check also treats `-1` as valid:** [6](#0-5) 

Only `=== 0` is excluded; `-1` is treated as a legitimate threshold value here too.

## Impact Explanation
A persisted `threshold = -1` corrupts the approver tree permanently. Any downstream approval-counting logic of the form `approvedCount >= threshold` is trivially satisfied for any non-negative count, meaning a transaction could advance to `WAITING_FOR_EXECUTION` with zero approvals. The update path enforces `@Min(1)`, so the corrupt record cannot be corrected via `PATCH` — the `@Min(1)` decorator would reject any attempt to set `threshold` to a valid value if the current stored value is negative and the update path re-reads it. This creates an unrecoverable integrity failure for that transaction's approval workflow.

## Likelihood Explanation
The precondition is minimal: a valid authenticated session and ownership of at least one transaction. `getCreatorsTransaction` only verifies creator identity, not any elevated role. [7](#0-6) 

Any normal authenticated user can reach this code path by creating a transaction and calling `POST /transactions/:id/approvers` with a crafted body.

## Recommendation
Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
@IsNumber()
@Min(1)        // ← add this
@IsOptional()
threshold?: number;
```

Additionally, update the service-level guard to use `< 1` instead of `=== 0`:

```typescript
// approvers.service.ts line 305
(dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold < 1)
```

And update `validateApprover` similarly:

```typescript
// line 695
(approver.threshold === null || isNaN(approver.threshold) || approver.threshold < 1)
```

## Proof of Concept

```http
POST /transactions/1/approvers
Authorization: Bearer <valid_creator_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 2 },
        { "userId": 3 }
      ]
    }
  ]
}
```

Expected (buggy) result: HTTP 201, `threshold = -1` persisted in `transaction_approver`. All downstream `approvedCount >= threshold` checks pass trivially.
