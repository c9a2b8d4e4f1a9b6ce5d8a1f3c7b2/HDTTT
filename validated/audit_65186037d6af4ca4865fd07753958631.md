Audit Report

## Title
`approveTransaction` always emits `emitTransactionUpdate` instead of `emitTransactionStatusUpdate` on approval due to stale in-memory `userApprovers.approved` check

## Summary
In `approvers.service.ts`, `approveTransaction` persists `approved: dto.approved` to the database but never updates the in-memory `userApprovers` objects. The subsequent branch condition `!dto.approved || userApprovers.every(a => a.approved)` always evaluates to `false` when `dto.approved` is `true`, causing `emitTransactionUpdate` (lightweight UI ping) to fire instead of `emitTransactionStatusUpdate` (full status-update notification pipeline) on every approval action.

## Finding Description
`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` follows this sequence:

1. **Lines 553–556**: Fetch `userApprovers` from the database (pre-write snapshot).
2. **Line 563**: Guard against double-approval using `userApprovers.every(a => a.signature)` — correct, uses the persisted `signature` field.
3. **Lines 599–610**: Persist the approval via a query builder `.update(TransactionApprover).set({ approved: dto.approved })`. The in-memory `userApprovers` objects are **not mutated**.
4. **Line 614**: Branch on `!dto.approved || userApprovers.every(a => a.approved)`.

When `dto.approved = true`:
- `!dto.approved` → `false`
- `userApprovers.every(a => a.approved)` → checks pre-write in-memory values (`null`/`false`/`undefined`) → `false`
- Combined: `false || false` → `false` → `else` branch → `emitTransactionUpdate`

`emitTransactionStatusUpdate` is only reached when `dto.approved = false` (rejection), because `!dto.approved` is then `true`.

The unit test at line 1278 does not catch this because it calls `approveTransaction` with both `approved: true` and `approved: false` in the same test and uses `toHaveBeenCalledWith` (not `toHaveBeenCalledTimes`). The rejection call satisfies the assertion; the approval call's incorrect behavior goes undetected.

## Impact Explanation
`emitTransactionStatusUpdate` publishes `TRANSACTION_STATUS_UPDATE`, which triggers `processTransactionStatusUpdateNotifications` in the notifications service. This handler:
- Deletes stale in-app notification indicators
- Creates new in-app notifications (e.g., `TRANSACTION_INDICATOR_EXECUTABLE`)
- Creates email notifications (e.g., `TRANSACTION_READY_FOR_EXECUTION`)
- Sends a `STATUS_UPDATE` event to connected clients

`emitTransactionUpdate` publishes `TRANSACTION_UPDATE`, which triggers only `processTransactionUpdateNotifications` — a lightweight handler that sends a `UPDATE` client ping with no in-app or email notification creation.

Because `emitTransactionStatusUpdate` is never called on approval, every approval action silently skips the full notification pipeline. Users and the UI do not receive immediate status-update notifications when a transaction is approved. The transaction status itself is eventually corrected by the periodic scheduler (`processTransactionStatus`), so the transaction is not permanently stuck, but the immediate notification delivery on approval is broken for every invocation.

## Likelihood Explanation
This code path is exercised on every call to `approveTransaction` with `dto.approved = true`. There are no conditions under which `userApprovers.every(a => a.approved)` can return `true` at line 614 when `dto.approved = true`: the double-approval guard at line 563 ensures that any approver whose `signature` is already set throws before reaching line 614, meaning all in-memory `userApprovers` have `approved` as `null`, `false`, or `undefined` at the point of the check. The bug is deterministic and affects every approval action without exception.

## Recommendation
Update the in-memory `userApprovers` objects immediately after the database write, before the branch condition is evaluated:

```typescript
// After the dataSource.transaction block (line 610)
userApprovers.forEach(a => (a.approved = dto.approved));

// Line 614 — now reads updated in-memory values
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

Alternatively, replace the stale in-memory check with the known value of `dto.approved` directly:

```typescript
if (!dto.approved || dto.approved) { // always true — simplify to always emit status update
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
}
```

If the intent is to emit `emitTransactionStatusUpdate` only when all of the user's approver records are now approved (i.e., the user has fully approved), the correct check after the write is `dto.approved` itself (since the write sets all `userApprovers` to `dto.approved`).

Additionally, add a dedicated unit test that asserts `emitTransactionStatusUpdate` is called when `approveTransaction` is invoked with `approved: true` alone, separate from the rejection case.

## Proof of Concept
Given a transaction with one approver record for `user.id` where `approved = null` and `signature = null`:

1. Call `approveTransaction({ approved: true, ... }, transactionId, user)`.
2. `userApprovers = [{ userId: user.id, approved: null, signature: null }]` (in-memory).
3. Guard at line 563: `userApprovers.every(a => a.signature)` → `false` → no throw.
4. DB write at lines 599–610: sets `approved = true` in the database. In-memory `userApprovers[0].approved` remains `null`.
5. Line 614: `!true || [{ approved: null }].every(a => a.approved)` → `false || false` → `false`.
6. `emitTransactionUpdate` is called. `emitTransactionStatusUpdate` is **not** called.
7. The notifications service receives only a `TRANSACTION_UPDATE` event. No in-app or email notifications are created for the approval. No `STATUS_UPDATE` client event is sent.
