## Audit Report

## Title
IDOR in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller as the creator of `transactionId`, then unconditionally deletes the approver record identified by `id` — without verifying that `id` belongs to `transactionId`. Any authenticated, verified user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause:** In `approvers.controller.ts`, `removeTransactionApprover` performs authorization on `transactionId` and mutation on `id` as two independent, uncorrelated operations:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts:108-109
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no ownership check
``` [5](#0-4) 

The service-level `removeTransactionApprover` only verifies the approver record exists, then deletes it:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts:534-543
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [6](#0-5) 

There is no check that `approver.transactionId` (or its root node's `transactionId`) matches the `transactionId` the caller was authorized against.

**Contrast with `updateTransactionApprover`**, which correctly performs this cross-validation before proceeding:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts:386-391
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The delete path has no equivalent guard.

## Impact Explanation
An attacker can silently remove any approver from any transaction they do not own. This directly corrupts the multi-signature approval workflow: a transaction requiring N approvals can have its approvers stripped, potentially allowing it to proceed without the intended governance controls, or permanently corrupting the approval tree of another user's transaction. This is an unauthorized, persistent state change with direct integrity impact on the transaction approval model.

## Likelihood Explanation
The attacker only needs to be an authenticated, verified user and the creator of at least one transaction — a normal product flow. Approver IDs are sequential integers (`id: number`), making enumeration trivial. No privileged access, leaked secrets, or special timing is required.

## Recommendation
In `removeTransactionApprover` (controller), after fetching the approver by `id`, verify that its root node's `transactionId` matches the authorized `transactionId` — mirroring the pattern already used in `updateTransactionApprover`:

```typescript
// After getCreatorsTransaction(transactionId, user):
const approver = await this.approversService.getTransactionApproverById(id);
const rootNode = await this.approversService.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
await this.approversService.removeTransactionApprover(id);
```

Alternatively, push this check into the `removeTransactionApprover` service method by accepting `transactionId` as a parameter and performing the cross-validation there, consistent with how `updateTransactionApprover` handles it.

## Proof of Concept

1. Attacker (user A) creates their own transaction — `txA` (id: `10`). They are its creator.
2. Victim (user B) creates `txB` (id: `20`) with approver records. Attacker enumerates or guesses an approver ID (`approverB_id: 99`) belonging to `txB`.
3. Attacker sends:
   ```
   DELETE /transactions/10/approvers/99
   Authorization: Bearer <attacker_jwt>
   ```
4. `getCreatorsTransaction(10, attacker)` passes — attacker IS the creator of `txA`.
5. `removeTransactionApprover(99)` fetches approver 99 (which belongs to `txB`), finds it exists, and deletes it — no further check.
6. Approver 99 is permanently removed from `txB` without user B's knowledge or consent.
