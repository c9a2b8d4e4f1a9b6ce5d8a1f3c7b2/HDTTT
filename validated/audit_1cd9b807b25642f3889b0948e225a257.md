### Title
Missing `VerifiedUserGuard` on `NotificationPreferencesController` Allows Unverified Users to Access Protected Endpoints

### Summary
The `NotificationPreferencesController` is missing the `VerifiedUserGuard` that every other protected controller in the API applies. As a result, a user who has registered but not yet been verified (status `NEW`) can read and modify their notification preferences, bypassing the user-verification access boundary that the rest of the API enforces.

### Finding Description

Every protected controller in the API applies the same three-guard chain:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
```

This is consistent across `TransactionsController`, `UsersController`, `UserKeysController`, `UserKeysAllController`, `CommentsController`, `TransactionGroupsController`, and `TransactionNodesController`.

`NotificationPreferencesController` applies only two guards:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
``` [1](#0-0) 

`VerifiedUserGuard` is absent. The guard's logic is:

```typescript
return user.status === UserStatus.NONE;
``` [2](#0-1) 

`UserStatus.NONE` is the verified state; `UserStatus.NEW` is the unverified/newly-registered state. Without `VerifiedUserGuard`, any user holding a valid JWT — including one whose account has never been verified — can reach both endpoints exposed by this controller:

- `PATCH /notification-preferences` — `updatePreferences`
- `GET /notification-preferences` — `getPreferencesOrCreate` [3](#0-2) 

The reference controllers that correctly include the guard for comparison: [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation

An unverified user (status `NEW`) who possesses a valid JWT — obtained immediately after calling `POST /auth/login` before admin verification — can:

1. Read their own notification preferences (`GET /notification-preferences`).
2. Create or overwrite their own notification preferences (`PATCH /notification-preferences`).

The data accessed is scoped to the authenticated user's own preferences, so cross-user data exposure is not possible. The concrete impact is an authorization boundary bypass: the system's invariant that only verified users may interact with protected resources is violated for this controller.

### Likelihood Explanation

The attacker precondition is minimal: register an account and log in. No privileged access, no leaked credentials, and no special tooling are required. The `POST /auth/login` endpoint is reachable without verification, and the returned JWT is immediately usable against the unguarded endpoints. Any newly registered user who has not yet been approved by an admin satisfies the precondition.

### Recommendation

Add `VerifiedUserGuard` to the controller-level `@UseGuards` decorator, consistent with every other protected controller:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
``` [1](#0-0) 

### Proof of Concept

1. Register a new user via `POST /auth/signup` (admin-initiated) or any registration flow; do **not** complete verification.
2. Log in via `POST /auth/login` with the new user's credentials. Receive a JWT.
3. Send `GET /notification-preferences` with `Authorization: Bearer <jwt>`. Observe HTTP 200 and the user's preferences returned — no 403.
4. Send `PATCH /notification-preferences` with a valid body (e.g., `{"type":"TRANSACTION_CREATED","email":false,"inApp":false}`). Observe HTTP 200 and the preference written to the database.
5. Repeat step 3 against any other protected endpoint (e.g., `GET /transactions`) with the same JWT. Observe HTTP 403, confirming `VerifiedUserGuard` blocks access there but not on the notification preferences controller.
