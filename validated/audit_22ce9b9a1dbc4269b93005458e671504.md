### Title
Missing Signer Identity Validation in `importSignatures` Allows Any Authorized User to Inject Signatures from Arbitrary Keys

### Summary

The `POST /transactions/signatures/import` endpoint (`importSignatures`) validates that submitted signatures are cryptographically valid but does not verify that the public keys in the signature map belong to the authenticated user. Any user with access to a transaction (including observers) can import cryptographically valid signatures produced by keys they do not own, injecting those signatures into the stored transaction bytes without creating a proper audit trail. This is the direct analog of the external report's "insufficient account validation" class: the `uploadSignatureMaps` path enforces key ownership, but `importSignatures` does not.

### Finding Description

**Root cause — missing ownership check in `importSignatures`:**

In `TransactionsService.importSignatures`, after `verifyAccess` confirms the user has *some* relationship to the transaction, `validateSignature` is called to confirm the signatures are cryptographically valid. The returned `publicKeys` are then added directly to the transaction bytes with no check that those keys are registered to the authenticated user: [1](#0-0) 

Compare this to the `uploadSignatureMaps` path (`SignersService`), which explicitly builds a `userKeyMap` from `user.keys` and throws `ErrorCodes.PNY` if any public key in the map is not found in that set: [2](#0-1) 

The `importSignatures` path has no equivalent guard.

**Access gate is too broad:**

`verifyAccess` grants access to observers, past signers, approvers, and the creator — not just users whose keys are required to sign: [3](#0-2) 

An observer with zero signing keys passes `verifyAccess` and can reach the signature injection code.

**Entry point:** [4](#0-3) 

The endpoint is protected only by JWT authentication and `VerifiedUserGuard`; there is no `@OnlyOwnerKey` interceptor (which is present on `createTransaction`): [5](#0-4) 

**Exploit flow:**

1. Attacker registers as an observer of a target transaction (or is legitimately added as one).
2. Attacker obtains a valid signature file (`.tx2`) for that transaction — e.g., intercepted from a shared channel, or exported by another user and forwarded.
3. Attacker calls `POST /transactions/signatures/import` with the signature map containing signatures from keys they do not own.
4. `validateSignature` confirms the signatures are cryptographically valid; no ownership check follows.
5. `sdkTransaction.addSignature` embeds those signatures into `transactionBytes`, which is persisted to the database.
6. No `TransactionSigner` record is created, so the audit trail does not reflect who actually submitted the signatures.

### Impact Explanation

- **Signature injection without ownership**: An observer can advance a transaction toward its required signature threshold by importing signatures from keys they do not own, bypassing the intended workflow where only key owners submit their own signatures.
- **Audit trail corruption**: Because `importSignatures` explicitly creates no `TransactionSigner` records, the injected signatures leave no record of who submitted them, breaking the accountability model.
- **Premature execution**: If the injected signatures satisfy the threshold, the transaction may be scheduled for execution without the key owners having explicitly authorized the submission through the system.

### Likelihood Explanation

The attacker must be an authenticated, verified user with observer (or other) access to the target transaction, and must possess a cryptographically valid signature from a key they do not own. In practice, signatures are shared as files (`.tx2`) for offline signing workflows; interception or forwarding of such files is a realistic scenario in a multi-party organization context. The cryptographic barrier (needing a valid signature) is the primary limiting factor, making this medium likelihood rather than high.

### Recommendation

Add the same ownership check used in `uploadSignatureMaps` to `importSignatures`. After `validateSignature` returns the valid public keys, verify each key is registered to the authenticated user before calling `addSignature`:

```typescript
// After validateSignature returns publicKeys:
await attachKeys(user, this.entityManager);
const userKeyMap = new Map(user.keys.map(k => [k.publicKey, k]));

for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  if (!userKeyMap.has(raw) && !userKeyMap.has(publicKey.toStringDer())) {
    throw new BadRequestException(ErrorCodes.PNY);
  }
  sdkTransaction.addSignature(publicKey, map);
}
```

Alternatively, apply `@OnlyOwnerKey` at the controller level if the DTO structure permits, consistent with how `createTransaction` is guarded. [6](#0-5) 

### Proof of Concept

**Preconditions:**
- User A owns Key A (required to sign Transaction T).
- User B is an observer of Transaction T (no signing keys).
- User A exports a signature file containing Key A's signature for Transaction T.

**Steps:**
1. User B obtains the signature file (forwarded, intercepted, or shared).
2. User B calls:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <User B's JWT>
   Body: [{ "id": <T's id>, "signatureMap": <Key A's signature map> }]
   ```
3. Server response: `[{ "id": <T's id> }]` (success).
4. Transaction T's `transactionBytes` in the database now contains Key A's signature, submitted by User B.
5. No `TransactionSigner` record exists for User B or Key A from this operation.
6. If Key A was the last required signature, Transaction T may now be eligible for execution — triggered by User B, not User A.

**Expected (correct) behavior:** The server should reject the request with `ErrorCodes.PNY` because Key A is not registered to User B.
