Audit Report

## Title
Transaction Creator Can Add Themselves as Approver and Self-Approve Their Own Transaction

## Summary
`createTransactionApprovers` in `approvers.service.ts` permits the transaction creator to designate their own `userId` as an approver. `approveTransaction` then allows that same user to satisfy the approval requirement, because neither function checks whether the approver and the creator are the same person. This collapses a multi-party approval workflow into a unilateral one.

## Finding Description

**Root cause — `createTransactionApprovers`:**

`getCreatorsTransaction` is called first to confirm the caller is the creator, but the inner `createApprover` loop only verifies that the supplied `userId` belongs to an existing user: [1](#0-0) 

There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. The creator's own `userId` passes this check without issue.

**Root cause — `approveTransaction`:**

The approval path filters approvers by the calling user's ID and proceeds if any match is found: [2](#0-1) 

There is no check that `user.id !== transaction.creatorKey.userId`. The transaction object is fetched later (line 575) but only to verify status and signature validity — never to compare the approver against the creator.

**`getCreatorsTransaction` does not help:** [3](#0-2) 

This guard is only called in `createTransactionApprovers` and `updateTransactionApprover`, not in `approveTransaction`, and it only confirms the caller *is* the creator — it says nothing about whether the approver being added is someone else.

**Exploit flow:**
1. Authenticated user creates a transaction; they become `creatorKey.userId`.
2. They call `POST /transactions/:id/approvers` with `approversArray: [{ userId: <own id> }]`. `getCreatorsTransaction` passes (caller is creator); the user-existence check passes (user exists). No other check fires.
3. They call `POST /transactions/:id/approvers/approve` with a valid signature. `approveTransaction` finds them in `userApprovers` and records the approval.
4. If the threshold is 1 or they are the sole approver, the transaction advances to execution with no independent review.

## Impact Explanation
The approval workflow exists to enforce multi-party authorization before a Hedera transaction is executed on-chain. A creator who self-approves bypasses this control entirely. Transactions that move funds or modify account keys can be executed without any independent oversight, nullifying the integrity guarantee of the approval step for any transaction the attacker creates.

**Impact: Medium** — the consequence (unilateral execution of transactions that should require independent approval) is a meaningful authorization bypass, but it is limited to authenticated organization members acting on their own transactions.

## Likelihood Explanation
Any authenticated user who can create transactions can exploit this. No elevated privileges, leaked credentials, or external dependencies are required. The steps are straightforward API calls available to every organization member.

**Likelihood: Medium** — the attacker must be a registered organization user, but the attack requires no special knowledge beyond normal API usage.

## Recommendation
Add an explicit check in `createTransactionApprovers` (inside `createApprover`) that rejects any `userId` equal to the creator's `userId`:

```typescript
// After getCreatorsTransaction resolves, capture the creator's userId
// then inside createApprover:
if (dtoApprover.userId === user.id)
  throw new Error('Transaction creator cannot be added as an approver');
```

Additionally, add a symmetric guard in `approveTransaction` to reject approval attempts by the transaction creator:

```typescript
if (transaction.creatorKey?.userId === user.id)
  throw new UnauthorizedException('Transaction creator cannot approve their own transaction');
```

Both guards are needed: the first prevents the misconfiguration from being created; the second provides defense-in-depth in case an approver record already exists (e.g., added before this fix).

## Proof of Concept

```
# Step 1 – Creator adds themselves as the sole approver
POST /transactions/42/approvers
Authorization: Bearer <creator_token>
{
  "approversArray": [{ "userId": <creator_user_id> }]
}
# → 201 Created (no error, user exists check passes)

# Step 2 – Creator approves their own transaction
POST /transactions/42/approvers/approve
Authorization: Bearer <creator_token>
{
  "userKeyId": <creator_key_id>,
  "signature": "<valid_signature_bytes>",
  "approved": true
}
# → 200 OK, approval recorded, transaction advances to execution
```
