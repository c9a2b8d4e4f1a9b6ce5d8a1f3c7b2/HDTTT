### Title
Missing Cryptographic Signature Validity Check in `processTransactionSignatures` Allows Invalid Signatures to Be Stored and Counted

### Summary
In `signers.service.ts`, the `processTransactionSignatures` method accepts a user-submitted `SignatureMap`, verifies that each public key belongs to the authenticated user, and then calls `sdkTransaction.addSignature(publicKey, map)` — but never cryptographically verifies that the signature bytes are actually valid for the transaction body. This is the direct analog of the external report's "signatory not checked" pattern: the recovered/verified identity is confirmed, but the cryptographic proof itself is not validated. Invalid signatures are stored in the database and counted toward the signing threshold, causing the transaction to fail on the Hedera network when submitted.

### Finding Description

**Root cause — `processTransactionSignatures` in `signers.service.ts`:** [1](#0-0) 

The method iterates over the submitted `SignatureMap`. For each public key it:
1. Checks the key belongs to the authenticated user via `userKeyMap` (lines 244–248) — this is the only check performed.
2. Calls `sdkTransaction.addSignature(publicKey, map)` (line 251) — the Hedera SDK's `addSignature` does **not** internally verify the signature bytes; it simply attaches them to the transaction.
3. Records the user as a signer and updates the stored transaction bytes. [2](#0-1) 

**The missing check — `validateSignature` exists but is never called here:**

A `validateSignature` function is defined in `transaction.ts` and performs the required cryptographic check (`publicKey.verify(bodyBytes, signature)`): [3](#0-2) 

This function is used elsewhere in the codebase (e.g., `transactions.service.ts`, `is-signature-map.decorator.ts`) but is **absent from the signature upload path** in `signers.service.ts`. A DTO-level decorator cannot perform this check because it has no access to the stored transaction bytes needed for `publicKey.verify(bodyBytes, signature)`.

**Exploit flow:**
1. Attacker (authenticated user) owns key `K` registered in the system.
2. Attacker crafts a `SignatureMap` containing key `K` paired with arbitrary/garbage signature bytes.
3. `POST /transactions/:id/signers` is called. The service confirms `K` belongs to the user and calls `addSignature` without verifying the bytes.
4. The invalid signature is stored in `transactionBytes`; a `TransactionSigner` record is inserted marking the user as having signed.
5. `processTransactionStatus` counts this signer toward the required threshold.
6. If threshold is met, the transaction is scheduled for submission to Hedera, which rejects it due to the invalid signature. [4](#0-3) 

### Impact Explanation
A malicious (but authenticated) user can deliberately submit invalid signature bytes for their own key. This causes:
- The transaction to be permanently poisoned with an invalid signature in the stored bytes.
- The signer record to be inserted, counting toward the threshold.
- The transaction to fail on the Hedera network upon submission, resulting in wasted fees and a failed/stuck multi-sig transaction.
- Other legitimate signers' valid signatures may be stripped or overwritten during `smartCollate` if the transaction exceeds size limits, compounding the damage. [5](#0-4) 

### Likelihood Explanation
Any authenticated user with at least one registered key can trigger this. No privileged access is required. The attacker only needs to construct a `SignatureMap` with their own public key and random bytes as the signature — a trivial operation using the Hedera SDK client-side. The endpoint is reachable via a standard authenticated `POST` request. [6](#0-5) 

### Recommendation
Call `validateSignature` (or equivalent per-entry `publicKey.verify(bodyBytes, signature)`) inside `processTransactionSignatures` **before** calling `addSignature`, using the transaction body bytes already available from `SDKTransaction.fromBytes(transaction.transactionBytes)`. Reject the entire upload if any signature fails verification.

```typescript
// Inside processTransactionSignatures, before addSignature:
const signatureValid = publicKey.verify(bodyBytes, signatureBytes);
if (!signatureValid) throw new Error(ErrorCodes.INVALID_SIGNATURE);
``` [7](#0-6) 

### Proof of Concept

**Preconditions:** Attacker has an account with a registered key pair (`publicKey`, `privateKey`) in the system.

1. Fetch the target transaction bytes via `GET /transactions/:id`.
2. Build a `SignatureMap` using the Hedera SDK with `publicKey` mapped to 64 bytes of `0xFF` (invalid signature).
3. Submit `POST /transactions/:id/signers` with the crafted map.
4. Observe: HTTP 201 returned; `TransactionSigner` record inserted for the attacker's key; `transactionBytes` updated to include the garbage signature.
5. Once the signing threshold is reached (counting the attacker's invalid entry), the chain service submits the transaction to Hedera.
6. Hedera returns `INVALID_SIGNATURE` status; the transaction fails permanently. [8](#0-7)

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};

export async function smartCollate(
  transaction: Transaction,
  requiredKeys: KeyList,
): Promise<SDKTransaction | null> {
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // HIP-1300: isTransactionOverMaxSize is fee-payer aware. Privileged payers
  // (0.0.2, 0.0.42-0.0.799) can carry up to 128 KB, so transactions under that
  // limit pass through here untouched and signatures are NOT stripped.
  if (await isTransactionOverMaxSize(sdkTransaction)) {
    const publicKeys = computeShortenedPublicKeyList(
      sdkTransaction._signerPublicKeys,
      requiredKeys,
    );

    const signatureMap = sdkTransaction.getSignatures();
    sdkTransaction.removeAllSignatures();

    for (const key of publicKeys) {
      sdkTransaction.addSignature(key, signatureMap);
    }

    // If the transaction is still too large,
    // set it to failed with the TRANSACTION_OVERSIZE status code
    // update the transaction, emit the event, and delete the timeout
    if (await isTransactionOverMaxSize(sdkTransaction)) {
      return null;
    }
  }

  return sdkTransaction;
}
