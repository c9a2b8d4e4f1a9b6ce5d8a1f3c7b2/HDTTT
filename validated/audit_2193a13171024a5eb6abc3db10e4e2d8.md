Audit Report

## Title
`hasValidSignatureKey` Incorrectly Returns `true` for Empty `KeyList`, Bypassing Signature Validation

## Summary

`hasValidSignatureKey` in `back-end/libs/common/src/utils/sdk/key.ts` returns `true` when passed an empty `KeyList` because the `forEach` loop is never entered and the final comparison `0 >= (null || 0)` evaluates to `true`. `computeSignatureKey` in `TransactionSignatureService` silently produces an empty `KeyList` when all mirror-node key lookups fail. Two downstream callers — `processTransactionStatus` and `getValidatedSDKTransaction` — treat the `true` return as proof of sufficient signatures, allowing unsigned transactions to be promoted to `WAITING_FOR_EXECUTION` and submitted to the Hedera network.

## Finding Description

**Root cause — `hasValidSignatureKey`:**

The function at lines 31–46 of `back-end/libs/common/src/utils/sdk/key.ts`:

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {          // never entered when keys.length === 0
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
    //     0               >= (null/undefined || 0)
    //     0               >= 0   →  true
  }
  ...
};
``` [1](#0-0) 

When `keys` is empty: `key.threshold` is `null`/`undefined` (no threshold was set on `new KeyList()`), `keys.length` is `0`, so the expression becomes `0 >= 0` → **`true`**, regardless of whether any public key was validated.

---

**How `signatureKey` becomes an empty `KeyList`:**

`computeSignatureKey` initialises `signatureKey = new KeyList()` and conditionally pushes keys into it. Every key-addition path silently swallows errors:

- `addFeePayerKey` (lines 82–99): wraps the mirror-node lookup in `try/catch` and returns without pushing if the lookup throws or returns no key.
- `addSigningAccountKeys` (lines 104–122): per-account `try/catch`, skips on error.
- `addReceiverAccountKeys` (lines 127–147): per-account `try/catch`, skips on error.
- `addNodeKeys` (lines 174–272): returns early if `!nodeInfo` or `!nodeInfo.admin_key`.
- `requirements.newKeys` may be an empty array. [2](#0-1) [3](#0-2) [4](#0-3) 

If the mirror node is unavailable (or returns no key for the fee-payer account), all `push` calls are skipped and `computeSignatureKey` returns an empty `KeyList`.

---

**Downstream impact — two call sites:**

**1. `processTransactionStatus`** (`back-end/libs/common/src/utils/transaction/index.ts`, lines 133–144):

```typescript
const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey          // ← may be empty KeyList → always true
);

if (isAbleToSign) {
  const collatedTx = await smartCollate(transaction, signatureKey);
  if (collatedTx !== null) {
    newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
  }
}
``` [5](#0-4) 

`smartCollate` with an empty `KeyList` and a normal-sized transaction returns the `sdkTransaction` directly (the size-check branch is not entered), so `collatedTx !== null` is also `true`. The transaction is promoted to `WAITING_FOR_EXECUTION` without any real signature check. This is confirmed by the existing test at line 403–407 of `back-end/libs/common/src/utils/sdk/privileged-payer.spec.ts`, which asserts that `smartCollate(entity, new KeyList())` returns non-null for a small transaction. [6](#0-5) 

**2. `getValidatedSDKTransaction`** (`back-end/libs/common/src/execute/execute.service.ts`, lines 216–220):

```typescript
const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);

if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
// ← never throws when signatureKey is empty KeyList
``` [7](#0-6) 

The guard that is supposed to block execution of improperly signed transactions is silently bypassed.

## Impact Explanation

An unsigned (or insufficiently signed) transaction can be submitted to the Hedera network. For high-value operations — account key rotations, node updates, large HBAR transfers — this means the network receives a transaction that should have been blocked at the application layer. The Hedera network will reject it with `INVALID_SIGNATURE`, but the application will have already: (1) promoted the transaction to `WAITING_FOR_EXECUTION`, (2) consumed transaction fees on the failed submission, and (3) emitted misleading status events to downstream consumers. The application's own signature-enforcement layer is completely defeated.

## Likelihood Explanation

The trigger condition — an empty `KeyList` from `computeSignatureKey` — is realistic:

- A transient mirror-node outage causes `addFeePayerKey` to throw, which is silently caught and the key is not pushed.
- Certain transaction types (e.g., a simple HBAR transfer where the receiver does not require a signature and there are no new keys) produce a very small key list; if the fee-payer lookup also fails, the list is empty.
- The chain service's scheduler calls `processTransactionStatus` on a timer, so every polling cycle during a mirror-node degradation is a trigger opportunity. [8](#0-7) 

## Recommendation

**Fix `hasValidSignatureKey`:** Treat an empty `KeyList` as unsatisfied (return `false`) rather than trivially satisfied. Add an explicit guard:

```typescript
if (key instanceof KeyList) {
  const keys = key.toArray();
  if (keys.length === 0) return false; // empty KeyList can never be satisfied
  let currentThreshold = 0;
  keys.forEach(key => {
    if (hasValidSignatureKey(publicKeys, key)) currentThreshold++;
  });
  return currentThreshold >= (key.threshold || keys.length);
}
```

**Fix `computeSignatureKey`:** After building the `KeyList`, verify it is non-empty before returning. If it is empty (all lookups failed), either throw an error or return a sentinel that callers treat as "unable to determine required keys — do not proceed."

**Fix callers:** Both `processTransactionStatus` and `getValidatedSDKTransaction` should treat a `null`/empty signature key result as an error condition, not as a pass.

## Proof of Concept

1. Configure the mirror-node client to be unreachable (e.g., point it at a non-existent host).
2. Create and persist a `Transaction` entity with `status = WAITING_FOR_SIGNATURES` and zero signatures on the `transactionBytes`.
3. Trigger `processTransactionStatus` (e.g., via the scheduler's polling cycle).
4. `computeSignatureKey` calls `addFeePayerKey`, which throws; the catch block logs and returns without pushing. All other add-key paths similarly produce nothing. The returned `KeyList` is empty.
5. `hasValidSignatureKey([], emptyKeyList)` evaluates `0 >= (null || 0)` → `true`.
6. `smartCollate(transaction, emptyKeyList)` returns the `sdkTransaction` (transaction is under max size).
7. The transaction's status is updated to `WAITING_FOR_EXECUTION`.
8. When `executeTransaction` is called, `getValidatedSDKTransaction` calls `hasValidSignatureKey` again with the same empty `KeyList` → `true` → no exception thrown.
9. The unsigned transaction is submitted to the Hedera network via `sdkTransaction.execute(client)`. [9](#0-8) [5](#0-4) [7](#0-6)

export async function smartCollate(
  transaction: Transaction,
  requiredKeys: KeyList,
): Promise<SDKTransaction | null> {
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // HIP-1300: isTransactionOverMaxSize is fee-payer aware. Privileged payers
  // (0.0.2, 0.0.42-0.0.799) can carry up to 128 KB, so transactions under that
  // limit pass through here untouched and signatures are NOT stripped.
  if (await isTransactionOverMaxSize(sdkTransaction)) {
    const publicKeys = computeShortenedPublicKeyList(
      sdkTransaction._signerPublicKeys,
      requiredKeys,
    );

    const signatureMap = sdkTransaction.getSignatures();
    sdkTransaction.removeAllSignatures();

    for (const key of publicKeys) {
      sdkTransaction.addSignature(key, signatureMap);
    }

    // If the transaction is still too large,
    // set it to failed with the TRANSACTION_OVERSIZE status code
    // update the transaction, emit the event, and delete the timeout
    if (await isTransactionOverMaxSize(sdkTransaction)) {
      return null;
    }
  }

  return sdkTransaction;
}
