# Audit Report

## Title
Plaintext Recovery Phrase Retained in Pinia Store Memory for Entire Session and Written Uncleared to Clipboard

## Summary
The `RecoveryPhrase` object stored in the global Pinia `storeUser` contains the full plaintext mnemonic word array (`words: string[]`) and the raw `Mnemonic` SDK object. Once set via `setRecoveryPhrase()`, this data persists in the Electron renderer process heap for the entire user session. The `Generate.vue` flow does not clear it after completing key generation. Additionally, the "Copy" button writes the full phrase to the system clipboard without ever clearing it. An attacker with local code execution on the same machine (e.g., malware running as the same OS user) can dump the V8 heap of the renderer process or read the clipboard to extract the plaintext mnemonic.

## Finding Description

**Root cause — plaintext mnemonic stored in Pinia reactive state:**

`createRecoveryPhrase()` in `userStoreHelpers.ts` constructs the object storing both the `Mnemonic` SDK object and the raw `words` array: [1](#0-0) 

The result is assigned to the Pinia store's reactive ref `recoveryPhrase`: [2](#0-1) 

via `setRecoveryPhrase()`: [3](#0-2) 

and exported globally from the store: [4](#0-3) 

---

**Code paths that do NOT clear `recoveryPhrase` after completing their operation:**

**1. `Generate.vue` — `handleGenerate()`**: After the user completes account setup (generates + verifies the phrase), the function calls `props.handleNext()` and navigates forward without clearing `user.recoveryPhrase`. The phrase remains in the store for the rest of the session. [5](#0-4) 

**2. `Generate.vue` — `correctWords` ref**: The full mnemonic is also held in a component-level `correctWords` ref that is set during verification and never zeroed on unmount. [6](#0-5) 

**3. `MigrateRecoveryPhraseHash.vue` — `handleContinue()`**: The component clears the phrase on mount (`onMounted` calls `setRecoveryPhrase(null)`), but after the user enters their phrase (triggering the watcher at line 131 which calls `handleVerify`), the phrase is set again. `handleContinue` then navigates to `transactions` without clearing `user.recoveryPhrase`. [7](#0-6) 

---

**Clipboard exposure — `handleCopyRecoveryPhrase()`:**

The "Copy" button in `Generate.vue` writes the entire 24-word phrase to the system clipboard as a comma-separated string. The clipboard is never cleared afterward. [8](#0-7) 

A second clipboard write path exists in `Summary.vue` (migration flow): [9](#0-8) 

---

**`logout()` does clear `recoveryPhrase`**, but only on explicit logout — not after completing individual key operations: [10](#0-9) 

## Impact Explanation

The Electron renderer process is a Chromium process with an inspectable V8 heap. Any process on the same machine with sufficient OS-level permissions (e.g., malware, a compromised dependency, or a process running as the same OS user) can:

1. Attach to the renderer process via the V8 inspector or OS memory APIs.
2. Dump the heap and search for the 24-word BIP-39 phrase stored in the reactive `recoveryPhrase.value.words` array.
3. Use the recovered mnemonic to derive all Ed25519/ECDSA private keys and drain any associated Hedera accounts.

The clipboard vector is even simpler: any process or browser tab can read the clipboard after the user clicks "Copy," with no memory forensics required. The phrase is retained in memory from the moment it is set until the user explicitly logs out, maximizing the window of exposure.

**Impact: Critical** — full, permanent loss of all keys derived from the exposed mnemonic.

## Likelihood Explanation

This is a desktop Electron application targeting Hedera Council members managing high-value accounts. The threat of malware on a developer or operator workstation is realistic and well-documented. The attacker does not need to compromise the application itself — only the OS user session. The clipboard vector requires no special privileges at all. Note that the SECURITY.md excludes "attacks requiring physical access to the victim device," but local code execution (malware) does not require physical access and is therefore not excluded.

**Likelihood: Medium** — requires local code execution or clipboard access, but no application-level privileges.

## Recommendation

1. **Clear `recoveryPhrase` immediately after use**: In `Generate.vue`'s `handleGenerate()`, call `user.setRecoveryPhrase(null)` before or immediately after `props.handleNext()`. Similarly, in `MigrateRecoveryPhraseHash.vue`'s `handleContinue()`, call `user.setRecoveryPhrase(null)` before `router.push`.
2. **Zero out `correctWords`**: In `Generate.vue`, set `correctWords.value = Array(24).fill('')` in an `onBeforeUnmount` hook.
3. **Clear the clipboard after a timeout**: After writing to the clipboard in `handleCopyRecoveryPhrase()`, schedule a `setTimeout` (e.g., 30–60 seconds) to overwrite the clipboard with an empty string via `navigator.clipboard.writeText('')`.
4. **Minimize in-memory lifetime**: Consider not storing the full `words` array in the Pinia store at all — store only the `hash` and derive keys on demand with a short-lived local variable.

## Proof of Concept

1. Launch the application and complete the account setup flow through `Generate.vue` (generate + verify a 24-word phrase, click "Next").
2. Using a script running as the same OS user, attach to the Electron renderer process via `--inspect` or OS memory APIs and search the V8 heap for the string array stored in `recoveryPhrase.value.words`. The 24-word array will be present for the entire session.
3. Alternatively, click the "Copy" button in `Generate.vue`. From any other process or browser tab running as the same OS user, call `navigator.clipboard.readText()` or the OS clipboard API. The full comma-separated mnemonic will be returned indefinitely until the clipboard is overwritten. [11](#0-10) [2](#0-1)
