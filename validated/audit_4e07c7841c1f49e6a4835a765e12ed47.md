Audit Report

## Title
`isNode()` Uniqueness Check Bypassed for Threshold Approver Nodes, Allowing Single-User Approval to Satisfy Multiple Threshold Requirements

## Summary
`ApproversService.isNode()` unconditionally returns `false` for any approver that lacks a `userId` (i.e., every threshold/tree node). Because the duplicate-detection guard in `createTransactionApprovers` relies solely on `isNode()`, a transaction creator can insert structurally identical threshold nodes. When `approveTransaction()` runs, it bulk-updates every row matching the approving user's ID across the entire tree, causing a single approval action to simultaneously satisfy all duplicate threshold nodes.

## Finding Description

**Root cause — `isNode()` never detects duplicate threshold nodes**

In `approvers.service.ts` at line 665:

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';
```

For a threshold node, `approver.userId` is `undefined`, so `typeof approver.userId === 'number'` evaluates to `false`. The function returns `false` regardless of `count`, meaning the guard at line 250 never fires for threshold nodes:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
``` [1](#0-0) 

**No database-level unique constraint exists** on `transaction_approver` for `(transactionId, threshold, listId)`. The entity only defines non-unique performance indexes: [2](#0-1) 

**Exploit path — creating duplicate threshold nodes**

A creator POSTs to `POST /transactions/:id/approvers` with:
```json
{
  "approversArray": [
    { "threshold": 1, "approvers": [{ "userId": 2 }] },
    { "threshold": 1, "approvers": [{ "userId": 2 }] }
  ]
}
```

`createTransactionApprovers` iterates `dto.approversArray` and calls `createApprover` for each entry: [3](#0-2) 

For each threshold node, `isNode()` returns `false` (userId is undefined), so both are inserted. The recursive child creation assigns each child a distinct `listId` (the ID of its parent threshold node): [4](#0-3) 

When `isNode()` is called for the child nodes (userId=2), the query filters on `listId` — since the two children have different `listId` values (one per duplicate threshold node), the count is 0 for each, so `isNode()` returns `false` for both children as well. All four rows are inserted successfully.

**Exploit path — single approval satisfies both nodes**

`approveTransaction` collects every row where `userId === user.id` across the entire approver tree: [5](#0-4) 

It then bulk-updates **all** of those rows in one query: [6](#0-5) 

Because User 2 appears as a child under both duplicate threshold nodes, a single call to `/approve` marks both child rows as signed. Both threshold nodes (each requiring 1-of-1) are now satisfied. If the root structure requires 2-of-2 (both threshold nodes), the transaction advances as if two independent approvals were received, when only one user acted.

## Impact Explanation
The Hedera Transaction Tool is used in Organization Mode to enforce multi-party authorization before Hedera network transactions (HBAR transfers, account operations, etc.) are submitted. A malicious creator can construct a deceptive approver tree that visually appears to require N independent approvals but structurally requires only 1. This allows the creator to execute Hedera transactions — which may move real HBAR or tokens — without obtaining the number of independent approvals the organization intended. This is an unauthorized state change and potential unauthorized movement of on-chain assets.

## Likelihood Explanation
- The attacker is the transaction creator, a role available to any authenticated organization user — no admin or privileged key is required.
- The exploit is a straightforward API call with a crafted JSON body; no timing, race condition, or cryptographic weakness is needed.
- There is no client-side or server-side control that prevents the duplicate payload from being submitted.
- The `isNode()` flaw is present in the current codebase with no compensating database-level unique constraint on threshold nodes.

## Recommendation
Fix `isNode()` to return `true` whenever a matching record exists, regardless of whether the approver is a user node or a threshold node:

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0; // remove the `&& typeof approver.userId === 'number'` condition
```

Additionally, add a database-level unique constraint on `(transactionId, threshold, listId)` in the `TransactionApprover` entity as a defense-in-depth measure to prevent duplicate threshold nodes at the persistence layer.

## Proof of Concept

1. Authenticate as a transaction creator (any org user with creator role).
2. Create a transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status.
3. POST to `/transactions/:id/approvers`:
   ```json
   {
     "approversArray": [
       { "threshold": 1, "approvers": [{ "userId": 2 }] },
       { "threshold": 1, "approvers": [{ "userId": 2 }] }
     ]
   }
   ```
4. Observe that both threshold nodes and both child rows for User 2 are inserted (no error thrown).
5. As User 2, POST to `/transactions/:id/approvers/approve` with a valid signature.
6. Observe that both child rows are marked as signed in a single request.
7. Both threshold nodes are now satisfied — the transaction advances as if two independent approvals were received, with only one user having acted.
