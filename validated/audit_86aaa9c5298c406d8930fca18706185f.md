## Audit Report

## Title
Unauthorized Deletion of Transaction Approvers via IDOR in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver belongs to that transaction. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause — controller** (`approvers.controller.ts` lines 102–113):

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ✓ checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ✗ deletes approver `id` with no cross-check
  return true;
}
``` [1](#0-0) 

**Root cause — service** (`approvers.service.ts` lines 534–544):

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  // approver.transactionId is NEVER compared to the URL's transactionId
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

The notification emits against `approver.transactionId` (the victim's transaction), not the attacker's `transactionId`, confirming the two values are never reconciled.

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-check (`approvers.service.ts` lines 386–391):

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);

/* Verifies that the root transaction is the same as the param */
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The `update` path has this guard; the `delete` path does not.

**Attack path:**
1. Attacker creates any transaction (Transaction A) — they are now its creator.
2. Attacker enumerates or guesses the approver `id` belonging to Transaction B (owned by a victim). Approver IDs are sequential integers.
3. Attacker calls `DELETE /transactions/{A}/approvers/{approver_id_from_B}`.
4. `getCreatorsTransaction(A, attacker)` passes — attacker owns A.
5. `removeTransactionApprover(approver_id_from_B)` executes — the approver (and its entire subtree via `removeNode`) is soft-deleted from Transaction B.

## Impact Explanation
An attacker can silently remove any approver (or entire approval subtrees) from any transaction they do not own. This undermines the multi-signature approval workflow that is the core security guarantee of the platform. A transaction that required N-of-M approvals can have its approvers stripped, potentially allowing it to proceed to execution without the intended governance controls. The `removeNode` function recursively soft-deletes the entire approver subtree, so a single call can wipe out a complex threshold approval tree. [5](#0-4) 

## Likelihood Explanation
- Requires only a valid authenticated session and having created at least one transaction — a low bar for any organization member.
- Approver IDs are sequential integers, making enumeration trivial.
- No rate limiting or anomaly detection is described for this endpoint.
- The attack is silent — the victim transaction's creator receives a status update notification but may not immediately notice an approver was removed. [6](#0-5) 

## Recommendation
Apply the same ownership cross-check that `updateTransactionApprover` already performs. In `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and verify its `transactionId` matches the URL's `transactionId`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Root transaction is not the same');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover`, mirroring the pattern already used by `updateTransactionApprover`. [7](#0-6) 

## Proof of Concept

**Setup:**
- Victim creates Transaction B (id=2) with approver id=99 requiring 2-of-3 signatures.
- Attacker creates Transaction A (id=1) — attacker is now its creator.

**Exploit request:**
```http
DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>
```

**Server execution trace:**
1. `getCreatorsTransaction(1, attacker)` → passes (attacker owns transaction 1).
2. `removeTransactionApprover(99)` → fetches approver 99 (belongs to transaction 2), calls `removeNode(99)`, soft-deletes the entire subtree. No check that `approver.transactionId (2) === transactionId (1)`.
3. Response: `true`.

**Result:** Approver 99 (and any child approvers) is permanently soft-deleted from Transaction B, which the attacker does not own, bypassing the multi-signature governance requirement. [2](#0-1)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,

    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],

      : this.dataSource.manager.findOne(Transaction, find));
