### Title
Organization Members Cannot Self-Remove — Permanent Lock Without Admin Cooperation

### Summary
The `DELETE /users/:id` endpoint in the backend API is exclusively gated behind `AdminGuard`, meaning only administrators can remove users from the organization. Regular users have no endpoint to remove themselves. This is a direct analog to the external report: just as DAO members cannot burn their membership tokens without `EXTERNAL_CALLER`, organization members cannot exit without an admin acting on their behalf.

### Finding Description
In `back-end/apps/api/src/users/users.controller.ts`, the only user-removal route is:

```ts
@UseGuards(AdminGuard)
@Delete('/:id')
removeUser(@GetUser() user: User, @Param('id', ParseIntPipe) id: number): Promise<boolean> {
  if (user.id === id) throw new BadRequestException(ErrorCodes.CRYFO);
  return this.usersService.removeUser(id);
}
``` [1](#0-0) 

`AdminGuard` passes only when `user.admin === true`:

```ts
canActivate(context: ExecutionContext) {
  const { user } = context.switchToHttp().getRequest();
  return user && user.admin;
}
``` [2](#0-1) 

There is no alternative self-removal route anywhere in the controller or auth module. The E2E test suite explicitly confirms the 403 response for non-admins:

```ts
it('(DELETE) should not remove a user if not admin', async () => {
  await endpoint.delete('2', userAuthToken).expect(403);
});
``` [3](#0-2) 

Additionally, the controller explicitly blocks even admins from removing themselves (`if (user.id === id) throw BadRequestException`), meaning the last admin in an organization with no other admins is also permanently locked in with no removal path at all. [4](#0-3) 

The underlying `removeUser` service performs a soft-delete of the user and all their associated `UserKey` records: [5](#0-4) 

This logic is correct and complete — it is simply never reachable by the user themselves.

### Impact Explanation
A regular user who wishes to leave the organization has no API path to do so. Their email address, public keys, and all associated metadata remain on the organization server indefinitely. The user's public keys stay visible to every other organization member via the contact list. If the admin is unavailable, unresponsive, or adversarial, the user's state is permanently locked — a direct match to the "permanent lock/freeze of user state" impact category in the RESEARCHER playbook.

### Likelihood Explanation
The precondition is simply being a non-admin authenticated user — no special setup required. Any user who wants to leave the organization will encounter this barrier. The only escape is admin cooperation, which introduces a dependency on a trusted third party that may not always be available or willing.

### Recommendation
Add a self-removal endpoint that allows an authenticated user to delete their own account, bypassing `AdminGuard` but scoped strictly to `user.id === requestingUser.id`. Example:

```ts
@Delete('/me')
removeSelf(@GetUser() user: User): Promise<boolean> {
  return this.usersService.removeUser(user.id);
}
```

This mirrors the pattern already used for `GET /me` (which is accessible without `AdminGuard`) and gives users a self-service exit path without granting any additional privilege.

### Proof of Concept
1. Register as a non-admin user and obtain a JWT (`userAuthToken`).
2. Send `DELETE /users/<own-id>` with `Authorization: Bearer <userAuthToken>`.
3. Observe HTTP 403 — the `AdminGuard` rejects the request.
4. There is no alternative endpoint to call. The user cannot leave the organization without an admin issuing `DELETE /users/<id>` on their behalf.
5. Confirm with the existing test: `(DELETE) should not remove a user if not admin → 403`. [3](#0-2)

  async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });

    if (!user) {
      throw new BadRequestException(ErrorCodes.UNF);
    }

    // Soft-delete all user keys first
    await this.repo.manager.softDelete(UserKey, { userId: id });

    // Then soft-delete the user
    await this.repo.softRemove(user);

    return true;
  }
