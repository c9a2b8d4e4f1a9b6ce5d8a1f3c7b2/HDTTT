### Title
Any Authenticated User Can Post Comments on Transactions They Have No Access To Due to Missing Authorization Check in `CommentsController`

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` enforces only authentication (JWT validity) but performs no authorization check to verify the requesting user has any relationship to the target transaction. Any verified user on the organization server can write comments on — and read comments from — any transaction, regardless of whether they are the creator, observer, signer, or approver. The developer explicitly acknowledged this gap with a `//TODO` comment in the source.

### Finding Description
The vulnerability class is **authorization bypass**: a request is accepted and serviced without verifying the requester has any legitimate relationship to the resource being acted upon. This is the direct analog to the external report's missing attribution field on `StoreChunksRequest`.

**Root cause — controller layer:**

`CommentsController` applies only three guards at the class level: `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. These confirm the JWT is valid and not blacklisted, and that the user account is verified. They say nothing about whether the user has any relationship to the transaction. [1](#0-0) 

The developer left an explicit acknowledgment of the missing check: [2](#0-1) 

**Root cause — service layer:**

`CommentsService.createComment()` performs zero access control. It creates the entity from the DTO, assigns the transaction ID and user, and immediately persists it. [3](#0-2) 

**Contrast with every other sensitive transaction endpoint:**

All other transaction sub-resources (`observers`, `approvers`, `signers`, `getTransactionWithVerifiedAccess`) call `verifyAccess()` or an equivalent check that confirms the user is the creator, observer, signer, or approver before proceeding. [4](#0-3) 

The `GET /transactions/:transactionId/comments` and `GET /transactions/comments/:id` endpoints are also unauthenticated with respect to transaction ownership — any authenticated user can enumerate comments on any transaction. [5](#0-4) 

**Exploit path:**
1. Attacker registers a valid account on the organization server (standard user flow).
2. Attacker enumerates or guesses a `transactionId` (integer IDs are sequential).
3. Attacker sends `POST /transactions/<id>/comments` with a valid JWT and any `message` string.
4. The comment is persisted to `transaction_comment` with the attacker's `userId` and the target `transactionId` — no relationship check is ever performed.
5. Attacker can also call `GET /transactions/<id>/comments` to read all comments on any transaction.

### Impact Explanation
- **Unauthorized state modification**: Any authenticated user can inject comments into transactions they have no legitimate access to, polluting the audit trail and communication channel for sensitive multi-party signing workflows.
- **Information disclosure**: Any authenticated user can read all comments on any transaction via the unguarded `GET` endpoints, leaking internal coordination details, key IDs discussed, and workflow state.
- **Integrity break**: The `transaction_comment` table is intended to record communications among transaction participants (creator, observers, signers, approvers). Allowing arbitrary users to write to it breaks the trust model of the organization workflow.

### Likelihood Explanation
The attacker precondition is only a valid, verified account on the organization server — the lowest privilege level available. Transaction IDs are sequential integers, making enumeration trivial. No special knowledge, keys, or privileges are required. The missing check is confirmed by the developer's own `//TODO` comment, meaning it was never implemented, not accidentally removed.

### Recommendation
In `CommentsService.createComment()`, add a call to `verifyAccess()` (already implemented in `TransactionsService`) before persisting the comment. Reject the request with `UnauthorizedException` if the user is not the creator, observer, signer, or approver of the target transaction. Apply the same check to `getTransactionComments()` and `getTransactionCommentById()`. [4](#0-3) 

### Proof of Concept
**Preconditions**: Two accounts exist — `victim` (creator of transaction ID 5) and `attacker` (no relationship to transaction 5). Both have valid JWTs.

```
# Step 1: Attacker posts a comment on a transaction they have no access to
POST /transactions/5/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "message": "attacker-controlled comment" }

# Expected (correct): 401 Unauthorized
# Actual: 201 Created — comment is persisted with attacker's userId and transactionId=5

# Step 2: Attacker reads all comments on the same transaction
GET /transactions/5/comments
Authorization: Bearer <attacker_jwt>

# Expected (correct): 401 Unauthorized
# Actual: 200 OK — full comment history returned
```

The `transaction_comment` table now contains a row with `userId = attacker.id` and `transactionId = 5`, despite the attacker having no creator/observer/signer/approver relationship to that transaction. [3](#0-2) [6](#0-5)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
