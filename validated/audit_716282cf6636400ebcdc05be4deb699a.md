### Title
IDOR in `DELETE /transactions/:transactionId/approvers/:id` Allows Any Transaction Creator to Delete Approvers from Transactions They Do Not Own

### Summary
The `removeTransactionApprover` endpoint verifies that the caller is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that the approver actually belongs to that transaction. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` in the URL and a victim's approver `id`.

### Finding Description

**Root cause — missing cross-reference check between `transactionId` and approver `id`.**

The controller handler is: [1](#0-0) 

```typescript
@Delete('/:id')
async removeTransactionApprover(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Param('id', ParseIntPipe) id: number,
) {
    await this.approversService.getCreatorsTransaction(transactionId, user);
    await this.approversService.removeTransactionApprover(id);
    return true;
}
```

Step 1 — `getCreatorsTransaction(transactionId, user)` — only checks that the authenticated user is the creator of the transaction identified by the URL's `:transactionId`: [2](#0-1) 

Step 2 — `removeTransactionApprover(id)` — fetches the approver by its own primary key `id` and deletes it with no check that `approver.transactionId === transactionId`: [3](#0-2) 

The deletion cascades through the entire approver sub-tree via a raw recursive SQL query: [4](#0-3) 

**Exploit path:**

1. Attacker registers a normal account and creates any transaction (transaction A). They are now the creator of A.
2. Victim creates transaction B and adds approvers (approver IDs are sequential integers, e.g. `42`).
3. Attacker sends:
   ```
   DELETE /transactions/{A_id}/approvers/42
   ```
4. The server verifies the attacker is the creator of A (true) and then unconditionally deletes approver `42`, which belongs to transaction B.

Approver IDs are auto-incremented integers, making them trivially enumerable. Additionally, if the attacker is legitimately an observer or signer of transaction B, they can read the exact approver IDs via `GET /transactions/{B_id}/approvers`.

### Impact Explanation

The approver workflow is the multi-signature authorization gate for Hedera transactions. Removing approvers from a transaction the attacker does not own:

- **Bypasses the approval threshold**: if all approvers are deleted, the transaction can proceed to execution without any required approvals, violating the multi-party authorization invariant.
- **Permanent state corruption**: once approvers are soft-deleted, the transaction's approval state is unrecoverably altered unless the creator manually re-adds them.
- **Cross-tenant integrity break**: any authenticated organization member who owns at least one transaction can sabotage any other transaction in the same organization.

### Likelihood Explanation

- **Precondition**: attacker must be an authenticated, verified user with at least one transaction they created. This is the normal baseline for any organization member.
- **No privileged access required**: no admin keys, no leaked credentials.
- **Approver IDs are guessable**: sequential integers starting from 1; enumeration requires at most a few hundred requests.
- **Alternatively zero-guess**: if the attacker is an observer or signer of the target transaction, they can read exact approver IDs from the `GET` endpoint.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the transaction the caller is authorized for:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // ← ADD THIS CHECK
    if (approver.transactionId !== transactionId)
        throw new UnauthorizedException('Approver does not belong to this transaction');

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
    return result;
}
```

Pass `transactionId` from the controller into the service call, mirroring the pattern already used in `updateTransactionApprover`: [5](#0-4) 

### Proof of Concept

**Setup:**
- User A (attacker) creates transaction `T_A` → becomes its creator.
- User B (victim) creates transaction `T_B` and adds approver with `id = 99`.

**Attack request (attacker's JWT):**
```
DELETE /transactions/{T_A_id}/approvers/99
Authorization: Bearer <attacker_jwt>
```

**Server execution:**
1. `getCreatorsTransaction(T_A_id, attacker)` → passes (attacker owns T_A).
2. `removeTransactionApprover(99)` → fetches approver 99 (belongs to T_B), deletes it.

**Result:** Approver 99 is permanently removed from T_B. If T_B required that approver's signature, the approval threshold is now unmet or the approver tree is corrupted, allowing T_B to proceed without the intended authorization.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

    // notifyTransactionAction(this.notificationsService);
  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
