### Title
Cross-Transaction Approver Deletion: Creator of Any Transaction Can Delete Approvers Belonging to Any Other Transaction

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController` verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then unconditionally deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system, bypassing the approval gate on transactions they do not own.

### Finding Description

**Root cause — decoupled authorization and action**

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `DELETE /:id` handler performs two independent operations:

```typescript
// approvers.controller.ts  lines 102-113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);  // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);                // ← deletes approver by id, no binding check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller owns the transaction referenced by the URL's `:transactionId` parameter:

```typescript
// approvers.service.ts  lines 624-644
async getCreatorsTransaction(transactionId, user, entityManager?) {
  ...
  if (transaction.creatorKey?.userId !== user.id)
    throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
  return transaction;
}
``` [2](#0-1) 

`removeTransactionApprover` then deletes the approver row identified by the separate `:id` parameter with no check that it belongs to the same transaction:

```typescript
// approvers.service.ts  lines 534-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [3](#0-2) 

The two URL parameters (`:transactionId` and `:id`) are never cross-validated. The authorization gate is satisfied by owning *any* transaction, while the destructive action operates on an *arbitrary* approver row.

This is the direct analog of the Aragon bug: just as a permission set on `address(this)` cascaded to any `_where` contract because the two checks were independent, here ownership of transaction A satisfies the guard that then deletes an approver from transaction B.

### Impact Explanation

An attacker who is the creator of at least one transaction (a normal, unprivileged product role) can:

1. Enumerate approver IDs for any target transaction (approver IDs are sequential integers; the `GET /transactions/:transactionId/approvers` endpoint is accessible to any participant of that transaction, and IDs are exposed in responses).
2. Call `DELETE /transactions/<attacker_own_tx_id>/approvers/<victim_approver_id>`.
3. The guard passes (attacker owns `<attacker_own_tx_id>`), and the victim approver row is hard-deleted from the database.

**Concrete consequences:**
- Approval requirements on any transaction can be silently gutted. A transaction requiring M-of-N approvals can be reduced to 0 approvers, allowing it to proceed to execution without the mandated governance sign-off.
- The attack is permanent and silent: the approver record is removed, notifications fire against the victim transaction, and there is no audit trail attributing the deletion to the attacker's transaction.
- In a multi-signature organizational workflow this directly enables unauthorized fund movement or unauthorized Hedera network operations (account updates, node operations) that the approval gate was designed to prevent.

### Likelihood Explanation

- **Attacker precondition**: authenticated, verified user who has created at least one transaction — the default state for any active organizational participant.
- **No privileged access required**: no admin key, no leaked credential, no internal network access.
- **Trivially reachable**: the endpoint is a standard REST `DELETE` call. Approver IDs are sequential integers discoverable by any participant of any transaction they legitimately belong to.
- **No rate-limit or anomaly barrier**: the request is structurally identical to a legitimate creator deleting their own approver.

### Recommendation

Inside `removeTransactionApprover` (or before calling it in the controller), verify that the approver being deleted actually belongs to the transaction whose ownership was checked:

```typescript
// In the controller, after getCreatorsTransaction succeeds:
const approver = await this.approversService.getTransactionApproverById(id);
if (!approver) throw new BadRequestException(ErrorCodes.ANF);

// Walk to root to get the owning transactionId
const root = await this.approversService.getRootNodeFromNode(approver.id);
if (!root || root.transactionId !== transactionId)
  throw new UnauthorizedException('Approver does not belong to this transaction');

await this.approversService.removeTransactionApprover(id);
```

Alternatively, push the binding check into `removeTransactionApprover` itself by accepting and enforcing `transactionId` as a required parameter, mirroring the pattern already used in `updateTransactionApprover` which calls `getRootNodeFromNode` and validates `rootNode.transactionId !== transactionId`. [4](#0-3) 

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (attacker-owned).
- User B creates Transaction 2 with a required approver (approver row ID = 42, `userId` = User C).

**Exploit:**
```
DELETE /transactions/1/approvers/42
Authorization: Bearer <User A's JWT>
```

**Result:**
- `getCreatorsTransaction(1, UserA)` → passes (User A owns Transaction 1).
- `removeTransactionApprover(42)` → deletes approver row 42 from Transaction 2.
- Transaction 2 now has no approvers; it proceeds to execution without User C's required approval.
- User A receives `true` as the response; no error is raised.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
