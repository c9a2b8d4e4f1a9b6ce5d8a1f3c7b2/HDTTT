# Audit Report

## Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows False Signer Registration and Transaction Corruption

## Summary
The `POST /transactions/:transactionId/signers` endpoint (`SignersService.uploadSignatureMaps`) verifies that submitted public keys belong to the authenticated user but never cryptographically verifies that the accompanying signature bytes are valid for the target transaction. An authenticated org user can submit a `SignatureMap` containing their public key paired with arbitrary or replayed signature bytes, be permanently recorded as a signer of the target transaction, and ‚Äî if the signing threshold is thereby satisfied ‚Äî cause the transaction to advance to `WAITING_FOR_EXECUTION` and be submitted to Hedera with an invalid signature.

## Finding Description

**Root cause ‚Äî `processTransactionSignatures` performs only an ownership check:**

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, `processTransactionSignatures` (lines 217‚Äì267) iterates over the submitted `SignatureMap`, confirms each public key belongs to the calling user via `userKeyMap`, and unconditionally appends the key to `userKeys` (the list of new signers to persist):

```typescript
let userKey = userKeyMap.get(raw);
if (!userKey) userKey = userKeyMap.get(publicKey.toStringDer());
if (!userKey) throw new Error(ErrorCodes.PNY);   // ownership check only

sdkTransaction = sdkTransaction.addSignature(publicKey, map); // no crypto verify

if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);   // recorded as signer unconditionally
}
``` [6](#0-5) 

The persistence guard in `persistSignatureChanges` only skips a result when **both** `isSameBytes` is `true` **and** `userKeys.length === 0`:

```typescript
if (isSameBytes && userKeys.length === 0) continue;
``` [5](#0-4) 

Because `userKeys` is populated before the bytes comparison, a new key is always inserted as a signer even when no valid signature bytes were added.

**Contrast ‚Äî `importSignatures` does verify:**

`TransactionsService.importSignatures` (lines 545‚Äì549 of `transactions.service.ts`) calls `validateSignature` before touching the transaction:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);
``` [3](#0-2) 

`validateSignature` (lines 213‚Äì248 of `transaction.ts`) calls `publicKey.verify(bodyBytes, signature)` and throws on any invalid signature: [2](#0-1) 

The `/signers` endpoint has no equivalent call.

**Two concrete attack paths:**

*Path 1 ‚Äî False signer record (unconditional):*
The attacker submits a `SignatureMap` containing their registered public key paired with any bytes (e.g., 64 zero bytes, or a signature from a different transaction). The ownership check passes. `addSignature` is called; because the Hedera SDK matches map entries by transaction ID and node account ID, a map from a different transaction will be a no-op on the target transaction's bytes (`isSameBytes = true`). However, `userKeys.length > 0`, so the guard does not skip, and the attacker is inserted into `TransactionSigner` as a signer of the target transaction without ever producing a valid signature for it.

*Path 2 ‚Äî Bytes corruption (targeted):*
The attacker constructs a `SignatureMap` whose keys match the target transaction's transaction ID and node account IDs but whose signature bytes are invalid (e.g., random bytes). `addSignature` appends these bytes to the transaction body (`isSameBytes = false`), the corrupted bytes are persisted, and the attacker is recorded as a signer. The transaction is now permanently unexecutable on-chain.

## Impact Explanation

- **False signer record / audit trail corruption**: The attacker is permanently recorded as having signed a transaction without producing a valid signature for it. This corrupts the audit trail and the signing threshold count.
- **Premature status advancement**: If the false signer record satisfies the required signing threshold, `processTransactionStatus` advances the transaction to `WAITING_FOR_EXECUTION`. The chain service then submits it to Hedera, which returns `INVALID_SIGNATURE`, and the transaction is marked `FAILED`. Legitimate signers cannot recover it.
- **Transaction bytes corruption** (Path 2): The stored `transactionBytes` are overwritten with bytes containing an invalid signature, making the transaction permanently unexecutable.
- **Denial of service on targeted transactions**: Any transaction the attacker is authorized to sign can be permanently broken.

## Likelihood Explanation

- The attacker must be an authenticated, verified org user with at least one registered public key ‚Äî no admin or privileged role is required.
- Path 1 requires only submitting a `SignatureMap` with the attacker's own public key and arbitrary signature bytes ‚Äî trivially achievable with the Hedera SDK.
- Path 2 additionally requires knowing the target transaction's transaction ID and node account IDs, both of which are visible to any user with access to the transaction.
- No rate limiting or anomaly detection is present on the `/signers` endpoint.
- The direct asymmetry between `/signers` (no crypto verify) and `/signatures/import` (crypto verify) strongly suggests an oversight.

## Recommendation

Add a `validateSignature` call inside `processTransactionSignatures`, mirroring the pattern already used in `importSignatures`:

```typescript
// Before calling addSignature, verify the signature bytes are valid
const { data: validPublicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

Only keys returned by `validateSignature` should be added via `addSignature` and recorded in `userKeys`. This ensures that the `/signers` endpoint enforces the same cryptographic guarantee as `/signatures/import`.
