Audit Report

## Title
`cutoffAt` Deadline Is Stored But Never Enforced in Signature Upload Paths, Allowing Signatures After the Intended Cutoff

## Summary
The `Transaction` entity stores a `cutoffAt` field documented as "The timestamp at which the transaction can no longer be signed by the signers." Neither `SignersService.validateTransactionStatus()` nor `TransactionsService.importSignatures()` checks this field before accepting a signature upload. Any authenticated signer can submit a valid signature after the `cutoffAt` deadline has passed, bypassing the intended access-control boundary entirely.

## Finding Description

**Root cause — `cutoffAt` is stored but never read during signature validation.**

The `cutoffAt` column is declared as a nullable column on the `Transaction` entity: [1](#0-0) 

It is accepted from the creator's DTO as an optional field: [2](#0-1) 

It is persisted during transaction creation: [3](#0-2) 

The database documentation explicitly states its purpose: [4](#0-3) 

**Signature upload path 1 — `POST /transactions/:id/signers` via `SignersService.uploadSignatureMaps()`**

`SignersService.validateTransactionStatus()` is the sole gate before signatures are accepted. It checks only the status enum and the Hedera SDK-level `isExpired()` (which reflects `validStart + transactionValidDuration`, typically 180 seconds). There is no `cutoffAt` check: [5](#0-4) 

**Signature upload path 2 — `POST /transactions/signatures/import` via `TransactionsService.importSignatures()`**

`TransactionsService.importSignatures()` performs the same two checks (status + `isExpired()`) and equally omits any `cutoffAt` comparison: [6](#0-5) 

## Impact Explanation

The `cutoffAt` deadline is the only application-level mechanism for a transaction creator to close the signing window before the Hedera validity window opens. Bypassing it allows:

- A signer who was intentionally excluded after the cutoff to add their signature and potentially push the transaction over the required threshold.
- Transactions to be promoted to `WAITING_FOR_EXECUTION` and executed on-chain after the creator's intended deadline, violating the creator's expressed intent and the documented invariant.
- In multi-signature governance workflows (the primary use-case of this tool), this breaks the trust model: the cutoff is the mechanism by which an organization freezes the signer set before execution.

## Likelihood Explanation

Any authenticated user whose public key is registered on the transaction can reach `POST /transactions/:id/signers` with no elevated privilege — only a valid JWT and a key belonging to the transaction's required signer set are needed. The `cutoffAt` field is optional (nullable), so only transactions where the creator explicitly set it are affected — but for those transactions the bypass is trivially reachable by any eligible signer after the deadline.

## Recommendation

Add a `cutoffAt` check in both signature upload paths. In `SignersService.validateTransactionStatus()`:

```typescript
private validateTransactionStatus(transaction: Transaction): string | null {
  if (
    transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
    transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
  ) {
    return ErrorCodes.TNRS;
  }

  // Enforce the creator-defined signing deadline
  if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
    return ErrorCodes.TE; // or a dedicated ErrorCode for cutoff exceeded
  }

  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
  if (isExpired(sdkTransaction)) {
    return ErrorCodes.TE;
  }

  return null;
}
```

Apply the same guard in `TransactionsService.importSignatures()` immediately after the status check: [6](#0-5) 

## Proof of Concept

1. Creator calls `POST /transactions` with `cutoffAt` set to `now + 1 minute` and `validStart` set to `now + 24 hours`.
2. Wait for `cutoffAt` to pass (1 minute).
3. An eligible signer calls `POST /transactions/:id/signers` with a valid signature map for their registered key.
4. `validateTransactionStatus()` is invoked:
   - `transaction.status` is `WAITING_FOR_SIGNATURES` → passes.
   - `isExpired(sdkTransaction)` returns `false` (the Hedera validity window, `validStart + 180s`, has not yet opened) → passes.
   - No `cutoffAt` check exists → passes.
5. The signature is merged into `transactionBytes` and persisted. [5](#0-4) 
6. If this signature tips the threshold, `processTransactionStatus()` promotes the transaction to `WAITING_FOR_EXECUTION` and the chain service executes it — after the creator's intended signing deadline.
