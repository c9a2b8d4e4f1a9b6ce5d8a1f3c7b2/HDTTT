# Audit Report

## Title
Transaction Creator Can Modify or Remove Approvers on In-Progress Transactions, Bypassing the Approval Workflow

## Summary
`createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` in `approvers.service.ts` perform no check on the current transaction status before mutating approver configuration. The sole authorization gate, `getCreatorsTransaction`, verifies only that the caller is the transaction creator — it never inspects `transaction.status`. A transaction creator can therefore remove a rejecting approver, lower the approval threshold, or replace an approver after approvals have already been collected, nullifying rejections and bypassing the organizational approval workflow.

## Finding Description

**Root cause — `getCreatorsTransaction` is status-blind**

`getCreatorsTransaction` fetches the transaction and checks only `transaction.creatorKey?.userId !== user.id`. There is no status inspection of any kind: [1](#0-0) 

Because this is the only authorization gate called before every approver mutation, all three mutation paths are reachable regardless of whether the transaction is `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `REJECTED`, or any other in-progress state.

---

**Path 1 — Remove a rejecting approver**

The controller calls `getCreatorsTransaction` (status-blind) and then immediately calls `removeTransactionApprover`, which performs no status check of its own: [2](#0-1) 

The service-level `removeTransactionApprover` also contains no status guard: [3](#0-2) 

---

**Path 2 — Lower the threshold after partial approvals**

`updateTransactionApprover` calls `getCreatorsTransaction` at line 394 and then unconditionally writes the new threshold with no status check: [4](#0-3) [5](#0-4) 

No existing approval records are cleared when the threshold is lowered, so previously collected approvals immediately satisfy the reduced requirement.

---

**Path 3 — Replace an approver who rejected**

When `dto.userId` is updated, the old approver's `signature`, `userKeyId`, and `approved` fields are unconditionally cleared: [6](#0-5) 

This erases a rejection (`approved = false`) and resets the slot to pending, effectively removing the veto.

---

**Path 4 — Inject approvers into a transaction already past the approval stage**

`createTransactionApprovers` calls `getCreatorsTransaction` and then proceeds to create approvers with no status check: [7](#0-6) 

---

**Contrast with `approveTransaction`**, which correctly guards against acting on a non-approvable transaction: [8](#0-7) 

No equivalent guard exists for any of the three mutation endpoints.

## Impact Explanation

The approval system is the organizational governance layer for multi-party Hedera transactions. By exploiting this gap, a transaction creator can:

1. **Nullify a rejection**: Remove or replace an approver who voted `approved = false`, erasing their veto.
2. **Retroactively lower the threshold**: After N-1 of N required approvers have approved, reduce the threshold to N-1, making the approval condition immediately satisfied without the final approver's consent.
3. **Inject approvers into a transaction already in `WAITING_FOR_EXECUTION`**: `createTransactionApprovers` has the same missing status check, allowing the creator to add approvers to a transaction that has already cleared the approval stage.

The result is that the approval workflow — the primary multi-party control mechanism — can be unilaterally bypassed by the transaction creator, who is supposed to be subject to it.

## Likelihood Explanation

The attacker is the transaction creator: a normal, authenticated, non-admin user. No leaked credentials, no admin access, and no cryptographic break are required. The attack path is a standard authenticated REST call (`DELETE /transactions/:transactionId/approvers/:id` or `PATCH /transactions/:transactionId/approvers/:id`). Any creator who faces a rejection has both the motive and the direct API access to exploit this. Likelihood is **high**.

## Recommendation

Add a transaction status guard in `getCreatorsTransaction`, or add an explicit check at the top of each mutation function (`createTransactionApprovers`, `updateTransactionApprover`, and the controller handler for `removeTransactionApprover`). The guard should reject mutations when the transaction is in any state other than `WAITING_FOR_SIGNATURES` (the only state where approver configuration is legitimately mutable). Modeled after the existing guard in `approveTransaction`:

```typescript
if (
  transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES
) {
  throw new BadRequestException(ErrorCodes.TNRA); // or a dedicated error code
}
```

This single check, applied consistently across all three mutation paths, closes the vulnerability.

## Proof of Concept

**Precondition**: Transaction T is in `WAITING_FOR_SIGNATURES`. Approver A has voted `approved = false` (rejected). The creator is authenticated as user C.

**Step 1 — Nullify the rejection via replace:**
```
PATCH /transactions/{T}/approvers/{A_id}
Authorization: Bearer <creator_token>
{ "userId": <new_user_id> }
```
`getCreatorsTransaction` passes (C is creator). `updateTransactionApprover` clears `approved`, `signature`, `userKeyId` for approver A and sets the new user. The rejection is erased.

**Step 2 — Retroactively lower the threshold:**
```
PATCH /transactions/{T}/approvers/{list_node_id}
Authorization: Bearer <creator_token>
{ "threshold": <current_approvals_count> }
```
`getCreatorsTransaction` passes. The threshold is written unconditionally. If N-1 approvals were already collected, the condition is now satisfied.

**Step 3 — Remove the rejecting approver entirely:**
```
DELETE /transactions/{T}/approvers/{A_id}
Authorization: Bearer <creator_token>
```
Controller calls `getCreatorsTransaction` (passes), then `removeTransactionApprover(A_id)`. The approver record is soft-deleted with no status check. The rejection is gone.

In all three cases, the transaction proceeds as if the rejection never occurred.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
