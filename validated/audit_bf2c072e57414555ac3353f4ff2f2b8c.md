### Title
Unbounded Recursive Approver Tree Creation Enables Authenticated Resource Exhaustion DoS

### Summary
The `POST /transactions/:transactionId/approvers` endpoint accepts a deeply nested, arbitrarily large approver tree with no depth or count limits enforced at either the DTO validation layer or the service layer. Any authenticated user who creates a transaction can submit a single crafted request containing an exponentially large tree of threshold-type approver nodes, triggering unbounded recursive function calls and a proportional number of expensive recursive-CTE database queries within a single held database transaction, causing severe server-side resource exhaustion.

### Finding Description

**Vulnerability class:** DoS / Resource Exhaustion (unbounded recursive input, no size cap)

**Root cause — DTO layer (no upper bound):**

`CreateTransactionApproverDto` and `CreateTransactionApproversArrayDto` impose only a minimum-size constraint on arrays; no `@ArrayMaxSize()` is present on either `approversArray` or the nested `approvers` field. [1](#0-0) 

`approversArray` has `@IsArray()` and `@ValidateNested()` only; `approvers` has `@ArrayMinSize(1)` but no upper bound. Both fields accept arbitrarily large, arbitrarily deep payloads.

**Root cause — service layer (unbounded recursion):**

`createTransactionApprovers` iterates `dto.approversArray` and calls the inner async `createApprover` for every element. `createApprover` recursively calls itself for every element of `dtoApprover.approvers`: [2](#0-1) 

Each recursive invocation issues multiple expensive database operations:
- `isNode` → `count` query on `transaction_approver`
- `findOne(TransactionApprover, ...)` for parent lookup
- `getRootNodeFromNode` → **recursive CTE** traversal up the tree
- `count(User, ...)` for user existence check
- `getApproversByTransactionId` → **recursive CTE** traversal down the tree
- `insert(TransactionApprover, ...)` [3](#0-2) 

All of this runs inside a single `dataSource.transaction(...)` call, holding a database transaction open for the entire duration.

**Critical bypass — `isNode` does not block threshold nodes:**

The duplicate-check guard `isNode` only returns `true` when `typeof approver.userId === 'number'`. For threshold-type nodes (no `userId`), it always returns `false`, meaning an unlimited number of threshold nodes can be inserted without triggering the "already exists" guard: [4](#0-3) 

**Attack path:**

1. Attacker registers as a normal organization user (no special privilege required).
2. Attacker creates any valid transaction via `POST /transactions` — they become the creator.
3. Attacker sends a single `POST /transactions/:id/approvers` with a crafted payload:

```json
{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{
      "threshold": 1,
      "approvers": [{
        "threshold": 1,
        "approvers": [ ... (depth D, branching factor B) ]
      }]
    }]
  }]
}
```

With branching factor B=10 and depth D=5, the total node count is ~111,111. Each node triggers ~5 database operations (including two recursive-CTE queries), yielding ~555,000 database operations inside a single held transaction. The Node.js event loop is blocked for the duration of the sequential `await` chain; the database connection pool is saturated; other requests time out.

**Authorization check does not mitigate the issue:**

`getCreatorsTransaction` correctly restricts the endpoint to the transaction creator, but any authenticated user can become a creator by creating their own transaction. The authorization check is not a meaningful barrier. [5](#0-4) 

### Impact Explanation

A single crafted HTTP request from any authenticated user causes:
- Unbounded recursive JavaScript call stack growth (potential stack overflow for very deep trees).
- Hundreds of thousands of sequential database queries inside one held transaction, saturating the PostgreSQL connection pool.
- The `api` service event loop is blocked for the duration, making it unresponsive to all other users.
- Repeated requests (one per transaction created) sustain the outage indefinitely.

This constitutes a complete, sustained denial of service of the `api` microservice achievable by any authenticated organization member.

### Likelihood Explanation

- **Attacker precondition:** Valid organization account only — no admin, no leaked credentials, no privileged keys.
- **Trigger:** A single HTTP POST request with a crafted JSON body.
- **Repeatability:** The attacker can create a new transaction and repeat the attack after each recovery.
- **Detection difficulty:** The request is structurally valid and passes all existing DTO validation; it will not be flagged by standard WAF rules.

### Recommendation

1. **Enforce array size limits at the DTO layer** — add `@ArrayMaxSize(N)` to both `approversArray` in `CreateTransactionApproversArrayDto` and `approvers` in `CreateTransactionApproverDto`. A reasonable cap (e.g., 20–50) matches real-world multi-sig use cases. [6](#0-5) 

2. **Enforce a maximum tree depth** — track recursion depth in `createApprover` and throw if it exceeds a configured limit (e.g., 5).

3. **Enforce a total-node budget** — count total nodes across the entire submitted tree before entering the database transaction and reject if it exceeds a cap.

4. **Fix `isNode` to also block duplicate threshold nodes** — the current guard only blocks duplicate user-based nodes, leaving threshold nodes uncapped even with a per-request limit.

### Proof of Concept

```bash
# Step 1: Authenticate and create a transaction (attacker becomes creator)
TOKEN="<valid_jwt>"
TX_ID=$(curl -s -X POST http://localhost:3000/transactions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ ... valid transaction body ... }' | jq .id)

# Step 2: Build a crafted approver tree (branching=5, depth=6 → 3906 nodes)
python3 -c "
import json
def build(depth):
    if depth == 0:
        return {'threshold': 1, 'approvers': [{'userId': 1}]}
    return {'threshold': 1, 'approvers': [build(depth-1) for _ in range(5)]}
print(json.dumps({'approversArray': [build(6)]}))
" > payload.json

# Step 3: Send the single crafted request
curl -X POST http://localhost:3000/transactions/$TX_ID/approvers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d @payload.json

# Expected outcome: request hangs for tens of seconds / minutes;
# concurrent requests to the api service time out;
# PostgreSQL connection pool exhausted.
```

The `createApprover` recursive function will be called ~3,906 times, each issuing up to 5 database operations (including two recursive-CTE queries), for a total of ~19,530 database operations inside a single held transaction — all from one unauthenticated-equivalent HTTP request.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
