### Title
The `approved` field in `ApproverChoiceDto` is not included in the signed payload, allowing a MITM or malicious backend to silently flip an approver's decision

### Summary
The transaction approval flow requires approvers to submit a cryptographic signature alongside an `approved: boolean` field. However, the signature is computed only over the raw transaction body bytes — the `approved` field is never included in the signed data. Because the same signature is equally valid for both `approved: true` and `approved: false`, any party who can intercept or modify the API request (a MITM, a compromised backend operator, or a rogue organization server) can flip the approver's intent without invalidating the signature.

### Finding Description

**Signing (front-end):**

In `ApproveTransactionController.vue`, the approver signs only the transaction body bytes. The `approved` prop is passed separately to `sendApproverChoice` and is never included in the signed payload:

```ts
// front-end/src/renderer/pages/TransactionDetails/ApproveTransactionController.vue:99-106
const signature = getTransactionBodySignatureWithoutNodeAccountId(privateKey, sdkTransaction);
await sendApproverChoice(
  user.selectedOrganization.serverUrl,
  transaction.id,
  orgKey.id,
  signature,
  props.approved,   // ← NOT signed, sent as a plain field
);
```

`getTransactionBodySignatureWithoutNodeAccountId` signs only `proto.TransactionBody.encode(transaction._makeTransactionBody(null)).finish()` — the `approved` boolean is absent from this byte string.

**DTO (no binding between signature and intent):**

```ts
// back-end/apps/api/src/transactions/dto/approver-choice.dto.ts:5-16
export class ApproverChoiceDto {
  userKeyId: number;
  signature: Buffer;   // signs transaction body bytes only
  approved: boolean;   // unsigned — not covered by signature
}
```

**Verification (back-end):**

`verifyTransactionBodyWithoutNodeAccountIdSignature` verifies the signature against the same transaction body bytes, with no awareness of `approved`:

```ts
// back-end/libs/common/src/utils/sdk/transaction.ts:266-284
export const verifyTransactionBodyWithoutNodeAccountIdSignature = (
  transaction: SDKTransaction,
  signature: string | Buffer,
  publicKey: string | PublicKey,
) => {
  const bodyBytes = getTransactionBodyBytes(transaction);
  return publicKey.verify(bodyBytes, signature);
};
```

After the signature passes, the server persists `dto.approved` verbatim:

```ts
// back-end/apps/api/src/transactions/approvers/approvers.service.ts:599-609
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({
    userKeyId: dto.userKeyId,
    signature: dto.signature,
    approved: dto.approved,   // ← taken from unsigned field
  })
  .whereInIds(userApprovers.map(a => a.id))
  .execute();
```

**End-to-end exploit path:**

1. Approver A submits `POST /transactions/{id}/approvers/approve` with `{userKeyId, signature, approved: true}`.
2. An attacker (MITM, rogue organization server, or compromised backend) intercepts the request and changes `approved: true` → `approved: false` (or vice versa), leaving `signature` unchanged.
3. `verifyTransactionBodyWithoutNodeAccountIdSignature` returns `true` — the signature is still valid because it covers only the transaction body bytes.
4. The server stores `approved: false`, recording the approver as having rejected the transaction they intended to approve.

The organization server URL is user-configurable (`user.selectedOrganization.serverUrl`), meaning a user connecting to a malicious organization server is directly exposed to this attack with no additional preconditions.

### Impact Explanation
An attacker controlling the organization server (or performing MITM on the HTTPS connection) can silently flip every approver's decision. In a threshold-approval workflow, this can:
- Block transactions that have legitimate approval by converting approvals to rejections.
- Force approval of transactions that approvers intended to reject.

The cryptographic signature provides no protection against this manipulation because it does not commit to the approver's intent.

### Likelihood Explanation
The organization server URL is supplied by the user and is not pinned or verified by the client. A malicious organization operator — a realistic attacker profile for a multi-tenant approval tool — can exploit this without any privileged access to the Hedera network or to user private keys. No brute force or cryptographic break is required.

### Recommendation
Include the `approved` field (and `transactionId`) in the signed payload. The approver should sign a structured message such as:

```
sign( SHA256( transactionBodyBytes || approved_byte || transactionId_bytes ) )
```

Both the front-end (`getTransactionBodySignatureWithoutNodeAccountId`) and the back-end (`verifyTransactionBodyWithoutNodeAccountIdSignature`) must be updated to incorporate these fields. The `ApproverChoiceDto` signature must cryptographically bind the approver's intent so that flipping `approved` invalidates the signature.

### Proof of Concept

1. Approver calls `getTransactionBodySignatureWithoutNodeAccountId(privateKey, sdkTransaction)` — this produces signature `S` over the transaction body bytes `B`.
2. Approver submits `{userKeyId: K, signature: S, approved: true}`.
3. Attacker intercepts and submits `{userKeyId: K, signature: S, approved: false}` to `POST /transactions/{id}/approvers/approve`.
4. Back-end calls `verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, S, publicKey)` → returns `true` (S is a valid signature over B regardless of the `approved` value).
5. Back-end persists `approved: false` — the approver's decision has been silently reversed with a cryptographically valid signature. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

export const verifyTransactionBodyWithoutNodeAccountIdSignature = (
  transaction: SDKTransaction,
  signature: string | Buffer,
  publicKey: string | PublicKey,
) => {
  const bodyBytes = getTransactionBodyBytes(transaction);

  /* Deserialize Public Key */
  publicKey = publicKey instanceof PublicKey ? publicKey : PublicKey.fromString(publicKey);

  /* Deserialize Signature */
  signature = typeof signature === 'string' ? decode(signature) : signature;

  try {
    return publicKey.verify(bodyBytes, signature);
  } catch (err) {
    console.log(err);
    return false;
  }
};
