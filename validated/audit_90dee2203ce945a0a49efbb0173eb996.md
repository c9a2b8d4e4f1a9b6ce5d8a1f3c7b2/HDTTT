### Title
Transaction Creator Can Self-Assign as Approver and Approve Their Own Transaction, Bypassing Multi-Party Approval Governance

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` contains no check preventing the transaction creator from assigning their own `userId` as an approver. Because the subsequent `approveTransaction` function only verifies that the caller is listed as an approver and that a valid cryptographic signature is provided — both conditions the creator can trivially satisfy — a malicious creator can unilaterally satisfy the approval requirement for their own transaction, defeating the separation-of-duties guarantee the approval workflow is designed to enforce.

### Finding Description

**Root cause — missing self-assignment guard in `createTransactionApprovers`:**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the function that creates approvers performs the following checks: [1](#0-0) 

1. The caller must be the transaction creator (`getCreatorsTransaction`).
2. The target `userId` must exist in the `User` table.

There is **no check** that `dtoApprover.userId !== user.id` (the creator's own ID). The creator can therefore POST to `transactions/:id/approvers` with their own `userId` in the body and be recorded as an approver of their own transaction.

**Exploit continuation — `approveTransaction` accepts the self-assigned approver:**

`approveTransaction` filters the approver list for records matching `a.userId === user.id` and then requires only that the caller supply a valid cryptographic signature over the transaction body: [2](#0-1) 

The creator already possesses the transaction bytes (they constructed them) and their own private key, so producing a valid signature is trivial. The signature verification step: [3](#0-2) 

…does not distinguish between a creator-signed approval and a genuine third-party approval.

**Controller entry point (no additional guard):** [4](#0-3) 

The only guards on the `POST /transactions/:id/approvers` route are JWT authentication and `VerifiedUserGuard` — no ownership-separation check exists.

### Impact Explanation
The approval mechanism exists to enforce multi-party governance: a designated second party must review and cryptographically endorse a transaction before it advances. A creator who self-assigns as the sole approver can satisfy this requirement unilaterally, reducing the approval workflow to a no-op. In an organization that relies on approvals for compliance, financial controls, or key-management governance, this allows a single compromised or malicious account to bypass the entire approval gate and push transactions to execution without any independent review.

### Likelihood Explanation
The attack requires only a valid authenticated session and creator ownership of a transaction — both conditions any normal organization user can achieve without elevated privileges. The exploit path is a straightforward two-step API call sequence (POST approver, POST approval) with no race conditions or timing dependencies. Any user who can create transactions can execute this bypass.

### Recommendation
In `createTransactionApprovers`, add an explicit guard rejecting any `userId` that matches the creator's `user.id`:

```typescript
if (dtoApprover.userId === user.id)
  throw new Error('The transaction creator cannot be assigned as an approver');
```

Apply the same check in `updateTransactionApprover` when `dto.userId` is being set, to prevent a creator from updating an existing approver slot to their own ID. Additionally, consider enforcing this invariant at the database constraint level (a check constraint or trigger) so that future service-layer changes cannot accidentally re-introduce the gap.

### Proof of Concept

**Preconditions:** Two accounts exist — `creator` (user ID 1) and `approver` (user ID 2). The organization has configured approvals as a governance requirement.

1. `creator` authenticates and obtains a JWT.
2. `creator` creates a transaction → receives `transactionId = 42`.
3. `creator` self-assigns as approver:
   ```
   POST /transactions/42/approvers
   Authorization: Bearer <creator_jwt>
   Content-Type: application/json

   { "approversArray": [{ "userId": 1 }] }
   ```
   Response: `201 Created` — no error, because `approvers.service.ts` only checks that user ID 1 exists in the `User` table.
4. `creator` signs the transaction body with their own private key, producing `<sig>`.
5. `creator` submits the approval:
   ```
   POST /transactions/42/approvers/approve
   Authorization: Bearer <creator_jwt>
   Content-Type: application/json

   { "userKeyId": 1, "signature": "<sig>", "approved": true }
   ```
   Response: `200 OK` — approval recorded, approval threshold satisfied.
6. The transaction advances to `WAITING_FOR_EXECUTION` with zero independent review, despite the approval workflow being active.

**Expected (correct) outcome:** Step 3 should return `400 Bad Request` — "The transaction creator cannot be assigned as an approver."

  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });
