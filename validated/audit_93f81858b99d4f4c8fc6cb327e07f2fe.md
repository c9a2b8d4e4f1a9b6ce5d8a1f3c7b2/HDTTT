Audit Report

## Title
Any Authenticated User Can Claim Another User's Public Key Without Proof of Ownership

## Summary
The `POST /user/:userId?/keys` endpoint registers any syntactically valid Hedera public key to the authenticated caller with no cryptographic proof that the caller holds the corresponding private key. Because the `userId` URL parameter is silently ignored on POST, and because the only ownership guard is a "key already taken" check, any verified organization member can preemptively register a victim's public key under their own account, permanently blocking the legitimate owner from registering it.

## Finding Description

**Controller** ā€” `UserKeysController.uploadKey` ignores the `:userId?` route parameter entirely for POST; the key is always assigned to `@GetUser() user`: [1](#0-0) 

**Service** ā€” `UserKeysService.uploadKey` looks up the key by `publicKey` value only. If no existing record is found, the key is created for whoever called the endpoint first ā€” with zero cryptographic verification: [2](#0-1) 

The only ownership guard fires only when the key is **already registered** to a different user: [3](#0-2) 

**DTO** ā€” `UploadUserKeyDto` validates only that `publicKey` is a syntactically valid Hedera key. No signature, challenge-response, or proof of private key possession is required: [4](#0-3) 

**Victim key disclosure** ā€” `getUserKeysRestricted` returns `publicKey` to all verified users; only `mnemonicHash` and `index` are hidden from non-owners: [5](#0-4) 

**E2E test confirms the URL param is ignored for POST** ā€” keys posted with arbitrary user IDs (`/123123123/keys`, `/5134/keys`) are always stored under the authenticated caller: [6](#0-5) 

## Impact Explanation
- **Key squatting / permanent DoS**: Once an attacker registers a victim's public key, the `ErrorCodes.PU` hard-block prevents the victim from ever registering that key in the organization. There is no admin recovery path exposed ā€” `removeUserKey` enforces `userKey.userId !== user.id`, so only the attacker (the registered owner) can delete it. [7](#0-6) 
- **Multi-sig workflow disruption**: If the attacker's account (holding the squatted key) is added as a required signer or approver, no valid signature can ever be produced for that key slot, permanently stalling any transaction that requires it.
- **Severity: Medium** ā€” requires an authenticated, verified org member; impact is availability and integrity of key registration and transaction workflows.

## Likelihood Explanation
- Any verified organization member can execute this attack with a single API call.
- Victim public keys are directly readable via `GET /user/:userId/keys`, accessible to all verified users.
- Public keys are also observable on the Hedera Mirror Node for any linked account.
- The attack is trivially scriptable.
- **Likelihood: Medium** ā€” attacker must be a verified org member, which is a low bar in any multi-user deployment.

## Recommendation
Require cryptographic proof of private key ownership at registration time. A standard approach is a server-issued challenge-response: the server issues a random nonce, the client signs it with the private key corresponding to the submitted public key, and the server verifies the signature before persisting the `UserKey` record. This ensures only the holder of the private key can register the corresponding public key.

## Proof of Concept
```
# Step 1 ā€” Attacker reads victim's registered public keys
GET /user/{victimId}/keys
Authorization: Bearer <attacker_jwt>
# Response: [{ "publicKey": "<victim_pubkey>", ... }]  (mnemonicHash/index omitted)

# Step 2 ā€” Attacker registers victim's public key under their own account
POST /user/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json
{ "publicKey": "<victim_pubkey>" }
# Response: 201 Created ā€” key now owned by attacker

# Step 3 ā€” Victim attempts to register their own key
POST /user/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json
{ "publicKey": "<victim_pubkey>" }
# Response: 400 Bad Request ā€” ErrorCodes.PU ("Public Key in use")
# Victim is permanently blocked with no recovery path
```
