### Title
Authorization Bypass in `removeTransactionApprover`: Approver ID Not Validated Against Authorized Transaction

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of `:transactionId`, but then passes the approver `:id` directly to `removeTransactionApprover()` without checking that the approver actually belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction by supplying their own `transactionId` for the authorization check and a victim's approver `id` for the deletion.

### Finding Description

**Root cause — decoupled authorization and action:**

In `approvers.controller.ts`, the `DELETE /:id` handler performs two independent calls:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts L102-L113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no cross-check
  return true;
}
``` [1](#0-0) 

The service method `removeTransactionApprover` receives only the approver `id` and performs no ownership verification:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts L534-L544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

The `transactionId` URL parameter is used only to authorize the caller as creator of *that* transaction; it is never compared against the `transactionId` stored on the approver being deleted. The `updateTransactionApprover` method in the same service correctly performs this cross-check:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts L389-L391
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The missing check in `removeTransactionApprover` is the direct analog of the `liquidateFrom` public-visibility bug: the outer function (`liquidate` / controller handler) performs a guard, but the inner function (`liquidateFrom` / `removeTransactionApprover`) is reachable with an arbitrary target ID that bypasses that guard.

**Exploit path:**

1. Attacker registers as a normal user and creates transaction A (attacker is creator).
2. Attacker enumerates or guesses approver ID `X` that belongs to victim transaction B (IDs are sequential integers).
3. Attacker sends: `DELETE /transactions/A/approvers/X`
4. Controller calls `getCreatorsTransaction(A, attacker)` — passes (attacker owns A).
5. Controller calls `removeTransactionApprover(X)` — deletes approver X from transaction B without any ownership check.

### Impact Explanation

An attacker can silently delete any approver node from any transaction in the system. This directly undermines the multi-signature approval workflow: required approvers can be removed, allowing transactions to advance to execution without the mandated approvals. In an organization context where high-value Hedera transactions (account updates, node operations, large transfers) require multi-party sign-off, this constitutes unauthorized state mutation that bypasses a core security control.

### Likelihood Explanation

The attacker requires only a valid authenticated account (normal user, no admin privileges). Approver IDs are sequential integers, making enumeration trivial. The endpoint is a standard REST `DELETE` call with no rate-limiting or anomaly detection visible in the codebase. Any registered user who has ever created a transaction satisfies the precondition.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver's root `transactionId` matches the URL-supplied `transactionId` and that the caller is the creator of that transaction — exactly as `updateTransactionApprover` already does:

```typescript
// approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  await this.getCreatorsTransaction(rootNode.transactionId, user); // ownership check

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Remove the redundant `getCreatorsTransaction` call from the controller once it is consolidated into the service method.

### Proof of Concept

**Preconditions:**
- Attacker account: `attacker@example.com` (normal user, JWT token `TOKEN_A`)
- Victim transaction ID: `42` (owned by another user, has approver ID `7`)
- Attacker-owned transaction ID: `99`

**Steps:**

```bash
# Step 1 — Attacker creates their own transaction to obtain a valid transactionId
POST /transactions
Authorization: Bearer TOKEN_A
# → returns { id: 99, ... }

# Step 2 — Attacker deletes approver 7 from victim transaction 42
# by supplying their own transactionId (99) for the authorization check
DELETE /transactions/99/approvers/7
Authorization: Bearer TOKEN_A

# Expected (current): HTTP 200 true  — approver 7 deleted from transaction 42
# Expected (fixed):   HTTP 401 Unauthorized — root transaction mismatch
```

**Outcome:** Approver ID 7 is permanently removed from transaction 42 without the victim's knowledge, potentially allowing that transaction to bypass its required approval threshold.
