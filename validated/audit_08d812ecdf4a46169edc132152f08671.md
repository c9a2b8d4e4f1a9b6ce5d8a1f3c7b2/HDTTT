Audit Report

## Title
Deleted Organization User's Transaction Approval Persists and Counts Toward Threshold

## Summary
When an organization user is removed via `removeUser`, only the `User` and `UserKey` records are soft-deleted. Any `TransactionApprover` rows belonging to that user — including those with `approved = true` and a stored `signature` — are left intact and continue to count toward the configured approval threshold on in-flight transactions.

## Finding Description

`removeUser` in `users.service.ts` performs exactly two cleanup operations: [1](#0-0) 

No `softDelete` or cascade is applied to `TransactionApprover` rows where `userId = id`.

The `getApproversByTransactionId` recursive CTE in `approvers.service.ts` only filters on `approverList."deletedAt" is null` — it never joins the `user` table to verify the referenced user is still active: [2](#0-1) 

The `approved` and `signature` columns are set during `approveTransaction` and are never cleared on user deletion: [3](#0-2) 

The frontend `isApproved` utility reads `approver.approved` directly from the persisted record, with no liveness check on the user: [4](#0-3) 

## Impact Explanation

Consider a transaction with a 2-of-3 threshold approver tree (users A, B, C):

1. User A approves → `TransactionApprover.approved = true`, `signature` set for A's record.
2. Admin removes User A via `DELETE /users/:id` → only `User` and `UserKey` rows are soft-deleted; A's `TransactionApprover` row remains with `deletedAt = null` and `approved = true`.
3. User B approves → the threshold of 2 is now met (A's stale approval + B's live approval).
4. The transaction proceeds to execution even though the admin's intent was to revoke A's organizational trust.

The admin action of removing a user has no effect on that user's already-submitted approvals on pending transactions.

## Likelihood Explanation

- Admin-initiated user removal is a standard, low-friction operation exposed via `DELETE /users/:id`.
- Any transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status with a threshold approver structure is affected.
- No special attacker capability is required: the approver only needs to approve before being removed, which is the normal workflow order.
- The `getApproversByTransactionId` query is called from multiple paths (`approveTransaction`, `shouldApproveTransaction`, `attachTransactionApprovers`, the notifications receiver), so the stale approval propagates everywhere approval status is evaluated. [5](#0-4) 

## Recommendation

In `removeUser`, add a soft-delete of all `TransactionApprover` rows for the removed user before soft-removing the user:

```ts
await this.repo.manager.softDelete(TransactionApprover, { userId: id });
```

This mirrors the existing `softDelete(UserKey, { userId: id })` pattern already present in the function. [6](#0-5) 

Alternatively (or additionally), `getApproversByTransactionId` can be hardened to join the `user` table and exclude approvers whose user has been soft-deleted, ensuring defense-in-depth even if the cleanup step is missed.

## Proof of Concept

1. Create a transaction with a threshold approver tree: `{ threshold: 2, approvers: [{ userId: A }, { userId: B }, { userId: C }] }`.
2. Authenticate as User A and call `POST /transactions/:id/approvers/approve` with a valid signature → `TransactionApprover.approved = true` for A's record.
3. As admin, call `DELETE /users/:id` for User A.
4. Query `getApproversByTransactionId(transactionId)` — A's `TransactionApprover` row is returned with `approved = true` because its own `deletedAt` is still `null`.
5. Authenticate as User B and approve → threshold of 2 is satisfied.
6. The transaction advances to execution with A's revoked approval still counted. [7](#0-6) [6](#0-5)

```typescript
  async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });

    if (!user) {
      throw new BadRequestException(ErrorCodes.UNF);
    }

    // Soft-delete all user keys first
    await this.repo.manager.softDelete(UserKey, { userId: id });

    // Then soft-delete the user
    await this.repo.softRemove(user);

    return true;
  }
```

```typescript
  async getApproversByTransactionId(
    transactionId: number,
    userId?: number,
    entityManager?: EntityManager,
  ): Promise<TransactionApprover[]> {
    if (typeof transactionId !== 'number' || (userId && typeof userId !== 'number')) return null;

    return (entityManager || this.repo).query(
      `
      with recursive approverList as
        (
          select * from transaction_approver 
          where "transactionId" = $1
            union all
              select approver.* from transaction_approver as approver
              join approverList on approverList."id" = approver."listId"
        )
      select * from approverList
      where approverList."deletedAt" is null
        ${userId ? 'and approverList."userId" = $2' : ''}
      `,
      userId ? [transactionId, userId] : [transactionId],
    );
  }
```

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
