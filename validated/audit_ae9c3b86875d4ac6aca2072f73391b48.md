Audit Report

## Title
`uploadSignatureMaps` Accepts Unverified Signatures, Allowing a Malicious Signer to Permanently Corrupt Transaction Bytes and Block Execution

## Summary
The `processTransactionSignatures` method in `signers.service.ts` calls `sdkTransaction.addSignature(publicKey, map)` without first cryptographically verifying that the submitted signature was produced over the correct transaction body bytes. A malicious registered signer can submit a signature made over a different transaction body; the server accepts it, records the user as a signer, and persists the corrupted bytes. When the transaction is later executed, the Hedera network rejects it with `INVALID_SIGNATURE`, permanently blocking execution.

## Finding Description

**Root cause — missing `validateSignature` call in the `uploadSignatureMaps` path**

`importSignatures` in `transactions.service.ts` correctly calls `validateSignature` before accepting any signature: [1](#0-0) 

`validateSignature` in `transaction.ts` explicitly calls `publicKey.verify(bodyBytes, signature)` for every entry in the map and throws on mismatch: [2](#0-1) 

The `uploadSignatureMaps` path (`POST /transactions/:id/signers`) goes through `validateAndProcessSignatures` → `processTransactionSignatures`, which **never calls `validateSignature`**. It calls `addSignature` directly with no cryptographic check: [3](#0-2) 

The developer comment in the same method confirms that `addSignature` only iterates and inserts raw bytes — no verification is performed: [4](#0-3) 

When `isSameBytes` is `false` (i.e., the attacker's crafted signature bytes were actually inserted into the transaction), the corrupted bytes are persisted to the database: [5](#0-4) 

## Impact Explanation

A malicious registered signer can permanently block execution of any transaction they are assigned to sign. Once the corrupted `transactionBytes` are written to the database, there is no mechanism to strip the bad signature and retry — the transaction will be marked `FAILED` by the chain service when the Hedera network returns `INVALID_SIGNATURE`. This affects all transaction types including high-value operations such as `FileUpdate`, `AccountUpdate`, `NodeCreate`, and HBAR transfers.

## Likelihood Explanation

The attacker must be a registered signer of the target transaction, which is a normal user role in organization mode — no admin or privileged access is required. The attack requires only a standard API call (`POST /transactions/:id/signers`) with a crafted `signatureMap` payload containing a signature over a different transaction body. The public key ownership check at line 248 is satisfied because the attacker uses their own legitimate key; only the signature bytes themselves are malicious. [6](#0-5) 

## Recommendation

Add a `validateSignature` call inside `processTransactionSignatures`, mirroring the pattern already used in `importSignatures`. Before calling `sdkTransaction.addSignature(publicKey, map)`, call `validateSignature(sdkTransaction, map)` and throw (or skip) if it returns an error. The utility is already imported and available in the common library: [7](#0-6) 

The fix should be applied at the top of `processTransactionSignatures`, before the loop that calls `addSignature`, consistent with how `importSignatures` gates on `validateSignature` before proceeding: [8](#0-7) 

## Proof of Concept

1. Attacker is a legitimate registered signer of transaction `T` (added by the creator in organization mode).
2. Attacker fetches `T`'s `transactionBytes` from the API.
3. Attacker locally constructs a **different** transaction body `T'` (e.g., different recipient or amount), freezes it, and signs it with their private key, producing `sigMap'`.
4. Attacker posts `sigMap'` to `POST /transactions/{id}/signers`.
5. `processTransactionSignatures` loads the original transaction from DB, finds the attacker's public key in `userKeyMap` (ownership check passes), and calls `sdkTransaction.addSignature(publicKey, sigMap')` — no cryptographic verification occurs. [9](#0-8) 
6. `isSameBytes` is `false` (the invalid signature bytes were inserted), so the corrupted `transactionBytes` are written to the database. [10](#0-9) 
7. The attacker is recorded as a signer. The transaction now carries an invalid signature in the DB.
8. When the chain service executes the transaction, the Hedera network returns `INVALID_SIGNATURE` and the transaction is permanently marked `FAILED` with no recovery path.

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
