Audit Report

## Title
Missing `@ArrayMinSize(1)` on `CreateTransactionApproversArrayDto.approversArray` Allows Empty Array Submission Triggering Spurious `TransactionStatusUpdate` NATS Notification

## Summary
`CreateTransactionApproversArrayDto.approversArray` lacks an `@ArrayMinSize(1)` constraint. An authenticated transaction creator can POST `{ "approversArray": [] }` to `POST /transactions/:id/approvers`, which passes DTO validation, performs zero database writes, but unconditionally fires a `TransactionStatusUpdate` NATS event — notifying downstream consumers of a state change that never occurred.

## Finding Description

**Missing constraint — confirmed at lines 25–29:**

`CreateTransactionApproversArrayDto` declares `approversArray` with only `@IsArray()`, `@ValidateNested({ each: true })`, and `@Type(...)` — no `@ArrayMinSize(1)`: [1](#0-0) 

**Contrast with sibling DTO** `CreateTransactionObserversDto`, which correctly guards its array field: [2](#0-1) 

**And the nested `approvers` field** inside `CreateTransactionApproverDto` also correctly applies `@ArrayMinSize(1)`: [3](#0-2) 

**Unconditional emit in the service — confirmed at lines 353–358:**

In `createTransactionApprovers`, the `for` loop at line 353 iterates over `dto.approversArray`. When the array is empty, the loop body never executes and the database transaction commits with zero writes. `emitTransactionStatusUpdate` at line 358 fires unconditionally — outside the loop, but still inside the `try` block — regardless of whether any approvers were created: [4](#0-3) 

Compare this with `updateTransactionApprover`, which correctly gates its emit behind an `updated` flag: [5](#0-4) 

**The endpoint** is `POST /transactions/:transactionId/approvers`, protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`: [6](#0-5) 

**Creator-only gate** is enforced via `getCreatorsTransaction` at line 239, meaning only the transaction's creator can reach the vulnerable code path: [7](#0-6) 

**The NATS emit helper** publishes to the `TRANSACTION_STATUS_UPDATE` subject unconditionally when called: [8](#0-7) 

## Impact Explanation
Every call with `{ "approversArray": [] }`:
1. Passes DTO validation (no `@ArrayMinSize(1)` guard).
2. Enters and exits the `dataSource.transaction` block with zero writes.
3. Publishes a `TransactionStatusUpdate` NATS event for a transaction whose state was not modified.

Downstream consumers — the WebSocket gateway and notification service — receive and process these events, triggering unnecessary re-fetches and UI updates for all connected clients observing that transaction. This is a concrete logic integrity failure: a state-change notification is emitted when no state change occurred.

## Likelihood Explanation
The precondition is minimal: any authenticated, verified user who has created at least one transaction can trigger this. No admin keys, no leaked credentials, and no privileged access are required. The endpoint is reachable via a standard authenticated HTTP POST.

## Recommendation
Add `@ArrayMinSize(1)` to `approversArray` in `CreateTransactionApproversArrayDto`, consistent with the pattern already used in `CreateTransactionObserversDto` and the nested `approvers` field:

```typescript
export class CreateTransactionApproversArrayDto {
  @IsArray()
  @ArrayMinSize(1)          // ← add this
  @ValidateNested({ each: true })
  @Type(() => CreateTransactionApproverDto)
  approversArray: CreateTransactionApproverDto[];
}
```

Additionally, consider making `emitTransactionStatusUpdate` in `createTransactionApprovers` conditional on at least one approver having been created (mirroring the `updated` flag pattern used in `updateTransactionApprover`), as defense-in-depth.

## Proof of Concept
```http
POST /transactions/42/approvers HTTP/1.1
Authorization: Bearer <valid_creator_jwt>
Content-Type: application/json

{ "approversArray": [] }
```

Expected (correct) behavior: HTTP 400 — validation error, no NATS event emitted.

Actual behavior: HTTP 201 — zero DB writes, `TransactionStatusUpdate` NATS event emitted for transaction `42`.
