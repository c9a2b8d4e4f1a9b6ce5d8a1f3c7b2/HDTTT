# Audit Report

## Title
Transaction Creator Can Delete Approvers Belonging to Any Other Transaction via Missing Cross-Transaction Authorization in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user owns `:transactionId`, but then deletes the approver record identified by `:id` without confirming that approver belongs to the authorized transaction. Any authenticated transaction creator can therefore delete approvers from transactions they do not own.

## Finding Description
**Root cause:** In `ApproversController.removeTransactionApprover` (`approvers.controller.ts`, lines 102–113), ownership is verified against `transactionId`, but the separate `id` parameter is forwarded directly to `ApproversService.removeTransactionApprover` with no membership check:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes by id — no membership check
```

In `ApproversService.removeTransactionApprover` (`approvers.service.ts`, lines 533–544), the service only verifies the approver record exists, then deletes it:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id); // deletes whatever id points to
  ...
}
```

There is no assertion that `approver.transactionId === transactionId` (or that the approver's root node belongs to `transactionId`). This is in direct contrast to `updateTransactionApprover` (`approvers.service.ts`, lines 386–391), which correctly calls `getRootNodeFromNode` and asserts:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
```

The delete path has no equivalent guard. Additionally, child approvers store `transactionId = null` (line 311), so a naive field comparison would be insufficient — the correct fix requires root-node traversal, as done in the update path.

## Impact Explanation
An attacker can silently remove required approvers from any transaction in the system. In the Hedera Transaction Tool, approvers gate whether a transaction may proceed to execution. Removing them can:
- Allow a transaction to advance to execution without required approval signatures.
- Permanently corrupt the approval workflow of another user's transaction.
- Cause transactions requiring multi-party approval to execute unilaterally.

This is an unauthorized state mutation with direct impact on transaction integrity and the trust model of the approval system.

## Likelihood Explanation
Any authenticated, verified user who has created at least one transaction can exploit this. No admin privileges, leaked secrets, or special network access are required. Approver IDs are sequential integers and discoverable via `GET /transactions/:id/approvers` for any transaction the attacker participates in. The attack requires only a crafted HTTP DELETE request.

## Recommendation
In `ApproversService.removeTransactionApprover`, add a cross-transaction membership check mirroring the one in `updateTransactionApprover`:

1. After fetching the approver, call `getRootNodeFromNode(approver.id)` to obtain the root node.
2. Assert `rootNode.transactionId === transactionId` (where `transactionId` is passed as a parameter to the service method).
3. Throw `UnauthorizedException` if the check fails.

Alternatively, pass `transactionId` into `removeTransactionApprover` and perform the root-node check there, keeping the authorization logic inside the service layer consistent with the update path.

## Proof of Concept
1. Attacker is an authenticated, verified user and creator of transaction A (`transactionId=1`).
2. Attacker observes approver record ID X belonging to victim's transaction B (e.g., via `GET /transactions/:id/approvers` for any transaction they can view, or by integer enumeration).
3. Attacker sends: `DELETE /transactions/1/approvers/X`
4. `getCreatorsTransaction(1, attacker)` passes — attacker owns transaction 1.
5. `removeTransactionApprover(X)` executes — deletes approver X, which belongs to transaction B, with no error.
6. Transaction B's approval workflow is silently corrupted.
