All referenced code is confirmed in the repository. The vulnerability is real and accurately described.

Audit Report

## Title
`IpThrottlerGuard` Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header

## Summary
`IpThrottlerGuard.getTracker` derives its Redis throttle key from the raw, client-supplied `X-Forwarded-For` HTTP header before falling back to `req.ip`. Because this header is fully attacker-controlled, any client can rotate its value on every request to obtain a perpetually fresh counter, bypassing the global rate limit entirely.

## Finding Description

**Root cause — `IpThrottlerGuard.getTracker`**

```ts
// back-end/apps/api/src/guards/ip-throttler.guard.ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;
}
```

`req.headers['x-forwarded-for']` is the raw HTTP header as sent by the client. It takes unconditional priority over `req.ip`, which Express populates only after applying the configured `trust proxy` chain. Because the raw header is never validated or sanitised, any string the attacker places there becomes the Redis key used to track request counts. [1](#0-0) 

**Global guard registration**

`IpThrottlerGuard` is registered as a global `APP_GUARD`, meaning it is the rate-limit layer for every HTTP route in the API service. [2](#0-1) 

**Throttler configuration**

Production and default deployments configure the guard with `GLOBAL_MINUTE_LIMIT=10000` and `GLOBAL_SECOND_LIMIT=1000`. [3](#0-2) [4](#0-3) 

## Impact Explanation

`IpThrottlerGuard` is the **only** rate-limit layer protecting all non-auth routes (transaction creation, key management, observer/signer assignment, notification queries, etc.). Bypassing it allows:

- **Brute-force / enumeration at scale** against any API endpoint with zero server-side throttling.
- **Resource exhaustion** of the PostgreSQL connection pool and NestJS worker threads, degrading service for all legitimate users.

The `EmailThrottlerGuard` on `/auth/login` and `/auth/reset-password` is keyed by `req.body.email` and is unaffected by this bypass — but all other routes lose their only rate-limit protection. [5](#0-4) 

## Likelihood Explanation

- **Attacker preconditions**: none — any unauthenticated HTTP client can set arbitrary headers.
- **Effort**: a single `curl -H 'X-Forwarded-For: <rotating-value>'` or a trivial loop script.
- **Detection difficulty**: high — each request arrives with a different apparent IP, making log-based detection and IP-based blocking ineffective.
- Even when Traefik appends the real client IP, the raw header value read by the guard is the full comma-separated string (e.g., `fake-ip, real-client-ip`). Each unique string is a distinct Redis key, so the bypass still works behind the proxy.

## Recommendation

Replace the raw header read with Express's `req.ip`, which is already proxy-aware (respects the `trust proxy` setting) and cannot be spoofed by the client when the proxy chain is correctly configured:

```ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

Additionally, ensure Express's `trust proxy` is set to the exact number of trusted proxy hops (e.g., `app.set('trust proxy', 1)` for a single Traefik hop) so that `req.ip` is correctly populated with the real client IP rather than the proxy's address.

## Proof of Concept

```bash
# Each request uses a different X-Forwarded-For value → fresh counter every time
for i in $(seq 1 50000); do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: attacker-ip-${i}" \
    -H "x-frontend-version: 1.0.0" \
    https://<target>/api/transactions
done
# All 50 000 requests succeed; no 429 is ever returned.
```

Each value `attacker-ip-1`, `attacker-ip-2`, … becomes a separate Redis key. The per-key counter never reaches the 10 000/minute limit, so the throttler never fires. [1](#0-0) [6](#0-5)
