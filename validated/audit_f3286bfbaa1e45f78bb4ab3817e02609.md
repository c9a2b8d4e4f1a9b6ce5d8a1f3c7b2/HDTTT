### Title
Cryptographically Insecure PRNG (`Math.random()`) Used to Generate Temporary User Passwords in `AuthService`

### Summary
The `generatePassword()` method in `back-end/apps/api/src/auth/auth.service.ts` uses `Math.random()` — a non-cryptographically-secure PRNG — to produce temporary passwords assigned to newly registered users. These passwords are stored as the user's initial credential and emailed to them. An attacker who can observe or predict the V8 xorshift128+ PRNG state can enumerate the temporary password for any newly registered account and authenticate as that user before they change it.

### Finding Description

**Root cause:** `Math.random()` is explicitly documented by MDN and the ECMAScript specification as not suitable for security-sensitive operations. V8 implements it using xorshift128+, a 128-bit state PRNG seeded once per process. Its output is predictable if any prior outputs from the same process are observable.

**Exact location:**

```typescript
// back-end/apps/api/src/auth/auth.service.ts, lines 147–154
private generatePassword() {
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () => String.fromCharCode(97 + Math.floor(Math.random() * 26))).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
``` [1](#0-0) 

The output space is 26^10 ≈ 1.4 × 10^14 combinations, but because `Math.random()` is seeded once per Node.js process and its internal state can be reconstructed from observed outputs, the effective security is far lower than this nominal figure.

**Exploit path:**

1. Admin calls `POST /auth/signup` to register a new user. [2](#0-1) 
2. `signUpByAdmin()` calls `this.generatePassword()` to produce `tempPassword`. [3](#0-2) 
3. `tempPassword` is stored as the user's hashed password and emailed to them. [4](#0-3) 
4. Until the user logs in and changes their password via `PATCH /auth/change-password`, the account is protected solely by this weak credential. [5](#0-4) 

**Other `Math.random()` usages** in `email.service.ts` (retry jitter, line 136) and `cache-management.service.ts` (cron jitter, line 58) are not security-sensitive and are not part of this finding.

### Impact Explanation
An attacker who reconstructs the PRNG state (by observing any prior `Math.random()` output from the same server process) can predict the temporary password of any newly registered user. With that password they can call `POST /auth/login`, obtain a valid JWT, and act as that user — including accessing Hedera transaction signing workflows, approving or submitting transactions, and reading all organization data visible to that user — before the victim ever logs in.

### Likelihood Explanation
The attack requires the attacker to:
1. Know or infer when a new user was registered (observable via timing or social engineering).
2. Recover the PRNG state. This is feasible: xorshift128+ state recovery from 3 consecutive 32-bit outputs is well-documented (Lemire et al., "Cracking `Math.random()`"). If the server exposes any endpoint that leaks `Math.random()` outputs (e.g., nonces, IDs, tokens generated with `Math.random()`), state recovery is straightforward. Even without that, the nominal 26^10 space is brute-forceable offline given a known bcrypt/argon2 hash — but the hash is not exposed, so the primary path is PRNG state recovery.
3. Authenticate before the victim changes their password. New users are in `UserStatus.NEW` and are expected to log in and change their password; this window may be hours to days in practice.

Likelihood is **medium**: requires PRNG state leakage or a brute-force window, but the window is real and the technique is publicly documented.

### Recommendation

Replace `Math.random()` with Node.js's `crypto` module, which provides a CSPRNG:

```typescript
import { randomBytes } from 'crypto';

private generatePassword(): string {
  const alphabet = 'abcdefghijklmnopqrstuvwxyz';
  const getRandomLetters = (length: number): string =>
    Array.from(
      { length },
      () => alphabet[randomBytes(1)[0] % alphabet.length],
    ).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
```

Long-term, consider replacing persistent temporary passwords with short-lived, single-use invitation tokens (e.g., a signed JWT with a 24-hour expiry that forces immediate password set on first use), consistent with the recommendation in the referenced report.

### Proof of Concept

1. Observe any sequence of `Math.random()` outputs from the target Node.js process (e.g., via a timing oracle or any endpoint using `Math.random()` for non-secret purposes).
2. Use the published xorshift128+ state-recovery algorithm to reconstruct the 128-bit PRNG state.
3. Fast-forward the recovered state to the point in time when `generatePassword()` was called for the target user (correlate with registration timestamp).
4. Reproduce the 10 `Math.random()` calls made inside `generatePassword()` to derive the exact `xxxxx-xxxxx` password.
5. Call `POST /auth/login` with `{ email: <target>, password: <derived> }`.
6. Receive a valid JWT and operate as the target user. [6](#0-5)
