### Title
Unbounded Per-Transaction Async Loop in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **all** non-terminal transactions from the database without any row limit, then performs expensive async operations (mirror-node account lookups, DB queries) for every transaction in a sequential loop before applying pagination. An authenticated user can accumulate transactions to make this endpoint progressively slower until it times out for all users.

### Finding Description

**Root cause — unbounded fetch + per-item async work:**

`getTransactionsToSign` issues an uncapped `repo.find` at line 295: [1](#0-0) 

No `take` / `limit` clause is passed, so the ORM returns every non-terminal transaction in the database that matches the filter. The result set is then iterated one-by-one: [2](#0-1) 

`userKeysToSign` → `keysRequiredToSign` → `transactionSignatureService.computeSignatureKey` resolves the signing key structure for the transaction, which internally issues per-account mirror-node HTTP calls (`addSigningAccountKeys`, `addReceiverAccountKeys`): [3](#0-2) 

Pagination (`offset`, `limit`) is applied **after** the entire loop completes: [4](#0-3) 

**Exploit path:**

1. Attacker registers as a normal user (no privilege required).
2. Attacker creates N transactions (e.g. `AccountUpdateTransaction`) targeting accounts with complex key structures (deep `KeyList` trees), maximising the per-iteration mirror-node call count.
3. Transactions remain in `WAITING_FOR_SIGNATURES` status indefinitely until they expire (attacker can set a far-future `validStart`).
4. Every call to `GET /transactions/sign?page=1&size=10` by any user must now iterate all N transactions, issuing O(N × accounts_per_tx) mirror-node calls before returning the first page.
5. As N grows, response time grows linearly until the HTTP gateway times out, making the endpoint permanently unavailable.

### Impact Explanation

- **Service unavailability**: The sign-queue endpoint becomes unusable for all organisation users once enough transactions accumulate.
- **Cascading effect**: Because `userKeysToSign` is called sequentially (not in parallel), a single slow mirror-node response per transaction multiplies across the entire dataset.
- **Persistence**: Transactions are not automatically cleaned up; the attacker's payload persists until expiry, requiring no sustained effort after the initial creation burst.

### Likelihood Explanation

- Any authenticated user can call `POST /transactions` to create transactions — no admin role required.
- The Hedera network imposes no meaningful cost on creating unsigned, never-submitted transactions stored only in the back-end database.
- The attack is gradual and hard to distinguish from legitimate usage until the endpoint degrades.

### Recommendation

1. **Push pagination into the database**: pass `take: limit` and `skip: offset` directly to `repo.find` so the DB returns only the page needed, then call `userKeysToSign` only for those rows.
2. **Separate count from data**: issue a `repo.count` for `totalItems` and a limited `repo.find` for the page, mirroring the pattern already used by other paginated endpoints in the service.
3. **Cap transaction creation**: enforce a per-user limit on the number of active (non-terminal) transactions to bound the worst-case loop size.
4. **Adopt the SQL-based approach**: the newer `getTransactionNodesQuery` path used by `TransactionNodesService` pushes all filtering into a single SQL query and avoids the in-memory loop entirely — the same approach should be applied here. [5](#0-4) 

### Proof of Concept

**Preconditions**: valid JWT for a normal (non-admin) user.

```
# Step 1 – create N transactions with complex key structures
for i in $(seq 1 5000); do
  curl -s -X POST https://<host>/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"transactionBytes":"<AccountUpdateTransaction bytes with deep KeyList>","mirrorNetwork":"mainnet",...}'
done

# Step 2 – observe degradation on the sign endpoint for ANY user
time curl -s "https://<host>/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer $VICTIM_TOKEN"
# Response time grows O(N) until gateway timeout (e.g. 30 s)
```

**Expected outcome**: after accumulating enough transactions, `GET /transactions/sign` returns HTTP 504 (gateway timeout) for all users, making the signing workflow permanently unavailable until transactions expire or are manually purged.
