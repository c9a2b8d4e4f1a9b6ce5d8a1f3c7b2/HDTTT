Audit Report

## Title
Unauthenticated Cross-Tenant Data Exposure via Unscoped `GET /transactions/history` Endpoint

## Summary
The `GET /transactions/history` endpoint returns all historical transactions across all users without any per-user ownership filter. Any authenticated user can paginate through the complete historical transaction record of the entire organization. The sibling `GET /transactions` endpoint correctly scopes results to the requesting user; `getHistoryTransactions` omits this scoping entirely.

## Finding Description

**Root cause confirmed in controller (`transactions.controller.ts` lines 142–154):**

The `/history` route handler has no `@GetUser()` decorator and passes no user identity to the service. [1](#0-0) 

Compare with `getTransactions` (lines 119–132), which correctly injects `@GetUser() user: User` and passes it to the service: [2](#0-1) 

**Root cause confirmed in service (`transactions.service.ts` lines 220–249):**

`getHistoryTransactions` accepts no `user` parameter and builds a `findOptions` object with only a status filter — no `signers.userId`, `observers.userId`, or `creatorKey.userId` clause: [3](#0-2) 

The correct pattern is visible in `getTransactions` (lines 159–173), which builds a `whereForUser` array scoped to all three ownership relations: [4](#0-3) 

**Compounding design issue — `verifyAccess` (lines 786–799):**

The `verifyAccess` method, used by the single-transaction `GET /:id` endpoint, explicitly returns `true` for every terminal status (EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED) without checking user ownership. This means the unscoped bulk-history endpoint is consistent with a broader design decision that treats all historical transactions as visible to any authenticated user — reinforcing rather than mitigating the exposure. [5](#0-4) 

## Impact Explanation
Any authenticated user — including low-trust members — can retrieve the complete historical transaction record of the entire organization: transaction IDs, types, raw transaction bytes, valid-start timestamps, mirror network identifiers, and group membership. Transaction bytes may encode account IDs, amounts, and key structures used in past Hedera treasury operations. Because `verifyAccess` also unconditionally grants access to individual historical transactions, the exposure is systemic across both the bulk-list and single-record endpoints.

## Likelihood Explanation
Exploitation requires only a valid user account — the lowest possible privilege level. The endpoint is a standard paginated REST `GET`, reachable by any HTTP client that can authenticate. No special tooling, timing, or privilege escalation is needed. The attack is silent and indistinguishable from normal API usage in access logs.

## Recommendation
1. **Add user scoping to `getHistoryTransactions`**: inject `@GetUser() user: User` in the controller and pass it to the service, then build a `whereForUser` array (mirroring `getTransactions`) that filters by `signers.userId`, `observers.userId`, and `creatorKey.userId`.
2. **Revisit `verifyAccess` for terminal statuses**: the unconditional `return true` for all terminal statuses means any authenticated user can also fetch any individual historical transaction via `GET /:id`. Ownership should be checked even for completed transactions unless the design explicitly intends organization-wide visibility.
