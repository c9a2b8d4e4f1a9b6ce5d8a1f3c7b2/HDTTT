### Title
`removeTransaction` Bypasses Status Enforcement That `cancelTransaction` Correctly Enforces, Allowing Creators to Force-Cancel Terminal-State Transactions

---

### Summary

`cancelTransaction` (exposed via `PATCH /transactions/cancel/:id`) enforces a `cancelableStatuses` allowlist before mutating a transaction's state. The parallel deletion path `removeTransaction` (exposed via `DELETE /transactions/:id`) performs the identical state mutation — forcibly setting status to `CANCELED` — but contains **no status check whatsoever**. A transaction creator can therefore call `DELETE /transactions/:id` to corrupt the recorded status of a transaction that is already `EXECUTED`, `FAILED`, `EXPIRED`, `ARCHIVED`, or `REJECTED`, breaking audit integrity and state invariants.

---

### Finding Description

**Root cause — inconsistent enforcement across two parallel state-mutation paths:**

`cancelTransactionWithOutcome` enforces a strict allowlist before any mutation:

```
cancelableStatuses = [NEW, WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]
``` [1](#0-0) 

If the transaction is not in one of those statuses, a `BadRequestException` is thrown and no mutation occurs. [2](#0-1) 

`removeTransaction`, the parallel path, performs the same `status = CANCELED` write but with **zero status validation**:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true) {
  const transaction = await this.getTransactionForCreator(id, user);
  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED }); // ← no guard
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction); // ← hard delete, also no guard
  }
  ...
}
``` [3](#0-2) 

This function is directly wired to the public `DELETE /transactions/:id` endpoint: [4](#0-3) 

The only check performed is `getTransactionForCreator`, which verifies ownership — not status. [5](#0-4) 

A second exposure exists in `removeTransactionGroup`, which calls `removeTransaction(..., false)` (hard delete) for every transaction in a group, also with no status check: [6](#0-5) 

---

### Impact Explanation

A transaction creator can:

1. Submit a transaction that reaches `EXECUTED` status on the Hedera network.
2. Immediately call `DELETE /transactions/:id` on the backend record.
3. The backend sets `status = CANCELED` and soft-deletes the row.

**Concrete consequences:**
- The audit log now shows the transaction as `CANCELED` even though it was executed on-chain. Other signers, approvers, and observers lose the true outcome record.
- Approvers and observers who relied on the backend state for compliance or reconciliation receive a false picture.
- The chain service emits a `CANCELED` status update notification to all subscribers for a transaction that actually executed.
- For the group path (`removeTransactionGroup`), the record is hard-deleted entirely — no trace remains in the database of an executed transaction.

---

### Likelihood Explanation

- **Attacker precondition:** Authenticated user who is the creator of any transaction. No privileged role required.
- **Trigger:** A single `DELETE /transactions/:id` HTTP call after the transaction has been executed.
- **Reachability:** The endpoint is fully public to any verified user with a key. [4](#0-3) 
- **Detectability:** The test suite for `removeTransaction` does not test any terminal-status transaction — it only mocks a transaction with no status field set, confirming the gap is untested. [7](#0-6) 

---

### Recommendation

Add the same `cancelableStatuses` guard to `removeTransaction` before performing any mutation. For the soft-remove path, reject the call if the transaction is not in a cancelable status. For the hard-remove path (used by `removeTransactionGroup`), either apply the same guard or introduce a separate `forceRemove` path that is not reachable by normal users.

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true) {
  const transaction = await this.getTransactionForCreator(id, user);

  if (!this.cancelableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OTIP);
  }

  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction);
  }
  ...
}
```

---

### Proof of Concept

1. Authenticate as a normal verified user with at least one key.
2. Create a transaction via `POST /transactions`.
3. Wait for or simulate the transaction reaching `EXECUTED` status (e.g., via the chain service or a direct DB update in a test environment).
4. Call `DELETE /transactions/:id` with the creator's auth token.
5. **Expected (correct) behavior:** `400 Bad Request` — transaction is not in a cancelable state.
6. **Actual behavior:** `200 OK` — the backend sets `status = CANCELED` and soft-deletes the record, overwriting the `EXECUTED` state.

Confirm by querying the DB with `withDeleted: true` — the row will show `status = CANCELED` and a non-null `deletedAt`, despite having been executed on-chain.
