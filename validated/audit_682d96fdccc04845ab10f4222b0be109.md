Audit Report

## Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header Renders Global Throttler Ineffective

## Summary
The global `IpThrottlerGuard` registered as `APP_GUARD` derives its throttle tracking key from the raw `X-Forwarded-For` HTTP request header, which any client can forge. Because this header takes unconditional priority over `req.ip`, an attacker can rotate the header value on every request to appear as a new IP, completely bypassing the only global rate-limiting protection across all API endpoints.

## Finding Description

`IpThrottlerGuard` is registered as a global `APP_GUARD` in `api.module.ts`, making it the sole rate-limiting protection for every HTTP route: [1](#0-0) 

The guard's `getTracker()` method resolves the throttle key as follows:

```ts
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [2](#0-1) 

`req.headers['x-forwarded-for']` is read directly from the raw HTTP headers — not from Express's trusted proxy-processed `req.ips` — and takes unconditional priority over `req.ip`. Because `X-Forwarded-For` is a plain HTTP header, any client can set it to an arbitrary value. The throttler stores hit counts keyed by this value, so an attacker who sends a different `X-Forwarded-For` value on each request (e.g., `1.2.3.1`, `1.2.3.2`, …) is treated as a distinct IP each time and never accumulates hits against the configured limits.

The `IpThrottlerModule` configures two throttlers (`global-minute` and `global-second`) whose limits are read from environment variables: [3](#0-2) 

Both limits are rendered meaningless once the tracking key is attacker-controlled.

The `EmailThrottlerGuard` is applied only to three auth endpoints:
- `/auth/signup` — requires admin JWT authentication first, limiting exposure
- `/auth/login` — keyed on `req.body.email`
- `/auth/reset-password` — keyed on `req.body.email` [4](#0-3) [5](#0-4) 

All other controllers carry no per-route throttle guard beyond the bypassable global IP guard.

Additionally, `UserThrottlerModule` and `UserThrottlerGuard` are defined but `UserThrottlerModule` is never imported into `api.module.ts` (confirmed: it appears only in its own file definition), and `UserThrottlerGuard` is not exported from the guards barrel (`guards/index.ts`), so they provide no protection: [6](#0-5) [7](#0-6) 

## Impact Explanation

An unauthenticated or authenticated attacker can send unlimited requests to any API endpoint:

- **OTP/reset-password flooding**: `/auth/reset-password` triggers email dispatch on each call. With IP throttling bypassed and email throttling bypassable by rotating target email addresses, an attacker can cause mass email spam and exhaust email-sending quotas.
- **Authentication brute-force at scale**: The email throttler on `/auth/login` limits attempts per email address, but with IP throttling bypassed, an attacker can flood the server with login attempts against many different email addresses simultaneously without any IP-level restriction.
- **Transaction and signature endpoint DoS**: `POST /transactions`, `POST /transactions/:id/signers`, `POST /transaction-groups`, and all other business-logic endpoints have no effective rate limiting. An attacker can flood these endpoints to exhaust database connections, CPU, and memory, causing service unavailability for all legitimate users.
- **Unbounded resource creation**: Authenticated users can create unlimited transactions, signature maps, and approver records, causing database bloat and degraded performance.

## Likelihood Explanation

The precondition is zero: setting an HTTP header requires no privilege, no leaked credential, and no special tooling. Any HTTP client (`curl`, Python `requests`, `k6`) can rotate `X-Forwarded-For` values in a loop. The attack is fully scriptable and deterministic. No cryptographic primitive needs to be broken.

## Recommendation

1. **Do not trust the raw `X-Forwarded-For` header directly.** If the application sits behind a trusted reverse proxy, configure Express's `trust proxy` setting appropriately and use `req.ip` (which Express populates correctly after proxy trust is configured) instead of reading `req.headers['x-forwarded-for']` directly.
2. **Update `getTracker()` to use only `req.ip`**, removing the `req.headers['x-forwarded-for']` branch:
   ```ts
   const clientIp = req.ip;
   ```
3. **Import and apply `UserThrottlerModule` and `UserThrottlerGuard`** to authenticated routes to add a secondary, non-bypassable rate limit layer keyed on authenticated user identity.
4. **Validate and pin the `X-Forwarded-For` header** at the reverse proxy/load balancer layer, stripping any client-supplied values before they reach the application.

## Proof of Concept

```bash
# Bypass IP rate limiting by rotating X-Forwarded-For on every request
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://target/auth/reset-password \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 1.2.3.$i" \
    -d '{"email":"victim@example.com"}'
done
```

Each request presents a unique IP to the throttler. The server processes all 1000 requests, triggering 1000 password-reset emails, while the `global-minute` and `global-second` limits are never reached for any single tracked key.
