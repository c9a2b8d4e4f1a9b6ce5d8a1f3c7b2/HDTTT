### Title
Unbounded In-Memory Array Accumulation in `getTransactionsToSign` Causes Authenticated DoS

### Summary
`getTransactionsToSign` in the back-end API fetches **all** non-terminal transactions from the database with no row limit, loads them entirely into server memory, and then performs expensive per-transaction deserialization and cryptographic key computation on each one before slicing for pagination. Any authenticated user can repeatedly call `GET /transactions/sign` to exhaust server memory and CPU, causing service degradation or crash.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method accepts pagination parameters but applies them only **after** loading the full unbounded result set:

```typescript
// Line 295-299: No `take` limit — fetches every non-terminal transaction
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});

// Line 301-308: Per-transaction deserialization + crypto for every row
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}

// Line 311-316: Pagination applied AFTER full load
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [1](#0-0) 

The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`) but is **not scoped to the requesting user's transactions** — it returns all non-terminal transactions across the entire organization. [2](#0-1) 

Each iteration calls `userKeysToSign`, which internally calls `SDKTransaction.fromBytes(transaction.transactionBytes)` (deserialization) and `transactionSignatureService.computeSignatureKey` (cryptographic key tree traversal), making the per-row cost non-trivial. [3](#0-2) 

The endpoint is exposed at `GET /transactions/sign` and is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin privilege required. [4](#0-3) 

Contrast this with `getTransactions` and `getHistoryTransactions`, which correctly pass `skip: offset, take: limit` into the database query before fetching. [5](#0-4) 

A secondary unbounded fetch exists in `UsersService.getUsers`, which calls `this.repo.find()` with no limit and returns all organization users in a single response. [6](#0-5) 

### Impact Explanation

- **Memory exhaustion**: Every non-terminal transaction row (including its `transactionBytes` blob and `groupItem` relation) is loaded into the Node.js heap simultaneously. In an organization with thousands of pending transactions, this can exhaust available memory.
- **CPU exhaustion**: Each row triggers `SDKTransaction.fromBytes` (protobuf deserialization) and cryptographic key-tree traversal. Concurrent requests multiply this cost.
- **Service crash / severe degradation**: The NestJS API process can OOM-crash or become unresponsive, denying service to all users.
- The endpoint comment `/* NO LONGER USED BY FRONT-END */` confirms it is vestigial but still fully reachable by any API client. [7](#0-6) 

### Likelihood Explanation

- **Attacker precondition**: A valid JWT session — obtainable by any registered user via normal login.
- **Attack complexity**: A single HTTP GET to `/transactions/sign?page=1&size=1` is sufficient to trigger the full unbounded load regardless of the requested page size.
- **Amplification**: Concurrent requests from one or more accounts multiply the effect linearly.
- No rate limiting is present on this endpoint in the reviewed code.

### Recommendation

1. **Apply the database-level limit before fetching**, mirroring the pattern used in `getTransactions`:
   ```typescript
   const transactions = await this.repo.find({
     where: whereForUser,
     relations: ['groupItem'],
     order,
     skip: offset,
     take: limit,
   });
   ```
   Because `userKeysToSign` filters further, `totalItems` must be computed separately (e.g., a `count` query with the same `where`), or the endpoint must be redesigned to push the user-key filter into the database query.

2. **Scope the query to the requesting user** by joining on `signers.userId = user.id` or `observers.userId = user.id`, reducing the candidate set before in-memory filtering.

3. **Remove or gate the endpoint** if it is truly no longer used by the front-end, to eliminate the attack surface entirely.

4. Apply the same fix to `UsersService.getUsers` — add pagination or a hard cap.

### Proof of Concept

**Precondition**: Attacker has a valid JWT (any registered, verified user).

**Steps**:
1. Register or log in to obtain a JWT.
2. Ensure the organization has a large number of non-terminal transactions (or create them via `POST /transactions`).
3. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <jwt>
   ```
4. Each request causes the server to load **all** non-terminal transactions into memory and run `SDKTransaction.fromBytes` + key-tree traversal on every row.
5. With sufficient concurrent requests or a large transaction table, the API process exhausts memory or CPU, causing 500 errors or an OOM crash for all users.

**Expected outcome**: Server memory and CPU spike proportionally to the number of non-terminal transactions in the database, with no per-request bound enforced.
