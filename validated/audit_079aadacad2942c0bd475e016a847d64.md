Audit Report

## Title
IDOR: `removeTransactionApprover` Deletes Approvers Across Transaction Boundaries Without Ownership Check

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController` authorizes the caller against `transactionId` but then deletes the approver identified by the unrelated `id` parameter with no verification that the approver belongs to that transaction. Any authenticated user who owns at least one transaction can delete approvers from any other user's transaction.

## Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent, uncorrelated operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
``` [1](#0-0) 

The service-level `removeTransactionApprover` simply fetches the approver by `id` and soft-deletes it via `removeNode`, with no reference to `transactionId` whatsoever:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

By contrast, `updateTransactionApprover` correctly resolves the root node of the approver and validates that `rootNode.transactionId === transactionId` before proceeding:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

`getCreatorsTransaction` only enforces that the caller is the creator of the transaction identified by `transactionId` — it makes no assertion about the `id` (approver) parameter: [4](#0-3) 

The root cause is a classic **Insecure Direct Object Reference (IDOR)**: the authorization gate (`transactionId`) and the resource being mutated (`id`) are decoupled, so passing the gate on one object grants write access to any object of the same type.

## Impact Explanation

An attacker can silently remove any approver from any transaction they do not own. This directly undermines the multi-signature approval workflow:

- Required approvers can be stripped from a victim's transaction, allowing it to advance past the approval gate with fewer approvals than intended.
- The approval tree of another user's transaction can be permanently corrupted, making it unexecutable or causing it to execute without the required consent. [5](#0-4) 

## Likelihood Explanation

The precondition is minimal: any registered, verified user who has created at least one transaction. No admin role, no leaked credentials, and no privileged access are required. The attack is a single crafted HTTP `DELETE` request with a valid JWT token. The approver `id` values are sequential integers, making enumeration trivial. [6](#0-5) 

## Recommendation

Mirror the fix already present in `updateTransactionApprover`: before calling `removeNode`, resolve the root node of the approver and assert that its `transactionId` matches the `transactionId` URL parameter.

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root and verify ownership boundary
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover` and update the call site accordingly. [3](#0-2) 

## Proof of Concept

1. Alice registers and creates **Transaction A** (she is its creator; `transactionId = A`).
2. Bob registers and creates **Transaction B**, then adds an approver (e.g., `id = 99`) to it.
3. Alice sends:
   ```
   DELETE /transactions/{A}/approvers/99
   Authorization: Bearer <alice_jwt>
   ```
4. `getCreatorsTransaction(A, alice)` succeeds — Alice is the creator of Transaction A.
5. `removeTransactionApprover(99)` executes with no further checks, soft-deleting Bob's approver (id=99) from Transaction B.
6. Bob's transaction now has a corrupted approval tree with no error or notification to Bob. [7](#0-6)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
