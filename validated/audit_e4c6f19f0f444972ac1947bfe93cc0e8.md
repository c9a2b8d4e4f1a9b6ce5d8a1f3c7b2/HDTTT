### Title
Any Authenticated User Can Post Comments on Transactions They Have No Access To

### Summary

`CommentsController.createComment()` contains an explicit `//TODO` comment acknowledging that no access control check exists to verify whether the calling user is a participant of the target transaction before allowing them to post a comment. This is a direct analog to the external report: just as the CrossAnchorBridge had commented-out `require` whitelist checks, this codebase has a self-documented, unimplemented authorization check on a state-mutating endpoint.

### Finding Description

The `CommentsController` applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level — these only verify that the caller is a valid, non-blacklisted, verified user. They do not verify that the user has any relationship to the specific transaction being commented on.

The `createComment` handler explicitly acknowledges this gap:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
``` [1](#0-0) 

The rest of the application enforces a strict access model: `verifyAccess()` in `TransactionsService` checks that a user is the creator, a signer, an observer, or an approver of a transaction before granting access. [2](#0-1) 

This check is used correctly in `getTransactionWithVerifiedAccess()` and `importSignatures()`, but is entirely absent from the `createComment` path. [3](#0-2) 

The `getComments` and `getCommentById` endpoints on the same controller also lack per-transaction access control, allowing any authenticated user to read comments on any transaction. [4](#0-3) 

### Impact Explanation

Any authenticated, verified organization user can:
1. Post comments on transactions they have no relationship to, polluting the audit trail and communication channel of sensitive multi-signature workflows.
2. Read comments on any transaction, leaking internal deliberation, key management decisions, and operational context that should be restricted to transaction participants.

The comment thread is part of the transaction's collaborative workflow. Unauthorized writes break the integrity of that record; unauthorized reads break confidentiality.

### Likelihood Explanation

Exploitation requires only a valid JWT token (i.e., any registered, verified user in the organization). The attacker needs only to know or enumerate a `transactionId` (integer, trivially guessable by incrementing). No elevated privileges, no race conditions, no special tooling — a single authenticated HTTP `POST` to `/transactions/:transactionId/comments` suffices.

### Recommendation

Before calling `commentsService.createComment()`, verify the user has access to the transaction using the existing `verifyAccess()` method from `TransactionsService`:

```typescript
@Post()
async createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  const transaction = await this.transactionsService.getTransactionById(transactionId);
  if (!(await this.transactionsService.verifyAccess(transaction, user))) {
    throw new UnauthorizedException();
  }
  return this.commentsService.createComment(user, transactionId, dto);
}
```

Apply the same check to `getComments` and `getCommentById`.

### Proof of Concept

1. Register two users (User A and User B) in the organization.
2. User A creates a transaction. User B is not added as a signer, observer, or approver.
3. User B authenticates and obtains a JWT.
4. User B sends:
   ```
   POST /transactions/1/comments
   Authorization: Bearer <user_b_jwt>
   Content-Type: application/json

   { "body": "Unauthorized comment" }
   ```
5. The comment is accepted and persisted. User B can also `GET /transactions/1/comments` and read all existing comments — despite having no legitimate access to the transaction. [5](#0-4)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
