### Title
Unbounded Recursive Approver Tree Causes Server Resource Exhaustion (DoS)

### Summary
The `createTransactionApprovers` endpoint accepts a deeply nested, arbitrarily large approver tree with no depth or node-count limit. The internal `createApprover` function processes each node recursively, issuing up to 7 sequential database queries per node. A malicious but authenticated transaction creator can submit a single crafted request containing hundreds of nested threshold nodes, exhausting the database connection pool and degrading or crashing the API service for all other users.

### Finding Description

**Root cause — no size or depth constraint on the approver DTO:**

`CreateTransactionApproverDto` allows unlimited recursive nesting through its `approvers` field: [1](#0-0) 

`@ArrayMinSize(1)` is present but there is no `@ArrayMaxSize` and no `@MaxDepth` equivalent. The outer array `approversArray` in `CreateTransactionApproversArrayDto` is equally unconstrained: [2](#0-1) 

**Exploit path — recursive processing with per-node DB queries:**

`createTransactionApprovers` calls the inner `createApprover` function recursively for every node in the submitted tree: [3](#0-2) 

For each node the function issues up to 7 sequential queries: `isNode` count, `findOne` for the parent, `getRootNodeFromNode` (a recursive CTE), `count` for the user, `getApproversByTransactionId` (another recursive CTE), `create`, and `insert`: [4](#0-3) 

**Why the duplicate-check does not protect against this:**

`isNode` only returns `true` (duplicate) when `typeof approver.userId === 'number'`. Pure threshold nodes (no `userId`, only `threshold` + nested `approvers`) are **never** flagged as duplicates, so an attacker can create an unlimited number of them: [5](#0-4) 

**Entry point:**

The endpoint is authenticated (`JwtAuthGuard`, `VerifiedUserGuard`) and restricted to the transaction creator, but any registered, verified user can create a transaction and then call this endpoint: [6](#0-5) 

### Impact Explanation

A single HTTP request containing a wide or deep threshold-only approver tree (e.g., 200 nodes, each triggering ~7 DB queries = ~1 400 sequential queries inside one database transaction) will:

- Hold a database connection open for the entire duration of the transaction.
- Saturate the TypeORM connection pool, causing all concurrent API requests that need a DB connection to queue or time out.
- Spike CPU on the database server due to repeated recursive CTEs (`getRootNodeFromNode`, `getApproversByTransactionId`).

The result is a denial of service for all organization users sharing the same API and database instance. The attacker pays only the cost of registering one account and creating one transaction.

### Likelihood Explanation

- **Attacker preconditions:** A valid, verified account — achievable by any user who can register on the platform.
- **Effort:** A single crafted POST request; no special tooling required.
- **Detection:** No rate limiting or request-size guard specific to the approver endpoint was found in the codebase.
- **Repeatability:** The attacker can repeat the request after each one completes (or in parallel from multiple accounts) to maintain sustained pressure.

### Recommendation

1. **Add `@ArrayMaxSize` to both `approversArray` and the nested `approvers` field** in the DTOs, e.g. a maximum of 20 root approvers and 20 children per node.
2. **Enforce a maximum tree depth** by passing a `depth` counter through the recursive `createApprover` call and throwing if it exceeds a threshold (e.g., 5).
3. **Add a global node-count guard** before entering the recursion: count total nodes in the submitted tree and reject if it exceeds a limit (e.g., 50).
4. **Apply API-level rate limiting** to the `POST /transactions/:id/approvers` endpoint.

### Proof of Concept

```
POST /transactions/1/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        { "threshold": 1, "approvers": [
            { "threshold": 1, "approvers": [
                ... (repeat 50+ levels deep, or 200 siblings wide)
            ]}
        ]}
      ]
    }
  ]
}
```

**Expected outcome:** The server enters a recursive loop issuing hundreds of sequential DB queries inside a single transaction, holding the connection pool slot for the full duration. Concurrent legitimate requests stall waiting for a free connection. Repeating the request (or sending it from multiple accounts simultaneously) degrades the service to unavailability.
