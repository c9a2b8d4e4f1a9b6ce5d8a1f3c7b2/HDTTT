### Title
Any Transaction Creator Can Delete Approvers from Transactions They Do Not Own (IDOR → Approval Workflow DoS)

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of `:transactionId`, but then deletes the approver record identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who owns at least one transaction can permanently remove approvers from any other transaction in the system, blocking its approval workflow.

### Finding Description

**Root cause — missing cross-object ownership check in `removeTransactionApprover`**

The controller delegates two independent calls:

```
// approvers.controller.ts  lines 102-113
await this.approversService.getCreatorsTransaction(transactionId, user); // ① checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // ② deletes approver by id — no transaction check
``` [1](#0-0) 

`removeTransactionApprover` fetches the approver purely by its primary key and soft-deletes it:

```
// approvers.service.ts  lines 533-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);   // no transactionId filter
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

`getTransactionApproverById` queries only on `id`:

```
// approvers.service.ts  lines 66-82
const find: FindOneOptions<TransactionApprover> = {
  relations: ['approvers'],
  where: { id },
};
``` [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly validates cross-ownership:

```
// approvers.service.ts  lines 386-394
const rootNode = await this.getRootNodeFromNode(approver.id, ...);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, ...);
``` [4](#0-3) 

The delete path has no equivalent guard.

**Exploit flow**

1. Attacker (Bob) registers an account and creates any transaction (transaction 1). This makes Bob the creator of transaction 1.
2. Bob learns approver ID `N` belonging to victim transaction 2 (IDs are sequential integers; Bob may also be an observer/approver of transaction 2 and can read them via `GET /transactions/2/approvers`).
3. Bob sends: `DELETE /transactions/1/approvers/N`
4. `getCreatorsTransaction(1, Bob)` passes — Bob owns transaction 1.
5. `removeTransactionApprover(N)` soft-deletes approver `N`, which belongs to transaction 2.
6. Transaction 2's approval threshold can no longer be met; it is permanently blocked.

### Impact Explanation

An attacker with no privileged access can permanently remove required approvers from any transaction in the system. If the deleted approver was the sole approver, or if the threshold can no longer be satisfied after deletion, the transaction can never reach `WAITING_FOR_EXECUTION` and will expire or be stuck indefinitely. This is a permanent, unrecoverable DoS on targeted transactions' approval workflows. [2](#0-1) 

### Likelihood Explanation

- **Precondition**: attacker must be an authenticated, verified user who has created at least one transaction — a normal product action.
- **Approver ID discovery**: approver IDs are auto-incremented integers. An attacker who is legitimately an observer or approver of any transaction can read approver IDs from `GET /transactions/:id/approvers`. Sequential enumeration is also trivial.
- **No race condition required**: unlike the EthenaMinting front-running scenario, this is a direct API call with no timing dependency.
- Likelihood is **high**. [1](#0-0) 

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the transaction the caller owns. Mirror the pattern already used in `updateTransactionApprover`:

```typescript
// approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // NEW: walk to root and confirm ownership
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller, which already holds the verified value. [4](#0-3) 

### Proof of Concept

**Setup**

- Admin creates transaction T2 with approver record ID = 42 (user Carol).
- Bob (unprivileged user) creates transaction T1 (any valid transaction).

**Attack**

```http
DELETE /transactions/T1/approvers/42
Authorization: Bearer <Bob's JWT>
```

**Expected (broken) behavior**

- `getCreatorsTransaction(T1, Bob)` → passes (Bob owns T1).
- `removeTransactionApprover(42)` → soft-deletes approver 42 (Carol on T2).
- Response: `200 true`

**Observed outcome**

- Approver Carol is removed from T2.
- T2's approval threshold can no longer be met.
- T2 is permanently blocked from execution.

**Verification**

```http
GET /transactions/T2/approvers
Authorization: Bearer <Admin JWT>
# Carol's approver record is gone
``` [1](#0-0) [2](#0-1)
