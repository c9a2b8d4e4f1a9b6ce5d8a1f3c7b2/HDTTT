Audit Report

## Title
Any Authenticated User Can Create and Read Comments on Arbitrary Transactions Without Authorization

## Summary
`CommentsController` in `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only JWT authentication guards at the controller level. No guard, interceptor, or service-layer check verifies that the requesting user has any permitted relationship (creator, approver, observer, signer) to the target `transactionId`. The developer explicitly acknowledged this gap with a TODO comment at line 22. All three endpoints — `createComment`, `getComments`, and `getCommentById` — are affected.

## Finding Description

**Root cause:** The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — all of which only confirm the caller holds a valid, non-blacklisted JWT with a verified account. No authorization check is performed against the `transactionId` route parameter. [1](#0-0) 

The developer-left TODO at line 22 reads:
```
//TODO need some sort of guard or check to ensure user can comment here
```
This confirms the missing check is known but unimplemented.

The `CommentsService.createComment()` performs zero access verification — it directly creates and persists the comment associated with the supplied `transactionId`: [2](#0-1) 

Similarly, `getTransactionComments` returns all comments for any `transactionId` without any user context: [3](#0-2) 

**Contrast with existing authorization patterns:** The codebase already implements proper access control elsewhere. `TransactionsService.getTransactionWithVerifiedAccess()` and `ApproversService.getVerifiedApproversByTransactionId()` both verify that the requesting user is a creator, observer, signer, or approver before granting access — but this pattern is entirely absent from the comments module. [4](#0-3) [5](#0-4) 

## Impact Explanation

- **Unauthorized state modification:** Any authenticated user can inject comments into transactions they have no business relationship with, polluting the audit trail and communication history.
- **Information disclosure:** `getComments` and `getCommentById` expose the full comment history of any transaction to any authenticated user, leaking potentially sensitive operational details (signing instructions, key references, business context) embedded in comments.
- **Integrity failure:** The comment thread is part of the transaction's organizational workflow. Unauthorized comments break the trust model that only involved parties interact with a transaction.

## Likelihood Explanation

- **Preconditions:** Attacker only needs a valid organization account — the lowest privilege level in the system.
- **Transaction ID enumeration:** IDs are sequential integers parsed via `ParseIntPipe`, making enumeration trivial with a simple loop.
- **No rate limiting** is applied to the comment endpoints.
- Any disgruntled or malicious employee with a valid account can exploit this immediately with no special tooling. [6](#0-5) 

## Recommendation

Apply the existing `getTransactionWithVerifiedAccess` pattern inside `CommentsService.createComment()` (and optionally for read endpoints) before persisting or returning data. Specifically:

1. Inject `TransactionsService` into `CommentsService`.
2. Call `transactionsService.getTransactionWithVerifiedAccess(transactionId, user)` at the start of `createComment`, `getTransactionComments`, and `getTransactionCommentById` to enforce that the requesting user is a creator, observer, signer, or approver of the target transaction before proceeding. [7](#0-6) 

## Proof of Concept

1. Register two accounts (User A and User B) in the same organization and obtain JWTs via `POST /auth/login`.
2. User A creates a transaction — note the returned `transactionId` (e.g., `42`). User B is not added as a creator, observer, signer, or approver.
3. User B sends:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <User B's JWT>
   Content-Type: application/json

   { "message": "unauthorized comment" }
   ```
4. The server responds with `201 Created` and the persisted comment — no ownership or membership check is performed.
5. User B can also read all comments:
   ```
   GET /transactions/42/comments
   Authorization: Bearer <User B's JWT>
   ```
   Returns the full comment history of a transaction User B has no role in. [8](#0-7)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
