### Title
`ApproversService`: `threshold ≤ approvers.length` invariant not enforced on the old parent after an approver is re-parented

### Summary
In `updateTransactionApprover`, when an approver is moved to a new parent (`dto.listId` is a number), the code validates the new parent but never checks or adjusts the **old** parent's threshold after the approver is detached from it. If the old parent's `threshold` was equal to its child count, removing one child leaves `threshold > approvers.length`, permanently breaking the approval invariant for that subtree.

### Finding Description

**Root cause — `approvers.service.ts`, lines 434–466**

When `dto.listId` is a number, the re-parenting path:

1. Validates the **new** parent (exists, has a threshold, is not a descendant of the approver being moved).
2. Writes the new `listId` to the database.
3. **Never touches the old parent** (`approver.listId` before the update). [1](#0-0) 

Contrast this with the `dto.listId === null` (detach) path, which **does** handle the old parent: [2](#0-1) 

The invariant `threshold ≤ approvers.length` is enforced when a threshold is set directly: [3](#0-2) 

…but there is no equivalent post-mutation check for the old parent after a re-parent operation.

**Exploit path**

1. Transaction creator builds a threshold tree: `Parent (threshold=2)` → children `[A, B]`.
2. Creator calls `PATCH /transactions/{txId}/approvers/{A.id}` with `{ listId: <other_tree_id> }`.
3. `A` is moved to the new parent. `Parent` now has `threshold=2` but only one child `[B]`.
4. The invariant `threshold ≤ approvers.length` is silently violated; `Parent` can never accumulate enough approvals.
5. Any transaction that requires approval through `Parent` is permanently unapprovable.

### Impact Explanation

A threshold node whose `threshold` exceeds its remaining child count can never satisfy its approval condition. Any transaction gated on that subtree is permanently locked in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` with no on-chain recourse. This constitutes **unrecoverable corruption of transaction state** — a category explicitly listed as high-value impact in `RESEARCHER.md`. [4](#0-3) 

### Likelihood Explanation

The precondition is being the **transaction creator** — a normal, unprivileged product role reachable by any registered user. The trigger is a single valid API call (`PATCH` with `{ listId: <id> }`) on an approver that currently has a parent. No special knowledge or timing is required. The `getCreatorsTransaction` guard confirms only creator-level access is needed. [5](#0-4) 

### Recommendation

In the re-parenting branch (`dto.listId` is a number), after detaching the approver from its old parent, apply the same logic already present in the `dto.listId === null` branch:

```typescript
// After updating approver.listId = dto.listId ...

if (approver.listId !== null) {          // approver had an old parent
  const oldParent = await transactionalEntityManager.findOne(TransactionApprover, {
    relations: ['approvers'],
    where: { id: approver.listId },      // old listId, before the update
  });
  if (oldParent) {
    const newChildCount = oldParent.approvers.length - 1;
    if (newChildCount === 0) {
      await transactionalEntityManager.softRemove(TransactionApprover, oldParent);
    } else if (newChildCount < oldParent.threshold) {
      await transactionalEntityManager.update(TransactionApprover, oldParent.id, {
        threshold: newChildCount,
      });
    }
  }
}
```

This mirrors the existing guard at lines 417–428 and ensures `threshold ≤ approvers.length` holds after every structural mutation. [2](#0-1) 

### Proof of Concept

**Setup**
```
POST /transactions/{txId}/approvers
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [{ "userId": 10 }, { "userId": 11 }]
  }]
}
```
This creates `Parent (id=1, threshold=2)` → `[Child_A (id=2, userId=10), Child_B (id=3, userId=11)]`.

**Trigger**
```
PATCH /transactions/{txId}/approvers/2
{ "listId": <any_other_valid_tree_id> }
```
Child_A (id=2) is moved to a different tree. `Parent (id=1)` now has `threshold=2` but only one child `[Child_B]`.

**Verify invariant is broken**
```sql
SELECT id, threshold FROM transaction_approver WHERE id = 1;
-- threshold = 2

SELECT COUNT(*) FROM transaction_approver WHERE "listId" = 1 AND "deletedAt" IS NULL;
-- count = 1   ← threshold (2) > children (1): invariant violated
```

**Observed outcome**: The transaction can never reach the required approval count through `Parent`, permanently blocking execution.
