Audit Report

## Title
`ApproversController.removeTransactionApprover()` Allows Cross-Transaction Approver Deletion

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns the transaction at `:transactionId`, but then deletes the approver at `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who has created at least one transaction can exploit this to silently remove approvers from any other transaction in the system.

## Finding Description

The controller handler performs two independent, uncorrelated operations:

**Step 1** — ownership check on the URL's `transactionId`: [1](#0-0) 

**Step 2** — deletion of the approver at `:id` with no cross-check against `transactionId`: [2](#0-1) 

The service-level `removeTransactionApprover` fetches the approver by `id` and calls `removeNode` immediately — there is no assertion that `approver.transactionId` (or its root node's `transactionId`) equals the `transactionId` URL parameter.

Contrast this with `updateTransactionApprover`, which correctly resolves the root node and validates it matches the URL parameter **before** proceeding: [3](#0-2) 

The `getCreatorsTransaction` guard only confirms ownership of the URL-supplied `transactionId`; it says nothing about the approver being deleted: [4](#0-3) 

**Root cause:** The authorization check and the destructive action operate on two different, unrelated entities. The check validates `transactionId`; the action targets `id`. No code ever asserts they are related.

## Impact Explanation

An attacker can silently remove required approvers from any transaction they do not own. This directly undermines the multi-signature approval workflow: a transaction requiring N approvals can have its approvers stripped, allowing it to proceed without the intended oversight or permanently corrupting another user's transaction approval state. The `removeNode` call performs a recursive soft-delete of the entire approver subtree, so a single request can wipe an entire approval tree belonging to a victim transaction.

**Impact: High** — direct integrity failure in the approval/signing workflow, which is the core security primitive of the system.

## Likelihood Explanation

Any authenticated, verified user who has created at least one transaction can exploit this. Approver IDs are sequential integers (`id` column in `transaction_approver`) and are trivially enumerable. No privileged access, leaked credentials, or special conditions are required beyond a valid JWT and having created one transaction.

**Likelihood: Medium** — requires an authenticated user and knowledge of a target approver ID, both trivially achievable.

## Recommendation

In `removeTransactionApprover` (service), after fetching the approver, resolve its root node via `getRootNodeFromNode` and assert that `rootNode.transactionId === transactionId` before calling `removeNode` — exactly mirroring the pattern already used in `updateTransactionApprover`:

```typescript
// approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // *** ADD THIS BLOCK ***
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
  // *** END ADDED BLOCK ***

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

Pass `transactionId` from the controller call site: [5](#0-4) 

## Proof of Concept

1. Alice authenticates and creates **Transaction A** (she is its creator).
2. Bob authenticates and creates **Transaction B**, adding an approver with `id = 99` belonging to Transaction B.
3. Alice sends:
   ```
   DELETE /transactions/{transactionA_id}/approvers/99
   Authorization: Bearer <alice_jwt>
   ```
4. `getCreatorsTransaction(transactionA_id, alice)` — **passes**, Alice owns Transaction A.
5. `removeTransactionApprover(99)` — **executes**, approver 99 (belonging to Transaction B) is soft-deleted via `removeNode` with no further check.
6. Transaction B's approval workflow is now corrupted; Bob's required approver is gone.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
