### Title
`archiveTransaction` Bypasses Terminal-State Guard via `isManual` Flag, Allowing Corruption of Executed/Failed/Canceled Transaction Records

---

### Summary

`archiveTransaction` in `TransactionsService` contains a flawed guard condition that uses a logical AND between the status check and `!transaction.isManual`. When `isManual` is `true`, the status check is entirely skipped, allowing the transaction creator to overwrite any terminal state — including `EXECUTED`, `FAILED`, `CANCELED`, and `EXPIRED` — with `ARCHIVED`. This is a direct state-transition analog to the GenesisGroup `purchase`-after-`launch` vulnerability: a mutating action that should be blocked after a terminal state is reached is instead permitted due to a missing guard.

---

### Finding Description

**Root cause — flawed boolean guard:** [1](#0-0) 

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← second operand of AND
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** conditions are true:
- status is not in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- **AND** `isManual` is `false`

When `isManual = true`, the second operand is `false`, so the entire expression is `false` and the throw is never reached — regardless of the current status. The function then unconditionally overwrites the status: [2](#0-1) 

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

**Terminal states that can be overwritten:** [3](#0-2) 

`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, and `REJECTED` are all terminal. None of them are guarded when `isManual = true`.

**Entry path:**

The only prerequisite is that the caller is the transaction creator: [4](#0-3) 

Any organization member who created a manual transaction satisfies this check. No admin or privileged role is required.

**Contrast with correct guards elsewhere:**

`cancelTransaction` uses a conditional SQL `WHERE status IN (...)` clause to prevent races and terminal-state overwrites: [5](#0-4) 

`uploadSignatureMaps` explicitly rejects non-active statuses: [6](#0-5) 

`archiveTransaction` is the only mutation path that omits this protection for manual transactions.

---

### Impact Explanation

A transaction creator with a `isManual = true` transaction that has already reached `EXECUTED`, `FAILED`, `CANCELED`, or `EXPIRED` can call `PATCH /transactions/:id/archive` and overwrite the terminal status to `ARCHIVED`.

Concrete consequences:
1. **Audit-trail corruption**: An `EXECUTED` transaction — one that was submitted to and confirmed by the Hedera network — is relabeled `ARCHIVED`. Observers and auditors see `ARCHIVED` instead of `EXECUTED`, misrepresenting the true outcome.
2. **Failure concealment**: A `FAILED` transaction can be archived, hiding the failure from observers who were added to the transaction.
3. **Irreversibility**: There is no "un-archive" path in the codebase. Once overwritten, the original terminal status is unrecoverable from the application layer.

The `executedAt` timestamp column remains set, creating an inconsistent record (a transaction with `executedAt` populated but `status = ARCHIVED`). [7](#0-6) 

---

### Likelihood Explanation

- **Attacker profile**: Any authenticated organization user who created a manual transaction. No admin key, no leaked credential, no privileged role required.
- **Trigger**: A single `PATCH /transactions/:id/archive` API call after the transaction has been executed or failed.
- **Precondition**: `isManual = true` on the transaction. Manual transactions are a standard product feature explicitly supported in the UI and API. [8](#0-7) 

The `executeTransaction` endpoint confirms that `isManual` transactions are a first-class, reachable flow for normal users.

---

### Recommendation

Separate the `isManual` check from the status guard. Terminal states must always be immutable regardless of the `isManual` flag:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  // Terminal states are never archivable
  const terminalStatuses = [
    TransactionStatus.EXECUTED,
    TransactionStatus.FAILED,
    TransactionStatus.CANCELED,
    TransactionStatus.EXPIRED,
    TransactionStatus.REJECTED,
    TransactionStatus.ARCHIVED,
  ];
  if (terminalStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  // For non-manual transactions, also require an active signing status
  if (
    !transaction.isManual &&
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION]
      .includes(transaction.status)
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
```

---

### Proof of Concept

1. Authenticated user Alice creates a transaction with `isManual: true` via `POST /transactions`.
2. The transaction proceeds through signing and is executed — status becomes `EXECUTED`, `executedAt` is set.
3. Alice calls `PATCH /transactions/:id/archive`.
4. `getTransactionForCreator` passes (Alice is the creator).
5. Guard evaluates: `status = EXECUTED` → not in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]` → first operand is `true`. `isManual = true` → `!isManual = false` → second operand is `false`. `true && false = false` → **no throw**.
6. `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes.
7. The database now records `status = ARCHIVED`, `executedAt = <timestamp>` — a contradictory and unrecoverable state.
8. Observers and auditors querying transaction history see `ARCHIVED` instead of `EXECUTED`. [9](#0-8)
