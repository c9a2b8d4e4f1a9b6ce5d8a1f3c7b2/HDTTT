Audit Report

## Title
Cross-Transaction Approver Deletion: Creator of Any Transaction Can Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller owns `:transactionId`, but then deletes the approver record identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can exploit this to delete approvers from transactions they do not own.

## Finding Description

**Root cause — missing scope check in the delete path:**

The controller's `removeTransactionApprover` handler performs two sequential calls:

```ts
await this.approversService.getCreatorsTransaction(transactionId, user); // line 108
await this.approversService.removeTransactionApprover(id);               // line 109
``` [1](#0-0) 

`getCreatorsTransaction` only validates that the authenticated user is the creator of the transaction identified by the URL parameter `transactionId`. It says nothing about the approver identified by `id`. [2](#0-1) 

`removeTransactionApprover` then fetches the approver by `id` and soft-deletes it with no check that `approver.transactionId` (or its root transaction) matches the `transactionId` from the URL: [3](#0-2) 

**Contrast with the update path:** `updateTransactionApprover` correctly performs this scope check — it fetches the root node and explicitly throws `UnauthorizedException` if `rootNode.transactionId !== transactionId`. The delete path has no equivalent guard. [4](#0-3) 

**Sequential integer primary key:** `TransactionApprover` uses `@PrimaryGeneratedColumn()`, making approver IDs sequential integers that are trivially enumerable. [5](#0-4) 

**Secondary inconsistency:** For nested approvers (`listId != null`), `approver.transactionId` is `null`. The `emitTransactionStatusUpdate` call at line 541 therefore fires with `entityId: null`, silently failing to notify the correct transaction's stakeholders. [6](#0-5) 

## Impact Explanation

An attacker can delete any approver from any transaction in the system regardless of ownership:

- **Approval threshold bypass**: Removing enough approvers from a threshold tree can cause the remaining count to fall below the threshold, causing the chain service to treat the transaction as fully approved and advance it to `WAITING_FOR_EXECUTION` without the required signatures.
- **Denial of approval**: Removing all approvers from a transaction permanently blocks it from reaching `WAITING_FOR_EXECUTION` status.
- **Integrity violation**: The multi-signature orchestration model — the core security guarantee of the system — is broken for any transaction whose approvers can be silently deleted by an unrelated party.

## Likelihood Explanation

- **Attacker precondition**: Must be a registered, verified user who has created at least one transaction. This is the normal user role — no elevated privilege required.
- **Knowledge required**: Approver IDs are sequential integers. An attacker can enumerate them by creating their own transactions and observing the IDs returned in API responses, then targeting nearby IDs belonging to other transactions.
- The attack is a single authenticated HTTP DELETE request with no rate-limiting or anomaly detection visible on this endpoint.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), resolve the root transaction of the approver being deleted and assert it matches the `transactionId` URL parameter — mirroring the check already present in `updateTransactionApprover`:

```ts
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root node and verify it belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

This also fixes the secondary inconsistency: using the validated `transactionId` parameter instead of `approver.transactionId` ensures the status update event is always emitted with the correct transaction ID, even for nested approvers where `approver.transactionId` is `null`.

## Proof of Concept

1. Alice registers and creates transaction `T_A` (ID = 1). She is its creator.
2. Bob creates transaction `T_B` (ID = 2) and adds an approver (approver ID = 5).
3. Alice observes that her own approvers have IDs 3 and 4 (returned in API responses), so she infers that nearby IDs may belong to other transactions.
4. Alice sends:
   ```
   DELETE /transactions/1/approvers/5
   Authorization: Bearer <alice_token>
   ```
5. `getCreatorsTransaction(1, alice)` passes — Alice is the creator of transaction 1.
6. `removeTransactionApprover(5)` executes — approver 5 (belonging to Bob's transaction 2) is soft-deleted with no ownership check.
7. Alice has successfully removed an approver from Bob's transaction, potentially bypassing its multi-signature approval requirement. [1](#0-0) [3](#0-2)
