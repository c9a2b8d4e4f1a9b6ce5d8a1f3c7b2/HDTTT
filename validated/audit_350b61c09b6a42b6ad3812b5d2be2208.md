Audit Report

## Title
Unbounded Database Fetch in `getTransactionsToSign` Causes Server-Side Resource Exhaustion

## Summary
`TransactionsService.getTransactionsToSign()` fetches every non-terminal transaction from the database with no row limit, then issues one sequential async `userKeysToSign()` database call per row before applying pagination in memory. As the transaction table grows, a single authenticated `GET /transactions/sign` request will exhaust server memory, hold open database connections, and degrade or crash the API for all users.

## Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` performs an unbounded `repo.find()` with no `take` constraint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then iterates the entire result set, issuing one sequential async database round-trip per transaction via `userKeysToSign()`:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

Pagination is applied only after the full in-memory result is built:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

Every other paginated endpoint in the service correctly passes `skip: offset, take: limit` to the ORM query. For example, `getHistoryTransactions()` and `getTransactionsToApprove()` both use `skip`/`take` at the database level: [4](#0-3) [5](#0-4) 

The HTTP controller exposes this as `GET /transactions/sign`, guarded only by JWT authentication — no admin role required. The `/* NO LONGER USED BY FRONT-END */` comment confirms the endpoint is still live and reachable: [6](#0-5) 

The `PaginationParams` decorator enforces `size <= 100` at the HTTP layer, but this limit is never forwarded to the database query, so it provides no protection against the unbounded fetch.

## Impact Explanation

With a large transaction table (thousands of non-terminal rows, realistic in an active organization deployment):

- The Node.js process allocates memory proportional to the full result set plus all loaded `groupItem` relations.
- Each iteration of the loop holds a database connection open for the duration of the sequential `userKeysToSign()` calls, which themselves issue additional DB queries.
- The request will time out at the HTTP layer, but the in-flight database work continues consuming resources.
- Repeated calls — even from a single user — can exhaust the database connection pool and heap memory, causing the API service to become unresponsive for all users.

**Impact category:** Service unavailability / severe degradation under realistic load.

## Likelihood Explanation

- **Precondition:** Any registered, verified user with at least one uploaded key. No privileged role is required.
- **Trigger:** A single `GET /transactions/sign?page=1&size=1` request. The `size` parameter is irrelevant because the DB query is unbounded regardless of what is passed.
- **Realism:** The endpoint remains active (confirmed by the "NO LONGER USED BY FRONT-END" comment — it is not removed, only abandoned by the UI). Any authenticated API client or integrator can reach it. As the platform accumulates non-terminal transactions over time, the impact worsens automatically without any additional attacker action.

## Recommendation

Apply database-level pagination inside `getTransactionsToSign()` analogous to the pattern already used in `getHistoryTransactions()` and `getTransactionsToApprove()`. The `skip`/`take` constraints must be passed to the ORM query, not applied after the fact in memory. Because `userKeysToSign()` filtering must happen post-query, a cursor-based or two-pass approach (fetch a page, filter, fetch more if needed) should be used, or the signing-key check should be pushed into the database query itself. Additionally, consider removing or formally deprecating the `GET /transactions/sign` endpoint if it is no longer used by the front-end.

## Proof of Concept

1. Register and verify a user account; upload at least one public key.
2. Ensure the database contains a large number of non-terminal transactions (e.g., thousands of `WAITING_FOR_SIGNATURES` rows — achievable by any user with create-transaction access over time, or by an attacker with a script).
3. Send a single authenticated request:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <valid_jwt>
   ```
4. Observe that the server fetches all non-terminal transactions from the database (no `LIMIT` clause), then issues one DB round-trip per row sequentially via `userKeysToSign()`, before returning only 1 item.
5. Monitor server memory and database connection pool — both grow proportionally to the total number of non-terminal transactions, not to the requested page size of 1.
6. Repeated requests (or concurrent requests from a single account) will exhaust the connection pool and heap, causing HTTP 503/timeout responses for all users.
