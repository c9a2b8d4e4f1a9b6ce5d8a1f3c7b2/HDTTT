### Title
Rate-Limit Bypass via Spoofable `x-forwarded-for` Header in `IpThrottlerGuard`

### Summary
`IpThrottlerGuard` derives the client identity for rate-limiting from the attacker-controlled `x-forwarded-for` request header, giving it unconditional priority over the actual socket IP (`req.ip`). This is the direct analog of the `tx.origin` pattern: a security decision is anchored to an untrusted, intermediary-supplied value rather than the verified connection identity. Any unauthenticated attacker can rotate through arbitrary fake IPs to bypass the global rate limiter and mount unlimited brute-force or enumeration attacks against every protected endpoint.

### Finding Description

`IpThrottlerGuard.getTracker()` resolves the tracking key as:

```ts
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [1](#0-0) 

`x-forwarded-for` is a plain HTTP header. Any HTTP client can set it to an arbitrary value. Because it is evaluated **before** `req.ip`, the real socket address is never consulted when the header is present. The guard then uses this attacker-supplied string as the Redis key for the global throttler: [2](#0-1) 

The guard is registered as the global `APP_GUARD` equivalent for the API service, so every endpoint — including unauthenticated ones — is subject to it. Sensitive endpoints that rely on this rate limiter include:

- `POST /auth/login` (password brute-force) — guarded by `LocalAuthGuard` + `EmailThrottlerGuard` + global IP throttler
- `POST /auth/reset-password` (OTP enumeration) — guarded by `EmailThrottlerGuard` + global IP throttler [3](#0-2) [4](#0-3) 

The `EmailThrottlerGuard` tracks by email address, so it provides a separate layer; however, the global IP throttler is the only control that limits requests from unknown/unauthenticated callers who rotate email addresses. Bypassing it removes the last line of defence against volumetric attacks.

### Impact Explanation

An attacker who rotates the `x-forwarded-for` value on each request (e.g., `X-Forwarded-For: 1.2.3.4`, then `1.2.3.5`, …) is never counted against any single bucket in Redis. The global per-minute and per-second limits configured in `GLOBAL_MINUTE_LIMIT` / `GLOBAL_SECOND_LIMIT` become entirely ineffective. Concrete consequences:

- **Credential brute-force**: unlimited password guesses against any known email address at `POST /auth/login`.
- **OTP enumeration**: unlimited TOTP guesses at `POST /auth/verify-reset`, bypassing the time-step window protection.
- **Resource exhaustion**: sustained high-rate requests to any endpoint without throttling.

### Likelihood Explanation

- Requires zero privileges — any HTTP client (curl, Python requests) can set arbitrary headers.
- The attack is trivially scriptable: increment a counter in the `X-Forwarded-For` header per request.
- No proxy or load-balancer stripping of the header is enforced in the codebase; the NestJS application does not call `app.set('trust proxy', ...)` with a validated proxy list, so Express does not sanitize the header before it reaches the guard.

### Recommendation

Do not trust `x-forwarded-for` unless the application is deployed behind a known, trusted reverse proxy that strips and re-writes the header. The fix is to remove the header lookup and rely solely on `req.ip`, or — if a proxy is in use — configure Express's `trust proxy` setting to a specific CIDR/count so that Express itself validates and resolves the real client IP before the guard runs:

```ts
// Option A: trust only req.ip (no proxy)
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

```ts
// Option B: configure Express trust proxy in main.ts
app.set('trust proxy', 1); // trust only the first hop (your known proxy)
// Then req.ip is already resolved correctly; remove the x-forwarded-for lookup in the guard.
``` [5](#0-4) 

### Proof of Concept

```bash
# Bypass rate limit by rotating X-Forwarded-For on every request
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<server>/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$i" \
    -H "x-frontend-version: 99.0.0" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
```

Each request is tracked under a distinct fake IP (`10.0.0.1`, `10.0.0.2`, …), so no bucket ever reaches the configured limit. The attacker can enumerate passwords or OTP codes at full network speed with no throttling applied.
