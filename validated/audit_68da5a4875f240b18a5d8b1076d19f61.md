## Audit Report

## Title
Cross-Transaction Approver Deletion: Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` verifies the caller is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who is the creator of *any* transaction can delete approvers from *any other* transaction.

## Finding Description

**Root Cause:**

In `removeTransactionApprover` in the controller, two independent operations are performed:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
```

`getCreatorsTransaction` only confirms the authenticated user is the creator of the URL-supplied `transactionId`: [1](#0-0) 

`removeTransactionApprover` then fetches and removes the approver by `id` alone, with no verification that it belongs to `transactionId`: [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly validates the cross-reference before proceeding: [3](#0-2) 

The `PATCH` path has this guard; the `DELETE` path does not. [4](#0-3) 

## Impact Explanation

- **Approval threshold bypass**: An attacker can delete required approvers from any transaction, reducing the effective approval threshold below what the transaction creator intended.
- **Disruption of multi-signature workflows**: Legitimate approvers are silently removed from transactions they are supposed to govern, breaking the organization's trust model.
- **Cross-tenant integrity failure**: Any user who has ever created a transaction gains the ability to tamper with approval structures of all other transactions in the system.

## Likelihood Explanation

- **Attacker precondition**: Only requires a valid authenticated account and having created at least one transaction — both are normal, unprivileged actions available to any organization member.
- **Approver IDs are sequential integers** (standard auto-increment primary keys), making enumeration trivial.
- No rate limiting or anomaly detection is needed to exploit this; a single crafted HTTP request is sufficient.

## Recommendation

In `removeTransactionApprover` in `approvers.service.ts`, add a cross-reference check analogous to the one already present in `updateTransactionApprover`. After fetching the approver by `id`, traverse to the root node and verify `rootNode.transactionId === transactionId` before proceeding with deletion. If they do not match, throw an `UnauthorizedException`. This mirrors the existing guard at lines 386–391 of `approvers.service.ts`. [5](#0-4) 

## Proof of Concept

1. Attacker authenticates and creates any transaction (e.g., `transactionId = 1`). This satisfies `getCreatorsTransaction`.
2. Attacker enumerates or guesses the `id` of an approver belonging to a victim's transaction (e.g., `approver.id = 99`, belonging to `transactionId = 42`).
3. Attacker sends: `DELETE /transactions/1/approvers/99`
4. `getCreatorsTransaction(1, attacker)` passes — attacker is creator of transaction 1.
5. `removeTransactionApprover(99)` executes — approver 99 (belonging to transaction 42) is removed with no ownership check. [6](#0-5)
