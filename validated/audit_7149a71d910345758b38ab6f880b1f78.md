Audit Report

## Title
Missing Access Control on Transaction Comments Endpoints Allows Any Authenticated User to Post and Read Comments on Any Transaction

## Summary
`CommentsController` applies only authentication-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) at the class level. Neither the controller nor the underlying `CommentsService` performs any per-transaction ownership or membership check. Any authenticated organization user can `POST` a comment to, or `GET` all comments from, any transaction they have no legitimate relationship to.

## Finding Description

**Root cause — `CommentsController`:**

The controller is decorated with only authentication guards and no authorization guard: [1](#0-0) 

The developer explicitly acknowledged the missing check with a TODO comment directly above `createComment`: [2](#0-1) 

**`getComments` and `getCommentById` — no user context at all:**

Neither endpoint receives a `user` parameter or performs any access check: [3](#0-2) 

**`CommentsService` — no authorization logic:**

`createComment` blindly associates the comment with the supplied `transactionId` without verifying the user's relationship to that transaction: [4](#0-3) 

`getTransactionComments` and `getTransactionCommentById` perform a plain database lookup with no caller identity check: [5](#0-4) 

**Contrast with other sub-resource services that do enforce access:**

`ObserversService.getTransactionObserversByTransactionId` explicitly throws `UnauthorizedException` if the requesting user is not the creator, a signer, an observer, or an approver of the transaction: [6](#0-5) 

`ApproversService.getVerifiedApproversByTransactionId` applies the identical pattern: [7](#0-6) 

The comments endpoints have no equivalent check.

## Impact Explanation

1. **Unauthorized state mutation**: Any authenticated user can `POST /transactions/:transactionId/comments` with an arbitrary `transactionId` and inject comments into transaction workflows they have no role in — creator, signer, observer, or approver.
2. **Information disclosure**: Transaction comments may contain sensitive organizational decisions, key coordination details, or approval rationale. Any authenticated user can enumerate all comments on any transaction by iterating `transactionId` values via `GET /transactions/:transactionId/comments`, or enumerate individual comments by iterating `id` values via `GET /transactions/:transactionId/comments/:id`.

## Likelihood Explanation

The only precondition is a valid, verified organization account — no admin privileges, no leaked secrets, no special role. The exploit is a direct HTTP `POST` or `GET` to a documented REST endpoint. The developer TODO comment at line 22 of the controller confirms the gap is known and unmitigated. Any malicious insider or compromised low-privilege account can exploit this immediately with no additional tooling.

## Recommendation

Implement a per-transaction membership check in `CommentsService` mirroring the pattern used in `ObserversService.getTransactionObserversByTransactionId` and `ApproversService.getVerifiedApproversByTransactionId`:

- Before `createComment`, `getTransactionComments`, and `getTransactionCommentById`, load the target `Transaction` with its `creatorKey`, `observers`, `signers`, and `approvers` relations.
- Verify the requesting user is the creator, a signer, an observer, or an approver. Throw `UnauthorizedException` otherwise.
- Pass the `user` object into `getComments` and `getCommentById` (currently absent) so the check can be performed.
- Remove the TODO comment once the guard is in place.

## Proof of Concept

```
# Step 1: Authenticate as any verified organization user (User A)
POST /auth/login
→ returns JWT token_A

# Step 2: Obtain a transactionId belonging to a transaction User A has no role in
# (e.g., by brute-forcing integers or observing network traffic)
transactionId = 42

# Step 3: Read all comments on that transaction
GET /transactions/42/comments
Authorization: Bearer token_A
→ 200 OK — returns all comments with no authorization error

# Step 4: Inject a comment into that transaction
POST /transactions/42/comments
Authorization: Bearer token_A
Content-Type: application/json
{"message": "Injected comment"}
→ 201 Created — comment is persisted with no authorization error
```
