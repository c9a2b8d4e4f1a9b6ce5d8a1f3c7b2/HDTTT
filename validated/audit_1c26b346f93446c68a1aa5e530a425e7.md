Audit Report

## Title
`getTransactionsToSign` Fetches All Non-Terminal Transactions Without Database-Level Pagination, Enabling Unbounded Resource Exhaustion

## Summary
`getTransactionsToSign` in `transactions.service.ts` accepts `{ page, limit, size, offset }` pagination parameters but never passes `take` or `skip` to the database query. The method loads every non-terminal transaction from PostgreSQL into memory, then calls the expensive `userKeysToSign` operation (which involves transaction deserialization, mirror-node key resolution, and additional DB queries) on each one, before finally slicing the result in memory. The endpoint `GET /transactions/sign` is live and reachable by any authenticated user.

## Finding Description

In `getTransactionsToSign`, the `repo.find()` call at line 295 omits both `take` and `skip`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no take, no skip
});
``` [1](#0-0) 

After loading the full unbounded result set, the code iterates over every transaction and calls `userKeysToSign` per entry:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [2](#0-1) 

Pagination is only applied in memory after all work is done:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

By contrast, the sibling methods `getTransactions` and `getHistoryTransactions` correctly pass `skip: offset, take: limit` to the database: [4](#0-3) [5](#0-4) 

And `getTransactionsToApprove` does the same: [6](#0-5) 

The endpoint is registered and protected only by standard session guards — no admin or elevated role is required: [7](#0-6) 

Each call to `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs:
1. `SDKTransaction.fromBytes()` — full transaction deserialization
2. `transactionSignatureService.computeSignatureKey()` — potentially involves mirror-node API calls
3. `entityManager.find(UserKey, ...)` — an additional database query per transaction [8](#0-7) 

## Impact Explanation
Any authenticated user can send `GET /transactions/sign?page=1&size=1`. Each request causes the server to load **all** active (non-terminal) transactions from PostgreSQL into memory, then execute the full `userKeysToSign` pipeline — including transaction deserialization, mirror-node key resolution, and a DB query — for every single transaction, regardless of the requested page size. As the transaction table grows, each request consumes proportionally more memory, CPU, and outbound mirror-node connections. Under sustained load or with a large transaction table, this can exhaust server resources and degrade or halt the API service.

## Likelihood Explanation
The endpoint requires only a valid authenticated session — no admin role, no special privilege. The controller comment explicitly notes it is `/* NO LONGER USED BY FRONT-END */`, meaning it is an unguarded, forgotten attack surface that has not been removed. The vulnerability is triggered by a single normal-looking API request, not a contrived exploit. Any registered user who discovers the endpoint (e.g., via the Swagger docs that are still served) can trigger it. [9](#0-8) 

## Recommendation
Apply one or both of the following fixes:

1. **Remove the endpoint entirely** — since the front-end no longer uses it, the simplest fix is to delete the `@Get('/sign')` route and the `getTransactionsToSign` method.

2. **Apply database-level pagination** — if the endpoint must remain, add `take: limit` and `skip: offset` to the `repo.find()` call, consistent with all sibling methods. Note that because the filtering step (`userKeysToSign`) happens after the DB fetch, in-memory pagination over a pre-filtered set is architecturally incorrect; the correct fix requires pushing the user-key filter into the query or accepting that `totalItems` will be approximate.

## Proof of Concept
```

GET /transactions/sign?page=1&size=1
Authorization: Bearer <valid_jwt>

# Server loads ALL non-terminal transactions, calls userKeysToSign on each,
# then returns exactly 1 item. Repeat to sustain load.
```

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];

  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
