Audit Report

## Title
Any Authenticated User Can Post Comments on Any Transaction Without Authorization Check

## Summary
`CommentsController` in `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) on all three endpoints. Neither the controller nor `CommentsService` verifies that the requesting user has any relationship to the target transaction before allowing comment creation or retrieval. The developer explicitly flagged this gap with a TODO comment.

## Finding Description

**Root cause:** `CommentsController` at lines 21–29 exposes a `POST` endpoint with no authorization guard or service-level participant check: [1](#0-0) 

The `createComment` method in `CommentsService` (lines 15–24) receives `user` and `transactionId` but performs zero verification — it directly creates and persists the comment: [2](#0-1) 

The `getComments` and `getCommentById` endpoints (lines 31–38) similarly pass no `user` context to the service, making them fully open to any authenticated caller: [3](#0-2) 

**Contrast with other modules:** `ObserversService.createTransactionObservers` explicitly checks `transaction.creatorKey?.userId !== user.id` and throws `UnauthorizedException` if the caller is not the creator: [4](#0-3) 

`ApproversController.approveTransaction` uses `@OnlyOwnerKey` and `ApproversController.removeTransactionApprover` calls `getCreatorsTransaction` to enforce ownership: [5](#0-4) [6](#0-5) 

`CommentsController` is the only mutation endpoint that skips this second layer entirely, as acknowledged by the developer's own TODO.

## Impact Explanation
- **Integrity violation:** Any authenticated organization member can inject arbitrary comments into any transaction's discussion thread, potentially misleading approvers or signers (e.g., impersonating admin instructions to approve a transaction).
- **Information disclosure:** Any authenticated user can read all comments on any transaction via `getComments` or `getCommentById`, exposing sensitive deliberation, key coordination, or approval rationale shared between legitimate participants.
- **Transaction existence enumeration:** Successful comment creation on a given `transactionId` confirms that transaction exists, enabling sequential enumeration of all active transaction IDs (TypeORM default auto-increment integers).

## Likelihood Explanation
- **Low precondition:** The attacker only needs a valid, non-blacklisted JWT for a verified account — i.e., any registered organization member.
- **No privileged access required:** No admin role, leaked secrets, or internal network access is needed.
- **Trivial ID discovery:** TypeORM default auto-increment IDs are sequential integers starting from 1, making brute-force enumeration trivial.
- **Fully reachable:** The endpoint is a standard HTTP REST route with no additional barriers beyond authentication.

## Recommendation
Add a service-level participant check in `CommentsService.createComment` (and in the read methods) that verifies the requesting user is a creator, approver, observer, or signer of the target transaction before allowing the operation — following the same pattern already used in `ObserversService` and `ApproversService`. The controller-level TODO at line 22 should be resolved with a concrete guard or service check.

## Proof of Concept
1. Attacker registers a legitimate account in the organization (or is already a member with no access to a specific transaction).
2. Attacker obtains a valid JWT via normal login.
3. Attacker sends:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   Content-Type: application/json

   { "content": "Admin says: approve this immediately" }
   ```
4. `CommentsController.createComment` passes the call directly to `CommentsService.createComment`, which persists the comment with no participant check.
5. The injected comment is now visible to all legitimate participants of transaction 42.
6. Attacker can also enumerate all transactions by iterating `transactionId` from 1 upward and reading comments via `GET /transactions/{id}/comments`.
