# Audit Report

## Title
Transaction Creator Can Add Themselves as an Approver, Bypassing the Multi-Party Approval Workflow

## Summary
`createTransactionApprovers` and `updateTransactionApprover` in `approvers.service.ts` enforce that only the transaction creator may manage approvers, but impose no check preventing the creator from designating their own `userId` as an approver. The creator can then immediately self-approve the transaction via `approveTransaction`, collapsing the multi-party approval mechanism to a single-party action.

## Finding Description

**Root cause — missing self-exclusion check in `createTransactionApprovers`:**

`createTransactionApprovers` first verifies the caller is the transaction creator: [1](#0-0) 

It then validates each approver DTO. The only user-existence check is: [2](#0-1) 

There is **no check** that `dtoApprover.userId !== user.id`. The creator can freely pass their own `userId` in the `approversArray` payload.

The same gap exists in `updateTransactionApprover`: after verifying the caller is the creator, it allows changing an existing approver's `userId` to any value, again without excluding the creator's own id: [3](#0-2) 

In `approveTransaction`, the user is accepted as a valid approver purely by matching `userId`: [4](#0-3) 

The controller exposes both endpoints to any verified JWT holder with no additional guard: [9](#0-8) [10](#0-9) 

## Impact Explanation
The approver system is the protocol's multi-party check-and-balance layer: independent parties must approve a transaction before it can be executed. A creator who self-approves collapses this to a single-party action, defeating the governance model entirely. Any transaction — including high-value transfers or account updates — can be unilaterally approved by its creator without any independent oversight.

## Likelihood Explanation
The attacker is a normal authenticated user (no admin privileges, no leaked secrets required). The exploit requires only two standard API calls after transaction creation. The path is deterministic and requires no race condition or timing dependency. Any organization member who can create transactions can exploit this immediately.

## Recommendation
In `createTransactionApprovers`, add a self-exclusion check before inserting each approver:

```typescript
if (dtoApprover.userId === user.id)
  throw new Error('Transaction creator cannot be added as an approver');
```

Apply the same guard in `updateTransactionApprover` when `dto.userId` is being set:

```typescript
if (dto.userId === user.id)
  throw new Error('Transaction creator cannot be set as an approver');
```

Optionally, add a redundant check in `approveTransaction` to reject approval if the approver's `userId` matches the transaction creator's `userId`, providing defense-in-depth.

## Proof of Concept

```
# Step 1: Creator creates a transaction
POST /transactions
Authorization: Bearer <creator_jwt>
→ { "id": 42, "status": "WAITING_FOR_SIGNATURES", ... }

# Step 2: Creator adds themselves as the sole approver
POST /transactions/42/approvers
Authorization: Bearer <creator_jwt>
Body: { "approversArray": [{ "userId": <creator_user_id> }] }
→ 201 Created — creator is now the sole approver

# Step 3: Creator self-approves with their own key signature
POST /transactions/42/approvers/approve
Authorization: Bearer <creator_jwt>
Body: { "userKeyId": <creator_key_id>, "signature": "<valid_sig>", "approved": true }
→ approveTransaction finds creator in userApprovers, verifies signature, marks approved

Result: Transaction approved by its creator alone, with zero independent oversight.
```
