Audit Report

## Title
IDOR in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId` but never validates that approver `:id` belongs to that transaction. Any authenticated user who created at least one transaction can delete approvers from any other transaction in the system.

## Finding Description
In the controller, the delete handler performs an ownership check on `transactionId`, then passes the approver `id` directly to the service with no cross-validation: [1](#0-0) 

`removeTransactionApprover` in the service fetches the approver purely by its own `id` and deletes it without any `transactionId` constraint: [2](#0-1) 

`getTransactionApproverById` queries with `where: { id }` only — no `transactionId` filter: [3](#0-2) 

The correct pattern exists in `updateTransactionApprover`, which explicitly validates the approver's root transaction matches the route parameter before proceeding: [4](#0-3) 

This guard is entirely absent from the delete path.

## Impact Explanation
An attacker who is the creator of any transaction (even a trivial one they created themselves) can delete approvers belonging to high-value transactions owned by other users. This removes required signers from multi-signature transactions, preventing them from ever reaching the approval threshold, and can silently stall or permanently block critical Hedera transactions awaiting multi-party approval.

## Likelihood Explanation
The attack requires only a valid JWT token and knowledge of an approver ID from another transaction. Approver IDs are sequential integers (`id: number`), making enumeration trivial. Any registered, verified user in an organization can exploit this immediately with no special privilege.

## Recommendation
In `removeTransactionApprover` (or in the controller before calling it), add the same cross-validation that `updateTransactionApprover` performs: resolve the root node of the target approver via `getRootNodeFromNode`, then assert `rootNode.transactionId === transactionId`. If they differ, throw `UnauthorizedException`. This mirrors the existing guard at lines 386–391 of `approvers.service.ts`. [5](#0-4) 

## Proof of Concept
1. Attacker (User A) creates Transaction T1 — they are now its creator.
2. Victim (User B) creates Transaction T2 with approver records (e.g., approver IDs 50, 51).
3. Attacker enumerates approver IDs (sequential integers) and identifies IDs 50 and 51 belong to T2.
4. Attacker sends:
   ```
   DELETE /transactions/{T1_id}/approvers/50
   Authorization: Bearer <attacker_jwt>
   ```
5. `getCreatorsTransaction(T1_id, userA)` passes — attacker owns T1.
6. `removeTransactionApprover(50)` executes — approver 50 (belonging to T2) is soft-deleted with no cross-check.
7. T2's multi-signature approval workflow is dismantled without User B's knowledge.
