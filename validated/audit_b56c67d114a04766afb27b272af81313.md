Audit Report

## Title
Old Parent Threshold Not Adjusted When Approver Is Re-Parented, Violating `threshold ≤ approvers.length` Invariant

## Summary
In `updateTransactionApprover`, the branch that detaches a child to root (`dto.listId === null`) correctly adjusts the old parent's threshold when its child count drops. The branch that re-parents a child to a new parent (`dto.listId = <number>`) fetches and validates only the *new* parent, leaving the *old* parent with a `threshold` that now exceeds its remaining child count. This permanently breaks the approval condition for any transaction whose approver tree contains that node.

## Finding Description
The invariant `threshold ≤ approvers.length` must hold for every threshold-type `TransactionApprover` node, because the approval check requires exactly `threshold` children to approve.

**Path 1 — `dto.listId === null`** correctly enforces the invariant: [1](#0-0) 

The old parent is fetched with its `approvers` relation, and if `newParentApproversLength < parent.threshold`, the threshold is reduced to match.

**Path 2 — `dto.listId = <number>`** does not enforce the invariant: [2](#0-1) 

Only the *new* parent is fetched and validated. The old parent (identified by `approver.listId` before the update) is never loaded, and its `threshold` is never adjusted. After the `update` at line 458, the old parent has lost one child but retains its original `threshold`, violating the invariant.

The threshold-update path does guard against setting `threshold > approvers.length` directly: [3](#0-2) 

But this guard is irrelevant here — the invariant is broken as a side-effect of the re-parenting operation, not through a direct threshold update.

## Impact Explanation
After the re-parenting call, the old parent node has `threshold = N` but only `N-1` children. The approval condition for that subtree can never be satisfied regardless of how many users approve. Any transaction whose approver tree contains this node is permanently blocked from reaching the approved state, causing a denial-of-service on transaction execution for the affected transaction and all approvers who have already signed it.

## Likelihood Explanation
The attacker only needs to be the **creator of a transaction** — a normal authenticated user role with no elevated privileges. The exploit requires two sequential API calls:

1. Create a threshold approver tree where a parent has `threshold = 2` and exactly 2 children (child A and child B).
2. Call `PATCH /transactions/:transactionId/approvers/:childA` with `{ "listId": <other_parent_id> }` to move child A to a different parent.

After step 2, the old parent has `threshold = 2` and only 1 child (child B). The approval condition is permanently unsatisfiable for that subtree.

## Recommendation
In the `dto.listId = <number>` branch, before performing the update, fetch the old parent (using `approver.listId`) with its `approvers` relation — mirroring exactly what the `dto.listId === null` branch already does — and apply the same threshold-adjustment logic:

```typescript
// Before the update at line 458, add:
if (approver.listId !== null) {
  const oldParent = await transactionalEntityManager.findOne(TransactionApprover, {
    relations: ['approvers'],
    where: { id: approver.listId },
  });
  if (oldParent) {
    const newOldParentLength = oldParent.approvers.length - 1;
    if (newOldParentLength === 0) {
      await transactionalEntityManager.softRemove(TransactionApprover, oldParent);
    } else if (newOldParentLength < oldParent.threshold) {
      await transactionalEntityManager.update(TransactionApprover, oldParent.id, {
        threshold: newOldParentLength,
      });
    }
  }
}
```

This mirrors the existing logic at lines 417–428 and restores the invariant for both re-parenting paths.

## Proof of Concept

// and a separate threshold node Q (threshold=1) with child C3

POST /transactions/T/approvers
{
  "approversArray": [
    { "threshold": 2, "approvers": [{ "userId": 1 }, { "userId": 2 }] },
    { "threshold": 1, "approvers": [{ "userId": 3 }] }
  ]
}
// P.id = 10, C1.id = 11, C2.id = 12, Q.id = 20

// Step 1: Re-parent C1 from P to Q
PATCH /transactions/T/approvers/11
{ "listId": 20 }

// Result:
// P now has threshold=2 but only 1 child (C2) → threshold > approvers.length
// Approval of P can never be satisfied; transaction T is permanently blocked
