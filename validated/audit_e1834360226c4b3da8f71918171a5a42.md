### Title
Scheduler Re-Processes All Pending Transactions Every 10 Seconds Without Change Detection, Causing Unnecessary Backend Load

### Summary
`TransactionSchedulerService` fetches and fully re-processes every pending transaction in a time window every 10 seconds, performing expensive deserialization and cryptographic operations on each transaction regardless of whether any new signatures have been added since the last cycle. This is the direct analog of the external report's finding: re-processing the same unchanged data repeatedly, rendering any implicit caching ineffective and placing unnecessary load on the backend.

### Finding Description

`handleTransactionsBetweenNowAndAfterThreeMinutes()` is decorated with `@Cron(CronExpression.EVERY_10_SECONDS)` and calls `updateTransactions()` on every tick. [1](#0-0) 

`updateTransactions()` fetches **all** transactions with status `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` in the time window, eagerly loading `creatorKey` and `groupItem.group` relations on every call: [2](#0-1) 

It then passes the full list to `processTransactionStatus()`. Inside that function, for **every** transaction in the list, regardless of whether any new signatures have arrived since the last cycle:

1. The transaction bytes are deserialized: `SDKTransaction.fromBytes(transaction.transactionBytes)` (CPU-intensive)
2. The required signature key is computed: `transactionSignatureService.computeSignatureKey(transaction)` (likely involves DB queries)
3. If signatures are sufficient, `smartCollate(transaction, signatureKey)` is called (CPU-intensive collation) [3](#0-2) 

The DB write is conditional (`if (transaction.status !== newStatus)`), so unnecessary writes are avoided. However, the expensive deserialization, signature-key computation, and collation steps execute unconditionally for every transaction on every 10-second tick. There is no mechanism to skip a transaction that has not received any new signatures since the previous cycle.

The same pattern repeats for `handleTransactionsBetweenThreeMinutesAndTenMinutes()` (every 30 seconds) and `handleTransactionsBetweenTenMinutesAndOneHour()` (every 10 minutes): [4](#0-3) 

### Impact Explanation
As the number of pending transactions in the near-term window grows, the CPU and database load from the scheduler grows proportionally. Every 10 seconds, the chain service performs N full transaction deserializations, N signature-key computations (with potential DB round-trips), and up to N `smartCollate` calls — all for transactions that may not have changed at all since the previous tick. Under high transaction volume, this can degrade overall system responsiveness and, in extreme cases, starve other operations of resources (DoS on the chain service).

### Likelihood Explanation
This is a realistic, reachable condition for any organization actively using the tool. Any deployment with a moderate number of pending transactions whose `validStart` falls within the ±3-minute window will trigger the full re-processing loop every 10 seconds. No attacker action is required; normal usage by legitimate users is sufficient to trigger the load.

### Recommendation
Introduce a lightweight change-detection mechanism so that transactions are only fully re-processed when their signature set has actually changed since the last cycle. One approach is to record a `lastSignatureHash` or `lastSignatureCount` on each transaction and skip `processTransactionStatus` for any transaction whose value has not changed. Alternatively, the scheduler could listen to signature-added events (already emitted via NATS) and maintain a dirty-set of transaction IDs that need re-evaluation, processing only those IDs on each tick rather than the entire window.

### Proof of Concept

1. Create a large number of transactions (e.g., 500) with `validStart` values within the next 3 minutes and status `WAITING_FOR_SIGNATURES`.
2. Do **not** add any new signatures.
3. Observe that every 10 seconds, `handleTransactionsBetweenNowAndAfterThreeMinutes` fires, fetches all 500 transactions with their relations, and calls `processTransactionStatus` — deserializing 500 transaction byte arrays, computing 500 signature keys, and potentially calling `smartCollate` 500 times.
4. Monitor CPU and DB query load on the chain service; it will spike every 10 seconds proportionally to the transaction count, with no useful work being done (no status changes occur because no signatures were added).

Relevant code path:
- `@Cron(EVERY_10_SECONDS) handleTransactionsBetweenNowAndAfterThreeMinutes()` → `updateTransactions(threeMinutesBefore, threeMinutesLater)` → `processTransactionStatus(repo, signatureService, allTransactions)` → per-transaction: `SDKTransaction.fromBytes(...)`, `computeSignatureKey(...)`, `smartCollate(...)` [5](#0-4)

  async updateTransactions(from: Date, to?: Date) {
    //Get the transaction, creatorKey, groupItem, and group. We need the group info upfront
    //in order to determine if the group needs to be processed together
    const transactions = await this.transactionRepo.find({
      where: {
        status: In([
          TransactionStatus.WAITING_FOR_SIGNATURES,
          TransactionStatus.WAITING_FOR_EXECUTION,
        ]),
        validStart: to ? Between(from, to) : MoreThan(from),
      },
      relations: {
        creatorKey: true,
        groupItem: {
          group: true,
        },
      },
      order: {
        validStart: 'ASC',
      },
    });

    const results = await processTransactionStatus(this.transactionRepo, this.transactionSignatureService, transactions);

    if (results.size > 0) {
      const events = Array.from(results.keys(), id => ({ entityId: id }));
      emitTransactionStatusUpdate(this.notificationsPublisher, events);
    }

    return transactions;
  }

export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }
