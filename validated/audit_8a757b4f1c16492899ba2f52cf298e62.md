### Title
Any Authenticated User Can Corrupt Another User's Transaction Group via Unauthorized `TransactionGroupItem` Deletion

### Summary
`removeTransactionGroup` in `TransactionGroupsService` fetches a `TransactionGroup` by ID with no ownership check, then deletes each `TransactionGroupItem` record **before** the ownership check on the underlying transaction is performed. When the ownership check fails (because the caller does not own the transaction), the group item has already been permanently deleted, leaving the group in a corrupted, inconsistent state. Any authenticated, verified user can trigger this against any group ID.

### Finding Description

**Root cause — missing pre-deletion ownership check in `removeTransactionGroup`:** [1](#0-0) 

The function at line 172 fetches the group with `findOneBy(TransactionGroup, { id })` — no user filter, no ownership assertion. It then fetches all `TransactionGroupItem` records for that group (lines 177–186), again with no user filter.

The critical sequence is inside the loop (lines 187–191):

```typescript
for (const groupItem of groupItems) {
  const transactionId = groupItem.transactionId;
  await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // ← hard-deleted first
  await this.transactionsService.removeTransaction(transactionId, user, false); // ← ownership checked here
}
```

`removeTransaction` delegates to `getTransactionForCreator`, which throws `UnauthorizedException` if `transaction.creatorKey?.userId !== user?.id`: [2](#0-1) 

By the time this check runs, the `TransactionGroupItem` row has already been hard-deleted from the database. The exception propagates before `remove(TransactionGroup, group)` (line 193) is reached, so the `TransactionGroup` shell remains — but with one fewer item than it should have.

**Attacker entry point — no guard beyond `VerifiedUserGuard`:** [3](#0-2) 

No `AdminGuard` or ownership guard is applied. Any authenticated, verified user can call `DELETE /transaction-groups/:id`.

**Group ID enumeration — `GET /transaction-groups` returns all groups:** [4](#0-3) 

`getTransactionGroups()` returns every `TransactionGroup` in the database with no user filter, allowing an attacker to enumerate all valid group IDs.

**Contrast with `cancelTransactionGroup` — which correctly checks ownership first:** [5](#0-4) 

`cancelTransactionGroup` calls `getTransactionGroup(user, groupId, false)` which runs a user-scoped SQL query and throws `UnauthorizedException` if the user has no access — before any mutation occurs. `removeTransactionGroup` skips this step entirely.

### Impact Explanation

An attacker who is a regular authenticated user can:

1. Enumerate all group IDs via `GET /transaction-groups`.
2. Call `DELETE /transaction-groups/:victimGroupId`.
3. The first `TransactionGroupItem` in the victim's group is permanently hard-deleted from the database.
4. The `TransactionGroup` record remains, but is now missing one or more items.

This causes **permanent, unrecoverable corruption** of another user's transaction group. In a multi-signature workflow, the group creator loses visibility into their grouped transactions, and the group's sequential execution logic is broken. The victim's pending Hedera transactions may become unexecutable through the group interface.

### Likelihood Explanation

- Attacker precondition: authenticated, verified organization user — the lowest privilege level in the system.
- No admin role, no leaked credentials, no privileged keys required.
- Group IDs are discoverable via the unauthenticated-to-role `GET /transaction-groups` endpoint.
- The attack is a single HTTP `DELETE` request.

### Recommendation

Move the ownership check **before** any mutation. The simplest fix is to call `getTransactionGroup(user, id, false)` at the start of `removeTransactionGroup` (exactly as `cancelTransactionGroup` does), which will throw `UnauthorizedException` if the user has no access to the group before any `TransactionGroupItem` is touched. Additionally, wrap the entire loop in a database transaction so that a mid-loop failure does not leave the group in a partially-deleted state.

### Proof of Concept

**Setup:**
- User A (attacker) and User B (victim) are both authenticated, verified members of the same organization.
- User B creates a transaction group (group ID = 5) containing two transactions.

**Attack:**
```
DELETE /transaction-groups/5
Authorization: Bearer <User A's JWT>
```

**Result:**
- `removeTransactionGroup` fetches group 5 (no ownership check).
- Fetches both `TransactionGroupItem` records.
- Deletes `TransactionGroupItem` for transaction 1 (hard delete, no ownership check).
- Calls `removeTransaction(tx1_id, userA, false)` → `getTransactionForCreator` throws `UnauthorizedException` because `tx1.creatorKey.userId === userB.id ≠ userA.id`.
- Exception propagates; `TransactionGroup` record 5 remains.
- **Net effect:** `TransactionGroupItem` for transaction 1 is permanently deleted. Group 5 now has only one item instead of two. User B's group is corrupted.

```typescript
  async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
    if (!group) {
      throw new Error('group not found');
    }
    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
      relations: {
        group: true,
      },
      where: {
        group: {
          id: group.id,
        },
      },
    });
    for (const groupItem of groupItems) {
      const transactionId = groupItem.transactionId;
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(transactionId, user, false);
    }

    await this.dataSource.manager.remove(TransactionGroup, group);

    emitTransactionUpdate(this.notificationsPublisher, groupItems.map(gi => ({ entityId: gi.transactionId })));

    return true;
  }
```
