Audit Report

## Title
Unvalidated Signature Submission in `uploadSignatureMaps` Allows Signer Impersonation and Transaction Sabotage

## Summary

`SignersService.uploadSignatureMaps` accepts and persists cryptographically invalid signatures. The `processTransactionSignatures` private method calls `sdkTransaction.addSignature(publicKey, map)` without first calling `validateSignature()`, unlike the parallel `importSignatures` path which does. An authenticated user with a registered key can submit a structurally valid but cryptographically garbage signature, corrupt the stored transaction bytes, be recorded as a legitimate signer, and cause the transaction to advance to `WAITING_FOR_EXECUTION` — where it will be permanently rejected by Hedera.

## Finding Description

**Root Cause**

Two code paths handle signature uploads:

1. `TransactionsService.importSignatures` calls `validateSignature()` before accepting any signature: [1](#0-0) 

`validateSignature` calls `publicKey.verify(bodyBytes, signature)` and throws on failure: [2](#0-1) 

2. `SignersService.processTransactionSignatures` calls `addSignature` directly with **no cryptographic check**: [3](#0-2) 

The only post-hoc check is a byte-equality comparison to detect whether bytes changed — not whether the signature is valid: [4](#0-3) 

**Input Validation Gap**

The `IsSignatureMap` decorator only validates structural correctness (non-empty bytes, valid account/transaction ID format, parseable public key). It performs no cryptographic verification: [5](#0-4) 

A 64-byte zero-filled payload passes all structural checks (`decodedSignature.length === 0` is false) and reaches `processTransactionSignatures` unchallenged.

**Persistence of Corrupted State**

Since `!isSameBytes` is true after `addSignature` embeds the garbage bytes, `persistSignatureChanges` writes the corrupted `transactionBytes` to the database and inserts a `TransactionSigner` row for the attacker: [6](#0-5) 

**Status Advancement**

`processTransactionStatus` reads `_signerPublicKeys` from the in-memory (already-corrupted) SDK transaction. `hasValidSignatureKey` only checks key presence, not signature validity: [7](#0-6) [8](#0-7) 

If the attacker's key satisfies the threshold, the transaction is promoted to `WAITING_FOR_EXECUTION`.

**Execution Failure**

`getValidatedSDKTransaction` in the chain service also only checks key presence via `hasValidSignatureKey`, not cryptographic validity: [9](#0-8) 

The transaction is submitted to Hedera, which rejects it. `_executeTransaction` marks it `FAILED`: [10](#0-9) 

**Existing E2E Test Does Not Cover This Case**

The existing negative test submits `'invalid-signature'` (a non-decodable string), which is caught by the `IsSignatureMap` structural check with `ErrorCodes.ISNMP`. It does not test a structurally valid but cryptographically invalid signature (e.g., 64 zero bytes hex-encoded), which bypasses the decorator and reaches `processTransactionSignatures`: [11](#0-10) 

## Impact Explanation

- **Transaction sabotage / permanent lock**: Any transaction requiring the attacker's key can be permanently destroyed. Once `FAILED`, the transaction cannot be re-executed.
- **Signer impersonation**: The attacker is recorded as a legitimate signer (`TransactionSigner` row inserted) without ever producing a valid cryptographic signature.
- **Integrity failure**: The stored `transactionBytes` in the database are corrupted with an invalid signature, violating the invariant that persisted transaction bytes contain only valid signatures.
- **High-value targets**: Governance or node-management transactions (e.g., `NodeUpdateTransaction`, `FreezeTransaction`) requiring specific council or admin keys can be permanently blocked by any registered user whose key is in the required key set.

## Likelihood Explanation

- **Attacker preconditions**: Must be a registered, verified user with at least one key registered. No privileged access required.
- **Reachability**: `POST /transactions/:transactionId/signers` is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard user-level authentication.
- **Effort**: Trivial. Construct a `SignatureMap` with the attacker's registered public key and 64 zero bytes as the signature value. The `IsSignatureMap` decorator accepts it (non-empty, valid format).
- **Detectability**: Silent. The server returns HTTP 201 with a valid signer record. The failure only surfaces later when the chain service attempts network submission.

## Recommendation

Add a call to `validateSignature()` inside `processTransactionSignatures`, immediately before `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before line 251:
const { data: validPublicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);

// Then proceed with addSignature only for validated keys
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
```

This ensures both upload paths enforce the same cryptographic invariant before mutating transaction state. [12](#0-11) [13](#0-12) 

## Proof of Concept

1. Register a user with key pair `(privK, pubK)` in the organization.
2. Obtain or create a multi-sig transaction that requires `pubK` as one of the required signers (status: `WAITING_FOR_SIGNATURES`).
3. Construct a `SignatureMap` payload:
   ```json
   {
     "0.0.3": {
       "0.0.XXXX@YYYY.ZZZZ": {
         "<pubK_DER_hex>": "<128 hex chars of 0x00>"
       }
     }
   }
   ```
   The 64-byte zero signature passes `IsSignatureMap` (non-empty, valid format).
4. Call `POST /transactions/{id}/signers` with this payload using the attacker's JWT.
5. Server responds HTTP 201. A `TransactionSigner` row is inserted for the attacker.
6. The corrupted `transactionBytes` (containing the zero-byte signature) are written to the database.
7. If the attacker's key satisfies the signing threshold, the transaction advances to `WAITING_FOR_EXECUTION`.
8. The chain service submits the transaction to Hedera; Hedera rejects it with `INVALID_SIGNATURE`.
9. The transaction is permanently marked `FAILED`.

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
  } else throw new Error(`Invalid key type`);
