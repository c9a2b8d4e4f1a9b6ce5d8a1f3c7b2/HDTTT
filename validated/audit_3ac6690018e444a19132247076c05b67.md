### Title
Unbounded O(N) Per-Request Computation in `GET /transactions/sign` Enables Authenticated DoS

### Summary
The `getTransactionsToSign` endpoint fetches **all** non-terminal transactions from the database without any row limit, then iterates over every one of them calling `userKeysToSign` â€” which deserializes SDK transaction bytes, computes a signature key, and issues a database query â€” before applying pagination in memory. Any authenticated user can repeatedly invoke this endpoint to force O(N) server-side work where N is the total number of active transactions in the organization, causing sustained CPU and database exhaustion that degrades service for all users.

### Finding Description

**Root cause â€” no `take` limit on the full-table scan:**

In `getTransactionsToSign`, the repository query has no `take` clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
}); // fetches every non-terminal transaction â€” no limit
``` [1](#0-0) 

**Root cause â€” per-transaction crypto + DB work inside the loop:**

For every row returned, the service calls `userKeysToSign`, which in turn calls `userKeysRequiredToSign` â†’ `keysRequiredToSign`. That function:
1. Deserializes the raw transaction bytes via `SDKTransaction.fromBytes()`
2. Calls `transactionSignatureService.computeSignatureKey()` (mirror-node / crypto work)
3. Issues a `entityManager.find(UserKey, ...)` database query [2](#0-1) [3](#0-2) 

**Root cause â€” pagination applied after all computation:**

The `limit`/`offset` slice happens only after the full loop completes, so requesting `page=1&size=1` still processes every active transaction:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit), // pagination is post-hoc
  ...
};
``` [4](#0-3) 

**Exposed endpoint â€” still reachable despite front-end deprecation:**

The route is decorated `/* NO LONGER USED BY FRONT-END */` but remains fully registered and protected only by JWT auth â€” no `@Throttle` override, no computation cap: [5](#0-4) 

**General throttler does not bound per-request work:**

The `UserThrottlerModule` allows 100 requests/minute and 10 requests/second per authenticated user. At 10 req/s, each triggering O(N) deserialization + crypto + DB work, the aggregate load scales linearly with the transaction table size. [6](#0-5) 

**Secondary surface â€” `importSignatures` array with no size cap:**

The developer explicitly noted the missing limit:

```typescript
//Added a batch mechanism, probably should limit this on the api side of things
const BATCH_SIZE = 500;
```

The `POST /transactions/signatures/import` endpoint accepts an unbounded `UploadSignatureMapDto[]`. Each element triggers `verifyAccess` (which calls `userKeysToSign`), SDK deserialization, and signature validation before the batch DB write. [7](#0-6) [8](#0-7) [9](#0-8) 

### Impact Explanation

A single authenticated user issuing repeated `GET /transactions/sign?page=1&size=1` requests forces the server to deserialize every active transaction's bytes, compute its signature key, and query the database â€” once per transaction, per request. As the organization accumulates transactions (which is the normal operational trajectory), each request becomes progressively more expensive. At scale this saturates the Node.js event loop and the PostgreSQL connection pool, causing request timeouts and service unavailability for all users. The `importSignatures` surface compounds this: a single POST with a large array of IDs triggers the same per-item crypto + DB work with no array-length guard.

### Likelihood Explanation

Preconditions: a valid JWT (any registered, verified organization user). No admin role, no leaked secrets, no physical access required. The attacker calls a standard REST endpoint that is publicly documented via Swagger. The general throttle of 100 req/min per user is generous enough that sustained load is achievable within normal rate limits, and multiple attacker accounts multiply the effect linearly.

### Recommendation

1. **Add a `take` limit to the `getTransactionsToSign` database query** so the full-table scan is bounded regardless of pagination intent. Pre-filter at the SQL level (e.g., join on `transaction_signer.userId`) instead of fetching all rows and filtering in application code.
2. **Apply pagination before the per-transaction crypto loop**, or restructure the query to identify candidate transactions via a set-based SQL join on user keys before deserializing any bytes.
3. **Enforce an array-size cap on `importSignatures`** (e.g., `@ArrayMaxSize(50)` on the DTO or an explicit guard in the controller) to bound per-request work on the secondary surface.
4. **Consider removing or gating the deprecated `GET /sign` endpoint** with a stricter per-endpoint `@Throttle` override if it must remain registered.

### Proof of Concept

```
# Attacker: any authenticated organization user
# Step 1 â€” obtain a valid JWT via normal login
TOKEN=$(curl -s -X POST https://<server>/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"attacker@org.com","password":"..."}' | jq -r .accessToken)

# Step 2 â€” repeatedly invoke the unbounded endpoint
# Each call forces O(N) deserialization + crypto + DB work
while true; do
  curl -s "https://<server>/transactions/sign?page=1&size=1" \
    -H "Authorization: Bearer $TOKEN" &
done
```

As the number of active (non-terminal) transactions in the organization grows, each iteration of the loop forces the server to process every one of them. With the 10 req/s per-user throttle and N active transactions, the server sustains N Ă— 10 deserialization + crypto + DB operations per second from a single attacker account. [10](#0-9)

  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,

  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }

export async function transformAndValidateDto<T extends object>(
  dtoClass: new (...args: any[]) => T,
  payload: T | T[],
): Promise<T[]> {
  const items = Array.isArray(payload) ? payload : [payload];
  const instances = items.map(item => plainToInstance(dtoClass, item));
  await Promise.all(instances.map(instance => validateOrReject(instance)));
  return instances;
