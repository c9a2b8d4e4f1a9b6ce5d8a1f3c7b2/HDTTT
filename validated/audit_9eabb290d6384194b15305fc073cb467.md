Audit Report

## Title
Cross-Transaction Approver Deletion via Missing Ownership Binding in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `:transactionId` but then deletes the approver identified by `:id` with no verification that the approver belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Controller** — `approvers.controller.ts` lines 102–113:

The controller calls `getCreatorsTransaction(transactionId, user)` to verify the caller owns the transaction at `:transactionId`, then immediately calls `removeTransactionApprover(id)` with the unvalidated `:id` param. [1](#0-0) 

**Service** — `approvers.service.ts` lines 533–544:

`removeTransactionApprover` fetches the approver by `id` alone and deletes it. It never checks that `approver.transactionId` (or its root node's `transactionId`) matches the `transactionId` that was authorized in the controller. The `transactionId` URL parameter is consumed only by the guard and then discarded. [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly performs the binding check before proceeding: [3](#0-2) 

The `update` path resolves the root node and asserts `rootNode.transactionId !== transactionId` before authorizing. The `delete` path has no equivalent check.

**`getTransactionApproverById`** fetches only by `id` with no transaction scope: [4](#0-3) 

## Impact Explanation

An attacker can silently remove approvers from any transaction they do not own:

- Multi-signature approval requirements for high-value Hedera transactions (account updates, token operations, etc.) can be stripped without the transaction creator's knowledge.
- A transaction requiring N-of-M approvals can be reduced to zero required approvers, allowing it to proceed to execution without the intended governance controls.
- The victim transaction creator receives no indication that their approver configuration was tampered with (the `emitTransactionStatusUpdate` in `removeTransactionApprover` fires against `approver.transactionId`, which is `null` for child nodes, further obscuring the action). [5](#0-4) 

## Likelihood Explanation

Attacker preconditions are minimal:

1. **Valid JWT** — any registered, verified user.
2. **Creator of at least one transaction** — trivially satisfied by creating any transaction.
3. **Knowledge of a target approver `id`** — approver IDs are sequential integers, enumerable by brute-force or by observing one's own approver IDs from the `GET /transactions/:transactionId/approvers` endpoint.

No admin role, no leaked credentials, and no special access is required. The endpoint is reachable by any authenticated user.

## Recommendation

In `removeTransactionApprover`, resolve the root node of the approver being deleted and assert that its `transactionId` matches the authorized `transactionId` parameter — exactly as `updateTransactionApprover` does at lines 386–394:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Binding check — mirrors updateTransactionApprover
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: rootNode.transactionId }]);
  return result;
}
```

The controller must also pass `transactionId` to the service call:
```typescript
await this.approversService.removeTransactionApprover(id, transactionId);
```

## Proof of Concept

```
# Attacker owns transaction T1 (id=1), victim owns transaction T2 (id=2)
# T2 has an approver with id=99

# Step 1 — Attacker discovers approver id=99 (e.g., by observing their own approver IDs
#           or by brute-forcing the sequential integer space)

# Step 2 — Attacker sends DELETE using their own transactionId=1 as the guard bypass,
#           but targets approver id=99 which belongs to T2

DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>

# Result:
# - getCreatorsTransaction(1, attacker) → PASSES (attacker owns T1)
# - removeTransactionApprover(99)       → DELETES approver 99 from T2 with no binding check
# - HTTP 200: true

# T2's approver configuration is now silently modified.
``` [6](#0-5)
