Audit Report

## Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Malicious Signer to Corrupt Multi-Sig Transactions

## Summary
The `POST /transactions/:transactionId/signers` endpoint routes through `SignersService.uploadSignatureMaps` → `processTransactionSignatures`, which calls `sdkTransaction.addSignature(publicKey, map)` without first cryptographically verifying that the submitted signature bytes are valid for the given public key and transaction body. A parallel code path — `TransactionsService.importSignatures` — correctly calls `validateSignature` before `addSignature`. Any JWT-authenticated, verified user who is a listed signer on a transaction can submit their own registered public key paired with arbitrary/invalid signature bytes. The system will accept the submission, record the user as a signer, overwrite the stored transaction bytes with the invalid signature, and potentially advance the transaction to `WAITING_FOR_EXECUTION`. When the chain service submits the transaction, the Hedera network rejects it, permanently corrupting the transaction.

## Finding Description

**Root cause — missing `validateSignature` call in the signers upload path**

`validateSignature` in `back-end/libs/common/src/utils/sdk/transaction.ts` is the only place in the codebase that performs a real cryptographic check. It iterates over every entry in the `SignatureMap` and calls `publicKey.verify(bodyBytes, signature)`, throwing `'Invalid signature'` on failure: [1](#0-0) 

The `importSignatures` path in `TransactionsService` correctly calls this function before calling `addSignature`: [2](#0-1) 

However, the primary signer upload path — `SignersService.processTransactionSignatures` — skips `validateSignature` entirely. It only checks whether the public key belongs to the authenticated user (via `userKeyMap` lookup), then immediately calls `addSignature`: [3](#0-2) 

The Hedera SDK's `addSignature` does not perform cryptographic verification; it blindly appends whatever bytes are in the map to the transaction's signed-transaction list. The `_signerPublicKeys` set is populated by `addSignature` regardless of whether the signature is valid.

**The downstream execution check does not catch this**

`ExecuteService.getValidatedSDKTransaction` calls `hasValidSignatureKey` against `sdkTransaction._signerPublicKeys`: [4](#0-3) 

`hasValidSignatureKey` only checks whether the public key string is present in the set — it performs no cryptographic verification of the associated signature bytes: [5](#0-4) 

So the execution gate passes even when the stored signature bytes are garbage.

**Attacker-controlled entry point**

The `POST /transactions/:transactionId/signers` endpoint is reachable by any JWT-authenticated, verified user: [6](#0-5) [7](#0-6) 

The `IsSignatureMap` decorator validates only structural format (account ID format, transaction ID format, non-empty byte length) — it performs no cryptographic check: [8](#0-7) 

**Corrupted bytes are persisted**

When `isSameBytes` is `false` (i.e., the invalid signature actually changed the transaction bytes), `persistSignatureChanges` writes the corrupted `transactionBytes` back to the database: [9](#0-8) 

## Impact Explanation

- **Transaction integrity destruction**: The stored `transactionBytes` are overwritten with invalid signature data. The original valid bytes are unrecoverable because no backup is kept after the update.
- **Threshold bypass / false advancement**: If the attacker's fake signature is the last one needed to meet the signing threshold, `processTransactionStatus` advances the transaction to `WAITING_FOR_EXECUTION` and it is submitted to the Hedera network, where it fails with `INVALID_SIGNATURE`.
- **Denial of service on transaction lifecycle**: The transaction enters `FAILED` state. All prior legitimate signers' work is wasted. Depending on the transaction's `validStart` window, re-creation may not be possible.
- **False audit trail**: The attacker is recorded as a legitimate signer in the `TransactionSigner` table despite never producing a valid signature.

## Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated user with a registered key pair who is listed as a signer on the target transaction. This is a normal, non-privileged role in the system.
- **No special tooling required**: The attacker only needs to craft a JSON body with their own DER public key and arbitrary hex bytes as the signature value. A standard HTTP client suffices.
- **No cryptographic break required**: The attack exploits a missing validation check, not a weakness in any cryptographic primitive.
- **Realistic scenario**: In an organization using this tool for governance or treasury transactions, a disgruntled or malicious signer can sabotage any transaction they are listed on.

## Recommendation

Add a `validateSignature` call inside `processTransactionSignatures` in `back-end/apps/api/src/transactions/signers/signers.service.ts`, mirroring the pattern already used in `TransactionsService.importSignatures`. Specifically, before calling `sdkTransaction.addSignature(publicKey, map)` at line 251, call `validateSignature(sdkTransaction, map)` and throw (or return an error) if it fails. This ensures that only cryptographically valid signatures are accepted and persisted, regardless of which upload path is used.

## Proof of Concept

1. Attacker is a registered, verified user with key pair (`publicKeyA` / `privateKeyA`) listed as a signer on transaction `T` (status: `WAITING_FOR_SIGNATURES`).

2. Attacker constructs a `SignatureMap` JSON body with `publicKeyA` (DER-encoded) mapped to 64 bytes of arbitrary data (e.g., all zeros hex-encoded). This passes the `decodedSignature.length === 0` guard in `IsSignatureMap`.

3. Attacker POSTs to `POST /transactions/T/signers` with a valid JWT:
```json
[{
  "id": <T>,
  "signatureMap": {
    "<nodeAccountId>": {
      "<transactionId>": {
        "<publicKeyA_DER>": "<64_zero_bytes_hex>"
      }
    }
  }
}]
```

4. `processTransactionSignatures` finds `publicKeyA` in `userKeyMap` (ownership check passes). No `validateSignature` call is made. `sdkTransaction.addSignature(publicKeyA, map)` is called — invalid bytes are written into the transaction's signed-transaction list.

5. `isSameBytes` is `false` (bytes changed), so `persistSignatureChanges` overwrites `transaction.transactionBytes` in the database with the corrupted bytes. A `TransactionSigner` record is inserted for the attacker.

6. `processTransactionStatus` runs. If the signing threshold is now met, the transaction advances to `WAITING_FOR_EXECUTION`.

7. `ExecuteService.getValidatedSDKTransaction` calls `hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey)`. Since `addSignature` populated `_signerPublicKeys` with `publicKeyA`, the key-presence check passes. The transaction is submitted to the Hedera network.

8. The Hedera network rejects the transaction with `INVALID_SIGNATURE`. The transaction enters `FAILED` state. The corrupted bytes remain in the database; the original valid bytes from other legitimate signers are permanently lost.

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```

export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
  } else throw new Error(`Invalid key type`);
```
