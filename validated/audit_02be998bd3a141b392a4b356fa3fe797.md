Audit Report

## Title
Missing Ownership Check in `removeTransactionGroup` Allows Any Authenticated User to Corrupt Any Transaction Group's State

## Summary
`removeTransactionGroup` in `transaction-groups.service.ts` accepts a `user` parameter but performs no ownership or authorization check on the target `TransactionGroup` before beginning destructive operations. Any authenticated, verified user can call `DELETE /transaction-groups/:id` against a group they do not own. `TransactionGroupItem` records are deleted **before** any per-transaction ownership check fires, resulting in permanent, unrecoverable partial deletion of another user's group state with no database transaction to roll back the damage.

## Finding Description

**Root cause confirmed:** `removeTransactionGroup` fetches the group by raw ID with no user filter and immediately begins deleting its items.

At [1](#0-0) , the function:

1. Fetches the group with `findOneBy(TransactionGroup, { id })` — no user filter.
2. Iterates over all `TransactionGroupItem` records.
3. **Calls `remove(TransactionGroupItem, groupItem)` first** (line 189), permanently hard-deleting the row.
4. Only then calls `removeTransaction(transactionId, user, false)` (line 190), which delegates to `getTransactionForCreator`.

`getTransactionForCreator` at [2](#0-1)  enforces ownership via `transaction.creatorKey?.userId !== user?.id`, throwing `UnauthorizedException`. But by the time this fires, the `TransactionGroupItem` row is already gone from the database.

There is **no wrapping `dataSource.transaction()`** around the loop in `removeTransactionGroup`, so each `remove()` call is committed immediately and cannot be rolled back.

**Contrast with `cancelTransactionGroup`** at [3](#0-2) , which correctly calls `getTransactionGroup(user, groupId, false)` — a user-filtered query — and then asserts `allOwnedByUser` before touching any data. `removeTransactionGroup` skips both steps entirely.

## Impact Explanation

An attacker who is a legitimate, verified organization member (no admin role required) can:

1. Call `DELETE /transaction-groups/:id` against a group owned by another user.
2. For the first `TransactionGroupItem` in the target group, the item is **permanently hard-deleted** from the database before the ownership check fires and throws.
3. The `TransactionGroup` record and the underlying `Transaction` records survive (the exception halts execution before `remove(TransactionGroup, group)` at line 193), but the first `TransactionGroupItem` linking row is gone.
4. The attacker can repeat this request to delete items one by one until the group has no items.
5. Once all items are deleted, `getTransactionGroup` throws `UnauthorizedException` at [4](#0-3)  because `groupItems.length === 0`, making the group permanently inaccessible and unmanageable by its legitimate owner.

## Likelihood Explanation

- **Attacker precondition**: A valid JWT for any verified organization user. No admin role, no leaked secrets.
- **Entry point**: `DELETE /transaction-groups/:id` — a standard REST endpoint reachable by any authenticated client, as seen in the controller at [5](#0-4) .
- **Enumeration**: `GET /transaction-groups` at [6](#0-5)  is marked "TESTING ONLY" but is guarded only by the class-level `VerifiedUserGuard` — no role restriction — and returns **all** groups via `getTransactionGroups()` which calls `find(TransactionGroup)` with no filter. Group IDs are sequential integers.
- **Effort**: One HTTP DELETE request per item to be deleted. Repeating the request N times destroys an N-item group.

## Recommendation

1. **Add an ownership check at the top of `removeTransactionGroup`**, mirroring `cancelTransactionGroup`: call `getTransactionGroup(user, id, false)` and verify `allOwnedByUser` before any destructive operation.
2. **Wrap the entire deletion loop in a `dataSource.transaction()`** so that any mid-loop failure rolls back all prior deletions atomically.
3. **Restrict or remove `GET /transaction-groups`**: the "TESTING ONLY" endpoint returns all groups to any verified user and enables trivial ID enumeration. Add a role guard (e.g., admin-only) or remove it from production builds.

## Proof of Concept

```
# Step 1: Enumerate all group IDs (no admin role needed)
GET /transaction-groups
Authorization: Bearer <attacker_jwt>
# Returns all TransactionGroup records including victim's group with id=42

# Step 2: Repeatedly delete items from victim's group one by one
DELETE /transaction-groups/42
Authorization: Bearer <attacker_jwt>
# First TransactionGroupItem for group 42 is permanently deleted.
# UnauthorizedException is thrown but damage is already committed.

# Repeat until group has 0 items — victim's group is now permanently inaccessible.
```

```typescript
  async removeTransactionGroup(user: User, id: number): Promise<boolean> {
    const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
    if (!group) {
      throw new Error('group not found');
    }
    const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
      relations: {
        group: true,
      },
      where: {
        group: {
          id: group.id,
        },
      },
    });
    for (const groupItem of groupItems) {
      const transactionId = groupItem.transactionId;
      await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(transactionId, user, false);
    }

    await this.dataSource.manager.remove(TransactionGroup, group);

    emitTransactionUpdate(this.notificationsPublisher, groupItems.map(gi => ({ entityId: gi.transactionId })));

    return true;
  }
```

```typescript
    // Verify the user is the creator of all transactions in the group
```
