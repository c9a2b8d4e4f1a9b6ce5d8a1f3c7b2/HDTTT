### Title
`removeTransactionGroup` Deletes Group Items Before Enforcing Creator Authorization, Enabling Partial State Corruption by Any Authenticated User

### Summary
`removeTransactionGroup` in `transaction-groups.service.ts` performs no ownership check on the group itself before beginning deletion. It deletes each `TransactionGroupItem` row first, then calls `removeTransaction` which enforces a creator-only check. Because the item deletion is committed outside any database transaction, a non-creator attacker can permanently orphan group items from a victim's group before the authorization check fires, leaving the group in an unrecoverable corrupted state.

### Finding Description

**Vulnerability class**: Authorization bypass → permanent state corruption.

**Root cause — split access control with wrong operation order**

The external report describes a function that gates entry with `onlyOwner` but then checks `msg.sender == seller/buyer` inside, making it impossible for any legitimate caller to succeed. The analog here is the inverse: `removeTransactionGroup` has *no* group-level ownership gate at entry, but the inner call `removeTransaction` *does* enforce creator ownership — after the destructive write has already been committed.

**Entry point — unauthenticated group enumeration**

`GET /transaction-groups` returns every group in the database to any authenticated user with no filtering: [1](#0-0) 

**Entry point — unguarded delete**

`DELETE /transaction-groups/:id` is protected only by JWT + `VerifiedUserGuard`. No ownership check exists on the group: [2](#0-1) 

**Destructive write before authorization check**

Inside `removeTransactionGroup`, the loop deletes the `TransactionGroupItem` row on line 189 *before* calling `removeTransaction` on line 190. There is no wrapping database transaction: [3](#0-2) 

**Creator check fires too late**

`removeTransaction` delegates immediately to `getTransactionForCreator`, which throws `UnauthorizedException` when the caller is not the creator: [4](#0-3) [5](#0-4) 

**Contrast with `cancelTransactionGroup`**

`cancelTransactionGroup` correctly checks ownership *before* any mutation: [6](#0-5) 

`removeTransactionGroup` has no equivalent guard.

### Impact Explanation

For each iteration of the loop, the attacker causes one `TransactionGroupItem` row to be permanently deleted from the victim's group before the `UnauthorizedException` is thrown and the loop aborts. The `TransactionGroup` record and the underlying `Transaction` records survive, but the group is now missing one or more of its items. The group is left in an unrecoverable corrupted state: it exists in the database but its item list is incomplete. The victim's group-based workflow (sequential/atomic execution, signing coordination) is permanently broken for those items. Because the operation is not wrapped in a database transaction, there is no rollback.

### Likelihood Explanation

- Attacker preconditions: any registered, verified user account — no admin or privileged role required.
- Group IDs are directly discoverable via `GET /transaction-groups`, which returns all groups with no per-user filtering.
- The attack requires a single HTTP `DELETE` request with a known group ID.
- No rate limiting or additional confirmation is required.
- The attack is fully deterministic and repeatable.

### Recommendation

1. **Add an upfront ownership check** in `removeTransactionGroup` before any mutation, mirroring `cancelTransactionGroup`. Verify that every transaction in the group belongs to the requesting user before touching any row.
2. **Wrap the entire deletion loop in a database transaction** so that any authorization failure rolls back all prior deletions atomically.
3. **Restrict `GET /transaction-groups`** to return only groups the requesting user has access to (or remove the endpoint entirely — it is already marked `TESTING ONLY`).

### Proof of Concept

**Setup**: User A creates a transaction group with two transactions. User B is any other authenticated, verified user.

**Steps**:

1. User B authenticates and calls:
   ```
   GET /transaction-groups
   ```
   Response includes User A's group with `id: 42`.

2. User B calls:
   ```
   DELETE /transaction-groups/42
   ```

3. Server executes `removeTransactionGroup(userB, 42)`:
   - Line 173: group 42 is found (no ownership check).
   - Line 177–186: all `TransactionGroupItem` rows for group 42 are loaded.
   - **First loop iteration**:
     - Line 189: `TransactionGroupItem` for transaction T1 is **deleted and committed**.
     - Line 190: `removeTransaction(T1, userB, false)` → `getTransactionForCreator` → `userB.id !== T1.creatorKey.userId` → throws `UnauthorizedException`.
   - Loop aborts. Group 42 and transaction T1 survive, but the `TransactionGroupItem` linking them is gone.

4. **Observed state**: Group 42 now has only one item instead of two. User A's group is permanently corrupted. Repeating the attack removes the next item.
