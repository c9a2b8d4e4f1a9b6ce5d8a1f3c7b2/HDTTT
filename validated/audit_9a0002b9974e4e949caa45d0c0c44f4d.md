Audit Report

## Title
IDOR: Transaction Creator Can Delete Approvers Belonging to Any Other Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` verifies the caller is the creator of `:transactionId`, but passes the raw `:id` (approver primary key) directly to `removeTransactionApprover()` with no check that the targeted approver actually belongs to `:transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description
The controller handler at `back-end/apps/api/src/transactions/approvers/approvers.controller.ts` lines 102–113 performs two independent, uncorrelated operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // only checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by raw id — no cross-check
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller created the transaction identified by `:transactionId` — it says nothing about the approver identified by `:id`: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by its own primary key and soft-deletes it with no assertion that `approver.transactionId === transactionId`: [3](#0-2) 

By contrast, `updateTransactionApprover` correctly walks up to the root node and validates `rootNode.transactionId !== transactionId` before proceeding — the delete path has no equivalent guard: [4](#0-3) 

## Impact Explanation
- An attacker who is the creator of any transaction (even a trivially created one) can enumerate approver IDs (sequential auto-increment integers) and issue `DELETE /transactions/{attacker_tx_id}/approvers/{victim_approver_id}`.
- This silently soft-deletes an approver belonging to a completely different transaction via `removeNode`, which sets `deletedAt = now()` on the entire approver subtree.
- Approval thresholds on victim transactions are silently reduced or eliminated, allowing transactions to advance to execution without the required signatures.
- In multi-sig workflows this directly undermines the security model: a transaction requiring N approvals can be stripped down to fewer, enabling unauthorized execution.
- The soft-delete corrupts the audit trail permanently for the affected approver records. [5](#0-4) 

## Likelihood Explanation
- **Attacker precondition**: only a valid, verified account is needed. Any user can create a transaction and become its creator.
- **Discovery**: approver IDs are auto-increment integers; blind enumeration across a small range is trivial.
- The attack requires a single authenticated HTTP request with no special privileges beyond account ownership.
- No cross-object ownership check exists on the delete path, making exploitation straightforward and reliable.

## Recommendation
In `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the targeted approver and assert that `rootNode.transactionId === transactionId` before proceeding with deletion — exactly as `updateTransactionApprover` does at lines 386–391. For example:

```typescript
// In the controller or service, before removeNode:
const rootNode = await this.approversService.getRootNodeFromNode(id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
``` [6](#0-5) 

## Proof of Concept
1. Attacker authenticates and creates Transaction A (`tx_id = 10`), becoming its creator.
2. Attacker discovers (via enumeration or the GET approvers endpoint on a transaction they can view) that Transaction B (`tx_id = 99`) has an approver with `id = 42`.
3. Attacker sends:
   ```
   DELETE /transactions/10/approvers/42
   Authorization: Bearer <attacker_token>
   ```
4. `getCreatorsTransaction(10, attacker)` passes — attacker is the creator of transaction 10.
5. `removeTransactionApprover(42)` fetches approver 42 (which belongs to transaction 99) and calls `removeNode(42)`, soft-deleting it and its entire subtree.
6. Transaction 99's approval threshold is now silently reduced or eliminated, with no error returned to the attacker and no alert to the victim. [7](#0-6)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

    // notifyTransactionAction(this.notificationsService);
  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
