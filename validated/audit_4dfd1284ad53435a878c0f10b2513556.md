Audit Report

## Title
Single Approval Satisfies Multiple Threshold Slots Due to Cross-Branch Duplicate Approver Placement

## Summary
`isNode` in `approvers.service.ts` only prevents placing the same user under the **same parent node** (`listId`). It does not enforce global uniqueness of a user across the entire approver tree for a transaction. Because `approveTransaction` bulk-updates **all** `TransactionApprover` rows belonging to the user in a single query, one approval call simultaneously satisfies multiple threshold slots across different branches, allowing a transaction to execute with fewer unique approvals than the tree structure implies.

## Finding Description

**Root cause — `isNode` scopes its duplicate check to `listId`:**

`isNode` in `approvers.service.ts` builds its `where` clause as:

```typescript
where: {
  listId: typeof approver.listId === 'number' ? approver.listId : null,
  userId: typeof approver.userId === 'number' ? approver.userId : null,
  ...
  transactionId: typeof approver.listId === 'number' ? null : transactionId,
},
``` [1](#0-0) 

Adding User A under `listId=10` queries `{listId:10, userId:A}`. Adding User A under `listId=20` queries `{listId:20, userId:A}`. Neither returns a count > 0 for the other, so both insertions succeed. The same user can occupy multiple leaf nodes across different branches of the same transaction's approver tree.

**Compounding issue — existing approval is copied to new branches on creation:**

When a user who has already approved is added to a new branch, `getApproversByTransactionId` returns the existing record and its `signature`, `userKeyId`, and `approved` fields are immediately copied into the new record:

```typescript
if (userApproverRecords.length > 0) {
  data.signature = userApproverRecords[0].signature;
  data.userKeyId = userApproverRecords[0].userKeyId;
  data.approved = userApproverRecords[0].approved;
}
``` [2](#0-1) 

The new branch is retroactively satisfied without any action from the approver.

**Approval bulk-update covers all slots for the user:**

In `approveTransaction`, `userApprovers` collects every `TransactionApprover` row where `userId === user.id` across the entire tree:

```typescript
const userApprovers = approvers.filter(a => a.userId === user.id);
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
``` [3](#0-2) 

If none have a signature yet, the guard passes. The subsequent `whereInIds` bulk-update then stamps the signature on **all** of the user's slots simultaneously:

```typescript
.whereInIds(userApprovers.map(a => a.id))
.execute();
``` [4](#0-3) 

## Impact Explanation
A malicious transaction creator can design an approver tree that visually and structurally implies a higher approval threshold than is actually enforced. Organizational members who inspect the tree see "threshold 2 of 3" and believe two independent parties must approve. In reality, a single user's approval satisfies two slots simultaneously, allowing the transaction to execute with fewer unique approvals than intended. This directly undermines the integrity of the multi-party approval governance model — the core security property of the organizational workflow.

## Likelihood Explanation
The attacker is the transaction creator, a role available to any authenticated organization user with no elevated privileges. The exploit requires only crafting a specific approver tree structure via the documented `POST /transactions/:id/approvers` API. No race conditions, cryptographic breaks, or external dependencies are required. Any creator who wants to reduce the effective approval threshold while concealing that reduction can do so deterministically.

## Recommendation

1. **Enforce global per-transaction uniqueness of user approvers.** In `isNode`, remove the `listId` constraint from the duplicate check when `userId` is set, or add a separate global check: query `TransactionApprover` for `{transactionId (via recursive CTE), userId}` across the entire tree and reject if any record already exists for that user.

2. **Do not copy approval state to newly created branches.** Remove lines 325–329 in `createTransactionApprovers`. Each approver slot should start unsigned; retroactively copying approval state defeats the purpose of adding a new slot.

3. **Scope the approval update to a single record.** In `approveTransaction`, instead of bulk-updating all records for the user, update only the specific approver record(s) that are legitimately unsatisfied, or enforce that a user may only appear once per transaction tree.

## Proof of Concept

**Setup (as transaction creator):**

```
POST /transactions/42/approvers
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [
      {
        "threshold": 1,
        "approvers": [{ "userId": 7 }]   // User A under branch 1
      },
      {
        "threshold": 1,
        "approvers": [{ "userId": 7 }]   // User A under branch 2 — isNode passes (different listId)
      },
      {
        "threshold": 1,
        "approvers": [{ "userId": 9 }]   // User B under branch 3
      }
    ]
  }]
}
```

`isNode` is called for User A under branch 1 (`listId=branch1.id`) → count=0 → passes.
`isNode` is called for User A under branch 2 (`listId=branch2.id`) → count=0 → passes.
Two `TransactionApprover` rows are created for User A with different `listId` values.

**Exploit (as User A):**

```
POST /transactions/42/approvers/approve
{ "userKeyId": 3, "signature": "<valid_sig>", "approved": true }
```

`userApprovers` = [row for branch 1, row for branch 2].
`userApprovers.every(a => a.signature)` → false (none signed yet) → guard passes.
`whereInIds([branch1_row_id, branch2_row_id])` → both rows updated with signature.

**Result:** Both child threshold nodes (threshold=1) are now satisfied. The root threshold of 2 is met. The transaction proceeds to execution without User B's approval, despite the displayed structure implying 2-of-3 independent approvers are required. [5](#0-4) [6](#0-5) [7](#0-6)
