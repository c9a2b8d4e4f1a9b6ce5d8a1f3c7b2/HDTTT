### Title
Cross-Transaction Approver Deletion via Missing Ownership Validation in `removeTransactionApprover`

### Summary

`removeTransactionApprover` in `approvers.service.ts` fetches and deletes an approver record by its own `id` without verifying that the approver belongs to the `transactionId` supplied in the URL. The controller only checks that the requesting user is the creator of the URL-parameter transaction, not that the target approver record is associated with it. Any authenticated user who owns at least one transaction can therefore delete approvers from any other transaction in the system.

### Finding Description

**Root cause — missing cross-reference check in the service layer**

The controller endpoint is:

```
DELETE /transactions/:transactionId/approvers/:id
``` [1](#0-0) 

It performs exactly one authorization check:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
```

`getCreatorsTransaction` verifies the caller is the creator of `transactionId` (the URL parameter). It says nothing about whether the approver identified by `id` belongs to that transaction. [2](#0-1) 

`removeTransactionApprover` then fetches the approver by its own primary key and immediately deletes it:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);   // no transactionId filter
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);            // deletes node + subtree
    emitTransactionStatusUpdate(this.notificationsPublisher,
        [{ entityId: approver.transactionId }]);                  // null for child nodes
    return result;
}
``` [3](#0-2) 

Contrast this with `updateTransactionApprover`, which correctly calls `getRootNodeFromNode` after fetching the approver and explicitly asserts `rootNode.transactionId !== transactionId` before proceeding: [4](#0-3) 

That guard is entirely absent in `removeTransactionApprover`.

**Secondary defect — silent notification failure for child-node deletions**

When the deleted approver is a child node (non-root), its `transactionId` column is `null` by design: [5](#0-4) 

`emitTransactionStatusUpdate` is therefore called with `entityId: null`, so the correct transaction never receives a status-update event after the deletion.

### Impact Explanation

An attacker who is the creator of any transaction (even a trivial one they created themselves) can:

1. Enumerate or guess approver IDs from other transactions (auto-increment integers).
2. Issue `DELETE /transactions/{own_tx_id}/approvers/{victim_approver_id}`.
3. The authorization check passes (they own `own_tx_id`); the victim approver is deleted.

Concrete consequences:
- **Approval bypass**: Remove required approvers from a victim transaction, reducing the effective threshold and allowing the transaction to proceed without the intended signatories.
- **Denial of service**: Permanently delete the entire approver subtree of any transaction, making it impossible to satisfy the approval requirement.
- **Cross-tenant integrity violation**: Any organization member who has ever created a transaction can corrupt the approval state of any other member's transaction.

### Likelihood Explanation

- Precondition: authenticated user with at least one created transaction — the lowest possible privilege level in the system.
- Approver IDs are sequential integers, trivially enumerable.
- No rate-limiting or anomaly detection is visible on this endpoint.
- The attack is a single HTTP DELETE request with no side-channel requirements.

### Recommendation

Inside `removeTransactionApprover` (or immediately after `getTransactionApproverById`), resolve the root node of the fetched approver and assert it belongs to the expected transaction, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // Mirror the guard in updateTransactionApprover
    const rootNode = await this.getRootNodeFromNode(approver.id);
    if (!rootNode || rootNode.transactionId !== transactionId)
        throw new UnauthorizedException('Approver does not belong to this transaction');

    await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher,
        [{ entityId: rootNode.transactionId }]);   // use root's transactionId, never null
}
```

Pass `transactionId` from the controller into the service call, and update the controller accordingly.

### Proof of Concept

**Setup:**
- User A creates transaction T1 (id=1).
- User B creates transaction T2 (id=2) with approver record id=7.

**Attack (as User A):**
```
DELETE /transactions/1/approvers/7
Authorization: Bearer <User A's JWT>
```

**Step-by-step:**
1. Controller calls `getCreatorsTransaction(1, userA)` → passes (A owns T1).
2. Controller calls `removeTransactionApprover(7)`.
3. Service fetches approver id=7 (belongs to T2, not T1) — no transaction check.
4. `removeNode(7)` soft-deletes approver 7 and its entire subtree.
5. `emitTransactionStatusUpdate` fires with `entityId: null` (approver 7 is a child node), so T2 receives no notification.
6. T2's approval requirement is silently corrupted; User B's transaction can no longer be approved as intended.

**Expected result:** HTTP 200, approver 7 deleted from T2 without User A ever owning T2.
