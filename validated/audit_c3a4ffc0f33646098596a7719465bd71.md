### Title
Cross-Transaction Approver Deletion via Decoupled Authorization Check

### Summary
The `removeTransactionApprover` controller verifies the caller is the creator of the transaction identified by the URL parameter `transactionId`, but then deletes the approver identified by the separate URL parameter `id` without verifying that approver belongs to that transaction. Any authenticated user who is the creator of any transaction can delete approvers from any other transaction, bypassing the intended approval requirements on transactions they do not own.

### Finding Description

**Root cause:** The authorization check and the destructive action operate on two different, unlinked objects.

In the controller:

```
DELETE /transactions/:transactionId/approvers/:id
``` [1](#0-0) 

```typescript
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
  return true;
}
```

`getCreatorsTransaction` only verifies the caller created the transaction at `transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches and deletes the approver by `id` alone, with no assertion that `approver.transactionId === transactionId`: [3](#0-2) 

The `removeNode` SQL cascade then soft-deletes the entire subtree rooted at that approver: [4](#0-3) 

**Contrast with `updateTransactionApprover`**, which correctly validates the root node's `transactionId` matches the URL parameter before acting: [5](#0-4) 

The `removeTransactionApprover` service path has no equivalent guard.

**Exploit flow:**
1. Attacker registers and creates any transaction `T_own` (becomes its creator).
2. Attacker observes or enumerates the integer approver `id` belonging to a victim transaction `T_victim` (IDs are sequential; the attacker may also be an observer/signer of `T_victim` and can retrieve approver IDs via `GET /transactions/:id/approvers`).
3. Attacker sends: `DELETE /transactions/{T_own}/approvers/{victim_approver_id}`.
4. `getCreatorsTransaction(T_own, attacker)` passes — attacker is creator of `T_own`.
5. `removeTransactionApprover(victim_approver_id)` executes with no ownership check, soft-deleting the approver (and its entire subtree) from `T_victim`.

### Impact Explanation

An attacker can silently strip all approvers from any transaction in the system. Once approvers are removed, the organizational approval gate for `T_victim` is gone. The transaction creator of `T_victim` loses the ability to enforce multi-party approval, and the transaction may proceed to execution without the required sign-offs. This constitutes unauthorized state mutation of another user's transaction and permanent destruction of approval requirements — a direct integrity failure in the trust model.

### Likelihood Explanation

The attacker precondition is minimal: any registered, verified user can create a transaction and become a creator. Approver IDs are sequential integers and trivially enumerable. No privileged access, leaked credentials, or cryptographic break is required. The attack is a single authenticated HTTP request.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the `transactionId` supplied in the URL:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve the root node to get the owning transactionId
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

This mirrors the guard already present in `updateTransactionApprover`. [5](#0-4) 

### Proof of Concept

**Setup:**
- User A creates transaction `T_victim` (id=10) and adds approver records for users B and C (approver ids=50, 51).
- User X creates any transaction `T_own` (id=99) — this is the only privilege required.

**Attack:**
```
DELETE /transactions/99/approvers/50   HTTP/1.1
Authorization: Bearer <User X token>
```

**Expected (correct) behavior:** 403 Unauthorized — approver 50 does not belong to transaction 99.

**Actual behavior:** 200 OK — approver 50 (and its subtree) is soft-deleted from `T_victim`, destroying User A's approval requirement without User A's knowledge or consent. [1](#0-0) [3](#0-2)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
