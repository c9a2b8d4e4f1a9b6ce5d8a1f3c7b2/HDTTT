Audit Report

## Title
Any Authenticated Verified User Can Enumerate All Organization Members' `mnemonicHash` and `index` via Unscoped `GET /user-keys`

## Summary
`UserKeysAllController.getUserKeys` (`GET /user-keys`) is accessible to any verified organization user and returns every `UserKey` row in the database — including `mnemonicHash` and `index` — with no per-user ownership filtering. This directly contradicts the access-control model enforced by `getUserKeysRestricted`, which hides those two fields from non-owners.

## Finding Description

**Root cause — missing user-scoping and field restriction in `getUserKeys`:**

`UserKeysAllController.getUserKeys` calls the service without passing the authenticated user:

```ts
// back-end/apps/api/src/user-keys/user-keys-all.controller.ts:37
return this.userKeysService.getUserKeys(paginationParams);
``` [1](#0-0) 

The service performs an unfiltered `findAndCount` with no `WHERE` clause and no `select` field restriction:

```ts
// back-end/apps/api/src/user-keys/user-keys.service.ts:145-148
const [items, total] = await this.repo.findAndCount({
  take: limit,
  skip: offset,
});
``` [2](#0-1) 

The serialization DTO exposes `mnemonicHash` and `index` unconditionally:

```ts
// back-end/apps/api/src/user-keys/dtos/user-key.dto.ts
@Expose() mnemonicHash?: string;
@Expose() index?: number;
``` [7](#0-6) 

Compare to `getUserKeysRestricted`, which explicitly gates those fields behind an ownership check:

```ts
// back-end/apps/api/src/user-keys/user-keys.service.ts:76-77
mnemonicHash: user.id === userId,
index: user.id === userId,
``` [4](#0-3) 

The controller is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard verified-user guards, not an `AdminGuard`: [5](#0-4) 

**Exploit path:**
1. Attacker registers or is invited as a normal organization user (no admin role required) and completes verification.
2. Attacker sends `GET /user-keys?page=1&size=100` with a valid JWT.
3. The response contains every `UserKey` row — including `publicKey`, `mnemonicHash`, and `index` for all users.
4. Attacker paginates through all pages to harvest the full key registry.

## Impact Explanation

- **`mnemonicHash`** — a hash of the user's BIP-39 mnemonic. The codebase explicitly restricts this to the key owner via `getUserKeysRestricted`. Exposure enables offline dictionary/brute-force attacks against the mnemonic, which if successful yields the private key.
- **`index`** — the BIP-44 derivation path index. Narrows the key-derivation search space for any brute-force attempt.
- **`publicKey`** — while public keys are not secret by nature, bulk enumeration enables targeted correlation attacks.

The design intent is unambiguous: `getUserKeysRestricted` hides `mnemonicHash` and `index` from non-owners. The `GET /user-keys` path bypasses this entirely, exposing the full record set to any authenticated peer.

## Likelihood Explanation

- **Attacker precondition:** valid JWT for any verified organization account (normal user, no admin role required).
- **Effort:** a single authenticated HTTP GET request with pagination.
- **Discoverability:** the endpoint is documented via Swagger (`@ApiTags('User Keys All')`), making it trivially discoverable. The front-end `getAllUserKeys` function also calls this endpoint directly, confirming it is reachable by all verified users. [6](#0-5) 

## Recommendation

Apply one of the following fixes:

1. **Add field restriction in `getUserKeys`**: Mirror the `select` logic from `getUserKeysRestricted` — set `mnemonicHash: false` and `index: false` since this endpoint is not scoped to a single owner. This is the minimal fix and aligns with the front-end's actual need (only `publicKey` and `userId` are used in `loadContacts`).

2. **Restrict to admin only**: Add `AdminGuard` to `UserKeysAllController` if the full record (including `mnemonicHash`/`index`) is ever legitimately needed server-side.

The front-end `loadContacts` only uses `publicKey` and `userId` from the response, so option 1 is the correct fix with no functional regression. [8](#0-7)
