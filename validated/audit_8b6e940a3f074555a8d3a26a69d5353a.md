### Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header Enables Unbounded Transaction and Group Creation

### Summary
The `IpThrottlerGuard` — the only rate-limiting mechanism applied globally to all API endpoints — uses the raw `x-forwarded-for` request header as its tracker key without validation or normalization. An authenticated user can rotate this header value on every request to bypass the IP-based throttle entirely. Combined with the fact that the `UserThrottlerModule`/`UserThrottlerGuard` are defined but never wired into the application, there is no effective per-user rate limit on the `POST /transactions` and `POST /transaction-groups` endpoints, allowing a malicious authenticated user to create unlimited transactions and groups.

### Finding Description

**Root cause — `IpThrottlerGuard` trusts attacker-controlled header:** [1](#0-0) 

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
```

The guard uses the full, unvalidated `x-forwarded-for` header string as the Redis throttle key. An attacker who sends `X-Forwarded-For: <rotating-value>` on each request gets a fresh throttle bucket every time, completely bypassing the limit. In a typical Kubernetes ingress setup (nginx/traefik), the ingress controller *appends* the real IP to an existing `x-forwarded-for` header rather than replacing it, so the full header value (and thus the Redis key) still changes with each rotation of the attacker-supplied prefix.

**This is the only global rate-limiting guard:** [2](#0-1) 

`IpThrottlerGuard` is registered as the sole `APP_GUARD` for rate limiting. The `UserThrottlerModule` and `UserThrottlerGuard` are defined but **never imported or applied anywhere**: [3](#0-2) [4](#0-3) 

`UserThrottlerModule` is absent from the `imports` array in `api.module.ts`, and `UserThrottlerGuard` is not applied to any controller or route.

**Unprotected resource-creation endpoints:**

`POST /transactions` (createTransaction): [5](#0-4) 

`POST /transaction-groups` (createTransactionGroup): [6](#0-5) 

Neither endpoint has a `UserThrottlerGuard` or any per-user creation limit.

**Exploit path:**
1. Attacker registers as a legitimate user (or is a malicious insider).
2. Attacker obtains a valid JWT and a registered user key.
3. Attacker sends `POST /transactions` in a loop, rotating the `X-Forwarded-For` header on each request (e.g., `1.0.0.1`, `1.0.0.2`, …).
4. Each request gets a fresh throttle bucket in Redis; the IP throttle never triggers.
5. Attacker creates thousands of transactions per second, each of which:
   - Persists a row in the `transaction` table.
   - Triggers `emitTransactionStatusUpdate` → NATS → Notifications service → WebSocket push to all connected clients.
   - Schedules a Redis reminder job if `reminderMillisecondsBefore` is set. [7](#0-6) 

### Impact Explanation

- **Database exhaustion:** Unlimited rows inserted into the `transaction` table, consuming PostgreSQL storage and degrading query performance for all users.
- **Notification spam / WebSocket flooding:** Every created transaction emits a NATS event that the Notifications service fans out to all connected WebSocket clients, causing client-side flooding and potential Notifications service overload.
- **Redis job exhaustion:** If `reminderMillisecondsBefore` is supplied, a Redis scheduler job is created per transaction, exhausting Redis memory.
- **Service degradation for all users:** Legitimate users experience slow queries, missed notifications, and UI unresponsiveness.

### Likelihood Explanation

- Precondition: valid authenticated account — achievable by any registered user.
- Exploit technique: setting an HTTP header — trivially scriptable with `curl` or any HTTP client.
- No privileged access, no cryptographic break, no physical access required.
- The `x-forwarded-for` bypass is a well-known technique; the code explicitly prefers that header over `req.ip`.

### Recommendation

1. **Fix `IpThrottlerGuard`** to extract only the rightmost (trusted) IP from `x-forwarded-for`, or configure the ingress controller to strip and re-set the header so the application always sees the real client IP via `req.ip`.
2. **Wire `UserThrottlerModule` and `UserThrottlerGuard`** into the application. Import `UserThrottlerModule` in `api.module.ts` and apply `@UseGuards(UserThrottlerGuard)` to `POST /transactions` and `POST /transaction-groups` (or register it as an `APP_GUARD` scoped to authenticated routes).
3. **Add a per-user transaction count cap** (e.g., max N active/pending transactions per user) enforced at the service layer as a defense-in-depth measure independent of throttling infrastructure.

### Proof of Concept

```bash
JWT="<valid_jwt>"
CREATOR_KEY_ID=1
TX_BYTES="<valid_signed_transaction_bytes_base64>"
SIG="<valid_signature_base64>"

for i in $(seq 1 10000); do
  curl -s -X POST https://<api-host>/transactions \
    -H "Authorization: Bearer $JWT" \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$((i % 255 + 1))" \
    -d "{
      \"creatorKeyId\": $CREATOR_KEY_ID,
      \"transactionBytes\": \"$TX_BYTES\",
      \"signature\": \"$SIG\",
      \"mirrorNetwork\": \"testnet\",
      \"name\": \"spam-$i\",
      \"isManual\": false
    }" &
done
```

Each request carries a different `X-Forwarded-For` value, giving it a fresh Redis throttle bucket. The IP throttle defined in `IpThrottlerModule` never fires. The result is thousands of transaction rows inserted, thousands of NATS events emitted, and WebSocket floods delivered to all connected clients.

```typescript
/* eslint-disable @typescript-eslint/no-explicit-any */
import { HttpException, HttpStatus, Injectable } from '@nestjs/common';
import { ThrottlerGuard } from '@nestjs/throttler';

@Injectable()
export class UserThrottlerGuard extends ThrottlerGuard {
  protected getTracker(req: Record<string, any>): Promise<string> {
    const user = req.user;
    if (!user) {
      throw new HttpException('No user connected.', HttpStatus.BAD_REQUEST);
    }
    // console.log('user id', user.id);
    return user.id;
  }
```
