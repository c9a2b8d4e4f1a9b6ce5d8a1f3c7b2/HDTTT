Audit Report

## Title
Unbounded Array Input in Bulk Signature Upload Endpoints Enables Server-Side Resource Exhaustion

## Summary
`POST /transactions/signers` and `POST /transactions/signatures/import` accept an unbounded array of `UploadSignatureMapDto` objects. Each element triggers CPU-intensive SDK transaction deserialization (`SDKTransaction.fromBytes()`), cryptographic signature validation, and database queries. The `validateAndProcessSignatures` method fans all items out concurrently via `Promise.all()`. A single authenticated user can exhaust server CPU, memory, and the database connection pool with one crafted request.

## Finding Description

**Root cause â€” no array size limit in `transformAndValidateDto`:**

`back-end/libs/common/src/dtos/index.ts` lines 13â€“20 maps over the caller-supplied array without any length check: [1](#0-0) 

**Entry point 1 â€” `POST /transactions/signers`:**

`SignersController.uploadSignatureMap` accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no size guard and passes the full array directly to `transformAndValidateDto`: [2](#0-1) 

**Entry point 2 â€” `POST /transactions/signatures/import`:**

`TransactionsController.importSignatures` has the identical pattern: [3](#0-2) 

**The DTO carries no `@ArrayMaxSize` decorator:**

`UploadSignatureMapDto` only validates `id` and `signatureMap` field types â€” no constraint on how many items can appear in the outer array: [4](#0-3) 

**Concurrent fan-out in `validateAndProcessSignatures`:**

`Promise.all()` is used to process every item simultaneously. Each item calls `validateTransactionStatus` (which calls `SDKTransaction.fromBytes()` at line 209) and `processTransactionSignatures` (which calls `SDKTransaction.fromBytes()` again at line 223): [5](#0-4) [6](#0-5) [7](#0-6) 

**`importSignatures` path (sequential but still unbounded):**

`TransactionsService.importSignatures` iterates over the unbounded array with a `for` loop, calling `SDKTransaction.fromBytes()` and `validateSignature` per item. Processing is sequential rather than concurrent, but the array is still unbounded: [8](#0-7) 

**Partial mitigation present but insufficient:**

A 2 MB body size limit is configured globally (`app.use(json({ limit: '2mb' }))`), which bounds the maximum payload: [9](#0-8) 

However, a minimal `UploadSignatureMapDto` item (small `id` + compact `signatureMap`) can be as small as ~100â€“300 bytes in JSON, meaning a 2 MB payload can still carry thousands of items â€” each triggering two `SDKTransaction.fromBytes()` calls and multiple DB round-trips.

**No throttler guards on these endpoints:**

The throttler modules (`UserThrottlerModule`, `IpThrottlerModule`) exist but are not applied to `SignersController` or `TransactionsController`. Both controllers are protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`: [10](#0-9) [11](#0-10) 

## Impact Explanation
A single authenticated user (no admin role required) can render the API service unavailable for all organization members by submitting one oversized request. The Node.js event loop is saturated by concurrent protobuf deserialization and cryptographic work; the PostgreSQL connection pool is exhausted by concurrent queries. Transaction signing, execution, and all other API operations are blocked for the duration. If the process OOMs, the service crashes entirely until restarted.

## Likelihood Explanation
Any verified user account is sufficient â€” the endpoints are protected only by `JwtAuthGuard` and `VerifiedUserGuard`. No admin access or special knowledge beyond the public API schema is required. The attack is a single HTTP POST with a large JSON array, reproducible with `curl` or any HTTP client. The 2 MB body limit is the only technical barrier, and it does not prevent the attack â€” it only bounds the maximum payload size per request.

## Recommendation
1. **Add an array size cap in `transformAndValidateDto`** or add a `@ArrayMaxSize(N)` decorator to `UploadSignatureMapDto` (e.g., limit to 50â€“100 items per request).
2. **Apply `UserThrottlerGuard` or `IpThrottlerGuard`** to `SignersController` and `TransactionsController` so that repeated bulk requests from the same user or IP are rate-limited.
3. **Consider processing items in bounded batches** (e.g., chunks of 10â€“20) rather than fanning out all items concurrently via `Promise.all()` in `validateAndProcessSignatures`.
4. **Consider reducing the body size limit** for these specific endpoints if large payloads are not a legitimate use case.

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
