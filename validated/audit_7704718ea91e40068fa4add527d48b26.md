### Title
`TransactionsService#importSignatures` Fails to Update Transaction Status and Signer Records After Signature Import

### Summary
`importSignatures` in `back-end/apps/api/src/transactions/transactions.service.ts` updates `transactionBytes` in the database after merging externally-supplied signatures, but never calls `processTransactionStatus` to advance the transaction's `status` field, and never inserts `TransactionSigner` records. This is a direct analog to the reported state-update-omission class: computed state is used locally but not written back to persistent storage. The result is that a transaction whose signature threshold is satisfied by an `importSignatures` call remains permanently stuck in `WAITING_FOR_SIGNATURES` from the DB's perspective until an unrelated cron job happens to re-evaluate it, and the signer ledger (`TransactionSigner`) is never updated at all.

### Finding Description

**Vulnerability class:** State-update omission (accounting / state-transition).

**Root cause — `importSignatures` (lines 493–626):** [1](#0-0) 

After adding each public key's signature to the in-memory SDK transaction and serialising the result back to `transactionBytes`, the function writes only the raw bytes to the DB: [2](#0-1) 

It then emits a notification event and returns: [3](#0-2) 

Two state mutations that **must** follow a successful signature merge are entirely absent:

1. **`processTransactionStatus` is never called.** This is the function that evaluates whether the collected signatures satisfy the required key structure and, if so, writes `status = WAITING_FOR_EXECUTION` to the DB. Every other code path that adds signatures calls it: [4](#0-3) [5](#0-4) 

2. **`TransactionSigner` rows are never inserted.** `uploadSignatureMaps` inserts a `TransactionSigner` record for every user key that contributed a new signature: [6](#0-5) 

`importSignatures` has no equivalent step.

**Exploit path:**
1. Authenticated user calls `POST /transactions/importSignatures` with a signature map that satisfies the transaction's required key structure.
2. `transactionBytes` is updated in the DB with the completed signatures.
3. `processTransactionStatus` is never called → `status` remains `WAITING_FOR_SIGNATURES`.
4. No `TransactionSigner` rows are inserted → the signer ledger is permanently inconsistent.
5. The chain service only submits transactions in `WAITING_FOR_EXECUTION`; the transaction is not submitted.
6. Status correction only occurs when a background cron job (`handleTransactionsBetweenTenMinutesAndOneHour`, `handleTransactionsBetweenOneDayAndOneWeek`, etc.) happens to re-evaluate the transaction — up to 10 minutes or 24 hours later depending on `validStart`. [7](#0-6) 

### Impact Explanation

- **Delayed / missed execution:** A transaction whose threshold is met via `importSignatures` is not submitted to the Hedera network until the next applicable cron window. For transactions with `validStart` more than one hour away the cron runs every 10 minutes; for those more than one day away it runs once per day. If the `validStart` window (180-second validity period) closes before the cron fires, the transaction expires and is permanently unexecutable.
- **Permanent signer-ledger corruption:** `TransactionSigner` rows are never written. Any feature that queries signers (audit trail, UI display of "who has signed", approver-threshold re-evaluation) will show incorrect data indefinitely — the cron does not repair this table.
- **Status inconsistency visible to all participants:** Observers and approvers see `WAITING_FOR_SIGNATURES` even though the transaction is fully signed, causing confusion and potential duplicate signing attempts.

### Likelihood Explanation

`importSignatures` is a documented, authenticated endpoint reachable by any organisation member. It is the intended path for importing offline/external signatures. Any user who imports a completing signature set triggers the bug automatically — no special privileges or timing attack required. The front-end exposes this flow directly: [8](#0-7) 

### Recommendation

After the batch `transactionBytes` update succeeds, call `processTransactionStatus` for every successfully updated transaction (mirroring `SignersService#updateStatusesAndNotify`) and insert the corresponding `TransactionSigner` records for each public key that was newly added. Wrap both operations in the same DB transaction as the bytes update to ensure atomicity.

```typescript
// After the bulk bytes update loop in importSignatures:
const transactionsToProcess = successfulUpdates.map(u => transactionMap.get(u.id));
const statusMap = await processTransactionStatus(
  this.txRepo,
  this.transactionSignatureService,
  transactionsToProcess,
);
// emit status-update or plain-update events based on statusMap, as done in SignersService
```

### Proof of Concept

1. Create an organisation transaction requiring a 1-of-1 key signature; note its `id` and `status = WAITING_FOR_SIGNATURES`.
2. Produce a valid offline signature for the required key.
3. Call `POST /api/transactions/importSignatures` with the signature map.
4. Immediately query `GET /api/transactions/{id}` — `status` is still `WAITING_FOR_SIGNATURES`.
5. Query `SELECT * FROM transaction_signer WHERE "transactionId" = {id}` — zero rows.
6. Wait for the applicable cron window to fire; only then does `status` advance to `WAITING_FOR_EXECUTION`.
7. If `validStart` expires before the cron fires, the transaction is permanently expired despite having a complete signature set embedded in `transactionBytes`.
