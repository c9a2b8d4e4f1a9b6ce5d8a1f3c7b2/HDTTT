Audit Report

## Title
Manual Transaction Permanently Locked When Creator Account Is Removed — Overly Strict Access Control on `executeTransaction`, `archiveTransaction`, and `cancelTransaction`

## Summary
`getTransactionForCreator` enforces a hard creator-only identity check with no admin bypass. All three transaction lifecycle endpoints (`execute`, `archive`, `cancel`) call this function unconditionally. When the creator's account is soft-deleted via the normal admin `DELETE /users/:id` flow, no authenticated user can ever satisfy the identity check, permanently locking any manual transaction that user created.

## Finding Description

**Root cause — `getTransactionForCreator` has no admin bypass:**

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts
async getTransactionForCreator(id: number, user: User) {
  const transaction = await this.getTransactionById(id);
  if (!transaction) throw new BadRequestException(ErrorCodes.TNF);
  if (transaction.creatorKey?.userId !== user?.id)
    throw new UnauthorizedException('Only the creator has access to this transaction');
  return transaction;
}
``` [1](#0-0) 

This function is called unconditionally by all three lifecycle operations:

- `executeTransaction` at line 737
- `archiveTransaction` at line 709
- `cancelTransactionWithOutcome` (called by `cancelTransaction`) at line 663 [2](#0-1) [3](#0-2) [4](#0-3) 

**The controller exposes these as plain `PATCH` endpoints with no `AdminGuard`:**

```typescript

**The scheduler explicitly skips manual transactions:**

```typescript
addExecutionTimeout(transaction: Transaction) {
  ...
  if (transaction.isManual) return;   // manual transactions are intentionally excluded
  ...

``` [7](#0-6) 

**User removal soft-deletes the user and all their keys:**

```typescript
async removeUser(id: number): Promise<boolean> {
  await this.repo.manager.softDelete(UserKey, { userId: id });
  await this.repo.softRemove(user);
  return true;
}
``` [8](#0-7) 

After soft-deletion, the user can no longer authenticate. The `creatorKey.userId` stored on the transaction still references the deleted user's ID, but no active session can ever present that ID. The identity check in `getTransactionForCreator` therefore throws `UnauthorizedException` for every caller — including admins — with no override path.

## Impact Explanation

Any manual transaction (`isManual: true`) created by a subsequently removed user is permanently stuck in `WAITING_FOR_EXECUTION` (or `WAITING_FOR_SIGNATURES`) state:

- `PATCH /transactions/execute/:id` — throws 401 for all callers
- `PATCH /transactions/archive/:id` — throws 401 for all callers
- `PATCH /transactions/cancel/:id` — throws 401 for all callers
- `DELETE /transactions/:id` — also calls `getTransactionForCreator`, throws 401

The chain service never touches manual transactions. The signed transaction bytes stored in the database can never be submitted to Hedera, and the record cannot be resolved. All signing effort by other organization members is wasted. For organizations using manual transactions for high-value operations (treasury transfers, account updates), this is a permanent operational lock with no in-system recovery path.

**Severity: Medium** — no on-chain funds are directly locked (the transaction was never submitted), but the signed transaction record is irrecoverable and the organization's workflow is permanently disrupted.

## Likelihood Explanation

The trigger is a routine, expected admin operation: removing a departed employee or a compromised account. No attacker capability is required. Any organization that:
- uses manual transactions, **and**
- ever removes a user who created one

will silently hit this. The admin has no warning that removing the user will permanently lock their pending manual transactions. The `removeUser` service performs no check for pending transactions owned by the user before deletion. [9](#0-8) 

## Recommendation

1. **Add an admin bypass to `getTransactionForCreator`** (or create a separate `getTransactionForAdmin` variant) that allows users with `admin: true` to access any transaction's lifecycle operations.
2. **Alternatively**, add dedicated admin-guarded endpoints (`PATCH /transactions/admin/cancel/:id`, etc.) that bypass the creator check.
3. **Optionally**, add a pre-deletion check in `removeUser` that warns or blocks deletion if the user has pending manual transactions in `WAITING_FOR_EXECUTION` or `WAITING_FOR_SIGNATURES` state.

## Proof of Concept

1. Admin creates User A (non-admin). User A creates a manual transaction (`isManual: true`) via `POST /transactions`.
2. Other organization members sign it; the transaction reaches `WAITING_FOR_EXECUTION`.
3. Admin calls `DELETE /users/:id` for User A — succeeds normally.
4. Any user (including admins) calls `PATCH /transactions/execute/:id`:
   - `getTransactionForCreator` loads the transaction; `transaction.creatorKey.userId` = User A's (now-deleted) ID.
   - No active session can present User A's ID.
   - Result: `401 Unauthorized — "Only the creator has access to this transaction"` for every caller.
5. Same result for `/cancel/:id` and `/archive/:id`.
6. The scheduler never picks up the transaction because `addExecutionTimeout` returns early for `isManual === true`.
7. The transaction record is permanently unresolvable. [1](#0-0) [7](#0-6)
