### Title
`IpThrottlerGuard` Uses Attacker-Controlled `x-forwarded-for` Header as Rate-Limit Key, Enabling Complete Bypass of Global Rate Limiting

### Summary
`IpThrottlerGuard.getTracker()` resolves the rate-limit key by reading `req.headers['x-forwarded-for']` before falling back to `req.ip`. Because `x-forwarded-for` is a plain HTTP header that any client can set to an arbitrary value, an attacker can rotate through fake IP strings on every request, making the global IP-based rate limit completely ineffective. This is the direct analog of the `flashProof`/`tx.origin` pattern: a spoofable shared identifier is used instead of a server-authoritative one.

### Finding Description

**Root cause — `IpThrottlerGuard`:**

```typescript
// back-end/apps/api/src/guards/ip-throttler.guard.ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;
}
```

`req.headers['x-forwarded-for']` is the raw HTTP header value. Any HTTP client can set it to any string. The guard prefers this header over `req.ip`, which is the value Express derives from the actual TCP connection (and which respects the `trust proxy` setting). Because the attacker-supplied header wins, the rate-limit bucket key is fully attacker-controlled.

**Global application — `api.module.ts`:**

```typescript
providers: [
  { provide: APP_GUARD, useClass: IpThrottlerGuard },
  ...
]
```

`IpThrottlerGuard` is registered as a global `APP_GUARD`, meaning it is the primary rate-limiting layer for every route in the API service.

**Exploit path:**

1. Attacker targets `POST /auth/login` (or `POST /auth/verify-reset` for OTP brute-force).
2. On each request the attacker sets `X-Forwarded-For: <random-or-incrementing-value>`.
3. Each request lands in a fresh, empty throttle bucket in Redis.
4. The `GLOBAL_MINUTE_LIMIT` and `GLOBAL_SECOND_LIMIT` counters are never exhausted for any single key.
5. The attacker can send unlimited requests with no throttling applied.

### Impact Explanation

- **Brute-force credential attacks**: `POST /auth/login` has no per-user lockout; the only global brake is `IpThrottlerGuard`. With the guard bypassed, an attacker can enumerate passwords for any known email address without restriction.
- **OTP brute-force**: `POST /auth/verify-reset` is guarded only by `JwtBlackListOtpGuard` and `OtpJwtAuthGuard`. The global IP throttle is the only rate control. A TOTP window is short but finite; with unlimited requests the attacker can exhaust the OTP space within the window.
- **Resource exhaustion**: Unlimited unauthenticated requests can be directed at any endpoint, degrading service for legitimate users.

Severity: **High** — authentication brute-force is directly enabled with no privilege required.

### Likelihood Explanation

- Requires only the ability to send HTTP requests with a custom header — trivially achievable with `curl`, any HTTP library, or a browser's `fetch` API.
- No credentials, tokens, or insider access needed.
- The attack is fully automatable and silent (no anomalous TCP behaviour).
- Deployments behind a reverse proxy that strips `x-forwarded-for` would be partially mitigated, but the code itself provides no such guarantee and the guard's own test suite explicitly validates the spoofable-header path.

### Recommendation

Replace the user-controlled header lookup with the server-authoritative `req.ip` only, and configure Express `trust proxy` correctly for the deployment topology:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip; // Express resolves this via trust proxy setting
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

If the service sits behind a trusted reverse proxy, set `app.set('trust proxy', 1)` (or the appropriate hop count) in the NestJS bootstrap so that `req.ip` is correctly populated from the last trusted proxy's forwarded header — without exposing the raw header to client manipulation.

### Proof of Concept

```bash
# Bypass global rate limit by rotating X-Forwarded-For on every request
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<api-host>/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$i" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
```

Every request receives a fresh throttle bucket keyed to `10.0.0.<i>`. None of the `GLOBAL_MINUTE_LIMIT` or `GLOBAL_SECOND_LIMIT` counters are ever reached, and the attacker can brute-force the login endpoint indefinitely.
