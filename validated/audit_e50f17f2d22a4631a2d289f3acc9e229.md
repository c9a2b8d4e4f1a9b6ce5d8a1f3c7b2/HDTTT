## Audit Report

## Title
`removeTransactionApprover` Does Not Update Parent Threshold When a Child Approver Is Deleted, Permanently Blocking Transaction Approval

## Summary
When a child node in a threshold-based approver tree is deleted via `removeTransactionApprover`, the parent node's `threshold` value is never decremented. If the remaining child count falls below the stored threshold, the approval condition becomes mathematically impossible to satisfy, permanently locking the transaction. A secondary bug causes the post-deletion status-update event to be emitted with `entityId: null` for any child node deletion, meaning the transaction's approval state is never re-evaluated even if the threshold were corrected.

## Finding Description

**Root cause â€” `removeNode` never touches the parent's `threshold`:**

`removeTransactionApprover` fetches the approver by id and delegates to `removeNode`: [1](#0-0) 

`removeNode` executes a recursive SQL soft-delete that marks the target node and all its descendants as deleted, but never reads or writes the parent row: [2](#0-1) 

The `WHERE` clause only matches `transaction_approver."id" = $1` (the target) and `approversToDelete."id" = transaction_approver."listId"` (descendants). The parent row â€” whose `id` is the deleted child's `listId` â€” is never updated.

**Contrast with `updateTransactionApprover` (the correct path):**

When a child is *detached* via `PATCH` (`dto.listId = null`), the code explicitly fetches the parent and adjusts its threshold: [3](#0-2) 

This guard is entirely absent in the `DELETE` path.

**Secondary compounding bug â€” status-update event emitted with `null` entity ID:**

For child nodes, `transactionId` is stored as `null` by design (only root nodes carry the `transactionId` FK): [4](#0-3) 

After deletion, the notification is fired as: [5](#0-4) 

`approver.transactionId` is `null` for any child node, so the status-update event is emitted with `entityId: null` â€” the transaction's approval state is never re-evaluated after the deletion.

**Approval evaluation uses the stale `threshold`:**

The `isApproved` helper reads the stored `threshold` field directly: [6](#0-5) 

If `threshold=2` but only 1 child remains, `approvals.length >= 2` can never be true â†’ the tree node is permanently unapproved.

**Controller performs no check that the supplied `id` is a root node:** [7](#0-6) 

The controller only verifies the caller is the transaction creator; it does not restrict deletion to root nodes, allowing any child approver to be targeted directly.

## Impact Explanation
A transaction creator (normal authenticated user, no admin role required) can permanently freeze any of their own transactions in a non-executable state. Once the parent's `threshold` exceeds the remaining child count, no combination of approvals can satisfy it, and there is no self-healing mechanism. The transaction will eventually expire, causing permanent loss of the intended on-chain Hedera operation (account updates, token transfers, file changes, etc.). In an organization context, other participants who have already signed or are waiting to sign are also affected.

## Likelihood Explanation
The trigger is a normal product workflow: a creator removes one approver from a threshold group intending to replace them. The `DELETE` endpoint is documented and reachable by any authenticated transaction creator. No special timing, race condition, or privileged access is required. The bug fires deterministically on every deletion of a child approver when `threshold > (children - 1)`.

## Recommendation
In `removeTransactionApprover` (or inside `removeNode`), before soft-deleting the target node, fetch the parent row (the row whose `id` equals the target's `listId`). After deletion, apply the same logic already present in `updateTransactionApprover`:

- If `newParentChildCount === 0`, soft-delete the parent as well.
- Else if `newParentChildCount < parent.threshold`, update `parent.threshold = newParentChildCount`.

Additionally, resolve the secondary bug by walking up to the root node (via `getRootNodeFromNode`) before emitting the status-update event, so `entityId` is always the root's `transactionId` rather than the child's `null`.

## Proof of Concept

1. Creator sets up a threshold approver tree:
   - Parent node: `threshold=2`, children = [User A (id=10), User B (id=11)]
2. Creator calls `DELETE /transactions/1/approvers/10` (User A's child approver row).
3. `removeNode(10)` soft-deletes row 10 and its descendants. Parent row retains `threshold=2`, `approvers=[User B]`.
4. `emitTransactionStatusUpdate` fires with `entityId: null` (child node has no `transactionId`) â€” no re-evaluation occurs.
5. User B approves â†’ `isApproved` evaluates: `approvals.length (1) >= threshold (2)` â†’ `false` â†’ tree node remains unapproved.
6. Transaction is permanently stuck in `WAITING_FOR_SIGNATURES`; it can never reach `WAITING_FOR_EXECUTION` or `EXECUTED` and will expire.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
