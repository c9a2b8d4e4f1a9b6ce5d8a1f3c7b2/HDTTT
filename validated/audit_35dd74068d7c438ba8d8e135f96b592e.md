Audit Report

## Title
Cross-Transaction Approver Deletion: Creator of Any Transaction Can Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of the transaction identified by `:transactionId`, but never verifies that the approver identified by `:id` actually belongs to that same transaction. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause:** The controller's `removeTransactionApprover` handler performs two independent, uncorrelated checks:

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller is the creator of the transaction referenced by `:transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by `id` and soft-deletes it with no check that `approver.transactionId` matches the URL's `:transactionId`: [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly performs the cross-check by walking up to the root node and comparing its `transactionId` against the URL parameter: [4](#0-3) 

The `removeTransactionApprover` service method accepts only an `id` parameter, making it structurally impossible to perform the binding check inside the service without a refactor.

## Impact Explanation

An attacker can silently remove required approvers from any transaction in the system. In the multi-signature organization workflow, approvers are the gatekeepers that must sign off before a transaction is submitted to the Hedera network. Removing an approver can:

- Reduce the required approval threshold, allowing a transaction to proceed with fewer authorizations than intended.
- Completely remove all approvers from a transaction, bypassing the approval gate entirely.
- Disrupt or permanently corrupt the approval state of any in-flight transaction.

This is an unauthorized state change with direct impact on transaction integrity and the multi-signature trust model.

## Likelihood Explanation

The precondition is minimal: the attacker only needs to be a registered, verified user and the creator of any one transaction (a normal product workflow). Approver IDs are sequential database integers and are returned in API responses (e.g., via `GET /transactions/:transactionId/approvers`), making enumeration trivial. No privileged access, leaked credentials, or cryptographic break is required.

## Recommendation

Mirror the pattern used in `updateTransactionApprover`. Before deleting, resolve the root node of the target approver and verify its `transactionId` matches the URL's `:transactionId`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Walk up to the root node and verify it belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Update the controller to pass `transactionId` to the service method, and update the service signature accordingly.

## Proof of Concept

1. Attacker registers as a normal user and creates Transaction A. They are now the creator of Transaction A (`A_id`).
2. Attacker calls `GET /transactions/{any_id}/approvers` (or observes API responses) to enumerate a victim approver ID (`victim_approver_id`) belonging to Transaction B, owned by another user.
3. Attacker sends:
   ```
   DELETE /transactions/{A_id}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction(A_id, attacker)` passes — attacker is creator of A.
5. `removeTransactionApprover(victim_approver_id)` executes with no further check — the victim's approver is soft-deleted from Transaction B.
6. Transaction B's approval state is now corrupted without the owner's knowledge.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
