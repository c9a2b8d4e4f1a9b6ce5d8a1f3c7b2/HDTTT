### Title
Approver Rejection Decision Is Irreversible — No Mechanism to Change Vote

### Summary
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction` function permanently locks an approver's decision once any choice (approve or reject) is submitted. An approver who rejects a transaction cannot reverse that decision and approve it, even while the transaction is still in an active, modifiable state. This is the direct analog of the external report's irreversible opt-out pattern.

### Finding Description
The guard at line 563 of `approvers.service.ts` is:

```typescript
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

The gate condition is `a.signature` — not `a.approved`. A `signature` is written to the `transaction_approver` row for **both** approve (`approved: true`) and reject (`approved: false`) paths, as seen at lines 599–610:

```typescript
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({
    userKeyId: dto.userKeyId,
    signature: dto.signature,   // written for both approve and reject
    approved: dto.approved,
  })
  .whereInIds(userApprovers.map(a => a.id))
  .execute();
```

Once the signature column is populated, the guard at line 563 fires unconditionally on any subsequent call, regardless of whether the approver wants to change from `false` → `true` or `true` → `false`. There is no separate code path that allows a reversal.

The only administrative escape hatch is for the transaction **creator** to call `updateTransactionApprover`, which resets `approved` to `undefined` when the `userId` is changed. This is not accessible to the approver themselves and requires creator intervention. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
An approver who submits `approved: false` (rejection) cannot subsequently submit `approved: true`, even if the transaction is still in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status. If the rejection threshold is met, the transaction moves to `REJECTED` — a terminal state — and can never be executed. The creator must cancel and recreate the entire transaction, losing all previously collected signatures. In a multi-signature workflow with many signers, this is a significant operational disruption. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The UI presents both "Approve transaction" and "Reject transaction" buttons with a confirmation dialog, but user error is realistic — an approver may click the wrong button, or may change their mind after a team discussion. The `approveTransaction` endpoint is directly reachable by any authenticated user who is listed as an approver. No elevated privilege is required to trigger the lock. The test at line 1301 of `approvers.service.spec.ts` explicitly confirms the `TAP` error is thrown on a second submission, demonstrating the behavior is intentional but without consideration for the reversal use case. [6](#0-5) 

### Recommendation
- **Short term:** In `approveTransaction`, change the guard from `a.signature` to a check that only blocks re-submission of the **same** choice, or remove the guard entirely and allow overwriting. For example:
  ```typescript
  // Allow re-vote only if the new choice differs from the stored one
  if (userApprovers.every(a => a.signature && a.approved === dto.approved))
    throw new BadRequestException(ErrorCodes.TAP);
  ```
- **Long term:** Add a dedicated `changeApproverVote` endpoint or extend `approveTransaction` to explicitly support vote reversal while the transaction is still in an active status (`WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION`). Add unit tests covering the reversal scenario (reject → approve and approve → reject). [7](#0-6) 

### Proof of Concept
1. Alice is listed as an approver on transaction `T` (status: `WAITING_FOR_SIGNATURES`).
2. Alice calls `POST /transactions/T/approvers/approve` with `{ approved: false, userKeyId: K, signature: S }`.
3. The `transaction_approver` row for Alice is updated: `signature = S`, `approved = false`.
4. Alice realizes she made a mistake and calls `POST /transactions/T/approvers/approve` again with `{ approved: true, userKeyId: K, signature: S }`.
5. The guard `userApprovers.every(a => a.signature)` evaluates to `true` (Alice's row now has a signature).
6. The server throws `BadRequestException(ErrorCodes.TAP)` — Alice's rejection is permanent.
7. If Alice's rejection satisfies the threshold, the transaction transitions to `REJECTED` and cannot be executed. [1](#0-0) [2](#0-1)
