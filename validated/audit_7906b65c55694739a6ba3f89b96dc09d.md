Audit Report

## Title
Duplicate `TransactionSigner` Records Insertable via TOCTOU Race on Missing Unique Constraint

## Summary
The `TransactionSigner` entity lacks a database-level unique constraint on `(transactionId, userKeyId)`. The application-level deduplication check in `loadTransactionData()` is performed outside the database transaction that executes the insert in `bulkInsertSigners()`. Two concurrent signature-upload requests from the same user for the same transaction will both pass the deduplication check and both successfully insert duplicate signer rows, corrupting the signer state for that transaction.

## Finding Description

**Root cause — missing unique constraint:**

`TransactionSigner` declares only a plain (non-unique) index on `(transactionId, userKeyId)`: [1](#0-0) 

The migration `IndexFixesForCaching1768289349311` confirms this — it creates a plain `CREATE INDEX` (not `CREATE UNIQUE INDEX`) for `transaction_signer`: [2](#0-1) 

By contrast, `TransactionObserver` correctly declares a **unique** index on the same kind of pair, both in the entity: [3](#0-2) 

and in the same migration: [4](#0-3) 

**TOCTOU window — check and insert are not atomic:**

Step 1 — existing signers are loaded **outside** any database transaction in `loadTransactionData()`: [5](#0-4) 

Step 2 — the deduplication check runs against that stale snapshot in `processTransactionSignatures()`: [6](#0-5) 

Step 3 — the insert runs inside a separate DB transaction with no `ON CONFLICT` clause in `bulkInsertSigners()`: [7](#0-6) 

## Impact Explanation
Duplicate signer rows corrupt the integrity of the signature-counting logic used by `processTransactionStatus` and `TransactionSignatureService`. If that logic counts all rows rather than distinct `userKeyId` values, a single user's key can be counted multiple times toward the required signature threshold, causing a transaction to advance to `WAITING_FOR_EXECUTION` prematurely — before all legitimately required signers have signed. This is an unauthorized state transition that bypasses the multi-signature security model of the platform. Even if the status logic deduplicates, the duplicate rows permanently corrupt the audit trail (who signed, when) and can cause incorrect UI display and notification behavior.

## Likelihood Explanation
The attacker is any authenticated user who is a valid signer on a transaction. No privileged access is required. The race window is the round-trip time of a single HTTP request — easily exploitable by sending two requests in parallel from a script. The missing unique constraint means the race always wins; there is no database-level backstop.

## Recommendation
1. **Add a unique constraint at the database level** on `transaction_signer(transactionId, userKeyId)` by changing the entity decorator from `@Index(['transactionId', 'userKeyId'])` to `@Index(['transactionId', 'userKeyId'], { unique: true })` and generating a corresponding migration.
2. **Add `ON CONFLICT DO NOTHING`** (or equivalent) to `bulkInsertSigners()` so that even if the race is won, the second insert is silently discarded rather than creating a duplicate row.
3. Optionally, wrap `loadTransactionData()` and the subsequent insert inside a single serializable database transaction with a `SELECT ... FOR UPDATE` on the relevant signer rows to eliminate the TOCTOU window entirely.

## Proof of Concept
```
# Terminal 1
curl -X POST https://host/transactions/42/signers \
  -H "Authorization: Bearer <valid_token>" \
  -d '{"signatureMap": <valid_map>}' &

# Terminal 2 (fired simultaneously)
curl -X POST https://host/transactions/42/signers \
  -H "Authorization: Bearer <valid_token>" \
  -d '{"signatureMap": <valid_map>}' &

wait

# Result: two rows in transaction_signer with identical (transactionId=42, userKeyId=<id>)
psql -c "SELECT * FROM transaction_signer WHERE \"transactionId\"=42;"
# Shows two rows with the same userKeyId, different primary key ids
```
Both requests execute `loadTransactionData()` before either commits, both see an empty `existingSignerIds` set, both pass the `!existingSignerIds.has(userKey.id)` check, and both call `bulkInsertSigners()` successfully because there is no unique constraint to reject the second insert.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
