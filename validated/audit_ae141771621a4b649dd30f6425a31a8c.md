### Title
Authenticated User Can Register Any Public Key Without Proof-of-Ownership, Enabling Key Squatting and Transaction Workflow DoS

### Summary
The `uploadKey` endpoint accepts any syntactically valid Hedera public key from any authenticated, verified organization member without requiring a cryptographic proof that the submitter controls the corresponding private key. An attacker who is a verified organization member can preemptively register another user's known public key under their own account. Once squatted, the legitimate owner receives `ErrorCodes.PU` ("Public key in use") on every registration attempt and is permanently locked out of participating in the organization's signing and approval workflows under their actual key.

### Finding Description

**Root cause — `uploadKey` in `user-keys.service.ts`:** [1](#0-0) 

The only checks performed are:
1. The submitted string must parse as a valid Hedera public key (`@IsHederaPublicKey()` in the DTO).
2. The key must not already be registered by a *different* user.

There is no challenge-response or signature proof that the submitter holds the private key. The controller wires this directly: [2](#0-1) 

The `userId` path parameter is completely ignored — the key is always registered for the JWT-authenticated caller, confirmed by the e2e test: [3](#0-2) 

**Exploit path:**

1. Attacker is a verified organization member (no admin role required).
2. Attacker discovers the victim's Hedera public key — public keys are visible via `GET /user/:userId/keys` for any verified user, and are also publicly indexed on Hedera mirror nodes.
3. Attacker calls `POST /user/any/keys` with `{ "publicKey": "<victim_public_key>" }` before the victim registers it.
4. The key is stored under the attacker's `userId`.
5. Victim calls the same endpoint with their own key → `ErrorCodes.PU` (HTTP 400) every time. [4](#0-3) 

**Removal is not available to the victim.** The only user-facing delete endpoint enforces ownership: [5](#0-4) 

The internal `removeKey` (no ownership check) is not exposed through the controller: [6](#0-5) 

**Skipped test confirms the missing ownership check is a known gap:** [7](#0-6) 

The test `'should throw if the signature key does not belong to the user'` is marked `.skip`, meaning the guard for key-ownership during approval was never enforced.

### Impact Explanation

- **Key squatting / permanent registration DoS**: The victim's real Hedera public key is permanently claimed by the attacker. The victim cannot register it and therefore cannot be added as a signer or approver using their on-chain identity.
- **Transaction workflow freeze**: If a transaction creator adds the attacker's user ID as an approver (expecting the squatted key to be the victim's), the attacker cannot produce a valid signature (they lack the private key), and the victim cannot sign either. The transaction is stuck in `WAITING_FOR_SIGNATURES` indefinitely.
- **Impersonation of key identity**: The attacker's account appears to "own" a public key associated with a known Hedera identity (e.g., a council member), corrupting the organization's key-to-user mapping.

### Likelihood Explanation

The attacker only needs to be a verified organization member — no admin or privileged role is required. Hedera public keys are public by design and trivially discoverable from mirror node APIs or from `GET /user/:userId/keys`. The attack requires a single API call and is executable the moment a new member is invited but before they complete key registration.

### Recommendation

Require a cryptographic proof-of-ownership at key registration time. The standard approach is a **sign-then-register** flow:

1. Server issues a short-lived random challenge nonce to the authenticated user.
2. Client signs the nonce with the private key corresponding to the public key being registered.
3. Server verifies the signature against the submitted public key before persisting the record.

This is consistent with how the approval flow already verifies signatures: [8](#0-7) 

Apply the same `verifyTransactionBodyWithoutNodeAccountIdSignature` pattern (or an equivalent challenge-response) to `uploadKey`.

Additionally, expose an admin-only endpoint backed by the existing `removeKey` method so administrators can revoke squatted keys without requiring attacker cooperation.

### Proof of Concept

**Preconditions**: Attacker (`attacker@org.com`) and victim (`victim@org.com`) are both verified organization members. Victim has not yet registered their key.

```
# Step 1 – Attacker learns victim's public key (publicly visible)
GET /user/<victim_id>/keys
Authorization: Bearer <attacker_jwt>
→ 200 { publicKey: "e0c8ec2758a5879ffac226a13c0c516b799e72e35141a0dd828f94d37988a4b7", ... }

# Step 2 – Attacker registers victim's public key under their own account
POST /user/1/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json
{ "publicKey": "e0c8ec2758a5879ffac226a13c0c516b799e72e35141a0dd828f94d37988a4b7" }
→ 201 Created  (key now owned by attacker's userId)

# Step 3 – Victim attempts to register their own key
POST /user/2/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json
{ "publicKey": "e0c8ec2758a5879ffac226a13c0c516b799e72e35141a0dd828f94d37988a4b7" }
→ 400 Bad Request  ErrorCodes.PU ("Public key in use")

# Step 4 – Victim cannot remove the squatted key (they don't own it)
DELETE /user/2/keys/<squatted_key_id>
Authorization: Bearer <victim_jwt>
→ 400 Bad Request  ErrorCodes.PNY ("Permission not yours")
```

Expected outcome: victim is permanently unable to register their Hedera public key in the organization, blocking all signing and approval participation under their real identity.

  async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
    await attachKeys(user, this.repo.manager);

    // Check if the user already has the maximum number of keys
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }

    // Find the userKey by the publicKey
    let userKey = await this.repo.findOne({
      where: { publicKey: dto.publicKey },
      withDeleted: true,
    });

    if (userKey) {
      // If the userKey found is owned by a different user,
      // or if the userKey has a non null hash or index that doesn't
      // match the hash or index provided
      // throw an error.
      if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
        throw new BadRequestException(ErrorCodes.PU);
      }
      // Set the hash and/or index (only if the current value is null)
      Object.assign(userKey, dto);
    } else {
      userKey = await this.repo.create(dto);
      userKey.user = user;
    }

    if (userKey.deletedAt) {
      await this.repo.recover(userKey);
    }
    return this.repo.save(userKey);
  }

  async removeUserKey(user: User, id: number): Promise<boolean> {
    const userKey = await this.getUserKey({ id });

    if (!userKey) {
      throw new BadRequestException(ErrorCodes.KNF);
    }

    if (userKey.userId !== user.id) {
      throw new BadRequestException(ErrorCodes.PNY);
    }

    await this.repo.softRemove(userKey);

    return true;
  }
