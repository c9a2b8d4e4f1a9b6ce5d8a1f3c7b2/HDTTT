### Title
Any Authenticated User Can Delete Approvers from Any Transaction They Do Not Own

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that `:id` belongs to `:transactionId`. Because these two checks are fully decoupled, any user who owns at least one transaction can delete approvers — including entire approval trees — from any other user's transaction.

### Finding Description

**Root cause — authorization check and mutation target are decoupled**

In `approvers.controller.ts`, the handler is:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` confirms the caller owns `transactionId`. But `removeTransactionApprover(id)` only checks that an approver with that `id` exists — it never verifies the approver belongs to `transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // deletes approver + full subtree
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

`removeNode` performs a recursive soft-delete of the entire approver subtree rooted at `id`: [3](#0-2) 

**Exploit path**

1. Attacker registers an account and creates any transaction (transaction A, id=1). This satisfies the `getCreatorsTransaction` check.
2. Attacker enumerates or guesses the integer `id` of an approver belonging to victim's transaction B (id=2). Because approver IDs are sequential integers, enumeration is trivial.
3. Attacker sends: `DELETE /transactions/1/approvers/{victim_approver_id}`
4. The controller confirms the attacker owns transaction 1 — check passes.
5. `removeTransactionApprover(victim_approver_id)` fetches the approver, finds it exists, and deletes it along with its entire child tree — no ownership check performed.
6. The victim's approval workflow for transaction B is now corrupted or destroyed.

### Impact Explanation

- **Unauthorized state mutation**: Any authenticated user can delete any approver record from any transaction in the system, regardless of ownership.
- **Approval workflow destruction**: Deleting a root approver node triggers `removeNode`, which recursively soft-deletes the entire approval tree via the CTE query. A single request can wipe all approvers for a transaction.
- **Multi-signature bypass**: If a transaction requires M-of-N approvals and an attacker deletes enough approver nodes, the threshold requirement can be undermined or the transaction left in an inconsistent state.
- **Permanent disruption**: Soft-deleted approvers are not automatically restored. The transaction creator would need to manually recreate the entire approval tree.

### Likelihood Explanation

- **Precondition**: The attacker only needs a valid authenticated account — no admin role, no special privilege. Any registered user qualifies.
- **Enumeration**: Approver IDs are auto-incremented integers. An attacker can trivially enumerate valid IDs by observing their own approver IDs or probing sequentially.
- **No rate limiting on DELETE**: Nothing in the controller or guards prevents bulk deletion across many transactions.
- **Realistic**: This is a standard IDOR pattern exploitable by any malicious registered user against any other user's transaction.

### Recommendation

Inside `removeTransactionApprover` (or before calling it in the controller), verify that the approver's root transaction matches the `transactionId` URL parameter. The `getRootNodeFromNode` helper already exists for exactly this purpose:

```typescript
async removeTransactionApprover(id: number, expectedTransactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // NEW: verify the approver belongs to the authorized transaction
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== expectedTransactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover`. This mirrors the pattern already correctly implemented in `updateTransactionApprover`, which does perform this cross-check: [4](#0-3) 

### Proof of Concept

**Setup:**
- User A creates transaction T_A (id=1) with no approvers.
- User B creates transaction T_B (id=2) and adds an approver tree with root approver id=7.

**Attack:**
```
DELETE /transactions/1/approvers/7
Authorization: Bearer <User A's JWT>
```

**Step-by-step:**
1. Controller calls `getCreatorsTransaction(1, userA)` → passes (A owns T_A).
2. Controller calls `removeTransactionApprover(7)`.
3. Service fetches approver id=7 → found (belongs to T_B, but no check performed).
4. `removeNode(7)` executes the recursive CTE, soft-deleting approver 7 and all its children.
5. T_B's entire approval tree is destroyed.

**Expected (correct) behavior:** Request should be rejected with 401/403 because approver 7 does not belong to transaction 1.

**Actual behavior:** Approver 7 (and its subtree) is deleted. User B's transaction T_B loses its approval requirements. [1](#0-0) [2](#0-1)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

    // notifyTransactionAction(this.notificationsService);
  }
