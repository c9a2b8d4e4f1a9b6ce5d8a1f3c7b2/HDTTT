Audit Report

## Title
Unbounded Recursive Depth in `createTransactionApprovers` Enables DoS via DB Exhaustion and Validation-Phase Stack Overflow

## Summary
The `createApprover` inner function inside `ApproversService.createTransactionApprovers` recurses into itself for every nested `approvers` entry with no depth limit. An authenticated transaction creator can submit a single POST request with an arbitrarily deep `approversArray` payload, exhausting the database with unbounded SQL queries and inserts inside a single open transaction, or triggering a synchronous stack overflow during the `class-transformer`/`class-validator` validation phase.

## Finding Description

**Root cause — unbounded recursion in `createApprover`:**

`createTransactionApprovers` defines an inner async function `createApprover` that calls itself for every child in `dtoApprover.approvers` with no depth counter or guard of any kind: [1](#0-0) 

**The DTO is self-referential with no depth constraint:**

`CreateTransactionApproverDto` references itself via `@ValidateNested({ each: true })` and `@Type(() => CreateTransactionApproverDto)`: [2](#0-1) 

This means `class-transformer` (object transformation) and `class-validator` (validation) both recurse **synchronously** through the entire nested structure before the service is reached. A payload with a linear chain of ~5,000–8,000 nested `approvers` objects (well within Express's default 100 KB body limit at ~15–20 bytes per level) triggers `RangeError: Maximum call stack size exceeded` in the synchronous validation pipeline.

**Important correction to the original report's stack overflow claim:** The `createApprover` function is `async`. In Node.js, each `await createApprover(...)` suspends the current frame and schedules a microtask continuation, so async recursion does **not** accumulate call-stack frames in the traditional sense. The stack overflow vector is in the **synchronous** `class-transformer`/`class-validator` phase, not in the service function itself.

**Per-level DB cost inside the recursive call:**

Each recursion level executes multiple queries inside a single open DB transaction:
- `isNode` → 1 `COUNT` query
- `getRootNodeFromNode` → 1 recursive CTE query (when `listId` is set)
- `transactionalEntityManager.insert` → 1 `INSERT`
- Optionally `getApproversByTransactionId` → 1 recursive CTE query (when `userId` is set) [3](#0-2) 

The `isNode` duplicate check does **not** protect against tree nodes (non-user nodes) because it returns `count > 0 && typeof approver.userId === 'number'` — tree nodes have no `userId`, so `isNode` always returns `false` for them, allowing unlimited tree node creation: [4](#0-3) 

**The exposed endpoint requires only a valid JWT and transaction ownership:** [5](#0-4) 

No throttler guard is applied to the approvers controller. The existing throttler guards (`UserThrottlerGuard`, `IpThrottlerGuard`) are only wired to the auth controller: [6](#0-5) 

## Impact Explanation

**Synchronous stack overflow (crash):** A payload with ~5,000–8,000 linearly nested `approvers` objects (≈75–120 KB, within Express's 100 KB default body limit) causes `RangeError: Maximum call stack size exceeded` inside the synchronous `class-transformer`/`class-validator` pipeline, crashing the NestJS worker process before the service is even reached.

**Database exhaustion (sustained):** At moderate depths (e.g., 200 levels, each with 3 children = 3^200 leaf nodes — or more practically, a linear chain of 500 nodes), the function issues 1,500–2,000 SQL queries and inserts inside a single long-lived DB transaction, holding connection-pool slots and table locks until the transaction times out or the server OOMs. Because the entire operation runs inside `dataSource.transaction(...)`, all locks are held for the full duration. [7](#0-6) 

**Severity: High** — a single authenticated HTTP request can crash the API process or render it unresponsive.

## Likelihood Explanation

**Attacker preconditions:**
- Valid JWT (registered, verified user) — achievable by any self-registered user.
- Ownership of at least one transaction — any user can create a transaction via the standard creation endpoint.

No admin or privileged role is required. The attack is a single crafted POST request. The payload is trivially generated programmatically. No rate-limiting is applied to this endpoint in the codebase. [8](#0-7) 

## Recommendation

1. **Enforce a maximum nesting depth** in `createTransactionApprovers` by passing a `depth` counter into `createApprover` and throwing a `BadRequestException` when it exceeds a reasonable limit (e.g., 10).
2. **Enforce a maximum total node count** across the entire `approversArray` to bound DB operations per request.
3. **Apply the `UserThrottlerGuard`** to the approvers controller or globally to limit per-user request rates.
4. **Configure a JSON body size limit** explicitly in `setupApp` (e.g., `app.use(express.json({ limit: '10kb' }))`) to reduce the attack surface for the validation-phase stack overflow.

## Proof of Concept

```python
import json, requests

# Build a linearly nested payload 500 levels deep
def build_nested(depth):
    if depth == 0:
        return {"userId": 1}
    return {"threshold": 1, "approvers": [build_nested(depth - 1)]}

payload = {"approversArray": [build_nested(500)]}

requests.post(
    "https://target/transactions/1/approvers",
    headers={"Authorization": "Bearer <valid_jwt>"},
    json=payload,
)
# Result: ~1,500 SQL queries inside a single open DB transaction,
# holding connection-pool slots until timeout.
# At depth ~6,000 (still under 100 KB): synchronous RangeError in
# class-transformer/class-validator crashes the NestJS process.
