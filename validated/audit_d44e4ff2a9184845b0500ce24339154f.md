### Title
Any Authenticated User Can Post Comments on Any Transaction Without Being a Participant

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` is protected only by authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but performs no authorization check to verify that the requesting user is actually a participant (creator, signer, observer, or approver) of the target transaction. The codebase itself acknowledges this gap with a `//TODO need some sort of guard or check to ensure user can comment here` comment. Any verified user can post comments on any transaction in the organization.

### Finding Description
**Root cause:** `createComment` in `CommentsService` performs no ownership or membership check before persisting the comment. It blindly associates the authenticated user and the caller-supplied `transactionId`.

**Code path:**

`CommentsController.createComment` (line 21–28) accepts `transactionId` as a URL parameter and `user` from the JWT, then delegates directly to `CommentsService.createComment` with no further authorization: [1](#0-0) 

The service creates and saves the comment without checking whether `user` is a creator, signer, observer, or approver of `transactionId`: [2](#0-1) 

The developer-left TODO on line 22 explicitly acknowledges the missing guard: [3](#0-2) 

Compare this to other sensitive transaction sub-resources (approvers, observers, signers) which all enforce participant-level authorization inside their service methods before mutating state. The `verifyAccess` helper used elsewhere checks creator, signer, observer, and approver membership: [4](#0-3) 

**Exploit flow:**
1. Attacker registers and is verified as a normal organization user.
2. Attacker enumerates or guesses a valid `transactionId` (IDs are sequential integers).
3. Attacker sends `POST /transactions/<id>/comments` with a valid JWT and any comment body.
4. The comment is persisted against the target transaction with no rejection.

### Impact Explanation
- **Unauthorized state modification**: Any verified user can inject arbitrary comments into transactions they have no legitimate relationship with, polluting the audit trail and potentially misleading approvers or signers.
- **Cross-tenant integrity break**: In an organization with multiple teams, a user from one team can comment on another team's sensitive transactions.
- **Workflow interference**: Misleading or spam comments on a pending transaction could influence approval decisions.

### Likelihood Explanation
- Precondition: attacker must be a registered, verified organization user — a realistic baseline for a multi-user organization tool.
- Transaction IDs are sequential integers, trivially enumerable.
- No rate-limiting or ownership check stands in the way.
- Exploitation requires only a valid JWT and a single HTTP request.

### Recommendation
Add a participant-membership check inside `CommentsService.createComment` (or as a guard/interceptor on the route) using the existing `verifyAccess` / `getTransactionWithVerifiedAccess` logic before allowing the comment to be saved. Only users who are the transaction creator, a signer, an observer, or an approver should be permitted to comment.

### Proof of Concept
```
# Step 1: Login as a normal user (not a participant of transaction #42)
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Post a comment on a transaction the attacker has no role in
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "content": "Attacker-injected comment" }

# Expected: 403 Forbidden
# Actual: 201 Created — comment is persisted against transaction 42
```

The comment is saved to the database with `user = attacker` and `transactionId = 42`, with no check that the attacker is a creator, signer, observer, or approver of that transaction. [5](#0-4) [2](#0-1)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
