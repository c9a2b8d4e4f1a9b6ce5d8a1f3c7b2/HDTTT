Audit Report

## Title
Unbounded Recursive Depth in `createTransactionApprovers` Enables DoS via Approver Tree Nesting

## Summary
The inner function `createApprover` inside `createTransactionApprovers` in `approvers.service.ts` recurses without any depth limit for every nested child in the submitted DTO. The `CreateTransactionApproverDto` imposes no maximum nesting depth, and the only global guard against oversized input is a 2 MB body limit. A single authenticated request with a deeply nested approver tree triggers a large number of sequential database queries inside one open TypeORM transaction, exhausting the DB connection pool and degrading or blocking the API service.

## Finding Description

**Unbounded recursive `createApprover`:**
The inner async function `createApprover` is defined at line 245 and calls itself at line 348 for every element of `dtoApprover.approvers`. No depth counter, maximum depth check, or iteration limit exists anywhere in the function or its callers. [1](#0-0) 

**DTO has no depth or size constraint:**
`CreateTransactionApproverDto` declares `approvers` as a self-referential optional array. It carries `@ArrayMinSize(1)` and `@ValidateNested` but no `@ArrayMaxSize` and no depth limit. `class-transformer` will fully deserialize and `class-validator` will fully validate an arbitrarily deep JSON tree before the handler is reached. [2](#0-1) 

**Only guard is a 2 MB body limit:**
`app.use(json({ limit: '2mb' }))` is the sole input-size control. A linear chain of 1 000 nesting levels requires roughly 30 KB of JSON (each level adds ~30 bytes for `{"threshold":1,"approvers":[`), well within the cap. [3](#0-2) 

**Per-level DB amplification inside a single open transaction:**
For each recursive invocation where `listId` is set (every nested level after the first), the function issues: a `COUNT` via `isNode`, a `findOne` for the parent, a recursive-CTE SQL query via `getRootNodeFromNode`, and an `INSERT`. At 1 000 nesting levels this produces thousands of sequential DB round-trips inside one open TypeORM transaction, holding a connection and a transaction lock for the entire duration. [4](#0-3) [5](#0-4) 

**Note on call-stack exhaustion claim:** The original report states the recursion exhausts the V8 call stack. Because `createApprover` is an `async` function and every recursive call is preceded by `await`, each invocation suspends and returns to the event loop before the next frame is pushed. Synchronous call-stack exhaustion does not occur for async recursive functions in Node.js. The real impact is DB connection pool starvation, not a stack overflow crash.

**Route requires only JWT authentication:**
`POST /transactions/:transactionId/approvers` is protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin or privileged role is required. [6](#0-5) 

## Impact Explanation
A single crafted request with a deeply nested approver tree (e.g., 500–1 000 levels, ~15–30 KB body) causes thousands of sequential DB queries to execute inside one open TypeORM transaction. This holds a PostgreSQL connection and transaction lock for the entire duration, starving the connection pool and blocking all other requests that require a DB connection. Because the attack is a single HTTP request, it can be repeated continuously by one attacker, causing sustained service degradation or a full denial of service for all other users.

## Likelihood Explanation
The attacker precondition is minimal: register an account, create any transaction (a standard user workflow), then POST a crafted approver payload. No privileged role, no leaked credentials, and no special network access are required. The attack is a single HTTP request and is fully reproducible.

## Recommendation
1. **Enforce a maximum nesting depth** in `CreateTransactionApproverDto` or at the start of `createTransactionApprovers`. A depth counter passed through each recursive call and checked against a configurable maximum (e.g., 10) is sufficient.
2. **Add `@ArrayMaxSize`** to the `approvers` field in `CreateTransactionApproverDto` to limit the branching factor per level.
3. **Validate total node count** before entering the TypeORM transaction by traversing the DTO tree iteratively and rejecting payloads that exceed a maximum total node count.

## Proof of Concept
```http
POST /transactions/1/approvers HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                ... // repeated ~500 times
                {
                  "userId": 1
                }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```
Each nesting level beyond the first triggers at minimum 3 sequential DB queries (`COUNT`, `findOne`, recursive-CTE `getRootNodeFromNode`) plus an `INSERT`, all inside a single open transaction. At 500 levels this is ~2 000 sequential queries holding one DB connection for the entire duration. [7](#0-6)
