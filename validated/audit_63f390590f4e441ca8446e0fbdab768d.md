Audit Report

## Title
Missing Ownership Check in `removeTransactionGroup` Allows Any Authenticated User to Corrupt Any Transaction Group's State

## Summary
`removeTransactionGroup` in `transaction-groups.service.ts` accepts a `user` parameter but performs no ownership or authorization check before beginning destructive operations. Any authenticated, verified user can call `DELETE /transaction-groups/:id` against a group they do not own. Because `TransactionGroupItem` records are deleted **before** the per-transaction ownership check fires, and there is no wrapping database transaction, the deletion is committed immediately and cannot be rolled back — permanently corrupting the target group's state.

## Finding Description

**Root cause:** `removeTransactionGroup` fetches the group by raw ID with no user filter and immediately begins deleting its items, relying on a downstream check inside `removeTransaction` that fires too late.

**Confirmed code path:**

`DELETE /transaction-groups/:id` → `TransactionGroupsController.removeTransactionGroup` → `TransactionGroupsService.removeTransactionGroup`

At `transaction-groups.service.ts` lines 172–198, the function fetches the group with no user filter:

```typescript
const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
// No ownership check
``` [1](#0-0) 

Then, inside the loop, the `TransactionGroupItem` is hard-deleted **before** `removeTransaction` is called:

```typescript
await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // deleted first
await this.transactionsService.removeTransaction(transactionId, user, false); // ownership check fires here — too late
``` [2](#0-1) 

There is no `this.dataSource.transaction(...)` wrapper around the loop, so each `remove` call is committed immediately and cannot be rolled back when `removeTransaction` subsequently throws.

`removeTransaction` delegates to `getTransactionForCreator`, which does enforce ownership:

```typescript
if (transaction.creatorKey?.userId !== user?.id) {
    throw new UnauthorizedException('Only the creator has access to this transaction');
}
``` [3](#0-2) 

But by the time this check throws, the first `TransactionGroupItem` row has already been hard-deleted. The exception propagates, breaking the loop, leaving the group in a partially corrupted state.

**Contrast with `cancelTransactionGroup`** (lines 200–212), which correctly calls `getTransactionGroup(user, groupId, false)` — a query that filters results by user visibility — and then explicitly asserts `allOwnedByUser` before touching any data:

```typescript
const group = await this.getTransactionGroup(user, groupId, false);
const allOwnedByUser = group.groupItems.every(
  item => item.transaction?.creatorKey?.userId === user.id,
);
if (!allOwnedByUser) {
  throw new UnauthorizedException('Only the creator can cancel all transactions in a group.');
}
``` [4](#0-3) 

`removeTransactionGroup` skips both of these steps entirely.

## Impact Explanation

An attacker who is a legitimate, verified organization member (no admin role required) can:

1. Enumerate any group ID via `GET /transaction-groups`, which is marked "TESTING ONLY" but has no role guard and returns all groups to any authenticated user.
2. Call `DELETE /transaction-groups/:id` against a group owned by another user.
3. The first `TransactionGroupItem` in the target group is **permanently hard-deleted** from the database before the ownership check fires.
4. The `TransactionGroup` record and the underlying `Transaction` records survive (because `removeTransaction` throws), but the first `TransactionGroupItem` linking row is gone.
5. If the group had only one item, the group shell now has zero items. `getTransactionGroup` throws `UnauthorizedException` when `groupItems.length === 0`, making the group invisible to its legitimate owner and impossible to cancel or manage. [5](#0-4) 

This is **permanent, unrecoverable data corruption** of another user's transaction group.

The `GET /transaction-groups` endpoint that enables enumeration: [6](#0-5) 

## Likelihood Explanation

- **Attacker precondition**: A valid JWT for any verified organization user. No admin role, no leaked secrets, no special privileges.
- **Entry point**: `DELETE /transaction-groups/:id` — a standard REST endpoint, reachable by any authenticated client.
- **Discovery**: `GET /transaction-groups` (marked "TESTING ONLY" but unguarded by any role check) returns all groups, making enumeration trivial. Group IDs are sequential integers.
- **Effort**: A single HTTP DELETE request per target group.

## Recommendation

1. **Add an ownership check at the start of `removeTransactionGroup`**, mirroring the pattern used in `cancelTransactionGroup`: call `getTransactionGroup(user, id, false)` to retrieve only groups visible to the caller, and verify `allOwnedByUser` before proceeding.
2. **Wrap the entire deletion loop in a database transaction** (`this.dataSource.transaction(async manager => { ... })`), so that any mid-loop failure rolls back all prior deletions atomically.
3. **Restrict or remove `GET /transaction-groups`**: the endpoint is labeled "TESTING ONLY" but has no role guard, enabling trivial enumeration of all group IDs.

## Proof of Concept

```
# Step 1: Enumerate all group IDs (no role required)
GET /transaction-groups
Authorization: Bearer <attacker_jwt>

# Step 2: Target a group owned by another user
DELETE /transaction-groups/42
Authorization: Bearer <attacker_jwt>

# Result:
# - The first TransactionGroupItem for group 42 is permanently deleted
# - If group 42 had only one item, the group is now inaccessible to its owner
# - No error is returned to the attacker for the partial deletion
```
