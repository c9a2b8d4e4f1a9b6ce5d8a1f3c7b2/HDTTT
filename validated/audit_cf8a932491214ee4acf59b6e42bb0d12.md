Audit Report

## Title
Stale Pre-Update `userApprovers` Snapshot Causes `emitTransactionStatusUpdate` to Never Fire on Approval

## Summary
In `approvers.service.ts`, `approveTransaction` reads `userApprovers` from the database before persisting the new approval, then evaluates the post-write notification branch using that stale in-memory snapshot. Because the snapshot still carries `approved = null`, the expression `userApprovers.every(a => a.approved)` is always `false` when `dto.approved = true`. Consequently, `emitTransactionStatusUpdate` is never emitted on an approval action; only `emitTransactionUpdate` is emitted.

## Finding Description
`userApprovers` is populated at lines 553–556, before any write: [1](#0-0) 

The database update at lines 599–610 persists `approved: dto.approved` but does not update the in-memory `userApprovers` array: [2](#0-1) 

The notification branch at line 614 then evaluates the stale snapshot: [3](#0-2) 

When `dto.approved = true`:
- `!dto.approved` → `false`
- `userApprovers.every(a => a.approved)` → `false` (stale `null`/`undefined` values)
- Combined: `false` → `emitTransactionUpdate` fires; `emitTransactionStatusUpdate` does not

The guard at line 563 (which checks `a.signature`) ensures the user has not yet signed, meaning the in-memory records will always have `approved = null` at the point of the stale check, making the bug deterministic: [4](#0-3) 

The existing test at line 1278 appears to cover this path but actually passes only because a second call with `approved: false` (line 1272) satisfies `!dto.approved = true` and triggers `emitTransactionStatusUpdate` — the approval call itself never does: [5](#0-4) 

## Impact Explanation
`emitTransactionStatusUpdate` is the event that signals the Chain service to re-evaluate whether a transaction has cleared all approval gates and should advance to execution. Because this event is never emitted when a user approves, a transaction can remain stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely after all required approvers have signed — unless an out-of-band trigger (e.g., an approver removal, which calls `emitTransactionStatusUpdate` unconditionally at line 541) happens to fire for the same transaction. In the normal approval-only workflow, no such trigger exists, making the freeze permanent within that workflow. [6](#0-5) 

## Likelihood Explanation
Every approval action (`dto.approved = true`) by every user on every transaction hits this code path. No special attacker capability or race condition is required — any legitimate approver exercising the normal approval workflow triggers the bug. The condition is deterministically wrong: the stale-data check can never return `true` for a first-time approval, so the wrong event is always emitted.

## Recommendation
After the database write, either re-fetch `userApprovers` from the database, or update the in-memory snapshot to reflect the newly written values before evaluating the notification branch. For example:

```typescript
// After the DB update, reflect the new approved value in-memory:
userApprovers.forEach(a => { a.approved = dto.approved; });

if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

Alternatively, re-query the approver rows after the write to get a fresh snapshot before the branch evaluation.

The test at line 1271–1278 should also be split into two separate test cases — one for `approved: true` and one for `approved: false` — each asserting the correct event independently, so the masking effect of the combined call is eliminated. [7](#0-6) 

## Proof of Concept
1. User A is the sole approver of transaction T (status: `WAITING_FOR_SIGNATURES`).
2. User A calls `POST /transactions/T/approvers/approve` with `{ approved: true, signature: <valid>, userKeyId: <valid> }`.
3. `approveTransaction` reads `userApprovers` → `[{ id: 1, userId: A, approved: null }]`.
4. The DB update sets `approved = true` for approver row 1.
5. Line 614 evaluates: `!true || [{ approved: null }].every(a => a.approved)` → `false || false` → `false`.
6. `emitTransactionUpdate` fires; `emitTransactionStatusUpdate` does not.
7. The Chain service is never notified. Transaction T remains in `WAITING_FOR_SIGNATURES` indefinitely despite having all required approvals.
8. No in-protocol recovery path exists through the normal approval workflow alone. [8](#0-7)
