### Title
`dualCompareHash` Throws Unhandled Exception on Bcrypt-Hashed Passwords, Permanently Locking Out Legacy Users

### Summary
The `dualCompareHash` helper used across the backend (`users.service.ts`, `auth.service.ts`) and frontend (`crypto.ts`) calls both `bcrypt.compare` and `argon2.verify` unconditionally and sequentially, with no exception handling. When a stored password hash is in bcrypt format (legacy users), `argon2.verify(bcrypt_hash, data)` throws because the argon2 library strictly validates that the hash starts with `$argon2`. This unhandled exception propagates out of `dualCompareHash` before the correct result can be returned, causing login to fail with a 500 error for every user whose password was hashed with bcrypt. Because the bcrypt→argon2 migration only occurs on a *successful* login, affected users are permanently locked out with no self-service recovery path.

### Finding Description

**Root cause — three identical implementations, no exception guard:**

`back-end/apps/api/src/users/users.service.ts` lines 179–182: [1](#0-0) 

`back-end/apps/api/src/auth/auth.service.ts` lines 168–172: [2](#0-1) 

`front-end/src/main/utils/crypto.ts` lines 60–64: [3](#0-2) 

All three follow the same pattern:
```typescript
const matchBcrypt = await bcrypt.compare(data, hash);   // step 1
const matchArgon2 = await argon2.verify(hash, data);    // step 2 — always runs
return { correct: matchBcrypt || matchArgon2, ... };
```

Step 2 is **not** short-circuited by step 1. For a user whose stored hash is a bcrypt string (e.g. `$2b$10$...`), `argon2.verify` receives a non-argon2 hash and throws `Error: The hash did not match the expected format` (or equivalent native binding error). This exception is never caught, so it propagates to the caller.

**Call sites that surface the failure:**

`getVerifiedUser` in `users.service.ts` (login path): [4](#0-3) 

`changePassword` in `auth.service.ts`: [5](#0-4) 

Neither call site wraps `dualCompareHash` in a try/catch, so the argon2 exception becomes an unhandled 500 Internal Server Error.

**Migration path is self-defeating:** The code intends to migrate bcrypt users to argon2 on first successful login (`if (isBcrypt) { await this.setPassword(user, password); }`). [4](#0-3) 

Because the exception fires before `{ correct, isBcrypt }` is ever returned, the migration branch is unreachable for bcrypt users — an exact structural analog to the external report's "fallback unreachable due to early return."

The same flaw exists in the Electron frontend's IPC handler `compareHash` and `compareDataToHashes`: [6](#0-5) 

### Impact Explanation

Any user whose password was stored with bcrypt (all users registered before the argon2 migration was introduced) cannot log in to either the organization backend or the local personal mode. Because the migration only fires on a successful login, there is no automatic recovery. An admin could manually reset the password, but the affected user has no self-service path. This is a **permanent account lockout** for the entire legacy user cohort — matching the "permanent lock / unrecoverable corruption of user state" impact category in RESEARCHER.md.

### Likelihood Explanation

The bcrypt→argon2 migration is explicitly coded and tested (test: "Should update the password if the password is encrypted with bcrypt"), confirming the developers expect bcrypt hashes to exist in deployed databases. [7](#0-6) 

Any deployment that was running before the argon2 migration was merged will have bcrypt-hashed users. No attacker action is required — the bug triggers on a normal login attempt by a legitimate user.

### Recommendation

Wrap each library call in a try/catch and treat a thrown exception as a non-match, preserving the dual-fallback semantics:

```typescript
async dualCompareHash(data: string, hash: string) {
  let matchBcrypt = false;
  let matchArgon2 = false;

  try { matchBcrypt = await bcrypt.compare(data, hash); } catch { /* not a bcrypt hash */ }
  try { matchArgon2 = await argon2.verify(hash, data); } catch { /* not an argon2 hash */ }

  return { correct: matchBcrypt || matchArgon2, isBcrypt: matchBcrypt };
}
```

Apply this fix to all three implementations: `back-end/apps/api/src/users/users.service.ts`, `back-end/apps/api/src/auth/auth.service.ts`, and `front-end/src/main/utils/crypto.ts`.

### Proof of Concept

1. Register a user while the system still stores bcrypt hashes (or directly insert a bcrypt-hashed password into the database: `$2b$10$<valid-bcrypt-hash>`).
2. Attempt to log in via `POST /auth/login` with the correct credentials.
3. `getVerifiedUser` calls `dualCompareHash(password, bcrypt_hash)`.
4. `bcrypt.compare` returns `true` (correct password, valid bcrypt hash).
5. `argon2.verify(bcrypt_hash, password)` throws because `$2b$10$...` is not a valid argon2 hash string.
6. The exception propagates; NestJS returns HTTP 500 instead of 200 with a JWT.
7. The user cannot log in. The migration branch (`if (isBcrypt) setPassword(...)`) is never reached. Repeated attempts produce the same result.
