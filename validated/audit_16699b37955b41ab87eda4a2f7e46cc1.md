Audit Report

## Title
TOCTOU Race in `archiveTransaction` Allows Unconditional Overwrite of `EXECUTED` Status with `ARCHIVED`

## Summary
`archiveTransaction` in `TransactionsService` performs a non-atomic read-validate-write sequence. The status is read and validated, but the subsequent `UPDATE` carries no `WHERE status IN (...)` predicate. A concurrent scheduler execution that transitions the transaction to `EXECUTED` between the read and the write is silently overwritten with `ARCHIVED`, permanently corrupting the audit trail.

## Finding Description

**Root cause — unguarded write in `archiveTransaction`:**

`archiveTransaction` reads the current status at line 709, validates it at lines 711–718, then unconditionally writes `ARCHIVED` at line 720 with no `WHERE status = <expected>` predicate:

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user); // line 709 — READ

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);                 // lines 711–718 — VALIDATE
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED }); // line 720 — UNCONDITIONAL WRITE
``` [1](#0-0) 

**Contrast with the correctly guarded `cancelTransactionWithOutcome`:**

`cancelTransactionWithOutcome` appends `.andWhere('status IN (:...statuses)', ...)` so the UPDATE is a no-op if the status changed between read and write:

```typescript
const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.CANCELED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses }) // guard
  .execute();
``` [2](#0-1) 

**The scheduler's own write is also correctly guarded:**

`_executeTransaction` in `ExecuteService` uses `WHERE id = :id AND status = :currentStatus` (WAITING_FOR_EXECUTION), so it will not overwrite a row that has already been archived. The problem is strictly the reverse direction:

```typescript
const updateResult = await this.transactionsRepo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: transactionStatus, executedAt, statusCode: transactionStatusCode })
  .where('id = :id AND status = :currentStatus', {
    id: transaction.id,
    currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
  })
  .execute();
``` [3](#0-2) 

**Exploit path:**

1. Creator submits a non-manual transaction; it reaches `WAITING_FOR_EXECUTION` (all required signatures collected).
2. Creator sends `PATCH /transactions/:id/archive` — `archiveTransaction` reads status `WAITING_FOR_EXECUTION`, passes the guard at lines 711–718.
3. Concurrently, the chain scheduler's `handleTransactionsBetweenNowAndAfterThreeMinutes` (runs every 10 seconds) calls `prepareTransactions` → `collateAndExecute` → `addExecutionTimeout` → `executeService.executeTransaction`. The guarded UPDATE sets status to `EXECUTED` in the database.
4. `archiveTransaction` then executes `repo.update({ id }, { status: ARCHIVED })` unconditionally, overwriting `EXECUTED`.

The scheduler runs on a 10-second cron for transactions whose `validStart` is within the ±3-minute window: [4](#0-3) 

`addExecutionTimeout` explicitly skips manual transactions, confirming the scheduler only auto-executes non-manual ones — the same class of transactions that can be archived via the status check: [5](#0-4) 

**State machine — no central transition table:**

`TransactionStatus` defines eight states with no single authoritative transition guard. Each service method independently re-implements its own ad-hoc status checks: [6](#0-5) 

## Impact Explanation

- A transaction that was successfully executed on the Hedera network (real-world effect already occurred) is recorded as `ARCHIVED` in the database.
- The `statusCode` field (Hedera response code, set by `_executeTransaction`) is absent from the `ARCHIVED` record, so the execution result is permanently lost. [7](#0-6) 
- All participants (signers, observers, approvers) receive `ARCHIVED` notifications instead of `EXECUTED` notifications, as the notification service maps status to notification type. [8](#0-7) 
- The audit trail is permanently corrupted: the creator can use this to conceal that a transaction was executed, misleading co-signers and observers about the outcome of a multi-party workflow.

## Likelihood Explanation

- **Attacker profile:** any authenticated user who is the creator of a transaction — no privileged access required.
- **Precondition:** the transaction must be in `WAITING_FOR_EXECUTION`, a normal, reachable state in every multi-signature workflow.
- **Timing:** `validStart` is stored in the database and returned to the creator, making the scheduler's 10-second execution window predictable. The creator can issue the archive request immediately before `validStart` to maximize the race probability.
- **Repeatability:** the request can be retried at zero cost until the race is won.

## Recommendation

Replace the unconditional `repo.update` in `archiveTransaction` with a guarded QueryBuilder update, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  // Re-read and surface the correct error (already executed, expired, etc.)
  const latest = await this.getTransactionForCreator(id, user);
  throw new ConflictException(
    `Cannot archive transaction in status: ${latest.status}`,
  );
}
```

Additionally, the `isManual` bypass in the current guard (lines 714–715) allows a manual transaction in any terminal status (e.g., `EXECUTED`, `FAILED`) to be archived without a race condition at all. The guarded write above also closes that path because `EXECUTED` and `FAILED` are not in `archivableStatuses`.

## Proof of Concept

```
T=0   Creator: GET /transactions/42  → status = WAITING_FOR_EXECUTION
T=1   Creator: PATCH /transactions/42/archive
        → archiveTransaction reads status = WAITING_FOR_EXECUTION  ✓ passes guard
        → [awaiting repo.update...]

T=2   Scheduler (10s cron fires):
        → _executeTransaction runs guarded UPDATE:
             UPDATE transaction SET status='EXECUTED', statusCode=22, executedAt=now
             WHERE id=42 AND status='WAITING FOR EXECUTION'
          → 1 row affected → status is now EXECUTED in DB

T=3   archiveTransaction resumes:
        → repo.update({ id: 42 }, { status: 'ARCHIVED' })   ← no WHERE guard
          → 1 row affected → status is now ARCHIVED in DB
          → statusCode is NULL (not set by archiveTransaction)
          → ARCHIVED notification emitted to all participants

Result: on-chain-executed transaction recorded as ARCHIVED; execution result lost.
