Audit Report

## Title
Missing Unique Database Constraint on `transaction_signer` Enables Duplicate Signer Insertion via Concurrent Requests (TOCTOU)

## Summary
The `transaction_signer` table has no unique constraint on `(transactionId, userKeyId)`. The application-level duplicate-signer check in `loadTransactionData` / `processTransactionSignatures` is a read-then-write pattern with no atomicity guarantee. Two concurrent `POST /transactions/:id/signers` requests from the same authenticated user can both pass the in-memory check and both insert duplicate `TransactionSigner` rows, corrupting the signer ledger for a transaction.

## Finding Description

**Root cause — no database-level uniqueness enforcement**

The `TransactionSigner` entity declares only a non-unique index:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← performance index, NOT unique
export class TransactionSigner { … }
``` [1](#0-0) 

Compare this to `TransactionObserver`, which correctly uses `{ unique: true }`:

```typescript
@Index(['userId', 'transactionId'], { unique: true })
``` [2](#0-1) 

The migration confirms no `UNIQUE` constraint exists on the table — only a plain `PRIMARY KEY` on `id`:

```sql
CREATE TABLE IF NOT EXISTS "transaction_signer" (
  "id" SERIAL NOT NULL, "transactionId" integer NOT NULL,
  "userKeyId" integer NOT NULL, "userId" integer NOT NULL, …
  CONSTRAINT "PK_…" PRIMARY KEY ("id")
)
``` [3](#0-2) 

The later index migration also creates only a non-unique index:

```sql
CREATE INDEX "IDX_f8cbac3e0cae3730a762e4905e" ON "transaction_signer" ("transactionId", "userKeyId")
``` [4](#0-3) 

**TOCTOU race in the application-level guard**

`uploadSignatureMaps` follows a read → validate → write pattern split across separate async steps:

**Step 1 — Read** (outside any DB transaction): `loadTransactionData` fetches existing signers and builds `signersByTransaction` at time T₁. [5](#0-4) [6](#0-5) 

**Step 2 — Validate**: `processTransactionSignatures` uses the stale snapshot to decide which keys are "new":

```typescript
if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);
}
``` [7](#0-6) 

**Step 3 — Write**: `bulkInsertSigners` inserts without any conflict-resolution clause:

```typescript
await manager.createQueryBuilder()
  .insert().into(TransactionSigner).values(signersToInsert)
  .returning('*').execute();
``` [8](#0-7) 

The DB transaction wrapping the write (line 317) does **not** include the read at line 104, so two concurrent requests that both read the same empty `signersByTransaction` snapshot will both classify the same key as "new" and both successfully insert a `TransactionSigner` row for the same `(transactionId, userKeyId)` pair. [9](#0-8) 

## Impact Explanation

- **Duplicate `TransactionSigner` rows** for the same `(transactionId, userKeyId)` pair are persisted permanently with no self-correction mechanism.
- Any downstream logic that counts `TransactionSigner` records to evaluate whether a signing threshold is met (e.g., `processTransactionStatus`) will see an inflated count, potentially advancing a transaction to `WAITING_FOR_EXECUTION` before all *distinct* required signers have actually signed.
- Notification dismissal (`bulkUpdateNotificationReceivers`) is triggered for each duplicate, causing spurious notification state changes visible to all organization members.
- The corrupted signer ledger cannot be corrected by normal users.

## Likelihood Explanation

Any authenticated user who owns at least one key registered on a transaction can trigger this by issuing two HTTP `POST` requests to the `uploadSignatureMap` endpoint simultaneously (e.g., with `Promise.all` or two browser tabs). No privileged access, no leaked credentials, and no special network position are required. The race window spans the entire async processing pipeline (~tens of milliseconds), which is wide enough to be reliably hit. [10](#0-9) 

## Recommendation

Apply both fixes together:

1. **Add a unique constraint at the database level** on `(transactionId, userKeyId)` in the `TransactionSigner` entity:
   ```typescript
   @Index(['transactionId', 'userKeyId'], { unique: true })
   ```
   and generate a corresponding migration.

2. **Add conflict-resolution in `bulkInsertSigners`** so that a race-induced duplicate is silently ignored rather than causing an error:
   ```typescript
   .orIgnore()   // or .onConflict('DO NOTHING')
   ```
   This mirrors the pattern already used in `cache.helper.ts`: [11](#0-10) 

Together these ensure that even if two concurrent requests race past the application-level check, the database will reject the second insert, and the application will handle it gracefully.

## Proof of Concept

```typescript
// Attacker owns key K registered on transaction txId=X
const payload = [{ id: X, signatureMap: signedMap }];

// Fire two concurrent POST requests
const [r1, r2] = await Promise.all([
  fetch('/transactions/signers', { method: 'POST', body: JSON.stringify(payload), headers }),
  fetch('/transactions/signers', { method: 'POST', body: JSON.stringify(payload), headers }),
]);

// Both return HTTP 201
// DB now contains two rows: { transactionId: X, userKeyId: K }
// SELECT COUNT(*) FROM transaction_signer WHERE "transactionId"=X AND "userKeyId"=K;
// → 2  (expected: 1)
```

Both requests independently call `loadTransactionData` before either has written, both see `signersByTransaction` as empty for key K, both classify K as a new signer, and both successfully execute `bulkInsertSigners` — resulting in two persisted rows for the same `(transactionId, userKeyId)` pair.
