### Title
Hardcoded Credentials and Weak Secrets Committed to Source Control

### Summary
The repository contains actual environment files and Kubernetes deployment manifests committed to source control with hardcoded, trivially weak secrets — including `JWT_SECRET=13123`, `OTP_SECRET=123`, and `POSTGRES_PASSWORD=postgres`. Anyone with read access to the repository can extract these secrets and forge JWT tokens to authenticate as any user, including administrators.

### Finding Description

Two categories of committed secrets are present:

**Category 1 — Actual `.env` file committed (not a template):**

`back-end/apps/api/.env.test` is a real environment file (not named `example.env`) committed directly to source control. It contains:
- `JWT_SECRET=13123` — a 5-character numeric string used to sign all JWTs
- `OTP_SECRET=123` — a 3-character numeric string used as the TOTP base secret for password reset [1](#0-0) 

**Category 2 — Hardcoded credentials in a committed Kubernetes dev deployment manifest:**

`back-end/k8s/dev/deployments/api-deployment.yaml` contains inline plaintext values for:
- `POSTGRES_PASSWORD: 'postgres'`
- `JWT_SECRET: 'some-secret-value'`
- `OTP_SECRET: 'some-secret-otp-value'` [2](#0-1) 

These secrets are consumed directly by the authentication service. `JWT_SECRET` is used to sign and verify all access tokens via `jwtService.sign()` and `jwtService.verifyAsync()`. `OTP_SECRET` is concatenated with a user's email to derive the per-user TOTP secret used for password reset flows. [3](#0-2) [4](#0-3) 

The production Kubernetes deployment (`back-end/k8s/back-end/api-deployment.yaml`) correctly uses `secretKeyRef` to pull from Kubernetes Secrets, but the dev manifest does not, and the `.env.test` file is committed with real (weak) values. [5](#0-4) 

No `.gitignore` was found in the repository, confirming `.env.test` was not excluded from version control.

### Impact Explanation

An attacker who obtains the repository (e.g., via a leaked clone, a former contributor, or a public fork) can:

1. **Forge JWT tokens**: With `JWT_SECRET=13123`, they can craft a valid signed JWT with any `userId` and `email` payload, bypassing all authentication. The `authenticateWebsocketToken` method and the standard login flow both rely on `jwtService.verifyAsync(accessToken)` with this secret.
2. **Generate valid OTP codes**: With `OTP_SECRET=123`, the per-user TOTP secret is `"123" + email`. An attacker can compute valid TOTP codes for any known user email and trigger a password reset, taking over any account.
3. **Access the PostgreSQL database directly**: With `POSTGRES_PASSWORD=postgres` and the host/port exposed in the manifests, direct database access is possible if the dev environment is reachable. [6](#0-5) 

### Likelihood Explanation

The `.env.test` file is committed and present in the repository today — no exploitation prerequisite beyond repository read access is required. The dev Kubernetes manifest is similarly committed. Any contributor, former employee, or attacker who obtained a clone of the repository at any point in its history has these credentials. The secrets are trivially weak (`13123`, `123`) and would also be cracked instantly by brute force even without source access.

### Recommendation

**Short term:**
- Remove `back-end/apps/api/.env.test` from source control and add `*.env` / `.env*` (excluding `example.env`) to a `.gitignore`.
- Remove hardcoded credential values from `back-end/k8s/dev/deployments/api-deployment.yaml` and replace with `secretKeyRef` references, matching the pattern used in the production manifest.
- Rotate `JWT_SECRET`, `OTP_SECRET`, and `POSTGRES_PASSWORD` immediately, as they are now considered compromised.

**Long term:**
- Adopt a secrets management solution (e.g., HashiCorp Vault, AWS Secrets Manager) for all environments including dev/test.
- Add a pre-commit hook or CI check (e.g., `trufflehog`, `gitleaks`) to prevent future secret commits.

### Proof of Concept

1. Clone the repository.
2. Read `back-end/apps/api/.env.test` — extract `JWT_SECRET=13123`.
3. Use any JWT library to sign a payload:
   ```js
   const jwt = require('jsonwebtoken');
   const token = jwt.sign({ userId: 1, email: 'admin@example.com' }, '13123', { expiresIn: '365d' });
   ```
4. Send the forged token as a `Bearer` token to any authenticated API endpoint. The server will accept it as a valid session for `userId: 1`.

For OTP takeover:
1. Extract `OTP_SECRET=123` from `.env.test`.
2. For a known user email (e.g., `victim@example.com`), compute `secret = "123" + "victim@example.com"`.
3. Generate a valid TOTP code using `otplib` with `digits: 8, step: 60, window: 20`.
4. Submit the code to the OTP verification endpoint to reset the victim's password. [7](#0-6) [8](#0-7) [9](#0-8)
