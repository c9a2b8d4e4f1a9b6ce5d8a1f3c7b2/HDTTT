## Audit Report

## Title
Sequential Transaction Group Ordering Not Enforced at Execution Time

## Summary
`executeTransactionGroup` silently drops group members that are not yet in `WAITING_FOR_EXECUTION` status before submitting the remaining ones. For sequential groups this means a later transaction can be submitted to the Hedera network before an earlier, still-unsigned transaction, breaking the causal dependency the group was designed to enforce.

## Finding Description

**Root cause — `execute.service.ts` lines 65–67:**

```typescript
transactionGroup.groupItems = transactionGroup.groupItems.filter(
  tx => tx.transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
);
```

This filter silently removes any group member that has not yet reached `WAITING_FOR_EXECUTION`. The sequential execution loop that follows operates only on the survivors, with no guard verifying that every preceding item (by `seq` or `validStart`) is also ready.

**Trigger path — `transaction-scheduler.service.ts` `prepareTransactions` (lines 163–199):**

The scheduler fires `collateGroupAndExecute` as soon as it finds *any* group member that is `WAITING_FOR_EXECUTION` with an executable `validStart`. There is no check that *all* members of the sequential group have reached `WAITING_FOR_EXECUTION` first.

**`collateGroupAndExecute` (lines 209–259):**

The callback iterates `for (const groupItem of transactionGroup.groupItems)` — all items, regardless of status — and calls `smartCollate` on each. `smartCollate` only checks transaction byte size; it does not inspect the transaction's `status` field. If all items pass the size check, `addGroupExecutionTimeout` is called, which eventually calls `executeTransactionGroup`, where the status filter silently drops the not-yet-signed items.

**Concrete scenario:**

| Item | seq | validStart | Signers required | Status when scheduler fires |
|------|-----|-----------|------------------|-----------------------------|
| Tx A | 1 | T | Alice + Bob | `WAITING_FOR_SIGNATURES` (Bob hasn't signed) |
| Tx B | 2 | T+1 s | Alice only | `WAITING_FOR_EXECUTION` |

1. Alice signs both; Bob has not yet signed Tx A.
2. The every-10-second cron finds Tx B in `WAITING_FOR_EXECUTION` with an executable `validStart`.
3. `prepareTransactions` triggers `collateGroupAndExecute` for the group.
4. `collateGroupAndExecute` runs `smartCollate` on both Tx A and Tx B — both pass (normal size).
5. `addGroupExecutionTimeout` → `executeTransactionGroup` is called.
6. `executeTransactionGroup` filters Tx A out (still `WAITING_FOR_SIGNATURES`).
7. Tx B is submitted to Hedera **before** Tx A.
8. If Tx B depends on the result of Tx A (e.g., a transfer to an account that Tx A was supposed to create), Tx B fails on-chain.

## Impact Explanation
Any sequential transaction group whose members have heterogeneous signature requirements can have its ordering guarantee silently violated. Dependent operations — account creation followed by a transfer, file creation followed by a file append, node creation followed by a node update — will fail on-chain with no clear error surfaced to the user. The scheduler does not report the ordering violation; it simply executes whatever is ready.

## Likelihood Explanation
Organization mode is explicitly designed for multi-signer workflows where different transactions in a group can have different threshold-key requirements. A group creator routinely assigns different signers to different items. The 10-second polling interval (`EVERY_10_SECONDS` cron at `transaction-scheduler.service.ts:87`) means the window is narrow but recurring; any time signatures arrive for a later item before an earlier item is fully signed, the race is entered. This is a normal operational pattern, not an edge case.

## Recommendation
In `prepareTransactions` (or at the entry of `executeTransactionGroup`), before dispatching a sequential group, verify that **all** group members have reached `WAITING_FOR_EXECUTION`. If any member is still in `WAITING_FOR_SIGNATURES` (or `NEW`), abort the dispatch for the entire group and wait for the next scheduler tick. A minimal guard would be:

```typescript
// In prepareTransactions, before calling collateGroupAndExecute for a sequential group:
const allReady = transactionGroup.groupItems.every(
  gi => gi.transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
);
if (!allReady) continue; // skip until all members are ready
```

Alternatively, enforce this invariant inside `executeTransactionGroup` itself: if the group is sequential and any item was filtered out, throw an error rather than silently proceeding with a partial set.

## Proof of Concept

1. Create an organization-mode sequential group with two transactions:
   - **Tx A** (`seq=1`, `validStart=T`): requires two signers (Alice + Bob).
   - **Tx B** (`seq=2`, `validStart=T+1s`): requires one signer (Alice only).
2. Alice signs both transactions. Bob does **not** sign Tx A.
3. After Alice's signature, Tx B transitions to `WAITING_FOR_EXECUTION`; Tx A remains `WAITING_FOR_SIGNATURES`.
4. Wait for the 10-second cron (`handleTransactionsBetweenNowAndAfterThreeMinutes`) to fire after `T+1s`.
5. Observe that `prepareTransactions` calls `collateGroupAndExecute` for the group (triggered by Tx B).
6. `collateGroupAndExecute` runs `smartCollate` on both items without status checks — both pass.
7. `executeTransactionGroup` filters Tx A out and submits only Tx B to Hedera.
8. Tx B executes on-chain before Tx A, violating the sequential ordering guarantee. If Tx B depends on Tx A's result, it fails on-chain with no ordering-violation error surfaced to the user.

  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },
            });
            // All the transactions for the group are now pulled. If there is an issue validating for even one
            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);
        }
      }
    }
  }

export async function smartCollate(
  transaction: Transaction,
  requiredKeys: KeyList,
): Promise<SDKTransaction | null> {
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // HIP-1300: isTransactionOverMaxSize is fee-payer aware. Privileged payers
  // (0.0.2, 0.0.42-0.0.799) can carry up to 128 KB, so transactions under that
  // limit pass through here untouched and signatures are NOT stripped.
  if (await isTransactionOverMaxSize(sdkTransaction)) {
    const publicKeys = computeShortenedPublicKeyList(
      sdkTransaction._signerPublicKeys,
      requiredKeys,
    );

    const signatureMap = sdkTransaction.getSignatures();
    sdkTransaction.removeAllSignatures();

    for (const key of publicKeys) {
      sdkTransaction.addSignature(key, signatureMap);
    }

    // If the transaction is still too large,
    // set it to failed with the TRANSACTION_OVERSIZE status code
    // update the transaction, emit the event, and delete the timeout
    if (await isTransactionOverMaxSize(sdkTransaction)) {
      return null;
    }
  }

  return sdkTransaction;
}
