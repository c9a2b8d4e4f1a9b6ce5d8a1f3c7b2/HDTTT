### Title
Sequential External Calls Per Transaction in Notification Processing Loop Causes Resource Exhaustion

### Summary
`receiver.service.ts` in the notifications app processes batches of transaction events in a `for` loop. For each event, it sequentially calls `keysRequiredToSign` → `computeSignatureKey` → `addSigningAccountKeys`/`addReceiverAccountKeys`, each of which makes one HTTP call to the Hedera Mirror Node **per account** inside its own inner loop. Additionally, a separate database transaction is opened per event. A normal user can create many transactions with reminder timestamps aligned to the same moment, causing the service to fan out O(N × M) sequential mirror-node HTTP calls and N sequential database transactions, degrading or stalling the notifications service.

### Finding Description

**Root cause — outer loop, one DB transaction per event:**

`processSignerReminders` iterates over every event in the batch. For each event it:
1. Calls `keysRequiredToSign` (which internally calls `computeSignatureKey`)
2. Opens a **new** `entityManager.transaction(...)` per event [1](#0-0) 

**Root cause — inner loops, one mirror-node HTTP call per account:**

`computeSignatureKey` calls `addSigningAccountKeys` and `addReceiverAccountKeys`, both of which iterate over every account in the transaction and `await` one HTTP call to the mirror node per account: [2](#0-1) [3](#0-2) 

The same pattern exists in `processTransactionStatusUpdateNotifications`, which also opens one DB transaction per event in its loop: [4](#0-3) 

**Exploit path:**
1. Attacker (normal authenticated user) creates N transactions, each with `reminderMillisecondsBefore` set to fire at the same moment.
2. At reminder time, the chain service emits N `TRANSACTION_REMIND_SIGNERS` events simultaneously.
3. `processSignerReminders` receives all N events in one batch and processes them sequentially.
4. For each of the N transactions, `keysRequiredToSign` → `computeSignatureKey` makes M sequential HTTP calls to the mirror node (M = number of signing + receiver accounts per transaction).
5. Then a separate DB transaction is opened per event.
6. Total cost: N × M sequential mirror-node HTTP calls + N sequential DB transactions, all blocking the notification worker.

### Impact Explanation
The notifications service worker is blocked for the entire duration of the sequential HTTP calls and DB transactions. With enough transactions (N) and accounts per transaction (M), the worker can be stalled long enough to:
- Miss or severely delay all other notification deliveries (signing reminders, execution alerts)
- Exhaust mirror-node rate limits, causing HTTP errors that propagate back as failed notification processing
- Cause DB connection pool exhaustion from N open transactions held simultaneously

This constitutes **service degradation / partial DoS** of the notifications subsystem, affecting all users of the organization backend.

### Likelihood Explanation
Any authenticated user of the organization backend can create transactions. The `reminderMillisecondsBefore` field is user-controlled. No admin privilege is required. The attacker only needs to register, create many transactions with aligned reminder times, and wait. The attack is low-cost and repeatable.

### Recommendation

1. **Cache the mirror-node result across accounts within a single `computeSignatureKey` call.** The `AccountCacheService` already exists; ensure it is warmed before the loop and that results are reused within the same invocation.
2. **Parallelize inner account lookups** using `Promise.all` instead of sequential `await` inside `addSigningAccountKeys` and `addReceiverAccountKeys`.
3. **Wrap the entire event batch in a single DB transaction** in `processSignerReminders` and `processTransactionStatusUpdateNotifications` instead of opening one transaction per event.
4. **Rate-limit transaction creation** per user to bound the maximum batch size an attacker can produce.

### Proof of Concept

1. Register as a normal user on the organization backend.
2. Create 500 transactions via the API, each with `reminderMillisecondsBefore` set to fire 60 seconds from now, and each `TransferTransaction` containing 5 distinct signing accounts.
3. Wait 60 seconds.
4. Observe the notifications service log: it will emit 500 sequential `keysRequiredToSign` calls, each making 5+ sequential HTTP requests to the mirror node (2500+ total), followed by 500 sequential DB `BEGIN`/`COMMIT` pairs.
5. During this window, all other notification deliveries are blocked; any concurrent signing or execution events are queued and delayed. [5](#0-4)

  private async processSignerReminders(
    events: NotificationEventDto[],
    isManual: boolean,
  ) {
    const ctx = await this.prepareEventContext(events);
    if (!ctx) return;

    const {
      cache,
      keyCache,
      transactionMap,
      deletionNotifications,
      inAppNotifications,
      emailNotifications,
      inAppReceiverIds,
      emailReceiverIds,
    } = ctx;

    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping signer reminder`);
        continue;
      }

      const allKeys = await keysRequiredToSign(
        transaction,
        this.transactionSignatureService,
        this.entityManager,
        false,
        null,
        keyCache,
      );

      // Filter out keys/users that have been soft-deleted to prevent notification failures
      const activeKeys = filterActiveUserKeys(allKeys);

      const userIds = new Set(activeKeys.map(k => k.userId).filter(Boolean));

      if (userIds.size === 0) continue;

      await this.entityManager.transaction(async entityManager => {
        await this.handleSignerReminderNotifications(
          entityManager,
          transaction,
          transactionId,
          userIds,
          cache,
          isManual,
          deletionNotifications,
          inAppNotifications,
          inAppReceiverIds,
          emailNotifications,
          emailReceiverIds,
        );
      });
    }

    await this.sendDeletionNotifications(deletionNotifications);
    await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
    await this.sendEmailNotifications(emailNotifications, emailReceiverIds);
  }

  private async addSigningAccountKeys(
    signatureKey: KeyList,
    transaction: Transaction,
    signingAccounts: Set<string>
  ): Promise<void> {
    for (const account of signingAccounts) {
      try {
        const accountInfo = await this.accountCacheService.getAccountInfoForTransaction(
          transaction,
          account,
        );
        if (accountInfo?.key) {
          signatureKey.push(accountInfo.key);
        }
      } catch (error) {
        this.logger.error(`Failed to get key for account ${account}: ${error.message}`);
      }
    }
  }
