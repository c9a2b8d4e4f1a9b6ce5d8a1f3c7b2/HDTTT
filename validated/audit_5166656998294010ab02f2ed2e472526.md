### Title
Missing Approver-to-Transaction Association Verification in `removeTransactionApprover` Allows Cross-Transaction Approver Deletion

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the authenticated user is the creator of the transaction identified by `:transactionId`, but never verifies that the approver identified by `:id` actually belongs to that same transaction. An attacker who is the creator of any transaction can supply their own `transactionId` (which they legitimately own) alongside the `id` of an approver belonging to a completely different transaction, causing that foreign approver to be silently deleted.

### Finding Description

The controller at `back-end/apps/api/src/transactions/approvers/approvers.controller.ts` handles the delete route as follows:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` only confirms the caller is the creator of the transaction referenced by `transactionId`. It says nothing about the approver referenced by `id`.

`removeTransactionApprover(id)` then deletes the approver tree rooted at `id` with no cross-check:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

There is no check that `approver.transactionId === transactionId` (or that the approver's root node belongs to `transactionId`). Compare this with `updateTransactionApprover`, which correctly performs this check:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The missing check is the direct analog of the external report: just as `process_approve_account` approves a token account without verifying `token_account.base.mint == mint_info.key`, `removeTransactionApprover` deletes an approver without verifying `approver.transactionId == transactionId`.

### Impact Explanation

An attacker who is the creator of **any** transaction (even a trivial one they created themselves) can delete approvers from **any other transaction** in the system. This directly undermines the multi-signature approval model:

- A transaction requiring N-of-M approvals can have its approvers silently removed, reducing the threshold to zero and allowing the transaction to proceed without the required authorizations.
- The victim transaction's creator receives no indication that their approver configuration was tampered with.
- The `emitTransactionStatusUpdate` call inside `removeTransactionApprover` will emit a status update for the **victim** transaction's ID (via `approver.transactionId`), not the attacker's `transactionId`, meaning the notification goes to the wrong transaction's participants — further obscuring the attack. [4](#0-3) 

### Likelihood Explanation

The attacker only needs to be a registered, verified user who has created at least one transaction (a trivially achievable precondition in normal product use). No privileged access, leaked credentials, or admin rights are required. The approver IDs are sequential integers, making enumeration straightforward. The attack is a single authenticated HTTP DELETE request.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver's root node belongs to the authorized transaction, mirroring the check already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the authorized transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

### Proof of Concept

1. Attacker registers as a normal user and creates Transaction **T1** (attacker is creator).
2. A separate user creates Transaction **T2** with a required approver record **A2** (approver ID = e.g. `42`).
3. Attacker sends:
   ```
   DELETE /transactions/T1/approvers/42
   Authorization: Bearer <attacker_jwt>
   ```
4. Server calls `getCreatorsTransaction(T1, attacker)` → passes (attacker owns T1).
5. Server calls `removeTransactionApprover(42)` → deletes approver 42 (belonging to T2) with no further check.
6. **Result:** Approver A2 is permanently soft-deleted from T2. T2's approval requirements are now reduced without the knowledge or consent of T2's creator or participants. [1](#0-0) [2](#0-1)
