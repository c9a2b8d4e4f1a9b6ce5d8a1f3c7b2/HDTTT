Audit Report

## Title
Any Authenticated User Can Corrupt Another User's Transaction Group via Missing Upfront Ownership Check in `removeTransactionGroup`

## Summary
`TransactionGroupsService::removeTransactionGroup` permanently deletes `TransactionGroupItem` rows before verifying that the requesting user owns the group. Because the ownership check inside `removeTransaction` → `getTransactionForCreator` is called only *after* the join-table row is already removed, any authenticated, verified user can iteratively strip group items from another user's transaction group one call at a time, permanently corrupting its structure.

## Finding Description
In `removeTransactionGroup`, the group is fetched with no user-ownership filter: [1](#0-0) 

Inside the loop, the `TransactionGroupItem` row is **permanently deleted first**, and only then is `removeTransaction` called to check ownership: [2](#0-1) 

`removeTransaction` delegates ownership verification to `getTransactionForCreator`: [3](#0-2) 

`getTransactionForCreator` throws `UnauthorizedException` when the caller is not the creator: [4](#0-3) 

The exception propagates and aborts the loop — but the `TransactionGroupItem` deleted on line 189 is already gone. Each call to `DELETE /transaction-groups/:id` removes exactly one item before throwing. Repeating the call N times strips all N items, leaving the `TransactionGroup` record as an empty shell.

The endpoint is reachable by any verified, authenticated user: [5](#0-4) [6](#0-5) 

## Impact Explanation
An attacker can permanently destroy the structural integrity of any other user's transaction group. Each call removes one `TransactionGroupItem` row. After N calls (where N is the number of items in the group), the group is an empty shell. The underlying `Transaction` records are not deleted (because `removeTransaction` throws before completing), but the group-to-transaction linkage is severed. This constitutes irreversible data corruption and denial of service against the victim's multi-signature workflow.

## Likelihood Explanation
- The endpoint requires only a valid JWT and verified-user status — no elevated privilege.
- Group IDs are auto-incremented integers, trivially enumerable by any authenticated user.
- The `GET /transaction-groups/:id` endpoint does filter by user (via `getTransactionGroupItemsQuery`), but the `DELETE` endpoint has no equivalent gate.
- No rate-limiting is applied to `DELETE /transaction-groups/:id`.
- The attack requires only repeated HTTP calls with a valid session token.

## Recommendation
Move the ownership check **before** any destructive database operation. The simplest fix is to call `getTransactionGroup(user, id)` (which already enforces user-scoped access via `getTransactionGroupItemsQuery`) at the very start of `removeTransactionGroup`, and throw if the result is empty or unauthorized. Alternatively, add an explicit ownership predicate to the initial `findOneBy` call (e.g., join through `TransactionGroupItem → Transaction → creatorKey → userId = user.id`). The entire loop should also be wrapped in a database transaction so that any mid-loop failure rolls back all prior deletes atomically.

## Proof of Concept
1. Attacker (user B) obtains a valid JWT for their account.
2. Attacker learns victim's group ID (e.g., `id = 42`) by guessing sequential integers.
3. Attacker sends `DELETE /transaction-groups/42`.
   - Line 173: group 42 is found (no user filter).
   - Line 189: the **first** `TransactionGroupItem` is deleted from the database.
   - Line 190 → 630 → 887: `getTransactionForCreator` throws `UnauthorizedException` because user B is not the creator.
   - HTTP 401 is returned to the attacker, but the DB row is already gone.
4. Attacker repeats step 3 for each item in the group (N total calls).
5. After N calls, all `TransactionGroupItem` rows for group 42 are deleted. The `TransactionGroup` record remains but has no items. The victim's multi-signature workflow is permanently broken.

  async getTransactionForCreator(id: number, user: User) {
    const transaction = await this.getTransactionById(id);

    if (!transaction) {
      throw new BadRequestException(ErrorCodes.TNF);
    }

    if (transaction.creatorKey?.userId !== user?.id) {
      throw new UnauthorizedException('Only the creator has access to this transaction');
    }

    return transaction;
  }
