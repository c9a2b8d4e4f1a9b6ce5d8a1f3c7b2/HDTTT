# Audit Report

## Title
`archiveTransaction` Incorrect Guard Condition Allows Creator to Archive Transactions in Terminal States via `isManual` Bypass

## Summary
The `archiveTransaction` function in `transactions.service.ts` uses a logically flawed compound guard that short-circuits the status check for any manual transaction. A creator who set `isManual: true` at creation time can call `PATCH /transactions/archive/:id` on a transaction in any status — including `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED` — forcing a backward state transition that the state machine is supposed to prohibit.

## Finding Description
The guard at lines 711–718 reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The throw fires only when **both** conditions are true: status is outside `[WFS, WFE]` **and** `isManual` is `false`. By De Morgan's law the **allow** condition is:

> `(status ∈ {WFS, WFE})` **OR** `(isManual === true)`

The second branch is an unconditional bypass. When `isManual` is `true`, the status check is never evaluated and execution falls through directly to:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [1](#0-0) 

The function's own comment — *"Archive the transaction if the transaction is sign only"* — confirms the intent is to restrict archiving to active, unsigned states, not to permit it unconditionally for manual transactions. [2](#0-1) 

The `isManual: true` flag is set by the creator at transaction creation time with no additional authorization, as confirmed by the e2e test: [3](#0-2) 

## Impact Explanation
1. **Audit trail corruption**: The database status transitions from `EXECUTED` → `ARCHIVED`. The Hedera network executed the transaction, but the system now records it as archived (never executed). Any downstream reporting, compliance, or history view reflects the falsified state.
2. **Misleading notifications**: `emitTransactionStatusUpdate` is called at line 721, broadcasting the false status change to all signers, approvers, and observers.
3. **Unrecoverable state inconsistency**: There is no reverse operation. Once archived, the record permanently diverges from on-chain reality. [4](#0-3) 

## Likelihood Explanation
- The attacker only needs to be an authenticated, verified user — the lowest privilege level in the system.
- `isManual: true` requires no additional authorization beyond normal transaction creation.
- The attack is a single authenticated `PATCH /transactions/archive/:id` request after the transaction reaches a terminal state.
- No race condition, timing dependency, or external oracle is required.

## Recommendation
Replace the compound boolean guard with an explicit status allowlist that applies unconditionally, regardless of `isManual`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!archivableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions are intentionally permitted to be archived from additional states (e.g., `WAITING_FOR_EXECUTION` only), those states should be explicitly enumerated in the allowlist rather than using `isManual` as a bypass.

## Proof of Concept
1. Authenticated user creates a transaction with `isManual: true`:
   ```
   POST /transactions
   { ..., isManual: true }
   → 201 { id: 42, status: "WAITING_FOR_SIGNATURES" }
   ```
2. The transaction is signed, executed on Hedera, and its status transitions to `EXECUTED`.
3. Creator calls:
   ```
   PATCH /transactions/archive/42
   ```
4. Guard evaluation: `status = EXECUTED` (not in `[WFS, WFE]`) → first condition is `true`; `isManual = true` → `!isManual = false` → second condition is `false`. Combined: `true && false = false` → **no exception thrown**.
5. `repo.update({ id: 42 }, { status: ARCHIVED })` executes.
6. Database now shows `ARCHIVED` for a transaction that was `EXECUTED` on-chain. `emitTransactionStatusUpdate` broadcasts the false state to all participants.
