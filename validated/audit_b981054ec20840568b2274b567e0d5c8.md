### Title
Cross-Transaction Approver Removal via Missing Ownership Validation in `removeTransactionApprover`

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of `:transactionId`, but then removes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can remove approvers from any other transaction in the system, bypassing the approval gate entirely.

### Finding Description

**Vulnerability class**: Authorization bypass / improper ownership validation (direct analog to the external report's "Position Collateral Zeroing" — a resource-action mismatch where the authorization check is performed on one entity while the destructive action is performed on a different, unrelated entity).

**Root cause — controller** (`back-end/apps/api/src/transactions/approvers/approvers.controller.ts`):

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of URL transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver `id` with NO cross-check
  return true;
}
``` [1](#0-0) 

**Root cause — service** (`back-end/apps/api/src/transactions/approvers/approvers.service.ts`):

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id); // fetches by id only

``` [2](#0-1) 

**Contrast with the correctly-protected `updateTransactionApprover`**, which explicitly validates that the approver's root transaction matches the URL parameter and that the user owns it:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, ...);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, ...);
``` [3](#0-2) 

The `removeTransactionApprover` path has no equivalent guard. The `getCreatorsTransaction` call in the controller only validates the URL's `:transactionId`, not the approver's actual owning transaction. [4](#0-3) 

### Impact Explanation

An attacker who removes all approvers from a victim's transaction eliminates the approval gate. The transaction can then proceed to `WAITING_FOR_EXECUTION` and be submitted to the Hedera network without any approver having signed off. For organizations using the approval workflow to enforce governance (e.g., multi-party authorization for large HBAR transfers, account key changes, or node operations), this completely bypasses that control. The impact is unauthorized state change / privilege escalation — a transaction executes without the required human approvals.

### Likelihood Explanation

**Attacker preconditions (no privilege required beyond a normal account):**
- Must be a registered user of the organization (standard authenticated user).
- Must have created at least one transaction (to pass `getCreatorsTransaction` on their own transaction).
- Must know a target approver's integer `id`. Approver IDs are sequential database integers; an attacker can enumerate them by creating their own approvers and observing the returned IDs, or by calling `GET /transactions/:id/approvers` on any transaction they have read access to.

The attack requires only two API calls and is fully deterministic. No race condition, no cryptographic break, no privileged role needed.

### Recommendation

In `removeTransactionApprover` (service), after fetching the approver, traverse to its root node and verify both that the root belongs to `transactionId` and that the requesting user is the creator — exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // ← ADD: same ownership check as updateTransactionApprover
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
  await this.getCreatorsTransaction(rootNode.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: rootNode.transactionId }]);
  return result;
}
```

The controller's redundant `getCreatorsTransaction` call can then be removed, or kept as a fast-fail pre-check.

### Proof of Concept

**Setup:**
- User A creates transaction `T_A` (id=1) — attacker-controlled.
- User B creates transaction `T_B` (id=2) with a required approver (approver id=5, userId=99).

**Attack:**
```
DELETE /transactions/1/approvers/5
Authorization: Bearer <User A token>
```

**Step-by-step execution:**
1. `getCreatorsTransaction(1, UserA)` → passes (UserA is creator of T_A).
2. `removeTransactionApprover(5)` → fetches approver id=5 (belongs to T_B), calls `removeNode(5)`.
3. Approver id=5 is deleted from T_B's approver tree.
4. T_B now has no approvers; the approval gate is gone.
5. T_B transitions to `WAITING_FOR_EXECUTION` on the next scheduler tick and is submitted to Hedera without any approval.

**Expected (correct) behavior:** Step 1 should fail or Step 2 should throw `ROOT_TRANSACTION_NOT_SAME` because approver 5 belongs to T_B, not T_A.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
