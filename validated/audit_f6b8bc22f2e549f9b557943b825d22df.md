## Audit Report

## Title
Balance Validation Check Bypassed in Organization Mode for HBAR Transfer Transactions

## Summary
In `TransferHbar.vue`, the computed property `createDisabled` gates the per-account balance guard on `user.selectedOrganization === null`. When operating in Organization mode the guard is silently dropped. Neither `preCreateAssert` nor the back-end `validateAndPrepareTransaction` compensates for the missing check. Any authenticated organization member can therefore submit a Transfer HBAR transaction whose debit amount exceeds the sender's on-chain balance, causing the transaction to fail on-chain after all required co-signers have already reviewed and signed it and after the fee-payer account has been charged the Hedera network fee.

## Finding Description

**Root cause — client-side guard is mode-gated:**

`createDisabled` in `front-end/src/renderer/components/Transaction/Create/TransferHbar/TransferHbar.vue` reads:

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    (user.selectedOrganization === null && anyTransfersExceedingBalance.value)
  );
});
``` [1](#0-0) 

The fourth condition is wrapped in `user.selectedOrganization === null`, so `anyTransfersExceedingBalance` is never evaluated when the user is in Organization mode. The "Sign & Submit" button remains enabled regardless of how large the debit is.

`anyTransfersExceedingBalance` correctly computes whether any debit leg exceeds the mirror-node balance for that account: [2](#0-1) 

**`preCreateAssert` does not fill the gap:**

`preCreateAssert` only validates that the net sum of all transfers equals zero and that the adjustment count is within bounds. There is no per-account balance check: [3](#0-2) 

**Back-end `validateAndPrepareTransaction` does not fill the gap:**

The server-side validation checks signature validity, expiry, size, and node membership — but performs no balance check against the Hedera mirror node or any other source: [4](#0-3) 

**Exploit flow:**

1. Attacker is a legitimate organization member (no privileged keys required).
2. Attacker opens the Transfer HBAR form in Organization mode.
3. Attacker sets a debit of, e.g., 1,000,000 HBAR from an account holding 1 HBAR. The "Sign & Submit" button remains enabled because `anyTransfersExceedingBalance` is not evaluated in Organization mode.
4. Attacker submits the transaction to the back-end. It passes all server-side checks (signature, expiry, size, nodes).
5. All required co-signers receive notifications, review, and sign the transaction.
6. The chain service submits the signed transaction to the Hedera network.
7. Hedera rejects it with `INSUFFICIENT_ACCOUNT_BALANCE`; the fee-payer account is charged the network fee.
8. The transaction is recorded as `FAILED` in the back-end.

## Impact Explanation

- **Fee drain:** Every failed submission charges the fee-payer account a Hedera network fee. An attacker who repeatedly creates such transactions can drain the organization's fee-payer account of cumulative amounts.
- **Workflow disruption / DoS:** Every invalid transaction triggers the full multi-signature collection workflow. Co-signers receive notifications, spend time reviewing, and sign transactions that are guaranteed to fail. At scale this degrades the organization's ability to process legitimate transactions.
- **Trust erosion:** Repeated failed transactions attributed to a member's submissions corrupt audit logs and erode trust in the platform's integrity checks.

The Hedera network itself prevents actual fund theft (it enforces balance at the protocol layer), so the severity is **Medium** — real but bounded impact.

## Likelihood Explanation

Any authenticated organization member can trigger this with zero special privileges and zero cryptographic capability beyond their normal membership. The attacker-controlled entry point is the standard Transfer HBAR UI form in Organization mode. No social engineering or insider access beyond normal membership is required. Likelihood is **High**.

## Recommendation

Remove the `user.selectedOrganization === null` condition from `createDisabled` so that `anyTransfersExceedingBalance` is evaluated unconditionally in both personal and organization modes:

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    anyTransfersExceedingBalance.value   // ← remove the mode gate
  );
});
```

Additionally, add a corresponding per-account balance assertion in `preCreateAssert` so that the guard is enforced even if the UI state is somehow bypassed:

```js
if (anyTransfersExceedingBalance.value) {
  throw new Error('One or more debit amounts exceed the sender account balance');
}
```

For defense-in-depth, consider adding a mirror-node balance check in `validateAndPrepareTransaction` on the back-end for `TransferTransaction` types, so that the server rejects over-balance transfers before they enter the co-signing workflow.

## Proof of Concept

1. Log in as a legitimate organization member in an organization with at least one linked account holding a small HBAR balance (e.g., 1 HBAR).
2. Navigate to **Create Transaction → Transfer HBAR** while in Organization mode (`user.selectedOrganization !== null`).
3. Add a debit leg from the linked account for an amount far exceeding its balance (e.g., 1,000,000 HBAR) and a matching credit leg to any recipient account.
4. Observe that the "Sign & Submit" button is **enabled** despite the debit exceeding the account balance. In personal mode, the same configuration disables the button.
5. Submit the transaction. Confirm it passes all back-end validation and enters the co-signing workflow.
6. After all required signers sign, the transaction is submitted to Hedera and returns `INSUFFICIENT_ACCOUNT_BALANCE`. The fee-payer account is charged the network fee and the transaction is recorded as `FAILED`.

const anyTransfersExceedingBalance = computed(() => {
  let result = false;
  for (const transfer of data.transfers) {
    if (transfer.amount.isNegative()) {
      const accountInfo = accountInfos.value[transfer.accountId.toString()];
      if (

        transfer.amount.negated().toBigNumber().isGreaterThan(accountInfo.balance.toBigNumber())
      ) {
        result = true;

  private async validateAndPrepareTransaction(
    dto: CreateTransactionDto,
    user: User,
    client: Client,
  ) {
    const creatorKey = user.keys.find(key => key.id === dto.creatorKeyId);

    if (!creatorKey) {
      throw new BadRequestException(`Creator key ${dto.creatorKeyId} not found`);
    }

    const publicKey = PublicKey.fromString(creatorKey.publicKey);

    // Verify signature
    const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
    if (!validSignature) {
      throw new BadRequestException(ErrorCodes.SNMP);
    }

    // Parse SDK transaction
    const sdkTransaction = SDKTransaction.fromBytes(dto.transactionBytes);

    // Check the transaction is frozen, cannot require it to be frozen, breaks backwards compatibility
    if (!sdkTransaction.isFrozen()) {
      sdkTransaction.freezeWith(client);
    }

    // Check if expired
    if (isExpired(sdkTransaction)) {
      throw new BadRequestException(ErrorCodes.TE);
    }

    // Check size
    if (isTransactionBodyOverMaxSize(sdkTransaction)) {
      throw new BadRequestException(ErrorCodes.TOS);
    }

    // Check nodes
    const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
    if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
      throw new BadRequestException(ErrorCodes.TNVN);
    }

    const transactionHash = await sdkTransaction.getTransactionHash();
    const transactionType = getTransactionTypeEnumValue(sdkTransaction);

    // Extract new keys if applicable
    let publicKeys: string[] | null = null;
    try {
      let keyToExtract: Key | null = null;

      if (transactionType === TransactionType.ACCOUNT_UPDATE) {
        keyToExtract = (sdkTransaction as AccountUpdateTransaction).key;
      } else if (transactionType === TransactionType.NODE_UPDATE) {
        keyToExtract = (sdkTransaction as NodeUpdateTransaction).adminKey;
      } else if (transactionType === TransactionType.NODE_CREATE) {
        keyToExtract = (sdkTransaction as NodeCreateTransaction).adminKey;
      }

      if (keyToExtract) {
        publicKeys = flattenKeyList(keyToExtract).map(pk => pk.toStringRaw());
      }
    } catch (error) {
      // Log but don't fail - publicKeys will remain null
      console.error(`Failed to extract public keys from transaction ${sdkTransaction.transactionId}:`, error);
    }

    return {
      name: dto.name,
      type: transactionType,
      description: dto.description,
      transactionId: sdkTransaction.transactionId.toString(),
      transactionHash: encodeUint8Array(transactionHash),
      transactionBytes: sdkTransaction.toBytes(),
      unsignedTransactionBytes: sdkTransaction.toBytes(),
      creatorKeyId: dto.creatorKeyId,
      signature: dto.signature,
      mirrorNetwork: dto.mirrorNetwork,
      validStart: sdkTransaction.transactionId.validStart.toDate(),
      isManual: dto.isManual,
      cutoffAt: dto.cutoffAt,
      publicKeys,
    };
  }
