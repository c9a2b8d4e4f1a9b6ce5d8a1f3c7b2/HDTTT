### Title
Cross-Transaction Approver Deletion: Any Transaction Creator Can Remove Approvers from Transactions They Do Not Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController` verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then passes the approver `:id` directly to `removeTransactionApprover()` without verifying that the approver belongs to that same transaction. Any verified user who owns at least one transaction can delete approvers from any other transaction in the system, bypassing the multi-signature approval workflow.

### Finding Description

**Root cause — missing cross-ownership check between `transactionId` and approver `id`:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations: [1](#0-0) 

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the transaction at `:transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver record identified by `:id`.

There is **no check** that the approver `id` belongs to `transactionId`. The service function `removeTransactionApprover` only checks that the approver exists: [2](#0-1) 

It fetches the approver by `id` and deletes it unconditionally. The `transactionId` URL parameter is consumed only by the creator-check and is never used to scope the deletion.

**Exploit path:**

1. Attacker (verified user) creates their own transaction → becomes creator of `transactionId=1`.
2. Victim has a separate transaction `transactionId=2` with a required approver record `id=99`.
3. Attacker calls: `DELETE /transactions/1/approvers/99`
4. `getCreatorsTransaction(1, attacker)` → passes (attacker owns tx 1).
5. `removeTransactionApprover(99)` → deletes approver 99 from tx 2 with no further check.

The `updateTransactionApprover` path correctly validates the root transaction matches the URL parameter (line 390 of `approvers.service.ts`), but `removeTransactionApprover` has no equivalent guard. [3](#0-2) 

### Impact Explanation

An attacker can silently remove required approvers from any transaction they do not own. This directly undermines the multi-signature approval workflow: a transaction that required N approvals can have its approvers stripped, potentially allowing it to proceed with fewer approvals than intended, or disrupting the workflow of other organizations' transactions. This constitutes unauthorized state mutation of another user's transaction data.

**Impact: High** — approval integrity is a core security invariant of the system.

### Likelihood Explanation

The attacker only needs to be a verified user (status `UserStatus.NONE`) and the creator of any one transaction. Both conditions are reachable by any normal registered user with no privileged access. The approver `id` values are sequential integers, trivially enumerable. No leaked secrets or admin keys are required.

**Likelihood: Medium** — requires a valid account and one self-owned transaction, both achievable by a normal user.

### Recommendation

Before calling `removeTransactionApprover(id)`, verify that the approver identified by `id` belongs to `transactionId`. The simplest fix is to fetch the approver first and assert `approver.transactionId === transactionId` (or that its root node's `transactionId` matches), rejecting the request if not. The `updateTransactionApprover` path already demonstrates this pattern at line 390 and should be replicated here:

```typescript
// In removeTransactionApprover controller handler:
await this.approversService.getCreatorsTransaction(transactionId, user);
const approver = await this.approversService.getTransactionApproverById(id);
if (!approver) throw new BadRequestException(ErrorCodes.ANF);
const root = await this.approversService.getRootNodeFromNode(approver.id);
if (root?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
await this.approversService.removeTransactionApprover(id);
```

### Proof of Concept

**Preconditions:**
- Attacker is a verified user (status `NONE`) with a valid JWT.
- Attacker has created transaction with `id=1` (so they pass the creator check).
- Victim's transaction `id=2` has an approver record with `id=99`.

**Steps:**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>
```

**Expected (correct) behavior:** 403/401 — approver 99 does not belong to transaction 1.

**Actual behavior:** 200 — approver 99 is soft-deleted from transaction 2. The victim's approval workflow is silently broken.

**Code path:**
1. `ApproversController.removeTransactionApprover(user, transactionId=1, id=99)` [4](#0-3) 
2. `getCreatorsTransaction(1, attacker)` → passes. [5](#0-4) 
3. `removeTransactionApprover(99)` → deletes approver 99 (belonging to tx 2) with no ownership check. [2](#0-1)

```typescript
  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
```

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
