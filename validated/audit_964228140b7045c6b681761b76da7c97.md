Audit Report

## Title
User Key Soft-Deletion Without Checking Active Approver References Disrupts Pending Transaction Approval Workflows

## Summary
When a `UserKey` is soft-deleted via `removeUserKey` (user self-service) or `removeUser` (admin), no check is performed against pending `TransactionApprover` records. After deletion, `approveTransaction` silently returns `false` when the approver's keys are gone, leaving the `TransactionApprover` row with `signature = null` and `approved = null` and requiring manual creator intervention to unblock the transaction.

## Finding Description

**Root cause — no reference check on key deletion:**

`removeUserKey` in `back-end/apps/api/src/user-keys/user-keys.service.ts` soft-removes the key with no query against `TransactionApprover`: [1](#0-0) 

`removeUser` in `back-end/apps/api/src/users/users.service.ts` bulk-soft-deletes all keys for a user, again with no check for pending approvals: [2](#0-1) 

**How the approval flow breaks after key deletion:**

`attachKeys` in `back-end/libs/common/src/utils/user/index.ts` fetches keys using a plain `find` with no `withDeleted: true`, so soft-deleted keys are excluded: [3](#0-2) 

`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` calls `attachKeys` and silently returns `false` at line 567 if `user.keys` is empty — no error, no state change, no notification: [4](#0-3) 

The `TransactionApprover` row with the deleted user's `userId` remains in the database with `signature = null` and `approved = null`. The transaction cannot advance past `WAITING_FOR_SIGNATURES` until the creator manually intervenes.

**The `UserKey` entity confirms the relationship:**

The `approvedTransactions` one-to-many relation shows that a single `UserKey` is the designated signing key for potentially many `TransactionApprover` records: [5](#0-4) 

**Notification service confirms soft-deleted keys are excluded from required signers:** [6](#0-5) 

This confirms the system intentionally excludes soft-deleted keys from signing requirements but does not cascade that exclusion back to unblock the stuck `TransactionApprover` record.

## Impact Explanation

A transaction in `WAITING_FOR_SIGNATURES` that requires approval from a user whose key has been soft-deleted is blocked. No signature can be recorded for that approver slot. The `approveTransaction` endpoint returns HTTP 200 with `false` — no error surfaced to the caller. The transaction creator must manually discover the issue and either call `removeTransactionApprover` or `updateTransactionApprover` to replace the approver. Until that manual remediation occurs, the transaction cannot advance. If the creator is unaware of the cause, the transaction may remain stuck indefinitely. Note: the impact is not strictly permanent — the creator retains the ability to remove or replace the approver, and the user can re-upload their key to recover it — but the silent failure and lack of any notification make discovery and remediation non-obvious. [7](#0-6) 

## Likelihood Explanation

The attack requires only a normal authenticated user who has been added as a transaction approver. The user calls `DELETE /users/:userId/keys/:keyId` — a standard self-service endpoint — after being assigned as an approver. No admin access, no leaked credentials, and no special timing is required. The attacker simply deletes their own key. This is a realistic, low-effort action by a malicious or uncooperative approver. [1](#0-0) 

## Recommendation

1. **Pre-deletion check**: Before soft-deleting a `UserKey`, query `TransactionApprover` for any records where `userKeyId = key.id` OR `userId = user.id` and `signature IS NULL` and `deletedAt IS NULL`. If any exist, either reject the deletion with an error or automatically nullify/remove those approver records and notify the transaction creator.
2. **Fix silent failure**: In `approveTransaction`, replace the silent `return false` when `user.keys.length === 0` with a `BadRequestException` (e.g., `ErrorCodes.KNF` or a dedicated error code) so the caller receives a meaningful error rather than a silent no-op.
3. **Cascade on deletion**: When a key is soft-deleted, emit a notification or event to inform transaction creators of any affected pending approvals so they can take remediation action promptly.

## Proof of Concept

```
1. Admin creates transaction T1 requiring approval from User A (userId=2).
   → TransactionApprover row: { transactionId: T1.id, userId: 2, signature: null, approved: null }

2. User A (userId=2) calls:
   DELETE /users/2/keys/5
   → removeUserKey executes repo.softRemove(userKey) with no check on TransactionApprover.
   → user_key row id=5: deletedAt = NOW()

3. User A attempts to approve:
   POST /transactions/T1/approvers/approve  { userKeyId: 5, signature: "...", approved: true }
   → approveTransaction calls attachKeys(user, manager)
   → attachKeys: entityManager.find(UserKey, { where: { userId: 2 } })
     → returns [] because deletedAt IS NOT NULL (soft-deleted, excluded by default)
   → user.keys.length === 0 → return false  (HTTP 200, body: false)
   → TransactionApprover row unchanged: signature=null, approved=null

4. Transaction T1 remains in WAITING_FOR_SIGNATURES.
   No error is raised, no notification is sent.
   Creator must manually discover the issue and call removeTransactionApprover or updateTransactionApprover.
``` [8](#0-7)

  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }
