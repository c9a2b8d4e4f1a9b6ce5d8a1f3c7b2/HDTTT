### Title
Skipped E2E Tests Leave File Transaction Types Untested in Organization/Complex-Key Mode

### Summary
The external report describes a test suite where certain transaction types are gated or skipped for a specific protocol version, leaving real execution paths unvalidated. The same pattern exists in this repository: multiple Playwright E2E tests covering `FileCreate`, `FileUpdate`, and `FileAppend` transactions with complex-key accounts in Organization Mode are permanently skipped via `test.skip()`, and the entire regression suite and all TTv1→TTv2 compatibility tests are also skipped. These are not conditional skips based on environment — they are unconditional, meaning these code paths are never exercised in CI.

### Finding Description
Three transaction-type tests in `organizationTransactionExecutionTests.test.ts` are unconditionally skipped:

- `test.skip('Verify user can execute file create with complex account'` [1](#0-0) 
- `test.skip('Verify user can execute file update with complex account'` [2](#0-1) 
- `test.skip('Verify user can execute file append with complex account'` [3](#0-2) 

The entire TTv1→TTv2 signature compatibility suite (3 tests) is also skipped: [4](#0-3) 

The entire regression suite (`test.describe.skip`) covering complex-key account update and transfer with 57-user threshold keys is also permanently disabled: [5](#0-4) 

The system has full backend model implementations for all three file transaction types: [6](#0-5) [7](#0-6) [8](#0-7) 

These transaction types are fully supported in production but their multi-signature collection, threshold evaluation, and execution flows in Organization Mode are never validated end-to-end.

### Impact Explanation
Bugs in the multi-signature workflow for `FileCreate`, `FileUpdate`, and `FileAppend` transactions — such as incorrect threshold key evaluation, missing signature propagation, or wrong status transitions — would not be caught by the test suite. A user relying on Organization Mode to collaboratively manage file transactions could silently experience incorrect behavior (e.g., a transaction marked ready for execution without sufficient signatures, or a valid transaction rejected). [1](#0-0) 

### Likelihood Explanation
The skips are unconditional — not environment-gated or feature-flagged. They will never run in CI. The file transaction types are fully implemented and exposed in the UI, meaning real users exercise these paths in production while the test suite provides no regression protection for them. [9](#0-8) 

### Recommendation
Remove the `test.skip()` annotations from the three file transaction tests and the TTv1→TTv2 compatibility tests, and remove `test.describe.skip` from the regression suite. If these tests are skipped due to infrastructure constraints (e.g., cost or flakiness), document the reason explicitly and create a tracking issue. At minimum, add unit-level tests for the `FileCreate`, `FileUpdate`, and `FileAppend` signature-collection and threshold-evaluation logic in Organization Mode to provide some regression coverage. [5](#0-4) 

### Proof of Concept
1. In `organizationTransactionExecutionTests.test.ts`, lines 94, 109, and 133 each begin with `test.skip(...)` — these tests will never execute in any test run. [9](#0-8) 
2. In `organizationTransactionCompatibilityTests.test.ts`, lines 118, 160, and 202 are all `test.skip(...)` — the entire TTv1→TTv2 compatibility surface is untested. [10](#0-9) 
3. In `organizationRegressionTests.test.ts`, line 34 wraps the entire suite in `test.describe.skip(...)`, disabling all regression tests for complex-key accounts. [11](#0-10) 
4. The backend models for `FileCreate`, `FileUpdate`, and `FileAppend` exist and are production-active, confirming these are real, reachable code paths with no test coverage in the organization multi-sig context. [6](#0-5)
