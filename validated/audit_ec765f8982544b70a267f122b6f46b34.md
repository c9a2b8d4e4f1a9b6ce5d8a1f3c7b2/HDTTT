Audit Report

## Title
Any Verified Organization Member Can Read All Other Users' `mnemonicHash` and `index` via Unguarded Bulk Key Endpoint

## Summary
The `GET /user-keys` endpoint (`UserKeysAllController`) returns every `UserKey` record in the database — including the sensitive `mnemonicHash` and `index` fields — to any authenticated, verified user. The parallel per-user endpoint (`GET /user/:userId/keys`) correctly strips those fields for non-owners, but the bulk endpoint bypasses that restriction entirely.

## Finding Description

**Two code paths with inconsistent access control:**

The per-user endpoint (`UserKeysController.getUserKeys`) calls `getUserKeysRestricted`, which uses a TypeORM `select` projection to conditionally hide `mnemonicHash` and `index` when the requester is not the owner: [1](#0-0) 

The bulk endpoint (`UserKeysAllController`) calls `getUserKeys` instead: [2](#0-1) 

`getUserKeys` issues a plain `findAndCount` with **no `select` projection and no `where` filter**, returning all columns for all users: [3](#0-2) 

The serialization DTO exposes both sensitive fields unconditionally via `@Expose()`: [4](#0-3) 

The only guard on the bulk endpoint is `VerifiedUserGuard` — any verified organization member satisfies it: [5](#0-4) 

**What the exposed fields reveal:**

`mnemonicHash` is the hash of a user's BIP-39 recovery phrase stored server-side to correlate keys to mnemonics during key recovery: [6](#0-5) 

`index` is the HD derivation index used alongside the mnemonic to reconstruct the private key: [7](#0-6) 

The e2e test suite confirms the per-user endpoint correctly hides these fields for non-owners, but no equivalent test exists for the bulk endpoint: [8](#0-7) 

## Impact Explanation
- **Cross-user sensitive metadata exposure**: every verified organization member can read the `mnemonicHash` and `index` of every other member's keys in a single paginated request.
- **Key-correlation attack**: an attacker learns which of a victim's keys share the same mnemonic (same `mnemonicHash`), reducing the search space for mnemonic brute-force from the full BIP-39 space to a targeted attack against one known mnemonic.
- **Full organization key enumeration**: the paginated endpoint leaks `userId` alongside the key metadata, giving the attacker a complete mapping of users to their key derivation parameters.

## Likelihood Explanation
- **Attacker precondition**: a valid organization account with verified status — the lowest privilege level in the system.
- **Exploit complexity**: a single authenticated HTTP `GET /user-keys?page=1&size=100` with the attacker's own JWT. No special tooling required.
- The front-end itself already calls this endpoint in a loop to harvest all keys: [9](#0-8) 

## Recommendation
Apply the same ownership-aware field restriction used in `getUserKeysRestricted` to the bulk endpoint. Specifically, `getUserKeys` should accept the requesting user as a parameter and apply a `select` projection that sets `mnemonicHash` and `index` to `false` for rows where `userId !== requestingUser.id`. Alternatively, if the bulk endpoint is intended only for administrative use, restrict it with an admin guard rather than `VerifiedUserGuard`.

## Proof of Concept
```
# 1. Obtain a JWT for any verified (non-admin) organization member
POST /auth/login  { "email": "member@org.com", "password": "..." }
# → { "token": "<JWT>" }

# 2. Harvest all user keys including mnemonicHash and index
GET /user-keys?page=1&size=100
Authorization: Bearer <JWT>

# Response includes for every user in the organization:
# { "id": 3, "userId": 7, "publicKey": "...", "mnemonicHash": "0xabc...", "index": 2 }
# { "id": 4, "userId": 8, "publicKey": "...", "mnemonicHash": "0xdef...", "index": 0 }
# ...

# 3. Use leaked mnemonicHash to identify which keys share the same mnemonic,
#    then use leaked index to target the exact HD derivation path for offline attack.
```

```typescript
        id: true,

        index: user.id === userId,
```

```typescript
export const getAllUserKeys = async (organizationServerUrl: string): Promise<IUserKey[]> => {
  let page = 1;
  const size = 100;
  let totalItems = 0;
  const allUserKeys: IUserKey[] = [];

  do {
    const { data, error } = await safeAwait(
      getUserKeysPaginated(organizationServerUrl, page, size),
    );
    if (data) {
      totalItems = data.totalItems;
      allUserKeys.push(...data.items);
      page++;
    }

    if (error) {
      break;
    }
  } while (allUserKeys.length < totalItems);

  return allUserKeys;
```
