### Title
Unbounded Recursive Iteration in `createTransactionApprovers` Enables Authenticated DoS via Exhaustive DB Operations

### Summary
The `POST /transactions/:id/approvers` endpoint accepts a deeply nested, arbitrarily large `approversArray` with no maximum-size constraint. The internal `createApprover` function is recursive and executes 5–6 sequential database queries per node. An authenticated transaction creator can craft a wide or deeply nested approver tree that triggers thousands of DB operations inside a single long-running transaction, exhausting database connections and CPU, and causing service degradation or unavailability for all other users.

### Finding Description

**Root cause — missing `@ArrayMaxSize` on both DTO fields:**

`CreateTransactionApproverDto` and `CreateTransactionApproversArrayDto` enforce only a minimum size (`@ArrayMinSize(1)`) but impose no upper bound on either `approversArray` or the recursive `approvers` field. [1](#0-0) 

**Unbounded recursive loop with per-node DB I/O:**

Inside `createTransactionApprovers`, the inner async function `createApprover` is called once per root entry in `dto.approversArray` and then calls itself for every element of `dtoApprover.approvers`. Each invocation performs:

1. `isNode` check (DB query)
2. Parent lookup (`findOne`)
3. `getRootNodeFromNode` traversal (recursive CTE query)
4. `count(User)` existence check
5. `getApproversByTransactionId` (recursive CTE query)
6. `insert(TransactionApprover)` (DB write)

All of this runs inside a single `dataSource.transaction(...)` block, holding a DB connection open for the entire duration. [2](#0-1) 

The outer loop that starts the recursion: [3](#0-2) 

**Body size is the only guard — and it is insufficient:**

The only server-wide protection is a 2 MB JSON body limit. A 2 MB payload can encode thousands of nested approver nodes (each node is a small JSON object), producing an exponential number of DB operations. [4](#0-3) 

**Secondary surface — observers endpoint:**

`CreateTransactionObserversDto` also lacks `@ArrayMaxSize`. The loop at line 49 of `observers.service.ts` iterates over every supplied `userId` in memory before a batch save. This is less severe than the approvers path but still allows a single request to create an arbitrarily large in-memory array. [5](#0-4) [6](#0-5) 

### Impact Explanation

A single crafted request can hold a PostgreSQL connection open for minutes while executing thousands of recursive CTE queries and inserts. With a small number of concurrent such requests, the database connection pool is exhausted, causing all other API operations (authentication, transaction reads, signature submissions) to queue and time out. This constitutes a full service denial for all organization users. The impact is permanent until the attacker stops sending requests; no admin action can recover the pool while the long transaction is running.

### Likelihood Explanation

The attacker must be an authenticated user who is the creator of at least one transaction — a condition reachable by any registered organization member. No elevated privilege is required. The attack requires only a single HTTP request with a crafted JSON body, which any HTTP client can produce. There is no rate limiting visible in the codebase. The attack is therefore trivially repeatable.

### Recommendation

1. Add `@ArrayMaxSize(N)` to both `approversArray` in `CreateTransactionApproversArrayDto` and `approvers` in `CreateTransactionApproverDto`. A reasonable bound (e.g., 50–100) matches real-world multi-sig use cases.
2. Add `@ArrayMaxSize(N)` to `userIds` in `CreateTransactionObserversDto`.
3. Enforce a maximum recursion depth inside `createApprover` (e.g., reject trees deeper than 5 levels).
4. Consider adding per-user or per-endpoint rate limiting at the API gateway or NestJS middleware layer.

### Proof of Concept

**Attacker steps:**

1. Register or log in as a normal user.
2. Create a transaction (obtain `transactionId`).
3. Send the following request (repeat concurrently 5–10 times to exhaust the DB pool):

```http
POST /transactions/1/approvers
Authorization: Bearer <valid_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        { "threshold": 1, "approvers": [
            { "threshold": 1, "approvers": [
                ... (repeat nesting to ~8 levels, each with 5 children)
            ]}
        ]}
      ]
    },
    ... (repeat root entries to fill ~1.9 MB)
  ]
}
```

**Expected outcome:** Each request triggers thousands of sequential recursive CTE queries and inserts inside a single DB transaction. The PostgreSQL connection pool is exhausted within seconds. Subsequent API calls from all users return 500/timeout errors until the attacker stops.
