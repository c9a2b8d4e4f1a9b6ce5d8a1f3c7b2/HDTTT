## Audit Report

## Title
Arbitrary `mirrorNetwork` URL Accepted Without Whitelist Enables SSRF via gRPC

## Summary
`CreateTransactionDto.mirrorNetwork` accepts any non-empty string with no allowlist constraint. Any authenticated user can supply an attacker-controlled host as the mirror network. The backend immediately makes an outbound gRPC `AddressBookQuery` connection to that host during transaction creation, and again at execution time via the chain service — constituting a persistent Server-Side Request Forgery (SSRF) vulnerability.

## Finding Description

**1. No allowlist on `mirrorNetwork` in `CreateTransactionDto`**

Only `@IsNotEmpty()` and `@IsString()` are applied. No `@IsEnum()`, `@IsIn()`, or URL-pattern constraint exists. [1](#0-0) 

**2. Raw user value passed directly to `getClientFromNetwork()` in `createTransactions()`**

The first DTO's `mirrorNetwork` is passed without any sanitization or validation to `getClientFromNetwork()`. [2](#0-1) 

**3. `getClientFromNetwork()` makes an outbound gRPC connection to any non-standard value**

When `mirrorNetwork` is not `mainnet`, `testnet`, `previewnet`, or `local-node`, the function constructs a gRPC client pointing at the attacker-supplied URL (via `MirrorNetworkGRPC.fromBaseURL`) and immediately executes an `AddressBookQuery` against it — an outbound network call to an arbitrary host. [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` confirms the default case appends `:443` and returns the raw attacker-supplied string as the gRPC endpoint: [4](#0-3) 

**4. Malicious value is persisted and re-triggered at execution time**

`transaction.mirrorNetwork` (the attacker-supplied value) is stored in the database and passed again to `getClientFromNetwork()` inside `_executeTransaction()` in the chain service, triggering a second outbound connection every time the chain service attempts to execute the transaction. [5](#0-4) 

**5. `ValidationPipe` with `whitelist: true` provides no protection**

`whitelist: true` only strips unknown DTO fields; it does not constrain the value of known string fields like `mirrorNetwork`. [6](#0-5) 

## Impact Explanation

1. **SSRF via gRPC** — the backend makes outbound gRPC connections to any `host:port` the attacker specifies. This allows probing internal services (Kubernetes sidecars, cloud metadata endpoints such as `169.254.169.254`, internal APIs) reachable from the backend pod but not from the public internet.
2. **Address-book poisoning** — if the attacker controls the gRPC endpoint and returns a crafted `NodeAddressBook`, `client.setNetworkFromAddressBook()` will configure the SDK client to route subsequent Hedera transactions to attacker-controlled nodes.
3. **Persistent trigger** — the malicious `mirrorNetwork` is stored in the database and re-triggered every time the chain service attempts to execute the transaction, multiplying SSRF calls without further attacker interaction. [7](#0-6) 

## Likelihood Explanation

- Requires only a valid authenticated session (normal user registration).
- No privileged keys, admin access, or leaked credentials needed.
- The attack path is a single API call (`POST /transactions`) with a crafted `mirrorNetwork` field.
- The `ValidationPipe` with `whitelist: true` strips unknown DTO fields but does **not** constrain the value of known string fields — so the guard provides no protection here. [1](#0-0) 

## Recommendation

Add an allowlist validator to `mirrorNetwork` in `CreateTransactionDto`. For example, using `class-validator`:

```typescript
import { IsIn, IsNotEmpty, IsString } from 'class-validator';

const ALLOWED_NETWORKS = ['mainnet', 'testnet', 'previewnet', 'local-node'];

@IsNotEmpty()
@IsString()
@IsIn(ALLOWED_NETWORKS)
mirrorNetwork: string;
```

If custom mirror networks are a required product feature, a separate admin-controlled allowlist should be maintained and validated against at the service layer before passing the value to `getClientFromNetwork()`.

## Proof of Concept

```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_user_jwt>
Content-Type: application/json

{
  "name": "SSRF Test",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<valid_hex>",
  "creatorKeyId": 1,
  "signature": "<valid_hex>",
  "mirrorNetwork": "169.254.169.254/latest/meta-data/iam/security-credentials/role"
}
```

Upon receipt, the backend calls `getClientFromNetwork("169.254.169.254/...")`, which reaches the `default` branch in `getClientFromNetwork()`, constructs a gRPC client with `MirrorNetworkGRPC.fromBaseURL("169.254.169.254/...")` as the mirror network, and executes `new AddressBookQuery().setFileId(FileId.ADDRESS_BOOK).execute(client)` — making an outbound gRPC connection to the attacker-specified host. The value is then persisted in `transaction.mirrorNetwork` and re-triggered on every execution attempt by the chain service. [3](#0-2) [5](#0-4)
