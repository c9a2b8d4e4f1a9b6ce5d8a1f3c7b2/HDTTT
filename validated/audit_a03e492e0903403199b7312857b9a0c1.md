Audit Report

## Title
Wrong Field Assignment in `handleContractOnBlur` Corrupts SystemDelete/SystemUndelete Form State

## Summary
In `SystemDataFormData.vue`, the `handleContractOnBlur` handler emits an update that writes to `fileId` instead of `contractId`, and reads from `props.data.fileId` instead of `props.data.contractId`. This means blurring the Contract ID input silently re-normalizes the File ID field (through the wrong formatter) while leaving the Contract ID field entirely un-normalized.

## Finding Description
In `front-end/src/renderer/components/Transaction/Create/SystemData/SystemDataFormData.vue` at line 23–25:

```typescript
function handleContractOnBlur() {
  emit('update:data', { ...props.data, fileId: formatContractId(props.data.fileId) });
  //                                   ^^^^^^ wrong key        ^^^^^^^^^^^^^^^^^ wrong source
}
```

The correct implementation should be:
```typescript
function handleContractOnBlur() {
  emit('update:data', { ...props.data, contractId: formatContractId(props.data.contractId) });
}
```

Two simultaneous wrong-field errors are present:
1. The emitted key is `fileId` instead of `contractId`
2. The source read is `props.data.fileId` instead of `props.data.contractId` [1](#0-0) 

The `SystemData` type has two distinct fields, `fileId` and `contractId`: [2](#0-1) 

`formatAccountId` and `formatContractId` are not equivalent — `formatAccountId` has an extra guard (`!/^0+/.test(accountId)`) that `formatContractId` does not: [3](#0-2) 

This component is shared by both `SystemDeleteFormData` and `SystemUndeleteFormData`: [4](#0-3) [5](#0-4) 

## Impact Explanation
When a user fills in the Contract ID field and blurs it:

1. **`contractId` is never normalized** — the raw user input remains in state, bypassing the `formatContractId` normalization step entirely.
2. **`fileId` is silently re-processed** through `formatContractId(props.data.fileId)` instead of `formatAccountId(props.data.fileId)`, applying the wrong normalization function to the File ID field.

The downstream `setSystemData` function uses both fields to build the on-chain transaction: [6](#0-5) 

The `createDisabled` guard in both `SystemDelete.vue` and `SystemUndelete.vue` checks `!isFileId(data.fileId) || !isContractId(data.contractId)`: [7](#0-6) [8](#0-7) 

Because `contractId` is never updated by the blur handler, any non-canonical contract ID entered by the user (e.g., `"123"` instead of `"0.0.123"`) is never normalized. If the raw input fails `isContractId`, the submit button is permanently disabled. Additionally, applying `formatContractId` to `fileId` instead of `formatAccountId` can produce a different normalized value for the File ID field, since `formatAccountId` has the `!/^0+/.test(accountId)` guard that `formatContractId` lacks — meaning IDs starting with `0` (the standard Hedera format) are handled differently by the two functions.

## Likelihood Explanation
This triggers on every normal user interaction: filling in the Contract ID field and clicking away is the standard workflow for SystemDelete and SystemUndelete transactions. No attacker involvement is required — any user of the tool who attempts these transaction types will encounter this bug.

## Recommendation
Fix `handleContractOnBlur` to use the correct field name and source:

```typescript
function handleContractOnBlur() {
  emit('update:data', { ...props.data, contractId: formatContractId(props.data.contractId) });
}
```

This mirrors the correct pattern already used in `handleFileOnBlur`: [9](#0-8) 

## Proof of Concept
1. Open the Hedera Transaction Tool and navigate to SystemDelete or SystemUndelete transaction creation.
2. Enter a valid File ID (e.g., `0.0.100`) in the File ID field and blur it — `handleFileOnBlur` fires correctly, emitting `{ fileId: formatAccountId("0.0.100"), contractId: "" }`.
3. Enter a valid Contract ID (e.g., `0.0.200`) in the Contract ID field and blur it — `handleContractOnBlur` fires and emits `{ fileId: formatContractId("0.0.100"), contractId: "" }` (the `contractId` key is never updated; `fileId` is re-processed through the wrong formatter).
4. Observe that `data.contractId` remains `""` (or whatever was previously set), causing `isContractId(data.contractId)` to return `false` and `createDisabled` to remain `true`, permanently disabling the submit button even though both IDs were validly entered. [10](#0-9)

```typescript
export const formatAccountId = (accountId: string) => {
  if (isAccountId(accountId) && !/^0+/.test(accountId)) {
    return AccountId.fromString(accountId).toString();
  } else {
    return accountId;
  }
};

export const formatContractId = (contractId: string) => {
  if (isContractId(contractId)) {
    return ContractId.fromString(contractId).toString();
  } else {
    return contractId;
  }
};
```
