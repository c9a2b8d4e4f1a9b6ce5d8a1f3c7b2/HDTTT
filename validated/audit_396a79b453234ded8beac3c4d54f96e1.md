## Audit Report

## Title
IDOR in `removeTransactionApprover`: Approver `id` Not Validated Against Authorized `transactionId`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller only against `:transactionId` but deletes the approver identified by `:id` without verifying that approver belongs to that transaction. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Controller — `approvers.controller.ts` lines 102–113:**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` confirms only that the caller is the creator of the URL-supplied `transactionId`. The approver `id` is then passed directly to `removeTransactionApprover` with no ownership check.

**Service — `approvers.service.ts` lines 533–544:**

```typescript

  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;

``` [2](#0-1) 

The service accepts only `id` and performs no cross-check between the approver's owning transaction and the transaction the caller was authorized against.

**Authorization guard — `approvers.service.ts` lines 623–644:** [3](#0-2) 

`getCreatorsTransaction` is scoped entirely to `transactionId` from the URL. The approver `id` being deleted is never validated against it.

**Contrast with `updateTransactionApprover`** — the sibling update path correctly performs the cross-check:

```typescript

if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [4](#0-3) 

This proves the pattern for the fix is already present in the codebase; it was simply omitted from the delete path.

## Impact Explanation
An attacker can silently remove any approver from any transaction in the system, including multi-signature governance transactions. This breaks the approval threshold, potentially allowing transactions to execute without the required number of approvals, or permanently corrupting the approval tree of critical transactions. The impact is unauthorized state mutation of other users' transaction approval workflows — a high-integrity violation.

## Likelihood Explanation
The attacker only needs to be a registered, authenticated user who has created at least one transaction. No admin privileges are required. Approver IDs are sequential integers, making enumeration trivial via `GET /transactions/:id/approvers` on any transaction the attacker has read access to, or by brute-force. The attack requires no special tooling — a single crafted HTTP DELETE request suffices.

## Recommendation
In `removeTransactionApprover` (controller), after fetching the approver, verify that its root transaction matches the authorized `transactionId` — mirroring the logic already present in `updateTransactionApprover`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  // Add: verify the approver belongs to transactionId
  await this.approversService.verifyApproverBelongsToTransaction(id, transactionId);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

In the service, implement `verifyApproverBelongsToTransaction` using the existing `getRootNodeFromNode` helper and throw `UnauthorizedException` (with `ROOT_TRANSACTION_NOT_SAME`) if the root's `transactionId` does not match — exactly as done in `updateTransactionApprover` at lines 386–391. [4](#0-3) 

## Proof of Concept

1. Attacker registers as a user and creates **Transaction A** (becomes its creator).
2. Attacker obtains approver ID `X` belonging to victim's **Transaction B** (via `GET /transactions/B/approvers` if they have read access, or by sequential integer brute-force).
3. Attacker sends:
   ```
   DELETE /transactions/A/approvers/X
   Authorization: Bearer <attacker_token>
   ```
4. `getCreatorsTransaction(A, attacker)` passes — attacker is creator of A.
5. `removeTransactionApprover(X)` executes — approver `X` from Transaction B is deleted with no error.
6. Transaction B's approval tree is now corrupted; its threshold may no longer be satisfiable or may be trivially met.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
