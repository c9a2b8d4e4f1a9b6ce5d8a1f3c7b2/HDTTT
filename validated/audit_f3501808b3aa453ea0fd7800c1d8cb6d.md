### Title
Any Authenticated User Can Remove Approvers from Transactions They Do Not Own via IDOR in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of `:transactionId`, but then removes the approver record identified by `:id` without verifying that the approver actually belongs to `:transactionId`. A regular authenticated user who owns any transaction can supply their own `transactionId` to pass the ownership check, then supply an approver `id` from a completely different transaction to delete it. No admin privilege is required.

### Finding Description

**Root cause:** In `ApproversController.removeTransactionApprover`, the authorization check and the deletion operate on two independent, unlinked identifiers.

```
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,  // attacker controls
  @Param('id', ParseIntPipe) id: number,                        // attacker controls
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId only
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id, no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only verifies `transaction.creatorKey?.userId === user.id` for the supplied `transactionId`: [2](#0-1) 

`removeTransactionApprover` then deletes the approver record identified by `id` with no check that it belongs to the verified transaction: [3](#0-2) 

By contrast, `updateTransactionApprover` correctly validates that the root transaction of the approver matches the URL `transactionId` parameter before proceeding: [4](#0-3) 

The delete path is missing this equivalent cross-check.

**Exploit flow:**
1. Attacker (regular authenticated user) creates any transaction — call it `TX_A`. They are now its creator.
2. Attacker learns an approver `id` from a victim's transaction `TX_B`. This is feasible because:
   - Approver IDs are sequential integers and can be enumerated.
   - If the attacker is an observer or signer of `TX_B`, they can call `GET /transactions/:TX_B_id/approvers` to retrieve the exact IDs.
3. Attacker sends: `DELETE /transactions/{TX_A_id}/approvers/{TX_B_approver_id}`
4. `getCreatorsTransaction(TX_A_id, attacker)` passes — attacker owns `TX_A`.
5. `removeTransactionApprover(TX_B_approver_id)` executes — the approver from `TX_B` is deleted.

### Impact Explanation
An attacker can silently remove required approvers from any transaction in the system. This breaks the multi-signature approval workflow: a transaction that required N-of-M approvals can have its approver list reduced, potentially allowing it to proceed to execution without the intended governance controls. This is a permanent state corruption of another user's transaction data, achievable by any authenticated user with no elevated privileges.

### Likelihood Explanation
The attacker only needs a valid JWT (any registered, verified user). Creating a transaction to obtain a valid `transactionId` is a normal product flow. Approver IDs are auto-increment integers and are discoverable by any user who shares observer or signer access on any transaction with the victim, or by enumeration. The endpoint is reachable without any special role.

### Recommendation
In `removeTransactionApprover` (or in the controller before calling it), verify that the approver record's root `transactionId` matches the URL `transactionId` parameter — exactly as `updateTransactionApprover` already does at line 390:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, entityManager);
if (rootNode?.transactionId !== transactionId)
  throw new UnauthorizedException('Approver does not belong to this transaction');
``` [4](#0-3) 

### Proof of Concept

**Preconditions:** Two registered users — Attacker and Victim. Victim has created `TX_B` with an approver (approver record `id = 42`). Attacker has created `TX_A` (approver record `id = 7`).

```
# Step 1: Attacker authenticates and gets JWT
POST /auth/login  { email: "attacker@org.com", password: "..." }
# → attacker_jwt

# Step 2: Attacker discovers approver id=42 on TX_B
# (e.g., attacker is an observer of TX_B, or enumerates IDs)
GET /transactions/{TX_B_id}/approvers  Authorization: attacker_jwt
# → [{ id: 42, userId: victim_approver_user_id, ... }]

# Step 3: Attacker deletes approver 42 using their own TX_A as the ownership anchor
DELETE /transactions/{TX_A_id}/approvers/42  Authorization: attacker_jwt
# → HTTP 200, true

# Result: approver id=42 is deleted from TX_B.
# TX_B's approval threshold is now unmet by a different set of approvers than intended.
```

Expected: HTTP 401 Unauthorized (approver does not belong to `TX_A`).
Actual: HTTP 200, approver deleted from `TX_B`.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
