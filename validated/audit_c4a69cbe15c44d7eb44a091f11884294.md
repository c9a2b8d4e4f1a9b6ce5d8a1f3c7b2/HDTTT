### Title
Creator of Any Transaction Can Delete Approvers Belonging to a Different Transaction

### Summary
The `removeTransactionApprover` endpoint verifies that the requesting user is the creator of the transaction identified by the URL parameter `transactionId`, but then deletes the approver record identified by the separate `id` parameter without confirming that approver belongs to that same transaction. Any verified user who has created at least one transaction can therefore delete approvers from transactions they do not own, bypassing approval requirements on those transactions.

### Finding Description

**Vulnerability class:** Authorization — missing cross-resource ownership check (analogous to the external report's dual-role authorization flaw, where one authorization check is performed on resource A while the destructive action is applied to resource B).

**Root cause:**

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```typescript
// Step 1 — verifies user is creator of :transactionId
await this.approversService.getCreatorsTransaction(transactionId, user);

// Step 2 — deletes approver by :id, with NO check that it belongs to :transactionId
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

`getCreatorsTransaction` only validates ownership of the transaction referenced by the URL parameter: [2](#0-1) 

`removeTransactionApprover` then deletes the approver record identified solely by `id`, with no check that `approver.transactionId === transactionId`: [3](#0-2) 

By contrast, `updateTransactionApprover` (the PATCH handler) correctly validates that the root node's `transactionId` matches the URL parameter before proceeding: [4](#0-3) 

The DELETE path omits this cross-check entirely.

**Exploit flow:**

1. Attacker (verified user) creates Transaction A — they become its creator.
2. Admin creates Transaction B and assigns User C as an approver (approver record ID = N).
3. Attacker discovers approver ID N (sequential integer; also visible on any transaction the attacker participates in as creator/observer/approver).
4. Attacker sends:
   ```
   DELETE /transactions/{A_id}/approvers/{N}
   ```
5. `getCreatorsTransaction(A_id, attacker)` passes — attacker owns Transaction A.
6. `removeTransactionApprover(N)` deletes approver N, which belongs to Transaction B.
7. Transaction B's approval requirement is silently removed.

### Impact Explanation

An attacker with a normal user account can unilaterally remove any approver from any transaction in the organization, including threshold-tree approver structures (the `removeNode` call removes entire subtrees). This bypasses the multi-signature approval workflow, potentially allowing transactions to advance to execution without the required approvals. The integrity of the organization's governance model is broken.

### Likelihood Explanation

- **Attacker preconditions:** Must be a registered, verified user — no admin or privileged role required.
- **Knowledge required:** Approver IDs are sequential integers. An attacker who has ever created a transaction and added approvers to it will have observed IDs in the same namespace. Blind enumeration across a small integer range is trivially feasible.
- **No rate-limiting or anomaly detection** is evident on this endpoint.
- The attack is a single authenticated HTTP request.

### Recommendation

Inside `removeTransactionApprover` (or before calling it in the controller), verify that the approver record's `transactionId` (or its root node's `transactionId`) matches the `transactionId` URL parameter, mirroring the check already present in `updateTransactionApprover`:

```typescript
// After getCreatorsTransaction passes:
const approver = await this.approversService.getTransactionApproverById(id);
const root = await this.approversService.getRootNodeFromNode(approver.id);
if (root?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
await this.approversService.removeTransactionApprover(id);
```

### Proof of Concept

1. Register two users: `attacker` and `victim`.
2. `victim` creates Transaction B and adds User C as an approver → note the returned approver `id` (e.g., `7`).
3. `attacker` creates Transaction A → note its `id` (e.g., `42`).
4. `attacker` sends:
   ```
   DELETE /transactions/42/approvers/7
   Authorization: Bearer <attacker_token>
   ```
5. Response: `200 true`
6. Query `GET /transactions/{B_id}/approvers` — approver 7 is gone from Transaction B.
7. Transaction B can now proceed without User C's approval.

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
