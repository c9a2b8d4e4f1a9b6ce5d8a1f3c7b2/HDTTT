Audit Report

## Title
Transaction Creator Can Arbitrarily Lower Approval Threshold or Replace Approvers After Signatures Are Collected

## Summary
`ApproversService` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` allows the transaction creator to mutate the approval threshold and replace individual approvers at any point in a transaction's lifecycle — including after the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status. Neither `updateTransactionApprover` nor `removeTransactionApprover` checks the current transaction status before applying mutations. A malicious creator can lower the threshold to meet it with fewer approvals than intended, or replace a rejecting approver to nullify their rejection.

## Finding Description

**Root Cause**

`updateTransactionApprover` (lines 367–531) calls `getCreatorsTransaction` at line 394 to verify ownership: [1](#0-0) 

`getCreatorsTransaction` (lines 624–644) only checks `creatorKey.userId !== user.id` — it never inspects `transaction.status`: [2](#0-1) 

The threshold update path (lines 467–488) allows the creator to set any valid threshold value with no status guard: [3](#0-2) 

The approver replacement path (lines 489–517) clears the existing `signature`, `userKeyId`, and `approved` fields when a new `userId` is set, effectively erasing a prior rejection: [4](#0-3) 

`removeTransactionApprover` (lines 533–544) also performs no status check — it only verifies the approver exists: [5](#0-4) 

The controller exposes both mutations to any authenticated transaction creator via `PATCH /:id` and `DELETE /:id`: [6](#0-5) 

By contrast, `approveTransaction` correctly guards against acting on a transaction that is not in an active state: [7](#0-6) 

No equivalent guard exists in `updateTransactionApprover` or `removeTransactionApprover`.

## Impact Explanation

The multi-sig approval workflow is the primary security control for organization transactions. Allowing the transaction creator to retroactively lower the threshold or replace rejecting approvers after votes have been cast completely undermines this control. A malicious creator can force any transaction through with fewer approvals than the organization intended, or nullify any rejection. Approvers believe their vote is binding, but the creator can silently invalidate it — a critical integrity failure in the trust model.

## Likelihood Explanation

Any authenticated user who creates a transaction can exploit this. No admin role, leaked credentials, or special privileges are required. The API endpoints (`PATCH /transactions/:id/approvers/:id` and `DELETE /transactions/:id/approvers/:id`) are standard REST calls available to any transaction creator. The attack requires only that the creator wait for partial approvals to be collected before mutating the threshold or approver list.

## Recommendation

Add a transaction status guard at the start of both `updateTransactionApprover` and `removeTransactionApprover` (and in the controller's `removeTransactionApprover` handler). After calling `getCreatorsTransaction`, check that the returned transaction's `status` is `WAITING_FOR_SIGNATURES` or earlier (e.g., `NEW` / `DRAFT`). If the transaction has already progressed to `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any terminal state, reject the mutation with a `BadRequestException`. This mirrors the guard already present in `approveTransaction` at lines 583–588.

## Proof of Concept

**Attack 1 — Threshold Lowering:**
1. Creator creates a transaction with a threshold tree node: `threshold=3`, children = [A, B, C, D, E].
2. Approvers A and B approve (`approved=true`). Approver C rejects (`approved=false`). Transaction remains in `WAITING_FOR_SIGNATURES`.
3. Creator sends `PATCH /transactions/:txId/approvers/:treeNodeId` with body `{ "threshold": 2 }`.
4. The threshold update path at lines 480–485 sets threshold to 2 with no status check. The chain service's status evaluator now sees 2 approvals ≥ threshold of 2 and transitions the transaction to `WAITING_FOR_EXECUTION`.
5. C's rejection is bypassed; the transaction proceeds to execution.

**Attack 2 — Approver Replacement to Nullify Rejection:**
1. Creator creates a transaction with approvers [A, B, C], threshold=2.
2. A approves, B rejects.
3. Creator sends `PATCH /transactions/:txId/approvers/:bApproverId` with body `{ "userId": D }`.
4. The replacement path at lines 500–516 sets `userKeyId: undefined`, `signature: undefined`, `approved: undefined` for B's slot and assigns D as the new approver.
5. D approves. Threshold is met. B's rejection is permanently erased from the record.
