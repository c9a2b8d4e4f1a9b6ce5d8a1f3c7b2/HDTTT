Audit Report

## Title
Weak Password Policy Allows Trivially Guessable Passwords in Organization Backend and Local Mode

## Summary
The organization backend's `ChangePasswordDto` and `NewPasswordDto` enforce only an 8-character minimum with zero complexity requirements via a misconfigured `@IsStrongPassword` decorator. The local-mode `changePassword` IPC handler in the Electron main process performs no password strength validation at all. Because the password also serves as the decryption key for stored Hedera private keys, a weak password directly threatens key material.

## Finding Description

**Organization backend (server-side):**

`ChangePasswordDto` (`back-end/apps/api/src/auth/dtos/change-password.dto.ts`, lines 8–19) and `NewPasswordDto` (`back-end/apps/api/src/auth/dtos/new-password.dto.ts`, lines 4–15) both use `@IsStrongPassword` with all complexity requirements explicitly zeroed:

```ts
@IsStrongPassword(
  { minLength: 8, minLowercase: 0, minNumbers: 0, minSymbols: 0, minUppercase: 0 },
  { message: 'Password is too weak, must contain at least 8 characters.' },
)
```

Any 8-character string (`aaaaaaaa`, `00000000`, `abcdefgh`) passes server-side DTO validation and is accepted by `PATCH /auth/change-password` and `PATCH /auth/set-password`. [1](#0-0) [2](#0-1) 

**Local mode (Electron main process):**

The `changePassword` function in `front-end/src/main/services/localUser/auth.ts` (lines 85–95) has no password strength check. It only verifies the old password is correct, then immediately persists the new one:

```ts
export const changePassword = async (userId, oldPassword, newPassword) => {
  const isOldCorrect = await comparePasswords(userId, oldPassword);
  if (isOldCorrect) {
    await updatePassword(userId, newPassword);  // no strength check
  } else {
    throw new Error('Incorrect current password');
  }
  await changeDecryptionPassword(userId, oldPassword, newPassword);
};
``` [3](#0-2) 

The IPC channel `localUser:changePassword` is registered unconditionally in the main process: [10](#0-9) 

The preload script exposes `changePassword` to the renderer world via `contextBridge.exposeInMainWorld`: [11](#0-10) [6](#0-5) 

The UI gate (`isPrimaryButtonDisabled` in `ProfileTab.vue`) is purely renderer-side and is bypassed by calling `window.electronAPI.local.localUser.changePassword(userId, currentPassword, 'aaa')` directly from the renderer context. [8](#0-7) 

The front-end `isPasswordStrong` function only checks for 10 characters minimum with no complexity requirements: [7](#0-6) 

## Impact Explanation

The password serves a dual purpose:
1. It authenticates the user at login.
2. It is used as the decryption key for stored Hedera private key pairs (passed directly to `changeDecryptionPassword`). [12](#0-11) 

- **Organization mode**: An attacker who can reach the organization API can brute-force `POST /auth/login` with a small dictionary. Upon success, they obtain a JWT and can sign or approve Hedera transactions on behalf of the victim.
- **Local mode**: If the encrypted SQLite database file is exfiltrated, the weak password makes offline brute-force of the encrypted private keys trivial, leading to full private key compromise and unauthorized Hedera transaction signing.

## Likelihood Explanation

- **Organization mode**: Any authenticated user can set a weak password through the normal UI or by directly calling `PATCH /auth/change-password`. No privilege escalation is required. The endpoint is reachable by any valid organization member.
- **Local mode**: The IPC bypass requires code execution in the renderer process. In production, `webContents.openDevTools()` is only called in dev mode (`VITE_DEV_SERVER_URL` set), raising the bar slightly. However, the weak front-end policy (10 chars, no complexity) is reachable by any local user through the normal UI without any bypass. [13](#0-12) 

## Recommendation

**Organization backend:**
- Remove the zeroed-out complexity overrides from `@IsStrongPassword` in both `ChangePasswordDto` and `NewPasswordDto`, or replace with explicit `@MinLength`, `@Matches` decorators enforcing uppercase, lowercase, number, and symbol requirements.
- Alternatively, adopt a minimum length of 12+ characters with at least one character from each class.

**Local mode:**
- Add server-side (main process) password strength validation inside the `changePassword` function in `front-end/src/main/services/localUser/auth.ts` before calling `updatePassword`, mirroring or exceeding the front-end `isPasswordStrong` check.
- Align the front-end `isPasswordStrong` utility to also require complexity (uppercase, lowercase, number, symbol) in addition to the 10-character minimum.

## Proof of Concept

**Organization backend:**
```bash
# After obtaining a valid JWT:
curl -X PATCH https://<org-server>/auth/change-password \
  -H "Authorization: Bearer <jwt>" \
  -H "Content-Type: application/json" \
  -d '{"oldPassword":"ValidPass1!","newPassword":"aaaaaaaa"}'
# Returns 200 OK — weak password accepted
```

**Local mode IPC bypass (renderer console):**
```js
// In Electron renderer DevTools console or compromised renderer:
await window.electronAPI.local.localUser.changePassword(
  '<userId>',
  '<currentPassword>',
  'aaa'  // 3-char password, no complexity — accepted by main process
);
// Succeeds: password and key decryption password both updated to 'aaa'
```

  result: boolean;
