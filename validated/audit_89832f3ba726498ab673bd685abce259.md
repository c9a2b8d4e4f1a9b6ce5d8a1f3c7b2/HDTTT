Audit Report

## Title
Stale Array-Index‚Äìto‚Äì`seq` Mapping in `updateGroup` Causes Permanent Transaction Draft Corruption After Group Item Removal

## Summary
`updateGroup` in `front-end/src/renderer/services/transactionGroupsService.ts` reconciles the in-memory `groupItems` array with the SQLite database by using raw array indices (`0`, `1`, `2`‚Ä¶) as `seq` lookup keys. After a user removes any item that is not the last one from a saved group, the in-memory array shifts but the database `seq` values do not. On the next save, `updateGroup` deletes the wrong DB record and overwrites the wrong draft, permanently corrupting the group's transaction data in SQLite.

## Finding Description

**File:** `front-end/src/renderer/services/transactionGroupsService.ts`, function `updateGroup` (lines 116‚Äì178)

**Root cause ‚Äî `updateGroup` conflates array position with DB `seq`:**

When a group is first saved, `addGroupWithDrafts` assigns `seq = index.toString()` sequentially: [1](#0-0) 

After the first save, `saveGroup` writes the DB `seq` back into each in-memory item: [2](#0-1) 

When the user removes an item, `removeGroupItem` splices the array but **never updates the `seq` fields of the surviving items**: [3](#0-2) 

After removal the in-memory array is `[A(seq='0'), C(seq='2')]` but the DB still holds rows `seq='0'`, `seq='1'`, `seq='2'`.

**Delete loop bug (line 133):** iterates `fetchedItems.entries()` and calls `deleteGroupItem(id, index.toString())` ‚Äî using the array position `index` rather than `item.seq`: [7](#0-6) 

With `fetchedItems.length=3` and `groupItems.length=2`, the loop skips `index=0` and `index=1`, then at `index=2` calls `deleteGroupItem(id, '2')` ‚Äî deleting C's DB slot instead of B's.

**Update loop bug (line 145):** iterates `groupItems.entries()` and calls `getGroupItem(id, index.toString())` ‚Äî using the array position `index` rather than `item.seq`: [5](#0-4) 

At `index=1`, `item` is C (seq='2'), but `getGroupItem(id, '1')` retrieves B's DB record and overwrites it with C's transaction bytes.

**Resulting DB state after the corrupted save (3-item group, middle item removed):**

| DB `seq` | Expected content | Actual content |
|---|---|---|
| `'0'` | A | A (correct) |
| `'1'` | deleted (B removed) | C's bytes (wrong) |
| `'2'` | C | deleted (wrong) |

## Impact Explanation
When the corrupted group is reloaded and submitted to the Hedera network, transaction B (which was supposed to be removed) is replaced with C's bytes and executed, while C's original slot is silently dropped. For groups containing HBAR transfers, token transfers, or account operations, this means the wrong transactions are submitted ‚Äî funds can be sent to wrong accounts or in wrong amounts with no warning to the user. The corruption is written to the local SQLite database and persists across sessions; there is no automatic recovery path.

## Likelihood Explanation
The trigger is a completely normal user workflow:
1. Create a transaction group with ‚â• 2 items.
2. Save it (Personal Mode, local SQLite).
3. Reload the group.
4. Remove any item that is **not** the last one.
5. Save again.

No attacker is required. The only condition that avoids corruption is removing only the last item (in which case `index` and `seq` still coincide by accident).

## Recommendation
In the delete loop, replace `index.toString()` with `item.seq`:
```ts
// Before (buggy):
await window.electronAPI.local.transactionGroups.deleteGroupItem(id, index.toString());

// After (correct):
await window.electronAPI.local.transactionGroups.deleteGroupItem(id, item.seq);
```

In the update loop, replace `index.toString()` with `item.seq`:
```ts
// Before (buggy):
const savedItem = await getGroupItem(id, index.toString());

// After (correct):
const savedItem = await getGroupItem(id, item.seq);
```

Additionally, `removeGroupItem` in `storeTransactionGroup.ts` should renumber the `seq` fields of all surviving items after removal to keep in-memory state consistent with what the next save will write to the DB.

## Proof of Concept

**Setup:** A saved group with 3 items. DB state: `seq='0'`‚ÜíA, `seq='1'`‚ÜíB, `seq='2'`‚ÜíC. In-memory: `[A(seq='0'), B(seq='1'), C(seq='2')]`.

**Step 1:** User clicks Delete on item B (index 1). `removeGroupItem(1)` is called:
```ts
// storeTransactionGroup.ts line 142-144
groupItems.value = [...groupItems.value.slice(0, 1), ...groupItems.value.slice(2)];
// Result: [A(seq='0'), C(seq='2')]  ‚Äî seq fields NOT updated
```

**Step 2:** User clicks Save. `saveGroup` detects `groupItems.value[0].groupId !== undefined` and calls `updateGroup(id, userId, ..., [A(seq='0'), C(seq='2')])`.

**Step 3:** Inside `updateGroup`, `fetchedItems = getGroupItems(id)` returns 3 rows: `[{seq:'0',...}, {seq:'1',...}, {seq:'2',...}]`.

**Step 4:** Delete loop ‚Äî `fetchedItems.length(3) > groupItems.length(2)`:
```
index=0: 0 < 2 ‚Üí continue
index=1: 1 < 2 ‚Üí continue
index=2: 2 >= 2 ‚Üí deleteGroupItem(id, '2')  ‚Üê deletes C's slot, not B's
```

**Step 5:** Update loop:
```
index=0, item=A(seq='0'): getGroupItem(id, '0') ‚Üí A's draft ‚Üí update with A's bytes ‚úì
index=1, item=C(seq='2'): getGroupItem(id, '1') ‚Üí B's draft ‚Üí overwrite with C's bytes ‚úó
```

**Final DB state:** `seq='0'`=A (correct), `seq='1'`=C's data (B's slot overwritten), `seq='2'`=gone (C's slot deleted). The group is permanently corrupted in SQLite.
