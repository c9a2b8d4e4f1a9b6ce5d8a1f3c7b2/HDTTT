### Title
Unbounded Batch Size in Signature Upload Endpoints Enables Authenticated DoS via Resource Exhaustion

### Summary
Two API endpoints — `POST /transactions/signers` and `POST /transactions/signatures/import` — accept an array of `UploadSignatureMapDto` objects with no enforced upper bound on array length. An authenticated user can submit a single request containing thousands of entries, triggering concurrent cryptographic operations, unbounded database queries, and memory allocation that exhausts server resources and renders the API unavailable for all users. A developer comment in the codebase explicitly acknowledges the missing limit.

### Finding Description

**Vulnerability type:** DoS — resource exhaustion via unbounded batch input.

**Entry points:**

1. `POST /transactions/signers` handled by `SignersController.uploadSignatureMap()`
2. `POST /transactions/signatures/import` handled by `TransactionsController.importSignatures()`

Both controllers accept `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array-size guard: [1](#0-0) [2](#0-1) 

The `UploadSignatureMapDto` class carries no `@ArrayMaxSize()` or equivalent constraint — only per-item field validators: [3](#0-2) 

`transformAndValidateDto` validates each element individually but never checks the array length, so arbitrarily large arrays pass validation silently.

**Dangerous processing in `SignersService.uploadSignatureMaps()`:**

The service fans out the entire input array with `Promise.all`, firing all cryptographic validations and DB lookups concurrently: [4](#0-3) 

Each concurrent item calls `SDKTransaction.fromBytes()`, `validateSignature()`, and `addSignature()` — all CPU-intensive cryptographic operations — plus DB reads per transaction ID.

**Sequential exhaustion in `TransactionsService.importSignatures()`:**

The import path iterates over every item in a `for...of` loop, calling `verifyAccess`, `SDKTransaction.fromBytes`, `validateSignature`, and `addSignature` per item, holding the request open for the full duration: [5](#0-4) 

**Developer acknowledgment of the missing limit:** [6](#0-5) 

The comment `//Added a batch mechanism, probably should limit this on the api side of things` confirms the developers identified the gap but never implemented the guard.

### Impact Explanation

A single crafted request with a large array (e.g., 10,000 entries referencing valid transaction IDs) causes:

- **CPU exhaustion** from concurrent cryptographic operations (`Promise.all` path) or a long-running synchronous loop (import path).
- **Database connection pool exhaustion** from the unbounded `IN (...)` query and per-item DB operations.
- **Memory pressure** from holding all `sdkTransaction` objects and result maps in memory simultaneously.

The result is API service unavailability for all users — transactions cannot be signed, imported, or processed — until the server recovers or is restarted. Because the attacker controls the array size, they can sustain the attack with repeated requests.

### Likelihood Explanation

- **Attacker precondition:** A valid JWT for any verified user account. No admin or privileged role is required.
- **Attack complexity:** Trivially low — craft a JSON array of arbitrary length and POST it to either endpoint.
- **Detectability:** No rate limiting or payload-size enforcement is visible on these routes beyond NestJS's default body-size limit (typically 100 KB, which still allows thousands of compact entries).

### Recommendation

1. **Enforce a hard array-size cap at the DTO/controller layer.** Add `@ArrayMaxSize(N)` (e.g., `N = 100`) to a wrapper DTO that wraps the array, and reject requests exceeding it with HTTP 400 before any processing begins.
2. **Apply rate limiting** (e.g., via `@nestjs/throttler`) to the signature upload endpoints.
3. **Replace `Promise.all` with a concurrency-limited executor** (e.g., process in chunks of 10–20) to bound simultaneous DB and CPU load even for legitimate large batches.

### Proof of Concept

**Precondition:** Attacker holds a valid JWT for any verified user account.

**Steps:**

```
POST /transactions/signers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1.000": { "<pubkey>": "<sig>" } } } },
  { "id": 1, "signatureMap": { ... } },
  ... (repeated 10,000 times)
]
```

**Expected outcome:**
- The server fans out 10,000 concurrent `Promise.all` branches in `validateAndProcessSignatures`.
- Each branch calls `SDKTransaction.fromBytes()` and `validateSignature()`.
- The DB connection pool is saturated by the batch `IN (...)` query and per-item lookups.
- The Node.js event loop is blocked; other requests time out.
- The API becomes unresponsive for all users until the request completes or the process is killed.

The same attack applies to `POST /transactions/signatures/import` via `TransactionsService.importSignatures()`, which processes the same unbounded array sequentially, holding the request thread for the full duration. [7](#0-6) [8](#0-7)
