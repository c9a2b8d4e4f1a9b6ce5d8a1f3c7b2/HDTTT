### Title
Unbounded Batch Array in Signature Upload Endpoints Enables Authenticated DoS

### Summary

Two authenticated endpoints accept an array of `UploadSignatureMapDto` objects with no enforced maximum length. An authenticated user can submit an arbitrarily large array, triggering concurrent cryptographic validation and bulk database operations for every element. The codebase even contains a self-admitted comment acknowledging the missing limit. This maps directly to the external report's finding of "no maximum length of targets defined."

### Finding Description

**Vulnerability class:** Missing input validation / no array size cap on batch endpoints (DoS via resource exhaustion).

**Entry point 1 — `POST /transactions/:transactionId?/signers`**

`SignersController.uploadSignatureMap` accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array size constraint: [1](#0-0) 

The body is passed through `transformAndValidateDto`, which only validates individual DTO field types — it applies no limit on the number of elements in the array: [2](#0-1) 

The resulting array is forwarded to `SignersService.uploadSignatureMaps`, which calls `validateAndProcessSignatures` using `Promise.all` — meaning **all N items are processed concurrently**: [3](#0-2) [4](#0-3) 

Each concurrent item performs: a DB lookup, `SDKTransaction.fromBytes` deserialization, cryptographic signature verification, and a bulk DB write. A single request with thousands of items saturates the DB connection pool, CPU, and heap simultaneously.

**Entry point 2 — `POST /transactions/signatures/import`**

`TransactionsController.importSignatures` has the identical missing limit: [5](#0-4) 

The service itself contains a developer comment explicitly acknowledging the missing guard: [6](#0-5) 

The `UploadSignatureMapDto` itself has no `@ArrayMaxSize` or any cardinality constraint: [7](#0-6) 

### Impact Explanation

Any authenticated user (no admin role required — only a valid JWT and at least one registered key, enforced by `JwtAuthGuard` + `VerifiedUserGuard`) can send a single HTTP request containing thousands of `UploadSignatureMapDto` entries. The `Promise.all` in `validateAndProcessSignatures` fans out all work concurrently, exhausting the PostgreSQL connection pool, Node.js heap, and CPU. The result is service unavailability for all other users until the request completes or the process crashes. The self-admitted comment at line 575 confirms the developers are aware the limit is missing.

### Likelihood Explanation

The attacker precondition is minimal: a valid account on the platform. No privileged keys, no leaked secrets, no internal access. The endpoint is reachable over standard HTTPS. The attack is a single crafted POST request and is trivially repeatable.

### Recommendation

1. Add a hard maximum array size at the controller layer before `transformAndValidateDto` is called. For example, reject any request where `Array.isArray(body) && body.length > MAX_BATCH_SIZE` (e.g., `MAX_BATCH_SIZE = 50`).
2. Alternatively, add an `@ArrayMaxSize(50)` decorator to a wrapper DTO that holds the array, so NestJS's validation pipe rejects oversized payloads before they reach the service.
3. For `SignersService.validateAndProcessSignatures`, replace the unbounded `Promise.all` with a chunked/sequential approach even after a size cap is added, to prevent a single large-but-within-limit batch from monopolizing the DB pool.

### Proof of Concept

```
POST /transactions/signers HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 2, "signatureMap": { ... } },
  ... (repeat 10,000 times with valid or invalid transaction IDs)
]
```

**Expected outcome:** The server fans out 10,000 concurrent DB queries and cryptographic operations via `Promise.all` in `validateAndProcessSignatures`. The PostgreSQL connection pool is exhausted, Node.js heap spikes, and the API becomes unresponsive to all other requests. The attack requires only a valid authenticated session — no privileged role.

```typescript
export async function transformAndValidateDto<T extends object>(
  dtoClass: new (...args: any[]) => T,
  payload: T | T[],
): Promise<T[]> {
  const items = Array.isArray(payload) ? payload : [payload];
  const instances = items.map(item => plainToInstance(dtoClass, item));
  await Promise.all(instances.map(instance => validateOrReject(instance)));
  return instances;
```
