Audit Report

## Title
Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController.removeTransactionApprover` authorizes the request by verifying the caller owns `transactionId`, but then deletes the approver identified by the independent `id` parameter with no check that the approver actually belongs to that transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

In `approvers.controller.ts`, the delete handler performs two sequential, unrelated operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
``` [1](#0-0) 

The service-level `removeTransactionApprover` only fetches the approver by its own primary key and soft-deletes it, with no validation that `approver.transactionId` matches the `transactionId` from the URL:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

The sibling `updateTransactionApprover` service method correctly performs this cross-reference check before proceeding:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The remove path has no equivalent guard, making the `transactionId` ownership check entirely ineffective at protecting approvers of other transactions.

## Impact Explanation
An authenticated but unprivileged user can silently delete approvers from any transaction they do not own. This directly undermines the multi-signature approval workflow: a transaction requiring N approvals can have its approver list arbitrarily gutted, potentially allowing it to advance to execution without the intended oversight. This is an unauthorized, irreversible (soft-delete) state change with direct integrity impact on the transaction lifecycle.

## Likelihood Explanation
The attacker only needs a valid account and at least one transaction of their own — the standard onboarding path. No admin credentials, leaked secrets, or privileged roles are required. Approver IDs are sequential integers stored in the `transaction_approver` table, making enumeration trivial via the public REST API. The attack is fully reachable with a single crafted HTTP request.

## Recommendation
In `removeTransactionApprover` (service), add the same cross-reference check that `updateTransactionApprover` already performs. After fetching the approver, resolve its root node and verify `rootNode.transactionId === transactionId` before calling `removeNode`. Concretely:

1. Call `getRootNodeFromNode(approver.id)` to get the root of the approver tree.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.
3. Only then proceed with `removeNode`.

This mirrors the existing guard at line 390–391 of `approvers.service.ts` and closes the authorization gap. [2](#0-1) 

## Proof of Concept

1. Attacker registers an account and creates **Transaction A** (they are its creator).
2. Victim's **Transaction B** has an approver with database ID = `X` (discoverable by enumeration or via the `GET /transactions/:transactionId/approvers` endpoint if the attacker has any read access).
3. Attacker sends:
   ```
   DELETE /transactions/A/approvers/X
   Authorization: Bearer <attacker_token>
   ```
4. `getCreatorsTransaction(A, attacker)` succeeds — attacker owns A. [4](#0-3) 
5. `removeTransactionApprover(X)` fetches approver `X` (which belongs to Transaction B) and calls `removeNode(X)` — no `transactionId` cross-check is performed. [2](#0-1) 
6. Approver `X` is soft-deleted from Transaction B. The server returns `true` (HTTP 200). [5](#0-4)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
