Audit Report

## Title
Any Authenticated Organization User Can Post and Read Comments on Transactions They Have No Access To

## Summary
`CommentsController` enforces only JWT authentication and user verification, but performs no transaction-level participant check before allowing a user to create or read comments on any transaction. This is in direct contrast to every other transaction sub-resource controller in the codebase, and is explicitly acknowledged by the developer with a `//TODO` comment in the controller itself.

## Finding Description

`CommentsController` applies only three guards at the class level — `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — which confirm the caller holds a valid, non-blacklisted JWT and has `UserStatus.NONE`. No guard or service-layer check verifies that the user has any relationship to the target transaction. [1](#0-0) 

The `createComment` handler passes `transactionId` directly to `CommentsService.createComment`, which immediately creates and persists the comment with no access verification: [2](#0-1) [3](#0-2) 

The developer explicitly flagged this gap: [4](#0-3) 

The `getComments` and `getCommentById` GET endpoints on the same controller also lack any transaction-level access check: [5](#0-4) [6](#0-5) 

By contrast, every other transaction sub-resource enforces participant-level access. `ObserversService.getTransactionObserversByTransactionId` throws `UnauthorizedException` if the user is not the creator, a signer, an observer, or an approver: [7](#0-6) 

`ApproversService.getVerifiedApproversByTransactionId` enforces the same pattern: [8](#0-7) 

## Impact Explanation

- **Unauthorized state mutation**: Any verified organization user can attach comments to any transaction in the system, including transactions they are not a party to. This pollutes the comment history of sensitive Hedera transactions (e.g., account creation, HBAR transfers, file updates).
- **Information disclosure**: Any verified user can enumerate and read all comments on any transaction via `GET /transactions/:transactionId/comments`, leaking internal deliberation, approval rationale, or sensitive operational notes intended only for transaction participants.
- The transaction isolation model — enforced for signers, observers, and approvers — is completely bypassed for the comments sub-resource.

## Likelihood Explanation

The attacker precondition is minimal: hold a valid JWT for any registered, verified organization account (`UserStatus.NONE`). No admin privilege, no leaked secret, and no special role is required. The endpoint is reachable via a standard HTTP POST to `POST /transactions/<any_id>/comments` with a bearer token. Any malicious insider or compromised low-privilege account can exploit this immediately.

## Recommendation

Add a participant verification check in `CommentsService.createComment` and `CommentsService.getTransactionComments`, mirroring the pattern already used in `ObserversService.getTransactionObserversByTransactionId` and `ApproversService.getVerifiedApproversByTransactionId`. Before creating or returning comments, fetch the transaction with its `creatorKey`, `observers`, `signers`, and `approvers` relations, then throw `UnauthorizedException` if the requesting user is none of those. The developer's own `//TODO` comment at line 22 of `comments.controller.ts` confirms this was a known, deferred gap.

## Proof of Concept

1. Register or obtain credentials for any verified organization user (non-admin, `UserStatus.NONE`).
2. Authenticate: `POST /auth/login` → receive `accessToken`.
3. Identify any transaction ID the user has no relationship to (e.g., a transaction created by another user with no shared signers/observers/approvers).
4. Post a comment:
   ```
   POST /transactions/<victim_transaction_id>/comments
   Authorization: Bearer <accessToken>
   Content-Type: application/json

   { "message": "unauthorized comment" }
   ```
5. Observe HTTP 201 — comment is created and persisted with no access check.
6. Read all comments on the same transaction:
   ```
   GET /transactions/<victim_transaction_id>/comments
   Authorization: Bearer <accessToken>
   ```
7. Observe HTTP 200 — all comments returned with no access check.

The same request against `GET /transactions/<victim_transaction_id>/observers` (a properly guarded endpoint) returns HTTP 401 `"You don't have permission to view this transaction"`, confirming the disparity.
