# Audit Report

## Title
Any Authenticated User Can Delete Approvers Belonging to Any Transaction via Broken Cross-Resource Authorization in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user is the creator of `:transactionId`, but then passes `:id` directly to `removeTransactionApprover(id)` without verifying the approver belongs to that transaction. An attacker who owns any one transaction can delete approvers from any other transaction by supplying their own `transactionId` (to pass the creator check) alongside an arbitrary victim approver `id`.

## Finding Description

**Root cause — controller decouples the creator check from the deletion target:**

In `approvers.controller.ts` lines 102–113, the handler performs two independent, unrelated calls:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
``` [4](#0-3) 

Step 1 confirms the caller is the creator of `transactionId`. Step 2 deletes the approver identified by `id`. No code ever checks that `approver.transactionId === transactionId`.

**Service method accepts any approver ID with no ownership binding:**

`removeTransactionApprover(id)` in `approvers.service.ts` lines 534–544 fetches the approver by `id` and calls `removeNode(approver.id)` — a recursive soft-delete of the entire subtree — without any reference to the URL's `transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

**Contrast with `updateTransactionApprover`, which does the check correctly:**

The update path at lines 386–394 explicitly verifies the approver's root transaction matches the URL parameter before proceeding:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The delete path has no equivalent check, making the `transactionId` URL parameter meaningless as a security boundary for the deletion operation.

## Impact Explanation

An attacker can silently delete any approver (or an entire approver subtree via `removeNode`) from any transaction in the organization. This:

- Removes required approval gates from transactions the attacker does not own, allowing those transactions to advance to execution without the intended approvals.
- Permanently corrupts the approval workflow state for victim transactions via soft-delete (`deletedAt`).
- Undermines the multi-signature trust model that is the core security guarantee of Organization Mode.

## Likelihood Explanation

- **Precondition**: The attacker must be an authenticated, verified organization user — the lowest privilege level above anonymous.
- **No special knowledge required**: Approver IDs are sequential integers and can be trivially enumerated.
- **Trigger**: Create one transaction (a normal user action), then issue `DELETE /transactions/{own_tx_id}/approvers/{victim_approver_id}` for any approver ID in the system.
- No rate limiting or anomaly detection is present on this endpoint.

## Recommendation

In `removeTransactionApprover` (service), add the same cross-resource ownership check that `updateTransactionApprover` already performs:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root node.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.
3. Only then proceed with `removeNode`.

Alternatively, pass `transactionId` into `removeTransactionApprover` and enforce the check there, keeping the authorization logic inside the service rather than the controller.

## Proof of Concept

1. Attacker (user A) creates Transaction T_A and obtains its `transactionId` (e.g., `42`).
2. Attacker observes or enumerates victim approver ID (e.g., `99`) belonging to Transaction T_B owned by user B.
3. Attacker sends:
   ```
   DELETE /transactions/42/approvers/99
   Authorization: Bearer <attacker_jwt>
   ```
4. Server calls `getCreatorsTransaction(42, userA)` — passes, because user A owns T_A.
5. Server calls `removeTransactionApprover(99)` — deletes approver 99 (belonging to T_B) with no further check.
6. Approver 99 is soft-deleted from T_B's approval workflow. T_B's required approval gate is silently removed.
