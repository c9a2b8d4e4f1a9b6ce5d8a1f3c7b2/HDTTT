### Title
Stale In-Memory `userApprovers` State Causes Wrong Notification Event in `approveTransaction()`

### Summary
In `approveTransaction()` inside `ApproversService`, the `userApprovers` array is fetched from the database before the approval is written. After the database update, the in-memory objects are never refreshed. A post-write condition check (`userApprovers.every(a => a.approved)`) therefore always evaluates against stale (pre-update) values, causing `emitTransactionUpdate` to be emitted instead of `emitTransactionStatusUpdate` every time a user approves a transaction. This is the TypeScript/NestJS analog of the Solidity `memory` vs `storage` bug: state is correctly persisted to the database, but the in-memory snapshot used for the branching decision is never updated.

### Finding Description

**Root cause — stale in-memory snapshot used after a database write**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction()` follows this sequence:

1. **Fetch** approvers from DB into `userApprovers` (each record has `approved = undefined/null` at this point).
2. **Write** the new `approved` value to the database via a query-builder `UPDATE`.
3. **Branch** on `userApprovers.every(a => a.approved)` — but `userApprovers` still holds the pre-update in-memory objects. [1](#0-0) [2](#0-1) [3](#0-2) 

The branching logic at line 614 is:

```typescript
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(...)   // triggers status-change notifications
} else {
  emitTransactionUpdate(...)         // triggers only a UI data refresh
}
```

**When `dto.approved = true`:**
- `!dto.approved` → `false`
- `userApprovers.every(a => a.approved)` → `false` (stale; `approved` was `undefined` before the write)
- Result: `false || false = false` → `emitTransactionUpdate` is **always** called, never `emitTransactionStatusUpdate`

The database row is correctly updated, but the in-memory `userApprovers` objects are never mutated to reflect the new `approved` value, so the condition that is supposed to detect "all approvers have now approved" is permanently broken for the positive-approval path.

### Impact Explanation

`emitTransactionStatusUpdate` publishes to the `TRANSACTION_STATUS_UPDATE` NATS subject, which the notifications service consumes to send in-app and email notifications to the transaction creator, signers, observers, and approvers about the status change. [4](#0-3) [5](#0-4) 

Because `emitTransactionStatusUpdate` is never called on the positive-approval path, users do not receive timely in-app or email notifications when an approver approves a transaction. The chain service's 10-second cron job (`handleTransactionsBetweenNowAndAfterThreeMinutes`) will eventually re-evaluate statuses, so transaction execution is not permanently blocked — but the notification pipeline is broken for every approval event where `dto.approved = true`. [6](#0-5) 

### Likelihood Explanation

This is triggered by every normal user approval (`approved: true`) of any organization transaction. No special attacker capability is required — any registered user who is a valid approver of a transaction will trigger this path. The code path is reachable via the public REST endpoint `POST /transactions/:transactionId/approvers/approve`. [7](#0-6) 

### Recommendation

After the database update, either:

**Option A** — Update the in-memory objects to reflect the written state before the branch:

```typescript
// After the DB update block (line 610)
userApprovers.forEach(a => {
  a.approved = dto.approved;
  a.userKeyId = dto.userKeyId;
  a.signature = dto.signature;
});
```

**Option B** — Re-fetch the approvers from the database after the write and use the fresh data for the condition check.

**Option C** — Replace the stale in-memory check with `dto.approved` alone (since the intent is to emit a status update whenever the user approves, not only when all their records are approved):

```typescript
if (!dto.approved) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

### Proof of Concept

**Preconditions:**
- A transaction exists in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status.
- User A is a registered approver for that transaction with `approved = null` in the DB.

**Steps:**
1. User A calls `POST /transactions/:id/approvers/approve` with `{ approved: true, userKeyId: X, signature: <valid> }`.
2. The DB row for User A's `TransactionApprover` is correctly updated: `approved = true`.
3. At line 614, `userApprovers.every(a => a.approved)` evaluates the pre-fetch in-memory object where `approved` is still `null`/`undefined` → returns `false`.
4. `emitTransactionUpdate` is called instead of `emitTransactionStatusUpdate`.
5. The notifications service never receives a `TRANSACTION_STATUS_UPDATE` event for this approval; in-app and email notifications are not dispatched.
6. The test at line 1278 in `approvers.service.spec.ts` asserts `emitTransactionStatusUpdate` is called, but only because the mock for `getVerifiedApproversByTransactionId` returns `[{ userId: user.id, transactionId: 1 }]` with `approved` already `undefined` — the test does not set `approved: true` on the mock, so it accidentally passes via the `!dto.approved` branch of the second call (`approved: false`), masking the bug. [8](#0-7)
