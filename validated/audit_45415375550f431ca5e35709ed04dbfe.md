Audit Report

## Title
Missing Cross-Reference Validation in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers Belonging to Other Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller by verifying they are the creator of `transactionId`, but then removes the approver identified by `:id` without verifying that approver actually belongs to `transactionId`. Any authenticated, verified user who has created at least one transaction can soft-delete approvers (and their entire subtrees) from any other transaction in the system.

## Finding Description

In `approvers.controller.ts`, the delete handler performs two sequential calls:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the caller is the creator of `transactionId` — it says nothing about the approver identified by `id`. [2](#0-1) 

`removeTransactionApprover` then fetches the approver by `id` and soft-deletes its entire subtree with no check that the approver belongs to `transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
    return result;
}
``` [3](#0-2) 

`removeNode` performs a recursive SQL soft-delete of the entire approver subtree rooted at `id`: [4](#0-3) 

The missing check is: **`rootNode.transactionId === transactionId`**. The `updateTransactionApprover` path correctly performs this check at line 390, but `removeTransactionApprover` does not:

```typescript
// Present in updateTransactionApprover — ABSENT in removeTransactionApprover
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [5](#0-4) 

## Impact Explanation

An attacker who is the creator of any transaction can:

1. Enumerate approver IDs (sequential integers) from any transaction via `GET /transactions/:transactionId/approvers`.
2. Call `DELETE /transactions/{their_own_txId}/approvers/{victim_approver_id}`.
3. The authorization check passes (they are the creator of `their_own_txId`).
4. The approver with `victim_approver_id` — belonging to a completely different transaction — is soft-deleted along with its entire subtree via `removeNode`.

This enables **unauthorized destruction of approval requirements on any transaction in the system**, allowing transactions to proceed without required approvals. It also causes **permanent corruption of the approver tree** for victim transactions, as soft-deleted records cannot be recovered without direct database intervention.

## Likelihood Explanation

Preconditions are minimal: the attacker must be a registered, verified user who has created at least one transaction — a normal user flow. Approver IDs are sequential integers and are trivially enumerable via the `GET /transactions/:transactionId/approvers` endpoint. No privileged access, leaked credentials, or external dependencies are required.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), add a cross-reference check analogous to what `updateTransactionApprover` already does at line 390:

1. After fetching the approver by `id`, resolve its root node via `getRootNodeFromNode`.
2. Verify `rootNode.transactionId === transactionId`.
3. If the check fails, throw `UnauthorizedException`.

This mirrors the existing pattern:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [5](#0-4) 

## Proof of Concept

1. Attacker (user A) creates `transactionA` — they are its creator.
2. Victim (user B) creates `transactionB` with approver tree rooted at approver ID `42`.
3. Attacker calls:
   ```
   DELETE /transactions/{transactionA_id}/approvers/42
   ```
4. `getCreatorsTransaction(transactionA_id, userA)` passes — user A is the creator of `transactionA`.
5. `removeTransactionApprover(42)` fetches approver `42` (belonging to `transactionB`), calls `removeNode(42)`, and soft-deletes approver `42` and its entire subtree.
6. `transactionB`'s approval requirements are silently destroyed. [1](#0-0) [3](#0-2)

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
