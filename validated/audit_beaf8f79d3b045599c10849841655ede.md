### Title
Missing Cross-Object Authorization in `removeTransactionApprover`: Any Transaction Creator Can Delete Approvers from Transactions They Don't Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController` verifies that the requesting user is the creator of `:transactionId`, but then deletes the approver record identified by `:id` without verifying that `:id` actually belongs to `:transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system.

### Finding Description
The root cause is in `approvers.controller.ts` at the `removeTransactionApprover` handler:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);  // checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);                // deletes approver by id — no cross-check
  return true;
}
``` [1](#0-0) 

The authorization check (`getCreatorsTransaction`) only confirms the caller is the creator of the transaction referenced by `:transactionId`. It does **not** confirm that the approver record `:id` is associated with `:transactionId`. The subsequent `removeTransactionApprover(id)` call deletes whatever approver record matches `:id`, regardless of which transaction it belongs to.

**Exploit path:**
1. Attacker registers as a normal user and creates Transaction A (attacker-owned).
2. Attacker enumerates or guesses the approver ID (`id = 42`) belonging to Victim's Transaction B.
3. Attacker sends: `DELETE /transactions/<A_id>/approvers/42`
4. `getCreatorsTransaction(A_id, attacker)` passes — attacker owns Transaction A.
5. `removeTransactionApprover(42)` executes — approver 42 (belonging to Transaction B) is deleted.
6. Transaction B now has a missing approver, potentially allowing it to bypass its approval threshold and be executed or manipulated.

### Impact Explanation
An attacker can silently remove required approvers from any transaction in the organization. This breaks the multi-signature approval workflow: a transaction that required N approvers can have those approvers stripped, potentially allowing it to proceed with fewer approvals than intended, or corrupting the approval state permanently. This is a direct integrity failure in the trust model of the multi-sig coordination system.

### Likelihood Explanation
The attacker only needs a valid authenticated account (any registered user who has created at least one transaction). No admin privileges, no leaked secrets, and no special network access are required. Approver IDs are sequential integers, making enumeration trivial. This is fully exploitable by any malicious normal user.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver record identified by `id` actually belongs to `transactionId`. For example:

```typescript
const approver = await this.approversService.getApproverById(id);
if (!approver || approver.transactionId !== transactionId) {
  throw new ForbiddenException();
}
await this.approversService.removeTransactionApprover(id);
```

This mirrors the fix in the external report: the authorization check must be applied to the **actual resource being mutated**, not a sibling resource.

### Proof of Concept
**Preconditions:** Two users (Attacker, Victim) are registered. Victim has Transaction B (id=10) with an approver record (id=42). Attacker has Transaction A (id=5).

**Steps:**
1. Attacker authenticates and obtains a JWT.
2. Attacker sends:
   ```
   DELETE /transactions/5/approvers/42
   Authorization: Bearer <attacker_jwt>
   ```
3. Server executes `getCreatorsTransaction(5, attacker)` → passes (attacker owns tx 5).
4. Server executes `removeTransactionApprover(42)` → deletes approver 42 from Victim's Transaction B.
5. `GET /transactions/10/approvers` now returns one fewer approver — the deletion succeeded without the attacker ever having access to Transaction B. [2](#0-1)
