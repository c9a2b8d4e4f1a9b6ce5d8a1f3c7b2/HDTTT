Audit Report

## Title
Unbounded Recursive Approver Tree Causes Server Resource Exhaustion (DoS)

## Summary
The `POST /transactions/:transactionId/approvers` endpoint accepts an arbitrarily large, deeply nested approver tree with no depth or node-count limit. The internal `createApprover` function processes each node recursively, issuing multiple sequential database queries per node inside a single database transaction. A malicious but authenticated transaction creator can submit a single crafted request containing hundreds of nested threshold nodes, exhausting the database connection pool and degrading the API service for all other users.

## Finding Description

**Root cause — no size or depth constraint on the approver DTO:**

`CreateTransactionApproverDto` allows unlimited recursive nesting through its `approvers` field. Only `@ArrayMinSize(1)` is present; there is no `@ArrayMaxSize` and no depth limit: [1](#0-0) 

The outer `approversArray` in `CreateTransactionApproversArrayDto` is equally unconstrained: [2](#0-1) 

The body size is capped at 2 MB globally, but 2 MB of JSON can encode thousands of nested threshold nodes: [3](#0-2) 

**Exploit path — recursive processing with per-node DB queries:**

`createTransactionApprovers` defines an inner `createApprover` closure and calls it recursively for every node in the submitted tree: [4](#0-3) 

For each node the function issues multiple sequential queries:
- `isNode` — a `COUNT` query (always executed): [5](#0-4) 
- `findOne` for the parent + `getRootNodeFromNode` (a recursive CTE) — executed when `listId` is set: [6](#0-5) 
- `count` for the user + `getApproversByTransactionId` (another recursive CTE) — executed when `userId` is set: [7](#0-6) 
- `insert` — always executed: [8](#0-7) 

All of this runs inside a single `dataSource.transaction(...)` call, holding one DB connection open for the entire duration: [9](#0-8) 

**Why the duplicate-check does not protect against this:**

`isNode` only returns `true` (duplicate) when `typeof approver.userId === 'number'`. Pure threshold nodes (no `userId`, only `threshold` + nested `approvers`) are **never** flagged as duplicates, so an attacker can create an unlimited number of them in a single request: [10](#0-9) 

**Entry point:**

The endpoint is authenticated (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) and restricted to the transaction creator, but any registered, verified user can create a transaction and then call this endpoint: [11](#0-10) 

**Rate limiting does not mitigate this:**

General IP and user throttlers exist but are not applied to this endpoint, and they govern request *count*, not the computational cost of a single request: [12](#0-11) 

## Impact Explanation

A single HTTP request containing a wide or deep threshold-only approver tree (e.g., 500 nodes, each triggering 2–4 DB queries = 1,000–2,000 sequential queries inside one database transaction) will:

- Hold a database connection open for the entire duration of the transaction.
- Saturate the TypeORM connection pool, causing all concurrent API requests that need a DB connection to queue or time out.
- Spike CPU on the database server due to repeated recursive CTEs (`getRootNodeFromNode`, `getApproversByTransactionId`).

The result is a denial of service for all organization users sharing the same API and database instance.

## Likelihood Explanation

- **Attacker preconditions:** A valid, verified account — achievable by any user who can register on the platform.
- **Effort:** A single crafted POST request; no special tooling required.
- **Detection:** No rate limiting or request-size guard specific to the approver endpoint was found. The global 2 MB body limit is the only constraint, but it is large enough to encode thousands of nested nodes.
- **Repeatability:** The attacker can repeat the request after each one completes (or in parallel from multiple accounts) to maintain sustained pressure.

## Recommendation

1. **Add `@ArrayMaxSize` to both `approvers` and `approversArray`** in the DTO to cap the number of children per node and the number of root-level approvers.
2. **Enforce a maximum tree depth** either in the DTO (via a custom validator) or at the start of `createTransactionApprovers` before entering the database transaction.
3. **Count total nodes before processing** and reject requests that exceed a reasonable limit (e.g., 50 total nodes).
4. **Apply the user-level throttler** to the approvers `POST` endpoint explicitly using `@Throttle`.

## Proof of Concept

```http
POST /transactions/1/approvers HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                ... (repeat 200+ levels deep, each with userId leaf at the bottom)
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

Each threshold node in the chain passes all validation checks (threshold ≤ children count, children present), is never blocked by `isNode` (no `userId`), and triggers at least 2–4 sequential DB queries. A 200-level deep chain with a leaf user node at the bottom results in ~600–800 sequential DB queries inside a single held transaction, blocking the connection pool for all concurrent users.
