Audit Report

## Title
Unbounded `groupItems` Array and Missing String Length Limits in Transaction API Enable Authenticated DoS

## Summary
The `POST /transaction-groups` endpoint accepts a `CreateTransactionGroupDto` whose `groupItems` array has no `@ArrayMaxSize()` constraint. Any authenticated user can submit a single request containing thousands of transaction items, each with full binary payloads, forcing the server to process all of them in parallel via `Promise.all`. Additionally, the `name` and `description` fields in `CreateTransactionDto` carry no `@MaxLength()` decorator, allowing arbitrarily long strings to reach the service layer before any DB-level rejection. The `POST /transactions/:transactionId/signers` endpoint similarly accepts an unbounded `UploadSignatureMapDto[]` array. All gaps are reachable by any registered user with no elevated privileges.

## Finding Description

**Root cause 1 — Unbounded `groupItems` array**

`CreateTransactionGroupDto` declares `groupItems` with only `@IsArray()`, `@IsNotEmpty()`, and `@ValidateNested({ each: true })`. There is no `@ArrayMaxSize()` guard: [1](#0-0) 

The controller passes the DTO directly to `createTransactionGroup`: [2](#0-1) 

`createTransactionGroup` extracts all transaction DTOs and immediately fans out to `createTransactions`: [3](#0-2) 

`createTransactions` runs `validateAndPrepareTransaction` for every item concurrently via `Promise.all`: [4](#0-3) 

Each call to `validateAndPrepareTransaction` performs: a cryptographic signature verification (`publicKey.verify`), protobuf deserialization (`SDKTransaction.fromBytes`), an async transaction hash computation (`sdkTransaction.getTransactionHash()`), and node/size validation: [5](#0-4) 

With thousands of items in one request, this saturates CPU, memory, and the database connection pool simultaneously.

**Root cause 2 — No `@MaxLength` on `name` and `description` in `CreateTransactionDto`**

Both fields carry only `@IsString()`: [6](#0-5) 

The database entity enforces `length: 50` and `length: 256` at the column level: [7](#0-6) 

But the DTO validation passes before the DB constraint fires, so the full string is deserialized, logged, and processed by the service layer on every request.

**Root cause 3 — Unbounded `UploadSignatureMapDto[]` array on the signers endpoint**

`POST /transactions/:transactionId/signers` accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array-size cap: [8](#0-7) 

The service layer also fans out via `Promise.all` over the array: [9](#0-8) 

**Global body limit is 2 MB — insufficient mitigation**

The only server-wide guard is: [10](#0-9) 

Within 2 MB, a minimal `CreateTransactionGroupItemDto` (small hex-encoded transaction, short strings) occupies roughly 300–500 bytes, allowing 4,000–6,000 items per request. Each item triggers protobuf deserialization, signature verification, and a DB write.

## Impact Explanation

A single authenticated user can submit one crafted `POST /transaction-groups` request that:
- Saturates all available CPU cores with parallel `Promise.all` cryptographic operations (`publicKey.verify`, `getTransactionHash`).
- Exhausts the PostgreSQL connection pool (configured at `POSTGRES_MAX_POOL_SIZE=2` in the example env) with a massive batch insert inside a single DB transaction.
- Causes the Node.js process to OOM-crash if enough items are packed into the 2 MB window.
- Degrades or denies service for all other users sharing the same API instance.

The `name`/`description` gap amplifies this: each item in the group can carry an arbitrarily long string that is fully deserialized and logged before the DB rejects it, adding memory pressure per item.

## Likelihood Explanation

Any user who has completed registration and email verification (`VerifiedUserGuard`) and has at least one uploaded key can reach this endpoint. No admin role is required. The IP-based throttler (`IpThrottlerModule`) limits request frequency but does not cap payload complexity; a single request within the rate limit is sufficient to trigger the issue: [11](#0-10) 

The default `GLOBAL_MINUTE_LIMIT=10000` and `GLOBAL_SECOND_LIMIT=1000` are extremely permissive: [12](#0-11) 

## Recommendation

1. **Add `@ArrayMaxSize(N)` to `groupItems`** in `CreateTransactionGroupDto` (e.g., `@ArrayMaxSize(50)`) to cap the number of items per request. [1](#0-0) 

2. **Add `@MaxLength(50)` to `name` and `@MaxLength(256)` to `description`** in `CreateTransactionDto` to enforce DB-level constraints at the DTO layer before any processing occurs. [6](#0-5) 

3. **Add `@ArrayMaxSize(N)` to the signers endpoint** or enforce a cap inside `transformAndValidateDto` before the `Promise.all` fan-out. [8](#0-7) 

4. **Consider processing `groupItems` sequentially or in bounded batches** rather than unbounded `Promise.all` to limit concurrent CPU and DB pressure. [4](#0-3) 

## Proof of Concept

```http
POST /transaction-groups HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "description": "dos",
  "atomic": false,
  "sequential": false,
  "groupItems": [
    {
      "seq": 1,
      "transaction": {
        "name": "A",
        "description": "B",
        "transactionBytes": "<minimal_valid_hex_encoded_hedera_tx>",
        "creatorKeyId": <valid_key_id>,
        "signature": "<valid_signature_hex>",
        "mirrorNetwork": "testnet"
      }
    },
    // ... repeat 4000–6000 times within the 2 MB body limit
  ]
}
```

Each item in `groupItems` triggers a concurrent `validateAndPrepareTransaction` call (protobuf parse + crypto verify + hash + DB lookup) via `Promise.all`. With 4,000+ items, the Node.js event loop stalls, CPU pegs at 100%, and the PostgreSQL connection pool is exhausted for the duration of the request, denying service to all other users.
