Audit Report

## Title
Authorization Bypass in `removeTransactionApprover`: Approver ID Not Validated Against Authorized Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller as creator of `:transactionId`, but then passes the approver `:id` directly to `removeTransactionApprover()` with no cross-check that the approver belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction.

## Finding Description

**Root cause — decoupled authorization and action:**

In `approvers.controller.ts`, the `DELETE /:id` handler performs two independent, uncorrelated calls: [1](#0-0) 

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
```

The service method `removeTransactionApprover` receives only the approver `id` and performs no ownership verification against any transaction: [2](#0-1) 

The `transactionId` URL parameter is used solely to authorize the caller as creator of *that* transaction; it is never compared against the `transactionId` stored on the approver being deleted.

By contrast, `updateTransactionApprover` in the same service correctly performs this cross-check before acting: [3](#0-2) 

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
```

The analogous guard is entirely absent from `removeTransactionApprover`.

## Impact Explanation

An attacker can silently delete any approver node (or entire approver subtree, via `removeNode`) from any transaction in the system. This directly undermines the multi-signature approval workflow: required approvers can be removed, allowing transactions to advance to execution without the mandated approvals. In an organization context where high-value Hedera transactions (account updates, node operations, large transfers) require multi-party sign-off, this constitutes unauthorized state mutation that bypasses a core security control. [4](#0-3) 

## Likelihood Explanation

The attacker requires only a valid authenticated account (normal user, no admin privileges). Approver IDs are sequential integers, making enumeration trivial. The endpoint is a standard REST `DELETE` call. Any registered user who has ever created a transaction satisfies the precondition.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and verify its `transactionId` matches the `transactionId` URL parameter — exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the authorized transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller call site:

```typescript
await this.approversService.removeTransactionApprover(id, transactionId);
``` [5](#0-4) 

## Proof of Concept

1. Attacker registers as a normal user and creates **Transaction A** (attacker is creator).
2. Attacker enumerates or guesses approver ID `X` belonging to **Transaction B** (owned by a victim). IDs are sequential integers.
3. Attacker sends:
   ```
   DELETE /transactions/A/approvers/X
   Authorization: Bearer <attacker_token>
   ```
4. Controller calls `getCreatorsTransaction(A, attacker)` — **passes** (attacker owns A).
5. Controller calls `removeTransactionApprover(X)` — **deletes approver X from Transaction B** with no ownership check.
6. Transaction B's approval structure is silently mutated; required approvers are gone. [1](#0-0) [2](#0-1)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
