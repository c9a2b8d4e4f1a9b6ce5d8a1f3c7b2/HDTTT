Audit Report

## Title
Creator of Any Transaction Can Remove Approvers Belonging to Any Other Transaction

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but passes the unrelated `:id` (approver ID) directly to `removeTransactionApprover()` with no check that the approver belongs to `:transactionId`. Any authenticated user who has created at least one transaction can soft-delete approvers from any other transaction.

## Finding Description

**Root cause — missing cross-resource ownership check in the delete handler**

In `approvers.controller.ts`, the handler:

```typescript
@Delete('/:id')
async removeTransactionApprover(...) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ① checks ownership of transactionId only
  await this.approversService.removeTransactionApprover(id);               // ② deletes approver `id` with no ownership check
  return true;
}
``` [1](#0-0) 

Step ① (`getCreatorsTransaction`) only confirms the caller is the creator of the URL-supplied `transactionId`. It does not inspect the approver record at all. [2](#0-1) 

Step ② (`removeTransactionApprover`) fetches the approver by `id` and soft-deletes it via `removeNode`. There is no assertion that `approver.transactionId === transactionId`. [3](#0-2) 

**The sibling `updateTransactionApprover` does perform this check**, making the omission in the delete path an inconsistency:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [4](#0-3) 

## Impact Explanation

- **Unauthorized state mutation**: Any authenticated user can remove approvers from any other user's transaction, corrupting the approval workflow.
- **Approval bypass**: If a transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`, the attacker can silently strip required approvers, allowing the transaction to proceed without the intended multi-party authorization.
- **Integrity side-effect**: `emitTransactionStatusUpdate` inside `removeTransactionApprover` fires with `approver.transactionId` — the *victim's* transaction ID — potentially triggering downstream status recalculations on a transaction the attacker has no legitimate access to. [5](#0-4) 

## Likelihood Explanation

- **Preconditions**: The attacker only needs a valid account and to have created at least one transaction — both are normal user actions requiring no elevated privilege.
- **Approver ID discovery**: IDs are auto-incremented integers. An attacker who is an observer, signer, or approver on any transaction can retrieve approver IDs via `GET /transactions/:id/approvers`, which calls `getVerifiedApproversByTransactionId` and returns approver IDs. Sequential IDs make brute-force trivial even without that. [6](#0-5) 

## Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and assert that its `transactionId` matches the URL-supplied `transactionId`, mirroring the guard already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // NEW: resolve root and verify ownership
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

This mirrors the pattern already used in `updateTransactionApprover` at lines 386–391 of `approvers.service.ts`. [7](#0-6) 

## Proof of Concept

1. Attacker registers as a normal user and creates transaction `TX_A` (becomes its creator).
2. Attacker discovers victim approver ID `V_ID` (e.g., by being an observer on any transaction and calling `GET /transactions/:id/approvers`, or by brute-forcing sequential integers).
3. Attacker sends:
   ```
   DELETE /transactions/{TX_A}/approvers/{V_ID}
   ```
4. `getCreatorsTransaction(TX_A, attacker)` passes — attacker is the creator of `TX_A`.
5. `removeTransactionApprover(V_ID)` fetches the approver with ID `V_ID` (which belongs to a completely different transaction) and soft-deletes it with no further check.
6. The victim's transaction now has one fewer approver, and `emitTransactionStatusUpdate` fires against the victim's transaction ID. [8](#0-7)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
