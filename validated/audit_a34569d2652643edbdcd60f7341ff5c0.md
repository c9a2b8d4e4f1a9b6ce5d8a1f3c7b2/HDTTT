### Title
Missing Bounds Validation on `reminderMillisecondsBefore` Causes State Inconsistency via Arithmetic Producing Invalid Date

### Summary
The `reminderMillisecondsBefore` field in `CreateTransactionDto` accepts any number without a minimum or maximum constraint. When a user supplies an extremely large value (e.g., `1e308`), the arithmetic `tx.validStart.getTime() - dto.reminderMillisecondsBefore` produces `-Infinity`, creating an invalid `Date`. This causes the Redis `PXAT` call to receive `NaN`, which Redis rejects with an error. Because the DB transaction commits before the reminder is scheduled, the transaction is permanently saved to the database while the API returns HTTP 400 to the caller. The user cannot retry (duplicate-transaction guard blocks it), leaving their transaction in an orphaned state.

### Finding Description

**Root cause — no bounds on `reminderMillisecondsBefore`:**

`CreateTransactionDto` declares the field with only `@IsOptional()`
