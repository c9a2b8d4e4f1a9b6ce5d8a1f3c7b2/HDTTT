Audit Report

## Title
Any Transaction Creator Can Delete Approvers from Transactions They Do Not Own (IDOR → Approval Workflow DoS)

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns `:transactionId`, but then deletes the approver record identified by `:id` without confirming that approver belongs to `:transactionId`. Any authenticated user who owns at least one transaction can permanently remove approvers from any other transaction in the system.

## Finding Description

**Root cause — missing cross-object ownership check in `removeTransactionApprover`**

The controller performs two independent, unlinked calls:

```typescript
// approvers.controller.ts lines 108-109
await this.approversService.getCreatorsTransaction(transactionId, user); // ① checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // ② deletes approver by id — no transaction check
``` [1](#0-0) 

`removeTransactionApprover` fetches the approver purely by its primary key and soft-deletes it with no `transactionId` filter:

```typescript
// approvers.service.ts lines 534-543
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);  // no transactionId filter
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

`getTransactionApproverById` queries only on `id`:

```typescript
// approvers.service.ts lines 72-74
const find: FindOneOptions<TransactionApprover> = {
  relations: ['approvers'],
  where: { id },
};
``` [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly validates cross-ownership before acting:

```typescript
// approvers.service.ts lines 386-394
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [4](#0-3) 

The delete path has no equivalent guard. The `removeNode` function also recursively soft-deletes the entire subtree rooted at the targeted approver, amplifying the impact. [5](#0-4) 

## Impact Explanation
An attacker with no privileged access can permanently remove required approvers from any transaction in the system. Because `removeNode` performs a recursive soft-delete of the entire approver subtree, a single request can wipe an entire approval tree. If the deleted approver(s) were required to meet the approval threshold, the transaction can never reach `WAITING_FOR_EXECUTION` and will expire or be permanently stuck. Soft-deletes are not easily reversible through the API, making this a permanent, unrecoverable DoS on targeted transactions' approval workflows. [5](#0-4) 

## Likelihood Explanation
- **Precondition**: attacker must be an authenticated, verified user who has created at least one transaction — a normal product action requiring no special privilege.
- **Approver ID discovery**: approver IDs are auto-incremented integers. Any user who is legitimately an observer, signer, or approver of any transaction can read approver IDs via `GET /transactions/:id/approvers`. Sequential enumeration is also trivial.
- **No race condition required**: this is a direct, synchronous API call with no timing dependency.

Likelihood is **high**. [6](#0-5) 

## Recommendation
In `removeTransactionApprover` (or in the controller before calling it), add the same cross-ownership guard already present in `updateTransactionApprover`:

1. After fetching the approver by `id`, call `getRootNodeFromNode(approver.id)` to obtain the root of the approver tree.
2. Assert `rootNode.transactionId === transactionId`. If not, throw `UnauthorizedException`.
3. Then proceed with `removeNode`.

This mirrors the existing pattern at lines 386–394 of `approvers.service.ts` and closes the IDOR gap. [4](#0-3) 

## Proof of Concept

```
# Setup
# Alice creates transaction 1 (Alice is creator)
# Alice adds approver with id=10 to transaction 1

# Bob creates transaction 2 (Bob is creator)
# Bob adds approver with id=20 to transaction 2 (required for approval)

# Attack: Bob deletes Alice's approver (id=10) using his own transaction (id=2) as the ownership anchor
DELETE /transactions/2/approvers/10
Authorization: Bearer <Bob's JWT>

# Step-by-step:
# 1. getCreatorsTransaction(2, Bob) → PASSES (Bob owns transaction 2)
# 2. removeTransactionApprover(10) → fetches approver id=10 (belongs to transaction 1), no transactionId check
# 3. removeNode(10) → soft-deletes approver 10 and its entire subtree
# 4. Transaction 1's approval workflow is permanently broken

# Symmetrically, Bob can target approver id=20 on transaction 1 using:
DELETE /transactions/2/approvers/20
# This removes Bob's own required approver from transaction 2 — or any other transaction's approver
``` [6](#0-5) [2](#0-1)

        (
