Audit Report

## Title
Cross-Transaction Approver Deletion via Missing Ownership Cross-Check on DELETE Endpoint

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the authenticated user owns the transaction identified by `transactionId`, but then deletes the approver identified by the separate URL parameter `id` without verifying that approver belongs to that same transaction. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Vulnerability class:** Authorization bypass — authorization check is performed on one entity while the destructive action is performed on a different, unvalidated entity.

**Root cause — `approvers.controller.ts` lines 102–113:**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
  return true;
}
``` [1](#0-0) 

**`removeTransactionApprover` in `approvers.service.ts` lines 534–544:**

```typescript

The service never checks whether the approver's root node `transactionId` matches the `transactionId` supplied in the URL.

**Contrast with `updateTransactionApprover`**, which correctly performs this cross-check at lines 386–391:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The `removeNode` helper at lines 205–231 recursively soft-deletes the target approver and all its children via a recursive CTE, regardless of which transaction they belong to. [4](#0-3) 

## Impact Explanation

An attacker who is the creator of **any** transaction can:

1. Enumerate approver IDs from a victim's transaction via `GET /transactions/:victimTxId/approvers` (accessible to any participant) or by sequential integer enumeration.
2. Call `DELETE /transactions/{attacker_tx_id}/approvers/{victim_approver_id}`.
3. `getCreatorsTransaction` passes — attacker owns `attacker_tx_id`.
4. `removeTransactionApprover(victim_approver_id)` executes, soft-deleting the victim's approver and its entire child subtree.

This allows stripping all approval requirements from any transaction, causing transactions that require multi-party approval to become executable without those approvals. In the Hedera context (transfers, key changes, file updates), this is a direct integrity failure in the trust model.

## Likelihood Explanation

- **Attacker preconditions:** Authenticated user with no admin privileges who has created at least one transaction (trivially achievable by any registered user).
- **Knowledge required:** The approver `id` of the target transaction. IDs are sequential integers, discoverable via the `GET` approvers endpoint or brute-force.
- **No privileged access required.** Any regular organization member can exploit this.

## Recommendation

Apply the same cross-ownership check already present in `updateTransactionApprover` to the `removeTransactionApprover` service method. Before calling `removeNode`, resolve the root node of the target approver and verify its `transactionId` matches the `transactionId` from the URL:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service method, mirroring the pattern used in `updateTransactionApprover`. [5](#0-4) 

## Proof of Concept

**Setup:**
- Attacker (User A) creates Transaction T_A and is its creator.
- Victim (User B) creates Transaction T_B with approver records (IDs e.g. 42, 43, 44).

**Exploit:**
```
DELETE /transactions/{T_A}/approvers/42
Authorization: Bearer <User A's token>
```

**Result:**
- `getCreatorsTransaction(T_A, UserA)` → passes (A owns T_A).
- `removeTransactionApprover(42)` → fetches approver 42 (belongs to T_B), calls `removeNode(42)`, recursively soft-deletes approver 42 and all its children from T_B.
- T_B's approval requirements are silently removed with no authorization error.
