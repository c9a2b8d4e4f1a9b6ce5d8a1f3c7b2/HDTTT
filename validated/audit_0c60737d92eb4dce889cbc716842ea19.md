### Title
`executeTransaction` Callable on Canceled (and Other Terminal-State) Transactions Due to Missing Status Guard

### Summary
The `executeTransaction` method in `TransactionsService` enforces only an `isManual` check before proceeding. It performs no validation of the transaction's current `status`. A transaction creator can cancel a transaction and then immediately call `PATCH /transactions/execute/:id` on it, bypassing the cancellation and either re-activating the transaction for automatic chain submission or directly submitting it to the Hedera network.

### Finding Description

**Root cause — no status guard in `executeTransaction`:** [1](#0-0) 

The function fetches the transaction, checks only `isManual`, then either flips `isManual = false` (future `validStart`) or calls `executeService.executeTransaction(transaction)` (past `validStart`). No check is made against `transaction.status`.

**`cancelTransaction` does NOT soft-delete — the row remains visible:** [2](#0-1) 

`cancelTransactionWithOutcome` only issues an `UPDATE … SET status = 'CANCELED'`. It never calls `softRemove`, so `deletedAt` stays `null`. TypeORM's default `find` therefore still returns the row, and `getTransactionForCreator` succeeds on a canceled transaction.

**Contrast with other state-guarded operations:**

`importSignatures` and `approveTransaction` both explicitly reject any status outside `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION`: [3](#0-2) [4](#0-3) 

`executeTransaction` has no equivalent guard.

**Exploit path:**
1. Attacker (normal authenticated user) creates a manual transaction (`isManual = true`). Transaction is stored with `status = WAITING_FOR_SIGNATURES`.
2. Attacker cancels it via `PATCH /transactions/cancel/:id`. Status becomes `CANCELED`; row is **not** soft-deleted.
3. Attacker calls `PATCH /transactions/execute/:id`. `getTransactionForCreator` finds the row (deletedAt is null). The `isManual` check passes. No status check exists.
4. Branch A (`validStart > now`): `isManual` is set to `false`, re-queuing the canceled transaction for automatic chain-service pickup.
   Branch B (`validStart ≤ now`): `executeService.executeTransaction(transaction)` is called directly on the canceled transaction, submitting it to the Hedera network.

The same path applies to `EXPIRED`, `FAILED`, `REJECTED`, and `ARCHIVED` transactions that have `isManual = true`.

**Secondary issue — `archiveTransaction` bypasses status check when `isManual = true`:** [5](#0-4) 

The guard throws only when the status is wrong **and** `isManual` is false. If `isManual = true`, any status (including `EXECUTED`, `CANCELED`, `FAILED`) can be archived, corrupting the audit trail.

### Impact Explanation

- **Branch A**: A canceled transaction is re-activated for automatic execution by the chain service, directly violating the cancellation invariant. Depending on the transaction type (e.g., `AccountUpdateTransaction`, `NodeUpdateTransaction`), this can cause unauthorized on-chain state changes.
- **Branch B**: A direct submission of a canceled or already-executed transaction to Hedera. Hedera will reject a duplicate `transactionId`, but the attempt itself is unauthorized and can cause confusion in the audit log and notification system.
- The secondary `archiveTransaction` issue allows a creator to overwrite a terminal status with `ARCHIVED`, permanently hiding the true outcome of a transaction from the audit trail.

### Likelihood Explanation

- Requires only a normal authenticated user account — no privileged role.
- The attacker is the transaction creator, which is a standard product role reachable by any registered user.
- The steps (create → cancel → execute) are all exposed via documented REST endpoints on the API controller. [6](#0-5) 

- No race condition or timing dependency is required; the attack is fully deterministic.

### Recommendation

Add an explicit status guard at the top of `executeTransaction` (and `archiveTransaction`) mirroring the pattern already used in `importSignatures` and `approveTransaction`:

```typescript
// executeTransaction — add before the isManual check
if (transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION) {
  throw new BadRequestException(ErrorCodes.OTIP);
}

// archiveTransaction — replace the current compound condition with
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

This removes the `isManual` bypass from `archiveTransaction` and adds the missing status guard to `executeTransaction`.

### Proof of Concept

```
# 1. Authenticate as a normal user and obtain JWT token.

# 2. Create a manual transaction
POST /transactions
{ "isManual": true, "mirrorNetwork": "testnet", ... }
→ Response: { "id": 42, "status": "WAITING_FOR_SIGNATURES", "isManual": true }

# 3. Cancel the transaction
PATCH /transactions/cancel/42
→ Response: true
# DB: status = 'CANCELED', deletedAt = NULL

# 4. Execute the canceled transaction (no status check blocks this)
PATCH /transactions/execute/42
→ Response: true   ← should have been rejected

# Observed: if validStart is in the future, DB now shows isManual = false
# for a CANCELED transaction, re-queuing it for chain-service execution.
# If validStart is in the past, executeService.executeTransaction() is
# called directly on the canceled transaction.
```

  async cancelTransactionWithOutcome(
    id: number,
    user: User,
  ): Promise<CancelTransactionOutcome> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (transaction.status === TransactionStatus.CANCELED) {
      return CancelTransactionOutcome.ALREADY_CANCELED;
    }

    if (!this.cancelableStatuses.includes(transaction.status)) {
      throw new BadRequestException(ErrorCodes.OTIP);
    }

    const updateResult = await this.repo
      .createQueryBuilder()
      .update(Transaction)
      .set({ status: TransactionStatus.CANCELED })
      .where('id = :id', { id })
      .andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })
      .execute();

    if (updateResult.affected && updateResult.affected > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        [{
          entityId: id,
          additionalData: {
            transactionId: transaction.transactionId,
            network: transaction.mirrorNetwork,
          },
        }],
      );

      return CancelTransactionOutcome.CANCELED;
    }

    // Race-safe fallback: state changed between read and update, so re-check current status.
    const latestTransaction = await this.getTransactionForCreator(id, user);
    if (latestTransaction.status === TransactionStatus.CANCELED) {
      return CancelTransactionOutcome.ALREADY_CANCELED;
    }
    if (!this.cancelableStatuses.includes(latestTransaction.status)) {
      throw new BadRequestException(ErrorCodes.OTIP);
    }
    throw new ConflictException('Cancellation conflict');

  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }

  async executeTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }
