### Title
Unauthenticated Transaction Signer Enumeration: Missing Access Control on `GET /transactions/:transactionId/signers`

### Summary
The `GET /transactions/:transactionId/signers` endpoint in `SignersController` performs no transaction-level access control, allowing any authenticated and verified user to retrieve the full list of signers (including their associated public keys) for any transaction in the system. The parallel endpoints for observers (`GET /transactions/:transactionId/observers`) and approvers (`GET /transactions/:transactionId/approvers`) both enforce per-transaction access checks, making this an exact analog of the external report's inconsistent validation pattern.

### Finding Description

**Root Cause ã Inconsistent Authorization Across Parallel Endpoints**

All three sub-resource controllers share the same controller-level guards:

```
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
```

These guards only confirm the caller is authenticated and email-verified. Transaction-level access control must be enforced inside the service layer. The observers and approvers services do this; the signers service does not.

**Observers** ã `ObserversService.getTransactionObserversByTransactionId` enforces access: [1](#0-0) 

**Approvers** ã `ApproversService.getVerifiedApproversByTransactionId` enforces access: [2](#0-1) 

**Signers** ã `SignersController.getSignaturesByTransactionId` passes **no user** to the service: [3](#0-2) 

The service method itself performs zero access control ã it issues a bare repository query keyed only on `transactionId`: [4](#0-3) 

**Exploit Path**

1. Attacker registers and verifies an account on the organization server (normal user flow, no privilege required).
2. Attacker enumerates transaction IDs (sequential integers starting from 1).
3. For each ID, attacker sends: `GET /transactions/<id>/signers` with a valid JWT.
4. The server returns all `TransactionSigner` rows for that transaction, including the `userKey` relation (public key material), with no check that the attacker is the creator, observer, signer, or approver of that transaction.

### Impact Explanation

Any authenticated user can:
- Enumerate every signer and their public keys for every transaction in the organization, including transactions they have no relationship to.
- Map public keys to user identities across the entire organization, breaking the confidentiality model that the observer/approver access controls are designed to enforce.
- Correlate signing patterns to infer organizational structure and key ownership.

This is a cross-tenant data exposure: the system's own access model (creator / observer / signer / approver) is enforced on observers and approvers but silently bypassed for signers.

### Likelihood Explanation

Exploitation requires only a valid JWT for a verified account ã the lowest privilege level in the system. The attack is a single authenticated HTTP GET request. No race condition, no chaining, no special timing. Any malicious insider or compromised low-privilege account can immediately exploit this.

### Recommendation

Apply the same access-control pattern used by `ObserversService` and `ApproversService`. Pass the authenticated `user` from the controller to the service, and gate the response on transaction membership:

In `SignersController.getSignaturesByTransactionId`, inject `@GetUser() user: User` and forward it to the service: [3](#0-2) 

In `SignersService`, add a pre-check mirroring `getTransactionObserversByTransactionId`: [5](#0-4) 

Reject the request if the user is not the creator, observer, signer, approver, or a required key-holder, and the transaction is not in a terminal (history) status.

    transactionId: number,
