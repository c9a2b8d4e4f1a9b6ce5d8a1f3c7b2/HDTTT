### Title
No Admin Role Revocation Mechanism — Permanent Privilege Escalation with No API-Level Remediation

### Summary
The Hedera Transaction Tool backend exposes a one-way admin elevation endpoint (`PATCH /auth/elevate-admin`) that permanently grants the `admin` flag to any user. There is no corresponding API endpoint or service method to revoke admin status. This is the direct analog to the Lybra `GovernanceTimelock` bug: privileged roles are granted but cannot be taken back through the application's own interface, leaving a compromised admin account with permanent, irrevocable system-wide privileges.

### Finding Description

The `AuthService.elevateAdmin` method unconditionally sets `admin: true` on a user record: [1](#0-0) 

The only exposed controller endpoint for role management is: [2](#0-1) 

There is no `PATCH /auth/revoke-admin`, no `demoteAdmin` service method, and no `updateUserById` call anywhere in the API layer that sets `admin: false`. The `AdminGuard` gates all privileged operations on this single boolean: [3](#0-2) 

The `User` entity stores this as a plain database column with no audit trail or time-bound expiry: [4](#0-3) 

Additionally, the bootstrap script `back-end/scripts/add-users.ts` hardcodes two admin accounts with the weak password `'159357'`: [5](#0-4) 

These accounts are created with `admin: true` and `status: UserStatus.NONE`, meaning they are immediately active and fully privileged with a known, weak credential.

### Impact Explanation

An admin account controls the entire user lifecycle of the organization backend:
- Creating new users (`POST /auth/signup`, guarded by `AdminGuard`)
- Elevating other users to admin (`PATCH /auth/elevate-admin`, guarded by `AdminGuard`)
- Deleting users (`DELETE /users/:id`) [6](#0-5) 

If an admin account is compromised, the attacker can:
1. Elevate attacker-controlled accounts to admin — creating persistent backdoors that survive the original account being soft-deleted.
2. Delete legitimate admin accounts, locking out the real operators.
3. Register new users and observe/approve transactions on their behalf.

Because there is no revocation endpoint, the only remediation path is direct PostgreSQL access (`UPDATE "user" SET admin = false WHERE id = ?`), which requires infrastructure-level access and is not documented as a recovery procedure.

### Likelihood Explanation

Medium. The attack requires an admin credential to be compromised first. However, the hardcoded weak-password admin accounts in `add-users.ts` (`password: '159357'`) lower the bar significantly if that script is ever run against a non-test environment. Even without that script, the absence of revocation means any credential leak — phishing, password reuse, session token theft — results in permanent privilege retention with no in-band remediation.

### Recommendation

1. **Add a revoke-admin endpoint**: Implement `PATCH /auth/revoke-admin` backed by `updateUserById(userId, { admin: false })`, protected by `AdminGuard` (admins can demote other admins, not themselves).
2. **Remove or gate the `add-users.ts` script**: The hardcoded weak-password admin accounts must never reach a production database. The script should be removed from the repository or gated behind an explicit `NODE_ENV=test` check.
3. **Soft-delete does not substitute for revocation**: Deleting a user removes their login ability but does not cleanly revoke the admin flag; a re-invited user via `signUpByAdmin` creates a fresh non-admin account, but this is not a documented or reliable revocation path.

### Proof of Concept

1. Admin A calls `PATCH /auth/elevate-admin` with `{ id: attackerUserId }` — attacker's account now has `admin: true`.
2. Admin A's account is compromised and the security team wants to revoke attacker's admin. They call every available API endpoint — no revocation endpoint exists.
3. Attacker uses their admin JWT to call `PATCH /auth/elevate-admin` with a second attacker-controlled account ID, creating a persistent backdoor.
4. Even if the original attacker account is soft-deleted via `DELETE /users/:id`, the second elevated account remains active with full admin privileges.
5. The only remediation requires direct DB access: `UPDATE "user" SET admin = false WHERE id = <attackerId>;` — an out-of-band operation with no in-application support. [1](#0-0) [5](#0-4)
