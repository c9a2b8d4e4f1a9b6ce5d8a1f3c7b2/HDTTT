Audit Report

## Title
Missing Parent Threshold Validation in `removeTransactionApprover` Allows Unsatisfiable Approval State

## Summary
`removeTransactionApprover` in `approvers.service.ts` deletes a child approver node via `removeNode` without checking or adjusting the parent threshold node. This is the exact guard that `updateTransactionApprover` applies in its detach path. The omission allows `parent.threshold > parent.approvers.length` to persist in the database, permanently blocking the transaction's approval workflow.

## Finding Description

The system enforces the invariant `threshold <= approvers.length` at creation time and in `updateTransactionApprover`. When `updateTransactionApprover` is called with `dto.listId = null` to detach a child, it explicitly adjusts the parent: [1](#0-0) 

The parallel deletion path, `removeTransactionApprover`, performs no such check. It fetches the approver and immediately delegates to `removeNode`: [2](#0-1) 

`removeNode` executes a recursive SQL soft-delete that marks the target node and all its descendants as deleted. It never reads or writes the parent record: [3](#0-2) 

The controller exposes this as `DELETE /transactions/:transactionId/approvers/:id`, gated only by a creator check: [4](#0-3) 

## Impact Explanation

After a child is removed, the parent's `threshold` column retains its original value while `approvers.length` decreases. The `isApproved` utility evaluates: [5](#0-4) 

When `approvals.length` can never reach `approver.threshold` (because there are fewer remaining children than the threshold requires), `isApproved` returns `null` indefinitely. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` with no recovery path short of expiry. All prior approver signatures are wasted and the multi-party workflow is broken.

## Likelihood Explanation

The transaction creator — a normal authenticated user — has direct access to `DELETE /transactions/:transactionId/approvers/:id`. No elevated privilege is required. The scenario arises naturally: a creator sets up a threshold group of N approvers with `threshold = N`, then removes one member (e.g., a user who left the organization), not realizing the threshold is now unsatisfiable. The system provides no warning or guard.

## Recommendation

In `removeTransactionApprover`, after calling `removeNode`, fetch the parent of the removed approver (via `approver.listId`) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If `newParentApproversLength === 0`, soft-remove the parent.
- Else if `newParentApproversLength < parent.threshold`, update `parent.threshold = newParentApproversLength`.

This should be wrapped in a database transaction to ensure atomicity. Additionally, fix the secondary bug on line 541: when the removed node is a child (`listId` is set), `approver.transactionId` is `null`, so the status-update event is emitted with `entityId: null`. The transaction ID should be resolved from the root node before deletion.

## Proof of Concept

1. Creator sets up a threshold group: parent node with `threshold = 2`, children `[A, B]`.
2. Creator calls `DELETE /transactions/1/approvers/<id_of_A>`.
3. `removeTransactionApprover` fetches A, calls `removeNode(A.id)` — A is soft-deleted.
4. Parent record in DB: `threshold = 2`, but only child B remains (`approvers.length = 1`).
5. User B approves — `isApproved` evaluates: `approvals.length (1) >= threshold (2)` → `false`. Returns `null`.
6. No further approvals are possible. The transaction never advances past `WAITING_FOR_SIGNATURES`.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
