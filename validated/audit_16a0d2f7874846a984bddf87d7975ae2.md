Audit Report

## Title
Cross-Transaction Authorization Bypass in `removeTransactionApprover` Allows Any Creator to Delete Approvers from Arbitrary Transactions

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller owns the transaction at `:transactionId`, but then deletes the approver at `:id` with no check that the approver belongs to that transaction. Any authenticated user who is the creator of at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Root cause — missing cross-transaction ownership check in the delete path:**

In `approvers.controller.ts`, `removeTransactionApprover` performs two independent, unlinked operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by raw id, no cross-check
``` [1](#0-0) 

`getCreatorsTransaction` only verifies that `transaction.creatorKey?.userId === user.id` for the URL-supplied `transactionId` — it says nothing about the approver `id`: [2](#0-1) 

`removeTransactionApprover` fetches the approver by primary key alone and soft-deletes its entire subtree via `removeNode`, with no check that the approver belongs to the verified transaction: [3](#0-2) 

`getTransactionApproverById` is entirely unscoped — it fetches any approver by `id` with no `transactionId` filter: [4](#0-3) 

**Contrast with `updateTransactionApprover`**, which correctly validates the cross-transaction boundary before acting:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [5](#0-4) 

The delete path is entirely missing this guard.

## Impact Explanation
An attacker with a valid authenticated session who has created at least one transaction can permanently soft-delete approvers from any other user's transaction. This:
- Breaks the multi-signature approval workflow — required approvers are silently removed, allowing transactions to proceed without the intended authorization threshold.
- Constitutes unauthorized state mutation on another user's data.
- Is irreversible in practice: `removeNode` sets `deletedAt` on the entire approver subtree, and all queries filter out soft-deleted records. [6](#0-5) 

## Likelihood Explanation
- **Precondition**: attacker must be a registered, verified user with at least one transaction of their own — a normal user role, no privilege escalation required.
- Approver IDs are sequential integers, observable from any transaction the attacker has read access to via `GET /transactions/:id/approvers`.
- The attack requires a single authenticated HTTP DELETE request with no special tooling.
- Likelihood: **High**.

## Recommendation
In `removeTransactionApprover` (service), after fetching the approver, resolve its root node and verify that `rootNode.transactionId === transactionId` before calling `removeNode` — exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover`. This mirrors the guard already present in `updateTransactionApprover`. [7](#0-6) 

## Proof of Concept

1. Eve registers an account and creates **Transaction A** — she is its creator.
2. Bob creates **Transaction B** with approver records (IDs are sequential integers, e.g., approver ID `42`).
3. Eve sends:
   ```
   DELETE /transactions/{transactionA_id}/approvers/42
   Authorization: Bearer <eve_token>
   ```
4. `getCreatorsTransaction(transactionA_id, eve)` passes — Eve is the creator of Transaction A.
5. `removeTransactionApprover(42)` executes — fetches approver 42 (Bob's approver on Transaction B), calls `removeNode(42)`, soft-deletes it and its entire subtree.
6. Bob's Transaction B now has fewer approvers than required, potentially allowing it to execute without the intended multi-party approval. [1](#0-0)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
