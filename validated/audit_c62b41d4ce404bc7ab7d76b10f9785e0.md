## Audit Report

## Title
`cutoffAt` Deadline Not Enforced in Signature Upload Path — Signers Can Submit After Creator-Set Cutoff

## Summary
The `Transaction` entity stores a `cutoffAt` field, documented as the hard deadline after which signers may no longer submit signatures. The server-side gate function `SignersService.validateTransactionStatus()`, which is the only validation executed before signatures are persisted via `uploadSignatureMaps`, never checks `cutoffAt`. Any authenticated signer can therefore submit signatures after the creator-intended deadline.

## Finding Description

**Root cause — missing `cutoffAt` check in `validateTransactionStatus`:**

`cutoffAt` is a nullable `Date` column on the `Transaction` entity: [1](#0-0) 

Its intended semantics are explicitly documented: [2](#0-1) 

It is accepted from the creator and persisted without modification at transaction creation time: [3](#0-2) 

When a signer calls `POST /transactions/:transactionId/signers` (i.e., `uploadSignatureMap` → `uploadSignatureMaps`), the only server-side validation before signatures are written is `validateTransactionStatus`: [4](#0-3) 

`validateTransactionStatus` checks only two conditions — status and SDK-level expiry — and returns `null` (no error) without ever consulting `cutoffAt`: [5](#0-4) 

The same gap exists in the secondary `importSignatures` path in `TransactionsService`, which also checks only status and SDK expiry: [6](#0-5) 

The controller endpoint that feeds into `uploadSignatureMaps` is: [7](#0-6) 

## Impact Explanation
The `cutoffAt` field is the mechanism by which a transaction creator defines a hard deadline for signature collection. Because it is never enforced server-side, any registered signer who holds a valid signing key can submit signatures after `cutoffAt` has elapsed. If enough signatures accumulate, `processTransactionStatus` may advance the transaction to `WAITING_FOR_EXECUTION`, and the chain service will subsequently submit it to the Hedera network. For financial transactions (token transfers, account updates, node operations), this means on-chain state — including asset balances — can be mutated after the creator believed the signature window was closed.

## Likelihood Explanation
The attacker is any authenticated, verified user who holds a signing key for the transaction and has received a signing notification. No privileged access, leaked credentials, or special network position is required. The signer simply delays their `POST /transactions/:transactionId/signers` call until after `cutoffAt`. The endpoint is reachable by any user who passes `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — all standard authenticated users. [8](#0-7) 

## Recommendation
Add a `cutoffAt` check inside `validateTransactionStatus` in `signers.service.ts`, immediately after the status check:

```typescript
if (transaction.cutoffAt && Date.now() > transaction.cutoffAt.getTime()) {
  return ErrorCodes.TCO; // Transaction cutoff exceeded
}
```

Apply the same guard in `importSignatures` in `transactions.service.ts`. A corresponding `ErrorCodes.TCO` (or equivalent) entry should be added to the error codes registry. The fix must be applied to both signing paths to close the gap completely.

## Proof of Concept

1. Creator creates a transaction with `cutoffAt` set to `T` (e.g., 1 hour from now).
2. The transaction is persisted with `status = WAITING_FOR_SIGNATURES`.
3. Time advances past `T`.
4. A signer calls:
   ```
   POST /transactions/:transactionId/signers
   Authorization: Bearer <valid_jwt>
   Content-Type: application/json

   { "id": <transactionId>, "signatureMap": { ... } }
   ```
5. `validateTransactionStatus` is called. It checks:
   - `status === WAITING_FOR_SIGNATURES` → passes.
   - `isExpired(sdkTransaction)` → passes (Hedera SDK expiry has not elapsed).
   - `cutoffAt` → **never checked**.
6. Signatures are persisted via `bulkInsertSigners`.
7. `processTransactionStatus` evaluates whether all required signatures are present; if so, it advances the transaction to `WAITING_FOR_EXECUTION`.
8. The chain service picks up the transaction and submits it to the Hedera network — after the creator's intended cutoff deadline.

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
