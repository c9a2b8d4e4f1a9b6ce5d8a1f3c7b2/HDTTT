### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Submission of Invalid Signatures to Advance Transaction State

### Summary
The `uploadSignatureMaps` endpoint in `signers.service.ts` verifies that submitted public keys belong to the authenticated user but **never cryptographically verifies that the signature bytes are valid** for the transaction body. The parallel `importSignatures` endpoint in `transactions.service.ts` explicitly calls `validateSignature` before accepting signatures. An authenticated required-signer can submit their own public key paired with arbitrary/invalid signature bytes, get recorded as having signed, advance the transaction to `WAITING_FOR_EXECUTION`, and cause it to fail permanently at the Hedera network — burning the fee payer's funds and destroying the transaction.

### Finding Description

**Root cause — missing validation in `processTransactionSignatures`:**

In `signers.service.ts`, the private method `processTransactionSignatures` (lines 217–267) iterates over the submitted `SignatureMap`, checks that each public key belongs to the authenticated user via `userKeyMap` lookup, and then unconditionally calls `sdkTransaction.addSignature(publicKey, map)`:

```typescript
let userKey = userKeyMap.get(raw);
if (!userKey) {
  userKey = userKeyMap.get(publicKey.toStringDer());
}
if (!userKey) throw new Error(ErrorCodes.PNY);

// Only add the signature once per unique key
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
``` [1](#0-0) 

There is **no call to `validateSignature`** or any equivalent cryptographic check on the signature bytes before they are added to the transaction.

**Contrast with `importSignatures`:**

The `importSignatures` method in `transactions.service.ts` (lines 545–553) explicitly validates the signature bytes before calling `addSignature`:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);

for (const publicKey of publicKeys) {
  sdkTransaction.addSignature(publicKey, map);
}
``` [2](#0-1) 

The existence of this explicit `validateSignature` call confirms that `addSignature` in the Hedera SDK does **not** perform cryptographic verification internally — otherwise the call would be redundant. The `uploadSignatureMaps` path is therefore missing this guard entirely.

**Exploit flow:**

1. Attacker is an authenticated user whose public key is listed as a required signer for a target transaction.
2. Attacker calls `POST /transactions/:transactionId/signers` with a `SignatureMap` containing their own public key but with **garbage/invalid signature bytes**.
3. `processTransactionSignatures` passes the `userKeyMap` check (key belongs to the user) and calls `sdkTransaction.addSignature(publicKey, map)` — the invalid bytes are embedded in the transaction.
4. `isSameBytes` is `false` (bytes changed), so the corrupted `transactionBytes` are persisted to the database. [3](#0-2) 
5. The attacker is recorded as a `TransactionSigner` — they will no longer receive signing reminders. [4](#0-3) 
6. `processTransactionStatus` is called; it checks `sdkTransaction._signerPublicKeys` against the required key list. Because `addSignature` added the attacker's public key to `_signerPublicKeys`, the threshold check passes and the transaction advances to `WAITING_FOR_EXECUTION`.
7. The chain service submits the transaction to the Hedera network. The network rejects it due to the invalid signature. The transaction is marked `FAILED` permanently.

**No access control on transaction ID:**

`loadTransactionData` fetches any transaction by ID without verifying the requesting user has any relationship to it:

```typescript
const transactions = await this.dataSource.manager.find(Transaction, {
  where: { id: In(transactionIds) },
});
``` [5](#0-4) 

Any authenticated user can attempt to submit signatures for any transaction ID they can enumerate or guess.

### Impact Explanation

- **Direct loss of funds**: The fee payer's HBAR is consumed when the Hedera network processes and rejects the transaction. The fee is non-refundable.
- **Permanent transaction destruction**: Once the transaction is submitted and fails, it cannot be re-executed (it is marked `FAILED`). High-value operations (e.g., `NodeUpdateTransaction`, `AccountUpdateTransaction`, treasury transfers) are permanently lost.
- **Integrity failure in multi-sig workflow**: The attacker is recorded as having signed, suppressing further signing reminders and deceiving other participants into believing the transaction is legitimately signed.

### Likelihood Explanation

- **Attacker preconditions**: A registered, authenticated user account. The attacker's public key must be a required signer for the target transaction (i.e., they were added as a signer by the transaction creator). This is a realistic scenario — a malicious insider or a compromised account.
- **No special privileges required**: Standard JWT authentication is sufficient. The `uploadSignatureMap` endpoint is protected only by `JwtAuthGuard` and `VerifiedUserGuard`. [6](#0-5) 
- **Ease of exploitation**: Constructing a `SignatureMap` with a valid public key and invalid signature bytes is trivial using the Hedera SDK.

### Recommendation

Add cryptographic signature verification in `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const { data: validatedKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

Additionally, enforce that the submitting user has an established relationship to the transaction (is a creator, signer, or observer) before processing any signature submission, to prevent enumeration-based attacks.

### Proof of Concept

1. Register two users (User A = transaction creator, User B = required signer).
2. User A creates a transaction requiring User B's signature. Transaction is in `WAITING_FOR_SIGNATURES`.
3. User B constructs a `SignatureMap` using their real public key but replaces the signature bytes with 64 zero bytes.
4. User B calls `POST /transactions/{id}/signers` with the malformed map.
5. **Expected (correct) behavior**: Server rejects with `ISNMPN` (invalid signature).
6. **Actual behavior**: Server accepts, persists corrupted `transactionBytes`, records User B as a signer, and advances the transaction to `WAITING_FOR_EXECUTION`.
7. The chain service submits the transaction; Hedera network returns `INVALID_SIGNATURE`. Transaction is permanently `FAILED`. Fee payer's HBAR is consumed.
