Audit Report

## Title
Transaction Creator Can Self-Approve Their Own Transaction by Adding Themselves as an Approver

## Summary
`ApproversService.createTransactionApprovers` verifies the caller is the transaction creator but does not prevent the creator from inserting their own `userId` into the `transaction_approver` table. Once inserted, `approveTransaction` — which only checks that the caller appears in the approvers list — accepts the creator's approval, completely bypassing the separation-of-duties control the approval workflow is designed to enforce.

## Finding Description

**Root Cause — `createTransactionApprovers`:**

`getCreatorsTransaction` is called at line 239 to confirm the *caller* is the creator. Inside the `createApprover` loop, the only user-identity check (lines 271–277) is a database existence count:

```ts
// approvers.service.ts:271-277
if (typeof dtoApprover.userId === 'number') {
  const userCount = await transactionalEntityManager.count(User, {
    where: { id: dtoApprover.userId },
  });
  if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
}
```

There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. The creator's own `userId` passes all validation and is inserted into `transaction_approver`. [2](#0-1) 

**Root Cause — `approveTransaction`:**

```ts
// approvers.service.ts:556-560
const userApprovers = approvers.filter(a => a.userId === user.id);
if (userApprovers.length === 0)
  throw new UnauthorizedException('You are not an approver of this transaction');
```

There is no check that `user.id !== transaction.creatorKey.userId`. Because the creator was inserted as an approver in the previous step, `userApprovers` is non-empty and the approval is recorded. [3](#0-2) 

**Secondary Path — `updateTransactionApprover`:**

When updating an existing approver's `userId` (lines 494–497), the same absence of a creator-identity check applies — only user existence is verified, not whether the new `userId` equals the creator's `userId`. [4](#0-3) 

## Impact Explanation
The approval workflow enforces separation of duties: an independent party must review and approve a transaction before it proceeds to execution. A creator who self-approves eliminates this control entirely. In an organizational context managing Hedera network operations (account creation, file operations, node management, fund transfers), this allows a single malicious or compromised user to unilaterally push transactions through the approval gate without any peer review, defeating the multi-party authorization model.

## Likelihood Explanation
The attacker is the transaction creator — a normal, authenticated user with no elevated privileges. The attack requires only two standard API calls that are part of the normal workflow (`POST /transactions/:id/approvers` then `POST /transactions/:id/approvers/approve`). No race condition, cryptographic break, or social engineering is needed. Any organization member who can create transactions can exploit this.

## Recommendation
Add an explicit self-approval guard in both `createTransactionApprovers` and `updateTransactionApprover` immediately after the user-existence check:

```ts
// In createTransactionApprovers, after line 276:
if (dtoApprover.userId === user.id)
  throw new Error('The transaction creator cannot be added as an approver');
```

```ts
// In updateTransactionApprover, after line 497:
if (dto.userId === user.id)
  throw new Error('The transaction creator cannot be set as an approver');
```

Optionally, add a complementary guard in `approveTransaction` as defense-in-depth:

```ts
// After line 560:
if (transaction.creatorKey?.userId === user.id)
  throw new UnauthorizedException('The transaction creator cannot approve their own transaction');
```

## Proof of Concept

1. **Setup:** User A is authenticated and belongs to an organization.
2. **Create transaction:** User A calls `POST /transactions` → becomes `creatorKey.userId` for transaction `T`.
3. **Self-add as approver:** User A calls:
   ```
   POST /transactions/T/approvers
   Body: { "approversArray": [{ "userId": A.id }] }
   ```
   `getCreatorsTransaction` passes (A is the creator). The user-existence check at line 272 passes (A exists). No self-approval guard exists. A is inserted into `transaction_approver` with `userId = A.id`. [1](#0-0) [2](#0-1) 

4. **Self-approve:** User A calls:
   ```
   POST /transactions/T/approvers/approve
   Body: { "userKeyId": A.keyId, "signature": <valid_sig>, "approved": true }
   ```
   `getVerifiedApproversByTransactionId` returns A's approver record. `userApprovers.filter(a => a.userId === A.id)` is non-empty. No creator-identity check exists. The approval is recorded. [6](#0-5) 

5. **Result:** User A has self-approved their own transaction, bypassing the organizational approval requirement entirely.
