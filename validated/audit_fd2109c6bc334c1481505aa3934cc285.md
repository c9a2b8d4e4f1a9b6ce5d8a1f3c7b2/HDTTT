### Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Overwriting `transactionBytes` of Already-Executed Transactions

### Summary
`SignersService.uploadSignatureMaps` reads transaction state, validates it, then writes updated `transactionBytes` in a separate step with no status guard in the SQL. The `bulkUpdateTransactions` helper issues an unconditional `UPDATE … WHERE id = ANY($N)` with no status condition. If the scheduler moves a transaction to `EXECUTED` (or `CANCELED`) between the initial read and the write, the stored bytes are silently overwritten, corrupting the audit record of what was actually submitted to Hedera.

### Finding Description

**Root cause — stale-read / no atomic check-and-set**

`uploadSignatureMaps` is split into four sequential async phases:

1. **Read** — `loadTransactionData` fetches the transaction row and existing signers outside any DB transaction. [1](#0-0) 

2. **Validate** — `validateTransactionStatus` checks the **stale in-memory snapshot** from step 1. [2](#0-1) 

3. **Persist** — `persistSignatureChanges` opens a DB transaction and calls `bulkUpdateTransactions`. [3](#0-2) 

4. **Write** — `bulkUpdateTransactions` issues:
```sql
UPDATE transaction
SET "transactionBytes" = CASE id … END,
    "updatedAt" = NOW()
WHERE id = ANY($N)          -- ← no status guard
``` [4](#0-3) 

There is no `AND status IN ('WAITING_FOR_SIGNATURES', 'WAITING_FOR_EXECUTION')` predicate. The write succeeds for any status the row currently holds.

**Contrast with the execution path**, which uses an atomic check-and-set:
```sql
WHERE id = :id AND status = :currentStatus
``` [5](#0-4) 

`executeTransaction` is also protected by a distributed `@MurLock`, but `uploadSignatureMaps` has no equivalent guard. [6](#0-5) 

**Race window**

```
T0  User calls POST /transactions/:id/signers
T1  loadTransactionData → status = WAITING_FOR_SIGNATURES  (snapshot)
T2  validateTransactionStatus → OK (uses T1 snapshot)
T3  Scheduler: processTransactionStatus → WAITING_FOR_EXECUTION
T4  Scheduler: executeTransaction → Hedera submit → status = EXECUTED
T5  persistSignatureChanges → bulkUpdateTransactions overwrites transactionBytes
    of the now-EXECUTED row (no status guard → UPDATE succeeds)
```

The scheduler runs every 10 seconds and the execution path is fast; the window is narrow but deterministically reachable by a signer who submits near `validStart`.

### Impact Explanation

- **Audit-trail corruption**: The `transactionBytes` stored in the database after step T5 contain a signature that was **not present** when the transaction was submitted to Hedera. The on-chain state is unaffected, but the backend's record of what was signed diverges from reality.
- **Falsified signing record**: A malicious signer can retroactively inject their key's signature into the stored bytes of an already-executed transaction, making it appear they signed when they did not (or vice-versa if the bytes are replaced with a version lacking a key).
- **Spurious status transitions**: `updateStatusesAndNotify` is called after the write using the stale in-memory transaction object. If `processTransactionStatus` re-evaluates the (now-corrupted) bytes and emits a status-change event for an `EXECUTED` transaction, downstream consumers (notifications, frontend) receive incorrect state. [7](#0-6) 

### Likelihood Explanation

- The attacker must be a **legitimate registered signer** for the target transaction (no privileged access required beyond normal user enrollment).
- The race window is narrow (~seconds around `validStart`) but the scheduler fires every 10 seconds and the signature upload endpoint is open to any authenticated user with a matching key.
- An attacker who controls the timing of their upload (e.g., submits just before `validStart`) can reliably hit the window without any special tooling — a simple retry loop suffices. [8](#0-7) 

### Recommendation

Add a status guard to `bulkUpdateTransactions` identical to the pattern used in `_executeTransaction`:

```sql
UPDATE transaction
SET "transactionBytes" = CASE id … END,
    "updatedAt" = NOW()
WHERE id = ANY($N)
  AND status IN ('WAITING_FOR_SIGNATURES', 'WAITING_FOR_EXECUTION')
```

Additionally, move the `loadTransactionData` call **inside** the DB transaction and use `SELECT … FOR UPDATE` (pessimistic lock) or re-validate the status inside the same transaction before writing, so the check and the write are atomic — mirroring the `AND status = :currentStatus` pattern already used in `_executeTransaction`. [9](#0-8) 

### Proof of Concept

1. Register as a normal user with a key that is a valid signer for transaction `T` (status `WAITING_FOR_SIGNATURES`, `validStart` = now + 15 s).
2. At `validStart − 2 s`, begin a loop that repeatedly calls:
   ```
   POST /transactions/T/signers
   Body: { signatureMap: <valid sig from attacker key> }
   ```
3. The scheduler fires at `validStart − 10 s` (`collateAndExecute`), then at `validStart + 5 s` (`addExecutionTimeout` → `executeTransaction`). The execution window overlaps with the upload loop.
4. Observe that after `T.status = EXECUTED`, at least one upload response returns HTTP 201 and the database row for `T.transactionBytes` now contains the attacker's signature — a signature that was **not** present in the bytes submitted to Hedera.
5. Confirm by comparing the stored bytes against the Hedera mirror-node record for the same transaction ID: the signatures differ.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (

      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

      return ErrorCodes.TE;
    }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }

  private async updateStatusesAndNotify(
    transactionsToProcess: Array<{ id: number; transaction: Transaction }>
  ) {
    if (transactionsToProcess.length === 0) return;

    // Process statuses in bulk
    let statusMap: Map<number, TransactionStatus>;
    try {
      statusMap = await processTransactionStatus(
        this.txRepo,
        this.transactionSignatureService,
        transactionsToProcess.map(t => t.transaction)
      );
    } catch (err) {
      console.error('Bulk status processing failed:', err);
      statusMap = new Map();
    }

    // Separate new statuses from unchanged
    const newStatusResults: number[] = [];
    const unchangedResults: number[] = [];

    for (const { id } of transactionsToProcess) {
      if (statusMap.has(id)) {
        newStatusResults.push(id);
      } else {
        unchangedResults.push(id);
      }
    }

    // Emit notifications
    if (newStatusResults.length > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        newStatusResults.map(id => ({ entityId: id }))
      );
    }
    if (unchangedResults.length > 0) {
      emitTransactionUpdate(
        this.notificationsPublisher,
        unchangedResults.map(id => ({ entityId: id }))
      );
    }
  }
