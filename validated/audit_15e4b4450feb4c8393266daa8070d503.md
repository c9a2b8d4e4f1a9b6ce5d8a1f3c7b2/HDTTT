### Title
Transaction Creator Can Manipulate Approval Outcome by Modifying `threshold` After Approvals Are Collected

### Summary

The `updateTransactionApprover` function in `approvers.service.ts` allows the transaction creator to change the `threshold` of an approver tree node at any time — including while the transaction is actively collecting approvals — with no check on the transaction's current status. Because the approval evaluation logic (`isApproved`) reads the **live** `threshold` value from the database rather than a value frozen at transaction creation, the creator can retroactively lower or raise the threshold to manipulate whether the transaction is considered approved.

### Finding Description

The `TransactionApprover` entity stores a mutable `threshold` column that determines how many child approvers in a tree must approve for that tree node to be considered satisfied. [1](#0-0) 

The `isApproved` utility function evaluates approval status by reading the **current** `threshold` directly from the approver object (which reflects the live database value): [2](#0-1) 

The `updateTransactionApprover` function in `approvers.service.ts` allows the transaction creator to update this `threshold` at any time. Critically, there is **no check on the transaction's current status** before the update is applied: [3](#0-2) 

The only guard is that the caller must be the transaction creator: [4](#0-3) 

This means the creator can freely change `threshold` while the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status — states where approvals are actively being collected and evaluated.

The `PATCH /:id` endpoint exposes this directly: [5](#0-4) 

### Impact Explanation

**Scenario A — Creator lowers threshold to force passage:**
1. Creator sets up a transaction with an approver tree requiring `threshold = 3` out of 5 approvers.
2. Only 2 approvers sign/approve.
3. Creator calls `PATCH /transactions/:id/approvers/:approverId` with `{ threshold: 2 }`.
4. The transaction is now considered approved with only 2 signatures, bypassing the originally agreed-upon 3-of-5 requirement.

**Scenario B — Creator raises threshold to block passage:**
1. Creator sets up a transaction with `threshold = 1` out of 3 approvers.
2. One approver approves, satisfying the original threshold.
3. Creator raises `threshold` to 3.
4. The transaction can no longer pass even though it met the threshold at the time of approval.

Both scenarios undermine the integrity of the multi-signature approval workflow, which is the core security guarantee of the system.

### Likelihood Explanation

The transaction creator is not a fully trusted role in the context of multi-party approval. The entire purpose of the approver system is to require independent sign-off from other users. The creator has direct API access to `PATCH /transactions/:transactionId/approvers/:id` and can exploit this at any point during the approval lifecycle. No special privileges beyond being the transaction creator are required.

### Recommendation

Add a transaction status check inside `updateTransactionApprover` before allowing any structural modification (threshold, userId, listId) to the approver tree. Modifications should only be permitted when the transaction is in a pre-approval state (e.g., `NEW`), not once it has entered `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

```typescript
// Inside updateTransactionApprover, after fetching the root transaction:
const tx = await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
if (
  tx.status !== TransactionStatus.NEW
) {
  throw new BadRequestException('Cannot modify approvers after approval has started');
}
```

### Proof of Concept

1. Creator creates a transaction (status: `NEW`) with an approver tree: `{ threshold: 3, approvers: [UserA, UserB, UserC, UserD, UserE] }`.
2. Transaction moves to `WAITING_FOR_SIGNATURES`.
3. UserA and UserB call `POST /transactions/:id/approvers/approve` — 2 of 3 required approvals collected.
4. Creator calls:
   ```
   PATCH /transactions/:id/approvers/:treeNodeId
   Body: { "threshold": 2 }
   ```
5. `updateTransactionApprover` updates `threshold` to `2` with no status check.
6. The chain service's next status evaluation reads the live `threshold = 2`, sees 2 approvals, and marks the transaction as fully approved — bypassing the original 3-of-5 requirement. [6](#0-5)

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
