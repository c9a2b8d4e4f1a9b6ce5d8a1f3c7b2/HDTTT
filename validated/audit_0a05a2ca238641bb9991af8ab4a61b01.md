Audit Report

## Title
Concurrent `uploadSignatureMaps` Calls Silently Overwrite Each Other's Signatures via Unlocked Read-Modify-Write on `transactionBytes`

## Summary
`SignersService.uploadSignatureMaps` reads `transactionBytes` outside any database transaction or row lock, deserializes the SDK transaction, appends new signatures, and writes the result back inside a separate database transaction. When two users sign the same transaction concurrently, both reads return the same original bytes, both compute their own updated blob, and the second write silently overwrites the first — permanently discarding one user's signature from the stored transaction.

## Finding Description

**Root cause — unlocked read-modify-write in `uploadSignatureMaps`:**

`loadTransactionData` fetches `transactionBytes` with a plain `find` (no `SELECT FOR UPDATE`): [1](#0-0) 

`processTransactionSignatures` deserializes those bytes and appends the caller's signatures: [2](#0-1) 

`bulkUpdateTransactions` (called from `persistSignatureChanges`) then writes the result back with a raw `UPDATE … SET "transactionBytes" = CASE id … END` — a direct overwrite, not an accumulation: [3](#0-2) 

The `dataSource.transaction(...)` block that wraps the write at line 317 does **not** cover the read at line 131, so the read-modify-write is not atomic: [4](#0-3) 

The same race exists in `importSignatures` in `transactions.service.ts`, which follows the identical pattern — plain `find` read, in-memory `addSignature`, then raw `UPDATE` write — all without any row lock: [5](#0-4) [6](#0-5) [7](#0-6) 

## Impact Explanation
Signatures are silently discarded. A multi-signature transaction that has collected the required number of signers in the `TransactionSigner` table may never have those signatures present in `transactionBytes`, so the Hedera network rejects it at execution time. The transaction is permanently stuck: it cannot be re-signed (the valid-start window expires) and cannot be executed. This constitutes an unrecoverable loss of a scheduled, coordinated transaction.

## Likelihood Explanation
Concurrent signing is the normal workflow in a multi-user organization: the UI notifies all required signers simultaneously, and they are expected to sign around the same time. No attacker capability beyond a valid authenticated session is required. A malicious co-signer can deliberately race their own signing request against another user's to suppress that user's signature, preventing the transaction from ever executing.

## Recommendation
Move the `transactionBytes` read **inside** the same `dataSource.transaction(...)` block as the write, and acquire a row-level lock using `SELECT … FOR UPDATE` (via TypeORM's `setLock('pessimistic_write')` or a raw `SELECT … FOR UPDATE` query). This ensures that concurrent requests serialize on the lock and each request reads the latest committed bytes before computing and writing its update. The same fix must be applied to `importSignatures` in `transactions.service.ts`.

## Proof of Concept

```
1. Transaction T requires signatures from User A and User B.
2. User A calls POST /transactions/T/signers
   → loadTransactionData reads transactionBytes (0 sigs) [line 131]
3. User B calls POST /transactions/T/signers concurrently
   → loadTransactionData reads the same transactionBytes (0 sigs) [line 131]
4. User A's request finishes first:
   → processTransactionSignatures appends A's sig [line 251]
   → bulkUpdateTransactions writes bytes containing only A's sig [line 365]
5. User B's request finishes second:
   → processTransactionSignatures appends B's sig to the original 0-sig bytes [line 251]
   → bulkUpdateTransactions writes bytes containing only B's sig [line 365]
   → A's signature is permanently gone from transactionBytes
6. The stored transaction now has 1 of 2 required signatures in transactionBytes;
   the Hedera network rejects it at execution time.
   The TransactionSigner table may show both signers, creating a false sense of completeness.
```
