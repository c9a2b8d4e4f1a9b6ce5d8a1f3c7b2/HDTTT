### Title
Remote DoS via Malicious `totalItems` in Server-Driven Pagination Loops

### Summary
The Electron front-end contains two functions — `getAllInAppNotifications` and `getAllUserKeys` — that drive pagination loops whose termination condition is controlled entirely by the `totalItems` field returned by the organization server. A malicious (or compromised) organization server can return an arbitrarily large `totalItems` while returning zero items per page, causing the loop to spin indefinitely. This freezes the Electron renderer process, making the application completely unresponsive.

### Finding Description

**Root cause — `getAllInAppNotifications`** [1](#0-0) 

```
let page = 1;
const pageSize = 100;
let final = false;
while (!final) {
    const { data } = await axiosWithCredentials.get(...);
    const totalItems = data.totalItems;          // ← re-read from server every iteration
    notifications.push(...data.items);
    final = notifications.length >= totalItems;  // ← termination fully server-controlled
    page++;
}
```

`totalItems` is re-fetched from the server on every iteration. There is no upper-bound cap, no page-count guard, and no break path when `data.items` is empty. If the server returns `totalItems = Number.MAX_SAFE_INTEGER` and `items = []`, `final` is never set to `true` and the loop runs forever, issuing HTTP requests in a tight async loop.

**Root cause — `getAllUserKeys`** [2](#0-1) 

```
do {
    const { data, error } = await safeAwait(getUserKeysPaginated(...));
    if (data) {
        totalItems = data.totalItems;            // ← updated from server each iteration
        allUserKeys.push(...data.items);
        page++;
    }
    if (error) { break; }
} while (allUserKeys.length < totalItems);       // ← termination fully server-controlled
```

Identical pattern: `totalItems` is overwritten from the server response on every successful page fetch. A server that returns a huge `totalItems` with an empty `items` array keeps the loop alive indefinitely.

**Call sites that trigger the loops**

`getAllInAppNotifications` is called inside `fetchNotifications()` in `storeNotifications.ts`, which runs for every organization the user is connected to: [3](#0-2) 

`getAllUserKeys` is called inside `loadContacts()` in `storeContacts.ts`, triggered whenever the user logs into an organization: [4](#0-3) 

**Analog to the Nimbus finding**

The Nimbus bug reads `lookupCgcFromPeer()` multiple times without caching; a peer returns a large value on the third call, causing an infinite loop in `handle_custody_groups`. Here, `data.totalItems` is the equivalent of `lookupCgcFromPeer()` — it is re-read from an external source on every iteration, and a large value causes an infinite loop. The vulnerability class is identical: **unbounded computation driven by an uncached, externally-controlled value**.

### Impact Explanation

The Electron renderer process runs the Vue 3 UI. An infinite async loop in the renderer blocks all UI interaction, making the application completely unresponsive (DoS). The user cannot sign transactions, manage keys, or close the organization session without force-killing the process. Because `fetchNotifications` is called on a recurring basis (watch triggers, WebSocket events), the loop can be re-triggered automatically after a restart if the user remains connected to the malicious server.

### Likelihood Explanation

The attacker must operate a server that the victim connects to as an organization server. This is a **malicious peer/integrator** scenario: no privileged access to the Hedera network or to the victim's machine is required — anyone can run a NestJS-compatible HTTP server. The attack surface is realistic in enterprise deployments where employees may be directed to connect to a new organization server URL. The exploit requires zero authentication bypass; the server simply returns a crafted JSON body `{ "totalItems": 9007199254740991, "items": [] }` for any paginated endpoint.

### Recommendation

1. **Cap the loop with a maximum page count** derived from a locally-trusted value (e.g., the first response's `totalItems` capped at a reasonable maximum such as 10,000 items):

```typescript
const MAX_ITEMS = 10_000;
// capture totalItems once from the first response and never update it
if (page === 1) totalItems = Math.min(data.totalItems, MAX_ITEMS);
```

2. **Add a stale-page guard**: if a page returns zero items but `totalItems` still exceeds `collected`, break immediately — a well-behaved server never returns an empty page before the total is reached.

3. **Validate `totalItems`** against a hard upper bound before entering the loop.

### Proof of Concept

1. Stand up a minimal HTTP server that responds to any `GET /notifications?...` or `GET /user-keys?...` with:
   ```json
   { "totalItems": 9007199254740991, "items": [], "page": 1, "size": 100 }
   ```
2. In the Hedera Transaction Tool, add this server as an organization server URL and log in.
3. Observe that `getAllInAppNotifications` (called by `fetchNotifications` in `storeNotifications.ts`) enters an infinite loop: the renderer process pegs one CPU core, the UI freezes, and no user interaction is possible.
4. The same result is reproducible via `getAllUserKeys` (called by `loadContacts` in `storeContacts.ts`) when the contacts view is loaded.
