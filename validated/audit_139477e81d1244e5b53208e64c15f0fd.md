Audit Report

## Title
`IpThrottlerGuard` Uses Attacker-Controlled `x-forwarded-for` Header as Rate-Limit Key, Enabling Complete Bypass of Global Rate Limiting

## Summary
`IpThrottlerGuard.getTracker()` resolves the throttle bucket key by reading `req.headers['x-forwarded-for']` before falling back to `req.ip`. Because `x-forwarded-for` is a plain HTTP header any client can set to an arbitrary value, an attacker can rotate through fake IP strings on every request, landing each request in a fresh, empty throttle bucket and making the global rate limit completely ineffective.

## Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`:**

`back-end/apps/api/src/guards/ip-throttler.guard.ts`, line 8:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [1](#0-0) 

`req.headers['x-forwarded-for']` is the raw, unvalidated HTTP header value. Any HTTP client can set it to any arbitrary string. The guard unconditionally prefers this header over `req.ip`, which is the value Express derives from the actual TCP connection (and which respects the `trust proxy` setting). Because the attacker-supplied header wins, the rate-limit bucket key is fully attacker-controlled.

**Global application — `api.module.ts`:**

`IpThrottlerGuard` is registered as a global `APP_GUARD`, making it the primary rate-limiting layer for every route in the API service. [2](#0-1) 

**Throttler configuration — `ip-throttler.module.ts`:**

The throttler uses Redis storage keyed by the tracker value, with `GLOBAL_MINUTE_LIMIT` and `GLOBAL_SECOND_LIMIT` as the enforced limits. [3](#0-2) 

**Test suite confirms the spoofable-header path is the intended behavior:** [4](#0-3) 

## Impact Explanation

- **Brute-force credential attacks**: `POST /auth/login` has no per-user lockout; the only global brake is `IpThrottlerGuard`. With the guard bypassed, an attacker can enumerate passwords for any known email address without restriction.
- **OTP brute-force**: `POST /auth/verify-reset` is guarded only by `JwtBlackListOtpGuard` and `OtpJwtAuthGuard`. The global IP throttle is the only rate control. With unlimited requests the attacker can exhaust the OTP space within the TOTP window.
- **Resource exhaustion**: Unlimited unauthenticated requests can be directed at any endpoint, degrading service for legitimate users.

Severity: **High** — authentication brute-force is directly enabled with no privilege required.

## Likelihood Explanation

- Requires only the ability to send HTTP requests with a custom header — trivially achievable with `curl`, any HTTP library, or a browser's `fetch` API.
- No credentials, tokens, or insider access needed.
- The attack is fully automatable and silent (no anomalous TCP behavior).
- Deployments behind a reverse proxy that strips or overrides `x-forwarded-for` would be partially mitigated, but the code itself provides no such guarantee, and the guard's own test suite explicitly validates the spoofable-header path as correct behavior.

## Recommendation

Replace the attacker-controlled header lookup with the server-authoritative `req.ip` value, which Express derives from the actual TCP connection and respects the configured `trust proxy` setting:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

If the application must operate behind a trusted reverse proxy that sets `x-forwarded-for`, configure Express's `trust proxy` setting appropriately (e.g., `app.set('trust proxy', 1)`). Express will then populate `req.ip` with the correct forwarded IP, and the guard can use `req.ip` exclusively without ever reading the raw header directly.

## Proof of Concept

```bash
# Send 1000 login attempts, each with a unique spoofed IP, with no throttling applied
for i in $(seq 1 1000); do
  curl -s -o /dev/null -X POST https://target/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$i" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
```

Each request uses a unique `X-Forwarded-For` value, creating a fresh Redis throttle bucket for each. The `GLOBAL_MINUTE_LIMIT` and `GLOBAL_SECOND_LIMIT` counters are never exhausted for any single key, and all 1000 requests are processed without any rate-limit response (`429`).
