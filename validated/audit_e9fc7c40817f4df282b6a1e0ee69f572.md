Audit Report

## Title
Wrong Property Names in `getComponentServiceEndpoint` Cause Node Endpoint Data to Always Be Empty, Breaking Transaction Integrity Comparison

## Summary

In `front-end/src/renderer/utils/sdk/getData.ts`, the function `getComponentServiceEndpoint` accesses `serviceEndpoint.getIpAddressV4`, `serviceEndpoint.getPort`, and `serviceEndpoint.getDomainName` — names that do not exist as properties on the `@hiero-ledger/sdk` `ServiceEndpoint` object. All three resolve to `undefined`, causing every endpoint to be extracted as `{ ipAddressV4: '', port: '', domainName: '' }`. This silently corrupts all node endpoint data and renders the `transactionsDataMatch` integrity check blind to endpoint substitution in `NodeCreateTransaction` and `NodeUpdateTransaction`.

## Finding Description

**Root cause — wrong property names:**

In `getComponentServiceEndpoint` at lines 227–232:

```typescript
const ipAddressV4 =
  serviceEndpoint.getIpAddressV4 && serviceEndpoint.getIpAddressV4.length > 0
    ? serviceEndpoint.getIpAddressV4.join('.')
    : '';
const port = serviceEndpoint.getPort ? serviceEndpoint.getPort.toString() : '';
const domainName = serviceEndpoint.getDomainName || '';
``` [1](#0-0) 

The `@hiero-ledger/sdk` `ServiceEndpoint` class exposes plain JavaScript properties (`ipAddressV4`, `port`, `domainName`) and setter methods (`setIpAddressV4`, `setPort`, `setDomainName`). This is confirmed by the rest of the codebase: `createTransactions.ts` reads from the plain-object `ComponentServiceEndpoint` using `serviceEndpoint.ipAddressV4`, `serviceEndpoint.domainName`, `serviceEndpoint.port`, then calls `setIpAddressV4`/`setDomainName`/`setPort` on the SDK object. [2](#0-1) 

The names `getIpAddressV4`, `getPort`, and `getDomainName` do not exist on the SDK object. Each evaluates to `undefined` (falsy), so every branch falls through to the empty-string default. `getComponentServiceEndpoint` always returns `{ ipAddressV4: '', port: '', domainName: '' }` regardless of actual transaction content.

**Propagation into `getNodeData`:**

`getNodeData` calls `getComponentServiceEndpoints` (which maps `getComponentServiceEndpoint`) for both `gossipEndpoints` and `serviceEndpoints`, and calls `getComponentServiceEndpoint` directly for `grpcWebProxyEndpoint`: [3](#0-2) 

Every `NodeCreateTransaction` and `NodeUpdateTransaction` is therefore extracted with all endpoint IP/port/domain fields blank.

**Propagation into `transactionsDataMatch`:**

`transactionsDataMatch` compares two transactions by serialising the output of `getAllData` to JSON: [4](#0-3) 

Because `getAllData` → `getNodeData` → `getComponentServiceEndpoints` → `getComponentServiceEndpoint` always produces empty endpoint strings, two `NodeCreateTransaction` objects with completely different gossip/service endpoints produce identical JSON for those fields. `transactionsDataMatch` returns `true` for transactions that are materially different in their endpoint configuration, as long as other fields (`nodeAccountId`, `description`, `gossipCaCertificate`, `certificateHash`, `adminKey`, `declineReward`) remain the same.

**UI deception:**

`NodeDetails.vue` renders endpoint tables by calling `getComponentServiceEndpoints` directly on the transaction's endpoint arrays. Since the function always returns empty strings, reviewers see blank IP/port/domain columns for every node transaction. [5](#0-4) 

## Impact Explanation

- **Signature collation bypass**: The `transactionsDataMatch` guard — the only client-side integrity check between signing rounds — is completely ineffective for node transactions when only endpoint fields differ. A substituted transaction with attacker-controlled endpoints passes the check undetected.
- **Node governance integrity**: `NodeCreateTransaction` and `NodeUpdateTransaction` are privileged governance transactions. Silently substituting gossip/service endpoints means the Hedera network could be directed to communicate with attacker-controlled infrastructure.
- **UI deception**: Reviewers see blank endpoint fields and cannot verify what they are signing, making informed approval impossible.

## Likelihood Explanation

- Any authenticated user of the organisation backend who can reach the transaction storage endpoint can attempt to substitute transaction bytes during the signing window.
- The bug is deterministic and fires on every node transaction — no special timing or cryptographic capability is required.
- The attacker only needs to preserve non-endpoint fields (`nodeAccountId`, `description`, certificates, `adminKey`, `declineReward`) while substituting endpoint data, which is straightforward.
- Node governance transactions are rare but high-value targets.

## Recommendation

Replace the non-existent Java-style getter names with the correct plain property names in `getComponentServiceEndpoint`:

```typescript
export const getComponentServiceEndpoint = (
  serviceEndpoint: ServiceEndpoint | null,
): ComponentServiceEndpoint | null => {
  if (!serviceEndpoint) return null;

  const ipAddressV4 =
    serviceEndpoint.ipAddressV4 && serviceEndpoint.ipAddressV4.length > 0
      ? Array.from(serviceEndpoint.ipAddressV4).join('.')
      : '';
  const port = serviceEndpoint.port ? serviceEndpoint.port.toString() : '';
  const domainName = serviceEndpoint.domainName || '';

  return { ipAddressV4, port, domainName };
};
``` [6](#0-5) 

## Proof of Concept

1. Create two `NodeCreateTransaction` objects with identical `nodeAccountId`, `description`, `gossipCaCertificate`, `certificateHash`, `adminKey`, and `declineReward`, but with entirely different `gossipEndpoints` (e.g., `10.0.0.1:50211` vs `192.168.99.1:9999`).
2. Call `transactionsDataMatch(tx1, tx2)`.
3. Both transactions produce `gossipEndpoints: [{ ipAddressV4: '', port: '', domainName: '' }]` in their `getAllData` output due to the wrong property names.
4. `JSON.stringify` of both outputs is identical; `transactionsDataMatch` returns `true`.
5. An attacker who replaces stored transaction bytes (keeping certificates/keys identical but changing endpoints) will pass the integrity check on every reviewing client.
