Audit Report

## Title
`archiveTransaction` Bypasses Status Guard for Manual Transactions, Allowing State Corruption from Terminal States

## Summary
The `archiveTransaction` method in `back-end/apps/api/src/transactions/transactions.service.ts` uses a logically flawed `&&` guard that is entirely bypassed when `transaction.isManual === true`. This allows any authenticated creator of a manual transaction to archive it from any status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `REJECTED` — corrupting the transaction lifecycle state machine.

## Finding Description

**Root cause — confirmed flawed guard logic:**

The guard at lines 711–718 reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [1](#0-0) 

When `isManual === true`, `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false` regardless of `transaction.status`. The `throw` is never reached, and the unconditional `repo.update` on line 720 overwrites the status to `ARCHIVED`.

**`isManual` is fully creator-controlled:**

`CreateTransactionDto` exposes `isManual` as an optional boolean field with no restrictions. Any authenticated user can set it at transaction creation time. [2](#0-1) 

**All terminal statuses confirmed in the enum:**

`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `REJECTED`, and `ARCHIVED` are all defined in `TransactionStatus`. None of them are in the guard's allowlist, so all are affected when `isManual=true`. [3](#0-2) 

**Test gap confirmed:**

The only e2e test for this guard (line 884) sets `status = EXECUTED` on a non-manual transaction and expects a 400. It does not test the `isManual: true` + terminal-status combination, so the bug is not caught by the test suite. [4](#0-3) 

## Impact Explanation

1. **Audit trail corruption:** A creator can archive an `EXECUTED` manual transaction, removing it from execution history. The `ARCHIVED` status is not treated as an execution record, so on-chain executions become hidden from the backend's history view.
2. **Terminal state overwrite:** `CANCELED`, `REJECTED`, `FAILED`, and `EXPIRED` each carry distinct lifecycle semantics. Overwriting any of them with `ARCHIVED` corrupts the integrity of the transaction state machine.
3. **False notification broadcast:** `emitTransactionStatusUpdate` is called unconditionally after the update (lines 721–730), broadcasting a false status change to all subscribers. [5](#0-4) 

## Likelihood Explanation

- **No elevated privileges required.** The attacker only needs a normal authenticated, verified user account.
- **`isManual` is a standard product workflow.** Any user can create a manual transaction via the standard `CreateTransactionDto`.
- **Trigger is trivial.** Create a transaction with `isManual: true`, allow or force it to reach any terminal state, then issue a single `PATCH /transactions/archive/:id` HTTP request.
- **No special tooling required.** A direct authenticated HTTP call suffices.

## Recommendation

Fix the guard logic to enforce the status check unconditionally, regardless of `isManual`. The corrected intent should be:

```typescript
// Allow archiving only from WAITING_FOR_SIGNATURES or WAITING_FOR_EXECUTION,
// regardless of isManual.
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions are intentionally allowed to be archived from additional states, those states must be explicitly enumerated in the allowlist — not derived by removing the status check entirely.

Additionally, add e2e test cases covering `isManual: true` transactions in each terminal state to prevent regression.

## Proof of Concept

1. Authenticate as any verified user.
2. Create a transaction with `isManual: true` via `POST /transactions`.
3. Manually update the transaction to a terminal status (e.g., `EXECUTED`, `CANCELED`, `FAILED`) — or wait for it to reach one naturally.
4. Issue `PATCH /transactions/archive/<id>` with the creator's auth token.
5. Observe: the response is `200 OK` and the transaction's status in the database is now `ARCHIVED`, overwriting the terminal state. The `emitTransactionStatusUpdate` notification is also broadcast to all subscribers. [6](#0-5)

  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
