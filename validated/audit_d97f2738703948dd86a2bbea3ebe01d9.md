Audit Report

## Title
Transaction Group Submission Accepts Intra-Batch Duplicate `transactionId` Entries, Bypassing the Uniqueness Guard

## Summary
`createTransactions()` in `transactions.service.ts` guards against duplicate `transactionId` values by querying the database before inserting. However, it never checks for duplicates **within the submitted batch itself**. A caller submitting a `POST /transaction-groups` request with multiple `groupItems` sharing identical `transactionBytes` (and thus the same Hedera `transactionId`) will have all items pass the guard and be persisted, because none exist in the database at check time.

## Finding Description

**Root cause — no intra-batch deduplication:**

`createTransactions()` collects all `transactionId` strings from the validated batch and issues a single `repo.find()` query: [1](#0-0) 

The guard throws only if any of those IDs already exist in the database. It does **not** check whether the `transactionIds` array itself contains duplicates. If a caller submits two DTOs whose `transactionBytes` encode the same payer account and `validStart`, both produce the same string at: [2](#0-1) 

Both pass the guard (neither is in the DB yet), and both are saved inside the database transaction: [3](#0-2) 

**No database-level safety net:**

The `Transaction` entity declares `transactionId` as a plain `@Column()` with no `@Unique()` decorator and no unique index: [4](#0-3) 

The class-level `@Index` decorators cover only `['status', 'mirrorNetwork']` and `['creatorKeyId']` — no unique constraint on `transactionId`: [5](#0-4) 

No migration in the codebase adds a `UNIQUE` constraint on the `transaction_id` column either.

**Attack entry point — `POST /transaction-groups`:**

`createTransactionGroup()` extracts all transaction DTOs from the request body and forwards them as a single batch to `createTransactions()`: [6](#0-5) 

**Access control — any authenticated user:**

The endpoint is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no privileged role is required: [7](#0-6) 

**What happens at execution time:**

When the duplicate row reaches execution, the chain service receives a `DuplicateTransaction` status code from Hedera and returns `null` without updating the row's status, leaving it permanently stuck in `WAITING_FOR_EXECUTION`: [8](#0-7) 

Note: this code path was designed for multi-pod race conditions, but its effect is identical for intentionally duplicated batch entries — the ghost row is never resolved.

## Impact Explanation

In an organization context, every transaction stored in the database is routed through the full signing and approval workflow. Duplicate entries with the same `transactionId` will:

1. Appear as separate pending items requiring signatures from all required key holders.
2. Consume approver time and attention for transactions that can never succeed on Hedera.
3. Leave ghost rows permanently in `WAITING_FOR_EXECUTION`, polluting dashboards and audit logs.
4. Scale linearly — a single malicious group submission with N identical items forces N−1 guaranteed failures through the full workflow.

## Likelihood Explanation

The attacker needs only a valid authenticated session (a normal organization member account). No privileged role is required. Crafting two group items with identical `transactionBytes` is trivial via a direct API call — the only constraint is a valid creator signature over the bytes, which the attacker controls. The `POST /transaction-groups` endpoint accepts arbitrary `transactionBytes` in the request body with no server-side uniqueness check within the batch.

## Recommendation

Add an intra-batch deduplication check in `createTransactions()` immediately after building the `transactionIds` array, before the database query:

```typescript
const transactionIds = validatedData.map(v => v.transactionId);
const uniqueIds = new Set(transactionIds);
if (uniqueIds.size !== transactionIds.length) {
  const seen = new Set<string>();
  const dupes = transactionIds.filter(id => seen.size === seen.add(id).size);
  throw new BadRequestException(
    `Duplicate transactionIds in request: ${[...new Set(dupes)].join(', ')}`,
  );
}
```

Additionally, add a `UNIQUE` constraint on the `transactionId` column in the `Transaction` table (scoped appropriately, e.g., excluding `CANCELED`/`REJECTED`/`ARCHIVED` rows via a partial index) as a defense-in-depth measure.

**Expected (correct) behavior:** 400 Bad Request — duplicate `transactionId` detected.

**Actual behavior:** 201 Created — both rows are inserted with the same `transactionId`. The second row will permanently remain in `WAITING_FOR_EXECUTION` after the first is executed on Hedera, as the `DuplicateTransaction` handler at `execute.service.ts:168-185` returns `null` without updating the row's status.
