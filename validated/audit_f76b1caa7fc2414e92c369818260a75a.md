### Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Concurrent Signature Writes to Silently Drop Signatures from Transaction Bytes

### Summary
The `uploadSignatureMaps` function in `signers.service.ts` reads transaction bytes from the database outside any lock or database transaction, computes a new signed version in memory, then writes it back unconditionally. Two concurrent `POST /transactions/signers` requests for the same transaction will both read the same stale snapshot, each independently merge their own signature, and the last writer will overwrite the first — silently dropping the earlier signature from the stored bytes. The `TransactionSigner` records for both users are inserted, creating a false appearance of full signing, while the actual `transactionBytes` column only contains one user's signature.

### Finding Description

**Root cause — three-phase read-modify-write with no locking:**

**Phase 1 — `loadTransactionData`** reads `transactionBytes` and existing signers outside any database transaction or row lock:

```
const transactions = await this.dataSource.manager.find(Transaction, {
  where: { id: In(transactionIds) },
});
``` [1](#0-0) 

**Phase 2 — `validateAndProcessSignatures`** merges the incoming signature map into the in-memory `sdkTransaction` object derived from the stale snapshot:

```
let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
...
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
``` [2](#0-1) 

**Phase 3 — `persistSignatureChanges` → `bulkUpdateTransactions`** writes the new bytes back with an unconditional `UPDATE` — no version column, no `WHERE "transactionBytes" = <old_value>` guard:

```sql
UPDATE transaction
SET "transactionBytes" = CASE id ${whenClauses} END,
    "updatedAt" = NOW()
WHERE id = ANY($N)
``` [3](#0-2) 

The three phases are not wrapped in a single serializable database transaction with a row-level lock. The outer `this.dataSource.transaction(...)` in `persistSignatureChanges` only covers the write phase, not the read phase in `loadTransactionData`. [4](#0-3) 

The public HTTP endpoint is:

```
POST /transactions/signers
``` [5](#0-4) 

Any authenticated organization user with a valid signing key for the transaction can call this endpoint. No privileged role is required.

### Impact Explanation

In an organization multi-signer workflow, two users (User A and User B) both submit their signatures for the same transaction concurrently:

1. Both requests execute `loadTransactionData` and read the same `transactionBytes_0`.
2. User A's request computes `bytes_A = bytes_0 + sig_A`.
3. User B's request computes `bytes_B = bytes_0 + sig_B`.
4. User A's write commits: DB stores `bytes_A`.
5. User B's write commits: DB stores `bytes_B` — **overwriting `bytes_A`**.
6. The `transaction_signer` table now has rows for both A and B (inserted independently), but the `transactionBytes` column only contains `sig_B`.
7. When the chain service collates and executes the transaction, it reads `transactionBytes` and finds only `sig_B`. `sig_A` is permanently lost.
8. If the transaction requires both signatures (threshold key), it will never reach execution — the transaction is permanently frozen in `WAITING_FOR_SIGNATURES` state.

This constitutes **permanent lock/freeze of a transaction** and **integrity failure in the signing state**, which are high-value impact categories per `RESEARCHER.md`. [6](#0-5) 

### Likelihood Explanation

- **Attacker preconditions**: Two authenticated organization users, each holding a valid signing key for the same transaction. This is the normal multi-signer workflow — no privilege escalation needed.
- **Trigger**: The front-end's "Sign All" feature for transaction groups sends batch signature uploads. Multiple users signing simultaneously is the expected use case, not an edge case.
- **Window**: The race window spans the entire async processing time of `validateAndProcessSignatures` (which includes key lookups and SDK operations), making it wide enough to be hit under normal concurrent load.
- **Detection**: The `TransactionSigner` table will show both users as having signed, masking the data loss from monitoring.

The automation test suite itself acknowledges this timing issue in the signing flow: [7](#0-6) 

### Recommendation

Wrap the entire read-modify-write sequence for each transaction in a single serializable database transaction with a pessimistic row-level lock (`SELECT ... FOR UPDATE`) on the `transaction` row. This prevents any concurrent writer from reading the same stale bytes:

```typescript
await this.dataSource.transaction('SERIALIZABLE', async manager => {
  // Lock the row
  const transaction = await manager
    .createQueryBuilder(Transaction, 't')
    .setLock('pessimistic_write')
    .where('t.id = :id', { id })
    .getOne();

  // Merge signatures into the locked, fresh bytes
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
  sdkTransaction.addSignature(publicKey, map);

  // Write back within the same transaction
  await manager.update(Transaction, id, {
    transactionBytes: Buffer.from(sdkTransaction.toBytes()),
  });
});
```

Alternatively, use optimistic locking with a `version` column on the `Transaction` entity and retry on conflict.

### Proof of Concept

```typescript
// Two authenticated users both sign transaction ID 42 simultaneously
const userAToken = '<jwt_user_a>';
const userBToken = '<jwt_user_b>';

const signatureMapA = buildSignatureMap(privateKeyA, sdkTransaction);
const signatureMapB = buildSignatureMap(privateKeyB, sdkTransaction);

// Fire both requests concurrently — no delay
await Promise.all([
  axios.post(`${serverUrl}/transactions/signers`, [{ id: 42, signatureMap: signatureMapA }],
    { headers: { Authorization: `Bearer ${userAToken}` } }),
  axios.post(`${serverUrl}/transactions/signers`, [{ id: 42, signatureMap: signatureMapB }],
    { headers: { Authorization: `Bearer ${userBToken}` } }),
]);

// Both return HTTP 201 and both appear in transaction_signer table
// But SELECT "transactionBytes" FROM transaction WHERE id = 42
// contains only ONE signature — whichever write committed last.
// The transaction is now permanently stuck if both signatures were required.
```

Expected outcome: `transaction_signer` has two rows (User A and User B), but `transaction.transactionBytes` contains only one signature. The chain service will fail to execute the transaction if the key threshold requires both.
