### Title
Unbounded Signature-Map Array in `importSignatures` Endpoint Enables Authenticated Resource-Exhaustion DoS

### Summary
The `POST /transactions/signatures/import` API endpoint accepts an unbounded array of `UploadSignatureMapDto` items with no server-side count limit. Each item in the array triggers expensive operations: a database lookup, `SDKTransaction.fromBytes()` protobuf deserialization, and `validateSignature()` cryptographic verification. An authenticated organization member can craft a single request containing hundreds of valid transaction IDs to force O(n) cryptographic work on the server, degrading or crashing the API service. The code itself acknowledges the missing limit with an inline comment.

### Finding Description

**Root cause — no count limit on the array:**

`POST /transactions/signatures/import` is handled by `TransactionsController.importSignatures`: [1](#0-0) 

The body is typed as `UploadSignatureMapDto[] | UploadSignatureMapDto` with no `@ArrayMaxSize` or equivalent constraint. The body parser accepts up to 2 MB of JSON, which can encode hundreds of items.

**Expensive per-item processing in `importSignatures`:**

For every item in the array the service performs:
1. A database read for the transaction row.
2. `SDKTransaction.fromBytes(transaction.transactionBytes)` — full protobuf deserialization of the stored transaction.
3. `validateSignature(sdkTransaction, map)` — cryptographic signature verification against the supplied map.
4. `sdkTransaction.addSignature(publicKey, map)` — another SDK operation. [2](#0-1) 

**Self-acknowledged missing limit:**

The batch-update block immediately below carries the comment: [3](#0-2) 

> "Added a batch mechanism, probably should limit this on the api side of things"

**Body-size limit does not prevent the attack:**

The global body limit is 2 MB: [4](#0-3) 

A 2 MB JSON payload can carry several hundred `{ id, signatureMap }` objects, each referencing a distinct valid transaction ID. The rate-limiter throttles by IP/second but does not bound the work-per-request.

**Rate-limiter configuration:** [5](#0-4) 

The limits (`GLOBAL_MINUTE_LIMIT`, `GLOBAL_SECOND_LIMIT`) are operator-configured. Even at 10 req/s, each request carrying 200 items yields 2 000 cryptographic operations per second from a single authenticated client.

**Same pattern exists in the signers upload path:** [6](#0-5) 

`uploadSignatureMaps` has the identical unbounded-array pattern.

### Impact Explanation

A single authenticated user can send repeated requests to `POST /transactions/signatures/import`, each containing the maximum number of valid transaction IDs that fit within the 2 MB body limit. The server performs O(n) protobuf deserialization and cryptographic signature verification per request. Sustained at the allowed request rate this exhausts CPU, causes response-time degradation for all other users, and can render the API service unavailable — a Denial-of-Service against the organization backend.

### Likelihood Explanation

Any registered, verified organization member can reach this endpoint — no admin or privileged key is required. The attacker only needs a valid JWT and knowledge of existing transaction IDs (obtainable via `GET /transactions`). The attack requires no special tooling beyond a standard HTTP client. The missing limit is explicitly acknowledged in the source code, confirming the gap is known but unmitigated.

### Recommendation

1. Add `@ArrayMaxSize(N)` (e.g., `N = 50`) to the `UploadSignatureMapDto[]` parameter in `TransactionsController.importSignatures` and the equivalent signers endpoint.
2. Apply the same constraint to `uploadSignatureMaps` in `SignersService`.
3. Consider adding per-endpoint rate limiting (e.g., via `@Throttle()`) that is tighter than the global IP throttler, specifically for the signature-import paths.
4. Enforce the array-size limit at the DTO level using `class-validator` so it is validated before any database or SDK work begins.

### Proof of Concept

```bash
# 1. Authenticate and obtain JWT
TOKEN=$(curl -s -X POST https://<api>/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"member@org.com","password":"..."}' | jq -r '.accessToken')

# 2. Collect 200 valid transaction IDs visible to this user
IDS=$(curl -s -H "Authorization: Bearer $TOKEN" \
  "https://<api>/transactions?limit=200" | jq '[.items[].id]')

# 3. Build a payload: 200 items, each with a crafted signatureMap
python3 - <<'EOF'
import json, requests

ids = <paste IDS here>
items = [{"id": i, "signatureMap": {}} for i in ids]

headers = {"Authorization": "Bearer <TOKEN>", "Content-Type": "application/json"}

# Send at the allowed rate; each request forces 200× SDK deserialization + crypto
for _ in range(60):          # 1 minute sustained
    requests.post("https://<api>/transactions/signatures/import",
                  json=items, headers=headers)
EOF
```

**Expected outcome:** API response times increase significantly for all users; under sustained load the Node.js event loop saturates and the service becomes unresponsive, constituting a Denial-of-Service against the organization backend.
