Audit Report

## Title
Approver Threshold Not Enforced Before Transaction Execution

## Summary
The Hedera Transaction Tool implements an organizational approver-threshold system where a configurable number of designated approvers must approve a transaction before it proceeds. However, `processTransactionStatus()` — the sole function responsible for promoting a transaction from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` — evaluates only cryptographic signature sufficiency, never the approver-threshold state. A transaction can therefore reach execution with zero approvals, bypassing the entire organizational governance control.

## Finding Description

**Root cause — `processTransactionStatus` ignores approver state**

`processTransactionStatus()` is the gate that sets `WAITING_FOR_EXECUTION`. Its entire decision logic is:

```typescript
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey
);

let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

if (isAbleToSign) {
  const collatedTx = await smartCollate(transaction, signatureKey);
  if (collatedTx !== null) {
    newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
  }
}
``` [1](#0-0) 

The decision is made entirely on `hasValidSignatureKey` (cryptographic key coverage) and `smartCollate` (signature collation). There is no call to any approver-status check. The function receives only a `Repository<Transaction>`, a `TransactionSignatureService`, and the transaction list — it has no access to `TransactionApprover` data and makes no attempt to query it. [2](#0-1) 

**The approver threshold is stored but never consulted at this gate**

The `transaction_approver` table stores a `threshold` integer and an `approved` boolean per approver node, forming a recursive tree structure. [3](#0-2) 

The `isApproved()` utility correctly evaluates the threshold tree recursively:

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    ...
  }
};
``` [4](#0-3) 

However, `isApproved()` lives entirely in the front-end and is only used for UI display in `ApproverStructureStatus.vue` and `ReadOnlyApproversList.vue`. [5](#0-4) 

A grep of the entire back-end codebase for `FEATURE_APPROVERS_ENABLED` returns zero matches — the flag is a front-end-only constant set to `false`. [6](#0-5) 

This means the back-end approver API endpoints (`POST /transactions/:id/approvers`, `POST /transactions/:id/approvers/approve`) are fully active regardless of the UI flag, but the threshold is never enforced server-side.

**Execution path never checks approver threshold**

After `processTransactionStatus` sets the status to `WAITING_FOR_EXECUTION`, the chain service's `prepareTransactions` picks it up and calls `collateAndExecute`: [7](#0-6) 

`collateAndExecute` calls `addExecutionTimeout` → `executeService.executeTransaction`. Neither function checks approver state: [8](#0-7) [9](#0-8) 

The manual-execution path (`executeTransaction` in `transactions.service.ts`) also only checks the `isManual` flag and `validStart`, not approver threshold: [10](#0-9) 

**Exploit path**

1. Organization admin creates a transaction and configures an approver tree with threshold N (e.g., 2-of-3 managers must approve) via `POST /transactions/:id/approvers`.
2. A user who holds the required Hedera signing keys signs the transaction cryptographically — without any approver having approved.
3. `processTransactionStatus()` is triggered (via `emitTransactionStatusUpdate` after signing). It sees sufficient cryptographic signatures → sets status to `WAITING_FOR_EXECUTION`.
4. The chain service executes the transaction on Hedera. The approver threshold was never evaluated.

## Impact Explanation

The approver-threshold system is the primary organizational governance control: it is the mechanism by which organizations enforce multi-party authorization for sensitive transactions (HBAR transfers, account updates, file updates). Bypassing it means:

- A single signer with the required cryptographic keys can unilaterally execute any transaction, regardless of the configured approval policy.
- Organizational funds or account configurations can be moved or changed without the required managerial sign-off.
- The entire approver workflow — including the `TransactionApprover` entity, the `approveTransaction` API, and the threshold tree — provides no actual security guarantee at the execution layer.

## Likelihood Explanation

Any authenticated organization user who holds the cryptographic keys required to satisfy the Hedera signature threshold can trigger this. In many deployments, signers and approvers overlap. A signer who disagrees with a pending rejection can simply sign and force execution. No privileged access, no leaked credentials, and no external network access are required — only a valid organization account with signing keys. The back-end approver API endpoints are active regardless of the `FEATURE_APPROVERS_ENABLED` front-end flag, so the attack surface exists in all deployments where approvers are configured via direct API calls.

## Recommendation

Enforce the approver threshold inside `processTransactionStatus()` before allowing the status transition to `WAITING_FOR_EXECUTION`. Specifically:

1. Query the `TransactionApprover` tree for each transaction being evaluated.
2. Port the `isApproved()` logic (currently front-end only) to a shared back-end utility.
3. In `processTransactionStatus()`, after `smartCollate` returns non-null, additionally verify that `isApproved()` returns `true` for every root-level approver node associated with the transaction. Only then set `newStatus = WAITING_FOR_EXECUTION`.
4. Apply the same check in the manual `executeTransaction` path in `transactions.service.ts`.

## Proof of Concept

```
1. Create a transaction via POST /transactions
2. Configure a 2-of-3 approver threshold via POST /transactions/:id/approvers
   (no approvers approve — approved field remains null for all)
3. Sign the transaction with sufficient cryptographic keys via the signing endpoint
4. Observe: processTransactionStatus() is triggered, sees hasValidSignatureKey = true
   and smartCollate != null, sets status = WAITING_FOR_EXECUTION
5. The chain service scheduler picks up the WAITING_FOR_EXECUTION transaction
   and calls collateAndExecute → addExecutionTimeout → executeService.executeTransaction
6. Transaction executes on Hedera with 0 approvals out of the required 2
```

export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }

  if (updatesByStatus.size > 0) {
    await Promise.all(
      Array.from(updatesByStatus.values()).map(async ({ newStatus, oldStatus, ids }) => {
        const result = await transactionRepo
          .createQueryBuilder()
          .update(Transaction)
          .set({ status: newStatus })
          .where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
          .returning('id')
          .execute();

        for (const row of result.raw) {
          statusChanges.set(row.id, newStatus);
        }
      })
    );
  }

  return statusChanges;
}

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};

    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing

              relations: {
                groupItems: {
                  transaction: true,
                },

                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },

          }
        } else {

  addExecutionTimeout(transaction: Transaction) {
    const name = `execution_timeout_${transaction.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    if (transaction.isManual) return;

    const timeToValidStart = transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransaction(transaction);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }
