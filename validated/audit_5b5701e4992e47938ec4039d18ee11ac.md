Audit Report

## Title
`CommentsService.createComment()` Missing Transaction Existence Check and Authorization Enforcement

## Summary
`CommentsService.createComment()` accepts a caller-supplied `transactionId` and saves a comment entity linked to it without (1) verifying the transaction exists, and (2) verifying the calling user has any relationship to that transaction. The controller itself contains a developer-authored `TODO` acknowledging the missing guard. Every other write-path transaction service enforces both invariants.

## Finding Description

`createComment()` at lines 15–24 constructs a `TransactionComment`, assigns `transactionId` directly, and calls `repo.save()` with no prior lookup: [1](#0-0) 

The controller that invokes it carries only JWT/verified-user guards and contains an explicit developer acknowledgment that a transaction-level check is absent: [2](#0-1) 

By contrast, `ObserversService.createTransactionObservers()` fetches the transaction and throws `BadRequestException(ErrorCodes.TNF)` when it is missing, and also verifies the caller is the creator: [3](#0-2) 

`ApproversService.createTransactionApprovers()` delegates to `getCreatorsTransaction()`, which performs the same existence + creator check: [4](#0-3) [5](#0-4) 

The `TransactionComment` entity declares a `@ManyToOne` relation to `Transaction`, which TypeORM materialises as a database-level foreign-key constraint: [6](#0-5) 

## Impact Explanation

**Unhandled 500 on non-existent transaction**: Because the FK constraint exists (via `@ManyToOne`), passing a non-existent `transactionId` causes `repo.save()` to throw a raw database FK-violation exception. This is not caught anywhere in `CommentsService` or the controller, so it propagates as an HTTP 500, leaking internal error details rather than returning a proper 400.

**Authorization bypass**: Any authenticated, verified user can POST `POST /transactions/:transactionId/comments` with an arbitrary integer `transactionId` — including transactions they did not create, are not a signer of, are not an observer of, and are not an approver of. This breaks the access-control model uniformly enforced by every other write-path service. Comments from unrelated users will appear on transactions, polluting audit trails and potentially leaking the existence of transaction IDs to unauthorized parties.

## Likelihood Explanation

The only precondition is a valid JWT for a verified account — no privileged role is required. The endpoint is reachable by any registered user. The developer TODO comment in the controller confirms the gap was known but unaddressed. Exploitation requires only a standard HTTP client.

## Recommendation

1. **Existence check**: Before saving, fetch the transaction and throw `BadRequestException(ErrorCodes.TNF)` if absent, consistent with `ObserversService` and `ApproversService`.
2. **Authorization check**: Verify the calling user is the creator, a signer, an observer, or an approver of the referenced transaction before allowing comment creation. The pattern used in `ObserversService.getTransactionObserversByTransactionId()` (lines 76–110) is a suitable reference.
3. **Error handling**: Wrap `repo.save()` in a try/catch and re-throw as `BadRequestException` to prevent raw DB errors from surfacing as 500s, consistent with the pattern in `ObserversService.createTransactionObservers()`.

## Proof of Concept

```
# Step 1: Authenticate as any registered user and obtain a JWT
POST /auth/login  { "email": "user@example.com", "password": "..." }
→ { "accessToken": "<jwt>" }

# Step 2: Post a comment to a transaction the user has no relationship to
POST /transactions/9999/comments
Authorization: Bearer <jwt>
Content-Type: application/json
{ "message": "unauthorized comment" }

# Result A (transactionId 9999 does not exist):
→ HTTP 500 — raw FK constraint violation leaks DB internals

# Result B (transactionId 9999 exists but belongs to another user):
→ HTTP 201 — comment is persisted; unauthorized user has written to another user's transaction
```

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
