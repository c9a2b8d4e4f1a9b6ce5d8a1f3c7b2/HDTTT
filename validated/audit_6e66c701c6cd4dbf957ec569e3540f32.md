### Title
Unbounded Array Input in `uploadSignatureMap` Endpoint Enables Single-Request DoS

### Summary

The `POST /transactions/:transactionId?/signers` endpoint (and the analogous `POST /transactions/signatures/import` endpoint) accepts an unbounded array of `UploadSignatureMapDto[]` with no maximum-size constraint. A single authenticated request containing thousands of entries causes the server to spawn thousands of concurrent promises, each performing CPU-intensive transaction deserialization and signature processing, exhausting server resources.

### Finding Description

The `uploadSignatureMap` controller method accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` as its body with no `@ArrayMaxSize()` or equivalent constraint: [1](#0-0) 

The `UploadSignatureMapDto` itself has no array-size decorator — only `@IsDefined()`, `@IsNumber()`, `@IsNotEmptyObject()`, and `@IsSignatureMap()`: [2](#0-1) 

After per-element validation, `uploadSignatureMaps` is called with the full unbounded array. Inside `validateAndProcessSignatures`, all entries are processed simultaneously via `Promise.all(dto.map(...))`: [3](#0-2) 

Each concurrent promise calls `validateTransactionStatus`, which invokes `SDKTransaction.fromBytes(transaction.transactionBytes)` — a CPU-intensive protobuf deserialization — once per array entry: [4](#0-3) 

Each entry then calls `processTransactionSignatures`, which iterates over a triple-nested loop (nodes → transaction IDs → public keys) with no depth or count limit: [5](#0-4) 

The same unbounded pattern exists in `importSignatures`: [6](#0-5) 

The global throttler limits requests per second/minute but does not constrain the number of elements within a single request body, so a single crafted request bypasses rate limiting entirely. [7](#0-6) 

### Impact Explanation

A single authenticated request with an array of N entries causes:
1. N concurrent `SDKTransaction.fromBytes()` deserializations (CPU-intensive protobuf parsing)
2. N concurrent triple-nested signature map iterations
3. N concurrent `sdkTransaction.addSignature()` calls

With N = 10,000 entries all referencing the same valid transaction ID, the server's event loop is saturated and memory is exhausted, rendering the API unavailable to all users. This directly mirrors the external report's pattern: an attacker-controlled count parameter drives unbounded resource consumption with no server-side cap.

### Likelihood Explanation

Any authenticated user (no admin privileges required) can reach this endpoint. The attack requires only a valid JWT token and knowledge of one valid transaction ID. The attacker constructs a JSON array of 10,000+ identical `{id: <valid_tx_id>, signatureMap: {...}}` entries and POSTs it in a single HTTP request. No special tooling is needed beyond `curl` or any HTTP client.

### Recommendation

1. Add `@ArrayMaxSize(N)` (e.g., N = 100) to the body array in both `uploadSignatureMap` and `importSignatures` controllers, or enforce the limit inside `transformAndValidateDto`.
2. Add a maximum-depth/entry-count check inside `processTransactionSignatures` on the number of nodes, transaction IDs, and public keys in the supplied `signatureMap` before iterating.
3. Consider processing the `dto` array sequentially or with a concurrency cap (e.g., `p-limit`) instead of `Promise.all` over the full unbounded input.

### Proof of Concept

```bash
# Attacker is a normal authenticated user with a valid JWT and one known transaction ID
TOKEN="<valid_jwt>"
TX_ID=42

# Build a payload with 10,000 identical entries
python3 -c "
import json
entry = {'id': $TX_ID, 'signatureMap': {'0.0.3': {'0.0.2@1234.0': {'302a...validDERkey': '0xdeadbeef...'}}}}
print(json.dumps([entry] * 10000))
" > payload.json

# Single request — bypasses per-request rate limiting
curl -X POST https://<api-host>/transactions/signers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d @payload.json
```

The server spawns 10,000 concurrent `SDKTransaction.fromBytes()` calls and 10,000 concurrent signature-processing loops, exhausting CPU and memory within the single request's lifetime.

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
