Audit Report

## Title
Unbounded Array Input in `POST /transactions/signatures/import` and `POST /transactions/:transactionId?/signers` Enables Authenticated DoS via Resource Exhaustion

## Summary
Both `importSignatures` and `uploadSignatureMaps` accept an attacker-controlled array of `UploadSignatureMapDto` objects with no enforced size limit. Processing each element involves sequential (or concurrent) database queries and CPU-intensive cryptographic operations. A single authenticated user can submit one request with thousands of entries to exhaust server CPU, database connections, and the Node.js event loop, degrading service for all users.

## Finding Description

**Entry point — `POST /transactions/signatures/import`:**

The controller accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no array size guard: [1](#0-0) 

The `UploadSignatureMapDto` class applies only field-level validators (`@IsDefined`, `@IsNumber`, `@IsSignatureMap`) — no `@ArrayMaxSize` or any cardinality constraint exists: [2](#0-1) 

**Unbounded sequential loop in `importSignatures`:**

The service iterates over every caller-supplied element sequentially: [3](#0-2) 

Per iteration:
1. `await this.verifyAccess(transaction, user)` — issues a DB round-trip via `userKeysToSign` for every non-terminal transaction.
2. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-bound SDK deserialization.
3. `validateSignature(sdkTransaction, map)` — CPU-bound cryptographic verification.
4. `sdkTransaction.addSignature(publicKey, map)` for each public key.

`verifyAccess` always calls `userKeysToSign` for non-terminal transactions, which in turn calls `userKeysRequiredToSign` → `attachKeys` (DB) + `keysRequiredToSign` (DB + crypto): [4](#0-3) [5](#0-4) 

**Developer acknowledgment of the missing limit:**

The developers left an explicit comment in the same function: [6](#0-5) 

**Second affected endpoint — concurrent variant (`POST /transactions/:transactionId?/signers`):**

`uploadSignatureMaps` uses `Promise.all(dto.map(...))`, firing all crypto operations simultaneously rather than sequentially: [7](#0-6) 

Both endpoints are guarded only by JWT authentication and email verification: [8](#0-7) [9](#0-8) 

## Impact Explanation

A single crafted request with N entries causes:
- **N sequential DB round-trips** in `importSignatures` (one `userKeysToSign` call per entry for non-terminal transactions).
- **N CPU-intensive SDK deserialization + cryptographic verification calls** per entry.
- The Node.js event loop is blocked for the duration of the entire sequential chain, delaying all other in-flight requests.
- For `uploadSignatureMaps`, N concurrent DB queries and crypto operations can exhaust the connection pool and CPU simultaneously, causing all other database operations to queue or fail.

At N = 10,000 (a trivially constructable JSON array), the server becomes unresponsive for the duration of the request. Repeated requests from a single authenticated account constitute a sustained service outage for all users.

**Impact: High** — complete service degradation achievable by a single authenticated user with no special privileges.

## Likelihood Explanation

Any user who has registered and verified their email address can reach both endpoints. No admin role, no leaked credentials, and no special account state is required beyond normal registration. The attack requires only a single HTTP request with a large JSON array body. The developer comment at line 575 confirms the team is already aware the limit is missing.

**Likelihood: High.**

## Recommendation

1. **Enforce an array size limit at the controller level** for both endpoints. Add a `@ArrayMaxSize(N)` decorator (e.g., N = 100 or a configurable value) to the array parameter, or validate the array length before passing it to the service.
2. **Add a global NestJS pipe** (e.g., `ValidationPipe` with `transform: true`) that enforces DTO constraints on array inputs.
3. **Rate-limit** both endpoints per authenticated user to prevent repeated large-payload attacks.
4. For `importSignatures`, consider batching the `verifyAccess` checks rather than issuing one DB round-trip per element sequentially.

## Proof of Concept

```http
POST /transactions/signatures/import HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1.000000000": { "<pubkey>": "<sig>" } } } },
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1.000000000": { "<pubkey>": "<sig>" } } } },
  ... (repeated 10,000 times)
]
```

Each entry triggers a sequential `await verifyAccess(...)` call (DB round-trip) plus `SDKTransaction.fromBytes` (CPU) plus `validateSignature` (CPU). The server event loop is blocked for the entire duration of the request, making the service unresponsive to all other users.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
