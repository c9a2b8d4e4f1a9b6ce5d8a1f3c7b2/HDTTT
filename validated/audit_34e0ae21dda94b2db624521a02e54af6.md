### Title
Transaction Creator Can Lower Approval Threshold After Approvals Are Cast, Bypassing Multi-Party Approval Requirement

### Summary
The `updateTransactionApprover` and `removeTransactionApprover` endpoints perform no check on the current transaction status before allowing the creator to mutate the approval tree. A malicious creator can cast the approval tree with a high threshold, wait for a subset of approvers to sign, then lower the threshold to match the already-collected approvals, causing the transaction to advance to `WAITING_FOR_EXECUTION` without the originally required quorum.

### Finding Description

**Root cause — missing status guard in `getCreatorsTransaction`**

`getCreatorsTransaction` is the sole authorization gate used by both the create, update, and remove approver paths. It only verifies creator identity; it never inspects `transaction.status`. [1](#0-0) 

**Update path — threshold can be lowered at any status**

`updateTransactionApprover` calls `getCreatorsTransaction` for authorization and then unconditionally writes the new threshold value. There is no guard preventing this when the transaction is already in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. [2](#0-1) [3](#0-2) 

**Remove path — approvers can be deleted at any status**

`removeTransactionApprover` in the controller calls `getCreatorsTransaction` (creator-only check, no status check) and then hard-removes the node tree. After removal it emits `emitTransactionStatusUpdate`, which triggers the chain service to re-evaluate whether the remaining approvals now satisfy the (possibly auto-reduced) threshold. [4](#0-3) [5](#0-4) 

**Contrast with `approveTransaction` — which does check status**

The approval submission endpoint correctly restricts itself to `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION`. The structural mutation endpoints have no equivalent guard. [6](#0-5) 

**Automatic threshold reduction on child detach**

When a child approver is detached (`listId` set to `null`), the parent's threshold is silently reduced to `newParentApproversLength` if it would otherwise exceed the remaining child count. This means removing non-approving children can automatically bring the threshold down to the number of already-approved children. [7](#0-6) 

### Impact Explanation

The approval tree is the only multi-party gate between a transaction being created and it being submitted to the Hedera network. Bypassing it lets the creator execute any transaction — including account key rotations, HBAR transfers, or file updates — with fewer approvals than the organization configured. This is an unauthorized state change / privilege escalation: the creator unilaterally satisfies a quorum that was never actually reached.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The exploit requires only two sequential API calls (`PATCH /transactions/{id}/approvers/{treeId}` with `{ threshold: K }` after K approvals are collected). No race condition, no cryptographic break, and no external dependency is needed. Any creator who wants to bypass an approval requirement they themselves set up can do so deterministically.

### Recommendation

Add a status guard to `getCreatorsTransaction` (or inline in each mutation path) that rejects structural changes once the transaction has left `NEW` status:

```typescript
// in getCreatorsTransaction or at the top of each mutation handler
if (
  transaction.status !== TransactionStatus.NEW
) {
  throw new BadRequestException('Approver structure cannot be modified after the transaction is active');
}
```

Alternatively, lock the approval tree the moment the first approval signature is recorded, mirroring the pattern already used in `approveTransaction`.

### Proof of Concept

1. Creator (User A) creates a transaction and sets up an approval tree: threshold = 3, approvers = [User B, User C, User D, User E, User F].
2. User B and User C call `POST /transactions/{id}/approvers/approve` — two approvals recorded, threshold not yet met, status remains `WAITING_FOR_SIGNATURES`.
3. Creator calls:
   ```
   PATCH /transactions/{id}/approvers/{treeNodeId}
   { "threshold": 2 }
   ```
   `updateTransactionApprover` accepts this because `getCreatorsTransaction` only checks creator identity, not status.
4. `emitTransactionStatusUpdate` fires; the chain service re-evaluates: 2 approvals ≥ threshold 2 → transaction advances to `WAITING_FOR_EXECUTION`.
5. Transaction is submitted to Hedera with only 2 of the originally required 3 approvals.

  async removeTransactionApprover(id: number): Promise<void> {

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
