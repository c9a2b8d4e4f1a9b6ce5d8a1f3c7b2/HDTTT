## Audit Report

## Title
Unbounded Loop in `getTransactionsToSign()` Enables Server-Side DoS via Per-Transaction Mirror Node I/O

## Summary
`getTransactionsToSign()` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then serially invokes `userKeysToSign()` — which calls `computeSignatureKey()` and makes async HTTP calls to the Hedera mirror node — for each transaction before applying pagination. As the number of active transactions grows, a single authenticated request can exhaust server CPU, memory, and outbound I/O.

## Finding Description

**1. No `take` limit on the database query.**

`getTransactionsToSign()` issues `repo.find()` with no `take` parameter, loading every non-terminal transaction into memory: [1](#0-0) 

**2. Unbounded per-item sequential async loop.**

Every loaded transaction is processed one-by-one through `userKeysToSign()`: [2](#0-1) 

**3. `userKeysToSign()` → `computeSignatureKey()` → mirror-node I/O per transaction.**

`userKeysToSign()` delegates to `userKeysRequiredToSign()` → `keysRequiredToSign()` → `transactionSignatureService.computeSignatureKey()`: [3](#0-2) [4](#0-3) 

`computeSignatureKey()` makes async HTTP calls to the Hedera mirror node via `AccountCacheService.getAccountInfoForTransaction()` for the fee-payer account, each signing account, and each receiver account of every transaction: [5](#0-4) [6](#0-5) [7](#0-6) 

**4. Pagination applied only after the full loop completes.**

The `slice()` for pagination happens only after every transaction has been processed, so the `limit` parameter provides no protection: [8](#0-7) 

**5. Endpoint is live and accessible to any authenticated user.**

The controller exposes `GET /transactions/sign`, guarded only by JWT authentication — no admin role required. The comment "NO LONGER USED BY FRONT-END" confirms it remains reachable: [9](#0-8) 

**Note on `AccountCacheService`:** The cache may reduce redundant mirror-node calls for repeated account lookups within a single request, but it does not eliminate the O(N) processing cost across N distinct transactions with distinct fee-payer accounts.

## Impact Explanation

A single authenticated request to `GET /transactions/sign` when N active transactions exist causes:
- N full transaction row deserializations into memory
- Up to N × (number of signing/receiver accounts per transaction) outbound HTTP calls to the Hedera mirror node
- The Node.js event loop is blocked on awaited I/O for the entire duration

With hundreds to thousands of active transactions — a realistic scenario for an organization using multi-sig workflows — the request never completes within any reasonable timeout, the server's connection pool and memory are exhausted, and the API becomes unresponsive for all users.

## Likelihood Explanation

- **Attacker precondition:** Any valid JWT — a normal registered user.
- **Trigger:** Create a large number of transactions (a normal product workflow), then call `GET /transactions/sign`. No special privilege is needed.
- **Amplification:** No evidence of application-layer rate limiting on this route; the attacker can repeat the call continuously.
- **Realistic scenario:** In an organization with hundreds of pending multi-sig transactions (a stated use case of the tool), even legitimate users will trigger severe latency degradation on every call to this endpoint.

## Recommendation

1. **Apply `take` at the database layer** before the processing loop, not after. Pre-filter candidates using a DB-level join on user keys to avoid loading irrelevant transactions.
2. **Remove or disable the endpoint** if it is truly no longer used by the front-end, as the comment states.
3. **Add application-layer rate limiting** on this route.
4. **Parallelize with a concurrency cap** (e.g., `p-limit`) if the loop must remain, to bound simultaneous mirror-node connections.

## Proof of Concept

```
# 1. Authenticate and obtain a JWT
POST /auth/login  →  { "token": "<jwt>" }

# 2. Create N non-terminal transactions (normal workflow)
for i in 1..N:
    POST /transactions  (with valid transaction bytes)

# 3. Trigger the unbounded loop
GET /transactions/sign
Authorization: Bearer <jwt>

# Result: server processes all N transactions serially,
# making up to N × M mirror-node HTTP calls before responding.
# With N=500 and M=3 accounts/tx → up to 1500 outbound HTTP calls per request.
# Repeated calls exhaust the event loop and connection pool.
```
