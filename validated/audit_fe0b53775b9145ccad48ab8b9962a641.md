### Title
`archiveTransaction` Bypasses Terminal-State Guard for Manual Transactions, Allowing Unauthorized State Transitions

### Summary

The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` uses a compound boolean condition that completely skips the status check whenever `isManual` is `true`. This allows any authenticated user who is the creator of a manual transaction to move it from any terminal state (`CANCELED`, `EXECUTED`, `REJECTED`, `EXPIRED`, `FAILED`) to `ARCHIVED`, violating the invariant that terminal states are permanent. The analog to the external report is direct: one operation path (`cancelTransaction`) enforces status guards consistently, while another (`archiveTransaction`) silently drops the guard for a specific transaction subtype.

### Finding Description

**Root cause — `archiveTransaction` in `transactions.service.ts`:** [1](#0-0) 

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual          // ← when isManual=true this whole guard is false
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

The guard throws only when **both** conditions are true:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`, **AND**
- `isManual` is `false`.

When `isManual = true`, the second operand is `false`, so the entire `&&` expression is `false` regardless of `status`. The guard never fires, and the `repo.update` call proceeds unconditionally.

**Contrast with the properly guarded path — `cancelTransactionWithOutcome`:** [2](#0-1) 

```typescript
async cancelTransactionWithOutcome(id, user) {
  const transaction = await this.getTransactionForCreator(id, user);

  if (transaction.status === TransactionStatus.CANCELED) {
    return CancelTransactionOutcome.ALREADY_CANCELED;
  }
  if (!this.cancelableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OTIP);   // always enforced
  }
  ...
}
```

`cancelTransaction` always validates status, independent of any other field. `archiveTransaction` does not — the same invariant is enforced inconsistently across two operations that both mutate transaction state.

**Terminal states defined by the service:** [3](#0-2) 

**The test suite confirms the gap** — the only `archiveTransaction` test that checks a terminal status uses a transaction where `isManual` is `undefined`/`false`: [4](#0-3) 

No test covers `isManual = true` with a terminal status.

**Exploit path:**

1. Attacker (normal authenticated user) creates a manual transaction via `POST /transactions` with `isManual: true`.
2. An admin or the approval workflow moves the transaction to `REJECTED` (or the system moves it to `CANCELED`, `EXECUTED`, `EXPIRED`, `FAILED`).
3. Attacker calls `PATCH /transactions/archive/:id`.
4. `getTransactionForCreator` succeeds (attacker is the creator).
5. The `isManual = true` short-circuits the status guard.
6. `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes unconditionally.
7. The transaction's status is now `ARCHIVED` instead of its prior terminal state.

### Impact Explanation

- **Audit-trail corruption**: An `EXECUTED` transaction can be relabeled `ARCHIVED`, hiding the fact that it was submitted to and confirmed on the Hedera network.
- **Rejection concealment**: A `REJECTED` transaction (rejected by an admin or approver) can be relabeled `ARCHIVED` by the creator, erasing the rejection record from the workflow history.
- **Invariant violation**: The system's own `terminalStatuses` list declares these states as permanent; `archiveTransaction` silently violates that contract for any manual transaction. [3](#0-2) 

### Likelihood Explanation

- **No privileged access required**: Any verified user can create a manual transaction (`isManual: true` is a normal creation-time field).
- **Reachable via standard API**: `PATCH /transactions/archive/:id` is a documented, authenticated endpoint.
- **Low complexity**: The attacker only needs to know their own transaction ID and call one endpoint. [5](#0-4) 

### Recommendation

Replace the compound condition with an explicit allowlist that is independent of `isManual`:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  const archivableStatuses = [
    TransactionStatus.WAITING_FOR_SIGNATURES,
    TransactionStatus.WAITING_FOR_EXECUTION,
  ];

  if (!archivableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
```

This mirrors the pattern used in `cancelTransactionWithOutcome`, where the status check is always enforced regardless of other transaction properties. Add a unit test covering `isManual = true` with each terminal status to prevent regression. [6](#0-5)
