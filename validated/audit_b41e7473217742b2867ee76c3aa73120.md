Audit Report

## Title
Transaction Creator Can Bypass Approval Governance by Modifying Approvers Mid-Lifecycle

## Summary
`ApproversService.getCreatorsTransaction` — the sole authorization gate for all three approver-mutation endpoints — verifies only creator identity and never checks `transaction.status`. This allows a transaction creator to swap approvers, reduce thresholds, or remove approvers at any lifecycle stage, including after approvals have already been collected, bypassing the multi-party governance model entirely.

## Finding Description

**Root cause — `getCreatorsTransaction` has no status guard:** [1](#0-0) 

The function fetches the transaction and checks only `creatorKey.userId !== user.id`. No check on `transaction.status` exists anywhere in the function.

All three mutation paths rely on it exclusively:

**1. `createTransactionApprovers`** — calls `getCreatorsTransaction` as its only authorization gate, with no status guard: [2](#0-1) 

**2. `updateTransactionApprover` — userId swap path** — replaces the approver's `userId` and clears their existing `signature`, `userKeyId`, and `approved`: [3](#0-2) 

The creator check inside `updateTransactionApprover` is also solely `getCreatorsTransaction`: [4](#0-3) 

**3. `updateTransactionApprover` — threshold reduction path** — only validates `threshold <= approvers.length && threshold != 0`, no status guard: [5](#0-4) 

**4. `removeTransactionApprover` controller** — calls `getCreatorsTransaction` then removes the approver with no status guard: [6](#0-5) 

**Attack Path A — Approver swap:**
1. Creator creates a transaction requiring approval from User A (threshold 1/1).
2. User A reviews and refuses to approve.
3. Creator calls `PATCH /transactions/:id/approvers/:approverId` with `{ userId: colludingUserB }`.
4. The update clears `signature`, `userKeyId`, and `approved` for that slot and assigns it to User B.
5. User B immediately approves.
6. Transaction advances to `WAITING_FOR_EXECUTION` without genuine approval from User A.

**Attack Path B — Threshold reduction:**
1. Creator creates a transaction with a tree approver requiring 3/3 approvals.
2. Two approvers approve; the third refuses.
3. Creator calls `PATCH /transactions/:id/approvers/:treeId` with `{ threshold: 2 }`.
4. The threshold check only validates `threshold <= approvers.length && threshold != 0` — no status guard.
5. The transaction now satisfies the reduced threshold and proceeds to execution.

For contrast, `approveTransaction` correctly enforces a status guard: [7](#0-6) 

This guard is entirely absent from the mutation endpoints.

## Impact Explanation
The approval mechanism is the primary governance control ensuring designated parties sign off before a transaction is submitted to the Hedera network. Bypassing it allows the creator to unilaterally execute transactions — fund transfers, account key updates, node changes — that were explicitly designed to require multi-party consent. The approval step provides no real security guarantee if the creator can retroactively rewrite who the approvers are or lower the required count after partial approvals have been collected.

## Likelihood Explanation
The attacker is the transaction creator — a normal, unprivileged user. No leaked credentials, admin access, or external collusion is required. The exploit requires only two sequential API calls: one to mutate the approver (swap or threshold reduction) and one for the colluding user to approve. Any creator who faces a non-cooperating approver can exploit this immediately.

## Recommendation
Add a transaction status guard inside `getCreatorsTransaction` (or at the entry point of each mutation method) that rejects modifications when the transaction is in `WAITING_FOR_EXECUTION`, or any terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). Approver structure should only be mutable while the transaction is in `WAITING_FOR_SIGNATURES` (or `NEW` if applicable), before any approvals have been recorded. Additionally, consider adding a separate check that rejects `userId` swaps or threshold reductions when any approver in the tree already has a non-null `signature`.

## Proof of Concept

```
# Attack Path A: Approver swap

# Step 1: Creator creates transaction with User A as required approver
POST /transactions/:txId/approvers
Body: { "approversArray": [{ "userId": <userA_id> }] }

# Step 2: User A refuses to approve (no action taken)

# Step 3: Creator swaps User A for colluding User B
PATCH /transactions/:txId/approvers/:approverId
Authorization: Bearer <creator_token>
Body: { "userId": <userB_id> }
# Result: signature, userKeyId, approved are cleared; userId is now userB

# Step 4: User B approves
POST /transactions/:txId/approvers/approve
Authorization: Bearer <userB_token>
Body: { "userKeyId": <userB_key_id>, "signature": <valid_sig>, "approved": true }
# Result: Transaction advances to WAITING_FOR_EXECUTION without User A's consent

---

# Attack Path B: Threshold reduction

# Step 1: Creator creates transaction with tree approver, threshold 3/3
POST /transactions/:txId/approvers
Body: { "approversArray": [{ "threshold": 3, "approvers": [
  { "userId": <userA_id> }, { "userId": <userB_id> }, { "userId": <userC_id> }
]}]}

# Step 2: Users A and B approve; User C refuses

# Step 3: Creator reduces threshold to 2
PATCH /transactions/:txId/approvers/:treeNodeId
Authorization: Bearer <creator_token>
Body: { "threshold": 2 }
# Result: 2/2 approvals now satisfied; transaction proceeds to execution
```

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
