Audit Report

## Title
`isNode()` Duplicate Guard Bypassed for Threshold/Tree Approvers Due to Incorrect Return Expression

## Summary
`ApproversService.isNode()` is the sole duplicate-existence check invoked before every approver insertion in `createTransactionApprovers()`. Its return expression unconditionally requires `typeof approver.userId === 'number'`, causing it to always return `false` for threshold/tree approvers (those with no `userId`). As a result, the duplicate guard is silently bypassed for every tree node, allowing a transaction creator to insert arbitrarily many identical threshold nodes into the approver tree.

## Finding Description
`isNode()` is defined at lines 647–666 of `back-end/apps/api/src/transactions/approvers/approvers.service.ts`:

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';
``` [1](#0-0) 

The `count` query correctly searches for existing records matching `listId`, `userId`, `threshold`, and `transactionId`. However, the result is then ANDed with `typeof approver.userId === 'number'`. For threshold/tree approvers, `userId` is `null`/`undefined`, so this condition is always `false`, making the entire function return `false` regardless of whether a duplicate exists in the database.

| Approver type | `count > 0` | `typeof userId === 'number'` | Return |
|---|---|---|---|
| User approver | true | true | **true** (duplicate detected) |
| Tree/threshold approver | true | **false** | **false** (duplicate missed) |

This function is the sole duplicate guard in `createTransactionApprovers()`:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
``` [2](#0-1) 

Because `isNode` never returns `true` for tree approvers, the `APPROVER_ALREADY_EXISTS` error is never thrown for them, and duplicate threshold nodes are freely inserted.

## Impact Explanation
A transaction creator can POST to `POST /transactions/approvers` with a threshold-type approver DTO (no `userId`, only `threshold` + nested `approvers`) multiple times. Each call inserts a duplicate root threshold node. The approval tree becomes structurally corrupt.

`getTreeStructure()` collects all root approvers by filtering for `listId === null`:

```typescript
const rootApprovers = Array.from(approverMap.values()).filter(
  approver => approver.listId === null,
);
``` [3](#0-2) 

Multiple duplicate root threshold nodes will all appear as roots. Any approval-counting or threshold-evaluation logic that iterates over root nodes will see multiple roots for the same transaction, potentially allowing a transaction to be marked "fully approved" with fewer real signatures than the threshold requires, or causing approval evaluation to behave unpredictably.

## Likelihood Explanation
The entry point is the authenticated `POST /transactions/approvers` endpoint, restricted to the transaction creator by the `getCreatorsTransaction()` guard:

```typescript
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
``` [4](#0-3) 

No special privilege beyond being the transaction creator is required. Any legitimate creator can trigger this unintentionally (e.g., by retrying a failed request) or maliciously. The bug is in a code path exercised every time an approver tree is created.

## Recommendation
Fix the return expression in `isNode()` to return `count > 0` unconditionally, removing the erroneous `userId` type check:

```typescript
// Before (buggy):
return count > 0 && typeof approver.userId === 'number';

// After (fixed):
return count > 0;
``` [1](#0-0) 

The `userId` type check was never part of the intended duplicate-detection logic; the `find` query already handles the `userId` field correctly by querying `null` when it is absent.

## Proof of Concept
1. Authenticate as a user who is the creator of transaction with `id = 1`.
2. POST to `/transactions/approvers` with body:
   ```json
   {
     "transactionId": 1,
     "approvers": [
       {
         "threshold": 1,
         "approvers": [{ "userId": 2 }]
       }
     ]
   }
   ```
3. Repeat step 2 identically.
4. Query the approvers for transaction `1`. Two identical root threshold nodes now exist in the database.
5. The approval tree is corrupt: both root nodes are returned by `getTreeStructure()`, and any threshold evaluation logic will process both, potentially satisfying the approval requirement with fewer actual signatures.
