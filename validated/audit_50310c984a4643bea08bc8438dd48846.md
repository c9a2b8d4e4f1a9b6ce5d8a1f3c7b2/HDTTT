### Title
Global IP Rate-Limit Bypass via Spoofed `X-Forwarded-For` Header Enables DoS on Resource-Intensive Endpoints

---

### Summary

The API's sole global throttling mechanism — `IpThrottlerGuard` — derives its per-client tracking key directly from the raw `x-forwarded-for` HTTP header, which is fully attacker-controlled. An authenticated user can rotate arbitrary values in this header to create unlimited independent rate-limit buckets in Redis, bypassing the global throttle entirely. Combined with the fact that `UserThrottlerModule` is dead code (never wired into the application), authenticated endpoints such as `POST /transactions/:transactionId/signers` — which perform unbounded cryptographic operations, DB writes, and NATS emissions per array element — are left completely unthrottled per-user, enabling a single authenticated client to exhaust server resources.

---

### Finding Description

**Root Cause 1 — Spoofable tracker key in `IpThrottlerGuard`**

`IpThrottlerGuard` is registered as a global `APP_GUARD` and is the only application-wide rate limiter: [1](#0-0) 

Its `getTracker()` method reads the raw `x-forwarded-for` header directly: [2](#0-1) 

The `x-forwarded-for` header is a client-supplied HTTP header. Any HTTP client can set it to an arbitrary string (e.g., `X-Forwarded-For: 1.2.3.4`). Because the raw header value becomes the Redis key for the throttle bucket, an attacker who rotates through different fake IP strings creates a fresh, empty bucket for each request, effectively making the rate limit non-functional. This is true even in production where `app.enable('trust proxy')` is set, because the guard reads `req.headers['x-forwarded-for']` (the raw header) rather than `req.ip` (the Express-resolved, proxy-aware value): [3](#0-2) 

**Root Cause 2 — `UserThrottlerModule` / `UserThrottlerGuard` are dead code**

A per-user throttler module and guard exist: [4](#0-3) [5](#0-4) 

Neither is imported into `api.module.ts` nor applied to any controller. The module imports only `IpThrottlerModule` and `EmailThrottlerModule`: [6](#0-5) 

**Root Cause 3 — Unbounded array input to resource-intensive endpoints**

`POST /transactions/:transactionId/signers` accepts an array of `UploadSignatureMapDto` with no declared maximum length: [7](#0-6) 

For each element in the array, `SignersService.uploadSignatureMaps` performs: a batch DB read, Hedera SDK deserialization (`SDKTransaction.fromBytes`), cryptographic signature addition (`sdkTransaction.addSignature`), a DB write transaction, and NATS event emission: [8](#0-7) 

The same pattern applies to `POST /transactions/signatures/import`: [9](#0-8) 

---

### Impact Explanation

An attacker with a valid account (or who has compromised any account) can:

1. Bypass the global IP throttle by rotating `X-Forwarded-For` header values on every request.
2. Submit repeated `POST /transactions/:transactionId/signers` calls with large arrays, each triggering multiple DB queries, Hedera SDK cryptographic operations, and NATS messages.
3. Exhaust PostgreSQL connection pool, Redis, CPU (crypto), and NATS throughput, degrading or denying service to all other organization users.

The 2 MB body size limit (`app.use(json({ limit: '2mb' }))`) provides only a weak bound — a 2 MB payload of signature map entries is still a significant workload per request: [10](#0-9) 

---

### Likelihood Explanation

- **High.** Setting an arbitrary `X-Forwarded-For` header requires zero special tooling — any HTTP client (curl, Python requests, etc.) supports it.
- The API is a publicly reachable REST service consumed by an Electron desktop app, so it is internet-exposed by design.
- A single authenticated user (even a low-privilege one) is sufficient to trigger the attack; no admin role is required.
- The `uploadSignatureMap` endpoint is a normal part of the signing workflow, so repeated calls are expected and will not appear anomalous without per-user rate limiting.

---

### Recommendation

1. **Fix `IpThrottlerGuard.getTracker()`** — use `req.ip` (Express's proxy-aware resolved IP) instead of the raw `req.headers['x-forwarded-for']` string. If multiple IPs appear in the header (proxy chain), take only the last trusted hop.

2. **Wire `UserThrottlerModule` and `UserThrottlerGuard` into the application** — import `UserThrottlerModule` in `api.module.ts` and apply `UserThrottlerGuard` as a global guard (or at minimum on all authenticated controllers) so that per-user limits are enforced independently of IP.

3. **Enforce array size limits** on `uploadSignatureMap` and `importSignatures` — use `@ArrayMaxSize(N)` on the body DTO or a `ParseArrayPipe({ items: UploadSignatureMapDto, maxItems: N })` to cap the number of items processed per request.

---

### Proof of Concept

```bash
# Attacker rotates X-Forwarded-For to bypass IP throttle
# Each request gets a fresh rate-limit bucket

for i in $(seq 1 1000); do
  curl -s -X POST https://api.example.com/transactions/42/signers \
    -H "Authorization: Bearer <valid_jwt>" \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$((i % 255))" \
    -d '[
      {"id": 42, "signatureMap": { ... large map ... }},
      {"id": 43, "signatureMap": { ... large map ... }},
      ... (up to 2MB of entries)
    ]' &
done
```

Each iteration uses a distinct fake IP, so the `IpThrottlerGuard` treats it as a new client. The server processes each request's full array — DB reads, `SDKTransaction.fromBytes`, `addSignature`, DB writes, NATS emissions — for every iteration, with no per-user counter ever incrementing.

import { HttpException, HttpStatus, Injectable } from '@nestjs/common';
import { ThrottlerGuard } from '@nestjs/throttler';

@Injectable()
export class UserThrottlerGuard extends ThrottlerGuard {
  protected getTracker(req: Record<string, any>): Promise<string> {
    const user = req.user;
    if (!user) {
      throw new HttpException('No user connected.', HttpStatus.BAD_REQUEST);
    }
    // console.log('user id', user.id);
    return user.id;
  }
