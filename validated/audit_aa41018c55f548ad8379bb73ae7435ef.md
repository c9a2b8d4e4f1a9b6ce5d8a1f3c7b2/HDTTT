Audit Report

## Title
Unbounded In-Memory Iteration in `getTransactionsToSign` Enables Server-Side DoS via Resource Exhaustion

## Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no SQL-level `LIMIT`, then issues one async database call per row inside a sequential `for` loop. Pagination is applied only after the full in-memory result is built. Any authenticated user can trigger this path; as the transaction table grows, each call consumes unbounded memory, CPU, and database connections, degrading or crashing the shared API process.

## Finding Description

**No `take`/`skip` on the initial query.**
`getTransactionsToSign` builds a `FindOptionsWhere` that filters only on `status` — it does **not** scope the query to the requesting user. Every non-terminal transaction in the entire database is loaded into the Node.js heap on each call. [1](#0-0) 

**N+1 async DB call per row.**
After loading the full result set, the code loops over every transaction and issues a sequential `await` to `userKeysToSign`, which internally calls `attachKeys` (a DB round-trip) and `computeSignatureKey` (a cryptographic + DB operation) for each row. [2](#0-1) [3](#0-2) 

**Pagination applied only after full scan.**
The `limit`/`offset` values received from the caller are used only to slice the already-materialised in-memory array; they have no effect on the database query. [4](#0-3) 

**`whereForUser` does not filter by user.**
Unlike `getTransactions`, which adds `signers.userId`, `observers.userId`, and `creatorKey.userId` conditions, `getTransactionsToSign` applies no user-scoping predicate to the initial query. User relevance is determined only inside the per-row loop. [5](#0-4) 

## Impact Explanation

- **Memory exhaustion**: Every non-terminal transaction row (with its `groupItem` relation) is materialised simultaneously in the Node.js heap. A large transaction table exhausts a typical container's memory limit.
- **CPU / event-loop starvation**: The sequential `await` inside the `for` loop serialises all N database round-trips, blocking the async event loop for the full duration of the request and degrading response times for all concurrent users.
- **Database connection pressure**: N sequential queries per request can exhaust the TypeORM connection pool, causing all other requests to queue or time out.
- **Denial of service for all users**: Because the API is a shared NestJS process, a single slow request that monopolises the event loop degrades every concurrent user.

**Severity: High** — complete service unavailability is achievable without any privileged access.

## Likelihood Explanation

- Any registered user with at least one key can reach this code path; no admin role is required.
- The initial DB query is not user-scoped, so even a user with zero transactions triggers a full table scan.
- Transactions accumulate naturally over time; in an active organisation the table grows continuously.
- An adversarial user can accelerate growth by bulk-creating transactions (subject only to whatever rate limits exist on the creation endpoint).
- The cost to the attacker is low: triggering the read endpoint is a normal product flow.

**Likelihood: Medium** — realistic in any moderately active deployment; trivially achievable by a malicious insider or external attacker with a valid account.

## Recommendation

1. **Push pagination into the database query**: Add `take: limit` and `skip: offset` to the `repo.find()` call. Because user-relevance filtering currently happens in-process, the query must first be restructured to express user-key matching in SQL (e.g., via a subquery or join on `transaction_signer`/`user_key`), mirroring the approach used in `getTransactions`.
2. **Eliminate the N+1 pattern**: Batch-fetch all required signing data for the page of transactions in a single query rather than one `await` per row.
3. **Add a hard server-side cap**: Enforce a maximum `limit` value (e.g., 100) in the controller or a guard so that no single request can request an arbitrarily large page.
4. **Apply rate limiting** to the "transactions to sign" endpoint to reduce the frequency at which any single user can trigger expensive scans during a migration period.

## Proof of Concept

```
1. Register as a normal user (no elevated privileges required).
2. Ensure the transaction table contains a large number of non-terminal
   transactions (they need not belong to the attacker; the query is
   not user-scoped).
3. Send a single authenticated GET request to the "ready to sign"
   endpoint, e.g.:
     GET /transactions?readyToSign=true&page=1&limit=10
4. Observe:
   a. The server executes SELECT * FROM transaction WHERE status NOT IN (...)
      with no LIMIT, loading all rows into the Node.js heap.
   b. For each row, one sequential async DB call is issued via
      userKeysToSign → userKeysRequiredToSign → keysRequiredToSign.
   c. The event loop is blocked for the duration; other requests time out.
   d. With sufficient rows, the Node.js process exhausts heap memory
      and crashes or becomes unresponsive for all users.
5. Repeat the request to sustain the outage.
```
