### Title
IDOR: Transaction Creator Can Delete Approvers Belonging to Any Other Transaction

### Summary
In `approvers.controller.ts`, the `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but then passes the raw `:id` (approver ID) directly to `removeTransactionApprover()` without ever confirming that the targeted approver actually belongs to `:transactionId`. Any authenticated user who owns at least one transaction can therefore delete approvers from any other transaction in the system.

### Finding Description
The controller handler at `back-end/apps/api/src/transactions/approvers/approvers.controller.ts` lines 102–113 performs two independent operations:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` only confirms the caller created the transaction identified by `:transactionId`. It says nothing about the approver identified by `:id`. [2](#0-1) 

`removeTransactionApprover(id)` then fetches the approver by its own primary key and deletes it — with no check that `approver.transactionId === transactionId`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [3](#0-2) 

Contrast this with `updateTransactionApprover`, which correctly validates that the root node of the targeted approver belongs to the URL's `transactionId` before proceeding: [4](#0-3) 

The delete path has no equivalent guard.

### Impact Explanation
An attacker who is the creator of any transaction (even a trivially created one) can enumerate approver IDs (sequential integers) and issue:

```
DELETE /transactions/{attacker_own_tx_id}/approvers/{victim_approver_id}
```

This silently removes an approver from a completely different transaction. Consequences:
- Approval thresholds on victim transactions are silently reduced or eliminated, allowing transactions to advance to execution without the required signatures.
- Permanent, irreversible deletion of approval records (soft-delete via `removeNode`), corrupting the audit trail.
- In multi-sig workflows this directly undermines the security model: a transaction requiring N approvals can be stripped down to fewer, enabling unauthorized execution.

### Likelihood Explanation
- **Attacker precondition**: only a valid, verified account is needed. Any user can create a transaction and become its creator.
- **Discovery**: approver IDs are auto-increment integers; blind enumeration across a small range is trivial.
- **No rate-limit or anomaly detection** is visible on this endpoint.
- The attack requires a single authenticated HTTP request.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the `transactionId` supplied in the URL. The simplest fix mirrors what `updateTransactionApprover` already does:

```typescript
// In removeTransactionApprover or the controller:
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and add the cross-check there, keeping the validation co-located with the deletion logic.

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (attacker-controlled).
- User B creates Transaction 2 with an approver (approver ID = 42, belonging to Transaction 2).

**Attack:**
```http
DELETE /transactions/1/approvers/42
Authorization: Bearer <User A's JWT>
```

**What happens:**
1. `getCreatorsTransaction(1, userA)` → passes (User A owns Transaction 1).
2. `removeTransactionApprover(42)` → fetches approver 42 (which belongs to Transaction 2), calls `removeNode(42)`, soft-deletes it.
3. Returns `true`.

**Result:** Approver 42 is deleted from Transaction 2 without User A having any relationship to Transaction 2. If Transaction 2 required that approver's signature before execution, the approval gate is now bypassed.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
