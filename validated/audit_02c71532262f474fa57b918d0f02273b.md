Audit Report

## Title
`isNode()` Duplicate Guard Bypassed for Threshold-Based Approver Trees, Allowing Repeated Insertion

## Summary

`ApproversService.isNode()` unconditionally returns `false` for every threshold-based (tree) approver node, regardless of whether an identical record already exists in the database. Because there is no database-level unique constraint as a backstop, a transaction creator can call `POST /transactions/:id/approvers` repeatedly with the same tree-node payload and insert it an unlimited number of times, corrupting the approval tree.

## Finding Description

The return expression of `isNode()` is:

```typescript
return count > 0 && typeof approver.userId === 'number';
``` [1](#0-0) 

For a threshold-based tree node, `approver.userId` is `undefined` (the payload carries `threshold` + child `approvers`, not a `userId`). Therefore `typeof approver.userId === 'number'` is always `false`, and `isNode()` always returns `false` for tree nodes — even when `count > 0` confirms an identical record already exists in the database.

The duplicate-guard in `createTransactionApprovers` relies entirely on this function:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
``` [2](#0-1) 

The guard is never triggered for tree nodes, so `transactionalEntityManager.insert(TransactionApprover, approver)` proceeds unconditionally for every repeated call. [3](#0-2) 

The entity definition carries no `@Unique` decorator and the table has only non-unique indexes on `transactionId` and `userId`, providing no database-level backstop: [4](#0-3) 

The recursive tree-building query and `getTreeStructure` both iterate over every non-deleted row for a transaction, so every duplicate root-level tree node appears as an independent root and every duplicate child node is attached to the parent: [5](#0-4) [6](#0-5) 

## Impact Explanation

- **Approval-status corruption**: Threshold satisfaction is checked per parent node. Duplicate parent nodes each independently demand their threshold to be met, so a transaction may never reach `WAITING_FOR_EXECUTION` even after all legitimate signers have approved.
- **Removal logic breakage**: `removeNode` soft-deletes a subtree by ID. With duplicates, removing one copy leaves the others intact, producing orphaned or incorrectly-thresholded nodes.
- **Spurious status events**: `emitTransactionStatusUpdate` is called after every successful `createTransactionApprovers` invocation, so repeated calls produce spurious status events. [7](#0-6) 

## Likelihood Explanation

The endpoint is restricted to the transaction creator via `getCreatorsTransaction`: [8](#0-7) 

Any authenticated user can create a transaction and is therefore its creator. No special privilege is required. The API is reachable over standard HTTPS with a valid JWT, making the attack straightforward for any registered user.

## Recommendation

Fix the return expression in `isNode()` to decouple the existence check from the `userId` type check:

```typescript
// Before (broken for tree nodes):
return count > 0 && typeof approver.userId === 'number';

// After:
return count > 0;
```

Additionally, consider adding a composite unique constraint on `(transactionId, listId, userId, threshold)` at the database level as a defense-in-depth measure.

## Proof of Concept

1. Authenticate as any user and create a transaction (becoming its creator).
2. `POST /transactions/:id/approvers` with a tree-node payload, e.g.:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": 2 }] }] }
   ```
3. Repeat step 2 identically N times.
4. `GET /transactions/:id/approvers` — observe N duplicate root-level tree nodes, each independently demanding threshold satisfaction.
5. Even after user 2 approves, the N−1 remaining duplicate root nodes are unsatisfied, permanently blocking `WAITING_FOR_EXECUTION`.
