Audit Report

## Title
Unbounded Batch Input and Dead Per-User Throttle Module Enable Authenticated Resource Exhaustion via `importSignatures`

## Summary
The `POST /transactions/signatures/import` endpoint accepts an arbitrarily large array of signature maps with no enforced size limit. The service layer contains a developer comment explicitly acknowledging the missing cap. Compounding this, the `UserThrottlerModule` — which defines per-user rate limits — exists in the codebase but is never exported from the throttlers barrel, never imported into `ApiModule`, and its guard is never registered as an `APP_GUARD`. The only active global throttle is IP-based, which is trivially bypassed by a single large-batch request.

## Finding Description

**Root cause 1 — Unbounded batch input:**

The controller accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no array length constraint: [1](#0-0) 

The service iterates over every element, performing per-item `SDKTransaction.fromBytes` deserialization (heap allocation) and `validateSignature` cryptographic verification: [2](#0-1) 

The developer explicitly flagged the missing input cap with a comment immediately before the `BATCH_SIZE` constant: [3](#0-2) 

`BATCH_SIZE = 500` only controls how many records are written per SQL statement — it does not cap the total number of entries accepted from the caller. An input of 5,000 entries produces 10 batched SQL writes, each with a `CASE id WHEN … THEN … END` clause, after 5,000 deserialization and crypto operations have already run. [4](#0-3) 

**Root cause 2 — `UserThrottlerModule` defined but never wired:**

The module exists with limits of 100 req/min and 10 req/sec per user: [5](#0-4) 

It is not exported from the throttlers barrel: [6](#0-5) 

It is not imported in `ApiModule`, and `UserThrottlerGuard` is never registered as an `APP_GUARD`. The only active global guard is `IpThrottlerGuard`: [7](#0-6) 

**Affected endpoint:**

`POST /transactions/signatures/import` is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no per-user or payload-size throttle: [8](#0-7) 

## Impact Explanation
A single authenticated user can submit one HTTP POST containing thousands of `UploadSignatureMapDto` entries. The server will:
1. Execute a `SELECT … WHERE id IN (…)` query with thousands of IDs.
2. Loop over every entry performing `SDKTransaction.fromBytes` (heap allocation) and `validateSignature` (asymmetric crypto) — both CPU and memory intensive.
3. Build and execute raw SQL `CASE id WHEN … THEN … END` statements that grow proportionally with input size.

This exhausts CPU, heap memory, and PostgreSQL connection pool slots, causing service degradation or crash for all users.

## Likelihood Explanation
Any user who has completed registration and email verification (`VerifiedUserGuard`) can reach this endpoint. No admin role or privileged credential is required — only a valid JWT obtained through the normal login flow. The attack is a single HTTP POST, so it stays within the per-IP request-count limit while causing disproportionate server-side work. The `UserThrottlerModule` that would have mitigated this is fully implemented but dead code.

## Recommendation
1. **Enforce an array size limit** at the controller or DTO level (e.g., `@ArrayMaxSize(50)` on the body, or an explicit check before delegating to the service).
2. **Wire `UserThrottlerModule`**: export it from `back-end/apps/api/src/throttlers/index.ts`, import it in `ApiModule`, and register `UserThrottlerGuard` as an `APP_GUARD` (or apply it specifically to authenticated endpoints).
3. **Add a payload body size limit** at the HTTP layer (e.g., NestJS `bodyParser` `limit` option) to prevent oversized raw request bodies.

## Proof of Concept
```http
POST /transactions/signatures/import HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 2, "signatureMap": { ... } },
  ... // repeat N thousand times with valid or invalid IDs
]
```

The server will process all N entries sequentially through `SDKTransaction.fromBytes` and `validateSignature` before attempting batched SQL writes. With N in the thousands, this saturates CPU and memory within a single request, with no per-user throttle to prevent repetition.
