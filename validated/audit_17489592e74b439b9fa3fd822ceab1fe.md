Audit Report

## Title
Cross-Resource Authorization Bypass: Transaction Approver Deletion Without Ownership Verification

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint validates that the requesting user is the creator of the transaction identified by `transactionId`, but performs the actual deletion on the approver identified by the separate `id` parameter without verifying that this approver belongs to `transactionId`. An attacker who owns any one transaction can delete approvers from any other transaction in the system.

## Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the delete handler: [1](#0-0) 

calls `getCreatorsTransaction(transactionId, user)` to verify the caller owns `transactionId`, then immediately calls `removeTransactionApprover(id)` with no further constraint.

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `removeTransactionApprover` fetches the approver solely by `id` and deletes it: [2](#0-1) 

There is no assertion that `approver.transactionId === transactionId`. The `transactionId` URL parameter is consumed only for the creator check; the deletion operates on whatever record `id` resolves to.

The `updateTransactionApprover` path in the same service explicitly enforces this relationship: [3](#0-2) 

The delete path has no equivalent guard.

## Impact Explanation
Any authenticated, verified user who is the creator of at least one transaction can delete approvers belonging to any other transaction. By supplying their own `transactionId` in the URL and a victim approver's `id` as the path parameter, the attacker bypasses the approval workflow of transactions they do not own. This can:
- Remove required approvers from a transaction awaiting multi-party approval, allowing it to proceed without the intended authorization.
- Silently corrupt the approval tree of any transaction, undermining the integrity of the multi-signature governance model.

The `removeNode` function performs a cascading soft-delete of the entire approver subtree rooted at the targeted `id`: [4](#0-3) 

So a single request can wipe an entire approval tree belonging to a victim transaction.

## Likelihood Explanation
The attacker only needs a valid JWT (any registered, verified user) and knowledge of a victim approver's numeric `id`. Approver IDs are sequential integers returned in API responses to any participant of a transaction (observer, signer, or approver). No privileged access is required beyond being a normal user of the platform.

## Recommendation
In `removeTransactionApprover` (or in the controller before calling it), retrieve the root node of the targeted approver and assert that its `transactionId` matches the URL parameter, mirroring the existing guard in `updateTransactionApprover`:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
```

This should be placed in `approvers.service.ts` inside `removeTransactionApprover`, after the approver is fetched and before `removeNode` is called. [2](#0-1) 

## Proof of Concept

1. Attacker (User A) creates Transaction T1 — they are its creator.
2. Victim (User B) creates Transaction T2 with an approver tree. The root approver node has `id = 99` and `transactionId = T2`.
3. Attacker learns approver `id = 99` (e.g., by being an observer/signer of T2, or by enumerating sequential IDs from their own transactions).
4. Attacker sends:
   ```
   DELETE /transactions/{T1}/approvers/99
   Authorization: Bearer <attacker_jwt>
   ```
5. Server executes:
   - `getCreatorsTransaction(T1, attacker)` → passes (attacker owns T1).
   - `removeTransactionApprover(99)` → fetches approver 99 (belongs to T2), calls `removeNode(99)`, cascading soft-deletes the entire approval tree of T2.
6. T2's approval workflow is silently destroyed. The attacker never needed any rights over T2.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
