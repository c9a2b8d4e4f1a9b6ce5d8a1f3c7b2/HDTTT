### Title
Unauthenticated Cross-Tenant Transaction Data Exposure via Unscoped History Endpoint and `verifyAccess` Terminal-Status Bypass

### Summary
Any authenticated user can enumerate all historical transactions belonging to all other users and organizations by calling `GET /transactions/history`. Additionally, `verifyAccess` unconditionally grants access to any authenticated user for transactions in a terminal status, allowing full detail retrieval of any completed transaction by ID. No ownership or membership check is performed in either path.

### Finding Description

**Root cause 1 — `GET /transactions/history` has no user scope.**

`TransactionsController.getHistoryTransactions` (line 143–154) accepts no `@GetUser()` argument and passes none to the service:

```typescript
@Get('/history')
getHistoryTransactions(
  @PaginationParams() paginationParams: Pagination,
  ...
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getHistoryTransactions(paginationParams, filter, sort);
}
```

`TransactionsService.getHistoryTransactions` (line 220–249) issues a query with no `userId` predicate — it returns every transaction whose status is in `{EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED}`:

```typescript
const findOptions: FindManyOptions<Transaction> = {
  where: {
    ...getWhere<Transaction>(filter),
    status: this.getHistoryStatusWhere(filter),
  },
  ...
};
```

No ownership, observer, signer, or approver membership filter is applied. [1](#0-0) [2](#0-1) 

**Root cause 2 — `verifyAccess` short-circuits to `true` for all terminal-status transactions.**

`TransactionsService.verifyAccess` (line 786–809) returns `true` for any caller when the transaction status is `EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, or `ARCHIVED`, before any membership check is evaluated:

```typescript
if (
  [EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED].includes(transaction.status)
)
  return true;   // ← no user check
```

`getTransactionWithVerifiedAccess` (line 754–763) relies solely on this function to gate `GET /transactions/:id`. An attacker who knows or guesses any completed transaction's integer ID receives its full payload — including `transactionBytes`, `transactionHash`, `transactionId`, account IDs, and signer public keys. [3](#0-2) [4](#0-3) 

The controller-level guard chain (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only confirms the caller holds a valid, non-blacklisted JWT and has `UserStatus.NONE`. It performs no resource-level authorization. [5](#0-4) 

### Impact Explanation

A malicious authenticated user (any registered organization member) can:

1. Call `GET /transactions/history` with pagination to dump every completed transaction across all organizations — including transaction bytes, Hedera account IDs, signer public keys, amounts, and memo fields.
2. Iterate integer IDs on `GET /transactions/:id` to retrieve full details of any completed transaction they were never a party to.

This constitutes **cross-tenant data exfiltration**: account balances, counterparty identities, and signing key fingerprints of unrelated organizations are fully exposed. In a multi-organization deployment this is a critical confidentiality breach.

### Likelihood Explanation

The attacker precondition is a valid JWT — i.e., any registered user account. Registration is admin-controlled (`POST /auth/signup` requires `AdminGuard`), but in any real deployment multiple users exist. The exploit requires only standard HTTP GET requests with a valid `Authorization` header; no privilege escalation, no cryptographic attack, and no race condition is needed. The history endpoint is paginated and filterable, making bulk extraction trivial.

### Recommendation

1. **Scope `getHistoryTransactions` to the requesting user.** Pass `user` into the service and add a `WHERE` clause identical to the one used in `getTransactions` (creator, signer, observer, or approver membership). [6](#0-5) 

2. **Remove the unconditional `return true` in `verifyAccess` for terminal statuses.** Terminal status should not bypass membership checks; the same signer/observer/creator/approver predicate must be evaluated regardless of status.

3. Optionally add an `AdminGuard`-gated endpoint for operators who legitimately need a global history view, separate from the user-scoped endpoint.

### Proof of Concept

```
# Step 1: Obtain a JWT for any registered user
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Dump all completed transactions across all organizations
GET /transactions/history?page=1&limit=100
Authorization: Bearer <JWT>
→ 200 OK — returns transactionBytes, transactionId, signers, account IDs
  for every organization's completed transactions

# Step 3: Retrieve full detail of any specific completed transaction by ID
GET /transactions/42
Authorization: Bearer <JWT>
→ 200 OK — full TransactionFullDto including bytes, hash, public keys
  even if the attacker was never a creator, signer, observer, or approver
```

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
