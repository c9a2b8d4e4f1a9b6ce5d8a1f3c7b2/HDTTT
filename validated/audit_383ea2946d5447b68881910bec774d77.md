Audit Report

## Title
`archiveTransaction` Bypasses Terminal-State Guard via `isManual` Flag, Allowing Corruption of Executed/Failed/Canceled Transaction Records

## Summary
`archiveTransaction` in `TransactionsService` contains a logically flawed guard that uses `&&` between the status check and `!transaction.isManual`. When `isManual` is `true`, the second operand short-circuits the entire condition to `false`, causing the guard to never throw — regardless of the transaction's current status. This allows any transaction creator to overwrite a terminal status (`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, `REJECTED`) with `ARCHIVED` on any manual transaction they own.

## Finding Description

**Root cause — flawed boolean guard:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `archiveTransaction` at line 711:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← second operand of AND
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard throws only when **both** conditions are true simultaneously:
- Status is not in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- **AND** `isManual` is `false`

When `isManual = true`, `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false`. The `throw` is never reached, regardless of the current status value.

**Unconditional status overwrite follows immediately:**

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

**Terminal states that can be overwritten** are defined in the service itself: [3](#0-2) 

`EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, `ARCHIVED`, and `REJECTED` are all terminal. None are protected when `isManual = true`.

**Entry path — creator check only:**

`getTransactionForCreator` is the sole prerequisite. No admin or privileged role is required. [4](#0-3) 

**Contrast with correct guards elsewhere:**

`cancelTransactionWithOutcome` uses a conditional SQL `WHERE status IN (...)` clause to prevent races and terminal-state overwrites: [5](#0-4) 

**The existing e2e test does not cover this case:**

The test `"should not archive a transaction if it's already executed"` uses `addedTransactions.userTransactions[0]`, which is a non-manual transaction. It correctly returns 400 for `isManual = false`, but the `isManual = true` + terminal-status combination is never tested. [6](#0-5) 

## Impact Explanation

A transaction creator who owns a `isManual = true` transaction that has already reached `EXECUTED`, `FAILED`, `CANCELED`, or `EXPIRED` can call `PATCH /transactions/archive/:id` and overwrite the terminal status to `ARCHIVED`.

Concrete consequences:
1. **Audit-trail corruption**: An `EXECUTED` transaction — one confirmed by the Hedera network — is relabeled `ARCHIVED`. Observers and auditors see `ARCHIVED` instead of `EXECUTED`, misrepresenting the true outcome.
2. **Failure concealment**: A `FAILED` transaction can be archived, hiding the failure from observers.
3. **Irreversibility**: There is no "un-archive" path in the codebase. Once overwritten, the original terminal status is unrecoverable from the application layer.

The `executedAt` timestamp column remains set, creating an inconsistent record (a transaction with `executedAt` populated but `status = ARCHIVED`). [7](#0-6) 

## Likelihood Explanation

- **Attacker profile**: Any authenticated organization user who created a manual transaction. No admin key, no leaked credential, no privileged role required.
- **Trigger**: A single `PATCH /transactions/archive/:id` API call after the transaction has reached a terminal state.
- **Precondition**: `isManual = true` on the transaction. Manual transactions are a standard, first-class product feature explicitly supported in the UI and API, as confirmed by the e2e test that creates one with `isManual: true`. [8](#0-7) 

## Recommendation

Replace the flawed `&&` guard with an explicit allowlist of archiveable statuses, independent of `isManual`:

```typescript
const archiveableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

// For manual transactions, also allow archiving from WAITING_FOR_EXECUTION
// but never from terminal states regardless of isManual
if (!archiveableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Or, if the intent is that manual transactions should additionally be archiveable from some other non-terminal state, define that explicitly:

```typescript
const archiveableStatuses = transaction.isManual
  ? [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION]
  : [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION];

if (!archiveableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The key fix is that the status check must be **unconditional** — `isManual` must never be used to bypass it. Additionally, add an e2e test that creates a manual transaction, advances it to `EXECUTED`/`FAILED`, and asserts that `PATCH /transactions/archive/:id` returns 400.

## Proof of Concept

1. Authenticate as any organization user (no special role needed).
2. Create a transaction with `isManual: true` via `POST /transactions`.
3. Allow or force the transaction to reach `EXECUTED` or `FAILED` status (e.g., via the execute flow or by direct DB update in a test environment).
4. Call `PATCH /transactions/archive/:id` as the transaction creator.
5. Observe: HTTP 200 is returned and the transaction status is now `ARCHIVED`, overwriting the terminal state.

The guard at line 711–718 evaluates `!transaction.isManual` as `false` (since `isManual = true`), making the entire condition `false`, so the `throw` is skipped and `repo.update` at line 720 unconditionally sets `status = ARCHIVED`. [9](#0-8)
