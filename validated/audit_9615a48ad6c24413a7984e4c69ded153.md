Audit Report

## Title
Missing Access Control on Transaction Comments Endpoints Allows Any Authenticated User to Read and Write Comments on Arbitrary Transactions

## Summary
The `CommentsController` exposes `POST /transactions/:transactionId/comments`, `GET /transactions/:transactionId/comments`, and `GET /transactions/:transactionId/comments/:id` with no authorization check verifying whether the requesting user has any relationship to the target transaction. Any verified, authenticated organization member can post comments to and read comments from any transaction they have no legitimate role in. The source code contains an explicit `//TODO need some sort of guard or check to ensure user can comment here` acknowledgment of the missing control.

## Finding Description

**Root cause — `back-end/apps/api/src/transactions/comments/comments.controller.ts`:**

The controller is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level. [1](#0-0) 

The `POST` handler carries an explicit TODO acknowledging the missing guard, and passes directly to `CommentsService.createComment` with no membership check: [2](#0-1) 

The `GET` handlers similarly perform no access verification: [3](#0-2) 

`CommentsService` contains no authorization logic whatsoever — `createComment` blindly saves the comment, and `getTransactionComments` returns all comments for any `transactionId`: [4](#0-3) 

**By contrast**, `ObserversService.getTransactionObserversByTransactionId` explicitly checks creator/signer/observer/approver membership and throws `UnauthorizedException` on failure: [5](#0-4) 

`ApproversService.getVerifiedApproversByTransactionId` applies the identical pattern: [6](#0-5) 

The canonical access model is defined in `TransactionsService.verifyAccess` (creator, signer, observer, approver, or terminal status), but it is **never invoked** by `CommentsService`: [7](#0-6) 

## Impact Explanation
- **Unauthorized write**: Any verified user can inject comments into any transaction in the organization, including sensitive multi-party signing workflows they have no legitimate role in. This corrupts the audit trail and communication record of the transaction.
- **Unauthorized read**: Any verified user can enumerate and read all comments on any transaction, exposing potentially sensitive coordination details (key identifiers, signing instructions, business context) intended only for transaction participants.
- Impact category: unauthorized state change + cross-user data exposure.

## Likelihood Explanation
- Attacker precondition: valid organization account (verified user). No admin or privileged role required.
- Attack path is a single authenticated HTTP request to `POST /transactions/<id>/comments` or `GET /transactions/<id>/comments` with any valid `transactionId`.
- Transaction IDs are sequential integers, making enumeration trivial.
- The `//TODO` comment in the source confirms the gap is known and unmitigated.

## Recommendation
Inject `TransactionsService` into `CommentsService` and call `verifyAccess` (or the equivalent `getTransactionWithVerifiedAccess`) at the start of `createComment`, `getTransactionComments`, and `getTransactionCommentById`. This mirrors the pattern already enforced by `ObserversService` and `ApproversService`. The check should load the transaction with its `creatorKey`, `observers`, `signers`, and `approvers` relations and throw `UnauthorizedException` if the requesting user is not among them (unless the transaction is in a terminal status, which `verifyAccess` already handles).

## Proof of Concept
```
# Attacker is a verified org member with no relationship to transaction ID 42
POST /transactions/42/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "message": "Injected comment into a transaction I have no role in" }

# Returns 201 Created — comment is persisted with no access check

GET /transactions/42/comments
Authorization: Bearer <attacker_jwt>

# Returns 200 OK with all comments on transaction 42 — no access check
```

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
