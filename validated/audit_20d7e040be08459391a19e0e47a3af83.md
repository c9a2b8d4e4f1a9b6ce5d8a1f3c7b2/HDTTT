Audit Report

## Title
Race Condition in `archiveTransaction` Allows Unconditional Status Overwrite, Corrupting Execution Audit Trail

## Summary
`archiveTransaction` in `transactions.service.ts` performs a status check followed by an unconditional `repo.update({ id }, { status: ARCHIVED })`. The chain service's `_executeTransaction` makes a multi-second external Hedera network call between its own status validation and its conditional write. A transaction creator can exploit this window to write `ARCHIVED` over a transaction that has already been irreversibly executed on Hedera, permanently corrupting the audit trail.

## Finding Description

**Unconditional write in `archiveTransaction`:**

The function reads the current status via `getTransactionForCreator`, checks that it is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`, then issues an update with no `WHERE status = ...` guard: [1](#0-0) 

Any status present in the DB at write time is silently overwritten.

**The chain service's conditional write:**

`_executeTransaction` correctly guards its own write: [2](#0-1) 

If the status is no longer `WAITING_FOR_EXECUTION` at write time, 0 rows are affected and the function returns `null` — silently suppressing the execution notification.

**The race window — external Hedera calls:**

Between `validateTransactionStatus` (which re-reads the DB and sees `WAITING_FOR_EXECUTION`) and the conditional `UPDATE`, there is a multi-second window occupied by two external network calls: [3](#0-2) 

`validateTransactionStatus` does check for `ARCHIVED` status and would throw — but this check runs *before* the Hedera calls, not after. Once it passes, the window is open. [4](#0-3) 

**`@MurLock` does not protect `archiveTransaction`:**

The distributed lock at line 41 serialises concurrent `executeTransaction` calls for the same transaction ID, but `archiveTransaction` acquires no lock and is not excluded by it. [5](#0-4) 

## Impact Explanation

- **Audit trail corruption:** The DB permanently records `ARCHIVED` for a transaction that was irreversibly committed on Hedera. Signers, approvers, and compliance reviewers cannot determine the true outcome from the application's own records.
- **Suppressed execution notification (Scenario A):** `_executeTransaction` returns `null` at line 198, so `emitTransactionStatusUpdate` is never called for the execution event. Downstream services and users are never informed the transaction executed.
- **False state (Scenario B):** If the chain service wins the write race first and records `EXECUTED`, the concurrent `archiveTransaction` (which read `WAITING_FOR_EXECUTION` before the execution write) then overwrites `EXECUTED` with `ARCHIVED` unconditionally, contradicting the already-emitted execution notification.

## Likelihood Explanation

- **Attacker profile:** Any authenticated user who is the creator of a transaction — no privileged role required.
- **Trigger:** The race window is the duration of the Hedera network round-trip (typically 2–10 seconds). A creator can poll the transaction status endpoint, detect the `WAITING_FOR_EXECUTION` transition, and immediately call `PATCH /transactions/archive/:id`. This is deliberate and scriptable with a standard HTTP client.
- **No special tooling required.**

## Recommendation

Add a conditional `WHERE status IN (WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION)` guard to the `repo.update` call in `archiveTransaction`, mirroring the pattern already used by `_executeTransaction`. After the update, check the affected row count; if 0, re-read the current status and throw an appropriate error (e.g., `ConflictException`). This eliminates the TOCTOU window by making the check and the write atomic at the database level.

```typescript
const result = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id AND status IN (:...allowedStatuses)', {
    id,
    allowedStatuses: [
      TransactionStatus.WAITING_FOR_SIGNATURES,
      TransactionStatus.WAITING_FOR_EXECUTION,
    ],
  })
  .returning('id')
  .execute();

if (result.raw.length === 0) {
  throw new ConflictException(ErrorCodes.OMTIP);
}
```

## Proof of Concept

1. Creator submits a transaction; it reaches `WAITING_FOR_EXECUTION`.
2. The chain service's scheduler picks it up, calls `validateTransactionStatus` (passes), and begins `sdkTransaction.execute(client)` — the Hedera call is now in flight.
3. Creator immediately calls `PATCH /transactions/archive/:id`.
4. `archiveTransaction` calls `getTransactionForCreator` → reads `WAITING_FOR_EXECUTION` → passes the guard → executes `repo.update({ id }, { status: ARCHIVED })` unconditionally.
5. Chain service receives the Hedera receipt, attempts `UPDATE ... WHERE id = :id AND status = 'WAITING_FOR_EXECUTION'` → 0 rows affected (status is now `ARCHIVED`) → returns `null`.
6. **Result:** Transaction is executed and irreversible on Hedera; the application DB shows `ARCHIVED`; no execution notification is emitted.
