### Title
Missing Tree-Node Existence Check in `isNode` Allows Duplicate Approval Trees, Corrupting Transaction Approval State

### Summary
`ApproversService.isNode()` is the sole guard against duplicate approver entries, but its return expression unconditionally returns `false` for threshold/tree nodes (any approver without a `userId`). A transaction creator can therefore call `POST /transactions/:id/approvers` repeatedly with the same threshold tree and each call inserts a fully-duplicated tree into the database. Because the chain service evaluates every root-level tree independently, the duplicated trees inflate the required approval work and can permanently stall a transaction in `WAITING_FOR_SIGNATURES`.

### Finding Description

**Root cause — `isNode` ignores tree nodes** [1](#0-0) 

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';   // ← always false for tree nodes
```

The database query (`find`) correctly locates an existing tree node by `{ listId: null, userId: null, threshold: N, transactionId: X }`, so `count > 0` is `true`. However, the second operand `typeof approver.userId === 'number'` is `false` for every threshold/tree node (they carry no `userId`). The `&&` short-circuits to `false`, so the guard never fires for tree nodes.

**Exploit path in `createTransactionApprovers`** [2](#0-1) 

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
```

Because `isNode` returns `false` for the root tree node, the `throw` is never reached. A new tree node is inserted (e.g., `id = 5`). The recursive loop then creates child leaf nodes under `listId = 5`: [3](#0-2) 

For each child, `isNode` is called with the **new** parent id (`listId = 5`). No existing record has `listId = 5`, so `count = 0` and the child is also inserted. The entire tree is cloned.

**No database-level unique constraint prevents this** [4](#0-3) 

`TransactionObserver` carries `@Index(['userId', 'transactionId'], { unique: true })`. No equivalent unique index exists on `transaction_approver`, so the database accepts the duplicate rows silently.

### Impact Explanation

Every root-level tree node is evaluated independently by the approval-status logic: [5](#0-4) 

With N duplicate trees, the transaction requires N independent rounds of approval from the same set of users. Because `approveTransaction` bulk-updates all records for a user at once: [6](#0-5) 

…all duplicate leaf records are signed simultaneously, so the transaction is not permanently blocked in the simple case. However:

1. **Permanent stall via large duplication**: A creator who submits hundreds of duplicate trees causes the approval-status query and recursive `isApproved` traversal to process thousands of rows. Under realistic database/CPU limits this can time out or produce incorrect status, leaving the transaction stuck in `WAITING_FOR_SIGNATURES` with no recovery path short of direct database intervention.
2. **Notification storm**: Every `createTransactionApprovers` call emits `emitTransactionStatusUpdate`, flooding observers and signers with spurious events.
3. **Corrupted audit trail**: The `transaction_approver` table contains ghost rows that cannot be removed without individually calling `removeTransactionApprover` for each duplicate node.

### Likelihood Explanation

- **Attacker profile**: Any authenticated user who has created a transaction. No admin or privileged role is required.
- **Trigger**: Call `POST /transactions/:id/approvers` with the same threshold-tree body more than once. The only server-side guard is `isNode`, which is broken for tree nodes.
- **Realistic scenario**: A malicious or buggy client retries the approver-creation request (e.g., on network timeout), or a user deliberately submits the same tree repeatedly to stall a multi-party transaction they no longer wish to proceed with.

### Recommendation

Fix the return expression in `isNode` to return `true` for **any** node type when a matching record already exists:

```typescript
// Before (broken for tree nodes):
return count > 0 && typeof approver.userId === 'number';

// After (correct for all node types):
return count > 0;
``` [1](#0-0) 

Additionally, add a composite unique index on `transaction_approver` (e.g., `(transactionId, listId, userId, threshold)`) as a defence-in-depth measure to prevent duplicates at the database level.

### Proof of Concept

1. Authenticate as a normal user and create a transaction (record its `id` as `TX_ID`).
2. `POST /transactions/TX_ID/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": 2 }] }] }
   ```
3. Repeat step 2 identically.
4. Query `SELECT * FROM transaction_approver WHERE "transactionId" = TX_ID` — two root tree nodes with `threshold = 1` and two child leaf nodes with `userId = 2` are present.
5. Repeat step 2 hundreds of times; observe that the approval-status evaluation degrades proportionally and the transaction remains in `WAITING_FOR_SIGNATURES` even after all legitimate approvers have signed, because the chain service must evaluate every duplicated tree root.

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }
```
