Audit Report

## Title
Recovery Phrase Written to System Clipboard in Plaintext via Explicit Copy Buttons

## Summary
The Hedera Transaction Tool desktop application writes the full 24-word BIP-39 recovery phrase to the OS-level system clipboard in plaintext through two separate, user-facing copy buttons. No expiry, clearing, or warning mechanism is applied. Any process on the same machine can silently read the clipboard at any time after the user clicks "Copy."

## Finding Description

**Path 1 — Account Setup (Onboarding):**

`front-end/src/renderer/pages/AccountSetup/components/Generate.vue`, `handleCopyRecoveryPhrase`, lines 92–94: [1](#0-0) 

Bound to a "Copy" button at line 188: [2](#0-1) 

**Path 2 — Migration Summary:**

`front-end/src/renderer/pages/Migrate/components/Summary.vue`, `copyRecoveryPhrase`, lines 62–67: [3](#0-2) 

Bound to a copy icon button at lines 162–168: [4](#0-3) 

In both cases, `navigator.clipboard.writeText()` places the full comma-separated 24-word mnemonic into the OS clipboard. There is no subsequent `clipboard.writeText('')` call, no timeout-based clearing, and no user warning that the data will persist indefinitely.

**Root cause:** The application treats the clipboard as a safe output channel for the root secret, with no lifecycle management after the write.

## Impact Explanation
The recovery phrase is the root secret from which all Hedera key pairs in the wallet are derived. Compromise of the recovery phrase gives an attacker full, permanent control over every key and account derived from it. The clipboard is a shared OS resource readable by any process without elevated privileges. Once written, the mnemonic persists until overwritten by another clipboard operation. This constitutes complete, irrecoverable loss of all assets controlled by keys derived from the phrase.

## Likelihood Explanation
The copy buttons are prominently placed in two high-traffic, normal-workflow flows: initial onboarding and post-migration summary. Clipboard managers (e.g., Windows Clipboard History, macOS ClipMenu, Ditto) are extremely common and persist clipboard history by default. Many legitimate applications — password managers, note-taking apps, cloud sync tools — read clipboard content. A malicious or compromised background process requires no elevated privileges to call the OS clipboard API. The user action (clicking "Copy") is explicitly invited by the UI, making this a realistic, high-frequency exposure path.

## Recommendation
1. **Clear the clipboard after a short delay** (e.g., 30–60 seconds) by calling `navigator.clipboard.writeText('')` via `setTimeout` immediately after the write.
2. **Warn the user** before writing, informing them that the phrase will be placed in the clipboard and advising them to clear it after use.
3. **Consider removing the one-click full-phrase copy entirely** in favor of requiring the user to manually select and copy individual words, reducing the attack surface.

## Proof of Concept
1. Launch the Hedera Transaction Tool and begin account setup onboarding.
2. Generate a recovery phrase. The 24-word phrase is displayed.
3. Click the "Copy" button (line 188 of `Generate.vue`).
4. From any other process on the machine (e.g., a terminal), read the clipboard:
   - **Linux:** `xclip -o` or `xdotool getclipboard`
   - **macOS:** `pbpaste`
   - **Windows:** `Get-Clipboard` (PowerShell)
5. The full plaintext mnemonic is returned with no further user interaction required.
6. The same result is reproducible via the migration summary copy button (`Summary.vue` line 165).
7. The clipboard content persists across application restarts until overwritten.
