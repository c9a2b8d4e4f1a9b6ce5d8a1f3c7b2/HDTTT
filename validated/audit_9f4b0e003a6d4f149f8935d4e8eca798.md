### Title
Silent Group Execution Abort via TOCTOU Between In-Memory Status Filter and DB Re-Validation in `executeTransactionGroup`

### Summary

`executeTransactionGroup` in `back-end/libs/common/src/execute/execute.service.ts` first filters group items using stale in-memory status, then re-reads each transaction's status from the database inside `validateTransactionStatus`. If any transaction's DB status has moved to a terminal state (e.g., `CANCELED`) between when the group was loaded and when execution runs, the function throws an unrecoverable error. That error is silently swallowed by the caller (`addGroupExecutionTimeout`), leaving every other transaction in the group permanently stuck in `WAITING_FOR_EXECUTION` until the expiry cron marks them `EXPIRED`. A malicious organization member can deliberately trigger this by canceling one transaction in a group during the execution window.

### Finding Description

**Root cause — TOCTOU between filter and DB re-read**

`executeTransactionGroup` performs two separate status checks on each group item:

1. **In-memory filter** (uses the object loaded by `prepareTransactions`): [1](#0-0) 

2. **DB re-read** inside `getValidatedSDKTransaction → validateTransactionStatus`: [2](#0-1) 

If a transaction is canceled after the group is loaded into memory (step 1 passes because in-memory status is still `WAITING_FOR_EXECUTION`) but before step 2 runs, `validateTransactionStatus` reads `CANCELED` from the DB and throws. The throw propagates out of the validation loop: [3](#0-2) 

**Silent swallow in caller**

The caller `addGroupExecutionTimeout` catches the thrown error and logs it, but takes no remediation action — no status update, no retry, no notification: [4](#0-3) 

The scheduler timeout is deleted in `finally`, so the group is never retried. All remaining transactions stay in `WAITING_FOR_EXECUTION` until `handleExpiredTransactions` marks them `EXPIRED` 3 minutes after their `validStart`.

**The TOCTOU window is large**

`prepareTransactions` loads the group and passes it to `collateGroupAndExecute`, which schedules a timeout 10 seconds before `validStart`. `addGroupExecutionTimeout` then schedules another timeout 5 seconds after `validStart`. The in-memory group object is never refreshed across this entire pipeline: [5](#0-4) [6](#0-5) [7](#0-6) 

The window between group load and actual execution can span tens of seconds to minutes, giving an attacker ample time to cancel one transaction.

**Cancellation is a legitimate user action**

Organization members can cancel individual transactions via the cancel API. The `cancelTransactionGroup` service confirms cancellation is a normal workflow: [8](#0-7) 

A malicious member cancels exactly one transaction in a group they belong to, triggering the abort of the entire group's execution.

### Impact Explanation

- All transactions in the group that were ready to execute are permanently blocked from executing.
- They eventually expire (`EXPIRED` status) without ever being submitted to the Hedera network.
- No error is surfaced to users — the group silently fails.
- For atomic/sequential groups (e.g., multi-step token operations, file updates), this means the entire coordinated workflow is destroyed.
- The attacker needs no special privilege beyond being an organization member with access to the group.

### Likelihood Explanation

- Any organization member who can view and cancel transactions in a group can trigger this.
- The TOCTOU window is large (tens of seconds) and predictable — the scheduler's cron fires every 10 seconds, and execution timeouts are deterministic relative to `validStart`.
- No cryptographic material, admin keys, or external network access is required.
- The attack is repeatable: the attacker can re-join or create new groups and repeat.

### Recommendation

1. **Refresh group item statuses from DB immediately before the validation loop** in `executeTransactionGroup`, rather than relying on the stale in-memory filter. Replace the in-memory filter with a fresh DB query.

2. **Do not throw on a single transaction validation failure for non-atomic groups.** Instead, skip the failed transaction, mark it individually as `FAILED`/`CANCELED`, and continue executing the remaining valid transactions. Only abort the entire group if `atomic: true` is set.

3. **Surface the error** — when `executeTransactionGroup` throws, `addGroupExecutionTimeout` should update all affected transactions to `FAILED` with an appropriate status code rather than silently swallowing the error.

### Proof of Concept

**Preconditions:**
- Organization with members Alice (legitimate) and Mallory (malicious).
- A sequential transaction group G with transactions T_1, T_2, T_3 all in `WAITING_FOR_EXECUTION`, `validStart` = now + 60 seconds.

**Steps:**

1. The chain service's `handleTransactionsBetweenNowAndAfterThreeMinutes` cron fires and calls `prepareTransactions`. Group G is loaded into memory with all three transactions showing `WAITING_FOR_EXECUTION`.

2. `collateGroupAndExecute` schedules a timeout at `validStart - 10s` (i.e., now + 50s). The in-memory `transactionGroup` object is captured in the closure.

3. At now + 30s, Mallory calls the cancel API for T_2. The DB now shows T_2 as `CANCELED`. The in-memory object still shows `WAITING_FOR_EXECUTION`.

4. At now + 50s, the `collateGroupAndExecute` timeout fires. `smartCollate` runs on all three transactions (including T_2, whose in-memory status is still `WAITING_FOR_EXECUTION`). All pass. `addGroupExecutionTimeout` is called.

5. At now + 65s, the `addGroupExecutionTimeout` timeout fires and calls `executeTransactionGroup(transactionGroup)`.

6. Inside `executeTransactionGroup`, the in-memory filter passes T_2 (status = `WAITING_FOR_EXECUTION` in memory). Then `getValidatedSDKTransaction(T_2)` calls `validateTransactionStatus`, which reads T_2 from DB → status = `CANCELED` → throws `"Transaction has been canceled."`.

7. The throw propagates to `executeTransactionGroup`'s catch block:
   ```
   throw new Error(`Transaction Group cannot be submitted. Error validating transaction 2: Transaction has been canceled.`);
   ```

8. `addGroupExecutionTimeout`'s catch block logs the error and deletes the timeout. T_1 and T_3 remain in `WAITING_FOR_EXECUTION`.

9. At now + 3 minutes past `validStart`, `handleExpiredTransactions` marks T_1 and T_3 as `EXPIRED`. The group's transactions are never submitted to Hedera.

```typescript
  private async validateTransactionStatus(transaction: Transaction) {
    const { status } = await this.transactionsRepo.findOne({
      where: { id: transaction.id },
      select: ['status'],
    });

    switch (status) {
      case TransactionStatus.NEW:
        throw new Error('Transaction is new and has not been signed yet.');
      case TransactionStatus.FAILED:
        throw new Error('Transaction has already been executed, but failed.');
      case TransactionStatus.EXECUTED:
        throw new Error('Transaction has already been executed.');
      case TransactionStatus.REJECTED:
        throw new Error('Transaction has already been rejected.');
      case TransactionStatus.EXPIRED:
        throw new Error('Transaction has been expired.');
      case TransactionStatus.CANCELED:
        throw new Error('Transaction has been canceled.');
      case TransactionStatus.ARCHIVED:
        throw new Error('Transaction is archived.');
    }
```
