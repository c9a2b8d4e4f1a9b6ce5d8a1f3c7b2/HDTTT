# Audit Report

## Title
Creator Can Delete Transactions in Terminal States, Bypassing Status-Transition Protections

## Summary
The `removeTransaction` function, exposed via `DELETE /transactions/:id`, allows a transaction creator to soft-delete a transaction and forcibly overwrite its status to `CANCELED` regardless of its current state — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, `ARCHIVED`, and `REJECTED`. The dedicated `cancelTransaction` endpoint enforces a `cancelableStatuses` guard that blocks this, but `removeTransaction` has no equivalent check, creating a bypass path that corrupts the off-chain audit trail.

## Finding Description

**Root cause — missing status guard in `removeTransaction`:**

`TransactionsService` defines two relevant status lists:

```typescript
// transactions.service.ts lines 92–105
private readonly cancelableStatuses = [
  TransactionStatus.NEW,
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

private readonly terminalStatuses = [
  TransactionStatus.EXECUTED,
  TransactionStatus.EXPIRED,
  TransactionStatus.FAILED,
  TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED,
  TransactionStatus.REJECTED,
];
``` [1](#0-0) 

`cancelTransactionWithOutcome` enforces the guard before any state mutation:

```typescript
// transactions.service.ts lines 669–671
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [2](#0-1) 

`removeTransaction`, called by `DELETE /transactions/:id`, performs **no such check** — it only verifies ownership via `getTransactionForCreator`, then unconditionally overwrites the status and soft-deletes:

```typescript
// transactions.service.ts lines 629–651
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user); // ownership only
  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction);
  }
  ...
}
``` [3](#0-2) 

The controller wires this directly to an authenticated endpoint with no additional guard:

```typescript
// transactions.controller.ts lines 313–316
@Delete('/:id')
deleteTransaction(@GetUser() user, @Param('id', ParseIntPipe) id: number): Promise<boolean> {
  return this.transactionsService.removeTransaction(id, user, true);
}
``` [4](#0-3) 

## Impact Explanation

The off-chain organizational record of an already-executed (or otherwise terminal) Hedera transaction is permanently corrupted. The `repo.update` call at line 633 overwrites the stored status to `CANCELED`, and `repo.softRemove` sets `deletedAt`, hiding the record from all standard queries. [5](#0-4) 

Observers, signers, and approvers who participated in the transaction lose visibility into its final state. The audit trail is silently falsified: an `EXECUTED` transaction is relabeled `CANCELED` and hidden. Any downstream logic (notifications, reporting, compliance checks) that relies on terminal-state records is affected. The on-chain Hedera transaction is immutable, but the tool's own integrity model is broken.

## Likelihood Explanation

Any authenticated, verified user who has ever created a transaction can trigger this. No elevated privileges are required — only a valid JWT and ownership of the transaction (i.e., `creatorKey.userId === user.id`). The endpoint is a standard REST `DELETE` call. The attacker does not need to race any window; they can call it at any time after the transaction reaches a terminal state.

## Recommendation

Add a status guard to `removeTransaction` (or at the controller level) that mirrors the one in `cancelTransactionWithOutcome`. Specifically, before performing any mutation, check that the transaction's current status is **not** in `terminalStatuses` (or equivalently, is only in `cancelableStatuses`):

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  // Add this guard:
  if (this.terminalStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OTIP); // or a dedicated error code
  }

  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction);
  }
  ...
}
```

The existing `terminalStatuses` array at lines 98–105 is already defined on the service and can be reused directly. [6](#0-5) 

## Proof of Concept

1. Authenticate as a normal user and obtain a valid JWT.
2. Create a transaction via `POST /transactions` — the user becomes its creator.
3. Allow the transaction to progress through signing and execution until its status is `EXECUTED`.
4. Issue `DELETE /transactions/:id` with the valid JWT.
5. `getTransactionForCreator` passes (user is the creator).
6. No status check is performed.
7. `repo.update(transaction.id, { status: TransactionStatus.CANCELED })` overwrites the status.
8. `repo.softRemove(transaction)` sets `deletedAt`, hiding the record from all standard queries.
9. The transaction now appears as `CANCELED` and is invisible to observers, signers, and approvers — while the on-chain execution remains immutable. [7](#0-6) [4](#0-3)
