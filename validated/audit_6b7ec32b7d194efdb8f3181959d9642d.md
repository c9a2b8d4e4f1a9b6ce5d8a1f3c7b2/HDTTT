Audit Report

## Title
`isManual` Boolean Bypasses Status Guard in `archiveTransaction`, Enabling Invalid State Transitions from Terminal States

## Summary
In `archiveTransaction`, the guard that prevents archiving a transaction in a terminal state is logically inverted. When `isManual=true`, the `&&` short-circuits and the guard never throws, allowing a transaction creator to overwrite the status of an `EXECUTED`, `FAILED`, `CANCELED`, or `EXPIRED` manual transaction to `ARCHIVED` via a direct API call.

## Finding Description

**Root cause — logical error in the archive guard:** [1](#0-0) 

```typescript
// transactions.service.ts lines 711–718
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual   // ← BUG: short-circuits the entire guard when isManual=true
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** sub-expressions are true:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- **AND** `isManual` is `false`

When `isManual=true`, `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false`. The guard never throws. Any manual transaction — regardless of whether its status is `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, or already `ARCHIVED` — passes the guard and has its status overwritten to `ARCHIVED`.

**`getTransactionForCreator` does not check status:** [2](#0-1) 

`getTransactionForCreator` only verifies that `creatorKey.userId === user.id`. It imposes no status restriction, so a creator can reach `archiveTransaction` for a transaction in any state.

**`verifyAccess` returns `true` unconditionally for terminal statuses:** [3](#0-2) 

`verifyAccess` grants access to `EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, and `ARCHIVED` transactions without any further check. This is not used by `getTransactionForCreator`, but confirms there is no secondary access barrier.

**The front-end correctly enforces the conjunction but is client-side only:** [4](#0-3) 

```typescript
const canArchive = computed(() => {
  const isManual = props.organizationTransaction?.isManual;
  return isManual && isCreator.value && transactionIsInProgress.value; // AND
});
```

The front-end requires `isManual=true` **AND** an in-progress status. The backend enforces only the disjunction. A direct API call bypasses the front-end guard entirely.

**The existing e2e test does not cover the vulnerable path:** [5](#0-4) 

The test at line 884 uses `addedTransactions.userTransactions[0]`, which is seeded at line 326 without `isManual: true`. [6](#0-5) 

Because `isManual` defaults to `false`, the test only validates the `isManual=false` + `EXECUTED` path (which correctly returns 400). The `isManual=true` + `EXECUTED` path — the actual bug — is never tested.

**The unit test also misses the case:** [7](#0-6) 

The "should throw if transaction status is not archiveable" unit test uses `status: CANCELED` without setting `isManual: true`. It passes because `isManual` is `undefined`/`false`, so the guard fires. A test with `isManual: true` and `status: CANCELED` would fail (no throw), exposing the bug.

## Impact Explanation

A transaction creator can:
1. Create a manual transaction (`isManual=true`).
2. Allow it to reach a terminal state (`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`).
3. Issue a direct `PATCH /transactions/archive/:id` API call.
4. The backend overwrites `status = ARCHIVED`, permanently corrupting the execution record.
5. The transaction is deprioritized in all subsequent lookups that treat `ARCHIVED` as inactive, effectively hiding the execution or failure from other organization members and audit views.

A `FAILED` transaction can be silently archived, concealing failure details from approvers and observers. An `EXECUTED` transaction can be re-labeled, corrupting the audit trail.

## Likelihood Explanation

- **Precondition:** Authenticated user who is the creator of any manual transaction that has reached a terminal state. No privileged role required.
- **Trigger:** Single direct API call to `PATCH /transactions/archive/:id`, bypassing the front-end guard.
- **`isManual`** is a normal field set at transaction creation time by the creator.
- **Likelihood: Medium.** The path is trivially reachable by any creator of a manual transaction; it requires only knowledge of the API endpoint.

## Recommendation

Replace the `&&` (AND) guard with an `||` (OR) guard so that the exception is thrown unless **both** conditions are satisfied — the transaction is manual **and** its status is in the allowed set:

```typescript
// Correct guard: throw unless isManual=true AND status is archiveable
if (
  !transaction.isManual ||
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Add a unit test covering `isManual=true` with each terminal status (`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, `ARCHIVED`) and an e2e test covering `isManual=true` + `EXECUTED`.
