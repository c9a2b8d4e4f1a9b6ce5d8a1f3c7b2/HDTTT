Audit Report

## Title
Any Authenticated User Can Post and Read Comments on Transactions They Have No Access To

## Summary
`CommentsController` enforces only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but performs zero transaction-level membership verification. Every analogous transaction sub-resource enforces that the requesting user is a creator, signer, approver, or observer. The comments endpoints skip this check entirely — a gap the codebase explicitly acknowledges with a developer `//TODO` comment.

## Finding Description

**Root cause:** `CommentsController` applies only authentication-level guards at the controller level. [1](#0-0) 

The `createComment` handler passes the request directly to `CommentsService.createComment` with no transaction existence check and no membership check. The developer-acknowledged gap is explicit: [2](#0-1) 

`CommentsService.createComment` creates and persists the comment with no access verification whatsoever: [3](#0-2) 

`getComments` and `getCommentById` similarly perform no access check: [4](#0-3) 

**Contrast with analogous operations that do enforce access:**

`GET /transactions/:id` calls `getTransactionWithVerifiedAccess`, which enforces creator/signer/approver/observer membership before returning data: [5](#0-4) 

`GET /transactions/:transactionId/approvers` calls `getVerifiedApproversByTransactionId`, which enforces the same membership check and throws `UnauthorizedException` on failure: [6](#0-5) [7](#0-6) 

`DELETE /transactions/:transactionId/approvers/:id` calls `getCreatorsTransaction` to confirm the user is the transaction creator: [8](#0-7) 

The `verifyAccess` logic used by the transaction service confirms the intended access model — only creators, signers, approvers, and observers are permitted: [9](#0-8) 

## Impact Explanation

- **Unauthorized state mutation:** Any authenticated user can inject comments into any transaction in the system, polluting the audit trail and potentially misleading legitimate participants.
- **Information disclosure:** Comments may contain sensitive operational context (key decisions, coordination details). Any authenticated user can read them for any transaction, breaking the confidentiality boundary enforced by the rest of the transaction access model.
- **Audit log integrity:** The comment `user` field is set to the attacker's user record, corrupting the audit trail with entries from users who have no legitimate relationship to the transaction.

## Likelihood Explanation

Exploitation requires only a valid, verified user account — the lowest privilege level in the system. Transaction IDs are sequential integers (auto-incremented primary keys), making enumeration trivial. No special tooling, timing, or race condition is required. The attack is a single authenticated HTTP request.

## Recommendation

Apply the same access verification used by `getTransactionWithVerifiedAccess` (or an equivalent service-layer check) inside `CommentsService` before creating or returning comments. Specifically:

1. **`createComment`**: Before persisting, verify the requesting user is a creator, signer, approver, or observer of the target transaction. Throw `UnauthorizedException` if not.
2. **`getComments` / `getCommentById`**: Require the requesting user's identity and apply the same membership check before returning comment data.

The existing `TransactionsService.verifyAccess` and `ApproversService.getVerifiedApproversByTransactionId` methods demonstrate the correct pattern to follow.

## Proof of Concept

```
# Step 1: Authenticate as any verified user (lowest privilege)
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "access_token": "<JWT>" }

# Step 2: Probe a transaction the attacker has no access to
GET /transactions/42
Authorization: Bearer <JWT>
→ 401 Unauthorized  (access control works here)

# Step 3: Post a comment to the same transaction — succeeds with no access check
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "message": "attacker comment" }
→ 201 Created  (comment persisted against transaction #42)

# Step 4: Read all comments on the same transaction — succeeds with no access check
GET /transactions/42/comments
Authorization: Bearer <JWT>
→ 200 OK  [ { "id": 1, "message": "...", "user": {...}, ... } ]
```

The `//TODO need some sort of guard or check to ensure user can comment here` comment at [10](#0-9)  confirms the developers are aware this check is missing.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
