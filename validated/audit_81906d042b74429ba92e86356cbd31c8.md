### Title
Missing Transaction Status Re-evaluation After Child Approver Removal

### Summary

`removeTransactionApprover` in `approvers.service.ts` emits `emitTransactionStatusUpdate` using `approver.transactionId`, which is `null` for child (non-root) approvers in a threshold tree. This causes the status-update event to be published with `entityId: null`, which the notification service silently drops, leaving the transaction's status stale after the approver tree is structurally modified. Other approver-mutating functions (`createTransactionApprovers`, `updateTransactionApprover`) correctly use the actual `transactionId` parameter.

### Finding Description

**Root cause — data model invariant:**

Child approvers in a threshold tree are stored with `transactionId = null` and `listId = <parent_id>`. Only root-level approvers carry a non-null `transactionId`. This is enforced at creation time: [1](#0-0) 

**Vulnerable function:**

`removeTransactionApprover` fetches the approver by its own `id`, soft-deletes it (and all its descendants via `removeNode`), then emits the status-update event using `approver.transactionId`: [2](#0-1) 

When the deleted approver is a child node, `approver.transactionId` is `null`. The published event therefore carries `{ entityId: null }`.

**How the notification service handles null:**

`processTransactionStatusUpdateNotifications` looks up the transaction by `entityId`. A null ID finds nothing in the map and the event is silently skipped with a `console.warn`: [3](#0-2) 

No status re-evaluation (`processTransactionStatus`) is triggered for the affected transaction.

  async removeTransactionApprover(id: number): Promise<void> {

    const result = await this.removeNode(approver.id);
