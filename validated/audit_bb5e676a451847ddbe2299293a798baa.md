### Title
Any Authenticated User Can Post Comments on Any Transaction Without Access Verification

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` performs no check to verify that the requesting user is a participant (creator, signer, observer, or approver) of the target transaction. Any authenticated organization user can post comments on any transaction by supplying an arbitrary `transactionId`. The code itself contains an explicit `//TODO` acknowledging the missing guard. Additionally, `GET /transactions/:transactionId/comments` and `GET /transactions/:transactionId/comments/:id` expose all comments for any transaction to any authenticated user without access verification.

### Finding Description

**Root cause — `CommentsController`:** [1](#0-0) 

The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level — these only confirm the caller holds a valid, non-blacklisted JWT and has a verified account. There is no guard or service-level check that the user is the creator, a signer, an observer, or an approver of the transaction identified by `:transactionId`. The developer acknowledged this gap inline:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}
```

**Root cause — `CommentsService.createComment`:** [2](#0-1) 

The service blindly creates and persists the comment for the supplied `transactionId` with no membership lookup. Compare this to `TransactionsService.verifyAccess`, which correctly checks creator, observer, signer, and approver membership before granting access: [3](#0-2) 

The `createComment` path never calls `verifyAccess` or any equivalent.

**Read endpoints are also unguarded:** [4](#0-3) 

`getComments` and `getCommentById` return comment data for any `transactionId` or comment `id` to any authenticated user, leaking the existence and content of comments on transactions the user has no business accessing.

**Exploit path:**
1. Attacker registers as a normal organization user (no admin, no special role).
2. Attacker obtains a valid JWT via `POST /auth/login`.
3. Attacker sends `POST /transactions/42/comments` with any `transactionId` and a comment body.
4. The request passes all guards (JWT valid, user verified) and `CommentsService.createComment` persists the comment without any membership check.
5. Attacker can also enumerate comments on any transaction via `GET /transactions/42/comments`.

### Impact Explanation
- **Unauthorized write on any transaction:** Any authenticated user can inject comments into transactions they have no role in, polluting the audit trail and potentially confusing legitimate participants.
- **Information disclosure:** Successfully posting or reading comments on a transaction confirms its existence and exposes its comment history to unauthorized users, leaking workflow and decision metadata.
- **Integrity degradation:** The comment history is part of the transaction's collaborative record. Unauthorized entries corrupt the integrity of that record.

### Likelihood Explanation
The attacker precondition is only a valid organization account — the lowest privilege level in the system. No admin access, no leaked secrets, and no special network position are required. The endpoint is reachable over the standard REST API. The `//TODO` comment confirms the developers are aware the check is missing, meaning the gap is not accidental but deferred, making it reliably reproducible.

### Recommendation
Add a membership check in `CommentsService.createComment` (and in the read endpoints) by calling `TransactionsService.verifyAccess` or an equivalent query before persisting or returning data:

```typescript
async createComment(user: User, transactionId: number, dto: CreateCommentDto) {
  const transaction = await this.transactionRepo.findOne({
    where: { id: transactionId },
    relations: ['creatorKey', 'observers', 'signers', 'approvers'],
  });
  if (!transaction) throw new NotFoundException();
  const hasAccess = await this.transactionsService.verifyAccess(transaction, user);
  if (!hasAccess) throw new UnauthorizedException();
  // ... existing create logic
}
```

Alternatively, introduce a dedicated `CommentAccessGuard` that replicates the `verifyAccess` logic and apply it to all three routes in `CommentsController`.

### Proof of Concept

**Preconditions:** Two accounts exist — `creator` (owns transaction ID 42) and `attacker` (has no role on transaction 42). Both have valid JWTs.

```http
# Step 1 — attacker logs in
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
# → 200 OK, returns attackerToken

# Step 2 — attacker posts a comment on a transaction they don't own
POST /transactions/42/comments
Authorization: Bearer <attackerToken>
Content-Type: application/json
{ "content": "Injected comment by unauthorized user" }
# → 201 Created  (expected: 401 Unauthorized)

# Step 3 — attacker reads all comments on the transaction
GET /transactions/42/comments
Authorization: Bearer <attackerToken>
# → 200 OK, returns full comment list including legitimate participants' comments
# (expected: 401 Unauthorized)
```

**Expected outcome:** Both requests succeed. The attacker's comment is persisted and visible to all participants, and the attacker can read all existing comments — without being the creator, signer, observer, or approver of transaction 42.

```typescript
  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
```
