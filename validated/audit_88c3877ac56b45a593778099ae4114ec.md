Audit Report

## Title
Critically Weak PBKDF2 Iteration Count (2560) Renders Password-Encrypted Private Keys Trivially Brute-Forceable

## Summary
The `deriveKey` function in `front-end/src/main/utils/crypto.ts` uses only **2560 iterations** of PBKDF2-SHA512 to derive the AES-256-GCM key that encrypts all Hedera private keys stored in the local SQLite database. OWASP recommends 210,000 iterations for PBKDF2-SHA512; 2560 is ~82× below that floor. Any attacker who obtains the SQLite database file can recover plaintext private keys by brute-forcing the user's password at GPU speeds that make the protection functionally equivalent to no encryption.

## Finding Description

**Root cause — `front-end/src/main/utils/crypto.ts`, lines 5–10:**

The `deriveKey` function hardcodes 2560 iterations:

```typescript
export function deriveKey(password: string, salt: Buffer) {
  const iterations = 2560;   // critically low
  const keyLength = 32;
  return crypto.pbkdf2Sync(password, salt, iterations, keyLength, 'sha512');
}
``` [1](#0-0) 

Both `encrypt()` and `decrypt()` in the same file call `deriveKey()` directly, making this the sole KDF for all password-based key protection: [2](#0-1) [3](#0-2) 

**Where the weak KDF is applied — `front-end/src/main/services/localUser/keyPairs.ts`:**

`storeKeyPair` calls `encrypt(keyPair.private_key, password)` (which internally calls `deriveKey`) when the keychain is not used and a password is provided. The resulting ciphertext is written to the local SQLite database via Prisma: [4](#0-3) 

The same weak KDF is used when re-encrypting keys in `changeDecryptionPassword`: [5](#0-4) 

**IPC exposure — `front-end/src/main/modules/ipcHandlers/localUser/keyPairs.ts`:**

Both `storeKeyPair` and `changeDecryptionPassword` are registered as live IPC channels callable from the renderer process, confirming this is the production code path: [6](#0-5) 

## Impact Explanation

Complete theft of all Hedera assets controlled by any private key stored in password-encrypted mode. The AES-256-GCM cipher is sound, but the key derivation is so weak that the cipher provides no meaningful protection against an offline attacker. Any user who chose the password path (not macOS keychain) and whose database file is ever exfiltrated — through malware, cloud backup compromise, or device resale — is fully exposed. This is a **critical** cryptographic integrity failure.

## Likelihood Explanation

- The password-based encryption path is the default for non-macOS users and for macOS users who decline keychain mode.
- Electron app data directories are routinely included in OS cloud backups (macOS Time Machine, iCloud Drive, Windows Backup); no special attacker capability beyond compromising a backup account is required.
- The attack is fully offline after obtaining the file; no interaction with the victim or the Hedera network is needed.
- GPU-accelerated PBKDF2 cracking tools (hashcat mode 12000) are publicly available and require no specialized knowledge.
- At 2560 iterations, a single RTX 4090 can test ~500,000–1,000,000 candidate passwords per second. A 6-character lowercase password space (~300 million candidates) is exhausted in under 10 minutes; common-password dictionaries (rockyou, etc.) are cracked in seconds.

## Recommendation

Increase the PBKDF2 iteration count in `deriveKey` to at minimum **210,000** (OWASP recommendation for PBKDF2-SHA512) or, preferably, replace PBKDF2 entirely with **Argon2id** (which the project already imports via the `argon2` package used in `hash()`/`verifyHash()`). Argon2id with `m=64MB, t=3, p=4` provides memory-hard resistance that GPU cracking cannot easily parallelize.

Additionally, implement a migration path to re-encrypt existing stored keys with the stronger KDF on next user login.

## Proof of Concept

1. User stores a Hedera private key via the normal onboarding flow (password mode, not keychain). `storeKeyPair` is called → `encrypt()` → `deriveKey()` with 2560 iterations and a random 64-byte salt → ciphertext written to SQLite.
2. Attacker obtains the SQLite database file (e.g., via cloud backup compromise).
3. Attacker extracts the base64 blob, parses out the 64-byte salt (bytes 0–63), 16-byte IV (bytes 64–79), 16-byte GCM tag (bytes 80–95), and ciphertext (bytes 96+).
4. Attacker runs: `hashcat -m 12000 -a 0 <hash_file> rockyou.txt` — hashcat mode 12000 is PBKDF2-SHA512.
5. At 2560 iterations on an RTX 4090, the full `rockyou.txt` dictionary (~14 million entries) is exhausted in **under 30 seconds**. A 6-character lowercase brute-force completes in **under 10 minutes**.
6. Recovered plaintext private key is used to sign Hedera transactions and drain all assets from the associated account.

export function encrypt(data: string, password: string) {
  const iv = crypto.randomBytes(16);
  const salt = crypto.randomBytes(64);

  const key = deriveKey(password, salt);

  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([cipher.update(data, 'utf8'), cipher.final()]);

  const tag = cipher.getAuthTag();

  return Buffer.concat([salt, iv, tag, encrypted]).toString('base64');
}

export function decrypt(data: string, password: string) {
  const bData = Buffer.from(data, 'base64');

  const salt = bData.subarray(0, 64);
  const iv = bData.subarray(64, 80);
  const tag = bData.subarray(80, 96);
  const text = bData.subarray(96).toString('base64');

  const key = deriveKey(password, salt);

  const decipher = crypto.createDecipheriv('aes-256-gcm', key, iv);
  decipher.setAuthTag(tag);

  const decrypted = decipher.update(text, 'base64', 'utf8') + decipher.final('utf8');

  return decrypted;
}
```
