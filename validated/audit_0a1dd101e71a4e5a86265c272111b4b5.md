Audit Report

## Title
Global IP Rate Limiter Bypassed via Attacker-Controlled `X-Forwarded-For` Header, Enabling Unlimited Brute-Force on Login

## Summary
`IpThrottlerGuard`, the sole global rate limiter, derives its tracking key from the unvalidated `X-Forwarded-For` request header. An unauthenticated attacker can rotate spoofed IP values in this header to assign each request to a fresh rate-limit bucket, completely nullifying the configured limits. Because `EmailThrottlerGuard` on `POST /auth/login` is ordered after `LocalAuthGuard` and never executes on failed credential attempts, the IP throttler is the only protection against brute-force ã and it is trivially bypassed.

## Finding Description

**Root cause ã `IpThrottlerGuard.getTracker()`:**

`getTracker()` unconditionally prefers the client-supplied `x-forwarded-for` header over the socket-level `req.ip`:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [1](#0-0) 

This header is never stripped or validated. The returned string becomes the Redis key used to count requests against the configured limits.

**Global registration ã `ApiModule`:**

`IpThrottlerGuard` is registered as `APP_GUARD`, making it the first guard executed on every incoming request:

```typescript
{
  provide: APP_GUARD,
  useClass: IpThrottlerGuard,
},
``` [2](#0-1) 

**Why `EmailThrottlerGuard` does not compensate on `/auth/login`:**

The login route is decorated as:

```typescript
@UseGuards(LocalAuthGuard, EmailThrottlerGuard)
``` [3](#0-2) 

NestJS executes guards in declaration order. `LocalAuthGuard` invokes the Passport local strategy via `LocalStrategy.validate()`, which calls `usersService.getVerifiedUser()` and throws `UnauthorizedException` on wrong credentials ã before `EmailThrottlerGuard` ever runs. [4](#0-3) 

The email-based throttle therefore only fires on *successful* logins, providing zero protection against failed credential attempts.

**`POST /auth/reset-password` secondary impact:**

This endpoint uses only `EmailThrottlerGuard` with no `LocalAuthGuard`:

```typescript
@UseGuards(EmailThrottlerGuard)
async createOtp(@Body() { email }: OtpLocalDto)
``` [5](#0-4) 

Bypassing the global IP throttler via `X-Forwarded-For` spoofing also bypasses the only unauthenticated-path guard here, enabling unlimited OTP email flooding.

## Impact Explanation

- **Credential brute-force**: An attacker can submit an unlimited number of password guesses against any known user email. Bcrypt/argon2 hashing is intentionally slow; at scale this also exhausts server CPU.
- **OTP email flooding**: Spoofing `X-Forwarded-For` on `POST /auth/reset-password` allows triggering unlimited OTP emails to any user's inbox.
- **Severity**: High ã direct path to account compromise via brute-force; no privileges required.

## Likelihood Explanation

- **Attacker preconditions**: None. The endpoint is public; the `X-Forwarded-For` header requires no special tooling ã a single `curl -H` flag suffices.
- **Detectability**: Each request appears to originate from a different IP, making the attack indistinguishable from normal traffic at the application layer.
- **Realistic scenario**: Any attacker with a list of registered email addresses can run an automated credential-stuffing or dictionary attack at arbitrary speed.

## Recommendation

1. **Do not trust `X-Forwarded-For` directly.** If the application sits behind a trusted reverse proxy, configure the proxy to set a single, canonical forwarded-IP header and use `req.ip` (which Express populates correctly when `trust proxy` is configured). Remove the `req.headers['x-forwarded-for']` fallback in `IpThrottlerGuard.getTracker()`. [6](#0-5) 

2. **Reorder guards on `/auth/login`.** Place `EmailThrottlerGuard` before `LocalAuthGuard` so the per-email rate limit is enforced regardless of whether credentials are correct:
   ```typescript
   @UseGuards(EmailThrottlerGuard, LocalAuthGuard)
   ``` [3](#0-2) 

3. **Add a per-email throttle to `POST /auth/reset-password`** (already present) and ensure the IP throttler cannot be bypassed as the sole unauthenticated-path guard.

## Proof of Concept

```bash
# Brute-force login with rotating spoofed IPs ã no rate limit ever triggers
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://target/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.$((i/256)).$((i%256))" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
```

Each iteration uses a fresh IP key in Redis; no bucket ever reaches the configured limit. `LocalAuthGuard` performs a full password hash comparison on every attempt.
