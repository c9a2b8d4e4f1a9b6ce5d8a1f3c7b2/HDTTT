### Title
Unbounded Iteration Over User-Controlled Arrays Causes Application-Layer DoS

### Summary
The external report describes a Solidity gas-overflow DoS where unbounded loops exhaust the block gas limit. The direct analog in this TypeScript/NestJS codebase is unbounded iteration over user-controlled or database-fetched arrays without any size cap, causing CPU exhaustion, memory pressure, and cascading database connection saturation on the API server. Two distinct entry points are confirmed.

---

### Finding Description

**Vector 1 — `GET /transactions/sign` (`getTransactionsToSign`)**

`getTransactionsToSign` fetches **all** non-terminal transactions from the database with no `take` (LIMIT) clause, then executes one async `userKeysToSign` call per transaction inside a sequential `for` loop. Pagination is applied only to the final in-memory slice, meaning the full unbounded set is always processed. [1](#0-0) 

The `find` call at line 295 has no `take` property, so it returns every matching row. The loop at line 301 then calls `userKeysToSign` (which itself performs async DB queries via `TransactionSignatureService`) for every row before pagination is applied at line 312. [2](#0-1) 

**Vector 2 — `POST /transactions/signatures/import` (`importSignatures`)**

`importSignatures` accepts an array of `UploadSignatureMapDto` with no `@ArrayMaxSize()` constraint. The developer explicitly acknowledged the missing limit in a comment at line 575:

> `//Added a batch mechanism, probably should limit this on the api side of things` [3](#0-2) 

The DTO has no maximum-size decorator: [4](#0-3) 

Each element in the array triggers: a transaction DB lookup, `SDKTransaction.fromBytes` deserialization, `validateSignature` (cryptographic work), and a bulk SQL `CASE` update. With a large array, this saturates the DB connection pool and blocks the Node.js event loop.

**Supporting — `POST /transaction-groups` (`createTransactionGroup`)**

`CreateTransactionGroupDto.groupItems` has `@IsArray()` and `@IsNotEmpty()` but no `@ArrayMaxSize()`: [5](#0-4) 

`createTransactions` is called with the full unbounded `groupItems` array, spawning one `validateAndPrepareTransaction` (SDK parse + network call) per item via `Promise.all`: [6](#0-5) 

**Supporting — `POST /transactions/:id/observers` (`createTransactionObservers`)**

`CreateTransactionObserversDto.userIds` has `@ArrayMinSize(1)` but no `@ArrayMaxSize()`: [7](#0-6) 

The service iterates over every `userId` in a `for` loop with no cap: [8](#0-7) 

---

### Impact Explanation

- **Vector 1**: A single authenticated user calling `GET /transactions/sign` when the system holds thousands of transactions causes the API process to execute thousands of sequential async DB queries, blocking the event loop and exhausting the DB connection pool for all other users. The endpoint is marked `/* NO LONGER USED BY FRONT-END */` but remains fully reachable via the API.
- **Vector 2**: A single `POST /transactions/signatures/import` request with an array of thousands of elements triggers thousands of cryptographic operations and DB writes in one request, starving the server of resources.
- Both vectors are reachable by any authenticated, verified user — no admin role required.

---

### Likelihood Explanation

Any registered user can authenticate and call these endpoints. Vector 2 requires only a valid JWT and a large JSON body; no special on-chain state is needed. Vector 1 is amplified naturally as the platform grows (more transactions = worse performance per request). The developer comment in `importSignatures` confirms awareness of the missing limit, indicating it was a known gap left unaddressed.

---

### Recommendation

1. **`getTransactionsToSign`**: Apply the `take: limit` and `skip: offset` pagination parameters directly to the `repo.find` call so the DB query is bounded, and move the `userKeysToSign` filtering to a separate paginated flow.
2. **`importSignatures` / `uploadSignatureMaps`**: Add `@ArrayMaxSize(N)` (e.g., 100) to the `UploadSignatureMapDto[]` body parameter and enforce it at the controller level.
3. **`CreateTransactionGroupDto.groupItems`**: Add `@ArrayMaxSize(N)` to cap the number of group items per request.
4. **`CreateTransactionObserversDto.userIds`**: Add `@ArrayMaxSize(N)` to cap the number of observers added per request.

---

### Proof of Concept

**Vector 1 (passive growth DoS):**
```
GET /transactions/sign HTTP/1.1
Authorization: Bearer <valid_jwt>
```
With 10,000 transactions in the system, the server executes 10,000 sequential `userKeysToSign` DB calls before returning any response. Repeated calls from multiple users saturate the DB connection pool.

**Vector 2 (active request DoS):**
```http
POST /transactions/signatures/import HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 1, "signatureMap": { ... } },
  ... (repeated 10,000 times)
]
```
Each element triggers `SDKTransaction.fromBytes`, `validateSignature`, and a SQL update. The server processes all 10,000 elements synchronously within a single request, blocking the event loop and exhausting DB connections for the duration.
