Audit Report

## Title
Any Authenticated User Can Corrupt Another User's Transaction Group via Unauthorized `TransactionGroupItem` Deletion

## Summary
`removeTransactionGroup` in `TransactionGroupsService` hard-deletes each `TransactionGroupItem` row **before** the ownership check on the underlying transaction is performed. When the ownership check fails (the caller does not own the transaction), the group item has already been permanently deleted, leaving the `TransactionGroup` in a corrupted, inconsistent state. Any authenticated, verified user can trigger this against any group ID.

## Finding Description

**Root cause — delete-before-check ordering in `removeTransactionGroup`:**

`removeTransactionGroup` fetches the group with no user filter:

```typescript
const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
``` [1](#0-0) 

It then fetches all `TransactionGroupItem` records for that group, again with no user filter: [2](#0-1) 

The critical sequence inside the loop:

```typescript
for (const groupItem of groupItems) {
  const transactionId = groupItem.transactionId;
  await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // ← hard-deleted first
  await this.transactionsService.removeTransaction(transactionId, user, false); // ← ownership checked here
}
``` [3](#0-2) 

`removeTransaction` immediately delegates to `getTransactionForCreator`, which throws `UnauthorizedException` if `transaction.creatorKey?.userId !== user?.id`: [4](#0-3) [5](#0-4) 

By the time this check runs, the `TransactionGroupItem` row has already been hard-deleted. The exception propagates before `remove(TransactionGroup, group)` (line 193) is reached, so the `TransactionGroup` shell remains — but permanently missing one item. [6](#0-5) 

**No ownership guard on the DELETE endpoint:**

The controller applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no `AdminGuard` or ownership guard. Any verified user can call `DELETE /transaction-groups/:id`: [7](#0-6) [8](#0-7) 

**Group ID enumeration — `GET /transaction-groups` returns all groups with no user filter:**

```typescript
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.dataSource.manager.find(TransactionGroup);
}
``` [9](#0-8) 

The comment in the controller even reads `/* TESTING ONLY: Get all transactions groups */`, confirming this endpoint was never intended for production use but remains accessible to all verified users: [10](#0-9) 

**Contrast with `cancelTransactionGroup` — which correctly checks ownership first:**

`cancelTransactionGroup` calls `this.getTransactionGroup(user, groupId, false)`, which runs a user-scoped SQL query (`getTransactionGroupItemsQuery` with user filter) and throws `UnauthorizedException` if the user has no access — before any mutation occurs. It then additionally verifies all transactions are owned by the caller before proceeding: [11](#0-10) 

`removeTransactionGroup` skips both of these steps entirely.

## Impact Explanation

An attacker who is a regular authenticated user can:

1. Enumerate all group IDs via `GET /transaction-groups`.
2. Call `DELETE /transaction-groups/:victimGroupId`.
3. The first `TransactionGroupItem` in the victim's group is permanently hard-deleted from the database.
4. The `TransactionGroup` record remains, but is now missing one or more items.

This causes **permanent, unrecoverable corruption** of another user's transaction group. In a multi-signature workflow, the group creator loses visibility into their grouped transactions, and the group's sequential execution logic is broken. The victim's pending Hedera transactions may become unexecutable through the group interface.

## Likelihood Explanation

- Attacker precondition: authenticated, verified organization user — the lowest privilege level in the system.
- No admin role, no leaked credentials, no privileged keys required.
- Group IDs are discoverable via the `GET /transaction-groups` endpoint accessible to all verified users.
- The attack is a single HTTP `DELETE` request.

## Recommendation

1. **Add a pre-deletion ownership check**: Before entering the deletion loop, call `this.getTransactionGroup(user, id, false)` (as `cancelTransactionGroup` does) to verify the caller has access to the group. This will throw `UnauthorizedException` before any mutation occurs.
2. **Wrap the entire operation in a database transaction**: Use `this.dataSource.transaction(...)` so that if any step fails (including the ownership check on any item), all prior deletions are rolled back atomically.
3. **Restrict `GET /transaction-groups`**: The endpoint is marked `TESTING ONLY` but is still live and unguarded. It should be removed or restricted to admin roles to prevent group ID enumeration.
4. **Reorder the loop**: At minimum, check ownership of all transactions before deleting any `TransactionGroupItem` rows.

## Proof of Concept

```
# Step 1: Enumerate all group IDs (no ownership filter)
GET /transaction-groups
Authorization: Bearer <attacker_jwt>
# Response: [{id: 42, ...}, {id: 43, ...}]  ← victim's group id = 43

# Step 2: Trigger the corrupting delete
DELETE /transaction-groups/43
Authorization: Bearer <attacker_jwt>

# Result:
# - TransactionGroupItem for the first item in group 43 is permanently hard-deleted
# - removeTransaction throws UnauthorizedException (attacker doesn't own the transaction)
# - TransactionGroup 43 still exists but is now missing one item
# - HTTP response: 401 Unauthorized (attacker sees an error, but damage is done)
```

The attacker receives a 401 response, but the database is already corrupted: the victim's `TransactionGroup` record (id=43) now has one fewer `TransactionGroupItem` than it should, permanently breaking the group's sequential execution logic.
