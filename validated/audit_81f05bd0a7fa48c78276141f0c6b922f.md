Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Per-Request Server Resource Exhaustion

## Summary
`GET /transactions/sign` loads every non-terminal transaction from the database with no row limit, then serially calls `userKeysToSign` on each row — an operation that deserializes transaction bytes and makes outbound Hedera mirror-node HTTP calls. Pagination parameters are accepted but only applied to an in-memory slice after all work is done. Any verified user can trigger this path.

## Finding Description

**Missing `take`/`skip` in the database query**

`getTransactionsToSign` in `transactions.service.ts` accepts `Pagination` parameters but never passes `take`/`skip` to the repository:

```typescript
// lines 295–299 — no take or skip
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

Every other list endpoint correctly passes `skip: offset, take: limit` to the query:

- `getTransactions` — [2](#0-1) 
- `getHistoryTransactions` — [3](#0-2) 
- `getTransactionsToApprove` — [4](#0-3) 

**Unbounded per-row async work**

After loading the full result set, the code iterates over every row and calls `userKeysToSign` sequentially:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [5](#0-4) 

**Pagination silently ignored at DB level**

The `page`/`size` parameters are only applied as an in-memory slice after all N rows are loaded and all N `userKeysToSign` calls complete:

```typescript
items: result.slice(offset, offset + limit),
``` [6](#0-5) 

**Endpoint is live and accessible**

The controller explicitly notes the endpoint is no longer used by the front-end, yet it remains publicly routed and protected only by standard user authentication:

```typescript
/* NO LONGER USED BY FRONT-END */
@Get('/sign')
getTransactionsToSign(...)
``` [7](#0-6) 

The controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) require only a valid, verified user — no elevated role. [8](#0-7) 

## Impact Explanation
A single authenticated user can cause the API server to load an unbounded number of transaction rows from the database, deserialize each transaction's bytes, and make one or more outbound mirror-node HTTP calls per row — all within a single synchronous request lifecycle. As the number of active (non-terminal) transactions grows through normal use or deliberate creation, individual requests block the Node.js event loop for the duration, starving all other concurrent requests. Repeated calls compound the effect, potentially causing service-wide unavailability.

## Likelihood Explanation
Any registered, verified user can create transactions (`POST /transactions`) and call `GET /transactions/sign`. No admin or privileged role is required. A large number of active transactions is a realistic steady-state in production organizations. The attacker does not need to flood the server — a single request against a sufficiently populated database is enough to cause a timeout or event-loop stall.

## Recommendation
1. **Apply DB-level pagination**: Pass `take: limit, skip: offset` to the `repo.find()` call in `getTransactionsToSign`, matching the pattern used by every other list endpoint.
2. **Remove or disable the endpoint**: The controller comment already acknowledges it is no longer used by the front-end. If it serves no active purpose, remove the route entirely to eliminate the attack surface.
3. **Parallelize or cap async work**: If the endpoint is retained, replace the serial `for...await` loop with bounded parallel execution (e.g., `Promise.all` over a limited batch) to reduce per-request latency.

## Proof of Concept
1. Register and verify a user account.
2. Call `POST /transactions` N times to create N transactions in `WAITING_FOR_SIGNATURES` status.
3. Call `GET /transactions/sign?page=1&size=10`.
4. The server loads all N rows from the database and calls `userKeysToSign` N times serially, regardless of the `size=10` parameter.
5. With a large N, the request hangs; other concurrent requests are starved for the duration.
