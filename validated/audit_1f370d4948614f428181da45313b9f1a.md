Audit Report

## Title
Unbounded SQL Query with No Pagination in `/transaction-nodes?collection=HISTORY` Causes Authenticated DoS

## Summary
The `GET /transaction-nodes` endpoint with `collection=HISTORY` executes a CTE-based SQL query against the entire `transaction` table with no `LIMIT` clause and no per-user scoping filter. Any authenticated, verified user can retrieve every terminal-status transaction in the database in a single HTTP request, exhausting server memory and database resources.

## Finding Description

**Root cause 1 — No LIMIT in the generated SQL:**

`getTransactionNodesQuery` in `transaction.queries.ts` builds a multi-CTE query whose final `SELECT` terminates with `ORDER BY rt.tx_created_at DESC` and no `LIMIT` or `OFFSET` clause. [1](#0-0) 

**Root cause 2 — HISTORY collection passes no user context:**

Every other collection (`READY_FOR_REVIEW`, `READY_TO_SIGN`, `READY_FOR_EXECUTION`, `IN_PROGRESS`) passes `user` and role flags to `getTransactionNodesQuery`. The `HISTORY` case omits both, passing only filter criteria. [2](#0-1) 

**Root cause 3 — `buildWhereClause` skips eligibility conditions when user is absent:**

When `userContext` is `undefined`, `buildEligibilityConditions` is never called. The only condition added to `eligibilityConditions` is the terminal-status override, so the final WHERE clause becomes `(status filter) AND (t.status = ANY($terminal_statuses))` — returning every terminal-status transaction across all users and all time. [3](#0-2) 

**Root cause 4 — Controller exposes the endpoint with no pagination guard:**

The controller accepts the `collection` enum and returns the full array. No `PaginationParams` decorator, no `size` cap, no throttler guard is applied to this route. [4](#0-3) 

**Confirmed by k6 load-test comments — intentional design, not an oversight:**

The k6 performance script explicitly documents that the endpoint "returns all items in single request" and asserts that 500+ items must be returned per call. [5](#0-4) 

**Secondary instance — `getTransactionsToSign` unbounded fetch + per-row loop:**

`transactions.service.ts` fetches every non-terminal transaction with no `take` limit, then iterates the full result set calling `userKeysToSign` per row — an O(n) async loop over an unbounded dataset. [6](#0-5) 

## Impact Explanation

- **Memory exhaustion**: The Node.js process must buffer the entire result set before serialization. With tens of thousands of historical transactions, a single request can consume hundreds of megabytes.
- **Database resource exhaustion**: The unbounded CTE holds a full sequential scan open, consuming PostgreSQL worker memory and a connection slot for the full duration of the query.
- **Cascading DoS**: Because NestJS processes requests synchronously within the event loop, concurrent requests from a single authenticated user can saturate the database connection pool and stall all other API consumers.
- **Permanent and worsening degradation**: The `transaction` table only grows; there is no archival or pruning mechanism. The attack surface increases monotonically with organizational usage.

## Likelihood Explanation

- **Attacker precondition**: A valid JWT for any `VerifiedUser` — the lowest privilege level enforced by `VerifiedUserGuard`.
- **Trigger**: A single HTTP GET: `GET /transaction-nodes?collection=HISTORY&network=mainnet`
- **No rate limiting**: The `transaction-nodes` controller applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. None of the `UserThrottlerGuard`, `IpThrottlerGuard`, or `EmailThrottlerGuard` present elsewhere in the API are applied here. [7](#0-6) 

- **Realistic scenario**: A production organization that has processed thousands of transactions over months will naturally accumulate a HISTORY collection large enough to make this query expensive. Repeated calls from a single attacker amplify the impact.

## Recommendation

1. **Add pagination to `getTransactionNodesQuery`**: Accept `limit` and `offset` parameters and append `LIMIT $n OFFSET $m` to the final `SELECT`.
2. **Enforce a maximum page size** at the controller level (e.g., `size ≤ 100`) consistent with other paginated endpoints.
3. **Scope HISTORY to the requesting user**: Pass `user` and `{ signer: true, creator: true, observer: true, approver: true }` to `getTransactionNodesQuery` for the HISTORY collection, as is done for all other collections.
4. **Apply a throttler guard** (e.g., `UserThrottlerGuard`) to the `transaction-nodes` controller.
5. **Fix `getTransactionsToSign`**: Add a `take` limit to the `repo.find()` call and perform the `userKeysToSign` check in the database query rather than in a post-fetch loop.

## Proof of Concept

```
# Prerequisites: valid JWT for any verified organization user
TOKEN="<jwt>"
NETWORK="mainnet"

# Single request returns every terminal-status transaction in the database
curl -s -H "Authorization: Bearer $TOKEN" \
  "https://<api-host>/transaction-nodes?collection=HISTORY&network=$NETWORK" \
  | wc -c   # observe response size growing with table size

# Concurrent amplification (10 parallel requests)
for i in $(seq 1 10); do
  curl -s -H "Authorization: Bearer $TOKEN" \
    "https://<api-host>/transaction-nodes?collection=HISTORY&network=$NETWORK" &
done
wait
# Observe API latency degradation / connection pool exhaustion
```
