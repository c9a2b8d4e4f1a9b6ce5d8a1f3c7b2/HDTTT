Audit Report

## Title
Creator Can Archive Already-Executed Manual Transactions, Corrupting Audit Trail

## Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a logic flaw in its status guard: for any manual transaction (`isManual = true`), the guard condition short-circuits to `false` and never throws, regardless of the transaction's current status. The subsequent database update has no status condition in the `WHERE` clause, allowing a transaction creator to unconditionally overwrite any terminal status (e.g., `EXECUTED`) with `ARCHIVED` by calling the API endpoint directly.

## Finding Description

**Root Cause — Flawed Guard Logic**

The guard in `archiveTransaction` is:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

For any manual transaction, `!transaction.isManual` evaluates to `false`. Because the condition uses `&&`, the entire expression is `false` regardless of `transaction.status`. The guard **never throws** for manual transactions — including those in `EXECUTED`, `FAILED`, `EXPIRED`, or `REJECTED` states.

**Unconditional Update**

The subsequent update has no status guard in the `WHERE` clause:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

This is in direct contrast to `cancelTransactionWithOutcome`, which uses an atomic conditional update:

```typescript
.andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })
``` [3](#0-2) 

No equivalent guard exists in `archiveTransaction`.

**Frontend Enforcement Is Insufficient**

The frontend restricts the archive button to in-progress manual transactions:

```typescript
const canArchive = computed(() => {
  const isManual = props.organizationTransaction?.isManual;
  return isManual && isCreator.value && transactionIsInProgress.value;
});
``` [4](#0-3) 

`isInProgressStatus` explicitly excludes `EXECUTED`, `FAILED`, `EXPIRED`, and `REJECTED`: [5](#0-4) 

This frontend check is client-side only and is trivially bypassed by calling the API endpoint directly.

## Impact Explanation

An executed Hedera transaction is immutable on-chain, but the organization's internal database record is mutable. By overwriting `EXECUTED` with `ARCHIVED`, a malicious creator can conceal that a transaction was executed — hiding an unauthorized fund transfer or account modification from post-hoc review. The corruption is permanent unless an administrator manually corrects the database. Auditors and observers relying on the organization's transaction history will see a misleading record.

## Likelihood Explanation

**High for the manual transaction case.** No race condition, no privileged access, and no special timing is required. Any registered organization user who creates a manual transaction can exploit this after execution by issuing a single direct HTTP `PATCH /transactions/{id}/archive` request. The only precondition is being the creator of a manual transaction.

A secondary TOCTOU issue also exists for non-manual transactions in `WAITING_FOR_EXECUTION`: if the chain service executes the transaction between the status read and the unconditional `repo.update`, the archive call overwrites `EXECUTED` with `ARCHIVED`. This requires precise timing and is lower likelihood.

## Recommendation

1. **Fix the guard logic** so that manual transactions are also subject to a status check. The intended behavior (based on the comment `/* Archive the transaction if the transaction is sign only */`) is that archiving should only be permitted for in-progress statuses:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

2. **Add a status guard to the `WHERE` clause** of the update, mirroring the pattern used in `cancelTransactionWithOutcome`:

```typescript
await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', {
    statuses: [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION],
  })
  .execute();
```

This atomically prevents both the manual-transaction bypass and the TOCTOU race for non-manual transactions.

## Proof of Concept

1. Register as a normal organization user (no admin privileges needed).
2. Create a manual transaction and add required signers/approvers.
3. Allow the transaction to collect signatures, reach `WAITING_FOR_EXECUTION`, and be executed on the Hedera network — status becomes `EXECUTED`.
4. Directly call `PATCH /transactions/{id}/archive` via HTTP (bypassing the frontend).
5. Backend reads `transaction.status = EXECUTED`, evaluates the guard: `!isManual = false` → the `&&` expression is `false` → guard does not throw.
6. `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` runs unconditionally.
7. The `EXECUTED` transaction is now recorded as `ARCHIVED` in the organization database, concealing its execution from the audit trail.
