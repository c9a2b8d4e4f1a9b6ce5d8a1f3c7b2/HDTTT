Audit Report

## Title
Unbounded Approvers Array in `POST /transactions/:transactionId/approvers` Enables Single-Request Resource Exhaustion DoS

## Summary
The `createTransactionApprovers` endpoint accepts a `CreateTransactionApproversArrayDto` whose `approversArray` field and the recursive `CreateTransactionApproverDto.approvers` field both lack any maximum-size constraint. The service processes the submitted tree via an unbounded recursive async function that issues one sequential database insert per node, all inside a single open database transaction. A single crafted request from any authenticated transaction creator can trigger thousands of sequential DB writes and deep call-stack recursion, exhausting server and database resources.

## Finding Description

**Root cause — missing `@ArrayMaxSize` at every level of the DTO:**

`CreateTransactionApproversArrayDto.approversArray` carries only `@IsArray` and `@ValidateNested({ each: true })` with no upper-bound decorator. [1](#0-0) 

`CreateTransactionApproverDto.approvers` carries `@IsArray`, `@ArrayMinSize(1)`, `@IsOptional`, and `@ValidateNested({ each: true })` — again, no `@ArrayMaxSize`. [2](#0-1) 

The private `validateApprover` method only rejects a completely empty approver object; it imposes no size or depth limit. [3](#0-2) 

**Service processes the tree with unbounded sequential recursion:**

The inner `createApprover` closure is called sequentially with `await` for every element of `dto.approversArray`, and then recursively for every element of each `dtoApprover.approvers`. Each call issues a `transactionalEntityManager.insert(TransactionApprover, approver)` — a synchronous DB round-trip — before descending into children. The entire tree is processed inside a single `dataSource.transaction`, holding the DB connection open for the full duration. [4](#0-3) 

**Controller exposes the endpoint to any authenticated transaction creator:**

Guards are `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` only. The sole additional restriction is `getCreatorsTransaction`, which limits callers to the creator of the target transaction — a role any registered user can hold. [5](#0-4) 

**IP throttler does not mitigate this:**

A global `IpThrottlerGuard` is registered, but its limits are environment-variable-driven (and set to 9999/99999 in the test configuration). More critically, the guard trusts the `x-forwarded-for` header without validation, making IP-based throttling bypassable. [6](#0-5) [7](#0-6) 

## Impact Explanation

- **Flat attack**: Sending `approversArray` with N=10,000 leaf approvers (each with a distinct, valid `userId`) causes N sequential DB inserts inside one open transaction, blocking a DB connection slot and the NestJS event loop for the entire duration.
- **Recursive (chain) attack**: A linear chain of depth D (each node has `threshold: 1` and one child) causes D sequential DB inserts and D levels of async call-stack depth. Node.js overflows its call stack at roughly 10,000–15,000 frames, crashing the request handler and potentially the process.
- **Cascading effect**: Because all inserts run inside a single `dataSource.transaction`, a crash or timeout leaves the DB transaction open until the connection is forcibly closed, consuming a DB connection slot for the entire duration.
- **Result**: API service unavailability or severe degradation for all users.

## Likelihood Explanation

Any registered, verified user who has created at least one transaction can reach this endpoint. No admin credentials, no leaked secrets, and no special role beyond transaction creator are required. The crafted payload is trivially constructed — a deeply nested JSON object or a large flat array — and can fit within the default NestJS body-size limit (100 KB) while still triggering thousands of recursive calls or DB inserts.

## Recommendation

1. **Add `@ArrayMaxSize` to both array fields** in the DTOs:
   - `CreateTransactionApproversArrayDto.approversArray`: e.g., `@ArrayMaxSize(50)`
   - `CreateTransactionApproverDto.approvers`: e.g., `@ArrayMaxSize(20)`
2. **Enforce a maximum recursion depth** inside `createApprover` by passing a depth counter and throwing if it exceeds a safe limit (e.g., 5–10 levels).
3. **Validate `x-forwarded-for`** in `IpThrottlerGuard` to prevent IP spoofing that bypasses rate limiting.
4. **Apply a per-user throttle** (a `UserThrottlerGuard` already exists in the codebase) to the approvers `POST` endpoint.

## Proof of Concept

**Flat attack (10,000 sequential DB inserts in one transaction):**
```http
POST /transactions/1/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    { "userId": 1 },
    { "userId": 2 },
    ...
    { "userId": 10000 }
  ]
}
```

**Recursive chain attack (stack overflow / deep call stack):**
```json
{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                "... (repeated ~10,000 levels deep) ..."
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

Each level satisfies the `threshold <= approvers.length` and `@ArrayMinSize(1)` constraints, so all validation passes. The recursive `createApprover` call at [8](#0-7) 
descends without bound until the Node.js call stack is exhausted.
