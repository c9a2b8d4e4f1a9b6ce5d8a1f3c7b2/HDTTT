## Audit Report

## Title
Unvalidated Signature Submission Corrupts Transaction State and Forces Premature Execution Failure

## Summary
`SignersService.processTransactionSignatures` calls `sdkTransaction.addSignature(publicKey, map)` without first calling `validateSignature`. A registered organization member can submit a signature map containing their valid registered public key paired with arbitrary (non-zero, valid-hex) garbage bytes. The SDK appends those bytes, the key is added to `_signerPublicKeys`, the corrupted bytes are persisted, the transaction is promoted to `WAITING_FOR_EXECUTION`, and the chain service submits it to Hedera — where it is rejected with `INVALID_SIGNATURE` and permanently marked `FAILED`.

## Finding Description

**Root cause — `processTransactionSignatures` calls `addSignature` without prior cryptographic verification:**

In `signers.service.ts`, the inner loop iterates the submitted `SignatureMap`, verifies only that the public key belongs to the authenticated user, and then unconditionally calls `addSignature`: [1](#0-0) 

There is no call to `validateSignature` anywhere in `signers.service.ts`.

**Contrast with the validated path:**

`TransactionsService.importSignatures` calls `validateSignature` before `addSignature`: [2](#0-1) 

`validateSignature` calls `publicKey.verify(bodyBytes, signature)` and throws `'Invalid signature'` on failure: [3](#0-2) 

**DTO-level validation does not close the gap:**

The `IsSignatureMap` decorator only checks that the decoded signature bytes are non-empty: [4](#0-3) 

An attacker submitting valid-hex garbage bytes (e.g., 64 zero bytes) passes this check. The existing e2e test that expects `ISNMP` uses the string `'invalid-signature'`, which fails hex decoding — it does not test the case of valid-hex-but-cryptographically-invalid bytes. [5](#0-4) 

**State propagation after the invalid upload:**

`persistSignatureChanges` skips a transaction only when `isSameBytes && userKeys.length === 0`. Because `addSignature` mutated the bytes (appending the garbage signature), `isSameBytes` is `false`, so the corrupted bytes are written to the database and the user is inserted as a `TransactionSigner`: [6](#0-5) 

`processTransactionStatus` then checks `hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey)`. Because `_signerPublicKeys` now contains the attacker's key (added by `addSignature` without verification), the threshold check passes and the transaction is promoted to `WAITING_FOR_EXECUTION`: [7](#0-6) 

`hasValidSignatureKey` checks only key presence, not cryptographic validity: [8](#0-7) 

The chain service's `getValidatedSDKTransaction` performs the same key-presence-only check before submitting: [9](#0-8) 

The transaction is submitted to Hedera, rejected with `INVALID_SIGNATURE`, and permanently set to `FAILED`. The `validateTransactionStatus` switch in `execute.service.ts` confirms `FAILED` transactions cannot be retried: [10](#0-9) 

## Impact Explanation

A malicious registered organization member who holds a public key required by a transaction can:

1. Submit a signature map with their registered public key but with arbitrary valid-hex garbage bytes via `POST /transactions/:id/signers`.
2. Cause the transaction bytes stored in the database to be corrupted with the invalid signature.
3. Advance the transaction to `WAITING_FOR_EXECUTION` without providing a real signature.
4. Force the chain service to submit the transaction to the Hedera network, where it is rejected.
5. Permanently mark the transaction as `FAILED` — it cannot be retried.

This is an irreversible sabotage of any transaction the attacker is a required signer for. For high-value governance or system-file transactions (fee schedule updates, node operations), this constitutes a critical integrity failure.

## Likelihood Explanation

The attacker preconditions are minimal: the attacker must be a registered organization member with at least one public key registered in the system — a normal, non-privileged role. The endpoint `POST /transactions/:id/signers` is authenticated but not otherwise restricted. The attack requires a single crafted HTTP request with a valid public key DER string and arbitrary valid-hex bytes as the signature value. The `UploadSignatureMapDto` format is documented in the OpenAPI spec and the `IsSignatureMap` decorator only rejects empty or non-hex values. [11](#0-10) 

## Recommendation

Add a call to `validateSignature` inside `processTransactionSignatures`, immediately before `sdkTransaction.addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before line 251:
const { data: validKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMP);
```

This ensures that only cryptographically valid signatures are accepted on the signer-upload path, consistent with the `importSignatures` path. [12](#0-11) 

## Proof of Concept

1. Register as an organization member with public key `PK_ATTACKER` (raw hex).
2. Identify a transaction `TX_ID` for which `PK_ATTACKER` is a required signer.
3. Craft a `SignatureMap` with the correct node account ID, transaction ID, and `PK_ATTACKER` as the public key, but with 64 zero bytes (`0x0000...00`) as the signature value.
4. `POST /transactions/TX_ID/signers` with `{ signatureMap: <crafted map> }`.
5. The `IsSignatureMap` decorator accepts the non-empty hex bytes.
6. `processTransactionSignatures` calls `addSignature` without `validateSignature`; the garbage bytes are appended to the transaction.
7. `isSameBytes` is `false`; the corrupted bytes are written to the database and the user is inserted as a `TransactionSigner`.
8. If `PK_ATTACKER` was the last required key, `processTransactionStatus` promotes the transaction to `WAITING_FOR_EXECUTION`.
9. The chain service submits the transaction to Hedera; Hedera rejects it with `INVALID_SIGNATURE`.
10. The transaction is permanently set to `FAILED` and cannot be retried.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
