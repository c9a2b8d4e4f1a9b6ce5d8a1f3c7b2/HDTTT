Audit Report

## Title
Any Verified User Can Enumerate All Organization User Keys via Unguarded `GET /user-keys` Endpoint

## Summary
`UserKeysAllController` exposes `GET /user-keys` protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. The `AdminGuard` is absent. Any verified (non-admin) organization member can retrieve a paginated list of every `UserKey` record in the database — including the sensitive `mnemonicHash` and `index` fields belonging to other users.

## Finding Description

**Root cause — missing `AdminGuard` on `UserKeysAllController`:**

The controller-level guard chain is:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class UserKeysAllController { ... }
```

`AdminGuard` is never applied at the controller or handler level. [1](#0-0) 

The handler calls the service with no user filter:

```typescript
return this.userKeysService.getUserKeys(paginationParams);
``` [2](#0-1) 

The `getUserKeys` service method issues an unscoped `findAndCount` with no `where` clause and no `select` restriction — returning all columns for all users:

```typescript
const [items, total] = await this.repo.findAndCount({ take: limit, skip: offset });
``` [3](#0-2) 

**Sensitive fields are serialized and returned:**

`UserKeyDto` marks `mnemonicHash` and `index` with `@Expose()`, so the `SerializeInterceptor` (which uses `excludeExtraneousValues: true`) will include them in the response: [4](#0-3) [5](#0-4) 

**Contrast with the scoped sibling controller:**

`UserKeysController.getUserKeys` calls `getUserKeysRestricted(user, userId)`, which filters by `userId` and only exposes `mnemonicHash`/`index` when `user.id === userId`: [6](#0-5) [7](#0-6) 

**`AdminGuard` is already used for other privileged operations:** [8](#0-7) [9](#0-8) [10](#0-9) 

Its absence on `UserKeysAllController` is an omission, not a design choice.

## Impact Explanation
A non-admin attacker obtains:
- All `publicKey` values for every user — enabling full account mapping on the Hedera network.
- All `mnemonicHash` values — allowing correlation of keys derived from the same mnemonic, identifying key families and high-value signers.
- All `index` values — providing complete key derivation metadata for every registered key.
- All `userId` values — linking keys to specific user identities.

This is cross-user data exposure. The `getUserKeysRestricted` design explicitly treats `mnemonicHash` and `index` as owner-only fields; this endpoint bypasses that control entirely for all users simultaneously. [11](#0-10) 

## Likelihood Explanation
Likelihood is **high**:
- The attacker only needs a valid verified-user JWT — no admin credentials, no special access.
- The endpoint is a standard `GET` with no required parameters beyond pagination.
- Any organization member who has completed verification satisfies the only enforced guard (`VerifiedUserGuard`).
- The attack is read-only and passive, leaving no audit trail distinguishing it from a legitimate admin key lookup.

## Recommendation
Add `AdminGuard` to `UserKeysAllController`, consistent with how it is applied to other privileged operations:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard, AdminGuard)
export class UserKeysAllController { ... }
```

Additionally, consider whether `getUserKeys` should include `mnemonicHash` and `index` in its result set at all, or whether a separate admin-scoped DTO should omit those fields. [12](#0-11)
