Audit Report

## Title
Any Authenticated User Can DoS Another User's Transaction Creation by Pre-Registering the Same `transactionId`

## Summary
The `createTransactions` function in `back-end/apps/api/src/transactions/transactions.service.ts` enforces `transactionId` uniqueness via an application-level check, but never validates that the payer account embedded in the `transactionId` belongs to the submitting user. Any authenticated attacker can craft transaction bytes using a victim's payer account ID and a predicted `validStart`, sign them with their own registered key, and submit first — permanently blocking the victim from registering that `transactionId`. Additionally, the check-then-insert is not atomic and there is no database-level unique constraint on `transactionId`, enabling a TOCTOU race that can produce duplicate rows.

## Finding Description

**Root cause — no ownership check on `transactionId`'s payer account:**

In `validateAndPrepareTransaction`, the only identity check is that `transactionBytes` were signed by the caller's own registered `creatorKey`: [1](#0-0) 

The `transactionId` is then extracted directly from the parsed SDK transaction with no check that the embedded payer account belongs to the submitting user: [2](#0-1) 

The Hedera SDK allows setting any account ID as the payer when constructing transaction bytes locally. Network-level validation only occurs on submission to the Hedera network, not here.

**Duplicate check is application-level only, outside the DB transaction:**

The uniqueness guard is a plain `repo.find` query executed **before** the database transaction begins: [3](#0-2) 

The subsequent insert is wrapped in a **separate** `entityManager.transaction` block: [4](#0-3) 

**No DB-level unique constraint on `transactionId`:**

The `transaction` table schema in the initial migration defines `transactionId` as `character varying NOT NULL` with no `UNIQUE` constraint: [5](#0-4) 

No subsequent migration adds a unique constraint on `transactionId`. The `Transaction` entity class also has no `@Unique` decorator on the `transactionId` column: [6](#0-5) 

## Impact Explanation

1. **Targeted DoS on transaction creation:** A malicious registered user can block any other user from registering a transaction for any payer account ID they know. Since payer account IDs are observable on the Hedera network, this is not a secret. The victim must choose a new `validStart`; the attacker can repeat the attack for each new attempt.

2. **Database integrity violation (TOCTOU):** Two concurrent requests with the same `transactionId` can both pass the `repo.find` check and both succeed at insert (no DB unique constraint), creating duplicate `transactionId` rows. This corrupts database state and may cause the chain service to attempt double-execution of the same Hedera transaction.

## Likelihood Explanation

- **Attacker preconditions:** Must be a registered user (no admin privileges required). Payer account IDs are public on the Hedera network.
- **Attack complexity:** Low. The attacker constructs standard Hedera SDK transaction bytes with an arbitrary payer account, signs the DTO with their own registered key, and submits via the standard `POST /transactions` endpoint.
- **Repeatability:** The attack can be repeated for every new `validStart` the victim attempts, making it a sustained DoS. The TOCTOU race requires only concurrent requests and no prediction.

## Recommendation

1. **Add an ownership check:** After parsing the SDK transaction, verify that the payer account ID embedded in `sdkTransaction.transactionId` corresponds to a Hedera account owned/registered by the submitting user, or at minimum belongs to the same organization context.

2. **Add a DB-level unique constraint:** Add a `UNIQUE` constraint on `transactionId` (scoped to active statuses, or as a partial index excluding `CANCELED`/`REJECTED`/`ARCHIVED`) so that concurrent inserts are rejected at the database level regardless of application logic.

3. **Move the duplicate check inside the DB transaction:** Perform the `repo.find` uniqueness check inside the `entityManager.transaction` block with a `SELECT ... FOR UPDATE` or rely on the DB unique constraint to eliminate the TOCTOU window.

## Proof of Concept

```
1. Attacker (registered user) learns victim's Hedera payer account ID
   (e.g., 0.0.12345, observable on the Hedera network).

2. Attacker constructs Hedera SDK transaction bytes with:
     transactionId = 0.0.12345@<targetValidStart>
   using any valid node account ID.

3. Attacker signs dto.transactionBytes with their own registered key
   (passes the publicKey.verify check at line 910).

4. Attacker submits POST /transactions before the victim.
   → validateAndPrepareTransaction passes (no payer ownership check).
   → repo.find finds no existing record.
   → entityManager.save inserts the row with transactionId = "0.0.12345@<targetValidStart>".

5. Victim submits POST /transactions with the same transactionId.
   → repo.find finds the attacker's record (status = WAITING_FOR_SIGNATURES).
   → BadRequestException: "Transactions already exist: 0.0.12345@<targetValidStart>" is thrown.

6. For the TOCTOU variant: two concurrent attacker requests with the same
   transactionId both pass the repo.find check simultaneously, and both
   succeed at entityManager.save, producing two rows with identical transactionId.
```
