Audit Report

## Title
Old Parent Threshold Not Adjusted When Approver Is Re-Parented, Violating `threshold ≤ approvers.length` Invariant

## Summary
In `updateTransactionApprover`, the `dto.listId === null` path correctly adjusts the old parent's threshold after removing a child, but the `dto.listId = <number>` (re-parenting) path never fetches or adjusts the old parent at all. This leaves the old parent with `threshold > approvers.length`, a state that can never be satisfied, permanently blocking approval for any transaction whose approver tree contains that node.

## Finding Description
The function handles two sub-cases when `dto.listId` is provided:

**Path 1 — detach to root (`dto.listId === null`, lines 398–431):**
The old parent is fetched with its `approvers` relation, and its threshold is corrected downward if the remaining child count falls below it. [1](#0-0) 

**Path 2 — re-parent to new node (`dto.listId = <number>`, lines 434–466):**
Only the *new* parent is fetched and validated. The old parent (`approver.listId` before the update) is never fetched, and its threshold is never adjusted. After the `update` call at line 458, the old parent has lost one child but retains its original `threshold`. [2](#0-1) 

The old parent's id is available as `approver.listId` immediately before the update, but it is never used to perform the same threshold-correction logic that Path 1 applies.

## Impact Explanation
After the re-parenting call, the old parent node has `threshold = N` but only `N-1` children. Because the approval check requires `threshold` children to approve, the condition can never be satisfied regardless of how many users approve. Any transaction whose approver tree contains this node is permanently blocked from reaching the approved state — a permanent, irreversible denial-of-service on that transaction's execution.

## Likelihood Explanation
The only privilege required is being the **creator of a transaction**, a standard authenticated user role. The exploit is two sequential API calls:

1. Create a threshold approver tree where a parent has `threshold = 2` and exactly 2 children.
2. Call `PATCH /transactions/:transactionId/approvers/:childId` with `{ "listId": <other_parent_id> }` to move one child to a different parent.

After step 2, the old parent has `threshold = 2` and `1` child. The state is deterministic, requires no race condition, and is not reversible through any exposed API path.

## Recommendation
Before executing the `update` at line 458, fetch the old parent using `approver.listId` (the value before it is overwritten) with its `approvers` relation, then apply the same threshold-correction logic already present in Path 1:

```typescript
// Before the update at line 458
const oldParentId = approver.listId;
if (oldParentId !== null) {
  const oldParent = await transactionalEntityManager.findOne(TransactionApprover, {
    relations: ['approvers'],
    where: { id: oldParentId },
  });
  if (oldParent) {
    const remaining = oldParent.approvers.length - 1;
    if (remaining === 0) {
      await transactionalEntityManager.softRemove(TransactionApprover, oldParent);
    } else if (remaining < oldParent.threshold) {
      await transactionalEntityManager.update(TransactionApprover, oldParent.id, {
        threshold: remaining,
      });
    }
  }
}
```

This mirrors the correction already applied in Path 1 and restores the `threshold ≤ approvers.length` invariant for both code paths. [3](#0-2) 

## Proof of Concept

```
1. Authenticate as a normal user (transaction creator).

2. POST /transactions/:txId/approvers
   Body: { threshold: 2, approvers: [{ userId: A }, { userId: B }] }
   → Creates parent node P (threshold=2) with children C_A and C_B.

3. POST /transactions/:txId/approvers
   Body: { threshold: 1, approvers: [{ userId: C }] }
   → Creates a second parent node P2 (threshold=1) with child C_C.

4. PATCH /transactions/:txId/approvers/:C_A_id
   Body: { "listId": <P2.id> }
   → Moves C_A under P2. Old parent P now has threshold=2 but only 1 child (C_B).

5. Attempt to approve the transaction as both user A and user B.
   → Approval check on P requires 2 approvals but only 1 child exists.
   → The transaction can never reach the approved state.
```

The old parent `P` is left in the corrupted state `threshold=2, approvers.length=1` with no API path to correct it, as the threshold-update endpoint at line 467 enforces `dto.threshold ≤ approver.approvers.length` and would reject any attempt to lower the threshold to 1 after the child count is already 1 — wait, actually that check would *allow* setting threshold to 1 since `1 ≤ 1`. However, the creator may not know the internal node id of `P` or that the invariant was broken, and in any case the damage is done silently with no error returned to the caller. [4](#0-3)
