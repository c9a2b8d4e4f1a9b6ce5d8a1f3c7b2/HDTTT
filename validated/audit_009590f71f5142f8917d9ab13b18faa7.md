### Title
Unbounded In-Memory Aggregation in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign()` fetches every non-terminal transaction from the database with no `take` limit, iterates over the full result set performing per-item async cryptographic work, and only then slices the array for pagination. Any authenticated user who is a signer on a large number of transactions can trigger unbounded memory and CPU consumption, degrading or crashing the API service.

### Finding Description

`GET /transactions/sign` is handled by `getTransactionsToSign()`. The function accepts a `Pagination` object whose `size` field is capped at 100 by `PaginationParams`: [1](#0-0) 

However, the database query issued inside the function carries **no `take` constraint**: [2](#0-1) 

The entire result set is then iterated, calling `userKeysToSign()` for every row: [3](#0-2) 

Pagination is applied only after all work is complete, in application memory: [4](#0-3) 

This is the same structural defect described in the external report: the size/count cap is enforced **after** full materialization, not before. The `PaginationParams` guard gives a false sense of safety — it limits the response slice, not the upstream DB fetch or the per-item processing loop.

A secondary acknowledgment of the missing input cap exists in `importSignatures`: [5](#0-4) 

That path is partially mitigated by the 2 MB body limit: [6](#0-5) 

But `getTransactionsToSign` has no equivalent guard.

### Impact Explanation
An authenticated user who is a signer on N transactions (or who creates many transactions and adds themselves as a signer) can issue a single `GET /transactions/sign?page=1&size=1` request. The server will load all N transaction rows into memory, deserialize each one, and run key-matching logic before returning a single-item page. With a large N this exhausts heap memory and saturates the event loop, causing the Node.js process to become unresponsive or be killed by the OOM killer, denying service to all other users.

### Likelihood Explanation
The endpoint is authenticated, so an anonymous attacker cannot exploit it directly. However, any registered and verified user can be a signer on an arbitrary number of transactions — there is no cap on how many transactions a user may be associated with. A malicious insider or a compromised account can trivially reach a transaction count large enough to trigger the condition. The endpoint is also marked `/* NO LONGER USED BY FRONT-END */` but remains fully reachable via the REST API. [7](#0-6) 

### Recommendation
Push the pagination limit into the database query by passing `take: limit` and `skip: offset` directly to `repo.find()`, mirroring the pattern used in `getTransactions` and `getTransactionsToApprove`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // add this
  skip: offset,  // add this
});
```

Because `userKeysToSign` filtering happens in application code, the `totalItems` count will need a separate `repo.count()` call (or a two-pass approach) to remain accurate. Additionally, consider adding a hard upper bound on the `size` parameter specifically for this endpoint if the per-item work remains in application code.

### Proof of Concept

1. Register and verify a user account on the API.
2. Create or be added as a signer to a large number of transactions (e.g., 50 000).
3. Issue: `GET /transactions/sign?page=1&size=1` with a valid JWT.
4. Observe: the server fetches all 50 000 transaction rows, deserializes each, and runs key-matching logic before returning one item. Memory usage spikes proportionally to N; with large `transactionBytes` blobs the process OOMs before the response is sent. [8](#0-7)
