Audit Report

## Title
`TransactionsService#importSignatures` Omits `processTransactionStatus` Call and `TransactionSigner` Insertion After Signature Merge

## Summary
`importSignatures` in `back-end/apps/api/src/transactions/transactions.service.ts` merges externally-supplied signatures into `transactionBytes` and persists the raw bytes, but never calls `processTransactionStatus` to advance the transaction `status` field, and never inserts `TransactionSigner` rows. The analogous path `SignersService#uploadSignatureMaps` performs both operations. The omission causes a transaction whose signature threshold is satisfied via `importSignatures` to remain stuck in `WAITING_FOR_SIGNATURES` until a background cron re-evaluates it, and leaves the `TransactionSigner` ledger permanently inconsistent (the cron does not repair it).

## Finding Description

**Root cause — `importSignatures` (lines 493–626, `transactions.service.ts`):**

After merging signatures and serialising the updated SDK transaction, the function performs a bulk SQL `UPDATE` that writes only `transactionBytes`: [1](#0-0) 

It then emits a notification event and returns — no status update, no signer records: [2](#0-1) 

**Contrast with `SignersService#uploadSignatureMaps`:**

`uploadSignatureMaps` calls `persistSignatureChanges` (which invokes `bulkInsertSigners` to write `TransactionSigner` rows) and then `updateStatusesAndNotify` (which calls `processTransactionStatus` to advance the DB `status`): [3](#0-2) 

`processTransactionStatus` evaluates the collected signatures against the required key structure and bulk-updates `status` to `WAITING_FOR_EXECUTION` when the threshold is met: [4](#0-3) 

`bulkInsertSigners` inserts the corresponding `TransactionSigner` rows: [5](#0-4) 

Neither of these steps exists anywhere in `importSignatures`.

## Impact Explanation

1. **Delayed / missed execution.** The chain service only submits transactions in `WAITING_FOR_EXECUTION`. After a completing `importSignatures` call the transaction stays in `WAITING_FOR_SIGNATURES` until the next applicable cron window in `TransactionSchedulerService`. For transactions with `validStart` between 10 minutes and 1 hour away the cron fires every 10 minutes; between 1 hour and 1 day, every 30 minutes; beyond 1 day, once per day at midnight. [6](#0-5) 
If the 180-second Hedera validity window closes before the cron fires, the transaction expires and is permanently unexecutable.

2. **Permanent `TransactionSigner` corruption.** No cron job inserts missing `TransactionSigner` rows. Any feature that queries signers (audit trail, UI "who has signed", approver-threshold re-evaluation) will show incorrect data indefinitely.

3. **Status inconsistency visible to all participants.** Observers and approvers see `WAITING_FOR_SIGNATURES` on a fully-signed transaction, causing confusion and potential duplicate signing attempts.

## Likelihood Explanation

`importSignatures` is a documented, authenticated endpoint reachable by any organisation member. It is the intended path for importing offline/external signatures. Any user who imports a completing signature set triggers both omissions automatically — no special privileges or timing attack required.

## Recommendation

After the batch `transactionBytes` update succeeds, add the two missing steps inside `importSignatures`:

1. Re-fetch the updated `Transaction` entities and call `processTransactionStatus(repo, transactionSignatureService, updatedTransactions)` to advance `status` where the threshold is now met.
2. For each public key that contributed a new signature, insert a corresponding `TransactionSigner` row (mirroring the `bulkInsertSigners` logic in `SignersService`).

Consider refactoring `importSignatures` to delegate to `SignersService#uploadSignatureMaps` (or a shared internal helper) so that both code paths remain consistent by construction.

## Proof of Concept

1. Create a multi-key transaction requiring two signatures; have one key sign via the normal `uploadSignatureMaps` path.
2. Authenticate as a second user and call `POST /transactions/importSignatures` with a `signatureMap` containing the second key's signature, satisfying the threshold.
3. Immediately query the transaction: `status` is still `WAITING_FOR_SIGNATURES`; `TransactionSigner` table has no row for the second key.
4. Query again after the next applicable cron window fires: `status` advances to `WAITING_FOR_EXECUTION`, but `TransactionSigner` still has no row for the second key — the ledger inconsistency is permanent.
5. For a transaction whose `validStart` is, e.g., 45 minutes away, the cron fires every 10 minutes. If the import happens just after a cron tick, the transaction may not be submitted before the 180-second Hedera validity window closes.

```typescript
export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }

  if (updatesByStatus.size > 0) {
    await Promise.all(
      Array.from(updatesByStatus.values()).map(async ({ newStatus, oldStatus, ids }) => {
        const result = await transactionRepo
          .createQueryBuilder()
          .update(Transaction)
          .set({ status: newStatus })
          .where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
          .returning('id')
          .execute();

        for (const row of result.raw) {
          statusChanges.set(row.id, newStatus);
        }
      })
    );
  }

  return statusChanges;
```
