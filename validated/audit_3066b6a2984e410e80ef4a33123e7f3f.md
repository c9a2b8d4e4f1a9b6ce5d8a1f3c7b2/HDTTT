Audit Report

## Title
`computeSignatureKey` Skips Node Admin Key for Node ID 0 Due to Falsy Check

## Summary
In `transaction-signature.service.ts`, the guard `if (requirements.nodeId)` uses a JavaScript truthiness check on a `number | null` value. When `nodeId` is `0` (a valid Hedera consensus node ID), the condition evaluates to `false`, causing `addNodeKeys` to be silently skipped. The node admin key is never added to the required signature set for any transaction targeting node 0.

## Finding Description
`computeSignatureKey` in `TransactionSignatureService` extracts a `nodeId: number | null` from the transaction model and then conditionally calls `addNodeKeys`:

```typescript
if (requirements.nodeId) {
  await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
}
``` [1](#0-0) 

The `SignatureRequirements` interface explicitly types `nodeId` as `number | null`: [2](#0-1) 

Both `NodeUpdateTransactionModel.getNodeId()` and `NodeDeleteTransactionModel.getNodeId()` check `if (this.transaction.nodeId)` where `nodeId` is a `Long` object from `@hiero-ledger/sdk`. A `Long` instance representing 0 is still a JavaScript object and therefore truthy, so both methods correctly return the number `0` for node 0: [3](#0-2) [4](#0-3) 

Once `getNodeId()` returns the primitive `0`, the guard `if (requirements.nodeId)` evaluates `0` as falsy in JavaScript, and `addNodeKeys` is never invoked. The correct guard is `if (requirements.nodeId !== null)`.

## Impact Explanation
The computed `signatureKey` is consumed in two critical paths:

**1. Execution gate** — `getValidatedSDKTransaction` calls `computeSignatureKey` and then checks `hasValidSignatureKey`. With the admin key absent from the `KeyList`, the check passes even when the node 0 admin key has not signed, allowing the transaction to be submitted to the network without the required authorization: [5](#0-4) 

**2. Status promotion** — `processTransactionStatus` uses the same service to determine whether a transaction has sufficient signatures to advance to `WAITING_FOR_EXECUTION`. With the admin key missing from the computed `KeyList`, a `NodeUpdateTransaction` or `NodeDeleteTransaction` targeting node 0 can reach `WAITING_FOR_EXECUTION` without the node admin key ever having signed: [6](#0-5) 

**3. Signing workflow** — `userKeysRequiredToSign` uses the same service to determine which user keys must sign. Users holding the node 0 admin key are never prompted, so the transaction can be executed without that key's signature. [7](#0-6) 

## Likelihood Explanation
Hedera consensus node IDs are zero-indexed; node 0 exists on every network (mainnet, testnet, previewnet). Any operator who creates a `NodeUpdateTransaction` or `NodeDeleteTransaction` for node 0 through this tool will trigger the bug. No special attacker capability is required — a normal authenticated user with creator-key access can reach this path.

## Recommendation
Replace the falsy guard with an explicit `null` check in `computeSignatureKey`:

```typescript
// Before (buggy):
if (requirements.nodeId) {

// After (correct):
if (requirements.nodeId !== null) {
``` [1](#0-0) 

Additionally, add a test case for `nodeId === 0` to the existing test suite in `transaction-signature.service.spec.ts` to prevent regression.

## Proof of Concept

1. Authenticate as a non-council user with creator access.
2. Create a `NodeUpdateTransaction` or `NodeDeleteTransaction` targeting node ID `0`.
3. Sign the transaction only with the fee payer key.
4. Observe that `processTransactionStatus` promotes the transaction to `WAITING_FOR_EXECUTION` — the node 0 admin key was never required.
5. Trigger execution via `getValidatedSDKTransaction`; `hasValidSignatureKey` passes because the admin key was never added to the `KeyList`.
6. The transaction is submitted to the network without the node 0 admin key's authorization.
