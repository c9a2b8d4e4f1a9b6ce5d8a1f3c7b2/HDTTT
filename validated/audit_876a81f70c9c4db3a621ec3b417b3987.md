### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every row with an async `userKeysToSign` call per transaction. Because the query is not scoped to the requesting user at the database level, the cost of every call to `GET /transactions/sign` grows linearly with the total number of active transactions in the system. An authenticated attacker can deliberately inflate the transaction table by bulk-creating valid transactions, causing the endpoint to time out or exhaust server memory for all users.

### Finding Description

**Root cause — unbounded `repo.find()` with no `take` limit:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // filters only by status — no user scope
  relations: ['groupItem'],
  order,
});

for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

The `whereForUser` predicate filters only by `status NOT IN (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED)`: [2](#0-1) 

There is no `take:` / `skip:` applied to this query, and no filter that restricts rows to the requesting user. The result is a full-table scan of every active transaction in the system, followed by one async `userKeysToSign` call per row. User-level filtering is done entirely in memory after the fact.

**Contrast with paginated endpoints** — every other listing endpoint in the same service correctly passes `take: limit` and `skip: offset` from the `Pagination` decorator, which enforces a hard cap of 100 rows: [3](#0-2) 

`getTransactionsToSign` is the only query that bypasses this guard.

**Attacker-controlled entry point — `POST /transactions` with no rate limit:** [4](#0-3) 

Any authenticated user can call this endpoint repeatedly. Transaction bytes can be crafted locally (no Hedera network interaction required for construction); only a valid creator key belonging to the caller is needed. Each successful call inserts a row that persists in the `WAITING_FOR_SIGNATURES` state until it expires (up to the Hedera valid-start window), meaning the attacker can maintain a large pool of active rows.

**Secondary unbounded path — `POST /transactions/signatures/import`:** [5](#0-4) 

The developer comment `//Added a batch mechanism, probably should limit this on the api side of things` confirms awareness that the array accepted by `importSignatures` has no size cap. A single request with thousands of entries triggers per-entry SDK deserialization and signature validation. [6](#0-5) 

### Impact Explanation

- **Availability:** As the attacker inflates the transaction table, every call to `GET /transactions/sign` by any user triggers an increasingly expensive full-table scan plus N async DB/crypto operations. At sufficient scale the request times out (Node.js default or reverse-proxy timeout), making the signing workflow unavailable for all organization members.
- **Server memory:** The entire result set is materialized in memory before pagination is applied (lines 311–313). A large table causes heap exhaustion.
- **Cascading effect:** The `getTransactionsToSign` result is used by the front-end to drive the signing UI; a broken endpoint blocks the entire multi-signature workflow.

### Likelihood Explanation

- **Preconditions:** A valid organization account (normal user, no admin privileges required).
- **Effort:** Scripted HTTP calls to `POST /transactions` with locally-crafted transaction bytes. No Hedera mainnet/testnet interaction is required for construction; the SDK can produce valid frozen bytes offline.
- **No rate limiting** is present on the transaction creation endpoint.
- **Persistence:** Rows remain active until their Hedera valid-start window expires, so the attacker can maintain pressure with a modest sustained request rate.

### Recommendation

1. **Immediate:** Add a user-scoped filter at the database level in `getTransactionsToSign` so the query only fetches transactions where the user is a signer/creator/observer, and apply `take: limit` / `skip: offset` consistent with every other listing endpoint.
2. **Short-term:** Add `@ArrayMaxSize(N)` to `CreateTransactionGroupDto.groupItems` and enforce a hard cap (e.g., 50–100) on the `importSignatures` / `uploadSignatureMap` array inputs.
3. **Long-term:** Introduce per-user rate limiting on `POST /transactions` and consider an index-backed approach for `getTransactionsToSign` that avoids loading all rows into memory.
