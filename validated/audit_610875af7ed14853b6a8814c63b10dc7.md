### Title
Transaction Creator Can Manipulate Approver Structure After Approvals Have Started, Permanently Blocking Execution

### Summary
The `ApproversService` allows the transaction creator to add, update, or remove approvers at any point in the transaction lifecycle without checking the transaction's current status. A malicious creator can exploit this to replace already-approved approvers with new users (clearing their signatures), remove approved approvers, or restructure the threshold tree after approvals have been collected — indefinitely preventing the approval threshold from being met and blocking transaction execution.

### Finding Description

**Vulnerability class:** State transition / missing status guard on creator-controlled parameter mutation.

**Root cause:** Three operations in `approvers.service.ts` and `approvers.controller.ts` gate access solely on "caller is the creator" (`getCreatorsTransaction`) but perform **no check on the transaction's current status**:

1. `createTransactionApprovers` — adds new approvers
2. `updateTransactionApprover` — replaces an approver's `userId` (clearing their signature/approval) or changes the threshold
3. `removeTransactionApprover` (via controller) — deletes an approver node [1](#0-0) 

`createTransactionApprovers` calls `getCreatorsTransaction` (creator-only check) and then proceeds with no status guard: [2](#0-1) 

`getCreatorsTransaction` only verifies `creatorKey.userId === user.id` — no status check. [3](#0-2) 

`updateTransactionApprover` similarly calls `getCreatorsTransaction` at line 394 with no status guard.

The most destructive path is the `userId` update branch, which explicitly clears the existing approval: [4](#0-3) 

When `dto.userId` is set to a different user, `signature`, `userKeyId`, and `approved` are all set to `undefined`, erasing the previously collected approval. No status check prevents this from happening while the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.

The `removeTransactionApprover` endpoint in the controller also only checks creator identity: [5](#0-4) 

### Impact Explanation

A malicious creator can permanently prevent a transaction from reaching the approval threshold:

- **Replace approved approvers:** Call `PATCH /transactions/:id/approvers/:approverId` with `{ userId: <new_user_id> }` after an approver has signed. This clears the existing signature and requires the new user to approve instead. The creator can repeat this for every approver who approves, making the threshold unreachable.
- **Remove approved approvers:** Call `DELETE /transactions/:id/approvers/:approverId` to remove an approver who has already approved, reducing the effective approval count.
- **Raise the threshold:** Add new approvers to a threshold tree (increasing the pool), then call `PATCH` with `{ threshold: N }` to raise the required count beyond what the existing approvers can satisfy.

The result is a permanently blocked transaction — it can never reach `WAITING_FOR_EXECUTION` and will eventually expire. Approvers waste effort and the transaction's intended on-chain action never executes.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond having created the transaction. Any user of the organization mode can create a transaction and become its creator. The attack requires only valid API calls through documented endpoints. The scenario is realistic in any multi-party workflow where the creator has a conflict of interest with the approvers (e.g., a creator who changes their mind after broadcasting, or a malicious insider).

**Likelihood: Medium. Impact: High. Overall: Medium.**

### Recommendation

Add a transaction status guard at the start of `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover`. Once a transaction has left `NEW`/`DRAFT` status (i.e., it is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`), the approver structure should be frozen:

```typescript
// In getCreatorsTransaction or at the top of each mutating method:
const MUTABLE_STATUSES = [TransactionStatus.NEW]; // or whatever draft status applies
if (!MUTABLE_STATUSES.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify approvers after transaction has been broadcasted');
}
```

Alternatively, allow only additive changes (adding new approvers) but never allow replacing or removing approvers who have already submitted approvals.

### Proof of Concept

1. Creator (User A) creates a transaction and sets approver structure: `{ threshold: 2, approvers: [UserB, UserC, UserD] }`.
2. Transaction is broadcasted → status becomes `WAITING_FOR_SIGNATURES`.
3. UserB approves → `TransactionApprover` row for UserB now has `signature` and `approved = true`.
4. Creator (User A) calls:
   ```
   PATCH /transactions/1/approvers/<UserB_approver_id>
   Authorization: Bearer <creator_token>
   { "userId": 999 }   // any valid user who will not approve
   ```
5. Server executes lines 500–513 of `approvers.service.ts`: UserB's `signature`, `userKeyId`, and `approved` are cleared; the approver row now points to User 999.
6. UserC approves. Creator repeats step 4 for UserC's approver row.
7. The threshold of 2 is never reached. The transaction expires without executing.
8. Creator can repeat this loop for every approval that comes in, with no rate limit or status guard stopping them. [6](#0-5) [7](#0-6)

```typescript
  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
