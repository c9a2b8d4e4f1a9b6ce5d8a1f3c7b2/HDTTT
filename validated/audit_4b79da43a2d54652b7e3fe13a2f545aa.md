Audit Report

## Title
Null Dereference on `getFeePayerAccountId()` in `extractSignatureRequirements` Causes Unhandled Crash in Signature Computation Pipeline

## Summary
`TransactionSignatureService.extractSignatureRequirements()` unconditionally calls `.toString()` on the return value of `getFeePayerAccountId()`, which is typed `AccountId | null`. When an SDK transaction with no `transactionId` is processed, this produces a `TypeError: Cannot read properties of null (reading 'toString')`, crashing the signature computation pipeline across three critical backend execution paths.

## Finding Description

`getFeePayerAccountId()` in `TransactionBaseModel` is explicitly typed to return `AccountId | null` and uses optional chaining, returning `null` when `transactionId` or `transactionId.accountId` is absent: [1](#0-0) 

The unit tests for this model explicitly assert that `null` is returned when `transactionId` is undefined: [2](#0-1) 

The caller `extractSignatureRequirements()` calls `.toString()` on this return value with no null guard at line 71: [3](#0-2) 

This is invoked from `computeSignatureKey()` at line 46: [4](#0-3) 

`computeSignatureKey` is called from three critical backend paths:

**1. Transaction execution validation** — `getValidatedSDKTransaction` in `execute.service.ts` line 216: [5](#0-4) 

**2. Scheduled collation/execution** — `collateAndExecute` in `transaction-scheduler.service.ts` line 280, with the `TypeError` swallowed by a generic `catch` at line 319: [6](#0-5) 

**3. Batch status processing** — `processTransactionStatus` in `utils/transaction/index.ts` line 133: [7](#0-6) 

## Impact Explanation

- In `collateAndExecute`, the `TypeError` is swallowed by the generic `catch` block (line 319 — `console.log(error)`), silently preventing the transaction from ever being executed. This is a permanent, unrecoverable silent DoS for that transaction with no status update or error surfaced to the user.
- In `getValidatedSDKTransaction`, the error propagates, blocking execution of any transaction whose bytes trigger this path.
- In `processTransactionStatus`, the error propagates mid-loop, halting status updates for all remaining transactions in the current batch, corrupting the signing/execution pipeline for the entire organization's pending transactions.

## Likelihood Explanation

Any authenticated user who can submit a transaction to the API controls the `transactionBytes` field. The Hedera protobuf format permits a `TransactionBody` without a `transactionId` field; `SDKTransaction.fromBytes()` can succeed on such bytes, producing an SDK transaction where `transactionId` is `undefined`. The optional chaining in `getFeePayerAccountId()` — and the explicit unit tests asserting `null` is returned — confirm the developers anticipated this null case, but the guard was never added at the call site in `extractSignatureRequirements()`. No admin privileges are required; crafting protobuf bytes that omit the `transactionId` field is straightforward using any protobuf library.

## Recommendation

Add a null guard in `extractSignatureRequirements()` before calling `.toString()`. For example:

```typescript
private extractSignatureRequirements(
  transactionModel: TransactionBaseModel<any>
): SignatureRequirements {
  const feePayerAccountId = transactionModel.getFeePayerAccountId();
  if (!feePayerAccountId) {
    throw new Error('Transaction is missing a transactionId / fee payer account.');
  }
  return {
    feePayerAccount: feePayerAccountId.toString(),
    signingAccounts: transactionModel.getSigningAccounts(),
    receiverAccounts: transactionModel.getReceiverAccounts(),
    newKeys: transactionModel.getNewKeys() ?? [],
    nodeId: transactionModel.getNodeId(),
  };
}
```

This surfaces the error explicitly and early, rather than crashing with a `TypeError` deep in the pipeline or being silently swallowed.

## Proof of Concept

1. Using any protobuf library, craft a `SignedTransaction` wrapping a `TransactionBody` that omits the `transactionId` field.
2. Submit this as `transactionBytes` when creating a transaction via the API (authenticated user, no special privileges).
3. When the scheduler calls `collateAndExecute` → `computeSignatureKey` → `extractSignatureRequirements`, `getFeePayerAccountId()` returns `null`.
4. `null.toString()` throws `TypeError: Cannot read properties of null (reading 'toString')`.
5. In `collateAndExecute`, the error is caught and logged only (`console.log(error)` at line 319), the transaction is permanently stuck — no status update, no retry, no user notification.
6. In `processTransactionStatus`, the same throw aborts the loop mid-batch, leaving all subsequent transactions in the batch unprocessed.

export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }
