### Title
Unbounded Full-Table Scan with Per-Row Async DB Calls in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in `TransactionsService` fetches every active transaction in the database with no row limit, then serially calls `userKeysToSign` (which deserializes transaction bytes and issues additional DB queries) for each row before applying pagination. Any authenticated user can trigger this endpoint; as the number of active transactions grows, the request consumes unbounded memory and CPU, degrading or hanging the API service for all users.

### Finding Description

**Root cause — no `take` limit on the initial query:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–299, the repository `find` call carries no `take` constraint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

**Unbounded serial async loop:**

Every row returned is then processed one-by-one in a `for…of` loop that `await`s `userKeysToSign` per iteration:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Each `userKeysToSign` call delegates to `userKeysRequiredToSign`, which:
1. Calls `attachKeys(user, entityManager)` — a DB query.
2. Deserializes raw transaction bytes via `SDKTransaction.fromBytes`.
3. Calls `transactionSignatureService.computeSignatureKey` — potentially a mirror-node network call.
4. Issues a `UserKey` lookup against the DB. [3](#0-2) 

**Pagination applied only after the full loop:**

The `slice` that enforces the page limit runs only after all N iterations complete:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [4](#0-3) 

So requesting page 1 with `limit=10` still forces the server to process every active transaction in the system.

### Impact Explanation

- **Memory exhaustion**: All active `Transaction` rows (with `groupItem` relations) are loaded into the Node.js heap simultaneously. In an organization with thousands of pending transactions, this can exhaust heap memory and crash the process.
- **CPU / event-loop starvation**: N sequential `await` calls (deserialization + DB + optional mirror-node I/O) block the event loop for the duration of the request. A single request with 10 000 active transactions can hold the server busy for tens of seconds, making the API unresponsive to all other users.
- **Cascading timeout**: NestJS/HTTP default timeouts will eventually kill the connection, but the server-side work continues, compounding resource pressure.

### Likelihood Explanation

- **Attacker precondition**: A valid JWT (any verified user). No admin role required.
- **Trigger**: A single `GET` request to the transactions-to-sign endpoint.
- **Realistic scale**: An organization running automated workflows can accumulate thousands of active transactions. The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`), so every in-flight transaction is included.
- **Amplification**: The attacker can send the request repeatedly; each request re-runs the full scan.

### Recommendation

Apply the database-level `take` limit before the loop, and push the filtering logic into the query rather than post-processing in application memory:

```typescript
// Apply take at the DB level, not after the loop
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // ← enforce page size at query time
  skip: offset,
});
```

For the per-row async work, replace the serial `for…of` with `Promise.all` (with a concurrency limiter such as `p-limit`) so that I/O is parallelised rather than serialised. Alternatively, push the "keys required to sign" logic into a single SQL/mirror-node batch call so the N+1 query pattern is eliminated entirely.

### Proof of Concept

**Preconditions**: Authenticated user with a valid JWT; organization has ≥ 1 000 active (non-terminal) transactions.

**Steps**:
1. Register/log in as any verified user.
2. Send `GET /transactions/sign` (the endpoint backed by `getTransactionsToSign`).
3. Observe: the server loads all active transactions into memory, then serially awaits `userKeysToSign` for each row.
4. With 5 000 active transactions and a mirror-node round-trip of ~50 ms per call, the single request occupies the server for ~250 seconds, rendering the API unresponsive for all concurrent users.
5. Repeating the request at a low rate (e.g., one request every 30 s) sustains the degradation indefinitely.

**Expected outcome**: API response times for all endpoints spike; under sufficient load the Node.js process OOMs or the HTTP gateway returns 504 errors to legitimate users.

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
};
