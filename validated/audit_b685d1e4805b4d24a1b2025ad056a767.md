### Title
Transaction Observers Can Be Added for Unverified (`UserStatus.NEW`) Users Without Validation

### Summary
The `createTransactionObservers` function in `observers.service.ts` accepts an array of arbitrary `userIds` from the request body and creates `TransactionObserver` records for each one without verifying that the target users are verified (`UserStatus.NONE`) members of the system. This is the direct analog to the external report: just as `addCollateral()` allows collateral to be assigned to non-whitelisted addresses, `createTransactionObservers()` allows observer roles to be assigned to non-functional (unverified) user accounts. The e2e test suite explicitly confirms this behavior succeeds with HTTP 201.

### Finding Description

**Root cause:** `ObserversService.createTransactionObservers` iterates over the caller-supplied `dto.userIds` array and creates a `TransactionObserver` row for each ID that is not already present, with no check that the target user exists in a verified state.

```
// back-end/apps/api/src/transactions/observers/observers.service.ts
for (const userId of dto.userIds) {
  if (!transaction.observers.some(o => o.userId === userId)) {
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);
  }
}
const result = await this.repo.save(observers);
``` [1](#0-0) 

The only guards on the endpoint are `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. `VerifiedUserGuard` checks only that the **caller** has `UserStatus.NONE`; it performs no check on the **target** user IDs supplied in the body. [2](#0-1) 

The `User` entity defines three statuses: `NEW` (registered by admin, account not yet set up), `RESET_PASSWORD`, and `NONE` (fully verified). [3](#0-2) 

The e2e test suite explicitly confirms that a `UserStatus.NEW` user (`userNew`) is successfully added as an observer (HTTP 201): [4](#0-3) 

**Exploit path:**
1. Attacker is a verified (`UserStatus.NONE`) user who has created a transaction.
2. Admin registers a new user via the sign-up flow; that user has `UserStatus.NEW` and cannot authenticate.
3. Attacker calls `POST /transactions/:id/observers` with `{ "userIds": [<newUserId>] }`.
4. The server creates a `TransactionObserver` row for the `NEW` user with `role: Role.FULL` — no error.
5. When the `NEW` user later completes registration and becomes `UserStatus.NONE`, they silently inherit `FULL` observer access to the transaction without ever being explicitly granted it at that point in time.

### Impact Explanation

- **Unintended access grant on registration completion:** A `NEW` user added as an observer gains `FULL` visibility into the transaction (all signers, approvers, status, bytes) the moment they complete account setup — without any deliberate grant at that time. This breaks the expected access control model where access is granted only to active, verified participants.
- **Notification delivery to non-functional accounts:** The notification pipeline emits `emitTransactionUpdate` events for the transaction; notification receivers tied to the `NEW` user's ID will be created and email/in-app alerts dispatched to an account that cannot yet log in.
- **Observer slot squatting:** Because the `(userId, transactionId)` pair has a unique index, a phantom observer record for a `NEW` userId permanently occupies that slot. If the `NEW` user is later deleted and their ID is reused (unlikely but possible in some DB configurations), a different real user could inherit the observer record. [5](#0-4) 

### Likelihood Explanation

- **Attacker preconditions:** Must be a verified user who is the creator of at least one transaction. This is a normal, reachable role — any registered and verified user can create transactions.
- **Knowledge required:** The attacker needs to know the numeric `userId` of a `NEW` user. User IDs are sequential integers; they are returned in API responses (e.g., the sign-up endpoint returns `{ id, email, createdAt }`), making enumeration trivial for an admin or any user who has seen another user's profile.
- **No privileged access required:** The `VerifiedUserGuard` is the only barrier, and the attacker already satisfies it as the transaction creator.

### Recommendation

In `createTransactionObservers`, before creating an observer record for each `userId`, verify that the target user exists and has `UserStatus.NONE`:

```typescript
const targetUser = await this.entityManager.findOne(User, {
  where: { id: userId, status: UserStatus.NONE },
});
if (!targetUser) {
  throw new BadRequestException(`User ${userId} is not a verified member`);
}
```

This mirrors the pattern already enforced by `VerifiedUserGuard` for the calling user, and is the direct analog to the external report's recommendation to "check that the receiver address is a whitelisted trader."

### Proof of Concept

```
# Step 1 – Admin registers a new user (status = NEW)
POST /auth/users
Authorization: Bearer <adminToken>
{ "email": "newuser@example.com" }
→ 201 { "id": 42, "email": "newuser@example.com", ... }

# Step 2 – Verified attacker creates a transaction
POST /transactions
Authorization: Bearer <attackerToken>
{ ... }
→ 201 { "id": 7, ... }

# Step 3 – Attacker adds the NEW user (id=42) as a FULL observer
POST /transactions/7/observers
Authorization: Bearer <attackerToken>
{ "userIds": [42] }
→ 201 [{ "id": 99, "transactionId": 7, "userId": 42, "role": "FULL", ... }]

# Step 4 – newuser@example.com completes registration (status → NONE)
# From this point, user 42 has FULL observer access to transaction 7
# with no further action required by the attacker.
```

The e2e test at `back-end/apps/api/test/spec/transaction-observers.e2e-spec.ts` lines 122–143 already reproduces step 3 with `userNew` (a `UserStatus.NEW` account) and asserts HTTP 201, confirming the absence of any recipient validation. [6](#0-5) [7](#0-6)

  async createTransactionObservers(
    user: User,
    transactionId: number,
    dto: CreateTransactionObserversDto,
  ): Promise<TransactionObserver[]> {
    const transaction = await this.entityManager.findOne(Transaction, {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user', 'observers'],
    });

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to delete it');

    const observers: TransactionObserver[] = [];

    for (const userId of dto.userIds) {
      if (!transaction.observers.some(o => o.userId === userId)) {
        const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
        observers.push(observer);
      }
    }

    if (observers.length === 0) {
      return [];
    }

    try {
      const result = await this.repo.save(observers);

      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);

      return result;
    } catch (error) {
      throw new BadRequestException(error.message);
    }
