Audit Report

## Title
Transaction Creator Can Manipulate Approver List at Any Time to Bypass Multi-Party Approval Requirements

## Summary
`createTransactionApprovers` and `updateTransactionApprover` in `approvers.service.ts` delegate authorization exclusively to `getCreatorsTransaction`, which verifies only that the caller is the transaction creator. No check on the transaction's lifecycle status is performed. A malicious creator can replace a pending approver slot with themselves after other approvers have already signed, then self-approve to satisfy the threshold and advance the transaction without genuine multi-party consent.

## Finding Description

`getCreatorsTransaction` is the sole authorization gate for both mutation endpoints. It fetches the transaction and checks `creatorKey.userId === user.id`, but returns the transaction regardless of its current `status`: [1](#0-0) 

`createTransactionApprovers` calls this gate and proceeds with no status guard: [2](#0-1) 

`updateTransactionApprover` calls the same gate with no status guard: [3](#0-2) 

When a `userId` update is applied, only that slot's `signature`, `userKeyId`, and `approved` fields are cleared — other already-approved slots remain intact: [4](#0-3) 

Neither `createTransactionApprovers` nor `updateTransactionApprover` prevents the creator from designating themselves as an approver. `approveTransaction` only checks that `user.id` appears in the approver list — no check that the approver is not the creator: [5](#0-4) 

The controller layer adds only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) — no status guard: [6](#0-5) 

## Impact Explanation

The approval system is the organizational governance control ensuring multiple designated parties consent before a transaction is submitted to the Hedera network. By replacing a pending approver slot with themselves after other approvers have already signed, the creator can satisfy any threshold requirement unilaterally. The transaction advances to `WAITING_FOR_EXECUTION` and is submitted to Hedera without the genuine consent of all required approvers. For Hedera Council use cases this could mean unauthorized fund transfers, account updates, or other privileged network operations.

## Likelihood Explanation

The attacker is any authenticated user who creates a transaction — no admin or privileged role is required. The API endpoints (`PATCH /transactions/:transactionId/approvers/:id` and `POST /transactions/:transactionId/approvers`) are standard authenticated REST calls. The attack requires only that the creator wait for at least one legitimate approver to sign, then replace the remaining unsigned slot with themselves. The steps are deterministic and require no race condition or special timing.

## Recommendation

1. **Status guard in `getCreatorsTransaction` (or at call sites):** Reject modification of approvers when the transaction status is `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any non-`NEW` status. Only allow approver list changes while the transaction is in `NEW` (draft) state.

2. **Prevent creator self-designation as approver:** In both `createTransactionApprovers` and `updateTransactionApprover`, add a check that `dtoApprover.userId !== creatorUserId` (obtained from the transaction's `creatorKey.userId`).

3. **Prevent creator from calling `approveTransaction`:** In `approveTransaction`, add a check that `user.id !== transaction.creatorKey.userId`.

## Proof of Concept

```
# Setup: Creator (userId=1) creates a transaction with approvers Alice (userId=2) and Bob (userId=3), threshold=2
POST /transactions/:txId/approvers
Body: { approversArray: [{ threshold: 2, approvers: [{ userId: 2 }, { userId: 3 }] }] }

# Step 1: Alice approves legitimately
POST /transactions/:txId/approvers/approve  (as Alice)
# Alice's approver slot now has signature set

# Step 2: Creator replaces Bob's slot with themselves (userId=1)
# Transaction is now in WAITING_FOR_SIGNATURES — no status check blocks this
PATCH /transactions/:txId/approvers/:bobApproverId  (as Creator)
Body: { userId: 1 }
# Bob's signature/userKeyId/approved are cleared; Alice's slot is untouched

# Step 3: Creator self-approves
POST /transactions/:txId/approvers/approve  (as Creator)
# approveTransaction finds userId=1 in approver list — no creator check
# Creator's signature is recorded

# Result: threshold=2 is now satisfied (Alice + Creator)
# Transaction advances to WAITING_FOR_EXECUTION and is submitted to Hedera
# without Bob's (or any second legitimate party's) consent
```

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
