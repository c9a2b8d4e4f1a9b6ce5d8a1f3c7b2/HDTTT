### Title
`archiveTransaction` Bypasses Status Guard for Manual Transactions, Allowing State Corruption of Terminal Transactions

### Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` uses a flawed boolean condition that completely skips the status validity check whenever `transaction.isManual` is `true`. This allows any authenticated creator of a manual transaction to archive it regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, and `REJECTED` — by calling the API endpoint directly, bypassing the frontend guard.

### Finding Description

**Vulnerability class:** State transition / access control — a status guard is short-circuited by an unrelated flag.

**Root cause:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `archiveTransaction` contains:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard throws only when **both** conditions are true: status is outside the allowed set **AND** the transaction is not manual. For any `isManual = true` transaction, `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false` — the guard never fires, and the archive proceeds unconditionally.

**Exploit path:**

1. Attacker authenticates as a normal user (no privilege required beyond being the creator of a manual transaction).
2. Attacker creates a manual transaction (`isManual: true`) via `POST /transactions`.
3. The transaction is executed by the chain service; its status becomes `EXECUTED` (or `FAILED`, `CANCELED`, etc.).
4. Attacker calls `PATCH /transactions/archive/:id` directly (bypassing the frontend).
5. The backend calls `archiveTransaction`, the status guard is skipped because `isManual = true`, and `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes unconditionally. [2](#0-1) 

**Frontend guard does not protect the backend:**

The frontend correctly restricts the archive button to in-progress transactions:

```typescript
const canArchive = computed(() => {
  const isManual = props.organizationTransaction?.isManual;
  return isManual && isCreator.value && transactionIsInProgress.value;
});
``` [3](#0-2) 

But this is a client-side check only. The REST endpoint `PATCH /transactions/archive/:id` is reachable directly. [4](#0-3) 

**Test gap confirms the bug:**

The existing unit test for `archiveTransaction` only covers the non-manual case (`isManual` is absent/falsy), so the guard fires and the test passes. No test covers `isManual = true` with a terminal status: [5](#0-4) 

### Impact Explanation

A transaction creator can overwrite the status of any of their manual transactions — including `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, or `REJECTED` — to `ARCHIVED`. This:

- **Corrupts the audit trail**: An `EXECUTED` transaction (with an on-chain record) can be relabeled `ARCHIVED`, hiding its true outcome from other organization members and auditors.
- **Conceals failures or cancellations**: A `FAILED` or `CANCELED` transaction can be archived, masking the failure from the organization's history view.
- **Breaks state machine invariants**: Terminal states are supposed to be immutable; this allows a user-controlled state regression.

### Likelihood Explanation

The attacker only needs to be an authenticated organization user who created at least one manual transaction — a normal, unprivileged role. The exploit requires a single direct HTTP `PATCH` request to a documented endpoint. No special tooling, timing, or race condition is needed.

### Recommendation

Remove the `isManual` short-circuit from the status guard. The status check must be enforced independently of `isManual`. If the intent is to allow manual transactions to be archived from `WAITING_FOR_EXECUTION` (in addition to `WAITING_FOR_SIGNATURES`), the current allowed-status list already covers that. The fix:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    )
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  // Optionally, additionally require isManual for WAITING_FOR_EXECUTION
  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
```

Add a unit test covering `isManual = true` with each terminal status to prevent regression.

### Proof of Concept

**Preconditions:** Attacker is an authenticated organization user with a valid JWT. They have previously created a manual transaction (id = `42`) that has since been executed (status = `EXECUTED`).

**Steps:**

```http
PATCH /transactions/archive/42
Authorization: Bearer <attacker_jwt>
```

**Expected (correct) behavior:** `400 Bad Request` with `ErrorCodes.OMTIP` — transaction is not in an archiveable state.

**Actual behavior:** `200 OK` — the transaction's status is updated from `EXECUTED` to `ARCHIVED` in the database, corrupting the organization's transaction history.

**Verification:** Query the database or call `GET /transactions/42` — the status field now reads `ARCHIVED` instead of `EXECUTED`.
