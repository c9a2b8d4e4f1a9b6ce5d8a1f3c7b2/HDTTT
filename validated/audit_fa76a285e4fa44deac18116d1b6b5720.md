Audit Report

## Title
Silent Key-Fetch Failure in `computeSignatureKey` Causes Empty `KeyList`, Allowing Unsigned Transactions to Advance to `WAITING_FOR_EXECUTION`

## Summary
`TransactionSignatureService.computeSignatureKey` silently swallows all mirror-node lookup errors, returning an empty `KeyList` when account-info fetches fail. The downstream `hasValidSignatureKey` function treats an empty `KeyList` as fully satisfied (`0 >= 0 = true`). As a result, any transaction can be promoted from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` whenever the mirror node is transiently unavailable, bypassing the multi-signature threshold requirement entirely.

## Finding Description

**Root cause 1 — Silent error swallowing in `computeSignatureKey`**

`addFeePayerKey` catches all exceptions and returns without adding any key to the accumulator `KeyList`: [1](#0-0) 

`addSigningAccountKeys` does the same per-account, continuing the loop silently: [2](#0-1) 

`addReceiverAccountKeys` and `addNodeKeys` follow the same catch-and-continue pattern: [3](#0-2) [4](#0-3) 

If every mirror-node call throws (network timeout, rate-limit, outage), `computeSignatureKey` returns a `new KeyList()` with zero entries: [5](#0-4) 

**Root cause 2 — `hasValidSignatureKey` treats empty `KeyList` as satisfied**

For an empty `KeyList`: `keys = []`, the `forEach` never executes so `currentThreshold` stays `0`, and `key.threshold` is `undefined`. The expression evaluates to `0 >= (undefined || 0)` = `0 >= 0` = `true`: [6](#0-5) 

**State-transition path in `processTransactionStatus`**

`processTransactionStatus` calls both functions in sequence. When `isAbleToSign = true` (from the empty-KeyList bypass) and `smartCollate` returns non-null (any transaction within the 6 KB size limit passes through untouched), `newStatus` is set to `WAITING_FOR_EXECUTION` and the DB is updated: [7](#0-6) 

The `smartCollate` function confirms: for a normal-sized transaction with an empty `KeyList`, it returns the SDK transaction directly (not null), as verified by the test `'returns the SDK transaction when small and within limits (normal payer)'`: [8](#0-7) 

**Trigger point 1 — User-triggered**

`SignersController.uploadSignatureMap` (POST `/transactions/:transactionId?/signers`, authenticated normal user) calls `SignersService.uploadSignatureMaps`, which calls `updateStatusesAndNotify`, which calls `processTransactionStatus`: [9](#0-8) [10](#0-9) 

**Trigger point 2 — Scheduler-triggered**

`handleTransactionsBetweenNowAndAfterThreeMinutes` runs every 10 seconds and calls `updateTransactions` → `processTransactionStatus` directly, requiring no user action: [11](#0-10) [12](#0-11) 

**Execution path continues the same flaw**

`ExecuteService.getValidatedSDKTransaction` also calls `computeSignatureKey` + `hasValidSignatureKey`. If the mirror node is still unavailable at execution time, the same empty-KeyList bypass passes, and the SDK transaction is submitted to Hedera without the required signatures: [13](#0-12) 

## Impact Explanation

- **State corruption**: A transaction is promoted to `WAITING_FOR_EXECUTION` without the required threshold of signatures, violating the core invariant of the multi-signature workflow.
- **Permanent lock / unrecoverable failure**: If the scheduler picks up the corrupted transaction and the mirror node is still down, the same bypass passes in `getValidatedSDKTransaction`, the transaction is submitted unsigned, Hedera rejects it, and the system permanently marks it `FAILED`. The transaction cannot be retried or recovered.
- **Audit integrity failure**: The `TransactionSigner` records and `transactionBytes` in the DB reflect a state that does not match the actual signature requirements.

## Likelihood Explanation

**Attacker preconditions (no privilege required)**:
- Hold a valid JWT (normal authenticated user).
- Upload a signature map to any transaction they are a signer on while the mirror node is transiently unavailable.

Mirror-node unavailability is a realistic condition: network partitions, rate-limiting, rolling restarts, or a targeted DDoS against the mirror node endpoint (which is a public service). The scheduler path requires no user action at all — it fires automatically every 10 seconds via `handleTransactionsBetweenNowAndAfterThreeMinutes`.

**Mitigating factor**: `AccountCacheService` may serve cached responses for recently-fetched accounts, reducing the window. However, cache misses (cold start, cache expiry, new accounts) leave the path fully open.

**Overall**: Low-to-moderate. Requires a transient infrastructure failure coinciding with a scheduler tick or a user upload, but the scheduler runs continuously, making the coincidence window non-negligible.

## Recommendation

1. **Propagate errors instead of swallowing them**: In `addFeePayerKey`, `addSigningAccountKeys`, `addReceiverAccountKeys`, and `addNodeKeys`, re-throw the caught exception (or return a sentinel error) so that `computeSignatureKey` itself throws when any required key cannot be fetched. `processTransactionStatus` and `getValidatedSDKTransaction` should then treat a thrown error as "unable to determine signature requirements" and abort the status transition rather than proceeding with an empty `KeyList`.

2. **Guard against empty `KeyList` in `hasValidSignatureKey`**: Add an explicit check — if the `KeyList` is empty (zero keys), return `false` unconditionally, since an empty required-key set is semantically meaningless and should never be considered satisfied.

3. **Guard in `processTransactionStatus`**: Before promoting a transaction to `WAITING_FOR_EXECUTION`, verify that the returned `KeyList` is non-empty as a defense-in-depth check.

## Proof of Concept

```
Precondition: Mirror node is unreachable (e.g., network partition or rate-limit).

1. Attacker (authenticated user, signer on transaction T) sends:
   POST /transactions/T/signers
   { "signatureMap": <any valid signature map> }

2. SignersService.uploadSignatureMaps() is called.
   → updateStatusesAndNotify() → processTransactionStatus()

3. processTransactionStatus() calls:
   signatureKey = await computeSignatureKey(T)
   // All accountCacheService calls throw → signatureKey = new KeyList() (empty)

4. hasValidSignatureKey([], new KeyList()) is called:
   keys = []; currentThreshold = 0;
   return 0 >= (undefined || 0)  →  0 >= 0  →  true

5. smartCollate(T, new KeyList()) is called:
   // Transaction is small (< 6 KB) → returns sdkTransaction (not null)

6. newStatus = WAITING_FOR_EXECUTION
   DB UPDATE: transaction T status → 'WAITING FOR EXECUTION'

7. Scheduler (every 10 s) picks up T, calls collateAndExecute(T):
   → getValidatedSDKTransaction(T)
   → computeSignatureKey(T) → empty KeyList (mirror still down)
   → hasValidSignatureKey → true
   → sdkTransaction.execute(client)  ← submitted to Hedera without required signatures
   → Hedera rejects → transaction T marked FAILED (unrecoverable)
```

export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
  } else throw new Error(`Invalid key type`);
};

export async function smartCollate(
  transaction: Transaction,
  requiredKeys: KeyList,
): Promise<SDKTransaction | null> {
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // HIP-1300: isTransactionOverMaxSize is fee-payer aware. Privileged payers
  // (0.0.2, 0.0.42-0.0.799) can carry up to 128 KB, so transactions under that
  // limit pass through here untouched and signatures are NOT stripped.
  if (await isTransactionOverMaxSize(sdkTransaction)) {
    const publicKeys = computeShortenedPublicKeyList(
      sdkTransaction._signerPublicKeys,
      requiredKeys,
    );

    const signatureMap = sdkTransaction.getSignatures();
    sdkTransaction.removeAllSignatures();

    for (const key of publicKeys) {
      sdkTransaction.addSignature(key, signatureMap);
    }

    // If the transaction is still too large,
    // set it to failed with the TRANSACTION_OVERSIZE status code
    // update the transaction, emit the event, and delete the timeout
    if (await isTransactionOverMaxSize(sdkTransaction)) {
      return null;
    }
  }

  return sdkTransaction;
}

  private async getValidatedSDKTransaction(
    transaction: Transaction,
  ): Promise<SDKTransaction> {
    /* Throws an error if the transaction is not found or in incorrect state */
    if (!transaction) throw new Error('Transaction not found');

    await this.validateTransactionStatus(transaction);

    /* Gets the SDK transaction from the transaction body */
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Gets the signature key */
    const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);

    /* Checks if the transaction has valid signatureKey */
    if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
      throw new Error('Transaction has invalid signature.');

    return sdkTransaction;
  }
