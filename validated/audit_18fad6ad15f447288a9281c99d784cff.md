Audit Report

## Title
Unchecked Execution Result in Sequential Transaction Group Allows Subsequent Transactions to Execute After Failure

## Summary
In `executeTransactionGroup`, the sequential execution loop calls `_executeTransaction` for each transaction and pushes every result into `rawResults`, but never inspects the returned status. When a transaction in a sequential group fails on the Hedera network, the loop continues executing all remaining transactions unconditionally. Additionally, the `atomic` flag — which implies all-or-nothing semantics — is never evaluated inside `executeTransactionGroup`, so atomic groups receive no abort-on-failure enforcement.

## Finding Description

**Root cause — `back-end/libs/common/src/execute/execute.service.ts` lines 86–91:**

```typescript
if (transactionGroup.sequential) {
  for (const { sdkTransaction, transaction } of transactions) {
    const delay = transaction.validStart.getTime() - Date.now();
    await sleep(delay);
    rawResults.push(await this._executeTransaction(transaction, sdkTransaction));
  }
}
```

`_executeTransaction` returns one of three values:
- `null` — duplicate or lost DB race
- `TransactionExecutedDto { status: FAILED, error: ... }` — Hedera rejected the transaction
- `TransactionExecutedDto { status: EXECUTED }` — success

The loop pushes every result into `rawResults` but **never reads it back**. There is no `break`, no early-return, and no conditional guard on the result before proceeding to the next iteration. [1](#0-0) 

`_executeTransaction` explicitly sets `transactionStatus = TransactionStatus.FAILED` and persists it to the database when the Hedera SDK throws, but **returns the DTO rather than throwing** — so the caller's loop never sees an exception and never stops. [2](#0-1) 

The `atomic` flag is accepted via the API, stored on the entity, and used in the scheduler to route groups into `executeTransactionGroup`. However, the word `atomic` does not appear anywhere inside `executeTransactionGroup`. Atomic groups fall through to the parallel execution `else` branch — no abort on failure, no sequential ordering. [3](#0-2) [4](#0-3) 

## Impact Explanation
A sequential transaction group is semantically ordered and dependency-bearing (e.g., tx1 creates a Hedera account, tx2 transfers HBAR to it; or tx1 sets an allowance, tx2 spends it). When tx1 fails on-chain:

1. The DB correctly records tx1 as `FAILED`.
2. The loop immediately submits tx2 to the Hedera network anyway.
3. tx2 may fail (wasted fees, confusing error) or — worse — succeed with unintended effects (e.g., HBAR sent to a wrong/non-existent account, or a state-dependent operation applied to stale state).

For `atomic` groups the impact is identical: a failure in any member transaction does not prevent the remaining transactions from being submitted. The all-or-nothing guarantee implied by the `atomic` flag is never enforced. Submitted transactions cannot be un-submitted, and HBAR transfers cannot be reversed — making any resulting state corruption permanent and unrecoverable. [5](#0-4) 

## Likelihood Explanation
No privileged access is required. Any user who creates a sequential or atomic transaction group and experiences a network-level failure (insufficient fees, account not found, transaction expired, node rejection) will trigger this path automatically via the scheduler. The scheduler path is fully automated and requires no further user interaction once the group is submitted. [6](#0-5) 

## Recommendation
Inside the sequential loop, inspect the result of each `_executeTransaction` call before proceeding:

```typescript
if (transactionGroup.sequential) {
  for (const { sdkTransaction, transaction } of transactions) {
    const delay = transaction.validStart.getTime() - Date.now();
    await sleep(delay);
    const result = await this._executeTransaction(transaction, sdkTransaction);
    rawResults.push(result);
    // Abort the sequence if the transaction failed
    if (result && result.status === TransactionStatus.FAILED) {
      break;
    }
  }
}
```

For `atomic` groups, add an explicit check inside `executeTransactionGroup` (e.g., `if (transactionGroup.atomic)`) and abort on any failure, marking remaining transactions accordingly. The `atomic` and `sequential` flags should have distinct, enforced semantics rather than both routing to the same unguarded loop.

## Proof of Concept

1. Create a transaction group with `sequential: true` containing two transactions: tx1 (e.g., a `CryptoTransfer` from an account with insufficient balance) and tx2 (any valid transaction).
2. Submit the group. The scheduler calls `executeTransactionGroup`.
3. `_executeTransaction` is called for tx1. The Hedera SDK throws (e.g., `INSUFFICIENT_ACCOUNT_BALANCE`). The catch block sets `transactionStatus = FAILED`, persists it, and returns `{ status: FAILED, error: "..." }`.
4. The loop pushes the failed result into `rawResults` and immediately proceeds to tx2 — no check, no break.
5. `_executeTransaction` is called for tx2. It is submitted to Hedera and executes (or fails independently).
6. Observe in the DB: tx1 is `FAILED`, tx2 is `EXECUTED` — the sequential dependency was not honored. [7](#0-6)
