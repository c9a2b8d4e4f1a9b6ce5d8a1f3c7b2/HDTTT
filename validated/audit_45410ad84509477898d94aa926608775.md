### Title
Missing Timeout on gRPC `AddressBookQuery` Causes Indefinite Hang in Transaction Execution Pipeline

### Summary

When a transaction is stored with a custom (non-standard) `mirrorNetwork` URL, the backend `ExecuteService` calls `getClientFromNetwork`, which issues a gRPC `AddressBookQuery` to the custom mirror node with no timeout configured. If the remote server accepts the TCP connection but never sends a response, the `execute()` promise never resolves, permanently freezing that transaction in `WAITING_FOR_EXECUTION` state and holding the distributed lock indefinitely. The same pattern exists in the frontend's `getNodeAddressBook` and `getClientFromNetwork` helpers.

### Finding Description

**Vulnerability class:** DoS / Permanent state freeze — missing network timeout on gRPC connection.

**Root cause — backend:**

`ExecuteService._executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` on every transaction execution: [1](#0-0) 

`getClientFromNetwork` in the backend common library, for any non-standard network (not `mainnet`/`testnet`/`previewnet`/`local-node`), issues a gRPC `AddressBookQuery` with **no timeout**: [2](#0-1) 

If the mirror node server at `mirrorNetwork` accepts the TCP connection but never sends a response, `AddressBookQuery.execute(client)` returns a promise that never settles. The entire `executeTransaction` call hangs indefinitely.

The `@MurLock(15000, 'transaction.id')` decorator on `executeTransaction` holds a distributed lock keyed to the transaction ID: [3](#0-2) 

Because the lock is never released (the function never returns), the transaction is permanently stuck in `WAITING_FOR_EXECUTION` — its status is never updated, and no other pod can acquire the lock to retry it.

**Root cause — frontend (two additional sites):**

`getClientFromNetwork` in the Electron main process has the identical pattern: [4](#0-3) 

`getNodeAddressBook` (exposed as an IPC handler) also issues an unbounded `AddressBookQuery`: [5](#0-4) 

**Attacker-controlled entry point:**

A registered organization user submits a transaction via the REST API with a `mirrorNetwork` value pointing to an attacker-controlled server. The field is passed directly from the frontend: [6](#0-5) 

No backend validation restricts `mirrorNetwork` to known/trusted endpoints. The attacker's server listens on the expected port, completes the TCP handshake, and then sends nothing.

### Impact Explanation

- The targeted transaction is permanently frozen in `WAITING_FOR_EXECUTION`. Its status is never written to the database, and no notification is emitted.
- The `@MurLock` for that transaction ID is never released, preventing any retry by other pods.
- If the scheduler re-queues the transaction (e.g., on service restart), each new execution attempt also hangs, compounding resource exhaustion (open gRPC connections, Node.js async handles).
- In the frontend, a hanging `getNodeAddressBook` IPC call freezes the renderer's network-setup flow indefinitely.

### Likelihood Explanation

- Any authenticated organization user can submit a transaction. No admin privilege is required.
- Setting a custom `mirrorNetwork` is a documented, normal product workflow (the UI explicitly supports custom node URLs).
- The attacker only needs to run a TCP server that accepts connections and never responds — trivial to implement.
- The attack is persistent: a single malicious transaction permanently corrupts that transaction's lifecycle state.

### Recommendation

Apply an explicit timeout to every `AddressBookQuery.execute()` call (and to `sdkTransaction.execute()` / `response.getReceipt()` in `_executeTransaction`). For example, wrap the call with a `Promise.race` against a rejection timer, or use the Hiero SDK's built-in `setRequestTimeout` / `setMaxAttempts` on the client before issuing the query:

```ts
// back-end/libs/common/src/utils/sdk/client.ts
const client = Client.forNetwork({})
  .setMirrorNetwork(MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0]))
  .setRequestTimeout(10_000);   // 10-second cap on any single RPC

const nodeAddressBook = await new AddressBookQuery()
  .setFileId(FileId.ADDRESS_BOOK)
  .execute(client);
```

Apply the same fix to `front-end/src/main/services/localUser/transactions.ts` and `front-end/src/main/services/localUser/sdk.ts`. Additionally, validate `mirrorNetwork` on the backend API to an allowlist or a well-formed URL pattern before storing it.

### Proof of Concept

1. Start a TCP server on an attacker-controlled host (e.g., `attacker.example.com:443`) that completes the TLS/TCP handshake but never sends any data.
2. As a normal authenticated organization user, call `POST /transactions` with:
   ```json
   {
     "mirrorNetwork": "attacker.example.com",
     "transactionBytes": "<valid signed tx bytes>",
     ...
   }
   ```
3. Wait for the transaction's `validStart` to pass. The chain service scheduler calls `executeTransaction`.
4. Inside `_executeTransaction`, `getClientFromNetwork("attacker.example.com")` is called.
5. `AddressBookQuery.execute(client)` opens a gRPC stream to `attacker.example.com:443`; the server accepts but never responds.
6. The promise never resolves. The `@MurLock` is held indefinitely.
7. Observe in the database: the transaction row remains in `WAITING_FOR_EXECUTION` forever. No status update or notification is ever emitted.
