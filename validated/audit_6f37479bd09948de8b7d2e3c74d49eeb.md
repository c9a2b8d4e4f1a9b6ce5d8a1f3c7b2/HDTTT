### Title
IP Rate-Limit State Not Scoped Per Real Identity — Attacker-Controlled `x-forwarded-for` Enables DoS Against Any User

### Summary
`IpThrottlerGuard` uses the `x-forwarded-for` request header as the rate-limit tracking key without any validation. Because this header is fully attacker-controlled, any unauthenticated actor can forge a victim's IP address and exhaust that IP's rate-limit quota, permanently blocking the victim from reaching any API endpoint until the window expires. This is the direct analog of the external report's "nonces not tracked per address" class: a shared counter (rate-limit bucket) is keyed by a value that is not authentically bound to the real caller.

### Finding Description
`IpThrottlerGuard.getTracker()` resolves the tracker key as:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
``` [1](#0-0) 

The `x-forwarded-for` header is a plain HTTP header that any HTTP client can set to an arbitrary string. The guard prefers it over `req.ip` (the actual socket-level IP), so a forged header always wins. This guard is registered as a global `APP_GUARD`, meaning it gates every single API route:

```typescript
{
  provide: APP_GUARD,
  useClass: IpThrottlerGuard,
},
``` [2](#0-1) 

The throttler module it enforces is backed by Redis and enforces two limits — a per-minute and a per-second bucket — both keyed by whatever string `getTracker` returns: [3](#0-2) 

Because the key is the forged IP string, every request the attacker sends with `x-forwarded-for: <victim_IP>` increments the victim's bucket in Redis. Once the bucket is full, all subsequent requests from the victim's real IP are rejected with HTTP 429 for the remainder of the TTL window.

### Impact Explanation
Any unauthenticated attacker can permanently deny a specific user (or the entire user base by cycling through known IPs) access to every API endpoint — login, transaction submission, signature upload, notification retrieval — for the duration of the rate-limit window. Because the throttler is global and Redis-backed, the DoS persists across all API service instances and survives restarts.

### Likelihood Explanation
The attack requires only the ability to send HTTP requests with a custom header, which is trivially achievable with `curl`, any scripting language, or a browser's `fetch` API. No authentication, no special knowledge, and no prior access is needed. The victim's IP address can be obtained from public sources (DNS, prior connections, etc.) or the attacker can simply target `127.0.0.1` or common internal ranges to affect all users behind a NAT.

### Recommendation
Never trust `x-forwarded-for` as a primary security boundary unless the application is deployed behind a trusted reverse proxy that strips and re-sets the header. The correct fix is one of:

1. **Remove `x-forwarded-for` from the tracker** and rely solely on `req.ip`, which Express populates from the actual TCP socket when `trust proxy` is not set.
2. **If a proxy is required**, configure Express's `trust proxy` setting to the exact number of trusted hops (e.g., `app.set('trust proxy', 1)`), which causes Express itself to validate and resolve `req.ip` correctly from the proxy chain — then use only `req.ip` in `getTracker`.

```typescript
// Corrected guard — no user-controlled header
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return clientIp;
}
```

### Proof of Concept

```bash
# Attacker exhausts the rate-limit bucket for victim IP 203.0.113.42
# (GLOBAL_MINUTE_LIMIT requests in under one minute)

for i in $(seq 1 200); do
  curl -s -o /dev/null \
    -H "x-forwarded-for: 203.0.113.42" \
    -H "x-frontend-version: 99.0.0" \
    https://api.example.com/auth/login
done

# Now the real user at 203.0.113.42 receives HTTP 429 on every request
curl -H "x-frontend-version: 99.0.0" https://api.example.com/auth/login
# → 429 Too Many Requests
```

The attacker's own real IP is never incremented; only the forged victim IP bucket in Redis is consumed. The victim is locked out for the full TTL window with no recourse. [4](#0-3) [5](#0-4) [6](#0-5)

/* eslint-disable @typescript-eslint/no-explicit-any */
import { HttpException, HttpStatus, Injectable } from '@nestjs/common';
import { ThrottlerGuard } from '@nestjs/throttler';

@Injectable()
export class IpThrottlerGuard extends ThrottlerGuard {
  protected getTracker(req: Record<string, any>): Promise<string> {
    const clientIp = req.headers['x-forwarded-for'] || req.ip;
    if (!clientIp) {
      throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
    }
    // console.log('client IP', clientIp.replace(/^.*:/, ''));
    return clientIp;
  }
