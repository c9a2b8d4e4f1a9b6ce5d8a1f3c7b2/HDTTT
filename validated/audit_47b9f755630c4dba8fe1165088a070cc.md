Audit Report

## Title
`changePassword` Endpoint Does Not Invalidate the Current JWT Session Token After Password Change

## Summary
The `PATCH /auth/change-password` endpoint successfully updates the user's password but never calls `blacklistService.blacklistToken` on the currently active JWT. Because the JWT payload contains only `userId` and `email` (no password hash), changing the password has zero effect on the validity of any already-issued token. Every other security-sensitive auth action in the same controller (`logout`, `verifyOtp`, `setPassword`) explicitly blacklists the relevant token after completing its action; `changePassword` is the sole exception.

## Finding Description
In `back-end/apps/api/src/auth/auth.controller.ts`, the handler is:

```typescript
// lines 115–119
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto): Promise<void> {
  return this.authService.changePassword(user, dto);
}
``` [1](#0-0) 

No `@Req() req` parameter is accepted, so the raw JWT is never extracted and never passed to `blacklistService.blacklistToken`. Compare with `logout`:

```typescript
// line 103
await this.blacklistService.blacklistToken(extractJwtAuth(req));
``` [2](#0-1) 

The `BlacklistService` stores the raw JWT string in Redis with a TTL equal to `JWT_EXPIRATION` days: [6](#0-5) 

`JwtBlackListAuthGuard` only checks Redis — it never adds tokens. Because `changePassword` never calls `blacklistToken`, the old token is never written to Redis and remains accepted by the guard for its full remaining lifetime. [7](#0-6) 

The JWT payload (`{ userId, email }`) contains no password material, so the password change itself provides no cryptographic invalidation of the token. [5](#0-4) 

## Impact Explanation
An attacker who has obtained a valid JWT (via XSS, network interception, logging, or a compromised device) retains full authenticated access to the API even after the victim changes their password. The password change — the primary remediation action available to a compromised user — provides no session revocation. The attacker can continue to create, sign, approve, and cancel Hedera transactions on behalf of the victim for the entire remaining lifetime of the stolen token (up to `JWT_EXPIRATION` days).

## Likelihood Explanation
JWT theft is a realistic and common attack vector. The `changePassword` flow is the natural response a user takes when they suspect compromise. No privileged attacker role is required — any user whose token is stolen is affected. The missing blacklist call is a straightforward, demonstrable omission consistent with the application's own security model.

## Recommendation
Inject `@Req() req: Request` into `changePassword` and call `await this.blacklistService.blacklistToken(extractJwtAuth(req))` after `authService.changePassword` succeeds, exactly as `logout` does. Optionally, also invalidate all other active sessions for the user if multi-session support is present.

## Proof of Concept
1. User A logs in via `POST /auth/login` and receives `tokenA`.
2. Attacker intercepts or steals `tokenA`.
3. User A, suspecting compromise, calls `PATCH /auth/change-password` with `tokenA` and valid `oldPassword`/`newPassword`. The server returns 200.
4. Attacker sends any authenticated request (e.g., `GET /transactions`) with `Authorization: Bearer tokenA`.
5. `JwtBlackListAuthGuard` checks Redis — `tokenA` is not present — and allows the request. `JwtAuthGuard` validates the signature and expiry — still valid. The attacker's request succeeds with full access.
