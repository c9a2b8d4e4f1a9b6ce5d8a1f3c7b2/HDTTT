### Title
`removeTransactionApprover` Does Not Update Parent Threshold After Child Node Deletion, Permanently Corrupting Approval State

### Summary

When a child `TransactionApprover` node (one with a non-null `listId`) is deleted via `DELETE /transactions/:transactionId/approvers/:id`, the `removeTransactionApprover` function soft-deletes the node and its entire subtree but never adjusts the parent node's `threshold`. This leaves the parent in a state where `threshold > remaining_children_count`, making it impossible to ever satisfy the approval requirement. The transaction becomes permanently stuck and unexecutable — an exact analog to the splay tree bug where a child reference is nulled without preserving the sibling subtree.

### Finding Description

**Vulnerability class:** State transition / accounting — orphaned invariant after node removal.

The `TransactionApprover` entity forms a tree: root nodes carry `transactionId` (non-null) and `listId = null`; child nodes carry `listId` pointing to their parent and `transactionId = null`. Tree (non-leaf) nodes carry a `threshold` value: the minimum number of children that must approve before the parent is considered approved.

**The invariant:** `threshold ≤ approvers.length` must hold at all times.

**The broken path — `removeTransactionApprover`:** [1](#0-0) 

This function fetches the approver and calls `removeNode(approver.id)`. `removeNode` executes a recursive SQL CTE that soft-deletes the target node and all its descendants: [2](#0-1) 

Critically, neither `removeTransactionApprover` nor `removeNode` touches the **parent** of the deleted node. The parent's `threshold` column is never decremented.

**Contrast with `updateTransactionApprover`**, which correctly handles this case when detaching a child: [3](#0-2) 

That branch decrements the threshold or soft-removes the parent if it becomes childless. `removeTransactionApprover` has no equivalent logic.

**The controller entry point** — reachable by any authenticated transaction creator: [4](#0-3) 

The only guard is `getCreatorsTransaction` (verifying the caller owns the transaction). There is no restriction preventing the caller from targeting a child node inside a threshold group.

**Secondary issue:** When the deleted node is a child (`transactionId = null`), `emitTransactionStatusUpdate` is called with `null` as the entity ID, so the status-update notification is silently dropped: [5](#0-4) 

### Impact Explanation

**Concrete scenario:**

1. Creator builds a threshold tree: parent node with `threshold = 2`, children `[A, B]`.
2. Creator calls `DELETE /transactions/1/approvers/A` (a valid, authenticated API call).
3. `removeNode(A.id)` soft-deletes `A`.
4. Parent still has `threshold = 2` but only child `B` remains.
5. The approval engine now requires 2 approvals from a group that has only 1 member.
6. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` — it can never be approved or executed.

This is permanent, unrecoverable state corruption. There is no API path to lower the threshold of a node that already has fewer children than its threshold (the `updateTransactionApprover` threshold-update path validates `dto.threshold ≤ approver.approvers.length`, so it cannot raise the threshold above children count, but the broken state is already below that floor). [6](#0-5) 

### Likelihood Explanation

The attacker precondition is: **authenticated user who is the creator of a transaction** — a standard, unprivileged role. The trigger is a single `DELETE` HTTP request targeting any child node inside a threshold group. No special knowledge beyond knowing the approver IDs (which are returned by `GET /transactions/:id/approvers`) is required. The scenario arises naturally when a creator legitimately tries to remove one approver from a threshold group, not realising the threshold is not auto-adjusted.

### Recommendation

In `removeTransactionApprover` (or inside `removeNode`), after soft-deleting the target node, fetch its parent (via `approver.listId`) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // NEW: adjust parent threshold before deletion
  if (approver.listId) {
    const parent = await this.dataSource.manager.findOne(TransactionApprover, {
      relations: ['approvers'],
      where: { id: approver.listId },
    });
    if (parent) {
      const remaining = parent.approvers.length - 1;
      if (remaining === 0) {
        await this.dataSource.manager.softRemove(TransactionApprover, parent);
      } else if (remaining < parent.threshold) {
        await this.dataSource.manager.update(TransactionApprover, parent.id, {
          threshold: remaining,
        });
      }
    }
  }

  const transactionId = approver.transactionId
    ?? (await this.getRootNodeFromNode(approver.id))?.transactionId;

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
}
```

The fix also resolves the secondary issue of `emitTransactionStatusUpdate` being called with `null` when the deleted node is a child.

### Proof of Concept

**Setup:**
1. Authenticate as user Alice (transaction creator).
2. `POST /transactions` → create transaction T (id=1).
3. `POST /transactions/1/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 2, "approvers": [{ "userId": 2 }, { "userId": 3 }] }] }
   ```
   This creates: root approver node R (threshold=2, listId=null, transactionId=1) with children A (userId=2) and B (userId=3).

**Trigger:**
4. `DELETE /transactions/1/approvers/{A.id}` — Alice removes child A.

**Observed state after deletion:**
- A is soft-deleted (`deletedAt` is set).
- R still has `threshold = 2`.
- R now has only one child: B.
- `threshold (2) > remaining children (1)` — invariant violated.

**Consequence:**
5. Users 2 and 3 attempt to approve. Only user 3 (B) can approve. The approval engine counts 1 approval against threshold=2 and never marks the transaction as fully approved.
6. Transaction remains permanently in `WAITING_FOR_SIGNATURES`. No execution path exists.
