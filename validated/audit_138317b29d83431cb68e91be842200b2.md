Audit Report

## Title
Missing Authorization Check on Transaction Comment Creation Allows Any Authenticated User to Comment on and Read Arbitrary Transactions

## Summary
The `POST /transactions/:transactionId/comments` and `GET /transactions/:transactionId/comments` endpoints apply only JWT authentication guards. No relationship check (creator, signer, observer, approver) is performed before allowing a user to write or read comments on any transaction. A developer-left TODO in the controller explicitly acknowledges this gap.

## Finding Description
In `comments.controller.ts`, the `createComment` handler is decorated only with `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level, and carries an explicit TODO acknowledging the missing check:

```
//TODO need some sort of guard or check to ensure user can comment here
``` [1](#0-0) 

The underlying `CommentsService.createComment` performs zero access validation — it directly creates and persists the comment with no check that the requesting user has any relationship to the target transaction: [2](#0-1) 

The `GET` routes similarly pass no user context into the service methods, meaning any authenticated user can enumerate all comments on any transaction by ID: [3](#0-2) 

By contrast, every other sensitive transaction sub-resource enforces a relationship check. For example, `ObserversService.getTransactionObserversByTransactionId` throws `UnauthorizedException` unless the requesting user is the creator, a signer, an observer, or an approver: [4](#0-3) 

Similarly, `ObserversService.createTransactionObservers` enforces creator-only access: [5](#0-4) 

## Impact Explanation
Any authenticated organization user can:
1. **Write comments** on any transaction by guessing or enumerating sequential integer transaction IDs, polluting comment threads and potentially injecting misleading information into multi-sig approval workflows.
2. **Read all comments** on any transaction, leaking sensitive operational context (e.g., internal notes about key holders, approval rationale, financial details) that the transaction creator intended to restrict to participants.

## Likelihood Explanation
Exploitation requires only a valid JWT, obtainable by any registered organization user. Transaction IDs are sequential integers in the database, making enumeration trivial. No elevated privilege, special tooling, or social engineering is required — a standard HTTP POST or GET to the endpoint is sufficient.

## Recommendation
Add a relationship check in `CommentsService` (or as a dedicated guard) before allowing read or write access to comments. The check should verify that the requesting user is one of: the transaction creator, a signer, an observer, or an approver — mirroring the pattern already implemented in `ObserversService.getTransactionObserversByTransactionId`. [6](#0-5) 

## Proof of Concept
1. Register two users, User A and User B, in the same organization.
2. User A creates a transaction (e.g., `transactionId = 42`) and does not add User B as observer, signer, or approver.
3. User B authenticates and obtains a JWT.
4. User B sends:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <user_b_jwt>
   Content-Type: application/json

   { "message": "Injected comment from unauthorized user" }
   ```
5. The server returns `201 Created` — the comment is persisted with no error.
6. User B sends:
   ```
   GET /transactions/42/comments
   Authorization: Bearer <user_b_jwt>
   ```
7. The server returns all comments on transaction 42, including any sensitive notes left by participants.
