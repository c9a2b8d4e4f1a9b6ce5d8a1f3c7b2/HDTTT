### Title
Missing Unique Constraint on `TransactionSigner` Allows Duplicate Signer Records via TOCTOU Race Condition in `uploadSignatureMaps`

### Summary
The `TransactionSigner` entity has no unique database constraint on `(transactionId, userKeyId)`. The duplicate-prevention check in `uploadSignatureMaps` is performed in application memory before the database transaction begins, creating a TOCTOU window. Two concurrent `POST /transactions/signers` requests from the same authenticated user can both pass the in-memory guard and each insert a separate `TransactionSigner` row for the same `(transactionId, userKeyId)` pair, corrupting the signing state of the transaction.

### Finding Description

**Root cause — no unique constraint on `TransactionSigner`:**

The entity definition carries only a non-unique index:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← plain index, NOT unique
export class TransactionSigner { ... }
``` [1](#0-0) 

The migration confirms the same — no `UNIQUE` constraint exists on the table:

```sql
CREATE TABLE IF NOT EXISTS "transaction_signer"
  ("id" SERIAL NOT NULL, "transactionId" integer NOT NULL,
   "userKeyId" integer NOT NULL, "userId" integer NOT NULL, ...)
``` [2](#0-1) 

The later migration adds only a plain index, still not unique:

```sql
CREATE INDEX "IDX_f8cbac3e0cae3730a762e4905e"
  ON "transaction_signer" ("transactionId", "userKeyId")
``` [3](#0-2) 

**TOCTOU window in `uploadSignatureMaps`:**

Step 1 — existing signers are loaded into an in-memory `Set` *before* any database transaction is opened:

```typescript
const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
  where: { transactionId: In(transactionIds) },
  select: ['transactionId', 'userKeyId'],
});
const signersByTransaction = new Map<number, Set<number>>();
for (const signer of existingSigners) {
  signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
}
``` [4](#0-3) 

Step 2 — the duplicate guard is evaluated against that in-memory snapshot:

```typescript
if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);   // ← only "new" signers are queued for insert
}
``` [5](#0-4) 

Step 3 — the insert is a plain `INSERT` with no `ON CONFLICT` clause:

```typescript
const result = await manager
  .createQueryBuilder()
  .insert()
  .into(TransactionSigner)
  .values(signersToInsert)
  .returning('*')
  .execute();
``` [6](#0-5) 

**Exploit flow:**

1. Attacker (authenticated user with key `K`) sends two concurrent `POST /transactions/signers` requests for the same `transactionId`.
2. Both requests execute `loadTransactionData` simultaneously; both read `existingSigners` and find no record for key `K` → both build an empty `existingSignerIds` set.
3. Both pass the `!existingSignerIds.has(userKey.id)` guard and queue key `K` for insertion.
4. Both execute `bulkInsertSigners` inside separate database transactions; since there is no unique constraint, both inserts succeed.
5. The `transaction_signer` table now contains two rows with identical `(transactionId, userKeyId)`.

### Impact Explanation

Duplicate `TransactionSigner` rows corrupt the authoritative record of who has signed a transaction. Downstream effects include:

- **Inflated signer count**: Any logic that counts distinct signers from the `TransactionSigner` table (e.g., threshold checks, audit queries) will double-count the attacker's key, potentially allowing a transaction to be considered "fully signed" with fewer real unique signers than required.
- **Audit trail corruption**: The signing history becomes unreliable; the same user/key appears multiple times, undermining non-repudiation guarantees.
- **Cascading state machine errors**: `processTransactionStatus` is invoked after every upload and drives the `WAITING_FOR_SIGNATURES → WAITING_FOR_EXECUTION` transition; if it relies on signer-record counts rather than pure cryptographic verification, a transaction could advance prematurely. [7](#0-6) 

### Likelihood Explanation

- **Attacker preconditions**: A valid, authenticated organization user with at least one registered key — no elevated privileges required.
- **Trigger**: Two concurrent HTTP requests to `POST /transactions/:transactionId/signers`. This is trivially achievable from any HTTP client (e.g., two parallel `fetch` calls in the front-end, or a script).
- **Reliability**: The race window spans the entire time between `loadTransactionData` and the commit of `persistSignatureChanges`, which includes signature validation and SDK processing — a window of tens to hundreds of milliseconds, making the race reliably winnable. [8](#0-7) 

### Recommendation

Apply both fixes:

1. **Add a unique database constraint** on `(transactionId, userKeyId)` in `TransactionSigner`:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'], { unique: true })  // enforce at DB level
export class TransactionSigner { ... }
``` [1](#0-0) 

2. **Use `INSERT ... ON CONFLICT DO NOTHING`** (or `upsert`) in `bulkInsertSigners` so that a concurrent duplicate insert is silently ignored rather than raising an unhandled error:

```typescript
await manager
  .createQueryBuilder()
  .insert()
  .into(TransactionSigner)
  .values(signersToInsert)
  .orIgnore()          // ON CONFLICT DO NOTHING
  .returning('*')
  .execute();
``` [9](#0-8) 

### Proof of Concept

```
Precondition:
  - User Alice is authenticated and has UserKey id=42.
  - Transaction id=7 is in WAITING_FOR_SIGNATURES status.
  - Alice has not yet signed transaction 7.

Steps:
  1. Alice sends two concurrent POST requests:
       POST /transactions/7/signers
       Body: { id: 7, signatureMap: <valid signature for key 42> }
     (both requests fired simultaneously before either completes)

  2. Both requests execute loadTransactionData at the same time.
     Both read transaction_signer WHERE transactionId=7 → empty result.
     Both build signersByTransaction.get(7) = Set{} (no existing signers).

  3. Both requests pass the guard:
       existingSignerIds.has(42) === false  →  userKeys = [key42]

  4. Both requests enter persistSignatureChanges and call bulkInsertSigners
     with { userId: Alice.id, transactionId: 7, userKeyId: 42 }.

  5. Both INSERTs succeed (no unique constraint to block the second one).

Expected (correct) result:
  SELECT * FROM transaction_signer WHERE "transactionId"=7 AND "userKeyId"=42;
  → 1 row

Actual (vulnerable) result:
  → 2 rows with identical (transactionId=7, userKeyId=42)

Impact:
  The signing record for transaction 7 is now corrupted.
  Any threshold check counting rows from transaction_signer
  will count Alice's key twice.
```

  private async updateStatusesAndNotify(
    transactionsToProcess: Array<{ id: number; transaction: Transaction }>
  ) {
    if (transactionsToProcess.length === 0) return;

    // Process statuses in bulk
    let statusMap: Map<number, TransactionStatus>;
    try {
      statusMap = await processTransactionStatus(
        this.txRepo,
        this.transactionSignatureService,
        transactionsToProcess.map(t => t.transaction)
      );
    } catch (err) {
      console.error('Bulk status processing failed:', err);
      statusMap = new Map();
    }

    // Separate new statuses from unchanged
    const newStatusResults: number[] = [];
    const unchangedResults: number[] = [];

    for (const { id } of transactionsToProcess) {
      if (statusMap.has(id)) {
        newStatusResults.push(id);
      } else {
        unchangedResults.push(id);
      }
    }

    // Emit notifications
    if (newStatusResults.length > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        newStatusResults.map(id => ({ entityId: id }))
      );
    }
    if (unchangedResults.length > 0) {
      emitTransactionUpdate(
        this.notificationsPublisher,
        unchangedResults.map(id => ({ entityId: id }))
      );
    }
  }
