### Title
Unpinned `actions/upload-artifact` Action in GitHub Workflow Enables Supply Chain Attack

### Summary
The `test-frontend.yaml` workflow uses `actions/upload-artifact@v7` (a mutable version tag) in four separate steps instead of pinning to an immutable SHA hash. All other actions in the repository are correctly pinned to full commit SHAs. An attacker who compromises the `actions/upload-artifact` repository and force-pushes a malicious commit under the `v7` tag would have their code executed in the CI environment.

### Finding Description
In `.github/workflows/test-frontend.yaml`, four steps reference `actions/upload-artifact@v7` without a SHA pin:

- Line 392: `uses: actions/upload-artifact@v7` — Upload Playwright Report
- Line 401: `uses: actions/upload-artifact@v7` — Upload Playwright Test Results
- Line 429: `uses: actions/upload-artifact@v7` — Upload Solo Diagnostics
- Line 442: `uses: actions/upload-artifact@v7` — Upload Kind Logs [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Every other action across all four workflows is correctly pinned to a full SHA, e.g.: [5](#0-4) [6](#0-5) 

The `v7` tag is a mutable pointer. GitHub does not prevent tag mutation on third-party action repositories.

### Impact Explanation
If `actions/upload-artifact@v7` is compromised (tag moved to a malicious commit), the injected code runs inside the CI runner with access to:
- All environment variables and secrets available to the job (e.g., `CODECOV_TOKEN`, `GITHUB_TOKEN`, `POSTGRES_PASSWORD`, `ORGANIZATION_URL`)
- The full checked-out source tree
- Any build artifacts already present on the runner

The workflow triggers on `push` to `main`/`release/**` and on `pull_request` to those branches, meaning every merge and every PR triggers the vulnerable steps. [7](#0-6) 

### Likelihood Explanation
The `actions/upload-artifact` action is maintained by GitHub itself, which lowers the probability of compromise compared to a random third-party action. However, the risk is non-zero: GitHub account compromises, insider threats, and dependency confusion attacks have all occurred against well-known organizations. The pattern is explicitly called out in supply-chain security guidance (e.g., SLSA, StepSecurity) and is the exact class of vulnerability described in the external report. The inconsistency — every other action is SHA-pinned while these four are not — suggests an oversight rather than a deliberate policy.

### Recommendation
Replace each `@v7` reference with the full commit SHA of the intended release. For example, for `actions/upload-artifact` v4 (the current stable major), the pinned form is:

```yaml
uses: actions/upload-artifact@65c4c4a1ddee5b72f698fdd19549f0f0fb45cf08 # v4.6.0
```

Apply the same pattern to all four occurrences in `.github/workflows/test-frontend.yaml` (lines 392, 401, 429, 442). The existing `dependabot.yml` already monitors `github-actions` packages daily, so Dependabot will keep the SHA up to date automatically once the pin is in place. [8](#0-7) 

### Proof of Concept

1. An attacker gains write access to the `actions/upload-artifact` repository (or the `actions` GitHub org).
2. They force-push a malicious commit to the `v7` tag that exfiltrates `$GITHUB_TOKEN` and all runner environment variables to an attacker-controlled endpoint.
3. The next `push` to `main` or any open PR against `main`/`release/**` triggers `test-frontend.yaml`.
4. The four `Upload …` steps execute the attacker's code with full access to the runner environment, secrets, and source tree.
5. The attacker receives the exfiltrated secrets and can use `GITHUB_TOKEN` (which has `contents: read` and `packages: write` in `publish-docker-images.yaml`) to push malicious Docker images to `ghcr.io`. [9](#0-8)
