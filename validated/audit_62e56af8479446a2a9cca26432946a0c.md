Audit Report

## Title
Creator of Any Transaction Can Delete Approvers Belonging to Other Transactions via Mismatched Authorization Check

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

**Controller — `approvers.controller.ts`, lines 102–113:**

The `removeTransactionApprover` handler performs two sequential calls with no relationship check between them:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks ownership of :transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver :id — no relationship check
``` [1](#0-0) 

Step 1 verifies the user is the creator of the transaction at `:transactionId`. Step 2 deletes the approver at `:id`. There is no check that the approver with `:id` belongs to the transaction with `:transactionId`.

**Service — `approvers.service.ts`, lines 534–544:**

`removeTransactionApprover` accepts only an `id`, fetches the approver, and removes it with no ownership or cross-transaction relationship validation:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

**Contrast with `updateTransactionApprover` — lines 386–394, which correctly validates:**

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The update path traverses to the root node, verifies the root's `transactionId` matches the URL parameter, and then verifies the user is the creator of *that* transaction. The delete path skips both checks entirely.

## Impact Explanation

Any authenticated, verified user who has created at least one transaction can delete approvers from **any** transaction in the system:

- An attacker can remove required approvers from a victim's transaction, causing it to bypass the intended approval threshold and proceed to execution without proper authorization.
- An attacker can remove all approvers from a transaction, permanently blocking it from reaching `WAITING_FOR_EXECUTION` status if approvers are required.
- This constitutes unauthorized state mutation of another user's transaction — a direct integrity failure in the multi-signature orchestration trust model.

## Likelihood Explanation

**High.** Attacker preconditions are minimal:
- Must be a registered, verified user (standard account creation flow).
- Must have created at least one transaction (to pass the `getCreatorsTransaction` check on their own transaction).
- No admin access, no leaked credentials, no privileged role required.
- The exploit is a single HTTP DELETE request with a crafted URL.

## Recommendation

Mirror the validation logic already present in `updateTransactionApprover`. In `removeTransactionApprover` (or in the controller before calling it), add the following checks:

1. Fetch the approver by `:id`.
2. Traverse to its root node via `getRootNodeFromNode`.
3. Assert `rootNode.transactionId === transactionId` — throw `UnauthorizedException` if not.
4. Call `getCreatorsTransaction(rootNode.transactionId, user)` to verify ownership.

This ensures the authorization check and the deletion target are always bound to the same transaction, consistent with how `updateTransactionApprover` already operates. [3](#0-2) 

## Proof of Concept

1. Attacker registers and verifies an account (User A).
2. User A creates Transaction A — this satisfies `getCreatorsTransaction` for Transaction A.
3. Victim (User B) creates Transaction B with approver records (e.g., approver with `id = 99` belonging to Transaction B).
4. Attacker sends:
   ```
   DELETE /transactions/{Transaction_A_id}/approvers/99
   Authorization: Bearer <User_A_token>
   ```
5. The server checks: "Is User A the creator of Transaction A?" → **Yes** (passes).
6. The server calls `removeTransactionApprover(99)` — deletes approver `99` which belongs to Transaction B, with no further checks.
7. Approver `99` is soft-deleted from Transaction B's approval workflow without User B's knowledge or consent. [4](#0-3)
