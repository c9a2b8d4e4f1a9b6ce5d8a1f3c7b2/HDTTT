## Audit Report

## Title
`TransferTransactionModel.getSigningAccounts()` Omits HTS Token and NFT Transfer Signing Accounts, Enabling Signature Bypass for Token Transfers

## Summary
`TransferTransactionModel.getSigningAccounts()` and `getReceiverAccounts()` only iterate over `hbarTransfersList`, completely ignoring `tokenTransfersList` and `nftTransfersList`. Because `TransactionSignatureService.computeSignatureKey()` derives the entire required `KeyList` from these two methods, any `TransferTransaction` that moves HTS fungible tokens or NFTs — but carries no HBAR debits — will produce an empty signing-accounts set. The backend will therefore treat such a transaction as fully signed with only the fee-payer's signature, allowing execution without the token owner's consent.

## Finding Description
`TransferTransaction` in the Hedera SDK exposes three independent transfer lists:

| List | Covers |
|---|---|
| `hbarTransfersList` | Native HBAR debits/credits |
| `tokenTransfersList` | HTS fungible-token debits/credits |
| `nftTransfersList` | HTS NFT ownership changes |

Each negative-amount entry in `tokenTransfersList` and each sender entry in `nftTransfersList` requires the owning account's key, exactly as a negative HBAR entry does.

`getSigningAccounts()` only processes `hbarTransfersList`:

```typescript
// back-end/libs/common/src/transaction-signature/model/transfer-transaction.model.ts
getSigningAccounts(): Set<string> {
  const accounts = new Set<string>();
  for (const transfer of this.transaction.hbarTransfersList) {  // ← HBAR only
    if (transfer.amount.isNegative() && !transfer.isApproved) {
      accounts.add(transfer.accountId.toString());
    }
  }
  return accounts;  // tokenTransfersList / nftTransfersList never touched
}
``` [4](#0-3) 

`getReceiverAccounts()` has the identical gap — it only reads `hbarTransfersList` for receiver-signature-required checks. [5](#0-4) 

`TransactionSignatureService.computeSignatureKey()` calls `extractSignatureRequirements()`, which delegates entirely to `transactionModel.getSigningAccounts()` and `transactionModel.getReceiverAccounts()`. The resulting `KeyList` is therefore empty for pure-token transactions (beyond the fee payer). [6](#0-5) 

The test suite confirms the blind spot: every test case in `transfer-transaction.model.spec.ts` mocks only `hbarTransfersList`; no test exercises `tokenTransfersList` or `nftTransfersList`. [7](#0-6) 

## Impact Explanation
An authenticated organization user can craft a `TransferTransaction` that debits HTS fungible tokens or NFTs from any account (e.g., a treasury or another member's account). Because `computeSignatureKey()` returns a `KeyList` that does not include the token owner's key, the backend's signature-completeness check passes with only the fee-payer's signature. The chain service then executes the transaction, draining tokens from the victim account without their consent.

## Likelihood Explanation
Any authenticated organization user can create and submit transactions — no privileged role is required. `TransferTransaction` is explicitly supported and listed in the `TransactionType` enum. The attack path is direct: create a `TransferTransaction` with a `tokenTransfersList` debit from a victim account, sign only as fee payer, upload the signature map, and the backend treats it as ready for execution.

## Recommendation
Extend both `getSigningAccounts()` and `getReceiverAccounts()` in `TransferTransactionModel` to also iterate over `tokenTransfersList` and `nftTransfersList`:

```typescript
getSigningAccounts(): Set<string> {
  const accounts = new Set<string>();

  for (const transfer of this.transaction.hbarTransfersList) {
    if (transfer.amount.isNegative() && !transfer.isApproved) {
      accounts.add(transfer.accountId.toString());
    }
  }

  // HTS fungible-token senders
  for (const tokenTransfer of this.transaction.tokenTransfersList) {
    for (const transfer of tokenTransfer.transfers) {
      if (transfer.amount.isNegative() && !transfer.isApproved) {
        accounts.add(transfer.accountId.toString());
      }
    }
  }

  // NFT senders
  for (const nftTransfer of this.transaction.nftTransfersList) {
    for (const transfer of nftTransfer.nftTransfers) {
      if (!transfer.isApproved) {
        accounts.add(transfer.senderAccountId.toString());
      }
    }
  }

  return accounts;
}
```

Apply the equivalent fix to `getReceiverAccounts()` for NFT receivers. Add unit tests covering pure-token and pure-NFT transfer scenarios.

## Proof of Concept

1. Attacker (account `A`) is an authenticated organization user.
2. Victim (account `V`) holds HTS tokens.
3. Attacker constructs a `TransferTransaction` with:
   - `tokenTransfersList`: debit `V` by N tokens, credit `A` by N tokens.
   - No HBAR transfers.
4. Attacker signs the transaction bytes with only their own fee-payer key.
5. Attacker uploads the signature map to the backend.
6. Backend calls `computeSignatureKey()` → `getSigningAccounts()` → iterates `hbarTransfersList` (empty) → returns `{}`.
7. Required `KeyList` contains only the fee-payer key (attacker's key), which is already present.
8. Signature-completeness check passes; the backend submits the transaction to the Hedera network.
9. The network executes the transfer, moving tokens from `V` to `A` without `V`'s signature.
