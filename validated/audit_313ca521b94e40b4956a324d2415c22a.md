Audit Report

## Title
Race Condition (Lost Update) in `importSignatures` Allows Silent Signature Drops

## Summary
The `importSignatures` method in `TransactionsService` performs a non-atomic read-modify-write on `transactionBytes` with no concurrency control. Two concurrent calls for the same transaction ID both read the same base bytes, independently append their signatures in memory, then race to write back — the second write silently overwrites the first, discarding valid signatures.

## Finding Description

The root cause is a non-atomic read-modify-write sequence in `importSignatures` with no database-level or application-level lock:

**Step 1 — Plain SELECT (no row lock):** [1](#0-0) 

**Step 2 — In-memory signature append:** [2](#0-1) 

**Step 3 — Blind overwrite (no version check, no optimistic lock):** [3](#0-2) 

There is no `entityManager.transaction()` wrapping `importSignatures` (unlike `createTransactions` which uses one at line 436), no `SELECT … FOR UPDATE`, no `@VersionColumn` on the `Transaction` entity, and no application-level mutex or distributed lock. The endpoint also has no rate-limit guard. [4](#0-3) 

**Interleaving that causes the lost update:**

| Time | Request A (legitimate signer) | Request B (attacker) |
|------|-------------------------------|----------------------|
| T0 | `find` → reads `bytes_0` | `find` → reads `bytes_0` |
| T1 | adds `sig_A` → `bytes_A` | adds `sig_B` → `bytes_B` |
| T2 | writes `bytes_A` | writes `bytes_B` (overwrites A) |
| T3 | — | `sig_A` is silently gone |

Because Node.js is single-threaded but `await` yields between the `find` and the `UPDATE`, both requests interleave freely at the I/O boundaries.

## Impact Explanation

1. **Silent signature drop** — the legitimate signer receives a `200 OK` with `{ id }`, unaware their signature was discarded.
2. **Transaction expiry** — Hedera transactions have a fixed `validStart` window. By repeatedly racing every legitimate import attempt, an attacker can prevent the required signature threshold from ever being reached before the deadline, permanently killing the transaction (expired transactions cannot be re-submitted with the same ID).
3. **Corrupt accumulated signature state** — the final `transactionBytes` stored in the DB may contain an incomplete signature set, causing downstream execution failures.

## Likelihood Explanation

- **Attacker precondition:** must be a registered, verified organization member with access to the target transaction — a realistic insider-threat profile.
- **Trigger:** two concurrent HTTP `POST /transactions/signatures/import` requests for the same transaction ID. Trivially achievable with any HTTP client (e.g., `Promise.all([fetch(…), fetch(…)])`).
- **No rate-limit protection** on the `/transactions/signatures/import` endpoint prevents sustained racing.
- **Silent success response** means the victim signer has no indication their signature was dropped.

## Recommendation

Wrap the entire read-modify-write in a serialized database transaction using `SELECT … FOR UPDATE` to acquire a row-level lock before reading `transactionBytes`:

```ts
await this.entityManager.transaction(async (em) => {
  // Use raw query or query builder with PESSIMISTIC_WRITE lock
  const transactions = await em.find(Transaction, {
    where: { id: In(ids) },
    lock: { mode: 'pessimistic_write' },
    relations: ['creatorKey', 'approvers', 'signers', 'observers'],
  });
  // ... modify and write back within the same transaction
});
```

Alternatively, add an optimistic-lock `@VersionColumn` to the `Transaction` entity and retry on conflict. Either approach eliminates the lost-update window.

## Proof of Concept

```ts
// Two concurrent requests for the same transaction ID
const payload = [{ id: TARGET_TX_ID, signatureMap: mySignatureMap }];

await Promise.all([
  fetch('POST /transactions/signatures/import', { body: payload, headers: authA }),
  fetch('POST /transactions/signatures/import', { body: payload, headers: authB }),
]);

// Both return 200 OK { id: TARGET_TX_ID }
// But only one signer's bytes survive in the DB — the other is silently dropped.
// Repeat until validStart expires → transaction is permanently dead.
```

The race is reproducible locally without any special tooling: fire two concurrent authenticated requests for the same transaction ID and inspect the resulting `transactionBytes` — it will contain only one signer's signature, not both.
