### Title
IDOR in `removeTransactionApprover`: Any Transaction Creator Can Delete Approvers Belonging to Other Users' Transactions

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController` performs an authorization check against `transactionId` (verifying the caller is the creator of that transaction), but then deletes the approver record identified by the separate `id` parameter without verifying that `id` actually belongs to `transactionId`. A malicious user who is the creator of any transaction can supply their own `transactionId` to pass the ownership check, while supplying a victim approver `id` from a completely different transaction to delete it.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `removeTransactionApprover` handler is:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

The authorization check `getCreatorsTransaction(transactionId, user)` only verifies the caller is the creator of the transaction identified by `transactionId`. The subsequent deletion `removeTransactionApprover(id)` operates on a completely independent `id` parameter with no check that this approver record belongs to `transactionId`. The two parameters are never correlated.

Root cause: authorization is checked on one object (`transactionId`) while the destructive action is performed on a different object (`id`), with no binding between them. [1](#0-0) 

### Impact Explanation
Any authenticated user who is the creator of at least one transaction can delete `TransactionApprover` records belonging to any other transaction in the system, regardless of who created that transaction. This allows:
- Removal of required approvers from pending multi-signature transactions, bypassing the approval workflow.
- Disruption of other organizations' transaction governance, potentially allowing transactions to proceed without required approvals or permanently corrupting the approval state of in-flight transactions.

This is a direct unauthorized state change on another user's/organization's assets.

### Likelihood Explanation
The attacker precondition is minimal: any verified, authenticated user who has created at least one transaction. No admin privileges are required. The attack requires only knowledge of a victim approver's numeric `id`, which is an auto-incremented integer and trivially enumerable. The endpoint is reachable via a standard authenticated HTTP `DELETE` request.

### Recommendation
After verifying the caller is the creator of `transactionId`, also verify that the approver record identified by `id` actually belongs to `transactionId` before deleting it. For example:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  // Add: verify approver `id` belongs to `transactionId`
  await this.approversService.verifyApproverBelongsToTransaction(id, transactionId);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

The `verifyApproverBelongsToTransaction` check should query the database for the approver record and confirm its `transactionId` field matches the URL parameter, throwing a `NotFoundException` or `ForbiddenException` if not.

### Proof of Concept

**Preconditions:**
- Attacker is a verified, authenticated user (User A).
- User A has created Transaction 1 (so they pass the `getCreatorsTransaction` check).
- Victim User B has created Transaction 2, which has an approver record with `id = 42`.

**Steps:**
1. Attacker authenticates and obtains a JWT.
2. Attacker enumerates approver IDs (sequential integers) to find `id = 42` belonging to Transaction 2.
3. Attacker sends:
   ```
   DELETE /transactions/1/approvers/42
   Authorization: Bearer <attacker_jwt>
   ```
4. The server calls `getCreatorsTransaction(1, attacker)` — passes, because attacker owns Transaction 1.
5. The server calls `removeTransactionApprover(42)` — deletes the approver record from Transaction 2 without any further check.
6. Transaction 2's approver with `id = 42` is permanently deleted, corrupting User B's transaction approval workflow. [1](#0-0)
