Audit Report

## Title
Missing Validation of `mirrorNetwork` Parameter Enables SSRF via Arbitrary Network Identifier

## Summary
`CreateTransactionDto` accepts a `mirrorNetwork` field validated only as a non-empty string. When a transaction is created, the backend immediately calls `getClientFromNetwork(dto.mirrorNetwork)`, which for any unrecognized value initiates a live outbound gRPC connection to `${mirrorNetwork}:443`. Any authenticated user can supply an arbitrary hostname or internal IP, causing the server to probe attacker-controlled or internal infrastructure (gRPC SSRF). If the attacker operates a gRPC stub that returns a valid address book, the stored `mirrorNetwork` is later used to construct `https://${mirrorNetwork}/api/v1/...` HTTP requests (HTTP SSRF).

## Finding Description

**Root cause â€” no whitelist on `mirrorNetwork`:**

`CreateTransactionDto` decorates `mirrorNetwork` only with `@IsNotEmpty()` and `@IsString()`: [1](#0-0) 

No `@IsIn([...])` or `@IsEnum(...)` guard exists. Any non-empty string passes DTO validation.

**Code path that triggers the outbound gRPC connection:**

`createTransactions` calls `getClientFromNetwork(dtos[0].mirrorNetwork)` unconditionally before any further validation: [2](#0-1) 

`getClientFromNetwork` falls through to the custom-network branch for any value that is not `mainnet`, `testnet`, `previewnet`, or `local-node`: [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` appends `:443` to the raw user-supplied string and uses it as the gRPC mirror-node endpoint: [4](#0-3) 

The `AddressBookQuery().execute(client)` call at lines 42â€“44 of `client.ts` then initiates a live gRPC TCP connection to `${mirrorNetwork}:443`.

**Secondary HTTP SSRF path:**

`MirrorNodeREST.fromBaseURL` constructs `https://${mirrorNetwork}` for any unrecognized value: [5](#0-4) 

`MirrorNodeClient.getMirrorNodeRESTURL` uses this directly to build request URLs: [6](#0-5) 

`fetchAccountInfo` and `fetchNodeInfo` then issue HTTP GET requests to `https://${mirrorNetwork}/api/v1/accounts/...` and `https://${mirrorNetwork}/api/v1/network/nodes?...`: [7](#0-6) [8](#0-7) 

## Impact Explanation

- **gRPC SSRF (immediate, no extra infrastructure):** Any authenticated user can force the API server to open a gRPC TCP connection to an arbitrary `host:443`. This enables internal port scanning, probing of internal gRPC services, and exfiltration of the server's egress IP.
- **HTTP SSRF (requires attacker-controlled gRPC server):** If the attacker operates a gRPC mirror-node stub that returns a syntactically valid address book, the transaction is persisted with the attacker-controlled `mirrorNetwork`. The chain service subsequently issues HTTP GET requests to `https://<attacker-host>/api/v1/accounts/...`, enabling access to internal REST services, cloud metadata endpoints (e.g., `169.254.169.254`), and credential theft in cloud environments.

## Likelihood Explanation

The attacker only needs a valid authenticated session (normal user account). No admin or privileged role is required. The gRPC SSRF path requires zero additional infrastructure â€” the connection attempt is made unconditionally during `POST /transactions`, even if it ultimately fails. The HTTP SSRF path requires the attacker to run a minimal gRPC stub that returns a syntactically valid address book, which is low effort.

## Recommendation

1. **Whitelist `mirrorNetwork` at the DTO layer.** Add `@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])` or an equivalent `@IsEnum(MirrorNetwork)` decorator to `CreateTransactionDto.mirrorNetwork`. [1](#0-0) 

2. **If custom networks are a required feature**, enforce a strict URL/hostname allowlist or require an admin-level role to supply custom network values. At minimum, validate the value against a regex that rejects IP addresses and internal hostnames before passing it to `getClientFromNetwork`.

## Proof of Concept

```
POST /transactions
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "name": "test",
  "description": "ssrf",
  "transactionBytes": "<valid_signed_tx_hex>",
  "creatorKeyId": <valid_key_id>,
  "signature": "<valid_sig_hex>",
  "mirrorNetwork": "169.254.169.254"
}
```

The backend immediately calls `getClientFromNetwork("169.254.169.254")`, which falls through to the custom-network branch, constructs a gRPC client targeting `169.254.169.254:443`, and executes `AddressBookQuery` against it â€” initiating a TCP connection to the cloud metadata endpoint from the server's network context. No admin privileges are required; any registered user can trigger this.
