Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

## Summary
`getTransactionsToSign` in `transactions.service.ts` issues an unbounded `repo.find()` with no row limit, fetching every non-terminal transaction in the system regardless of the requesting user. Per-row async `userKeysToSign` calls are then executed in memory. An authenticated attacker who bulk-creates transactions can inflate the active-transaction table, causing every call to `GET /transactions/sign` to perform an increasingly expensive full-table scan and N async operations, exhausting server memory and/or timing out the endpoint for all users.

## Finding Description

**Root cause — unbounded `repo.find()` with no `take`:**

The `whereForUser` predicate filters only by `status NOT IN (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED)` — it does not scope rows to the requesting user: [1](#0-0) 

The subsequent `repo.find()` carries no `take:` or `skip:` option, so it materialises the entire active-transaction table into memory: [2](#0-1) 

User-level filtering is then performed in-memory, one async `userKeysToSign` call per row, before pagination is applied via `result.slice()`: [3](#0-2) 

**Contrast with every other listing endpoint**, which correctly passes `take: limit` and `skip: offset` at the database level:

- `getTransactions` — `take: limit` at line 180 [4](#0-3) 
- `getHistoryTransactions` — `take: limit` at line 235 [5](#0-4) 
- `getTransactionsToApprove` — `take: limit` at line 349 [6](#0-5) 

`getTransactionsToSign` is the only query that bypasses this guard.

**The endpoint remains active.** Although the controller comment reads `/* NO LONGER USED BY FRONT-END */`, the route is still registered and reachable by any authenticated API client: [7](#0-6) 

**Secondary unbounded path — `POST /transactions/signatures/import`:**

The developer comment at line 575 explicitly acknowledges that the array accepted by `importSignatures` has no size cap at the API layer. The `BATCH_SIZE = 500` constant governs only the DB update batching, not the size of the incoming request array: [8](#0-7) 

A single request with thousands of entries triggers per-entry SDK deserialization and signature validation before any batching occurs.

**Rate limiting:** A global `IpThrottlerGuard` is registered as an `APP_GUARD`, which provides IP-level throttling on all routes including `POST /transactions`: [9](#0-8) 

However, IP-based throttling does not prevent the attack: once the transaction table is inflated (achievable over time from a single IP within throttle limits, or trivially from multiple IPs), every subsequent `GET /transactions/sign` call by *any* user triggers the expensive full-table scan. The throttler limits request rate, not the cost of each request.

## Impact Explanation

- **Availability:** As the attacker inflates the transaction table, every call to `GET /transactions/sign` by any user triggers an increasingly expensive full-table scan plus N sequential async DB/crypto operations. At sufficient scale the request times out, making the signing workflow unavailable for all organisation members.
- **Server memory:** The entire result set is materialised in the Node.js heap before pagination is applied (line 313). A large table causes heap exhaustion.
- **Cascading effect:** The `getTransactionsToSign` result drives the multi-signature workflow; a broken endpoint blocks signing for all users.

## Likelihood Explanation

- **Preconditions:** A valid organisation account — no admin privileges required.
- **Effort:** Scripted HTTP calls to `POST /transactions` with locally-crafted, frozen transaction bytes. No Hedera mainnet/testnet interaction is required; the SDK produces valid frozen bytes offline.
- **Rate limiting:** The global `IpThrottlerGuard` slows but does not prevent table inflation; multiple IPs or a sustained low-rate script from a single IP are both viable.
- **Persistence:** Rows remain in `WAITING_FOR_SIGNATURES` until their Hedera valid-start window expires, so the attacker can maintain pressure with a modest sustained request rate.

## Recommendation

1. **Add `take: limit` and `skip: offset` to the `repo.find()` call inside `getTransactionsToSign`**, mirroring the pattern used by every other listing endpoint. Pre-filter at the database level using a JOIN or subquery on `transaction_signer.userId = :userId` so that only rows relevant to the requesting user are fetched.
2. **If the endpoint is truly unused by the front-end, remove or disable it** to eliminate the attack surface entirely.
3. **Cap the input array size for `POST /transactions/signatures/import`** at the controller level (e.g., a `@MaxArraySize` class-validator decorator or an explicit length check), as acknowledged by the developer comment at line 575.
4. **Consider a per-user throttler** (`UserThrottlerGuard` already exists but is not applied globally) on `POST /transactions` to limit the rate at which any single authenticated user can create transactions.

## Proof of Concept

```
# Step 1 – Inflate the transaction table (repeat until table is large)
for i in $(seq 1 5000); do
  curl -s -X POST https://<server>/transactions \
    -H "Authorization: Bearer <attacker_token>" \
    -H "Content-Type: application/json" \
    -d '{
      "name": "flood-'$i'",
      "type": "TRANSFER",
      "description": "flood",
      "body": "<valid_frozen_tx_bytes_hex>",
      "status": "NEW",
      "creatorKeyId": <attacker_key_id>,
      "network": "testnet",
      "signature": "<valid_sig>",
      "validStart": "<future_timestamp>",
      "cutoffAt": "<future_timestamp>"
    }' > /dev/null
done

# Step 2 – Trigger the full-table scan as any user (including victim)
time curl -s "https://<server>/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer <victim_token>"
# Expected: request times out or returns after exhausting server memory
```

Each `POST /transactions` row persists in `WAITING_FOR_SIGNATURES` until expiry, so the attacker can maintain the inflated table with a low sustained rate. Every call to `GET /transactions/sign` by any user — including legitimate organisation members — will scan the entire inflated table.
