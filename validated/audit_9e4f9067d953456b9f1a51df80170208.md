Audit Report

## Title
Missing Group-Level Authorization in `removeTransactionGroup` Causes Partial `TransactionGroupItem` Data Corruption

## Summary
`removeTransactionGroup` in `transaction-groups.service.ts` fetches a `TransactionGroup` by ID with no user-ownership filter, then unconditionally deletes `TransactionGroupItem` records before any ownership check fires. A non-owner attacker can permanently corrupt another user's transaction group by orphaning its items, even though the `Transaction` records themselves are protected by a downstream check.

## Finding Description
In `back-end/apps/api/src/transactions/groups/transaction-groups.service.ts`, `removeTransactionGroup` (lines 172–198) performs no ownership check on the group before entering the deletion loop:

```typescript
const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
// ← No check: is `user` the owner of this group?
if (!group) { throw new Error('group not found'); }
``` [3](#0-2) 

Inside the loop, `TransactionGroupItem` is removed **before** the ownership check fires via `removeTransaction`:

```typescript
await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // deleted, no auth check
await this.transactionsService.removeTransaction(transactionId, user, false); // auth check here
``` [4](#0-3) 

`removeTransaction` calls `getTransactionForCreator`, which enforces `transaction.creatorKey?.userId !== user?.id`: [2](#0-1) 

Because there is no wrapping database transaction, the `TransactionGroupItem` deletion is committed before the `UnauthorizedException` is thrown. The group and its `Transaction` records survive, but the group is left in a corrupted state with a missing item.

By contrast, `cancelTransactionGroup` correctly calls `getTransactionGroup(user, groupId, false)` first, which runs a user-scoped SQL query via `getTransactionGroupItemsQuery`, and then asserts `allOwnedByUser` before any mutation: [5](#0-4) 

The `DELETE /transaction-groups/:id` controller endpoint is guarded only by JWT authentication — no ownership guard: [6](#0-5) 

## Impact Explanation
Any authenticated user can issue `DELETE /transaction-groups/<victim_group_id>` with their own valid JWT. The first `TransactionGroupItem` in the victim's group is permanently deleted before the `UnauthorizedException` is thrown. The `TransactionGroup` record and `Transaction` records survive, but the group is left in an inconsistent state (missing items), which can break group-level operations that depend on item completeness. Repeated calls targeting different groups can corrupt multiple groups. This is a data integrity failure, not full data destruction.

## Likelihood Explanation
- **Attacker precondition:** Any valid JWT — no admin role required.
- **Group ID discovery:** Sequential integers; trivially enumerable from the attacker's own groups or by probing.
- **No rate limiting or ownership guard** on the `DELETE` endpoint.
- The endpoint is publicly documented via `@ApiTags('Transaction Groups')`.

## Recommendation
Add a group-level ownership check at the start of `removeTransactionGroup`, mirroring `cancelTransactionGroup`. Either:
1. Call `getTransactionGroup(user, id, false)` first (which enforces user-scoped access via `getTransactionGroupItemsQuery` and throws `UnauthorizedException` if the user has no relationship to the group), or
2. Wrap the entire loop in a database transaction so that any mid-loop failure rolls back already-deleted `TransactionGroupItem` records atomically.

Both fixes should be applied together: the ownership check prevents unauthorized access, and the database transaction prevents partial state corruption.

## Proof of Concept
1. Attacker (user B) authenticates and obtains a valid JWT.
2. Attacker enumerates group IDs (e.g., observes ID `5` from their own group, infers victim's group is ID `4`).
3. Attacker issues: `DELETE /transaction-groups/4` with their JWT.
4. Server enters the loop, deletes the first `TransactionGroupItem` for group 4 (committed to DB).
5. Server calls `removeTransaction(T1.id, userB, false)` → `getTransactionForCreator` finds `T1.creatorKey.userId = victimId ≠ userB.id` → throws `UnauthorizedException`.
6. Response: HTTP 401. But the first `TransactionGroupItem` of the victim's group is permanently gone, leaving the group in a corrupted state.
