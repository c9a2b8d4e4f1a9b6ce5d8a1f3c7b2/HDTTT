Audit Report

## Title
Raw Database Error Messages Exposed in `POST /transactions/signatures/import` Response

## Summary
The `importSignatures` method in `transactions.service.ts` embeds raw database/ORM error messages directly into the API response body via the `SignatureImportResultDto.error` field. Because the error is placed in a return value rather than thrown, the global `BadRequestExceptionFilter` never intercepts it, and the raw message reaches the client.

## Finding Description
In `back-end/apps/api/src/transactions/transactions.service.ts`, when a batch database save fails, the catch block constructs a message by concatenating a prefix (`SAVE_ERROR_PREFIX`) with the raw `err.message` from the ORM/database layer and stores it in the result map returned to the caller. The `SignatureImportResultDto` exposes this `error` field via `@Expose()`, and the controller serializes and returns the DTO array directly to the client. [3](#0-2) 

The `BadRequestExceptionFilter` only intercepts *thrown* `BadRequestException` instances and sanitizes them to a generic message. Since the error here is embedded in the return value, the filter never runs. [4](#0-3) 

## Impact Explanation
Raw database error messages from TypeORM/PostgreSQL can reveal internal schema details such as table names, column names, constraint names, and ORM query structure. This information assists an attacker in mapping the database schema and crafting more targeted injection or manipulation payloads.

## Likelihood Explanation
The endpoint `POST /transactions/signatures/import` is reachable by any authenticated, verified user (guarded by `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). No admin privileges are required. A user can trigger the batch save failure path by submitting a malformed or oversized signature map payload that causes the raw SQL batch update to fail.

## Recommendation
In the catch block of `importSignatures`, do **not** include `err.message` in the response. Replace the error message with a static, generic string (e.g., `SAVE_ERROR_PREFIX` alone, without appending `err.message`). Log the full error server-side using a logger. This ensures internal schema details are never surfaced to the client regardless of how the error is returned.

## Proof of Concept
1. Authenticate as any verified user and obtain a valid JWT.
2. Submit a `POST /transactions/signatures/import` request with a signature map payload crafted to trigger a database constraint violation or ORM error (e.g., a duplicate key, oversized field, or invalid foreign key reference).
3. Observe the response body: the `error` field in the returned `SignatureImportResultDto` array will contain the raw TypeORM/PostgreSQL error message, including constraint names, table names, or column names.

```typescript
@Catch(BadRequestException)
export class BadRequestExceptionFilter implements ExceptionFilter {
  catch(exception: BadRequestException, host: ArgumentsHost) {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    const status = exception.getStatus();

    const isErrorCode = ErrorMessages[exception.message];

    response.status(status).json({
      statusCode: status,
      message: 'The request is invalid. Please check the request body and try again.',
      code: isErrorCode ? exception.message : ErrorCodes.UNKWN,
    });
  }
```
