### Title
Race Condition in `approveTransaction` Allows Malicious Approver to Approve Before Removal Takes Effect

### Summary
The `approveTransaction` function in `approvers.service.ts` performs a non-atomic read-check-write sequence. A malicious approver who learns they are about to be removed can race to submit their approval before the soft-delete is committed, potentially advancing the transaction to `WAITING_FOR_EXECUTION` status before the creator's revocation takes effect. This is the direct analog of the external report's `setSignatureValidatorApproval` race condition, mapped to the approver permission lifecycle in this protocol.

### Finding Description

**Vulnerability class:** TOCTOU / race condition in permission revocation (state transition).

**Root cause — `approveTransaction` is not atomic with `removeTransactionApprover`:**

`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` executes in three distinct, non-atomic phases:

**Phase 1 — Read (outside any DB transaction):** [1](#0-0) 

`getVerifiedApproversByTransactionId` calls `getApproversByTransactionId`, which filters `deletedAt IS NULL`: [2](#0-1) 

**Phase 2 — Check (on the in-memory snapshot from Phase 1):** [3](#0-2) 

**Phase 3 — Write (inside a DB transaction, but `whereInIds` does NOT re-check `deletedAt`):** [4](#0-3) 

**The removal path (`removeTransactionApprover`) is also non-atomic at the controller level:** [5](#0-4) 

`getCreatorsTransaction` (auth check) and `removeTransactionApprover` (soft-delete) are two separate awaited calls with no locking between them.

**The service-level removal:** [6](#0-5) 

**Race window exploit path:**

1. Creator (Alice) adds User B as an approver via `POST /transactions/:id/approvers`.
2. Alice discovers User B is malicious and calls `DELETE /transactions/:id/approvers/:approverId`.
3. User B observes (via the notification system or polling) that a removal is in flight.
4. User B immediately calls `POST /transactions/:id/approvers/approve`.
5. User B's request reaches Phase 1 (line 553) **before** the soft-delete commits — User B is still found as a live approver.
6. User B's request passes Phase 2 checks (lines 559–563) — no signature yet, user is an approver.
7. Alice's removal commits — User B's record gets `deletedAt = now()`.
8. User B's Phase 3 write (line 608) executes `whereInIds` with no `deletedAt` guard — **the approval is written to the now-live record** (if the write races before the soft-delete) or to the soft-deleted record (if after).
9. `emitTransactionStatusUpdate` fires at line 615, triggering the chain service to re-evaluate.

**Critical sub-case — approval written before soft-delete:**
If User B's Phase 3 write commits before Alice's soft-delete, the approval record has `deletedAt = NULL` and `approved = true`. The chain service's threshold evaluation counts it. If User B's approval was the threshold-completing vote, the transaction advances to `WAITING_FOR_EXECUTION`: [7](#0-6) 

Alice's subsequent removal fires another `emitTransactionStatusUpdate` (line 541), but if the chain service scheduler has already picked up the `WAITING_FOR_EXECUTION` transaction for submission to the Hedera network in the interval between the two events, the transaction executes with the malicious approval intact.

### Impact Explanation

A malicious approver can approve a transaction in the window between when the creator initiates their removal and when the soft-delete is committed. If their approval satisfies the threshold, the transaction advances to `WAITING_FOR_EXECUTION`. Depending on the chain service scheduler's polling interval, the transaction may be submitted to the Hedera network before the creator's removal triggers a re-evaluation. The result is an unauthorized transaction execution approved by a party the creator explicitly revoked.

### Likelihood Explanation

The attacker must be a legitimate approver on the transaction — a normal user role, not a privileged one. The race window is narrow (milliseconds between the two DB operations), but it is deterministically exploitable: the attacker can monitor the notification stream (`emitTransactionUpdate` / `emitTransactionStatusUpdate`) or poll the approvers endpoint to detect when a removal is in flight, then immediately submit their approval. No cryptographic break or admin access is required. The attack is a standard TOCTOU against a REST API with no request serialization or optimistic locking.

### Recommendation

**Short term:** Wrap the entire `approveTransaction` read-check-write sequence in a single serializable database transaction with a `SELECT ... FOR UPDATE` lock on the approver rows. Add an explicit `AND deletedAt IS NULL` predicate to the `whereInIds` update so that a soft-deleted approver record cannot receive a new approval even if the IDs were captured before deletion.

**Long term:** Introduce a database-level unique partial index or advisory lock keyed on `(transactionId, userId)` to serialize concurrent approval and removal operations for the same approver. Consider making `removeTransactionApprover` in the controller a single atomic service call rather than two separate awaited calls (`getCreatorsTransaction` + `removeTransactionApprover`).

### Proof of Concept

**Setup:**
- Transaction T exists with status `WAITING_FOR_SIGNATURES`.
- Alice (creator) and User B (approver) are registered. User B is the sole required approver (threshold = 1).

**Steps:**

1. Alice sends `DELETE /transactions/T/approvers/B_approver_id` (HTTP request in flight, not yet committed).
2. Simultaneously, User B sends `POST /transactions/T/approvers/approve` with a valid signature.
3. User B's request executes `getVerifiedApproversByTransactionId` (line 553) — returns User B as a live approver (soft-delete not yet committed).
4. User B's request passes the "already approved" check (line 563) — no signature on record.
5. Alice's soft-delete commits (`deletedAt = now()` on User B's approver row).
6. User B's `whereInIds` update (line 608) writes `approved = true`, `signature = <valid>` to the row — **no `deletedAt` guard prevents this**.
7. `emitTransactionStatusUpdate` fires (line 615).
8. Chain service evaluates: if the write at step 6 raced before step 5, the row has `deletedAt = NULL` and the threshold is met → transaction status advances to `WAITING_FOR_EXECUTION`.
9. Chain service scheduler submits the transaction to Hedera before Alice's removal re-evaluation reverts the status.

**Expected (broken) outcome:** Transaction T is executed on the Hedera network with User B's approval, despite Alice having removed User B as an approver.

**Expected (correct) outcome:** The approval write should be rejected if the approver row has been soft-deleted, and the entire read-check-write should be serialized within a single locked transaction.
