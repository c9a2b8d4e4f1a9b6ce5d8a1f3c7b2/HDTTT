### Title
`createTransactionObservers` Accepts Duplicate `userIds` in Input Array, Bypassing Deduplication Check

### Summary

The `createTransactionObservers` function in `ObserversService` iterates over the caller-supplied `dto.userIds` array and checks each entry only against the already-persisted `transaction.observers` list. Because the check never compares against the local `observers` array being built in the same loop, a transaction creator can supply the same `userId` multiple times in a single request and cause duplicate `TransactionObserver` rows to be inserted for the same user on the same transaction. This is the direct analog of the external report's unchecked-duplicate-array vulnerability class.

### Finding Description

**Root cause — `observers.service.ts` lines 49–54:**

```typescript
for (const userId of dto.userIds) {
  if (!transaction.observers.some(o => o.userId === userId)) {  // ← checks DB state only
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);   // ← local array never consulted on next iteration
  }
}
```

`transaction.observers` is the snapshot loaded from the database before the loop starts. It is never updated as the loop progresses. If `dto.userIds = [victimId, victimId, victimId]` and `victimId` is not already an observer, all three iterations pass the guard and three identical `TransactionObserver` objects are pushed into `observers`. All three are then persisted in a single `repo.save(observers)` call. [1](#0-0) 

**Entry point — `observers.controller.ts` line 44–50:**

The `POST /transactions/:transactionId/observers` endpoint accepts a `CreateTransactionObserversDto` whose `userIds` field is a plain array with no uniqueness constraint enforced at the DTO/validation layer. [2](#0-1) 

**Persistence — `observers.service.ts` line 61:**

```typescript
const result = await this.repo.save(observers);
```

TypeORM's `save` on an array issues individual `INSERT` statements. If the database has no `UNIQUE(transactionId, userId)` constraint, all duplicates are written. If such a constraint exists, the entire save fails and a `BadRequestException` is thrown, preventing even the legitimate (first) observer entry from being created. [3](#0-2) 

### Impact Explanation

**Path A — no DB unique constraint (data integrity):**
Multiple `TransactionObserver` rows exist for the same `(transactionId, userId)` pair. Every downstream query that iterates `transaction.observers` (e.g., access-control checks in `getTransactionObserversByTransactionId`, notification fan-out) processes the victim user multiple times. Notification services that fan out per-observer record will deliver duplicate notifications to the victim for every subsequent transaction event, constituting targeted notification spam with no rate limit.

**Path B — DB unique constraint present (availability):**
The `repo.save` call throws a database error. The catch block re-throws it as a `BadRequestException`. The entire batch — including legitimately new observers — is rejected. A malicious creator can permanently prevent any observer from being added to a transaction by always including a duplicate in the request, since the error aborts the whole save. [4](#0-3) 

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The attack requires only a crafted JSON body to the `POST /transactions/:id/observers` endpoint, which is a standard product workflow. No special tooling, timing, or cryptographic capability is needed. Any user who creates a transaction can immediately exploit this.

### Recommendation

Deduplicate `dto.userIds` before the loop, or track seen IDs within the loop itself:

```typescript
const seenUserIds = new Set<number>();
for (const userId of dto.userIds) {
  if (seenUserIds.has(userId)) continue;
  seenUserIds.add(userId);
  if (!transaction.observers.some(o => o.userId === userId)) {
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);
  }
}
```

Additionally, enforce a `UNIQUE(transactionId, userId)` constraint at the database level as a defence-in-depth measure.

### Proof of Concept

**Request:**
```
POST /transactions/1/observers
Authorization: Bearer <creator_jwt>
Content-Type: application/json

{ "userIds": [42, 42, 42] }
```

**Expected (correct) behaviour:** one `TransactionObserver` row for userId=42 is created.

**Actual behaviour (no DB constraint):** three `TransactionObserver` rows for userId=42 are created. Every subsequent notification event for transaction 1 delivers three notifications to user 42.

**Actual behaviour (DB constraint):** the save throws a unique-violation error; the `BadRequestException` is returned and no observers are added at all — including any legitimate new observers that were also in the array.

The deduplication gap is confirmed by the existing test at line 103–119 of `observers.service.spec.ts`, which only tests that a userId already present in `transaction.observers` (the DB snapshot) is skipped — it does not test the case where the same userId appears twice within the same incoming `dto.userIds` array. [5](#0-4)
