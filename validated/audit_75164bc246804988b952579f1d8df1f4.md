### Title
`emitTransactionStatusUpdate` Fired Unconditionally After Batch DB Update Failure in `importSignatures`, Causing False State Notifications

### Summary
In `transactions.service.ts`, the `importSignatures` method emits a `TRANSACTION_STATUS_UPDATE` notification for every transaction in `updateArray` regardless of whether the underlying batch database update succeeded or failed. If a batch `execute()` throws, the per-batch `catch` block marks those results as errored, but the `emitTransactionStatusUpdate` call outside the loop still fires for all IDs — including those whose `transactionBytes` were never persisted. This is the direct TypeScript analog of M-11: a downstream event is emitted even when the underlying state mutation failed.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `importSignatures` method processes signature uploads in batches:

```typescript
// lines 580–613
for (let i = 0; i < updateArray.length; i += BATCH_SIZE) {
  const batch = updateArray.slice(i, i + BATCH_SIZE);
  try {
    await this.entityManager
      .createQueryBuilder()
      .update(Transaction)
      .set({ transactionBytes: () => caseSQL })
      .where('id IN (:...ids)', { ids: batch.map(u => u.id) })
      .setParameters(params)
      .execute();
    batch.forEach(u => results.set(u.id, { id: u.id }));
  } catch (err) {
    // marks results with error — DB write did NOT happen
    batch.forEach(u => results.set(u.id, { id: u.id, error: message }));
  }
}

// lines 616–622 — fires for ALL updateArray entries, no check on per-batch success
emitTransactionStatusUpdate(
  this.notificationsPublisher,
  updateArray.map(r => ({
    entityId: r.id,
    additionalData: { transactionId: r.transactionId, network: r.network },
  })),
);
``` [1](#0-0) 

The `emitTransactionStatusUpdate` call is placed after the loop and outside any error guard. It iterates over `updateArray` — the full set of intended updates — not over a set of confirmed-successful updates. If any batch's `execute()` throws (transient DB error, constraint violation, connection loss), the catch block records the failure in `results` but does not remove those IDs from `updateArray`. The notification is still published for every ID.

The notification is consumed by `receiver.service.ts` → `processTransactionStatusUpdateNotifications`, which reads the transaction's current DB state and dispatches in-app and email notifications to signers, observers, and approvers. [2](#0-1) 

The `emitTransactionStatusUpdate` helper itself only logs a warning on publish failure; it does not throw or prevent downstream processing. [3](#0-2) 

A secondary, narrower instance exists in `archiveTransaction` (lines 720–730): `repo.update()` is called without checking its `affected` count, and `emitTransactionStatusUpdate` fires unconditionally. Under a TOCTOU race (status changed between the pre-check read and the update), the notification fires for a row that was not actually archived. [4](#0-3) 

### Impact Explanation
Downstream consumers (notification service, websocket gateway, email sender) receive `TRANSACTION_STATUS_UPDATE` events for transactions whose `transactionBytes` were never persisted. This causes:
- Signers, observers, and approvers to receive "transaction updated" in-app and email notifications for a state change that did not occur.
- The notification service reads the stale DB row and dispatches notifications based on the pre-failure status, creating a permanent inconsistency between what users are told and what is actually stored.
- In multi-batch scenarios (>500 transactions), a failure in one batch silently contaminates the notification set for the entire call.

### Likelihood Explanation
The trigger is a transient or persistent DB error during the batch `execute()` call — connection pool exhaustion, statement timeout, or a PostgreSQL constraint violation on `transactionBytes`. These are realistic operational conditions, not exotic attacker inputs. Any authenticated user who submits a large `importSignatures` request during a DB degradation event will trigger the inconsistency. No privileged access is required; the endpoint is reachable by any user with a valid session and access to at least one transaction.

### Recommendation
Track which batch IDs were successfully written and emit notifications only for those:

```typescript
const successfulIds = new Set<number>();

for (let i = 0; i < updateArray.length; i += BATCH_SIZE) {
  const batch = updateArray.slice(i, i + BATCH_SIZE);
  try {
    await this.entityManager...execute();
    batch.forEach(u => {
      results.set(u.id, { id: u.id });
      successfulIds.add(u.id);          // ← track confirmed writes
    });
  } catch (err) {
    batch.forEach(u => results.set(u.id, { id: u.id, error: message }));
  }
}

const toNotify = updateArray.filter(r => successfulIds.has(r.id));
if (toNotify.length > 0) {
  emitTransactionStatusUpdate(
    this.notificationsPublisher,
    toNotify.map(r => ({ entityId: r.id, additionalData: { ... } })),
  );
}
```

For `archiveTransaction`, check `updateResult.affected` before emitting, mirroring the pattern already used correctly in `cancelTransactionWithOutcome` (lines 681–691). [5](#0-4) 

### Proof of Concept

1. Authenticated user calls `POST /transactions/importSignatures` with >500 valid transaction IDs (triggering multi-batch processing).
2. During the second batch's `execute()`, the DB connection is interrupted (or a statement timeout fires).
3. The catch block at line 605 records errors for all IDs in that batch.
4. Execution continues to line 616; `emitTransactionStatusUpdate` publishes events for **all** IDs in `updateArray`, including the failed batch.
5. `receiver.service.ts` → `processTransactionStatusUpdateNotifications` processes the events, reads the stale DB rows (unchanged `transactionBytes`), and dispatches in-app and email notifications to all associated users.
6. Users receive "transaction updated" notifications for transactions that were not actually modified.

    const {

      transactionMap,
      approversMap,
      deletionNotifications,
      inAppNotifications,
      emailNotifications,
      inAppReceiverIds,
      emailReceiverIds,
      affectedUsers,

    // Process each event
    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping status-update notifications`);
        continue;
      }
      const approvers = approversMap.get(transactionId) || [];

      const syncType = this.getInAppNotificationType(transaction.status);
      const emailType = this.getEmailNotificationType(transaction.status);

      await this.entityManager.transaction(async entityManager => {
        await this.handleTransactionStatusUpdateNotifications(
          entityManager,
          transaction,
          approvers,
          syncType,
          emailType,
          cache,
          keyCache,
          deletionNotifications,
          inAppNotifications,
          inAppReceiverIds,
          emailNotifications,
          emailReceiverIds,
          affectedUsers,
          transactionId,
        );
      });

    await this.sendDeletionNotifications(deletionNotifications);
    await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
