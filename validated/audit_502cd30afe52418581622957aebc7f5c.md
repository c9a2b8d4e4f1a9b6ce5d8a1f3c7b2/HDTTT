### Title
Testing-Only `GET /transaction-groups` Endpoint Left in Production Exposes All Organization Transaction Groups to Any Authenticated User

### Summary
The `GET /transaction-groups` endpoint in `transaction-groups.controller.ts` is explicitly marked `/* TESTING ONLY */` in source code, carries no `@ApiOperation` documentation (unlike every other production endpoint in the file), and delegates to a service method that accepts no user parameter — returning every transaction group in the organization. Any authenticated, verified user can call it to enumerate all groups, bypassing the per-user access controls enforced by every other group-related endpoint.

### Finding Description

**Root cause — controller:**

`back-end/apps/api/src/transactions/groups/transaction-groups.controller.ts`, lines 52–57:

```typescript
/* TESTING ONLY: Get all transactions groups */
@Get()
@Serialize(TransactionGroupDto)
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroups();
}
``` [1](#0-0) 

The comment is the developer's own admission that this endpoint has no production purpose. It was left in by accident.

**Contrast with the adjacent production endpoint** (lines 67–75), which correctly passes `user` to enforce per-user access:

```typescript
getTransactionGroup(
  @GetUser() user: User,
  @Param('id', ParseIntPipe) groupId: number,
  ...
): Promise<TransactionGroup> {
  return this.transactionGroupsService.getTransactionGroup(user, groupId, full ?? true);
}
``` [2](#0-1) 

The testing-only handler passes **no user** to `getTransactionGroups()`, so the service performs no ownership or membership check — it returns all rows.

**Controller-level guards** (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) are applied at the class level: [3](#0-2) 

These guards only require a valid, non-blacklisted JWT from a verified account. They impose no ownership or role restriction on which groups are returned. Any verified user satisfies them.

**Exploit path:**
1. Attacker registers as a normal user and verifies their account (standard sign-up flow via `POST /auth/signup` + password set).
2. Attacker logs in (`POST /auth/login`) and obtains a JWT.
3. Attacker sends `GET /transaction-groups` with the JWT.
4. The response contains every `TransactionGroup` in the organization — including groups the attacker has no legitimate access to — with their names, descriptions, statuses, and associated transaction metadata.

### Impact Explanation
Every transaction group created by every user in the organization is disclosed. `TransactionGroupDto` includes group metadata and its linked transactions. An attacker learns the full set of pending and historical multi-signature workflows, their participants, and their statuses — information that is explicitly gated per-user on every other endpoint. This is unauthorized cross-tenant data exposure.

### Likelihood Explanation
The precondition is only a valid, verified account — the lowest privilege level in the system. No admin role, no special key, no internal access is required. The endpoint is a standard HTTP GET with no rate-limiting or additional guard. Any registered user can trigger it immediately after login.

### Recommendation
**Remove the endpoint entirely.** The developer comment `/* TESTING ONLY */` confirms it has no intended production use. It is the direct analog of the BancorSwapper `trade()` function: dead code left in by accident that is callable by any user. Deleting the handler and its corresponding service method (`getTransactionGroups`) eliminates the exposure with no functional regression.

If a paginated admin-only listing is ever needed in the future, it must be added behind `AdminGuard` with explicit user-scoped filtering.

### Proof of Concept

```
# Step 1 – obtain a JWT for any verified user
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2 – enumerate ALL organization transaction groups
GET /transaction-groups
Authorization: Bearer <JWT>

# Expected (vulnerable) response:
[
  { "id": 1, "description": "Q1 treasury rekey", "groupItems": [...] },
  { "id": 2, "description": "Node update batch", "groupItems": [...] },
  ...  ← groups belonging to OTHER users, fully disclosed
]
```

The attacker receives all groups without being a creator, observer, signer, or approver of any of them.
