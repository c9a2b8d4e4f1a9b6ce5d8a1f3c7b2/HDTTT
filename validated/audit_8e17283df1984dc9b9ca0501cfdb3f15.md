Audit Report

## Title
TOCTOU Race Condition in `archiveTransaction` Allows Overwriting Terminal Transaction Status

## Summary
`archiveTransaction` in `TransactionsService` performs a check-then-act pattern without atomicity. An unconditional `UPDATE` at line 720 carries no status guard, allowing it to overwrite a terminal `EXECUTED` or `FAILED` status written by the chain service's conditional update between the read and the write.

## Finding Description

`archiveTransaction` in `back-end/apps/api/src/transactions/transactions.service.ts` reads the transaction status, validates it, then issues an unconditional update:

**Step 1 — Read & validate (lines 709–718):** [1](#0-0) 

**Step 2 — Unconditional UPDATE (line 720):** [2](#0-1) 

There is no `WHERE status IN (...)` guard on the `UPDATE`. Between the read and the write, `_executeTransaction` in `back-end/libs/common/src/execute/execute.service.ts` can atomically transition the same row to `EXECUTED` or `FAILED` via its own conditional update: [3](#0-2) 

After the chain service wins its conditional update, `archiveTransaction`'s unconditional `UPDATE` still fires and stamps `ARCHIVED` over `EXECUTED`/`FAILED`, erasing `executedAt`, `statusCode`, `response`, `receipt`, and `receiptBytes`.

This contrasts directly with `cancelTransactionWithOutcome`, which correctly guards its update: [4](#0-3) 

And with `processTransactionStatus`, which also uses a conditional update: [5](#0-4) 

`archiveTransaction` is the only mutating path that omits this guard.

Note also: `executeTransaction` is decorated with `@MurLock(15000, 'transaction.id')` to prevent concurrent executions of itself, but `archiveTransaction` acquires no such lock, so the distributed lock provides no protection against this race. [6](#0-5) 

## Impact Explanation
A transaction creator can cause the database to record `ARCHIVED` for a transaction that was successfully submitted to and executed on the Hedera network. The on-chain execution is permanent and unaffected, but the backend's audit record is corrupted: the `executedAt` timestamp, `statusCode`, `response`, `receipt`, and `receiptBytes` fields written by `_executeTransaction` are all lost. Downstream consumers — notifications, history views, compliance exports — will see the transaction as archived rather than executed, creating a false audit trail.

## Likelihood Explanation
The race window is narrow but structurally predictable. The chain service schedules execution via `addExecutionTimeout` at exactly `validStart + 5 seconds`: [7](#0-6) 

A creator who knows their transaction's `validStart` can time an archive request to land in the window between the status check passing (`WAITING_FOR_EXECUTION`) and the chain service's conditional update completing. In a multi-pod deployment (Kubernetes/Helm is present in this repo), the API pod and chain pod run independently, making the race a real concurrent scenario rather than a theoretical one.

## Recommendation
Replace the unconditional `repo.update` in `archiveTransaction` with a conditional query-builder update that guards on the expected pre-archive statuses, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
const result = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', {
    statuses: [
      TransactionStatus.WAITING_FOR_SIGNATURES,
      TransactionStatus.WAITING_FOR_EXECUTION,
    ],
  })
  .execute();

if (!result.affected || result.affected === 0) {
  // Re-read and surface the actual current status to the caller
  throw new ConflictException('Transaction status changed; archive aborted.');
}
```

For manual transactions (where the `isManual` bypass applies), the same guard should be applied or the bypass logic should be explicitly scoped to only the statuses that are safe to archive.

## Proof of Concept

1. Creator submits a non-manual transaction; it reaches `WAITING_FOR_EXECUTION`.
2. Creator notes the `validStart` timestamp.
3. At `validStart + 4.9 s`, creator sends `PATCH /transactions/archive/:id`.
4. `archiveTransaction` calls `getTransactionForCreator` → reads `WAITING_FOR_EXECUTION` → check passes.
5. At `validStart + 5 s`, the chain service's `addExecutionTimeout` fires, `executeTransaction` → `_executeTransaction` executes on Hedera, then issues:
   ```sql
   UPDATE transaction SET status='EXECUTED', executedAt=now(), statusCode=22
   WHERE id=:id AND status='WAITING_FOR_EXECUTION'
   -- affected: 1 row
   ```
6. `archiveTransaction` resumes and issues:
   ```sql
   UPDATE transaction SET status='ARCHIVED' WHERE id=:id
   -- affected: 1 row  ← overwrites EXECUTED
   ```
7. Database now shows `status=ARCHIVED`, `executedAt=NULL`, `statusCode=NULL`, `receipt=NULL` for a transaction that was successfully executed on-chain.
