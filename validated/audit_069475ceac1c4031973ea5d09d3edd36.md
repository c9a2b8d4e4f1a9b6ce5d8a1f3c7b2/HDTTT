### Title
`getTransactionsToSign()` Fetches All Active Transactions Without Pagination, Causing Unbounded Per-Transaction Async Work on Every Request

### Summary
`getTransactionsToSign()` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every active transaction in the system without any database-level limit, then serially calls `userKeysToSign()` — which deserializes transaction bytes and queries the database — for each one before applying pagination in memory. Any authenticated user can trigger this endpoint repeatedly, causing CPU, memory, and database connection exhaustion proportional to the total number of active transactions in the system.

### Finding Description

**Root cause — unbounded `repo.find()` with no `take`/`skip`:**

In `getTransactionsToSign()`, the database query at lines 295–299 has no `take` or `skip` clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

Compare this to `getHistoryTransactions()` (lines 227–235) and `getTransactionsToApprove()` (lines 342–350), both of which correctly pass `skip: offset, take: limit` to the ORM query. `getTransactionsToSign()` is the only paginated endpoint that omits these constraints.

**Unbounded per-transaction async work:**

After loading all rows, the function loops over every transaction and awaits `userKeysToSign()` for each one (lines 301–309):

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

`userKeysToSign()` calls `userKeysRequiredToSign()` → `keysRequiredToSign()`, which per transaction:
1. Deserializes `transaction.transactionBytes` via `SDKTransaction.fromBytes()` (CPU-intensive protobuf decode)
2. Calls `transactionSignatureService.computeSignatureKey()`, which makes mirror-node/account-cache lookups
3. Queries the database for `UserKey` entities matching the required public keys

Pagination is only applied after the full loop completes (line 313):
```typescript
items: result.slice(offset, offset + limit),
```

**Attacker-controlled entry path:**

The endpoint `GET /transactions/sign` is exposed to any verified (non-admin) user:

```typescript
@Get('/sign')
getTransactionsToSign(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
  ...
)
```

The controller comment at line 157 even notes `/* NO LONGER USED BY FRONT-END */`, meaning there is no legitimate traffic to justify keeping this endpoint open, yet it remains fully reachable.

**Array growth path:**

Any authenticated user can create transactions via `POST /transactions`. As the number of active (non-terminal-status) transactions grows, the cost of each call to `GET /transactions/sign` grows linearly in both DB load and CPU time.

### Impact Explanation

A malicious authenticated user can:
1. Create a large number of transactions (each `POST /transactions` is a normal user action)
2. Repeatedly call `GET /transactions/sign` concurrently

Each request forces the server to load all active transactions into memory, deserialize each transaction's bytes, and perform per-transaction DB queries — all before returning a single page of results. With N active transactions, each request performs O(N) deserialization operations and O(N) database round-trips. Concurrent requests multiply this load. The result is:
- API service memory exhaustion (all transaction rows loaded into heap per request)
- Database connection pool saturation (N queries per request × concurrent requests)
- CPU exhaustion from repeated protobuf deserialization
- Effective denial of service for all API users, not just the attacker

### Likelihood Explanation

- **Attacker precondition**: Only a valid JWT (any verified user account). No admin access required.
- **Trigger**: A single HTTP GET to `/transactions/sign` with any pagination parameters.
- **Amplification**: The attacker can pre-seed the system with transactions (a normal user action) and then issue concurrent requests. The endpoint has no rate limit specific to this operation beyond the global `GLOBAL_MINUTE_LIMIT: "10000"` / `GLOBAL_SECOND_LIMIT: "1000"` configured in the Helm values, which are far too permissive to prevent this attack.
- **Persistence**: The endpoint remains active despite the front-end no longer using it, so there is no operational reason for it to exist.

### Recommendation

1. **Immediate**: Remove or disable the `GET /transactions/sign` endpoint entirely, since the front-end no longer uses it (as noted in the controller comment).
2. **If the endpoint must be kept**: Apply database-level pagination by adding `take: limit, skip: offset` to the `repo.find()` call, and push the signing-key filter into the SQL query (as `getTransactionsToApprove()` does with its recursive CTE) rather than doing it in application code.
3. **Defense in depth**: Add a per-endpoint rate limit on this route to prevent rapid repeated calls.

### Proof of Concept

**Setup**: Register as a normal user. Create N (e.g., 10,000) transactions via `POST /transactions` using valid transaction bytes.

**Trigger**:
```
for i in $(seq 1 50); do
  curl -s -H "Authorization: Bearer <user_jwt>" \
    "https://<api>/transactions/sign?page=1&size=10" &
done
wait
```

**Expected outcome**: The API service becomes unresponsive or OOM-crashes. Each concurrent request loads all N transaction rows, deserializes all N transaction byte arrays, and issues N database queries for `UserKey` lookups — all before returning 10 items. With 50 concurrent requests and 10,000 active transactions, this is 500,000 deserialization operations and 500,000 DB queries issued simultaneously.

**File:** back-end/libs/common/src/utils/transaction/index.ts (L20-94)
```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
```
