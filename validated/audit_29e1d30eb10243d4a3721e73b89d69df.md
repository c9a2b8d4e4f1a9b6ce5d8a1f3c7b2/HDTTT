Audit Report

## Title
Signature Bytes Not Cryptographically Verified in `uploadSignatureMaps`, Allowing Permanent Transaction Sabotage

## Summary
The `POST /transactions/:id/signers` endpoint (`uploadSignatureMaps` → `processTransactionSignatures`) accepts and persists signature maps without cryptographically verifying that the submitted signature bytes are valid over the transaction body. A required signer can submit their registered public key paired with arbitrary garbage bytes. The system records them as a valid signer, potentially advances the transaction to `WAITING_FOR_EXECUTION`, and then submits it to the Hedera network where it is permanently rejected with `INVALID_SIGNATURE`, marking the transaction as `FAILED` — a terminal, unrecoverable state.

## Finding Description

Two distinct code paths handle signature uploads, and they apply inconsistent validation:

**Path 1 — `importSignatures`** in `transactions.service.ts`:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);
``` [1](#0-0) 

This calls `validateSignature`, which performs a real cryptographic check at line 235 of `transaction.ts`:

```typescript
const signatureValid = publicKey.verify(bodyBytes, signature);
if (signatureValid) {
  signerPublicKeys.push(publicKey);
} else {
  throw new Error('Invalid signature');
}
``` [2](#0-1) 

**Path 2 — `processTransactionSignatures`** in `signers.service.ts`:

```typescript
let userKey = userKeyMap.get(raw);
if (!userKey) userKey = userKeyMap.get(publicKey.toStringDer());
if (!userKey) throw new Error(ErrorCodes.PNY);

sdkTransaction = sdkTransaction.addSignature(publicKey, map);
``` [3](#0-2) 

This path only verifies that the public key belongs to the authenticated user (`userKeyMap.get(raw)`). There is **no call to `validateSignature`** and no call to `publicKey.verify(...)`. The signature bytes themselves are never checked against the transaction body. `addSignature` in the Hedera SDK blindly appends whatever bytes are in the map and adds the public key to `_signerPublicKeys`.

At execution time, `getValidatedSDKTransaction` in `execute.service.ts` calls `hasValidSignatureKey`:

```typescript
if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
``` [4](#0-3) 

`hasValidSignatureKey` only checks whether the public key **string** is present in `_signerPublicKeys`:

```typescript
} else if (key instanceof PublicKey) {
  return publicKeys.includes(key.toStringRaw());
}
``` [5](#0-4) 

It performs no cryptographic verification of the signature bytes. The pre-execution check therefore passes, the transaction is submitted to Hedera with invalid signature bytes, and Hedera rejects it permanently.

The same `hasValidSignatureKey` pattern is also used in `processTransactionStatus` to decide whether to advance the transaction to `WAITING_FOR_EXECUTION`: [6](#0-5) 

## Impact Explanation

A required signer submits invalid signature bytes via `POST /transactions/:id/signers`. The system:
1. Accepts the submission — the public key ownership check passes
2. Persists the garbage bytes as the transaction's `transactionBytes`
3. Advances the transaction to `WAITING_FOR_EXECUTION` (because `hasValidSignatureKey` passes on key-string presence alone)
4. Submits the transaction to Hedera, which rejects it with `INVALID_SIGNATURE`
5. Marks the transaction as `FAILED` — a terminal state with no recovery path

This permanently destroys any multi-signature transaction the attacker is a required signer for. For high-value operations (node management, system freeze, large transfers), this is a critical sabotage vector.

## Likelihood Explanation

Any authenticated user whose public key is a required signer on a transaction can exploit this. No special privileges are needed beyond being a registered participant. The endpoint `POST /transactions/:id/signers` is a standard, documented API endpoint. The attacker only needs to craft a `signatureMap` with their registered public key and arbitrary garbage bytes — a trivial operation using the Hedera SDK. The asymmetry between the two paths (`importSignatures` vs `uploadSignatureMaps`) makes this easy to miss and confirms the missing check is not intentional design.

## Recommendation

Add a call to `validateSignature` inside `processTransactionSignatures` in `signers.service.ts`, mirroring the pattern already used in `importSignatures`:

```typescript
// After confirming the key belongs to the user, before addSignature:
const { data: _, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

This ensures that signature bytes are cryptographically verified against the transaction body before they are persisted, consistent with the behavior of `importSignatures`. [7](#0-6) 

## Proof of Concept

```typescript
import {
  AccountCreateTransaction,
  AccountId,
  PrivateKey,
  PublicKey,
  SignatureMap,
  TransactionId,
} from '@hiero-ledger/sdk';

// 1. Attacker has a registered key pair
const attackerKey = PrivateKey.generateECDSA();
const attackerPublicKey = attackerKey.publicKey;

// 2. Build a valid frozen transaction (as stored in the DB)
const tx = new AccountCreateTransaction()
  .setTransactionId(TransactionId.generate('0.0.2'))
  .setNodeAccountIds([AccountId.fromString('0.0.3')])
  .freeze();

// 3. Craft a SignatureMap with the attacker's real public key
//    but with garbage (all-zero) signature bytes
const garbageSignature = new Uint8Array(64); // 64 zero bytes
const fakeMap = new SignatureMap();
fakeMap.addSignature(
  AccountId.fromString('0.0.3'),
  tx.transactionId,
  attackerPublicKey,
  garbageSignature,
);

// 4. POST { id: <transactionId>, signatureMap: fakeMap }
//    to /transactions/:id/signers
//
// processTransactionSignatures will:
//   - find attackerPublicKey in userKeyMap (ownership check passes)
//   - call tx.addSignature(attackerPublicKey, fakeMap)
//     → garbage bytes are stored, key added to _signerPublicKeys
//   - isSameBytes = false → transaction bytes are persisted
//
// hasValidSignatureKey passes (key string is present).
// Transaction advances to WAITING_FOR_EXECUTION.
// Hedera rejects with INVALID_SIGNATURE.
// Transaction status → FAILED (terminal).
```

```typescript
  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
