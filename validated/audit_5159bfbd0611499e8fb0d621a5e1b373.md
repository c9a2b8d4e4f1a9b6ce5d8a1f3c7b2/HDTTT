Audit Report

## Title
Unbounded Full-Table Scan with Per-Row Async I/O in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

## Summary
`TransactionsService.getTransactionsToSign()` loads every non-terminal transaction in the database into memory with no row limit, then serially calls `userKeysToSign()` — which performs CPU-bound SDK deserialization and multiple async mirror-node/cache I/O operations — for each row before applying pagination. Any authenticated, verified user can trigger this endpoint to exhaust database connections, memory, and CPU proportional to the total number of active transactions in the system.

## Finding Description

**Root cause 1 — no `take` limit on the initial query:**

The `repo.find()` call in `getTransactionsToSign()` omits `skip`/`take`, loading every non-terminal transaction into the Node.js heap:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no skip / take
});
``` [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED), meaning every `NEW` and `WAITING_FOR_SIGNATURES` transaction across the entire organization is returned. [2](#0-1) 

By contrast, the sibling method `getHistoryTransactions()` correctly applies `skip`/`take` before querying: [3](#0-2) 

**Root cause 2 — serial async I/O loop over the full result set:**

Every returned transaction is processed serially with `await` inside a `for...of` loop:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [4](#0-3) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `keysRequiredToSign`, which calls `transactionSignatureService.computeSignatureKey()`. [5](#0-4) 

Each `computeSignatureKey()` call:
- Deserializes `transactionBytes` via `SDKTransaction.fromBytes()` (CPU-bound)
- Makes async calls to `accountCacheService.getAccountInfoForTransaction()` for the fee payer, signing accounts, and receiver accounts
- Potentially calls `nodeCacheService.getNodeInfoForTransaction()` for node transactions [6](#0-5) 

Additionally, `keysRequiredToSign` itself also calls `SDKTransaction.fromBytes()` independently: [7](#0-6) 

**Root cause 3 — pagination applied only after the full loop completes:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ← sliced after full scan
  ...
};
``` [8](#0-7) 

**Reachable entry point:**

The endpoint `GET /transactions/sign` is live and the controller class is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard authenticated user access, no admin role required: [9](#0-8) [10](#0-9) 

The controller comment explicitly acknowledges the endpoint is `/* NO LONGER USED BY FRONT-END */`, making it a forgotten, permanently-open attack surface: [11](#0-10) 

**Rate limiting does not mitigate this:**

The `UserThrottlerGuard` (100 req/min, 10 req/sec) is defined but is **not applied** to the `TransactionsController` — the grep for `UserThrottlerGuard` usage returns only the guard definition and its spec file, with no application to any controller. Even if it were applied, the damage is per-request (N deserialization operations + N×M async I/O calls), not per-request-volume. [12](#0-11) 

## Impact Explanation

With a large number of active transactions in the system:

- **Memory exhaustion**: All transaction rows and their `groupItem` relations are loaded into the Node.js heap simultaneously.
- **CPU exhaustion**: `SDKTransaction.fromBytes()` is CPU-bound and is called at least once per transaction per request (twice, in fact — once in `keysRequiredToSign` and once inside `computeSignatureKey`).
- **Database connection starvation**: Each `computeSignatureKey()` call issues multiple DB queries (account cache misses, UserKey lookups). With N transactions in the loop, this can saturate the DB connection pool.
- **Cascading failure**: The serial `await` inside `for...of` holds an open HTTP connection and DB resources for the entire duration, blocking other requests.

A single authenticated user making one request can render the API unresponsive for all other users.

## Likelihood Explanation

- **Attacker precondition**: Any verified user account — the lowest privilege level in the system. No admin, no special role.
- **Trigger**: One HTTP GET request to `GET /transactions/sign?page=1&size=1`.
- **Realistic scenario**: In an organization that has processed thousands of transactions over time, even a fraction remaining in non-terminal states (e.g., `WAITING_FOR_SIGNATURES`) is sufficient to cause severe degradation. The attacker does not need to create any transactions themselves.
- **No rate limiting on this endpoint**: As noted, `UserThrottlerGuard` is not applied to `TransactionsController`, so there is no per-user request cap at all on this endpoint.

## Recommendation

1. **Apply database-level pagination before the expensive loop**: Add a `take`/`skip` to the `repo.find()` call, or restructure the query to filter by user keys at the SQL level (as `getTransactionsToApprove` does with a subquery).
2. **Remove or disable the endpoint**: Since the comment explicitly states it is `NO LONGER USED BY FRONT-END`, the endpoint should be removed or protected with an admin-only guard to eliminate the attack surface entirely.
3. **Parallelize or batch the async loop**: Replace the serial `for...of` with `Promise.all` with a concurrency limit, or move the key-matching logic into the database query.
4. **Apply `UserThrottlerGuard` to the controller**: Ensure rate limiting is actually enforced on this and similar endpoints.

## Proof of Concept

```
GET /transactions/sign?page=1&size=1
Authorization: Bearer <valid_verified_user_jwt>
```

With N active (non-terminal) transactions in the database, this single request causes:
1. A full-table scan loading all N rows + `groupItem` relations into memory.
2. N sequential calls to `computeSignatureKey()`, each performing `SDKTransaction.fromBytes()` + 1–3 async I/O calls to the account/node cache.
3. The response returns only 1 item (as requested), but the server has already paid the full O(N) cost.

Repeating this request (even once per second, since no throttle is applied) sustains the resource exhaustion indefinitely.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
