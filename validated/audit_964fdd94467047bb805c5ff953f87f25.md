### Title
`archiveTransaction()` Bypasses Status Guard for Manual Transactions, Enabling Terminal-State Overwrites

### Summary
The `archiveTransaction` function in `transactions.service.ts` contains a flawed boolean guard that completely skips status validation when `isManual = true`. This allows any transaction creator to archive a manual transaction regardless of its current status — including terminal states like `EXECUTED`, `FAILED`, `CANCELED`, and `EXPIRED` — by calling the API endpoint directly. The result is an unauthorized state transition that corrupts the transaction's audit record and misleads all observers.

### Finding Description

**Root cause — incorrect guard logic:** [1](#0-0) 

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual          // ← short-circuits the entire check
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

The condition uses `&&`. When `transaction.isManual === true`, `!transaction.isManual` evaluates to `false`, making the entire `if` condition `false` regardless of `transaction.status`. The status check is never evaluated. Any status — `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, even `ARCHIVED` again — passes through to the unconditional `repo.update(...)` call.

**The `repo.update` call has no SQL-level status guard:** [2](#0-1) 

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

There is no `.andWhere('status IN (:...statuses)', ...)` clause, so the database row is unconditionally overwritten.
