Audit Report

## Title
Missing Unique Constraint on `TransactionSigner` Allows Duplicate Signer Records via Race Condition

## Summary
The `TransactionSigner` entity declares only a plain (non-unique) composite index on `(transactionId, userKeyId)`. The application-level duplicate check in `SignersService.uploadSignatureMaps` reads existing signers into an in-memory `Set` outside any database transaction, creating a TOCTOU window. Two concurrent `POST /transactions/:id/signers` requests from the same user can both pass the in-memory check and both insert identical signer rows, corrupting the signer ledger.

## Finding Description

**Root cause 1 — Missing unique constraint on `TransactionSigner`:**

`TransactionSigner` uses a plain index with no `unique: true` option: [1](#0-0) 

Compare with `TransactionObserver`, which correctly enforces uniqueness at the database level: [2](#0-1) 

**Root cause 2 — TOCTOU in the duplicate check:**

`uploadSignatureMaps` calls `loadTransactionData` first, which reads existing signers into an in-memory `Set` using `dataSource.manager.find` — entirely outside any database transaction: [3](#0-2) [4](#0-3) 

The "new signer" decision is made in `processTransactionSignatures` by checking that stale in-memory set: [5](#0-4) 

The actual database insert happens later, inside a separate `dataSource.transaction(...)` call in `persistSignatureChanges` → `bulkInsertSigners`, with no conflict-handling clause (`ON CONFLICT DO NOTHING` or similar): [6](#0-5) 

## Impact Explanation

Duplicate `TransactionSigner` rows corrupt the authoritative signer ledger. Any downstream logic that counts or iterates signer records — including `processTransactionStatus` and the notification/dismissal pipeline — will observe inflated signer counts. In a multi-signature threshold workflow this can cause a transaction to be advanced to `WAITING_FOR_EXECUTION` (or treated as fully signed) before the required number of *distinct* keys have actually signed, breaking the core multi-sig integrity guarantee. The duplicate rows persist permanently, corrupting the audit trail even after the transaction is executed. [7](#0-6) 

## Likelihood Explanation

Any authenticated user who is a required signer on a transaction can trigger this with two simultaneous HTTP requests — no elevated privileges, no leaked secrets, no special tooling beyond a basic HTTP client. The race window spans multiple async I/O operations: two DB reads in `loadTransactionData`, SDK signature processing in `validateAndProcessSignatures`, and then a separate DB write transaction in `persistSignatureChanges`. Under normal network latency the race is reliably winnable.

## Recommendation

Two complementary fixes are required — neither alone is sufficient:

1. **Add a unique constraint at the database level** on `TransactionSigner(transactionId, userKeyId)`, mirroring `TransactionObserver`:
   ```ts
   @Index(['transactionId', 'userKeyId'], { unique: true })
   ```
   This is the hard safety net that prevents duplicates regardless of application logic.

2. **Use `ON CONFLICT DO NOTHING` (or `orIgnore()`) in `bulkInsertSigners`** so that concurrent inserts that race past the application check are silently deduplicated rather than throwing an error:
   ```ts
   manager.createQueryBuilder()
     .insert()
     .into(TransactionSigner)
     .values(signersToInsert)
     .orIgnore()          // ON CONFLICT DO NOTHING
     .returning('*')
     .execute();
   ```

## Proof of Concept

```
# Terminal 1
curl -X POST https://<host>/transactions/42/signers \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '[{"id":42,"signatureMap":<map>}]' &

# Terminal 2 (simultaneous)
curl -X POST https://<host>/transactions/42/signers \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '[{"id":42,"signatureMap":<map>}]' &

wait

# Verify duplicates
curl https://<host>/transactions/42/signers -H "Authorization: Bearer <token>"
# Returns two TransactionSigner rows with identical transactionId and userKeyId
```

Both requests call `loadTransactionData` concurrently, both see an empty `signersByTransaction` set for the key, both pass the `!existingSignerIds.has(userKey.id)` check, and both execute `bulkInsertSigners`. Because there is no unique constraint and no conflict clause, the database accepts both inserts.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
