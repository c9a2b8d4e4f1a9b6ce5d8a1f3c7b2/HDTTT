Audit Report

## Title
Unauthenticated Attacker Can Exhaust Per-Email Rate Limit on `/auth/reset-password`, Locking Out Any User's Password Reset

## Summary
`POST /auth/reset-password` is protected only by `EmailThrottlerGuard`, which keys its throttle bucket on the attacker-supplied `req.body.email`. Because no authentication is required to reach this endpoint, any unauthenticated attacker who knows a victim's email can exhaust the per-email rate limit (3 requests per 60-second window in production) and permanently block the victim from requesting a password-reset OTP for as long as the attack is sustained.

## Finding Description

**Root cause — `EmailThrottlerGuard.getTracker` uses attacker-supplied email as bucket key**

`EmailThrottlerGuard.getTracker` returns `req.body.email` directly as the throttle-bucket key: [1](#0-0) 

The bucket limits are configured in `EmailThrottlerModule` using `ANONYMOUS_MINUTE_LIMIT` and `ANONYMOUS_FIVE_SECOND_LIMIT`: [2](#0-1) 

Production configs confirm the live limits are `ANONYMOUS_MINUTE_LIMIT=3` and `ANONYMOUS_FIVE_SECOND_LIMIT=1`: [3](#0-2) [4](#0-3) [5](#0-4) 

**Vulnerable endpoint — no authentication before the throttle guard**

`POST /auth/reset-password` applies only `EmailThrottlerGuard` — no JWT guard, no credential validation: [6](#0-5) 

Compare with `POST /auth/login`, where `LocalAuthGuard` (credential validation) runs *before* `EmailThrottlerGuard`, so an unauthenticated attacker cannot exhaust the login bucket without valid credentials: [7](#0-6) 

**Secondary issue — `IpThrottlerGuard` trusts attacker-controlled `x-forwarded-for`**

The global IP throttler reads `x-forwarded-for` directly from the request header without validation, allowing an attacker to rotate arbitrary values and get a fresh bucket on every request: [8](#0-7) 

`IpThrottlerGuard` is registered as a global `APP_GUARD`, meaning it applies to all routes: [9](#0-8) 

## Impact Explanation

A user who has forgotten their organization password is locked out of the password-reset flow for as long as the attacker sustains the attack (3 HTTP requests per minute). The victim cannot obtain an OTP, cannot set a new password, and therefore cannot authenticate to the organization server. All pending multi-signature transactions requiring that user's approval or signature are stalled. The attack is silent — the victim sees only HTTP 429 with no indication of malicious activity.

## Likelihood Explanation

- **No authentication required**: the endpoint is fully public.
- **Attacker precondition**: knowledge of the victim's email address only (often obtainable from the organization's user list or public sources).
- **Attack cost**: 3 HTTP requests per minute — trivially sustainable from a single machine or a cron job.
- **Global IP throttler bypassable**: `IpThrottlerGuard` trusts the attacker-controlled `x-forwarded-for` header, so the attacker can rotate fake IPs to avoid the global rate limit entirely, compounding the issue.
- The attack is persistent and requires no special tooling.

## Recommendation

1. **Add IP-based throttling to `/auth/reset-password`** in addition to email-based throttling, so that exhausting the email bucket also requires exhausting the attacker's own IP bucket (which cannot be spoofed if `x-forwarded-for` is sanitized).
2. **Fix `IpThrottlerGuard`** to use only `req.ip` (set by the framework/proxy after trusted proxy configuration) rather than blindly trusting `req.headers['x-forwarded-for']`. Configure NestJS/Express `trust proxy` settings appropriately so `req.ip` reflects the real client IP behind a reverse proxy.
3. **Consider CAPTCHA or proof-of-work** on the `/auth/reset-password` endpoint to raise the cost of automated abuse.
4. **Decouple the throttle key** from the attacker-supplied input: for example, key the email bucket on `IP + email` combined, so exhausting one victim's email bucket also consumes the attacker's IP quota.

## Proof of Concept

```bash
VICTIM="victim@org.com"
SERVER="https://org-server.example.com"

# Exhaust the per-email bucket (3 requests per 60s window)
for i in 1 2 3; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST "$SERVER/auth/reset-password" \
    -H "Content-Type: application/json" \
    -H "x-frontend-version: 0.24.0" \
    -H "x-forwarded-for: 10.0.0.$i" \
    -d "{\"email\":\"$VICTIM\"}"
done

# Victim's own legitimate request now receives 429
curl -s -o /dev/null -w "%{http_code}\n" \
  -X POST "$SERVER/auth/reset-password" \
  -H "Content-Type: application/json" \
  -H "x-frontend-version: 0.24.0" \
  -d "{\"email\":\"$VICTIM\"}"
# → 429 Too Many Requests

# Attacker repeats every 60 seconds to maintain the lockout
```

The `x-forwarded-for` rotation (lines 3 and 8 of the loop) bypasses the global `IpThrottlerGuard`, allowing the attacker to send unlimited requests from the server's perspective while the victim's email bucket remains exhausted.
