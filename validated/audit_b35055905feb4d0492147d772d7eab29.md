Audit Report

## Title
Empty `groupItems` Array Bypasses Validation in `POST /transaction-groups`, Allowing Creation of Empty Transaction Groups

## Summary
`CreateTransactionGroupDto` uses `@IsNotEmpty()` on the `groupItems` field. In class-validator, `@IsNotEmpty()` only rejects `null`, `undefined`, and `""` — an empty array `[]` passes silently. The service layer performs no secondary length check. Any authenticated organization user can POST `groupItems: []` directly to `POST /transaction-groups`, persisting an empty, permanently inaccessible transaction group to the database.

## Finding Description

**Wrong decorator on `groupItems`:**

`@IsNotEmpty()` is applied to `groupItems` in `CreateTransactionGroupDto`: [1](#0-0) 

`@IsNotEmpty()` calls class-validator's internal `isEmpty()`, which returns `true` only for `''`, `null`, and `undefined`. An empty array `[]` is not considered empty by this definition and passes the check. The correct decorator for "array must have at least one element" is `@ArrayMinSize(1)`, which is already used correctly in two other DTOs in the same codebase: [2](#0-1) [3](#0-2) 

**Service performs no secondary length check:**

`createTransactionGroup` maps directly over `dto.groupItems` with no guard. With an empty array, `transactionDtos` is `[]`, `createTransactions` is called with an empty list, and the group is saved to the database with zero items: [4](#0-3) 

**Controller entry point — no additional guard:** [5](#0-4) 

**Test spec confirms the gap:**

The DTO spec for `CreateTransactionGroupDto` explicitly tests the empty-array mapping case and asserts `groupItems.length === 0` passes through — but deliberately omits a `validateSync()` call: [6](#0-5) 

By contrast, the approver DTO spec explicitly asserts that an empty array fails validation: [7](#0-6) 

## Impact Explanation

An authenticated organization user can flood the organization's transaction group list with empty groups, corrupting shared state visible to all members. Additionally, any empty group created this way becomes permanently inaccessible: `getTransactionGroup` throws `UnauthorizedException` when `groupItems.length === 0` after retrieval, meaning the record cannot be viewed or managed after creation: [8](#0-7) 

This results in data integrity corruption and persistent orphaned records with no recovery path through the normal API.

**Severity: Medium** — data integrity corruption and persistent inaccessible records; no direct asset theft.

## Likelihood Explanation

Any verified, authenticated organization user can exploit this with a single crafted HTTP POST, bypassing the frontend guard trivially. No special privileges, leaked credentials, or complex setup are required — only a valid JWT session, which is the baseline for any organization member.

## Recommendation

Replace `@IsNotEmpty()` with `@ArrayMinSize(1)` on the `groupItems` field in `CreateTransactionGroupDto`:

```ts
// back-end/apps/api/src/transactions/dto/create-transaction-group.dto.ts
@IsArray()
@ArrayMinSize(1)          // replaces @IsNotEmpty()
@ValidateNested({ each: true })
@Type(() => CreateTransactionGroupItemDto)
groupItems: CreateTransactionGroupItemDto[];
```

Additionally, add a defensive length check at the top of `createTransactionGroup` in `TransactionGroupsService` as a belt-and-suspenders guard, and add a `validateSync()` assertion to the `CreateTransactionGroupDto` spec for the empty-array case (mirroring the pattern already used in `create-transaction-approver.dto.spec.ts`).

## Proof of Concept

```http
POST /transaction-groups HTTP/1.1
Host: <org-server>
Content-Type: application/json
Cookie: <valid-jwt-session>

{
  "description": "empty group",
  "atomic": false,
  "sequential": false,
  "groupItems": []
}
```

Expected (correct) behavior: `400 Bad Request` — `groupItems` must contain at least one element.

Actual behavior: `201 Created` — an empty `TransactionGroup` is persisted. Any subsequent `GET /transaction-groups/:id` for this group throws `401 Unauthorized` ("You don't have permission to view this group"), making it permanently inaccessible.
