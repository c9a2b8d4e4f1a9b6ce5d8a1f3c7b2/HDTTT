### Title
Missing `timeout-minutes` on All GitHub Actions Jobs Enables Indefinite Runner Exhaustion

### Summary
All four GitHub Actions workflow files in this repository define jobs without a `timeout-minutes` field. GitHub's default job timeout is **6 hours**. Because two of these workflows trigger on `pull_request` events (open to any contributor), a malicious or buggy PR can cause jobs to hang indefinitely, consuming self-hosted runner capacity for the full 6-hour window.

### Finding Description
Every job across all four workflow files omits `timeout-minutes`:

- `pr-formatting.yaml` — job `title-check`, no timeout [1](#0-0) 

- `publish-docker-images.yaml` — job `build-and-push-image`, no timeout [2](#0-1) 

- `test-backend.yaml` — job `test`, no timeout [3](#0-2) 

- `test-frontend.yaml` — jobs `build`, `test`, and `unit-test`, none have a timeout [4](#0-3) [5](#0-4) [6](#0-5) 

The `test-frontend.yaml` `test` job is the highest-risk case: it provisions a full Kind/Kubernetes cluster, deploys a Solo Hedera network, builds and starts a Docker Compose backend, and then runs Playwright E2E tests — all without any time bound. [7](#0-6) 

Both `test-backend.yaml` and `test-frontend.yaml` trigger on `pull_request` against `main` and `release/**`, meaning any external contributor can open a PR to trigger these jobs. [8](#0-7) [9](#0-8) 

All jobs run on the self-hosted runner label `transaction-tools-linux-medium`, so a hung job directly blocks that runner pool. [10](#0-9) 

**Note on permissions:** Unlike the external report's description, explicit `permissions` blocks *are* present in all four workflows, so the permissions half of the finding does not apply here. The sole valid analog is the missing timeout. [11](#0-10) 

### Impact Explanation
A PR that introduces a test, build step, or script that hangs (e.g., an infinite loop in a test file, a `sleep` call, or a network call to an unreachable host) will occupy the self-hosted runner for up to 6 hours per matrix shard. The `test-frontend.yaml` `test` job has a matrix of 2 shards, so a single malicious PR could consume 2 runner-slots × 6 hours = 12 runner-hours, effectively denying CI to all other contributors for that period.

### Likelihood Explanation
The trigger is `pull_request` — any GitHub user can open a PR against `main` or `release/**`. No maintainer approval is required to *start* the workflow (unlike `pull_request_target` with explicit approval gates). The `test-frontend.yaml` `test` job already contains long-running infrastructure setup steps (Kind cluster, Solo deploy, Docker Compose) that are natural hang points if any upstream service is slow or unavailable. [12](#0-11) 

### Recommendation
Add `timeout-minutes` to every job. Suggested values based on observed job complexity:

```yaml
# pr-formatting.yaml
jobs:
  title-check:
    timeout-minutes: 5

# publish-docker-images.yaml
jobs:
  build-and-push-image:
    timeout-minutes: 30

# test-backend.yaml
jobs:
  test:
    timeout-minutes: 20

# test-frontend.yaml
jobs:
  build:
    timeout-minutes: 20
  test:
    timeout-minutes: 60   # Solo + backend deploy + Playwright
  unit-test:
    timeout-minutes: 20
```

### Proof of Concept
1. Open a PR against `main` that modifies any file under `back-end/` or `front-end/`.
2. The `test-backend.yaml` `test` job and `test-frontend.yaml` `build`/`test`/`unit-test` jobs are triggered automatically.
3. If the PR introduces a test with `while(true) {}` or a script step with `sleep infinity`, the job runs for the full GitHub default of **360 minutes** before being killed.
4. During that window, the `transaction-tools-linux-medium` self-hosted runner is fully occupied, blocking all other CI runs on that runner pool. [5](#0-4) [13](#0-12)
