### Title
`cutoffAt` Deadline Is Stored But Never Enforced in Signature Upload Paths, Allowing Signatures After the Intended Cutoff

### Summary
The `Transaction` entity stores a `cutoffAt` field documented as "The timestamp at which the transaction can no longer be signed by the signers." However, neither `SignersService.validateTransactionStatus()` nor `TransactionsService.importSignatures()` checks this field before accepting a signature upload. Any authenticated signer can submit a valid signature after the `cutoffAt` deadline has passed, bypassing the intended access-control boundary entirely.

### Finding Description

**Root cause — `cutoffAt` is stored but never read during signature validation.**

The `cutoffAt` column is declared on the entity: [1](#0-0) 

It is accepted from the creator's DTO and persisted: [2](#0-1) 

The database documentation explicitly states its purpose: [3](#0-2) 

**Signature upload path 1 — `POST /transactions/:id/signers`**

`SignersService.validateTransactionStatus()` is the sole gate before signatures are accepted. It checks only the status enum and the Hedera SDK-level `isExpired()` (which reflects `validStart + transactionValidDuration`, typically 180 seconds). There is no `cutoffAt` check: [4](#0-3) 

**Signature upload path 2 — `POST /transactions/signatures/import`**

`TransactionsService.importSignatures()` performs the same two checks (status + `isExpired()`) and equally omits any `cutoffAt` comparison: [5](#0-4) 

**Exploit flow:**

1. Creator submits a transaction with `cutoffAt` set to time T (e.g., 23 hours before `validStart`).
2. After time T, a signer — who should be locked out — calls `POST /transactions/:id/signers` with a valid signature map.
3. `validateTransactionStatus()` passes: status is still `WAITING_FOR_SIGNATURES` and `isExpired()` returns `false` (the Hedera validity window has not yet opened).
4. The signature is merged into `transactionBytes` and persisted.
5. If this late signature tips the threshold, `processTransactionStatus()` promotes the transaction to `WAITING_FOR_EXECUTION`, and the chain service executes it — after the creator's intended signing deadline.

### Impact Explanation

The `cutoffAt` deadline is the only application-level mechanism for a transaction creator to close the signing window before the Hedera validity window opens. Bypassing it allows:

- A signer who was intentionally excluded after the cutoff to add their signature and potentially push the transaction over the required threshold.
- Transactions to be promoted to `WAITING_FOR_EXECUTION` and executed on-chain after the creator's intended deadline, violating the creator's expressed intent and the documented invariant.
- In multi-signature governance workflows (the primary use-case of this tool), this breaks the trust model: the cutoff is the mechanism by which an organization freezes the signer set before execution.

### Likelihood Explanation

Any authenticated user whose public key is registered on the transaction can reach `POST /transactions/:id/signers` with no elevated privilege. The only precondition is a valid JWT and a key that belongs to the transaction's required signer set. The `cutoffAt` field is optional (nullable), so only transactions where the creator explicitly set it are affected — but for those transactions the bypass is trivially reachable by any eligible signer after the deadline.

### Recommendation

Add a `cutoffAt` check inside `SignersService.validateTransactionStatus()` and `TransactionsService.importSignatures()` immediately after the status check:

```typescript
// In validateTransactionStatus() — signers.service.ts
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  return ErrorCodes.TC; // Transaction Cutoff (new error code)
}
```

Apply the same guard in `importSignatures()` in `transactions.service.ts` after the status check at line 539. Both paths must be patched; fixing only one leaves the other exploitable.

### Proof of Concept

1. Authenticate as user A (creator) and `POST /transactions` with a valid transaction body that includes `"cutoffAt": "<timestamp 1 minute from now>"`.
2. Wait for the cutoff to pass (> 1 minute).
3. Authenticate as user B (a registered signer on the transaction) and `POST /transactions/:id/signers` with a valid signature map for user B's key.
4. Observe HTTP 201 and the signer record persisted in the database — the cutoff was not enforced.
5. If user B's signature satisfies the threshold, the transaction status advances to `WAITING_FOR_EXECUTION` and will be submitted to the Hedera network at `validStart`.

    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

      return ErrorCodes.TE;
    }
