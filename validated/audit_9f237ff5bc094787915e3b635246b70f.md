### Title
Missing Cryptographic Signature Verification in `uploadSignatureMap` Allows False Signing Records and Transaction Corruption

### Summary
The `POST /transactions/:transactionId/signers` endpoint (`SignersService.uploadSignatureMaps`) accepts a signature map, verifies only that the submitted public key belongs to the authenticated user, and then calls `sdkTransaction.addSignature(publicKey, map)` without ever cryptographically verifying that the signature bytes are a valid signature over the transaction body. A registered user can submit a signature map containing their legitimate public key paired with arbitrary/invalid signature bytes, causing the server to record them as a signer, update the stored transaction bytes with the invalid signature, and potentially advance the transaction status — all without the user ever possessing the corresponding private key.

### Finding Description

**Root cause:** `SignersService.processTransactionSignatures` (lines 217–267 of `back-end/apps/api/src/transactions/signers/signers.service.ts`) performs only an ownership check — confirming the public key is in `userKeyMap` — before calling `sdkTransaction.addSignature(publicKey, map)`. The Hedera SDK's `addSignature` method appends signature bytes to the transaction without verifying their cryptographic validity; that verification is deferred to the network. No call to `publicKey.verify(bodyBytes, signature)` exists anywhere in this code path.

```
uploadSignatureMap (controller)
  → uploadSignatureMaps (service)
    → validateAndProcessSignatures
      → processTransactionSignatures
          for each publicKey in map:
            userKey = userKeyMap.get(raw)   // ownership check only
            if (!userKey) throw PNY
            sdkTransaction.addSignature(publicKey, map)  // NO crypto verify
            userKeys.push(userKey)          // recorded as signer
```

**Contrast with `importSignatures`** (`transactions.service.ts` lines 545–549), which explicitly calls `validateSignature(sdkTransaction, map)` — a function that calls `publicKey.verify(bodyBytes, signature)` and throws on failure — before calling `addSignature`. The two code paths are inconsistent.

**Exploit flow:**
1. Attacker is an authenticated user with a registered public key that is a required signer for transaction T.
2. Attacker constructs a `SignatureMap` containing their legitimate public key but with random/garbage bytes as the signature value.
3. Attacker submits `POST /transactions/T/signers` with this map.
4. Server: public key found in `userKeyMap` → ownership check passes.
5. `addSignature` appends the invalid bytes to the transaction → `isSameBytes` is `false`.
6. A `TransactionSigner` row is inserted for the attacker's key; the corrupted transaction bytes are persisted.
7. `processTransactionStatus` may advance the transaction to `WAITING_FOR_EXECUTION`.
8. When the chain service submits the transaction to Hedera, it fails with `INVALID_SIGNATURE`.

### Impact Explanation

- **False signing record**: The attacker is permanently recorded as having signed the transaction without possessing the private key. This corrupts the audit trail and the multi-signature workflow.
- **Transaction corruption**: The stored `transactionBytes` are overwritten with an invalid signature. Legitimate signers cannot undo this; the transaction bytes now carry a bad signature.
- **Transaction execution failure / DoS**: If the transaction advances to `WAITING_FOR_EXECUTION` with the corrupted bytes, the chain service will submit it to Hedera and receive `INVALID_SIGNATURE`. The transaction moves to `FAILED` status, permanently blocking execution of a potentially high-value Hedera operation (transfers, account updates, node operations).
- **Notification dismissal**: The `bulkUpdateNotificationReceivers` call marks the attacker's signing notification as read, suppressing reminders to legitimate signers.

### Likelihood Explanation

The attacker needs only:
- A valid account on the backend (any registered user).
- Their public key to be one of the required signers for the target transaction (a normal condition in multi-sig workflows).
- The ability to craft a `SignatureMap` with their public key and arbitrary bytes — trivially done with the Hedera SDK or by hand-crafting the JSON body accepted by `IsSignatureMap`.

No privileged access, no leaked credentials, and no cryptographic break are required. The attack is a single authenticated HTTP POST.

### Recommendation

Add cryptographic signature verification in `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, after ownership check:
const { data: _, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

Alternatively, inline the verification per key:
```typescript
const bodyBytes = /* extract body bytes for this node/tx */;
if (!publicKey.verify(bodyBytes, signatureBytes)) {
  throw new Error(ErrorCodes.ISNMPN);
}
```

The `validateSignature` utility already exists at `back-end/libs/common/src/utils/sdk/transaction.ts` lines 213–248 and is used correctly in `importSignatures`. Apply it consistently to `uploadSignatureMap`.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

```typescript
export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```
