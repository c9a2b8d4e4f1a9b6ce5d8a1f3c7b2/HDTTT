# Audit Report

## Title
Missing User Key Ownership Check in `importSignatures` Allows Injection of Unregistered Signatures into Transaction Bytes

## Summary
The `POST /transactions/signatures/import` endpoint in `transactions.controller.ts` does not enforce key ownership. Any authenticated user with transaction access can inject cryptographically valid signatures from keys they control — including keys never registered in the system — directly into the stored `transactionBytes`. The parallel signing path (`uploadSignatureMaps` in `signers.service.ts`) enforces this ownership check explicitly; `importSignatures` skips it entirely.

## Finding Description

**Ownership check in `uploadSignatureMaps` (present):**

In `signers.service.ts`, `validateAndProcessSignatures` builds a `userKeyMap` exclusively from `user.keys`: [1](#0-0) 

Then in `processTransactionSignatures`, if any public key in the submitted signature map is absent from `userKeyMap`, it throws `ErrorCodes.PNY`: [2](#0-1) 

**Ownership check in `importSignatures` (absent):**

The `POST /transactions/signatures/import` controller handler carries only the class-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). There is no `@UseGuards(HasKeyGuard)` and no `@OnlyOwnerKey` decorator on the endpoint: [3](#0-2) [4](#0-3) 

Compare this to `createTransaction`, which explicitly applies `@OnlyOwnerKey<CreateTransactionDto>('creatorKeyId')` and `@UseGuards(HasKeyGuard)`: [5](#0-4) 

The controller's own API description confirms the design gap: *"No signature entities will be created"* — meaning no `TransactionSigner` record is written, so the audit trail is bypassed by design on this path: [6](#0-5) 

## Impact Explanation

- **Audit trail bypass**: No `TransactionSigner` record is created for imported signatures, so the system's record of who signed is incomplete or incorrect.
- **Authorization model bypass**: A user who is only an observer (not a designated signer) can inject a valid signature from a key they control, potentially satisfying a required signing slot and advancing the transaction to `WAITING_FOR_EXECUTION` without being a registered signer.
- **State machine manipulation**: If the injected signature satisfies the threshold, `processTransactionStatus` may advance the transaction and it can be submitted to the Hedera network with a signature from an unregistered, untracked key, violating the multi-sig governance invariant the system is designed to enforce.

## Likelihood Explanation

- **Attacker preconditions**: Authenticated organization user with any access role on the target transaction (observer, approver, or signer). No privileged access required beyond a valid JWT.
- The attacker must control a key pair whose public key is in the transaction's required signing key list. This is realistic when the attacker is the owner of a Hedera account involved in the transaction (e.g., the fee payer or an account being updated).
- The endpoint is reachable via standard REST API with a valid JWT token. No special tooling is needed.

## Recommendation

Apply the same ownership enforcement used in `uploadSignatureMaps` to the `importSignatures` path:

1. Add `@UseGuards(HasKeyGuard)` to the `POST /transactions/signatures/import` endpoint in `transactions.controller.ts`.
2. Inside `transactionsService.importSignatures`, build a `userKeyMap` from `user.keys` (as done in `signers.service.ts` lines 162–165) and reject any public key in the submitted signature map that is absent from that map (as done in `signers.service.ts` line 248).
3. Consider whether `importSignatures` should also create `TransactionSigner` records for accountability, or document explicitly why it intentionally does not and what access controls compensate for that.

## Proof of Concept

1. Authenticate as any organization user (observer role is sufficient) and obtain a JWT.
2. Identify a target transaction in `WAITING_FOR_SIGNATURES` status.
3. Generate a fresh key pair locally (not registered in the system under any account).
4. Sign the raw `transactionBytes` with that key pair locally.
5. Submit:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <jwt>
   Content-Type: application/json

   [{ "id": <transactionId>, "signatureMap": <crafted_map> }]
   ```
6. `validateSignature` passes (signature is cryptographically valid). `verifyAccess` passes (user has transaction access). No ownership check is performed.
7. The signature is added to `transaction.transactionBytes` and persisted. No `TransactionSigner` record is created.
8. If the injected key satisfies a required signing slot, `processTransactionStatus` advances the transaction to `WAITING_FOR_EXECUTION`.
