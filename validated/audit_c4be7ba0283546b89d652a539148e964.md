Audit Report

## Title
Any Transaction Creator Can Remove Approvers From Arbitrary Transactions Due to Missing Cross-Reference Check

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `:transactionId` but deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who is the creator of at least one transaction can remove approvers from any other transaction in the system.

## Finding Description

**Root cause — authorization and operation are decoupled:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent steps: [1](#0-0) 

Step 1 (`getCreatorsTransaction`) verifies the caller owns `:transactionId`. Step 2 (`removeTransactionApprover(id)`) deletes the approver by `:id` with no check that the approver belongs to `:transactionId`.

The service method `removeTransactionApprover` only checks existence, never ownership relative to the authorized transaction: [2](#0-1) 

`approver.transactionId` (line 541) is the *victim* transaction's ID — it is never compared to the `transactionId` that was authorized. The notification fires against the victim transaction, confirming the operation targets the wrong transaction.

**Contrast with the correct pattern in `updateTransactionApprover`:**

The update path correctly resolves the root node of the approver and cross-checks it against the authorized transaction before proceeding: [3](#0-2) 

The delete path omits this entire cross-reference step entirely.

## Impact Explanation

An attacker who is the creator of Transaction A can:
1. Call `GET /transactions/:B/approvers` to enumerate approver IDs from any Transaction B they have visibility of.
2. Call `DELETE /transactions/:A/approvers/:approverB_id`.
3. The guard passes (attacker is creator of A); the service deletes the approver from Transaction B.

**Concrete consequences:**
- Required approvers are silently removed from victim transactions, allowing those transactions to proceed without the intended approval threshold — bypassing the multi-party authorization model entirely.
- An attacker can reduce a threshold-based approver tree to zero approvers, making a transaction auto-approvable or permanently broken.
- The notification emitted references `approver.transactionId` (the victim's transaction), leaving no audit trail pointing to the attacker's transaction.

## Likelihood Explanation

- **Attacker preconditions:** Must be an authenticated, verified organization user and the creator of at least one transaction — a normal user role with no elevated privileges.
- **Attack complexity:** Trivial — a single crafted HTTP DELETE request with a mismatched `:transactionId` and `:id`.
- **Discoverability:** Approver IDs are returned by `GET /transactions/:id/approvers`, accessible to any user with transaction visibility. IDs are sequential integers, making enumeration straightforward.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and assert that its `transactionId` matches the authorized `transactionId`. This is exactly the pattern already used in `updateTransactionApprover`:

```typescript
// After getCreatorsTransaction(transactionId, user):
const approver = await this.approversService.getTransactionApproverById(id);
if (!approver) throw new BadRequestException(ErrorCodes.ANF);

const rootNode = await this.approversService.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId)
  throw new UnauthorizedException('Approver does not belong to this transaction');

await this.approversService.removeTransactionApprover(id);
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and perform the check inside the service, consistent with how `updateTransactionApprover` handles it at [4](#0-3) .

## Proof of Concept

**Setup:**
- Attacker (User A) is the creator of Transaction 1 (ID: `1`).
- Victim (User B) is the creator of Transaction 2 (ID: `2`) with approver ID `99` assigned.

**Attack:**
```http
DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_jwt>
```

**Result:**
- `getCreatorsTransaction(1, userA)` passes — attacker owns Transaction 1.
- `removeTransactionApprover(99)` executes — approver `99` (belonging to Transaction 2) is soft-deleted.
- `emitTransactionStatusUpdate` fires against Transaction 2's ID.
- Transaction 2 loses approver `99` with no record of the attacker's involvement.

    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
