Audit Report

## Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated DoS

## Summary
`getTransactionsToSign` in the API back-end fetches every non-terminal transaction from the database with no `LIMIT`, then performs CPU-bound protobuf deserialization and async signature-key computation for each transaction in a sequential loop. Pagination is applied only in-memory after all work is done. Any authenticated, verified user can trigger this with a single request, causing memory exhaustion, CPU saturation, and database connection starvation proportional to the total number of active transactions.

## Finding Description

**Unbounded `repo.find()` — confirmed at lines 295–299:**

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

No `take` (LIMIT) is passed. Compare to `getTransactions` (lines 175–181) and `getHistoryTransactions` (lines 234–235), which both pass `skip: offset, take: limit`, and `getTransactionsToApprove` (lines 342–350), which also passes `skip: offset, take: limit`. `getTransactionsToSign` is the sole outlier. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

**`whereForUser` only excludes five terminal statuses** — every `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `NEW`, and `SIGN_ONLY` transaction in the entire system is loaded into the Node.js heap. [5](#0-4) 

**Sequential async loop over all N transactions — confirmed at lines 301–309:**

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
```

Per iteration, `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs:
1. `attachKeys(user, entityManager)` — a database query per call
2. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-bound protobuf deserialization
3. `transactionSignatureService.computeSignatureKey(transaction, showAll)` — async signature key computation [6](#0-5) [7](#0-6) [8](#0-7) 

**In-memory pagination only — confirmed at lines 311–316:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
```

The `limit`/`offset` parameters control only the final slice; they do not constrain the database fetch or the processing loop. [9](#0-8) 

**Endpoint is live and confirmed `/* NO LONGER USED BY FRONT-END */`**, protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard authenticated-user guards applied at the controller class level. [10](#0-9) [11](#0-10) 

**Rate limiting is insufficient.** The global `IpThrottlerGuard` is the only guard applied globally. Its limits are configured via `GLOBAL_MINUTE_LIMIT` and `GLOBAL_SECOND_LIMIT` environment variables. The default/production values are 10,000 req/min and 1,000 req/sec — far too high to prevent a single-request resource exhaustion attack. No per-user throttler is applied to the transactions controller. [12](#0-11) [13](#0-12) [14](#0-13) 

## Impact Explanation

A single authenticated user sending one request to `GET /transactions/sign?page=1&size=1` causes the server to execute a full PostgreSQL table scan of all active transactions, load the entire result set into the Node.js heap, and then perform O(N) sequential async operations (protobuf deserialization + signature key computation + DB lookups) before returning a single item. With thousands of active transactions — realistic in an enterprise multi-sig deployment — this exhausts Node.js heap memory, saturates the event loop, and starves the PostgreSQL connection pool (configured at `POSTGRES_MAX_POOL_SIZE=2` in production), making the API unavailable to all users. [15](#0-14) 

## Likelihood Explanation

The endpoint is reachable by any verified organization member with no special privileges. The comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is still live and registered. The IP-based rate limiter at 10,000 req/min does not prevent a single-request attack — one request is sufficient to trigger the full O(N) work. A malicious insider or a compromised member account can trigger this with a single HTTP request at negligible cost. [16](#0-15) 

## Recommendation

1. **Add database-level pagination** to `getTransactionsToSign` by passing `skip: offset, take: limit` to `repo.find()`, consistent with every other paginated query in the same service.
2. **Pre-filter at the database level** — push the user-key membership check into the SQL query (as `getTransactionsToApprove` does with its recursive CTE) so only transactions actually requiring the user's signature are fetched.
3. **Remove or disable the endpoint** if it is confirmed to be unused by the front-end, eliminating the attack surface entirely.
4. **Apply a per-user rate limiter** to the transactions controller to limit the blast radius even if the unbounded query is not immediately fixed.

## Proof of Concept

```

#     transactionSignatureService.computeSignatureKey(tx)
# - Returns: { totalItems: K, items: [1 item], page: 1, size: 1 }

export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
