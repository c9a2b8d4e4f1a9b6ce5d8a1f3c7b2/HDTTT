### Title
Any Verified User Can Post Comments on Any Transaction Without Ownership or Participation Check

### Summary
The `createComment` endpoint in `CommentsController` is intended to restrict comment creation to users who are participants of a given transaction (creator, signer, approver, or observer). However, no such check exists at the controller or service call level — any verified user can post a comment on any transaction by supplying an arbitrary `transactionId`. The developer explicitly acknowledged this gap with a `//TODO` comment in the source.

### Finding Description
**Vulnerability class:** Missing access control (authorization bypass on a state-mutating endpoint).

**Root cause:**

In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the `createComment` handler is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the controller level. These guards confirm the caller is a logged-in, verified user — but they do not verify any relationship between the caller and the target transaction. [1](#0-0) 

The developer's own inline comment at line 22 reads:

```
//TODO need some sort of guard or check to ensure user can comment here
``` [2](#0-1) 

**Exploit path:**

1. Attacker registers and is verified as a normal user (no admin privileges needed).
2. Attacker discovers or enumerates any valid `transactionId` (integer, sequential).
3. Attacker sends `POST /transactions/:transactionId/comments` with an arbitrary comment body.
4. The comment is persisted against the target transaction with no ownership or participation check.

Additionally, `getComments` and `getCommentById` carry the same absence of relationship enforcement, allowing any verified user to read comments on any transaction: [3](#0-2) 

### Impact Explanation
- Any verified user can inject arbitrary comments into any transaction in the system, including transactions belonging to other organizations or users they have no relationship with.
- This breaks the integrity of the transaction audit trail — comments are used to communicate context and decisions around sensitive financial transactions on the Hedera network.
- An attacker can flood any transaction's comment thread, obscuring legitimate communication between signers and approvers, disrupting the transaction approval workflow.
- Sensitive information disclosed in comments (e.g., signing rationale, key references) becomes readable by any verified user, not just transaction participants.

### Likelihood Explanation
- **Preconditions:** Attacker only needs a valid verified user account. No admin access, no leaked credentials, no privileged role required.
- **Transaction ID enumeration:** IDs are sequential integers (`ParseIntPipe`), trivially enumerable by iterating from 1 upward.
- **No rate-limit specific to this endpoint** is visible in the controller.
- The developer's own `//TODO` comment confirms this is a known, unresolved gap — not a theoretical concern.

### Recommendation
Add a service-level or guard-level check that verifies the requesting user is a participant of the target transaction (creator, signer, approver, or observer) before allowing comment creation or retrieval. For example:

```typescript
@Post()
async createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  await this.commentsService.assertUserIsParticipant(user, transactionId);
  return this.commentsService.createComment(user, transactionId, dto);
}
```

Apply the same participation check to `getComments` and `getCommentById`.

### Proof of Concept

**Setup:** Two accounts — `victim` (transaction creator) and `attacker` (unrelated verified user). A transaction with `id=42` exists and belongs to `victim`.

**Steps:**
1. Attacker logs in: `POST /auth/login` → receives `attackerToken`.
2. Attacker posts a comment on victim's transaction:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <attackerToken>
   Content-Type: application/json

   { "body": "Attacker-injected comment" }
   ```
3. **Expected (correct) behavior:** `403 Forbidden` — attacker is not a participant.
4. **Actual behavior:** `201 Created` — comment is persisted against transaction 42 with no relationship check.
5. Attacker can also read all comments: `GET /transactions/42/comments` → returns all comments including sensitive participant communications. [4](#0-3)
