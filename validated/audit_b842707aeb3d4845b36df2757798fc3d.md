Audit Report

## Title
Rate Limiter Bypass via Attacker-Controlled `x-forwarded-for` Header

## Summary
`IpThrottlerGuard`, registered as a global `APP_GUARD` on every API endpoint, derives its throttle key from the attacker-controlled `x-forwarded-for` HTTP request header without any validation. An unauthenticated attacker can rotate arbitrary fake IP values in this header to make every request appear to originate from a different IP, completely nullifying the global rate limiter.

## Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`**

In `back-end/apps/api/src/guards/ip-throttler.guard.ts`, line 8:

```typescript
const clientIp = req.headers['x-forwarded-for'] || req.ip;
```

The `x-forwarded-for` header is checked **before** `req.ip`. Because it is a plain HTTP header, any client can set it to any arbitrary string. The real socket IP (`req.ip`) is never consulted when the header is present. [1](#0-0) 

**Global application — every route is affected**

`IpThrottlerGuard` is registered as `APP_GUARD` in `back-end/apps/api/src/api.module.ts`, making it the first throttle check on every single endpoint in the API service. [2](#0-1) 

**Throttler configuration — limits stored per attacker-controlled key**

The configured limits (`GLOBAL_MINUTE_LIMIT`, `GLOBAL_SECOND_LIMIT`) are stored per-key in Redis. Because the attacker controls the key, those counters are never incremented for the same attacker across requests. [3](#0-2) 

**High-value endpoints exposed**

- `POST /auth/login` — protected by `LocalAuthGuard` + `EmailThrottlerGuard` at the route level. The global `IpThrottlerGuard` is bypassed. The `EmailThrottlerGuard` still throttles per email address (not per IP), so brute-force against a single account is still limited by email-keyed throttling — but the global IP-based rate limit is fully bypassed. [4](#0-3) 

- `POST /auth/reset-password` — unauthenticated endpoint protected only by `EmailThrottlerGuard` at the route level. The global `IpThrottlerGuard` is bypassed. The `EmailThrottlerGuard` tracks by `req.body.email`, so it still limits OTP requests per email address. However, an attacker can still send OTP emails to **many different victim email addresses** without any IP-based throttling. [5](#0-4) 

**Note on `EmailThrottlerGuard`**: It tracks by email address (`req.body.email`), not by IP. [6](#0-5) 
This means the `EmailThrottlerGuard` is **not** subject to the same IP-header bypass — it is a separate, email-keyed throttle. The IP bypass does not nullify the per-email throttle on `reset-password` for a single target email. However, the global IP rate limit is still fully bypassed for all endpoints.

## Impact Explanation

- **Global rate limit bypass**: The `IpThrottlerGuard` global guard is completely nullified for any attacker who sets `x-forwarded-for`. This removes the server-wide request rate limit for all endpoints.
- **Credential stuffing / brute-force on `/auth/login`**: The global IP throttle is bypassed. The per-email `EmailThrottlerGuard` still limits attempts per email address, but the global IP-based defense is gone. An attacker targeting many accounts (credential stuffing) faces no IP-level throttling.
- **Email bombing on `/auth/reset-password`**: An attacker can send OTP emails to many different registered email addresses without any IP-level throttling. The per-email throttle still limits requests to a single target email, but mass-targeting of many accounts is unrestricted.

## Likelihood Explanation

- **Attacker preconditions**: None. No account, no token, no privileged access required.
- **Technique**: Setting an HTTP header is trivial with any HTTP client (`curl`, Python `requests`, etc.).
- **Detection difficulty**: Rotating fake IPs in `x-forwarded-for` produces no repeated-IP signal in logs, making automated detection harder.
- **Well-known class**: This is a documented and commonly attempted rate-limiter bypass technique.

## Recommendation

1. **Do not trust `x-forwarded-for` directly.** Only use it if the application is deployed behind a trusted reverse proxy (e.g., nginx, AWS ALB), and even then, extract only the **first** IP in the comma-separated list (the leftmost, which is the original client IP as set by the proxy — not the attacker-appended value).
2. **Prefer `req.ip`** (Express's built-in, which respects the `trust proxy` setting) over raw header access. Configure Express's `trust proxy` setting appropriately for your deployment topology.
3. **Alternatively**, if the app is always behind a trusted proxy, configure NestJS/Express with `app.set('trust proxy', 1)` and use `req.ip` exclusively — Express will then correctly resolve the real client IP from `x-forwarded-for` using only the trusted proxy hop.

## Proof of Concept

```bash
# Bypass global IP rate limit by rotating x-forwarded-for on every request
for i in $(seq 1 1000); do
  curl -s -X POST https://target/auth/reset-password \
    -H "Content-Type: application/json" \
    -H "x-forwarded-for: 10.0.0.$i" \
    -H "x-frontend-version: 0.21.0" \
    -d '{"email": "victim1@example.com"}'
done

# For credential stuffing: rotate both x-forwarded-for and target email
for i in $(seq 1 1000); do
  curl -s -X POST https://target/auth/login \
    -H "Content-Type: application/json" \
    -H "x-forwarded-for: 10.0.0.$i" \
    -H "x-frontend-version: 0.21.0" \
    -d "{\"email\": \"victim${i}@example.com\", \"password\": \"Password123\"}"
done
```

Each request uses a unique fake IP as the throttle key in Redis, so the global rate limit counter is never incremented for the same attacker. The `EmailThrottlerGuard` per-email limit is still active for single-email targeting, but the global IP-based defense is fully bypassed.
