## Audit Report

## Title
`removeTransactionApprover` Does Not Re-Validate Parent Threshold After Child Deletion, Enabling Permanent Approval Deadlock

## Summary
`removeTransactionApprover` in `approvers.service.ts` soft-deletes a node and all its descendants via `removeNode` without adjusting the parent threshold node. This leaves the parent in a state where `threshold > remaining_children`, making the transaction permanently unapprovable. The sibling path `updateTransactionApprover` (with `listId: null`) correctly performs this adjustment, confirming the omission is a defect rather than a design choice.

## Finding Description

**Root cause — constraint enforced at creation and in `updateTransactionApprover`, but not in `removeTransactionApprover`.**

At creation time, `createTransactionApprovers` enforces `threshold ≤ approvers.length`: [1](#0-0) 

When a child is detached via `updateTransactionApprover` (`listId: null`), the parent threshold is correctly re-evaluated: [2](#0-1) 

However, `removeTransactionApprover` takes a completely different path — it fetches the approver and calls `removeNode`, with **no parent threshold adjustment**: [3](#0-2) 

`removeNode` issues only a recursive SQL soft-delete and never touches the parent row: [4](#0-3) 

The controller exposes this endpoint to any transaction creator with no status gate and no check that the target node is a root: [5](#0-4) 

`getTransactionApproverById` loads only child relations (`approvers`), not the parent, so the service has no visibility into the parent's threshold at deletion time: [6](#0-5) 

**Secondary defect:** When a child node is deleted, `approver.transactionId` is `null` (child nodes store `transactionId: null` by design), so the `emitTransactionStatusUpdate` call fires with `entityId: null`, meaning the status-update notification for the affected transaction is silently dropped. [7](#0-6) 

## Impact Explanation

After a child leaf node is deleted, the parent threshold node retains its original `threshold` value (e.g., `2`) while having only one remaining child. The `isApproved` utility confirms the deadlock: [8](#0-7) 

With `threshold=2` and one remaining child who approves, `approvals.length (1) >= threshold (2)` evaluates to `false`. The threshold node's `approved` field is never set to `true`, the transaction status never advances past `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`, and the state is **unrecoverable** — no approver action can satisfy the corrupted constraint.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user exercising a documented API endpoint (`DELETE /transactions/:txId/approvers/:id`). No elevated privileges, leaked credentials, or external dependencies are required. The only precondition is setting up a threshold approver tree, which is a standard workflow. The endpoint imposes no transaction-status gate, so the attack is executable at any point while the transaction is pending.

## Recommendation

In `removeTransactionApprover`, after calling `removeNode`, fetch the deleted node's parent (via `listId`) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If the parent has no remaining children after deletion, soft-delete the parent.
- If `remaining_children < parent.threshold`, update `parent.threshold = remaining_children`.

This logic should be wrapped in a database transaction to ensure atomicity. Additionally, the controller should verify that the approver being deleted belongs to the supplied `transactionId` before proceeding.

## Proof of Concept

1. Creator creates a transaction and adds a threshold approver tree: `threshold=2`, children = `[User A (id=10), User B (id=11)]`.
2. User A calls `POST /transactions/:txId/approvers/approve` and approves. The parent threshold node now has 1 of 2 approvals.
3. Creator calls `DELETE /transactions/:txId/approvers/10` (User A's leaf node).
4. `removeNode(10)` soft-deletes User A's record. The parent threshold node still has `threshold=2` but now has only 1 child (User B).
5. User B calls `POST /transactions/:txId/approvers/approve` and approves.
6. `isApproved` evaluates `approvals.length (1) >= threshold (2)` → `false`. The threshold node's `approved` field is never set.
7. The transaction is permanently stuck — it can never reach the approved state.

  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,

  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
