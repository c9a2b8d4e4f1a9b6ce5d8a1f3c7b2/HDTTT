Audit Report

## Title
Transaction Creator Can Lower Approval Threshold After Approvals Are Cast, Bypassing Multi-Party Approval Requirement

## Summary
`getCreatorsTransaction` — the sole authorization gate for the approver mutation endpoints — checks only creator identity and never inspects `transaction.status`. A malicious creator can wait for a subset of approvers to sign, then lower the threshold (via `PATCH /transactions/{id}/approvers/{treeId}`) or remove unapproved approvers (via `DELETE /transactions/{id}/approvers/{id}`) to match the already-collected approvals, causing the transaction to advance to `WAITING_FOR_EXECUTION` without the originally required quorum.

## Finding Description

**`getCreatorsTransaction` — no status guard**

`getCreatorsTransaction` only verifies `transaction.creatorKey?.userId !== user.id`. It returns the transaction regardless of its current status (`NEW`, `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, etc.). [1](#0-0) 

**`updateTransactionApprover` — threshold written unconditionally**

After calling `getCreatorsTransaction` at line 394, the function proceeds to write the new threshold value with no guard on the transaction's current status. The only validation is that the new threshold does not exceed the current child count — it can freely be lowered to 1. [2](#0-1) [3](#0-2) 

**`removeTransactionApprover` — no status check, triggers immediate re-evaluation**

The controller calls `getCreatorsTransaction` (creator-only check, no status check) and then hard-removes the node. After removal, `emitTransactionStatusUpdate` is called, which triggers the chain service to immediately re-evaluate whether the remaining approvals satisfy the (now possibly auto-reduced) threshold. [4](#0-3) [5](#0-4) 

**Automatic threshold reduction on child detach**

When a child approver is detached (`listId` set to `null`), the parent's threshold is silently reduced to `newParentApproversLength` if it would otherwise exceed the remaining child count. Removing a non-approving child can automatically bring the threshold down to the number of already-approved children. [6](#0-5) 

**Contrast: `approveTransaction` does check status**

The approval submission endpoint correctly restricts itself to `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION`. The structural mutation endpoints have no equivalent guard. [7](#0-6) 

## Impact Explanation

The approval tree is the only multi-party gate between a transaction being created and it being submitted to the Hedera network. Bypassing it lets the creator execute any transaction — including account key rotations, HBAR transfers, or file updates — with fewer approvals than the organization configured. This is an unauthorized state change and privilege escalation: the creator unilaterally satisfies a quorum that was never actually reached.

## Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges beyond owning the transaction. The exploit requires only sequential API calls with no race condition, no cryptographic break, and no external dependency. Any creator who wants to bypass an approval requirement they themselves set up can do so deterministically.

## Recommendation

Add a status guard at the top of both `updateTransactionApprover` and `removeTransactionApprover` (and in `createTransactionApprovers`) that rejects mutations when the transaction is not in `NEW` or `REJECTED` status. The simplest fix is to extend `getCreatorsTransaction` to accept an optional set of allowed statuses and throw if the transaction's current status is not in that set — mirroring the guard already present in `approveTransaction`.

## Proof of Concept

1. Creator creates a transaction with an approval tree: root node (threshold = 3) with children [Alice, Bob, Carol].
2. Alice and Bob approve — two approvals are now recorded in the DB.
3. Transaction is in `WAITING_FOR_SIGNATURES` (threshold not yet met).
4. Creator sends:
   ```
   PATCH /transactions/{id}/approvers/{rootNodeId}
   { "threshold": 2 }
   ```
   `updateTransactionApprover` calls `getCreatorsTransaction` (passes — creator identity confirmed, no status check), then writes `threshold = 2` to the DB.
5. The next `emitTransactionStatusUpdate` event (e.g., triggered by the scheduler every 10 seconds, or by the creator also calling `DELETE` on Carol's approver node which immediately calls `emitTransactionStatusUpdate`) causes the chain service to re-evaluate: 2 approvals collected ≥ threshold 2 → transaction advances to `WAITING_FOR_EXECUTION`.
6. Transaction is submitted to Hedera with only 2 of the originally required 3 approvals.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
