# Audit Report

## Title
Approver Rejection Does Not Gate Transaction Execution — Approval System Bypassed in Execution Path

## Summary
The organizational approver system records approvals and rejections in the `transaction_approver` table, but neither the status-transition function (`processTransactionStatus`) nor the execution-time validator (`getValidatedSDKTransaction`) ever consults approver state. A transaction with pending or explicitly rejected approvals is executed on the Hedera network as soon as the required cryptographic signatures are collected, making the entire approver workflow a no-op with respect to actual fund movement.

## Finding Description

**Root Cause 1 — `processTransactionStatus` ignores approver state**

In `back-end/libs/common/src/utils/transaction/index.ts`, the function that promotes a transaction from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` checks only cryptographic signatures: [1](#0-0) 

There is no query against `transaction_approver.approved`. A transaction whose approvers have all set `approved = false` will still be promoted to `WAITING_FOR_EXECUTION` the moment signatures are sufficient.

**Root Cause 2 — `getValidatedSDKTransaction` ignores approver state**

In `back-end/libs/common/src/execute/execute.service.ts`, at execution time the service validates only the cryptographic signature key: [2](#0-1) 

No approval check is performed before `sdkTransaction.execute(client)` is called.

**Root Cause 3 — `approveTransaction` never sets `Transaction.status` to `REJECTED`**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, when an approver rejects (`dto.approved = false`), the service updates only the `TransactionApprover` row and emits a notification; the parent `Transaction.status` is never changed to `REJECTED`: [3](#0-2) 

The `validateTransactionStatus` switch in `execute.service.ts` does guard against `REJECTED` status at line 239 — but that status is never written, so the guard is never reached. [4](#0-3) 

**Execution pipeline — no approver check at any stage**

The scheduler's `prepareTransactions` → `collateAndExecute` → `addExecutionTimeout` → `executeTransaction` chain contains no approver state check at any point: [5](#0-4) 

## Impact Explanation

The approver system is presented to users as a mandatory organizational gate (the UI renders "Approve transaction" / "Reject transaction" actions and the `ApproverChoiceDto` has an `approved` boolean). In practice it provides zero enforcement: any transaction that collects the required cryptographic signatures will be executed on the Hedera network regardless of approver decisions. This allows a transaction creator to move funds without the oversight the approver system is supposed to guarantee. Treasury or governance workflows that rely on the approver feature for multi-party authorization are entirely unprotected.

## Likelihood Explanation

Any authenticated user who can create a transaction and whose transaction accumulates the required Hedera cryptographic signatures can trigger this. No privileged access is required beyond being a registered organization user. The approver API endpoint (`POST /transactions/:transactionId/approvers/approve`) is reachable by any registered organization user. The scenario is realistic in any deployment where the approver feature is used for treasury or governance oversight. [6](#0-5) 

## Recommendation

1. **Set `Transaction.status = REJECTED` on rejection.** In `approveTransaction`, after persisting `approved: false`, evaluate whether the rejection makes approval impossible (e.g., a mandatory single approver rejected, or enough members of a threshold group rejected to make the threshold unreachable). If so, update `Transaction.status` to `REJECTED` within the same database transaction.

2. **Gate `processTransactionStatus` on approver state.** Before promoting a transaction to `WAITING_FOR_EXECUTION`, query `transaction_approver` for the transaction and verify that no required approver has `approved = false` and that all required approvers have `approved = true`. Only promote if both signature sufficiency and approval sufficiency are met.

3. **Gate `getValidatedSDKTransaction` on approver state.** As a defense-in-depth measure, add an approver check inside `getValidatedSDKTransaction` (or `validateTransactionStatus`) that throws if any required approver has rejected or if approval is still pending.

## Proof of Concept

1. Alice (creator) submits a `TransferTransaction` moving HBAR from a multi-sig treasury account.
2. Alice adds Bob as a mandatory approver via `POST /transactions/:id/approvers`.
3. Bob calls `POST /transactions/:id/approvers/approve` with `approved: false` — rejection is recorded in `transaction_approver` (`approved = false`) but `Transaction.status` stays `WAITING_FOR_SIGNATURES`.
4. The required Hedera signers upload their cryptographic signatures via the signers endpoint.
5. `processTransactionStatus` (called by the chain scheduler every 10 seconds) sees valid signatures, sets status to `WAITING_FOR_EXECUTION` — no approval check performed. [7](#0-6) 
6. The scheduler's `prepareTransactions` picks up the `WAITING_FOR_EXECUTION` transaction and calls `collateAndExecute` → `addExecutionTimeout` → `executeService.executeTransaction`.
7. `getValidatedSDKTransaction` validates only the cryptographic signature key and calls `sdkTransaction.execute(client)`. [8](#0-7) 
8. HBAR is transferred on-chain despite Bob's explicit rejection.

  private async getValidatedSDKTransaction(
    transaction: Transaction,
  ): Promise<SDKTransaction> {
    /* Throws an error if the transaction is not found or in incorrect state */
    if (!transaction) throw new Error('Transaction not found');

    await this.validateTransactionStatus(transaction);

    /* Gets the SDK transaction from the transaction body */
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Gets the signature key */
    const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);

    /* Checks if the transaction has valid signatureKey */
    if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
      throw new Error('Transaction has invalid signature.');

    return sdkTransaction;
  }

  async prepareTransactions(transactions: Transaction[]) {
    const processedGroupIds = new Set<number>();

    for (const transaction of transactions) {
      const waitingForExecution = transaction.status === TransactionStatus.WAITING_FOR_EXECUTION;

      if (waitingForExecution && this.isValidStartExecutable(transaction.validStart)) {
        if (transaction.groupItem && (transaction.groupItem.group.atomic || transaction.groupItem.group.sequential)) {
          if (!processedGroupIds.has(transaction.groupItem.groupId)) {
            processedGroupIds.add(transaction.groupItem.groupId);
            // Now that we are sure this transaction group needs to be processed together, get it
            // and being the processing
            const transactionGroup = await this.transactionGroupRepo.findOne({
              where: { id: transaction.groupItem.groupId },
              relations: {
                groupItems: {
                  transaction: true,
                },
              },
              order: {
                groupItems: {
                  transaction: {
                    validStart: 'ASC',
                  },
                },
              },
            });
            // All the transactions for the group are now pulled. If there is an issue validating for even one
            // transaction, the group will not be executed. This is handled in executeTransactionGroup
            this.collateGroupAndExecute(transactionGroup);
          }
        } else {
          this.collateAndExecute(transaction);
