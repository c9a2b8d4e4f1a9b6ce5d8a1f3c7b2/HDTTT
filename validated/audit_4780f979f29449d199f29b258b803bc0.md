### Title
Any Authenticated User Can Enumerate All Organization Members' Cryptographic Key Metadata via Unscoped `GET /user-keys`

### Summary
The `GET /user-keys` endpoint in `UserKeysAllController` is accessible to any authenticated, verified user but returns all `UserKey` records from the database with no per-user filtering. This allows any organization member to enumerate every other member's public keys, mnemonic hashes, and key derivation indices — sensitive cryptographic metadata that the rest of the codebase explicitly restricts on a per-user basis.

### Finding Description

**Root cause — missing user-scoping in `getUserKeys`:**

`UserKeysAllController.getUserKeys` calls the service without passing the authenticated user: [1](#0-0) 

The service implementation performs an unfiltered `findAndCount` with no `WHERE userId = ?` clause: [2](#0-1) 

Compare this to `getUserKeysRestricted`, which the codebase uses elsewhere and explicitly gates `mnemonicHash` and `index` behind an ownership check (`user.id === userId`): [3](#0-2) 

The controller is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard authenticated-user guards, not an `AdminGuard` or any ownership check: [4](#0-3) 

**Exploit path:**
1. Attacker registers or is invited as a normal organization user (no admin role required).
2. Attacker sends `GET /user-keys?page=1&limit=100` with a valid JWT.
3. The response contains every `UserKey` row in the database — including `publicKey`, `mnemonicHash`, and `index` for all users.
4. Attacker paginates through all pages to harvest the full key registry.

### Impact Explanation

- **`publicKey`** — reveals every user's Hedera public key, enabling targeted phishing or key-correlation attacks.
- **`mnemonicHash`** — a hash of the user's BIP-39 mnemonic. If the hash function is weak or the mnemonic space is small, this enables offline brute-force to recover the mnemonic and derive the private key.
- **`index`** — reveals the BIP-44 derivation path index, narrowing the key-derivation search space for any brute-force attempt.

The design intent is clear: `getUserKeysRestricted` hides `mnemonicHash` and `index` from non-owners. The unscoped `getUserKeys` path bypasses this entirely, exposing the full record set to any authenticated peer.

### Likelihood Explanation

- **Attacker precondition:** valid JWT for any organization account (normal user, no admin).
- **Effort:** a single authenticated HTTP GET request with pagination.
- **Discoverability:** the endpoint is documented via Swagger (`@ApiTags('User Keys All')`), making it trivially discoverable. [5](#0-4) 

### Recommendation

Replace the unscoped `getUserKeys` call with the user-scoped variant. Pass the authenticated user into the service and filter by `userId`:

```typescript
// In UserKeysAllController.getUserKeys:
@Get()
@Serialize(withPaginatedResponse(UserKeyDto))
getUserKeys(
  @GetUser() user: User,          // inject authenticated user
  @PaginationParams() paginationParams: Pagination,
): Promise<PaginatedResourceDto<UserKey>> {
  return this.userKeysService.getUserKeysByUser(user, paginationParams);
}
```

Add a new service method that mirrors `getUserKeysRestricted` but with pagination, filtering results to `userId = user.id` and applying the same field-level restrictions (`mnemonicHash`/`index` only for owner).

Alternatively, if this endpoint is intended for admin use only, add `@UseGuards(AdminGuard)` to the controller and document it accordingly.

### Proof of Concept

```
# Step 1: Login as any normal (non-admin) organization user
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Enumerate all user keys in the organization
GET /user-keys?page=1&limit=100
Authorization: Bearer <JWT>

# Response: full UserKey records for ALL users, including:
# { id, userId, publicKey, mnemonicHash, index, ... }
# Repeat with page=2, page=3, ... to harvest all records.
```

The response will include `mnemonicHash` and `index` for every user — fields that `getUserKeysRestricted` explicitly hides from non-owners — because `getUserKeys` issues a bare `findAndCount` with no user filter. [6](#0-5)
