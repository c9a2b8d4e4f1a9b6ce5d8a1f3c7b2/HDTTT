Audit Report

## Title
Unbounded Loop Over All Non-Terminal Transactions in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

## Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction from the database with no row limit, then iterates over the full result set performing expensive per-transaction operations (byte deserialization, async signature-key computation, and mirror-node cache calls) before applying pagination. As the transaction table grows — which any authenticated user can influence — each call to this endpoint consumes proportionally more CPU and memory, enabling a sustained resource-exhaustion DoS.

## Finding Description

**Unbounded DB fetch:**

`getTransactionsToSign` issues a `repo.find` with no `take` (limit) clause. The `whereForUser` filter only excludes terminal statuses; it does **not** scope results to the requesting user, meaning the query returns **all** non-terminal transactions across all users. [1](#0-0) 

**Full-table loop before pagination:**

Every returned row is processed by `userKeysToSign` before any slicing occurs. [2](#0-1) 

Pagination is applied only after the loop completes: [3](#0-2) 

**Per-iteration cost:**

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `keysRequiredToSign`. That function deserializes raw transaction bytes via `SDKTransaction.fromBytes` and then calls `transactionSignatureService.computeSignatureKey`: [4](#0-3) 

`computeSignatureKey` makes one or more async calls to `accountCacheService.getAccountInfoForTransaction` (mirror-node backed) for the fee payer, signing accounts, and receiver accounts: [5](#0-4) 

**Endpoint is reachable by any verified user:**

`GET /transactions/sign` is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. The comment "NO LONGER USED BY FRONT-END" does not remove the route. [6](#0-5) 

**Permissive rate limits:** [7](#0-6) 

`GLOBAL_MINUTE_LIMIT=10000` and `GLOBAL_SECOND_LIMIT=1000` give an attacker ample request budget. The PostgreSQL pool is constrained to `POSTGRES_MAX_POOL_SIZE=2`, making connection exhaustion a secondary impact vector. [8](#0-7) 

## Impact Explanation

Every call to `GET /transactions/sign` forces the server to load and process the entire non-terminal transaction table (scoped only by status, not by user). With N active transactions, the server performs N deserialization operations and up to N×(accounts per transaction) async mirror-node lookups per request. This saturates the Node.js event loop and the PostgreSQL connection pool, causing request timeouts and API unavailability for all users. The unbounded memory allocation for the full result set also makes OOM termination a realistic outcome in constrained deployments.

## Likelihood Explanation

- **Attacker precondition**: A valid, verified account — the lowest privilege level in the system.
- **Inflation vector**: Any verified user can create transactions via `POST /transactions`, inflating the table that `getTransactionsToSign` scans with no per-user scope.
- **Trigger**: Repeatedly calling `GET /transactions/sign` with any pagination parameters. Server-side cost is independent of the requested page size.
- **No special knowledge required**: Standard REST endpoint, documented in the Swagger API.

## Recommendation

1. **Push pagination into the database query**: Add `skip: offset, take: limit` to the `repo.find` call in `getTransactionsToSign`, mirroring the pattern already used in `getTransactions` and `getHistoryTransactions`.
2. **Pre-filter by user at the DB level**: Scope the query to transactions where the user is a signer, creator, or observer before fetching, so the result set is bounded by user activity rather than global table size.
3. **Enforce a hard server-side maximum page size**: Cap `limit` to a reasonable value (e.g., 50–100) regardless of client input.
4. **Consider removing or gating the endpoint**: The "NO LONGER USED BY FRONT-END" comment suggests this endpoint may be a candidate for removal or stricter access control.

## Proof of Concept

```
# 1. Register and verify a low-privilege account
# 2. Create thousands of transactions via POST /transactions
# 3. Repeatedly call the endpoint:
for i in $(seq 1 100); do
  curl -s -H "Authorization: Bearer <token>" \
    "https://<host>/transactions/sign?page=1&limit=10" &
done
wait
# Result: server CPU/memory spikes; legitimate requests time out
```

Each concurrent request triggers a full-table scan and N×M mirror-node lookups. With the default `POSTGRES_MAX_POOL_SIZE=2`, the database connection pool is exhausted after just two concurrent requests reach the query phase, blocking all other database operations across the API.

```typescript
  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
