Audit Report

## Title
Any Authenticated User Can Post Comments on Arbitrary Transactions Due to Missing Authorization Check

## Summary
`CommentsController.createComment` enforces only authentication (JWT + verified user) but performs no check that the requesting user has any relationship to the target transaction. Any authenticated, verified user can `POST /transactions/<id>/comments` against any transaction ID in the system.

## Finding Description

**Root cause:**

`CommentsController` applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level — these confirm only that the caller holds a valid, non-blacklisted, verified JWT. [1](#0-0) 

The `createComment` handler carries a developer-acknowledged `//TODO need some sort of guard or check to ensure user can comment here` comment, confirming the authorization boundary is intentionally absent and unresolved. [2](#0-1) 

`CommentsService.createComment` performs no participant lookup. It directly creates and persists the comment record with the supplied `transactionId` and the authenticated user, with no check against the transaction's creator, signers, observers, or approvers. [3](#0-2) 

A `verifyAccess` method exists in `TransactionsService` that correctly checks creator, signer, observer, and approver membership, but it is never invoked from the comments code path. [4](#0-3) 

**Exploit path:**

1. Attacker registers as a normal user and obtains a valid JWT via the standard sign-up/login flow.
2. Attacker enumerates or guesses any integer `transactionId` (IDs are sequential auto-incremented integers).
3. Attacker sends `POST /transactions/<any_id>/comments` with a valid JWT and arbitrary comment body.
4. `CommentsService.createComment` persists the comment to the database, associated with that transaction — no participant check is performed at any layer.

## Impact Explanation

- **Unauthorized persistent state modification**: Any authenticated user can inject comment records into the audit trail of any transaction in the organization. The SECURITY.md explicitly states "persistent plain text injection" is **not** excluded from scope.
- **Workflow integrity degradation**: Signers and approvers rely on the comment thread for coordination. Injected comments from unauthorized users corrupt this channel, potentially causing confusion or misdirection during time-sensitive signing rounds.
- **Audit trail unreliability**: The comment table accumulates records that violate the intended trust model (only participants should comment), making forensic audit trails unreliable.

## Likelihood Explanation

- **Preconditions**: Only a valid, verified user account is required — the lowest privilege level in the system.
- **Effort**: A single authenticated HTTP `POST` request with a known or guessed integer transaction ID.
- **Transaction ID discoverability**: IDs are sequential integers. A user who legitimately participates in even one transaction can infer the ID range and probe others.
- **No rate-limit or anomaly detection** is present on this endpoint beyond standard authentication guards.

## Recommendation

Call `TransactionsService.verifyAccess` (or an equivalent participant check) inside `CommentsService.createComment` before persisting the comment. The logic should mirror the existing check used elsewhere in the codebase: [5](#0-4) 

Alternatively, introduce a dedicated guard (as the TODO suggests) that loads the transaction and verifies the requesting user is a creator, signer, observer, or approver before the handler is invoked.

## Proof of Concept

```
# 1. Obtain a JWT for any verified user
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# 2. Post a comment on an arbitrary transaction (e.g., id=42)
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "message": "Injected comment on transaction 42" }

# 3. Expected (incorrect) result: HTTP 201, comment persisted
# CommentsService.createComment saves the record with no participant check.
# The comment now appears in the audit trail of transaction 42.
```

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
