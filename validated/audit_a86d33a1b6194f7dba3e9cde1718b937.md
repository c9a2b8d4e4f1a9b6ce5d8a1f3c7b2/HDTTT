Audit Report

## Title
Cross-Transaction Approver Deletion via IDOR in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId` but never validates that approver `:id` belongs to that transaction. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system.

## Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the delete handler performs two independent operations:

```typescript
// lines 108–109
await this.approversService.getCreatorsTransaction(transactionId, user);
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the caller is the creator of the URL-supplied `transactionId` — it says nothing about the approver `id`. [2](#0-1) 

`removeTransactionApprover` receives only the approver `id`, fetches it by primary key, and soft-deletes it with no ownership or transaction-membership check: [3](#0-2) 

By contrast, `updateTransactionApprover` correctly resolves the root node and rejects the request if `rootNode.transactionId !== transactionId`: [4](#0-3) 

This guard is entirely absent in the delete path.

## Impact Explanation

Removing an approver eliminates that party's required approval from the multi-signature workflow. If the deleted approver was the sole or threshold-completing approver, the transaction can proceed to execution without the intended authorization. This directly undermines the integrity of the organization-mode approval model — the core security guarantee of the product.

## Likelihood Explanation

Any authenticated, verified user who has created at least one transaction can exploit this. No elevated privileges are required. Approver IDs are sequential integers and are disclosed through `getVerifiedApproversByTransactionId` to any transaction participant. The attack requires only a valid JWT and a target approver ID, both realistic for a malicious organization member.

## Recommendation

In `removeTransactionApprover` (or in the controller before calling it), resolve the root node of the approver being deleted and assert that `rootNode.transactionId === transactionId`, mirroring the guard already present in `updateTransactionApprover`:

```typescript
const rootNode = await this.getRootNodeFromNode(id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and perform the check inside the service method, keeping authorization logic co-located with the deletion logic.

## Proof of Concept

1. Attacker registers as a normal user and creates **Transaction A** (becomes its creator, obtaining `transactionId_A`).
2. Attacker obtains a victim approver ID (`victim_approver_id`) from **Transaction B** via `GET /transactions/{transactionId_B}/approvers` (accessible to any participant) or by brute-forcing sequential integers.
3. Attacker sends:
   ```
   DELETE /transactions/{transactionId_A}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction(transactionId_A, attacker)` passes — attacker is the creator of Transaction A.
5. `removeTransactionApprover(victim_approver_id)` executes with no further checks, soft-deleting the victim's approver record from Transaction B.
6. Transaction B's approval threshold is now unmet or bypassed, allowing it to proceed without the deleted approver's authorization.
