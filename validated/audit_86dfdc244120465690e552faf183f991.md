Audit Report

## Title
Cross-Transaction Approver Deletion via Missing Ownership Binding in `removeTransactionApprover`

## Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `:transactionId` but deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who owns at least one transaction can silently delete approvers from any other transaction in the system.

## Finding Description

**Root cause — decoupled authorization and deletion target**

`ApproversController.removeTransactionApprover` performs two independent, unlinked operations:

```typescript
// approvers.controller.ts lines 108-109
await this.approversService.getCreatorsTransaction(transactionId, user); // checks caller owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no binding
``` [1](#0-0) 

`ApproversService.removeTransactionApprover` accepts only the approver `id`, fetches the approver record, and immediately calls `removeNode` — there is no assertion that `approver.transactionId === transactionId`:

```typescript
// approvers.service.ts lines 534-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // no ownership check
  ...
}
``` [2](#0-1) 

`getCreatorsTransaction` only validates that the caller created the URL-supplied `transactionId`; it returns that transaction and the result is never compared against the approver's actual `transactionId`: [3](#0-2) 

`removeNode` executes a recursive SQL soft-delete on the approver tree rooted at the supplied `id` with no transaction ownership guard: [4](#0-3) 

**Exploit path**

1. Attacker (User A) creates their own transaction → receives `transactionId = 7`.
2. Attacker learns approver `id = 42` belonging to victim's `transactionId = 3` (sequential integer IDs; attacker may also be an observer/signer on transaction 3 and can enumerate via `GET /transactions/3/approvers`).
3. Attacker sends:
   ```
   DELETE /transactions/7/approvers/42
   Authorization: Bearer <attacker_jwt>
   ```
4. `getCreatorsTransaction(7, attacker)` passes — attacker owns transaction 7.
5. `removeTransactionApprover(42)` fetches approver 42 (belonging to transaction 3) and calls `removeNode(42)`, soft-deleting approver 42 and its entire child subtree from transaction 3 with no further check.

## Impact Explanation
Removing an approver eliminates a required approval gate. In a threshold-based approval tree, deleting enough approvers can reduce the required threshold below the remaining count, or remove the entire tree, allowing a transaction to proceed to execution without the intended multi-party consent. This directly undermines the core security invariant of the platform (HIP-1300 multi-signature workflows) and can result in unauthorized Hedera network transactions being submitted on behalf of an organization.

## Likelihood Explanation
- **Preconditions**: The attacker must be an authenticated, verified user (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard` are all satisfied by any normal account) and must own at least one transaction of their own.
- **Approver ID discovery**: Approver IDs are sequential database integers. An attacker who is a signer, observer, or approver on any shared transaction can call `GET /transactions/:id/approvers` to enumerate IDs. Even without that, sequential IDs are trivially brute-forced.
- **No rate limiting or anomaly detection** is present in the controller path.

This is a realistic attack for any registered user in a multi-user organization deployment.

## Recommendation
In `ApproversController.removeTransactionApprover`, after fetching the approver (or as part of `removeTransactionApprover`), assert that the approver's root transaction matches the URL-supplied `transactionId` before proceeding with deletion. Concretely, in `ApproversService.removeTransactionApprover`, accept `transactionId` as a second parameter and verify:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id);
if (rootNode?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

This mirrors the existing guard already present in `createTransactionApprovers` and `updateTransactionApprover`, where `getRootNodeFromNode` is used to verify `root?.transactionId !== transactionId` before any mutation. [5](#0-4) 

## Proof of Concept

```
# Step 1: Attacker authenticates and creates their own transaction (receives transactionId=7)
POST /transactions
Authorization: Bearer <attacker_jwt>
{ ... }

# Step 2: Attacker enumerates approvers on victim's transaction (if they have any access)
GET /transactions/3/approvers
Authorization: Bearer <attacker_jwt>
# Response includes approver with id=42

# Step 3: Attacker deletes victim's approver using their own transactionId as the auth anchor
DELETE /transactions/7/approvers/42
Authorization: Bearer <attacker_jwt>

# Result: HTTP 200 true
# Approver id=42 (and its entire child subtree) is soft-deleted from transaction 3,
# bypassing the multi-signature approval requirement on a transaction the attacker does not own.
```

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

    // notifyTransactionAction(this.notificationsService);
  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
