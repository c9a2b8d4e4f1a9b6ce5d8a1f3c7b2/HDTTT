Audit Report

## Title
Missing Ownership Check in `removeTransactionGroup` Allows Any Authenticated User to Corrupt Another User's Transaction Group State

## Summary
`removeTransactionGroup` in `TransactionGroupsService` performs no ownership or authorization check before mutating database state. It unconditionally deletes `TransactionGroupItem` records before delegating to `removeTransaction`, which does check ownership. Because the item deletion precedes the authorization check in each loop iteration, an attacker can progressively strip group items from any group they did not create, corrupting the group's structural integrity without ever successfully deleting the underlying transactions.

## Finding Description

**Root cause ŌĆö missing pre-authorization in `removeTransactionGroup`:**

`removeTransactionGroup` fetches the target group with no user filter:

```typescript
const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
``` [1](#0-0) 

It then fetches all group items with no user filter:

```typescript
const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
  relations: { group: true },
  where: { group: { id: group.id } },
});
``` [2](#0-1) 

Inside the loop, it **deletes the `TransactionGroupItem` first**, then calls `removeTransaction` (which does check ownership):

```typescript
for (const groupItem of groupItems) {
  const transactionId = groupItem.transactionId;
  await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // ŌåÉ no auth, executes unconditionally
  await this.transactionsService.removeTransaction(transactionId, user, false); // ŌåÉ throws if not creator
}
``` [3](#0-2) 

`removeTransaction` calls `getTransactionForCreator`, which throws `UnauthorizedException` when the caller is not the transaction creator:

```typescript
if (transaction.creatorKey?.userId !== user?.id) {
  throw new UnauthorizedException('Only the creator has access to this transaction');
}
``` [4](#0-3) 

By the time this exception is thrown, the `TransactionGroupItem` for that iteration has already been permanently deleted. The `TransactionGroup` itself is not removed (line 193 is never reached), but one group item is gone per request.

**Contrast with `cancelTransactionGroup`**, which correctly gates on user-scoped data retrieval and an explicit ownership assertion before any mutation:

```typescript
const group = await this.getTransactionGroup(user, groupId, false); // user-filtered query
const allOwnedByUser = group.groupItems.every(
  item => item.transaction?.creatorKey?.userId === user.id,
);
if (!allOwnedByUser) {
  throw new UnauthorizedException('Only the creator can cancel all transactions in a group.');
}
``` [5](#0-4) 

`getTransactionGroup` uses `getTransactionGroupItemsQuery` with a user-scoped `WHERE` clause that filters by creator, signer, observer, and approver roles:

```typescript
const query = getTransactionGroupItemsQuery(this.sqlBuilder, id, user);
``` [6](#0-5) 

`getTransactionGroupItemsQuery` passes `{ signer: true, creator: true, observer: true, approver: true }` to the eligibility filter: [7](#0-6) 

`removeTransactionGroup` uses none of this ŌĆö it bypasses the user-scoped query entirely.

The `DELETE /transaction-groups/:id` endpoint is protected only by JWT authentication and email verification ŌĆö no role or ownership guard: [8](#0-7) [9](#0-8) 

## Impact Explanation

- **Unauthorized state mutation**: Any authenticated user can destroy the `TransactionGroupItem` membership records of any group, regardless of ownership.
- **Broken atomic/sequential group execution**: The scheduler relies on `groupItems` being intact to collate and execute atomic or sequential groups. Removing items breaks this invariant, causing the scheduler to execute an incomplete set or skip execution entirely.
- **Orphaned transactions**: The underlying `Transaction` records survive but lose their group context, making coordinated multi-transaction workflows (e.g., FileUpdate + FileAppend) impossible to execute as a unit.
- **Severity**: High ŌĆö unauthorized persistent state corruption of other users' transaction groups.

## Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (registered, email-verified organization user). No admin, no leaked secrets, no special role.
- **Group ID discovery**: `TransactionGroup.id` is a sequential auto-increment integer. An attacker can enumerate IDs by observing their own group IDs or by probing the `GET /transaction-groups/:id` endpoint (which does apply user filtering, but the integer range is trivially guessable).
- **Repeatability**: Each `DELETE` call removes exactly one group item before failing. The attacker simply loops N times to fully strip a group with N items.
- **No rate-limit bypass needed**: The operation is cheap and the endpoint is standard REST.

## Recommendation

Add an ownership check at the start of `removeTransactionGroup`, before any mutation occurs. The simplest correct fix mirrors `cancelTransactionGroup`: call `this.getTransactionGroup(user, id, false)` first, which uses the user-scoped `getTransactionGroupItemsQuery` and throws `UnauthorizedException` if the user has no access. Then additionally assert that the user is the creator of all transactions in the group before proceeding with any deletions. All mutations should occur only after authorization is fully established.

## Proof of Concept

1. Attacker (authenticated, email-verified user) creates their own group to observe sequential group IDs (e.g., their group is ID 10, so victim's group is likely ID 9 or 11).
2. Attacker sends `DELETE /transaction-groups/9` (victim's group ID).
3. Server executes line 189: `this.dataSource.manager.remove(TransactionGroupItem, groupItem)` ŌĆö first group item is permanently deleted from the database.
4. Server executes line 190: `this.transactionsService.removeTransaction(transactionId, user, false)` ŌĆö `getTransactionForCreator` finds the attacker is not the creator and throws `UnauthorizedException`.
5. The HTTP response is a 401 error, but the `TransactionGroupItem` deletion at step 3 is already committed.
6. Attacker repeats steps 2ŌĆō5 N times for a group with N items.
7. All `TransactionGroupItem` records are deleted; the `TransactionGroup` shell and the underlying `Transaction` records remain, but the group is structurally empty and broken. Any scheduler or workflow depending on the group's items will find an empty set.
