Audit Report

## Title
Lost-Update Race Condition in `SignersService.uploadSignatureMaps` Silently Discards Signatures

## Summary
`uploadSignatureMaps()` reads `transactionBytes` into an in-memory snapshot outside any database transaction or distributed lock, processes signatures, then blindly overwrites the stored bytes with no optimistic or pessimistic locking. Two concurrent HTTP requests for the same transaction will each read the same pre-signature bytes, independently append their own signature, and the second write will silently overwrite the first — permanently losing a signer's contribution from the stored transaction bytes.

## Finding Description

The three-phase flow in `uploadSignatureMaps` is:

**Phase 1 — Snapshot read (outside DB transaction)**

`loadTransactionData()` reads the current `transactionBytes` and existing signers into memory using a plain `dataSource.manager.find()` with no row-level lock. [1](#0-0) 

**Phase 2 — Concurrent in-memory processing**

`validateAndProcessSignatures()` fans out with `Promise.all()`, each branch calling `processTransactionSignatures()` which deserializes `transaction.transactionBytes` from the same in-memory snapshot and calls `sdkTransaction.addSignature()` on it. [2](#0-1) [3](#0-2) 

**Phase 3 — Blind overwrite**

`bulkUpdateTransactions()` issues an unconditional `UPDATE transaction SET "transactionBytes" = CASE id ... END WHERE id = ANY(...)` with no version check, no `WHERE "transactionBytes" = $old`, and no `SELECT FOR UPDATE` guard. [4](#0-3) 

**No `@MurLock` protects `uploadSignatureMaps`**, in contrast to `executeTransaction` and `executeTransactionGroup` which are explicitly guarded with a distributed Redis lock: [5](#0-4) [6](#0-5) 

## Impact Explanation

The `transactionBytes` blob is the authoritative source for which cryptographic signatures are embedded in the Hedera transaction. The `TransactionSigner` table records (inserted via `bulkInsertSigners`) are metadata only — the actual signature bytes that will be submitted on-chain come exclusively from `transactionBytes`. When Request B's write overwrites Request A's write, the DB will show both users as signers in `transaction_signer`, but the stored `transactionBytes` will only contain B's signature. When the transaction is eventually submitted to the Hedera network, it will be missing A's signature and will be rejected with `INVALID_SIGNATURE`. For threshold-key or `KeyList` transactions — the primary use case of this tool — a missing signature means the threshold is never reached. There is no automatic recovery: the lost signer must re-submit, but if the transaction's `validStart` window has expired, the transaction is unrecoverable. [7](#0-6) 

## Likelihood Explanation

The `POST /transactions/:transactionId/signers` endpoint is the normal path for every organization signer. Multi-signature workflows are the core purpose of the tool; concurrent signing by multiple users is the expected, documented workflow. The race window spans the entire async signature-processing time between `loadTransactionData()` and the DB write — easily tens to hundreds of milliseconds in a real deployment. No special attacker capability is required; two legitimate signers acting simultaneously trigger the condition. [8](#0-7) 

## Recommendation

Apply one of the following mitigations:

1. **Distributed lock (consistent with existing pattern):** Add `@MurLock(15000, 'dto[0].id.toString()')` (or per-transaction-id locking) to `uploadSignatureMaps`, matching the pattern already used on `executeTransaction` and `executeTransactionGroup`. [5](#0-4) 

2. **Optimistic locking:** Change `bulkUpdateTransactions` to include a `WHERE "transactionBytes" = $old_bytes` clause and retry on conflict.

3. **Pessimistic locking:** Move the `loadTransactionData` read inside the `dataSource.transaction()` block and use `SELECT ... FOR UPDATE` to lock the row before reading, preventing concurrent reads from seeing the same pre-signature state. [9](#0-8) 

## Proof of Concept

```
Time →
Request A (User A signs TX #1)          Request B (User B signs TX #1)
──────────────────────────────────────────────────────────────────────
loadTransactionData()
  → transactionBytes = B0 (no sigs)
                                         loadTransactionData()
                                           → transactionBytes = B0 (no sigs)

processTransactionSignatures()
  → sdkTx = fromBytes(B0)
  → sdkTx.addSignature(pubKeyA, mapA)
  → newBytes = B1 = B0 + sig_A
                                         processTransactionSignatures()
                                           → sdkTx = fromBytes(B0)
                                           → sdkTx.addSignature(pubKeyB, mapB)
                                           → newBytes = B2 = B0 + sig_B

bulkUpdateTransactions(B1)
  → DB: transactionBytes = B1 ✓
                                         bulkUpdateTransactions(B2)
                                           → DB: transactionBytes = B2 ✗
                                             (overwrites B1; sig_A is gone)

Final DB state:
  transaction_signer: [userA_keyId, userB_keyId]  ← both recorded
  transactionBytes:   B2 = B0 + sig_B only         ← sig_A permanently lost

On-chain submission → INVALID_SIGNATURE (threshold not met)
```

The race is triggered by two legitimate users signing the same transaction within the same async processing window, with no special privileges required.

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
