Audit Report

## Title
Missing `@Min(1)` Constraint on `threshold` in `CreateTransactionApproverDto` Allows Negative Threshold to Bypass Multi-Signature Approval Requirement

## Summary
`CreateTransactionApproverDto.threshold` accepts any integer, including negative values, because it lacks a `@Min(1)` decorator. The service-level guard only rejects `threshold === 0` and `threshold > approvers.length`. A negative value (e.g., `-1`) passes all validation, is persisted to the database, and causes the approval-tree evaluation to treat the node as permanently satisfied — effectively bypassing the multi-signature approval gate.

## Finding Description

**Root cause — DTO missing lower-bound constraint**

`CreateTransactionApproverDto` decorates `threshold` with only `@IsNumber()` and `@IsOptional()`: [1](#0-0) 

`UpdateTransactionApproverDto` correctly adds `@Min(1)` on the same field, demonstrating the fix was known but not applied to the create path: [2](#0-1) 

**Service-level guard does not catch negative values**

Inside `createTransactionApprovers`, the only threshold range check is: [3](#0-2) 

For `threshold = -1` with two approvers: `-1 > 2` is `false` and `-1 === 0` is `false`, so neither branch fires and execution continues.

The private `validateApprover` helper also fails to catch it: [4](#0-3) 

The condition `approver.threshold === null || isNaN(approver.threshold) || approver.threshold === 0` evaluates to `false` for `-1`, so the "empty approver" guard is not triggered either.

**Negative threshold is persisted**

The assignment that stores the threshold uses a JavaScript truthiness check: [5](#0-4) 

In JavaScript, `-1` is truthy, so `dtoApprover.threshold && dtoApprover.approvers` evaluates to the approvers array (truthy), and the negative value is written to `transaction_approver.threshold`.

The `threshold` column on the entity has no database-level constraint either: [6](#0-5) 

**Approval evaluation with negative threshold**

The `isApproved` function used to evaluate approval status computes: [7](#0-6) 

With `threshold = -1` and zero approvals: `-1` is truthy in JavaScript, so `approver.threshold || approvals.length` resolves to `-1`. The condition `0 >= -1` is immediately `true`, and the node is considered fully approved with zero actual approvals.

## Impact Explanation
Any authenticated user who creates a transaction can unilaterally bypass the multi-signature approval requirement. Transactions that should require N-of-M approvals from designated approvers can be treated as approved with zero signatures. This is a direct integrity failure in the trust model: the approval gate — the primary organizational control over transaction execution — is rendered ineffective for any transaction whose creator supplies a crafted payload.

## Likelihood Explanation
The attack requires only a valid authenticated session and the ability to create a transaction (a normal user capability). The crafted payload is a single integer field set to a negative number. No privileged access, leaked secrets, or special tooling are needed. Any user who can call `POST /transactions/:id/approvers` can trigger this.

## Recommendation
Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring what is already done in `UpdateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
import { IsArray, IsNumber, IsOptional, Min, ValidateNested, ArrayMinSize } from 'class-validator';

@IsNumber()
@Min(1)
@IsOptional()
threshold?: number;
```

Additionally, add a `CHECK (threshold >= 1)` constraint at the database level on the `transaction_approver.threshold` column to enforce the invariant independently of application logic.

## Proof of Concept

```http
POST /transactions/:id/approvers
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 2 },
        { "userId": 3 }
      ]
    }
  ]
}
```

1. `class-validator` passes — no `@Min` constraint on `CreateTransactionApproverDto.threshold`.
2. `validateApprover` passes — `-1` is not `null`, not `NaN`, not `0`.
3. Range guard passes — `-1 > 2` is `false`; `-1 === 0` is `false`.
4. Truthiness check `dtoApprover.threshold && dtoApprover.approvers` evaluates to the approvers array (truthy), so `threshold = -1` is written to the database.
5. `isApproved` evaluates `approvals.length >= (approver.threshold || approvals.length)` → `0 >= -1` → `true`. The approval node is immediately satisfied with zero approvals.
