### Title
Missing enforcement of `cutoffAt` deadline check in signature upload and approval flows allows post-deadline signing

### Summary
The `Transaction` entity defines a `cutoffAt` field representing the deadline by which signatures must be submitted. However, neither `SignersService.uploadSignatureMaps()`, `ApproversService.approveTransaction()`, nor `TransactionsService.importSignatures()` enforces this deadline against the current time. Signers and approvers can submit signatures after the creator-specified cutoff, violating the creator's intent and allowing transactions to be executed with signatures collected beyond the intended deadline.

### Finding Description

The `Transaction` entity declares `cutoffAt` as a nullable `Date` column representing the signing deadline: [1](#0-0) 

This value is accepted from the creator at transaction creation time and persisted: [2](#0-1) 

However, in `SignersService.validateTransactionStatus()` — the gating function called before every signature upload — only the status enum and SDK-level expiry are checked. There is no comparison of `transaction.cutoffAt` against `Date.now()`: [3](#0-2) 

The same omission exists in `ApproversService.approveTransaction()`. After verifying the approver identity and transaction status, the method proceeds to record the approval signature without ever checking `cutoffAt`: [4](#0-3) 

And in `TransactionsService.importSignatures()`, the same pattern: status and SDK expiry are checked, but `cutoffAt` is not: [5](#0-4) 

**Exploit path:**
1. Creator creates a transaction with `cutoffAt` set to a past or near-future timestamp, intending to freeze signature collection at that point.
2. After `cutoffAt` passes, a signer (or approver) calls `POST /transactions/:id/signers` (or the approver endpoint) with a valid signature map.
3. `validateTransactionStatus` passes because the transaction status is still `WAITING_FOR_SIGNATURES` and the Hedera SDK-level `validStart + validDuration` window has not yet expired.
4. The signature is accepted and persisted, potentially pushing the transaction into `WAITING_FOR_EXECUTION` and triggering automatic submission — all after the creator's intended deadline.

### Impact Explanation
**Medium**: The creator's `cutoffAt` deadline is silently ignored. A transaction intended to be frozen at a specific time can still accumulate signatures and be executed after that deadline. For time-sensitive multi-sig workflows (e.g., governance votes, time-bounded fund transfers), this breaks the creator's explicit intent and the integrity of the signing window. The transaction may be submitted to the Hedera network with signatures that should not have been accepted.

### Likelihood Explanation
**Medium**: The `cutoffAt` field is optional (nullable), so this only affects transactions where the creator explicitly sets a deadline. Any authenticated signer or approver with a valid key can trigger this after the deadline passes, with no privileged access required beyond normal participation in the transaction.

### Recommendation
Add a `cutoffAt` guard in all three signature-acceptance paths:

In `SignersService.validateTransactionStatus()`:
```typescript
private validateTransactionStatus(transaction: Transaction): string | null {
  if (
    transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
    transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
  ) {
    return ErrorCodes.TNRS;
  }

  // Enforce cutoffAt deadline
  if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
    return ErrorCodes.TE; // or a dedicated ErrorCodes.CUTOFF_PASSED
  }

  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
  if (isExpired(sdkTransaction)) {
    return ErrorCodes.TE;
  }

  return null;
}
```

Apply the same guard in `ApproversService.approveTransaction()` after the status check at line 584, and in `TransactionsService.importSignatures()` after the status check at line 535.

### Proof of Concept

1. Authenticated user A creates a transaction via `POST /transactions` with `cutoffAt` set to `Date.now() + 5000` (5 seconds from now).
2. Wait 6 seconds so that `cutoffAt` is in the past.
3. Authenticated user B (a designated signer) calls `POST /transactions/:id/signers` with a valid `signatureMap`.
4. **Expected**: Request rejected with a deadline-passed error.
5. **Actual**: `validateTransactionStatus` passes (status is `WAITING_FOR_SIGNATURES`, SDK expiry has not elapsed), signature is persisted, and if threshold is now met, the transaction transitions to `WAITING_FOR_EXECUTION` and is submitted to the Hedera network — after the creator's intended cutoff.   private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
