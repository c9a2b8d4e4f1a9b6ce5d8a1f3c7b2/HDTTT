### Title
Creator Can Archive a Manual Transaction From Any Status, Including Terminal States

### Summary

The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a flawed guard condition. Because the status-range check and the `isManual` flag are combined with a logical AND (`&&`), setting `isManual: true` on a transaction completely bypasses the status check. Any authenticated user who is the creator of a manual transaction can therefore call `PATCH /transactions/archive/:id` and force the status to `ARCHIVED` regardless of the transaction's current state — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED`.

### Finding Description

**Root cause — `archiveTransaction`, lines 711–718:**

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← entire guard is short-circuited when isManual = true
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard is supposed to reject archive requests when the transaction is not in an archiveable state. However, the condition only throws when **both** sub-expressions are true:

| `status` not in allowed list | `!isManual` | throws? |
|---|---|---|
| true | true (non-manual) | ✓ |
| true | false (manual) | **✗ — bypassed** |
| false | true | ✗ |
| false | false | ✗ |

When `isManual` is `true`, the second operand is `false`, so the entire `&&` expression evaluates to `false` and the guard never fires. The unconditional update on line 720 then runs:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

**Exploit path:**

1. Attacker (any authenticated, verified user) creates a transaction with `isManual: true` via `POST /transactions`.
2. The transaction proceeds through the normal lifecycle and reaches `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`.
3. Attacker calls `PATCH /transactions/archive/:id`.
4. `getTransactionForCreator` passes (attacker is the creator).
5. The guard is bypassed because `isManual = true`.
6. Status is overwritten to `ARCHIVED`.

Any authenticated user can create a manual transaction, so no elevated privilege is required.

### Impact Explanation

- **Audit-trail corruption:** Terminal states (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) represent the definitive outcome of a Hedera network transaction. Overwriting them with `ARCHIVED` permanently hides the true outcome from all other participants (signers, observers, approvers) who rely on the stored status.
- **Integrity failure in multi-signature workflows:** In an organizational context, signers and observers depend on accurate status to understand whether a transaction succeeded or failed. A creator can conceal a failed or expired transaction by archiving it, undermining the trust model of the multi-party signing workflow.
- **Irreversible in normal operation:** There is no API endpoint to un-archive a transaction; the corruption persists unless an administrator directly modifies the database.

### Likelihood Explanation

- **Preconditions:** The attacker must be an authenticated, verified user and the creator of a manual transaction. Both conditions are reachable by any normal user — manual transactions are a standard product feature (`isManual: true` in the create payload).
- **No special knowledge required:** The flaw is triggered by a single `PATCH` request to a documented endpoint.
- **Realistic scenario:** A user whose transaction failed or expired could archive it to hide the failure from co-signers or auditors.

### Recommendation

Separate the two concerns in the guard. The status check should always apply; `isManual` should only expand the set of archiveable statuses if that is the intended design, not bypass the check entirely:

```typescript
// Option A: isManual transactions are archiveable from the same statuses only
const archiveableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];
if (!archiveableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}

// Option B: if isManual is meant to allow a wider set, enumerate it explicitly
const archiveableStatuses = transaction.isManual
  ? [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION]
  : [TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION];
if (!archiveableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

### Proof of Concept

```
# 1. Authenticate as any verified user and obtain a JWT token.

# 2. Create a manual transaction.
POST /transactions
Authorization: Bearer <token>
{ ..., "isManual": true }
→ { "id": 42, "status": "WAITING FOR SIGNATURES", "isManual": true }

# 3. (Simulate or wait for) the transaction to reach EXECUTED status.
#    In a test environment this can be forced via a direct DB update.

# 4. Archive the executed transaction.
PATCH /transactions/archive/42
Authorization: Bearer <token>
→ 200 OK

# 5. Verify the status has been overwritten.
GET /transactions/42
→ { "id": 42, "status": "ARCHIVED" }   ← was EXECUTED
```

```typescript
  /* Archive the transaction if the transaction is sign only. */
  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
```
