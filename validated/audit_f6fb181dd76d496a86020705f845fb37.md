### Title
`archiveTransaction` Status Guard Bypassed for Manual Transactions, Enabling Terminal-State Corruption

### Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` uses a logically flawed guard that completely skips the status check for any transaction where `isManual = true`. This allows a transaction creator to move their own manual transactions from any terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `REJECTED`) into `ARCHIVED`, corrupting the audit trail and enabling re-creation of transactions whose `transactionId` should be permanently retired.

### Finding Description

**Root cause — flawed boolean guard:** [1](#0-0) 

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← short-circuits the entire status check
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** conditions are true:
1. Status is outside `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
2. `isManual` is `false`

When `isManual = true`, condition 2 is always `false`, so the compound `&&` is always `false` and the guard **never throws**, regardless of the current status. The function then unconditionally writes `ARCHIVED` to the database: [2](#0-1) 

**Cancellation only updates `status`, never `isManual`:** [3](#0-2) 

So after cancellation, `isManual` remains `true`, and the creator can immediately call `archiveTransaction` on the now-`CANCELED` transaction.

**`ARCHIVED` is treated as an inactive status, unlocking `transactionId` re-use:** [4](#0-3) [5](#0-4) 

Once a transaction is `ARCHIVED`, `createTransactions` no longer considers its `transactionId` as "taken", so a new transaction record can be inserted with the same Hedera `transactionId`.

**Exploit flow:**

1. Creator submits a manual transaction (`isManual = true`). Signers sign it. It reaches `WAITING_FOR_EXECUTION`.
2. Creator calls `PATCH /transactions/execute/:id` → transaction is submitted to Hedera → chain service sets status to `EXECUTED`.
3. Creator calls `PATCH /transactions/archive/:id` → guard passes because `isManual = true` → status is overwritten to `ARCHIVED`.
4. The `EXECUTED` record is now invisible to any query that filters out `ARCHIVED`; the audit trail shows the transaction was "archived" rather than executed.
5. Because `ARCHIVED` is in `inactiveStatuses`, the creator can now `POST /transactions` with the same Hedera `transactionId` string, creating a fresh record that appears active.

### Impact Explanation

- **Audit trail integrity**: A creator can silently reclassify any of their own manual transactions from `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, or `REJECTED` to `ARCHIVED`. Downstream reporting, compliance views, and the history endpoint (`getHistoryTransactions`) will reflect the wrong terminal state.
