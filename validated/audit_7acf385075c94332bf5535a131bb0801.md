Audit Report

## Title
TOCTOU Race Condition in `approveTransaction` Allows Approver to Submit Conflicting Votes

## Summary
`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` performs its "already voted" guard check outside any database transaction. Two concurrent HTTP requests from the same authenticated approver can both pass the guard simultaneously, allowing both writes to execute — with the last commit winning — bypassing the one-vote-per-approver invariant.

## Finding Description

**Root cause — check-then-act is not atomic:**

The guard at line 563 reads approver state from outside the DB transaction:

```
Line 553: const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);
Line 556: const userApprovers = approvers.filter(a => a.userId === user.id);
Line 563: if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

The actual mutation at lines 599–610 is wrapped in a DB transaction, but the guard is not part of it and there is no `SELECT FOR UPDATE`, optimistic lock, or re-check inside the transaction:

```
Line 599: await this.dataSource.transaction(async transactionalEntityManager => {
Line 600:   await transactionalEntityManager
Line 601:     .createQueryBuilder()
Line 602:     .update(TransactionApprover)
Line 603:     .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
Line 608:     .whereInIds(userApprovers.map(a => a.id))
Line 609:     .execute();
Line 610: });
```

The `TransactionApprover` entity has only non-unique indexes on `transactionId` and `userId` — no unique constraint on `(transactionId, userId)` exists to prevent both writes from succeeding at the DB level. [1](#0-0) [2](#0-1) [3](#0-2) 

**Exploit path:**

1. Attacker is a legitimate approver of transaction T (no privilege beyond being an approver).
2. Attacker sends two concurrent HTTP requests to `POST /transactions/T/approvers/approve`:
   - Request A: `{ approved: true, signature: sigA, userKeyId: K }`
   - Request B: `{ approved: false, signature: sigB, userKeyId: K }`
3. Both execute `getVerifiedApproversByTransactionId` before either completes — both read `userApprovers` with `signature = null`.
4. Both pass the guard at line 563 (`userApprovers.every(a => a.signature)` is `false` for both).
5. Both proceed through signature verification (lines 565–596) — a wide async window.
6. Both reach the DB update. Both execute successfully. Last commit wins.
7. Attacker controls the final `approved` value by controlling which request completes last. [4](#0-3) [5](#0-4) 

**Secondary impact — stale notification snapshot:**

Line 614 evaluates `userApprovers.every(a => a.approved)` using the pre-update snapshot, so status-update events may be emitted based on stale state, corrupting downstream notification logic. [6](#0-5) 

## Impact Explanation

An approver can submit two conflicting votes simultaneously and have the last write determine the final `approved` value. This corrupts the integrity of the multi-signature approval workflow:

- A transaction that legitimately reached its approval threshold can be retroactively rejected by an approver flipping their vote via the race.
- A transaction that was rejected can be retroactively approved.
- Notification events are emitted based on stale pre-update state, further corrupting downstream consumers. [7](#0-6) 

## Likelihood Explanation

- **Attacker preconditions**: Must be a registered, verified user listed as an approver of the target transaction. No admin or privileged access is required.
- **Exploit complexity**: Sending two concurrent HTTP requests is trivial (`Promise.all` in JavaScript, two parallel `curl` calls). The race window spans multiple async DB operations (lines 553–596), making it wide and reliably exploitable.
- **No rate limiting or idempotency key** is present on the approve endpoint.
- **No DB-level guard** (unique constraint, `SELECT FOR UPDATE`, or optimistic lock) prevents both writes from succeeding. [3](#0-2) 

## Recommendation

Move the guard check inside the database transaction and use a pessimistic lock (`SELECT FOR UPDATE`) to make the read-check-write atomic:

```typescript
await this.dataSource.transaction(async transactionalEntityManager => {
  // Re-read with a row-level lock inside the transaction
  const lockedApprovers = await transactionalEntityManager
    .getRepository(TransactionApprover)
    .find({
      where: { id: In(userApprovers.map(a => a.id)) },
      lock: { mode: 'pessimistic_write' },
    });

  if (lockedApprovers.every(a => a.signature)) {
    throw new BadRequestException(ErrorCodes.TAP);
  }

  await transactionalEntityManager
    .createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .whereInIds(lockedApprovers.map(a => a.id))
    .execute();
});
```

Additionally, consider adding a unique constraint on `(transactionId, userId)` in `TransactionApprover` as a defense-in-depth measure. [8](#0-7) [3](#0-2)

  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }
