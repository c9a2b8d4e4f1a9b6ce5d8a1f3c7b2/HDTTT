### Title
Unbounded Full-Table Scan with Per-Row Async DB Queries in `getTransactionsToSign` Causes Server Resource Exhaustion

### Summary

`GET /transactions/sign` fetches every non-terminal transaction in the database without any SQL-level `LIMIT`, then executes multiple asynchronous database queries for each row before applying pagination in memory. Any verified user can trigger this endpoint, causing the API server to consume unbounded memory, CPU, and database connections proportional to the total number of active transactions in the system, degrading or crashing the service for all users.

### Finding Description

**Root cause — `getTransactionsToSign` in `transactions.service.ts`:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({   // NO take/LIMIT — full table scan
  where: whereForUser,
  relations: ['groupItem'],
  order,
});

for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);  // DB query per row
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { console.log(error); }
}

return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // pagination applied AFTER full scan
  ...
};
```

The `repo.find()` call has no `take` option, so TypeORM emits a `SELECT` with no `LIMIT`. Every active (non-terminal) transaction in the database is loaded into Node.js heap memory.

For each of the N loaded transactions, `userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` is called: [2](#0-1) 

This chain performs:
1. `attachKeys(user, entityManager)` — a DB query to load user keys
2. `transactionSignatureService.computeSignatureKey(transaction, showAll)` — cryptographic computation + potential DB access
3. `entityManager.find(UserKey, { where: { publicKey: In(flatPublicKeys) } })` — another DB query [3](#0-2) 

Total cost per request: **1 unbounded SELECT + O(N) async DB round-trips**, where N = number of active transactions system-wide.

**Exposed endpoint:** [4](#0-3) 

The comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is abandoned but still live and fully reachable by any authenticated user.

### Impact Explanation

A single HTTP request from any verified user causes the API process to:
- Load the entire active transaction table into heap (memory exhaustion)
- Issue O(N) sequential async DB queries, holding DB connection pool slots for the duration
- Block the Node.js event loop with CPU-bound cryptographic work (`computeSignatureKey`) per transaction

At scale (thousands of active transactions), a single request can exhaust Node.js heap, starve the PostgreSQL connection pool, and cause the API service to become unresponsive for all users. Repeated requests (even at low rate) compound the effect. This constitutes **service unavailability / severe degradation** under realistic attacker input.

### Likelihood Explanation

- Attacker precondition: valid account + email verification — achievable by any self-registering user.
- Attack path: `GET /transactions/sign` — a single unauthenticated-to-authenticated HTTP request.
- No rate limiting or request-level resource cap is visible on this endpoint.
- The endpoint is still registered and guarded only by standard JWT auth guards. [5](#0-4) 

### Recommendation

1. **Apply a database-level LIMIT before the loop.** Add `take: limit` and `skip: offset` to the `repo.find()` call, and push the key-filtering logic into a SQL join or subquery so pagination happens at the DB layer.
2. **Remove or gate the endpoint.** Since the front-end no longer uses `GET /transactions/sign`, remove it or protect it with an admin-only guard.
3. **Batch the per-transaction DB queries.** If the endpoint must remain, pre-fetch all relevant `UserKey` rows in a single `IN (...)` query before the loop rather than querying once per transaction.

### Proof of Concept

**Preconditions:** A system with a large number of active (non-terminal) transactions. Attacker has a valid verified user account.

**Steps:**
1. Register and verify a user account.
2. Obtain a JWT via `POST /auth/login`.
3. Issue: `GET /transactions/sign?page=1&size=1` with the JWT.
4. Despite requesting only 1 result, the server executes `SELECT * FROM transaction WHERE status NOT IN (...)` with no LIMIT, loads all rows into memory, then calls `userKeysToSign` for every row sequentially.

**Expected outcome:** With N = 10,000 active transactions, the server performs ~10,000 DB round-trips and holds all transaction bytes in heap for the duration of the request. Repeated calls (or concurrent calls from multiple attacker accounts) exhaust the DB connection pool and Node.js heap, causing 503 errors for all users. [6](#0-5)

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
```
