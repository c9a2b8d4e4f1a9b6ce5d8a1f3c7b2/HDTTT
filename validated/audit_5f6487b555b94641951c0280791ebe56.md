Audit Report

## Title
`createTransactionApprovers` passes original DTO to recursive `createApprover` call, making the `threshold: null` leaf-node fix dead code

## Summary
In `approvers.service.ts`, the inner `createApprover` function builds a corrected copy (`nestedApprover`) of each child DTO and conditionally sets `threshold: null` for leaf nodes. However, the recursive call on line 348 re-spreads the **original** `nestedDtoApprover` instead of passing `nestedApprover`. The corrected copy is never used, so any leaf-node DTO that carries a numeric `threshold` will hit the `CHILDREN_REQUIRED` validation guard and abort the entire database transaction.

## Finding Description
Inside the loop that recurses over child approvers:

```typescript
// lines 341–349, approvers.service.ts
for (const nestedDtoApprover of dtoApprover.approvers) {
  const nestedApprover = { ...nestedDtoApprover, listId: approver.id }; // corrected copy

  if (!nestedDtoApprover.approvers || nestedDtoApprover.approvers.length === 0) {
    nestedApprover.threshold = null; // ← dead code: nestedApprover is never used
  }

  await createApprover({ ...nestedDtoApprover, listId: approver.id }); // ← original DTO re-spread
}
``` [1](#0-0) 

`nestedApprover` is assigned and mutated but never passed anywhere. The call on line 348 constructs a new object from `nestedDtoApprover` directly, so the `threshold: null` assignment on line 345 has zero effect.

The validation guard that this fix was meant to bypass:

```typescript
// lines 296–300
if (
  typeof dtoApprover.threshold === 'number' &&
  (!dtoApprover.approvers || dtoApprover.approvers.length === 0)
)
  throw new Error(this.CHILDREN_REQUIRED);
``` [2](#0-1) 

Because the threshold is never nulled out before the recursive call, any leaf-node DTO with a numeric `threshold` reaches this guard with the original value intact and throws.

## Impact Explanation
When a caller submits a nested approver tree where a leaf node (no `approvers` children) carries a numeric `threshold`, the recursive `createApprover` call throws `CHILDREN_REQUIRED`. The entire operation is wrapped in a `dataSource.transaction(...)` block, so all previously inserted parent nodes are rolled back. The outer `catch` re-throws as a `BadRequestException`. No approver structure is persisted for that transaction. [3](#0-2) 

## Likelihood Explanation
The endpoint is reachable by any authenticated transaction creator via `POST /transactions/:transactionId/approvers`. The `CreateTransactionApproverDto` permits `threshold` as an optional field on any node at the DTO-validation layer, so a frontend or API consumer that copies a parent-node DTO shape for a child node will produce exactly this input. The unused `nestedApprover` variable is a deterministic code defect — it fires on every request that includes a leaf node with a numeric `threshold`, not a race condition or edge case. [4](#0-3) 

## Recommendation
Replace the call on line 348 with `nestedApprover` (the already-corrected copy) instead of re-spreading the original:

```typescript
// Before (buggy)
await createApprover({ ...nestedDtoApprover, listId: approver.id });

// After (correct)
await createApprover(nestedApprover);
```

This ensures the `threshold: null` assignment for leaf nodes is actually passed into the recursive call, allowing the validation guard to pass for leaf nodes whose DTOs carry a stale `threshold` value.

## Proof of Concept
Send the following request body to `POST /transactions/1/approvers` as an authenticated transaction creator:

```json
{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "userId": 1,
          "threshold": 1
        }
      ]
    }
  ]
}
```

The child `{ "userId": 1, "threshold": 1 }` is a leaf node (no `approvers`). Because `nestedApprover` is dead code, `createApprover` receives `{ userId: 1, threshold: 1, listId: <parentId> }`. The guard at line 296–300 evaluates `typeof 1 === 'number'` (true) and `!approvers || approvers.length === 0` (true), throws `CHILDREN_REQUIRED`, rolls back the transaction, and returns a `400 Bad Request`. The parent node is also rolled back, leaving the transaction with no approvers.
