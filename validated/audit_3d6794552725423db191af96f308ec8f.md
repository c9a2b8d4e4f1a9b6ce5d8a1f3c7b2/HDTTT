### Title
Any Authenticated User Can Spoof a Privileged Fee Payer Account to Bypass the 6 KB Transaction Size Limit

### Summary
The backend's transaction size validation in `validateAndPrepareTransaction` derives the allowed size limit from the fee payer account embedded in the attacker-controlled `TransactionId`. Because the backend never verifies that the submitting user actually controls the fee payer account, any authenticated user can set the fee payer to a privileged governance account (e.g., `0.0.2` or any account in `0.0.42–0.0.799`) and have their transaction accepted at the 128 KB limit instead of the 6 KB limit. This is the direct analog of M-08: a special account class grants elevated privileges, and any user can claim membership in that class without authorization.

### Finding Description

**Root cause — fee payer is attacker-controlled and never ownership-verified**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `validateAndPrepareTransaction` performs the size check at line 929:

```typescript
// Check size
if (isTransactionBodyOverMaxSize(sdkTransaction)) {
  throw new BadRequestException(ErrorCodes.TOS);
}
```

`isTransactionBodyOverMaxSize` (defined in `back-end/libs/common/src/utils/sdk/transaction.ts` lines 328–333) reads the fee payer directly from the SDK transaction's `transactionId.accountId`:

```typescript
export function isTransactionBodyOverMaxSize(transaction: SDKTransaction) {
  const bodyBytes = getTransactionBodyBytes(transaction);
  const maxSize = getMaxTransactionSizeForTransaction(transaction);  // reads fee payer
  return bodyBytes.length > maxSize;
}
```

`getMaxTransactionSizeForTransaction` → `isPrivilegedFeePayer` grants 128 KB to accounts `0.0.2` and `0.0.42–0.0.799`:

```typescript
export function isPrivilegedFeePayer(accountId: AccountId | null | undefined): boolean {
  if (!accountId) return false;
  if (accountId.shard.toNumber() !== 0 || accountId.realm.toNumber() !== 0) return false;
  const num = accountId.num.toNumber();
  return num === 2 || (num >= 42 && num <= 799);
}
```

The only checks performed before the size gate are:

1. `creatorKeyId` belongs to the authenticated user's registered keys
2. The user's public key signed `dto.transactionBytes`

Neither check verifies that the fee payer account in the `TransactionId` is owned or controlled by the submitting user. The `TransactionId` is fully attacker-controlled: any Hedera SDK call such as `TransactionId.withValidStart(AccountId.fromString('0.0.2'), validStart)` sets the fee payer to the treasury account without requiring any proof of ownership.

**Exploit flow**

1. Attacker registers a key pair on the organization backend (normal user, no admin role).
2. Attacker constructs a `FileUpdateTransaction` with:
   - `transactionId` fee payer = `0.0.2` (or any account in `0.0.42–0.0.799`)
   - `contents` = ~100 KB payload (well under 128 KB, so the privileged check passes)
3. Attacker signs `transaction.toBytes()` with their own registered private key.
4. Attacker POSTs to `POST /transactions` with their own `creatorKeyId`.
5. Backend validates:
   - Signature: **passes** (attacker's key signed the bytes)
   - Expiry: **passes**
   - Size: `isTransactionBodyOverMaxSize` reads fee payer `0.0.2` → limit = 131,072 bytes → 100 KB < 128 KB → **passes**
   - Nodes: **passes** if valid node IDs are used
6. The 100 KB transaction is persisted to the database.

### Impact Explanation

- **Storage exhaustion / resource abuse**: Any authenticated user can store transactions up to 128 KB instead of 6 KB — a 21× amplification factor per transaction. Repeated submissions fill the PostgreSQL `transactionBytes` / `unsignedTransactionBytes` `bytea` columns far beyond the intended quota.
- **Integrity violation**: The 6 KB limit is a security invariant protecting the backend from oversized payloads. Spoofing a privileged fee payer silently nullifies it for any user.
- **Downstream chain-service waste**: The chain service will attempt to execute these transactions; they will fail on the Hedera network (the real fee payer `0.0.2` never signed), but the backend has already consumed storage and processing resources.
- **No fund loss**, but permanent database bloat and potential denial-of-service against the organization backend.

### Likelihood Explanation

- **Attacker precondition**: Only a valid authenticated account on the organization backend — no admin, no special role.
- **Effort**: Trivial. The Hedera SDK makes it a one-liner to set an arbitrary fee payer in a `TransactionId`. No cryptographic material for the spoofed account is needed.
- **Detectability**: The backend logs no warning when a non-privileged user submits a transaction with a privileged fee payer. The attack is silent.
- **Realistic scenario**: Any disgruntled or malicious organization member can execute this immediately after registration.

### Recommendation

After parsing the SDK transaction, verify that the fee payer account in the `TransactionId` matches the creator's registered Hedera account, or — if the tool must support governance accounts submitting on behalf of others — enforce that the creator key is associated with a known privileged account in the organization's user database. Concretely, in `validateAndPrepareTransaction`:

```typescript
const feePayer = sdkTransaction.transactionId?.accountId;
if (isPrivilegedFeePayer(feePayer)) {
  // Verify the creator's key is actually linked to this privileged account
  if (!creatorKey.accountId || creatorKey.accountId !== feePayer.toString()) {
    throw new BadRequestException(ErrorCodes.SNMP); // or a dedicated error code
  }
}
```

Alternatively, ignore the fee payer when computing the size limit for user-submitted transactions and always apply the standard 6 KB limit unless the organization backend itself is the submitter.

### Proof of Concept

```typescript
import {
  FileUpdateTransaction,
  FileId,
  TransactionId,
  AccountId,
  Timestamp,
  PrivateKey,
} from '@hiero-ledger/sdk';

// Attacker's own registered key pair
const attackerKey = PrivateKey.generateED25519();

// Craft a ~100 KB transaction with treasury (0.0.2) as fee payer
const tx = new FileUpdateTransaction()
  .setFileId(FileId.fromString('0.0.150'))
  .setContents(Buffer.alloc(100_000, 0x41))          // 100 KB payload
  .setTransactionId(
    TransactionId.withValidStart(
      AccountId.fromString('0.0.2'),                  // spoofed privileged fee payer
      Timestamp.fromDate(new Date()),
    ),
  );

const txBytes = tx.toBytes();
const signature = attackerKey.sign(txBytes);

// POST /transactions with attacker's own creatorKeyId
// → backend accepts: isTransactionBodyOverMaxSize returns false
//   because fee payer 0.0.2 → limit = 131,072 bytes > 100,000 bytes
const dto = {
  name: 'Oversized',
  description: '',
  transactionBytes: Buffer.from(txBytes).toString('hex'),
  creatorKeyId: ATTACKER_KEY_ID,   // attacker's registered key
  signature: Buffer.from(signature).toString('hex'),
  mirrorNetwork: 'testnet',
};
// Expected: HTTP 201 Created — 100 KB transaction stored in DB
// Without fix: HTTP 400 TOS would be the correct response
```

**Expected result without fix**: The 100 KB transaction is accepted and stored.
**Expected result with fix**: The backend rejects the transaction with `ErrorCodes.TOS` (transaction over size) because the submitting user does not own `0.0.2`.
