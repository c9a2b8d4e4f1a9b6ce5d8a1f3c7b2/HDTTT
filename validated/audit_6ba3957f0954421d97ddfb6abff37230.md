Audit Report

## Title
Manual Transaction Creator Can Archive Transactions in Terminal States Due to Logical AND Error in Status Guard

## Summary
In `archiveTransaction` within `back-end/apps/api/src/transactions/transactions.service.ts`, the guard condition joins two sub-conditions with a logical AND (`&&`). When `transaction.isManual === true`, the expression `!transaction.isManual` evaluates to `false`, collapsing the entire `if` to `false` and preventing the exception from being thrown. As a result, a transaction creator can archive their own manual transaction regardless of its current status — including terminal states `EXECUTED`, `FAILED`, `EXPIRED`, and `CANCELED`.

## Finding Description

**Root cause — `archiveTransaction` guard logic:**

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts (lines 711–718)
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← short-circuits the entire check when isManual=true
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The intended semantics are: "throw unless the status is in the allowed set OR the transaction is manual." The correct operator is `||`, not `&&`. With `&&`, when `isManual=true`:

- `!transaction.isManual` → `false`
- `<status check> && false` → `false`
- The `if` body is never entered; `repo.update` proceeds unconditionally. [2](#0-1) 

**Test gap — unit test does not set `isManual: true` for the terminal-state case:**

The "should throw" unit test uses `{ status: TransactionStatus.CANCELED }` with no `isManual` field (defaults to falsy), so the guard fires correctly and the test passes — but the bug path is never exercised. [3](#0-2) 

**Test gap — e2e "already executed" test uses a non-manual transaction:**

The e2e test that asserts a 400 response for an executed transaction operates on `addedTransactions.userTransactions[0]`, which is not a manual transaction. A manual transaction in `EXECUTED` state would return 200 and overwrite the status. [4](#0-3) 

**Frontend guard is purely cosmetic:**

The frontend `canArchive` computed in `TransactionDetailsHeader.vue` does enforce a status check even for manual transactions, but it is client-side only and trivially bypassed by calling `PATCH /transactions/archive/:id` directly. [5](#0-4) 

## Impact Explanation

**Severity: High**

The `status` field is the sole authoritative lifecycle record for a transaction in the PostgreSQL database. Overwriting `EXECUTED` → `ARCHIVED` on a manual transaction:

- **Erases execution evidence**: Co-signers, observers, and approvers lose the `EXECUTED` record and can no longer confirm whether a Hedera network transaction was submitted.
- **Conceals failures**: A `FAILED` transaction can be silently archived, hiding errors from auditors.
- **Corrupts group transaction history**: In transaction groups, individual item statuses feed into group-level reporting; corrupting one item's status corrupts the group view. [6](#0-5) 

## Likelihood Explanation

**Likelihood: High**

- No elevated privilege is required — any authenticated user who created a manual transaction can exploit this.
- The exploit is a single direct HTTP `PATCH /transactions/archive/:id` request.
- `isManual` is a standard field set at transaction creation time, not restricted to any special role.
- The frontend guard is purely cosmetic and provides no server-side enforcement. [7](#0-6) 

## Recommendation

Change the logical operator in the guard from `&&` to `||` so that both conditions must independently fail before archiving is permitted:

```typescript
// Corrected guard
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual   // ← change to: the status check alone is sufficient
) {
```

The correct fix is to remove the `isManual` short-circuit entirely, or restructure the guard so that the status allowlist is enforced unconditionally:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!archivableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Additionally, add unit and e2e tests covering a manual transaction in each terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) to assert that `archiveTransaction` throws `ErrorCodes.OMTIP`. [8](#0-7) 

## Proof of Concept

1. Authenticate as any user and create a manual transaction:
   ```
   POST /transactions
   { ..., "isManual": true }
   ```
2. Allow or force the transaction to reach `EXECUTED` status (or directly update via the DB / another API path).
3. Call the archive endpoint directly (bypassing the frontend):
   ```
   PATCH /transactions/archive/<id>
   Authorization: Bearer <creator_token>
   ```
4. Observe HTTP 200 response and confirm in the database that `status` is now `ARCHIVED` — the `EXECUTED` record has been overwritten.

The guard at lines 711–718 evaluates `<status_not_in_allowlist> && !isManual` → `true && false` → `false`, so the exception is never raised and `repo.update` executes unconditionally. [9](#0-8)
