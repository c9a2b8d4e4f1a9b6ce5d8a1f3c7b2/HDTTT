Audit Report

## Title
TOCTOU Race Condition in Signature Upload Allows Duplicate `TransactionSigner` Records, Corrupting Transaction State

## Summary
The `uploadSignatureMaps` flow in `SignersService` reads existing signers as a point-in-time snapshot, then inserts new signers with no conflict handling and no unique database constraint on `(transactionId, userKeyId)`. Two concurrent signature-upload requests for the same transaction by the same user can both pass the duplicate check and both insert the same signer record, producing duplicate rows that corrupt transaction status tracking.

## Finding Description

**1. Non-unique composite index on `transaction_signer`**

The `TransactionSigner` entity carries only a plain (non-unique) composite index:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // NOT unique
export class TransactionSigner { … }
``` [1](#0-0) 

The corresponding migration confirms this — the index is created without `UNIQUE`:

```sql
CREATE INDEX "IDX_f8cbac3e0cae3730a762e4905e"
ON "transaction_signer" ("transactionId", "userKeyId")
``` [2](#0-1) 

No `UNIQUE` constraint exists anywhere in the migration history for this table. The initial schema migration also creates the table with only a plain primary key and no uniqueness on `(transactionId, userKeyId)`. [3](#0-2) 

**2. Duplicate check is a pre-loaded snapshot (TOCTOU)**

`loadTransactionData` reads all existing signers once, before any processing begins:

```typescript
const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
  where: { transactionId: In(transactionIds) },
  select: ['transactionId', 'userKeyId'],
});
``` [4](#0-3) 

That snapshot is passed into `processTransactionSignatures`, which uses it to decide whether a key is "new":

```typescript
if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);
}
``` [5](#0-4) 

Two concurrent requests both read the same snapshot before either has committed, both see the key as absent, and both mark it for insertion.

**3. `bulkInsertSigners` has no conflict handling**

```typescript
private async bulkInsertSigners(manager: any, signersToInsert: any[]) {
  const result = await manager
    .createQueryBuilder()
    .insert()
    .into(TransactionSigner)
    .values(signersToInsert)
    .returning('*')
    .execute();   // no .orIgnore(), no ON CONFLICT clause
  return result.raw;
}
``` [6](#0-5) 

By contrast, the `insertKeys` helper used for cached account/node keys explicitly uses `.orIgnore()` for idempotency:

```typescript
await this.dataSource
  .createQueryBuilder()
  .insert()
  .into(keyEntity)
  .values(…)
  .orIgnore()
  .execute();
``` [7](#0-6) 

The inconsistency is clear: the codebase already knows how to write idempotent inserts but did not apply the pattern to `bulkInsertSigners`.

## Impact Explanation

Duplicate `TransactionSigner` rows for the same `(transactionId, userKeyId)` corrupt the state that `processTransactionStatus` and `TransactionSignatureService` rely on to determine whether a transaction has collected enough signatures to advance. The same key's signature is counted more than once, which can cause:

- A transaction to be incorrectly promoted to `WAITING_FOR_EXECUTION` before all required keys have actually signed.
- Downstream execution of a transaction that is not fully authorized.
- Persistent database state corruption that cannot be self-healed without manual intervention.

Additionally, if a `UNIQUE` constraint is later added as a remediation, the absence of `.orIgnore()` in `bulkInsertSigners` means concurrent requests would throw a constraint-violation exception, caught by the outer `catch` block at line 342–344 and re-thrown as `ErrorCodes.FST`, causing the entire signature-upload batch to fail. [8](#0-7) 

## Likelihood Explanation

The `POST /transactions/:transactionId/signers` endpoint is accessible to any verified JWT holder who is a signer on a transaction. The race window is the time between `loadTransactionData` completing and the commit of `bulkInsertSigners`. In a multi-instance deployment (the repo ships Kubernetes manifests for multiple API replicas), two requests handled by different pods will never share an in-process lock, making the race reliably reproducible by submitting the same signature map twice in rapid succession — something that can happen accidentally (double-click, client retry logic) or deliberately.

## Recommendation

Apply both fixes together:

1. **Add a `UNIQUE` constraint** on `(transactionId, userKeyId)` in `transaction_signer` via a new migration:
   ```sql
   CREATE UNIQUE INDEX "UQ_transaction_signer_tx_key"
   ON "transaction_signer" ("transactionId", "userKeyId");
   ```

2. **Add `.orIgnore()`** (or an explicit `ON CONFLICT DO NOTHING`) to `bulkInsertSigners` in `signers.service.ts` to make concurrent inserts idempotent, consistent with the pattern already used in `cache.helper.ts`:
   ```typescript
   .values(signersToInsert)
   .orIgnore()
   .returning('*')
   .execute();
   ``` [9](#0-8) 

## Proof of Concept

1. User A is a signer on transaction T with key K (not yet signed).
2. Client sends two identical `POST /transactions/T/signers` requests simultaneously (e.g., double-click or retry).
3. Both requests enter `loadTransactionData` concurrently and both read the same snapshot: key K is absent from `transaction_signer`.
4. Both pass the `!existingSignerIds.has(userKey.id)` check and both add key K to `signersToInsert`.
5. Both execute `bulkInsertSigners` — since there is no unique constraint and no `.orIgnore()`, both inserts succeed.
6. `transaction_signer` now contains two rows with identical `(transactionId=T, userKeyId=K)`.
7. `processTransactionStatus` counts both rows, double-counting key K's signature, and may incorrectly advance the transaction to `WAITING_FOR_EXECUTION` even if other required signers have not yet signed. [10](#0-9) [6](#0-5)
