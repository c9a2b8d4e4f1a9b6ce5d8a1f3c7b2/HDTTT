### Title
AES-256-ECB Without Authentication in Legacy Mnemonic Decryption Allows Undetected Ciphertext Tampering

### Summary
`decryptLegacyMnemonic` in `front-end/src/main/services/localUser/dataMigration.ts` decrypts the legacy `recovery.aes` file using AES-256-ECB — a mode with no IV, no authentication tag, and no MAC. An attacker with local filesystem access (e.g., malware) can silently tamper with the ciphertext before migration. The decryption will succeed without any integrity error, and the corrupted or attacker-influenced mnemonic will be imported into the wallet. The primary `encrypt`/`decrypt` path in `front-end/src/main/utils/crypto.ts` already uses AES-256-GCM, but this legacy migration path was never upgraded.

### Finding Description

In `front-end/src/main/services/localUser/dataMigration.ts`, the function `decryptLegacyMnemonic` reads the legacy encrypted recovery file and decrypts it using AES-256-ECB: [1](#0-0) 

Specifically at lines 108–114:

```ts
const salt = Buffer.from([1, 2, 3, 4, 5, 6, 7, 8]);  // hardcoded static salt
const key = crypto.pbkdf2Sync(password, salt, iterationCount, keyLength, 'sha256');
// ...
const decipher = crypto.createDecipheriv('aes-256-ecb', key, null);
const decrypted = Buffer.concat([decipher.update(data), decipher.final()]);
return decrypted.toString('utf-8');
``` [2](#0-1) 

ECB mode encrypts each 16-byte block independently with no chaining, no IV, and no authentication tag. This means:

1. **No integrity check**: Any modification to the ciphertext file is silently accepted. The decryption call will not throw.
2. **Block-level malleability**: An attacker can rearrange, duplicate, or zero-out 16-byte blocks. Each block decrypts independently, so the output is a deterministic transformation of the attacker's chosen block arrangement.
3. **Compounding issue — hardcoded salt**: The PBKDF2 salt is the static literal `[1, 2, 3, 4, 5, 6, 7, 8]` for every user. This allows precomputed rainbow tables against common passwords, lowering the bar for a full plaintext-recovery attack.

This function is called from `decryptMigrationMnemonic` when `isLegacy = true`: [3](#0-2) 

The result is split by spaces and returned as the user's mnemonic word list, which is then used to derive Hedera private keys.

By contrast, the current production encryption path correctly uses AES-256-GCM with a random salt and authentication tag: [4](#0-3) 

### Impact Explanation

If an attacker tampers with the `recovery.aes` file before the user runs the migration wizard:

- **Corrupted mnemonic import**: Rearranging ECB blocks produces a deterministically corrupted plaintext. The user imports a mnemonic that does not correspond to their actual keys, permanently losing access to their Hedera wallet funds.
- **Partial block substitution**: If the attacker knows the ECB-encrypted form of specific BIP39 words (obtainable by encrypting known words with the same static salt and a guessed/brute-forced password), they can substitute individual 16-byte blocks to inject attacker-controlled words into the mnemonic, potentially redirecting derived keys to an attacker-controlled wallet.
- **Silent failure**: Because there is no authentication tag or MAC, neither the application nor the user receives any error. The migration completes "successfully" with a corrupted or attacker-influenced seed.

### Likelihood Explanation

The `recovery.aes` file lives on the local filesystem at a predictable path (`~/Documents/TransactionTools/Files/.System/recovery.aes`). For a desktop Electron wallet app, malware with user-level filesystem access is a well-established and realistic threat model. No privileged OS access is required — only read/write access to the user's Documents folder, which any process running as the user possesses. The migration wizard is a one-time flow, but users who have not yet migrated from the old tool are exposed for the entire window between installation and migration.

### Recommendation

1. **Replace AES-256-ECB with AES-256-GCM** in `decryptLegacyMnemonic`, matching the pattern already used in `decryptMnemonic` (lines 121–153 of the same file), which correctly uses GCM with an authentication tag.
2. **Remove the hardcoded static salt** (`[1, 2, 3, 4, 5, 6, 7, 8]`). If the legacy file format cannot be changed, document this as a known limitation and warn users to use a strong, unique password.
3. **Verify the decrypted mnemonic** against a BIP39 word list before importing, as a defense-in-depth measure to catch corrupted output.

### Proof of Concept

**Precondition**: Attacker has user-level filesystem write access (e.g., via malware) before the user runs the migration wizard.

**Steps**:

1. Locate `~/Documents/TransactionTools/Files/.System/recovery.aes`.
2. Read the file as raw bytes. Each 16-byte chunk is an independent ECB block.
3. Rearrange or zero-out blocks — for example, swap block 0 and block 1:
   ```python
   data = open('recovery.aes', 'rb').read()
   block = 16
   tampered = data[block:2*block] + data[0:block] + data[2*block:]
   open('recovery.aes', 'wb').write(tampered)
   ```
4. User launches the app and runs the migration wizard, entering their correct password.
5. `decryptLegacyMnemonic` decrypts the tampered file with no error. The first two mnemonic words are swapped (or corrupted).
6. The corrupted mnemonic is imported. The derived Hedera private key does not match the user's actual key. Funds are inaccessible. [5](#0-4)

export async function decryptLegacyMnemonic(
  inputPath: string,
  password: string,
): Promise<string | null> {
  /* Read the encrypted data from the file */
  const data = await fs.promises.readFile(inputPath, { flag: 'r' });

  const iterationCount = 65536;
  const keyLength = 32; // 256 bits
  const salt = Buffer.from([1, 2, 3, 4, 5, 6, 7, 8]);

  const key = crypto.pbkdf2Sync(password, salt, iterationCount, keyLength, 'sha256');
  const header = Buffer.from('AES|256|CBC|PKCS5Padding|', 'utf-8');
  if (data.length < header.length || !header.equals(data.slice(0, header.length))) {
    const decipher = crypto.createDecipheriv('aes-256-ecb', key, null);
    const decrypted = Buffer.concat([decipher.update(data), decipher.final()]);
    return decrypted.toString('utf-8');
  }

  return null;
}

export async function decryptMigrationMnemonic(password: string): Promise<string[] | null> {
  const content = await fs.promises.readFile(getPropertiesPath(), { encoding: 'utf-8', flag: 'r' });
  const parsedContent = parseUserProperties(content);

  const token = parsedContent.hash;
  if (!token) throw Error('No hash found at location');

  const isLegacy = parsedContent.legacy;

  let words: string | null;
  if (isLegacy) {
    words = await decryptLegacyMnemonic(getMnemonicPath(), password);
  } else {
    words = await decryptMnemonic(getMnemonicPath(), token, password);
  }

  if (words) return words.split(' ');

  return null;
}

export function encrypt(data: string, password: string) {
  const iv = crypto.randomBytes(16);
  const salt = crypto.randomBytes(64);

  const key = deriveKey(password, salt);

  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([cipher.update(data, 'utf8'), cipher.final()]);

  const tag = cipher.getAuthTag();

  return Buffer.concat([salt, iv, tag, encrypted]).toString('base64');
}
