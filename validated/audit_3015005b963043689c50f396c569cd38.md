### Title
TOCTOU Race Condition in `approveTransaction` Allows a User to Submit Conflicting Approval Decisions, Corrupting Transaction Approval State

### Summary
The `approveTransaction` method in `approvers.service.ts` performs a "has the user already approved?" check and the subsequent database write as two separate, non-atomic operations. A malicious authenticated approver can send two concurrent HTTP requests with opposite `approved` values. Both requests pass the guard before either write completes, and the final persisted state is whichever write lands last — non-deterministically overwriting the first decision. This is the direct analog of the external report's "function callable multiple times without a guard" class, applied to the approval state-transition in this protocol.

### Finding Description

**Root cause — non-atomic check-then-act:**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` reads the current approver state, checks whether the user has already signed, and then — after several more async operations — writes the new state inside a separate database transaction:

The guard at line 563 checks the in-memory snapshot loaded at line 553. Between those two points and the actual write at line 599, there are three `await` calls (`attachKeys`, `findOne`, `verifyTransactionBodyWithoutNodeAccountIdSignature`). During any of those awaits, a second concurrent request from the same user reads the same snapshot (signature still `null`), also passes the guard, and proceeds independently to the write.

**Exploit flow:**

1. Attacker is a legitimate approver on transaction `T`.
2. Attacker sends **Request A** (`approved: true`) and **Request B** (`approved: false`) concurrently to `POST /transactions/:id/approvers/approve`.
3. Both requests call `getVerifiedApproversByTransactionId` before either write; both see `signature: null`; both pass line 563.
4. Request A writes `approved=true, signature=sigA`.
5. Request B writes `approved=false, signature=sigB`, overwriting Request A's row.
6. Final DB state: `approved=false` — the attacker has effectively rejected the transaction after the system (and other approvers) may have already acted on the `approved=true` event emitted by Request A.

The notification emission at lines 614–618 uses the stale in-memory `userApprovers` array (loaded before either write), so the status-update event fired by Request A is based on pre-write data, compounding the inconsistency. [3](#0-2) 

### Impact Explanation

- **Approval state corruption**: A user can have their `approved=true` record silently overwritten with `approved=false` (or vice versa) within the same logical "approval action," producing a final DB state that does not reflect any single coherent decision.
- **Threshold bypass / sabotage**: If the approval threshold is met by the first write and the chain service begins acting on it, the second write can flip the approver's record to `rejected` retroactively, creating an inconsistency between the on-chain action and the stored approval state.
- **Audit trail integrity**: The `TransactionApprover` row is the authoritative record of who approved and with what signature. A race-corrupted row breaks the audit trail.

### Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated user who is listed as an approver on the target transaction — a normal, non-privileged role.
- **Trigger**: Two concurrent HTTP POST requests to the same endpoint. Achievable with any HTTP client (e.g., `Promise.all([fetch(…), fetch(…)])` or a simple script). No special tooling required.
- **Race window**: The window spans three sequential `await` calls (lines 566, 575, 594), making it wide enough to be reliably triggered even over a normal network connection.

### Recommendation

Replace the read-check-write pattern with a single atomic conditional update. Use a database-level `UPDATE … WHERE signature IS NULL … RETURNING id` and treat zero affected rows as "already approved":

```typescript
const result = await this.dataSource.transaction(async manager => {
  return manager
    .createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .where('id IN (:...ids) AND signature IS NULL', { ids: userApprovers.map(a => a.id) })
    .returning('id')
    .execute();
});

if (result.raw.length === 0) {
  throw new BadRequestException(ErrorCodes.TAP); // already approved
}
```

This makes the guard and the write a single atomic operation at the database level, eliminating the TOCTOU window entirely.

### Proof of Concept

```typescript
// Attacker is an authenticated approver on transaction ID 42.
// Send two concurrent requests with opposite decisions:
const [resA, resB] = await Promise.all([
  fetch('/api/transactions/42/approvers/approve', {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ userKeyId: 1, signature: sigA, approved: true }),
  }),
  fetch('/api/transactions/42/approvers/approve', {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
    body: JSON.stringify({ userKeyId: 1, signature: sigB, approved: false }),
  }),
]);

// Expected (correct): second request returns 400 ErrorCodes.TAP
// Actual (buggy):     both return 200; DB row ends up with whichever
//                     write landed last — approved=false overwrites approved=true
//                     or vice versa, non-deterministically.
```

**Expected outcome**: Query `SELECT approved, signature FROM transaction_approver WHERE id = <approver_id>` after both requests complete. The value will reflect whichever concurrent write landed last, not a single coherent decision. The `emitTransactionStatusUpdate` event fired by the first write will have already propagated to the chain service with stale data.
