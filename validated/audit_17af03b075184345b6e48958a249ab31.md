### Title
`transactionId` URL Parameter Not Verified Against Approver's Stored `transactionId` in `removeTransactionApprover`

### Summary
In `ApproversController.removeTransactionApprover`, the URL-supplied `transactionId` is used only to verify the caller is the creator of *that* transaction, but the approver record being deleted is fetched solely by its own `id` with no check that `approver.transactionId === transactionId`. Any authenticated user who is the creator of at least one transaction can delete approvers belonging to any other transaction by supplying their own `transactionId` in the URL and a victim approver's `id` in the path.

### Finding Description

**Root cause — controller (`approvers.controller.ts` lines 102–113):**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);  // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);                // ← deletes approver by id, no cross-check
  return true;
}
``` [1](#0-0) 

**Service (`approvers.service.ts` lines 534–544):**

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(..., [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

`removeTransactionApprover` never asserts `approver.transactionId === transactionId`. The `transactionId` from the URL is consumed entirely by `getCreatorsTransaction`, which only confirms the caller created *that* transaction. [3](#0-2) 

**Exploit path:**

1. Attacker (authenticated, verified user) creates their own transaction → receives `transactionA_id`.
2. Attacker enumerates or guesses `approverB_id` belonging to a victim's transaction (approver PKs are sequential integers via `@PrimaryGeneratedColumn()`).
3. Attacker sends: `DELETE /transactions/{transactionA_id}/approvers/{approverB_id}`
4. `getCreatorsTransaction(transactionA_id, attacker)` passes — attacker owns transaction A.
5. `removeTransactionApprover(approverB_id)` executes — deletes the victim's approver with no ownership check. [4](#0-3) 

### Impact Explanation

Deleting an approver from a victim's transaction removes a required approval gate. If the victim's transaction had a threshold-based approver tree, removing nodes can reduce the threshold requirement or eliminate it entirely, allowing the transaction to advance to `WAITING_FOR_EXECUTION` or `EXECUTED` without the intended approvals. This is an unauthorized state mutation on another user's transaction — a direct integrity failure in the approval workflow.

### Likelihood Explanation

Any registered, verified user can create a transaction (no special privilege required). Approver IDs are sequential integers, making enumeration trivial. The attacker needs only one valid transaction of their own and knowledge of any approver ID on any other transaction. No leaked credentials or admin access are required.

### Recommendation

After fetching the approver record inside `removeTransactionApprover` (or in the controller before calling it), assert that `approver.transactionId === transactionId`. Throw `UnauthorizedException` if they do not match. The same cross-check should be applied to `updateTransactionApprover` to close the same class of issue on the `PATCH` route.

```typescript
// In the controller, after getCreatorsTransaction:
const approver = await this.approversService.getTransactionApproverById(id);
if (!approver || approver.transactionId !== transactionId) {
  throw new BadRequestException(ErrorCodes.ANF);
}
await this.approversService.removeTransactionApprover(id);
```

### Proof of Concept

**Preconditions:** Two users, Alice (victim) and Bob (attacker), both authenticated and verified. Alice has a transaction (id=10) with an approver record (id=7). Bob has a transaction (id=20).

**Steps:**

1. Bob sends: `DELETE /transactions/20/approvers/7` with Bob's JWT.
2. Server calls `getCreatorsTransaction(20, Bob)` → passes (Bob owns tx 20).
3. Server calls `removeTransactionApprover(7)` → fetches approver id=7 (belongs to Alice's tx 10), deletes it.
4. Alice's transaction 10 now has its approver removed without Alice's or the approver's consent.
5. Depending on the approver tree structure, Alice's transaction may now proceed to execution without the required approval.

**Expected (correct) behavior:** Server should reject with 400/401 because approver id=7 does not belong to transaction id=20.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
