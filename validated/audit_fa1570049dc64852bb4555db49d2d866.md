Audit Report

## Title
Single Approval Satisfies Multiple Threshold Slots Due to Cross-Branch Duplicate Approver Placement

## Summary
The `isNode` duplicate-check in `createTransactionApprovers` only prevents adding the same user under the **same parent node** (`listId`). It does not prevent the same user from appearing as a leaf node in **different branches** of the approver tree. When `approveTransaction` is called, it bulk-updates **all** `TransactionApprover` rows belonging to that user in a single query, causing one approval action to simultaneously satisfy multiple threshold slots across different branches.

## Finding Description

**Root cause — `isNode` only checks same-parent uniqueness:**

`isNode` in `approvers.service.ts` constructs its `where` clause including `listId`:

```typescript
where: {
  listId: typeof approver.listId === 'number' ? approver.listId : null,
  userId: typeof approver.userId === 'number' ? approver.userId : null,
  ...
  transactionId: typeof approver.listId === 'number' ? null : transactionId,
},
```

Adding User A under `listId=10` checks for `{listId:10, userId:A}`. Adding User A under `listId=20` checks for `{listId:20, userId:A}`. These are independent queries; neither returns a count > 0 for the other, so both insertions succeed. The same user can therefore occupy multiple leaf nodes across different branches of the same transaction's approver tree. [1](#0-0) 

**Compounding issue — existing approval is copied to new branches on creation:**

When the creator adds a user who has **already approved** to a new branch, `getApproversByTransactionId` returns the existing record and its `signature`, `userKeyId`, and `approved` fields are immediately copied into the new record. The new branch is retroactively satisfied without any further action from the approver. [2](#0-1) 

**Approval bulk-update covers all slots for the user:**

In `approveTransaction`, `getVerifiedApproversByTransactionId` returns the full flat list of all approver rows for the transaction. `userApprovers` then collects every row where `userId === user.id` — across all branches. The duplicate-approval guard uses `every`:

```typescript
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

If none have a signature yet, the guard passes. The subsequent `whereInIds(userApprovers.map(a => a.id))` bulk-update then stamps the signature on **all** of the user's slots simultaneously — one HTTP call, multiple threshold slots satisfied. [3](#0-2) 

## Impact Explanation
A malicious transaction creator can design an approver tree that visually and structurally implies a higher approval threshold than is actually enforced. Organizational members who inspect the tree see "threshold 2 of 3" and believe two independent parties must approve. In reality, a single user's approval satisfies two slots simultaneously, allowing the transaction to execute with fewer unique approvals than intended. This undermines the integrity of the multi-party approval governance model — the core security property of the organizational workflow.

## Likelihood Explanation
The attacker is the transaction creator, a role available to any authenticated organization user with no elevated privileges. The exploit requires only crafting a specific approver tree structure via the documented `POST /transactions/:id/approvers` API. No race conditions, cryptographic breaks, or external dependencies are required. Any creator who wants to reduce the effective approval threshold while concealing that reduction can do so deterministically.

## Recommendation

1. **Enforce global uniqueness of `(transactionId, userId)` at the tree level during creation.** Before inserting a new leaf node, check whether the `userId` already exists anywhere in the transaction's approver tree (not just under the same `listId`). The existing `getApproversByTransactionId(transactionId, userId)` helper already performs this recursive query and can be reused — if it returns any records, reject the insertion. [4](#0-3) 

2. **Remove the approval-copy behavior on creation.** The block at lines 318–329 that copies `signature`/`userKeyId`/`approved` from an existing record into a newly created branch node should be removed. A new branch slot should always start unsigned. [2](#0-1) 

3. **Update `approveTransaction` to update only one slot per approval call**, or at minimum ensure the `whereInIds` update is scoped to a single record rather than all records for the user across the entire tree. [5](#0-4) 

## Proof of Concept

**Setup (creator calls `POST /transactions/:id/approvers`):**
```json
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [
      {
        "threshold": 1,
        "approvers": [{ "userId": A }]
      },
      {
        "threshold": 1,
        "approvers": [{ "userId": A }]
      },
      {
        "threshold": 1,
        "approvers": [{ "userId": B }]
      }
    ]
  }]
}
```

- `isNode` is called for the first `userId: A` with `listId = <child_threshold_node_1_id>` → count = 0, insertion succeeds.
- `isNode` is called for the second `userId: A` with `listId = <child_threshold_node_2_id>` → count = 0 (different `listId`), insertion succeeds. [6](#0-5) 

**Exploit (User A calls `POST /transactions/:id/approvers/approve` once):**

1. `getVerifiedApproversByTransactionId` returns all approver rows including both User A leaf nodes (in branch 1 and branch 2).
2. `userApprovers = [record_branch_1, record_branch_2]` — both have no signature yet, so the `every(a => a.signature)` guard passes.
3. `whereInIds([record_branch_1.id, record_branch_2.id])` stamps the signature on both rows in one query.
4. Both child threshold nodes (threshold=1) are now satisfied; the root threshold of 2 is met.
5. Transaction proceeds to execution without User B's approval, despite the displayed structure implying 2-of-3 independent approvers are required. [7](#0-6)

  async getApproversByTransactionId(
    transactionId: number,
    userId?: number,
    entityManager?: EntityManager,
  ): Promise<TransactionApprover[]> {
    if (typeof transactionId !== 'number' || (userId && typeof userId !== 'number')) return null;

    return (entityManager || this.repo).query(
      `
      with recursive approverList as
        (
          select * from transaction_approver 
          where "transactionId" = $1
            union all
              select approver.* from transaction_approver as approver
              join approverList on approverList."id" = approver."listId"
        )
      select * from approverList
      where approverList."deletedAt" is null
        ${userId ? 'and approverList."userId" = $2' : ''}
      `,
      userId ? [transactionId, userId] : [transactionId],
    );
  }
