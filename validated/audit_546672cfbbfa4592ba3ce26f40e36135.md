Based on my investigation, I found a concrete analog to the external report's vulnerability class (missing input validation on a critical numeric parameter) in the `CreateTransactionApproverDto`.

---

### Title
`CreateTransactionApproverDto.threshold` Lacks Minimum Value Validation, Allowing Negative Threshold to Bypass Organizational Approval Workflow

### Summary
The `CreateTransactionApproverDto` accepts a `threshold` field with only `@IsNumber()` and `@IsOptional()` decorators — no `@Min(1)` constraint. The service-level validation only rejects `threshold === 0`, not negative values. A malicious authenticated user (transaction creator) can submit `threshold: -1` when creating an approver tree, which passes all validation and is persisted to the database. Since any count of approvals (≥ 0) satisfies a negative threshold, the approval requirement is trivially bypassed. This is directly analogous to the external report: a critical numeric parameter lacks lower-bound validation, making the enforcement mechanism ineffective.

### Finding Description

**Root cause — DTO layer:**

`CreateTransactionApproverDto` has no `@Min` constraint on `threshold`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
@IsNumber()
@IsOptional()
threshold?: number;   // ← no @Min(1)
``` [1](#0-0) 

By contrast, the **update** DTO correctly enforces `@Min(1)`:

```typescript
// back-end/apps/api/src/transactions/dto/update-transaction-approver.dto.ts
@IsNumber()
@Min(1)
@IsOptional()
threshold?: number;
``` [2](#0-1) 

The inconsistency shows `@Min(1)` was intentionally added to the update path but omitted from the create path.

**Root cause — service layer:**

The service validation only rejects `threshold === 0` and `threshold > approvers.length`. A negative value (e.g., `-1`) satisfies neither condition and is silently accepted:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
``` [3](#0-2) 

For `threshold = -1` with 1 child approver: `-1 > 1` is `false`, `-1 === 0` is `false` → the check passes and the record is inserted.

**Exploit flow:**

1. Attacker (authenticated user) creates a transaction via `POST /transactions`.
2. Attacker calls `POST /transactions/:id/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": -1, "approvers": [{ "userId": 99 }] }] }
   ```
3. `CreateTransactionApproverDto` passes NestJS `ValidationPipe` (no `@Min` check).
4. `approvers.service.ts` `createTransactionApprovers` passes the service-level check (only rejects `=== 0`).
5. A `TransactionApprover` row with `threshold = -1` is inserted into PostgreSQL.
6. Any subsequent approval-count evaluation of the form `approvalCount >= threshold` is trivially satisfied (0 ≥ -1), so the transaction is treated as fully approved without any approver actually approving it.

### Impact Explanation

The organizational approval workflow — the primary multi-party authorization control before a transaction is submitted to the Hedera network — is bypassed. A transaction creator can mark their own transaction as approved without any designated approver taking action. This enables unauthorized transaction execution under the guise of a legitimately approved workflow, constituting an unauthorized state change and integrity failure in the trust model.

### Likelihood Explanation

Any authenticated user who can create transactions (a normal product flow) can trigger this. No privileged access, no leaked credentials, and no special tooling are required — a crafted JSON body to a documented API endpoint is sufficient. The inconsistency between the create and update DTOs suggests this was an oversight rather than intentional design, making it likely to remain undetected without targeted testing.

### Recommendation

Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
import { IsArray, IsNumber, IsOptional, Min, ValidateNested, ArrayMinSize } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @IsOptional()
  listId?: number;

  @IsNumber()
  @Min(1)          // ← add this
  @IsOptional()
  threshold?: number;
  ...
}
```

Additionally, update the service-level guard to explicitly reject non-positive thresholds (`threshold <= 0`) rather than only `threshold === 0`, providing defense-in-depth.

### Proof of Concept

**Preconditions:** Attacker has a valid authenticated session (normal user account).

**Steps:**

1. Create a transaction:
   ```
   POST /transactions
   { ...valid transaction body... }
   → returns { id: 42 }
   ```

2. Submit a negative-threshold approver tree:
   ```
   POST /transactions/42/approvers
   Authorization: Bearer <attacker_token>
   Content-Type: application/json

   {
     "approversArray": [{
       "threshold": -1,
       "approvers": [{ "userId": 99 }]
     }]
   }
   ```

3. **Expected (correct) behavior:** HTTP 400 — threshold must be ≥ 1.

4. **Actual behavior:** HTTP 201 — approver tree created with `threshold = -1` in the database.

5. **Result:** The approval-count check (`approvalCount >= -1`) is always true. The transaction proceeds as "approved" without user 99 (or anyone) actually approving it, bypassing the organizational authorization control. [4](#0-3) [3](#0-2) [5](#0-4)
