### Title
Missing Cryptographic Signature Binding in `processTransactionSignatures` Allows Fraudulent Signer Registration

### Summary

`processTransactionSignatures` in `signers.service.ts` validates that a submitted public key belongs to the authenticated user but never verifies that the accompanying signature bytes are cryptographically valid for the stored transaction body. A registered organization user can submit a signature map containing their public key paired with arbitrary bytes, be recorded as a signer, and advance the transaction to `WAITING_FOR_EXECUTION` — where it will ultimately fail at the Hedera network level.

### Finding Description

The vulnerability class is **missing binding constraint**: individual components (public key ownership) are checked, but the relationship between those components and the source value (the transaction body) is never enforced.

**Vulnerable path — `uploadSignatureMaps` → `processTransactionSignatures`:** [1](#0-0) 

The loop iterates over every public key in the submitted `SignatureMap`. For each key it:
1. Checks the key belongs to the authenticated user via `userKeyMap.get(raw)` — authorization only.
2. Calls `sdkTransaction.addSignature(publicKey, map)` — the Hedera SDK's `addSignature` does **not** cryptographically verify the signature bytes; it simply appends them and registers the public key in `_signerPublicKeys`.

There is no call to `validateSignature` anywhere in this path.

**Correct path — `importSignatures`:** [2](#0-1) 

This path calls `validateSignature` before `addSignature`. `validateSignature` retrieves the actual `bodyBytes` for each (node, transactionId) slot and calls `publicKey.verify(bodyBytes, signature)`, throwing on failure. [3](#0-2) 

**Downstream execution check — `hasValidSignatureKey`:** [4](#0-3) 

`hasValidSignatureKey` only checks that the required public keys are present in `_signerPublicKeys`. Because `addSignature` populates `_signerPublicKeys` regardless of signature validity, this check passes even when the signature bytes are garbage. [5](#0-4) 

### Impact Explanation

1. An authenticated organization user submits `POST /transactions/:id/signers` with their registered public key but with arbitrary/invalid signature bytes.
2. `processTransactionSignatures` records them as a signer in the `transaction_signer` table.
3. `processTransactionStatus` calls `hasValidSignatureKey` against `_signerPublicKeys` — which now includes the fraudulent key — and may advance the transaction to `WAITING_FOR_EXECUTION`.
4. The Chain Service picks up the transaction, calls `getValidatedSDKTransaction`, which again passes `hasValidSignatureKey`, and submits the transaction to Hedera.
5. Hedera rejects the transaction with an invalid-signature error, marking it `FAILED`.

Consequences: fraudulent signer registration, premature status advancement, forced transaction failure, and disruption of sequential `TransactionGroup` pipelines where one failed transaction blocks all subsequent ones.

### Likelihood Explanation

Any authenticated organization user who has at least one registered `UserKey` can reach `POST /transactions/:id/signers`. No elevated role is required. The attacker only needs to craft a `SignatureMap` with their real public key and substitute the signature bytes with zeros or random data. This is straightforward with the Hedera SDK.

### Recommendation

Add a call to `validateSignature` inside `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const { data: validKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMP);
```

Alternatively, inline the per-key verification: for each `(publicKey, signature)` pair, call `publicKey.verify(bodyBytes, signature)` and reject the entire batch if any signature is invalid.

### Proof of Concept

1. Create a transaction via `POST /transactions` (legitimate).
2. Freeze the SDK transaction locally but **do not sign it** — instead construct a `SignatureMap` manually with the user's registered public key and 64 zero bytes as the signature.
3. Submit `POST /transactions/:id/signers` with this forged map.
4. Observe: the server returns HTTP 201, the `transaction_signer` row is inserted, and `processTransactionStatus` may flip the status to `WAITING_FOR_EXECUTION`.
5. The Chain Service submits the transaction to Hedera; Hedera returns `INVALID_SIGNATURE`, and the transaction is marked `FAILED` — despite the signer never having held the private key. [6](#0-5) [2](#0-1)

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
  } else throw new Error(`Invalid key type`);
};
