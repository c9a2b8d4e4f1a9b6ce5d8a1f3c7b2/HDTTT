### Title
IPC Mechanism Mismatch: `searchPublicKeysAbort` Silently Fails Due to `ipcRenderer.send` vs `ipcMain.handle` Mismatch

---

### Summary

The `abortPublicKeySearch` handler is registered in the Electron main process using `ipcMain.handle` (which only responds to `ipcRenderer.invoke`), but the preload layer calls it using `ipcRenderer.send` (which only triggers `ipcMain.on` listeners). These two IPC mechanisms are incompatible. The abort call is silently dropped — the handler is never executed — making it impossible to cancel an in-progress public key search.

---

### Finding Description

In the main process IPC handler registration file, `abortPublicKeySearch` is registered via `createIPCChannel`, which internally calls `ipcMain.handle`: [1](#0-0) 

`createIPCChannel` exclusively uses `ipcMain.handle` for all entries: [2](#0-1) 

However, in the preload layer, the abort is dispatched using `ipcRenderer.send` — a fire-and-forget mechanism that only triggers `ipcMain.on` listeners, **not** `ipcMain.handle` listeners: [3](#0-2) 

Electron's IPC model is strict: `ipcMain.handle` ↔ `ipcRenderer.invoke` form one pair; `ipcMain.on` ↔ `ipcRenderer.send` form another. Mixing them means the message is silently discarded with no error thrown on either side.

---

### Impact Explanation

Every call to `searchPublicKeysAbort()` from the renderer is silently dropped. The `abortPublicKeySearch` service function is never invoked. Any in-progress file-system scan for public keys continues running indefinitely regardless of user intent to cancel. This can cause:

- UI state desync (abort button appears to work but the operation continues)
- Uncontrolled resource consumption (file I/O, CPU) from a scan that cannot be stopped
- Potential application hangs if the renderer waits on a cancelled state that never arrives

---

### Likelihood Explanation

This code path is triggered any time a user initiates and then cancels a public key search — a normal, expected UI interaction. The bug is deterministic: every abort attempt fails silently. No special conditions or attacker involvement are required.

---

### Recommendation

Change the main process registration to use `ipcMain.on` instead of routing through `createIPCChannel`/`ipcMain.handle` for the abort signal, since abort is a one-way notification with no return value:

```typescript
// In publicKeyMappings.ts (main process)
ipcMain.on('publicKeyMapping:searchPublicKeys:abort', () => {
  abortPublicKeySearch();
});
```

Or, alternatively, change the preload to use `ipcRenderer.invoke` to match the existing `ipcMain.handle` registration:

```typescript
// In publicKeyMapping.ts (preload)
searchPublicKeysAbort: () => ipcRenderer.invoke('publicKeyMapping:searchPublicKeys:abort'),
```

The first option (using `ipcMain.on` / `ipcRenderer.send`) is semantically more appropriate for a cancel signal that requires no response.

---

### Proof of Concept

1. In the renderer, trigger a public key search (calls `publicKeyMapping:searchPublicKeys` via `ipcRenderer.invoke` — this works correctly).
2. Before the search completes, call `searchPublicKeysAbort()`.
3. The preload dispatches `ipcRenderer.send('publicKeyMapping:searchPublicKeys:abort')`.
4. The main process has `ipcMain.handle('publicKeyMapping:searchPublicKeys:abort', ...)` registered — this listener is **never triggered** by `ipcRenderer.send`.
5. `abortPublicKeySearch()` is never called; the scan continues running.

```typescript
export const createIPCChannel = (
  channelName: string,
  handlers: Array<(...args: any[]) => void>,
) => {
  const createChannelName = (...props) => [channelName, ...props].join(':');

  for (let i = 0; i < handlers.length; i++) {
    const handler = handlers[i];
    const name = handler.name;
    if (!name) {
      throw new Error('IPC handler must have a name');
    }
    ipcMain.handle(createChannelName(name), (_e, ...args) => handler(...args));
  }
```
