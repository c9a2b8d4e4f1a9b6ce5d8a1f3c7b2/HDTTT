### Title
Uncaught Re-throw from Mirror Node Account Lookup in Frontend `computeSignatureKey` Causes Permanent Signing DoS When Any Referenced Account Is Deleted

### Summary

In `front-end/src/renderer/utils/transactionSignatureModels/transaction.model.ts`, the `computeSignatureKey()` method catches errors from `accountInfoCache.lookup()` for the fee payer, signing accounts, and receiver accounts — but immediately re-throws them. When any of those accounts is deleted from the Hedera network, the mirror node returns a 404 that propagates as a thrown error, causing `computeSignatureKey()` to abort entirely. This is the direct analog of the external report: just as a burned referrer profile causes `ownerOf()` to revert and blocks collection for everyone, a deleted Hedera account causes the mirror node lookup to throw and blocks the signing-key computation for all other participants in that multi-sig transaction.

### Finding Description

**Root cause — frontend `transaction.model.ts`:** [1](#0-0) 

```
if (feePayerAccountId) {
  try {
    const accountInfo = await accountInfoCache.lookup(…);
    …
  } catch (error) {
    logger.warn('Failed to resolve fee payer key', { error });
    throw error;          // ← re-thrown, aborts entire computeSignatureKey()
  }
}
```

The same re-throw pattern appears for signing accounts and receiver accounts: [2](#0-1) [3](#0-2) 

**Why the mirror node throws on a deleted account:**

The backend `MirrorNodeClient.fetchAccountInfo()` does not retry on 404 and re-throws the error: [4](#0-3) 

The frontend's `AccountByIdCache` calls the same REST endpoint; a 404 for a deleted account propagates as a thrown rejection.

**Propagation path:**

`accountInfoCache.lookup()` throws → caught in `transaction.model.ts` → **re-thrown** → `AppCache.computeSignatureKey()` throws (no try/catch there): [5](#0-4) 

**Callers that are affected:**

`usersPublicRequiredToSign()` calls `appCache.computeSignatureKey()` with no try/catch of its own: [6](#0-5) 

`collectMissingSignerKeys()` also calls `appCache.computeSignatureKey()` with no try/catch: [7](#0-6) 

**Contrast with the backend**, which handles this correctly — each account lookup is wrapped in an independent try/catch that logs and continues, so one missing account never aborts the whole computation: [8](#0-7) 

### Impact Explanation

When a Hedera account referenced in a pending multi-sig transaction (as fee payer, signing account, or receiver) is deleted from the network:

1. The mirror node returns 404 for that account.
2. `computeSignatureKey()` throws instead of skipping the missing account.
3. `isSignableTransaction()` returns `false` for all remaining participants — they cannot sign even if they hold all other required keys.
4. `collectMissingSignerKeys()` throws, breaking any workflow that calls it.
5. The transaction is permanently stuck in the UI for all organization members.

The transaction is effectively frozen in the tool's signing workflow with no recovery path short of a code fix.

### Likelihood Explanation

Hedera accounts can be deleted by their owner at any time using `AccountDeleteTransaction`. A participant in a multi-sig workflow can:

1. Be added as a required signer to a pending organization transaction.
2. Delete their own Hedera account.
3. All other participants immediately lose the ability to compute signing requirements for that transaction.

No privileged access is required. Any normal user with a Hedera account can trigger this condition. The scenario is realistic in any organization that manages multi-sig transactions with external or departing participants.

### Recommendation

Replace the re-throw pattern with the same graceful-skip approach used in the backend's `TransactionSignatureService`. When a mirror node lookup fails for a signing account, log the error and continue processing the remaining accounts rather than aborting the entire computation:

```typescript
// fee payer
try {
  const accountInfo = await accountInfoCache.lookup(feePayerAccountId.toString(), mirrorNodeLink);
  if (accountInfo?.key) { … }
} catch (error) {
  logger.warn('Failed to resolve fee payer key — skipping', { error });
  // do NOT re-throw
}

// signing accounts
for (const accountId of accounts) {
  try {
    const accountInfo = await accountInfoCache.lookup(accountId, mirrorNodeLink);
    if (accountInfo?.key && !hasKey(accountInfo.key)) { … }
  } catch (error) {
    logger.warn('Failed to resolve account signing key — skipping', { accountId, error });
    // do NOT re-throw
  }
}
```

This mirrors the already-correct backend implementation and ensures that one deleted or unreachable account does not abort the entire signature-key computation.

### Proof of Concept

1. Create an organization multi-sig transaction in the tool that requires signatures from accounts A and B.
2. From account B's owner, submit an `AccountDeleteTransaction` on Hedera to delete account B.
3. As account A's owner, open the pending transaction in the tool and attempt to sign it.
4. `computeSignatureKey()` calls `accountInfoCache.lookup('B', …)`, receives a 404, and re-throws.
5. `isSignableTransaction()` catches the error and returns `false`; the "Sign" button is disabled.
6. Account A's owner cannot sign the transaction despite holding a valid key, and the transaction is permanently stuck in `WAITING_FOR_SIGNATURES` state in the UI.
