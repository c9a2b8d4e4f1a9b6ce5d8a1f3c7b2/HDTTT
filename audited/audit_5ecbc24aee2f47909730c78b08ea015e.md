Based on my review of all controllers and services, I found two clear missing access control instances. The most impactful one is in `user-keys-all.controller.ts`.

---

### Title
Any Verified User Can Enumerate All Organization Users' Public Keys via Unguarded `/user-keys` Endpoint

### Summary
The `GET /user-keys` endpoint in `UserKeysAllController` is protected only by authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but lacks an `AdminGuard`. Any verified organization member can call this endpoint and retrieve the full paginated list of every user's public keys across the entire organization, with no ownership or role restriction enforced at the controller or service layer.

### Finding Description

**Root cause:** The `getUserKeys` handler in `UserKeysAllController` passes only pagination parameters to the service — no `user` context is injected, and no `AdminGuard` is applied.

```
@Controller('user-keys')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)  // ← no AdminGuard
export class UserKeysAllController {
  @Get()
  @Serialize(withPaginatedResponse(UserKeyDto))
  getUserKeys(
    @PaginationParams() paginationParams: Pagination,
    // ← no @GetUser(), no ownership filter
  ): Promise<PaginatedResourceDto<UserKey>> {
    return this.userKeysService.getUserKeys(paginationParams);
  }
}
```

Contrast this with the sibling controller `UserKeysController` (`GET /user/:userId/keys`), which correctly injects `@GetUser()` and delegates to `getUserKeysRestricted(user, userId)` — a method that enforces ownership.

The `UserKeysAllController` calls `getUserKeys(paginationParams)` with no user argument, meaning the service returns all keys for all users unfiltered.

**Exploit path:**
1. Attacker registers or is invited as a normal (non-admin) organization member.
2. Attacker sends `GET /user-keys?page=1&limit=100` with a valid JWT.
3. The response contains every user's `publicKey`, `mnemonicHash`, `index`, and `userId` for the entire organization — no admin privilege required.
4. Attacker iterates pages to enumerate the complete key inventory. [1](#0-0) [2](#0-1) 

### Impact Explanation

- **Cross-user data exposure:** Every verified user can read the Hedera public keys of every other user in the organization. Public keys directly identify Hedera accounts and are used to verify transaction signatures.
- **Reconnaissance for targeted attacks:** An attacker learns which accounts exist, their associated key indices, and mnemonic hashes — enabling targeted phishing, key-correlation attacks, or impersonation attempts.
- **Integrity of the multi-sig trust model is undermined:** The organization's key management model assumes keys are visible only to their owner and admins; this endpoint breaks that invariant for all verified users.

### Likelihood Explanation

- **Precondition:** Attacker only needs a valid, verified organization account — the lowest privilege level in the system.
- **No special tooling required:** A standard HTTP client with a valid JWT is sufficient.
- **Endpoint is publicly documented** via Swagger (`@ApiTags('User Keys All')`), making discovery trivial.

### Recommendation

Add `AdminGuard` to the controller-level `@UseGuards` decorator so only administrators can enumerate all keys:

```typescript
@Controller('user-keys')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard, AdminGuard)
export class UserKeysAllController { ... }
```

Alternatively, if non-admin access is intentional, inject `@GetUser()` and pass the user to a filtered service method (analogous to `getUserKeysRestricted`) so users can only see their own keys.

### Proof of Concept

**Attacker preconditions:** Valid JWT for any verified (non-admin) user.

```
GET /user-keys?page=1&limit=100
Authorization: Bearer <non-admin-jwt>

HTTP 200 OK
{
  "data": [
    { "id": 1, "userId": 2, "publicKey": "302a...", "mnemonicHash": "...", "index": 0 },
    { "id": 2, "userId": 3, "publicKey": "302a...", "mnemonicHash": "...", "index": 0 },
    ...
  ],
  "total": 87
}
```

The response contains keys belonging to other users — no admin privilege was used. [1](#0-0)
