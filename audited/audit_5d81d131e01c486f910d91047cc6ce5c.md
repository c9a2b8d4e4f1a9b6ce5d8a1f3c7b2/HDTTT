### Title
Missing Transaction-Level Access Control in `CommentsController::createComment` Allows Any Authenticated User to Post Comments on Arbitrary Transactions

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` applies only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but performs no check that the requesting user has any relationship to the target transaction. Any authenticated user — regardless of whether they are the creator, observer, signer, or approver of a transaction — can post comments on it. The developer acknowledged this gap with an explicit TODO comment in the source. The `GET` endpoints for comments also lack transaction-level access control, enabling cross-user data exposure.

### Finding Description
**Root cause:** `CommentsService::createComment` directly persists a comment for any `transactionId` supplied by the caller without verifying the caller's membership in that transaction.

**Code path:**

`CommentsController` at [1](#0-0)  applies only three guards at the controller level — `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — none of which check transaction membership. The `createComment` handler carries a developer-acknowledged TODO:

```
//TODO need some sort of guard or check to ensure user can comment here
``` [2](#0-1) 

`CommentsService::createComment` then blindly creates and saves the comment: [3](#0-2) 

**Contrast with the rest of the codebase:** Every other sensitive transaction operation enforces membership. For example, `TransactionsService::getTransactionWithVerifiedAccess` throws `UnauthorizedException` if the user is not creator, observer, signer, or approver: [4](#0-3) 

The `verifyAccess` helper that backs this check: [5](#0-4) 

The comments subsystem is the only area that skips this check entirely.

Additionally, `getComments` and `getCommentById` also have no transaction-level access control: [6](#0-5) 

**Exploit flow:**
1. Attacker registers a valid account and obtains a JWT.
2. Attacker enumerates sequential integer `transactionId` values (e.g., 1, 2, 3…).
3. For each ID, attacker calls `POST /transactions/<id>/comments` with arbitrary content.
4. The comment is persisted with the attacker's user identity attached to a transaction they have no legitimate relationship to.
5. Attacker also calls `GET /transactions/<id>/comments` to read all comments on any transaction, leaking internal discussion and metadata.

### Impact Explanation
- **Unauthorized state mutation:** Any authenticated user can inject comments into transactions they do not own, observe, sign, or approve — polluting the audit trail and collaboration history of other users' transactions.
- **Information disclosure:** Any authenticated user can read all comments on any transaction via `GET /transactions/:transactionId/comments`, exposing internal discussion, key decisions, and workflow context that should be scoped to transaction participants.
- **Transaction ID enumeration confirmation:** Successful comment creation on an ID confirms the transaction exists, leaking existence information across organizational boundaries.

### Likelihood Explanation
The attacker precondition is only a valid registered account and JWT — the lowest possible privilege bar in this system. The endpoint is a standard REST route with no rate-limiting or membership check. The developer's own TODO comment confirms the gap was known and unaddressed. Any malicious registered user can exploit this immediately.

### Recommendation
In `CommentsService::createComment`, call `TransactionsService::getTransactionWithVerifiedAccess` (or an equivalent membership check) before persisting the comment:

```typescript
async createComment(user: User, transactionId: number, dto: CreateCommentDto) {
  // Throws UnauthorizedException if user has no relationship to the transaction
  await this.transactionsService.getTransactionWithVerifiedAccess(transactionId, user);

  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);
}
```

Apply the same guard to `getComments` and `getCommentById` — either inline or via a shared guard — so that comment reads are also scoped to transaction participants.

### Proof of Concept

```
# Step 1: Register attacker account and obtain JWT
POST /auth/login
{ "email": "attacker@evil.com", "password": "..." }
→ { "accessToken": "<attacker_jwt>" }

# Step 2: Post a comment on a transaction the attacker has no access to
POST /transactions/1/comments
Authorization: Bearer <attacker_jwt>
Content-Type: application/json
{ "content": "Attacker-injected comment" }

# Expected (current): 201 Created — comment persisted on transaction #1
# Expected (fixed):   403 Unauthorized

# Step 3: Read all comments on any transaction
GET /transactions/1/comments
Authorization: Bearer <attacker_jwt>

# Expected (current): 200 OK — full comment history returned
# Expected (fixed):   403 Unauthorized
```

The comment is saved with `comment.user = user` (the attacker's identity), so it appears as a legitimate participant comment in the transaction's history, corrupting the audit trail.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
```
