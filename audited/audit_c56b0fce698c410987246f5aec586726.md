### Title
Unbounded Recursive Depth in `createTransactionApprovers` Enables DoS via Stack Overflow and DB Exhaustion

### Summary
The `createApprover` inner function inside `ApproversService.createTransactionApprovers` recurses into itself for every nested `approvers` entry in the request body, with no depth limit enforced. An authenticated transaction creator can submit a single POST request with an arbitrarily deep `approversArray` payload, causing a Node.js call-stack overflow that crashes the API process, or exhausting the database with thousands of inserts and recursive SQL queries inside a single transaction.

### Finding Description

**Root cause — unbounded recursion in `createApprover`:**

`createTransactionApprovers` defines an inner async function `createApprover` that calls itself for every child in `dtoApprover.approvers`:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
const createApprover = async (dtoApprover: CreateTransactionApproverDto) => {
  // ... validation, DB insert ...
  if (dtoApprover.approvers) {
    for (const nestedDtoApprover of dtoApprover.approvers) {
      await createApprover({ ...nestedDtoApprover, listId: approver.id }); // ← recursive, no depth guard
    }
  }
};
``` [1](#0-0) 

No depth counter, no maximum-depth constant, and no guard of any kind exists anywhere in this function or its callers.

**The DTO is self-referential with no depth constraint:**

```typescript
export class CreateTransactionApproverDto {
  @IsArray() @ArrayMinSize(1) @IsOptional()
  @ValidateNested({ each: true })
  @Type(() => CreateTransactionApproverDto)
  approvers?: CreateTransactionApproverDto[];
}
``` [2](#0-1) 

`class-validator` will itself recurse through the entire nested structure during deserialization and validation before the service is even reached.

**The exposed endpoint requires only a valid JWT and transaction ownership:** [3](#0-2) 

**Per-level DB cost inside the recursive call:**

Each recursion level executes:
1. `isNode` → 1 `COUNT` query
2. `getRootNodeFromNode` → 1 recursive SQL CTE query
3. `transactionalEntityManager.insert` → 1 `INSERT`
4. Optionally `getApproversByTransactionId` → 1 recursive SQL CTE query

All of these run inside a single open DB transaction. [4](#0-3) 

### Impact Explanation

**Call-stack overflow (crash):** Node.js has a default call-stack limit of ~10,000–15,000 frames. A payload with a linear chain of ~10,000 nested `approvers` objects triggers `RangeError: Maximum call stack size exceeded` inside the open DB transaction. The uncaught exception propagates out of the `dataSource.transaction` callback, crashes the async context, and can take down the NestJS API worker process.

**Database exhaustion (sustained):** Even at moderate depths (e.g., 500 levels, each with 5 children = 3,125 leaf nodes), the function issues thousands of SQL queries and inserts inside a single long-lived DB transaction, holding locks and consuming connection-pool slots until the transaction times out or the server OOMs.

**Severity: High** — a single authenticated HTTP request can crash the API service or render it unresponsive.

### Likelihood Explanation

**Attacker preconditions:**
- Valid JWT (registered, verified user) — achievable by any self-registered user.
- Ownership of at least one transaction — any user can create a transaction via the standard creation endpoint.

No admin or privileged role is required. The attack is a single crafted POST request. The payload is trivially generated programmatically (a deeply nested JSON object). There is no rate-limiting visible on this endpoint in the codebase.

### Recommendation

1. **Enforce a maximum nesting depth** inside `createApprover` by passing a `depth` counter and throwing if it exceeds a small constant (e.g., 5–10):

```typescript
const MAX_APPROVER_DEPTH = 5;
const createApprover = async (dto, depth = 0) => {
  if (depth > MAX_APPROVER_DEPTH)
    throw new Error('Approver tree exceeds maximum allowed depth');
  // ...
  for (const nested of dto.approvers ?? []) {
    await createApprover({ ...nested, listId: approver.id }, depth + 1);
  }
};
```

2. **Add a `@Max()` constraint on array sizes** in `CreateTransactionApproverDto` to cap the branching factor per level.

3. **Add a total-node cap** (e.g., reject if `approversArray` flattened exceeds 50 nodes) before entering the recursive loop.

### Proof of Concept

**Attacker steps:**

1. Register and verify a user account; obtain a JWT.
2. Create a transaction via `POST /transactions` to obtain a `transactionId`.
3. Craft a deeply nested payload:

```python
def make_nested(depth):
    if depth == 0:
        return {"userId": 1}
    return {"threshold": 1, "approvers": [make_nested(depth - 1)]}

payload = {"approversArray": [make_nested(15000)]}
```

4. Send:
```
POST /transactions/{transactionId}/approvers
Authorization: Bearer <jwt>
Content-Type: application/json

{"approversArray": [{"threshold":1,"approvers":[{"threshold":1,"approvers":[... 15000 levels ...]}]}]}
```

**Expected outcome:** The API process throws `RangeError: Maximum call stack size exceeded` inside `createApprover`, the NestJS exception filter may not recover the worker, and the service becomes unavailable to all users until restarted.
