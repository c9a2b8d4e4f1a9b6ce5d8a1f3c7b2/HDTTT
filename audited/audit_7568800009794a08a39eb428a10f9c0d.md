### Title
`COUNCIL_ACCOUNTS` Membership Check Always Returns False — Admin Key Bypass Protection for Council-Initiated `NodeDeleteTransaction` Is Completely Ineffective

### Summary

`COUNCIL_ACCOUNTS` is defined as a JavaScript **array** (`['0.0.2', '0.0.50', '0.0.55']`), but the membership check uses the `in` operator (`payerId.toString() in COUNCIL_ACCOUNTS`). The `in` operator on an array tests for **index keys** (`'0'`, `'1'`, `'2'`), not element values. As a result, `isCouncilAccount` is always `false` for every real council account ID, and the admin key is unconditionally required for every `NodeDeleteTransaction` — including those legitimately initiated by council accounts. The unit test that is supposed to validate this protection is also broken in the same way, giving false confidence that the guard works.

### Finding Description

**Root cause — wrong operator on wrong data structure**

`COUNCIL_ACCOUNTS` is declared as an array: [1](#0-0) 

Both the back-end model and the service perform the membership test with `in`: [2](#0-1) [3](#0-2) 

The front-end model contains the identical pattern: [4](#0-3) 

In JavaScript/TypeScript, `value in array` checks whether `value` is a **property name** of the array object — i.e., a numeric index string (`'0'`, `'1'`, `'2'`) or `'length'`. It never checks element values. Therefore:

```
'0.0.2'  in ['0.0.2', '0.0.50', '0.0.55']  // → false  (not an index)
'0.0.50' in ['0.0.2', '0.0.50', '0.0.55']  // → false
'0.0.55' in ['0.0.2', '0.0.50', '0.0.55']  // → false
```

`isCouncilAccount` is **always `false`**, so the guard branch that skips the admin key requirement is **never entered**.

**Consequence in the back-end service**

Because `isCouncilAccount` is always `false`, `signatureKey.push(nodeInfo.admin_key)` is always executed for `NodeDeleteTransaction`, regardless of who the fee payer is: [5](#0-4) 

**The test gives false confidence**

The spec uses `Object.keys(COUNCIL_ACCOUNTS)[0]` to obtain a "council account ID": [6](#0-5) 

`Object.keys` on an array returns its **index strings** (`['0', '1', '2']`), so `Object.keys(COUNCIL_ACCOUNTS)[0]` yields `'0'`, not `'0.0.2'`. The expression `'0' in ['0.0.2', '0.0.50', '0.0.55']` is `true` because `'0'` is a valid array index — so the test passes while testing a completely different (and meaningless) input. The actual council account IDs are never tested.

This is the direct structural analog to H-13: a protection mechanism that is supposed to work, a test that appears to validate it, but both are silently broken.

### Impact Explanation

The intended design is: when a privileged council account (`0.0.2`, `0.0.50`, `0.0.55`) is the fee payer of a `NodeDeleteTransaction`, its signature already satisfies the authorization requirement and the node admin key is not needed. Because the check is always `false`, the tool always demands the admin key signature.

Concrete effects:
1. **Governance operation lock**: Any council-initiated `NodeDeleteTransaction` submitted through the tool is permanently stuck in the "waiting for signatures" state unless the admin key holder co-signs. If the admin key holder is unavailable or adversarial, the operation cannot complete through the tool.
2. **Unintended veto power**: The admin key holder gains de-facto veto power over council node deletions — a privilege that the design explicitly intended to exclude.
3. **Integrity failure in the trust model**: The tool's computed required-signature set diverges from what the Hedera network actually requires, breaking the correctness guarantee of the signature workflow.

### Likelihood Explanation

Every `NodeDeleteTransaction` where the fee payer is a council account triggers this bug — there are no preconditions an attacker needs to satisfy. The broken test means the defect has survived code review and CI. Any organization using the tool for council-governed node management will encounter this on the first such transaction.

### Recommendation

Change `COUNCIL_ACCOUNTS` from an array to a `Set` (or a plain object used as a lookup map), and use the correct membership operator:

```typescript
// constants/index.ts
export const COUNCIL_ACCOUNTS = new Set(['0.0.2', '0.0.50', '0.0.55']);

// usage
const isCouncilAccount = payerId && COUNCIL_ACCOUNTS.has(payerId.toString());
```

Fix the unit test to use actual council account ID strings (e.g., `'0.0.2'`) rather than `Object.keys(COUNCIL_ACCOUNTS)[0]`, and add a negative test that confirms a non-council account ID (e.g., `'0.0.999'`) is correctly rejected.

Apply the same fix to the front-end model at: [4](#0-3) 

### Proof of Concept

```typescript
import { COUNCIL_ACCOUNTS } from '@app/common';
// COUNCIL_ACCOUNTS = ['0.0.2', '0.0.50', '0.0.55']  (array)

const payerId = '0.0.2';  // legitimate council account

// Broken check (current code):
console.log(payerId in COUNCIL_ACCOUNTS);
// → false  (checks index '0.0.2', which does not exist)

// The test's broken probe:
const testInput = Object.keys(COUNCIL_ACCOUNTS)[0];  // → '0'
console.log(testInput in COUNCIL_ACCOUNTS);
// → true   (checks index '0', which exists — test passes for wrong reason)

// Correct check:
const councilSet = new Set(COUNCIL_ACCOUNTS);
console.log(councilSet.has(payerId));
// → true   (correct)
```

**End-to-end exploit path**:
1. A council account (`0.0.2`) creates a `NodeDeleteTransaction` and submits it through the tool.
2. `computeSignatureKey` calls `addNodeKeys`, which evaluates `'0.0.2' in ['0.0.2', '0.0.50', '0.0.55']` → `false`.
3. `nodeInfo.admin_key` is pushed into the required key list.
4. The transaction is blocked in the tool's workflow waiting for the admin key signature.
5. If the admin key holder refuses to sign, the council cannot complete the node deletion through the tool — the admin key holder has acquired an unintended veto.
