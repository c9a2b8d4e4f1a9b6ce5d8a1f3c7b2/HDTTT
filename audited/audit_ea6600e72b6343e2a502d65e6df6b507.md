### Title
Unbounded Loop in `getTransactionsToSign` Causes DoS as Transaction Count Grows

### Summary
`TransactionsService.getTransactionsToSign` fetches **all** non-terminal transactions from the database with no row limit, then performs an async `userKeysToSign` call (involving DB queries and cryptographic computation) for every single transaction before applying pagination. As the number of active transactions grows, each request to `GET /transactions/sign` consumes unbounded memory and CPU, degrading or crashing the API service for all users.

### Finding Description
In `getTransactionsToSign`, the database query at line 295 has no `take` constraint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / limit
});
```

The method then loops over every returned transaction and awaits an expensive per-transaction check:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Pagination is only applied **after** the full loop completes:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
```

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `TransactionSignatureService.computeSignatureKey` — a cryptographic and DB-heavy operation — for every transaction in the unbounded set. [1](#0-0) 

The endpoint is publicly reachable by any authenticated user via `GET /transactions/sign`: [2](#0-1) 

### Impact Explanation
With N active transactions in the system, a single request to `GET /transactions/sign` triggers N sequential async DB + crypto operations in memory. At scale this causes:
- Memory exhaustion (all N transaction objects loaded simultaneously)
- CPU saturation (N cryptographic key computations per request)
- Request timeouts cascading to other API endpoints
- Complete API unavailability if requests are repeated

Any authenticated user — not just admins — can trigger this endpoint. [3](#0-2) 

### Likelihood Explanation
The endpoint is accessible to every verified user (guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). In an organization with many concurrent transactions, the condition is naturally met without any adversarial intent. A malicious user could also deliberately create many transactions to amplify the effect. [4](#0-3) 

### Recommendation
Apply the database-level limit **before** the loop, not after. Push the filtering logic into SQL rather than application memory:

1. Add `take: limit` and `skip: offset` to the `repo.find()` call, or rewrite the query to join signing requirements at the DB level.
2. If in-memory filtering is unavoidable, cap the fetch with a configurable `MAX_FETCH` constant and document the trade-off.
3. Consider a dedicated indexed query that pre-filters transactions requiring the user's keys, eliminating the per-row async call entirely.

### Proof of Concept
1. Register as a verified user on the API.
2. Create a large number of transactions (e.g., 10,000) in `WAITING_FOR_SIGNATURES` status.
3. Call `GET /transactions/sign?page=1&limit=10` repeatedly.
4. Observe that each request loads all 10,000 transactions into memory and performs 10,000 sequential `userKeysToSign` calls before returning 10 results — response time grows linearly with N, and memory usage spikes proportionally. [5](#0-4)

```typescript
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
