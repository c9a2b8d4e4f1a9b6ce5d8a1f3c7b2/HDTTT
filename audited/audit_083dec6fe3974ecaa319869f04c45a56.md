### Title
`isNode()` in `ApproversService` Never Returns `true` for Tree (Threshold) Approvers, Allowing Duplicate Tree Nodes

### Summary
`ApproversService.isNode()` is the duplicate-existence check called before every approver insertion. Its return expression unconditionally appends `&& typeof approver.userId === 'number'`, so it always returns `false` for threshold/tree approvers (those with no `userId`). The duplicate guard is therefore silently bypassed for every tree node, allowing an attacker who is the transaction creator to insert arbitrarily many identical threshold nodes into the approver tree.

### Finding Description
`isNode()` is defined at line 647–666 of `back-end/apps/api/src/transactions/approvers/approvers.service.ts`:

```typescript
async isNode(
  approver: CreateTransactionApproverDto,
  transactionId: number,
  entityManager?: EntityManager,
) {
  const find: FindManyOptions<TransactionApprover> = {
    where: {
      listId: typeof approver.listId === 'number' ? approver.listId : null,
      userId: typeof approver.userId === 'number' ? approver.userId : null,
      threshold: ...,
      transactionId: ...,
    },
  };

  const count = await (entityManager || this.repo).count(TransactionApprover, find);
  return count > 0 && typeof approver.userId === 'number';   // ← bug
}
```

The final expression `count > 0 && typeof approver.userId === 'number'` means:

| Approver type | `count > 0` | `typeof userId === 'number'` | Return value |
|---|---|---|---|
| User approver (has `userId`) | true | true | **true** ✓ |
| Tree approver (has `threshold`, no `userId`) | true | **false** | **false** ✗ |

For tree approvers the function always returns `false`, regardless of whether an identical node already exists in the database.

This function is the sole duplicate guard in `createTransactionApprovers()`:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
```

Because `isNode` never throws for tree approvers, the same threshold node can be inserted repeatedly.

### Impact Explanation
A transaction creator can POST to `POST /transactions/approvers` with a threshold-type approver DTO (no `userId`, only `threshold` + nested `approvers`) multiple times. Each call inserts a duplicate root threshold node. The approval tree becomes structurally corrupt: the chain service and any approval-counting logic now sees multiple root nodes for the same transaction, potentially allowing a transaction to be marked "fully approved" with fewer real signatures than the threshold requires, or causing the approval evaluation to behave unpredictably.

### Likelihood Explanation
The entry point is the authenticated `POST /transactions/approvers` endpoint, restricted to the transaction creator. Any legitimate creator can trigger this unintentionally or maliciously. No special privilege beyond being the transaction creator is required. The bug is in a code path exercised every time an approver tree is created.

### Recommendation
Remove the `&& typeof approver.userId === 'number'` guard from the return expression. The function should return `true` whenever a matching record already exists, regardless of approver type:

```typescript
// Before (buggy)
return count > 0 && typeof approver.userId === 'number';

// After (correct)
return count > 0;
```

### Proof of Concept
1. Authenticate as a user who owns a transaction (creator).
2. POST to `/transactions/approvers` with a threshold-type body:
   ```json
   {
     "approversArray": [{
       "threshold": 1,
       "approvers": [{ "userId": 42 }]
     }]
   }
   ```
3. Repeat the identical POST request.
4. Query the `transaction_approver` table — two identical root threshold nodes now exist for the same `transactionId`.
5. Observe that `isNode` returned `false` on the second call despite `count = 1`, because `typeof approver.userId === 'number'` evaluated to `false` for the threshold node. [1](#0-0) [2](#0-1)
