### Title
Missing Array Size Limit on Bulk Signature Endpoints Enables Authenticated DoS via Resource Exhaustion

### Summary
Both `POST /transactions/signatures/import` and `POST /transactions/:transactionId/signers` accept an unbounded array of `UploadSignatureMapDto` objects with no maximum-size guard. Any authenticated user can submit a single request containing thousands of entries, forcing the server to perform proportional cryptographic operations, SDK deserialization, and heavy relational database queries, exhausting CPU, memory, and DB connection resources. The codebase itself acknowledges the gap with the comment: *"Added a batch mechanism, probably should limit this on the api side of things."*

### Finding Description

**Root cause — no `ArrayMaxSize` on the body array**

`UploadSignatureMapDto` has no array-level size constraint: [1](#0-0) 

Both controllers accept `UploadSignatureMapDto | UploadSignatureMapDto[]` with no validation pipe that caps the array length: [2](#0-1) [3](#0-2) 

**Exploit path in `importSignatures`**

For every element in the unbounded array the service:
1. Issues a single `IN (…)` query loading all matching `Transaction` rows **with four eager relations** (`creatorKey`, `approvers`, `signers`, `observers`): [4](#0-3) 

2. Loops over every entry, calling `SDKTransaction.fromBytes()` (protobuf decode), `validateSignature()` (ECDSA/ED25519 verification), and `addSignature()` for each: [5](#0-4) 

3. Builds a raw SQL `CASE … END` bulk-update statement whose size grows linearly with the input array: [6](#0-5) 

The developer comment at line 575 explicitly flags the missing limit: [7](#0-6) 

The same unbounded pattern exists in `uploadSignatureMaps`: [8](#0-7) 

### Impact Explanation

A single crafted POST request with N=10,000 entries causes:
- One massive relational DB query (`IN` with 10,000 IDs, four JOINs) — potential DB OOM / connection exhaustion.
- 10,000 protobuf deserializations and cryptographic signature verifications — CPU spike proportional to N.
- A raw SQL `CASE` statement with 10,000 branches — potential query-planner failure or DB timeout.

The API process can be rendered unresponsive for all other users for the duration of the request, constituting a **service availability denial** for the entire organization backend.

### Likelihood Explanation

Any user who has completed registration (`VerifiedUserGuard`) can reach both endpoints — no admin or privileged role is required. The attacker only needs a valid JWT and knowledge of any transaction IDs (or can supply arbitrary IDs, since the service handles missing ones gracefully and still iterates). The attack is a single HTTP request, requires no special tooling, and is repeatable.

### Recommendation

1. Add `@ArrayMaxSize(N)` (e.g., `N = 100`) to the body array in both controllers, or introduce a dedicated wrapper DTO with the constraint.
2. Enforce the limit at the `transformAndValidateDto` helper level so it applies uniformly.
3. Consider rate-limiting the two endpoints per user.

### Proof of Concept

```
POST /transactions/signatures/import
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 2, "signatureMap": { ... } },
  ... // repeat 10,000 times with valid or arbitrary IDs
]
```

Expected outcome: the server spends seconds-to-minutes executing `SDKTransaction.fromBytes` + `validateSignature` for each entry, saturating the Node.js event loop and the PostgreSQL connection pool, making the API unresponsive to concurrent requests. [9](#0-8) [10](#0-9)
