### Title
Transaction Creator Can Self-Approve Their Own Transaction by Adding Themselves as an Approver

### Summary
The `createTransactionApprovers` function in `ApproversService` enforces that only the transaction creator can add approvers, but it does not prevent the creator from adding themselves as an approver. Once added, the creator can call `approveTransaction` — which only checks that the caller is in the approvers list — to approve their own transaction. This completely bypasses the organizational oversight that the approval workflow is designed to enforce.

### Finding Description

**Root Cause:**

`createTransactionApprovers` calls `getCreatorsTransaction` to verify the *caller* is the creator, but performs no check that the `userId` being added as an approver is different from the creator's own `userId`. [1](#0-0) 

Inside the inner `createApprover` loop, the only user-existence check is: [2](#0-1) 

There is no guard of the form `if (dtoApprover.userId === transaction.creatorKey.userId) throw ...`. The creator's own `userId` passes all validation and is inserted into `transaction_approver`.

`approveTransaction` then only checks that the caller appears in the approvers list: [3](#0-2) 

There is no check that `user.id !== transaction.creatorKey.userId`. Because the creator was inserted as an approver in step 1, this check passes and the creator's approval is recorded.

A second path exists via `updateTransactionApprover`: when updating an existing approver's `userId`, the same absence of a creator-identity check applies: [4](#0-3) 

**Exploit Flow:**

1. Attacker (User A) creates a transaction — they become the `creatorKey.userId`.
2. User A calls `POST /transactions/:id/approvers` with body `{ "approversArray": [{ "userId": A.id }] }`. This succeeds because `getCreatorsTransaction` only verifies the *caller* is the creator, not that the *target userId* is someone else.
3. User A calls `POST /transactions/:id/approvers/approve` with a valid signature. `approveTransaction` finds User A in `userApprovers`, passes all checks, and records the approval.
4. User A has now self-approved their own transaction.

### Impact Explanation
The approval workflow exists to enforce a separation of duties: a second, independent party must review and approve a transaction before it can proceed to execution. A creator who self-approves eliminates this control entirely. In an organizational context managing Hedera network operations (account creation, file operations, node management, fund transfers), this allows a single malicious or compromised user to unilaterally push transactions through the approval gate without any peer review, defeating the multi-party authorization model.

### Likelihood Explanation
The attacker is the transaction creator — a normal, authenticated user with no elevated privileges. The attack requires only two standard API calls that are part of the normal workflow. No race condition, cryptographic break, or social engineering is needed. Any organization member who can create transactions can exploit this.

### Recommendation
In `createTransactionApprovers`, after `getCreatorsTransaction` resolves the transaction, compare each `dtoApprover.userId` against `transaction.creatorKey.userId` and throw if they match:

```typescript
const transaction = await this.getCreatorsTransaction(transactionId, user);
// inside createApprover:
if (dtoApprover.userId === transaction.creatorKey.userId)
  throw new Error('The transaction creator cannot be added as an approver');
```

Apply the same guard in `updateTransactionApprover` when `dto.userId` is being set: [5](#0-4) 

Optionally, add a redundant check in `approveTransaction` that rejects the call if `user.id === transaction.creatorKey.userId`, providing defense-in-depth: [6](#0-5) 

### Proof of Concept

**Preconditions:** User A is a registered, verified organization member.

**Steps:**

```
# Step 1 — Create a transaction (User A becomes creator)
POST /transactions
Authorization: Bearer <User_A_JWT>
{ "creatorKeyId": <A_key_id>, "transactionBytes": "...", ... }
→ Response: { "id": 42, ... }

# Step 2 — Add self as approver
POST /transactions/42/approvers
Authorization: Bearer <User_A_JWT>
{ "approversArray": [{ "userId": <User_A_id> }] }
→ Response: [{ "id": 7, "userId": <User_A_id>, "transactionId": 42, ... }]
# Succeeds: getCreatorsTransaction passes (A is creator), no userId≠creator check exists

# Step 3 — Self-approve
POST /transactions/42/approvers/approve
Authorization: Bearer <User_A_JWT>
{ "userKeyId": <A_key_id>, "signature": "<valid_sig>", "approved": true }
→ Response: true
# Succeeds: userApprovers = [{ userId: A.id }], no creator-identity check in approveTransaction
```

**Expected (correct) behavior:** Step 2 should be rejected with an error such as `"The transaction creator cannot be added as an approver"`.

**Actual behavior:** Both steps succeed; User A's approval is recorded, bypassing the organizational approval gate.
