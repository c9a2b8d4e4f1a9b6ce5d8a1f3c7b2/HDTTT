### Title
Unbounded Per-Transaction Iteration in `getTransactionsToSign` Enables Authenticated DoS via O(N) Server-Side Work

### Summary
`TransactionsService.getTransactionsToSign()` fetches every active transaction in the database without a row limit, then iterates over all of them in a `for...of` loop, calling `userKeysToSign()` per transaction. Each call involves database queries and cryptographic deserialization. Pagination is applied only after the full loop completes. Any authenticated user can trigger this endpoint to force the server to perform unbounded work proportional to the total number of active transactions in the system.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` issues an unconstrained `repo.find()` with no `take` limit, then iterates over every returned transaction: [1](#0-0) 

The `find()` call at line 295 applies only a status filter — it has no `take` bound. Every active (non-terminal) transaction in the system is loaded into memory. The loop at line 301 then calls `userKeysToSign()` for each transaction: [2](#0-1) 

`userKeysToSign()` delegates to `userKeysRequiredToSign()`, which calls `attachKeys(user, entityManager)` on every iteration (a DB query per transaction), then calls `keysRequiredToSign()`: [3](#0-2) 

`keysRequiredToSign()` deserializes transaction bytes via `SDKTransaction.fromBytes()`, calls `transactionSignatureService.computeSignatureKey()`, and may issue additional DB queries to fetch `UserKey` records: [4](#0-3) 

Pagination is applied only after the full loop completes: [5](#0-4) 

There is no per-user transaction creation limit visible in `createTransaction()` or `createTransactions()`: [6](#0-5) 

### Impact Explanation

For N active transactions in the system, a single request to the `getTransactionsToSign` endpoint causes:
- N `SDKTransaction.fromBytes()` deserializations (CPU)
- N `computeSignatureKey()` calls (cryptographic computation)
- Up to N+1 `attachKeys()` database queries
- Potential additional `UserKey` DB lookups per transaction

An attacker who creates a large number of transactions (all in `WAITING_FOR_SIGNATURES` status, which never expires until the `validStart` window passes) and then repeatedly calls this endpoint can cause sustained CPU and database exhaustion, degrading or denying service to all other users of the API service.

### Likelihood Explanation

The attacker only needs a valid registered user account — no admin or privileged role is required. The `createTransaction` endpoint is accessible to any authenticated user with no visible per-user rate limit or transaction count cap. The attacker can pre-populate the system with many transactions and then repeatedly trigger the unbounded loop. The endpoint is a standard read operation that users are expected to call frequently (to check what they need to sign), making it a natural and unsuspicious target.

### Recommendation

Apply the `take` (limit) and `skip` (offset) pagination parameters directly to the `repo.find()` call, before the loop, rather than slicing the result afterward. Because `userKeysToSign` must be evaluated to determine inclusion, a two-phase approach is needed: first fetch a larger candidate window, evaluate keys, then return the paginated slice. Alternatively, push the key-matching logic into the database query (e.g., via a subquery on `publicKeys` stored on the transaction) to avoid per-row application-layer processing. At minimum, add a hard cap (e.g., `take: 500`) on the initial `find()` to bound worst-case work per request.

### Proof of Concept

1. Register a user account via the API.
2. Create a large number of transactions (e.g., 5,000) using `POST /transactions`, each with a future `validStart` so they remain in `WAITING_FOR_SIGNATURES` status.
3. Call `GET /transactions/sign` (the endpoint backed by `getTransactionsToSign`) with any pagination parameters (e.g., `page=1&size=10`).
4. Observe that the server processes all 5,000 transactions — performing 5,000 `SDKTransaction.fromBytes()` calls, 5,000 `computeSignatureKey()` calls, and up to 5,001 `attachKeys()` DB queries — before returning 10 results.
5. Repeat the request in a loop. Each request forces O(N) server-side work regardless of the requested page size, causing CPU and DB saturation and degrading service for all other users.

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
```

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
};
```
