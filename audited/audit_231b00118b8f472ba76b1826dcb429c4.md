### Title
Unbounded Approver Array in `createTransactionApprovers` Enables Authenticated DoS via DB Connection Exhaustion

### Summary
The `POST /transactions/:id/approvers` endpoint accepts an arbitrarily large and arbitrarily deep nested approver tree with no size or depth cap. The recursive `createApprover` function issues multiple database queries per node. A verified user can craft a single request containing thousands of approver nodes, exhausting the database connection pool and causing request timeouts that degrade service for all concurrent users.

### Finding Description

**Vulnerability class:** DoS — unbounded resource consumption via attacker-controlled collection size.

**Root cause — missing array size constraints in DTOs:**

`CreateTransactionApproversArrayDto` has no `@ArrayMaxSize()` on `approversArray`, and `CreateTransactionApproverDto` has no `@ArrayMaxSize()` on the nested `approvers` field. [1](#0-0) 

Both arrays accept unlimited entries. The outer array can contain thousands of root-level approvers; each root-level approver can itself carry an arbitrarily deep nested `approvers` tree.

**Root cause — recursive processing with per-node DB queries:**

`createTransactionApprovers` calls the inner async `createApprover` recursively for every node in the tree: [2](#0-1) 

For each node the function issues at minimum:
1. `isNode` check (DB query)
2. `getRootNodeFromNode` recursive CTE (DB query)
3. `count(User)` existence check (DB query)
4. `getApproversByTransactionId` recursive CTE (DB query)
5. `insert(TransactionApprover)` (DB write)

With N approver nodes, the single HTTP request triggers ≥ 5N synchronous database round-trips inside one TypeORM transaction, holding a connection for the entire duration.

**Attack path:**

1. Attacker registers a normal verified user account (no privilege required).
2. Attacker creates a valid transaction — `POST /transactions`.
3. Attacker sends `POST /transactions/:id/approvers` with a body containing thousands of flat root-level approver nodes, e.g.:
```json
{
  "approversArray": [
    { "userId": 1 },
    { "userId": 2 },
    ...  // repeated N thousand times
  ]
}
```
4. The server processes all N nodes sequentially inside a single DB transaction, issuing 5N+ queries while holding one connection from the pool.
5. Repeated requests from the attacker saturate the connection pool; legitimate users' queries queue or time out.

A deeply nested variant (each node wrapping another in `approvers`) additionally risks a Node.js call-stack overflow because `createApprover` is a recursive async function with no depth guard.

**No size guard exists anywhere in the call chain** — not in the DTO, not in the controller, not in the service. [3](#0-2) 

### Impact Explanation

**Impact: High**

- A single crafted request with ~5,000 approver nodes issues ~25,000 DB queries while holding a connection, stalling the pool for all concurrent users.
- Repeated requests cause sustained service unavailability (HTTP 503 / timeouts) for the entire API.
- A deeply nested tree (e.g., depth 10,000) triggers a JavaScript call-stack overflow, crashing the NestJS worker process.
- No data theft is required; the attacker only needs a valid account.

### Likelihood Explanation

**Likelihood: Low**

- Requires a registered, verified user account — not fully open to anonymous callers.
- However, user registration is a standard product flow with no stated rate limit or invite-only gate visible in the codebase.
- The attack is trivially scriptable: one HTTP request with a large JSON body.
- No existing guard (DTO validator, middleware, or service logic) prevents it.

### Recommendation

1. **Add `@ArrayMaxSize(N)` to both DTO arrays** — choose a small practical limit (e.g., 50 root approvers, 50 children per node):

```typescript
// create-transaction-approver.dto.ts
import { ArrayMaxSize, ArrayMinSize, IsArray, IsNumber, IsOptional, ValidateNested } from 'class-validator';

export class CreateTransactionApproverDto {
  // ...
  @IsArray()
  @ArrayMinSize(1)
  @ArrayMaxSize(50)   // add this
  @IsOptional()
  @ValidateNested({ each: true })
  @Type(() => CreateTransactionApproverDto)
  approvers?: CreateTransactionApproverDto[];
}

export class CreateTransactionApproversArrayDto {
  @IsArray()
  @ArrayMaxSize(50)   // add this
  @ValidateNested({ each: true })
  @Type(() => CreateTransactionApproverDto)
  approversArray: CreateTransactionApproverDto[];
}
```

2. **Add a recursion depth guard** inside `createApprover` and reject requests that exceed a maximum nesting depth (e.g., 5 levels).

3. **Apply the same fix to `CreateTransactionObserversDto`** — `userIds` also has no `@ArrayMaxSize()`: [4](#0-3) 

### Proof of Concept

**Preconditions:** Attacker has a valid JWT for a verified user who owns transaction ID `42`.

**Request:**
```http
POST /transactions/42/approvers
Authorization: Bearer <attacker_token>
Content-Type: application/json

{
  "approversArray": [
    { "userId": 1 }, { "userId": 2 }, { "userId": 3 },
    ... (5000 entries)
  ]
}
```

**Expected outcome:**
- The server enters `createTransactionApprovers`, acquires a DB connection, and begins issuing ≥ 25,000 sequential queries.
- The DB connection pool (default TypeORM pool size: 10) is saturated within 2–3 concurrent such requests.
- All other API calls that require a DB connection (login, transaction fetch, signing) queue indefinitely or return 503/timeout errors.
- The attacker can sustain the DoS by repeating the request as each prior one eventually times out.

**Deeply nested variant (stack overflow):**
```json
{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{
      "threshold": 1,
      "approvers": [{
        ... // 10,000 levels deep
      }]
    }]
  }]
}
```
This triggers unbounded recursion in `createApprover` → `createApprover` → … until Node.js throws `RangeError: Maximum call stack size exceeded`, crashing the worker. [5](#0-4)
