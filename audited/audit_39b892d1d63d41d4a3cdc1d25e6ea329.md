### Title
Any Authenticated User Can Enumerate All Organization User Keys Due to Missing Ownership Filter in `GET /user-keys`

### Summary
The `UserKeysAllController` exposes a `GET /user-keys` endpoint that is protected by authentication guards but contains no ownership or authorization filter. The underlying `UserKeysService.getUserKeys()` method fetches every `UserKey` record in the database without scoping the query to the requesting user. Any authenticated organization member can paginate through all user keys belonging to all other users in the system.

### Finding Description

**Root cause — controller passes no user identity to the service:**

`back-end/apps/api/src/user-keys/user-keys-all.controller.ts` lines 32–38:

```typescript
@Get()
@Serialize(withPaginatedResponse(UserKeyDto))
getUserKeys(
  @PaginationParams() paginationParams: Pagination,
): Promise<PaginatedResourceDto<UserKey>> {
  return this.userKeysService.getUserKeys(paginationParams);
}
```

The `@GetUser()` decorator is never used. The authenticated user's identity is silently discarded. [1](#0-0) 

**Root cause — service performs an unfiltered `findAndCount`:**

`back-end/apps/api/src/user-keys/user-keys.service.ts` lines 139–156:

```typescript
async getUserKeys({ page, limit, size, offset }: Pagination) {
  const [items, total] = await this.repo.findAndCount({
    take: limit,
    skip: offset,
  });
  ...
}
```

There is no `where: { userId: ... }` clause. Every row in the `user_key` table is returned. [2](#0-1) 

**Contrast with the correctly scoped sibling method** `getUserKeysRestricted`, which filters by `userId` and conditionally hides `mnemonicHash` / `index` from non-owners: [3](#0-2) 

**The controller-level guards only verify authentication, not authorization:** [4](#0-3) 

`JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` confirm the caller holds a valid, non-blacklisted JWT and has an active account status. None of them restrict which records the caller may read.

**Exploit path:**
1. Attacker registers (or is invited) as a normal organization user and obtains a valid JWT.
2. Attacker sends `GET /user-keys?page=1&limit=100` with their bearer token.
3. The endpoint returns paginated `UserKey` records for **all** users in the organization — public keys, and any additional fields serialized by `UserKeyDto` (potentially `mnemonicHash`, `index`).
4. Attacker repeats with increasing `page` values to enumerate the entire table.

### Impact Explanation
An attacker with a valid organization account can harvest the public keys of every other user. Public keys are used to identify signers in multi-signature Hedera transactions; knowing them allows targeted transaction manipulation (e.g., crafting transactions that appear to require a specific signer's key). If `UserKeyDto` serializes `mnemonicHash` or `index`, the exposure is more severe: `mnemonicHash` is a hash of the user's BIP-39 mnemonic and could be used for offline dictionary attacks to recover the private key. [3](#0-2) 

### Likelihood Explanation
Likelihood is **high**. The attacker precondition is only a valid organization account — no admin role, no leaked secrets, no internal network access. The endpoint is reachable over standard HTTPS. The exploit is a single authenticated HTTP GET request.

### Recommendation
Pass the authenticated user into `getUserKeys` and add a `where: { userId: user.id }` clause, mirroring the pattern already used in `getUserKeysRestricted`:

```typescript
// user-keys-all.controller.ts
@Get()
@Serialize(withPaginatedResponse(UserKeyDto))
getUserKeys(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
): Promise<PaginatedResourceDto<UserKey>> {
  return this.userKeysService.getUserKeys(user, paginationParams);
}

// user-keys.service.ts
async getUserKeys(user: User, { page, limit, size, offset }: Pagination) {
  const [items, total] = await this.repo.findAndCount({
    where: { userId: user.id },
    take: limit,
    skip: offset,
  });
  ...
}
```

Additionally, audit `UserKeyDto` to ensure `mnemonicHash` and `index` are excluded from the serialized output for this endpoint, consistent with the field-level restrictions already present in `getUserKeysRestricted`. [3](#0-2) 

### Proof of Concept

**Preconditions:** Two organization accounts exist — `victim@org.com` (has uploaded keys) and `attacker@org.com` (normal user, no special role).

**Steps:**
1. Attacker logs in: `POST /auth/login` → receives `accessToken`.
2. Attacker sends:
   ```
   GET /user-keys?page=1&limit=100
   Authorization: Bearer <attacker_accessToken>
   ```
3. **Expected (correct) behavior:** Response contains only the attacker's own keys.
4. **Actual behavior:** Response contains `UserKey` records for all users in the organization, including the victim's `publicKey` (and potentially `mnemonicHash`, `index`).
