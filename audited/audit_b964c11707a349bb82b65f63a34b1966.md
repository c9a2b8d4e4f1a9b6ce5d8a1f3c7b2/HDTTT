### Title
Unvalidated `mirrorNetwork` Field Causes Transactions to Permanently Stall in `WAITING_FOR_EXECUTION` via External Call Outside Error Boundary

### Summary

The `mirrorNetwork` field in `CreateTransactionDto` accepts arbitrary string values with no whitelist enforcement. When the chain service executes a transaction, `_executeTransaction` calls `getClientFromNetwork(transaction.mirrorNetwork)` **outside** the try-catch block. For any non-standard network value, this triggers an external `AddressBookQuery` to the attacker-controlled URL. If that call throws or hangs, the exception escapes the error boundary entirely — the transaction status is never updated and the record is permanently stranded in `WAITING_FOR_EXECUTION`.

### Finding Description

**Step 1 — No whitelist on `mirrorNetwork` at ingestion.**

`CreateTransactionDto` only requires the field to be a non-empty string: [1](#0-0) 

No `@IsIn([...])` or equivalent guard exists. Any authenticated user may supply an arbitrary hostname.

**Step 2 — The raw value is persisted to the database.**

`createTransactions` stores `dto.mirrorNetwork` verbatim: [2](#0-1) 

**Step 3 — During creation, `getClientFromNetwork` is also called with the custom value.** [3](#0-2) 

For a custom network, this issues an `AddressBookQuery` to the supplied URL. An attacker who controls the target host can make it respond successfully here, allowing the transaction to be saved.

**Step 4 — `getClientFromNetwork` for custom networks makes an outbound call to the stored URL.** [4](#0-3) 

**Step 5 — During execution, the same call is made OUTSIDE the try-catch.**

In `_executeTransaction`, `getClientFromNetwork` is invoked at line 132, before the `try` block that handles execution errors and updates the transaction status: [5](#0-4) 

If `getClientFromNetwork` throws (or hangs indefinitely because the attacker's server stops responding), the exception propagates out of `_executeTransaction` without ever reaching the `update` call at lines 187–196 that would set the status to `FAILED`. The `finally { client.close() }` block is also never reached because `client` was never assigned.

**Step 6 — The scheduler swallows the error and removes the timeout, leaving no retry.**

The scheduler's callback wraps the execution in a try-catch that only logs the error: [6](#0-5) 

The timeout is deleted in `finally`. No retry is scheduled. The transaction record remains in `WAITING_FOR_EXECUTION` indefinitely with no automated recovery path.

### Impact Explanation

A transaction stuck in `WAITING_FOR_EXECUTION` cannot be canceled by the creator (only `WAITING_FOR_SIGNATURES` and similar pre-execution statuses are cancelable), cannot be re-executed, and is invisible to the normal history view. If the attacker's server hangs rather than refusing the connection, the execution goroutine blocks for the full SDK timeout duration, consuming a thread/connection slot on every retry attempt. In an organization with many signers, a single malicious transaction can tie up execution resources and prevent legitimate transactions from being processed by the scheduler.

### Likelihood Explanation

Any authenticated user of the backend API can submit a `POST /transactions` request with an arbitrary `mirrorNetwork` value. The attacker needs:
1. A valid account on the backend (normal user, no admin role required).
2. A server they control that responds to the `AddressBookQuery` during creation but hangs or drops connections during execution.

This is a low-privilege, single-request attack. No brute force or timing precision is required.

### Recommendation

**Short term:** Move `getClientFromNetwork` inside the try-catch in `_executeTransaction` so that any failure is caught, the transaction is marked `FAILED`, and the status code is recorded:

```typescript
try {
  const client = await getClientFromNetwork(transaction.mirrorNetwork);
  try {
    const response = await sdkTransaction.execute(client);
    ...
  } finally {
    client.close();
  }
} catch (error) {
  // mark transaction FAILED, record statusCode
}
```

**Long term:** Enforce a whitelist of allowed `mirrorNetwork` values in `CreateTransactionDto` using `@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])` or a configurable allowlist, preventing arbitrary external URLs from being stored and later contacted by the execution service. [1](#0-0) 

### Proof of Concept

1. Stand up a server at `attacker.example.com` that responds to gRPC `AddressBookQuery` with a valid address book (so creation succeeds).
2. Authenticate to the backend API and submit:
   ```json
   POST /transactions
   {
     "name": "test",
     "description": "poc",
     "transactionBytes": "<valid signed tx bytes>",
     "creatorKeyId": <your key id>,
     "signature": "<valid sig>",
     "mirrorNetwork": "attacker.example.com"
   }
   ```
3. Confirm the transaction is saved with status `WAITING_FOR_SIGNATURES`.
4. Have enough signers sign it so it transitions to `WAITING_FOR_EXECUTION`.
5. At this point, configure `attacker.example.com` to drop all connections (or never respond).
6. Observe: the scheduler fires `collateAndExecute`, which calls `_executeTransaction`. `getClientFromNetwork("attacker.example.com")` hangs until the SDK timeout, then throws. The scheduler logs the error and deletes the timeout. The transaction status remains `WAITING_FOR_EXECUTION` permanently — it cannot be canceled, re-executed, or automatically recovered.
