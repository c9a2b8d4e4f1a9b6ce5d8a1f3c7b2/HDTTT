### Title
Missing Creator Authorization Check in `removeTransactionApprover` Allows Any Authenticated User to Delete Approvers from Any Transaction

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` performs no ownership or creator verification before deleting a transaction approver node. Every other mutating approver operation (`createTransactionApprovers`, `updateTransactionApprover`) gates itself behind `getCreatorsTransaction`, which enforces `transaction.creatorKey?.userId !== user.id`. `removeTransactionApprover` accepts only an approver `id`, never a `User`, and therefore cannot perform — and does not perform — any such check. Any authenticated user can delete approvers belonging to transactions they did not create.

### Finding Description

The vulnerability class from the external report is **inverted/absent authorization check on a privileged state-mutating operation**. In the external Solidity report the guard used `!=` instead of `==`, letting everyone through except the legitimate owner. The analog here is that the guard is **entirely absent**.

`createTransactionApprovers` and `updateTransactionApprover` both call `getCreatorsTransaction` before mutating state: [1](#0-0) [2](#0-1) 

`getCreatorsTransaction` enforces the ownership invariant: [3](#0-2) 

`removeTransactionApprover`, however, accepts no `User` argument and performs no such check: [4](#0-3) 

The function only verifies the approver record exists, then immediately deletes it and emits a status update. Because the `User` is never passed in, no ownership check is structurally possible inside the function. [5](#0-4) 

### Impact Explanation

An authenticated attacker (any registered user with a valid JWT) can:

1. Enumerate approver IDs for any transaction (approver IDs are sequential integers).
2. Call `DELETE /transactions/:transactionId/approvers/:id` with an approver ID belonging to a transaction they did not create.
3. The approver record — and its entire subtree via `removeNode` — is permanently deleted.

This breaks the multi-signature approval workflow for the victim transaction: required approvers are silently removed, allowing a transaction to advance through the approval gate with fewer (or zero) approvals than the creator intended. This is a **permanent, unrecoverable state corruption** of the approval structure for any active transaction. [4](#0-3) 

### Likelihood Explanation

- **Attacker precondition**: valid JWT only — no admin role, no leaked credentials, no privileged access.
- **Attack path**: single authenticated HTTP DELETE request with a guessable integer approver ID.
- **Discoverability**: the asymmetry between `create`/`update` (which require creator check) and `remove` (which does not) is immediately visible to any code reviewer or attacker reading the API surface.
- **Likelihood: High** — the entry point is reachable by any registered user with no additional preconditions.

### Recommendation

Add a creator ownership check to `removeTransactionApprover` by accepting a `User` parameter and calling `getCreatorsTransaction` before deletion, mirroring the pattern already used in `createTransactionApprovers` and `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Enforce creator-only access, consistent with create/update
  await this.getCreatorsTransaction(approver.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

The corresponding controller handler must also pass the authenticated `user` from `@GetUser()` to the service call. [3](#0-2) 

### Proof of Concept

**Preconditions:**
- User A creates a transaction with approver records (approver IDs are integers, e.g. `42`).
- User B is any other authenticated user.

**Steps:**
1. User B authenticates and obtains a JWT.
2. User B sends: `DELETE /transactions/1/approvers/42` with their JWT.
3. `removeTransactionApprover(42)` is called — no user check occurs.
4. `removeNode(42)` soft-deletes approver `42` and its entire subtree.
5. The approval threshold for User A's transaction is now broken; the transaction may proceed with insufficient approvals.

**Expected result:** `403 Unauthorized` — only the transaction creator may remove approvers.
**Actual result:** `200 OK` — approver deleted, approval workflow permanently corrupted. [4](#0-3)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
