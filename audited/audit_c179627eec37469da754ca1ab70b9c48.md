### Title
`removeTransactionApprover` Does Not Update Parent Threshold After Child Removal, Causing Unsatisfiable Approval Requirements

### Summary
When a child approver node is removed from a threshold-based approver tree via `removeTransactionApprover`, the parent node's `threshold` value is never decremented. This leaves the parent with a threshold that exceeds the number of remaining child approvers, making the approval requirement permanently unsatisfiable for that transaction. The `updateTransactionApprover` path (detach-to-root) correctly adjusts the parent threshold, but the delete path does not, creating an inconsistency that can permanently lock a transaction in `WAITING_FOR_SIGNATURES`.

### Finding Description

**Root cause — `removeTransactionApprover` / `removeNode` never touches the parent's threshold:**

`removeTransactionApprover` fetches the target approver and delegates to `removeNode`: [1](#0-0) 

`removeNode` executes a recursive CTE that soft-deletes the target node and all its descendants, but never reads or updates the parent row: [2](#0-1) 

**Contrast with `updateTransactionApprover` (detach path), which does update the parent:**

When an approver is detached from its parent via `dto.listId = null`, the code explicitly recalculates and writes the parent's threshold: [3](#0-2) 

The delete path has no equivalent logic.

**Entry point — no transaction-status guard on the delete endpoint:**

`getCreatorsTransaction` only verifies the caller is the creator; it does not check whether the transaction is still in a mutable state: [4](#0-3) 

So the creator can call `DELETE /transactions/:transactionId/approvers/:id` at any point, including after the transaction has entered `WAITING_FOR_SIGNATURES`. [5](#0-4) 

**Threshold invariant enforced at creation but not at deletion:**

At creation time the system validates `threshold ≤ approvers.length`: [6](#0-5) 

No equivalent check runs after a child is removed, so the invariant can be silently violated.

### Impact Explanation

Consider a threshold tree: `Root (threshold=2) → [User A, User B]`.

1. Creator calls `DELETE /transactions/T/approvers/{User_A_id}`.
2. `removeNode` soft-deletes User A's record.
3. Root still has `threshold = 2` in the database, but only User B remains.
4. User B approves — the system still requires 2 approvals.
5. No second approver exists; the threshold can never be reached.
6. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` unless the creator manually lowers the threshold via a separate `PATCH` call — a non-obvious recovery step.

All other participants (signers, observers, the intended approvers) are blocked from seeing the transaction progress. The transaction cannot be executed and cannot be automatically expired or cleaned up by the chain service, which only acts on `WAITING_FOR_EXECUTION` status. [7](#0-6) 

### Likelihood Explanation

The creator is a normal authenticated user with no special privileges. The delete endpoint is a standard REST call. The inconsistency between the delete path (no threshold update) and the detach-to-root path (threshold updated) means a creator who removes a child approver expecting the threshold to auto-adjust — a reasonable expectation given the UI's `getThreshold` helper does exactly that — will silently corrupt the approval tree. [8](#0-7) 

The front-end `getThreshold` function caps the threshold at `newApprovers.length` on every UI removal, so users are conditioned to expect automatic adjustment. The back-end delete path violates this expectation.

### Recommendation

In `removeTransactionApprover` (or inside `removeNode`), after soft-deleting the target node, fetch the parent (if any) and apply the same threshold-correction logic already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id); // includes .approvers relation
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // If this node has a parent, fix the parent's threshold before deleting
  if (approver.listId) {
    const parent = await this.repo.findOne({
      relations: ['approvers'],
      where: { id: approver.listId },
    });
    if (parent) {
      const newChildCount = parent.approvers.length - 1;
      if (newChildCount === 0) {
        await this.repo.softRemove(parent);
      } else if (newChildCount < parent.threshold) {
        await this.repo.update(parent.id, { threshold: newChildCount });
      }
    }
  }

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
}
```

### Proof of Concept

1. Authenticate as User C (creator).
2. `POST /transactions` — create transaction T.
3. `POST /transactions/T/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 2, "approvers": [{"userId": 1}, {"userId": 2}] }] }
   ```
   This creates Root (threshold=2) with children User 1 and User 2.
4. `DELETE /transactions/T/approvers/{child_user_1_id}` — removes User 1.
5. Query the database: Root still has `threshold = 2`, but only User 2 remains as a child.
6. User 2 calls `POST /transactions/T/approvers/approve` — approval is recorded.
7. Observe: transaction never advances to `WAITING_FOR_EXECUTION` because the threshold of 2 is never reached with only 1 approver available.
8. Transaction is permanently stuck in `WAITING_FOR_SIGNATURES`. [9](#0-8)
