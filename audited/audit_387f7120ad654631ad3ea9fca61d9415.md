### Title
Missing Lower-Bound Validation on `threshold` in `CreateTransactionApproverDto` Allows Negative Threshold to Bypass Approval Requirements

### Summary
The `CreateTransactionApproverDto` accepts any numeric value for `threshold`, including zero and negative integers, because no `@Min(1)` constraint is applied at the DTO layer. The service-level guard in `ApproversService.createTransactionApprovers` only rejects `threshold === 0` and `threshold > approvers.length`, leaving negative values unchecked. A negative threshold stored in the database causes the approval-count comparison (`approvedCount >= threshold`) to be trivially satisfied, allowing a transaction creator to bypass the multi-signature approval requirement entirely.

### Finding Description

**Root cause — DTO layer:**

`CreateTransactionApproverDto.threshold` carries only `@IsNumber()` and `@IsOptional()`. There is no `@Min(1)` decorator. [1](#0-0) 

Compare with `UpdateTransactionApproverDto`, which correctly adds `@Min(1)`: [2](#0-1) 

**Root cause — service layer:**

The service guard checks `threshold === 0` and `threshold > approvers.length`, but never checks `threshold < 0`: [3](#0-2) 

A value of `-1` with one child approver evaluates as:
- `-1 === 0` → `false`
- `-1 > 1` → `false`

Both guards pass, and the negative threshold is persisted to the database.

**Exploit path:**

1. Attacker is a normal authenticated user who creates a transaction in an organization.
2. Attacker calls `POST /transactions/{id}/approvers` with body `{ "approversArray": [{ "threshold": -1, "approvers": [{ "userId": <victim_id> }] }] }`.
3. The DTO passes class-validator (no `@Min` constraint). The service guard passes (neither `=== 0` nor `> 1`). The record is inserted with `threshold = -1`.
4. When the chain service evaluates whether the approval tree is satisfied, any non-negative approval count satisfies `approvedCount >= -1`, so the tree is considered approved with zero actual approvals.
5. The transaction proceeds to execution without the required multi-party sign-off. [4](#0-3) 

### Impact Explanation
A transaction creator can unilaterally bypass the organization's multi-signature approval gate by injecting a negative threshold. This breaks the core trust model of the organization mode: transactions that are supposed to require N-of-M approvals from other members execute with zero approvals. This constitutes an unauthorized state change and integrity failure in the approval/consensus flow.

### Likelihood Explanation
Any authenticated user who can create a transaction (i.e., every normal organization member) can trigger this. No privileged credentials are required. The API endpoint is reachable over HTTPS, and the malformed payload is trivially constructed. The discrepancy between the CREATE and UPDATE DTOs (`@Min(1)` present only in the latter) suggests this was an oversight rather than intentional design.

### Recommendation
Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring the existing constraint in `UpdateTransactionApproverDto`:

```typescript
// create-transaction-approver.dto.ts
import { IsArray, IsNumber, IsOptional, Min, ValidateNested, ArrayMinSize } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @IsOptional()
  listId?: number;

  @IsNumber()
  @Min(1)          // ← add this
  @IsOptional()
  threshold?: number;
  ...
}
```

Additionally, harden the service guard to reject any `threshold < 1` explicitly:

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold < 1 || dtoApprover.threshold > dtoApprover.approvers.length)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
``` [3](#0-2) 

### Proof of Concept

```
POST /api/transactions/42/approvers
Authorization: Bearer <normal_user_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 99 }
      ]
    }
  ]
}
```

Expected (correct) behavior: `400 Bad Request` — threshold must be ≥ 1.

Actual behavior: `201 Created` — approver tree stored with `threshold = -1`. The approval gate is trivially satisfied with zero approvals, and the transaction advances to execution without any member sign-off.
