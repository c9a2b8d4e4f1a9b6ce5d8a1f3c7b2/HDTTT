### Title
Unbounded Full-Table Scan with Per-Row Async Computation in `getTransactionsToSign` Enables DoS

### Summary
`GET /transactions/sign` calls `getTransactionsToSign`, which fetches **every non-terminal transaction in the entire database** with no row limit, then performs an expensive async `computeSignatureKey` call (deserialization + external account/node cache lookups) for each row sequentially. Pagination is applied only after the full in-memory scan completes. Any authenticated user can create many transactions to grow this set, causing the endpoint to time out or exhaust server memory and CPU, degrading service for all users.

### Finding Description

**Root cause — no `take` limit on the DB query:**

In `getTransactionsToSign`, the TypeORM `find` call has no `take` or `skip` clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters by status, not by user
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope to the requesting user's transactions — it returns every active transaction in the system.

**Root cause — O(N) expensive async work per row:**

After loading all rows, the code iterates sequentially and calls `userKeysToSign` for each:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [2](#0-1) 

`userKeysToSign` → `keysRequiredToSign` → `computeSignatureKey`, which for every transaction:
1. Deserializes raw bytes via `SDKTransaction.fromBytes`
2. Makes one or more async calls to `accountCacheService.getAccountInfoForTransaction`
3. Optionally calls `nodeCacheService.getNodeInfoForTransaction` [3](#0-2) 

**Pagination is applied after the full scan:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // sliced in-memory, after full scan
  ...
};
``` [4](#0-3) 

**The endpoint is reachable by any authenticated, verified user:**

```typescript
@Get('/sign')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
getTransactionsToSign(...)
``` [5](#0-4) 

**Transaction creation has no per-user count limit.** Any user with a registered key can call `POST /transactions` repeatedly to grow the active transaction set. [6](#0-5) 

### Impact Explanation

With N active transactions in the system, each `GET /transactions/sign` request:
- Loads all N transaction rows (including `transactionBytes` blobs) into memory
- Performs N sequential async round-trips to the account/node cache service
- Holds the Node.js event loop for the full duration

A sufficiently large N causes the request to exceed the HTTP server timeout (default 30 s in NestJS/Express), returning a 5xx to all callers. Because the query is system-wide (not per-user), a single attacker inflating the transaction count degrades the endpoint for every user. The server process may also exhaust heap memory if transaction bytes are large.

**Impact category:** Service unavailability / severe degradation for all users of the `GET /transactions/sign` endpoint.

### Likelihood Explanation

- **Attacker preconditions:** Register an account, verify it, add one key pair — all normal product flows.
- **Attack action:** Call `POST /transactions` in a loop to create thousands of active transactions. Each transaction only needs a valid Hedera transaction ID and bytes; the attacker controls both.
- **No privileged access required.** The endpoint is open to any `VerifiedUser`.
- The controller comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is still live and reachable via the API. [7](#0-6) 

### Recommendation

1. **Apply the DB-level limit before processing.** Add `take: limit` and `skip: offset` to the `repo.find` call so only the requested page is loaded and processed — mirroring how `getTransactions` and `getHistoryTransactions` already work. [8](#0-7) 

2. **Scope the query to the requesting user.** Filter by `signers.userId`, `observers.userId`, or `creatorKey.userId` before fetching, so the result set is bounded by the user's own transactions.

3. **Move the expensive `computeSignatureKey` call out of the hot path.** Pre-compute and cache signature key requirements when a transaction is created or updated, rather than recomputing on every list request.

4. **Add a rate limit** on `POST /transactions` per user to prevent unbounded transaction creation.

### Proof of Concept

**Attacker steps (no privileged access):**

```
# 1. Register and verify a user account, add one key pair (normal product flow)

# 2. Create N active transactions (loop):
for i in $(seq 1 5000); do
  curl -X POST https://api-host/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -d '{"creatorKeyId": 1, "transactionBytes": "<valid_bytes>", "signature": "<sig>", ...}'
done

# 3. Trigger the vulnerable endpoint:
curl -X GET "https://api-host/transactions/sign?page=1&size=10" \
  -H "Authorization: Bearer $TOKEN"
# → Server loads all 5000 transactions, calls computeSignatureKey 5000 times sequentially
# → Request times out (HTTP 504) for attacker and all other users
```

**Expected outcome:** With a large enough N, `getTransactionsToSign` exhausts the request timeout or server heap, returning errors to all callers of `GET /transactions/sign`. Because the query is not scoped to the requesting user, organic growth of the transaction table (even without a deliberate attacker) will eventually cause the same degradation. [9](#0-8)
