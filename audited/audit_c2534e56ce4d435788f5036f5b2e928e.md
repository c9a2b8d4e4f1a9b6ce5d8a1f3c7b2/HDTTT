### Title
`getUserShouldApprove()` Always Returns `false` and `FEATURE_APPROVERS_ENABLED` Is Hardcoded to `false`, Permanently Disabling the Transaction Approval Workflow

### Summary
The function `getUserShouldApprove()` in `front-end/src/renderer/services/organization/transaction.ts` always returns `false` with the actual API call commented out, directly mirroring the `isClosed()` pattern from the external report. Compounding this, the constant `FEATURE_APPROVERS_ENABLED` in `front-end/src/shared/constants/featureFlags.ts` is hardcoded to `false`. Together, these two defects permanently disable the entire approver/rejection workflow: no user is ever prompted to approve a transaction, and no approval is ever submitted to the backend, regardless of whether the transaction's policy requires it.

### Finding Description

**Root cause 1 — `getUserShouldApprove` always returns `false`:** [1](#0-0) 

The function accepts `serverUrl` and `transactionId` parameters (both marked `@typescript-eslint/no-unused-vars`), ignores them, and unconditionally returns `false`. The real API call (`GET /transactions/approve/:id`) is commented out with a `//TODO` note acknowledging the feature is unimplemented.

**Root cause 2 — `FEATURE_APPROVERS_ENABLED` is hardcoded `false`:** [2](#0-1) 

This constant gates every approver UI path. Because it is `false` at compile time, no conditional branch that depends on it can ever be true.

**Affected call sites:**

*`TransactionDetailsHeader.vue`* — `canApprove` computed property: [3](#0-2) 

Because `FEATURE_APPROVERS_ENABLED` is `false`, `canApprove` is always `false`, so the Approve/Reject buttons are never added to `visibleButtons`. [4](#0-3) 

*`TransactionGroupDetails.vue`* — group-level approve buttons: [5](#0-4) 

*`ApproveAllController.vue`* — even when the controller is invoked, `getUserShouldApprove` always returns `false`, so `sendApproverChoice` is never called for any transaction: [6](#0-5) 

### Impact Explanation

The approver workflow is designed as a mandatory gate: designated approvers must explicitly approve (or reject) a transaction before it advances. With both defects in place:

1. The Approve/Reject buttons are never rendered, so approvers have no UI path to record their decision.
2. Even if the controller were triggered programmatically, `getUserShouldApprove` returns `false` for every transaction, so `sendApproverChoice` is never invoked and no approval record is ever written to the backend.

The result is a permanent, silent bypass of the approval gate. Transactions that the protocol intends to hold in an "awaiting approval" state either stall indefinitely (DoS on the workflow) or, if the backend does not independently enforce the approval count before execution, proceed to execution without any approver sign-off — an unauthorized state transition equivalent to the `isClosed()` market-stop bypass.

### Likelihood Explanation

This is not a probabilistic attack — it is a deterministic code defect present in every build. Any organization that configures transactions to require approver sign-off is affected on every such transaction. No attacker capability is required; the bypass is automatic.

### Recommendation

1. **Fix `getUserShouldApprove`**: Restore the commented-out API call so the function actually queries the backend for the current user's approval obligation:

```typescript
export const getUserShouldApprove = async (
  serverUrl: string,
  transactionId: number,
): Promise<boolean> =>
  commonRequestHandler(async () => {
    const { data } = await axiosWithCredentials.get(
      `${serverUrl}/${controller}/approve/${transactionId}`,
    );
    return data;
  }, 'Failed to get if user should approve the transaction');
```

2. **Fix `FEATURE_APPROVERS_ENABLED`**: Replace the hardcoded `false` with a runtime-configurable value (environment variable, backend capability flag, or user-setting) so the feature can be enabled when the backend supports it.

3. **Add backend enforcement**: Regardless of frontend state, the backend chain/scheduler service should refuse to execute a transaction that has not collected all required approver signatures, making the approval gate robust against frontend defects.

### Proof of Concept

1. Configure an organization transaction that requires at least one approver.
2. Log in as the designated approver in the Electron client.
3. Navigate to the transaction's detail page (`TransactionDetails`) or the group page (`TransactionGroupDetails`).
4. Observe: no Approve or Reject button is rendered — `FEATURE_APPROVERS_ENABLED = false` causes `canApprove` / `visibleButtons` to exclude them unconditionally.
5. Inspect `getUserShouldApprove` in DevTools or unit tests: it returns `Promise<false>` for every `(serverUrl, transactionId)` pair without making any network request.
6. Consequence: `sendApproverChoice` is never called; the backend never receives an approval record; the transaction either stalls or executes without the required approval, depending on backend enforcement.
