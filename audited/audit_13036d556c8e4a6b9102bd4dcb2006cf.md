### Title
`archiveTransaction` Incorrect Guard Condition Allows Creator to Archive Transactions in Terminal States via `isManual` Bypass

### Summary

The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a flawed guard condition that uses `!transaction.isManual` as a short-circuit, completely bypassing the status check for manual transactions. This allows the creator of any manual transaction to force it into `ARCHIVED` status regardless of its current state — including after it has already been `EXECUTED` on the Hedera network — corrupting the transaction audit trail and producing an inconsistent system state.

### Finding Description

The guard at lines 711–718 reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The throw condition is: `(status NOT in [WFS, WFE]) AND (NOT isManual)`.

By De Morgan's law, the **allow** condition is: `(status in [WFS, WFE]) OR (isManual == true)`.

The second branch — `isManual == true` — is an unconditional bypass. When a transaction was created with `isManual: true` (a field the creator supplies at creation time, as confirmed by the e2e test at `back-end/apps/api/test/spec/transaction.e2e-spec.ts` line 855), the status check is never evaluated. The function proceeds directly to:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

This means a creator can call `PATCH /transactions/archive/:id` on a manual transaction whose status is `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`, forcing a backward state transition that the state machine is supposed to prohibit.

The function's own comment — *"Archive the transaction if the transaction is sign only"* — confirms the intent is to restrict archiving to active, unsigned states, not to permit it unconditionally for manual transactions.

The analogous flaw in the external report is `paused == false` being checked instead of `marketStatus == Trading`: in both cases, an insufficient boolean condition replaces a proper status-set membership check, creating a bypass for a state that should be locked.

### Impact Explanation

A creator who archives an `EXECUTED` transaction causes:

1. **Audit trail corruption**: The database status changes from `EXECUTED` → `ARCHIVED`. The Hedera network executed the transaction, but the system now records it as archived (never executed). Any downstream reporting, compliance, or history view reflects the falsified state.
2. **Misleading notifications**: `emitTransactionStatusUpdate` is called, broadcasting the status change to all signers, approvers, and observers of the transaction, who receive a false picture of the transaction's lifecycle.
3. **Unrecoverable state inconsistency**: There is no reverse operation. Once archived, the record diverges permanently from the on-chain reality. [1](#0-0) 

### Likelihood Explanation

- The attacker only needs to be an authenticated, verified user — the lowest privilege level in the system.
- `isManual: true` is set by the creator at transaction creation time with no additional authorization required, as shown in the e2e test.
- The attack path is a single authenticated `PATCH` request after the transaction reaches a terminal state.
- No race condition, timing dependency, or external oracle is needed. [2](#0-1) 

### Recommendation

Remove the `isManual` short-circuit from the guard. The status check must apply unconditionally:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    )
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  // ...
}
```

If manual transactions genuinely need a broader set of archivable statuses (e.g., `FAILED`), enumerate those statuses explicitly rather than bypassing the check entirely.

### Proof of Concept

1. Authenticate as any verified user (no admin role required).
2. Create a transaction with `isManual: true` via `POST /transactions`.
3. Allow the transaction to reach `EXECUTED` status (either via the chain service or by directly updating the DB in a test environment).
4. Call `PATCH /transactions/archive/<id>` with the creator's JWT.
5. Observe: HTTP 200 is returned; the transaction's `status` in the database is now `ARCHIVED`; `emitTransactionStatusUpdate` fires, notifying all participants of the false state change.

Expected (correct) behavior: HTTP 400 with `ErrorCodes.OMTIP`, because `EXECUTED` is not in the allowed set. [3](#0-2) [4](#0-3)

```typescript
  /* Archive the transaction if the transaction is sign only. */
  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
```
