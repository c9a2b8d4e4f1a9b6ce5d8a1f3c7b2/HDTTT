### Title
Stale Approval State Persists After Approver `userId` Replacement Due to TypeORM `undefined`-Skip Behavior

### Summary

In `updateTransactionApprover`, when the transaction creator replaces an approver's `userId`, the code intends to clear the old approver's `signature`, `userKeyId`, and `approved` fields by passing `undefined` for those values to TypeORM's `EntityManager.update()`. TypeORM silently skips `undefined` values when building the SQL `SET` clause, so those three columns are never written to the database. The new user inherits the old user's approval state, permanently blocking the new user from approving and corrupting the transaction's approval integrity.

### Finding Description

**Root cause — TypeORM skips `undefined` in `update()`**

TypeORM's `EntityManager.update()` builds its SQL `SET` clause only from properties whose value is not `undefined`. Passing `undefined` for a column is a no-op; the column retains its current database value.

In `approvers.service.ts` lines 501–512:

```typescript
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: undefined,   // ← silently dropped by TypeORM
  signature: undefined,   // ← silently dropped by TypeORM
  approved: undefined,    // ← silently dropped by TypeORM
};
await transactionalEntityManager.update(TransactionApprover, approver.id, data);
```

The actual SQL executed is:
```sql
UPDATE transaction_approver SET "userId" = $1 WHERE id = $2
```

`userKeyId`, `signature`, and `approved` are never touched. [1](#0-0) 

**Downstream effect — new approver is permanently blocked**

`approveTransaction` checks at line 563:

```typescript
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

Because the stale `signature` from the old user is still present in the row now attributed to the new `userId`, this guard fires immediately when the new user attempts to approve, throwing "already approved" and making it impossible for the new user to ever submit their approval. [2](#0-1) 

**Access control — only the transaction creator can trigger this**

`getCreatorsTransaction` enforces that only the user who created the transaction may call `updateTransactionApprover`. The transaction creator is a normal authenticated user — no admin or operator role is required. [3](#0-2) 

**Exploit flow**

1. Attacker (malicious transaction creator) creates a transaction with approver User A.
2. User A approves — `signature`, `userKeyId`, `approved=true` are written to the `transaction_approver` row.
3. Attacker calls `PATCH /transactions/:txId/approvers/:approverId` with `{ userId: userBId }`.
4. TypeORM issues `UPDATE transaction_approver SET "userId" = userBId WHERE id = approverId` — the three approval fields are untouched.
5. User B's approver row now contains User A's `signature`, `userKeyId`, and `approved=true`.
6. User B calls `POST /transactions/:txId/approvers/approve` → `userApprovers.every(a => a.signature)` is `true` → `ErrorCodes.TAP` is thrown. User B can never approve.
7. The transaction's approval record falsely attributes User A's cryptographic signature to User B.

**The unit test at line 808 confirms the bug is present and untested against real DB behavior** — the mock simply records that `update` was called with `undefined` values, but never verifies the database outcome. [4](#0-3) 

### Impact Explanation

- **Approval DoS**: Any approver whose `userId` is replaced after they have signed is permanently blocked from approving under the new identity. The transaction cannot collect the required approvals and is stuck.
- **Approval integrity violation**: The `transaction_approver` row records a cryptographic `signature` and `userKeyId` belonging to User A but attributed to User B. Downstream logic that evaluates whether the approval threshold is met operates on falsified data.
- **Transaction status corruption**: The chain service and status-update logic (`emitTransactionStatusUpdate`) consume approval state to determine whether a transaction is ready for execution. Stale `approved=true` on a replaced approver can cause premature status transitions. [5](#0-4) 

### Likelihood Explanation

The trigger path is a standard authenticated API call available to any user who has created a transaction with approvers. No elevated privilege is required. The scenario — replacing an approver after they have already signed — is a realistic operational action (e.g., the original approver is unavailable and must be substituted). The bug is silent: no error is raised, the API returns success, and the corruption is invisible until the new approver attempts to approve. [6](#0-5) 

### Recommendation

Replace `undefined` with `null` for the three fields that must be cleared. TypeORM includes `null` values in the SQL `SET` clause, which correctly writes `NULL` to the nullable columns:

```typescript
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: null,    // ← writes NULL to DB
  signature: null,    // ← writes NULL to DB
  approved: null,     // ← writes NULL to DB
};
await transactionalEntityManager.update(TransactionApprover, approver.id, data);
``` [7](#0-6) 

### Proof of Concept

**Preconditions**: Two registered users (User A, User B). User C is the transaction creator.

1. User C creates a transaction and adds User A as an approver via `POST /transactions/:txId/approvers`.
2. User A approves: `POST /transactions/:txId/approvers/approve` with valid `{ userKeyId, signature, approved: true }`. The `transaction_approver` row now has `userId=A`, `signature=<A_sig>`, `userKeyId=<A_key>`, `approved=true`.
3. User C replaces the approver: `PATCH /transactions/:txId/approvers/:approverId` with body `{ userId: B }`. The API returns 200. Database state: `userId=B`, `signature=<A_sig>`, `userKeyId=<A_key>`, `approved=true` — stale fields remain.
4. User B attempts to approve: `POST /transactions/:txId/approvers/approve`. The server evaluates `userApprovers.every(a => a.signature)` → `true` (stale `<A_sig>`) → throws `ErrorCodes.TAP` ("Transaction already approved"). User B is permanently blocked.
5. Direct DB query confirms: `SELECT "userId","signature","userKeyId","approved" FROM transaction_approver WHERE id = :approverId` returns `userId=B` with User A's non-null `signature`, `userKeyId`, and `approved=true`. [8](#0-7) [9](#0-8)
