### Title
TOCTOU Race Condition in `executeTransaction` Allows Double Submission of Hedera Transactions

### Summary
In `TransactionsService.executeTransaction`, the service reads transaction state, performs checks, and then calls the external Hedera network (`executeService.executeTransaction`) without first atomically updating the transaction's status to a terminal or in-progress state. Two concurrent HTTP requests for the same transaction ID can both pass all checks and both invoke the external execution call, violating the Checks-Effects-Interactions pattern analog in this backend context.

### Finding Description

The vulnerable function is `executeTransaction` in `back-end/apps/api/src/transactions/transactions.service.ts`: [1](#0-0) 

The execution flow is:

1. **Check**: `getTransactionForCreator` reads the transaction from DB and verifies `isManual === true` (line 737–741).
2. **Branch**: If `validStart > now`, it updates `isManual = false` (line 744) — a state write before the notification. This branch is safe.
3. **Vulnerable branch**: If `validStart <= now` (i.e., the transaction is ready to execute), it calls `this.executeService.executeTransaction(transaction)` (line 747) **without first writing any state change** to the database. [2](#0-1) 

Because Node.js yields at every `await`, two concurrent HTTP requests targeting the same `id` can both:
1. Complete `getTransactionForCreator` and see `isManual = true`.
2. Both evaluate `validStart <= Date.now()` as true.
3. Both reach `this.executeService.executeTransaction(transaction)` and submit the transaction to Hedera.

No guard (status flag, optimistic lock, or conditional DB update) is set before the external call to prevent re-entry.

A secondary instance of the same pattern exists in `createTransactions`: [3](#0-2) 

The uniqueness check (`repo.find` for existing `transactionId`) is performed **outside** the `entityManager.transaction` block that saves the new records (line 436). Two concurrent requests with the same `transactionId` can both pass the uniqueness check before either has committed, then both proceed to save — creating duplicate transaction records if no DB-level unique constraint exists. [4](#0-3) 

By contrast, `cancelTransactionWithOutcome` correctly uses a conditional atomic update (`andWhere('status IN (:...statuses)')`) to avoid this class of bug: [5](#0-4) 

### Impact Explanation

**Primary (`executeTransaction`)**: Two concurrent calls submit the same Hedera transaction bytes twice. Hedera deduplicates by `transactionId` (accountId@validStart), so the second submission is rejected at the network layer. However, the `executeService` likely updates the transaction status after the call returns. With two concurrent executions, the status update from the failed (duplicate-rejected) call may overwrite the successful one, leaving the transaction in a `FAILED` state even though it executed successfully on-chain. This is a state integrity failure: the backend's recorded status diverges from the actual on-chain outcome, causing the transaction to appear failed to all users and potentially triggering incorrect retry or notification logic.

**Secondary (`createTransactions`)**: If no DB-level unique constraint exists on `transactionId`, concurrent requests can insert duplicate active transaction records for the same Hedera transaction, corrupting the signing and approval workflow.

### Likelihood Explanation

The `executeTransaction` endpoint is callable by any authenticated transaction creator. A user (or a client with a network glitch causing a retry) submitting two rapid requests for the same transaction ID is a realistic scenario requiring no special privileges. The race window is the duration of the `executeService.executeTransaction` call (a Hedera network round-trip, typically hundreds of milliseconds), which is wide enough for a concurrent request to enter.

### Recommendation

**For `executeTransaction`**: Before calling `executeService.executeTransaction`, atomically transition the transaction to an in-progress or terminal status using a conditional update (mirroring the pattern in `cancelTransactionWithOutcome`):

```typescript
const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.EXECUTING }) // or a suitable guard state
  .where('id = :id', { id })
  .andWhere('isManual = true')
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  throw new ConflictException('Transaction already being executed');
}

await this.executeService.executeTransaction(transaction);
```

**For `createTransactions`**: Move the uniqueness check inside the `entityManager.transaction` block, or rely on a DB-level unique constraint on `transactionId` (with appropriate status scoping) to make the check-and-insert atomic.

### Proof of Concept

1. Create a manual transaction `T` with `validStart` in the past (so `validStart <= now`).
2. As the transaction creator, send two simultaneous `POST /transactions/{id}/execute` requests (e.g., using `Promise.all` or two parallel `curl` processes).
3. Both requests complete `getTransactionForCreator` before either updates state.
4. Both enter the `else` branch and call `executeService.executeTransaction(transaction)`.
5. Request A's Hedera submission succeeds; Request B's is rejected by Hedera as a duplicate.
6. Depending on `executeService` error handling, the status update from Request B's failure may set the transaction to `FAILED`, overwriting Request A's successful `EXECUTED` status.
7. Observe the transaction in the DB showing `FAILED` despite successful on-chain execution.

  async executeTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (!transaction.isManual) {
      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }
