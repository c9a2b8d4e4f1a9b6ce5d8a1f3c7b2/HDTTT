### Title
Missing User Key Ownership Validation in `importSignatures` Allows Any Authorized User to Inject Foreign Signatures

### Summary
The `importSignatures` endpoint (`POST /transactions/signatures/import`) validates that submitted signatures are cryptographically correct but does not verify that the public keys in the signature map belong to the authenticated user. The parallel `uploadSignatureMaps` endpoint enforces this ownership check. Any user with read access to a transaction (including observers) can inject cryptographically valid signatures from keys they do not own, advancing the transaction status without proper authorization and leaving no audit trail.

### Finding Description

**Root cause — missing ownership check in `importSignatures`:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `importSignatures` performs three checks before accepting a signature map:

1. `verifyAccess` — confirms the user has any access to the transaction (creator, observer, signer, approver, or has keys to sign).
2. Status check — transaction must be `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.
3. `validateSignature` — confirms the signature bytes are cryptographically valid for the given public key. [1](#0-0) 

What is **absent** is any check that the public keys in the submitted signature map belong to the authenticated user. The function accepts any cryptographically valid `(publicKey, signature)` pair regardless of who owns that key.

**Contrast with `uploadSignatureMaps` in `signers.service.ts`:**

The normal signing path builds a `userKeyMap` from `user.keys` and, inside `processTransactionSignatures`, throws `ErrorCodes.PNY` ("Public key not yours") if a submitted public key is not found in that map: [2](#0-1) [3](#0-2) 

`importSignatures` has no equivalent guard.

**Additional gap — no `TransactionSigner` record is created:**

`uploadSignatureMaps` persists a `TransactionSigner` row for every accepted key, creating an auditable record of who signed. `importSignatures` only updates `transactionBytes` directly via a raw SQL `CASE` statement and emits a status-update notification — no signer record is written. [4](#0-3) 

**Exploit path:**

1. Attacker is an observer on a multi-signature transaction (observers pass `verifyAccess`).
2. The transaction creator shares a transaction file externally (a supported workflow — see `front-end/src/renderer/services/organization/transaction.ts` `importSignatures`). The file contains signatures from other users' keys.
3. Attacker calls `POST /transactions/signatures/import` with the signature map extracted from that file.
4. `validateSignature` passes (signatures are cryptographically valid for the transaction body bytes).
5. The foreign signatures are merged into `transactionBytes`; the transaction may advance to `WAITING_FOR_EXECUTION`.
6. No `TransactionSigner` record is created, so the audit log shows no signer for those keys. [5](#0-4) 

### Impact Explanation

- **Unauthorized status advancement:** An observer (read-only role) can push a transaction from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` by injecting signatures from keys they do not own, bypassing the intended multi-party signing workflow.
- **Audit trail corruption:** Because no `TransactionSigner` rows are written, the system cannot determine which user submitted which signature. Compliance and forensic capabilities are broken.
- **Integrity of the signing model:** The invariant that "only the owner of a key may add that key's signature to a transaction" is violated on this code path.

### Likelihood Explanation

The attacker must be an authenticated user with any access to the target transaction (observer is sufficient). They must also possess a cryptographically valid signature for the specific transaction body from a key they do not own. This second requirement is realistic because:

- Transaction files containing signatures are explicitly designed to be shared externally (the front-end `importSignatures` flow exists for this purpose).
- An observer who receives such a file can replay its signatures via this endpoint.

No privileged access, no secret leakage, and no cryptographic break are required — only a valid transaction file that has been partially or fully signed by other participants.

### Recommendation

Add the same user key ownership check that `uploadSignatureMaps` enforces. Before accepting any `(publicKey, signature)` pair in `importSignatures`, verify that the public key exists in `user.keys`:

```typescript
// Build user key lookup (same pattern as signers.service.ts)
await attachKeys(user, this.entityManager);
const userKeyMap = new Map(user.keys.map(k => [k.publicKey, k]));

// Inside the per-transaction loop, after validateSignature:
for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  const der = publicKey.toStringDer();
  if (!userKeyMap.has(raw) && !userKeyMap.has(der)) {
    throw new BadRequestException(ErrorCodes.PNY);
  }
  sdkTransaction.addSignature(publicKey, map);
}
```

Additionally, consider creating `TransactionSigner` records in `importSignatures` (or delegating to `uploadSignatureMaps`) so that the audit trail is preserved regardless of which code path is used.

### Proof of Concept

**Preconditions:**
- User A is the creator of transaction T (id=42), which requires signatures from keys K1 (User A) and K2 (User B).
- User C is an observer on transaction T.
- User B signs T offline and shares the transaction file with User C.

**Steps:**
1. User C extracts the signature map from the shared file (contains K2's valid signature over T's body bytes).
2. User C calls:
   ```
   POST /transactions/signatures/import
   Authorization: Bearer <User C's JWT>
   Body: [{ "id": 42, "signatureMap": { "0.0.3": { "<txId>": { "<K2_DER>": "<sig_hex>" } } } }]
   ```
3. `verifyAccess` returns `true` (User C is an observer).
4. `validateSignature` returns `[K2_PublicKey]` (signature is cryptographically valid).
5. `sdkTransaction.addSignature(K2_PublicKey, map)` succeeds.
6. `transactionBytes` is updated in the database with K2's signature.
7. If K1 was already signed, the transaction advances to `WAITING_FOR_EXECUTION`.
8. No `TransactionSigner` record for K2 is created — the audit log shows no signer.

**Expected (correct) behavior:** Step 2 should return `400 PNY` because K2 does not belong to User C. [6](#0-5) [7](#0-6)

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
