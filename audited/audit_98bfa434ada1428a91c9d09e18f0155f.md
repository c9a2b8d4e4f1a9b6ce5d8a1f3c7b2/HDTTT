### Title
Transaction Creator Can Arbitrarily Lower Approval Threshold or Replace Approvers After Signatures Are Collected

### Summary
The `ApproversService` in the backend API allows the transaction creator to modify the approval threshold and replace individual approvers at any point during a transaction's lifecycle — including after the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status. There is no status guard preventing these mutations once approvals have been collected. A malicious transaction creator can lower the threshold to bypass the required number of approvals, or replace a rejecting approver with a new one to nullify the rejection, undermining the multi-sig trust model the organization relies on.

### Finding Description

**Root Cause**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, both `updateTransactionApprover` and `removeTransactionApprover` verify only that the caller is the transaction creator, but perform **no check on the current transaction status** before mutating the approval structure.

`updateTransactionApprover` (lines 367–531): [1](#0-0) 

The function calls `getCreatorsTransaction` to verify ownership, but `getCreatorsTransaction` (lines 624–644) only checks `creatorKey.userId === user.id` — it never inspects `transaction.status`: [2](#0-1) 

The threshold update path (lines 467–488) allows the creator to set any valid threshold value with no status guard: [3](#0-2) 

The approver replacement path (lines 489–517) clears the existing `signature`, `userKeyId`, and `approved` fields when a new `userId` is set, effectively erasing a prior rejection: [4](#0-3) 

`removeTransactionApprover` (lines 533–544) also performs no status check: [5](#0-4) 

The controller exposes both mutations to any authenticated transaction creator: [6](#0-5) 

By contrast, `approveTransaction` correctly guards against acting on a transaction that is not in an active state: [7](#0-6) 

No equivalent guard exists in `updateTransactionApprover` or `removeTransactionApprover`.

**Exploit Flow**

**Attack 1 — Threshold Lowering:**
1. Creator creates a transaction with a threshold tree node: `threshold=3`, children = [A, B, C, D, E].
2. Approvers A and B approve (`approved=true`). Approver C rejects (`approved=false`). Transaction remains in `WAITING_FOR_SIGNATURES`.
3. Creator sends `PATCH /transactions/:txId/approvers/:treeNodeId` with body `{ "threshold": 2 }`.
4. The threshold is updated to 2. The chain service's status evaluator now sees 2 approvals ≥ threshold of 2 and transitions the transaction to `WAITING_FOR_EXECUTION`.
5. C's rejection is bypassed; the transaction proceeds to execution.

**Attack 2 — Approver Replacement to Nullify Rejection:**
1. Creator creates a transaction with approvers [A, B, C], threshold=2.
2. A approves, B rejects.
3. Creator sends `PATCH /transactions/:txId/approvers/:bApproverId` with body `{ "userId": D }`.
4. B's `signature`, `userKeyId`, and `approved` are cleared. D is now the approver in B's slot.
5. D approves. Threshold is met. The rejection is erased from the record.

### Impact Explanation

The multi-sig approval workflow is the primary security control for organization transactions. Allowing the transaction creator to retroactively lower the threshold or replace rejecting approvers after votes have been cast completely undermines this control. A malicious creator can force any transaction through with fewer approvals than the organization intended, or nullify any rejection. This is a critical integrity failure in the trust model: approvers believe their vote is binding, but the creator can silently invalidate it.

### Likelihood Explanation

Any authenticated user who creates a transaction can exploit this. No admin role, leaked credentials, or special privileges are required. The API endpoints (`PATCH /transactions/:id/approvers/:id` and `DELETE /transactions/:id/approvers/:id`) are standard REST calls available to any transaction creator. The attack requires only that the creator wait for partial approvals to be collected before mutating the threshold or approver list.

### Recommendation

Add a transaction status guard at the start of `updateTransactionApprover`, `removeTransactionApprover`, and `createTransactionApprovers`. Once a transaction has left `NEW` status (i.e., is in `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any terminal state), the approval structure should be immutable. Specifically:

```typescript
const transaction = await this.getCreatorsTransaction(transactionId, user, entityManager);
if (transaction.status !== TransactionStatus.NEW) {
  throw new BadRequestException('Cannot modify approvers after the transaction has been submitted');
}
```

This mirrors the guard already present in `approveTransaction` and aligns with the principle that approval parameters must be fixed before the approval process begins.

### Proof of Concept

**Prerequisites:** Two authenticated users — Creator and Approver B (who will reject).

```
# Step 1: Creator creates transaction with threshold=2, approvers=[A, B, C]
POST /transactions/:txId/approvers
Authorization: Bearer <creator_token>
{ "approversArray": [{ "threshold": 2, "approvers": [{"userId":1},{"userId":2},{"userId":3}] }] }

# Step 2: Approver A approves
POST /transactions/:txId/approvers/approve
Authorization: Bearer <approver_a_token>
{ "userKeyId": 1, "signature": "<sig>", "approved": true }

# Step 3: Approver B rejects
POST /transactions/:txId/approvers/approve
Authorization: Bearer <approver_b_token>
{ "userKeyId": 2, "signature": "<sig>", "approved": false }

# Step 4: Creator lowers threshold to 1 (bypassing B's rejection)
PATCH /transactions/:txId/approvers/:treeNodeId
Authorization: Bearer <creator_token>
{ "threshold": 1 }

# Result: Transaction transitions to WAITING_FOR_EXECUTION with only 1 approval,
# B's rejection is bypassed.
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
