### Title
Approver Rejection Does Not Gate Transaction Execution — Approval System Bypassed in Execution Path

### Summary
The organizational approver system records approvals and rejections in the `transaction_approver` table, but neither the status-transition function (`processTransactionStatus`) nor the execution-time validator (`getValidatedSDKTransaction`) ever consults approver state. A transaction with pending or explicitly rejected approvals is executed on the Hedera network as soon as the required cryptographic signatures are collected, making the entire approver workflow a no-op with respect to actual fund movement.

### Finding Description

**Root cause — `processTransactionStatus` ignores approver state**

The function that promotes a transaction from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` checks only cryptographic signatures:

```typescript
// back-end/libs/common/src/utils/transaction/index.ts
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey
);
let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;
if (isAbleToSign) {
  const collatedTx = await smartCollate(transaction, signatureKey);
  if (collatedTx !== null) {
    newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
  }
}
``` [1](#0-0) 

There is no check against `transaction_approver.approved`. A transaction whose approvers have all set `approved = false` will still be promoted to `WAITING_FOR_EXECUTION` the moment signatures are sufficient.

**Root cause — `getValidatedSDKTransaction` ignores approver state**

At execution time the chain service validates only the cryptographic signature key:

```typescript
// back-end/libs/common/src/execute/execute.service.ts
const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);
if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
``` [2](#0-1) 

No approval check is performed before `sdkTransaction.execute(client)` is called.

**Root cause — `approveTransaction` never sets transaction status to `REJECTED`**

When an approver rejects (`dto.approved = false`), the service updates only the `TransactionApprover` row and emits a notification; the parent `Transaction.status` is never changed to `REJECTED`:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
  .whereInIds(userApprovers.map(a => a.id))
  .execute();
// No transaction.status = REJECTED here
``` [3](#0-2) 

The `validateTransactionStatus` switch in `execute.service.ts` does guard against `REJECTED` status — but that status is never written, so the guard is never reached. [4](#0-3) 

**End-to-end exploit path**

1. Alice (creator) submits a `TransferTransaction` moving HBAR from a multi-sig treasury account.
2. Alice adds Bob as a mandatory approver via `POST /transactions/:id/approvers`.
3. Bob calls `PATCH /transactions/:id/approvers/:approverId` with `approved: false` — rejection is recorded in `transaction_approver` but `Transaction.status` stays `WAITING_FOR_SIGNATURES`.
4. The required Hedera signers upload their signatures via `POST /transactions/signers`.
5. `processTransactionStatus` (called by the chain scheduler) sees valid signatures, sets status to `WAITING_FOR_EXECUTION` — no approval check.
6. The chain service calls `executeTransaction`, which calls `getValidatedSDKTransaction` — no approval check — and submits the transaction to Hedera.
7. HBAR is transferred on-chain despite Bob's explicit rejection.

### Impact Explanation

The approver system is presented to users as a mandatory organizational gate (the UI renders an "Awaiting Approval" stepper step and the `TransactionApproverDto` has an `approved` boolean). In practice it provides zero enforcement: any transaction that collects the required cryptographic signatures will be executed on the Hedera network regardless of approver decisions. This allows a malicious or negligent transaction creator to move funds without the oversight the approver system is supposed to guarantee, directly analogous to the external report's "funds released before settlement is guaranteed" pattern. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

Any authenticated user who can create a transaction and whose transaction accumulates the required Hedera cryptographic signatures can trigger this. No privileged access is required. The approver API endpoints are reachable by any registered organization user. The scenario is realistic in any deployment where the approver feature is used for treasury or governance oversight.

### Recommendation

1. **Set `Transaction.status = REJECTED` when any approver rejects.** In `approveTransaction`, after persisting `approved = false`, update the parent transaction status to `REJECTED` so `validateTransactionStatus` in the execution path will block it.

2. **Gate `processTransactionStatus` on approver satisfaction.** Before promoting a transaction to `WAITING_FOR_EXECUTION`, query `transaction_approver` and verify that the approval tree is satisfied (i.e., `isApproved(rootApprover) === true
