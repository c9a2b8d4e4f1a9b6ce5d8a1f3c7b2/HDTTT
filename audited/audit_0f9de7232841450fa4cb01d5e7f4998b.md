### Title
Missing Authorization Check in `createComment` Allows Any Verified User to Post Comments on Arbitrary Transactions

### Summary
The `CommentsController.createComment` endpoint accepts a `transactionId` parameter and creates a comment on that transaction without verifying that the requesting user has any relationship to it (creator, signer, approver, or observer). The codebase itself acknowledges this with a `//TODO need some sort of guard or check to ensure user can comment here` comment. Any authenticated, verified user can post comments on any transaction in the system, and similarly read comments on any transaction via `getComments` and `getCommentById` with no access check.

### Finding Description
The `CommentsController` applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the controller level. These guards confirm the user is authenticated and email-verified — nothing more. The `@Post()` handler for `createComment` has no additional guard or service-level ownership check:

```
back-end/apps/api/src/transactions/comments/comments.controller.ts
Lines 21-29:

  @Post()
  //TODO need some sort of guard or check to ensure user can comment here
  createComment(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Body() dto: CreateCommentDto,
  ) {
    return this.commentsService.createComment(user, transactionId, dto);
  }
```

The service layer performs no authorization check either:

```
back-end/apps/api/src/transactions/comments/comments.service.ts
Lines 15-24:

  async createComment(user, transactionId, dto): Promise<TransactionComment> {
    const comment = this.repo.create(dto);
    comment['transaction'].id = transactionId;
    comment.user = user;
    return this.repo.save(comment);
  }
```

The `getComments` (line 31-34) and `getCommentById` (line 36-39) endpoints are equally unguarded — any authenticated user can read comments on any transaction ID.

Compare this to how other sensitive transaction sub-resources are protected: `ApproversService.getCreatorsTransaction` enforces creator ownership before mutations, `ObserversService` checks user relationship before returning data, and `TransactionsService.getTransactionWithVerifiedAccess` calls `verifyAccess` which checks creator/signer/approver/observer membership. Comments have none of this.

### Impact Explanation
1. **Unauthorized write to any transaction**: A verified user with no relationship to a transaction can inject arbitrary comment content into it, polluting audit trails and potentially misleading approvers/signers about transaction intent.
2. **Information disclosure**: `getComments` and `getCommentById` expose comment content (which may contain sensitive operational notes) to any authenticated user, bypassing the transaction-level access model enforced everywhere else.
3. **Spam / resource exhaustion**: An attacker can flood any transaction's comment thread at will, degrading usability and potentially causing database growth issues.

### Likelihood Explanation
The attacker precondition is only a valid, verified account — the lowest privilege level in the system. No admin access, no special role, no leaked secrets are required. The endpoint is a standard REST `POST /transactions/:transactionId/comments` reachable over HTTPS. The developer-left TODO comment confirms the gap is known but unresolved, making accidental exploitation by any registered user realistic.

### Recommendation
Add a service-level authorization check in `CommentsService.createComment` (or as a guard on the controller method) that verifies the requesting user has an existing relationship to the transaction before allowing the comment to be created. The existing `TransactionsService.verifyAccess` method already implements this logic (checking creator, signer, approver, observer membership) and should be reused. Similarly, `getComments` and `getCommentById` should call `verifyAccess` before returning data, consistent with how `getTransactionWithVerifiedAccess` protects the transaction read endpoint.

### Proof of Concept

1. Register and verify two accounts: `attacker@example.com` and `victim@example.com`.
2. As `victim`, create a transaction (note its `id`, e.g. `42`). Do not add `attacker` as observer, approver, or signer.
3. As `attacker`, obtain a JWT via `POST /auth/login`.
4. Send:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <attacker_jwt>
   Content-Type: application/json

   { "content": "Injected comment by attacker" }
   ```
5. Observe HTTP 201 and the comment persisted on transaction 42.
6. Send `GET /transactions/42/comments` as `attacker` — all comments are returned with no access denial.

Expected (correct) behavior: HTTP 401/403 because `attacker` has no relationship to transaction 42. [1](#0-0) [2](#0-1) [3](#0-2)
