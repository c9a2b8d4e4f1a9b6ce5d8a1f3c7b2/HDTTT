### Title
Transaction Creator Can Delete Approvers Belonging to Transactions They Do Not Own (Cross-Transaction IDOR in `removeTransactionApprover`)

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that the approver actually belongs to that transaction. A user who is the creator of any transaction can therefore delete approvers from any other transaction in the system by supplying their own `transactionId` and a victim approver's `id`. This is the direct analog of the SKALE "node can unlink validator" bug: a secondary entity (approver) can be removed by a party that has no ownership over the primary resource (the victim transaction), because the authorization check and the state-mutation target are decoupled.

### Finding Description

**Root cause — missing cross-transaction ownership check in the delete path**

In `approvers.controller.ts`, the delete handler is:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver `id` with no ownership check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only verifies that `user` is the creator of the transaction identified by `transactionId`:

```typescript
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
``` [2](#0-1) 

`removeTransactionApprover` then deletes the approver by its own `id` with no check that it belongs to the verified transaction:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [3](#0-2) 

**Contrast with the update path — the fix already exists there**

`updateTransactionApprover` performs the missing check:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [4](#0-3) 

The delete path simply omits this guard entirely.

**Exploit path**

1. Attacker registers as a normal user (no admin required).
2. Attacker creates their own transaction → receives `transactionId_A`.
3. Attacker learns `approver_id_B` for a victim transaction `transactionId_B`. This is achievable if the attacker is an observer or signer of `transactionId_B` (both roles can call `getVerifiedApproversByTransactionId`), or by sequential integer enumeration of approver IDs.
4. Attacker sends:
   ```
   DELETE /transactions/{transactionId_A}/approvers/{approver_id_B}
   ```
5. `getCreatorsTransaction(transactionId_A, user)` passes — attacker owns A.
6. `removeTransactionApprover(approver_id_B)` executes — approver from B is deleted.

### Impact Explanation

An attacker can silently delete any approver (including entire approval trees via `removeNode`'s recursive SQL) from any transaction they do not own. This permanently disrupts the multi-signature approval workflow for victim transactions: required approvers disappear, threshold conditions can no longer be met, and the transaction may become permanently stuck or require the creator to rebuild the entire approver tree. Because `removeNode` is a recursive soft-delete, the damage covers the full subtree rooted at the targeted approver. [5](#0-4) 

### Likelihood Explanation

- **Precondition**: attacker must be a registered user (no admin, no special role).
- **Approver ID discovery**: approver IDs are sequential integers; any user who is an observer or signer of the victim transaction can retrieve them directly via the `GET /transactions/:id/approvers` endpoint.
- **No rate-limit barrier**: the delete endpoint is a standard authenticated REST call.
- **Realistic scenario**: a malicious participant added as an observer to a high-value multi-sig transaction can destroy its entire approval tree before execution.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted belongs to the transaction whose ownership was already verified:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Mirror the check already present in updateTransactionApprover
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
}
```

Pass `transactionId` from the controller into the service call, exactly as `updateTransactionApprover` already does.

### Proof of Concept

**Setup**

- Admin creates two users: `alice` (creator of transaction B) and `bob` (attacker, observer of transaction B, creator of transaction A).
- Alice creates transaction B and adds approver with `approver_id = 42`.
- Alice adds Bob as an observer of transaction B.

**Attack**

```bash
# Bob lists approvers of transaction B (allowed because he is an observer)
GET /transactions/B/approvers
# → returns approver id=42

# Bob creates his own transaction A
POST /transactions  →  { id: 99 }

# Bob deletes Alice's approver using his own transactionId
DELETE /transactions/99/approvers/42
Authorization: Bearer <bob_token>
# → HTTP 200 true
```

**Result**: Approver 42 (and its entire subtree) is permanently soft-deleted from transaction B. Alice's multi-sig approval workflow is broken. Bob never needed to own or be the creator of transaction B.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
