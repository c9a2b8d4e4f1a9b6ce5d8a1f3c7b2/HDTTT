### Title
Deleted Organization User's Transaction Approval Persists and Counts Toward Threshold

### Summary
When an organization user is removed (soft-deleted), their existing `TransactionApprover` records are not cleaned up. Any prior approval (`approved = true`, `signature` set) they submitted on pending transactions remains active and continues to count toward the configured approval threshold. This is the direct analog of the oDAO "kicked member votes still count" pattern.

### Finding Description

The `removeUser` function in `users.service.ts` soft-deletes the user and their `UserKey` records, but performs no cleanup of `TransactionApprover` records: [1](#0-0) 

Specifically, only these two operations occur on removal:
- `softDelete(UserKey, { userId: id })` — removes keys
- `softRemove(user)` — removes the user

No `softDelete` or cascade is applied to `TransactionApprover` rows where `userId = id`.

The `getApproversByTransactionId` query that feeds all approval-status checks only filters on `TransactionApprover.deletedAt is null` — it does **not** join the `user` table to check whether the referenced user is still active: [2](#0-1) 

The `approved` and `signature` columns on `TransactionApprover` are set when a user approves and are never cleared on user deletion: [3](#0-2) 

The approval-status evaluation (both frontend `isApproved` and the backend tree evaluation) reads the persisted `approved` field directly: [4](#0-3) 

### Impact Explanation

Consider a transaction configured with a 2-of-3 threshold approver tree (users A, B, C):

1. User A (malicious or later-untrusted) approves → `TransactionApprover.approved = true` for A's record.
2. Admin removes User A from the organization (`DELETE /users/:id`).
3. User A's `TransactionApprover` record is **not** soft-deleted; `approved = true` persists.
4. User B approves → threshold of 2 is now met (A's stale approval + B's live approval).
5. The transaction proceeds to execution even though the admin intended to revoke A's participation.

The admin's intent in removing a user — to strip their organizational trust — is not reflected in the approval state of in-flight transactions.

### Likelihood Explanation

- Admin-initiated user removal is a standard operation exposed via `DELETE /users/:id` (called from `deleteUser` in the frontend).
- Any transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status with a threshold approver structure is affected.
- No special attacker capability is required beyond being an approver who approves before being removed.

### Recommendation

In `removeUser`, after soft-deleting the user and their keys, also soft-delete (or nullify the `approved`/`signature` fields of) all `TransactionApprover` records belonging to that user:

```typescript
await this.repo.manager.softDelete(UserKey, { userId: id });
// Add: clear stale approvals for the removed user
await this.repo.manager.update(TransactionApprover, { userId: id }, {
  approved: null,
  signature: null,
  userKeyId: null,
});
await this.repo.softRemove(user);
```

Alternatively, the `getApproversByTransactionId` query should join the `user` table and exclude approvals from soft-deleted users.

### Proof of Concept

1. Create a transaction with a threshold approver structure: `threshold: 2`, approvers `[userA, userB, userC]`.
2. Log in as `userA` and call `POST /transactions/:id/approvers/approve` → `approved = true` stored.
3. Admin calls `DELETE /users/:userAId` → user soft-deleted, `TransactionApprover` row for A untouched.
4. Log in as `userB` and approve → threshold of 2 is met.
5. Observe the transaction advances to `WAITING_FOR_EXECUTION` despite `userA` no longer being an active organization member.

Root cause: [1](#0-0)  does not cascade to `TransactionApprover`, and [5](#0-4)  does not filter by user deletion status.

```typescript
  async getApproversByTransactionId(
    transactionId: number,
    userId?: number,
    entityManager?: EntityManager,
  ): Promise<TransactionApprover[]> {
    if (typeof transactionId !== 'number' || (userId && typeof userId !== 'number')) return null;

    return (entityManager || this.repo).query(
      `
      with recursive approverList as
        (
          select * from transaction_approver 
          where "transactionId" = $1
            union all
              select approver.* from transaction_approver as approver
              join approverList on approverList."id" = approver."listId"
        )
      select * from approverList
      where approverList."deletedAt" is null
        ${userId ? 'and approverList."userId" = $2' : ''}
      `,
      userId ? [transactionId, userId] : [transactionId],
    );
  }
```

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
