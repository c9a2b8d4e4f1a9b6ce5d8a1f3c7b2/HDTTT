### Title
Authorization Bypass in `verifyAccess` Grants Any Authenticated User Full Read Access to All Terminal-Status Transactions

### Summary
The `verifyAccess` function in `TransactionsService` unconditionally returns `true` for any transaction in a terminal status (EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED) without verifying that the requesting user has any relationship to that transaction. Because `GET /transactions/:id` relies solely on this function for authorization, any authenticated user can read the full details — including `transactionBytes`, `publicKeys`, `signature`, and all relational data — of any other organization's completed, canceled, or failed transaction by guessing or enumerating its integer ID.

### Finding Description
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `verifyAccess` method contains the following branch:

```typescript
if (
  [
    TransactionStatus.EXECUTED,
    TransactionStatus.EXPIRED,
    TransactionStatus.FAILED,
    TransactionStatus.CANCELED,
    TransactionStatus.ARCHIVED,
  ].includes(transaction.status)
)
  return true;   // ← no user identity check whatsoever
``` [1](#0-0) 

For non-terminal transactions the function correctly checks creator, observer, signer, and approver membership. That entire block is skipped for terminal transactions.

The controller wires `GET /transactions/:id` directly through this function:

```
GET /transactions/:id
  → TransactionsController.getTransaction()
  → TransactionsService.getTransactionWithVerifiedAccess()
  → getTransactionById(id)          // no user filter
  → attachTransactionApprovers()
  → verifyAccess(transaction, user) // returns true for any terminal tx
  → returns full Transaction entity
``` [2](#0-1) [3](#0-2) 

`getTransactionById` performs no user-scoped filtering; it fetches by raw integer or `TransactionId` string. [4](#0-3) 

A second, independent exposure exists on `GET /transactions/history`: the controller passes no user context to `getHistoryTransactions`, which returns all terminal-status transactions across all users with no ownership filter. [5](#0-4) [6](#0-5) 

### Impact Explanation
An attacker who is a legitimate but unprivileged user of the platform can:

1. Call `GET /transactions/history` (no user filter) to enumerate all integer IDs of terminal-status transactions across every organization.
2. Call `GET /transactions/:id` for each ID and receive the full `TransactionFullDto`, which includes `transactionBytes` (the raw serialized Hedera transaction body), `publicKeys` (the public keys of all required signers), `signature`, `mirrorNetwork`, creator key metadata, observer list, signer list, and approver tree.

This constitutes **cross-tenant data exfiltration**: one organization's completed transaction data — including the public keys of its members and the exact bytes of every executed Hedera transaction — is readable by any other authenticated user. In a multi-organization deployment this breaks the fundamental confidentiality boundary between tenants.

### Likelihood Explanation
- **Attacker precondition**: a valid JWT for any account on the platform. No admin, no leaked credentials, no privileged role.
- **Effort**: trivial. The history endpoint paginates all terminal transactions; IDs are sequential integers. A single authenticated HTTP session is sufficient.
- **Detection**: the request is indistinguishable from a legitimate user viewing their own history.
- **Prevalence**: every transaction that has ever been executed, expired, failed, canceled, or archived is permanently exposed.

### Recommendation
Remove the unconditional early-return for terminal statuses in `verifyAccess`. Terminal transactions should apply the same ownership checks (creator, observer, signer, approver) as active ones:

```typescript
async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
  if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

  // Do NOT short-circuit for terminal statuses.
  // Apply the same membership checks regardless of status.
  const userKeysToSign = await this.userKeysToSign(transaction, user, true);

  return (
    userKeysToSign.length !== 0 ||
    transaction.creatorKey?.userId === user.id ||
    !!transaction.observers?.some(o => o.userId === user.id) ||
    !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
    !!transaction.approvers?.some(a => a.userId === user.id)
  );
}
```

Additionally, `getHistoryTransactions` must be scoped to the requesting user by adding the same `whereForUser` predicate used in `getTransactions`. [7](#0-6) 

### Proof of Concept

**Prerequisites**: two accounts, `alice` (owner of transaction #42, now EXECUTED) and `bob` (unrelated user). Both have valid JWTs.

1. Bob authenticates and obtains a JWT.
2. Bob calls:
   ```
   GET /transactions/history?page=1&limit=100
   Authorization: Bearer <bob_jwt>
   ```
   Response includes transaction IDs for all organizations, e.g. `[{id: 42, status: "EXECUTED", ...}]`.

3. Bob calls:
   ```
   GET /transactions/42
   Authorization: Bearer <bob_jwt>
   ```
   Expected (correct) behavior: `401 Unauthorized`.
   Actual behavior: `200 OK` with the full `TransactionFullDto` for Alice's transaction, including `transactionBytes`, `publicKeys`, `signature`, creator key, signers, observers, and approvers.

The root cause is the `return true` at line 798 of `transactions.service.ts`, which is never exercised by a test case where the requesting user has no relationship to the transaction — the exact "uncovered branch" pattern described in the external report. [1](#0-0)
