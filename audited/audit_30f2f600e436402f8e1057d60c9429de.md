### Title
Global IP Rate Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header in `IpThrottlerGuard`

### Summary
The `IpThrottlerGuard`, registered as a global `APP_GUARD` for all API endpoints, derives its rate-limit tracker key from the raw `req.headers['x-forwarded-for']` HTTP header. Because this header is fully attacker-controlled, any client can rotate arbitrary fake source IPs on every request, completely bypassing the global IP-based rate limit. This renders the only global throttling mechanism ineffective for all API routes.

### Finding Description
**Root cause — `IpThrottlerGuard.getTracker()` trusts an attacker-supplied header:**

```typescript
// back-end/apps/api/src/guards/ip-throttler.guard.ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;   // raw header string used as throttle key
}
``` [1](#0-0) 

The guard reads `req.headers['x-forwarded-for']` directly — the raw, unparsed HTTP header — and uses it as the Redis throttle-storage key. An attacker simply sets `X-Forwarded-For: <any_ip>` on each request to get a fresh counter bucket.

**Global registration — every endpoint is affected:**

`IpThrottlerGuard` is registered as `APP_GUARD`, meaning it is the sole global rate-limit enforcement layer for all routes. [2](#0-1) 

**Configured limits that are bypassed:**

```
GLOBAL_MINUTE_LIMIT=10000
GLOBAL_SECOND_LIMIT=1000
``` [3](#0-2) 

These limits are per-tracker-key (i.e., per fake IP). By rotating `X-Forwarded-For` values, an attacker gets a fresh 10,000 req/min budget for each new fake IP.

**Production `trust proxy` setting does not help:**

In production, `app.enable('trust proxy')` is set, which instructs Express to parse `X-Forwarded-For` and populate `req.ip` with the leftmost non-trusted IP. However, the guard ignores `req.ip` and reads the raw header directly, bypassing Express's proxy-trust logic entirely. [4](#0-3) 

**`UserThrottlerModule` is dead code — never imported:**

A per-user throttler module exists but is neither exported from the throttlers index nor imported in `ApiModule`, so it provides no protection. [5](#0-4) [6](#0-5) 

**Exploit path:**

1. Attacker sends `POST /auth/reset-password` with `X-Forwarded-For: 1.0.0.1` → gets a fresh throttle bucket.
2. Repeats with `X-Forwarded-For: 1.0.0.2`, `1.0.0.3`, … — each request is treated as a new IP.
3. The `EmailThrottlerGuard` on `/auth/reset-password` limits to 3 OTP emails per minute **per email address**, but the attacker can target thousands of distinct registered email addresses, triggering OTP emails to all of them with no IP-level brake.
4. For authenticated endpoints (transactions, notifications, etc.) that have **only** the IP throttler, an authenticated attacker can flood them without bound. [7](#0-6) 

### Impact Explanation
- **Email abuse / OTP spam**: An attacker can trigger OTP password-reset emails to an unbounded number of registered users per minute, causing user harassment and exhausting email-sending quotas.
- **Resource exhaustion on authenticated endpoints**: Transactions, notifications, and user-key endpoints have no per-user throttle (the `UserThrottlerModule` is unused). A single authenticated attacker can flood these endpoints, degrading service for all users.
- **Complete nullification of the global rate-limit control**: The only global security control against request flooding is rendered ineffective.

### Likelihood Explanation
- **No privilege required**: Any HTTP client can set arbitrary headers. No authentication, no special network position.
- **Trivially scriptable**: A single `curl` loop rotating `X-Forwarded-For` values is sufficient.
- **Directly reachable**: The guard is on every route; no internal access is needed.

### Recommendation
1. **Use `req.ip` instead of the raw header** in `IpThrottlerGuard.getTracker()`. When `trust proxy` is correctly configured, Express sets `req.ip` to the verified client IP and cannot be overridden by the client.
2. **Scope `trust proxy` to known proxy IPs** (e.g., Cloudflare CIDR ranges) rather than `app.enable('trust proxy')` (which trusts all proxies). This prevents `X-Forwarded-For` injection when requests bypass the proxy.
3. **Wire `UserThrottlerModule` into `ApiModule`** and apply `UserThrottlerGuard` to authenticated endpoints so that per-user limits remain effective even if IP spoofing occurs.
4. **Export `UserThrottlerModule`** from `back-end/apps/api/src/throttlers/index.ts` so it is available for import.

### Proof of Concept

```bash
# Bypass global IP rate limit by rotating X-Forwarded-For on every request
for i in $(seq 1 500); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<api-host>/auth/reset-password \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.${i}.1" \
    -d '{"email":"victim@example.com"}'
done
```

**Expected (broken) behavior**: All 500 requests return `200 OK`. Each request is assigned a fresh throttle bucket keyed to the fake IP, so the `ANONYMOUS_MINUTE_LIMIT=3` email throttler is the only remaining brake — and it only limits per-email, not per-IP. Rotating target emails removes even that constraint.

**Expected (correct) behavior**: After 3 requests from the same real IP within 60 seconds, subsequent requests should return `429 Too Many Requests`.
