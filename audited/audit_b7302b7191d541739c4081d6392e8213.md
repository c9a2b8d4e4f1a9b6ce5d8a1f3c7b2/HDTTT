### Title
Any Authenticated User Can Post Unlimited Comments on Any Transaction — Missing Authorization Check in `createComment`

### Summary

The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` enforces only authentication (JWT validity), but performs no check that the requesting user has any relationship to the target transaction. Any verified organization member can inject an unlimited number of arbitrarily large comments into any transaction's comment thread — including transactions they have no role in — corrupting the integrity of transaction audit trails and growing the database without bound. The developer acknowledged this gap with an explicit `//TODO` in the source.

### Finding Description

**Root cause — missing authorization in `CommentsController.createComment`:** [1](#0-0) 

The controller applies three guards at the class level: [2](#0-1) 

These guards verify only that the JWT is valid and not blacklisted, and that the user's status is `NONE` (i.e., a normal verified member). There is no check that the user is a signer, observer, approver, or creator of the transaction identified by `:transactionId`.

The service layer performs no such check either — it blindly creates and persists the comment: [3](#0-2) 

**No message length constraint:**

`CreateCommentDto` validates only that `message` is a string, with no `@MaxLength()` decorator: [4](#0-3) 

**No per-user rate limiting on this endpoint:**

`ApiModule` registers only `IpThrottlerModule` and `EmailThrottlerModule` as global providers. The `UserThrottlerModule` (which defines 100 req/min and 10 req/sec per-user limits) is defined but **never imported** in `ApiModule`: [5](#0-4) 

The only active global throttle is IP-based, which is trivially bypassed via proxies or multiple accounts from different IPs.

**Developer acknowledgment:** [6](#0-5) 

### Impact Explanation

1. **Cross-tenant data integrity violation**: Any authenticated user can inject arbitrary comments into any transaction in the organization — including transactions belonging to other users or groups — corrupting the audit trail that transaction participants rely on.
2. **Unbounded database growth**: With no message length cap and only IP-based throttling, an attacker with a valid JWT can flood the `transaction_comment` table with large payloads, degrading PostgreSQL query performance and eventually exhausting disk space.
3. **Transaction enumeration**: The endpoint accepts any integer `transactionId`, allowing an attacker to probe and comment on all transactions by iterating IDs, confirming their existence.

### Likelihood Explanation

- **Precondition**: Attacker needs only a valid organization account (any verified user). No admin or privileged role required.
- **Exploit path**: Trivially scriptable — obtain a JWT via `POST /auth/login`, then loop `POST /transactions/{id}/comments` with a large `message` payload.
- **No mitigations in place**: No per-user rate limit on this route, no message length cap, no relationship check.

### Recommendation

1. **Add a relationship guard**: Before persisting a comment, verify the requesting user is a signer, observer, approver, or creator of the target transaction. Reject with `403 Forbidden` otherwise.
2. **Add `@MaxLength()`** to `CreateCommentDto.message` (e.g., 4096 characters).
3. **Apply `UserThrottlerModule`** to the comments endpoint, or import it globally in `ApiModule`.
4. **Validate transaction existence** before inserting a comment to avoid orphaned records.

### Proof of Concept

```bash
# Step 1: Obtain a JWT for any valid organization user
TOKEN=$(curl -s -X POST http://<host>/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"attacker@org.com","password":"password"}' \
  | jq -r '.accessToken')

# Step 2: Spam comments on a victim's transaction (id=42)
# with a large payload, bypassing all authorization checks
LARGE_MSG=$(python3 -c "print('A'*100000)")

for i in $(seq 1 500); do
  curl -s -X POST http://<host>/transactions/42/comments \
    -H "Authorization: Bearer $TOKEN" \
    -H 'Content-Type: application/json' \
    -d "{\"message\": \"$LARGE_MSG\"}" &
done
wait
```

Expected result: 500 comment rows, each ~100 KB, are inserted into the `transaction_comment` table for transaction 42 — a transaction the attacker has no legitimate relationship to — with no authorization error returned.
