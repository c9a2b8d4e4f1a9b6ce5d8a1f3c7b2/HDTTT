### Title
`hasValidSignatureKey` and `ableToSign` Return `true` for Empty `KeyList`, Bypassing Signature Validation

### Summary
Both `hasValidSignatureKey` (back-end) and `ableToSign` (front-end) recursively evaluate a `KeyList` to determine whether sufficient signatures exist. Neither function guards against an **empty `KeyList`** (zero keys, no threshold). When an empty `KeyList` is encountered as a nested entry, the arithmetic `0 >= (null || 0)` evaluates to `true`, causing the function to report the signature requirement as satisfied with zero actual signatures. This is the direct analog of the `CircularBuffer::contains` false-positive: an "empty slot" in the key structure is treated as a valid, satisfied entry.

### Finding Description

**Root cause — `hasValidSignatureKey`:**

```typescript
// back-end/libs/common/src/utils/sdk/key.ts  lines 31-46
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();          // [] for an empty KeyList
    let currentThreshold = 0;

    keys.forEach(key => {                // loop never executes
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
    //     0                >= (null          || 0          )
    //     0                >= 0   → TRUE  ← false positive
  }
  ...
};
``` [1](#0-0) 

The identical logic exists in the front-end:

```typescript
// front-end/src/renderer/utils/sdk/index.ts  lines 109-126
export const ableToSign = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;
    keys.forEach(key => { if (ableToSign(publicKeys, key)) currentThreshold++; });
    return currentThreshold >= (key.threshold || keys.length);
    //     same 0 >= 0 → TRUE for empty KeyList
  }
  ...

``` [2](#0-1) 

**Critical call sites:**

1. **Transaction execution gate** — `execute.service.ts` line 219 uses `hasValidSignatureKey` as the sole check before allowing a transaction to proceed to the Hedera network:

```typescript
if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
``` [3](#0-2) 

2. **Status promotion** — `processTransactionStatus` in `transaction/index.ts` uses the same function to decide whether to promote a transaction from `WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION`:

```typescript
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey
);
if (isAbleToSign) { ... newStatus = TransactionStatus.WAITING_FOR_EXECUTION; }
``` [4](#0-3) 

**Exploit path:**

The `signatureKey` is assembled by `TransactionSignatureService.computeSignatureKey`, which pushes account keys fetched from the Hedera mirror node into a `KeyList`. If any participating account's on-chain key is (or contains) an empty `KeyList`, that empty `KeyList` is pushed into `signatureKey`. When `hasValidSignatureKey` recurses into it, it returns `true` with zero signatures present, satisfying the threshold check for the entire tree. [5](#0-4) 

### Impact Explanation

A transaction that has collected **zero signatures** for an account whose on-chain key resolves to an empty `KeyList` will pass the `hasValidSignatureKey` gate in `execute.service.ts` and be submitted to the Hedera network. Simultaneously, `processTransactionStatus` will promote the transaction to `WAITING_FOR_EXECUTION`, triggering the chain service to attempt submission. If the Hedera network also treats an empty `KeyList` as trivially satisfied (vacuous truth), the transaction executes without the intended authorization. Even if the network rejects it, the tool's internal state is corrupted: the transaction is incorrectly marked ready for execution, bypassing the multi-signature coordination workflow the tool is designed to enforce.

### Likelihood Explanation

Hedera accounts can be created with arbitrary `KeyList` structures, including nested empty `KeyList` entries. A malicious user operating within the normal product flow (no privileged access required) can:
1. Create a Hedera account whose key contains an empty nested `KeyList`.
2. Submit a transaction through the tool that involves this account as a fee payer or required signer.
3. The mirror-node-fetched key propagates the empty `KeyList` into `signatureKey`, triggering the false positive.

No admin keys, leaked credentials, or out-of-scope capabilities are needed.

### Recommendation

Add an explicit guard at the top of both `hasValidSignatureKey` and `ableToSign` to treat an empty `KeyList` as **not satisfied** (analogous to the PR#14 fix in the external report):

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    if (keys.length === 0) return false; // ← guard: empty KeyList is never satisfied
    let currentThreshold = 0;
    keys.forEach(k => { if (hasValidSignatureKey(publicKeys, k)) currentThreshold++; });
    return currentThreshold >= (key.threshold || keys.length);
  }
  ...
};
```

Apply the same guard to `ableToSign` in `front-end/src/renderer/utils/sdk/index.ts`. Note that `isKeyListValid` already contains this guard (`keys.length === 0 → false`) but it is not called from either of the two vulnerable functions. [6](#0-5) 

### Proof of Concept

1. On Hedera testnet, create account `A` with key = `new KeyList([])` (empty threshold key list).
2. In the tool (Organization Mode), create an `AccountUpdateTransaction` where `A` is the fee payer.
3. Submit the transaction without providing any signatures.
4. Observe: `computeSignatureKey` returns a `KeyList` containing the empty `KeyList` for account `A`.
5. `hasValidSignatureKey([], emptyKeyList)` → evaluates `0 >= (null || 0)` → `true`.
6. `processTransactionStatus` promotes the transaction to `WAITING_FOR_EXECUTION` with zero signatures collected.
7. `getValidatedSDKTransaction` passes the same check and submits the unsigned transaction to the Hedera network.

Expected (correct) behavior: the transaction should remain in `WAITING_FOR_SIGNATURES` and the execution gate should throw `'Transaction has invalid signature.'`

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
