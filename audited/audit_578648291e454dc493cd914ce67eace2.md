### Title
Approvers and Observers Can Be Added to Terminal-State Transactions Without Status Validation

### Summary
The `createTransactionApprovers` and `createTransactionObservers` functions do not validate the transaction's current status before mutating its state. A transaction creator can add approvers or observers to a transaction that is already in a terminal state (`CANCELED`, `EXECUTED`, `FAILED`, `EXPIRED`, `ARCHIVED`, `REJECTED`), corrupting the database state and triggering spurious status-update events for finished transactions. This is the direct analog of the external report's `depositCollateral` missing a pool-status guard.

### Finding Description

**Root cause — missing status guard in `getCreatorsTransaction`**

`createTransactionApprovers` delegates its authorization check to `getCreatorsTransaction`: [1](#0-0) 

`getCreatorsTransaction` only verifies existence and creator identity; it never inspects `transaction.status`: [2](#0-1) 

After inserting the approvers, `emitTransactionStatusUpdate` is unconditionally fired for the (already-finished) transaction: [3](#0-2) 

The same pattern exists in `createTransactionObservers`: creator identity is checked but transaction status is never validated before persisting new observer records: [4](#0-3) 

**Contrast with correct guards elsewhere**

Every other mutating path in the codebase that touches a transaction checks status before acting. `approveTransaction` explicitly rejects non-active statuses: [5](#0-4) 

`cancelTransaction` and `archiveTransaction` in `transactions.service.ts` both gate on `cancelableStatuses` / active statuses before writing: [6](#0-5) 

`validateTransactionStatus` in `signers.service.ts` rejects any non-`WAITING_FOR_SIGNATURES` / non-`WAITING_FOR_EXECUTION` transaction: [7](#0-6) 

The status enum defines the terminal states that should block further mutation: [8](#0-7) 

**Exploit flow**

1. Creator submits a transaction; it reaches `CANCELED` (or `EXECUTED`, `FAILED`, etc.).
2. Creator calls `POST /transactions/:id/approvers` with a valid `CreateTransactionApproversArrayDto`.
3. `createTransactionApprovers` calls `getCreatorsTransaction`, which passes (user is creator, transaction exists).
4. New `TransactionApprover` rows are inserted into the database for the finished transaction.
5. `emitTransactionStatusUpdate` fires, causing the chain/notification service to process a status-change event for a transaction that is already in a terminal state.
6. Same flow applies to `POST /transactions/:id/observers`.

### Impact Explanation
- **Database state corruption**: `TransactionApprover` and `TransactionObserver` rows are permanently written for transactions that have already concluded. There is no cleanup path; `cancelTransaction` only sets the status field and does not purge related approver/observer records.
- **Spurious downstream events**: `emitTransactionStatusUpdate` is published for finished transactions, causing the chain service and notification service to process events that should never occur, potentially producing incorrect in-app notifications or UI state for all participants of the transaction.
- **Audit-trail integrity**: The approver/observer history for a finished transaction becomes unreliable, undermining the trust model of the multi-signature coordination workflow.

### Likelihood Explanation
The attacker precondition is simply being the creator of any transaction — a normal, unprivileged product role. No special keys, credentials, or internal access are required. The trigger is a standard authenticated API call. The scenario can occur by mistake (creator adds an approver after canceling) or deliberately.

### Recommendation
Add a terminal-status guard at the top of both `createTransactionApprovers` and `createTransactionObservers`, mirroring the pattern already used in `approveTransaction`:

```typescript
// In createTransactionApprovers, after getCreatorsTransaction():
if (this.terminalStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP); // or a dedicated error code
}

// In createTransactionObservers, after the creator check:
const terminalStatuses = [
  TransactionStatus.EXECUTED, TransactionStatus.EXPIRED,
  TransactionStatus.FAILED,   TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED, TransactionStatus.REJECTED,
];
if (terminalStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
```

The same guard should be applied to `updateTransactionApprover`, which also delegates to `getCreatorsTransaction` without a status check. [9](#0-8) 

### Proof of Concept

**Preconditions**: authenticated user who is the creator of transaction `id=42`; transaction status is `CANCELED`.

```
# Step 1 – cancel the transaction (or wait for it to reach a terminal state)
PATCH /transactions/42/cancel
→ 200 OK, status becomes CANCELED

# Step 2 – add an approver to the now-finished transaction
POST /transactions/42/approvers
Authorization: Bearer <creator-jwt>
Content-Type: application/json

{
  "approversArray": [{ "userId": 7 }]
}

# Expected (correct): 400 Bad Request – transaction is not in an active state
# Actual: 201 Created – TransactionApprover row inserted for CANCELED transaction,
#         emitTransactionStatusUpdate fired for transaction 42
```

**Verification**: query `SELECT * FROM transaction_approver WHERE "transactionId" = 42` — the row exists despite the transaction being in a terminal state.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
