### Title
Creator Signature Over Transaction Bytes Does Not Bind to Network, Enabling Cross-Network Replay

### Summary
When a user creates a transaction via the organization back-end, the creator's signature is verified only over the raw `transactionBytes`. The `mirrorNetwork` field (which identifies the target Hedera network: testnet, mainnet, previewnet) is submitted separately and is **not** included in the signed data. Because the same server handles transactions for multiple networks, an authenticated user can replay a signature produced for one network against a different network, causing a transaction to be stored and potentially executed on an unintended network.

### Finding Description

**Root cause — `validateAndPrepareTransaction`:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, the creator signature is verified as:

```typescript
const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
``` [1](#0-0) 

The `dto.mirrorNetwork` value (e.g. `"testnet"`, `"mainnet"`) is accepted from the request body and stored directly into the database record without being part of the signed payload:

```typescript
mirrorNetwork: dto.mirrorNetwork,
``` [2](#0-1) 

The server then uses `dto.mirrorNetwork` to select the appropriate Hedera client: [3](#0-2) 

**Root cause — `approveTransaction`:**

The approval signature is verified via `verifyTransactionBodyWithoutNodeAccountIdSignature`, which signs/verifies over the Hedera transaction body bytes with the node account ID stripped out — no network identifier is included: [4](#0-3) 

The underlying helper: [5](#0-4) 

**Exploit flow:**

1. Attacker (authenticated user) constructs an unfrozen Hedera transaction and signs the raw bytes with their private key.
2. Attacker submits `POST /transactions` with `mirrorNetwork: "testnet"` — this succeeds and the signature is recorded.
3. Attacker replays the **identical** `transactionBytes` + `signature` in a second `POST /transactions` request, but sets `mirrorNetwork: "mainnet"`.
4. `publicKey.verify(dto.transactionBytes, dto.signature)` passes — the signature is over the bytes, not the network.
5. The server calls `getClientFromNetwork("mainnet")`, freezes the transaction with mainnet node IDs, validates nodes, and stores the transaction as a **mainnet** transaction.
6. The transaction proceeds through the mainnet approval/execution pipeline.

For unfrozen transactions this is fully exploitable: the server freezes the transaction with the target network's nodes after signature verification, so the node-validation check (`isTransactionValidForNodes`) does not block the replay.

### Impact Explanation

A malicious authenticated user can cause a transaction they signed for testnet to be stored and executed on mainnet (or any other supported network) without ever explicitly signing for that network. This can result in:

- Unintended fund transfers or account mutations on mainnet triggered by a testnet-intended signature.
- Approval signatures (`verifyTransactionBodyWithoutNodeAccountIdSignature`) being replayed across networks for the same transaction body, allowing an approver's testnet approval to count as a mainnet approval.

Severity: **High** — direct unauthorized state change on a different network using a legitimately obtained signature.

### Likelihood Explanation

Any authenticated user of the organization server can perform this attack with no special privileges. The only requirement is the ability to call `POST /transactions` twice with the same `transactionBytes` + `signature` but different `mirrorNetwork` values. No cryptographic break, no admin access, and no leaked secrets are required.

### Recommendation

Include the `mirrorNetwork` (or a canonical Hedera ledger/network identifier) in the data that is signed by the creator. The signature should cover a domain-separated message such as:

```
sign( hash(transactionBytes || mirrorNetwork) )
```

Similarly, the approval signature in `verifyTransactionBodyWithoutNodeAccountIdSignature` should incorporate the network identifier so that an approval for testnet cannot be accepted as an approval for mainnet.

### Proof of Concept

```
1. Authenticate as user U on the organization server (supports testnet + mainnet).

2. Build an unfrozen AccountCreateTransaction T with payer 0.0.X.

3. Sign raw bytes: sig = privateKey.sign(T.toBytes())

4. POST /transactions
   { transactionBytes: T.toBytes(), signature: sig, mirrorNetwork: "testnet", creatorKeyId: K, ... }
   → 201 Created, transaction stored as testnet TX id=1

5. POST /transactions  (replay, only mirrorNetwork changed)
   { transactionBytes: T.toBytes(), signature: sig, mirrorNetwork: "mainnet", creatorKeyId: K, ... }
   → 201 Created, transaction stored as mainnet TX id=2
     (server freezes T with mainnet nodes; publicKey.verify passes because sig is over original bytes)

6. TX id=2 now enters the mainnet execution pipeline.
``` [6](#0-5)

/* Get transaction body bytes without node account id */
export const getTransactionBodyBytes = (transaction: SDKTransaction) => {
  // @ts-expect-error - _makeTransactionBody is a private method
  const transactionBody = transaction._makeTransactionBody(null);
  return proto.TransactionBody.encode(transactionBody).finish();
};

/* Verify the signature of the transaction body without node account id */
export const verifyTransactionBodyWithoutNodeAccountIdSignature = (
  transaction: SDKTransaction,
  signature: string | Buffer,
  publicKey: string | PublicKey,
) => {
  const bodyBytes = getTransactionBodyBytes(transaction);

  /* Deserialize Public Key */
  publicKey = publicKey instanceof PublicKey ? publicKey : PublicKey.fromString(publicKey);

  /* Deserialize Signature */
  signature = typeof signature === 'string' ? decode(signature) : signature;

  try {
    return publicKey.verify(bodyBytes, signature);
  } catch (err) {
    console.log(err);
    return false;
  }
};
