### Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Concurrent Signature Writes to Overwrite Each Other, Causing Lost Signatures in `transactionBytes`

### Summary
`SignersService.uploadSignatureMaps` reads `transactionBytes` from the database outside any locking boundary, processes signatures in memory, and writes the result back with an unconditional `UPDATE` that has no optimistic-concurrency guard. When two or more authenticated users upload signatures to the same transaction concurrently, the last writer silently overwrites the first writer's signature bytes. The `TransactionSigner` rows are inserted correctly, so the system believes the transaction is fully signed, but the actual `transactionBytes` stored in the database may be missing one or more cryptographic signatures, causing the Hedera submission to fail.

### Finding Description

**Root cause — three-phase read-modify-write with no locking:**

**Phase 1 — Read (outside any DB transaction):** [1](#0-0) 

`loadTransactionData` fetches the current `transactionBytes` snapshot for every requested transaction ID in a plain `find` call with no row-level lock.

**Phase 2 — Modify (in memory, async):** [2](#0-1) 

`processTransactionSignatures` deserialises the snapshot, calls `sdkTransaction.addSignature(publicKey, map)` for each key, and returns the new bytes. This entire step operates on the in-memory snapshot; it has no awareness of concurrent writers.

**Phase 3 — Write (unconditional UPDATE, no version check):** [3](#0-2) 

`bulkUpdateTransactions` issues:
```sql
UPDATE transaction
SET "transactionBytes" = CASE id WHEN X THEN $1::bytea END,
    "updatedAt" = NOW()
WHERE id = ANY($2)
```
There is no `WHERE "transactionBytes" = <expected_old_value>`, no version column, and no row-level lock. The last `UPDATE` to arrive wins unconditionally.

**Status check is also outside the DB transaction:** [4](#0-3) 

`validateTransactionStatus` checks the in-memory snapshot, not a freshly locked DB row, so the check and the write are not atomic.

**Exploit timeline (two concurrent users, same transaction):**

```
T=0  User A reads transactionBytes (original, no sigs)
T=0  User B reads transactionBytes (original, no sigs)
T=1  User A adds sig_A → newBytesA
T=1  User B adds sig_B → newBytesB
T=2  User A writes newBytesA to DB  ✓
T=3  User B writes newBytesB to DB  ← overwrites A's write; sig_A is gone
T=4  TransactionSigner rows: both A and B present  ← system thinks it's fully signed
T=5  Hedera submission fails: sig_A missing from transactionBytes
```

The `bulkInsertSigners` call inside the DB transaction inserts both users' `TransactionSigner` rows correctly (different `userKeyId` values, no conflict), so `processTransactionStatus` concludes the threshold is met and schedules execution. The Hedera node rejects the transaction because the required key is absent from the actual bytes.

### Impact Explanation

- **Integrity failure**: The `transactionBytes` persisted in the database diverges from the set of `TransactionSigner` records. The system's own invariant — "if enough `TransactionSigner` rows exist, the transaction can be executed" — is silently violated.
- **Transaction execution failure**: The Hedera network rejects the submission because a required signature is missing from the bytes, even though the backend reports the transaction as `WAITING_FOR_EXECUTION` or `EXECUTED`.
- **Unrecoverable state**: Once the transaction's `validStart` window expires, the transaction cannot be re-signed or re-submitted. The funds or account-update intent are permanently lost for that transaction window.
- **No privileged access required**: Any two authenticated organisation members who are both required signers can trigger this by uploading signatures within the same request window.

### Likelihood Explanation

In an organisation with multiple required signers, it is normal and expected for several users to upload their signatures in a short time window (e.g., after receiving an email/WebSocket notification). The race window is the entire async processing time of `processTransactionSignatures` (SDK deserialisation + signature addition), which is on the order of tens to hundreds of milliseconds — wide enough to be hit in normal production use without any deliberate attack. A malicious insider can trivially widen the window by submitting a large signature map to slow Phase 2.

### Recommendation

Apply one of the following:

1. **Optimistic locking**: Add a `version` column (TypeORM `@VersionColumn`) to `Transaction`. Change `bulkUpdateTransactions` to `WHERE id = X AND version = <expected_version>` and retry on conflict.
2. **Pessimistic row-level lock**: Inside the DB transaction in `persistSignatureChanges`, re-fetch the transaction rows with `SELECT … FOR UPDATE` before computing and writing the new bytes, so the read-modify-write is atomic.
3. **Merge instead of replace**: Re-read the current `transactionBytes` inside the DB transaction (after acquiring a row lock), merge the new signatures into the freshly read bytes, and write the merged result. This is the safest approach because it is also correct when more than two concurrent writers race.

### Proof of Concept

**Preconditions**: Organisation transaction `T` requires signatures from User A (key `kA`) and User B (key `kB`). Both users receive the sign notification simultaneously.

**Steps**:
1. User A calls `POST /transactions/T/signers` with `signatureMap` containing `kA`'s signature.
2. Before User A's request completes Phase 3, User B calls `POST /transactions/T/signers` with `signatureMap` containing `kB`'s signature.
3. Both requests complete Phase 1 with the same original `transactionBytes` (no signatures).
4. User A's request writes `bytes(original + sig_A)` to the DB.
5. User B's request writes `bytes(original + sig_B)` to the DB, overwriting step 4.
6. DB state: `transactionBytes = original + sig_B` only; `TransactionSigner` has rows for both A and B.
7. `processTransactionStatus` sees two signers, transitions status to `WAITING_FOR_EXECUTION`.
8. Chain service submits `bytes(original + sig_B)` to Hedera; Hedera returns `INVALID_SIGNATURE` because `kA` is required but absent.
9. Transaction is marked `FAILED`; the operation is permanently lost within that `validStart` window. [5](#0-4) [6](#0-5)

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
