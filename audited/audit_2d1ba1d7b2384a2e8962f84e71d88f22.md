### Title
Any Authenticated User Can DoS Another User's Transaction Creation by Pre-Registering the Same `transactionId`

### Summary

The `createTransactions` function in `back-end/apps/api/src/transactions/transactions.service.ts` enforces uniqueness of `transactionId` via an application-level check, but does not validate that the payer account embedded in the `transactionId` belongs to the submitting user. Any authenticated attacker can craft transaction bytes using a victim's payer account ID and a predicted `validStart`, sign them with their own registered key, and submit first — permanently blocking the victim from registering a transaction with that `transactionId`. Additionally, the check-then-insert is not atomic and there is no database-level unique constraint on `transactionId`, compounding the issue.

---

### Finding Description

**Root cause — no ownership check on `transactionId`'s payer account:**

In `validateAndPrepareTransaction`, the API:
1. Verifies that `transactionBytes` were signed by the caller's registered `creatorKey` [1](#0-0) 
2. Parses the SDK transaction and extracts `transactionId` as a plain string (`payerAccountId@validStart`) [2](#0-1) 

There is **no check** that the payer account embedded in the `transactionId` matches the submitting user's Hedera account. The Hedera SDK allows setting any account ID as the payer when constructing transaction bytes locally — network-level validation only occurs on submission to the Hedera network, not here.

**Duplicate check is application-level only, with no DB unique constraint:**

The uniqueness guard is a plain `repo.find` query executed **outside** the database transaction: [3](#0-2) 

The subsequent insert is wrapped in a separate `entityManager.transaction` block: [4](#0-3) 

The `transaction` table schema (confirmed in the migration) defines `transactionId` as `character varying NOT NULL` with **no `UNIQUE` constraint**: [5](#0-4) 

The `Transaction` entity class also has no `@Unique` decorator on `transactionId`: [6](#0-5) 

**Attack path:**

1. Attacker learns victim's Hedera payer account ID (observable from the network or organization user list).
2. Attacker constructs Hedera SDK transaction bytes with `transactionId = victimPayerAccount@targetValidStart`.
3. Attacker signs those bytes with their own registered key (signature verification passes because it only checks the creator's key, not the payer account).
4. Attacker submits `POST /transactions` before the victim.
5. The attacker's record is stored with `transactionId = victimPayerAccount@targetValidStart`.
6. Victim submits the same `transactionId` → the `repo.find` check finds the attacker's record → `BadRequestException: Transactions already exist` is thrown.
7. Victim must choose a new `validStart`; attacker repeats the attack indefinitely.

---

### Impact Explanation

- **Persistent DoS on transaction creation:** A malicious registered user can block any other user from registering a transaction for any payer account ID they know. Since payer account IDs are observable on the Hedera network, this is not a secret.
- **Database integrity violation (TOCTOU):** Two concurrent requests with the same `transactionId` can both pass the `repo.find` check and both succeed at insert (no DB unique constraint), creating duplicate `transactionId` rows. This corrupts the database state and may cause the chain service to attempt double-execution of the same Hedera transaction.
- **Severity:** High — targeted DoS against specific users, and potential database corruption.

---

### Likelihood Explanation

- **Attacker preconditions:** Must be a registered user (no admin privileges required). Payer account IDs are public on the Hedera network.
- **Attack complexity:** Low. The attacker constructs standard Hedera SDK transaction bytes with an arbitrary payer account, signs with their own key, and submits via the standard `POST /transactions` endpoint.
- **Repeatability:** The attack can be repeated for every new `validStart` the victim attempts, making it a sustained DoS.

---

### Recommendation

1. **Validate payer account ownership:** After parsing the SDK transaction, verify that the payer account in `sdkTransaction.transactionId.accountId` corresponds to a Hedera account owned by the submitting user (e.g., cross-reference against the user's registered accounts).

2. **Add a database-level unique constraint on `transactionId`** (scoped to active statuses, or unconditionally) to prevent duplicate entries regardless of race conditions:
   ```sql
   CREATE UNIQUE INDEX ON "transaction" ("transactionId")
   WHERE status NOT IN ('CANCELED', 'REJECTED', 'ARCHIVED');
   ```

3. **Make the check-then-insert atomic:** Move the `repo.find` duplicate check inside the `entityManager.transaction` block, or use an `INSERT ... ON CONFLICT DO NOTHING` / upsert pattern so the check and insert are a single atomic operation.

---

### Proof of Concept

**Setup:** Two registered users, Alice (victim, payer account `0.0.1234`) and Bob (attacker).

**Steps:**

1. Bob observes that Alice's payer account is `0.0.1234`.
2. Bob constructs transaction bytes:
   ```typescript
   const tx = new AccountCreateTransaction()
     .setTransactionId(
       TransactionId.withValidStart(
         AccountId.fromString('0.0.1234'), // Alice's account
         Timestamp.fromDate(new Date())    // current time
       )
     )
     .toBytes();
   ```
3. Bob signs `tx` with his own registered private key and submits:
   ```
   POST /transactions
   { transactionBytes: <Bob's bytes>, creatorKeyId: <Bob's keyId>, signature: <Bob's sig>, ... }
   ```
   → Server stores `transactionId = "0.0.1234@<timestamp>"` under Bob's creator key. Signature check passes because it only verifies Bob's key signed Bob's bytes.

4. Alice submits her legitimate transaction with the same `transactionId`:
   ```
   POST /transactions
   { transactionBytes: <Alice's bytes>, creatorKeyId: <Alice's keyId>, signature: <Alice's sig>, ... }
   ```
   → Server finds Bob's record at line 415–427 → throws `BadRequestException: Transactions already exist: 0.0.1234@<timestamp>`.

5. Bob repeats for every new timestamp Alice tries.

**Expected outcome:** Alice is permanently blocked from creating transactions for her own payer account as long as Bob keeps frontrunning.
