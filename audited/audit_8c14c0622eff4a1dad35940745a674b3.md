### Title
Any Authenticated User Can Register Another User's Public Key Without Proof of Possession

### Summary
The `uploadKey` endpoint in the Hedera Transaction Tool accepts any public key string from any authenticated user without requiring cryptographic proof that the submitter controls the corresponding private key. An attacker who knows a victim's public key can register it under their own account first, permanently blocking the victim from registering their own key and corrupting the key-ownership state used to coordinate multi-signature workflows.

### Finding Description
**Root Cause**

In `back-end/apps/api/src/user-keys/user-keys.service.ts`, the `uploadKey` method stores whatever `publicKey` string is supplied in the DTO, bound to the currently authenticated user, with no signature challenge or proof-of-possession step: [1](#0-0) 

The only ownership guard is a post-hoc collision check at line 52: [2](#0-1) 

This check fires only **after** a key has already been claimed. The first authenticated user to `POST /user/:userId?/keys` with a given `publicKey` value wins ownership permanently. There is no challenge-response, no signature verification, and no other mechanism to confirm the submitter holds the private key.

**Controller entry point** — the endpoint is reachable by any verified, logged-in user: [3](#0-2) 

**Exploit flow**

1. Eve (any authenticated user) learns Alice's Hedera public key. Public keys are routinely shared in multi-sig coordination (e.g., shared in a group chat, visible in a transaction proposal, or derived from a known account ID on the public Hedera ledger).
2. Eve calls `POST /user/keys` with `{ publicKey: "<alice_public_key>", mnemonicHash: "<arbitrary_hash>", index: <arbitrary_index> }`.
3. The server creates a `UserKey` row binding Alice's public key to Eve's `userId`.
4. Alice later calls the same endpoint with her own public key → receives `BadRequestException(ErrorCodes.PU)` ("Public key already in use") and is permanently blocked from registering her key.
5. The system's key-ownership table now shows Eve as the owner of Alice's public key. Any transaction workflow that looks up "who owns this key" will resolve to Eve, not Alice.

### Impact Explanation

**State corruption / permanent lock-out of legitimate key owner**

- Alice cannot register her own public key. The only remediation path is for Eve to voluntarily delete the key (`DELETE /user/keys/:id`), which Eve controls.
- Any multi-signature transaction that lists Alice's public key as a required signer will resolve Eve as the responsible party. Eve cannot actually sign on the Hedera network (she lacks the private key), so those transactions stall or fail.
- If Alice is a required approver in a time-sensitive transaction group, the entire group is blocked.
- The `mnemonicHash` and `index` fields stored by Eve are arbitrary, corrupting the key-derivation metadata the front-end uses to locate keys in the local wallet. [4](#0-3) 

### Likelihood Explanation

- **Attacker precondition**: Must be a registered, verified user of the same organization instance. This is a low bar — the system is designed for multi-user organizations, so any member can attack any other member.
- **Knowledge required**: The victim's Hedera public key. Public keys are not secrets; they are shared during multi-sig setup and are derivable from public Hedera account IDs.
- **Timing**: No race condition is required. The attacker simply registers the key before the victim does. In a new-user onboarding flow, the window is wide open.
- **Detection**: The victim receives a generic "Public key already in use" error with no indication of who claimed it, making diagnosis difficult.

### Recommendation

Implement **proof of possession** before accepting a public key registration:

1. **Challenge-response**: When a user submits a public key, the server issues a random nonce. The client must sign the nonce with the corresponding private key. The server verifies the signature using the submitted public key before persisting the record.
2. **Alternatively**, enforce that the `publicKey` submitted matches a key already associated with the user's Hedera account on-chain (verifiable via the Hedera Mirror Node API), so the ledger itself acts as the authority.
3. Add a negative test: attempt to register a public key as User B that was already registered by User A, and assert the correct error; also attempt to register a key as User A without a valid proof-of-possession signature.

### Proof of Concept

**Setup**: Two accounts exist — `alice@example.com` (victim) and `eve@example.com` (attacker). Alice's Hedera public key is `302a300506032b6570032100<alice_bytes>` (publicly known).

**Step 1** — Eve logs in and obtains a JWT:
```
POST /auth/login
{ "email": "eve@example.com", "password": "..." }
→ { "accessToken": "<eve_jwt>" }
```

**Step 2** — Eve registers Alice's public key under her own account:
```
POST /user/keys
Authorization: Bearer <eve_jwt>
{
  "publicKey": "302a300506032b6570032100<alice_bytes>",
  "mnemonicHash": "arbitrary_hash_eve_controls",
  "index": 0
}
→ 201 Created  { "id": 42, "publicKey": "302a...", "userId": <eve_id> }
```

**Step 3** — Alice logs in and attempts to register her own key:
```
POST /user/keys
Authorization: Bearer <alice_jwt>
{
  "publicKey": "302a300506032b6570032100<alice_bytes>",
  "mnemonicHash": "<alice_real_hash>",
  "index": 0
}
→ 400 Bad Request  { "message": "PU" }   ← permanently blocked
```

**Step 4** — Any transaction requiring Alice's key now resolves Eve as the owner. Eve cannot sign on-chain; the transaction stalls indefinitely. [5](#0-4)

  async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
    await attachKeys(user, this.repo.manager);

    // Check if the user already has the maximum number of keys
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }

    // Find the userKey by the publicKey
    let userKey = await this.repo.findOne({
      where: { publicKey: dto.publicKey },
      withDeleted: true,
    });

    if (userKey) {
      // If the userKey found is owned by a different user,
      // or if the userKey has a non null hash or index that doesn't
      // match the hash or index provided
      // throw an error.
      if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
        throw new BadRequestException(ErrorCodes.PU);
      }
      // Set the hash and/or index (only if the current value is null)
      Object.assign(userKey, dto);
    } else {
      userKey = await this.repo.create(dto);
      userKey.user = user;
    }

    if (userKey.deletedAt) {
      await this.repo.recover(userKey);
    }
    return this.repo.save(userKey);
  }
