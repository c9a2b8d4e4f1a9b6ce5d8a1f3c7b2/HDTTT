### Title
`isNode` Duplicate-Check Bypass for Threshold Approver Nodes Due to Incorrect Boolean Logic

### Summary
The `isNode` function in `approvers.service.ts` is intended to prevent duplicate approver nodes from being inserted into the approval tree. However, its return expression appends an extra `&& typeof approver.userId === 'number'` condition that causes the function to unconditionally return `false` for threshold (tree) nodes — regardless of whether a matching record already exists in the database. This is the same vulnerability class as the external report: a validation guard that uses incorrect boolean logic, causing it to silently pass for a specific category of inputs.

### Finding Description

The root cause is in `isNode`: [1](#0-0) 

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';
```

A threshold/tree approver node has no `userId` (it is `null` or `undefined`). Therefore `typeof approver.userId === 'number'` is always `false` for such nodes, making the entire expression always `false` — even when `count > 0` (i.e., a duplicate already exists).

This function is called as the sole duplicate-existence guard inside `createTransactionApprovers`: [2](#0-1) 

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
```

Because `isNode` always returns `false` for threshold nodes, the `APPROVER_ALREADY_EXISTS` error is never thrown for them, and the duplicate is silently inserted.

### Impact Explanation

A transaction creator (normal authenticated user) can call `POST /transactions/:transactionId/approvers` multiple times with the same threshold-node structure. Each call bypasses the duplicate check and inserts additional identical threshold nodes into the `transaction_approver` table. This corrupts the approval tree:

1. `getTreeStructure` reconstructs the tree from the flat list and will produce duplicate branches.
2. The duplicated threshold nodes are returned to all participants via `getApproversByTransactionId`, corrupting the displayed approval state.
3. Approval-counting logic that traverses the tree (e.g., determining whether the threshold is satisfied) may produce incorrect results, potentially allowing a transaction to be marked ready for execution with fewer real approvals than required, or permanently blocking execution.

### Likelihood Explanation

The attacker is the transaction creator — a normal, unprivileged authenticated user. No special keys or admin access are required. The exploit path is a standard API call (`POST /transactions/:transactionId/approvers`) with a threshold-only approver DTO (no `userId`). The condition is trivially reachable and requires no race condition or timing dependency.

### Recommendation

Remove the extra `&& typeof approver.userId === 'number'` condition. The duplicate check should fire for **all** node types:

```typescript
// Before (incorrect):
return count > 0 && typeof approver.userId === 'number';

// After (correct):
return count > 0;
``` [3](#0-2) 

### Proof of Concept

1. Authenticate as a transaction creator.
2. Create a transaction and note its `transactionId`.
3. Call `POST /transactions/:transactionId/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": 2 }] }] }
   ```
4. Call the same endpoint again with the identical body.
5. Call `GET /transactions/:transactionId/approvers`.
6. **Expected**: second call rejected with `APPROVER_ALREADY_EXISTS`.
7. **Actual**: second call succeeds; the response contains two identical threshold nodes, corrupting the approval tree.

  /* Check if the approver node already exists */
  async isNode(
    approver: CreateTransactionApproverDto,
    transactionId: number,
    entityManager?: EntityManager,
  ) {
    const find: FindManyOptions<TransactionApprover> = {
      where: {
        listId: typeof approver.listId === 'number' ? approver.listId : null,
        userId: typeof approver.userId === 'number' ? approver.userId : null,
        threshold:
          typeof approver.threshold === 'number' && approver.threshold !== 0
            ? approver.threshold
            : null,
        transactionId: typeof approver.listId === 'number' ? null : transactionId,
      },
    };

    const count = await (entityManager || this.repo).count(TransactionApprover, find);
    return count > 0 && typeof approver.userId === 'number';
```
