### Title
JWT Token Not Blacklisted After Password Change Allows Continued Use of Compromised Tokens

### Summary
The JWT blacklist mechanism is applied inconsistently across authentication operations. While `logout` correctly blacklists the current token, `changePassword` does not — leaving the old JWT fully valid for its entire remaining lifetime after a password change. This is the direct analog to the USDM blocklist issue: a security control (the blacklist) is enforced in some operations but silently skipped in others.

### Finding Description

The `BlacklistService` provides `blacklistToken` and `isTokenBlacklisted` to manage a Redis-backed JWT denylist. [1](#0-0) 

The `logout` endpoint correctly calls `blacklistToken` on the current auth JWT immediately after the operation: [2](#0-1) 

The OTP flow also correctly blacklists the OTP token after `verifyOtp` and `setPassword`: [3](#0-2) 

However, the `changePassword` endpoint — which is the primary mechanism a user would invoke to revoke access after a suspected compromise — **never blacklists the current auth token**: [4](#0-3) 

The underlying service method also performs no token invalidation: [5](#0-4) 

The JWT is signed with an expiry of `JWT_EXPIRATION` days, and the blacklist TTL mirrors this same value: [6](#0-5) [7](#0-6) 

This means a stolen token remains usable for the full configured expiry window even after the victim changes their password.

### Impact Explanation
An attacker who obtains a valid JWT (via session hijacking, token leakage, or insider access) retains full API access even after the legitimate user changes their password. The password change — the standard remediation action for a compromised account — provides no actual revocation of the stolen credential. All protected endpoints guarded by `JwtBlackListAuthGuard` + `JwtAuthGuard` remain accessible to the attacker.

### Likelihood Explanation
This is a realistic, low-complexity attack path. Token theft is a common threat (XSS, logging, network interception). Users routinely change passwords specifically to revoke access after suspecting compromise. The gap between user intent ("I changed my password to lock out the attacker") and actual behavior ("the old token still works") is concrete and exploitable without any special privileges.

### Recommendation
In `auth.controller.ts`, after a successful `changePassword` call, blacklist the current auth token and optionally issue a fresh one:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(
  @GetUser() user: User,
  @Body() dto: ChangePasswordDto,
  @Req() req: Request,
): Promise<void> {
  await this.authService.changePassword(user, dto);
  await this.blacklistService.blacklistToken(extractJwtAuth(req)); // ADD THIS
}
```

### Proof of Concept

1. Attacker obtains victim's JWT token `T` (e.g., via XSS or network interception).
2. Victim detects suspicious activity and calls `PATCH /auth/change-password` with valid credentials — request succeeds.
3. `changePassword` in `auth.controller.ts` (line 117) calls `authService.changePassword` but **never calls** `blacklistService.blacklistToken`.
4. Attacker uses token `T` on any guarded endpoint (e.g., `GET /users`, `POST /transactions`).
5. `JwtBlackListAuthGuard` checks Redis — token `T` is **not** in the blacklist — access is granted.
6. Attacker retains full API access until `T` naturally expires (`JWT_EXPIRATION` days later).

  async changePassword(user: User, { oldPassword, newPassword }: ChangePasswordDto): Promise<void> {
    if (oldPassword === newPassword) throw new BadRequestException(ErrorCodes.NPMOP);

    const { correct } = await this.dualCompareHash(oldPassword, user.password);
    if (!correct) throw new BadRequestException(ErrorCodes.INOP);

    if (user.status === UserStatus.NEW && user.keys.length === 0) {
      emitUserStatusUpdateNotifications(this.notificationsPublisher, { entityId: user.id, additionalData: { username: user.email } });
    }

    await this.usersService.setPassword(user, newPassword);
  }
