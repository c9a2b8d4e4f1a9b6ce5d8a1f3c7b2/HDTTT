### Title
`cutoffAt` Deadline Not Enforced in Signature Upload Path — Signers Can Submit After Creator-Set Cutoff

### Summary
The `Transaction` entity stores a `cutoffAt` field set by the transaction creator to define a hard deadline for signature collection. However, `SignersService.validateTransactionStatus()` — the only server-side gate before signatures are persisted — never checks whether `Date.now()` has passed `cutoffAt`. Any registered signer can therefore submit signatures after the creator-intended deadline, causing the transaction to accumulate signatures and potentially execute on the Hedera network past the cutoff window.

### Finding Description

**Root cause — missing `cutoffAt` check in `validateTransactionStatus`:**

`cutoffAt` is a nullable `Date` column on the `Transaction` entity: [1](#0-0) 

It is accepted from the creator at transaction-creation time and persisted without modification: [2](#0-1) 

When a signer calls `POST /transactions/sign` (i.e., `uploadSignatureMaps`), the only server-side validation before signatures are written is `validateTransactionStatus`: [3](#0-2) 

This method checks only:
1. Transaction status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.
2. The Hedera SDK-level expiry (`isExpired`).

There is **no check** of the form `if (transaction.cutoffAt && Date.now() > transaction.cutoffAt.getTime()) return ErrorCodes.XXX`. The `cutoffAt` value is loaded from the database but never consulted in the signing path.

**Exploit flow:**
1. Creator creates a transaction with `cutoffAt = T`.
2. Time advances past `T`.
3. A signer calls `POST /transactions/sign` with a valid `signatureMap`.
4. `validateTransactionStatus` passes (status is still `WAITING_FOR_SIGNATURES`, Hedera expiry has not elapsed).
5. Signatures are persisted; `processTransactionStatus` may flip the transaction to `WAITING_FOR_EXECUTION`.
6. The chain service schedules and submits the transaction to the Hedera network — after the creator's intended cutoff.

### Impact Explanation
The creator's `cutoffAt` deadline is the mechanism by which an organization controls the signature-collection window. Bypassing it allows a signer to unilaterally push a transaction past the cutoff into execution. For financial transactions (token transfers, account updates, node operations), this means funds or account state can be mutated on the Hedera mainnet after the creator believed the transaction was effectively dead. The impact is unauthorized state change / unintended asset movement.

### Likelihood Explanation
The attacker is any registered user who holds a signing key for the transaction and has received a signing notification. No privileged access is required. The signer simply delays their API call until after `cutoffAt`. The endpoint is reachable by any authenticated, verified user with a matching key.

### Recommendation
Add a `cutoffAt` guard inside `validateTransactionStatus` in `back-end/apps/api/src/transactions/signers/signers.service.ts`:

```typescript
private validateTransactionStatus(transaction: Transaction): string | null {
  if (
    transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
    transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
  ) {
    return ErrorCodes.TNRS;
  }

  // Enforce creator-set cutoff deadline
  if (transaction.cutoffAt && Date.now() > transaction.cutoffAt.getTime()) {
    return ErrorCodes.TC; // add a suitable "Transaction Cutoff" error code
  }

  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
  if (isExpired(sdkTransaction)) {
    return ErrorCodes.TE;
  }

  return null;
}
```

The same guard should be applied in the `approveTransaction` path in `approvers.service.ts` if `cutoffAt` is intended to gate approvals as well.

### Proof of Concept

1. Creator submits `POST /transactions` with `cutoffAt` set to `now + 60s`.
2. Wait 61 seconds.
3. Signer calls `POST /transactions/sign` with a valid `signatureMap` for the transaction.
4. **Expected**: `400 Bad Request` — cutoff has passed.
5. **Actual**: `200 OK` — signatures are persisted; `validateTransactionStatus` at lines 201–215 returns `null` (no error) because neither the status check nor `isExpired` catches the expired cutoff.
6. If the signer's key was the last required key, `processTransactionStatus` transitions the transaction to `WAITING_FOR_EXECUTION` and the chain service submits it to the Hedera network. [4](#0-3) [1](#0-0)

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
