### Title
Unbounded ZIP Decompression in File Import Paths Enables Zip-Bomb Resource Exhaustion

### Summary
The ZIP extraction functions in `front-end/src/main/utils/files.ts` perform streaming decompression with no limit on the decompressed output size. A crafted zip bomb ŌĆö a small compressed file that expands to gigabytes ŌĆö can exhaust the Electron process's heap memory (via `extractUnzipperFileToBuffer`) or fill the user's disk (via `extractUnzipperFile`). Both code paths are reachable through normal, documented import workflows without any privileged access.

### Finding Description

**Vulnerability class**: DoS via resource exhaustion during decompression of attacker-controlled compressed input (direct analog of the zstd streaming decompression bomb in the external report).

**Root cause**: The `unzipper` library performs streaming decompression. Neither extraction function checks the uncompressed size before or during extraction.

`extractUnzipperFileToBuffer` accumulates every decompressed chunk into an in-memory `Buffer[]` array with no bound:

```
// front-end/src/main/utils/files.ts lines 138-154
stream.on('data', (d: Buffer) => {
  // only abort check ŌĆö no size guard
  result.push(d);   // unbounded accumulation
})
.on('end', () => resolve(Buffer.concat(result)));
``` [1](#0-0) 

`extractUnzipperFile` pipes the decompressed stream directly to a `WriteStream` with no size guard:

```
stream.pipe(fs.createWriteStream(dist))  // unbounded disk write
``` [2](#0-1) 

**Exploit flow ŌĆö path 1 (memory, via V1 import)**:

`filterCandidate()` in `importV1.ts` calls `extractUnzipperFileToBuffer` for both the `.tx` and `.json` entries inside the ZIP, with no size check on either:

```
const txBytes   = await extractUnzipperFileToBuffer(txFile);   // line 32
const jsonBytes = await extractUnzipperFileToBuffer(jsonFile); // line 37
``` [3](#0-2) 

A zip bomb whose inner entry decompresses to, say, 2 GB will cause the Node.js heap to grow until the Electron process is OOM-killed.

**Exploit flow ŌĆö path 2 (disk, via public-key search)**:

`unzipFile()` calls `extractUnzipperFile` for every matching file in the archive, writing decompressed bytes to a temp directory under `app.getPath('temp')` with no size limit: [4](#0-3) 

A zip bomb here fills the user's disk.

### Impact Explanation

- **Memory path**: Electron/Node.js process is OOM-killed, crashing the application mid-workflow. Any unsigned or unsaved transaction state is lost.
- **Disk path**: Temp directory fills the OS partition, potentially making the entire system unresponsive or preventing other writes (e.g., key storage, transaction drafts).
- Both impacts are local to the user's machine, but the application becomes completely unavailable until the user manually cleans up.

### Likelihood Explanation

Both import workflows are standard, documented features of the application. The attacker precondition is only that the victim opens a malicious ZIP file through one of these flows:

1. **V1 transaction import**: A malicious organization participant sends a crafted `.zip` claiming to be a transaction export. The victim uses the "Import V1 transactions" workflow.
2. **Public key import**: A malicious peer sends a crafted `.zip` claiming to contain public keys. The victim uses the "Import public keys" modal.

No privileged access, no credentials, and no network access to the victim's machine are required. The attacker only needs to deliver a file ŌĆö a realistic social-engineering vector in a multi-party transaction signing tool where file exchange is the primary collaboration mechanism.

### Recommendation

1. **Check `uncompressedSize` before extraction**: The `unzipper.File` object exposes `uncompressedSize`. Reject any entry whose declared uncompressed size exceeds a reasonable threshold (e.g., 10 MB for `.tx`/`.json`/`.pub` files) before calling `file.stream()`.
2. **Add a streaming byte counter**: Even if `uncompressedSize` is spoofed or zero, track bytes received in the `data` handler and call `stream.destroy()` + reject the promise once the limit is exceeded ŌĆö analogous to the "bulk decompression with size cap" recommendation in the external report.
3. **Apply limits in both functions**: Both `extractUnzipperFile` and `extractUnzipperFileToBuffer` need independent guards.

### Proof of Concept

```python
import zipfile, io

# Create a zip bomb: 1 KB compressed ŌåÆ ~1 GB decompressed
payload = b'\x00' * (1024 * 1024 * 1024)  # 1 GB of zeros
buf = io.BytesIO()
with zipfile.ZipFile(buf, 'w', zipfile.ZIP_DEFLATED) as zf:
    zf.writestr('transaction.tx', payload)   # triggers memory path
    zf.writestr('signatures.json', payload)  # triggers memory path
with open('bomb.zip', 'wb') as f:
    f.write(buf.getvalue())
```

1. Deliver `bomb.zip` to a victim user.
2. Victim opens Hedera Transaction Tool ŌåÆ "Import V1 transactions" ŌåÆ selects `bomb.zip`.
3. `filterCandidate()` calls `extractUnzipperFileToBuffer(txFile)` ŌåÆ Node.js heap grows to ~1 GB ŌåÆ Electron process is OOM-killed.

For the disk path, use the same file via "Import public keys" ŌåÆ `searchFiles()` ŌåÆ `unzipFile()` ŌåÆ `extractUnzipperFile()` ŌåÆ temp directory fills disk.

const extractUnzipperFile = (file: unzipper.File, dist: string, abortSignal?: AbortSignal) => {
  return new Promise((resolve, reject) => {
    const stream = file.stream();
    stream
      .on('data', () => {
        if (abortSignal?.aborted) {
          stream.destroy();
          reject('File extraction aborted');
        }
      })
      .pipe(fs.createWriteStream(dist))
      .on('close', () => resolve(dist))
      .on('error', reject);
  });
}
```

export const extractUnzipperFileToBuffer = (file: unzipper.File, abortSignal?: AbortSignal): Promise<Buffer> => {
  return new Promise((resolve, reject) => {
    const result: Buffer[] = [];
    const stream = file.stream();
    stream
      .on('data', (d: Buffer) => {
        if (abortSignal?.aborted) {
          stream.destroy();
          reject('File extraction aborted');
        } else {
          result.push(d);
        }
      })
      .on('end', () => resolve(Buffer.concat(result)))
      .on('error', reject);
  });
}
