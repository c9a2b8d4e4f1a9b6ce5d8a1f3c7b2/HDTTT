### Title
Unbounded In-Memory Iteration Over All Active Transactions in `getTransactionsToSign` Enables Authenticated Resource Exhaustion DoS

### Summary
The `getTransactionsToSign` function in `TransactionsService` fetches **all** non-terminal transactions system-wide from the database without any user-scoping filter, then iterates over every one of them in an unbounded loop, performing per-transaction deserialization and database queries. An authenticated user can pre-populate the system with many long-lived transactions, causing every subsequent call to this endpoint by any user to consume unbounded server CPU and database connections, degrading or timing out the service for all users.

### Finding Description

**Root Cause — Missing User Filter at DB Level**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` constructs a `whereForUser` clause that filters only by transaction status, not by the requesting user: [1](#0-0) 

```typescript
const whereForUser: FindOptionsWhere<Transaction> = {
  ...where,
  status: Not(In([EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED])),
};

const transactions = await this.repo.find({
  where: whereForUser,   // ← no userId filter; fetches ALL active transactions
  relations: ['groupItem'],
  order,
});
```

Compare this to `getTransactions`, which correctly scopes to the user: [2](#0-1) 

**Unbounded Loop with Per-Iteration DB + CPU Work**

After loading all N active transactions, the function iterates over every one: [3](#0-2) 

Each call to `userKeysToSign` → `keysRequiredToSign` performs:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU deserialization
2. `transactionSignatureService.computeSignatureKey(transaction)` — potentially expensive key tree traversal
3. `entityManager.find(UserKey, ...)` — a database round-trip [4](#0-3) 

**Pagination Applied After Full Scan**

Pagination parameters are accepted but applied only after the full in-memory scan completes, providing no protection: [5](#0-4) 

**Attacker-Controlled Growth of the Transaction Pool**

Any authenticated user can call `createTransaction` with valid Hedera transaction bytes and a future `validStart` (up to one week ahead per the scheduler). There is no per-user transaction count cap enforced in the creation path: [6](#0-5) 

Transactions with future `validStart` remain in `WAITING_FOR_SIGNATURES` status and are not expired by the scheduler until their valid start passes: [7](#0-6) 

### Impact Explanation

Every call to `GET /transactions/sign` (or equivalent) by **any** user triggers O(N) database queries and O(N) cryptographic deserialization operations, where N is the total number of active transactions in the system. An attacker who pre-creates a large number of long-lived transactions causes:

- Sustained high CPU and database connection usage on the backend API server
- Response timeouts for all users attempting to retrieve their signing queue
- Cascading degradation of other endpoints sharing the same database connection pool

This constitutes a **service availability** impact affecting all organization users, not just the attacker.

### Likelihood Explanation

- **Precondition**: A valid registered account on the organization backend — no admin privileges required.
- **Cost**: Creating each transaction requires a valid Hedera SDK transaction object signed with the user's key. This is computationally cheap and can be scripted. Transactions with `validStart` set 7 days in the future persist for the full week.
- **Amplification**: A single attacker session creating thousands of transactions permanently degrades the endpoint for all users until those transactions expire or are manually cleaned up.
- **No rate limiting** on transaction creation is visible in the reviewed code paths.

### Recommendation

1. **Scope the DB query to the requesting user** — mirror the pattern used in `getTransactions` by adding user-scoped `WHERE` clauses (creator, signer, observer, approver) at the database level before fetching rows.
2. **Apply `take`/`skip` at the DB level** — pass `limit` and `offset` into `repo.find()` rather than slicing in memory after a full scan.
3. **Add a per-user transaction creation rate limit** — enforce a maximum number of active (non-terminal) transactions per user account to bound the pool size.
4. **Consider a server-side cap** on the total number of transactions returned by this endpoint regardless of system size.

### Proof of Concept

1. Register a user account on the backend API.
2. Script the creation of, e.g., 10,000 transactions each with `validStart` set 7 days in the future (valid Hedera transaction bytes, signed with the user's registered key).
3. As any user (including a different account), call `GET /transactions/sign`.
4. Observe: the server fetches all 10,000 transactions, performs 10,000 × (deserialization + DB query) operations before returning any response. Response time grows linearly with N; at sufficient scale the request times out and the endpoint becomes unavailable for all users.
