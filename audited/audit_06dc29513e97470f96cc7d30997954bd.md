### Title
Missing Validation of `mirrorNetwork` Parameter Enables SSRF via Arbitrary Network Identifier

### Summary

The `CreateTransactionDto` accepts a `mirrorNetwork` field that is only validated as a non-empty string with no whitelist enforcement against supported networks. When a transaction is created, the backend calls `getClientFromNetwork(dto.mirrorNetwork)`, which for any unrecognized value makes an outbound gRPC connection to `${mirrorNetwork}:443`. Any authenticated user can supply an arbitrary hostname or internal IP address, causing the backend to initiate outbound gRPC connections to attacker-controlled or internal infrastructure — a Server-Side Request Forgery (SSRF).

### Finding Description

**Root cause — no whitelist on `mirrorNetwork`:**

`CreateTransactionDto` decorates `mirrorNetwork` only with `@IsNotEmpty()` and `@IsString()`: [1](#0-0) 

No `@IsIn([...])` or `@IsEnum(...)` guard exists. Any string passes DTO validation.

**Code path that triggers the outbound connection:**

`validateAndPrepareTransaction` calls `getClientFromNetwork(dto.mirrorNetwork)`: [2](#0-1) 

`getClientFromNetwork` falls through to the custom-network branch for any value that is not `mainnet`, `testnet`, `previewnet`, or `local-node`: [3](#0-2) 

`MirrorNetworkGRPC.fromBaseURL` appends `:443` to the raw user-supplied string and uses it as the gRPC mirror-node endpoint: [4](#0-3) 

The `AddressBookQuery.execute(client)` call at line 43–44 of `client.ts` then initiates a live gRPC connection to `${mirrorNetwork}:443`.

**Secondary HTTP SSRF path (if gRPC succeeds):**

If the attacker controls a gRPC server that returns a valid address book, the transaction is persisted. The chain service later calls `MirrorNodeClient.fetchAccountInfo` / `fetchNodeInfo` with the stored `mirrorNetwork`. `getMirrorNodeRESTURL` constructs: [5](#0-4) 

which calls: [6](#0-5) 

producing `https://${mirrorNetwork}/api/v1/accounts/...` — a full HTTP SSRF to any URL the attacker embeds in `mirrorNetwork`.

### Impact Explanation

- **gRPC SSRF (immediate, no extra infrastructure):** Any authenticated user can force the API server to open a gRPC TCP connection to an arbitrary host:443. This enables internal port scanning, probing of internal gRPC services, and exfiltration of the server's egress IP.
- **HTTP SSRF (requires attacker-controlled gRPC server):** If the attacker operates a gRPC mirror-node stub, the transaction is stored and the chain service subsequently issues HTTP GET requests to `https://<attacker-host>/api/v1/accounts/...`, enabling access to internal REST services, cloud metadata endpoints (e.g., `169.254.169.254`), and credential theft in cloud environments.

### Likelihood Explanation

The attacker only needs a valid authenticated session (normal user account). No admin or privileged role is required. The gRPC SSRF path requires zero additional infrastructure — the connection attempt is made unconditionally during `POST /transactions`, even if it ultimately fails. The HTTP SSRF path requires the attacker to run a minimal gRPC stub that returns a syntactically valid address book, which is low effort.

### Recommendation

Add an allowlist validator to `mirrorNetwork` in `CreateTransactionDto`. For example:

```typescript
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

If custom mirror-node URLs must be supported, apply strict URL parsing and block RFC-1918 / link-local address ranges before any outbound connection is made.

### Proof of Concept

```
POST /transactions
Authorization: Bearer <valid_user_jwt>
Content-Type: application/json

{
  "name": "ssrf-test",
  "description": "ssrf",
  "transactionBytes": "<valid_hedera_tx_hex>",
  "creatorKeyId": <valid_key_id>,
  "signature": "<valid_sig_hex>",
  "mirrorNetwork": "169.254.169.254"
}
```

**Expected (vulnerable) behaviour:**
The API server calls `getClientFromNetwork("169.254.169.254")`, which calls `MirrorNetworkGRPC.fromBaseURL("169.254.169.254")` → `["169.254.169.254:443"]`, then executes `AddressBookQuery` against that endpoint. A TCP/gRPC connection to `169.254.169.254:443` is initiated from the backend host — confirming SSRF. On AWS, this reaches the instance metadata service.

**Expected (fixed) behaviour:**
DTO validation rejects the request with HTTP 400 before any network call is made
