### Title
Any Authenticated User Can Post Comments on and Read Comments from Transactions They Do Not Participate In

### Summary
`CommentsController` exposes `POST /transactions/:transactionId/comments` and `GET /transactions/:transactionId/comments` without any transaction-participation check. Any verified, authenticated user can inject comments into — and read comments from — any transaction in the system, regardless of whether they are the creator, signer, observer, or approver of that transaction. The code itself acknowledges the gap with a `//TODO need some sort of guard or check to ensure user can comment here` comment that was never resolved.

### Finding Description

**Root cause:** `CommentsController` applies only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the controller level. These guards confirm the caller is a logged-in, verified user — nothing more. Neither the controller nor `CommentsService` verifies that the caller has any relationship to the target transaction.

**Affected file:** `back-end/apps/api/src/transactions/comments/comments.controller.ts`

```
@Controller('transactions/:transactionId?/comments')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class CommentsController {

  @Post()
  //TODO need some sort of guard or check to ensure user can comment here  ← acknowledged gap
  createComment(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Body() dto: CreateCommentDto,
  ) {
    return this.commentsService.createComment(user, transactionId, dto);
  }

  @Get()
  getComments(@Param('transactionId', ParseIntPipe) transactionId: number) {
    return this.commentsService.getTransactionComments(transactionId);   // no user passed
  }
``` [1](#0-0) 

**`CommentsService.createComment`** performs no transaction-participation check — it directly creates and saves the comment:

```ts
async createComment(user, transactionId, dto) {
  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);   // no access check
}
``` [2](#0-1) 

**`CommentsService.getTransactionComments`** similarly performs no access check — it returns all comments for any `transactionId`: [3](#0-2) 

**Contrast with every other transaction sub-resource**, which all enforce participation checks:

- `ObserversService.getTransactionObserversByTransactionId` throws `UnauthorizedException` if the caller is not creator/observer/signer/approver. [4](#0-3) 

- `TransactionsService.getTransactionWithVerifiedAccess` calls `verifyAccess` before returning any data. [5](#0-4) 

- `ApproversService.getCreatorsTransaction` throws if the caller is not the creator. [6](#0-5) 

**Exploit path:**
1. Attacker registers and verifies an account (normal product flow).
2. Attacker enumerates or guesses any `transactionId` (integer IDs are sequential).
3. `POST /transactions/{victimTransactionId}/comments` with `{"message": "..."}` — comment is persisted to the victim's transaction.
4. `GET /transactions/{victimTransactionId}/comments` — all comments (including sensitive workflow notes from legitimate participants) are returned to the attacker.

### Impact Explanation

- **Unauthorized state mutation:** Any verified user can inject arbitrary comment records into any transaction they have no relationship to. This pollutes the audit trail and can disrupt multi-signature workflows (e.g., injecting misleading instructions or approvals into a Hedera Council transaction).
- **Information disclosure:** Comments on private, in-progress transactions (status `WAITING_FOR_SIGNATURES`, etc.) are readable by any authenticated user. Other endpoints correctly gate this data behind participation checks; comments do not.
- **Cross-tenant integrity break:** The system's trust model assumes only participants can interact with a transaction. Comments violate this invariant.

### Likelihood Explanation

- Attacker precondition: a valid, verified account — achievable through the normal signup flow (admin creates account, user sets password).
- No privileged keys, no leaked credentials, no internal network access required.
- Transaction IDs are sequential integers, trivially enumerable.
- The vulnerability is reachable via a single authenticated HTTP request.

### Recommendation

In `CommentsService.createComment`, call `TransactionsService.getTransactionWithVerifiedAccess(transactionId, user)` (or an equivalent participation check) before saving the comment. Apply the same check in `getTransactionComments` by passing the authenticated user and verifying access, mirroring the pattern used by `ObserversService` and `ApproversService`.

### Proof of Concept

```
# Step 1 – log in as a user who has NO relationship to transaction #42
POST /auth/login  { "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<TOKEN>" }

# Step 2 – inject a comment into a transaction the attacker does not own
POST /transactions/42/comments
Authorization: Bearer <TOKEN>
Content-Type: application/json
{ "message": "Attacker-controlled message" }
→ 201 Created  (comment persisted to victim transaction)

# Step 3 – read all comments on the same private transaction
GET /transactions/42/comments
Authorization: Bearer <TOKEN>
→ 200 OK  [ { "id": ..., "message": "...", "user": {...} }, ... ]
  (returns comments from participants the attacker has no right to see)
```

Expected outcome: both requests succeed with 2xx responses. The correct outcome should be 401 Unauthorized for a caller who is not a creator, signer, observer, or approver of transaction 42.
