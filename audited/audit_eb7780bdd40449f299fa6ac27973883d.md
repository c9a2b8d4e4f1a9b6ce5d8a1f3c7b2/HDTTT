### Title
Transaction Creator Can Modify or Remove Approvers on In-Progress Transactions, Bypassing the Approval Workflow

### Summary
The `updateTransactionApprover`, `removeTransactionApprover`, and `createTransactionApprovers` functions in `approvers.service.ts` perform no check on the current transaction status before mutating approver parameters. A transaction creator — a normal authenticated user — can remove a rejecting approver, lower the approval threshold, or replace an approver after approvals have already been collected. This nullifies rejections and breaks the integrity of the organizational approval workflow that is the protocol's primary governance control over multi-party transactions.

### Finding Description

**Root cause**: `getCreatorsTransaction` — the only authorization gate called before every approver mutation — checks only that the caller is the transaction creator. It never inspects `transaction.status`. [1](#0-0) 

Because of this, all three mutation paths are reachable regardless of whether the transaction is `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, or any other in-progress state.

**Path 1 — Remove a rejecting approver (controller)** [2](#0-1) 

`getCreatorsTransaction` is called to confirm creator identity, then `removeTransactionApprover` is called with no status guard: [3](#0-2) 

**Path 2 — Lower the threshold after partial approvals**

`updateTransactionApprover` calls `getCreatorsTransaction` (line 394) and then unconditionally writes the new threshold: [4](#0-3) 

No existing approval records are cleared when the threshold is lowered, so previously collected approvals now satisfy the reduced requirement.

**Path 3 — Replace an approver who rejected**

When `dto.userId` is updated, the old approver's `signature`, `userKeyId`, and `approved` fields are cleared: [5](#0-4) 

This erases a rejection (`approved = false`) and resets the slot to pending, effectively removing the veto.

**Contrast with `approveTransaction`**, which correctly guards against acting on a terminal or non-approvable transaction: [6](#0-5) 

No equivalent guard exists for the mutation endpoints.

### Impact Explanation

The approval system is the organizational governance layer for multi-party Hedera transactions. Approvers are designated to review and either approve or reject a transaction before it proceeds. By exploiting this gap, a transaction creator can:

1. **Nullify a rejection**: Remove or replace an approver who voted `approved = false`, erasing their veto.
2. **Retroactively lower the threshold**: After N-1 of N required approvers have approved, reduce the threshold to N-1, making the approval condition immediately satisfied without the final approver's consent.
3. **Add new approvers to a transaction already in `WAITING_FOR_EXECUTION`**: `createTransactionApprovers` has the same missing status check, allowing the creator to inject approvers into a transaction that has already cleared the approval stage.

The result is that the approval workflow — the primary multi-party control mechanism — can be unilaterally bypassed by the transaction creator, who is supposed to be subject to it.

### Likelihood Explanation

The attacker is the transaction creator: a normal, authenticated, non-admin user. No leaked credentials, no admin access, and no cryptographic break are required. The attack path is a standard authenticated REST call (`DELETE /transactions/:transactionId/approvers/:id` or `PATCH /transactions/:transactionId/approvers/:id`). Any creator who faces a rejection has both the motive and the direct API access to exploit this. Likelihood is high.

### Recommendation

Add a transaction status guard at the start of `updateTransactionApprover`, `removeTransactionApprover` (or in the controller before calling it), and `createTransactionApprovers`. Approver mutations should only be permitted when the transaction is in `NEW` or `WAITING_FOR_SIGNATURES` status — before any approvals have been collected. Once any approver has acted (`approved` is non-null or `signature` is set), the approver structure should be frozen.

```typescript
// Example guard to add inside getCreatorsTransaction or at the top of each mutation:
const MUTABLE_STATUSES = [TransactionStatus.NEW, TransactionStatus.WAITING_FOR_SIGNATURES];
if (!MUTABLE_STATUSES.includes(transaction.status)) {
  throw new BadRequestException('Approvers cannot be modified after approval collection has begun');
}
```

### Proof of Concept

**Setup**: Organization with users Alice (creator), Bob, and Carol. Transaction T requires threshold-2-of-2 approval from Bob and Carol.

1. Alice creates transaction T with two approvers: Bob and Carol, threshold = 2.
2. Bob approves (`POST /transactions/T/approvers/approve`, `approved: true`).
3. Carol rejects (`POST /transactions/T/approvers/approve`, `approved: false`).
4. Alice calls `DELETE /transactions/T/approvers/<carol_approver_id>` — no status check blocks this.
5. Carol's rejection record is soft-deleted via `removeNode`.
6. Alice calls `POST /transactions/T/approvers` to add a new approver Dave (or lowers the threshold to 1 via `PATCH /transactions/T/approvers/<list_id>` with `{threshold: 1}`).
7. The approval requirement is now satisfied with only Bob's approval, bypassing Carol's explicit rejection.

**Expected**: `400 Bad Request` — approvers cannot be modified after approval collection has begun.
**Actual**: `200 OK` — Carol's rejection is silently erased. [2](#0-1) [3](#0-2) [7](#0-6)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
