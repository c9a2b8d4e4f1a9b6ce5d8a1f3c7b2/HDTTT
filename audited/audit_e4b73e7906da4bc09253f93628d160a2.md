### Title
`NotificationPreferencesController` Missing `VerifiedUserGuard` — Unverified Users Can Access Notification Preferences

---

### Summary

Every other API controller in the application applies `VerifiedUserGuard` as part of its class-level guard chain. `NotificationPreferencesController` applies only `JwtBlackListAuthGuard` and `JwtAuthGuard`, silently omitting `VerifiedUserGuard`. This is a direct structural analog to the ENS `acceptRegistrarTransfer` bug: one function/class in a family is missing the access-control modifier that all its siblings carry.

---

### Finding Description

All controllers in `back-end/apps/api/src/` follow the same guard pattern at the class level:

```
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
```

Confirmed across:

- `UsersController` [1](#0-0) 
- `TransactionsController` [2](#0-1) 
- `ApproversController` [3](#0-2) 
- `ObserversController` [4](#0-3) 
- `SignersController` [5](#0-4) 
- `TransactionGroupsController` [6](#0-5) 
- `UserKeysController` [7](#0-6) 
- `UserKeysAllController` [8](#0-7) 

`NotificationPreferencesController` breaks this pattern — it applies only two guards, dropping `VerifiedUserGuard`:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)   // VerifiedUserGuard absent
``` [9](#0-8) 

The two exposed endpoints are:

- `PATCH /notification-preferences` — `updatePreferences` [10](#0-9) 
- `GET /notification-preferences` — `getPreferencesOrCreate` [11](#0-10) 

The codebase explicitly uses `@AllowNonVerifiedUser()` on the handful of endpoints that are intentionally accessible to unverified users (e.g., `GET /users/me`, `GET /users/public-owner/:publicKey`, `POST /users/version-check`). [12](#0-11)  The absence of both `VerifiedUserGuard` **and** `@AllowNonVerifiedUser()` on `NotificationPreferencesController` is inconsistent and indicates an oversight, not an intentional design choice.

---

### Impact Explanation

A user who has registered an account but has **not** completed email verification can:

1. Read their notification preferences (`GET /notification-preferences`).
2. Mutate their notification preferences (`PATCH /notification-preferences`), silencing or enabling any notification type.

In a multi-signature workflow tool, notification preferences govern whether a user is alerted to pending signing/approval requests. An unverified (and therefore untrusted) account can suppress its own notifications, reducing auditability and potentially hiding activity from administrators.

---

### Likelihood Explanation

Any attacker who can register an account (the `/auth/signup` endpoint is admin-gated, but the password-reset flow creates temporary OTP-authenticated sessions) and skip email verification can immediately reach these endpoints. The missing guard is reachable via a standard HTTP client with a valid JWT — no special privilege or race condition required.

---

### Recommendation

Add `VerifiedUserGuard` to the class-level guard chain in `NotificationPreferencesController`, consistent with every other controller in the application:

```typescript
// back-end/apps/api/src/notification-preferences/notification-preferences.controller.ts

@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)  // add VerifiedUserGuard
```

If either endpoint genuinely needs to be accessible to unverified users, annotate it explicitly with `@AllowNonVerifiedUser()` so the intent is clear and auditable.

---

### Proof of Concept

1. Register a new user account (via admin `POST /auth/signup`).
2. Do **not** complete email verification — the account remains unverified.
3. Log in (`POST /auth/login`) to obtain a JWT.
4. Issue `GET /notification-preferences` with the JWT → **200 OK**, preferences returned.
5. Issue `PATCH /notification-preferences` with a body disabling all notifications → **200 OK**, preferences mutated.
6. Repeat step 4 on any endpoint protected by `VerifiedUserGuard` (e.g., `GET /transactions`) → **403 Forbidden**, confirming the guard is enforced everywhere else.
