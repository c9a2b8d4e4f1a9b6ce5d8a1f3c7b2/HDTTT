### Title
Unbounded Full-Table Scan with Per-Row Async I/O in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

### Summary
`TransactionsService.getTransactionsToSign()` fetches every active transaction in the database with no row limit, then serially calls `userKeysToSign()` — which performs deserialization and async mirror-node/cache I/O — for each row before applying pagination. Any authenticated, verified user can trigger this endpoint to exhaust database connections, memory, and CPU proportional to the total number of active transactions in the system, causing severe service degradation or complete unavailability for all users.

### Finding Description

**Root cause — no `take` limit on the initial query:**

In `getTransactionsToSign()`, the `repo.find()` call at line 295 omits `skip`/`take`, so it loads every non-terminal transaction in the entire database into memory:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no skip / take
});
``` [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED), meaning every `NEW` and `WAITING_FOR_SIGNATURES` transaction across the entire organization is returned. [2](#0-1) 

**Root cause — serial async I/O loop over the full result set:**

The code then iterates over every returned transaction and calls `userKeysToSign()` for each one:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [3](#0-2) 

`userKeysToSign` calls `userKeysRequiredToSign`, which calls `keysRequiredToSign`, which calls `transactionSignatureService.computeSignatureKey()`. Each `computeSignatureKey()` call:
- Deserializes `transactionBytes` via the Hedera SDK
- Makes one or more async calls to `accountCacheService.getAccountInfoForTransaction()` (mirror node or DB) for the fee payer, signing accounts, and receiver accounts
- Potentially calls `nodeCacheService.getNodeInfoForTransaction()` for node transactions [4](#0-3) 

**Pagination is applied only after the full loop completes:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ← sliced after full scan
  ...
};
``` [5](#0-4) 

**Reachable entry point:**

The endpoint `GET /transactions/sign` is live and guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard authenticated user access, no admin role required: [6](#0-5) [7](#0-6) 

The controller comment even acknowledges the endpoint is "NO LONGER USED BY FRONT-END" — meaning it is a forgotten, permanently-open attack surface. [8](#0-7) 

**Rate limiting does not mitigate this:**

The `UserThrottlerGuard` allows 100 requests/minute and 10 requests/second per user. A single request to this endpoint with N active transactions causes N deserialization operations and up to N×(signing accounts + 1) async I/O calls. The damage is per-request, not per-request-volume. [9](#0-8) 

### Impact Explanation

With a large number of active transactions (e.g., thousands in a busy organization):

- **Memory exhaustion**: All transaction rows and their `groupItem` relations are loaded into the Node.js heap simultaneously.
- **Database connection starvation**: Each `computeSignatureKey()` call may issue multiple DB queries (UserKey lookups, account cache misses). With N transactions in the loop, this can saturate the DB connection pool.
- **CPU exhaustion**: Hedera SDK deserialization (`SDKTransaction.fromBytes`) is CPU-bound and is called once per transaction per request.
- **Cascading failure**: Because the loop is serial (`await` inside `for...of`), the request holds an open HTTP connection and DB resources for the entire duration, blocking other requests from being served.

A single authenticated user making one request can render the API unresponsive for all other users.

### Likelihood Explanation

- **Attacker precondition**: Any verified user account — the lowest privilege level in the system. No admin, no special role.
- **Trigger**: One HTTP GET request to `GET /transactions/sign?page=1&size=1`.
- **Realistic scenario**: In an organization that has processed thousands of transactions over time, even a fraction remaining in non-terminal states (e.g., `WAITING_FOR_SIGNATURES`) is sufficient to cause severe degradation. The attacker does not need to create any transactions themselves.
- **Amplification**: The attacker can repeat the request up to 100 times per minute (within the throttle limit), sustaining the DoS.

### Recommendation

1. **Apply a database-level limit before the loop.** Add `take` and `skip` to the `repo.find()` call so only the page-sized slice is fetched and processed:
   ```typescript
   const transactions = await this.repo.find({
     where: whereForUser,
     relations: ['groupItem'],
     order,
     skip: offset,
     take: limit,
   });
   ```
   This requires moving the signing-key check into the DB query (e.g., via a subquery or a separate indexed column), or accepting that `totalItems` will be approximate.

2. **Alternatively, remove or gate the endpoint.** Since the front-end no longer uses `GET /transactions/sign`, disable it or protect it with an admin-only guard until it is properly reimplemented.

3. **Apply a hard cap on active transactions returned.** Even if exact pagination is not feasible, add a `take: MAX_SAFE_LIMIT` to prevent unbounded loads.

### Proof of Concept

**Preconditions:**
- An organization with ≥1,000 active (non-terminal) transactions in the database.
- One valid verified-user JWT token.

**Steps:**
1. Authenticate as any verified user and obtain a JWT.
2. Send: `GET /transactions/sign?page=1&size=1` with the JWT.
3. Observe: The server loads all N active transactions, calls `computeSignatureKey()` N times (each involving SDK deserialization + async mirror-node/cache I/O), and holds the connection open for the entire duration.
4. Repeat up to 100 times per minute (within the throttle window).

**Expected outcome:** API response latency for all endpoints increases proportionally to N. With N in the thousands, the Node.js event loop stalls, DB connection pool is exhausted, and the service becomes unavailable to all users for the duration of each request.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
