### Title
Transaction Creator Can Delete Approvers Belonging to Any Other Transaction via Missing Cross-Transaction Authorization in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to the authorized `:transactionId`. Any authenticated transaction creator can therefore delete approvers from transactions they do not own, bypassing the approval workflow of arbitrary transactions.

### Finding Description
**Root cause:** The controller's `removeTransactionApprover` handler performs ownership verification on the URL's `transactionId` parameter, then passes the separate `id` (approver ID) parameter directly to the service — which performs no cross-transaction membership check.

**Controller** (`back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, lines 102–113):

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no membership check
  return true;
}

**Service** (`back-end/apps/api/src/transactions/approvers/approvers.service.ts`, lines 533–544):

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // ← deletes whatever approver.id points to
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

The service only checks that the approver record exists — it never validates `approver.transactionId === transactionId`. The `getCreatorsTransaction` guard is therefore trivially bypassed by supplying the attacker's own `transactionId` in the URL while targeting a victim's approver `id`.

**Contrast with `updateTransactionApprover`**, which correctly validates cross-transaction membership:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The delete path has no equivalent guard.

**Exploit path:**
1. Attacker is an authenticated, verified user and the creator of transaction A (`transactionId=1`).
2. Attacker discovers approver record ID X (e.g., via the `GET /transactions/:id/approvers` endpoint for any transaction they can observe, or by enumeration).
3. Attacker sends: `DELETE /transactions/1/approvers/X`
4. `getCreatorsTransaction(1, attacker)` passes — attacker owns transaction 1.
5. `removeTransactionApprover(X)` executes — deletes approver X, which belongs to victim's transaction B.

### Impact Explanation
An attacker can silently remove required approvers from any transaction in the system. In the Hedera Transaction Tool, approvers gate whether a transaction may proceed to execution. Removing them can:
- Allow a transaction to advance to execution without the required approval signatures.
- Permanently corrupt the approval workflow of another user's transaction.
- Cause transactions requiring multi-party approval to execute unilaterally.

This is an unauthorized state mutation with direct impact on transaction integrity and the trust model of the approval system.

### Likelihood Explanation
Any authenticated, verified user who has created at least one transaction can exploit this. No admin privileges, leaked secrets, or special network access are required. Approver IDs are sequential integers and discoverable via the `GET /transactions/:id/approvers` endpoint for transactions the attacker participates in. The attack requires only a crafted HTTP DELETE request.

### Recommendation
In `removeTransactionApprover` (service), after fetching the approver, assert that it belongs to the authorized transaction before deleting:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the authorized transaction
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, mirroring the pattern already used in `updateTransactionApprover`.

### Proof of Concept

**Setup:**
- User A creates transaction 1 (attacker-owned).
- User B creates transaction 2 with approver record ID=99 (victim-owned).

**Attack request (as User A):**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer <User A's JWT>
```

**Expected (correct) behavior:** 403/401 — approver 99 does not belong to transaction 1.

**Actual behavior:** 200 `true` — approver 99 is deleted from transaction 2 without User A having any ownership or permission over transaction 2. [1](#0-0) [2](#0-1)
