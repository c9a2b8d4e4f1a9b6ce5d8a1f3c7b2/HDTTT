### Title
Cross-Transaction Approver Deletion via IDOR in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the caller against `transactionId` (the URL parameter) but then deletes the approver record identified by `:id` without verifying that the approver actually belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` for authorization and a victim's approver `id` for deletion.

### Finding Description

**Root cause — authorization check is decoupled from the resource being mutated.**

In `approvers.controller.ts`, `removeTransactionApprover` performs two independent operations:

```
await this.approversService.getCreatorsTransaction(transactionId, user);  // authorizes on transactionId
await this.approversService.removeTransactionApprover(id);                // deletes approver by id
``` [1](#0-0) 

`getCreatorsTransaction` only checks that the calling user is the creator of the transaction identified by `transactionId` (the URL parameter): [2](#0-1) 

`removeTransactionApprover` then deletes the approver record identified by `id` with **no cross-check** that the approver belongs to the authorized transaction: [3](#0-2) 

Contrast this with `updateTransactionApprover`, which correctly validates that the approver's root node belongs to the same `transactionId` before proceeding: [4](#0-3) 

**Exploit path:**
1. Attacker registers as a normal user and creates Transaction A (they become its creator).
2. Attacker discovers approver ID `N` belonging to Transaction B (owned by another user). Approver IDs are sequential integers; they can be enumerated by calling `GET /transactions/:id/approvers` on any transaction the attacker has legitimate access to, or by brute-forcing the integer space.
3. Attacker sends `DELETE /transactions/{A_id}/approvers/{N}`.
4. Controller calls `getCreatorsTransaction(A_id, attacker)` → passes (attacker is creator of A).
5. Controller calls `removeTransactionApprover(N)` → deletes approver `N` from Transaction B without any ownership check.

### Impact Explanation

An attacker can silently remove any approver from any transaction in the organization, bypassing the multi-signature approval workflow. Concretely:

- A transaction requiring M-of-N approvals can have its approvers stripped down, allowing it to proceed with fewer (or zero) required approvals.
- The transaction creator's approval tree is corrupted without their knowledge.
- This is an unauthorized state change that breaks the integrity of the organization's transaction governance model.

### Likelihood Explanation

- **Preconditions**: Only a valid, verified organization account is required. No admin or elevated privileges are needed.
- **Discovery of approver IDs**: IDs are sequential integers. Any approver ID visible to the attacker (e.g., from their own transactions) gives a starting point for enumeration. The attacker can also observe approver IDs from transactions where they are a signer or observer.
- **No rate limiting on DELETE**: The endpoint has no special throttling beyond standard JWT auth.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted belongs to the transaction that was authorized. The fix mirrors what `updateTransactionApprover` already does correctly:

```typescript
// In removeTransactionApprover or in the controller:
const rootNode = await this.getRootNodeFromNode(approver.id);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and perform the ownership check inside the service method, keeping authorization logic co-located with the mutation.

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (A is creator).
- User B creates Transaction 2 with an approver tree; approver record has ID = 7.

**Attack:**
```http
DELETE /transactions/1/approvers/7
Authorization: Bearer <User A's JWT>
```

**Expected (correct) behavior:** 403/401 — approver 7 does not belong to transaction 1.

**Actual behavior:** 200 — approver 7 is deleted from Transaction 2's approval tree, with no error, because `getCreatorsTransaction(1, userA)` passes and `removeTransactionApprover(7)` performs no cross-transaction ownership check. [1](#0-0) [3](#0-2)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
