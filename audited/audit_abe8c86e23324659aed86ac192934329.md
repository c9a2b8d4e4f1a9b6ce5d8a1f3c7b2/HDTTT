### Title
IDOR in `removeTransactionApprover`: Creator Check on Wrong Resource Allows Any Approver Deletion

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that this approver actually belongs to that transaction. An attacker who is the creator of any transaction can delete approvers belonging to any other user's transaction by supplying their own `transactionId` in the URL while targeting a victim's approver `id`.

### Finding Description
The controller method `removeTransactionApprover` in `back-end/apps/api/src/transactions/approvers/approvers.controller.ts` performs two sequential operations:

```typescript
// Line 108: checks user is creator of :transactionId
await this.approversService.getCreatorsTransaction(transactionId, user);
// Line 109: deletes approver by :id — no cross-check that :id belongs to :transactionId
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

The service method `removeTransactionApprover` accepts only the approver `id` and performs no ownership or transaction-membership check:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    ...
}
``` [2](#0-1) 

Contrast this with `updateTransactionApprover`, which correctly validates that the approver's root transaction matches the URL parameter before proceeding:

```typescript
if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, transactionalEntityManager);
``` [3](#0-2) 

The `removeTransactionApprover` service method has no equivalent cross-check, making it an IDOR. The authorization check is applied to `transactionId` (the URL parameter the attacker controls) while the destructive action is applied to `id` (the approver ID, which can belong to any transaction).

### Impact Explanation
An authenticated attacker who is the creator of at least one transaction can:
1. Delete any approver record from any other user's transaction.
2. Remove required approvers from a pending multi-signature transaction, collapsing the threshold structure and potentially allowing the transaction to advance to execution without the intended approvals.
3. Silently corrupt another organization member's transaction workflow — the victim's transaction may become executable with fewer approvals than intended, or the approver tree may be left in an inconsistent state.

This is an unauthorized state change that directly undermines the multi-signature trust model, which is the primary security invariant of the system.

### Likelihood Explanation
- The attacker only needs a valid authenticated session and to be the creator of any one transaction (a normal user capability).
- Approver IDs are sequential auto-increment integers in PostgreSQL, making enumeration trivial.
- No special privileges, leaked credentials, or physical access are required.
- The attack is a single HTTP DELETE request.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted belongs to the transaction the user is authorized to modify. The simplest fix mirrors what `updateTransactionApprover` already does:

```typescript
// In the controller, after getCreatorsTransaction:
const approver = await this.approversService.getTransactionApproverById(id);
const root = await this.approversService.getRootNodeFromNode(approver.id);
if (root.transactionId !== transactionId) throw new UnauthorizedException(...);
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and enforce the membership check inside the service, consistent with how `updateTransactionApprover` handles it. [4](#0-3) 

### Proof of Concept

**Preconditions:**
- Attacker is a normal authenticated user (no admin required).
- Attacker has created at least one transaction (`txA`) — they are its creator.
- Victim has a transaction (`txB`) with an approver whose ID is `approverB_id` (discoverable by enumeration or by being an observer/approver of `txB`).

**Steps:**
1. Attacker authenticates and obtains a JWT.
2. Attacker sends:
   ```
   DELETE /transactions/{txA_id}/approvers/{approverB_id}
   Authorization: Bearer <attacker_token>
   ```
3. `getCreatorsTransaction(txA_id, attacker)` succeeds — attacker is creator of `txA`.
4. `removeTransactionApprover(approverB_id)` executes with no further check.
5. The approver belonging to `txB` is soft-deleted from the database.

**Expected (correct) behavior:** Request should be rejected with 401/403 because `approverB_id` does not belong to `txA`.

**Actual behavior:** The approver is deleted. The victim's transaction `txB` now has a corrupted approver structure, potentially allowing it to proceed without the required signatures.

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
