### Title
Signature Upload Writes to Transactions Without Atomic Status Guard, Enabling Inconsistent Signer State

### Summary
`uploadSignatureMaps` in `SignersService` validates transaction status from a stale in-memory snapshot loaded at the start of the request, but the subsequent database writes (`bulkUpdateTransactions`, `bulkInsertSigners`) contain no status guard in their SQL. This is a direct analog to the `TokenMigration` pattern: a precondition is checked at one point in time, but the state mutation is not gated on that precondition at write time. Under concurrent signing requests, this produces a split state where `TransactionSigner` records exist for signers whose cryptographic signatures have been silently overwritten from `transactionBytes`, causing the transaction to fail execution even though the system believes it is fully signed.

### Finding Description

**Root cause — stale read, unguarded write:**

`loadTransactionData` fetches all transactions into an in-memory `Map` at the start of the request: [1](#0-0) 

`validateAndProcessSignatures` then checks the status from that in-memory snapshot: [2](#0-1) 

`validateTransactionStatus` itself only reads the object already in memory — it never re-queries the database: [3](#0-2) 

The actual database write in `bulkUpdateTransactions` is a raw SQL `UPDATE` with **no status predicate**: [4](#0-3) 

Compare this to `cancelTransaction`, which correctly uses an atomic status guard in its `WHERE` clause to prevent TOCTOU: [5](#0-4) 

`bulkInsertSigners` similarly has no guard: [6](#0-5) 

**Exploit flow — concurrent signers:**

1. Transaction T is in `WAITING_FOR_SIGNATURES` with `transactionBytes = B0`.
2. Signer A and Signer B both call `POST /transactions/:id/signers` concurrently.
3. Both requests execute `loadTransactionData` and read `B0`.
4. Both pass `validateTransactionStatus` (status is valid in both snapshots).
5. A's request computes `B0 + A_sig` and B's request computes `B0 + B_sig`.
6. A's `bulkUpdateTransactions` writes `B0 + A_sig` to the DB.
7. B's `bulkUpdateTransactions` writes `B0 + B_sig` to the DB — **overwriting A's signatures**.
8. Both `bulkInsertSigners` succeed, recording both A and B as `TransactionSigner` rows.

**Result:** The DB records A and B as signers, but `transactionBytes` contains only B's signature. When `processTransactionStatus` or `execute.service.ts` evaluates whether the transaction has sufficient signatures, it reads `transactionBytes` and finds A's signature missing, even though A is listed as a signer. [7](#0-6) 

**Secondary exploit — write to canceled transaction:**

1. Transaction T is in `WAITING_FOR_SIGNATURES`.
2. Signer loads T; status check passes.
3. Creator concurrently cancels T (status → `CANCELED`).
4. `bulkUpdateTransactions` writes new `transactionBytes` to the now-canceled T.
5. `bulkInsertSigners` inserts a `TransactionSigner` row for the canceled T.

This produces orphaned signer records and mutated bytes on a terminal-state transaction, corrupting audit trails.

### Impact Explanation

In the concurrent-signer scenario, one signer's cryptographic signatures are silently dropped from `transactionBytes` while their `TransactionSigner` row persists. The system then believes the transaction is fully signed (because it counts signer rows), but execution fails or produces an invalid transaction because the required signature is absent from the bytes. In a multi-signature Hedera transaction requiring threshold keys, this can permanently block execution of a valid transaction — the transaction expires without being submitted, and the Hedera operation is lost. This is a state integrity failure directly analogous to the double-mint in the `TokenMigration` report: the accounting (signer records) diverges from the actual state (transaction bytes).

### Likelihood Explanation

The Hedera Transaction Tool is explicitly designed for multi-user, multi-signer organizational workflows. Multiple signers acting on the same transaction concurrently is the normal operating mode, not an edge case. The API is a REST service with no per-transaction locking, making concurrent `POST /transactions/:id/signers` requests routine. No privileged access is required — any authenticated organization member who is a required signer can trigger this path.

### Recommendation

Add a status predicate to `bulkUpdateTransactions` so the `UPDATE` is a no-op if the transaction has left a signable state between the check and the write:

```sql
UPDATE transaction
SET "transactionBytes" = CASE id ... END,
    "updatedAt" = NOW()
WHERE id = ANY($N)
  AND status IN ('WAITING_FOR_SIGNATURES', 'WAITING_FOR_EXECUTION')
```

Similarly, gate `bulkInsertSigners` on the current DB status, or use a `SELECT ... FOR UPDATE` lock on the transaction row at the start of `loadTransactionData` to serialize concurrent signing requests per transaction. The pattern already used in `cancelTransaction` (optimistic `WHERE status IN (...)` guard) and in `processTransactionStatus` (optimistic `AND status = :oldStatus` guard) should be applied consistently here. [8](#0-7) 

### Proof of Concept

1. Create a transaction T requiring two signers (A and B).
2. Have A and B simultaneously submit their signatures via `POST /transactions/T/signers` (e.g., using two parallel HTTP clients with identical timing).
3. After both requests complete with HTTP 201, query `TransactionSigner` — both A and B are listed.
4. Decode `transaction.transactionBytes` from the DB — only one signer's signature is present in the bytes.
5. Attempt to execute T — execution fails or the Hedera network rejects it due to missing signatures, even though the system reports it as `WAITING_FOR_EXECUTION`. [9](#0-8)

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
