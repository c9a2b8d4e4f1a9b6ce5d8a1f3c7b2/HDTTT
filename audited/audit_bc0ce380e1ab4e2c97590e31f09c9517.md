Looking at the vulnerability class — **Invalid Validation / missing access control check** — I need to find endpoints where a user-scoping check is absent when it should be present (or present when it shouldn't be). Let me examine the key finding.

### Title
Missing User-Scoped Access Control in `getHistoryTransactions` Exposes All Users' Terminal Transactions

### Summary
The `GET /transactions/history` endpoint in the API service returns all terminal-state transactions (executed, failed, expired, canceled, archived) from **all users** to any authenticated requester. Unlike the sibling `GET /transactions` endpoint, which correctly filters results to the requesting user, `getHistoryTransactions` accepts no `user` parameter and applies no per-user filter, making every organization member's historical transaction data visible to every other member.

### Finding Description

**Root cause — missing user parameter and filter:**

In `transactions.controller.ts`, the `getHistoryTransactions` handler has no `@GetUser()` decorator and passes no user to the service: [1](#0-0) 

```typescript
@Get('/history')
getHistoryTransactions(
  @PaginationParams() paginationParams: Pagination,
  @SortingParams(transactionProperties) sort?: Sorting[],
  @FilteringParams({...}) filter?: Filtering[],
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getHistoryTransactions(paginationParams, filter, sort);
}
```

The service method signature confirms no user is accepted: [2](#0-1) 

```typescript
async getHistoryTransactions(
  { page, limit, size, offset }: Pagination,
  filter: Filtering[] = [],
  sort: Sorting[] = [],
): Promise<PaginatedResourceDto<Transaction>> {
  // No user filter — returns ALL terminal transactions
  const findOptions: FindManyOptions<Transaction> = {
    where: { ...getWhere<Transaction>(filter), status: this.getHistoryStatusWhere(filter) },
    ...
  };
  const [transactions, total] = await this.repo.createQueryBuilder()
    .setFindOptions(findOptions).getManyAndCount();
  ...
}
```

**Contrast with the correctly-scoped sibling endpoint:**

`getTransactions` (the non-history variant) correctly receives `user: User` and restricts results to transactions where the user is creator, signer, observer, or approver: [3](#0-2) [4](#0-3) 

The history endpoint is the only one in the controller that omits `@GetUser()` entirely.

**Analog to the external report:**
The external report identified `balance_of()` as having an *overly restrictive* check (`ensure_system`) that blocked legitimate callers. Here the analog is the *inverse*: `getHistoryTransactions` has a *missing* restriction — the user-scoping check present in every other data-retrieval endpoint is simply absent, allowing any authenticated caller to read data belonging to all other users.

### Impact Explanation

Any authenticated organization member can call `GET /transactions/history` (with optional pagination/filter/sort query parameters) and receive the full set of terminal-state transactions for **every other user** in the organization. This includes:

- Transaction IDs, types, amounts, and counterparties
- Creator key associations (linking keys to users)
- Transaction bytes (the raw signed Hedera transaction payload)
- Mirror network and valid-start timestamps

This constitutes a **cross-tenant data exposure** of sensitive financial records. An attacker can enumerate the entire organization's transaction history with a single authenticated HTTP request.

### Likelihood Explanation

- **Precondition**: valid JWT for any organization account — achievable by any registered user.
- **Attack path**: `GET /transactions/history` with no special parameters.
- **No privilege escalation required**: the endpoint is behind `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` only — the same guards as every other transaction endpoint.
- The front-end comment `/* NO LONGER USED BY FRONT-END */` on `getTransactions` suggests `getHistoryTransactions` may now be the primary history view, increasing real-world exposure.

### Recommendation

Pass the authenticated user into `getHistoryTransactions` and apply the same user-scoping logic used in `getTransactions`:

1. Add `@GetUser() user: User` to the controller handler.
2. Pass `user` to `transactionsService.getHistoryTransactions(...)`.
3. In the service, add a `WHERE` clause restricting results to transactions where `creatorKey.userId = user.id OR observers.userId = user.id OR signers.userKey.userId = user.id OR approvers.userId = user.id`, mirroring the logic in `getTransactions`.

### Proof of Concept

1. Register two accounts (User A and User B) in the same organization.
2. As User A, create and execute a transaction. It moves to `EXECUTED` status.
3. As User B, send:
   ```
   GET /transactions/history
   Authorization: Bearer <User B's JWT>
   ```
4. **Expected (secure)**: response contains only User B's own history.
5. **Actual (vulnerable)**: response contains User A's executed transaction, including its `transactionBytes`, `transactionId`, `creatorKey`, and all metadata. [1](#0-0) [2](#0-1)

```typescript
  async getTransactions(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<PaginatedResourceDto<Transaction>> {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser = [
      { ...where, signers: { userId: user.id } },
      {
        ...where,
        observers: {
          userId: user.id,
        },
      },
      {
        ...where,
        creatorKey: {
          userId: user.id,
        },
      },
    ];

    const findOptions: FindManyOptions<Transaction> = {
      where: whereForUser,
      order,
      relations: ['creatorKey', 'groupItem', 'groupItem.group'],
      skip: offset,
      take: limit,
    };

    const whereBrackets = new Brackets(qb =>
      qb.where(where).andWhere(
        `
        (
          with recursive "approverList" as
            (
              select * from "transaction_approver"
              where "transaction_approver"."transactionId" = "Transaction"."id"
                union all
                  select "approver".* from "transaction_approver" as "approver"
                  join "approverList" on "approverList"."id" = "approver"."listId"
            )
          select count(*) from "approverList"
          where "approverList"."deletedAt" is null and "approverList"."userId" = :userId
        ) > 0
        `,
        {
          userId: user.id,
        },
      ),
    );

    const [transactions, total] = await this.repo
      .createQueryBuilder()
      .setFindOptions(findOptions)
      .orWhere(whereBrackets)
      .getManyAndCount();

    return {
      totalItems: total,
      items: transactions,
      page,
      size,
    };
  }
```
