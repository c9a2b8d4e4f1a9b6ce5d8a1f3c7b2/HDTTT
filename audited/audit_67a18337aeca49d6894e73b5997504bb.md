### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Malicious Signer to Permanently Fail Transactions

### Summary
The `processTransactionSignatures` method in `signers.service.ts` accepts and persists signature bytes from an authenticated user without verifying their cryptographic validity against the transaction body. A malicious user whose key is part of a transaction's required signing set can submit invalid signature bytes, satisfy the signing threshold, and cause the transaction to be submitted to the Hedera network with a bad signature — permanently failing it. The parallel `importSignatures` path in `transactions.service.ts` does call `validateSignature`, making this an inconsistency between two code paths handling the same operation.

### Finding Description

**Root Cause**

`processTransactionSignatures` in `signers.service.ts` iterates over the submitted `SignatureMap`, checks only that each public key belongs to the authenticated user, then unconditionally calls `sdkTransaction.addSignature(publicKey, map)`: [1](#0-0) 

There is no call to `publicKey.verify(transactionBytes, signatureBytes)` or any equivalent cryptographic check. The Hedera SDK's `addSignature` appends whatever bytes are in the map to the transaction without validating them.

The `IsSignatureMap` decorator that parses the incoming request body only checks that the decoded signature is non-empty: [2](#0-1) 

This is purely a structural check — it does not verify the signature is cryptographically correct for the claimed public key and transaction body.

**Contrast with `importSignatures`**

The `importSignatures` path in `transactions.service.ts` does call `validateSignature` before adding signatures: [3](#0-2) 

The primary signer upload path (`POST /transactions/:transactionId/signers` → `uploadSignatureMaps`) has no equivalent check.

**Exploit Flow**

1. Attacker is an authenticated user with a key registered in the system that is part of a transaction's required signing key list (e.g., a 2-of-3 threshold key).
2. One legitimate signer has already signed. The threshold requires one more signature.
3. Attacker calls `POST /transactions/:transactionId/signers` with a `signatureMap` containing their valid public key but random/invalid signature bytes.
4. `processTransactionSignatures` finds the public key in `userKeyMap` (ownership check passes), calls `sdkTransaction.addSignature(publicKey, map)` with the invalid bytes, and returns `isSameBytes = false` (bytes changed) and `userKeys = [attackerKey]`.
5. `persistSignatureChanges` updates `transactionBytes` in the database with the corrupted signature and inserts the attacker as a `TransactionSigner`.
6. `processTransactionStatus` sees the threshold is now met and advances the transaction to `WAITING_FOR_EXECUTION`.
7. The chain service submits the transaction to Hedera. Hedera returns `INVALID_SIGNATURE`. The transaction is permanently marked `FAILED`. [4](#0-3) [5](#0-4) 

### Impact Explanation

A malicious required signer can permanently fail any transaction they are part of by submitting invalid signature bytes. Unlike simply refusing to sign (which leaves the transaction open until expiry), this attack:
- Immediately corrupts the stored `transactionBytes` with an invalid signature
- Satisfies the signing threshold, locking out other legitimate signers
- Forces the chain service to submit a doomed transaction to Hedera
- Results in a permanent `FAILED` state with no recovery path

For high-value organizational transactions (system file updates, account key rotations, large transfers), this is an unrecoverable integrity failure.

### Likelihood Explanation

The attacker must be an authenticated organization member with a key registered in the system that appears in the transaction's key list. This is a normal user role — no admin or privileged access is required. Any member who is designated as a required signer for a transaction can execute this attack. The API endpoint is reachable with a standard JWT token and a crafted JSON body.

### Recommendation

Add cryptographic signature verification inside `processTransactionSignatures` before calling `addSignature`. For each `(publicKey, signatureBytes)` pair in the map, call `publicKey.verify(transactionBodyBytes, signatureBytes)` and reject the entire submission if any signature fails verification. This mirrors the protection already present in the `importSignatures` path via `validateSignature`. [6](#0-5)

```
POST /transactions/42/signers

  "id": 42,
  "signatureMap": {
    "0.0.3": {
      "0.0.1001@1700000000.000000000": {
        "302a300506032b6570032100<attacker_pubkey_hex>": "deadbeefdeadbeefdeadbeefdeadbeef..."
      }
    }
  }
}]
```

**Expected (current) behavior**: Server returns HTTP 201, inserts attacker as `TransactionSigner`, updates `transactionBytes` with the invalid signature, advances transaction to `WAITING_FOR_EXECUTION`, chain service submits to Hedera → `INVALID_SIGNATURE` → transaction permanently `FAILED`.

**Expected (fixed) behavior**: Server verifies `publicKey.verify(txBodyBytes, deadbeef...)` returns `false`, rejects the request with a 400 error, transaction state is unchanged.

```typescript
  private async processTransactionSignatures(

    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
```
