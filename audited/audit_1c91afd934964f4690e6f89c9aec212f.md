### Title
Unbounded Full-Table Scan with Per-Row Async Work in `getTransactionsToSign` Causes Server Resource Exhaustion

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction in the database without any row limit, then performs a sequential async DB call (`userKeysToSign`) for each row before applying pagination. Any authenticated, verified user can trigger this endpoint (`GET /transactions/sign`). As the number of active transactions in the organization grows, each request consumes unbounded CPU and database connections, degrading or hanging the API service.

### Finding Description

**Root cause — no `take` on the initial query:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / no limit
});
```

The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope to the requesting user's own transactions — it returns every active transaction in the entire organization.

**Unbounded sequential async loop:** [2](#0-1) 

For each of the N returned rows, `userKeysToSign` is awaited sequentially. Internally this calls `keysRequiredToSign`, which:
- Deserializes the Hedera SDK transaction from raw bytes (CPU)
- Calls `transactionSignatureService.computeSignatureKey(transaction)` — an async DB query per transaction [3](#0-2) 

**Pagination applied only after full scan:** [4](#0-3) 

```typescript
items: result.slice(offset, offset + limit),
```

The caller receives one page of results, but the server has already processed every active transaction.

**Exposed endpoint — no rate limiting visible:** [5](#0-4) 

The route `GET /transactions/sign` is protected only by JWT + `VerifiedUserGuard`. No rate-limit decorator or middleware is applied at the controller level. The comment `/* NO LONGER USED BY FRONT-END */` confirms the endpoint is still live and reachable.

### Impact Explanation

Each call to `GET /transactions/sign` causes the server to:
1. Load all N active transactions from PostgreSQL (full table scan on non-terminal rows)
2. Execute N sequential async operations (SDK deserialization + DB query per transaction)

With a large enough active-transaction set, a single request can exhaust the Node.js event loop and the PostgreSQL connection pool, causing timeouts for all concurrent users. Because the endpoint is still publicly reachable and requires only a valid JWT, a single malicious authenticated user can trigger this repeatedly to degrade or hang the API service.

### Likelihood Explanation

- **Attacker precondition:** A valid, verified account in the organization — the lowest privilege level the application grants.
- **Attack vector:** A single HTTP GET request to `GET /transactions/sign`. No special tooling required.
- **Amplification:** The attacker can first create many transactions via `POST /transactions` (no creation limit is enforced), then repeatedly call the sign endpoint. Alternatively, in a busy organization with many legitimate active transactions, the endpoint is already slow without any attacker-created data.
- **No rate limiting** is present on this endpoint.

### Recommendation

1. **Apply `take` at the database level** before the loop — push the filtering logic into SQL rather than application memory:
   ```typescript
   const transactions = await this.repo.find({
     where: whereForUser,
     relations: ['groupItem'],
     order,
     take: limit,
     skip: offset,
   });
   ```
   This requires moving the `userKeysToSign` check into the query (e.g., a subquery or a pre-filter on public keys), or accepting that `totalItems` is an estimate.

2. **Alternatively, cap the scan** with a hard maximum (e.g., 200 rows) and document the behavior.

3. **Add rate limiting** to this endpoint (e.g., via the existing Redis/`MurLock` infrastructure already present in the stack).

4. **Consider removing the endpoint** entirely if it is confirmed to be unused by the front-end, as noted in the controller comment.

### Proof of Concept

**Preconditions:** Attacker has a verified account (`userAuthToken`).

**Steps:**

1. Create a large number of active transactions:
   ```bash
   for i in $(seq 1 5000); do
     curl -X POST https://<api>/transactions \
       -H "Authorization: Bearer $userAuthToken" \
       -H "Content-Type: application/json" \
       -d '{"name":"tx","type":"TRANSFER","transactionBytes":"...","creatorKeyId":1,...}'
   done
   ```

2. Repeatedly call the unbounded endpoint:
   ```bash
   while true; do
     curl -s "https://<api>/transactions/sign?page=1&size=10" \
       -H "Authorization: Bearer $userAuthToken"
   done
   ```

**Expected outcome:** Each request causes the server to load all 5,000+ transactions and call `userKeysToSign` 5,000+ times sequentially. API response times degrade for all users; under sustained load the Node.js process becomes unresponsive and the PostgreSQL connection pool is exhausted.
