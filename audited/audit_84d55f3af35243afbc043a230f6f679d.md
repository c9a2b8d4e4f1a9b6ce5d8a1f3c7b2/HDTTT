### Title
Unbounded Sequential Per-Row DB Queries in `getTransactionsToSign` Enable Authenticated DoS

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches every non-terminal transaction in the database with no DB-level row limit, then iterates over the full result set making one sequential async database call per row. Any authenticated user can trigger this via `GET /transactions/sign`, causing O(N) sequential DB round-trips where N is the total count of active transactions. As the platform scales, each request consumes proportionally more server CPU, DB connections, and wall-clock time, eventually causing request timeouts and degrading service for all users.

### Finding Description

**Root cause — no `take` on the initial query:**

`this.repo.find({where: whereForUser, relations: ['groupItem'], order})` is called with no `take` or `skip` option, so TypeORM issues a `SELECT … FROM transaction` with no `LIMIT` clause, returning every active transaction in the system. [1](#0-0) 

**Root cause — sequential async call per row:**

The result set is then iterated with a `for…of` loop that `await`s `this.userKeysToSign(transaction, user)` on every iteration. This is not parallelised; each iteration blocks on the previous one. [2](#0-1) 

**What `userKeysToSign` does per iteration:**

Each call resolves to `userKeysRequiredToSign`, which calls `attachKeys` (a DB query to load the user's keys), then `keysRequiredToSign`, which calls `transactionSignatureService.computeSignatureKey` (an async mirror-node or DB operation) and potentially `entityManager.find(UserKey, …)` (another DB query). So each loop iteration can issue 2–3 DB round-trips. [3](#0-2) 

**Pagination is applied only after all work is done:**

The `PaginationParams` decorator enforces `size ≤ 100`, but that only limits the slice returned to the caller. The full O(N) computation always runs first. [4](#0-3) [5](#0-4) 

**The endpoint is live and requires only a valid JWT:**

The controller exposes `GET /transactions/sign` protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin role, no special permission. The comment "NO LONGER USED BY FRONT-END" confirms the endpoint is still registered and reachable. [6](#0-5) 

**No rate limiting on this route:**

Throttle guards exist in the codebase but are applied only to authentication endpoints (login, email flows). The `/transactions/sign` route has no throttle decorator. [7](#0-6) 

### Impact Explanation

With N active transactions in the database, a single `GET /transactions/sign?page=1&size=1` request forces the server to:
1. Load all N transaction rows from PostgreSQL (one large query).
2. Execute up to 3 × N sequential async DB/network calls inside the loop.

At moderate scale (e.g., 1 000 active transactions), each request holds a DB connection pool slot and CPU for the entire duration. An attacker sending a modest stream of such requests can saturate the connection pool, starve legitimate requests of DB connections, and cause HTTP timeouts across the entire API — affecting all users and all endpoints that share the same NestJS process and TypeORM pool.

### Likelihood Explanation

**Preconditions**: A valid, verified user account — obtainable by any registered user of the platform.  
**Trigger**: A single HTTP GET request with no special parameters.  
**Attacker profile**: Malicious normal user abusing a valid product flow (RESEARCHER.md profile 2).  
**Scaling**: The impact grows automatically as the platform accumulates transactions; no attacker action is needed to worsen it over time.

### Recommendation

Apply DB-level pagination before the loop, or rewrite the query to push the key-matching logic into SQL so the loop is eliminated entirely:

1. **Short-term**: Add `take: limit, skip: offset` to the `this.repo.find(…)` call so the DB returns at most `size` rows. Accept that `totalItems` will be approximate or require a separate `COUNT` query.
2. **Long-term**: Rewrite `getTransactionsToSign` to use a single SQL query (similar to how `getTransactionsToApprove` uses a `WITH RECURSIVE` subquery) that filters transactions by whether the user's public keys appear in the required signers, eliminating the per-row application-layer loop entirely.
3. **Immediate mitigation**: Apply the existing `UserThrottlerGuard` to the `/transactions/sign` route to limit request rate per user.

### Proof of Concept

**Setup**: An organization with 500+ active (non-terminal) transactions in the database.

**Attacker steps**:
1. Register and verify a normal user account.
2. Obtain a JWT via `POST /auth/login`.
3. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <jwt>
   ```

**Expected outcome**: Each request causes the server to load all 500+ transaction rows and execute up to 1 500+ sequential async DB calls before returning. Response latency grows linearly with transaction count. Concurrent requests from the same or different users exhaust the TypeORM connection pool, causing `QueryFailedError: remaining connection slots are reserved` or HTTP 504 timeouts for all API consumers.

**Verification**: Instrument `getTransactionsToSign` with a counter; observe that `transactions.length` equals the total active transaction count regardless of the `size` query parameter. [8](#0-7)

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
```
