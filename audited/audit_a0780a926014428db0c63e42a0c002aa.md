### Title
Missing User-Scoped Authorization on `GET /transactions/history` Allows Any Authenticated User to Enumerate All Organization Transactions

### Summary
The `getHistoryTransactions` endpoint in the Hedera Transaction Tool API returns all historical transactions (executed, failed, expired, canceled, archived) without filtering by the requesting user. Any authenticated user — regardless of their role or relationship to those transactions — can enumerate the full transaction history of the entire organization. This is compounded by `verifyAccess` unconditionally returning `true` for all terminal-status transactions, meaning `GET /transactions/:id` also exposes full transaction details to any authenticated user.

### Finding Description

**Root Cause 1 — Missing user filter in `getHistoryTransactions`:**

In `back-end/apps/api/src/transactions/transactions.controller.ts`, the `/history` route does not pass the authenticated user to the service: [1](#0-0) 

The `@GetUser()` decorator is absent. The service method `getHistoryTransactions` in `back-end/apps/api/src/transactions/transactions.service.ts` performs no user-scoped filtering whatsoever: [2](#0-1) 

Every transaction in a terminal status is returned to every caller, regardless of whether the caller created, signed, observed, or approved those transactions.

**Root Cause 2 — `verifyAccess` unconditionally grants access to terminal-status transactions:** [3](#0-2) 

`verifyAccess` returns `true` for any user when the transaction status is `EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, or `ARCHIVED`. This is used by `getTransactionWithVerifiedAccess`, which backs the `GET /transactions/:id` endpoint: [4](#0-3) 

**Contrast with non-history endpoints:** The standard `getTransactions` endpoint correctly scopes results to the requesting user (creator, signer, observer, or approver): [5](#0-4) 

The history endpoint omits this scoping entirely.

**Exploit Path:**
1. Attacker registers as a normal user in the organization (no privileged access required).
2. Attacker calls `GET /transactions/history` (authenticated, no other preconditions).
3. The API returns all historical transactions from all users — including transaction bytes, transaction IDs, mirror network, valid start times, and creator key references.
4. Attacker calls `GET /transactions/:id` for any returned ID to retrieve full transaction details including signer lists, observer lists, and approver trees.

### Impact Explanation
An unprivileged authenticated user can exfiltrate the complete historical transaction record of the entire organization: transaction bytes (which encode the full Hedera transaction payload), all participant identities (signers, observers, approvers), transaction IDs, and network metadata. In an organization managing sensitive Hedera operations (node management, account updates, large HBAR transfers), this constitutes a critical cross-tenant confidentiality breach. Transaction bytes may also reveal key structures used in threshold signing schemes.

### Likelihood Explanation
The attack requires only a valid authenticated session — no admin keys, no leaked credentials, no privileged role. Any user who can register and log in to the organization's API can trigger this. The endpoint is a standard REST `GET` with no additional guards beyond `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`: [6](#0-5) 

The `VerifiedUserGuard` only checks `user.status === UserStatus.NONE`, which is the baseline for any registered user: [7](#0-6) 

### Recommendation
1. **Add user-scoped filtering to `getHistoryTransactions`**: Accept a `User` parameter and apply the same `whereForUser` bracket logic used in `getTransactions` — restricting results to transactions where the user is creator, signer, observer, or approver.
2. **Fix `verifyAccess` for terminal-status transactions**: Remove the blanket `return true` for terminal statuses. Instead, apply the same ownership/role check regardless of status. Historical transactions are not less sensitive than active ones.
3. **Pass `@GetUser()` to the history controller method** and thread it through to the service.

### Proof of Concept
```
# Step 1: Authenticate as any registered user
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "token": "<JWT>" }

# Step 2: Enumerate all organization historical transactions
GET /transactions/history
Authorization: Bearer <JWT>
→ 200 OK: { items: [ ...ALL org transactions... ], totalItems: N }

# Step 3: Retrieve full details of any transaction
GET /transactions/42
Authorization: Bearer <JWT>
→ 200 OK: { transactionBytes: "...", signers: [...], observers: [...], ... }
```

Expected result: full transaction history of all users in the organization is returned. The attacker has no relationship to these transactions.

    { page, limit, size, offset }: Pagination,
    filter: Filtering[] = [],
    sort: Sorting[] = [],
  ): Promise<PaginatedResourceDto<Transaction>> {
    const order = getOrder(sort);

    const findOptions: FindManyOptions<Transaction> = {
      where: {
        ...getWhere<Transaction>(filter),
        status: this.getHistoryStatusWhere(filter),
      },
      order,
      relations: ['groupItem', 'groupItem.group'],
      skip: offset,
      take: limit,
    };

    const [transactions, total] = await this.repo
      .createQueryBuilder()
      .setFindOptions(findOptions)
      .getManyAndCount();

    return {
      totalItems: total,
      items: transactions,
      page,
      size,
    };
  }
