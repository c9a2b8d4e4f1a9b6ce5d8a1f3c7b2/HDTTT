### Title
Unrestricted Access to All User Keys — Any Authenticated User Can Enumerate System-Wide Key Data

### Summary
The `GET /user-keys` endpoint in `UserKeysAllController` applies only basic JWT authentication guards but performs no ownership or role check before returning all `UserKey` records for every user in the system. Any authenticated, verified user can paginate through the complete key inventory of all other users, exposing public keys, mnemonic hashes, and key indices across all accounts.

### Finding Description

**Root cause — missing caller restriction on a sensitive read endpoint.**

The analog to the RSIManager "too open" pattern is `UserKeysAllController.getUserKeys`: [1](#0-0) 

The controller is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — the same baseline guards applied to every authenticated endpoint. There is no `AdminGuard`, no ownership filter, and no user context passed to the service call:

```ts
// user-keys-all.controller.ts line 34-38
getUserKeys(
  @PaginationParams() paginationParams: Pagination,
): Promise<PaginatedResourceDto<UserKey>> {
  return this.userKeysService.getUserKeys(paginationParams);   // no user arg
}
``` [2](#0-1) 

Contrast this with the sibling controller `UserKeysController`, which correctly passes the authenticated user and calls the restricted variant:

```ts
// user-keys.controller.ts line 52-57
getUserKeys(@GetUser() user: User, @Param('userId', ParseIntPipe) userId: number) {
  return this.userKeysService.getUserKeysRestricted(user, userId);
}
``` [3](#0-2) 

`getUserKeys` (the unrestricted variant) accepts only pagination parameters and returns the full cross-user key set. The `AdminGuard` exists in the codebase and is applied to sensitive user-management endpoints (e.g., `PATCH /users/:id`, `DELETE /users/:id`): [4](#0-3) 

But it is absent from `GET /user-keys`.

**Exploit path:**
1. Attacker registers as a normal user and obtains a valid JWT.
2. Attacker sends `GET /user-keys?page=1&limit=100` with their bearer token.
3. The endpoint returns paginated `UserKey` records for all users — public keys, mnemonic hashes, and key indices — with no restriction.
4. Attacker iterates pages to harvest the complete key inventory.

### Impact Explanation

Every `UserKey` record contains the user's public key, the mnemonic hash (a fingerprint of the mnemonic phrase used to derive the key), and the derivation index. An attacker who collects mnemonic hashes can:

- Correlate keys across users and transactions, breaking pseudonymity.
- Use mnemonic hashes in offline dictionary/rainbow-table attacks to recover the original mnemonic phrase, which would allow full key compromise.
- Map the entire organizational key structure, identifying high-value signers for targeted attacks.

This is unauthorized cross-tenant data exposure with a realistic path to full key compromise.

### Likelihood Explanation

**High.** The attacker precondition is only a valid account (any registered, verified user). No elevated privilege, no leaked credential, and no special network position is required. The endpoint is a standard REST `GET` call. The `AdminGuard` is already implemented and used elsewhere in the same codebase, so the omission here is an oversight, not a design choice.

### Recommendation

Apply `AdminGuard` to `GET /user-keys` to restrict it to administrators, or — if the endpoint must be accessible to normal users — pass the authenticated user to the service and filter results to only that user's keys (matching the pattern in `UserKeysController`):

```ts
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard, AdminGuard)
@Get()
getUserKeys(@PaginationParams() paginationParams: Pagination): Promise<...> {
  return this.userKeysService.getUserKeys(paginationParams);
}
```

Or, for a non-admin scoped version:

```ts
@Get()
getUserKeys(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
): Promise<...> {
  return this.userKeysService.getUserKeysRestricted(user, user.id, paginationParams);
}
```

### Proof of Concept

**Preconditions:** Two accounts exist — `victim` and `attacker`. Both are registered and verified.

**Steps:**
1. Authenticate as `attacker` → receive JWT `attacker_token`.
2. Send:
   ```
   GET /user-keys?page=1&limit=100
   Authorization: Bearer attacker_token
   ```
3. **Expected (secure) behavior:** 403 Forbidden or only attacker's own keys returned.
4. **Actual behavior:** HTTP 200 with paginated `UserKey` records for all users, including `victim`'s public key, mnemonic hash, and key index. [1](#0-0)
