### Title
Soft-Deleted User Can Bypass Admin Removal by Re-Registering with the Same Email

### Summary
When an admin removes a user from the organization, the removal is implemented as a soft-delete (`deletedAt` timestamp). However, the `createUser` function in `users.service.ts` explicitly detects a soft-deleted record and re-activates it upon re-registration with the same email. This allows any removed user to immediately re-join the organization without admin approval, completely circumventing the admin's removal action.

### Finding Description
The `removeUser` function performs a soft-delete on the user record: [1](#0-0) 

The `createUser` function fetches users including soft-deleted ones (`withDeleted = true`) and contains an explicit branch that re-activates a soft-deleted user: [2](#0-1) 

Specifically, lines 36–39 check: if the user exists **and** `deletedAt` is set, re-activate by calling `updateUser` with `{ ..., status: UserStatus.NEW, deletedAt: null }`. This resets the soft-delete and restores the user to an active `NEW` state.

The `removeUser` endpoint is admin-only: [3](#0-2) 

But the registration path that calls `createUser` is in the auth service and is publicly accessible (no authentication required for self-registration). A removed user simply re-submits a registration request with their original email to be re-activated.

**Exploit path:**
1. Admin calls `DELETE /users/:id` → user record is soft-deleted (`deletedAt` set, keys soft-deleted).
2. Removed user calls the public registration endpoint (e.g., `POST /auth/register`) with the same email and any password.
3. `createUser` finds the soft-deleted record, sees `deletedAt !== null`, and calls `updateUser(user, { email, password, status: UserStatus.NEW, deletedAt: null })`.
4. The user is fully re-activated in the organization with a fresh password they control.

### Impact Explanation
An admin-removed user can immediately re-join the organization without any admin approval. This constitutes an **unauthorized privilege escalation / authorization bypass**: the admin's explicit removal decision is rendered ineffective. The re-activated user regains access to all organization resources, transaction workflows, and multi-signature operations they were removed from. If the user was removed due to a security incident (e.g., compromised account, insider threat), this bypass directly undermines the incident response.

### Likelihood Explanation
Likelihood is **high**. The attacker (removed user) requires no privileged access — only knowledge of their own email address, which they already possess. The registration endpoint is public. The re-activation is immediate and requires a single HTTP request. There is no rate-limiting or admin notification described in the code path.

### Recommendation
In `createUser`, do not re-activate a soft-deleted user. Instead, treat a soft-deleted record the same as a permanently removed one and throw an error (e.g., `UnprocessableEntityException('Email already exists.')`) or require explicit admin re-invitation. The check at line 37 should be removed or inverted:

```typescript
// Current (vulnerable):
if (!user.deletedAt) throw new UnprocessableEntityException('Email already exists.');
return this.updateUser(user, { email, password, status: UserStatus.NEW, deletedAt: null });

// Fixed:
throw new UnprocessableEntityException('Email already exists.');
```

If re-registration of previously removed users is a desired feature, it must require explicit admin approval rather than being self-service.

### Proof of Concept
1. Admin authenticates and calls `DELETE /users/42` — user with email `removed@org.com` is soft-deleted.
2. Attacker (the removed user) sends:
   ```
   POST /auth/register
   { "email": "removed@org.com", "password": "NewPassword123!" }
   ```
3. `UsersService.createUser` executes:
   - `getUser({ email: 'removed@org.com' }, true)` → returns the soft-deleted record.
   - `user.deletedAt` is non-null → branch at line 38 executes.
   - `updateUser(user, { email, password, status: UserStatus.NEW, deletedAt: null })` → user is fully restored.
4. Attacker logs in with `removed@org.com` / `NewPassword123!` and has active organization membership. [4](#0-3)

  async createUser(email: string, password: string): Promise<User> {
    let user = await this.getUser({ email }, true);
    password = await this.getSaltedHash(password);

    if (user) {
      if (!user.deletedAt) throw new UnprocessableEntityException('Email already exists.');
      return this.updateUser(user, { email, password, status: UserStatus.NEW, deletedAt: null });
    }

    user = this.repo.create({
      email,
      password,
    });

    return await this.repo.save(user);
  }
