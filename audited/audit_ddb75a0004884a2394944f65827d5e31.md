### Title
Mutually Exclusive `atomic` and `sequential` Transaction Group Modes Can Be Activated Simultaneously, Silently Dropping the Atomic Execution Guarantee

### Summary
`TransactionGroup` exposes two operationally exclusive boolean flags — `atomic` and `sequential` — that govern how a group of transactions is executed. The `CreateTransactionGroupDto` applies no mutual-exclusivity validation, so a caller can submit `{ atomic: true, sequential: true }` simultaneously. In the execution path, only `sequential` is checked; `atomic` is silently ignored. Organization members who sign a group believing it carries an atomic (all-or-nothing) guarantee are misled: partial execution can occur.

### Finding Description

**Root cause — no mutual-exclusivity guard in the DTO:**

`CreateTransactionGroupDto` declares both flags as independent, optional booleans with no cross-field constraint: [1](#0-0) 

The entity mirrors this with two independent columns, both defaulting to `false`: [2](#0-1) 

**Scheduler treats both flags as equivalent triggers:**

`prepareTransactions` routes any group where `atomic || sequential` is truthy into `collateGroupAndExecute`, so a group with both flags set is processed identically to one with only one flag set: [3](#0-2) 

**Execution service checks only `sequential`, silently dropping `atomic`:**

Inside `executeTransactionGroup`, the branching logic reads only `transactionGroup.sequential`. The `atomic` flag is never consulted: [4](#0-3) 

This means:
- `{ atomic: true, sequential: false }` → parallel execution (no atomicity enforced, no rollback)
- `{ atomic: true, sequential: true }` → sequential execution (atomic guarantee silently dropped, `sequential` wins)
- `{ atomic: false, sequential: true }` → sequential execution (correct)

The front-end store's own comment confirms `atomic` semantics are unresolved: [5](#0-4) 

**Exploit path:**

1. Authenticated organization member calls `POST /transaction-groups` with `{ atomic: true, sequential: true, groupItems: [...] }`.
2. The API accepts the payload — no validation rejects the combination.
3. The group is persisted with both flags set to `true`.
4. Other organization members query the group, observe `atomic: true`, and sign under the assumption of all-or-nothing execution.
5. At execution time, `executeTransactionGroup` branches on `sequential: true` and executes transactions one-by-one with no rollback. If transaction N fails, transactions 0…N-1 have already been irreversibly submitted to Hedera.
6. The atomic guarantee the signers relied on was never enforced.

### Impact Explanation
Signers in an organization co-sign transactions based on the declared execution semantics of the group. If they believe the group is atomic, they may sign transactions they would otherwise refuse to sign individually (e.g., a multi-step asset transfer where partial execution is harmful). Because `atomic` is silently overridden by `sequential`, partial execution can occur on the Hedera network — an irreversible ledger — violating the signers' intent and potentially causing unrecoverable asset movement.

### Likelihood Explanation
Any authenticated organization member can create a transaction group via the standard API endpoint. No elevated privileges are required. The malicious payload (`atomic: true, sequential: true`) is a trivially crafted JSON body. Other signers have no mechanism to detect the flag conflict because the API and front-end both expose `atomic` and `sequential` as independent readable fields without warning.

### Recommendation
Enforce mutual exclusivity at the DTO validation layer using a class-validator `@ValidateIf` / custom decorator, or replace the two booleans with a single enum:

```typescript
// Option A: Enum (preferred)
export enum TransactionGroupMode {
  NONE = 'NONE',
  ATOMIC = 'ATOMIC',
  SEQUENTIAL = 'SEQUENTIAL',
}

// Option B: Cross-field guard in the DTO
@ValidateIf(o => o.sequential === true)
@Equals(false, { message: 'atomic and sequential are mutually exclusive' })
atomic: boolean;
```

Additionally, `executeTransactionGroup` should explicitly check and enforce the `atomic` flag's semantics (all-or-nothing with rollback/cancellation on partial failure), or remove the flag entirely until its semantics are defined.

### Proof of Concept

```http
POST /transaction-groups HTTP/1.1
Authorization: Bearer <valid_org_member_token>
Content-Type: application/json

{
  "description": "Misleading atomic group",
  "atomic": true,
  "sequential": true,
  "groupItems": [
    { "seq": 0, "transaction": { ... } },
    { "seq": 1, "transaction": { ... } }
  ]
}
```

**Expected (correct) behavior:** Server rejects the request with a validation error — `atomic` and `sequential` cannot both be `true`.

**Actual behavior:** Server accepts the payload. The group is stored with `atomic=true, sequential=true`. At execution time, `executeTransactionGroup` branches on `sequential=true` only, executes transactions in order with no rollback. If transaction at seq=1 fails, seq=0 has already been submitted to Hedera irreversibly. The `atomic` guarantee is never enforced.
