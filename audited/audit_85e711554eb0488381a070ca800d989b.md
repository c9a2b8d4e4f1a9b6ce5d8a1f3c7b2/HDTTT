### Title
Email Case Normalization Missing in Backend Causes Permanent Account Lockout and Duplicate User Registration

### Summary
The backend API stores email addresses exactly as supplied by the admin during user registration, with no case normalization. Because PostgreSQL's `UNIQUE` constraint is case-sensitive, `User@Test.com` and `user@test.com` are treated as distinct identities. The client-side login flow unconditionally lowercases the email before sending it to the server. If an admin registers a user with any uppercase characters in the email, the resulting account is permanently inaccessible: the user's login attempt (lowercased by the client) will never match the stored mixed-case email.

### Finding Description

**Root cause — no normalization at registration:**

`SignUpUserDto` applies only `@IsEmail()` (format check), with no `@Transform` to lowercase: [1](#0-0) 

`AuthService.signUpByAdmin()` passes `dto.email` verbatim into `usersService.createUser()`: [2](#0-1) 

`UsersService.createUser()` stores the email as-is and the duplicate check (`getUser({ email })`) is also case-sensitive, so `User@Test.com` passes the uniqueness check when `user@test.com` already exists: [3](#0-2) 

The `User` entity declares `@Column({ unique: true })` on `email`, but PostgreSQL's default unique index is case-sensitive: [4](#0-3) 

**Root cause — client always lowercases on login:**

`OrganizationLogin.vue` calls `.toLocaleLowerCase().trim()` before sending credentials: [5](#0-4) 

So a user registered as `User@Test.com` will always attempt to authenticate as `user@test.com`. The backend lookup `getUser({ email: 'user@test.com' })` returns `null` because the stored value is `User@Test.com`. Login fails unconditionally.

**Duplicate account path:**

Because the uniqueness check is case-sensitive, an admin can register both `user@test.com` and `User@test.com` as separate accounts. Both receive invitation emails. Only the all-lowercase account is ever reachable via the client.

### Impact Explanation

- **Permanent account lockout:** Any user registered with a mixed-case email address can never log in. The account exists in the database but is unreachable. There is no self-service recovery path because the OTP/password-reset flow (`createOtp`) also performs a case-sensitive lookup: [6](#0-5) 
- **Duplicate identity:** Two accounts for the same real-world email address can coexist, each with independent keys, transaction signing rights, and notification preferences, violating the one-identity-per-email invariant the system relies on.

### Likelihood Explanation

The trigger is a normal admin workflow (registering a new organization user). Any admin who pastes an email address from an email client, a spreadsheet, or a directory that auto-capitalizes the first character will silently create an inaccessible account. No malicious intent is required. The `@IsEmail()` validator accepts mixed-case local parts, so no error is surfaced. [7](#0-6) 

### Recommendation

Normalize the email to lowercase at the earliest server-side entry point. Apply a `@Transform` decorator in `SignUpUserDto` and in `LoginDto`, and also normalize inside `UsersService.createUser()` and `UsersService.getVerifiedUser()` as a defense-in-depth measure:

```typescript
// sign-up-user.dto.ts
import { Transform } from 'class-transformer';
import { IsEmail } from 'class-validator';

export class SignUpUserDto {
  @IsEmail()
  @Transform(({ value }) => value?.toLowerCase().trim())
  email: string;
}
```

Apply the same `@Transform` to `LoginDto`. Additionally, consider adding a case-insensitive unique index at the database level (`LOWER(email)`) to enforce the invariant independently of application logic.

### Proof of Concept

1. Admin sends `POST /auth/signup` with body `{ "email": "Alice@Example.com" }` using a valid admin JWT.
2. Server stores the user with `email = 'Alice@Example.com'` (HTTP 201 returned).
3. The invited user opens the desktop client, enters `alice@example.com` (or any lowercase variant) and the temporary password.
4. `OrganizationLogin.vue` sends `email = 'alice@example.com'` to `POST /auth/login`.
5. `UsersService.getVerifiedUser('alice@example.com', ...)` calls `getUser({ email: 'alice@example.com' })`.
6. PostgreSQL returns no row (stored value is `Alice@Example.com`).
7. Server throws `UnauthorizedException('Please check your login credentials')`.
8. The user is permanently locked out. No password reset is possible because `createOtp('alice@example.com')` also returns `null` and silently exits. [8](#0-7)
