### Title
Missing Terminal-State Guard in `archiveTransaction` Allows Creator to Overwrite Executed Transaction Status

### Summary
The `archiveTransaction` function in `transactions.service.ts` contains a flawed state-transition guard that is entirely bypassed for manual transactions (`isManual = true`). A transaction creator can call archive on a manual transaction already in a terminal state (`EXECUTED`, `FAILED`, `CANCELED`, `REJECTED`, `EXPIRED`, or even `ARCHIVED`), overwriting the terminal status with `ARCHIVED`. This corrupts the authoritative status record that other participants (observers, signers) rely on to confirm Hedera network outcomes.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the guard inside `archiveTransaction` is:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The condition throws only when **both** sub-expressions are true. For any manual transaction (`isManual = true`), `!transaction.isManual` evaluates to `false`, making the entire `&&` expression `false` regardless of the current status. The guard is therefore completely inert for manual transactions.

Immediately after the guard, the function unconditionally writes:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

This overwrites any existing terminal status — including `EXECUTED` — with `ARCHIVED`.

The `TransactionStatus` enum defines the full set of terminal states that should be immutable once reached: [3](#0-2) 

The `cancelTransactionWithOutcome` function, by contrast, correctly enforces state guards using an explicit allowlist and a conditional-update query that atomically checks the current status: [4](#0-3) 

`archiveTransaction` applies no equivalent protection for manual transactions.

### Impact Explanation

A transaction creator can change the recorded status of their own manual transaction from `EXECUTED` (or any other terminal state) to `ARCHIVED` at any time after execution. Because observers and signers rely on the application's status field to confirm that a transaction was submitted to the Hedera network, overwriting `EXECUTED` with `ARCHIVED` misleads every other participant about the true outcome. The corruption is persistent in the database and is not self-correcting. The integrity of the audit trail — the primary trust artifact of the tool — is violated.

### Likelihood Explanation

The preconditions are minimal: the attacker must be the creator of a manual transaction that has already reached a terminal state. No privileged role, no leaked credentials, and no special network access are required. The operation is reachable through the standard authenticated API endpoint. Any user who creates manual transactions can trigger this at will.

### Recommendation

Remove the `!transaction.isManual` short-circuit from the guard. The allowed-status check should apply unconditionally to all transactions, manual or not:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Additionally, use a conditional-update query (as done in `cancelTransactionWithOutcome`) so the status check and the write are atomic, preventing a TOCTOU race between the read and the update.

### Proof of Concept

1. Authenticate as a normal user and create a manual transaction (`isManual = true`).
2. Allow or force the transaction to reach `EXECUTED` status (e.g., via the chain service or direct DB update in a test environment).
3. Call `PATCH /transactions/archive/:id` with the creator's auth token.
4. Observe that the guard does not throw and the DB row now shows `status = ARCHIVED`.
5. Log in as an observer or signer of the same transaction and confirm the status is displayed as `ARCHIVED` rather than `EXECUTED`, hiding the fact that the transaction was submitted to Hedera. [1](#0-0)

  /* Archive the transaction if the transaction is sign only. */
  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
