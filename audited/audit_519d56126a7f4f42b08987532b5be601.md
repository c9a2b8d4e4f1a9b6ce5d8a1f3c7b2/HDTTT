### Title
Hard-coded Rate-Limit Values in `UserThrottlerModule` While Sibling Throttlers Are Configurable

### Summary
`back-end/apps/api/src/throttlers/user-throttler.module.ts` embeds the per-authenticated-user rate-limit thresholds (`100` per minute, `10` per second) as bare numeric literals directly in source code. The sibling `IpThrottlerModule` and `EmailThrottlerModule` read their equivalent limits from environment variables via `ConfigService`. The magic numbers carry no explanatory name, no comment, and cannot be tuned without a code change and full redeployment.

### Finding Description
In `user-throttler.module.ts` lines 17–18, the throttler limits are hard-coded inline:

```typescript
{ name: 'user-minute', ttl: seconds(60), limit: 100 },
{ name: 'user-second', ttl: seconds(1),  limit: 10  },
```

In contrast, `ip-throttler.module.ts` lines 17–20 and `email-throttler.module.ts` lines 16–21 read their limits from environment variables:

```typescript
limit: configService.getOrThrow<number>('GLOBAL_MINUTE_LIMIT'),
limit: configService.getOrThrow<number>('GLOBAL_SECOND_LIMIT'),
```

The inconsistency is structural: two of the three throttler modules treat their limits as operator-configurable policy; the third silently bakes them in. There is no named constant, no comment explaining the basis for `100` or `10`, and no environment variable that an operator can adjust at deploy time.

Additionally, `front-end/src/main/utils/crypto.ts` lines 6–7 hard-code the PBKDF2 iteration count and key length used to derive the AES-256-GCM encryption key for local data:

```typescript
const iterations = 2560;
const keyLength = 32;
```

No comment explains why `2560` was chosen. OWASP currently recommends ≥ 210,000 iterations for PBKDF2-SHA512; 2560 is roughly 82× below that floor, making offline brute-force of any captured encrypted blob orders of magnitude cheaper.

### Impact Explanation
**Rate-limit path:** Because `UserThrottlerModule` limits cannot be changed without a code push, an operator responding to an active credential-stuffing or API-abuse incident cannot tighten the per-user window without a full release cycle. Conversely, if `100/min` is too restrictive for a legitimate workload, there is no knob to loosen it. The hard-coded values also create a silent divergence from the documented pattern used by the other two throttlers, making it easy for a future maintainer to overlook that these limits exist at all.

**Crypto path:** Any attacker who obtains the local encrypted SQLite/file data (e.g., via physical access, malware, or a backup leak) can mount an offline dictionary attack against the PBKDF2-derived key at a rate that is orders of magnitude higher than if a standard iteration count were used, potentially recovering private key material.

### Likelihood Explanation
The `UserThrottlerModule` inconsistency is already present in the shipped code and requires no attacker action to be in effect — it is a standing misconfiguration. The `crypto.ts` iteration count is equally static; any user whose device is compromised is immediately exposed to the accelerated brute-force risk.

### Recommendation
- In `user-throttler.module.ts`, replace the inline literals with `configService.getOrThrow<number>('USER_MINUTE_LIMIT')` and `configService.getOrThrow<number>('USER_SECOND_LIMIT')`, mirroring the pattern already used in `IpThrottlerModule` and `EmailThrottlerModule`. Add the new keys to the Joi validation schema in `api.module.ts`.
- In `crypto.ts`, replace `2560` with a named constant (e.g., `PBKDF2_ITERATIONS`) set to a value meeting current OWASP guidance (≥ 210,000 for PBKDF2-SHA512), and add a comment citing the standard. Provide a migration path for data encrypted under the old count.
- For `auth.service.ts` TOTP options (`digits: 8`, `step: 60`, `window: 20`), extract each to a named constant with a comment explaining the chosen value.
