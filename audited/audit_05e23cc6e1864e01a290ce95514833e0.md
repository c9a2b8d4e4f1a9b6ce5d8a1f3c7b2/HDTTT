### Title
TOCTOU Race Condition in `createTransactions` Allows Duplicate Transaction Records for the Same Hedera Transaction ID

### Summary
`createTransactions` in `back-end/apps/api/src/transactions/transactions.service.ts` performs a uniqueness check for `transactionId` outside any database-level lock or atomic operation, then saves the record in a separate step. Because the `transaction` table has no `UNIQUE` constraint on `transactionId`, two concurrent requests carrying the same Hedera transaction ID both pass the check and both succeed in writing to the database, producing duplicate rows. This is the direct TypeScript/NestJS analog of the Hermez check-effects-interactions violation described in the external report.

### Finding Description

**Root cause — no atomic guard between check and write.**

`createTransactions` executes in two distinct async phases:

**Phase 1 — uniqueness check (lines 415–427):** [1](#0-0) 

This `repo.find` is a plain `SELECT` with no row-level lock (`SELECT ... FOR UPDATE`) and is executed **outside** the DB transaction that follows.

**Phase 2 — write (lines 436–462):** [2](#0-1) 

The `entityManager.transaction` block wraps only the `INSERT`; it does not re-validate uniqueness inside the same serializable scope.

**No database-level safety net.**

The `Transaction` entity declares `transactionId` as a plain `@Column()` with no `unique: true` option: [3](#0-2) 

The initial migration confirms the column is created without a `UNIQUE` constraint: [4](#0-3) 

There is no migration that adds one later. The `grep` for `UNIQUE.*transactionId` returns only hits in cache-related migrations, not the `transaction` table.

**Exploit path:**

1. Attacker holds a valid JWT (any normal user account — no admin required).
2. Attacker constructs a valid Hedera transaction and signs it with their key.
3. Attacker fires two concurrent `POST /transactions` requests carrying identical `transactionBytes` (same Hedera `transactionId`).
4. Both requests reach `repo.find` before either has committed; both see zero existing rows and pass the guard.
5. Both proceed to `entityManager.save`; both succeed because there is no DB-level uniqueness enforcement.
6. Two rows with the same `transactionId` now exist in `WAITING_FOR_SIGNATURES` status.

### Impact Explanation

- **Duplicate execution risk**: The chain service polls for `WAITING_FOR_EXECUTION` transactions. With two rows for the same Hedera transaction ID, it may attempt to submit the same transaction to the Hedera network twice. The second submission will fail at the network level (Hedera rejects duplicate transaction IDs), but the second row's status will be set to `FAILED`, creating a permanently inconsistent record.
- **Signature-collection split**: Signers who sign one duplicate row do not satisfy the threshold on the other. Neither row may ever accumulate enough signatures to execute, permanently locking the transaction in `WAITING_FOR_SIGNATURES`.
- **State corruption**: Observers, approvers, and notification records attached to one duplicate are invisible to queries against the other, breaking the integrity of the entire transaction lifecycle.
- **Severity**: High — permanent state corruption and potential double-submission to the Hedera network, reachable by any authenticated user.

### Likelihood Explanation

- **Attacker precondition**: A valid JWT token — obtainable by any registered user.
- **Trigger mechanism**: Two simultaneous HTTP POST requests, trivially achievable with `Promise.all`, `curl --parallel`, or any load-testing tool.
- **No special timing required**: Node.js's event loop and async I/O make the window between the `SELECT` and the `INSERT` reliably exploitable under normal server load; no millisecond-precision timing is needed.
- **Likelihood**: High.

### Recommendation

1. **Add a `UNIQUE` constraint on `transactionId` in the `transaction` table** (scoped to active statuses if partial indexes are acceptable, or unconditionally). This is the only reliable defense because it enforces the invariant at the database level regardless of application-layer races.

```sql
CREATE UNIQUE INDEX uq_transaction_transactionId
  ON "transaction" ("transactionId")
  WHERE status NOT IN ('CANCELED', 'REJECTED', 'ARCHIVED');
```

2. **Move the uniqueness check inside the DB transaction** using a `SELECT ... FOR UPDATE` or an `INSERT ... ON CONFLICT DO NOTHING RETURNING *` pattern, so the check and the write are atomic.

3. **Add the `@Index({ unique: true })` decorator** (or a scoped unique index via migration) to the `transactionId` column in the `Transaction` entity so TypeORM enforces the constraint at the ORM layer as well. [5](#0-4) 

### Proof of Concept

```typescript
// Attacker script — requires only a valid JWT token
import axios from 'axios';

const token = '<valid_jwt>';
const txBytes = '<base64_encoded_transaction_bytes>'; // same bytes for both requests

const payload = {
  name: 'dup-test',
  description: 'race',
  transactionBytes: txBytes,
  creatorKeyId: 1,
  signature: '<valid_sig>',
  mirrorNetwork: 'testnet',
};

// Fire both requests concurrently — both will pass the uniqueness check
const [r1, r2] = await Promise.all([
  axios.post('https://<api>/transactions', payload, { headers: { Authorization: `Bearer ${token}` } }),
  axios.post('https://<api>/transactions', payload, { headers: { Authorization: `Bearer ${token}` } }),
]);

// Expected (broken) outcome: both return HTTP 201 with different `id` values
// but the same `transactionId` string — two rows now exist in the DB.
console.log(r1.data.id, r2.data.id);           // different IDs
console.log(r1.data.transactionId === r2.data.transactionId); // true
```

**Expected result**: Both requests return `201 Created` with distinct database `id` values but identical `transactionId` strings, confirming two duplicate rows were inserted. A subsequent `SELECT * FROM transaction WHERE "transactionId" = '<id>'` returns two rows.
