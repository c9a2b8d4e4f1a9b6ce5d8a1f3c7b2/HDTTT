### Title
No Admin Demotion Mechanism — Irrevocable Admin Privilege Escalation

### Summary
The `elevateAdmin` function in `auth.service.ts` is strictly one-directional: it can only promote a user to admin (`admin: true`) and there is no corresponding API endpoint or service method to revoke or demote admin status. Once a user is elevated to admin, their privilege cannot be removed through any exposed API surface. This is the direct analog to the commented-out ownership transfer in `Swap.vy`.

### Finding Description
In `auth.service.ts`, the `elevateAdmin` method unconditionally sets `admin: true` with no inverse operation:

```ts
async elevateAdmin(userId: number): Promise<void> {
  await this.usersService.updateUserById(userId, { admin: true });
}
```

The only endpoint that triggers this is `PATCH /auth/elevate-admin` in `auth.controller.ts`, which is guarded by `AdminGuard` and only calls `authService.elevateAdmin`. There is no `demoteAdmin`, `revokeAdmin`, or equivalent method anywhere in `auth.service.ts` or `auth.controller.ts`.

The `users.controller.ts` exposes a `PATCH /users/:id` endpoint that calls `updateUserById` with an `UpdateUserDto`, but this endpoint's description says "Update the admin state of a user" — however, the `elevate-admin` dedicated path is the only documented and intended mechanism for admin role management, and no demotion path is exposed or documented.

Additionally, `removeUser` in `users.controller.ts` only guards against self-removal:

```ts
if (user.id === id) throw new BadRequestException(ErrorCodes.CRYFO);
```

There is no guard preventing the removal of the last admin. If the last admin removes all other admins and then their own account becomes inaccessible, or if a compromised admin account needs to be revoked, there is no in-protocol mechanism to do so.

### Impact Explanation
- A compromised admin account cannot be demoted through the API. The compromised admin retains full admin privileges indefinitely (can add/remove users, elevate others, etc.).
- If the last admin is removed (by another admin), no new admins can be created, no new users can be signed up (`POST /auth/signup` requires `AdminGuard`), and the organization backend becomes permanently locked from administrative operations.
- All admin-gated operations — user creation, admin elevation, user removal — become permanently inaccessible.

### Likelihood Explanation
Admin accounts are long-lived and credentials can be compromised. Organizations routinely need to rotate privileged accounts. The absence of a demotion path is a realistic operational gap, not a theoretical one. The scenario of a sole admin losing access or being compromised is a standard operational risk.

### Recommendation
1. Add a `demoteAdmin(userId: number)` method to `auth.service.ts` that sets `admin: false`, with a guard preventing demotion of the last remaining admin.
2. Add a corresponding `PATCH /auth/demote-admin` endpoint in `auth.controller.ts` protected by `AdminGuard`.
3. Add a last-admin guard in `removeUser` in `users.controller.ts` to prevent deletion of the sole admin.

**Scenario:**
1. Admin A elevates User B to admin via `PATCH /auth/elevate-admin`.
2. Admin B's credentials are compromised.
3. Admin A has no API call available to revoke Admin B's privileges — `elevateAdmin` only accepts `admin: true`, and no demotion endpoint exists.
4. Compromised Admin B retains full admin access permanently.
