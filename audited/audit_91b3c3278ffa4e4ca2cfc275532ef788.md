### Title
Unbounded Batch Signature Processing Enables Authenticated DoS via CPU Exhaustion

### Summary
The `POST /transactions/:id/signers` and `POST /transactions/signatures/import` endpoints accept an unbounded array of `UploadSignatureMapDto` objects, each containing a `signatureMap` with no entry-count constraint. All DTOs are processed concurrently via `Promise.all()`, and each triggers multiple expensive cryptographic operations. An authenticated user can craft a single request with a large array of DTOs or a deeply populated signature map to saturate the Node.js event loop and degrade service for all users. This is the direct analog of the external report's "poison order" pattern: instead of a contract consuming all EVM gas, a crafted payload consumes all server CPU.

### Finding Description

**Root cause 1 — No size limit on the `signatureMap` field**

The `IsSignatureMap` decorator iterates over all three nesting levels of the attacker-supplied JSON object with no count guard: [1](#0-0) 

For every leaf entry the decorator calls `PublicKey.fromString()` (an asymmetric-key parse), `decode()`, and `signatureMap.addSignature()`. There is no `@MaxLength`, `@ArrayMaxSize`, or entry-count check anywhere in `UploadSignatureMapDto`: [2](#0-1) 

**Root cause 2 — No size limit on the DTO array**

Both controllers accept `UploadSignatureMapDto | UploadSignatureMapDto[]` with no `@ArrayMaxSize` guard: [3](#0-2) [4](#0-3) 

**Root cause 3 — Concurrent processing of all DTOs**

`validateAndProcessSignatures` fans out all DTOs simultaneously with `Promise.all()`: [5](#0-4) 

For each DTO, `processTransactionSignatures` runs a triple-nested loop over the signature map and calls `sdkTransaction.addSignature()` (protobuf re-serialization) for every unique public key, followed by `sdkTransaction.toBytes()`: [6](#0-5) 

**Exploit path**

1. Attacker registers as a verified user and registers one public key.
2. Attacker constructs a JSON body that is an array of N `UploadSignatureMapDto` objects. Each DTO references a real (or non-existent) transaction ID and a `signatureMap` with the attacker's own public key spread across many node-account-ID × transaction-ID combinations.
3. A single POST to `/transactions/signers` triggers:
   - `IsSignatureMap` decorator: `PublicKey.fromString()` × (nodes × txIds × keys) per DTO
   - `Promise.all()` launches all N processing coroutines simultaneously
   - Each coroutine calls `SDKTransaction.fromBytes()` twice, iterates the full map, and calls `sdkTransaction.addSignature()` + `sdkTransaction.toBytes()`
4. The IP throttler (`IpThrottlerModule`) limits request rate but does not constrain payload size or per-request CPU cost: [7](#0-6) 

### Impact Explanation
A single crafted request can pin the Node.js event loop for an extended period, causing all concurrent API requests to time out or queue indefinitely. Because the API service is the sole entry point for transaction management, signature submission, and user operations, sustained CPU exhaustion constitutes a full service outage for all organization members. No funds are at risk, but the availability of the multi-signature workflow is completely disrupted.

### Likelihood Explanation
Any verified user of the organization can trigger this. Becoming a verified user requires admin invitation, but once inside, the attacker needs only a standard HTTP client. The IP throttler allows a configurable number of requests per second; each request can carry a payload sized up to the HTTP body limit (NestJS default: 100 KB), which is sufficient to embed hundreds of signature-map entries or dozens of DTOs. No privileged keys, no leaked secrets, and no internal network access are required.

### Recommendation

1. **Cap the DTO array length** — add `@ArrayMaxSize(N)` (e.g., 50) to the array parameter in both controllers, or enforce it inside `transformAndValidateDto`.
2. **Cap signature-map entries** — inside `IsSignatureMap`, count total entries across all three loops and throw `BadRequestException` above a reasonable threshold (e.g., 500 total leaf entries).
3. **Process DTOs sequentially or with bounded concurrency** — replace the unbounded `Promise.all()` in `validateAndProcessSignatures` with a concurrency-limited executor (e.g., process at most 10 DTOs at a time).
4. **Apply a per-user throttle to the signature upload endpoints** — the existing `UserThrottlerGuard` is defined but not applied to these routes; adding it would limit burst abuse per account.

### Proof of Concept

```
POST /transactions/signers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  // Repeat this object 200 times, each with a different `id`
  {
    "id": 1,
    "signatureMap": {
      "0.0.3":   { "0.0.1001@1700000000.000000000": { "<attacker_pubkey_DER>": "<valid_sig_hex>" } },
      "0.0.4":   { "0.0.1001@1700000000.000000000": { "<attacker_pubkey_DER>": "<valid_sig_hex>" } },
      // ... repeat for 50 node account IDs
    }
  },
  { "id": 2, "signatureMap": { ... } },
  ...
]
```

Expected outcome: the server spawns 200 concurrent coroutines, each deserializing an SDK transaction, iterating 50 map entries with `PublicKey.fromString()`, and calling `addSignature()` + `toBytes()`. The event loop stalls; subsequent requests from legitimate users receive timeouts. The IP throttler does not prevent this because it is a single request.
