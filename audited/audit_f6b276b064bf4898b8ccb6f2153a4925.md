### Title
Race Condition in `uploadSignatureMaps` Allows Silent Signature Loss via Unguarded Concurrent `transactionBytes` Overwrite

---

### Summary

`SignersService.uploadSignatureMaps` reads transaction state, validates it, and then unconditionally overwrites `transactionBytes` in the database with no atomic guard. Two concurrent signing requests for the same transaction can both pass the status check, both compute new `transactionBytes` from the same stale base, and then both write back — with the second write silently discarding the first's signatures. This is the direct analog of the Vyper `@nonreentrant` bug: a guard that is supposed to prevent unsafe concurrent state mutation does not actually prevent it.

---

### Finding Description

The `uploadSignatureMaps` flow in `SignersService` is:

1. **`loadTransactionData`** — reads `transaction.status` and existing `TransactionSigner` rows into memory (no lock, no `SELECT FOR UPDATE`).
2. **`validateTransactionStatus`** — checks the in-memory status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. This is the intended "guard."
3. **`processTransactionSignatures`** — deserializes `transactionBytes`, calls `sdkTransaction.addSignature(...)`, producing new bytes.
4. **`persistSignatureChanges` → `bulkUpdateTransactions`** — issues an unconditional `UPDATE transaction SET "transactionBytes" = ... WHERE id = ANY(...)` with **no status predicate**.
5. **`updateStatusesAndNotify` → `processTransactionStatus`** — re-reads `transactionBytes` from DB to determine whether to advance status.

The critical flaw is in step 4. The SQL issued is:

```sql
UPDATE transaction
SET "transactionBytes" = CASE id WHEN <id> THEN $1::bytea END,
    "updatedAt" = NOW()
WHERE id = ANY($2)
```

There is no `AND status = 'WAITING FOR SIGNATURES'` predicate and no row-level lock. Any concurrent request that passed step 2 will unconditionally overwrite whatever bytes are currently in the row.

**Race window:**

| Time | Request A (User A, key K1) | Request B (User B, key K2) |
|------|---------------------------|---------------------------|
| T1 | reads `transactionBytes` = `B0` | reads `transactionBytes` = `B0` |
| T2 | validates status = OK | validates status = OK |
| T3 | computes `B_A` = `B0 + sig(K1)` | computes `B_B` = `B0 + sig(K2)` |
| T4 | writes `B_A` to DB | — |
| T5 | — | writes `B_B` to DB (overwrites `B_A`) |
| T6 | `TransactionSigner` row for K1 inserted | `TransactionSigner` row for K2 inserted |

After T5, the DB contains `B_B` (only K2's signature). K1's signature is permanently lost. Both `TransactionSigner` rows exist, but `processTransactionStatus` at step 5 reads `_signerPublicKeys` from `transactionBytes` — it sees only K2, so the transaction may never advance to `WAITING_FOR_EXECUTION`.

A secondary issue: `TransactionSigner` has only a non-unique index on `(transactionId, userKeyId)`:

```typescript
@Index(['transactionId', 'userKeyId'])
export class TransactionSigner { ... }
```

The migration confirms no `UNIQUE` constraint exists on this pair. `bulkInsertSigners` uses a plain `INSERT` with no `ON CONFLICT` clause, so duplicate signer rows can be inserted by concurrent requests.

---

### Impact Explanation

A transaction that has collected all required signatures can be permanently stuck in `WAITING_FOR_SIGNATURES` because the last concurrent write discarded earlier signatures from `transactionBytes`. The transaction will never reach `WAITING_FOR_EXECUTION` and will eventually expire. Signatures contributed by legitimate organization members are silently lost with no error returned to the caller — both requests return HTTP 200. In a multi-signature workflow requiring M-of-N keys, this can permanently block execution of any transaction where two signers submit concurrently.

---

### Likelihood Explanation

In Organization Mode, multiple users are explicitly expected to sign the same transaction, often in rapid succession after receiving a WebSocket notification that a new signature is needed. The frontend's real-time notification system (`notifications` service via NATS → WebSocket) actively triggers concurrent signing. No special attacker capability is required — two legitimate organization members signing at the same time is the normal use case. The race window spans the entire async processing pipeline (signature validation + SDK deserialization + DB write), which is wide enough to be hit in practice.

---

### Recommendation

Replace the unconditional `bulkUpdateTransactions` with an atomic compare-and-swap pattern. Use PostgreSQL's `SELECT ... FOR UPDATE` to lock the transaction row at the start of `loadTransactionData`, holding the lock for the duration of the DB transaction in `persistSignatureChanges`. Alternatively, use an optimistic locking column (e.g., `version`) and include `AND version = :expectedVersion` in the UPDATE predicate, retrying on conflict. The DB transaction in `persistSignatureChanges` must encompass the read, validate, and write steps atomically.

For `bulkInsertSigners`, add a `UNIQUE` constraint on `(transactionId, userKeyId)` in the database and use `ON CONFLICT DO NOTHING` (or `ON CONFLICT DO UPDATE`) in the insert query.

---

### Proof of Concept

**Step 1 — Setup:** Create an organization transaction requiring two signers (User A with key K1, User B with key K2).

**Step 2 — Concurrent signing:** Send two simultaneous POST requests to the signature upload endpoint:
- Request A: User A uploads `signatureMap` containing K1's signature
- Request B: User B uploads `signatureMap` containing K2's signature

Both requests enter `uploadSignatureMaps` at the same time.

**Step 3 — Both pass the guard:** `validateTransactionStatus` at [1](#0-0) 
reads from the in-memory `transactionMap` loaded before either write. Both see `WAITING_FOR_SIGNATURES` and proceed.

**Step 4 — Both compute new bytes from the same stale base:** `processTransactionSignatures` at [2](#0-1) 
deserializes `transactionBytes` from the snapshot loaded at T1. Request A produces `B0 + sig(K1)`, Request B produces `B0 + sig(K2)`.

**Step 5 — Last writer wins:** `bulkUpdateTransactions` at [3](#0-2) 
issues an unconditional UPDATE. Whichever request commits second overwrites the other's bytes. The losing signature is gone.

**Step 6 — Status never advances:** `processTransactionStatus` at [4](#0-3) 
reads `_signerPublicKeys` from the surviving `transactionBytes`. It sees only one key, so `hasValidSignatureKey` returns false, and the status remains `WAITING_FOR_SIGNATURES`. The transaction expires.

**Step 7 — Duplicate signer rows:** `bulkInsertSigners` at [5](#0-4) 
inserts without conflict handling. The `TransactionSigner` entity has only a non-unique index: [6](#0-5) 
confirmed by the migration DDL: [7](#0-6) 
Both requests insert their signer rows successfully, creating a misleading audit trail showing both users signed, while only one signature is actually present in `transactionBytes`.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
