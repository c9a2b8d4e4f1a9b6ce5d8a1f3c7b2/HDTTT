### Title
Unbounded In-Memory Iteration Over All Active Transactions in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign()` in the API service fetches **every non-terminal transaction in the entire system** from the database with no row limit, then performs a sequential async DB round-trip per transaction (`userKeysToSign`) before applying pagination in memory. A verified user who creates a large number of transactions can cause this endpoint to exhaust server memory and CPU, making it unavailable for all users.

### Finding Description

**Root cause — unbounded DB fetch + O(N) async loop:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–309, the function fetches all transactions whose status is not in the terminal set, with no `take` (SQL `LIMIT`) applied:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters by status, NOT by user
  relations: ['groupItem'],
  order,
});                      // ← no `take` / `skip` — full table scan

for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [1](#0-0) 

The `whereForUser` predicate only excludes terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope the query to the requesting user's transactions: [2](#0-1) 

For each transaction in the result set, `userKeysToSign` is called, which internally calls `userKeysRequiredToSign` → `keysRequiredToSign`, which:
1. Deserializes the transaction bytes via the Hiero SDK
2. Calls `transactionSignatureService.computeSignatureKey()` (async)
3. Issues an `entityManager.find(UserKey, ...)` DB query [3](#0-2) 

Pagination (`offset`/`limit`) is applied **after** all N iterations complete, in memory: [4](#0-3) 

**Exploit path:**

1. Attacker registers and verifies an account (normal user flow).
2. Attacker creates thousands of transactions via `POST /transactions`. Each transaction is cheap to create and remains in a non-terminal state (`NEW` or `WAITING_FOR_SIGNATURES`) indefinitely until it expires (3 minutes past `validStart`). The attacker can set `validStart` far in the future to keep transactions active.
3. Any user (including the attacker) calls `GET /transactions/sign`. The server loads all N active transactions into memory and performs N sequential async DB queries.
4. With enough transactions, the Node.js process hits memory limits or the request times out, making the endpoint unavailable for all users.

**The controller endpoint is accessible to any verified user:** [5](#0-4) 

### Impact Explanation

- The `GET /transactions/sign` endpoint becomes progressively slower as the active transaction count grows, eventually timing out or causing OOM in the Node.js process.
- Because the query is not scoped to the requesting user, **all users share the same unbounded result set**. A single attacker inflating the transaction table degrades the endpoint for every user in the organization.
- The endpoint is a core workflow feature (showing which transactions need signing), so its unavailability disrupts normal operations.

### Likelihood Explanation

- Any verified user can create transactions — no admin or privileged role is required.
- Creating transactions is a normal, expected action; there is no rate limit visible in the controller.
- The attack is cheap: each transaction creation is a single API call, and transactions with a far-future `validStart` remain active for a long time.
- The endpoint is called frequently by the frontend during normal use.

### Recommendation

Apply the database-level `take`/`skip` before the loop, scoped to the requesting user. The correct fix is to push the user-relevance filter into the SQL query (joining on signers/observers/creatorKey) and apply `LIMIT`/`OFFSET` at the DB layer, eliminating the need to load and iterate over the entire active transaction set in memory. The existing `getTransactions` and `getTransactionsToApprove` methods already demonstrate the correct pattern using `skip`/`take` in `FindManyOptions`. [6](#0-5) 

### Proof of Concept

1. Register and verify a user account.
2. Obtain a JWT token via `POST /auth/login`.
3. In a loop, call `POST /transactions` with a `validStart` set 30 days in the future, creating 10,000 transactions. Each call succeeds and the transaction enters `NEW` status.
4. Call `GET /transactions/sign` (with any pagination params, e.g., `?page=1&size=10`).
5. Observe: the server loads all 10,000 transactions from the DB, then performs 10,000 sequential async calls to `userKeysToSign`. Response time grows linearly with transaction count. At sufficient scale the request times out or the process crashes with an OOM error, returning 500 or no response to all concurrent users.

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
```
