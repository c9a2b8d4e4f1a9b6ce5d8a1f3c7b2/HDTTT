### Title
Stale Parent `threshold` After Child Approver Removal Causes Permanently Unsatisfiable Approval Condition

### Summary
`removeTransactionApprover()` in `ApproversService` soft-deletes a target approver node and its entire subtree via `removeNode()`, but never updates the **parent** node's `threshold` field. When a child node of a threshold-tree is removed, the parent retains its original threshold value even though it now has fewer children. This makes the approval condition permanently unsatisfiable and locks the transaction in `WAITING_FOR_SIGNATURES`. Additionally, `emitTransactionStatusUpdate` is called with `{ entityId: null }` when the removed node is a non-root child (because `approver.transactionId` is `null` for child nodes), silently corrupting the notification event.

### Finding Description

**Root cause — `removeTransactionApprover` never touches the parent's `threshold`:** [1](#0-0) 

`removeNode` executes a recursive SQL soft-delete that marks the target node and all its descendants as deleted, but performs no `UPDATE` on the parent row: [2](#0-1) 

Contrast this with `updateTransactionApprover` (the "detach child" path, `dto.listId === null`), which **does** correctly adjust the parent's threshold or soft-removes the parent when it becomes childless: [3](#0-2) 

`removeTransactionApprover` contains no equivalent logic. The controller passes any caller-supplied approver `id` directly, with no guard that the target must be a root node: [4](#0-3) 

**Secondary bug — null `entityId` in status notification:**
Child approvers store `transactionId = null` (only root nodes carry `transactionId`). After deletion, line 541 emits `{ entityId: null }` to the chain service, which silently drops or misroutes the status-update event. [5](#0-4) 

### Impact Explanation

**Concrete scenario:**

1. Creator builds a threshold tree: `Parent (threshold=2)` → `[Child A (userId=1), Child B (userId=2)]`.
2. User 1 approves — Child A receives a valid signature.
3. Creator calls `DELETE /transactions/:txId/approvers/:childBId` (Child B's id).
4. `removeNode` soft-deletes Child B. Parent still has `threshold = 2`, but now has only one non-deleted child (Child A).
5. The approval query (`getApproversByTransactionId`) filters `deletedAt IS NULL`, so it returns only Child A.
6. The chain service evaluates: "2 of 1 approvers must approve" — a condition that can **never** be satisfied.
7. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` (or until it expires on-chain), wasting all prior signing effort and blocking any dependent workflow.

The invariant `threshold ≤ count(non-deleted children)` is violated and never repaired.

### Likelihood Explanation

The trigger path requires only that the caller be the authenticated creator of the transaction — a normal, unprivileged user role. The `DELETE /transactions/:transactionId/approvers/:id` endpoint accepts any approver `id` without validating whether it is a root or a child node. No special credentials, admin rights, or race conditions are needed. Any creator who has set up a threshold-tree approver structure and later removes a child node will silently corrupt the parent's threshold.

### Recommendation

In `removeTransactionApprover`, after soft-deleting the target node, fetch the parent (if any) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If `parent.approvers.length - 1 === 0`, soft-remove the parent.
- Else if `parent.threshold > parent.approvers.length - 1`, update `parent.threshold` to `parent.approvers.length - 1`.

Also replace `approver.transactionId` in the `emitTransactionStatusUpdate` call with the actual root node's `transactionId`, obtained via `getRootNodeFromNode(approver.id)` before deletion.

### Proof of Concept

```
# Setup
POST /transactions/:txId/approvers
Body: { approversArray: [{ threshold: 2, approvers: [{ userId: 1 }, { userId: 2 }] }] }
# → creates Parent (id=10, threshold=2) → Child A (id=11, userId=1), Child B (id=12, userId=2)

# User 1 approves

# Creator removes Child B (a non-root child)

# → removeNode(12) soft-deletes Child B only
# → Parent (id=10) still has threshold=2, but only Child A remains

# State after removal:
# Parent: threshold=2, children=[Child A]  ← INVARIANT VIOLATED
# emitTransactionStatusUpdate called with { entityId: null }  ← SILENT BUG

# Transaction is permanently stuck in WAITING_FOR_SIGNATURES.
```

```typescript
  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
```

```typescript
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);
```
