### Title
Broken `getUserShouldApprove` Implementation Always Returns `false`, Silently Disabling the Entire Approval Workflow

### Summary
The `getUserShouldApprove` function in `front-end/src/renderer/services/organization/transaction.ts` is supposed to query the backend API (`GET /transactions/approve/:transactionId`) to determine whether the authenticated user must approve a given transaction. Instead, the real API call is commented out and the function unconditionally returns `false`. This is a direct analog to the external report: a function that is supposed to compute a meaningful value based on real state instead returns a hardcoded incorrect result, silently breaking the feature that depends on it.

### Finding Description

**Root cause — hardcoded stub:**

```typescript
// front-end/src/renderer/services/organization/transaction.ts  lines 155-172
export const getUserShouldApprove = async (
  _serverUrl: string,
  _transactionId: number,
): Promise<boolean> =>
  commonRequestHandler(async () => {
    //TODO Approve is not implemented yet ...
    // const { data } = await axiosWithCredentials.get(
    //   `${serverUrl}/${controller}/approve/${transactionId}`,
    // );
    // return data;

    return false;          // ← always false, regardless of actual approver status
  }, 'Failed to get if user should approve the transaction');
``` [1](#0-0) 

**Call-site 1 — `ApproveAllController.vue`:**

The "Approve All" / "Reject All" bulk-action controller iterates every group item and calls `getUserShouldApprove` to decide whether to send an approval signature. Because the function always returns `false`, the inner body — including `sendApproverChoice` — is **never executed**. The loop completes silently, a success toast is shown, and no approval is ever submitted to the backend.

```typescript
for (const item of group.groupItems) {
  if (await getUserShouldApprove(serverUrl, item.transaction.id)) {
    // ... sign transaction body
    await sendApproverChoice(...);   // ← never reached
  }
}
``` [2](#0-1) 

**Call-site 2 — `TransactionDetailsHeader.vue`:**

The transaction detail view uses the return value to decide whether to show the approve/reject button to the user. Because it always returns `false`, designated approvers never see the button.

```typescript
const approvePromise: Promise<boolean> = FEATURE_APPROVERS_ENABLED
  ? getUserShouldApprove(user.selectedOrganization.serverUrl, transaction.id)
  : Promise.resolve(false);
// ...
results[1].status === 'fulfilled' && (shouldApprove.value = results[1].value);
``` [3](#0-2) 

**The backend is fully implemented and correct.** The endpoint `GET /transactions/approve/:transactionId` exists and `shouldApproveTransaction` in `transactions.service.ts` correctly queries the approver table and returns the real status. The stub is purely a front-end oversight. [4](#0-3) [5](#0-4) 

### Impact Explanation

1. **Approval workflow permanently broken for all group transactions.** Any transaction group that requires approvals before execution will never receive them through the "Approve All" UI path. The transaction remains stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely — a permanent freeze of organizational transaction state.

2. **Approve/Reject button never rendered for designated approvers.** Even for single-transaction views, `shouldApprove` is always set to `false`, so the UI never presents the approval action to users who are legitimately required to approve.

3. **Silent failure with false success feedback.** `ApproveAllController.vue` shows a success toast after the loop, giving approvers false confidence that their approval was recorded when nothing was sent.

Severity: **High** — the approver role is a core multi-signature governance control. Its complete silent failure means multi-party authorization requirements can never be satisfied through the intended UI, permanently locking transactions that depend on approvals.

### Likelihood Explanation

- Any organization that has configured transaction approvers and uses the "Approve All" or "Reject All" UI will be affected on every use.
- No special attacker capability is required; this is triggered by normal product usage.
- The backend API and approver data model are fully wired up, so the feature appears complete to administrators who set it up, making the silent failure especially deceptive.

### Recommendation

Uncomment and restore the real API call in `getUserShouldApprove`:

```typescript
export const getUserShouldApprove = async (
  serverUrl: string,
  transactionId: number,
): Promise<boolean> =>
  commonRequestHandler(async () => {
    const { data } = await axiosWithCredentials.get(
      `${serverUrl}/${controller}/approve/${transactionId}`,
    );
    return data;
  }, 'Failed to get if user should approve the transaction');
```

Remove the underscore prefixes from the parameters (they were added only because the parameters were unused in the stub). Add an integration test that verifies the function actually calls the backend endpoint and returns its response.

### Proof of Concept

1. Configure an organization with a transaction that has a designated approver (user A).
2. Log in as user A.
3. Navigate to the transaction group containing that transaction.
4. Click "Approve All."
5. Observe: the success toast fires, but no `POST` to `/transactions/approve` is made (verify via network inspector or backend logs).
6. Refresh the transaction — its approval status is unchanged; `sendApproverChoice` was never called because `getUserShouldApprove` returned `false` for every item in the loop. [1](#0-0)

  /* Check whether the user should approve the transaction */
  async shouldApproveTransaction(transactionId: number, user: User) {
    const transaction = await this.getTransactionById(transactionId);
    if (!transaction) {
      throw new BadRequestException(ErrorCodes.TNF);
    }

    if (this.terminalStatuses.includes(transaction.status)) {
      return false;
    }

    /* Get all the approvers */
    const approvers = await this.approversService.getApproversByTransactionId(transactionId);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0) return false;

    /* Check if the user has already approved the transaction */
    return !userApprovers.every(a => a.signature);
  }
