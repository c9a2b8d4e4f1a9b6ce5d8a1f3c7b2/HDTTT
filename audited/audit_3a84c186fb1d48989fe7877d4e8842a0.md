### Title
Any Authenticated User Can Create Comments on Arbitrary Transactions Without Access Verification

### Summary
`CommentsController.createComment()` in `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies JWT authentication at the controller level but performs no authorization check to verify the requesting user has any relationship to the target transaction. Any authenticated organization user can post comments on any transaction — including transactions they were never added to as a creator, approver, observer, or signer. The code itself contains an explicit TODO acknowledging this gap.

### Finding Description
**Root cause:** The controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only confirm the caller holds a valid, non-blacklisted JWT and has a verified account. No guard, interceptor, or service-layer check verifies that the authenticated user has any permitted relationship to the `transactionId` supplied in the route parameter. [1](#0-0) 

The developer-left TODO at line 22 reads:
```
//TODO need some sort of guard or check to ensure user can comment here
```

This confirms the missing check is known but unimplemented.

**Exploit path:**
1. Attacker registers a legitimate organization account and obtains a JWT via `POST /auth/login`.
2. Attacker enumerates or guesses a `transactionId` for a transaction they have no role in (IDs are sequential integers).
3. Attacker calls `POST /transactions/<targetId>/comments` with a crafted body — no ownership or membership check is performed.
4. The comment is persisted to the database and associated with the target transaction.

Additionally, `getComments` and `getCommentById` also pass no user context to the service, meaning any authenticated user can read all comments on any transaction: [2](#0-1) 

### Impact Explanation
- **Unauthorized state modification:** Any authenticated user can inject comments into transactions they have no business relationship with, polluting the audit trail and communication history of other users' transactions.
- **Information disclosure:** `getComments` and `getCommentById` expose the full comment history of any transaction to any authenticated user, leaking potentially sensitive operational details (e.g., signing instructions, key references, business context) embedded in comments.
- **Integrity failure:** The comment thread is part of the transaction's organizational workflow. Unauthorized comments break the trust model that only involved parties (creator, approvers, observers, signers) interact with a transaction.

### Likelihood Explanation
- **Preconditions:** Attacker only needs a valid organization account — the lowest privilege level in the system.
- **Transaction ID enumeration:** IDs are sequential integers (`ParseIntPipe`), making enumeration trivial with a simple loop.
- **No rate limiting** is applied to the comment endpoints.
- Any disgruntled or malicious employee with a valid account can exploit this immediately.

### Recommendation
Add an access-verification step in `CommentsService.createComment()` (or as a guard/interceptor) that confirms the requesting user is the creator, an approver, an observer, or a signer of the target transaction — mirroring the pattern already used in `TransactionsService.getTransactionWithVerifiedAccess()`: [3](#0-2) 

Apply the same `verifyAccess` check before allowing comment creation or retrieval.

### Proof of Concept
```
# Step 1 – Obtain a JWT as a low-privilege user
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2 – Post a comment on a transaction the attacker has no role in
POST /transactions/42/comments
Authorization: Bearer <JWT>
{ "content": "Injected comment on transaction 42" }
→ HTTP 201 Created  (no 401/403 returned)

# Step 3 – Read all comments on any transaction
GET /transactions/42/comments
Authorization: Bearer <JWT>
→ HTTP 200 OK  [ { "id": 1, "content": "...", ... }, ... ]
```

The server returns `201` for the write and `200` for the read with no ownership enforcement, confirming the missing authorization check at [4](#0-3) .
