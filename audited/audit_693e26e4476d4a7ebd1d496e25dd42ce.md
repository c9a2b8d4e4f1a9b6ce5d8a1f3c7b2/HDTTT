### Title
`importSignatures()` Emits `emitTransactionStatusUpdate` for Transactions Whose DB Write Failed

### Summary
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `importSignatures()` function processes signature imports in batches. When a batch DB write fails (caught in the inner `catch` block), the affected transactions are correctly marked as failed in the `results` map, but `emitTransactionStatusUpdate` is still called with the full `updateArray` — including the transactions whose DB write failed. This causes the notification service to emit spurious status-update events for transactions that were never actually updated, misleading observers, approvers, and signers into believing a signature import succeeded when it did not.

### Finding Description

The `importSignatures()` function in `transactions.service.ts` follows this flow:

1. **Validation loop** (lines 525–572): Each signature import request is validated. Passing items are added to the `updates` map; failing items are recorded with errors in `results`.

2. **Batch DB write loop** (lines 580–614): Items from `updates` are written to the DB in batches of 500. Each batch is wrapped in its own `try/catch`. If the DB write **fails**, the catch block at line 605 marks those items with errors in `results` — but does **not** remove them from `updateArray`.

3. **Notification emission** (lines 616–622): `emitTransactionStatusUpdate` is called unconditionally with the **entire** `updateArray`, regardless of which batches succeeded or failed. [1](#0-0) 

The root cause is that `updateArray` is derived from `updates.values()` before the batch loop runs, and the batch loop's `catch` block only updates `results` — it never filters failed items out of `updateArray`. So the notification call at line 616 always uses the pre-failure list. [2](#0-1) 

`emitTransactionStatusUpdate` publishes a NATS `TRANSACTION_STATUS_UPDATE` message to the notification service: [3](#0-2) 

The notification service (`processTransactionStatusUpdateNotifications`) then reads the transaction from the DB and dispatches in-app and email notifications to all relevant parties (creator, approvers, observers, signers): [4](#0-3) 

### Impact Explanation

When a DB batch write fails, the `results` map correctly reports failure to the API caller. However, the notification service is simultaneously told that those same transactions were updated. It re-reads the (unchanged) transaction state from the DB and sends notifications to all stakeholders — approvers, observers, and signers — indicating a status change that never occurred. In a multi-signature workflow, this can cause:

- Approvers or signers to believe a signature was successfully recorded and act on that assumption (e.g., proceeding to execution).
- Observers to receive false "transaction updated" alerts, eroding trust in the notification system.
- Downstream automation or scripts that consume these events to make incorrect decisions about transaction readiness.

The integrity invariant violated: **a status-update notification must only be emitted when the underlying state change was durably persisted.**

### Likelihood Explanation

Any authenticated user with access to at least one transaction can call `importSignatures`. The DB write failure path (line 605) is triggered by transient infrastructure issues (DB connection drops, query timeouts, constraint violations on the raw SQL CASE update). These are realistic in production under load or during DB maintenance. No privileged access is required — a normal user submitting a valid signature import request is sufficient to reach this code path.

### Recommendation

Filter `updateArray` to exclude items from failed batches before calling `emitTransactionStatusUpdate`. Track which batch IDs succeeded and only emit notifications for those:

```typescript
const successfulIds = new Set<number>();

for (let i = 0; i < updateArray.length; i += BATCH_SIZE) {
  const batch = updateArray.slice(i, i + BATCH_SIZE);
  try {
    await this.entityManager.createQueryBuilder()...execute();
    batch.forEach(u => {
      results.set(u.id, { id: u.id });
      successfulIds.add(u.id);  // only track successful writes
    });
  } catch (err) {
    batch.forEach(u => results.set(u.id, { id: u.id, error: message }));
    // do NOT add to successfulIds
  }
}

const successfulUpdates = updateArray.filter(r => successfulIds.has(r.id));
if (successfulUpdates.length > 0) {
  emitTransactionStatusUpdate(
    this.notificationsPublisher,
    successfulUpdates.map(r => ({
      entityId: r.id,
      additionalData: { transactionId: r.transactionId, network: r.network },
    })),
  );
}
```

### Proof of Concept

1. Authenticated user submits a valid `importSignatures` request for N transactions (N > 0, all passing validation).
2. All N items are added to `updates`; `updateArray` has N entries.
3. During the batch DB write, the DB connection drops — the `catch` block at line 605 fires, marking all N items as failed in `results`.
4. Execution continues to line 616: `emitTransactionStatusUpdate` is called with all N transaction IDs.
5. The NATS notification service receives the event, fetches the (unchanged) transactions from DB, and dispatches in-app/email notifications to all approvers, observers, and signers for those N transactions.
6. The API caller receives N error results — correctly indicating failure — but all N stakeholders have already received spurious "transaction updated" notifications. [5](#0-4)

  async processTransactionStatusUpdateNotifications(events: NotificationEventDto[]) {
    const ctx = await this.prepareEventContext(events, true);
    if (!ctx) return;

    const {
      cache,
      keyCache,
      transactionMap,
      approversMap,
      deletionNotifications,
      inAppNotifications,
      emailNotifications,
      inAppReceiverIds,
      emailReceiverIds,
      affectedUsers,
    } = ctx;

    // Process each event
    for (const { entityId: transactionId } of events) {
      const transaction = transactionMap.get(transactionId);
      if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping status-update notifications`);
        continue;
      }
      const approvers = approversMap.get(transactionId) || [];

      if (transaction.deletedAt && transaction.status !== TransactionStatus.CANCELED) {
        console.error(
          `Soft-deleted transaction ${transactionId} has unexpected status: ${transaction.status} (expected CANCELED)`
        );
        transaction.status = TransactionStatus.CANCELED;
      }

      const syncType = this.getInAppNotificationType(transaction.status);
      const emailType = this.getEmailNotificationType(transaction.status);

      // Single transaction for both notification types
      await this.entityManager.transaction(async entityManager => {
        await this.handleTransactionStatusUpdateNotifications(
          entityManager,
          transaction,
          approvers,
          syncType,
          emailType,
          cache,
          keyCache,
          deletionNotifications,
          inAppNotifications,
          inAppReceiverIds,
          emailNotifications,
          emailReceiverIds,
          affectedUsers,
          transactionId,
        );
      });
    }

    // Send all notifications in batch
    await this.sendDeletionNotifications(deletionNotifications);
    await this.sendInAppNotifications(inAppNotifications, inAppReceiverIds);
    await this.sendEmailNotifications(emailNotifications, emailReceiverIds);
    await this.sendNotifyClients(affectedUsers, TRANSACTION_EVENT_TYPE.STATUS_UPDATE);
