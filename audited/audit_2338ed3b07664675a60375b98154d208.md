### Title
`cutoffAt` Deadline Is Stored But Never Enforced in Signing and Approval Flows

### Summary
The `Transaction` entity exposes a `cutoffAt` field intended as a hard deadline for the signing window. This field is accepted in the creation DTO, persisted to the database, and surfaced in the API response — but it is never checked in either the signature-upload path (`SignersService.validateTransactionStatus`) or the approval path (`ApproversService.approveTransaction`). Any authenticated signer or approver can submit after the cutoff has passed, making the deadline completely non-functional.

### Finding Description

**Root cause — field is stored but never read at enforcement time.**

`cutoffAt` is declared as a nullable `Date` column on the entity: [1](#0-0) 

It is accepted as an optional DTO field: [2](#0-1) 

It is persisted when a transaction is created: [3](#0-2) 

**Signing path — `validateTransactionStatus` never reads `cutoffAt`.**

The only gate before signatures are accepted is a status check and an SDK-level expiry check. There is no comparison against `cutoffAt`: [4](#0-3) 

**Approval path — `approveTransaction` never reads `cutoffAt`.**

The approval handler checks status and signature validity, but again performs no `cutoffAt` comparison: [5](#0-4) 

The database documentation confirms `cutoffAt` is populated with a real timestamp (e.g., `2024-05-24 18:13:28`) that precedes `validStart` (`2024-05-24 19:12:00`), confirming it is intended as a meaningful pre-execution deadline: [6](#0-5) 

**Exploit flow:**

1. Creator submits a transaction with `cutoffAt = T` (e.g., 30 minutes before `validStart`).
2. At time `T`, only 2 of 3 required signatures have been collected; the transaction should be frozen/rejected.
3. Because `cutoffAt` is never checked, a third signer calls `POST /transactions/:id/signers` at `T + 1 minute`.
4. `validateTransactionStatus` passes (status is still `WAITING_FOR_SIGNATURES`, SDK expiry has not elapsed).
5. The signature is accepted, the threshold is met, and the transaction proceeds to execution — violating the creator's explicit deadline intent.

The same applies to the approval flow via `POST /transactions/:id/approvers/approve`.

### Impact Explanation
The `cutoffAt` deadline is the mechanism by which a transaction creator controls the signing window. With it unenforced, any signer or approver can unilaterally extend the signing window past the creator's stated cutoff. This can cause a transaction to be executed that the creator intended to expire unsigned, or allow a late approver to override an implicit rejection that had already been reached by the cutoff. In multi-party treasury or governance workflows this constitutes unauthorized state change: a transaction executes when the creator's policy said it should not.

### Likelihood Explanation
The attacker is any authenticated organization member who holds a signing key for the transaction. No elevated privileges are required. The attack requires only submitting a normal signing request after the `cutoffAt` timestamp — a trivially reachable condition for any signer who is late or deliberately delays their submission.

### Recommendation
Add a `cutoffAt` guard in both enforcement points:

**In `SignersService.validateTransactionStatus`** (`signers.service.ts`, after the status check):
```typescript
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  return ErrorCodes.TC; // Transaction Cutoff (new error code)
}
```

**In `ApproversService.approveTransaction`** (`approvers.service.ts`, after the status check at line 588):
```typescript
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  throw new BadRequestException(ErrorCodes.TC);
}
```

Add a corresponding `TC` entry to `ErrorCodes` and document the new error in the API.

### Proof of Concept

1. Authenticate as an organization user and create a transaction with `cutoffAt` set 1 minute in the future:
   ```
   POST /transactions
   { ..., "cutoffAt": "<now + 60s>" }
   ```
2. Wait 61 seconds (past the cutoff).
3. Submit a valid signature for the transaction:
   ```
   POST /transactions/:id/signers
   { "signatureMap": { ... } }
   ```
4. **Expected (correct) behavior:** HTTP 400 with a cutoff-exceeded error.
5. **Actual behavior:** HTTP 201 — signature is accepted, `TransactionSigner` row is inserted, and the transaction status is updated as if the deadline never existed.
