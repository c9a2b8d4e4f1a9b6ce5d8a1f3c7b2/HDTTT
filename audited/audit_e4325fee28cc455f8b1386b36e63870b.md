### Title
Incorrect State Transition Guard in `archiveTransaction` Allows Terminal-State Transactions to Be Archived

### Summary
The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a logically flawed guard condition. Because the status check is short-circuited by `!transaction.isManual`, any transaction created with `isManual: true` can be moved to `ARCHIVED` regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `REJECTED`. This is a direct analog to the external report's incorrect state-transition bug: a successful operation (execution) should produce an immutable terminal state, but the wrong guard condition allows an additional, unintended state transition out of it.

### Finding Description

In `archiveTransaction`, the guard is:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The exception is thrown only when **both** conditions are true:
- Status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- `isManual` is `false`

When `isManual` is `true`, the entire status check is bypassed. The function then unconditionally writes `status: ARCHIVED` to the database:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

The `isManual` flag is a user-controlled field supplied at transaction creation time via `CreateTransactionDto`:

```typescript
@IsOptional()
@IsBoolean()
isManual?: boolean;
``` [3](#0-2) 

It is stored directly on the `Transaction` entity and is never reset to `false` after execution completes. The chain service's `_executeTransaction` only writes `status`, `executedAt`, and `statusCode` — it does not touch `isManual`: [4](#0-3) 

Therefore, after a manual transaction reaches `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, or `REJECTED`, its creator can still call `PATCH /transactions/archive/:id` and successfully transition it to `ARCHIVED`.

The `ARCHIVED` state is treated as a terminal "done" state in the history collection: [5](#0-4) 

### Impact Explanation

- An `EXECUTED` transaction — which represents a confirmed, on-chain Hedera operation — can be silently moved to `ARCHIVED` by its creator, removing it from the normal `EXECUTED` history bucket and placing it in `ARCHIVED`. This corrupts the audit trail and allows a user to obscure the fact that a transaction was executed.
- A `FAILED` or `EXPIRED` transaction can similarly be archived, hiding failure evidence.
- A `CANCELED` transaction (which may have been canceled by an admin or through a group operation) can be re-classified as `ARCHIVED` by the original creator, undermining the cancellation record.

### Likelihood Explanation

The attack requires no privileged access. Any authenticated user can:
1. Create a transaction with `isManual: true` (a standard, documented field in the API).
2. Wait for the transaction to reach any terminal state.
3. Call `PATCH /transactions/archive/:id` as the creator.

The endpoint is reachable by any verified user and the `isManual` flag is set at creation time with no restrictions. [6](#0-5) 

### Recommendation

The status check must be unconditional. The `isManual` flag should not gate the status guard. The correct condition is:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If the intent was to allow manual transactions in `WAITING_FOR_EXECUTION` to be archived (which is already covered by the status list), the `!transaction.isManual` clause is entirely redundant and should be removed.

### Proof of Concept

1. Authenticate as a normal user and create a transaction with `isManual: true`:
   ```
   POST /transactions
   { ..., "isManual": true }
   ```
2. The chain service executes the transaction; its status becomes `EXECUTED`.
3. Call:
   ```
   PATCH /transactions/archive/<id>
   ```
   as the creator.
4. The guard at lines 711–718 evaluates: status check fails (`EXECUTED` not in list) **but** `!transaction.isManual` is `false` (since `isManual` is `true`), so the `&&` is `false` and no exception is thrown.
5. The transaction status is written as `ARCHIVED`, confirmed by querying the database — the `EXECUTED` record is gone from the execution history. [7](#0-6)

  /* Archive the transaction if the transaction is sign only. */
  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
