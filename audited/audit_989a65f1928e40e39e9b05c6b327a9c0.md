### Title
Unbounded Full-Table Scan in `getTransactionsToSign()` Enables Authenticated DoS

### Summary
`getTransactionsToSign()` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no `take`/`limit` clause, then performs a per-transaction async `userKeysToSign()` call inside a loop. Any authenticated user can trigger this endpoint, causing the server to execute an unbounded number of sequential database queries proportional to the total number of active transactions in the system. An attacker who creates many transactions (or exploits an already-large dataset) can degrade or crash the API service.

### Finding Description

**Root cause — missing `take` on the initial fetch:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` builds a `whereForUser` filter that excludes only terminal statuses (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED). It then calls `this.repo.find()` with **no `take` parameter**, loading every matching row:

```typescript
// lines 295-299 — no take/limit applied
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

It then iterates over every returned transaction and issues an async `userKeysToSign()` call per row:

```typescript
// lines 301-308 — N sequential async DB calls
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

Pagination is applied only **after** the full scan:

```typescript
// lines 311-316 — pagination is post-hoc, not at DB level
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
```

**Attacker-controlled growth path:**

The `whereForUser` filter contains **no user-scoping clause** — it matches all non-terminal transactions system-wide. Any authenticated user can:
1. Register an account (standard sign-up flow).
2. Create a large number of transactions via `POST /transactions` (each is a valid API call requiring only a valid `creatorKeyId` and `transactionBytes`).
3. Repeatedly call `GET /transactions/sign` — each call loads all active transactions and fires N async DB queries.

The `PaginationParams` decorator enforces `size ≤ 100` on the response, but this guard is irrelevant because the unbounded work happens before pagination is applied.

**Contrast with properly paginated endpoints:**

`getTransactions()` and `getHistoryTransactions()` both pass `skip: offset, take: limit` directly into the TypeORM `FindManyOptions`, so the database enforces the page size. `getTransactionsToSign()` does not.

### Impact Explanation

- **Availability:** Each request to `GET /transactions/sign` causes the API process to issue O(N) sequential async database queries where N is the total number of non-terminal transactions. With thousands of transactions, each request can hold a DB connection pool slot and CPU for seconds to minutes, starving other users.
- **Memory:** All N transaction rows are loaded into Node.js heap simultaneously before any filtering.
- **Cascading:** Because `userKeysToSign()` itself performs additional DB lookups (key resolution, approver tree traversal), the actual DB query count per request is a multiple of N.
- **Severity:** Medium-High. The endpoint is authenticated, limiting the attacker surface, but any verified user can exploit it. The service can be degraded without any admin-level access.

### Likelihood Explanation

- Any registered, verified user can reach `GET /transactions/sign` — it is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`.
- Transaction creation (`POST /transactions`) is also available to any verified user with a registered key.
- An attacker needs only one account and the ability to create transactions in bulk. No privileged role is required.
- The attack is repeatable and deterministic: more transactions → longer response time → eventual timeout/OOM.

### Recommendation

Apply the `take`/`skip` limit at the database query level, not after in-memory processing. The correct fix mirrors how `getTransactions()` works:

```typescript
// Apply limit at DB level, then filter in-memory only within that page
const [transactions, total] = await this.repo.findAndCount({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  skip: offset,
  take: limit,
});
```

Because `userKeysToSign()` is needed to determine *which* transactions require the user's signature, a more robust fix is to push the key-matching logic into the SQL query (as the newer `transaction-nodes` endpoint does via `getTransactionNodesQuery`), eliminating the in-memory loop entirely.

Additionally, enforce a rate limit on `GET /transactions/sign` per authenticated user.

### Proof of Concept

1. Register and verify a user account via the standard API flow.
2. Obtain a JWT token via `POST /auth/login`.
3. Create a registered `UserKey` for the account.
4. In a loop, call `POST /transactions` with valid `transactionBytes` and `creatorKeyId` to create N transactions (e.g., N = 10,000). Each transaction will remain in `WAITING_FOR_SIGNATURES` status.
5. Call `GET /transactions/sign?page=1&size=10`.
6. Observe that the server loads all N transactions from the database and calls `userKeysToSign()` N times before returning the first 10 results. Response time grows linearly with N. At large N, the request times out or the Node.js process runs out of memory/DB connections.
