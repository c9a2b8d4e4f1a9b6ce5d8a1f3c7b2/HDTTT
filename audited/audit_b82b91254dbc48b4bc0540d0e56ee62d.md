### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Malicious Signer to Corrupt Multi-Sig Transactions

---

### Summary

The `POST /transactions/:id/signers` endpoint routes through `SignersService.uploadSignatureMaps` → `processTransactionSignatures`, which calls `sdkTransaction.addSignature(publicKey, map)` without first cryptographically verifying that the submitted signature bytes are valid for the given public key and transaction body. A parallel code path — `TransactionsService.importSignatures` — correctly calls `validateSignature` before `addSignature`. Any authenticated user who is a listed signer on a transaction can submit their own registered public key paired with arbitrary/invalid signature bytes. The system will accept the submission, record the user as a signer, update the stored transaction bytes with the invalid signature, and potentially advance the transaction to `WAITING_FOR_EXECUTION`. When the chain service submits the transaction, the Hedera network rejects it, permanently corrupting the transaction.

---

### Finding Description

**Root cause — missing `validateSignature` call in the signers upload path**

`validateSignature` in `back-end/libs/common/src/utils/sdk/transaction.ts` is the only place in the codebase that performs a real cryptographic check (`publicKey.verify(bodyBytes, signature)`): [1](#0-0) 

The `importSignatures` path in `TransactionsService` correctly calls this function before calling `addSignature`: [2](#0-1) 

However, the primary signer upload path — `SignersService.processTransactionSignatures` — skips `validateSignature` entirely. It only checks whether the public key belongs to the authenticated user, then immediately calls `addSignature`: [3](#0-2) 

The Hedera SDK's `addSignature` does not perform cryptographic verification; it blindly appends whatever bytes are in the map to the transaction's signed-transaction list. The `_signerPublicKeys` set is populated by `addSignature` regardless of whether the signature is valid, so the downstream execution check in `execute.service.ts`: [4](#0-3) 

…also passes, because it only checks key presence, not cryptographic validity.

**Attacker-controlled entry point**

The `POST /transactions/:id/signers` endpoint is reachable by any JWT-authenticated, verified user: [5](#0-4) 

The `IsSignatureMap` decorator validates only structural format (account ID format, transaction ID format, non-empty byte length) — it performs no cryptographic check: [6](#0-5) 

**Exploit flow**

1. Attacker is a registered user with a valid key pair (`publicKey A / privateKey A`) listed as a signer on transaction T.
2. Attacker constructs a `SignatureMap` containing `publicKey A` paired with 64 or 96 bytes of arbitrary garbage (e.g., all zeros) — passes the `decodedSignature.length === 0` guard.
3. Attacker POSTs this to `POST /transactions/T/signers`.
4. `processTransactionSignatures` finds `publicKey A` in `userKeyMap` (ownership check passes), calls `sdkTransaction.addSignature(publicKey A, map)` — invalid bytes are written into the transaction.
5. A `TransactionSigner` record is inserted for the attacker; `isSameBytes` is `false` so the corrupted `transactionBytes` are persisted to the database.
6. `processTransactionStatus` runs; if the threshold is now met, the transaction advances to `WAITING_FOR_EXECUTION`.
7. The chain service submits the transaction to the Hedera network; the network rejects it with `INVALID_SIGNATURE`.
8. The transaction enters `FAILED` state. The original valid bytes are overwritten and unrecoverable.

---

### Impact Explanation

- **Transaction integrity destruction**: A multi-sig transaction that has collected legitimate signatures from other parties is permanently corrupted. The stored `transactionBytes` are overwritten with invalid signature data and cannot be recovered.
- **Threshold bypass / false advancement**: If the attacker's fake signature is the last one needed to meet the signing threshold, the transaction is prematurely advanced to `WAITING_FOR_EXECUTION` and submitted to the network, where it fails.
- **Denial of service on transaction lifecycle**: The transaction enters `FAILED` state. All prior legitimate signers' work is wasted. Depending on the transaction's `validStart` window, re-creation may not be possible.
- **False audit trail**: The attacker is recorded as a legitimate signer in the `TransactionSigner` table despite never producing a valid signature.

---

### Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated user with a registered key pair who is listed as a signer or observer on the target transaction. This is a normal, non-privileged role in the system.
- **No special tooling required**: The attacker only needs to craft a JSON body with their own DER public key and arbitrary hex bytes as the signature value. Standard HTTP client suffices.
- **No cryptographic break required**: The attack exploits a missing check, not a weakness in any cryptographic primitive.
- **Realistic scenario**: In an organization using this tool for governance or treasury transactions, a disgruntled or malicious signer can sabotage any transaction they are listed on.

---

### Recommendation

Add a call to `validateSignature` inside `processTransactionSignatures` in `signers.service.ts`, before calling `addSignature`, mirroring the pattern already used in `TransactionsService.importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const signatureValid = publicKey.verify(bodyBytes, signatureBytes);
if (!signatureValid) {
  throw new Error(ErrorCodes.ISNMPN); // or a dedicated error code
}
```

Alternatively, call the existing `validateSignature` utility on the full map before iterating:

```typescript
import { validateSignature } from '@app/common';

// At the top of processTransactionSignatures:
validateSignature(sdkTransaction, map); // throws on invalid signature
```

This aligns the signers upload path with the already-correct `importSignatures` path and ensures no invalid signature bytes are ever persisted to the database.

### Proof of Concept

**Preconditions**: Attacker has a valid account, a registered key pair (`publicKey A`), and is listed as a signer on transaction T (status `WAITING_FOR_SIGNATURES`).

**Steps**:

```bash
# 1. Construct a signature map with publicKey A and 64 zero bytes as the "signature"
FAKE_SIG=$(python3 -c "print('0x' + '00'*64)")

# 2. POST to the signers endpoint
curl -X POST https://<host>/api/transactions/T/signers \
  -H "Authorization: Bearer <attacker_jwt>" \
  -H "Content-Type: application/json" \
  -d "[{
    \"id\": <T>,
    \"signatureMap\": {
      \"<nodeAccountId>\": {
        \"<transactionId>\": {
          \"<publicKeyA_DER>\": \"$FAKE_SIG\"
        }
      }
    }
  }]"
```

**Expected (correct) behavior**: Server returns `400` with `ISNMPN` — invalid signature.

**Actual behavior**: Server returns `201`. A `TransactionSigner` record is created for the attacker. The transaction's `transactionBytes` in the database are updated to include the invalid signature. If the threshold is met, the transaction advances to `WAITING_FOR_EXECUTION` and is submitted to the Hedera network, where it is rejected with `INVALID_SIGNATURE`, permanently failing the transaction.
