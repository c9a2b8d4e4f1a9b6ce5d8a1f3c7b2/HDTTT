### Title
`getTransactionsToSign` Fetches and Processes All Active Transactions Without Bound, Causing Server Resource Exhaustion

### Summary

`getTransactionsToSign` in the API service loads every non-terminal transaction from the database in a single unbounded query, then iterates over all of them in a sequential `for...of` loop making one async DB call per transaction via `userKeysToSign`. Pagination is applied only after the full in-memory scan. Any authenticated user can trigger this endpoint (`GET /transactions/sign`) to exhaust server memory, DB connection pool, and CPU, degrading or hanging the API service for all users.

### Finding Description

**Root cause — no `take` limit on the DB query:**

In `getTransactionsToSign`, the `repo.find()` call at line 295 has no `take` field, unlike every other paginated query in the same file (e.g., `getTransactions` at line 180 passes `take: limit`):

```ts
// back-end/apps/api/src/transactions/transactions.service.ts:295-299
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` — loads ALL non-terminal transactions
});
``` [1](#0-0) 

**Unbounded sequential async loop:**

The code then iterates over every returned transaction, making one async `userKeysToSign` call per row:

```ts
// back-end/apps/api/src/transactions/transactions.service.ts:301-309
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

**Pagination applied only after full scan:**

The `limit` and `offset` from `PaginationParams` are used only to slice the already-built in-memory result:

```ts
// back-end/apps/api/src/transactions/transactions.service.ts:311-316
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [3](#0-2) 

The `PaginationParams` decorator enforces `size <= 100`, but this only limits the final slice — it does not bound the DB query or the loop. [4](#0-3) 

**Reachable entry point:**

The endpoint is still active and requires only a valid authenticated session:

```ts
// back-end/apps/api/src/transactions/transactions.controller.ts:165-178
@Get('/sign')
@Serialize(withPaginatedResponse(TransactionToSignDto))
getTransactionsToSign(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
  ...
) {
  return this.transactionsService.getTransactionsToSign(user, paginationParams, sort, filter);
}
``` [5](#0-4) 

The comment "NO LONGER USED BY FRONT-END" confirms the front-end has moved away from it, but the route remains live and unenforced.

### Impact Explanation

- **Memory exhaustion**: All non-terminal transactions (with their `groupItem` relations) are loaded into the Node.js heap at once. In a large organization with thousands of active transactions, this can exhaust heap memory and crash the API process.
- **DB connection starvation**: The sequential `await userKeysToSign(...)` loop holds a DB connection (or issues N queries) for the entire duration of the request, starving other concurrent requests of DB connections.
- **Request timeout / API hang**: A single call with a large transaction set can block the event loop long enough to cause HTTP timeouts for all concurrent users.
- **Severity**: High — results in service unavailability for all users of the API, triggered by a single authenticated request.

### Likelihood Explanation

- Any registered, authenticated user can call `GET /transactions/sign?page=1&size=1`.
- No admin or privileged role is required.
- In an organization with many active transactions (e.g., bulk transaction workflows), the condition is naturally met without any attacker-specific setup.
- An attacker can also deliberately create many transactions (via `POST /transactions`) to inflate the table before triggering the endpoint.

### Recommendation

Apply the `take: limit` and `skip: offset` constraints directly in the `repo.find()` call, and move the key-eligibility check into the SQL query (as a join or subquery), mirroring the pattern used by `getTransactionsToApprove`:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  skip: offset,
  take: limit,   // ← add this
});
```

Alternatively, push the `userKeysToSign` logic into a SQL predicate so the DB filters eligible rows before returning them, eliminating the N+1 loop entirely.

### Proof of Concept

1. Register and authenticate as a normal user.
2. Create N transactions (e.g., N = 10,000) via `POST /transactions` with `status = WAITING_FOR_SIGNATURES`.
3. Issue a single request: `GET /transactions/sign?page=1&size=1`.
4. The server loads all N transactions from the DB, then executes N sequential async `userKeysToSign` calls.
5. Observe: the request hangs for an extended period (proportional to N), DB connection pool is saturated, and concurrent API requests from other users time out or are rejected.
