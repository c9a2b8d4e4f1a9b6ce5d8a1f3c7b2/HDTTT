### Title
Missing Input Validation on CSV-Supplied `txValidDuration` and `transactionFee` Causes Entire Transaction Groups to Fail on Submission

### Summary
In `ImportCSVController.vue`, the `txValidDuration` and `transactionFee` fields are read verbatim from a user-supplied CSV file and passed directly to `setTransactionValidDuration()` and `setMaxTransactionFee()` with no bounds or sanity checks. A malicious or malformed CSV can set these to values that cause every transaction in the imported group to be rejected by the Hedera network, silently wasting the user's signing effort and making the entire batch unexecutable.

### Finding Description

In `handleImportCsv()`, both parameters are captured from CSV rows without validation: [1](#0-0) 

They are then used directly when constructing each `TransferTransaction`: [2](#0-1) 

**`txValidDuration` attack path:**
- Hedera enforces `transactionValidDuration` in the range `[1, 180]` seconds.
- A CSV row with `transaction valid duration, 0` causes `Number.parseInt("0")` = `0` (the string `"0"` is truthy, so the fallback `180` is never used). `setTransactionValidDuration(0)` produces a transaction that is immediately expired before it can be submitted.
- A CSV row with `transaction valid duration, 99999` produces a duration the Hedera network rejects outright (`TRANSACTION_EXPIRED` or `INVALID_TRANSACTION_DURATION`).
- A CSV row with `transaction valid duration, abc` causes `Number.parseInt("abc")` = `NaN`, which is passed to the SDK and may throw or produce an invalid transaction body.

**`transactionFee` attack path:**
- A CSV row with `transaction fee, 0` creates `new Hbar("0", HbarUnit.Tinybar)` = 0 ℏ max fee. Every transaction in the group is submitted with a zero max fee and the Hedera network rejects all of them with `INSUFFICIENT_TX_FEE`.
- A CSV row with `transaction fee, -1` creates a negative Hbar value; the SDK may throw or produce a malformed transaction.

No validation exists anywhere in the import pipeline for these two fields. The normal UI path (`BaseTransaction.vue`) does check that `maxTransactionFee > 0`: [3](#0-2) 

But the CSV import path bypasses this guard entirely.

### Impact Explanation

Every transaction in the imported group is built with the invalid parameter. After the user reviews and signs the group, all submissions to the Hedera network fail. The user's signing effort is wasted, the transaction group is unexecutable, and the user receives no clear error at import time explaining why. In an organizational multi-signature workflow this also wastes the signing effort of all co-signers who approved the group before execution is attempted.

### Likelihood Explanation

The attack requires delivering a crafted CSV to a user who then imports it via the normal "Import CSV" workflow. This is a realistic social-engineering vector (e.g., a malicious file sent by email or shared via a collaboration tool). The feature is explicitly designed to accept external CSV files, so no privileged access is required — only the ability to influence which file the user imports.

### Recommendation

Add explicit validation of both fields immediately after they are parsed from the CSV, before any `TransferTransaction` is constructed:

1. **`txValidDuration`**: Reject or clamp to `[1, 180]`. If the parsed value is `NaN`, ≤ 0, or > 180, emit a `toastManager.error` and return `null` (matching the pattern already used for invalid account IDs).
2. **`transactionFee`**: Reject if the parsed tinybar value is ≤ 0 or not a finite number. Emit a `toastManager.error` and return `null`.

Apply these checks in `handleImportCsv()` in `ImportCSVController.vue` at the point where `txValidDuration` and `transactionFee` are assigned from `rowInfo[1]`.

### Proof of Concept

Craft a CSV file with the following header section:

```
transaction description, Test Group
sender account, 0.0.1234
fee payer account, 0.0.1234
sending time, 12:00:00
transaction fee, 0
transaction valid duration, 0
memo, test
```

followed by a normal transfer data row. Import this CSV via the "Import CSV" feature in the Create Transaction Group page. The import completes successfully with a "Import complete" toast. The user signs the group. On submission, every transaction is rejected by the Hedera network with `INSUFFICIENT_TX_FEE` (due to 0 max fee) and/or `INVALID_TRANSACTION_DURATION` (due to 0 valid duration). No error is surfaced at import time. [4](#0-3)
