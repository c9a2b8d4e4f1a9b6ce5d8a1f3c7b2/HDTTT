### Title
Transaction Creator Can Bypass Approval Governance by Modifying Approvers Mid-Lifecycle

### Summary
The `ApproversService` in the Hedera Transaction Tool allows the transaction creator to add, remove, or replace approvers — and reduce approval thresholds — at any point during the transaction lifecycle, including after approvals have already been collected. Because `getCreatorsTransaction` only verifies creator identity and never checks the transaction's current status, a creator can swap out a non-cooperating approver for a colluding one, or reduce the threshold after partial approvals, forcing a transaction to execution without genuine sign-off from the originally designated oversight parties.

### Finding Description

The root cause is in `getCreatorsTransaction`, which is the sole authorization gate for all three approver-mutation endpoints: [1](#0-0) 

It only checks that the caller is the creator. It performs **no check on `transaction.status`**. All three mutation paths rely on it exclusively:

**1. `createTransactionApprovers`** — adds new approvers at any lifecycle stage: [2](#0-1) 

**2. `updateTransactionApprover`** — replaces an approver's `userId` (clearing their existing signature/approval) or reduces the `threshold` of a tree node: [3](#0-2) [4](#0-3) [5](#0-4) 

**3. `removeTransactionApprover`** (controller) — removes an approver with no status guard: [6](#0-5) 

**Attack path A — Approver swap:**
1. Creator creates a transaction requiring approval from User A (an oversight party), threshold 1/1.
2. User A reviews and refuses to approve.
3. Creator calls `PUT /transactions/:id/approvers/:approverId` with `{ userId: colludingUserB }`.
4. The update clears `signature`, `userKeyId`, and `approved` for that slot and assigns it to User B.
5. User B immediately approves.
6. Transaction advances to `WAITING_FOR_EXECUTION` without genuine approval from User A.

**Attack path B — Threshold reduction:**
1. Creator creates a transaction with a tree approver requiring 3/3 approvals.
2. Two approvers approve; the third refuses.
3. Creator calls `PUT /transactions/:id/approvers/:treeId` with `{ threshold: 2 }`.
4. The threshold check only validates `threshold <= approvers.length && threshold != 0` — no status guard.
5. The transaction now satisfies the (reduced) threshold and proceeds to execution. [5](#0-4) 

### Impact Explanation

The approval mechanism is the primary governance control ensuring designated parties sign off before a transaction is submitted to the Hedera network. Bypassing it allows the creator to unilaterally execute transactions (e.g., fund transfers, account key updates, node changes) that were explicitly designed to require multi-party consent. This is a direct integrity failure in the trust model: the approval step provides no real security guarantee if the creator can retroactively rewrite who the approvers are or lower the required count.

### Likelihood Explanation

The attacker is the transaction creator — a normal, unprivileged user. No leaked credentials, admin access, or external collusion is required. The exploit requires only two sequential API calls: one to mutate the approver (swap or threshold reduction) and one for the colluding user to approve. Any creator who faces a non-cooperating approver can exploit this immediately.

### Recommendation

Add a transaction status guard inside `getCreatorsTransaction` (or at the start of each mutation function) that rejects modifications once the transaction has left the `NEW` or `WAITING_FOR_SIGNATURES` state. Specifically, approver mutations should be forbidden when `status` is `WAITING_FOR_EXECUTION`, `EXECUTED`, `EXPIRED`, `CANCELED`, or `ARCHIVED`. For example:

```typescript
async getCreatorsTransaction(transactionId, user, entityManager?) {
  // ... existing lookup ...
  const terminalOrLockedStatuses = [
    TransactionStatus.WAITING_FOR_EXECUTION,
    TransactionStatus.EXECUTED,
    TransactionStatus.EXPIRED,
    TransactionStatus.CANCELED,
    TransactionStatus.ARCHIVED,
  ];
  if (terminalOrLockedStatuses.includes(transaction.status))
    throw new BadRequestException('Cannot modify approvers after transaction is locked');
  // ... existing creator check ...
}
```

### Proof of Concept

**Setup:** Transaction T requires approval from User A (threshold 1/1). User A is an oversight party who refuses to approve.

**Steps:**
1. Creator authenticates and calls:
   ```
   PUT /transactions/T/approvers/<approverIdForUserA>
   Body: { "userId": <colludingUserBId> }
   ```
   `updateTransactionApprover` calls `getCreatorsTransaction` (passes — creator identity only), then executes: [7](#0-6) 
   User A's `signature`, `userKeyId`, and `approved` are cleared; slot now belongs to User B.

2. User B authenticates and calls:
   ```
   POST /transactions/T/approvers/approve
   Body: { "userKeyId": <userBKeyId>, "signature": <validSig>, "approved": true }
   ```
   `approveTransaction` finds User B in the approver list, verifies the signature, and records approval. [8](#0-7) 

3. `emitTransactionStatusUpdate` fires; the status engine sees 1/1 approvals satisfied and advances T to `WAITING_FOR_EXECUTION`.

**Result:** Transaction T executes on the Hedera network without User A's approval, defeating the governance control entirely.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
