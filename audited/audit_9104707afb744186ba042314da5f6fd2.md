### Title
Transaction Approval Bypass via Unrestricted Approver Removal Without Minimum Threshold Enforcement

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` contains no minimum-threshold guard and no service-level authorization check. A transaction creator (a normal authenticated user) can remove every approver from a transaction one by one, leaving zero approvers. After the last removal, `emitTransactionStatusUpdate` fires and the chain service re-evaluates approval status against an empty approver set. Because JavaScript's `Array.prototype.every()` returns `true` on an empty array, any downstream check of the form `approvers.every(a => a.approved)` vacuously passes, allowing the transaction to advance to execution without any approval — directly bypassing the multi-party governance the approver system is designed to enforce.

### Finding Description

**Root cause — missing minimum threshold check in `removeTransactionApprover`:** [1](#0-0) 

The function accepts only an approver `id`; it carries no `user: User` parameter and calls neither `getCreatorsTransaction` nor any equivalent guard. Compare with `createTransactionApprovers` and `updateTransactionApprover`, both of which accept a `User` argument and explicitly call `getCreatorsTransaction` to enforce creator-only access: [2](#0-1) [3](#0-2) 

After `removeNode` soft-deletes the approver subtree, `emitTransactionStatusUpdate` is unconditionally emitted: [4](#0-3) 

This triggers the chain service to re-evaluate the transaction's approval state. With zero remaining approvers, `getApproversByTransactionId` returns `[]`: [5](#0-4) 

The frontend `isApproved` utility illustrates the vacuous-truth problem: when `approver.approvers` is an empty array, `approvals.length` (0) is compared against `approver.threshold || approvals.length` — if threshold is 0 or undefined, the condition `0 >= 0` is `true`, returning `approved`: [6](#0-5) 

The `TransactionApprover` entity allows `threshold` and `userId` to be `null`, meaning a root node with no children and no threshold is a valid database state after all approvers are removed: [7](#0-6) 

**Exploit path:**
1. Attacker (transaction creator, a normal user) creates a transaction and adds one or more approvers via `POST /transactions/:id/approvers`.
2. Before approvers sign, the creator calls `DELETE /transactions/approvers/:approverId` for each approver in sequence.
3. No minimum-threshold guard exists; each call succeeds.
4. After the last deletion, `emitTransactionStatusUpdate` fires with the transaction ID.
5. The chain service evaluates approval status against an empty approver list; vacuous-truth evaluation marks the transaction as fully approved.
6. The transaction advances to `WAITING_FOR_EXECUTION` or `EXECUTED` without any approver signature.

### Impact Explanation
The approver system is the sole multi-party governance control for organizational Hedera transactions (account updates, file operations, transfers). Bypassing it allows a transaction creator to execute any transaction — including high-value transfers or account key changes — without the required co-signers. This constitutes unauthorized state change and potential direct asset movement, the highest-severity impact category in the RESEARCHER playbook.

### Likelihood Explanation
The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The exploit requires only repeated calls to a standard REST endpoint (`DELETE /transactions/approvers/:id`) that the creator already has legitimate access to. No leaked secrets, no admin keys, and no race conditions are required. The path is fully deterministic and reproducible.

### Recommendation
1. **Enforce minimum threshold in `removeTransactionApprover`**: After soft-deleting the node, count remaining root-level approvers for the transaction. If the count would reach zero, reject the request with a `BadRequestException`.
2. **Add service-level authorization**: Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` (matching the pattern in `createTransactionApprovers` and `updateTransactionApprover`) to ensure only the transaction creator can remove approvers.
3. **Guard status re-evaluation**: In the chain service's approval-status handler, treat an empty approver list as "not yet approved" (pending), not as vacuously approved.

### Proof of Concept

```
# Step 1 – Creator creates a transaction with two approvers
POST /transactions/1/approvers
Body: { approversArray: [{ userId: 2 }, { userId: 3 }] }
→ 201 Created, approverIds: [10, 11]

# Step 2 – Creator removes first approver (no minimum check)
DELETE /transactions/approvers/10
→ 200 OK

# Step 3 – Creator removes second approver (no minimum check, zero approvers remain)
DELETE /transactions/approvers/11
→ 200 OK
→ emitTransactionStatusUpdate fires with transactionId: 1

# Step 4 – Chain service re-evaluates: getApproversByTransactionId(1) returns []
# every() on [] → true → transaction marked approved
# Transaction advances to WAITING_FOR_EXECUTION without any approver signature
```

Expected result without the fix: transaction executes with zero approvals.
Expected result with the fix: `DELETE /transactions/approvers/11` returns `400 Bad Request – at least one approver must remain`.

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
