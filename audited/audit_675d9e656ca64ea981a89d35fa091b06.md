### Title
Unbounded `reminderMillisecondsBefore` Causes Immediate Reminder Notification Bypass via Arithmetic Underflow in `createTransactions`

### Summary
`CreateTransactionDto.reminderMillisecondsBefore` accepts any numeric value with no upper or lower bound validation. In `transactions.service.ts`, the expression `tx.validStart.getTime() - dto.reminderMillisecondsBefore` is computed without guards. Supplying a value larger than `validStart.getTime()` produces a negative Unix timestamp, which Redis's `PXAT` option treats as already-expired, causing the reminder notification to fire immediately upon transaction creation — bypassing the intended "remind N ms before valid start" semantics entirely.

### Finding Description

**Root cause — missing bounds on DTO field:**

`CreateTransactionDto` declares `reminderMillisecondsBefore` with only `@IsNumber()` and `@IsOptional()`. No `@Min(1)` or `@Max()` constraint exists. [1](#0-0) 

**Unchecked arithmetic in service:**

In `createTransactions`, the reminder date is computed directly from user-supplied input:

```typescript
const remindAt = new Date(tx.validStart.getTime() - dto.reminderMillisecondsBefore);
``` [2](#0-1) 

If `dto.reminderMillisecondsBefore` > `tx.validStart.getTime()` (e.g., attacker sends `9007199254740991` — `Number.MAX_SAFE_INTEGER`), then `remindAt` becomes a `Date` with a large negative millisecond value (a timestamp far in the past).

**Scheduler stores the poisoned date in Redis:**

`SchedulerService.addReminder` passes `date.getTime()` directly as the Redis `PXAT` argument:

```typescript
await this.pubClient.set(key, key, 'PXAT', date.getTime());
``` [3](#0-2) 

Redis `PXAT` interprets the value as a Unix timestamp in milliseconds. A negative or past timestamp causes the key to expire immediately, firing the keyspace notification at once.

**Immediate notification dispatch:**

`ReminderHandlerService.handleTransactionReminder` is invoked by the keyspace event. It checks transaction status and deduplication, then calls `emitTransactionRemindSigners` — sending a "please sign" notification to all signers the moment the transaction is created, not at the intended reminder time. [4](#0-3) 

**Exploit flow:**
1. Authenticated organization user POSTs to `POST /transactions` with `reminderMillisecondsBefore: 9007199254740991`.
2. Server computes `remindAt = new Date(validStart - 9007199254740991)` → large negative timestamp.
3. Redis key is set with `PXAT = <negative>` → expires immediately.
4. Keyspace notification fires → `handleTransactionReminder` runs → `emitTransactionRemindSigners` dispatches notifications to all signers immediately.
5. Attacker can repeat this for every transaction they create, flooding signers with spurious notifications.

### Impact Explanation
Every signer assigned to any transaction created with a crafted `reminderMillisecondsBefore` receives an immediate, unsolicited "sign this transaction" notification. At scale, an attacker can create many transactions in rapid succession, generating a notification storm that degrades the notification system and desensitizes signers to legitimate reminders — undermining the integrity of the multi-signature workflow. The deduplication check (line 51–58 of `reminder-handler.service.ts`) only prevents a second reminder for the same transaction, not the initial spurious one.

### Likelihood Explanation
Any authenticated organization member can reach `POST /transactions` with a crafted body. No privilege escalation is required. The API accepts arbitrary numeric values for `reminderMillisecondsBefore` because the DTO only enforces `@IsNumber()`. The attack requires no special tooling — a single crafted HTTP request suffices.

### Recommendation
Add `@Min(1)` and a reasonable `@Max` (e.g., `@Max(7 * 24 * 60 * 60 * 1000)` for one week) to `reminderMillisecondsBefore` in `CreateTransactionDto`:

```typescript
@IsOptional()
@IsNumber()
@Min(1)
@Max(7 * 24 * 60 * 60 * 1000) // max 1 week
reminderMillisecondsBefore?: number;
```

Additionally, add a server-side guard in `createTransactions` before computing `remindAt`:

```typescript
const remindAt = new Date(tx.validStart.getTime() - dto.reminderMillisecondsBefore);
if (remindAt.getTime() <= Date.now()) {
  // skip scheduling — reminder time is already in the past
  return null;
}
``` [1](#0-0)

**Expected (correct) behavior:** reminder fires N ms before `validStart`.

**Observed (buggy) behavior:** `remindAt = new Date(validStart.getTime() - 9007199254740991)` → negative timestamp → Redis key expires immediately → `emitTransactionRemindSigners` fires at transaction creation time, sending spurious notifications to all assigned signers.
