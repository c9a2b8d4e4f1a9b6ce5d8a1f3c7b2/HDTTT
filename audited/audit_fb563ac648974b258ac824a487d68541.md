### Title
Unbounded Recursive Approver Tree Submission Causes Server-Side DoS via Resource Exhaustion

### Summary
The `POST /transactions/:transactionId/approvers` endpoint accepts an arbitrarily large and arbitrarily deep nested approver tree with no maximum size or depth constraint. The service processes each node of this tree recursively, executing 4–6 synchronous database queries per node inside a single database transaction. A malicious authenticated user can submit a crafted payload with thousands of nested nodes, exhausting the database connection pool and causing service unavailability for all users.

### Finding Description

**Root Cause — Missing `@ArrayMaxSize` on both DTO fields:**

`CreateTransactionApproversArrayDto.approversArray` and `CreateTransactionApproverDto.approvers` carry only `@ArrayMinSize(1)` and no upper bound: [1](#0-0) 

Neither `approversArray` (the top-level list) nor the recursive `approvers` field has an `@ArrayMaxSize` decorator. There is also no depth limit enforced anywhere in the DTO or service.

**Exploit Path — Recursive DB-heavy processing per node:**

`createTransactionApprovers` defines an inner async function `createApprover` that is called recursively for every node in the submitted tree: [2](#0-1) 

For each node the function executes:
1. `isNode(...)` — a DB query to check for duplicates
2. `transactionalEntityManager.findOne(TransactionApprover, ...)` — parent lookup
3. `getRootNodeFromNode(...)` — a recursive SQL CTE query
4. `transactionalEntityManager.count(User, ...)` — user existence check
5. `getApproversByTransactionId(...)` — existing approver lookup
6. `transactionalEntityManager.insert(TransactionApprover, ...)` — DB write

All of this runs inside a single `dataSource.transaction(...)` block, holding one database connection open for the entire duration of the request.

**Entry Point — Reachable by any verified user:**

The controller applies only authentication guards; no rate-limiting guard is present on this route: [3](#0-2) 

The service additionally checks that the caller is the transaction creator: [4](#0-3) 

A malicious user only needs to: register an account, get verified, create one transaction, then POST a crafted approver tree.

**Secondary surface — `CreateTransactionObserversDto.userIds` also unbounded:** [5](#0-4) 

The observers endpoint iterates over `dto.userIds` with no upper bound, performing a DB check and a bulk `repo.save()` for an arbitrarily large list: [6](#0-5) 

### Impact Explanation
A single crafted HTTP request with a deeply nested or very wide approver tree (e.g., 1 000+ nodes) triggers thousands of synchronous database queries within one open transaction. This:
- Holds a database connection for the full duration of the request
- Saturates the database connection pool, blocking all other queries
- Causes request timeouts and HTTP 500 errors for all concurrent users
- Can be repeated by the same user (or multiple users) to sustain the outage

The result is complete service unavailability — a permanent denial of service until the server is restarted or the connection pool recovers.

### Likelihood Explanation
The attacker preconditions are minimal: a registered, verified account and one created transaction. Both are achievable through normal product flows with no privileged access. The payload is a standard JSON body. No special tooling is required — a single `curl` or HTTP client call suffices. The absence of any rate-limiting guard on the approvers controller means the attack can be repeated without throttling.

### Recommendation
1. **Add `@ArrayMaxSize` to both DTO fields** — enforce a reasonable upper bound (e.g., 50) on `approversArray` and on each nested `approvers` array in `CreateTransactionApproverDto`.
2. **Enforce a maximum recursion depth** — add a `depth` counter to `createApprover` and throw if it exceeds a safe limit (e.g., 5).
3. **Add `@ArrayMaxSize` to `CreateTransactionObserversDto.userIds`** — cap the number of observers that can be added in a single request.
4. **Apply a rate-limiting guard** (e.g., `UserThrottlerGuard`) to the approvers and observers controllers to limit request frequency per user.

### Proof of Concept

**Attacker steps:**

1. Register and verify a user account via the normal sign-up flow.
2. Create a transaction (any valid type) to obtain a `transactionId`.
3. POST the following payload to `POST /transactions/{transactionId}/approvers` with a valid JWT:

```json
{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                ... (repeat nesting 500+ levels deep, each with userId: <valid_user_id>)
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**Expected outcome:** The server opens a database transaction and executes 4–6 DB queries per level of nesting. With 500 levels, this is 2 000–3 000 synchronous DB queries in a single request, holding a connection open for seconds to minutes. Concurrent legitimate requests time out or fail. Repeating the request sustains the outage.

**Alternatively (wide tree):** Submit `approversArray` with 10 000 top-level entries (each `{ "userId": <valid_id> }`). The DTO validation passes (only `@ArrayMinSize(1)` is checked), and the service iterates all 10 000 entries with DB queries.

          this.validateApprover(dtoApprover);

          /* Check if the approver already exists */
          if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))

          if (typeof dtoApprover.listId === 'number') {

              where: { id: dtoApprover.listId },

            if (!parent) throw new Error(this.PARENT_APPROVER_NOT_FOUND);

            /* Check if the root transaction is the same */

            if (root?.transactionId !== transactionId)
              throw new Error(this.ROOT_TRANSACTION_NOT_SAME);
          }

          /* Check if the user exists */
          if (typeof dtoApprover.userId === 'number') {
            const userCount = await transactionalEntityManager.count(User, {
              where: { id: dtoApprover.userId },

            if (userCount === 0) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
          }

          /* Check if there are sub approvers */
          if (
            typeof dtoApprover.userId === 'number' &&
            dtoApprover.approvers &&
            dtoApprover.approvers.length > 0

            throw new Error(this.ONLY_USER_OR_TREE);

          /* Check if the approver has threshold when there are children */
          if (
            dtoApprover.approvers &&
            dtoApprover.approvers.length > 0 &&
            (dtoApprover.threshold === null || isNaN(dtoApprover.threshold))

            throw new Error(this.THRESHOLD_REQUIRED);

          /* Check if the approver has children when there is threshold */
          if (

          /* Check if the approver threshold is less or equal to the number of approvers */
