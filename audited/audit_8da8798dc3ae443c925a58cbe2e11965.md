### Title
`TransferTransactionModel` Ignores HTS Token and NFT Transfer Signing Accounts, Allowing Signature Bypass for Token Transfers

---

### Summary

`TransferTransactionModel.getSigningAccounts()` only iterates over `hbarTransfersList` and completely omits `tokenTransfersList` and `nftTransfersList`. This mirrors the external report's root cause exactly: a parser that handles one transfer variant (`transfer`/HBAR) while silently ignoring others (`transferFrom`/HTS tokens/NFTs). As a result, `computeSignatureKey` in `TransactionSignatureService` never adds the token-sender's key to the required signature set, so the backend will mark a `TransferTransaction` containing only HTS fungible-token or NFT debits as fully signed without the actual token owner's signature.

---

### Finding Description

`TransferTransaction` in the Hedera SDK carries three independent transfer lists:

| List | Covers |
|---|---|
| `hbarTransfersList` | Native HBAR debits/credits |
| `tokenTransfersList` | HTS fungible-token debits/credits |
| `nftTransfersList` | HTS NFT ownership changes |

Each debit entry in `tokenTransfersList` and each sender entry in `nftTransfersList` requires the owning account's key, exactly as a negative-amount HBAR entry does.

`TransferTransactionModel.getSigningAccounts()` only processes `hbarTransfersList`:

```typescript
// back-end/libs/common/src/transaction-signature/model/transfer-transaction.model.ts
getSigningAccounts(): Set<string> {
  const accounts = new Set<string>();
  for (const transfer of this.transaction.hbarTransfersList) {   // ← only HBAR
    if (transfer.amount.isNegative() && !transfer.isApproved) {
      accounts.add(transfer.accountId.toString());
    }
  }
  return accounts;   // tokenTransfersList / nftTransfersList never touched
}
```

`getReceiverAccounts()` has the same gap — it only reads `hbarTransfersList` for receiver-signature-required checks.

`TransactionSignatureService.computeSignatureKey()` calls `transactionModel.getSigningAccounts()` and `transactionModel.getReceiverAccounts()` and adds the resulting keys to the required `KeyList`. Because those sets are empty for pure-token transactions, the token-sender's key is never added.

The test suite confirms the blind spot — every test case in `transfer-transaction.model.spec.ts` mocks only `hbarTransfersList`; no test exercises `tokenTransfersList` or `nftTransfersList`.

---

### Impact Explanation

An authenticated organization user can craft a `TransferTransaction` that debits HTS tokens from **any account** (e.g., a treasury or another member's account) and submit it to the backend. Because `computeSignatureKey` returns a `KeyList` that does not include the token-owner's key, the backend's signature-completeness check will pass with only the fee-payer's signature. The chain service will then execute the transaction, draining tokens from the victim account without their consent.

---

### Likelihood Explanation

Any authenticated user of the organization backend can create and submit transactions. No privileged role is required. The `TransferTransaction` type is explicitly supported and listed in `TransactionType` enum. The attack path is direct: create a `TransferTransaction` with a `tokenTransfersList` debit from a victim account, sign only as fee payer, upload the signature map, and the backend will treat it as ready for execution.

---

### Recommendation

Extend `getSigningAccounts()` to also iterate `tokenTransfersList` (negative amounts, non-approved) and `nftTransfersList` (senders), mirroring the existing HBAR logic:

```typescript
getSigningAccounts(): Set<string> {
  const accounts = new Set<string>();

  for (const transfer of this.transaction.hbarTransfersList) {
    if (transfer.amount.isNegative() && !transfer.isApproved) {
      accounts.add(transfer.accountId.toString());
    }
  }

  // Add token transfer senders
  for (const transfer of this.transaction.tokenTransfersList) {
    if (transfer.amount < 0 && !transfer.isApproved) {
      accounts.add(transfer.accountId.toString());
    }
  }

  // Add NFT transfer senders
  for (const transfer of this.transaction.nftTransfersList) {
    if (!transfer.isApproved) {
      accounts.add(transfer.senderAccountId.toString());
    }
  }

  return accounts;
}
```

Apply the same fix to `getReceiverAccounts()` for token/NFT receivers that have `receiverSignatureRequired`.

```typescript
  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
