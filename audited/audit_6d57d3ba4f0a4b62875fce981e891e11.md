### Title
Missing Access Control on Transaction Comments Endpoints Allows Any Authenticated User to Post and Read Comments on Any Transaction

### Summary

`CommentsController` in `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only authentication guards at the class level but performs no per-transaction ownership or membership check. Any authenticated organization user can `POST` a comment to any transaction they have no relationship to, and `GET` all comments from any transaction. The developers themselves flagged this with a `//TODO need some sort of guard or check to ensure user can comment here` comment directly in the code.

### Finding Description

**Root cause — `CommentsController`:** [1](#0-0) 

The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level — these only verify that the caller is a valid, non-blacklisted, verified organization user. They do **not** verify that the user is the creator, signer, observer, or approver of the target transaction.

**`createComment` — unauthorized state mutation:** [2](#0-1) 

The `//TODO need some sort of guard or check to ensure user can comment here` comment is a developer acknowledgment that no access control exists. Any authenticated user can `POST /transactions/:transactionId/comments` with an arbitrary `transactionId` and inject a comment into a transaction they have no relationship to.

**`getComments` and `getCommentById` — unauthorized information disclosure:** [3](#0-2) 

Neither `getComments` nor `getCommentById` receives a `user` parameter. There is no access check — any authenticated user can enumerate comments on any transaction by iterating `transactionId` or comment `id`.

**Contrast with other controllers that do enforce per-transaction access:**

The `ObserversService.getTransactionObserversByTransactionId` and `ApproversService.getVerifiedApproversByTransactionId` both explicitly verify that the requesting user is the creator, a signer, an observer, or an approver before returning data. [4](#0-3) 

The comments endpoints have no equivalent check.

### Impact Explanation

1. **Unauthorized state mutation**: Any authenticated user can post comments on any transaction in the organization, injecting misleading, disruptive, or harassing content into transaction workflows they have no legitimate role in.
2. **Cross-tenant information disclosure**: Transaction comments may contain sensitive organizational decisions, key coordination details, or approval rationale. Any authenticated user can read all comments on any transaction by iterating `transactionId` values, bypassing the access control that protects all other transaction sub-resources (observers, approvers, signers, transaction bytes).

### Likelihood Explanation

The attacker precondition is only a valid organization account — no admin privileges, no leaked secrets. The exploit is a direct HTTP `POST` or `GET` to a documented REST endpoint. The developer TODO comment confirms the gap is known and unmitigated. Any malicious insider or compromised low-privilege account can exploit this immediately.

### Recommendation

Apply the same access-verification pattern used by `ObserversService` and `ApproversService`:

- In `createComment`: verify the requesting user is the transaction creator, a signer, an observer, or an approver before allowing the comment to be posted.
- In `getComments` / `getCommentById`: pass the `user` to the service and reject requests from users with no relationship to the transaction (mirroring `getTransactionObserversByTransactionId`).

A shared `verifyTransactionAccess(transactionId, user)` helper (analogous to `TransactionsService.verifyAccess`) should be called at the top of each comments service method.

### Proof of Concept

1. Register two users: `alice` (creator of transaction `#42`) and `bob` (no relationship to transaction `#42`).
2. Authenticate as `bob` and obtain a valid JWT.
3. `POST /transactions/42/comments` with body `{ "text": "Injected comment" }` and `bob`'s JWT → HTTP 201, comment is persisted on Alice's transaction.
4. `GET /transactions/42/comments` with `bob`'s JWT → HTTP 200, all comments (including sensitive organizational discussion) are returned.
5. Repeat step 4 for `transactionId` values 1–N to enumerate comments across all organization transactions.
