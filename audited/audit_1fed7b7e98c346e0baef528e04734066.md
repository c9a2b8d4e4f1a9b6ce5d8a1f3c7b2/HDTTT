### Title
Severely Misconfigured Coverage Collection Leaves Critical Back-End Apps and Front-End Logic Untested

### Summary
The repository has extensive spec files, but the coverage collection configuration is critically misconfigured: the root back-end Jest config only collects coverage from `libs/common`, entirely excluding the three application services (`api`, `chain`, `notifications`), and the front-end Vitest config tracks coverage for only a single file (`logger.ts`). This mirrors the BeamNetwork P1-H02 finding where components existed but had no effective test coverage tracking.

### Finding Description

**Back-End (`back-end/jest.config.ts`, line 11):**

The `collectCoverageFrom` field is scoped exclusively to `libs/common/src`:

```ts
collectCoverageFrom: ['<rootDir>/libs/common/src/**/*.{!(module),}.(t|j)s'],
roots: ['<rootDir>/libs/'],
```

This means the three core application services — `apps/api` (authentication, transaction CRUD, user management), `apps/chain` (Hedera network submission, scheduling), and `apps/notifications` (fan-out, WebSocket, email) — are **completely excluded from coverage reporting**, even though spec files exist for them. Coverage CI gates never apply to these apps. [1](#0-0) 

**Front-End (`front-end/src/tests/vitest.config.main.ts`, lines 19–28):**

The coverage `include` list contains only one file:

```ts
coverage: {
  provider: 'v8',
  include: ['src/main/modules/logger.ts'],
  ...
  thresholds: { branches: 90, functions: 90, lines: 90, statements: 90 },
},
```

All IPC handlers, services, utilities, and window management code in `src/main/` are excluded from coverage measurement. The 90% threshold applies only to `logger.ts`, giving a false sense of coverage quality. [2](#0-1) 

**Codecov threshold is also low:**

The global Codecov config sets the acceptable coverage range at `60..80`, meaning 60% overall coverage is treated as passing. [3](#0-2) 

### Impact Explanation
The `apps/chain` service is responsible for submitting signed transactions to the Hedera network and managing scheduling. The `apps/api` service handles JWT authentication, transaction approvals, and multi-sig coordination. Regressions or logic errors in these components — e.g., incorrect signature validation, double-execution of scheduled transactions, or broken approval flows — will not be caught by coverage-gated CI because these paths are invisible to the coverage tool. A developer could delete or break a critical guard in `apps/chain` or `apps/api` and the coverage report would remain unchanged.



### Likelihood Explanation
This is not theoretical — the misconfiguration is already in place and active. Any PR that modifies `apps/api`, `apps/chain`, or `apps/notifications` source files will not trigger a coverage regression, because those files are not in `collectCoverageFrom`. The front-end renderer components, stores, composables, and IPC handlers are similarly invisible. Given the active development pace of the project, the probability of an undetected regression in these uncovered paths is high. [4](#0-3) 

### Recommendation

1. **Back-end:** Expand `collectCoverageFrom` in `back-end/jest.config.ts` to include all application source files:
   ```ts
   collectCoverageFrom: [
     '<rootDir>/apps/**/src/**/*.{!(module),}.(t|j)s',
     '<rootDir>/libs/common/src/**/*.{!(module),}.(t|j)s',
   ],
   roots: ['<rootDir>/apps/', '<rootDir>/libs/'],
   ```

2. **Front-end:** Expand the `include` list in `vitest.config.main.ts` to cover all of `src/main/` (not just `logger.ts`), and add equivalent coverage configs for `vitest.config.renderer.ts` and `vitest.config.shared.ts`.

3. **Codecov:** Raise the minimum threshold from `60` to at least `80` and enforce per-flag thresholds for each app. [5](#0-4) 

### Proof of Concept

1. Open `back-end/jest.config.ts` — `collectCoverageFrom` references only `libs/common/src`. [1](#0-0) 
2. Open `back-end/apps/chain/src/` — contains `transaction-scheduler`, `transaction-reminder`, `cache-management` with spec files, but none of these source files appear in `collectCoverageFrom`. 
3. Open `front-end/src/tests/vitest.config.main.ts` — `include` under `coverage` lists only `src/main/modules/logger.ts`. All other main-process source files (IPC handlers, services, utilities) are excluded. [6](#0-5) 
4. Running `jest --coverage` from `back-end/` will produce a report showing 0 lines from `apps/api`, `apps/chain`, or `apps/notifications`, regardless of how many spec files pass.
