### Title
Sensitive Credentials and Cryptographic Secrets Hardcoded in Source Control

### Summary
The repository commits a live `.env.test` file and a test utility file containing hardcoded Hedera private keys directly into source control. Any party with repository read access — employees, contractors, or an attacker who obtains a copy — can extract these secrets and use them to forge authentication tokens or control Hedera accounts used in the test/CI pipeline.

### Finding Description

**Finding 1 — Committed `.env.test` with weak JWT and OTP secrets**

The file `back-end/apps/api/.env.test` is committed to the repository (not listed in `.gitignore`) and contains plaintext cryptographic secrets:

```
JWT_SECRET=13123
OTP_SECRET=123
``` [1](#0-0) 

`JWT_SECRET` is the HMAC signing key used to issue and verify all JSON Web Tokens in the API. `OTP_SECRET` is the seed used to generate and validate one-time passwords for email verification. Both values are trivially guessable (3–5 digit integers) and are now permanently embedded in the repository's git history.

**Finding 2 — Hardcoded Hedera private keys in test utility**

The file `back-end/apps/api/test/utils/hederaUtils.ts` exports five `HederaAccount` objects, each constructed with a hardcoded private key:

```typescript
export const localnet2 = new HederaAccount()
  .setAccountId('0.0.2')
  .setPrivateKey(
    '302e020100300506032b65700422042091132178e72057a1d7528025956fe39b0b847f200ab59b2fdd367017f3087137',
  );

export const localnet1002 = new HederaAccount()
  .setAccountId('0.0.1002')
  .setPrivateKey('0x7f109a9e3b0d8ecfba9cc23a3614433ce0fa7ddcc80f2a8f10b222179a5a80d6', 'ECDSA');
``` [2](#0-1) 

These keys are used to sign and submit real Hedera transactions in the e2e test suite against a local node. Account `0.0.2` is the genesis/treasury account of the Hedera local node; its key controls the initial HBAR supply on that node. If the same local node is ever exposed to a network (e.g., in a shared CI environment), an attacker with the key can drain all HBAR from the genesis account.

### Impact Explanation

- **JWT_SECRET=13123**: An attacker who reads the repository can sign arbitrary JWT payloads with this key. Because the API trusts any token signed with `JWT_SECRET`, the attacker can impersonate any user — including administrators — without knowing their password. This grants full API access: reading all transactions, approving/rejecting signatures, and managing organization members.
- **OTP_SECRET=123**: An attacker can pre-compute valid TOTP codes for any user's email verification step, bypassing the second-factor entirely.
- **Hedera private keys**: Full control over the corresponding Hedera accounts in any environment where these keys are reused (CI, staging, or a misconfigured local node exposed to the internet).

### Likelihood Explanation

The `.env.test` file is already committed and present in the repository's git history. Any developer, contractor, or attacker who clones the repository immediately has the secrets. The risk escalates if:
1. A developer accidentally copies `.env.test` to `.env` when standing up a staging or production instance (a common mistake when `example.env` and `.env.test` coexist).
2. The CI/CD pipeline uses `.env.test` directly and the test API endpoint is reachable from outside the CI network.

The `example.env` file shows the intended placeholder (`JWT_SECRET=some-very-secret-string`), confirming that the committed `.env.test` values were never meant to be production-grade — but they are now permanently in git history. [3](#0-2) 

### Recommendation

**Short term:**
1. Rotate `JWT_SECRET` and `OTP_SECRET` in all environments immediately.
2. Add `**/.env.test`, `**/.env`, and `**/.env.local` to `.gitignore`.
3. Remove the committed `.env.test` from git history following [GitHub's guide on removing sensitive data](https://docs.github.com/en/authentication/keeping-your-account-and-data-secure/removing-sensitive-data-from-a-repository) (e.g., using `git filter-repo`).
4. Replace hardcoded private keys in `hederaUtils.ts` with keys loaded from environment variables or a secrets manager at test runtime.

**Long term:**
- Store all secrets (JWT keys, OTP seeds, database passwords) in a secrets management solution (e.g., HashiCorp Vault, AWS Secrets Manager) and inject them at runtime via environment variables that are never committed to source control.
- Add a pre-commit hook or CI secret-scanning step (e.g., `gitleaks`, GitHub secret scanning) to prevent future credential commits.

### Proof of Concept

1. Clone the repository: `git clone https://github.com/0xOyakhilome/hedera-transaction-tool--003`
2. Read `back-end/apps/api/.env.test` — `JWT_SECRET=13123` is immediately visible.
3. Craft a forged JWT:
   ```javascript
   const jwt = require('jsonwebtoken');
   // Forge an admin token
   const token = jwt.sign({ sub: '<admin-user-id>', role: 'admin' }, '13123', { expiresIn: '365d' });
   console.log(token);
   ```
4. Use the forged token in an `Authorization: Bearer <token>` header against any protected API endpoint. The server will accept it as a valid session because it is signed with the correct `JWT_SECRET`. [4](#0-3) [5](#0-4)
