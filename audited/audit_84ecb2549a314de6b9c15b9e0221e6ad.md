### Title
Missing Upper-Bound Validation on Batch Signature Import Array Enables Authenticated DoS

### Summary
The `POST /transactions/signatures/import` endpoint accepts an unbounded array of `UploadSignatureMapDto` objects with no maximum size constraint. Each element triggers CPU-intensive SDK transaction deserialization, cryptographic signature validation, and per-item database queries. Any authenticated verified user can submit a single request with an arbitrarily large array, exhausting server CPU and database connection pool resources. The developer's own inline comment explicitly acknowledges this gap.

### Finding Description

**Root cause:** No `@ArrayMaxSize` constraint exists on the array accepted by `importSignatures`, and no server-side count cap is enforced before processing begins.

**Entry point:** `TransactionsController.importSignatures` at `back-end/apps/api/src/transactions/transactions.controller.ts` accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no size limit: [1](#0-0) 

`transformAndValidateDto` validates each element's schema individually but imposes no count cap on the array itself.

**Processing path:** `TransactionsService.importSignatures` at `back-end/apps/api/src/transactions/transactions.service.ts` iterates over every element in the caller-controlled array. Per item, it executes:
- `this.verifyAccess(transaction, user)` — an additional DB query per item
- `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive SDK deserialization
- `isExpired(sdkTransaction)` — SDK computation
- `validateSignature(...)` — cryptographic validation
- `sdkTransaction.addSignature(...)` — cryptographic operation [2](#0-1) 

The developer left an explicit acknowledgment of this gap at line 575: [3](#0-2) 

The same pattern exists on `POST /transactions/:transactionId/signers` (`SignersController.uploadSignatureMap`), which also accepts an unbounded `UploadSignatureMapDto[]`: [4](#0-3) 

The `CreateTransactionObserversDto` DTO has `@ArrayMinSize(1)` but no `@ArrayMaxSize`, and the service loops over all `userIds` performing DB inserts per entry: [5](#0-4) [6](#0-5) 

### Impact Explanation
A single HTTP request with a large array (e.g., 10,000 entries) causes the server to execute N cryptographic deserialization operations, N per-item DB queries (via `verifyAccess`), and N SDK signature operations synchronously within one request handler. This saturates the Node.js event loop, exhausts the TypeORM database connection pool, and renders the API unavailable for all other users — a complete service-level DoS. No admin privilege is required; any registered and verified organization user can trigger this.

### Likelihood Explanation
The attacker precondition is only a valid JWT for a verified user account — a normal registration flow. The endpoint is documented in the OpenAPI spec. The developer's own comment (`//Added a batch mechanism, probably should limit this on the api side of things`) confirms the absence of a limit is a known gap, not an intentional design choice.

### Recommendation
Add `@ArrayMaxSize(N)` (e.g., `N = 100`) to the array parameter in `UploadSignatureMapDto` usage, or enforce a hard count check at the top of `importSignatures` and `uploadSignatureMaps` before any processing loop begins. Apply the same fix to `CreateTransactionObserversDto` and `CreateTransactionGroupDto`.

### Proof of Concept
1. Register and verify a user account on the organization server.
2. Obtain a valid JWT via the login endpoint.
3. Send the following request:
```
POST /transactions/signatures/import
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 2, "signatureMap": { ... } },
  ... (repeat 10,000 times with valid or invalid IDs)
]
```
4. The server enters the processing loop at `transactions.service.ts:525`, executing `verifyAccess` + `SDKTransaction.fromBytes` + `validateSignature` for each of the 10,000 entries synchronously.
5. The Node.js event loop is blocked; all concurrent API requests time out; the database connection pool is exhausted.
6. The service is unavailable until the request completes or the process is restarted.
