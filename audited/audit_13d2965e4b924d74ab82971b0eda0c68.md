### Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Concurrent Signature Writes to Silently Corrupt Transaction Bytes

### Summary

`SignersService.uploadSignatureMaps` reads transaction state (bytes + status) without a row-level lock, performs async signature processing, then writes updated bytes back to the database with no status guard and no optimistic-lock check. When two authenticated users sign the same transaction concurrently — a normal multi-signer workflow — the second writer's `UPDATE` silently overwrites the first writer's signatures, causing permanent signature loss and potential status/bytes desynchronization.

### Finding Description

**Root cause — check and write are not atomic**

In `uploadSignatureMaps` the flow is:

1. **Read** — `loadTransactionData` fetches the transaction row and existing signers with a plain `SELECT` (no `FOR UPDATE` / `SKIP LOCKED`). [1](#0-0) 

2. **Check** — `validateTransactionStatus` inspects the in-memory snapshot's `status` field. [2](#0-1) 

3. **Process** — `processTransactionSignatures` builds new bytes by calling `addSignature` on the snapshot bytes. [3](#0-2) 

4. **Write** — `bulkUpdateTransactions` issues an unconditional `UPDATE … WHERE id = ANY(…)` with **no status guard and no version/ETag check**. [4](#0-3) 

The same pattern exists in `importSignatures` in `transactions.service.ts`, where the bytes update also carries no status guard: [5](#0-4) 

**Contrast with the protected path**

`processTransactionStatus` correctly uses optimistic locking (`AND status = :oldStatus`) so status transitions are safe. The bytes column has no equivalent protection. [6](#0-5) 

**Exploit flow (two concurrent signers)**

```
Time →
User A (Key A):  loadData → validateStatus(OK) → processSignatures(bytes+A) → bulkUpdate(bytes+A)
User B (Key B):  loadData → validateStatus(OK) → processSignatures(bytes+B) → bulkUpdate(bytes+B)
                                                                                      ↑
                                                                         overwrites bytes+A
```

Both requests read the same original bytes (no Key A, no Key B). User A writes `bytes+A`; User B, whose computation was based on the same original snapshot, then writes `bytes+B`, silently discarding Key A's signature. The DB now holds bytes with only Key B's signature.

Meanwhile, `updateStatusesAndNotify` for User A ran against the in-memory object that held `bytes+A` — if that was sufficient to satisfy the threshold, it may have advanced the status to `WAITING_FOR_EXECUTION` via `processTransactionStatus`. The stored bytes, however, only contain Key B's signature and cannot actually be executed.

### Impact Explanation

- **Permanent signature loss**: Key A's signature is silently discarded from the stored transaction bytes. There is no error, no log entry visible to users, and no automatic retry.
- **Status/bytes desynchronization**: The transaction status can advance to `WAITING_FOR_EXECUTION` while the stored bytes lack the required signatures, causing execution to fail or the transaction to be permanently stuck.
- **Integrity failure in the multi-signature trust model**: The core invariant — "a transaction in `WAITING_FOR_EXECUTION` has all required signatures in its stored bytes" — is violated without any privileged access.

### Likelihood Explanation

- **No privilege required**: Any two authenticated organization members who are both required signers of the same transaction can trigger this by signing simultaneously — a routine workflow in multi-signer organizations.
- **Realistic trigger**: The frontend's "Sign All" flow and automated signing scripts make concurrent submissions highly probable in practice.
- **No special timing needed**: The race window spans the entire async signature-processing duration (network I/O, crypto operations), which is wide enough to be hit reliably.

### Recommendation

Apply a **pessimistic row-level lock** at the point of reading the transaction, holding it through the write:

```typescript
// In loadTransactionData, inside a dataSource.transaction():
const transaction = await manager
  .createQueryBuilder(Transaction, 't')
  .where('t.id IN (:...ids)', { ids: transactionIds })
  .setLock('pessimistic_write')
  .getMany();
```

Move the entire `loadTransactionData → validateAndProcessSignatures → persistSignatureChanges` sequence inside a single `dataSource.transaction()` block so the lock is held until the write commits.

Alternatively, add an **optimistic-lock guard** to `bulkUpdateTransactions` analogous to what `processTransactionStatus` already does:

```sql
UPDATE transaction
SET "transactionBytes" = CASE id … END, "updatedAt" = NOW()
WHERE id = ANY($n)
  AND status IN ('WAITING_FOR_SIGNATURES', 'WAITING_FOR_EXECUTION')
```

and verify `rowsAffected` equals the expected count, returning an error to the caller if not.

### Proof of Concept

**Preconditions**: Organization transaction T requires signatures from User A (Key A) and User B (Key B). T is in `WAITING_FOR_SIGNATURES`.

1. User A calls `POST /transactions/signers` with Key A's signature map.
2. Before User A's request completes its DB write, User B calls `POST /transactions/signers` with Key B's signature map.
3. Both requests pass `validateTransactionStatus` (both see `WAITING_FOR_SIGNATURES`).
4. Both compute new bytes from the same original snapshot (no signatures).
5. User A's `bulkUpdateTransactions` writes `bytes+A` to the DB.
6. User B's `bulkUpdateTransactions` writes `bytes+B` to the DB, overwriting `bytes+A`.
7. **Observed**: DB holds `bytes+B` only. Key A's signature is gone. Both API calls return HTTP 201 success with no error.
8. If the threshold requires both keys, the transaction is now stuck: status may say `WAITING_FOR_EXECUTION` (set by User A's `processTransactionStatus` run against the in-memory `bytes+A`) but the stored bytes cannot satisfy execution.

      if (!signersByTransaction.has(signer.transactionId)) {

      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
