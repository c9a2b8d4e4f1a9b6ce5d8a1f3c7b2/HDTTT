### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Invalid Signatures to Advance Transaction State

### Summary
The `uploadSignatureMaps` endpoint (`POST /transactions/:id/signers`) in `signers.service.ts` accepts and persists signature maps without verifying that the submitted signature bytes are cryptographically valid over the transaction body. Any authenticated user can submit a garbage byte string as their signature, which the server records as a valid signing event, advances the transaction status toward execution, and ultimately causes the transaction to fail irrecoverably on the Hedera network. This is the direct analog of the external report's missing verification check: one code path performs the check (`importSignatures` calls `validateSignature`), while the primary signing path does not.

### Finding Description

**Root cause — missing `publicKey.verify()` call in `processTransactionSignatures`:**

`signers.service.ts` `processTransactionSignatures` (lines 217–267) iterates the submitted `SignatureMap`, confirms the public key belongs to the authenticated user via `userKeyMap`, then unconditionally calls `sdkTransaction.addSignature(publicKey, map)` without ever verifying the signature bytes are cryptographically valid: [1](#0-0) 

The only checks performed are:
1. Public key belongs to the authenticated user (`userKeyMap.get(raw)`)
2. Transaction is in the right status
3. Transaction is not expired

No call to `publicKey.verify(bodyBytes, signature)` is made anywhere in this path.

**Contrast with `importSignatures` which DOES verify:**

`transactions.service.ts` `importSignatures` (lines 545–549) calls `validateSignature` before adding any signature: [2](#0-1) 

`validateSignature` in `back-end/libs/common/src/utils/sdk/transaction.ts` (lines 213–248) explicitly calls `publicKey.verify(bodyBytes, signature)` and throws on failure: [3](#0-2) 

**Execution path check is also bypassed:**

`execute.service.ts` `getValidatedSDKTransaction` (lines 219–220) only checks that required public keys are *present* in `_signerPublicKeys`, not that their signatures are cryptographically valid: [4](#0-3) 

`_signerPublicKeys` is populated by `addSignature` regardless of whether the signature bytes are valid, so this check is bypassed by the invalid signature.

**`IsSignatureMap` decorator does not verify either:**

The DTO-level `IsSignatureMap` decorator only validates structural format (account ID, transaction ID, non-empty bytes) — it performs no cryptographic check: [5](#0-4) 

### Impact Explanation

An authenticated user who is a required signer for a transaction can:

1. Submit a signature map containing their registered public key paired with arbitrary/invalid signature bytes via `POST /transactions/:id/signers`.
2. The server records them as a signer in `TransactionSigner` and updates `transactionBytes` with the invalid signature.
3. `processTransactionStatus` may advance the transaction to `WAITING_FOR_EXECUTION` if the threshold is now met.
4. When the transaction is submitted to the Hedera network, it is rejected because the signature is cryptographically invalid.
5. The transaction is marked `FAILED` — an irrecoverable terminal state.

The transaction is permanently destroyed. Any HBAR or token operations it was meant to perform are lost. If the attacker is a co-signer on a multi-sig transaction (e.g., a threshold key account), they can unilaterally sabotage any transaction they are required to sign.

### Likelihood Explanation

- **Attacker preconditions**: Must be an authenticated user with at least one registered public key (`UserKey`) in the system. No admin or privileged access required.
- **Attack complexity**: Trivial. The attacker simply crafts a `POST /transactions/:id/signers` request with their real public key but a random 64-byte hex string as the signature value.
- **Reachability**: The endpoint is a standard authenticated API endpoint, reachable by any registered organization user.
- **Detection**: No server-side error is raised; the request succeeds with HTTP 201.

### Recommendation

Add cryptographic signature verification inside `processTransactionSignatures` in `signers.service.ts`, mirroring the existing `validateSignature` call used in `importSignatures`. Before calling `sdkTransaction.addSignature(publicKey, map)`, call `validateSignature(sdkTransaction, map)` (or an equivalent per-key `publicKey.verify(bodyBytes, signature)` check) and reject the request if any signature is invalid.

```typescript
// In processTransactionSignatures, before addSignature:
const { data: validKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

This is the same pattern already used in `transactions.service.ts` lines 546–549.

### Proof of Concept

1. Register as a user in the organization and upload a public key (e.g., ED25519 key `pubKeyRaw`).
2. Obtain a transaction ID where `pubKeyRaw` is a required signer.
3. Craft a `POST /transactions/:id/signers` body:
```json
[{
  "id": <transactionId>,
  "signatureMap": {
    "0.0.3": {
      "<transactionId@validStart>": {
        "<DER-encoded pubKeyRaw>": "0xdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeefdeadbeef"
      }
    }
  }
}]
```
4. Server responds HTTP 201. The `TransactionSigner` record is created and `transactionBytes` is updated with the invalid signature.
5. If threshold is met, transaction advances to `WAITING_FOR_EXECUTION`.
6. Transaction is submitted to Hedera and rejected with `INVALID_SIGNATURE`. Transaction is marked `FAILED` — permanently unrecoverable.
