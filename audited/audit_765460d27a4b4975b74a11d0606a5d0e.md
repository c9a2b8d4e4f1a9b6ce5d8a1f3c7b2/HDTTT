### Title
Transaction Creator Can Modify Approver Structure After Approvals Are Committed, Bypassing Multi-Approval Requirement

### Summary
The `getCreatorsTransaction` helper used to authorize all approver mutations (create, update, delete) performs only an identity check — it never validates the transaction's current status. This allows the transaction creator, a regular unprivileged user, to add, remove, or restructure approvers at any point in the transaction lifecycle, including after approvers have already signed. The creator can therefore unilaterally dismantle the approval gate that other parties relied upon.

### Finding Description

**Vulnerability class:** Authorization / state-transition invariant violation (direct analog of the external "owner has too many permissions" class).

**Root cause — `getCreatorsTransaction`:** [1](#0-0) 

The function fetches the transaction and verifies `creatorKey.userId === user.id`. It returns the transaction unconditionally — there is **no check on `transaction.status`**. Every approver-mutation path delegates its authorization to this single function.

**Three mutation paths that call it with no additional status guard:**

1. `createTransactionApprovers` — adds new approvers: [2](#0-1) 

2. `removeTransactionApprover` (controller) — deletes an approver: [3](#0-2) 

3. `updateTransactionApprover` — changes threshold or re-parents a node: [4](#0-3) 

**Threshold auto-reduction on child removal** (makes the exploit self-completing):

When a child approver is removed and the remaining count falls below the parent threshold, the service automatically lowers the threshold to match: [5](#0-4) 

**Approver approval is gated only on transaction status, not on approver immutability:** [6](#0-5) 

The `approveTransaction` path accepts `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` — both reachable states where the creator can still mutate approvers.

**Transaction entity — `cutoffAt` and `status` are stored but never consulted by approver mutations:** [7](#0-6) 

### Impact Explanation

A transaction creator can:

1. **Remove a required approver who has not yet signed** — the threshold is automatically reduced, so the transaction can reach `WAITING_FOR_EXECUTION` with fewer approvals than originally required.
2. **Remove all approvers** — the approval gate disappears entirely; the chain service sees no pending approvers and can proceed.
3. **Add themselves as an approver and self-approve** — `createTransactionApprovers` has no check preventing the creator from being listed as an approver, and `approveTransaction` has no check preventing the creator from approving their own transaction.

The multi-approval workflow is the primary trust mechanism for organization transactions. Bypassing it allows a single user to execute transactions that were explicitly designed to require independent review, constituting a critical integrity failure.

### Likelihood Explanation

- **Attacker profile:** Any authenticated, verified user who has created at least one organization transaction. No admin key, no leaked credential, no network access required.
- **Trigger:** Three standard API calls (`DELETE /transactions/:id/approvers/:approverId`, `POST /transactions/:id/approvers`, `POST /transactions/:id/approvers/approve`).
- **Detection difficulty:** The mutations emit `emitTransactionUpdate` / `emitTransactionStatusUpdate` events, but there is no audit trail distinguishing a legitimate approver change from a malicious mid-flight one.

### Recommendation

Add a status guard inside `getCreatorsTransaction` (or at the top of each mutation function) that rejects modifications once the transaction has left the `WAITING_FOR_SIGNATURES` state:

```typescript
const IMMUTABLE_STATUSES = [
  TransactionStatus.WAITING_FOR_EXECUTION,
  TransactionStatus.EXECUTED,
  TransactionStatus.CANCELED,
  TransactionStatus.EXPIRED,
  TransactionStatus.ARCHIVED,
  TransactionStatus.FAILED,
];

if (IMMUTABLE_STATUSES.includes(transaction.status))
  throw new BadRequestException('Approver structure cannot be modified at this stage');
```

Additionally, prevent the creator from adding themselves as an approver on their own transaction.

### Proof of Concept

**Preconditions:** Attacker is a registered, verified user (creator). Transaction T has a 2-of-2 approver requirement (approver A and approver B). Approver A has signed; approver B has not.

1. **Attacker authenticates** → receives JWT.
2. **Attacker calls** `DELETE /transactions/T/approvers/<approverB_id>` — removes approver B. The service calls `getCreatorsTransaction` (passes, creator check only), then `removeNode`. Because the parent threshold (2) now exceeds the remaining child count (1), the threshold is automatically reduced to 1.
3. **Transaction status** is now satisfiable with only approver A's existing signature. The chain service's next poll moves T to `WAITING_FOR_EXECUTION`.
4. **Result:** Transaction executes with one approval instead of the required two, without approver B's knowledge or consent.

**Alternative (zero-approval bypass):**

1. Attacker creates transaction T with approver A.
2. Attacker calls `DELETE /transactions/T/approvers/<approverA_id>` before A signs.
3. No approvers remain; the chain service sees no pending approval requirement.
4. Transaction executes with zero approvals.

```typescript
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
