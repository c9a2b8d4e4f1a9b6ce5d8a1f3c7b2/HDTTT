### Title
Unbounded Array Input in `importSignatures` and `uploadSignatureMaps` Enables Resource Exhaustion by Authenticated Users

### Summary
The `importSignatures` function in `transactions.service.ts` and `uploadSignatureMaps` in `signers.service.ts` both accept user-supplied arrays (`UploadSignatureMapDto[]`) with no upper-bound limit. Each element triggers expensive operations: database queries, cryptographic deserialization (`SDKTransaction.fromBytes`), and signature validation. A single authenticated user can submit one crafted request with thousands of entries, exhausting server CPU, memory, and database connections. A developer comment in the code explicitly acknowledges the missing limit.

### Finding Description

**Root cause — `importSignatures` (`transactions.service.ts`, line 493):**

The function signature accepts an unbounded slice of DTOs:

```ts
async importSignatures(
  dto: UploadSignatureMapDto[],
  user: User,
): Promise<SignatureImportResultDto[]>
``` [1](#0-0) 

For every element in `dto`, the function performs:
- A batch DB fetch (`entityManager.find`) over all supplied IDs
- `SDKTransaction.fromBytes(transaction.transactionBytes)` — protobuf deserialization
- `validateSignature` — cryptographic verification
- `sdkTransaction.addSignature` — per-key signature attachment [2](#0-1) 

The developer's own comment at line 575 explicitly acknowledges the missing guard:

```ts
//Added a batch mechanism, probably should limit this on the api side of things
const BATCH_SIZE = 500;
``` [3](#0-2) 

The `BATCH_SIZE = 500` only controls how many SQL `CASE` clauses are emitted per UPDATE statement — it does **not** cap the total number of items accepted from the caller. An attacker can still submit 50,000 items; they will be processed in 100 batches of 500, each batch still executing the full crypto pipeline.

**Root cause — `uploadSignatureMaps` (`signers.service.ts`, line 99):**

```ts
async uploadSignatureMaps(
  dto: UploadSignatureMapDto[],
  user: User,
): Promise<{ signers: TransactionSigner[]; notificationReceiverIds: number[] }>
``` [4](#0-3) 

The function calls `validateAndProcessSignatures`, which fans out into `Promise.all(dto.map(...))` — all items are processed concurrently with no concurrency cap or array size limit. [5](#0-4) 

**Root cause — `createTransactionGroup` (`transaction-groups.service.ts`, line 40):**

`groupItems` in `CreateTransactionGroupDto` is accepted without an `ArrayMaxSize` constraint. Each item triggers `createTransactions`, which performs signature verification, SDK deserialization, and DB writes per entry. [6](#0-5) 

**DTO validation gap:**

The grep for `ArrayMaxSize` across all backend DTOs returned zero matches. The only array decorators present are `@IsArray` and `@ArrayNotEmpty` — neither enforces an upper bound. [7](#0-6) 

### Impact Explanation

A single authenticated user submitting one HTTP request with a large `UploadSignatureMapDto[]` array (e.g., 10,000 entries) will:
1. Force the server to deserialize and cryptographically process thousands of Hedera SDK transactions.
2. Saturate the PostgreSQL connection pool with concurrent queries.
3. Cause the Node.js event loop to stall under `Promise.all` fan-out with no concurrency limit.

This results in **service unavailability** for all other users during the attack window. Because the attack requires only one request from one valid account, it is a severe degradation path, not a volumetric DDoS.

### Likelihood Explanation

Any registered user of the Organization Mode can reach these endpoints. No elevated privilege is required. The attacker needs only a valid session token and knowledge of the API (which is documented). The developer comment in the code confirms the team is aware the limit is missing but has not yet implemented it.

### Recommendation

1. Add `@ArrayMaxSize(N)` to `UploadSignatureMapDto[]` parameters in the controller layer (e.g., `N = 100` to match the NFT cap in the referenced fix).
2. Add `@ArrayMaxSize(N)` to `groupItems` in `CreateTransactionGroupDto`.
3. Replace the unbounded `Promise.all` in `validateAndProcessSignatures` with a concurrency-limited executor (e.g., `p-limit`).
4. Enforce the limit at the controller/pipe level so it is rejected before reaching service logic.

### Proof of Concept

**Preconditions:** Attacker has a valid authenticated session (normal user account).

**Request:**
```
POST /transactions/import-signatures
Authorization: Bearer <valid_token>
Content-Type: application/json

{
  "signatures": [
    { "id": 1, "signatureMap": { ... } },
    { "id": 2, "signatureMap": { ... } },
    ... // repeated 10,000 times with valid or invalid transaction IDs
  ]
}
```

**Expected outcome:**
- Server enters `importSignatures` with `dto.length === 10000`.
- `entityManager.find` is called with `In([...10000 ids])`.
- The loop at line 525 iterates 10,000 times, calling `SDKTransaction.fromBytes` and `validateSignature` for each.
- Server CPU spikes; PostgreSQL connection pool is saturated; concurrent legitimate requests time out.
- All other users experience service degradation or complete unavailability for the duration of the request. [8](#0-7)
