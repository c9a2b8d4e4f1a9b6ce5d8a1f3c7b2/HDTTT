### Title
Organization Members Cannot Self-Remove from the Server — Permanent Lock Without Admin Cooperation

### Summary
The `DELETE /users/:id` endpoint in `UsersController` is exclusively gated behind `AdminGuard`. No self-removal endpoint (e.g., `DELETE /users/me`) exists. A regular organization member who wishes to leave the organization cannot do so unilaterally — their server-side account, registered keys, and transaction history remain permanently active until an admin removes them. This is a direct analog to the DAO membership exit vulnerability described in the external report.

### Finding Description

**Root cause:** The only user-deletion path in the API is `DELETE /users/:id`, which requires `AdminGuard`. The guard checks `user && user.admin` and returns `false` for any non-admin, resulting in a `403 Forbidden`. [1](#0-0) [2](#0-1) 

The handler also explicitly throws `BadRequestException(ErrorCodes.CRYFO)` if an admin attempts to delete themselves, meaning **no user — admin or regular — can self-remove via this endpoint**. [3](#0-2) 

The frontend service layer acknowledges this restriction with the comment `/* ADMIN ONLY: Delete a user */` and only exposes `deleteUser` through admin-gated UI flows. [4](#0-3) 

The `handleRemove()` function in `ContactList.vue` only calls `deleteUser` when `user.selectedOrganization.admin` is `true`, confirming no non-admin path exists in the UI either. [5](#0-4) 

**The local "disconnect" is not equivalent to leaving:** `handleDeleteConnection` in `OrganizationsTab.vue` only removes the local SQLite connection record (credentials, key pairs, contacts). It does **not** call the server-side `DELETE /users/:id`. The user's account, keys, and transaction history remain live on the organization server. [6](#0-5) [7](#0-6) 

**E2E tests confirm the access control:** `(DELETE) should not remove a user if not admin` asserts a `403` response for non-admin callers. [8](#0-7) 

### Impact Explanation

A regular organization member who wants to leave cannot remove their own server-side account. Their email, registered public keys, and full transaction history remain on the organization server indefinitely. If the admin is unresponsive, hostile, or the organization is abandoned, the user has no recourse — their membership state is permanently locked. This satisfies the RESEARCHER.md criterion of **"Permanent lock, freeze, or unrecoverable corruption of user/project state."**

### Likelihood Explanation

This affects every non-admin user who wants to exit an organization. No special attacker capability is required — any authenticated regular user who attempts `DELETE /users/<own_id>` receives a `403`. The condition is trivially reachable by any member of any organization.

### Recommendation

Add a `DELETE /users/me` endpoint (no `AdminGuard`) that allows an authenticated user to soft-delete their own account and associated keys. The existing `removeUser` service method already implements the correct soft-delete logic and can be reused directly. [9](#0-8) 

### Proof of Concept

1. Register as a non-admin user on an organization server and obtain a JWT.
2. Send `DELETE /users/<own_id>` with the JWT as `Authorization: Bearer <token>`.
3. Observe `403 Forbidden` — the request is rejected by `AdminGuard`.
4. In the desktop client, use Settings → Organizations → "Delete Connection" to remove the local record.
5. Confirm via `GET /users/<own_id>` (using an admin token) that the server-side account, keys, and transaction records are still fully present and active.
6. There is no API endpoint or UI flow that allows the user to complete server-side removal without admin cooperation.

  async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });

    if (!user) {
      throw new BadRequestException(ErrorCodes.UNF);
    }

    // Soft-delete all user keys first
    await this.repo.manager.softDelete(UserKey, { userId: id });

    // Then soft-delete the user
    await this.repo.softRemove(user);

    return true;
  }
