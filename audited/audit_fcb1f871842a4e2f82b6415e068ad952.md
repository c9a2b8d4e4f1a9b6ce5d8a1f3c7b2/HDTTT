### Title
Unbounded Full-Table Scan with Per-Row External I/O in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in `TransactionsService` fetches every active transaction from the database with no row limit, then performs CPU-intensive deserialization and multiple external mirror-node HTTP calls for each row. Any authenticated user can trigger this endpoint repeatedly, causing memory exhaustion, connection-pool starvation, and server unresponsiveness that grows proportionally with the total number of active transactions in the system.

### Finding Description

**Root cause — no database-level limit on the initial fetch:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–299, the repository query has no `take` / `limit` clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

Compare this to every other paginated method in the same file (`getTransactions`, `getHistoryTransactions`, `getTransactionsToApprove`) which all pass `skip: offset, take: limit` to the query. [1](#0-0) [2](#0-1) 

**Per-row expensive work — deserialization + external I/O:**

After loading all N transactions, the code loops over every one and calls `userKeysToSign`, which internally calls `computeSignatureKey`: [3](#0-2) 

`computeSignatureKey` performs, for each transaction:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive protobuf deserialization
2. One or more `accountCacheService.getAccountInfoForTransaction()` calls — outbound HTTP to the Hedera mirror node
3. Optionally `nodeCacheService.getNodeInfoForTransaction()` — another outbound HTTP call [4](#0-3) 

**Pagination is applied only after the full scan:**

The `page`/`limit` parameters are accepted but only used to slice the already-computed in-memory `result` array:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [5](#0-4) 

**The endpoint is still publicly reachable:**

The controller registers `GET /transactions/sign` for any authenticated, verified user. The comment "NO LONGER USED BY FRONT-END" confirms the frontend has moved on, but the route remains active and callable by any API client: [6](#0-5) 

**The `whereForUser` filter does not bound the result set:**

The only filter applied is `status NOT IN (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED)`. In a production organization with many concurrent transactions, this set can be arbitrarily large. There is no per-user scoping at the DB level — the query returns all active transactions across all users. [7](#0-6) 

### Impact Explanation

- **Memory exhaustion**: All active transaction rows (including their `transactionBytes` blobs) are loaded into the Node.js heap simultaneously. With thousands of transactions, a single request can consume hundreds of MB.
- **CPU exhaustion**: Protobuf deserialization (`SDKTransaction.fromBytes`) is synchronous and CPU-bound; N transactions means N serial deserializations blocking the event loop.
- **Connection pool / mirror-node starvation**: Each transaction triggers one or more outbound HTTP calls to the Hedera mirror node. N transactions = N+ concurrent or sequential HTTP connections, exhausting the HTTP agent pool and degrading all other mirror-node-dependent operations.
- **Cascading denial of service**: Because Node.js is single-threaded, a blocked event loop from repeated calls to this endpoint makes the entire API service unresponsive for all users, not just the attacker.

Severity: **High** — complete API unavailability achievable by a single authenticated user with no special privileges.

### Likelihood Explanation

- **Attacker precondition**: Only a valid session cookie / JWT is required — any registered, verified organization member qualifies.
- **No rate-limit evidence found** in the controller or service for this specific endpoint.
- **Grows automatically**: As the organization accumulates transactions over time, the cost per request increases without any operator action.
- **Trivially scriptable**: A simple loop of `GET /transactions/sign?page=1&size=1` requests is sufficient; the `size=1` parameter does not reduce server-side work at all.

### Recommendation

Apply the database-level `take`/`limit` at the query layer, not after the fact. The correct fix mirrors the pattern already used in `getTransactions` and `getTransactionsToApprove`:

```typescript
// Replace the unbounded find:
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ADD:
  skip: offset,
  take: limit,
});
```

However, because the eligibility check (`userKeysToSign`) cannot be pushed into SQL, a true fix requires either:
1. Moving the signing-key check into a SQL subquery (preferred), or
2. Implementing a cursor-based scan with a hard cap (e.g., process at most 500 rows per request and return a `nextCursor` token), or
3. Pre-computing and caching which transactions each user must sign (event-driven, updated on signature events).

Additionally, the endpoint should be rate-limited per user.

### Proof of Concept

**Preconditions**: Authenticated as any verified organization user. The organization has ≥ 500 active (non-terminal) transactions.

**Steps**:
1. Obtain a valid session token via `POST /auth/login`.
2. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <token>
   ```
3. Each request causes the server to:
   - Load all N active transactions from PostgreSQL into memory.
   - Deserialize all N transaction byte blobs.
   - Issue N+ HTTP calls to the Hedera mirror node.
4. With 10 concurrent requests from the same user, the server's event loop saturates and all other API endpoints begin timing out.

**Expected outcome**: HTTP 504 / connection timeouts for all other users; Node.js process memory climbs to OOM; mirror-node HTTP pool exhausted.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
