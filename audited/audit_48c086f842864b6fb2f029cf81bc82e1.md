### Title
Transaction Creator Can Add Themselves as Approver and Self-Approve Their Own Transaction

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` enforces that only the transaction creator can add approvers, but contains no check preventing the creator from designating themselves (`user.id`) as an approver. Because `approveTransaction` only verifies that the caller appears in the approver list, the creator can add themselves, then immediately approve their own transaction â€” collapsing the entire multi-party approval workflow into a single-actor action.

### Finding Description
**Root cause:** `createTransactionApprovers` (line 234) calls `getCreatorsTransaction` to confirm the caller is the transaction creator, then iterates over the submitted approver DTOs. For each DTO with a `userId`, it only checks that the user exists in the database (line 271â€“276). There is no guard of the form `if (dtoApprover.userId === user.id) throw ...`. [1](#0-0) [2](#0-1) 

The same gap exists in `updateTransactionApprover` when `dto.userId` is supplied â€” it checks existence but not self-assignment: [3](#0-2) 

**Exploit flow:**

1. Attacker (normal user) creates a transaction via `POST /transactions` â€” they become the creator.
2. Attacker calls `POST /transactions/:id/approvers` with `approversArray: [{ userId: <own id> }]`. The service confirms the caller is the creator and that the userId exists â€” both pass. The attacker is now an approver of their own transaction.
3. Attacker calls `POST /transactions/:id/approvers/approve` with a valid signature from one of their own keys. `approveTransaction` filters approvers by `userId === user.id`, finds the self-inserted record, and records the approval. [4](#0-3) 

The approval threshold check in `processTransactionStatus` then sees the required approvals satisfied, advancing the transaction status without any independent party ever reviewing it.

### Impact Explanation
The approval workflow is the primary mechanism for enforcing multi-party authorization on sensitive Hedera transactions (account updates, node operations, large fund movements). A creator who self-approves bypasses this control entirely: no independent reviewer is required, and the transaction can proceed to execution as if it had been legitimately approved. This constitutes an **unauthorized state change / privilege escalation** â€” the creator unilaterally satisfies an approval requirement that was intended to involve a separate party.

### Likelihood Explanation
The attack requires only a standard authenticated session as a normal user â€” no admin keys, no leaked credentials, no race conditions. The two API calls (`POST approvers`, `POST approvers/approve`) are part of the normal documented workflow. Any user who can create a transaction can exploit this immediately.

### Recommendation
In `createTransactionApprovers`, reject any approver DTO whose `userId` equals the requesting user's id:

```typescript
if (dtoApprover.userId === user.id)
  throw new Error('Creator cannot add themselves as an approver');
```

Apply the same guard in `updateTransactionApprover` when processing a `dto.userId` update:

```typescript
if (dto.userId === user.id)
  throw new Error('Creator cannot assign themselves as an approver');
``` [5](#0-4)
