### Title
Missing Null Guard on `signatureKey` in `approveTransaction` Causes Unhandled Exception (DoS)

### Summary
In `approvers.service.ts`, the `approveTransaction` method searches `user.keys` for a key matching the caller-supplied `dto.userKeyId`. If no match is found, `signatureKey` is `undefined`. The code then immediately calls `PublicKey.fromString(signatureKey?.publicKey)` — passing `undefined` to the SDK — without any guard. The Hedera SDK throws on invalid input, producing an unhandled exception and a 500 Internal Server Error for every such request.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction` method performs the following sequence:

```typescript
// line 567
if (user.keys.length === 0) return false;

// line 569-572
const signatureKey = user.keys.find(key => key.id === dto.userKeyId);
/* Gets the public key that the signature belongs to */
const publicKey = PublicKey.fromString(signatureKey?.publicKey);
``` [1](#0-0) 

The guard at line 567 only handles the case where the user has **zero** keys. It does not handle the case where the user has keys but none of them matches `dto.userKeyId`. In that scenario:

- `user.keys.find(...)` returns `undefined`
- `signatureKey?.publicKey` evaluates to `undefined`
- `PublicKey.fromString(undefined)` is called — the Hedera SDK throws a `TypeError` or similar

There is no `try/catch` around this block inside `approveTransaction`, so the exception propagates unhandled to the NestJS runtime, which returns HTTP 500.

The analogous pattern in the external report is a constructor or function that accepts an address/key parameter and uses it without first asserting it is non-zero/non-null.

### Impact Explanation
Any authenticated user who holds the **approver** role on at least one transaction can send a well-formed `POST` request to the approve-transaction endpoint with a `userKeyId` that is a valid integer but does not belong to their key set. The server will throw an unhandled exception and return HTTP 500 for every such request. This constitutes a targeted denial-of-service against the approval endpoint, and it produces misleading 500 errors in server logs that obscure legitimate failures.

### Likelihood Explanation
The attack requires only that the caller be an authenticated approver — a role that is assigned by the transaction creator and is therefore reachable by ordinary organization members. No privileged access is needed beyond being listed as an approver. Supplying an out-of-range `userKeyId` is trivial (e.g., `userKeyId: 0` or any integer not in the user's key list).

### Recommendation
Add an explicit guard immediately after the `find` call, consistent with the pattern already used elsewhere in the codebase (e.g., `removeUserKey` in `user-keys.service.ts`):

```typescript
const signatureKey = user.keys.find(key => key.id === dto.userKeyId);
if (!signatureKey) {
  throw new BadRequestException(ErrorCodes.KNF); // "Key not found"
}
const publicKey = PublicKey.fromString(signatureKey.publicKey);
``` [2](#0-1) 

### Proof of Concept

1. Register and log in as a user (`userA`) who has been added as an approver on transaction `T`.
2. Obtain `userA`'s valid JWT.
3. Send:
   ```
   POST /transactions/{T}/approvers/approve
   Authorization: Bearer <userA-jwt>
   {
     "userKeyId": 999999,   // valid integer, but not in userA's key set
     "signature": "<any bytes>",
     "approved": true
   }
   ```
4. The server executes `user.keys.find(key => key.id === 999999)` → `undefined`, then calls `PublicKey.fromString(undefined)`, which throws.
5. Response: `HTTP 500 Internal Server Error`.
