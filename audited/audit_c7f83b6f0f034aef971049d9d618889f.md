### Title
`removeTransactionApprover` Does Not Update Parent Threshold After Child Removal, Permanently Locking Transaction Approval State

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` removes an approver node from the tree without checking or adjusting the parent list's `threshold`. If the parent's threshold exceeds the remaining child count after removal, the approval condition becomes permanently unsatisfiable and the transaction is irreversibly stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. This is the direct analog of the external report's missing power-threshold sanity check in `updateValset`.

### Finding Description

**Root cause — missing threshold adjustment in `removeTransactionApprover`:** [1](#0-0) 

The function accepts only an `id` parameter (no `user`), calls `removeNode`, and emits a status event. It never reads the parent node, never checks whether the parent's `threshold` still satisfies `threshold ≤ remaining_children`, and never updates the parent record.

Contrast this with the `updateTransactionApprover` path that handles the `dto.listId === null` (detach) case — it **does** perform the threshold correction: [2](#0-1) 

That branch explicitly reduces the parent threshold when `newParentApproversLength < parent.threshold`. The `removeTransactionApprover` code path has no equivalent guard.

**Concrete stuck-state scenario:**

1. Transaction creator builds an approver tree: `parent (threshold = 2)` → children `[UserA, UserB]`.
2. Creator (or any caller with access to the DELETE endpoint) calls `removeTransactionApprover(UserA.id)`.
3. `removeNode` soft-deletes `UserA`. Parent now has `threshold = 2`, `children = [UserB]`.
4. Only 1 approver remains; the threshold of 2 can never be reached.
5. The transaction is permanently stuck — it can never advance past the approval gate.

**Secondary concern — missing creator authorization in the service function:**

`createTransactionApprovers` and `updateTransactionApprover` both call `getCreatorsTransaction(transactionId, user, ...)` to enforce that only the transaction creator may modify the approver tree: [3](#0-2) [4](#0-3) 

`removeTransactionApprover` accepts no `user` parameter and makes no such call, meaning creator-level authorization is absent at the service layer. If the controller does not compensate, any authenticated user can delete any approver node.

### Impact Explanation

A transaction whose approver tree has an unsatisfiable threshold is permanently locked in the approval state. It cannot be approved, executed, or progressed. The only escape would be for the creator to cancel the transaction (if that path is still reachable), but the approval state itself is corrupted. This maps directly to the "contract effectively stuck" impact described in the external report.

### Likelihood Explanation

The trigger is a normal, in-scope API call (`DELETE /transactions/:id/approvers/:approverId`). No cryptographic break or privileged key is required. Any authenticated user who can reach the endpoint — potentially including non-creators if the controller lacks a creator guard — can trigger the stuck state. The scenario is reachable in any organization workflow that uses threshold approver trees.

### Recommendation

1. **In `removeTransactionApprover`**, after soft-deleting the node, load the parent (if any), count its remaining non-deleted children, and if `remaining < parent.threshold`, either reduce `parent.threshold` to `remaining` or reject the removal with an error explaining the threshold would become unsatisfiable.
2. **Add a `user` parameter** to `removeTransactionApprover` and call `getCreatorsTransaction` (matching the pattern used by `createTransactionApprovers` and `updateTransactionApprover`) to enforce that only the transaction creator can remove approver nodes.

### Proof of Concept

```
POST /transactions/42/approvers
Body: { approversArray: [{ threshold: 2, approvers: [{ userId: 10 }, { userId: 11 }] }] }
→ Creates parent (id=1, threshold=2) with children UserA(id=2), UserB(id=3)

DELETE /transactions/42/approvers/2          ← removes UserA
→ removeTransactionApprover(2) called
→ removeNode(2) soft-deletes UserA
→ parent (id=1) still has threshold=2, but only child UserB remains
→ approveTransaction() for UserB succeeds, but threshold counter reaches 1 < 2
→ transaction never advances; permanently stuck
```
