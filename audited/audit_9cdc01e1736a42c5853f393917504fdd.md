### Title
Stale Pre-Update State in `approveTransaction` Causes Wrong Notification Event on Every Approval

### Summary

In `approvers.service.ts`, the `approveTransaction` function evaluates the post-update notification branch using the **pre-update** in-memory snapshot of `userApprovers`. Because the guard at line 563 guarantees at least one approver record lacks a signature before the update, the condition `userApprovers.every(a => a.approved)` is always `false` when `dto.approved = true`. This means every approval unconditionally emits `emitTransactionUpdate` instead of `emitTransactionStatusUpdate`, suppressing the status-recomputation event that is supposed to fire when new approvals are collected.

### Finding Description

**Root cause — wrong variable, analogous to the GaugeController report:**

```
// approvers.service.ts L553-556  (snapshot taken BEFORE the DB write)
const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);
const userApprovers = approvers.filter(a => a.userId === user.id);

// L563 — guard: throws if ALL already have a signature
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

// L599-610 — DB is updated here
await this.dataSource.transaction(async transactionalEntityManager => {
  await transactionalEntityManager
    .createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .whereInIds(userApprovers.map(a => a.id))
    .execute();
});

// L614 — decision uses the STALE snapshot, not the post-update state
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
``` [1](#0-0) [2](#0-1) [3](#0-2) 

**Why the condition is always wrong when `dto.approved = true`:**

The guard at line 563 (`userApprovers.every(a => a.signature)`) ensures execution only continues when at least one approver record has no signature yet. That same record therefore has `approved = undefined/null`. Consequently `userApprovers.every(a => a.approved)` is always `false` at line 614 when the user is submitting a new approval. The branch taken is always `emitTransactionUpdate`, never `emitTransactionStatusUpdate`.

When `dto.approved = false` (rejection), `!dto.approved` is `true`, so `emitTransactionStatusUpdate` is correctly called. The logic is inverted: **rejections trigger status re-evaluation; approvals never do.**

**Correct post-update state is never consulted.** The DB write at lines 599-610 updates `approved` to `dto.approved`, but `userApprovers` in memory is never refreshed. The decision at line 614 should either re-query the DB or simply check `dto.approved` directly.

### Impact Explanation

`emitTransactionStatusUpdate` is the event that causes the notification/chain pipeline to re-evaluate whether a transaction has collected enough approvals to advance its status (e.g., `WAITING_FOR_SIGNATURES → WAITING_FOR_EXECUTION`). [4](#0-3) 

Because approvals never emit `emitTransactionStatusUpdate`, a transaction can collect every required approval and still remain permanently stuck in `WAITING_FOR_SIGNATURES`. The status transition is only triggered by unrelated events (approver creation/removal, cancellation). [5](#0-4) 

This constitutes a **permanent lock of transaction state** reachable through the normal approval workflow — no privileged access required.

### Likelihood Explanation

The bug is triggered on every call to `approveTransaction` where `dto.approved = true`, which is the primary, expected user action in the approval workflow. Any authenticated approver submitting a valid approval hits this path. No special conditions or attacker capabilities are needed.

### Recommendation

Replace the stale in-memory check with the value that was just written to the database:

```typescript
// Before (wrong — uses pre-update snapshot):
if (!dto.approved || userApprovers.every(a => a.approved)) {

// After (correct — uses the value that was just persisted):
if (!dto.approved) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
}
// Simplified: always emit status update after any approval decision
emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
```

Or, if the intent is to distinguish "new approval" from "no-op":

```typescript
if (!dto.approved || !userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

The safest fix is to always call `emitTransactionStatusUpdate` after any approval decision, since status re-evaluation is idempotent.

### Proof of Concept

1. Create a transaction with a threshold approver group requiring 1-of-2 approvals.
2. User A (an approver) calls `POST /transactions/{id}/approvers/approve` with `{ approved: true, signature: <valid>, userKeyId: <id> }`.
3. The DB is updated: `approved = true`, `signature = <bytes>`.
4. At line 614: `!dto.approved` = `false`; `userApprovers.every(a => a.approved)` = `false` (pre-update snapshot had `approved = null`).
5. `emitTransactionUpdate` is called — **not** `emitTransactionStatusUpdate`.
6. The notification pipeline receives only a data-change event, not a status-recomputation event.
7. The transaction remains in `WAITING_FOR_SIGNATURES` indefinitely, even though the 1-of-2 threshold is now satisfied.
8. Contrast: if User A calls with `{ approved: false }`, `emitTransactionStatusUpdate` IS called and the pipeline re-evaluates status correctly.
