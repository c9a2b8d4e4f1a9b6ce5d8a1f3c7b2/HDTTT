### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows Invalid Signatures to Advance Transaction State

---

### Summary

The `uploadSignatureMaps` flow in `SignersService` accepts and persists signature maps without performing any cryptographic verification of the signature bytes. An authenticated user can submit a `SignatureMap` containing their registered public key paired with arbitrary/invalid signature bytes. The system will accept the submission, record the user as a signer, update the stored transaction bytes, and potentially advance the transaction status to `WAITING_FOR_EXECUTION`. When the chain service later attempts to execute the transaction on the Hedera network, it will fail because the signature is cryptographically invalid.

This is the direct analog of the external report's finding: after a signature is added/replaced, there is no re-validation step to confirm the new signature is cryptographically sound.

---

### Finding Description

There are **two distinct code paths** for submitting signatures to a transaction:

**Path 1 — `importSignatures` (correct):**
In `transactions.service.ts`, `importSignatures` calls `validateSignature` before adding any signature:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);
```

`validateSignature` in `transaction.ts` performs actual cryptographic verification:

```typescript
const signatureValid = publicKey.verify(bodyBytes, signature);
if (signatureValid) {
  signerPublicKeys.push(publicKey);
} else {
  throw new Error('Invalid signature');
}
```

**Path 2 — `uploadSignatureMaps` (vulnerable):**
In `signers.service.ts`, `processTransactionSignatures` only checks that the public key belongs to the authenticated user, then immediately calls `addSignature` with no cryptographic check:

```typescript
let userKey = userKeyMap.get(raw);
if (!userKey) {
  userKey = userKeyMap.get(publicKey.toStringDer());
}
if (!userKey) throw new Error(ErrorCodes.PNY);

// Only add the signature once per unique key
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
```

The Hedera SDK's `addSignature` does not verify the signature bytes — it unconditionally adds the public key to `_signerPublicKeys` and stores the raw bytes. After this, `processTransactionStatus` evaluates readiness using:

```typescript
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey
);
```

`hasValidSignatureKey` only checks whether the required public keys are present in `_signerPublicKeys` — it does not verify the cryptographic validity of the associated signature bytes. If the threshold is satisfied, the transaction is promoted to `WAITING_FOR_EXECUTION`.

---

### Impact Explanation

A registered user can submit a `SignatureMap` with their legitimate public key but with garbage/forged signature bytes via `POST /transactions/:id/signers`. The system will:

1. Accept the submission and record the user as a signer in `TransactionSigner`
2. Overwrite `transaction.transactionBytes` with the corrupted signature embedded
3. Potentially advance the transaction to `WAITING_FOR_EXECUTION` if the threshold is met
4. Cause the transaction to fail on the Hedera network at execution time with an invalid signature error

This corrupts the stored transaction state and causes guaranteed execution failure for any transaction where this path is the deciding factor for threshold satisfaction. It also permanently overwrites the `transactionBytes` with invalid data, blocking legitimate execution even if other signers later provide valid signatures (since the corrupted bytes are what gets submitted).

---

### Likelihood Explanation

The attack requires only an authenticated user account with at least one registered `UserKey`. No elevated privileges are needed. The endpoint `POST /transactions/:id/signers` is accessible to any verified user who is a signer on the transaction. The attack is trivial to execute: construct a `SignatureMap` with the correct public key but random bytes as the signature value.

---

### Recommendation

Add cryptographic signature verification inside `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`. Specifically, call `validateSignature(sdkTransaction, map)` and only proceed with `addSignature` for keys that pass verification:

```typescript
// Validate signatures cryptographically before adding
const { data: validPublicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

Then restrict `addSignature` calls to only the keys returned by `validateSignature`.

---

### Proof of Concept

1. Register a user account and add a public key `PK` (corresponding to private key `SK`).
2. Create a transaction that requires `PK` to sign.
3. Construct a `SignatureMap` with `PK` as the key but with 64 bytes of `0x00` as the signature value (invalid).
4. Submit `POST /transactions/:id/signers` with this map.
5. Observe: the server accepts the request, inserts a `TransactionSigner` record for the user, and updates `transactionBytes` with the invalid signature embedded.
6. If `PK` was the last required signer, observe the transaction status advance to `WAITING_FOR_EXECUTION`.
7. When the chain service executes the transaction, it fails on the Hedera network with a signature verification error.

```typescript
export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
  } else throw new Error(`Invalid key type`);
};
```
