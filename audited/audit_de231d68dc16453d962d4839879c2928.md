### Title
No Mechanism to Demote an Admin User in the Organization

### Summary
The Hedera Transaction Tool's organization backend exposes a one-way `elevateAdmin` function that can only promote a user to admin (`admin: true`). There is no corresponding demote/revoke endpoint or UI action. Once a user is elevated to admin, their privileges cannot be revoked through any application-level interface. The UI itself explicitly warns: *"This action cannot be undone."*

### Finding Description
The `elevateAdmin` function in `back-end/apps/api/src/auth/auth.service.ts` unconditionally sets `admin: true` with no inverse operation:

```typescript
// auth.service.ts line 162-165
async elevateAdmin(userId: number): Promise<void> {
  await this.usersService.updateUserById(userId, { admin: true });
}
```

The corresponding controller endpoint `PATCH /auth/elevate-admin` is the only admin-role-management endpoint in the auth controller:

```typescript
// auth.controller.ts line 186-190
@Patch('/elevate-admin')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, AdminGuard)
async elevateUser(@Body() dto: ElevateAdminDto): Promise<void> {
  return this.authService.elevateAdmin(dto.id);
}
```

There is no `PATCH /auth/demote-admin` or equivalent endpoint. The UI enforces this irreversibility explicitly in `ElevateContactModal.vue`:

> *"Are you sure you want to assign this Contact as an admin? **This action cannot be undone.**"*

The `ContactDetails.vue` component only renders the "Assign Admin" button when `!contact.user.admin`, meaning once a user is admin, no revocation button is ever shown:

```html
<!-- ContactDetails.vue line 212-219 -->
<template v-if="!contact.user.admin">
  <AppButton @click="$emit('elevate-to-admin')">Assign Admin</AppButton>
</template>
```

The front-end service `elevateUserToAdmin` in `front-end/src/renderer/services/organization/auth.ts` only calls the elevate endpoint and has no demote counterpart.

### Impact Explanation
Admin users in this system hold critical privileges: they can add new users to the organization (`POST /auth/signup`), remove existing users, and elevate other users to admin. If an admin account is compromised or the admin acts maliciously, no other admin can revoke their privileges through the application. The compromised admin retains full organizational control indefinitely. The only workaround — deleting the user entirely — destroys all associated keys and transaction history, which is a destructive and disproportionate action.

### Likelihood Explanation
Admin accounts are long-lived credentials. Credential compromise (phishing, password reuse, session hijacking) is a realistic and common threat. Organizations with multiple admins have no recourse if one admin account is taken over. The irreversibility is not a design choice hidden in code — it is explicitly documented in the UI warning, confirming the developers are aware of the one-way nature but have not provided a remedy.

### Recommendation
Add a `demoteAdmin` function in `AuthService` and a corresponding `PATCH /auth/demote-admin` endpoint (protected by `AdminGuard`) that sets `admin: false` for the target user. Add a "Remove Admin" button in `ContactDetails.vue` (visible only when `contact.user.admin` is true and the acting user is also an admin). Ensure an admin cannot demote themselves to prevent accidental lockout.

### Proof of Concept

1. Admin A elevates User B to admin via `PATCH /auth/elevate-admin` with `{ id: B_id }`.
2. User B's account is compromised by an attacker.
3. Admin A attempts to revoke B's admin status — no endpoint or UI action exists to do so.
4. The attacker, operating as admin B, can now add rogue users (`POST /auth/signup`), remove legitimate users (`DELETE /users/:id`), or elevate additional attacker-controlled accounts to admin.
5. Admin A's only recourse is to delete User B entirely (`DELETE /users/:id`), which permanently destroys B's keys and organization data — a destructive, irreversible action that may itself cause operational harm.
