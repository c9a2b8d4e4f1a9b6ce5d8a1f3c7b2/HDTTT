### Title
Unbounded Recursive Approver Tree Creation Enables DoS via Stack Overflow

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` uses an internally recursive `createApprover` helper with no depth limit. The `CreateTransactionApproverDto` DTO is self-referential with no nesting cap enforced at the validation layer. Any authenticated user who owns a transaction can submit an arbitrarily deeply nested `approversArray` payload, exhausting the Node.js call stack and crashing the API service process.

### Finding Description

**Root cause — no depth limit on recursive DTO and recursive handler:**

The DTO `CreateTransactionApproverDto` is self-referential:

```
approvers?: CreateTransactionApproverDto[];
```

with only `@ValidateNested({ each: true })` and `@Type(() => CreateTransactionApproverDto)` — no `@ArrayMaxSize`, no depth cap. [1](#0-0) 

The service's inner `createApprover` function calls itself for every nested child:

```ts
await createApprover({ ...nestedDtoApprover, listId: approver.id });
``` [2](#0-1) 

The only structural validation performed is `validateApprover`, which only checks that the approver is not completely empty — it does not check nesting depth. [3](#0-2) 

**Exploit path:**

1. Attacker registers a normal user account and creates a transaction (standard API flow).
2. Attacker calls `POST /transactions/:transactionId/approvers` with a JSON body containing a `approversArray` whose `approvers` field is nested thousands of levels deep, each level containing `threshold: 1` and a single child.
3. NestJS deserializes and validates the payload recursively via `class-transformer`/`class-validator` (itself unbounded), then passes it to `createTransactionApprovers`.
4. `createApprover` recurses once per nesting level with no guard. At sufficient depth (~10 000+ frames on a default V8 stack), the process throws `RangeError: Maximum call stack size exceeded`.
5. Because the error propagates outside the `try/catch` that only wraps the `dataSource.transaction` callback, or because the stack overflow is a hard V8 error, the Node.js worker/process crashes.

The controller endpoint is guarded only by JWT auth and `VerifiedUserGuard` — no rate-limit or payload-depth guard is present. [4](#0-3) 

### Impact Explanation
A single authenticated user can crash the API service process with one HTTP request, causing a complete denial of service for all other users. Because the crash is a hard V8 stack overflow (not a caught exception), NestJS's global exception filter cannot intercept it; the process must be restarted by the process manager (e.g., Kubernetes pod restart). Repeated requests keep the service unavailable.

### Likelihood Explanation
The precondition is only a valid JWT token and ownership of any transaction — both are reachable by any registered user with no elevated privileges. The attack requires crafting a single deeply nested JSON object, which is trivial. No rate-limiting or payload-depth validation exists at any layer between the HTTP boundary and the recursive call.

### Recommendation

1. **Enforce a maximum nesting depth in the DTO** — add `@ArrayMaxSize(N)` at each level and introduce a custom depth-limit decorator, or flatten the DTO to disallow inline nesting entirely (children must be added via separate API calls referencing `listId`).
2. **Add a depth counter to `createApprover`** — pass a `depth` parameter and throw a `BadRequestException` when it exceeds a safe limit (e.g., 10):
   ```ts
   const createApprover = async (dto, depth = 0) => {
     if (depth > MAX_DEPTH) throw new Error('Approver tree too deep');
     ...
     await createApprover(child, depth + 1);
   };
   ```
3. **Add a global request body size / JSON depth limit** at the NestJS/Express layer (e.g., `json({ limit: '100kb' })` and a custom JSON depth parser).

### Proof of Concept

```
POST /transactions/1/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            ... // repeated ~15 000 times
            { "userId": 1 }
          ]
        }
      ]
    }
  ]
}
```

**Expected outcome:** The API process throws `RangeError: Maximum call stack size exceeded` inside `createApprover` (or inside `class-transformer`'s recursive `plainToInstance`), crashing the NestJS worker. All in-flight requests fail and the service is unavailable until the process manager restarts it. The attacker can repeat the request immediately after each restart to maintain the outage.
