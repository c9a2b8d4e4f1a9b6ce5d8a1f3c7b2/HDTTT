### Title
`isNode()` Uniqueness Check Bypassed for Threshold Approver Nodes, Allowing Single-User Approval to Satisfy Multiple Threshold Requirements

### Summary
`ApproversService.isNode()` is the sole server-side guard against duplicate approver entries. Its return expression unconditionally short-circuits to `false` for any approver that lacks a `userId` (i.e., every threshold/tree node). A transaction creator can therefore POST duplicate threshold nodes containing the same user(s). Because `approveTransaction()` updates **all** database rows that match `userId === user.id`, a single approval action simultaneously satisfies every duplicate threshold node, allowing a transaction to advance with fewer real approvals than the structure visually implies.

### Finding Description

**Root cause — `isNode()` never detects duplicate threshold nodes** [1](#0-0) 

```typescript
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';   // ← always false for threshold nodes
```

For a threshold node `userId` is `undefined`, so `typeof approver.userId === 'number'` is `false`. The function returns `false` regardless of `count`, meaning the duplicate guard at line 250 never fires for threshold nodes. [2](#0-1) 

**Exploit path — creating duplicate threshold nodes**

A creator POSTs to `POST /transactions/:id/approvers` with:

```json
{
  "approversArray": [
    { "threshold": 1, "approvers": [{ "userId": 2 }] },
    { "threshold": 1, "approvers": [{ "userId": 2 }] }
  ]
}
```

`createTransactionApprovers` iterates `dto.approversArray` and calls `createApprover` for each entry. [3](#0-2) 

Because `isNode()` returns `false` for both threshold entries, both are inserted. The database has no unique constraint on `(transactionId, threshold, listId)` for `transaction_approver`, so the inserts succeed. The result is two structurally identical root threshold nodes, each with its own child row for `userId=2`.

**Exploit path — single approval satisfies both nodes**

`approveTransaction` collects every row where `userId === user.id` across the entire approver tree: [4](#0-3) 

It then bulk-updates **all** of those rows in one query: [5](#0-4) 

Because User 2 appears in both duplicate threshold nodes, a single call to `/approve` marks both child rows as signed. Both threshold nodes (each requiring 1-of-1) are now satisfied. If the root structure requires 2-of-2 (both threshold nodes), the transaction advances as if two independent approvals were received, when only one user acted.

### Impact Explanation

The Hedera Transaction Tool is used in Organization Mode to enforce multi-party authorization before Hedera network transactions (HBAR transfers, account operations, etc.) are submitted. A malicious creator can construct a deceptive approver tree that visually appears to require N independent approvals but structurally requires only 1. This allows the creator to execute Hedera transactions — which may move real HBAR or tokens — without obtaining the number of independent approvals the organization intended. This is an unauthorized state change and potential unauthorized movement of on-chain assets.

### Likelihood Explanation

- The attacker is the transaction creator, a role available to any authenticated organization user — no admin or privileged key is required.
- The exploit is a straightforward API call with a crafted JSON body; no timing, race condition, or cryptographic weakness is needed.
- There is no client-side or server-side control that prevents the duplicate payload from being submitted.
- The `isNode()` flaw is present in the current codebase with no compensating database-level unique constraint on threshold nodes.

### Recommendation

Fix `isNode()` to return `true` whenever a matching record exists, regardless of whether `userId` is set:

```typescript
// current (broken)
return count > 0 && typeof approver.userId === 'number';

// fixed
return count > 0;
```

Additionally, add a database-level partial unique index on `transaction_approver` to prevent duplicate threshold nodes at the persistence layer:

```sql
CREATE UNIQUE INDEX uq_threshold_node
  ON transaction_approver ("transactionId", "threshold", "listId")
  WHERE "userId" IS NULL AND "deletedAt" IS NULL;
```

### Proof of Concept

**Setup**: Organization with two users — Alice (creator, `userId=1`) and Bob (`userId=2`). Alice creates a transaction that is supposed to require Bob's approval plus one other independent approval.

**Step 1 — Alice creates the transaction** (normal flow).

**Step 2 — Alice POSTs duplicate threshold nodes**:

```http
POST /transactions/42/approvers
Authorization: Bearer <alice-token>
Content-Type: application/json

{
  "approversArray": [
    { "threshold": 1, "approvers": [{ "userId": 2 }] },
    { "threshold": 1, "approvers": [{ "userId": 2 }] }
  ]
}
```

Server inserts four rows: two threshold nodes (IDs 10, 11) and two child user-approver rows (IDs 12, 13), both pointing to `userId=2`.

**Step 3 — Bob approves once**:

```http
POST /transactions/42/approvers/approve
Authorization: Bearer <bob-token>
Content-Type: application/json

{ "userKeyId": 5, "signature": "...", "approved": true }
```

`approveTransaction` finds `userApprovers = [row 12, row 13]` and updates both with Bob's signature in a single query.

**Expected outcome**: Both threshold nodes are now marked satisfied. The transaction status transitions as if two independent approvals were received, even though only Bob acted. Alice's transaction proceeds without the intended second independent approver. [1](#0-0) [2](#0-1) [4](#0-3) [5](#0-4)
