### Title
Unbounded Scheduler Query Over Attacker-Inflatable Pending Transactions Causes Chain Service Resource Exhaustion

### Summary

The `updateTransactions` method in `TransactionSchedulerService` fetches **all** pending transactions matching a status and time-range filter with no row limit. Any authenticated user can create arbitrarily many transactions with future `validStart` dates through the normal API. As the pending-transaction table grows, every cron tick loads the entire unbounded result set into memory and runs CPU-intensive per-row processing (`SDKTransaction.fromBytes`, `computeSignatureKey`, `smartCollate`), degrading or crashing the chain service and preventing legitimate transaction execution.

---

### Finding Description

**Root cause — unbounded `find` in the scheduler**

`updateTransactions` issues a TypeORM `find` with no `take` (LIMIT) clause: [1](#0-0) 

The call at line 134 has no `take` field, so TypeORM emits `SELECT … WHERE status IN (…) AND validStart … ORDER BY validStart ASC` with no `LIMIT`. Every matching row is loaded into the Node.js process.

The most dangerous invocation is `handleInitialTransactionStatusUpdate`, which passes only a lower bound (`MoreThan(from)`) and no upper bound, meaning it fetches **every** pending transaction from three minutes ago to the end of time: [2](#0-1) 

The every-10-second cron `handleTransactionsBetweenNowAndAfterThreeMinutes` also calls `updateTransactions` and then `prepareTransactions`, iterating the full result set: [3](#0-2) 

For each transaction in the result, `processTransactionStatus` deserializes the raw bytes, calls `computeSignatureKey`, and calls `smartCollate` — all CPU-intensive operations: [4](#0-3) 

**Attacker-controlled inflation path**

Any authenticated user can call `POST /transactions` with a batch of `CreateTransactionDto` objects. The service method accepts an unbounded array: [5](#0-4) 

The only rate-limiting in place is a global IP-based throttler (minute and second limits): [6](#0-5) 

This throttler limits request frequency but does not cap the total number of transactions a user may accumulate over time. By submitting batches of transactions with far-future `validStart` values (keeping them perpetually in `WAITING_FOR_SIGNATURES` status), an attacker steadily inflates the pending-transaction table. The transactions never expire because their `validStart` is in the future, so the expiry cron never removes them: [7](#0-6) 

The pagination decorator that caps page size at 100 applies only to read endpoints, not to the create endpoint or the internal scheduler: [8](#0-7) 

---

### Impact Explanation

Once the pending-transaction table is sufficiently large, every cron tick of the chain service:

1. Executes a full-table scan loading all rows into Node.js heap — memory exhaustion.
2. Runs `SDKTransaction.fromBytes` + `computeSignatureKey` + `smartCollate` for every row — CPU starvation.
3. Blocks the event loop, preventing legitimate transactions from being collated and executed on Hedera.

The result is a complete or severe degradation of the chain service's ability to execute any transaction, including those from unrelated users. This is a permanent condition as long as the inflated rows remain in the database.

---

### Likelihood Explanation

- **Attacker precondition**: A valid account on the organization back-end. Registration is the only barrier.
- **No privileged access required**: `POST /transactions` is a standard authenticated endpoint.
- **Gradual, stealthy attack**: The attacker spreads requests over hours or days, staying within per-minute throttle limits. No burst is needed.
- **Persistence**: Transactions with far-future `validStart` are never expired by the scheduler, so the inflated state is permanent until an admin manually purges rows.

---

### Recommendation

1. **Add a `take` limit to `updateTransactions`**: Process pending transactions in bounded batches (e.g., 500 rows per cron tick) using cursor-based or offset pagination.
2. **Cap per-user pending transaction count**: Enforce a maximum number of non-terminal transactions per user at creation time (e.g., reject if the user already has more than N pending transactions).
3. **Limit batch size on `createTransactions`**: Add a `@ArrayMaxSize(N)` class-validator constraint to the DTO array accepted by the create endpoint.
4. **Add a maximum `validStart` horizon**: Reject transactions whose `validStart` is more than, say, 30 days in the future, to bound how long rows remain in the pending state.

---

### Proof of Concept

1. Register a user account on the organization back-end.
2. Obtain a JWT via the login endpoint.
3. In a loop (respecting the per-minute throttle), POST batches of transactions to `POST /transactions`, each with `validStart` set 365 days in the future and `status: WAITING_FOR_SIGNATURES`. Each batch can contain multiple DTOs in the array body.
4. After accumulating tens of thousands of rows, observe the chain service's memory and CPU usage spike on every 10-second cron tick of `handleTransactionsBetweenNowAndAfterThreeMinutes`.
5. Legitimate transactions submitted by other users will fail to be collated and executed because the event loop is saturated.

The unbounded `find` at `transaction-scheduler.service.ts:134` with no `take` is the direct root cause, mirroring the unbounded `for` loop over `undeletedProposalStart` in the original `Gov.sol` report.

  async updateTransactions(from: Date, to?: Date) {
    //Get the transaction, creatorKey, groupItem, and group. We need the group info upfront
    //in order to determine if the group needs to be processed together
    const transactions = await this.transactionRepo.find({
      where: {
        status: In([
          TransactionStatus.WAITING_FOR_SIGNATURES,
          TransactionStatus.WAITING_FOR_EXECUTION,
        ]),
        validStart: to ? Between(from, to) : MoreThan(from),
      },
      relations: {
        creatorKey: true,
        groupItem: {
          group: true,
        },
      },
      order: {
        validStart: 'ASC',
      },
    });

    const results = await processTransactionStatus(this.transactionRepo, this.transactionSignatureService, transactions);

    if (results.size > 0) {
      const events = Array.from(results.keys(), id => ({ entityId: id }));
      emitTransactionStatusUpdate(this.notificationsPublisher, events);
    }

    return transactions;
  }
