### Title
Missing Proof-of-Key-Ownership in `uploadKey` Allows Public Key Squatting, Blocking Legitimate Users from Multi-Sig Participation

### Summary
The `POST /user/:userId?/keys` endpoint in the organization backend accepts any Hedera public key from any authenticated user without requiring a cryptographic proof that the submitter controls the corresponding private key. A malicious organization member can register another user's public key under their own account before the legitimate owner does, permanently preventing the legitimate owner from registering that key and participating in multi-signature transaction workflows.

### Finding Description

**Root cause:** `uploadKey` in `back-end/apps/api/src/user-keys/user-keys.service.ts` performs no proof-of-ownership check. The only guard is a uniqueness check: if the key already exists in the database under a *different* user, it throws `ErrorCodes.PU`. If the key does not yet exist, it is registered for whoever submits it first. [1](#0-0) 

The DTO accepted by the endpoint contains only `publicKey`, `mnemonicHash`, and `index` — no signature field, no challenge-response, no proof that the submitter holds the private key. [2](#0-1) 

The controller exposes this endpoint to any verified, authenticated user: [3](#0-2) 

**Contrast with transaction creation:** When a transaction is *created*, the backend correctly verifies that the submitter controls the creator key by checking `publicKey.verify(dto.transactionBytes, dto.signature)`. No equivalent check exists for key registration. [4](#0-3) 

**Exploit flow:**

1. Alice's Hedera public key is visible on-chain (standard for Hedera accounts).
2. Bob, a malicious but authenticated organization member, calls `POST /user/123/keys` with Alice's public key before Alice registers it.
3. The key is stored under Bob's `userId`.
4. When Alice later calls the same endpoint with her own public key, the service finds the key already exists under a different user and throws `ErrorCodes.PU`. [5](#0-4) 

5. Alice can never register her key in the organization backend.
6. When a multi-sig transaction requires Alice's key, the `processTransactionSignatures` flow builds a `userKeyMap` from each user's registered keys. Since Alice's key is registered under Bob's account, Alice cannot upload a valid signature for it — the system will reject her signature upload with `ErrorCodes.PNY` (key does not belong to user). [6](#0-5) 

### Impact Explanation

Any multi-signature transaction that requires Alice's key is permanently blocked from completion through the organization backend. For the Hedera Council use case — where the tool is used for high-stakes governance operations (file updates, account management, network parameter changes) — blocking a required signer's key registration can halt critical network operations indefinitely. The legitimate key owner has no recovery path within the application; an admin would need to manually intervene at the database level.

### Likelihood Explanation

- **Attacker precondition**: Must be an authenticated, verified organization member. This is a realistic insider-threat scenario.
- **Key discovery**: Hedera public keys are on-chain and publicly queryable via the mirror node API. An attacker can enumerate all organization members' Hedera account IDs and retrieve their public keys before those users register them.
- **Timing**: The attack only needs to succeed once, before the victim registers their key. In a newly onboarded organization, this window is wide open.
- **No special tooling required**: A single authenticated HTTP POST is sufficient.

### Recommendation

Require a proof-of-ownership signature when registering a public key. The submitter should sign a deterministic challenge (e.g., `SHA256("register-key" || userId || publicKey)`) with the private key corresponding to the public key being registered. The backend verifies this signature before persisting the key. This is the same pattern already used for transaction creation (`publicKey.verify(dto.transactionBytes, dto.signature)`). [4](#0-3) 

The `UploadUserKeyDto` should be extended with a mandatory `ownershipProof` field (the signature bytes), and `uploadKey` should verify it before the uniqueness check.

### Proof of Concept

```
# Step 1: Bob (attacker) obtains Alice's public key from the Hedera mirror node
GET https://mainnet-public.mirrornode.hedera.com/api/v1/accounts/0.0.ALICE_ACCOUNT

# Step 2: Bob registers Alice's public key under his own account
POST https://org-backend/user/BOB_USER_ID/keys
Authorization: Bearer BOB_JWT
Content-Type: application/json

{ "publicKey": "ALICE_PUBLIC_KEY_RAW" }

# Response: 201 Created — key is now owned by Bob

# Step 3: Alice attempts to register her own key
POST https://org-backend/user/ALICE_USER_ID/keys
Authorization: Bearer ALICE_JWT
Content-Type: application/json

{ "publicKey": "ALICE_PUBLIC_KEY_RAW" }

# Response: 400 Bad Request — ErrorCodes.PU ("Public key already in use")

# Step 4: Any transaction requiring Alice's signature is now permanently blocked.
# Alice cannot upload signatures; the system maps her public key to Bob's userId,
# and her upload attempt fails with ErrorCodes.PNY.
```
