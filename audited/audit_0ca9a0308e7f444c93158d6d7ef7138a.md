### Title
Approver Can Flip Vote After Submission When Assigned to Multiple Positions in the Approver Tree

### Summary

The `approveTransaction` function in `ApproversService` guards against re-submission using `userApprovers.every(a => a.signature)`. This guard only blocks re-approval when **all** of a user's approver records already carry a signature. Because the system permits the same user to occupy multiple positions in the approver tree (e.g., as a direct approver and as a member of a threshold group), a user with at least one unsigned record can call `approveTransaction` again, and the bulk update will overwrite **all** their records â€” including already-signed ones â€” with a new `approved` value. This is the direct analog of the `GraphGovernance.updateProposal` issue: a resolution that should be permanent can be silently reversed.

---

### Finding Description

**Root cause â€” guard logic** [1](#0-0) 

```ts
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

`userApprovers` is the slice of all `TransactionApprover` rows whose `userId` matches the caller. The guard throws only when **every** row already has a signature. If the user holds two rows and only one is signed, `every` returns `false`, the guard is bypassed, and execution continues.

**Bulk overwrite** [2](#0-1) 

```ts
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
  .whereInIds(userApprovers.map(a => a.id))   // ALL rows, including already-signed ones
  .execute();
```

The `whereInIds` clause targets every row in `userApprovers`, so the already-signed row is overwritten with whatever `approved` value the caller supplies in the second request.

**How a user ends up in multiple positions**

`createTransactionApprovers` deduplicates via `isNode`, which matches on the exact combination of `(userId, listId, threshold, transactionId)`. [3](#0-2) 

A user added as a **direct approver** (`listId = null, transactionId = X`) and also as a **member of a threshold group** (`listId = Y, transactionId = null`) produces two distinct rows that both pass `isNode`, so the system creates both without error.

**Exploit flow**

1. Transaction creator builds an approver tree that includes User B as a direct approver **and** as a member of a threshold group â€” both are valid, distinct records.
2. User B calls `POST /transactions/:transactionId/approvers/approve` with `approved: true`. The direct-approver row gets `signature` + `approved = true`. The threshold-group row remains unsigned.
3. User B calls the same endpoint again with `approved: false`. Guard check: `every(a => a.signature)` â†’ `false` (threshold-group row is still unsigned). Guard passes.
4. The bulk update overwrites **both** rows with `approved = false`, reversing the already-submitted approval.
5. The downstream `processTransactionStatus` scheduler now sees User B's vote as a rejection, potentially blocking a transaction that had already met its approval threshold.

---

### Impact Explanation

An approver can silently reverse a previously submitted approval or rejection while the transaction is still in `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` status. This undermines the integrity of the multi-signature approval process: a threshold that was legitimately reached can be undone, or a rejection can be converted to an approval, without any audit trail distinguishing the flip from an ordinary first-time vote.

---

### Likelihood Explanation

The precondition â€” a user appearing in multiple approver positions â€” is reachable through the normal API without any privileged access. The transaction creator (who may be the attacker, or who may innocently construct a complex threshold tree) can place the same user in two positions. No leaked credentials, no admin keys, and no cryptographic breaks are required. The attacker only needs a valid JWT for an account that holds at least two approver records on the target transaction.

---

### Recommendation

Change the bulk update to target **only** the unsigned rows:

```ts
.whereInIds(userApprovers.filter(a => !a.signature).map(a => a.id))
```

Additionally, strengthen the guard to throw if **any** of the user's approver records already carries a signature, making the first submission permanent:

```ts
if (userApprovers.some(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

This mirrors the intent documented in the guard comment ("Check if the user has already approved the transaction") and closes the multi-record bypass.
