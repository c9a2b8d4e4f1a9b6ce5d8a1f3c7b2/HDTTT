### Title
Missing `@Min(1)` on `threshold` in `CreateTransactionApproverDto` Allows Negative Threshold to Bypass Multi-Signature Approval Requirement

---

### Summary

`CreateTransactionApproverDto` does not enforce a minimum value of `1` on the `threshold` field, while the analogous `UpdateTransactionApproverDto` correctly applies `@Min(1)`. The service-level guard in `approvers.service.ts` only rejects `threshold === 0` and `threshold > approvers.length`, leaving negative threshold values unchecked. A negative threshold stored in the database causes the approval-count check to be trivially satisfied, effectively bypassing the multi-signature approval requirement for a transaction.

---

### Finding Description

**DTO-level gap — `CreateTransactionApproverDto`:** [1](#0-0) 

The `threshold` field carries only `@IsNumber()` and `@IsOptional()`. No `@Min(1)` decorator is present.

**Contrast with `UpdateTransactionApproverDto`:** [2](#0-1) 

The update path correctly applies `@Min(1)`, demonstrating that the constraint is known and intentional — it was simply omitted from the create path.

**Service-level guard is incomplete:** [3](#0-2) 

The condition `dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0` catches zero and values exceeding the child count, but **not negative values**. For example, `threshold = -1` with two child approvers evaluates as:
- `-1 > 2` → `false`
- `-1 === 0` → `false`

No exception is thrown; the negative threshold is persisted to the database.

---

### Impact Explanation

The `threshold` field controls how many child approvers must approve before a transaction is considered approved. If a negative threshold (e.g., `-1`) is stored, any approval-count comparison of the form `approvedCount >= threshold` is trivially true for any non-negative count, including zero. This means a transaction configured with a negative threshold approver tree requires **zero actual approvals** to proceed, completely defeating the multi-signature workflow that is the core security guarantee of the tool.

---

### Likelihood Explanation

The API endpoint is authenticated (requires a valid JWT), so an anonymous attacker cannot exploit this directly. However, any authenticated organization user who is the creator of a transaction can submit the `POST /transactions/:id/approvers` request with a crafted negative `threshold`. The front-end UI constrains the select widget to values `≥ 1`, but the back-end performs no equivalent enforcement at the DTO layer, making a direct API call the trivial attack path. [4](#0-3) 

---

### Recommendation

Add `@Min(1)` to the `threshold` field in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-approver.dto.ts
import { IsArray, IsNumber, IsOptional, ValidateNested, ArrayMinSize, Min } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @IsOptional()
  listId?: number;

  @IsNumber()
  @Min(1)          // <-- add this
  @IsOptional()
  threshold?: number;
  ...
}
```

Additionally, harden the service-level guard to explicitly reject non-positive thresholds:

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold <= 0)
)
  throw new Error(...);
```

### Proof of Concept

```http
POST /transactions/1/approvers HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": -1,
      "approvers": [
        { "userId": 2 },
        { "userId": 3 }
      ]
    }
  ]
}
```

**Expected:** `400 Bad Request` — threshold must be ≥ 1.

**Actual:** `201 Created` — the approver tree is persisted with `threshold = -1`. Any subsequent approval-count evaluation against this tree is trivially satisfied, allowing the transaction to advance without collecting the required signatures. [5](#0-4) [6](#0-5) [7](#0-6)
