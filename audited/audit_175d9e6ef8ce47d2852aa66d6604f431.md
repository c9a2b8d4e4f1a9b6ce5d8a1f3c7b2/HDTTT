### Title
Unbounded Loop Over All Non-Terminal Transactions in `getTransactionsToSign` Causes Server-Side Resource Exhaustion DoS

### Summary
`getTransactionsToSign` in `TransactionsService` fetches every non-terminal transaction from the database with no server-side row limit, then iterates over the full result set performing expensive per-transaction operations (byte deserialization, async signature-key computation, and potential mirror-node cache calls) before applying pagination. As the transaction table grows — which any authenticated user can influence by creating transactions — each call to this endpoint consumes proportionally more CPU and memory, enabling a sustained resource-exhaustion DoS against the API service.

### Finding Description

**Root cause — unbounded DB fetch followed by a full-table loop:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues a `repo.find` with no `take` (limit) clause:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then iterates over every returned row, calling `userKeysToSign` for each one:

```ts
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

Pagination is applied only **after** the loop completes:

```ts
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

**Per-iteration cost — each loop body is expensive:**

`userKeysToSign` delegates to `userKeysRequiredToSign`, which calls `transactionSignatureService.computeSignatureKey`. That function:
1. Deserializes raw transaction bytes via `SDKTransaction.fromBytes`.
2. Makes one or more async calls to `accountCacheService.getAccountInfoForTransaction` (mirror-node backed).
3. Constructs a `KeyList` from the results. [4](#0-3) 

**Endpoint is still publicly reachable:**

The controller exposes `GET /transactions/sign` to any authenticated, verified user. The comment "NO LONGER USED BY FRONT-END" does not remove the route: [5](#0-4) 

**Rate limits are permissive:**

The deployed configuration sets `GLOBAL_MINUTE_LIMIT: "10000"` and `GLOBAL_SECOND_LIMIT: "1000"`, giving an attacker ample request budget. [6](#0-5) 

### Impact Explanation

Every call to `GET /transactions/sign` forces the server to load and process the entire non-terminal transaction table. With N active transactions, the server performs N deserialization operations and up to N×(accounts per transaction) async cache/mirror-node lookups per request. At scale this saturates the Node.js event loop and the PostgreSQL connection pool (`POSTGRES_MAX_POOL_SIZE: "3"`), causing request timeouts and API unavailability for all users — not just the attacker. The API pod's memory limit is only 250 Mi in the default deployment, making OOM termination a realistic outcome. [7](#0-6) 

### Likelihood Explanation

- **Attacker precondition**: A valid, verified account — the lowest privilege level in the system.
- **Inflation vector**: Any verified user can create transactions (`POST /transactions`). Creating thousands of transactions inflates the table that `getTransactionsToSign` scans.
- **Trigger**: Repeatedly calling `GET /transactions/sign` with any pagination parameters. The server-side cost is independent of the requested page size.
- **No special knowledge required**: The endpoint path and authentication mechanism are standard REST conventions documented in the API.

### Recommendation

Apply the database-level `take` limit **before** the loop, not after. The correct pattern is to push the key-matching logic into the database query (as a subquery or join against `transaction_signer`/`user_key` tables), mirroring the approach already used by `getTransactionsToApprove` which uses a proper `getManyAndCount` with `skip`/`take`: [8](#0-7) 

If the in-process key-matching loop must be retained, enforce a hard cap on the number of rows fetched (e.g., `take: MAX_SCAN_ROWS`) and document the limitation. Additionally, consider removing or gating the `/transactions/sign` endpoint entirely since the front-end no longer uses it.

### Proof of Concept

1. Register and verify a user account on the API.
2. Create a large number of transactions (e.g., 10,000) via `POST /transactions` using the verified account's JWT. Each transaction only needs to be syntactically valid.
3. From any authenticated session, repeatedly issue:
   ```
   GET /transactions/sign?page=1&size=1
   ```
4. **Observed**: Each request causes the server to load all N non-terminal transactions, deserialize each one, and compute signature keys — regardless of the `size=1` parameter. CPU usage climbs linearly with N; at sufficient scale the Node.js process stalls or the pod is OOM-killed, returning 503/504 errors to all concurrent users.
5. **Expected**: The query should be bounded at the database level so that only `size` rows are processed per request.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
