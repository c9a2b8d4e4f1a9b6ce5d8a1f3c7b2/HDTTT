### Title
Missing Cryptographic Signature Validation and Access Control in `uploadSignatureMaps` Allows Any Authenticated User to Corrupt Transaction Bytes

### Summary
The `uploadSignatureMaps` function in `signers.service.ts` does not cryptographically verify submitted signatures before adding them to stored transaction bytes, and does not enforce that the submitting user has access to the target transaction. Any authenticated user can submit an invalid signature for their own registered key against any transaction, corrupting the stored `transactionBytes` and causing the transaction to fail permanently when executed on-chain. This is the direct analog to the external report's pattern: arbitrary input data bypasses validation, causing unauthorized state mutation.

### Finding Description

**Vulnerability class**: Input validation bypass → unauthorized state mutation → permanent DoS on target transaction.

**Root cause — missing cryptographic validation:**

In `signers.service.ts`, `processTransactionSignatures` iterates the caller-supplied `SignatureMap`, checks only that each public key belongs to the authenticated user, and then unconditionally calls `sdkTransaction.addSignature(publicKey, map)` without verifying that the signature bytes are cryptographically valid for the transaction body: [1](#0-0) 

The Hedera SDK's `addSignature` appends the signature bytes to the inner signed transactions without performing cryptographic verification — that is left to the network at submission time. The comment in the code itself confirms this is a raw byte-append operation.

Contrast this with `importSignatures` in `transactions.service.ts`, which explicitly calls `validateSignature` before any `addSignature` call: [2](#0-1) 

The `validateSignature` utility in `back-end/libs/common/src/utils/sdk/transaction.ts` performs `publicKey.verify(bodyBytes, signature)` for each entry and throws on failure: [3](#0-2) 

The `/signers` path (`uploadSignatureMaps`) has no equivalent check.

**Root cause — missing access control:**

`loadTransactionData` fetches transactions by ID with no user filter: [4](#0-3) 

`validateAndProcessSignatures` performs no `verifyAccess` call: [5](#0-4) 

Again, contrast with `importSignatures`, which gates every transaction on `verifyAccess`: [6](#0-5) 

**Combined exploit path:**

1. Attacker (any authenticated, verified user) picks a target transaction ID (sequential integers, trivially enumerable).
2. Attacker constructs a `SignatureMap` containing their own registered public key paired with an all-zero or otherwise invalid signature byte string.
3. Attacker POSTs to `POST /transactions/{id}/signers` (protected only by JWT + verified-user guards): [7](#0-6) 

4. `processTransactionSignatures` finds the attacker's public key in `userKeyMap` (ownership check passes), calls `addSignature` with the invalid bytes, and returns `isSameBytes = false`.
5. `persistSignatureChanges` writes the corrupted `transactionBytes` to the database: [8](#0-7) 

6. The attacker is also recorded as a `TransactionSigner`, falsely advancing the signer count.
7. When the chain service later executes the transaction, the Hedera network rejects it because the invalid signature fails verification. The transaction status becomes `FAILED` — a terminal, unrecoverable state. [9](#0-8) 

### Impact Explanation

- **Permanent DoS on any in-flight transaction**: Once corrupted `transactionBytes` are persisted, the transaction cannot be recovered. The `FAILED` status is terminal.
- **False signer records**: The attacker is recorded as a legitimate signer, corrupting audit trails and potentially satisfying threshold counts prematurely.
- **No asset recovery path**: Hedera transactions are immutable once submitted; a failed transaction cannot be re-executed with the same transaction ID.
- **Scope**: Every transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status is reachable by any authenticated user.
