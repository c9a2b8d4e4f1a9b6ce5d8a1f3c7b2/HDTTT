### Title
Transaction Group Creator Can Delete Transactions in `WAITING_FOR_EXECUTION` State, Causing DB/Chain State Divergence

### Summary
`removeTransactionGroup` hard-deletes all transactions in a group without checking their current status. When a transaction is already in `WAITING_FOR_EXECUTION` state and the chain service has scheduled it for on-chain submission, the creator can delete the DB record while the Hedera transaction still executes on-chain. This produces an irrecoverable divergence between the organization's database and the Hedera network state, permanently destroying the audit trail for an executed transaction.

### Finding Description

**Root cause — missing status guard in `removeTransaction` when called with `softRemove=false`:**

`removeTransactionGroup` iterates over every group item and calls `removeTransaction` with the hard-delete flag:

```typescript
// transaction-groups.service.ts line 187-191
for (const groupItem of groupItems) {
  const transactionId = groupItem.transactionId;
  await this.dataSource.manager.remove(TransactionGroupItem, groupItem);
  await this.transactionsService.removeTransaction(transactionId, user, false); // softRemove=false
}
``` [1](#0-0) 

`removeTransaction` with `softRemove=false` performs a raw `repo.remove()` with **no status check**:

```typescript
// transactions.service.ts line 629-637
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);
  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction);   // ← no status guard
  }
``` [2](#0-1) 

Contrast this with `cancelTransaction`, which explicitly rejects non-cancelable statuses:

```typescript
// transactions.service.ts line 669-671
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [3](#0-2) 

**Chain service execution path:**

The chain service's scheduler runs every 10 seconds, picks up `WAITING_FOR_EXECUTION` transactions, and registers named `setTimeout` callbacks that submit the transaction to Hedera:

```typescript
// transaction-scheduler.service.ts line 87-97
@Cron(CronExpression.EVERY_10_SECONDS, ...)
async handleTransactionsBetweenNowAndAfterThreeMinutes() {
  const transactions = await this.updateTransactions(...);
  await this.prepareTransactions(transactions);
}
``` [4](#0-3) 

`collateAndExecute` loads the transaction object into memory and registers a timeout. The timeout fires using the **in-memory** transaction bytes, independent of the DB record: [5](#0-4) 

Once the timeout is registered, deleting the DB row does not cancel the scheduled submission. The Hedera network will still receive and process the transaction.

**Exploit flow:**

1. Attacker (transaction creator) creates a transaction group with a HBAR-transfer or token-operation transaction.
2. Required signers sign; transaction advances to `WAITING_FOR_EXECUTION`.
3. Chain service scheduler picks it up and registers a `smart_collate_timeout_<id>` callback.
4. Creator immediately calls `DELETE /transaction-groups/:id`.
5. `removeTransactionGroup` → `removeTransaction(id, user, false)` hard-deletes the DB row — no status check fires.
6. The scheduled timeout fires; the chain service submits the transaction to Hedera using the in-memory bytes.
7. Hedera executes the transaction (fees deducted, state changed on-chain).
8. The chain service attempts to update the now-deleted DB row → silent failure or unhandled error.
9. The organization's DB has zero record of the executed transaction.

### Impact Explanation

- **Irrecoverable audit trail loss**: An on-chain Hedera transaction executes with no corresponding DB record. The organization cannot reconcile its internal state with the network.
- **Corruption of multi-party workflow integrity**: Signers, approvers, and observers who participated in the transaction lose all evidence of their participation and the outcome.
- **Silent on-chain execution**: Because the chain service holds the transaction bytes in memory at the time of deletion, the Hedera transaction is submitted regardless. The creator can use this to execute a transaction and then erase the organizational record of it.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The window is the period between the chain service scheduling the timeout (up to 3 minutes before `validStart`) and the actual submission. The `DELETE /transaction-groups/:id` endpoint is a standard REST call. No race condition tooling is required; the creator simply needs to delete the group after signatures are collected but before `validStart`. [6](#0-5) 

### Recommendation

Add a status guard inside `removeTransaction` (or at the call site in `removeTransactionGroup`) that rejects deletion when the transaction is in a non-cancelable or actively-executing state:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  const nonDeletableStatuses = [
    TransactionStatus.WAITING_FOR_EXECUTION,
    TransactionStatus.EXECUTED,
    TransactionStatus.FAILED,
  ];
  if (nonDeletableStatuses.includes(transaction.status)) {
    throw new BadRequestException('Cannot delete a transaction in its current state.');
  }
  // ... rest of function
}
```

Alternatively, mirror the pattern already used in `cancelTransaction` and `cancelTransactionGroup`, which explicitly enumerate cancelable statuses and reject anything outside that set. [7](#0-6) 

### Proof of Concept

1. Authenticate as a normal user (the transaction creator).
2. `POST /transaction-groups` — create a group with a transaction whose `validStart` is ~5 minutes in the future.
3. Have required signers sign the transaction; wait for status to become `WAITING_FOR_EXECUTION`.
4. Observe the chain service scheduler log: `smart_collate_timeout_<id>` is registered.
5. Immediately call `DELETE /transaction-groups/:id`.
6. Observe: the API returns `200 OK`; the DB row is gone.
7. Wait for `validStart`; observe the chain service submitting the transaction to Hedera (visible in Hedera mirror node explorer).
8. Query the DB for the transaction — it does not exist.
9. Result: on-chain state changed, DB has no record.

```typescript
  collateAndExecute(transaction: Transaction) {
    const name = `smart_collate_timeout_${transaction.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart = transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        const requiredKeys = await this.transactionSignatureService.computeSignatureKey(transaction);

        const sdkTransaction = await smartCollate(transaction, requiredKeys);

        // If the transaction is still too large,
        // set it to failed with the TRANSACTION_OVERSIZE status code
        // update the transaction, emit the event, and delete the timeout
        if (sdkTransaction === null) {
          const result = await this.transactionRepo
            .createQueryBuilder()
            .update(Transaction)
            .set({
              status: TransactionStatus.FAILED,
              executedAt: new Date(),
              statusCode: Status.TransactionOversize._code,
            })
            .where('id = :id AND status = :currentStatus', {
              id: transaction.id,
              currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
            })
            .returning('id')
            .execute();

          if (result.raw.length > 0) {
            emitTransactionStatusUpdate(
              this.notificationsPublisher,
              result.raw.map(row => ({ entityId: row.id })),
            );
          }
          return;
        }

        // TODO then make sure that front end doesn't allow chunks larger than 2k'
        //NOTE: the transactionBytes are set here but are not to be saved. Otherwise,
        // any signatures that were removed in order to make the transaction fit
        // would be lost.
        transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());

        this.addExecutionTimeout(transaction);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart - 10 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }
```
