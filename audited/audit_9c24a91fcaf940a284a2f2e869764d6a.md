### Title
Unsanitized `v-html` Rendering of Hedera Transaction Memo Enables XSS in Transaction Group View

### Summary
`CreateTransactionGroup.vue` renders transaction data — including the raw `transactionMemo` field decoded from Hedera transaction bytes — directly into the DOM via Vue's `v-html` directive without any sanitization. An attacker can craft a Hedera transaction with an XSS payload in its memo field, share the transaction bytes with a victim, and trigger script execution when the victim views the transaction group in the Electron renderer process.

### Finding Description

In `front-end/src/renderer/pages/CreateTransactionGroup/CreateTransactionGroup.vue`, lines 421–429, a single `v-html` binding renders one of several unsanitized values depending on the transaction type:

```vue
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
``` [1](#0-0) 

Three of the four branches are attacker-influenced:

1. **`Transaction.fromBytes(groupItem.transactionBytes).transactionMemo`** — The Hedera SDK's `transactionMemo` field is a free-form string embedded in the transaction bytes. Any party who creates a transaction (including a malicious peer in an organization workflow) controls this field entirely. The raw string is passed to `v-html` with no sanitization.

2. **`groupItem.description`** — This field is populated from CSV import (`ImportCSVController.vue`) or from the organization back-end. A malicious CSV row or a crafted organization transaction can inject HTML here.

3. **`makeTransfer(index)`** — Calls `formatHbarTransfers`, which returns an HTML string built from transfer data. If that function interpolates account IDs or amounts into HTML without escaping, it is also a vector.

The `AppStepper.vue` component has a secondary instance of the same pattern: `getBubbleContent` constructs `<i class="bi bi-${item.bubbleIcon}"></i>` and returns it for `v-html` rendering. [2](#0-1) [3](#0-2) 

### Impact Explanation

This is an Electron desktop application. The renderer process runs in a Chromium context. Even with `contextIsolation` enabled, XSS in the renderer allows an attacker to:

- Steal locally stored secrets (recovery phrases, private keys, session tokens) by calling `window.electronAPI` / IPC preload-exposed methods.
- Exfiltrate the SQLite local database contents via exposed IPC handlers.
- Redirect the user to a malicious page or silently submit transactions.

If `nodeIntegration` is enabled in `mainWindow.ts` (which could not be confirmed in this search), the impact escalates to full Remote Code Execution on the host OS.

### Likelihood Explanation

The attack requires no privileged access. In the organization workflow, any authenticated organization member can submit a transaction with a crafted memo. The victim (another org member) opens the transaction group view, which immediately triggers the `v-html` render. In the personal (local) workflow, the attacker needs the victim to import a crafted `.bytes` transaction file — a realistic social-engineering step. No cryptographic break is required; the Hedera SDK faithfully decodes whatever memo bytes are present.

### Recommendation

Replace `v-html` with safe text interpolation (`{{ }}`) wherever the content is not intentionally authored HTML. For the one case where HTML output is genuinely needed (e.g., `makeTransfer` returning formatted HTML), sanitize the string with DOMPurify before binding:

```ts
import DOMPurify from 'dompurify';
// ...
v-html="DOMPurify.sanitize(makeTransfer(index))"
```

Apply the same fix to `AppStepper.vue`'s `getBubbleContent` if `bubbleIcon`/`bubbleLabel` can ever originate from user-supplied data.

### Proof of Concept

1. Craft a Hedera `TransferTransaction` (or any transaction type) with the memo set to:
   ```
   <img src=x onerror="document.title='XSS'">
   ```
2. Serialize it to bytes and share the file (or submit it through the organization API as a group item).
3. The victim opens **Create Transaction Group** in the Electron app and adds/loads the crafted transaction.
4. `CreateTransactionGroup.vue` evaluates the `v-html` binding at line 421–429; because `groupItem.description` is empty and the type is not `Transfer Transaction`, it falls through to `Transaction.fromBytes(...).transactionMemo`.
5. The raw memo string `<img src=x onerror="document.title='XSS'">` is injected into the DOM via `v-html` with no sanitization, and the `onerror` handler executes in the Electron renderer process. [1](#0-0)
