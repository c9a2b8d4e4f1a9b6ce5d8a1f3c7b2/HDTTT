### Title
Any Authenticated User Can Post and Read Comments on Transactions They Have No Access To

### Summary
The `CommentsController` endpoints (`POST /transactions/:transactionId/comments`, `GET /transactions/:transactionId/comments`, `GET /transactions/:transactionId/comments/:id`) perform no transaction-level access verification. Every other transaction sub-resource (approvers, the transaction itself) enforces that the requesting user is a creator, signer, approver, or observer before granting access. The comments endpoints skip this check entirely — a fact the codebase itself acknowledges with a `//TODO` comment.

### Finding Description
**Vulnerability class:** Authorization bypass (missing access control on a specific operation while analogous operations enforce it).

**Root cause:**

`CommentsController` is protected only by the controller-level guards `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. These guards confirm the user is authenticated and not blacklisted, but they say nothing about whether the user has any relationship to the target transaction. [1](#0-0) 

The `createComment` handler passes the request straight to `CommentsService.createComment`, which creates and persists the comment with no existence check on the transaction and no membership check on the user. [2](#0-1) 

The `getComments` and `getCommentById` handlers similarly perform no access check — they return comment data for any `transactionId` supplied by the caller. [3](#0-2) 

**Contrast with analogous operations that do enforce access:**

- `GET /transactions/:id` calls `getTransactionWithVerifiedAccess`, which confirms the user is a creator, signer, approver, or observer before returning data. [4](#0-3) 

- `GET /transactions/:transactionId/approvers` calls `getVerifiedApproversByTransactionId`, which enforces the same membership check. [5](#0-4) 

- `DELETE /transactions/:transactionId/approvers/:id` calls `getCreatorsTransaction` to confirm the user is the transaction creator. [6](#0-5) 

The codebase itself flags the gap with an explicit developer note: [7](#0-6) 

**Exploit path:**

1. Attacker registers and is verified as a normal user (no admin or special role needed).
2. Attacker enumerates transaction IDs (sequential integers) by probing `GET /transactions/:id` — they will be denied for transactions they have no access to.
3. Attacker calls `POST /transactions/<any_id>/comments` with an arbitrary body. The request succeeds and the comment is persisted against a transaction the attacker has no legitimate access to.
4. Attacker calls `GET /transactions/<any_id>/comments` to read all comments on any transaction, including those containing sensitive operational notes left by legitimate participants.

### Impact Explanation
- **Unauthorized state mutation:** Any authenticated user can inject comments into any transaction in the system, polluting the audit trail and potentially misleading legitimate participants.
- **Information disclosure:** Comments may contain sensitive operational context (key decisions, legal notes, coordination details). Any authenticated user can read them for any transaction, breaking the confidentiality boundary that the rest of the transaction access model enforces.
- **Audit log integrity:** The comment author field is set to the attacker's user record, so the audit trail is corrupted with entries from users who have no legitimate relationship to the transaction.

### Likelihood Explanation
Exploitation requires only a valid, verified user account — the lowest privilege level in the system. Transaction IDs are sequential integers, making enumeration trivial. No special tooling, timing, or race condition is required. The attack is a single authenticated HTTP request.

### Recommendation
Add a transaction membership check inside `CommentsService.createComment` (and the two `get` handlers) that mirrors the logic already used by `getTransactionWithVerifiedAccess` / `verifyAccess`. Concretely:

1. Resolve the transaction by ID and throw `404` if it does not exist.
2. Call the existing `verifyAccess(transaction, user)` method (or an equivalent) and throw `403` if it returns `false`.
3. Apply the same check to `getTransactionComments` and `getTransactionCommentById` so that comment reads are also gated on transaction membership.

### Proof of Concept
**Preconditions:** Two accounts — `victim` (creator of transaction ID 42) and `attacker` (no relationship to transaction 42). Both are verified users.

```
# Step 1 – confirm attacker is denied the transaction itself
GET /transactions/42
Authorization: Bearer <attacker_token>
→ 403 Forbidden  (verifyAccess enforced)

# Step 2 – attacker posts a comment on the same transaction
POST /transactions/42/comments
Authorization: Bearer <attacker_token>
Content-Type: application/json
{"message": "attacker comment"}
→ 201 Created   ← no access check performed

# Step 3 – attacker reads all comments (including victim's private notes)
GET /transactions/42/comments
Authorization: Bearer <attacker_token>
→ 200 OK  [{"id":1,"message":"victim private note",...}, {"id":2,"message":"attacker comment",...}]
```

The attacker successfully reads confidential comments and injects arbitrary content into the audit trail of a transaction they have no legitimate access to.
