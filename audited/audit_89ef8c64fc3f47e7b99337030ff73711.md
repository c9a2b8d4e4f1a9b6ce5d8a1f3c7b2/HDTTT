### Title
`changePassword` Endpoint Does Not Invalidate the Current JWT Session Token After Password Change

---

### Summary

The `changePassword` endpoint in `auth.controller.ts` successfully updates the user's password but **never blacklists the currently active JWT auth token**. This is a direct state-transition analog to the external report: a security-sensitive action is performed, but the corresponding protective state update (token invalidation) is skipped. An attacker who has obtained a valid JWT can continue using it indefinitely after the victim changes their password.

---

### Finding Description

In `back-end/apps/api/src/auth/auth.controller.ts`, the `changePassword` handler calls `authService.changePassword` but makes no call to `blacklistService.blacklistToken`:

```typescript
// back-end/apps/api/src/auth/auth.controller.ts  (lines 115–119)
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto): Promise<void> {
  return this.authService.changePassword(user, dto);
}
``` [1](#0-0) 

Compare this to every other security-sensitive auth action in the same controller, all of which **do** blacklist the relevant token:

- **`logout`** — blacklists the auth token:
  ```typescript
  await this.blacklistService.blacklistToken(extractJwtAuth(req));
  ``` [2](#0-1) 

- **`setPassword`** — blacklists the OTP token after setting a new password:
  ```typescript
  await this.authService.setPassword(user, dto.password);
  await this.blacklistService.blacklistToken(extractJwtOtp(req));
  ``` [3](#0-2) 

- **`verifyOtp`** — blacklists the OTP token after OTP verification:
  ```typescript
  await this.blacklistService.blacklistToken(extractJwtOtp(req));
  ``` [4](#0-3) 

The `BlacklistService` stores invalidated tokens in Redis with a TTL equal to `JWT_EXPIRATION` days. The guard `JwtBlackListAuthGuard` checks this store on every protected request. Because `changePassword` never calls `blacklistToken`, the old token is never added to Redis and remains accepted by the guard for its full lifetime. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An attacker who has stolen or intercepted a user's JWT (e.g., via XSS, network interception, or a compromised device) retains full authenticated access to the API even after the victim changes their password. The password change — the primary remediation action available to a compromised user — provides no actual session revocation. The attacker can continue to create, sign, approve, and cancel Hedera transactions on behalf of the victim for the entire remaining lifetime of the stolen token.

---

### Likelihood Explanation

Token theft is a realistic and common attack vector (XSS, logging, compromised endpoints). The `changePassword` flow is the natural response a user takes when they suspect compromise. The missing blacklist call is a straightforward omission consistent with the pattern seen in the external report. No privileged attacker role is required — any user whose token is stolen is affected.

---

### Recommendation

Add the token blacklist call in `changePassword`, mirroring the pattern used in `logout` and `setPassword`:

```diff
// back-end/apps/api/src/auth/auth.controller.ts
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
-async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto): Promise<void> {
-  return this.authService.changePassword(user, dto);
+async changePassword(@GetUser() user: User, @Body() dto: ChangePasswordDto, @Req() req: Request): Promise<void> {
+  await this.authService.changePassword(user, dto);
+  await this.blacklistService.blacklistToken(extractJwtAuth(req));
}
```

---

### Proof of Concept

1. User A logs in → receives `JWT_A`.
2. Attacker steals `JWT_A` (any realistic vector).
3. User A notices the compromise and calls `PATCH /auth/change-password` with `JWT_A` and a new password. The call succeeds.
4. Attacker immediately calls any protected endpoint (e.g., `GET /transactions`) with `JWT_A`.
5. `JwtBlackListAuthGuard` checks Redis — `JWT_A` is **not** present because `blacklistToken` was never called — and grants access.
6. Attacker retains full authenticated access until `JWT_A` naturally expires. [1](#0-0) [6](#0-5)

  async blacklistToken(jwt: string) {
    const expirationDays = this.configService.get<number>('JWT_EXPIRATION');
    const expirationSeconds = Number(expirationDays) * 24 * 60 * 60;

    await this.client.set(jwt, this.BLACKLISTED, 'EX', expirationSeconds);
  }

  async isTokenBlacklisted(jwt: string) {
    const data = await this.client.get(jwt);
    return data === this.BLACKLISTED;
  }
