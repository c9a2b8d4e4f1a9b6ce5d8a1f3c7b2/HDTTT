### Title
`createTransactionObservers`: Duplicate `userId` entries in a single request bypass the deduplication guard, inserting duplicate observer rows

### Summary
`ObserversService.createTransactionObservers` checks for existing observers against a snapshot of `transaction.observers` loaded at the start of the function. The guard is never applied against the `observers` array being built within the same loop iteration. A transaction creator who sends a single `POST /transactions/{id}/observers` request with a repeated `userId` in `dto.userIds` will insert multiple `TransactionObserver` rows for the same user, corrupting the observer list and triggering duplicate notifications.

### Finding Description

In `back-end/apps/api/src/transactions/observers/observers.service.ts`, `createTransactionObservers` loads the transaction (with its current observers) once at the top of the function: [1](#0-0) 

It then iterates over `dto.userIds` and checks each `userId` against that **static snapshot**: [2](#0-1) 

The guard `!transaction.observers.some(o => o.userId === userId)` only looks at rows already committed to the database before this request arrived. It never checks the `observers` array that is being accumulated within the current loop. Therefore, if `dto.userIds` is `[5, 5]` (or any repeated value), both iterations pass the check, two `TransactionObserver` objects are pushed into `observers`, and both are persisted: [3](#0-2) 

The same race-condition variant exists: two concurrent requests from the creator, each carrying the same `userId`, both read the same pre-insert snapshot, both pass the guard, and both call `repo.save`. Without a database-level unique constraint on `(transactionId, userId)` the rows are silently duplicated.

### Impact Explanation

1. **Duplicate DB rows** — `transaction_observer` contains multiple rows for the same `(transactionId, userId)` pair, corrupting the authoritative observer list.
2. **Duplicate notifications** — `receiver.service.ts` maps `transaction.observers` directly to `observerUserIds` without deduplication before building notification recipient lists for several notification types (e.g. `TRANSACTION_INDICATOR_REJECTED`, `TRANSACTION_INDICATOR_EXECUTED`), causing the same user to receive the same notification multiple times: [4](#0-3) [5](#0-4) 

3. **UI corruption** — `getTransactionObserversByTransactionId` returns `transaction.observers` verbatim; the same user appears twice in the observer list.
4. **Removal leaves ghost rows** — `removeTransactionObserver` removes a single row by `id`; the second duplicate row remains, so the user cannot be fully removed from the observer list without manual DB intervention.

### Likelihood Explanation

The attacker precondition is simply being the creator of any transaction — a normal, unprivileged user role. The exploit requires only a single crafted HTTP request:

```
POST /transactions/{id}/observers
{ "userIds": [5, 5] }
```

No special tooling, no race timing, no privileged access. The creator role is reachable by any registered organization user.

### Recommendation

Add an intra-request deduplication step before the loop, and also check the accumulating `observers` array (not just the DB snapshot) inside the loop:

```typescript
const uniqueUserIds = [...new Set(dto.userIds)];
for (const userId of uniqueUserIds) {
  const alreadyExists =
    transaction.observers.some(o => o.userId === userId) ||
    observers.some(o => o.userId === userId);   // ← check in-flight list too
  if (!alreadyExists) {
    observers.push(this.repo.create({ userId, transactionId, role: Role.FULL }));
  }
}
```

Additionally, add a database-level unique constraint on `(transactionId, userId)` in the `transaction_observer` table as a defence-in-depth measure.

### Proof of Concept

**Preconditions:** Authenticated as the creator of transaction with `id = 1`. User with `id = 7` is not yet an observer.

**Request:**
```http
POST /transactions/1/observers
Authorization: Bearer <creator_token>
Content-Type: application/json

{ "userIds": [7, 7] }
```

**Expected (correct) behaviour:** One `TransactionObserver` row inserted for userId 7.

**Actual behaviour:** Two `TransactionObserver` rows inserted for userId 7 (both pass the check at line 50 because neither is in `transaction.observers` at read time). Subsequent notification events dispatch two copies of every observer-targeted notification to user 7. Calling `DELETE /transactions/observers/{id}` on one row leaves the second row, so user 7 remains an observer despite the deletion. [6](#0-5)
