### Title
Unbounded Recursive Approver Tree Enables Single-Request DoS via Quadratic DB Query Amplification

### Summary
The `POST /transactions/:transactionId/approvers` endpoint accepts a self-referential, arbitrarily deep nested approver tree with no depth or node-count limit. The server-side handler processes this tree recursively, and each node insertion triggers a `getRootNodeFromNode` traversal that issues O(depth) database queries. A single crafted request from a verified user can generate O(depth²) database queries, exhausting the connection pool and degrading the API service for all users.

### Finding Description

**Root cause — no size or depth constraint on the DTO:**

`CreateTransactionApproverDto` is self-referential. The `approvers` field carries only `@ArrayMinSize(1)` and no `@ArrayMaxSize`. The outer wrapper `CreateTransactionApproversArrayDto.approversArray` also carries no `@ArrayMaxSize`. [1](#0-0) 

**Root cause — unbounded recursive processing in the service:**

`createTransactionApprovers` defines an inner async function `createApprover` that calls itself for every child node. There is no depth counter, no total-node counter, and no guard of any kind. [2](#0-1) 

**DB query amplification per node:**

For every node inserted, the code calls `getRootNodeFromNode(dtoApprover.listId, ...)` which traverses the parent chain up to the root. For a linear chain of depth D, this traversal issues O(D) queries per node, yielding O(D²) total queries for the full chain. [3](#0-2) 

**Exploit path:**

1. Attacker registers and becomes a verified user (required by `VerifiedUserGuard`).
2. Attacker creates a transaction (becomes its creator).
3. Attacker sends one `POST /transactions/:id/approvers` with a deeply nested threshold tree:

```json
{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{
      "threshold": 1,
      "approvers": [{
        "threshold": 1,
        "approvers": [
          ... (depth N)
          { "userId": <any_valid_user_id> }
        ]
      }]
    }]
  }]
}
```

Each threshold node is valid (passes all structural checks: `THRESHOLD_REQUIRED`, `CHILDREN_REQUIRED`, `THRESHOLD_LESS_OR_EQUAL_APPROVERS`). No userId duplication check blocks threshold-only nodes. The call stack grows to depth N and the DB receives O(N²) queries within a single HTTP request.

**Rate limiting does not mitigate this:**

The `UserThrottlerModule` limits to 10 requests/second and 100 requests/minute. A single request with N=500 nodes already issues ~125,000 DB queries before the throttle window resets. [4](#0-3) 

The controller applies no additional guard beyond authentication and user verification. [5](#0-4) 

### Impact Explanation

A single authenticated verified user can issue one HTTP request that:
- Exhausts the database connection pool for the duration of the request.
- Blocks all other API operations that require a DB connection (transaction creation, signing, execution polling).
- Potentially causes a JavaScript call-stack overflow for very large depths, crashing the API process.

This constitutes service unavailability for all users of the organization server, not just the attacker.

### Likelihood Explanation

The attacker preconditions are low: register an account, pass email verification, create one transaction. All of these are normal product flows available to any user. No privileged access, no leaked credentials, and no external infrastructure are required. The attack is a single HTTP POST with a crafted JSON body.

### Recommendation

1. **Add `@ArrayMaxSize` to both DTO arrays.** Apply a hard cap (e.g., 50) to `approversArray` in `CreateTransactionApproversArrayDto` and to `approvers` in `CreateTransactionApproverDto`.
2. **Add a maximum recursion depth guard** inside `createApprover` — pass a depth counter and throw if it exceeds a threshold (e.g., 10).
3. **Add a total-node counter** across the entire transaction to cap the cumulative number of approver nodes (e.g., 200).
4. **Flatten `getRootNodeFromNode` traversal** or cache the root per transaction to avoid O(depth) queries per insertion.

### Proof of Concept

```http
POST /transactions/1/approvers
Authorization: Bearer <valid_verified_user_token>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                ... (repeat 500 levels deep)
                { "userId": 2 }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**Expected outcome:** The server enters a recursive loop issuing ~125,000 sequential DB queries within the single request. The database connection pool is saturated. Concurrent requests from other users receive connection-timeout errors. If depth exceeds the Node.js call-stack limit (~10,000 frames), the API process crashes with `RangeError: Maximum call stack size exceeded`.
