### Title
Transaction Creator Can Mutate Approver and Observer Lists on Terminal-State Transactions

### Summary
`ApproversService.createTransactionApprovers()`, `ApproversService.updateTransactionApprover()`, `ObserversService.createTransactionObservers()`, `ObserversService.updateTransactionObserver()`, and `ObserversService.removeTransactionObserver()` all lack transaction-status guards. A transaction creator can add, update, or remove approvers and observers on transactions that are already in terminal states (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`), corrupting the integrity of the approval and audit record for finalized transactions.

### Finding Description

The root cause is that the shared helper `getCreatorsTransaction()` — used as the sole gate in both `createTransactionApprovers` and `updateTransactionApprover` — only verifies creator ownership; it never inspects `transaction.status`. [1](#0-0) 

`createTransactionApprovers` calls this helper and then proceeds to insert new approver records with no status check: [2](#0-1) 

`updateTransactionApprover` follows the same pattern — it calls `getCreatorsTransaction` and then mutates the approver tree regardless of whether the transaction is already finalized: [3](#0-2) 

`removeTransactionApprover` is even more permissive: it accepts only an approver `id`, performs no ownership check, and performs no status check before soft-deleting the approver node: [4](#0-3) 

`createTransactionObservers` in `ObserversService` similarly checks only creator ownership and never the transaction status before inserting observer records: [5](#0-4) 

`updateTransactionObserver` and `removeTransactionObserver` delegate to `getUpdateableObserver`, which again checks only creator ownership: [6](#0-5) 

By contrast, the signing path (`validateTransactionStatus`) and the approval-choice path (`approveTransaction`) both correctly gate on `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` before accepting mutations: [7](#0-6) [8](#0-7) 

The approver/observer mutation paths are inconsistent with these guards.

### Impact Explanation

A transaction creator can retroactively alter the approver tree or observer list of an `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, or `ARCHIVED` transaction. This:

1. **Corrupts the audit trail** — the persisted approver/observer records no longer reflect who was actually required to approve or observe the transaction at the time it was finalized.
2. **Breaks downstream integrity checks** — any component that re-reads approver state to determine whether a transaction was properly authorized will see a falsified record.
3. **`removeTransactionApprover` has no ownership check** — any authenticated user (not just the creator) can delete any approver from any transaction in any state, including active ones, potentially unblocking a transaction that should remain blocked.

### Likelihood Explanation

The attacker is the transaction creator — a normal, unprivileged user. No special credentials are required. The API endpoints for creating/updating/removing approvers and observers are standard REST calls available to any authenticated user who created a transaction. The scenario is reachable in normal product usage (e.g., a transaction expires or is canceled, and the creator subsequently calls the approver-management endpoints).

### Recommendation

Add a terminal-state guard in `getCreatorsTransaction` (or inline in each mutating method) that mirrors the pattern already used in `approveTransaction` and `validateTransactionStatus`:

```typescript
const TERMINAL_STATUSES = [
  TransactionStatus.EXECUTED,
  TransactionStatus.FAILED,
  TransactionStatus.EXPIRED,
  TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED,
];

if (TERMINAL_STATUSES.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify a finalized transaction');
}
```

Apply this check in:
- `ApproversService.createTransactionApprovers`
- `ApproversService.updateTransactionApprover`
- `ApproversService.removeTransactionApprover` (also add an ownership check here)
- `ObserversService.createTransactionObservers`
- `ObserversService.getUpdateableObserver` (covers update and remove)

### Proof of Concept

1. Authenticated user Alice creates a transaction (becomes creator).
2. The transaction progresses and reaches `EXECUTED` status.
3. Alice calls `POST /transactions/:id/approvers` with a new approver payload.
4. `createTransactionApprovers` calls `getCreatorsTransaction`, which passes (Alice is the creator).
5. No status check occurs; new `TransactionApprover` rows are inserted for the already-executed transaction.
6. The approval record now shows approvers who were never part of the original workflow, corrupting the audit trail.
7. Separately, any authenticated user (not just Alice) calls `DELETE /transactions/approvers/:approverId` for an approver on an active transaction — `removeTransactionApprover` has no user parameter and performs no ownership or status check, so the approver is silently deleted.   async createTransactionObservers(
    user: User,
    transactionId: number,
    dto: CreateTransactionObserversDto,
  ): Promise<TransactionObserver[]> {
    const transaction = await this.entityManager.findOne(Transaction, {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user', 'observers'],
    });

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to delete it');

    const observers: TransactionObserver[] = [];

    for (const userId of dto.userIds) {
      if (!transaction.observers.some(o => o.userId === userId)) {
        const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
        observers.push(observer);
      }
    }

    if (observers.length === 0) {
      return [];
    }

    try {
      const result = await this.repo.save(observers);

      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);

      return result;
    } catch (error) {
      throw new BadRequestException(error.message);
    }
  }
