### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side DoS

### Summary
The `getTransactionsToSign` method in the API service fetches every non-terminal transaction in the entire database without any SQL-level limit, then performs an async `userKeysToSign` call for each row in a sequential loop. Any authenticated user can trigger this endpoint. As the number of transactions in the system grows, each request consumes proportionally more CPU, memory, and database connections, eventually degrading or crashing the API service for all users.

### Finding Description

**Root cause — missing `take` in the repository query:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` (lines 295–309) issues an unbounded `find()` with no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `skip` / `take` here
});
```

Compare this to every other paginated method in the same file, e.g. `getTransactions` (line 175–181) and `getHistoryTransactions` (line 227–235), which both pass `skip: offset, take: limit` to the query.

The `whereForUser` filter only excludes terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** filter by the requesting user, so the result set is every active transaction in the entire system.

**Unbounded async loop:**

After loading the full result set, the code iterates over every row and awaits an async call per iteration:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
```

`userKeysToSign` involves cryptographic key resolution and potentially additional database queries per transaction. Pagination is applied only after the loop completes:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  page,
  size,
};
```

**The endpoint is still publicly reachable:**

The controller exposes this at `GET /transactions/sign` with only standard JWT authentication guards — no admin role required. The comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end stopped calling it, but the route remains active and callable by any authenticated API client. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

- **Service-wide DoS:** Because the query is not scoped to the requesting user, a single request loads every active transaction in the system. With N active transactions, each request performs N async `userKeysToSign` calls sequentially. At sufficient scale this exhausts the Node.js event loop, database connection pool, and process memory, making the API unresponsive for all users.
- **Amplification:** Any authenticated user can repeatedly call the endpoint, compounding the load. Multiple concurrent callers multiply the effect.
- **No self-recovery:** Transactions only leave the unbounded set when they reach a terminal status. A growing organization accumulates more active transactions over time, making the problem progressively worse.

### Likelihood Explanation

- **Attacker precondition:** Only a valid JWT token is required — any registered user qualifies.
- **Trigger:** A single HTTP `GET /transactions/sign` request is sufficient to trigger the full scan. No special timing or coordination is needed.
- **Growth vector:** Every transaction created by any user in the organization (not just the attacker) enlarges the scanned set. In an active organization this grows organically without any attacker action.
- **Realistic scenario:** A malicious insider or a compromised account can repeatedly call the endpoint to sustain the DoS. Even without malicious intent, a large organization will eventually hit performance degradation naturally.

### Recommendation

1. **Apply SQL-level pagination to the initial query** — pass `skip: offset, take: limit` to `this.repo.find(...)` just as `getTransactions` and `getHistoryTransactions` do. This caps the number of rows loaded and the number of `userKeysToSign` calls per request.
2. **Scope the query to the requesting user** — add a join/filter so only transactions where the user is a potential signer are fetched, reducing the result set further.
3. **Remove or formally deprecate the endpoint** — since the front-end no longer uses it (per the controller comment), consider removing the route entirely to eliminate the attack surface.

### Proof of Concept

**Setup:** An organization with a large number of active (non-terminal) transactions (e.g., 10,000 rows in `WAITING_FOR_SIGNATURES` status, created by any users).

**Steps:**
1. Obtain a valid JWT token for any registered, verified user.
2. Send repeated requests:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <valid_jwt>
   ```
3. **Observed behavior:** Each request causes the server to:
   - Execute `SELECT * FROM transaction WHERE status NOT IN (...)` with no LIMIT — loading all 10,000 rows.
   - Sequentially await `userKeysToSign(transaction, user)` 10,000 times.
4. **Expected outcome:** API response latency grows linearly with the number of active transactions. Under concurrent requests the Node.js process becomes CPU-bound and the PostgreSQL connection pool is exhausted, causing HTTP 503/timeout responses for all users of the service. [4](#0-3) [5](#0-4)
