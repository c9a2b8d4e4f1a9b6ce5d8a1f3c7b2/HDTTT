### Title
Unbounded Transaction Queue Iteration in `getTransactionsToSign` Enables Authenticated DoS

### Summary
The `getTransactionsToSign` function in `transactions.service.ts` fetches **all** active transactions from the database with no row limit, then iterates over every result with an async `userKeysToSign` call per row. An authenticated attacker who repeatedly calls `POST /transactions` — which has no rate-limiting guard — can grow this set arbitrarily, causing `GET /transactions/sign` to consume unbounded server resources and eventually time out or degrade service for all users.

### Finding Description

**Root cause — unbounded DB fetch with O(n) async loop:**

In `getTransactionsToSign`, the repository query at line 295 has no `take` constraint:

```ts
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

Every row returned is then processed in a sequential async loop:

```ts
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

Pagination is applied **after** the full in-memory scan (line 313), so the DB and CPU cost is always O(total active transactions), not O(page size).

**Entry point — unthrottled transaction creation:**

`POST /transactions` carries only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`, `HasKeyGuard`). No `@Throttle` decorator is present on the controller or the endpoint: [3](#0-2) 

The throttler modules that exist (`UserThrottlerModule`, `IpThrottlerModule`) are wired only to the auth and notification controllers — confirmed by the absence of any throttle reference in `transactions.controller.ts`. [4](#0-3) 

### Impact Explanation

Each call to `GET /transactions/sign` triggers one unbounded SQL query loading all active transactions plus one async `userKeysToSign` invocation per row. With thousands of attacker-created transactions in `WAITING_FOR_SIGNATURES` status, the endpoint response time grows linearly. At sufficient scale this causes:

- Request timeouts for all legitimate users calling `GET /transactions/sign`
- Sustained high CPU/memory pressure on the API process
- Potential cascading slowdown of the shared NestJS event loop, degrading other endpoints

### Likelihood Explanation

The attacker must be an authenticated organization user with at least one registered key — a realistic "malicious insider" or compromised account scenario. Each `POST /transactions` call requires a valid Hedera SDK transaction bytes payload with a unique transaction ID, but these are trivially generated client-side. Because there is no per-user transaction count cap and no rate limit on the creation endpoint, an attacker can accumulate tens of thousands of pending transactions over hours without triggering any existing control.

### Recommendation

1. **Apply a `take` limit to the DB query in `getTransactionsToSign`** and perform the `userKeysToSign` filter at the database level rather than in application memory, so cost is bounded by the page size.
2. **Add a `@UseGuards(UserThrottlerGuard)` (or equivalent `@Throttle`) decorator** to `POST /transactions` in `TransactionsController` to cap per-user creation rate.
3. **Enforce a per-user cap** on the number of active (non-terminal) transactions to prevent indefinite accumulation.

### Proof of Concept

**Preconditions:** Attacker holds a valid JWT and one registered `UserKey` in the organization.

**Steps:**

1. Generate N distinct Hedera `AccountCreateTransaction` objects, each with a unique `TransactionId` (different valid-start timestamps).
2. POST each to `POST /transactions` with the attacker's `creatorKeyId`. No rate limit blocks this loop.
3. All N transactions enter `WAITING_FOR_SIGNATURES` status and are included in the `whereForUser` filter (status not in the excluded terminal set).
4. Any user (including victims) calls `GET /transactions/sign?page=1&size=10`.
5. `getTransactionsToSign` issues `repo.find()` with no `take`, loading all N rows, then awaits `userKeysToSign` N times sequentially.
6. With N = 10,000, the endpoint takes orders of magnitude longer than normal, eventually timing out or exhausting the Node.js event loop, denying service to all legitimate signers. [5](#0-4) [6](#0-5)
