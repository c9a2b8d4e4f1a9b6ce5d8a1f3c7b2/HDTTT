### Title
Authorization Bypass: `verifyAccess` Unconditionally Grants Access to All Terminal-Status Transactions, Enabling Cross-Tenant Data Exposure

### Summary
The `verifyAccess` function in `transactions.service.ts` short-circuits all ownership and relationship checks and returns `true` for any transaction whose status is `EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, or `ARCHIVED`. Because `GET /transactions/:id` delegates its authorization entirely to this function, any authenticated user can retrieve the full details — including raw transaction bytes, signer keys, observer identities, and creator key metadata — of any completed transaction belonging to any other user or organization. The `GET /transactions/history` endpoint compounds this by returning all terminal-status transactions system-wide with no per-user filtering whatsoever.

### Finding Description

**Root cause — `verifyAccess` bypasses all ownership checks for terminal states** [1](#0-0) 

```
async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;          // ← no user relationship check at all
    ...
```

The early `return true` fires before any of the ownership predicates (`creatorKey.userId`, `observers`, `signers`, `approvers`) are evaluated. Any authenticated user who supplies a valid transaction ID receives a `true` result and is served the full transaction object.

**Call site — `GET /transactions/:id`** [2](#0-1) 

`getTransactionWithVerifiedAccess` calls `verifyAccess` and throws only when it returns `false`. For every terminal-status transaction it always returns `true`, so the guard never fires.

**Compounding issue — `GET /transactions/history` has no per-user filter** [3](#0-2) 

`getHistoryTransactions` builds a `FindManyOptions` with only a status filter and no `userId` predicate. Every authenticated user receives a paginated dump of all terminal-status transactions across all tenants.

**Controller — both endpoints are reachable by any verified JWT holder** [4](#0-3) 

The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard session guards — but no ownership guard. Registration is open to any user invited by an admin, and the attacker only needs a valid session token.

### Impact Explanation

A malicious authenticated user can:

1. **Enumerate all completed transactions** via `GET /transactions/history` (paginated, no ownership check) and obtain transaction IDs, types, mirror networks, and group metadata for every transaction ever processed by the platform.
2. **Retrieve full transaction detail** for any enumerated ID via `GET /transactions/:id`, obtaining:
   - Raw `transactionBytes` (the full signed Hedera transaction body)
   - `transactionHash`
   - `publicKeys` embedded in the transaction
   - Signer user-key associations (`signers[].userKey`)
   - Observer user identities
   - Creator key and linked user identity

This constitutes **cross-tenant data exfiltration**: organizational transaction history, key-to-user mappings, and signing patterns of unrelated users are fully exposed. Severity is **High** (confidentiality breach across all tenants; no asset theft, but complete organizational intelligence leak).

### Likelihood Explanation

- **Attacker precondition**: a valid, non-admin account — obtainable by any user the admin has ever invited.
- **Effort**: a single authenticated HTTP request to `GET /transactions/history` returns the first page; subsequent pages enumerate the entire history. No brute-force of IDs is required.
- **Detection difficulty**: the requests are indistinguishable from legitimate history browsing.
- **Likelihood**: **High**.

### Recommendation

1. **Fix `verifyAccess`**: Remove the unconditional `return true` for terminal statuses. Apply the same ownership predicate (creator, signer, observer, approver) regardless of status. Terminal-status transactions are not inherently public.

```typescript
// Remove this block entirely:
if ([EXECUTED, EXPIRED, FAILED, CANCELED, ARCHIVED].includes(transaction.status))
  return true;
```

2. **Fix `getHistoryTransactions`**: Add a per-user `WHERE` clause identical to the one used in `getTransactions` (creator OR signer OR observer OR approver).

3. **Add an integration test** asserting that `GET /transactions/:id` and `GET /transactions/history` return `401`/empty results for a user with no relationship to the queried transactions.

### Proof of Concept

**Step 1** — Attacker registers (or is invited) and obtains a JWT via `POST /auth/login`.

**Step 2** — Attacker calls:
```
GET /transactions/history?page=1&limit=50
Authorization: Bearer <attacker_jwt>
```
Response: paginated list of **all** terminal-status transactions across all users/organizations, including IDs, types, networks, and group metadata.

**Step 3** — For any `id` returned, attacker calls:
```
GET /transactions/42
Authorization: Bearer <attacker_jwt>
```
`verifyAccess` is invoked; because transaction 42 is `EXECUTED`, it returns `true` immediately. The full transaction object — bytes, hash, public keys, signer/observer/creator identities — is returned to the attacker with HTTP 200.

**Expected (correct) behavior**: HTTP 401 `"You don't have permission to view this transaction"`.

**Actual behavior**: HTTP 200 with full transaction detail. [5](#0-4) [3](#0-2) [6](#0-5) [7](#0-6)
