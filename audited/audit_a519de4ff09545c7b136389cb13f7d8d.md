### Title
Any Verified User Can Delete Any Transaction Approver Without Ownership Check

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` accepts only an approver `id` and performs no check that the calling user is the creator of the associated transaction. Every other approver-mutating operation (`createTransactionApprovers`, `updateTransactionApprover`) explicitly verifies creator ownership via `getCreatorsTransaction`, but the delete path omits this check entirely. Any authenticated, verified user who knows (or can enumerate) an approver record ID can silently remove it from any transaction they do not own.

### Finding Description

**Root cause — missing ownership check in `removeTransactionApprover`**

`createTransactionApprovers` and `updateTransactionApprover` both accept a `user: User` argument and call `getCreatorsTransaction` to assert the caller owns the transaction before mutating approver state. [1](#0-0) [2](#0-1) 

`removeTransactionApprover`, by contrast, accepts only `id: number` — no `user` parameter, no ownership assertion: [3](#0-2) 

The controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only verify that the JWT is valid and the user's status is `NONE` (verified). They do not verify transaction ownership. [4](#0-3) 

**Exploit path**

1. Attacker registers and verifies their account (status `NONE`).
2. Attacker observes or enumerates a `TransactionApprover` record ID belonging to a transaction they did not create (IDs are sequential integers).
3. Attacker sends `DELETE /transactions/{transactionId}/approvers/{id}` with their valid JWT.
4. `removeTransactionApprover(id)` is called; it finds the approver, calls `removeNode`, and deletes it — no ownership check is ever performed.

**Analog to external report**

The external report shows that `_update` checks `from` (which is `address(0)` during mint) instead of the actual restricted actor. Here, the delete path checks nothing about the actor at all, while the create/update paths correctly check `user`. The pattern is identical: one code path in the same authorization boundary silently omits the actor-identity check that all sibling paths enforce.

### Impact Explanation

An attacker can remove approvers from any transaction in the system:

- **Approval threshold sabotage**: Removing approvers from a threshold-based approval tree can reduce the required signers below the intended threshold, allowing a transaction to proceed to execution without the required organizational sign-off.
- **Denial of service on transactions**: Removing all approvers from a transaction that requires approval leaves it permanently stuck in `WAITING_FOR_SIGNATURES` or forces the creator to rebuild the approver tree.
- **Integrity violation**: The approval workflow is the core trust mechanism of the platform. Silent removal of approvers by an unauthorized party breaks the multi-signature coordination guarantee the system is designed to provide.

### Likelihood Explanation

- Attacker precondition: a valid, verified account — the lowest privilege level in the system.
- Approver IDs are sequential integers, trivially enumerable via the GET approvers endpoint (which is accessible to any verified user who is a participant in at least one transaction).
- No rate limiting or anomaly detection is described for this endpoint.
- The attack requires a single authenticated HTTP DELETE request.

### Recommendation

Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` (or an equivalent ownership assertion) before executing the delete, exactly as `updateTransactionApprover` does:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root transaction and assert caller is creator
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  await this.getCreatorsTransaction(rootNode.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Update the controller's DELETE handler to pass `@GetUser() user: User` to the service call.

### Proof of Concept

**Preconditions**
- `attackerToken`: JWT for a verified user (status `NONE`) who is NOT the creator of the target transaction.
- `victimApproverIds`: one or more `TransactionApprover` IDs belonging to a transaction created by another user (obtainable by being an observer/signer on any transaction and calling `GET /transactions/{id}/approvers`, or by sequential enumeration).

**Steps**
```
DELETE /transactions/{transactionId}/approvers/{victimApproverId}
Authorization: Bearer <attackerToken>
```

**Expected (correct) behavior**: `401 Unauthorized` — caller is not the transaction creator.

**Actual behavior**: `200 OK` — approver record is deleted.

The approver is removed from the transaction's approval tree without any ownership verification, as `removeTransactionApprover` never receives or checks the calling user's identity. [3](#0-2)
