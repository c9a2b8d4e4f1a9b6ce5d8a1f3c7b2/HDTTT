### Title
Unbounded `groupItems` Array in `POST /transaction-groups` Enables Authenticated DoS

### Summary
The `CreateTransactionGroupDto` imposes no upper bound on the `groupItems` array. An authenticated user can submit a transaction group containing thousands of items, triggering unbounded parallel mirror-node network calls in `createTransactions`, unbounded sequential database round-trips in `removeTransactionGroup`, and unbounded iteration in `executeTransactionGroup`. This can exhaust server memory, connection pools, and CPU, causing sustained service degradation or crash for all users.

### Finding Description

**Root cause — missing `@ArrayMaxSize()` in the DTO:**

`CreateTransactionGroupDto` validates that `groupItems` is a non-empty array and that each element is a valid `CreateTransactionGroupItemDto`, but applies no maximum-length constraint: [1](#0-0) 

**Exploit path 1 — unbounded `Promise.all` in `createTransactions`:**

`createTransactionGroup` extracts every item's transaction DTO and passes the entire unbounded slice to `createTransactions`: [2](#0-1) 

`createTransactions` then fans out a `Promise.all` over every DTO, each of which calls `validateAndPrepareTransaction` — a function that performs mirror-node network I/O per item: [3](#0-2) 

With 10 000 items this spawns 10 000 concurrent outbound HTTP connections, exhausting the Node.js event loop and the mirror-node connection pool.

**Exploit path 2 — O(n) sequential DB round-trips in `removeTransactionGroup`:**

`removeTransactionGroup` iterates over every group item and issues two sequential `await` database calls per item with no batching or bound: [4](#0-3) 

A group with thousands of items holds a long-lived database connection and blocks the event loop for the duration.

**Exploit path 3 — unbounded iteration in `executeTransactionGroup`:**

The chain service's `executeTransactionGroup` iterates over every group item to validate and then execute each transaction, with no cap: [5](#0-4) 

### Impact Explanation
A single malicious authenticated user can POST one request with an arbitrarily large `groupItems` array to `/transaction-groups`. This causes:
- **Memory exhaustion**: thousands of in-flight SDK transaction objects and mirror-node responses held simultaneously.
- **Connection pool starvation**: all available database and HTTP connections consumed, blocking legitimate requests.
- **Event-loop stall**: the sequential `removeTransactionGroup` loop holds the loop for minutes, making the API unresponsive for all other users.

The result is a complete or severe denial of service for the entire organization backend.

### Likelihood Explanation
Any registered organization user — the baseline attacker in this system — can reach `POST /transaction-groups` after a normal login. No admin privilege, no leaked secret, and no special role is required. The endpoint is protected only by JWT authentication (`JwtAuthGuard`, `VerifiedUserGuard`), which any legitimate account satisfies. A single crafted request is sufficient to trigger the condition. [6](#0-5) 

### Recommendation
1. Add `@ArrayMaxSize(N)` (e.g., `N = 100`) to `groupItems` in `CreateTransactionGroupDto` to enforce a hard cap at the validation layer.
2. In `removeTransactionGroup`, replace the per-item sequential loop with a single bulk `DELETE` query.
3. Consider adding a global NestJS `ValidationPipe` option `forbidUnknownValues: true` and request-body size limits at the HTTP layer as defence-in-depth.

### Proof of Concept

```bash
# Authenticated as any normal org user
TOKEN="<valid_jwt>"
SERVER="https://<org-server>"

# Build a payload with 5000 identical group items
python3 -c "
import json, sys
item = {
  'seq': 0,
  'transaction': {
    'name': 'x',
    'description': 'x',
    'transactionBytes': '0a' * 100,   # minimal hex bytes
    'mirrorNetwork': 'testnet',
    'signature': 'aa' * 64,
    'creatorKeyId': 1
  }
}
body = {'description': 'dos', 'atomic': False, 'sequential': False,
        'groupItems': [dict(item, seq=i) for i in range(5000)]}
print(json.dumps(body))
" > payload.json

curl -s -X POST "$SERVER/transaction-groups" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d @payload.json
```

**Expected outcome:** The server spawns 5 000 concurrent mirror-node validation calls. Memory and connection-pool usage spike; subsequent requests to any endpoint time out or return 503 for the duration of the operation.

  @ApiResponse({
    status: 201,
    type: TransactionGroupDto,

  @Post()
  @Serialize(TransactionGroupDto)
  createTransactionGroup(
    @GetUser() user: User,
    @Body() dto: CreateTransactionGroupDto,

    return this.transactionGroupsService.createTransactionGroup(user, dto);
