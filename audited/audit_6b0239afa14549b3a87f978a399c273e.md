### Title
Missing Creator Authorization Check in `removeTransactionApprover` Allows Any Authenticated User to Remove Approvers from Any Transaction

### Summary
`removeTransactionApprover` in `approvers.service.ts` performs no authorization check to verify the calling user is the creator of the transaction. Every other approver-mutating function (`createTransactionApprovers`, `updateTransactionApprover`) calls `getCreatorsTransaction` to enforce creator-only access, but `removeTransactionApprover` accepts only an approver `id` with no `user` parameter and no equivalent guard. Any authenticated user who knows or can enumerate an approver ID can silently delete required approvers from any other user's transaction, bypassing the approval workflow entirely.

### Finding Description

**Root cause — missing authorization in `removeTransactionApprover`:** [1](#0-0) 

The function signature is `removeTransactionApprover(id: number): Promise<void>` — it accepts no `user` argument and calls no authorization helper. It fetches the approver, calls `removeNode`, and emits a status update. No identity check is performed.

**Contrast with `createTransactionApprovers`**, which immediately calls `getCreatorsTransaction`: [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which also calls `getCreatorsTransaction` inside the DB transaction: [3](#0-2) 

**`getCreatorsTransaction` is the sole authorization gate** — it enforces `transaction.creatorKey?.userId === user.id`: [4](#0-3) 

Because `removeTransactionApprover` never calls this gate and accepts no `user` parameter, the controller cannot pass a user for service-level enforcement. The service-level gap is structural, not just a controller omission.

**Exploit flow:**
1. Attacker registers as a normal user (no privilege required).
2. Attacker observes or enumerates a `transactionId` they can see (e.g., via `getTransactions`, which returns transactions where the user is a signer, observer, or approver).
3. Attacker calls `GET /transactions/:id/approvers` (accessible to any participant) to retrieve approver IDs.
4. Attacker calls `DELETE /transactions/:transactionId/approvers/:approverId` — the service removes the approver with no creator check.
5. Required approvers are gone; the transaction may proceed without the intended approval gate, or the approval tree is corrupted.

### Impact Explanation

- **Approval workflow bypass**: An attacker can remove all required approvers from a victim's transaction, allowing it to advance to execution without the intended multi-party sign-off.
- **Governance/workflow DoS**: Repeatedly removing approvers from active transactions disrupts the organization's signing coordination, forcing manual remediation.
- **Integrity violation**: The approval tree is a trust model for multi-signature coordination. Silent deletion by an unauthorized party breaks the integrity invariant that only the transaction creator controls approver configuration.

Severity: **High** — direct unauthorized state mutation with no privilege required beyond a valid user account.

### Likelihood Explanation

- Attacker precondition: valid authenticated session only (normal user account).
- Approver IDs are integers, discoverable via the approvers list endpoint accessible to any transaction participant.
- No rate limiting or anomaly detection is described for this endpoint.
- The attack is silent — no notification distinguishes "creator removed approver" from "attacker removed approver."

Likelihood: **High** — low barrier, no special knowledge required beyond being a registered user.

### Recommendation

Add the creator authorization check to `removeTransactionApprover`, mirroring the pattern used in `createTransactionApprovers` and `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve the root transaction and enforce creator-only access
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  await this.getCreatorsTransaction(rootNode.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Update the controller to pass the authenticated `user` to this method, consistent with how `createTransactionApprovers` and `updateTransactionApprover` are called.

### Proof of Concept

**Setup:**
- User A creates a transaction and adds User B and User C as required approvers (approver IDs 10 and 11).
- User D is a participant (e.g., observer or signer) on the same transaction and can list approvers.

**Attack:**
```
# Step 1: List approvers (accessible to any participant)
GET /transactions/42/approvers
→ [{ id: 10, userId: B }, { id: 11, userId: C }]

# Step 2: Remove approvers as User D (not the creator)
DELETE /transactions/42/approvers/10   → 200 OK
DELETE /transactions/42/approvers/11   → 200 OK
```

**Result:** Both required approvers are deleted. The transaction's approval gate is destroyed. User A's transaction can now proceed (or be executed) without the intended approvals, or the approval state is corrupted beyond recovery without manual intervention. No error is returned to the attacker; no notification distinguishes this from a legitimate creator action.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
