### Title
Missing `HasKeyGuard` on `importSignatures` Allows Key-less Users to Bypass Key Requirement

### Summary
The `TransactionsController` applies `HasKeyGuard` to `createTransaction` to enforce that only users with at least one registered key can interact with transactions. However, the `importSignatures` endpoint on the same controller omits this guard entirely. Any authenticated, verified user — even one with zero registered keys — can call `POST /transactions/signatures/import` directly, bypassing the key-ownership requirement that protects `createTransaction`.

### Finding Description
In `back-end/apps/api/src/transactions/transactions.controller.ts`, the controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) are applied to all routes: [1](#0-0) 

The `createTransaction` method additionally enforces `HasKeyGuard`, which calls `UserKeysService.getUserKeysCount` and throws `UnauthorizedException` if the user has no keys: [2](#0-1) 

However, `importSignatures` — which also performs a privileged, state-mutating action (injecting signature maps into transactions) — carries **no** `HasKeyGuard`: [3](#0-2) 

The `HasKeyGuard` itself confirms the enforcement logic that is absent from `importSignatures`: [4](#0-3) 

The exploit path is direct:
1. Attacker registers and verifies an account but registers **no** user keys.
2. Attacker calls `POST /transactions/signatures/import` with a crafted `UploadSignatureMapDto[]` body.
3. The controller-level JWT guards pass (user is authenticated and verified).
4. `HasKeyGuard` is never evaluated — the request reaches `transactionsService.importSignatures`.

### Impact Explanation
A key-less user can inject arbitrary signature maps into existing transactions, corrupting or manipulating the signing state of transactions they do not own and for which they hold no cryptographic material. This undermines the integrity of the multi-signature approval pipeline, which is the core security invariant of the system.

**Impact: Medium** — unauthorized state mutation of transaction signing records without possessing any valid key.

### Likelihood Explanation
The attacker only needs a valid, verified account — no privileged access, no leaked secrets. Registration is an open flow (`POST /auth/signup` by an admin, or the standard onboarding path). The missing guard is a single missing decorator, making the bypass trivially reachable by any authenticated user.

**Likelihood: High**

### Recommendation
Add `@UseGuards(HasKeyGuard)` to the `importSignatures` method, mirroring the pattern used on `createTransaction`:

```typescript
@Post('/signatures/import')
@HttpCode(201)
@UseGuards(HasKeyGuard)   // <-- add this
@Serialize(SignatureImportResultDto)
async importSignatures(...) { ... }
``` [5](#0-4) 

### Proof of Concept
```
# Step 1 – Obtain a JWT for a verified user with zero registered keys
POST /auth/login
{ "email": "keyless@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2 – Confirm no keys exist for this user
GET /user-keys
Authorization: Bearer <JWT>
→ []   (empty — HasKeyGuard would block createTransaction here)

# Step 3 – Call importSignatures directly (no HasKeyGuard present)
POST /transactions/signatures/import
Authorization: Bearer <JWT>
Content-Type: application/json
[
  {
    "transactionId": 42,
    "signatureMap": { ... }
  }
]
→ HTTP 201  ← succeeds despite user having no keys
```

The request succeeds and `transactionsService.importSignatures` is invoked, mutating transaction signing state, while the equivalent `POST /transactions` call would have been rejected with `"You should have at least one key to perform this action."` [6](#0-5)
