### Title
Any Authenticated User Can Delete Approvers Belonging to Transactions They Do Not Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of `:transactionId`, but then deletes the approver record identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` and a victim's approver `id`.

### Finding Description

**Root cause — decoupled ownership check and resource deletion:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

1. Verifies the caller is the creator of the URL-supplied `transactionId`.
2. Deletes the approver record identified by the URL-supplied `id`. [1](#0-0) 

The service method `removeTransactionApprover` receives only the approver `id` and performs no cross-reference to confirm that approver belongs to the transaction whose ownership was just verified: [2](#0-1) 

Contrast this with `updateTransactionApprover`, which explicitly validates that the approver's root transaction matches the URL `transactionId` before proceeding: [3](#0-2) 

The delete path has no equivalent guard. The `getCreatorsTransaction` call in the controller only proves the caller owns `transactionId`; it says nothing about whether approver `id` belongs to that transaction. [4](#0-3) 

**Exploit flow:**

1. Attacker registers as a normal user (no admin required).
2. Attacker creates any transaction → receives `transactionId = A` (attacker is creator).
3. Attacker learns a victim approver ID `B` (sequential integer IDs; or attacker is an observer/signer of another transaction and can read its approvers via `GET /transactions/:victimTxId/approvers`).
4. Attacker sends: `DELETE /transactions/A/approvers/B`
5. Server verifies attacker owns transaction `A` ✓, then deletes approver `B` (which belongs to victim transaction `C`) ✗.
6. Approver `B` is permanently removed from transaction `C`.

### Impact Explanation

Deleting an approver from a transaction the attacker does not own breaks the multi-signature approval workflow. An attacker can:

- Remove required approvers from any pending transaction, causing it to proceed with fewer approvals than intended or to become permanently stuck.
- Silently sabotage organizational governance by stripping approval requirements from high-value transactions before they execute.
- Trigger unintended `emitTransactionStatusUpdate` events for arbitrary transactions, polluting the notification system.

This constitutes unauthorized state mutation of another user's transaction — a direct integrity failure in the trust model.

### Likelihood Explanation

- **Preconditions**: Only a valid JWT (any authenticated, verified user). No admin role, no special privilege.
- **Attacker entry point**: Standard REST endpoint, reachable from any HTTP client.
- **Approver ID discovery**: IDs are sequential integers. An attacker who is an observer or signer on any shared transaction can read approver IDs directly via `GET /transactions/:id/approvers`. Blind enumeration is also trivial.
- **No rate-limit barrier**: The endpoint is not rate-limited beyond standard throttling.

Likelihood is **high**.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the ownership-verified transaction. Mirror the guard already present in `updateTransactionApprover`:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root node and verify it belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller to the service method, matching the pattern already used by `updateTransactionApprover`.

### Proof of Concept

**Setup:**
- User A creates transaction `txA` (ID = 10). User A is the creator.
- User B creates transaction `txB` (ID = 20) with approver record (ID = 99) assigned to User C.

**Attack (as User A):**
```http
DELETE /transactions/10/approvers/99
Authorization: Bearer <User A's JWT>
```

**Expected (correct) behavior:** 403 Unauthorized — approver 99 does not belong to transaction 10.

**Actual behavior:** 200 OK — approver 99 is deleted from transaction 20, removing User C's required approval from User B's transaction. [1](#0-0) [2](#0-1)

```typescript
  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
