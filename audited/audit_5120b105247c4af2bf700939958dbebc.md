### Title
IP Throttler Bypass via Spoofed `X-Forwarded-For` Header Enables Unbounded DoS on Resource-Intensive API Endpoints

---

### Summary

The global `IpThrottlerGuard` — the only rate-limiting protection applied to all API routes — uses the client-supplied `X-Forwarded-For` header as its tracking key without any trusted-proxy validation. An authenticated user can trivially bypass this guard by rotating the header value on every request, effectively removing all rate limiting. This allows flooding of expensive endpoints (notably `POST /transactions/signatures/import`) that perform array-based DB writes, causing resource exhaustion and degraded service for all users.

---

### Finding Description

`IpThrottlerGuard` is registered as the sole global `APP_GUARD` in `api.module.ts`: [1](#0-0) 

Its tracker implementation reads the `X-Forwarded-For` header directly from the request: [2](#0-1) 

Because `req.headers['x-forwarded-for']` is evaluated first and Express has no trusted-proxy configuration set in `setup-app.ts`, a client can supply any arbitrary string as this header. The throttler stores rate-limit counters keyed by this string. By sending a unique `X-Forwarded-For` value with each request (e.g., `1.1.1.1`, `1.1.1.2`, …), the attacker creates a fresh counter bucket every time and never accumulates hits against the configured limit. [3](#0-2) 

A `UserThrottlerModule` and `UserThrottlerGuard` exist with hardcoded per-user limits (100 req/min, 10 req/sec), but `UserThrottlerModule` is **never imported** in `api.module.ts` and `UserThrottlerGuard` is **never applied** to any controller or endpoint — leaving the IP guard as the only line of defense. [4](#0-3) 

The most impactful target is `POST /transactions/signatures/import`: [5](#0-4) 

This endpoint accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` (up to the 2 MB JSON body limit), calls `transformAndValidateDto` on every element, and then delegates to `transactionsService.importSignatures` which performs multiple DB writes per item. With the throttler bypassed, an attacker can submit back-to-back large arrays, multiplying DB load linearly.

---

### Impact Explanation

An authenticated user can exhaust database connection pool capacity and CPU by flooding `POST /transactions/signatures/import` (or any other DB-backed endpoint such as `GET /transactions/history`, `GET /transactions`, `POST /transactions`) with no effective rate limit. This degrades or denies service for all other users of the organization server — disrupting transaction creation, signing, and approval workflows.

---

### Likelihood Explanation

- Requires a valid authenticated session (reduces open-internet exposure).
- Any verified user account suffices — no admin privilege needed.
- The bypass is trivial: a single `curl` loop with an incrementing `X-Forwarded-For` header value is sufficient.
- The `UserThrottlerGuard` infrastructure already exists in the codebase, indicating the developers intended per-user limiting but never wired it up, making this an oversight rather than a deliberate design choice.

---

### Recommendation

1. **Fix the tracker**: Configure Express trusted-proxy settings so that `req.ip` reflects the real client IP from the TCP connection, and remove the unconditional trust of `X-Forwarded-For`:

   ```typescript
   // In setup-app.ts, before other middleware
   app.set('trust proxy', false); // or a specific trusted proxy count/CIDR
   ```

   Then in `IpThrottlerGuard`, use only `req.ip`:

   ```typescript
   protected getTracker(req: Record<string, any>): Promise<string> {
     if (!req.ip) throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
     return Promise.resolve(req.ip);
   }
   ```

2. **Wire up `UserThrottlerGuard`**: Import `UserThrottlerModule` in `api.module.ts` and apply `UserThrottlerGuard` (either globally or on expensive endpoints) so that per-user limits are enforced independently of IP.

3. **Add endpoint-specific limits** on `POST /transactions/signatures/import` to cap both request rate and array payload size.

---

### Proof of Concept

```bash
# Authenticated attacker floods importSignatures with a unique spoofed IP per request
TOKEN="<valid_jwt>"
for i in $(seq 1 10000); do
  curl -s -o /dev/null \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.$((i/256)).$((i%256))" \
    -d '[{"transactionId":1,"publicKey":"...","signature":"..."}]' \
    http://<server>/transactions/signatures/import &
done
```

Each request lands in a fresh throttler bucket (keyed by the unique spoofed IP), so the `GLOBAL_MINUTE_LIMIT` and `GLOBAL_SECOND_LIMIT` counters are never incremented for any single key, and all 10 000 requests are processed without throttling.
