### Title
`getTransactionsToSign` Fetches All Transactions Without Database-Level Pagination, Enabling DoS via Unbounded Processing

### Summary

`getTransactionsToSign` in `transactions.service.ts` accepts pagination parameters but never applies them to the database query. It loads **every** non-terminal transaction from the database, calls the expensive `userKeysToSign` operation on each one, and only then slices the result in memory. This is the same vulnerability class as M-25: a bounded-processing function ignores its bounding parameters, causing unbounded work proportional to total data volume.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method receives `{ page, limit, size, offset }` from the `PaginationParams` decorator (which caps `size` at 100), but the database query at line 295 omits both `take` and `skip`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take`, no `skip`
});
```

After loading the full unbounded result set, the code iterates over every transaction and calls `userKeysToSign` for each one — a cryptographic key-matching operation — before finally applying pagination in memory:

```typescript
items: result.slice(offset, offset + limit),
```

Compare this to the other paginated methods in the same file (`getTransactions`, `getHistoryTransactions`, `getTransactionsToApprove`) which all correctly pass `take: limit` and `skip: offset` to the database query.

The endpoint `GET /transactions/sign` is still registered and reachable by any authenticated user, even though the controller comment notes it is "NO LONGER USED BY FRONT-END." [1](#0-0) [2](#0-1) [3](#0-2)

### Impact Explanation

Any authenticated user can send repeated requests to `GET /transactions/sign?page=1&size=1`. Each request causes the server to:
1. Load **all** active (non-terminal) transactions from PostgreSQL into memory.
2. Call `userKeysToSign` (cryptographic key derivation and matching) for every transaction.
3. Return only 1 item.

As the transaction table grows, each request consumes proportionally more memory and CPU. Under sustained load or with a large transaction table, this can exhaust server resources and degrade or halt the API service — the same chain-halt class of impact described in M-25.

### Likelihood Explanation

The endpoint requires only a valid authenticated session — no admin role, no special privilege. Any registered user can trigger it. The endpoint is still live despite the front-end no longer using it, meaning it is an unguarded attack surface. The vulnerability is triggered by normal API usage patterns (requesting the sign queue), not a contrived exploit.

### Recommendation

Apply database-level pagination to the `getTransactionsToSign` query, consistent with all other paginated methods in the same service:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // ← add this
  skip: offset,  // ← add this
});
```

Note that applying DB-level pagination here changes the semantics of `totalItems` (it would no longer reflect the true count of signable transactions). The correct fix is to either:
- Run a separate `count` query for `totalItems`, or
- Restructure the query to push the key-matching logic into SQL (as the newer `transaction-nodes` service does with `getTransactionNodesQuery`).

If the endpoint is truly deprecated, it should be removed entirely.

### Proof of Concept

1. Register as any user with at least one key.
2. Create a large number of transactions (or wait for the platform to accumulate them).
3. Send repeated requests: `GET /transactions/sign?page=1&size=1`
4. Observe that server CPU and memory scale with the total number of active transactions, not with the requested page size of 1. [5](#0-4) [6](#0-5)

```typescript
  /* Get the transactions that a user needs to sign */
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
