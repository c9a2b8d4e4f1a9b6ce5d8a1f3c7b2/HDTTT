### Title
Unbounded `signatureMap` Entries and Unbounded `UploadSignatureMapDto[]` Array Enable Authenticated DoS on Signature Upload Endpoints

### Summary
The `POST /transactions/signers` and `POST /transactions/signatures/import` endpoints accept an unbounded outer array of `UploadSignatureMapDto` objects, each containing a `signatureMap` with no limit on the number of node account IDs, transaction IDs, or public keys. No `@ArrayMaxSize()` constraint exists on either the outer array or the inner map entries. A malicious authenticated user can submit a single crafted request that forces the server to perform unbounded concurrent processing via `Promise.all`, exhaust event-loop time during synchronous validation, and saturate database connection pools — degrading or denying service to all other users of the organization backend.

### Finding Description

**Root cause 1 — Unbounded outer array, concurrent fan-out**

`SignersController.uploadSignatureMap` accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no size guard: [1](#0-0) 

`transformAndValidateDto` iterates over every element and calls `plainToInstance` + `validateOrReject` per item. After validation, `validateAndProcessSignatures` fans out all items concurrently: [2](#0-1) 

`Promise.all(dto.map(...))` spawns one async task per array element simultaneously. There is no concurrency cap (no `p-limit` or equivalent). A 100-element array spawns 100 concurrent DB-deserialize-sign-status-update pipelines in a single request.

**Root cause 2 — Unbounded `signatureMap` entries, expensive synchronous validation**

`UploadSignatureMapDto.signatureMap` is decorated only with `@IsSignatureMap()`: [3](#0-2) 

The `IsSignatureMap` `Transform` function iterates over every `nodeAccountId → transactionId → publicKey` triple, calling `AccountId.fromString`, `TransactionId.fromString`, `PublicKey.fromString`, hex-decoding the signature, and calling `signatureMap.addSignature` — all synchronously on the Node.js event loop — with no limit on the number of entries: [4](#0-3) 

`processTransactionSignatures` then iterates over the same entries again: [5](#0-4) 

**Root cause 3 — Same pattern on `importSignatures`**

`TransactionsController.importSignatures` has the identical unbounded-array pattern: [6](#0-5) 

**Root cause 4 — Unbounded `approversArray` with unbounded recursive depth**

`CreateTransactionApproversArrayDto.approversArray` has `@ArrayMinSize` but no `@ArrayMaxSize`, and `CreateTransactionApproverDto.approvers` is also unbounded: [7](#0-6) 

The service processes this with an unbounded recursive function that issues one `INSERT` per node: [8](#0-7) 

A deeply nested tree causes unbounded recursion and unbounded sequential DB inserts within a single database transaction.

### Impact Explanation

- **Event-loop starvation**: The synchronous `IsSignatureMap` transform blocks the Node.js event loop proportionally to the number of entries. A single request with a large map can freeze the API for all concurrent users.
- **Memory exhaustion**: Each `SignatureMap` object and each deserialized `SDKTransaction` is held in memory for the duration of processing. A large outer array with large inner maps can exhaust heap memory.
- **DB connection pool saturation**: `Promise.all` with a large outer array spawns concurrent DB operations that can exhaust the TypeORM connection pool, causing all subsequent requests to queue or time out.
- **Cascading service degradation**: The API, Chain, and Notifications microservices share the same NATS bus; a flood of notification emissions from a single large request propagates load to downstream services.

### Likelihood Explanation

The attacker only needs a valid JWT for a verified user account — the lowest privilege level in the system. No admin role, no special key, no internal access is required. The endpoints are reachable over standard HTTPS. A single HTTP POST is sufficient to trigger the condition. Any registered user of an organization deployment can exploit this.

### Recommendation

1. **Enforce `@ArrayMaxSize` on the outer DTO array** in both `uploadSignatureMap` and `importSignatures` (e.g., max 50 items per request).
2. **Enforce entry-count limits inside `IsSignatureMap`**: cap the number of `nodeAccountId`, `transactionId`, and `publicKey` entries (e.g., max 20 nodes × max 5 transaction IDs × max 10 keys).
3. **Replace `Promise.all` with a concurrency-limited alternative** (e.g., `p-limit`) in `validateAndProcessSignatures`.
4. **Enforce `@ArrayMaxSize` on `approversArray`** and add a maximum recursion depth guard in `createApprover`.
5. **Add a global NestJS body-size limit** and per-user rate limiting on mutation endpoints.

### Proof of Concept

**Step 1**: Register and verify a normal user account on the organization backend.

**Step 2**: Obtain a valid JWT via `POST /auth/login`.

**Step 3**: Craft a request body with a large outer array. Each item references the same valid transaction ID and contains a `signatureMap` with many valid-looking entries (valid account IDs, transaction IDs, and public keys can be generated trivially):

```json
POST /transactions/signers
Authorization: Bearer <jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1000000000.000000000": { "<valid_der_pubkey_1>": "<hex_sig>", "<valid_der_pubkey_2>": "<hex_sig>", ... /* 50 keys */ } }, "0.0.4": { ... }, ... /* 20 nodes */ } } },
  { "id": 1, "signatureMap": { ... same large map ... } },
  ... /* 50 items */
]
```

**Expected outcome**: The server's event loop is blocked during synchronous `IsSignatureMap` validation of 50 × 20 × 5 × 50 = 250,000 SDK parse operations, followed by 50 concurrent async pipelines each deserializing transaction bytes and issuing DB queries. Concurrent legitimate requests experience severe latency or timeout. Repeating the request in a loop (even at low rate) sustains the degradation.
