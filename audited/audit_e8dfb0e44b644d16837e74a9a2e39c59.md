### Title
`isSignableStatus` Front-End Guard Restricts Signing to `WAITING_FOR_SIGNATURES` While the Back-End Accepts Signatures in `WAITING_FOR_EXECUTION` Too

### Summary
The front-end utility `isSignableStatus` only permits the sign action when a transaction is in the `WAITING_FOR_SIGNATURES` state. However, the back-end signing endpoint (`SignersService.validateTransactionStatus`) explicitly accepts signatures for transactions in both `WAITING_FOR_SIGNATURES` **and** `WAITING_FOR_EXECUTION`. Additionally, the back-end collection constant `READY_TO_SIGN` includes `WAITING_FOR_EXECUTION`, meaning the server itself surfaces those transactions to users as signable. The result is a state-transition mismatch: a transaction in `WAITING_FOR_EXECUTION` appears in the "Ready to Sign" tab but the sign button is never rendered, silently blocking legitimate signers.

---

### Finding Description

**Back-end allows signing in both states**

`SignersService.validateTransactionStatus` in `signers.service.ts`:

```typescript
private validateTransactionStatus(transaction: Transaction): string | null {
  if (
    transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
    transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION   // ← also accepted
  ) {
    return ErrorCodes.TNRS;
  }
  ...
}
``` [1](#0-0) 

The back-end `READY_TO_SIGN` collection also explicitly includes `WAITING_FOR_EXECUTION`:

```typescript
READY_TO_SIGN: [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,   // ← included
],
``` [2](#0-1) 

**Front-end guard is more restrictive**

`isSignableStatus` in `transactionStatusGuards.ts` only returns `true` for `WAITING_FOR_SIGNATURES`:

```typescript
export const isSignableStatus = (status: TransactionStatus | null | undefined): boolean =>
  status === TransactionStatus.WAITING_FOR_SIGNATURES;   // WAITING_FOR_EXECUTION excluded
``` [3](#0-2) 

`canSign` in `TransactionDetailsHeader.vue` gates the sign button on this guard:

```typescript
const canSign = computed(() => {
  ...
  if (!isSignableStatus(props.organizationTransaction.status)) return false;
  ...
});
``` [4](#0-3) 

The test suite explicitly encodes this narrower intent:

```typescript
test('signable only when waiting for signatures', () => {
  expect(isSignableStatus(TransactionStatus.WAITING_FOR_SIGNATURES)).toBe(true);
  expect(isSignableStatus(TransactionStatus.WAITING_FOR_EXECUTION)).toBe(false);
});
``` [5](#0-4) 

---

### Impact Explanation

A transaction moves to `WAITING_FOR_EXECUTION` once the minimum required signature threshold is reached. However, the back-end intentionally continues to accept additional signatures in that state (as evidenced by both `validateTransactionStatus` and the `READY_TO_SIGN` collection). A user whose key is still required (or whose additional signature is desired) will see the transaction listed in the "Ready to Sign" tab — because the back-end returns it — but will find no sign button on the details page. The only workaround is a raw API call, which ordinary users cannot be expected to perform. This silently blocks legitimate signers from completing their signing obligation.

---

### Likelihood Explanation

This is triggered in the normal multi-signer workflow whenever the first signer's submission causes the status to advance to `WAITING_FOR_EXECUTION` before all intended signers have acted. Given that the `READY_TO_SIGN` collection surfaces these transactions to users, the scenario is a routine part of the intended workflow, not an edge case.

---

### Recommendation

Align `isSignableStatus` with the back-end's accepted states:

```typescript
export const isSignableStatus = (status: TransactionStatus | null | undefined): boolean =>
  [
    TransactionStatus.WAITING_FOR_SIGNATURES,
    TransactionStatus.WAITING_FOR_EXECUTION,   // add this
  ].includes(status as TransactionStatus);
```

This mirrors the pattern already used by `isApprovableStatus`, which correctly includes both states. [6](#0-5) 

---

### Proof of Concept

1. User A creates a multi-signer transaction; status is `WAITING_FOR_SIGNATURES`.
2. User A signs first; the back-end calls `processTransactionStatus` and advances the status to `WAITING_FOR_EXECUTION`.
3. User B (also a required signer) opens the app. The "Ready to Sign" tab shows the transaction (back-end `READY_TO_SIGN` collection includes `WAITING_FOR_EXECUTION`).
4. User B clicks the transaction. `canSign` evaluates `isSignableStatus('WAITING_FOR_EXECUTION')` → `false`. The sign button is not rendered.
5. User B cannot sign through the UI. Calling `POST /transactions/:id/signers` directly with a valid signature map succeeds (back-end accepts it), confirming the front-end guard is the sole blocker.
