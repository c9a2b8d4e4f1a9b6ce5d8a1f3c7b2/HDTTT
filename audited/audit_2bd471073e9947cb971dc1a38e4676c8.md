### Title
Any Authenticated User Can Delete Approvers from Transactions They Do Not Own via Mismatched `transactionId` / Approver `id` Parameters

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but never verifies that the approver identified by `:id` actually belongs to that same transaction. An attacker who is the creator of any transaction can supply their own `transactionId` (passing the ownership check) while supplying an approver `id` from a completely different user's transaction, causing unauthorized deletion of that approver.

### Finding Description

**Root cause — missing cross-entity ownership check in the delete path.**

The controller handler is: [1](#0-0) 

Step 1 (`getCreatorsTransaction`) verifies the authenticated user is the creator of the URL-supplied `transactionId`. Step 2 (`removeTransactionApprover(id)`) then deletes the approver record identified by the URL-supplied `id` with no check that this approver belongs to `transactionId`: [2](#0-1) 

The service only checks that the approver record exists, then removes it. There is no assertion that `approver.transactionId === transactionId`.

**Contrast with the update path**, which correctly performs this cross-entity check: [3](#0-2) 

The update path explicitly rejects requests where the approver's root transaction does not match the URL `transactionId`. The delete path has no equivalent guard.

### Impact Explanation

Any verified, authenticated user who is the creator of at least one transaction (a normal product workflow) can:

1. Enumerate or guess approver IDs from another user's transaction (IDs are sequential integers).
2. Call `DELETE /transactions/{own_tx_id}/approvers/{victim_approver_id}`.
3. The ownership check passes (they own `own_tx_id`), and the victim's approver is silently deleted.

Concrete consequences:
- **Approval workflow sabotage**: Removing required approvers from a victim's transaction causes it to skip mandatory approval gates and potentially advance to execution without the required sign-off.
- **Unauthorized state mutation**: The victim's transaction state is permanently altered without their knowledge or consent.
- **Integrity failure**: The multi-signature/approval trust model of the platform is broken for any transaction whose approvers can be deleted by an unrelated user.

### Likelihood Explanation

- Attacker precondition: must be a registered, verified user who has created at least one transaction — a normal user action with no privilege required.
- Approver IDs are auto-incremented integers, trivially enumerable.
- No rate-limiting or anomaly detection prevents repeated probing.
- The attack is a single authenticated HTTP request with no side-channel requirements.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the resolved approver's root transaction matches the URL-supplied `transactionId`, mirroring the guard already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Mirror the check in updateTransactionApprover
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (owns it).
- User B creates Transaction 2 with Approver record `id = 42` (e.g., requiring User C's approval).

**Attack (executed by User A):**
```
DELETE /transactions/1/approvers/42
Authorization: Bearer <User A's JWT>
```

**Execution trace:**
1. `getCreatorsTransaction(1, userA)` → passes; User A is creator of Transaction 1. [4](#0-3) 
2. `removeTransactionApprover(42)` → fetches approver 42 (belongs to Transaction 2), finds it exists, deletes it. No `transactionId` comparison is performed. [2](#0-1) 

**Result:** Approver 42 is deleted from User B's Transaction 2. User B's approval workflow is permanently corrupted. User A receives HTTP 200 `true`.

```typescript
  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
```
