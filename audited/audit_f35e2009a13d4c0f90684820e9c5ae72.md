### Title
`removeTransactionApprover` Authorization Check Performed on Wrong Entity Allows Cross-Transaction Approver Deletion

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the authenticated user is the creator of `:transactionId`, but then passes the unrelated `:id` parameter directly to `removeTransactionApprover()` without verifying that the targeted approver actually belongs to the authorized transaction. An attacker who is the creator of any transaction can delete approvers from transactions they do not own by supplying their own `transactionId` for the authorization check while targeting a foreign approver `id`.

### Finding Description

**Root cause:** In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the URL-supplied `transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver with the URL-supplied `id`. [1](#0-0) 

These two checks are **decoupled**. The service method `removeTransactionApprover` fetches the approver by `id` and soft-deletes it without ever verifying that `approver.transactionId` matches the `transactionId` that was authorized: [2](#0-1) 

Contrast this with `updateTransactionApprover`, which **does** perform the cross-entity check: [3](#0-2) 

The `updateTransactionApprover` path explicitly checks `rootNode.transactionId !== transactionId` before proceeding. The `removeTransactionApprover` path has no equivalent guard.

**Exploit flow:**

1. Attacker (User A) creates Transaction T_A — they are its creator.
2. Attacker learns the approver `id` of an approver record belonging to Transaction T_B (owned by User B). This is feasible because:
   - Approver IDs are sequential integers and can be enumerated.
   - If the attacker is an observer or signer on T_B, `getVerifiedApproversByTransactionId` returns the approver records including their IDs.
3. Attacker sends: `DELETE /transactions/{T_A_id}/approvers/{approver_from_T_B_id}`
4. `getCreatorsTransaction(T_A_id, user)` passes — attacker is creator of T_A.
5. `removeTransactionApprover(approver_from_T_B_id)` executes — the approver from T_B is soft-deleted with no ownership check. [4](#0-3) 

### Impact Explanation

An attacker can unilaterally remove required approvers from any transaction in the system, including transactions they have no legitimate access to. This directly undermines the multi-signature approval workflow: by deleting approvers, the attacker can reduce or eliminate the approval threshold for a transaction, potentially allowing it to proceed to execution without the required organizational sign-off. This is an unauthorized state change with direct impact on transaction integrity and governance.

### Likelihood Explanation

The attacker only needs to be a registered user who has created at least one transaction (a normal product flow). Approver IDs are sequential integers, making enumeration trivial. If the attacker is also an observer or signer on the target transaction, they can obtain the exact approver `id` without guessing. No privileged access is required.

### Recommendation

In `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted belongs to the authorized `transactionId`. Mirror the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the authorized transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [5](#0-4) 

### Proof of Concept

**Preconditions:**
- Attacker is a registered, authenticated user.
- Attacker has created Transaction T_A (so they pass the creator check).
- Target Transaction T_B exists with at least one approver (approver `id` = `X`). The attacker can obtain `X` by being an observer on T_B, or by integer enumeration.

**Steps:**
1. Authenticate as the attacker.
2. Send: `DELETE /transactions/{T_A_id}/approvers/{X}`
3. **Expected (correct) behavior:** 403 Unauthorized — approver X does not belong to T_A.
4. **Actual behavior:** 200 OK — approver X is soft-deleted from T_B.

**Verification:** Query the approver table for T_B; the approver with `id = X` now has a non-null `deletedAt`, removing it from T_B's approval requirements. [1](#0-0) [2](#0-1)
