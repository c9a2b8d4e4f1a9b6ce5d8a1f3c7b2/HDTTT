### Title
Missing Transaction Status Check in `createTransactionApprovers` and `updateTransactionApprover` Allows Approver Tree Modification on Terminal-State Transactions

### Summary
The `createTransactionApprovers` and `updateTransactionApprover` functions in `approvers.service.ts` delegate authorization to `getCreatorsTransaction`, which only verifies transaction existence and creator ownership — it never validates the transaction's current status. As a result, a transaction creator can add or restructure approvers on transactions that are already in terminal states (`EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, `ARCHIVED`) or in the `WAITING_FOR_EXECUTION` state, corrupting the audit trail and triggering unintended status recalculations.

### Finding Description

**Root cause — `getCreatorsTransaction` performs no status check:** [1](#0-0) 

The function only asserts the transaction exists and that `creatorKey.userId === user.id`. No status guard is present.

**`createTransactionApprovers` relies solely on this check:** [2](#0-1) 

After the single `getCreatorsTransaction` call, the function proceeds to insert new `TransactionApprover` rows regardless of whether the transaction is `EXECUTED`, `CANCELED`, or any other terminal state. [3](#0-2) 

It then calls `emitTransactionStatusUpdate`, which publishes a NATS event that triggers status recalculation on the now-modified transaction.

**`updateTransactionApprover` has the same gap:** [4](#0-3) 

The update path also calls `getCreatorsTransaction` with no status validation, allowing structural changes (re-parenting nodes, changing thresholds, swapping users) on transactions in any state.

**Contrast with `approveTransaction`, which correctly guards status:** [5](#0-4) 

The approval action explicitly rejects anything outside `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION`. The create/update paths have no equivalent guard.

**Pre-population of approval state on new approver nodes:** [6](#0-5) 

When a new approver node is created for a user who already has an existing approver record on the same transaction, the new node inherits the existing `signature`, `userKeyId`, and `approved` values. On a `WAITING_FOR_EXECUTION` transaction this means a creator can inject a structurally new approver node that arrives pre-approved, silently satisfying threshold requirements without the designated user performing any action.

### Impact Explanation

1. **Audit-trail corruption on terminal transactions**: Approver records can be inserted or restructured after a transaction has been `EXECUTED`, `CANCELED`, `FAILED`, or `EXPIRED`, making the historical record of who approved what unreliable.

2. **Forced status regression on `WAITING_FOR_EXECUTION` transactions**: Adding an unapproved approver node to a transaction that has already gathered all required approvals triggers `emitTransactionStatusUpdate`. The chain service recalculates status and can move the transaction back to `WAITING_FOR_SIGNATURES`, indefinitely stalling execution until the newly injected approver acts.

3. **Phantom pre-approved nodes**: Because the code copies existing approval state into newly created approver nodes (lines 325–329), a creator can manufacture approver nodes that appear already-approved without the target user performing any signing action, silently satisfying threshold counts.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The endpoint `POST /transactions/:transactionId/approvers` is reachable by any verified user for their own transactions. No special tooling is required; a standard HTTP client suffices. The condition (a transaction in `WAITING_FOR_EXECUTION` or a terminal state) arises in normal operation.

### Recommendation

Add an explicit status guard inside `createTransactionApprovers` and `updateTransactionApprover` (or inside `getCreatorsTransaction` itself) that rejects the operation when the transaction is not in a mutable state. Only `NEW`, `WAITING_FOR_SIGNATURES`, and `WAITING_FOR_EXECUTION` should permit structural approver changes, and even `WAITING_FOR_EXECUTION` should be reconsidered given the regression risk.

```typescript
const allowedStatuses = [TransactionStatus.NEW, TransactionStatus.WAITING_FOR_SIGNATURES];
if (!allowedStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP); // Operation not permitted in this state
}
```

Additionally, the pre-population of `signature`/`approved` from existing approver records (lines 318–329) should be removed or gated behind an explicit policy decision, as it allows approval state to be silently inherited by newly created nodes.

### Proof of Concept

1. User A creates a transaction; it progresses to `WAITING_FOR_EXECUTION` (all required approvals collected).
2. User A sends `POST /transactions/{id}/approvers` with a new approver payload for User B (who has not yet approved).
3. The request succeeds — `createTransactionApprovers` inserts the new approver row and emits `emitTransactionStatusUpdate`.
4. The chain/API service recalculates status; the transaction now lacks the new approver's signature and is demoted to `WAITING_FOR_SIGNATURES`.
5. Execution is blocked until User B approves. User A can repeat step 2 with a different user to stall indefinitely.

Alternatively, for the phantom-approval variant:
1. User A creates a transaction; User B approves it (approver record now has `signature` and `approved = true`).
2. User A sends `POST /transactions/{id}/approvers` adding a second approver node for User B at a different position in the tree.
3. Lines 318–329 copy User B's existing `signature`/`approved` into the new node.
4. The new node counts as approved without User B performing any additional action, satisfying threshold requirements at the new tree position.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
