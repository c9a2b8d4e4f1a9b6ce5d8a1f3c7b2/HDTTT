### Title
`archiveTransaction()` Bypasses Status Guard When `isManual` Is True, Allowing Terminal-State Transactions to Be Archived

### Summary
The `archiveTransaction()` function in `TransactionsService` contains a logically flawed guard condition. Because the status check and the `isManual` flag are combined with `&&` (AND), any transaction with `isManual: true` bypasses the status check entirely. This allows the creator of a manual transaction to archive it even after it has reached a terminal state (`EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, `REJECTED`), corrupting the application's audit trail and state integrity.

### Finding Description

**Root cause — flawed boolean guard:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `archiveTransaction()` reads:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

The guard throws **only when both** conditions are true:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- **AND** `isManual === false`

The logical negation means the function proceeds (does NOT throw) when **either**:
- status IS in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`, OR
- `isManual === true` — **regardless of status**

When `isManual === true`, the status check is completely short-circuited. A manual transaction in `EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, or `REJECTED` state passes the guard and gets its status overwritten to `ARCHIVED`.

**Exploit path:**
1. Authenticated user creates a manual transaction (`isManual: true`) via `POST /transactions`.
2. The transaction reaches a terminal state (e.g., `EXECUTED` after on-chain execution, or `CANCELED`).
3. The creator calls `PATCH /transactions/archive/:id`.
4. `archiveTransaction()` fetches the transaction via `getTransactionForCreator()` (ownership verified), evaluates the guard — since `isManual === true`, the status check is skipped — and executes `repo.update({ id }, { status: TransactionStatus.ARCHIVED })`.
5. The transaction's status is now `ARCHIVED`, overwriting its true terminal state.

The controller endpoint is publicly reachable by any authenticated, verified user who is the transaction creator: [2](#0-1) 

The test suite confirms the happy path only tests `isManual: true` with `WAITING_FOR_EXECUTION`, and the negative test only tests `isManual` unset (falsy) with `CANCELED` — the combination of `isManual: true` + terminal status is never tested: [3](#0-2) 

### Impact Explanation

A transaction creator can overwrite the recorded outcome of any of their manual transactions from a terminal state to `ARCHIVED`. Concretely:

- An `EXECUTED` transaction (confirmed on-chain) can be re-labeled `ARCHIVED`, hiding that it was executed from observers, approvers, and audit logs within the application.
- A `FAILED` or `CANCELED` transaction can be re-labeled `ARCHIVED`, obscuring its true outcome.
- Notification and status-update events are emitted with the new `ARCHIVED` status, propagating the corrupted state to all subscribers.

This is an unauthorized state transition that violates the application's state integrity invariant: terminal states should be immutable.

### Likelihood Explanation

- **Attacker profile**: Any authenticated, verified user who has created at least one manual transaction. No privileged role required.
- **Preconditions**: The attacker must be the creator of a `isManual: true` transaction that has reached a terminal state. Manual transactions are a supported, documented feature.
- **Trigger**: A single `PATCH /transactions/archive/:id` HTTP request. No race condition or timing dependency required.

### Recommendation

Fix the guard to enforce status independently of `isManual`. The correct intent appears to be: archive is only allowed when the transaction is in an active (non-terminal) state, with `isManual` being an additional requirement (not a bypass):

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  const archivableStatuses = [
    TransactionStatus.WAITING_FOR_SIGNATURES,
    TransactionStatus.WAITING_FOR_EXECUTION,
  ];

  if (!archivableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  if (!transaction.isManual) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }
  // ...
}
```

This ensures that both conditions must be satisfied independently, and no value of `isManual` can bypass the status check.

### Proof of Concept

1. Authenticate as a normal user and create a manual transaction:
   ```
   POST /transactions  { ..., isManual: true }
   → 201 { id: 42, status: "NEW", isManual: true }
   ```
2. Upload sufficient signatures so the scheduler moves it to `WAITING_FOR_EXECUTION`, then trigger execution:
   ```
   PATCH /transactions/execute/42
   → 200 true
   ```
   Transaction status is now `EXECUTED` (or simulate by directly setting status in a test environment).
3. Immediately call archive:
   ```
   PATCH /transactions/archive/42
   → 200 true
   ```
4. Fetch the transaction:
   ```
   GET /transactions/42
   → { id: 42, status: "ARCHIVED", isManual: true }
   ```

Expected: `400 Bad Request` (transaction is in a terminal state).
Actual: `200 true`, status overwritten to `ARCHIVED`. [4](#0-3)

  /* Archive the transaction if the transaction is sign only. */
  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
