### Title
TOCTOU Race Condition in `approveTransaction` Allows an Approver to Bypass the "Already Approved" Guard and Corrupt Approval State

### Summary

`approveTransaction` in `approvers.service.ts` performs its duplicate-approval check outside of any database transaction and without a conditional `WHERE signature IS NULL` guard on the subsequent `UPDATE`. Two concurrent HTTP requests from the same authenticated approver can both pass the check and both write conflicting `approved` values, allowing the approver to effectively change their vote after it has been cast and corrupt the multi-signature approval state.

### Finding Description

**Root cause — TOCTOU between check and write**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` follows this sequence:

1. **Read** approver records (outside any DB transaction): [1](#0-0) 

2. **Check** whether the user has already voted (outside any DB transaction): [2](#0-1) 

3. **Write** the new approval decision — inside a DB transaction, but with **no** `WHERE signature IS NULL` condition: [3](#0-2) 

Because steps 1–2 are not inside the same serializable transaction as step 3, two concurrent requests from the same user can both read `signature = null`, both pass the guard at line 563, and both execute the `UPDATE`. The last write wins unconditionally.

**No unique constraint prevents duplicate writes**

The `transaction_approver` table has no unique constraint on `(transactionId, userId)`: [4](#0-3) 

The migration for `transaction_signer` similarly creates only a non-unique index: [5](#0-4) 

**Exploit flow**

```
Approver Alice (userId=5) is listed for transaction T (threshold = 1 of 1).

t=0  Request A: GET approvers → userApprovers[0].signature = null → passes guard
t=0  Request B: GET approvers → userApprovers[0].signature = null → passes guard
t=1  Request A: UPDATE transaction_approver SET approved=true, signature=sigA WHERE id=X
t=2  Request B: UPDATE transaction_approver SET approved=false, signature=sigB WHERE id=X
     → DB now shows approved=false, but the chain service may have already
       advanced the transaction to WAITING_FOR_EXECUTION based on Request A's write.
```

The endpoint is reachable by any authenticated user who is listed as an approver — no elevated privilege is required: [6](#0-5) 

### Impact Explanation

- **Vote manipulation**: An approver can race `approved: true` and `approved: false` simultaneously. The last write wins, allowing them to change their vote after the fact, bypassing the `ErrorCodes.TAP` guard entirely.
- **Inconsistent approval state**: If Request A's `approved: true` causes the chain service to advance the transaction to `WAITING_FOR_EXECUTION`, and Request B's `approved: false` then overwrites the approver record, the system is left in a state where the transaction is queued for execution but the approver record shows a rejection. This violates the core multi-signature integrity invariant.
- **Threshold manipulation**: In a 1-of-1 or N-of-N threshold scenario, a single malicious approver can approve to trigger advancement and then immediately overwrite with a rejection, corrupting the approval audit trail.

### Likelihood Explanation

The attacker is a legitimate approver — no privilege escalation is required. Sending two concurrent HTTP requests is trivial from any HTTP client (e.g., `Promise.all([fetch(...), fetch(...)])`). The race window spans the entire async execution path from the read at line 553 to the write at line 609, which in a Node.js/NestJS service under normal load is reliably exploitable. No special network conditions or timing precision is needed.

### Recommendation

1. **Move the duplicate-approval check inside the DB transaction and add a conditional WHERE clause** so the `UPDATE` is a no-op if the signature is already set:

```sql
UPDATE transaction_approver
SET "userKeyId" = $1, "signature" = $2, "approved" = $3
WHERE id = ANY($4)
  AND "signature" IS NULL   -- atomic guard
```

2. **Wrap the entire read-check-write sequence in a single serializable (or `REPEATABLE READ`) database transaction** so concurrent reads see a consistent snapshot and the write is gated on the check atomically.

3. **Add a unique constraint** on `(transactionId, userId)` in `transaction_approver` to enforce at the database level that a user can only have one approver record per transaction.

### Proof of Concept

```typescript
// Attacker is a valid approver for transactionId=42
const url = `${serverUrl}/transactions/42/approvers/approve`;
const headers = { Authorization: `Bearer ${attackerJwt}`, 'Content-Type': 'application/json' };

const approvePayload  = { userKeyId: 7, signature: sigTrue,  approved: true  };
const rejectPayload   = { userKeyId: 7, signature: sigFalse, approved: false };

// Fire both concurrently — both pass the signature=null guard
const [r1, r2] = await Promise.all([
  fetch(url, { method: 'POST', headers, body: JSON.stringify(approvePayload) }),
  fetch(url, { method: 'POST', headers, body: JSON.stringify(rejectPayload) }),
]);

// Expected (correct): second call throws TAP (already approved)
// Actual: both return 200; DB shows whichever write landed last,
//         while the chain service may have already acted on the first write.
```

**Expected outcome**: The `transaction_approver` row for the attacker ends up with `approved = false` (or `true`, depending on scheduling), while the transaction status may have already advanced to `WAITING_FOR_EXECUTION` based on the first write — producing an irreconcilable inconsistency in the approval audit trail.
