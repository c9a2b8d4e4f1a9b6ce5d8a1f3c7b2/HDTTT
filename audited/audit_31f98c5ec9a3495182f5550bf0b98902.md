### Title
Approver Signature Replay Across Organization Server Deployments via Missing Domain Separation in `verifyTransactionBodyWithoutNodeAccountIdSignature`

### Summary
The approval signature schema used in `approvers.service.ts` signs and verifies only the raw Hedera transaction body bytes (protobuf-encoded, with node account ID stripped). No domain separator — such as the server URL, organization ID, or network identifier — is included in the signed payload. This means a valid approver signature obtained from one organization server deployment is cryptographically valid on any other deployment that holds the same transaction bytes, enabling cross-server signature replay to forge approvals without the approver's knowledge or consent.

### Finding Description

**Root cause — signing function:**

`getTransactionBodySignatureWithoutNodeAccountId` in the front-end signs only the raw protobuf body bytes:

```ts
// front-end/src/renderer/utils/sdk/index.ts:330-339
const transactionBody = transaction._makeTransactionBody(null); // null strips node account ID
const bodyBytes = proto.TransactionBody.encode(transactionBody).finish();
const signature = privateKey.sign(bodyBytes);
``` [1](#0-0) 

**Root cause — verification function:**

`verifyTransactionBodyWithoutNodeAccountIdSignature` in the back-end verifies against the same raw body bytes with no additional context:

```ts
// back-end/libs/common/src/utils/sdk/transaction.ts:266-285
const bodyBytes = getTransactionBodyBytes(transaction); // _makeTransactionBody(null)
return publicKey.verify(bodyBytes, signature);
``` [2](#0-1) 

**Where it is enforced — approver flow:**

`approveTransaction` in `approvers.service.ts` calls this verification as the sole cryptographic check before persisting the approval:

```ts
// back-end/apps/api/src/transactions/approvers/approvers.service.ts:592-596
if (!verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey))
  throw new BadRequestException(ErrorCodes.SNMP);
``` [3](#0-2) 

The signed payload contains only the Hedera transaction body (payer account ID, timestamp, transaction type, parameters). It contains **no** server URL, organization identifier, or Hedera network/chain identifier. The `mirrorNetwork` field is stored in the database separately and is never included in the signed data.

**Exploit flow:**

1. Eve operates a malicious organization server (a second deployment of this same software).
2. Bob is a registered user on both Eve's server and the legitimate organization server, and is listed as an approver on a transaction on the legitimate server.
3. Eve creates a transaction on her server with **identical transaction body bytes** (same payer, same timestamp, same parameters) as the pending transaction on the legitimate server.
4. Eve adds Bob as an approver on her server's transaction and asks him to approve it (e.g., framing it as a test or routine approval).
5. Bob approves on Eve's server. His client calls `getTransactionBodySignatureWithoutNodeAccountId` and sends the signature to Eve's server.
6. Eve extracts Bob's signature and submits it to the legitimate server's `POST /transactions/:id/approvers` endpoint with Bob's `userKeyId`.
7. The legitimate server calls `verifyTransactionBodyWithoutNodeAccountIdSignature` — the signature is valid because the body bytes are identical. The approval is persisted as if Bob approved it on the legitimate server.

### Impact Explanation

An attacker who controls a second deployment of the organization server can forge approvals from any user who is registered on both servers. Since the approval workflow gates transaction execution (transactions move from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION` once approval thresholds are met), a forged approval can cause unauthorized Hedera transactions — including fund transfers, account key changes, or file operations — to be executed on the Hedera network without the approver's genuine consent for that specific server's transaction. [4](#0-3) 

### Likelihood Explanation

The attacker preconditions are realistic:
- Running a second instance of the open-source Hedera Transaction Tool backend requires no privileged access.
- Users of organizational multi-sig tools routinely participate in multiple organizations/deployments.
- Constructing a transaction with identical body bytes is straightforward: the attacker only needs to know the payer account ID and timestamp of the target transaction, both of which are visible to any user who can query the legitimate server's transaction list.
- The attack requires no cryptographic break and no leaked secrets.

### Recommendation

Include a domain separator in the signed payload that binds the signature to a specific server deployment and network. For example, sign:

```
HMAC or hash( serverUrl || mirrorNetwork || transactionBodyBytes )
```

or use a structured signed message:

```
sign( "hedera-tx-tool-approval" || serverUrl || mirrorNetwork || transactionBodyBytes )
```

Both `getTransactionBodySignatureWithoutNodeAccountId` (front-end) and `verifyTransactionBodyWithoutNodeAccountIdSignature` (back-end) must be updated consistently. The server URL or a stable deployment-unique identifier should be configured server-side and communicated to the client at session establishment so the client can include it in the signed payload. ### Proof of Concept

**Setup:**
- Legitimate server: `https://org.example.com` — has transaction T1 with body bytes `B`, pending approval from Bob (userKeyId=42).
- Attacker server: `https://evil.example.com` — Eve controls this deployment.

**Steps:**

1. Eve queries the legitimate server's transaction list (Bob is a registered user and can see T1) to obtain the transaction bytes `B`.

2. Eve creates transaction T2 on her server with the same bytes `B`:
   ```
   POST https://evil.example.com/transactions
   { transactionBytes: B, creatorKeyId: <Eve's key>, mirrorNetwork: "testnet", ... }
   ```

3. Eve adds Bob as an approver on T2 on her server.

4. Eve asks Bob to approve T2 on her server. Bob's client calls:
   ```ts
   getTransactionBodySignatureWithoutNodeAccountId(bobPrivateKey, sdkTx)
   // => signature S (over body bytes of B, no server context)
   ```
   Bob's client sends `S` to `https://evil.example.com/transactions/T2/approvers`.

5. Eve intercepts or reads `S` from her server's database.

6. Eve submits Bob's signature to the legitimate server:
   ```
   POST https://org.example.com/transactions/T1/approvers
   Authorization: Bearer <Eve's JWT on the legitimate server>
   { userKeyId: 42, signature: S, approved: true }
   ```
   *(Eve must be a registered user on the legitimate server to hold a valid JWT; she does not need Bob's JWT.)*

7. The legitimate server executes:
   ```ts
   verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTx_T1, S, bobPublicKey)
   // => true, because body bytes of T1 == body bytes of T2 == B
   ```
   Bob's approval is recorded. If the approval threshold is now met, the transaction proceeds to execution.

**Expected outcome:** Bob's approval is forged on the legitimate server. Bob never consented to approving T1 on `org.example.com`.

export const getTransactionBodyBytes = (transaction: SDKTransaction) => {
  // @ts-expect-error - _makeTransactionBody is a private method
  const transactionBody = transaction._makeTransactionBody(null);
  return proto.TransactionBody.encode(transactionBody).finish();

/* Verify the signature of the transaction body without node account id */
export const verifyTransactionBodyWithoutNodeAccountIdSignature = (
  transaction: SDKTransaction,
  signature: string | Buffer,
  publicKey: string | PublicKey,
) => {
  const bodyBytes = getTransactionBodyBytes(transaction);

  /* Deserialize Public Key */
  publicKey = publicKey instanceof PublicKey ? publicKey : PublicKey.fromString(publicKey);

  /* Deserialize Signature */
  signature = typeof signature === 'string' ? decode(signature) : signature;

  try {
    return publicKey.verify(bodyBytes, signature);
  } catch (err) {
    console.log(err);
    return false;
  }
};
```
