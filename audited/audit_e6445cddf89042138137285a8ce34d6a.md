### Title
Non-Verified User Can Be Set as Transaction Approver, Causing Transactions to Become Stuck

### Summary
`createTransactionApprovers` and `updateTransactionApprover` in `approvers.service.ts` validate that a designated approver user *exists* but do not validate that the user is *verified* (`status === UserStatus.NONE`). Because `VerifiedUserGuard` blocks non-verified users from calling `approveTransaction`, a transaction can be permanently stuck in `WAITING_FOR_SIGNATURES` if a non-verified user is set as a required approver — directly analogous to the external report where a product could be whitelisted with a non-whitelisted collateral, causing downstream operations to fail.

---

### Finding Description

**Vulnerability class:** Incomplete prerequisite validation (state transition)

The system enforces a two-tier user status model:

- `UserStatus.NEW` — user exists but has not completed verification/setup
- `UserStatus.NONE` — user is fully verified and can access protected endpoints

`VerifiedUserGuard` enforces this at the controller level: [1](#0-0) 

The `ApproversController` applies this guard globally to all its routes, including `POST /approve`: [2](#0-1) 

However, when `createTransactionApprovers` adds a user as an approver, it only checks that the user *exists* (count > 0), not that the user is verified: [3](#0-2) 

The same incomplete check exists in `updateTransactionApprover` when reassigning an approver to a different user: [4](#0-3) 

When the non-verified approver later attempts to call `POST /transactions/:id/approvers/approve`, `VerifiedUserGuard` returns HTTP 403, permanently blocking them from fulfilling their approver role.

---

### Impact Explanation

A transaction requiring approval from a non-verified user will remain stuck in `WAITING_FOR_SIGNATURES` indefinitely. If the transaction has a `cutoffAt` deadline or the Hedera transaction's `validStart` window expires, the transaction will expire without ever being executed. The creator must manually detect the misconfiguration, remove the broken approver, and re-add a verified one — a non-trivial recovery path that may not be obvious.

---

### Likelihood Explanation

This is realistic in organization mode: an admin invites a new user to the platform, the new user registers but has not yet completed account setup (`UserStatus.NEW`), and a transaction creator immediately adds them as an approver. The creator has no UI-level warning that the designated approver cannot yet act. The e2e test suite confirms `userNew` (status `NEW`) is blocked from all verified-user endpoints: [5](#0-4) 

---

### Recommendation

In `createTransactionApprovers` and `updateTransactionApprover`, extend the user existence check to also assert `status === UserStatus.NONE`:

```typescript
// In approvers.service.ts — createTransactionApprovers
const user = await transactionalEntityManager.findOne(User, {
  where: { id: dtoApprover.userId },
});
if (!user) throw new Error(this.USER_NOT_FOUND(dtoApprover.userId));
if (user.status !== UserStatus.NONE)
  throw new Error(`User with id: ${dtoApprover.userId} is not verified and cannot be an approver`);
```

Apply the same guard in `updateTransactionApprover` at the analogous location.

---

### Proof of Concept

1. Admin (verified) creates a transaction requiring approver sign-off.
2. Admin calls `POST /transactions/:id/approvers` with `{ approversArray: [{ userId: <newUserId> }] }` where `newUserId` belongs to a user with `status = UserStatus.NEW`.
3. The call succeeds — `createTransactionApprovers` only checks `userCount > 0`. [3](#0-2) 

4. The non-verified user attempts `POST /transactions/:id/approvers/approve` — `VerifiedUserGuard` returns HTTP 403. [6](#0-5) 

5. The transaction remains in `WAITING_FOR_SIGNATURES` with no path to execution unless the creator manually intervenes.
