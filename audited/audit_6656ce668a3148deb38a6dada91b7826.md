### Title
Missing Lower-Bound Validation on `threshold` in `CreateTransactionApproverDto` Allows Negative Threshold Storage

### Summary
The `CreateTransactionApproverDto` class applies no lower-bound constraint on the `threshold` field. The service-level guard in `approvers.service.ts` only rejects `threshold === 0` and `threshold > approvers.length`, leaving negative values (e.g., `-1`) completely unchecked. Any authenticated user who creates a transaction can submit a negative threshold that is persisted to the database, corrupting the approval-tree invariant that `1 ≤ threshold ≤ child_count`.

### Finding Description

**Root cause — DTO layer (`CreateTransactionApproverDto`)**

`threshold` carries only `@IsNumber()` and `@IsOptional()`. No `@Min(1)` is present. [1](#0-0) 

Compare with `UpdateTransactionApproverDto`, which does apply `@Min(1)`: [2](#0-1) 

The protection is inconsistently applied: the update path is guarded, the create path is not.

**Root cause — service-layer guard (`approvers.service.ts`)**

The only runtime check for an invalid threshold is: [3](#0-2) 

For `threshold = -1` with two child approvers the expression evaluates to:

```
true && (-1 > 2 || -1 === 0)  →  true && (false || false)  →  false
```

No exception is thrown. The negative value then flows directly into the database write: [4](#0-3) 

**Exploit path**

1. Attacker authenticates as a normal user (no privilege required).
2. Attacker creates a transaction (standard workflow).
3. Attacker calls `POST /transactions/:id/approvers` with body:
   ```json
   {
     "approversArray": [{
       "threshold": -1,
       "approvers": [{"userId": 2}, {"userId": 3}]
     }]
   }
   ```
4. `createTransactionApprovers` calls `getCreatorsTransaction` — passes because the attacker is the creator.
5. `validateApprover` passes (threshold is a number, not null/NaN).
6. The threshold check at line 302–307 passes (see arithmetic above).
7. `threshold: -1` is persisted to `transaction_approver`. [5](#0-4) 

### Impact Explanation

A stored `threshold` of `-1` violates the invariant `1 ≤ threshold ≤ child_count` that the entire approval-tree evaluation relies on. Any downstream logic that computes `approvedCount >= threshold` will treat the condition as permanently satisfied (any integer ≥ −1), allowing the transaction to advance through the approval workflow with zero actual approvals. This constitutes an unauthorized state change: a transaction that should require multi-party approval can be self-approved by its creator alone by manipulating the threshold.

### Likelihood Explanation

The attacker only needs a valid account and the ability to create a transaction — both are standard, unprivileged operations available to every registered user. The payload is a single crafted JSON body. No special timing, race condition, or external dependency is required. Likelihood is high given the low barrier to entry.

### Recommendation

Add `@Min(1)` to `threshold` in `CreateTransactionApproverDto`, mirroring the constraint already present in `UpdateTransactionApproverDto`:

```typescript
// create-transaction-approver.dto.ts
import { IsArray, IsNumber, IsOptional, Min, ValidateNested, ArrayMinSize } from 'class-validator';

export class CreateTransactionApproverDto {
  @IsNumber()
  @IsOptional()
  listId?: number;

  @IsNumber()
  @Min(1)          // ← add this
  @IsOptional()
  threshold?: number;
  ...
}
```

Additionally, strengthen the service-level guard to be explicit:

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold < 1 || dtoApprover.threshold > dtoApprover.approvers.length)
)
  throw new Error(...);
``` [3](#0-2) 

### Proof of Concept

**Request** (authenticated as transaction creator):
```
POST /api/transactions/42/approvers
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "approversArray": [{
    "threshold": -1,
    "approvers": [{"userId": 10}, {"userId": 11}]
  }]
}
```

**Expected (correct) behaviour:** HTTP 400 — threshold must be ≥ 1.

**Actual behaviour:** HTTP 201 — approver tree created with `threshold = -1` stored in `transaction_approver`. Any subsequent approval-count evaluation against this node resolves as satisfied with zero approvals, bypassing the multi-signature requirement for the transaction.
