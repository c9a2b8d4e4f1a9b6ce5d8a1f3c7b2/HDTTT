### Title
Mutable Approver Structure Allows Transaction Creator to Nullify Rejections After Approvals Are Submitted

### Summary
The `updateTransactionApprover()` and `removeTransactionApprover()` functions in `approvers.service.ts` allow the transaction creator to modify the approver tree — including threshold values, approver identities, and tree structure — at any time, with no check on whether approvals have already been submitted. A malicious creator can exploit this to remove rejecting approvers or swap them out (which clears their `approved`/`signature` state), effectively nullifying rejections and bypassing the multi-party governance the approval system is designed to enforce.

### Finding Description

**Root cause**: Neither `updateTransactionApprover()` nor `removeTransactionApprover()` checks the transaction's current status or whether any approvers have already submitted their decisions before allowing structural changes to the approver tree.

`getCreatorsTransaction()` — the only guard called before these mutations — verifies only that the caller is the transaction creator. It performs no status check: [1](#0-0) 

In `updateTransactionApprover()`, the threshold can be lowered at any time: [2](#0-1) 

An approver's `userId` can be replaced at any time, and doing so **explicitly clears** the existing approval state (`signature`, `userKeyId`, `approved` are set to `undefined`): [3](#0-2) 

`removeTransactionApprover()` has no status check at all: [4](#0-3) 

The controller confirms `getCreatorsTransaction` is the only guard before removal: [5](#0-4) 

**Exploit flow**:

1. Creator creates a transaction and sets up approvers: User A, User B, User C with threshold = 2.
2. User A approves (`approved = true`). User B rejects (`approved = false`).
3. Creator calls `PATCH /transactions/:txId/approvers/:approverId` with `{ userId: userD }` to replace User B. This clears User B's rejection — the new record for User D has `signature = null`, `approved = null`.
4. User D approves.
5. Threshold of 2 is now met with User A and User D. The rejection from User B is gone.

Alternatively, the creator calls `DELETE /transactions/:txId/approvers/:approverId` to remove User B entirely, reducing the effective approver pool and potentially meeting the threshold with only the remaining approvals.

### Impact Explanation

The approval system is the primary governance mechanism preventing a transaction creator from unilaterally executing transactions that require organizational consensus. By nullifying rejections through approver swaps or removals, the creator can force any transaction through regardless of how many approvers reject it. This is a **critical integrity failure in the trust model**: the approval workflow provides no meaningful protection against a creator who is willing to cycle out rejecting approvers.

Concrete impact:
- A creator can execute a transaction that was explicitly rejected by designated approvers.
- The rejection history is silently erased (no audit trail of the removed approver's decision).
- Organizational multi-party approval policies are rendered unenforceable.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no special privileges. The attack path requires only standard API calls (`PATCH` or `DELETE` on approver endpoints) that the creator is already authorized to make. No leaked credentials, no admin access, and no race conditions are required. The creator can observe rejections in real time via the notification system and respond immediately.

### Recommendation

Add a transaction status guard to both `updateTransactionApprover()` and `removeTransactionApprover()` (and their controller entry points). Modifications to the approver structure should only be permitted while the transaction is in a state where no approvals have yet been submitted — for example, `WAITING_FOR_SIGNATURES` with zero submitted approver decisions, or a dedicated `NEW`/`DRAFT` state.

```typescript
// In getCreatorsTransaction or at the top of updateTransactionApprover / removeTransactionApprover:
const hasSubmittedApprovals = await entityManager.count(TransactionApprover, {
  where: { transactionId, signature: Not(IsNull()) },
});
if (hasSubmittedApprovals > 0) {
  throw new BadRequestException('Cannot modify approvers after approvals have been submitted');
}
```

Alternatively, freeze the approver structure once the first approval or rejection is recorded, mirroring the pattern used in `approveTransaction()` which already correctly guards against acting on non-actionable statuses. [6](#0-5) 

### Proof of Concept

**Preconditions**: Two registered users (creator, approverB). Creator has an active transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status with approvers [A, B, C], threshold = 2.

**Steps**:
1. User B calls `POST /transactions/:txId/approvers/approve` with `{ approved: false, ... }`. User B's record now has `approved = false`.
2. Creator calls `PATCH /transactions/:txId/approvers/:approverB_id` with body `{ userId: userD_id }`.
3. **Expected (secure)**: Request rejected — approvals already submitted.
4. **Actual**: Request succeeds. User B's `approved`, `signature`, `userKeyId` are cleared. User D is now the approver with a blank record.
5. User D calls `POST /transactions/:txId/approvers/approve` with `{ approved: true, ... }`.
6. Threshold of 2 is met (User A + User D). Transaction proceeds despite User B's explicit rejection. [7](#0-6)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
