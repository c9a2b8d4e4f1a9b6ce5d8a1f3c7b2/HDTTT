### Title
Any Verified User Can Permanently Block Another User's Public Key Registration via Pre-Registration

### Summary
The `uploadKey` endpoint in `UserKeysService` enforces public key uniqueness at the application level only, with no database-level `UNIQUE` constraint on `publicKey`. Any verified user can register any Hedera public key — including keys belonging to other users — before the legitimate owner does. Once registered under the attacker's account, the legitimate owner is permanently blocked from registering that key, locking them out of all organization transaction workflows.

### Finding Description

**Root cause:** `uploadKey` in `back-end/apps/api/src/user-keys/user-keys.service.ts` performs a `findOne` lookup on `publicKey` (with `withDeleted: true`) and rejects the request if the found record belongs to a different user. There is no proof-of-ownership check — any authenticated user can submit any syntactically valid public key string. The `user_key` table has only a non-unique index on `publicKey`, not a `UNIQUE` constraint, so the enforcement is entirely application-level. [1](#0-0) 

The DTO accepts any valid Hedera public key with no proof of private-key ownership: [2](#0-1) 

The database schema confirms there is no `UNIQUE` constraint on `publicKey` — only a plain index: [3](#0-2) 

The entity definition mirrors this — `@Index()` without `{ unique: true }`: [4](#0-3) 

**Exploit flow:**

1. Attacker registers as a verified user on the organization server.
2. Attacker obtains the victim's public key from any public source (Hedera mirror node, the organization's `GET /user/:id/keys` endpoint, or any signed transaction visible on-chain).
3. Attacker calls `POST /user/keys` with `{ "publicKey": "<victim_public_key>" }`.
4. The key is now stored in `user_key` with `userId = attacker_id`.
5. When the victim calls `POST /user/keys` with their own key, `findOne` returns the attacker's record. The check `userKey.userId !== user.id` is true → `throw new BadRequestException(ErrorCodes.PU)`.
6. The victim is permanently blocked.

**Why the block is permanent:** The `findOne` query uses `withDeleted: true`, so even if the attacker soft-deletes their `user_key` record, the deleted row is still returned and the ownership check still fails for the victim. [5](#0-4) 

### Impact Explanation

Without a registered `user_key`, a victim user cannot:
- **Create transactions** — `createTransaction` requires a `creatorKeyId` that must be a registered key belonging to the user.
- **Be added as a signer or approver** — signers and approvers are identified by their registered `user_key` records.
- **Participate in any organization transaction workflow.**

This is a permanent, unrecoverable denial-of-service against a targeted user's ability to use the organization mode of the tool. The attacker does not need to use the key for anything — the sole purpose is to occupy the slot. The victim has no recourse because the system provides no mechanism to reclaim a key registered by another user. [6](#0-5) 

### Likelihood Explanation

- **Attacker preconditions:** Only a verified organization account is required — no admin or privileged role.
- **Key discovery:** Hedera public keys are public by design. Any key used to sign a Hedera transaction is visible on the mirror node. An attacker can also enumerate keys already registered in the organization via `GET /user/:id/keys`.
- **Effort:** A single HTTP POST request per targeted key. An attacker can pre-register keys for all known organization members in bulk.
- **No timing constraint:** The attack can be performed at any time before the victim registers their key, including before the victim even joins the organization. [7](#0-6) 

### Recommendation

Require proof of private-key ownership at registration time. The standard approach is a **challenge-response**: the server issues a random nonce, and the client must sign it with the private key corresponding to the public key being registered. Only if the signature verifies does the server accept the registration.

Alternatively, add a database-level `UNIQUE` constraint on `publicKey` in `user_key` and change the ownership-conflict logic so that a soft-deleted key owned by a different user can be reclaimed by the legitimate owner (verified via challenge-response).

### Proof of Concept

```
# Step 1: Attacker registers (normal signup flow, gets JWT)
POST /auth/login  →  attacker_jwt

# Step 2: Attacker learns victim's public key (e.g., from mirror node or GET /user/2/keys)
victim_pubkey = "302a300506032b6570032100<victim_hex>"

# Step 3: Attacker pre-registers victim's key
POST /user/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "publicKey": "302a300506032b6570032100<victim_hex>" }

→ 201 Created  (key now owned by attacker)

# Step 4: Victim attempts to register their own key
POST /user/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json

{ "publicKey": "302a300506032b6570032100<victim_hex>" }

→ 400 Bad Request  { "code": "PU" }   ← permanent block

# Victim can never register their key. They cannot create or sign transactions.
``` [8](#0-7)

  async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
    await attachKeys(user, this.repo.manager);

    // Check if the user already has the maximum number of keys
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }

    // Find the userKey by the publicKey
    let userKey = await this.repo.findOne({
      where: { publicKey: dto.publicKey },
      withDeleted: true,
    });

    if (userKey) {
      // If the userKey found is owned by a different user,
      // or if the userKey has a non null hash or index that doesn't
      // match the hash or index provided
      // throw an error.
      if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
        throw new BadRequestException(ErrorCodes.PU);
      }
      // Set the hash and/or index (only if the current value is null)
      Object.assign(userKey, dto);
    } else {
      userKey = await this.repo.create(dto);
      userKey.user = user;
    }

    if (userKey.deletedAt) {
      await this.repo.recover(userKey);
    }
    return this.repo.save(userKey);
