### Title
Unfiltered `GET /user-keys` Endpoint Exposes All Users' Key Metadata Including `mnemonicHash` and `index` to Any Verified Organization Member

### Summary
The `GET /user-keys` endpoint served by `UserKeysAllController` calls `getUserKeys()` which performs an unfiltered database query returning every `UserKey` record across all users, including the sensitive `mnemonicHash` and `index` fields. This directly contradicts the ownership-scoped `getUserKeysRestricted()` method used by the sibling `UserKeysController`, which deliberately hides `mnemonicHash` and `index` from non-owners. Any verified organization member can silently enumerate all other users' public keys and their mnemonic correlation metadata with a single authenticated HTTP request.

### Finding Description

**Root cause — missing caller-identity filter in `getUserKeys()`:**

`UserKeysService.getUserKeys()` performs a bare `findAndCount` with no `userId` constraint: [1](#0-0) 

The query is `{ take: limit, skip: offset }` — no `where` clause, no caller identity, no field restriction. It returns every row in the `user_key` table.

**Contrast with the restricted path:**

`getUserKeysRestricted()` — used by `UserKeysController` at `GET /user/:userId/keys` — explicitly scopes by `userId` and suppresses `mnemonicHash`/`index` for non-owners: [2](#0-1) 

**The unprotected controller:**

`UserKeysAllController` at `GET /user-keys` passes no user identity to the service: [3](#0-2) 

Guards applied are only `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — the last of which only checks `user.status === UserStatus.NONE`: [4](#0-3) 

No ownership check, no admin check, no scoping to the caller's own keys.

**What the response DTO exposes:**

`UserKeyDto` serializes `id`, `userId`, `mnemonicHash`, `index`, `publicKey`, and `deletedAt` — all fields, for all users: [5](#0-4) 

The `mnemonicHash` field is the hash of a user's 24-word BIP-39 recovery phrase, and `index` is the HD derivation index: [6](#0-5) 

**Exploit path:**

```
Attacker (verified org member) →
  GET /user-keys?page=1&size=100
  Authorization: Bearer <valid_jwt>

Response: all UserKey rows for all users, including:
  { userId, publicKey, mnemonicHash, index, ... }
```

The front-end client already uses this endpoint to fetch all keys for transaction signing coordination: [7](#0-6) 

### Impact Explanation

1. **Cross-user key enumeration**: Any verified member can map every public key to its owning `userId`, building a complete picture of the organization's key infrastructure.
2. **`mnemonicHash` leakage**: The hash of each user's mnemonic phrase is exposed. This reveals which keys share the same seed wallet, enabling correlation of a user's key portfolio. It also provides a preimage target for offline dictionary/brute-force attacks against the mnemonic.
3. **`index` leakage**: The HD derivation index is exposed, revealing the key derivation structure of other users' wallets.

The `getUserKeysRestricted` method was explicitly designed to prevent exactly this — hiding `mnemonicHash` and `index` from non-owners — but `UserKeysAllController` bypasses that protection entirely.

### Likelihood Explanation

- **Attacker precondition**: Valid JWT for any verified organization member (`status = NONE`). No admin role required.
- **Exploitation**: Single HTTP GET request. No special tooling.
- **Discoverability**: The endpoint is registered in the NestJS module, documented via Swagger (`@ApiTags('User Keys All')`), and actively called by the front-end client. Any member who inspects network traffic or the API docs will find it.

### Recommendation

`getUserKeys()` must be scoped to the calling user's identity. Pass the authenticated user into the service method and add a `where: { userId: caller.id }` constraint, mirroring `getUserKeysRestricted()`. Additionally, apply the same field-level restriction (`mnemonicHash: false`, `index: false`) that `getUserKeysRestricted()` enforces for non-owners, or remove `mnemonicHash` and `index` from the `UserKeyDto` used by this endpoint entirely.

```typescript
// UserKeysAllController — pass caller identity
getUserKeys(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
) {
  return this.userKeysService.getUserKeys(user, paginationParams);
}

// UserKeysService.getUserKeys — scope to caller
async getUserKeys(caller: User, { limit, offset, page, size }: Pagination) {
  const [items, total] = await this.repo.findAndCount({
    where: { userId: caller.id },
    select: { id: true, userId: true, publicKey: true },
    take: limit,
    skip: offset,
  });
  return { totalItems: total, items, page, size };
}
```

### Proof of Concept

**Precondition**: Attacker holds a valid JWT for any verified organization member (e.g., obtained via normal login).

**Request**:
```http
GET /user-keys?page=1&size=100 HTTP/1.1
Host: <org-server>
Authorization: Bearer <attacker_jwt>
```

**Expected response** (HTTP 200):
```json
{
  "totalItems": N,
  "items": [
    {
      "id": 1,
      "userId": 1,
      "publicKey": "e0c8ec2758a5879ff...",
      "mnemonicHash": "7a40e67733edaec4...",
      "index": 0,
      "deletedAt": null
    },
    {
      "id": 2,
      "userId": 3,
      "publicKey": "5a7245ff4fbbc301...",
      "mnemonicHash": "9b21f78844bcde12...",
      "index": 1,
      "deletedAt": null
    }
    ...
  ]
}
```

The attacker receives `mnemonicHash` and `index` for every user in the organization — data that `GET /user/:userId/keys` explicitly withholds from non-owners. The e2e test at line 46–56 confirms that `GET /user/:adminId/keys` as a non-owner correctly strips `mnemonicHash` and `index`, proving the design intent is ownership-scoped access — which `GET /user-keys` violates.
