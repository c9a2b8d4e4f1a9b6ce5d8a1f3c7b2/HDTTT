### Title
Empty `groupItems` Array Bypasses Validation in `POST /transaction-groups`, Allowing Creation of Empty Transaction Groups

### Summary
`CreateTransactionGroupDto` decorates `groupItems` with `@IsNotEmpty()`, which in class-validator does **not** reject an empty array `[]` — it only rejects `null`, `undefined`, and `""`. Any authenticated organization user can POST directly to `POST /transaction-groups` with `groupItems: []`, bypassing the frontend guard and persisting an empty transaction group to the database. The service layer performs no additional length check.

### Finding Description

**Root cause — wrong decorator:**

`@IsNotEmpty()` from class-validator calls `isEmpty(value)`, which returns `true` only for `''`, `null`, and `undefined`. An empty array `[]` is **not** empty by this definition, so it passes the check silently. [1](#0-0) 

The correct decorator for "array must contain at least one element" is `@ArrayMinSize(1)`, which is already used correctly elsewhere in the same codebase: [2](#0-1) [3](#0-2) 

**Service performs no secondary check:**

`createTransactionGroup` maps directly over `dto.groupItems` with no length guard. With an empty array, `transactionDtos` is `[]`, `createTransactions` is called with an empty list, and the group is saved to the database with zero items. [4](#0-3) 

**Frontend check is client-side only:**

The UI enforces a non-empty check before calling the API, but this is trivially bypassed by sending a raw HTTP request. [5](#0-4) 

**Test spec confirms the gap:**

The DTO spec for `CreateTransactionGroupDto` tests the empty-array mapping case but deliberately omits a `validateSync()` call on it — unlike the approver DTO spec, which explicitly asserts that an empty array fails validation. [6](#0-5) [7](#0-6)

### Impact Explanation
An authenticated organization user can flood the organization's transaction group list with empty groups. This corrupts the shared state visible to all organization members, degrades the integrity of the group management workflow, and may cause downstream confusion or errors in any code that assumes a group always has at least one item (e.g., `getTransactionGroup` throws `UnauthorizedException` when `groupItems.length === 0`, making the empty group permanently inaccessible after creation). [9](#0-8) 

Severity: **Medium** — data integrity corruption and persistent inaccessible records, no direct asset theft.

### Likelihood Explanation
Any verified, authenticated organization user can exploit this with a single crafted HTTP POST. No special privileges, no leaked credentials, and no complex setup are required. The attacker only needs a valid JWT session, which is the baseline for any organization member.

### Recommendation
Replace `@IsNotEmpty()` with `@ArrayMinSize(1)` on the `groupItems` field in `CreateTransactionGroupDto`, consistent with how `CreateTransactionApproverDto` and `CreateTransactionObserversDto` already enforce non-empty arrays:

```typescript
// back-end/apps/api/src/transactions/dto/create-transaction-group.dto.ts
import {
  ArrayMinSize,   // add this
  IsArray,
  IsBoolean,
  IsOptional,
  IsString,
  ValidateNested,
} from 'class-validator';

@IsArray()
@ArrayMinSize(1)  // replace @IsNotEmpty()
@ValidateNested({ each: true })
@Type(() => CreateTransactionGroupItemDto)
groupItems: CreateTransactionGroupItemDto[];
```

Also add a corresponding `validateSync` assertion to the DTO spec for the empty-array case.

### Proof of Concept

```bash
# Attacker is a legitimate authenticated organization user
curl -X POST https://<server>/transaction-groups \
  -H "Authorization: Bearer <valid_jwt>" \
  -H "Content-Type: application/json" \
  -d '{
    "description": "empty group",
    "atomic": false,
    "sequential": false,
    "groupItems": []
  }'

# Expected (current): HTTP 201, empty TransactionGroup persisted to DB
# Expected (fixed):   HTTP 400, validation error: groupItems must contain at least 1 elements
```

The created group is then permanently inaccessible via `GET /transaction-groups/:id` (throws `UnauthorizedException` due to the empty-items check in `getTransactionGroup`), leaving orphaned records in the database.

  async createTransactionGroup(
    user: User,
    dto: CreateTransactionGroupDto,
  ): Promise<TransactionGroup> {
    const group = this.dataSource.manager.create(TransactionGroup, dto);

    // Extract all transaction DTOs
    const transactionDtos = dto.groupItems.map(item => item.transaction);

    // Batch create all transactions
    const transactions = await this.transactionsService.createTransactions(
      transactionDtos,
      user,
    );

    await this.dataSource.transaction(async manager => {
      // Create group items with corresponding transactions
      const groupItems = transactions.map((transaction, index) => {
        const groupItemDto = dto.groupItems[index];
        const groupItem = manager.create(TransactionGroupItem, groupItemDto);
        groupItem.transaction = transaction;
        groupItem.group = group;
        return groupItem;
      });

      // Save everything
      await manager.save(TransactionGroup, group);
      await manager.save(TransactionGroupItem, groupItems);

      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        transactions.map(tx => ({
          entityId: tx.id,
        })),
      );
    });

    return group;
  }
