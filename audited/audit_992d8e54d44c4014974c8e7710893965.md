### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in the API back-end fetches every non-terminal transaction from the database in a single query with no database-level `LIMIT`, then performs expensive per-transaction async work (transaction deserialization + external account-cache lookups) in a sequential loop. Any authenticated, verified user can trigger this endpoint repeatedly, causing memory exhaustion, CPU saturation, and database connection starvation proportional to the total number of active transactions in the system.

### Finding Description

**Root cause — unbounded `repo.find()` with no `take`:**

In `getTransactionsToSign`, the database query at lines 295–299 omits the `take` (LIMIT) clause entirely:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

Compare this to every other query in the same file — `getTransactions`, `getHistoryTransactions`, and `getTransactionsToApprove` all pass `skip: offset, take: limit` to the database. [2](#0-1) 

The `whereForUser` filter only excludes five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). Every `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `NEW`, and `SIGN_ONLY` transaction in the entire organization is loaded into the Node.js process heap. [3](#0-2) 

**Per-transaction async work in a sequential loop:**

After loading all N transactions, the code iterates over every one and calls `userKeysToSign`:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [4](#0-3) 

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` performs, for each transaction:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-bound protobuf deserialization
2. `transactionSignatureService.computeSignatureKey(transaction)` — which calls `accountCacheService.getAccountInfoForTransaction` for the fee-payer, every signing account, and every receiver account (potentially multiple external/cache lookups per transaction) [5](#0-4) [6](#0-5) 

**Pagination is applied only in-memory after all work is done:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [7](#0-6) 

The `PaginationParams` decorator enforces `size ≤ 100`, but this only controls the final slice — it does not limit the database fetch or the per-transaction processing loop. [8](#0-7) 

**Attacker entry path:**

The endpoint is `GET /transactions/sign`, protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — standard authenticated-user guards. [9](#0-8) 

The global `IpThrottlerGuard` applies a per-IP rate limit, but:
- The limit values are configurable environment variables (`GLOBAL_MINUTE_LIMIT`, `GLOBAL_SECOND_LIMIT`) with no hardcoded floor. [10](#0-9) 
- A single request with a large N is sufficient to cause significant resource consumption; the attacker does not need to flood the endpoint.
- The `UserThrottlerModule` and `UserThrottlerGuard` exist but are **not** applied to the transactions controller. [11](#0-10) 

### Impact Explanation

A single authenticated user sending one request to `GET /transactions/sign?page=1&size=1` causes the server to:
- Execute a full table scan of all active transactions (unbounded PostgreSQL result set loaded into Node.js heap)
- Perform O(N) protobuf deserializations and O(N × accounts_per_tx) external account-cache lookups sequentially

With thousands of active transactions (realistic in an enterprise multi-sig deployment), this exhausts Node.js heap memory, saturates the event loop, and starves the PostgreSQL connection pool — making the API unavailable to all users. The attacker needs only a valid account, which any organization member possesses.

### Likelihood Explanation

The endpoint is reachable by any verified organization member with no special privileges. The comment `/* NO LONGER USED BY FRONT-END */` in the controller confirms the endpoint is still live and registered. A malicious insider or a compromised member account can trigger this with a single HTTP request. The cost to the attacker is negligible; the cost to the server scales with the organization's transaction history.

### Recommendation

Apply the database-level `take`/`skip` before the loop, and push the signing-eligibility filter into SQL rather than doing it in application memory:

1. **Add `take: limit, skip: offset` to the `repo.find()` call** as a first approximation, accepting that `totalItems` will be approximate (or use a two-query count + paginated fetch pattern).
2. **Rewrite the query to filter signing eligibility in SQL** (similar to how `getTransactions` uses a recursive CTE to check approver membership at the database level), so only transactions the user actually needs to sign are returned — eliminating the in-memory loop entirely.
3. **Apply `UserThrottlerGuard` to the transactions controller** to bound per-user request rates independently of IP.

### Proof of Concept

**Preconditions:** Attacker has a valid JWT for any verified organization member account. The organization has N active (non-terminal) transactions.

**Steps:**
1. Authenticate and obtain a JWT.
2. Send: `GET /transactions/sign?page=1&size=1` with `Authorization: Bearer <jwt>`
3. The server executes `SELECT * FROM transaction WHERE status NOT IN (...)` with no LIMIT, loading all N rows.
4. For each of the N rows, the server deserializes transaction bytes and performs account-cache lookups.
5. With N in the thousands, Node.js heap grows proportionally; response latency spikes; concurrent requests from legitimate users time out.
6. Repeat at the IP throttle boundary to sustain the degradation.

**Expected outcome:** API response time degrades linearly with N; at sufficient N the process OOMs or the PostgreSQL connection pool is exhausted, causing 500 errors for all users.

    transaction: Transaction,

    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
