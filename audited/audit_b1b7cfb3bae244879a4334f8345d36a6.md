### Title
Missing Cryptographic Signature Validation in `uploadSignatureMaps` Allows Forged Signer Registration and Transaction Corruption

### Summary
The `processTransactionSignatures` function in `signers.service.ts` accepts and persists signature maps without verifying that the submitted signature bytes are cryptographically valid for the claimed public key. Any authenticated user with a registered key can submit a properly-formatted but cryptographically invalid signature, be recorded as a legitimate signer, and corrupt the stored transaction bytes — potentially causing the transaction to fail on the Hedera network or advance to execution prematurely.

### Finding Description

There are two distinct signature upload paths in the codebase:

**Path 1 — `importSignatures` in `transactions.service.ts` (safe):** [1](#0-0) 

This path calls `validateSignature(sdkTransaction, map)` before calling `addSignature`, ensuring cryptographic correctness.

**Path 2 — `uploadSignatureMaps` → `processTransactionSignatures` in `signers.service.ts` (vulnerable):** [2](#0-1) 

This path only checks whether the public key in the submitted map belongs to the authenticated user (`userKeyMap.get(raw)`), then immediately calls `sdkTransaction.addSignature(publicKey, map)` with no cryptographic validation of the signature bytes. The Hedera SDK's `addSignature` is a byte-insertion method — it does not verify that the signature is valid over the transaction body.

The `isSameBytes` check at line 262–264 does not protect against this: if the attacker submits invalid-but-different bytes, `isSameBytes` will be `false`, causing the corrupted bytes to be persisted. [3](#0-2)

### Impact Explanation

1. **False signer registration**: The attacker is inserted into the `TransactionSigner` table as a legitimate signer despite never producing a valid cryptographic signature. This breaks the integrity of the multi-signature approval workflow.
2. **Transaction byte corruption**: The stored `transactionBytes` are overwritten with bytes containing an invalid signature, permanently corrupting the transaction record.
3. **Execution failure / DoS on a transaction**: If the corrupted transaction advances to `WAITING_FOR_EXECUTION` (because the required signer count is met), the chain service will submit it to Hedera, which will reject it due to the invalid signature. The transaction cannot be recovered. [5](#0-4) 

### Likelihood Explanation

The attacker must be an authenticated user with at least one registered public key in the system. This is a normal, non-privileged user role. The attack requires no special access beyond a valid JWT and a registered key. The attacker constructs a `SignatureMap` containing their registered public key paired with arbitrary (invalid) signature bytes of the correct length and encoding to pass DTO validation. This is a straightforward, low-effort manipulation of a standard API call.

### Recommendation

Add cryptographic signature validation in `processTransactionSignatures` before calling `addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before sdkTransaction.addSignature(publicKey, map):
const isValid = publicKey.verify(transactionBodyBytes, signatureBytes);
if (!isValid) throw new Error(ErrorCodes.ISNMP);
```

Alternatively, extract and reuse the existing `validateSignature` utility from `back-end/libs/common/src/utils/sdk/transaction.ts` to ensure both upload paths apply the same validation logic. [6](#0-5) 

### Proof of Concept

1. Register as a normal user and add a public key (e.g., ECDSA key `K`) to the account.
2. Obtain a transaction in `WAITING_FOR_SIGNATURES` status that requires a signature from `K`.
3. Construct a `SignatureMap` with the correct structure (node account ID → transaction ID → public key `K`) but with 64 bytes of zeroes (or any arbitrary bytes) as the signature value — valid encoding, invalid cryptography.
4. POST to `POST /transactions/:transactionId/signers` with this map.
5. **Expected (correct) behavior**: 400 error — invalid signature rejected.
6. **Actual behavior**: 201 success — the user is recorded as a signer in `TransactionSigner`, and the transaction bytes are updated with the invalid signature. If this was the last required signature, the transaction advances to `WAITING_FOR_EXECUTION` and will be rejected by the Hedera network upon submission.
