### Title
Missing Array Bounds Check in CSV Row Parser Causes Silent Partial Import and Uninformative Failure

### Summary
The `handleImportCsv()` function in `ImportCSVController.vue` parses CSV transaction rows and accesses `rowInfo[1]` (the transfer amount field) without first verifying that the parsed row contains at least two columns. When a transaction data row is malformed (fewer than 2 columns), `rowInfo[1]` is `undefined`, and calling `.replace()` on it throws a `TypeError`. This exception is caught by a generic outer handler that shows only "Failed to import CSV file" — with no indication of which row failed or that the transaction group is now in a partially-populated state. Unlike the receiver-account-lookup failure path, the outer catch does **not** call `transactionGroup.clearGroup()`, leaving stale partial transactions in the group.

### Finding Description

**Root cause — missing bounds check before indexed access:**

In `front-end/src/renderer/pages/CreateTransactionGroup/ImportCSVController.vue`, the `handleImportCsv()` function splits each CSV row with a regex and stores the result in `rowInfo`: [1](#0-0) 

For transaction data rows (the `default` branch of the `switch`), the code immediately accesses `rowInfo[1]` to read the transfer amount and calls `.replace()` on it: [2](#0-1) 

There is **no guard** checking `rowInfo.length >= 2` before this access. If a data row has fewer than 2 comma-separated fields, `rowInfo[1]` is `undefined`, and `undefined.replace(...)` throws:

```
TypeError: Cannot read properties of undefined (reading 'replace')
```

**Failure path — generic catch, no group cleanup:**

The TypeError propagates to the outer `try/catch`: [3](#0-2) 

This shows only `"Failed to import CSV file"` — no row number, no field name, no indication that the group is incomplete. Critically, `transactionGroup.clearGroup()` is **not** called here (it is only called in the specific receiver-account-lookup failure path at line 131): [4](#0-3) 

The `finally` block then fires `props.callback()`, which may trigger downstream processing on the now-partial group: [5](#0-4) 

**Secondary instances — header rows also unguarded:**

The same pattern applies to every header-row branch. For example, `rowInfo[1]` is read for `'sender account'`, `'fee payer account'`, `'sending time'`, `'transaction fee'`, `'transaction valid duration'`, and `'memo'` without a length check: [6](#0-5) 

For header rows the consequence is that `senderAccount`, `feePayer`, etc. are set to `undefined`, which then propagates silently into transaction construction.

**Contrast with the one guarded access:**

The only bounds-checked access in the entire function is the optional memo field at index 3: [7](#0-6) 

This confirms the developer was aware of the pattern for index 3 but did not apply it to the mandatory fields at indices 1 and 2.

### Impact Explanation

A user who imports a CSV file where any transaction data row is missing the amount column (column index 1) will:

1. Receive only the generic toast `"Failed to import CSV file"` with no actionable detail.
2. Have a transaction group that is **silently partially populated** — all rows processed before the bad row are already added via `transactionGroup.addGroupItem()`.
3. If the user does not notice the partial state and proceeds to submit the group, only a subset of the intended HBAR transfers will be executed on the Hedera network, causing unintended financial outcomes (e.g., some recipients receive funds while others do not, or the sender's balance is partially debited without completing the full intended distribution).

### Likelihood Explanation

Any user who imports a CSV file with a missing amount column — whether due to a formatting mistake, a spreadsheet export artifact, or a crafted file — will trigger this. The CSV import feature is a documented, normal workflow. The generic error message provides no guidance to diagnose or recover from the partial state, making it likely that a user will not realize the group is incomplete before submitting.

### Recommendation

Add an explicit minimum-length check before accessing mandatory indexed fields in the transaction data row branch, mirroring the pattern already used for the optional memo field:

```typescript
default: {
  if (row === '') continue;

  // Require at least: receiverAccount (0), amount (1), startDate (2)
  if (rowInfo.length < 3) {
    toastManager.error(`Malformed transaction row (expected at least 3 columns): "${row}"`);
    transactionGroup.clearGroup();
    return null;
  }

  const startDate = rowInfo[2];
  // ...
  const transferAmount = rowInfo[1].replace(/,/g, '');
  // ...
}
```

Additionally, the outer `catch` block should call `transactionGroup.clearGroup()` to prevent partial state from persisting when any unhandled error occurs during import.

### Proof of Concept

Craft a CSV file with the following content (the third data row is missing the amount column):

```
Sender Account,0.0.1001,,
Sending Time,14:35,,
Node IDs,,,
AccountID,Amount,Start Date,memo
0.0.1002,100,9/4/24,payment 1
0.0.1003,200,9/4/24,payment 2
0.0.1004,,9/4/24,payment 3
```

**Steps:**
1. Open the Hedera Transaction Tool desktop application.
2. Navigate to Create Transaction Group → Import CSV.
3. Select the crafted file above.
4. Observe: the first two transaction rows (`0.0.1002`, `0.0.1003`) are silently added to the group; on the third row `rowInfo[1]` is `undefined`; `undefined.replace(...)` throws a `TypeError`; the outer catch displays `"Failed to import CSV file"`; the group retains the two partial entries.
5. The user can proceed to submit the group, executing only 2 of the 3 intended transfers with no clear indication that the third was dropped. [8](#0-7)
