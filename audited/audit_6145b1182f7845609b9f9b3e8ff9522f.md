### Title
`isManual` Boolean Bypasses Status Guard in `archiveTransaction`, Enabling Invalid State Transitions from Terminal States

### Summary
The `Transaction` entity uses two separate state variables — a `TransactionStatus` enum and an `isManual` boolean — to govern lifecycle behavior. In `archiveTransaction`, the guard condition is logically inverted: when `isManual=true`, the status check is entirely bypassed, allowing a transaction creator to archive their own transaction even when it is already in a terminal state (`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`). This is the direct analog to the `saleOpen`/`presaleOpen` pattern: two independent state variables whose interaction is not correctly enforced, producing reachable invalid state transitions.

### Finding Description

**Root cause — logical error in the archive guard:** [1](#0-0) 

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    ) &&
    !transaction.isManual          // ← BUG: OR-logic, not AND-logic
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }
  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

The guard throws only when **both** conditions are true:
- status is NOT in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- AND `isManual` is `false`

By De Morgan's law, the guard **passes** when:
- status IS in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]` (regardless of `isManual`), **OR**
- `isManual` is `true` (regardless of status)

The second branch is the bug. Any manual transaction — regardless of whether it is `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, or already `ARCHIVED` — passes the guard and gets its status overwritten to `ARCHIVED`.

**The two-variable state model:** [2](#0-1) [3](#0-2) 

`TransactionStatus` (enum) and `isManual` (boolean) together define the effective lifecycle phase of a transaction. The front-end correctly enforces the conjunction: [4](#0-3) 

```typescript
const canArchive = computed(() => {
  const isManual = props.organizationTransaction?.isManual;
  return isManual && isCreator.value && transactionIsInProgress.value; // AND
});
```

But the backend enforces only the disjunction (`||`), not the conjunction (`&&`). The front-end guard is client-side only and trivially bypassed by a direct API call.

**Access control does not block terminal-status transactions:** [5](#0-4) 

`verifyAccess` returns `true` unconditionally for `EXECUTED`, `EXPIRED`, `FAILED`, `CANCELED`, and `ARCHIVED` statuses, so `getTransactionForCreator` does not block the call.

**Downstream effect — terminal transactions become "inactive":** [6](#0-5) 

```typescript
const inactiveStatuses = [TransactionStatus.CANCELED, TransactionStatus.REJECTED, TransactionStatus.ARCHIVED];
const transaction =
  transactions.find(t => !inactiveStatuses.includes(t.status)) ??
  transactions[0];
```

Once an `EXECUTED` transaction is re-labeled `ARCHIVED`, it is treated as inactive and deprioritized in all subsequent lookups.

### Impact Explanation

A transaction creator can:
1. Submit a direct `PATCH /transactions/:id/archive` (or equivalent) API call after their manual transaction has been `EXECUTED` or `FAILED`.
2. The backend overwrites `status = ARCHIVED`, permanently corrupting the execution record.
3. The transaction is now treated as "inactive" in all queries, effectively hiding the execution or failure from other organization members and audit views.
4. A `FAILED` transaction can be silently archived, concealing failure details from approvers and observers.

Impact category: **unauthorized state mutation / audit-trail corruption** for the creator's own transactions within an organization.

### Likelihood Explanation

- Attacker precondition: authenticated user who is the creator of any manual transaction that has reached a terminal state. No privileged role required.
- Trigger: single direct API call bypassing the front-end guard.
- The `isManual` flag is set at transaction creation time and is a normal, user-controlled field.
- Likelihood is **medium**: requires a malicious or curious creator, but the path is trivially reachable.

### Recommendation

Replace the disjunctive guard with a conjunctive one that enforces **both** conditions simultaneously:

```typescript
// Correct: archive is only valid for manual transactions in an in-progress status
if (
  !transaction.isManual ||
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

This mirrors the front-end `canArchive` logic and eliminates the bypass path for terminal-state manual transactions.

### Proof of Concept

1. Authenticate as a normal user (no admin role required).
2. Create a manual transaction (`isManual: true`) and let it reach `EXECUTED` status through the normal signing/execution flow.
3. Issue a direct API call: `PATCH /transactions/<id>/archive` (bypassing the front-end).
4. The backend `archiveTransaction` evaluates: `status=EXECUTED` (not in `[WFS, WFE]`) **AND** `isManual=false` → condition is `true AND false = false` → guard does **not** throw.
5. `repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes.
6. Subsequent calls to `getTransactionById` for the same `transactionId` now deprioritize this record as "inactive", hiding the execution record from organization members. [7](#0-6)
