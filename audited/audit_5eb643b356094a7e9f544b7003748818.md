### Title
Self-Referential `listId` in `updateTransactionApprover` Bypasses Cycle Guard, Enabling Approver Tree Corruption and API DoS

### Summary
In `approvers.service.ts`, `updateTransactionApprover` accepts a `dto.listId` value equal to the approver's own `id`. The existing cycle guard only checks whether `dto.listId` is a *descendant* of the approver — it never checks whether `dto.listId === approver.id`. A transaction creator can therefore set a tree-type approver's parent to itself, creating a self-referential cycle in the `TransactionApprover` table. Any subsequent call that traverses the approver tree via `getRootNodeFromNode` on that node will loop infinitely, hanging the API worker and causing a denial-of-service for all users of the platform.

### Finding Description

**Root cause — missing self-reference guard**

`updateTransactionApprover` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` handles the `dto.listId` (numeric) branch as follows: [1](#0-0) 

The three guards that exist are:

1. **Parent must exist and have a threshold** (line 441–442) — passes when `dto.listId === approver.id` and the approver is a tree node (has `threshold` set).
2. **Parent must not be a descendant of the approver** (line 445–450) — `getTransactionApproversById(approver.id)` returns the approver's *children*, not the approver itself, so `approverList.some(a => a.id === dto.listId)` evaluates to `false` when `dto.listId === approver.id`.
3. **Parent's root transaction must match** (line 453–455) — `getRootNodeFromNode(dto.listId)` where `dto.listId === approver.id` returns the same root, so this check also passes.

No guard checks `dto.listId !== approver.id`.

**Resulting state after the update** (line 458–461):

```
approver.listId      = approver.id   // self-reference
approver.transactionId = null        // removed from root list
```

The `TransactionApprover` entity has no database-level constraint preventing `listId = id`: [2](#0-1) 

**Infinite loop trigger**

Every subsequent operation on this transaction that calls `getRootNodeFromNode(approver.id)` will follow `listId → approver.id → listId → approver.id → …` indefinitely, blocking the Node.js event loop and hanging the API process. [3](#0-2) 

**Attacker-controlled entry point**

The endpoint is: [4](#0-3) 

The only authorization check is that the caller is the *creator* of the transaction — a normal authenticated organization member role, not an admin. [5](#0-4) 

### Impact Explanation

1. **Approver tree permanently corrupted** — the affected transaction can never be approved or have its approver tree traversed again without triggering the infinite loop.
2. **API server DoS** — `getRootNodeFromNode` is called inside a database transaction on every `updateTransactionApprover`, `createTransactionApprovers`, and `approveTransaction` call. An infinite loop inside a DB transaction holds a connection open and blocks the Node.js event loop, degrading or crashing the API for all users.
3. **Unrecoverable without direct DB intervention** — there is no application-level endpoint to break the cycle once written.

### Likelihood Explanation

Any authenticated organization member who has created a transaction and added a threshold-based (tree) approver to it can trigger this with a single crafted PATCH request. No elevated privileges, leaked secrets, or race conditions are required. The attacker only needs to know their own approver's `id`, which is returned by the create-approver API response.

### Recommendation

Add an explicit self-reference guard immediately after fetching the approver, before the `dto.listId` branch:

```typescript
// In the dto.listId === number branch, before fetching newParent:
if (dto.listId === approver.id)
  throw new Error('An approver cannot be its own parent');
```

Additionally, add cycle detection to `getRootNodeFromNode` (e.g., a visited-ID set) as a defense-in-depth measure to prevent any future cycle from causing an infinite loop. [6](#0-5) 

### Proof of Concept

**Preconditions:**
- Attacker is an authenticated organization member.
- Attacker has created a transaction (becomes its creator).
- Attacker has added a tree approver (threshold ≥ 1, with at least one child) to the transaction. The tree approver's `id` is returned in the API response — call it `TREE_ID`.

**Steps:**

1. `POST /transactions` → create a transaction, note `transactionId = T`.
2. `POST /transactions/approvers/T` with body `{ approversArray: [{ threshold: 1, approvers: [{ userId: X }] }] }` → note the tree approver's `id = TREE_ID`.
3. `PATCH /transactions/approvers/T/TREE_ID` with body `{ listId: TREE_ID }`.

**Expected (correct) behavior:** Request rejected with "An approver cannot be its own parent."

**Actual behavior:** Request succeeds. The DB row for `TREE_ID` now has `listId = TREE_ID` and `transactionId = null`.

4. Any subsequent call that invokes `getRootNodeFromNode(TREE_ID)` — e.g., another `PATCH /transactions/approvers/T/TREE_ID` — enters an infinite loop, hanging the API worker indefinitely. [7](#0-6)
