### Title
Excessively Large TOTP `window` Value Allows OTP Reuse and Extended Validity Window in Password Reset Flow

### Summary
The TOTP configuration in `auth.service.ts` sets `window: 20` alongside `step: 60`. In otplib, `window` defines how many time-steps before and after the current step are accepted as valid. This makes every OTP code valid for approximately **41 minutes** (41 steps × 60 seconds) instead of the typical ~3 minutes (window=1). An attacker who intercepts a password-reset OTP email has a vastly extended window to exploit it, and the same OTP code can be submitted multiple times within that window since there is no single-use invalidation of the TOTP code itself.

### Finding Description

In `back-end/apps/api/src/auth/auth.service.ts`, the global TOTP options are set at module load time:

```typescript
totp.options = {
  digits: 8,
  step: 60,
  window: 20,   // ← accepts codes from 20 steps before to 20 steps after current time
};
``` [1](#0-0) 

With `step: 60` and `window: 20`, otplib accepts any TOTP code that is valid within ±20 steps of the current time. That is **41 steps × 60 seconds = 2,460 seconds ≈ 41 minutes** of total validity per code.

The password-reset flow is:
1. `createOtp()` generates a TOTP from `getOtpSecret(user.email)` and emails it to the user.
2. `verifyOtp()` calls `totp.check(token, secret)` — which uses the global `window: 20` — to validate the submitted code. [2](#0-1) 

There is no single-use enforcement on the TOTP code itself. The blacklist service operates on JWT access tokens, not on the 8-digit OTP value. This means the same OTP code can be submitted multiple times within the 41-minute window.

A secondary indicator of the misconfiguration is the dead-code in `getOtpToken`:

```typescript
const expires = new Date();
expires.setSeconds(expires.getSeconds() + totp.options.step * (totp.options.window as number));
// = 60 * 20 = 1200 seconds — computed but never used
``` [3](#0-2) 

The developer clearly intended the OTP lifetime to be `step × window = 1200 seconds`, but the `expires` variable is never passed to `jwtService.sign()`. The JWT expiry is instead driven by the `OTP_EXPIRATION` config variable (in minutes). The TOTP code itself, however, remains valid for the full 41-minute window regardless of the JWT.

### Impact Explanation

- **Password account takeover**: An attacker who intercepts the OTP email (e.g., via a compromised mail relay, email forwarding rule, or network interception) has ~41 minutes to submit the code and reset the victim's password — far beyond the ~3-minute window that `window: 1` would provide.
- **OTP reuse**: Because `totp.check()` does not consume or invalidate the code, the same 8-digit value can be submitted multiple times within the window. If the first submission fails for any reason (network error, wrong JWT), the attacker can retry.
- **Severity**: High — successful exploitation leads to full account takeover without any privileged access.

### Likelihood Explanation

- The attacker only needs to intercept or observe the OTP email. This is realistic via: compromised email infrastructure, misconfigured email forwarding, or a malicious insider with email access.
- No privileged application access is required.
- The 41-minute window is long enough to make manual exploitation trivial.

### Recommendation

Set `window` to `1` (the otplib default), which allows ±1 step (±60 seconds) of clock drift — sufficient for legitimate use while closing the extended attack window:

```typescript
totp.options = {
  digits: 8,
  step: 60,
  window: 1,  // ±60 seconds of clock drift tolerance
};
```

Additionally, enforce single-use OTP codes by storing the used TOTP counter value in Redis and rejecting any code whose counter has already been consumed.

### Proof of Concept

1. Attacker triggers password reset for `victim@example.com` via `POST /auth/otp`.
2. The server generates a TOTP from `OTP_SECRET + "victim@example.com"` and emails it.
3. Attacker intercepts the email and obtains the 8-digit OTP code.
4. Up to 41 minutes later, attacker submits the OTP code to `POST /auth/verify-otp` with the OTP JWT.
5. `totp.check(token, secret)` returns `true` because `window: 20` still covers the time of code generation.
6. The server resets the victim's password to attacker-controlled value, completing account takeover.
