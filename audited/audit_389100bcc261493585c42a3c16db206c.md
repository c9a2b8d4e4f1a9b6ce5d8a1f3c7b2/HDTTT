### Title
No Upper-Bound Validation on `validStart` Allows Transactions to Be Permanently Stuck in Unrecoverable State

### Summary
The `validateAndPrepareTransaction` function in `transactions.service.ts` rejects already-expired transactions via `isExpired()`, but applies no upper-bound check on how far in the future `validStart` may be. An authenticated user can submit transaction bytes with a `validStart` set decades ahead. The scheduler's cron windows never cover such a timestamp, and the expiry cron never fires for it, leaving the transaction permanently stuck in `WAITING_FOR_SIGNATURES` with no automated recovery path.

### Finding Description

**Root cause — missing upper-bound check at ingestion:**

In `validateAndPrepareTransaction`, the only temporal guard is the `isExpired` call: [1](#0-0) 

`isExpired` is defined as: [2](#0-1) 

It only checks whether `now >= validStart + duration` (i.e., already expired). It does **not** check whether `validStart` is unreasonably far in the future. A transaction with `validStart` = year 2076 passes this check trivially.

`validStart` is extracted directly from user-supplied bytes with no further constraint: [3](#0-2) 

**Why the scheduler never recovers it:**

The chain service processes transactions only within bounded time windows. The widest window is one day: [4](#0-3) [5](#0-4) 

A transaction 50 years in the future is never within any of these windows and is never passed to `prepareTransactions` or `addExecutionTimeout`.

**Why the expiry cron never fires:**

`handleExpiredTransactions` marks a transaction expired only when `validStart < 3 minutes ago`: [6](#0-5) 

A transaction with `validStart` 50 years in the future never satisfies this condition. It remains in `WAITING_FOR_SIGNATURES` indefinitely.

**DTO has no upper-bound constraint on `cutoffAt` or `reminderMillisecondsBefore` either:** [7](#0-6) 

`reminderMillisecondsBefore` carries only `@IsNumber()` — no `@Min(0)` — so a negative value causes `remindAt = validStart - (negative) = validStart + |value|`, scheduling a reminder arbitrarily far in the future and consuming a scheduler slot indefinitely.

### Impact Explanation

Any authenticated organization user can submit a transaction whose `validStart` is decades in the future. The record is permanently stored in `WAITING_FOR_SIGNATURES` status. No automated process ever executes or expires it. If the transaction belongs to a sequential `TransactionGroup`, the entire group's execution is blocked indefinitely. Recovery requires direct database intervention by an administrator. This constitutes permanent, unrecoverable corruption of transaction state.

### Likelihood Explanation

Low. Requires an authenticated organization user — no admin or operator role needed — to either deliberately or accidentally encode a far-future `validStart` in the Hedera SDK transaction bytes before submission. The Hedera SDK imposes no upper bound on `validStart`, making the crafted input trivially constructable.

### Recommendation

In `validateAndPrepareTransaction`, immediately after the `isExpired` check, add an upper-bound guard:

```typescript
const MAX_FUTURE_MS = 365 * 24 * 60 * 60 * 1_000; // 1 year
const validStartMs = sdkTransaction.transactionId.validStart.toDate().getTime();
if (validStartMs > Date.now() + MAX_FUTURE_MS) {
  throw new BadRequestException(ErrorCodes.TE); // or a dedicated error code
}
```

Additionally, add `@Min(0)` to `reminderMillisecondsBefore` in `CreateTransactionDto` to prevent negative values from scheduling reminders beyond `validStart`.

### Proof of Concept

1. Authenticate as a normal organization user (no elevated role required).
2. Using the Hedera SDK, build any valid transaction (e.g., `TransferTransaction`) and set its `transactionId` with `validStart = new Timestamp(50_years_from_now_seconds, 0)`.
3. Freeze, sign, and serialize the transaction to bytes.
4. `POST /transactions` with the serialized bytes, a valid `creatorKeyId`, and a valid `signature`.
5. The server calls `isExpired(sdkTransaction)` → returns `false` (transaction has not expired; it is in the future). All other checks pass.
6. The transaction is saved with `validStart` = 50 years from now and `status = WAITING_FOR_SIGNATURES`.
7. Observe: no scheduler cron (`handleTransactionsBetweenNowAndAfterThreeMinutes`, `…TenMinutesAndOneHour`, `…OneHourAndOneDay`) ever includes this transaction in its `Between(from, to)` query.
8. Observe: `handleExpiredTransactions` never marks it expired because `validStart` is not less than `3 minutes ago`.
9. The transaction record is permanently stuck — unexecuted and unexpired — with no automated recovery.

```typescript
export const isExpired = (transaction: SDKTransaction) => {
  if (!transaction.transactionId?.validStart) {
    return true;
  }

  const validStart = transaction.transactionId.validStart.toDate();
  const duration = transaction.transactionValidDuration;

  return new Date().getTime() >= validStart.getTime() + duration * 1_000;
};
```
