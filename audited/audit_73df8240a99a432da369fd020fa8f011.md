### Title
Threshold Reduction After Partial Approval Allows Transaction Creator to Bypass Multi-Signature Requirement

### Summary
The `updateTransactionApprover` function allows the transaction creator to lower the approval threshold on an in-progress approver tree without resetting the `approved` and `signature` fields of child approver records. Because previously collected approvals persist unchanged, a malicious creator who is also an approver can satisfy the new (lower) threshold using only their own pre-existing approval, bypassing the organizational multi-signature control entirely.

Additionally, `createTransactionApprovers` explicitly carries over existing `signature`, `userKeyId`, and `approved` values when re-adding a user to a new approver tree, compounding the same issue.

### Finding Description

**Root cause — threshold update does not reset child approval state**

`updateTransactionApprover` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` handles a `dto.threshold` update by writing only the new threshold value to the database:

```typescript
await transactionalEntityManager.update(TransactionApprover, approver.id, {
  threshold: dto.threshold,
});
``` [1](#0-0) 

No child approver records are touched. Their `approved`, `signature`, and `userKeyId` columns remain exactly as they were before the threshold change.

**Root cause — signature carryover in `createTransactionApprovers`**

When the creator calls `createTransactionApprovers` to build a new approver tree for the same transaction, the code explicitly copies the existing approval state for any user already present:

```typescript
if (typeof dtoApprover.userId === 'number') {
  const userApproverRecords = await this.getApproversByTransactionId(
    transactionId,
    dtoApprover.userId,
    transactionalEntityManager,
  );

  if (userApproverRecords.length > 0) {
    data.signature = userApproverRecords[0].signature;
    data.userKeyId = userApproverRecords[0].userKeyId;
    data.approved = userApproverRecords[0].approved;
  }
}
``` [2](#0-1) 

After inserting the new tree, `emitTransactionStatusUpdate` is called immediately, triggering a status re-evaluation with the stale approvals already in place. [3](#0-2) 

**Authorization gate — creator-only, no status guard**

Both `updateTransactionApprover` and `createTransactionApprovers` gate on `getCreatorsTransaction`, which only verifies the caller is the transaction creator. There is no check on the current transaction status (`WAITING_FOR_SIGNATURES`, etc.), so the creator can modify the approver tree at any point while the transaction is live. [4](#0-3) 

**`TransactionApprover` entity — the persisted fields**

The `approved` and `signature` columns on `TransactionApprover` are the sole source of truth for whether a user has approved: [5](#0-4) 

### Impact Explanation

A malicious transaction creator who is also listed as an approver can unilaterally approve and execute any transaction they create, regardless of the threshold set at creation time. This completely defeats the organizational multi-signature control: the creator sets a high threshold to appear legitimate, collects their own approval, then lowers the threshold to 1 (or whatever they have already satisfied). The transaction proceeds to execution with only the creator's signature, with no action required from any other approver.

Concrete impact: unauthorized execution of Hedera transactions (account updates, token operations, transfers, etc.) that were supposed to require consensus from multiple organizational members.

### Likelihood Explanation

The attacker is a normal authenticated user — the transaction creator. No admin keys, no leaked secrets, and no privileged access are required. The creator role is reachable by any organization member who submits a transaction. The two API calls needed (`POST /transactions/:id/approvers` or `PATCH /transactions/:id/approvers/:approverId`) are standard product flows. The attack requires only that the creator is also listed as one of the approvers, which is a common organizational pattern.

### Recommendation

1. **Reset child approval state on threshold change.** When `updateTransactionApprover` lowers a threshold, set `approved = NULL`, `signature = NULL`, and `userKeyId = NULL` on all direct child approver records of that tree node before persisting the new threshold.

2. **Do not carry over approval state in `createTransactionApprovers`.** Remove lines 325–329 in `approvers.service.ts`. A new approver tree must start with a clean slate; any prior approval is invalid under the new structure.

3. **Lock the approver tree once signing begins.** Add a status guard in both `createTransactionApprovers` and `updateTransactionApprover` that rejects modifications when the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status.

### Proof of Concept

**Setup:** Organization with users Alice (creator + approver), Bob, Charlie. Threshold requirement: 2-of-3.

**Step 1 — Creator sets up approver tree (threshold = 2):**
```
POST /transactions/42/approvers
{ approversArray: [{ threshold: 2, approvers: [
    { userId: Alice },
    { userId: Bob },
    { userId: Charlie }
]}]}
```

**Step 2 — Alice approves (her own transaction):**
```
POST /transactions/42/approvers/approve
{ userKeyId: aliceKeyId, signature: "0xALICE_SIG", approved: true }
```
DB state: Alice's approver record → `approved=true, signature=0xALICE_SIG`.

**Step 3 — Creator (Alice) lowers threshold to 1:**
```
PATCH /transactions/42/approvers/<tree_node_id>
{ threshold: 1 }
```
`updateTransactionApprover` writes `threshold=1` only. Alice's child record still has `approved=true`.

**Step 4 — Status re-evaluation (triggered on next approval event or scheduler tick):**
The system evaluates the approver tree: threshold=1, Alice.approved=true → threshold met → transaction status advances to `WAITING_FOR_EXECUTION` → chain service executes the transaction.

**Result:** Transaction executes with only Alice's signature. Bob and Charlie never approved.
