### Title
Any Verified Organization Member Can Read All Other Users' `mnemonicHash` and `index` via Unguarded Bulk Key Endpoint

### Summary

The `GET /user-keys` endpoint (`UserKeysAllController`) returns every `UserKey` record in the database — including the `mnemonicHash` and `index` fields — to any authenticated, verified user. A parallel endpoint (`GET /user/:userId/keys`) correctly strips those fields for keys that do not belong to the requesting user, but the bulk endpoint bypasses that restriction entirely. Any organization member can therefore enumerate the mnemonic-hash and HD-derivation index of every other member's registered keys.

### Finding Description

**Root cause — two code paths with inconsistent access control**

The per-user endpoint calls `getUserKeysRestricted`, which uses a TypeORM `select` projection to hide `mnemonicHash` and `index` when the requester is not the owner: [1](#0-0) 

The bulk endpoint (`UserKeysAllController`) calls `getUserKeys` instead: [2](#0-1) 

`getUserKeys` issues a plain `findAndCount` with **no `select` projection and no `where` filter**: [3](#0-2) 

The serialization DTO exposes both sensitive fields unconditionally: [4](#0-3) 

The only guard on the bulk endpoint is `VerifiedUserGuard` — any verified member of the organization satisfies it: [5](#0-4) 

**What the exposed fields reveal**

`mnemonicHash` is the hash of a user's BIP-39 recovery phrase stored server-side so the system can correlate keys to mnemonics during key recovery: [6](#0-5) 

`index` is the HD derivation index used alongside the mnemonic to reconstruct the private key: [7](#0-6) 

Together, `mnemonicHash + index` uniquely fingerprint a victim's key derivation path. An attacker who also obtains (or brute-forces) the mnemonic can use the leaked `index` to derive the exact private key without any further guessing.

The e2e test suite confirms the per-user endpoint correctly hides these fields for non-owners, but no equivalent test exists for the bulk endpoint: [8](#0-7) 

### Impact Explanation

- **Cross-user sensitive metadata exposure**: every verified organization member can read the `mnemonicHash` and `index` of every other member's keys in a single paginated request.
- **Key-correlation attack**: an attacker learns which of a victim's keys share the same mnemonic (same `mnemonicHash`), reducing the search space for mnemonic brute-force from the full BIP-39 space to a targeted attack against one known mnemonic.
- **Collusion amplification** (analog to the NuCypher report): a malicious organization member who is also a transaction approver/signer can combine leaked `mnemonicHash + index` with knowledge of the victim's public key to mount a targeted offline attack, then impersonate the victim's signing key in multi-signature workflows without the victim's consent.
- **Full organization key enumeration**: the paginated endpoint leaks `userId` alongside the key metadata, so the attacker also learns the complete mapping of users to their key derivation parameters.

### Likelihood Explanation

- **Attacker precondition**: a valid organization account (verified status). This is the lowest privilege level in the system — any employee or contractor added to the organization qualifies.
- **Exploit complexity**: a single authenticated HTTP GET to `/user-keys?page=1&size=100` with the attacker's own JWT. No special tooling required.
- **No rate-limit barrier specific to this endpoint** beyond the general API rate limiter, so the full key list can be harvested in seconds.

### Recommendation

Apply the same field-level ownership check used in `getUserKeysRestricted` to the bulk endpoint. The simplest fix is to add a `userId` filter so the bulk endpoint only returns the requesting user's own keys, or to add a `select` projection that strips `mnemonicHash` and `index` for rows not owned by the requester. Concretely:

1. In `getUserKeys` (or a new `getUserKeysForUser` variant), accept the requesting `User` and filter by `userId` **and** apply the same `select` mask as `getUserKeysRestricted`.
2. Update `UserKeysAllController.getUserKeys` to inject `@GetUser() user: User` and pass it to the service.
3. Add an e2e test asserting that `GET /user-keys` does **not** return `mnemonicHash` or `index` for keys belonging to other users.

### Proof of Concept

```
# 1. Attacker logs in as any verified organization member
POST /auth/login  { "email": "attacker@org.com", "password": "..." }
# → receives JWT: eyJ...

# 2. Attacker calls the unguarded bulk endpoint
GET /user-keys?page=1&size=100
Authorization: Bearer eyJ...

# 3. Response includes ALL users' keys with sensitive fields:
{
  "totalItems": 12,
  "items": [
    { "id": 3, "userId": 5, "publicKey": "302a...", "mnemonicHash": "0xabc123...", "index": 2 },
    { "id": 4, "userId": 7, "publicKey": "302a...", "mnemonicHash": "0xdef456...", "index": 0 },
    ...
  ]
}

# 4. Attacker now knows victim userId=5 used mnemonic with hash 0xabc123... at index 2.
#    Combined with any offline mnemonic leak or brute-force, the attacker can derive
#    the victim's private key and sign transactions on their behalf.
```
