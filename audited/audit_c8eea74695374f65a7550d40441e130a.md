### Title
Plaintext Mnemonic Words Persist in Pinia Store After Recovery Phrase Migration Completes

### Summary
The global Pinia user store (`storeUser.ts`) holds a `RecoveryPhrase` object containing the 24 plaintext mnemonic words in the renderer process heap. In `MigrateRecoveryPhraseHash.vue`, the `handleContinue` handler successfully completes the migration and navigates away but never clears `user.recoveryPhrase`. The plaintext mnemonic remains in the JavaScript heap for the entire session until the user explicitly logs out, mirroring the iOS/Android vulnerability described in the external report.

### Finding Description

**Root cause — global store retains plaintext words:**

`storeUser.ts` declares `recoveryPhrase` as a persistent Pinia ref:

```ts
const recoveryPhrase: Ref<RecoveryPhrase|null> = ref(null);
``` [1](#0-0) 

`createRecoveryPhrase` (called by `setRecoveryPhrase`) stores the full plaintext word array alongside the `Mnemonic` object:

```ts
return { mnemonic, words, hash };
``` [2](#0-1) 

**Specific missing clear — `handleContinue` in `MigrateRecoveryPhraseHash.vue`:**

`handleSkip` (line 51) correctly calls `await user.setRecoveryPhrase(null)` before navigating away: [3](#0-2) 

But `handleContinue` (the success path) calls `router.push({ name: 'transactions' })` without ever clearing the phrase: [4](#0-3) 

The asymmetry is explicit: the developer remembered to clear on skip but not on success. After `handleContinue` returns, `user.recoveryPhrase.words` (24 plaintext BIP-39 words) lives in the V8 heap for the rest of the session.

**Secondary instance — `GenerateOrImport.vue` `handleImport`:**

After the user imports their phrase and clicks "Next", `handleImport` calls `props.handleNext()` without clearing: [5](#0-4) 

**Only `logout` clears the phrase:** [6](#0-5) 

### Impact Explanation
An attacker who obtains a V8 heap snapshot or crash dump of the Electron renderer process (e.g., via an integrated crash-reporting service such as Sentry, or via a local process memory read) will find the 24-word mnemonic in plaintext. With the mnemonic, the attacker can derive all private keys at any index and drain all associated Hedera accounts. Impact is total wallet compromise.

### Likelihood Explanation
The Electron renderer process is a Chromium renderer; V8 heap snapshots are a standard debugging artifact. If the application integrates any crash reporter that captures renderer heap data (common in Electron apps), the mnemonic is transmitted off-device on every crash. Even without a crash reporter, the mnemonic persists in memory for the entire session — potentially hours — rather than the seconds needed for the migration operation. The trigger path (user completes recovery phrase migration → navigates to transactions → app crashes or heap is inspected) is a realistic, low-privilege scenario.

### Recommendation
In `MigrateRecoveryPhraseHash.vue`, call `await user.setRecoveryPhrase(null)` immediately after `updateKeyPairsHash` succeeds and before `router.push`:

```ts
const handleContinue = async () => {
  ...
  if (!error) {
    await user.setRecoveryPhrase(null); // add this
    toastManager.success('Recovery phrase hash updated successfully');
    await router.push({ name: 'transactions' });
  }
  ...
};
```

Apply the same fix to `GenerateOrImport.vue` `handleImport` and audit every other call site where `setRecoveryPhrase` is set but the corresponding clear is missing after the operation completes. Additionally, consider zeroing the `words` array elements before nulling the ref, since JavaScript GC does not guarantee immediate memory reclamation.

### Proof of Concept

1. Open the Hedera Transaction Tool desktop app with an existing account that has legacy (non-argon2) mnemonic hashes, triggering the migration flow.
2. Enter the 24-word recovery phrase in `MigrateRecoveryPhraseHash.vue`.
3. Click "Continue" — `handleContinue` runs, migration succeeds, app navigates to the transactions page.
4. Attach a Node.js/V8 inspector to the Electron renderer process (`--inspect` flag or via `electron --remote-debugging-port=9229`).
5. Take a heap snapshot via Chrome DevTools → Memory → Take Snapshot.
6. Search the snapshot for any of the 24 mnemonic words (e.g., search string objects for a known word).
7. Observe that `user.recoveryPhrase.words` (the full 24-word array) is present in the heap, confirming the mnemonic was never cleared after migration completed.
