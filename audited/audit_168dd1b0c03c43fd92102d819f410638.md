### Title
Concurrent Signature Uploads Silently Overwrite Each Other's `transactionBytes`, Permanently Losing Signatures

### Summary
`uploadSignatureMaps` in `signers.service.ts` reads the current `transactionBytes` from the database, merges new signatures into an in-memory copy, then writes the result back with an unconditional `UPDATE` that has no status guard or optimistic-concurrency check. When two signers submit signatures for the same transaction concurrently, the second write silently replaces the first, discarding the earlier signer's contribution. The same flaw exists in `importSignatures` in `transactions.service.ts`. Because the `TransactionSigner` row is still inserted, the system believes both signers have signed, but the actual cryptographic bytes only contain one signer's signature, so `processTransactionStatus` will never promote the transaction to `WAITING_FOR_EXECUTION`, permanently blocking execution.

### Finding Description

**Root cause — unconditional byte overwrite**

`loadTransactionData` fetches a snapshot of `transactionBytes` at time T₀: [1](#0-0) 

`processTransactionSignatures` merges the caller's signatures into that snapshot in memory: [2](#0-1) 

`bulkUpdateTransactions` then writes the result back with a bare `UPDATE … WHERE id = ANY(…)` — no `AND status = …` guard, no `AND "transactionBytes" = <expected>` optimistic-lock: [3](#0-2) 

The identical pattern exists in `importSignatures`: [4](#0-3) 

**Contrast with correctly guarded paths**

`_executeTransaction` and `processTransactionStatus` both use `AND status = :currentStatus` / `AND status = :oldStatus` to detect concurrent changes and abort if the row was already mutated: [5](#0-4) [6](#0-5) 

`executeTransactionGroup` is additionally protected by a distributed `@MurLock`: [7](#0-6) 

`uploadSignatureMaps` has **no equivalent protection**.

**Exploit flow**

1. Transaction T requires signatures from User A and User B.
2. Both users click "Sign" at roughly the same time (or the "Sign All" automation fires for both).
3. Both requests call `loadTransactionData` and receive the same `transactionBytes` snapshot (no signatures yet).
4. Both independently merge their own signature into the snapshot in memory.
5. Request A commits first: DB now holds bytes containing only A's signature.
6. Request B commits second: DB now holds bytes containing only B's signature — **A's signature is gone**.
7. Both `TransactionSigner` rows exist in the DB, so the system believes both have signed.
8. `processTransactionStatus` calls `computeSignatureKey` and `hasValidSignatureKey`; the bytes fail the check because A's signature is missing.
9. The transaction is never promoted to `WAITING_FOR_EXECUTION` and can never be executed. [8](#0-7) 

### Impact Explanation

- **Permanent DoS on specific transactions**: once the race is lost, the missing signature cannot be re-added because the signer's `TransactionSigner` row already exists (duplicate-key constraint prevents re-insertion) and the bytes no longer contain their signature. The transaction is stuck in `WAITING_FOR_SIGNATURES` until it expires.
- **Silent integrity failure**: the system reports success to both signers and shows both as having signed, yet the transaction is cryptographically incomplete.
- **Affects organization multi-signer workflows**, which are the primary use-case for the backend service.

### Likelihood Explanation

- Multi-signer organization transactions are the core workflow; concurrent signing is expected and actively encouraged by the UI ("Sign All" button, group signing).
- The k6 load-test suite explicitly exercises concurrent API calls.
- No special attacker capability is required — two legitimate users signing at the same time is sufficient.
- The window is wide: `validateAndProcessSignatures` is async and involves mirror-node lookups, giving ample time for a second request to interleave. [9](#0-8) 

### Recommendation

1. **Optimistic concurrency on `bulkUpdateTransactions`**: add `AND "transactionBytes" = <original_bytes>` (or a hash/version column) to the `WHERE` clause. If `affected = 0`, retry by re-reading the latest bytes and re-merging.
2. **Alternatively, use a per-transaction advisory lock** (e.g., `SELECT … FOR UPDATE` on the transaction row) inside the DB transaction that wraps `persistSignatureChanges`, so concurrent writes are serialised.
3. **Apply the same fix to `importSignatures`** in `transactions.service.ts`.

### Proof of Concept

```
1. Create a transaction T requiring two signers (User A, User B).
2. Simultaneously POST to /transactions/:id/signers with User A's signature map
   and POST to /transactions/:id/signers with User B's signature map.
3. Observe: both requests return HTTP 201 and both TransactionSigner rows exist.
4. Fetch transaction bytes; verify only one signer's signature is present.
5. Observe: transaction remains in WAITING_FOR_SIGNATURES indefinitely and
   eventually expires without executing.
```

The race is deterministic under load: the k6 suite at `automation/k6/helpers/sign-transactions.ts` already provides the tooling to reproduce it at scale. [3](#0-2) [10](#0-9)

    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };

export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }
