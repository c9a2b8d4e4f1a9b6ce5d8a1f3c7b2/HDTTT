### Title
Sequential Transaction Group Execution Ignores `seq` Ordering Field, Using `validStart` Instead

### Summary
The `TransactionGroupItem.seq` field is the documented mechanism for defining execution order within a sequential transaction group. However, when the Chain Service loads a group for execution, it orders items by `transaction.validStart ASC` and completely ignores `seq`. In `executeTransactionGroup`, sequential groups then execute in that `validStart`-derived order. Any group where the `seq` order diverges from the `validStart` order will execute transactions out of the intended sequence, causing dependent operations to fail or execute incorrectly.

### Finding Description

**Root cause — ordering metadata is discarded on load:**

In `prepareTransactions`, when a sequential or atomic group is fetched from the database, the TypeORM query sorts by `transaction.validStart ASC`: [1](#0-0) 

The `seq` column on `TransactionGroupItem` — which is the field explicitly documented as "Sequence of transaction execution within the transaction group" — is never referenced in this query or anywhere downstream. [2](#0-1) 

**Execution path — order is consumed directly:**

`executeTransactionGroup` iterates `transactionGroup.groupItems` in the order they were loaded. For sequential groups it executes them one-by-one in that order: [3](#0-2) 

`collateGroupAndExecute` also anchors its timing to `groupItems[0].transaction.validStart`, assuming the first item by `validStart` is the first to execute: [4](#0-3) 

**Exploit flow:**

1. A malicious (or mistaken) creator submits a sequential group via `POST /transaction-groups` with `sequential: true`.
2. They assign `seq` values that do not match the ascending `validStart` order — e.g., `seq=0` on the transaction with the later `validStart`, `seq=1` on the one with the earlier `validStart`.
3. Other organization members see the `seq`-ordered list in the UI and sign, believing execution will follow `seq` order.
4. The Chain Service loads the group ordered by `validStart ASC` and executes in that order — the opposite of what signers approved. [5](#0-4) 

### Impact Explanation

For sequential groups, execution order is semantically critical. Dependent operations (e.g., `FileCreate` → `FileAppend`, token association → token transfer, account creation → HBAR transfer) will fail or produce incorrect on-chain state if executed out of order. Because Hedera transactions expire at `validStart + 180 s`, failed transactions cannot be retried — the window is gone. This constitutes an unrecoverable corruption of the intended transaction sequence and potential financial loss for the organization.

### Likelihood Explanation

Any authenticated organization user can create a transaction group. No privileged access is required. The divergence between `seq` and `validStart` order can arise naturally (e.g., a creator sets `validStart` values for scheduling reasons that do not match the intended `seq` order) or deliberately (a malicious creator deceives co-signers). The `seq` field is surfaced in the UI and API response, so signers have no way to detect that the actual execution order will differ.

### Recommendation

Replace the `validStart`-based sort in `prepareTransactions` with a sort on `groupItems.seq`:

```typescript
order: {
  groupItems: {
    seq: 'ASC',   // use the documented ordering field
  },
},
```

Additionally, in `executeTransactionGroup`, add an explicit sort on `seq` before iterating, so the field is authoritative regardless of how the caller loaded the group:

```typescript
transactionGroup.groupItems.sort((a, b) => a.seq - b.seq);
```

### Proof of Concept

1. Log in as an organization user.
2. Create a sequential transaction group with two transactions:
   - Item A: `seq=0`, `validStart = now + 120 s` (e.g., `FileCreateTransaction`)
   - Item B: `seq=1`, `validStart = now + 60 s` (e.g., `FileAppendTransaction` referencing the file from A)
3. Have required signers approve (they see A listed first by `seq`).
4. Wait for the Chain Service scheduler to pick up the group.
5. Observe: the service loads items ordered by `validStart ASC` → B executes first at `now+60s`, fails with `INVALID_FILE_ID` because the file does not exist yet; A executes at `now+120s` but the append window has already been missed or the group is already marked failed.
6. Both transactions expire; the intended operation is permanently unexecutable. [6](#0-5) [7](#0-6)

# transaction_group_item

**Description**: A transaction group is a collection of transactions that need to be executed simultaneously. This allows for a coordinated and synchronized execution of multiple related transactions. The `transaction_group_item` table stores information about the individual transaction in the transaction group and its relative sequence to other transactions in the group.

## Columns

| Column Name       | Type      | Description                                                                                                                      |

| **seq**           | Integer   | Sequence of transaction execution within the transaction group, if it matter. If the sequence is not important this can be null. |
| **transactionId** | String    | The transaction id (Foreign key to transaction)                                                                                  |
| **groupId**       | Timestamp | The group id of the transaction ( Foreign key to transaction_group )                                                             |
