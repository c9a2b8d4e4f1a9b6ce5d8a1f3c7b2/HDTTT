### Title
`approveTransaction` Silently Returns `false` When User Has No Keys, Leaving Approval Unrecorded and Caller Uninformed

### Summary
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction` function silently returns `false` when `user.keys.length === 0` after calling `attachKeys`. No exception is thrown, no error is propagated, and the database approval record is never written. The controller passes this `false` directly back to the HTTP client as a `200 OK` response, giving the caller no actionable signal that the approval was not recorded.

### Finding Description
In `approveTransaction` (line 547–621), after all authorization checks pass (user is a valid approver, has not already approved, transaction is in the correct status), the function calls `attachKeys` to populate `user.keys` and then immediately checks:

```typescript
await attachKeys(user, this.dataSource.manager);
if (user.keys.length === 0) return false;   // line 567
```

If `user.keys` is empty at this point, the function exits early. The database update that records the approval (lines 599–610) is never reached. No exception is thrown. The controller at `approvers.controller.ts` line 67–73 returns the service result directly:

```typescript
approveTransaction(...): Promise<boolean> {
  return this.approversService.approveTransaction(body, transactionId, user);
}
```

So the HTTP client receives `HTTP 200 OK` with body `false`. There is no error code, no message, and no indication that the approval was silently dropped.

The `OnlyOwnerKey` guard on the endpoint checks that `dto.userKeyId` belongs to the authenticated user at request time. However, `attachKeys` is called inside the service after the guard has already passed. A race condition — where the user's key is deleted between the guard check and the `attachKeys` call — can produce `user.keys.length === 0` even though the guard passed. In that window, the approval is silently discarded.

### Impact Explanation
- The approval is not persisted. If this approver's signature is required for the transaction to advance to `WAITING_FOR_EXECUTION`, the transaction is permanently stuck in `WAITING_FOR_SIGNATURES`.
- The HTTP client receives `200 OK` with `false`, which is indistinguishable from a legitimate "approval recorded but returned false for another reason" response. Callers that check only the HTTP status code will believe the approval succeeded.
- No notification event is emitted (the `emitTransactionStatusUpdate` / `emitTransactionUpdate` calls at lines 614–618 are also skipped), so observers and the notification service are never informed.

### Likelihood Explanation
The primary trigger is a race condition: a user's key is deleted (or soft-deleted) between the `OnlyOwnerKey` guard evaluation and the `attachKeys` call inside the service. This is a realistic scenario in a multi-user organization where an admin can remove keys. The window is small but deterministically reachable by a concurrent delete request. Additionally, any future code path that bypasses the guard (e.g., internal service-to-service calls) would expose this silently.

### Recommendation
Replace the silent `return false` with a thrown exception that clearly communicates the failure reason:

```typescript
await attachKeys(user, this.dataSource.manager);
if (user.keys.length === 0) {
  throw new BadRequestException('User has no registered keys and cannot approve this transaction.');
}
```

This ensures the HTTP client receives a `400 Bad Request` with a descriptive message, the approval is never silently dropped, and the caller can take corrective action.

### Proof of Concept
1. Authenticate as User A who is a valid approver for Transaction T.
2. Concurrently, have an admin delete all of User A's keys.
3. User A sends `POST /transactions/:transactionId/approvers/approve` with a valid `userKeyId` (the guard passes because the key existed at guard-evaluation time).
4. Inside `approveTransaction`, `attachKeys` now returns an empty array because the key was deleted.
5. Line 567 triggers: `return false`.
6. HTTP response: `200 OK`, body: `false`.
7. The `TransactionApprover` row for User A remains with `signature = null` and `approved = null`.
8. Transaction T never advances past `WAITING_FOR_SIGNATURES`.
