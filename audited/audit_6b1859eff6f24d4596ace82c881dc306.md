### Title
Missing Transaction Status Guard in `createTransactionApprovers` Allows Approver Mutation on Terminal-State Transactions

### Summary
The `createTransactionApprovers` and `updateTransactionApprover` functions in `approvers.service.ts` perform no check on the current transaction status before mutating approver state. Any authenticated user who is the creator of a transaction can add or update approvers after the transaction has reached a terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). This mirrors the external report's pattern exactly: a state-mutating function callable without a required state precondition, enabling fraudulent event emission and persistent data corruption.

### Finding Description

**Root cause — `createTransactionApprovers` (no status guard):**

`createTransactionApprovers` calls only `getCreatorsTransaction`, which exclusively verifies the caller is the transaction creator. It performs no check on `transaction.status`. [1](#0-0) 

`getCreatorsTransaction` itself only enforces creator identity, never inspects status: [2](#0-1) 

After the approver tree is inserted, the function unconditionally emits a `emitTransactionStatusUpdate` notification event regardless of the transaction's current state: [3](#0-2) 

**Same root cause in `updateTransactionApprover`:**

`updateTransactionApprover` also calls `getCreatorsTransaction` and has no status guard, allowing approver tree restructuring on terminal transactions: [4](#0-3) 

**Contrast with functions that do enforce status:**

`approveTransaction` correctly rejects calls when the transaction is not in an active state: [5](#0-4) 

`validateTransactionStatus` in `signers.service.ts` similarly enforces status before signature upload: [6](#0-5) 

The terminal statuses that should block approver mutation are defined in `transactions.service.ts`: [7](#0-6) 

**Exploit path:**
1. Attacker (any authenticated user) creates a transaction → becomes creator.
2. Transaction progresses to `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`.
3. Attacker calls `POST /transactions/:transactionId/approvers` with a crafted `CreateTransactionApproversArrayDto`.
4. `createTransactionApprovers` succeeds: new `TransactionApprover` rows are inserted into the database for a completed transaction.
5. `emitTransactionStatusUpdate` fires, pushing a spurious status-change notification to all subscribers of that transaction.
6. Steps 3–5 can be repeated indefinitely.

The REST endpoint is guarded only by JWT authentication and creator identity: [8](#0-7) 

### Impact Explanation

1. **Fraudulent notification events**: `emitTransactionStatusUpdate` is emitted for transactions already in terminal states, causing all observers, signers, and the creator to receive spurious "status changed" notifications. This corrupts the notification stream and can cause clients to re-render or re-fetch stale/incorrect state.
2. **Persistent approver data corruption**: `TransactionApprover` rows are permanently inserted for completed transactions. Any downstream logic that queries approvers by `transactionId` (e.g., `getApproversByTransactionId`, `shouldApproveTransaction`) will return stale or fabricated approver records for transactions that are no longer actionable.
3. **Integrity failure in the state machine**: The invariant that "a terminal-state transaction is immutable" is violated. The approver tree — which is part of the transaction's authorization model — can be rewritten after execution, undermining audit trails.

### Likelihood Explanation

The attacker precondition is minimal: any registered, verified user who has ever created a transaction. No privileged keys, no admin access, and no leaked credentials are required. The endpoint `POST /transactions/:transactionId/approvers` is a standard REST call documented in the API. The creator role is reachable by any user of the system.

### Recommendation

Add an explicit status guard at the top of `createTransactionApprovers` and `updateTransactionApprover`, mirroring the pattern already used in `approveTransaction`:

```typescript
async createTransactionApprovers(user, transactionId, dto) {
  const transaction = await this.getCreatorsTransaction(transactionId, user);

  const terminalStatuses = [
    TransactionStatus.EXECUTED,
    TransactionStatus.EXPIRED,
    TransactionStatus.FAILED,
    TransactionStatus.CANCELED,
    TransactionStatus.ARCHIVED,
    TransactionStatus.REJECTED,
  ];
  if (terminalStatuses.includes(transaction.status)) {
    throw new BadRequestException('Cannot modify approvers of a completed transaction');
  }
  // ... rest of function
}
```

Apply the same guard to `updateTransactionApprover` and `removeTransactionApprover`.

### Proof of Concept

1. Register two users: `creator` and `observer`.
2. As `creator`, call `POST /transactions` to create a valid transaction. Record `transactionId`.
3. Wait for or trigger the transaction to reach `EXECUTED` or `CANCELED` status.
4. As `creator`, call:
   ```
   POST /transactions/:transactionId/approvers
   Authorization: Bearer <creator_jwt>
   Body: { "approversArray": [{ "userId": <any_valid_user_id> }] }
   ```
5. **Expected (correct) behavior**: `400 Bad Request` — transaction is in a terminal state.
6. **Actual behavior**: `201 Created` — approver row inserted; `emitTransactionStatusUpdate` fires; `observer` receives a spurious status-change notification for the already-completed transaction.
7. Repeat step 4 indefinitely to continuously emit fraudulent events and accumulate approver rows.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
