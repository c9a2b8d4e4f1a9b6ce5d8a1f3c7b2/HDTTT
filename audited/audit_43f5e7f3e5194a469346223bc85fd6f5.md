### Title
Duplicate `TransactionSigner` Records Insertable via TOCTOU Race on Missing Unique Constraint

### Summary
The `TransactionSigner` entity lacks a database-level unique constraint on `(transactionId, userKeyId)`. The application-level deduplication check in `loadTransactionData()` is performed outside the database transaction that executes the insert in `bulkInsertSigners()`. Two concurrent signature-upload requests from the same user for the same transaction will both pass the deduplication check and both successfully insert duplicate signer rows, corrupting the signer state for that transaction.

### Finding Description

**Root cause — missing unique constraint:**

`TransactionSigner` declares only a plain index on `(transactionId, userKeyId)`: [1](#0-0) 

Compare with `TransactionObserver`, which correctly declares a **unique** index on the same kind of pair: [2](#0-1) 

The migration confirms no `UNIQUE` constraint exists on `transaction_signer`: [3](#0-2) 

**TOCTOU window — check and insert are not atomic:**

Step 1 — existing signers are loaded **outside** any database transaction: [4](#0-3) 

Step 2 — the deduplication check runs against that snapshot, also outside the DB transaction: [5](#0-4) 

Step 3 — the insert runs inside a separate DB transaction with no conflict clause: [6](#0-5) 

**Exploit path:**
1. Authenticated user holds a valid key for transaction T.
2. User sends two concurrent `POST /transactions/T/signers` requests with the same signature map.
3. Both requests execute `loadTransactionData()` before either commits — both see the same (empty or partial) `existingSignerIds` set.
4. Both pass the `!existingSignerIds.has(userKey.id)` check and push the same `userKey` into `signersToInsert`.
5. Both call `bulkInsertSigners()`. Because there is no `ON CONFLICT DO NOTHING` and no unique constraint, both inserts succeed.
6. The `transaction_signer` table now contains two rows with identical `(transactionId, userKeyId)`.

### Impact Explanation

Duplicate signer rows corrupt the integrity of the signature-counting logic used by `processTransactionStatus` and `TransactionSignatureService`. If that logic counts all rows rather than distinct `userKeyId` values, a single user's key can be counted multiple times toward the required signature threshold, causing a transaction to advance to `WAITING_FOR_EXECUTION` prematurely — i.e., before all legitimately required signers have signed. This is an unauthorized state transition that bypasses the multi-signature security model of the platform.

Even if the status logic deduplicates, the duplicate rows permanently corrupt the audit trail (who signed, when) and can cause incorrect UI display and notification behavior.

### Likelihood Explanation

The attacker is any authenticated user who is a valid signer on a transaction. No privileged access is required. The race window is the round-trip time of a single HTTP request — easily exploitable by sending two requests in parallel from a script. The missing unique constraint means the race always wins; there is no database-level backstop.

### Recommendation

1. **Add a unique constraint** on `(transactionId, userKeyId)` in `TransactionSigner`, mirroring what `TransactionObserver` already does:

```typescript
@Index(['transactionId', 'userKeyId'], { unique: true })
export class TransactionSigner { ... }
```

2. **Use `INSERT ... ON CONFLICT DO NOTHING`** in `bulkInsertSigners()` so concurrent inserts are idempotent:

```typescript
await manager
  .createQueryBuilder()
  .insert()
  .into(TransactionSigner)
  .values(signersToInsert)
  .orIgnore()   // ON CONFLICT DO NOTHING
  .returning('*')
  .execute();
```

Both fixes together eliminate the TOCTOU window and enforce the invariant at the storage layer, consistent with how `TransactionObserver` is already protected.

### Proof of Concept

```
1. Obtain a valid JWT for user U who holds key K registered for transaction T
   (T must be in WAITING_FOR_SIGNATURES status).

2. Build a valid signatureMap for key K over transaction T.

3. Fire two concurrent requests:
     POST /transactions/T/signers  { signatureMap: ... }   # request A
     POST /transactions/T/signers  { signatureMap: ... }   # request B

4. Both return HTTP 201.

5. Query the database:
     SELECT * FROM transaction_signer
     WHERE "transactionId" = T AND "userKeyId" = K_id;

   Expected (correct): 1 row
   Actual (vulnerable): 2 rows with identical (transactionId, userKeyId)

6. The duplicate row inflates the signer count, potentially causing
   processTransactionStatus to transition T to WAITING_FOR_EXECUTION
   before all required co-signers have signed.
```
