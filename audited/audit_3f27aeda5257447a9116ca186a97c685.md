### Title
Cross-Transaction Approver Deletion via Missing Ownership Validation in `removeTransactionApprover`

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint in the API verifies that the authenticated user is the creator of `:transactionId`, but then calls `removeTransactionApprover(id)` without ever verifying that the approver `:id` actually belongs to `:transactionId`. This allows any verified user who owns at least one transaction to delete approvers — including entire approval trees — from any other user's transaction, completely bypassing multi-signature requirements.

### Finding Description

**Vulnerability class:** Invalid Validation / Authorization bypass on state mutation (direct analog of H-02's "authorized on entity A, mutates entity B without cross-validation").

**Root cause — controller does not cross-validate approver ownership:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts (lines 102-113)
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver `id` with NO cross-check
  return true;
}
``` [1](#0-0) 

Step 1 (`getCreatorsTransaction`) only confirms the user created `transactionId`. It does not return any data used to constrain step 2.

**Root cause — service deletes by approver ID alone:**

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts (lines 533-544)

The `update` path has this guard; the `delete` path does not.

**`removeNode` recursively soft-deletes the entire approver tree:**

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts (lines 204-231)
async removeNode(listId: number): Promise<void> {
  await this.repo.query(`
    with recursive approversToDelete AS (
      select "id", "listId", "deletedAt" from transaction_approver where "id" = $1
      union all
        select transaction_approver."id", ...
        from transaction_approver, approversToDelete
        where approversToDelete."id" = transaction_approver."listId"
    )
    update transaction_approver set "deletedAt" = now()
    from approversToDelete
    where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
  `, [listId]);
}
``` [5](#0-4) 

Targeting a root approver of another transaction deletes the entire approval tree for that transaction.

### Impact Explanation

- **Unauthorized state mutation:** Any verified user can delete approvers from any transaction they did not create, bypassing the "only creator can modify" invariant.
- **Multi-signature bypass:** By deleting all approvers from a target transaction, an attacker can eliminate the approval requirement entirely, allowing the transaction to proceed without the required signatures.
- **Permanent corruption:** Soft-deleted approvers are gone from the active approval workflow. The transaction's approval state is irrecoverably corrupted unless manually restored at the database level.
- **Scope:** All transactions in the system are affected, not just the attacker's own.

### Likelihood Explanation

- **Attacker preconditions:** Must be a registered, verified user with at least one created transaction. This is the normal user baseline — no admin access required.
- **Approver ID discovery:** The `GET /transactions/:transactionId/approvers` endpoint returns approver IDs for any transaction the user has access to (as observer, signer, or approver). IDs are sequential integers and may be guessable even without direct access.
- **No rate limiting or anomaly detection** is visible on this endpoint.
- **Trivially exploitable:** A single crafted `DELETE` request is sufficient.

### Recommendation

In `removeTransactionApprover` (service), after fetching the approver, verify that it belongs to the authorized transaction. The fix mirrors what `updateTransactionApprover` already does correctly:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Get root node and verify it belongs to the authorized transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to the specified transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service method, and update the controller call accordingly.

### Proof of Concept

**Setup:**
- User A creates Transaction 1 with no approvers (or a trivial one).
- User B creates Transaction 2 and adds a multi-sig approval tree (approver IDs 10, 11, 12 where 10 is the root).

**Attack:**
1. Attacker (User A) calls `GET /transactions/2/approvers` (if they have any access) or enumerates approver IDs.
2. Attacker sends: `DELETE /transactions/1/approvers/10` (using their own `transactionId=1` but targeting approver `id=10` from Transaction 2).
3. Server checks: Is User A the creator of Transaction 1? **Yes** → passes.
4. Server calls `removeTransactionApprover(10)` → fetches approver 10 (belongs to Transaction 2) → calls `removeNode(10)` → recursively soft-deletes approvers 10, 11, 12.
5. Transaction 2 now has zero approvers. Its approval requirement is gone.

**Expected outcome:** `HTTP 200 true` — Transaction 2's entire approval tree is deleted without User A ever being the creator of Transaction 2.
