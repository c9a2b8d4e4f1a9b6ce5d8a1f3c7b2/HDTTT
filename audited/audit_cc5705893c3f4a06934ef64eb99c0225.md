### Title
TOCTOU Race Condition in `archiveTransaction` Allows Overwriting Terminal Transaction Status

### Summary

`archiveTransaction` in `TransactionsService` reads the transaction status, validates it, then issues an unconditional `UPDATE` with no status guard. A concurrent execution by the chain service can transition the same transaction to `EXECUTED` between the read and the write, causing the archive to silently overwrite the terminal status — corrupting the audit trail.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `archiveTransaction` follows a classic check-then-act pattern without atomicity:

**Step 1 — Read & Check (line 709–718):**
```typescript
const transaction = await this.getTransactionForCreator(id, user);

if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

**Step 2 — Unconditional Update (line 720):**
```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [2](#0-1) 

There is **no `WHERE status IN (...)` guard** on the `UPDATE`. Between the read and the write, the chain service's `_executeTransaction` can atomically transition the same row to `EXECUTED`:

```typescript
await this.transactionsRepo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: transactionStatus, executedAt, statusCode: transactionStatusCode })
  .where('id = :id AND status = :currentStatus', {
    id: transaction.id,
    currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
  })
  ...
``` [3](#0-2) 

After the chain service wins its conditional update, the creator's unconditional `UPDATE` still fires and stamps `ARCHIVED` over `EXECUTED`.

This contrasts directly with `cancelTransactionWithOutcome`, which correctly guards its update:
```typescript
.where('id = :id', { id })
.andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })
``` [4](#0-3) 

And with `processTransactionStatus`, which also uses a conditional update:
```typescript
.where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
``` [5](#0-4) 

`archiveTransaction` is the only mutating path that omits this guard.

### Impact Explanation

A transaction creator can cause the database to record `ARCHIVED` for a transaction that was successfully submitted to and executed on the Hedera network. The on-chain execution is permanent and unaffected, but the backend's audit record is corrupted: the `executedAt` timestamp, `statusCode`, `response`, and `receipt` fields written by `_executeTransaction` are all lost. Downstream consumers (notifications, history views, compliance exports) will see the transaction as archived rather than executed.

### Likelihood Explanation

The race window is narrow but structurally predictable. The chain service schedules execution via `addExecutionTimeout` at exactly `validStart + 5 seconds`:

```typescript
const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
``` [6](#0-5) 

A creator who knows their transaction's `validStart` can time an archive request to land in the window between the status check passing (`WAITING_FOR_EXECUTION`) and the chain service's conditional update completing. In a multi-pod deployment (Kubernetes/Helm is present in this repo), the API pod and chain pod run independently, making the race a real concurrent scenario rather than a theoretical one.

### Recommendation

Replace the unconditional update in `archiveTransaction` with a conditional update that only succeeds when the row is still in an archivable status, mirroring the pattern used in `cancelTransactionWithOutcome`:

```typescript
const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', {
    statuses: [
      TransactionStatus.WAITING_FOR_SIGNATURES,
      TransactionStatus.WAITING_FOR_EXECUTION,
    ],
  })
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

For `isManual` transactions the allowed statuses should be expanded accordingly, matching the existing check logic.

### Proof of Concept

1. Creator submits a transaction with `validStart = T` and `isManual = false`. Status becomes `WAITING_FOR_EXECUTION`.
2. At `T − 1 s`, creator sends `PATCH /transactions/:id/archive`. The service reads status = `WAITING_FOR_EXECUTION` → check passes.
3. At `T + 5 s`, the chain service fires `executeTransaction`. `_executeTransaction` submits to Hedera, receives a receipt, and runs:
   ```sql
   UPDATE transaction
   SET status = 'EXECUTED', executedAt = now(), statusCode = 22
   WHERE id = :id AND status = 'WAITING_FOR_EXECUTION'
   -- affected = 1 ✓
   ```
4. The creator's `archiveTransaction` resumes and runs:
   ```sql
   UPDATE transaction SET status = 'ARCHIVED' WHERE id = :id
   -- no status guard → affected = 1, overwrites EXECUTED
   ```
5. Database now shows `ARCHIVED`; `executedAt`, `receipt`, and `statusCode` are gone. The Hedera network retains the executed transaction permanently.
