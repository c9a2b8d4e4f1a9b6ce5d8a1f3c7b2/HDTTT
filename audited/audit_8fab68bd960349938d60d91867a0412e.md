### Title
Unbounded Per-Transaction Cryptographic Processing in `getTransactionsToSign` Enables Authenticated Resource Exhaustion DoS

### Summary
`GET /transactions/sign` fetches every transaction in the database matching a filter with no row limit, then performs a cryptographic signature-key computation (`userKeysToSign`) for each row in memory before applying pagination. An authenticated user who pre-populates the database with a large number of transactions can force the server to perform O(N) cryptographic operations per request, exhausting CPU and memory and degrading service for all users. This is the direct analog of the gas-tracking DoS: attacker-controlled persistent state (transaction rows) drives unbounded per-request resource consumption with no adequate cost control.

### Finding Description

**Root cause — no `take` limit on the initial DB query:**

In `getTransactionsToSign`, the repository `find()` call has no `take` parameter, so it returns every matching row:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
}); // ← no take / limit
```

Pagination is applied only *after* the full in-memory loop:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
return {
  items: result.slice(offset, offset + limit), // ← pagination here, too late
  ...
};
``` [1](#0-0) 

**Per-row cost — `userKeysToSign` → `computeSignatureKey`:**

For every fetched transaction, `userKeysToSign` calls `userKeysRequiredToSign`, which in turn calls `transactionSignatureService.computeSignatureKey`. This involves deserializing the Hedera SDK transaction bytes and computing the required signing key set — a non-trivial cryptographic and parsing operation repeated N times per HTTP request. [2](#0-1) 

**No cap on transaction creation:**

`createTransaction` / `createTransactions` enforce only uniqueness of `transactionId` and basic validity checks. There is no per-user quota or total-count limit, so an attacker can pre-load the database with an arbitrarily large number of transactions. [3](#0-2) 

**Secondary surface — `importSignatures` unbounded array:**

`POST /transactions/signatures/import` accepts `UploadSignatureMapDto[] | UploadSignatureMapDto` with no enforced size limit. The code itself contains the comment *"probably should limit this on the api side of things"*, confirming the gap. Each element triggers a DB lookup, `SDKTransaction.fromBytes()` deserialization, `validateSignature()`, and `addSignature()` — all per array entry. [4](#0-3) [5](#0-4) 

**Rate limiting does not bound per-request cost:**

The IP throttler enforces a per-second and per-minute request count, but does nothing to bound the work done inside a single request. One request against N=10,000 transactions is as damaging as N requests against 1 transaction, while consuming only 1 rate-limit token. [6](#0-5) 

### Impact Explanation

A single authenticated user can:
1. Pre-load the database with thousands of valid Hedera transactions (one-time cost, bounded only by Hedera network fees for valid transaction IDs).
2. Repeatedly call `GET /transactions/sign` (or `POST /transactions/signatures/import` with a large array).
3. Each call forces the API process to deserialize and cryptographically process every stored transaction before returning even a single page of results.

This causes sustained CPU and memory exhaustion on the API service, degrading or denying service to all other users. Because the attacker's transactions persist in the database, the attack is self-amplifying over time and requires no ongoing Hedera network interaction after setup.

**Severity: High** — authenticated, persistent, amplifying DoS against a shared service.

### Likelihood Explanation

- **Preconditions**: A valid registered account and the ability to submit Hedera transactions (standard user capability, no privilege required).
- **Attack cost**: Creating N transactions requires N valid Hedera transaction IDs (each with a unique `validStart`). This is cheap: a single Hedera account can generate thousands of distinct transaction IDs by varying the nanosecond `validStart` field.
- **Detectability**: The attack looks like normal API usage until the server degrades; no anomalous error codes are produced.
- **Existing mitigations**: None that bound per-request work. The IP throttler only limits request frequency, not request cost.

### Recommendation

1. **Apply `take` at the DB layer in `getTransactionsToSign`**: Pass `limit` directly to `repo.find({ take: limit, skip: offset, ... })` and perform the `userKeysToSign` filter only on the returned page. Accept that the `totalItems` count may require a separate, cheaper COUNT query.
2. **Enforce a per-user transaction quota**: Add a database-level or service-level cap on the number of active (non-terminal) transactions per user.
3. **Limit `importSignatures` array size**: Validate `body.length <= MAX_BATCH` (e.g., 100) at the controller before delegating to the service, as the inline comment already acknowledges is needed.
4. **Add per-request work budgets**: Consider a middleware that rejects requests whose estimated cost (e.g., number of IDs in a batch body) exceeds a configurable threshold.

### Proof of Concept

**Setup (one-time):**
```
for i in 1..10000:
  POST /transactions
  body: { transactionBytes: <valid Hedera tx with validStart = now + i nanoseconds>, ... }
```

**Attack loop (repeated, consumes 1 rate-limit token per call):**
```
GET /transactions/sign?page=1&limit=10
```

**Expected outcome:** Each request causes the server to fetch all 10,000 transaction rows, call `userKeysToSign()` 10,000 times (each involving `computeSignatureKey` + SDK deserialization), and return only 10 results. Server CPU spikes to 100% per request; concurrent legitimate users experience timeouts or 503 errors. [7](#0-6)

  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
