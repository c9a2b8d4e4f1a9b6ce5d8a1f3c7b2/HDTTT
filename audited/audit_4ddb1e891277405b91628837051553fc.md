### Title
Public Key Squatting Allows Malicious Organization Member to Permanently Block Legitimate User Key Registration

### Summary
The `uploadKey` function in `user-keys.service.ts` enforces global uniqueness of public keys across all users via an application-level check-then-insert pattern with no database-level locking or atomic guarantee. A malicious registered user who knows a target user's public key can register it under their own account first, permanently preventing the legitimate owner from registering that key. This is a direct analog to the front-running key registration vulnerability in the external report: instead of a blockchain mempool race, the attack exploits the globally-unique public key constraint in a web API context.

---

### Finding Description

**Root Cause**

In `back-end/apps/api/src/user-keys/user-keys.service.ts`, the `uploadKey` method performs a non-atomic check-then-insert: [1](#0-0) 

```
Step 1 – Read:  findOne({ where: { publicKey: dto.publicKey }, withDeleted: true })
Step 2 – Check: if (userKey.userId !== user.id) → throw ErrorCodes.PU
Step 3 – Write: repo.create(dto) + repo.save(userKey)
```

There is no database transaction, advisory lock, or `INSERT … ON CONFLICT` guard wrapping steps 1–3. The uniqueness invariant is enforced only in application code, not at the database level.

**Exploit Flow**

1. Attacker (a registered organization member, `userId=A`) learns the target user's (`userId=B`) Hedera public key. Public keys are observable from the Hedera network, from shared transaction bytes, or via the `getOwnerOfPublicKey` helper already present in the codebase. [2](#0-1) 

2. Attacker sends `POST /user/:userId/keys` with `{ publicKey: "<victim's key>" }` while authenticated as user A. Because no record exists yet, `findOne` returns `null`, the else-branch executes, and the key is saved with `userId = A`. [3](#0-2) 

3. Victim (user B) later calls `POST /user/:userId/keys` with the same `publicKey`. `findOne` now returns the existing record owned by A. The ownership check fires: [4](#0-3) 

   `userKey.userId (A) !== user.id (B)` → `throw new BadRequestException(ErrorCodes.PU)`

4. The victim's registration is permanently rejected. No recovery path exists in the API.

**Controller Entry Point**

The endpoint is authenticated (JWT + verified-user guards) but requires no admin privilege — any organization member can reach it: [5](#0-4) 

---

### Impact Explanation

- The victim cannot register their Hedera public key with the organization backend.
- Without a registered key, the victim cannot be set as a `creatorKey` for any transaction, cannot appear as a required signer, and cannot participate in any multi-signature coordination workflow — the core purpose of the tool.
- The `MAX_USER_KEYS` cap means the attacker can squatt up to that many keys per victim, potentially blocking all keys the victim might use.
- The state is permanent: there is no admin endpoint to forcibly reassign or evict a squatted key from another user's account. [6](#0-5) 

---

### Likelihood Explanation

- **Attacker precondition**: Must be a registered organization member (no admin role required). This is a realistic "malicious insider" scenario explicitly listed in `RESEARCHER.md` as an in-scope attacker profile.
- **Key discovery**: Hedera public keys are public by design and are embedded in every signed transaction visible on the network or shared within the organization workflow. The codebase itself exposes a `getOwnerOfPublicKey` query, confirming keys are treated as known identifiers.
- **Attack complexity**: A single authenticated HTTP POST. No timing precision required — the attacker only needs to act before the victim registers. [7](#0-6) 

---

### Recommendation

1. **Enforce uniqueness at the database layer**: Add a `UNIQUE` constraint on the `publicKey` column of the `user_key` table and handle the constraint-violation error gracefully (map it to `ErrorCodes.PU`). This eliminates the TOCTOU window entirely.

2. **Wrap check-and-insert in a serializable transaction**: If a DB-level constraint is not immediately feasible, wrap the `findOne` + `save` sequence in `this.repo.manager.transaction('SERIALIZABLE', ...)` so concurrent inserts are serialized.

3. **Bind public key to the registering user at insert time**: Consider whether a public key should truly be globally unique across all users, or whether the uniqueness constraint should be `(publicKey, userId)`. If a key can only be registered by its cryptographic owner (verified via a challenge-response), squatting becomes impossible regardless of uniqueness scope.

---

### Proof of Concept

**Setup**: Two registered, verified organization members — Alice (`userId=1`, JWT-A) and Bob (`userId=2`, JWT-B). Bob's Hedera public key is `302a300506032b6570032100<bob_pubkey_hex>` (observable from any Hedera transaction Bob has signed).

**Step 1 — Alice squats Bob's key**:
```http
POST /user/1/keys
Authorization: Bearer <JWT-A>
Content-Type: application/json

{ "publicKey": "302a300506032b6570032100<bob_pubkey_hex>" }
```
Response: `201 Created` — key saved with `userId=1`.

**Step 2 — Bob attempts to register his own key**:
```http
POST /user/2/keys
Authorization: Bearer <JWT-B>
Content-Type: application/json

{ "publicKey": "302a300506032b6570032100<bob_pubkey_hex>" }
```
Response: `400 Bad Request` — `ErrorCodes.PU` ("Public key already in use").

**Result**: Bob is permanently unable to register his own public key. He cannot be assigned as a creator or signer in any transaction. Alice has achieved a targeted denial-of-service against Bob's participation in the organization's multi-signature workflows, using only a single authenticated API call. [8](#0-7)

  async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
    await attachKeys(user, this.repo.manager);

    // Check if the user already has the maximum number of keys
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }

    // Find the userKey by the publicKey
    let userKey = await this.repo.findOne({
      where: { publicKey: dto.publicKey },
      withDeleted: true,
    });

    if (userKey) {
      // If the userKey found is owned by a different user,
      // or if the userKey has a non null hash or index that doesn't
      // match the hash or index provided
      // throw an error.
      if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
        throw new BadRequestException(ErrorCodes.PU);
      }
      // Set the hash and/or index (only if the current value is null)
      Object.assign(userKey, dto);
    } else {
      userKey = await this.repo.create(dto);
      userKey.user = user;
    }

    if (userKey.deletedAt) {
      await this.repo.recover(userKey);
    }
    return this.repo.save(userKey);
  }
