### Title
Balance Validation Bypassed in Organization Mode for HBAR Transfer Transactions

### Summary
In `TransferHbar.vue`, the guard that prevents submitting an HBAR transfer when a sender's balance is insufficient is explicitly disabled when the user is operating in Organization Mode. The `createDisabled` computed property conditions the balance check on `user.selectedOrganization === null`, meaning any organization-mode user can create, sign, and submit a transfer transaction whose debit amount exceeds the sender account's on-chain balance. The Hedera network will reject such a transaction at execution time, but the payer account still loses the transaction fee, and the organization backend records a submitted-but-failed transaction, creating a state inconsistency.

### Finding Description

**Root cause — `TransferHbar.vue` lines 45–51:**

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    (user.selectedOrganization === null && anyTransfersExceedingBalance.value) // ← only blocks in personal mode
  );
});
``` [1](#0-0) 

The short-circuit `user.selectedOrganization === null &&` means `anyTransfersExceedingBalance` is never evaluated when an organization is selected. The button is enabled regardless of whether the debit amount exceeds the sender's balance.

**`anyTransfersExceedingBalance` correctly detects the problem but is never consulted in org mode:**

```js
const anyTransfersExceedingBalance = computed(() => {
  for (const transfer of data.transfers) {
    if (transfer.amount.isNegative()) {
      const accountInfo = accountInfos.value[transfer.accountId.toString()];
      if (accountInfo &&
          transfer.amount.negated().toBigNumber().isGreaterThan(accountInfo.balance.toBigNumber())) {
        result = true; break;
      }
    }
  }
  return result;
});
``` [2](#0-1) 

**`preCreateAssert` also has no balance check**, so even the server-side-style assertion path does not catch this: [3](#0-2) 

**Exploit path:**
1. User logs into Organization Mode (`user.selectedOrganization !== null`).
2. User opens the Transfer HBAR form and adds a debit entry for an account whose balance is, e.g., 0 HBAR, with a transfer amount of 100 HBAR.
3. `createDisabled` evaluates to `false` (the balance guard is skipped); the "Sign & Submit" button is active.
4. The transaction is signed and submitted to the organization backend via `OrganizationRequestHandler.vue` → `submitTransaction`.
5. Approvers sign the transaction without any balance warning.
6. The Hedera network executes the transaction, returns `INSUFFICIENT_ACCOUNT_BALANCE`, and charges the payer account the transaction fee.
7. The organization backend records the transaction as submitted; the on-chain result is a failure. [4](#0-3) 

### Impact Explanation

- **Transaction fee loss**: The payer account is charged the Hedera network fee for every failed transaction. In a multi-approver workflow this can be triggered repeatedly.
- **System state inconsistency**: The organization backend records the transaction as submitted while the network records it as failed, diverging the two states.
- **Approver time waste / griefing**: A malicious or careless organization member can force all required approvers to sign a transaction that is guaranteed to fail on-chain.
- **No on-chain fund loss for the intended transfer** (the network rejects it), but fee loss and state divergence are concrete and measurable.

### Likelihood Explanation

Any authenticated organization-mode user can trigger this without any special privileges. The UI provides no warning about insufficient balance in organization mode (the `errorMessage` in `TransferHbarFormData.vue` is display-only and does not block submission). The condition is reachable in normal product usage — a user simply selects an organization, enters a transfer amount larger than the sender's balance, and clicks submit. [1](#0-0) 

### Recommendation

Remove the `user.selectedOrganization === null &&` guard so the balance check applies in both personal and organization modes:

```js
const createDisabled = computed(() => {
  return (
    !totalBalance.value.toBigNumber().isEqualTo(0) ||
    totalBalanceAdjustments.value > 10 ||
    totalBalanceAdjustments.value === 0 ||
    anyTransfersExceedingBalance.value   // enforce in all modes
  );
});
```

Additionally, add a balance assertion inside `preCreateAssert` so the check is enforced even if the UI guard is bypassed:

```js
const preCreateAssert = () => {
  // ... existing checks ...
  if (anyTransfersExceedingBalance.value) {
    throw new Error('Insufficient balance for one or more sender accounts');
  }
};
```

### Proof of Concept

1. Start the application and log in to an organization (`user.selectedOrganization !== null`).
2. Navigate to **Transfer HBAR**.
3. Add a "From" account whose on-chain balance is 0 HBAR (or any amount less than the intended transfer).
4. Set the transfer amount to exceed that balance (e.g., 100 HBAR from a 0-HBAR account).
5. Observe that the "Sign & Submit" button is **enabled** and no balance warning blocks submission.
6. Submit the transaction. The organization backend accepts it; approvers can sign it.
7. When the scheduled execution window arrives, the Hedera network returns `INSUFFICIENT_ACCOUNT_BALANCE`; the payer account is debited the transaction fee; the organization backend shows the transaction as submitted with a failed on-chain result. [1](#0-0) [2](#0-1) [3](#0-2)
