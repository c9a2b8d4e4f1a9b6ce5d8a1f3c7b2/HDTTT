### Title
Unbounded Recursive Approver Tree in `createTransactionApprovers` Causes Server-Side Resource Exhaustion DoS

### Summary
The `POST /transactions/:id/approvers` endpoint accepts a deeply nested, arbitrarily large approver tree with no depth or size limit enforced at the DTO or service layer. The `createTransactionApprovers` function processes this tree via unbounded recursion, issuing multiple database queries per node. A transaction creator (any authenticated user) can submit a crafted payload that triggers thousands to hundreds of thousands of sequential database queries within a single long-running transaction, exhausting the database connection pool and degrading or halting service for all users.

### Finding Description

**Root cause — no size/depth limit on the approver DTO:**

`CreateTransactionApproverDto.approvers` is a self-referential field validated only with `@ArrayMinSize(1)` (a lower bound) and no `@ArrayMaxSize()` or depth guard: [1](#0-0) 

`CreateTransactionApproversArrayDto.approversArray` similarly has no upper-bound constraint: [2](#0-1) 

**Root cause — unbounded recursive processing with per-node DB queries:**

`createTransactionApprovers` defines an inner async function `createApprover` that calls itself for every child in `dtoApprover.approvers`, with no recursion depth counter or total-node guard: [3](#0-2) 

Each invocation of `createApprover` issues up to **4–6 sequential database queries** (one `isNode` count, one `count(User)`, one recursive-CTE `getApproversByTransactionId`, one `insert`, and optionally one `findOne` + one `getRootNodeFromNode` CTE): [4](#0-3) 

All of this runs inside a single `dataSource.transaction(...)` block, holding a DB transaction open for the entire duration: [5](#0-4) 

**Exploit path:**

1. Attacker registers a normal user account (no privilege required).
2. Attacker creates a transaction — a standard user action.
3. Attacker sends `POST /transactions/:id/approvers` with a crafted payload: a tree of branching factor 5 and depth 5 produces 3,125 leaf nodes + 781 internal nodes, all within ~57 KB (well under the default 100 KB body limit).
4. The server processes ~14,000 sequential DB queries inside one open transaction.
5. The DB connection pool is saturated; all other requests queue or time out.

No rate-limiting `@Throttle` decorator is present on the approvers endpoint (unlike `remind-signers`): [6](#0-5) 

### Impact Explanation
A single crafted request from any authenticated user exhausts the PostgreSQL connection pool (configured at 3 connections in dev/prod values files) and holds a long-running transaction, causing complete service unavailability for all concurrent users. Increasing tree depth to 7 (branching factor 5) yields ~350,000 DB queries from a ~1.4 MB payload — achievable if the body-size limit is raised, which is common in production. The impact is **permanent service degradation** until the request times out or the server is restarted. [7](#0-6) 

### Likelihood Explanation
Any registered user can create a transaction and immediately call the approvers endpoint. No admin key, no leaked credential, and no special role is required. The attack requires a single HTTP request with a crafted JSON body. The absence of a `@Throttle` decorator on this endpoint means the attack can be repeated immediately after each timeout.

### Recommendation
1. **Enforce a maximum array size** on `approversArray` and `approvers` using `@ArrayMaxSize(N)` in the DTO (e.g., N = 20–50).
2. **Enforce a maximum recursion depth** by passing a depth counter into `createApprover` and throwing if it exceeds a safe threshold (e.g., 5).
3. **Add a `@Throttle` decorator** to the `POST /transactions/:id/approvers` endpoint, consistent with the pattern already used on `remind-signers`.
4. **Add a total-node budget** across the entire `createTransactionApprovers` call (e.g., reject if the flattened tree exceeds 100 nodes total).

### Proof of Concept

Send the following request as any authenticated transaction creator (replace `TOKEN` and `TX_ID`):

```
POST /transactions/TX_ID/approvers
Authorization: Bearer TOKEN
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        { "threshold": 1, "approvers": [
            { "threshold": 1, "approvers": [
                { "threshold": 1, "approvers": [
                    { "threshold": 1, "approvers": [
                        {"userId": 1}, {"userId": 2}, {"userId": 3}, {"userId": 4}, {"userId": 5}
                    ]},
                    ... (repeat 4 more times)
                ]},
                ... (repeat 4 more times)
            ]},
            ... (repeat 4 more times)
        ]},
        ... (repeat 4 more times)
      ]
    }
  ]
}
```

A branching-factor-5, depth-5 tree (3,906 total nodes, ~57 KB JSON) triggers approximately **14,000 sequential database queries** inside a single open transaction. Expected outcome: the API becomes unresponsive for all users until the transaction times out or the server is restarted.
