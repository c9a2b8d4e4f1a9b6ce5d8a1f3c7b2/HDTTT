### Title
Any Transaction Creator Can Delete Approvers Belonging to Transactions They Do Not Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by the URL parameter `transactionId`, but then deletes the approver identified by the separate URL parameter `id` without verifying that this approver actually belongs to `transactionId`. This is the same cross-resource authorization bypass class as the external report: ownership of one resource is validated, but the operation is performed on a different, unrelated resource.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the delete handler is:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` confirms the caller is the creator of the transaction at `transactionId`. However, `removeTransactionApprover(id)` then deletes the approver record identified solely by `id`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [1](#0-0) [2](#0-1) 

There is **no check** that `approver.transactionId === transactionId`. The `transactionId` URL parameter is used only to gate the creator check; the actual deletion operates on whatever approver `id` points to, regardless of which transaction it belongs to.

Contrast this with `updateTransactionApprover`, which explicitly enforces the cross-resource relationship:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The update path is protected; the delete path is not.

### Impact Explanation
Any authenticated, verified user who is the creator of **at least one** transaction can delete approvers from **any** transaction in the system. By supplying their own `transactionId` (one they created) in the URL and a victim's approver `id` as the path parameter, the attacker bypasses the approval workflow of transactions they do not own. This can:
- Remove required approvers from a transaction awaiting multi-party approval, allowing it to proceed without the intended authorization.
- Silently corrupt the approval tree of any transaction, undermining the integrity of the multi-signature governance model.

### Likelihood Explanation
The attacker only needs a valid JWT (any registered, verified user) and knowledge of a victim approver's numeric `id`. Approver IDs are sequential integers returned in API responses to participants of a transaction. An attacker who is an observer, signer, or approver of any transaction can enumerate IDs from those responses. No privileged access is required beyond being a normal user of the platform.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver's root transaction matches the `transactionId` URL parameter, mirroring the check already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover` and enforce the same `rootNode.transactionId !== transactionId` guard that `updateTransactionApprover` already uses. [4](#0-3) 

### Proof of Concept

1. Attacker registers and logs in as `userA`. Victim registers as `userB`.
2. `userB` creates transaction `T_victim` and adds approver record with `id = 42` (e.g., requiring `userC` to approve before execution).
3. `userA` creates any transaction `T_attacker` (they become its creator).
4. `userA` sends:
   ```
   DELETE /transactions/{T_attacker.id}/approvers/42
   Authorization: Bearer <userA_token>
   ```
5. The service confirms `userA` is the creator of `T_attacker` ✓, then deletes approver `42` (which belongs to `T_victim`) without any cross-check ✗.
6. `T_victim`'s approval requirement is silently removed. The transaction can now proceed without `userC`'s approval.
