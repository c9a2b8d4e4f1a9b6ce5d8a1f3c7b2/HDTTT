### Title
Hardcoded Cryptographic Secrets Committed to Version Control in `.env.test`

### Summary
The file `back-end/apps/api/.env.test` is committed directly to the repository (not as an example/template) and contains real, hardcoded values for `JWT_SECRET` and `OTP_SECRET`. These secrets govern JWT session token signing and TOTP-based two-factor authentication for the entire back-end API. Any party with read access to the repository can extract these values and forge authentication tokens or generate valid OTP codes.

### Finding Description
Unlike the `example.env` files (which use placeholder strings and are clearly templates), `back-end/apps/api/.env.test` is a live environment file committed to version control with concrete secret values:

```
JWT_SECRET=13123
OTP_SECRET=123
``` [1](#0-0) 

There is no `.gitignore` in the repository to exclude `.env*` files, meaning this file is fully tracked and its history is permanently retained in Git. [2](#0-1) 

The `JWT_SECRET` is consumed by the JWT strategy and the auth module to sign and verify all user session tokens. [3](#0-2) 

The `OTP_SECRET` is used by `otplib`'s TOTP engine to generate and verify one-time passwords for the two-factor authentication flow. [4](#0-3) 

The production and Kubernetes deployment configurations correctly pull these values from Kubernetes Secrets via `secretKeyRef`, but the `.env.test` file short-circuits this by providing trivially weak, committed values. [5](#0-4) 

### Impact Explanation
An attacker who reads the committed `JWT_SECRET=13123` can craft arbitrary, cryptographically valid JWT tokens for any user ID or role, achieving full authentication bypass — including admin impersonation — without knowing any password. An attacker who reads `OTP_SECRET=123` can independently generate valid TOTP codes, defeating the two-factor authentication gate entirely. Together, these two secrets allow complete account takeover of any user in the system.

### Likelihood Explanation
The file is committed to the repository with no `.gitignore` protection. Any developer, CI system, or external party with repository read access can retrieve it instantly. The secrets are also trivially weak (`13123`, `123`), making brute-force a secondary attack path even without repository access.

### Recommendation
- **Immediately** remove `back-end/apps/api/.env.test` from version control and purge it from Git history (e.g., using `git filter-repo`).
- Add `.env.test` and all `.env*` patterns to a root `.gitignore`.
- Replace the file with a `example.env.test` template containing only placeholder strings, consistent with the pattern used by `back-end/apps/api/example.env`.
- For CI/CD pipelines that require a real `.env.test`, inject secrets via environment variables from a secrets manager (e.g., GitHub Actions Secrets, HashiCorp Vault) at runtime, never from a committed file.
- Rotate both `JWT_SECRET` and `OTP_SECRET` immediately, as the committed values must be considered permanently compromised.

### Proof of Concept

1. Clone or browse the repository.
2. Read `back-end/apps/api/.env.test` — no special access required beyond repository read.
3. Extract `JWT_SECRET=13123`.
4. Craft a signed JWT:
   ```js
   const jwt = require('jsonwebtoken');
   const token = jwt.sign({ sub: 1, role: 'admin' }, '13123', { expiresIn: '365d' });
   console.log(token); // Valid admin session token accepted by the API
   ```
5. Use the token in an `Authorization: Bearer <token>` header against any protected API endpoint — the server will accept it as a legitimate admin session.
6. Separately, extract `OTP_SECRET=123` and use `otplib` to generate a valid TOTP code, bypassing the OTP verification step for any account. [1](#0-0) [6](#0-5)
