### Title
Silent Approval Drop in `approveTransaction()` When User Has No Keys

### Summary

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction()` function silently returns `false` when the authenticated approver has no registered keys, instead of throwing an error. This is a direct analog to the reported vulnerability: a critical state-changing action (recording an approval or rejection) is silently skipped with no error propagated to the caller, leaving the approver believing their action succeeded when it did not.

### Finding Description

The `approveTransaction()` function performs the following sequence:

1. Verifies the user is a designated approver — throws if not.
2. Verifies the user has not already approved — throws if they have.
3. Calls `attachKeys()` to load the user's keys from the database.
4. **Silently returns `false` if `user.keys.length === 0`** — no exception, no error code. [1](#0-0) 

The `attachKeys` utility only fetches keys if `user.keys` is empty or undefined: [2](#0-1) 

If the user has no keys in the database, `user.keys` is set to `[]` and the function returns `false` at line 567 — **before** any signature validation, **before** any database write, and **without** any error being thrown.

The full approval flow that is silently aborted: [3](#0-2) 

The caller receives `false` as the HTTP response body. No `BadRequestException` or `UnauthorizedException` is raised. The approver's intent (approve or reject) is permanently lost for that call.

### Impact Explanation

An approver who has no keys registered (or whose keys were deleted after being designated as an approver) can call the approval endpoint and receive a `false` response with no error. Their approval or rejection is never written to the `TransactionApprover` table. The transaction remains blocked in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely if that approver's vote was required to meet the threshold. The approver has no indication from the API that their action was silently dropped — the response is `false`, not an exception.

### Likelihood Explanation

This is reachable by any authenticated user who:
1. Is designated as a `TransactionApprover` for a transaction (a normal, non-privileged role).
2. Has no keys registered in the database at the time of the call.

This can occur legitimately when:
- A user is added as an approver before they register any keys.
- A user deletes all their keys after being designated as an approver.
- There is a TOCTOU window: `HasKeyGuard` (which checks key count at the guard layer) passes, but by the time `attachKeys` runs inside the service, the user's keys have been deleted. [4](#0-3) 

The guard and the service perform two separate database reads, creating a race window.

### Recommendation

Replace the silent `return false` with a thrown exception that clearly communicates the failure reason:

```typescript
// Before (line 567):
if (user.keys.length === 0) return false;

// After:
if (user.keys.length === 0) throw new BadRequestException(ErrorCodes.UHK); // or a new "no keys" error code
```

This mirrors the fix described in the external report: instead of silently skipping the action, revert/throw so the caller knows their intent was not fulfilled and can correct the situation.

### Proof of Concept

1. Register User A and designate them as an approver for Transaction T (threshold = 1, only approver).
2. Delete all of User A's registered keys via `DELETE /user/:userId/keys/:id`.
3. As User A, call `POST /transactions/:transactionId/approvers/approve` with any `dto` (e.g., `{ userKeyId: 1, signature: <any bytes>, approved: true }`).
4. **Expected (correct):** API returns a `400 Bad Request` with an error indicating no keys are available.
5. **Actual (buggy):** API returns HTTP 200 with body `false`. No approval is recorded. Transaction T remains blocked. User A has no indication their approval was dropped. [1](#0-0)

  /* Approves a transaction */
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }
