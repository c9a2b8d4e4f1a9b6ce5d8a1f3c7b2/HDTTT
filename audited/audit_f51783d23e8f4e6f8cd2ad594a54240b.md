### Title
Approver Signature Does Not Commit to the `approved` Intent, Allowing Transmitter to Flip Approval Decision

### Summary
In `approvers.service.ts`, the `approveTransaction` function verifies a user's signature against only the **transaction body bytes**. The `approved` boolean field — which determines whether the user is approving or rejecting the transaction — is **not included in the signed data**. A transmitter (the party calling the API on behalf of the user) can flip `approved: true` to `approved: false` (or vice versa) while the signature still passes verification, because the signature never commits to the user's intent.

### Finding Description

In `approvers.service.ts`, the `approveTransaction` function accepts an `ApproverChoiceDto` containing three fields: `userKeyId`, `signature`, and `approved`. The signature is verified as follows: [1](#0-0) 

The verification function `verifyTransactionBodyWithoutNodeAccountIdSignature` signs only the **transaction body bytes**: [2](#0-1) 

After the signature passes, the `approved` field from the DTO is stored directly into the database without any cryptographic binding: [3](#0-2) 

The signed message is `transactionBodyBytes` — it does **not** include the `approved` boolean. This means:
- A user signs the transaction body bytes (proving key ownership)
- The transmitter submits the request with `approved: false` even though the user intended to approve
- The signature check passes because `approved` was never part of the signed payload

### Impact Explanation
**Medium.** A user who delegates transaction submission to a third-party transmitter (e.g., an organization admin or relay service) cannot trustlessly express their approval intent via signature. The transmitter can silently flip the approval decision. In an organization workflow where approvals gate transaction execution, this breaks the trustless nature of the approval mechanism — a user's approval can be turned into a rejection, or a rejection into an approval, without the user's knowledge.

### Likelihood Explanation
**Medium.** This only matters when a user relies on a transmitter to submit the `approveTransaction` call on their behalf. Users who call the API directly are unaffected. However, in organization/multi-party workflows — which is the primary use case of this tool — delegated submission is a realistic and common scenario.

### Recommendation
Include the `approved` boolean in the data that the user signs, so the signature commits to both the transaction identity and the user's intent. For example, the signed payload should be:

```
signature = sign(transactionBodyBytes || approved)
```

The verification in `approveTransaction` should be updated to reconstruct this combined payload and verify against it, rather than verifying only `transactionBodyBytes`.

### Proof of Concept

1. User wants to **approve** transaction ID `42`. They sign `transactionBodyBytes` of transaction 42 with their private key and hand the signature to a transmitter.
2. The transmitter calls `POST /transactions/42/approvers/{approverId}/approve` with:
   ```json
   {
     "userKeyId": 7,
     "signature": "<user_signature_over_tx_body>",
     "approved": false
   }
   ```
3. `approveTransaction` in `approvers.service.ts` calls `verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)` — this passes because the signature is valid over the transaction body bytes.
4. The database is updated with `approved: false`, recording the user as having **rejected** the transaction, despite the user's intent being to approve it. [4](#0-3) [5](#0-4)

/* Get transaction body bytes without node account id */
export const getTransactionBodyBytes = (transaction: SDKTransaction) => {
  // @ts-expect-error - _makeTransactionBody is a private method
  const transactionBody = transaction._makeTransactionBody(null);
  return proto.TransactionBody.encode(transactionBody).finish();
};

/* Verify the signature of the transaction body without node account id */
export const verifyTransactionBodyWithoutNodeAccountIdSignature = (
  transaction: SDKTransaction,
  signature: string | Buffer,
  publicKey: string | PublicKey,
) => {
  const bodyBytes = getTransactionBodyBytes(transaction);

  /* Deserialize Public Key */
  publicKey = publicKey instanceof PublicKey ? publicKey : PublicKey.fromString(publicKey);

  /* Deserialize Signature */
  signature = typeof signature === 'string' ? decode(signature) : signature;

  try {
    return publicKey.verify(bodyBytes, signature);
  } catch (err) {
    console.log(err);
    return false;
  }
};
