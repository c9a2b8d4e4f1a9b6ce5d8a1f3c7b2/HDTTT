### Title
Unbounded O(n) Full-Table Scan in `getTransactionsToSign` Enables DoS via Transaction Flooding

### Summary
`getTransactionsToSign` fetches every non-terminal transaction in the system without a database-level row limit, then performs an expensive per-transaction computation (SDK deserialization + mirror-node signature-key resolution + DB query) for each row before applying pagination. Any verified user can flood the `transaction` table with non-terminal transactions via `POST /transactions`, causing `GET /transactions/sign` to time out or exhaust server resources for all users — a direct structural analog to the Nomad queue-iteration DoS.

### Finding Description

**Root cause — unbounded fetch with no `take` limit**

`getTransactionsToSign` in `transactions.service.ts` issues a `repo.find()` with no `take` parameter:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters by status, not by user
  relations: ['groupItem'],
  order,
});                      // ← no take/limit
``` [1](#0-0) 

The `whereForUser` predicate excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope the query to the requesting user's transactions — it returns every non-terminal transaction in the entire system. [2](#0-1) 

**O(n) per-transaction computation**

For every row returned, the service calls `userKeysToSign`, which:
1. Deserializes the full transaction bytes via `SDKTransaction.fromBytes`
2. Calls `transactionSignatureService.computeSignatureKey` (hits the Hedera mirror node)
3. Issues a `UserKey` DB query

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [3](#0-2) 

Pagination is applied only **after** the full scan completes:

```typescript
items: result.slice(offset, offset + limit),
``` [4](#0-3) 

**Attacker-controlled bloat path**

Any verified user with at least one registered key can call `POST /transactions` to create transactions. Created transactions enter `WAITING_FOR_SIGNATURES` status and remain there indefinitely if the required signers never sign. There is no per-user transaction creation limit or rate limit on this endpoint. [5](#0-4) 

**Exposed endpoint**

`GET /transactions/sign` is guarded only by JWT authentication and email verification — no admin role required. [6](#0-5) 

### Impact Explanation

An attacker who creates N transactions causes every subsequent call to `GET /transactions/sign` by any user to:
- Load N rows from PostgreSQL (with `groupItem` relation)
- Perform N SDK deserializations
- Issue N mirror-node HTTP calls (or N cache misses)
- Issue N `UserKey` DB queries

At a few thousand transactions the endpoint will exceed typical HTTP timeout thresholds (30 s), rendering the "transactions to sign" workflow completely unavailable for all users of the organization. Server memory and connection-pool exhaustion are secondary effects. The endpoint is marked `/* NO LONGER USED BY FRONT-END */` but remains live and callable.

### Likelihood Explanation

- **Attacker preconditions**: a valid account + at least one registered `UserKey` — both achievable through the normal self-registration flow.
- **Cost**: issuing N `POST /transactions` HTTP requests with valid (but otherwise arbitrary) `transactionBytes`. No privileged access, no leaked secrets.
- **Persistence**: transactions stay in `WAITING_FOR_SIGNATURES` indefinitely; the attacker does not need to remain active after the initial flood.
- **No existing mitigations**: no `@Throttle` decorator on `POST /transactions`, no per-user transaction count cap, no `take` limit in `getTransactionsToSign`.

### Recommendation

1. **Apply a database-level limit before the loop.** Pass `take` to `repo.find()` so the DB returns at most `limit` candidate rows. This requires restructuring the query to push the "does this user need to sign?" predicate into SQL (e.g., join on `transaction_signer` or use the existing recursive approver CTE pattern).

2. **Scope the query to the requesting user.** Add a join condition so only transactions where the user is a potential signer are fetched, mirroring the approach already used in `getTransactions`.

3. **Rate-limit `POST /transactions`** with NestJS `@Throttle` to bound the rate at which any single user can inflate the table.

4. **Consider removing the endpoint** if it is genuinely no longer used by the front-end (as the comment states), eliminating the attack surface entirely.

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
