### Title
Negative Threshold Bypass in Transaction Approver Validation Allows Approval Requirement to Be Circumvented

### Summary
The `createTransactionApprovers` and `updateTransactionApprover` functions in `approvers.service.ts` validate that a threshold is not greater than the number of approvers and not equal to zero, but they do not reject negative threshold values. A malicious transaction creator can supply `threshold: -1` (or any negative integer) to store an invalid threshold in the database. Because the approval-check logic evaluates `approvals.length >= threshold`, a negative threshold is always satisfied, meaning the approval gate is bypassed with zero actual approvals.

### Finding Description

**Root cause — missing lower-bound check on `threshold`**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the guard that is supposed to enforce a valid threshold is:

```typescript
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(dtoApprover.approvers.length));
``` [1](#0-0) 

The check only rejects values that are **too large** (`> approvers.length`) or **exactly zero**. A negative value such as `-1` satisfies neither condition (`-1 > N` is false; `-1 === 0` is false), so it passes silently and is persisted to the database.

The identical gap exists in the update path:

```typescript
if (
  approver.approvers &&
  (dto.threshold > approver.approvers.length || dto.threshold === 0)
)
  throw new Error(this.THRESHOLD_LESS_OR_EQUAL_APPROVERS(approver.approvers.length));
``` [2](#0-1) 

The `validateApprover` helper also does not catch negative values — it only rejects `null`, `NaN`, or `0`: [3](#0-2) 

**How the negative threshold is consumed**

The `isApproved` utility that drives approval-status evaluation reads:

```typescript
if (approvals.length >= (approver.threshold || approvals.length)) {
  return true;
}
``` [4](#0-3) 

Because `-1` is truthy in JavaScript, `approver.threshold || approvals.length` evaluates to `-1`. `approvals.length >= -1` is always `true` (array length is always ≥ 0), so the approval gate is permanently open regardless of how many approvers have actually signed.

The same threshold value stored in the database is used by the backend chain/scheduler service when it evaluates whether a transaction has accumulated sufficient approvals before execution.

**Exploit path**

1. Attacker (a normal authenticated user) creates a transaction.
2. Attacker calls `POST /transactions/:id/approvers` with body:
   ```json
   {
     "approversArray": [{
       "threshold": -1,
       "approvers": [{ "userId": 2 }, { "userId": 3 }]
     }]
   }
   ```
3. The service-level guard passes (−1 is neither > 2 nor === 0).
4. The record is inserted with `threshold = -1`.
5. The approval-check logic immediately evaluates the tree as approved (0 ≥ −1).
6. The transaction proceeds to execution without any of the designated approvers ever signing.

### Impact Explanation
A transaction creator can silently nullify the multi-party approval requirement that is the core organizational governance control of the tool. Transactions that are supposed to require N-of-M approvals from designated organization members execute immediately with zero approvals. This directly enables unauthorized movement of HBAR or other assets on the Hedera network, bypassing the multi-signature coordination the system is designed to enforce.

### Likelihood Explanation
The attack requires only a valid authenticated session as a transaction creator — no admin keys, no leaked credentials, no privileged role. The malicious payload is a single integer field in a standard API request. Any user who can create a transaction can exploit this immediately.

### Recommendation
Add an explicit lower-bound check (`threshold < 1`) to both the create and update paths, and enforce it at the DTO layer with a `@Min(1)` class-validator decorator on the `threshold` field in `CreateTransactionApproverDto` and `UpdateTransactionApproverDto`:

```typescript
// In the service guard (both create and update):
if (
  dtoApprover.approvers &&
  (dtoApprover.threshold < 1 || dtoApprover.threshold > dtoApprover.approvers.length)
)
  throw new Error(...);
```

```typescript
// In the DTO:
@IsInt()
@Min(1)
threshold?: number;
```

### Proof of Concept

**Request:**
```http
POST /api/transactions/42/approvers
Authorization: Bearer <creator-token>
Content-Type: application/json

{
  "approversArray": [{
    "threshold": -1,
    "approvers": [
      { "userId": 10 },
      { "userId": 11 }
    ]
  }]
}
```

**Expected (correct) behaviour:** `400 Bad Request` — threshold must be ≥ 1.

**Actual behaviour:** `201 Created` — approver tree stored with `threshold = -1`.

**Verification:** Query `GET /api/transactions/42/approvers` — the returned tree shows `threshold: -1`. The `isApproved` evaluation immediately returns `true` with zero approvals, and the chain service schedules the transaction for execution without waiting for any approver signatures.
