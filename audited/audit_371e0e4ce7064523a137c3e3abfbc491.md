### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side Resource Exhaustion (DoS)

### Summary
`TransactionsService.getTransactionsToSign` fetches every non-terminal transaction from the database with no row limit, then performs an expensive per-transaction async signature-key computation inside a sequential loop. Any authenticated user can trigger this endpoint (`GET /transactions/sign`) to exhaust server memory and CPU, degrading or crashing the API service. Pagination is applied only after the full in-memory scan, providing no protection.

### Finding Description

**Root cause — no database-level limit on the query:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues a `find` call with no `take` parameter: [1](#0-0) 

This loads every transaction whose status is not in the excluded set (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED) into memory in a single query.

**Unbounded loop with per-iteration async DB work:**

The result set is then iterated in full, calling `userKeysToSign` — which internally calls `TransactionSignatureService.computeSignatureKey` and performs additional entity-manager queries — for every transaction: [2](#0-1) 

**Pagination applied only after full scan:**

The `offset`/`limit` slice is applied to the already-computed in-memory result, not to the database query: [3](#0-2) 

**Contrast with the correctly paginated sibling method:**

`getTransactionsToApprove` correctly passes `skip: offset, take: limit` to the database query, so only the requested page is loaded: [4](#0-3) 

**Exposed HTTP endpoint:**

The endpoint is reachable by any authenticated, verified user — no admin role required: [5](#0-4) 

The controller comment `/* NO LONGER USED BY FRONT-END */` confirms the front-end has moved on, but the route remains live and unprotected.

### Impact Explanation

As the number of active (non-terminal) transactions grows, each call to `GET /transactions/sign` forces the API process to:

1. Load the entire active transaction table into Node.js heap memory.
2. Execute N sequential async DB round-trips (one `computeSignatureKey` call per transaction).

A single authenticated user repeatedly calling this endpoint can saturate the API server's memory and CPU, causing request timeouts for all other users and potentially crashing the NestJS process. Because the work scales linearly with the total transaction count — a quantity that only grows over the lifetime of the platform — the impact worsens over time without any attacker escalation.

### Likelihood Explanation

- **Attacker precondition:** A valid JWT for any non-admin user. Registration is a normal product flow.
- **Trigger:** A single HTTP GET to `/transactions/sign` (or a low-rate loop of such requests).
- **No special knowledge required:** The endpoint is documented in the Swagger spec and follows the standard REST pattern of the API.
- **Amplification:** The cost per request grows automatically as the organization accumulates transactions; no attacker action is needed to increase severity over time.

### Recommendation

Apply the database-level `skip`/`take` pagination to the `find` call, mirroring the pattern already used in `getTransactionsToApprove`:

```typescript
// In getTransactionsToSign — replace the current find call:
const [transactions, total] = await this.repo.findAndCount({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  skip: offset,
  take: limit,
});
```

Because `userKeysToSign` requires per-row async work that cannot be pushed into SQL, the total count must be obtained separately (e.g., via a lightweight `count` query) and the loop must only iterate over the page-sized slice. Alternatively, push the "does this user need to sign?" predicate into the database query itself (as a subquery or join), eliminating the in-memory loop entirely.

### Proof of Concept

**Preconditions:**
- An organization with N active (non-terminal) transactions (N grows naturally with usage).
- One valid user account (any role).

**Steps:**

1. Authenticate and obtain a JWT:
   ```
   POST /auth/login  →  { access_token: "<jwt>" }
   ```

2. Issue the unbounded scan request:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <jwt>
   ```

3. Observe that despite `size=10`, the server loads all N transactions from the database and calls `computeSignatureKey` N times before returning 10 results.

4. Repeat the request in a loop (or simply wait for the transaction count to grow). With thousands of active transactions, each request will consume hundreds of MB of heap and block the event loop for seconds, causing HTTP timeouts for concurrent users and eventual OOM crashes of the API pod.

**Expected outcome:** API response latency grows linearly with the total active transaction count; at sufficient scale, the service becomes unavailable to all users from a single authenticated attacker's requests.
