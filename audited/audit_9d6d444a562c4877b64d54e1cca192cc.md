### Title
Silent Failure in `computeSignatureKey` Returns Empty `KeyList`, Causing `hasValidSignatureKey` to Pass Vacuously and Promote Unsigned Transactions to Execution

### Summary

`TransactionSignatureService.computeSignatureKey` silently swallows all errors from mirror-node account/node lookups and returns an empty `KeyList` when those lookups fail. `hasValidSignatureKey` treats an empty `KeyList` as vacuously satisfied (`0 >= 0 → true`), so `processTransactionStatus` incorrectly promotes transactions to `WAITING_FOR_EXECUTION` and the chain service subsequently attempts to execute them on Hedera. Hedera rejects them with `INVALID_SIGNATURE`, permanently marking them `FAILED` and burning the fee-payer's HBAR.

### Finding Description

**Root cause — silent error suppression in `computeSignatureKey`**

Every key-gathering helper in `TransactionSignatureService` catches its own errors and returns without adding any key to the list: [1](#0-0) 

`addSigningAccountKeys` and `addReceiverAccountKeys` follow the same pattern. If the mirror node is unreachable, slow, or returns `null` for an account, the key is simply not pushed. The function returns a `KeyList` that may be completely empty.

**Root cause — vacuous truth in `hasValidSignatureKey`** [2](#0-1) 

When called with an empty `KeyList`:
- `keys = []`, `currentThreshold = 0`
- `key.threshold` is `undefined`, `keys.length = 0`
- Returns `0 >= (undefined || 0)` → `0 >= 0` → **`true`**

An empty required-key list is treated as "all requirements satisfied."

**Propagation through `processTransactionStatus`** [3](#0-2) 

When `computeSignatureKey` returns an empty `KeyList`, `isAbleToSign` is `true`, `smartCollate` is called, and if it returns non-null the transaction is promoted to `WAITING_FOR_EXECUTION`. This function is called both by the 10-second cron job in the chain service and immediately after every signature upload. [4](#0-3) 

**Same flaw repeated at execution time** [5](#0-4) 

The identical empty-KeyList check is performed again before execution. It passes for the same reason, so the transaction is submitted to Hedera.

### Impact Explanation

1. **Permanent transaction failure**: Hedera rejects the transaction with `INVALID_SIGNATURE`; the system marks it `FAILED`. The transaction cannot be retried.
2. **HBAR loss**: The fee-payer account is charged the Hedera transaction fee for a submission that was never valid.
3. **State integrity violation**: The transaction lifecycle state machine transitions to `WAITING_FOR_EXECUTION` and then `FAILED` without the required signatures ever being present — a direct invariant violation.

### Likelihood Explanation

The trigger condition is any transient or permanent failure of the mirror-node account-info lookup:

- **Mirror-node lag**: Newly created accounts are not immediately indexed; a transaction created seconds after account creation can hit this window.
- **Mirror-node downtime / rate-limiting**: The cron job runs every 10 seconds across all pending transactions; a brief outage or throttle response causes every in-flight transaction to be evaluated with an empty `KeyList`.
- **Deleted fee-payer account**: If the fee-payer account is deleted, the lookup returns `null`, the key is omitted, and the empty-list check passes.

No privileged access is required. A normal registered user who creates a transaction during a mirror-node hiccup is sufficient.

### Recommendation

1. **Distinguish "no keys required" from "key lookup failed"**: `computeSignatureKey` should propagate or surface lookup errors rather than silently omitting keys. At minimum, return a sentinel or throw so callers can abort.

2. **Guard against empty `KeyList` in `hasValidSignatureKey`**: An empty required-key list should not be considered satisfied. Add an explicit check:

   ```typescript
   if (key instanceof KeyList) {
     const keys = key.toArray();
     if (keys.length === 0) return false; // no requirements defined — treat as unsatisfied
     ...
   }
   ```

3. **Guard in `processTransactionStatus`**: Before promoting a transaction, verify that `signatureKey` is non-empty:

   ```typescript
   const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
   if (signatureKey.toArray().length === 0) continue; // skip — key resolution failed
   ```

### Proof of Concept

1. Register a user and create a `CryptoTransfer` transaction via the API.
2. Do **not** upload any signatures.
3. Temporarily make the mirror node unreachable (e.g., block the configured mirror-node URL at the network level, or point it to a non-existent host).
4. Wait up to 10 seconds for `handleTransactionsBetweenNowAndAfterThreeMinutes` to fire.
5. `updateTransactions` calls `processTransactionStatus`; `computeSignatureKey` returns `[]`; `hasValidSignatureKey([], emptyKeyList)` returns `true`; `smartCollate` is called; the transaction is updated to `WAITING_FOR_EXECUTION`.
6. `collateAndExecute` schedules execution; `getValidatedSDKTransaction` calls `hasValidSignatureKey` again with the same empty list — passes; the transaction is submitted to Hedera.
7. Hedera returns `INVALID_SIGNATURE`; the transaction is marked `FAILED`.
8. Restore the mirror node — the transaction is permanently `FAILED` and the fee-payer's HBAR has been consumed.

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    return currentThreshold >= (key.threshold || keys.length);
  } else if (key instanceof PublicKey) {
    return publicKeys.includes(key.toStringRaw());
  } else throw new Error(`Invalid key type`);
};
```
