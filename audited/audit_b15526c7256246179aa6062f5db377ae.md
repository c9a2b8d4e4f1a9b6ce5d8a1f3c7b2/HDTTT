### Title
Unbounded Array Input in Bulk Signature Upload Endpoints Enables Server-Side Resource Exhaustion

### Summary
The `POST /transactions/signers` and `POST /transactions/signatures/import` endpoints accept an array of `UploadSignatureMapDto` objects with no upper-bound size constraint. Each array element triggers CPU-intensive SDK transaction deserialization (`SDKTransaction.fromBytes()`), cryptographic signature validation, and database queries. The `validateAndProcessSignatures` method fans all items out concurrently via `Promise.all()`, multiplying resource consumption. A single authenticated user can exhaust server CPU, memory, and the database connection pool with one crafted request, causing service unavailability for all users.

### Finding Description

**Root cause — no array size limit in `transformAndValidateDto`:**

`back-end/libs/common/src/dtos/index.ts` lines 13–20 is the shared helper used by both bulk endpoints. It maps over the caller-supplied array without any length check:

```ts
export async function transformAndValidateDto<T extends object>(
  dtoClass: new (...args: any[]) => T,
  payload: T | T[],
): Promise<T[]> {
  const items = Array.isArray(payload) ? payload : [payload];
  const instances = items.map(item => plainToInstance(dtoClass, item));
  await Promise.all(instances.map(instance => validateOrReject(instance)));
  return instances;
}

**Entry point 1 — `POST /transactions/signers`:**

`SignersController.uploadSignatureMap` accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no size guard: [2](#0-1) 

**Entry point 2 — `POST /transactions/signatures/import`:**

`TransactionsController.importSignatures` has the identical pattern: [3](#0-2) 

**The DTO itself carries no `@ArrayMaxSize` decorator:** [4](#0-3) 

**Concurrent fan-out amplifies the damage:**

`validateAndProcessSignatures` wraps every item in `Promise.all()`, so N items produce N simultaneous `SDKTransaction.fromBytes()` calls (CPU-heavy protobuf deserialization), N cryptographic signature verifications, and N database round-trips: [5](#0-4) 

**Exploit flow:**
1. Attacker registers a normal user account (no privilege required).
2. Attacker sends `POST /transactions/signers` with a JSON body that is an array of thousands of `{ id: <valid_tx_id>, signatureMap: {...} }` objects.
3. `transformAndValidateDto` accepts the array without length validation.
4. `uploadSignatureMaps` → `loadTransactionData` → `validateAndProcessSignatures` fans all items out via `Promise.all()`.
5. The Node.js event loop is saturated by concurrent protobuf deserialization and cryptographic work; the PostgreSQL connection pool is exhausted by concurrent queries.
6. The API service becomes unresponsive for all other users until the request completes or the process OOMs.

The same path exists through `importSignatures` → `TransactionsService.importSignatures`, which also iterates over the unbounded array and calls `SDKTransaction.fromBytes()` and `validateSignature` per item: [6](#0-5) 

### Impact Explanation
A single authenticated user (no admin role needed) can render the API service unavailable for all organization members by submitting one oversized request. Transaction signing, execution, and all other API operations are blocked for the duration. If the process OOMs, the service crashes entirely. This is a permanent service disruption until the server is restarted or the request times out.

### Likelihood Explanation
Any verified user account is sufficient — the endpoints are protected only by `JwtAuthGuard` and `VerifiedUserGuard`. No leaked credentials, no admin access, and no special knowledge beyond the public API schema are required. The attack is a single HTTP POST with a large JSON array, reproducible with `curl` or any HTTP client.

### Recommendation
1. Add `@ArrayMaxSize(N)` (e.g., `N = 50`) to `UploadSignatureMapDto` when it is used as an array element, or enforce the limit inside `transformAndValidateDto`:
   ```ts
   if (items.length > MAX_BATCH_SIZE) {
     throw new BadRequestException(`Batch size exceeds limit of ${MAX_BATCH_SIZE}`);
   }
   ```
2. Replace `Promise.all()` in `validateAndProcessSignatures` with a bounded concurrency primitive (e.g., process items in chunks of a fixed size) to prevent fan-out even if a large batch somehow passes validation.
3. Apply the same fix to `importSignatures` in `TransactionsService`.

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
