### Title
Transaction Creator Can Modify Approver List and Threshold After Approvals Have Been Collected, Bypassing Multi-Signature Integrity

### Summary
The `updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover` functions in `approvers.service.ts` allow the transaction creator to modify the approver list, replace individual approvers, and change the approval threshold at any time — including after the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status and after other users have already submitted approvals or rejections. There is no transaction-status guard on any of these mutation paths.

### Finding Description

**Root cause — missing status check in all three approver-mutation paths:**

`updateTransactionApprover` (lines 367–531) verifies only that the caller is the transaction creator and that the approver belongs to the correct transaction. It never reads or checks `transaction.status`. [1](#0-0) 

`createTransactionApprovers` (lines 234–364) calls `getCreatorsTransaction` for ownership but performs no status check before inserting new approvers. [2](#0-1) 

`removeTransactionApprover` (lines 534–544) only verifies the approver exists; the controller calls `getCreatorsTransaction` for ownership but again no status check is present. [3](#0-2) 

The controller wires these three endpoints without any status guard: [4](#0-3) 

By contrast, `approveTransaction` correctly checks `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` before accepting a signature — but the mutation endpoints that control *who* must sign and *how many* must sign have no equivalent guard. [5](#0-4) 

**Exploit flows:**

1. **Rejection bypass** — Transaction T requires approval from User A and User B. User B rejects (`approved=false`). Creator calls `DELETE /transactions/T/approvers/<B-id>` to remove User B, then `POST` to add User C. User C approves. The rejection is silently erased and the transaction proceeds.

2. **Threshold reduction** — Transaction T has a threshold tree requiring 2-of-3 approvals. Only 1 approval has been collected. Creator calls `PATCH /transactions/T/approvers/<tree-id>` with `{ "threshold": 1 }`. The single existing approval now satisfies the requirement; the scheduler's next `updateTransactions` run will advance the transaction to execution. [6](#0-5) 

3. **Approver substitution** — Creator replaces an approver who has already signed with a different user. The `updateTransactionApprover` path for `userId` explicitly clears `signature`, `userKeyId`, and `approved`, forcing re-approval and potentially stalling or redirecting the transaction. [7](#0-6) 

### Impact Explanation

The multi-signature approval workflow is the core trust mechanism of the Hedera Transaction Tool. Approvers agree to sign (or reject) a specific transaction body. If the creator can silently remove rejecting approvers, reduce the required threshold, or swap in compliant approvers after the fact, the entire approval process is rendered meaningless. A malicious creator can force through any transaction that other organization members have explicitly rejected, causing unauthorized Hedera network transactions (account updates, token transfers, contract calls, etc.) to be submitted on behalf of the organization.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The attack requires only standard API calls (`PATCH`, `DELETE`, `POST` on `/transactions/:id/approvers`) that are already part of the documented workflow. No special tooling is needed. Any organization member who creates transactions can exploit this against their co-signers.

### Recommendation

Add a transaction-status guard at the top of `updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover`. Approver-list mutations should only be permitted while the transaction is in a pre-approval state (e.g., `NEW` / `DRAFT`). Once the transaction reaches `WAITING_FOR_SIGNATURES` or later, the approver list and threshold must be immutable:

```typescript
const transaction = await transactionalEntityManager.findOne(Transaction, {
  where: { id: transactionId },
});
if (
  transaction.status !== TransactionStatus.NEW // or whatever the initial state is
) {
  throw new BadRequestException('Approvers cannot be modified after signing has begun');
}
```

Apply the same guard in the controller's `removeTransactionApprover` handler before calling the service.

### Proof of Concept

**Setup:** Organization with creator (User C) and two approvers (User A, User B). Transaction T is created requiring both A and B to approve (threshold 2/2).

1. User A calls `POST /transactions/T/approvers/approve` — approves successfully.
2. User B calls `POST /transactions/T/approvers/approve` with `approved: false` — rejects.
3. Creator (User C) calls:
   ```
   DELETE /transactions/T/approvers/<B-approver-id>
   ```
   Response: `200 true`. No status check fires.
4. Creator calls:
   ```
   POST /transactions/T/approvers
   { "approversArray": [{ "userId": <User-D-id> }] }
   ```
5. User D calls `POST /transactions/T/approvers/approve` — approves.
6. Transaction now has 2 approvals (A + D). The scheduler's `updateTransactions` advances it to `WAITING_FOR_EXECUTION` and submits it to the Hedera network — despite User B's explicit rejection. [8](#0-7) [9](#0-8)
