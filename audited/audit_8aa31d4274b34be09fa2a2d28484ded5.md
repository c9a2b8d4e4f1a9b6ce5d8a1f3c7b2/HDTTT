### Title
`removeTransactionApprover` Emits Status Update with `null` entityId for Child Approvers, Silently Dropping Transaction State Re-evaluation

### Summary
In `removeTransactionApprover`, the notification is emitted using `approver.transactionId` as the `entityId`. For child approvers (those nested under a threshold tree, where `listId` is set and `transactionId` is `null`), this causes `emitTransactionStatusUpdate` to be called with `{ entityId: null }`. The notification service silently drops this event, meaning the transaction's approval status is never re-evaluated and no users are notified after a child approver is removed.

### Finding Description
The root cause is in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` at `removeTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
    return result;
}
```

The `TransactionApprover` entity uses a tree structure: root approvers have `transactionId` set and `listId = null`; child approvers have `listId` set to their parent's ID and `transactionId = null`. When `getTransactionApproverById` fetches a child approver, `approver.transactionId` is `null`.

The emitted event `{ entityId: null }` reaches `processTransactionStatusUpdateNotifications` in the notification service:

```typescript
for (const { entityId: transactionId } of events) {
    const transaction = transactionMap.get(transactionId); // transactionMap.get(null) → undefined
    if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping status-update notifications`);
        continue; // silently dropped
    }
    ...
}
```

The controller at the call site has the correct `transactionId` available but does not pass it to the service:

```typescript
@Delete('/:id')
async removeTransactionApprover(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Param('id', ParseIntPipe) id: number,
) {
    await this.approversService.getCreatorsTransaction(transactionId, user);
    await this.approversService.removeTransactionApprover(id); // transactionId not forwarded
    return true;
}
```

This is the direct analog of M-01: the wrong data field (`approver.transactionId`) is used instead of the correct one (the root transaction ID), causing the intended state update to be silently skipped.

### Impact Explanation
When a child approver is removed from a nested approval tree:
1. The `emitTransactionStatusUpdate` call fires with `entityId: null`.
2. The notification service silently skips the event.
3. The transaction's approval status is never re-evaluated by `processTransactionStatus`.
4. If removing the child approver causes the threshold to be automatically reduced (see `updateTransactionApprover` lines 423–427 which reduce parent threshold when children are removed), the transaction may now satisfy its approval requirements but will never advance to `WAITING_FOR_EXECUTION`.
5. Signers and observers receive no notification of the structural change.

The transaction can become permanently stuck in `WAITING_FOR_SIGNATURES` even after the approval conditions are met, constituting a state transition integrity failure.

### Likelihood Explanation
Any authenticated user can create a transaction and set up a nested approval tree. The attacker precondition is simply: be the creator of a transaction with at least one child approver (a threshold tree with nested approvers). The DELETE endpoint is reachable by any authenticated user who owns the transaction. No privileged access is required.

### Recommendation
Pass the `transactionId` from the controller into `removeTransactionApprover` and use it directly for the notification, rather than relying on `approver.transactionId`:

```typescript
// In the service:
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
    return result;
}

// In the controller:
await this.approversService.removeTransactionApprover(id, transactionId);
```

### Proof of Concept
1. Authenticate as User A and create a transaction (e.g., `POST /transactions`).
2. Create a threshold approval tree with a root approver and at least one child approver (`POST /transactions/:txId/approvers` with nested `approvers` array).
3. In the DB, confirm the child approver has `transactionId = null` and `listId = <root_id>`.
4. Call `DELETE /transactions/:txId/approvers/:childApproverId` as User A.
5. Observe in the notification service logs: `"Transaction null not found, skipping status-update notifications"`.
6. Confirm the transaction status is not re-evaluated and no notifications are dispatched to signers/observers.
7. If the removal caused the parent threshold to drop to the number of remaining approvers (all of whom have already approved), the transaction remains stuck in `WAITING_FOR_SIGNATURES` indefinitely despite approval conditions being satisfied.
