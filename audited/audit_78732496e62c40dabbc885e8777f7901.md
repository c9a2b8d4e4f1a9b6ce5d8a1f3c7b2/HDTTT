### Title
Creator of Any Transaction Can Remove Approvers Belonging to Any Other Transaction

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of `:transactionId`, but then passes the unrelated `:id` (approver ID) directly to `removeTransactionApprover()` without verifying that the approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can therefore soft-delete approvers from any other user's transaction by supplying their own `transactionId` and a victim's `approverId`.

### Finding Description

**Root cause — missing cross-resource ownership check**

The controller handler in `approvers.controller.ts`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);  // ① only checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);                // ② deletes approver `id` with no ownership check
  return true;
}
``` [1](#0-0) 

Step ① (`getCreatorsTransaction`) only confirms the caller is the creator of the URL-supplied `transactionId`. It does not touch the approver at all. [2](#0-1) 

Step ② (`removeTransactionApprover`) fetches the approver by `id` and soft-deletes it. There is no assertion that `approver.transactionId === transactionId`. [3](#0-2) 

By contrast, `updateTransactionApprover` — the sibling mutation — **does** perform this check:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [4](#0-3) 

The guard was added for updates but omitted for deletes, creating an inconsistent and exploitable authorization boundary.

**Exploit path**

1. Attacker registers as a normal user and creates any transaction (becomes a creator, satisfying the `getCreatorsTransaction` check for their own `transactionId`).
2. Attacker discovers a victim approver ID. Approver IDs are sequential integers. The attacker can also enumerate them by being an observer/signer on another transaction and calling `GET /transactions/:id/approvers`, which returns approver IDs.
3. Attacker calls:
   ```
   DELETE /transactions/{attacker_tx_id}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction(attacker_tx_id, attacker)` passes — attacker is creator of their own transaction.
5. `removeTransactionApprover(victim_approver_id)` soft-deletes the victim's approver with no further check.

### Impact Explanation

- **Unauthorized state mutation**: Any authenticated user can remove approvers from any other user's transaction, corrupting the approval workflow.
- **Approval bypass**: If a transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` and requires approvals before the creator considers it ready, the attacker can silently strip those approvers, allowing the transaction to proceed without the intended multi-party authorization.
- **Integrity violation**: The `emitTransactionStatusUpdate` call inside `removeTransactionApprover` will fire against the *victim's* transaction ID, potentially triggering downstream status recalculations on a transaction the attacker has no legitimate access to. [5](#0-4) 

### Likelihood Explanation

- **Preconditions**: The attacker only needs a valid account and to have created at least one transaction — both are normal user actions with no privilege requirement.
- **Approver ID discovery**: IDs are auto-incremented integers. An attacker who is an observer or signer on any transaction can retrieve approver IDs via `GET /transactions/:id/approvers`. Sequential IDs make brute-force trivial even without that.
- **No rate limiting or anomaly detection** is visible in the codebase for this endpoint.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), assert that the fetched approver belongs to the transaction the caller is authorized for:

```typescript
async removeTransactionApprover(id: number, expectedTransactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Mirror the guard already present in updateTransactionApprover
  const root = await this.getRootNodeFromNode(approver.id);
  if (root?.transactionId !== expectedTransactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into `removeTransactionApprover`, consistent with how `updateTransactionApprover` already handles this. [6](#0-5) 

### Proof of Concept

**Setup**:
- User A creates transaction `TX_A` (id = 1). User A is now a valid creator.
- User B creates transaction `TX_B` (id = 2) and adds an approver (approver id = 7, `userId = UserC`).

**Attack**:
```http
DELETE /transactions/1/approvers/7
Authorization: Bearer <User A's JWT>
```

**Expected result (correct)**: 403/401 — approver 7 does not belong to transaction 1.

**Actual result (vulnerable)**: 200 — approver 7 is soft-deleted from `TX_B`. User B's transaction now has no approver, and `UserC` is no longer required to approve it. The approval workflow for `TX_B` is silently corrupted by an unrelated user. [1](#0-0) [3](#0-2)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
