### Title
Soft-Deleted User Re-Registration Reuses Same ID, Granting New User Inherited Transaction Approver Rights

### Summary
When a user is soft-deleted and a new account is created with the same email address, the system reactivates the original database record — preserving the same primary key (`id`). Because `TransactionApprover` records are never cleaned up on user deletion, the new user silently inherits all pending approver roles that belonged to the deleted user, allowing them to approve transactions they were never authorized to approve.

### Finding Description

**Root cause — `createUser` reuses the deleted user's primary key:**

In `back-end/apps/api/src/users/users.service.ts`, `createUser` first queries for any user with the given email, including soft-deleted ones (`withDeleted = true`). If a soft-deleted record is found, it calls `updateUser` to clear `deletedAt` and reset the password — keeping the same `id`: [1](#0-0) 

This means a new registrant who happens to use a previously-deleted user's email address receives the **exact same numeric `id`** as the deleted user.

**Orphaned approver records survive user deletion:**

`removeUser` soft-deletes the user and their `UserKey` records, but does **not** touch `TransactionApprover` rows: [2](#0-1) 

`TransactionApprover` rows that reference the deleted user's `userId` remain active in the database: [3](#0-2) 

**New user inherits approval rights at `approveTransaction`:**

`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` fetches approvers by `transactionId` and filters by `userId === user.id`. Because the new user's `id` equals the old user's `id`, the orphaned approver rows match: [4](#0-3) 

The new user then registers a fresh `UserKey`, signs the transaction body with it, and the signature verification passes because it only checks that the submitted signature is valid for the submitted public key against the transaction body — it does not verify that the approver was ever explicitly authorized for this transaction: [5](#0-4) 

**End-to-end exploit path:**

1. Admin creates User A (`id=5`, `email=alice@org.com`) and adds them as an approver to Transaction T (a high-value HBAR transfer).
2. Admin removes User A — `removeUser` soft-deletes the user and their keys; `TransactionApprover` rows for Transaction T remain with `userId=5`.
3. Attacker (or new employee) registers with `alice@org.com`. `createUser` finds the soft-deleted record, clears `deletedAt`, and returns the same `id=5`.
4. New user registers a new `UserKey`, signs the transaction body, and calls `POST /transactions/:id/approvers/approve` with `approved: true`.
5. `approveTransaction` finds the orphaned approver row (`userId=5`), verifies the signature, and records the approval — the transaction advances toward execution.

### Impact Explanation

A new user who re-registers a previously-deleted email address can approve any pending transaction that the deleted user was assigned to. In an organization managing Hedera account transfers, file operations, or node management transactions, this constitutes **unauthorized state change / privilege escalation**: the new user can unilaterally satisfy an approval threshold and cause a transaction to execute that the organization never intended them to authorize.

### Likelihood Explanation

- No privileged access is required beyond normal user registration.
- Email addresses are frequently reused in organizations (employee turnover, role reassignment).
- The attacker only needs to know a previously-used email address, which may be guessable or discoverable from notification emails or public records.
- The window of exploitation is any time between user deletion and the transaction reaching a terminal status (`EXECUTED`, `EXPIRED`, `CANCELED`, etc.).

### Recommendation

Apply one or both of the following fixes:

1. **Nullify `userId` on orphaned approver rows when a user is deleted.** In `removeUser`, after soft-deleting the user, update all `TransactionApprover` rows with `userId = id` to set `userId = NULL` (or soft-delete them).

2. **Do not reuse the deleted user's primary key.** Instead of reactivating the old record in `createUser`, insert a new row with a fresh `id`. This prevents any foreign-key-based inheritance of the deleted user's roles.

Option 1 is the minimal fix; Option 2 is the more robust architectural change.

### Proof of Concept

```
1. POST /auth/register  { email: "alice@org.com", password: "pass1" }
   → User created: id=5

2. Admin: PUT /transactions/42/approvers  { userId: 5 }
   → TransactionApprover row: { id=1, transactionId=42, userId=5, deletedAt=null }

3. Admin: DELETE /users/5
   → User row: { id=5, deletedAt=<now> }
   → UserKey rows for userId=5: { deletedAt=<now> }
   → TransactionApprover row: { id=1, transactionId=42, userId=5, deletedAt=null }  ← UNCHANGED

4. POST /auth/register  { email: "alice@org.com", password: "pass2" }
   → createUser finds deleted record (withDeleted=true), calls updateUser({ deletedAt: null })
   → User row: { id=5, deletedAt=null }  ← SAME ID REUSED

5. New user registers a UserKey (id=99, publicKey=<attacker_key>)

6. New user signs transaction body with attacker_key → signature=<sig>

7. POST /transactions/42/approvers/approve  { userKeyId: 99, signature: <sig>, approved: true }
   → getApproversByTransactionId(42, userId=5) returns row id=1  ← ORPHANED ROW MATCHES
   → verifyTransactionBodyWithoutNodeAccountIdSignature passes
   → TransactionApprover updated: { userKeyId=99, signature=<sig>, approved=true }
   → Transaction 42 advances toward execution
```

  async removeUser(id: number): Promise<boolean> {
    const user = await this.getUser({ id });

    if (!user) {
      throw new BadRequestException(ErrorCodes.UNF);
    }

    // Soft-delete all user keys first
    await this.repo.manager.softDelete(UserKey, { userId: id });

    // Then soft-delete the user
    await this.repo.softRemove(user);

    return true;
  }
