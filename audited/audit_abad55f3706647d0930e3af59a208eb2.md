### Title
Unbounded Sequential Per-Transaction Computation in `getTransactionsToSign` Causes Server Resource Exhaustion

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then sequentially calls `userKeysToSign` for every transaction in a `for` loop. Each `userKeysToSign` call invokes `userKeysRequiredToSign`, which itself calls `attachKeys` (a DB query) and `computeSignatureKey` (an expensive async operation involving key resolution). With a large number of transactions in the system, a single authenticated API request causes O(N) sequential expensive operations, exhausting server CPU, memory, and DB connection resources.

### Finding Description

**Root cause — no pagination before the per-item loop:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip` — fetches every non-terminal transaction
});

for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
```

**Per-iteration cost — `userKeysToSign` → `userKeysRequiredToSign`:** [2](#0-1) 

`userKeysRequiredToSign` is called once per transaction: [3](#0-2) 

Inside `userKeysRequiredToSign`, `attachKeys(user, entityManager)` is called on every iteration (a DB round-trip each time), and then `keysRequiredToSign` is called, which invokes `transactionSignatureService.computeSignatureKey(transaction, showAll)` — an async operation that resolves the Hedera key structure for the transaction (potentially involving mirror-node queries): [4](#0-3) 

**Exploit path:**
1. Attacker (or any authenticated user) creates a large number of transactions (or the system accumulates them organically).
2. Attacker calls `GET /transactions/sign`.
3. The server fetches all N non-terminal transactions, then sequentially awaits `userKeysToSign` for each — performing at minimum 2 DB queries + 1 `computeSignatureKey` call per transaction.
4. With N = 10,000 transactions, the server performs ≥ 20,000 sequential DB queries + 10,000 key-resolution operations in a single request, holding the event loop and DB connection pool for the entire duration.

**The endpoint is still registered and reachable**, even though it is annotated as no longer used by the front-end: [5](#0-4) 

### Impact Explanation
A single authenticated HTTP request to `GET /transactions/sign` causes the server to perform O(N) sequential async DB queries and key-resolution operations with no upper bound. This degrades or fully blocks the NestJS API process for the duration of the request, starves the DB connection pool for other concurrent users, and can cause request timeouts or OOM crashes under realistic transaction volumes. The impact is **service unavailability / severe degradation** for all users sharing the backend.

### Likelihood Explanation
Any authenticated user — with no special privileges — can trigger this by issuing a single HTTP GET request. No crafted payload is required. The number of transactions grows organically as the platform is used; the vulnerability worsens over time. The endpoint requires only a valid session token, which every registered user possesses.

### Recommendation
1. **Apply a hard database-level limit** before the loop: add `take: limit` and `skip: offset` to the `repo.find` call so the query is paginated before any per-item processing begins.
2. **Move pagination filtering to the database layer**: push the "does this user need to sign?" predicate into the SQL query (as is already done in `getTransactions` with the recursive CTE approver check) rather than fetching all rows and filtering in application code.
3. **Deduplicate `attachKeys` calls**: call `attachKeys` once before the loop (it is already called at line 285) and remove the redundant call inside `userKeysRequiredToSign` when invoked from this context.
4. **Consider removing or rate-limiting the endpoint** since it is already marked as no longer used by the front-end.

### Proof of Concept

**Preconditions:** Authenticated user account; system contains a large number of non-terminal transactions (e.g., 5,000+).

**Steps:**
1. Obtain a valid JWT for any registered user.
2. Send: `GET /api/transactions/sign` with `Authorization: Bearer <token>`.
3. Observe: the server fetches all 5,000 transactions, then sequentially awaits `userKeysToSign` for each — performing ≥ 10,000 DB queries serially.
4. **Expected outcome:** The request takes tens of seconds to minutes; the DB connection pool is saturated; concurrent requests from other users time out or queue indefinitely.

**Quantified loop cost (analogous to the external report's calculation):**
- N = number of non-terminal transactions in the DB (unbounded)
- Per iteration: 1× `attachKeys` DB query + 1× `computeSignatureKey` (async, potentially mirror-node) + 1× `entityManager.find(UserKey, ...)` DB query
- Total DB operations: **3 × N** sequential round-trips per single API call, with no cap on N.

```typescript
export const userKeysRequiredToSign = async (
  transaction: Transaction,
  user: User,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
): Promise<number[]> => {
  await attachKeys(user, entityManager);
  if (user.keys.length === 0) return [];

  const userKeysRequiredToSign = await keysRequiredToSign(
    transaction,
    transactionSignatureService,
    entityManager,
    showAll,
    user.keys
  );

  return userKeysRequiredToSign.map(k => k.id);
};
```
