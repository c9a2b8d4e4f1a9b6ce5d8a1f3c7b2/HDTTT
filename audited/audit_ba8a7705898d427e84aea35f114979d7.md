### Title
Raw Database Error Messages Exposed in `POST /transactions/signatures/import` Response

### Summary
The `importSignatures` method in `transactions.service.ts` returns raw database error messages directly in the API response body when a batch save operation fails. This bypasses the global `BadRequestExceptionFilter` (which sanitizes thrown exceptions) because the error is embedded in the return value rather than thrown.

### Finding Description
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `importSignatures` method processes signature maps in batches. When a batch database update fails, the catch block constructs a message by concatenating a prefix with the raw `err.message` from the database/ORM layer, then stores it directly in the result map that is returned to the caller: [1](#0-0) 

```ts
const SAVE_ERROR_PREFIX = 'An unexpected error occurred while saving the signatures';
const message =
  err instanceof Error && err.message
    ? `${SAVE_ERROR_PREFIX}: ${err.message}`
    : SAVE_ERROR_PREFIX;

batch.forEach(u => results.set(u.id, { id: u.id, error: message }));
```

This result map is then returned directly from the service: [2](#0-1) 

The `SignatureImportResultDto` exposes the `error` field via `@Expose()`: [3](#0-2) 

The controller endpoint `POST /transactions/signatures/import` serializes and returns this DTO array directly to the client: [4](#0-3) 

The global `BadRequestExceptionFilter` only intercepts *thrown* `BadRequestException` instances and sanitizes them to generic messages: [5](#0-4) 

Because the error here is embedded in the return value (not thrown), the filter never runs, and the raw database error string — which may include PostgreSQL constraint names, column names, table names, or TypeORM internals — is sent to the client.

### Impact Explanation
A raw database error message can reveal internal schema details (table names, column names, constraint names), ORM internals (TypeORM query structure), and PostgreSQL-specific error codes. This information assists an attacker in mapping the database schema and crafting more targeted injection or manipulation payloads.

### Likelihood Explanation
The endpoint `POST /transactions/signatures/import` is reachable by any authenticated, verified user (guarded by `JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). A user can trigger the batch save failure path by submitting a malformed or oversized signature map payload that causes the raw SQL batch update to fail. No admin privileges are required.

### Recommendation
Replace the raw `err.message` concatenation with a static generic message and log the full error server-side only:

```ts
// Instead of:
const message = err instanceof Error && err.message
  ? `${SAVE_ERROR_PREFIX}: ${err.message}`
  : SAVE_ERROR_PREFIX;

// Use:
this.logger.error('Batch signature save failed', err);
const message = 'An unexpected error occurred while saving the signatures';
```

Apply the same pattern to any other location where `err.message` is embedded in response-bound objects rather than thrown exceptions.

### Proof of Concept
1. Authenticate as a valid organization user and obtain a JWT.
2. Submit a `POST /transactions/signatures/import` request with valid transaction IDs and a signature map that passes per-transaction validation but causes the batch SQL update to fail (e.g., by triggering a constraint violation or a malformed bytea value in the batch).
3. Observe the response body contains an `error` field of the form:
   ```json
   [{ "id": 42, "error": "An unexpected error occurred while saving the signatures: duplicate key value violates unique constraint \"transaction_pkey\"" }]
   ```
   The suffix after the colon is the raw PostgreSQL/TypeORM error message, exposing internal schema details.
