### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Per-Request Server Resource Exhaustion

### Summary
`GET /transactions/sign` fetches every non-terminal transaction in the database with no row limit, then serially calls `computeSignatureKey` on each one — an operation that deserializes transaction bytes and makes external Hedera mirror-node HTTP calls. Any authenticated user can trigger this path. As the number of active transactions grows (through normal use or deliberate creation), a single request causes unbounded CPU, memory, and outbound network consumption on the API server, degrading or hanging the service for all users.

### Finding Description

**Root cause — missing `take` on the database query**

`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` accepts `Pagination` parameters but never passes `take`/`skip` to the repository query:

```typescript
// lines 295-299 — no `take` or `skip`
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

Every other list endpoint (`getTransactions`, `getHistoryTransactions`, `getTransactionsToApprove`) correctly passes `skip: offset, take: limit` to the query. This one does not. [2](#0-1) 

**Unbounded per-row async work**

After loading the full result set, the code iterates over every row and calls `userKeysToSign` sequentially:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  ...
}
``` [3](#0-2) 

`userKeysToSign` → `computeSignatureKey` deserializes the transaction bytes and makes one or more outbound HTTP calls to the Hedera mirror node (fee-payer account, signing accounts, receiver accounts, node account) per transaction: [4](#0-3) 

**Exploit path**

1. Attacker registers as a normal user (no privilege required).
2. Attacker calls `POST /transactions` repeatedly to create N active transactions (status `WAITING_FOR_SIGNATURES`). Each transaction is valid and accepted by the server.
3. Attacker calls `GET /transactions/sign?page=1&size=10`.
4. The server loads all N transactions from the database, then calls `computeSignatureKey` N times, each making mirror-node HTTP calls. The `page`/`size` query parameters are accepted but only applied to the in-memory slice at line 313 — they have no effect on the database load or the loop.
5. With a large enough N, the request hangs the Node.js event loop for the duration, starving other requests. Repeated calls compound the effect.

**Endpoint is still live**

The controller comment explicitly notes the endpoint is no longer used by the front-end, but it remains publicly routed and authenticated:

```typescript
/* NO LONGER USED BY FRONT-END */
@Get('/sign')
getTransactionsToSign(...)
``` [5](#0-4) 

### Impact Explanation
A single authenticated user can cause the API server to spend unbounded CPU time deserializing transaction bytes and making unbounded outbound mirror-node HTTP calls per request. With enough active transactions, individual requests time out or block the event loop, causing service degradation or unavailability for all users of the organization. Because the pagination parameters are silently ignored at the database level, there is no client-side workaround.

### Likelihood Explanation
Any registered, verified user can create transactions and call this endpoint. No admin or privileged role is required. Transaction creation is a core workflow, so a large number of active transactions is a realistic steady-state condition in production organizations, not an adversarial precondition. The attacker does not need to send a flood of requests — a single request against a sufficiently populated database is sufficient to cause a timeout.

### Recommendation
Apply database-level pagination to the query, matching the pattern used by every other list endpoint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // Add these:
  skip: offset,
  take: limit,
});
```

Because the in-application filtering (checking which transactions actually require the user's keys) changes the effective result count, the `totalItems` field will be approximate unless a two-pass approach is used. Alternatively, push the signer-key check into the SQL query (as the newer `getTransactionNodes` path does) so that pagination can be applied correctly at the database level. If the endpoint is truly unused by the front-end, it should be removed entirely.

### Proof of Concept

1. Authenticate as a normal verified user; obtain a JWT.
2. Create 500 transactions via `POST /transactions` (each with valid `transactionBytes`, `creatorKeyId`, etc.).
3. Issue: `GET /transactions/sign?page=1&size=10`
4. Observe: the server loads all 500 rows from the database, then calls `computeSignatureKey` 500 times (each making mirror-node HTTP calls) before returning the 10-item page. Response time scales linearly with the number of active transactions. With enough transactions the request exceeds the server's HTTP timeout, returning a 504/connection reset to the caller while the server-side work continues consuming resources.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
