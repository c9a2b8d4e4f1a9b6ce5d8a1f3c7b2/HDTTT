### Title
Transaction Creator Can Lower Approver Threshold After Partial Approval, Bypassing Multi-Approval Requirement

### Summary
The `updateTransactionApprover` function in `approvers.service.ts` allows the transaction creator to modify the threshold of an approver group at any point during the approval lifecycle, including after approvals have already been collected. Because no transaction-status guard is enforced before the threshold write, a malicious creator can retroactively lower the required approval count to match the number of approvals already received, causing the transaction to be treated as fully approved without the originally-required number of sign-offs.

### Finding Description

**Root cause — missing status guard in `updateTransactionApprover`**

`updateTransactionApprover` enforces only three invariants before writing a new threshold value:

1. The caller is the transaction creator (`getCreatorsTransaction`, line 394).
2. The approver node is a threshold-type node (line 469).
3. The new threshold is `≥ 1` and `≤ number of children` (lines 473–477). [1](#0-0) 

There is **no check** that the transaction is in a pre-approval state (e.g., `WAITING_FOR_SIGNATURES`). The function proceeds to write the new threshold unconditionally: [2](#0-1) 

After the write, `emitTransactionUpdate` is fired, which triggers downstream status recalculation with the new, lower threshold. [3](#0-2) 

**Contrast with `approveTransaction`**, which correctly gates on transaction status: [4](#0-3) 

**Approval-status computation** (front-end, mirrored by back-end logic) treats a threshold node as approved when `approvals.length >= threshold`: [5](#0-4) 

Lowering the threshold to equal the count of already-collected approvals therefore immediately satisfies this condition.

**Exploit path**

1. Creator creates a transaction and sets up a threshold approver group: `threshold = 3`, children = [Alice, Bob, Carol].
2. Alice and Bob approve (`approved = true`, signatures stored).
3. Carol has not yet approved.
4. Creator calls `PATCH /transactions/{txId}/approvers/{groupId}` with body `{ "threshold": 2 }`.
5. The service writes `threshold = 2` with no status check.
6. The approval-status evaluator now sees `approvals (2) >= threshold (2)` → **approved**.
7. The transaction advances to execution without Carol's approval.

The same technique works with `createTransactionApprovers` (also missing a status guard): [6](#0-5) 

A creator can add a new approver node whose `userId` already has an existing approval record; the service copies the existing `signature`/`approved` values into the new node (lines 318–329), inflating the apparent approval count. [7](#0-6) 

**Secondary issue — `removeTransactionApprover` has no authorization check at all** [8](#0-7) 

The function accepts only an approver `id` with no `user` parameter, meaning any authenticated user can delete any approver from any transaction. This is a separate but related integrity failure.

### Impact Explanation

A malicious transaction creator can unilaterally reduce the number of approvals required for their own transaction after the approval round has begun. In an organizational multi-signature workflow, this defeats the entire purpose of the approval gate: a single actor (the creator) can force a transaction to execution that the organization's policy required multiple independent parties to authorize. Depending on the transaction type (e.g., large HBAR transfers, account key rotations), this can result in unauthorized fund movement or unauthorized account changes on the Hedera network.

### Likelihood Explanation

The attacker is the transaction creator — a normal, authenticated user with no elevated privileges beyond owning the transaction. The exploit requires only a standard authenticated API call (`PATCH /transactions/{id}/approvers/{approverId}`) with a crafted `threshold` value. No special tooling, timing, or collusion is needed. The window of opportunity is the entire duration between transaction creation and final execution.

### Recommendation

Add a transaction-status guard at the top of `updateTransactionApprover`, `createTransactionApprovers`, and `removeTransactionApprover` that rejects modifications once the transaction has entered any approval-active state (e.g., `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`):

```typescript
if (
  transaction.status === TransactionStatus.WAITING_FOR_SIGNATURES ||
  transaction.status === TransactionStatus.WAITING_FOR_EXECUTION
) {
  throw new BadRequestException('Cannot modify approvers after approval has started');
}
```

Additionally, `removeTransactionApprover` must accept a `user` parameter and call `getCreatorsTransaction` before proceeding, matching the pattern already used in `createTransactionApprovers` and `updateTransactionApprover`.

### Proof of Concept

**Preconditions**: Two organization users exist (Alice, Bob, Carol). Creator has a valid JWT.

```
# Step 1 – Create transaction with threshold=3 approver group
POST /transactions/{txId}/approvers
Body: { "approversArray": [{ "threshold": 3, "approvers": [
  { "userId": <alice> }, { "userId": <bob> }, { "userId": <carol> }
]}]}

# Step 2 – Alice and Bob approve (normal flow)
POST /transactions/{txId}/approvers/approve  (as Alice)
POST /transactions/{txId}/approvers/approve  (as Bob)

# Step 3 – Creator lowers threshold to 2 (malicious)
PATCH /transactions/{txId}/approvers/{groupNodeId}
Body: { "threshold": 2 }

# Expected: 400 Bad Request (transaction already in approval phase)
# Actual:   200 OK — threshold written, transaction now considered approved
#           Carol's approval is no longer required
```

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
