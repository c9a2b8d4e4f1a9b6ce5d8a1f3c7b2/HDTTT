### Title
Missing HBAR Transfer Amount and Recipients in Transaction Signing Confirmation Dialog

### Summary
The transaction signing confirmation dialog in `ConfirmTransactionHandler.vue` omits the actual HBAR transfer amounts and recipient accounts for `TransferTransaction` types. A user approving a transfer sees only the transaction type label and the max fee — not the financial substance of what they are signing. The same omission applies to the `initialBalance` field of `AccountCreateTransaction`.

### Finding Description
`ConfirmTransactionHandler.vue` is the modal shown to every signer immediately before they enter their password and cryptographically sign a transaction. The template branch that handles a `TransactionRequest` (lines 102–163) renders four fields unconditionally:

- Type of Transaction
- Transaction ID
- Valid Start
- Max Transaction Fee (only if set) [1](#0-0) 

The `transaction` computed property deserializes the raw bytes into a generic SDK `Transaction` object but never casts it to `TransferTransaction` to read `hbarTransfersList`. [2](#0-1) 

`TransferTransactionModel` already iterates `hbarTransfersList` to determine which accounts must sign, proving the data is accessible from the deserialized bytes: [3](#0-2) 

Yet none of that information (sender account, receiver account, amount per leg) is surfaced in the confirmation dialog. The same gap exists for `AccountCreateTransaction.initialBalance` — the HBAR seeded into a new account — which is set during creation: [4](#0-3) 

but never rendered in the confirmation modal.

### Impact Explanation
A signer in an organization workflow receives a transaction created by another party. The only information available at signing time is the confirmation dialog. Because the dialog shows only `"Transfer Transaction"` and a max fee, the signer has no way to verify — without leaving the dialog — how much HBAR is being moved or to which accounts. A malicious or compromised transaction creator can embed an arbitrarily large transfer (e.g., draining the payer account) and the signer's confirmation screen will look identical to a routine 1-HBAR transfer. The signature is irreversible once submitted to the Hedera network.

### Likelihood Explanation
The organization multi-signer workflow is the primary use-case of this tool. Transactions are routinely created by one role and signed by another. The signer is expected to rely on the confirmation dialog as their last checkpoint. The omission is present for every `TransferTransaction` and every `AccountCreateTransaction` with a non-zero `initialBalance` — both are among the most common transaction types in the application. No special attacker capability is required beyond being a legitimate transaction creator in the organization.

### Recommendation
In `ConfirmTransactionHandler.vue`, after deserializing the transaction, check its concrete type and render type-specific financial fields:

- For `TransferTransaction`: display each entry in `hbarTransfersList` (account ID + signed HBAR amount).
- For `AccountCreateTransaction`: display `initialBalance`.
- For `AccountAllowanceApproveTransaction`: display each entry in `hbarApprovals` (spender + amount).

The SDK objects are already available from the deserialized `transaction` computed value; only the template needs to be extended with type-guarded branches, mirroring the pattern already used in `TransferDetails.vue`: [5](#0-4) 

### Proof of Concept

1. In an organization, create a `TransferTransaction` that sends 10 000 HBAR from the payer account to an attacker-controlled account.
2. Submit it for signing by a second user.
3. The second user opens the confirmation dialog — it displays:
   - **Type of Transaction**: Transfer Transaction
   - **Transaction ID**: `0.0.X@...`
   - **Valid Start**: `<date>`
   - **Max Transaction Fee**: `2 ℏ`
4. No transfer amounts or recipient accounts appear anywhere in the dialog.
5. The signer clicks **Confirm**, enters their password, and the 10 000 HBAR transfer is signed and submitted.
