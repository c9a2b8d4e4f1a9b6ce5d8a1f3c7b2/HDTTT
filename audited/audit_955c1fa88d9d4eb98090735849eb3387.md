### Title
Admin Privilege Elevation Is Irreversible — No Mechanism to Demote Admin Users

### Summary
The `PATCH /auth/elevate-admin` endpoint unconditionally sets `admin: true` with no corresponding demote/revoke path. The UI explicitly warns that the action "cannot be undone." Once a user is elevated to admin, the only recourse to remove their privileges is to delete their account entirely — a more destructive action that removes all associated data. This mirrors the external report's "arbitrator cannot be revoked" pattern.

### Finding Description

**Root cause — one-way elevation endpoint:**

`back-end/apps/api/src/auth/auth.service.ts`, line 163–165:
```typescript
async elevateAdmin(userId: number): Promise<void> {
  await this.usersService.updateUserById(userId, { admin: true });
}
```
The call hardcodes `{ admin: true }`. There is no `demoteAdmin()` method, no `PATCH /auth/demote-admin` route, and no parameter to pass `false`. [1](#0-0) 

**Controller — only one direction exposed:**

`back-end/apps/api/src/auth/auth.controller.ts`, line 186–190:
```typescript
@Patch('/elevate-admin')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, AdminGuard)
async elevateUser(@Body() dto: ElevateAdminDto): Promise<void> {
  return this.authService.elevateAdmin(dto.id);
}
```
No `PATCH /auth/demote-admin` or equivalent route exists anywhere in the controller. [2](#0-1) 

**UI explicitly encodes irreversibility:**

`front-end/src/renderer/components/Contacts/ElevateContactModal.vue`, line 26:
```
Are you sure you want to assign this Contact as an admin? This action cannot be undone.
``` [3](#0-2) 

**UI shows no demote button:**

`front-end/src/renderer/components/Contacts/ContactDetails.vue`, line 212–219 — the "Assign Admin" button is only rendered when `!contact.user.admin`. There is no corresponding "Demote Admin" button rendered when `contact.user.admin` is `true`. [4](#0-3) 

**Frontend service — elevation only:**

`front-end/src/renderer/services/organization/auth.ts`, line 125–131:
```typescript
export const elevateUserToAdmin = (organizationServerUrl: string, id: number) =>
  commonRequestHandler(async () => {
    await axiosWithCredentials.patch(`${organizationServerUrl}/${authController}/elevate-admin`, {
      id,
    });
  }, 'Failed to assign user as admin');
```
No `demoteAdmin` service function exists. [5](#0-4) 

**Exploit flow:**
1. Admin A legitimately elevates User B to admin via `PATCH /auth/elevate-admin`.
2. User B's account is later compromised, or User B turns malicious.
3. Admin A attempts to demote User B — no such endpoint or UI action exists.
4. The only available remediation is `DELETE /users/:id` (soft-delete), which removes the account and all associated keys/data — a far more disruptive action.
5. During the window between compromise detection and account deletion, User B (or the attacker controlling their session) retains full admin capabilities: creating users, elevating more accounts to admin, accessing all transactions.

**Note on partial mitigation:** The `PATCH /users/:id` endpoint (guarded by `AdminGuard`) calls `updateUserById(userId, dto)` with `UpdateUserDto`. The e2e test description "update a user's admin status" and the test body `{ admin: true }` suggest the DTO may accept an `admin` boolean, which would allow raw API demotion. However, this path is not exposed in the UI, not documented as a demote mechanism, and the system design (dedicated endpoint + modal warning) communicates that elevation is permanent. This ambiguity does not eliminate the vulnerability in the intended application flow. [6](#0-5) 

### Impact Explanation
A compromised or malicious admin account cannot be demoted through any supported UI or dedicated API path. The only recourse — account deletion — is destructive and removes all user keys and associated data. During the remediation window, the attacker retains full admin access: they can register new users, elevate additional accounts to admin (creating persistent backdoors), and access all organization transactions. This constitutes a permanent privilege escalation with no least-privilege rollback path.

### Likelihood Explanation
Realistic. Admin elevation is a normal operational action in any multi-user organization. Insider threats and account compromise are standard risk scenarios. The inability to demote means any elevation decision is irreversible, increasing the blast radius of any future compromise. No special attacker capability is required beyond having been legitimately elevated — the vulnerability activates the moment elevation occurs and a reason to revoke arises.

### Recommendation
Add a dedicated `PATCH /auth/demote-admin` endpoint (or extend `ElevateAdminDto` to accept a boolean `admin` field) that sets `admin: false`, protected by `AdminGuard`. Add a corresponding "Demote Admin" button in `ContactDetails.vue` (shown when `contact.user.admin === true`). Update `ElevateContactModal.vue` to remove the "cannot be undone" language. Ensure at least one admin always remains (guard against self-demotion or last-admin demotion).

### Proof of Concept

**Setup:**
1. Deploy the backend. Create admin user A and regular user B.
2. Admin A calls `PATCH /auth/elevate-admin` with `{ id: B.id }` → User B is now admin.

**Trigger:**
3. User B's account is compromised. Admin A attempts to demote User B.
4. Search all routes: no `PATCH /auth/demote-admin` exists. The UI shows no "Demote" button for User B.
5. Admin A's only option: `DELETE /users/B.id` — which soft-deletes the account and all keys.

**Observed outcome:** Admin B's privileges cannot be revoked without destroying their account. If Admin A delays deletion, the attacker controlling User B can call `PATCH /auth/elevate-admin` with additional user IDs, creating new admin backdoors that persist even after User B is deleted.

**Expected outcome:** A `PATCH /auth/demote-admin` (or equivalent) should set `admin: false` for User B, revoking privileges while preserving the account.
