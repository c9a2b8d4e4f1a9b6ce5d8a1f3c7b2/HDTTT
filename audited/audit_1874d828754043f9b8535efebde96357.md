### Title
Wrong Field Assignment in `handleContractOnBlur` Corrupts SystemDelete/SystemUndelete Form State

### Summary
In `SystemDataFormData.vue`, the `handleContractOnBlur` handler — triggered when a user leaves the Contract ID input — emits an update that writes to `fileId` instead of `contractId`, and reads from `props.data.fileId` instead of `props.data.contractId`. This is the direct TypeScript analog of the Solidity `governance = _guardian` bug: a setter function that silently mutates the wrong field.

### Finding Description
In `front-end/src/renderer/components/Transaction/Create/SystemData/SystemDataFormData.vue`:

```typescript
function handleFileOnBlur() {
  emit('update:data', { ...props.data, fileId: formatAccountId(props.data.fileId) });   // ✓ correct
}

function handleContractOnBlur() {
  emit('update:data', { ...props.data, fileId: formatContractId(props.data.fileId) });  // ✗ wrong field, wrong source
}
```

The correct implementation should be:
```typescript
function handleContractOnBlur() {
  emit('update:data', { ...props.data, contractId: formatContractId(props.data.contractId) });
}
```

Two simultaneous wrong-field errors:
1. The emitted key is `fileId` instead of `contractId`
2. The source read is `props.data.fileId` instead of `props.data.contractId` [1](#0-0) 

This component is shared by both `SystemDeleteFormData` and `SystemUndeleteFormData`, which both use `SystemDataFormData` to render the File ID and Contract ID inputs. [2](#0-1) 

The `SystemData` type has two distinct fields: [3](#0-2) 

`formatContractId` and `formatAccountId` are distinct normalization functions: [4](#0-3) 

Note that `formatAccountId` has an extra guard (`!/^0+/.test(accountId)`) that `formatContractId` does not, meaning the two functions are not equivalent even for IDs that parse as both types.

### Impact Explanation
When a user fills in the Contract ID field and blurs it:

1. **`contractId` is never normalized** — the raw user input remains in state, bypassing the `formatContractId` normalization step entirely.
2. **`fileId` is silently re-processed** through `formatContractId(props.data.fileId)` instead of `formatAccountId(props.data.fileId)`, potentially producing a different normalized value for the File ID field.

The downstream `setSystemData` function uses both fields to build the on-chain transaction: [5](#0-4) 

The `createDisabled` guard in both SystemDelete and SystemUndelete checks `!isFileId(data.fileId) || !isContractId(data.contractId)`: [6](#0-5) 

If the blur-induced `fileId` corruption causes `isFileId(data.fileId)` to return `false`, the submit button is permanently disabled even when both IDs are valid — making SystemDelete/SystemUndelete transactions impossible to submit. Conversely, if a non-canonical `contractId` passes `isContractId` but fails SDK parsing at transaction build time, the transaction is built with malformed data.

### Likelihood Explanation
This triggers on every normal user interaction: filling in the Contract ID field and clicking away is the standard workflow for SystemDelete and SystemUndelete transactions. No attacker involvement is required — any user of the tool who attempts these transaction types will hit this bug.

### Recommendation
Fix `handleContractOnBlur` to update the correct field from the correct source:

```typescript
function handleContractOnBlur() {
  emit('update:data', { ...props.data, contractId: formatContractId(props.data.contractId) });
}
```

### Proof of Concept
1. Open the SystemDelete or SystemUndelete transaction form.
2. Enter a valid File ID (e.g. `0.0.111`) in the File ID field and blur — `handleFileOnBlur` correctly normalizes `fileId`.
3. Enter a valid Contract ID (e.g. `0.0.222`) in the Contract ID field and blur — `handleContractOnBlur` fires.
4. Observe: the emitted `update:data` event carries `{ fileId: formatContractId("0.0.111"), contractId: "0.0.222" }` — `contractId` is never normalized, and `fileId` was re-processed through the wrong formatter.
5. The parent component (`SystemDelete.vue` / `SystemUndelete.vue`) calls `Object.assign(data, newData)`, persisting the corrupted state into the reactive `data` object used to build the transaction. [7](#0-6)

export const formatAccountId = (accountId: string) => {
  if (isAccountId(accountId) && !/^0+/.test(accountId)) {
    return AccountId.fromString(accountId).toString();
  } else {
    return accountId;
  }
};

export const formatContractId = (contractId: string) => {
  if (isContractId(contractId)) {
    return ContractId.fromString(contractId).toString();
  } else {
    return contractId;
  }
};
