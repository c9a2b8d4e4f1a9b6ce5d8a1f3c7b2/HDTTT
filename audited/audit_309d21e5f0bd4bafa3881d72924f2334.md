### Title
TOCTOU Race Condition in `archiveTransaction` Allows Overwriting Terminal Transaction States

### Summary
`archiveTransaction` in `TransactionsService` reads the transaction status, validates it, then issues an unconditional `UPDATE` with no status guard in the `WHERE` clause. Between the read and the write, the chain service's cron jobs can transition the transaction to a terminal state (`EXECUTED`, `EXPIRED`, `FAILED`). The archive write then silently overwrites that terminal state with `ARCHIVED`, permanently corrupting the audit record.

### Finding Description

**Root cause — missing status guard in the UPDATE**

`archiveTransaction` follows a read-check-write pattern without atomicity:

```
back-end/apps/api/src/transactions/transactions.service.ts
```

Step 1 — read and check (line 709–718):
```typescript
const transaction = await this.getTransactionForCreator(id, user);
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) && !transaction.isManual
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

Step 2 — unconditional write (line 720):
```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
```

There is **no** `AND status IN (...)` guard in the `WHERE` clause. Any status present at write time is overwritten.

**Contrast with the safe pattern used elsewhere**

`cancelTransactionWithOutcome` (line 673–679) and `_executeTransaction` (line 187–196) both include a status guard in the `WHERE` clause so the update is a no-op if the status changed:

```typescript
// cancelTransactionWithOutcome
.andWhere('status IN (:...statuses)', { statuses: this.cancelableStatuses })

// _executeTransaction
.where('id = :id AND status = :currentStatus', {
  id: transaction.id,
  currentStatus: TransactionStatus.WAITING_FOR_EXECUTION,
})
```

`archiveTransaction` has no equivalent guard.

**Concurrent state changers that create the race window**

The chain service runs multiple cron jobs that mutate transaction status without coordination with the API service:

- `handleExpiredTransactions` — runs **every 10 seconds**, bulk-sets `EXPIRED` for transactions whose `validStart < now - 3 min`
- `handleTransactionsBetweenNowAndAfterThreeMinutes` — runs **every 10 seconds**, calls `collateAndExecute` which can set `EXECUTED` or `FAILED`

**Exploit sequence**

1. Transaction T is in `WAITING_FOR_EXECUTION` state; its `validStart` is within seconds of expiry.
2. Creator calls `PATCH /transactions/archive/:id`.
3. `getTransactionForCreator` returns status `WAITING_FOR_EXECUTION` — check passes.
4. Chain service's `handleExpiredTransactions` cron fires and sets T's status to `EXPIRED` (or `collateAndExecute` sets it to `EXECUTED`).
5. `archiveTransaction` issues `UPDATE transactions SET status = 'ARCHIVED' WHERE id = :id` — no status guard — overwrites `EXPIRED`/`EXECUTED` with `ARCHIVED`.

### Impact Explanation

A transaction that was successfully executed on the Hedera network (status `EXECUTED`) or that expired (status `EXPIRED`) or failed (status `FAILED`) has its terminal state permanently overwritten with `ARCHIVED`. Consequences:

- **Audit trail corruption**: The organization loses the authoritative record that the transaction was executed on-chain. The `executedAt` timestamp and `statusCode` remain in the DB but the status field no longer reflects reality.
- **Operational confusion**: Users and downstream integrations that rely on `status = EXECUTED` to confirm on-chain settlement will instead see `ARCHIVED`, potentially triggering duplicate submission attempts.
- **Irreversibility**: There is no recovery path; the terminal state is gone.

### Likelihood Explanation

The race window is narrow but mechanically guaranteed to exist. The chain service cron jobs run every 10 seconds in a separate NestJS process (the `chain` microservice) with no distributed lock coordinating with the `api` service. Any transaction near its `validStart` boundary is at risk on every archive call. A transaction creator who routinely archives transactions near their execution window will eventually trigger this. No attacker privilege beyond being the transaction creator is required — this is a normal authenticated API call.

### Recommendation

Add a status guard to the `UPDATE` in `archiveTransaction`, mirroring the pattern already used in `cancelTransactionWithOutcome`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

const updateResult = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  .execute();

if (!updateResult.affected || updateResult.affected === 0) {
  // Re-read and surface the actual current status to the caller
  const latest = await this.getTransactionForCreator(id, user);
  throw new ConflictException(
    `Cannot archive: transaction is now in status ${latest.status}`,
  );
}
```

Apply the same fix to `removeTransaction` (line 633), which has an identical unconditional `update` call with no status guard.

### Proof of Concept

**Setup**: Two concurrent async operations against the same transaction ID.

```
T=0ms  API:   GET /transactions/:id  → status = WAITING_FOR_EXECUTION  (check passes)
T=5ms  Chain: handleExpiredTransactions cron fires
               UPDATE transactions SET status='EXPIRED'
               WHERE status IN ('NEW','REJECTED','WAITING_FOR_EXECUTION','WAITING_FOR_SIGNATURES')
               AND validStart < now() - interval '3 minutes'
               → affected = 1  (transaction is now EXPIRED)
T=6ms  API:   UPDATE transactions SET status='ARCHIVED' WHERE id = :id
               → affected = 1  (no status guard — overwrites EXPIRED)
T=7ms  DB:    SELECT status FROM transactions WHERE id = :id
               → 'ARCHIVED'   ← terminal state lost
```

**Deterministic reproduction**: Set a transaction's `validStart` to `now() - 179 seconds` (one second before the 3-minute expiry threshold). Call `PATCH /transactions/archive/:id` and simultaneously trigger `handleExpiredTransactions`. The archive write will race the expiry write; whichever lands second wins. Because the archive write has no status guard, it always wins if it lands after the expiry write, producing a permanently incorrect `ARCHIVED` status for an expired transaction.   @Cron(CronExpression.EVERY_10_SECONDS, {
    name: 'status_update_expired_transactions',
  })
  async handleExpiredTransactions() {
    const result = await this.transactionRepo
      .createQueryBuilder()
      .update(Transaction)
      .set({ status: TransactionStatus.EXPIRED })
      .where('status IN (:...statuses) AND validStart < :before', {
        statuses: [
          TransactionStatus.NEW,
          TransactionStatus.REJECTED,
          TransactionStatus.WAITING_FOR_EXECUTION,
          TransactionStatus.WAITING_FOR_SIGNATURES,
        ],
        before: this.getThreeMinutesBefore(),
      })
      .returning(['id'])
      .execute();

    if (result.raw.length > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        result.raw.map(t => ({
          entityId: t.id,
        })),
      );
    }
  }
