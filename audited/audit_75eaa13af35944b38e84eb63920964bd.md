### Title
Any Authenticated User Can Delete Transaction Approvers Without Ownership Verification

### Summary
`removeTransactionApprover` in `approvers.service.ts` accepts only an approver `id` and performs no check that the calling user is the creator of the associated transaction. Every other mutating approver operation (`updateTransactionApprover`, `createTransactionApprovers`) enforces creator-only access via `getCreatorsTransaction`, but the delete path omits this check entirely. Any authenticated organization member can silently delete approvers from any transaction they did not create, bypassing the multi-signature approval workflow.

### Finding Description

**Root cause — missing user parameter and ownership check in `removeTransactionApprover`:** [1](#0-0) 

The function signature is `removeTransactionApprover(id: number)`. It fetches the approver by `id`, verifies it exists, and deletes it — with no `user` argument and no call to `getCreatorsTransaction`.

**Contrast with `updateTransactionApprover`, which correctly enforces creator ownership:** [2](#0-1) 

`updateTransactionApprover` takes a `user` parameter and calls `getCreatorsTransaction`, which throws `UnauthorizedException` if the caller is not the transaction creator: [3](#0-2) 

**Controller-level guards only enforce authentication, not ownership:** [4](#0-3) 

The controller applies `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — these confirm the caller is a logged-in, verified user, but impose no transaction-ownership constraint. Because `removeTransactionApprover` never receives the `user` object, no service-layer ownership check is possible.

**Exploit path:**
1. Attacker registers as a normal organization user (no admin required).
2. Attacker observes or guesses a valid `transactionApprover.id` (IDs are sequential integers; the attacker can enumerate them via their own transactions or via the `getApproversByTransactionId` endpoint on transactions they legitimately participate in).
3. Attacker sends `DELETE /transactions/{anyTransactionId}/approvers/{victimApproverId}` with their own valid JWT.
4. The service fetches the approver, finds it, and deletes it — no ownership check fires.

### Impact Explanation
Deleting an approver from a transaction under `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status directly corrupts the approval threshold tree. If the deleted node was a required approver or a threshold-list node, the transaction may:
- Become permanently unapprovable (locked state), or
- Silently satisfy its threshold with fewer approvals than intended, allowing execution without the required consent.

This constitutes unauthorized state mutation of the transaction lifecycle and can cause permanent corruption of the multi-signature approval workflow — a critical integrity failure.

### Likelihood Explanation
The attacker needs only a valid JWT for any organization account (no admin, no leaked secrets). Approver IDs are sequential integers and are exposed to any participant of their own transactions. The HTTP verb `DELETE` on the approvers sub-resource is a standard REST pattern; no special knowledge of internals is required. Likelihood is **high**.

### Recommendation
Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` (or the equivalent inline check) before performing the deletion, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Enforce creator-only access, same as updateTransactionApprover
  await this.getCreatorsTransaction(approver.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Update the controller's DELETE handler to pass `@GetUser() user: User` to the service call.

### Proof of Concept

**Preconditions:**
- Two accounts: `admin` (transaction creator) and `attacker` (any verified org user).
- Admin creates a transaction with an approver assigned (approver record ID = `42`).

**Steps:**
```http
# Step 1 – Attacker logs in and obtains JWT
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "eyJ..." }

# Step 2 – Attacker deletes admin's transaction approver
DELETE /transactions/7/approvers/42
Authorization: Bearer eyJ...
→ HTTP 200 (no error, approver deleted)
```

**Expected (correct) behavior:** HTTP 401 Unauthorized — only the transaction creator may delete approvers.

**Actual behavior:** HTTP 200 — approver is deleted regardless of caller identity, corrupting the approval tree of a transaction the attacker did not create.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
