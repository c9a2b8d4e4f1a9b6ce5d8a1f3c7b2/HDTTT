### Title
Stale Approval State Silently Carried Over When Same User Is Added to a New Approver Position

### Summary
In `createTransactionApprovers`, when a transaction creator adds a user to a new position in the approver tree, the service unconditionally copies that user's existing `signature`, `userKeyId`, and `approved` state from any prior approver record for the same transaction. This means a user's prior approval decision is silently reused at the new structural position without the user's knowledge or re-consent — a direct analog to the "approvals not cleared on transfer" class.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `createTransactionApprovers` function contains the following block: [1](#0-0) 

When a new `TransactionApprover` record is being inserted for a given `userId`, the code calls `getApproversByTransactionId(transactionId, dtoApprover.userId, ...)` to find any existing approver records for that user in the same transaction. If any exist, it copies `signature`, `userKeyId`, and `approved` verbatim into the new record.

The duplicate-check guard (`isNode`) only blocks creation when the new record has the **same** `listId`, `userId`, `threshold`, and `transactionId` as an existing record: [2](#0-1) 

If the user is being added to a **different** position (e.g., moved from root level to inside a threshold group, or added to a second threshold group), `isNode` returns `false`, the creation proceeds, and the stale approval state is copied in.

The `approved` field is the authoritative signal used by both the frontend `isApproved` utility and the backend status evaluation: [3](#0-2) 

The `TransactionApprover` entity stores `approved`, `signature`, and `userKeyId` as nullable columns that are only supposed to be populated after the user explicitly calls `approveTransaction`: [4](#0-3) 

### Impact Explanation
A transaction creator (no privileged role required — any authenticated user can create transactions) can:

1. Create transaction T with User A as a root-level approver.
2. Wait for User A to call `approveTransaction` (sets `approved=true`, `signature=<bytes>`, `userKeyId=<id>` on the root-level record).
3. Call `createTransactionApprovers` to add User A into a new threshold group (different `listId`). The `isNode` guard passes because `listId` differs.
4. The new approver record for User A inside the threshold group is immediately created with `approved=true` and the old signature — without User A taking any action.
5. The threshold group now counts User A as having approved, potentially satisfying the threshold and advancing the transaction status without the required number of independent approvals.

This violates the core invariant that each approver position must be independently approved by the designated user.

### Likelihood Explanation
The attacker is the transaction creator — a normal, unprivileged organization member. The steps require only standard API calls (`POST /transactions/:id/approvers`) that are available to any creator. No admin access, leaked credentials, or race conditions are needed. The scenario is fully deterministic and reproducible.

### Recommendation
Remove the approval-state carry-over block entirely from `createTransactionApprovers`. New approver records should always be created with `signature`, `userKeyId`, and `approved` set to `null`/`undefined`, regardless of whether the user has prior records for the same transaction. [1](#0-0) 

If preserving prior approval state across structural reorganizations is intentional, it must be gated behind explicit re-confirmation by the approver user, not silently inherited by the creator's structural edit.

### Proof of Concept

**Preconditions**: Two organization users — `creator` and `userA`. A pending transaction `T` with `userA` as a root-level approver.

1. `userA` calls `POST /transactions/T/approvers/choice` → `approved=true`. The root-level `TransactionApprover` row for `userA` now has `approved=true`, `signature=<S>`, `userKeyId=<K>`.
2. `creator` calls `POST /transactions/T/approvers` with body:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": <userA_id> }] }] }
   ```
   This creates a new threshold-group node (root, `listId=null`) and a child node for `userA` with `listId=<new_group_id>`.
3. `isNode` check: the child node has `listId=<new_group_id>` (not `null`), so it does not match the existing root-level record. Guard passes.
4. `getApproversByTransactionId(T, userA_id)` returns the existing root-level record with `approved=true`.
5. The new child record is inserted with `approved=true`, `signature=<S>`, `userKeyId=<K>` — copied from step 1.
6. The threshold group (1-of-1) is now immediately satisfied. `userA` never re-approved.
7. `GET /transactions/T` shows the new threshold group as fully approved, advancing the transaction status without `userA`'s re-consent.

```typescript
export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
```
