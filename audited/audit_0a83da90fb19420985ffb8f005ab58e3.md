### Title
`approveTransaction()` Reads Stale In-Memory Approver State Instead of Post-Update DB State, Suppressing `emitTransactionStatusUpdate` on Every Approval

### Summary
In `approvers.service.ts`, the `approveTransaction()` function fetches `userApprovers` from the database before writing the approval to the database. After the write, the post-write notification branch at line 614 evaluates `userApprovers.every(a => a.approved)` against the stale pre-write in-memory array. Because the in-memory records still carry `approved: null/undefined`, this condition is always `false` when `dto.approved = true`, so `emitTransactionStatusUpdate` is never emitted on an approval — only `emitTransactionUpdate` is. This is the direct analog of the external report: state is read from the wrong location (stale in-memory object instead of the updated database row).

### Finding Description

**Root cause — stale object read:**

`userApprovers` is populated at line 553 from a DB query, before the DB write:

```
const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);
const userApprovers = approvers.filter(a => a.userId === user.id);
```

The DB write happens at lines 599–610:

```
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
  .whereInIds(userApprovers.map(a => a.id))
  .execute();
```

Immediately after, the notification branch at line 614 evaluates the stale in-memory array:

```
if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

**Why the condition is always wrong when `dto.approved = true`:**

- `!dto.approved` → `false`
- `userApprovers.every(a => a.approved)` → evaluates the pre-write objects, where `approved` is still `null` or `undefined` → `false`
- Combined: `false || false` → `false`
- Result: `emitTransactionUpdate` is always called; `emitTransactionStatusUpdate` is **never** called when a user approves.

**When `dto.approved = false` (rejection):**
- `!dto.approved` → `true` → `emitTransactionStatusUpdate` is called correctly.

So the bug is one-sided: rejections emit the correct event; approvals never do. [1](#0-0) [2](#0-1) 

### Impact Explanation

`emitTransactionStatusUpdate` is the event that drives downstream status re-evaluation. The notifications service routes `TRANSACTION_STATUS_UPDATE` to `processTransactionStatusUpdateNotifications`, while `TRANSACTION_UPDATE` routes to `processTransactionUpdateNotifications` — a lighter path that does not trigger the same status-transition logic. [3](#0-2) 

When the last required approver approves a transaction, the system should emit `emitTransactionStatusUpdate` to signal that the approval threshold has been met and the transaction can advance (e.g., from `WAITING_FOR_SIGNATURES` to `WAITING_FOR_EXECUTION`). Because the stale-data check always suppresses this event, the status transition notification is never sent in real time for any approval. The transaction is only picked up by the chain service's cron job (every 10 seconds), causing a systematic delay in all approval-driven status transitions and breaking real-time client notifications for every approval action. [4](#0-3) 

### Likelihood Explanation

This fires on **every single approval** (`dto.approved = true`) submitted by any authenticated user who is a valid approver. No special attacker capability is required — any normal user exercising the standard approval workflow triggers the bug. The condition `userApprovers.every(a => a.approved)` can never be `true` against pre-write data when the user has not yet approved, so the wrong branch is taken 100% of the time on approval.

### Recommendation

Re-fetch the approver records from the database after the write, or mutate the in-memory objects to reflect the update before evaluating the notification branch:

```typescript
// After the DB update, re-read from DB:
const updatedApprovers = await this.getApproversByTransactionId(transactionId, user.id);

if (!dto.approved || updatedApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

Alternatively, mutate the in-memory array before the check:

```typescript
userApprovers.forEach(a => { a.approved = dto.approved; });
``` [5](#0-4) 

### Proof of Concept

1. Set up two users: User A (creator) and User B (approver).
2. User A creates a transaction and adds User B as an approver.
3. User B calls `POST /transactions/:id/approvers/approve` with `{ approved: true, ... }`.
4. Observe server logs / NATS messages: only `TRANSACTION_UPDATE` is published; `TRANSACTION_STATUS_UPDATE` is never published.
5. Repeat with `{ approved: false }`: `TRANSACTION_STATUS_UPDATE` is correctly published.
6. Confirm: the existing test at line 1271–1278 of `approvers.service.spec.ts` only asserts `emitTransactionStatusUpdate` was called after the *rejection* call (second call), masking the fact that the approval call (first call) never triggers it independently. [6](#0-5)
