### Title
Unbounded Recursive `createApprover` Call Allows Authenticated User to Crash the API Service via Stack Overflow

### Summary
The `createTransactionApprovers` function in `approvers.service.ts` uses an inner async function `createApprover` that calls itself recursively to process nested approver DTOs. No depth limit is enforced on this recursion. An authenticated user can submit a single HTTP request with an arbitrarily deep nested `approvers` JSON payload, exhausting the Node.js call stack and crashing the API process for all users.

### Finding Description
The root cause is in `back-end/apps/api/src/transactions/approvers/approvers.service.ts`. The inner function `createApprover` is defined at line 245 and calls itself at line 348 for every element in `dtoApprover.approvers`: [1](#0-0) 

There is no guard on recursion depth anywhere in this function or in the DTO validation layer: [2](#0-1) 

The `CreateTransactionApproverDto` accepts an `approvers` field that itself contains `CreateTransactionApproverDto` objects, allowing arbitrary nesting depth in the request body. The only validation performed is `validateApprover`, which checks for empty approvers but imposes no depth limit: [3](#0-2) 

**Exploit path:**
1. Attacker registers as a normal user (no privilege required beyond authentication).
2. Attacker creates a transaction (standard workflow).
3. Attacker sends a `POST /transactions/{id}/approvers` request with a JSON body containing thousands of levels of nested `approvers` arrays, e.g.:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "threshold": 1, "approvers": [{ ... }] }] }] }
   ```
4. The server processes the payload by recursively calling `createApprover` for each level.
5. At sufficient depth (Node.js default stack ~10,000â€“15,000 frames), a `RangeError: Maximum call stack size exceeded` is thrown.
6. Because this is thrown inside a `dataSource.transaction` callback and caught by the outer `catch (error)` block at line 359, the error is re-thrown as a `BadRequestException`. However, depending on the exact depth and async frame behavior, the uncaught stack overflow can crash the entire Node.js process. [4](#0-3) 

### Impact Explanation
A single crafted HTTP request from any authenticated user can crash the NestJS API service process. This causes complete service unavailability for all users until the process is restarted. Because the crash is triggered by a single request (not volumetric traffic), it is not a traditional DDoS and is not excluded by the SECURITY.md rule on "Impacts that only require DDoS." The impact is **permanent service freeze** until operator intervention.

### Likelihood Explanation
Any authenticated user who can create a transaction â€” which is a standard, unprivileged workflow â€” can trigger this. No admin keys, no leaked credentials, and no special role is required. The attack requires crafting a single JSON payload, which is trivial. The attacker precondition is: registered account + one created transaction.

### Recommendation
Add a `depth` parameter to `createApprover` and reject the request when it exceeds a safe threshold (e.g., 10):

```typescript
const createApprover = async (dtoApprover: CreateTransactionApproverDto, depth = 0) => {
  if (depth > MAX_APPROVER_DEPTH) {
    throw new Error('Approver tree exceeds maximum allowed depth');
  }
  // ...
  for (const nestedDtoApprover of dtoApprover.approvers) {
    await createApprover({ ...nestedDtoApprover, listId: approver.id }, depth + 1);
  }
};
```

Additionally, enforce the same depth limit at the DTO validation layer (e.g., via a custom class-validator decorator) so the request is rejected before any database transaction is opened.
