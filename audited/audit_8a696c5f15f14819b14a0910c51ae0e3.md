### Title
Unbounded Full-Table Scan with Per-Row External Network Calls in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`TransactionsService.getTransactionsToSign` fetches **every** non-terminal transaction from the database without any row limit, then iterates over the entire result set making external mirror-node HTTP calls per transaction to compute signature keys. Pagination is applied only after the full loop completes. Any authenticated, verified user can repeatedly invoke `GET /transactions/sign` to exhaust server CPU, memory, and outbound HTTP connections, degrading or crashing the API service.

### Finding Description

**Root cause — unbounded DB fetch with no `take` limit:**

In `getTransactionsToSign`, the TypeORM `find` call at line 295 omits `take`/`skip`, so it returns every row in the `transaction` table that is not in a terminal status:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip`
});
``` [1](#0-0) 

**Per-row external network calls inside the loop:**

For every row returned, the code calls `this.userKeysToSign(transaction, user)`:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` → `transactionSignatureService.computeSignatureKey(transaction)`, which makes outbound HTTP calls to the Hedera mirror node for every account involved in each transaction: [3](#0-2) [4](#0-3) 

**Pagination applied post-hoc — never limits the work:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),  // ← slicing happens AFTER full loop
  page,
  size,
};
``` [5](#0-4) 

The `PaginationParams` decorator caps `size` at 100, but this only limits the response payload — it does not limit the DB query or the loop: [6](#0-5) 

**Exposed endpoint — still reachable by any authenticated user:**

```typescript
@Get('/sign')
getTransactionsToSign(
  @GetUser() user: User,
  @PaginationParams() paginationParams: Pagination,
  ...
)
``` [7](#0-6) 

The controller comment says "NO LONGER USED BY FRONT-END" but the route is fully active and protected only by JWT + `VerifiedUserGuard` — any registered, verified user can call it. [8](#0-7) 

### Impact Explanation

- **CPU/memory exhaustion**: With N active transactions in the system, every call to `GET /transactions/sign?page=1&size=1` forces the server to load all N rows into memory and process each one.
- **Outbound connection exhaustion**: Each transaction triggers multiple async mirror-node HTTP calls. With thousands of transactions, a single request spawns thousands of concurrent outbound connections.
- **API unavailability**: Repeated requests from one or more authenticated users can saturate the Node.js event loop and the mirror-node HTTP client pool, causing timeouts and 5xx errors for all users.
- **Cascading failure**: The chain service and notifications service share infrastructure; a saturated API service degrades the entire platform.

### Likelihood Explanation

- **Attacker precondition**: Only a valid JWT token from a registered, verified user is required — no admin or special role.
- **Trigger**: A single HTTP GET request. Automation (e.g., a loop sending requests every second) amplifies the impact trivially.
- **Realistic scenario**: A disgruntled organization member, a compromised user account, or a malicious API client can exploit this without any privileged access.
- **Amplification**: The cost scales with the total number of non-terminal transactions in the database, which grows over time in normal operation.

### Recommendation

Apply the database-level limit **before** the loop, not after:

```typescript
// Apply take/skip at the DB query level
const [transactions, total] = await this.repo.findAndCount({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  skip: offset,
  take: limit,
});

// Then filter by keysToSign — but note: this changes semantics
// A better approach: push the key-check into a SQL subquery or
// use a separate endpoint for the "do I need to sign?" check per transaction.
```

The cleanest fix is to push the `userKeysToSign` filter into the database query (joining on `transaction_signer` and the user's public keys), eliminating the in-process loop and all per-row mirror-node calls from this endpoint entirely. Alternatively, add a hard cap on the number of rows processed per request and return a cursor-based pagination token.

### Proof of Concept

**Preconditions**: The organization has ≥ 500 active (non-terminal) transactions. Attacker has a valid JWT for any verified user account.

**Steps**:
1. Obtain a valid JWT: `POST /auth/login` with any verified user credentials.
2. Send repeated requests:
   ```bash
   while true; do
     curl -s -H "Authorization: Bearer $TOKEN" \
       "https://api.example.com/transactions/sign?page=1&size=1" &
   done
   ```
3. Each request causes the server to:
   - Load all 500+ transaction rows from PostgreSQL (with `groupItem` relation).
   - Call `computeSignatureKey` for each → multiple mirror-node HTTP calls per transaction.
   - Allocate all results in memory before slicing to 1 item.
4. **Expected outcome**: API response times degrade to timeouts; other users receive 5xx errors; Node.js process memory climbs until OOM or the event loop stalls.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
