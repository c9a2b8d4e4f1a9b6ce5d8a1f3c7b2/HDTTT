### Title
IDOR in `removeTransactionApprover`: Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

### Summary
The `removeTransactionApprover` service method does not verify that the approver being deleted belongs to the transaction specified in the request. The controller enforces that the caller is the creator of `transactionId`, but then passes the unrelated approver `id` directly to the service, which removes it without any ownership check. A malicious user who is the creator of any transaction can delete approvers from transactions they do not own, bypassing approval requirements on those transactions.

### Finding Description

**Vulnerability class:** Authorization bypass / IDOR (analogous to the external report's "incorrect assumption about parameter scope" — the code assumes the approver `id` belongs to the transaction `transactionId` without verifying it, just as `handleAcrossMessage` assumed `amount` represented the full fill without verifying completeness).

**Root cause — controller:**

In `approvers.controller.ts`, the `DELETE /:id` handler first verifies the caller is the creator of `transactionId`, then calls `removeTransactionApprover(id)` with the raw approver `id` from the URL:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ✓ checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ✗ no check that id ∈ transactionId
  return true;
}
``` [1](#0-0) 

**Root cause — service:**

`removeTransactionApprover` fetches the approver by `id` alone (no `transactionId` filter) and deletes it unconditionally:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id); // fetches ANY approver by id
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);          // deletes it, no ownership check
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

`getTransactionApproverById` queries only by `id`, with no `transactionId` constraint: [3](#0-2) 

By contrast, `updateTransactionApprover` correctly validates `rootNode.transactionId !== transactionId` before proceeding: [4](#0-3) 

**Exploit path:**

1. Attacker registers as a normal user and creates any transaction (Transaction A). This makes them the creator of A.
2. Attacker becomes an observer or signer of victim's Transaction B (or enumerates sequential approver IDs).
3. Attacker calls `GET /transactions/B/approvers` to retrieve approver IDs for Transaction B.
4. Attacker calls `DELETE /transactions/A/approvers/{approver_id_from_B}`.
5. The controller verifies the attacker is creator of A (passes). The service deletes the approver belonging to B without any cross-check.
6. The approval requirement on Transaction B is silently removed.

### Impact Explanation

Removing an approver from a transaction eliminates that approval gate. If the transaction required N-of-M approvals and the attacker removes enough approvers, the transaction can proceed to execution without the intended organizational oversight. Since approvers gate Hedera network transactions (account updates, fund transfers, node operations), this allows an attacker to cause unauthorized Hedera transactions to be executed by bypassing the approval workflow entirely. [5](#0-4) 

### Likelihood Explanation

The attacker only needs to be a registered user on the organization server and creator of at least one transaction (a normal workflow action). Approver IDs are sequential integers and are exposed via the `GET /transactions/{id}/approvers` endpoint to any user with visibility of that transaction (creator, signer, observer, or approver). No privileged access, leaked credentials, or cryptographic breaks are required. [6](#0-5) 

### Recommendation

Add an ownership check inside `removeTransactionApprover` (or before calling it) to verify that the approver being deleted belongs to the transaction specified in the request. The simplest fix mirrors what `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Verify the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, and update the controller accordingly.

### Proof of Concept

**Setup:**
- User Alice creates Transaction A (she is its creator).
- User Bob creates Transaction B and adds Approver X (id=42) as a required approver.
- Alice is added as an observer of Transaction B (or guesses/enumerates approver id=42).

**Steps:**
1. Alice calls `GET /transactions/B/approvers` → receives `[{ id: 42, userId: X, ... }]`.
2. Alice calls `DELETE /transactions/A/approvers/42` with her JWT.
3. Controller: `getCreatorsTransaction(A, Alice)` → passes (Alice owns A).
4. Service: `getTransactionApproverById(42)` → returns Approver X (belonging to B).
5. Service: `removeNode(42)` → soft-deletes Approver X from Transaction B.
6. **Result:** Transaction B's approval requirement is silently removed. Bob's transaction can now proceed to execution without Approver X's sign-off. [1](#0-0) [2](#0-1)
