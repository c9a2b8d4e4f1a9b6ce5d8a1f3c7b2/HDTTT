### Title
Any Authenticated Organization User Can Post Comments on Transactions They Have No Access To

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` enforces only authentication (JWT + verified user), but performs no check that the requesting user is a participant (creator, signer, observer, or approver) of the target transaction. Any registered organization user can post comments on — and read comments from — any transaction, bypassing the transaction-level access control model enforced everywhere else in the codebase. The developer acknowledged this gap with a `//TODO` comment directly in the controller.

### Finding Description

The `CommentsController` applies only three guards at the class level: `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. These confirm the caller holds a valid, non-blacklisted JWT and has `UserStatus.NONE`. No guard or service-layer check verifies that the user has any relationship to the transaction. [1](#0-0) 

The `createComment` handler passes the `transactionId` directly to `CommentsService.createComment`, which immediately creates and persists the comment with no access verification: [2](#0-1) 

The developer explicitly flagged this gap: [3](#0-2) 

By contrast, every other transaction sub-resource enforces participant-level access. For example, `getVerifiedApproversByTransactionId` and `getTransactionObserversByTransactionId` both throw `UnauthorizedException` if the user is not the creator, a signer, an observer, or an approver: [4](#0-3) [5](#0-4) 

The `getComments` and `getCommentById` GET endpoints on the same controller also lack any transaction-level access check, enabling any authenticated user to read comments on any transaction: [6](#0-5) 

### Impact Explanation

- **Unauthorized state mutation**: Any verified organization user can attach comments to any transaction in the system, including transactions they are not a party to. This pollutes the comment history of sensitive, high-stakes Hedera transactions (e.g., account creation, file updates, HBAR transfers managed by the Hedera Council).
- **Information disclosure**: Any verified user can enumerate and read all comments on any transaction, leaking internal deliberation, approval rationale, or sensitive operational notes that are intended only for transaction participants.
- The transaction isolation model — enforced for signers, observers, and approvers — is completely bypassed for the comments sub-resource.

### Likelihood Explanation

The attacker precondition is minimal: hold a valid JWT for any registered, verified organization account. No admin privilege, no leaked secret, and no special role is required. The endpoint is reachable via a standard HTTP POST to `POST /transactions/<any_id>/comments` with a bearer token. Any malicious insider or compromised low-privilege account can exploit this immediately.

### Recommendation

In `CommentsService.createComment`, call `transactionsService.getTransactionWithVerifiedAccess(transactionId, user)` (which already implements the full participant check via `verifyAccess`) before persisting the comment. Apply the same check in `getTransactionComments` and `getTransactionCommentById`. [7](#0-6) 

Alternatively, add a dedicated guard analogous to the one used in `ApproversController` and `ObserversController` that verifies transaction participation before the handler executes.

### Proof of Concept

1. Register two users in the organization: `alice` (creator of transaction `#42`) and `bob` (has no relationship to transaction `#42`).
2. Log in as `bob` and obtain a JWT.
3. Send:
   ```
   POST /transactions/42/comments
   Authorization: Bearer <bob_jwt>
   Content-Type: application/json

   { "message": "I should not be able to post this" }
   ```
4. Observe HTTP 201 and the comment persisted in the database linked to transaction `#42`.
5. Send `GET /transactions/42/comments` as `bob` and observe all existing comments returned — including any sensitive deliberation from `alice` and other participants.

  async getTransactionWithVerifiedAccess(transactionId: number | TransactionId, user: User) {
    const transaction = await this.getTransactionById(transactionId);

    await this.attachTransactionApprovers(transaction);

    if (!(await this.verifyAccess(transaction, user))) {
      throw new UnauthorizedException('You don\'t have permission to view this transaction');
    }
    return transaction;
  }
