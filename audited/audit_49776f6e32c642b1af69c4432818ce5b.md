### Title
Missing Ownership Check in `removeTransactionGroup` Allows Any Authenticated User to Corrupt Another User's Transaction Group State

### Summary
`removeTransactionGroup` in `TransactionGroupsService` performs no ownership or authorization check before operating on a `TransactionGroup`. It deletes `TransactionGroupItem` records unconditionally before delegating to `removeTransaction`, which does check ownership. Because the item deletion precedes the authorization check in each loop iteration, an attacker can progressively strip group items from any group they did not create, corrupting the group's structural integrity without ever successfully deleting the underlying transactions.

### Finding Description

**Root cause — missing pre-authorization in `removeTransactionGroup`:**

`removeTransactionGroup` fetches the target group with no user filter:

```typescript
const group = await this.dataSource.manager.findOneBy(TransactionGroup, { id });
```

It then fetches all group items with no user filter:

```typescript
const groupItems = await this.dataSource.manager.find(TransactionGroupItem, {
  relations: { group: true },
  where: { group: { id: group.id } },
});
```

Inside the loop, it **deletes the `TransactionGroupItem` first**, then calls `removeTransaction` (which does check ownership):

```typescript
for (const groupItem of groupItems) {
  const transactionId = groupItem.transactionId;
  await this.dataSource.manager.remove(TransactionGroupItem, groupItem); // ← no auth, executes unconditionally
  await this.transactionsService.removeTransaction(transactionId, user, false); // ← throws if not creator
}
``` [1](#0-0) 

When the attacker is not the creator of the transactions, `removeTransaction` throws `UnauthorizedException` via `getTransactionForCreator`: [2](#0-1) 

But by that point, the `TransactionGroupItem` for that iteration has already been permanently deleted. The `TransactionGroup` itself is not removed (line 193 is never reached), but one group item is gone per request.

**Contrast with `cancelTransactionGroup`**, which correctly gates on user-scoped data retrieval and an explicit ownership assertion before any mutation:

```typescript
const group = await this.getTransactionGroup(user, groupId, false); // user-filtered query
const allOwnedByUser = group.groupItems.every(
  item => item.transaction?.creatorKey?.userId === user.id,
);
if (!allOwnedByUser) {
  throw new UnauthorizedException('Only the creator can cancel all transactions in a group.');
}
``` [3](#0-2) 

`getTransactionGroup` uses `getTransactionGroupItemsQuery` with a user-scoped `WHERE` clause that filters by creator, signer, observer, and approver roles: [4](#0-3) 

`removeTransactionGroup` uses none of this — it bypasses the user-scoped query entirely.

**Exploit path (N-call attack):**
1. Attacker (authenticated, verified user) discovers victim's group ID (sequential integer, trivially enumerable).
2. Attacker sends `DELETE /transaction-groups/:id` — the first `TransactionGroupItem` is deleted before `removeTransaction` throws.
3. Attacker repeats N times for a group with N items.
4. All `TransactionGroupItem` records are deleted; the `TransactionGroup` shell and the underlying `Transaction` records remain, but the group is structurally empty and broken.

The `DELETE /transaction-groups/:id` endpoint is protected only by JWT authentication and email verification — no role or ownership guard: [5](#0-4) 

### Impact Explanation

- **Unauthorized state mutation**: Any authenticated user can destroy the `TransactionGroupItem` membership records of any group, regardless of ownership.
- **Broken atomic/sequential group execution**: The chain scheduler (`transaction-scheduler.service.ts`) relies on `groupItems` being intact to collate and execute atomic or sequential groups. Removing items breaks this invariant, causing the scheduler to execute an incomplete set or skip execution entirely. [6](#0-5) 

- **Orphaned transactions**: The underlying `Transaction` records survive but lose their group context, making coordinated multi-transaction workflows (e.g., FileUpdate + FileAppend) impossible to execute as a unit.
- **Severity**: High — unauthorized persistent state corruption of other users' transaction groups.

### Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (registered, email-verified organization user). No admin, no leaked secrets, no special role.
- **Group ID discovery**: `TransactionGroup.id` is a sequential auto-increment integer. An attacker can enumerate IDs by observing their own group IDs or by probing the `GET /transaction-groups/:id` endpoint (which does apply user filtering, but the integer range is trivially guessable).
- **Repeatability**: Each `DELETE` call removes exactly one group item before failing. The attacker simply loops N times to fully strip a group with N items.
- **No rate-limit bypass needed**: The operation is cheap and the endpoint is standard REST.

### Recommendation

Add an ownership check at the start of `removeTransactionGroup`, mirroring the pattern used in `cancelTransactionGroup`. Use `getTransactionGroup(user, id, false)` (which applies the user-scoped SQL filter) and then assert that all transactions in the group belong to the calling user before performing any deletions. Additionally, wrap the entire loop in a database transaction so that a failed `removeTransaction` rolls back any already-deleted group items atomically.

```typescript
async removeTransactionGroup(user: User, id: number): Promise<boolean> {
  // 1. Fetch group with user-scoped filter (throws if user has no access)
  const group = await this.getTransactionGroup(user, id, false);

  // 2. Assert ownership of all transactions
  const allOwnedByUser = group.groupItems.every(
    item => item.transaction?.creatorKey?.userId === user.id,
  );
  if (!allOwnedByUser) {
    throw new UnauthorizedException('Only the creator can remove a transaction group.');
  }

  // 3. Perform deletions inside a single DB transaction for atomicity
  await this.dataSource.transaction(async manager => {
    for (const groupItem of group.groupItems) {
      await manager.remove(TransactionGroupItem, groupItem);
      await this.transactionsService.removeTransaction(groupItem.transactionId, user, false);
    }
    await manager.remove(TransactionGroup, { id });
  });

  emitTransactionUpdate(...);
  return true;
}
```

### Proof of Concept

**Setup**: Alice (victim) creates a transaction group with 3 items via `POST /transaction-groups`. Group is assigned `id = 42`.

**Attack** (Bob, any authenticated user):

```
DELETE /transaction-groups/42   HTTP/1.1
Authorization: Bearer <bob_jwt>
```

**Iteration 1**: `TransactionGroupItem` for item 1 is deleted. `removeTransaction` throws (Bob is not creator). Response: 500/401.

**Iteration 2**: Same request. `TransactionGroupItem` for item 2 is deleted. Throws again.

**Iteration 3**: Same request. `TransactionGroupItem` for item 3 is deleted. Throws again.

**Result**: `SELECT * FROM transaction_group_item WHERE "groupId" = 42` returns 0 rows. The `TransactionGroup` record with `id = 42` still exists but has no items. Alice's 3 `Transaction` records still exist but are no longer part of any group. Any scheduled atomic/sequential execution for group 42 will find no items and fail silently or execute nothing.
