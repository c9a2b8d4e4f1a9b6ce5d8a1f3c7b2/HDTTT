### Title
Any Authenticated User Can Register Any Public Key Without Proof-of-Ownership, Blocking Legitimate Key Registration

### Summary
The `uploadKey` endpoint in the Hedera Transaction Tool's backend accepts any valid Hedera public key string and registers it under the authenticated caller's account without requiring any cryptographic proof that the caller controls the corresponding private key. Because the system enforces a one-owner-per-public-key constraint, an attacker who registers a victim's public key first permanently blocks the victim from registering that key in the organization. This corrupts the key registry and disrupts the signing workflow for any transaction requiring that key.

### Finding Description

**Root cause:** `uploadKey` in `user-keys.service.ts` performs no proof-of-ownership check. The only validation is that the submitted string is a syntactically valid Hedera public key (`@IsHederaPublicKey()`). No challenge-response, no signature over a nonce, nothing. [1](#0-0) 

The service then looks up whether the public key already exists. If it does not, it creates a new `UserKey` record owned by the authenticated caller: [2](#0-1) 

The ownership guard only fires when the key is **already registered**: [3](#0-2) 

If the key is not yet registered, there is no check at all — the caller becomes the owner.

**Exploit path:**

1. Attacker (authenticated org member) observes a victim's Hedera public key. Public keys are visible on the Hedera mirror node, in transaction bytes, and via the `/users/public-owner/:publicKey` endpoint.
2. Attacker calls `POST /user/<any-id>/keys` with `{ "publicKey": "<victim_public_key>" }`. The `userId` path parameter is ignored by the controller — the key is always registered to the JWT-authenticated caller. [4](#0-3) 

3. The victim's public key is now stored in `user_key` with `userId = attacker_id`.
4. When the victim later tries to register their own key, the service finds the existing record, sees `userKey.userId !== user.id`, and throws `ErrorCodes.PU` (key in use). The victim is permanently blocked. [5](#0-4) 

The e2e test suite itself confirms that the `userId` in the URL is completely ignored and that any key can be registered regardless of the path parameter: [6](#0-5) 

### Impact Explanation

1. **Denial of Service on key registration (concrete):** The victim can never register their own public key in the organization. Because `creatorKeyId` must reference a key owned by the submitting user, the victim cannot create transactions in the organization using that key. [7](#0-6) 

2. **Key registry corruption:** The `/users/public-owner/:publicKey` endpoint returns the attacker's email as the owner of the victim's public key, breaking the trust model that other users rely on to verify key ownership.

3. **Signing workflow disruption:** `keysRequiredToSign` resolves `UserKey` records by public key string to determine who must sign a transaction. With the attacker registered as owner of the victim's key, signing notifications and routing are directed to the wrong account. [8](#0-7) 

Note: The attacker cannot forge signatures (they lack the private key), so they cannot create or sign transactions on the victim's behalf. The impact is DoS and registry integrity, not direct fund theft.

### Likelihood Explanation

- **Attacker precondition:** Any verified organization member. No admin or privileged role required.
- **Key discoverability:** Hedera public keys are public by definition — visible on the mirror node, in transaction bytes, and via the organization's own API.
- **Effort:** A single authenticated HTTP POST. No race condition or timing dependency required; the attacker simply needs to act before the victim registers their key.
- **Scalability:** An attacker can pre-register all public keys they observe, blocking multiple victims simultaneously.

### Recommendation

Require the caller to prove possession of the private key before registering a public key. The standard approach is a **challenge-response**:

1. The server issues a short-lived random nonce tied to the session.
2. The client signs the nonce with the private key corresponding to the public key being registered.
3. The server verifies the signature using the submitted public key before persisting the record.

This is consistent with how transaction creation already enforces key ownership — `validateAndPrepareTransaction` verifies `publicKey.verify(dto.transactionBytes, dto.signature)` before accepting a `creatorKeyId`. [9](#0-8) 

Apply the same pattern to `uploadKey`: add a `signature` field to `UploadUserKeyDto` containing a signature over a server-issued nonce, and verify it in `UserKeysService.uploadKey` before saving.

### Proof of Concept

**Preconditions:** Two accounts in the organization — `attacker` (verified) and `victim` (verified). The victim's public key `victim_pubkey` is known (e.g., from the mirror node or from `GET /user/<victim_id>/keys`).

**Steps:**

```
# Step 1: Attacker registers victim's public key under their own account
POST /user/1/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{ "publicKey": "<victim_pubkey>" }

# Response: 201 Created — key is now owned by attacker

# Step 2: Victim attempts to register their own key
POST /user/2/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json

{ "publicKey": "<victim_pubkey>" }

# Response: 400 Bad Request — ErrorCodes.PU ("Public key in use")
# Victim is permanently blocked from registering their own key.

# Step 3: Confirm registry corruption
GET /users/public-owner/<victim_pubkey>
Authorization: Bearer <any_jwt>

# Response: attacker's email — not the victim's
```

**Expected outcome:** The victim cannot register their own public key, cannot use it as `creatorKeyId` for new transactions, and the system reports the attacker as the key's owner.
