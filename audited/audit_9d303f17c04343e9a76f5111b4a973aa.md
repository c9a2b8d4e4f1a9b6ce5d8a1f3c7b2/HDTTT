### Title
Plaintext Temporary Password Transmitted via Unauthenticated/Unencrypted SMTP Enables New-User Account Takeover

### Summary
The `signUpByAdmin` flow generates a temporary password and embeds it in plaintext in a registration email. The SMTP transport is configured with `EMAIL_API_SECURE=false` as the documented default and with optional (unauthenticated) SMTP credentials. An attacker with network access to the SMTP path can intercept the unencrypted connection and read the temporary password before the victim does, then log in as the newly registered user and take full control of the account. This is a direct analog to the external report's vulnerability class: authentication credential exposure via an insecure email channel.

### Finding Description

**Step 1 — Temporary password generated and emailed in plaintext**

`signUpByAdmin` in `AuthService` generates a temporary password and passes it directly into the registration email payload: [1](#0-0) 

The `generatePassword()` method produces a password of the form `xxxxx-xxxxx` (10 lowercase letters + 1 dash), giving only ~26^10 ≈ 1.4 × 10^14 combinations — and the fixed format further reduces effective entropy: [2](#0-1) 

The email template embeds the raw `tempPassword` value directly in the HTML body: [3](#0-2) 

**Step 2 — SMTP transport defaults to unencrypted**

`EmailService` constructs the nodemailer transporter from `EMAIL_API_SECURE`, which the official documentation and example configuration set to `false`: [4](#0-3) 

The README example explicitly shows `EMAIL_API_SECURE=false`: [5](#0-4) 

**Step 3 — SMTP authentication is optional**

`EMAIL_API_USERNAME` and `EMAIL_API_PASSWORD` are marked `optional: true` in the Kubernetes deployment manifest, meaning the service can be deployed with no SMTP authentication at all: [6](#0-5) 

`getAuthConfig()` silently omits the `auth` block when credentials are absent: [7](#0-6) 

**Step 4 — No SPF/DKIM/DMARC enforcement**

`SENDER_EMAIL` is operator-configurable with no domain validation anywhere in the codebase. The code sets the `from` field to an arbitrary string: [8](#0-7) 

There is no code-level enforcement of SPF, DKIM, or DMARC records for the sender domain. If the deployer's domain lacks these DNS records, an attacker can spoof registration emails from the same `SENDER_EMAIL` address, embedding a known temporary password, and the victim's mail client will display it as legitimate.

### Impact Explanation

A successful attack yields full account takeover of a newly registered organization member. In the Hedera Transaction Tool context, this means the attacker can:
- Approve or sign multi-signature Hedera transactions on behalf of the victim
- Observe all pending transactions and organization membership
- Participate in threshold-key signing workflows, potentially unblocking transactions that require the victim's key

The impact is high because the tool is explicitly designed for managing financial transactions on the Hedera network.

### Likelihood Explanation

Two independent attack paths exist:

1. **Network interception**: An attacker on the network path between the `notifications` service and the SMTP relay (e.g., a corporate network, cloud VPC, or ISP) can read the unencrypted SMTP session and extract the temporary password. This is realistic in enterprise deployments where the SMTP relay is internal.

2. **Email spoofing**: If the deployer's domain lacks SPF/DKIM/DMARC (a common misconfiguration, directly matching the external report), an attacker can forge a registration email with a known temporary password. The victim receives what appears to be a legitimate onboarding email and logs in with the attacker-controlled credential.

The temporary password is also low-entropy (`xxxxx-xxxxx`, lowercase only), making offline brute-force feasible if the hash is ever exposed.

### Recommendation

1. **Enforce TLS for SMTP**: Set `EMAIL_API_SECURE=true` and require STARTTLS or SMTPS. Remove the `false` default from documentation and example configs.
2. **Require SMTP authentication**: Remove the `optional: true` flags from `EMAIL_API_USERNAME` and `EMAIL_API_PASSWORD` in the Kubernetes manifests and make them required.
3. **Do not transmit credentials in email**: Replace the temporary-password-in-email pattern with a time-limited, single-use invite link (e.g., a signed JWT URL) that lets the user set their own password on first login.
4. **Document SPF/DKIM/DMARC requirements**: Add a deployment prerequisite requiring operators to configure these DNS records for the `SENDER_EMAIL` domain before going live.
5. **Increase password entropy**: If temporary passwords must be used, generate them with a cryptographically secure random source and include mixed-case letters, digits, and symbols.

### Proof of Concept

**Path A — SMTP interception (unencrypted transport)**

1. Admin calls `POST /auth/signup` with `{ "email": "victim@corp.com" }`.
2. `signUpByAdmin` generates `tempPassword = "abcde-fghij"` and calls `emitUserRegistrationEmail(...)` with `{ tempPassword }` in `additionalData`.
3. The `notifications` service sends the email over an unencrypted SMTP connection (`EMAIL_API_SECURE=false`).
4. Attacker on the network path captures the raw SMTP `DATA` command and reads:
   ```
   Temporary Password
   abcde-fghij
   ```
5. Attacker calls `POST /auth/login` with `{ "email": "victim@corp.com", "password": "abcde-fghij" }` before the victim does.
6. Server returns a valid JWT. Attacker has full session as `victim@corp.com`.

**Path B — Email spoofing (missing SPF/DKIM/DMARC)**

1. Attacker determines the `SENDER_EMAIL` domain (e.g., `no-reply@org.example.com`) lacks SPF/DKIM/DMARC records.
2. Attacker crafts a spoofed "Hedera Transaction Tool Registration" email to `victim@corp.com` with `tempPassword = "aaaaa-bbbbb"` (attacker-chosen).
3. Victim receives the email, which passes mail-client authentication checks (no DMARC rejection), and logs in with `aaaaa-bbbbb`.
4. Attacker, who already knows the password, logs in simultaneously and takes over the account. [9](#0-8) [10](#0-9)

  async signUpByAdmin(dto: SignUpUserDto, url: string): Promise<User> {
    const rawRepoUrl = this.configService.get<string>('FRONTEND_REPO_URL');
    const repoUrl = rawRepoUrl ? rawRepoUrl.replace(/\/+$/, '') : '';
    const downloadUrl = `${repoUrl}/latest`;
    const tempPassword = this.generatePassword();

    const existingUser = await this.usersService.getUser({ email: dto.email }, true);
    let user: User;

    if (existingUser && !existingUser.deletedAt && existingUser.status === UserStatus.NEW) {
      const hashedPass = await this.usersService.getSaltedHash(tempPassword);
      user = await this.usersService.updateUserById(existingUser.id, { password: hashedPass });
    } else {
      user = await this.usersService.createUser(dto.email, tempPassword);
    }

    this.logger.log(`User ${user.id} registered and temporary password generated.`);

    emitUserRegistrationEmail(this.notificationsPublisher, [{ email: user.email, additionalData: { url, tempPassword, downloadUrl } }]);

    return user;
  }

  private generatePassword() {
    const getRandomLetters = (length: number) =>
      Array.from({ length }, () => String.fromCharCode(97 + Math.floor(Math.random() * 26))).join(
        '',
      );

    return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
