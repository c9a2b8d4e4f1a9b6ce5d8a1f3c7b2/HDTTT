### Title
Any Authenticated User Can Claim Another User's Public Key Without Proof of Ownership

### Summary
The `uploadKey` function in `UserKeysService` allows any authenticated, verified organization member to register any valid Hedera public key as their own without proving possession of the corresponding private key. This mirrors the external report's pattern exactly: a resource (public key) is registered to the caller without verifying the caller actually owns it. A malicious user can preemptively claim a victim's public key, permanently blocking the legitimate owner from registering it and disrupting multi-signature workflows.

### Finding Description
The `POST /user/:userId?/keys` endpoint is handled by `UserKeysController.uploadKey`, which passes only the authenticated user and the request body to the service: [1](#0-0) 

The service `UserKeysService.uploadKey` performs the following checks: [2](#0-1) 

The only ownership guard is: if the public key is **already registered** to a different user, reject it. If the key is **not yet registered**, it is created for whoever calls the endpoint first — with zero cryptographic proof that the caller holds the corresponding private key.

The DTO only validates that `publicKey` is a syntactically valid Hedera key: [3](#0-2) 

No challenge-response, no signature verification, no proof of private key possession.

**Attack path:**
1. Attacker (any verified org member) calls `GET /user/{victimId}/keys` — public keys are returned to all verified users (only `mnemonicHash`/`index` are hidden from non-owners): [4](#0-3) 
2. Attacker calls `POST /user/keys` with `{ "publicKey": "<victim's public key>" }` using their own JWT.
3. The key is now registered under the attacker's account (201 Created).
4. Victim calls `POST /user/keys` with their own public key → receives 400 `ErrorCodes.PU` ("Public Key in use") because `userKey.userId !== user.id`.
5. Victim is permanently blocked from registering their key in the organization.

The e2e test explicitly confirms the `userId` URL parameter is ignored for POST — the key always goes to the authenticated caller: [5](#0-4) 

### Impact Explanation
- **Key squatting / DoS**: A malicious verified user can preemptively claim any public key not yet registered, permanently preventing the legitimate owner from registering it (the `ErrorCodes.PU` check is a hard block with no recovery path for the victim).
- **Multi-sig disruption**: If the attacker is subsequently added as a required transaction signer or approver with the claimed key, they cannot produce a valid signature (they lack the private key). The `approveTransaction` function cryptographically verifies signatures, so the attacker cannot forge one — but the transaction becomes stuck waiting for a signature that can never arrive.
- **Severity: Medium** — requires an authenticated org member; impact is availability/integrity of key registration and transaction workflows.

### Likelihood Explanation
- Any authenticated, verified organization member can perform this attack with no special privileges.
- Victim public keys are directly readable via `GET /user/:userId/keys`, which is accessible to all verified users.
- Public keys are also observable on the Hedera Mirror Node for any account the victim has linked.
- The attack is trivially scriptable and requires only a single API call.
- **Likelihood: Medium** — attacker must be a verified org member, but that is a low bar in any multi-user organization deployment.

### Recommendation
Require cryptographic proof of private key possession at upload time. Implement a challenge-response flow:
1. Server issues a random nonce tied to the session/user.
2. Client signs the nonce with the private key corresponding to the public key being registered.
3. Server verifies the signature against the submitted public key before persisting the record.

Alternatively, at minimum, validate that the `mnemonicHash` field (when provided) is consistent with the public key — though a full challenge-response is the correct fix.

### Proof of Concept
```
# Step 1: Attacker obtains victim's public key
GET /user/2/keys
Authorization: Bearer <attacker_jwt>
→ 200 OK: [{ "publicKey": "302a300506032b6570032100<victim_pubkey>", ... }]

# Step 2: Attacker claims victim's public key
POST /user/keys
Authorization: Bearer <attacker_jwt>
Content-Type: application/json
{ "publicKey": "302a300506032b6570032100<victim_pubkey>" }
→ 201 Created: key now owned by attacker

# Step 3: Victim attempts to register their own key
POST /user/keys
Authorization: Bearer <victim_jwt>
Content-Type: application/json
{ "publicKey": "302a300506032b6570032100<victim_pubkey>" }
→ 400 Bad Request: ErrorCodes.PU ("Public Key in use")
   Victim is permanently blocked from registering their key.
```

Root cause: `back-end/apps/api/src/user-keys/user-keys.service.ts` lines 42–65 — no proof of private key ownership is required before associating a public key to a user account. [6](#0-5)
