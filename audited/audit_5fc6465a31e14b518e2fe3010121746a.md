### Title
Approver Deletion Authorization Bypass: Any Authenticated User Can Remove Approvers from Transactions They Do Not Own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that this approver actually belongs to `:transactionId`. An attacker who owns any transaction can supply their own `transactionId` to pass the ownership check, while supplying an approver `id` from a victim's transaction to delete it.

### Finding Description

**Root cause — mismatched authorization scope in the controller:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```typescript
// Step 1: verify user owns transactionId (from URL param)
await this.approversService.getCreatorsTransaction(transactionId, user);

// Step 2: delete approver by id (from URL param) — no cross-check
await this.approversService.removeTransactionApprover(id);
``` [1](#0-0) 

The service method `removeTransactionApprover` accepts only an approver `id` and performs no ownership verification:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  // approver.transactionId is never compared to the URL's transactionId
``` [2](#0-1) 

**Contrast with `updateTransactionApprover`**, which correctly cross-checks that the approver's root transaction matches the URL parameter before proceeding:

```typescript
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
``` [3](#0-2) 

The delete path has no equivalent guard. The authorization check validates ownership of one resource (`transactionId`) while the destructive action targets a different, unvalidated resource (the approver `id`).

**Exploit path:**
1. Attacker registers as a normal user and creates Transaction A (they become its creator).
2. Attacker enumerates or discovers approver IDs belonging to Transaction B (owned by a victim). Approver IDs are sequential integers; the `GET /transactions/:transactionId/approvers` endpoint returns approver IDs to any user who has access to that transaction (observer, signer, or approver role).
3. Attacker sends:
   ```
   DELETE /transactions/{transactionA_id}/approvers/{victimApprover_id}
   ```
4. `getCreatorsTransaction(transactionA_id, attacker)` passes — attacker owns Transaction A.
5. `removeTransactionApprover(victimApprover_id)` deletes the victim's approver with no further check.

### Impact Explanation

An attacker can silently delete any approver from any transaction in the system. This breaks the multi-signature approval workflow: if a required approver is removed, the threshold may no longer be reachable, permanently blocking a transaction from being approved and executed. Alternatively, removing approvers from a threshold tree can lower the effective approval requirement, allowing a transaction to proceed with fewer approvals than the creator intended. This is an unauthorized state change with direct impact on transaction integrity and the trust model of the organization mode.

### Likelihood Explanation

The attacker requires only a valid authenticated account (no admin or privileged role). Approver IDs are sequential integers and are returned in API responses to any user with read access to a transaction. The attack requires no special timing, cryptographic break, or physical access. Any malicious organization member can execute this against any transaction they can observe.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted belongs to the `transactionId` supplied in the URL. The simplest fix mirrors the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the authorized transaction
  const root = await this.getRootNodeFromNode(approver.id);
  if (root?.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
``` [2](#0-1) 

### Proof of Concept

**Preconditions:** Two registered, verified users — `attacker` and `victim`. Both are members of the same organization.

1. `victim` creates Transaction B and adds `approverUser` as an approver via `POST /transactions/{B}/approvers`. The response includes `{ id: 42, userId: approverUser.id, transactionId: B }`.

2. `attacker` creates Transaction A via `POST /transactions`. Response: `{ id: 99 }`.

3. `attacker` sends:
   ```http
   DELETE /transactions/99/approvers/42
   Authorization: Bearer <attacker_token>
   ```

4. **Expected (secure) behavior:** 401/403 — approver 42 does not belong to transaction 99.

5. **Actual behavior:** 200 OK. Approver 42 is soft-deleted from Transaction B. `victim`'s approval workflow is disrupted.

The controller's `getCreatorsTransaction(99, attacker)` succeeds (attacker owns transaction 99), and `removeTransactionApprover(42)` deletes approver 42 without any cross-check. [1](#0-0) [2](#0-1)
