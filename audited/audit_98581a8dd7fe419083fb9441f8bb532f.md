### Title
Transaction Creator Can Bypass Agreed Approval Threshold by Modifying Approver Structure After Signatures Are Collected

### Summary
The transaction creator can modify the approval threshold or remove approvers at any point during a transaction's lifecycle — including after other approvers have already signed — because neither `updateTransactionApprover` nor `removeTransactionApprover` enforce any transaction-status guard. This allows a malicious creator to lower the required approval count to match already-collected signatures, causing a transaction to reach `WAITING_FOR_EXECUTION` with fewer approvals than the other participants agreed to.

### Finding Description

**Vulnerability class**: State transition integrity failure (analogous to M03 — privileged actor bypasses agreed constraints by mutating governance parameters after participants have committed).

**Root cause**: `updateTransactionApprover` and `removeTransactionApprover` only verify that the caller is the transaction creator (`getCreatorsTransaction`). Neither function checks the current transaction status before allowing structural changes to the approver tree.

`createTransactionApprovers` — no status guard: [1](#0-0) 

`updateTransactionApprover` — only creator check, no status guard, threshold can be freely lowered: [2](#0-1) 

Threshold update path — no status check before writing: [3](#0-2) 

`removeTransactionApprover` in the controller — only creator check, no status guard: [4](#0-3) 

`getCreatorsTransaction` — confirms it only checks creator identity, not status: [5](#0-4) 

**Exploit flow**:

1. Creator creates a transaction and adds 3 approvers under a threshold-3 tree node (all 3 must approve).
2. Approver A and Approver B review the approval structure and sign/approve.
3. Before Approver C acts, the creator calls:
   ```
   PATCH /transactions/{id}/approvers/{treeNodeId}
   Body: { "threshold": 2 }
   ```
4. The threshold is now 2. Two approvals already exist. `processTransactionStatus` detects sufficient signatures and transitions the transaction to `WAITING_FOR_EXECUTION`.
5. The transaction executes without Approver C's consent — the agreed 3-of-3 requirement was silently bypassed.

Alternatively, the creator can `DELETE /transactions/{id}/approvers/{C_id}` to remove an unsigned approver, then lower the threshold to match already-collected signatures.

### Impact Explanation

A malicious creator can execute any transaction (account key updates, token transfers, Hedera system file modifications) with fewer approvals than the other participants agreed to. Approvers who reviewed and signed based on the original approval structure have no mechanism to detect or prevent the post-hoc modification. In organization workflows where high-value or irreversible Hedera operations require multi-party approval, this completely undermines the integrity of the approval process.

### Likelihood Explanation

The attacker is a normal authenticated user (the transaction creator) — no admin keys, no leaked credentials, no privileged access required. The attack requires only: (1) creating a transaction, (2) waiting for at least one approver to sign, (3) issuing a standard PATCH or DELETE API call. The endpoint is reachable over the normal REST API with a valid JWT. This is a realistic, low-effort attack by any malicious creator.

### Recommendation

Add a transaction-status guard at the start of `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover`. Structural changes to the approver tree should only be permitted while the transaction is in the `NEW` state (before any approver has acted). Once the transaction advances to `WAITING_FOR_SIGNATURES` or beyond, the approver structure should be immutable.

```typescript
// In getCreatorsTransaction or at the top of each mutating method:
const allowedStatuses = [TransactionStatus.NEW];
if (!allowedStatuses.includes(transaction.status)) {
  throw new BadRequestException('Approver structure cannot be modified after signing has begun');
}
```

### Proof of Concept

**Preconditions**: Two registered users (Creator, ApproverA, ApproverB, ApproverC) in an organization. Creator has a valid JWT.

1. Creator POSTs `/transactions` → gets `transactionId=42`.
2. Creator POSTs `/transactions/42/approvers` with a threshold-3 tree containing ApproverA, ApproverB, ApproverC.
3. ApproverA calls `POST /transactions/42/approvers/approve` → signs.
4. ApproverB calls `POST /transactions/42/approvers/approve` → signs.
5. Creator calls:
   ```
   PATCH /transactions/42/approvers/{treeNodeId}
   Authorization: Bearer <creator_jwt>
   { "threshold": 2 }
   ```
   — returns 200, threshold updated with no error.
6. `processTransactionStatus` runs (or is triggered) and finds 2-of-2 signatures satisfied → status transitions to `WAITING_FOR_EXECUTION`.
7. Transaction executes on Hedera without ApproverC's consent.

**Expected (correct) behavior**: Step 5 should be rejected with an error because the transaction has already entered the signing phase. [6](#0-5) [4](#0-3)

```typescript
  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
