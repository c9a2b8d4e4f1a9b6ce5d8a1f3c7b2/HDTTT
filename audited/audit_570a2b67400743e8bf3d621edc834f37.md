### Title
Unbounded In-Memory Loop in `getTransactionsToSign` Enables Authenticated DoS via Transaction Accumulation

### Summary
The `getTransactionsToSign` endpoint in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **all** active transactions from the database with no row limit, then performs a sequential async database call (`userKeysToSign`) for every transaction before applying pagination in memory. Any authenticated user can create an unbounded number of transactions to inflate this set, causing the endpoint to exhaust server resources (CPU, DB connections, memory) and become unresponsive for all users.

### Finding Description

**Root cause — no DB-level `take` limit before the per-row async loop:**

In `getTransactionsToSign`, the repository query has no `take` constraint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
// ← no `take` / limit
```

Every row returned is then processed one-by-one with an awaited async call:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Pagination is applied only **after** the full loop completes:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
```

`userKeysToSign` internally calls `keysRequiredToSign`, which performs signature-key computation and additional DB queries per transaction. The cost of one request therefore scales linearly — and unboundedly — with the total number of active transactions in the system.

**Attacker-controlled entry path:**

The `POST /transactions` endpoint (guarded only by JWT + `HasKeyGuard`) allows any authenticated user with a registered key to create transactions. Each transaction requires a unique Hedera `transactionId`, which the attacker can generate freely by varying the account ID or timestamp component. Transactions remain in the active pool (status `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`) until their `validStart` expires (180 seconds window), but an attacker can continuously create new ones to maintain a large active set, or set `validStart` far in the future.

**Contrast with the bounded `importSignatures` path:**

The `importSignatures` endpoint (`POST /transactions/signatures/import`) also accepts an unbounded array, but the processing is bounded by what the caller explicitly submits per request. The `getTransactionsToSign` bug is worse: the server itself fetches and processes the entire unbounded set on every GET request, regardless of the requested page size. [1](#0-0) [2](#0-1) 

### Impact Explanation

Every call to `GET /transactions/sign` (or the equivalent controller route) triggers a full table scan of all active transactions followed by N sequential async DB round-trips. With a sufficiently large active transaction set:

- The Node.js event loop is blocked for the duration of the loop.
- The PostgreSQL connection pool is saturated with concurrent queries from parallel user requests.
- Response times grow without bound, causing HTTP timeouts for all users of the API service.
- The Chain service's `updateTransactions` cron (which also iterates over active transactions) is similarly degraded.

The result is **service unavailability** for all organization users — a permanent degradation that persists as long as the attacker maintains a large active transaction set. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

- **Attacker preconditions:** A valid JWT (i.e., a registered, verified account). No admin or privileged role is required.
- **Attack cost:** Creating transactions is a normal product workflow. The attacker needs only to register, obtain a Hedera key pair, and submit transactions in a loop via the public API.
- **Persistence:** Transactions with future `validStart` values remain active indefinitely until expiry. The attacker can continuously replenish the pool.
- **No rate-limit evidence found** on `POST /transactions` in the controller or middleware. [5](#0-4) [6](#0-5) 

### Recommendation

1. **Apply a DB-level `take` limit before the loop.** Fetch only the page-sized slice from the database, not the entire table. This requires moving the `userKeysToSign` filter logic into the query (e.g., a subquery or a join on the signature-key table) so that the DB can filter and paginate correctly.

2. **As an interim fix**, add a hard cap on the number of rows fetched (e.g., `take: 500`) and document that `totalItems` is approximate.

3. **Add rate-limiting** on `POST /transactions` per authenticated user to limit how quickly the active transaction pool can be inflated.

4. **Apply the same fix to `importSignatures`** (`POST /transactions/signatures/import`) and `uploadSignatureMaps` (`POST /transactions/signers`), both of which accept unbounded arrays with no server-side size cap. [1](#0-0) [7](#0-6) 

### Proof of Concept

**Setup:** Register two accounts — `attacker` and `victim`. Both are normal authenticated users.

**Step 1 — Inflate the active transaction pool:**
```
for i in 1..10000:
  POST /transactions
  Authorization: Bearer <attacker_token>
  {
    "transactionBytes": "<valid_hedera_tx_bytes with unique transactionId>",
    "validStart": "<now + 3 minutes>",
    "creatorKeyId": <attacker_key_id>,
    ...
  }
```
Each request succeeds (HTTP 201). The transactions enter `WAITING_FOR_SIGNATURES` status.

**Step 2 — Trigger the unbounded loop as victim:**
```
GET /transactions/sign?page=1&size=10
Authorization: Bearer <victim_token>
```

**Expected outcome:** The server fetches all 10,000 active transactions, then calls `userKeysToSign` 10,000 times sequentially. The request times out or takes tens of seconds. Concurrent requests from other users queue behind this operation, causing API-wide degradation.

**Root cause confirmed at:**
- `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–309 (unbounded fetch + sequential async loop)
- `back-end/apps/api/src/transactions/transactions.controller.ts` lines 155–175 (the `GET /transactions/sign` route that calls this function) [1](#0-0) [8](#0-7)
