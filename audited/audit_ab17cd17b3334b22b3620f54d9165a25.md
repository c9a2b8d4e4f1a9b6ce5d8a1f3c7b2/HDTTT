### Title
Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

### Summary
In `approvers.controller.ts`, the `removeTransactionApprover` endpoint verifies the requesting user is the creator of the `transactionId` supplied in the URL, but then deletes the approver identified by the separate `id` URL parameter without verifying that approver actually belongs to that transaction. A user who owns any transaction can therefore delete approvers from transactions they do not own.

### Finding Description
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `ApproversController.removeTransactionApprover` performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the authenticated user is the creator of the transaction identified by `transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver identified by `id`. [1](#0-0) 

There is no check that the approver with `id` belongs to the transaction with `transactionId`. The service-level `removeTransactionApprover` only looks up the approver by its own primary key and deletes it: [2](#0-1) 

By contrast, the sibling `updateTransactionApprover` service method correctly validates `rootNode.transactionId !== transactionId` before proceeding: [3](#0-2) 

The remove path has no equivalent cross-reference check, making the `transactionId` authorization guard ineffective for protecting approvers of other transactions.

**Exploit path:**
1. Attacker creates transaction A (they are its creator).
2. Victim's transaction B has an approver with database ID = `X`.
3. Attacker sends `DELETE /transactions/A/approvers/X`.
4. `getCreatorsTransaction(A, attacker)` succeeds — attacker owns A.
5. `removeTransactionApprover(X)` deletes approver `X` from transaction B with no further check.

### Impact Explanation
An authenticated but unprivileged user (any transaction creator) can silently remove approvers from any other transaction in the system. This breaks the multi-signature approval workflow: a transaction that required N approvals can have its approver list gutted, potentially allowing it to advance to execution without the intended oversight. This is an unauthorized state change with direct integrity impact on the transaction lifecycle.

### Likelihood Explanation
The attacker only needs a valid account and at least one transaction of their own (the normal onboarding path). No admin credentials, leaked secrets, or privileged roles are required. The approver IDs are sequential integers, making enumeration trivial. The attack is fully reachable via the public REST API.

### Recommendation
Inside `removeTransactionApprover` (either in the controller or the service), verify that the approver being deleted actually belongs to the transaction the caller is authorized for. The simplest fix mirrors what `updateTransactionApprover` already does:

```typescript
// In approvers.controller.ts removeTransactionApprover, after getCreatorsTransaction:
const approver = await this.approversService.getTransactionApproverById(id);
if (!approver || approver.transactionId !== transactionId) {
  throw new BadRequestException('Approver does not belong to this transaction');
}
await this.approversService.removeTransactionApprover(id);
```

Alternatively, add a `transactionId` filter inside `removeTransactionApprover` in the service layer, consistent with how `updateTransactionApprover` validates `rootNode.transactionId`.

### Proof of Concept

1. Register two users: **Alice** (victim) and **Bob** (attacker).
2. Alice creates transaction `T_alice` and adds an approver; note the approver's `id` = `X` from the response.
3. Bob creates any transaction `T_bob` (he just needs to be its creator).
4. Bob sends:
   ```
   DELETE /transactions/{T_bob.id}/approvers/{X}
   Authorization: Bearer <Bob's JWT>
   ```
5. The server responds `200 true`.
6. Alice's approver `X` is now deleted from `T_alice`, even though Bob has no relationship to `T_alice`. [1](#0-0) [2](#0-1)

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
