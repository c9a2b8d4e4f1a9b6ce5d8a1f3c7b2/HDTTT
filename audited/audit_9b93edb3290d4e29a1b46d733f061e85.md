### Title
Missing Post-State Validation When Reparenting an Approver Node Leaves Parent Threshold Invariant Broken

### Summary

`updateTransactionApprover` in `approvers.service.ts` validates the **destination** parent when moving a child approver to a new parent (`dto.listId = newParentId`), but never adjusts or validates the **source** parent after it loses a child. The invariant `threshold ≤ number_of_children` can therefore be silently violated on the source parent, leaving the approval tree in a permanently unsatisfiable state. The `dto.listId = null` branch in the same function correctly handles this case, making the omission in the reparenting branch a clear inconsistency.

### Finding Description

`updateTransactionApprover` handles three mutually exclusive update modes. The `dto.listId === null` branch (detach to root) explicitly compensates the old parent:

```
// lines 417-428 — old parent is adjusted when child is detached to root
if (parent) {
  const newParentApproversLength = parent.approvers.length - 1;
  if (newParentApproversLength === 0) {
    await transactionalEntityManager.softRemove(TransactionApprover, parent);
  } else if (newParentApproversLength < parent.threshold) {
    await transactionalEntityManager.update(TransactionApprover, parent.id, {
      threshold: newParentApproversLength,
    });
  }
}
``` [1](#0-0) 

The `dto.listId = newParentId` branch (reparent to a different tree node) performs no equivalent adjustment on the source parent:

```
// lines 457-464 — child is moved to new parent; old parent is never touched
await transactionalEntityManager.update(TransactionApprover, approver.id, {
  listId: dto.listId,
  transactionId: null,
});
approver.listId = dto.listId;
approver.transactionId = null;
updated = true;
return approver;
``` [2](#0-1) 

After this update the source parent retains its original `threshold` value while its `approvers` array has shrunk by one. If `threshold` was equal to the original child count, it now exceeds the remaining child count, violating the invariant enforced everywhere else in the service:

```
// line 305 — invariant enforced at creation time
(dtoApprover.threshold > dtoApprover.approvers.length || dtoApprover.threshold === 0)
``` [3](#0-2) 

The endpoint is reachable by any authenticated transaction creator via `PATCH /transactions/:transactionId/approvers/:id`. [4](#0-3) 

### Impact Explanation

A source parent whose `threshold` exceeds its remaining child count can never satisfy its approval condition. Any transaction whose approval tree routes through that parent is permanently stuck in `WAITING_FOR_SIGNATURES`. Because `removeTransactionApprover` deletes the entire subtree of the node passed to it, the creator cannot surgically remove only the broken parent without also deleting its remaining legitimate children. The only recovery path is to update the broken parent's threshold separately — an action that requires the creator to notice the corruption and take a corrective step that the system does not prompt for.

Concrete outcome: a transaction creator can (intentionally or accidentally) render a multi-party approval tree permanently unsatisfiable, causing a DoS on the transaction's approval workflow and locking all parties' participation in that transaction.

### Likelihood Explanation

The attacker is the transaction creator — a normal authenticated user with no elevated privileges. The trigger is a single valid API call (`PATCH /transactions/:transactionId/approvers/:id` with `{ listId: <otherId> }`) on a transaction they own. No special timing, race condition, or external dependency is required. The scenario arises naturally when a creator reorganises an approval tree after one or more approvers have already been assigned, which is an expected product workflow.

### Recommendation

In the `dto.listId = newParentId` branch, after updating the approver's `listId`, fetch the source parent (the approver's previous `listId`) and apply the same compensation logic already present in the `dto.listId === null` branch:

1. If the source parent has no remaining children after the move, soft-delete it.
2. If the source parent's `threshold` now exceeds its remaining child count, reduce `threshold` to `newChildCount`.

This mirrors the existing pattern at lines 417–428 and restores the invariant `threshold ≤ number_of_children` after every reparenting operation. [5](#0-4) 

### Proof of Concept

**Setup:**
- Transaction T is created by User C.
- Approver tree: `Parent` (threshold=2) → [`ChildA` (userId=1), `ChildB` (userId=2)]
- A second root-level tree node `OtherParent` (threshold=1) → [`ChildX`] also exists on T.

**Trigger:**
```
PATCH /transactions/T/approvers/ChildA
Body: { "listId": <OtherParent.id> }
```

**Result after the call:**
- `ChildA.listId` = `OtherParent.id` ✓
- `Parent.threshold` = 2, `Parent.approvers` = [`ChildB`] (length=1) — **invariant violated**

**Consequence:**
- The approval condition for `Parent` requires 2 approvals but only 1 approver (`ChildB`) exists under it.
- `ChildB` can approve, but the threshold can never be reached.
- Transaction T is permanently stuck in `WAITING_FOR_SIGNATURES` through the `Parent` path.
- `emitTransactionStatusUpdate` fires after the update, potentially marking the transaction as needing approvals that can never be collected. [6](#0-5)
