### Title
Unbounded Full-Table Scan and Per-Row Async Processing in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`TransactionsService.getTransactionsToSign` fetches every non-terminal transaction from the database with no row limit, then iterates over all of them calling the expensive `userKeysToSign` (which deserializes transaction bytes and makes mirror-node network calls) for each row. Pagination is applied only after the full scan completes. Any authenticated user can trigger this endpoint repeatedly; as the transaction table grows, the server exhausts memory, CPU, and open connections, causing sustained service degradation or crash.

### Finding Description

**Root cause — missing `take` on the database query:**

In `getTransactionsToSign`, the `repo.find()` call has no `take` (SQL `LIMIT`) clause: [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED), so every `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION` transaction in the entire organization is loaded into the Node.js process heap.

**Unbounded per-row async loop:**

Every loaded transaction is then processed individually in a sequential `for...of` loop: [2](#0-1) 

Each iteration calls `userKeysToSign`, which internally calls `computeSignatureKey`: [3](#0-2) 

`computeSignatureKey` performs:
1. `SDKTransaction.fromBytes(transaction.transactionBytes)` — CPU-intensive protobuf deserialization
2. `accountCacheService.getAccountInfoForTransaction(...)` — outbound mirror-node HTTP call per signing account
3. Optionally `nodeCacheService.getNodeInfoForTransaction(...)` — another outbound call

**Pagination applied only after the full scan:** [4](#0-3) 

The `PaginationParams` decorator enforces a maximum page size of 100 on the *response*, but this has no effect on the database query or the loop: [5](#0-4) 

**Attacker-controlled entry path:**

The endpoint is reachable by any authenticated, verified user — no admin role required. The controller exposes it under `GET /transactions/sign`: [6](#0-5) 

A malicious user creates a large number of transactions (the `POST /transactions` endpoint is also available to any verified user with a key), then repeatedly polls `GET /transactions/sign`. Each poll forces the server to load and process every active transaction.

### Impact Explanation
- **Memory exhaustion**: Loading thousands of transaction rows (each containing serialized transaction bytes) into the Node.js heap can trigger OOM kills.
- **CPU exhaustion**: Protobuf deserialization of every transaction byte blob on every request saturates the event loop.
- **Connection pool exhaustion**: Concurrent mirror-node HTTP calls per transaction row can exhaust the outbound connection pool, blocking other services.
- **Cascading denial of service**: Because the loop is sequential and awaited, a single slow mirror-node response stalls the entire request, holding the HTTP connection open and blocking NestJS worker threads.
- All other API users are affected while the server is degraded.

### Likelihood Explanation
Any authenticated organization member can trigger this. The load-test seed data already demonstrates 500 active transactions as a realistic scale: [7](#0-6) 

At 500 active transactions, each request to `GET /transactions/sign` triggers 500 deserialization operations and up to 500+ outbound mirror-node calls. No special privilege or exploit is required beyond a valid JWT.

### Recommendation
1. **Apply `take` to the database query** before the loop, using the same `limit` from `Pagination`. Pre-filter at the SQL level using a join or subquery on `transaction_signer` to identify only transactions where the user's keys are relevant.
2. **Move the key-eligibility check into the database query** (e.g., join on `user_key` and `transaction_signer`) so the application layer never loads rows it will discard.
3. **Add a hard cap** on the number of rows processed per request, independent of the pagination size, as a defense-in-depth measure.

### Proof of Concept

**Preconditions:** Attacker is a registered, verified organization user with at least one user key.

1. Attacker calls `POST /transactions` 500+ times to create transactions in `WAITING_FOR_SIGNATURES` status (or waits for organic growth of the transaction table).
2. Attacker sends repeated `GET /transactions/sign?page=1&size=1` requests.
3. Despite requesting only 1 result, the server executes:
   - `SELECT * FROM transaction WHERE status NOT IN (...)` — returns all 500+ rows
   - 500+ iterations of `SDKTransaction.fromBytes(...)` + `computeSignatureKey(...)` + mirror-node HTTP calls
4. Server response time grows linearly with the number of active transactions. At sufficient scale, the Node.js process OOMs or the request times out, returning 5xx to all concurrent users.

  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
