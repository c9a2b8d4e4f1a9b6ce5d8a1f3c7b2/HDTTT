### Title
`handleContractOnBlur` in `SystemDataFormData.vue` incorrectly updates `fileId` instead of `contractId`, corrupting form state on blur

### Summary
In `front-end/src/renderer/components/Transaction/Create/SystemData/SystemDataFormData.vue`, the `handleContractOnBlur` function is a direct copy-paste of `handleFileOnBlur` with neither the target field name nor the source field corrected. When the Contract ID input loses focus, the handler emits an update that re-formats `props.data.fileId` using `formatContractId` and writes the result back into `fileId`, while `contractId` is never normalized. This is the exact same wrong-variable-assignment class as the external report.

### Finding Description
`SystemData` has two distinct fields:

```typescript
export type SystemData = {
  fileId: string;
  contractId: string;
};
``` [1](#0-0) 

The component defines two blur handlers:

```typescript
function handleFileOnBlur() {
  emit('update:data', { ...props.data, fileId: formatAccountId(props.data.fileId) });
}

function handleContractOnBlur() {
  emit('update:data', { ...props.data, fileId: formatContractId(props.data.fileId) });
  //                                   ^^^^^^ wrong field name          ^^^^^^ wrong source field
}
``` [2](#0-1) 

`handleContractOnBlur` is bound to the Contract ID `<AppInput>` via `@blur="handleContractOnBlur"`: [3](#0-2) 

When the user tabs out of the Contract ID field, two wrong things happen simultaneously:
1. `props.data.fileId` is read and passed through `formatContractId` (a formatter intended for contract IDs, not file IDs), and the result is written back to `fileId` — potentially corrupting the file ID value.
2. `contractId` is never touched, so the user's raw contract ID input is never normalized.

This shared form component is used by both `SystemDelete` and `SystemUndelete` transaction flows: [4](#0-3) [5](#0-4) 

### Impact Explanation
Every time a user fills in the Contract ID field and moves focus away, the reactive `data` object held by the parent (`SystemDelete` / `SystemUndelete`) receives a corrupted `fileId` (re-formatted with the wrong formatter) while `contractId` remains un-normalized. The downstream `setSystemData` function reads both fields to build the actual Hedera SDK transaction: [6](#0-5) 

A corrupted `fileId` can cause `isFileId()` to return false, which silently drops the file ID from the transaction or causes `preCreateAssert` to throw, blocking transaction creation entirely. The `contractId` is never normalized, so it may be submitted in a non-canonical format that the Hedera network rejects. Both `SystemDeleteTransaction` and `SystemUndeleteTransaction` are privileged system-level operations; incorrect or missing IDs mean the intended system file or contract is not acted upon.

### Likelihood Explanation
This triggers on every normal user interaction: entering a Contract ID and pressing Tab or clicking elsewhere. No attacker involvement is required — any user of the System Delete or System Undelete transaction forms will hit this on every use. The bug is deterministic and 100% reproducible.

### Recommendation
Fix `handleContractOnBlur` to read from and write to `contractId`:

```diff
function handleContractOnBlur() {
-  emit('update:data', { ...props.data, fileId: formatContractId(props.data.fileId) });
+  emit('update:data', { ...props.data, contractId: formatContractId(props.data.contractId) });
}
``` [7](#0-6) 

### Proof of Concept
1. Open the application and navigate to **System Delete** (or **System Undelete**).
2. Enter a valid File ID (e.g., `0.0.101`) in the File ID input.
3. Enter a valid Contract ID (e.g., `0.0.1234`) in the Contract ID input.
4. Click outside the Contract ID input (blur event fires).
5. Observe: `handleContractOnBlur` emits `{ fileId: formatContractId("0.0.101"), contractId: "0.0.1234" }`. The `fileId` is now processed by `formatContractId` (wrong formatter), potentially changing its value. The `contractId` remains as raw user input, never normalized.
6. Attempt to create the transaction — `preCreateAssert` may throw `"Invalid File ID"` even though the user entered a valid one, or the transaction is built with a non-canonical contract ID.
