### Title
TOCTOU Race Condition Allows Duplicate `transactionId` Registration in `createTransactions`

### Summary
The `createTransactions` function in `back-end/apps/api/src/transactions/transactions.service.ts` performs a duplicate `transactionId` check **outside** the database transaction that saves the new records. Because there is no database-level unique constraint on the `transactionId` column, two concurrent requests carrying the same `transactionId` can both pass the check and both be persisted, creating duplicate records. This is the direct analog of the external report's `orderId` replay vulnerability: an identifier that is supposed to be unique is not validated atomically, allowing it to be registered more than once.

### Finding Description

**Root cause — check/use separation (TOCTOU):**

In `createTransactions`, the duplicate guard reads from the database at lines 415–427, then the actual `INSERT` happens inside a separate `entityManager.transaction()` block at line 436–462. There is no database-level `UNIQUE` constraint on `transactionId` (the `@Column()` decorator carries no `unique: true`), so nothing prevents a second concurrent writer from winning the race.

```
[Request A] repo.find(transactionId) → [] (no existing)
[Request B] repo.find(transactionId) → [] (no existing, A not yet saved)
[Request A] entityManager.save(Transaction) → row inserted
[Request B] entityManager.save(Transaction) → second row inserted (same transactionId)
``` [1](#0-0) 

The `Transaction` entity confirms no uniqueness enforcement at the column level: [2](#0-1) 

**Exploit path:**

1. Attacker (or any authenticated user) constructs a valid Hedera transaction with a chosen `transactionId` (format: `payerAccountId@validStart`, e.g. `0.0.1003@1716577920.000000000`).
2. Sends two concurrent `POST /transactions` requests with identical `transactionBytes` (and therefore identical `transactionId`).
3. Both requests pass `validateAndPrepareTransaction`, both pass the `repo.find` duplicate check, and both reach `entityManager.save`.
4. Two `Transaction` rows with the same `transactionId` are committed to PostgreSQL.

**State after exploit:**

- Both rows enter `WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION`.
- The chain service submits the first to Hedera (succeeds, row → `EXECUTED`).
- The chain service submits the second; Hedera returns `DUPLICATE_TRANSACTION`. The execute service silently drops the update: [3](#0-2) 

- The second row is **permanently stranded** in `WAITING_FOR_EXECUTION` (or `FAILED` depending on timing), corrupting the database state.

`getTransactionById` has a partial workaround (prefer non-inactive rows), but it does not remove the phantom row: [4](#0-3) 

### Impact Explanation

- **Database integrity corruption**: Multiple rows share a `transactionId` that the schema documents as unique. Downstream queries (signers, approvers, observers, notifications) that join on `transactionId` may operate on the wrong row.
- **Phantom transactions**: The stranded duplicate row remains visible in the UI and API responses, showing a transaction that will never complete.
- **Notification/audit integrity**: Off-chain consumers (audit logs, notification subscribers) receive duplicate `transaction_created` events for the same Hedera transaction, mirroring the regulatory/database-integrity concern described in the external report.
- **Scheduler resource waste**: The scheduler may repeatedly attempt to execute the stranded row.

### Likelihood Explanation

- **Preconditions**: Any authenticated, verified user. No privileged access required.
- **Ease**: A single HTTP client sending two concurrent requests with the same body is trivially achievable (e.g., `Promise.all([fetch(...), fetch(...)])`).
- **No rate-limit barrier**: The API has no per-`transactionId` idempotency key or request-level mutex.
- **Window**: The race window spans the entire async path from `repo.find` to `entityManager.save`, which includes network I/O to the Hedera mirror node (`getClientFromNetwork`, `validateAndPrepareTransaction`), making the window wide enough to be reliably hit.

### Recommendation

1. **Add a database-level unique constraint** on `transactionId` in the `Transaction` entity and a corresponding migration. This is the only reliable guard:
   ```typescript
   @Column({ unique: true })
   transactionId: string;
   ```
   With this in place, the second concurrent `INSERT` will throw a unique-violation error, which the existing `catch` block can surface as `BadRequestException(ErrorCodes.FST)`.

2. **Move the duplicate check inside the database transaction** (or remove it in favour of relying on the constraint + error handling), eliminating the TOCTOU window entirely.

3. Optionally, add an application-level idempotency key per request to reject retries before they reach the database.

### Proof of Concept

```typescript
// Two concurrent requests with identical transactionBytes (same transactionId)
const body = {
  name: 'Test',
  description: 'PoC',
  transactionBytes: '<hex of frozen AccountCreateTransaction with fixed validStart>',
  creatorKeyId: 1,
  signature: '<valid signature>',
  mirrorNetwork: 'testnet',
};

const [res1, res2] = await Promise.all([
  fetch('https://<api>/transactions', { method: 'POST', body: JSON.stringify(body), headers }),
  fetch('https://<api>/transactions', { method: 'POST', body: JSON.stringify(body), headers }),
]);

// Expected (correct): one 201, one 400 "Transactions already exist"
// Actual (vulnerable): two 201 responses; two rows in the Transaction table
//   with the same transactionId, one of which will be permanently stranded.
console.log(res1.status, res2.status); // 201 201
```

After the two requests complete, querying the database directly:
```sql
SELECT id, "transactionId", status FROM "transaction"
WHERE "transactionId" = '0.0.1003@1716577920.000000000';
-- Returns 2 rows instead of 1
``` [5](#0-4)

    try {
      // Validate all DTOs upfront

        dtos.map(dto => this.validateAndPrepareTransaction(dto, user, client)),

      // Batch check for existing transactions
      const transactionIds = validatedData.map(v => v.transactionId);
      const existing = await this.repo.find({
        where: {
          transactionId: In(transactionIds),
          status: Not(
            In([
              TransactionStatus.CANCELED,
              TransactionStatus.REJECTED,
              TransactionStatus.ARCHIVED,
            ]),
          ),
        },
        select: ['transactionId'],
      });

      if (existing.length > 0) {
        throw new BadRequestException(

      const savedTransactions = await this.entityManager.transaction(async (entityManager) => {
        const transactions = validatedData.map(data =>
