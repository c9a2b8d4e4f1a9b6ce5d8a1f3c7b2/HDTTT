### Title
Manual Transaction `archiveTransaction` Bypasses Terminal-State Guard, Allowing Unauthorized State Transition

### Summary
`archiveTransaction` in `transactions.service.ts` is intended to only operate on transactions in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` states. However, a logical flaw in the guard condition causes the entire status check to be skipped whenever `transaction.isManual === true`. This allows the transaction creator to archive a manual transaction that is already in a terminal state (`EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, `REJECTED`), overwriting the terminal status with `ARCHIVED` and corrupting the authoritative state record.

### Finding Description

**Root cause — flawed boolean guard:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 711–718:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← when isManual=true this is false
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard is a conjunction (`A && B`). When `isManual === true`, `B` (`!transaction.isManual`) evaluates to `false`, making the entire expression `false` regardless of `A` (the status check). The exception is never thrown, and execution falls through to:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [1](#0-0) 

The codebase itself defines `terminalStatuses` as a closed set that includes `EXECUTED`, `FAILED`, `CANCELED`, `EXPIRED`, `ARCHIVED`, and `REJECTED`: [2](#0-1) 

Transitioning from one terminal state to another (`EXECUTED → ARCHIVED`, `FAILED → ARCHIVED`, etc.) is never intended and has no guard anywhere else in the pipeline.

**Exploit path:**

1. Attacker (the transaction creator) creates a manual transaction (`isManual = true`).
2. The transaction is executed on the Hedera network; its status becomes `EXECUTED`.
3. Attacker calls `PATCH /transactions/archive/:id` (or equivalent endpoint backed by `archiveTransaction`).
4. Because `isManual === true`, the status guard is bypassed.
5. The DB record is updated: `status = ARCHIVED`.
6. The authoritative record of the on-chain execution is permanently overwritten.

The same path applies to `FAILED`, `CANCELED`, `EXPIRED`, and `REJECTED` manual transactions.

**Contrast with correct paths:**

- `cancelTransactionWithOutcome` explicitly checks `cancelableStatuses` and throws `OTIP` for terminal states. [3](#0-2) 
- `signers.service.ts::validateTransactionStatus` correctly rejects any status outside `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION`. [4](#0-3) 
- `approvers.service.ts::approveTransaction` explicitly checks for `WAITING_FOR_SIGNATURES` / `WAITING_FOR_EXECUTION` and throws `TNRA` otherwise. [5](#0-4) 

`archiveTransaction` is the only state-mutating function that omits this pattern for manual transactions.

### Impact Explanation

A transaction creator can permanently overwrite the status of any manual transaction — including one that has already been `EXECUTED` on the Hedera network — with `ARCHIVED`. This constitutes:

- **Unauthorized state change**: A terminal, immutable state (`EXECUTED`, `FAILED`) is silently replaced, violating the invariant that terminal states are final.
- **Audit-trail corruption**: The system's record of whether a transaction succeeded or failed is destroyed. Downstream consumers (notifications, history views, compliance logs) will see `ARCHIVED` instead of the true outcome.
- **Irreversibility**: There is no un-archive operation; the original terminal status cannot be recovered from the database.

### Likelihood Explanation

- **Attacker profile**: Any authenticated organization user who is the creator of a manual transaction. No privileged access is required.
- **Precondition**: The attacker must own the transaction (`getTransactionForCreator` enforces creator-only access). This is a normal, reachable condition for any user who creates transactions.
- **Trigger**: A single authenticated API call to the archive endpoint with a valid transaction ID.
- **Detection difficulty**: The operation emits a `TransactionStatusUpdate` notification, but the notification payload does not expose the previous status, so the overwrite is not self-evidently anomalous to observers.

### Recommendation

Replace the flawed conjunction with an explicit allowlist that is independent of `isManual`:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

if (!archivableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions genuinely need a broader archivable set (e.g., also `NEW`), enumerate those states explicitly rather than using `isManual` as a bypass flag. The `isManual` flag controls execution routing, not lifecycle permissions, and must not be used as a status-check bypass.

### Proof of Concept

**Preconditions**: Authenticated as the creator of a manual transaction that has been executed (status = `EXECUTED`).

```
# Step 1 – Create a manual transaction (isManual=true) and let it execute.
# After execution, DB status = EXECUTED.

# Step 2 – Call the archive endpoint as the creator:
PATCH /transactions/archive/{transactionId}
Authorization: Bearer <creator_jwt>

# Expected (correct) response: 400 Bad Request (OMTIP)
# Actual response:             200 OK

# Step 3 – Verify:
GET /transactions/{transactionId}
# Returns status: "ARCHIVED" instead of "EXECUTED"
```

**Deterministic reasoning**: `isManual === true` → `!transaction.isManual === false` → `(statusCheck) && false === false` → guard never throws → `repo.update({ status: ARCHIVED })` executes unconditionally. [1](#0-0)

  async archiveTransaction(id: number, user: User): Promise<boolean> {
    const transaction = await this.getTransactionForCreator(id, user);

    if (
      ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
        transaction.status,
      ) &&
      !transaction.isManual
    ) {
      throw new BadRequestException(ErrorCodes.OMTIP);
    }

    await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }
