### Title
`createTransactions()` Uses Only the First DTO's Network to Validate All Transactions in a Batch, Enabling Cross-Network Validation Bypass

### Summary
`TransactionsService.createTransactions()` accepts an array of `CreateTransactionDto` objects, each carrying its own `mirrorNetwork` field. However, a single Hedera `Client` is constructed from `dtos[0].mirrorNetwork` and then reused to validate every DTO in the batch. Transactions belonging to a different network than the first DTO are frozen and node-validated against the wrong network's client, causing either silent mis-validation or incorrect rejection of legitimate transactions.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the batch creation path is:

```typescript
// line 405 — client built from FIRST dto only
const client = await getClientFromNetwork(dtos[0].mirrorNetwork);

// lines 409-411 — ALL dtos validated with that single client
const validatedData = await Promise.all(
  dtos.map(dto => this.validateAndPrepareTransaction(dto, user, client)),
);
``` [1](#0-0) 

Inside `validateAndPrepareTransaction`, the shared `client` is used for two critical operations:

```typescript
// line 920 — freezes with wrong-network client if not already frozen
if (!sdkTransaction.isFrozen()) {
  sdkTransaction.freezeWith(client);
}

// lines 934-936 — node validation uses wrong-network's node set
const allowedNodes = getNodeAccountIdsFromClientNetwork(client);
if (!isTransactionValidForNodes(sdkTransaction, allowedNodes)) {
  throw new BadRequestException(ErrorCodes.TNVN);
}
``` [2](#0-1) 

`getClientFromNetwork` builds a network-specific client whose node set is entirely determined by the network name (mainnet, testnet, previewnet, or custom): [3](#0-2) 

The caller `createTransactionGroup` passes all group-item transactions in a single `createTransactions` call: [4](#0-3) 

**Exploit path:**
1. Attacker (or any normal user) creates a transaction group via `POST /transaction-groups` with two group items: item 0 has `mirrorNetwork: "testnet"`, item 1 has `mirrorNetwork: "mainnet"`.
2. `createTransactions` builds a testnet `Client`.
3. Item 1 (a mainnet transaction) is validated with the testnet client: its mainnet node IDs are checked against testnet's allowed node set → `TNVN` error, rejecting the entire batch even though the mainnet transaction is perfectly valid.
4. Conversely, if item 0 is mainnet and item 1 is testnet, the testnet transaction is frozen with the mainnet client (wrong node/ledger context) and its testnet node IDs are checked against mainnet nodes — the transaction may pass node validation incorrectly if node IDs happen to overlap, and is then stored with `mirrorNetwork: "testnet"` but frozen for mainnet, causing execution failure later.

### Impact Explanation
- **Incorrect rejection (DoS):** Any batch containing transactions for more than one network will fail with `TNVN` for all non-first-network transactions, making multi-network transaction groups permanently unusable.
- **Silent mis-freeze:** If a non-first DTO's transaction is not yet frozen, it is frozen with the wrong network's client, embedding incorrect node/ledger metadata. The transaction is then stored and later submitted to the wrong network, causing execution failure or unexpected behavior.
- **Integrity violation:** The stored `transactionBytes` may contain node IDs from the wrong network, creating a persistent inconsistency between the stored `mirrorNetwork` field and the actual transaction bytes.

### Likelihood Explanation
The vulnerability is reachable by any authenticated user with no special privileges. The `createTransactionGroup` endpoint is a standard user-facing API. A user who legitimately wants to group a mainnet and a testnet transaction (or any two different networks) will trigger this bug. The code path is exercised every time a transaction group is created with heterogeneous networks.

### Recommendation
Replace the single shared client with per-DTO client creation inside `validateAndPrepareTransaction`, or validate that all DTOs share the same `mirrorNetwork` before creating the client and reject the batch if they differ:

```typescript
// Option A: reject mixed networks upfront
const networks = new Set(dtos.map(d => d.mirrorNetwork));
if (networks.size > 1) {
  throw new BadRequestException('All transactions in a batch must share the same mirrorNetwork');
}

    const publicKey = PublicKey.fromString(creatorKey.publicKey);

    const validSignature = publicKey.verify(dto.transactionBytes, dto.signature);

export const getClientFromNetwork = async (
  mirrorNetwork: string | string[],
  ledgerId?: string | LedgerId,
) => {
  if (!Array.isArray(mirrorNetwork)) {
    mirrorNetwork = [mirrorNetwork];
  }

  mirrorNetwork = mirrorNetwork.map(network => network.toLocaleLowerCase());
  if ([MAINNET, TESTNET, PREVIEWNET].includes(mirrorNetwork[0])) {
    return Client.forName(mirrorNetwork[0]);
  }

  if (mirrorNetwork[0] === LOCAL_NODE) {
    return Client.forNetwork(getLocalClientNetwork(process.env.NODE_ENV)).setMirrorNetwork(
      MirrorNetworkGRPC.LOCAL_NODE,
    );
  }

  const client = Client.forNetwork({}).setMirrorNetwork(
    MirrorNetworkGRPC.fromBaseURL(mirrorNetwork[0]),
  );

  const nodeAddressBook = await new AddressBookQuery()
    .setFileId(FileId.ADDRESS_BOOK)
    .execute(client);

  client.setNetworkFromAddressBook(nodeAddressBook);

  if (ledgerId) {
    client.setLedgerId(ledgerId);
  }

  return client;
};
