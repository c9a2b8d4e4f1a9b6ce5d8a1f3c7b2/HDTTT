### Title
Attacker-Controlled Throttle Key in `EmailThrottlerGuard` Enables Targeted DoS on Password Reset for Any User

### Summary
The `POST /auth/reset-password` endpoint is unauthenticated and protected only by `EmailThrottlerGuard`, which keys its rate-limit counter on the **email address supplied in the request body**. Because the attacker fully controls that value, they can exhaust the per-email rate-limit window for any victim's email address with cheap, unauthenticated HTTP requests, permanently blocking that user from initiating a password reset for as long as the attacker keeps sending requests.

### Finding Description

**Vulnerability class:** DoS — attacker-controlled rate-limit key (resource exhaustion via shared-limit exhaustion; direct analog to the Teleportation `maxTransferAmountPerDay` exhaustion).

**Root cause — `EmailThrottlerGuard.getTracker()`**

`back-end/apps/api/src/guards/email-throttler.guard.ts`, lines 7–13:

```typescript
protected getTracker(req: Record<string, any>): Promise<string> {
  const email = req.body.email;   // ← attacker-supplied value becomes the throttle key
  if (!email) {
    throw new HttpException('No email specified.', HttpStatus.BAD_REQUEST);
  }
  return email;
}
```

The NestJS `ThrottlerGuard` uses the value returned by `getTracker()` as the Redis key for counting requests. Here that key is the victim's email address, not the attacker's IP.

**Rate-limit configuration — `EmailThrottlerModule`**

`back-end/apps/api/src/throttlers/email-throttler.module.ts`, lines 14–22:

```typescript
throttlers: [
  { name: 'anonymous-minute',      ttl: seconds(60), limit: ANONYMOUS_MINUTE_LIMIT },
  { name: 'anonymous-five-second', ttl: seconds(5),  limit: ANONYMOUS_FIVE_SECOND_LIMIT },
],
```

Both limits are env-var-driven. Once either bucket is full for a given email key, every subsequent request for that email returns HTTP 429 until the TTL expires.

**Vulnerable endpoint — `AuthController.createOtp()`**

`back-end/apps/api/src/auth/auth.controller.ts`, lines 134–139:

```typescript
@Post('/reset-password')
@HttpCode(200)
@UseGuards(EmailThrottlerGuard)          // ← only guard; no IP guard, no auth
async createOtp(@Body() { email }: OtpLocalDto) {
  return this.authService.createOtp(email);
}
```

No `JwtAuthGuard`, no `IpThrottlerGuard`, no CAPTCHA — the endpoint is fully open to unauthenticated callers.

**End-to-end exploit path:**

1. Attacker sends `ANONYMOUS_MINUTE_LIMIT + 1` POST requests to `/auth/reset-password` with body `{ "email": "victim@example.com" }` within a 60-second window.
2. The Redis counter for key `victim@example.com` reaches the limit.
3. Every subsequent request from **anyone** (including the legitimate victim) for that email returns HTTP 429 for the remainder of the TTL window.
4. Attacker repeats the burst every 60 seconds to maintain the block indefinitely.
5. The victim is permanently unable to initiate a password reset.

### Impact Explanation

The victim loses access to the only self-service account-recovery mechanism. If the victim has forgotten their password (the exact scenario this endpoint exists for), they are permanently locked out of the organization backend with no alternative recovery path. The attacker can sustain the block at negligible cost (unauthenticated HTTP requests, no tokens, no funds required).

### Likelihood Explanation

- **No authentication required** — any internet-reachable client can call the endpoint.
- **No cost** — only HTTP requests; no tokens, no credentials, no on-chain assets.
- **Victim email is not secret** — organization members' emails are typically known within the org.
- **Sustained attack is trivial** — a simple loop or cron job maintains the block indefinitely.
- **No IP-based back-pressure** — `IpThrottlerGuard` is not applied to this endpoint, so the attacker's own IP is never throttled.

### Recommendation

Replace the email-keyed tracker with the **attacker's IP address** for the anonymous `/auth/reset-password` endpoint. The email key is appropriate only for endpoints where the caller is already authenticated (so the email identifies the caller, not a victim). Concretely:

1. Apply `IpThrottlerGuard` (already implemented at `back-end/apps/api/src/guards/ip-throttler.guard.ts`) to `POST /auth/reset-password` instead of, or in addition to, `EmailThrottlerGuard`.
2. If per-email limiting is still desired to prevent email-bombing, apply it **after** IP limiting and ensure the IP limit is strict enough to prevent exhaustion of the email bucket from a single source.

### Proof of Concept

```bash
# Exhaust the per-email rate-limit window for victim@example.com
# (replace LIMIT with the value of ANONYMOUS_MINUTE_LIMIT in production)
for i in $(seq 1 $((LIMIT + 1))); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<api-host>/auth/reset-password \
    -H 'Content-Type: application/json' \
    -d '{"email":"victim@example.com"}'
done

# Expected output: 200 repeated LIMIT times, then 429 on the final request.
# After this point, the legitimate victim receives 429 for every reset attempt
# until the 60-second TTL expires — which the attacker refreshes continuously.
```

**Expected outcome:** The victim's `POST /auth/reset-password` returns HTTP 429 for the full TTL window. Repeating the burst every 60 seconds maintains the block indefinitely with no privileged access required. [1](#0-0) [2](#0-1) [3](#0-2)
