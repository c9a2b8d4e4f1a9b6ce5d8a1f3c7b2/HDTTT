### Title
Any Transaction Creator Can Remove Approvers From Other Users' Transactions

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` verifies that the calling user is the creator of the transaction identified by `:transactionId`, but then passes the unrelated `:id` (approver ID) directly to `removeTransactionApprover()` without verifying that the approver belongs to the authorized transaction. A malicious authenticated user can use their own `transactionId` to pass the creator check, then supply an approver ID from any other transaction to delete it.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, the `removeTransactionApprover` handler performs two sequential calls:

```
await this.approversService.getCreatorsTransaction(transactionId, user);  // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);                // deletes approver by id — no binding check
``` [1](#0-0) 

The first call confirms the authenticated user is the creator of `transactionId`. The second call, `removeTransactionApprover(id)`, only looks up the approver by its primary key and deletes it:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
``` [2](#0-1) 

There is **no check** that the approver with `id` belongs to the transaction with `transactionId`. The `updateTransactionApprover` service method correctly performs this cross-reference check (`rootNode.transactionId !== transactionId`), but `removeTransactionApprover` does not. [3](#0-2) 

The authorization check and the destructive action are decoupled: the `transactionId` path parameter is used only for the creator check, while the `id` path parameter is used for the actual deletion with no binding between the two.

### Impact Explanation

A malicious authenticated organization member who is the creator of at least one transaction can:

1. Enumerate approver IDs from other transactions (sequential integer IDs, discoverable via `GET /transactions/:id/approvers` on any transaction they have observer/signer access to, or by brute-force).
2. Call `DELETE /transactions/{their_own_tx_id}/approvers/{victim_approver_id}`.
3. The creator check passes (they own `their_own_tx_id`), and the victim's approver is deleted.

This allows the attacker to silently remove required approvers from any transaction in the organization, breaking the approval threshold, causing transactions to execute without required sign-off, or permanently corrupting the approval workflow for high-value governance transactions.

### Likelihood Explanation

- **Attacker preconditions**: Must be a registered, verified organization user and creator of at least one transaction (trivially achievable by creating any transaction).
- **No privileged access required**: Standard user role is sufficient.
- **Approver ID discovery**: IDs are sequential integers. Any transaction the attacker can view (as observer, signer, or approver) leaks approver IDs via the `GET` endpoint. Brute-force over a small integer range is also feasible.
- **No cryptographic barrier**: The attack is a straightforward HTTP DELETE with a crafted URL.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the authorized `transactionId`. Mirror the check already present in `updateTransactionApprover`:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, entityManager);
if (!rootNode || rootNode.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
``` [4](#0-3) 

### Proof of Concept

1. Attacker (User A) registers and creates Transaction A (`id=10`). Attacker is its creator.
2. Victim (User B) creates Transaction B (`id=20`) with an approver record (`approver.id=99`, `approver.transactionId=20`). Attacker learns `approver.id=99` by observing Transaction B or by brute-forcing the integer range.
3. Attacker sends:
   ```
   DELETE /transactions/10/approvers/99
   Authorization: Bearer <attacker_jwt>
   ```
4. `getCreatorsTransaction(10, attackerUser)` → passes (attacker owns tx 10).
5. `removeTransactionApprover(99)` → fetches approver 99 (belongs to tx 20), calls `removeNode(99)`, soft-deletes it.
6. Approver 99 is permanently removed from Transaction B's approval chain without any authorization from Transaction B's creator. [1](#0-0) [2](#0-1)

  /* Removes the transaction approver by id */
  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }
