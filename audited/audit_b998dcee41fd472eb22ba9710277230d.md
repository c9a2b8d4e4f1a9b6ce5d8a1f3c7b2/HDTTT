### Title
`TransactionsService.getTransactionsToSign`: Unbounded Full-Table Scan Enables Attacker-Controlled Resource Exhaustion

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no row limit, then iterates over every row in application memory to determine whether the requesting user must sign each one. Because any authenticated user can create transactions (growing the dataset), a malicious user can inflate the transaction table to make this endpoint consume unbounded memory and CPU for every other user who calls it, causing request timeouts or server-wide degradation.

### Finding Description

**Root cause — missing `take` limit on a full-table query:**

In `getTransactionsToSign`, the `whereForUser` predicate filters only on `status` (not on user identity), and the subsequent `repo.find` call carries no `take` / `limit` clause: [1](#0-0) 

```typescript
const whereForUser: FindOptionsWhere<Transaction> = {
  ...where,
  status: Not(In([EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED])),
};
// ↓ No `take:` — loads every non-terminal transaction in the system
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
```

Every row returned is then processed in a sequential `for` loop that calls `userKeysToSign` (a cryptographic operation) per transaction: [2](#0-1) 

Pagination is applied only **after** the full in-memory scan: [3](#0-2) 

**Exploit path:**

1. Attacker registers as a normal user (no privilege required).
2. Attacker repeatedly calls `POST /transactions`, creating valid Hedera transactions. Each transaction is cheap to produce and persists in `WAITING_FOR_SIGNATURES` status indefinitely (until it expires or is canceled by the creator).
3. Any victim user who calls `GET /transactions/sign` triggers a full scan of the ever-growing transaction table plus one `userKeysToSign` cryptographic call per row.
4. As the table grows, response latency increases linearly; eventually requests time out or the Node.js process runs out of memory, denying service to all users of the endpoint.

The attacker does not need to target a specific victim — the degradation is system-wide.

### Impact Explanation

- **Availability**: `getTransactionsToSign` becomes unusable for all organization members once the transaction table is large enough. Users cannot discover which transactions require their signature, blocking the core signing workflow.
- **Severity**: High. The signing workflow is the central feature of the application. Blocking it permanently freezes all pending multi-signature transactions.
- **No data theft**: The attacker cannot read or modify other users' data; the impact is purely availability.

### Likelihood Explanation

- **Attacker preconditions**: Only a valid account in the organization is required — no admin keys, no leaked credentials.
- **Cost**: Creating each transaction requires a valid Hedera transaction body signed with the attacker's own key. On Hedera mainnet this costs a small fee per transaction; on testnet it is free. The attacker can accumulate thousands of transactions over time even under rate limiting.
- **Persistence**: Transactions remain in `WAITING_FOR_SIGNATURES` until their `validStart` expires (Hedera transactions have a 3-minute validity window by default, but the application stores them and they remain in the DB). The attacker simply needs to keep creating new ones faster than they expire.

### Recommendation

1. **Push the user-relevance filter to the database.** Add a subquery or join so only transactions where the user's registered public keys appear in `publicKeys` (or where the user is already a signer/observer/creator) are fetched. This makes the query O(relevant rows) instead of O(all rows).

2. **Apply `take` at the database level**, not after the in-memory loop. The current pattern of `result.slice(offset, offset + limit)` after a full scan defeats pagination entirely.

3. **Add a hard cap** on the number of transactions a single user may have in non-terminal states simultaneously (e.g., via a database constraint or a pre-insert count check in `createTransactions`).

### Proof of Concept

```
# Attacker (authenticated as user A) runs in a loop:
for i in $(seq 1 5000); do
  POST /transactions  { valid Hedera tx bytes signed by A's key, mirrorNetwork: "testnet" }
done

# Victim (user B) calls:
GET /transactions/sign?page=1&size=10

# Expected: returns 10 items quickly
# Actual: server loads all 5000+ transactions into memory, calls userKeysToSign
#         for each, request times out or OOM-kills the process
```

The exact threshold depends on server memory and the configured HTTP request timeout, but the growth is linear and unbounded. [4](#0-3)

```typescript
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
