### Title
Docker Containers Run as Root Due to Missing `USER` Instruction Across All Service Dockerfiles

### Summary
All five Dockerfiles in the `back-end/` directory are missing a `USER` instruction. Every container — API, chain, notifications, typeorm migration, and Redis — starts and runs its entrypoint process as `root` (uid 0). An attacker who achieves code execution inside any of these containers gains full root privileges within the container, bypassing any file-permission controls on sensitive assets such as TLS certificates and environment-injected secrets.

### Finding Description
None of the five production Dockerfiles issue a `USER` instruction before their `CMD`/`ENTRYPOINT`:

- `back-end/apps/api/Dockerfile` — no `USER` before `CMD ["node", "dist/apps/api/main"]` [1](#0-0) 

- `back-end/apps/chain/Dockerfile` — no `USER` before `CMD ["node", "dist/apps/chain/main"]` [2](#0-1) 

- `back-end/apps/notifications/Dockerfile` — no `USER` before `CMD ["node", "dist/apps/notifications/main"]` [3](#0-2) 

- `back-end/typeorm/Dockerfile` — no `USER` before `CMD ["pnpm", "-F", "typeorm", "migration:run"]` [4](#0-3) 

- `back-end/redis/Dockerfile` — no `USER` before `CMD ["redis-server", ...]` [5](#0-4) 

The base image `node:22.12.0-alpine` ships with a built-in unprivileged user `node` (uid 1000), but it is never activated. The `api` production stage additionally copies a `cert/` directory (TLS certificates) into the image at root-owned paths, and all services consume `.env` files containing database credentials, Hedera operator keys, and other secrets via `env_file` directives. [6](#0-5) [7](#0-6) 

### Impact Explanation
If an attacker exploits a vulnerability in the Node.js application (e.g., a deserialization flaw, a dependency with an RCE CVE, or a path-traversal that allows arbitrary code execution), they drop into the container as `root`. From that position they can:
- Read all environment variables (Hedera operator private keys, database passwords, JWT secrets).
- Read TLS private keys copied into the `api` image under `cert/`.
- Write to any path in the container filesystem, including overwriting the application binary.
- Potentially escape the container if the host kernel has a known privilege-escalation vulnerability, since root inside the container maps to root on the host when user-namespace remapping is not configured.

### Likelihood Explanation
The Node.js ecosystem has a history of supply-chain and dependency RCE vulnerabilities. The services use `pnpm install --prod` without a lockfile integrity check at runtime, and the `chain` service interacts directly with the Hedera network, making it an attractive target. Running as root is the default when no `USER` instruction is present, so this condition is guaranteed to be active in every deployment using these Dockerfiles.

### Recommendation
In each Dockerfile's production stage, add a `USER` instruction before the `CMD`. The `node:alpine` base image already provides the `node` user:

```dockerfile
# After all COPY/RUN steps, before CMD:
USER node
CMD ["node", "dist/apps/api/main"]
```

For the `redis` image, use the built-in `redis` user provided by the official Redis base image:

```dockerfile
FROM redis
COPY ./redis.conf /usr/local/etc/redis/redis.conf
USER redis
CMD ["redis-server", "/usr/local/etc/redis/redis.conf"]
```

Ensure that any files the process needs to read at runtime (e.g., `cert/`) are `COPY`-ed before the `USER` switch and have permissions readable by the unprivileged user.

### Proof of Concept

1. Build and start the production stack: `docker compose -f back-end/docker-compose.prod.yaml up --build -d`
2. Exec into the API container: `docker exec -it <api_container_id> sh`
3. Run `id` — output will be `uid=0(root) gid=0(root) groups=0(root)`, confirming root execution.
4. Run `cat /proc/1/environ | tr '\0' '\n'` to dump all environment variables, including any Hedera operator private key or database password injected via `env_file`. [8](#0-7)
