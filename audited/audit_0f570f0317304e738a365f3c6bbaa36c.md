### Title
Inverted Boolean Condition in `approveTransaction` Causes Approval Events to Never Trigger Transaction Status Re-evaluation

### Summary

In `approvers.service.ts`, the `approveTransaction` method contains an inverted boolean condition that determines which NATS event to emit after an approver submits their choice. When a user **approves** a transaction (`dto.approved = true`), only a lightweight `emitTransactionUpdate` is emitted (a UI-only data refresh). The `emitTransactionStatusUpdate` — which triggers the chain service to re-evaluate whether the transaction can proceed to execution — is only emitted on **rejection** (`dto.approved = false`). This is the direct analog of the external report's `burn` flag inversion: the flag controlling a critical downstream action is evaluated with the wrong polarity.

### Finding Description

**Root cause — inverted condition on stale data:**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts` lines 598–618, after the DB update is committed, the notification branch is:

```typescript
// userApprovers was fetched BEFORE the DB update (line 556)
// After the update, in-memory approved fields are still null/undefined

if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
``` [1](#0-0) 

`userApprovers` is populated at line 556 **before** the DB write at lines 599–610. After the write, the in-memory array still holds the pre-update values (`approved: null/undefined`). Therefore `userApprovers.every(a => a.approved)` is always `false` (falsy `null`/`undefined` fails the predicate). [2](#0-1) 

The condition therefore collapses to:

| `dto.approved` | `!dto.approved` | `every(...)` | Branch taken |
|---|---|---|---|
| `true` (approve) | `false` | `false` (stale) | `emitTransactionUpdate` ← **wrong** |
| `false` (reject) | `true` | irrelevant | `emitTransactionStatusUpdate` ← correct |

**What each event does:**

- `emitTransactionStatusUpdate` → publishes to `TRANSACTION_STATUS_UPDATE` NATS subject → consumed by the chain service and notifications service to re-evaluate transaction status (check if all approvals are satisfied, advance to execution, send approval notifications).
- `emitTransactionUpdate` → publishes to `TRANSACTION_UPDATE` NATS subject → consumed only for UI refresh; no status re-evaluation occurs. [3](#0-2) 

**Exploit path:**

1. An organization transaction is created requiring one or more approvers.
2. All designated approvers call `POST /{transactionId}/approvers/approve` with `approved: true`.
3. Each call reaches `approveTransaction`, writes the approval to the DB, but emits only `emitTransactionUpdate`.
4. The chain service never receives `TRANSACTION_STATUS_UPDATE` for this transaction.
5. The transaction status is never re-evaluated; it remains permanently in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`.
6. The transaction never proceeds to execution.

### Impact Explanation

Every transaction that requires approvals and receives only approvals (no rejections) is permanently frozen. The chain service is never signaled to advance the transaction to execution. The only way `emitTransactionStatusUpdate` fires is if at least one approver **rejects** the transaction. This means:

- Approved transactions are permanently stuck — a complete denial of the approval-to-execution workflow.
- Funds or operations gated behind the approval flow can never be executed through normal means.
- The state corruption is silent: the DB correctly records `approved: true`, but the downstream state machine is never triggered.

**Impact: 5** — permanent, unrecoverable lock of the transaction execution flow for all approval-gated transactions.

### Likelihood Explanation

Every organization user who is a designated approver and submits an approval triggers this path. No special attacker capability is required — any legitimate approver exercising the normal approval workflow hits this bug. The endpoint is authenticated but reachable by any user who has been added as an approver.

**Likelihood: 5** — triggered by normal, expected user behavior on every approval action.

### Recommendation

The condition must be corrected so that `emitTransactionStatusUpdate` is emitted whenever the approval state changes (both approval and rejection), and `emitTransactionUpdate` is used only when no status-relevant change occurred. The stale `userApprovers` array must not be used to gate the status update:

```typescript
// After the DB update, always emit a status update so the chain
// service can re-evaluate whether the transaction can proceed.
emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
```

If the intent was to emit `emitTransactionUpdate` for partial approvals (not yet meeting threshold) and `emitTransactionStatusUpdate` only when the threshold is met or on rejection, the check must be performed against **fresh DB data** fetched after the write, not the stale pre-update `userApprovers` array.

### Proof of Concept

1. Create an organization transaction requiring a single approver (User A).
2. User A calls `POST /transactions/{id}/approvers/approve` with body `{ userKeyId: X, signature: "...", approved: true }`.
3. The DB record for the `TransactionApprover` is updated: `approved = true`, `signature = <valid>`.
4. Observe NATS traffic: only `TRANSACTION_UPDATE` is published; `TRANSACTION_STATUS_UPDATE` is **not** published.
5. The chain service receives no event; the transaction status remains `WAITING_FOR_EXECUTION` indefinitely.
6. Contrast: if User A calls with `approved: false`, `TRANSACTION_STATUS_UPDATE` **is** published and the chain service correctly marks the transaction as rejected.

The test at `approvers.service.spec.ts` line 1271–1278 inadvertently confirms this: it calls `approveTransaction` with both `approved: true` and `approved: false` in sequence, and only asserts that `emitTransactionStatusUpdate` was called — but the mock for `approved: false` is what actually satisfies that assertion, masking the bug for the `approved: true` case.
