### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated Resource Exhaustion DoS

### Summary
`getTransactionsToSign` in `TransactionsService` fetches **every non-terminal transaction in the entire database** without any SQL-level `LIMIT`, then performs a per-transaction async cryptographic key-computation call (`userKeysToSign`) for each row before applying pagination in application memory. An authenticated attacker who creates a large number of transactions causes every subsequent call to `GET /transactions/sign` to perform O(N) database queries and O(N) cryptographic operations, progressively degrading and eventually making the API server unresponsive.

### Finding Description

**Root cause — unbounded `repo.find()` with no `take`:**

In `getTransactionsToSign`, the database query at line 295 has no `take` (SQL `LIMIT`) clause:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters by status, not by user
  relations: ['groupItem'],
  order,
  // ← no `take` / no `limit`
});
``` [1](#0-0) 

The `whereForUser` filter excludes only `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `ARCHIVED` statuses. Every transaction in `NEW`, `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, and `REJECTED` states — across **all users** — is loaded into memory.

**Per-row async work:**

For every loaded transaction, `userKeysToSign` is called, which internally calls `computeSignatureKey` (a DB-backed cryptographic key resolution):

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [2](#0-1) 

**Pagination applied after full scan:**

The `limit`/`offset` from the request is applied only after the full in-memory result is built:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [3](#0-2) 

So even a request for `page=1&size=1` causes the server to load and process every non-terminal transaction in the database.

**The endpoint remains registered and reachable:**

The controller comment says "NO LONGER USED BY FRONT-END" but the route is still active and protected only by JWT + verified-user guards — no additional throttle decorator is applied to this specific endpoint: [4](#0-3) 

**Global IP throttler does not prevent this:**

The `IpThrottlerGuard` limits the number of requests per IP per time window, but a single request with N=100,000 transactions in the DB still causes the server to do O(N) work per call. [5](#0-4) 

**Transaction creation has no per-user cap:**

Any authenticated user with at least one registered key can create transactions. Each transaction must have a unique Hedera transaction ID (account + timestamp), which is trivially varied. There is no limit on how many transactions a single user may create. [6](#0-5) 

### Impact Explanation

- **CPU exhaustion**: Each call to `GET /transactions/sign` triggers O(N) calls to `userKeysToSign` → `computeSignatureKey`, which involves deserialization of Hedera SDK transaction bytes and multiple DB lookups per transaction.
- **Memory exhaustion**: All N transaction rows (including their `transactionBytes` blobs) are loaded into the Node.js heap simultaneously.
- **Cascading degradation**: As N grows, response latency increases linearly. At sufficient scale, the NestJS event loop stalls, causing all API endpoints to time out — not just `/transactions/sign`.
- **Persistence**: Transactions in `WAITING_FOR_SIGNATURES` remain in the database until they expire (Hedera transaction validity window is up to 3 minutes by default, but the record persists in the DB after expiry in `EXPIRED` status — however `EXPIRED` is excluded). An attacker can continuously create new transactions to maintain a large active set.

### Likelihood Explanation

- **Attacker preconditions**: A registered, verified account with one uploaded key. This is the normal onboarding flow — no privileged access required.
- **Attack cost**: Creating each transaction requires constructing a valid Hedera SDK transaction and signing it with the registered key. This is fully scriptable using the open-source Hedera SDK. The attacker's only cost is the time to register and upload a key.
- **No rate limit on transaction creation per user**: The `POST /transactions` endpoint is subject only to the global IP throttler (configurable `GLOBAL_MINUTE_LIMIT`), which is designed for general abuse prevention, not for preventing bulk transaction creation by a single authenticated user.
- **Amplification**: One attacker creating N transactions degrades the experience for all users of the platform, since the scan is not scoped to the requesting user's transactions.

### Recommendation

Apply the SQL-level `take` limit **before** the application-level loop. The correct fix is to push the key-eligibility check into the database query (as a join or subquery), mirroring the approach already used in `getTransactions` and `getTransactionsToApprove`:

```typescript
// Instead of fetching all and filtering in-memory:
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // ← add SQL-level limit
  skip: offset,  // ← add SQL-level offset
});
```

For accurate `totalItems`, a separate `count()` query with the same `where` clause should be issued. The per-transaction `userKeysToSign` check should be moved into a SQL predicate (joining on `transaction_signer` or `public_keys`) so that only relevant rows are fetched, consistent with how `getTransactionsToApprove` uses a recursive CTE. [7](#0-6) 

### Proof of Concept

1. Register a user account and upload one ECDSA key pair via `POST /user-keys`.
2. Using the Hedera SDK, generate 10,000 `AccountCreateTransaction` objects, each with a unique `TransactionId` (vary the `validStart` timestamp by nanoseconds). Sign each with the registered private key.
3. Submit each via `POST /transactions` (scriptable; the global throttler allows bursts).
4. Once the transactions are stored (status `WAITING_FOR_SIGNATURES`), call `GET /transactions/sign?page=1&size=1` from any authenticated session.
5. **Observed**: The server loads all 10,000 transaction rows, calls `userKeysToSign` 10,000 times (each involving DB lookups and SDK deserialization), and the response time grows linearly with N. At sufficient N, the Node.js event loop stalls and all API endpoints become unresponsive.
