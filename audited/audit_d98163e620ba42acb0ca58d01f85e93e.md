### Title
IDOR in `removeTransactionApprover`: Any Authenticated User Can Delete Approvers from Any Transaction

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the caller is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without checking that this approver actually belongs to `:transactionId`. An attacker who owns any one transaction can delete approvers from any other user's transaction by supplying their own `transactionId` (to pass the creator check) and a victim's approver `id` (to perform the deletion).

### Finding Description

The controller handler in `approvers.controller.ts` performs two independent operations:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks user owns transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only validates that the authenticated user is the creator of the transaction identified by `:transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by its own `id` and deletes it — with no check that `approver.transactionId` matches the `:transactionId` URL parameter: [3](#0-2) 

`getTransactionApproverById` performs a plain lookup by `id` only: [4](#0-3) 

**Contrast with `updateTransactionApprover`**, which correctly validates the cross-ownership before acting:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, ...);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
await this.getCreatorsTransaction(rootNode.transactionId, user, ...);
``` [5](#0-4) 

The `removeTransactionApprover` path is missing this exact cross-check, making it an IDOR.

### Impact Explanation

An attacker can silently delete approvers from any transaction in the system. The approval workflow is a core security control: transactions require a threshold of approvers to sign before execution. Removing approvers from a victim's transaction can:

1. **Bypass the approval requirement** — if all approvers are deleted, the transaction may proceed to `WAITING_FOR_EXECUTION` without the intended governance checks.
2. **Corrupt the approval tree** — `removeNode` performs a recursive soft-delete of the entire approver subtree, so a single call can wipe an entire multi-level approval structure.
3. **Cause unauthorized state changes** — `emitTransactionStatusUpdate` is called after deletion, potentially triggering downstream state transitions in the Chain Service. [6](#0-5) 

### Likelihood Explanation

- **Attacker preconditions**: Only a valid authenticated account (registered user with a verified JWT). No admin or privileged role required.
- **Discovery of victim approver IDs**: Approver IDs are sequential integers. An attacker can enumerate them by creating their own transactions and observing the IDs returned, or by calling `GET /transactions/:id/approvers` on any transaction they have access to.
- **Attack execution**: Trivial — one HTTP `DELETE` request with the attacker's own `transactionId` and the victim's approver `id`.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), resolve the root transaction of the approver being deleted and assert it matches the `:transactionId` URL parameter, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode) throw new BadRequestException(ErrorCodes.RANF);
  if (rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

### Proof of Concept

**Setup:**
- Attacker registers and logs in → obtains JWT `attacker_token`.
- Victim creates Transaction B with approver IDs `[42, 43]` (discovered via enumeration or a shared/public endpoint).
- Attacker creates their own Transaction A → receives `transactionId = 99`.

**Exploit request:**
```http
DELETE /transactions/99/approvers/42
Authorization: Bearer attacker_token
```

**Step-by-step:**
1. `getCreatorsTransaction(99, attacker)` → passes, attacker owns transaction 99.
2. `removeTransactionApprover(42)` → fetches approver 42 (belongs to victim's transaction B), calls `removeNode(42)`, recursively soft-deletes approver 42 and all its children.
3. Response: `200 true`.

**Expected (correct) behavior:** `403 Unauthorized` — approver 42 does not belong to transaction 99.

**Actual behavior:** Approver 42 (and its subtree) is permanently deleted from the victim's transaction, bypassing the approval workflow.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
