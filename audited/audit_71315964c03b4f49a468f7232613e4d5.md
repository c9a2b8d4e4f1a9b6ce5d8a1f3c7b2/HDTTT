### Title
TOCTOU Race Condition in Signature Upload Allows Duplicate `TransactionSigner` Records, Corrupting Transaction State

---

### Summary

The `uploadSignatureMaps` flow in `SignersService` checks for existing signers using a pre-loaded snapshot, then inserts new signers with no conflict handling and no unique database constraint on `(transactionId, userKeyId)`. Two concurrent signature-upload requests for the same transaction by the same user can both pass the duplicate check and both insert the same signer record, producing duplicate rows that corrupt transaction status tracking.

---

### Finding Description

The vulnerability spans three cooperating weaknesses:

**1. Non-unique index on `TransactionSigner`**

`transaction_signer` carries only a plain (non-unique) composite index on `(transactionId, userKeyId)`:

```typescript
@Entity()
@Index(['transactionId', 'userKeyId'])   // ← NOT unique
export class TransactionSigner { … }
``` [1](#0-0) 

There is no `{ unique: true }` option and no corresponding `UNIQUE` constraint in any migration. The database will silently accept multiple rows with the same `(transactionId, userKeyId)` pair.

**2. Duplicate check is a pre-loaded snapshot (TOCTOU)**

`loadTransactionData` reads all existing signers once, before any processing begins:

```typescript
const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
  where: { transactionId: In(transactionIds) },
  select: ['transactionId', 'userKeyId'],
});
``` [2](#0-1) 

That snapshot is passed into `processTransactionSignatures`, which uses it to decide whether a key is "new":

```typescript
if (!existingSignerIds.has(userKey.id)) {
  userKeys.push(userKey);
}
``` [3](#0-2) 

Two concurrent requests both read the same snapshot (before either has committed), both see the key as absent, and both mark it for insertion.

**3. `bulkInsertSigners` has no conflict handling**

```typescript
private async bulkInsertSigners(manager: any, signersToInsert: any[]) {
  const result = await manager
    .createQueryBuilder()
    .insert()
    .into(TransactionSigner)
    .values(signersToInsert)
    .returning('*')
    .execute();          // ← no .orIgnore(), no ON CONFLICT clause
  return result.raw;
}
``` [4](#0-3) 

Compare with the correctly-written `insertKeys` helper used for cached account/node keys, which explicitly uses `.orIgnore()` for idempotency:

```typescript
await this.dataSource
  .createQueryBuilder()
  .insert()
  .into(keyEntity)
  .values(…)
  .orIgnore()           // ← idempotent
  .execute();
``` [5](#0-4) 

The inconsistency is clear: the codebase already knows how to write idempotent inserts but did not apply the pattern to `bulkInsertSigners`.

---

### Impact Explanation

Duplicate `TransactionSigner` rows for the same `(transactionId, userKeyId)` corrupt the state that `processTransactionStatus` and `TransactionSignatureService` rely on to determine whether a transaction has collected enough signatures to advance. The same key's signature is counted more than once, which can cause:

- A transaction to be incorrectly promoted to `WAITING_FOR_EXECUTION` before all required keys have actually signed.
- Downstream execution of a transaction that is not fully authorized.
- Persistent database state corruption that cannot be self-healed without manual intervention.

Additionally, if a `UNIQUE` constraint is later added to fix the data integrity issue (the correct remediation), the absence of `.orIgnore()` in `bulkInsertSigners` means concurrent requests would throw a constraint-violation exception, caught by the outer `catch` block and re-thrown as `ErrorCodes.FST`, causing the entire signature-upload batch to fail — a direct analog to the epoch-end abort described in the external report. [6](#0-5) 

---

### Likelihood Explanation

The `POST /transactions/:transactionId/signers` endpoint is unauthenticated beyond a valid JWT, meaning any verified user who is a signer on a transaction can trigger this. The race window is the time between `loadTransactionData` and the commit of `bulkInsertSigners`. In a multi-instance deployment (the repo ships Kubernetes manifests for multiple API replicas), two requests handled by different pods will never share an in-process lock, making the race reliably reproducible by simply submitting the same signature map twice in rapid succession — something that can happen accidentally (double-click, retry logic) or deliberately. [7](#0-6) 

---

### Recommendation

Apply both fixes together:

1. **Add a unique constraint** on `transaction_signer.(transactionId, userKeyId)` via a new TypeORM migration:
   ```sql
   CREATE UNIQUE INDEX "UQ_transaction_signer_tx_key"
     ON "transaction_signer" ("transactionId", "userKeyId");
   ```

2. **Add `.orIgnore()` to `bulkInsertSigners`** so that concurrent or duplicate inserts are silently skipped rather than crashing the batch:
   ```typescript
   .insert()
   .into(TransactionSigner)
   .values(signersToInsert)
   .orIgnore()   // ← add this
   .returning('*')
   .execute();
   ```

This mirrors the pattern already used in `insertKeys` in `cache.helper.ts` and makes the operation idempotent.

---

### Proof of Concept

1. Obtain a valid JWT for user U who is a signer on transaction T (key K).
2. Sign transaction T with key K to produce a `signatureMap`.
3. Send two concurrent `POST /transactions/T/signers` requests with the same `signatureMap`, timed so both reach `loadTransactionData` before either commits.
4. Both requests pass the `!existingSignerIds.has(userKey.id)` check (line 254) because neither has committed yet.
5. Both call `bulkInsertSigners` and both succeed (no unique constraint).
6. Query the database:
   ```sql
   SELECT * FROM transaction_signer
   WHERE "transactionId" = T AND "userKeyId" = K;
   -- Returns 2 rows with identical (transactionId, userKeyId)
   ```
7. Observe that `processTransactionStatus` now counts key K's signature twice, potentially advancing the transaction status incorrectly.
