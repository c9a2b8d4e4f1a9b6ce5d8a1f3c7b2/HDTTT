### Title
Unbounded Full-Table Scan in `getTransactionsToSign()` Causes Server Resource Exhaustion via Single Authenticated Request

### Summary
`TransactionsService.getTransactionsToSign()` fetches **all** active (non-terminal) transactions from the database in a single query with no row limit, then iterates over every result calling the async `userKeysToSign()` per transaction. Pagination is applied only after the full in-memory loop completes. Any authenticated, verified user can trigger `GET /transactions/sign` to force the server to load and process an unbounded number of transactions, exhausting memory, CPU, and database connections — directly analogous to the OOG loop in the external report.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign()` issues an unconstrained `find()` with no `take` parameter:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

It then loops over every returned row, performing an async `userKeysToSign()` call per transaction:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

Pagination (`offset`/`limit`) is applied only after the loop finishes:

```typescript
items: result.slice(offset, offset + limit),
``` [3](#0-2) 

The controller exposes this as `GET /transactions/sign`, guarded only by standard JWT + verified-user guards — no admin role required: [4](#0-3) 

The endpoint is still live even though the comment marks it `/* NO LONGER USED BY FRONT-END */`: [5](#0-4) 

### Impact Explanation

With N active (non-terminal) transactions in the system, a single `GET /transactions/sign` request causes the API process to:
1. Load all N transaction rows (plus `groupItem` relations) into heap memory.
2. Serially await N async `userKeysToSign()` calls, each of which performs cryptographic key-structure traversal.

At scale this exhausts heap memory and saturates the Node.js event loop, causing request timeouts and degraded or unavailable service for all users. Because the loop is sequential and awaited, a single request holds the event loop busy proportional to N, blocking other requests from being processed in a timely manner.

**Impact: High** — complete API service degradation achievable by a single authenticated user.

### Likelihood Explanation

**Likelihood: Low-Medium.** The attacker must hold a valid, verified organization account (no admin privilege needed). The endpoint is still reachable via the REST API despite the front-end no longer calling it. The attack requires only one HTTP request; no flooding or DDoS tooling is needed. The condition worsens naturally as the organization accumulates active transactions over time.

### Recommendation

Apply the database-level `take` limit **before** the loop, mirroring the pattern used in `getTransactions()` and `getHistoryTransactions()`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // push pagination into the DB query
  skip: offset,
});
```

If per-transaction key filtering must remain in application code (because it cannot be expressed as a SQL predicate), implement cursor-based or keyset pagination so the server never processes more than one page of rows per request. Alternatively, if the endpoint is truly unused, remove it entirely from the controller.

### Proof of Concept

**Preconditions:**
- Attacker holds a valid verified-user JWT (any normal organization member).
- The organization has accumulated a large number of active (non-`EXECUTED`/`FAILED`/`EXPIRED`/`CANCELED`/`ARCHIVED`) transactions.

**Steps:**
1. Authenticate as a verified user and obtain a JWT.
2. Issue: `GET /transactions/sign?page=1&size=10` (any small page size).
3. The server ignores the `size` parameter for the DB query and fetches **all** active transactions, then loops through every one calling `userKeysToSign()`.
4. With tens of thousands of active transactions, the Node.js process heap grows proportionally and the event loop stalls for the duration of the loop, causing timeouts for concurrent requests.

**Expected outcome:** Server memory and CPU spike; concurrent API requests time out; service is degraded for all users until the request completes or the process OOMs.
