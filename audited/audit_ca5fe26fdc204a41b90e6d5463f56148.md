### Title
Unbounded String Fields in Backend DTOs Allow Authenticated Users to Exhaust Database Storage and Cause Memory Pressure

### Summary
Multiple backend API DTOs accept string fields with no `@MaxLength()` constraint, meaning any authenticated organization member can POST arbitrarily large strings that are persisted directly to PostgreSQL. The most critical instance is `CreateCommentDto.message`, which has no length limit at the DTO level or the database entity level. Unlike the front-end, which enforces length limits client-side for transaction `name` and `description`, the backend enforces nothing, and the comment endpoint has zero client-side validation either. A single authenticated user can exhaust database disk space or cause server-side memory pressure when the stored data is later loaded.

### Finding Description

**Root cause — missing `@MaxLength()` in DTOs and no column length in entities:**

`CreateCommentDto` accepts an unbounded `message` string: [1](#0-0) 

The TypeORM entity maps this to a plain `@Column()` with no `length` argument, which defaults to PostgreSQL `text` — no practical upper bound: [2](#0-1) 

The controller passes the DTO directly to the service with no intermediate sanitization: [3](#0-2) 

The service calls `repo.create(dto)` and `repo.save(comment)` immediately: [4](#0-3) 

The same pattern exists in `CreateTransactionDto` for `name` and `description`: [5](#0-4) 

And in `CreateTransactionGroupDto` for `description`: [6](#0-5) 

**Why front-end guards do not help:**

The front-end `ValidateRequestHandler.vue` checks `name.length > 50` and `description.length > 256`, but these are client-side only: [7](#0-6) 

There is no equivalent check anywhere in the front-end for comment `message`. An attacker bypasses the front-end entirely by calling the REST API directly.

**Exploit path:**

1. Attacker registers as a normal organization user (no admin privileges required).
2. Attacker obtains a valid JWT via `POST /auth/login`.
3. Attacker sends `POST /transactions/:transactionId/comments` with a `message` field containing a multi-megabyte string (up to the configured Express body-parser limit, which defaults to 100 KB but is often raised for applications that handle binary `transactionBytes` payloads).
4. The backend validates only that `message` is a string (`@IsString()`), passes validation, and persists the full payload to PostgreSQL.
5. Repeating this across many transactions fills the database disk. When any user or service later calls `getTransactionComments`, the full text is loaded into Node.js heap memory.

### Impact Explanation

- **Database disk exhaustion**: Each comment row can hold up to the body-parser limit per request. Repeated calls fill the `transaction_comment` table with no per-user or per-transaction quota.
- **Server-side memory pressure**: `getTransactionComments` loads all comment rows for a transaction into memory with no pagination or size guard, so a transaction with many large comments can cause OOM in the API process.
- **Degraded availability for all users**: Once the database disk is full or the API process OOMs, the entire organization's transaction workflow is disrupted — approvals, signing, and execution all depend on the same API service.

### Likelihood Explanation

Any verified organization member can reach the comment endpoint — no admin role is required. The guard chain is `JwtBlackListAuthGuard → JwtAuthGuard → VerifiedUserGuard`, meaning any user who has completed email verification can exploit this. The attack requires only a standard HTTP client (curl, Postman) to bypass the front-end validation. No cryptographic material, no privileged access, and no special knowledge beyond a valid JWT is needed. [8](#0-7) 

### Recommendation

1. Add `@MaxLength(N)` to every unbounded string field in backend DTOs. A reasonable cap for `message` is 2000–4000 characters; for transaction `name` match the front-end limit of 50; for `description` match 256.
2. Add a `length` argument to the TypeORM `@Column()` on `TransactionComment.message` (e.g., `@Column({ length: 4000 })`) so the database itself enforces the constraint as a second layer.
3. Confirm the Express body-parser `limit` is set to the minimum value needed for legitimate `transactionBytes` payloads and document it explicitly.
4. Consider adding per-user rate limiting on the comment creation endpoint.

### Proof of Concept

```bash
# 1. Obtain a JWT for a normal verified user
TOKEN=$(curl -s -X POST http://localhost:3000/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"user@org.com","password":"Password1"}' | jq -r .accessToken)

# 2. Build a payload with a 90 KB message (within default 100 KB body limit)
LARGE_MSG=$(python3 -c "print('A' * 90000)")

# 3. POST the oversized comment — passes @IsString() validation, stored in DB
curl -X POST http://localhost:3000/transactions/1/comments \
  -H "Authorization: Bearer $TOKEN" \
  -H 'Content-Type: application/json' \
  -d "{\"message\": \"$LARGE_MSG\"}"

# 4. Repeat in a loop across multiple transactions to exhaust disk
# 5. GET /transactions/1/comments now loads the full 90 KB into heap per row
```

The `POST` returns HTTP 201 and the full payload is written to `transaction_comment.message` with no truncation or rejection, confirming the missing length enforcement.
