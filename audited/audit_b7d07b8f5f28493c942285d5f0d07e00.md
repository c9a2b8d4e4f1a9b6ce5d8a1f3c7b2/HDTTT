### Title
Concurrent `uploadSignatureMaps` Calls Silently Overwrite Each Other's Signatures via Unlocked Read-Modify-Write on `transactionBytes`

### Summary
`SignersService.uploadSignatureMaps` reads `transactionBytes` from the database, deserializes the SDK transaction, appends new signatures, and writes the result back — all without any row-level lock. When two users sign the same transaction concurrently, both reads return the same original bytes, both compute their own updated byte blob, and the second write silently overwrites the first. The net result is that one user's signature is permanently lost from the stored transaction, mirroring the external report's "value overwritten instead of accumulated" pattern.

### Finding Description

**Root cause — unlocked read-modify-write in `uploadSignatureMaps`:**

`loadTransactionData` fetches `transactionBytes` with a plain `find` (no `SELECT FOR UPDATE`): [1](#0-0) 

`processTransactionSignatures` deserializes those bytes and appends the caller's signatures: [2](#0-1) 

`persistSignatureChanges` then writes the result back with a raw `UPDATE … SET "transactionBytes" = CASE id … END` — a direct overwrite, not an accumulation: [3](#0-2) 

The database transaction that wraps the write (line 317) does **not** cover the read at line 131, so the read-modify-write is not atomic.

**Exploit flow:**

1. Transaction T requires signatures from User A and User B.
2. User A calls `POST /transactions/T/signers` — reads `transactionBytes` (0 sigs).
3. User B calls `POST /transactions/T/signers` concurrently — reads the same `transactionBytes` (0 sigs).
4. User A's request finishes first: writes `transactionBytes` containing only A's signature.
5. User B's request finishes second: writes `transactionBytes` containing only B's signature — **A's signature is gone**.
6. The stored transaction now has 1 of 2 required signatures; it can never reach the execution threshold.

The same race exists in `importSignatures` in `transactions.service.ts`, which follows the identical read-modify-write pattern without locking: [4](#0-3) 

### Impact Explanation

Signatures are silently discarded. A multi-signature transaction that has collected the required number of signers in the `TransactionSigner` table may never have those signatures present in `transactionBytes`, so the Hedera network rejects it at execution time. The transaction is permanently stuck: it cannot be re-signed (the valid-start window expires) and cannot be executed. This constitutes an unrecoverable loss of a scheduled, coordinated transaction — matching the "permanent lock/freeze of user/project state" impact category in RESEARCHER.md.

### Likelihood Explanation

Concurrent signing is the **normal workflow** in a multi-user organization: the UI notifies all required signers simultaneously, and they are expected to sign around the same time. No attacker capability beyond a valid authenticated session is required. A malicious co-signer can deliberately race their own signing request against another user's to suppress that user's signature, preventing the transaction from ever executing.

### Recommendation

Acquire a row-level exclusive lock on the `transaction` row before reading `transactionBytes`. In PostgreSQL/TypeORM this is `SELECT … FOR UPDATE`:

```typescript
const transaction = await manager.findOne(Transaction, {
  where: { id },
  lock: { mode: 'pessimistic_write' },
});
```

Move the read inside the same `dataSource.transaction` block that performs the write, so the lock is held for the entire read-modify-write cycle. Alternatively, use optimistic locking (a `version` column) and retry on conflict.

### Proof of Concept

```
# Setup: transaction T requires signatures from user_a and user_b

# Fire both requests simultaneously (e.g., with curl --parallel or two terminal tabs)
curl -X POST https://api/transactions/T/signers \
  -H "Authorization: Bearer <user_a_token>" \
  -d '{"signatureMap": <user_a_sig_map>}' &

curl -X POST https://api/transactions/T/signers \
  -H "Authorization: Bearer <user_b_token>" \
  -d '{"signatureMap": <user_b_sig_map>}' &

wait

# Expected: transactionBytes contains both A's and B's signatures
# Actual:   transactionBytes contains only the last writer's signature
#           TransactionSigner table shows 2 rows (both recorded as signers)
#           but the stored bytes only carry 1 signature
#           → Hedera rejects the transaction at execution time
```

The discrepancy between the `TransactionSigner` table (which correctly records both signers via the insert path) and the actual `transactionBytes` blob (which only retains the last writer's signatures) is the observable proof of the overwrite. [5](#0-4) [6](#0-5)

  private async persistSignatureChanges(
    validationResults: any[],
    user: User,
  ) {
    const signers = new Set<TransactionSigner>();
    let notificationsToDismiss: number[] = [];

    // Prepare batched operations
    const transactionsToUpdate: { id: number; transactionBytes: Buffer }[] = [];
    const notificationsToUpdate: { userId: number; transactionId: number }[] = [];
    const signersToInsert: { userId: number; transactionId: number; userKeyId: number }[] = [];
    const transactionsToProcess: { id: number; transaction: Transaction }[] = [];

    for (const result of validationResults) {
      if (result.error) {
        console.error(`[TX ${result.id}] Validation failed: ${result.error}`);
        continue;
      }

      const { id, transaction, sdkTransaction, userKeys, isSameBytes } = result;

      // Skip if nothing to do - no signatures were added to the transaction
      // AND no new signers were inserted (the signature can be present on the transaction
      // if collated by an outside or 'offline' method)
      if (isSameBytes && userKeys.length === 0) continue;

      // Collect updates
      if (!isSameBytes) {
        transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
        transactionsToUpdate.push({ id, transactionBytes: transaction.transactionBytes });
      }

      // Collect inserts
      if (userKeys.length > 0) {
        const newSigners = userKeys.map(userKey => ({
          userId: user.id,
          transactionId: id,
          userKeyId: userKey.id,
        }));
        signersToInsert.push(...newSigners);
      }

      transactionsToProcess.push({ id, transaction });
      notificationsToUpdate.push({ userId: user.id, transactionId: transaction.id });
    }

    // Execute in single transaction
    try {
      await this.dataSource.transaction(async manager => {
        // Bulk update transactions
        if (transactionsToUpdate.length > 0) {
          await this.bulkUpdateTransactions(manager, transactionsToUpdate);
        }

        // Bulk update notifications
        if (notificationsToUpdate.length > 0) {
          const updatedNotificationReceivers = await this.bulkUpdateNotificationReceivers(manager, notificationsToUpdate);

          // To maintain backwards compatibility and multi-machine support, we send off a dismiss event.
          emitDismissedNotifications(
            this.notificationsPublisher,
            updatedNotificationReceivers,
          );

          notificationsToDismiss = updatedNotificationReceivers.map(nr => nr.id);
        }

        // Bulk insert signers
        if (signersToInsert.length > 0) {
          const results = await this.bulkInsertSigners(manager, signersToInsert);
          results.forEach(signer => signers.add(signer));
        }
      });
    } catch (err) {
      console.error('Database transaction failed:', err);
      throw new BadRequestException(ErrorCodes.FST);
    }

    return {
      transactionsToProcess,
      signers,
      notificationsToDismiss,
    };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
