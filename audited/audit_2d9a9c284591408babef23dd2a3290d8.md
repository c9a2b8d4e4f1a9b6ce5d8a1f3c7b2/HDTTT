### Title
Duplicate `userId` Across Threshold Tree Branches Allows Single-User Bypass of Multi-Party Approval

### Summary

The `createTransactionApprovers` function in `ApproversService` does not enforce uniqueness of `userId` values across different branches of a nested threshold approver tree. A malicious transaction creator can craft a threshold structure where the same user appears as an approver in multiple sub-branches. Because `approveTransaction` updates **all** records matching `userId = user.id` in a single operation, one approval action satisfies multiple branches simultaneously, bypassing the intended multi-party governance requirement.

### Finding Description

**Root Cause — `isNode` only checks `(listId, userId)` uniqueness, not global `userId` uniqueness:** [1](#0-0) 

The `isNode` duplicate check queries:
```
WHERE listId = <parentId> AND userId = X AND transactionId IS NULL
```

This only prevents the same user from appearing twice under the **same parent node**. It does not prevent the same `userId` from appearing under **different parent nodes** within the same tree.

**Vulnerable creation path:** [2](#0-1) 

When the recursive `createApprover` processes nested children, each child's `isNode` call uses the newly-assigned `listId` (the parent's DB id). Two children under different threshold nodes with the same `userId` will each pass the `isNode` check because their `listId` values differ.

**Exploit path — `approveTransaction` updates ALL records for the user:** [3](#0-2) 

`userApprovers` collects every `TransactionApprover` row where `userId === user.id`. The bulk update at line 600–609 marks **all** of them as approved in one call. If `userId: 1` appears in two branches, a single approval from user 1 marks both rows as approved.

**Concrete malicious structure:**

```json
POST /transactions/42/approvers
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [
      { "threshold": 1, "approvers": [{ "userId": 1 }] },
      { "threshold": 1, "approvers": [{ "userId": 1 }] }
    ]
  }]
}
```

This creates a `2-of-2` structure that visually appears to require two independent approvals, but user 1's single approval satisfies both sub-branches simultaneously.

**`TransactionApprover` entity has no unique constraint on `(transactionId, userId)`:** [4](#0-3) 

Only `transactionId` and `userId` are individually indexed, not as a unique composite constraint, so the database does not reject the duplicate insert.

### Impact Explanation

A transaction creator (normal authenticated user) can construct a threshold approver tree that **appears** to require N distinct approvers but actually requires only 1. This undermines the organization's multi-party approval governance model: a transaction can be marked as fully approved by a single user while the approval UI and audit trail show a multi-party threshold structure. Any downstream logic that gates transaction execution on approval status is bypassed.

### Likelihood Explanation

The attacker must be the transaction creator — a normal, authenticated organization user with no elevated privileges. The API endpoint `POST /transactions/:transactionId/approvers` is reachable by any creator. The payload is a standard JSON body accepted by `CreateTransactionApproversArrayDto` with no additional validation. No special tooling is required beyond a crafted HTTP request. [5](#0-4) 

### Recommendation

Add a global uniqueness check for `userId` across the entire submitted approver tree before any DB inserts. In `createTransactionApprovers`, before the recursive `createApprover` loop, collect all leaf `userId` values from `dto.approversArray` recursively and reject the request if any `userId` appears more than once:

```typescript
function collectUserIds(approvers: CreateTransactionApproverDto[]): number[] {
  return approvers.flatMap(a =>
    typeof a.userId === 'number'
      ? [a.userId]
      : a.approvers ? collectUserIds(a.approvers) : []
  );
}

const allUserIds = collectUserIds(dto.approversArray);
const uniqueUserIds = new Set(allUserIds);
if (allUserIds.length !== uniqueUserIds.size) {
  throw new BadRequestException('Duplicate userId values are not allowed in the approver tree');
}
```

Additionally, add a database-level unique constraint on `(transactionId, userId)` in `TransactionApprover` (accounting for soft-deletes) to enforce this invariant at the persistence layer.

### Proof of Concept

**Step 1** — Creator authenticates and creates a transaction (any type).

**Step 2** — Creator posts a malicious approver tree to `POST /transactions/:id/approvers`:
```json
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [
      { "threshold": 1, "approvers": [{ "userId": 5 }] },
      { "threshold": 1, "approvers": [{ "userId": 5 }] }
    ]
  }]
}
```
Both inserts succeed because `isNode` checks `listId=<node_A_id>, userId=5` and `listId=<node_B_id>, userId=5` separately — both return 0 rows.

**Step 3** — User 5 calls `POST /transactions/:id/approvers/approve` once with a valid signature.

**Step 4** — `approveTransaction` filters `approvers.filter(a => a.userId === 5)` → returns **2 rows**. Both are updated to `approved=true` in the bulk update.

**Step 5** — The `2-of-2` threshold is now satisfied. The transaction proceeds to execution with only one unique approver having acted, defeating the multi-party approval requirement.
