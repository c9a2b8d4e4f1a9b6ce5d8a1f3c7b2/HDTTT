### Title
Transaction Creator Can Remove Approvers From Any Other Transaction, Bypassing Approval Requirements

### Summary
The `removeTransactionApprover` controller verifies the caller is the creator of the transaction ID supplied in the URL, but then removes the approver by a separate `id` parameter without verifying that approver belongs to the same transaction. Any authenticated user who owns at least one transaction can exploit this to delete the entire approver tree of any other transaction, eliminating its approval requirements.

### Finding Description

**Root cause — missing ownership check in `removeTransactionApprover`:**

The controller at `back-end/apps/api/src/transactions/approvers/approvers.controller.ts` lines 102–113 performs two independent operations:

```typescript
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // removes approver by id — no cross-check
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the caller is the creator of the `transactionId` path parameter. It says nothing about whether the approver `id` belongs to that transaction. [2](#0-1) 

`removeTransactionApprover` in the service fetches the approver by `id` and calls `removeNode`, which recursively soft-deletes the node and all its children. There is no assertion that `approver.transactionId === transactionId`. [3](#0-2) 

`removeNode` performs a recursive soft-delete of the entire subtree rooted at the given `listId`: [4](#0-3) 

**Secondary issue — threshold not recalculated after direct removal:**

When `removeTransactionApprover` deletes a leaf approver (a `userId`-based node), the parent threshold node's `threshold` value is never decremented. The threshold-adjustment logic only exists in `updateTransactionApprover` (the `listId = null` detach path): [5](#0-4) 

`removeTransactionApprover` has no equivalent adjustment, so after removing a leaf the parent can retain a `threshold` value that exceeds the remaining child count, permanently locking the approval process for that transaction.

### Impact Explanation

**Primary (authorization bypass):** An attacker who is the creator of any transaction (even a trivial one they created themselves) can issue:

```
DELETE /transactions/{own_tx_id}/approvers/{root_approver_id_of_victim_tx}
```

This passes the `getCreatorsTransaction` check (attacker owns `own_tx_id`) and then removes the entire approver tree of the victim transaction. With no approvers remaining, the victim transaction's approval gate is eliminated. Depending on how `processTransactionStatus` handles zero-approver transactions, this either allows the transaction to advance to execution without any approvals, or permanently corrupts its state.

**Secondary (DoS):** Removing a leaf approver without adjusting the parent threshold leaves the threshold permanently unsatisfiable, locking the transaction in `WAITING_FOR_SIGNATURES` forever.

### Likelihood Explanation

Any authenticated organization user can create a transaction (no special privilege required). Approver IDs are sequential integers returned by the API to any user with read access to a transaction. An attacker needs only:
1. Create one transaction (to obtain a valid `transactionId` they own).
2. Observe or enumerate the approver IDs of a target transaction (readable via `GET /transactions/{id}/approvers` if they have any access to it).
3. Issue the `DELETE` request with their own `transactionId` and the victim's approver `id`.

This is a low-effort, no-privilege-escalation attack path reachable by any normal authenticated user.

### Recommendation

In `removeTransactionApprover` (service), after fetching the approver, assert that its root transaction matches the `transactionId` supplied by the caller before proceeding with deletion. Concretely:

1. Call `getRootNodeFromNode(approver.id)` to obtain the root.
2. Assert `rootNode.transactionId === transactionId`; throw `UnauthorizedException` if not.
3. After removing a leaf node, fetch the parent and decrement its `threshold` if it now exceeds the remaining child count (mirror the logic already present in `updateTransactionApprover`).

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (attacker-owned).
- User B creates Transaction 2 with a threshold approver tree: root node (threshold=2, children=[UserC, UserD]).
- The root approver of Transaction 2 has `id = 42`.

**Attack:**
```
DELETE /api/transactions/1/approvers/42
Authorization: Bearer <User A's token>
```

**What happens:**
1. `getCreatorsTransaction(1, UserA)` → passes (User A owns Transaction 1).
2. `removeTransactionApprover(42)` → fetches approver 42 (belongs to Transaction 2), calls `removeNode(42)`.
3. `removeNode` recursively soft-deletes approver 42 and its children (UserC, UserD).
4. Transaction 2 now has zero approvers. Its approval gate is gone. [6](#0-5) [1](#0-0)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
