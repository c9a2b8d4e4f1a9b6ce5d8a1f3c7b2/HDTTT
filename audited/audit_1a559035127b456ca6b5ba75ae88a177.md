### Title
Unpinned Docker Base Images Enable Silent Supply-Chain Substitution in CI and Production

### Summary
Multiple Dockerfiles and Docker Compose files in the repository reference Docker images with no tag at all (defaulting to `latest`) or with mutable version tags, rather than pinning to immutable SHA256 digests. This is the direct analog of the external report's finding about mutable Git tags on third-party GitHub Actions. If Docker Hub is compromised or an image maintainer silently pushes malicious content under the same tag, the CI pipeline and production deployments will pull and execute that malicious image without any indication of change.

### Finding Description

**Completely untagged images (default to `latest`):**

`back-end/redis/Dockerfile` line 1 uses `FROM redis` with no tag: [1](#0-0) 

`back-end/docker-compose.yaml` references three additional completely untagged images: [2](#0-1) [3](#0-2) [4](#0-3) 

These are `natsio/nats-box`, `postgres`, and `dpage/pgadmin4` — all resolving to `latest`.

**Mutable version-tagged images:**

`back-end/docker-compose.yaml` line 73 uses `nats:2.10`: [5](#0-4) 

`.github/workflows/test-migrations.yaml` line 32 uses `postgres:15` as a service container: [6](#0-5) 

All four application Dockerfiles (`api`, `chain`, `notifications`, `typeorm`) use `node:22.12.0-alpine` — a specific version string but not a SHA256-pinned digest: [7](#0-6) 

The `docker-compose.yaml` is the same file used by the CI pipeline's "Deploy Back-end" step: [8](#0-7) 

**Note on GitHub Actions:** The workflow files themselves largely follow best practice — most third-party actions are already pinned to full commit hashes (e.g., `step-security/harden-runner@8d3c67de8e2fe68ef647c8db1e6a09f647780f40`). The only unpinned action usages are `actions/upload-artifact@v7`, which is an official GitHub-organization action and is generally considered lower risk. [9](#0-8) 

### Impact Explanation
The `redis` service built from `FROM redis` (no tag) is a core runtime dependency of the `api`, `chain`, and `notifications` services in production: [10](#0-9) 

If Docker Hub's `redis:latest` (or `postgres:latest`, `natsio/nats-box:latest`, `dpage/pgadmin4:latest`) is silently replaced with a malicious image, the next `docker compose build` or `docker compose up` will pull and execute that image in both CI and production environments. This could result in credential theft (database passwords, API secrets from `.env` files), data exfiltration, or persistent backdoors in the running infrastructure.

### Likelihood Explanation
Docker Hub has a documented history of image compromise incidents. Images with no tag default to `latest`, which is updated by maintainers on every release — making silent substitution trivially easy for a compromised maintainer account. The `natsio/nats-box` and `dpage/pgadmin4` images are third-party community images with smaller maintainer teams, increasing the risk surface. The CI pipeline actively pulls and builds these images on every PR targeting `main` or `release/**` branches. [11](#0-10) 

### Recommendation
- **Short term**: Pin every Docker image reference to its SHA256 digest. For example:
  - `FROM redis` → `FROM redis:7.4.2@sha256:<digest>`
  - `image: postgres` → `image: postgres:16@sha256:<digest>`
  - `image: natsio/nats-box` → `image: natsio/nats-box:0.14.5@sha256:<digest>`
  - `image: dpage/pgadmin4` → `image: dpage/pgadmin4:8@sha256:<digest>`
- **Long term**: Add `docker` as a `package-ecosystem` in `.github/dependabot.yml` to automate digest updates. The existing Dependabot config covers `github-actions` and `npm` but not Docker: [12](#0-11) 

### Proof of Concept
1. An attacker compromises the Docker Hub account for `redis` (or any of the other untagged images).
2. They push a malicious image under the `latest` tag that exfiltrates environment variables on startup.
3. The next time the CI pipeline runs `docker compose build redis` (triggered by any PR to `main`), the malicious image is pulled silently.
4. The malicious container starts alongside `api`, `chain`, and `notifications`, which all connect to it — exposing database credentials, JWT secrets, and any other secrets present in the `.env` files mounted into those containers. [13](#0-12)
