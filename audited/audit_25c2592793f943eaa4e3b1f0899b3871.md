### Title
Unbounded Sequential Database Iteration in `getTransactionsToSign` Causes Denial of Service

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** matching transactions from the database without any SQL-level limit, then executes one additional async database query per transaction inside a `for` loop to compute signing keys. Pagination is applied only after the full in-memory iteration. As the number of transactions grows — driven by any authenticated user creating transactions — this endpoint degrades into N+1 query exhaustion, causing HTTP timeouts, database connection pool starvation, and server memory pressure that degrades service for all users.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method accepts `Pagination` parameters but does **not** pass `take`/`skip` to the ORM query:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` or `skip`
});
```

It then iterates every returned row and issues an async database call per transaction:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Pagination is applied only after the full loop completes:

```typescript
items: result.slice(offset, offset + limit),
```

`userKeysToSign` delegates to `userKeysRequiredToSign`, which performs key-lookup queries against the database for each transaction. With N active transactions visible to the requesting user, this endpoint issues N+1 database queries per HTTP request. [1](#0-0) 

### Impact Explanation

- **Memory exhaustion**: All matching transactions are loaded into the Node.js process heap simultaneously.
- **Database connection pool starvation**: N sequential `await` calls hold or re-acquire connections from the pool, blocking other requests.
- **HTTP timeout / 503**: The request handler runs until the Node.js HTTP timeout fires, returning an error to the caller and leaving the DB work in-flight.
- **Cascading degradation**: Because the loop is sequential and awaited, a single large request can monopolize the event loop long enough to delay all other in-flight requests on the same process.

Any authenticated user who is a creator, signer, observer, or approver of a large number of transactions can trigger this condition. The endpoint is a standard paginated GET route, so repeated calls amplify the effect.

### Likelihood Explanation

- **No privileged access required**: Any registered user can create transactions via the normal API flow.
- **Self-amplifying**: The more transactions a user creates (or is added to), the worse each call to this endpoint becomes — there is no server-side cap on how many transactions are fetched.
- **Reachable from the public API**: The endpoint is a standard REST GET route, callable by any authenticated session.
- **No rate-limit mitigation visible in the service layer** that would prevent repeated calls.

### Recommendation

Apply the SQL-level `take`/`skip` before the loop. Because `userKeysToSign` must be called to filter results, the correct fix is to move the key-check logic into a SQL subquery or a pre-filtered query so that only the page-sized slice is loaded and iterated:

1. Add `take: limit` and `skip: offset` to the `repo.find()` call, and accept that `totalItems` will require a separate `count()` query.
2. Alternatively, pre-filter at the SQL level using the existing key-matching logic (similar to the `whereBrackets` subquery already used in `getTransactions`) so that only transactions the user actually needs to sign are returned, then paginate those.
3. If the per-row async check cannot be eliminated, enforce a hard server-side cap (e.g., `MAX_TRANSACTIONS_PER_REQUEST = 200`) and return an error if the unfiltered count exceeds it.

### Proof of Concept

**Preconditions**: Attacker is a registered user with a valid JWT.

**Steps**:
1. Attacker calls `POST /transactions` repeatedly (or is added as observer/signer to many transactions by other users) until N transactions (e.g., 10,000) are visible to their account.
2. Attacker calls `GET /transactions/to-sign?page=1&size=10`.
3. The server executes `repo.find()` returning all 10,000 rows, then issues 10,000 sequential `await userKeysToSign(...)` calls.
4. The HTTP request times out; the database connection pool is saturated; other users' requests queue or fail.
5. Repeating step 2 continuously keeps the server in a degraded state.

**Expected outcome**: The endpoint becomes unresponsive for all users as transaction count grows, with no attacker capability beyond normal authenticated API usage. [2](#0-1)

```typescript
  async getTransactionsToSign(
    user: User,
    { page, limit, size, offset }: Pagination,
    sort?: Sorting[],
    filter?: Filtering[],
  ): Promise<
    PaginatedResourceDto<{
      transaction: Transaction;
      keysToSign: number[];
    }>
  > {
    const where = getWhere<Transaction>(filter);
    const order = getOrder(sort);

    const whereForUser: FindOptionsWhere<Transaction> = {
      ...where,
      status: Not(
        In([
          TransactionStatus.EXECUTED,
          TransactionStatus.FAILED,
          TransactionStatus.EXPIRED,
          TransactionStatus.CANCELED,
          TransactionStatus.ARCHIVED,
        ]),
      ),
    };

    const result: {
      transaction: Transaction;
      keysToSign: number[];
    }[] = [];

    /* Ensures the user keys are passed */
    await attachKeys(user, this.entityManager);
    if (user.keys.length === 0) {
      return {
        totalItems: 0,
        items: [],
        page,
        size,
      };
    }

    const transactions = await this.repo.find({
      where: whereForUser,
      relations: ['groupItem'],
      order,
    });

    for (const transaction of transactions) {
      /* Check if the user should sign the transaction */
      try {
        const keysToSign = await this.userKeysToSign(transaction, user);
        if (keysToSign.length > 0) result.push({ transaction, keysToSign });
      } catch (error) {
        console.log(error);
      }
    }

    return {
      totalItems: result.length,
      items: result.slice(offset, offset + limit),
      page,
      size,
    };
  }
```
