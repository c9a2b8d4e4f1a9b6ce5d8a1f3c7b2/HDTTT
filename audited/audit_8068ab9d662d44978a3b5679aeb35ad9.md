### Title
Silent Signature Upload Failure in Batch `uploadSignatureMaps` Masked by Flawed Frontend Detection, Causing Transactions to Expire Without User Notification

### Summary
When `uploadSignatureMaps` processes a batch of signature uploads, per-item validation failures (expired transaction, wrong status, not found) are silently logged to console and skipped. The HTTP response always returns HTTP 201 with only the successfully created `TransactionSigner` records and no per-item error information. The frontend `signItems` function then attempts to detect unsigned items using a unit-mismatched comparison (`items.length - signers.length`) that can evaluate to zero even when some transactions were silently rejected, completely hiding the failure from the user.

### Finding Description

**Root cause — backend silent skip:**

In `persistSignatureChanges`, any item whose validation produced an error is silently skipped with only a `console.error`:

```
back-end/apps/api/src/transactions/signers/signers.service.ts  lines 282-286
``` [1](#0-0) 

The outer `uploadSignatureMaps` then returns only the successfully created signers:

```
back-end/apps/api/src/transactions/signers/signers.service.ts  lines 120-123
``` [2](#0-1) 

The controller always responds HTTP 201 with no per-item error field: [3](#0-2) 

**Root cause — frontend flawed detection:**

`signItems` tries to detect unsigned items using:

```typescript
const unsignedCount = items.length - signers.length;
if (unsignedCount > 0) { ... }
``` [4](#0-3) 

`items` is `SignatureItem[]` — one element per transaction. `signers` is `TransactionSigner[]` — one element **per key per transaction**. These are different units. When a multi-key transaction succeeds and a single-key transaction fails, the counts can coincidentally match, making `unsignedCount = 0`, so the inner detection loop never runs and the failed transaction is completely hidden.

**Concrete failing scenario:**

| Submitted | Keys | Outcome |
|---|---|---|
| Transaction A | key1, key2 | Succeeds → 2 `TransactionSigner` records |
| Transaction B | key3 | Expired → silently dropped, 0 records |

- `items.length = 2`, `signers.length = 2` → `unsignedCount = 0`
- Inner loop never executes
- Transaction B's failure is invisible to the user

The validation that rejects expired transactions is correct: [5](#0-4) 

But the result is never surfaced to the caller.

### Impact Explanation
A user submits a batch of signatures and receives HTTP 201. They believe all signatures were recorded. In the scenario above, Transaction B never receives the required signature. If no other signer covers it, Transaction B expires without being executed. The transaction cannot be re-submitted with the same transaction ID. This is an unrecoverable state loss for the transaction — a permanent freeze of that transaction's lifecycle — caused by a silent failure the user had no way to detect.

### Likelihood Explanation
This is a realistic race condition: a user fetches a list of transactions to sign, one of them expires in the window between fetch and submission (Hedera transactions have a short validity window, typically 120–180 seconds). The user submits a batch that includes the now-expired transaction alongside a multi-key transaction. The flawed `unsignedCount` check masks the failure. No privileged access is required; any normal authenticated user can trigger this through ordinary signing workflows.

### Recommendation

**Backend**: Change `uploadSignatureMaps` to return a per-item result array (similar to `importSignatures` which already returns `SignatureImportResultDto[]` with per-item `error` fields): [6](#0-5) 

**Frontend**: Replace the unit-mismatched `unsignedCount` check with a direct set-membership check that does not depend on the count comparison:

```typescript
// Replace:
const unsignedCount = items.length - signers.length;
if (unsignedCount > 0) { ... }

// With:
for (const item of items) {
  if (!signedTransactionIds.has(item.transactionId)) {
    result.push(item);
  }
}
```

### Proof of Concept

1. Authenticate as a normal user with two keys (`key1`, `key2`) and a third key (`key3`).
2. Create Transaction A requiring `key1` and `key2`. Create Transaction B requiring `key3`.
3. Wait until Transaction B is within seconds of expiry.
4. Call `POST /transactions/signers` with a batch containing both A and B.
5. Observe: HTTP 201 is returned. `signers` array contains 2 records (both for Transaction A).
6. In `signItems`: `items.length = 2`, `signers.length = 2`, `unsignedCount = 0` → inner loop skipped.
7. The caller of `signItems` receives an empty array (no unsigned items reported).
8. Transaction B expires without its required signature. It transitions to `EXPIRED` status and cannot be executed.
