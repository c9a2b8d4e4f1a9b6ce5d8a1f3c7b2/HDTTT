### Title
`isApproved` Rejection Logic Incorrectly Marks Threshold Approver Groups as Rejected with Small Thresholds, Enabling a Single Approver to Suppress Multi-Party Approval

### Summary

The `isApproved` function in `front-end/src/renderer/utils/sdk/index.ts` uses a flawed rejection condition for hierarchical threshold approver groups. When the threshold is small relative to the total number of approvers (e.g., 1-of-3), a single rejection causes the entire group to be displayed as definitively rejected — even though the remaining approvers could still satisfy the threshold. This is the direct analog of the external report's "small threshold" failure: incorrect threshold evaluation causes a premature terminal state.

### Finding Description

The root cause is in `isApproved` at line 357:

```typescript
return rejections.length >= (approver.threshold || rejections.length) ? false : null;
``` [1](#0-0) 

The condition `rejections.length >= threshold` is mathematically wrong for the purpose of determining *definitive* rejection. A threshold group is only definitively rejected when it is **impossible** to reach the threshold — i.e., when `(total_approvers - rejections) < threshold`. The current code instead marks the group as rejected as soon as the rejection count equals the threshold, regardless of how many approvers have not yet voted.

**Concrete example — 1-of-3 threshold group:**

| State | Approver A | Approver B | Approver C | `isApproved` result |
|---|---|---|---|---|
| A rejects, B and C pending | `false` | `null` | `null` | `false` (WRONG — B or C can still approve) |
| Correct answer | — | — | — | `null` (still pending) |

Trace:
- `approvals.length = 0`, `rejections.length = 1`, `threshold = 1`
- Line 354: `0 >= 1` → `false` (not approved yet — correct)
- Line 357: `1 >= (1 || 1)` → `1 >= 1` → `true` → returns `false` (rejected — **wrong**)

The correct rejection condition is:
```typescript
const remaining = approver.approvers.length - rejections.length;
if (remaining < approver.threshold) return false; // impossible to reach threshold
return null; // still pending
```

This incorrect result propagates into every UI component that calls `isApproved`:

- `ReadOnlyApproversList.vue` applies `bg-danger` class (red badge) to the approver group [2](#0-1) 
- `ApproverStructureStatus.vue` renders a red X icon on the group [3](#0-2) 
- `TransactionDetailsHeader.vue` and `TransactionGroupDetails.vue` consume the same function to drive approval-state display

The `ITransactionApprover` interface confirms `threshold` is a plain optional number, so any value ≥ 1 and less than the child count is a valid small threshold: [4](#0-3) 

The backend enforces that `threshold ≥ 1` and `threshold ≤ approvers.length`, so threshold=1 with N>1 approvers is a fully valid, server-accepted configuration: [5](#0-4) 

### Impact Explanation

Any organization using a threshold approver group where `threshold < total_approvers` (e.g., 1-of-3, 2-of-5) is affected. When a single approver in such a group rejects, the UI immediately displays the entire approval group as definitively rejected (red badge, X icon). Other approvers who have not yet voted see a terminal "rejected" state and are likely to conclude no further action is needed, effectively suppressing the approval process. The transaction remains stuck in a state where the backend has not actually rejected it, but the UI presents it as rejected — a soft-lock of the multi-party approval workflow.

### Likelihood Explanation

The attacker is a **malicious normal user** who is a legitimate member of a threshold approver group. No privileged access is required. The exploit requires only submitting a rejection via the standard `POST /approvers/:id/approve` endpoint with `approved: false`. This is a reachable, no-privilege action within the normal product workflow. Threshold approver groups with small thresholds (e.g., 1-of-N) are a natural and common configuration for organizations that want any one of several designated approvers to be able to unblock a transaction.

### Recommendation

Replace the rejection condition in `isApproved` with a mathematically correct impossibility check:

```typescript
// Current (wrong):
return rejections.length >= (approver.threshold || rejections.length) ? false : null;

// Correct:
const totalApprovers = approver.approvers.length;
const remaining = totalApprovers - rejections.length;
if (remaining < (approver.threshold ?? totalApprovers)) return false;
return null;
```

This ensures the group is only marked as rejected when it is provably impossible for the remaining pending approvers to satisfy the threshold.

### Proof of Concept

**Setup**: Create a transaction with a threshold approver group: threshold=1, approvers=[UserA, UserB, UserC].

**Steps**:
1. Log in as UserA and reject the transaction (`approved: false`).
2. Open the transaction details page in the UI.
3. Observe: the approver group displays with a red badge and X icon (`isApproved` returns `false`).
4. Log in as UserB (who has not voted). The UI shows the approval group as already rejected.
5. UserB, seeing the rejected state, does not submit their approval.

**Expected**: The group should display as pending (`null`) because UserB and UserC have not voted and either could satisfy the 1-of-3 threshold.

**Actual**: The group displays as definitively rejected after a single rejection, suppressing participation from the remaining 2 approvers. [6](#0-5)

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }

  return null;
};
