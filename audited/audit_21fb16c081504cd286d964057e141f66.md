### Title
Unauthorized Signature Injection via Missing Transaction Access Control in `uploadSignatureMaps`

### Summary
The `uploadSignatureMaps` function in `signers.service.ts` does not verify that the requesting user has any legitimate relationship to a transaction before allowing them to inject their cryptographic signature into it. Any authenticated, verified user with at least one registered key can upload their signature to any active transaction in the system. This is a direct access-control omission, inconsistent with the parallel `importSignatures` path which correctly calls `verifyAccess` before processing.

### Finding Description

**Root Cause**

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, the `validateAndProcessSignatures` private method performs no transaction-level authorization check: [1](#0-0) 

The only checks performed are:
1. Transaction exists in the DB
2. Transaction status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`
3. The public key in the submitted `SignatureMap` belongs to the requesting user's registered keys

There is no check that the user is the creator, a designated signer, an observer, or an approver of the transaction.

**Contrast with `importSignatures`**

The parallel endpoint `POST /transactions/signatures/import` in `transactions.service.ts` correctly calls `verifyAccess` before processing: [2](#0-1) 

The `verifyAccess` function enforces that the user must be the creator, observer, signer, approver, or hold a key required by the transaction: [3](#0-2) 

**Attack Path**

1. Attacker registers as a normal authenticated user and uploads at least one key pair via `POST /user/:userId/keys`.
2. Attacker discovers an active transaction ID. Transaction IDs are sequential integers; the attacker can enumerate them by submitting `POST /transactions/:id/signers` and observing whether the response is `TNF` (not found) or a status error — both confirm the transaction exists.
3. Attacker constructs a valid `SignatureMap` containing their own public key and a valid signature over the target transaction's body bytes (which they can compute locally since the Hedera SDK is open source and the transaction body format is deterministic).
4. Attacker submits `POST /transactions/:transactionId/signers` with the crafted payload.
5. `validateAndProcessSignatures` accepts the payload because the public key is in the attacker's registered key set.
6. `processTransactionSignatures` calls `sdkTransaction.addSignature(publicKey, map)`, mutating the transaction bytes.
7. The mutated `transactionBytes` are written back to the database via `bulkUpdateTransactions`.
8. A `TransactionSigner` record is inserted linking the attacker's user ID to the transaction.
9. `processTransactionStatus` is invoked, potentially advancing the transaction's status. [4](#0-3) [5](#0-4) 

The controller applies only authentication guards — no transaction-ownership guard exists at this layer: [6](#0-5) 

### Impact Explanation

**Unauthorized state mutation of any active transaction.** An attacker can:

- **Inject their signature into any active transaction**, permanently modifying the stored `transactionBytes` in the database. This is an integrity violation of the transaction record.
- **Create a fraudulent signer record** (`transaction_signer` row) linking their user ID to a transaction they have no legitimate role in, corrupting the audit trail and notification system.
- **Advance transaction status**: `processTransactionStatus` is called after every upload. If the attacker's key happens to satisfy a required signing key in the transaction's Hedera key structure (e.g., the attacker controls a key that is part of a threshold key on the paying account), the transaction can be moved to `WAITING_FOR_EXECUTION` and subsequently submitted to the Hedera network — causing unauthorized fund movement or account state changes.
- **Inflate transaction size**: Repeated signature injection can push `transactionBytes` past the Hedera network's size limit, causing the transaction to fail permanently with `TRANSACTION_OVERSIZE`. [7](#0-6) 

### Likelihood Explanation

**High reachability, low precondition.** The attacker needs only:
- A valid account on the organization server (normal user, no admin role required)
- At least one registered key pair (required to use the system at all)
- Knowledge of a target transaction ID (sequential integers, discoverable via the response code difference between `TNF` and `TNRS`/`TE`)

No privileged access, no leaked credentials, and no cryptographic break are required. The endpoint is publicly reachable over HTTPS by any verified user.

### Recommendation

Add a `verifyAccess` call inside `validateAndProcessSignatures` in `back-end/apps/api/src/transactions/signers/signers.service.ts`, mirroring the check already present in `importSignatures`. The transaction must be loaded with the required relations (`creatorKey`, `observers`, `signers`, `approvers`) — note that `loadTransactionData` currently loads transactions **without** these relations, so the query must be updated: [8](#0-7) 

The fix should:
1. Load transactions with `relations: ['creatorKey', 'approvers', 'signers', 'observers']` in `loadTransactionData`.
2. Call `verifyAccess(transaction, user)` at the top of the per-transaction processing loop in `validateAndProcessSignatures`, returning `{ id, error: ErrorCodes.TNF }` on failure (consistent with how `importSignatures` handles it).

### Proof of Concept

**Preconditions:**
- Attacker has a valid JWT (`attacker_token`) for a normal verified user account
- Attacker has registered public key `PK_attacker` with key ID `k1`
- Target transaction ID is `42`, status `WAITING_FOR_SIGNATURES`
- Attacker is NOT the creator, observer, signer, or approver of transaction `42`

**Steps:**

```
# 1. Attacker fetches transaction bytes to construct a valid signature
#    (transaction bytes are not secret; the attacker can obtain them
#     via the GET /transactions/:id endpoint if they can guess the ID,
#     or construct a valid signature over the known body format)

# 2. Attacker signs the transaction body bytes with their private key SK_attacker
#    producing signature SIG_attacker

# 3. Attacker submits the signature map
POST /transactions/42/signers
Authorization: Bearer <attacker_token>
Content-Type: application/json

[{
  "id": 42,
  "signatureMap": {
    "0.0.3": {
      "<transactionId>": {
        "<PK_attacker_DER>": "<SIG_attacker_hex>"
      }
    }
  }
}]

# Expected (correct) response: 401 Unauthorized
# Actual response: 201 Created
# Result: transaction_signer row inserted for attacker's userId,
#         transactionBytes updated with attacker's signature,
#         processTransactionStatus called
```

**Verification:** Query the `transaction_signer` table — a row with `userId = attacker_id` and `transactionId = 42` will exist. The `transactionBytes` column of transaction `42` will contain the attacker's signature. [9](#0-8)

  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
