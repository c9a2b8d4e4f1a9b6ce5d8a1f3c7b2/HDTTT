### Title
`archiveTransaction` Bypasses Terminal-State Guard for Manual Transactions, Allowing Creators to Corrupt Executed Transaction Status

### Summary
The `archiveTransaction` service method contains a logically flawed guard condition that completely skips the status check whenever `isManual` is `true`. Because `isManual` is a user-controlled field supplied at transaction creation time, any transaction creator who set `isManual: true` can call `PATCH /transactions/archive/:id` on a transaction in any terminal state — including `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, or `REJECTED` — and overwrite that status with `ARCHIVED`. This violates the intended state machine and corrupts the off-chain audit trail for all other participants (signers, observers, approvers).

### Finding Description

**Root cause — flawed boolean guard in `archiveTransaction`:** [1](#0-0) 

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← short-circuits the entire status check
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** conditions are true: status is outside the allowed set **and** `isManual` is `false`. The moment `isManual` is `true`, the status check is never evaluated and the `repo.update` call that writes `ARCHIVED` proceeds unconditionally.

**Attacker-controlled entry point — `isManual` is a user-supplied field:** [2](#0-1) 

Any authenticated user can set `isManual: true` when creating a transaction via `POST /transactions`. The field is optional but fully user-controlled with no server-side restriction.

**Exposed HTTP endpoint:** [3](#0-2) 

**Exploit flow:**
1. Attacker (creator) submits a transaction with `isManual: true`.
2. Other participants sign and approve; the scheduler executes it — status becomes `EXECUTED`.
3. Attacker calls `PATCH /transactions/archive/:id`.
4. The `isManual = true` flag short-circuits the status guard; `repo.update({ id }, { status: ARCHIVED })` runs unconditionally.
5. The transaction's DB status is now `ARCHIVED` instead of `EXECUTED`.

**Downstream effect on `getTransactionById`:** [4](#0-3) 

`ARCHIVED` is classified as an "inactive" status. All subsequent lookups by `transactionId` return the record with status `ARCHIVED`, so every participant — signers, observers, approvers — now sees the transaction as archived rather than executed.

**Test gap — the existing unit test only covers `isManual` unset (falsy):** [5](#0-4) 

No test exercises the case `{ status: EXECUTED, isManual: true }`, so the bypass is untested.

### Impact Explanation

A transaction creator can unilaterally rewrite the recorded outcome of any manual transaction — including one that has already been executed on the Hedera network — from `EXECUTED` (or `FAILED`, `EXPIRED`, `CANCELED`, `REJECTED`) to `ARCHIVED`. This:

- Corrupts the off-chain audit trail for all other participants who relied on the system's status to confirm execution.
- Causes the `HISTORY` view and any downstream reporting to misclassify the transaction's final state.
- Breaks the invariant that terminal states are immutable once reached, which is the core guarantee of the transaction state machine.

The on-chain Hedera record is unaffected, but the off-chain management system — which is the authoritative source for organizational workflows — is permanently corrupted for that record.

### Likelihood Explanation

The preconditions are minimal: the attacker must be an authenticated user who created a transaction with `isManual: true`. This is a standard, documented workflow (the front-end explicitly supports manual transactions). No elevated privileges, leaked credentials, or external dependencies are required. The endpoint `PATCH /transactions/archive/:id` is a normal authenticated REST call.

### Recommendation

Remove the `!transaction.isManual` short-circuit from the guard. The status check must apply unconditionally, regardless of the `isManual` flag:

```typescript
// Correct: always enforce the status allowlist
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

If manual transactions genuinely require a different set of archiveable states, enumerate them explicitly rather than bypassing the check entirely. Add unit tests covering `{ status: EXECUTED, isManual: true }` and all other terminal states with `isManual: true` to prevent regression.

### Proof of Concept

**Preconditions:** Two authenticated users — `creator` and `signer` — exist in the system.

1. `creator` calls `POST /transactions` with body `{ ..., isManual: true }`. Server responds with `{ id: 42, status: "WAITING FOR SIGNATURES" }`.
2. `signer` signs the transaction; status advances to `WAITING FOR EXECUTION`.
3. The scheduler executes the transaction on Hedera; status becomes `EXECUTED`.
4. `creator` calls `PATCH /transactions/archive/42` (authenticated as creator).
5. **Expected (correct) behavior:** `400 Bad Request` — terminal state cannot be archived.
6. **Actual behavior:** `200 OK` — `repo.update({ id: 42 }, { status: "ARCHIVED" })` executes because `isManual = true` short-circuits the guard at line 715.
7. `signer` calls `GET /transactions/42` and receives `{ status: "ARCHIVED" }` instead of `{ status: "EXECUTED" }`. [6](#0-5)
