### Title
Observers and Approvers Can Be Added to Terminal-State Transactions

### Summary
The `createTransactionObservers` function in `ObserversService` and the `createTransactionApprovers` function in `ApproversService` perform no check on the transaction's current status before persisting new records. A transaction creator can therefore add observers or approvers to transactions that are already in a terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`), violating the state-machine invariant that terminal transactions are immutable. This is the direct analog of the Futureswap M02 finding: a mutating operation on a closed/terminal entity is not gated by a state guard.

### Finding Description

**Root cause — `createTransactionObservers`**

`ObserversService.createTransactionObservers` fetches the transaction, checks existence and creator identity, then immediately persists new `TransactionObserver` rows: [1](#0-0) 

The only guards are:
- `if (!transaction) throw …` (line 42)
- `if (transaction.creatorKey?.userId !== user.id) throw …` (line 44)

There is **no** `transaction.status` check. A transaction with status `EXECUTED`, `CANCELED`, `FAILED`, `EXPIRED`, or `ARCHIVED` passes both guards and new observers are saved.

**Root cause — `createTransactionApprovers`**

`ApproversService.createTransactionApprovers` delegates its authorization to `getCreatorsTransaction`: [2](#0-1) 

`getCreatorsTransaction` only checks existence and creator identity — no status guard: [3](#0-2) 

**Root cause — `updateTransactionObserver` / `removeTransactionObserver`**

Both delegate to `getUpdateableObserver`, which again only checks existence and creator identity, with no status guard: [4](#0-3) 

**Contrast with correct guards elsewhere**

Every other mutating operation on a transaction correctly rejects terminal states. For example, `validateTransactionStatus` in the execute path: [5](#0-4) 

And `approveTransaction` in `ApproversService`: [6](#0-5) 

The terminal-state set is well-defined in the codebase: [7](#0-6) 

**Exploit path**

1. Attacker (transaction creator) creates a transaction; it reaches `EXECUTED` or `CANCELED`.
2. Attacker calls `POST /transactions/:id/observers` with `{ userIds: [victimId] }`.
3. `createTransactionObservers` passes both guards (transaction exists, caller is creator) and persists the new `TransactionObserver` row.
4. `emitTransactionUpdate` fires, sending a spurious notification to the victim.
5. The victim now appears in the observer list of a terminal transaction, corrupting the immutable audit record.
6. Because `getTransactionObserversByTransactionId` short-circuits access control for terminal transactions (lines 92–101 of `observers.service.ts`), the newly added observer can immediately retrieve the full observer list without any further check. [8](#0-7) 

### Impact Explanation

- **Audit-trail corruption**: The observer and approver lists for a terminal transaction are part of its immutable historical record. Post-execution mutation breaks the integrity of that record.
- **Unauthorized access grant**: Adding an observer to a terminal transaction causes the short-circuit access-control path (lines 92–101) to return the full observer list to the newly added user, granting them visibility they should not receive after the transaction closed.
- **Notification spam / confusion**: `emitTransactionUpdate` is called on every successful observer addition, sending real-time notifications about transactions that are already in a terminal state, which can mislead users and downstream consumers.
- **Approver record manipulation**: Approvers can be added to or removed from already-executed transactions, retroactively altering who was recorded as an approver.

### Likelihood Explanation

The attacker precondition is minimal: any authenticated, verified user who is the creator of at least one transaction. No privileged role is required. The endpoint `POST /transactions/:transactionId/observers` is a standard REST call reachable by any creator. The transaction's terminal status is never surfaced as a rejection reason, so the operation succeeds silently. This is a straightforward, low-effort call that any creator can make at any time after their transaction closes.

### Recommendation

Add a terminal-state guard at the top of `createTransactionObservers`, `updateTransactionObserver`, `removeTransactionObserver`, `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` (or centralize it in `getUpdateableObserver` and `getCreatorsTransaction`):

```typescript
const TERMINAL_STATUSES = [
  TransactionStatus.EXECUTED,
  TransactionStatus.FAILED,
  TransactionStatus.EXPIRED,
  TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED,
];

if (TERMINAL_STATUSES.includes(transaction.status)) {
  throw new BadRequestException('Cannot modify a transaction in a terminal state');
}
```

This mirrors the pattern already used in `validateTransactionStatus` and `approveTransaction`.

### Proof of Concept

**Preconditions**: Authenticated as the creator of transaction ID `42`, which has status `EXECUTED`.

```http
POST /transactions/42/observers
Authorization: Bearer <creator_jwt>
Content-Type: application/json

{ "userIds": [99] }
```

**Expected (correct) response**: `400 Bad Request` — transaction is in a terminal state.

**Actual response**: `201 Created` — observer row for user `99` is inserted into `transaction_observer`, `emitTransactionUpdate` fires, and user `99` can now call `GET /transactions/42/observers` and receive the full observer list due to the terminal-state short-circuit at lines 92–101 of `observers.service.ts`.

The same request pattern applies to `POST /transactions/:id/approvers` for the approver variant.
