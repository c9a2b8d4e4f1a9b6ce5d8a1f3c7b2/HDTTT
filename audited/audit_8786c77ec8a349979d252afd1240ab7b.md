### Title
Incorrect `in` Operator Usage in `addNodeKeys` Causes Council Account Bypass to Never Trigger for `NodeDeleteTransaction`

### Summary
In `back-end/libs/common/src/transaction-signature/transaction-signature.service.ts`, the `addNodeKeys` method uses the JavaScript `in` operator to check whether a fee payer is a council account. Because `in` checks for **property indices** on arrays (not values), the check `payerId.toString() in COUNCIL_ACCOUNTS` always evaluates to `false`. As a result, the admin key is always added as a required signature for `NodeDeleteTransaction`, even when the fee payer is a council account — directly contradicting the intended bypass logic.

### Finding Description

The `addNodeKeys` method contains the following logic for `NodeDeleteTransaction`:

```typescript
const payerId = sdkTransaction.transactionId?.accountId;
const isCouncilAccount = payerId && payerId.toString() in COUNCIL_ACCOUNTS;

if (!isCouncilAccount) {
    signatureKey.push(nodeInfo.admin_key);
}
return;
``` [1](#0-0) 

`COUNCIL_ACCOUNTS` is defined as a plain JavaScript array:

```typescript
export const COUNCIL_ACCOUNTS = ['0.0.2', '0.0.50', '0.0.55'];
``` [2](#0-1) 

In JavaScript/TypeScript, the `in` operator checks whether a **key (index or property name)** exists in an object. For arrays, valid keys are `'0'`, `'1'`, `'2'`, `'length'`, etc. — not the stored values. Therefore:

```
'0.0.2' in ['0.0.2', '0.0.50', '0.0.55']  // → false (not an index)
'0'     in ['0.0.2', '0.0.50', '0.0.55']  // → true  (index 0 exists)
```

Since `payerId.toString()` returns a Hedera account ID string like `'0.0.2'`, the expression `payerId.toString() in COUNCIL_ACCOUNTS` is **always `false`**, making `isCouncilAccount` always falsy. The `if (!isCouncilAccount)` branch always executes, and `nodeInfo.admin_key` is always pushed into the required signature key list.

The intended behavior (documented in the inline comment) is:

> "if fee payer is council_accounts, it will already be added to the required list and the admin key is not required (as the fee payer will already approve it)" [3](#0-2) 

This bypass never fires. The correct check should be `COUNCIL_ACCOUNTS.includes(payerId.toString())`.

### Impact Explanation

The `signatureKey` KeyList built by `addNodeKeys` is consumed by `computeSignatureKey`, which is called by `keysRequiredToSign` and ultimately by `processTransactionStatus` to determine whether a transaction has collected enough signatures to advance to `WAITING_FOR_EXECUTION`. [4](#0-3) 

When a council account (e.g., `0.0.2`) is the fee payer for a `NodeDeleteTransaction`, the system incorrectly requires the node admin key in addition to the fee payer's key. The transaction will remain stuck in `WAITING_FOR_SIGNATURES` indefinitely unless the admin key also signs — even though the protocol design explicitly exempts council-account-initiated node deletions from this requirement.

### Likelihood Explanation

Any `NodeDeleteTransaction` submitted with a council account as the fee payer triggers this bug unconditionally. The `COUNCIL_ACCOUNTS` list (`0.0.2`, `0.0.50`, `0.0.55`) represents privileged Hedera governance accounts that are expected to be able to delete nodes without requiring the admin key. This is a realistic operational scenario in a Hedera network management context.

### Recommendation

Replace the `in` operator with `Array.prototype.includes`:

```typescript
// Before (broken):
const isCouncilAccount = payerId && payerId.toString() in COUNCIL_ACCOUNTS;

// After (correct):
const isCouncilAccount = payerId && COUNCIL_ACCOUNTS.includes(payerId.toString());
``` [5](#0-4) 

### Proof of Concept

Given:
- `COUNCIL_ACCOUNTS = ['0.0.2', '0.0.50', '0.0.55']`
- A `NodeDeleteTransaction` with `transactionId.accountId = AccountId.fromString('0.0.2')`

Execution trace:
1. `payerId.toString()` → `'0.0.2'`
2. `'0.0.2' in ['0.0.2', '0.0.50', '0.0.55']` → `false` (JS `in` checks indices, not values)
3. `isCouncilAccount` → `false`
4. `!isCouncilAccount` → `true`
5. `signatureKey.push(nodeInfo.admin_key)` — admin key incorrectly required
6. `processTransactionStatus` sees admin key as missing → transaction stays `WAITING_FOR_SIGNATURES`
7. Council-account-initiated node deletion is permanently blocked without admin key co-signature [6](#0-5)
