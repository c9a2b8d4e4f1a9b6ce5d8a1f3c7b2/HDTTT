### Title
Unbounded Transaction Fetch in `getTransactionsToSign` Causes Per-Transaction External Network Calls, Enabling Resource Exhaustion DoS

### Summary
The `GET /transactions/sign` endpoint in the API service fetches **all** non-terminal transactions from the database with no database-level row limit, then performs a per-transaction async `computeSignatureKey` call (which makes external Hedera mirror-node network requests) for every row before applying pagination in memory. Any authenticated, verified user can trigger this endpoint. As the number of active transactions in the system grows, a single request can exhaust server CPU, memory, and outbound mirror-node connections, degrading or halting the service.

### Finding Description

**Root cause — no `take` limit on the database query:**

In `getTransactionsToSign`, the repository `find` call has no `take` parameter, so it returns every non-terminal transaction in the entire system: [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});  // ← no `take` / limit
```

**Per-transaction async processing with external I/O:**

For every row returned, `userKeysToSign` is called, which calls `computeSignatureKey`, which makes one or more outbound HTTP calls to the Hedera mirror node (fee-payer key, signing-account keys, receiver-account keys, node admin key): [2](#0-1) [3](#0-2) 

**Pagination applied in-memory after full processing:**

The `PaginationParams` decorator enforces `size ≤ 100`, but that limit is only applied to the final slice — the entire table scan and per-row processing happens unconditionally first: [4](#0-3) 

**Developer-acknowledged absence of API-side limits:**

The same pattern is acknowledged in a comment in `importSignatures`: [5](#0-4) 

> `//Added a batch mechanism, probably should limit this on the api side of things`

**Entry point — any verified user:**

The endpoint is protected only by standard JWT + verified-user guards; no admin role is required: [6](#0-5) 

The `PaginationParams` decorator rejects `size > 100` but places no constraint on the underlying database scan: [7](#0-6) 

### Impact Explanation

A single `GET /transactions/sign?page=1&size=1` request causes the server to:

1. Load **all** non-terminal transactions (potentially thousands) from PostgreSQL into memory.
2. Deserialize each transaction's bytes (`SDKTransaction.fromBytes`).
3. Issue one or more outbound HTTP calls to the Hedera mirror node per transaction.
4. Hold all results in a JavaScript array before slicing to the requested page.

With a large transaction backlog, this exhausts server heap memory, saturates the outbound mirror-node connection pool, and blocks the Node.js event loop — degrading or crashing the API service for all users. Repeated requests amplify the effect without any per-user rate limiting visible in the codebase.

### Likelihood Explanation

- **Attacker precondition:** A valid JWT for any verified organization user with at least one registered key — a normal product account.
- **Trigger:** A single HTTP GET request to `/transactions/sign`.
- **Amplification:** The attacker does not need to create the transactions; any existing backlog of active transactions in the organization is sufficient. The attacker can also create transactions themselves (each requires a valid Hedera transaction signed with their own key, which is a low-cost operation).
- **No rate limiting** on this endpoint is visible in the codebase.

### Recommendation

1. **Apply a database-level `take` limit** in `getTransactorsToSign` before the loop, mirroring the pattern used in `getTransactions` and `getHistoryTransactions` which correctly pass `skip`/`take` to the ORM query.
2. **Filter at the database level** — push the "user must sign" predicate into SQL (joining on `transaction_signer` / `transaction_approver`) rather than fetching all rows and filtering in memory.
3. **Add an API-side array-size cap** on `POST /transactions/signers` and `POST /transactions/signatures/import` (the developer comment at line 575 already flags this gap).
4. **Apply rate limiting** (e.g., NestJS `ThrottlerGuard`) to all transaction-query endpoints.

### Proof of Concept

**Preconditions:**
- Organization has ≥ 1,000 active (non-terminal) transactions.
- Attacker holds a valid JWT for any verified user with one registered key.

**Steps:**

```bash
# Repeat rapidly to sustain load
for i in $(seq 1 20); do
  curl -s -o /dev/null \
    -H "Authorization: Bearer <VALID_JWT>" \
    "https://<api-host>/transactions/sign?page=1&size=1" &
done
wait
```

**Expected outcome:** Each request triggers a full table scan of all non-terminal transactions followed by N mirror-node calls. With 1,000 active transactions and 20 concurrent requests, the server issues ≥ 20,000 outbound mirror-node calls simultaneously, exhausts the Node.js heap, and returns 500 errors or becomes unresponsive for all users.

**Code path:**
`GET /transactions/sign` → `TransactionsController.getTransactionsToSign` → `TransactionsService.getTransactionsToSign` → `repo.find({where, relations, order})` (no `take`) → `for (transaction of transactions) { userKeysToSign(transaction, user) }` → `computeSignatureKey` → `accountCacheService.getAccountInfoForTransaction` (mirror-node HTTP call per account). [8](#0-7)

```typescript
  async computeSignatureKey(
    transaction: Transaction,
    showAll: boolean = false,
  ): Promise<KeyList> {
    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const transactionModel = TransactionFactory.fromTransaction(sdkTransaction);

    // Extract signature requirements from the transaction model
    const requirements = this.extractSignatureRequirements(transactionModel);

    // Build the key list
    const signatureKey = new KeyList();

    await this.addFeePayerKey(signatureKey, transaction, requirements.feePayerAccount);
    await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
    await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);

    if (requirements.nodeId) {
      await this.addNodeKeys(signatureKey, transaction, requirements.nodeId);
    }

    signatureKey.push(...requirements.newKeys);

    return signatureKey;
  }
```
