### Title
Approver Approval Status Is Not Enforced Before Transaction Execution

### Summary
The Hedera Transaction Tool supports an approver workflow where designated users must approve a transaction before it proceeds. However, the execution path — both the status-transition engine (`processTransactionStatus`) and the chain-service executor (`executeTransaction`) — never checks whether required approvers have actually approved. A transaction with pending or rejected approvers will still be promoted to `WAITING_FOR_EXECUTION` and submitted to the Hedera network, completely bypassing the organizational approval gate. This is the direct analog of the HolographOperator missing-fee bug: a business requirement is documented and partially wired up, but the enforcement step is absent from the critical code path.

### Finding Description

**Root cause — status transition does not consult approvers**

`processTransactionStatus` is the function that decides when a transaction moves from `WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION`. It checks only whether the collected cryptographic signatures satisfy the Hedera key structure: [1](#0-0) 

```
const isAbleToSign = hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey);
let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;
if (isAbleToSign) {
  const collatedTx = await smartCollate(transaction, signatureKey);
  if (collatedTx !== null) {
    newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
  }
}
```

There is no call to fetch approvers, no check of `approved` flags, and no guard that blocks promotion when approvers have not yet acted.

**Root cause — execution service does not consult approvers**

`ExecuteService.executeTransaction` calls `getValidatedSDKTransaction`, which again only validates the cryptographic signature key: [2](#0-1) 

No approver-status check exists anywhere in this path.

**Stub that confirms the gap**

`getTransactionApproversForTransactions` — the method that would supply approver data to the execution pipeline — is explicitly stubbed out and returns an empty array: [3](#0-2) 

```typescript
//To be implemented when approver functionality is added.
return [];
```

**Approver recording works, enforcement does not**

`ApproversService.approveTransaction` correctly records each user's approval or rejection in the database: [4](#0-3) 

But nothing downstream reads those records before execution. The `approved` column described in the schema is populated but never consulted by the chain service. [5](#0-4) 

### Impact Explanation

Any transaction that has approvers assigned — including threshold-based approval trees — will be executed on the Hedera network as soon as it accumulates sufficient cryptographic signatures, regardless of whether the designated approvers have approved, rejected, or not yet responded. This completely nullifies the organizational approval control: a transaction creator can set up approvers to satisfy a compliance or governance requirement, but those approvers have no actual power to block execution. Unauthorized or premature on-chain transactions (account updates, token transfers, node changes, etc.) can be submitted without the required human approval.

### Likelihood Explanation

The approver feature is reachable through the normal API (`POST /transactions/:id/approvers`). Any authenticated user who is a transaction creator can attach approvers to a transaction. Once the transaction collects enough Hedera key signatures, the chain service scheduler (`updateTransactions` → `processTransactionStatus` → `collateAndExecute`) will automatically promote and execute it, ignoring all approver state. No privileged access is required beyond being a normal organization user. [6](#0-5) 

### Recommendation

Before promoting a transaction to `WAITING_FOR_EXECUTION` inside `processTransactionStatus`, query the `transaction_approver` table and verify that the root-level approver tree evaluates to `approved = true` (using the same recursive threshold logic already implemented in `isApproved`). If any root approver has `approved = false` or `approved = null`, the transaction must remain in `WAITING_FOR_SIGNATURES` (or a dedicated `WAITING_FOR_APPROVAL` status). Apply the same guard at the start of `ExecuteService.executeTransaction`. Remove the stub in `getTransactionApproversForTransactions` and replace it with the real query already used in `ApproversService.getApproversByTransactionId`.

### Proof of Concept

1. Authenticate as a normal organization user (User A).
2. Create a transaction (e.g., a `CryptoTransfer`) via `POST /transactions`.
3. Attach an approver (User B) via `POST /transactions/:id/approvers` with `{ approversArray: [{ userId: B }] }`.
4. User A signs the transaction with their Hedera key — satisfying the cryptographic signature requirement.
5. Observe: `processTransactionStatus` promotes the transaction to `WAITING_FOR_EXECUTION` without consulting User B's approval status.
6. The chain-service scheduler calls `collateAndExecute`, which calls `executeTransaction`, and the transaction is submitted to the Hedera network.
7. User B's approval was never required, never checked, and the on-chain action completes without their consent. [7](#0-6) [3](#0-2)

```typescript
export async function processTransactionStatus(
  transactionRepo: Repository<Transaction>,
  transactionSignatureService: TransactionSignatureService,
  transactions: Transaction[],
): Promise<Map<number, TransactionStatus>> {
  const statusChanges = new Map<number, TransactionStatus>();

  // Group intended updates by [newStatus, oldStatus] so we can bulk update
  // only rows that still have the expected current status
  const updatesByStatus = new Map<string, { newStatus: TransactionStatus, oldStatus: TransactionStatus, ids: number[] }>();

  for (const transaction of transactions) {
    if (!transaction) continue;

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
    const isAbleToSign = hasValidSignatureKey(
      [...sdkTransaction._signerPublicKeys],
      signatureKey
    );

    let newStatus = TransactionStatus.WAITING_FOR_SIGNATURES;

    if (isAbleToSign) {
      const collatedTx = await smartCollate(transaction, signatureKey);
      if (collatedTx !== null) {
        newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
      }
    }

    if (transaction.status !== newStatus) {
      const key = `${transaction.status}->${newStatus}`;
      if (!updatesByStatus.has(key)) {
        updatesByStatus.set(key, { newStatus, oldStatus: transaction.status, ids: [] });
      }
      updatesByStatus.get(key)!.ids.push(transaction.id);
    }
  }

  if (updatesByStatus.size > 0) {
    await Promise.all(
      Array.from(updatesByStatus.values()).map(async ({ newStatus, oldStatus, ids }) => {
        const result = await transactionRepo
          .createQueryBuilder()
          .update(Transaction)
          .set({ status: newStatus })
          .where('id IN (:...ids) AND status = :oldStatus', { ids, oldStatus })
          .returning('id')
          .execute();

        for (const row of result.raw) {
          statusChanges.set(row.id, newStatus);
        }
      })
    );
  }

  return statusChanges;
}
```

```typescript
  /* Checks if the signers are enough to sign the transactions and update their statuses */
  async updateTransactions(from: Date, to?: Date) {
    //Get the transaction, creatorKey, groupItem, and group. We need the group info upfront
    //in order to determine if the group needs to be processed together
    const transactions = await this.transactionRepo.find({
      where: {
        status: In([
          TransactionStatus.WAITING_FOR_SIGNATURES,
          TransactionStatus.WAITING_FOR_EXECUTION,
        ]),
        validStart: to ? Between(from, to) : MoreThan(from),
      },
      relations: {
        creatorKey: true,
        groupItem: {
          group: true,
        },
      },
      order: {
        validStart: 'ASC',
      },
    });

    const results = await processTransactionStatus(this.transactionRepo, this.transactionSignatureService, transactions);

    if (results.size > 0) {
      const events = Array.from(results.keys(), id => ({ entityId: id }));
      emitTransactionStatusUpdate(this.notificationsPublisher, events);
    }

    return transactions;
```
