### Title
`archiveTransaction` Guard Bypassed for Manual Transactions, Allowing Illegal State Transitions from Terminal States

### Summary
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `archiveTransaction` method uses a logically flawed guard condition that is entirely bypassed when `transaction.isManual === true`. As a result, any authenticated user who is the creator of a manual transaction can archive it regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `REJECTED`. This violates the protocol's own terminal-state invariant and corrupts the audit trail.

### Finding Description

The guard in `archiveTransaction` is:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← second operand of AND
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
``` [1](#0-0) 

Because the two sub-expressions are joined with `&&`, the exception is thrown only when **both** are true. When `isManual === true`, the second operand evaluates to `false`, making the entire condition `false` — the throw is never reached, regardless of the current status.

The codebase explicitly defines terminal states that must not be re-transitioned:

```typescript
private readonly terminalStatuses = [
  TransactionStatus.EXECUTED,
  TransactionStatus.EXPIRED,
  TransactionStatus.FAILED,
  TransactionStatus.CANCELED,
  TransactionStatus.ARCHIVED,
  TransactionStatus.REJECTED,
];
``` [2](#0-1) 

None of these are excluded from the bypass. After the guard, the code unconditionally writes `ARCHIVED` to the database:

```typescript
await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
``` [3](#0-2) 

The `isManual` flag is never cleared during execution — `_executeTransaction` only updates `status`, `executedAt`, and `statusCode`: [4](#0-3) 

So after a manual transaction reaches `EXECUTED`, `isManual` remains `true`, and the bypass is fully reachable.

The endpoint is exposed to any authenticated user via `PATCH /transactions/archive/:id`: [5](#0-4) 

### Impact Explanation

A transaction creator can overwrite the status of their own manual transaction from any terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `REJECTED`) to `ARCHIVED`. Concretely:

- **Audit trail corruption**: An `EXECUTED` transaction — one that has already been submitted to and confirmed by the Hedera network — can be re-labelled `ARCHIVED`, erasing the execution record from the application's perspective.
- **Terminal-state invariant broken**: The protocol's own `terminalStatuses` list is rendered meaningless for manual transactions; any of those states can be silently overwritten.
- **Downstream logic affected**: Any component that relies on `EXECUTED` status for reporting, compliance, or dispute resolution will receive incorrect data.

### Likelihood Explanation

- **No privileged access required**: the attacker only needs to be an authenticated user and the creator of a manual transaction — a standard, reachable role.
- **Manual transactions are a supported, documented feature**: `isManual: true` is set at creation time by any user.
- **Execution does not clear `isManual`**: the window between execution and archiving is unbounded; the attacker can call the archive endpoint at any time after execution.
- **The existing unit test does not cover this path**: the only "should throw" test uses `isManual` unset (falsy) with `CANCELED` status, so the bypass is untested. [6](#0-5) 

### Recommendation

Remove the `&& !transaction.isManual` operand from the guard. The status check should be unconditional:

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  )
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

This ensures that only transactions genuinely awaiting signatures or execution can be archived, regardless of the `isManual` flag.

### Proof of Concept

1. Authenticate as a regular user and create a manual transaction:
   ```
   POST /transactions
   { ..., isManual: true }
   → 201, { id: 42, status: "WAITING FOR EXECUTION", isManual: true }
   ```

2. Execute the transaction (validStart in the past):
   ```
   PATCH /transactions/execute/42
   → 200 true
   ```
   Database row: `status = "EXECUTED"`, `isManual = true`.

3. Immediately archive the now-executed transaction:
   ```
   PATCH /transactions/archive/42
   → 200 true
   ```

4. Verify:
   ```
   GET /transactions/42
   → { status: "ARCHIVED" }   ← was "EXECUTED"
   ```

Guard evaluation at step 3:
- `!['WAITING FOR SIGNATURES','WAITING FOR EXECUTION'].includes('EXECUTED')` → `true`
- `!true` (isManual) → `false`
- `true && false` → `false` → **no exception thrown**
- `repo.update({ id: 42 }, { status: 'ARCHIVED' })` executes unconditionally.
