### Title
Missing User-Based Access Control on `GET /transactions/history` Exposes All Organization Transactions to Any Authenticated User

### Summary

The `getHistoryTransactions` handler in `TransactionsController` omits the authenticated user from its service call, returning **all** historical transactions across the entire organization to any verified user — regardless of whether they participated in those transactions. This is the same vulnerability class as the external report: a function designed to serve a restricted caller has no access-control check enforcing that restriction.

### Finding Description

`TransactionsController` applies class-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) to all routes. However, the `getHistoryTransactions` handler does not inject `@GetUser()` and passes no user identity to the service: [1](#0-0) 

```typescript
@Get('/history')
getHistoryTransactions(
  @PaginationParams() paginationParams: Pagination,
  ...
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getHistoryTransactions(paginationParams, filter, sort);
}
```

Compare this with the sibling `getTransactions` handler, which correctly passes the authenticated user for filtering: [2](#0-1) 

```typescript
@Get()
getTransactions(
  @GetUser() user: User,
  ...
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getTransactions(user, paginationParams, sort, filter);
}
```

The `getHistoryTransactions` call omits `user` entirely, so the service has no basis to restrict results to the caller's transactions.

A second instance of the same pattern exists in `TransactionGroupsController`, where a route explicitly commented `/* TESTING ONLY */` also omits `@GetUser()` and returns all groups: [3](#0-2) 

```typescript
/* TESTING ONLY: Get all transactions groups */
@Get()
getTransactionGroups(): Promise<TransactionGroup[]> {
  return this.transactionGroupsService.getTransactionGroups();
}
```

A third instance exists in `CommentsController`, where the developer left an explicit TODO acknowledging the missing check: [4](#0-3) 

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
)
```

### Impact Explanation

Any authenticated, verified organization member can call `GET /transactions/history` and enumerate **every** completed, failed, or expired transaction in the organization — including transactions they were never a party to. Each `TransactionDto` exposes transaction bytes, status, creator, valid-start timestamps, and associated Hedera account IDs. In an enterprise multi-signature workflow tool, this leaks the full financial activity of the organization to any low-privilege member.

The "TESTING ONLY" `GET /transaction-groups` endpoint similarly exposes all group metadata. The `POST /transactions/:transactionId/comments` endpoint allows any verified user to write comments to transactions they have no role in.

### Likelihood Explanation

Exploitation requires only a valid JWT for a verified user account — the lowest privilege level in the system. No admin rights, no special role, no key ownership is needed. The endpoint is reachable over standard HTTPS. An attacker who registers and gets verified (or compromises any low-privilege account) can immediately enumerate all historical transaction data.

### Recommendation

1. **`getHistoryTransactions`**: Inject `@GetUser() user: User` and pass it to `transactionsService.getHistoryTransactions(user, ...)`. The service should filter results to transactions where the user is creator, signer, approver, or observer — mirroring the pattern used in `getTransactions`.
2. **`getTransactionGroups`**: Either remove the "TESTING ONLY" route from production builds or apply the same user-scoping used in `getTransactionGroup(user, groupId)`.
3. **`createComment`**: Resolve the acknowledged TODO by verifying the requesting user has a role (creator, signer, approver, or observer) on the target transaction before allowing the comment to be persisted.

### Proof of Concept

```
# Step 1: Register and verify any user account (lowest privilege)
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Enumerate ALL historical transactions — no participant check
GET /transactions/history?page=1&size=50
Authorization: Bearer <JWT>

# Response: full list of every completed/failed/expired transaction
# across the entire organization, including those the attacker
# was never invited to sign, approve, or observe.
```

The root cause is identical to the external report: the function is designed to serve only authorized callers (transaction participants), but the implementation contains no check enforcing that restriction, so any caller at the authentication boundary can invoke it freely.
