### Title
Any Transaction Creator Can Delete Approvers Belonging to Other Transactions via IDOR in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the requesting user is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who is the creator of at least one transaction can supply an arbitrary approver `:id` from any other transaction and delete it, bypassing the intended authorization boundary.

### Finding Description

**Vulnerability class:** Authorization bypass / IDOR

**Root cause — controller (`approvers.controller.ts` lines 102–113):**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` only confirms the caller is the creator of the URL-supplied `transactionId`. It says nothing about whether the approver `id` belongs to that transaction.

**Service method (`approvers.service.ts` lines 533–544):**

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

The service method accepts only `id` — no `transactionId` parameter — and performs no cross-check between the approver's owning transaction and the transaction the caller was authorized against.

**Authorization check (`approvers.service.ts` lines 623–644):**

```typescript
async getCreatorsTransaction(transactionId, user, entityManager?) {
  ...
  if (transaction.creatorKey?.userId !== user.id)
    throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
  return transaction;
}
``` [3](#0-2) 

This guard is scoped entirely to `transactionId` from the URL. The `id` of the approver being deleted is never validated against it.

**Exploit path:**
1. Attacker creates transaction A (becomes its creator).
2. Attacker discovers approver ID `X` belonging to victim's transaction B (approver IDs are sequential integers, enumerable via `GET /transactions/B/approvers` if the attacker has read access, or by brute-force).
3. Attacker sends: `DELETE /transactions/A/approvers/X`
4. `getCreatorsTransaction(A, attacker)` passes — attacker is creator of A.
5. `removeTransactionApprover(X)` executes — approver X from transaction B is deleted.

### Impact Explanation
An attacker can silently remove any approver from any transaction in the system, including multi-signature governance transactions. This breaks the approval threshold, potentially allowing transactions to execute without the required number of approvals, or permanently corrupting the approval tree of critical transactions. The impact is unauthorized state mutation of other users' transaction approval workflows — a high-integrity violation.

### Likelihood Explanation
The attacker only needs to be a registered, authenticated user who has created at least one transaction. No admin privileges are required. Approver IDs are sequential integers, making enumeration trivial. The attack requires no special tooling — a single crafted HTTP DELETE request suffices.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the authorized `transactionId`. Concretely, after fetching the approver, assert:

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, entityManager);
if (rootNode?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to this transaction');
}
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and enforce the ownership check inside the service method, mirroring the pattern already used in `updateTransactionApprover`. [4](#0-3) 

### Proof of Concept

**Preconditions:**
- Attacker is a registered user (no admin required).
- Attacker has created transaction A (ID = 10).
- Victim has transaction B (ID = 20) with approver ID = 99.

**Steps:**
```
# 1. Attacker authenticates
POST /auth/login  { email: "attacker@example.com", password: "..." }
# → receives JWT

# 2. Attacker deletes victim's approver by supplying their own transactionId
DELETE /transactions/10/approvers/99
Authorization: Bearer <attacker_JWT>

# Expected (correct) response: 401 Unauthorized
# Actual response: 200 true  — approver 99 from transaction 20 is deleted
```

The `getCreatorsTransaction(10, attacker)` check passes because the attacker owns transaction 10. `removeTransactionApprover(99)` then deletes approver 99 from transaction 20 without any further validation.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
