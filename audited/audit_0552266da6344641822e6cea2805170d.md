### Title
TOCTOU Race Condition in `approveTransaction` Allows Approval Decision Override

### Summary
`approveTransaction` in `approvers.service.ts` checks whether a user has already approved a transaction using stale in-memory data fetched before the database write. Two concurrent requests from the same approver both pass the "already approved" guard because neither sees the other's write, allowing the second request to silently overwrite the first and flip the approval decision.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `approveTransaction` executes the following sequence:

1. **Line 553** — fetches `userApprovers` from the database into memory via `getVerifiedApproversByTransactionId`.
2. **Line 563** — guards against double-approval: `if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);`
3. **Lines 599–609** — writes the new `signature`, `userKeyId`, and `approved` values back to the database. [1](#0-0) [2](#0-1) 

The guard at step 2 reads the **in-memory snapshot** captured at step 1. If two HTTP requests from the same approver arrive concurrently, both snapshots show `signature = null`, both pass the guard, and both proceed to the write. The second write unconditionally overwrites the first via `.whereInIds(userApprovers.map(a => a.id))` with no conditional `WHERE signature IS NULL` clause. There is no distributed lock, no database-level row lock, and no unique constraint that serialises these two writes.

This is structurally analogous to the reported `MysteryBox` bug: a guard reads a state field that is not yet set at the time of the check, so the guard always passes for concurrent callers, allowing the protected action to execute more than once.

### Impact Explanation
A registered approver can race two concurrent POST requests to `POST /transactions/:transactionId/approvers/approve` — one with `approved: true` and one with `approved: false`. The second write wins, silently flipping the stored approval decision. Depending on the threshold logic, this can:

- Cause a transaction that reached its approval threshold to be retroactively rejected.
- Cause a transaction that was rejected to appear approved.

This is an unauthorized state change to the approval record of a live transaction, directly affecting whether the transaction proceeds to execution.

### Likelihood Explanation
The attacker only needs to be a legitimately registered approver for any transaction — a normal, unprivileged role. Sending two concurrent HTTP requests is trivial from any HTTP client (`Promise.all`, `curl` in parallel, etc.). No special timing precision is required; the window is the entire duration of the database round-trip at step 1, which is tens to hundreds of milliseconds.

### Recommendation
Replace the unconditional `UPDATE … WHERE id IN (…)` with a conditional update that only writes when `signature IS NULL`:

```typescript
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
  .whereInIds(userApprovers.map(a => a.id))
  .andWhere('signature IS NULL')   // ← atomic guard
  .execute();
```

Then check `updateResult.affected` and throw `ErrorCodes.TAP` if zero rows were updated (meaning a concurrent request already wrote the signature). This collapses the check-then-act into a single atomic database operation, eliminating the TOCTOU window. Alternatively, use the Redis `MurLock` already present in the codebase to serialise concurrent calls for the same `(transactionId, userId)` pair.

### Proof of Concept
1. User A is a registered approver for transaction T (status `WAITING_FOR_SIGNATURES`).
2. User A fires two concurrent requests:
   - Request R1: `POST /transactions/T/approvers/approve` with `{ approved: true, signature: sig1, userKeyId: k }`
   - Request R2: `POST /transactions/T/approvers/approve` with `{ approved: false, signature: sig2, userKeyId: k }`
3. Both R1 and R2 call `getVerifiedApproversByTransactionId` and read `signature = null` → both pass the guard at line 563.
4. R1 writes `approved = true, signature = sig1`.
5. R2 writes `approved = false, signature = sig2`, overwriting R1.
6. The stored approval record now shows `approved = false` even though User A's first (and intended) decision was `approved = true`.
7. If the threshold required User A's approval, the transaction is now blocked despite User A having approved it. [3](#0-2)

```typescript
  async approveTransaction(
    dto: ApproverChoiceDto,
    transactionId: number,
    user: User,
  ): Promise<boolean> {
    /* Get all the approvers */
    const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);

    /* If user is approver, filter the records that belongs to the user */
    const userApprovers = approvers.filter(a => a.userId === user.id);

    /* Check if the user is an approver */
    if (userApprovers.length === 0)
      throw new UnauthorizedException('You are not an approver of this transaction');

    /* Check if the user has already approved the transaction */
    if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);

    /* Ensures the user keys are passed */
    await attachKeys(user, this.dataSource.manager);
    if (user.keys.length === 0) return false;

    const signatureKey = user.keys.find(key => key.id === dto.userKeyId);

    /* Gets the public key that the signature belongs to */
    const publicKey = PublicKey.fromString(signatureKey?.publicKey);

    /* Get the transaction body */
    const transaction = await this.dataSource.manager.findOne(Transaction, {
      where: { id: transactionId },
      relations: { creatorKey: true, observers: true },
    });

    /* Check if the transaction exists */
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    /* Checks if the transaction is requires approval */
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    )
      throw new BadRequestException(ErrorCodes.TNRA);

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    /* Verify the signature matches the transaction */
    if (
      !verifyTransactionBodyWithoutNodeAccountIdSignature(sdkTransaction, dto.signature, publicKey)
    )
      throw new BadRequestException(ErrorCodes.SNMP);

    /* Update the approver with the signature */
    await this.dataSource.transaction(async transactionalEntityManager => {
      await transactionalEntityManager
        .createQueryBuilder()
        .update(TransactionApprover)
        .set({
          userKeyId: dto.userKeyId,
          signature: dto.signature,
          approved: dto.approved,
        })
        .whereInIds(userApprovers.map(a => a.id))
        .execute();
    });

    const notificationEvent = [{ entityId: transaction.id }];

    if (!dto.approved || userApprovers.every(a => a.approved)) {
      emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
    } else {
      emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
    }

    return true;
  }
```
