### Title
Cross-Transaction Approver Deletion: Any Transaction Creator Can Delete Approvers Belonging to Other Transactions

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` verifies the caller is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver belongs to `:transactionId`. Any authenticated user who is the creator of *any* transaction can delete approvers from *any other* transaction, bypassing the approval threshold requirement for that transaction.

### Finding Description

**Root Cause:**

In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations: [1](#0-0) 

```
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no cross-check
```

`getCreatorsTransaction` only confirms the authenticated user is the creator of the URL-supplied `transactionId`: [2](#0-1) 

`removeTransactionApprover` then fetches and deletes the approver by `id` with no verification that it belongs to `transactionId`: [3](#0-2) 

**Contrast with `updateTransactionApprover`**, which correctly validates the cross-reference: [4](#0-3) 

```typescript
const rootNode = await this.getRootNodeFromNode(approver.id, transactionalEntityManager);
if (rootNode.transactionId !== transactionId)
  throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);
```

The `PATCH` path has this guard; the `DELETE` path does not.

**Exploit Flow:**

1. Attacker registers as a normal user and creates any transaction (e.g., `transactionId = 1`). This satisfies `getCreatorsTransaction`.
2. Attacker enumerates or guesses the `id` of an approver belonging to a victim's transaction (e.g., `approver.id = 99`, belonging to `transactionId = 42`).
3. Attacker sends: `DELETE /transactions/1/approvers/99`
4. `getCreatorsTransaction(1, attacker)` passes — attacker is creator of transaction 1.
5. `removeTransactionApprover(99)` executes — approver 99 (belonging to transaction 42) is soft-deleted with no ownership check.

### Impact Explanation

- **Approval threshold bypass**: An attacker can delete required approvers from any transaction, reducing the effective approval threshold below what the transaction creator intended. If a transaction requires 3-of-5 approvals and 2 approvers are deleted, it may become executable with fewer approvals than designed.
- **Disruption of multi-signature workflows**: Legitimate approvers are silently removed from transactions they are supposed to govern, breaking the organization's trust model.
- **Cross-tenant integrity failure**: Any user who has ever created a transaction (a normal, unprivileged action) gains the ability to tamper with approval structures of all other transactions in the system.

### Likelihood Explanation

- **Attacker precondition**: Only requires a valid authenticated account and having created at least one transaction — both are normal, unprivileged actions available to any organization member.
- **Approver IDs are sequential integers** (standard auto-increment primary keys), making enumeration trivial.
- No rate limiting or anomaly detection is needed to exploit this; a single crafted HTTP request is sufficient.

### Recommendation

In `removeTransactionApprover` (service), add the same cross-reference check that `updateTransactionApprover` already performs:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Add: verify the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException(this.ROOT_TRANSACTION_NOT_SAME);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller into the service call, mirroring the pattern already used in `updateTransactionApprover`.

### Proof of Concept

```
# Step 1: Attacker creates their own transaction (gets transactionId=1)
POST /transactions
Authorization: Bearer <attacker_token>
{ ...valid transaction body... }

# Step 2: Attacker discovers approver id=99 belongs to victim's transaction (id=42)
# (via sequential enumeration or leaked data)

# Step 3: Attacker deletes victim's approver using their own transactionId as the path param
DELETE /transactions/1/approvers/99
Authorization: Bearer <attacker_token>

# Result: HTTP 200 true
# Approver 99 is soft-deleted from transaction 42.
# Transaction 42's approval threshold is now unmet by fewer approvers than intended.
```

Expected: `403 Unauthorized` — approver 99 does not belong to transaction 1.
Actual: `200 true` — approver 99 is deleted with no cross-ownership check.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
