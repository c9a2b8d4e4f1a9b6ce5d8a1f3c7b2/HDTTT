### Title
Missing Authorization Check Allows Any Authenticated User to Post Comments on Any Transaction

### Summary
The `CommentsController.createComment` endpoint in the API service lacks an authorization check to verify that the requesting user has any relationship to the target transaction. Any authenticated, verified user can post comments on any transaction in the system, regardless of whether they are a creator, approver, signer, or observer of that transaction. A `// TODO` comment in the code explicitly acknowledges this gap.

### Finding Description
**Root cause:** `CommentsController` applies only authentication guards at the class level (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`), but the `createComment` handler has no authorization check to verify the requesting user is a participant in the target transaction. [1](#0-0) 

The `// TODO` comment on line 22 explicitly acknowledges the missing check: [2](#0-1) 

The service layer performs no relationship check either — it directly creates and persists the comment with the caller-supplied `transactionId`: [3](#0-2) 

**Exploit flow:**
1. Attacker registers as a normal user and obtains a valid JWT.
2. Attacker enumerates transaction IDs (integer IDs, trivially guessable or discoverable via `GET /transactions`).
3. Attacker sends `POST /transactions/:transactionId/comments` with any `transactionId` and arbitrary comment content.
4. The comment is persisted and visible to all participants of that transaction.

Contrast this with other controllers such as `ApproversController`, which calls `getCreatorsTransaction` to enforce ownership before mutations: [4](#0-3) 

No equivalent check exists in `CommentsController`.

### Impact Explanation
An attacker with a valid account can inject arbitrary comments into any transaction's discussion thread — including high-value multi-signature transactions awaiting approval. Approvers reading the comment thread may be misled by fabricated status updates, false urgency claims, or spoofed instructions, causing them to approve or reject transactions based on attacker-controlled information. This constitutes an unauthorized state-change path (influencing approval decisions) and a cross-tenant data integrity failure.

### Likelihood Explanation
The precondition is only a valid authenticated account — the lowest possible privilege level in the system. Transaction IDs are sequential integers, making enumeration trivial. The `// TODO` comment confirms the gap is known and unaddressed. Any registered user can exploit this immediately with a single HTTP request.

### Recommendation
In `CommentsController.createComment`, verify that the requesting user is a participant (creator, approver, signer, or observer) of the target transaction before persisting the comment. The existing `TransactionsService.getTransactionWithVerifiedAccess` or a similar participant-check method should be called before delegating to `CommentsService.createComment`. Remove the `// TODO` comment once the check is implemented.

### Proof of Concept
```

POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Post a comment on a transaction the attacker has no relation to
POST /transactions/1/comments
Authorization: Bearer <JWT>
{ "content": "Approved by admin — please proceed immediately." }

# Expected (current): 201 Created — comment persisted on transaction #1
# Expected (fixed):   403 Forbidden — user is not a participant
