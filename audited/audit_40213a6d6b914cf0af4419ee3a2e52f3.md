### Title
Unbounded In-Memory Loop in `getTransactionsToSign` Causes Server-Side Resource Exhaustion

### Summary

The `getTransactionsToSign` method in the API backend fetches **all** non-terminal transactions from the database without any row limit, then iterates over every result making an async database call per transaction to compute signing requirements. Pagination is applied only after the full loop completes. This is a direct analog to the external report's unbounded-loop pattern: a single authenticated API request triggers O(N) sequential database queries where N is the total number of active transactions in the system, causing memory and CPU exhaustion that degrades or crashes the service.

### Finding Description

**Root cause — `back-end/apps/api/src/transactions/transactions.service.ts`, lines 295–313:**

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← NO `take` / `skip` applied here
});

for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}

return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit), // ← pagination applied AFTER full loop
  page,
  size,
};
``` [1](#0-0) 

The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope the query to the requesting user's transactions — it returns every non-terminal transaction in the system. [2](#0-1) 

For each transaction returned, `userKeysToSign` is called inside the loop. This function performs additional database lookups (signature key computation, signer resolution) per transaction — creating an N+1 query pattern with no upper bound.

The `{ page, limit, size, offset }` pagination parameters accepted by the method signature are only used at line 313 (`result.slice(offset, offset + limit)`) — **after** the entire dataset has been loaded and iterated. [3](#0-2) 

### Impact Explanation

A single HTTP request to the "transactions to sign" endpoint causes:

1. A full unbounded `SELECT` of all non-terminal transactions in the database (no `LIMIT`).
2. One async `userKeysToSign` DB call per row — O(N) sequential round-trips to PostgreSQL.
3. All N transaction objects held in the Node.js heap simultaneously.

As N grows, each request consumes proportionally more memory and CPU. At sufficient scale, the Node.js event loop stalls, response times spike to timeouts, and the API service becomes unavailable to all users — a complete service outage.

### Likelihood Explanation

**Attacker preconditions:** A valid authenticated account (normal user, no admin privileges required). The `getTransactionsToSign` endpoint is a standard product workflow reachable by any verified user.

**Escalation path:** A malicious user can deliberately inflate N by creating large numbers of transactions (each with a future `validStart` so they remain in `WAITING_FOR_SIGNATURES` status). Alternatively, in any active organization deployment, N grows naturally over time. Either way, repeated calls to this endpoint by one or more users will exhaust server resources.

### Recommendation

Apply the database-level `take`/`skip` pagination **before** the loop, not after. Because `userKeysToSign` must be evaluated to determine inclusion, the correct fix is to push the signing-key check into the SQL query (e.g., via a subquery or join against the user's keys and the transaction's required signers), so the database returns only the relevant page of results. If an in-process check is unavoidable, enforce a hard cap on the number of transactions fetched per request and document that `totalItems` is an estimate.

### Proof of Concept

1. Register a normal (non-admin) user account.
2. Create a large number of transactions (e.g., 10,000) with future `validStart` values so they remain in `WAITING_FOR_SIGNATURES` status.
3. Call `GET /transactions/sign?page=1&size=10` (or equivalent endpoint backed by `getTransactionsToSign`).
4. Observe: the server executes one `repo.find()` returning all 10,000 rows, followed by 10,000 sequential `userKeysToSign` DB calls, before returning 10 results.
5. Monitor Node.js heap and response latency — both grow linearly with the number of active transactions, confirming unbounded resource consumption from a single request. [4](#0-3)
