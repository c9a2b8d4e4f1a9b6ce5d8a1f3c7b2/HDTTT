### Title
`removeTransactionApprover` Does Not Re-Validate Parent Threshold After Child Deletion, Enabling Permanent Approval Deadlock

### Summary
When a child approver node is deleted via `removeTransactionApprover`, the parent threshold node's `threshold` value is never re-evaluated against the new (reduced) child count. This mirrors the DN404 `_burn` pattern exactly: a constraint (`threshold ≤ children`) is enforced at creation time but not re-checked after the state-modifying deletion. A transaction creator can exploit this to leave a threshold node in a state where `threshold > remaining_children`, making the transaction permanently unapprovable.

### Finding Description

**Root cause — constraint checked at creation, never after deletion.**

At creation time, `createTransactionApprovers` enforces the invariant:

```
threshold ≤ approvers.length
``` [1](#0-0) 

When a child is detached via `updateTransactionApprover` (setting `listId: null`), the code correctly re-evaluates the parent and adjusts its threshold:

```typescript
const newParentApproversLength = parent.approvers.length - 1;
if (newParentApproversLength === 0) {
    await transactionalEntityManager.softRemove(TransactionApprover, parent);
} else if (newParentApproversLength < parent.threshold) {
    await transactionalEntityManager.update(TransactionApprover, parent.id, {
        threshold: newParentApproversLength,
    });
}
``` [2](#0-1) 

However, `removeTransactionApprover` takes a completely different path. It fetches the approver and calls `removeNode`, which recursively soft-deletes the target and all its descendants — with **no parent threshold adjustment whatsoever**:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(...);
    return result;
}
``` [3](#0-2) 

`removeNode` only issues a recursive SQL soft-delete; it never touches the parent row: [4](#0-3) 

The controller exposes this endpoint to any transaction creator without status-gating: [5](#0-4) 

**Exploit path:**

1. Creator creates a transaction and sets up a threshold approver tree: `threshold=2`, children = `[User A, User B]`.
2. User A signs and approves. The tree now has 1 of 2 approvals.
3. Creator calls `DELETE /transactions/:txId/approvers/:userAId` (User A's leaf node).
4. `removeNode` soft-deletes User A's record. The parent threshold node still has `threshold=2` but now has only 1 child (User B).
5. Even if User B approves, `isApproved` evaluates `approvals.length (1) >= threshold (2)` → `false`.
6. The transaction is permanently stuck — it can never reach the approved state.

The `isApproved` utility confirms the deadlock:

```typescript
if (approvals.length >= (approver.threshold || approvals.length)) {
    return true;
}
``` [6](#0-5) 

### Impact Explanation

The threshold invariant (`threshold ≤ children`) is permanently violated in the database. Any transaction whose approval tree is corrupted this way is frozen in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` indefinitely. Approvers who have already signed cannot recover the transaction. This constitutes **unrecoverable corruption of transaction state** — a high-value impact category per `RESEARCHER.md`.

### Likelihood Explanation

The attacker is a normal registered user (transaction creator) exercising a valid API endpoint. No privileged keys, no leaked credentials, and no external dependencies are required. The endpoint is reachable with a standard authenticated HTTP DELETE request. The only precondition is that the creator has set up a threshold approver tree, which is a documented and common workflow.

### Recommendation

In `removeTransactionApprover`, after soft-deleting the target node, fetch the deleted node's parent (if any) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

- If `newChildCount === 0`, soft-remove the parent.
- If `newChildCount < parent.threshold`, update `parent.threshold = newChildCount`.

Alternatively, gate the deletion: reject the request if removing the node would leave the parent with `threshold > remaining_children`, forcing the creator to lower the threshold first.

### Proof of Concept

**Setup:**
```
POST /transactions/:txId/approvers
{
  "approversArray": [{
    "threshold": 2,
    "approvers": [{ "userId": 10 }, { "userId": 11 }]
  }]
}
```
Parent node `P` is created with `threshold=2`, children `[A(userId=10), B(userId=11)]`.

**Trigger:**
```
DELETE /transactions/:txId/approvers/:A_id
```
`removeNode(A_id)` soft-deletes node A. Parent `P` still has `threshold=2` in the DB.

**Observe:**
```
GET /transactions/:txId/approvers
→ P: { threshold: 2, approvers: [B] }
```
User B approves. `isApproved(P)` → `1 >= 2` → `false`. Transaction never advances. State is permanently corrupted.
