### Title
Any Transaction Creator Can Delete Approvers Belonging to Other Transactions via Missing Cross-Ownership Check

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint in `approvers.controller.ts` verifies that the caller is the creator of the transaction identified by `:transactionId`, but then passes the unrelated `:id` (approver ID) directly to `removeTransactionApprover()` without verifying that the targeted approver actually belongs to that transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system by substituting a foreign approver ID into the URL.

### Finding Description
In `approvers.controller.ts`, the `removeTransactionApprover` handler performs two independent operations:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts  lines 102-113
@Delete('/:id')
async removeTransactionApprover(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Param('id', ParseIntPipe) id: number,
) {
    await this.approversService.getCreatorsTransaction(transactionId, user);
    await this.approversService.removeTransactionApprover(id);
    return true;
}
``` [1](#0-0) 

`getCreatorsTransaction(transactionId, user)` confirms the caller owns `transactionId`, but `removeTransactionApprover(id)` then deletes the approver row identified by `id` with no check that it belongs to `transactionId`:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts  lines 534-543
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(...);
    return result;
}
``` [2](#0-1) 

The authorization check and the deletion target are completely decoupled. The `transactionId` URL segment is used only to gate the caller's identity; the actual deletion is driven solely by the `id` segment, which is never cross-validated against `transactionId`.

Approver IDs are sequential integers (auto-increment primary key), making enumeration trivial.

### Impact Explanation
An attacker who is the creator of any transaction can:
1. Enumerate approver IDs belonging to other transactions (IDs are sequential integers).
2. Call `DELETE /transactions/{own_transactionId}/approvers/{victim_approverId}`.
3. The system confirms the caller owns `own_transactionId` (true) and then deletes `victim_approverId` without further checks.

This allows the attacker to silently dismantle the approval workflow of any transaction in the organization — removing required approvers, reducing threshold trees, or deleting entire approval trees via `removeNode`. A transaction that required multi-party approval can be reduced to requiring no approval at all, enabling unauthorized execution.

### Likelihood Explanation
- Attacker precondition: a valid, verified organization account (no admin privilege required).
- Approver IDs are sequential integers — no brute-force difficulty.
- The endpoint is a standard REST `DELETE` call with no rate-limiting specific to this action.
- The bug is reachable by any normal user in Organization Mode.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the transaction the caller owns:

```typescript
// In approvers.service.ts
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // Cross-check: the approver must belong to the authorized transaction
    const root = await this.getRootNodeFromNode(approver.id);
    if (!root || root.transactionId !== transactionId)
        throw new UnauthorizedException('Approver does not belong to this transaction');

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(...);
    return result;
}
```

Pass `transactionId` from the controller to the service method, and update the controller call accordingly.

### Proof of Concept
**Setup:**
- User A creates Transaction 1 (owns it).
- User B creates Transaction 2 and adds Approver X (approver ID = 42) to it.

**Attack:**
```
DELETE /transactions/1/approvers/42
Authorization: Bearer <User A's JWT>
```

**Expected behavior:** 403 Unauthorized — User A does not own Transaction 2.

**Actual behavior:** 200 OK — `getCreatorsTransaction(1, userA)` passes (User A owns Transaction 1), then `removeTransactionApprover(42)` deletes Approver X from Transaction 2 without any further check. [1](#0-0) [2](#0-1)
