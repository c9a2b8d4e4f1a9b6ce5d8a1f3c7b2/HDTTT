### Title
Cross-Transaction Approver Deletion: Creator of Any Transaction Can Delete Approvers Belonging to Other Transactions

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the authenticated user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that this approver actually belongs to `:transactionId`. Any user who has created at least one transaction can exploit this to delete approvers from transactions they do not own, breaking the multi-signature approval invariant for those transactions.

### Finding Description

**Root cause — implicit cross-module restriction not enforced:**

The controller performs an authorization check against `transactionId` (the URL parameter), but the service deletes the approver by its own `id` with no scope check:

`back-end/apps/api/src/transactions/approvers/approvers.controller.ts` lines 102–113: [1](#0-0) 

```
await this.approversService.getCreatorsTransaction(transactionId, user); // checks user owns transactionId
await this.approversService.removeTransactionApprover(id);               // deletes approver by id — no scope check
```

`back-end/apps/api/src/transactions/approvers/approvers.service.ts` lines 533–544: [2](#0-1) 

`removeTransactionApprover` only checks that the approver record exists; it never verifies that `approver.transactionId` (or the root transaction of a nested approver) matches the `transactionId` supplied in the URL.

The `getCreatorsTransaction` check (lines 624–644) only validates ownership of the URL-supplied `transactionId`, not the approver's actual parent transaction: [3](#0-2) 

**Exploit path:**

1. Attacker (Alice) registers and creates any transaction T_A — she becomes its creator.
2. Victim (Bob) creates transaction T_B and adds approvers (approver IDs are sequential integers, trivially enumerable).
3. Alice sends:
   ```
   DELETE /transactions/{T_A_id}/approvers/{approver_of_T_B_id}
   ```
4. `getCreatorsTransaction(T_A_id, alice)` passes — Alice is the creator of T_A.
5. `removeTransactionApprover(approver_of_T_B_id)` executes — the approver belonging to T_B is soft-deleted with no ownership check.
6. Alice has successfully removed an approver from Bob's transaction.

The `TransactionApprover` entity uses a sequential integer primary key, making approver IDs enumerable: [4](#0-3) 

**Secondary inconsistency (bonus):** For nested approvers (`listId != null`), `approver.transactionId` is `null`. The `emitTransactionStatusUpdate` call at line 541 therefore fires with `entityId: null`, silently failing to notify the correct transaction's stakeholders: [5](#0-4) 

### Impact Explanation

An attacker can delete any approver from any transaction in the system, regardless of ownership. Concrete consequences:

- **Approval threshold bypass**: Remove enough approvers from a threshold tree to make the remaining count fall below the threshold, causing the chain service to treat the transaction as fully approved and advance it to `WAITING_FOR_EXECUTION` without the required signatures.
- **Denial of approval**: Remove all approvers from a transaction, permanently blocking it from reaching `WAITING_FOR_EXECUTION` status.
- **Integrity violation**: The multi-signature orchestration model — the core security guarantee of the system — is broken for any transaction whose approvers can be silently deleted by an unrelated party.

### Likelihood Explanation

- **Attacker precondition**: Must be a registered, verified user who has created at least one transaction. This is the normal user role — no elevated privilege required.
- **Knowledge required**: Approver IDs are sequential integers. An attacker can enumerate them by creating their own transactions and observing the IDs returned, then targeting nearby IDs belonging to other transactions.
- **No rate-limiting or anomaly detection** is visible on this endpoint.
- The attack is a single authenticated HTTP DELETE request.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), resolve the root transaction of the approver being deleted and assert it matches the `transactionId` from the URL:

```typescript
// In the controller or service, after fetching the approver:
const rootNode = await this.approversService.getRootNodeFromNode(approver.id, entityManager);
if (rootNode?.transactionId !== transactionId) {
  throw new UnauthorizedException('Approver does not belong to the specified transaction');
}
```

The `getRootNodeFromNode` helper already exists in the service and is used correctly in `updateTransactionApprover` (lines 386–391): [6](#0-5) 

Apply the same pattern to `removeTransactionApprover`.

### Proof of Concept

**Setup:**
- User Alice registers and creates transaction T_A (ID = 1). She is its creator.
- User Bob creates transaction T_B (ID = 2) and adds an approver (approver record ID = 5, `transactionId = 2`).

**Attack:**
```http
DELETE /transactions/1/approvers/5
Authorization: Bearer <alice_jwt>
```

**Expected (correct) behavior:** 403 Unauthorized — approver 5 does not belong to transaction 1.

**Actual behavior:**
1. `getCreatorsTransaction(1, alice)` → passes (Alice owns T_A).
2. `removeTransactionApprover(5)` → fetches approver 5, calls `removeNode(5)`, soft-deletes it.
3. Returns `true`.

Bob's transaction T_B now has its approver deleted. If T_B required that approver's signature to reach `WAITING_FOR_EXECUTION`, it is now permanently blocked or its threshold is silently auto-reduced (per the `updateTransactionApprover` parent-threshold adjustment logic at lines 417–428): [7](#0-6)

  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
