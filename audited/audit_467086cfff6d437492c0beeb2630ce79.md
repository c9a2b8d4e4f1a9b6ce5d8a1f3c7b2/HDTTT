### Title
Any Authenticated User Can Access Transaction Signers Without Authorization Check

### Summary
The `GET /transactions/:transactionId/signers` endpoint in `SignersController` performs no access verification against the requesting user's relationship to the transaction. Any verified organization member can supply an arbitrary `transactionId` and retrieve the full list of signers, their public keys, and cryptographic signature bytes for transactions they have no business accessing. This is the direct analog of the external whitelist-bypass report: a function that should be restricted to authorized parties (creator, signer, observer, approver) is reachable by any authenticated user.

### Finding Description

**Root cause — missing access control in `getSignaturesByTransactionId`:**

`SignersController` is mounted at `transactions/:transactionId?/signers` and protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. [1](#0-0) 

The `GET /` handler receives the `transactionId` from the URL path and immediately delegates to `signaturesService.getSignaturesByTransactionId(transactionId, true)` — no `@GetUser()` decorator, no access check, no call to `verifyAccess`. [2](#0-1) 

**Contrast with the access control that exists elsewhere:**

Every other sensitive transaction endpoint calls `verifyAccess`, which enforces that the requesting user must be the creator, a signer, an observer, or an approver of the transaction. [3](#0-2) 

The `getTransactionWithVerifiedAccess` wrapper used by `GET /transactions/:id` enforces this check before returning any data. [4](#0-3) 

The signers endpoint has no equivalent gate.

**Exploit path:**

1. Attacker registers as a normal organization user and obtains a valid JWT.
2. Attacker iterates integer `transactionId` values starting from 1: `GET /transactions/1/signers`, `GET /transactions/2/signers`, …
3. For every transaction in the system — including those the attacker has no relationship to — the server returns the full `TransactionSigner[]` array, including user key IDs, DER-encoded public keys, and raw signature bytes.

### Impact Explanation

- **Cross-tenant data exposure**: An attacker learns which users signed which transactions across the entire organization, revealing internal approval workflows and key usage patterns.
- **Cryptographic material leakage**: Raw signature bytes and public keys are returned. While signatures alone do not allow key recovery, they provide material for offline analysis and reduce the attacker's uncertainty about the organization's key infrastructure.
- **Organizational intelligence**: Knowing which users are signers on high-value transactions (e.g., node update or account update transactions) enables targeted social engineering or privilege escalation attempts.

### Likelihood Explanation

- **Preconditions**: Only a valid JWT for any verified user account is required — no admin role, no special privilege.
- **Transaction ID enumeration**: IDs are sequential integers, making full enumeration trivial with a simple loop.
- **No rate-limit specific to this endpoint**: The general throttler guards apply but do not prevent systematic enumeration over time.
- **Realistic attacker**: A malicious insider, a compromised low-privilege account, or a contractor with minimal access can execute this immediately after obtaining credentials.

### Recommendation

Add the same `verifyAccess` gate used by the main transaction endpoint to `getSignaturesByTransactionId`. Inject the current user via `@GetUser()` and call `transactionsService.verifyAccess(transaction, user)` (or an equivalent check in `SignersService`) before returning any signer data. Reject with `UnauthorizedException` if the user has no relationship to the transaction.

```typescript
@Get()
@HttpCode(200)
async getSignaturesByTransactionId(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
): Promise<TransactionSigner[]> {
  await this.signaturesService.verifyUserAccess(transactionId, user); // new gate
  return this.signaturesService.getSignaturesByTransactionId(transactionId, true);
}
```

### Proof of Concept

**Setup**: Two users exist — `alice` (creator of transaction #42) and `bob` (no relationship to transaction #42).

**Steps**:
1. `bob` authenticates: `POST /auth/login` → receives JWT.
2. `bob` calls: `GET /transactions/42/signers` with `Authorization: Bearer <bob_jwt>`.
3. Server returns HTTP 200 with the full `TransactionSigner[]` for transaction #42, including Alice's public key and signature bytes.

**Expected (correct) behavior**: HTTP 401 — `bob` has no relationship to transaction #42.

**Actual behavior**: HTTP 200 — full signer list returned. [2](#0-1)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
