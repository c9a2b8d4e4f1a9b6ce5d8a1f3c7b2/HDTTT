### Title
`uploadKey()` Limit Check Fires Before Existence Check, Blocking Legitimate Key Updates for Users at `MAX_USER_KEYS`

### Summary
The `uploadKey()` function in `UserKeysService` serves a dual purpose: it creates a new `UserKey` record **or** updates an existing one (e.g., to attach a `mnemonicHash`/`index` to a previously uploaded key). However, the `MAX_USER_KEYS` guard is evaluated unconditionally before the code determines whether the operation is a creation or an update. A user who has reached the key limit is therefore blocked from updating any of their existing keys via this endpoint, even though the update does not increase the key count.

### Finding Description

**Root cause — wrong ordering of the limit check:**

In `back-end/apps/api/src/user-keys/user-keys.service.ts`, `uploadKey()` executes as follows:

1. `attachKeys()` loads the user's **non-deleted** keys into `user.keys`.
2. The limit check fires immediately.
3. Only *after* the check does the code query the DB to see whether the submitted `publicKey` already exists. [1](#0-0) 

```
async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
    await attachKeys(user, this.repo.manager);          // loads non-deleted keys

    if (user.keys.length >= MAX_USER_KEYS) {            // ← fires unconditionally
      throw new BadRequestException(ErrorCodes.UMK);
    }

    let userKey = await this.repo.findOne({             // ← existence check is AFTER
      where: { publicKey: dto.publicKey },
      withDeleted: true,
    });
    ...
    if (userKey) {
      Object.assign(userKey, dto);   // UPDATE path – no new key created
    } else {
      userKey = this.repo.create(dto);  // CREATE path – count increases
      userKey.user = user;
    }
```

`attachKeys` loads only active (non-soft-deleted) keys: [2](#0-1) 

So when `user.keys.length === MAX_USER_KEYS` and the caller submits a `publicKey` that already belongs to them (e.g., to attach a `mnemonicHash` that was omitted on first upload), the function throws `ErrorCodes.UMK` at line 37–39 and never reaches the update branch at line 47–56. [3](#0-2) 

The unit tests for the update path always mock `attachKeys` to return `user.keys = []`, masking this defect: [4](#0-3) 

**Concrete trigger scenario:**

1. User uploads 500 keys (reaching `MAX_USER_KEYS`).
2. One or more of those keys was uploaded without a `mnemonicHash`/`index` (a supported workflow, as shown in the e2e test at line 165–178).
3. User calls `POST /user/:id/keys` with the existing `publicKey` plus the missing `mnemonicHash`/`index`.
4. `user.keys.length` = 500 ≥ `MAX_USER_KEYS` → `BadRequestException` is thrown.
5. The user can never attach recovery metadata to that key via this endpoint. [5](#0-4) 

### Impact Explanation

A user at the key limit is permanently unable to update `mnemonicHash` or `index` on any of their existing keys through the `uploadKey` endpoint. Because `mnemonicHash` is the recovery phrase hash used to restore private keys, a user who cannot attach it to an existing key loses the ability to recover that key through the standard organization workflow. This is a **permanent, self-inflicted DoS on key recovery** for users at the limit — a state that cannot be resolved without first deleting a key.

### Likelihood Explanation

`MAX_USER_KEYS` is 500. [6](#0-5) 

The system is designed for organizations managing many Hedera accounts. A power user or automated integration that uploads keys in bulk can reach this limit. The two-step upload pattern (key first, `mnemonicHash` later) is an explicitly supported and tested workflow, making the collision realistic.

### Recommendation

Move the `MAX_USER_KEYS` check to **after** the existence lookup, and apply it only on the creation branch:

```typescript
async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
  await attachKeys(user, this.repo.manager);

  let userKey = await this.repo.findOne({
    where: { publicKey: dto.publicKey },
    withDeleted: true,
  });

  if (userKey) {
    // UPDATE path — no new key is created, limit check not needed
    if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
      throw new BadRequestException(ErrorCodes.PU);
    }
    Object.assign(userKey, dto);
  } else {
    // CREATE path — enforce the limit only here
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }
    userKey = this.repo.create(dto);
    userKey.user = user;
  }

  if (userKey.deletedAt) {
    await this.repo.recover(userKey);
  }
  return this.repo.save(userKey);
}
```

### Proof of Concept

**Preconditions:** Authenticated user with exactly `MAX_USER_KEYS` (500) active keys, at least one of which was uploaded without a `mnemonicHash`.

**Steps:**

1. Upload 500 keys, one without `mnemonicHash`:
   ```
   POST /user/2/keys  { "publicKey": "<KEY_A>" }          → 201
   POST /user/2/keys  { "publicKey": "<KEY_2..500>" }     → 201 (×499 more)
   ```
2. Attempt to attach `mnemonicHash` to `KEY_A`:
   ```
   POST /user/2/keys  { "publicKey": "<KEY_A>", "mnemonicHash": "abc", "index": 0 }
   ```
3. **Expected (correct):** 201 — existing key updated, count unchanged.
4. **Actual (buggy):** 400 — `"User must have up to 500 keys"` — thrown at line 37 before the existence check at line 42. [7](#0-6)
