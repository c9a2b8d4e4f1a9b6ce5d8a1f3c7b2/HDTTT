### Title
Throw-on-Any-Invalid-Status Inside `executeTransactionGroup` Loop Causes Entire Group Execution to Fail (TOCTOU DoS)

### Summary

`executeTransactionGroup` in `back-end/libs/common/src/execute/execute.service.ts` first filters group items using **stale in-memory** status, then re-validates each item against the **live database** inside a loop. If any single transaction's DB status has changed between those two points (e.g., user cancels one transaction while the timeout is pending), the entire group execution throws and all remaining valid transactions are skipped for that cycle. This is the direct TypeScript analog of the Solidity `require`-in-loop pattern from the external report.

### Finding Description

**Root cause — TOCTOU between in-memory filter and DB re-validation inside a loop.**

`collateGroupAndExecute` fetches the `TransactionGroup` from the database and then schedules `addGroupExecutionTimeout` to fire at `validStart + 5 seconds`: [1](#0-0) 

The `transactionGroup` object (with its in-memory statuses) is captured in the closure. When the timeout fires, `executeTransactionGroup` is called:

**Step 1 — filter uses stale in-memory status:** [2](#0-1) 

**Step 2 — loop validates each item with a fresh DB lookup:** [3](#0-2) 

**Step 3 — `validateTransactionStatus` throws for any non-`WAITING_FOR_EXECUTION` DB status:** [4](#0-3) 

**Step 4 — the catch block re-throws, aborting the entire group:** [5](#0-4) 

**Step 5 — `addGroupExecutionTimeout` silently swallows the error and deletes the timeout:** [6](#0-5) 

**Exploit path:**
1. User creates a transaction group (A1, A2, A3), all reach `WAITING_FOR_EXECUTION`.
2. Scheduler runs `collateGroupAndExecute`; group is fetched with all three statuses as `WAITING_FOR_EXECUTION` in memory.
3. User calls `DELETE /transactions/:id` or `PATCH cancel` on A2 — DB status becomes `CANCELED`.
4. The `addGroupExecutionTimeout` fires at `validStart + 5s`; the in-memory filter passes A2 (still `WAITING_FOR_EXECUTION` in memory).
5. `validateTransactionStatus` re-queries DB for A2, finds `CANCELED`, throws.
6. The catch block re-throws; A1 and A3 are never executed in this cycle.
7. The timeout is deleted; the error is only logged.

  addGroupExecutionTimeout(transactionGroup: TransactionGroup) {
    const name = `group_execution_timeout_${transactionGroup.id}`;

    if (this.schedulerRegistry.doesExist('timeout', name)) return;

    const timeToValidStart =
      transactionGroup.groupItems[0].transaction.validStart.getTime() - Date.now();

    const callback = async () => {
      try {
        await this.executeService.executeTransactionGroup(transactionGroup);
      } catch (error) {
        console.log(error);
      } finally {
        this.schedulerRegistry.deleteTimeout(name);
      }
    };

    const timeout = setTimeout(callback, timeToValidStart + 5 * 1_000);
    this.schedulerRegistry.addTimeout(name, timeout);
  }

  private async validateTransactionStatus(transaction: Transaction) {
    const { status } = await this.transactionsRepo.findOne({
      where: { id: transaction.id },
      select: ['status'],
    });

    switch (status) {
      case TransactionStatus.NEW:
        throw new Error('Transaction is new and has not been signed yet.');
      case TransactionStatus.FAILED:
        throw new Error('Transaction has already been executed, but failed.');
      case TransactionStatus.EXECUTED:
        throw new Error('Transaction has already been executed.');
      case TransactionStatus.REJECTED:
        throw new Error('Transaction has already been rejected.');
      case TransactionStatus.EXPIRED:
        throw new Error('Transaction has been expired.');
      case TransactionStatus.CANCELED:
        throw new Error('Transaction has been canceled.');
      case TransactionStatus.ARCHIVED:
        throw new Error('Transaction is archived.');
    }
  }
