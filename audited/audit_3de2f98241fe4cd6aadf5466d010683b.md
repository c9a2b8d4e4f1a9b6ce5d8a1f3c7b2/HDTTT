### Title
TOCTOU Race Condition in `archiveTransaction` Allows Overwriting Terminal Execution State

### Summary
`archiveTransaction` in `transactions.service.ts` reads the transaction status to validate it, then performs an unconditional `UPDATE` that does not re-assert the status in the `WHERE` clause. Between the read and the write, the chain service can execute the transaction on the Hedera network and set the status to `EXECUTED`. The subsequent unconditional update then overwrites `EXECUTED` with `ARCHIVED`, permanently corrupting the database record while the transaction has already settled on-chain.

### Finding Description

The vulnerability class is **TOCTOU (state-transition integrity failure)** — the direct analog of the Checks-Effects-Interactions violation in the reference report. In Solidity the pattern is: check state → call external contract → update state. Here the pattern is: check state → await async I/O → update state without re-asserting the guard.

**Root cause — `archiveTransaction`:** [1](#0-0) 

```
1. READ  → getTransactionForCreator(id, user)          // status = WAITING_FOR_EXECUTION ✓
2. CHECK → status must be WAITING_FOR_SIGNATURES | WAITING_FOR_EXECUTION
3. (async gap — chain service executes tx, writes EXECUTED to DB)
4. WRITE → this.repo.update({ id }, { status: ARCHIVED })  // no WHERE on status
```

The `update` call at line 720 uses only `{ id }` as the filter — it will match the row regardless of what status it currently holds. Every other status-mutating path in the same codebase guards against this with an atomic conditional `WHERE`:

- `cancelTransactionWithOutcome` — `.andWhere('status IN (:...statuses)', ...)` [2](#0-1) 
- `_executeTransaction` — `.where('id = :id AND status = :currentStatus', ...)` [3](#0-2) 
- `collateAndExecute` — `.where('id = :id AND status = :currentStatus', ...)` [4](#0-3) 

`archiveTransaction` is the only status-mutating path that omits this guard. [5](#0-4) 

**Concurrent execution path:**

The chain service schedules execution via `collateAndExecute` → `addExecutionTimeout` → `_executeTransaction`. The execution timeout fires when `validStart` is reached. A user who calls `PATCH /transactions/archive/{id}` in the seconds around `validStart` can win or lose the race:

- If `archiveTransaction` writes `ARCHIVED` first → `_executeTransaction`'s conditional `WHERE status = WAITING_FOR_EXECUTION` finds zero rows, returns `null`, emits no event — but the SDK transaction was already submitted to Hedera. The on-chain effect is real; the DB record says `ARCHIVED`. [6](#0-5) 
- If `_executeTransaction` writes `EXECUTED` first → `archiveTransaction`'s unconditional update overwrites `EXECUTED` with `ARCHIVED`, erasing `executedAt`, `statusCode`, and `receiptBytes` from the record. [5](#0-4) 

### Impact Explanation

**Concrete impact:**

1. **State corruption / loss of execution record** — A transaction that settled on the Hedera ledger is permanently recorded as `ARCHIVED` in the database. `executedAt`, `statusCode`, and receipt data are lost or never written.
2. **Integrity failure** — The system's view of the transaction lifecycle diverges from the actual Hedera network state. Observers, approvers, and auditors see `ARCHIVED` for a transaction that moved real assets or created accounts on-chain.
3. **Unrecoverable** — Once the row is overwritten there is no automated reconciliation path; the system has no mechanism to re-query Hedera and correct the record.

### Likelihood Explanation

**Attacker preconditions (no privilege required):**
- Must be the authenticated creator of the transaction (a normal registered user).
- Must call `PATCH /transactions/archive/{id}` within the execution window around `validStart`.

The window is deterministic: the chain service schedules execution exactly at `validStart - 10 seconds` via `setTimeout`. A creator who knows their own transaction's `validStart` (it is returned in the API response) can time the archive call precisely. No brute-force or guessing is needed.

### Recommendation

Replace the unconditional `repo.update` with a conditional query-builder update that re-asserts the allowed statuses atomically, and verify that at least one row was affected — exactly as `cancelTransactionWithOutcome` does:

```typescript
const archivableStatuses = [
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];

const result = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', { statuses: archivableStatuses })
  // also allow isManual = true if that path is intentional
  .execute();

if (!result.affected || result.affected === 0) {
  throw new BadRequestException(ErrorCodes.OMTIP); // or ConflictException
}
```

This makes the check and the write atomic at the database level, eliminating the TOCTOU window.

### Proof of Concept

**Setup:** A transaction `T` with `id=42` is in `WAITING_FOR_EXECUTION` status. Its `validStart` is `T+30s` from now. The chain service has already scheduled `addExecutionTimeout` for `T+20s`.

**Steps:**

1. At `T+19s`, the creator sends `PATCH /api/transactions/archive/42`.
2. `archiveTransaction` calls `getTransactionForCreator(42, user)` — returns status `WAITING_FOR_EXECUTION`. Check passes.
3. At `T+20s`, the chain service timeout fires; `_executeTransaction` submits the SDK transaction to Hedera. Hedera accepts it. The service then issues:
   ```sql
   UPDATE transaction SET status='EXECUTED', executedAt=NOW(), statusCode=22
   WHERE id=42 AND status='WAITING_FOR_EXECUTION'
   -- 1 row affected
   ```
4. `archiveTransaction` resumes and issues:
   ```sql
   UPDATE transaction SET status='ARCHIVED' WHERE id=42
   -- 1 row affected — overwrites EXECUTED
   ```

**Expected outcome:** DB row for transaction 42 shows `status=ARCHIVED`, `executedAt=NULL`, `statusCode=NULL`. The transaction is settled on Hedera. The system has permanently lost the execution record.
