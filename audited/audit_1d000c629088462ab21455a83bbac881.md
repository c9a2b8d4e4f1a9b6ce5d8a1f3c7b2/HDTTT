### Title
Missing Access Control on Transaction Comment Creation — Any Authenticated User Can Comment on Any Transaction

### Summary
`CommentsController#createComment` allows any authenticated organization member to post a comment on any transaction, regardless of whether they have any relationship to it. The controller itself contains a developer `TODO` acknowledging the missing check. This is the direct analog of the external report's vulnerability class: a write operation that verifies the caller is authenticated but fails to verify the caller has authorization over the specific resource being modified.

---

### Finding Description

In `back-end/apps/api/src/transactions/comments/comments.controller.ts`, the `POST` route accepts a `transactionId` from the URL and delegates to `CommentsService.createComment` with no ownership or access verification:

```typescript
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Body() dto: CreateCommentDto,
  ) {
    return this.commentsService.createComment(user, transactionId, dto);
  }
``` [1](#0-0) 

The service performs no access check either — it simply creates and saves the comment:

```typescript
async createComment(user, transactionId, dto) {
    const comment = this.repo.create(dto);
    comment['transaction'].id = transactionId;
    comment.user = user;
    return this.repo.save(comment);
}
``` [2](#0-1) 

Compare this to every other transaction-related write operation in the codebase, which all enforce ownership. For example, `ObserversService.createTransactionObservers` explicitly checks:

```typescript
if (transaction.creatorKey?.userId !== user.id)
    throw new UnauthorizedException('Only the creator of the transaction is able to delete it');
``` [3](#0-2) 

And `ApproversService.createTransactionApprovers` calls `getCreatorsTransaction` which enforces the same pattern: [4](#0-3) 

The `getCreatorsTransaction` helper throws `UnauthorizedException` if the caller is not the transaction creator: [5](#0-4) 

The comments module has no equivalent guard. Additionally, `getComments` and `getCommentById` also lack access control, allowing any authenticated user to read all comments on any transaction: [6](#0-5) 

---

### Impact Explanation

Any authenticated organization member can:
1. Post comments on transactions they have no relationship to (not creator, signer, observer, or approver), interfering with transaction workflows.
2. Read all comments on any transaction, potentially exposing sensitive deliberation or workflow information.
3. Spam or inject misleading comments to disrupt multi-signature coordination.

---

### Likelihood Explanation

High. The endpoint is reachable by any user who passes `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — all of which only verify authentication, not authorization over the specific transaction. The developer `TODO` comment confirms this is a known, unresolved gap. An attacker needs only a valid organization account and a known transaction ID (which can be enumerated via integer IDs).

---

### Recommendation

Add an access check in `CommentsService.createComment` mirroring the pattern used in `ObserversService` and `ApproversService`:

```typescript
async createComment(user: User, transactionId: number, dto: CreateCommentDto) {
    const transaction = await this.entityManager.findOne(Transaction, {
        where: { id: transactionId },
        relations: ['creatorKey', 'observers', 'signers', 'signers.userKey'],
    });
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    const hasAccess =
        transaction.creatorKey?.userId === user.id ||
        transaction.observers?.some(o => o.userId === user.id) ||
        transaction.signers?.some(s => s.userKey?.userId === user.id);

    if (!hasAccess) throw new UnauthorizedException("You don't have permission to comment on this transaction");

    const comment = this.repo.create(dto);
    comment['transaction'].id = transactionId;
    comment.user = user;
    return this.repo.save(comment);
}
```

Apply the same access check to `getComments` and `getCommentById`.

---

### Proof of Concept

1. Alice creates a transaction. She is the sole creator; Bob has no relationship to it.
2. Bob authenticates and obtains a valid JWT.
3. Bob sends `POST /transactions/{aliceTransactionId}/comments` with a comment body.
4. The server creates the comment with no error — Bob's comment is now attached to Alice's transaction.
5. Bob also calls `GET /transactions/{aliceTransactionId}/comments` and reads all of Alice's transaction comments.

No special privileges are required — only a valid authenticated account in the organization.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
