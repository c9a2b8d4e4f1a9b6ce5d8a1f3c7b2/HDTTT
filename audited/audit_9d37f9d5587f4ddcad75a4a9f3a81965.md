### Title
Any Authenticated User Can Grief Transaction Submission by Pre-Claiming a Victim's `transactionId`

### Summary
The `createTransaction` endpoint in the Hedera Transaction Tool validates that a submitted transaction's `transactionId` is unique across active records, but it does not verify that the submitting user's identity has any relationship to the payer account embedded in the transaction bytes. Any authenticated organization member can craft transaction bytes with an arbitrary payer account ID and valid-start timestamp, sign them with their own registered key, and submit them first — permanently blocking the legitimate payer from submitting their own transaction with that same `transactionId`.

### Finding Description

**Root cause — no ownership check between creator key and transaction payer**

In `createTransactions`, the service extracts the `transactionId` directly from the submitted SDK transaction bytes: [1](#0-0) 

The only identity check performed is that `dto.signature` is a valid signature of `dto.transactionBytes` by the creator's registered public key: [2](#0-1) 

There is **no check** that the payer account embedded in the transaction bytes (`sdkTransaction.transactionId.accountId`) corresponds to any Hedera account owned or controlled by the submitting user.

**Uniqueness enforcement blocks the legitimate payer**

After validation, the service performs a batch uniqueness check: [3](#0-2) 

Transactions in `WAITING_FOR_SIGNATURES`, `WAITING_FOR_EXECUTION`, `EXECUTED`, and `FAILED` statuses all count as "existing." Only `CANCELED`, `REJECTED`, and `ARCHIVED` are excluded. A bad actor's transaction sitting in `WAITING_FOR_SIGNATURES` (its initial status) permanently blocks the victim.

**End-to-end exploit path**

1. In a multi-sig workflow, the transaction creator shares serialized transaction bytes with co-signers for signing. The bytes contain the `transactionId` (`payerAccountId@validStart`).
2. A malicious co-signer (bad actor) extracts the `transactionId` from the shared bytes.
3. The bad actor constructs their own transaction bytes containing the identical `transactionId` (same payer account ID and valid-start timestamp), signs them with their own registered key (`creatorKeyId`), and submits to `POST /transactions` before the legitimate creator.
4. The system stores the bad actor's transaction with that `transactionId` in `WAITING_FOR_SIGNATURES` status.
5. When the legitimate creator submits their transaction, the uniqueness check at lines 415–433 finds the existing record and throws `BadRequestException: Transactions already exist`.

The bad actor's transaction never progresses to execution (they lack the payer's Hedera private key), so it remains in `WAITING_FOR_SIGNATURES` indefinitely, sustaining the block.

### Impact Explanation

The legitimate transaction creator is permanently prevented from submitting their pre-agreed transaction through the tool. In a multi-sig organization workflow, this disrupts coordinated signing flows — the agreed-upon transaction bytes (with a specific `transactionId`) cannot be registered, and all collected signatures become useless. The bad actor can repeat this for every new transaction the victim attempts to create, causing sustained denial of service for that user's transaction submissions. No financial theft occurs, but critical workflow integrity is broken.

### Likelihood Explanation

The attacker must be a registered organization member with a registered key — a realistic insider threat. The attack requires knowing the victim's `transactionId` (payer account + valid-start), which is directly available to any co-signer who receives the transaction bytes for signing. The Hedera account ID of any user is also public on-chain, and the valid-start timestamp is observable from shared bytes. No privileged access, leaked secrets, or cryptographic breaks are required.

### Recommendation

Add a check that the payer account ID embedded in the submitted transaction bytes is associated with a Hedera account registered to the submitting user. Concretely, after parsing the SDK transaction, verify:

```typescript
const payerAccountId = sdkTransaction.transactionId.accountId.toString();
// Ensure the creator's registered Hedera accounts include payerAccountId
const userOwnsPayerAccount = user.keys.some(k => k.hederaAccountId === payerAccountId);
if (!userOwnsPayerAccount) {
  throw new BadRequestException('Transaction payer account does not belong to the submitting user');
}
```

If the data model does not currently store a mapping from `UserKey` to Hedera account ID, that mapping should be introduced. Alternatively, scope the `transactionId` uniqueness constraint per-user rather than globally, so one user's submission cannot block another's.

### Proof of Concept

**Preconditions**: Two registered users, Alice (victim) and Bob (bad actor), both members of the same organization with registered keys.

1. Alice creates a `FileCreateTransaction` with `transactionId = 0.0.1234@1716577920.000000000`, signs it with her key, and shares the bytes with Bob for co-signing.
2. Bob extracts the `transactionId` from Alice's bytes. Bob constructs his own `FileCreateTransaction` with the same `transactionId` (`0.0.1234@1716577920.000000000`), signs it with his own registered key, and submits:
   ```
   POST /transactions
   {
     "transactionBytes": "<Bob's bytes with Alice's transactionId>",
     "creatorKeyId": <Bob's key ID>,
     "signature": "<Bob's valid signature>",
     "mirrorNetwork": "testnet",
     ...
   }
   ```
   Response: `201 Created` — Bob's transaction is stored.
3. Alice submits her original transaction:
   ```
   POST /transactions
   {
     "transactionBytes": "<Alice's original bytes>",
     "creatorKeyId": <Alice's key ID>,
     "signature": "<Alice's valid signature>",
     "mirrorNetwork": "testnet",
     ...
   }
   ```
   Response: `400 Bad Request — Transactions already exist: 0.0.1234@1716577920.000000000`

Alice's transaction is permanently blocked. Bob's transaction sits in `WAITING_FOR_SIGNATURES` indefinitely since Bob cannot provide the payer's (Alice's) Hedera signature for actual network execution. [4](#0-3) [5](#0-4)
