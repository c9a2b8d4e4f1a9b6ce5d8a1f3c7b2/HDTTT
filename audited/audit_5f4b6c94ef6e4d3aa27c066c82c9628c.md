### Title
Cross-Transaction Approver Deletion via Missing Ownership Binding in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can delete approvers from any other transaction in the system by supplying their own `transactionId` and a victim approver's `id`.

### Finding Description

**Vulnerability class:** Authorization bypass — caller-controlled parameter selects the victim object; the ownership check is performed on a different object.

**Root cause — controller (`approvers.controller.ts` lines 102–113):**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no binding check
  return true;
}
``` [1](#0-0) 

**Root cause — service (`approvers.service.ts` lines 533–544):**

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);          // ← no check: approver.transactionId === callerOwnedTransactionId
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

The `getCreatorsTransaction` guard only confirms the caller owns `transactionId` (the URL path parameter). It does not bind the approver `id` to that transaction. The service's `removeTransactionApprover` accepts any approver `id` globally and deletes it unconditionally.

By contrast, `updateTransactionApprover` performs a `getRootNodeFromNode` check that verifies `root?.transactionId === transactionId`, providing the binding that `removeTransactionApprover` lacks. [3](#0-2) 

### Impact Explanation

An attacker who is the creator of any transaction (even a trivial one they created themselves) can delete approvers from any other transaction in the system. Consequences:

- **Approval bypass:** Removing a required approver from a victim's transaction can reduce the approval threshold, allowing the transaction to advance to execution without the intended authorization.
- **Workflow disruption:** Permanently removes approver records (soft-delete), forcing the victim transaction creator to re-add approvers and restart the approval cycle.
- **Integrity failure:** The approval tree — which is the core trust model of the multi-signature workflow — can be silently mutated by an unauthorized party.

### Likelihood Explanation

- **Precondition:** The attacker must be a registered, verified user who has created at least one transaction. This is the normal user role — no elevated privilege is required.
- **Approver ID discovery:** Approver IDs are sequential integers. A user who is an approver or observer on any transaction can enumerate IDs. The `GET /transactions/:transactionId/approvers` endpoint returns approver IDs to any user with access to that transaction.
- **No rate limiting or anomaly detection** is visible on this endpoint.
- The attack is a single authenticated HTTP `DELETE` request.

### Recommendation

In the controller, after `getCreatorsTransaction` succeeds, verify that the approver being deleted belongs to the same transaction before calling `removeTransactionApprover`:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  // Add: verify approver belongs to this transaction
  const root = await this.approversService.getRootNodeFromNode(id);
  if (!root || root.transactionId !== transactionId) {
    throw new BadRequestException('Approver does not belong to this transaction');
  }
  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

Alternatively, add the binding check inside `removeTransactionApprover(id, transactionId)` at the service level, mirroring the pattern already used in `updateTransactionApprover`.

### Proof of Concept

**Setup:**
- Attacker (User A) creates `Transaction_A` and is its creator.
- Victim (User B) creates `Transaction_B` with an approver record having `id = 42`.
- User A is an observer on `Transaction_B` (or can enumerate approver IDs via any shared transaction).

**Attack:**
```http
DELETE /transactions/{Transaction_A_id}/approvers/42
Authorization: Bearer <User_A_JWT>
```

**Expected (broken) behavior:**
1. `getCreatorsTransaction(Transaction_A_id, UserA)` → passes (User A owns Transaction_A).
2. `removeTransactionApprover(42)` → fetches approver 42 (belongs to Transaction_B), soft-deletes it.
3. Returns `true`.

**Result:** Approver 42 is permanently removed from `Transaction_B` without User B's knowledge or consent. If approver 42 was the sole required approver, `Transaction_B` can now proceed to execution without the intended authorization.
