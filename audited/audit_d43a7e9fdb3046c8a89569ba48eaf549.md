### Title
Admin Can Demote Other Admins or Self-Demote via `PATCH /users/:id`, Enabling Privilege Takeover or Permanent Admin Lockout

### Summary
The `PATCH /users/:id` endpoint in `UsersController` allows any authenticated admin to set `admin: false` on any user record — including other admins and themselves — with no guard against self-demotion or last-admin demotion. This is the direct analog of the Escher `grantRole` / `revokeRole` vulnerability: an admin can elevate a second user to admin via `/auth/elevate-admin`, then the new admin can demote the original admin, gaining sole control. Alternatively, a malicious or mistaken admin can self-demote, permanently locking the organization out of all admin-gated functions.

### Finding Description

**Root cause — `updateUser` lacks the self-check that `removeUser` has**

`removeUser` explicitly blocks self-deletion: [1](#0-0) 

`updateUser` has no equivalent guard. The requesting user's identity is never injected (`@GetUser()` is absent), so the service receives only the target `userId` from the URL and the `dto` from the body: [2](#0-1) 

`UpdateUserDto` accepts `admin: boolean` — both `true` and `false` are valid: [3](#0-2) 

`updateUserById` performs no admin-count check before persisting: [4](#0-3) 

**Attack path A — Privilege takeover (analog to Escher M-13)**

1. Admin A legitimately elevates User B to admin via `PATCH /auth/elevate-admin`: [5](#0-4) 
2. User B (now admin) sends `PATCH /users/{admin_A_id}` with body `{ "admin": false }`.
3. Admin A is demoted to a regular user. User B holds sole admin control.

**Attack path B — Permanent admin lockout (DoS)**

1. The sole admin sends `PATCH /users/{own_id}` with `{ "admin": false }`.
2. No admin exists in the organization.
3. All admin-gated endpoints (`POST /auth/signup`, `DELETE /users/:id`, `PATCH /users/:id`, `PATCH /auth/elevate-admin`) are permanently inaccessible — all require `AdminGuard`: [6](#0-5) 

### Impact Explanation

- **Privilege takeover**: A newly elevated admin can immediately strip the original admin of their role, gaining exclusive control over user management (add, remove, promote users), with no recourse for the demoted admin.
- **Permanent organizational lockout**: If the last admin self-demotes, no user can ever regain admin status through the API. All admin-gated operations are permanently frozen. Recovery requires direct database intervention.

Both outcomes constitute unauthorized state changes and/or permanent lock of organizational state.

### Likelihood Explanation

- Attack path A requires the attacker to first be elevated to admin by an existing admin — a realistic scenario (e.g., a newly onboarded admin who turns malicious, or a social-engineering scenario where a user convinces an admin to elevate them).
- Attack path B requires only that the attacker already holds admin status, which is a normal operational state.
- The API call is a single authenticated `PATCH` request with a trivial body. No special tooling is needed.
- The `removeUser` endpoint already has a self-harm guard, proving the developers considered this class of issue — making the omission in `updateUser` a concrete oversight rather than a design choice.

### Recommendation

1. **Block self-demotion**: In `updateUser`, inject `@GetUser()` and reject requests where `userId === requestingUser.id` and `dto.admin === false`, mirroring the self-deletion guard in `removeUser`.
2. **Guard against last-admin demotion**: Before persisting `admin: false`, count remaining admins. If the target is the last admin, throw a `BadRequestException`.

```typescript
// users.controller.ts
@UseGuards(AdminGuard)
@Patch('/:id')
@Serialize(UserDto)
updateUser(
  @GetUser() requestingUser: User,
  @Param('id', ParseIntPipe) userId: number,
  @Body() dto: UpdateUserDto,
): Promise<User> {
  if (requestingUser.id === userId && dto.admin === false) {
    throw new BadRequestException('Cannot demote yourself');
  }
  return this.usersService.updateUserById(userId, dto);
}
```

And in `updateUserById` (or a dedicated service method), verify at least one admin will remain after the update.

### Proof of Concept

**Attack path A — Privilege takeover**

```
# Step 1: Admin A elevates User B
PATCH /auth/elevate-admin
Authorization: Bearer <admin_A_token>
{ "id": <user_B_id> }
→ 200 OK

# Step 2: User B demotes Admin A
PATCH /users/<admin_A_id>
Authorization: Bearer <user_B_token>
{ "admin": false }
→ 200 OK  (no error — Admin A is now a regular user)

# Step 3: Verify Admin A is locked out
PATCH /users/<any_id>
Authorization: Bearer <admin_A_token>
→ 403 Forbidden
```

**Attack path B — Self-demotion lockout**

```
# Sole admin self-demotes
PATCH /users/<admin_own_id>
Authorization: Bearer <admin_token>
{ "admin": false }
→ 200 OK  (organization now has zero admins)

# All admin operations are now permanently inaccessible
POST /auth/signup
Authorization: Bearer <any_token>
→ 403 Forbidden (AdminGuard blocks all users)
```
