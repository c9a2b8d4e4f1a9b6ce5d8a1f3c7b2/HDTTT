### Title
Cross-Transaction Authorization Bypass in `removeTransactionApprover` Allows Any Creator to Delete Approvers of Other Transactions

### Summary

The `removeTransactionApprover` function in the back-end API service removes a `TransactionApprover` record identified only by its own `id`, without verifying that the approver belongs to the transaction the caller is authorized to modify. The controller verifies the caller is the creator of the URL-supplied `transactionId`, but the service then deletes whichever approver matches the separate `id` parameter — which can belong to a completely different transaction. Additionally, when the deleted approver is a child node (with `transactionId = null`), the status-update notification is emitted with `entityId: null`, silently dropping the re-evaluation of the affected transaction's state.

### Finding Description

**Entry point** — `DELETE /transactions/:transactionId/approvers/:id` [1](#0-0) 

The controller performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the transaction identified by the URL's `:transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver identified by `:id`. [2](#0-1) 

Inside `removeTransactionApprover`, the approver is fetched by its own primary key with no ownership check: [3](#0-2) 

There is no assertion that `approver.transactionId === transactionId` (or that the approver's root node belongs to `transactionId`). The two parameters are completely decoupled.

**Secondary state-tracking defect** — The `TransactionApprover` data model stores `transactionId = null` on every child node (non-root node in an approval tree); only root nodes carry the real transaction FK. [4](#0-3) 

When a child approver is the target of deletion, `approver.transactionId` is `null`, so the status-update event is emitted as `{ entityId: null }`: [5](#0-4) 

The notification receiver skips any event whose transaction cannot be found, meaning the affected transaction's status is never re-evaluated after the removal: [6](#0-5) 

The commented-out line in the controller (`// await this.approversService.emitSyncIndicators(transactionId);`) is evidence that this gap was previously recognized but never resolved: [7](#0-6) 

### Impact Explanation

**Authorization bypass (primary):** Any authenticated user who has created at least one transaction can delete approvers belonging to any other transaction in the system. This allows:

- Removal of required approvers from a victim transaction, bypassing the intended multi-signature approval gate.
- Forcing a transaction that previously required N-of-M approvals to proceed with fewer approvers than intended, or to become permanently stuck if the approval tree is corrupted.
- Permanent, irreversible corruption of another user's transaction approval state (soft-delete is used, but the approval workflow cannot be restored without creator intervention).

**Incomplete state tracking (secondary):** When a child approver is removed, the parent transaction's status is never re-evaluated via the event pipeline. The transaction may remain stuck in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` even if the removal changed the approval threshold in a way that should advance the state. The periodic scheduler (`updateTransactions`) will eventually catch up, but the immediate notification and status transition are silently lost. [8](#0-7) 

### Likelihood Explanation

- **Precondition:** The attacker must be a registered, verified user who has created at least one transaction (to pass the `getCreatorsTransaction` check). This is a normal user action with no elevated privileges.
- **Approver ID enumeration:** `TransactionApprover.id` is a sequential integer primary key. An attacker can enumerate IDs trivially by observing their own approver IDs and probing adjacent values.
- **No rate limiting or anomaly detection** is visible on this endpoint.
- The attack requires only a single authenticated HTTP `DELETE` request.

### Recommendation

1. **In `removeTransactionApprover`**, resolve the root transaction of the approver being deleted and assert it matches the `transactionId` the caller is authorized to modify:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Walk up to the root and verify ownership
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root || root.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  await this.removeNode(approver.id);

  // Always emit with the authoritative transactionId, not approver.transactionId
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
}
```

2. **In the controller**, pass `transactionId` to `removeTransactionApprover` so the service can perform the ownership check and emit the correct entity ID.

3. **Remove the commented-out `emitSyncIndicators` line** and replace it with the corrected `emitTransactionStatusUpdate` call using the URL's `transactionId`.

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (id=1) with no approvers.
- User B creates Transaction 2 (id=2) and adds an approver (User C, approver id=7, root node with `transactionId=2`).

**Attack:**
```
DELETE /transactions/1/approvers/7
Authorization: Bearer <User A's JWT>
```

**Step-by-step:**
1. `getCreatorsTransaction(1, userA)` → passes (User A is creator of Transaction 1).
2. `removeTransactionApprover(7)` → fetches approver id=7 (belongs to Transaction 2, `transactionId=2`).
3. `removeNode(7)` → soft-deletes approver 7 and all its children.
4. `emitTransactionStatusUpdate(..., [{ entityId: 2 }])` → Transaction 2's status is re-evaluated without its required approver.
5. Transaction 2 now has no approvers; if signatures are already sufficient, it may advance to `WAITING_FOR_EXECUTION` without User C's approval.

**For the child-node / null-entityId variant:**
- Replace approver id=7 with a child approver id (e.g., id=8, `transactionId=null`, `listId=7`).
- Step 4 becomes `emitTransactionStatusUpdate(..., [{ entityId: null }])` → notification silently dropped, Transaction 2's status is never re-evaluated. [2](#0-1) [1](#0-0)

  getTransactionApproverById(
    id: number,

    const find: FindOneOptions<TransactionApprover> = {
      relations: ['approvers'],
      where: { id },
    };

    if (entityManager) {
      return entityManager.findOne(TransactionApprover, find);
    }

    return this.repo.findOne(find);
  }
