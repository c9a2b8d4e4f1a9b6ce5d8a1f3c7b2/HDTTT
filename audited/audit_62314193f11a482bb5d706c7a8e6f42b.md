### Title
Unbounded User-Controlled Batch Size in Signature Upload Endpoints Enables Authenticated DoS

---

### Summary

The `POST /transactions/:id/signers` and `POST /transactions/signatures/import` endpoints accept an unbounded array of `UploadSignatureMapDto` objects, and the `IsSignatureMap` decorator iterates over an unbounded number of entries within each signature map — all without any count or size limit. This is the direct analog of the external report's missing `sz` bounds check: a user-controlled size/count parameter is consumed without validation before a data-processing operation. Any authenticated user can exploit this to exhaust server CPU and memory.

---

### Finding Description

**Root cause 1 — Unbounded array at the controller layer**

Both signature endpoints accept `UploadSignatureMapDto | UploadSignatureMapDto[]` with no maximum array length enforced:

`back-end/apps/api/src/transactions/signers/signers.controller.ts` lines 102–107:
```typescript
async uploadSignatureMap(
  @Body() body: UploadSignatureMapDto | UploadSignatureMapDto[],
  ...
``` [1](#0-0) 

`back-end/apps/api/src/transactions/transactions.controller.ts` lines 96–106:
```typescript
async importSignatures(
  @Body() body: UploadSignatureMapDto[] | UploadSignatureMapDto,
  ...
``` [2](#0-1) 

**Root cause 2 — Unbounded iteration inside `IsSignatureMap` decorator**

For every item in the array, the `IsSignatureMap` decorator iterates over all `nodeAccountId → transactionId → publicKey` entries in the user-supplied object with no entry-count limit. For each entry it calls `PublicKey.fromString()`, `AccountId.fromString()`, `TransactionId.fromString()`, and `decode()` — all CPU-intensive operations: [3](#0-2) 

There is a check that `decodedSignature.length === 0` (line 51) but **no upper-bound check** on the decoded signature size or on the total number of entries.

**Root cause 3 — Developer-acknowledged missing limit**

The service itself contains a comment that explicitly acknowledges the missing API-side limit: [4](#0-3) 

```typescript
//Added a batch mechanism, probably should limit this on the api side of things
const BATCH_SIZE = 500;
```

The `BATCH_SIZE = 500` only controls the database write batch, not the number of items accepted from the caller.

---

### Impact Explanation

An authenticated user sends a single POST request to `/transactions/signatures/import` (or `/transactions/:id/signers`) with:
- An array of N `UploadSignatureMapDto` objects (no limit enforced)
- Each containing a signature map with M entries (no limit enforced)

The server performs O(N × M) cryptographic parsing operations (`PublicKey.fromString`, `AccountId.fromString`, `TransactionId.fromString`, `decode`) synchronously during DTO validation, before any business-logic guard can reject the request. This exhausts CPU and heap memory on the API service, causing denial of service for all other users.

---

### Likelihood Explanation

The endpoints are protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — i.e., any verified, logged-in user can reach them. No admin role is required. The attack requires a single HTTP request with a crafted JSON body and is trivially repeatable.

---

### Recommendation

1. **Enforce a maximum array length** at the controller level before calling `transformAndValidateDto`, e.g.:
   ```typescript
   if (Array.isArray(body) && body.length > 100) {
     throw new BadRequestException('Too many items in batch');
   }
   ```
2. **Enforce a maximum entry count** inside `IsSignatureMap` — reject any signature map whose total `nodeAccountId × transactionId × publicKey` entry count exceeds a safe threshold (e.g., 50).
3. **Add an upper-bound check** on `decodedSignature.length` (e.g., reject signatures longer than 128 bytes) to prevent oversized allocation per entry.

---

### Proof of Concept

```http
POST /transactions/signatures/import HTTP/1.1
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  {
    "id": 1,
    "signatureMap": {
      "0.0.3": {
        "0.0.1@1700000000.000000000": {
          "<64-char-pubkey-1>": "aabb...aabb",
          "<64-char-pubkey-2>": "aabb...aabb",
          ... // repeat M times
        }
      },
      "0.0.4": { ... },
      ... // repeat across N node IDs
    }
  },
  ... // repeat the outer object N times
]
```

Each request triggers O(N × M) calls to `PublicKey.fromString`, `AccountId.fromString`, `TransactionId.fromString`, and `decode` inside the `IsSignatureMap` decorator — with no server-side limit on N or M — exhausting CPU and memory on the API process.
