### Title
Creator Permanently Locked Out of Transaction Management After `creatorKey` Soft-Deletion

### Summary
When a user creates a transaction, their specific `UserKey` is stored as `creatorKey` on the `Transaction` entity. All creator-only operations (execute, manage approvers, etc.) verify identity by checking `transaction.creatorKey?.userId === user.id`. However, `getTransactionById` loads the `creatorKey` relation **without** `withDeleted: true`, so if the `creatorKey` is soft-deleted, TypeORM returns `null` for that relation. The creator is then permanently locked out of all creator-only operations on their active transactions, with no recovery path.

### Finding Description

**Root cause — `getTransactionById` does not load soft-deleted `creatorKey`:** [1](#0-0) 

The `relations` array includes `'creatorKey'` and `'creatorKey.user'` but there is no `withDeleted: true` option. TypeORM's soft-delete mechanism means that once a `UserKey` has its `deletedAt` set, it is excluded from relation loading by default.

**`UserKey` is soft-deletable with no guard against active `creatorKey` usage:** [2](#0-1) [3](#0-2) 

`removeUserKey` performs no check for whether the key is currently referenced as a `creatorKey` on any active transaction.

**Creator-only operations fail when `creatorKey` is null:** [4](#0-3) 

When `transaction.creatorKey` is `null` (soft-deleted), `transaction.creatorKey?.userId` evaluates to `undefined`, which is never equal to `user.id`, so `UnauthorizedException` is always thrown.

The same pattern exists in the approvers service: [5](#0-4) 

**Trigger paths for key deletion:**

1. User self-deletes their own key via `DELETE /users/:userId/keys/:keyId`
2. Admin removes a user from the organization, which soft-deletes **all** their keys: [6](#0-5) 

The codebase itself acknowledges this behavior in a comment: [7](#0-6) 

### Impact Explanation

Once the `creatorKey` is soft-deleted, the creator permanently loses the ability to:
- Execute manual transactions (`PATCH /transactions/execute/:id`)
- Manage approvers on their transactions (add/remove/update)
- Any other creator-gated operation

The transaction enters a permanently unmanageable state from the creator's perspective. There is no API endpoint to reassign a `creatorKey` or recover creator access. This is a **permanent lock** on active transactions.

### Likelihood Explanation

This is triggered by normal product flows available to any authenticated user:
- A user rotating or cleaning up their keys (a routine operation) will lock themselves out of all transactions created with the deleted key.
- An admin removing a departed team member from the organization will soft-delete all that user's keys, permanently locking all their active transactions.

No privileged attacker capability is required — a regular user can trigger this on themselves, and an admin can trigger it on any user.

### Recommendation

1. **Guard key deletion against active `creatorKey` usage**: Before soft-deleting a `UserKey`, check whether it is referenced as `creatorKey` on any non-terminal transaction and reject the deletion (or require the transaction to be canceled first).
2. **Load `creatorKey` with `withDeleted: true`**: In `getTransactionById`, add `withDeleted: true` to the relation load so that the `userId` is still accessible even after soft-deletion, preserving creator identity for authorization checks.
3. **Allow `creatorKey` reassignment**: Provide an admin-only endpoint to reassign the `creatorKey` on a transaction to another key belonging to the same user, enabling recovery.

### Proof of Concept

1. User A creates a transaction using `UserKey` with `id=5`. The transaction's `creatorKeyId` is set to `5`.
2. User A calls `DELETE /users/2/keys/5` — the key is soft-deleted (`deletedAt` is set). No error is returned.
3. User A calls `PATCH /transactions/execute/{transactionId}`.
4. `executeTransaction` calls `getTransactionForCreator`, which calls `getTransactionById`.
5. `getTransactionById` loads the transaction with `relations: ['creatorKey', ...]` — TypeORM returns `null` for `creatorKey` because it is soft-deleted.
6. The check `transaction.creatorKey?.userId !== user.id` evaluates to `undefined !== 2` → `true`.
7. `UnauthorizedException: 'Only the creator has access to this transaction'` is thrown.
8. The transaction is now permanently stuck — it cannot be executed, and approvers cannot be modified by the creator.
