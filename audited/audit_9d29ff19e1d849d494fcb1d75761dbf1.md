### Title
TOCTOU Race in `archiveTransaction` Allows Creator to Overwrite `EXECUTED` Status with `ARCHIVED`, Corrupting Audit Trail

### Summary
`archiveTransaction` in `TransactionsService` reads the transaction status, validates it, then performs an unconditional `UPDATE` with no `WHERE status IN (...)` guard. A concurrent scheduler execution that transitions the transaction to `EXECUTED` between the read and the write is silently overwritten with `ARCHIVED`. The creator of a transaction can exploit this timing window to make an on-chain-executed transaction appear as merely archived, corrupting the audit trail for all participants.

### Finding Description

**Root cause — non-atomic state transition in `archiveTransaction`:** [1](#0-0) 

The function reads the current status at line 709, validates it at lines 711–718, then unconditionally writes `ARCHIVED` at line 720 with no `WHERE status = <expected>` predicate. There is no optimistic-lock clause preventing the update from landing on a row whose status has already changed.

**Contrast with the correctly guarded `cancelTransaction`:** [2](#0-1) 

`cancelTransaction` appends `.andWhere('status IN (:...statuses)', ...)` so the UPDATE is a no-op if the status changed between read and write. `archiveTransaction` omits this guard entirely.

**Exploit path:**

1. Creator submits a transaction; it reaches `WAITING_FOR_EXECUTION` (all required signatures collected).
2. Creator sends `PATCH /transactions/:id/archive` — the service reads status `WAITING_FOR_EXECUTION`, passes the guard.
3. Concurrently, the chain scheduler's `updateTransactions` / `executeTransaction` path sets status to `EXECUTED` in the database.
4. `archiveTransaction` then executes `repo.update({ id }, { status: ARCHIVED })` unconditionally, overwriting `EXECUTED`.

The scheduler runs on a fixed interval and processes transactions whose `validStart` is approaching: [3](#0-2) 

Because `validStart` is visible to the creator, the timing window is predictable and the race can be attempted repeatedly at negligible cost.

**State machine context — no central transition table:**

The `TransactionStatus` enum defines eight states but there is no single authoritative transition table or guard modifier: [4](#0-3) 

Each service method independently re-implements its own ad-hoc status checks, exactly the pattern flagged in the external report. `archiveTransaction` is the one place where the check is present but the write is not guarded.

### Impact Explanation

- A transaction that was successfully executed on the Hedera network (real-world effect already occurred) is recorded as `ARCHIVED` in the database.
- All participants (signers, observers, approvers) see the wrong terminal state; notifications fire for `ARCHIVED` instead of `EXECUTED`. [5](#0-4) 

- The audit trail is permanently corrupted: `ARCHIVED` has no `statusCode` (Hedera response code), so the execution result is lost.
- The creator can use this to conceal that a transaction was executed, misleading co-signers and observers about the outcome of a multi-party workflow.

### Likelihood Explanation

- **Attacker profile:** any authenticated user who is the creator of a transaction — no privileged access required.
- **Precondition:** the transaction must be in `WAITING_FOR_EXECUTION`, which is a normal, reachable state in every multi-signature workflow.
- **Timing:** `validStart` is stored in the database and returned to the creator, making the scheduler's execution window predictable. The creator can issue the archive request immediately before `validStart` to maximize the race probability.
- **Repeatability:** the request can be retried at zero cost until the race is won.

### Recommendation

Add a conditional `WHERE status IN (WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION)` predicate to the `archiveTransaction` UPDATE, mirroring the pattern already used in `cancelTransaction`:

```typescript
const result = await this.repo
  .createQueryBuilder()
  .update(Transaction)
  .set({ status: TransactionStatus.ARCHIVED })
  .where('id = :id', { id })
  .andWhere('status IN (:...statuses)', {
    statuses: [
      TransactionStatus.WAITING_FOR_SIGNATURES,
      TransactionStatus.WAITING_FOR_EXECUTION,
    ],
  })
  .execute();

if (!result.affected || result.affected === 0) {
  // Status changed concurrently; re-read and throw appropriate error
  throw new ConflictException('Transaction state changed; cannot archive.');
}
```

More broadly, define a single authoritative transition table (the finite-state-machine pattern recommended in the external report) and enforce every status write through it, so no individual service method can accidentally omit the guard.

### Proof of Concept

1. Authenticate as a user who owns a transaction currently in `WAITING_FOR_EXECUTION`.
2. Observe the transaction's `validStart` timestamp (returned in the transaction payload).
3. At `validStart - ε` (e.g., 200 ms before), send:
   ```
   PATCH /transactions/{id}/archive
   ```
4. The scheduler fires at `validStart`, calls `executeTransaction`, and sets status to `EXECUTED`.
5. If the archive request's `repo.update` executes after the scheduler's write, the database row now shows `status = ARCHIVED`.
6. Query `GET /transactions/{id}` — status is `ARCHIVED`; no `statusCode` is present, and the execution is invisible to all other participants.
