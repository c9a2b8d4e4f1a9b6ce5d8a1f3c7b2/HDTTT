### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Authenticated DoS

### Summary
`TransactionsService.getTransactionsToSign()` fetches every non-terminal transaction in the database without any row limit, then performs a per-row async `userKeysToSign()` call (which deserializes transaction bytes and queries the DB) before applying pagination. Any authenticated user can repeatedly call this endpoint to exhaust server CPU, memory, and DB connections, degrading service for all users.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts` at lines 295–309, the function `getTransactionsToSign` issues an unbounded `repo.find()` with no `take` constraint:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no take / skip
});

for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
``` [1](#0-0) 

Pagination is applied only **after** the full iteration:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [2](#0-1) 

Each iteration calls `userKeysToSign`, which in turn calls `keysRequiredToSign`. That function:
1. Deserializes the full transaction bytes via `SDKTransaction.fromBytes()`
2. Calls `transactionSignatureService.computeSignatureKey()` (may involve mirror node I/O)
3. Issues a DB query to fetch matching `UserKey` rows [3](#0-2) 

The `whereForUser` filter excludes only terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). All `WAITING_FOR_SIGNATURES` and `WAITING_FOR_EXECUTION` transactions — across **all users** in the organization — are loaded into memory and processed per request. [4](#0-3) 

The endpoint is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — any verified organization member can reach it. [5](#0-4) 

A second related surface exists in `POST /transactions/:transactionId?/signers` (`uploadSignatureMap`), which accepts an unbounded array of `UploadSignatureMapDto` objects. The DTO has no `@ArrayMaxSize` decorator and no server-side count cap, so a single request can force the server to process thousands of signature maps in parallel via `Promise.all`. [6](#0-5) [7](#0-6) 

### Impact Explanation
- **Service degradation / DoS**: As the number of active (non-terminal) transactions grows, each call to `GET /transactions/sign` forces a full table scan plus N async DB+CPU operations. Repeated calls from one or more authenticated users can saturate the Node.js event loop, exhaust the PostgreSQL connection pool, and make the API unresponsive for all users.
- **Memory exhaustion**: All matching transaction rows (including `transactionBytes` blobs) are loaded into the API process heap simultaneously with no bound.
- Severity: **High** — permanent service unavailability is achievable under realistic load without any privileged access.

### Likelihood Explanation
- Precondition: a valid organization account (any verified user). No admin or special role required.
- The attack is trivially repeatable: a script calling `GET /transactions/sign` in a tight loop is sufficient.
- The severity grows organically: the more transactions the organization accumulates, the worse each individual call becomes, even without an active attacker.

### Recommendation
1. **Apply the DB-level limit before iteration**: pass `take: limit` and `skip: offset` to `repo.find()` and restructure the query so the signer-key check is pushed into the database (e.g., a subquery or a pre-filtered join), eliminating the in-process loop entirely.
2. **Cap the `uploadSignatureMap` array**: add `@ArrayMaxSize(N)` to the DTO or validate array length in the controller before delegating to the service.
3. **Add rate limiting** on these endpoints (the infrastructure already uses Redis for rate limiting elsewhere).

### Proof of Concept

```
# Attacker: any verified organization user
# Step 1 – create many transactions (or rely on existing ones)
# Step 2 – hammer the endpoint
while true; do
  curl -s -H "Authorization: Bearer $TOKEN" \
    "https://<api>/transactions/sign?page=1&size=10" > /dev/null
done
```

With N active transactions in the DB, each request triggers N `SDKTransaction.fromBytes()` deserializations and N DB queries inside `keysRequiredToSign`. At a few hundred transactions the API response time degrades noticeably; at a few thousand it becomes unresponsive. The attacker needs only a single valid JWT and the ability to send HTTP requests. [8](#0-7)

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
```
