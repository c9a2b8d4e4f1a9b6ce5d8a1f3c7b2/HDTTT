### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Server-Side Resource Exhaustion (DoS)

### Summary

`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **every non-terminal transaction in the entire database** without any SQL-level `LIMIT`, then iterates over all of them in application memory — calling an async cryptographic operation (`userKeysToSign`) per row — before applying pagination via an in-memory slice. Any authenticated user can trigger this endpoint. As the transaction table grows, a single request causes unbounded memory allocation, CPU consumption, and async I/O, eventually timing out or crashing the Node.js process.

### Finding Description

**Root cause — missing `take` in the DB query:** [1](#0-0) 

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` / `skip`
});
```

The `whereForUser` predicate filters only on `status NOT IN (terminal statuses)`. It does **not** scope results to the requesting user's transactions. Every active transaction in the system is returned.

**Per-row async cryptographic work:** [2](#0-1) 

For each row, `userKeysToSign` is awaited. Internally this calls `transactionSignatureService.computeSignatureKey`, which deserializes Hedera SDK transaction bytes and may perform mirror-node lookups: [3](#0-2) [4](#0-3) 

Pagination is applied only **after** all rows are processed: [5](#0-4) 

**Contrast with every other method in the same file**, which correctly passes `take: limit` to the DB query: [6](#0-5) [7](#0-6) 

**Controller endpoint** — reachable by any authenticated user: [8](#0-7) 

The `PaginationParams` decorator enforces `size ≤ 100` for the response slice, but this cap is never applied to the underlying DB query inside `getTransactionsToSign`: [9](#0-8) 

### Impact Explanation

A single HTTP request to `GET /transactions/sign?page=1&size=10` causes the server to:
1. Load **all** active transactions (potentially tens of thousands of rows with `transactionBytes` blobs) into Node.js heap.
2. Await one async cryptographic deserialization + signature-key computation per row, sequentially.

This produces unbounded memory growth and CPU saturation proportional to the total number of active transactions in the system. At sufficient scale the API process times out or OOMs, making the signing endpoint — and potentially the entire API service — unavailable to all users.

### Likelihood Explanation

- Any registered user can create transactions; there is no apparent per-user or global cap on active transaction count.
- A single malicious (or even unintentional) accumulation of active transactions triggers the condition on the next call to this endpoint.
- The attacker needs only a valid session — no admin privileges, no leaked credentials.
- The endpoint is part of the normal signing workflow and is polled by the front-end, so it is called frequently.

### Recommendation

Apply DB-level pagination inside `getTransactionsToSign` the same way every other method in the file does. Because the user-key filtering must happen in application code (it requires SDK deserialization), a two-phase approach is needed:

1. Add a configurable hard cap (e.g., `take: MAX_ACTIVE_TX_SCAN`) to the initial `repo.find` call to bound the worst-case load.
2. Alternatively, push the user-key matching into SQL (join against `transaction_signer` keyed on the user's public keys) so the DB can filter and paginate before rows reach the application layer, matching the pattern used by `getTransactionsToApprove`.

### Proof of Concept

**Preconditions:** Two accounts — `attacker` (creates transactions) and `victim` (any user with at least one registered key).

1. `attacker` authenticates and creates N (e.g., 10 000) transactions in `WAITING_FOR_SIGNATURES` status via `POST /transactions`.
2. `victim` (or `attacker` themselves) sends:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <victim_token>
   ```
3. The server executes `repo.find({ where: { status: NOT IN [...terminal] } })` — returning all N rows — then awaits `userKeysToSign` for each.
4. **Expected result:** request times out or the Node.js process exhausts available heap, causing a 500 or process crash, denying service to all users of the API. [10](#0-9)
