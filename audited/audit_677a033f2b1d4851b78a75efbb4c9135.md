### Title
`cutoffAt` Signing Deadline Is Defined But Never Enforced — Signers Can Submit Signatures After the Cutoff Time

### Summary
The `Transaction` entity defines a `cutoffAt` field documented as "the timestamp at which the transaction can no longer be signed by the signers." However, neither of the two server-side signature submission paths — `SignersService.uploadSignatureMaps()` nor `TransactionsService.importSignatures()` — checks whether the current time has passed `cutoffAt` before accepting a signature. Any authenticated signer can submit signatures after the creator-defined deadline has expired.

### Finding Description

**Root cause:** The `cutoffAt` field is stored on the `Transaction` entity and surfaced in the API response, but the enforcement check is entirely absent from both signing code paths.

**Path 1 — `POST /transactions/:transactionId/signers`:**

`SignersController.uploadSignatureMap()` delegates to `SignersService.uploadSignatureMaps()`. [1](#0-0) 

Inside `uploadSignatureMaps()`, per-transaction validation is performed by `validateTransactionStatus()`: [2](#0-1) 

This method checks only two things: the transaction's status enum and whether the Hedera SDK transaction bytes are expired. It never reads `transaction.cutoffAt`.

**Path 2 — `importSignatures` in `TransactionsService`:** [3](#0-2) 

Same pattern: status check + `isExpired()` check, no `cutoffAt` check.

**The field is defined and documented:** [4](#0-3) [5](#0-4) 

The database documentation explicitly states `cutoffAt` is "The timestamp at which the transaction can no longer be signed by the signers." The field is accepted in `CreateTransactionDto`, persisted, and returned in API responses — but the server never enforces it. [6](#0-5) 

### Impact Explanation

A transaction creator sets `cutoffAt` to define a signing window — for example, to ensure that only signatures collected before a governance deadline are counted. After `cutoffAt` passes, any signer who still has access to the transaction (i.e., it has not yet expired per the Hedera SDK's 180-second validity window, or is a long-lived transaction) can submit a valid signature that will be accepted and persisted. This violates the creator's explicit access-control intent: signatures submitted after the cutoff are incorporated into `transactionBytes`, counted toward the required threshold, and can cause the transaction to advance to `WAITING_FOR_EXECUTION` — all after the deadline the creator set.

### Likelihood Explanation

The attack requires no privilege escalation. Any authenticated user who is a designated signer on a transaction can exploit this by simply calling `POST /transactions/:id/signers` after `cutoffAt` has passed. The transaction must still be in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status and not yet Hedera-expired. This is a realistic window: `cutoffAt` is typically set well before `validStart`, meaning the Hedera expiry has not yet triggered. The attacker needs only a valid JWT and a key registered on the transaction.

### Recommendation

Add a `cutoffAt` check inside `validateTransactionStatus()` in `signers.service.ts` and the equivalent block in `importSignatures()` in `transactions.service.ts`:

```typescript
// In validateTransactionStatus():
if (transaction.cutoffAt && new Date() > transaction.cutoffAt) {
  return ErrorCodes.TC; // Transaction Cutoff (new error code)
}
```

Apply the same guard in `importSignatures()` immediately after the status check. This mirrors the pattern already used for `isExpired()` and directly enforces the documented semantics of `cutoffAt`.

### Proof of Concept

1. Creator submits a transaction with `cutoffAt` set to `T+60s` and `validStart` set to `T+300s`.
2. At `T+61s` (after cutoff, before Hedera expiry), an authenticated signer calls:
   ```
   POST /transactions/:id/signers
   Body: { signatureMap: <valid signature map> }
   ```
3. `validateTransactionStatus()` is called. It checks status (`WAITING_FOR_SIGNATURES` — passes) and `isExpired()` (false, since `validStart + 180s` is still in the future — passes). `cutoffAt` is never read.
4. The signature is accepted, persisted to `TransactionSigner`, and `transactionBytes` is updated with the late signature.
5. If this late signature completes the required threshold, the transaction advances to `WAITING_FOR_EXECUTION`, executing after the creator's intended signing window has closed.

  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
