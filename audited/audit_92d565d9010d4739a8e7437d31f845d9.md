### Title
TOCTOU Race Condition in `uploadSignatureMaps` Allows Signatures to Be Written to Expired or Executed Transactions

### Summary
The `uploadSignatureMaps` flow in `SignersService` reads and validates transaction status at one point in time (`loadTransactionData`), then writes signature data to the database later (`persistSignatureChanges`) without re-checking the current status. The chain service's `handleExpiredTransactions` cron (runs every 10 seconds) and the execution scheduler can change a transaction's status between the check and the write, causing signatures and signer records to be persisted against transactions that are already EXPIRED or EXECUTED.

### Finding Description

The `uploadSignatureMaps` method in `back-end/apps/api/src/transactions/signers/signers.service.ts` is split into four sequential async phases:

**Phase 1 — Read (T1):** `loadTransactionData` fetches the transaction row and existing signers from the database. [1](#0-0) 

**Phase 2 — Validate (T1 snapshot):** `validateAndProcessSignatures` calls `validateTransactionStatus`, which checks `transaction.status` from the in-memory snapshot captured at T1. It also calls `computeSignatureKey`, which makes async mirror-node HTTP calls that can take 5–15 seconds (documented in the test suite comment). [2](#0-1) 

**Phase 3 — Write (T2):** `persistSignatureChanges` executes `bulkUpdateTransactions` and `bulkInsertSigners` inside a DB transaction. Critically, `bulkUpdateTransactions` issues a raw SQL `UPDATE` with **no status guard**: [3](#0-2) 

`bulkInsertSigners` similarly inserts signer records with no status guard: [4](#0-3) 

**Concurrent actor — Expiry cron (every 10 seconds):** `handleExpiredTransactions` in the chain service issues a guarded bulk `UPDATE … SET status = EXPIRED` for any `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` transaction whose `validStart` has passed. [5](#0-4) 

**Race window:** Between T1 (status read) and T2 (write), the cron can mark the transaction EXPIRED in the database. Because the write in Phase 3 has no `AND status IN (...)` guard, it overwrites `transactionBytes` and inserts `TransactionSigner` rows for a transaction that is now in a terminal state.

### Impact Explanation

- **`transactionBytes` of an EXPIRED/EXECUTED transaction are overwritten** with a new signature-bearing payload. This corrupts the audit trail: the stored bytes no longer match the bytes that were actually submitted to (or rejected by) the Hedera network.
- **`TransactionSigner` rows are inserted** for a terminal transaction, creating false accounting of who signed what and when.
- **`processTransactionStatus`** is then called with the stale in-memory `transaction` object (old status). Its `AND status = :oldStatus` guard prevents an incorrect DB status promotion, but the bytes and signer records are already corrupted.
- Any downstream system or audit query that reads `transactionBytes` or `TransactionSigner` for an expired transaction will see inconsistent, attacker-influenced data.

### Likelihood Explanation

- Any authenticated organization user can call `POST /transactions/:id/signers` — no elevated privilege required.
- The `computeSignatureKey` path makes mirror-node HTTP calls that routinely take 5–15 seconds (confirmed by the comment in `GroupPage.ts` lines 396–400).
- `handleExpiredTransactions` fires every 10 seconds.
- A transaction whose `validStart` is within the 3-minute expiry window is a natural target: the user legitimately tries to sign it just before expiry, the cron fires during the mirror-node lookup, and the write lands after the status flip.
- No special tooling is needed — a standard HTTP client racing a normal sign request against the cron window is sufficient.

### Recommendation

Add a status guard to `bulkUpdateTransactions` so the `UPDATE` is a no-op if the transaction has already left a signable state:

```sql
UPDATE transaction
SET "transactionBytes" = CASE id ... END,
    "updatedAt" = NOW()
WHERE id = ANY($N)
  AND status IN ('WAITING_FOR_SIGNATURES', 'WAITING_FOR_EXECUTION')
```

Apply the same guard to `bulkInsertSigners` by wrapping the insert in a subquery that checks the current status, or by re-reading the transaction status inside the DB transaction before writing. The status re-read must happen **inside** the same `dataSource.transaction(...)` block to be atomic. [6](#0-5) 

### Proof of Concept

1. Identify a transaction in `WAITING_FOR_SIGNATURES` status whose `validStart` is approximately 3 minutes in the future (so the expiry cron will fire during the request).
2. As an authenticated user who is a signer on that transaction, send `POST /transactions/:id/signers` with a valid signature map.
3. The server enters `computeSignatureKey`, which makes mirror-node HTTP calls (5–15 s).
4. During that window, `handleExpiredTransactions` fires and sets `status = EXPIRED` for the transaction in the DB.
5. `persistSignatureChanges` completes: `bulkUpdateTransactions` writes new `transactionBytes` to the now-EXPIRED row; `bulkInsertSigners` inserts a `TransactionSigner` record.
6. Query the DB: the EXPIRED transaction has updated `transactionBytes` and a new `TransactionSigner` row — state that should be impossible for a terminal transaction.

```typescript
  private async loadTransactionData(dto: UploadSignatureMapDto[]) {
    const transactionIds = dto.map(item => item.id);

    // Batch load all transactions
    const transactions = await this.dataSource.manager.find(Transaction, {
      where: { id: In(transactionIds) },
    });

    const transactionMap = new Map(transactions.map(t => [t.id, t]));

    // Batch load all existing signers
    const existingSigners = await this.dataSource.manager.find(TransactionSigner, {
      where: { transactionId: In(transactionIds) },
      select: ['transactionId', 'userKeyId'],
    });

    // Group by transaction ID
    const signersByTransaction = new Map<number, Set<number>>();
    for (const signer of existingSigners) {
      if (!signersByTransaction.has(signer.transactionId)) {
        signersByTransaction.set(signer.transactionId, new Set());
      }
      signersByTransaction.get(signer.transactionId).add(signer.userKeyId);
    }

    return { transactionMap, signersByTransaction };
  }
```

```typescript
  private validateTransactionStatus(transaction: Transaction): string | null {
    if (
      transaction.status !== TransactionStatus.WAITING_FOR_SIGNATURES &&
      transaction.status !== TransactionStatus.WAITING_FOR_EXECUTION
    ) {
      return ErrorCodes.TNRS;
    }

    const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
    if (isExpired(sdkTransaction)) {
      return ErrorCodes.TE;
    }

    return null;
  }
```

```typescript
  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
```

```typescript
  @Cron(CronExpression.EVERY_10_SECONDS, {
    name: 'status_update_expired_transactions',
  })
  async handleExpiredTransactions() {
    const result = await this.transactionRepo
      .createQueryBuilder()
      .update(Transaction)
      .set({ status: TransactionStatus.EXPIRED })
      .where('status IN (:...statuses) AND validStart < :before', {
        statuses: [
          TransactionStatus.NEW,
          TransactionStatus.REJECTED,
          TransactionStatus.WAITING_FOR_EXECUTION,
          TransactionStatus.WAITING_FOR_SIGNATURES,
        ],
        before: this.getThreeMinutesBefore(),
      })
      .returning(['id'])
      .execute();

    if (result.raw.length > 0) {
      emitTransactionStatusUpdate(
        this.notificationsPublisher,
        result.raw.map(t => ({
          entityId: t.id,
        })),
      );
    }
  }
```
