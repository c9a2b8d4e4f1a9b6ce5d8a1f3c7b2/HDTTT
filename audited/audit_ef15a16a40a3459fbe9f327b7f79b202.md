### Title
Unbounded `groupItems` Array in Transaction Group Creation Enables Authenticated Resource Exhaustion Against Chain Service

### Summary
The `CreateTransactionGroupDto` accepts a `groupItems` array with no upper-bound constraint. The chain service iterates over every item in the group during both the collation phase (computing signature keys, smart-collating) and the execution phase (submitting each transaction to the Hedera network). An authenticated user can submit a single crafted request containing thousands of group items, causing the chain service to exhaust CPU, memory, and Hedera network connections, degrading or blocking transaction processing for all users.

### Finding Description

**Root cause — missing `@ArrayMaxSize` on `groupItems`:**

`CreateTransactionGroupDto` validates the `groupItems` field with only `@IsArray()` and `@IsNotEmpty()`. There is no `@ArrayMaxSize()` decorator, so the array is unbounded. [1](#0-0) 

**Iteration 1 — collation phase (chain service scheduler):**

`collateGroupAndExecute` iterates over every group item, calling `computeSignatureKey` and `smartCollate` for each one. These are async operations involving DB queries and SDK serialization. [2](#0-1) 

**Iteration 2 — execution phase (execute service):**

`executeTransactionGroup` first validates every item (DB + SDK work), then either executes them all in parallel via `Promise.all` (non-sequential) or one-by-one (sequential). Each execution involves a live gRPC call to the Hedera network. [3](#0-2) 

**Exploit path:**

1. Attacker registers as a normal user (no privilege required).
2. Attacker POSTs to `POST /transaction-groups` with `groupItems` containing N=10,000 entries, each with minimal but valid `transactionBytes`.
3. The API persists all 10,000 items without any size check.
4. The chain scheduler picks up the group and calls `collateGroupAndExecute`, iterating all 10,000 items.
5. `executeTransactionGroup` then fires 10,000 parallel Hedera gRPC calls (`Promise.all`), exhausting the connection pool and blocking all other transaction processing.

**Secondary unbounded surface — approver tree:**

`CreateTransactionApproversArrayDto.approversArray` and the nested `CreateTransactionApproverDto.approvers` also have no `@ArrayMaxSize()`. The recursive `createApprover` function performs 4–5 DB queries per node. A deeply nested or wide tree causes proportional DB load within a single DB transaction. [4](#0-3) [5](#0-4) 

### Impact Explanation

- **Chain service DoS**: A single request with thousands of group items causes the chain service to issue thousands of parallel Hedera network calls, exhausting the gRPC connection pool and blocking all legitimate transaction execution for every user on the platform.
- **DB exhaustion via approvers**: A crafted approver tree with hundreds of nodes causes hundreds of DB queries inside a single transaction, holding DB connections and potentially timing out the connection pool.
- **Severity: High** — complete disruption of the core transaction scheduling and execution pipeline, affecting all users.

### Likelihood Explanation

- Any registered user can create a transaction group; no admin or privileged role is required.
- The attack requires a single HTTP POST with a crafted JSON body — trivially automatable.
- No rate limiting or size guard is present at the DTO, service, or infrastructure layer (none found in the codebase).
- **Likelihood: High**.

### Recommendation

1. Add `@ArrayMaxSize(N)` to `groupItems` in `CreateTransactionGroupDto` (e.g., `N = 100`): [1](#0-0) 

2. Add `@ArrayMaxSize(N)` to `approversArray` in `CreateTransactionApproversArrayDto` and to `approvers` in `CreateTransactionApproverDto`, and enforce a maximum recursion depth in `createApprover`: [6](#0-5) 

3. Add `@ArrayMaxSize(N)` to `userIds` in `CreateTransactionObserversDto`: [7](#0-6) 

### Proof of Concept

```http
POST /transaction-groups
Authorization: Bearer <valid_user_token>
Content-Type: application/json

{
  "description": "dos-group",
  "atomic": false,
  "sequential": false,
  "groupItems": [
    { "transactionBytes": "<minimal_valid_tx_hex>", "mirrorNetwork": "testnet", ... },
    // ... repeated 10,000 times
  ]
}
```

**Expected outcome**: The API accepts the request (no size validation). The chain scheduler picks up the group and calls `executeTransactionGroup`, which fires 10,000 parallel `_executeTransaction` calls via `Promise.all`, exhausting the Hedera gRPC connection pool and blocking all other transaction execution on the server.
