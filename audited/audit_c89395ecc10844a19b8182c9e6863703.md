### Title
Any Authenticated User Can Enumerate and Read All Other Users' Historical Transactions

### Summary
The `GET /transactions/history` endpoint returns all terminal-status transactions (EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED) from every user in the organization without any per-user access filtering. Additionally, `verifyAccess` unconditionally grants read access to any terminal-status transaction by ID. Any authenticated organization member can enumerate the full transaction history of all other users and retrieve their transaction bytes, names, descriptions, and network metadata.

### Finding Description

**Root Cause 1 — `getHistoryTransactions` has no user scope**

The controller passes no `user` argument to the service: [1](#0-0) 

The service method accepts no `User` parameter and queries the database with only a status filter, returning every matching row regardless of who created, signed, or observed it: [2](#0-1) 

Compare this to `getTransactions`, which correctly scopes results to the requesting user via `signers.userId`, `observers.userId`, and `creatorKey.userId`: [3](#0-2) 

**Root Cause 2 — `verifyAccess` bypasses all ownership checks for terminal statuses**

When a caller requests `GET /transactions/:id`, `verifyAccess` returns `true` immediately for any transaction whose status is EXECUTED, EXPIRED, FAILED, CANCELED, or ARCHIVED — without verifying that the requesting user is the creator, a signer, an observer, or an approver: [4](#0-3) 

**Combined exploit path:**
1. Attacker authenticates as any valid organization user (no elevated role required).
2. `GET /transactions/history` returns a paginated list of **all** terminal-status transactions across the entire organization, including their IDs, names, descriptions, and network metadata.
3. For each returned `id`, `GET /transactions/:id` returns the full `TransactionFullDto` — including raw `transactionBytes` — because `verifyAccess` returns `true` unconditionally for terminal statuses.

### Impact Explanation

Every historical transaction in the organization is readable by any authenticated member. Exposed data includes: transaction names and descriptions (which may contain business-sensitive labels), raw `transactionBytes` (revealing account IDs, amounts, memo fields, and key structures), `transactionId`, `transactionHash`, `mirrorNetwork`, and `validStart`. This constitutes cross-tenant data exposure and breaks the confidentiality model of the multi-user organization workflow.

### Likelihood Explanation

The attacker precondition is only a valid JWT for any organization account — the lowest possible privilege level. The exploit requires two standard REST calls with no special tooling. The endpoint is documented in Swagger and is actively used by the frontend. Likelihood is high.

### Recommendation

1. **`getHistoryTransactions`**: Accept a `User` parameter and apply the same ownership-scoped `WHERE` clause used in `getTransactions` (creator, signer, observer, approver membership). The controller must pass `@GetUser()` to the service.

2. **`verifyAccess`**: Remove the blanket `return true` for terminal statuses. Replace it with the same ownership check applied to active transactions (creator, signer, observer, approver). If a public history view is intentionally desired, make it an explicit, separately guarded endpoint with clear documentation.

### Proof of Concept

```
# Step 1 – authenticate as any low-privilege org user
POST /auth/login  { "email": "attacker@org.com", "password": "..." }
# → receives JWT

# Step 2 – enumerate all historical transactions of all users
GET /transactions/history
Authorization: Bearer <attacker_jwt>
# → returns paginated list of ALL terminal-status transactions
#   including IDs, names, descriptions belonging to other users

# Step 3 – read full details of any returned transaction
GET /transactions/42
Authorization: Bearer <attacker_jwt>
# → verifyAccess returns true (status = EXECUTED), full TransactionFullDto returned
#   including transactionBytes, transactionHash, creatorKey, signers, observers
```

Expected result: attacker receives complete transaction data for records they did not create, sign, observe, or approve.

    { page, limit, size, offset }: Pagination,
    filter: Filtering[] = [],
    sort: Sorting[] = [],
  ): Promise<PaginatedResourceDto<Transaction>> {
    const order = getOrder(sort);

    const findOptions: FindManyOptions<Transaction> = {
      where: {
        ...getWhere<Transaction>(filter),
        status: this.getHistoryStatusWhere(filter),
      },
      order,
      relations: ['groupItem', 'groupItem.group'],
      skip: offset,
      take: limit,
    };

    const [transactions, total] = await this.repo
      .createQueryBuilder()
      .setFindOptions(findOptions)
      .getManyAndCount();

    return {
      totalItems: total,
      items: transactions,
      page,
      size,
    };

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
