### Title
Stored XSS via Unsanitized `v-html` Rendering of User-Controlled Transaction Memo and Description in `CreateTransactionGroup.vue`

### Summary
The `CreateTransactionGroup.vue` page renders user-controlled transaction data — specifically `transactionMemo` and `groupItem.description` — directly into the DOM using Vue's `v-html` directive without any HTML sanitization. In organization mode, a malicious member can craft a transaction with an HTML/JavaScript payload in the memo field, store it via the backend, and have it execute in the Electron renderer of any other organization member who opens the Create Transaction Group page. This is stored XSS, not self-XSS.

### Finding Description

In `front-end/src/renderer/pages/CreateTransactionGroup/CreateTransactionGroup.vue`, lines 421–429, the template renders group item content using `v-html`:

```html
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
``` [1](#0-0) 

Two distinct user-controlled values are injected raw into the DOM:

1. **`groupItem.description`** — a free-text description field set by any organization member when creating or editing a transaction group item.
2. **`Transaction.fromBytes(groupItem.transactionBytes).transactionMemo`** — the Hedera transaction memo, which is an arbitrary string embedded in the serialized transaction bytes. Any organization member who can submit a transaction can set this field to arbitrary content.

Neither value is passed through any HTML-escaping or sanitization function before being handed to `v-html`. Vue's `v-html` bypasses its own template escaping and sets `innerHTML` directly, so any HTML tags or `<script>` content in these strings execute in the renderer process. [2](#0-1) 

The Electron main window security configuration is defined in `front-end/src/main/windows/mainWindow.ts`, which references `nodeIntegration`, `contextIsolation`, and `webSecurity` settings. Even with `contextIsolation: true` and `nodeIntegration: false` (the safer defaults), XSS in the renderer still allows full DOM manipulation, credential harvesting from the UI, and potential IPC abuse through the exposed preload API surface. [3](#0-2) 

### Impact Explanation

A malicious organization member crafts a transaction whose `transactionMemo` contains `<img src=x onerror="...">` or an inline `<script>` block. This transaction is saved to the shared backend. When any other organization member opens the Create Transaction Group page and the group item is rendered, the payload executes in their Electron renderer process. Consequences include:

- **Credential theft**: The renderer has access to all data displayed in the UI, including account IDs, public keys, and any sensitive values rendered on-screen.
- **Transaction manipulation**: The attacker's script can silently alter transaction parameters displayed to the victim before signing, causing the victim to sign a transaction they did not intend to authorize — a direct analog to the FilSnap finding where Markdown injection caused users to sign inaccurate messages.
- **IPC abuse**: The preload script exposes IPC channels to the renderer. Depending on what handlers are registered, XSS can invoke privileged main-process operations (file system access, key operations, etc.).

### Likelihood Explanation

The attack requires only a normal organization member account — no admin or privileged role. Creating a transaction with a crafted memo is a standard, reachable workflow. The payload is stored in the backend and delivered to every organization member who views the affected page. No user interaction beyond opening the page is required to trigger execution.

### Recommendation

Replace `v-html` with safe text interpolation (`{{ }}`) for all user-controlled fields. Vue's double-brace syntax HTML-encodes output automatically:

```html
<!-- Replace this -->
<div v-html="groupItem.description"></div>

<!-- With this -->
<div>{{ groupItem.description }}</div>
```

If rich HTML rendering is genuinely required for any field, pass the value through a strict allowlist sanitizer (e.g., DOMPurify with a minimal tag allowlist) before binding to `v-html`. Apply the same fix to the `makeTransfer` return value from `formatHbarTransfers`, which is also rendered via `v-html` and should be audited for injection vectors. [1](#0-0) 

### Proof of Concept

**Preconditions**: Attacker has a valid organization member account. Victim is another organization member.

1. Attacker creates a Hedera transaction (e.g., a Transfer Transaction) and sets the `transactionMemo` to:
   ```
   <img src=x onerror="document.title=document.cookie">
   ```
2. Attacker adds this transaction to a shared transaction group via the organization backend.
3. Victim opens the **Create Transaction Group** page in their Electron client and loads the shared group.
4. The `v-html` binding at line 426 sets `innerHTML` to the attacker's payload.
5. The browser engine inside Electron fires the `onerror` handler, executing arbitrary JavaScript in the victim's renderer process.
6. **Expected outcome**: Attacker's script runs in the victim's Electron renderer, with access to all DOM content, Pinia store state, and preload-exposed IPC channels.
