### Title
Unauthenticated Cross-Tenant Transaction History Exposure via `GET /transactions/history`

### Summary
The `GET /transactions/history` endpoint in the API service returns **all** historical transactions from all users in the organization to any authenticated user. Unlike every other transaction-listing endpoint, `getHistoryTransactions` accepts no `user` parameter and applies no per-user access filter. Any low-privilege authenticated user can enumerate the full transaction history of the entire organization.

### Finding Description

**Root cause:** `getHistoryTransactions` in `transactions.service.ts` takes no `user` argument and performs no ownership/visibility check. It queries all transactions whose status falls in the terminal set (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`) and returns them to the caller. [1](#0-0) 

The controller wires this endpoint without passing the authenticated user: [2](#0-1) 

**Contrast with the correctly scoped endpoints.** Every other listing method (`getTransactions`, `getTransactionsToSign`, `getTransactionsToApprove`) receives the `user` object and restricts results to transactions where the user is creator, signer, observer, or approver: [3](#0-2) 

The controller comments on three of those endpoints explicitly state `/* NO LONGER USED BY FRONT-END */`, indicating the codebase has evolved and the front-end has migrated to different entry points — yet the `/history` endpoint, which was left without user-scoping, remains fully active and reachable. [4](#0-3) 

The only guard on the controller class is `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — all of which only confirm the caller is a valid, non-blacklisted, verified user. No ownership or visibility check is applied to `/history`. [5](#0-4) 

**Exploit path:**
1. Attacker registers or is invited as a normal organization user (no admin role required).
2. Attacker sends `GET /transactions/history?page=1&limit=100` with their JWT.
3. The server returns all terminal-status transactions from all users in the organization, including transaction IDs, account IDs, amounts, network, creator key references, and group membership.
4. Attacker iterates pages to dump the complete transaction history.

### Impact Explanation
Any authenticated user can read the full historical transaction record of every other user and every organization account. This exposes:
- Hedera account IDs and transaction IDs of all parties
- Transaction types, amounts, and network details
- Creator key associations (linking keys to users)
- Group/batch transaction membership

This is a **cross-tenant confidentiality breach** with no privilege requirement beyond a valid session token.

### Likelihood Explanation
The attacker precondition is only a valid, verified user account — the lowest possible privilege level in the system. The endpoint is a standard REST `GET` with no additional guards. Any user who can log in can immediately exploit this. The `filter` and `sort` query parameters further allow targeted enumeration (e.g., filter by `mirrorNetwork` or `type`).

### Recommendation
Pass the authenticated `user` into `getHistoryTransactions` and apply the same visibility filter used by `getTransactions` — restricting results to transactions where the requesting user is creator, signer, observer, or approver:

```typescript
// controller
getHistoryTransactions(
  @GetUser() user: User,   // add this
  @PaginationParams() paginationParams: Pagination,
  ...
) {
  return this.transactionsService.getHistoryTransactions(user, paginationParams, filter, sort);
}

// service
async getHistoryTransactions(user: User, ...) {
  // apply the same whereForUser bracket used in getTransactions()
}
```

```
# Response: 200 OK

  "items": [

```

The response contains records belonging to all users in the organization, not just the attacker's own transactions.
