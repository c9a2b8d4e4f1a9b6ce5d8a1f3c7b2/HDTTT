### Title
JWT Token Not Revoked on Password Change — Stolen Session Persists After Credential Rotation

### Summary
The `PATCH /auth/change-password` endpoint successfully updates the user's password but never blacklists the caller's current JWT. Any attacker holding a stolen bearer token retains full API access for up to 365 days even after the victim rotates their credentials. The JWT blacklist mechanism exists and is correctly applied on logout and OTP-based password reset, but the password-change path is silently omitted — an inconsistent blacklist directly analogous to the external report's finding.

### Finding Description

**Root cause — inconsistent blacklist application**

Three security events exist that should revoke the current session token. Two are handled; one is not:

| Endpoint | Blacklists current token? |
|---|---|
| `POST /auth/logout` | Yes — `blacklistService.blacklistToken(extractJwtAuth(req))` |
| `POST /auth/verify-reset` | Yes — `blacklistService.blacklistToken(extractJwtOtp(req))` |
| `PATCH /auth/set-password` | Yes — `blacklistService.blacklistToken(extractJwtOtp(req))` |
| **`PATCH /auth/change-password`** | **No — nothing** | [1](#0-0) 

The `changePassword` controller delegates to `authService.changePassword()` and returns. Neither the controller nor the service calls `blacklistService.blacklistToken()`. [2](#0-1) 

**Why the JWT cannot be invalidated by any other means**

The JWT payload contains only `userId` and `email` — no password hash, no credential version, no session ID: [3](#0-2) 

The JWT validation strategy only checks whether the user record still exists in the database: [4](#0-3) 

Changing the password does not delete the user record, so the stolen token passes validation indefinitely until it naturally expires.

**Token lifetime**

The default `JWT_EXPIRATION` is 365 days in every deployment configuration: [5](#0-4) 

The `BlacklistService` correctly stores revoked tokens in Redis with a TTL equal to `JWT_EXPIRATION` days, but it is never called from the password-change path: [6](#0-5) 

### Impact Explanation

An attacker who obtains a user's bearer token (via XSS, network interception, compromised device, etc.) retains complete API access — read all transactions, sign on behalf of the user, view all organization data — for up to **365 days** after the victim changes their password. The victim has no mechanism to revoke the stolen session short of contacting an administrator to delete their account.

### Likelihood Explanation

The attacker precondition is possession of a valid JWT, which is a realistic outcome of XSS, token leakage in logs, or a compromised client device. No privileged access is required. The victim's natural response to a suspected compromise — changing their password — is entirely ineffective, making the window of exploitation the full remaining token lifetime (up to 365 days). The attack path requires no special tooling: a single HTTP request with the stolen `Authorization: Bearer <token>` header suffices.

### Recommendation

In `AuthController.changePassword`, after a successful password update, blacklist the caller's current JWT:

```typescript
@Patch('/change-password')
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
async changePassword(
  @GetUser() user: User,
  @Body() dto: ChangePasswordDto,
  @Req() req: Request,   // add request parameter
): Promise<void> {
  await this.authService.changePassword(user, dto);
  await this.blacklistService.blacklistToken(extractJwtAuth(req)); // revoke current session
}
```

This mirrors the pattern already used in `logout` and `setPassword`. [7](#0-6) 

### Proof of Concept

1. Authenticate as `victim@example.com` → receive `TOKEN_A`.
2. Simulate token theft: copy `TOKEN_A` to an attacker-controlled client.
3. As the victim, call `PATCH /auth/change-password` with valid `oldPassword` / `newPassword` using `TOKEN_A`. Response: `200 OK`.
4. As the attacker, call `GET /transactions/history?page=1&size=10` with `Authorization: Bearer TOKEN_A`. **Expected (secure): `401 Unauthorized`. Actual: `200 OK`** — full transaction data returned.
5. Repeat step 4 at any point within the next 365 days; the result remains `200 OK`.

  /* Change the password for the given user */
  async changePassword(user: User, { oldPassword, newPassword }: ChangePasswordDto): Promise<void> {
    if (oldPassword === newPassword) throw new BadRequestException(ErrorCodes.NPMOP);

    const { correct } = await this.dualCompareHash(oldPassword, user.password);
    if (!correct) throw new BadRequestException(ErrorCodes.INOP);

    if (user.status === UserStatus.NEW && user.keys.length === 0) {
      emitUserStatusUpdateNotifications(this.notificationsPublisher, { entityId: user.id, additionalData: { username: user.email } });
    }

    await this.usersService.setPassword(user, newPassword);
  }
