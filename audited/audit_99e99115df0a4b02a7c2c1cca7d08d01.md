### Title
Transaction Approver Modification Allowed Regardless of Transaction Status

### Summary

The `createTransactionApprovers`, `updateTransactionApprover`, and `removeTransactionApprover` operations in the approvers service do not check the current `transaction.status` before allowing modifications. This is directly analogous to the external report: a state flag (`transaction.status`) controls what operations should be permitted, but the flag is never enforced in the approver-mutation paths. A transaction creator can add, update, or remove approvers on a transaction that has already reached `WAITING_FOR_EXECUTION` status (i.e., all required approvals have been collected), triggering a status recalculation that can revert the transaction back to `WAITING_FOR_SIGNATURES` and block execution indefinitely.

---

### Finding Description

The `approveTransaction` function correctly enforces a status gate: [1](#0-0) 

However, the three mutation paths that modify the approver set share a common helper, `getCreatorsTransaction`, which only verifies that the caller is the transaction creator — it never inspects `transaction.status`: [2](#0-1) 

`createTransactionApprovers` calls this helper and proceeds without any status check: [3](#0-2) 

`updateTransactionApprover` does the same: [4](#0-3) 

`removeTransactionApprover` in the controller calls `getCreatorsTransaction` but again adds no status guard: [5](#0-4) 

After every mutation, a `emitTransactionStatusUpdate` event is fired, which triggers a status recalculation: [6](#0-5) [7](#0-6) 

The same missing status check exists in the observers service (`createTransactionObservers`, `getUpdateableObserver`), though with lower impact since observers are read-only participants: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

The `TransactionStatus` enum defines a clear lifecycle: [10](#0-9) 

Once a transaction reaches `WAITING_FOR_EXECUTION`, all required approvals have been collected and the transaction is ready to be submitted to the Hedera network. Because `createTransactionApprovers` and `removeTransactionApprover` both fire `emitTransactionStatusUpdate` after mutating the approver set, adding a new (unapproved) approver or removing an existing one at this stage causes the status engine to recalculate and can revert the transaction to `WAITING_FOR_SIGNATURES`. This effectively lets the creator stall or permanently block execution of a transaction that all other parties have already approved.

---

### Likelihood Explanation

The attack requires the caller to be the transaction creator (a legitimate, authenticated user), so it is not exploitable by an arbitrary third party. However, any creator — whether acting maliciously or by mistake — can trigger this at any time via the standard REST API (`POST /transactions/:id/approvers`, `PATCH /transactions/:id/approvers/:id`, `DELETE /transactions/:id/approvers/:id`). No special privilege beyond being the creator is needed, and there is no time window or race condition required.

---

### Recommendation

Add a status guard inside `getCreatorsTransaction` (or at the top of each mutation method) that rejects modifications when the transaction is in a terminal or post-approval state. For example, reject if `status` is not in `[WAITING_FOR_SIGNATURES]` (or whatever states are appropriate for approver setup):

```typescript
const mutableStatuses = [TransactionStatus.WAITING_FOR_SIGNATURES];
if (!mutableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP); // or a dedicated error code
}
```

Apply the same guard to `createTransactionObservers` and `getUpdateableObserver` in `observers.service.ts`.

---

### Proof of Concept

1. User A (creator) creates a transaction; it enters `WAITING_FOR_SIGNATURES`.
2. Creator adds User B as an approver via `POST /transactions/:id/approvers`.
3. User B approves via `POST /transactions/:id/approvers/approve`; the status engine advances the transaction to `WAITING_FOR_EXECUTION`.
4. Creator calls `POST /transactions/:id/approvers` again, adding User C as a new approver — **no status check blocks this**.
5. `emitTransactionStatusUpdate` fires; the status engine sees an unapproved approver (User C) and reverts the transaction to `WAITING_FOR_SIGNATURES`.
6. The transaction can never reach execution unless User C also approves, which the creator can prevent by repeating step 4 with a new user indefinitely.

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }

  async createTransactionObservers(
    user: User,
    transactionId: number,
    dto: CreateTransactionObserversDto,
  ): Promise<TransactionObserver[]> {
    const transaction = await this.entityManager.findOne(Transaction, {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user', 'observers'],
    });

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to delete it');

    const observers: TransactionObserver[] = [];

    for (const userId of dto.userIds) {
      if (!transaction.observers.some(o => o.userId === userId)) {
        const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
        observers.push(observer);
      }
    }

    if (observers.length === 0) {
      return [];
    }

    try {
      const result = await this.repo.save(observers);

      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);

      return result;
    } catch (error) {
      throw new BadRequestException(error.message);
    }
