### Title
Incorrect Boolean Guard in `archiveTransaction` Allows Creators to Archive Manual Transactions in Any Terminal State

### Summary

The `archiveTransaction` function in `back-end/apps/api/src/transactions/transactions.service.ts` contains a logically flawed guard condition. Due to a misplaced `&&` operator, any creator of a `isManual = true` transaction can force it into `ARCHIVED` status regardless of its current state — including `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`. This corrupts the transaction state machine and audit trail without requiring any privileged access.

### Finding Description

**Root cause — incorrect compound guard:** [1](#0-0) 

```typescript
if (
  ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
    transaction.status,
  ) &&
  !transaction.isManual          // ← second operand of &&
) {
  throw new BadRequestException(ErrorCodes.OMTIP);
}
```

The guard throws only when **both** sub-conditions are true:
- `status` is not in `[WAITING_FOR_SIGNATURES, WAITING_FOR_EXECUTION]`
- **AND** `isManual` is `false`

When `isManual` is `true`, the second operand is `false`, so the entire `&&` expression is `false` and the guard **never throws** — regardless of the transaction's current status. A creator of any manual transaction can therefore call `PATCH /transactions/archive/:id` after the transaction has already been `EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`.

**Exploit path:**

1. Attacker (normal authenticated user) creates a transaction with `isManual: true` via `POST /transactions`.
2. The transaction reaches a terminal state (`EXECUTED`, `FAILED`, `EXPIRED`, or `CANCELED`) through normal protocol flow.
3. Attacker calls `PATCH /transactions/archive/:id`.
4. `getTransactionForCreator` passes (attacker is the creator).
5. The guard at lines 711–718 evaluates to `false` because `!transaction.isManual` is `false`.
6. `this.repo.update({ id }, { status: TransactionStatus.ARCHIVED })` executes unconditionally. [2](#0-1) 

The existing unit and e2e tests only exercise the non-manual path (`isManual` absent/`false`), so the bug is untested: [3](#0-2) [4](#0-3) 

Neither test sets `isManual: true` on the transaction under test, so the faulty branch is never exercised.

**Analog to external report:** The GooseFX bug was an incorrect state-application function that wrote values to the wrong variables, corrupting protocol state. Here the analog is an incorrect state-guard that fails to block writes, allowing invalid state transitions — same vulnerability class: **state transition integrity failure**.

### Impact Explanation

- A creator can move an already-`EXECUTED` transaction to `ARCHIVED`, hiding it from dashboards and audit views that filter by terminal status.
- A creator can move a `FAILED` or `EXPIRED` transaction to `ARCHIVED`, masking execution failures from observers and approvers.
- The `ARCHIVED` status is treated as a success-like terminal state by the notification system: [5](#0-4) 

This means observers and approvers receive misleading "archived" notifications for transactions that actually failed or were already executed, breaking the integrity of the audit trail.

### Likelihood Explanation

- **Attacker preconditions**: authenticated, verified user; creator of any manual transaction. No privileged keys or admin access required.
- **Trigger**: a single `PATCH /transactions/archive/:id` HTTP request after the transaction reaches a terminal state.
- **Reachability**: the endpoint is publicly reachable under `JwtAuthGuard` + `VerifiedUserGuard` only: [6](#0-5) 

Any organization user who creates manual transactions can exploit this immediately.

### Recommendation

Replace the compound `&&` guard with two independent checks that enforce both invariants separately:

```typescript
async archiveTransaction(id: number, user: User): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  // Only manual transactions may be archived
  if (!transaction.isManual) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  // Only archive from in-progress states
  if (
    ![TransactionStatus.WAITING_FOR_SIGNATURES, TransactionStatus.WAITING_FOR_EXECUTION].includes(
      transaction.status,
    )
  ) {
    throw new BadRequestException(ErrorCodes.OMTIP);
  }

  await this.repo.update({ id }, { status: TransactionStatus.ARCHIVED });
  ...
}
```

Add a unit test covering `isManual: true` with each terminal status (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`) to confirm the guard rejects them.

### Proof of Concept

**Preconditions**: valid JWT for user A; organization server running.

```bash
# 1. Create a manual transaction
POST /transactions
{ ..., "isManual": true }
# → { "id": 42, "status": "WAITING FOR SIGNATURES" }

# 2. Simulate terminal state (or wait for natural expiry / execution)
# (attacker observes status change to EXECUTED via WebSocket notification)

# 3. Archive the already-executed transaction
PATCH /transactions/archive/42
Authorization: Bearer <userA_token>

# Expected (correct): 400 Bad Request
# Actual (buggy):     200 OK  — transaction.status is now "ARCHIVED"
```

**Verification**: query the database — `SELECT status FROM transaction WHERE id = 42` returns `ARCHIVED` despite the transaction having been `EXECUTED`.
