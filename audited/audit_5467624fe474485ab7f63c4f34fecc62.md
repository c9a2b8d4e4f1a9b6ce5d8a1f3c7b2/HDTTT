### Title
`removeTransactionApprover` Does Not Update Parent `threshold` When a Child Approver Is Deleted, Making Approval Threshold Permanently Unsatisfiable

---

### Summary

When a child approver node is deleted from a threshold tree via `removeTransactionApprover`, the parent node's `threshold` value is never reduced. If the threshold was set equal to the original number of children (e.g., unanimous approval), removing even one child makes the threshold mathematically impossible to satisfy, permanently blocking the transaction.

---

### Finding Description

The `removeTransactionApprover` function in `approvers.service.ts` calls `removeNode(approver.id)`, which recursively soft-deletes the target approver and all its descendants via a raw SQL query. However, it never fetches or updates the **parent** node's `threshold`. [1](#0-0) 

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);   // ← parent threshold never touched
    emitTransactionStatusUpdate(...);
    return result;
}
```

`removeNode` only soft-deletes the subtree rooted at the given ID: [2](#0-1) 

The contrast is stark: `updateTransactionApprover` (the "detach" path, `dto.listId === null`) **does** reduce the parent's threshold when a child is moved out: [3](#0-2) 

```typescript
if (newParentApproversLength < parent.threshold) {
    await transactionalEntityManager.update(TransactionApprover, parent.id, {
        threshold: newParentApproversLength,
    });
}
```

This correction is entirely absent from the delete path.

The `DELETE /:id` endpoint is reachable by any authenticated transaction creator: [4](#0-3) 

The `TransactionApprover` entity stores `threshold` as a plain nullable integer column with no database-level constraint that enforces `threshold ≤ count(children)` after a delete: [5](#0-4) 

---

### Impact Explanation

**Permanently unsatisfiable approval threshold.** Consider a threshold tree with 3 children (Alice, Bob, Charlie) and `threshold = 3` (unanimous). If Alice is deleted:

- Remaining children: Bob, Charlie (2 total)
- Stored `threshold`: still 3
- Maximum achievable approvals: 2
- `2 < 3` → the threshold can **never** be met

The transaction is permanently stuck in `WAITING_FOR_SIGNATURES`. There is no mechanism in the service to auto-correct the threshold after deletion. The `isApproved` utility confirms approval is evaluated against the stored `threshold`: [6](#0-5) 

```typescript
if (approvals.length >= (approver.threshold || approvals.length)) {
    return true;
}
```

A threshold that exceeds the remaining child count can never be reached.

---

### Likelihood Explanation

The `DELETE /transactions/:transactionId/approvers/:id` endpoint accepts **any** approver ID — root or child — without restriction. A transaction creator who sets up a unanimous (N-of-N) threshold tree and later removes one member via the delete endpoint will silently corrupt the tree. This is an easy mistake to make during normal transaction management (e.g., a user leaves the organization). No special privileges beyond being the transaction creator are required.

---

### Recommendation

In `removeTransactionApprover`, after soft-deleting the node, fetch the deleted approver's parent (if any) and apply the same threshold-correction logic already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // If this node has a parent, correct the parent's threshold
    if (approver.listId) {
        const parent = await this.repo.findOne(TransactionApprover, {
            relations: ['approvers'],
            where: { id: approver.listId },
        });
        if (parent) {
            const newCount = parent.approvers.length - 1;
            if (newCount === 0) {
                await this.repo.softRemove(parent);
            } else if (newCount < parent.threshold) {
                await this.repo.update(parent.id, { threshold: newCount });
            }
        }
    }

    await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
}
```

### Proof of Concept

1. Creator sets up transaction T with a threshold tree:
   - Parent node: `threshold = 3`, children = [Alice (id=10), Bob (id=11), Charlie (id=12)]

2. Creator calls:
   ```
   DELETE /transactions/T/approvers/10
   ```

3. `removeNode(10)` soft-deletes Alice. Parent node still has `threshold = 3`.

4. Bob and Charlie both call `POST /transactions/T/approvers/approve` — only 2 approvals are possible.

5. `isApproved` evaluates `2 >= 3` → `false`. The transaction can never advance past `WAITING_FOR_SIGNATURES`. [1](#0-0) [2](#0-1) [4](#0-3)

export const isApproved = (approver: ITransactionApprover): boolean | null => {
  if (approver.approved === false) {
    return false;
  }

  if (approver.approved === true) {
    return true;
  }

  if (approver.approvers) {
    const approvals = approver.approvers.filter(approver => isApproved(approver) === true);
    const rejections = approver.approvers.filter(approver => isApproved(approver) === false);
    if (approvals.length >= (approver.threshold || approvals.length)) {
      return true;
    }
    return rejections.length >= (approver.threshold || rejections.length) ? false : null;
  }
