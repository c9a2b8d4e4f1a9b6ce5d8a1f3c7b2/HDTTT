### Title
Unverified (`NEW` Status) Users Can Access Notification Preferences Endpoints Due to Missing `VerifiedUserGuard`

### Summary

The `NotificationPreferencesController` applies only `JwtBlackListAuthGuard` and `JwtAuthGuard`, omitting the `VerifiedUserGuard` that every other sensitive controller uses. A user whose account is in `UserStatus.NEW` state (i.e., has not completed onboarding by setting their own password) can authenticate, obtain a valid JWT, and freely call `GET /notification-preferences` and `PATCH /notification-preferences` — endpoints that are supposed to be restricted to fully-onboarded (`UserStatus.NONE`) users. This is the direct analog of the external report's "endpoints that did not check KYC status" pattern.

---

### Finding Description

**Root cause — missing guard on the controller**

`NotificationPreferencesController` is decorated with:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard)
``` [1](#0-0) 

Every other protected controller in the API adds `VerifiedUserGuard` as the third guard:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
``` [2](#0-1) [3](#0-2) [4](#0-3) 

**What `VerifiedUserGuard` enforces**

The guard blocks any user whose `status` is not `UserStatus.NONE`:

```typescript
return user.status === UserStatus.NONE;
``` [5](#0-4) 

**The three statuses**

```typescript
export enum UserStatus {
  NEW = 'NEW',
  RESET_PASSWORD = 'RESET_PASSWORD',
  NONE = 'NONE',
}
``` [6](#0-5) 

Users are created with `UserStatus.NEW` by default and transition to `UserStatus.NONE` only after they set their own password via `setPassword()`:

```typescript
await this.updateUser(user, {
  password: await this.getSaltedHash(newPassword),
  status: UserStatus.NONE,
});
``` [7](#0-6) 

**Login does not check status**

`getVerifiedUser` validates only the password — it does not reject `NEW` or `RESET_PASSWORD` users — so a `NEW` user can obtain a valid JWT immediately after being invited: [8](#0-7) 

**Exposed endpoints**

With a valid JWT as a `NEW` user, the following routes are reachable without restriction:

| Method | Route | Action |
|--------|-------|--------|
| `GET` | `/notification-preferences` | Read own notification preferences |
| `PATCH` | `/notification-preferences` | Mutate own notification preferences | [9](#0-8) 

---

### Impact Explanation

A `NEW` user (invited but not yet onboarded) can read and overwrite their own notification preferences before the platform considers them a verified member. The access control boundary that `VerifiedUserGuard` is designed to enforce is bypassed for this controller. The data exposed is limited to the user's own notification settings (not financial data, not other users' data), so the severity is **Low**. However, the broken access control pattern is real and demonstrable — the guard omission is inconsistent with the explicit design intent of the rest of the API.

---

### Likelihood Explanation

Any user who has been added to the organization by an admin (i.e., exists in the database with `UserStatus.NEW`) and knows their temporary password can immediately exploit this. No privilege escalation, no leaked secrets, and no special tooling are required — a standard HTTP client suffices. Likelihood is **Medium** (requires a valid account, but no further preconditions).

---

### Recommendation

Add `VerifiedUserGuard` to `NotificationPreferencesController`'s `@UseGuards` decorator, consistent with every other protected controller:

```typescript
// back-end/apps/api/src/notification-preferences/notification-preferences.controller.ts
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
``` [10](#0-9) 

---

### Proof of Concept

1. Admin creates a new user via the invite flow; the user is stored with `status = 'NEW'`.
2. The `NEW` user logs in with their temporary credentials — `getVerifiedUser` succeeds and returns a JWT.
3. Using that JWT, the attacker sends:
   ```
   GET /notification-preferences
   Authorization: Bearer <jwt_of_NEW_user>
   ```
   Expected (per design): `403 Forbidden` from `VerifiedUserGuard`.
   Actual: `200 OK` — notification preferences are returned.
4. Similarly:
   ```
   PATCH /notification-preferences
   Authorization: Bearer <jwt_of_NEW_user>
   Content-Type: application/json
   { "type": "TRANSACTION_READY_FOR_EXECUTION", "email": false, "inApp": false }
   ```
   Actual: `200 OK` — preferences are mutated.

The `NEW` user has bypassed the verification gate that all other endpoints enforce. [11](#0-10)

  async getVerifiedUser(email: string, password: string): Promise<User> {
    let user: User;

    try {
      user = await this.getUser({ email });
    } catch {
      throw new InternalServerErrorException('Failed to retrieve user.');
    }

    if (!user) {
      throw new UnauthorizedException('Please check your login credentials');
    }

    const { correct, isBcrypt } = await this.dualCompareHash(password, user.password);

    if (!correct) {
      throw new UnauthorizedException('Please check your login credentials');
    }

    if (isBcrypt) {
      await this.setPassword(user, password);
    }

    return user;
  }
