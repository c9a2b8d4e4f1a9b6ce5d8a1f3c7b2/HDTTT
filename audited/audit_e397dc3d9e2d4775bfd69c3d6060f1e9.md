### Title
Unbounded Full-Table Fetch with N+1 DB Queries in `getTransactionsToSign` Enables Authenticated DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **every** non-terminal transaction from the database with no `skip`/`take` limit, then executes an additional async DB query and full transaction deserialization for each row to determine signing eligibility. Pagination is applied only in-memory after all work is done. As the transaction table grows, any authenticated user can trigger memory exhaustion, CPU saturation, and DB connection pool starvation by calling `GET /transactions/sign?page=1&size=10`.

### Finding Description

**Root cause — unbounded DB fetch:**

In `back-end/apps/api/src/transactions/transactions.service.ts` the `getTransactionsToSign` method calls `this.repo.find(...)` with no `skip` or `take`:

```typescript
// lines 295-299 — no skip/take, loads entire table
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
});
``` [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses; it does **not** scope to the requesting user's transactions. Every active transaction in the system is loaded into the Node.js heap.

**Root cause — N+1 async queries per row:**

After loading all rows, the code iterates and calls `userKeysToSign` for each transaction:

```typescript
// lines 301-309 — one async DB round-trip + SDK deserialization per transaction
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

`userKeysToSign` calls `userKeysRequiredToSign` → `keysRequiredToSign`, which deserializes the transaction bytes via `SDKTransaction.fromBytes` and issues a `entityManager.find(UserKey, ...)` query: [3](#0-2) 

**Root cause — pagination applied only after full scan:**

```typescript
// line 313 — in-memory slice, not DB-level pagination
items: result.slice(offset, offset + limit),
``` [4](#0-3) 

**Contrast with correctly paginated sibling methods:**

`getTransactions` and `getHistoryTransactions` both pass `skip: offset, take: limit` directly to the ORM query: [5](#0-4) 

**Exposed endpoint:**

The controller exposes this as an authenticated `GET /transactions/sign` route, enforced only by `JwtAuthGuard` and `VerifiedUserGuard` — no admin role required: [6](#0-5) 

The `PaginationParams` decorator enforces `size ≤ 100` at the HTTP layer, but this limit is never propagated to the DB query — it only controls the final in-memory slice: [7](#0-6) 

### Impact Explanation

Each call to `GET /transactions/sign?page=1&size=10` with N active transactions in the system causes:

1. **Memory exhaustion** — all N transaction rows (including `transactionBytes` blobs) loaded into the Node.js heap.
2. **CPU exhaustion** — N synchronous `SDKTransaction.fromBytes(...)` deserializations.
3. **DB connection pool starvation** — N sequential `entityManager.find(UserKey, ...)` queries, each holding a connection.

A single authenticated user issuing repeated requests can degrade or crash the API service for all other users. The impact scales linearly with the number of active transactions on the platform.

### Likelihood Explanation

- **Attacker precondition:** Any registered, email-verified user. No admin or privileged role required.
- **Trigger:** A single HTTP GET request to a documented, publicly reachable endpoint.
- **Amplification:** The attacker does not need to create the transactions themselves; transactions created by other users accumulate naturally over time, making the attack progressively easier as the platform grows.
- **No rate-limit evidence** was found in the controller or middleware for this specific endpoint.

### Recommendation

Apply DB-level pagination before the per-row loop. The simplest fix is to push the signing-eligibility check into SQL (as `getTransactionNodes` already does via `getTransactionNodesQuery` with `{ signer: true, onlyUnsigned: true }`), eliminating the N+1 pattern entirely: [8](#0-7) 

If the in-memory approach must be kept, at minimum add `skip` and `take` to the `repo.find` call and accept that `totalItems` will be approximate. A proper fix moves the user-key matching into the SQL query so the DB returns only the rows the user can sign, then applies `LIMIT`/`OFFSET` at the DB level.

### Proof of Concept

1. Register and verify two accounts (attacker + any other user).
2. Using any account, create 50,000 transactions in `WAITING_FOR_SIGNATURES` status (or wait for organic accumulation).
3. As the attacker, issue repeated requests:
   ```
   GET /transactions/sign?page=1&size=10
   Authorization: Bearer <attacker_jwt>
   ```
4. **Observed:** Each request causes the API process to load all 50,000 transaction rows into memory, deserialize each blob, and issue 50,000 sequential DB queries. Server memory climbs to OOM or response times exceed gateway timeouts, making the API unavailable to all users.
5. **Expected (correct behaviour):** The DB query should return at most `size=10` rows, matching the behaviour of `getTransactions` and `getHistoryTransactions`.
