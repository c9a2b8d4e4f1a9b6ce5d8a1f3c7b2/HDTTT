### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Enables Authenticated DoS via Resource Exhaustion

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction in the entire database with no `take` limit, then iterates over all of them in-memory calling the async `userKeysToSign` per transaction before applying pagination. This is the direct analog of the `MAX_DELEGATES` DoS: instead of iterating over 1024 delegates exhausting EVM gas, here iterating over an unbounded global transaction set exhausts server CPU, memory, and database connections. Any authenticated user can trigger this by issuing a single `GET /transactions/sign` request after the active transaction count has grown large.

### Finding Description
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `getTransactionsToSign` method issues a `repo.find()` call with **no `take` (LIMIT) clause**:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters out terminal statuses — no user scope
  relations: ['groupItem'],
  order,
  // ← no take / limit
});
```

The `whereForUser` condition only excludes five terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** filter by the requesting user's ownership, so the query returns every active transaction in the system. The result set is then iterated synchronously, calling the async `userKeysToSign` for each row:

```typescript
for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Pagination is applied only after the full in-memory loop completes:

```typescript
items: result.slice(offset, offset + limit),
```

The `PaginationParams` decorator caps `size` at 100, but this cap is never propagated to the database query — it only limits the final slice. The endpoint `GET /transactions/sign` is still exposed to all authenticated, verified users even though it is marked `/* NO LONGER USED BY FRONT-END */`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
A single authenticated HTTP request to `GET /transactions/sign?page=1&size=1` causes the server to:
1. Execute a full-table scan of the `transaction` table (no DB-level limit).
2. Deserialize every row plus its `groupItem` relation into memory.
3. Invoke `userKeysToSign` (an async, potentially mirror-node-querying operation) for every row sequentially.

As the active transaction count grows, each request consumes proportionally more CPU, heap memory, and database connection time. Repeated requests, or a single request against a large dataset, can saturate the Node.js event loop and the PostgreSQL connection pool, causing service unavailability for all users of the organization backend.

### Likelihood Explanation
- **Attacker precondition**: a valid JWT (any verified organization member). No admin role required.
- **Trigger**: `GET /transactions/sign` — a standard, documented REST endpoint still wired in the router.
- **Amplification**: transactions accumulate globally. A malicious user who creates many transactions (no per-user creation rate limit was found) inflates the scan for every subsequent caller. Even without a malicious actor, organic growth of the transaction table degrades this endpoint for all users.
- **Repeatability**: the attacker can issue the request in a tight loop; each call re-executes the full scan. [4](#0-3) 

### Recommendation
1. **Push pagination into the database query**: pass `take: limit` and `skip: offset` directly to `repo.find()` so the DB returns at most 100 rows per request, matching the existing `PaginationParams` cap.
2. **Scope the query to the requesting user**: add a user-scoped join condition (e.g., `signers: { userId: user.id }`) at the DB level so the scan is bounded by the user's own transactions, not the global active set.
3. **Remove or gate the deprecated endpoint**: since the controller already documents `/* NO LONGER USED BY FRONT-END */`, consider removing the route or protecting it with an additional guard to reduce the attack surface.

### Proof of Concept
1. Register and verify two organization accounts (attacker A, victim V).
2. As attacker A, call `POST /transactions` in a loop to create N transactions (e.g., N = 10,000) with status `WAITING_FOR_SIGNATURES`. No rate limit prevents this.
3. As victim V (or attacker A), issue:
   ```
   GET /transactions/sign?page=1&size=1
   Authorization: Bearer <jwt>
   ```
4. Observe: the server executes `SELECT * FROM transaction WHERE status NOT IN (...)` returning all N rows, then calls `userKeysToSign` N times before returning a single-item page. Response time grows linearly with N; at sufficient scale the Node.js process becomes unresponsive and subsequent requests from all users time out. [5](#0-4) [6](#0-5)
