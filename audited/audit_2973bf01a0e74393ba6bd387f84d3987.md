### Title
Unvalidated Signature Submission Corrupts Transaction State and Forces Premature Execution Failure

### Summary
The `uploadSignatureMaps` endpoint in `SignersService` records a user as a transaction signer and updates the stored transaction bytes without cryptographically verifying the submitted signature. A registered organization member can submit a signature map containing their valid public key paired with garbage signature bytes. The Hedera SDK's `addSignature` blindly appends those bytes, the public key is added to `_signerPublicKeys`, the transaction advances to `WAITING_FOR_EXECUTION`, and the chain service submits it to the Hedera network — where it is rejected with `INVALID_SIGNATURE` and permanently marked `FAILED`. The sister endpoint `importSignatures` correctly calls `validateSignature` before `addSignature`; the signer-upload path does not.

### Finding Description

**Vulnerability class:** State update without corresponding validated action (accounting / state-transition integrity).

**Root cause — missing signature validation in `processTransactionSignatures`:**

`SignersService.processTransactionSignatures` iterates the submitted `SignatureMap`, looks up each public key in the authenticated user's registered keys, and calls `sdkTransaction.addSignature(publicKey, map)` unconditionally: [1](#0-0) 

The Hedera SDK's `addSignature` appends whatever bytes are present in the map for that key — it performs no cryptographic verification. After the call, the public key is added to the SDK transaction's internal `_signerPublicKeys` set regardless of whether the signature is valid.

**Contrast with the validated path:**

`TransactionsService.importSignatures` calls `validateSignature` before `addSignature`: [2](#0-1) [3](#0-2) 

`validateSignature` calls `publicKey.verify(bodyBytes, signature)` and throws on failure: [4](#0-3) 

**State propagation after the invalid upload:**

`persistSignatureChanges` skips a transaction only when `isSameBytes && userKeys.length === 0`. Because `addSignature` mutated the bytes (appending the garbage signature), `isSameBytes` is `false`, so the corrupted bytes are written to the database and the user is inserted as a `TransactionSigner`: [5](#0-4) 

`processTransactionStatus` then checks `hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey)`. Because `_signerPublicKeys` now contains the attacker's key (added by `addSignature` without verification), the threshold check passes and the transaction is promoted to `WAITING_FOR_EXECUTION`: [6](#0-5) 

The chain service's `getValidatedSDKTransaction` also checks only key presence, not cryptographic validity: [7](#0-6) 

The transaction is submitted to Hedera, rejected with `INVALID_SIGNATURE`, and permanently set to `FAILED`.

### Impact Explanation

A malicious registered organization member who holds a public key required by a transaction can:

1. Submit a signature map with their registered public key but with arbitrary/garbage signature bytes via `POST /transactions/:id/signers`.
2. Cause the transaction bytes stored in the database to be corrupted with the invalid signature.
3. Advance the transaction to `WAITING_FOR_EXECUTION` without providing a real signature.
4. Force the chain service to submit the transaction to the Hedera network, where it is rejected.
5. Permanently mark the transaction as `FAILED` — it cannot be retried (`validateTransactionStatus` throws for `FAILED` status).

This is an irreversible sabotage of any transaction the attacker is a required signer for. For high-value governance or system-file transactions (fee schedule updates, node operations), this constitutes a critical integrity failure. [8](#0-7) 

### Likelihood Explanation

The attacker preconditions are minimal: the attacker must be a registered organization member with at least one public key registered in the system — a normal, non-privileged role. The endpoint `POST /transactions/:id/signers` is authenticated but not otherwise restricted. The attack requires a single crafted HTTP request with a valid public key DER string and arbitrary bytes as the signature value. No special tooling is needed; the `UploadSignatureMapDto` format is documented in the OpenAPI spec. [9](#0-8) 

### Recommendation

Add a call to `validateSignature` inside `processTransactionSignatures`, immediately before `sdkTransaction.addSignature`, mirroring the pattern already used in `importSignatures`:

```typescript
// In processTransactionSignatures, before addSignature:
const { data: validKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

This ensures that only cryptographically valid signatures can mutate transaction bytes or advance transaction state. [10](#0-9) 

### Proof of Concept

1. Register as an organization member with public key `PK_attacker` (e.g., an ED25519 key).
2. Obtain the ID of a transaction that requires `PK_attacker` as a signer.
3. Construct a `SignatureMap` with the correct node account ID, transaction ID, and `PK_attacker` as the public key, but with 64 bytes of `0x00` as the signature value.
4. `POST /transactions/{id}/signers` with `{ signatureMap: <crafted map> }`.
5. Observe: the server returns HTTP 201, the `TransactionSigner` row is inserted, and the transaction bytes in the database now contain the invalid signature.
6. If `PK_attacker` was the last required signer, observe the transaction status advance to `WAITING FOR EXECUTION`.
7. The chain service submits the transaction to Hedera; Hedera returns `INVALID_SIGNATURE`; the transaction is permanently set to `FAILED`.

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
