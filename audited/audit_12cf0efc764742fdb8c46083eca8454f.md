### Title
Missing Ownership Check on Imported Public Keys in `importSignatures` Allows Any Authorized User to Inject Signatures from Keys They Do Not Own

### Summary

The `importSignatures` endpoint in `transactions.service.ts` validates that submitted signatures are cryptographically valid but never verifies that the public keys producing those signatures belong to the authenticated user. This is the direct analog of the external "signatory not checked" report: the recovered/verified public key identity is never compared against the caller's registered keys.

### Finding Description

The codebase has two paths for submitting signatures to a transaction:

**Path 1 — `uploadSignatureMaps` (`signers.service.ts`)** correctly enforces ownership: [1](#0-0) 

It builds a `userKeyMap` exclusively from `user.keys`, then in `processTransactionSignatures` rejects any public key not found in that map: [2](#0-1) 

**Path 2 — `importSignatures` (`transactions.service.ts`)** does NOT enforce ownership: [3](#0-2) 

`validateSignature` returns every public key whose signature bytes are cryptographically valid for the transaction body: [4](#0-3) 

Those public keys are then added directly to the transaction bytes with no check that any of them belong to the calling user. The only gate before this point is `verifyAccess`, which checks whether the user has *any* relationship to the transaction (creator, observer, approver, or has keys to sign), not whether the submitted keys are theirs: [5](#0-4) 

The endpoint is publicly reachable by any verified JWT holder: [6](#0-5) 

### Impact Explanation

An authenticated user who has any access to a transaction (e.g., as an observer or a signer for one of their own keys) can call `POST /transactions/signatures/import` and supply a signature map containing cryptographically valid signatures produced by **other users' keys**. Because `validateSignature` only checks mathematical validity and `importSignatures` never cross-references the recovered public keys against `user.keys`, those foreign signatures are written directly into `transactionBytes`. This can:

- Advance a transaction's signing threshold using signatures the caller does not legitimately control for this workflow.
- Bypass the multi-signature approval process that the organization configured, since the transaction bytes are updated and `processTransactionStatus` will subsequently treat the transaction as sufficiently signed.
- Leave no `TransactionSigner` audit record (unlike `uploadSignatureMaps`), making the injection invisible in the signer log.

### Likelihood Explanation

The scenario is realistic in the context of this tool's own offline/multi-device signing workflow: transaction bytes are routinely shared between participants so they can sign locally. A participant who receives a copy of transaction bytes already signed by another party can extract those signatures and replay them through `importSignatures` against the live transaction, even if they were not the intended submitter of those signatures. The attacker only needs a valid JWT and any access relationship to the target transaction.

### Recommendation

Inside `importSignatures`, after `validateSignature` returns the list of valid public keys, cross-reference each key against the authenticated user's registered keys (mirroring the `userKeyMap` pattern in `uploadSignatureMaps`). Reject the request if any submitted public key does not belong to the caller:

```typescript
// After attachKeys(user, ...) or equivalent:
const userKeySet = new Set(user.keys.map(k => k.publicKey));

for (const publicKey of publicKeys) {
  const raw = publicKey.toStringRaw();
  const der = publicKey.toStringDer();
  if (!userKeySet.has(raw) && !userKeySet.has(der)) {
    throw new BadRequestException(ErrorCodes.PNY);
  }
  sdkTransaction.addSignature(publicKey, map);
}
```

### Proof of Concept

1. User A (observer on transaction T) and User B (required signer on transaction T) both have accounts.
2. User B signs transaction T offline and shares the signed transaction bytes with User A (normal workflow).
3. User A extracts User B's signature from the signed bytes and formats it as an `UploadSignatureMapDto` keyed under User B's public key.
4. User A calls `POST /transactions/signatures/import` with their own JWT and the crafted payload.
5. `verifyAccess` passes (User A is an observer).
6. `validateSignature` passes (the signature is mathematically valid for transaction T's body bytes).
7. User B's signature is written into `transactionBytes` with no `TransactionSigner` record created.
8. `processTransactionStatus` now sees the threshold met and advances the transaction to `WAITING_FOR_EXECUTION` — without User B having explicitly submitted their signature through the intended workflow.

```typescript
export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};
```
