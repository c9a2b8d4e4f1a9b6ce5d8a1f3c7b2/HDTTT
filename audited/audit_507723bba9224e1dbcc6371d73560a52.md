### Title
Missing Authorization Check in `removeTransactionApprover` Allows Any Authenticated User to Remove Approvers from Any Transaction

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` performs no ownership or identity verification before deleting an approver record. Every other mutation in the same service (`createTransactionApprovers`, `updateTransactionApprover`, `approveTransaction`) accepts a `User` parameter and calls `getCreatorsTransaction` or an equivalent guard. `removeTransactionApprover` accepts only an approver `id`, making it callable by any authenticated user against any transaction's approver tree.

### Finding Description

**Root cause — missing `user` parameter and ownership check:** [1](#0-0) 

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

The function signature takes only `id: number` — no `User` argument, no ownership assertion.

**Contrast with every other mutation in the same service:**

`createTransactionApprovers` calls `getCreatorsTransaction` before any write: [2](#0-1) 

`updateTransactionApprover` also calls `getCreatorsTransaction` inside its transaction block: [3](#0-2) 

`getCreatorsTransaction` enforces that `transaction.creatorKey?.userId === user.id`: [4](#0-3) 

**Exploit path:**
1. Attacker registers a normal user account (no privilege required).
2. Attacker learns or enumerates an approver `id` for a transaction they did not create (IDs are sequential integers).
3. Attacker calls `DELETE /transactions/{transactionId}/approvers/{id}` (or equivalent route).
4. The service calls `removeNode(approver.id)`, which recursively soft-deletes the entire approver subtree: [5](#0-4) 
5. The approver (and any nested threshold tree) is permanently removed from the transaction without the transaction creator's knowledge or consent.

### Impact Explanation
Removing approvers from a transaction directly undermines the multi-signature approval workflow. If a threshold approver tree is deleted, the transaction may advance to `WAITING_FOR_EXECUTION` or be executed without the required organizational sign-off. This constitutes an unauthorized state change that can bypass governance controls on high-value Hedera transactions. The `emitTransactionStatusUpdate` call after deletion means downstream services immediately act on the corrupted state. [6](#0-5) 

### Likelihood Explanation
Any registered user can reach this code path. Approver IDs are sequential database integers, making enumeration trivial. The attacker needs only a valid JWT (obtainable by registering a free account) and a guessed or observed approver ID. No admin key, leaked credential, or privileged role is required.

### Recommendation
Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction(approver.transactionId, user)` before invoking `removeNode`, exactly as `createTransactionApprovers` and `updateTransactionApprover` already do. This ensures only the transaction creator can delete approvers.

### Proof of Concept

**Preconditions:** Two accounts — `victim` (transaction creator) and `attacker` (any registered user).

1. `victim` creates a transaction and adds an approver tree (returns approver IDs, e.g., `id=42`).
2. `attacker` authenticates and sends:
   ```
   DELETE /transactions/7/approvers/42
   Authorization: Bearer <attacker_jwt>
   ```
3. `removeTransactionApprover(42)` is called. No user check occurs. `removeNode(42)` soft-deletes the entire approver subtree.
4. `victim`'s transaction now has no approvers; the approval gate is bypassed.
5. `emitTransactionStatusUpdate` fires, and the chain service may immediately advance the transaction to execution.

**Expected (correct) behavior:** HTTP 401/403 — only the transaction creator may remove approvers.
**Actual behavior:** HTTP 200 — approver deleted regardless of caller identity.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
