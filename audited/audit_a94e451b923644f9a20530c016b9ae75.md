### Title
`removeTransaction` Executes on Terminal-State Transactions, Enabling Audit Trail Corruption

### Summary
The `removeTransaction` function in `transactions.service.ts` performs no status check before soft-deleting a transaction. A transaction creator can invoke this endpoint against transactions already in terminal states (`EXECUTED`, `FAILED`, `EXPIRED`), overwriting their status to `CANCELED` and hiding them from all normal queries. For a governance tool used by the Hedera Council, this is a direct integrity failure in the state transition model.

### Finding Description

**Root cause — missing status guard in `removeTransaction`:**

`removeTransaction` delegates its authorization check entirely to `getTransactionForCreator`, which only verifies creator ownership: [1](#0-0) [2](#0-1) 

`getTransactionForCreator` checks only `creatorKey.userId === user.id` — no status validation is performed. The function then unconditionally:
1. Overwrites `status` → `CANCELED`
2. Calls `softRemove` (sets `deletedAt`)

**Contrast with other operations that do check status:**

`cancelTransaction` explicitly guards against non-cancelable states: [3](#0-2) [4](#0-3) 

`executeService.validateTransactionStatus` also explicitly rejects all terminal states before execution: [5](#0-4) 

`removeTransaction` has no equivalent guard, making it the only write path that silently accepts terminal-state transactions.

**Effect on query visibility:**

After soft-deletion, `getTransactionById` does not use `withDeleted: true`, so the record disappears from all normal lookups. The preference logic that favors non-inactive statuses further ensures the deleted record is never surfaced: [6](#0-5) 

**Defined terminal states that should be immutable:** [7](#0-6) 

`EXECUTED`, `FAILED`, and `EXPIRED` are all terminal. `removeTransaction` allows transitioning any of them to `CANCELED` + soft-deleted.

### Impact Explanation

- An `EXECUTED` transaction represents a Hedera network operation that has already been submitted and confirmed. Overwriting its status to `CANCELED` and hiding it from the database is a false representation of on-chain history.
- All other participants (observers, signers, approvers) lose visibility into a transaction they were party to, with no notification.
- For a governance tool used by the Hedera Council for multi-signature operations, audit trail integrity is a core security property. A creator can unilaterally erase evidence of any transaction they originated, regardless of its outcome.
- The soft-delete is not recoverable through any API endpoint — only direct database access can restore the record.

### Likelihood Explanation

- Attacker precondition: authenticated user who created at least one transaction. No elevated privileges required.
- Attack path: single authenticated `DELETE /transactions/{id}` API call after the transaction reaches a terminal state.
- The creator role is a standard user role reachable by any registered participant in the system.

    const transaction = await this.getTransactionForCreator(id, user);

    if (softRemove) {
      await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
      await this.repo.softRemove(transaction);
    } else {
      await this.repo.remove(transaction);
    }

    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{
        entityId: transaction.id,
        additionalData: {
          transactionId: transaction.transactionId,
          network: transaction.mirrorNetwork,
        },
      }],
    );

    return true;
  }

  private async validateTransactionStatus(transaction: Transaction) {
