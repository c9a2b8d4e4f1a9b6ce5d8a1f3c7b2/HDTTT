### Title
Unbounded In-Memory Iteration Over All Active Transactions in `getTransactionsToSign` Causes Authenticated DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions system-wide from the database without any row limit, then iterates over every row in application memory calling the expensive `userKeysToSign` (which deserializes transaction bytes and computes signature keys) for each one. Pagination is applied only after the full scan. As the number of active transactions grows, each call to this endpoint consumes proportionally more memory, CPU, and DB time, eventually causing request timeouts or OOM crashes that degrade service for all users.

### Finding Description

**Root cause — no `take` limit on the DB query:**

In `back-end/apps/api/src/transactions/transactions.service.ts` lines 295–309, `getTransactionsToSign` issues an unbounded `repo.find()`:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only excludes terminal statuses — no user scope, no row cap
  relations: ['groupItem'],
  order,
});                      // ← no `take` / LIMIT clause

for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}

return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),   // pagination applied in-memory, post-scan
  ...
};
``` [1](#0-0) 

The `whereForUser` filter only excludes terminal statuses (`EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, `ARCHIVED`). It does **not** scope to the requesting user's transactions at the database level: [2](#0-1) 

Contrast this with `getTransactions`, which correctly passes `skip: offset, take: limit` to the ORM so the database enforces pagination: [3](#0-2) 

**Per-row cost — `userKeysToSign` is expensive:**

For every row returned, `userKeysToSign` is called, which calls `computeSignatureKey` and deserializes raw transaction bytes via the Hedera SDK: [4](#0-3) 

This is O(N) SDK deserialization + O(N) key-tree traversal where N is the total number of active transactions in the system.

**Exploit path:**

1. Attacker registers as a normal user (no privilege required).
2. Attacker calls `POST /transactions` repeatedly, creating many transactions with future `validStart` values. These remain in `WAITING_FOR_SIGNATURES` status indefinitely until they expire.
3. Attacker (or any other authenticated user) calls `GET /transactions/to-sign`. The handler fetches every non-terminal transaction in the database, iterates over all of them, and only then slices the result for the requested page.
4. With enough accumulated transactions, the Node.js process exhausts heap memory or the DB query times out, crashing or hanging the API service for all users.

### Impact Explanation

A single authenticated API call to `GET /transactions/to-sign` can cause the API service to become unresponsive or crash due to unbounded memory allocation and CPU usage proportional to the total number of active transactions. Because the API is a shared NestJS service, a crash or sustained hang affects all concurrent users. The impact is **service unavailability / severe degradation** — a permanent DoS until the service is restarted and the transaction backlog is cleared.

### Likelihood Explanation

Any registered user can reach this endpoint. Transaction creation is unrestricted in volume per user. An attacker needs only a valid account and the ability to submit HTTP requests. No privileged access, leaked credentials, or external dependencies are required. The degradation also occurs organically as legitimate usage grows, meaning the vulnerability can be triggered without any malicious intent.

### Recommendation

Apply the database-level `take` limit **before** the in-memory loop, and push the user-eligibility filter into the SQL query (as done in `getTransactions` and `getTransactionsToApprove`). Specifically:

1. Add `take: limit` and `skip: offset` to the `repo.find()` call inside `getTransactionsToSign`, and add a SQL-level join/subquery to filter only transactions where the requesting user has a required signing key — mirroring the approver subquery pattern already used in `getTransactionsToApprove`.
2. Alternatively, rewrite the query using the `getTransactionNodesQuery` SQL builder (already used in `transaction-nodes.service.ts`) which supports user-scoped, paginated queries at the DB level.
3. Add a hard cap on the maximum `size` parameter accepted by `PaginationParams` to prevent a client from requesting an arbitrarily large page.

### Proof of Concept

**Setup:** Deploy the backend locally. Register user A.

**Step 1 — inflate the active-transaction table:**
```bash
for i in $(seq 1 5000); do
  curl -s -X POST http://localhost:3000/transactions \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"mirrorNetwork":"testnet","validStart":"2099-01-01T00:00:00Z",...}'
done
```

**Step 2 — trigger the unbounded scan:**
```bash
time curl -s "http://localhost:3000/transactions/to-sign?page=1&size=10" \
  -H "Authorization: Bearer $TOKEN"
```

**Expected outcome:** Response time grows linearly with the number of active transactions. At sufficient scale (tens of thousands of rows), the Node.js process exhausts heap memory (`JavaScript heap out of memory`) or the request times out, returning 500 to all concurrent users and potentially crashing the API pod.
