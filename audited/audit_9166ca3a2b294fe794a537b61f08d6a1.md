### Title
`MAX_USER_KEYS` Guard in `uploadKey` Fires Before Existence Check, Blocking Legitimate Updates and Allowing Limit Overflow via Concurrent Requests

### Summary

`uploadKey` in `back-end/apps/api/src/user-keys/user-keys.service.ts` enforces the `MAX_USER_KEYS` cap before it determines whether the incoming request would create a new key or merely update an already-registered one. This produces two symmetric defects that are the direct analog of the external AMM report: (1) a user who has reached the cap is rejected even when uploading a key they already own (e.g., to attach a mnemonic hash), because no new row would be inserted; and (2) two concurrent upload requests for distinct new keys, submitted when the count is exactly `MAX_USER_KEYS − 1`, both pass the guard and both insert, pushing the stored count above the intended maximum.

### Finding Description

**Root cause — guard fires before existence check** [1](#0-0) 

The execution order is:

1. `attachKeys` loads the user's current active keys into `user.keys`.
2. The guard `if (user.keys.length >= MAX_USER_KEYS)` throws immediately.
3. Only *after* that guard would the code call `this.repo.findOne({ where: { publicKey: dto.publicKey }, withDeleted: true })` to discover whether the key already exists.

Because step 2 precedes step 3, a user sitting at exactly `MAX_USER_KEYS` keys who submits a `POST /keys` request for a public key they already own (to set `mnemonicHash` or recover a soft-deleted entry) is rejected with `ErrorCodes.UMK` even though the operation would not insert a new row.

**Issue 1 — legitimate update blocked at the cap**

When `user.keys.length === MAX_USER_KEYS` and the submitted `publicKey` already exists in the database owned by the same user, the `if (userKey)` branch at line 47 would execute `Object.assign(userKey, dto)` and call `this.repo.save(userKey)` — no new key is created. The guard at line 37 prevents this path from ever being reached. [2](#0-1) [3](#0-2) 

**Issue 2 — concurrent requests exceed the cap**

There is no database-level transaction or row-level lock wrapping the read-then-write sequence. Two simultaneous `uploadKey` calls for two distinct new public keys, both arriving when `user.keys.length === MAX_USER_KEYS − 1`, will:

1. Both call `attachKeys` and observe `user.keys.length = MAX_USER_KEYS − 1`.
2. Both pass the guard (`MAX_USER_KEYS − 1 < MAX_USER_KEYS`).
3. Both call `findOne` and find no existing row.
4. Both call `this.repo.save(newKey)`.
5. The user ends up with `MAX_USER_KEYS + 1` keys, violating the invariant. [4](#0-3)

### Impact Explanation

**Issue 1**: Any user who has filled all `MAX_USER_KEYS` slots and subsequently needs to attach a mnemonic hash to an existing key (a normal workflow after importing a key without its derivation metadata) receives a permanent `400 Bad Request`. The key metadata cannot be updated through any other endpoint, creating an unrecoverable functional lock on that user's key management.

**Issue 2**: A user (or a script acting on behalf of a user) can exceed the per-user key cap by racing two upload requests. While the direct harm is limited to the user's own key count, it breaks the server-side invariant that `user.keys.length ≤ MAX_USER_KEYS`, which downstream logic (e.g., `HasKeyGuard`, key-count checks in other services) may rely on.

### Likelihood Explanation

Issue 1 is triggered by normal product usage: a user imports a public key without a mnemonic, fills the remaining slots, then tries to attach the mnemonic later. No attacker capability is required — any authenticated user at the limit hits this. Issue 2 requires two concurrent HTTP requests from the same authenticated session, which is trivially achievable with any HTTP client or browser tab duplication.

### Recommendation

Move the `MAX_USER_KEYS` guard to *after* the existence check, and only enforce it when the operation would actually insert a new row:

```typescript
async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
  await attachKeys(user, this.repo.manager);

  let userKey = await this.repo.findOne({
    where: { publicKey: dto.publicKey },
    withDeleted: true,
  });

  const isNewKey = !userKey || (userKey.userId !== user.id);

  // Only enforce the cap when a genuinely new key would be created
  if (isNewKey && user.keys.length >= MAX_USER_KEYS) {
    throw new BadRequestException(ErrorCodes.UMK);
  }
  // ... rest of logic
}
```

To close the race condition, wrap the read-check-insert sequence in a database transaction with a `SELECT ... FOR UPDATE` or use a unique constraint on `(userId, publicKey)` combined with an upsert, so concurrent inserts are serialised at the database level.

Add non-skipped tests asserting: (a) a user at the cap can still update an existing key's mnemonic hash, and (b) `user.keys.length` never exceeds `MAX_USER_KEYS` after any sequence of concurrent uploads.

### Proof of Concept

**Issue 1 — update blocked at cap**

1. Register a user and upload exactly `MAX_USER_KEYS` public keys (without mnemonic hashes).
2. `POST /users/{id}/keys` with `{ publicKey: <one of the already-uploaded keys>, mnemonicHash: "abc" }`.
3. Observe `400 Bad Request` with `ErrorCodes.UMK`, even though no new key would be created.

**Issue 2 — concurrent overflow**

1. Register a user and upload `MAX_USER_KEYS − 1` keys.
2. Fire two simultaneous `POST /users/{id}/keys` requests with two distinct new public keys (e.g., via `Promise.all`).
3. Both return `201 Created`.
4. `GET /users/{id}/keys` returns `MAX_USER_KEYS + 1` entries, violating the cap.

  async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
    await attachKeys(user, this.repo.manager);

    // Check if the user already has the maximum number of keys
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }

    // Find the userKey by the publicKey
    let userKey = await this.repo.findOne({
      where: { publicKey: dto.publicKey },
      withDeleted: true,
    });

    if (userKey) {
      // If the userKey found is owned by a different user,
      // or if the userKey has a non null hash or index that doesn't
      // match the hash or index provided
      // throw an error.
      if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
        throw new BadRequestException(ErrorCodes.PU);
      }
      // Set the hash and/or index (only if the current value is null)
      Object.assign(userKey, dto);
    } else {
      userKey = await this.repo.create(dto);
      userKey.user = user;
    }

    if (userKey.deletedAt) {
      await this.repo.recover(userKey);
    }
    return this.repo.save(userKey);
  }
