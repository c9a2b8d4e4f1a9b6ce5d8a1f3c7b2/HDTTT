### Title
Missing Caller Authorization in `removeTransactionApprover` — Any Authenticated User Can Remove Any Transaction Approver

### Summary
The `removeTransactionApprover` function in `ApproversService` performs no caller-identity check before deleting an approver node. Every sibling mutating function (`createTransactionApprovers`, `updateTransactionApprover`) enforces that only the transaction creator may act, via `getCreatorsTransaction`. `removeTransactionApprover` accepts no `user` parameter and calls no equivalent guard, so any authenticated user who knows an approver `id` can silently delete it from any transaction they did not create.

### Finding Description
`createTransactionApprovers` and `updateTransactionApprover` both accept a `User` argument and call `getCreatorsTransaction`, which throws `UnauthorizedException` if the caller is not the transaction creator. [1](#0-0) [2](#0-1) 

`removeTransactionApprover`, by contrast, accepts only `id: number`. It fetches the approver and immediately deletes the entire subtree rooted at that node — no `user` parameter, no `getCreatorsTransaction` call, no ownership check of any kind. [3](#0-2)

Because `removeTransactionApprover` never invokes this helper, the caller's identity is never verified against the transaction's `creatorKey.userId`.

### Impact Explanation
An attacker who is a legitimate member of the organization (any authenticated, verified user) can:
1. Observe a transaction they are an approver or signer of (they have read access via `getVerifiedApproversByTransactionId`).
2. Learn the numeric `id` of any approver node on that transaction.
3. Call `DELETE /transactions/:transactionId/approvers/:id` to remove that approver — or an entire approval subtree — from a transaction they do not own.

This breaks the multi-signature approval workflow: required approvers can be stripped, lowering or eliminating the approval threshold for high-value Hedera transactions without the creator's knowledge.

### Likelihood Explanation
The endpoint is reachable by any authenticated, verified user — no admin role is required. The approver IDs are integers returned in normal API responses visible to participants of a transaction. The attack requires no special tooling beyond a standard HTTP client.

### Recommendation
Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` (or an equivalent ownership check) before invoking `removeNode`, mirroring the pattern already used in `createTransactionApprovers` and `updateTransactionApprover`.

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve the root node to get the transactionId, then verify ownership
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root) throw new BadRequestException(ErrorCodes.RANF);
  await this.getCreatorsTransaction(root.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

The controller calling this function must also be updated to pass the authenticated `user` object.

### Proof of Concept

1. Alice creates a transaction and adds Bob as a required approver (approver record `id = 42`).
2. Carol, an unrelated authenticated user, sends:
   ```
   DELETE /transactions/7/approvers/42
   Authorization: Bearer <carol_jwt>
   ```
3. `removeTransactionApprover(42)` is called with no user check.
4. `removeNode(42)` soft-deletes the approver subtree rooted at id 42.
5. The transaction now has no required approvers; it can proceed to execution without Bob's signature. [3](#0-2)

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
