### Title
Users With `MAX_USER_KEYS` Active Keys Are Permanently Blocked From Updating Existing Keys via `uploadKey`

### Summary
In `back-end/apps/api/src/user-keys/user-keys.service.ts`, the `uploadKey` function enforces the `MAX_USER_KEYS` limit **before** determining whether the incoming request would add a new key or merely update an existing one. This mirrors the exact vulnerability class in the external report: a guard check placed before the reuse-detection logic permanently blocks a legitimate operation (updating an existing key's mnemonic hash) for any user who has already reached the key limit.

### Finding Description

**Root cause — check before reuse logic:**

In `uploadKey`, the execution order is:

1. `attachKeys(user, ...)` — loads the user's active keys into `user.keys` [1](#0-0) 
2. **Guard check** — throws if `user.keys.length >= MAX_USER_KEYS` [2](#0-1) 
3. **Reuse check** — queries the DB for an existing key with the same `publicKey` (including soft-deleted rows) [3](#0-2) 
4. If found and owned by the same user → **updates** the existing row (no new key created) [4](#0-3) 
5. If soft-deleted → **recovers** the existing row (no new key created) [5](#0-4) 

The guard at step 2 fires unconditionally, before the code ever reaches step 3–4. This means that when a user at the limit calls `POST /user/:userId/keys` with a `publicKey` they already own (the documented path for adding a mnemonic hash to an existing key), the request is rejected with `UMK` even though **no new database row would be created**.

**Confirmed normal workflow that triggers the bug:**

The e2e test suite explicitly documents this two-step pattern as the intended usage:

```
POST /keys  { publicKey }                        → 201 (key created, no mnemonic)
POST /keys  { publicKey, mnemonicHash, index }   → 201 (key updated with mnemonic)
``` [6](#0-5) 

A user who uploads all `MAX_USER_KEYS` keys in the first step and then tries to enrich them with mnemonic hashes in the second step will be blocked on every single second-step call.

**Analog to external report:**

| External report (`wstTAO.sol`) | This codebase (`user-keys.service.ts`) |
|---|---|
| `require(array.length < max)` before loop | `if (user.keys.length >= MAX_USER_KEYS) throw` before reuse check |
| Loop finds empty/reusable slots | `repo.findOne` finds existing owned key to update |
| Require blocks even when slots exist | Guard blocks even when no new key would be added |

### Impact Explanation

Any user who has reached `MAX_USER_KEYS` is permanently unable to add mnemonic hashes (or update indices) to their existing keys via the `POST /user/:userId/keys` endpoint. Since mnemonic hashes are used for key recovery and signing workflows, this degrades the user's ability to participate in multi-signature transaction flows. The only workaround is to delete a key first, which itself may break active signing obligations on in-flight transactions.

### Likelihood Explanation

This is a realistic, no-privilege scenario. A normal user who imports many hardware-wallet or derived keys (without mnemonic metadata) and later tries to annotate them hits this bug deterministically. No attacker involvement is required — the user triggers it through the documented two-step upload workflow. [7](#0-6) 

### Recommendation

Move the `MAX_USER_KEYS` guard to **after** the reuse check, and only enforce it on the code path that would actually insert a new row:

```typescript
async uploadKey(user: User, dto: UploadUserKeyDto): Promise<UserKey> {
  await attachKeys(user, this.repo.manager);

  let userKey = await this.repo.findOne({
    where: { publicKey: dto.publicKey },
    withDeleted: true,
  });

  if (userKey) {
    if (userKey.userId !== user.id || (userKey.index && userKey.index !== dto.index)) {
      throw new BadRequestException(ErrorCodes.PU);
    }
    Object.assign(userKey, dto);
  } else {
    // Only enforce the limit when a genuinely new key would be created
    if (user.keys.length >= MAX_USER_KEYS) {
      throw new BadRequestException(ErrorCodes.UMK);
    }
    userKey = await this.repo.create(dto);
    userKey.user = user;
  }

  if (userKey.deletedAt) {
    await this.repo.recover(userKey);
  }
  return this.repo.save(userKey);
}
```

This matches the fix recommended in the external report: place the guard immediately before the push/insert, not before the loop/reuse check.
