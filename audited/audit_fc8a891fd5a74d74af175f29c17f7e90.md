### Title
Duplicate `userId` Entries in `createTransactionObservers` Cause Inconsistent Observer State

### Summary
`ObserversService.createTransactionObservers` accepts an array of `userIds` and checks each one only against the **already-persisted** `transaction.observers` loaded at the start of the call. It performs no deduplication within the caller-supplied `dto.userIds` array itself. A transaction creator can submit `{ "userIds": [N, N, N] }` to insert multiple `TransactionObserver` rows for the same user on the same transaction, corrupting the observer list state. This is the direct TypeScript analog of the Solidity `MappedSinglyLinkedList.initialize` vulnerability described in the external report.

### Finding Description

**Root cause — `observers.service.ts` lines 49–54:**

```typescript
for (const userId of dto.userIds) {
  if (!transaction.observers.some(o => o.userId === userId)) {
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);
  }
}
```

`transaction.observers` is the snapshot loaded from the database at line 37–40 **before** the loop begins. The guard `!transaction.observers.some(o => o.userId === userId)` is evaluated against that static snapshot for every element of `dto.userIds`. If the same `userId` appears twice in the input array and is not already in the DB snapshot, both iterations pass the guard and two `TransactionObserver` objects are pushed into `observers`. All of them are then persisted in a single `repo.save(observers)` call at line 61. [1](#0-0) 

The DTO validation only enforces `@IsArray()`, `@IsNumber({}, { each: true })`, and `@ArrayMinSize(1)` — no `@ArrayUnique()` or equivalent constraint is applied. [2](#0-1) 

**Contrast with the approvers service**, which runs inside a `dataSource.transaction` block and calls `isNode()` using the same `transactionalEntityManager`, so a just-inserted duplicate is visible to the next iteration's count query and is correctly rejected. [3](#0-2) 

The observers path has no equivalent transactional guard.

**Exploit path:**

1. Attacker is the creator of transaction ID `T`.
2. Attacker sends:
   ```
   POST /transactions/T/observers
   { "userIds": [5, 5, 5] }
   ```
3. `transaction.observers` is loaded (userId 5 is absent).
4. All three iterations pass the guard; three `TransactionObserver` rows `{userId:5, transactionId:T}` are pushed.
5. `repo.save([...])` persists all three rows (assuming no DB-level unique constraint on `(userId, transactionId)`).
6. The observer list for transaction `T` now contains three duplicate entries for user 5.

### Impact Explanation

- **Inconsistent state:** The observer list for a transaction contains duplicate rows for the same user, violating the intended one-observer-per-user invariant.
- **Removal difficulty:** `removeTransactionObserver` operates by row `id`, not by `userId`. Each duplicate must be individually identified and deleted; a single removal call leaves the remaining duplicates intact.
- **UI/API corruption:** `getTransactionObserversByTransactionId` returns `transaction.observers` directly; callers receive a list with repeated entries for the same user, breaking any client-side deduplication assumptions.
- **Notification skew:** `emitTransactionUpdate` fires once regardless of how many duplicate rows were created, so downstream consumers receive an inconsistent view. [4](#0-3) 

### Likelihood Explanation

- **Attacker profile:** Any authenticated, verified user who is the creator of at least one transaction. No admin or privileged role is required.
- **Entry point:** `POST /transactions/:transactionId/observers` — a standard, documented API endpoint reachable over HTTPS.
- **Precondition:** The attacker must own the transaction (enforced at line 44). This is trivially satisfied by creating a transaction first.
- **Effort:** A single crafted HTTP request with a repeated `userId` in the array. [5](#0-4) 

### Recommendation

Deduplicate `dto.userIds` against itself **before** the loop, in addition to the existing check against persisted observers:

```typescript
const uniqueIncoming = [...new Set(dto.userIds)];

for (const userId of uniqueIncoming) {
  if (!transaction.observers.some(o => o.userId === userId)) {
    const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
    observers.push(observer);
  }
}
```

Additionally, add a `@ArrayUnique()` decorator (from `class-validator`) to `CreateTransactionObserversDto.userIds` to reject duplicate entries at the validation layer before they reach the service:

```typescript
import { ArrayMinSize, ArrayUnique, IsArray, IsNumber } from 'class-validator';

export class CreateTransactionObserversDto {
  @IsArray()
  @IsNumber({}, { each: true })
  @ArrayMinSize(1)
  @ArrayUnique()
  userIds: number[];
}
``` [2](#0-1) 

### Proof of Concept

**Preconditions:** Authenticated as user A (the transaction creator). Transaction ID `T` exists and was created by user A. User ID `5` is a valid user not yet an observer.

**Request:**
```
POST /transactions/T/observers
Authorization: Bearer <token_for_user_A>
Content-Type: application/json

{ "userIds": [5, 5, 5] }
```

**Expected (correct) behavior:** One `TransactionObserver` row created for userId 5.

**Actual behavior:** Three `TransactionObserver` rows created for userId 5 on transaction T, because the loop iterates over `[5, 5, 5]` and each iteration finds userId 5 absent from the pre-loaded `transaction.observers` snapshot.

**Verification:** `GET /transactions/T/observers` returns three entries with `userId: 5`, confirming the inconsistent state. [6](#0-5)
