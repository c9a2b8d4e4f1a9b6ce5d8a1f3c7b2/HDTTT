### Title
Unbounded Recursive Approver Tree Causes Server-Side DoS via Deeply Nested JSON Payload

### Summary
The `CreateTransactionApproverDto` class accepts a self-referential `approvers` field with no depth limit or array size cap. Any authenticated user can create a transaction and submit an arbitrarily deep nested approver tree to `POST /transactions/:id/approvers`. The NestJS pipeline (`class-transformer` + `class-validator`) and the `createApprover` service function all recurse into the tree without bound, enabling a single crafted HTTP request to exhaust server CPU, stack memory, or trigger a stack overflow — denying service to all other users.

### Finding Description

**Root cause — no depth or size constraint on the recursive DTO:**

`CreateTransactionApproverDto` is self-referential. The `approvers` field carries only `@ArrayMinSize(1)` (a lower bound) and no upper bound on either array length or nesting depth. [1](#0-0) 

The outer wrapper `CreateTransactionApproversArrayDto` also has no `@ArrayMaxSize` on `approversArray`. [2](#0-1) 

**Processing path — three recursive traversals per request:**

1. **`class-transformer`** (`@Type(() => CreateTransactionApproverDto)`) recursively instantiates every nested node during body deserialization.
2. **`class-validator`** (`@ValidateNested({ each: true })`) recursively validates every node.
3. **`createApprover`** inside `createTransactionApprovers` recursively processes each child node, issuing at least one database query per node (existence check, parent lookup, user count, insert). [3](#0-2) 

The spec confirms that multi-level nesting is an intended, exercised code path: [4](#0-3) 

**Attacker-controlled entry point:**

The endpoint is reachable by any authenticated, verified user. The only gate is `getCreatorsTransaction`, which checks that the caller created the target transaction — a condition the attacker trivially satisfies by creating their own transaction first. [5](#0-4) 

**Payload construction:**

A linear chain of depth N costs roughly 30 bytes per level in JSON. A 3,000-level chain fits in ~90 KB (within typical body-parser limits) and forces 3,000 recursive calls plus 3,000+ sequential database round-trips inside a single database transaction. A binary-branching tree of depth 20 forces 2^20 ≈ 1 M recursive calls and is still expressible in a few hundred KB.

### Impact Explanation

A single authenticated user can send one HTTP request that:
- Causes a JavaScript call-stack overflow (RangeError: Maximum call stack size exceeded) crashing the NestJS worker process, or
- Pins the event loop for seconds-to-minutes processing thousands of recursive DB queries inside a serializable transaction, blocking all other requests on the same connection pool.

Either outcome constitutes a full denial of service for all users of the API service. Because the attacker only needs a valid account (open registration), the attack is repeatable at negligible cost.

### Likelihood Explanation

- **Attacker preconditions:** register an account, verify email, create one transaction. All are normal product flows requiring no privilege.
- **Attack cost:** one HTTP POST with a crafted JSON body.
- **Detection difficulty:** the payload looks structurally valid; no anomaly-detection rule fires on a single well-formed request.
- **Repeatability:** the attacker can re-register or reuse the same account to repeat the attack after any recovery.

### Recommendation

1. **Add `@ArrayMaxSize(N)` to both array fields** in `CreateTransactionApproverDto` and `CreateTransactionApproversArrayDto`. A reasonable cap (e.g., 50) matches real-world approval structures.

```typescript
import { IsArray, IsNumber, IsOptional, ValidateNested,
         ArrayMinSize, ArrayMaxSize } from 'class-validator';

export class CreateTransactionApproverDto {
  // ...
  @IsArray()
  @ArrayMinSize(1)
  @ArrayMaxSize(50)          // ← add this
  @IsOptional()
  @ValidateNested({ each: true })
  @Type(() => CreateTransactionApproverDto)
  approvers?: CreateTransactionApproverDto[];
}

export class CreateTransactionApproversArrayDto {
  @IsArray()
  @ArrayMaxSize(50)          // ← add this
  @ValidateNested({ each: true })
  @Type(() => CreateTransactionApproverDto)
  approversArray: CreateTransactionApproverDto[];
}
```

2. **Enforce a maximum nesting depth** in the service before any DB work begins — reject payloads whose tree depth exceeds a fixed constant (e.g., 10).

3. **Apply a global request body size limit** in `main.ts` / `setup-app.ts` that is tighter than the default (e.g., 64 KB) to reduce the surface for all similar attacks.

### Proof of Concept

```
POST /transactions/<attacker-owned-tx-id>/approvers
Authorization: Bearer <valid-jwt>
Content-Type: application/json

{
  "approversArray": [
    {
      "threshold": 1,
      "approvers": [
        {
          "threshold": 1,
          "approvers": [
            {
              "threshold": 1,
              "approvers": [
                ... (repeat 3000 levels) ...
                { "userId": 1 }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

**Expected outcome:** The NestJS process either throws `RangeError: Maximum call stack size exceeded` during `class-transformer` deserialization / `class-validator` validation, or enters a multi-second recursive DB-query loop that starves the connection pool, returning 500 errors or timeouts to all concurrent users.
