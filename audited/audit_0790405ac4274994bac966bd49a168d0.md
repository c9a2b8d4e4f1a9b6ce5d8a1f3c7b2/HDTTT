### Title
`hasValidSignatureKey` Incorrectly Returns `true` When `signatureKey` Is an Empty `KeyList`, Bypassing Signature Validation

---

### Summary

`hasValidSignatureKey` in `back-end/libs/common/src/utils/sdk/key.ts` returns `true` when given an empty `KeyList` because the inner `forEach` loop is bypassed and the final comparison `0 >= (null || 0)` evaluates to `true`. `computeSignatureKey` in `TransactionSignatureService` can silently produce an empty `KeyList` when all account/node key lookups fail. Downstream callers — `processTransactionStatus` and `getValidatedSDKTransaction` — treat the `true` return as proof that sufficient signatures exist, allowing unsigned transactions to be promoted to `WAITING_FOR_EXECUTION` and submitted to the Hedera network.

---

### Finding Description

**Root cause — `hasValidSignatureKey`:**

```typescript
// back-end/libs/common/src/utils/sdk/key.ts
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();
    let currentThreshold = 0;

    // @audit when keys.length === 0, forEach is never entered
    keys.forEach(key => {
      if (hasValidSignatureKey(publicKeys, key)) {
        currentThreshold++;
      }
    });

    // 0 >= (undefined || 0)  =>  0 >= 0  =>  true
    return currentThreshold >= (key.threshold || keys.length);
  }
  ...
};
``` [1](#0-0) 

When `keys` is empty: `key.threshold` is `null`/`undefined`, `keys.length` is `0`, so the expression becomes `0 >= (null || 0)` → `0 >= 0` → **`true`**, regardless of whether any public key has actually been validated.

---

**How `signatureKey` becomes an empty `KeyList`:**

`computeSignatureKey` starts with `new KeyList()` and conditionally pushes keys into it. Every key-addition path silently swallows errors:

- `addFeePayerKey` wraps the mirror-node lookup in `try/catch` and returns without pushing if the lookup fails.
- `addSigningAccountKeys` and `addReceiverAccountKeys` do the same per-account.
- `addNodeKeys` returns early if `nodeInfo` or `nodeInfo.admin_key` is absent.
- `requirements.newKeys` may be an empty array. [2](#0-1) [3](#0-2) 

If the mirror node is unavailable (or returns no key for the fee-payer account), all `push` calls are skipped and `computeSignatureKey` returns an empty `KeyList`.

---

**Downstream impact — two call sites:**

**1. `processTransactionStatus`** (`back-end/libs/common/src/utils/transaction/index.ts`):

```typescript
const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],
  signatureKey          // ← may be empty KeyList
);

if (isAbleToSign) {
  // transaction is promoted to WAITING_FOR_EXECUTION
  newStatus = TransactionStatus.WAITING_FOR_EXECUTION;
}
``` [4](#0-3) 

An unsigned transaction is promoted to `WAITING_FOR_EXECUTION` without any real signature check.

**2. `getValidatedSDKTransaction`** (`back-end/libs/common/src/execute/execute.service.ts`):

```typescript
const signatureKey = await this.transactionSignatureService.computeSignatureKey(transaction);

if (!hasValidSignatureKey([...sdkTransaction._signerPublicKeys], signatureKey))
  throw new Error('Transaction has invalid signature.');
// ← never throws when signatureKey is empty KeyList
``` [5](#0-4) 

The guard that is supposed to block execution of improperly signed transactions is silently bypassed.

---

### Impact Explanation

An unsigned (or insufficiently signed) transaction can be submitted to the Hedera network. For high-value operations — account key rotations, node updates, large HBAR transfers — this means the network receives a transaction that should have been blocked. Depending on whether the Hedera network itself rejects it (it will if the on-chain key check fails), the worst case is a failed-but-submitted transaction that consumes fees and emits misleading status events; the best case is a silent no-op. Either way, the protocol's own signature-enforcement layer is completely defeated.

---

### Likelihood Explanation

The trigger condition — an empty `KeyList` from `computeSignatureKey` — is realistic:

- A transient mirror-node outage causes `addFeePayerKey` to throw, which is silently caught.
- Certain transaction types (e.g., a simple HBAR transfer where the receiver does not require a signature and there are no new keys) may produce a very small key list; if the fee-payer lookup also fails, the list is empty.
- The chain service's scheduler (`updateTransactions`) calls `processTransactionStatus` on a timer, so every polling cycle during a mirror-node degradation is a trigger opportunity. [6](#0-5) 

---

### Recommendation

Add an explicit empty-list guard at the top of `hasValidSignatureKey`:

```typescript
export const hasValidSignatureKey = (publicKeys: string[], key: Key) => {
  if (key instanceof KeyList) {
    const keys = key.toArray();

    // An empty KeyList means no signature requirements could be determined;
    // treat this as "not satisfied" rather than vacuously true.
    if (keys.length === 0) return false;

    let currentThreshold = 0;
    keys.forEach(k => {
      if (hasValidSignatureKey(publicKeys, k)) currentThreshold++;
    });
    return currentThreshold >= (key.threshold || keys.length);
  }
  ...
};
```

Additionally, `computeSignatureKey` should propagate or surface errors rather than silently swallowing them, so callers can distinguish "no keys required" from "key lookup failed."

---

### Proof of Concept

```typescript
import { KeyList } from '@hiero-ledger/sdk';
import { hasValidSignatureKey } from '../utils/sdk/key';

describe('hasValidSignatureKey empty KeyList', () => {
  it('incorrectly returns true for an empty KeyList with no public keys', () => {
    const emptyKeyList = new KeyList(); // no keys, no threshold

    // No public keys provided — user has signed nothing
    const result = hasValidSignatureKey([], emptyKeyList);

    // BUG: returns true; should return false
    expect(result).toBe(true);
  });
});
```

The empty `KeyList` is exactly what `computeSignatureKey` returns when all mirror-node lookups fail. `processTransactionStatus` then sets `isAbleToSign = true` and promotes the transaction to `WAITING_FOR_EXECUTION`, and `getValidatedSDKTransaction` passes the signature guard, allowing `executeTransaction` to submit the transaction to the Hedera network without valid signatures.
