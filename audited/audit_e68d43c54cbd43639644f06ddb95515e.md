### Title
Any Authenticated User Can Retrieve All Historical Transactions Belonging to Other Users

### Summary
The `GET /transactions/history` endpoint in `TransactionsController` does not pass the authenticated user to the underlying service method, unlike every other transaction-listing endpoint in the same controller. As a result, any verified, authenticated user can enumerate all executed, failed, and expired transactions across the entire organization, regardless of whether they are the creator, signer, approver, or observer of those transactions.

### Finding Description
In `back-end/apps/api/src/transactions/transactions.controller.ts`, the controller class is protected at the class level by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. All other transaction-listing endpoints that return user-scoped data correctly inject `@GetUser() user: User` and forward it to the service:

- `getTransactions` (line 121) calls `this.transactionsService.getTransactions(user, ...)` — user-scoped.
- `getTransactionsToSign` (line 167) calls `this.transactionsService.getTransactionsToSign(user, ...)` — user-scoped.
- `getTransactionsToApprove` (line 210) calls `this.transactionsService.getTransactionsToApprove(user, ...)` — user-scoped.

However, `getHistoryTransactions` (lines 143–154) omits the user entirely:

```ts
getHistoryTransactions(
  @PaginationParams() paginationParams: Pagination,
  @SortingParams(transactionProperties) sort?: Sorting[],
  @FilteringParams(...) filter?: Filtering[],
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getHistoryTransactions(paginationParams, filter, sort);
}
```

No `@GetUser()` parameter is declared, and no user argument is forwarded to the service. The service therefore has no basis to filter results by the requesting user and must return the full unfiltered history. [1](#0-0)

The asymmetry is the root cause: the same controller, the same guard stack, but one endpoint silently drops the user context.

### Impact Explanation
Any authenticated, verified user — including a low-privilege member who has never created or participated in any transaction — can call `GET /transactions/history` and receive paginated, filterable access to every executed, failed, and expired transaction in the organization. This exposes:

- Transaction bytes (which may encode account IDs, amounts, memo fields, and key structures).
- Creator identities, timestamps, and execution outcomes.
- Organizational financial activity in aggregate.

This is a cross-tenant data exposure with direct confidentiality impact on all organization members' transaction history.

### Likelihood Explanation
The attacker precondition is only a valid, verified account — the lowest privilege level the application grants. No admin role, no leaked credentials, and no special knowledge are required. The endpoint is reachable via a standard authenticated HTTP GET request. The filtering and pagination parameters are attacker-controlled, enabling targeted enumeration.

### Recommendation
Add `@GetUser() user: User` to `getHistoryTransactions` and forward it to `transactionsService.getHistoryTransactions`, then update the service to apply the same user-scoping logic used by `getTransactions`. If the history endpoint is intentionally organization-wide (e.g., for audit purposes), restrict it with `@UseGuards(AdminGuard)` and document that intent explicitly.

```ts
// Corrected controller method
@Get('/history')
@Serialize(withPaginatedResponse(TransactionDto))
getHistoryTransactions(
  @GetUser() user: User,           // <-- add this
  @PaginationParams() paginationParams: Pagination,
  @SortingParams(transactionProperties) sort?: Sorting[],
  @FilteringParams({...}) filter?: Filtering[],
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getHistoryTransactions(user, paginationParams, filter, sort);
}
```

### Proof of Concept
1. Register two accounts (User A and User B) on the organization back-end.
2. As User A (admin or creator), create and execute several transactions.
3. Log in as User B (a plain verified user with no relation to those transactions).
4. Send: `GET /transactions/history` with User B's JWT in the `Authorization` header.
5. Observe that the response contains all of User A's executed/failed/expired transactions, including their full transaction details.
6. Contrast with `GET /transactions` (same credentials), which returns only transactions User B is involved in — confirming the missing user-scoping is specific to the history endpoint. [3](#0-2) [1](#0-0)
