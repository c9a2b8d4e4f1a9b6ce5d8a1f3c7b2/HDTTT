### Title
Missing Cryptographic Signature Verification in `uploadSignatureMaps` Allows False Signer Registration and Transaction Corruption

### Summary
The `uploadSignatureMaps` path in `signers.service.ts` accepts and persists signature bytes from a user-controlled `SignatureMap` without ever verifying that those bytes are a valid cryptographic signature over the transaction body. Any authenticated user with a registered public key can submit a `SignatureMap` containing their real public key paired with arbitrary/garbage signature bytes. The system records them as a legitimate signer and overwrites the stored transaction bytes with the invalid signature, permanently corrupting the transaction and causing it to fail when submitted to Hedera.

### Finding Description

There are two distinct code paths for uploading signatures:

**Path A — `importSignatures` in `transactions.service.ts` (SAFE):** [1](#0-0) 

`validateSignature` explicitly calls `publicKey.verify(bodyBytes, signature)` and throws `'Invalid signature'` if the bytes do not verify. Only after this check does the caller invoke `addSignature`. [2](#0-1) 

**Path B — `uploadSignatureMaps` → `processTransactionSignatures` in `signers.service.ts` (VULNERABLE):** [3](#0-2) 

The only check performed is whether the public key belongs to the authenticated user (`userKeyMap.get(raw)`). There is no call to `publicKey.verify()` on the signature bytes. After confirming key ownership, `sdkTransaction.addSignature(publicKey, map)` is called unconditionally with the attacker-supplied bytes, and the user is immediately pushed into `userKeys` (new signers list).

The `IsSignatureMap` decorator validates structural format (valid account IDs, non-empty byte arrays) but performs no cryptographic verification: [4](#0-3) 

The Hedera SDK's `addSignature` does not verify signatures — it blindly appends the provided bytes to the transaction's signature map. The `isSameBytes` comparison only detects whether bytes changed, not whether they are valid: [5](#0-4) 

### Impact Explanation

1. **False signer registration**: A `TransactionSigner` database record is created for the attacker, falsely attesting they provided a valid signature. This breaks the audit-log integrity guarantee of the system.
2. **Transaction byte corruption**: `transaction.transactionBytes` is overwritten with bytes containing the invalid signature. Once persisted, the stored transaction is permanently corrupted.
3. **Permanent transaction failure / DoS**: When the chain service submits the corrupted transaction to Hedera, the network rejects it with `INVALID_SIGNATURE`. The transaction cannot be recovered because the unsigned bytes are overwritten. For threshold-key transactions, this permanently locks funds or blocks the intended operation. [6](#0-5) 

### Likelihood Explanation

The attacker only needs to be an authenticated user with at least one registered public key — the lowest privilege level in the system. No admin access, leaked credentials, or physical access is required. The `/signers` endpoint is a normal product workflow. The attacker constructs a `SignatureMap` with their real public key and replaces the signature bytes with 64 bytes of zeros or random data. The `IsSignatureMap` decorator accepts this (non-empty bytes pass the length check). The attack is deterministic and reproducible.

### Recommendation

Apply the same `publicKey.verify(bodyBytes, signature)` check used in `validateSignature` inside `processTransactionSignatures`, before calling `addSignature`. Specifically, for each `(publicKey, signature)` pair extracted from the map, verify the signature against the corresponding `bodyBytes` from `_signedTransactions` and throw if invalid — mirroring the logic already present in `validateSignature`: [7](#0-6) 

### Proof of Concept

1. Register as a normal user and add a public key (e.g., ED25519 key pair `K`).
2. Obtain a transaction ID `T` that is in `WAITING_FOR_SIGNATURES` status and requires a signature from `K`.
3. Construct a `SignatureMap` JSON body with the correct node account ID, transaction ID, and the DER-encoded public key of `K`, but set the signature value to `"0000...0000"` (64 zero bytes hex-encoded).
4. POST to `/<T>/signers` with this body using the authenticated session.
5. The `IsSignatureMap` decorator accepts the payload (non-empty bytes). `processTransactionSignatures` finds `K` in `userKeyMap`, calls `addSignature` with the zero bytes, and pushes the user into `userKeys`.
6. `persistSignatureChanges` writes a `TransactionSigner` row for the user and overwrites `transaction.transactionBytes` with the corrupted bytes.
7. Observe: the database now records the user as a signer; the stored transaction bytes contain an invalid signature. When the chain service later submits the transaction to Hedera, it receives `INVALID_SIGNATURE` and the transaction permanently fails.

export const validateSignature = (transaction: SDKTransaction, signatureMap: SignatureMap) => {
  const signerPublicKeys: PublicKey[] = [];

  const { rowLength, nodeAccountIdRow, transactionIdCol } =
    getSignedTransactionsDimensions(transaction);

  for (const [nodeAccountId, transactionIds] of signatureMap._map) {
    for (const [transactionId, publicKeys] of transactionIds._map) {
      for (const [publicKeyDer, signature] of publicKeys._map) {
        const publicKey = PublicKey.fromString(publicKeyDer);
        const publicKeyHex = publicKey.toStringRaw();

        const alreadySigned =
          transaction._signerPublicKeys.has(publicKeyHex) ||
          transaction._signerPublicKeys.has(publicKeyDer);

        if (!alreadySigned) {
          const row = nodeAccountIdRow[nodeAccountId];
          const col = transactionIdCol[transactionId];

          const bodyBytes = transaction._signedTransactions.get(col * rowLength + row).bodyBytes;

          const signatureValid = publicKey.verify(bodyBytes, signature);

          if (signatureValid) {
            signerPublicKeys.push(publicKey);
          } else {
            throw new Error('Invalid signature');
          }
        }
      }
    }
  }

  return signerPublicKeys;
};

  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
