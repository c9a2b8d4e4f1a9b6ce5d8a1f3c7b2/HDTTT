### Title
Unbounded Full-Table Scan in `getTransactionsToSign` Causes Memory and CPU Exhaustion for Any Authenticated User

### Summary

`TransactionsService::getTransactionsToSign()` fetches every non-terminal transaction from the database in a single query with no `take`/`skip` limit, then iterates over the entire result set in application memory — calling `userKeysToSign()` (which itself issues additional DB queries) for each row — before finally slicing the result for pagination. Any authenticated user can trigger this endpoint. As the transaction table grows, a single request will consume unbounded memory and CPU, degrading or crashing the API service.

### Finding Description

**Root cause — missing database-level pagination before in-memory iteration:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues an unbounded `find()` with no `take` or `skip`: [1](#0-0) 

All returned rows are then iterated in a `for` loop, with `userKeysToSign()` called per transaction: [2](#0-1) 

Pagination is applied only after the full scan, in memory: [3](#0-2) 

The endpoint is exposed at `GET /transactions/sign` and is reachable by any verified, authenticated user: [4](#0-3) 

The controller is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — no admin or special role is required: [5](#0-4) 

**Secondary unbounded query — `getUsers`:**

`UsersService::getUsers()` also issues an unbounded `find()` with no pagination, returning the entire user table to any authenticated caller: [6](#0-5) 

### Impact Explanation

For `getTransactionsToSign`: with N non-terminal transactions in the database, a single HTTP request causes the Node.js process to:
1. Load all N transaction rows into heap memory.
2. Execute up to N additional database round-trips via `userKeysToSign()`.

At scale this exhausts heap memory (OOM kill) or saturates the DB connection pool, making the API unavailable for all users. A single authenticated user issuing repeated requests can sustain the degradation. The effect is permanent service unavailability until the transaction table is pruned or the service is restarted.

### Likelihood Explanation

- **Attacker precondition:** valid account (any registered, verified user). Registration is a normal product flow.
- **Attack complexity:** a single `GET /transactions/sign` request with default pagination parameters is sufficient; no special tooling required.
- **Natural growth:** even without a malicious actor, a production deployment accumulates transactions over time, making this a reliability issue that degrades organically.

### Recommendation

Apply database-level `take`/`skip` before the in-memory loop. The correct pattern (already used in `getTransactions` and `getHistoryTransactions`) is:

```typescript
// Pass take/limit and skip/offset directly to the DB query
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit,   // ← add this
  skip: offset,  // ← add this
});
```

Because `userKeysToSign` requires per-row async work that cannot be pushed into SQL, the total count must be obtained via a separate `count()` query scoped to the same `where` clause, and the in-memory loop operates only on the already-paginated page.

For `getUsers`, add a `PaginationParams`-based `take`/`skip` or restrict the endpoint to admin-only callers.

### Proof of Concept

1. Register and verify a user account via the normal sign-up flow.
2. Ensure the database contains a large number of non-terminal transactions (e.g., created by other users or via the bulk-create endpoint).
3. Issue:
   ```
   GET /transactions/sign?page=1&limit=10
   Authorization: Bearer <valid_jwt>
   ```
4. Observe: the API process loads all non-terminal transactions into memory and calls `userKeysToSign` for each before returning 10 results. With tens of thousands of transactions, the request times out or the process OOMs. Repeated requests sustain the degradation.
