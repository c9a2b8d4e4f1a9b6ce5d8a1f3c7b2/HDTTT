### Title
Lost-Update Race Condition in `SignersService.uploadSignatureMaps` Silently Discards Signatures

### Summary
`SignersService.uploadSignatureMaps()` reads `transactionBytes` into an in-memory snapshot outside any database transaction, processes signatures, then blindly overwrites the stored bytes with no optimistic or pessimistic locking. Two concurrent HTTP requests for the same transaction will each read the same pre-signature bytes, independently append their own signature, and the second write will silently overwrite the first — permanently losing a signer's contribution.

### Finding Description
The vulnerability class is **TOCTOU / lost-update** (state-transition category), directly analogous to the beacon-kit `RangeDB.Set()` race where concurrent goroutines wrote to shared state without synchronization.

The three-phase flow in `uploadSignatureMaps` is:

**Phase 1 — Snapshot read (outside DB transaction)**

`loadTransactionData()` reads the current `transactionBytes` and existing signers into memory. [1](#0-0) 

**Phase 2 — Concurrent in-memory processing**

`validateAndProcessSignatures()` fans out with `Promise.all()`, each branch operating on the same in-memory snapshot. [2](#0-1) 

**Phase 3 — Blind overwrite**

`bulkUpdateTransactions()` issues an unconditional `UPDATE … SET "transactionBytes" = …` with no version check, no `WHERE "transactionBytes" = $old`, and no `SELECT FOR UPDATE` guard. [3](#0-2) 

**Race sequence for two concurrent requests (User A and User B signing the same transaction):**

```
Request A                          Request B
─────────────────────────────────────────────────────
loadTransactionData → bytes = B0   loadTransactionData → bytes = B0
addSignature(S_A)  → bytes = B1
                                   addSignature(S_B)  → bytes = B2
bulkUpdateTransactions(B1)
                                   bulkUpdateTransactions(B2)   ← overwrites B1
```

Final DB state: `B2 = B0 + S_A_missing + S_B`. Signature `S_A` is permanently lost.

No `@MurLock` decorator protects `uploadSignatureMaps`, in contrast to `executeTransaction` and `executeTransactionGroup` which are explicitly guarded. [4](#0-3) [5](#0-4) 

### Impact Explanation
Lost signatures corrupt the transaction's accumulated signature set. For threshold-key or `KeyList` transactions — the primary use case of this tool — a missing signature means the threshold is never reached. The transaction will remain stuck in `WAITING_FOR_SIGNATURES` indefinitely, or if it advances to `WAITING_FOR_EXECUTION`, it will be rejected on-chain with an `INVALID_SIGNATURE` error. There is no automatic recovery: the lost signer must re-submit, but if the transaction's `validStart` window has expired by then, the transaction is unrecoverable.

### Likelihood Explanation
The `POST /transactions/:transactionId/signers` endpoint is the normal path for every organization signer. Multi-signature workflows are the core purpose of the tool; concurrent signing by multiple users is the expected, documented workflow. The race window spans the entire async signature-processing time between `loadTransactionData` and the DB write — easily tens to hundreds of milliseconds. No special attacker capability is required; two legitimate signers acting simultaneously trigger the condition. [6](#0-5) 

### Recommendation
Apply a `@MurLock` (Redis distributed lock) keyed on `transaction.id` to `uploadSignatureMaps`, matching the pattern already used for `executeTransaction`. Alternatively, move the `loadTransactionData` read inside the DB transaction and add a `SELECT … FOR UPDATE` on the transaction row so the snapshot and the write are atomic. A third option is to replace the blind `UPDATE` with a compare-and-swap: `WHERE id = $id AND "transactionBytes" = $old_bytes`, retrying on conflict.

The Redis `MurLock` infrastructure is already present and configured in the codebase. [7](#0-6) 

### Proof of Concept

Two authenticated organization users submit signatures for the same transaction simultaneously:

```
# User A (signer 1)
POST /transactions/1/signers
Authorization: Bearer <token_A>
Body: [{ "id": 1, "signatureMap": <sig_map_A> }]

# User B (signer 2) — concurrent
POST /transactions/1/signers
Authorization: Bearer <token_B>
Body: [{ "id": 1, "signatureMap": <sig_map_B> }]
```

Both requests pass `validateTransactionStatus` (both see `WAITING_FOR_SIGNATURES`). Both compute new bytes from the same `B0` snapshot. The second `bulkUpdateTransactions` call overwrites the first. The `TransactionSigner` row for User A may be inserted (if the unique constraint allows it), creating an inconsistency where the DB records User A as a signer but the stored `transactionBytes` contain no trace of their signature. [8](#0-7) [9](#0-8)

  private async persistSignatureChanges(
    validationResults: any[],
    user: User,
  ) {
    const signers = new Set<TransactionSigner>();
    let notificationsToDismiss: number[] = [];

    // Prepare batched operations
    const transactionsToUpdate: { id: number; transactionBytes: Buffer }[] = [];
    const notificationsToUpdate: { userId: number; transactionId: number }[] = [];
    const signersToInsert: { userId: number; transactionId: number; userKeyId: number }[] = [];
    const transactionsToProcess: { id: number; transaction: Transaction }[] = [];

    for (const result of validationResults) {
      if (result.error) {
        console.error(`[TX ${result.id}] Validation failed: ${result.error}`);
        continue;
      }

      const { id, transaction, sdkTransaction, userKeys, isSameBytes } = result;

      // Skip if nothing to do - no signatures were added to the transaction
      // AND no new signers were inserted (the signature can be present on the transaction
      // if collated by an outside or 'offline' method)
      if (isSameBytes && userKeys.length === 0) continue;

      // Collect updates
      if (!isSameBytes) {
        transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
        transactionsToUpdate.push({ id, transactionBytes: transaction.transactionBytes });
      }

      // Collect inserts
      if (userKeys.length > 0) {
        const newSigners = userKeys.map(userKey => ({
          userId: user.id,
          transactionId: id,
          userKeyId: userKey.id,
        }));
        signersToInsert.push(...newSigners);
      }

      transactionsToProcess.push({ id, transaction });
      notificationsToUpdate.push({ userId: user.id, transactionId: transaction.id });
    }

    // Execute in single transaction
    try {
      await this.dataSource.transaction(async manager => {
        // Bulk update transactions
        if (transactionsToUpdate.length > 0) {
          await this.bulkUpdateTransactions(manager, transactionsToUpdate);
        }

        // Bulk update notifications
        if (notificationsToUpdate.length > 0) {
          const updatedNotificationReceivers = await this.bulkUpdateNotificationReceivers(manager, notificationsToUpdate);

          // To maintain backwards compatibility and multi-machine support, we send off a dismiss event.
          emitDismissedNotifications(
            this.notificationsPublisher,
            updatedNotificationReceivers,
          );

          notificationsToDismiss = updatedNotificationReceivers.map(nr => nr.id);
        }

        // Bulk insert signers
        if (signersToInsert.length > 0) {
          const results = await this.bulkInsertSigners(manager, signersToInsert);
          results.forEach(signer => signers.add(signer));
        }
      });
    } catch (err) {
      console.error('Database transaction failed:', err);
      throw new BadRequestException(ErrorCodes.FST);
    }

    return {
      transactionsToProcess,
      signers,
      notificationsToDismiss,
    };
  }

  private async bulkUpdateTransactions(
    manager: any,
    transactionsToUpdate: { id: number; transactionBytes: Buffer }[]
  ) {
    const whenClauses = transactionsToUpdate
      .map((t, index) => `WHEN ${t.id} THEN $${index + 1}::bytea`)
      .join(' ');

    const ids = transactionsToUpdate.map(t => t.id);
    const bytes = transactionsToUpdate.map(t => t.transactionBytes);

    await manager.query(
      `UPDATE transaction
     SET "transactionBytes" = CASE id ${whenClauses} END,
         "updatedAt" = NOW()
     WHERE id = ANY($${bytes.length + 1})`,
      [...bytes, ids]
    );
  }
