### Title
Creator Can Delete Transactions in Terminal States, Bypassing Status-Transition Protections

### Summary
The `removeTransaction` function (exposed via `DELETE /transactions/:id`) allows a transaction creator to soft-delete and forcibly set the status of a transaction to `CANCELED` regardless of its current status — including terminal states such as `EXECUTED`, `FAILED`, `EXPIRED`, `ARCHIVED`, and `REJECTED`. The dedicated `cancelTransaction` endpoint explicitly enforces a `cancelableStatuses` guard that blocks this, but `removeTransaction` has no equivalent check, creating a bypass path that corrupts the off-chain audit trail.

### Finding Description

**Root cause — missing status guard in `removeTransaction`:**

`cancelTransaction` → `cancelTransactionWithOutcome` enforces:

```typescript
private readonly cancelableStatuses = [
  TransactionStatus.NEW,
  TransactionStatus.WAITING_FOR_SIGNATURES,
  TransactionStatus.WAITING_FOR_EXECUTION,
];
// ...
if (!this.cancelableStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
``` [1](#0-0) [2](#0-1) 

`removeTransaction`, called by `DELETE /transactions/:id`, performs no such check:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user); // only checks ownership
  if (softRemove) {
    await this.repo.update(transaction.id, { status: TransactionStatus.CANCELED });
    await this.repo.softRemove(transaction);
  } else {
    await this.repo.remove(transaction);
  }
  ...
}
``` [3](#0-2) 

The controller wires this directly to an authenticated endpoint with no additional guard:

```typescript
@Delete('/:id')
deleteTransaction(@GetUser() user, @Param('id', ParseIntPipe) id: number): Promise<boolean> {
  return this.transactionsService.removeTransaction(id, user, true);
}
``` [4](#0-3) 

**Exploit path:**
1. Attacker (normal authenticated user) creates a transaction — becomes its creator.
2. Transaction progresses through signing and is executed on-chain (`EXECUTED` status).
3. Attacker calls `DELETE /transactions/:id`.
4. `getTransactionForCreator` passes (attacker is the creator).
5. No status check is performed.
6. The record is updated to `CANCELED` and soft-deleted (`deletedAt` set).
7. The transaction disappears from all standard queries for observers, signers, and approvers.

### Impact Explanation

The off-chain organizational record of an already-executed Hedera transaction is permanently corrupted. Observers, signers, and approvers who participated in the transaction lose visibility into its final state. The audit trail — which is the primary value of the organizational back-end — is silently falsified: an `EXECUTED` transaction is relabeled `CANCELED` and hidden. Any downstream logic (notifications, reporting, compliance checks) that relies on terminal-state records is affected. The on-chain Hedera transaction is immutable, but the tool's own integrity model is broken.

### Likelihood Explanation

Any authenticated, verified user who has ever created a transaction can trigger this. No elevated privileges are required — only a valid JWT and ownership of the transaction. The endpoint is a standard REST `DELETE` call. The attacker does not need to race any window; they can call it at any time after execution.

### Recommendation

Add the same `cancelableStatuses` guard to `removeTransaction` before performing any state mutation:

```typescript
async removeTransaction(id: number, user: User, softRemove: boolean = true): Promise<boolean> {
  const transaction = await this.getTransactionForCreator(id, user);

  if (!this.cancelableStatuses.includes(transaction.status)) {
    throw new BadRequestException(ErrorCodes.OTIP);
  }
  // ... rest of function
}
``` [3](#0-2) 

Alternatively, if deletion of terminal-state records is intentional (e.g., for cleanup), it should be a separate, explicitly scoped endpoint with its own authorization model, not silently reachable through the same `DELETE` path used for pre-execution cancellation.

### Proof of Concept

1. Authenticate as a normal user and obtain a JWT.
2. `POST /transactions` — create a transaction. Note the returned `id`.
3. Allow the transaction to reach `EXECUTED` status (via signing and chain execution, or by directly updating the DB row in a test environment).
4. `DELETE /transactions/{id}` with the user's JWT.
5. **Expected (correct) behavior**: `400 Bad Request` with `ErrorCodes.OTIP`, matching the behavior of `PATCH /transactions/cancel/{id}` for the same status.
6. **Actual behavior**: `200 OK`. The transaction row now has `status = CANCELED` and `deletedAt` set. It no longer appears in any standard query for observers or signers. The audit trail shows the transaction was canceled, not executed. [5](#0-4) [6](#0-5)
