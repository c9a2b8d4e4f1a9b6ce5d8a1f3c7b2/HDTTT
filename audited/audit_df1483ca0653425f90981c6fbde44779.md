### Title
Cross-Transaction Approver Deletion via Mismatched Authorization Key

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller is the creator of `transactionId`, but then deletes the approver record identified by `id` without verifying that approver belongs to `transactionId`. Any authenticated user who owns at least one transaction can delete approvers from any other user's transaction by supplying their own `transactionId` for the auth check and a victim's approver `id` for the deletion.

### Finding Description

**Vulnerability class:** Authorization bypass via wrong-key state mutation (direct analog of the external report).

**Root cause — controller (`approvers.controller.ts` lines 102–113):**

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← auth uses transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletion uses id (no cross-check)
  return true;
}
``` [1](#0-0) 

The authorization gate (`getCreatorsTransaction`) confirms the caller owns `transactionId`. The deletion call (`removeTransactionApprover(id)`) then operates on a completely independent approver record identified only by `id`.

**Root cause — service (`approvers.service.ts` lines 533–544):**

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // ← deletes by id, no transactionId check
  ...
}
``` [2](#0-1) 

`removeTransactionApprover` never verifies `approver.transactionId === transactionId`. The `removeNode` function then recursively soft-deletes the entire approver subtree rooted at `id`. [3](#0-2) 

**Exploit path:**
1. Attacker registers and creates Transaction T_attacker (becomes its creator).
2. Victim creates Transaction T_victim with a required approval tree (approver IDs are sequential auto-increment integers; they are also returned by `GET /transactions/:transactionId/approvers` which any authenticated user can call for any transaction they can observe).
3. Attacker sends:
   ```
   DELETE /transactions/{T_attacker}/approvers/{approver_id_from_T_victim}
   ```
4. `getCreatorsTransaction(T_attacker, attacker)` passes — attacker owns T_attacker.
5. `removeTransactionApprover(approver_id_from_T_victim)` executes — deletes the victim's approver (and its entire subtree) with no further check.

### Impact Explanation

- An attacker can silently remove required approvers from any transaction they do not own, bypassing the multi-signature approval workflow.
- `removeNode` is recursive, so a single call can wipe an entire approval tree, not just a leaf node.
- A transaction that previously required N-of-M approvals can be reduced to 0 required approvals, allowing it to proceed to execution without any authorization.
- This is a critical integrity failure in the transaction approval trust model.

### Likelihood Explanation

- Precondition: attacker must be a registered, verified user who has created at least one transaction — a normal product workflow, no privilege required.
- Victim approver IDs are sequential integers discoverable by calling `GET /transactions/:transactionId/approvers` on any transaction the attacker can observe, or by brute-forcing small integers.
- The attack requires a single crafted HTTP DELETE request. No race condition, no complex setup.

### Recommendation

Add a cross-check in the controller (or inside `removeTransactionApprover`) that the approver being deleted actually belongs to the authorized `transactionId`:

```typescript
// In the controller, after getCreatorsTransaction passes:
const approver = await this.approversService.getTransactionApproverById(id);
if (!approver || approver.transactionId !== transactionId) {
  throw new BadRequestException(ErrorCodes.ANF);
}
await this.approversService.removeTransactionApprover(id);
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and enforce the check there, mirroring the pattern already used in `updateTransactionApprover` which does verify `rootNode.transactionId !== transactionId` before mutating state. [4](#0-3) 

### Proof of Concept

**Setup:**
- User A (attacker) registers, creates Transaction T1 (id=1). T1 now has no approvers.
- User B (victim) registers, creates Transaction T2 (id=2) and adds an approver (approver id=5, userId=B).

**Attack:**
```http
DELETE /transactions/1/approvers/5
Authorization: Bearer <attacker_jwt>
```

**Expected (correct) behavior:** 403/400 — approver 5 does not belong to transaction 1.

**Actual behavior:** 200 `true` — approver 5 (belonging to T2) is soft-deleted via `removeNode`. T2's approval requirement is now gone. T2 can proceed to execution without User B's approval.

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
