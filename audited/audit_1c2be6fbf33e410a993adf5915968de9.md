### Title
TOCTOU Race Condition in `approveTransaction` Allows Approver to Bypass Single-Submission Guard and Corrupt Approval State

### Summary
The `approveTransaction` function in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` performs its "already approved" guard check outside of any database transaction, while the actual write occurs in a separate, later database transaction. Two concurrent HTTP requests from the same authenticated approver can both pass the guard simultaneously, allowing the approver to submit two conflicting approval decisions (`approved=true` and `approved=false`) in a single race window. If the chain service's threshold check fires between the two writes, the transaction's execution state will not match the approvers' final recorded states — a direct state integrity violation.

### Finding Description

**Root cause:** The guard check and the state mutation are not atomic.

In `approvers.service.ts`, `approveTransaction` (lines 547–621):

- **Line 553** — reads approver state via `getVerifiedApproversByTransactionId` **outside** any DB transaction: [1](#0-0) 

- **Line 563** — the "already approved" guard is evaluated on the stale snapshot, also **outside** any DB transaction: [2](#0-1) 

- **Lines 565–596** — multiple async operations (`attachKeys`, `findOne`, `verifyTransactionBodyWithoutNodeAccountIdSignature`) execute between the guard and the write, widening the race window: [3](#0-2) 

- **Lines 599–610** — the actual UPDATE runs inside a separate DB transaction, but the guard at line 563 is not re-evaluated inside it: [4](#0-3) 

**Exploit flow:**

1. Approver reads `userApprovers` — `signature = null` for both concurrent requests.
2. Both requests pass `if (userApprovers.every(a => a.signature))` because neither has written yet.
3. Both requests independently verify their signatures against the same transaction body (the same key can produce two valid signatures for the same body).
4. Request A writes `{approved: true, signature: sig_A}`.
5. The chain service's threshold evaluator fires, sees the threshold met, and transitions the transaction to `WAITING_FOR_EXECUTION`.
6. Request B writes `{approved: false, signature: sig_B}`, overwriting the approver's record.
7. The transaction is now in `WAITING_FOR_EXECUTION` (and will be executed) while the approver's final recorded state is `approved = false`.

The controller endpoint that exposes this path: [5](#0-4) 

The `TransactionApprover` entity confirms `approved` and `signature` are nullable columns updated in-place: [6](#0-5) 

### Impact Explanation
An approver can cause a transaction to be executed on the Hedera network while their final recorded approval state shows `approved = false` (rejected). Conversely, an approver can record `approved = true` while the transaction is blocked. This breaks the core invariant of the approval workflow: **a transaction must not advance to execution unless all threshold approvers have a final `approved = true` state**. Because the chain service reads the `approved` field to determine threshold satisfaction, a race-induced state mismatch directly corrupts the trust model of the multi-party approval system.

### Likelihood Explanation
The attacker is a legitimate approver — no privilege escalation is required. Sending two concurrent HTTP requests is trivially achievable with a script, `curl --parallel`, or two browser tabs. The race window spans multiple async operations (key attachment, DB read, signature verification), making it wide enough to be reliably triggered. The approver can sign the same transaction body twice with the same key to produce two independently valid signatures, satisfying the signature verification check in both requests.

### Recommendation
Move the "already approved" guard inside the same database transaction that performs the UPDATE, using a `SELECT ... FOR UPDATE` (pessimistic lock) on the approver rows. This ensures the check and the write are atomic:

```typescript
await this.dataSource.transaction(async em => {
  // Re-read with a row-level lock
  const lockedApprovers = await em.query(
    `SELECT * FROM transaction_approver WHERE id = ANY($1) FOR UPDATE`,
    [userApprovers.map(a => a.id)]
  );
  if (lockedApprovers.every(a => a.signature !== null))
    throw new BadRequestException(ErrorCodes.TAP);

  await em.createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .whereInIds(lockedApprovers.map(a => a.id))
    .execute();
});
```

### Proof of Concept

**Preconditions:**
- Transaction T exists with status `WAITING_FOR_SIGNATURES`.
- User A is the sole approver; threshold = 1.
- User A holds a valid key pair registered in the system.

**Steps:**
1. User A signs the transaction body twice with the same private key, producing `sig_approve` and `sig_reject` (both are valid signatures over the same bytes).
2. User A fires two concurrent POST requests to `POST /transactions/{T}/approvers/approve`:
   - Request 1: `{ approved: true,  signature: sig_approve, userKeyId: k }`
   - Request 2: `{ approved: false, signature: sig_reject,  userKeyId: k }`
3. Both requests read `userApprovers` before either write completes; both see `signature = null`.
4. Both pass the guard at line 563.
5. Request 1 commits `approved = true` → chain service threshold check fires → transaction transitions to `WAITING_FOR_EXECUTION`.
6. Request 2 commits `approved = false` → approver record now shows `approved = false`.
7. **Observed:** Transaction is queued for on-chain execution; approver's recorded decision is `rejected`.
