### Title
XSS via Unsanitized `transactionMemo` and `description` Rendered with `v-html` in Transaction Group Signing UI

### Summary
The `CreateTransactionGroup.vue` page renders `groupItem.description` and `transactionMemo` (extracted from raw transaction bytes) directly into the DOM using Vue's `v-html` directive without any sanitization. This is the direct analog of the reported MetaMask Snaps `text()` Markdown injection: in both cases, attacker-controlled transaction data is rendered in a UI the user relies on to decide what to sign, and the rendered output can be manipulated to mislead the user. In an Electron renderer process, this also constitutes a stored XSS that can execute arbitrary JavaScript.

### Finding Description

In `front-end/src/renderer/pages/CreateTransactionGroup/CreateTransactionGroup.vue`, lines 421–429, the template uses `v-html` to render the display label for each transaction group item:

```vue
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
``` [1](#0-0) 

Both `groupItem.description` and `transactionMemo` are attacker-controlled:

- **`transactionMemo`** is a free-text field (up to 100 chars) set by any user when creating a transaction. It is embedded in the serialized transaction bytes and later extracted via `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo`. [2](#0-1) 

- **`groupItem.description`** is set by organization members when submitting group items to the backend. [3](#0-2) 

Neither value is HTML-escaped or sanitized before being passed to `v-html`. Vue's `v-html` bypasses its normal text-escaping and injects the string as raw HTML, so any HTML tags or JavaScript event handlers in the value are parsed and executed by the browser engine embedded in Electron.

The `makeTransfer` path (for Transfer Transactions) calls `formatHbarTransfers`, which returns a template string that also ends up in `v-html`. [4](#0-3) 

### Impact Explanation

**UI deception (direct analog to the report):** A malicious organization member crafts a transaction whose `transactionMemo` contains HTML such as `<span style="display:none">0.0.9999 → 1000 HBAR</span><span>0.0.1234 → 0.001 HBAR</span>`. The victim user sees fabricated transfer details in the group list and confirms/signs a transaction they did not intend to authorize.

**Stored XSS in Electron renderer:** Because this is an Electron application, JavaScript injected via `v-html` executes in the renderer process. Depending on `nodeIntegration` / `contextIsolation` settings, this can escalate to full Node.js API access (filesystem, child_process), constituting Remote Code Execution on the victim's machine. Even with `contextIsolation: true`, the attacker can still exfiltrate local data, manipulate the UI state, or intercept IPC messages.

### Likelihood Explanation

Any authenticated organization member can create a transaction with an arbitrary `transactionMemo` (the field is a plain text input with only a `maxlength="100"` client-side constraint). [5](#0-4)  The malicious bytes are stored in the backend and fetched by any other member who opens the group. No privileged role is required — a regular organization member is sufficient. The attack is fully automated and requires no user interaction beyond the victim opening the Create Transaction Group page.

### Recommendation

Replace `v-html` with Vue's safe text interpolation (`{{ }}`) for all user-controlled fields. If HTML formatting is genuinely needed for the transfer display, sanitize the output with a library such as `DOMPurify` before passing it to `v-html`. Specifically:

1. Replace the `v-html` binding at lines 421–429 of `CreateTransactionGroup.vue` with a plain text binding using `{{ }}`.
2. Refactor `formatHbarTransfers` to return a plain string (no HTML tags) and render it with `{{ }}`.
3. Apply the same fix to `AppStepper.vue` line 58 where `getBubbleContent` returns raw HTML strings — use a component or conditional rendering instead of `v-html`. [6](#0-5) 

### Proof of Concept

1. Log in as any organization member.
2. Create a new transaction (e.g., Transfer HBAR) and set the **Transaction Memo** to:
   ```
   <img src=x onerror="alert(document.title)">
   ```
3. Add this transaction to a group and submit it to the organization backend.
4. As a second organization member (the victim), open **Create Transaction Group** and load or view the group containing the crafted transaction.
5. The `onerror` handler fires, executing `alert(document.title)` in the Electron renderer — confirming arbitrary JavaScript execution. Replace the payload with `require('child_process').exec('calc.exe')` (if `nodeIntegration` is enabled) to demonstrate RCE, or with DOM manipulation to demonstrate UI deception before the victim clicks "Confirm". export function formatHbarTransfers(transfers: Transfer[]): string {
  if (transfers.length === 0) {
    return 'No transfers';
  }

  if (transfers.length === 1) {
    const amount = transfers[0].amount;
    if (amount.isNegative()) {
      return 'Missing receiver';
    } else {
      return 'Missing sender';
    }
  }

  if (transfers.length === 2) {
    // the JS SDK sorts the order of the transfers by account ID. We don't want this. We want the sender to be on the
    // left and the receiver to be on the right. So we need to check if the amount is negative or positive and then
    // arrange the transfers accordingly
    let sender = transfers[0];
    let receiver = transfers[1];
    if (receiver.amount.isNegative()) {
      sender = transfers[1];
      receiver = transfers[0];
    }
    return `${sender.accountId} --> ${stringifyHbarWithFont(receiver.amount)} --> ${receiver.accountId}`;
  }

  return 'Multiple transfers';
```
