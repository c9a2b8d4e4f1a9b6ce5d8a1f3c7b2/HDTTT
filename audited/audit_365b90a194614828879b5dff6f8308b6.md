### Title
Any Authenticated User Can Post Comments on Arbitrary Transactions Without Authorization Check

### Summary
The `createComment` endpoint in `back-end/apps/api/src/transactions/comments/comments.controller.ts` applies only authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) but performs no authorization check to verify that the requesting user is actually a participant (creator, signer, approver, or observer) of the target transaction. Any verified organization member can post comments on any transaction by supplying an arbitrary `transactionId`. The codebase itself acknowledges this gap with an explicit `//TODO need some sort of guard or check to ensure user can comment here` comment.

### Finding Description
**Root cause:**

The controller-level guards only confirm the caller holds a valid, non-blacklisted JWT and has `UserStatus.NONE` (verified). No guard or service-layer check confirms the caller has any relationship to the transaction identified by `:transactionId`. [1](#0-0) 

The `CommentsService.createComment` receives the caller's `User` object and the `transactionId` and persists the comment without verifying membership: [2](#0-1) 

**Exploit flow:**

1. Attacker registers as a normal organization user (no admin privileges needed).
2. Attacker obtains a valid JWT via `POST /auth/login`.
3. Attacker enumerates or guesses a `transactionId` (sequential integer IDs are typical in TypeORM/PostgreSQL).
4. Attacker sends `POST /transactions/<victim_transactionId>/comments` with any comment body.
5. The comment is persisted against the target transaction with no rejection.

**Existing checks and why they fail:**

`JwtBlackListAuthGuard` → confirms token is not revoked.
`JwtAuthGuard` → confirms token is valid.
`VerifiedUserGuard` → confirms `user.status === UserStatus.NONE`.

None of these checks whether the authenticated user is a creator, signer, approver, or observer of the transaction. The `AdminGuard` is also absent, so even non-admin users can reach this path. [3](#0-2) 

### Impact Explanation
- **Unauthorized state mutation**: Any organization member can inject comments into transactions they have no business relationship with, corrupting the audit trail and workflow communication for those transactions.
- **Information disclosure**: Successfully posting a comment to a guessed `transactionId` confirms that transaction exists, leaking organizational transaction metadata to unauthorized users.
- **Workflow disruption**: Spam comments on pending multi-signature transactions can confuse signers and approvers, potentially delaying or disrupting time-sensitive transaction execution.

### Likelihood Explanation
- **Attacker precondition**: Only a valid organization account is required — no admin role, no leaked secrets, no internal network access.
- **Attack complexity**: Low. A single authenticated HTTP POST with an integer `transactionId` is sufficient.
- **Discoverability**: The vulnerability is self-documented in the source code with a `//TODO` comment, meaning any developer or auditor reading the file immediately sees the gap. [4](#0-3) 

### Recommendation
Add a service-layer or guard-level check inside `CommentsService.createComment` (or as a dedicated `TransactionParticipantGuard`) that queries whether the requesting user appears in any of the following relations for the given `transactionId`:
- `transaction.creatorId === user.id`
- `TransactionSigner` table: user has a signing key linked to this transaction
- `TransactionApprover` table: user is listed as an approver
- `TransactionObserver` table: user is listed as an observer

If none of these conditions hold, return `401 Unauthorized` before persisting the comment. This mirrors the pattern already used in the observers controller, which checks creator identity before allowing mutations.

### Proof of Concept
```
# Step 1 – Obtain JWT as a non-privileged verified user
POST /auth/login
{ "email": "attacker@org.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2 – Post a comment on a transaction the attacker has no role in
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "message": "Unauthorized comment on transaction 42" }

# Expected (correct) response: 401 Unauthorized
# Actual response: 201 Created — comment persisted
```

The integer `transactionId` (42 in the example) can be discovered by brute-force or by observing IDs from transactions the attacker legitimately participates in, since PostgreSQL auto-increment sequences are contiguous.
