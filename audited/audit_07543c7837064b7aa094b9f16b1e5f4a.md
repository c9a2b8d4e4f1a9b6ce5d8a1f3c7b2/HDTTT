### Title
Incomplete State Reset in `updateTransactionApprover` Allows Stale Signature to Permanently Block Replacement Approver

### Summary
When a transaction creator replaces an approver's `userId` via `updateTransactionApprover`, the code attempts to clear `signature`, `userKeyId`, and `approved` by setting them to `undefined` in the TypeORM update payload. However, TypeORM's `EntityManager.update()` silently skips fields set to `undefined` — they are never included in the SQL `UPDATE` statement. As a result, the previous approver's signature data persists in the database under the new user's identity, and the replacement approver is permanently blocked from submitting their own approval.

### Finding Description

**Root cause — TypeORM skips `undefined` in updates:**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `updateTransactionApprover` function handles the case where a `userId` is changed: [1](#0-0) 

```typescript
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: undefined,   // ← TypeORM SKIPS this
  signature: undefined,   // ← TypeORM SKIPS this
  approved: undefined,    // ← TypeORM SKIPS this
};
await transactionalEntityManager.update(TransactionApprover, approver.id, data);
```

TypeORM's `EntityManager.update()` only includes fields with **defined** (non-`undefined`) values in the generated SQL. Fields set to `undefined` are omitted entirely from the `UPDATE` statement. The correct way to clear a nullable column is to use `null`, not `undefined`.

The effective SQL executed is:
```sql
UPDATE transaction_approver SET "userId" = $1 WHERE id = $2
-- signature, userKeyId, approved are NOT touched
```

**Downstream check that gets tripped:**

In `approveTransaction`, the guard against double-approval reads: [2](#0-1) 

```typescript
if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
```

Because `signature` was never cleared in the database, this check evaluates to `true` for the replacement approver, and the `BadRequestException` is thrown unconditionally.

**Why the unit test does not catch this:**

The test at line 808–813 of `approvers.service.spec.ts` only asserts that `dataSource.manager.update` was *called* with the `undefined` values — it does not test actual TypeORM persistence behavior, because the manager is fully mocked: [3](#0-2) 

**Entity definition confirms columns are nullable (can accept `null`):** [4](#0-3) 

All three fields (`userKeyId`, `signature`, `approved`) are declared `nullable: true`, so setting them to `null` via SQL is valid and is the correct fix.

### Impact Explanation

A transaction requiring approval is permanently stuck. After the creator replaces approver User A with User B:

- User B calls `POST /transactions/{id}/approvers/approve` and receives `ErrorCodes.TAP` ("Transaction Already Processed") every time.
- The transaction cannot advance past the approval gate.
- The creator cannot directly clear the stale signature (no API endpoint exists for that).
- If the transaction's `validStart` window expires while stuck, it transitions to `EXPIRED` status and can never be executed.

This constitutes **permanent lock / unrecoverable corruption of transaction state** for any transaction where an approver was replaced after the original approver had already signed.

### Likelihood Explanation

The scenario is realistic in normal organization workflows:

1. A transaction is created with approver User A.
2. User A approves (writes `signature`, `userKeyId`, `approved` to DB).
3. The creator decides to change the approver to User B (e.g., User A is unavailable, or the wrong person was assigned).
4. The creator calls `PATCH /transactions/{id}/approvers/{id}` with `{ userId: UserB_id }` — a documented, supported API operation.
5. The bug triggers silently; no error is returned.
6. User B is permanently blocked.

No privileged access is required. The attacker role here is a **malicious normal user** who is the transaction creator — a role reachable by any authenticated organization member.

### Recommendation

Replace `undefined` with `null` for the fields that must be cleared when `userId` changes:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts
const data: DeepPartial<TransactionApprover> = {
  userId: dto.userId,
  userKeyId: null,    // null is included in SQL UPDATE; undefined is not
  signature: null,
  approved: null,
};
```

This ensures TypeORM generates:
```sql
UPDATE transaction_approver
SET "userId" = $1, "userKeyId" = NULL, "signature" = NULL, "approved" = NULL
WHERE id = $2
```

Also update the in-memory object assignments on lines 508–510 to use `null` for consistency, and add an integration test that verifies the database state (not just the mock call) after a `userId` replacement on an already-signed approver.

### Proof of Concept

**Steps:**

1. Authenticate as the transaction creator (User C).
2. Create a transaction with approver User A via `POST /transactions/{txId}/approvers`.
3. Authenticate as User A; call `POST /transactions/{txId}/approvers/approve` with a valid signature. DB now has `signature = <bytes>`, `approved = true`, `userKeyId = <id>` for the approver row.
4. Authenticate as User C; call `PATCH /transactions/{txId}/approvers/{approverId}` with body `{ "userId": <UserB_id> }`. Returns 200 OK.
5. Inspect the DB: `SELECT "userId","signature","approved" FROM transaction_approver WHERE id = <approverId>` — `userId` is now User B's ID, but `signature` and `approved` still hold User A's values.
6. Authenticate as User B; call `POST /transactions/{txId}/approvers/approve` with a valid signature.
7. **Observed:** `400 Bad Request` with `ErrorCodes.TAP` — User B is permanently blocked.
8. **Expected:** User B's approval is accepted and the stale signature is cleared. [5](#0-4) [6](#0-5)
