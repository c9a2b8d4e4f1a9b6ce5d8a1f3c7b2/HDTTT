### Title
Missing HTTP Request Timeout in Mirror Node and Organization API Calls Enables Indefinite Hang (DoS)

### Summary
The codebase makes HTTP requests to external mirror nodes and organization servers without configuring any request timeout. A slow or non-responsive server causes these calls to block indefinitely, freezing the frontend UI or stalling backend processing. The pattern is inconsistently applied — `health.ts` correctly sets a 3-second timeout, but all other HTTP call sites do not.

### Finding Description

**Frontend — `mirrorNodeDataService.ts`:**

`getAccountsByPublicKey` and `getNetworkNodes` both execute paginated `while` loops that call `axios.get()` with no `timeout` and no `AbortController` signal: [1](#0-0) [2](#0-1) 

`getNodeInfo` similarly calls `axios.get()` with no timeout and no signal: [3](#0-2) 

**Frontend — `axios.ts` (global instance):**

No default timeout is set on the global `axios` instance, and the `axiosWithCredentials` wrapper (used for all organization API calls — login, transactions, user keys, etc.) passes no `timeout` field: [4](#0-3) 

**Backend — `MirrorNodeClient.getMirrorNodeData`:**

The backend mirror node client issues `axiosRef.get()` with no timeout. The retry wrapper (`fetchWithRetry`) adds up to 3 retries with exponential backoff, but each individual attempt can hang indefinitely if the server is slow rather than refusing the connection: [5](#0-4) 

**Contrast — `health.ts` (correctly protected):** [6](#0-5) 

The health check explicitly sets `timeout: 3000`, demonstrating the team is aware of the pattern but has not applied it consistently.

### Impact Explanation
- **Frontend:** A slow mirror node causes `getAccountsByPublicKey` or `getNetworkNodes` to block the async call chain indefinitely inside a `while` loop. The UI awaiting these calls (account lookup, node display, exchange rate) freezes with no recovery path.
- **Backend:** `MirrorNodeClient` is used in the cache-management and transaction-signature flows. A hanging `axiosRef.get()` holds a Node.js async context open indefinitely. With 3 retries and no per-request timeout, a single slow mirror node can tie up backend workers for an unbounded duration.
- **Organization API calls:** All calls through `axiosWithCredentials` (auth, transactions, user keys) have no timeout, so a slow organization server stalls the entire operation.

### Likelihood Explanation
Mirror nodes are external infrastructure. Network degradation, rate-limiting, or a misconfigured/malicious mirror node URL (user-supplied in organization settings) can produce slow responses without connection refusal. The paginated loops amplify the risk: each page fetch is a new unbounded wait. No trusted-role compromise is required.

### Recommendation
1. Set a global default timeout on the axios instance in `front-end/src/renderer/utils/axios.ts`:
   ```ts
   axios.defaults.timeout = 10000; // 10 seconds
   ```
2. Add per-request timeouts to the paginated loops in `getAccountsByPublicKey` and `getNetworkNodes` in `mirrorNodeDataService.ts`.
3. Add a `timeout` option to `getMirrorNodeData` in `back-end/libs/common/src/transaction-signature/mirror-node.client.ts`:
   ```ts
   const response = await this.httpService.axiosRef.get<T>(url, {
     headers,
     timeout: 10000,
     validateStatus: ...
   });
   ```

### Proof of Concept

1. Stand up a TCP server that accepts connections but never sends a response (simulating a slow mirror node).
2. Configure the application to use this server as the mirror node URL.
3. Trigger `getAccountsByPublicKey` or `getNetworkNodes` from the frontend (e.g., open the Accounts page).
4. Observe: the call never resolves, the UI hangs indefinitely, and no timeout error is thrown.
5. For the backend: trigger a cache refresh that calls `MirrorNodeClient.fetchAccountInfo`. Observe the NestJS worker is blocked for the full retry duration (up to ~21 seconds of backoff) with no per-request timeout boundary. [7](#0-6) [8](#0-7) [9](#0-8)

export const getAccountsByPublicKey = async (
  mirrorNodeURL: string,
  publicKey: string,
): Promise<AccountInfo[]> => {
  let accounts: AccountInfo[] = [];

  try {
    let nextUrl: string | null =
      `${withAPIPrefix(mirrorNodeURL)}/accounts/?account.publickey=${publicKey}&limit=100&order=asc`;

    while (nextUrl) {
      const res = await axios.get<AccountsResponse>(nextUrl);
      const data: AccountsResponse = res.data;
      accounts = accounts.concat(data.accounts || []);

      if (data.links?.next) {
        nextUrl = `${withAPIPrefix(mirrorNodeURL)}${data.links.next.slice(data.links.next.indexOf('/accounts'))}`;
      } else {
        nextUrl = null;
      }
    }

    return accounts;
  } catch (error) {
    logger.error('Failed to get accounts by public key', { error });
    return accounts;
  }

  private async fetchWithRetry<T>(
    url: string,
    etag?: string,
    attempt = 1,
  ): Promise<HttpResult<T>> {
    try {
      return await this.getMirrorNodeData<T>(url, etag);
    } catch (error) {
      const shouldRetry = this.isRetryableError(error);

      if (!shouldRetry || attempt >= RETRY_CONFIG.MAX_RETRIES) {
        this.logger.error(
          `Request failed after ${attempt} attempt(s) for ${url}: ${error.message}`
        );
        throw new HttpException(
          `Mirror node request failed: ${error.message}`,
          error.response?.status || HttpStatus.SERVICE_UNAVAILABLE,
        );
      }

      const delay = this.calculateBackoffDelay(attempt);
      this.logger.warn(
        `Request failed (attempt ${attempt}/${RETRY_CONFIG.MAX_RETRIES}), ` +
        `retrying in ${delay}ms: ${error.message}`
      );

      await this.delay(delay);
      return this.fetchWithRetry<T>(url, etag, attempt + 1);
    }
