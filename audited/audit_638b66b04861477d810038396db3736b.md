### Title
Cross-Transaction Approver Removal via IDOR in `removeTransactionApprover` — Creator of Any Transaction Can Delete Approvers Belonging to Other Transactions

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the calling user is the creator of `:transactionId`, but then removes the approver record identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any registered user who has created at least one transaction can exploit this to silently delete approvers from transactions they do not own, bypassing the approval gate and potentially allowing those transactions to advance to execution without required sign-off.

### Finding Description

**Root cause — decoupled ownership check and object mutation**

The controller performs two independent operations:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts  lines 102-113
@Delete('/:id')
async removeTransactionApprover(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Param('id', ParseIntPipe) id: number,
) {
    await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
    await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no cross-check
    return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller owns `transactionId`; it says nothing about whether approver `id` belongs to that transaction.

`removeTransactionApprover` then fetches the approver purely by its primary key and deletes it:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts  lines 534-543
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id); // fetches by PK only
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);          // deletes without ownership check
    emitTransactionStatusUpdate(...);
    return result;
}
``` [2](#0-1) 

`getTransactionApproverById` queries only on `id`, with no `transactionId` filter:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts  lines 66-82

**Exploit flow**

1. Attacker registers as a normal user and creates any transaction (becomes its creator).
2. Attacker enumerates or guesses the integer `id` of an approver belonging to a victim's transaction (IDs are sequential auto-increment integers).
3. Attacker sends:
   ```
   DELETE /transactions/{attacker_own_tx_id}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction(attacker_own_tx_id, attacker)` passes — attacker owns that transaction.
5. `removeTransactionApprover(victim_approver_id)` executes — deletes the victim's approver record with no further check.
6. The victim's transaction now has fewer (or zero) required approvers; the chain service may advance it to `WAITING_FOR_EXECUTION` or `EXECUTED` without the intended approval gate.

This is the direct analog of the external report: just as `send_nft` bypasses the payment settlement enforced by `transfer_nft`, the `DELETE` endpoint bypasses the per-transaction ownership check by splitting the authorization check and the mutation into two unlinked calls.

### Impact Explanation

- **Unauthorized state mutation**: Any registered user can remove approvers from any other user's transaction, destroying the multi-signature approval requirement.
- **Approval gate bypass**: Once required approvers are removed, the chain service can execute a Hedera transaction that was never properly approved, resulting in unauthorized on-chain actions (fund transfers, account updates, node operations, etc.).
- **No asset recovery**: Hedera transactions are irreversible once submitted to the network.

### Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (any registered, verified user). No admin or privileged role required.
- **Enumeration**: `TransactionApprover` IDs are sequential integers. An attacker can trivially iterate over a small range to find valid approver IDs belonging to other transactions.
- **No rate limiting observed** on the delete endpoint, making bulk enumeration practical.
- The attack is silent — the victim receives a status-update notification but no indication that an approver was removed by an unauthorized party.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver being deleted actually belongs to the transaction whose ownership was checked:

```typescript
// In approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // Resolve the root node to get the owning transactionId
    const root = await this.getRootNodeFromNode(approver.id);
    if (!root || root.transactionId !== transactionId) {
        throw new UnauthorizedException('Approver does not belong to this transaction');
    }

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
    return result;
}
```

Pass `transactionId` from the controller and enforce the cross-reference before any deletion occurs.

### Proof of Concept

**Setup**:
- User A creates `transaction_1` and adds User C as an approver (approver record gets `id = 42`).
- User B creates `transaction_2` (any valid transaction; B is its creator).

**Attack** (as User B):
```http
DELETE /transactions/transaction_2/approvers/42
Authorization: Bearer <user_B_jwt>
```

**Expected (broken) behavior**:
- `getCreatorsTransaction(transaction_2, userB)` → passes (B owns `transaction_2`).
- `removeTransactionApprover(42)` → deletes approver `id=42` which belongs to `transaction_1`.
- Response: `true`.

**Observed outcome**: User C's approver record for `transaction_1` is permanently deleted. `transaction_1` can now proceed to execution without User C's approval, violating the multi-signature requirement set by User A. [1](#0-0) [2](#0-1) [3](#0-2)
