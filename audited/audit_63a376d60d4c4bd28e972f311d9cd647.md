### Title
Any Authenticated User Can Post Comments on Arbitrary Transactions Due to Missing Authorization Check

### Summary

The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` performs no authorization check to verify that the requesting user has any relationship to the target transaction. Any verified, authenticated user can write comments to — and read comments from — any transaction in the system, regardless of whether they are the creator, signer, approver, or observer of that transaction. The developer acknowledged this gap with an explicit `//TODO` comment in the controller.

### Finding Description

**Root Cause**

The `createComment` handler in `CommentsController` applies only the standard authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). These guards confirm the user is logged in and verified, but they do not verify any relationship between the user and the target transaction. [1](#0-0) 

The developer explicitly flagged this gap: [2](#0-1) 

The underlying `CommentsService.createComment` performs zero authorization checks — it does not verify that the transaction exists, that the user is a creator/signer/approver/observer, or that the transaction is in a state that permits comments: [3](#0-2) 

**Contrast with other services**

Every other transaction sub-resource enforces relationship-based access. For example, `ObserversService.getTransactionObserversByTransactionId` and `ApproversService.getVerifiedApproversByTransactionId` both throw `UnauthorizedException` if the user is not a creator, signer, approver, or observer: [4](#0-3) [5](#0-4) 

The `CommentsService` has no equivalent check.

**Read path is also unprotected**

The `GET /transactions/:transactionId/comments` endpoint also performs no transaction-level authorization: [6](#0-5) 

**Exploit Path**

1. Attacker registers and verifies an account on the organization backend (standard user flow).
2. Attacker obtains a valid JWT.
3. Attacker sends `POST /transactions/{victimTransactionId}/comments` with any `message` body.
4. The comment is persisted to the `transaction_comment` table linked to the victim transaction — no relationship check is performed.
5. Attacker also sends `GET /transactions/{victimTransactionId}/comments` to read all comments on any transaction, including those containing sensitive operational details.

No privileged access, no leaked credentials, and no special knowledge beyond a valid transaction ID (which are sequential integers) is required.

### Impact Explanation

- **Unauthorized state modification**: Any verified user can inject comments into transactions they have no business relationship with, polluting the audit trail and potentially confusing legitimate participants.
- **Information disclosure**: Any verified user can read all comments on any transaction, including comments that may contain sensitive operational details (key decisions, financial amounts, internal notes).
- **Integrity violation**: The comment history — which is part of the transaction audit record — can be corrupted by arbitrary users.

### Likelihood Explanation

Exploitation requires only a valid, verified account (the normal registration flow). Transaction IDs are sequential integers, making enumeration trivial. No special tooling is needed — a single HTTP request with a valid JWT is sufficient. The attack is fully reachable by a malicious normal user abusing a valid product flow.

### Recommendation

Add a transaction membership check inside `CommentsService.createComment` (and `getTransactionComments`) that mirrors the pattern used by `ObserversService` and `ApproversService`: fetch the transaction with its `creatorKey`, `observers`, `signers`, and `approvers` relations, then throw `UnauthorizedException` if the requesting user does not appear in any of those roles. The existing `TransactionsService.verifyAccess` method already implements this logic and can be reused: [7](#0-6) 

### Proof of Concept

```
# Step 1: Login as any verified user
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2: Write a comment to a transaction the attacker has no role in
POST /transactions/42/comments
Authorization: Bearer <JWT>
Content-Type: application/json
{ "message": "Attacker-injected comment" }
→ 201 Created  (no 401/403)

# Step 3: Read all comments on any transaction
GET /transactions/42/comments
Authorization: Bearer <JWT>
→ 200 OK  [ { "id": 1, "message": "...", ... } ]
```

The `CommentsService.createComment` will persist the comment without any relationship verification, confirming the missing access control. [3](#0-2)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
