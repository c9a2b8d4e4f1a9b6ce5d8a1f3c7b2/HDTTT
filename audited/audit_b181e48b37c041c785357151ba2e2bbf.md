### Title
`getTransactionsToSign` Fetches All Non-Terminal Transactions Without Limit, Causing Unbounded Per-Transaction DB Queries and API DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every non-terminal transaction from the database with no row limit, then sequentially calls `userKeysToSign()` — which issues multiple database queries and a cryptographic signature-key computation — for each transaction. Pagination is applied only after the full scan completes. As the number of active transactions grows, each call to this endpoint consumes proportionally more server resources, eventually causing request timeouts and degrading the API for all users.

### Finding Description

**Root cause — no `take` limit on the initial fetch:**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` issues a `repo.find()` with no `take` (limit) parameter:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no `take` here; returns every non-terminal transaction
});
``` [1](#0-0) 

**Unbounded sequential loop with per-iteration DB I/O:**

Every transaction returned is then processed one-by-one:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) {
    console.log(error);
  }
}
``` [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which:
1. Calls `attachKeys(user, entityManager)` — a DB query to load the user's keys.
2. Calls `keysRequiredToSign()` — deserializes the transaction bytes, calls `transactionSignatureService.computeSignatureKey()` (mirror-node I/O or cryptographic work), and issues `entityManager.find(UserKey, ...)`. [3](#0-2) [4](#0-3) 

**Pagination applied after the full scan:**

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),   // ← sliced after O(N) work
  page,
  size,
};
``` [5](#0-4) 

The caller receives only one page of results, but the server has already paid the full O(N × DB_queries) cost.

**Contrast with other endpoints:** `getTransactions`, `getHistoryTransactions`, and `getTransactionsToApprove` all pass `skip`/`take` directly into the query and return immediately. [6](#0-5) 

### Impact Explanation

- **Service degradation / DoS for all users.** The Node.js event loop is occupied for the duration of the sequential `await` chain. With thousands of active transactions, a single request can hold the event loop for tens of seconds, making the API unresponsive to every other user.
- **Memory exhaustion.** All non-terminal transaction rows (including their `transactionBytes` blobs) are loaded into the process heap simultaneously.
- **Cascading DB pressure.** Each iteration fires multiple queries against PostgreSQL; under concurrent load this saturates the connection pool (`POSTGRES_MAX_POOL_SIZE: "3"` in the default config). [7](#0-6) 

### Likelihood Explanation

- **No privilege required.** Any authenticated user can create transactions (normal product workflow). Transactions remain non-terminal until executed, expired, or canceled.
- **Self-inflicted at scale.** Even without a malicious actor, an organization that accumulates thousands of pending transactions will trigger this degradation organically every time any user opens the "Ready to Sign" tab.
- **Rate limits do not protect.** The configured `GLOBAL_MINUTE_LIMIT: 10000` and `GLOBAL_SECOND_LIMIT: 1000` are far too high to prevent a single expensive request. [8](#0-7) 

### Recommendation

Apply the database-level limit **before** the loop, mirroring the pattern used by every other paginated method in the same service:

```typescript
// 1. Fetch only a bounded window of candidates
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  take: limit * CANDIDATE_MULTIPLIER,   // e.g. 5× the page size
  skip: offset,
});

// 2. Filter in-memory (cheap, bounded)
for (const transaction of transactions) { ... }
```

Alternatively, push the "does this user need to sign?" predicate into SQL (as `getTransactionsToApprove` does with its recursive CTE) so the database returns only matching rows and no application-level loop is needed.

### Proof of Concept

**Preconditions:** Authenticated user account; N active (non-terminal) transactions exist in the system (creatable via normal API calls).

**Trigger path:**
1. Authenticated user calls `GET /transactions/sign?page=1&size=10` (or equivalent endpoint that invokes `getTransactionsToSign`).
2. `getTransactionsToSign` issues `repo.find()` with no limit → PostgreSQL returns all N rows.
3. The service loops over all N rows, awaiting `userKeysToSign()` for each → O(N) sequential DB round-trips.
4. With N = 10,000 transactions and ~5 ms per iteration, the request takes ~50 seconds — well beyond any HTTP timeout — blocking the Node.js event loop throughout.

**Expected outcome:** The API endpoint times out; concurrent requests from other users are also delayed or dropped; the server's PostgreSQL connection pool is saturated. [9](#0-8)

      }
    }

      totalItems: result.length,
      items: result.slice(offset, offset + limit),

    };

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
};
