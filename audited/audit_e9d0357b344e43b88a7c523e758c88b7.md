### Title
Any Authenticated User Can Upload Signatures to Any Transaction Without Authorization Check

### Summary
The `uploadSignatureMaps` function in `SignersService` does not verify that the calling user is a designated signer, creator, observer, or approver of the target transaction before accepting and persisting their signature. Any authenticated user with a registered key can sign any transaction in the system. This is the direct analog to the "anyone can mint" pattern: just as an ERC20 contract with no `onlyMinter` guard lets anyone mint, this endpoint has no "only designated signer" guard.

### Finding Description

In `back-end/apps/api/src/transactions/signers/signers.service.ts`, `uploadSignatureMaps` performs three checks before accepting a signature:

1. Transaction exists
2. Transaction status is `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`
3. The public key in the submitted `SignatureMap` belongs to one of the calling user's own registered keys [1](#0-0) 

The critical missing check is any call to `verifyAccess`, which enforces that the user is the creator, observer, signer, approver, or holds a key actually required by the transaction. [2](#0-1) 

By contrast, the parallel `importSignatures` endpoint in `TransactionsService` **does** call `verifyAccess` before processing: [3](#0-2) 

The `uploadSignatureMaps` path is the primary signing endpoint exposed at `POST /transactions/:transactionId/signers`: [4](#0-3) 

**Exploit flow:**

1. Attacker registers as a normal user and registers any key pair.
2. Attacker learns (or enumerates) a transaction ID belonging to another user/organization.
3. Attacker signs the raw transaction bytes with their private key and submits the resulting `SignatureMap` to `POST /transactions/<victim_tx_id>/signers`.
4. `processTransactionSignatures` finds the attacker's public key in `userKeyMap`, calls `sdkTransaction.addSignature`, and returns the attacker's `UserKey`. [5](#0-4) 

5. `persistSignatureChanges` writes the modified transaction bytes back to the database and inserts a `TransactionSigner` row linking the attacker to the victim transaction. [6](#0-5) 

6. Because `verifyAccess` checks `transaction.signers?.some(s => s.userKey?.userId === user.id)`, the attacker now has permanent read access to the victim transaction. [7](#0-6) 

### Impact Explanation

- **Cross-tenant data exposure**: The attacker gains `verifyAccess` to any transaction they sign, exposing transaction bytes, signers, observers, and comments belonging to other organizations or users.
- **Transaction byte corruption**: The stored `transactionBytes` in the database are permanently modified to include the attacker's signature. While Hedera ignores extra signatures at the network level, the application's internal state is corrupted and the stored bytes no longer match the original `unsignedTransactionBytes`.
- **Signer record pollution**: The attacker is permanently recorded as a signer, which can interfere with audit trails, notification logic, and any downstream logic that iterates over signers.
- **Status advancement**: `processTransactionStatus` is called after every signature upload. If the attacker's key happens to satisfy a threshold in the Hedera transaction's key structure, the status can be advanced to `WAITING_FOR_EXECUTION` without the legitimate required signers having signed. [8](#0-7) 

### Likelihood Explanation

- **Attacker preconditions**: Only a valid JWT (any registered user) and one registered key pair are required — both are normal onboarding steps.
- **Transaction ID discovery**: Transaction IDs are sequential integers (`id: number`). An attacker can enumerate them trivially.
- **No privileged access required**: The attack requires no admin role, no leaked credentials, and no internal network access.
- **Reachable entry point**: The endpoint is a standard authenticated REST route with no additional guards beyond `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. [9](#0-8) 

### Recommendation

Add an authorization check at the start of `validateAndProcessSignatures` (or in `uploadSignatureMaps` before processing) that calls the equivalent of `verifyAccess` to confirm the user is a creator, observer, designated signer, or approver of the transaction — mirroring the guard already present in `importSignatures`:

```typescript
// In signers.service.ts, before processing each transaction:
if (!(await this.transactionsService.verifyAccess(transaction, user))) {
  return { id, error: ErrorCodes.TNF }; // or a dedicated UNAUTHORIZED code
}
```

This aligns `uploadSignatureMaps` with the access control already enforced by `importSignatures` in `TransactionsService`. [3](#0-2) 

### Proof of Concept

```
# 1. Register as a normal user (attacker) and obtain a JWT token.
POST /auth/login  →  { token: "<attacker_jwt>" }

# 2. Register a key pair for the attacker account.
POST /user-keys  →  { id: 42, publicKey: "<attacker_pubkey>" }

# 3. Enumerate a victim transaction (sequential IDs).
GET /transactions/7  →  { id: 7, transactionBytes: "<hex>", status: "WAITING_FOR_SIGNATURES" }

# 4. Sign the raw transaction bytes with the attacker's private key.
signatureMap = attackerPrivKey.sign(fromHex(transactionBytes))

# 5. Submit the signature — no access check is performed.
POST /transactions/7/signers
Authorization: Bearer <attacker_jwt>
Body: { signatureMap: <formatted_map> }

# Expected result: HTTP 201, attacker is now a recorded signer of transaction 7.
# Attacker can now GET /transactions/7 and receive full transaction details.
```

The attack succeeds because `uploadSignatureMaps` never calls `verifyAccess` and the only guard is that the submitted public key must match a key registered to the attacker — a condition the attacker trivially satisfies with their own key. [10](#0-9)

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
