### Title
Unbounded Batch Signature Upload Enables Amplified DoS on the API Server

### Summary
The `POST /transactions/:transactionId?/signers` (and `POST /transactions/signatures/import`) endpoints accept an **unbounded array** of `UploadSignatureMapDto` objects, each of which itself contains an **unbounded nested signature map** (nodes × transactionIds × public keys). Any authenticated, verified user can craft a single request that forces the server to perform an arbitrarily large number of cryptographic operations and database queries, directly analogous to the IN3 n² amplification pattern.

### Finding Description

**Entry point — unbounded array accepted with no size cap:**

The controller accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no `@ArrayMaxSize` guard. [1](#0-0) 

The DTO itself has no array-size constraint: [2](#0-1) 

**Unbounded signature map — O(nodes × txIds × keys) work per entry:**

The `@IsSignatureMap()` decorator iterates over every node account ID, every transaction ID, and every public key in the submitted map with no depth or count limit: [3](#0-2) 

**Per-entry cryptographic + DB work in `processTransactionSignatures`:**

For every unique public key found in the map, the service calls `sdkTransaction.addSignature` (an expensive SDK operation) and performs DB lookups: [4](#0-3) 

**Batch amplification — all entries processed in parallel:**

All items in the submitted array are processed concurrently via `Promise.all`, meaning a 1 000-item array with large signature maps triggers 1 000 parallel cryptographic + DB pipelines simultaneously: [5](#0-4) 

**Secondary vector — unbounded recursive approver tree:**

`CreateTransactionApproversArrayDto` has no `@ArrayMaxSize` on `approversArray`, and the nested `approvers` field also has no max size. The service processes this with a recursive function that issues multiple DB queries per node: [6](#0-5) [7](#0-6) 

**Throttling does not mitigate this:**

The existing throttlers limit *request count* per user/IP, not *work per request*. A single request with a 10 000-entry array or a deeply nested approver tree exhausts server resources within the throttle window. [8](#0-7) 

### Impact Explanation
A single authenticated user can submit one HTTP request containing thousands of `UploadSignatureMapDto` entries, each with a large signature map. The server will perform O(N × M × K) cryptographic operations (N = array length, M = nodes per map, K = keys per node) and a proportional number of DB queries, exhausting CPU and database connection pools. This can render the API unresponsive for all other users (DoS). The `importSignatures` endpoint (`POST /transactions/signatures/import`) is identically affected. [9](#0-8) 

### Likelihood Explanation
Any user who has completed email verification (`VerifiedUserGuard`) can reach these endpoints. No admin role or special privilege is required. The attack requires only a crafted JSON body — no special tooling. The pattern is well-known and easy to reproduce. [10](#0-9) 

### Recommendation
1. Add `@ArrayMaxSize(N)` (e.g., `N = 50`) to the top-level array accepted by `uploadSignatureMap` and `importSignatures`.
2. Add size limits inside `UploadSignatureMapDto` — cap the number of node account IDs, transaction IDs, and public keys in the signature map (e.g., via a custom validator that counts entries before processing).
3. Add `@ArrayMaxSize` to `CreateTransactionApproversArrayDto.approversArray` and `CreateTransactionApproverDto.approvers`, and enforce a maximum recursion depth in `createApprover`.
4. Consider moving the cryptographic validation work to a background queue so a single request cannot block the request-handling thread pool.

### Proof of Concept

```http
POST /transactions/signers HTTP/1.1
Authorization: Bearer <valid_verified_user_jwt>
Content-Type: application/json

[
  {
    "id": 1,
    "signatureMap": {
      "0.0.3":   { "0.0.2@1700000000.000000000": { "<pubkey_1>": "<sig>", "<pubkey_2>": "<sig>", ... /* 500 keys */ } },
      "0.0.4":   { "0.0.2@1700000000.000000000": { "<pubkey_1>": "<sig>", ... } },
      ... /* 100 node entries */
    }
  },
  ... /* repeat 500 times */
]
```

Each of the 500 array entries causes the server to iterate 100 nodes × 1 transaction × 500 keys = 50 000 `addSignature` calls and corresponding DB lookups, for a total of **25 000 000 cryptographic operations** from a single HTTP request — directly mirroring the n² amplification pattern described in the IN3 report. [11](#0-10)
