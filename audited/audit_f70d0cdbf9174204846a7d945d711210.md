### Title
Unauthenticated Transaction Group Enumeration via Unguarded "Testing Only" Endpoint

### Summary
The `GET /transaction-groups` endpoint in `TransactionGroupsController` is explicitly marked `/* TESTING ONLY */` in source code but remains live in production, accessible to any authenticated user. It returns **all** transaction groups with no user-based access control, allowing any authenticated user to enumerate every organizational transaction group — bypassing the per-user authorization enforced by every other endpoint in the same controller.

### Finding Description
In `back-end/apps/api/src/transactions/groups/transaction-groups.controller.ts`, the `getTransactionGroups` handler is decorated with a plain `@Get()` and is protected only by the class-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). The developer comment explicitly marks it as testing-only, yet it was never removed or restricted. [1](#0-0) 

The method calls `this.transactionGroupsService.getTransactionGroups()` with **no user argument**, so the service has no basis to filter results by the requesting user.

Contrast this with every other endpoint in the same controller, which receives `@GetUser() user: User` and passes it to the service for ownership/access verification: [2](#0-1) [3](#0-2) [4](#0-3) 

The same pattern of missing user-based access control also appears in `SignersController.getSignaturesByTransactionId`, which returns all signers for any transaction ID without verifying the requesting user has any relationship to that transaction: [5](#0-4) 

Compare this to `TransactionsService.getTransactionWithVerifiedAccess`, which explicitly calls `verifyAccess` before returning transaction data: [6](#0-5) 

### Impact Explanation
Any authenticated user — regardless of role or organizational relationship — can call `GET /transaction-groups` and receive the full list of every transaction group in the organization. Transaction groups contain group items that reference individual transactions, exposing names, descriptions, sequencing, and participant metadata for transactions the caller has no legitimate access to. This breaks the access-control invariant enforced by all other transaction-related endpoints and enables reconnaissance of the entire organization's transaction pipeline.

### Likelihood Explanation
High. The endpoint requires only a valid JWT (any verified user). It is registered under the Swagger `@ApiTags('Transaction Groups')` tag, making it discoverable via the API documentation. No elevated privilege, admin role, or special condition is required.

### Recommendation
Remove the `getTransactionGroups` endpoint entirely — the developer comment already acknowledges it should not be in production. If a listing endpoint is genuinely needed, it must accept `@GetUser() user: User` and pass it to the service to filter results to groups the authenticated user created or is a participant in, consistent with the pattern used by `getTransactions` and `getTransactionsToApprove`. [7](#0-6) 

### Proof of Concept
1. Register and verify any user account on the organization backend.
2. Authenticate and obtain a JWT token via `POST /auth/login`.
3. Send `GET /transaction-groups` with the `Authorization: Bearer <token>` header.
4. Observe that the response contains **all** transaction groups in the organization, including those created by other users, with no filtering applied.
