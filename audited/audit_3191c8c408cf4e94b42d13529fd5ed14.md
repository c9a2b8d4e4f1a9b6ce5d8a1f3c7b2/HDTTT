### Title
IP Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header Renders Global Throttler Ineffective

### Summary
The global `IpThrottlerGuard` registered as `APP_GUARD` across all API endpoints derives its rate-limit tracking key from the attacker-controlled `X-Forwarded-For` request header. Because any HTTP client can forge this header, an attacker can trivially rotate the tracking key on every request, completely bypassing the only global rate-limiting protection. This leaves all API endpoints â€” including authentication, transaction creation, and signature upload â€” exposed to unlimited request flooding.

### Finding Description

The `IpThrottlerGuard` is registered as a global `APP_GUARD` in `api.module.ts`, making it the sole rate-limiting protection for every HTTP route: [1](#0-0) 

The guard's `getTracker()` method resolves the throttle key as follows: [2](#0-1) 

`req.headers['x-forwarded-for']` is evaluated **first** and takes priority over `req.ip`. Because `X-Forwarded-For` is a plain HTTP header, any client can set it to an arbitrary value. The throttler stores hit counts keyed by this value, so an attacker who sends a different `X-Forwarded-For` value on each request (e.g., `1.2.3.1`, `1.2.3.2`, â€¦) is treated as a distinct IP each time and never accumulates hits against the configured limits.

The `IpThrottlerModule` configures two throttlers (`global-minute` and `global-second`) whose limits are read from environment variables: [3](#0-2) 

Both limits are rendered meaningless once the tracking key is attacker-controlled.

The `EmailThrottlerGuard` is applied only to three auth endpoints (`/auth/signup`, `/auth/login`, `/auth/reset-password`): [4](#0-3) 

All other controllers â€” `TransactionsController`, `SignersController`, `ApproversController`, `TransactionGroupsController`, `UsersController`, `UserKeysController`, `NotificationPreferencesController` â€” carry **no per-route throttle guard** beyond the bypassable global IP guard: [5](#0-4) [6](#0-5) 

Additionally, `UserThrottlerModule` and `UserThrottlerGuard` are defined but never imported into `api.module.ts` and never applied to any route, so they provide no protection: [7](#0-6) [8](#0-7) 

### Impact Explanation

An unauthenticated or authenticated attacker can send unlimited requests to any API endpoint:

- **Authentication brute-force**: `/auth/login` has `EmailThrottlerGuard` keyed on `req.body.email`, but the global IP guard is bypassed. An attacker targeting a single account is still constrained by the email throttler, but can flood the server with login attempts against many different email addresses without IP-level restriction.
- **OTP/reset-password flooding**: `/auth/reset-password` triggers email dispatch on each call. With IP throttling bypassed and email throttling bypassable by rotating target emails, an attacker can cause mass email spam and exhaust email-sending quotas.
- **Transaction and signature endpoint DoS**: `POST /transactions`, `POST /transactions/:id/signers`, `POST /transaction-groups`, and all other business-logic endpoints have no effective rate limiting. An attacker can flood these endpoints to exhaust database connections, CPU, and memory, causing service unavailability for all legitimate users.
- **Unbounded resource creation**: Authenticated users can create unlimited transactions, signature maps, and approver records, causing database bloat and degraded performance.

### Likelihood Explanation

The precondition is zero: setting an HTTP header requires no privilege, no leaked credential, and no special tooling. Any HTTP client (curl, Python requests, k6) can rotate `X-Forwarded-For` values in a loop. The attack is fully scriptable and deterministic. No cryptographic primitive needs to be broken.

### Recommendation

1. **Do not trust `X-Forwarded-For` directly.** Configure NestJS/Express `trust proxy` to a specific, known proxy count or CIDR range so that `req.ip` reflects the real client IP as set by the trusted reverse proxy, and remove the explicit `req.headers['x-forwarded-for']` read from `getTracker()`.
2. **Use `req.ip` exclusively** as the throttle key in `IpThrottlerGuard`, relying on the framework's proxy-trust configuration to populate it correctly.
3. **Import and apply `UserThrottlerModule`/`UserThrottlerGuard`** to authenticated endpoints so that per-user rate limits are enforced independently of IP.
4. **Apply explicit `@Throttle()` overrides** on high-cost endpoints (transaction creation, signature upload) with tighter limits.

### Proof of Concept

```bash
# Bypass global IP rate limit by rotating X-Forwarded-For on every request
for i in $(seq 1 10000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST https://<api-host>/auth/reset-password \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.0.$((i % 255))" \
    -d '{"email":"victim@example.com"}'
done
# Expected: all requests return 200 (never 429), flooding victim's inbox
# and exhausting the email-sending service quota.

# Same technique against unauthenticated transaction listing or any endpoint:
for i in $(seq 1 10000); do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: 192.168.$((i/256 % 256)).$((i % 256))" \
    https://<api-host>/transactions
done
# Expected: server processes all 10,000 requests without ever returning 429.
```

Each iteration presents a fresh IP to the throttler, so the hit counter for any single key never reaches the configured limit, and the rate limiter never fires.

  protected getTracker(req: Record<string, any>): Promise<string> {
    const clientIp = req.headers['x-forwarded-for'] || req.ip;
    if (!clientIp) {
      throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
    }
    // console.log('client IP', clientIp.replace(/^.*:/, ''));
    return clientIp;
  }
