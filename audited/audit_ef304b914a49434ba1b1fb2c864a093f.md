### Title
Vue DevTools Plugin Unconditionally Bundled into Production Builds

### Summary
`vite-plugin-vue-devtools` is registered in `vite.config.ts` without any environment guard, causing it to be included in production Electron builds of the Hedera Transaction Tool. This is the direct analog of the external report: a development-only artifact ships in production, exposing internal application state — including Pinia store data that may contain private keys, mnemonics, and session tokens — to anyone with the Vue DevTools browser extension installed.

### Finding Description
In `front-end/vite.config.ts`, the config function receives `{ command }` and derives `isServe` and `isBuild` flags, but the `vueDevTools()` plugin is added to the `plugins` array unconditionally:

```ts
// front-end/vite.config.ts  lines 19-43
const isServe = command === 'serve';
const isBuild = command === 'build';
...
plugins: [
  vue(),
  vueDevTools(),   // ← no isServe / !isBuild guard
  electron({ ... }),
  eslint(),
],
``` [1](#0-0) [2](#0-1) 

`vite-plugin-vue-devtools` injects a client-side overlay and hooks into Vue's internal devtools API. When not gated on `isServe`, it is compiled into the production bundle. The `isBuild` variable is defined but never used to exclude this plugin.

### Impact Explanation
The Hedera Transaction Tool manages cryptographic key material and Hedera network transactions. Vue DevTools exposes the full Pinia store tree at runtime, which in this application includes account keys, mnemonics, organization credentials, and active session state. Any user with the Vue DevTools browser extension installed on a production build can inspect this data without any authentication. In an Electron context the renderer runs with elevated trust, making this exposure more severe than in a typical web app.

### Likelihood Explanation
The production Electron binary is distributed via GitHub Releases (`electron-builder.yml` publishes to `hashgraph/hedera-transaction-tool`). Any end-user who installs the Vue DevTools browser extension (a common developer tool) and opens the app will have the devtools overlay available. No attacker-controlled server or network access is required — the attack surface is entirely local to the installed application. [3](#0-2) 

### Recommendation
Gate `vueDevTools()` on the `isServe` flag so it is only active during local development:

```ts
// front-end/vite.config.ts
plugins: [
  vue(),
  ...(isServe ? [vueDevTools()] : []),
  electron({ ... }),
  eslint(),
],
```

This mirrors the fix described in the external report: programmatically exclude development-only tooling from production builds.

### Proof of Concept
1. Run `pnpm build` in `front-end/` to produce a production Electron bundle.
2. Install the [Vue DevTools browser extension](https://devtools.vuejs.org/).
3. Launch the packaged application.
4. Open Vue DevTools — the component tree, Pinia stores (including any key/credential state), router history, and timeline events are fully accessible.
