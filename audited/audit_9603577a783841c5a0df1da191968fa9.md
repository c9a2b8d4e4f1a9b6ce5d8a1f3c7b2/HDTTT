### Title
Unauthenticated Cross-Tenant Data Exposure via Unscoped `GET /transactions/history` Endpoint

### Summary
The `GET /transactions/history` endpoint returns all historical transactions from all users without any per-user access control. Any authenticated user can enumerate every other user's completed, failed, expired, canceled, and archived transactions. The sibling `GET /transactions` endpoint correctly scopes results to the requesting user, but `getHistoryTransactions` omits this scoping entirely — a missing invariant that proper test coverage would have caught.

### Finding Description

**Root cause:** `getHistoryTransactions` in the service accepts no `user` parameter and applies no ownership filter to the database query.

In `transactions.controller.ts`, the `/history` route passes no user context to the service:

```typescript
// back-end/apps/api/src/transactions/transactions.controller.ts  lines 142-154
@Get('/history')
@Serialize(withPaginatedResponse(TransactionDto))
getHistoryTransactions(
  @PaginationParams() paginationParams: Pagination,
  @SortingParams(transactionProperties) sort?: Sorting[],
  @FilteringParams({...}) filter?: Filtering[],
): Promise<PaginatedResourceDto<Transaction>> {
  return this.transactionsService.getHistoryTransactions(paginationParams, filter, sort);
}
```

No `@GetUser()` decorator is present. Compare this to `getTransactions` (line 119), which correctly injects `@GetUser() user: User` and passes it to the service.

In `transactions.service.ts`, the service method applies only a status filter — no user ownership clause:

```typescript
// back-end/apps/api/src/transactions/transactions.service.ts  lines 220-249
async getHistoryTransactions(
  { page, limit, size, offset }: Pagination,
  filter: Filtering[] = [],
  sort: Sorting[] = [],
): Promise<PaginatedResourceDto<Transaction>> {
  const findOptions: FindManyOptions<Transaction> = {
    where: {
      ...getWhere<Transaction>(filter),
      status: this.getHistoryStatusWhere(filter),  // only status filter
    },
    ...
  };
  const [transactions, total] = await this.repo
    .createQueryBuilder()
    .setFindOptions(findOptions)
    .getManyAndCount();   // returns ALL matching rows
  ...
}
```

The `getTransactions` method (lines 159-173) shows the correct pattern — it builds a `whereForUser` array scoped to `signers.userId`, `observers.userId`, and `creatorKey.userId`. `getHistoryTransactions` has no equivalent.

**Exploit path:**
1. Attacker registers a valid account and authenticates (obtains JWT).
2. Attacker sends `GET /transactions/history?page=1&limit=100` with their JWT.
3. Server returns all historical transactions across all users with no ownership check.
4. Attacker iterates pages to enumerate the full transaction history of the organization.

The endpoint is protected by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` (controller-level, lines 56-57), so the attacker must be a valid authenticated user — but no higher privilege is required.

### Impact Explanation
Any authenticated user can read the complete historical transaction record of every other user in the organization: transaction IDs, types, bytes, valid-start timestamps, mirror network, and group membership. For an organization using this tool to manage Hedera treasury operations, this leaks the full financial activity history to any member, including low-trust participants. Transaction bytes may also reveal account IDs, amounts, and key structures used in past operations.

### Likelihood Explanation
Exploitation requires only a valid user account — the lowest possible privilege level. The endpoint is a standard REST `GET` with pagination, reachable by any client that can authenticate. No special tooling or timing is needed. The attack is silent and leaves no anomalous trace beyond normal API access logs.

### Recommendation
Inject the authenticated user into `getHistoryTransactions` and apply the same ownership scoping used by `getTransactions`. Specifically:

1. Add `@GetUser() user: User` to the controller handler and pass it to the service.
2. In the service, add a `whereForUser` clause (matching `creatorKey.userId`, `signers.userId`, `observers.userId`, or approver membership) identical to the pattern in `getTransactions` (lines 159-173).
3. Add a unit test asserting that `GET /transactions/history` for user A does not return transactions owned exclusively by user B.
