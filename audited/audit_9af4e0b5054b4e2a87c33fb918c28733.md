### Title
Unbounded Array Input in `POST /transactions/signatures/import` Enables Authenticated DoS via Sequential Resource Exhaustion

### Summary
The `importSignatures` endpoint accepts an attacker-controlled array of `UploadSignatureMapDto` objects with no enforced size limit. For every element in the array, the service performs a sequential async database query (`verifyAccess` → `userKeysToSign`) and CPU-intensive cryptographic operations (`SDKTransaction.fromBytes`, `validateSignature`, `addSignature`). Any authenticated user can submit a single request with thousands of entries to exhaust server CPU, database connections, and Node.js event-loop capacity, degrading or blocking service for all other users. The developers themselves flagged this gap in a code comment.

### Finding Description

**Vulnerability class:** DoS — unbounded iteration over attacker-controlled input causing resource exhaustion (direct analog to the external report's nested loop OOG pattern, translated from EVM gas to Node.js CPU/DB resources).

**Entry point:**

`POST /transactions/signatures/import` is guarded only by JWT authentication and email verification. Any registered, verified user can reach it. [1](#0-0) 

The body is typed as `UploadSignatureMapDto[] | UploadSignatureMapDto`. No `ArrayMaxSize`, `Max`, or any cardinality constraint is applied anywhere in the DTO or the controller before the array is forwarded to the service. [2](#0-1) 

**Unbounded loop in the service:**

`importSignatures` iterates over every element of the caller-supplied `dto` array sequentially: [3](#0-2) 

Per iteration the service:
1. Calls `await this.verifyAccess(transaction, user)` — an async method that itself issues a database query via `userKeysToSign`.
2. Calls `SDKTransaction.fromBytes(transaction.transactionBytes)` — deserializes and parses a Hedera SDK transaction object (CPU-bound).
3. Calls `validateSignature(sdkTransaction, map)` — performs cryptographic signature verification (CPU-bound).
4. Iterates over all returned public keys and calls `sdkTransaction.addSignature(publicKey, map)` for each.

`verifyAccess` always issues at least one DB round-trip for non-terminal transactions: [4](#0-3) 

**Developer acknowledgment of the missing limit:**

The developers left an explicit comment in the same function noting the array should be bounded at the API layer: [5](#0-4) 

**Second affected endpoint — concurrent variant:**

`POST /transactions/:transactionId?/signers` (`uploadSignatureMaps`) has the same missing size cap but uses `Promise.all(dto.map(...))`, firing all DB queries and crypto operations simultaneously rather than sequentially. This can exhaust the PostgreSQL connection pool in a single request. [6](#0-5) [7](#0-6) 

### Impact Explanation

A single crafted request with N entries causes:
- **N sequential DB round-trips** (one `userKeysToSign` query per entry in `importSignatures`).
- **N CPU-intensive SDK deserialization + cryptographic verification calls** per entry.
- The Node.js event loop is blocked for the duration of the entire sequential chain, delaying all other in-flight requests.
- For `uploadSignatureMaps`, N concurrent DB queries can exhaust the connection pool, causing all other database operations to queue or fail.

At N = 10,000 (a trivially constructable JSON array), the server becomes unresponsive for the duration of the request. Repeated requests from a single authenticated account constitute a sustained service outage for all users of the organization backend.

**Impact: High** — complete service degradation achievable by a single authenticated user with no special privileges.

### Likelihood Explanation

Any user who has registered and verified their email address can reach both endpoints. No admin role, no leaked credentials, and no special account state is required beyond normal registration. The attack requires only a single HTTP request with a large JSON array body. The developer comment at line 575 confirms the team is already aware the limit is missing.

**Likelihood: High.**

### Recommendation

1. **Enforce an array size cap at the DTO/controller layer** for both endpoints. Add `@ArrayMaxSize(100)` (or a configurable constant) to the array parameter before it reaches the service. NestJS `class-validator` supports this natively.
2. **For `importSignatures`**, move the `verifyAccess` call outside the per-element loop where possible, or batch the access check into a single query.
3. **For `uploadSignatureMaps`**, replace the unbounded `Promise.all` with a chunked concurrency pattern (e.g., process at most 50 items concurrently).
4. **Add a global NestJS body-size limit** and consider rate-limiting the signature import endpoints per user.

### Proof of Concept

**Preconditions:** Attacker has a valid JWT for any verified user account.

**Request:**
```http
POST /transactions/signatures/import
Authorization: Bearer <valid_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { "0.0.3": { "0.0.1@1000000.000000000": { "<valid_pubkey_hex>": "<valid_sig_hex>" } } } },
  { "id": 1, "signatureMap": { ... } },
  ... (repeated 10,000 times)
]
```

**Expected outcome:** The server enters a sequential loop of 10,000 iterations, each performing at least one DB query and one cryptographic operation. The Node.js event loop is saturated for the duration, causing all concurrent requests from other users to time out. The attack is repeatable with no cooldown.

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
