### Title
`removeTransactionApprover` Does Not Adjust Parent Threshold After Child Deletion, Permanently Locking Transaction Approval

### Summary
When a child approver node is deleted via `removeTransactionApprover`, the parent threshold-tree node's `threshold` value is never decremented. If the parent was configured with `threshold == children.length` (a fully-saturated threshold), removing one child leaves `threshold > remaining_children`, making the approval condition permanently unsatisfiable and the transaction permanently stuck in a waiting state.

### Finding Description

**Vulnerability class:** State-transition invariant violation / DoS (permanent lock).

**Root cause ‚Äî `removeNode` has no threshold adjustment:**

`removeTransactionApprover` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` (lines 533‚Äì544) fetches the target approver and delegates directly to `removeNode`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);
  ...
}
```

`removeNode` (lines 204‚Äì231) issues a recursive SQL soft-delete of the target node and all its descendants, but **never touches the parent row's `threshold` column**:

```sql
update transaction_approver
set "deletedAt" = now()
from approversToDelete
where approversToDelete."id" = transaction_approver."listId"
   or transaction_approver."id" = $1;
```

No parent lookup, no threshold recalculation.

**Contrast with the correct path:**

`updateTransactionApprover` with `dto.listId = null` (lines 417‚Äì428) correctly handles the analogous "detach child" case:

```typescript
const newParentApproversLength = parent.approvers.length - 1;
if (newParentApproversLength === 0) {
  await transactionalEntityManager.softRemove(TransactionApprover, parent);
} else if (newParentApproversLength < parent.threshold) {
  await transactionalEntityManager.update(TransactionApprover, parent.id, {
    threshold: newParentApproversLength,
  });
}
```

The delete path has no equivalent guard.

**Reachable entry point:**

The `DELETE /:id` controller endpoint (lines 102‚Äì113) accepts any approver `id` ‚Äî including child nodes (those with `listId` set) ‚Äî after only verifying the caller is the transaction creator:

```typescript
@Delete('/:id')
async removeTransactionApprover(...) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

There is no restriction preventing the creator from targeting a nested child node.

**Exploit path:**

1. Creator builds a threshold tree: parent with `threshold = 2`, children `[UserA, UserB]` (2-of-2).
2. Creator calls `DELETE /transactions/{txId}/approvers/{UserA_id}`.
3. `removeNode` soft-deletes UserA. Parent still has `threshold = 2`, but only `[UserB]` remains.
4. The approval condition `threshold = 2` can never be satisfied with 1 approver.
5. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES`.

### Impact Explanation

The transaction's approval gate becomes permanently unsatisfiable. No combination of approver actions can advance the transaction to `WAITING_FOR_EXECUTION` or `EXECUTED`. The transaction is effectively frozen. Any Hedera operation (account update, file create, token operation, etc.) tied to that transaction is permanently blocked. The creator must notice the broken state and manually call `updateTransactionApprover` with a corrected threshold ‚Äî an implicit, undocumented recovery step that the system does not prompt for or enforce.

### Likelihood Explanation

The trigger is a normal, documented workflow action (deleting an approver) performed by the transaction creator. No privileged access beyond being the transaction creator is required. The scenario where `threshold == children.length` (a fully-saturated threshold, e.g., 2-of-2 or 3-of-3) is a common and legitimate configuration. Any creator who removes one approver from such a tree without manually correcting the threshold will silently lock their transaction.

### Recommendation

In `removeTransactionApprover` (or inside `removeNode`), after soft-deleting the target node, look up the parent (if any) and apply the same threshold-adjustment logic already present in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id); // loads with relations: ['approvers']
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Adjust parent threshold before deletion
  if (approver.listId) {
    const parent = await this.repo.findOne(TransactionApprover, {
      relations: ['approvers'],
      where: { id: approver.listId },
    });
    if (parent) {
      const newCount = parent.approvers.length - 1;
      if (newCount === 0) {
        await this.repo.softRemove(parent);
      } else if (newCount < parent.threshold) {
        await this.repo.update(parent.id, { threshold: newCount });
      }
    }
  }

  await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
}
```

### Proof of Concept

**Setup:**
- Transaction T1 created by User C.
- Approver tree: root threshold-node P (`threshold=2`), children: approver node A (`userId=1`), approver node B (`userId=2`).
- State: `P.threshold = 2`, `P.approvers.length = 2` ‚Äî valid.

**Trigger:**
```
DELETE /transactions/T1/approvers/{A.id}
Authorization: Bearer <User C JWT>
```

**Resulting DB state:**
- Node A: `deletedAt = now()` (soft-deleted).
- Node P: `threshold = 2` (unchanged), `approvers = [B]` (length = 1).
- Invariant violated: `P.threshold (2) > P.approvers.length (1)`.

**Observable impact:**
- `POST /transactions/T1/approvers/approve` by User B records approval.
- Chain service evaluates: `approved_count (1) < threshold (2)` ‚Üí transaction never advances to execution.
- Transaction T1 is permanently stuck in `WAITING_FOR_SIGNATURES`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
