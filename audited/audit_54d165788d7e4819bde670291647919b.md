### Title
`ApproversController.removeTransactionApprover()` allows any transaction creator to delete approvers belonging to transactions they do not own

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the authenticated user is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that this approver actually belongs to `:transactionId`. Any user who has created at least one transaction can exploit this to delete approvers from any other transaction in the system, bypassing multi-signature approval workflows.

### Finding Description

The controller handler in `removeTransactionApprover` performs two independent operations:

1. `getCreatorsTransaction(transactionId, user)` — verifies the caller is the creator of the transaction at `:transactionId`.
2. `removeTransactionApprover(id)` — deletes the approver at `:id` with no cross-check against `:transactionId`. [1](#0-0) 

The service-level `removeTransactionApprover` simply fetches the approver by `id` and deletes it, with no check that the approver's `transactionId` matches the URL parameter: [2](#0-1) 

Contrast this with `updateTransactionApprover`, which correctly validates that the approver's root transaction matches the URL parameter before proceeding: [3](#0-2) 

The authorization check in `getCreatorsTransaction` only confirms ownership of the URL-supplied `transactionId`, not of the approver being deleted: [4](#0-3) 

**Exploit path:**
1. Attacker (Alice) creates transaction A — she is its creator.
2. Bob creates transaction B and assigns approvers (e.g., approver with `id = 99` belonging to transaction B).
3. Alice sends: `DELETE /transactions/{transactionA_id}/approvers/99`
4. `getCreatorsTransaction(transactionA_id, alice)` passes — Alice owns transaction A.
5. `removeTransactionApprover(99)` executes — approver 99 (belonging to transaction B) is soft-deleted with no further check.

### Impact Explanation

An attacker can silently remove required approvers from any transaction they do not own. This directly undermines the multi-signature approval workflow: a transaction that requires N approvals can have its approvers stripped, potentially allowing it to proceed without the intended oversight, or permanently disrupting the approval state of another user's transaction. The impact is unauthorized state mutation of another user's transaction data.

**Impact: High** — direct integrity failure in the approval/signing workflow, which is the core security primitive of the system.

### Likelihood Explanation

Any authenticated, verified user who has created at least one transaction can exploit this. Approver IDs are sequential integers and can be enumerated. No privileged access, leaked credentials, or special conditions are required beyond holding a valid JWT and having created one transaction.

**Likelihood: Medium** — requires an authenticated user and knowledge of a target approver ID, but both are trivially achievable.

### Recommendation

Inside `removeTransactionApprover` in the controller, verify that the fetched approver's root transaction matches the URL-supplied `transactionId` before deletion — exactly as `updateTransactionApprover` does in the service:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);
  // Add: verify the approver belongs to transactionId before removing
  const approver = await this.approversService.getTransactionApproverById(id);
  const rootNode = await this.approversService.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId) {
    throw new UnauthorizedException('Approver does not belong to this transaction');
  }
  await this.approversService.removeTransactionApprover(id);
  return true;
}
```

Alternatively, pass `transactionId` into `removeTransactionApprover` and enforce the ownership check at the service level, consistent with how `updateTransactionApprover` handles it.

### Proof of Concept

**Preconditions:**
- Alice is a verified user with JWT token `tokenA`.
- Alice has created transaction with `id = 1` (she is its creator).
- Bob has created transaction with `id = 2`, which has an approver with `id = 99`.

**Steps:**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer tokenA
```

**Expected (correct) behavior:** 403 Unauthorized — approver 99 does not belong to transaction 1.

**Actual behavior:** 200 OK — approver 99 is soft-deleted from transaction 2 without any error.

**Verification:** Query the `transaction_approver` table; the record with `id = 99` will have `deletedAt` set, and transaction 2's approval workflow is now broken. [1](#0-0) [2](#0-1)

  async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);

    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);

    return result;
  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
