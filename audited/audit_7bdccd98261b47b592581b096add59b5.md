### Title
TOCTOU Race Condition in `createTransactions` Allows Duplicate Transaction Creation via User-Controlled `transactionId`

### Summary

The `createTransactions` function in `transactions.service.ts` performs a duplicate-check query against the database and then saves the new transaction in two separate, non-atomic steps. Because no database-level unique constraint exists on the `transactionId` column, two concurrent API requests carrying the same user-controlled `transactionId` can both pass the check and both be persisted, producing duplicate transaction rows. Each duplicate can independently advance to execution and be submitted to the Hedera network, enabling double-execution of the underlying Hedera transaction.

### Finding Description

**Root cause — check-then-act without atomicity or a DB unique constraint**

In `createTransactions`, the uniqueness guard is a plain `SELECT` followed, after an `await` yield point, by a separate `INSERT` wrapped in a TypeORM `entityManager.transaction()`:

```
// Step 1 – check (lines 415-427)
const existing = await this.repo.find({
  where: {
    transactionId: In(transactionIds),
    status: Not(In([CANCELED, REJECTED, ARCHIVED])),
  },
  select: ['transactionId'],
});
if (existing.length > 0) throw new BadRequestException(...);

// Step 2 – save (lines 436-462, inside entityManager.transaction())
return await entityManager.save(Transaction, transactions);
``` [1](#0-0) [2](#0-1) 

The `entityManager.transaction()` call wraps only the `INSERT`; the `SELECT` that guards it runs outside that transaction boundary. Two concurrent HTTP requests that arrive within the same scheduling window will both execute Step 1 against an empty result set, both conclude no duplicate exists, and both proceed to Step 2 — where both `INSERT`s succeed because there is no database-level `UNIQUE` constraint on the `transactionId` column.

**No DB-level unique constraint on `transactionId`**

The earliest migration that creates the `Transaction` table defines `transaction_id TEXT NOT NULL` with no `UNIQUE` modifier and no unique index: [3](#0-2) 

The TypeORM entity grep confirms no `@Unique` or `@Index({unique:true})` decorator is applied to `transactionId` in the back-end entity layer. The `catch` block inside `entityManager.save` only re-throws as `BadRequestException(ErrorCodes.FST)`, so even if a constraint existed the error surface would be opaque; without one, both rows are silently committed.

**User-controlled `transactionId`**

The `transactionId` stored in the database is extracted directly from the user-supplied `transactionBytes` field of `CreateTransactionDto`. In Hedera, a transaction ID is `payerAccountId@validStartTime` — both values are chosen by the submitting client. Any registered user can craft transaction bytes with an arbitrary payer account and valid-start timestamp, giving full control over the resulting `transactionId` string. [4](#0-3) 

**Exploit path**

1. Attacker (registered user) observes or predicts a victim's `transactionId` (payer account is often a known organizational account; valid-start is a timestamp the attacker can race or guess).
2. Attacker sends `POST /transactions` with transaction bytes encoding the same `payerAccountId@validStartTime`, signed with their own creator key.
3. Both requests pass the `repo.find` check concurrently.
4. Both `entityManager.save` calls succeed — two `Transaction` rows with identical `transactionId` are now in the database.
5. The chain service's scheduler picks up both rows and submits both to the Hedera network. The second submission receives `DUPLICATE_TRANSACTION` from Hedera (handled gracefully by the execute service), but the first submission executes the underlying Hedera transaction — potentially a token transfer, account creation, etc. — twice from the platform's perspective, corrupting accounting state.

### Impact Explanation

- **Duplicate transaction rows** in the PostgreSQL database break the platform's invariant that each Hedera `transactionId` maps to exactly one managed transaction.
- **Double-execution risk**: both rows enter `WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION`. The chain service will attempt to execute both. The Hedera network deduplicates at the protocol level, but the platform's own state (status, executedAt, receipts) becomes inconsistent across the two rows.
- **Griefing / DoS**: an attacker who races a victim's submission causes the victim's row to be rejected at the Hedera level (duplicate) while the attacker's row is marked `EXECUTED`, effectively hijacking the execution record of the victim's transaction.
- **Accounting / audit integrity failure**: downstream reporting, fee accounting, and notification systems receive two events for one Hedera transaction.

### Likelihood Explanation

- **Attacker preconditions**: only a valid JWT (registered user account) is required — no admin or privileged role.
- **Triggering the race**: sending two HTTP requests within the same Node.js event-loop tick is trivially achievable with any HTTP client (`Promise.all([fetch(...), fetch(...)])`). No mempool or blockchain timing is involved.
- **Predicting the victim's `transactionId`**: organizational deployments typically use well-known payer accounts (treasury, operator). An attacker who knows the payer account can enumerate valid-start timestamps in a tight window. Alternatively, the attacker can race their own duplicate submission without targeting a specific victim.

### Recommendation

1. **Add a database-level unique constraint** on `transactionId` (scoped to non-terminal statuses, or unconditionally with a partial index excluding `CANCELED`/`REJECTED`/`ARCHIVED`). This makes the second concurrent `INSERT` fail at the DB layer regardless of application-level timing.
2. **Move the duplicate check inside the `entityManager.transaction()` block** and use a `SELECT ... FOR UPDATE` (pessimistic lock) or an `INSERT ... ON CONFLICT DO NOTHING RETURNING` pattern so the check and insert are atomic.
3. As a defence-in-depth measure, catch the unique-constraint violation in the `catch (error)` block inside `entityManager.save` and surface it as a meaningful `409 Conflict` rather than the generic `ErrorCodes.FST`.

export const createTransactionId = (
  accountId: AccountId | string,
  validStart: Timestamp | string | Date | null,
) => {
  if (typeof accountId === 'string') {
    accountId = AccountId.fromString(accountId);
  }

  if (!validStart) {
    validStart = Timestamp.generate();
  }

  if (typeof validStart === 'string' || validStart instanceof Date) {
    const date = validStart instanceof Date ? validStart : new Date(validStart);
    const expirationDate = new Date(date.getTime() + 1000 * 60 * 3);
    validStart = Timestamp.fromDate(expirationDate < new Date() ? new Date() : date);
  }

  return TransactionId.withValidStart(accountId, validStart);
