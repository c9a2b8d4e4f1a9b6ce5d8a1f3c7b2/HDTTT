### Title
Transaction Group Submission Accepts Intra-Batch Duplicate `transactionId` Entries, Bypassing the Uniqueness Guard

---

### Summary

`createTransactions()` in `transactions.service.ts` checks for duplicate `transactionId` values by querying the database before inserting. However, it never checks for duplicates **within the submitted batch itself**. Because none of the batch items exist in the database at check time, all items pass the guard and are persisted. A malicious authenticated user can submit a `POST /transaction-groups` request whose `groupItems` array contains multiple entries sharing the same `transactionId`, causing all of them to be stored and routed through the organization's signing/approval workflow, while only the first can ever succeed on the Hedera network.

---

### Finding Description

**Root cause — no intra-batch deduplication:**

`createTransactions()` collects all `transactionId` strings from the validated batch and issues a single `repo.find()` query against the database: [1](#0-0) 

The guard throws only if any of those IDs already exist in the database. It does **not** check whether the `transactionIds` array itself contains duplicates. If a caller submits two DTOs whose `transactionBytes` encode the same payer account and `validStart` (i.e., the same Hedera `transactionId`), both produce the same string at line 967: [2](#0-1) 

Both pass the guard (neither is in the DB yet), and both are saved inside the database transaction: [3](#0-2) 

**No database-level safety net:**

The `Transaction` entity declares `transactionId` as a plain `@Column()` with no `@Unique()` decorator and no unique index: [4](#0-3) 

There is therefore no constraint at the PostgreSQL level to reject the second insert.

**Attack entry point — `POST /transaction-groups`:**

`createTransactionGroup()` extracts all transaction DTOs from the request body and forwards them as a single batch to `createTransactions()`: [5](#0-4) 

Any authenticated user can reach this endpoint: [6](#0-5) 

**What happens at execution time:**

The chain service detects a `DuplicateTransaction` status code from Hedera and silently skips updating the duplicate row, leaving it permanently stuck in `WAITING_FOR_EXECUTION`: [7](#0-6) 

---

### Impact Explanation

In an organization context, every transaction stored in the database is routed through the full signing and approval workflow. Duplicate entries with the same `transactionId` will:

1. Appear as separate pending items requiring signatures from all required key holders.
2. Consume approver time and attention for transactions that can never succeed on Hedera.
3. Leave ghost rows permanently in `WAITING_FOR_EXECUTION` (the duplicate-detection path returns `null` without updating status), polluting dashboards and audit logs.
4. Scale linearly — a single malicious group submission with N identical items forces N−1 guaranteed failures through the full workflow.

---

### Likelihood Explanation

The attacker needs only a valid authenticated session (a normal organization member account). No privileged role is required. The `POST /transaction-groups` endpoint is publicly documented in `front-end/docs/api.md` and accepts arbitrary `transactionBytes` in the request body. Crafting two group items with identical bytes is trivial — the frontend's own "Duplicate" button demonstrates the concept, though the frontend assigns unique `validStart` values; a direct API call has no such constraint. [8](#0-7) 

---

### Recommendation

Add an intra-batch uniqueness check immediately after extracting `transactionIds`, before the database query:

```typescript
// In createTransactions(), after line 414
const uniqueIds = new Set(transactionIds);
if (uniqueIds.size !== transactionIds.length) {
  throw new BadRequestException(
    'Batch contains duplicate transactionId values',
  );
}
```

Additionally, add a database-level unique index on `transactionId` (scoped to non-terminal statuses, or unconditionally) to provide a hard safety net independent of application logic:

```typescript
// In transaction.entity.ts
@Index({ unique: true })
@Column()
transactionId: string;
```

---

### Proof of Concept

**Precondition:** Attacker holds a valid JWT for any organization member account with at least one registered key.

**Steps:**

1. Construct a valid, signed Hedera `AccountCreateTransaction` with a fixed `transactionId` (e.g., `0.0.1234@1700000000.000000000`). Serialize to hex as `TX_BYTES`.

2. Send:

```http
POST /transaction-groups
Authorization: Bearer <attacker_token>
Content-Type: application/json

{
  "description": "dup-test",
  "atomic": false,
  "sequential": false,
  "groupItems": [
    { "seq": 0, "transaction": { "transactionBytes": "TX_BYTES", "signature": "SIG", "creatorKeyId": 1, "name": "a", "description": "", "mirrorNetwork": "testnet" } },
    { "seq": 1, "transaction": { "transactionBytes": "TX_BYTES", "signature": "SIG", "creatorKeyId": 1, "name": "b", "description": "", "mirrorNetwork": "testnet" } }
  ]
}
```

**Expected (correct) behavior:** 400 — duplicate `transactionId` in batch.

**Actual behavior:** 201 — two `Transaction` rows are created with identical `transactionId`. Both appear in the organization's pending queue. The second will permanently fail on Hedera with `DUPLICATE_TRANSACTION` and remain stuck in `WAITING_FOR_EXECUTION`.
