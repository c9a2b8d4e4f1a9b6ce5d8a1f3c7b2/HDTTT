### Title
Convoluted Implementation of Freeze and Sign Operations in `signTransaction`

### Summary
The `signTransaction` function in `front-end/src/main/services/localUser/transactions.ts` merges two distinct operations — freezing a transaction and signing it — into a single function controlled by a boolean `needsFreeze` flag. A dedicated `freezeTransaction` function already exists separately. This dual-purpose design mirrors the M07 pattern: behavior changes silently based on an argument, increasing complexity, error-proneness, and the risk of incorrect state transitions.

### Finding Description
The `signTransaction` function at line 91 of `front-end/src/main/services/localUser/transactions.ts` accepts a `needsFreeze` parameter (defaulting to `true`):

```typescript
export const signTransaction = async (
  transactionBytes: Uint8Array,
  publicKeys: string[],
  userId: string,
  userPassword: string | null,
  needsFreeze = true,   // <-- controls two distinct behaviors
) => {
  const transaction = Transaction.fromBytes(transactionBytes);

  if (needsFreeze) {
    transaction.freezeWith(client);  // freeze path
  }
  // ... signing path always runs
``` [1](#0-0) 

A completely separate `freezeTransaction` function already exists at line 82 of the same file:

```typescript
export const freezeTransaction = async (transactionBytes: Uint8Array) => {
  const transaction = Transaction.fromBytes(transactionBytes);
  transaction.freezeWith(client);
  return transaction.toBytes();
};
``` [2](#0-1) 

The two distinct call sites demonstrate the dual-use nature:

1. `SignPersonalRequestHandler.vue` calls `signTransaction` without `needsFreeze` (defaults to `true`), meaning it freezes-then-signs in one call. [3](#0-2) 

2. `SignTransactionFileModal.vue` calls `signTransaction` with `needsFreeze = false` (line 85), meaning it only signs — because the transaction from the file is already frozen. [4](#0-3) 

The renderer-side wrapper in `transactionService.ts` passes `needsFreeze` through to the IPC layer, propagating the dual-purpose design across the entire stack. [5](#0-4) 

### Impact Explanation
If a caller passes `needsFreeze = false` for a transaction that has not yet been frozen, the Hedera SDK will attempt to sign an unfrozen transaction. In the Hedera SDK, signing an unfrozen transaction either throws an error or produces transaction bytes that are not valid for submission, silently corrupting the signing flow. Conversely, if `needsFreeze = true` is passed for an already-frozen transaction, calling `freezeWith` a second time may reset or corrupt the transaction's internal state. Either mistake causes a transaction to fail at execution time, with no compile-time or obvious runtime guard to catch the misuse.

### Likelihood Explanation
Any developer adding a new call site to `signTransaction` must independently know the freeze state of the transaction bytes being passed in. There is no type-level or API-level enforcement. The default of `needsFreeze = true` is a silent footgun for callers working with pre-frozen transactions (e.g., from files, from the organization backend, or from external signers). The existing codebase already has two divergent call patterns, confirming that the ambiguity is actively exercised.

### Recommendation
Split `signTransaction` into two clearly named, single-responsibility functions:

- `freezeAndSignTransaction(transactionBytes, publicKeys, userId, userPassword)` — for the standard personal flow where the transaction must be frozen before signing.
- `signFrozenTransaction(transactionBytes, publicKeys, userId, userPassword)` — for flows where the transaction is already frozen (external signing, file-based signing).

This eliminates the boolean flag, makes each call site's intent explicit, and removes the possibility of passing the wrong freeze mode.

### Proof of Concept

**Scenario — silent corruption via wrong `needsFreeze`:**

1. A developer adds a new call site for signing a transaction received from the organization backend (already frozen and serialized).
2. They call `signTransaction(bytes, keys, userId, password)` — omitting `needsFreeze`, which defaults to `true`.
3. `freezeWith(client)` is called on an already-frozen transaction. Depending on SDK version, this either throws immediately or resets node account IDs and transaction ID, invalidating the existing signatures embedded in the bytes.
4. The subsequent `transaction.sign(privateKey)` call succeeds, but the resulting bytes represent a different transaction than the one approved by the organization, breaking the multi-sig integrity guarantee.
