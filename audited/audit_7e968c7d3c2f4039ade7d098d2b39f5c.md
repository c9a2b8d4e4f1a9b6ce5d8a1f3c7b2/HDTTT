### Title
Missing Cryptographic Signature Verification in `processTransactionSignatures` Allows False Signer Registration

### Summary

The `SignersService.processTransactionSignatures` method in `back-end/apps/api/src/transactions/signers/signers.service.ts` records a user as a transaction signer and updates transaction bytes without ever verifying that the submitted signature bytes are cryptographically valid over the transaction body. Any authenticated user who owns a registered key can submit a signature map containing their public key paired with garbage or cross-transaction signature bytes, be recorded as a legitimate signer, and potentially advance the transaction's status to `WAITING_FOR_EXECUTION` — causing the transaction to fail on-chain when the chain service attempts to submit it.

### Finding Description

**Root cause:** `processTransactionSignatures` checks only that the public key in the submitted `SignatureMap` belongs to the authenticated user (via `userKeyMap` lookup), then unconditionally pushes the user into `userKeys` and calls `sdkTransaction.addSignature(publicKey, map)`. It never calls `publicKey.verify(bodyBytes, signature)`.

The codebase already contains a correct cryptographic verification helper, `validateSignature`, in `back-end/libs/common/src/utils/sdk/transaction.ts` (lines 213–248), which calls `publicKey.verify(bodyBytes, signature)` and throws on failure:

if (signatureValid) {

} else {

``` [2](#0-1) 

This helper is correctly used in the parallel `importSignatures` path in `transactions.service.ts`:

```typescript
const { data: publicKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(this, sdkTransaction, map),
);
if (error) throw new BadRequestException(ErrorCodes.ISNMPN);
``` [3](#0-2) 

But it is **absent** from the `uploadSignatureMaps` → `validateAndProcessSignatures` → `processTransactionSignatures` call chain in `signers.service.ts`. [4](#0-3) 

**Exploit flow:**

1. Attacker registers a normal account and adds a key pair (public key `K_attacker`).
2. Attacker observes any transaction in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION` status (no access-control check exists in `loadTransactionData` — it fetches by ID only).
3. Attacker crafts a `SignatureMap` containing `K_attacker` mapped to arbitrary/garbage bytes (or bytes from a completely different transaction).
4. Attacker calls `POST /transactions/:id/signers` with this map.
5. `processTransactionSignatures` finds `K_attacker` in `userKeyMap` (ownership check passes), calls `addSignature`, and pushes the user into `userKeys`.
6. In `persistSignatureChanges`, because `userKeys.length > 0`, the signer record is inserted even if `isSameBytes` is true (i.e., even if `addSignature` silently rejected the invalid bytes).
7. `processTransactionStatus` is called; if the attacker's false signer record satisfies the last required signature, the transaction advances to `WAITING_FOR_EXECUTION`.
8. The chain service attempts on-chain submission; Hedera rejects the transaction because the required signature is missing or invalid, leaving the transaction in `FAILED` state permanently. [5](#0-4) 

### Impact Explanation

- **False signer record**: The attacker is permanently recorded as having signed a transaction they did not sign.
- **Premature status advancement**: If the attacker's false record satisfies the last required signature threshold, the transaction moves to `WAITING_FOR_EXECUTION` before all legitimate signatures are collected.
- **Irreversible transaction failure**: The chain service submits the transaction to Hedera with a missing or invalid signature; Hedera rejects it, and the transaction is marked `FAILED` — a state from which there is no recovery path in the system.
- **Denial of Service against legitimate users**: Legitimate signers can no longer complete the transaction; the funds or operation it represents are permanently blocked.

### Likelihood Explanation

- Attacker precondition: a valid account with any registered key — the lowest possible privilege level.
- The vulnerable endpoint `POST /transactions/:id/signers` is publicly reachable by any authenticated user.
- No rate limiting or additional authorization is required.
- The attack requires only crafting a `SignatureMap` with a valid public key and arbitrary signature bytes, which is trivial given the SDK's public API.
- The discrepancy between `importSignatures` (which verifies) and `uploadSignatureMaps` (which does not) makes this a straightforward oversight to exploit.

### Recommendation

Call `validateSignature` (already present in `back-end/libs/common/src/utils/sdk/transaction.ts`) inside `processTransactionSignatures` before calling `addSignature`, mirroring the pattern used in `importSignatures`:

```typescript
// In processTransactionSignatures, before addSignature:
const { data: validKeys, error } = safe<PublicKey[]>(
  validateSignature.bind(null, sdkTransaction, map),
);
if (error) throw new Error(ErrorCodes.ISNMPN);
```

Only keys returned by `validateSignature` (i.e., those whose signatures are cryptographically valid over the transaction body bytes) should be added and recorded as signers. [6](#0-5) 

### Proof of Concept

**Setup:** Two users — `victim` (transaction creator) and `attacker` (any registered user with key `K_attacker`).

1. `victim` creates a transaction requiring `K_victim`'s signature. Transaction ID = `42`, status = `WAITING_FOR_SIGNATURES`.
2. `attacker` constructs a `SignatureMap`:
   ```
   { nodeId: "0.0.3", txId: "<victim's txId>", publicKey: K_attacker_DER, signature: <32 zero bytes> }
   ```
3. `attacker` sends:
   ```
   POST /transactions/42/signers
   Authorization: Bearer <attacker_jwt>
   Body: { id: 42, signatureMap: <crafted map> }
   ```
4. Server responds `201`. A `TransactionSigner` row is inserted for `attacker`'s key against transaction `42`.
5. If `K_attacker` was the last required signer (e.g., attacker also controls the fee-payer account), `processTransactionStatus` advances the transaction to `WAITING_FOR_EXECUTION`.
6. Chain service submits to Hedera → rejected → transaction status = `FAILED`.
7. `victim` can no longer execute the transaction; their funds/operation are permanently blocked. [7](#0-6) [8](#0-7)

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }

```typescript
  private async processTransactionSignatures(
    transaction: Transaction,
    map: SignatureMap,
    userKeyMap: Map<string, UserKey>,
    existingSignerIds: Set<number>
  ) {
    let sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

    const userKeys: UserKey[] = [];
    const processedRawKeys = new Set<string>();

    // To explain what is going on here, we need to understand how sdkTransaction.addSignature works.
    // The addSignature method will go through each inner transaction, then go through the map
    // and pull the signatures for the supplied public key belonging to that inner transaction
    // (denoted by the node and transaction id), add the signatures to the inner transactions.
    // So we need to go through the map and get each unique publicKey and call addSignature one time
    // per key.
    for (const nodeMap of map.values()) {
      for (const txMap of nodeMap.values()) {
        for (const publicKey of txMap.keys()) {
          const raw = publicKey.toStringRaw();

          // Skip duplicates across node/tx maps, and already-processed keys
          if (processedRawKeys.has(raw)) continue;
          processedRawKeys.add(raw);

          // Look up key (raw first, then DER)
          let userKey = userKeyMap.get(raw);
          if (!userKey) {
            userKey = userKeyMap.get(publicKey.toStringDer());
          }
          if (!userKey) throw new Error(ErrorCodes.PNY);

          // Only add the signature once per unique key
          sdkTransaction = sdkTransaction.addSignature(publicKey, map);

          // Only return "new" signers (not already persisted)
          if (!existingSignerIds.has(userKey.id)) {
            userKeys.push(userKey);
          }
        }
      }
    }

    // Finally, compare the resulting transaction bytes to see if any signatures were actually added
    const isSameBytes = Buffer.from(sdkTransaction.toBytes()).equals(
      transaction.transactionBytes
    );

    return { sdkTransaction, userKeys, isSameBytes };
  }
