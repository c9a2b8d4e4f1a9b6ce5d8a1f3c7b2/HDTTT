### Title
Single-User Approval Satisfies Multi-Party Threshold via Duplicate userId Across Approver Tree Branches

### Summary

The `isNode` duplicate-check in `approvers.service.ts` only prevents the same `userId` from appearing twice under the **same parent node** (`listId`). It does not prevent the same user from appearing in multiple distinct branches of the approval tree. Because `approveTransaction` then bulk-updates **all** `TransactionApprover` rows belonging to that user, a single approval action simultaneously satisfies every branch the user inhabits, collapsing a multi-party threshold requirement to a single-user action.

This is the direct analog of the external `Ticket._beforeTokenTransfer()` bug: just as `from == to` caused a balance to be decremented and re-incremented (net duplication), placing the same `userId` in multiple tree branches causes one approval to be counted multiple times, defeating the threshold invariant.

### Finding Description

**Root cause — `isNode` check is scoped to a single parent:** [1](#0-0) 

The `WHERE` clause includes `listId` (the immediate parent's ID), so the count only detects a duplicate within the same parent. A user with `listId=branchA.id` and another record with `listId=null` (root level) are treated as two distinct, non-duplicate entries. The function returns `false` for both insertions, allowing the same `userId` to be inserted at multiple positions in the tree.

**Insertion path — no global uniqueness check:** [2](#0-1) 

`createApprover` calls `isNode` before each insert. Because `isNode` is parent-scoped, the creator can submit a tree such as:

```
root (threshold=2)
├── branchA (threshold=1)
│   ├── userA   ← inserted with listId=branchA.id  ✓ passes isNode
│   └── userB
└── userA       ← inserted with listId=null         ✓ passes isNode (different listId)
```

**Approval path — bulk-updates every record for the user:** [3](#0-2) 

`getVerifiedApproversByTransactionId` returns the full recursive list. `approvers.filter(a => a.userId === user.id)` collects **both** userA rows. The subsequent `UPDATE … WHERE id IN (…)` marks both rows `approved=true` in one call.

**Result:** After userA's single approval:
- `branchA` has 1 approved child → threshold=1 satisfied → branchA counts as approved
- Root has 2 approved children (branchA + userA-at-root) → threshold=2 satisfied

The entire tree is satisfied by one user, despite the tree visually advertising a 2-of-N requirement.

### Impact Explanation

The multi-party approval governance model is completely bypassed. A transaction creator can construct an approver tree that **appears** to require N distinct approvals but actually requires only one colluding user (or the creator themselves, since there is no check preventing the creator from adding their own `userId` as an approver). Any observer or auditor inspecting the tree structure would see a legitimate-looking threshold requirement that is silently defeatable. The transaction proceeds to execution with a fraudulently satisfied approval state.

### Likelihood Explanation

The attacker must be the transaction creator — a normal, unprivileged role reachable by any registered user. No admin keys, leaked secrets, or race conditions are required. The creator has full, legitimate API access to `POST /transactions/:id/approvers` and can craft the malicious tree in a single request. The `isNode` bypass is deterministic and requires no timing or probabilistic exploitation.

### Recommendation

Replace the parent-scoped duplicate check with a **global uniqueness check** across the entire transaction's approver tree. Before inserting any user-type approver node, query whether that `userId` already exists anywhere in the tree for the given `transactionId`:

```typescript
// In isNode or a new helper:
const globalCount = await entityManager.count(TransactionApprover, {
  where: { userId: approver.userId, /* join/subquery to transactionId */ }
});
if (globalCount > 0) throw new Error(this.APPROVER_ALREADY_EXISTS);
```

The recursive CTE in `getApproversByTransactionId` already traverses the full tree and can be reused for this check. Additionally, the `approveTransaction` method should assert that `userApprovers.length === 1` or otherwise enforce that a user can only occupy one position in the tree.

### Proof of Concept

1. Register two users: **creator** and **userA**.
2. Creator creates a transaction and posts the following approver tree to `POST /transactions/:id/approvers`:
   ```json
   {
     "approversArray": [{
       "threshold": 2,
       "approvers": [
         {
           "threshold": 1,
           "approvers": [
             { "userId": <userA_id> },
             { "userId": <userB_id> }
           ]
         },
         { "userId": <userA_id> }
       ]
     }]
   }
   ```
3. `isNode` passes for both `userA` insertions because they have different `listId` values.
4. userA calls `POST /transactions/:id/approvers/approve` once with their key and `approved: true`.
5. `approveTransaction` collects both `TransactionApprover` rows for userA and bulk-updates both to `approved=true`.
6. The threshold=1 sub-tree is satisfied (userA approved), and the root threshold=2 is satisfied (sub-tree + userA-at-root both approved).
7. The transaction advances to execution status — with only one user's approval — despite the tree requiring two. [4](#0-3) [5](#0-4)

  async isNode(
    approver: CreateTransactionApproverDto,
    transactionId: number,
    entityManager?: EntityManager,
  ) {
    const find: FindManyOptions<TransactionApprover> = {
      where: {
        listId: typeof approver.listId === 'number' ? approver.listId : null,
        userId: typeof approver.userId === 'number' ? approver.userId : null,
        threshold:
          typeof approver.threshold === 'number' && approver.threshold !== 0
            ? approver.threshold
            : null,
        transactionId: typeof approver.listId === 'number' ? null : transactionId,
      },
    };

    const count = await (entityManager || this.repo).count(TransactionApprover, find);
    return count > 0 && typeof approver.userId === 'number';
  }
