### Title
Cross-Transaction Approver Deletion via IDOR in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies that the authenticated user is the creator of `:transactionId`, but then calls `removeTransactionApprover(id)` without verifying that the approver record identified by `:id` actually belongs to `:transactionId`. Any transaction creator can therefore delete approvers from transactions they do not own, bypassing the multi-signature approval workflow of other users' transactions.

### Finding Description

**Root cause — two independent lookups with no cross-check:**

In `approvers.controller.ts`, the delete handler performs two sequential calls:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.controller.ts  lines 102-113
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user);  // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);                // ← deletes approver by id, no ownership check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only confirms the caller owns `:transactionId`. It says nothing about whether the approver record `:id` belongs to that transaction.

`removeTransactionApprover` in the service then fetches the approver by its own primary key and deletes it unconditionally:

```typescript
// back-end/apps/api/src/transactions/approvers/approvers.service.ts  lines 533-544
async removeTransactionApprover(id: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);
  const result = await this.removeNode(approver.id);   // ← no transactionId check
  emitTransactionStatusUpdate(...);
  return result;
}
``` [2](#0-1) 

There is no assertion that `approver.transactionId === transactionId` (or that the root node of the approver tree belongs to the caller's transaction). The `updateTransactionApprover` path does perform such a root-transaction check (`parentRoot?.transactionId !== transactionId`), making the omission in the delete path a clear inconsistency. [3](#0-2) 

**Exploit path:**

1. Attacker (Alice) is a verified organization member and has created at least one transaction — call it `txA` (she is its creator).
2. Victim (Bob) created `txB`, which has a required approver record with `id = 99`.
3. Alice sends:
   ```
   DELETE /transactions/{txA.id}/approvers/99
   Authorization: Bearer <alice-jwt>
   ```
4. `getCreatorsTransaction(txA.id, alice)` passes — Alice owns `txA`.
5. `removeTransactionApprover(99)` deletes approver `99`, which belongs to `txB`.
6. Bob's transaction now has a missing approver, potentially satisfying a lower threshold or removing a required signer entirely.

### Impact Explanation

Removing an approver from another user's transaction directly undermines the multi-signature approval workflow. If the deleted approver was the sole required approver, the transaction can advance to execution without any approval. If it was part of a threshold tree, the threshold may be auto-reduced (see `removeNode` cascade logic), silently weakening the governance requirement. This constitutes an unauthorized state change against another user's transaction with direct integrity impact on the approval/execution pipeline. [4](#0-3) 

### Likelihood Explanation

The attacker must be a registered, verified member of the same organization — a low but realistic bar (any internal user qualifies). Approver IDs are sequential integers assigned by the database, making enumeration straightforward. No privileged role (admin/owner) is required. The attack requires only a single authenticated HTTP request.

### Recommendation

Inside `removeTransactionApprover` (or in the controller before calling it), assert that the resolved approver's root transaction matches the caller-supplied `transactionId`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root node and verify it belongs to the expected transaction
  const root = await this.getRootNodeFromNode(id, ...);
  if (root?.transactionId !== transactionId)
    throw new BadRequestException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  ...
}
```

This mirrors the existing guard already present in `updateTransactionApprover`. [3](#0-2) 

### Proof of Concept

**Preconditions:**
- Alice is a verified org member; she has created `txA` (id = 1).
- Bob has created `txB` (id = 2) with a required approver record (id = 5, userId = Carol).

**Steps:**

```http
DELETE /transactions/1/approvers/5
Authorization: Bearer <alice-jwt>
```

**Expected (correct) behavior:** 403 Unauthorized — approver 5 does not belong to transaction 1.

**Actual behavior:** 200 OK — approver 5 (belonging to Bob's transaction 2) is soft-deleted. Bob's transaction now has its approval requirement silently removed or its threshold reduced, and the transaction may proceed to execution without Carol's approval. [1](#0-0) [2](#0-1)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }
