### Title
Unbounded In-Memory Iteration Over All Non-Terminal Transactions in `getTransactionsToSign` Enables Resource Exhaustion DoS

### Summary
`TransactionsService.getTransactionsToSign` fetches every non-terminal transaction in the entire system from the database without a row-level limit, then performs an expensive per-transaction cryptographic operation loop in memory, applying pagination only after the full scan completes. A registered attacker can flood the system with transactions to make this endpoint progressively slower for every user who calls it, eventually causing request timeouts or server-side resource exhaustion.

### Finding Description

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` builds a `whereForUser` filter that contains only a status exclusion — no user-scoping predicate: [1](#0-0) 

The resulting `repo.find(...)` call has no `take` clause, so it returns every non-terminal transaction in the database: [2](#0-1) 

For each row returned, the loop calls `this.userKeysToSign(transaction, user)`, which in turn calls `keysRequiredToSign`. That function deserializes the raw transaction bytes via the Hedera SDK, calls `transactionSignatureService.computeSignatureKey` (a potentially expensive mirror-node or cryptographic operation), flattens the resulting `KeyList`, and issues a database query for matching `UserKey` rows: [3](#0-2) 

Pagination (`slice(offset, offset + limit)`) is applied only after the entire in-memory scan finishes: [4](#0-3) 

**Exploit path:**
1. Attacker registers as a normal user (no privilege required).
2. Attacker submits a large number of transactions via `POST /transactions`, keeping them in `WAITING_FOR_SIGNATURES` status (never signing them, so they never reach a terminal state).
3. Any user — including the attacker — who calls `GET /transactions/sign` now triggers a full table scan plus one SDK deserialization + one `computeSignatureKey` call + one DB query per row.
4. As the non-terminal transaction count grows, response latency grows linearly. At sufficient scale, the Node.js event loop is saturated, causing timeouts for all concurrent users of the endpoint.

### Impact Explanation

Every call to `GET /transactions/sign` by any authenticated user becomes O(N) in the total number of non-terminal transactions in the system, where each iteration involves CPU-bound SDK deserialization and at least one additional database round-trip. An attacker who creates enough transactions can make the endpoint time out for all users, effectively denying the signing workflow to the entire organization. Because the transactions remain in a non-terminal state indefinitely (the attacker never signs or cancels them), the degradation is permanent until an administrator manually purges the rows.

### Likelihood Explanation

Any registered user can reach this path with no elevated privileges. Transaction creation is a core product feature with no apparent hard cap on the number of open transactions per user visible in the reviewed code. The attacker cost is low (API calls), the amplification is high (every other user's signing-list request is affected), and the effect is persistent.

### Recommendation

Apply a database-level `take` limit before the in-memory loop, or — better — rewrite the query to push the user-key matching into the database so that only transactions actually requiring the requesting user's keys are returned. The current pattern of fetching all rows and filtering in application memory is the root cause:

```typescript
// Instead of:
const transactions = await this.repo.find({ where: whereForUser, ... }); // no limit
for (const transaction of transactions) { ... }

// Apply take at the DB level, or use a subquery that joins on user keys
const transactions = await this.repo.find({
  where: whereForUser,
  take: MAX_SAFE_BATCH,
  ...
});
```

Additionally, add a per-user rate limit on `GET /transactions/sign` and enforce a maximum number of open (non-terminal) transactions per creator account.

### Proof of Concept

1. Register two accounts: `attacker` and `victim`.
2. As `attacker`, submit 10,000 `AccountCreateTransaction` objects via `POST /transactions`, each signed only by the attacker's key so they remain in `WAITING_FOR_SIGNATURES`.
3. As `victim`, call `GET /transactions/sign` and measure response time — it will scale linearly with the 10,000 rows because `getTransactionsToSign` fetches all of them and calls `userKeysToSign` for each before returning.
4. Increase the count until the endpoint exceeds the server's request timeout, at which point the signing workflow is unavailable to all users. [5](#0-4)

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
```
