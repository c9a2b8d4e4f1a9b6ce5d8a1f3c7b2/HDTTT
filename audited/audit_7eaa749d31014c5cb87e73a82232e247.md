### Title
`removeUserKey` Allows Deletion of Keys Required by Pending Transactions, Permanently Blocking Execution

### Summary
Any verified organization user can soft-delete their own signing key via `DELETE /users/:userId/keys/:keyId` without any check for pending transactions that require that key. Once the key is soft-deleted, `keysRequiredToSign` silently excludes it (TypeORM filters `deletedAt IS NULL`), leaving transactions in `WAITING_FOR_SIGNATURES` with no path to completion. The creator must cancel and recreate the transaction, and a malicious user can exploit this to deliberately sabotage any transaction they are required to sign.

### Finding Description

**Root cause — `removeUserKey` has no pending-transaction guard:** [1](#0-0) 

The function only verifies ownership (`userKey.userId !== user.id`) and then calls `softRemove`. There is no check for active transactions whose embedded public key matches the key being deleted.

**How the key becomes required:**

When a transaction is created, the Hedera SDK transaction bytes embed the public keys that must sign. The backend resolves which `UserKey` rows correspond to those public keys via `keysRequiredToSign`: [2](#0-1) 

TypeORM's default `find` excludes soft-deleted rows (`WHERE deletedAt IS NULL`). After the key is soft-deleted, this query returns nothing for that public key — the system no longer knows any user owns it.

**Effect on transaction status:**

`processTransactionStatus` checks whether the already-collected signatures satisfy the Hedera key requirement: [3](#0-2) 

If the required key has not yet signed, `hasValidSignatureKey` returns false and the transaction stays in `WAITING_FOR_SIGNATURES`. Because the key is now soft-deleted, no user can be prompted to sign it, and no user can provide the signature through the tool. The transaction remains stuck until its Hedera `validStart` expires.

**Notification service also silently drops the deleted key:** [4](#0-3) 

Soft-deleted keys are filtered out of reminder notifications, so no one is ever alerted that the transaction is blocked.

**The endpoint is reachable by any verified user (no admin required):** [1](#0-0) 

Confirmed by the e2e test suite: [5](#0-4) 

**Admin path (`removeUser`) has the same gap:** [6](#0-5) 

`softDelete(UserKey, { userId: id })` bulk-deletes all keys for the removed user with no check for pending transactions.

### Impact Explanation

Any transaction in `WAITING_FOR_SIGNATURES` whose required public key is owned by the deleting user becomes permanently uncompletable through the tool. The transaction will sit in that state until its Hedera `validStart` window expires, at which point it is marked `EXPIRED`. The transaction creator must cancel and recreate the transaction, losing any signatures already collected from other parties. A malicious required signer can exploit this to reliably block any multi-party transaction they are assigned to sign, with no recourse for other participants.

**Impact: Medium** — workflow disruption and forced transaction recreation; no direct fund theft, but targeted DoS of specific transaction flows is reliable and repeatable.

### Likelihood Explanation

The endpoint `DELETE /users/:userId/keys/:keyId` is available to every verified organization user. No elevated privilege is needed. A user who wants to avoid signing a transaction (or sabotage a workflow) only needs to delete their key after being added as a required signer. The scenario also occurs accidentally when a user rotates keys without realizing they are a required signer on pending transactions.

**Likelihood: Medium** — the action is a normal product feature (key deletion), the malicious variant requires only a verified account, and accidental triggering during key rotation is realistic.

### Recommendation

Before soft-deleting a `UserKey`, query for active transactions (`WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`) whose `publicKeys` array contains the key's public key, or whose `TransactionSigner`/`TransactionApprover` records reference the key. If any exist, either reject the deletion with an error listing the affected transactions, or require the caller to explicitly cancel those transactions first.

The same guard should be applied in `removeUser` before bulk-deleting all keys for a user.

### Proof of Concept

1. Admin creates a multi-signature transaction (e.g., `AccountUpdateTransaction`) that requires User B's public key to sign. Transaction status becomes `WAITING_FOR_SIGNATURES`.
2. User B (a regular verified user) calls `DELETE /users/{userBId}/keys/{keyId}` — their own key — before signing. The call succeeds with HTTP 200.
3. The key row now has `deletedAt` set. `keysRequiredToSign` no longer returns it.
4. The transaction scheduler runs `processTransactionStatus`; `hasValidSignatureKey` returns false (User B's key never signed). Status remains `WAITING_FOR_SIGNATURES`.
5. The notification service calls `filterActiveUserKeys`, excludes the deleted key, and sends no signing reminders.
6. The transaction expires at its `validStart` deadline. The creator must cancel and recreate it, losing all previously collected signatures. [7](#0-6) [8](#0-7) [9](#0-8)

```typescript
export const keysRequiredToSign = async (
  transaction: Transaction,
  transactionSignatureService: TransactionSignatureService,
  entityManager: EntityManager,
  showAll: boolean = false,
  userKeys?: UserKey[],
  cache?: Map<string, UserKey>,
): Promise<UserKey[]> => {
  if (!transaction) return [];

  /* Deserialize the transaction */
  const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);

  // list of just public keys that have already signed the transaction
  const signerKeys = sdkTransaction._signerPublicKeys;

  const signature = await transactionSignatureService.computeSignatureKey(transaction, showAll);
  // flatten the key list to an array of public keys
  // and filter out any keys that have already signed the transaction
  const flatPublicKeys = flattenKeyList(signature)
    .map(pk => pk.toStringRaw())
    .filter(pk => !signerKeys.has(pk));

  if (flatPublicKeys.length === 0) return [];

  let results: UserKey[] = [];
  // Now if userKeys is provided, filter out any keys that are not in the flatPublicKeys array
  // this way a user requesting required keys will only see their own keys that are required
  // Otherwise, fetch all UserKeys that are in flatPublicKeys
  if (userKeys) {
    results = userKeys.filter(publicKey =>
        flatPublicKeys.includes(publicKey.publicKey)
    );
  } else {
    if (cache) {
      const cachedKeys: Set<UserKey> = new Set();
      const missingPublicKeys: Set<string> = new Set();

      for (const publicKey of flatPublicKeys) {
        const cached = cache.get(publicKey);
        if (cached) {
          cachedKeys.add(cached);
        } else {
          missingPublicKeys.add(publicKey);
        }
      }

      let fetchedKeys: UserKey[] = [];
      if (missingPublicKeys.size > 0) {
        try {
          fetchedKeys = await entityManager.find(UserKey, {
            where: { publicKey: In([...missingPublicKeys]) },
            relations: ['user'],
          });
          // Store fetched keys in cache
          for (const key of fetchedKeys) {
            cache.set(key.publicKey, key);
          }
        } catch (error) {
          console.error('Error fetching missing user keys:', error);
          throw error;
        }
      }

      results = [...cachedKeys, ...fetchedKeys];
    } else {
      results = await entityManager.find(UserKey, {
        where: { publicKey: In(flatPublicKeys) },
        relations: ['user'],
      });
    }
  }

  return results;
```
