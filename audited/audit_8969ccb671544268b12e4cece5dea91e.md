### Title
TOCTOU Race Condition in `approveTransaction` Allows Approver to Bypass "Already Voted" Guard and Flip Vote

### Summary
`approveTransaction` in `back-end/apps/api/src/transactions/approvers/approvers.service.ts` reads the approver's current state and performs the "already approved" guard check entirely outside any database transaction. Two concurrent POST requests to `POST /transactions/:transactionId/approvers/approve` from the same authenticated approver can both pass the guard simultaneously, allowing the approver to submit two conflicting votes (e.g., `approved=true` and `approved=false`) and have the last write win — bypassing the one-vote-per-approver invariant.

### Finding Description

**Root cause — check-then-act is not atomic:**

In `approveTransaction` (`back-end/apps/api/src/transactions/approvers/approvers.service.ts`):

```

Line 556: const userApprovers = approvers.filter(a => a.userId === user.id);
Line 563: if (userApprovers.every(a => a.signature)) throw new BadRequestException(ErrorCodes.TAP);
...
Line 599: await this.dataSource.transaction(async transactionalEntityManager => {
Line 600:   await transactionalEntityManager
Line 601:     .createQueryBuilder()
Line 602:     .update(TransactionApprover)
Line 603:     .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
Line 608:     .whereInIds(userApprovers.map(a => a.id))
Line 609:     .execute();
Line 610: });
```

The guard at line 563 reads stale data from outside the DB transaction. The actual mutation at lines 599–610 is inside a DB transaction, but the guard is not. There is no database-level lock, `SELECT FOR UPDATE`, or optimistic concurrency check between the read and the write.

**Exploit path:**

1. Attacker is a legitimate approver of transaction T (no privilege required beyond being an approver).
2. Attacker sends two concurrent HTTP requests to `POST /transactions/T/approvers/approve`:
   - Request A: `{ approved: true, signature: sigA, userKeyId: K }`
   - Request B: `{ approved: false, signature: sigB, userKeyId: K }`
3. Both requests execute `getVerifiedApproversByTransactionId` before either completes — both read `userApprovers` with `signature = null`.
4. Both pass the guard at line 563 (`userApprovers.every(a => a.signature)` is `false` for both).
5. Both proceed through signature verification and reach the DB update.
6. Both updates execute. The last one to commit wins.
7. Final DB state reflects whichever `approved` value was written last — the attacker controls this by timing.

**No unique constraint or DB-level guard exists** on `TransactionApprover` that would prevent both updates from succeeding. The entity has no unique index on `(transactionId, userId)`.

### Impact Explanation

An approver can flip their vote from `approved=true` to `approved=false` (or vice versa) after already submitting, by racing two concurrent requests. This corrupts the integrity of the multi-signature approval workflow:

- A transaction that legitimately reached its approval threshold can be retroactively rejected by an approver flipping their vote.
- A transaction that was rejected can be retroactively approved.
- The notification logic at line 614 uses the stale pre-update `userApprovers` snapshot, so status-update events may be emitted incorrectly, further corrupting downstream state. [5](#0-4) 

### Likelihood Explanation

- **Attacker preconditions**: Must be a registered, verified user who is listed as an approver of the target transaction. No admin or privileged access is required.
- **Exploit complexity**: Sending two concurrent HTTP requests is trivial from any HTTP client (e.g., `Promise.all` in JavaScript, two parallel `curl` calls). The race window spans multiple async DB operations (lines 553–596), making it wide enough to be reliably exploitable.
- **No rate limiting or idempotency key** is present on the approve endpoint.

### Recommendation

Move the "already approved" guard **inside** the database transaction and use a pessimistic lock (`SELECT FOR UPDATE`) or an optimistic concurrency check (conditional `WHERE signature IS NULL`) on the update itself:

```typescript
await this.dataSource.transaction(async em => {
  // Re-read with a row lock inside the transaction
  const freshApprovers = await em
    .createQueryBuilder(TransactionApprover, 'a')
    .setLock('pessimistic_write')
    .where('a.id IN (:...ids)', { ids: userApprovers.map(a => a.id) })
    .getMany();

  if (freshApprovers.every(a => a.signature)) {
    throw new BadRequestException(ErrorCodes.TAP);
  }

  await em.createQueryBuilder()
    .update(TransactionApprover)
    .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
    .where('id IN (:...ids) AND signature IS NULL', { ids: freshApprovers.map(a => a.id) })
    .execute();
});
```

Alternatively, add a unique partial index on `(transactionId, userId) WHERE signature IS NOT NULL` at the database level to enforce the one-vote invariant.

### Proof of Concept

```typescript
// Attacker is an approver of transaction ID 42
const url = 'https://api.example.com/transactions/42/approvers/approve';
const headers = { Authorization: 'Bearer <attacker_jwt>' };

// Send two concurrent requests with opposite votes
const [r1, r2] = await Promise.all([
  fetch(url, {
    method: 'POST',
    headers,
    body: JSON.stringify({ userKeyId: K, signature: sigA, approved: true }),
  }),
  fetch(url, {
    method: 'POST',
    headers,
    body: JSON.stringify({ userKeyId: K, signature: sigB, approved: false }),
  }),
]);

// Both return HTTP 200. The DB now reflects whichever approved value
// was written last — the attacker controls the outcome by timing.
console.log(await r1.json()); // true
console.log(await r2.json()); // true
// Final DB state: approved = false (or true), bypassing the one-vote guard.
```

**Expected outcome**: Both requests succeed (HTTP 200). The `TransactionApprover` row is updated twice, with the final `approved` value determined by write order. The "already approved" guard (`ErrorCodes.TAP`) is never triggered because both reads see `signature = null`.
