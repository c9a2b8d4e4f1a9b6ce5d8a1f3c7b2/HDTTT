### Title
Manual Transaction Execution Allowed in `WAITING_FOR_SIGNATURES` State Due to Blocklist Status Guard

### Summary

`executeTransaction` in `transactions.service.ts` performs no transaction-status check before delegating to `ExecuteService`. The inner `validateTransactionStatus` guard uses a **blocklist** (throws on specific bad states) rather than an **allowlist** (only permits `WAITING_FOR_EXECUTION`). Because `WAITING_FOR_SIGNATURES` is absent from the blocklist, a creator can force-submit a manual transaction to the Hedera network while the database still records it as `WAITING_FOR_SIGNATURES`, producing a permanent state split: the network considers the transaction executed while the database eventually marks it `EXPIRED`.

### Finding Description

**Root cause — two cooperating gaps:**

**Gap 1:** `executeTransaction` in `transactions.service.ts` checks only `isManual`; it never asserts the transaction is in `WAITING_FOR_EXECUTION`. [1](#0-0) 

**Gap 2:** `validateTransactionStatus` in `execute.service.ts` uses a switch-case blocklist. Every terminal/invalid state is listed explicitly, but `WAITING_FOR_SIGNATURES` is not — so it falls through silently and the transaction proceeds to network submission. [2](#0-1) 

**Contrast with the correct pattern:** `executeTransactionGroup` explicitly filters to `WAITING_FOR_EXECUTION` before processing, and `_executeTransaction`'s DB update is conditioned on `status = WAITING_FOR_EXECUTION`. [3](#0-2) [4](#0-3) 

**Exploit path:**

1. Attacker (transaction creator) creates a manual transaction.
2. Required signers upload their signatures; `processTransactionStatus` has not yet promoted the row to `WAITING_FOR_EXECUTION` (or the creator races ahead of that update).
3. Creator calls `POST /transactions/:id/execute`. `executeTransaction` passes the `isManual` check.
4. `validateTransactionStatus` reads `WAITING_FOR_SIGNATURES` — not in the blocklist — and returns without throwing.
5. `hasValidSignatureKey` passes because the signatures are already present in the transaction bytes.
6. `sdkTransaction.execute(client)` submits the transaction to the Hedera network. **The transaction is now live on-chain.**
7. The DB update inside `_executeTransaction` targets `status = WAITING_FOR_EXECUTION`; the row is still `WAITING_FOR_SIGNATURES`, so `affected = 0` and the function returns `null`. No DB update, no notification.
8. `processTransactionStatus` later promotes the row to `WAITING_FOR_EXECUTION`. The scheduler picks it up and re-submits.
9. Hedera returns `DUPLICATE_TRANSACTION`; the duplicate-handling path skips the DB update, leaving the row in `WAITING_FOR_EXECUTION`.
10. After the 3-minute valid-start window closes, the row is marked `EXPIRED`. The transaction succeeded on-chain but the database permanently records it as expired. [5](#0-4) 

### Impact Explanation

- **Permanent state split:** A successfully executed Hedera transaction is recorded as `EXPIRED` in the database. All downstream logic (notifications, audit trails, UI status) reflects a false failure.
- **Double network submission:** The transaction is submitted twice; the second attempt consumes network fees and produces a `DUPLICATE_TRANSACTION` receipt that is silently discarded.
- **Integrity failure:** The organization's authoritative record of which transactions executed is corrupted for any transaction the creator races through this path.

    const transaction = await this.getTransactionForCreator(id, user);

      throw new BadRequestException(ErrorCodes.IO);
    }

    if (transaction.validStart.getTime() > Date.now()) {
      await this.repo.update({ id }, { isManual: false });
      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transaction.id }]);
    } else {
      await this.executeService.executeTransaction(transaction);
    }

    return true;
  }

  private async validateTransactionStatus(transaction: Transaction) {
    const { status } = await this.transactionsRepo.findOne({
      where: { id: transaction.id },
      select: ['status'],
    });

    switch (status) {
      case TransactionStatus.NEW:
        throw new Error('Transaction is new and has not been signed yet.');
      case TransactionStatus.FAILED:
        throw new Error('Transaction has already been executed, but failed.');
      case TransactionStatus.EXECUTED:
        throw new Error('Transaction has already been executed.');
      case TransactionStatus.REJECTED:
        throw new Error('Transaction has already been rejected.');
      case TransactionStatus.EXPIRED:
        throw new Error('Transaction has been expired.');
      case TransactionStatus.CANCELED:
        throw new Error('Transaction has been canceled.');
      case TransactionStatus.ARCHIVED:
        throw new Error('Transaction is archived.');
    }
  }
