### Title
No Service Pause During Database Schema Migration Allows Concurrent Transaction Processing

### Summary
The Hedera Transaction Tool's Kubernetes deployment runs database schema migrations as a standalone `Job` with no mechanism to halt or pause the API and Chain services. During the window when `migration-job` is altering the PostgreSQL schema, both the API service and the Chain service continue accepting and processing transactions. This mirrors the external report's root cause: state is read/written against a schema that is simultaneously being restructured, risking data corruption and an invalid system state.

### Finding Description
The migration job is defined as an independent Kubernetes `batch/v1` Job: [1](#0-0) 

It has no `initContainer`, no `readinessGate`, and no coordination hook that would prevent the API or Chain deployments from serving traffic while it runs. The API and Chain services are separate, always-running `Deployment` objects: [2](#0-1) [3](#0-2) 

The migration checklist confirms there is no step to stop running services before applying a migration: [4](#0-3) 

Migrations such as `TransactionNewKeyArray` and `TransactionCachedAccountIsReceiver` perform structural DDL changes (column additions, constraint modifications) on the live `transaction` table: [5](#0-4) [6](#0-5) 

While these DDL statements execute, the Chain service's transaction scheduler continues polling and writing to the same tables, and the API service continues accepting new transaction submissions.

### Impact Explanation
A DDL statement that adds a `NOT NULL` column, drops a column, or alters a constraint on the `transaction` table will conflict with in-flight `INSERT`/`UPDATE` queries from the API and Chain services. Possible outcomes include:

- **Data corruption**: rows written mid-migration may violate the new schema constraints, leaving the database in a partially valid state.
- **Application errors**: the Chain service's scheduler may attempt to read columns that no longer exist (or have been renamed), causing runtime failures and dropped transaction executions.
- **Silent data loss**: if the migration drops a column that the running service is still writing to, those writes are silently discarded.

### Likelihood Explanation
Every schema-changing release triggers this window. The production Helm chart and the dev k8s manifests both use the same standalone `Job` pattern with no traffic-draining step. Any operator following the documented checklist would apply the migration while services are live, making this a near-certain occurrence on every schema upgrade.

### Recommendation
**Short term:** Add a pre-migration step (e.g., a Kubernetes `initContainer` on the API/Chain pods, or a Helm hook with `pre-upgrade` weight) that scales the API and Chain deployments to zero replicas before the migration Job runs, and a `post-upgrade` hook that scales them back up after the Job completes successfully. Update `MIGRATION_CHECKLIST.md` to require this step explicitly.

**Long term:** Adopt an expand/contract migration pattern: first deploy a backward-compatible schema change (expand), then deploy the new application code, then remove the old schema artifacts (contract). This eliminates the need for a full service outage during migrations and ensures the system state remains valid throughout the upgrade process.

### Proof of Concept
1. Deploy the system to a Kubernetes cluster using the manifests in `back-end/k8s/dev/deployments/`.
2. While the API and Chain services are running and actively processing transactions (e.g., the Chain service's scheduler is polling), apply the migration job: `kubectl apply -f back-end/k8s/dev/deployments/migration-job.yaml`.
3. The migration job begins altering the `transaction` table (e.g., `1770998025293-TransactionNewKeyArray.ts` modifies key-related columns).
4. Simultaneously, the Chain service writes a new transaction row using the old schema column layout.
5. Observe that the write either fails with a constraint violation or succeeds with stale/incorrect column values, leaving the `transaction` table in an inconsistent state that does not reflect the intended post-migration schema. [7](#0-6)
