### Title
Cross-Transaction Approver Deletion via Missing Authorization Check in `removeTransactionApprover`

### Summary

The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but then deletes the approver record identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who has created at least one transaction can delete approvers from any other transaction on the server, bypassing the multi-party approval requirement.

### Finding Description

**Vulnerability class**: Authorization bypass / cross-resource state manipulation (direct analog to the external report's "one user acting on behalf of / against another user's state").

**Root cause — controller** (`back-end/apps/api/src/transactions/approvers/approvers.controller.ts`, lines 102–113):

```typescript
@Delete('/:id')
async removeTransactionApprover(
    @GetUser() user: User,
    @Param('transactionId', ParseIntPipe) transactionId: number,
    @Param('id', ParseIntPipe) id: number,
) {
    await this.approversService.getCreatorsTransaction(transactionId, user); // ✓ checks user owns transactionId
    await this.approversService.removeTransactionApprover(id);               // ✗ deletes approver `id` with NO transactionId check
    return true;
}

**Exploit path**:

1. Attacker registers as a normal user and creates any transaction (becomes its creator).
2. Attacker enumerates approver IDs (sequential integers) or learns a target approver ID through any observable channel.
3. Attacker calls:
   ```
   DELETE /transactions/{attacker_own_tx_id}/approvers/{victim_approver_id}
   ```
4. `getCreatorsTransaction` passes (attacker owns `attacker_own_tx_id`).
5. `removeTransactionApprover(victim_approver_id)` executes with no cross-check — the approver belonging to a completely different transaction is soft-deleted along with its entire subtree.
6. `emitTransactionStatusUpdate` fires for the victim transaction, potentially triggering a status re-evaluation that advances the transaction past the approval gate.

### Impact Explanation

Deleting the root approver node of a victim transaction removes the entire approval tree for that transaction. The `TransactionApprover` entity is the sole enforcement layer for multi-party approval workflows. Removing it allows the transaction to advance through the lifecycle (`WAITING_FOR_SIGNATURES` → `WAITING_FOR_EXECUTION`) without the required approvals, bypassing the organizational authorization control. This constitutes unauthorized state change and privilege escalation — a malicious user can unilaterally remove approval requirements from any transaction on the server. [5](#0-4) 

### Likelihood Explanation

- **Attacker precondition**: Must be a registered, verified user on the organization server — no admin or privileged access required.
- **Must have created at least one transaction**: trivially achievable by any user.
- **Must know a victim approver ID**: IDs are sequential auto-incremented integers starting from 1; blind enumeration is sufficient. No read access to the victim transaction is required.
- The attack requires a single authenticated HTTP DELETE request. It is repeatable and leaves no obvious trace beyond the soft-deleted record.

### Recommendation

In `removeTransactionApprover` (service), after fetching the approver, verify that the approver's root transaction matches the `transactionId` the caller is authorized to modify:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // NEW: verify the approver belongs to the authorized transaction
    const root = await this.getRootNodeFromNode(approver.id);
    if (!root || root.transactionId !== transactionId)
        throw new UnauthorizedException('Approver does not belong to this transaction');

    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);
    return result;
}
```

Pass `transactionId` from the controller into the service call, and update the controller accordingly. [1](#0-0) 

### Proof of Concept

**Setup**:
- User A creates transaction `T1` (attacker-owned).
- User B creates transaction `T2` with an approver tree (approver root ID = `42`).

**Attack**:
```http
DELETE /transactions/T1/approvers/42
Authorization: Bearer <attacker_jwt>
```

**Expected (correct) behavior**: 403 Unauthorized — approver 42 does not belong to T1.

**Actual behavior**: 200 OK — approver 42 (and its entire subtree) is soft-deleted from T2. `emitTransactionStatusUpdate` fires for T2, potentially advancing it past the approval gate without User B's required approval. [2](#0-1) [1](#0-0)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );
