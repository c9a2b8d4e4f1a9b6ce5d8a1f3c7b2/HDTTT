### Title
`IpThrottlerGuard` Rate-Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header

### Summary
The global `IpThrottlerGuard` derives its rate-limit tracking key from `req.headers['x-forwarded-for']` before falling back to `req.ip`. Because `X-Forwarded-For` is a client-supplied HTTP header, an attacker can rotate its value on every request to present a fresh identity to the throttler, bypassing the global rate limit entirely. This is the direct analog of the external report's mismatch between `_user` (the identity whose counter is reset) and `msg.sender` (the identity whose counter is checked): here the intended identity is the real client IP, but the enforced identity is an attacker-controlled string.

### Finding Description

**Root cause — `IpThrottlerGuard.getTracker`** [1](#0-0) 

```ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.headers['x-forwarded-for'] || req.ip;
  ...
  return clientIp;
}
```

`req.headers['x-forwarded-for']` is the raw, unvalidated HTTP header sent by the client. It takes priority over `req.ip` (which Express populates from the same header only after applying the configured `trust proxy` chain). An attacker can set `X-Forwarded-For` to any arbitrary string — `1.2.3.4`, `fake-ip-001`, etc. — and each distinct value becomes a separate Redis key in the throttler storage, giving the attacker a completely fresh counter.

**Where the guard is applied**

`IpThrottlerGuard` is registered as a global `APP_GUARD`, meaning it applies to every HTTP route in the API service. [2](#0-1) 

The throttler it enforces is configured with a limit of 10 000 requests/minute and 1 000 requests/second in production. [3](#0-2) 

**Exploit flow**

1. Attacker sends `POST /auth/login` with `X-Forwarded-For: 1.1.1.1` — counter for key `1.1.1.1` increments.
2. Attacker sends next request with `X-Forwarded-For: 1.1.1.2` — counter for key `1.1.1.2` starts at 0.
3. By cycling through N fake IPs the attacker can issue N × 10 000 requests/minute with zero throttling.

Even when the application sits behind Traefik (which appends the real client IP), the raw header value read by the guard is the full comma-separated string including any attacker-prepended values, e.g. `fake-ip, real-client-ip`. Each unique string is a distinct Redis key, so the bypass still works. [4](#0-3) 

### Impact Explanation

The global IP throttler is the only rate-limit layer protecting endpoints that do **not** carry the `EmailThrottlerGuard` (all non-auth routes: transaction creation, key management, observer/signer assignment, notification queries, etc.). Bypassing it allows:

- **Brute-force / enumeration at scale** against any API endpoint without hitting any server-side limit.
- **Resource exhaustion** of the PostgreSQL connection pool and NestJS worker threads, degrading service for legitimate users.

The `EmailThrottlerGuard` on `/auth/login` and `/auth/reset-password` is a separate guard keyed by `req.body.email` and is unaffected by this bypass — but all other routes lose their only rate-limit protection. [5](#0-4) 

### Likelihood Explanation

- **Attacker preconditions**: none — any unauthenticated HTTP client can set arbitrary headers.
- **Effort**: a single line of `curl -H 'X-Forwarded-For: <rotating-value>'` or a trivial script.
- **Detection difficulty**: high — each request arrives with a different apparent IP, making log-based detection ineffective.

### Recommendation

Replace the raw header read with Express's `req.ip`, which respects the `trust proxy` setting and returns the first untrusted IP in the chain:

```ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip;   // populated correctly when trust proxy is configured
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return clientIp;
}
```

Additionally, ensure the NestJS application sets `app.set('trust proxy', 1)` (or the appropriate hop count) so that `req.ip` reflects the real client IP behind Traefik/load-balancer, not the proxy's address. [6](#0-5) 

### Proof of Concept

```bash
# Bypass the global rate limit by rotating X-Forwarded-For on every request
for i in $(seq 1 50000); do
  curl -s -o /dev/null \
    -H "X-Forwarded-For: 10.0.0.$((i % 255))" \
    -X GET https://<api-host>/some-endpoint
done
```

Each request uses a different tracker key in Redis. The throttler never accumulates enough hits on any single key to trigger a 429 response, so all 50 000 requests succeed regardless of the configured `GLOBAL_MINUTE_LIMIT`. [3](#0-2)

/* eslint-disable @typescript-eslint/no-explicit-any */
import { HttpException, HttpStatus, Injectable } from '@nestjs/common';
import { ThrottlerGuard } from '@nestjs/throttler';

@Injectable()
export class IpThrottlerGuard extends ThrottlerGuard {
  protected getTracker(req: Record<string, any>): Promise<string> {
    const clientIp = req.headers['x-forwarded-for'] || req.ip;
    if (!clientIp) {
      throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
    }
    // console.log('client IP', clientIp.replace(/^.*:/, ''));
    return clientIp;
  }
}
