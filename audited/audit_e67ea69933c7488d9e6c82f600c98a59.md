### Title
Missing Authorization Check in `removeTransactionApprover` Allows Any Authenticated User to Delete Any Transaction Approver

### Summary
The `removeTransactionApprover` function in `ApproversService` performs no ownership or creator verification before deleting a transaction approver. Unlike the sibling `createTransactionApprovers` and `updateTransactionApprover` functions — which both accept a `user` parameter and call `getCreatorsTransaction` to enforce that only the transaction creator can modify approvers — `removeTransactionApprover` accepts only an approver `id` and deletes it after a bare existence check. Any authenticated user can delete any approver on any transaction they did not create, bypassing the approval workflow entirely.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the three approver-mutation functions show a clear inconsistency:

`createTransactionApprovers` accepts `user: User` and immediately calls `getCreatorsTransaction` to enforce creator-only access: [1](#0-0) 

`updateTransactionApprover` also accepts `user: User` and calls `getCreatorsTransaction` inside the transaction: [2](#0-1) 

`removeTransactionApprover`, however, accepts only `id: number` — no `user` parameter — and performs only an existence check before deleting: [3](#0-2) 

Because the function signature carries no `user` context, there is no path by which the service can enforce that the caller owns the transaction. The `getCreatorsTransaction` guard used by the other two mutations is structurally unreachable here.

The `getCreatorsTransaction` helper that enforces creator-only access is defined at: [4](#0-3) 

### Impact Explanation
An authenticated attacker (any registered user) can:
1. Enumerate approver IDs (approver IDs are sequential integers returned by the API).
2. Call `DELETE /transactions/:transactionId/approvers/:id` for approvers on transactions they do not own.
3. Remove all approvers from a transaction, collapsing the approval requirement and allowing the transaction to proceed without the intended multi-party authorization.
4. Selectively remove specific approvers to manipulate threshold logic, causing a transaction to reach `WAITING_FOR_EXECUTION` without the required approvals.

This constitutes unauthorized state mutation of the approval workflow — a core security invariant of the system.

### Likelihood Explanation
The precondition is only a valid JWT (any registered, verified user). Approver IDs are sequential integers exposed in API responses. No privileged access, leaked credentials, or physical access is required. The attack is trivially scriptable.

### Recommendation
Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` before deletion, mirroring the pattern already used in `createTransactionApprovers` and `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Enforce creator-only access, matching create/update behavior
  await this.getCreatorsTransaction(approver.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

The calling controller endpoint must also be updated to pass the authenticated user.

### Proof of Concept

1. Attacker registers and logs in as `userA` (no admin privileges).
2. `userB` (the transaction creator) creates a transaction and adds `userC` as a required approver. The API response includes the approver's `id` (e.g., `42`).
3. `userA` sends:
   ```
   DELETE /transactions/1/approvers/42
   Authorization: Bearer <userA_jwt>
   ```
4. The request reaches `removeTransactionApprover(42)`. The approver exists, so it is deleted without any check that `userA` owns the transaction.
5. `userC` is no longer an approver. The approval threshold is reduced. The transaction can advance without `userC`'s signature, violating the creator's intended approval policy.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
