### Title
Unbounded Full-Table Fetch with N+1 Async DB Calls in `getTransactionsToSign` Causes Authenticated DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches **all** non-terminal transactions from the database with no `take`/`skip` limit, then performs a sequential async `userKeysToSign` call (involving SDK deserialization and a DB query) for every row before applying pagination in memory. As the transaction table grows, any authenticated user can trigger this endpoint to exhaust server memory, CPU, and database connections with a single request. The endpoint remains live and reachable despite being marked "NO LONGER USED BY FRONT-END."

### Finding Description

**Root cause — unbounded fetch with no DB-level limit:**

`getTransactionsToSign` calls `this.repo.find(...)` at line 295 with no `take` or `skip` option:

```typescript
const transactions = await this.repo.find({
  where: whereForUser,
  relations: ['groupItem'],
  order,
  // ← no take, no skip
});
``` [1](#0-0) 

This loads every transaction whose status is not in `[EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED]` — i.e., every active transaction in the system — into the Node.js process heap.

**N+1 async calls per row:**

The code then iterates over every fetched transaction and awaits `userKeysToSign` for each one:

```typescript
for (const transaction of transactions) {
  try {
    const keysToSign = await this.userKeysToSign(transaction, user);
    if (keysToSign.length > 0) result.push({ transaction, keysToSign });
  } catch (error) { ... }
}
``` [2](#0-1) 

`userKeysToSign` → `userKeysRequiredToSign` → `keysRequiredToSign` deserializes the raw transaction bytes via the Hedera SDK (`SDKTransaction.fromBytes`) and calls `transactionSignatureService.computeSignatureKey(transaction)` — a non-trivial operation — for every row. [3](#0-2) [4](#0-3) 

**Pagination applied only after full processing:**

The `offset`/`limit` from the request are used only at the very end:

```typescript
return {
  totalItems: result.length,
  items: result.slice(offset, offset + limit),
  ...
};
``` [5](#0-4) 

The `PaginationParams` decorator enforces `size ≤ 100`, but this cap is irrelevant because the DB fetch and the N+1 loop are unbounded regardless of the requested page size. [6](#0-5) 

**Endpoint is live and reachable:**

`GET /transactions/sign` is exposed in the controller, guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` — any registered, verified user can call it: [7](#0-6) 

The comment "NO LONGER USED BY FRONT-END" confirms the front-end migrated to the optimized `/transaction-nodes` endpoint, but the old endpoint was never removed and remains accessible to any API client.

**Contrast with the fixed endpoint:**

The replacement `TransactionNodesService.getTransactionNodes` issues a single parameterized SQL query with all filtering done in the database and returns results directly — no in-process loop, no N+1 calls: [8](#0-7) 

### Impact Explanation

With N active transactions in the system:
- **Memory**: the entire active transaction set (including `transactionBytes` blobs) is loaded into the Node.js heap per request.
- **CPU**: N SDK deserialization operations (`SDKTransaction.fromBytes`) are performed sequentially.
- **DB connections**: N sequential queries are issued against the database, holding a connection slot for the full duration.
- **Amplification**: concurrent requests from a single user multiply the effect linearly.

At scale (thousands of active transactions), a single request can cause the API process to OOM-crash or become unresponsive for all users, constituting a full service outage.

### Likelihood Explanation

- **Attacker precondition**: a valid account on the organization server (registered + verified user). No admin or privileged role required.
- **Attack path**: `GET /transactions/sign?page=1&size=1` — the smallest valid request still triggers the full unbounded fetch and N+1 loop.
- **Realistic scenario**: an organization accumulates thousands of active transactions over time (normal operation). Any user — including a malicious insider or a compromised account — can then trigger the DoS with a single HTTP request.
- **No rate-limit evidence** was found in the codebase for this endpoint.

### Recommendation

1. **Remove or disable the endpoint** — since the front-end no longer uses `GET /transactions/sign`, the simplest fix is to delete or gate it behind a feature flag.
2. **If the endpoint must be kept**, rewrite `getTransactionsToSign` to push all filtering (including the "user must sign" predicate) into the database query, applying `take`/`skip` at the DB level — mirroring the approach used in `getTransactionNodesQuery`.
3. **Do not apply pagination after in-memory processing** — `result.slice(offset, offset + limit)` is not a substitute for DB-level pagination when the dataset is unbounded.

### Proof of Concept

**Preconditions:**
- Organization has ≥ 10,000 active (non-terminal) transactions.
- Attacker has a valid verified user account.

**Steps:**
1. Authenticate and obtain a JWT token.
2. Issue: `GET /transactions/sign?page=1&size=1` with the JWT.
3. The server executes `this.repo.find(...)` with no limit, loading all 10,000+ transaction rows (including `transactionBytes` blobs) into heap.
4. The server then sequentially awaits `userKeysToSign(tx, user)` for each of the 10,000+ rows, each deserializing the SDK transaction bytes and querying the DB.
5. The request takes tens of seconds to minutes; concurrent identical requests exhaust the event loop, memory, and DB connection pool, causing 503s for all other users.

**Expected outcome:** API becomes unresponsive or crashes (OOM) under a realistic transaction volume, triggered by a single authenticated user with no privileged access.
