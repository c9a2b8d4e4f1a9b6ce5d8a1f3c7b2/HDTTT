### Title
`TransferTransactionModel.getSigningAccounts()` Only Inspects HBAR Transfers, Missing Token and NFT Transfer Senders

### Summary
Both the back-end and front-end `TransferTransactionModel` implementations compute required signers by iterating exclusively over `hbarTransfersList`. A Hedera `TransferTransaction` can also carry fungible-token transfers and NFT transfers, whose senders are never added to the signing-accounts set. This mirrors the external report's root cause exactly: a function that claims to handle all transfer variants only handles one.

### Finding Description

**Vulnerability class:** Incomplete type/variant coverage in signing-account computation.

**Affected files and lines:**

- `back-end/libs/common/src/transaction-signature/model/transfer-transaction.model.ts` lines 10–19
- `front-end/src/renderer/utils/transactionSignatureModels/transfer-transaction.model.ts` lines 6–16

Both implementations are identical in their omission:

```ts
getSigningAccounts(): Set<string> {
  const accounts = new Set<string>();
  // ONLY hbarTransfersList is iterated
  for (const transfer of this.transaction.hbarTransfersList) {
    if (transfer.amount.isNegative() && !transfer.isApproved) {
      accounts.add(transfer.accountId.toString());
    }
  }
  return accounts;   // tokenTransferList and nftTransfersList are never touched
}
``` [1](#0-0) [2](#0-1) 

The Hedera SDK `TransferTransaction` exposes three distinct transfer lists:
- `hbarTransfersList` — HBAR transfers
- `tokenTransferList` — fungible HTS token transfers
- `nftTransfersList` — NFT transfers

Only the first is consulted. The same omission applies to `getReceiverAccounts()`, which also only reads `hbarTransfersList`, so token/NFT receivers with `receiverSignatureRequired = true` are also silently dropped. [3](#0-2) 

The result of `getSigningAccounts()` is consumed directly by `TransactionSignatureService.computeSignatureKey()` to build the `KeyList` of required signatures for the organization multi-sig workflow:

```ts
const requirements = this.extractSignatureRequirements(transactionModel);
await this.addSigningAccountKeys(signatureKey, transaction, requirements.signingAccounts);
await this.addReceiverAccountKeys(signatureKey, transaction, requirements.receiverAccounts, showAll);
``` [4](#0-3) 

When the computed `KeyList` is incomplete, the tool either (a) submits a transaction to Hedera that is missing required cryptographic signatures — causing an on-chain rejection — or (b) in the case where the token sender is also the fee payer, silently relies on the fee-payer signature to satisfy the token-sender requirement, bypassing the organization's explicit per-role approval accounting.

### Impact Explanation

**Primary impact — transaction failure / workflow DoS:** Any `TransferTransaction` that moves HTS fungible tokens or NFTs where the token sender is not the fee payer will be submitted to Hedera without the sender's signature. Hedera will reject it with `INVALID_SIGNATURE`. In an organization context this wastes transaction fees and permanently blocks that transaction from executing through the tool's workflow.

**Secondary impact — approval-threshold bypass:** When the token sender is the same account as the fee payer, the tool collects that account's key once (as fee payer) and never adds it again as a "signing account." The organization's multi-sig threshold is computed from the union of keys in the `KeyList`. If the intended security model requires the token sender to be an explicit, separately-counted approver (e.g., in a threshold key structure), the threshold is silently under-counted, allowing the transaction to be marked ready with fewer approvals than the policy requires.

### Likelihood Explanation

Any user of the organization mode who creates a `TransferTransaction` containing HTS token or NFT transfers triggers this path. The Hedera ecosystem heavily uses HTS tokens; this is a routine operation, not an edge case. No privilege is required — a normal authenticated user can create such a transaction.

### Recommendation

Extend both `getSigningAccounts()` implementations to also iterate over `tokenTransferList` and `nftTransfersList`, adding senders (negative-amount entries for fungible tokens, `senderAccountId` for NFTs) that are not approved transfers. Apply the same fix to `getReceiverAccounts()` for token/NFT receivers. The `AccountAllowanceApproveTransactionModel` already demonstrates the correct pattern of covering all three approval types:

```ts
this.transaction.hbarApprovals.forEach(...);
this.transaction.tokenApprovals.forEach(...);
this.transaction.tokenNftApprovals.forEach(...);
``` [5](#0-4) 

Apply the same exhaustive-coverage pattern to `TransferTransactionModel` in both the back-end and front-end.

### Proof of Concept

1. In organization mode, create a `TransferTransaction` that transfers an HTS fungible token from account `A` (token sender, not the fee payer) to account `B`.
2. Observe that `TransferTransactionModel.getSigningAccounts()` returns an empty set (or only the fee payer).
3. The tool's signature-collection workflow does not prompt account `A` to sign.
4. The transaction is submitted to Hedera without account `A`'s signature.
5. Hedera returns `INVALID_SIGNATURE` and the transaction fails.

Alternatively, set the token sender = fee payer = account `A`, observe that the tool marks the transaction ready after collecting only the fee-payer signature, and confirm that the organization's approval record shows zero "signing account" approvals collected — demonstrating the approval-threshold under-count.
