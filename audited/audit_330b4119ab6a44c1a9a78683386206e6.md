### Title
`importSignatures` Silently Discards All Imported Signatures Due to Ignored `addSignature` Return Value

### Summary
In `back-end/apps/api/src/transactions/transactions.service.ts`, the `importSignatures` function calls `sdkTransaction.addSignature(publicKey, map)` at line 552 but discards the return value. The Hedera SDK's `addSignature` method returns the updated transaction object; without capturing it, the local `sdkTransaction` variable remains the original unsigned transaction. The subsequent `sdkTransaction.toBytes()` at line 555 then serializes the original, unmodified bytes into the database. Every signature import silently succeeds (HTTP 200 with no error) while the stored transaction bytes are never updated.

### Finding Description
**Root cause:** `SDKTransaction.addSignature()` in the `@hiero-ledger/sdk` returns the transaction with the signature applied. The caller must reassign the result. This is correctly done in `signers.service.ts`:

```typescript
// signers.service.ts line 251 — CORRECT
sdkTransaction = sdkTransaction.addSignature(publicKey, map);
```

But in `transactions.service.ts` the return value is thrown away:

```typescript
// transactions.service.ts line 552 — BUG
sdkTransaction.addSignature(publicKey, map);   // return value ignored
```

Immediately after, the stale object is serialized and written to the database:

```typescript
// transactions.service.ts line 555
transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
```

**Exploit flow:**
1. Authenticated user (no special role required) calls the `importSignatures` endpoint with a valid signature map.
2. `importSignatures` validates the signatures successfully (line 546–549).
3. `addSignature` is called but its return value is discarded (line 552).
4. `sdkTransaction.toBytes()` serializes the original, unsigned transaction (line 555).
5. The database is updated with the original bytes — signatures are gone.
6. The API returns `{ id }` (success) with no error indication.
7. The transaction never accumulates the required signatures and can never be executed.

### Impact Explanation
Any signature imported through this endpoint is permanently lost. The transaction bytes stored in the database remain identical to what they were before the import. Because the API returns success, neither the caller nor any monitoring system detects the failure. The transaction is permanently stuck in `WAITING_FOR_SIGNATURES` state and can never be executed, constituting an unrecoverable corruption of transaction state for every user of this feature.

### Likelihood Explanation
The `importSignatures` endpoint is a standard, documented user-facing feature (importing signatures from external `.sigmap` files). No elevated privileges are required — any authenticated user with access to a transaction can trigger this path. The bug fires on every invocation of the endpoint, not just under edge-case conditions.

### Recommendation
Capture the return value of `addSignature`, mirroring the correct pattern already used in `signers.service.ts`:

```typescript
// transactions.service.ts — fix
for (const publicKey of publicKeys) {
  sdkTransaction = sdkTransaction.addSignature(publicKey, map);  // capture return value
}
transaction.transactionBytes = Buffer.from(sdkTransaction.toBytes());
```

### Proof of Concept
1. Create a transaction in `WAITING_FOR_SIGNATURES` state.
2. Sign it locally to produce a valid `SignatureMap`.
3. POST the signature map to the `importSignatures` endpoint.
4. Observe the API returns `{ id: <n> }` (success).
5. Fetch the stored `transactionBytes` from the database.
6. Deserialize them and call `getSignatures()` — the signature map is empty, proving the bytes were never updated.
