### Title
Any Authenticated User Can Retrieve All Historical Transactions Without Authorization

### Summary
The `GET /transactions/history` endpoint in `TransactionsController` applies no user-based access control. Any authenticated user can retrieve the complete history of every transaction in the system — regardless of whether they are the creator, signer, approver, or observer of those transactions. This is the same vulnerability class as the external report (auth — missing caller restriction), mapped to this protocol's API layer.

### Finding Description

**Root cause:** `getHistoryTransactions` in the controller passes no user identity to the service, and the service applies no user-scoped `WHERE` clause.

Controller — no `@GetUser()` parameter, no user passed to service: [1](#0-0) 

Service — query filters only on `status`, never on `userId`, `creatorKey`, `signers`, or `observers`: [2](#0-1) 

**Contrast with `getTransactions`**, which correctly scopes results to the requesting user via `signers.userId`, `observers.userId`, and `creatorKey.userId`: [3](#0-2) 

The class-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only verify that the caller holds a valid JWT — they do not restrict which transactions are returned: [4](#0-3) 

**Exploit path:**
1. Attacker registers a legitimate account and obtains a valid JWT.
2. Attacker sends `GET /transactions/history` (optionally with pagination/filter params).
3. The endpoint returns all `EXECUTED`, `FAILED`, `EXPIRED`, `CANCELED`, and `ARCHIVED` transactions from every user in the organization — names, types, statuses, `transactionId`, `mirrorNetwork`, `validStart`, `executedAt`, etc.

### Impact Explanation
An attacker with any valid account can enumerate the full operational history of the organization: which Hedera accounts are managed, what transaction types are executed (account updates, node operations, file updates), when they ran, and whether they succeeded or failed. In an organization managing treasury or node operations on Hedera, this constitutes a material confidential data breach. The impact is concrete and immediately reproducible — no further exploitation steps are required.

### Likelihood Explanation
Likelihood is high. The attacker precondition is only a valid user account — the lowest possible privilege level in the system. No special role, no leaked credentials, no internal network access. The endpoint is actively used by the front-end (it carries no `/* NO LONGER USED BY FRONT-END */` comment, unlike the sibling endpoints at lines 110, 157, and 199). Any registered user can trigger this with a single HTTP GET.

### Recommendation
Apply the same user-scoping pattern used in `getTransactions`. Pass the authenticated user into `getHistoryTransactions` and add a `WHERE` clause that restricts results to transactions where the user is the creator, a signer, an observer, or an approver — mirroring the access model enforced by `verifyAccess`: [5](#0-4) 

### Proof of Concept

```
# Step 1 — obtain a JWT for any registered user
POST /auth/login
{ "email": "attacker@example.com", "password": "..." }
→ { "accessToken": "<JWT>" }

# Step 2 — call the unguarded history endpoint
GET /transactions/history?page=1&limit=50
Authorization: Bearer <JWT>

# Step 3 — observe full transaction history for ALL users
→ 200 OK
{
  "totalItems": 312,
  "items": [
    { "id": 1, "name": "Treasury HBAR Transfer", "type": "CRYPTO_TRANSFER",
      "status": "EXECUTED", "transactionId": "0.0.1234@...", ... },
    { "id": 2, "name": "Node Admin Key Rotation", "type": "NODE_UPDATE",
      "status": "FAILED", ... },
    ...
  ]
}
```

The attacker receives records belonging to users they have no relationship with, with no error or access denial.

  async getHistoryTransactions(
    { page, limit, size, offset }: Pagination,
    filter: Filtering[] = [],
    sort: Sorting[] = [],
  ): Promise<PaginatedResourceDto<Transaction>> {
    const order = getOrder(sort);

    const findOptions: FindManyOptions<Transaction> = {
      where: {
        ...getWhere<Transaction>(filter),
        status: this.getHistoryStatusWhere(filter),
      },
      order,
      relations: ['groupItem', 'groupItem.group'],
      skip: offset,
      take: limit,
    };

    const [transactions, total] = await this.repo
      .createQueryBuilder()
      .setFindOptions(findOptions)
      .getManyAndCount();

    return {
      totalItems: total,
      items: transactions,
      page,
      size,
    };
  }

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
  }
