### Title
Missing Transaction Existence and Authorization Validation in `createComment` Allows Any Authenticated User to Comment on Arbitrary Transactions

### Summary
`CommentsService.createComment` performs a state-mutating database write using a caller-supplied `transactionId` without first verifying that the transaction exists or that the requesting user has any relationship to it. This is the direct analog to the missing `onlyValidCollectionId()` modifier: a function accepts an ID parameter and mutates state without validating the ID refers to a real, accessible entity. Any authenticated user can create comments on transactions they have no involvement in.

### Finding Description
The root cause is in `back-end/apps/api/src/transactions/comments/comments.service.ts`, `createComment` (lines 15–23):

```typescript
async createComment(
  user: User,
  transactionId: number,
  dto: CreateCommentDto,
): Promise<TransactionComment> {
  const comment = this.repo.create(dto);
  comment['transaction'].id = transactionId;
  comment.user = user;
  return this.repo.save(comment);
}
``` [1](#0-0) 

There are two missing checks:

1. **No transaction existence check**: The function never queries the database to confirm a `Transaction` with `id = transactionId` exists before writing. Every other analogous service method performs this check explicitly. For example, `createTransactionObservers` does:
   ```typescript
   const transaction = await this.entityManager.findOne(Transaction, { where: { id: transactionId }, ... });
   if (!transaction) throw new BadRequestException(ErrorCodes.TNF);
   ``` [2](#0-1) 

2. **No authorization check**: The function never verifies that `user` is the creator, observer, signer, or approver of the target transaction. All other transaction-scoped write operations enforce this. `createTransactionObservers` checks `transaction.creatorKey?.userId !== user.id`; `getTransactionObserversByTransactionId` checks all four roles before allowing access. [3](#0-2) [4](#0-3) 

The exploit path is:
1. Attacker authenticates as any valid user (no privilege required).
2. Attacker sends `POST /transactions/<victim_transactionId>/comments` with arbitrary body.
3. `createComment` is called with the victim's `transactionId`.
4. No existence or authorization check is performed.
5. The comment is written to the database linked to the victim transaction.

### Impact Explanation
- **Unauthorized state mutation**: Any authenticated user can inject comments into transactions they have no relationship to (not creator, observer, signer, or approver). This corrupts the comment history of any transaction in the system.
- **Data integrity**: If the database lacks a foreign key constraint on `transactionId`, comments can be created for non-existent transaction IDs, producing orphaned records. If a FK constraint exists, the DB error is unhandled and leaks internal error details.
- **Confidentiality side-channel**: By observing whether the save succeeds or fails, an attacker can enumerate valid transaction IDs.

### Likelihood Explanation
- Attacker precondition: a valid JWT token (any registered, verified user).
- The endpoint is reachable via standard HTTP with no additional privilege.
- The `transactionId` is a sequential integer (visible from the attacker's own transactions), making enumeration trivial.
- No rate limiting or additional guard is applied at the controller level beyond `JwtAuthGuard` and `VerifiedUserGuard`. [5](#0-4) 

### Recommendation
Add the two missing guards at the start of `createComment`, mirroring the pattern used in `createTransactionObservers`:

```typescript
async createComment(user: User, transactionId: number, dto: CreateCommentDto) {
  // 1. Validate transaction exists
  const transaction = await this.entityManager.findOne(Transaction, {
    where: { id: transactionId },
    relations: ['creatorKey', 'observers', 'signers'],
  });
  if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

  // 2. Validate user has a relationship to the transaction
  const isAuthorized =
    transaction.creatorKey?.userId === user.id ||
    transaction.observers.some(o => o.userId === user.id) ||
    transaction.signers.some(s => s.userKey?.userId === user.id);
  if (!isAuthorized) throw new UnauthorizedException('No permission to comment on this transaction');

  const comment = this.repo.create(dto);
  comment.transaction = transaction;
  comment.user = user;
  return this.repo.save(comment);
}
```

### Proof of Concept

**Precondition**: Attacker has a valid JWT (`attacker_token`). Victim transaction ID is `42` (attacker has no relationship to it).

```http
POST /transactions/42/comments
Authorization: Bearer <attacker_token>
Content-Type: application/json

{ "message": "injected comment" }
```

**Expected (current behavior)**: HTTP 201, comment record created in DB with `transactionId = 42`, `userId = attacker_id`. The attacker has successfully written to a transaction they have no access to.

**Expected (after fix)**: HTTP 401 Unauthorized.

The same request with a non-existent `transactionId` (e.g., `99999`) currently either produces an unhandled DB error or creates an orphaned record, depending on FK constraint configuration. [1](#0-0)
