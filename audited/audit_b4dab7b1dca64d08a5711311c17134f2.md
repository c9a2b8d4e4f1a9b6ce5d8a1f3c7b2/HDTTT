### Title
`removeTransactionApprover` Passes `null` `transactionId` to `emitTransactionStatusUpdate` for Child Approver Nodes, Silently Dropping Status-Update Notifications

---

### Summary

In `ApproversService.removeTransactionApprover`, after deleting an approver node, the code emits a transaction status-update notification using `approver.transactionId`. By design, child approver nodes (those with a `listId` set) always have `transactionId = null`. When a child node is removed, `emitTransactionStatusUpdate` is called with `entityId: null`, causing the downstream notification pipeline to silently skip the event. All transaction participants (creator, signers, observers, other approvers) receive no notification about the approver-tree change.

---

### Finding Description

**Root cause — wrong field used as the notification key:**

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `removeTransactionApprover` is:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);
    const result = await this.removeNode(approver.id);
    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{ entityId: approver.transactionId }],   // ← BUG
    );
    return result;
}
``` [1](#0-0) 

**Why `approver.transactionId` is `null` for child nodes:**

The `TransactionApprover` entity explicitly documents the invariant: *"If the approver has a `listId`, then `transactionId` should be null."* [2](#0-1) 

This is confirmed by the service's own `createTransactionApprovers` logic, which sets `transactionId: null` for every non-root node: [3](#0-2) 

And by the unit-test assertions that verify child nodes are inserted with `transactionId: null`: [4](#0-3) 

**How the null propagates to a silent drop:**

`emitTransactionStatusUpdate` publishes `{ entityId: null }` to NATS. The consumer, `processTransactionStatusUpdateNotifications`, does:

```typescript
for (const { entityId: transactionId } of events) {
    const transaction = transactionMap.get(transactionId); // get(null) → undefined
    if (!transaction) {
        console.warn(`Transaction ${transactionId} not found, skipping...`);
        continue;                                           // silently skipped
    }
    ...
}
``` [5](#0-4) 

No in-app notification, no email, no WebSocket push is sent to any participant.

**Contrast with the correct pattern used elsewhere:**

`updateTransactionApprover` correctly resolves the root node first and uses `rootNode.transactionId`: [6](#0-5) 

`createTransactionApprovers` also uses the `transactionId` parameter directly (always the root ID): [7](#0-6) 

---

### Impact Explanation

When any child approver node is deleted, all transaction participants (creator, signers, observers, other approvers) receive zero notification. The multi-signature orchestration workflow depends on real-time WebSocket and email notifications to coordinate signing. Silently dropping the event means:

- Participants are not informed that approval requirements have changed.
- The transaction's approval-tree state diverges from what participants believe it to be.
- The integrity of the multi-signature trust model is broken for every removal of a non-root approver.

---

### Likelihood Explanation

Any authenticated user who is the creator of a transaction with a nested approver tree (threshold/list structure) can trigger this by removing a child approver node via the DELETE approver endpoint. Nested approver trees are a documented, first-class feature. The bug fires deterministically every time a child node (one with `listId != null`) is removed — no race condition or special timing required.

---

### Recommendation

Before emitting the notification, resolve the root node to obtain the correct `transactionId`, exactly as `updateTransactionApprover` already does:

```typescript
async removeTransactionApprover(id: number): Promise<void> {
    const approver = await this.getTransactionApproverById(id);
    if (!approver) throw new BadRequestException(ErrorCodes.ANF);

    // Resolve the root to get the correct transactionId
    const rootNode = await this.getRootNodeFromNode(approver.id);
    const transactionId = rootNode?.transactionId ?? approver.transactionId;

    const result = await this.removeNode(approver.id);

    emitTransactionStatusUpdate(
      this.notificationsPublisher,
      [{ entityId: transactionId }],   // ← correct
    );
    return result;
}
```

---

### Proof of Concept

1. Authenticate as a user and create a transaction.
2. Add a nested approver tree (a threshold list with at least one child user-approver). The root node gets `transactionId = <tx_id>`, child nodes get `transactionId = null`.
3. Call `DELETE /transactions/:txId/approvers/:childApproverId` where `:childApproverId` is the ID of any child node.
4. Observe that `removeNode` succeeds (the DB row is soft-deleted), but `emitTransactionStatusUpdate` is called with `entityId: null`.
5. In the notifications service log, observe the warning: `Transaction null not found, skipping status-update notifications`.
6. Confirm that no WebSocket event, in-app notification, or email is delivered to any transaction participant. [1](#0-0) [2](#0-1) [5](#0-4)
