### Title
Any Authenticated User Can Remove Transaction Approvers Without Ownership Verification

### Summary
The `removeTransactionApprover()` function in the approvers service performs no creator/ownership check before deleting a `TransactionApprover` record. Every other mutating operation on approvers (`createTransactionApprovers`, `updateTransactionApprover`) enforces that the calling user is the transaction creator via `getCreatorsTransaction()`, but `removeTransactionApprover()` omits this check entirely. Any authenticated user can delete any approver from any transaction they do not own.

### Finding Description
**Root cause:** `removeTransactionApprover()` accepts only an approver `id` and performs no user-identity check. [1](#0-0) 

The function fetches the approver by ID, verifies it exists, then immediately deletes it and emits a status update — with no check that the caller owns the parent transaction.

Contrast with `createTransactionApprovers()`, which calls `getCreatorsTransaction()` as its first action: [2](#0-1) 

And `updateTransactionApprover()`, which also calls `getCreatorsTransaction()` inside the transaction block: [3](#0-2) 

`getCreatorsTransaction()` enforces the ownership invariant: [4](#0-3) 

The controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only enforce **authentication** (valid JWT, not blacklisted, email verified) — not **authorization** (caller owns the transaction). [5](#0-4) 

**Exploit path:**
1. Attacker registers as a normal user and obtains a valid JWT.
2. Attacker learns (or enumerates) a `TransactionApprover` ID belonging to a transaction they do not own. Approver IDs are sequential integers returned in API responses visible to any authenticated user who is an observer/signer on any transaction.
3. Attacker sends `DELETE /transactions/{transactionId}/approvers/{approverId}` with their own JWT.
4. The service deletes the approver record and emits a status update — no ownership check fires.

### Impact Explanation
An attacker can silently strip required approvers from any transaction in the system. For a transaction requiring a threshold of approvals (e.g., 2-of-3), removing one approver lowers the effective threshold or eliminates a required signer, allowing the transaction to proceed to execution without the intended governance controls. This is an unauthorized state change that breaks the integrity of the multi-party approval workflow — the core security property of the organizational mode.

### Likelihood Explanation
The attacker only needs a valid organization account (normal user, no admin). Approver IDs are sequential integers exposed in API responses. No special timing, race condition, or cryptographic break is required. The attack is a single authenticated HTTP DELETE request.

### Recommendation
Add the creator ownership check to `removeTransactionApprover()`, passing the calling `User` and verifying via `getCreatorsTransaction()` before executing the delete, consistent with `createTransactionApprovers()` and `updateTransactionApprover()`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root transaction and verify caller is creator
  const root = await this.getRootNodeFromNode(approver.id);
  await this.getCreatorsTransaction(root.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

The controller's delete handler must also be updated to inject and pass the `@GetUser()` user.

### Proof of Concept
**Preconditions:** Two accounts — `victim` (transaction creator) and `attacker` (normal user). Victim has created a transaction with approver record ID `42`.

```
# Step 1: Attacker logs in
POST /auth/login  { email: "attacker@org.com", password: "..." }
→ { accessToken: "attacker_jwt" }

# Step 2: Attacker deletes victim's approver
DELETE /transactions/7/approvers/42
Authorization: Bearer attacker_jwt

→ HTTP 200 OK
```

**Expected (correct) behavior:** HTTP 401/403 — caller is not the transaction creator.
**Actual behavior:** Approver record 42 is deleted; transaction approval threshold is silently broken. [1](#0-0)

```typescript
  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
```
