### Title
Global IP Rate Limiter Bypassed via Attacker-Controlled `X-Forwarded-For` Header, Enabling Unlimited Brute-Force on Login

### Summary
`IpThrottlerGuard`, the sole global rate limiter protecting all API endpoints, derives its tracking key from the attacker-controlled `X-Forwarded-For` request header without any validation. By rotating arbitrary spoofed IP values in this header, an unauthenticated attacker can assign each request to a fresh rate-limit bucket, completely nullifying the 100 req/min and 10 req/sec limits. Because the per-email `EmailThrottlerGuard` on `/auth/login` is ordered after `LocalAuthGuard` and therefore never executes on failed credential attempts, the IP throttler is the only protection against brute-force — and it is trivially bypassed.

### Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`:**

```typescript
// back-end/apps/api/src/guards/ip-throttler.guard.ts:8
const clientIp = req.headers['x-forwarded-for'] || req.ip;
```

The guard unconditionally prefers the `x-forwarded-for` header over the socket-level `req.ip`. This header is set by the HTTP client and is never stripped or validated by the server. The returned string becomes the Redis key used to count requests against the configured limits. [1](#0-0) 

**Global registration — `ApiModule`:**

`IpThrottlerGuard` is registered as `APP_GUARD`, making it the first guard executed on every incoming request across the entire API surface. [2](#0-1) 

**Why `EmailThrottlerGuard` does not compensate on `/auth/login`:**

The login route is decorated as:
```typescript
@UseGuards(LocalAuthGuard, EmailThrottlerGuard)
```
NestJS executes guards in declaration order. `LocalAuthGuard` invokes the Passport local strategy and throws `UnauthorizedException` on wrong credentials — before `EmailThrottlerGuard` ever runs. The email-based throttle therefore only fires on *successful* logins, providing zero protection against failed credential attempts. [3](#0-2) 

**End-to-end exploit path:**

1. Attacker targets `POST /auth/login` with a known or enumerated email address.
2. Each request includes `X-Forwarded-For: <unique_random_ip>` (e.g., `1.2.3.4`, `1.2.3.5`, …).
3. `IpThrottlerGuard.getTracker()` returns the spoofed IP; Redis increments a counter for that key.
4. Because each request uses a fresh key, no bucket ever reaches the limit.
5. `LocalAuthGuard` performs bcrypt/argon2 comparison on every attempt, consuming CPU.
6. The attacker iterates a password list at arbitrary speed with no server-side enforcement.

### Impact Explanation

- **Credential brute-force**: The attacker can submit an unlimited number of password guesses against any known user email. Bcrypt/argon2 hashing is intentionally slow; at scale this also exhausts server CPU.
- **OTP email flooding**: `POST /auth/reset-password` uses only `EmailThrottlerGuard` (no `LocalAuthGuard` before it), but the global IP throttler is still the only unauthenticated-path guard. Spoofing `X-Forwarded-For` bypasses it, allowing the attacker to trigger unlimited OTP emails to any user's inbox.
- **Severity**: High — direct path to account compromise via brute-force; no privileges required. [4](#0-3) 

### Likelihood Explanation

- **Attacker preconditions**: None. The endpoint is public; the `X-Forwarded-For` header requires no special tooling — a single `curl -H` flag suffices.
- **Detectability**: The attack is indistinguishable from normal traffic at the application layer because each request appears to originate from a different IP.
- **Realistic scenario**: Any attacker with a list of registered email addresses (obtainable via the signup flow's error responses or other enumeration) can run an automated credential-stuffing or dictionary attack.

### Recommendation

1. **Never trust `X-Forwarded-For` directly.** If the API sits behind a known, trusted reverse proxy (nginx, AWS ALB, etc.), configure Express's `trust proxy` setting to the exact number of trusted hops and use `req.ip` exclusively — Express will then resolve the correct client IP from the rightmost untrusted entry in the chain.

2. **Fix `IpThrottlerGuard.getTracker()`** to use only `req.ip`:
   ```typescript
   protected getTracker(req: Record<string, any>): Promise<string> {
     const clientIp = req.ip;
     if (!clientIp) {
       throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
     }
     return Promise.resolve(clientIp);
   }
   ```

3. **Reorder guards on `/auth/login`** so `EmailThrottlerGuard` runs *before* `LocalAuthGuard`, ensuring failed attempts are counted:
   ```typescript
   @UseGuards(EmailThrottlerGuard, LocalAuthGuard)
   ```

4. **Add account-level lockout** (e.g., lock after N consecutive failures) as a defence-in-depth measure independent of network-layer throttling.

### Proof of Concept

```bash
# Brute-force /auth/login with rotating spoofed IPs — no 429 is ever returned
for i in $(seq 1 1000); do
  curl -s -o /dev/null -w "%{http_code}\n" \
    -X POST http://<api-host>/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.${i}.1" \
    -d '{"email":"victim@example.com","password":"guess'$i'"}'
done
# Expected: all responses are 401 (wrong password), never 429 (rate limited)
# Without the fix: attacker iterates the full password list unimpeded
```

Each iteration presents a distinct key to `IpThrottlerGuard`, so the Redis counter for every bucket stays at 1 — permanently below the configured limits of 100/min and 10/sec. [1](#0-0) [5](#0-4) [6](#0-5)
