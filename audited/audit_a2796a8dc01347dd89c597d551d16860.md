### Title
Cryptographically Weak PRNG (`Math.random()`) Used for Temporary Password Generation in `AuthService`

### Summary
`AuthService.generatePassword()` in `back-end/apps/api/src/auth/auth.service.ts` uses JavaScript's `Math.random()` — a non-cryptographically-secure PRNG (V8's xorshift128+) — to generate temporary passwords for newly registered users. These passwords are emailed to users and serve as the sole credential until the user changes them. An attacker who can observe enough `Math.random()` outputs from the same Node.js process can reconstruct the PRNG state and predict past or future temporary passwords, enabling account takeover of any newly registered user before they change their password.

### Finding Description
The root cause is in `generatePassword()`:

```typescript
// back-end/apps/api/src/auth/auth.service.ts, lines 147-154
private generatePassword() {
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () => String.fromCharCode(97 + Math.floor(Math.random() * 26))).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
```

`Math.random()` in V8 uses xorshift128+ with a 128-bit internal state. It is explicitly documented as not cryptographically secure. The generated password has the format `[a-z]{5}-[a-z]{5}` (10 lowercase letters), yielding 26^10 ≈ 1.4×10^14 theoretical combinations — but the effective entropy is bounded by the PRNG's predictability, not the output space.

This password is used directly in `signUpByAdmin()`:

```typescript
// lines 52-66
const tempPassword = this.generatePassword();
// ...
user = await this.usersService.createUser(dto.email, tempPassword);
// ...
emitUserRegistrationEmail(..., { url, tempPassword, downloadUrl });
```

The temporary password is the only credential for the new account until the user explicitly changes it. The same Node.js process also calls `Math.random()` in other service paths (e.g., `cache-management.service.ts`, `mirror-node.client.ts`, `email.service.ts`), which share the same PRNG state. An attacker who can trigger or observe those outputs can reconstruct the xorshift128+ state and predict the next `generatePassword()` output.

### Impact Explanation
An attacker who reconstructs the PRNG state can predict the temporary password for any newly registered user. They can authenticate as that user before the user logs in and changes their password. Once authenticated, the attacker has full access to that user's account — including their Hedera transaction history, organization memberships, and the ability to sign or approve transactions on their behalf. This constitutes unauthorized account takeover and unauthorized state changes with no privileged preconditions on the attacker's side.

### Likelihood Explanation
V8's `Math.random()` state reconstruction is a well-documented technique. Tools such as `z3` or purpose-built xorshift128+ crackers can recover the full 128-bit state from as few as 3 consecutive 64-bit outputs (or ~5 truncated outputs as produced by `Math.floor(Math.random() * 26)`). The attacker needs only to observe a handful of `Math.random()`-derived values from the same server process — achievable through any API endpoint that leaks such values — then predict the next `generatePassword()` call. The window of exploitation is the time between user registration and the user's first login and password change, which can be hours or days.

### Recommendation
Replace `Math.random()` with `crypto.randomBytes()` from Node's built-in `crypto` module, which is a CSPRNG:

```typescript
import { randomBytes } from 'crypto';

private generatePassword(): string {
  const chars = 'abcdefghijklmnopqrstuvwxyz';
  const getRandomLetters = (length: number) =>
    Array.from({ length }, () => chars[randomBytes(1)[0] % 26]).join('');
  return `${getRandomLetters(5)}-${getRandomLetters(5)}`;
}
```

Alternatively, use `randomBytes` to generate a longer hex or base64 token, which provides higher entropy and eliminates modulo bias.

### Proof of Concept

1. Observe any sequence of `Math.random()` outputs from the target Node.js API process (e.g., by triggering code paths in `cache-management.service.ts` or `mirror-node.client.ts` that call `Math.random()` and return observable results).
2. Feed the observed outputs into an xorshift128+ state-recovery tool (e.g., https://github.com/d0nutptr/v8_rand_buster or a Z3-based solver).
3. Advance the recovered PRNG state to the point of the next `generatePassword()` call (10 calls to `Math.random()`).
4. Compute `String.fromCharCode(97 + Math.floor(predicted * 26))` for each of the 10 positions to reconstruct the temporary password.
5. Use the predicted password to authenticate as the newly registered user via `POST /auth/login` before the user changes it.
