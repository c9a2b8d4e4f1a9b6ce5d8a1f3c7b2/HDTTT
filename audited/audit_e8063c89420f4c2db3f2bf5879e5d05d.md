### Title
IP-Based Rate Limit Bypass via Attacker-Controlled `X-Forwarded-For` Header in `IpThrottlerGuard`

### Summary
The `IpThrottlerGuard` uses the raw `req.headers['x-forwarded-for']` header as the primary rate-limit tracking key. Because this header is fully attacker-controlled, any external client can rotate arbitrary spoofed IP values in the header to generate unlimited independent rate-limit buckets, completely bypassing the global IP throttle. This is the direct analog of the Sybil attack described in the external report: instead of generating multiple ephemeral peer IDs, the attacker generates multiple fake IP identities via a single HTTP header.

### Finding Description

**Root cause — `IpThrottlerGuard.getTracker()`:** [1](#0-0) 

The tracker key is resolved as `req.headers['x-forwarded-for'] || req.ip`. The raw `X-Forwarded-For` header is read first and used verbatim as the Redis throttle key. An attacker sets this header to any arbitrary string on every request, creating a fresh rate-limit bucket each time.

**Global application — `ApiModule`:** [2](#0-1) 

`IpThrottlerGuard` is registered as `APP_GUARD`, meaning it is the first guard executed on every HTTP route, including all unauthenticated endpoints.

**`trust proxy: true` does not protect the raw header:** [3](#0-2) 

In production, `app.enable('trust proxy')` is equivalent to trusting all proxy hops unconditionally. Express uses this setting to populate `req.ip` from `X-Forwarded-For`, but it does not sanitize or restrict the raw `req.headers['x-forwarded-for']` value. The guard reads the raw header before `req.ip`, so the `trust proxy` setting provides no protection.

**Exploit path against unauthenticated endpoints:**

The two highest-value unauthenticated targets are:

- `POST /auth/login` — guarded by `LocalAuthGuard` + `EmailThrottlerGuard` (email-keyed) after the global IP guard. [4](#0-3) 

- `POST /auth/reset-password` — guarded only by `EmailThrottlerGuard` (email-keyed) after the global IP guard. [5](#0-4) 

The `EmailThrottlerGuard` tracks by `req.body.email`: [6](#0-5) 

An attacker who simultaneously rotates both the `X-Forwarded-For` header and the target email address bypasses both throttle layers entirely, enabling unlimited credential-stuffing or OTP-spam requests.

### Impact Explanation

- **Credential brute-force**: The global IP throttle is the only rate-limit layer that applies across all email addresses. Bypassing it allows an attacker to attempt passwords against any account at the maximum rate the server can process, limited only by network bandwidth.
- **OTP/email spam**: `POST /auth/reset-password` triggers an outbound email per call. With both throttles bypassed, an attacker can flood any user's inbox or exhaust the email-sending quota.
- **Sustained CPU/DB load**: Each login attempt invokes bcrypt password comparison (expensive by design). Unlimited requests from a single attacker IP can saturate CPU.
- **No automatic ban or reputation tracking**: There is no progressive penalty or temporary ban mechanism; each spoofed IP bucket is treated as a fresh identity.

### Likelihood Explanation

- **Zero privilege required**: The attack requires only the ability to send HTTP requests with a custom header — available to any external client.
- **Trivially scriptable**: A single `curl` loop rotating `X-Forwarded-For: <random_ip>` on each iteration is sufficient.
- **No detection**: The server logs the spoofed IP, not the real source, making forensic attribution difficult.
- **Production-only amplification**: The `trust proxy: true` flag is only set in production (`NODE_ENV === 'production'`), which is exactly where the attack matters most.

### Recommendation

Replace the raw header read with Express's already-resolved `req.ip`, which correctly reflects the real client IP when `trust proxy` is configured:

```typescript
// back-end/apps/api/src/guards/ip-throttler.guard.ts
protected getTracker(req: Record<string, any>): Promise<string> {
  const clientIp = req.ip; // Use Express-resolved IP only
  if (!clientIp) {
    throw new HttpException('Unable to determine client IP', HttpStatus.INTERNAL_SERVER_ERROR);
  }
  return Promise.resolve(clientIp);
}
```

Additionally, change `app.enable('trust proxy')` (trust all) to `app.set('trust proxy', 1)` (trust exactly one hop — the immediate load balancer/reverse proxy) so that `req.ip` is set to the real client IP as reported by the trusted proxy, not to an attacker-supplied value: [3](#0-2) 

The same raw-header pattern appears in `FrontendVersionGuard` (used only for logging, not for access control, so lower severity): [7](#0-6) 

### Proof of Concept

```bash
# Bypass IP rate limit on /auth/login by rotating X-Forwarded-For
# Each request gets a fresh rate-limit bucket

for i in $(seq 1 10000); do
  curl -s -X POST https://<server>/auth/login \
    -H "Content-Type: application/json" \
    -H "X-Forwarded-For: 10.0.${i%256}.${i}" \
    -H "X-Frontend-Version: <valid_version>" \
    -d '{"email":"victim@example.com","password":"attempt'$i'"}' &
done
```

**Expected outcome**: Every request is accepted by `IpThrottlerGuard` because each carries a unique tracker key. The `EmailThrottlerGuard` bucket for `victim@example.com` is exhausted after `ANONYMOUS_MINUTE_LIMIT` requests within 60 seconds, but by rotating the target email as well, that secondary control is also defeated. The server processes bcrypt comparisons for every request with no throttle enforcement.
