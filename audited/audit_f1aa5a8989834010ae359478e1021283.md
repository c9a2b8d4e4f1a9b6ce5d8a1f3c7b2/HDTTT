### Title
Unbounded Recursive Approver Tree in `createTransactionApprovers` Causes Server-Side DoS via Resource Exhaustion

### Summary
The `POST /transactions/:transactionId/approvers` endpoint accepts a user-controlled, arbitrarily deep and wide nested approver tree with no size or depth limit enforced at the DTO or service layer. The `createApprover` inner function recurses without bound, executing multiple database queries per node inside a single long-lived database transaction. A malicious authenticated user who owns a transaction can craft a single request with thousands of nested nodes to exhaust server CPU, memory, and database connections, causing severe service degradation or a process crash.

### Finding Description

**Root Cause — No array size or depth limit on the approver DTO:**

`CreateTransactionApproverDto` is a self-referential class with no `@ArrayMaxSize()` decorator on its `approvers` field, and `CreateTransactionApproversArrayDto` has no `@ArrayMaxSize()` on `approversArray`. [1](#0-0) 

The only constraint is `@ArrayMinSize(1)`, which prevents empty arrays but places no upper bound on size or nesting depth.

**Unbounded recursive execution path:**

`createTransactionApprovers` defines an inner async function `createApprover` that calls itself recursively for every element in `dtoApprover.approvers`: [2](#0-1) 

For every single node in the tree, the function performs up to **five sequential database round-trips** inside the same open database transaction:
1. `this.isNode()` — a `COUNT` query
2. `transactionalEntityManager.findOne(TransactionApprover, ...)` — parent lookup
3. `this.getRootNodeFromNode(...)` — a recursive CTE query
4. `transactionalEntityManager.count(User, ...)` — user existence check
5. `transactionalEntityManager.insert(TransactionApprover, ...)` — the actual insert [3](#0-2) 

The outer loop over `dto.approversArray` is also unbounded: [4](#0-3) 

**Entry point:**

The endpoint is guarded only by JWT authentication and a check that the caller is the transaction creator: [5](#0-4) 

Any registered user who has created at least one transaction satisfies this precondition.

### Impact Explanation

A single crafted HTTP request with a deeply nested or very wide approver tree causes:

1. **Stack overflow / process crash** — Node.js has a finite call stack. A tree nested ~10,000 levels deep will throw `RangeError: Maximum call stack size exceeded`, crashing the request handler and potentially the process.
2. **Database connection starvation** — Each node holds the open TypeORM transaction while awaiting sequential DB queries. A wide tree (e.g., 50,000 leaf nodes) keeps a DB connection occupied for minutes, starving other requests.
3. **CPU and memory exhaustion** — The recursive object traversal and repeated `await` chains consume heap memory proportional to tree size, degrading the entire API service for all users.

The entire API service becomes unavailable or severely degraded for the duration of the attack, affecting all users of the platform.

### Likelihood Explanation

- **Attacker precondition:** Must be a registered, authenticated user who has created at least one transaction. This is a normal, unprivileged user action.
- **Attack complexity:** Trivially low — a single `curl` or HTTP client call with a crafted JSON body.
- **No rate limiting** is visible on this endpoint in the controller or service code.
- The recursive DTO structure is explicitly documented and supported (the API docs show nested approver trees), making the attack surface obvious to any user reading the documentation. [6](#0-5) 

### Recommendation

1. **Add `@ArrayMaxSize(N)` to both DTO fields** — enforce a reasonable maximum (e.g., 100) on `approversArray` and on each nested `approvers` array:

```typescript
// create-transaction-approver.dto.ts
@IsArray()
@ArrayMinSize(1)
@ArrayMaxSize(100)          // add this
@IsOptional()
@ValidateNested({ each: true })
@Type(() => CreateTransactionApproverDto)
approvers?: CreateTransactionApproverDto[];
```

2. **Enforce a maximum nesting depth** — pass a `depth` counter into `createApprover` and throw if it exceeds a fixed limit (e.g., 10).

3. **Add `@ArrayMaxSize(N)` to `approversArray`** in `CreateTransactionApproversArrayDto`.

### Proof of Concept

**Preconditions:** Attacker is a registered user with a valid JWT and has created transaction with ID `1`.

**Request (wide flat attack — 50,000 nodes):**
```http
POST /transactions/1/approvers
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{
  "approversArray": [
    { "userId": 1 },
    { "userId": 2 },
    ... repeated 50000 times with valid userId values ...
  ]
}
```

Each entry triggers 3–5 sequential DB queries inside a single open transaction. The DB connection is held for the entire duration, starving the connection pool.

**Request (deep nested attack — stack overflow):**
```http
POST /transactions/1/approvers
Authorization: Bearer <attacker_jwt>
Content-Type: application/json

{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{
      "threshold": 1,
      "approvers": [{
        ... nested ~15000 levels deep ...
        "userId": 1
      }]
    }]
  }]
}
```

**Expected outcome:** Node.js throws `RangeError: Maximum call stack size exceeded` inside the unguarded recursive `createApprover` call at line 348 of `approvers.service.ts`, crashing the request and potentially the worker process, making the API unavailable. [7](#0-6)
