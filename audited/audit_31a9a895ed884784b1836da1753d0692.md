### Title
Any Verified User Can Claim Another User's Public Key, Permanently Blocking Their Key Registration

### Summary

The `uploadKey` endpoint in the organization back-end accepts any valid Hedera public key from any authenticated, verified user without requiring proof of ownership (i.e., no signature challenge). Because the system enforces global uniqueness of public keys across all users, a malicious verified user can pre-register a victim's public key under their own account, permanently preventing the victim from registering that key and blocking their ability to participate in transaction signing workflows.

### Finding Description

**Root Cause**

`uploadKey` in `UserKeysService` performs a global uniqueness check on `publicKey` across all users. If the key already exists and is owned by a different user, it throws `ErrorCodes.PU`. There is no cryptographic proof-of-ownership step — the caller only needs to supply a syntactically valid Hedera public key string. [1](#0-0) 

The `UploadUserKeyDto` validates only that the string is a parseable Hedera public key. No signature or challenge-response is required. [2](#0-1) 

The controller exposes this as a `POST /user/:userId?/keys` endpoint, guarded only by JWT authentication and `VerifiedUserGuard`. The `userId` URL parameter is completely ignored — the key is always assigned to the authenticated caller. [3](#0-2) 

This is confirmed by the e2e test comment: *"should upload your key if verified whatever user id is passed"* — the URL `userId` is irrelevant. [4](#0-3) 

**Exploit Flow**

1. Attacker (verified user) calls `GET /user/:victimId/keys` to enumerate the victim's registered public keys (this endpoint is accessible to all verified users).
2. Attacker calls `POST /user/keys` with `{ publicKey: <victim's key> }` before the victim has registered it (or after the victim deletes it).
3. The key is now owned by the attacker's `userId` in the `user_key` table.
4. When the victim attempts `POST /user/keys` with their own public key, `uploadKey` finds the key already owned by a different user and throws `ErrorCodes.PU` (Bad Request).
5. The victim can never register their key in the organization back-end.

### Impact Explanation

The victim is permanently blocked from registering their public key in the organization back-end. Since transaction signing workflows rely on `user.keys` to identify signers and verify signatures, the victim cannot:
- Be identified as a required signer for transactions (`userKeysRequiredToSign`)
- Submit signatures via `approveTransaction`, which looks up the key by `dto.userKeyId` from the user's own key list [5](#0-4) 

This constitutes a targeted DoS against any specific user's ability to participate in multi-signature transaction workflows within an organization.

### Likelihood Explanation

- Attacker precondition: must be a verified user in the same organization (no admin/privileged access required).
- Victim's public keys are discoverable via `GET /user/:id/keys`, which is accessible to all verified users.
- The attack is a single authenticated HTTP POST — trivially automatable.
- The victim has no recourse: the attacker owns the key record and can keep it alive (or soft-delete and re-register it) indefinitely.

### Recommendation

Require proof-of-ownership before registering a public key. The standard approach is a **challenge-response**: the server issues a random nonce, the client signs it with the private key corresponding to the public key being registered, and the server verifies the signature before persisting the key. This ensures only the holder of the private key can register the corresponding public key.

Alternatively, at minimum, enforce that a `publicKey` can only be registered once globally and cannot be claimed by a different user even after soft-deletion, and add rate-limiting to prevent bulk pre-registration attacks.

### Proof of Concept

```
# Step 1: Attacker learns victim's public key
GET /user/2/keys  (with attacker's JWT)
→ 200 OK: [{ publicKey: "abc123...", ... }]

# Step 2: Attacker claims victim's key (before victim registers it)
POST /user/keys  (with attacker's JWT)
Body: { "publicKey": "abc123..." }
→ 201 Created: key now owned by attacker's userId

# Step 3: Victim tries to register their own key
POST /user/keys  (with victim's JWT)
Body: { "publicKey": "abc123..." }
→ 400 Bad Request: ErrorCodes.PU ("Public key in use")

# Result: Victim cannot register their key; cannot sign transactions
# in the organization back-end.
```
