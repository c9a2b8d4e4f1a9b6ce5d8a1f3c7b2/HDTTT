### Title
Unbounded Transaction Creation Enables O(N) Full-Table Scan DoS in `getTransactionsToSign`

### Summary
`getTransactionsToSign` in `back-end/apps/api/src/transactions/transactions.service.ts` fetches **all** non-terminal transactions from the database with no upper bound, then iterates over every row calling `userKeysToSign` per transaction. A malicious verified user can create an unbounded number of transactions, causing this endpoint to degrade linearly for every user in the organization. This is the direct analog of the Ditto validator-registration DoS: unrestricted entity creation inflates a structure that is traversed in linear time on every subsequent request.

---

### Finding Description

**Root cause — unbounded fetch with no user-scoping:** [1](#0-0) 

```typescript
// whereForUser contains NO user-id filter — it matches every non-terminal tx
const whereForUser: FindOptionsWhere<Transaction> = {
  ...where,
  status: Not(In([EXECUTED, FAILED, EXPIRED, CANCELED, ARCHIVED])),
};

const transactions = await this.repo.find({   // ← no `take`, no user filter
  where: whereForUser,
  relations: ['groupItem'],
  order,
});

for (const transaction of transactions) {     // ← O(N) over ALL transactions
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}
```

Every call to `GET /transactions/sign` (the "Ready to Sign" tab) loads the entire non-terminal transaction table into memory and calls `userKeysToSign` — which itself performs per-transaction DB work — for each row. There is no `take`/`skip` guard and no pre-filter by user identity at the query level.

**No hard limit on transaction creation:**

The only rate-limiting in place is the general throttler: [2](#0-1) 

100 requests/minute and 10 requests/second. This slows accumulation but does not cap the total number of live transactions. A verified user can create 100 transactions per minute; over hours this yields tens of thousands of rows that persist until they expire or are manually canceled.

**Contrast with the guarded `uploadKey` path:**

User keys do have a hard cap: [3](#0-2) 

No equivalent cap exists for transactions.

**Attack flow:**

1. Attacker registers as a normal verified user (standard product flow; no admin required).
2. Attacker submits transactions at the throttle ceiling (100/min) continuously.
3. Each transaction remains in a non-terminal state until it expires on the Hedera network or is manually canceled.
4. Every legitimate user who opens the "Ready to Sign" tab triggers `getTransactionsToSign`, which now scans the entire inflated table and calls `userKeysToSign` for each row.
5. Response time grows linearly with the number of attacker-created transactions; at sufficient scale the service times out or exhausts DB connection pool resources.

---

### Impact Explanation

- **Availability**: The "Ready to Sign" endpoint becomes progressively slower for all organization users, eventually timing out entirely. This blocks the core multi-signature workflow.
- **Scope**: The scan is global (not per-user), so a single attacker degrades the experience for every user in the organization.
- **Severity**: High — the signing workflow is the critical path of the application; blocking it prevents any multi-sig transaction from being executed.

---

### Likelihood Explanation

- **Attacker preconditions**: Only a valid verified-user account is required. No admin, no leaked secrets.
- **Effort**: Low. The attacker simply calls the transaction-creation endpoint at the rate limit ceiling. No special tooling beyond a scripted HTTP client is needed.
- **Persistence**: Transactions accumulate over time; the attacker does not need to maintain a high request rate indefinitely — a burst campaign is sufficient to leave a large residual table.
- **Detection difficulty**: The behavior looks like normal transaction creation traffic; no anomaly is obvious until latency degrades.

---

### Recommendation

1. **Add a hard cap on active (non-terminal) transactions per user** — analogous to `MAX_USER_KEYS` — enforced at creation time in the transaction service.
2. **Push the user-identity filter into the database query** in `getTransactionsToSign` so only transactions where the user is a signer/creator are fetched, eliminating the global scan entirely.
3. **Add `take`/`skip` pagination** to the `repo.find` call so the result set is bounded regardless of table size.
4. **Consider a background job** to aggressively expire or archive stale transactions rather than relying solely on Hedera network expiry.

---

### Proof of Concept

**Preconditions**: attacker holds a valid JWT for a verified user account.

```bash
# Step 1 – flood transaction creation at the throttle ceiling
for i in $(seq 1 10000); do
  curl -s -X POST https://<org-backend>/transactions \
    -H "Authorization: Bearer <attacker_jwt>" \
    -H "Content-Type: application/json" \
    -d '{"name":"dos-'$i'","transactionBytes":"<valid_bytes>","mirrorNetwork":"testnet",...}' &
  # 10 concurrent, 100/min ceiling → ~6000/hour
done

# Step 2 – measure victim latency on the signing endpoint
time curl -s https://<org-backend>/transactions/sign \
  -H "Authorization: Bearer <victim_jwt>"
# Expected: response time grows proportionally to number of attacker transactions
# At ~50 000 rows: timeout or OOM on the API pod
```

**Expected outcome**: `getTransactionsToSign` response time increases linearly with the number of non-terminal transactions in the database. At sufficient scale the endpoint times out, blocking all users from accessing the "Ready to Sign" workflow.
