### Title
Single User Approval Satisfies Multiple Threshold Positions via Duplicate Approver Entries

### Summary
The `approveTransaction` function in `approvers.service.ts` updates **all** `TransactionApprover` records belonging to the calling user in a single operation. Because `createTransactionApprovers` only prevents duplicate `(userId, listId)` pairs — not duplicate `userId` values across different parent nodes — a malicious transaction creator can insert the same user under multiple distinct threshold groups. One approval call from that user then simultaneously satisfies every position they occupy, collapsing a multi-party threshold to a single-party approval.

### Finding Description

**Root cause — duplicate approver insertion is not fully prevented**

`createTransactionApprovers` calls `isNode` before inserting each leaf:

```typescript
// approvers.service.ts:250
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
```

`isNode` queries for a record matching `(listId, userId, transactionId)`:

```typescript
// approvers.service.ts:652-665
const find: FindManyOptions<TransactionApprover> = {
  where: {
    listId: typeof approver.listId === 'number' ? approver.listId : null,
    userId: typeof approver.userId === 'number' ? approver.userId : null,
    ...
    transactionId: typeof approver.listId === 'number' ? null : transactionId,
  },
};
const count = await (entityManager || this.repo).count(TransactionApprover, find);
return count > 0 && typeof approver.userId === 'number';
``` [1](#0-0) 

The guard only rejects a user who already exists **under the same parent (`listId`)**. The same `userId` can be inserted under two different parent nodes (different `listId` values) without triggering the duplicate check.

**Root cause — single approval updates all positions for the user**

`approveTransaction` fetches every approver record for the user across the entire recursive tree and updates them all in one query:

```typescript
// approvers.service.ts:553-608
const approvers = await this.getVerifiedApproversByTransactionId(transactionId, user);
const userApprovers = approvers.filter(a => a.userId === user.id);
// ...
await transactionalEntityManager
  .createQueryBuilder()
  .update(TransactionApprover)
  .set({ userKeyId: dto.userKeyId, signature: dto.signature, approved: dto.approved })
  .whereInIds(userApprovers.map(a => a.id))
  .execute();
``` [2](#0-1) 

`getVerifiedApproversByTransactionId` uses a recursive CTE that returns every node in the tree, so `userApprovers` contains **all** records for the user regardless of depth or parent: [3](#0-2) 

**Exploit path**

A malicious creator builds a tree such as:

```
Root threshold: 2 of 2
  ├── Group A (threshold 1 of 1) → userId: Alice
  └── Group B (threshold 1 of 1) → userId: Alice   ← same user, different listId
```

Both `Group A → Alice` and `Group B → Alice` pass `isNode` because they have different `listId` values. When Alice calls `POST /transactions/:id/approvers/approve` once, `userApprovers` contains both records and both are stamped with her signature. Group A and Group B are both satisfied, so the root 2-of-2 threshold is met — with only Alice's single approval.

### Impact Explanation
A malicious transaction creator can bypass the organization's multi-party approval requirement entirely. Any transaction — including high-value transfers or administrative actions — can be approved by a single colluding user, defeating the purpose of the threshold approver system. The impact is unauthorized transaction execution without the consent of the other designated approvers.

### Likelihood Explanation
The attacker must be a legitimate user who can create transactions (no admin privileges required). The `createTransactionApprovers` endpoint is available to any authenticated transaction creator. The attack requires only crafting a specific approver tree payload — a straightforward API call — and then calling the approve endpoint once. No cryptographic weakness or race condition is needed.

### Recommendation
Enforce global uniqueness of `userId` within a transaction's entire approver tree, not just within a single parent node. Before inserting a leaf approver, query whether the `userId` already exists anywhere in the tree for that `transactionId`:

```typescript
// In createTransactionApprovers, before inserting a userId leaf:
const existingForUser = await this.getApproversByTransactionId(
  transactionId, dtoApprover.userId, transactionalEntityManager
);
if (existingForUser.length > 0)
  throw new Error('A user may only appear once in the approver tree');
```

Alternatively, change `approveTransaction` so that it only updates the **single** approver record that is still unsigned (not all records for the user), and require a separate approval call for each position the user occupies — though the former fix is cleaner.

### Proof of Concept

1. Authenticated user (creator) creates a transaction.
2. Creator calls `POST /transactions/:id/approvers` with:
   ```json
   {
     "approversArray": [{
       "threshold": 2,
       "approvers": [
         { "threshold": 1, "approvers": [{ "userId": 42 }] },
         { "threshold": 1, "approvers": [{ "userId": 42 }] }
       ]
     }]
   }
   ```
   Both inner nodes have `userId: 42` but different `listId` values — `isNode` passes for both.
3. User 42 calls `POST /transactions/:id/approvers/approve` once with a valid signature.
4. `userApprovers` contains two records (one per inner node). Both are updated with the same signature.
5. Both inner thresholds (1-of-1) are satisfied, satisfying the outer 2-of-2 threshold.
6. The transaction proceeds to execution with only one unique approver's consent. [4](#0-3) [5](#0-4)
