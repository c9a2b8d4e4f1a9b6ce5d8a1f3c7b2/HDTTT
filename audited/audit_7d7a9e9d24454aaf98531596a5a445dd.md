### Title
Missing Transaction Status Validation Allows Modification of Terminal-State Transactions

### Summary

The `createTransactionApprovers` and `createTransactionObservers` service functions do not validate the transaction's current status before executing. A transaction creator can successfully add approvers or observers to transactions already in terminal states (`EXECUTED`, `CANCELED`, `EXPIRED`, `FAILED`, `ARCHIVED`). This is the direct analog of M-05: just as an inactive KNOT could call `stake()` without a state guard, a creator can mutate terminal transactions without a status guard.

Additionally, `removeTransactionApprover` performs no authorization check at all — it accepts no `user` parameter — meaning any authenticated user can delete any approver from any transaction, bypassing the approval workflow entirely.

---

### Finding Description

**Vector 1 — Missing status check in `createTransactionApprovers`**

`ApproversService.createTransactionApprovers` delegates its only pre-condition check to `getCreatorsTransaction`, which verifies existence and creator ownership but never inspects `transaction.status`. [1](#0-0) 

`getCreatorsTransaction` itself: [2](#0-1) 

No status guard exists between the ownership check and the `INSERT` into `transaction_approver`.

**Vector 2 — Missing status check in `createTransactionObservers`**

`ObserversService.createTransactionObservers` checks existence and creator ownership, then immediately proceeds to insert observers with no status validation: [3](#0-2) 

**Vector 3 — Missing authorization AND status check in `removeTransactionApprover`**

`removeTransactionApprover` accepts only an approver `id` — no `user` parameter — and performs no ownership or status check before soft-deleting the entire approver subtree: [4](#0-3) 

Compare this to `updateTransactionApprover`, which correctly accepts a `user` parameter and calls `getCreatorsTransaction`: [5](#0-4) 

The contrast confirms the omission in `removeTransactionApprover` is unintentional.

**Contrast with functions that do check status**

`approveTransaction` and `importSignatures` both guard on status before proceeding: [6](#0-5) [7](#0-6) 

The terminal-status set used elsewhere in the codebase is well-defined: [8](#0-7)

### Impact Explanation

**Vector 1 & 2 (status bypass):**
- Approvers added to an `EXECUTED` or `CANCELED` transaction receive real approval-request notifications for a transaction that can never be acted upon, corrupting the audit trail and confusing users.
- `emitTransactionStatusUpdate` fires after the insert, propagating the corrupted state to all connected clients.
- `shouldApproveTransaction` does guard on terminal status and returns `false`, so the approval gate itself is not bypassed — but the data and notification integrity are broken.

**Vector 3 (authorization bypass):**
- Any authenticated user (not just the creator) can call the delete endpoint and remove all approvers from any in-flight transaction.
- Removing all approvers from a `WAITING_FOR_SIGNATURES` transaction eliminates the approval requirement, potentially allowing the transaction to advance to `WAITING_FOR_EXECUTION` without the intended governance controls.

---

### Likelihood Explanation

- Vectors 1 & 2 require the transaction creator to call the endpoint after the transaction reaches a terminal state. This can happen accidentally (race condition between execution and a UI action) or deliberately.
- Vector 3 requires only a valid JWT — any registered user. The approver `id` is an auto-incremented integer, making enumeration trivial.

---

### Recommendation

1. **Add a terminal-status guard** at the top of `createTransactionApprovers`, `createTransactionObservers`, `updateTransactionApprover`, and `removeTransactionApprover`:

```typescript
if (this.terminalStatuses.includes(transaction.status)) {
  throw new BadRequestException(ErrorCodes.OTIP);
}
```

2. **Add a `user` parameter and creator check** to `removeTransactionApprover`, mirroring the pattern already used in `updateTransactionApprover`:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  const rootNode = await this.getRootNodeFromNode(approver.id);
  await this.getCreatorsTransaction(rootNode.transactionId, user); // ownership + status check
  ...
}
```

### Proof of Concept

**Vector 1 — Add approver to executed transaction:**

```
POST /transactions          → creates transaction T (id=42)
[transaction T executes]    → status becomes EXECUTED
POST /transactions/42/approvers
  Body: { approversArray: [{ userId: 99 }] }
  Auth: creator's JWT

→ HTTP 201 Created
→ TransactionApprover row inserted for EXECUTED transaction
→ User 99 receives "approval required" notification for a closed transaction
```

**Vector 3 — Any user removes an approver:**

```
GET /transactions/42/approvers  → reveals approver id=7
DELETE /transactions/approvers/7
  Auth: ANY valid JWT (not the creator)

→ HTTP 200 OK
→ Approver subtree soft-deleted; approval requirement silently removed
→ Transaction may advance without required governance sign-off
```

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;

      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user', 'observers'],

    const observers: TransactionObserver[] = [];

      if (!transaction.observers.some(o => o.userId === userId)) {
        const observer = this.repo.create({ userId, transactionId, role: Role.FULL });
        observers.push(observer);
      }

      return [];
    }

      const result = await this.repo.save(observers);

      emitTransactionUpdate(this.notificationsPublisher, [{ entityId: transactionId }]);

      return result;
