### Title
Unpinned Dependency Versions and Missing Lock File Expose Build Pipeline to Supply Chain Attack

### Summary
The `automation/package.json` uses caret (`^`) version ranges for critical Hedera SDK dependencies, the root `package.json` uses a caret range for `prettier`, no `pnpm-lock.yaml` files are committed to the repository, and the back-end CI pipeline runs `pnpm install` without `--frozen-lockfile`. Together, these gaps allow silently updated or compromised dependency versions to enter the build and test pipeline without any detection mechanism.

### Finding Description

**1. Caret ranges in `automation/package.json`**

Three dependencies allow automatic minor/patch version resolution: [1](#0-0) 

- `"@hiero-ledger/proto": "^2.28.0"` — any `2.x.x >= 2.28.0` is accepted
- `"@hiero-ledger/sdk": "^2.82.0"` — any `2.x.x >= 2.82.0` is accepted
- `"jszip": "^3.10.1"` — any `3.x.x >= 3.10.1` is accepted

`@hiero-ledger/sdk` is the Hedera network SDK used to construct and submit transactions. A malicious minor release would be silently pulled in.

**2. Caret range in root `package.json`** [2](#0-1) 

`"prettier": "^3.8.3"` — the workspace root also accepts any `3.x.x >= 3.8.3`.

**3. No `pnpm-lock.yaml` committed to the repository**

A glob search across the entire repository for `**/pnpm-lock.yaml` returned zero results. The CI workflows reference these paths as cache keys: [3](#0-2) [4](#0-3) 

But since no lock files are committed, every `pnpm install` resolves the full transitive dependency graph fresh from the npm registry. Even exact-pinned direct dependencies (as used in `front-end/package.json` and `back-end/package.json`) do not protect against transitive dependency drift without a committed lock file.

**4. Back-end CI runs `pnpm install` without `--frozen-lockfile`** [5](#0-4) 

The back-end test workflow uses plain `pnpm install`, which will silently resolve and install whatever the current registry serves for transitive dependencies. By contrast, the front-end and automation workflows do use `--frozen-lockfile`: [6](#0-5) 

But `--frozen-lockfile` is only meaningful if a lock file is committed — without one, it either fails or generates a new one, providing no reproducibility guarantee.

### Impact Explanation

The back-end NestJS services (`api`, `chain`, `notifications`) handle transaction signing, multi-signature orchestration, and direct submission of transactions to the Hedera network. If a transitive dependency of any back-end package is compromised (e.g., a dependency of `@nestjs/core`, `typeorm`, or `@hiero-ledger/sdk`), the malicious code would be silently included in every CI build and Docker image without any integrity check. The `chain` service in particular submits signed transaction bytes directly to the Hedera network. [7](#0-6) 

### Likelihood Explanation

Supply chain attacks against npm packages are well-documented and increasing in frequency. The attack surface here is realistic:
- `@hiero-ledger/sdk` is a high-value target (Hedera network SDK).
- The automation package explicitly allows any `^2.82.0` minor release.
- The absence of committed lock files means every CI run is a fresh resolution — no human review is required for a transitive dependency to change.
- The back-end CI does not use `--frozen-lockfile`, removing the last line of defense.

### Recommendation

1. **Commit lock files**: Ensure `pnpm-lock.yaml` is committed for all three workspaces (`front-end/`, `back-end/`, `automation/`). Remove any `.gitignore` entries that exclude them.
2. **Remove caret/tilde ranges**: Change `automation/package.json` lines 45–46 and 57 to exact versions (`"@hiero-ledger/sdk": "2.82.0"`, etc.). Fix root `package.json` line 3 similarly.
3. **Add `--frozen-lockfile` to back-end CI**: Change `pnpm install` to `pnpm install --frozen-lockfile` in `.github/workflows/test-backend.yaml` line 62.
4. **Enable Dependabot**: Configure automated PR-based dependency update reviews rather than silent resolution.
5. **Add `pnpm audit` step** to all CI workflows to catch known CVEs on every build.

### Proof of Concept

```
# Attacker publishes @hiero-ledger/sdk@2.83.0 with malicious payload
# (or compromises any transitive dep of @nestjs/core, typeorm, etc.)

# Developer or CI runs:
cd automation && pnpm install
# pnpm resolves ^2.82.0 → picks up 2.83.0 automatically
# No lock file diff to review; no frozen-lockfile check fails

# Back-end CI:
cd back-end && pnpm install   # no --frozen-lockfile
# Transitive deps resolved fresh; malicious transitive package installed
# Docker image built and deployed with compromised dependency
# chain service submits transactions to Hedera network using compromised SDK
```

Concrete file references:
- `automation/package.json` lines 45–46, 57 — caret ranges on `@hiero-ledger/sdk`, `@hiero-ledger/proto`, `jszip`
- `package.json` line 3 — caret range on `prettier`
- `.github/workflows/test-backend.yaml` line 62 — `pnpm install` without `--frozen-lockfile`
- No `pnpm-lock.yaml` present anywhere in the repository
