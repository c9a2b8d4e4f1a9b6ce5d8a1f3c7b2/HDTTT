### Title
Any Authenticated User Can Enumerate All User Keys via Missing Access Control on `GET /user-keys`

### Summary
The `UserKeysAllController` exposes a `GET /user-keys` endpoint that retrieves all user keys in the system without restricting access to administrators. Any authenticated, verified user can call this endpoint and receive the full paginated list of every user's keys. A parallel endpoint (`GET /user/:userId/keys`) correctly uses `getUserKeysRestricted(user, userId)` to enforce per-user scoping, but the `/user-keys` route bypasses this entirely by calling `getUserKeys(paginationParams)` with no user filter and no `AdminGuard`.

### Finding Description

**Root cause — missing caller restriction:**

The `UserKeysAllController` is guarded only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard`. No `AdminGuard` is present. [1](#0-0) 

The service call passes no user identity:

```ts
return this.userKeysService.getUserKeys(paginationParams);
``` [2](#0-1) 

Compare this to the correctly scoped sibling endpoint in `UserKeysController`, which passes the authenticated user and calls the restricted variant:

```ts
return this.userKeysService.getUserKeysRestricted(user, userId);
``` [3](#0-2) 

The existence of a `getUserKeysRestricted` method confirms that `getUserKeys` is the unrestricted, all-users variant. The `UserKeysController` (route: `user/:userId?/keys`) correctly enforces ownership; the `UserKeysAllController` (route: `user-keys`) does not. [4](#0-3) 

The `AdminGuard` is available and used elsewhere in the codebase (e.g., `PATCH /users/:id`, `DELETE /users/:id`) for privileged operations: [5](#0-4) 

### Impact Explanation

Any authenticated, verified user — with no admin privilege — can issue `GET /user-keys` and receive a paginated dump of every user key registered in the organization. User keys include public keys and mnemonic hash metadata used for signing Hedera transactions. This enables:

- **Cross-user key enumeration**: An attacker learns which public keys belong to which accounts across the entire organization.
- **Targeted attack surface mapping**: Knowing all registered keys allows an attacker to craft targeted signature-import or transaction-signing attacks against specific key IDs.
- **Privacy violation**: Organizational key ownership is confidential operational data. [6](#0-5) 

### Likelihood Explanation

**High.** The attacker preconditions are minimal: possess any valid, non-blacklisted JWT for a verified user account. No admin role, no special privilege, no leaked credentials required. The endpoint is a standard REST `GET` with no additional friction. Any user who has completed registration can exploit this immediately. [7](#0-6) 

### Recommendation

Add `AdminGuard` to `UserKeysAllController` so that only administrators can retrieve the full, unfiltered key list:

```ts
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard, AdminGuard)
export class UserKeysAllController { ... }
```

Alternatively, if this endpoint is intended for all users but should be scoped to the requesting user, replace the service call with `getUserKeysRestricted(user, paginationParams)` and inject `@GetUser() user: User` — matching the pattern already used in `UserKeysController`. [8](#0-7) 

### Proof of Concept

1. Register or obtain credentials for any non-admin verified user account.
2. Authenticate: `POST /auth/login` → receive `accessToken`.
3. Issue: `GET /user-keys` with `Authorization: Bearer <accessToken>`.
4. Observe: the response contains a paginated list of **all** user keys in the organization, belonging to all users, not just the requesting user.
5. Contrast: `GET /user/999/keys` (where 999 is another user's ID) correctly returns only what `getUserKeysRestricted` permits for the requesting user. [6](#0-5) [3](#0-2)

@Injectable()
export class AdminGuard implements CanActivate {
  canActivate(context: ExecutionContext) {
    const { user } = context.switchToHttp().getRequest();
    return user && user.admin;
