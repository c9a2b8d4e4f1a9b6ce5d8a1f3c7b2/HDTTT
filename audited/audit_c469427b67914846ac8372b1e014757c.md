### Title
Transaction Creator Can Add Themselves as an Approver and Self-Approve Their Own Transaction

### Summary
In `ApproversService.createTransactionApprovers`, the only authorization check is that the caller is the transaction creator. There is no guard preventing the creator from supplying their own `userId` in the `approversArray`. Once added, the creator can call `approveTransaction` and satisfy the approval requirement unilaterally, bypassing the multi-party oversight that the approver workflow is designed to enforce.

### Finding Description

**Root cause — `createTransactionApprovers` has no self-referral guard:** [1](#0-0) 

The function opens by verifying the caller is the transaction creator: [2](#0-1) 

It then validates the supplied `userId` only to confirm the user *exists* in the database: [3](#0-2) 

There is no check of the form `if (dtoApprover.userId === user.id) throw ...`. The creator's own `id` passes every validation and is inserted as a valid approver record.

**Exploitation path — `approveTransaction` accepts the self-inserted record:** [4](#0-3) 

`approveTransaction` filters approver rows by `userId === user.id`. Because the creator inserted themselves, `userApprovers` is non-empty and the `UnauthorizedException` is never thrown. The creator then supplies a valid signature over the transaction body (which they already possess, having created the transaction) and the approval is recorded.

**API surface — reachable by any authenticated user:** [5](#0-4) 

Both endpoints are protected only by JWT authentication (`JwtAuthGuard`, `VerifiedUserGuard`). No admin role is required.

### Impact Explanation

The approver workflow exists to enforce independent oversight before a transaction is executed. A creator who self-approves satisfies the approval gate without any second party ever reviewing the transaction. In an organization context this completely nullifies the multi-party control model: a single user can create, self-approve, and drive a transaction to execution status (`WAITING_FOR_EXECUTION`) without any colleague's involvement. Depending on the Hedera transaction type (e.g., large HBAR transfers, account key updates, node admin operations), this can result in unauthorized asset movement or unauthorized network state changes.

### Likelihood Explanation

The attack requires only a valid JWT for any organization user who has created at least one transaction — no elevated privileges, no leaked secrets, no social engineering. The two API calls needed are documented in the project's own API guide: [6](#0-5) 

Any authenticated user who can create transactions can exploit this immediately.

### Recommendation

In `createTransactionApprovers`, before inserting an approver record, reject any `userId` that equals the calling user's id:

```typescript
if (typeof dtoApprover.userId === 'number' && dtoApprover.userId === user.id) {
  throw new Error('Transaction creator cannot add themselves as an approver');
}
```

Apply the same guard in `updateTransactionApprover` when `dto.userId` is being set, to prevent a creator from rotating an existing approver slot to themselves after the fact. [7](#0-6) 

### Proof of Concept

1. Authenticate as user A (the transaction creator). Obtain a JWT.
2. Create a transaction via `POST /transactions`. Note the returned `transactionId` and user A's `userId`.
3. Self-insert as approver:
   ```
   POST /transactions/<transactionId>/approvers
   Authorization: Bearer <tokenA>
   Content-Type: application/json

   { "approversArray": [{ "userId": <userA_id> }] }
   ```
   Expected: `201 Created` — no error, because `createTransactionApprovers` only checks the user exists, not that they differ from the creator.
4. Self-approve:
   ```
   POST /transactions/<transactionId>/approvers/approve
   Authorization: Bearer <tokenA>
   Content-Type: application/json

   { "userKeyId": <userA_keyId>, "signature": "<validSig>", "approved": true }
   ```
   Expected: `200 true` — `approveTransaction` finds user A in the approver list and records the approval.
5. Observe that the transaction advances through the approval gate with zero independent oversight.
