### Title
Missing Creator Authorization Check in `removeTransactionApprover` Allows Any Authenticated User to Remove Approvers from Any Transaction

### Summary
The `removeTransactionApprover` function in `approvers.service.ts` performs no authorization check to verify the caller is the transaction creator. Unlike `createTransactionApprovers` and `updateTransactionApprover`, which both enforce creator-only access via `getCreatorsTransaction`, the remove path accepts only an approver `id` with no `user` parameter and no ownership verification. Any authenticated user can silently dismantle the approval tree of any transaction by supplying a valid approver ID.

### Finding Description

**Vulnerability class:** Authorization — missing access control on a state-transition function (direct analog to the Aave report's permissioned/permissionless mismatch on a critical state-change path).

**Root cause — inconsistent authorization pattern across CRUD operations on approvers:**

`createTransactionApprovers` enforces creator-only access: [1](#0-0) 

`updateTransactionApprover` enforces creator-only access: [2](#0-1) 

`removeTransactionApprover` — **no `user` parameter, no creator check, no authorization at all:** [3](#0-2) 

The function signature is `removeTransactionApprover(id: number)`. Because the `user` is not even accepted as a parameter, no caller — including the controller — can inject an ownership check through this path. The service immediately fetches the approver and calls `removeNode` without any identity verification.

Compare the signatures side-by-side:
- `createTransactionApprovers(user: User, transactionId: number, dto: ...)` → calls `getCreatorsTransaction`
- `updateTransactionApprover(id, dto, transactionId, user: User)` → calls `getCreatorsTransaction`
- `removeTransactionApprover(id: number)` → **no user, no check**

The controller-level guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`) only confirm the caller is a valid, verified user — they do not enforce transaction ownership. [4](#0-3) 

**Exploit path:**
1. Attacker registers as a normal user and authenticates (JWT obtained).
2. Attacker learns a target transaction's approver IDs. These are sequential integers and can be brute-forced, or obtained legitimately if the attacker is an observer/signer on any transaction (the `getVerifiedApproversByTransactionId` endpoint returns IDs for accessible transactions).
3. Attacker calls `DELETE /transactions/{transactionId}/approvers/{approverId}` (or equivalent route) for each approver on a victim transaction.
4. All approvers are removed. The approval tree is destroyed. The transaction can now proceed to `WAITING_FOR_EXECUTION` without any approvals having been collected.

### Impact Explanation

Removing all approvers from a transaction eliminates the multi-party approval gate that is the primary governance control in Organization Mode. A transaction that required N-of-M approvals before execution can be stripped of all approvers by any authenticated user, allowing it to advance to execution with zero approvals. On the Hedera network this translates to unauthorized execution of arbitrary transaction types (account updates, token transfers, file operations, node operations) that the organization intended to gate behind human review.

### Likelihood Explanation

The attacker requires only a valid, verified account — the lowest possible privilege level in the system. Approver IDs are sequential integers, making enumeration trivial even without prior knowledge of specific transactions. The attack is silent (no notification is sent to the transaction creator on approver removal based on the code path), and it is non-reversible once the approver records are soft-deleted.

### Recommendation

Add a `user: User` parameter to `removeTransactionApprover` and call `getCreatorsTransaction` (or the equivalent `getCreatorsTransaction` helper already used by create and update) before executing the removal:

```typescript
async removeTransactionApprover(id: number, user: User): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Resolve root transaction and verify caller is the creator
  const root = await this.getRootNodeFromNode(approver.id);
  if (!root) throw new BadRequestException(ErrorCodes.RANF);
  await this.getCreatorsTransaction(root.transactionId, user);

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

The controller must also be updated to extract and forward the authenticated user to this service method, consistent with how `createTransactionApprovers` and `updateTransactionApprover` are already wired.

### Proof of Concept

**Preconditions:** Two accounts — `victim` (transaction creator) and `attacker` (any verified user).

1. `victim` creates a transaction and adds three approvers (IDs returned: 42, 43, 44).
2. `attacker` authenticates and obtains a JWT.
3. `attacker` sends:
   ```
   DELETE /transactions/approvers/42   (Authorization: Bearer <attacker_jwt>)
   DELETE /transactions/approvers/43
   DELETE /transactions/approvers/44
   ```
4. All three requests succeed with HTTP 200.
5. The transaction's approval tree is now empty. The chain service processes it as if approvals are satisfied, advancing it toward execution without any approver having signed.

**Expected (correct) behavior:** Each DELETE should return HTTP 401/403 because `attacker` is not the creator of the transaction owning approver IDs 42–44.
