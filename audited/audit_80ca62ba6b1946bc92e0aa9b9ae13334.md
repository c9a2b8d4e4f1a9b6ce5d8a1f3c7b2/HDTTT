### Title
Decrypted Private Key Persists in Renderer DOM Without Timeout and Is Accessible Without Re-Authentication During Cached Password Window

### Summary
Once a user authenticates with their password for any operation (e.g., signing a transaction), the password is cached in the Pinia store for a 10-minute sliding window. During this window, any click on the "show private key" eye icon in `Settings > Keys` (`KeysTab.vue`) decrypts and renders the raw private key in the DOM **without prompting for the password again**. The decrypted key is stored in a Vue reactive ref (`decryptedKeys`) with **no timeout** — it remains in the DOM until the user manually hides it. A low-privileged process running as the same OS user (e.g., malware installed via phishing) can exploit this via OS accessibility APIs or screenshot capture to steal the private key without physical access.

### Finding Description

**Root cause 1 — 10-minute sliding password cache:**

`storeUser.ts` initializes `passwordStoreDuration` to 10 minutes:

```
const passwordStoreDuration = ref<number>(10 * 60 * 1_000);
```

`getPassword()` returns the cached password and resets the expiry on every call:

```typescript
personal.value.passwordExpiresAt = new Date(Date.now() + passwordStoreDuration.value);
return personal.value.password;
```

`isLoggedInWithValidPassword()` in `userStoreHelpers.ts` only checks `passwordExpiresAt < now`:

```typescript
const isExpired = user.passwordExpiresAt && new Date(user.passwordExpiresAt) < new Date();
return !isExpired;
```

**Root cause 2 — `usePersonalPassword.getPassword()` bypasses the modal when password is cached:**

```typescript
const personalPassword = user.getPassword();
if (!personalPassword && !user.personal.useKeychain) {
  // opens modal
  return false;
}
return personalPassword;  // returned silently, no re-prompt
```

**Root cause 3 — `KeysTab.vue` `handleShowPrivateKey` → `decrypt()` uses cached password:**

```typescript
const handleShowPrivateKey = async (publicKey: string) => {
  publicKeysPrivateKeyToDecrypt.value = publicKey;
  await decrypt();  // no additional auth gate
};

const decrypt = async () => {
  const personalPassword = getPassword(decrypt, { ... });
  if (passwordModalOpened(personalPassword)) return;
  // if password is cached, proceeds directly:
  const decryptedKey = await decryptPrivateKey(user.personal.id, personalPassword, ...);
  decryptedKeys.value.push({ publicKey: ..., decrypted: decryptedKey });
  // stored indefinitely — no timeout
};
```

**Root cause 4 — Decrypted key rendered in DOM with no timeout:**

The template renders the raw private key string directly from `decryptedKeys` ref and keeps it there until the user manually clicks the hide icon. There is no automatic expiry or clearing mechanism.

**Exploit flow:**

1. User logs in and performs any password-requiring action (e.g., signs a transaction) → password cached for 10 minutes, sliding window.
2. Malware (running as the same OS user, no elevated privileges needed) uses OS accessibility APIs (Windows UI Automation / Linux AT-SPI) to enumerate the Electron window, locate the eye icon button (`data-testid="span-show-modal-{index}"`), and programmatically click it.
3. `handleShowPrivateKey` fires → `decrypt()` calls `getPassword()` → cached password returned → IPC call to `decryptPrivateKey` in main process → raw private key returned to renderer.
4. Malware reads the DOM text content of `span[data-testid="span-private-key-{index}"]` via accessibility tree, or captures a screenshot.
5. Raw private key is exfiltrated → complete Hedera account compromise.

### Impact Explanation
The attacker obtains the raw ED25519 or ECDSA private key controlling one or more Hedera accounts. With the private key, the attacker can sign arbitrary transactions (HBAR transfers, token operations, account updates) and drain all assets. Impact is **critical**: direct, irreversible theft of funds and full account takeover.

### Likelihood Explanation
Requires: (a) malware installed on the victim's machine (achievable remotely via phishing/drive-by, no physical access needed), and (b) the user to have authenticated within the past 10 minutes (normal usage pattern — any signing operation triggers this). On Windows and Linux, same-user processes can access the accessibility tree and capture screenshots without elevated privileges. The `data-testid` attributes on the eye icon and private key span make programmatic targeting trivial. Likelihood is **medium-high** for users who are actively using the application.

### Recommendation

1. **Require re-authentication before displaying the raw private key.** The `decrypt()` function in `KeysTab.vue` should always open the password modal regardless of cache state. Clear `user.personal.password` before calling `getPassword()` in this specific context, or add a dedicated `requireFreshPassword()` path that always prompts.

2. **Implement a display timeout for decrypted keys.** After decrypting and displaying a private key, start a short timer (e.g., 30–60 seconds) after which the entry is removed from `decryptedKeys` and the key is cleared from the DOM.

3. **Reduce the password cache window or scope it.** The 10-minute sliding window is broad. Consider scoping the cached password to signing operations only, not to key display. Alternatively, reduce the duration or require re-entry for sensitive display operations.

4. **Avoid storing the decrypted private key string in reactive state.** Instead of pushing the plaintext key into `decryptedKeys`, consider displaying it only transiently (e.g., in a modal that is destroyed on close) so it is not retained in the component's reactive graph.

### Proof of Concept

**Preconditions:** Hedera Transaction Tool is open; user has signed a transaction within the last 10 minutes (password cached); malware process runs as the same OS user.

**Steps:**

1. Malware enumerates the Electron renderer window via Windows UI Automation or Linux AT-SPI.
2. Malware locates the element with `data-testid="span-show-modal-0"` (the eye icon for the first key pair).
3. Malware programmatically fires a click event on that element.
4. `handleShowPrivateKey` executes → `decrypt()` → `getPassword()` returns cached password (no modal shown) → IPC `decryptPrivateKey` call → raw private key pushed to `decryptedKeys`.
5. Malware reads the text content of `span[data-testid="span-private-key-0"]` from the accessibility tree.
6. Raw private key (e.g., `302e020100300506032b657004220420...`) is exfiltrated.
7. Attacker uses the key to sign a `CryptoTransferTransaction` draining the account.
