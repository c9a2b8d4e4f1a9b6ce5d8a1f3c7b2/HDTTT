### Title
IDOR in `removeTransactionApprover` Allows Any Transaction Creator to Delete Approvers from Arbitrary Transactions

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint authorizes the request by verifying the caller is the creator of the transaction identified by `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver belongs to the authorized transaction. Any authenticated user who owns at least one transaction can delete approvers from any other transaction in the system, bypassing multi-signature approval requirements.

### Finding Description

**Vulnerability class:** Authorization bypass / IDOR (analogous to the external report's missing sender validation on the burn path).

The controller handler in `approvers.controller.ts` lines 102–113:

```typescript
@Delete('/:id')
async removeTransactionApprover(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Param('id', ParseIntPipe) id: number,
) {
  await this.approversService.getCreatorsTransaction(transactionId, user); // ← checks ownership of transactionId
  await this.approversService.removeTransactionApprover(id);               // ← deletes approver by id, no cross-check
  return true;
}
``` [1](#0-0) 

`getCreatorsTransaction` only verifies the caller is the creator of the transaction referenced by the URL's `:transactionId` parameter: [2](#0-1) 

`removeTransactionApprover` then fetches the approver by `:id` and deletes it with no check that it belongs to the authorized transaction: [3](#0-2) 

`removeNode` performs a recursive soft-delete purely by approver ID, with no transaction ownership guard: [4](#0-3) 

**The fix already exists in the sibling `updateTransactionApprover` function**, which correctly cross-checks the root node's `transactionId` against the URL parameter before proceeding: [5](#0-4) 

The delete path simply omits this check entirely.

### Impact Explanation

An attacker who is the creator of any transaction (a normal, unprivileged product flow) can:

1. Enumerate or guess approver IDs belonging to another user's transaction (IDs are sequential integers).
2. Issue `DELETE /transactions/<attacker_own_txId>/approvers/<victim_approver_id>`.
3. The authorization check passes (attacker owns `<attacker_own_txId>`), and the victim's approver is soft-deleted.

Concrete consequences:
- **Bypass multi-sig approval gates**: Removing all required approvers from a victim's transaction causes the approval threshold to be unmet, preventing execution or corrupting the approval state.
- **Sabotage organizational workflows**: In an organization context, an attacker can silently strip approvers from any pending transaction, causing it to stall or execute without the intended oversight.
- **Unauthorized state mutation**: The attacker modifies another user's transaction state without any authorization over that transaction.

### Likelihood Explanation

- **Attacker precondition**: Only requires a valid authenticated account and the ability to create one transaction (standard product flow, no privilege required).
- **Approver ID discovery**: IDs are auto-incremented integers; an attacker can enumerate them trivially or observe IDs from their own transactions to estimate the range.
- **No rate limiting or anomaly detection** is visible on this endpoint.
- The attack is silent — the victim receives a status-update notification but no indication that an unauthorized deletion occurred.

### Recommendation

In `removeTransactionApprover` (service), add the same cross-check already present in `updateTransactionApprover`: resolve the root node of the target approver and assert its `transactionId` matches the URL-supplied `transactionId` before deleting.

Concretely, mirror the pattern at lines 385–391 of `approvers.service.ts`:

```typescript
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Cross-check: ensure the approver belongs to the authorized transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(this.notificationsPublisher, [{ entityId: approver.transactionId }]);
  return result;
}
```

Pass `transactionId` from the controller (it is already available as a parameter) into the service method.

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (attacker-owned).
- User B creates Transaction 2 with an approver (approver ID = 99) required before execution.

**Attack:**
```
DELETE /transactions/1/approvers/99
Authorization: Bearer <User A's JWT>
```

**What happens:**
1. `getCreatorsTransaction(1, userA)` → passes (User A owns Transaction 1).
2. `removeTransactionApprover(99)` → fetches approver 99 (belongs to Transaction 2), calls `removeNode(99)`, soft-deletes it.
3. Transaction 2's required approver is gone. User B's multi-sig requirement is silently bypassed.

**Expected (correct) behavior:** The request should be rejected with an authorization error because approver 99 does not belong to Transaction 1. [1](#0-0) [3](#0-2)

  /* Soft deletes approvers' tree */
  async removeNode(listId: number): Promise<void> {
    if (!listId || typeof listId !== 'number') return null;

    await this.repo.query(
      `
      with recursive approversToDelete AS
        (
          select "id", "listId", "deletedAt"
          from transaction_approver
          where "id" = $1
  
            union all
              select transaction_approver."id", transaction_approver."listId", transaction_approver."deletedAt"
              from transaction_approver, approversToDelete     
              where approversToDelete."id" = transaction_approver."listId"
      
        )
      update transaction_approver
      set "deletedAt" = now()
      from approversToDelete
      where approversToDelete."id" = transaction_approver."listId" or transaction_approver."id" = $1;
    `,
      [listId],
    );

  }

  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
