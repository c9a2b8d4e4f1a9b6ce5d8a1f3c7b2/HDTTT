### Title
Stale Cached Account Key Mixed With Current Transaction Signatures Causes Incorrect Status Transitions and Permanent Transaction Failure

### Summary

`processTransactionStatus` and `getValidatedSDKTransaction` both call `computeSignatureKey`, which resolves required signing keys from a TTL-based cache (`AccountCacheService` / `NodeCacheService`, default TTL 10 seconds). The result is compared against `sdkTransaction._signerPublicKeys`, which reflects the actual signatures embedded in the current transaction bytes. When an account's key is rotated on-chain between the time a transaction is signed and the time the scheduler evaluates it, the two views diverge: the cache still holds the old key while the Hedera network has already accepted the new key. This mismatch causes the system to incorrectly promote a transaction to `WAITING_FOR_EXECUTION` and then submit it to Hedera with signatures that are no longer valid, resulting in a permanent `FAILED` status.

```typescript
const sdkTransaction = SDKTransaction.fromBytes(transaction.transactionBytes);
const signatureKey = await transactionSignatureService.computeSignatureKey(transaction);
const isAbleToSign = hasValidSignatureKey(
  [...sdkTransaction._signerPublicKeys],   // ← current: signatures already embedded
  signatureKey                              // ← potentially stale: from cache
);
```

`sdkTransaction._signerPublicKeys` is derived from the transaction bytes stored in the database — it reflects the keys that actually signed the transaction at signing time. `signatureKey` is built by `computeSignatureKey` by querying `AccountCacheService.getAccountInfoForTransaction`, which serves cached data up to `CACHE_STALE_THRESHOLD_MS` (default 10 000 ms) old before refreshing.
