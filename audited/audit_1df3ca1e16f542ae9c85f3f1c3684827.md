### Title
Unbounded Full-Table Scan Loop in `getTransactionsToSign` Causes Server-Side DoS

### Summary
`getTransactionsToSign` in `transactions.service.ts` fetches every active transaction from the database with no row limit, then iterates over the entire result set making one async database call per transaction before applying pagination. As the transaction table grows, any authenticated user can trigger this endpoint to exhaust server memory and CPU, causing denial of service.

### Finding Description
In `getTransactionsToSign`, the repository query at line 295 uses `this.repo.find()` with no `take` or `skip` constraint — it returns every transaction whose status is not in the terminal set. The result is then iterated in a `for` loop at line 301, calling the async `userKeysToSign` helper for each row. Pagination is applied only after the loop completes, via `result.slice(offset, offset + limit)` at line 313. [1](#0-0) 

The loop bound is the total count of non-terminal transactions in the database. There is no enforced maximum. Each iteration of the loop performs at least one additional async operation (`userKeysToSign`), so the work done per request scales linearly — and unboundedly — with the size of the transaction table.

The controller exposes this function at `GET /transactions/sign`, which is reachable by any authenticated, verified user: [2](#0-1) 

The `PaginationParams` decorator enforces `size ≤ 100` for the response slice, but this limit is never applied to the database query itself: [3](#0-2) 

A secondary instance exists in `importSignatures`. The controller accepts an unbounded `UploadSignatureMapDto[]` array with no array-length cap, and the service itself contains a developer comment acknowledging the missing limit: [4](#0-3) 

### Impact Explanation
An attacker (or a misconfigured client) repeatedly calling `GET /transactions/sign` against a deployment with a large transaction history will force the server to load the entire active transaction table into Node.js heap memory and execute N sequential async calls. This causes:
- Heap exhaustion / OOM crash of the API process.
- CPU saturation from deserializing and processing thousands of ORM objects.
- Cascading latency for all other API consumers sharing the same process.

Because the endpoint is paginated at the HTTP level, users and operators have no visible signal that the underlying query is unbounded.

### Likelihood Explanation
The endpoint is reachable by any authenticated organization member — no admin role is required. In a production deployment with months of transaction history, the active-transaction count grows continuously. The vulnerability requires no special knowledge beyond a valid JWT and a single HTTP GET request. The developer comment in `importSignatures` ("probably should limit this on the api side of things") confirms awareness that array-size limits are missing, suggesting the pattern is systemic.

### Recommendation
1. **Apply `take`/`skip` at the database layer** in `getTransactionsToSign`. Push the key-eligibility check into a SQL subquery (as already done in `getTransactionsToApprove` with its recursive CTE) so the database returns only the page of rows the user actually needs.
2. **Enforce an array-length cap** on the `UploadSignatureMapDto[]` body in `importSignatures` using a class-validator `@ArrayMaxSize` decorator before the service is invoked.
3. **Add an integration test** that asserts response time stays bounded as the transaction count grows.

### Proof of Concept

```
# 1. Authenticate and obtain a JWT
POST /auth/login  →  { "accessToken": "<JWT>" }

# 2. Create thousands of transactions (or wait for organic growth)
# 3. Trigger the unbounded scan
GET /transactions/sign?page=1&size=1
Authorization: Bearer <JWT>
```

The server will load every non-terminal transaction row into memory and call `userKeysToSign` for each before returning a single-item page. With 10 000 active transactions the Node.js process heap will spike proportionally; with 100 000+ rows the process will OOM or the request will time out, blocking the event loop for all concurrent users.
