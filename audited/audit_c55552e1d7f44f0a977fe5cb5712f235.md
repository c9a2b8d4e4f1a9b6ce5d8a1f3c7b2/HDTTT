### Title
Missing Transaction Access Check on `createComment` Allows Any Authenticated User to Post Comments on Arbitrary Transactions

### Summary
The `POST /transactions/:transactionId/comments` endpoint in `CommentsController` has no authorization check verifying that the requesting user has any relationship (creator, observer, signer, or approver) to the target transaction before creating a comment. Any authenticated, verified user can post comments on any transaction in the organization, including transactions they have no legitimate access to. The developers themselves flagged this gap with a `TODO` comment in the code.

### Finding Description
The `createComment` handler in `CommentsController` is protected only by controller-level authentication guards (`JwtBlackListAuthGuard`, `JwtAuthGuard`, `VerifiedUserGuard`). These guards confirm the caller is a valid, non-blacklisted, verified user — but they perform no check that the user is a participant in the target transaction. [1](#0-0) 

The developer explicitly acknowledged the missing check with a `TODO` comment on line 22. The underlying service method `CommentsService.createComment` performs no access verification either — it directly creates and saves the comment entity without querying whether the user is the creator, observer, signer, or approver of the transaction. [2](#0-1) 

By contrast, every other sensitive transaction sub-resource enforces access. For example, `ApproversService.getVerifiedApproversByTransactionId` throws `"You don't have permission to view this transaction"` for unauthorized users, and `TransactionsService.getTransactionWithVerifiedAccess` enforces the same pattern. The comments subsystem has no equivalent guard. [3](#0-2) 

A second, related missing-check exists on `GET /transaction-groups`, which is explicitly marked `/* TESTING ONLY */` in the controller but is deployed and accessible to any authenticated user. The service method `getTransactionGroups()` returns every `TransactionGroup` record in the database with no user-scoping whatsoever. [4](#0-3) [5](#0-4) 

### Impact Explanation
**`createComment`**: An authenticated user who is not a creator, observer, signer, or approver of a transaction can post arbitrary comment messages on that transaction. This constitutes unauthorized state mutation — comments are persisted to the database and associated with the transaction. In an organization workflow, this pollutes the audit trail, can be used to inject misleading information into the transaction record, and allows an attacker to probe the existence of arbitrary transaction IDs (a successful `201` response confirms the transaction exists).

**`getTransactionGroups`**: Any authenticated user can enumerate all transaction groups in the organization — their IDs, descriptions, creation timestamps, and `atomic`/`sequential` flags — regardless of whether they are a participant in any of those groups. This leaks organizational transaction metadata to unauthorized users.

### Likelihood Explanation
Both attack paths require only a valid, non-blacklisted, verified user account — the lowest privilege level in the system. No admin access, no leaked credentials, and no special knowledge beyond a valid JWT are required. The attacker simply sends a `POST /transactions/{id}/comments` with any `message` body, or a `GET /transaction-groups`. Both are trivially reachable from the public API.

### Recommendation
**For `createComment`**: Before saving the comment, verify the requesting user has access to the transaction using the existing `TransactionsService.getTransactionWithVerifiedAccess` (or `verifyAccess`) method. Throw `UnauthorizedException` if the user is not a creator, observer, signer, or approver of the target transaction.

**For `getTransactionGroups`**: Either remove the endpoint entirely (it is marked "TESTING ONLY") or add a user-scoped filter so it only returns groups containing transactions the requesting user is a participant in, consistent with how `getTransactionGroup` (singular) works via `getTransactionGroupItemsQuery`.

### Proof of Concept
**`createComment`**:
1. Register or log in as User A (no relationship to any transaction).
2. Obtain a valid JWT for User A.
3. Send:
   ```
   POST /transactions/1/comments
   Authorization: Bearer <User_A_JWT>
   Content-Type: application/json

   { "message": "Unauthorized comment on transaction 1" }
   ```
4. Observe `201 Created` — the comment is persisted to the database linked to transaction 1, even though User A is not a creator, observer, signer, or approver of that transaction.

**`getTransactionGroups`**:
1. Log in as any verified user.
2. Send:
   ```
   GET /transaction-groups
   Authorization: Bearer <any_valid_JWT>
   ```
3. Observe a `200 OK` response containing all `TransactionGroup` records in the organization, regardless of the requesting user's participation in any of them.

  /* Get the transaction by id and verifies that the user is the creator */
  async getCreatorsTransaction(
    transactionId: number,
    user: User,
    entityManager?: EntityManager,
  ): Promise<Transaction> {
    const find: FindOneOptions<Transaction> = {
      where: { id: transactionId },
      relations: ['creatorKey', 'creatorKey.user'],
    };

    const transaction = await (entityManager
      ? entityManager.findOne(Transaction, find)
      : this.dataSource.manager.findOne(Transaction, find));

    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (transaction.creatorKey?.userId !== user.id)
      throw new UnauthorizedException('Only the creator of the transaction is able to modify it');

    return transaction;
  }
