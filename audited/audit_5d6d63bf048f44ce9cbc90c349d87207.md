### Title
Unauthorized Cross-Transaction Approver Deletion via Missing Ownership Validation in `removeTransactionApprover`

### Summary
The `DELETE /transactions/:transactionId/approvers/:id` endpoint verifies the caller is the creator of `:transactionId`, but then deletes the approver identified by `:id` without verifying that approver actually belongs to `:transactionId`. Any authenticated user who is the creator of *any* transaction can delete approvers from *any other* transaction by supplying their own `transactionId` for the ownership check and a foreign approver `id` for the deletion target. The `updateTransactionApprover` path performs this cross-transaction check correctly, making this a direct analog to the external report's asymmetric validation pattern.

### Finding Description

**Root cause — controller decouples ownership check from deletion target:**

In `approvers.controller.ts` lines 102–113, the `removeTransactionApprover` handler:
1. Calls `getCreatorsTransaction(transactionId, user)` — verifies the caller owns `:transactionId`.
2. Calls `removeTransactionApprover(id)` — deletes the approver node identified by `:id`. [1](#0-0) 

There is no check that the approver with `:id` belongs to `:transactionId`. The service function `removeTransactionApprover` simply fetches the approver by its own primary key and deletes it: [2](#0-1) 

**Contrast with `updateTransactionApprover` — which does the check correctly:**

The update path explicitly validates that the approver's root transaction matches the supplied `transactionId` before proceeding: [3](#0-2) 

This is the exact asymmetry described in the external report: one mutating operation (update) enforces the ownership invariant; the other (delete) does not.

**Exploit path:**
1. Attacker (User A) is the creator of Transaction 1.
2. Transaction 2 belongs to User B and has an approver tree with approver record `id = 42`.
3. Attacker discovers approver ID 42 (sequential integers; or by being an approver/observer of Transaction 2 and calling `GET /transactions/2/approvers`).
4. Attacker sends: `DELETE /transactions/1/approvers/42`.
5. `getCreatorsTransaction(1, userA)` passes — User A owns Transaction 1.
6. `removeTransactionApprover(42)` executes — approver 42 (belonging to Transaction 2) is soft-deleted with no further check.

### Impact Explanation
An attacker can silently remove approval requirements from any transaction in the system. Removing approvers from a transaction that requires a threshold of approvals before execution can allow the transaction to advance to `WAITING_FOR_EXECUTION` or be executed without the intended governance controls. This constitutes unauthorized state mutation and a bypass of the multi-signature approval workflow, which is a core security invariant of the system.

### Likelihood Explanation
- Precondition: attacker must be an authenticated, verified user who has created at least one transaction (low barrier).
- Approver IDs are sequential database integers and can be enumerated, or obtained legitimately if the attacker is also an approver/observer on the target transaction (the `GET` endpoint returns them).
- No privileged role is required. The attack is fully reachable by a normal user.

### Recommendation
Inside `removeTransactionApprover` (or in the controller before calling it), verify that the approver's root transaction matches the supplied `transactionId`, mirroring the check already present in `updateTransactionApprover`:

```typescript
// approvers.service.ts — removeTransactionApprover
async removeTransactionApprover(id: number, transactionId: number): Promise<void> {
  const approver = await this.getTransactionApproverById(id);
  if (!approver) throw new BadRequestException(ErrorCodes.ANF);

  // Add: verify the approver belongs to the expected transaction
  const rootNode = await this.getRootNodeFromNode(approver.id);
  if (!rootNode || rootNode.transactionId !== transactionId)
    throw new UnauthorizedException('Approver does not belong to this transaction');

  const result = await this.removeNode(approver.id);
  emitTransactionStatusUpdate(...);
  return result;
}
```

Pass `transactionId` from the controller call site:
```typescript
// approvers.controller.ts
await this.approversService.removeTransactionApprover(id, transactionId);
```

### Proof of Concept

**Setup:**
- User A creates Transaction 1 (id=1).
- User B creates Transaction 2 (id=2) with an approver record (id=42, userId=UserC).

**Attack:**
```http
DELETE /transactions/1/approvers/42
Authorization: Bearer <User A JWT>
```

**Expected (correct) behavior:** 403 Unauthorized — approver 42 does not belong to transaction 1.

**Actual behavior:** 200 OK — approver 42 is soft-deleted from Transaction 2's approval tree. Transaction 2's approval threshold may now be satisfiable without User C's approval, or the approval tree may be corrupted, allowing Transaction 2 to proceed without the intended governance check.
