### Title
Undocumented Override Intent in `TransactionBaseModel` Hierarchy Risks Silent Signature Omission

### Summary
`TransactionBaseModel` defines and implements concrete (non-abstract) methods — `getSigningAccounts()`, `getReceiverAccounts()`, `getNewKeys()`, and `getNodeId()` — that return empty/null defaults and are silently intended to be overridden by subclasses. No documentation communicates this intent, and subclasses override these methods without the TypeScript `override` keyword. A developer adding a new transaction type who fails to override `getSigningAccounts()` will silently produce an empty signer set, causing the signature-collection pipeline to skip required signers for that transaction type.

### Finding Description
`TransactionBaseModel` is the root of the transaction model hierarchy used by the back-end to compute signing requirements:

```typescript
// back-end/libs/common/src/transaction-signature/model/transaction-base.model.ts
export abstract class TransactionBaseModel<T extends SDKTransaction> {
  getSigningAccounts(): Set<string> { return new Set<string>(); }
  getReceiverAccounts(): Set<string> { return new Set<string>(); }
  getNewKeys(): Key[] { return []; }
  getNodeId(): number | null { return null; }
}
``` [1](#0-0) 

None of these methods are marked `abstract`, none carry a doc-comment explaining they must be overridden, and none of the subclasses use the TypeScript `override` keyword:

```typescript
// AccountUpdateTransactionModel — no `override` keyword
getNewKeys(): Key[] { ... }
getSigningAccounts(): Set<string> { ... }
``` [2](#0-1) 

```typescript
// AccountDeleteTransactionModel — no `override` keyword
getSigningAccounts(): Set<string> { ... }
getReceiverAccounts(): Set<string> { ... }
``` [3](#0-2) 

```typescript
// NodeCreateTransactionModel — no `override` keyword
getNewKeys() { ... }
``` [4](#0-3) 

Contrast this with `EntityCache`, where the same pattern is handled more defensively: the base `load()` method throws an explicit error and subclasses use `protected override async load(...)`: [5](#0-4) [6](#0-5) 

`TransactionBaseModel` provides no such safety net — the base methods return empty collections silently.

### Impact Explanation
`getSigningAccounts()` drives the back-end's determination of which accounts must provide signatures before a transaction is eligible for execution. An empty return from the base class means the system records zero required signers for that transaction type. A transaction that should require one or more account signatures would be treated as fully signed immediately, bypassing the multi-signature approval workflow. [7](#0-6) 

### Likelihood Explanation
The Hedera network regularly introduces new transaction types (e.g., `NodeCreateTransaction`, `NodeUpdateTransaction`, `NodeDeleteTransaction` were recently added). Each addition requires a new model subclass. Because the base methods are undocumented and lack `abstract` enforcement, a developer can write a syntactically valid, test-passing subclass that simply omits the override — and the system will silently treat that transaction type as requiring no signers.

### Recommendation
1. Mark `getSigningAccounts()`, `getReceiverAccounts()`, `getNewKeys()`, and `getNodeId()` as `abstract` in `TransactionBaseModel`, removing the default implementations. This forces every concrete subclass to explicitly provide an implementation.
2. If a default empty return is genuinely correct for some subclasses, keep the concrete implementation but add a JSDoc comment on each method stating: *"Override in subclasses to specify accounts/keys required for this transaction type. The base implementation returns an empty collection."*
3. Enable `"noImplicitOverride": true` in `tsconfig.json` so TypeScript enforces the `override` keyword on all subclass method overrides, making intentional overrides explicit and catching accidental shadowing at compile time.
4. Add a unit test asserting that every registered transaction model type returns a non-empty result from `getSigningAccounts()` where applicable, to catch omissions at CI time.

### Proof of Concept
A developer adds support for a hypothetical `FreezeTransaction` model:

```typescript
export class FreezeTransactionModel extends TransactionBaseModel<FreezeTransaction> {
  static readonly TRANSACTION_TYPE = 'FreezeTransaction';
  // Developer forgets to override getSigningAccounts()
}
```

The `transaction-signature.service` calls `model.getSigningAccounts()`, receives `new Set()`, records zero required signers, and marks the transaction as fully signed. The transaction proceeds to execution without collecting any account signatures — silently, with no error or warning anywhere in the pipeline. [8](#0-7)
