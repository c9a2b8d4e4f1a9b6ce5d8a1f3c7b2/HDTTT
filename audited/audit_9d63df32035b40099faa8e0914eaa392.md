### Title
Unvalidated `mirrorNetwork` Field Enables SSRF and False Transaction Execution State

### Summary
The `mirrorNetwork` field in `CreateTransactionDto` accepts any arbitrary string with no allowlist enforcement. The backend uses this attacker-controlled value to open gRPC connections to external hosts (SSRF) and the chain service later re-uses the stored value to actually submit the transaction, meaning a transaction can be "executed" against an attacker-controlled node rather than the real Hedera network, corrupting the transaction's status record.

### Finding Description

**Root cause — no allowlist on `mirrorNetwork`:**

`CreateTransactionDto` decorates `mirrorNetwork` only with `@IsNotEmpty()` and `@IsString()`. There is no `@IsIn([...])`, `@IsEnum(...)`, or any other constraint that restricts the value to the known networks (`mainnet`, `testnet`, `previewnet`, `local-node`). [1](#0-0) 

**Step 1 — backend opens a gRPC connection to the attacker-supplied URL:**

`createTransactions` passes `dtos[0].mirrorNetwork` directly to `getClientFromNetwork`. [2](#0-1) 

Inside `getClientFromNetwork`, any value that does not match the four known constants falls through to the `default` branch, which constructs a live gRPC client pointed at the attacker-controlled host and immediately executes an `AddressBookQuery` against it. [3](#0-2) 

The attacker's server controls the address book that is returned, so `getNodeAccountIdsFromClientNetwork` will return attacker-chosen node IDs, and the subsequent `isTransactionValidForNodes` check will pass for any node IDs the attacker embeds in the fake address book. [4](#0-3) 

**Step 2 — the malicious `mirrorNetwork` is persisted:**

After validation, `mirrorNetwork: dto.mirrorNetwork` is stored verbatim in the `Transaction` row.
