### Title
`addClaim` Unconditionally Creates Duplicate Records When Claim Already Exists

### Summary
`addClaim` in `front-end/src/main/services/localUser/claim.ts` checks whether a claim already exists but then calls `prisma.claim.create` unconditionally regardless of the result. The comment says "claim will be overwritten," but no overwrite occurs ŌĆö a duplicate record is inserted instead. Because the `Claim` model in `schema.prisma` has no `@@unique([user_id, claim_key])` constraint, SQLite accepts the duplicate silently. All subsequent reads return the original (stale) value, making the intended overwrite permanently ineffective.

### Finding Description

**Root cause ŌĆö separated check and mutation:**

In `front-end/src/main/services/localUser/claim.ts` lines 18ŌĆō33:

```typescript
const alreadyAddedCount = await prisma.claim.count({
  where: { user_id: userId, claim_key: claimKey },
});

if (alreadyAddedCount > 0) logger.info('Claim already exists, claim will be overwritten');

return await prisma.claim.create({   // ŌåÉ always executed, even when count > 0
  data: { user_id: userId, claim_key: claimKey, claim_value: claimValue },
});
``` [1](#0-0) 

The `if` block only logs; it does not branch away from the `create` call. The `create` is outside and after the `if`, so it always runs ŌĆö identical to the FTN-3 pattern where the mutation was outside the conditional block.

**No unique constraint in schema:**

The `Claim` model has no `@@unique([user_id, claim_key])` directive:

```prisma
model Claim {
  id          String   @id @default(uuid())
  user_id     String
  claim_key   String
  claim_value String
  ...
}
``` [2](#0-1) 

SQLite therefore accepts the duplicate row without error, making the bug silent.

**Stale value is always returned:**

`getStoredClaim` and `getUseKeychainClaim` both take the first element of the result array:

```typescript
const [claim] = await get({ where });
return claim?.claim_value;
``` [3](#0-2) 

```typescript
if (flags.length === 0) throw new Error('Keychain mode not initialized');
return flags[0].claim_value === 'true';
``` [4](#0-3) 

Without an `ORDER BY`, SQLite returns rows in insertion order, so the original (first-inserted) value is always returned. The "overwrite" never takes effect.

**Reachable call sites:**

`addClaim` is called directly from `dataMigration.ts` for security-critical keys:

```typescript
await safeAwait(addClaim(userId, DEFAULT_MAX_TRANSACTION_FEE_CLAIM_KEY, ...));
await safeAwait(addClaim(userId, SELECTED_NETWORK, defaultNetwork));
await safeAwait(addClaim(userId, UPDATE_LOCATION, updatesLocation));
``` [5](#0-4) 

`addClaim` is also exposed as an IPC channel (`claim:add`) callable from the renderer process:

```typescript
createIPCChannel('claim', [renameFunc(addClaim, 'add'), ...]);
``` [6](#0-5) 

### Impact Explanation

When `addClaim` is called for an already-existing key (e.g., during a second migration run, or any renderer call to `claim:add`):

1. A duplicate row is silently inserted into the `Claim` table.
2. All subsequent reads return the **original** stale value ŌĆö the intended new value is permanently shadowed.
3. For the `USE_KEYCHAIN` claim, this means the encryption mode (system keychain vs. password-based) cannot be changed once set. If the user switches modes, the app continues using the old mode, potentially decrypting credentials with the wrong method and causing authentication failures or data inaccessibility.
4. For `SELECTED_NETWORK`, the active Hedera network cannot be updated via `addClaim`, causing transactions to be submitted to the wrong network.
5. Duplicate rows accumulate indefinitely, growing the local SQLite database.

### Likelihood Explanation

The bug is triggered by any second call to `addClaim` with the same `(userId, claimKey)` pair. `dataMigration.ts` calls `addClaim` for three keys every time migration runs. If migration is triggered more than once for the same user (e.g., re-import, re-login, or app reinstall without DB wipe), all three claims are duplicated. The IPC channel `claim:add` is also directly callable from the renderer, making it reachable through normal application flows. No privileged access is required.

### Recommendation

Move the `create` call inside an `else` branch, and perform an `update` when the claim already exists ŌĆö matching the intent expressed in the log message:

```typescript
if (alreadyAddedCount > 0) {
  return await prisma.claim.update({
    where: { /* find by userId + claimKey */ },
    data: { claim_value: claimValue },
  });
} else {
  return await prisma.claim.create({
    data: { user_id: userId, claim_key: claimKey, claim_value: claimValue },
  });
}
```

Additionally, add a `@@unique([user_id, claim_key])` constraint to the `Claim` model in `schema.prisma` to enforce the invariant at the database level and prevent silent duplicates from any future code path.

### Proof of Concept

1. App starts fresh; `addClaim(userId, 'USE_KEYCHAIN', 'true')` is called ŌĆö one row inserted, `claim_value = 'true'`.
2. User triggers migration again (or renderer calls `claim:add` with `USE_KEYCHAIN = 'false'`).
3. `addClaim` counts 1 existing row, logs "claim will be overwritten," then calls `prisma.claim.create` ŌĆö a **second** row is inserted with `claim_value = 'false'`.
4. `getUseKeychainClaim()` calls `getClaims` ŌåÆ returns both rows ŌåÆ takes `flags[0]` ŌåÆ returns `'true'` (the original).
5. The app continues using keychain mode even though the user set it to `'false'`. The intended state change has no effect. [7](#0-6) [2](#0-1)
