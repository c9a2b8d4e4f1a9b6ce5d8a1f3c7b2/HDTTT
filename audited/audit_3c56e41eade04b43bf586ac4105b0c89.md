### Title
`approveTransaction()` Uses Stale In-Memory State to Decide Status Notification, Preventing Transaction Status Transition After Approval

### Summary
In `approvers.service.ts`, the `approveTransaction()` function fetches approver records into memory, writes the approval to the database, then evaluates a stale in-memory snapshot to decide whether to emit a `emitTransactionStatusUpdate` or `emitTransactionUpdate` event. Because the in-memory snapshot always reflects the pre-update state (at least one approver without a signature, enforced by the guard at line 563), the condition `userApprovers.every(a => a.approved)` is always `false` when `dto.approved = true`. As a result, `emitTransactionStatusUpdate` is never emitted for approvals, and the transaction status is never transitioned to `WAITING_FOR_EXECUTION` through the approval path, leaving approved transactions permanently stuck.

### Finding Description

In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, the `approveTransaction()` function:

**Step 1 — Snapshot fetch (stale after DB write):** [1](#0-0) 

`userApprovers` is an in-memory snapshot of approver rows at the time of the query.

**Step 2 — Guard that guarantees at least one approver is unsigned:** [2](#0-1) 

If all approvers already had a signature, the function throws. Therefore, when execution reaches the DB write, at least one `userApprover` in the snapshot has `approved = undefined/null`.

**Step 3 — DB write updates the database but NOT the in-memory snapshot:** [3](#0-2) 

After this write, the database rows have `approved = dto.approved`, but the in-memory `userApprovers` array still holds the old values.

**Step 4 — Stale snapshot used to decide notification type:** [4](#0-3) 

When `dto.approved = true`:
- `!dto.approved` → `false`
- `userApprovers.every(a => a.approved)` → evaluated on stale data where at least one entry has `approved = undefined/null` → always `false`
- Result: `emitTransactionUpdate` is always dispatched instead of `emitTransactionStatusUpdate`

`emitTransactionStatusUpdate` is the event that triggers `processTransactionStatusUpdateNotifications` in the notification service, which in turn calls `processTransactionStatus` to evaluate and persist the new `WAITING_FOR_EXECUTION` status: [5](#0-4) 

Because this event is never emitted on the approval path, the transaction status is never advanced to `WAITING_FOR_EXECUTION` after all approvers have approved.

### Impact Explanation

Transactions that require approver sign-off are permanently stuck in `WAITING_FOR_SIGNATURES` even after all required approvers have approved. The chain service's scheduler only executes transactions in `WAITING_FOR_EXECUTION` status: [6](#0-5) 

This constitutes an unrecoverable lock of transaction state — approved transactions can never be automatically executed. Any organization relying on the approval workflow for multi-sig transaction execution is fully blocked.

**Impact: High** — complete denial of the approval-gated execution flow.

### Likelihood Explanation

This is triggered on every approval (`dto.approved = true`) submitted through the normal API flow. No special attacker preconditions are required — any legitimate approver exercising their normal role triggers the bug. The condition is deterministic and reproducible.

**Likelihood: High** — occurs on every approval action.

### Recommendation

After the database write, update the in-memory `userApprovers` to reflect the new state before evaluating the notification condition, or re-query the database. For example:

```typescript
// After the DB update, reflect the new approved state in memory
userApprovers.forEach(a => { a.approved = dto.approved; a.signature = dto.signature; });

if (!dto.approved || userApprovers.every(a => a.approved)) {
  emitTransactionStatusUpdate(this.notificationsPublisher, notificationEvent);
} else {
  emitTransactionUpdate(this.notificationsPublisher, notificationEvent);
}
```

Alternatively, re-fetch the approver rows from the database after the write to get the authoritative post-update state before deciding which event to emit.

### Proof of Concept

1. Create a transaction requiring one approver (User A).
2. User A calls `POST /transactions/:id/approvers/approve` with `{ approved: true, userKeyId: ..., signature: ... }`.
3. The DB write at lines 599–610 sets `approved = true` for User A's approver row.
4. At line 614, `userApprovers.every(a => a.approved)` evaluates the stale snapshot where User A's entry still has `approved = undefined` → condition is `false`.
5. `emitTransactionUpdate` is dispatched; `emitTransactionStatusUpdate` is never called.
6. The notification service never invokes `processTransactionStatus` for this transaction.
7. The transaction remains in `WAITING_FOR_SIGNATURES` indefinitely and is never scheduled for execution by the chain service.
