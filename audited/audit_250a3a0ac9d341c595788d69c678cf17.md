### Title
Missing Duplicate Check for Threshold Approver Nodes in `createTransactionApprovers` Allows Approval Tree Corruption

### Summary
The `isNode()` function in `approvers.service.ts` is intended to prevent duplicate approver nodes from being inserted, but it unconditionally returns `false` for threshold/tree nodes due to a logic flaw. A transaction creator (a normal authenticated user) can call the `POST /transactions/:id/approvers` endpoint multiple times with the same threshold tree payload, inserting duplicate threshold nodes and corrupting the approval tree structure for all assigned approvers.

### Finding Description
In `back-end/apps/api/src/transactions/approvers/approvers.service.ts`, `createTransactionApprovers` calls `isNode()` at line 250 to guard against duplicate insertions:

```typescript
if (await this.isNode(dtoApprover, transactionId, transactionalEntityManager))
  throw new Error(this.APPROVER_ALREADY_EXISTS);
``` [1](#0-0) 

The `isNode()` function (lines 647–665) performs a database count of matching records, but its return statement is:

```typescript
return count > 0 && typeof approver.userId === 'number';
``` [2](#0-1) 

For threshold/tree nodes, `approver.userId` is `undefined` or `null`, so `typeof approver.userId === 'number'` is always `false`. This means `isNode()` returns `false` for every threshold node regardless of whether a matching record already exists in the database. The duplicate guard is completely bypassed for all non-leaf approver nodes.

The `transaction_approver` table has no unique constraint on `(transactionId, threshold, listId)` — only a primary key — so the database will not reject the duplicate insert either. [3](#0-2) 

A transaction creator can POST to `/transactions/:id/approvers` repeatedly with the same threshold tree payload. Each call passes the `isNode()` check and inserts a new set of duplicate threshold nodes. [4](#0-3) 

### Impact Explanation
Duplicate root threshold nodes are returned by the recursive `getApproversByTransactionId` query and assembled into the approval tree by `getTreeStructure`. The approval workflow now contains multiple identical root trees. Approvers assigned to the transaction may be unable to satisfy the inflated/duplicated approval requirements, permanently stalling the transaction in `WAITING_FOR_SIGNATURES` status. This constitutes a corruption of the transaction's approval state that affects all participants (approvers, observers) who are not the creator. [5](#0-4) 

### Likelihood Explanation
Any authenticated user who creates a transaction is the transaction creator and can call the approvers endpoint. No elevated privileges are required. The attacker only needs to send the same valid `POST /transactions/:id/approvers` payload twice. This is a trivially reachable code path with no rate limiting or idempotency protection observed. [6](#0-5) 

### Recommendation
Remove the `&& typeof approver.userId === 'number'` condition from the return statement of `isNode()`. The duplicate check should apply to all node types, not just user-leaf nodes:

```typescript
// Before (buggy):
return count > 0 && typeof approver.userId === 'number';

// After (fixed):
return count > 0;
``` [7](#0-6) 

### Proof of Concept
1. Authenticate as a normal user and create a transaction (obtain `transactionId`).
2. POST to `/transactions/{transactionId}/approvers`:
   ```json
   { "approversArray": [{ "threshold": 1, "approvers": [{ "userId": 1 }] }] }
   ```
3. POST the identical payload a second time to the same endpoint.
4. Observe: no `"Approver already exists"` error is returned; HTTP 201 is returned both times.
5. Query the `transaction_approver` table: two root threshold nodes (both with `transactionId` set and `listId` null, `threshold = 1`) now exist for the same transaction, along with two duplicate child user-leaf nodes for `userId = 1`.
6. The approval tree is now corrupted — the transaction requires satisfying two independent threshold trees, which may behave unexpectedly in the approval status evaluation logic. [8](#0-7)
