### Title
`getTransactionsToSign` Fetches All Non-Terminal Transactions Without a DB-Level Limit, Enabling Unbounded Resource Exhaustion

### Summary

`TransactionsService.getTransactionsToSign` loads every non-terminal transaction in the database into memory and then issues one async `userKeysToSign()` DB call per row before applying pagination. Because there is no per-user transaction creation limit and no `take` clause on the bulk fetch, any authenticated user can inflate the transaction table and make a single API call that forces the server to execute an unbounded number of sequential database queries, exhausting memory and connection-pool resources for all users.

### Finding Description

**Root cause — unbounded DB fetch followed by per-row async DB calls**

In `back-end/apps/api/src/transactions/transactions.service.ts`, `getTransactionsToSign` (lines 295–313):

```typescript
const transactions = await this.repo.find({
  where: whereForUser,   // only filters by status ≠ terminal
  relations: ['groupItem'],
  order,
  // ← NO `take` / `skip` here
});

for (const transaction of transactions) {
  const keysToSign = await this.userKeysToSign(transaction, user);
  if (keysToSign.length > 0) result.push({ transaction, keysToSign });
}

return {
  ...
  items: result.slice(offset, offset + limit),   // pagination applied AFTER
};
``` [1](#0-0) 

The `whereForUser` filter only excludes five terminal statuses; it does **not** scope the query to the requesting user's transactions. Every non-terminal transaction in the system is loaded. [2](#0-1) 

`userKeysToSign` delegates to `userKeysRequiredToSign`, which performs its own DB lookups per transaction. With N transactions in the table, the endpoint issues O(N) sequential async DB calls before returning anything. [3](#0-2) 

**Attacker-controlled growth path**

There is no per-user transaction creation limit. A normal authenticated user can call `POST /transactions` repeatedly, each time creating a new transaction that remains in `WAITING_FOR_SIGNATURES` (non-terminal) indefinitely until it expires or is manually canceled. The `CreateTransactionDto` imposes no count cap.

**Exposed endpoint**

The controller exposes `GET /transactions/sign` to any verified user. The comment in the controller even acknowledges the endpoint is no longer used by the front-end, meaning it is a forgotten but fully live attack surface. [4](#0-3) 

The `PaginationParams` decorator enforces a maximum page `size` of 100, but this only limits the *response slice*; it has no effect on the unbounded DB fetch that precedes it. [5](#0-4) 

### Impact Explanation

- **Memory exhaustion**: Loading tens of thousands of transaction rows with their `groupItem` relations into the Node.js heap can exhaust server memory and crash the process.
- **DB connection starvation**: O(N) sequential `await` calls to `userKeysToSign` hold DB connections for the entire duration of the request, starving other concurrent requests.
- **Cascading DoS**: Because the fetch is system-wide (not scoped to the requesting user), a single attacker inflating the transaction table degrades the endpoint for every user who calls it.
- **No cost to attacker**: Transactions can be created with minimal valid input. After the lock/expiry period the attacker's transactions naturally expire, but the damage to other users occurs in the interim.

### Likelihood Explanation

- Precondition: a registered, verified account — the lowest privilege level in the system.
- Attack steps: create many transactions via `POST /transactions`, then call `GET /transactions/sign` once.
- No special knowledge, no privileged keys, no external dependencies.
- The endpoint is still live despite the front-end no longer using it, so it is reachable by any API client.

### Recommendation

1. **Apply `take`/`skip` at the DB layer** inside `getTransactionsToSign` before the loop, mirroring the pattern used in `getTransactions` and `getTransactionsToApprove`.
2. **Scope the query to the requesting user** (join on `signers.userId` or `creatorKey.userId`) so the fetch is bounded by the user's own activity, not the entire table.
3. **Enforce a per-user active-transaction limit** in `createTransaction` to prevent unbounded table growth.
4. **Remove or gate the `/transactions/sign` endpoint** if it is genuinely no longer needed by any client, reducing the attack surface.

### Proof of Concept

1. Register and verify a normal user account.
2. In a loop, call `POST /transactions` with a valid `transactionBytes` payload, creating N transactions (e.g., N = 10,000). Each transaction will sit in `WAITING_FOR_SIGNATURES` status.
3. Call `GET /transactions/sign?page=1&size=10` with the attacker's JWT.
4. Observe: the server executes one `repo.find` that returns all N rows, then issues N sequential `userKeysToSign` DB calls before returning the 10-item page. Server CPU and DB connection usage spike proportionally to N. Concurrent requests from other users time out or fail.
5. The attacker's transactions eventually expire (no cost beyond the initial registration), but all users who called the endpoint during that window experienced degraded or failed responses.
