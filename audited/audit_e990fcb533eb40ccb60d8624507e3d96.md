### Title
Missing Access Control on Transaction Comments Endpoints Allows Any Authenticated User to Read and Write Comments on Arbitrary Transactions

### Summary
The `CommentsController` exposes three endpoints (`POST`, `GET`, `GET /:id`) under `/transactions/:transactionId/comments` with no authorization check to verify whether the requesting user has any relationship to the target transaction. Any verified, authenticated user can post comments to and read comments from transactions they are not a creator, signer, observer, or approver of. The code itself contains an explicit `//TODO need some sort of guard or check to ensure user can comment here` acknowledgment of the missing control.

### Finding Description
All other transaction sub-resources (approvers, observers, signers) enforce access by verifying the requesting user is the transaction creator, a signer, an observer, or an approver before allowing reads or writes. The `CommentsController` performs none of these checks.

**Root cause — `back-end/apps/api/src/transactions/comments/comments.controller.ts`:**

```
@Post()
//TODO need some sort of guard or check to ensure user can comment here
createComment(
  @GetUser() user: User,
  @Param('transactionId', ParseIntPipe) transactionId: number,
  @Body() dto: CreateCommentDto,
) {
  return this.commentsService.createComment(user, transactionId, dto);
}

@Get()
getComments(@Param('transactionId', ParseIntPipe) transactionId: number) {
  return this.commentsService.getTransactionComments(transactionId);
}

@Get('/:id')
getCommentById(@Param('id', ParseIntPipe) id: number) {
  return this.commentsService.getTransactionCommentById(id);
}
``` [1](#0-0) 

The controller is protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` at the class level — meaning any verified organization member passes all guards. [2](#0-1) 

By contrast, `ObserversService.getTransactionObserversByTransactionId` and `ApproversService.getVerifiedApproversByTransactionId` both explicitly check creator/signer/observer/approver membership before returning data, and throw `UnauthorizedException` on failure. [3](#0-2) [4](#0-3) 

The `verifyAccess` method in `TransactionsService` defines the canonical access model (creator, signer, observer, approver, or terminal status), but it is never invoked by the comments service. [5](#0-4) 

### Impact Explanation
- **Unauthorized write**: Any verified user can inject comments into any transaction in the organization, including sensitive multi-party signing workflows they have no legitimate role in. This corrupts the audit trail and communication record of the transaction.
- **Unauthorized read**: Any verified user can enumerate and read all comments on any transaction, exposing potentially sensitive coordination details (key identifiers, signing instructions, business context) that are intended only for transaction participants.
- Impact category: unauthorized state change + cross-user data exposure.

### Likelihood Explanation
- Attacker precondition: valid organization account (verified user). No admin or privileged role required.
- Attack path is a single authenticated HTTP request to `POST /transactions/<id>/comments` or `GET /transactions/<id>/comments` with any valid `transactionId`.
- Transaction IDs are sequential integers, making enumeration trivial.
- The TODO comment in the source confirms the gap is known and unmitigated.

### Recommendation
Apply the same access-verification pattern used by observers and approvers. Before processing any comment operation, call `TransactionsService.verifyAccess(transaction, user)` (or an equivalent inline check) and throw `UnauthorizedException` if the user is not the creator, a signer, an observer, or an approver of the transaction. For the `GET` endpoints, pass the authenticated user and perform the same check.

### Proof of Concept
1. Register two users (User A and User B) in the organization. User B has no relationship to any of User A's transactions.
2. User A creates a transaction (returns `transactionId = N`).
3. User B authenticates and obtains a valid JWT.
4. User B sends:
   ```
   POST /transactions/N/comments
   Authorization: Bearer <user_b_token>
   { "content": "attacker comment" }
   ```
   Expected (correct): `401 Unauthorized`
   Actual: `201 Created` — comment is persisted against User A's transaction.
5. User B sends:
   ```
   GET /transactions/N/comments
   Authorization: Bearer <user_b_token>
   ```
   Expected (correct): `401 Unauthorized`
   Actual: `200 OK` — all comments for the transaction are returned. [1](#0-0)

  async verifyAccess(transaction: Transaction, user: User): Promise<boolean> {
    if (!transaction) throw new BadRequestException(ErrorCodes.TNF);

    if (
      [
        TransactionStatus.EXECUTED,
        TransactionStatus.EXPIRED,
        TransactionStatus.FAILED,
        TransactionStatus.CANCELED,
        TransactionStatus.ARCHIVED,
      ].includes(transaction.status)
    )
      return true;

    const userKeysToSign = await this.userKeysToSign(transaction, user, true);

    return (
      userKeysToSign.length !== 0 ||
      transaction.creatorKey?.userId === user.id ||
      !!transaction.observers?.some(o => o.userId === user.id) ||
      !!transaction.signers?.some(s => s.userKey?.userId === user.id) ||
      !!transaction.approvers?.some(a => a.userId === user.id)
    );
