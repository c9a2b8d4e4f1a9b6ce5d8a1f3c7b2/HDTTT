### Title
Critically Low PBKDF2 Iteration Count (2,560) Enables Fast Offline Brute-Force of Encrypted Private Keys

### Summary
The `deriveKey` function in `front-end/src/main/utils/crypto.ts` uses only **2,560 PBKDF2-SHA512 iterations** to derive the AES-256-GCM key used to encrypt all user private keys stored in the local SQLite database. This is approximately 82–410× below current minimum recommendations (OWASP: 210,000; NIST SP 800-132: 1,000,000 for PBKDF2-SHA512). If an attacker exfiltrates the SQLite database file — via malware, backup exposure, or physical access — they can brute-force the user's password at a rate orders of magnitude faster than a properly hardened scheme would allow, recovering all stored Hedera private keys.

### Finding Description

**Root cause — `front-end/src/main/utils/crypto.ts`, lines 5–9:**

```typescript
export function deriveKey(password: string, salt: Buffer) {
  const iterations = 2560;   // ← critically low
  const keyLength = 32;
  return crypto.pbkdf2Sync(password, salt, iterations, keyLength, 'sha512');
}

**Where the encrypted data lives — `front-end/src/main/services/localUser/keyPairs.ts`, lines 61–87:**

`storeKeyPair` calls `encrypt(keyPair.private_key, password)` and persists the result to the SQLite `KeyPair` table as `private_key`. Every Hedera ED25519 private key the user manages is protected solely by this weak KDF. [3](#0-2) 

Organization credentials (the user's backend password) are also encrypted via the same `encrypt()` path through `organizationCredentials.ts`: [4](#0-3) 

The identical weak `deriveKey` (2,560 iterations) is duplicated in the automation utility: [5](#0-4) 

**Exploit path:**
1. Attacker exfiltrates the SQLite database file (user-level read access — malware, cloud backup, stolen device).
2. Parses any `KeyPair.private_key` blob: first 64 bytes = salt, next 16 = IV, next 16 = GCM auth tag, remainder = ciphertext.
3. Runs offline PBKDF2-SHA512 with 2,560 iterations per password candidate against the known salt.
4. Confirms a correct guess when AES-256-GCM decryption succeeds (auth tag validates).
5. Recovers the plaintext ED25519 private key and gains full control of the associated Hedera account.

### Impact Explanation

A modern GPU (e.g., RTX 4090) achieves roughly **1–2 billion PBKDF2-SHA512 iterations per second**. At 2,560 iterations per password guess, that yields **~390,000–780,000 password candidates per second per GPU**. A common 8-character lowercase password space (~200 billion candidates) is exhausted in under 3 days on a single consumer GPU. Dictionary attacks against typical user passwords succeed in seconds to minutes. Successful exploitation gives the attacker the raw ED25519 private key, enabling them to sign arbitrary Hedera transactions and drain any associated accounts — a direct, irreversible asset loss.

### Likelihood Explanation

The SQLite database is stored in the Electron app's user-data directory, readable by any process running as the same OS user. Malware (a realistic, no-privilege attacker) can silently exfiltrate the file. Cloud backup services (iCloud, Google Drive, OneDrive) may also sync the directory. The attacker needs no interaction with the running application and no knowledge of the app's internals beyond the public blob format (salt || IV || tag || ciphertext), which is trivially reverse-engineered from the open-source code.

### Recommendation

Replace the 2,560-iteration PBKDF2 with a memory-hard KDF already used elsewhere in the codebase. The `dataMigration.ts` file already demonstrates correct Argon2id usage:

```typescript
// dataMigration.ts — correct pattern already in the codebase
const options = {
  type: argon2.argon2id,
  memoryCost: 262144, // 256 MB
  timeCost: 3,
  parallelism: 1,
  salt,
  raw: true,
};
``` [6](#0-5) 

Replace `deriveKey` in `front-end/src/main/utils/crypto.ts` with Argon2id (matching the parameters above), and implement a data-migration flow (analogous to the existing `changeDecryptionPassword`) to re-encrypt stored keys on next login. If PBKDF2 must be retained for compatibility, the minimum is 600,000 iterations for PBKDF2-SHA256 or 210,000 for PBKDF2-SHA512 per current OWASP guidance.

With 2,560 iterations, a single RTX 4090 tests ~500,000 candidates/second. A 10-million-entry password list is exhausted in ~20 seconds.

export function encrypt(data: string, password: string) {
  const iv = crypto.randomBytes(16);
  const salt = crypto.randomBytes(64);

  const key = deriveKey(password, salt);

  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([cipher.update(data, 'utf8'), cipher.final()]);

  const tag = cipher.getAuthTag();

  return Buffer.concat([salt, iv, tag, encrypted]).toString('base64');
}

export async function generateArgon2id(password: string, salt: Buffer) {
  const options = {
    type: argon2.argon2id,
    memoryCost: 262144, // 256MB
    timeCost: 3, // iterations
    parallelism: 1, // threads
    salt,
    raw: true, // get the raw bytes
