### Title
Transaction Creator Can Bypass Approval Threshold by Exploiting Inherited Approval State in New Approver Entries

### Summary
In `approvers.service.ts`, the `createTransactionApprovers` function automatically copies a user's existing `approved`, `signature`, and `userKeyId` state into any newly created approver entry for that same user on the same transaction. Because the duplicate-check (`isNode`) only blocks re-adding a user to the **same parent node**, a malicious transaction creator can add an already-approved user to a **new** threshold node, causing that node to inherit `approved: true` without the user re-approving. Combined with the unrestricted ability to lower a threshold at any time, the creator can satisfy the approval requirement with fewer real approvals than the structure demands.

### Finding Description

**Root cause — `createTransactionApprovers`, lines 318-329:**

```typescript
if (typeof dtoApprover.userId === 'number') {
  const userApproverRecords = await this.getApproversByTransactionId(
    transactionId,
    dtoApprover.userId,
    transactionalEntityManager,
  );

  if (userApproverRecords.length > 0) {
    data.signature = userApproverRecords[0].signature;
    data.userKeyId = userApproverRecords[0].userKeyId;
    data.approved = userApproverRecords[0].approved;   // ← inherited unconditionally
  }
}
``` [1](#0-0) 

Whenever a new leaf approver is inserted, the service queries all existing approver records for that `(transactionId, userId)` pair and blindly copies the first record's approval state into the new row.

**Why the duplicate guard does not protect against this:**

`isNode` only blocks re-adding a user when the new entry would have the **same `listId`** (same parent) as an existing entry:

```typescript
const find: FindManyOptions<TransactionApprover> = {
  where: {
    listId: typeof approver.listId === 'number' ? approver.listId : null,
    userId: ...,
    transactionId: ...,
  },
};
return count > 0 && typeof approver.userId === 'number';
``` [2](#0-1) 

Adding the same user under a **different** parent (a new threshold node) passes the guard and still inherits the approval state.

**No status gate on approver creation or threshold update:**

`getCreatorsTransaction` — the only guard called before both `createTransactionApprovers` and `updateTransactionApprover` — checks only that the caller is the creator; it never checks the transaction's current status:

```typescript
if (transaction.creatorKey?.userId !== user.id)
  throw new UnauthorizedException('Only the creator of the transaction is able to modify it');
``` [3](#0-2) 

This means the creator can restructure the approver tree and lower thresholds **after** approvals have already been collected.

**The behavior is explicitly tested and confirmed:**

```typescript
it('should create basic transaction approver that is already added', async () => {
  jest.spyOn(service, 'getApproversByTransactionId').mockResolvedValueOnce([
    { userId: 1, signature: Buffer.from('0x123'), userKeyId: 1, approved: true },
  ]);
  // ...
  expect(dataSource.manager.create).toHaveBeenCalledWith(TransactionApprover, {
    userId: 1, signature: Buffer.from('0x123'), userKeyId: 1, approved: true,
  });
``` [4](#0-3) 

### Impact Explanation

A malicious transaction creator (a normal, unprivileged user) can cause a transaction to advance to `WAITING_FOR_EXECUTION` — and ultimately be submitted to the Hedera network — without collecting the number of genuine approvals that the approver structure was designed to require. Concretely:

- A transaction requiring approval from both UserA **and** UserB (threshold=2) can be executed after only UserA approves.
- The creator adds UserA to a new threshold node (threshold=1); that node inherits `approved: true`; the creator lowers the root threshold to 1; the transaction is now fully "approved."
- Any on-chain action encoded in the transaction bytes (token transfers, account updates, file changes, etc.) executes without the consent of the bypassed approvers.

### Likelihood Explanation

- **Attacker profile:** the transaction creator — a standard authenticated user with no elevated privileges.
- **Preconditions:** the creator must have created a transaction with at least one approver who has already approved. This is the normal operating state of any multi-approver transaction mid-lifecycle.
- **Steps required:** two API calls after the first approval arrives — one `POST /transactions/:id/approvers` and one `PATCH /transactions/:id/approvers/:approverId`. Both endpoints are reachable by any authenticated creator.
- **No cryptographic break required:** the inherited `signature` bytes are a real, valid signature already stored in the database; the chain service will accept them.

### Recommendation

1. **Do not inherit approval state into new approver entries.** When `createTransactionApprovers` inserts a new leaf node, omit `signature`, `userKeyId`, and `approved` entirely. If the intent is to avoid requiring a re-signature when a user is moved within the tree, handle that explicitly only for `updateTransactionApprover` (which already clears these fields when the `userId` changes).

2. **Gate approver-structure mutations on transaction status.** In `getCreatorsTransaction` (or at the start of `createTransactionApprovers` and `updateTransactionApprover`), reject modifications when the transaction is in `WAITING_FOR_SIGNATURES` or `WAITING_FOR_EXECUTION`. Structural changes after approvals have been collected invalidate the integrity of the approval process.

### Proof of Concept

**Setup:** Transaction T requires threshold=2 approvals from UserA and UserB. UserA has approved (their DB record has `approved: true`, `signature: <sigA>`).

**Step 1 — Add a new threshold node containing UserA:**
```
POST /transactions/T/approvers
Authorization: Bearer <creator-token>
{
  "approversArray": [{
    "threshold": 1,
    "approvers": [{ "userId": <UserA_id> }, { "userId": <UserC_id> }]
  }]
}
```
Result: a new `TransactionApprover` row for UserA is inserted with `approved: true` (inherited), satisfying the new threshold node (threshold=1).

**Step 2 — Lower the root threshold to 1:**
```
PATCH /transactions/T/approvers/<root_node_id>
Authorization: Bearer <creator-token>
{ "threshold": 1 }
```
Result: the root threshold is now 1. The new threshold node (already `approved: true`) satisfies it.

**Outcome:** The transaction status update listener evaluates the approver tree, finds the root threshold met, and transitions the transaction to `WAITING_FOR_EXECUTION` — then submits it to the Hedera network — without UserB ever approving.
