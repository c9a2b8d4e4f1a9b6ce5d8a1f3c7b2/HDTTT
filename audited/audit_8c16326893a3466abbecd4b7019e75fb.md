### Title
Missing `AdminGuard` on `GET /user-keys` Allows Any Authenticated User to Enumerate All Users' Cryptographic Key Metadata

### Summary
The `GET /user-keys` endpoint in `UserKeysAllController` is guarded only by JWT authentication and verified-user checks, but is missing the `AdminGuard`. The underlying service method `getUserKeys` performs an unfiltered database query returning every `UserKey` record in the system — including `mnemonicHash` and `index` for all users — to any authenticated non-admin user. This is a direct analog to the external report's missing `onlyOwner` pattern: a privileged-scope operation is reachable by an unprivileged caller.

---

### Finding Description

**Root cause — missing `AdminGuard` on the controller:**

`UserKeysAllController` at `back-end/apps/api/src/user-keys/user-keys-all.controller.ts` applies only three guards:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, VerifiedUserGuard)
export class UserKeysAllController {
``` [1](#0-0) 

`AdminGuard` is absent. Any verified, authenticated user — not just admins — can reach the `getUserKeys` handler.

**The handler passes no user context to the service:**

```typescript
getUserKeys(
  @PaginationParams() paginationParams: Pagination,
): Promise<PaginatedResourceDto<UserKey>> {
  return this.userKeysService.getUserKeys(paginationParams);
}
``` [2](#0-1) 

No `@GetUser()` is injected; the caller's identity is never forwarded.

**The service performs a completely unfiltered query:**

```typescript
async getUserKeys({ page, limit, size, offset }: Pagination) {
  const [items, total] = await this.repo.findAndCount({
    take: limit,
    skip: offset,
  });
``` [3](#0-2) 

There is no `where` clause. Every `UserKey` row in the database is returned, paginated only by `limit`/`offset`.

**The response DTO exposes sensitive fields for all users:**

```typescript
export class UserKeyDto {
  @Expose() id: number;
  @Expose() userId: number;
  @Expose() mnemonicHash?: string;
  @Expose() index?: number;
  @Expose() publicKey: string;
  @Expose() deletedAt?: Date;
}
``` [4](#0-3) 

`mnemonicHash` is the hash of a user's 24-word BIP-39 mnemonic recovery phrase; `index` is the HD derivation index. Both are exposed for every user in the organization.

**Contrast with the correctly scoped per-user endpoint:**

The sibling controller `UserKeysController` at `GET /user/:userId/keys` uses `getUserKeysRestricted`, which filters by `userId` and conditionally hides `mnemonicHash`/`index` unless the requester is the owner:

```typescript
select: {
  mnemonicHash: user.id === userId,
  index: user.id === userId,
  ...
}
``` [5](#0-4) 

The `GET /user-keys` endpoint bypasses all of this scoping entirely.

---

### Impact Explanation

Any verified, non-admin user can issue `GET /user-keys?page=1&limit=100` (and paginate further) to retrieve the full `UserKey` table for the entire organization. The response includes:

- `publicKey` — the on-chain Hedera public key for every user
- `mnemonicHash` — the hash of every user's 24-word mnemonic recovery phrase
- `index` — the HD derivation index for every user's key

An attacker who obtains `mnemonicHash` values can use them to correlate or brute-force mnemonic phrases offline (if the hash function is weak or the mnemonic space is constrained). Combined with `publicKey` and `index`, this gives an attacker a complete map of the organization's key infrastructure, enabling targeted attacks against individual users' wallets and signing keys. This is a cross-tenant confidentiality breach affecting all users in the organization.

---

### Likelihood Explanation

The attacker precondition is minimal: a valid, verified account on the organization's backend — the lowest privilege level that exists after login. No admin access, no special role, no leaked secrets are required. The exploit is a single unauthenticated HTTP GET request after login. Any malicious insider, compromised low-privilege account, or attacker who registers a user account can immediately exploit this.

---

### Recommendation

Add `AdminGuard` to `UserKeysAllController` so that only administrators can enumerate all user keys:

```typescript
@UseGuards(JwtBlackListAuthGuard, JwtAuthGuard, AdminGuard, VerifiedUserGuard)
export class UserKeysAllController {
``` [1](#0-0) 

Alternatively, if this endpoint is not intended for admin use, remove it entirely and rely on the properly scoped `GET /user/:userId/keys` endpoint which already enforces per-user access control via `getUserKeysRestricted`.

---

### Proof of Concept

**Preconditions:** Attacker has a valid, verified (non-admin) account on the organization server.

**Steps:**

1. Attacker logs in:
   ```
   POST /auth/login
   { "email": "attacker@example.com", "password": "..." }
   → { "accessToken": "<JWT>" }
   ```

2. Attacker calls the unguarded endpoint:
   ```
   GET /user-keys?page=1&limit=100
   Authorization: Bearer <JWT>
   ```

3. Response contains all user keys across the organization:
   ```json
   {
     "totalItems": 42,
     "items": [
       {
         "id": 1,
         "userId": 2,
         "mnemonicHash": "7a40e67733edaec462d6b4a31f026cc37f96f767d6a581e41f71785516d42929",
         "index": 0,
         "publicKey": "5a7245ff4fbbc301ec0a4b4c9d04117d7527759431d8fab1729a7dbfb715094f",
         "deletedAt": null
       },
       ...
     ]
   }
   ``` [6](#0-5) [7](#0-6) 

The attacker receives `mnemonicHash`, `index`, and `publicKey` for every user in the system — data that the correctly scoped `getUserKeysRestricted` endpoint explicitly withholds from non-owners.
