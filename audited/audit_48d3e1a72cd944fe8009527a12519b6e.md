### Title
Keychain Mode Provides No Per-Operation Authentication, Enabling Unauthorized Transaction Signing on an Unattended Desktop

### Summary
When the Hedera Transaction Tool is configured in keychain mode (macOS), Electron's `safeStorage` API decrypts private keys silently with zero user-presence verification. The application also auto-logs in on startup without any credential prompt. An attacker who gains access to an unlocked, running instance of the application can sign and submit Hedera transactions — including HBAR transfers — without entering any password or passing any authentication challenge. No idle timeout exists for keychain-mode users.

### Finding Description

**Two-part root cause:**

**Part 1 — Automatic login with no credential gate (keychain mode)**

`useAutoLogin.ts` runs on mount and, when `useKeychain` is `true`, calls `keychainLogin()` which immediately calls `user.login(staticUser.id, staticUser.email, true)` with no password: [1](#0-0) 

The `login()` action in the Pinia store sets `personal.value` with `password: null` and `useKeychain: true`, completing the session with no secret ever verified: [2](#0-1) 

**Part 2 — Transaction signing bypasses all authentication for keychain users**

`signTransaction` in the main-process service checks `useKeychain` first and, when true, calls `safeStorage.decryptString()` directly — no password argument is consulted, no user prompt is shown: [3](#0-2) 

The renderer-side composable `usePersonalPassword` mirrors this: when `useKeychain` is `true`, `getPasswordV2` immediately invokes the callback with `null` — the password modal is never opened: [4](#0-3) 

`SignPersonalRequestHandler.vue` explicitly skips the password requirement for keychain users before calling `signTransaction`: [5](#0-4) 

**Part 3 — No idle timeout for keychain users**

The `passwordStoreDuration` / `passwordExpiresAt` expiry mechanism in the store only applies to password-mode users. `isLoggedInWithValidPassword` is never consulted for keychain users because `getPassword()` is never called in that path: [6](#0-5) [7](#0-6) 

The 10-minute expiry window is never enforced for keychain users; the session is permanent for the lifetime of the process.

### Impact Explanation
An attacker with physical access to an unlocked desktop where the application is running (or can be launched) in keychain mode can:
1. Open the Hedera Transaction Tool (auto-login completes silently).
2. Navigate to any pending transaction or create a new transfer.
3. Click "Sign Transaction" — private keys are decrypted and the transaction is signed and submitted to the Hedera network with no authentication challenge.

This results in direct, irreversible theft of HBAR or HTS tokens from the victim's Hedera accounts. The impact is equivalent to the ZecWallet scenario: complete fund loss with no authentication barrier.

### Likelihood Explanation
- **Keychain mode is a first-class, documented option** presented to users on macOS during initial setup (`KeychainOption.vue`). Many users will choose it to avoid typing a password repeatedly.
- Desktop machines are routinely left unlocked (brief absence, shared office, etc.) — a more common scenario than a stolen locked phone.
- The attacker needs only momentary unsupervised access to the running application; no technical skill beyond basic UI navigation is required.
- macOS is the primary supported platform for keychain mode (`process.platform === 'darwin'`), and macOS is the dominant platform for professional/enterprise Hedera users. [8](#0-7) 

### Recommendation
1. **Short term**: For keychain mode, require macOS Touch ID (via `LocalAuthentication` / `node-mac-auth` or Electron's `systemPreferences.promptTouchID`) before each private-key decryption operation. This is the direct analog of the TouchID recommendation in the ZecWallet report.
2. **Short term**: Implement an idle timeout for keychain-mode sessions (e.g., 5–10 minutes of inactivity) that locks the application and requires Touch ID or OS-level re-authentication before resuming.
3. **Long term**: Treat keychain-mode decryption as a privileged operation gated by OS biometrics, not merely by OS session login. Encrypting the keychain entry with a user-supplied PIN or biometric-derived key would require the attacker to authenticate even if the OS session is unlocked.

### Proof of Concept

**Preconditions**: macOS machine, Hedera Transaction Tool installed and configured in keychain mode, application running (or launchable), victim's desktop session unlocked, victim briefly absent.

**Steps**:
1. Sit at the victim's unlocked desktop. Open the Hedera Transaction Tool (if not already open). The `useAutoLogin` composable fires on mount, calls `keychainLogin()`, and the user is fully logged in — no password prompt appears.
2. Navigate to **Transactions** → select any pending transaction awaiting signature, or create a new HBAR transfer to an attacker-controlled account.
3. Click **Sign Transaction**. `SignPersonalRequestHandler.vue` calls `user.getPassword()`, which returns `null`; because `user.personal.useKeychain === true`, no error is thrown and no modal is shown. `signTransaction` is called with `userPassword = null`.
4. Inside `signTransaction` (main process), `useKeychain` is `true`, so `safeStorage.decryptString(buffer)` is called silently. The private key is decrypted and the transaction is signed.
5. The signed transaction is submitted to the Hedera network. Funds are transferred. No authentication was ever requested.

**Expected (secure) behavior**: Step 3 should trigger a Touch ID prompt or equivalent before decrypting any private key.
**Actual behavior**: Signing completes silently with no user-presence verification.
