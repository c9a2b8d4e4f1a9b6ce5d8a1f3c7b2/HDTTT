### Title
Unbounded Array Input in `POST /transactions/signers` Enables Single-Request Server Resource Exhaustion

### Summary
The `uploadSignatureMap` endpoint accepts an array of `UploadSignatureMapDto` objects with no enforced maximum size. A single authenticated request containing a large array causes the server to concurrently process all items via `Promise.all`, performing unbounded DB queries, cryptographic validation, and bulk writes. A developer comment in the codebase explicitly acknowledges the missing API-side limit. Any verified user can trigger this without elevated privileges.

### Finding Description

**Entry point:**
`POST /transactions/signers` — protected only by `JwtBlackListAuthGuard`, `JwtAuthGuard`, and `VerifiedUserGuard` (any verified user). [1](#0-0) 

The controller accepts `UploadSignatureMapDto | UploadSignatureMapDto[]` with no array size constraint: [2](#0-1) 

No `@ArrayMaxSize()` or equivalent guard exists on the DTO. The body is passed directly to `uploadSignatureMaps`: [3](#0-2) 

Inside `validateAndProcessSignatures`, all items are processed **concurrently** via `Promise.all(dto.map(...))`: [4](#0-3) 

Each item triggers: a transaction DB lookup, signature deserialization (`SDKTransaction.fromBytes`), cryptographic validation, and a DB write. With N items in the array, N concurrent async operations are spawned simultaneously.

A developer comment in the parallel `importSignatures` path explicitly acknowledges the missing limit: [5](#0-4) 

> `//Added a batch mechanism, probably should limit this on the api side of things` [6](#0-5) 

A secondary unbounded input exists in `CreateTransactionObserversDto`, which accepts `userIds: number[]` with only `@ArrayMinSize(1)` and no `@ArrayMaxSize()`: [7](#0-6) 

The `createTransactionObservers` service iterates over all `userIds` and bulk-saves them: [8](#0-7) 

### Impact Explanation

A single crafted POST request with an array of thousands of entries causes:
- **Unbounded concurrent DB connections** — `Promise.all` spawns all DB queries simultaneously, exhausting the PostgreSQL connection pool.
- **Memory exhaustion** — all `SDKTransaction.fromBytes` deserialized objects are held in memory simultaneously.
- **CPU saturation** — concurrent cryptographic signature validation across all items.
- **Service unavailability** — the NestJS event loop stalls under memory/CPU pressure, making the API unresponsive for all users.

This is not a volumetric DDoS; a single authenticated HTTP request is sufficient to trigger the condition.

### Likelihood Explanation

- **Attacker precondition**: Any registered and verified user account — no admin or special role required.
- **Attack complexity**: Trivially low. The attacker constructs one HTTP POST with a large JSON array body.
- **No rate-limit evidence found** on this specific endpoint that would prevent a large single payload (rate limiting typically counts requests, not payload item count).
- The developer's own comment confirms awareness that the API-side limit is missing.

### Recommendation

1. Add `@ArrayMaxSize(N)` to `UploadSignatureMapDto[]` at the controller level (e.g., `N = 50`) and to `CreateTransactionObserversDto.userIds`.
2. Replace `Promise.all(dto.map(...))` with a concurrency-limited alternative (e.g., `p-limit` with a cap of 10) in `validateAndProcessSignatures`.
3. Apply a global NestJS body size limit and a per-array-field size guard via a validation pipe.

### Proof of Concept

```
POST /transactions/signers
Authorization: Bearer <valid_verified_user_jwt>
Content-Type: application/json

[
  { "id": 1, "signatureMap": { ... } },
  { "id": 2, "signatureMap": { ... } },
  ... (repeat 10,000 times with arbitrary transaction IDs)
]
```

**Expected server behavior:**
- `loadTransactionData` issues `IN (10000 ids)` DB query.
- `validateAndProcessSignatures` spawns 10,000 concurrent `Promise.all` branches, each calling `SDKTransaction.fromBytes` and DB lookups.
- Node.js heap grows proportionally; PostgreSQL connection pool is exhausted.
- API becomes unresponsive for all concurrent users until the request completes or the process OOMs.

  private async validateAndProcessSignatures(
    dto: UploadSignatureMapDto[],
    user: User,
    transactionMap: Map<number, Transaction>,
    signersByTransaction: Map<number, Set<number>>
  ) {
    // Build user key lookup once
    const userKeyMap = new Map<string, UserKey>();
    for (const key of user.keys) {
      userKeyMap.set(key.publicKey, key);
    }

    return Promise.all(
      dto.map(async ({ id, signatureMap: map }) => {
        try {
          const transaction = transactionMap.get(id);
          if (!transaction) return { id, error: ErrorCodes.TNF };

          // Validate transaction status
          const statusError = this.validateTransactionStatus(transaction);
          if (statusError) return { id, error: statusError };

          // Process signatures
          const { sdkTransaction, userKeys, isSameBytes } = await this.processTransactionSignatures(
            transaction,
            map,
            userKeyMap,
            signersByTransaction.get(id) || new Set()
          );

          return {
            id,
            transaction,
            sdkTransaction,
            userKeys,
            isSameBytes,
            error: null,
          };
        } catch (err) {
          console.error(`[TX ${id}] Error:`, err.message);
          return { id, error: err.message };
        }
      })
    );
  }
